# 0.36 Status (as of 2026-03-01)

## 0.36 Scope

This status tracks grouped execution hardening and aggregate-surface expansion
work for `0.36.x` from:

- `docs/design/0.36-ordered-group-execution.md`
- `docs/design/0.36-having.md`

`0.36` extends grouped execution foundations from `0.35` with ordered grouping
strategy formalization, HAVING support, DISTINCT aggregate semantics, streaming
eligibility enforcement, and grouped projection contract hardening.

- ordered vs hash grouped execution strategy routing

## 0.36 Contract Freeze (`0.36.x`)

As of `2026-03-01`, the following surfaces are frozen for `0.36.x` patch
releases:

1. grouped projection schema surface
2. explain grouped output shape
3. fingerprint hashing surface used for grouped continuation signatures
4. grouped continuation token wire format/version behavior (`v1` shape:
   `signature`, `last_group_key`, `direction`, `initial_offset`)
5. grouped DISTINCT typed error surface
6. streaming-eligibility downgrade behavior (`OrderedGroup` hint may downgrade
   to `HashGroup`; executor revalidation must not upgrade beyond planner
   eligibility)

`0.36.x` patch releases may fix defects but must not introduce semantic or shape
drift in these frozen surfaces. Any intentional surface change is deferred to a
new minor line (`0.37+`).

## Cross-Feature Matrix Audit (`0.36` closeout)

The following cross-feature combinations were audited for deterministic,
bounded behavior and are explicitly covered:

1. Ordered + DISTINCT + HAVING (policy gate): rejected deterministically
   (`GroupPlanError::DistinctHavingUnsupported`)
   - `db::query::intent::tests::grouped_having_with_distinct_is_rejected_for_ordered_eligible_shape`
2. Hash + DISTINCT + HAVING (policy gate): rejected deterministically
   (`GroupPlanError::DistinctHavingUnsupported`)
   - `db::query::intent::tests::grouped_having_with_distinct_is_rejected`
3. Ordered + HAVING only: ordered grouped strategy remains eligible for
   supported HAVING operators
   - `db::query::explain::tests::explain_grouped_strategy_keeps_ordered_group_for_supported_having_operator`
4. DISTINCT hitting per-group cap: typed grouped failure
   (`GroupError::DistinctBudgetExceeded { resource: "distinct_values_per_group", .. }`)
   - `db::executor::aggregate::tests::contracts::grouped_aggregate_state_enforces_distinct_values_per_group_limit`
5. DISTINCT hitting total cap: typed grouped failure
   (`GroupError::DistinctBudgetExceeded { resource: "distinct_values_total", .. }`)
   - `db::executor::aggregate::tests::contracts::grouped_aggregate_state_enforces_distinct_values_total_limit`
6. Pagination mid-group with DISTINCT: no group split across pages/continuation
   drift
   - `db::executor::tests::paged_builder::grouped_fluent_execute_count_distinct_pagination_does_not_split_single_group`
7. Pagination with HAVING filter-out between pages: continuation resumes to next
   matching group without boundary drift
   - `db::executor::tests::paged_builder::grouped_fluent_execute_having_pagination_skips_filtered_middle_group`
8. Strategy downgrade on explain surface: ordered hint downgrades to hash for
   ineligible HAVING operators
   - `db::query::explain::tests::explain_grouped_strategy_downgrades_to_hash_for_unsupported_having_operator`
