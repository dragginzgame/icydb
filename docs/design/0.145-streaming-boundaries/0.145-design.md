# IcyDB 0.145 — Streaming Boundaries & Materialization Elimination

## Status

Proposed

## Summary

0.145 reduces avoidable row materialization in the query execution spine.

The current execution model is structurally clean, but several runtime paths
still collect rows, group candidates, or projected values into `Vec`s before
the next stage immediately consumes them. This release focuses on moving owned
allocation to real boundaries only:

* public SQL/fluent response construction
* `ORDER BY` sorting
* `DISTINCT` key retention
* grouped result ordering where all groups must be compared
* cursor token construction

Every other stage should prefer incremental or visitor-based handoffs,
bounded heaps, or borrowed row access until ownership is required. `RowView`
remains a read-only view over row data; it is not a general streaming
primitive.

## Goals

* Reduce peak memory use for large result sets and blob-heavy rows.
* Avoid full-row ownership when only selected fields, row keys, or aggregate
  inputs are needed.
* Stream bounded grouped and count windows directly into top-k state.
* Push SQL `RETURNING` ownership to the exact returned-field boundary.
* Preserve the existing query planning, ordering, cursor, and response shapes.

## Non-Goals

* No public API changes.
* No query semantic changes.
* No new planner features.
* No cursor format changes.
* No persisted format changes.
* No broad executor rewrite.
* No speculative streaming abstraction where a local visitor or fold is enough.

## Streaming Rule

Rows should flow through executor stages.

They should be collected only when a stage genuinely needs ownership of the
whole candidate set. A `Vec` is justified when the stage must sort, retain
distinct keys or rows, build a public response, encode cursor state, or expose
an owned API boundary.

If a stage only filters, projects, counts, compares, or inserts into a bounded
heap, it should consume rows incrementally.

Materialization is not the problem. Accidental materialization is. Owned
buffers remain the right tool at semantic and public API boundaries; 0.145
removes staging buffers that do not define such a boundary.

## Execution Modes

Every executor stage touched by 0.145 must be classified into one of three
execution modes.

### Streaming

Rows are consumed incrementally without retaining prior candidates.

Allowed operations:

* filtering
* projection
* single-pass aggregation
* simple comparisons

Not allowed:

* global ordering
* `DISTINCT` accumulation
* cursor boundary construction that requires ordered retained state

### Bounded Streaming

Rows are consumed incrementally into a bounded structure.

Allowed operations:

* `ORDER BY ... LIMIT` when a bounded heap preserves the same ordering
* grouped top-k selection
* bounded grouped windows

Memory usage must be proportional to the bound, not to the number of input
rows or groups.

### Materialized

All candidates are collected before producing output.

Required operations:

* `DISTINCT` retention
* unbounded `ORDER BY`
* cursor token construction requiring ordered retained state
* final public response ownership
* unbounded grouped output that must be globally sorted

## Ownership Boundaries

### Must Materialize

These boundaries are intentionally owned:

* SQL response rows (`Vec<Vec<Value>>`)
* fluent public result vectors
* `ORDER BY` candidates that cannot be streamed in final order
* `DISTINCT` key sets and accepted output rows
* cursor token payloads
* unbounded grouped output that must be globally sorted

### Should Stream

These boundaries should not require full intermediate row ownership:

* grouped bounded top-k candidate selection
* grouped count bounded windows
* field-list `RETURNING` projection
* aggregate helpers that only need keys or one field value
* route-satisfied page scans with no sort/cursor materialization need

## Initial Hotspots

### 1. Grouped Count Bounded Window

Location:

```text
crates/icydb-core/src/db/executor/aggregate/runtime/grouped_fold/count/window.rs
```

Current shape:

```text
all grouped counts
  -> qualifying Vec
  -> bounded heap
  -> output rows
```

Target shape:

```text
all grouped counts
  -> HAVING/resume check
  -> bounded heap
  -> output rows
```

This is the safest first patch because it removes one staging allocation
without changing grouping, ordering, or result construction.

### 2. Grouped Page Candidate Finalization

Location:

```text
crates/icydb-core/src/db/executor/aggregate/runtime/grouped_fold/page_finalize.rs
```

Current shape:

```text
finalized groups
  -> Vec<GroupedPageCandidate>
  -> bounded heap
```

Target shape:

```text
finalized groups
  -> candidate visitor
  -> bounded heap
```

Unbounded grouped output may still collect because global ordering requires a
full candidate set.

### 3. SQL RETURNING Field Projection

Location:

```text
crates/icydb-core/src/db/session/sql/execute/write_returning.rs
```

Current shape:

```text
full owned row
  -> clone selected fields
  -> response row
```

Target shape:

```text
selected fields only
  -> response row
```

For `RETURNING *`, full-row ownership remains correct. For `RETURNING field`,
especially blob-heavy entities, the executor should not copy non-returned
fields.

Mutation semantics must remain unchanged. The projected values must reflect
the same inserted, replaced, updated, or deleted row state as today, in the
same order. Selected fields may be moved out of owned rows only when no later
logic needs the full row.

### 4. Aggregate Helper Row Staging

Location:

```text
crates/icydb-core/src/db/executor/aggregate/helpers.rs
crates/icydb-core/src/db/executor/aggregate/mod.rs
```

Current shape:

```text
materialized rows
  -> projected key/value Vec
  -> aggregate or order statistic
```

Target shape:

```text
row stream
  -> key/value visitor
  -> aggregate or bounded/order statistic state
```

`min_max` can stream comparisons. `nth` and `median` still need an ordering
structure, but they do not need full raw row staging before projection.

Streaming aggregate helpers must preserve numeric coercion behavior, `NULL`
handling, and comparison semantics exactly. This is especially important for
`min`/`max`, where field-target comparison can differ from generic value
comparison.

### 5. Kernel Page Scan Boundary

Location:

```text
crates/icydb-core/src/db/executor/terminal/page/scan.rs
crates/icydb-core/src/db/executor/terminal/page/post_access.rs
crates/icydb-core/src/db/executor/terminal/page/post_scan.rs
```

Current shape:

```text
scan
  -> Vec<KernelRow>
  -> post-access filtering/windowing/sort
  -> structural payload Vec
```

Target shape:

```text
scan
  -> streaming sink for route-satisfied simple pages
  -> owned Vec only when sort/cursor/post-access requires it
```

The initial sink is local to page execution and non-composable. It is not a
general executor-wide streaming abstraction in 0.145.

Streaming applies only to route-satisfied scans where the prepared plan proves
the access path already satisfies the required order and no DISTINCT,
cursor-wide materialization, or post-access sort is required. The existing
materialized kernel-row path remains the fallback for every other case.

This is the highest-impact but highest-risk slice. It should come after the
smaller bounded-window and returning fixes.

## Design Constraints

### Preserve Fail-Closed Decode

Streaming must not weaken persisted-row decode errors. Any borrowed row view
must still come from the same checked decode path currently used by the owned
row path.

### Preserve Ordering Contracts

Streaming is allowed only when the prepared plan already proves the access path
is in the required final order, or when the stage is order-insensitive.

If the executor must compare all candidates to determine output order, the
stage remains a materialization boundary.

### Preserve Cursor Contracts

Cursor validation and cursor boundary construction remain prepared-plan
owned. Streaming may reduce pre-cursor staging only when the cursor contract
does not require the full row set.

### Preserve DISTINCT Contracts

`DISTINCT` may hash borrowed rows or values, but accepted keys and retained
rows remain owned as needed. The distinct accumulator is a real retention
boundary, not an accidental allocation.

### Preserve Response Shapes

SQL and fluent APIs still return owned rows and values. 0.145 only changes how
late those rows become owned.

## Slice Plan

### Slice 1: Bounded Grouped Count

Remove the `qualifying` staging vector from grouped count bounded windows and
feed qualifying candidates directly into the existing bounded heap.

Validation:

```bash
cargo fmt --all
cargo test -p icydb-core grouped --features sql -- --nocapture
cargo test -p icydb-core sql_grouped --features sql -- --nocapture
cargo check -p icydb-core --tests --features sql
cargo clippy -p icydb-core --all-targets --features sql -- -D warnings
git diff --check
```

### Slice 2: Generic Grouped Candidate Visitor

Replace bounded grouped page candidate pre-collection with a visitor/fold that
feeds the bounded heap directly.

Keep unbounded grouped finalization materialized.

### Slice 3: SQL RETURNING Field Ownership

Avoid constructing or cloning full response rows when SQL `RETURNING` names a
subset of fields.

Focus blob-heavy rows first:

* `INSERT ... RETURNING blob_field`
* `UPDATE ... RETURNING thumbnail`
* `REPLACE ... RETURNING id, blob_field`
* `DELETE ... RETURNING blob_field`

### Slice 4: Aggregate Helper Streaming

Stream aggregate helper inputs where the helper only needs keys or one field
value.

Do not change aggregate semantics or ordered statistic behavior.

### Slice 5: Kernel Page Streaming Boundary

Introduce a narrow streaming sink for route-satisfied simple page scans.

Only apply when no post-access sort, DISTINCT retention, or cursor-wide
materialization is required.

Do not generalize this into an executor-wide streaming framework.

## Remaining 0.145 Follow-Up Work

The initial streaming-boundary cleanup landed the local materialization wins
first. The remaining 0.145 work is still in scope, but each item needs an
explicit boundary decision before implementation:

* predicate-aware index streaming
* `INSERT ... SELECT` streaming rewrite
* DISTINCT streaming improvements beyond borrowed hashing already present
* cursor boundary streaming
* aggregate fallback redesign

These require broader cross-layer work than the initial cleanup slices and
should land as deliberate follow-ups rather than incidental changes inside an
unrelated patch.

## Validation Strategy

Every slice must run:

```bash
cargo fmt --all
cargo check -p icydb-core --tests --features sql
cargo clippy -p icydb-core --all-targets --features sql -- -D warnings
git diff --check
```

Run targeted tests for each slice:

* grouped tests for grouped candidate changes
* SQL write tests for `RETURNING`
* SQL select/projection tests for page boundary changes
* blob SQL tests for large-row ownership changes
* `ORDER BY ... LIMIT` tests with stable tie-breaking
* cursor pagination tests that compare page boundaries and tokens

Run the SQL perf harness after slices that affect SQL row flow.

## Regression Checks

Before closing 0.145, verify:

* `ORDER BY` cases still materialize where required.
* `ORDER BY ... LIMIT` produces identical rows, including tie-breaking.
* `DISTINCT` cases retain stable deduplication behavior.
* cursor pagination emits identical pages and cursor tokens.
* blob `INSERT`/`UPDATE`/`REPLACE`/`DELETE RETURNING` tests do not copy
  unreturned fields unnecessarily.
* grouped bounded and unbounded outputs remain identical.

## Expected Outcome

After 0.145:

* bounded grouped paths hold fewer intermediate candidates
* blob-heavy SQL write responses avoid unnecessary full-row copies
* aggregate helpers stop staging raw rows where projected inputs are enough
* the remaining materialization points are explicit, justified boundaries

This prepares the executor for future larger streaming work without changing
query behavior or public API contracts.
