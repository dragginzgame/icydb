# 0.189 Audit Fallout Cleanup

Status: tentative design.

Theme: minimum viable IC-native reliability cleanup after the 0.187 adversarial
database audit.

This line responds to `audit-results.md` and `chatgpt-evaluation.md`. The audit
is useful, but it must be translated from a SQLite/POSIX file-database frame
into IcyDB's Internet Computer execution and persistence model.

0.189 should not make IcyDB look like SQLite. It should apply SQLite-level
seriousness to IC-native failure modes: stable memory layout, commit-marker
replay, trap and instruction-limit interruption, resource exhaustion,
upgrade/recovery boundaries, virtual-memory partitioning, and API surfaces that
can lead users into data loss.

## Context

The audit found no proven critical corruption bug. It did find credible gaps:

- no-default test configurations do not compile;
- marker/journal recovery has a plausible design but lacks adversarial
  failpoint proof;
- raw journal batch bytes are stored as an unbounded stable-structure value
  before codec size checks run;
- deterministic marker/journal corruption tests are incomplete;
- volatile `heap()` stores and non-atomic batch helpers are easy to misuse;
- process-global recovery hints deserve multi-registry proof;
- broader persisted-format fuzzing, Miri coverage, resource modeling, and
  operator docs remain necessary but are too broad for one patch line.

The ChatGPT evaluation correctly pushes back on POSIX-only expectations. IcyDB
does not need fsync, directory sync, file rename ordering, or multi-process file
locks unless it starts claiming to be a normal embedded file database. The real
question is whether IcyDB's IC-native durability contract is adversarially
proven.

## IC Platform Baseline

0.189 assumes the current IcyDB contract:

- Mutations execute synchronously inside one Internet Computer update message.
- IcyDB mutation execution must not `await`, yield, or allow reentrancy.
- Canister update messages are serialized for one canister.
- IcyDB does not rely on IC trap rollback for database correctness.
- Guarded read and write entrypoints must run recovery before normal access if
  a marker may be present.
- Stable memory is the durable substrate, partitioned by memory IDs and virtual
  memories, not by host files.
- Heap stores are volatile and are not a durability mechanism.

The IC-native failure model for this line is:

- returned storage/recovery errors at durable phase boundaries;
- panic or unwind after a durable stable-memory write;
- trap-like interruption where host tests can approximate it;
- stale or malformed stable-memory bytes from bugs, upgrades, test fixtures, or
  future backup/restore/import tooling;
- interrupted recovery that must be safe to repeat;
- multiple logical registries in host tests even if production canisters use one
  serialized execution domain.

The POSIX failure model is not part of 0.189:

- no fsync/fdatasync work;
- no directory fsync work;
- no rename-atomicity work;
- no cross-process file-locking work;
- no OS permission or ENOSPC taxonomy unless a future backend actually exposes
  those errors.

## Recovery Oracle

0.189 failpoint tests must compare recovered storage against an explicit
expected model, not merely assert that recovery succeeds.

If interruption occurs before the commit marker is durably visible, guarded
reentry must expose the pre-mutation state.

If interruption occurs after the commit marker is durably visible, guarded
reentry must expose the full marker-authorized post-commit state after recovery.

Intermediate states are not acceptable merely because they are internally
consistent. Row data, derived indexes, fold watermarks, marker state, and
journal-tail visibility must agree with the expected model.

Every failpoint test must state:

- durable phase reached;
- expected recovery state;
- whether the marker is expected to remain or clear after recovery;
- whether journal tail records are expected to replay, fold, or be ignored;
- whether derived indexes are expected to rebuild or already be valid.

This is the difference between proving that recovery did not panic and proving
that recovery preserved atomicity.

## Durable Phase Boundaries

Failpoints are classified by durable authority, not by source location alone.

| Phase | Durable authority exists? | Expected recovery state |
| --- | --- | --- |
| Before marker write | No | Pre-mutation state. |
| After marker write | Yes: marker payload | Full marker-authorized post-commit state. |
| Before first marker-bound journal batch append | Yes: marker payload | Recovery republishes or folds the full commit. |
| After prefix of marker-bound journal batches | Yes: marker payload plus partial journal publication | No duplicate apply; missing batches are reconstructed or replayed safely. |
| Before row-op replay | Yes: marker and/or journal authority | Full marker-authorized post-commit state. |
| After fold watermark persist before tail cleanup | Yes: folded canonical state plus watermark | Folded records do not replay twice; tail cleanup is idempotent. |
| Before marker clear | Yes: marker remains authoritative | Recovery repeats safely and reaches full post-commit state. |
| After marker clear before index-ready marking | Yes: canonical data/index state has already passed integrity validation | Full post-commit state; guarded reentry must rebuild or mark volatile readiness before reads. |

The last phase is intentionally explicit. If the marker is gone, recovery can no
longer rely on marker authority; the durable state must already be sufficient to
reconstruct derived readiness and prevent reads from observing an unready index.

## Goal

Turn the highest-risk audit fallout into bounded reliability slices that improve
production confidence without rewriting the storage engine or derailing the
0.188 mutation candidate collector.

0.189 should:

- make supported feature/test configurations compile in CI;
- prove the marker/journal commit boundary with an IC-native host failpoint
  harness;
- harden or disprove the raw journal batch oversized-allocation risk;
- add deterministic marker/journal corruption coverage;
- harden docs/examples around volatile and non-atomic write lanes.

## Minimum Viable 0.189 Closeout

0.189 is intentionally smaller than the full audit backlog. The minimum viable
closeout is:

1. Supported no-default matrix is green or explicitly de-supported.
2. Host failpoint harness covers marker write, journal append, and marker clear.
3. Recovery oracle proves pre-marker versus marker-authorized post-commit state.
4. Raw journal batch oversized allocation risk is fixed or precisely proven not
   exploitable.
5. Deterministic corrupt/truncate/oversize tests exist for marker and journal.
6. Heap/non-atomic docs and examples are hardened.

Everything else belongs to 0.189.x or 0.190 unless it is small and directly
supports those six items.

These items must not be cut from 0.189:

| Priority | Item | Reason |
| --- | --- | --- |
| P0 | Feature matrix cleanup | The supported cfg matrix must be explicit and green before reliability work is credible. |
| P0 | Marker write, journal append, and marker clear failpoints | These are the smallest durable-authority interruption windows that prove the core commit boundary. |
| P0 | Recovery oracle assertions | Failpoint tests must prove the expected state, not only that recovery returns. |
| P0 | Raw journal batch allocation-bound proof or hardening | Oversized journal values must not bypass the configured allocation bound. |
| P1 | Deterministic marker/journal corrupt, truncate, and oversize tests | Marker and journal are the first persisted envelopes that recovery depends on. |
| P1 | Heap/non-atomic docs and examples | These are cheap, high-value data-loss footgun reductions. |

0.189 proves the minimal IC-native durability boundary:

- supported cfg matrix is explicit and green;
- marker/journal/marker-clear interruption has an oracle-driven host harness;
- oversized journal values cannot bypass the configured allocation bound;
- marker/journal corrupt bytes fail safely;
- obvious API data-loss footguns are documented.

## Non-Goals

- Do not implement SQLite-style transactions.
- Do not introduce multi-message transactions.
- Do not make pagination snapshot-isolated.
- Do not change the existing full-batch commit-window contract.
- Do not add POSIX file-system durability machinery.
- Do not change stable-memory layout compatibility as a side effect of tests.
- Do not implement broad persisted-format fuzzing in the first 0.189 closeout.
- Do not use checksums as a substitute for failpoint recovery tests or
  deterministic corruption tests.

## Work Track A: Feature Matrix Cleanup

Fix the no-default test failure first. It is bounded, not IC-specific, and
prevents future cfg drift from hiding real bugs.

Design decision:

- `cargo check --no-default-features` should always be supported for public
  crates.
- `cargo test -p icydb-core --no-default-features` should be supported unless
  the project explicitly declares that specific test helpers require
  SQL/diagnostics.
- Full workspace no-default tests do not need to include audit canisters that
  inherently assume SQL, blob, diagnostics, or generated audit features.
- If full no-default tests are not supported, CI must still run a no-default
  compile-only job and a minimal no-default smoke test that exercises non-SQL
  core storage behavior.

Likely work:

- Align `#[cfg(test)]` and `#[cfg(feature = "sql")]` gates for SQL-only test
  helpers and re-exports.
- Keep SQL audit canisters out of no-default workspace tests unless their
  required features are enabled.
- Avoid broad cfg aliases that accidentally pull SQL into core no-default tests.

Verification:

- no-default core test or minimal no-default core smoke test;
- public-crate no-default check jobs;
- all-features workspace tests;
- clippy over the supported feature matrix.

## Work Track B: Host Recovery Failpoint Harness

The 0.189 harness proves the smallest useful marker/journal matrix. Wider fold,
index rebuild, PocketIC, and instruction-budget cases move to 0.190.

Minimum 0.189 failpoint sites:

- marker write;
- marker-bound journal append;
- marker clear.

Each site needs failpoints immediately before and immediately after durable
authority changes where the code can distinguish those positions.

Failpoint modes are distinct:

- `ReturnError` exercises explicit error propagation.
- `PanicUnwind` exercises abrupt interruption through Rust control flow.
- `AbortLikeTrap` approximates IC-style termination where host-feasible.
- `InstructionBudgetExhaustion` belongs to later host/PocketIC budget work.

The minimal 0.189 harness must cover `ReturnError` and `PanicUnwind` for marker
write, journal append, and marker clear. Instruction-budget and PocketIC
coverage may land after the host harness proves the semantic matrix.

Test contract:

- run a mutation from an explicit pre-state;
- interrupt at a named failpoint and mode;
- construct a fresh guarded access path;
- run recovery;
- compare rows, indexes, marker state, journal visibility, and fold watermarks
  against the declared recovery oracle.

Deferred failpoint sites for 0.190:

- row-op replay;
- journal tail fold before watermark persist;
- after fold watermark persist before tail cleanup;
- live projection rebuild;
- secondary-index rebuild;
- journaled index materialized-view fold;
- repeated interruption during recovery itself.

## Work Track C: Raw Journal Batch Proof And Hardening

Raw journal batch hardening must first prove where oversized-value allocation
occurs. A decoder-side size check is insufficient if stable-structures
materializes the full value before IcyDB-owned validation.

Required proof task:

1. Determine whether allocation occurs before IcyDB-owned decode.
2. Determine whether allocation occurs inside IcyDB-owned decode.
3. Determine whether allocation occurs inside stable-structures value
   materialization.
4. Determine whether allocation occurs inside the underlying memory adapter.

If allocation occurs before IcyDB decode, the fix must change the stable value
shape, for example by using bounded chunks, a bounded envelope, or another
representation that prevents unbounded allocation before validation.

Requirements:

- A malformed journal-tail value must not require allocating the full malicious
  value before IcyDB can reject it by size.
- `MAX_JOURNAL_BATCH_BYTES` must be enforced before or at the first IcyDB-owned
  materialization boundary.
- The PR must include an inspection note or adversarial test proving the
  allocation boundary.

## Work Track D: Marker/Journal Corruption Corpus

0.189 covers deterministic marker and journal corruption tests. Broader fuzzing
belongs to 0.190.

Required marker/journal cases:

- empty bytes where a marker or journal batch is expected;
- truncated header;
- bad magic;
- unsupported version;
- length mismatch;
- unknown record tag;
- oversized encoded value;
- duplicate or conflicting journal batch sequence where applicable.

Required invariant:

- malformed bytes produce structured corruption or incompatible-format errors;
- malformed bytes do not panic;
- malformed bytes do not allocate unbounded memory;
- malformed bytes do not silently produce a valid but wrong logical state.

Persisted format change classification:

Any change to marker, journal, row, schema, or index persisted bytes must be
classified in the PR as one of:

- test-only harness change;
- internal pre-1.0 hard cut;
- backward-compatible reader extension;
- format-breaking migration.

The PR must state which category applies.

## Work Track E: API Footgun Docs And Examples

These docs are early, cheap, and do not need to wait for the failpoint harness.

Volatile heap stores:

- Review examples and docs for accidental `heap()` normalization.
- Ensure durable examples use `journaled` unless volatility is the point.
- Consider naming, docs, diagnostics, or feature gating that makes production
  heap usage explicit.

Non-atomic batch helpers:

- Keep `*_many_non_atomic` semantics explicit.
- Add examples that show prefix commits after a later failure.
- Ensure API docs route users to `*_many_atomic` when they need all-or-nothing
  behavior.

Transaction and pagination semantics:

- Do not redesign them in 0.189.
- Keep the docs clear that IcyDB has no transaction blocks, no rollback on
  returned `Err`, and no snapshot pagination.
- Add characterization tests only where public examples might imply otherwise.

## Work Track F: Multi-Registry Recovery Proof

This is P1 for 0.189.x unless it proves small enough to land during the minimum
closeout.

Do not move this into the broad 0.190 reliability bucket. If the test is small,
land it in 0.189. If it needs harness support, track it as 0.189.x. The purpose
is to prove or dismiss one precise process-global recovery-hint concern, not to
start the full fuzz/Miri/resource-budget program.

The suspected risk is not IC production concurrency. The suspected risk is:

> A process-global recovered flag or marker-presence hint could cause registry B
> to skip recovery because registry A recovered earlier.

Required test shape:

1. Construct registry A and registry B with independent virtual memory domains.
2. Put a pending commit marker only in registry B.
3. Trigger guarded access on registry A and complete recovery or no-op recovery.
4. Trigger guarded access on registry B.
5. Assert registry B does not skip marker recovery because of process-global
   state.
6. Repeat with registry A having a marker and registry B having no marker.
7. Repeat after clearing marker-presence hints.

Acceptance:

- recovery state is scoped by storage domain, or
- process-global fast paths are proven not to suppress marker checks for other
  registries.

## Work Track G: Resource Failure Classification

0.189 classifies resource failures so tests do not treat unlike failures as the
same thing. Deeper resource-budget characterization moves to 0.190.

Structured resource failures:

- anything the storage layer can report before durable mutation;
- anything a test storage adapter can inject as `Result::Err`.

Trap-like resource failures:

- stable-memory grow trap if not catchable;
- instruction-limit exhaustion;
- heap allocation OOM;
- canister-level termination behavior.

Scalability budget failures:

- recovery or index fold completes correctly but may exceed practical IC message
  limits.

Acceptance rules:

- structured resource failure before marker leaves no visible mutation;
- structured resource failure after marker leaves marker-authorized recovery
  authority;
- trap-like interruption after marker converges through guarded reentry;
- scalability budget failure produces measured limits or a follow-up streaming
  design.

## Slice Order

0.189 should land in this order:

1. Feature matrix cleanup.
2. Heap/non-atomic API footgun docs and examples.
3. Minimal failpoint harness around marker write, journal append, and marker
   clear.
4. Recovery oracle assertions for pre-marker and marker-authorized post-commit
   states.
5. Raw journal batch allocation-bound proof and hardening.
6. Deterministic corrupt/truncate/oversize marker and journal tests.
7. Multi-registry recovery proof if it remains small; otherwise 0.189.x.

0.190 owns the executable proof expansion:

- fold/index rebuild failpoints;
- repeated recovery interruption;
- structural decode fuzzing;
- Miri;
- PocketIC trap/instruction-budget scenarios;
- large-index recovery threshold or streaming design evidence.

0.191 owns the productization line informed by 0.190 evidence:

- backup/restore/import threat model;
- checksum decision;
- operator durability page.

## Acceptance Criteria

0.189 is complete when:

- supported no-default configurations compile and are covered by CI, or the
  unsupported configurations are explicitly removed from the support matrix;
- each failpoint test declares the durable phase reached and validates against
  an explicit pre-marker or marker-authorized post-commit oracle;
- host failpoint tests cover returned error and panic/unwind modes for marker
  write, journal append, and marker clear;
- raw journal batch hardening includes evidence that oversized values cannot be
  materialized before the configured bound is enforced;
- deterministic marker/journal corruption tests cover corrupt, truncated,
  oversized, incompatible, and unknown-tag cases;
- any persisted-format change is classified as test-only, pre-1.0 hard cut,
  backward-compatible extension, or format-breaking migration;
- API footgun docs/examples for heap stores and non-atomic batch helpers land
  independently of the later operator durability page.

## Validation Plan

Routine validation:

- format check;
- all-features workspace tests;
- supported no-default tests or no-default smoke tests;
- clippy on supported feature combinations;
- invariant checks.

Reliability validation:

- marker/journal/marker-clear failpoint matrix;
- recovery oracle comparison against explicit expected model;
- deterministic marker/journal corruption corpus;
- oversized raw journal batch allocation-bound test or inspection proof.

Optional 0.189.x validation:

- multi-registry recovery test;
- minimal model-based random operation test with guarded reopen/recover.

## Relationship To 0.188

0.188 remains the mutation candidate collector line. It should not absorb the
0.189 durability harness or persisted-format hardening work.

The overlap is limited:

- 0.188 must preserve current commit-window atomicity and failure semantics.
- 0.189 can later use the failpoint harness to validate write paths touched by
  0.188.
- Neither line should introduce chunked durable commits without a new atomicity
  design.

## Relationship To 0.190

0.190 is the follow-up reliability line. It should start from the 0.189 harness,
oracle vocabulary, persisted-format classification rule, and raw journal batch
allocation proof.

0.190 should not reopen the 0.189 minimum closeout unless the 0.189 work proves
one of its assumptions false.

## Exit Note

The audit's useful conclusion is not "IcyDB lacks SQLite mechanics." The useful
conclusion is that IcyDB has a plausible IC-native durability design that is not
yet adversarially proven. 0.189 should close the minimum proof gap with an
oracle-driven marker/journal failpoint harness, raw journal batch hardening, and
cheap API-footgun cleanup. Broader fuzzing, Miri, and resource-budget proof
belong to 0.190. Backup/restore, checksum, and operator-contract decisions
belong to 0.191.
