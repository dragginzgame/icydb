# 0.98 Parameterized Queries and Prepared Execution

## Thesis

`0.98` should introduce **parameterized queries** by separating **query structure** from **runtime values**, allowing the engine to compile one semantic query shape once and execute it many times with different bound arguments.

> **0.98 makes query values runtime inputs instead of structural plan identity, enabling true prepared execution, better cache reuse, and a cleaner separation between semantic query shape and execution-time data.**

This slice builds on the stable semantic and execution model established in `0.96` and the completed SQL result-pipeline work in `0.97`.

---

## Problem statement

After `0.97`, the engine has:

* unified semantic ownership in planner `Expr`
* compiled execution across production query paths
* stable SQL DISTINCT / ordering / paging behavior
* query-plan caching already present in the system

However, queries are still largely cached and compiled at the level of **fully literalized query text / intent**.

That means these queries are treated as different compiled shapes:

```sql
SELECT name FROM Character WHERE strength > 10;
SELECT name FROM Character WHERE strength > 20;
SELECT name FROM Character WHERE strength > 30;
```

Even though they differ only by runtime value.

This creates avoidable cost and fragmentation:

* structurally identical queries produce separate cache entries
* repeated value variation churns the query cache
* plan reuse is weaker than it should be
* the engine lacks a first-class prepared-query execution model
* future APIs that want “prepare once, execute many” do not have a clean foundation

The current system therefore has **query caching**, but not yet **true parameterized prepared execution**.

---

## Design goal

Introduce parameterization so that:

* query **structure** is compiled once
* query **values** are supplied at execution time
* compiled plans can be reused across value variation
* explain / planning / cache identity reflect query shape rather than bound literal values

The goal is:

> **one structural query shape, one compiled prepared plan, many executions with different bound values.**

---

## Non-goals

This slice does **not** aim to:

* add JOIN
* add subqueries
* add window functions
* redesign planner or executor architecture
* introduce a generic VM / bytecode engine
* support arbitrary dynamic SQL construction at runtime
* rework all cache infrastructure from scratch
* widen SQL semantics beyond parameter placeholders and bound execution
* change DISTINCT / ORDER BY / paging semantics

This is a **prepared execution** slice, not a broader SQL feature slice.

---

## Why now

`0.96` and `0.97` finished the hard prerequisites:

* semantic convergence onto planner `Expr`
* compiled execution across runtime paths
* stable projection / ordering / distinct pipeline
* clarified result-stream semantics

Parameterization now fits naturally because the engine already has:

* a semantic IR (`Expr`)
* compiled execution forms
* plan caching
* well-defined execution contexts

So `0.98` does **not** need to invent a new execution model. It only needs to extend the existing model so some values are **late-bound**.

---

# Part I — Current model

## Current query identity

Today, compiled plan reuse is primarily keyed on a query with literals already embedded into the semantic shape.

Conceptually:

```text
SELECT ... WHERE strength > 10  !=  SELECT ... WHERE strength > 20
```

for cache / plan purposes.

That is acceptable for literal execution, but it is not true prepared execution.

## Current execution model

The engine already supports:

```text
query text / fluent intent
  -> lower to semantic Expr / StructuralQuery
  -> validate
  -> compile to prepared execution plan
  -> execute
```

That core flow is correct and should remain.

## Missing capability

The missing piece is:

```text
query shape
  -> compile once
bound values
  -> supplied later at execution time
```

---

# Part II — Target model

## Before

```text
Query with literals
  -> lower
  -> compile
  -> cache by full literalized shape
  -> execute
```

## After

```text
Parameterized query shape
  -> lower to Expr with Param slots
  -> infer / validate parameter contracts
  -> compile once
  -> cache by structural parameterized shape
  -> execute many times with bound values
```

### Example

Before:

```sql
SELECT name FROM Character WHERE strength > 10;
SELECT name FROM Character WHERE strength > 20;
```

Two distinct compiled plans.

After:

```sql
SELECT name FROM Character WHERE strength > ?;
```

One compiled plan, executed with:

```text
[10]
[20]
[30]
```

---

# Part III — Core design

## 1. Introduce parameter placeholders in semantic expressions

Planner `Expr` needs a parameter slot form.

Example:

```rust
Expr::Param(ParamSlot { index: 0 })
```

or equivalent.

### Contract

A parameter expression represents:

* a runtime value supplied at execution time
* a fixed parameter position within the query shape
* a value whose type/compatibility is validated against its usage sites

### Explicit rule

> Parameters are semantic expression leaves, not post-planning text substitution.

---

## 2. SQL parser/lowering admits placeholders

SQL should admit a bounded placeholder surface, for example:

```sql
SELECT name FROM Character WHERE strength > ?;
SELECT name FROM Character WHERE name = ?;
SELECT name FROM Character WHERE strength BETWEEN ? AND ?;
SELECT COUNT(*) FROM Character HAVING COUNT(*) > ?;
```

The exact placeholder syntax can remain simple for this slice.

### Parser responsibility

The parser should:

* recognize placeholder tokens
* preserve placeholder position/order

### Lowering responsibility

Lowering should:

* convert placeholders into planner `Expr::Param(...)`
* assign stable parameter indexes
* feed those into validation / type inference

### Explicit rule

> Parameter placeholders lower directly into planner-owned parameter expressions and are never expanded by text replacement after parsing.

---

## 3. Value-sensitive planning boundary

The engine already contains predicate families where literal values influence
normalized intent and access routing, especially in prefix-sensitive text
families.

For `0.98`, parameterization is restricted to **value-insensitive planning
positions only**.

### v1 contract

A parameterized query is admissible only when every placeholder position:

* does not influence normalized query shape
* does not influence access route selection

If a placeholder appears in a value-sensitive planning position, prepare must
reject the query fail-closed.

### Allowed examples

```sql
WHERE strength > ?
WHERE name = ?
WHERE strength BETWEEN ? AND ?
HAVING COUNT(*) > ?
```

### Rejected examples

```sql
WHERE name LIKE ?
WHERE name ILIKE ?
WHERE name STARTS WITH ?
WHERE field IN (?)
```

### Rationale

This preserves the `0.98` v1 invariant:

* prepare once
* fix one route
* execute many times without reopening normalization or routing

It also keeps the current literal SQL caching story intact for value-sensitive
predicate families that are still literal-only in `0.98`.

### Follow-on

A later slice may widen prepared execution with explicit parameterized route
templates for value-sensitive predicate families, but that is out of scope for
`0.98`.

---

## 4. Parameter values are not part of structural plan identity

This is the heart of the slice.

Two queries with identical structure and placeholder positions share one plan:

```sql
SELECT name FROM Character WHERE strength > ?;
```

with `[10]` and `[20]` is one compiled plan, many executions.

### Cache key rule

The cache key must include:

* parameterized query structure
* schema fingerprint
* any existing route/significant execution-shape identity

The cache key must **not** include:

* specific bound parameter values

### Explicit rule

> Bound parameter values affect execution, not compiled plan identity.

---

## 5. Parameter contract inference

The planner must infer a parameter contract from usage.

Examples:

```sql
strength > ?              -> parameter must match the compare contract for strength
name = ?                  -> parameter must match the compare contract for name
strength BETWEEN ? AND ?  -> both parameters must match the same compare contract
COUNT(*) > ?              -> parameter must match the aggregate-compare contract
```

This does not require full global HM-style inference. It only requires the existing expression/type machinery to infer enough contract for admitted usages.

### Parameter contract should include

* parameter index
* expected runtime type family
* exact coercion rules, if any
* null-admission behavior
* usage role, such as compare-position, arithmetic-position, or aggregate-compare-position

This contract is frozen at prepare time and reused unchanged for every
execution of the prepared shape.

### Explicit rule

> Parameter validation happens against semantic usage, not against placeholder syntax alone.

---

## 6. Bound parameter validation at execution time

At execution time, the caller supplies a parameter vector.

The engine must validate:

* parameter count matches compiled contract
* parameter types are compatible
* any coercion rules are honored
* invalid bindings fail before execution

### NULL rule

`NULL` bindings are allowed in admitted parameter positions and follow ordinary
SQL null semantics during evaluation. They do not affect plan shape, route
selection, or compiled plan identity.

### Explicit rule

> A compiled parameterized plan must not execute with missing, extra, or type-incompatible parameter bindings.

---

## 7. Compiled execution forms must accept parameter values

The compiled runtime forms already exist:

* `PredicateProgram`
* scalar projection compiled forms
* grouped/global compiled forms
* order-related compiled seams where applicable

These do **not** need to be redesigned.

They need to be able to read bound values from a runtime parameter environment when evaluating parameter leaves.

### Runtime model

Execution receives:

* compiled plan
* parameter bindings
* normal execution context (rows, aggregates, etc.)

Compiled evaluators resolve `Expr::Param(index)` through the bound parameter environment.

### Explicit rule

> Parameter lookup is a runtime data read inside existing compiled evaluators, not a second semantic lowering pass.

---

## 8. Prepared query API model

Internally, `0.98` should formalize:

```text
prepare(parameterized query shape) -> prepared parameterized plan
execute(prepared plan, bindings) -> result
```

Even if the full public API evolves incrementally, this internal model should be explicit.

### Important separation

* prepare = parse/lower/validate/compile
* execute = bind/validate values and run

### Explicit rule

> Execution of a prepared parameterized plan must not trigger replanning due only to parameter value changes.

---

## 9. Literal queries remain supported

Existing literal SQL and fluent paths must continue to work.

A non-parameterized query is still valid:

```sql
SELECT name FROM Character WHERE strength > 10;
```

This may continue to compile as a literalized plan.

`0.98` adds parameterized plans; it does not require all queries to be rewritten.

### Optional internal normalization

The engine may later choose to normalize some literal queries into parameterized shape internally, but that is out of scope for `0.98`.

---

# Part IV — Explain changes

Yes — explain needs to change, but in a bounded and deliberate way.

## 1. Explain must show parameterized structure, not bound values, for prepared plans

If a query is parameterized, explain should reflect the parameterized query shape.

Examples:

```sql
SELECT name FROM Character WHERE strength > ?;
```

Explain should show a parameter slot / placeholder, not a specific runtime bound value.

This preserves the core invariant:

> explain describes the compiled structural plan

not one incidental execution.

---

## 2. Explain should surface parameter metadata where useful

For parameterized prepared queries, explain may include:

* parameter count
* parameter slot indexes
* inferred parameter kinds / compatibility hints

For example:

```text
params:
  $0: UInt-compatible
  $1: Text-compatible
```

The exact rendering can be modest, but explain should make parameterization visible.

---

## 3. Explain execution mode may include bound values only if explicitly execution-scoped

If you support an explain form that describes one concrete execution invocation, there are two possible views:

### Structural explain

* shows placeholders / param slots only

### Execution explain

* may optionally include bound values for that invocation if you choose

The important rule is consistency.

### Explicit rule

> Structural explain remains parameterized; any bound-value rendering must be clearly execution-scoped and must not change plan identity semantics.

---

## 4. Explain / JSON / execution front doors must stay aligned

As with DISTINCT, all explain entry points must obey the same parameterization contract:

* `EXPLAIN`
* `EXPLAIN JSON`
* `EXPLAIN EXECUTION`

No front door should silently de-parameterize or reinterpret the query differently.

---

# Part V — Planner and cache identity

## 1. Structural fingerprinting must treat parameters as stable slots

Fingerprints should distinguish:

```sql
strength > ?
strength = ?
? > strength
strength BETWEEN ? AND ?
```

because they are different shapes.

But fingerprints should *not* distinguish `[10]` from `[20]` for the same parameterized plan.

### Explicit rule

> Parameter slot position and usage are part of structural identity; bound values are not.

---

## 2. Cache identity must preserve semantic distinctions

These should remain distinct:

```sql
WHERE strength > ?
WHERE strength >= ?
WHERE strength > ? AND name = ?
WHERE strength > ? AND name = ?
```

same shape for different values, but distinct from:

```sql
WHERE strength > ? AND name = ? AND active = ?
```

### Explicit rule

> Parameterization reduces value variance, not semantic shape variance.

---

## 3. Schema fingerprint still gates reuse

A prepared parameterized plan must still be invalidated / missed when schema changes make the plan unsafe.

No change here.

---

# Part VI — Execution contracts

## 1. Parameter binding environment

Execution must carry a parameter binding environment accessible by compiled evaluators.

That environment should be:

* ordered by parameter slot
* validated before query execution begins
* immutable for the duration of execution

---

## 2. Predicate execution with parameters

WHERE / boolean evaluation must support parameters inside compiled predicate programs.

Examples:

```sql
WHERE strength > ?
WHERE name = ?
WHERE age BETWEEN ? AND ?
WHERE active = ?
```

No separate predicate model is needed.

---

## 3. Projection execution with parameters

Projection expressions must support parameters where admitted.

Examples:

```sql
SELECT strength + ?
SELECT COALESCE(name, ?)
```

assuming those functions/operators are otherwise supported.

---

## 4. Ordering with parameters

If the SQL surface admits parameterized ORDER BY expressions, those should be compiled and evaluated consistently through the normal compiled ordering seams.

If the admitted SQL surface does not yet allow meaningful parameterized ORDER BY, no widening is required in `0.98`.

---

## 5. Grouped / post-aggregate paths

If parameterized expressions are admitted inside grouped/global aggregate contexts, they must follow existing phase rules.

Examples:

```sql
SELECT COUNT(*) FROM Character HAVING COUNT(*) > ?;
```

This is a natural and useful admitted case.

No clause-specific evaluator should be introduced.

---

# Part VII — Validation and error behavior

## 1. Placeholder count errors

Reject at prepare or execute time as appropriate:

* too few bindings
* too many bindings

## 2. Type incompatibility errors

Reject executions such as:

```sql
WHERE strength > ?    with bound text
WHERE name = ?        with bound list
```

unless an existing coercion rule intentionally admits them.

## 3. Unsupported placement

If some SQL forms do not yet admit placeholders safely, reject them fail-closed rather than partially supporting them.

Value-sensitive planning positions must be rejected at prepare time rather than
quietly falling back to per-execution replanning.

### Explicit rule

> `0.98` should support parameterization only in admitted expression positions and fail closed elsewhere.

---

# Part VIII — Scope of initial admitted parameter surface

To keep `0.98` tractable, parameterization should begin with the highest-value expression positions.

## Initial admitted positions

* WHERE compare-family predicates in value-insensitive planning positions
* simple scalar projection expressions where parameter type/coercion contracts freeze cleanly
* HAVING compare-family expressions in grouped/global aggregate contexts where existing phase rules already admit them
* `BETWEEN ? AND ?` only where both slots freeze to the same compare contract as the surrounding scalar comparison

## Deferred positions if needed

* `LIKE`, `ILIKE`, `STARTS WITH`, and related prefix-sensitive text families
* membership families such as `IN (...)`
* schema / DDL contexts
* unusual metadata commands
* parameterized `ORDER BY` unless it is proven to stay on the existing scalar order seam with fixed compiled ordering semantics
* any place where placeholders would require special parser/control-flow treatment

### Explicit rule

> `0.98` should prioritize ordinary query expression positions, not every possible SQL statement form.

---

# Part IX — Fluent/query API impact

## 1. SQL surface is the primary entry point for `0.98`

That keeps the slice concrete and bounded.

## 2. Fluent can remain literal/value-driven initially

A future slice may add a fluent prepared-query builder model, but `0.98` does not need to redesign the fluent API around placeholders unless that is already natural.

### Explicit rule

> `0.98` introduces the prepared-parameter model without forcing a broad fluent API redesign.

# Part X — Testing requirements

## Functional tests

Add coverage for:

```sql
SELECT name FROM Character WHERE strength > ?;
SELECT name FROM Character WHERE name = ?;
SELECT COUNT(*) FROM Character HAVING COUNT(*) > ?;
```

with multiple bindings over the same prepared shape.

## Cache / reuse tests

Prove:

* same parameterized query shape reuses the same compiled plan
* different bound values do not create distinct compiled plans
* different query structures still do create distinct plans

## Error tests

Add coverage for:

* wrong parameter count
* wrong parameter type
* unsupported placeholder positions
* value-sensitive prepare-time rejections such as `LIKE ?`
* explain consistency on parameterized queries

## Explain tests

Add coverage for:

* `EXPLAIN` shows parameterized shape
* `EXPLAIN JSON` shows parameter slots/metadata consistently
* `EXPLAIN EXECUTION` behavior is explicit and consistent

# Part XII — Risks

## Risk 1 — parameter contract ambiguity

If parameter contracts are too loose, executions may fail late or inconsistently.

Mitigation:

* infer simple, explicit compatibility contracts from usage
* fail closed on ambiguous/unsupported cases

## Risk 2 — accidental value leakage into cache identity

If bound values sneak into cache keys, you lose most of the benefit.

Mitigation:

* make parameterized shape the only identity input
* test cache reuse explicitly

## Risk 3 — explain drift

Explain can easily show different things across front doors.

Mitigation:

* define structural vs execution-scoped explain clearly
* test all explain surfaces

## Risk 4 — over-widening scope

Trying to parameterize every SQL command form will bloat the slice.

Mitigation:

* focus on normal query expression positions
* reject unusual forms until explicitly designed

---

# Part XIII — Out of scope

The following stay out of `0.98`:

* JOIN
* subqueries
* window functions
* generic VM / bytecode engine
* full fluent prepared-query redesign
* automatic literal-to-parameter normalization for all existing queries
* broad optimizer redesign
* prepared DDL / metadata command parameterization unless trivial
* bind-time specialization or value-sensitive route-template planning

---

# Part XIV — Follow-on

After `0.98`, the engine should have:

* stable semantic expression ownership
* compiled runtime execution
* correct result-pipeline behavior
* true prepared execution across value variation

That enables later work in:

* higher-level prepared query APIs
* fluent prepared-query ergonomics
* batching / repeated execution optimization
* explicit prepared route templates for value-sensitive predicate families
* eventual larger SQL features without cache churn from value variance

---

## Thesis sentence

> **0.98 introduces parameterized queries for value-insensitive planning positions by separating query structure from runtime values, compiling semantic query shape once into a reusable prepared plan, and executing it repeatedly with validated bound arguments while keeping explain, cache identity, and runtime evaluation consistent with the existing compiled execution model.**
