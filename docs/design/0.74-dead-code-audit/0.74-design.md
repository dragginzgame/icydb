# 0.74 — Dead Code and Redundancy Audit

## 1. Overview

0.74 is a broad, fine-toothed cleanup line focused on eliminating **dead code, parallel abstractions, disagreeing logic paths, and stale compatibility seams** that accumulated across recent versions.

The purpose of 0.74 is not feature expansion. It is to make the codebase structurally harder to drift, easier to reason about, and cheaper to extend after the 0.73 grouped-planning work.

This line assumes that 0.73 finishes the grouped-planning ownership shift:

* planner-authored grouped semantics
* downstream projection from `GroupedPlanStrategy`
* explicit grouped fallback reasons

With that in place, 0.74 can treat disagreement and redundancy as a first-class engineering target rather than an incidental cleanup task.

---

## 2. Problem Statement

The codebase is now large enough that redundancy can accumulate in several forms:

* dead compatibility code left behind after migrations
* old helpers that still encode superseded policy
* multiple representations of the same concept
* explain/runtime/fingerprint projections that silently diverge
* executor-side re-derivations of planner-owned semantics
* stale tests that lock obsolete behavior
* comments and naming that preserve outdated ownership models

These problems do not always fail loudly. More often they create:

* hidden disagreement between layers
* increased change cost
* false confidence from duplicated tests
* hesitation to remove code because ownership is unclear

0.74 addresses this directly.

---

## 3. Primary Goal

Reduce the number of places where behavior can be represented, decided, or projected inconsistently.

In practical terms, 0.74 should:

* remove dead code
* collapse duplicate abstractions
* make ownership boundaries explicit
* delete stale compatibility seams
* align naming with actual ownership
* strengthen the rule that semantic decisions are authored once and projected elsewhere

---

## 4. Non-Goals

0.74 is not a feature line.

Out of scope:

* new SQL feature expansion
* broad planner redesign unrelated to redundancy removal
* cost-model work
* join-planning work
* semantic widening of grouped admission beyond what 0.73 lands
* cosmetic refactors with no ownership or redundancy payoff

---

## 5. Audit Thesis

The core audit principle is:

> Remove places where code can disagree.

This is broader and more useful than “remove unused code.”

The most dangerous redundancy is not merely unreferenced code. It is **live code that re-encodes the same meaning in multiple places**.

Examples:

* planner decides behavior, but executor re-checks the same policy
* explain reconstructs decisions instead of projecting from planner artifacts
* fingerprint shape is derived from a fallback default rather than the canonical strategy
* route capability reasons and planner fallback reasons partially overlap without a clear boundary

0.74 treats this as an architectural problem, not just a linter problem.

---

## 6. Core Ownership Rule

A central rule should guide deletion and retention decisions:

> Semantic decisions are authored once. Other layers consume, project, validate, or enforce capability.

For current architecture this means:

### Planner / semantics layer

Owns:

* strategy selection
* fallback selection
* semantic rejection reasons
* canonical behavior artifacts

### Executor / route layer

Owns:

* capability checks
* execution contracts
* runtime guardrails
* observability projection

### Explain / fingerprint / continuation layers

Own:

* projection from canonical artifacts
* no semantic re-derivation

Code that violates this boundary is a cleanup candidate.

---

## 7. Audit Categories

The audit should classify findings into explicit categories.

### 7.1 Dead Code

Code that is no longer referenced or reachable.

Examples:

* obsolete helper functions
* dead compatibility adapters
* unused enum variants
* stale modules left after migrations

Action:

* delete outright

### 7.2 Parallel Abstractions

Two or more public or semi-public representations of the same meaning.

Examples:

* canonical strategy object plus legacy hint object
* multiple enums for the same fallback reason family
* executor-local shadow strategy representations

Action:

* retain exactly one canonical abstraction
* demote or delete the others

### 7.3 Re-Derivation Paths

Code that recomputes semantics already authored upstream.

Examples:

* executor recomputing grouped ordered eligibility from raw facts
* explain inferring planner decisions instead of projecting them
* fingerprint defaulting into shape selection without planner reason

Action:

* delete or reduce to projection-only logic

### 7.4 Compatibility Seams

Old translation or wrapper layers that were needed during migration but now preserve outdated models.

Examples:

* legacy strategy projection helpers
* shim APIs kept beyond the migration window
* temporary translation code between old and new plan artifacts

Action:

* remove once no longer required

### 7.5 Naming Debt

Code whose names preserve outdated ownership assumptions even when behavior is now correct.

Examples:

* files or functions whose names imply executor ownership of planner semantics
* “eligibility” functions that now only project contracts

Action:

* rename after behavioral cleanup to prevent reintroduction of bad patterns

### 7.6 Stale Test Locks

Tests that encode old abstractions or duplicate behavior checks at the wrong layer.

Examples:

* tests validating a compatibility shim that should no longer exist
* route-layer tests asserting planner policy through downstream reconstruction

Action:

* delete obsolete tests
* relocate remaining assertions to the correct ownership layer

---

## 8. Audit Method

0.74 should not be an ad hoc hunt. It should be run as a structured sweep.

### 8.1 Phase 1 — Inventory

Build a list of candidate hotspots by searching for:

* `Hint`
* `Legacy`
* `Compat`
* `Deprecated`
* `Fallback`
* `Eligibility`
* `Strategy`
* `Reason`
* `grouped`
* `ordered`
* `streaming`
* `materialized`

Also inspect modules that historically carried transitions between planner, executor, explain, and fingerprint.

Output:

* file-by-file inventory of suspect abstractions and seams

### 8.2 Phase 2 — Ownership Classification

For each candidate, classify it as one of:

* canonical semantic source
* projection
* capability enforcement
* runtime mechanics
* compatibility shim
* dead code

Only the first four are expected to survive.

### 8.3 Phase 3 — Delete or Collapse

For each non-canonical artifact, answer:

* does it express unique behavior?
* does that behavior belong here?
* can the same behavior be obtained from a canonical artifact?

If no unique value remains:

* delete it

If value remains but ownership is wrong:

* move or collapse it into the canonical layer

### 8.4 Phase 4 — Boundary Renaming

After behavior is cleaned up, rename remaining files/functions/types whose names encode obsolete architecture.

This prevents future contributors from reintroducing old patterns.

### 8.5 Phase 5 — Test Realignment

Ensure tests assert behavior at the correct layer:

* planner tests for planner-authored semantics
* executor tests for capability enforcement and runtime execution
* explain tests for projection fidelity
* fingerprint tests for canonical projection stability

---

## 9. Deletion Heuristic

A simple rule should guide 0.74 decisions:

> If deleting this code would not change behavior because the canonical artifact already determines it, the code is redundant.

This is stronger than “is it referenced.”

Referenced code can still be redundant if it merely re-derives decisions that should already be fixed upstream.

---

## 10. Priority Targets

Initial likely targets include:

### 10.1 Grouped Planning Residue

After 0.73, inspect for:

* executor-side grouped eligibility derivation
* leftover grouped hint-era translations
* grouped fallback defaults outside `GroupedPlanStrategy`
* route observability code that recomputes grouped semantics

### 10.2 Route Reason Families

Ensure separation between:

* planner fallback reasons
* route capability reasons

Inspect for overlap, drift, or silent translation.

### 10.3 Explain Projection Paths

Audit for any explain logic that reconstructs instead of projecting.

### 10.4 Fingerprint / Continuation Projection

Audit for any fallback defaults or independently derived shape fields.

### 10.5 Old Migration Helpers

Look for stale adapter functions and temporary wrappers left behind from earlier lines.

---

## 11. Safety Constraints

0.74 should be aggressive, but not reckless.

Do not remove:

* runtime invariant checks that protect correctness
* capability enforcement that is distinct from semantic selection
* observability surfaces that project canonical information
* tests that still lock important invariants

The goal is not to minimize LOC blindly. The goal is to minimize **semantic duplication and disagreement risk**.

---

## 12. Slice Structure

0.74 should be landed as a sequence of small, auditable cleanup slices rather than one giant patch.

### 0.74.0 — Inventory and Classification

* produce audit inventory
* identify canonical vs redundant grouped surfaces
* lock deletion criteria

### 0.74.1 — Remove Parallel Grouped Abstractions

* delete residual grouped compatibility seams
* collapse duplicate grouped fallback representations

### 0.74.2 — Remove Executor Re-Derivation

* reduce grouped executor logic to capability enforcement and projection only
* delete semantic recomputation paths

### 0.74.3 — Explain / Fingerprint Projection Cleanup

* remove remaining reconstruction logic
* ensure projection is direct and canonical

### 0.74.4 — Naming and Boundary Cleanup

* rename misleading files/functions/types
* make ownership boundaries legible in code organization

### 0.74.5 — Broad Redundancy Sweep Beyond Grouped

* use the same method on adjacent subsystems where older duplication patterns may exist

---

## 13. Success Criteria

0.74 is successful if:

* canonical semantic sources are obvious
* no parallel public abstractions remain for the same meaning
* executor no longer re-derives planner semantics
* explain and fingerprint project directly from canonical artifacts
* stale compatibility shims are gone
* misleading ownership names are removed
* the number of places where behavior can disagree is materially reduced

---

## 14. One-Line Definition

> 0.74 removes dead code, parallel abstractions, and disagreeing derivation paths so that semantic behavior is authored once and projected consistently across the codebase.
