# IcyDB 0.177 - Explicit Schema Version Contracts

## Status

Active design.

0.177 defines what accepted entity schema identity means when schema shape can
change. It is the version-contract follow-on to 0.176, not an extension of the
0.176 hot-path catalog-consumption work.

0.176 answers where accepted schema authority lives and how query compile/plan
paths consume it cheaply. 0.177 answers what source-declared schema identity
means and when a changed generated proposal is allowed to become a candidate
accepted schema transition.

## Thesis

Schema version should be an intentional source/DDL contract, not a
storage-side auto-increment artifact.

Use:

```text
schema_version = human-declared contract
schema_fingerprint = machine-computed shape guard
```

The version tells IcyDB that the author intentionally changed the schema
contract. The fingerprint proves what changed. Neither one is enough alone:

- version without fingerprint is too weak to guard shape drift;
- fingerprint without declared version is too silent for source-controlled
  schema evolution;
- storage-side auto-increment hides authorship and makes diffs harder to
  audit.

## Why This Is 0.177

0.146 introduced persistent schema identity and layout authority.
0.147 moved runtime row-layout authority onto accepted schema snapshots.
0.152 moved schema mutation reasoning toward catalog-native mutation plans.
0.176 names accepted schema catalog authority and its cheap runtime identity.

Those lines still leave a policy question open: when generated source changes,
what proves the source author intended a new schema contract?

0.177 answers that with an explicit source-level `schema_version`.

## Non-Goals

0.177 must not add:

- public migration DDL;
- broad `ALTER TABLE` semantics;
- a migration execution framework;
- online row rewrites;
- per-row schema versions;
- auto-incremented schema versions;
- version gaps by default;
- generated-model fallback authority;
- diagnostics-derived schema authority;
- query-planner hot-path changes beyond consuming the identity defined here.

Schema mutation compatibility rules remain owned by the schema mutation layer.
0.177 only decides whether a changed generated proposal is eligible to become a
candidate transition.

## Identity Tuple

Accepted entity schema identity should be:

```text
entity_tag
entity_path
store_path
schema_version
fingerprint_method_version
schema_fingerprint
```

The minimum comparison tuple for one accepted entity is:

```text
entity_tag
schema_version
fingerprint_method_version
schema_fingerprint
```

`entity_path` and `store_path` should remain part of the broader catalog
identity because they protect namespace and storage binding. They may already
be implied by the existing model/store registration boundary, but diagnostics
and rejection messages should report them explicitly.

`entity_tag` is the durable entity identity. `entity_path` is the source and
namespace binding. `store_path` is the storage binding. 0.177 does not define
entity rename or store rebinding semantics. A changed `entity_path` or
`store_path` under an existing accepted `entity_tag` should be reported as an
identity mismatch unless a future explicit rename/rebind design admits it.

The fingerprint remains the machine guard used by cache, commit, cursor, and
planner invalidation. The declared version is the source contract that makes
shape changes intentional.

`fingerprint_method_version` is part of runtime schema identity, not merely a
diagnostic label. A raw fingerprint value without its method version is not
sufficient cache, commit, cursor, or planner invalidation identity. Callers
must either use a fingerprint type that embeds the method version or carry the
method version as an adjacent identity field everywhere fingerprints are
compared.

The accepted version, fingerprint-method version, and fingerprint must be
available as cheap accepted catalog identity/header facts. Query
compile/plan/cache paths must not decode the full accepted snapshot, construct
`SchemaInfo`, inspect generated models, or run schema reconciliation merely to
obtain `schema_version` or `schema_fingerprint`.

## Version Semantics

`schema_version` is declared in source or explicit mutation publication intent.

Rules:

- it starts at `1`;
- it is required for every generated entity;
- it is persisted in the accepted schema snapshot;
- it is compared during startup reconciliation and schema mutation admission;
- it is never inferred from storage contents;
- it is never auto-incremented by schema publication;
- it is not allocation identity;
- it is not a row-format version;
- it is not a per-row version.

The existing schema-owned `SchemaVersion` type can remain the internal carrier,
but 0.177 changes the contract of its source: generated proposals and explicit
mutation publication intents must provide the value.

## Fingerprint Semantics

`schema_fingerprint` is computed from the accepted persisted schema contract
that runtime planning, writing, decode, indexes, and relation policy consume.

`schema_version` is not an input to `schema_fingerprint`. The declared version
is compared alongside the fingerprint, not inside it. This is required so
schema admission can distinguish unchanged shape with a version bump, changed
shape without a version bump, and changed shape with an intentional version
bump.

The 0.177 canonicalization decision is:

```text
schema_fingerprint = accepted runtime shape and contract identity
accepted catalog identity = entity/store binding + declared version + method + fingerprint
```

It should include accepted shape facts that can change runtime behavior:

- `entity_tag` if treated as durable logical schema identity by the current
  fingerprint method;
- primary-key contract;
- durable field IDs;
- field names and row-layout slots;
- persisted field kinds, codecs, nullability, defaults, and write policies;
- accepted index contracts, including expression-index semantics;
- accepted relation contracts;
- schema metadata that changes write, read, or planning admissibility.

It should not include:

- `schema_version`;
- `store_path`;
- storage allocation identity;
- diagnostics formatting;
- storage diagnostics snapshots;
- generated model data that was not accepted;
- migration intent before publication;
- migration publication metadata that is not part of the accepted runtime
  shape.

`entity_path` is part of accepted catalog identity. If the existing accepted
runtime fingerprint method already hashes `entity_path`, 0.177 may preserve
that method under its current `fingerprint_method_version`. If 0.177 changes
the fingerprint method, the preferred split is to keep namespace binding in
accepted catalog identity and keep the runtime schema fingerprint focused on
shape/contract facts. In either case, a changed `entity_path` or `store_path`
under the same `entity_tag` is an identity mismatch, not an ordinary schema
shape transition.

The accepted runtime schema fingerprint must be computed from a canonical
accepted contract representation. Canonicalization must define:

- whether the current method preserves existing `entity_path` hashing or moves
  it exclusively to adjacent catalog identity;
- whether field order is meaningful or fields are sorted by durable field ID;
- whether index order is meaningful or indexes are sorted by stable index
  identity;
- how relation contracts are ordered;
- how defaults and write policies are encoded;
- how expression-index semantics are normalized;
- which fingerprint-method version protects future canonicalization changes.

Field names are part of the accepted schema contract because query planning and
source-level access resolve through accepted field names. A field rename is a
shape change unless a future explicit rename mechanism says otherwise.

The fingerprint may be represented by the existing commit/cache schema
fingerprint type if that remains the accepted runtime schema fingerprint. If
separate fingerprints are later introduced, cache/commit/cursor identity must
still use the runtime accepted-schema fingerprint, not a generated proposal
fingerprint.

Before implementation, identify the current accepted runtime schema fingerprint
function and prove whether `fingerprint_method_version` is embedded in the
fingerprint value or must be carried as a separate cache-key dimension. Do not
treat raw fingerprint bytes without their method version as complete schema
identity.

## Transition Rules

When comparing a stored accepted snapshot with the current generated proposal
or explicit mutation intent:

| Stored Version | Stored Method | Stored Fingerprint | Candidate Version | Candidate Method | Candidate Fingerprint | Result |
| --- | --- | --- | --- | --- | --- | --- |
| `N` | `M` | `F` | `N` | `M` | `F` | same schema |
| `N` | `M` | `F` | `N` | `M` | different | reject: shape changed without version bump |
| `N` | `M` | `F` | `> N` | `M` | `F` | reject by default: version bump without shape change |
| `N` | `M` | `F` | `N + 1` | `M` | different | valid candidate transition, subject to migration rules |
| `N` | `M` | `F` | `> N + 1` | `M` | different | reject by default: version gap |
| `N` | `M` | `F` | `< N` | `M` | any | reject: source version went backwards |
| any | `M` | any | any | different | any | reject or schema-owned hard-cut recompute; never a query/cache bridge |

The default policy is exactly `+1` for changed shape. Version gaps should be
rejected unless a later explicit governance decision introduces a narrow,
audited import or recovery mode.

The matrix assumes stored and candidate fingerprints use the same
`fingerprint_method_version`. A method-version mismatch is not an ordinary
schema transition. It must be handled by a schema-owned hard-cut/recompute path
or rejected with an explicit fingerprint-method mismatch diagnostic. Normal
query/cache paths must never bridge fingerprint-method versions.

Passing the version/method/fingerprint gate does not publish a schema. It only
permits the candidate to enter existing schema mutation compatibility
classification. Unsupported mutation classes still reject after a valid version
bump.

## Rejection Diagnostics

Rejections should tell the author exactly which contract failed.

Required diagnostic classes:

- fingerprint-method version mismatch;
- generated proposal changed but `schema_version` did not change;
- `schema_version` changed but accepted shape fingerprint did not change;
- `schema_version` jumped by more than one;
- `schema_version` moved backwards;
- entity identity changed under an existing accepted schema identity;
- stored accepted schema and generated proposal disagree after declaring the
  same version.

Diagnostics should include:

- entity name and entity path;
- entity tag when available;
- store path;
- stored schema version;
- candidate schema version;
- stored fingerprint-method version;
- candidate fingerprint-method version;
- stored fingerprint;
- candidate fingerprint;
- first human-readable shape difference when available.

Diagnostics do not become authority. They report the
version/method/fingerprint decision made by schema reconciliation or mutation
admission.

When multiple rejection facts overlap, diagnostics should report a stable
primary class and may include secondary facts. Precedence:

1. entity identity mismatch;
2. fingerprint-method mismatch;
3. rollback;
4. changed shape without version bump;
5. empty bump;
6. version gap;
7. valid `N + 1` changed-shape gate;
8. unsupported mutation after a valid version gate.

For example, candidate version `N + 2` with unchanged fingerprint is primarily
an empty bump because the accepted shape did not change; the diagnostic may
also report that the requested version skips `N + 1`.

## Generated Macro Surface

Generated entities should declare schema version at the entity attribute:

```rust
#[entity(
    name = "Character",
    schema_version = 4,
)]
```

Initial generated entities should declare:

```rust
schema_version = 1
```

Rules:

- no implicit default in generated code;
- no silent version choice;
- version `0` is invalid;
- omitted `schema_version` is a compile-time error once 0.177 is implemented;
- the derive output must carry the declared version into the compiled schema
  proposal;
- generated model helpers must not synthesize a version for runtime fallback.

If the implementation needs a transition patch for existing tests/entities,
that patch should add `schema_version = 1` explicitly everywhere rather than
teaching the macro or runtime to guess.

## DDL And Mutation Intent

Future catalog-native DDL or schema mutation APIs must carry an explicit target
schema version when they submit a changed candidate for accepted-snapshot
publication.

Rules:

- DDL does not auto-increment;
- mutation runners do not auto-increment;
- publication rejects changed shape without declared target version `N + 1`;
- publication rejects target version `N + 1` when the resulting fingerprint is
  unchanged, unless a later design admits metadata-only version stamps with a
  separate explicit reason;
- migration plans consume accepted snapshots and target versions, not generated
  model fallback authority.

Unpublished authoring or draft-edit commands do not have to carry a next schema
version if they cannot publish accepted schema changes. The explicit target
version is required at the publication/admission boundary that can turn the
candidate into a visible accepted snapshot.

Until metadata-only version stamps are explicitly designed, changes excluded
from the runtime schema fingerprint are not schema-versioned changes. Bumping
`schema_version` for such changes is rejected as an empty bump.

0.177 does not design public DDL syntax. It only reserves the requirement that
future DDL publication must be explicit about target schema version.

## Catalog And Cache Identity

0.176's accepted catalog identity should include the 0.177 tuple:

```text
entity_tag / entity_path / store_path
schema_version
fingerprint_method_version
schema_fingerprint
```

Cache keys may continue to key primarily by fingerprint for correctness, but
diagnostic and miss-attribution surfaces should retain the declared version and
fingerprint method version.

Cache identity must include the accepted runtime schema fingerprint together
with its `fingerprint_method_version`, either because the method version is
embedded in the fingerprint type or because it is an explicit adjacent cache-key
dimension. A raw fingerprint value without its method version is not sufficient
cache, commit, cursor, or planner invalidation identity.

Accepted version, fingerprint-method version, and fingerprint must be persisted
or otherwise available as cheap accepted catalog identity/header data. Query
compile/plan/cache paths must not decode the full accepted snapshot, construct
`SchemaInfo`, inspect generated models, or run reconciliation merely to obtain
cache identity.

Cache semantics:

- same method, same version, and same fingerprint can reuse plans if all other
  cache key dimensions match;
- same method and same version with different fingerprint must fail schema
  admission before query planning can observe it as a normal cache miss;
- same method and higher version with same fingerprint must fail admission by
  default;
- same method and version `N + 1` with changed fingerprint invalidates plans
  and can only plan after mutation compatibility accepts the transition;
- different fingerprint-method versions must not share cache entries and must
  not be bridged by query paths.

The planner should not decide version compatibility. It consumes accepted
catalog identity after schema-owned admission.

Candidate version/method/fingerprint comparison is an authority-producing
operation. It belongs to startup reconciliation, explicit schema mutation
admission, and tests that construct candidate proposals. Normal query
compile/plan/cache paths consume only already accepted identity.

## Persistence Rules

Accepted schema snapshots should persist the declared schema version, the
fingerprint-method version, the accepted runtime schema fingerprint, and the
accepted shape that produces that fingerprint. Schema store identity/header
lookup should expose the version, fingerprint-method version, and fingerprint
without requiring full accepted snapshot decode or planner projection
construction.

For pre-0.177 accepted snapshots that do not persist declared `schema_version`
or `fingerprint_method_version`, 0.177 must hard-cut the internal format.
Production code must not infer missing versions from storage order, generated
models, or `SchemaVersion::initial()`. Existing tests and fixtures should be
updated to persist explicit version `1`; obsolete internal snapshots should
return a clear obsolete-schema-format error or be handled only by an explicitly
named test/support upgrade path.

Schema store keys may continue to include schema version for ordered latest
lookup, but the store must not assign the next version. It only persists the
accepted version chosen by schema authority.

Latest lookup remains:

```text
entity -> latest accepted snapshot
```

It does not mean:

```text
entity -> storage chooses next schema version
```

Query cache-key construction should use the corresponding latest accepted
identity/header lookup, not full snapshot decode:

```text
entity -> latest accepted identity/header
```

## Compatibility With Existing Designs

### 0.176 Boundary

0.176 creates or consolidates the cheap accepted catalog context. 0.177
defines additional identity facts that context must carry.

Do not fold 0.177 implementation into the 0.176 hot-path work unless the patch
is only a type/field preparation that does not change version policy.

Implementation must use the completed 0.176 accepted catalog identity/header
path. Do not add query-path decode, `SchemaInfo` construction, generated-model
inspection, or schema reconciliation to obtain `schema_version`,
`fingerprint_method_version`, or `schema_fingerprint`. 0.177 policy belongs to
generated proposal construction, startup reconciliation, explicit schema
mutation admission, diagnostics, and tests.

### 0.152 Mutation Boundary

0.152 owns mutation classification and rebuild requirements. 0.177 owns the
version gate before a changed shape can be considered a candidate transition.

The order is:

```text
stored accepted identity
  + candidate declared identity
  -> version/method/fingerprint admission
  -> mutation compatibility classification
  -> publication or rejection
```

Passing the version/method/fingerprint admission gate is not publication. It only
allows the candidate to enter mutation compatibility classification. Existing
unsupported transition classes, rebuild requirements, validation failures, and
publication preflight checks still reject after a valid version bump.

### Pre-1.0 Hard Cut

Before `1.0.0`, internal schema formats should hard-cut to the latest contract.
Do not add compatibility fallbacks that infer missing versions from generated
models or storage keys.

Production code must not infer missing declared schema versions from old
snapshots, storage-key order, generated models, or `SchemaVersion::initial()`.
Tests and fixtures should be updated to write explicit `schema_version = 1`.
Old internal snapshots should fail clearly unless an explicitly named test or
support upgrade path owns the conversion.

## Work Areas

### 1. Add Declared Version To Generated Entity Metadata

Extend the entity macro/node/model pipeline so generated schema proposals carry
an explicit declared `schema_version`.

Expected shape:

- parse `schema_version = <positive integer>`;
- reject omission;
- reject zero;
- include the value in generated proposal construction;
- keep initial entities explicit with `schema_version = 1`.

### 2. Hash Candidate Accepted Shape Before Version Admission

Schema reconciliation needs both candidate version and candidate fingerprint.
It should compare stored accepted identity against candidate identity before
mutation compatibility rules decide whether a transition can publish.

Candidate fingerprint computation belongs only in authority-producing paths:
startup reconciliation, explicit schema mutation admission, and tests. Normal
query paths must not compute candidate fingerprints and must not inspect
generated `EntityModel` or `IndexModel` metadata to obtain
version/fingerprint-method/fingerprint identity.

### 3. Enforce The Version/Fingerprint Matrix

Implement the matrix in schema-owned reconciliation/admission code.

The default changed-shape path should require exactly `N + 1`.

If stored and candidate fingerprint-method versions differ, do not enter the
ordinary version/method/fingerprint matrix. Either recompute both sides into the
current method under a schema-owned hard-cut path or reject with an explicit
fingerprint-method mismatch diagnostic. Query/cache code must not perform this
bridge.

### 4. Improve Diagnostics

Update schema rejection diagnostics to distinguish:

- version unchanged with changed shape;
- version changed with unchanged shape;
- version gap;
- version rollback;
- fingerprint-method mismatch;
- ordinary unsupported mutation after a valid version bump.

### 5. Keep Storage Passive

Source-audit schema store publication and mutation code so storage does not
choose versions, increment versions, or repair missing versions.

### 6. Keep Query Paths Passive

Query compile/plan/cache paths consume accepted identity. They must not decide
whether a generated proposal should have bumped schema version.

They must also not decode the full accepted snapshot, build `SchemaInfo`, call
schema reconciliation, or inspect generated models merely to read accepted
version/fingerprint-method/fingerprint identity. Cache-key construction should
consume the accepted catalog identity/header prepared by 0.176.

## Recommended Implementation Sequence

### Slice 1: Identity/Header Alignment

Make accepted identity/header carry `schema_version`,
`fingerprint_method_version`, and `schema_fingerprint` in the final 0.177
shape.

Rules:

- no version policy yet;
- no macro rejection yet;
- no migration behavior yet;
- no query-path schema decode.

### Slice 2: Macro Surface And Explicit Declarations

Add `schema_version = N` parsing and generated proposal carrying. Patch
existing generated entities and test fixtures to declare `schema_version = 1`
explicitly, then reject omitted, zero, and non-integer declarations.

### Slice 3: Candidate Fingerprint Admission Inputs

Compute candidate accepted-shape fingerprint only inside schema-owned
authority-producing paths: startup reconciliation, explicit mutation admission,
and tests. Query/session cache paths must not compute candidate fingerprints.

### Slice 4: Transition Matrix Enforcement

Enforce the version/method/fingerprint matrix before mutation compatibility.
The valid changed-shape case reaches compatibility classification; it does not
publish by itself.

### Slice 5: Diagnostics And Hot-Path Regression Guards

Make rejection diagnostics actionable and prove 0.176 did not regress:
accepted identity load stays before `SchemaInfo`, warmed cache hits do not
decode accepted snapshots or inspect generated models, and normal query paths
do not compute candidate fingerprints.

## Source-Audit Candidates

Version synthesis should be suspicious outside tests:

```bash
rg -n "SchemaVersion::initial\\(|SchemaVersion::new\\(|schema_version" \
  crates/icydb-core/src/db/schema \
  crates/icydb-core/src/db/session \
  crates/icydb-schema-derive/src
```

Expected:

- generated proposal construction receives explicit source version;
- first-use bootstrap uses declared version `1`, not implicit
  `SchemaVersion::initial()`;
- tests may construct snapshots directly, but production reconciliation should
  not infer versions.

Storage auto-increment should not exist:

```bash
rg -n "next_schema|next_version|increment|saturating_add\\(1\\)|\\+ 1" \
  crates/icydb-core/src/db/schema
```

Expected:

- `+1` appears only in admission checks or tests, not in storage publication as
  an auto-assigned value.

Generated fallback should remain disallowed:

```bash
rg -n "compiled_schema_proposal_for_model|EntityModel|IndexModel|cached_for_generated_entity_model" \
  crates/icydb-core/src/db/session \
  crates/icydb-core/src/db/query \
  crates/icydb-core/src/db/executor
```

Expected:

- generated model use is proposal, compatibility proof, model-only test, or
  explicitly named bridge code;
- runtime accepted identity never falls back to generated metadata to pick a
  version.

Accepted identity hot-path scans should make these patterns suspect:

```bash
rg -n "SchemaInfo::from_accepted_snapshot|latest_persisted_snapshot|ensure_accepted_schema_snapshot|compiled_schema_proposal_for_model|StoreDiagnostics" \
  crates/icydb-core/src/db/session \
  crates/icydb-core/src/db/query \
  crates/icydb-core/src/db/executor
```

Expected:

- accepted version/fingerprint-method/fingerprint identity loading does not
  build `SchemaInfo`;
- warmed cache keys do not decode full accepted snapshots for identity;
- normal query paths do not compute candidate fingerprints;
- normal query paths do not inspect generated models or diagnostics to obtain
  accepted version/fingerprint-method/fingerprint identity.

## Proof Targets

0.177 should prove:

- every generated entity declares `schema_version`;
- omitted schema version fails at macro/compile time;
- initial accepted schema persists declared version `1`;
- same fingerprint-method version, same schema version, and same fingerprint
  admits as same schema;
- same fingerprint-method version and same schema version with changed
  fingerprint rejects with a clear no-version-bump diagnostic;
- same accepted shape with different declared schema versions produces the same
  `schema_fingerprint`;
- bumped version and unchanged fingerprint rejects by default;
- empty bump rejection proves `schema_version` is not hashed into
  `schema_fingerprint`;
- changed fingerprint with version gap rejects by default;
- changed fingerprint with exactly `+1` reaches mutation compatibility checks;
- fingerprint-method mismatch rejects or enters an explicitly schema-owned
  hard-cut/recompute path before the ordinary transition matrix;
- changed shape with no version bump rejects before planner/cache code is
  invoked;
- schema store publication does not auto-increment;
- accepted identity load does not build `SchemaInfo`;
- accepted identity load does not decode the full accepted snapshot when an
  identity/header path exists;
- warmed cache hits consume version/fingerprint-method/fingerprint header data
  without generated model inspection, schema reconciliation, diagnostics DTO
  construction, or `SchemaInfo` construction;
- query/cache keys compare accepted fingerprints together with
  `fingerprint_method_version`;
- normal query paths do not compute candidate fingerprints;
- query cache identity consumes accepted
  version/fingerprint-method/fingerprint after schema admission and does not
  own version policy.

## Closeout Standard

0.177 can close when:

- generated entity declarations require explicit positive `schema_version`;
- accepted schema snapshots persist the declared version;
- accepted version/fingerprint-method/fingerprint identity is loadable as cheap
  catalog identity/header data before `SchemaInfo`;
- `schema_version` is proven not to be an input to `schema_fingerprint`;
- startup reconciliation and mutation admission enforce the
  version/method/fingerprint matrix;
- fingerprint-method mismatch has an explicit schema-owned hard-cut/recompute
  or rejection policy and is never bridged by query/cache paths;
- storage remains passive and never chooses schema versions;
- candidate fingerprint comparison is isolated to reconciliation/mutation
  admission and tests;
- diagnostics distinguish method mismatch, missing bump, empty bump, gap,
  rollback, and unsupported transition cases;
- cache/catalog diagnostics expose accepted schema version,
  fingerprint-method version, and fingerprint;
- no generated-model fallback can synthesize runtime schema identity.

## Deferred Work

These remain outside 0.177:

- public migration DDL syntax;
- version-gap import tooling;
- metadata-only version stamps;
- multi-version row decode;
- online schema migration execution;
- automatic source rewrite tooling for bumping versions;
- semantic versioning of schemas beyond monotonic integer contracts.
