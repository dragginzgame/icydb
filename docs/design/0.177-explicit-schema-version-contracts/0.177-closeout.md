# 0.177 Explicit Schema Version Contracts Closeout

## Status

Closeout-ready as of 2026-06-03.

0.177 closes the explicit schema-version contract for generated schema
proposals, accepted snapshots, schema admission, cache identity, storage
publication, and diagnostics. It does not ship public migration DDL,
version-gap import tooling, metadata-only version stamps, multi-version row
decode, online migration execution, automatic source rewrite tooling, or
semantic schema versions beyond monotonic integers. Those remain deferred to
0.178 or later designs.

The release-engineering follow-through in the later 0.177 patches tightened
CI and local developer setup. That work is intentionally outside the runtime
schema-version contract.

## Completed Surface

- Generated entities must declare an explicit positive `schema_version`.
- Accepted schema snapshots persist the declared version and expose
  method-qualified fingerprint identity.
- Schema admission compares stored and candidate version, fingerprint-method
  version, and accepted-shape fingerprint before mutation compatibility.
- Admission fingerprints ignore `schema_version` while still tracking accepted
  shape, index, relation, layout, and write-contract changes.
- Storage publication remains passive: it persists the snapshot-declared
  version and does not assign the next version.
- Query, cache, session, and executor runtime paths consume accepted identity
  after schema-owned admission; they do not compute candidate fingerprints or
  read generated declared versions for runtime identity.
- Diagnostics expose version, fingerprint-method version, and fingerprint
  facts without becoming authority.

## Source Audit Notes

The closeout audit re-ran the source scans named in the 0.177 design. The
remaining `SchemaVersion::initial()` and version-construction hits are confined
to tests, explicitly constructed accepted snapshots, range sentinels, schema
admission comparison, or helper paths that do not infer missing persisted
versions. The remaining `+ 1` and `saturating_add(1)` hits in schema code are
admission checks, field/index count comparisons, counters, or tests; no schema
store publication path assigns a next version.

Generated `EntityModel` and `IndexModel` uses remain in proposal,
model-only/test convenience, or existing planner/executor model lanes. Runtime
schema-version identity remains accepted-snapshot owned after an accepted
snapshot exists.

## Deferred Boundary

0.178 should start from this closed 0.177 boundary:

- public DDL publication must carry explicit target schema version intent;
- migration admission consumes the 0.177 version/method/fingerprint gate before
  mutation compatibility;
- version gaps, metadata-only stamps, and online row migration remain
  unsupported until separately designed;
- query/cache paths continue to consume accepted identity after admission and
  must not bridge fingerprint-method versions.
