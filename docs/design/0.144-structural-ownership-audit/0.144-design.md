# 0.144 Structural Ownership Audit

## Status

Proposed

## Motivation

Recent work has converged several high-traffic systems around clearer execution
and semantic ownership:

```text
query construction
  -> prepared planning
  -> terminal-specific execution adapters
```

The next risk is not raw performance. The next risk is structural drift:

* large files accumulating unrelated responsibilities
* helpers living in caller modules instead of owner modules
* orchestration layers owning semantic decisions
* deep imports turning internal submodules into accidental APIs
* compatibility shims surviving after their callers have converged

Optimizing on top of that shape would make the codebase harder to move later.
`0.144` is therefore a structure-first release.

## Goal

Perform a full module-by-module audit of the codebase and make ownership
boundaries explicit before starting another optimization cycle.

The goal is not to make the code smaller at all costs. The goal is:

* each subsystem has one clear owner
* each module exports only the surface it intends to support
* code lives where the domain decision belongs
* stale shims and compatibility paths are removed
* large files are split only when responsibilities are genuinely mixed
* small files are merged only when they add indirection without ownership value

## Non-Goals

Do not:

* optimize runtime performance
* change query behavior
* change storage or cursor formats
* redesign public APIs for their own sake
* keep compatibility shims for old internal paths
* introduce broad abstraction layers
* rename modules without a clear ownership improvement

## Scope

Primary scope:

* `crates/icydb-core/src/db`
* `crates/icydb-core/src/value`
* `crates/icydb-schema`
* `crates/icydb-schema-derive`
* `crates/icydb-build`
* test and audit harness modules that expose stale production structure

Secondary scope:

* root crate re-exports
* changelog and design docs affected by moved ownership boundaries
* scripts only if they encode stale module assumptions

## Audit Method

Each module is reviewed with the same questions:

1. What domain responsibility does this module own?
2. What should callers be allowed to import?
3. Which submodules are implementation details?
4. Are helpers located in the owner module or in a caller module?
5. Does orchestration code contain semantic, storage, or planner decisions?
6. Does the module expose test-only or obsolete compatibility surfaces?
7. Are there files that should be split, merged, or moved?
8. Are imports crossing deep boundaries that should be re-exported instead?
9. Can one core concept be traced end-to-end without crossing unclear
   ownership boundaries?

The trace question is the dynamic check. For example:

```text
SQL
  -> canonical expression
  -> plan
  -> descriptor
  -> executor
  -> result
```

If that path crosses modules that each reinterpret the same concept, the
boundary is not sealed yet.

## Audit Order

Run the audit from upstream ownership toward downstream implementation:

1. `db/query`
2. `db/sql`
3. `db/session`
4. `db/executor`
5. `db/data`
6. `db/index`
7. `value`
8. schema crates
9. build/test harnesses
10. docs/scripts

This keeps planner and lowering decisions stable before touching session,
executor, value, and schema boundaries that depend on those upstream choices.

## Subsystem Interpretation

The same audit framework applies across the codebase, but ownership must be
judged relative to the subsystem's role.

### Execution Path

Examples:

* `db/query`
* `db/sql`
* `db/session`
* `db/executor`

Focus:

* ownership versus orchestration
* semantic leakage
* polymorphic or branch-heavy execution dispatch
* boundary clarity between planning, session, and executor layers

Execution modules answer:

```text
Who owns this behavior or semantic decision?
```

### Data / Value / Storage Path

Examples:

* `value`
* `db/data`
* persisted row codecs
* storage-key and structural-field helpers

Focus:

* encoding and decoding logic leaking outward
* format assumptions in callers
* duplicated conversion logic
* schema versus runtime representation mismatch

Data modules answer:

```text
Who owns the representation of this data?
```

For example, decimal normalization belongs in the value/data layer, not in
query execution.

### Schema / Build / Derive Tooling

Examples:

* `icydb-schema`
* `icydb-schema-derive`
* `icydb-build`

Focus:

* codegen leaking runtime internals
* derive macros encoding runtime assumptions
* build helpers duplicating logic from runtime or schema validation

Tooling modules answer:

```text
Is this generating structure, or re-implementing logic that belongs elsewhere?
```

### Priority

High priority:

* `db/query`
* `db/sql`
* `db/session`
* `db/executor`
* `value`

Medium priority:

* `db/data`
* `db/index`
* schema crates

Lower priority:

* build scripts
* test harnesses
* docs/scripts, unless they encode stale production structure

## Decision Rule

Each audit must end in a decision, not just an observation:

* if ownership is ambiguous, choose one owner and move the code there
* if a file contains multiple owners, split along the ownership boundary
* if a helper lives in a caller module, move it to the domain owner
* if a module has no clear owner, defer it explicitly with the reason
* if a suspected issue is only stylistic, leave it alone

The status tracker records the owner, the supported import surface, internal
submodules, the issue found, and the action taken.

## Structural Rules

### Ownership vs Orchestration

Owner modules define semantics, invariants, and data meaning.

Orchestration modules compose behavior. They may wire owners together, but they
must not define meaning.

Orchestration must not:

* interpret persisted data formats
* define semantic equivalence
* perform domain validation
* own planner or executor invariants

Orchestration should:

* delegate to owner modules
* keep control flow explicit
* expose only the boundary needed by callers

### Ownership First

Move code toward the module that owns the concept.

Examples:

* query meaning belongs in query planning or SQL lowering, not execution wiring
* storage format logic belongs in data/persistence modules, not callers
* terminal orchestration delegates to session/executor boundaries
* test fixtures stay under test owners unless they are production APIs

### Module Roots Define Surfaces

External callers should import from module roots, not deep implementation paths.
If a nested item is part of a module's supported surface, the module root should
re-export it intentionally.

Public and `pub(crate)` surfaces must be intentional. During an audit, run a
visibility scan for the module and narrow anything that exists only because an
old caller needed it.

Example visibility check:

```bash
rg "pub " crates/icydb-core/src/db
```

### Layering Rule

Higher-level modules must not depend on:

* executor internals
* storage layout details
* codec implementations

Lower-level modules must not depend on:

* planner semantics
* query construction
* session orchestration

If a layer needs information from another layer, pass it through an explicit
boundary type owned by the correct layer.

### Stability Rule

Do not move code unless:

* ownership is clearly wrong
* the target module is clearly the correct owner
* the move reduces coupling or removes an unclear boundary

Avoid:

* speculative reorganizations
* renaming without ownership gain
* moving code only for symmetry
* splitting files only because they are long
* merging files only because they are short

### Split Only Mixed Responsibility

Large files are not automatically wrong. Split them when they contain multiple
owners, such as:

* syntax plus semantics plus execution mapping
* storage codec plus runtime traversal plus tests that belong elsewhere
* planner state plus executor adapters

### Merge Fake Modules

Small modules are not automatically wrong. Merge a module only when it:

* does not define a domain boundary
* does not hide meaningful complexity
* forces callers to import deeper paths without reducing coupling
* creates a naming layer without clarifying ownership

### Delete Stale Paths

Backwards compatibility for superseded internal paths is not a goal.

Once a canonical path exists and all internal callers have migrated:

* remove the old path immediately
* do not keep aliases, re-exports, or compatibility wrappers
* move tests to the canonical path too
* delete temporary bridges instead of documenting them as permanent structure

## Target Outcomes

By the end of `0.144`:

* major subsystem boundaries are documented in the status tracker
* obvious misplaced code has been moved
* overloaded modules have been split where the owner boundary is clear
* fake modules and low-value shims have been deleted
* public and `pub(crate)` visibility has been narrowed where possible
* deep imports from outside module boundaries have been reduced
* optimization candidates are listed, but not implemented

## Validation

Run validation appropriate to each structural slice:

```bash
cargo fmt --all
cargo check -p icydb-core --tests --features sql
cargo clippy -p icydb-core --all-targets --features sql -- -D warnings
git diff --check
```

For broader moves, also run:

```bash
cargo test -p icydb-core --features sql -- --nocapture
cargo test --workspace
```

PocketIC-backed tests should be run outside the sandbox when a slice touches
integration harness structure.

## Deliverables

* module-by-module status in `0.144-status.md`
* for each audited module: owner, allowed imports, internal-only modules,
  issues found, and action taken
* small structural patches grouped by ownership boundary
* deferred follow-up list for risky or optimization-oriented work
* changelog entries only for changes that affect maintainers or users

## Stop Condition

Stop refactoring when:

* module ownership is clear enough for a new contributor to follow
* no remaining change would clearly reduce cross-module coupling
* remaining issues are either documented or intentionally deferred
* the next changes would be optimization, not structural ownership cleanup
