# 0.206 Redo-Only Schema Index Publication

Status: proposed; design cut on 2026-07-15; implementation starts after 0.205 closeout

## Context

IcyDB currently publishes accepted schema through a durable commit marker and
rebuilds secondary indexes from authoritative rows during recovery. Index DDL,
however, still contains operation-specific rollback machinery:

- field-path index addition stages store mutations and retains reverse actions;
- expression-index addition records inserted keys and removes them on rejection;
- index drop snapshots original keys and values, rewrites compacted ordinals, and
  restores the before-image if publication rejects;
- recovery snapshots the complete index store before clear-and-rebuild, then
  restores that snapshot if rebuilding fails.

Those paths are locally defensible, but together they create two competing
transaction models:

1. the accepted-schema marker owns durable publication and recovery; and
2. individual index operations own physical undo.

The second model is unnecessary for derived index state. Accepted schema
snapshots are runtime authority, rows are authoritative data, and secondary
indexes can be deterministically reconstructed from those two inputs. Retaining
per-key undo therefore adds state, failure branches, proof obligations, and
tests without adding an independent source of truth.

This design makes schema/index publication redo-only. All fallible index
derivation and validation happens before apply. On recovery-capable durable
stores, no physical index mutation happens before the durable marker. Once the
marker exists, the operation has one legal direction: complete the
accepted-after schema and rebuild its derived index projection. Non-journaled
heap stores use the equivalent prepared, infallible apply discipline described
below.

This is a pre-1.0 hard cut. The existing index rollback representations are
deleted when their last index-DDL consumer is removed. No compatibility shim,
dual path, or legacy preservation test remains.

## Problem Statement

Current index DDL is difficult to prove because a physical mutation can happen
before accepted-schema publication is durably owned. A later publication
rejection then needs exact reversal of every physical effect:

- inserted index keys must be removed;
- overwritten values must be restored;
- ordinal-compacted keys must be moved back;
- collisions and partial progress must be distinguished;
- an interrupted rollback must itself be recoverable;
- schema identity must be checked so stale rollback cannot overwrite newer
  accepted state.

The implementation consequently needs operation-specific before-images and a
classifier that decides whether undo remains safe. Recovery repeats the pattern
at store scale by snapshotting the full index store before rebuild.

This is the wrong cost model for derived data. The durable marker already
provides a publication boundary, and recovery already has enough authority to
recompute the accepted-after indexes. The missing rule is simply that physical
index state must not be touched until the marker owns the operation.

## Design Decision

Recovery-capable durable index DDL becomes marker-first and redo-only:

> Before marker persistence, accepted schema and physical indexes remain at
> accepted-before. While the marker is present, recovery completes
> accepted-after. After verified marker clear, accepted schema and physical
> indexes are both accepted-after. No index key is rolled back across that
> boundary.

Before writing the marker, the operation constructs and validates one complete
raw projection of the affected entity's user-index domain under the
accepted-after schema. The stage contains both the exact raw keys currently
occupying that domain and the final raw entries that must replace them. After
writing the marker, apply consumes only those staged raw values. If apply is
interrupted, recovery clears and deterministically rebuilds derived indexes from
authoritative rows and the marker-owned accepted schema.

The affected user-index domain is deliberately broader than the target index:

- CREATE INDEX produces the final projection for every accepted-after index on
  that entity;
- DROP INDEX naturally produces compacted accepted-after ordinals without
  snapshotting or remapping before-images;
- field-path and expression indexes use the same physical apply shape;
- the IndexKeyKind::System reverse-relation family remains untouched;
- entries for other entity tags in the same index store remain untouched.

This design chooses bounded replacement of one user-index domain instead of
shadow generations or a complete store rewrite during normal apply. It reuses
the existing accepted-schema candidate and marker format and avoids a new
persisted index-generation protocol. If measurement proves that domain
replacement is not viable, implementation stops for a new design decision. It
does not silently restore per-key undo.

The staged deletion keys are a forward apply set, not a before-image. They carry
no prior values or prior readiness state, are never retained after apply, and
cannot restore the accepted-before projection. If deletion is interrupted,
recovery reconstructs accepted-after from rows.

## Alternatives Considered

### Keep Operation-Specific Rollback

Rejected. It retains two transaction authorities, requires exact before-images,
and makes a rollback interruption a second recovery problem. Derived index
state does not justify that proof surface.

### Mutate Only The New Or Removed Index

Rejected as the common physical shape. It leaves CREATE and DROP with different
ordinal, collision, and interruption rules. Replacing the complete user-index
domain gives both operations one accepted-after representation while preserving
unrelated key families.

### Clear And Rebuild The Complete Index Store During Normal Apply

Rejected for normal publication. A backing store can contain multiple entity
tags and IndexKeyKind::System reverse-relation entries. Clearing it would
broaden DDL work and couple one index change to unrelated derived domains.
Complete-store clear remains appropriate for global recovery, where every
derived domain is rebuilt under the accepted catalog.

### Shadow Index Generations

Deferred. Shadow generations can provide a single selector flip, but they
require a persisted generation identity, cleanup protocol, query lookup change,
and cursor/diagnostic review. Those costs are not justified until bounded
domain replacement is measured and shown insufficient.

### Persist Staged Index Entries In The Marker

Rejected. It duplicates reconstructable data, pressures the bounded marker and
journal codecs, and creates a second persisted index payload. Accepted-after
schema plus authoritative rows is already the recovery recipe.

## Goals

- Make all expected index-DDL rejection happen before durable or physical
  mutation.
- Make physical index apply a mechanical replacement of a prevalidated raw
  projection.
- Give every post-marker interruption one recovery direction: forward.
- Remove operation-specific before-images and rollback plans from field-path
  create, expression create, and index drop.
- Remove full index-store snapshot restoration from recovery.
- Preserve accepted schema as the sole runtime schema authority.
- Preserve rows as authoritative data and indexes as reconstructable derived
  state.
- Preserve SQL behavior, generated API shape, and query semantics.
- Preserve all current persisted formats.
- Fail closed whenever index readiness or marker recovery is incomplete.
- Reduce conceptual owners and recovery branches even if precommit staging does
  more work.

## Non-Goals

0.206 does not:

- redesign row-layout mutations such as DROP COLUMN;
- change entity, relation, field, row, index-key, index-value, journal, marker,
  accepted-root, allocation, or cursor formats;
- add shadow index generations or a persisted generation selector;
- add online or resumable multi-message index builds;
- yield, await, or spill staging across messages;
- change query planning or the query hot path;
- add a new public SQL statement, option, error code, or configuration setting;
- make direct unguarded store access part of the atomicity contract;
- put staged index entries into the commit marker;
- introduce publication sinks, runtime schema epochs, capability traits, or
  alternate physical plans;
- retain a legacy rollback path as a fallback;
- delete rollback support still required by out-of-scope row mutations.

Metadata-only schema changes that do not alter raw index keys continue to use
normal accepted-schema publication. They do not stage or replace an entity
projection merely because an index definition is mentioned.

## Authority Model

The authority order is:

1. a durable commit marker owns an in-progress accepted-schema candidate;
2. after recovery, the accepted schema snapshot owns runtime schema meaning;
3. authoritative rows own stored record values;
4. the physical index store is a derived materialized projection;
5. generated EntityModel and IndexModel values remain proposal,
   reconciliation, model-only convenience, and test inputs.

No generated model reconstructs runtime authority. No physical index snapshot
can override the marker or accepted schema.

### Publication Identity

Precommit captures the exact accepted-before identity used to derive the
candidate. Immediately before marker persistence it rechecks:

- accepted-before identity is unchanged;
- the staged entity identity still resolves to the same authoritative entity;
- the candidate fingerprint is unchanged.

Row scan, projection staging, final schema recheck, and marker persistence
execute in one non-awaiting message. That serialized execution is the row
snapshot guarantee; 0.206 does not add a second row fingerprint or persisted row
revision merely to restate it. Introducing an await, yield, or multi-message
stage invalidates this design and requires a new snapshot-identity decision.

### Marker Authority

Before the marker, the accepted-before schema and physical projection remain
untouched. A validation error returns normally.

After the marker, an error is not an ordinary publication rejection and cannot
select accepted-before. It represents interruption or recovery-required state.
The marker remains authoritative until recovery has completed the accepted-after
schema and derived projection and has safely cleared the marker.

An injected ReturnError after the marker tests interruption semantics. It does
not authorize rollback and must not be treated as a normal DDL rejection.

## Atomicity Invariant

The maintained invariant is:

> Every fallible input-dependent decision occurs before marker persistence.
> Every post-marker physical step consumes prevalidated raw data or performs a
> deterministic recovery rebuild. Guarded entrypoints observe neither a partial
> schema nor a partial index projection.

Normal post-marker apply must contain no:

- row or schema decoding;
- expression parsing or evaluation;
- predicate evaluation;
- field-path traversal;
- uniqueness decision;
- key derivation;
- ordinal planning;
- allocation decision;
- collision interpretation;
- candidate validation;
- operation-specific rollback classification.

The production apply call graph must be infallible. Store return values cannot
drive a recoverable post-boundary decision, and helpers that currently return a
Result for validation must be split into fallible precommit plus mechanical
apply. Test-only failpoints may interrupt post-marker execution to prove
recovery, but they do not model a permitted production rejection path.

## Target Publication Flow

### Phase 1: Resolve Accepted-Before

The operation resolves the currently accepted snapshot and checks the expected
schema identity. It also requires the backing IndexStore to be Ready before
reading its physical before projection. It derives the accepted-after candidate
catalog-natively.

SQL DDL is a frontend into this phase, not the source of mutation semantics.
Startup reconciliation lowers to the same candidate and staging owner.

No durable mutation occurs.

### Phase 2: Stage The User-Index Domain Replacement

One concrete staging owner scans authoritative rows and constructs both the
expected accepted-before and final accepted-after raw entries for every user
index on the affected entity. It also scans the current physical domain,
defined exactly by entity tag plus IndexKeyKind::User. The observed domain must
match the derived accepted-before projection byte-for-byte before its keys can
become the deletion set.

Before and after projections should be derived in one row traversal where
possible. This is one staging operation, not two independently observable
passes.

Conceptually:

~~~text
StagedUserIndexDomainReplacement
  store path and entity identity
  accepted-before identity
  accepted-after candidate fingerprint
  sorted raw deletion keys
  sorted final raw key/value entries
  deletion, insertion, and byte counts
~~~

This is not a report, trait, or abstract publication plan. It is the exact raw
payload consumed by mechanical apply.

Staging performs:

- bounded and fallible row decoding;
- field-path traversal;
- expression and predicate evaluation;
- null and filtered-row handling;
- composite-key encoding;
- raw index-key derivation using accepted-after ordinals;
- uniqueness detection;
- duplicate and collision validation;
- current-domain key decoding and ownership validation;
- exact current-domain equality with the row-derived accepted-before projection;
- proof that insertion collisions are contained in the deletion domain;
- deterministic sorting and duplicate raw-key rejection;
- entry-count and byte-budget enforcement;
- final projection completeness checks.

The stage contains no IndexStore writes and does not change IndexState.

### Phase 3: Final Precommit Recheck

Immediately before marker persistence, the operation rechecks publication and
candidate identities. A mismatch discards the in-memory stage and returns a
typed precommit error. The no-await critical section remains the row-consistency
proof.

Because no marker or physical write exists, no rollback is required.

### Phase 4: Persist The Existing Schema Marker

The operation persists the existing AcceptedSchemaPublish candidate through the
normal commit protocol.

The marker does not contain index entries. The combination of candidate schema
and authoritative rows is sufficient to deterministically reconstruct them.

No marker codec or payload change is expected. If implementation discovers that
recovery cannot identify the accepted-after candidate with the current marker,
that is a design blocker and persisted-format review is required.

### Phase 5: Mechanical Forward Apply

With the marker durable, normal apply:

1. appends and applies the marker-bound accepted-schema journal batch using the
   existing publication machinery;
2. marks the complete backing IndexStore Building, matching its current
   store-wide readiness granularity;
3. removes the staged raw deletion keys;
4. inserts the staged, sorted final raw entries;
5. updates derived cardinality/generation bookkeeping and marks the complete
   IndexStore Ready;
6. verifies and clears the marker.

Only the affected entity's user-index domain is replaced. IndexKeyKind::System
reverse-relation entries and other entity tags sharing the physical store must
retain byte-for-byte identical entries.

Raw remove and insert return values do not select control flow after the
boundary. Their exact expected state was proved during staging. If a store
primitive cannot offer an infallible mechanical form after that proof, it must
be split or replaced before this flow can land.

The exact ordering of accepted-schema application and projection replacement is
not a visibility promise. The durable marker and non-Ready state gate every
guarded entrypoint until both agree.

### Phase 6: Forward Recovery

If execution stops after marker persistence, the next guarded entrypoint runs
recovery before serving reads or accepting writes.

Recovery:

1. reads and validates the marker;
2. completes marker-bound journal append/fold idempotently;
3. establishes the accepted-after schema candidate;
4. marks derived index state unavailable;
5. clears every recovery-owned derived index store in deterministic store-path
   order;
6. reconstructs indexes deterministically from authoritative rows under the
   accepted-after schema;
7. validates rebuilt readiness and integrity;
8. marks indexes Ready;
9. verifies and clears the marker.

Recovery never restores accepted-before index entries.

If recovery fails, it leaves the marker present and indexes non-Ready. A later
retry restarts from a clean derived-store clear and repeats the same forward
rebuild. Guarded operations continue to fail closed.

## CREATE INDEX Semantics

Field-path and expression index creation share the same publication flow.

The candidate schema contains the new accepted-after index definition and
ordinal. Staging derives the final user-index domain under that candidate. This
means uniqueness and expression failures are detected before the marker and
before any physical write.

The following current distinctions remain semantic inputs, not physical
transaction implementations:

- field-path versus expression key derivation;
- unique versus non-unique;
- filtered versus unfiltered;
- scalar, null, and composite key behavior;
- startup proposal versus SQL DDL frontend.

There is no inserted-key rollback list. A rejected candidate simply drops its
in-memory stage.

## DROP INDEX Semantics

Index drop no longer removes target keys and remaps later ordinals in place.

The accepted-after candidate omits the target index and assigns final ordinals
according to the maintained catalog rules. Staging derives the entire final
user-index domain using those ordinals. Mechanical apply replaces the old
domain with that projection.

This eliminates:

- original-key/value snapshots;
- remapped-key before-images;
- reverse remap order;
- partial-remap collision recovery;
- stale accepted-before identity rollback.

First, middle, and last index drops use the same apply implementation.

## Index Readiness And Visibility

The durable marker is the primary publication gate. IndexState is a secondary
fail-closed guard for physical readiness.

Required behavior:

- a marker is checked before guarded reads and writes;
- Building or otherwise non-Ready indexes cannot serve queries;
- no query may fall back to a scan merely to hide incomplete publication unless
  that scan is already the normal planner choice under a fully recovered state;
- no accepted-after index is exposed before its physical projection is Ready;
- no accepted-before projection is exposed after a marker owns accepted-after;
- stale plans and cursors continue to invalidate through existing accepted
  publication identity rules.

Direct internal store reads that bypass both marker recovery and readiness
checks remain outside the public atomicity contract and must be structurally
audited.

## Recovery Without A Schema Marker

Recovery may also rebuild derived indexes when no schema marker is present, for
example after readiness detection finds an incomplete derived view.

That path is also redo-only:

- mark indexes unavailable;
- clear the rebuild domain;
- reconstruct from the accepted snapshot and authoritative rows;
- mark Ready only after successful validation.

It does not snapshot and restore the prior physical index store. If the rebuild
fails because authoritative data is corrupt or resources are exhausted, the
database remains fail-closed and a retry starts from clear again.

This is safe because physical indexes are not authority. Restoring an old
derived projection would only make it available without proving that it matches
the accepted schema and rows.

## Heap And Non-Journaled Stores

Stores without durable recovery capability still follow the same precommit
discipline:

- stage and validate the complete user-index-domain replacement before physical
  mutation;
- perform one mechanical replacement;
- do not retain an operation-specific rollback plan.

No durable marker is fabricated for a store that does not support one. The
in-memory apply must be structured so that all input-dependent failures are
excluded before mutation and raw replacement is infallible within that store's
contract.

All heap allocation and candidate construction that can fail must occur while
staging. Heap publication then consumes prepared schema and index values through
an infallible, non-awaiting replacement. If the current heap APIs cannot exclude
partial returned-error behavior, a store-local prepared replacement primitive
is an implementation prerequisite. 0.206 cannot close while retaining heap
per-key rollback, and IC trap rollback is not a fallback.

## Bounds And Resource Ownership

The domain replacement is bounded. Unbounded deletion-key or insertion-entry
vectors are not acceptable.

Before Slice 2 implementation, one existing resource-budget owner must be
selected for:

- maximum staged entry count;
- maximum staged raw bytes;
- maximum staged deletion-key count and bytes;
- maximum decoded row bytes;
- maximum expression and predicate work;
- maximum deterministic-sort workspace.

The limits must be checked incrementally while staging so an oversized
projection fails before marker persistence. The error is typed and contains no
physical side effect.

0.206 does not add a public tuning surface. If no existing internal budget can
represent this work coherently, the implementation records a narrow internal
limit decision in this document before proceeding.

## Failure-State Matrix

| Failure boundary | Marker | Accepted schema | Physical projection | Required next action |
|---|---|---|---|---|
| Candidate derivation or stage validation | Absent | Before | Before | Return typed rejection |
| Final publication-identity recheck | Absent | Before | Before | Return typed stale-state error |
| Before marker write | Absent | Before | Before | Return interruption/error |
| After marker write, before journal append | Present | Before or candidate pending | Before | Recover forward |
| During marker-bound journal append/fold | Present | Before or after | Before | Recover forward idempotently |
| After accepted-schema apply, before domain deletion | Present | After | Before, marker-gated | Recover forward |
| After user-index-domain deletion | Present | After | Target domain empty/Building | Recover forward |
| During staged raw insertion | Present | After | Partial/Building | Clear and rebuild forward |
| After insertion, before Ready | Present | After | After/Building | Recover or validate forward |
| After Ready, before marker clear | Present | After | After/Ready | Verify and clear marker |
| After marker clear | Absent | After | After/Ready | Operation is complete |

At no post-marker row may a caller restore the before projection.

## Idempotence Rules

- Marker-bound journal append remains identity checked and idempotent.
- Accepted-schema apply remains candidate identity checked and idempotent.
- Recovery rebuild always starts by making the derived domain unavailable and
  clearing it, so a partial prior insertion is not interpreted.
- Raw staged insertion is deterministic for a fixed candidate and authoritative
  row set.
- Ready is written only after all expected entries and invariants validate.
- Marker clear remains verified against the candidate it is completing.
- Repeated recovery at every failpoint converges on the same accepted-after
  schema and byte-identical physical projection.

Normal apply need not make each individual raw insertion independently
idempotent because recovery does not resume at an insertion offset. It clears
and rebuilds the derived domain.

## Persisted Compatibility

0.206 expects no persisted-format change:

- no accepted-schema codec change;
- no commit-marker codec or bound change;
- no journal record change;
- no row or entity key change;
- no index key or value change;
- no allocation change;
- no cursor change.

The stage is transient memory only. It is never decoded after restart.

If implementation requires a persisted index generation, staged-entry journal,
or new marker variant, work stops for a separate persisted-format design and
explicit size review.

## Proposed Module Shape

The exact file layout follows the repository's current module boundaries, but
production ownership should converge on:

- one pure accepted-after user-index-domain replacement builder;
- one marker-bound user-index-domain apply function;
- one redo-only recovery rebuild owner;
- one concrete prepared accepted-schema/index publication owner built on the
  existing marker protocol;
- existing field-path, expression, predicate, and key encoders as derivation
  dependencies rather than transaction owners.

The design specifically rejects:

- a generic schema sink;
- a runtime publication epoch layer;
- a trait hierarchy for staged mutation capabilities;
- a callback-based or capability-based publication indirection;
- separate field-path and expression physical transaction protocols;
- a rollback plan embedded in the physical runner report;
- a full-store recovery snapshot.

Metrics such as staged entry count, bytes, and instruction observations are
derived from the concrete projection. They are not a second execution plan.

## Hard-Cut Cleanup

After every index-DDL consumer has moved to redo-only publication, delete:

- field-path index reverse-action and before-image types used only for index
  creation;
- expression inserted-key rollback bookkeeping;
- DROP INDEX original/remapped snapshot and reverse-remap code;
- index-specific publication rejection classifiers;
- recovery full-index-store snapshot capture and restore;
- tests whose only purpose is preserving those removed rollback shapes;
- diagnostics and comments that describe index rollback as maintained behavior.

Keep and strengthen tests for the maintained current surface:

- rejection before marker leaves before state;
- interruption after marker converges forward;
- physical projections exactly match accepted schema and rows;
- incomplete index state fails closed.

Shared code required by DROP COLUMN or other out-of-scope mutations remains
until its own design removes it.

Exit condition: drop has no operation-specific undo and produces byte-identical
accepted-after keys under normal apply and recovery.

### Slice 5: Redo-Only Recovery

- Remove full index-store snapshot restoration.
- Make clear-and-rebuild explicitly fail-closed and retryable.
- Ensure marker-owned recovery always selects accepted-after.
- Add missing failpoints around schema apply, domain deletion, staged insertion,
  Ready, and marker clear where existing failpoints are insufficient.
- Prove repeated ReturnError and PanicUnwind recovery converges.

Exit condition: recovery has no derived-store before-image and no backward
branch.

### Slice 6: Cleanup And Closeout

- Delete dead index rollback representations, helpers, comments, and
  preservation-only tests.
- Update ATOMICITY wording if needed to distinguish ordinary precommit
  rejection from post-boundary interruption.
- Complete the 0.204 evidence rows.
- Re-run focused correctness, performance, heap, and wasm measurements.
- Record the final complexity delta and any deferred generation design.

Exit condition: maintained production index publication has one staging owner,
one forward apply shape, and one redo-only recovery direction.

## Validation Plan

### Structural Audit

Focused searches must prove:

- no IndexStore mutation occurs while building a staged projection;
- no index-DDL code calls physical apply before marker persistence;
- no create/drop index rollback plan remains;
- no recovery index snapshot restore remains;
- no generated model is used as runtime schema authority;
- no post-marker normal apply performs fallible derivation;
- no unbounded persisted decoding or staging collection is introduced;
- no new persisted marker or journal variant exists.

### Operation Matrix

The maintained matrix covers:

- startup field-path index creation;
- SQL field-path index creation;
- SQL expression index creation;
- SQL drop of first, middle, and last index;
- unique and non-unique indexes;
- filtered and unfiltered indexes;
- nullable and non-null values;
- composite keys;
- expression results with null and filtered rows;
- multiple indexes on one entity;
- multiple entities sharing the physical store;
- empty, one-row, and multi-row entities.

### Precommit Rejection

Each rejection must assert marker absent, accepted-before unchanged, and the
complete physical store byte-identical:

- duplicate unique key;
- invalid or unsupported expression result;
- field-path derivation failure;
- raw-key collision;
- backing IndexStore is not Ready at precommit;
- current physical user-index domain diverges from rows and accepted-before;
- stale accepted-before identity;
- staged entry-count limit;
- staged deletion-key limit;
- staged byte limit;
- malformed or oversized authoritative input.

Tests assert typed errors or error variants, never error strings.

### Interruption Matrix

Every applicable boundary runs in both ReturnError and PanicUnwind modes:

- before marker write;
- after marker write;
- before and after marker-bound journal append;
- before and after journal fold;
- after accepted-schema apply;
- before and after user-index-domain deletion;
- during raw entry insertion at deterministic prefixes;
- after raw insertion;
- before and after Ready;
- before and after marker clear.

For every post-marker interruption, the oracle is accepted-after after guarded
reentry. No test accepts accepted-before as a valid recovered outcome.

### Recovery And Reentry

Tests must prove:

- guarded read triggers recovery before observing state;
- guarded write triggers recovery before accepting another mutation;
- repeated recovery from the same failpoint is idempotent;
- partial insertion is discarded by clear-and-rebuild;
- marker remains when deterministic recovery fails;
- non-Ready index state fails closed;
- successful retry clears the marker only after Ready;
- the recovered projection is byte-identical to a clean accepted-after build;
- other entity tags and non-user key kinds remain byte-identical.

### Projection Oracles

For each accepted entity:

- derive expected raw entries independently from authoritative rows and accepted
  schema;
- compare exact sorted key/value bytes with physical storage;
- verify no removed-index ordinal survives;
- verify every accepted index has complete entries;
- verify uniqueness and predicate semantics;
- verify IndexState is Ready only for a complete projection.

The oracle must not call the production projection builder as its expected-value
implementation.

### Focused Validation Boundary

Implementation work runs only focused package, target, and named-test
selections for the changed slice. Full repository and workspace suites remain
user-owned push validation.

## Performance And Size Budget

### Runtime Shape

Staging a final user-index domain costs:

- time: O(current domain entries plus rows multiplied by the union of
  accepted-before and accepted-after indexes);
- transient entries: O(old and final user-index entries for the affected
  entity);
- apply writes: O(old affected-domain entries plus final entries);
- recovery: O(rows multiplied by accepted indexes) after derived-store clear.

This can be more work than target-only create or in-place ordinal remap. The
trade buys a single atomicity proof and removes before-images. It must still fit
the established instruction and heap envelopes.

### Measurement Matrix

Measure at least:

- rows: 0, 1, 16, 256, and 2048 where fixtures permit;
- indexes per entity: 1, 2, and 8;
- field-path and expression;
- unique and non-unique;
- filtered and unfiltered;
- create plus drop-first, drop-middle, and drop-last;
- normal apply and recovery rebuild.

Record:

- staging instructions;
- apply instructions;
- recovery instructions;
- raw entry count and bytes;
- peak transient heap;
- physical writes removed and inserted;
- failure position if a bound rejects.

### Query Hot Path

No query hot-path regression is expected. Marker and readiness checks already
belong to guarded entrypoints. 0.206 must not add per-row or per-index work to a
fully recovered query.

### Wasm

Raw non-gzipped wasm bytes are the primary size measure. Gzip is secondary
context.

The expected direction is neutral or down after operation-specific rollback
systems and recovery snapshots are deleted. A material increase requires an
owner-by-owner explanation and cannot be justified solely by adding another
abstraction layer.

### Complexity

Closeout reports:

- files touched;
- approximate production and test line delta;
- number of production staging, apply, rollback, and recovery owners before and
  after;
- whether implementation became simpler, stayed neutral, or became more
  complex.

The required conceptual result is simpler:

- zero index-DDL rollback owners;
- one concrete user-index-domain replacement stage;
- one forward physical replacement shape;
- one redo-only recovery direction.

## Risks And Mitigations

### Complete User-Index-Domain Staging Is Too Expensive

Mitigation: enforce incremental bounds and measure representative rows and
index counts before cutting every operation over. If it cannot fit, stop for a
separate shadow-generation design. Do not retain an undocumented rollback
fallback.

### Hidden Fallibility Remains After The Marker

Mitigation: audit the apply call graph and keep it limited to raw store
operations and already encoded values. Inject test-only interruptions around
each physical boundary. Any Result-returning or input-dependent production
branch moves back into staging or is replaced with an infallible primitive.

### Candidate Or Rows Drift Between Stage And Marker

Mitigation: keep row scan through marker persistence in one non-awaiting
critical section and recheck accepted identity plus candidate fingerprint
immediately before marker write. Any future await or multi-message staging
requires an explicit row snapshot identity and a new design.

### Accepted Schema Becomes Visible Before Projection Is Ready

Mitigation: the marker gates all guarded entrypoints and IndexState remains
non-Ready until the already validated raw replacement completes. Neither state
is independently servable.

### Recovery Repeats A Deterministic Failure

Mitigation: retain the marker, keep indexes unavailable, return a typed
fail-closed error, and allow operator correction or resource recovery before
retry. Never expose or restore an unproved old projection.

### Marker Does Not Persist Staged Entries

Mitigation: recovery does not need the transient stage. Accepted-after schema
plus authoritative rows is the deterministic source of the same derived
projection. Projection oracle tests prove this equivalence.

### Shared Cleanup Breaks Row DDL

Mitigation: inventory every rollback consumer and delete only index-specific
ownership. DROP COLUMN and other row-layout publication remain explicitly out
of scope.

### Other Entities Are Damaged By Replacement

Mitigation: entity ownership and IndexKeyKind::User are part of every raw key
selection, and tests snapshot other entity tags and non-user key kinds
byte-for-byte before and after create, drop, interruption, and recovery.

### Tests Certify Only Happy-Path Forward Completion

Mitigation: exercise each boundary in ReturnError and PanicUnwind modes, recover
through guarded read and write entrypoints, and compare against an independent
accepted-after physical oracle.

## Closeout Checklist

0.206 is complete only when all of the following are true:

- [ ] accepted-before and candidate identities are rechecked before marker
      write;
- [ ] row scan through marker write contains no await or yield;
- [ ] staging begins only from a Ready backing IndexStore;
- [ ] staging is bounded and performs no physical index mutation;
- [ ] all supported field-path and expression validation finishes pre-marker;
- [ ] normal post-marker apply consumes only prevalidated raw entries;
- [ ] CREATE INDEX has no per-key rollback plan;
- [ ] DROP INDEX has no before-image or reverse ordinal remap;
- [ ] recovery has no full-index-store snapshot restore;
- [ ] every marker-owned interruption recovers to accepted-after;
- [ ] recovery without a marker is redo-only and fail-closed;
- [ ] Ready is unreachable for a partial projection;
- [ ] other entity tags and non-user key kinds remain byte-identical;
- [ ] first, middle, and last drop ordinals are covered;
- [ ] ReturnError and PanicUnwind cover every applicable boundary;
- [ ] repeated recovery is idempotent;
- [ ] projection oracles are independent from production staging;
- [ ] 0.204 owns the final correctness evidence rows;
- [ ] no persisted format or public API changed;
- [ ] no compatibility alias, shim, or legacy fallback remains;
- [ ] shared row-mutation rollback code was not removed accidentally;
- [ ] focused validation passes;
- [ ] instruction, heap, raw wasm, and complexity deltas are recorded;
- [ ] active contracts and diagnostics describe redo-only index publication.

## Closeout Decision

The intended 0.206 result is not merely less rollback code. It is one
unambiguous publication law:

> Reject while everything is still staged, or persist the marker and finish
> forward.

For secondary indexes, accepted schema and authoritative rows already contain
everything required to reach the final state. Operation-specific physical undo
therefore has no authority role and no maintained place in the design.
