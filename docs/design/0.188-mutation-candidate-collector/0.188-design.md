# 0.188 Mutation Candidate Collector

Status: closed for 0.188.5 release prep.

This line follows the 0.184 query-engine audit write-path notes and the 0.187
second query-engine audit closeout. It is a write-path candidate collection
line, not a durable commit rewrite. Runtime behavior should stay unchanged
until a slice has parity coverage for affected row counts, `RETURNING` rows,
rejection codes, ordering, and failure atomicity.

## Context

The 0.184 query-engine audit identified three write paths that still stage
complete candidate sets before mutation:

- SQL `UPDATE` executes a structural selector, projects primary keys, and then
  builds a mutation batch.
- SQL/typed `DELETE` resolves a key stream, materializes rows, applies
  delete-owned post-access filtering/selection, and then prepares commit
  payloads.
- `INSERT ... SELECT` executes the source query and turns every source row into
  an insert patch before commit.

0.184 and 0.187 intentionally cleaned smaller seams first:

- SQL `UPDATE`, `INSERT`, and `DELETE` now share candidate row bound helpers at
  the SQL write boundary.
- DELETE scanned-row attribution is owned by the shared delete candidate
  resolver.
- Typed DELETE, count-only structural DELETE, and DELETE `RETURNING` share a
  generic delete output contract and final commit-window handoff.

The 0.188 implementation closes this slice by giving bounded write candidate
collection shared accounting and diagnostics at the executor/session handoff
while keeping payload abstractions narrow.

## 0.188.0 Progress

The first implementation slice establishes parity coverage before changing
write accounting:

- bounded DELETE has characterization tests for exact row bounds, over-limit
  atomicity, `RETURNING` order, and current staged/`RETURNING` row-cap error
  precedence;
- bounded structural DELETE now checks the selected candidate count after
  delete post-access filtering/selection and before rollback/`RETURNING`
  packaging or commit preflight;
- bounded UPDATE has characterization tests for exact row bounds, over-limit
  atomicity, `RETURNING` order, and staged/`RETURNING` row-cap collision
  precedence;
- bounded UPDATE now passes its staged-row bound into structural mutation-batch
  collection and still revalidates at the final mutation-batch handoff;
- source guards assert the intended DELETE and UPDATE accounting boundaries.

The 0.188.0 slice does not yet introduce a public
`MutationCandidateCollection` type. That broader wrapper should wait until the
DELETE and UPDATE candidate-accounting seams are proven stable enough to share
payload vocabulary without becoming too generic.

## 0.188 Follow-up And Closeout Progress

Follow-up implementation now extends the bounded collection seam to
`INSERT ... SELECT`:

- `INSERT ... SELECT` passes the existing INSERT candidate bounds into the
  structural source-row mutation-batch collector;
- the selector-source mutation-batch collector has one bounded mode instead of
  separate bounded and unbounded helper paths;
- `INSERT ... VALUES` still uses the direct source-size precheck instead of the
  selector-style collector;
- final mutation-batch validation remains in place as a second guard at the
  write handoff.

Follow-up implementation also names candidate accounting as the shared
successful-write row-count seam:

- SQL write metrics now flow through `SqlWriteCandidateAccounting`;
- semantic candidate counts are distinct from matched, mutated, and
  `RETURNING` row counts before they are projected into the existing public
  metrics event;
- the exported metrics fields and row-count behavior remain unchanged.

Follow-up implementation adds the first internal candidate diagnostic
vocabulary:

- staged-row bound checks identify INSERT VALUES source admission,
  selector-source mutation-batch collection, or final mutation-batch handoff;
- over-limit diagnostics preserve the semantic candidate count and source stage
  before projecting to the existing `StagedRowsTooMany` public error;
- no public metrics field, diagnostic code, or statement result shape changes.

Follow-up implementation extends selector-source diagnostics:

- `SqlWriteCandidateCollection` now carries the internal diagnostics while
  `SqlWriteMutationBatch` owns only row payloads;
- the collection retains projected selector source-row counts alongside
  semantic mutation-candidate counts;
- selector-source collection records the projected source-row count once when
  SQL projection materialization hands rows to mutation-batch construction;
- this remains internal diagnostic state and does not change public SQL write
  metrics or statement output.

Follow-up implementation moves structural DELETE toward the same vocabulary:

- `StructuralDeleteCandidateCollection` records loaded structural rows and
  post-access selected candidates separately;
- bounded structural DELETE validation consumes the selected-candidate count
  from that collection after delete post-access filtering/selection;
- structural DELETE row-limit checks now retain internal over-limit source
  stages for post-access selection and final projection validation;
- commit preparation and public DELETE count/`RETURNING` behavior remain
  unchanged.

Closeout keeps the final row-limit vocabulary internal:

- structural DELETE over-limit diagnostics distinguish post-access selection
  from final projection validation;
- the public error remains the existing staged-row SQL write-boundary error;
- the 0.188 line does not expose new metrics, statement payload fields, cache
  identity, cursor identity, or persistence formats.

## Design Principle

0.188 should not start by building "a generic mutation collector." It should
start by building one authoritative semantic accounting boundary for public
bounded writes, with DELETE proving the hardest case first.

The first useful sharing target is accounting, diagnostics, ordering proof, and
bound policy. Payload abstraction can generalize later after DELETE proves the
boundary.

## Problem

The current write paths enforce public staged-row and `RETURNING` bounds, but
the enforcement point is still downstream of path-specific materialization.

That creates three risks:

- memory/cycle pressure from collecting more candidate rows than a public
  bounded write will later accept;
- drift between UPDATE and DELETE diagnostics/metrics for equivalent bounded
  write policy failures;
- future chunking or streaming work starting from three different candidate
  collection shapes instead of one contract.

DELETE is the subtle case. It is unsafe to reject based only on raw access-key
count because residual filters, cursor/order selection, consistency policy, and
delete post-access selection can remove rows before they become actual delete
candidates. The limit proof must consume the same post-access semantics that
execution consumes.

## Goal

Introduce a shared mutation-candidate collector contract for bounded public
`UPDATE` and `DELETE` execution while keeping the current full-batch commit
window.

The first implementation line should:

- keep storage mutation atomicity exactly as it is today;
- keep public SQL/fluent/generated write behavior unchanged;
- centralize staged-row and `RETURNING` row bound accounting;
- reject as soon as the collector can prove that the number of semantic
  mutation candidates exceeds the relevant bound;
- preserve deterministic row order where the public surface observes it;
- feed existing commit-window preflight with a complete bounded batch.

## Non-Goals

- Do not implement chunked durable commits.
- Do not change the mutation commit window's full-batch preflight requirement.
- Do not widen generated endpoint exposure or silently bound broad
  session/admin SQL writes.
- Do not merge `INSERT ... VALUES` into selector-style collection.
- Do not force `INSERT ... SELECT` into the first slice.
- Do not change `RETURNING` result shape, column order, row order, or response
  size enforcement.
- Do not introduce cost-based write routing.

## Candidate Contract

The exact type names are not fixed, but the shared contract should represent
candidate collection facts rather than SQL parser details.

Possible vocabulary:

```text
MutationCandidateCollection
  write_family
  surface_policy
  source_identity
  source_route_diagnostics
  candidate_payload
  accounting
  diagnostics
  ordering_proof
  collection_safety
  commit_mode
```

Candidate payload should be explicit:

```text
MutationCandidatePayload
  UpdatePatchRows(Vec<(PrimaryKey, StructuralPatch)>)
  DeleteRows(Vec<DeleteCandidateRow>)
```

Policy, accounting, diagnostics, ordering, and safety should be first-class:

```text
BoundedWritePolicy
  staged_row_bound: Option<RowBound>
  returning_row_bound: Option<RowBound>
  public_surface
  broad_write_policy
  error_family_policy

CandidateAccounting
  semantic_candidate_count
  returning_row_count
  over_limit: Option<OverLimitKind>

CandidateDiagnostics
  keys_resolved
  rows_loaded
  rows_missing_or_stale
  rows_rejected_by_consistency
  rows_rejected_by_residual_filter
  rows_rejected_by_post_access_selection
  staged_candidates
  returning_rows
  over_limit_at

OrderingProof
  Unobservable
  SourceStreamOrderStable
  ExplicitOrderSatisfied
  CursorWindowApplied
  PostAccessSelectionOrderPreserved

CollectionSafety
  StreamCountable
  RequiresFullPostAccessSelection
  RequiresFullOrderingSelection
  RequiresFullMaterialization

CommitMode
  FullBatchCommitWindow
```

`DeleteCandidateRow` should be the post-access selected row representation,
not a raw key. It must carry enough information for:

- primary key identity;
- accepted-schema row snapshot or the existing row reference needed by current
  delete preflight;
- rollback row ownership;
- relation validation;
- commit row-op preparation;
- count-only DELETE;
- DELETE `RETURNING` projection.

The first slice must preserve the current row identity/snapshot/reference
contract handed to delete preflight. If the current path re-reads rows during
commit preflight, the collector must not accidentally make the earlier selected
row snapshot authoritative. If the current path relies on a selected row
snapshot, the collector must not downgrade it to a weaker reference without
equivalent generation validation.

The contract should sit at the executor/session handoff. SQL can be the first
frontend to build or consume it, but the type must not carry SQL AST nodes,
SQL parser token state, or SQL-only coercion traces.

`CommitMode` is deliberately boring in 0.188. The only admitted mode is the
existing full-batch commit window. A general-looking atomicity enum should not
be added until a later line actually designs another commit mode.

## Candidate Accounting Order

The collector owns bound accounting only over semantic mutation candidates.

For DELETE, a semantic mutation candidate is a row that has survived the same
stages that currently decide whether the row would be deleted:

```text
access key resolved
  -> row loaded through the structural consistency boundary
  -> missing/stale/consistency-rejected row handled
  -> residual filter applied
  -> delete-owned post-access selection applied
  -> ordering/cursor semantics applied when they affect selected rows
  -> DeleteCandidateRow created
  -> staged-row count incremented
  -> staged-row bound checked
  -> returning-row count incremented when RETURNING is active
  -> returning-row bound checked
  -> candidate appended
```

The exact implementation may fuse stages, but the accounting boundary must not
move earlier. Raw access keys, rows rejected by consistency, rows rejected by a
residual filter, and rows excluded by post-access selection must not count
unless current public behavior already counts them.

For UPDATE, a semantic mutation candidate is the primary-key/patch pair
produced by the existing structural projection and patch-construction path.
Source rows that fail projection, coercion, schema acceptance, residual
filtering, or generated-field preparation must preserve existing error and
counting behavior.

Bounds should be checked by comparing the semantic candidate count with the
configured public bound. Implementation should avoid relying on unchecked
`max_rows + 1` arithmetic; conceptually the collector rejects when count first
exceeds the bound.

When both staged-row and `RETURNING` row bounds are present, the collector must
preserve the current public error precedence. If no explicit precedence is
captured today, define one before implementation and add collision tests. The
default intended rule is:

- staged-row bound wins when the write cardinality itself is not allowed;
- `RETURNING` bound wins when the mutation cardinality is allowed but the
  response cardinality is not allowed.

A `RETURNING` row counts once the mutation candidate is known to produce one
public `RETURNING` row under the existing result-shaping rules. The collector
must preserve whether current behavior performs `RETURNING` projection before
or after response-bound rejection, including projection error precedence.

Minimum error-precedence cases to characterize before migration:

| Condition | Expected Public Error | Existing Parity Test |
| --- | --- | --- |
| staged bound exceeded only | Preserve current staged-row over-limit family | Required |
| `RETURNING` bound exceeded only | Preserve current returning/response over-limit family | Required |
| staged and `RETURNING` bounds both exceeded | Preserve or define collision precedence | Required |
| projection fails before staged bound exceeded | Preserve projection/schema error | Required |
| projection fails on the same candidate that would exceed staged bound | Preserve current precedence | Required |
| `RETURNING` projection fails on the same row that would exceed `RETURNING` bound | Preserve current precedence | Required |
| schema/patch construction fails before bound exceeded | Preserve schema/patch error | Required |
| schema/patch construction fails on the first candidate beyond the bound | Preserve current precedence | Required |

## Selection And Ordering Safety

The collector must classify selection as stream-countable or requiring complete
selection before it may short-circuit.

It may stop early only after all operators that can remove, reorder, or window
candidates have run. If a selector requires global knowledge, the collector
must not short-circuit before that selector has produced the final semantic
candidate stream.

Order is observable when it affects:

- which rows are selected;
- affected-row ordering;
- `RETURNING` row order;
- cursor/window semantics.

For any order-observable write surface, the collector must carry an explicit
`OrderingProof` or use a `CollectionSafety` mode that requires full selection
before bound accounting. Parity tests must compare returned primary-key order
against the pre-collector implementation.

## Duplicate And Missing Row Semantics

Candidate bounds must count the same logical mutation targets that the existing
write path and commit preflight would accept.

Before implementation, each first-slice path must answer:

- whether the source key/query stream can produce duplicate primary keys;
- whether current behavior mutates once per unique primary key or once per
  appearance;
- where any existing deduplication occurs;
- whether the bound currently counts source appearances or final mutation
  targets.

If duplicates can occur, duplicate handling must happen at the same semantic
stage as today, and parity tests must cover duplicate source candidates.

Missing, stale, or consistency-rejected rows must preserve current behavior for
all of:

- scanned/read metrics;
- selected/staged candidate counts;
- affected-row counts;
- returned rows;
- error family.

## Collection And Commit Boundary

The collector builds a bounded batch before the commit window. The commit
window remains responsible for revalidating all facts that are currently
commit-window authority before durable apply. 0.188 must not make
collector-produced facts newly authoritative unless an equivalent preflight
guard already exists.

The collector produces candidates from the current read/selection path. The
commit window remains authoritative for:

- relation validation;
- uniqueness checks;
- row and index generation guards;
- journal/marker preparation;
- rollback safety;
- storage mutation.

If storage state changes between candidate collection and commit preflight, the
write must behave exactly as the current path behaves: existing generation
guards or preflight checks reject drift, or the current accepted behavior is
preserved. Even though IC execution is single-threaded per message, this
boundary should be explicit so future durability/concurrency changes do not
weaken it.

## Pre-Implementation Parity Discovery

Before moving a write path onto the collector, add characterization tests for
the current path covering:

- duplicate candidates;
- missing, stale, or consistency-rejected rows;
- ordering and cursor-sensitive selection;
- affected-row counts;
- `RETURNING` row count and row order;
- staged-row and `RETURNING` bound precedence;
- projection, schema, and patch-construction error precedence;
- metrics and diagnostic payload shape where public or test-observable.

Collector implementation for that path may begin only after these tests
document the current observable behavior. If current behavior is inconsistent
or undesirable, the change must be explicitly promoted from parity-preserving
refactor to behavior change and documented as such.

## Boundary Split

### Frontend/Session Responsibilities

- Build the accepted schema/write descriptor.
- Choose public execution bounds from generated/session policy.
- Lower SQL/fluent write intent into existing structural query or mutation
  patch inputs.
- Convert collector output into the existing public statement result.

### Collector Responsibilities

- Execute the source candidate query/stream through existing planner/executor
  authority.
- Apply the same residual/post-access semantics as execution.
- Count selected candidates and returned rows at one shared boundary.
- Reject once the semantic candidate count exceeds a public bound, but only
  when the selected collection safety mode permits early rejection.
- Return the same public over-limit error family, code, and diagnostic shape as
  the existing bounded write path before any commit-window preflight, journal
  marker creation, side-effecting row-op preparation, or storage mutation can
  occur.
- Emit shared candidate metrics and diagnostics.

### Commit Window Responsibilities

- Keep full-batch preflight.
- Validate relations and uniqueness.
- Build journal/marker payloads.
- Apply row/index mutations atomically as today.

0.188 must not move these responsibilities out of the commit window.

## First Slice

Start with bounded public `DELETE`, because it is the audit finding with the
clearest memory/cycle pressure and the strictest semantic ordering requirement.

Use a narrow `BoundedDeleteCandidateCollector` first, but give it the
accounting, diagnostics, ordering, policy, and safety structures intended to
become shared. Do not force UPDATE into the same payload enum until DELETE
behavior is proven.

The slice should:

1. Add a delete candidate collector that accepts an optional candidate bound.
2. Resolve keys through the existing scalar key-stream resolver.
3. Read rows through the existing structural consistency boundary.
4. Apply delete post-access filtering/selection before incrementing the
   selected-row count.
5. Stop and return the existing staged-row-too-many style error at the first
   selected semantic candidate beyond the bound.
6. Feed the existing generic delete output contract for count-only, typed, and
   `RETURNING` deletes.
7. Preserve rows-scanned attribution and add/keep selected-row diagnostics.
8. Avoid building unnecessary `RETURNING` payloads for the over-limit row when
   current error precedence permits rejection before projection.

This should not change broad unbounded session/admin DELETE behavior. When no
bound exists, the collector may continue to materialize the full selected set
as today, and this path must remain behaviorally identical to the current
unbounded session/admin path unless explicitly moved under parity coverage.

## Second Slice

Move bounded SQL `UPDATE` source collection onto the same candidate accounting
contract.

The slice should:

- keep the existing structural projection source query;
- count staged update rows as projected primary-key/patch candidates;
- reject at the first primary-key/patch candidate beyond the bound before
  building/opening the mutation execution wrapper;
- preserve `RETURNING` row caps separately from staged-row caps;
- preserve current affected-row count and row order.

This slice should not change patch construction, generated field handling,
accepted schema row contracts, or save/commit preflight.

Error precedence must be preserved when a row that would exceed a bound would
also fail projection, patch construction, schema validation, or generated-field
preparation. Add parity tests before moving the counting boundary.

## Later Slices

- Add shared diagnostics using the candidate diagnostic taxonomy for source
  route, keys resolved, rows loaded, consistency rejection, residual filtering,
  post-access filtering, staged rows, returned rows, and over-limit reason.
- Consider chunked preflight/apply only after the shared collector exists.
- Design chunk-level journal/rollback only if full-batch preflight becomes a
  proven bottleneck.

## Invariants

- A failed multi-row write must not partially mutate storage.
- Public bounded write policies must reject over-limit candidate sets before
  opening the mutation commit window.
- `RETURNING` row limits must remain distinct from staged-row limits.
- DELETE limits must count rows after residual filtering and post-access
  selection, not raw access keys.
- UPDATE limits must count rows after source projection produces actual
  primary-key/patch candidates.
- Candidate ordering must remain explicit where public results observe row
  order.
- Existing commit-window generation guards remain the authority for detecting
  index-store drift between preflight and apply.
- Accepted schema snapshots remain runtime authority for row contracts.

## Acceptance Matrix

Minimum tests for the first DELETE slice:

- count-only DELETE within bound;
- count-only DELETE over bound rejects before commit;
- DELETE `RETURNING` within staged-row and returning-row bounds;
- DELETE `RETURNING` over staged-row bound rejects before commit;
- DELETE `RETURNING` over returning-row bound preserves the existing error
  family;
- DELETE with residual filter where raw key count exceeds the bound but selected
  rows are within the bound;
- DELETE with residual filter where selected rows exceed the bound by one;
- DELETE with `bound = 0` and zero selected rows succeeds;
- DELETE with `bound = 0` and one selected row rejects before commit;
- DELETE with `bound = N`, exactly `N` selected rows succeeds;
- DELETE with `bound = N`, `N + 1` selected rows rejects before commit;
- DELETE where raw key stream contains duplicate keys preserves current
  affected-row and bound behavior;
- DELETE where the key stream includes missing/stale/consistency-rejected rows
  preserves current affected-row and metric behavior;
- DELETE `RETURNING` where staged-row and returning-row bounds are both
  exceeded preserves the defined/current error precedence;
- DELETE `RETURNING` over limit does not write journal/marker/commit artifacts;
- ordered DELETE / cursor-sensitive shape preserves row order;
- DELETE with order/cursor selection where early stopping before post-access
  selection would select the wrong rows;
- typed/fluent bounded DELETE preserves affected count, rejection code,
  exposed `RETURNING` behavior, metrics, and failure atomicity;
- generated bounded DELETE endpoint preserves rejection code and atomicity;
- unbounded session/admin DELETE remains unchanged.

Minimum tests for the UPDATE slice:

- UPDATE within bound;
- UPDATE over staged-row bound rejects before commit;
- UPDATE `RETURNING` within staged-row and returning-row bounds;
- UPDATE with residual/source filter preserves affected rows;
- UPDATE with `bound = 0` and zero matched rows succeeds;
- UPDATE with `bound = 0` and one projected patch rejects before commit;
- UPDATE with `bound = N`, exactly `N` projected patches succeeds;
- UPDATE with `bound = N`, `N + 1` projected patches rejects before commit;
- UPDATE with duplicate projected primary keys preserves current behavior;
- UPDATE where source rows exceed the bound but projected patch candidates do
  not preserves current behavior;
- UPDATE where patch construction fails on the same row that would exceed a
  bound preserves existing error precedence;
- UPDATE `RETURNING` where staged and returning bounds collide preserves the
  defined/current error precedence;
- UPDATE over-limit rejection does not instantiate or enter the mutation
  execution wrapper if that is observable in tests;
- generated bounded UPDATE endpoint preserves rejection code and atomicity;
- unbounded session/admin UPDATE remains unchanged.

Cross-cutting validation:

- no commit marker is written on over-limit rejection;
- row/index stores are unchanged after over-limit rejection;
- relation and index stores are byte-equivalent or logically equivalent after
  over-limit rejection;
- SQL write metrics report the same row counts as before for accepted writes;
- metrics parity is covered for SQL, typed/fluent, and generated surfaces that
  expose the bounded write path;
- EXPLAIN/route/cache identity is unchanged;
- error code, error message family, and diagnostic payload parity are covered;
- bound arithmetic is overflow-safe for large configured bounds.

## Deferred Work

0.188 intentionally leaves broader write-pipeline work to later lines:

- extract a public or cross-family `MutationCandidateCollection` wrapper only
  after the SQL and structural DELETE payload vocabulary is stable enough to
  avoid a vague generic container;
- decide which internal candidate diagnostics should become public metrics or
  query-visible diagnostics, with naming and versioning handled as public
  contract work;
- route unbounded session/admin writes through the same collection vocabulary
  only after separate parity coverage for unbounded behavior;
- design chunked durable commits and chunk-level journal/rollback separately
  from candidate accounting;
- add cost/selectivity write routing only as planner behavior work, not as a
  mutation-accounting cleanup;
- pursue more aggressive streaming or early-stop behavior only where ordering,
  cursor, residual, and post-access selection proofs make it semantic-preserving.

`INSERT ... SELECT` is no longer deferred from this line: it now uses the
bounded selector-source collection path. `INSERT ... VALUES` remains on direct
source-size admission by design.

## Closeout Gate

0.188 is considered closed when:

- characterization tests document current behavior for duplicate candidates,
  stale/missing rows, ordering, affected counts, `RETURNING`, and error
  precedence before collector migration;
- bounded DELETE and UPDATE share candidate accounting and diagnostics;
- over-limit bounded writes reject before commit-window open;
- DELETE and UPDATE preserve current duplicate, stale/missing row, ordering, and
  error-precedence behavior;
- existing public behavior, row order, `RETURNING` payloads, and error families
  are covered by parity tests;
- deferred work is explicitly limited to generic/public collector extraction,
  public diagnostic exposure, unbounded session/admin routing, chunked durable
  commits, chunk-level journal/rollback, cost/selectivity routing, and
  semantics-proven streaming/early-stop improvements.
