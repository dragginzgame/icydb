# IcyDB 0.58 Slice D: Structural Plan Cut

Status: **Planned**.

Goal: promote the planner's existing `AccessPlan<Value>` representation into
the canonical query/executor plan model, then confine typed primary-key logic
to one executor boundary sidecar.

## Invariant

After this slice:

- `AccessPlannedQuery` is structural
- `ContinuationContract` is structural
- executor runtime consumes structural plan data
- typed primary-key behavior exists only in one executor-owned sidecar

The sidecar is helper state, not plan state.

## Type Transformations

The cut is:

```rust
AccessPath<K>           -> AccessPath<Value> as canonical plan shape
AccessPlan<K>           -> AccessPlan<Value> as canonical plan shape
AccessPlannedQuery<K>   -> AccessPlannedQuery
ContinuationContract<K> -> ContinuationContract
ExecutablePlan<E>       -> structural plan + typed access sidecar
```

Planner/query boundaries already produce `AccessPlan<Value>` today. This slice
promotes that representation instead of introducing a new plan form.

## Boundary Model

Old flow:

```text
planner
  -> AccessPlan<Value>
  -> AccessPlan<K>
  -> AccessPlannedQuery<K>
  -> executor
```

New flow:

```text
planner
  -> AccessPlan<Value>
  -> AccessPlannedQuery
  -> ExecutablePlan<E> {
       plan,
       typed_access_sidecar,
     }
  -> executor
```

## Sidecar Contract

The executor-owned sidecar exists only to support typed physical traversal.

It may contain:

- typed primary-key conversion helpers
- typed access-path payloads for physical scans
- typed traversal hooks required by `Context<E>` and primary-key/index readers

It must not contain:

- predicate semantics
- projection semantics
- continuation semantics
- planner routing logic
- a second copy of plan structure

## Execution Order

1. make `AccessPlannedQuery` structural
2. make `ContinuationContract` structural
3. attach typed access sidecar to `ExecutablePlan<E>`
4. switch `ExecutionPreparation` and `ExecutionInputs` to structural plans
5. switch grouped runtime to structural plan + sidecar

## Validation Focus

This slice must preserve:

- access-path ordering
- continuation compatibility
- query fingerprint stability
- explain output semantics

## Development Loop

While Slice D is in flight, use native build/test validation as the default
inner loop:

```bash
cargo check -p icydb-core --lib --locked
cargo test -p icydb-core --lib --locked
```

Do not run wasm size builds on every intermediate change. Wasm analysis should
wait until one structural batch is complete and the executor/runtime surface is
clean again.

It is acceptable for typed executor traversal to remain behind the sidecar
boundary during the migration. It is not acceptable to keep type parameters in
the canonical plan model.
