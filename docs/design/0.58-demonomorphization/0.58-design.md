### Practical Implication for `0.58`

`0.58` is a continuation and hardening line, not a first-pass redesign.

The remaining gap is to complete the hard cut so runtime identity and hot-path
execution are consistently non-generic and `EntityTag`-driven everywhere in
scope, without fallback islands.

## Intent

`0.58` is a hard-cut runtime simplification line.

The primary objective is to remove runtime generic specialization pressure and
string-based entity identity from execution-heavy paths.

This line replaces runtime `EntityName` and runtime `E: EntityKind` execution
coupling with one compact runtime identity:

- `EntityTag(u64)`

This is an intentional breaking line with no compatibility shims.

## Hard Rules

1. No backward compatibility layer.
2. No dual identity representation in runtime flows.
3. No runtime hashing for entity identity.
4. Runtime identity must be constant, copyable, and O(1) comparable.
5. Runtime scan/execution paths must not depend on `EntityName`.
6. Runtime scan/execution paths must not be generic over `E: EntityKind`.

## Runtime Identity Contract

Introduce:

```rust
#[derive(Clone, Copy, Debug, Eq, PartialEq, Hash, Ord, PartialOrd)]
pub struct EntityTag(pub u64);
```

Extend `EntityKind`:

```rust
pub trait EntityKind: EntitySchema + EntityPlacement + Kind + TypeKind {
    const ENTITY_TAG: EntityTag;
}
```

Every entity must declare one fixed compile-time tag constant.

## EntityTag Assignment Contract

Required invariant:

- `EntityTag` values must be globally unique and stable across builds.

Assignment and enforcement rules:

1. tags are assigned at build/codegen time
2. tags are deterministic and reproducible
3. tags are fixed compile-time constants in generated/declared entity impls
4. a build validation step must:
   - collect all entities
   - verify tag uniqueness
   - fail the build on any collision

Explicitly prohibited:

- runtime tag generation
- runtime hash-based tag derivation
- tag assignment that depends on mutable hash algorithms or unstable ordering

## Scope

In-scope runtime domains for de-monomorphization:

- `db/index/*`
- `db/cursor/*`
- `db/executor/*`
- `db/access/*`

Identity replacement target:

- remove runtime `EntityName` usage from key/runtime execution flow
- move runtime dispatch identity to `EntityTag`

Out of scope:

- compatibility migration adapters
- preserving old persisted `EntityName` key wire format
- incremental dual-mode fallbacks

## DataKey Cut

`DataKey` runtime identity must move from:

```rust
entity: EntityName
```

to:

```rust
entity: EntityTag
```

Generic constructors such as `DataKey::from_key::<E>` / `try_new::<E>` should
be removed from runtime-facing paths and replaced by explicit identity-bearing
constructors:

```rust
DataKey::new(entity_tag, storage_key)
```

## Persistence Format Rule (Hard Cut)

`0.58` selects one explicit persistence contract:

- Option A (selected): persist `EntityTag` only.
- pre-`0.58` persisted `EntityName`-keyed data is invalid for this line and
  requires reset/rebuild.

Option B is explicitly rejected for this line:

- no mixed-format decode support
- no dual write/read identity format
- no compatibility branch in runtime decode paths

## De-Monomorphization Boundary Rule

Runtime flow functions in scan/index/cursor/executor/access must not carry
`<E: EntityKind>` parameters.

Typed entity boundaries remain only at outer API/planner/decode adapters.

Boundary conversion pattern:

- outer typed layer supplies `E::ENTITY_TAG`
- runtime layer executes on `EntityTag + raw key/value primitives`
- final typed decode/materialization happens at adapter boundary

Typed decode placement rule (strict):

- typed decode must happen after scan completion or at batch boundary
- typed decode must not occur per row inside scan loops

Disallowed pattern:

```rust
for row in scan_rows(...) {
    let entity = decode::<E>(row)?;
    // ...
}
```

Required pattern:

```rust
let raw_rows = scan_rows(...)?;
let entities = decode_batch::<E>(raw_rows)?;
```

## Runtime Guardrails (CI Hard Fail)

In runtime-heavy modules:

- `db/index/*`
- `db/cursor/*`
- `db/executor/*`
- `db/access/*`

any usage of `EntityKind` or `EntityName` must fail CI.

Enforcement mechanism:

- add one CI invariant script (grep-based guard is acceptable)
- fail CI when disallowed runtime-type identity dependencies are detected
- run this guard in the standard checks pipeline

## Work Slices

Patch labels are intentionally omitted.

### Slice A: Identity Primitive and Trait Contract

- add `EntityTag`
- add `EntityKind::ENTITY_TAG`
- assign compile-time tags for every entity
- implement deterministic build/codegen tag assignment contract
- add uniqueness guard (build failure plus test/static assertion)

### Slice B: Core Key Identity Rewrite

- rewrite `DataKey` identity field to `EntityTag`
- remove `EntityName` from runtime key flow
- update key encode/decode contracts to `EntityTag` representation

### Slice C: Runtime Signature Cut

- remove `<E: EntityKind>` from runtime-heavy functions in:
  - `db/index`
  - `db/cursor`
  - `db/executor`
  - `db/access`
- replace with explicit `EntityTag` parameters

### Slice D: Call-Site Boundary Rewire

- update planner/API boundary calls to pass `E::ENTITY_TAG`
- keep typed boundaries only where decode/materialization requires `E`

### Slice E: Cleanup and Guardrails

- remove now-dead generic helpers from runtime paths
- enforce no `EntityName` in runtime index/cursor/executor/access flows
- enforce no runtime `<E: EntityKind>` usage in targeted runtime modules
- enforce CI hard-fail checks for disallowed `EntityKind`/`EntityName` imports
  in runtime-heavy modules

## Acceptance Criteria

1. Runtime paths in `db/index`, `db/cursor`, `db/executor`, `db/access` no
   longer use `EntityName` identity.
2. Runtime paths in those modules no longer expose `<E: EntityKind>` in hot
   execution signatures.
3. Entity identity in runtime flow is `EntityTag` only.
4. All entities have compile-time tag constants and uniqueness checks.
5. Persistence uses `EntityTag` only with no mixed-format decode path.
6. CI fails on any `EntityKind`/`EntityName` usage in targeted runtime modules.
7. Workspace build/tests pass under the new contract.
8. Wasm analysis shows reduced monomorphized duplication in runtime paths.

## Expected Impact (Sanity Check)

Current partial-runtime refactor baseline (before hard-cut completion) showed
size regressions in quickstart/minimal wasm builds.

Expected outcome after completing this hard-cut contract:

- remove repeated N-by-entity duplication in scan/continuation/decode control
  flow
- recover roughly `~5KB` to `~20KB` wasm size depending on entity count and
  path mix
- flatten future wasm growth by preventing generic/runtime identity creep

## Risks

1. Over-broad signature churn across non-runtime layers.
2. Hidden `EntityKind` couplings reintroduced via helper modules.
3. Identity collisions if tag uniqueness is not enforced.
4. Runtime regressions if typed decode boundary drifts into runtime loops.
5. Generic reintroduction via helper traits (for example `FieldValue`,
   projection helpers, decode helpers) re-infects runtime loops.

Mitigations:

- enforce module-boundary checks
- add explicit no-generic/no-`EntityName` guards in targeted runtime modules
- fail build/test on duplicate `EntityTag` assignments
- keep typed decode at explicit adapter boundary only
- audit helper traits/functions for hidden generic runtime re-entry points

## Exit Criteria

`0.58` is complete when runtime execution flow is identity-driven by
`EntityTag` and no longer monomorphized on entity type across hot runtime
paths.
