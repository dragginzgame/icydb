# 0.127 Naming Cleanup

## Thesis

`0.127` should standardize type and helper naming across the codebase so
ownership boundaries are easier to read and future cleanup does not keep
reintroducing near-synonyms for the same kind of thing.

This is not a semantics slice.

It is not a behavior change slice.

It is not a broad repo-wide polish pass.

It is a naming-boundary slice whose job is to align names with architectural
role and remove historical drift in the most important type and helper
families.

The target end state is:

* adjacent modules use the same name family for the same kind of concept
* type names communicate ownership role, not history
* helper verbs are consistent about what kind of work they describe
* new cleanup work has one naming rubric to extend instead of local precedent
* the same naming rubric applies repo-wide even when rename work is staged by
  subsystem

---

## Why `0.127` Exists

The recent structural cleanup work improved ownership, but the names now show
where older layers grew independently.

The same kind of thing is often called different names across nearby modules:

* `Plan`
* `Contract`
* `Context`
* `Facts`
* `Capability`
* `Decision`
* `Shape`
* `Analysis`

That is not purely cosmetic.

In this codebase, names are part of the architecture:

* they tell readers which layer owns meaning
* they tell maintainers whether something is a frozen snapshot, a final
  executable payload, a local walk bundle, or a proof surface
* they tell future patches whether a new helper belongs in an existing family
  or is inventing another one

Without a naming pass, the codebase will keep paying three costs.

### 1. Architectural Meaning Is Harder To Read

If `Facts`, `Context`, `Plan`, and `Contract` are used loosely, a reader must
inspect fields and call sites before understanding the role of a type.

### 2. Cleanup Work Leaves Behind Historical Drift

Refactors often improve ownership but preserve old names to avoid churn.

That is rational locally, but over several cleanup slices it leaves behind
"better structure, older names."

### 3. New Helpers Recreate Old Ambiguity

When there is no stable naming rubric, new code defaults to:

* another `resolve_*`
* another `prepare_*`
* another `Context`
* another `Shape`

even when the new item is really a `Facts`, `Decision`, `Analysis`, or
`classify_*` helper.

---

## Goal

Standardize the high-value naming families that describe architectural role.

`0.127` should deliver:

* one naming rubric for the main structural type families
* one naming rubric for helper verbs
* a concrete audit of the highest-value drift families
* a bounded rename pass focused on high-signal architectural surfaces

This slice should improve readability and future consistency without turning
into a style-only repo churn pass.

---

## Core Invariant

Names should encode architectural role before historical lineage.

Corollaries:

* if two adjacent types play the same role, they should use the same family
  name
* if two helpers perform different classes of work, they should not share the
  same verb casually
* no new helper or type should introduce another near-synonym when an adopted
  family already exists

---

## Scope

### In Scope

* naming families for architectural type roles
* naming families for helper verbs
* the full codebase
* planner, route, covering, lowering, prepared execution, cursor,
  continuation, executor pipeline, session, and adjacent public/internal
  boundary surfaces as the highest-priority families
* renames that materially improve ownership readability

### Out Of Scope

* broad stylistic rewrites with no family-level payoff
* one-off renames with no family-level payoff
* semantics or behavior changes hidden behind naming churn
* test-only naming cleanup unless it supports a production family rename

## Naming Rubric

### Type Families

#### `*Plan`

Use `*Plan` for:

* executable payloads
* planner-selected payloads
* phase outputs consumed by the next stage

Do not use `*Plan` for:

* local walk bundles
* proof-only surfaces
* lightweight classification snapshots

#### `*Contract`

Use `*Contract` for:

* proof or admission surfaces
* cross-layer guarantees consumed by another subsystem
* reusable shape agreements

Do not use `*Contract` for final chosen outcomes if they include fallback or
decision state.

#### `*Decision`

Use `*Decision` for:

* final chosen outcome objects
* structures that pair selected mode/route plus reason

#### `*Facts`

Use `*Facts` for:

* frozen classification snapshots
* derived supporting inputs reused by more than one downstream helper

#### `*Context`

Use `*Context` for:

* local traversal bundles
* short-lived, owner-local execution or derivation inputs

Do not use `*Context` for reusable classification records when `*Facts` would
be clearer.

#### `*Shape`

Use `*Shape` for:

* compact enum-style structural families
* admitted/rejected structural forms

#### `*Analysis`

Use `*Analysis` for:

* richer descriptive results with multiple derived fields
* structural answers that are more than one enum or bool

### Helper Verbs

#### `classify_*`

Use when returning:

* a discrete family
* an enum
* a compact result category

#### `analyze_*`

Use when returning:

* a richer multi-field struct
* a composite descriptive result

#### `derive_*`

Use when:

* computing a new fact or output from inputs
* materializing a result that did not already exist

#### `resolve_*`

Use when:

* looking up or selecting one already-existing thing
* mapping an input onto an existing slot, symbol, path, or component

Do not use `resolve_*` for broad multi-step derivation pipelines.

#### `prepare_*`

Use when:

* freezing reusable state for a later phase
* precomputing a reusable bundle for subsequent execution or planning

Do not use `prepare_*` for final public outputs when `derive_*` would be
clearer.

#### `canonicalize_*`

Use only for:

* normalization to an equivalent stable form

Do not use it for arbitrary validation, lookup, or shaping.

## Highest-Value Audit Families

`0.127` covers the entire codebase.

The families below are the first rename batches because they have the highest
payoff-to-churn ratio, not because the rest of the repo is out of scope.

### 1. Route Family

The route layer currently mixes:

* `Facts`
* `Capabilities`
* `Decision`
* `Reason`
* `Contract`

This family needs one clear role split:

* `*Facts` = supporting snapshot inputs
* `*Decision` = selected route outcome
* `*Capabilities` = reusable boolean/eligibility surface
* `*Reason` = explanation enum for fallback or choice
* `*Contract` = proof/admission surface shared across layers

Likely audit targets:

* `LoadRouteCapabilityFacts`
* `RouteCapabilities`
* `LoadOrderRouteDecision`
* `LoadOrderRouteContract`
* `LoadOrderRouteReason`

The goal is not to rename these blindly, but to confirm the family is truly
parallel and fix any member whose role name no longer matches its usage.

### 2. Covering Family

The covering layer now has a much cleaner structure, but it still carries a
mix of:

* `Plan`
* `ExecutionPlan`
* `Context`
* `Facts`
* `Policy`

Likely audit targets:

* `CoveringProjectionContext`
* `PrimaryStoreCoveringAccessFacts`
* `CoveringProjectionSourceContext`
* `CoveringReadPlan`
* `CoveringReadExecutionPlan`
* `CoveringProjectionFieldSourcePolicy`

This family should confirm:

* which items are local walk bundles
* which items are frozen snapshots
* which items are phase outputs

### 3. Order / Grouped Structural Family

The order family mixes:

* `Contract`
* `Match`
* `Shape`
* `Analysis`

Likely audit targets:

* `DeterministicSecondaryOrderContract`
* `DeterministicSecondaryIndexOrderMatch`
* `GroupedIndexOrderContract`
* `GroupedIndexOrderMatch`
* `GroupedCanonicalOrderShape`
* `GroupedOrderExprAnalysis`

This family should standardize:

* `*OrderContract` for proof surfaces
* `*OrderMatch` for classification results
* `*Shape` for compact structural enums
* `*Analysis` for richer descriptive structs

### 4. Helper Verb Family

Several modules still blur:

* `resolve_*`
* `derive_*`
* `prepare_*`
* `analyze_*`
* `classify_*`

Likely audit zones:

* SQL lowering
* route planning
* covering helpers
* execution page/materialization helpers

This family should ensure new helpers say what class of work they actually do.

### 5. Prepared Execution Family

The prepared execution layer contains a dense cluster of names around:

* `Core`
* `Shared`
* `Parts`
* `Plan`
* `RuntimeParts`

Likely audit targets:

* `PreparedExecutionPlanCoreShared`
* `PreparedExecutionPlanCore`
* `PreparedScalarRuntimeParts`
* `PreparedGroupedRuntimeParts`
* `PreparedAccessPlanParts`
* `SharedPreparedProjectionRuntimeParts`
* `SharedPreparedExecutionPlan`
* `PreparedLoadPlan`
* `PreparedAggregatePlan`

This family should answer:

* when is something a `*Plan` versus `*Parts` payload?
* when does `Shared*` mean generic-free shared shell versus shared cloned
  residents?
* whether `Core` is a meaningful family name here or just historical layering

The likely end state is not one rename rule, but one clearer split between:

* prepared executable payloads
* extracted runtime handoff bundles
* shared generic-free shells

### 6. Continuation / Cursor Family

The continuation layer currently mixes:

* `Contract`
* `Window`
* `Projection`
* `Action`
* `Plan`
* `Cursor`

Likely audit targets:

* `PlannedContinuationContract`
* `GroupedContinuationWindow`
* `GroupedWindowProjection`
* `GroupedCursorAction`
* `ScalarAccessWindowPlan`
* `PlannedCursor`
* `GroupedPlannedCursor`

This family should answer:

* whether `ScalarAccessWindowPlan` is truly a `Plan` or a projected contract
* whether `GroupedWindowProjection` should be a more explicit continuation
  projection/facts name
* whether planned cursor names remain parallel with the continuation contract
  family

### 7. Lowered SQL Shape Family

The SQL lowering layer contains multiple adjacent names built around:

* `Shape`
* `Analysis`
* `OrderTerm`
* `DescriptorShape`

Likely audit targets:

* `LoweredSelectShape`
* `LoweredBaseQueryShape`
* `LoweredExprAnalysis`
* `LoweredSqlOrderTerm`
* `PreparedSqlScalarAggregateDescriptorShape`
* `LoweredSqlAggregateShape`

This family should answer:

* when `Shape` means a structural lowering payload
* when `Analysis` means a descriptive fact bundle
* whether some `*Shape` items are really closer to lowered plans/contracts
* whether descriptor-shape naming should stay specialized or be folded into
  the wider structural rubric

### 8. Codebase-Wide Sweep Family

After the first rename batches land, `0.127` should continue through the rest
of the repo under the same rubric.

That broader sweep should include:

* session-layer contracts and visibility surfaces
* executor pipeline contracts, payloads, and contexts
* access planning and capability helpers
* intent and query-shaping names
* explain and observability payload families
* mutation, save, and delete preparation helpers
* aggregate runtime payload and boundary names

The key rule is that "not first" does not mean "not covered."

These are not the entire slice.

They are the first bounded batch that should establish the naming standard for

### Covering

Possible audit questions:

* is `CoveringProjectionContext` really a local context, or is it closer to
  `CoveringProjectionFacts`?
* does `prepare_covering_index_projection_plan(...)` describe a reusable
  preparation step, or should it become `derive_covering_index_projection_facts(...)`
  or `prepare_covering_index_projection_context(...)`?

### Route

Possible audit questions:

* does `LoadOrderRouteContract` truly represent a contract, or is it closer to
  a route mode/shape enum?
* is `RouteCapabilities` the right plural surface, or are some fields really
  route facts or decision outputs?

### Helper Verbs

Possible audit questions:

* helpers that currently say `resolve_*` but actually compute new composite
  outputs may need `derive_*`
* helpers that currently say `prepare_*` but actually return a final result may
  need `derive_*`
* helpers that currently say `classify_*` but return a rich struct may need
  `analyze_*`

### Prepared Execution

Possible audit questions:

* is `SharedPreparedExecutionPlan` better understood as a shared prepared shell
  or a shared prepared core?
* are `PreparedScalarRuntimeParts`, `PreparedGroupedRuntimeParts`, and
  `PreparedAccessPlanParts` one coherent `*RuntimeParts` / `*PlanParts` family,
  or are they mixing extraction payloads with phase outputs?
* does `PreparedExecutionPlanCoreShared` communicate a meaningful role, or is
  it historical implementation layering that should become a clearer
  `SharedPreparedExecutionCore`-style family?

### Continuation / Cursor

Possible audit questions:

* is `ScalarAccessWindowPlan` really a plan, or is it closer to a continuation
  window contract/projection?
* is `GroupedWindowProjection` too generic relative to
  `GroupedContinuationWindow`?
* do `PlannedCursor` and `GroupedPlannedCursor` still line up cleanly with the
  surrounding continuation-contract naming?

### Lowered SQL

Possible audit questions:

* are `LoweredSelectShape` and `LoweredBaseQueryShape` the right `Shape`
  family, or are they functioning more like lowered query payloads?
* does `PreparedSqlScalarAggregateDescriptorShape` communicate a stable family,
  or is it layering too many historical qualifiers into one name?
* are `LoweredExprAnalysis` and `LoweredSqlAggregateShape` a clean
  `Analysis`/`Shape` contrast that should become the model for the rest of SQL
  lowering?

## What `0.127` Should Not Do

`0.127` should not:

* rename stable names just because they are slightly awkward
* mix large behavior changes into naming work
* optimize for theoretical perfection over consistency
* churn low-value leaf helpers with no family-level payoff
* flatten intentionally meaningful distinctions between `Plan`, `Contract`,
  `Facts`, `Context`, `Decision`, `Shape`, and `Analysis`

---

## Migration Strategy

1. Define the naming rubric first.
2. Audit the highest-value naming families.
3. Rename the first bounded batch that establishes the family standard.
4. Continue the same rubric through the remaining codebase families.
5. Rename only where the new name removes real ambiguity.
6. Update call sites and docs in the same slice.
7. Stop when the major repo-wide families are internally consistent.

This slice should proceed by family, not by random one-off rename.

---

## Completion Criteria

`0.127` is complete when:

* the main architectural type families have one consistent naming rubric
* the route family is internally consistent
* the covering family is internally consistent
* order/grouped structural names are internally consistent
* prepared execution names are internally consistent
* continuation and cursor names are internally consistent
* lowered SQL naming families are internally consistent
* the remaining codebase families have been audited under the same rubric
* helper verb usage follows the adopted meaning boundaries
* no major public or internal family still mixes historical near-synonyms
  without reason

---

## Expected Payoff

If `0.127` succeeds, the codebase gains:

* faster architectural readability
* clearer ownership boundaries
* less historical drift after cleanup refactors
* less temptation to introduce new near-synonym families
* a naming rubric that future slices can extend instead of reinventing

This is a small but durable maintenance slice: it does not change what the
system does, but it makes the architecture easier to read, review, and evolve.

### FE Impact Rule

For `0.127`, any rename that touches these public families must be treated as a
frontend-affecting change even if the same rename family is mostly internal
elsewhere.

That does not mean "leave the old name alone."

It means:

* rename it if the new name is materially better
* make the rename explicit and intentional
* update the public facade re-exports in the same slice
* update public docs and examples in the same slice
* reflect the rename clearly in the changelog
* add FE/client migration notes when the rename changes named public types
  rather than only internal helpers
