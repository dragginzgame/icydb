# 0.45 - Feature Cleanup

## Status

Draft  
Target: `0.45`  
Dependency: `0.44` optimization closure baseline (complete)

---

## 1. Objective

`0.45` is a cleanup release line focused on quality, consistency, and
maintainability of previously shipped features.

The goal is to ensure existing behavior is:

- correct
- aligned across planner/access/executor/diagnostics layers
- implemented using current repository best practices
- covered by explicit tests and clear diagnostics

Primary `0.45` themes:

- API consistency
- schema correctness
- internal maintainability

`0.45` is not a feature-expansion release.

---

## 2. Non-Goals

- no new query terminals
- no new access-path variants
- no new planner semantic families
- no new executor routing branches
- no backward-compatibility shims
- no broad architecture redesign unrelated to cleanup scope

---

## 3. Cleanup Principles

### 3.1 No Half-Finished Flows

Every existing feature touched in `0.45` must end in exactly one explicit
state:

- cleanup completed and tested
- explicitly deferred with owner and rationale
- explicitly removed with migration note (if applicable)

No implicit "partial" state is allowed at release close.

### 3.2 Layer Authority Alignment

Authority boundaries remain:

`intent -> query/plan -> access -> executor -> index/storage`

- planner owns semantics
- access owns shape/capability
- executor owns runtime mechanics
- diagnostics own observability surfaces

Cleanup work must remove policy duplication across boundaries.

### 3.3 Public Surface Discipline

Prefer narrow visibility:

- `pub(crate)` over `pub` when cross-crate access is not required
- accessors over public fields
- module-root re-exports over deep-path coupling

Public API growth in `0.45` is treated as a regression unless justified.

### 3.4 No Compatibility Shims

Legacy forwarding helpers, duplicate aliases, and compatibility-only fields are
cleanup targets.

New work should consume canonical contracts directly.

### 3.5 Observability and Explain Consistency

If runtime behavior changes, EXPLAIN/trace/test surfaces must remain aligned.

EXPLAIN DTO construction remains immutable (`ExplainPlan::new(...)` style).

## 5. Workstreams

### 5.1 Visibility and API Surface Cleanup

- audit `pub` items added during recent feature lines
- reduce visibility where cross-crate use is not required
- keep public struct fields minimal; prefer accessors
- document intentional public exposure when retained

### 5.2 Shim and Compatibility Surface Removal

- remove compatibility-only wrappers and duplicate forwarding helpers
- remove duplicate route/executor contract fields where one canonical source exists
- add structural tests that block shim reintroduction

### 5.3 Layer Contract Alignment

- remove repeated policy derivation across layers
- keep planner decisions planner-owned and consumed downstream
- ensure route/access/executor each consume, not recompute, upstream policy

### 5.4 Explain/Trace and Runtime Parity

- verify explain descriptors reflect shipped runtime behavior
- verify trace optimization markers remain explicit and stable
- add parity tests for cleaned paths (not only happy path)

### 5.5 Test and Documentation Hardening

- add regression coverage for each cleanup change
- keep changelog entries clear by audience:
  - root `CHANGELOG.md`: junior-friendly impact summary
  - `docs/changelog/0.45.md`: implementation details for maintainers
- update status tracker so no cleanup row remains implicitly open

### 5.6 Relation Identity Naming Convention Restoration

Recent schema changes experimented with relation field names without `_id` and
`_ids` suffixes.

This introduced avoidable ambiguity and API drift:

- ambiguity between identity references and loaded entities
- inconsistent constructor and accessor naming
- frontend/API confusion around relation semantics
- schema meaning drifting from storage representation

IcyDB relation fields store identity references, not loaded entity objects.
Naming must therefore encode identity semantics directly.

Canonical convention:

- singular relation fields end in `_id`:
  - `user_id`
  - `project_id`
  - `owner_id`
  - `parent_id`
- plural relation fields end in `_ids`:
  - `order_ids`
  - `member_ids`
  - `tag_ids`

Schema enforcement:

- `item(rel = "...")` fields must end in `_id`
- `many item(rel = "...")` fields must end in `_ids`
- violations must fail at compile time in schema macros

Example (invalid):

`field(ident = "user", value(item(rel = "User", prim = "Ulid")))`

Expected:

`field(ident = "user_id", value(item(rel = "User", prim = "Ulid")))`

Generated endpoint/codegen alignment:

- generation follows canonical field names directly (no semantic aliasing)
- singular helper style:
  - `user_id()`
  - `set_user_id(...)`
- plural helper style:
  - `order_ids()`
  - `add_order_id(...)`
  - `remove_order_id(...)`

This keeps schema, API surface, and storage representation aligned.

### 5.7 Temporal Type Return Normalization

Current query responses may expose raw numeric temporal storage primitives
(for example `u64` timestamps), which leaks storage encoding into client APIs.

Canonical return model:

- return semantic temporal types at query/runtime API boundaries:
  - `Timestamp`
  - `Date`
  - `Duration`
- do not expose raw numeric storage counters as the primary client-facing type

Storage vs return separation:

- storage encoding remains unchanged
  - timestamp remains integer-backed in storage
  - duration remains integer-backed in storage
- returned/query-facing values convert to semantic temporal types before
  crossing client API boundaries

Scope in `0.45`:

- define and lock canonical temporal return type contracts
- migrate executor/query result surfaces to those canonical temporal types
- align explain/trace/test surfaces where temporal values are projected
- ensure serialized representations are stable for client consumption

Storage encodings must remain stable.

### 5.8 Error Surface Normalization

Current errors can surface inconsistently across:

- query
- planner
- executor
- index
- storage

`0.45` normalizes public error boundaries:

- planner failures surface as `QueryError`
- executor failures surface as `ExecutionError`
- storage/index failures surface as `InternalError`

Enforcement:

- no executor errors may leak planner semantics
- no storage/index raw error types may leak to public query surfaces
- mapping rules are covered with boundary regression tests

### 5.9 Access-Shape Coordination Cleanup

Access-shape coordination currently spans multiple runtime contracts, creating a
high-change coordination surface.

`0.45` introduces a canonical executor-facing route descriptor
(`ExecutionRouteShape`) consumed downstream.

Executor consumers should read canonical shape contracts instead of repeatedly
matching across separate shape/signature/continuation variants.

### 5.10 Fast-Path Telemetry Unification

Fast-path diagnostics currently include path-specific test counters and trace
markers that can drift in naming and intent.

`0.45` unifies fast-path telemetry under one internal optimization taxonomy
shared by:

- runtime trace markers
- test diagnostics/counters

Target taxonomy remains explicit (for example PK count, bytes fast path,
covering projection, top-N seek), with one authority for naming.

### 5.11 Macro Validation Hardening

Beyond `_id`/`_ids` naming enforcement, `0.45` hardens compile-time schema
validation in macros:

- relation type compatibility checks
- index field existence checks
- plural relation shape correctness checks

Violations must fail at compile time with explicit diagnostics.

### 5.11.1 Planner Validation Coverage

Planner validation rules must be explicitly covered by regression tests.

This includes rules governing:

- grouped query legality
- cursor pagination requirements
- ordering constraints for mutation operations
- unsupported feature detection

Tests must assert invalid queries fail during planning rather than reaching
execution with undefined behavior.

### 5.12 Canonical Query Fingerprint Determinism

`0.45` guarantees deterministic fingerprints for logically equivalent queries.

Required stability coverage includes:

- predicate reordering equivalence
- map-ordering equivalence
- equivalent literal coercion/normalization cases

Goal: same logical query must produce the same fingerprint.

### 5.13 Cursor Token Stability Audit

Continuation contracts remain a high-risk core subsystem.

`0.45` locks cursor invariants across:

- signature construction and validation
- decoding and error classification
- resume-window and ordering invariants

Coverage must include fast paths, limit pushdown paths, and ordering variants.

### 5.14 EXPLAIN Output Stability

`0.45` adds stability tests for key explain surfaces, including snapshot-style
coverage for:

- `ExplainExecutionNodeDescriptor`
- `ExplainAggregateTerminalPlan`

Target outcomes:

- prevent unintentional descriptor drift
- protect downstream tooling and debugging workflows

### 5.15 Module Gravity-Well Splits

Large concentration files are cleanup targets when they impede review and
ownership clarity.

`0.45` allows behavior-preserving splits of oversized modules into focused
submodules with clear authority boundaries.

No semantic/runtime behavior change is in scope for these splits.

### 5.16 Storage Encoding Consistency Audit

`0.45` audits encoding contracts across value hashing, predicate encoding, and
index key encoding to enforce:

- canonical map ordering everywhere
- consistent value hashing semantics
- no duplicate encode logic paths with diverging behavior

This is a consistency cleanup; storage format stability must be preserved.

### 5.17 Test Matrix Rationalization

`0.45` rationalizes large matrix-style tests into consistent domain grouping and
naming conventions across query/route/pagination coverage.

Goal:

- keep matrix coverage breadth
- reduce fragmentation and naming drift
- improve maintainability and discoverability
- enforce explicit ordering assertions for order-guaranteed query shapes
- avoid implicit ordering assumptions where ordering is undefined
- require explicit ordering assertions for range scans, ordered index scans, and
  cursor paging coverage

### 5.18 Serialization Contract Stabilization

Query responses, explain outputs, and DTO contracts must serialize in a stable
and predictable form for client consumption.

`0.45` stabilizes serialization contracts for public-facing types.

Scope includes:

- explicit serde representation attributes on applicable DTO enums (for example
  `#[serde(rename_all = "snake_case")]` and `#[serde(tag = "type")]`)
- preventing accidental field renaming in serialized contracts
- ensuring new internal fields do not serialize unintentionally
- ensuring timestamp/date/duration types serialize consistently

Goals:

- prevent accidental wire-format drift
- maintain compatibility for frontend and client tooling
- keep serialization stable across patch releases

### 5.19 Executor/Planner Overlap Cleanup

`0.45` includes a dedicated companion cleanup doc for planner/executor overlap:

- `docs/design/0.45-executor-planner-overlap.md`

Scope in this line:

- remove semantic overlap where planner authority already exists
- keep executor runtime focused on consuming canonical planner/route outputs
- add anti-regression coverage that blocks overlap reintroduction

## 6. Safety Constraints

Cleanup must preserve:

- result correctness
- ordering guarantees
- pagination and cursor semantics
- stale-key and missing-row behavior
- error class/origin behavior at public boundaries

Every cleanup path must retain deterministic fallback behavior where fallback
exists.

Cleanup must also preserve:

- public error contract stability at layer boundaries
- cursor token replay stability across equivalent query shapes

## 7. Implementation Slices (`0.45.x`)

Patch labels remain patch-agnostic.

- Slice A: visibility/public-surface audit and reduction
- Slice B: error-surface and schema-macro normalization
- Slice C: access-shape/telemetry/fingerprint contract convergence
- Slice D: cursor + explain stability lock and regression expansion
- Slice E: module/encoding/test-matrix maintainability cleanup
- Slice F: final closure pass and release documentation

Recommended order: `A -> B -> C -> D -> E -> F`.

---

## 8. Complexity Guardrails

During `0.45`:

- no new query-surface features
- no new access-path variants
- no new execution terminals
- no new executor routing branches
- no new policy duplication between layers
- schema macro enforcement may add compile-time validation but must not expand
  runtime execution paths

If cleanup requires violating a guardrail, scope must be explicitly split into a
future feature line.

---

## 9. Exit Criteria

`0.45` closes when all conditions are true:

- cleanup matrix rows are either `complete` or explicitly `deferred` with owner+rationale
- no compatibility shim remains for touched areas
- no intentionally-public field remains without justification
- explain/trace outputs remain aligned with runtime behavior for touched paths
- regression coverage exists for each cleaned boundary
- error boundary mapping is explicit and enforced by tests
- canonical query fingerprint invariants are locked for normalization cases
- cursor token stability suite passes across fast-path and pushdown variants
- root and detailed changelog entries are updated for shipped cleanup work
