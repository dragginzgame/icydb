# 0.87 Addendum Design Doc

## Grouped Expression Evaluator Unification

## Document Status

* status: proposed
* release: follow-up after `0.87.0`
* scope: grouped post-aggregate evaluator cleanup
* primary areas:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/grouped.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/model.rs`

## Purpose

This addendum proposes unifying grouped projection evaluation and grouped
`HAVING` evaluation onto one shared slot-based compiled expression runtime.

Today, grouped post-aggregate execution has two parallel evaluation paths:

* grouped projection evaluation already compiles into
  `GroupedProjectionExpr` and evaluates over finalized grouped rows
* grouped `HAVING` evaluation still walks `GroupHavingExpr` directly in the
  aggregate runtime

That split is now the main grouped post-aggregate duplication seam.

The target end state is:

```text
GroupedOutputRow
  -> GroupedEvalContext
  -> CompiledGroupedExpr
  -> eval(...)
```

with:

* projection using the resulting `Value` directly
* `HAVING` enforcing boolean/truthy semantics over that same evaluated result

The goal is to reduce duplicate evaluation code, tighten semantics, lower wasm
size, and make later grouped `ORDER BY` / post-aggregate expression work easier
to extend.

---

## Problem Statement

Grouped projection and grouped `HAVING` now share most of the same expression
surface:

* grouped key access
* aggregate output access
* literals
* arithmetic
* function calls

But they still execute through different runtime paths:

* grouped projection compiles `Expr` into `GroupedProjectionExpr`
* grouped `HAVING` interprets `GroupHavingExpr` directly and uses its own
  recursive evaluator

That creates four problems:

1. duplicated logic
2. semantic drift risk between projection and `HAVING`
3. larger wasm / instruction footprint
4. harder follow-up work for grouped `ORDER BY` and future compiled expression
   caching

This is not just cleanup. It is an authority problem:

* projection owns one post-aggregate evaluator
* `HAVING` owns another

There should be one.

---

## Design Goal

Introduce one shared grouped compiled-expression model and one evaluator for
post-aggregate grouped rows.

That means:

* one grouped evaluation context
* one slot model
* one compiled grouped expression IR
* one `eval(...) -> Value`

`HAVING` should stop owning a dedicated evaluator.

---

## Non-Goals

This addendum does not aim to:

* widen grouped expression semantics
* add arbitrary grouped `ORDER BY` support
* redesign grouped planner validation
* add full expression compilation caching in the same slice
* change grouped continuation encoding
* introduce implicit full-sort grouped execution

This is an evaluator unification slice, not a behavior-widening slice.

---

## Target Runtime Model

### Shared evaluation context

Grouped post-aggregate evaluation should read from one explicit runtime input:

```rust
struct GroupedEvalContext<'a> {
    group_keys: &'a [Value],
    aggregates: &'a [Value],
}
```

This makes the grouped execution boundary explicit and keeps runtime evaluation
 independent of field names or aggregate-expression matching.

### Shared slot model

Compiled grouped expressions should resolve leaves onto slots:

```rust
enum GroupedSlot {
    GroupKey(usize),
    Aggregate(usize),
}
```

Runtime evaluation must be slot-only.

No field-name or aggregate-symbol resolution should happen in the hot path.

### Shared compiled IR

Grouped projection and `HAVING` should compile into one shared grouped
expression form, conceptually:

```rust
enum CompiledGroupedExpr {
    Slot(GroupedSlot),
    Literal(Value),
    FunctionCall { ... },
    Unary { ... },
    Binary { ... },
    Compare { ... },
    And(Vec<Self>),
}
```

Exact shape can vary, but the core rule is:

* one compiled grouped expression type
* no separate `GroupedProjectionExpr` vs `GroupHavingExpr` runtime evaluator

### Shared evaluator

Runtime execution should become:

```rust
fn eval(expr: &CompiledGroupedExpr, ctx: &GroupedEvalContext<'_>) -> Value
```

Projection then does:

```rust
let value = eval(projection_expr, ctx);
```

`HAVING` does:

```rust
let value = eval(having_expr, ctx);
if !truthy(value) { ... }
```

---

## Boolean Boundary

`HAVING` must still enforce boolean semantics explicitly.

The runtime boundary should not silently reinterpret arbitrary values as truthy
unless validation has already proven that grouped `HAVING` expressions always
produce boolean-compatible outputs.

Acceptable shapes:

* validate boolean result at planner/type-check time, then treat runtime
  boolean mismatch as internal error
* or use one explicit runtime truthiness boundary:

```rust
fn truthy(v: Value) -> bool
```

But this must be deliberate and single-owned.

---

## Migration Strategy

This should not be attempted as one giant slice.

### Slice 1: shared grouped evaluator core

Introduce:

* `GroupedEvalContext`
* `GroupedSlot`
* shared grouped compiled IR
* shared `eval(...) -> Value`

Then migrate grouped projection onto that shared evaluator first.

This keeps the external grouped `HAVING` behavior unchanged while proving the
engine extraction boundary.

### Slice 2: grouped `HAVING` migration

Compile `GroupHavingExpr` into the shared grouped compiled IR and replace the
dedicated grouped `HAVING` runtime evaluator in:

* `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/mod.rs`

At that point:

* `eval_group_having_expr(...)` should disappear
* `eval_group_having_value_expr(...)` should disappear
* grouped runtime should use the shared evaluator plus one explicit boolean
  boundary

### Slice 3: optional cleanup / reuse follow-through

After projection and `HAVING` share the same evaluator, follow-up work can
consider:

* compiled expression caching
* grouped `ORDER BY` reuse
* constant folding / specialization

This is intentionally not part of the first unification cut.

---

## Why This Should Be Split

Trying to do all of this in one slice would simultaneously change:

* grouped compiled expression IR
* grouped projection compilation
* grouped `HAVING` compilation
* grouped runtime evaluation
* boolean/truthiness enforcement
* planner/runtime tests across projection, `HAVING`, explain, and fingerprint

That is too much for one clean, reviewable slice.

The high-value boundary is:

1. extract the shared evaluator
2. then migrate `HAVING`

That keeps the change reviewable and fail-closed.

---

## Expected Wins

If completed, this unification should give:

* less duplicated evaluator code
* lower instruction count on grouped post-aggregate paths
* smaller wasm
* fewer semantic drift bugs between projection and `HAVING`
* cleaner future path for grouped `ORDER BY` and expression caching

---

## Acceptance Criteria

### After Slice 1

* grouped projection evaluates through the shared grouped evaluator
* no field-name or aggregate-symbol resolution happens during grouped runtime
  evaluation
* grouped projection behavior is unchanged

### After Slice 2

* grouped `HAVING` evaluates through the same shared grouped evaluator
* no dedicated grouped `HAVING` evaluator remains
* one explicit boolean/truthiness boundary exists
* grouped projection and grouped `HAVING` share identical post-aggregate
  arithmetic/function/null semantics

### Full closeout

* one grouped compiled-expression evaluator exists
* no parallel grouped projection vs `HAVING` runtime evaluators remain
* grouped tests are green
