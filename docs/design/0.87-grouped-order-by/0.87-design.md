# 0.87 Grouped ORDER BY Expression Admission and Boundedness

## Thesis

`0.87` should define a **strict, provable admission model** for grouped `ORDER BY` expressions.

The goal is not to allow arbitrary grouped expressions in `ORDER BY`.

The real target is:

> admit only those grouped `ORDER BY` expressions whose ordering semantics can be proven to preserve the current grouped boundedness and continuation guarantees.

That means:

* keep grouped execution bounded
* keep grouped continuation/cursor correctness intact
* keep grouped `ORDER BY` fail-closed unless order preservation is provable
* do **not** introduce implicit full materialization or full sort behavior
* do **not** allow aggregate-valued ordering just because `0.85` and `0.86` made those expressions evaluable elsewhere

`0.87` is not another expression-widening slice.

It is an **order-proof slice**.

---

## Problem statement

After `0.85` and `0.86`, grouped SQL supports post-aggregate expressions in:

* projection
* `HAVING`

So the engine can now evaluate grouped expressions like:

```sql
ROUND(AVG(strength), 4)
```

```sql
COUNT(*) + 1
```

```sql
MAX(level) - MIN(level)
```

But grouped `ORDER BY` must remain narrower.

That is because grouped `ORDER BY` is not just an output expression surface. It is part of the execution contract:

* it affects row order
* it affects pagination
* it affects continuation correctness
* it affects whether execution can remain bounded

So there is now an asymmetry:

* grouped projection can evaluate rich post-aggregate expressions
* grouped `HAVING` can evaluate rich post-aggregate boolean expressions
* grouped `ORDER BY` cannot safely admit those same expressions without a proof story

This is now the main remaining grouped SQL seam.

---

## Architectural position

### `ORDER BY` is not “just another expression surface”

`0.85` and `0.86` widened **evaluation** surfaces.

`0.87` is about **ordering semantics**.

That distinction is essential.

The question for grouped `ORDER BY` is not:

> can the engine evaluate this expression?

The question is:

> can the engine guarantee that ordering by this expression preserves the grouped execution contract without introducing unbounded sort behavior or breaking continuation semantics?

That is a different class of problem.

---

### Preserve current grouped boundedness guarantees

`0.87` should preserve the current grouped `ORDER BY` contract:

* grouped `ORDER BY` still requires `LIMIT`
* grouped execution must not gain implicit full-sort behavior
* continuation/cursor correctness must remain intact
* grouped ordering must stay tied to canonical grouped-key order unless a broader expression can be proven to preserve it

This slice should not widen execution strategy.

It should only widen **admission** where proof is straightforward and sound.

---

### Do not let expression evaluability imply orderability

After `0.85` and `0.86`, the engine may be able to evaluate:

```sql
AVG(strength) * 2
```

That does **not** mean the engine can safely admit:

```sql
ORDER BY AVG(strength) * 2
```

Likewise, the engine may be able to evaluate:

```sql
ROUND(AVG(strength), 2)
```

That does **not** mean it is safe to sort grouped rows by it under the current boundedness model.

`0.87` must make this separation explicit.

---

## Scope

### In scope

* formalize grouped `ORDER BY` expression admission as an **order-preservation proof**
* admit a narrow set of grouped `ORDER BY` expressions only when they provably preserve canonical grouped-key order
* keep execution strategy unchanged
* keep unsupported grouped `ORDER BY` expressions fail-closed
* define the classifier/rules that distinguish:

  * safe order-preserving expressions
  * unsupported expressions
  * explicitly rejected expressions that would require full-sort semantics

### Out of scope

* arbitrary grouped `ORDER BY` expressions
* ordering by aggregate outputs
* ordering by computed aggregate expressions
* bounded top-k sort over grouped aggregates
* relaxing the grouped `LIMIT` requirement
* changing continuation encoding
* changing grouping/aggregation mechanics
* joins
* window functions
* subqueries

---

## Current state

Before `0.87`, the grouped path already has a narrow `ORDER BY` admission model tied to canonical grouped-key order.

That is the right baseline.

Grouped order admission today is deliberately conservative because grouped order is part of the execution-proof layer, not just an expression layer.

`0.87` should preserve that philosophy, but make the rule more explicit and slightly more capable where proof is clean.

---

## Design goal

### Introduce an explicit grouped order-preservation model

The engine should gain an explicit classifier for grouped `ORDER BY` expressions with this shape:

1. determine what grouped inputs the expression depends on
2. determine whether the expression preserves canonical grouped-key order
3. admit only expressions that are provably safe
4. reject everything else fail-closed

Conceptually:

```text
ORDER BY expr
   -> classify(expr)
   -> preserves canonical grouped-key order?
      yes -> allow
      no  -> reject
```

This keeps grouped ordering an execution-proof problem, not an open-ended expression problem.

---

## Proposed model

### Order-preserving grouped `ORDER BY` expressions

For `0.87`, grouped `ORDER BY` expressions should be admitted only if they satisfy all of the following:

1. they depend only on grouped key inputs
2. they do not depend on aggregate outputs
3. they preserve canonical grouped-key order
4. they do not require materializing and sorting grouped rows independently of grouped-key order

This implies a very small admitted family in `v1`.

---

### Expression classes

#### Class 1: exact grouped-key reference

These should remain allowed:

```sql
ORDER BY class_name
```

```sql
ORDER BY class_name, level
```

They are the canonical baseline and preserve grouped-key order by definition.

---

#### Class 2: grouped-key plus constant offset

These may be admitted if the type semantics are already well-defined and monotonic:

```sql
ORDER BY level + 1
```

```sql
ORDER BY level - 2
```

These preserve order because they are monotonic affine transforms of one grouped key.

This is likely the largest safe widening worth taking in `0.87`.

---

#### Class 3: grouped-key scalar wrappers with unproven order semantics

These should remain rejected in `0.87` unless proof is explicit and type-local:

```sql
ORDER BY ROUND(level, 2)
```

```sql
ORDER BY LOWER(class_name)
```

```sql
ORDER BY ABS(level)
```

Even when evaluable, they do not automatically preserve canonical order.

`0.87` should not admit them by accident.

---

#### Class 4: aggregate output ordering

These must remain rejected:

```sql
ORDER BY AVG(strength)
```

```sql
ORDER BY MAX(level) - MIN(level)
```

```sql
ORDER BY ROUND(AVG(strength), 2)
```

These are exactly the expression families that would push grouped order into a different execution regime, likely requiring full materialization and bounded-top-k or full-sort semantics.

That is a later slice.

---

## Admission rule

### Core rule

A grouped `ORDER BY` expression is valid in `0.87` only if it is a provably order-preserving transform of the canonical grouped-key prefix order.

That means:

* grouped-key leaves only
* one grouped-key authority path at a time unless exact prefix semantics remain obvious
* no aggregate leaves
* no wrappers/functions unless explicitly proven monotonic and stable under the type contract
* no expression families that collapse distinct key values or reorder them unpredictably

---

### Recommended `v1` admission family

`0.87 v1` should support:

* exact grouped-key references
* grouped-key plus integer/decimal constant offset
* grouped-key minus integer/decimal constant offset

Recommendation:

Start there and stop there.

That gives a real but bounded widening without dragging in string collation, numeric non-monotonic wrappers, or aggregate-order semantics.

---

## Validation rules

### Allowed leaves

A grouped `ORDER BY` expression leaf must be:

* a grouped key field
* a literal constant

No aggregate output leaves are allowed in `0.87`.

### Allowed forms in `v1`

A grouped `ORDER BY` expression may be:

* `group_key`
* `group_key + constant`
* `group_key - constant`

where type compatibility is valid and monotonicity is preserved.

### Still invalid in `0.87`

These should fail closed:

```sql
SELECT class_name, AVG(strength)
FROM character
GROUP BY class_name
ORDER BY AVG(strength)
LIMIT 100;
```

```sql
SELECT class_name, MAX(level), MIN(level)
FROM character
GROUP BY class_name
ORDER BY MAX(level) - MIN(level)
LIMIT 100;
```

```sql
SELECT class_name, COUNT(*)
FROM character
GROUP BY class_name
ORDER BY COUNT(*) + 1
LIMIT 100;
```

```sql
SELECT class_name, COUNT(*)
FROM character
GROUP BY class_name
ORDER BY LOWER(class_name)
LIMIT 100;
```

```sql
SELECT level, COUNT(*)
FROM character
GROUP BY level
ORDER BY ABS(level)
LIMIT 100;
```

---

## Parser changes

### Current problem

After `0.85` and `0.86`, the parser may be able to represent broader grouped expressions generally, but grouped `ORDER BY` cannot simply inherit that support.

### Target

Allow grouped `ORDER BY` to parse as an expression surface if needed, but keep validation/lowering fail-closed around the order-preserving subset.

The rule becomes:

> grouped `ORDER BY` may parse broadly, but admission is governed by the grouped order-preservation classifier.

This keeps parsing flexible while preserving execution safety.

---

## Lowering changes

### Current problem

Grouped order admission is likely currently embedded in special-case structural checks.

### Target

Introduce a grouped order classifier pass:

1. inspect each grouped `ORDER BY` expression
2. determine whether it is:

   * canonical grouped key order
   * order-preserving affine transform of grouped key order
   * unsupported / unsafe
3. lower only admitted expressions into grouped ordering specs

A useful internal form:

```text
GroupedOrderExprClass {
  CanonicalGroupKey,
  GroupKeyPlusConstant,
  Unsupported,
}
```

or a richer equivalent that directly captures proof status.

The important part is not the enum shape; it is that the proof becomes explicit and testable.

---

## Executor changes

### Execution model

Grouped execution should remain unchanged.

`0.87` should not add:

* a grouped sort buffer
* a top-k heap over aggregate-computed order expressions
* a second ordering strategy
* a full materialization path

Execution remains:

1. scan source rows
2. build/accumulate grouped state
3. finalize aggregate outputs
4. evaluate grouped `HAVING`
5. evaluate projection expressions
6. emit rows under the existing grouped order contract

The only change is that slightly more `ORDER BY` expressions are admitted when they can be proven equivalent to canonical grouped-key ordering.

---

## Continuation and cursor semantics

This is the core reason `0.87` must stay narrow.

Grouped continuation correctness depends on stable, canonical interpretation of ordering.

So `0.87` must preserve:

* the existing grouped cursor/continuation contract
* the existing grouped-key prefix discipline
* the existing notion of deterministic row progression

Any expression family that weakens those guarantees is out of scope.

That is why aggregate-based ordering is not part of `0.87`.

---

## Ordering direction

`0.87` should preserve whatever current grouped direction rules already exist.

If grouped ordering direction is currently constrained to the canonical grouped-key order contract, keep it that way.

Do not expand direction semantics as part of this slice.

If direction-sensitive order-preservation proof is needed later, that should be a separate review.

---

## Phased execution plan

### Phase A: formalize grouped order-proof model

* define the grouped `ORDER BY` admission classifier
* make order-preservation explicit in validation/lowering
* keep fail-closed defaults

### Phase B: admit exact grouped-key references under the new model

* preserve current behavior through the new classifier
* prove no behavioral change for the baseline path

### Phase C: widen to affine grouped-key transforms

* admit `group_key + constant`
* admit `group_key - constant`
* keep type families narrow and explicit

### Phase D: harden error/reporting path

* unsupported grouped `ORDER BY` expressions should fail clearly
* distinguish “expression evaluable but not order-admissible” from generic parse failure

---

## Guardrails

### Guardrail 1: no aggregate-valued grouped ordering

Do not allow:

```sql
ORDER BY AVG(strength)
```

or any computed form over aggregate outputs in this slice.

### Guardrail 2: no implicit bounded-sort fallback

Do not introduce:

* top-k grouped sort
* full grouped sort
* materialize-then-sort fallback

Those belong to a later design slice.

### Guardrail 3: no proof by evaluator reuse

An expression being evaluable does not make it safe for grouped ordering.

Do not derive order admission from general expression support.

### Guardrail 4: keep continuation semantics primary

If an expression’s ordering behavior is not trivially compatible with the current grouped continuation model, reject it.

### Guardrail 5: stay narrow on function families

Do not admit scalar wrapper functions unless monotonicity and stability are explicit, local, and easily testable.

---

## Acceptance criteria

### Functional

* grouped `ORDER BY` continues to accept canonical grouped-key references
* grouped `ORDER BY` newly accepts grouped-key plus/minus constant where type-compatible
* grouped aggregate ordering remains rejected
* non-monotonic or unproven grouped-key wrappers remain rejected

### Architectural

* grouped execution strategy remains unchanged
* no new sort/materialization path is introduced
* grouped order admission is driven by an explicit proof/classifier step

### Validation

* unsupported grouped order expressions fail cleanly
* aggregate leaves in grouped `ORDER BY` remain fail-closed
* grouped-key authority remains preserved

### Performance

* no additional grouped sort cost is introduced
* no additional full materialization path is introduced
* current grouped boundedness guarantees remain intact

---

## Follow-on

This slice intentionally leaves a larger future step open:

* bounded grouped sort over aggregate/computed order expressions

That would be the point where queries like:

```sql
ORDER BY AVG(strength) DESC LIMIT 10
```

could be considered under an explicit bounded top-k execution design.

That is not `0.87`.

## Follow-on

This slice intentionally leaves a larger future step open:

* bounded grouped sort over aggregate/computed order expressions

That would be the point where queries like:

```sql
ORDER BY AVG(strength) DESC LIMIT 10
```

could be considered under an explicit bounded top-k execution design.

That is not `0.87`.

## Thesis sentence

> **`0.87` should define and admit only a provably order-preserving subset of grouped `ORDER BY` expressions, preserving current grouped boundedness and continuation guarantees without introducing aggregate-based sort semantics or implicit full-materialization behavior.**
