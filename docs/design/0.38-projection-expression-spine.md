# 0.38 — Projection & Expression Spine

## Status

Draft — Architectural Design
Target: 0.38
Scope: Complete projection semantics and introduce a unified expression spine
Prerequisite: 0.37 aggregate builder convergence complete

---

# 1. Objective

Complete the SELECT surface by introducing:

* A unified expression model
* Fully deterministic projection semantics
* Projection-aware fingerprinting
* Expression-safe execution
* Explicit GROUP BY compatibility enforcement

0.38 finalizes projection without:

* Introducing relational joins
* Introducing window functions
* Introducing cost-based planning
* Expanding beyond single-entity algebra

---

# 2. Non-Goals (0.38)

The following are explicitly out of scope:

* Joins
* Subqueries
* Window functions
* Recursive queries
* Cost-based optimization
* Expression pushdown into storage
* Dynamic function registration
* Implicit expression evaluation in index layer

Projection expansion must not destabilize planner simplicity.

---

# 3. Current Gaps

Today:

* Structural projection exists.
* Aggregate builder exists.
* GROUP BY validation exists.
* HAVING exists.

Missing:

* Unified expression AST
* Expression type inference
* Projection fingerprint normalization
* Expression-aware continuation stability

# 4. Expression Model

## 4.1 Canonical Expression Enum

Introduce a single expression type:

```rust
enum Expr {
    Field(FieldId),
    Literal(Value),
    Unary { op: UnaryOp, expr: Box<Expr> },
    Binary { op: BinaryOp, left: Box<Expr>, right: Box<Expr> },
    Aggregate(AggregateExpr),
    Alias { expr: Box<Expr>, name: Alias },
}
```

Constraints:

* Tree structure only.
* No cyclic references.
* No runtime-evaluated closures.
* No storage-layer references.

---

## 4.2 Supported Expression Categories

0.38 supports:

### Arithmetic

* + (addition)
* - (subtraction)
* * (multiplication)
* / (division)
* - (unary negation)

### Boolean

* AND
* OR
* NOT

### Comparison

* =
* !=
* <
* <=
* >
* > =

### Literals

* Numeric
* Boolean
* String (if already supported)
* Null

No string/date function library in 0.38.

# 5. Type Inference

Introduce deterministic type resolution:

* Binary operations enforce compatible types.
* Numeric coercion follows unified decimal model.
* Null propagation rules defined.
* Aggregate result types explicitly defined.
* No implicit widening beyond schema-defined scale.

Type inference must be:

* Deterministic
* Planner-visible
* Fingerprint-stable

# 6. Projection Semantics

## 6.1 Structural Projection

* Explicit field list defines output shape.
* Order preserved as declared.
* Duplicate aliases disallowed.

## 6.2 Aggregate Projection

When GROUP BY is present:

* Non-aggregate expressions must reference grouped fields.
* Aggregate expressions allowed.
* Mixed grouped + aggregate expressions allowed.
* Validation occurs in planner stage.

Invalid cases rejected at plan construction.

---

## 6.3 Alias Semantics

* Aliases apply only to output naming.
* Alias does not affect fingerprint identity of underlying expression.
* Alias collisions disallowed.

---

# 7. Expression Evaluation Engine

Introduce a pure in-memory evaluator:

* Executes after row materialization.
* Does not access storage layer.
* Deterministic evaluation order.
* No dynamic dispatch via trait objects.

Evaluation occurs:

* After predicate filtering.
* After grouping (for aggregate expressions).
* Before ordering (for computed projection fields used in ORDER BY).

---

# 8. Planner Integration

## 8.1 Expression Compatibility Rules

Planner must:

* Validate GROUP BY compatibility.
* Validate aggregate placement.
* Detect storage-pushdown eligibility (structural only).
* Forbid expression-based pushdown.

Expression pushdown is explicitly disallowed in 0.38.

---

## 8.2 Capability Preservation

Planner remains rule-based.

Expressions:

* Do not introduce plan enumeration.
* Do not affect index scoring.
* Do not require cost estimation.

# 9. Fingerprinting

Projection shape must be included in plan fingerprint.

Fingerprint components:

* Entity
* Predicate
* Group spec
* Aggregate spec
* Projection expression tree (normalized)
* Order spec
* Limit spec

Expression fingerprinting rules:

* Canonical ordering of commutative operations.
* Literal normalization.
* Alias excluded from structural identity.
* Aggregate expressions fingerprint through AggregateExpr spine.

Continuation envelopes must include projection hash.

# 10. Continuation Stability

Continuation must encode:

* Plan fingerprint
* Ordering boundary
* Grouping state (if grouped)
* Projection shape hash

Any projection change invalidates continuation reuse.

# 11. Execution Model Interaction

Projection must not:

* Change streaming eligibility.
* Introduce hidden buffering.
* Modify grouping strategy.
* Alter downgrade semantics.

Computed expressions evaluated post-stream where necessary.

Streaming grouping remains unaffected.

# 13A. Single Projection Spine Hardening Spec

This section is the structural hardening contract for projection unification.
It is additive to the 0.38 goals above and is intended to be enforced by tests.

## Objective

Guarantee that projection semantics are:

* Constructed exactly once (planner-owned)
* Represented by a single canonical semantic structure
* Consumed mechanically by executor
* Fingerprinted semantically (no metadata coupling)
* Impossible to reconstruct in executor or fluent layers

Projection shape must have a single semantic authority.

## 1. Canonical Projection Model

**Invariant P1 — Single Canonical Projection Struct**

There must exist exactly one semantic projection representation, for example:

```rust
pub struct ProjectionSpec {
    pub fields: Vec<ProjectionField>,
}

pub enum ProjectionField {
    Scalar {
        expr: Expr,
        alias: Option<String>, // metadata only
    },
    Aggregate {
        kind: AggregateKind,
        target: Option<FieldId>,
        distinct: bool,
        alias: Option<String>, // metadata only
    },
}
```

No alternate structural projection enums may exist in executor.
If any executor-local projection enum exists, `P1` is violated.

## 2. Planner Ownership

**Invariant P2 — Planner Is the Only Projection Constructor**

`ProjectionSpec` (or equivalent) must only be constructed inside `db::query::plan`.

Disallowed construction sites:

* `executor/*`
* `fluent/*`
* aggregate adapters
* distinct adapters
* `load/mod.rs`
* grouped execution modules

All fluent terminals must route through planner shape-building helpers.
If projection struct construction occurs outside planner, `P2` is violated.

## 3. Executor Consumption Model

**Invariant P3 — Executor Is Shape Consumer Only**

Executor must:

* Iterate `ProjectionSpec.fields`
* Evaluate `Expr` or aggregate slot
* Materialize values

Executor must NOT:

* Construct projection shapes
* Transform grouped shape into projection shape
* Branch on projection structural variants (beyond direct evaluation mechanics)
* Pattern-match on fluent-specific projection enums

If executor matches over projection variants beyond evaluation mechanics, `P3` is violated.

## 4. DISTINCT and GROUP BY Integration

**Invariant P4 — DISTINCT and GROUP BY Must Normalize Into ProjectionSpec**

Global `DISTINCT` must be expressible via `ProjectionField` semantics.
`GROUP BY` + aggregates must result in `ProjectionSpec`.

There must be no:

* Executor-only `DISTINCT` shape
* Grouped-specific projection struct
* Alternate grouped result container for projection

If `DISTINCT` path constructs grouped builder shape outside planner, `P4` is violated.

## 5. Fingerprint Derivation

**Invariant P5 — Fingerprint Must Derive From ProjectionSpec Only**

Fingerprint logic must:

* Traverse `ProjectionSpec`
* Hash semantic fields only
* Exclude alias/explain/presentation metadata

Disallowed:

* Hashing based on fluent builder enums
* Hashing explain tags
* Hashing alias strings
* Hashing projection variant display labels

If fingerprint reads from any structure other than `ProjectionSpec` (or semantic equivalent), `P5` is violated.

## 6. HAVING and Aggregate Alignment

**Invariant P6 — HAVING References Canonical Aggregate Semantics**

`HAVING` evaluation must reference:

* Aggregate semantics defined in planner
* `ProjectionSpec` aggregate entries

Executor must not duplicate aggregate shape logic.
If `HAVING` evaluation re-derives aggregate structure in executor, `P6` is violated.

## 7. Fluent Layer Restrictions

**Invariant P7 — Fluent Terminals Are Intent-Only**

Fluent helpers must:

* Record projection intent
* Delegate to planner shape builder

Fluent helpers must NOT:

* Construct `ProjectionField` directly
* Construct `AggregateKind` directly
* Construct grouped builder structures
* Construct fingerprint-relevant structures

If fluent constructs semantic projection shape, `P7` is violated.

## 8. Structural Guards (No String Scans)

Required structural enforcement (no text-pattern scanning):

**Guard G1 — Projection Construction Boundary Is Type-Enforced**

`ProjectionSpec` and `ProjectionField` construction must be planner-owned by
module visibility and constructor placement.

Enforcement:

* constructors are non-public outside planner boundary
* compile-fail/visibility tests prove executor/fluent cannot construct canonical projection semantics directly

**Guard G2 — Executor Boundary Is Signature-Enforced**

Executor entry points accept planned semantic projection shape only.
Executor modules must not expose APIs that accept fluent/builder projection
shape inputs.

Enforcement:

* module boundary API tests for accepted types at executor entry points
* invariant tests that invalid shapes can only arrive via explicit planner bypass

**Guard G3 — Fingerprint Boundary Is Semantic-Input Enforced**

Aggregate/projection fingerprint helpers accept semantic shape inputs only and
do not consume presentation metadata.

Enforcement:

* function signatures for fingerprint hashing exclude alias/explain/display metadata inputs
* regression matrix proves alias/explain/presentation changes do not alter semantic fingerprint identity

## 9. Done Criteria

Projection spine is unified when:

* Exactly one projection semantic struct exists
* Only planner constructs it
* Executor consumes it without structural branching
* `DISTINCT` and `GROUP BY` normalize through it
* Fingerprint derives exclusively from it
* Structural guards (type/signature/invariant) enforce boundaries without string scans
* All projection-related tests pass

When these are true, projection duplication risk is eliminated.
