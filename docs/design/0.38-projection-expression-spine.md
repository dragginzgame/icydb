# 0.38 — Projection & Expression Spine

## Status

0.38.1: completed (projection hardening and slot unification)

0.38.2+: planned (canonical expression spine and projection semantics completion)

Scope: complete projection semantics and unified expression authority for single-entity queries.

Prerequisite satisfied: 0.37 aggregate builder convergence.

## 1. 0.38.1 — Projection Hardening (Completed)

0.38.1 completed structural projection authority hardening and slot unification.

Completed outcomes:

- Fluent field-target terminals dispatch through planner-resolved slots.
- Production fallback dispatch paths were removed from fluent terminal routing.
- String-based executor entrypoints for field-target terminals were removed from production query-layer routes.
- Planner remains the semantic authority for grouped aggregate and HAVING contracts.
- Executor-side semantic checks are invariant-only defenses for planner-bypassed invalid shapes.
- Aggregate fingerprinting is semantically isolated to aggregate kind, target field, and distinct modifier.
- Compile-fail boundary tests enforce private module and API visibility boundaries.

Explicit boundary statement:

- 0.38.1 completed structural projection authority hardening but did not introduce the unified expression AST.

## 2. Objective

Complete 0.38 by introducing a unified, planner-owned expression and projection spine that provides deterministic query semantics for:

- projection evaluation
- GROUP BY and HAVING compatibility enforcement
- fingerprint derivation
- continuation compatibility

The architecture must preserve planner simplicity and execution determinism while remaining single-entity only.

## 3. Non-Goals (0.38)

The following are explicitly out of scope:

- joins
- subqueries
- window functions
- recursive queries
- cost-based planning
- expression pushdown into storage or index layers
- dynamic function registration
- relational expansion beyond single-entity algebra

Expression spine work must not introduce relational features or optimizer complexity.

## 4. Scope Boundary

0.38 applies to query planning and executor projection semantics for single-entity algebra only.

Required scope:

- canonical expression model
- canonical projection semantic model
- planner-owned semantic construction
- executor consumption-only model
- semantic fingerprint and continuation stability

Out-of-scope mechanics remain unchanged unless required to enforce these semantic boundaries.

## 5. Canonical Expression Model (Target)

0.38 requires one canonical expression enum owned by planner/query semantics.

Canonical shape:

```rust
enum Expr {
    Field(FieldId),
    Literal(Value),
    Unary { op: UnaryOp, expr: Box<Expr> },
    Binary { op: BinaryOp, left: Box<Expr>, right: Box<Expr> },
    Aggregate(AggregateExpr),
    Alias { expr: Box<Expr>, name: Alias },
}
```

Expression constraints:

- tree structure only
- no cycles
- no runtime closures
- no trait-object expression dispatch
- no storage/index-layer references

Supported categories for 0.38:

- arithmetic: `+`, `-`, `*`, `/`, unary negation
- boolean: `AND`, `OR`, `NOT`
- comparison: `=`, `!=`, `<`, `<=`, `>`, `>=`
- literals: numeric, boolean, string (existing type support only), null
- aggregate expression embedding through canonical aggregate spine

No function-library expansion is part of 0.38.

## 6. Type Inference Contract

Type inference must be deterministic and planner-visible.

Required behavior:

- binary operation compatibility checks
- numeric coercion through unified decimal semantics
- explicit null propagation behavior
- aggregate output type definition per aggregate kind
- no implicit widening outside schema-defined numeric constraints

Type inference output is part of semantic identity and must remain fingerprint-stable.

## 7. Canonical Projection Semantics (Target)

0.38 requires one canonical projection semantic structure.

Canonical shape:

```rust
pub struct ProjectionSpec {
    pub fields: Vec<ProjectionField>,
}

pub enum ProjectionField {
    Scalar {
        expr: Expr,
        alias: Option<String>,
    },
    Aggregate {
        kind: AggregateKind,
        target: Option<FieldId>,
        distinct: bool,
        alias: Option<String>,
    },
}
```

Required semantics:

- field order is declaration order
- duplicate aliases are rejected
- alias is presentation metadata only
- alias does not affect semantic identity

## 8. Evaluation Model

0.38 requires a pure in-memory evaluation engine for expressions.

Required execution model:

- evaluates post-materialization only
- does not access storage/index layers
- deterministic evaluation order
- no trait-object dynamic evaluation

Required phase ordering:

- after filtering
- after grouping for aggregate expressions
- before ordering when computed projection expressions are order-referenced

## 9. GROUP BY and HAVING Compatibility Model (Target)

0.38 requires expression-based grouped compatibility validation.

Required rules:

- non-aggregate expressions in grouped projection must reference grouped fields only
- aggregate expressions are allowed
- mixed grouped-field and aggregate expressions are allowed
- nested aggregate expressions are disallowed
- invalid grouped expression shapes are rejected at planner construction time

HAVING must operate over canonical expression semantics and planner-owned aggregate identity.

## 10. Fingerprint and Continuation Model (Target)

Projection semantics are part of query identity.

Fingerprint inputs must include:

- entity identity
- predicate semantics
- grouped semantics
- aggregate semantics
- projection expression tree (normalized)
- order semantics
- limit semantics

Required projection fingerprint rules:

- semantic tree hashing only
- canonical ordering for commutative operators
- literal normalization
- alias excluded from semantic identity

Continuation requirements:

- continuation envelope includes projection hash
- changing projection semantics invalidates continuation reuse
- equivalent projection semantics preserve continuation compatibility

Explicit removals for completion:

- explain-surface-based projection hashing is removed as projection authority
- `ProjectionDefault` marker dependency is removed as projection identity source

## 11. Planner and Executor Boundary

Planner owns semantic construction.

Planner must:

- construct `Expr`
- construct `ProjectionSpec`
- validate grouped and HAVING expression compatibility
- emit semantic projection handles consumed by execution

Executor must:

- consume planner-emitted projection semantics only
- evaluate expressions and aggregate outputs mechanically
- enforce invariant-only guards for invalid shapes that can occur only via planner bypass

Executor must not:

- construct alternate projection semantics
- reconstruct aggregate semantics from ad-hoc runtime models
- own user-shape semantic validation

## 12. Validation Ownership and Layering

Validation ownership is explicit.

Planner-owned:

- user-shape semantics
- grouped compatibility
- aggregate placement
- projection semantic validity

Executor-owned:

- structural and invariant defenses
- runtime safety checks for bypassed invalid shapes

Layering constraints:

- no upward dependencies from planner semantics to executor internals
- no string-based error taxonomy classification at query/facade boundaries

## 13. Architectural Invariants

Required invariants for 0.38 closure:

- exactly one canonical projection semantic struct exists
- canonical `Expr` exists and is used for projection semantics
- planner is the only semantic projection constructor
- executor is projection-shape consumer only
- no expression logic exists in storage or index layers
- no alternate aggregate/projection semantic shapes exist in production boundaries
- fingerprint is derived from semantic projection tree and aggregate semantics only
- continuation compatibility includes projection semantic hash
- alias and explain metadata are excluded from semantic fingerprint identity

## 14. Enforcement and Test Requirements

Required guard classes:

Required behavioral coverage:

- unknown field failures before scan-budget consumption
