# 0.38 — Projection & Expression Spine

## Status

Draft — Architectural Design
Target: 0.38
Scope: Complete projection semantics and introduce a unified expression spine
Prerequisite: 0.37 aggregate builder convergence complete

---

# 1. Objective

Complete the SELECT surface by introducing:

* A unified expression model
* Fully deterministic projection semantics
* Projection-aware fingerprinting
* Expression-safe execution
* Explicit GROUP BY compatibility enforcement

0.38 finalizes projection without:

* Introducing relational joins
* Introducing window functions
* Introducing cost-based planning
* Expanding beyond single-entity algebra

---

# 2. Non-Goals (0.38)

The following are explicitly out of scope:

* Joins
* Subqueries
* Window functions
* Recursive queries
* Cost-based optimization
* Expression pushdown into storage
* Dynamic function registration
* Implicit expression evaluation in index layer

Projection expansion must not destabilize planner simplicity.

---

# 3. Current Gaps

Today:

* Structural projection exists.
* Aggregate builder exists.
* GROUP BY validation exists.
* HAVING exists.

Missing:

* Unified expression AST
* Expression type inference
* Projection fingerprint normalization
* Expression-aware continuation stability
* Formal projection validation model

---

# 4. Expression Model

## 4.1 Canonical Expression Enum

Introduce a single expression type:

```rust
enum Expr {
    Field(FieldId),
    Literal(Value),
    Unary { op: UnaryOp, expr: Box<Expr> },
    Binary { op: BinaryOp, left: Box<Expr>, right: Box<Expr> },
    Aggregate(AggregateExpr),
    Alias { expr: Box<Expr>, name: Alias },
}
```

Constraints:

* Tree structure only.
* No cyclic references.
* No runtime-evaluated closures.
* No storage-layer references.

---

## 4.2 Supported Expression Categories

0.38 supports:

### Arithmetic

* + (addition)
* - (subtraction)
* * (multiplication)
* / (division)
* - (unary negation)

### Boolean

* AND
* OR
* NOT

### Comparison

* =
* !=
* <
* <=
* >
* > =

### Literals

* Numeric
* Boolean
* String (if already supported)
* Null

No string/date function library in 0.38.

# 5. Type Inference

Introduce deterministic type resolution:

* Binary operations enforce compatible types.
* Numeric coercion follows unified decimal model.
* Null propagation rules defined.
* Aggregate result types explicitly defined.
* No implicit widening beyond schema-defined scale.

Type inference must be:

* Deterministic
* Planner-visible
* Fingerprint-stable

---

# 6. Projection Semantics

## 6.1 Structural Projection

* Explicit field list defines output shape.
* Order preserved as declared.
* Duplicate aliases disallowed.

## 6.2 Aggregate Projection

When GROUP BY is present:

* Non-aggregate expressions must reference grouped fields.
* Aggregate expressions allowed.
* Mixed grouped + aggregate expressions allowed.
* Validation occurs in planner stage.

Invalid cases rejected at plan construction.

---

## 6.3 Alias Semantics

* Aliases apply only to output naming.
* Alias does not affect fingerprint identity of underlying expression.
* Alias collisions disallowed.

---

# 7. Expression Evaluation Engine

Introduce a pure in-memory evaluator:

* Executes after row materialization.
* Does not access storage layer.
* Deterministic evaluation order.
* No dynamic dispatch via trait objects.

Evaluation occurs:

* After predicate filtering.
* After grouping (for aggregate expressions).
* Before ordering (for computed projection fields used in ORDER BY).

---

# 8. Planner Integration

## 8.1 Expression Compatibility Rules

Planner must:

* Validate GROUP BY compatibility.
* Validate aggregate placement.
* Detect storage-pushdown eligibility (structural only).
* Forbid expression-based pushdown.

Expression pushdown is explicitly disallowed in 0.38.

---

## 8.2 Capability Preservation

Planner remains rule-based.

Expressions:

* Do not introduce plan enumeration.
* Do not affect index scoring.
* Do not require cost estimation.

---

# 9. Fingerprinting

Projection shape must be included in plan fingerprint.

Fingerprint components:

* Entity
* Predicate
* Group spec
* Aggregate spec
* Projection expression tree (normalized)
* Order spec
* Limit spec

Expression fingerprinting rules:

* Canonical ordering of commutative operations.
* Literal normalization.
* Alias excluded from structural identity.
* Aggregate expressions fingerprint through AggregateExpr spine.

Continuation envelopes must include projection hash.

---

# 10. Continuation Stability

Continuation must encode:

* Plan fingerprint
* Ordering boundary
* Grouping state (if grouped)
* Projection shape hash

Any projection change invalidates continuation reuse.

---

# 11. Execution Model Interaction

Projection must not:

* Change streaming eligibility.
* Introduce hidden buffering.
* Modify grouping strategy.
* Alter downgrade semantics.

Computed expressions evaluated post-stream where necessary.

Streaming grouping remains unaffected.

---

# 12. Testing Requirements

## 12.1 Expression Evaluation

* Arithmetic correctness
* Null propagation
* Boolean evaluation
* Aggregate + expression interaction

## 12.2 GROUP BY Validation

* Valid combinations accepted
* Invalid combinations rejected

## 12.3 Fingerprint Stability

* Equivalent expressions produce identical fingerprint.
* Alias changes do not affect fingerprint.
* Helper terminals equal builder equivalents.

## 12.4 Continuation Stability

* Changing projection invalidates continuation.
* Identical projection preserves continuation.

---

# 13. Architectural Invariants

After 0.38:

* All projection logic flows through Expr.
* No aggregate logic outside AggregateExpr.
* No expression logic in storage layer.
* No duplicated projection validation paths.
* Planner/executor boundary preserved.
* No cost-based logic introduced.

---

# 14. Migration Notes

* Existing helper projection APIs remain.
* Builder path becomes canonical.
* Internal direct construction removed where redundant.
* Public API remains source-compatible.

---

# 15. Success Criteria

0.38 is complete when:

* Expression AST is canonical.
* Projection fingerprint is stable.
* GROUP BY compatibility enforced via expression model.
* All tests pass with no semantic regression.
* No planner complexity growth detected.
* No layering violations introduced.

# Summary

0.38 completes the SELECT surface by:

* Introducing a unified expression spine
* Making projection first-class and deterministic
* Preserving planner simplicity
* Maintaining streaming-first guarantees
* Avoiding relational expansion

It is the last major language-surface milestone before numeric consolidation and identity freeze.
