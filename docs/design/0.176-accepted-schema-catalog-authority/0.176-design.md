# IcyDB 0.176 - Accepted Schema Catalog Authority

## Status

Active design.

0.176 is the schema-catalog follow-on to the 0.175 journaled-store hot-path
hardening line. It should make accepted schema behave like a cheap runtime
catalog consumed by query compile/plan paths, not as authority that each query
reconstructs from schema stores, generated models, diagnostics, or
reconciliation helpers.

This line is about authority, identity, and hot-path shape. It is not a public
migration-syntax line.

## Thesis

Query compile/plan should move from this shape:

```text
query compile/plan
  -> ensure accepted schema snapshot
  -> reconstruct/manifest schema authority
  -> maybe hit cache
```

to this shape:

```text
query compile/plan
  -> read accepted catalog identity/snapshot
  -> cache lookup
  -> plan only on miss
```

An accepted schema snapshot remains the persisted truth, but the runtime should
consume it through a catalog-shaped boundary:

- cheap identity first;
- bounded latest-snapshot load;
- one schema-owned projection for planner-visible facts;
- no generated-model fallback authority;
- no diagnostics or reconciliation DTOs in normal query hot paths.

## Why This Is 0.176

0.147 made accepted schema the metadata and runtime row-layout authority while
keeping generated models as proposals and generated-compatible bridges.

0.170 made store capabilities derive from accepted catalog declarations and
registry handles, not generated models or diagnostics DTOs.

0.175 hardened journaled read paths so normal reads do not rebuild live caches,
replay journal tails, fold, construct diagnostics, or materialize full overlay
snapshots before bounded scans.

0.175.6 also removed warmed-query recovery/schema reconciliation drift from
the journaled `LIMIT 1` path. If the 0.175 line leaves first-query cold
compile/plan cost high, 0.176 owns that follow-up only at the accepted catalog
boundary: make cold query setup consume cheap accepted catalog identity/context
instead of reconstructing schema authority before cache lookup or first plan
construction.

The remaining leak is schema authority consumption during query compile/plan:
some paths can still make accepted schema feel like something to ensure,
reconstruct, or re-manifest before the cache can answer. 0.176 should close
that boundary without expanding migration semantics.

## Sanity Check

The tentative direction is sound if 0.176 stays narrow:

- query cache lookup needs accepted catalog identity, not a rebuilt schema
  authority object;
- compile and plan should share the same accepted catalog context instead of
  loading or manifesting schema independently;
- `SchemaInfo` and visible-index views are projections of accepted catalog
  authority, not alternate authority;
- schema store persistence remains the durable source, but latest lookup must
  be streaming and bounded for query paths;
- generated `EntityModel` and `IndexModel` remain proposal and compatibility
  inputs, never runtime fallback authority;
- diagnostics can render accepted catalog facts, but rendered facts do not
  feed back into planning or capability decisions.

The main risk is replacing one reconstruction step with a differently named
reconstruction step. 0.176 should name the catalog identity/context and then
prove cache hits can return before building planner projections.

## Non-Goals

0.176 must not add:

- public migration DDL;
- broad `ALTER TABLE` semantics;
- automatic schema-version increments;
- a schema migration framework;
- generated-model fallback authority;
- diagnostics-derived schema authority;
- journaled protocol changes unless a hard authority bug requires them;
- row-format compatibility branches for obsolete internal schema shapes.

0.176 should not change SQL DDL frontend scope. SQL DDL remains a frontend over
catalog-native schema mutation, not the owner of mutation semantics.

## Vocabulary

### Generated Schema Proposal

Generated `EntityModel` and `IndexModel` describe what the compiled code wants
the schema and index contracts to be.

Allowed uses:

- first bootstrap proposal when no accepted snapshot exists;
- reconciliation input;
- generated-compatible proof construction;
- model-only tests and convenience views.

Disallowed uses:

- runtime fallback authority after an accepted snapshot exists;
- planner cache identity;
- diagnostics backfill for missing accepted metadata;
- schema-store reconstruction in query hot paths.

### Accepted Runtime Schema

The accepted runtime schema is the accepted persisted snapshot for one entity
after schema-owned validation. It owns durable schema facts:

- entity path and entity name;
- schema version as accepted metadata;
- durable field IDs;
- primary-key field IDs;
- row-layout slots;
- persisted field kinds and leaf codecs;
- nullable/default/write-policy metadata;
- accepted index contracts;
- relation metadata and SQL capability projection.

Accepted runtime schema is the authority. It is not generated metadata plus
runtime fixes.

### Accepted Schema Catalog

The accepted schema catalog is the runtime boundary that exposes accepted
schema authority cheaply to readers.

It should be treated as a schema-owned context for one entity:

```text
AcceptedSchemaCatalog
  -> AcceptedSchemaCatalogIdentity
  -> AcceptedSchemaSnapshot
  -> planner-visible projection on miss
  -> row/write/decode contracts at mutation or executor boundaries
```

The catalog may internally retain or derive projections, but callers should not
reconstruct those projections independently.

### Accepted Catalog Identity

Catalog identity is the runtime identity of one accepted schema catalog entry.
It should be cheap to clone and sufficient for cache invalidation.

The identity should include:

- entity identity, such as entity path and/or entity tag;
- accepted schema version for observability and ordering context;
- accepted runtime schema fingerprint;
- identity-method version if the meaning of the fingerprint changes.

The fingerprint, not the numeric schema version alone, is cache authority. A
schema version can help diagnostics, but cache correctness depends on hashing
the accepted persisted contract that planning consumes.

### Planner View

The planner view is the catalog-owned projection that query planning uses for
field typing, scalar normalization, visible accepted indexes, index key
contracts, expression-index contracts, and relation visibility.

`SchemaInfo` may remain the concrete projection type, but it must be consumed
as a projection of the accepted catalog. It must not become an independent
authority that query paths rebuild before each cache lookup.

### Diagnostics Rendering

Diagnostics rendering is a read-only projection of accepted catalog state,
store state, and capability state.

Diagnostics may render:

- accepted catalog identity;
- schema version;
- accepted fields and indexes;
- store allocation metadata;
- cache attribution;
- storage footprints.

Diagnostics must not provide schema authority back to compile, plan, write, or
capability policy paths.

### Schema Store Persistence

The schema store persists accepted schema snapshots. It is durable storage, not
the query planner's authority API.

Query paths may load the latest accepted snapshot for one entity through a
bounded latest lookup. They must not scan all entities or materialize a
latest-by-entity map just to build query cache keys.

### Migration Intent

Migration intent is the proposed transition from existing accepted schema to a
new generated proposal or explicit mutation plan.

Migration intent is not accepted runtime schema. It becomes runtime authority
only after schema-owned validation publishes a new accepted snapshot.

## Authority Model

Accepted schema catalog ownership belongs in schema/session/registry
boundaries, not in query planner internals.

Allowed owners:

- schema reconciliation and schema mutation runners;
- schema store latest-snapshot APIs;
- runtime store registry handles after accepted registration;
- session entrypoints that create an accepted catalog context for one query or
  write operation.

Disallowed owners:

- query planner cache internals;
- SQL DDL frontend code;
- executor diagnostics;
- generated model helpers;
- storage diagnostics DTOs;
- recovery/fold helpers.

The query layer may consume catalog facts, but it should not know how to
ensure, reconcile, or reconstruct them.

## Runtime Shape

### Startup And Mutation Paths

Startup, reconciliation, and schema mutation paths can ensure and publish
accepted snapshots. They own proposal comparison and transition policy.

Expected shape:

```text
generated proposal / mutation intent
  -> schema-owned validation
  -> accepted snapshot publication
  -> registry/session catalog refresh
```

These paths may inspect generated models and migration intent because they are
authority-producing paths.

### Query Compile/Plan Paths

Query compile/plan paths should not ensure schema by reconciliation unless they
are serving an empty first-use bootstrap path.

Expected shape:

```text
query entrypoint
  -> accepted catalog context for entity
  -> catalog identity
  -> structural query cache key
  -> shared plan cache lookup
  -> on miss: planner view from same catalog context
  -> plan
```

The accepted catalog context should be created once per query entrypoint and
shared by compile, visible-index, explain, and plan-cache code that participate
in the same operation.

### Cache Hit Path

On a cache hit, query execution should not:

- build `SchemaInfo` from an accepted snapshot;
- project accepted expression-index DTOs;
- inspect generated index models;
- call schema reconciliation;
- call storage diagnostics;
- allocate all-entity catalog metadata;
- invoke recovery or fold helpers.

The hit path may:

- load or borrow accepted catalog identity;
- check store planner visibility;
- build a structural query cache key;
- clone or borrow the shared prepared plan;
- execute with authority already carried by the prepared plan.

### Cache Miss Path

On a miss, planning may use the catalog planner view. The miss path should
still derive that view once and hand it through to all consumers that need it.

Compile and plan must not separately manifest schema authority from the same
accepted snapshot.

Cold first-query work may still be materially more expensive than warmed query
execution after 0.175. That cost belongs here only when it is schema/catalog
manifestation or first-plan construction work. 0.176 should not re-open
journaled traversal, recovery, fold, or diagnostics hot-path fixes unless they
turn out to be direct accepted-catalog authority leaks.

## Cache Identity

The shared query-plan cache key should include:

- cache method version;
- entity identity;
- accepted catalog fingerprint;
- planner-visible store/index readiness;
- structural query identity.

Filtered queries may need predicate normalization to avoid accidental cache
misses. That normalization must be a bounded semantic-key operation over the
accepted catalog context, not a side effect of full plan construction.

Cache key construction must not depend on:

- generated model index membership;
- diagnostics DTOs;
- schema-store allocation metadata;
- all-entity latest snapshot maps;
- migration proposal details that were not accepted.

The prepared plan should carry the accepted catalog fingerprint it was planned
against so executor/cursor/continuation guards can reject stale plans after an
accepted schema change.

## Schema Store Loading

`SchemaStore::latest_persisted_snapshot(entity)` is the query-path shape. It
should:

- constrain lookup to one entity;
- read in descending version order where the backend supports it;
- early-stop after the latest visible snapshot;
- merge journaled canonical/live/tombstone overlays without materializing a
  full snapshot map;
- decode only the selected latest snapshot.

All-entity helpers such as allocation metadata and latest-by-entity maps are
catalog diagnostics, registry setup, or capability derivation surfaces. They
must not be part of normal query cache-key construction.

## Planner Consumption

Planner entrypoints should receive either:

- an accepted catalog context; or
- an explicit pair of catalog identity plus catalog-owned planner view on miss.

They should not receive only a generated model and then discover accepted
schema internally.

Visible indexes should be derived from accepted index contracts plus store
visibility:

```text
accepted catalog planner view
  + store index readiness
  -> planner-visible indexes
```

Expression-index metadata remains accepted-catalog metadata. Generated
`IndexModel` may be used to propose or test expression contracts, but ordinary
planning should not reopen generated index metadata when accepted contracts
exist.

## Boundary Rules

Generated proposal, accepted runtime schema, diagnostics rendering, schema
store persistence, and migration intent have different jobs:

| Surface | Owns | Does Not Own |
| --- | --- | --- |
| Generated schema proposal | desired code schema, bootstrap proposal, compatibility proof input | runtime authority after acceptance |
| Accepted runtime schema | durable field/index/relation/write/decode contracts | migration intent before publication |
| Accepted schema catalog | cheap runtime identity and schema-owned projections | query planning semantics |
| Planner view | query planning field/index facts projected from catalog | accepted schema publication |
| Diagnostics rendering | human/operator visibility | compile/plan/write authority |
| Schema store persistence | accepted snapshot storage | generated fallback reconstruction |
| Migration intent | proposed transition | runtime authority before acceptance |

The planner can ask the catalog for facts. It cannot decide that generated
facts are close enough when accepted facts are missing or expensive to load.

## Work Areas

### 1. Name The Catalog Context

Introduce or consolidate a named accepted catalog context for one entity query
entrypoint.

The context should make this pairing explicit:

- accepted snapshot;
- accepted catalog identity/fingerprint;
- accepted executor authority when needed;
- lazy or cached planner view for misses;
- visible-index projection tied to store readiness.

The point is not a large abstraction. The point is preventing call sites from
building these pieces independently and accidentally paying for them twice.

### 2. Move Cache Lookup Before Planner Projection

Shared query-plan cache lookup should happen after catalog identity and
structural query identity are known, but before full planner projection.

For cache hits, this should skip:

- accepted `SchemaInfo` construction;
- expression-index projection;
- generated-compatible proof rebuilding;
- access-choice finalization work not needed to execute the cached plan.

### 3. Collapse Compile/Plan Schema Manifestation

SQL compile, typed query planning, visible-index construction, explain, and
plan hash paths should share the catalog context created for the operation.

The same operation should not call both:

- accepted snapshot load;
- schema-info reconstruction;
- accepted entity authority construction;
- visible-index projection;

from separate helper stacks when one catalog context can carry them.

### 4. Keep Generated Models Out Of Runtime Fallbacks

Generated model paths should be source-audited so generated `EntityModel` and
`IndexModel` uses remain proposal, test, compatibility-proof, or
model-only-convenience uses.

Any runtime fallback that says "if accepted schema is absent or expensive,
reconstruct from generated model" should be removed or isolated to first-use
bootstrap for empty stores.

### 5. Keep Diagnostics Out Of Authority Paths

Diagnostics should render catalog facts after the fact. Query cache keys,
planner visibility, schema fingerprints, and capability decisions should not
consume diagnostics DTOs or storage reports.

### 6. Keep Schema Store Latest Lookup Bounded

Schema store latest lookup should remain a one-entity streaming lookup with
early stop. Full latest-by-entity materialization belongs to diagnostics,
allocation metadata, or registry capability derivation, not query compile/plan.

### 7. Add Guard Evidence

0.176 should add focused tests or source-audit guards that prove:

- query cache hit does not reconstruct accepted schema;
- compile and plan do not both manifest schema independently;
- accepted schema lookup is cheap and bounded;
- generated `EntityModel`/`IndexModel` are not fallback runtime authority;
- diagnostics do not become schema authority;
- schema store latest lookup is streaming, not full-map materialization;
- planner cache key includes accepted schema identity.

## Source-Audit Candidates

Query/session/executor hot-path scans should make these patterns suspect:

```bash
rg -n "ensure_accepted_schema_snapshot|latest_raw_snapshots_by_entity|allocation_metadata|catalog_metadata|storage_report|StoreDiagnostics" \
  crates/icydb-core/src/db/session \
  crates/icydb-core/src/db/query \
  crates/icydb-core/src/db/executor
```

Expected:

- query hit paths do not call reconciliation-style ensure helpers;
- query cache key construction does not call all-entity metadata helpers;
- diagnostics/reporting helpers do not feed planner authority.

Accepted schema projection scans should make repeated manifestation visible:

```bash
rg -n "SchemaInfo::from_accepted_snapshot|from_accepted_schema|AcceptedRowLayoutRuntimeContract::from_generated_compatible_schema" \
  crates/icydb-core/src/db/session \
  crates/icydb-core/src/db/query \
  crates/icydb-core/src/db/executor
```

Expected:

- one operation creates one accepted catalog context;
- full planner projection is miss-only where possible;
- generated-compatible proof construction is not part of query cache hits.

Generated fallback scans should stay narrow:

```bash
rg -n "cached_for_generated_entity_model|EntityModel|IndexModel|model\\.indexes|MODEL\\.indexes" \
  crates/icydb-core/src/db/session \
  crates/icydb-core/src/db/query \
  crates/icydb-core/src/db/executor
```

Expected:

- generated model use in query/planner tests is allowed;
- production use is limited to structural query ownership, proposal identity,
  and explicitly named compatibility bridges;
- accepted index contracts drive runtime planning when accepted schema exists.

## Closeout Standard

0.176 can close when:

- query cache hits can return without accepted schema reconstruction;
- cache lookup happens before full planner-view manifestation on eligible
  paths;
- compile, plan, explain, and plan-hash paths share accepted catalog context
  instead of independently loading or manifesting schema facts;
- first-query cold compile/plan attribution shows schema/catalog authority is
  manifested once per operation and not before an eligible cache lookup;
- the cache key includes accepted catalog identity and misses on accepted
  schema fingerprint changes;
- schema-store latest lookup for query paths is one-entity, streaming, and
  early-stopping;
- generated model and diagnostics fallbacks are source-audited out of runtime
  authority paths;
- 0.175 journaled hot-path guarantees remain intact.

## Deferred Work

These remain outside 0.176 unless a direct authority bug forces a narrow fix:

- public schema migration syntax;
- online migration planning;
- index rebuild orchestration;
- fold scheduling and compaction policy;
- richer diagnostics formatting;
- broad SQL DDL expansion;
- multi-version row decode;
- compatibility fallback for obsolete internal schema formats.
