# IcyDB 0.64 Feature Design

## Structural Field-Level Mutation & Serialization

Status: **In Progress (0.64 API landed in core + façade)**
Scope: **Runtime + schema/model integration (no generator expansion)**

---

## 1. Intent

Introduce a **single structural field-level pipeline** that supports:

* partial updates (**UpdatePatch**)
* per-field serialization/deserialization
* future diff-based updates

without:

* per-entity generated logic
* per-field generated helpers
* wasm size regression

---

## 2. Motivation

After `0.62`:

* execution is **structural**
* generator surface is **minimized**
* metadata is **trait-owned**

However:

* updates still require **full-row reconstruction**
* serialization is **row-oriented**
* logic for field access is **duplicated across paths**

This feature consolidates:

> projection, update, and serialization into a **single slot-driven field pipeline**

---

## 3. Core Design Principle

> **All field-level operations are expressed structurally over slot-addressed rows,
> while the public API remains field-name-driven**

No per-entity logic. No generated mutation helpers.

Internally, persisted-row and row replay still operate on stable slot-addressed
field updates. Public callers do not address slots directly; they build an
`UpdatePatch` by model field name and let the structural mutation boundary
resolve names into the canonical slot patch.

    model: &EntityModel,

## 7. Execution Strategy

### Phase 1 (0.64)

* **full-row rewrite model**

  * read row → apply patch → re-encode row
* safe, simple, no layout coupling

### Phase 2 (future)

* partial-write optimization:

  * only rewrite affected fields
  * requires stable field offsets

---

## 8. API Surface

### 8.1 Structural API (primary)

```rust
session.mutate_structural(key, patch, mode)
```

Where `mode` is explicit:

```rust
enum StructuralMutationMode {
    Insert,
    Update,
    Replace,
}
```

Semantics:

* `Insert` requires one full after-image patch and fails if the row already exists
* `Update` applies one delta patch over one existing row and fails if missing
* `Replace` requires one full after-image patch, rebuilds from an empty row
  image, and inserts if the row is missing

Example:

```rust
let patch = UpdatePatch::new()
    .set_field(User::MODEL, "id", Value::Ulid(user_id))?
    .set_field(User::MODEL, "email", Value::String("alice@example.com".into()))?;

let saved = session.mutate_structural::<User>(
    user_id,
    patch,
    StructuralMutationMode::Update,
)?;
```

---

### 8.2 Typed Builder (optional sugar)

```rust
User::patch()
    .set_field(User::MODEL, "name", "Alice".into())
    .set_field(User::MODEL, "age", 42.into())
```

Constraints:

* remains model-owned instead of slot-owned
* no generated per-field methods

---

### 8.3 Closure API (optional future)

```rust
db.update::<User>(id, |u| {
    u.name = "Alice".into();
});
```

Implementation:

* decode → mutate → diff → patch → apply

## 9. Integration Points

### 9.1 Executor

* patch becomes a mutation input
* reuse:

  * validation
  * index maintenance
  * commit hooks

---

### 9.2 Validation

* reuse existing validation
* no duplicate validation logic

---

### 9.3 Index System

* patch triggers:

  * index recompute
  * unique constraint validation

---

## 10. Non-Goals

* no per-entity UpdateView structs
* no generated mutation helpers (`set_name`, etc.)
* no per-field generated logic
* no partial-write optimization in 0.63

---

## 11. Constraints

### 11.1 No Codegen Expansion

* derive/build must not emit:

  * patch logic
  * serialization helpers
  * per-field mutation code

---

### 11.2 Structural Authority Only

All logic flows through:

* `EntityModel`
* `FieldSlot`
* `Value`

---

### 11.3 No Duplication

* patch, serialization, and projection must share:

  * field access logic
  * slot resolution
  * value handling

---

## 12. Expected Outcomes

### 12.1 Functional

* ergonomic partial updates
* field-level serialization support
* foundation for diff-based updates

---

### 12.2 Architectural

* unified field pipeline
* reduced duplication across:

  * projection
  * serialization
  * mutation

---

### 12.3 Wasm Impact

* neutral or slightly positive
* no per-entity slope increase
* potential gzip improvement via shared logic

---

## 13. Risks

| Risk                            | Mitigation                     |
| ------------------------------- | ------------------------------ |
| Hidden duplication reintroduced | enforce shared helpers         |
| Validation drift                | reuse existing validation path |
| Over-engineering early          | start with full-row rewrite    |
| API creep                       | keep structural API primary    |

---

## 14. Acceptance Criteria

* all existing tests pass
* new patch-based updates validated
* no increase in `ten_simple` slope
* no reintroduction of generated helper surface
* projection/update/serialization share field pipeline

---

## 15. Summary

This feature introduces:

> **a single structural field-level pipeline for read, write, and serialization**

It enables partial updates while:

* preserving wasm efficiency
* avoiding generator regression
* reducing system-wide duplication
