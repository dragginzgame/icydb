# 0.158 Schema DDL and Field Evolution

## Status

Proposed.

Implemented slices:

- `0.158.0`: parse and route `ALTER TABLE ... ADD COLUMN ...` while failing
  closed before publication.
- `0.158.1`: generated-vs-accepted schema check endpoint and CLI report;
  persisted field-origin metadata exposed through schema describe surfaces,
  with reconciliation preserving accepted DDL-owned fields absent from
  generated Rust models.
- `0.158.2`: database-first defaults cleanup.
- `0.158.3`: executable `ALTER TABLE ... ADD COLUMN ...` publication for
  nullable no-default fields and supported encodable scalar defaults; required
  fields without defaults remain fail-closed.

## Purpose

0.157 closed the index-DDL line: SQL can now create, inspect, drop, and
idempotently manage DDL-owned secondary indexes through accepted-catalog
mutation.

0.158 should continue DDL at the table/field boundary. This is a different
risk class from index lifecycle work because field DDL changes row decoding,
nullability, accepted database defaults, retained slots, write encoding, and
accepted schema transition policy.

The default semantics themselves are part of the 0.158 DDL line: valid
defaults are encoded as accepted schema metadata, and runtime missing-field
decode uses accepted nullability/default contracts only.

The target is:

IcyDB can expose narrow, catalog-native schema-evolution DDL while keeping
accepted schema snapshots as runtime authority and rejecting unsupported field
transitions before row data is decoded or rewritten.

## Scope

Already implemented:

- schema drift inspection before executable field DDL
- `icydb schema check` or equivalent CLI report comparing generated schema,
  accepted schema, and DDL-owned accepted additions
- parsing/routing `ALTER TABLE entity ADD COLUMN field type` while failing
  closed for unsupported transition classes
- parsing/routing `ALTER TABLE entity ADD COLUMN field type DEFAULT value`
  while failing closed before accepted schema publication
- persisted field-origin metadata exposed through schema describe surfaces
- reconciliation preserving accepted DDL-owned fields absent from generated
  Rust models
- executable metadata-only publication for nullable no-default additive fields
- executable metadata-only publication for additive fields with accepted
  encoded defaults

Next executable field-DDL candidates:

- `DESCRIBE` / schema endpoint visibility for added fields, including
  nullability and accepted default metadata
- stale query-plan and cursor rejection after field-schema publication

SQL-added defaults are executable only when the `ADD COLUMN` publication path
validates and stores the default as encoded accepted schema metadata.
Unsupported defaults still reject before accepted schema mutation.

Later candidates:

- `ALTER TABLE entity ALTER COLUMN field SET/DROP DEFAULT`
- `ALTER TABLE entity ALTER COLUMN field SET/DROP NOT NULL`
- retained-slot field removal
- field rename metadata, if field identity remains stable

## Non-Goals

0.158 should not:

- treat SQL text as schema authority
- rebuild rows through ad hoc generated-model reconstruction
- add compatibility readers for older pre-1.0 schema formats
- silently coerce missing persisted fields without an accepted default or
  nullable policy
- redesign database default semantics outside the 0.158 DDL line
- use generated Rust `Default` or generated model fallback expressions as
  runtime decode authority
- publish SQL `ADD COLUMN DEFAULT` without accepted encoded default metadata
  and transition validation
- adopt DDL-owned fields into generated ownership; generated/DDL collision must
  fail closed until an explicit adoption command or reconciliation rule exists
- implement physical descending index key ordering; that belongs to the 0.159
  compact key/index encoding work
- redesign planner transparency; that belongs to 0.160

## Current Publication Boundary

As of `0.158.3`, SQL `ADD COLUMN` publishes accepted schema metadata for
nullable no-default additions and supported encodable defaults.

Unsupported shapes still fail closed before accepted schema publication.

The next executable slice may relax this only for transition classes that are
provably safe:

- future required additions with explicit physical backfill/rewrite

All other field transitions must continue to fail before accepted schema
mutation.

## Database Default Semantics Dependency

Defaults are database/schema defaults, not Rust construction fallbacks. The Rust
schema macro is the schema authoring frontend, while accepted schema snapshots
remain runtime authority.

Generated Rust `Default` is construction convenience only. It must not
influence accepted-runtime decode, schema drift handling, or SQL DDL
publication.

For field evolution, the relevant accepted-schema states are:

- no accepted default: `FieldDatabaseDefault::None`
- accepted persisted default: `FieldDatabaseDefault::EncodedSlotPayload(...)`

Missing persisted fields must be interpreted only through accepted schema
metadata:

- nullable missing field with no accepted default materializes null
- missing field with accepted encoded database default materializes that default
- required missing field without an accepted default fails closed

This applies equally to generated additive fields and future SQL-added fields.

## Hard Invariants

- Accepted schema snapshots remain runtime authority.
- Field identity must be stable and explicit.
- Field origin must be explicit before executable SQL field DDL. Generated
  fields and DDL-owned fields are different authority lanes.
- Missing persisted fields must decode only through accepted nullability or
  accepted default contracts.
- Accepted database defaults must be stored as encoded schema metadata, not as
  generated Rust expressions.
- Generated Rust `Default` and generated `PersistedRow` fallback behavior must
  never be accepted-runtime decode authority.
- Required additive fields are publishable only when the accepted transition
  can prove an accepted encoded default or an explicit backfill/rewrite policy.
- Nullable additive fields may be publishable without a default only when the
  accepted absence policy materializes null.
- After publication, write/insert encoding must use the accepted schema
  contract for omitted nullable/defaulted fields, not the generated Rust
  construction path.
- Unsupported field transitions fail before physical rewrite or publication.
- Generated models remain proposal inputs, not runtime fallback authority.
- DDL-owned accepted fields must be preserved across canister upgrades even
  when the generated Rust schema does not mention them.
- A generated field that collides with an existing DDL-owned field must fail
  closed until an explicit adoption rule is implemented.
- Successful schema DDL must be visible through `DESCRIBE`, schema endpoint
  payloads, query planning metadata, and cache/cursor identity.

## Schema Check Prerequisite

Executable field DDL should not ship before developers have a way to inspect
generated-vs-accepted schema drift.

Add a CLI check surface such as:

```text
icydb schema check demo_rpg --environment demo
```

The command should compare:

- generated schema from the current local build
- accepted schema currently stored in the target canister
- DDL-owned accepted fields and indexes

The report should distinguish expected drift from dangerous drift:

```text
IcyDB schema check
  environment: demo
  canister: demo_rpg
  status: drift

entity Character
  generated fields:
    id          Nat64      required
    name        Text       required
    level       Nat32      required default=1

  accepted fields:
    id          Nat64      required
    name        Text       required
    level       Nat32      required default=1
    nickname    Text       nullable origin=ddl
    title       Text       required default="Novice" origin=ddl

drift
  + nickname  accepted-only  origin=ddl
  + title     accepted-only  origin=ddl default="Novice"

recommendation
  ok: DDL-owned fields will be preserved across upgrade
  action: add nickname or title to Rust schema only if you intend to adopt them
```

Default rendering should prefer readable canonical literals when possible. When
a default payload cannot be losslessly rendered as a concise literal, the
schema-check surface may fall back to a stable encoded/hash representation, but
it must still distinguish `no default` from `explicit accepted default`.

Drift classes:

- `accepted-only origin=ddl`: valid drift; preserve across upgrade.
- `accepted-only origin=generated`: removed generated field; fail closed until
  retained-slot removal policy exists.
- `generated-only`: pending additive generated field; allowed only when
  accepted transition policy can prove nullable/default handling.
- `generated-only required with encoded default`: potentially publishable
  additive field after accepted transition validation.
- `generated-only required without default`: error until backfill/rewrite
  policy exists.
- `generated-only nullable without default`: potentially publishable additive
  field after accepted transition validation using accepted null
  materialization.
- `same-name different contract`: error.
- `same-name different default contract`: error unless explicit `ALTER DEFAULT`
  transition exists.
- `same-name default removed`: error until `ALTER COLUMN DROP DEFAULT` policy
  exists.
- `same-name nullability tightened`: error until `ALTER COLUMN SET NOT NULL`
  and validation/backfill policy exists.
- `generated collides with ddl-owned`: error until an explicit adoption rule is
  defined.
- `accepted-only ddl-owned defaulted field`: valid drift; preserve across
  upgrade.
- `accepted ddl-owned index`: valid drift; preserve across upgrade.
- `accepted generated-owned index missing from generated`: error unless a
  supported generated-index rename/removal reconciliation applies.

This check should be useful even before executable `ADD COLUMN`, because 0.157
already allows DDL-owned indexes that intentionally differ from generated
schema.

## ADD COLUMN Transition Rules

Executable field DDL must classify additive fields before accepted schema
publication.

For a new field:

1. Nullable with no default

   Publishable if the accepted absence policy can materialize null for older
   rows.

2. Required with encoded default

   Publishable if the SQL/default value can be encoded as one deterministic
   persisted slot payload and stored in accepted schema metadata.

3. Required with no default

   Not publishable without an explicit physical backfill/rewrite policy.

4. Unsupported or non-deterministic default

   Rejected before accepted schema publication.

5. Default expression that cannot encode as one slot payload

   Rejected before accepted schema publication.

Publication must allocate stable field identity and storage/slot metadata before
the accepted schema is written. The new identity must not be derived from field
position alone, and it must remain stable across generated-schema
reconciliation.

Publication must update schema identity so stale plans/cursors are rejected.

## Validation Expectations

Schema DDL slices should include:

- schema check coverage for accepted/generated drift classes
- parser and binding coverage for accepted and rejected SQL shapes
- accepted schema transition coverage
- accepted default metadata coverage for generated and DDL-owned additive fields
- persisted row decode coverage for missing nullable/defaulted fields
- missing-field decode coverage for nullable, defaulted, and invalid required
  additions
- write encoding coverage after schema publication
- schema check coverage for default/nullability drift classes
- SQL `ADD COLUMN DEFAULT` rejection/publication boundary coverage
- tests proving generated Rust `Default` is not runtime authority
- tests proving SQL-added defaults, once implemented, encode through accepted
  schema metadata
- generated canister endpoint coverage when generated behavior changes
- stale cache/cursor coverage when schema identity changes
- clippy and formatting checks for touched crates
