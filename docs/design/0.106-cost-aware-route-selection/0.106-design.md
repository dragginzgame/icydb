0.106 Cost-Aware Route Selection

Thesis

`0.106` should not introduce full cost-based optimization.

That would be too large and too early for the current planner line.

Instead, `0.106` introduces a minimal, bounded cost model that chooses
between already-valid execution routes without changing semantic admission,
predicate extraction rules, or expression ownership.

This slice is about choosing which valid plan to run, not inventing new plans.

Problem statement

After `0.104` and `0.105`:

* prepared fallback typing is aligned with planner expression typing
* predicate extraction is stable
* residual and explain plan shapes are structurally consistent

That means the planner can now ask a cleaner question:

* given multiple semantically valid routes, which one is better?

Without this slice, route selection remains effectively first-valid or
rule-priority driven, which can make residual-heavy queries choose poor routes
even when a better already-valid access path exists.

Design goal

Make route selection directionally better by choosing between already-correct
candidate plans using small structural heuristics.

That means:

* route alternatives become explicit planner candidates
* candidate routes are scored with a bounded heuristic model
* residual-heavy routes remain valid but score worse
* equal or unclear cases still fall back to deterministic rule-based choice

Non-goals

`0.106` does not aim to:

* introduce full cost-based optimization
* collect or maintain statistics
* add runtime feedback loops
* reorder boolean expressions for cost
* rewrite predicates for selectivity
* introduce join ordering
* add a global optimizer
* change query semantics
* broaden SQL admission
* change predicate extraction behavior

This is a route-quality slice, not an optimizer slice.

Why `0.106` comes after `0.104` and `0.105`

Because now the planner has:

* aligned prepared and non-prepared typing contracts
* stable predicate extraction
* consistent residual plan ownership

Before those slices, route comparison was less stable because equivalent query
shapes could still differ in extracted predicate or residual form.

Normative model

Rule 1 - cost chooses only among already-valid plans

`0.106` may choose between semantically equivalent routes using cost
heuristics, but it must not change query meaning.

Cost selection must only compare routes that are already valid under the
existing planner and validator contracts.

It must not:

* invent new access paths
* broaden predicate extraction
* introduce speculative rewrites
* change residual semantics

Rule 2 - route alternatives are explicit planner candidates

If multiple valid routes already exist, the planner should make those
alternatives explicit before selection.

The normative shipped candidate set for `0.106` is limited to existing
planner-owned route families:

* full scan
* primary-key lookup
* index lookup
* equality-prefix index scan
* order-preserving index range scan

No other candidate families are introduced in this slice.

The model is:

* candidate routes are enumerated
* invalid routes are discarded
* valid routes are scored
* one deterministic winner is selected

Rule 3 - cost model remains structural and bounded

The cost model should use only simple structural heuristics.

Example signals include:

* index-backed access versus full scan
* number of pushdown predicates
* expected row reduction before residual filtering
* presence and size of residual work
* access-path ordering compatibility

It should not depend on:

* histograms
* cardinality estimation infrastructure
* runtime sampling
* adaptive feedback

The comparison contract for `0.106` should be explicit and ordered:

* stronger access family outranks weaker access family
* greater pushdown coverage outranks smaller pushdown coverage
* lower residual penalty outranks higher residual penalty
* order-preserving routes outrank otherwise comparable routes that require more
  downstream ordering work
* remaining ties fall back to the existing deterministic planner tie-break
  contract

Rule 4 - residual work is a planner-visible penalty

The predicate/residual split now matters to route quality.

Routes with stronger pushdown and smaller residual work should score better
than routes that defer more work to residual filtering.

For example:

```sql
SELECT name
FROM character
WHERE class_name = 'Wizard' AND (strength + dexterity) > 24
```

This should prefer:

* index on `class_name` plus residual filter

over:

* full scan plus full residual filter

Residual-heavy plans remain valid, but they should score worse.

The residual penalty contract should stay bounded and planner-visible:

* no residual
* derived residual predicate only
* residual scalar expression
* residual work plus additional downstream ordering/materialization burden

`0.106` does not need numeric residual costing, but it should use one stable
category ordering so residual penalties are auditable and repeatable.

Rule 5 - tie-breaking stays deterministic

If cost signals are equal or too weak to distinguish candidates, the planner
must fall back to deterministic rule-based selection.

Cost-awareness improves route quality, but it must not make planner choice
nondeterministic.

The tie-break authority is the existing canonical planner ranking contract.

`0.106` must not introduce a second independent fallback selector.

Rule 6 - explain exposes lightweight route reasoning

Explain should show enough information to make route choice understandable,
without introducing a full numeric cost surface.

It is sufficient to expose:

* chosen route
* why that route won in broad terms
* what predicate pushdown was available
* what residual work remained

Numeric costs are not required in `0.106`.

Where possible, this slice should reuse the existing access-choice explain
surface rather than inventing a parallel route-explain contract.

Examples

Indexed route preferred over full scan

```sql
SELECT name
FROM character
WHERE class_name = 'Wizard' AND (strength + dexterity) > 24
```

Expected planner behavior:

* index access on `class_name` is considered
* full scan is also a valid fallback route
* the indexed route wins because it reduces rows before residual evaluation

Order-preserving route preferred when it avoids extra work

```sql
SELECT name
FROM character
WHERE class_name = 'Wizard'
ORDER BY class_name ASC, id ASC
LIMIT 20
```

Expected planner behavior:

* an order-preserving route may outrank a semantically equivalent unordered
  route that would require more downstream work
* this still depends only on already-valid route alternatives

Prepared execution boundary

No change:

* `0.106` does not widen prepared templates
* `0.106` does not change prepared fallback typing
* prepared and non-prepared planning may share route scoring logic, but cost
  selection must not create a new prepared lane

Identity boundary

Cost-aware selection is planner-time route choice only.

It must not:

* change semantic query identity
* widen admission
* create a new prepared identity surface
* become cache or fingerprint identity by itself

Likely code seams

* planner candidate-route enumeration
* planner candidate scoring
* access-choice ranking and deterministic tie-breaking
* explain route-reason projection

Risks

Risk 1 - semantic leakage

If route scoring starts depending on speculative rewrites or widened
extraction, `0.106` will stop being a route-quality slice and start becoming an
optimizer slice.

Risk 2 - hidden complexity

A bounded heuristic model can still become too large if it starts modeling too
many signals too early.

Risk 3 - nondeterministic planner choice

If equal-score handling is not explicit, cost selection could make route choice
unstable across equivalent plans.

Relationship to earlier slices

Slice | Role
--- | ---
`0.100` | surface completion
`0.102` | semantic admission
`0.103` | execution clarity and explain split
`0.104` | prepared typing alignment
`0.105` | extraction stability
`0.106` | route quality selection

What comes after `0.106`

Only after this slice would larger optimization work make sense, such as:

* better selectivity heuristics
* limited statistics
* join planning

Verification matrix

Area | Required proof
--- | ---
Candidate routes | multiple already-valid routes are enumerated explicitly when available
Cost selection | structurally better routes outrank worse routes under the bounded heuristic model
Residual penalty | residual-heavier routes score worse than otherwise comparable pushdown-heavier routes
Determinism | equal or unclear cost cases still select the same route deterministically
Explain | chosen route and lightweight planner reason are publicly visible
Boundary | no new semantic admission, extraction, or prepared-template widening occurs

Acceptance criteria

`0.106` is complete when:

* the planner can score multiple already-valid routes
* route choice is guided by bounded structural heuristics
* residual-heavy routes are penalized relative to pushdown-heavier routes
* deterministic tie-breaking remains explicit
* explain can describe route choice at a lightweight public level
* no semantic, extraction, or prepared-boundary widening occurs

Thesis sentence

`0.106` introduces a bounded cost-aware route selection model that chooses
between semantically valid execution strategies using simple structural
heuristics, without changing query semantics, predicate extraction rules, or
expression ownership.
