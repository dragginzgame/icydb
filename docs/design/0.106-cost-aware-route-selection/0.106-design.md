0.106 should not be “full cost-based optimization”

That would be too big and too early.

Instead, 0.106 should be:

0.106 Cost-Aware Route Selection (Minimal Planner Cost Model)
Core idea

0.106 introduces a minimal, bounded cost model to choose between already-valid execution routes, without changing semantic admission, predicate extraction rules, or expression ownership.

This is about:

choosing which valid plan to run
not inventing new plans
Why 0.106 comes after 0.104/0.105

Because now you have:

prepared fallback typing aligned with planner expression typing
stable predicate extraction
consistent plan shapes

So now you can ask:

“Given multiple valid routes, which one is better?”

Before this, that question wasn’t even well-defined.

What 0.106 should actually do
1. Introduce route alternatives (but only existing ones)

You likely already have multiple implicit routes:

full scan
indexed lookup
prefix scan
maybe order-preserving scans

0.106 should make this explicit:

candidate_routes = [
  index_lookup(class_name = 'Wizard'),
  full_scan
]
2. Add a very small cost model

Not statistics-heavy. Just structural heuristics.

Example signals:

predicate selectivity (heuristic, not measured)
number of pushdown predicates
index vs full scan
expected row reduction before residual

Think:

cost(index_lookup) = low
cost(full_scan) = high

Not:

histograms
cardinality estimation
runtime feedback
3. Define route selection contract

This is key:

The planner may choose among semantically equivalent routes using cost heuristics, but must not change query meaning.

Also:

If cost signals are equal or unclear, fall back to deterministic rule-based choice.

4. Integrate with predicate/residual model

Now the triad matters:

routes are evaluated based on predicate coverage
residual cost is a penalty

Example:

WHERE class_name = 'Wizard' AND (strength + dexterity) > 24

Plans:

index on class_name + residual filter → GOOD
full scan + full residual → WORSE

So:

residual-heavy plans are still valid, but scored worse

5. Explain must show route reasoning (lightweight)

Not full cost breakdown — just enough:

Route: index_lookup(class_name)
Reason: predicate pushdown available
Residual: (strength + dexterity) > 24

No numeric costs required yet.

What 0.106 explicitly does NOT do
no statistics collection
no runtime feedback loops
no join ordering
no reordering boolean expressions for cost
no predicate rewriting for selectivity
no global optimizer
Why this slice is important

Without 0.106:

all plans are “first valid wins”
residual-heavy queries may accidentally pick poor routes
performance feels inconsistent or surprising

With 0.106:

performance becomes directionally correct
without sacrificing simplicity or correctness
Clean 0.106 thesis

0.106 introduces a bounded cost-aware route selection model that chooses between semantically valid execution strategies using simple structural heuristics, without changing query semantics, predicate extraction rules, or expression ownership.

Important constraint

The most important rule for 0.106 is:

Cost must only choose between already-correct plans. It must never introduce new semantics or rely on speculative rewrites.

If you keep that invariant, you won’t destabilize the system.
