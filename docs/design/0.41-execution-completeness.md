## 4. Execution Vocabulary Contract

Execution nodes (stable vocabulary):

- Access:
  `ByKeyLookup`, `ByKeysLookup`, `PrimaryKeyRangeScan`, `IndexPrefixScan`,
  `IndexRangeScan`, `FullScan`, `Union`, `Intersection`
- Predicate evaluation:
  `IndexPredicatePrefilter`, `ResidualPredicateFilter`
- Ordering:
  `OrderByAccessSatisfied`, `OrderByMaterializedSort`
- Distinct:
  `DistinctPreOrdered`, `DistinctMaterialized`
- Projection:
  `ProjectionMaterialized`, `ProjectionIndexOnly`
- Cursor / limit:
  `LimitOffset`, `CursorResume`, `TopNSeek`, `IndexRangeLimitPushdown`
- Aggregates:
  `AggregateCount`, `AggregateExists`, `AggregateMin`, `AggregateMax`,
  `AggregateFirst`, `AggregateLast`, `AggregateSum`, `AggregateSeekFirst`,
  `AggregateSeekLast`
- Grouped aggregates:
  `GroupedAggregateHashMaterialized`, `GroupedAggregateOrderedMaterialized`
- Index ordering:
  `SecondaryOrderPushdown`

`0.41` slices below are selected because they materially affect this vocabulary.

## 4.1 Predicate Execution Model

Predicate evaluation occurs in two stages:

1. `IndexPredicatePrefilter`
   - Predicate portion compiled into index scan keys or index-side filtering.
   - Executed during index traversal.
2. `ResidualPredicateFilter`
   - Remaining predicate conditions evaluated against materialized rows.
   - Applied after row fetch when predicate components cannot be pushed into the index.

`EXPLAIN` reflects both stages explicitly so users can see which predicate
components were pushed into index traversal and which remained post-access.

## 5. Landed In 0.41 (Current)

## 5.1 A) COUNT(predicate) strict index prefilter pushdown

Design:

- allow index-backed count fast path with predicate only when strict
  all-or-none index predicate compilation succeeds
- route strict compiled predicate into index stream descriptor as
  `IndexPredicateExecution`

Why this matters:

- introduces/solidifies `IndexPredicatePrefilter` behavior for count windows
- avoids post-`EXPLAIN` shape churn for count predicate execution plans

Current invariants:

- no strict compile -> do not take count index prefilter fast path
- strict compile -> prefilter may apply, parity with fallback must hold

## 5.2 B) Field-target MAX bounded probe parity (safe subset)

Design:

- enable bounded `MAX(field)` probe only for tie-free targets
- centralized rule: `is_tie_free_probe_target(...)`

Tie-free rules:

- target is primary key, or
- target is backed by unique single-field leading index

Why this matters:

- closes MIN/MAX probe asymmetry where MAX was always conservative
- preserves correctness for non-tie-free targets by keeping them unbounded

Current invariants:

- non tie-free `MAX(field)` remains conservative
- tie-free `MAX(field)` gets one-key bounded probe hint

## 5.3 C) Index-range ORDER satisfaction extension

Design:

- extend secondary ORDER pushdown applicability so index-range shapes can be
  considered eligible when ORDER matches the index range suffix + PK tie-break
- treat eligible index-range ordered shapes as access-order-satisfied in route
  and post-access ordering gates

Why this matters:

- removes avoidable post-access materialized sorting for eligible index-range
  ordered windows
- stabilizes `OrderByAccessSatisfied` behavior before `EXPLAIN` freeze

Current invariants:

- non-matching index-range ORDER shapes remain explicit non-eligible
- matching index-range ORDER shapes skip materialized sort safely

## 5.4 D) Secondary-index IN multi-key pushdown

Design:

- add dedicated access node: `AccessPath::IndexMultiLookup`
- lower secondary `IN` predicates to one index-backed multi-lookup node per
  eligible leading-field index (instead of expanding one union child per value)
- execute one multi-lookup node by consuming one lowered prefix spec per lookup
  value and merging/deduplicating emitted keys

Why this matters:

- stabilizes explain-facing access vocabulary before `EXPLAIN` freeze
- replaces noisy `Union(IndexPrefix...)` expansion with one explicit node

Current invariants:

- singleton secondary `IN` collapses to canonical `IndexPrefix`
- multi-value secondary `IN` canonicalizes value sets (sorted + deduped)
- planner/executor parity remains equivalent to prior union-expanded behavior

## 5.5 E) ORDER BY + LIMIT top-N seek

Design:

- add explicit route contract `TopNSeekSpec { fetch }` for eligible ordered
  scalar load windows
- derive top-N seek eligibility in route hints/feasibility once, then project
  bounded fetch hints through route-owned scan-hint contracts
- keep unpaged `execute()` semantics explicit by reducing bounded fetch to
  keep-count (`offset + limit`) instead of paged lookahead fetch (`+1`)

Why this matters:

- stabilizes top-N seek semantics as an explicit pre-`EXPLAIN` route contract
- removes ad-hoc one-off hinting and keeps bounded ordered-window behavior
  centralized under route/executor contracts

Current invariants:

- top-N seek contract only derives for ordered load windows that are streaming
  safe and non-continuation
- paged load windows retain fetch-count (`offset + limit + 1`) behavior for
  continuation lookahead
- unpaged `execute()` windows use keep-count (`offset + limit`) bounded fetch
  while preserving canonical row parity

## 6. Remaining 0.41 Slices

## 7. Contract and Safety Requirements

- preserve strict vs ignore missing-row behavior
- maintain deterministic tie-break semantics (field value, then PK ascending)
- keep strict predicate compilation boundary explicit (`StrictAllOrNone`)
- keep fallback parity tests for every new bounded fast path

## 8. Test Strategy (Per Slice)

For execution-shape slices:

- route-contract tests: verify execution mode, hints, and seek/pushdown flags
- parity tests: compare optimized path against canonical materialized baseline
- stale-key/missing-row tests: prove correctness under `Ignore` and `Error`
- scan-budget tests: prove bounded paths reduce scans where contract allows
