# 0.198 Read Intent Ergonomics

Status: proposed follow-up after 0.197 exact-key canonicalization.

0.198 should make caller-facing reads easier to write safely by replacing
ad-hoc raw `.limit(...)` caps with a small set of semantic read intents.

The first implementation must stay simple:

- no custom read policies;
- no user-defined caps;
- no policy builder API;
- no endpoint-specific policy overrides;
- no app-owned admission configuration.

IcyDB should own the first set of limits internally. Application developers
should choose what the endpoint promises, not which magic row cap happens to
satisfy read admission.

## Context

Current read admission correctly makes public non-primary-key reads fail closed
unless the planner can prove a bounded admitted shape. That policy should stay
strict.

The developer-facing problem is that application code can satisfy the policy by
adding raw limits such as:

```rust
.limit(1_000)
.limit(10_000)
```

Those numbers are not inherently meaningful. They are endpoint safety caps, and
they can become silent partial-result bugs when the endpoint appears to return a
complete result.

These endpoint contracts are different:

- return one exact row;
- return whether a row exists;
- return one public cursor page;
- return a complete small set or fail;
- return an exact count or sum;
- process trusted admin batches.

They should not all be expressed as:

```rust
.limit(N)
.collect()
```

0.197 owns exact-key canonicalization: strict scalar primary-key equality and
finite primary-key `IN` canonicalize to `ByKey`, `ByKeys`, or `Empty` when
accepted-schema authority proves the key bound. 0.198 owns the broader
read-intent ergonomics problem for non-primary-key pages, complete small reads,
aggregates, and trusted/admin batch reads.

0.196 ordered-read pushdown also matters here: IcyDB already separates semantic
correctness from route-dependent performance facts such as pushed ordered
reads, materialized ordering, and limit-stop attribution. 0.198 must preserve
that separation. It should make safe read shapes easier to spell, not make
public reads less safe.

## Problem

Developers have too much opportunity to cargo-cult arbitrary caps.

Bad patterns:

```rust
// Bad: the endpoint probably means "all collections", but this silently
// truncates when the result has more than 1,000 rows.
db!()
    .load::<Collection>()
    .filter(FilterExpr::eq("project_id", project_id))
    .limit(1_000)
    .collect()?;
```

```rust
// Bad: the endpoint means "exact count", but this counts at most 10,000
// loaded rows.
let count = db!()
    .load::<Token>()
    .filter(FilterExpr::eq("collection_id", collection_id))
    .limit(10_000)
    .collect()?
    .len();
```

```rust
// Bad: the endpoint means "does this exist?", but it is expressed as a
// row page.
let exists = !db!()
    .load::<User>()
    .filter(FilterExpr::eq("email", email))
    .limit(1)
    .collect()?
    .is_empty();
```

These workarounds satisfy a safety mechanism, but they do not encode intent.
The dangerous behavior is not having a cap. The dangerous behavior is returning
a partial result through an API that looks complete.

## Design Goal

0.198 introduces hardcoded, engine-owned read intents:

- exact lookup;
- existence check;
- public page;
- complete small set or typed error;
- exact aggregate or typed unsupported/too-expensive diagnostic;
- trusted/admin batch.

Developers should choose one of these semantic read intents. IcyDB should apply
the relevant hardcoded admission behavior internally.

The first slice should improve ergonomics without introducing a general policy
configuration system.

## Core Rule

Paged reads may be partial because the return type says they are partial.

Complete reads must be complete or return a typed error.

Aggregates must use aggregate execution/proof or return unsupported or too
expensive. They must not be implemented by loading capped rows and aggregating
in application code.

Admin batch reads must be explicit and trusted/admin-gated.

Raw high `.limit(...)` should become an advanced low-level primitive, not the
normal public endpoint pattern.

## Non-Goals

0.198.0 does not include:

- custom read policies;
- user-defined page-size caps;
- app-defined complete-set caps;
- endpoint-specific policy builders;
- runtime policy registration;
- per-entity policy overrides;
- per-canister policy overrides;
- cost-based optimization;
- weakening public read admission;
- exact-key canonicalization, which belongs to 0.197;
- persisted-format changes;
- cursor-token format changes;
- count or sum implemented by capped row collection;
- silent clamping for complete reads;
- broad SQL surface expansion;
- automatic migration of every downstream app call site.

## Hardcoded Policy Stance

0.198.0 should use hardcoded engine-owned constants.

Tentative internal names:

```rust
pub(crate) const ICYDB_PUBLIC_PAGE_DEFAULT_ROWS: usize = /* existing default */;
pub(crate) const ICYDB_PUBLIC_PAGE_MAX_ROWS: usize = /* existing public max */;
pub(crate) const ICYDB_PUBLIC_PAGE_MAX_RESPONSE_BYTES: usize =
    /* existing bytes cap */;

pub(crate) const ICYDB_COMPLETE_SMALL_MAX_ROWS: usize = /* engine-owned cap */;
pub(crate) const ICYDB_COMPLETE_SMALL_LOOKAHEAD_ROWS: usize = 1;

pub(crate) const ICYDB_ADMIN_BATCH_ROWS: usize = /* engine-owned admin cap */;
pub(crate) const ICYDB_ADMIN_BATCH_MAX_RESPONSE_BYTES: usize =
    /* existing admin/trusted cap if any */;
```

The implementation should prefer existing read-admission constants where they
already exist rather than inventing conflicting values. If a current hardcoded
admission default already exists, 0.198 should wrap and document it. It should
not create a second cap authority for the same concept.

The first slice must not expose:

```rust
ReadPolicy::new(...)
ReadPolicyBuilder
.with_max_rows(...)
.with_max_bytes(...)
.with_custom_policy(...)
```

or any equivalent public override surface.

Future versions may add advanced policy configuration after the intent API has
settled, but that is out of scope for 0.198.0.

## Read Intent Model

Conceptually, public read execution should pass through one of these intents:

```text
ReadIntent
  ExactLookup
  Exists
  PublicPage
  CompleteSmallSet
  ExactAggregate
  AdminBatch
  TrustedUnchecked
```

The names are conceptual. The final implementation should reuse existing IcyDB
naming conventions where possible.

Terminals select intent. Planner and admission still own proof.

## Intent Contracts

### Exact Lookup

Exact lookup is the intent for zero-or-one primary-key reads.

Preferred spelling today:

```rust
db!()
    .load::<PrincipalToUser>()
    .by_id(Id::from_key(pid))
    .try_one()
```

After 0.197, this should also work when strict accepted-schema key proof exists:

```rust
db!()
    .load::<PrincipalToUser>()
    .filter(FilterExpr::eq("pid", pid))
    .try_one()
```

0.198 should not add caps to exact lookup. Exactness is proved by the selected
access path, not by `.limit(1)`.

### Exists

Existence checks should be their own terminal, not `.limit(1).collect()`.

Tentative API:

```rust
db!()
    .load::<User>()
    .filter(FilterExpr::eq("email", email))
    .exists()
```

Expected behavior:

- returns `bool`;
- uses exact-key proof, index proof, or an admitted bounded existence route;
- avoids materializing rows where the engine can prove a cheaper route;
- fails closed if the route is not admitted;
- does not require users to spell `.limit(1)`.

If an existence check would require an unbounded public scan, it should fail
with an actionable read-admission diagnostic.

### Public Page

Public pages are the normal shape for public list endpoints.

Tentative API:

```rust
let page = db!()
    .load::<Collection>()
    .filter(FilterExpr::eq("project_id", project_id))
    .order_by(Collection::created_at.desc().then(Collection::id.asc()))
    .page(PageRequest {
        limit: requested_limit,
        cursor,
    })?;
```

The page terminal may accept a requested limit, but that is not a custom policy.
It is a caller request clamped by IcyDB's hardcoded page maximum.

Expected behavior:

- clamps requested page size to `ICYDB_PUBLIC_PAGE_MAX_ROWS`;
- uses `ICYDB_PUBLIC_PAGE_DEFAULT_ROWS` if no limit is supplied;
- requires deterministic order unless the selected access path has a documented
  deterministic default order;
- returns a page type that makes partiality explicit;
- returns cursor or continuation metadata when more rows may exist;
- never pretends to return a complete set.

Tentative return shape:

```rust
pub struct Page<T> {
    pub rows: Vec<T>,
    pub next_cursor: Option<CursorToken>,
    pub limit: usize,
    pub has_more: bool,
}
```

The exact public fields should follow IcyDB's DTO stability policy.

### Complete Small Set

Complete small reads are for endpoints that genuinely promise "all matching
rows," but only for data that should be small.

Tentative API:

```rust
let collections = db!()
    .load::<Collection>()
    .filter(FilterExpr::eq("project_id", project_id))
    .order_by(Collection::id.asc())
    .collect_complete()?;
```

0.198.0 should not accept a caller-provided cap:

```rust
// Not in 0.198.0
.collect_complete_with_cap(10_000)
.collect_all_or_too_many(10_000)
.collect_with(ReadPolicy::...)
```

Instead, IcyDB applies `ICYDB_COMPLETE_SMALL_MAX_ROWS` internally.

Expected behavior:

- returns all matching rows if the result fits within the hardcoded
  complete-set cap;
- reads at most cap plus lookahead logical matching rows where possible;
- returns a typed `TooManyRows` error if more rows exist;
- never silently returns a partial vector;
- requires deterministic order if result ordering is exposed;
- preserves existing read-admission safety.

Tentative error:

```rust
pub enum CompleteReadError {
    TooManyRows {
        cap: usize,
        observed_at_least: usize,
    },
    ReadAdmission(ReadAdmissionError),
}
```

The key property:

```text
Vec<T> from collect_complete() means complete.
TooManyRows means not complete.
```

There is no partial success state for complete reads.

### Exact Aggregate

Aggregate endpoints should use aggregate terminals, not row loading.

Tentative APIs:

```rust
let count = db!()
    .load::<Token>()
    .filter(FilterExpr::eq("collection_id", collection_id))
    .count_exact()?;

let total = db!()
    .load::<LedgerEntry>()
    .filter(FilterExpr::eq("account_id", account_id))
    .sum_exact(LedgerEntry::amount)?;
```

Expected behavior:

- uses existing aggregate execution where available;
- uses exact cardinality/index proof where available;
- fails with a typed unsupported/too-expensive diagnostic when exact aggregate
  execution is not admitted;
- does not call `collect()` and aggregate in app code;
- does not return capped partial aggregate results.

Tentative error:

```rust
pub enum ExactAggregateError {
    UnsupportedShape {
        reason: AggregateUnsupportedReason,
    },
    RequiresUnboundedScan {
        suggested_intent: Option<ReadIntentSuggestion>,
    },
    ReadAdmission(ReadAdmissionError),
}
```

0.198.0 does not need to make every aggregate fast. It only needs to stop
encouraging fake exact aggregates built from capped row collections.

### Admin Batch

Trusted/admin operations sometimes need to process larger sets. Those should be
explicitly batched.

Tentative API:

```rust
let batch = db!()
    .load::<LedgerEntry>()
    .filter(FilterExpr::eq("status", Status::Pending))
    .order_by(LedgerEntry::id.asc())
    .admin_batch(AdminBatchRequest {
        cursor,
    })?;
```

0.198.0 should not expose custom batch sizes:

```rust
// Not in 0.198.0
.admin_batch_size(5_000)
.for_each_batch(batch_size, max_total, ...)
```

Instead, IcyDB uses `ICYDB_ADMIN_BATCH_ROWS` internally.

Expected behavior:

- trusted/admin lane only;
- hardcoded batch size;
- deterministic order required;
- cursor returned for continuation;
- never available as a normal public list shortcut;
- diagnostics clearly distinguish admin batch from public page.

The first implementation may make admin batch docs-only or generated-admin-only
if the trusted/admin API boundary is not ready.

### Trusted Unchecked

IcyDB may keep an explicit escape hatch for internal/admin uses, but 0.198
should make it socially and mechanically uncomfortable to use accidentally.

Expected properties:

- not used in public examples;
- requires a reason string or typed reason enum if that fits existing API
  conventions;
- clearly documented as bypassing normal public read ergonomics;
- excluded from generated public endpoints;
- easy to grep.

Tentative spelling:

```rust
.trusted_read_unchecked(TrustedReadReason::MaintenanceJob)
```

Do not improve this as the normal solution. It exists to make rare bypasses
auditable.

## API Shape

The preferred developer-facing API is terminal-based, not policy-based.

Good:

```rust
.try_one()
.exists()
.page(req)
.collect_complete()
.count_exact()
.sum_exact(field)
.admin_batch(req)
```

Avoid in 0.198.0:

```rust
.read_with(policy)
.with_policy(policy)
.with_max_rows(...)
.with_max_response_bytes(...)
.policy(ReadPolicy::...)
```

The API should be easy for application developers to choose from a small menu:

| Endpoint need | Use |
| --- | --- |
| One row | `try_one()` |
| Boolean existence | `exists()` |
| Public list | `page(...)` |
| All rows from a small set | `collect_complete()` |
| Count or sum | `count_exact()` / `sum_exact(...)` |
| Admin scan | `admin_batch(...)` |

## Raw `.limit(...)` Position

Raw `.limit(...)` remains a low-level query modifier, but 0.198 should stop
teaching it as the normal way to satisfy read admission.

Rules:

- `.limit(1)` should not be necessary for exact key lookup.
- `.limit(N).collect()` must not be recommended for complete reads.
- `.limit(N).collect().len()` must not be recommended for counts.
- high raw limits should be treated as migration debt unless wrapped by a
  semantic terminal.

0.198.0 may add a docs/test report for suspicious examples:

```text
.limit(1000)
.limit(1_000)
.limit(10000)
.limit(10_000)
```

This should start as a warning/reporting tool, not necessarily a hard CI
failure, unless the repository already has a lint convention for this.

## Read Admission Behavior

0.198 must not weaken read admission.

| Intent | Admission basis |
| --- | --- |
| `ExactLookup` | Exact key proof from `by_id(...)` or 0.197 canonicalized `ByKey` |
| `Exists` | Exact key, index, or bounded existence proof; otherwise fail closed |
| `PublicPage` | Bounded page size, deterministic order, admitted route |
| `CompleteSmallSet` | Hardcoded cap plus lookahead/proof; too many becomes typed error |
| `ExactAggregate` | Aggregate proof/execution path; unsupported if unbounded |
| `AdminBatch` | Trusted/admin lane plus hardcoded batch size |
| `TrustedUnchecked` | Explicit bypass reason, not public |

If the engine cannot prove an admitted shape for the chosen intent, it should
fail with an actionable diagnostic rather than suggesting arbitrary caps.

## Diagnostics

Read-admission diagnostics should suggest semantic intent.

Example diagnostic for an unbounded public read:

```text
This public read is not admitted because it may return an unbounded row set.

Choose one of the supported read intents:

- exact lookup: use by_id(...) or canonicalized primary-key equality;
- existence check: use exists();
- public list: use page(...) with deterministic order;
- complete small set: use collect_complete(), which fails if too many rows exist;
- exact aggregate: use count_exact() or sum_exact(...);
- trusted/admin batch: use admin_batch(...) from an admin lane.
```

Diagnostics must avoid telling users to "just add `.limit(10_000)`."

For complete reads that exceed the cap:

```text
This complete read matched more rows than IcyDB's hardcoded complete-set cap.

Use page(...) for public list semantics, count_exact() for counts, or an admin
batch if this is a maintenance path.
```

For aggregate fallback:

```text
This exact aggregate is not admitted for the selected shape.

Do not compute it by loading a capped row set. Add an index, use a supported
aggregate route, or expose a paged endpoint instead.
```

## Return Types

Return types must make partiality explicit.

Good:

```rust
Page<T>
CompleteReadError::TooManyRows
ExactAggregateError::UnsupportedShape
AdminBatch<T>
```

Bad:

```rust
Vec<T> // when silently capped
usize  // when computed from a capped row load
```

If an API returns `Vec<T>` from a complete terminal, that vector must be
complete for the requested shape.

## Generated Endpoint Guidance

0.198 should update generated endpoint recommendations, but generated endpoint
redesign can be incremental.

Preferred generated public list shape:

```rust
#[query]
fn list_collections(req: PageRequest) -> Result<Page<CollectionDto>, Error>
```

Avoid generating:

```rust
#[query]
fn list_collections() -> Vec<CollectionDto>
```

unless the endpoint is explicitly a complete-small-set endpoint and will return
`TooManyRows` on overflow.

Tentative generated endpoint categories:

```rust
#[icydb_query(one)]
#[icydb_query(exists)]
#[icydb_query(page)]
#[icydb_query(complete_small)]
#[icydb_query(count_exact)]
#[icydb_admin_query(batch)]
```

These attributes are conceptual. The first slice may only update docs/examples
if codegen changes would make 0.198 too large.

## Implementation Shape

Preferred flow:

```text
SQL/fluent/query input
  -> existing semantic binding
  -> developer selects terminal/read intent
  -> read intent maps to hardcoded IcyDB admission lane
  -> planner/admission proves or rejects the shape
  -> executor runs selected route
  -> result type reflects intent
```

Do not make this a separate policy layer that application developers must
learn. Do not duplicate planner proof logic inside terminals. Terminals select
intent; planner and admission still own proof.

## First Slice Scope

0.198.0 should be intentionally small.

In scope:

- add or standardize `exists()`;
- add or standardize `page(PageRequest)`;
- add `collect_complete()` with hardcoded cap and `TooManyRows`;
- add or document `count_exact()` and optionally `sum_exact(...)`;
- add hardcoded internal constants for public page and complete-small-set caps;
- improve read-admission diagnostics to suggest semantic terminals;
- update docs/examples away from raw high `.limit(...)`;
- add test/report grep for suspicious raw high limits in docs/examples.

Optional in 0.198.0 if small:

- `admin_batch(AdminBatchRequest)`;
- generated endpoint examples for page/complete/count shapes;
- query-safe EXPLAIN field showing selected read intent.

Out of scope for 0.198.0:

- custom policies;
- public `ReadPolicy` type;
- policy builders;
- per-endpoint cap overrides;
- per-canister cap overrides;
- broad generated endpoint redesign;
- new optimizer proof families;
- persisted-format changes;
- general aggregate planner rewrite.

Stop and split if implementing all terminals requires broad query planner
changes. The first slice may land docs, diagnostics, `page(...)`, and
`collect_complete()` before aggregate or admin batch support if needed.

## Error Model

Use typed errors instead of silent partial results.

Tentative error categories:

```rust
pub enum ReadIntentError {
    ReadAdmission(ReadAdmissionError),
    TooManyRows {
        cap: usize,
        observed_at_least: usize,
    },
    MissingDeterministicOrder {
        intent: ReadIntentKind,
    },
    UnsupportedExactAggregate {
        reason: AggregateUnsupportedReason,
    },
    AdminOnly {
        intent: ReadIntentKind,
    },
}
```

The exact type layout should follow existing IcyDB error conventions.

Important distinction:

```text
TooManyRows is not corruption.
TooManyRows is not a partial success.
TooManyRows means the caller chose a complete-small-set intent for data that is
no longer small.
```

## Tests

Minimum tests:

Exact lookup guidance:

- docs/examples for exact lookup do not use `.limit(1)`;
- 0.197 exact-key examples remain the authority for `filter(pk = value)`.

Exists:

- `exists()` returns true for a matching row;
- `exists()` returns false for no row;
- unsupported public existence shape fails admission instead of scanning
  unboundedly;
- `exists()` does not materialize full rows when a cheaper route exists, if
  attribution can prove that.

Public page:

- request with no limit uses hardcoded default;
- request above max clamps to hardcoded max;
- response returns explicit cursor/has_more metadata;
- deterministic order is required unless the selected route documents a
  deterministic order;
- response does not claim completeness.

Complete small set:

- returns all rows when under cap;
- returns all rows when exactly at cap;
- returns `TooManyRows` when more than cap rows match;
- does not silently truncate;
- observes deterministic order;
- covers heap and journaled stores if both surfaces exist.

Exact aggregate:

- `count_exact()` uses aggregate/exact route where available;
- unsupported count returns typed diagnostic;
- no implementation uses `collect_complete()` or capped row loading internally
  to fake an exact count;
- same coverage for `sum_exact(...)` if implemented.

Admin batch:

- unavailable or rejected on normal public lane;
- works on trusted/admin lane if implemented;
- uses hardcoded batch size;
- returns cursor/continuation metadata;
- deterministic order required.

Diagnostics:

- unbounded public read diagnostic suggests semantic terminals;
- diagnostic does not recommend arbitrary `.limit(10_000)`;
- complete overflow diagnostic suggests `page(...)`, aggregate, or admin batch
  as appropriate.

Docs/lint:

- docs/examples do not contain raw `.limit(1_000)` / `.limit(10_000)` as a
  recommended public pattern;
- allowed raw-limit examples are explicitly marked low-level or anti-pattern.

## Performance And Behavior Evidence

0.198 is mostly an API/admission ergonomics line, not an optimizer line. It
should still produce behavior evidence.

Required focused behavior matrix:

| Scenario | Expected |
| --- | --- |
| Public page under requested max | Returns page with requested size |
| Public page over max | Clamps to hardcoded max |
| Complete set under cap | Returns complete vector |
| Complete set exactly at cap | Returns complete vector |
| Complete set over cap | Returns `TooManyRows` |
| Exact count supported | Returns exact count |
| Exact count unsupported | Typed unsupported/too-expensive diagnostic |
| Raw high-limit examples | Replaced or marked anti-pattern |
| Unsupported public read | Diagnostic suggests semantic intent |

For performance evidence:

- page and complete reads should not introduce material overhead beyond existing
  admission/execution paths;
- complete-small-set should stop after cap plus lookahead where possible;
- exact aggregate must not load capped rows and count them;
- no broad deterministic matrix regression should be introduced if a full matrix
  is run;
- any performance claim must include before/after artifacts.

0.198 should not claim query-engine speedups unless actual route facts or
instruction counters prove them.

## Migration Guidance

For each raw high-limit call site, classify intent:

- exact lookup: use `by_id(...)` or 0.197 canonicalized primary-key equality;
- public list: use `page(...)`;
- complete small set: use `collect_complete()` only when the endpoint truly
  promises all rows and the data should stay small;
- count or sum: use `count_exact()` / `sum_exact(...)`;
- maintenance/admin flow: use `admin_batch(...)`;
- real escape hatch: use `trusted_read_unchecked(...)` with an explicit reason.

Do not migrate raw caps mechanically to `collect_complete()` unless the
endpoint really promises a complete small set.

## Risks

### Too Much API Surface

Even without custom policies, adding several terminals can confuse users.

Mitigation:

- keep the docs table simple;
- prefer names that describe endpoint promise;
- do not add builder types in 0.198.0.

### Hidden Partial Results

The largest risk is preserving silent truncation under a nicer name.

Mitigation:

- only `page(...)` may be partial;
- `collect_complete()` must return `TooManyRows`;
- aggregate terminals must not use capped row collection.

### Cap Values Become Controversial

Hardcoded values may not fit every app.

Mitigation:

- 0.198.0 intentionally prioritizes simplicity;
- advanced overrides can be a later design after the core intent vocabulary is
  stable;
- developers with large data should use pagination or admin batches, not bigger
  public caps.

### Admission Drift

Terminals must not duplicate proof logic.

Mitigation:

- terminals select intent;
- planner and admission still own proof;
- tests assert unsupported shapes fail closed.

### Aggregate Overreach

Exact aggregate support may not be ready for every shape.

Mitigation:

- `count_exact()` / `sum_exact(...)` may return unsupported or too expensive;
- do not fake exactness by loading capped rows.

## Success Criteria

0.198.0 is complete when:

- public docs teach read intent instead of arbitrary raw limits;
- exact lookup examples do not use fake `.limit(1)`;
- public list examples use `page(...)`;
- complete-small-set examples use `collect_complete()` and return `TooManyRows`
  on overflow;
- count examples use `count_exact()` and do not load capped rows;
- hardcoded page and complete-set caps live in one internal authority;
- no public custom policy or override API is exposed;
- read-admission diagnostics suggest semantic terminals;
- raw high `.limit(...)` call sites in docs/examples are removed or explicitly
  marked as anti-patterns;
- existing read-admission safety is preserved.

## Implementation Sequencing

| Step | Patch | Closeout |
| --- | --- | --- |
| 1 | Docs and diagnostic wording | Read-admission errors suggest semantic intents, not raw caps |
| 2 | `exists()` terminal | Existence no longer requires `.limit(1).collect()` examples |
| 3 | `page(PageRequest)` terminal | Public page clamps to hardcoded max and returns explicit page DTO |
| 4 | `collect_complete()` terminal | Complete reads are complete or `TooManyRows` |
| 5 | `count_exact()` docs/API | Counts are not implemented by capped row loads |
| 6 | Optional `admin_batch(...)` | Trusted/admin batch reads have hardcoded batch size |
| 7 | Migration/doc cleanup | Raw high limits removed from examples or marked anti-pattern |

Stop and split if implementing all terminals requires broad query planner
changes.

## Implementation Prompt Addendum

0.198 is a read-intent ergonomics slice.

Do not add custom policies.

Do not expose public cap override APIs.

Do not add `ReadPolicy`, `ReadPolicyBuilder`, `.with_max_rows(...)`,
`.with_max_bytes(...)`, or equivalent user-configurable policy surfaces in
0.198.0.

Use hardcoded IcyDB-owned constants. Prefer existing read-admission constants
where available.

Do not weaken read admission.

Do not silently clamp complete reads.

Do not implement count/sum by loading capped rows.

Do not implement 0.197 exact-key canonicalization here.

Do not introduce persisted-format or cursor-token changes.

The goal is to make developers choose one of a few semantic terminals:

```text
try_one
exists
page
collect_complete
count_exact / sum_exact
admin_batch
trusted_read_unchecked
```

instead of choosing arbitrary `.limit(...)` values.
