# 0.198 Supplemental Status

Date: 2026-07-07

0.198 is the read-intent ergonomics line. Its implemented surface is the
semantic terminal vocabulary for caller-facing read promises, not a custom read
policy framework.

## Closed Surface

- Exact rows: primary-key access plus `try_one()`.
- Existence: `exists()` / `not_exists()`.
- Public pages: request-owned `PageRequest` through `page(...)?.execute()`.
- Complete small sets: `collect_complete()`.
- Exact aggregates:
  - `count_exact()`;
  - `sum_exact(field)`;
  - `min_exact()`;
  - `min_exact_by(field)`;
  - `max_exact()`;
  - `max_exact_by(field)`;
  - `avg_exact(field)`.
- Trusted maintenance batches: `trusted_read_unchecked().admin_batch(...)`.

Semantic terminals own their bounds. A prior `partial_window(...)` is rejected
for existence, page, complete-small-set, exact aggregate, and admin-batch
terminals because accepting it would blur the endpoint contract.

## Low-Level Row Windows

`partial_window(...)` is the explicit low-level partial row-window primitive
for `execute()` and `execute_rows()`.

The older public load `.limit(...)` spelling and public partial-window
aggregate aliases are not retained. Developers choose semantic terminals for
semantic answers, or `partial_window(...)` only when the endpoint deliberately
returns a partial row window.

## Resolved Out Of Scope

The following are resolved as outside the 0.198 architecture:

- public `ReadPolicy` / `ReadPolicyBuilder` APIs;
- caller-configurable read caps;
- generated public SQL endpoints;
- broad diagnostic taxonomy redesign;
- persisted-format changes;
- cursor-token format changes.

They remain intentionally outside 0.198 because the line's architecture is that
application developers choose the endpoint promise, while IcyDB owns the hard
caps for public semantic terminals.

## Validation Expectations

0.198 closeout validation should prove:

- exact aggregate helpers compile on the public fluent surface;
- exact aggregate helpers reject prior `partial_window(...)`;
- exact aggregate EXPLAIN helpers report `ReadIntentKind::ExactAggregate`;
- exact primary-key filtered reads can use exact aggregate helpers without fake
  row-window limits;
- read-admission docs and examples teach semantic read intent before raw
  limits.
