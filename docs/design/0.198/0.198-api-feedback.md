# 0.198 Supplemental API Feedback

Date: 2026-07-07

Status: feedback captured for 0.198 API cleanup. This document records review
feedback; it is not itself an implementation result.

0.198 made the read-intent model much clearer, but the public facade still
mixes too many audiences in one reference surface:

- normal public endpoint authors;
- trusted/admin endpoint authors;
- diagnostics and EXPLAIN users;
- low-level generated, policy, and internal callers.

The facade should teach a small intent-first API for ordinary endpoint code and
move advanced, trusted, diagnostic, and generated/internal surfaces into their
own tiers.

## Desired Public Shape

Ordinary endpoint authors should mostly see these shapes:

```rust
db.load::<User>()
    .by_id(id)
    .try_one()
```

```rust
db.load::<User>()
    .filter_eq(User::status, Status::Active)
    .order_asc(User::created_at)
    .page(PageRequest::first(50))
```

```rust
db.load::<User>()
    .filter_eq(User::kind, Kind::SmallStaticSet)
    .collect_complete()
```

```rust
db.load::<User>()
    .filter_eq(User::status, Status::Active)
    .count_exact()
```

The target mental model is:

- normal endpoint authors choose intent terminals;
- trusted/admin users enter an explicit trusted lane;
- diagnostics live in their own namespace or section;
- generated/internal helpers are listed only for audit, not taught as normal
  user API.

## Tiered Facade Model

Classify every public method into one of four tiers.

| Tier | Audience | Description |
| --- | --- | --- |
| Tier 1: Normal endpoint API | Ordinary app endpoints | Safe, intent-first commands endpoint authors should reach for first. |
| Tier 2: Trusted/admin API | Maintenance, migrations, internal canister tasks | Visible escape hatches that require caller-owned authorization and resource policy. |
| Tier 3: Diagnostics API | EXPLAIN, trace, plan inspection, attribution | Debugging and observability surfaces, not normal endpoint commands. |
| Tier 4: Generated/internal API | Macros, SQL validators, policy wrappers, generated glue | Not taught as normal user API; listed only where auditability requires it. |

Tier 1 should stay small:

- `load::<E>()`;
- `by_id(id)` / `by_ids(ids)`;
- `filter(...)`, `filter_eq(...)`, and adjacent typed filter helpers;
- `order_asc(...)`, `order_desc(...)`, `order_term(...)`,
  `order_terms(...)`;
- `try_one()`;
- `execute_rows()` primarily for exact key sets or explicitly partial windows;
- `page(request)`;
- `collect_complete()`;
- `exists()` / `not_exists()`;
- `count_exact()`;
- `sum_exact(field)`;
- `min_exact_by(field)`;
- `max_exact_by(field)`;
- `avg_exact(field)`.

Do not lead ordinary endpoint docs with `execute()`, trusted variants, tracing,
plan inspection, direct SQL helpers, or response narrowing helpers.

## API Recommendations

### 1. Use One Public Paging Spelling

Current surface includes both:

- `page(request)`;
- `execute_paged(request)`.

Status: accepted in 0.198.9 and completed by the trusted-lane cleanup.
`execute_paged(request)` and `execute_paged_trusted(request)` are removed from
the fluent load surface. Use `page(request)?.execute()` for public pages. For
trusted maintenance code, enter `trusted_read_unchecked()` first and then use
the same `page(request)?.execute()` spelling.

Suggested rule:

> Use `.page(request)?.execute()` in public endpoint code. Keep one spelling per
> cursor-page behavior.

### 2. Partial Window Naming

Status: accepted. The public facade should use `partial_window(...)` for this
low-level read shape.

`partial_window(n).execute_rows()` makes the endpoint contract explicit. It is
not a page, not complete, and not exact.

```rust
db.load::<Event>()
    .order_desc(Event::created_at)
    .partial_window(100)
    .execute_rows()
```

Docs should consistently describe it as a deliberate partial row-window
contract.

Status update: `partial_window(...)` now returns `PartialWindowLoadQuery`.
That wrapper exposes partial materialization and diagnostics, but not semantic
terminals such as `page(...)`, `collect_complete()`, `exists()`, or exact
aggregate helpers. This makes the partial-result contract visible in the type
shape instead of relying only on runtime intent checks.

### 3. Demote `execute()`

`execute()` returning `QueryResponse<E>` is powerful but does not express an
endpoint promise. Most endpoint code should prefer a terminal that names the
contract.

Suggested docs rule:

> Most endpoint code should not call `execute()` directly. Prefer `try_one()`,
> `page(...)`, `collect_complete()`, or an exact aggregate helper. `execute()`
> is for advanced callers that intentionally handle multiple response shapes.

### 4. Collapse Trusted Naming

Original trusted surface included:

- `trusted_read_unchecked()`;
- `execute_trusted()`;
- `execute_rows_trusted()`;
- `admin_batch(request)`.

Status: accepted. Fluent load queries now use `trusted_read_unchecked()` as the
trusted-lane marker and keep the same normal terminal spelling afterward.

The preferred mental model is:

```rust
db.load::<E>()
    .trusted_read_unchecked()
    .admin_batch(AdminBatchRequest::new())
```

Avoid teaching trusted variants of every terminal. That encourages the wrong
pattern: "try the public API, then add `_trusted` when rejected."

### 5. Split SQL Surface Visibility

SQL should be documented in hard tiers:

SQL admin API:

- `execute_sql_query`;
- `execute_sql_update`;
- `execute_sql_ddl`.

SQL policy/internal API:

- generated or validated policy wrappers;
- hidden SQL helpers;
- generated endpoint glue.

Accepted: internal/generated SQL helpers live under the Tier 4
internal/generated API section in the public facade reference. They are listed
for audit completeness, not taught as ordinary endpoint commands.

### 6. Clarify Exact Aggregate Names

The fieldless extrema helpers were ambiguous unless their target was
documented. The accepted spelling makes primary-key extrema explicit:

- `min_id_exact()`;
- `max_id_exact()`.

`min_id_exact()` and `max_id_exact()` operate on the entity's primary-key
ordering. Use `min_exact_by(field)` / `max_exact_by(field)` for field extrema.

### 7. Use `singleton()` For Singleton Entities

`singleton()` is valid for `SingletonEntity` types, and delete has its own
`singleton()` shape. This replaces the older `only()` spelling so the method
does not read like a generic one-row terminal.

Docs should say:

> `singleton()` is only for `SingletonEntity` types. It is not a one-row query
> terminal. For normal one-row reads, use `.by_id(id).try_one()`.

### 8. Make Delete Intentionally Different

Delete uses `.max_affected(n)`, which makes the mutation safety cap visibly
different from public read materialization bounds.

```rust
db.delete::<Session>()
    .filter_lt(Session::expires_at, now)
    .max_affected(500)
    .execute()
```

Accepted term: `max_affected(n)`.

### 9. Move Response Helpers Out Of The Main Facade Reference

Response helpers are useful, but listing all of them beside query commands
makes the facade feel much larger:

- `rows.count()`;
- `rows.exists()`;
- `rows.require_one()`;
- `rows.entity()`;
- `rows.entities()`;
- projection helpers;
- mutation result narrowing helpers.

Accepted: response helpers live in the response-types appendix instead of the
normal endpoint command path.

### 10. Remove Fluent Offset From Normal Public Reads

Status: accepted. The public `icydb` facade no longer exposes fluent
`.offset(...)` on load queries. SQL/direct-query offset support remains a
planner/executor concept and public read admission still rejects non-zero
offset for ordinary public reads.

Caller-facing list endpoints should use cursor continuation through
`PageRequest` instead of offset pagination.

### 11. Add Endpoint Cookbook

Status: accepted. The public facade reference now has an endpoint cookbook
covering exact lookup, exact ID sets, public pages, complete small sets, exact
aggregates, existence checks, deliberate partial row windows, and trusted
maintenance batches.

The cookbook is the Tier 1 teaching surface; diagnostics, SQL, trusted/admin,
and generated/internal helpers remain in separate tiers.

### 12. Type-State Read Intent Boundary

Status: first slice accepted. `PartialWindowLoadQuery` is the initial
read-intent type-state boundary. It prevents accidental chaining from a partial
row-window intent into semantic terminals without requiring a full query-builder
type-state rewrite.

Future type-state work should be driven by landed API pain, not speculative
surface expansion.

## Revised Read Intent Summary

| Endpoint promise | Preferred command | Notes |
| --- | --- | --- |
| Exact row by primary key | `load::<E>().by_id(id).try_one()` | Proved by key shape, not by truncation. |
| Exact rows by primary keys | `load::<E>().by_ids(ids).execute_rows()` | Exact key set. |
| Existence | `load::<E>().exists()` / `not_exists()` | Avoid materializing rows. |
| Public cursor page | `load::<E>().order_term(...).page(PageRequest::first(n))` | Preferred public list shape. |
| Complete small set | `load::<E>().collect_complete()` | Returns all admitted rows or fails. |
| Deliberate partial window | `load::<E>().partial_window(n).execute_rows()` | Not a page and not complete. Use only when partial results are the contract. |
| Exact aggregate | `count_exact()`, `sum_exact(field)`, `min_id_exact()`, `min_exact_by(field)`, `max_id_exact()`, `max_exact_by(field)`, `avg_exact(field)` | Aggregate over the admitted full result, not a materialized page. |
| Trusted maintenance batch | `trusted_read_unchecked().admin_batch(AdminBatchRequest::new())` | Admin/maintenance only. |

## Concrete Hard-Cut Candidates

| Current | Recommendation | Reason |
| --- | --- | --- |
| `page(request)` and `execute_paged(request)` | Accepted: keep only `page(request)` | One spelling per behavior. |
| `partial_window(n)` | Accepted spelling | Communicates partial-result contract. |
| fluent `.offset(n)` | Accepted: remove from the public facade | Caller-facing pages use cursors, not offset pagination. |
| `partial_window(n)` returning the normal load query | Accepted: return `PartialWindowLoadQuery` | Keeps semantic terminals off the partial-window path. |
| `execute()` | Keep but demote to advanced | Does not express endpoint promise. |
| `execute_trusted`, `execute_rows_trusted` | Accepted: remove from fluent load surface | Avoid casual trusted bypasses. |
| delete affected-row cap | Accepted: use `.max_affected(n)` | Distinguishes mutation cap from read limit. |
| fieldless exact min/max | Accepted: use `min_id_exact()` / `max_id_exact()` | The method name now states that the returned value is an identifier. |
| singleton entity shape | Accepted: use `singleton()` | Avoids confusing singleton access with a generic one-row terminal. |
| hidden SQL helpers | Accepted: move to internal/generated appendix | Prevent accidental direct use. |

## Bottom Line

The 0.198 read-intent API now has a tiered public reference: ordinary endpoint
authors get an intent-first Tier 1 path and endpoint cookbook, trusted/admin
work enters a visible Tier 2 lane, diagnostics live in Tier 3, generated/policy
helpers are listed only in Tier 4 for audit completeness, and the partial
row-window path is separated by `PartialWindowLoadQuery`.
