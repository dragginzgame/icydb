# 0.156 IcyDB TOML Config

Tentative design draft.

## Purpose

0.155 moved local SQL reads toward a standard controller-gated admin SQL
endpoint and left one remaining configuration question open: where should
canister/demo SQL opt-ins live?

0.156 should introduce a small `icydb.toml` project manifest that can answer
those build-time questions without making TOML parsing part of the canister or
CLI runtime payload.

The closeout target for 0.156 is:

An IcyDB project can declare whether the demo SQL connection and per-canister
admin SQL endpoint are active in `icydb.toml`; build tooling reads that file on
the host, validates it, and injects typed generated constants into generated
actor/project metadata glue. The final wasm and normal executable target graph
must not include the TOML parser or its dependency suite.

## Non-Goals

0.156 should not add:

- runtime TOML loading inside canisters
- runtime TOML loading in the normal `icydb` executable path
- a broad permission model
- per-principal SQL allowlists
- row-level policy
- dynamic endpoint toggles that can change without rebuild/reinstall
- migration scripting
- schema mutation semantics
- compatibility fallbacks for older config layouts before `1.0.0`

Changing `icydb.toml` is a build input change. It should require rebuild and,
for canister endpoint shape changes, reinstall/upgrade through the normal
project flow.

## Config Shape

The first manifest should stay deliberately small:

```toml
[demo.sql]
active = true
environment = "demo"
canister = "demo_rpg"

[canisters.demo_rpg.sql]
admin = true
```

Field meanings:

- `demo.sql.active`: whether repository demo SQL defaults are installed into
  generated project/demo metadata
- `demo.sql.environment`: default environment used by demo SQL helpers when
  `active = true`
- `demo.sql.canister`: default canister used by demo SQL helpers when
  `active = true`
- `canisters.<name>.sql.admin`: whether the generated actor glue for that
  canister includes the standard controller-gated admin SQL query endpoint

Omitted booleans default to `false`. Omitted `demo.sql.environment` and
`demo.sql.canister` are rejected when `demo.sql.active = true`; parsed inactive
demo fields do not appear in the generated runtime-facing model.

The nested canister SQL shape is intentional even though v1 has only one SQL
knob. It keeps the first implementation narrow while leaving an obvious
namespace for later SQL-surface controls without turning `admin` into a
one-off canister property.

## Hard Invariants

- The accepted schema snapshot remains runtime authority.
- `icydb.toml` controls opt-in surfaces and local developer defaults only; it is
  not schema authority.
- Admin SQL config decides whether the endpoint is generated. Runtime
  authorization still requires a controller caller.
- Unsupported or unknown config fields fail at build time.
- Config parsing happens only in host build tooling.
- Generated runtime code contains typed constants/enums, not raw TOML text.
- The final canister wasm must not depend on `toml`, `toml_edit`, or equivalent
  parser crates.
- The normal CLI executable should not carry a TOML parser unless a later
  design explicitly adds runtime project-config discovery.
- Before `1.0.0`, config formats hard-cut to the latest version instead of
  carrying compatibility readers.

## Build-Time Boundary

`icydb.toml` must be read by a host-only build helper. The parser dependency
belongs to a build-only package, not to `icydb`, `icydb-core`, `icydb-cli`, or
target actor glue.

Preferred shape:

```text
canister build.rs
  -> icydb-config-build reads icydb.toml
  -> validates IcydbProjectConfig
  -> writes OUT_DIR/icydb_config.rs
  -> icydb::build! consumes typed config when generating actor.rs
  -> icydb::start! includes generated actor.rs
```

The generated file should contain ordinary Rust constants, for example:

```rust
pub const DEMO_SQL_ACTIVE: bool = true;
pub const DEMO_SQL_ENVIRONMENT: &str = "demo";
pub const DEMO_SQL_CANISTER: &str = "demo_rpg";
pub const ICYDB_ADMIN_SQL_ENABLED: bool = true;
```

The exact names can change during implementation. The required property is that
runtime code reads typed generated Rust, not TOML.

The initial parsed model should be shaped like:

```rust
pub struct IcydbProjectConfig {
    pub demo_sql: DemoSqlConfig,
    pub canisters: BTreeMap<CanisterName, CanisterConfig>,
}

pub struct DemoSqlConfig {
    pub active: bool,
    pub environment: Option<String>,
    pub canister: Option<String>,
}

pub struct CanisterConfig {
    pub sql: CanisterSqlConfig,
}

pub struct CanisterSqlConfig {
    pub admin: bool,
}
```

Validation should lower that into a stricter generated model:

```rust
pub struct GeneratedIcydbConfig {
    pub demo_sql: Option<GeneratedDemoSqlDefaults>,
    pub canisters: BTreeMap<CanisterName, GeneratedCanisterConfig>,
}
```

That keeps inactive demo defaults out of runtime-facing generated code.

## Dependency Rule

Do not add TOML parser crates to a package that is compiled into canister wasm
or the ordinary CLI binary.

Acceptable dependency placement:

- a new host-only `icydb-config-build` crate used from `build-dependencies`
- optionally a tiny `icydb-config` typed-model crate with no TOML dependency,
  if shared types are needed
- a build-script-only helper binary run by `build.rs`
- target-specific build dependencies that are never compiled for
  `wasm32-unknown-unknown`

Rejected dependency placement:

- `crates/icydb/Cargo.toml` normal dependencies
- `crates/icydb-core/Cargo.toml` dependencies
- generated actor module dependencies
- CLI runtime dependencies, unless a later runtime-config design explicitly
  accepts the binary size tradeoff

The default implementation answer is a new host-only `icydb-config-build`
crate. Do not put TOML parsing into `icydb-build` unless the implementation
first proves that `icydb-build` is purely host-only in the actual target graph.
Even then, prefer the split crate because the invariant is easier to audit:

```text
canister crate build-dependencies:
  icydb-config-build

canister crate dependencies:
  icydb
  no toml
```

CI should include a dependency-graph guard that proves the wasm/package graph
does not include TOML parser crates.

## Admin SQL Generation

`sql.admin = true` should become the source of truth for whether generated
actor glue exports:

```text
icydb_admin_sql_query(sql: text) -> Result<SqlQueryPerfResult, Error>
```

When `sql.admin = false` or omitted:

- generated actor glue does not export the endpoint
- helper dispatch may be absent entirely
- the canister still builds with SQL support if its Cargo features request SQL
- CLI calls to that endpoint fail with the normal missing-method path

When `sql.admin = true`:

- generated actor glue includes the endpoint
- the endpoint remains read-only
- the endpoint rejects non-controller callers
- mutation SQL remains fail-closed through the existing query-lane rejection
- routing remains canister-owned and accepted-schema-backed

The old manual `icydb::admin_sql_query!()` opt-in must be removed or changed to
`compile_error!` with migration text in the same hard-cut line. No shim should
remain. If the macro can still export a method by being present in source, it
is still a second source of truth and undermines this design.

## Demo SQL Defaults

`demo.sql.active = true` should control repository demo defaults, not public SQL
authorization.

When active, generated project metadata may keep the current boring default:

```text
environment = "demo"
canister = "demo_rpg"
```

When inactive:

- demo helper commands should not assume `demo_rpg`
- SQL shell defaults should require explicit canister/environment or use a
  separately defined non-demo default
- missing demo config should not trigger stale-wasm or fixture-reload advice

This separates "the demo SQL connection exists" from "a canister exports admin
SQL." The former is a local developer default. The latter controls generated
canister surface area.

The normal `icydb-cli` binary should not embed arbitrary project
`icydb.toml` values at CLI compile time. That only works for the repository
demo and would not be a general project mechanism. For 0.156, CLI consumption
should be one of:

- explicit `--environment` and `--canister`
- generated project metadata emitted by build tooling in a stable location
- deferred until a later runtime project-metadata design

The CLI must not parse `icydb.toml` directly in this slice.

## Validation And Diagnostics

Build-time validation should reject:

- unknown top-level sections
- unknown fields
- non-boolean `active` or `sql.admin`
- active demo SQL without both default target fields
- a `canisters.<name>` entry that does not match a generated canister name
- duplicate or ambiguous canister names after normalization

Diagnostics should identify the config path, section, and key. Tests should
assert typed error variants or structured diagnostics, not string fragments.

## Cargo Integration

Build scripts that read `icydb.toml` must emit:

```text
cargo:rerun-if-changed=icydb.toml
```

If the config search walks up to the workspace root, it must also emit
`rerun-if-changed` for the resolved file and avoid broad directory watches.

The initial implementation should prefer one resolved manifest:

1. explicit path from build script input, if supplied
2. crate-local `icydb.toml`
3. workspace-root `icydb.toml`
4. absent config, treated as all booleans false

Absent config should be valid. Malformed present config should fail the build.
If both crate-local and workspace-root manifests exist, crate-local wins
entirely. 0.156 must not merge manifests or apply per-section overrides across
files.

## Example Build Flow

For demo RPG:

```text
canisters/demo/rpg/icydb.toml
  -> [demo.sql] active=true, environment=demo, canister=demo_rpg
  -> [canisters.demo_rpg.sql] admin=true
  -> build.rs emits typed config constants
  -> generated actor.rs exports icydb_admin_sql_query
  -> generated project metadata can target demo_rpg in demo
```

For a production canister that does not want SQL:

```text
[canisters.ledger.sql]
admin = false
```

Result:

- no admin SQL endpoint export
- no TOML parser in wasm
- no runtime config read
- accepted schema/runtime behavior unchanged

## Implementation Order

The recommended order is:

1. Add `icydb-config-build` with strict parse, validate, and generated-Rust
   writing.
2. Support absent config and one explicit config path.
3. Generate typed Rust constants into `OUT_DIR`.
4. Wire one demo or fixture canister build to the generated config.
5. Move admin endpoint generation behind the generated boolean.
6. Remove or neuter `icydb::admin_sql_query!()` with a hard-cut migration
   error.
7. Add dependency-graph and raw non-gzipped wasm size guards.
8. Only then wire demo SQL defaults through generated project metadata.

This proves parser dependency isolation and generated endpoint shape before the
less critical demo-default cleanup.

## Validation Target

Implementation should include focused checks that prove:

- absent `icydb.toml` builds and disables all optional config surfaces
- malformed `icydb.toml` fails during build tooling validation
- `sql.admin = false` produces no exported admin SQL endpoint
- `sql.admin = true` produces the controller-gated endpoint
- demo SQL defaults are generated only when `demo.sql.active = true`
- changing `icydb.toml` reruns the build script
- the wasm target dependency graph does not include TOML parser crates
- raw non-gzipped wasm size does not regress from parser dependencies

## Open Questions

- What stable location should generated project metadata use if CLI demo
  defaults are consumed before a broader project-metadata design?
- Should `icydb-config` exist as a TOML-free typed-model crate, or should
  `icydb-config-build` own both the parsed model and generated model privately
  for the first implementation?
