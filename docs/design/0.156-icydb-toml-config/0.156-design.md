# 0.156 IcyDB TOML Config

Tentative design draft.

## Purpose

0.155 moved local SQL reads toward standard controller-gated SQL endpoints and
left one remaining configuration question open: where should per-canister DB SQL
surface opt-ins live?

0.156 should introduce a small `icydb.toml` project manifest that can answer
those build-time questions without making TOML parsing part of the canister or
generated actor runtime.

The closeout target for 0.156 is:

An IcyDB project can declare whether per-canister read-only SQL and DDL/write
SQL surfaces are active in `icydb.toml`; build tooling reads that file on the
host, validates it, and injects typed generated constants into generated actor
glue. The final wasm target graph must not include the TOML parser or its
dependency suite.

## Non-Goals

0.156 should not add:

- runtime TOML loading inside canisters
- a broad permission model
- per-principal SQL allowlists
- row-level policy
- dynamic endpoint toggles that can change without rebuild/reinstall
- migration scripting
- schema mutation semantics
- compatibility fallbacks for older config layouts before `1.0.0`

Changing `icydb.toml` is a build input change. It should require rebuild and,
for canister endpoint shape changes, reinstall/upgrade through the normal
project flow.

## Config Shape

The first manifest should stay deliberately small:

```toml
[canisters.demo_rpg.sql]
readonly = true
ddl = true
fixtures = true

[canisters.demo_rpg.metrics]
enabled = true
reset = true

[canisters.demo_rpg.snapshot]
enabled = true
```

Field meanings:

- `canisters.<name>.sql.readonly`: whether the generated actor glue for that
  canister includes the standard controller-gated read-only SQL query endpoint
- `canisters.<name>.sql.ddl`: whether the generated actor glue for that
  canister includes the controller-gated DDL/write endpoint
- `canisters.<name>.sql.fixtures`: whether the generated actor glue for that
  canister includes the controller-gated fixture lifecycle endpoints
- `canisters.<name>.metrics.enabled`: whether the generated actor glue for that
  canister includes the metrics report endpoint
- `canisters.<name>.metrics.reset`: whether the generated actor glue for that
  canister includes the metrics reset endpoint
- `canisters.<name>.snapshot.enabled`: whether the generated actor glue for
  that canister includes the storage snapshot endpoint

Omitted booleans default to `false`. There is no IcyDB-owned default target;
`icydb sql` requires an explicit `--canister`.

The nested canister DB shape keeps the first implementation narrow while
leaving obvious namespaces for later endpoint-surface controls.

## Hard Invariants

- The accepted schema snapshot remains runtime authority.
- `icydb.toml` controls DB SQL opt-in surfaces only; it is not schema authority
  and does not own canister target mapping.
- SQL config decides whether endpoint families are generated. Runtime
  authorization still requires a controller caller.
- Unsupported or unknown config fields fail at build time.
- Config parsing happens only in host build tooling.
- Generated runtime code contains typed constants/enums, not raw TOML text.
- The final canister wasm must not depend on `toml`, `toml_edit`, or equivalent
  parser crates.
- The normal CLI executable may parse `icydb.toml` for config inspection and
  sync checks, but SQL execution still requires explicit `--canister`.
- Before `1.0.0`, config formats hard-cut to the latest version instead of
  carrying compatibility readers.

## Build-Time Boundary

`icydb.toml` must be read by a host-only build helper. The parser dependency
belongs to a build-only package, not to `icydb`, `icydb-core`, `icydb-cli`, or
target actor glue.

Preferred shape:

```text
canister build.rs
  -> icydb-config-build reads icydb.toml
  -> validates IcydbProjectConfig
  -> writes OUT_DIR/icydb_config.rs
  -> icydb::build_with_options! consumes typed config when generating actor.rs
  -> icydb::start! includes generated actor.rs
```

The generated file should contain ordinary Rust constants, for example:

```rust
pub const ICYDB_SQL_READONLY_ENABLED: bool = true;
pub const ICYDB_SQL_DDL_ENABLED: bool = true;
pub const ICYDB_SQL_FIXTURES_ENABLED: bool = true;
pub const ICYDB_METRICS_ENABLED: bool = true;
pub const ICYDB_METRICS_RESET_ENABLED: bool = true;
pub const ICYDB_SNAPSHOT_ENABLED: bool = true;
```

The exact names can change during implementation. The required property is that
runtime code reads typed generated Rust, not TOML.

The initial parsed model should be shaped like:

```rust
pub struct IcydbProjectConfig {
    pub canisters: BTreeMap<CanisterName, CanisterConfig>,
}

pub struct CanisterConfig {
    pub sql: CanisterSqlConfig,
    pub metrics: CanisterMetricsConfig,
    pub snapshot: CanisterSnapshotConfig,
}

pub struct CanisterSqlConfig {
    pub readonly: bool,
    pub ddl: bool,
    pub fixtures: bool,
}

pub struct CanisterMetricsConfig {
    pub enabled: bool,
    pub reset: bool,
}

pub struct CanisterSnapshotConfig {
    pub enabled: bool,
}
```

Validation should lower that into a stricter generated model:

```rust
pub struct GeneratedIcydbConfig {
    pub canisters: BTreeMap<CanisterName, GeneratedCanisterConfig>,
}
```

That keeps target selection outside runtime-facing generated code.

## Dependency Rule

Do not add TOML parser crates to a package that is compiled into canister wasm.
The ordinary CLI may use the host parser for explicit `icydb config` inspection
commands, but generated canister code must only consume typed constants.

Acceptable dependency placement:

- a new host-only `icydb-config-build` crate used from `build-dependencies`
- optionally a tiny `icydb-config` typed-model crate with no TOML dependency,
  if shared types are needed
- a build-script-only helper binary run by `build.rs`
- target-specific build dependencies that are never compiled for
  `wasm32-unknown-unknown`

Rejected dependency placement:

- `crates/icydb/Cargo.toml` normal dependencies
- `crates/icydb-core/Cargo.toml` dependencies
- generated actor module dependencies
- CLI runtime dependencies, unless a later runtime-config design explicitly
  accepts the binary size tradeoff

The default implementation answer is a new host-only `icydb-config-build`
crate. Do not put TOML parsing into `icydb-build` unless the implementation
first proves that `icydb-build` is purely host-only in the actual target graph.
Even then, prefer the split crate because the invariant is easier to audit:

```text
canister crate build-dependencies:
  icydb-config-build

canister crate dependencies:
  icydb
  no toml
```

CI should include a dependency-graph guard that proves the wasm/package graph
does not include TOML parser crates.

## SQL Surface Generation

`sql.readonly = true` should become the source of truth for whether generated
actor glue exports the read-only SQL query endpoint:

```text
__icydb_query(sql: text) -> Result<SqlQueryPerfResult, Error>
```

Generated endpoint Rust function names are the exported canister method names.
They use the reserved project prefix `__icydb_*` and do not use endpoint
`name = ...` attribute overrides. User-provided hooks remain plain, non-exported
`icydb_*` functions called by the generated wrapper.

`sql.ddl = true` should become the source of truth for whether generated actor
glue exports the DDL/write endpoint:

```text
__icydb_ddl(sql: text) -> Result<SqlQueryResult, Error>
```

`sql.fixtures = true` should become the source of truth for whether generated
actor glue exports the fixture lifecycle endpoints:

```text
__icydb_fixtures_reset() -> Result<(), Error>
__icydb_fixtures_load() -> Result<(), Error>
```

`metrics.enabled = true` should become the source of truth for whether
generated actor glue exports the metrics report endpoint:

```text
__icydb_metrics(window_start_ms: opt nat64) -> Result<EventReport, Error>
```

`metrics.reset = true` should become the source of truth for whether generated
actor glue exports the metrics reset endpoint:

```text
__icydb_metrics_reset() -> Result<(), Error>
```

`snapshot.enabled = true` should become the source of truth for whether
generated actor glue exports the storage snapshot endpoint:

```text
__icydb_snapshot() -> Result<StorageReport, Error>
```

The generated `__icydb_snapshot` endpoint walks recovered stores and reports
store/entity/index inventory, byte estimates, index state, and corruption
counters. That makes it useful for diagnostics and audits, but larger and
semantically different from in-memory runtime metrics.

When a switch is `false` or omitted:

- generated actor glue does not export that endpoint family
- helper dispatch may be absent entirely
- the canister still builds with SQL support if its Cargo features request SQL
- CLI calls to absent endpoints fail with the normal missing-method path

When a switch is `true`:

- generated actor glue includes the corresponding endpoint family
- generated endpoints reject non-controller callers
- mutation SQL remains fail-closed through the existing query-lane rejection
- routing remains canister-owned and accepted-schema-backed

Generated actor glue is the only source of SQL endpoint shape. Source files
should provide only the lifecycle hooks referenced by enabled config surfaces.

The normal `icydb-cli` binary may parse project `icydb.toml` at command runtime
for inspection, sync checks, and local endpoint-generation preflight
diagnostics. It must not embed arbitrary project
`icydb.toml` values at CLI compile time. That only works for the repository
demo and would not be a general project mechanism. For 0.156, SQL execution
should require explicit target selection:

- explicit `--environment` and `--canister`
- no IcyDB-owned default canister target

The CLI config preflight is only a local user diagnostic. Generated actor glue
and the canister's actual exported methods remain the runtime authority for
which endpoint calls can execute.

## Validation And Diagnostics

Build-time validation should reject:

- unknown top-level sections
- unknown fields
- non-boolean `sql.readonly`, `sql.ddl`, `sql.fixtures`,
  `metrics.enabled`, `metrics.reset`, or `snapshot.enabled`
- `[demo.sql]` or other default-target config
- a `canisters.<name>` entry that does not match a generated canister name
- duplicate or ambiguous canister names after normalization

Diagnostics should identify the config path, section, and key. Tests should
assert typed error variants or structured diagnostics, not string fragments.

## Cargo Integration

Build scripts that read `icydb.toml` must emit:

```text
cargo:rerun-if-changed=icydb.toml
```

If the config search walks up to the workspace root, it must also emit
`rerun-if-changed` for the resolved file and avoid broad directory watches.

The initial implementation should prefer one resolved manifest:

1. explicit path from build script input, if supplied
2. nearest `icydb.toml` found by walking up from the canister crate
3. absent config, treated as all booleans false

Absent config should be valid. Malformed present config should fail the build.
If both crate-local and ancestor manifests exist, the nearest manifest wins
entirely. 0.156 must not merge manifests or apply per-section overrides across
files.

## Example Build Flow

For demo RPG:

```text
icydb.toml
  -> [canisters.demo_rpg.sql] readonly=true, ddl=true
  -> build.rs emits typed config constants
  -> generated actor.rs exports the selected controller-gated SQL surfaces
```

For a production canister that does not want SQL:

```text
# Omit [canisters.ledger.sql], or leave both switches false.
```

Result:

- no generated SQL endpoint exports
- no TOML parser in wasm
- no runtime config read
- accepted schema/runtime behavior unchanged

## Implementation Order

The recommended order is:

1. Add `icydb-config-build` with strict parse, validate, and generated-Rust
   writing.
2. Support absent config and one explicit config path.
3. Generate typed Rust constants into `OUT_DIR`.
4. Wire one demo or fixture canister build to the generated config.
5. Move SQL endpoint generation behind the generated booleans.
6. Keep SQL endpoint shape owned by generated actor glue and config.
7. Add dependency-graph and raw non-gzipped wasm size guards.
8. Add CLI config display/check commands for the resolved DB SQL config.

This proves parser dependency isolation and generated endpoint shape before the
less critical operator UX cleanup.

## Validation Target

Implementation should include focused checks that prove:

- absent `icydb.toml` builds and disables all optional config surfaces
- malformed `icydb.toml` fails during build tooling validation
- `sql.readonly = true` produces the controller-gated readonly SQL endpoint
- `sql.ddl = true` produces the controller-gated DDL endpoint
- `sql.fixtures = true` produces the controller-gated fixture lifecycle endpoints
- unknown config sections and fields fail closed
- changing `icydb.toml` reruns the build script
- the wasm target dependency graph does not include TOML parser crates
- raw non-gzipped wasm size does not regress from parser dependencies

## Open Questions

- Should `icydb-config` exist as a TOML-free typed-model crate, or should
  `icydb-config-build` own both the parsed model and generated model privately
  for the first implementation?
