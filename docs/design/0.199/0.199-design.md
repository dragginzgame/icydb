# 0.199 Technical Debt Audit

Status: placeholder after 0.198 read-intent ergonomics.

0.199 is reserved for a focused technical-debt audit after the 0.197 exact-key
canonicalization and 0.198 read-intent ergonomics lines have settled.

This is an audit line, not an implementation line. The goal is to identify the
next small, high-leverage cleanup slices without mixing them into the
planner/admission/API work in 0.197 and 0.198.

## Context

0.197 changes exact primary-key proof ownership across planner, SQL/fluent
lowering, public read admission, EXPLAIN, cache identity, and executor route
selection.

0.198 changes developer-facing read intent vocabulary and may add or standardize
terminals such as `page(...)`, `collect_complete()`, and exact aggregate
helpers.

Those lines intentionally leave broader cleanup decisions out of scope. 0.199
should revisit the resulting code shape after the proof and API boundaries are
clear.

## Audit Goals

0.199 should answer:

- where planner/admission/executor proof ownership is still duplicated or
  drift-prone;
- whether 0.198 read-intent terminals introduced API or error-shape sprawl;
- whether raw `.limit(...)` migration guidance created obvious follow-up slices;
- where diagnostics, EXPLAIN, and read-admission wording still diverge;
- which modules are carrying the highest complexity, churn, or velocity cost;
- which technical-debt items are small enough to land as focused follow-up
  slices.

## Non-Goals

0.199 should not:

- implement 0.197 exact-key canonicalization;
- implement 0.198 read-intent ergonomics;
- introduce new public read policy builders;
- weaken read admission;
- change persisted formats, cursor formats, or schema snapshot authority;
- bundle broad refactors into one cleanup patch.

## Candidate Audit Tracks

- Planner/admission/executor proof ownership.
- Fluent and SQL read-intent API surface.
- Error taxonomy and diagnostics drift.
- EXPLAIN/read-admission evidence consistency.
- Raw-limit migration debt in docs, tests, and downstream examples.
- Complexity and module-structure pressure in query/session/executor code.
- Wasm and performance cost for any new 0.197/0.198 surfaces.

## Expected Outputs

0.199 should produce:

- an audit report or design update with prioritized findings;
- a short list of proposed cleanup slices;
- explicit "do not do now" items for broad or risky refactors;
- validation notes for any command-backed findings;
- wasm/perf impact guidance only where the audited surfaces changed size or
  runtime behavior.

## Closeout Criteria

0.199 can close when:

- the audit scope has been run against the current post-0.198 tree;
- findings are prioritized by risk and implementation size;
- each recommended cleanup has a clear owner boundary;
- no finding relies on speculative 0.197 or 0.198 behavior that has not landed;
- follow-up implementation work is split into separate design or cleanup slices.
