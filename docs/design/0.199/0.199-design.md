# 0.199 Technical Debt Audit

Status: planned audit line after 0.197 exact-key canonicalization and 0.198
read-intent ergonomics have landed and stabilized.

0.199 is a focused technical-debt audit across the current IcyDB codebase, with
special attention to the new proof, read-admission, read-intent, diagnostics,
and executor boundaries introduced or clarified by 0.197 and 0.198.

This is an audit line, not an implementation line. The goal is to produce an
evidence-backed cleanup plan: small, high-leverage follow-up slices that reduce
drift, duplication, API sprawl, diagnostic inconsistency, and module complexity
without reopening planner/admission/API design decisions.

The concern behind 0.199 is codebase control. IcyDB is gaining more layers of
semantic authority: planner proofs, SQL/fluent lowering, public read admission,
EXPLAIN evidence, cache identity, executor route choice, read terminals, cursor
semantics, and wasm-facing behavior. 0.199 should verify that these
responsibilities are still cleanly owned, testable, explainable, and cheap to
evolve.

## Reviewer Feedback

- The scope is pointed in the right direction: it frames 0.199 as an audit line
  and resists turning it into a cleanup mega-patch.
- The strongest artifact is the boundary matrix. Treat it as required closeout
  evidence, not as an optional checklist item.
- The main risk is audit sprawl. Seven tracks plus several inventories can
  become too broad unless every finding maps to either a small follow-up slice
  or an explicit "do not do now" item.
- The doc should keep forcing findings to distinguish landed behavior from
  intended 0.197/0.198 behavior. That guardrail is important while 0.198 work is
  still visible in the dirty tree.
- Recommended cleanup categories should not be treated as pre-approved patches.
  Each concrete slice still needs evidence, owner boundary, validation, and
  explicit forbidden changes.
- Keep wasm/perf observations evidence-labeled. When size is measured, raw
  non-gzipped `.wasm` bytes should be the primary signal; gzip is secondary
  context.

## Context

- 0.197 changes exact primary-key proof ownership across planner, SQL/fluent
  lowering, public read admission, EXPLAIN, cache identity, and executor route
  selection.
- 0.198 changes developer-facing read intent vocabulary and may add or
  standardize terminals such as `page(...)`, `collect_complete()`, and exact
  aggregate helpers.
- Those lines intentionally leave broader cleanup decisions out of scope. 0.199
  should revisit the resulting code shape after the proof and API boundaries are
  clear.
- 0.199 should not assume behavior that has not landed. Findings must be based
  on the actual post-0.198 tree, not on intended 0.197 or 0.198 designs.

## Audit Objective

0.199 should answer one primary question:

> Is IcyDB's current query/read path still simple enough to reason about,
> maintain, diagnose, test, and extend?

The audit should identify:

- Places where semantic ownership is duplicated.
- Places where public API vocabulary is spreading faster than the underlying
  model.
- Places where diagnostics, EXPLAIN, read admission, docs, and tests describe
  the same behavior differently.
- Modules that are becoming high-churn or high-complexity risk centers.
- Cleanup opportunities that are small enough to land independently.
- Risky broad refactors that should explicitly be deferred.

The expected result is not a giant cleanup patch. The expected result is a
prioritized cleanup map.

## Core Audit Questions

### 1. Proof Ownership and Semantic Authority

0.199 should determine whether exact-key proof ownership has a single source of
truth.

Answer:

- Where is exact primary-key proof created?
- Where is it consumed?
- Where is it re-derived, mirrored, copied, normalized, or inferred again?
- Are planner, read admission, EXPLAIN, cache identity, and executor route
  selection using the same proof representation or merely equivalent-looking
  local logic?
- Are there places where SQL and fluent APIs can accidentally produce divergent
  proof evidence?
- Are there places where executor route choice depends on a weaker, looser, or
  differently canonicalized condition than public read admission?
- Are tests proving the invariant once at the right boundary, or repeatedly
  patching symptoms at lower layers?

Preferred outcome:

- One clear owner for proof construction.
- One or a very small number of explicit proof consumers.
- No hidden proof reconstruction from strings, diagnostics, SQL fragments,
  lowered filters, or cache keys.
- Tests that assert boundary behavior rather than duplicate implementation
  details.

### 2. Planner/Admission/Executor Drift

0.199 should identify whether the planner, public read admission, and executor
have started to encode the same eligibility rules in multiple forms.

Check for drift between:

- Planner proof eligibility.
- SQL lowering proof metadata.
- Fluent lowering proof metadata.
- Read-admission checks.
- EXPLAIN evidence.
- Cache identity construction.
- Executor route selection.
- Test fixtures and expected diagnostics.

The audit should flag any pair of modules where changing one semantic rule would
require a developer to remember to update another manually.

High-risk examples:

- Exact-key logic repeated in both admission and executor.
- EXPLAIN reporting proof evidence that is not the same evidence admission used.
- Cache identity canonicalization depending on a representation that differs
  from executor routing.
- SQL path and fluent path producing equivalent behavior through different
  intermediate concepts.
- Tests blessing inconsistent wording or evidence shape across surfaces.

### 3. Read-Intent API Surface and Terminal Vocabulary

0.198 may introduce or standardize developer-facing read intent terminals such
as `page(...)`, `collect_complete()`, and exact aggregate helpers. 0.199 should
audit whether these terminals made the model clearer or merely added more public
API shapes.

Answer:

- Is the intended vocabulary small and coherent?
- Are terminal names aligned across fluent API, SQL behavior, diagnostics, docs,
  examples, and tests?
- Do terminals encode distinct semantics, or are some just aliases for old
  `.limit(...)` habits?
- Are incomplete reads, bounded reads, exact aggregates, and complete collection
  reads distinguishable in diagnostics?
- Are there overlapping helpers that produce the same admission behavior but
  different error messages?
- Is there any new ambiguity between "bounded page," "complete result," "exact
  aggregate," and "raw limit"?
- Is the API forcing callers to express intent at the correct boundary, or are
  internals still guessing intent from shape?

Preferred outcome:

- Minimal public terminal set.
- Each terminal has one semantic purpose.
- Old raw-limit semantics are either clearly legacy, internal, or documented as
  migration-only.
- Diagnostics guide users toward the right terminal without leaking internal
  planner terms.

### 4. Raw `.limit(...)` Migration Debt

0.199 should audit whether raw `.limit(...)` guidance introduced obvious
follow-up work.

Check:

- Docs that still recommend raw `.limit(...)` for reads that should use
  `page(...)`.
- Examples that rely on `.limit(...)` to satisfy read admission.
- Tests that encode legacy `.limit(...)` behavior without explaining why.
- Downstream-facing snippets that now teach the wrong model.
- Diagnostics that mention limit mechanics instead of read intent.
- SQL examples that do not map cleanly to fluent read terminals.
- Migration notes that are missing, stale, or too vague.

The audit should separate three categories:

- **Safe to modernize now:** docs, examples, tests, names, fixtures.
- **Needs design follow-up:** behavior where `.limit(...)` still has ambiguous
  public semantics.
- **Do not touch in 0.199:** actual behavior changes, cursor format changes,
  persisted read semantics.

### 5. Diagnostics, EXPLAIN, and Admission Wording Consistency

0.199 should verify that developers see a consistent explanation of read
behavior across failure paths, success introspection, and documentation.

Audit:

- Read-admission errors.
- Planner diagnostics.
- SQL/fluent error messages.
- EXPLAIN output.
- Cache/debug labels.
- Test fixture names.
- Docs and examples.
- Wasm-visible errors, if applicable.

Answer:

- Do these surfaces use the same vocabulary?
- Do they distinguish exact-key proof, bounded page, complete collection, and
  exact aggregate consistently?
- Are error messages actionable?
- Do messages identify the missing condition at the right abstraction level?
- Are internal details leaking into public errors?
- Is EXPLAIN describing the same proof/admission facts used by execution?
- Are diagnostics stable enough to test without making refactors painful?

Preferred outcome:

- One canonical diagnostic vocabulary.
- No parallel wording for the same concept.
- EXPLAIN and admission evidence agree structurally, not just textually.
- Public errors guide API users toward the correct terminal or query shape.

### 6. Complexity, Churn, and Module Pressure

0.199 should identify modules carrying disproportionate maintenance cost.

The audit should inspect, at minimum:

- Query planner modules.
- SQL lowering modules.
- Fluent query/session modules.
- Public read admission.
- Executor route selection.
- Cache identity/canonicalization.
- EXPLAIN generation.
- Error/diagnostic construction.
- Wasm bindings or wasm-facing API surfaces.
- Docs and public examples.
- Test fixtures around read admission and exact-key behavior.

For each pressure point, assess:

- Size and branching complexity.
- Number of concepts owned by the module.
- Churn during 0.197 and 0.198.
- Number of downstream test failures caused by small changes.
- Duplication of semantic checks.
- Public API exposure.
- Wasm/perf sensitivity.
- Amount of fixture text that must be updated for wording-only changes.
- Whether the module is a true owner or an accidental coordinator.

The audit should call out "god modules," "translation layers that became policy
layers," and "policy layers that became formatting layers."

### 7. Test Shape and Regression Confidence

0.199 should audit whether the test suite protects the new semantics without
overfitting implementation details.

Answer:

- Are exact-key proof invariants tested at the planner/admission boundary?
- Are executor route tests dependent on proof ownership or on local
  reconstruction?
- Do SQL and fluent paths have parallel coverage where they should?
- Do they share common expected semantics where they should?
- Are EXPLAIN tests proving meaningful evidence, or just snapshotting fragile
  text?
- Are diagnostic tests too brittle to allow cleanup?
- Are raw-limit migration tests clearly marked as legacy, compatibility, or
  active intended behavior?
- Are wasm tests covering public behavior without duplicating native internals?

The audit should propose test cleanup slices where test structure itself is
causing technical debt.

### 8. Wasm and Performance Impact

0.199 is not a performance optimization line, but it should flag debt that could
affect wasm size, runtime cost, or developer experience.

Audit wasm/perf only where 0.197 or 0.198 changed surface area or runtime
behavior.

Check:

- New public API symbols exposed to wasm.
- New diagnostic/error payload size.
- EXPLAIN output growth.
- Additional planner/admission passes.
- Canonicalization work repeated across layers.
- Aggregate helper overhead.
- Duplicated route-selection checks.
- Test-only helpers accidentally compiled into production or wasm builds.
- Added dependencies or formatting machinery.

Guidance:

- Do not optimize speculatively.
- Do not block small cleanup on unmeasured wasm concerns.
- Do flag obvious repeated canonicalization or large error/EXPLAIN payload
  growth.
- Perf claims must be command-backed or explicitly marked as code-inspection
  risk.

## Non-Goals

0.199 should not:

- Implement 0.197 exact-key canonicalization.
- Implement 0.198 read-intent ergonomics.
- Introduce new public read policy builders.
- Weaken read admission.
- Change persisted formats.
- Change cursor formats.
- Change schema snapshot authority.
- Change cache identity semantics except as a separately approved follow-up.
- Broaden or relax exact-key eligibility.
- Redesign SQL or fluent APIs.
- Add new planner capabilities.
- Rewrite the query planner.
- Rewrite the executor.
- Merge planner, admission, and executor responsibilities into one broad
  abstraction.
- Perform broad naming churn without a specific diagnostic or ownership win.
- Bundle unrelated cleanup into a single mega-patch.

## Evidence Requirements

Every finding should identify its evidence type.

Acceptable evidence types:

- **Command-backed:** produced by tests, linting, coverage, size checks,
  benchmarks, snapshot diffs, dependency reports, or code search.
- **Code-reading-backed:** based on direct inspection of module
  responsibilities, duplicated branches, API signatures, or call graph shape.
- **Fixture-backed:** based on expected outputs, test fixtures, EXPLAIN
  snapshots, diagnostic snapshots, or docs examples.
- **Design-backed:** based on conflict with accepted 0.197 or 0.198 design
  decisions.
- **Risk-only:** plausible risk found during inspection, but not yet proven by
  command or failing test.

Risk-only findings are allowed, but they should not be promoted to
implementation work without a validation step.

## Prioritization Rubric

Each finding should be scored on four axes.

### Risk

- **High:** likely to cause correctness drift, admission bypass, executor
  misrouting, cache identity inconsistency, or public API confusion.
- **Medium:** likely to cause maintenance cost, inconsistent diagnostics,
  brittle tests, or migration confusion.
- **Low:** naming, layout, fixture, or documentation debt with limited semantic
  impact.

### Implementation Size

- **Small:** one owner boundary, one module family, no public behavior change.
- **Medium:** multiple adjacent modules, fixture updates, no semantic redesign.
- **Large:** broad refactor, public API consequences, persisted/cursor/cache
  implications, or unclear blast radius.

### Confidence

- **High:** directly observed in code, tests, commands, or snapshots.
- **Medium:** strongly indicated by code shape but needs validation.
- **Low:** suspected pattern requiring deeper inspection.

### Urgency

- **Now:** should be the next cleanup slice before more planner/API work lands.
- **Soon:** should happen before the next related feature line.
- **Later:** worth tracking, but not blocking.
- **Do not do now:** too broad, too risky, or dependent on unsettled design.

Recommended cleanup candidates should generally be high or medium risk, small or
medium size, and high or medium confidence.

## Suggested Priority Classes

### P0: Correctness or Semantic Authority Risk

Use P0 for findings where two layers may disagree about whether a read is valid,
exact, cacheable, explainable, or executable through a fast/exact route.

Examples:

- Admission and executor use different exact-key eligibility checks.
- Cache identity is derived from a non-canonical representation.
- EXPLAIN evidence can claim exactness that admission did not prove.
- SQL and fluent paths produce different proof semantics for the same logical
  query.

P0 findings should produce small, immediate cleanup slices.

### P1: Public API or Diagnostic Drift

Use P1 for issues that make the system harder for users or downstream developers
to understand.

Examples:

- `page(...)`, `collect_complete()`, exact aggregate helpers, and `.limit(...)`
  are described inconsistently.
- Admission errors use old raw-limit vocabulary.
- Docs recommend patterns that 0.198 is trying to move away from.
- Wasm-facing errors differ from native errors without a clear reason.

P1 findings should usually become docs, diagnostics, or API consistency slices.

### P2: Maintenance and Test Brittleness

Use P2 for issues that slow development but do not currently threaten semantics.

Examples:

- Brittle EXPLAIN snapshots.
- Duplicate fixture blocks across SQL and fluent tests.
- Large test files that encode multiple policies at once.
- Helper functions whose names no longer match their role.
- Modules that coordinate too many concepts but are not yet causing semantic
  bugs.

P2 findings should become focused test or module-boundary cleanup slices.

### P3: Cosmetic or Deferred Debt

Use P3 for cleanup that is real but not urgent.

Examples:

- Local naming cleanup.
- Comment updates.
- Minor file organization.
- Opportunistic helper extraction.
- Broad refactors that need a later design line.

P3 findings should not block current roadmap work.

## Cleanup Slice Rules

Recommended slices should be narrow enough to review safely.

A good 0.199 follow-up slice should have:

- One owner boundary.
- One primary invariant.
- No persisted format changes.
- No cursor format changes.
- No admission weakening.
- No broad public API redesign.
- Clear tests.
- Clear rollback path.
- Minimal fixture churn.

Bad cleanup slices:

- "clean up planner"
- "simplify read admission"
- "refactor executor"
- "unify all diagnostics"
- "modernize query API"
- "remove all raw limit usage"
- "rewrite EXPLAIN output"

Better cleanup slices:

- "make EXPLAIN exact-key evidence read from planner proof instead of
  reconstructing local evidence"
- "rename internal diagnostic helper to match 0.198 terminal vocabulary"
- "split read-admission error construction from admission decision logic"
- "replace docs examples that use raw `.limit(...)` for paged reads"
- "deduplicate SQL/fluent exact-key proof fixture setup"
- "add a boundary test proving executor route selection consumes admitted proof"

## Explicit Do Not Do Now Items

0.199 should explicitly defer:

- Any query planner rewrite.
- Any executor rewrite.
- Any broad SQL/fluent API redesign.
- New public read policy builders.
- Changes to persisted data formats.
- Changes to cursor format or cursor compatibility.
- Changes to schema snapshot authority.
- Broad cache identity changes.
- Broad error taxonomy redesign unless limited to wording and ownership cleanup.
- Performance work without a measured regression or repeated runtime cost.
- Wasm API pruning that would become a breaking public change.
- Removing raw `.limit(...)` behavior unless already approved elsewhere.

The audit may recommend later design lines for these, but should not smuggle them
into cleanup slices.

## Closeout Criteria

0.199 can close when:

- The audit has been run against the actual current post-0.198 tree.
- The audited commit or branch is recorded.
- 0.197 and 0.198 landed behavior is distinguished from speculative intended
  behavior.
- Planner/admission/executor proof ownership has been mapped.
- SQL and fluent read-intent surfaces have been inventoried.
- Diagnostics, EXPLAIN, read admission, and docs wording have been compared.
- Raw `.limit(...)` migration debt has been classified.
- High-pressure modules have been identified.
- Findings are prioritized by risk, implementation size, confidence, and
  urgency.
- Each recommended cleanup has a clear owner boundary.
- Each recommended cleanup is split into a separate follow-up slice.
- Broad or risky refactors are listed under "do not do now."
- Validation notes distinguish command-backed findings from code-reading-only
  findings.
- Wasm/perf guidance is included only where audited surfaces changed size,
  exported API, or runtime behavior.

## Proposed Follow-Up Slice Categories

The audit should produce a short list of concrete slices. The actual names
should come from findings, but likely categories are:

### 1. Proof Ownership Cleanup

Goal: remove duplicated exact-key proof checks or reconstruction.

Owner boundary: planner/admission/executor.

Allowed:

- Helper relocation.
- Proof consumer cleanup.
- Boundary tests.
- Internal naming cleanup.

Forbidden:

- Changing exact-key eligibility.
- Changing read admission policy.
- Changing cache identity semantics.

### 2. Read-Intent Vocabulary Cleanup

Goal: align internal names, diagnostics, docs, and tests with 0.198 terminal
semantics.

Owner boundary: fluent/SQL API plus diagnostics/docs.

Allowed:

- Wording cleanup.
- Example migration.
- Fixture renames.
- Helper names.
- Docs updates.

Forbidden:

- New public read policy builders.
- Broad terminal redesign.
- Behavior changes to `.limit(...)`.

### 3. EXPLAIN Evidence Consistency Cleanup

Goal: ensure EXPLAIN reports the same facts used by planner/admission/executor.

Owner boundary: EXPLAIN generation.

Allowed:

- Read from existing proof/admission evidence.
- Reduce local reconstruction.
- Adjust tests to assert evidence rather than incidental wording.

Forbidden:

- Redesigning EXPLAIN format broadly.
- Adding new planner behavior.
- Weakening admission.

### 4. Admission Error Taxonomy Cleanup

Goal: reduce diagnostic drift and make errors actionable.

Owner boundary: read admission and public diagnostics.

Allowed:

- Consolidate error constructors.
- Standardize terms.
- Improve migration hints.
- Update fixtures.

Forbidden:

- Changing what is admitted.
- Introducing new public policy APIs.
- Broad error-code redesign without a separate design line.

### 5. Raw-Limit Docs and Test Migration Cleanup

Goal: stop examples and tests from teaching legacy `.limit(...)` as the
preferred read-intent mechanism.

Owner boundary: docs/tests/examples.

Allowed:

- Update examples to `page(...)` or `collect_complete()`.
- Mark legacy tests clearly.
- Add migration notes.

Forbidden:

- Removing compatibility behavior.
- Changing cursor or read semantics.

### 6. Module Pressure Relief Cleanup

Goal: split accidental coordination responsibilities from policy ownership in
one high-pressure module.

Owner boundary: whichever module the audit identifies as worst.
