# 0.120 Pipeline Unification & Bypass Elimination

## Thesis

`0.120` should eliminate execution pipelines that bypass or duplicate the
canonical:

```text
lowering -> planner intent -> staged route planning -> execution
```

After semantic centralization in `0.116` and `0.117`, flow contraction in
`0.118`, and subtractive cleanup in `0.119`, the next remaining velocity drag
is no longer semantic ambiguity.

It is pipeline multiplicity.

`0.120` should make the canonical staged pipeline the only meaningful route to
execution.

This is not a semantic slice.

It is not a feature slice.

It is not an executor redesign slice.

It is a topology-contraction slice whose job is to remove parallel execution
routes and force all shipped query families through one dominant planning and
execution spine.

---

## Why `0.120` Exists

The current line now reads roughly as:

* `0.116` centralizes truth-condition semantics
* `0.117` centralizes expression-family semantics
* `0.118` collapses the dominant expression-flow path
* `0.119` deletes obsolete expression-flow scaffolding and one abandoned
  prepared-session lane

That leaves a narrower but still important structural problem:

* multiple partially-overlapping paths can still reach execution
* route construction still has more than one entry shape
* prepared/session or diagnostics surfaces can still read as if they own part
  of planning topology
* equivalent logical operations still risk being reached through different
  physical pipeline shapes

The system can therefore be semantically correct and physically smaller than
before, while still carrying too many execution topologies.

That is why `0.120` exists.

It is the slice that should remove remaining bypass and duplicate pipeline
shapes rather than continuing to centralize meaning.

### Audit Alignment

#### 1. Canonical Semantic Authority Audit

The current canonical semantic authority line already reads as healthy:

* no missing canonical typed models
* no raw-string semantic reparse authority
* no major competing semantic owner

That matters because `0.120` should not reopen semantic authority work.

The job here is not to define meaning again.

It is to make execution reachability obey the authority shape that already
exists.

#### 2. Crosscutting Completeness Audit

The completeness line now reads as bounded rather than product-starved:

* no emergency prepared gap
* no obvious missing canonical model
* the remaining visible seams are structural rather than capability-shaped

That means `0.120` is not justified by a missing feature row.

It is justified by removing duplicated ways to reach the same runtime outcome.

#### 3. Velocity Preservation Audit

The velocity line remains the clearest direct motivator:

* median blast radius is still above the `<= 8` target
* route and expression-adjacent layers still behave as gravity wells
* the remaining cost is increasingly about how many entrypoints and route
  shapes remain, not just which layer owns semantics

That means `0.120` should be evaluated primarily as a velocity slice.

#### 4. Complexity Accretion Audit

Complexity pressure now reads less like semantic duplication and more like
parallel path maintenance:

* multiple route construction seams
* duplicated execution reachability paths
* planning decisions appearing in more than one place

The safest next complexity move is to remove duplicate topology, not invent
new abstraction.

#### 5. DRY Consolidation Audit

At this point, DRY pressure is best read at the pipeline level:

* duplicated route construction
* duplicate plan-entry helpers
* separate diagnostic or prepared walks over the same planned structure

`0.120` should remove those path-level duplications instead of coordinating
them better.

---

## Problem

The system still allows more than one way to reach execution for related
logical work.

Broadly, the remaining shapes read like:

* the canonical staged planner path
* mutation-oriented entrypoints that can bypass parts of staged planning
* load-oriented entrypoints that split fast-path and normal-path construction
* prepared/session-adjacent flows that can still read as partial planning
  participants
* diagnostics or explain walks that can still resemble parallel route
  traversals

These paths are not all equal in severity.

Some are hard bypasses.

Some are partial shadow paths.

Some are only dual-walk consumers.

But together they still create the same architectural cost:

* more modules touched per change
* duplicated route construction logic
* harder audit interpretation
* long-term LoC growth that no longer pays for correctness or capability

The problem is not “how many helper functions exist.”

The problem is:

> how many distinct ways execution can be reached for the same logical family

`0.120` should drive that count toward one canonical answer.

---

## Goal

Enforce a single dominant execution pipeline topology:

```text
lowering -> planner intent -> staged route planning -> execution
```

All shipped query families should reach execution through that topology.

Adjacent layers may still perform:

* syntax adaptation
* binding
* diagnostics formatting
* runtime shell transport

But they must not define parallel route-construction or execution-entry
pipelines of their own.

---

## Core Invariant

No execution plan may be constructed outside staged route planning.

Corollaries:

* no direct entrypoint-to-execution shortcut for a shipped query family
* no duplicated route construction paths for the same logical operation
* no second-stage planning in prepared/session layers
* no alternate structural walk that recreates planning topology for the same
  semantic structure

A thin adapter branch is allowed.

A second planning topology is not.

---

## Scope

### In Scope

* route planner entrypoints
* mutation execution path
* load planning entrypoints
* prepared/session planning flow
* access, explain, and diagnostics traversal shapes where they duplicate route
  topology

### Out Of Scope

* new SQL features
* semantic changes
* executor redesign
* storage or index redesign
* planner semantic authority work already handled in earlier slices

---

## Ownership Boundary

`0.120` should make the pipeline owner split explicit.

### Lowering Owns

* parsed statement adaptation
* construction of planner-compatible intent inputs

Lowering must not own:

* alternate route construction
* direct execution-plan assembly

### Planner Owns

* intent interpretation
* staged route planning
* final execution topology construction

This is the only owner that should decide how execution is reached.

### Prepared / Session Owns

* binding
* contract transport
* execution wiring

Prepared/session must not own:

* second-stage planning
* alternate route construction
* local re-derivation of planner topology

### Diagnostics / Explain Own

* rendering
* reporting
* consumption of planner-owned route topology

Diagnostics and explain must not own:

* parallel route walks that recreate planning meaning
* alternate structural planning traversals for the same route family

---

## Pipeline Inventory

### Canonical Pipeline To Keep

* lowering -> intent -> staged planning -> execution

### Pipelines To Remove Or Collapse

#### 1. Mutation Bypass

Current read:

* mutation entrypoint
* direct execution-oriented construction
* partial or missing staged planner participation

Required contraction:

* mutation should produce planner intent
* mutation should flow through staged route planning
* direct execution construction should disappear

This is the strongest first target because it is the clearest hard bypass.

#### 2. Load Dual Entry

Current read:

* load path A
* load path B or fast path
* duplicated entry decisions split across helpers

Required contraction:

* one entrypoint
* planner-owned fast-path decision inside the canonical route flow

#### 3. Prepared / Session Shadow Planning

Current read:

* prepared/session still risk reading as if they own plan-shape decisions
* local flow branching can still look like partial route construction

Required contraction:

* prepared/session become binding, contract transport, and execution wiring
  only
* planner produces final structure

#### 4. Access / Explain Dual Traversal

Current read:

* planner consumes one access topology
* explain or diagnostics can still look like a second structural walk

Required contraction:

* one canonical traversal contract
* explain/diagnostics consume planner-owned artifacts or one shared traversal

#### 5. Access Choice vs Projection Divergence

Current read:

* multiple consumers can still walk access structure differently enough to act
  like distinct topology readers

Required contraction:

* one planner-owned access traversal contract

---

## Concrete Actions

### 1. Enforce One Route Construction Point

All execution plans must originate from staged planner ownership.

Delete or redirect:

* direct execution-plan construction helpers outside planner
* any route-entry helper that bypasses staged planning

### 2. Collapse Mutation Into The Canonical Pipeline

Mutation should:

* produce planner-owned intent
* flow through staged planning
* arrive at execution through the same route topology as the rest of the
  system

### 3. Collapse Load Entry Points

Merge load entry helpers so:

* there is one load route-entry surface
* fast-path eligibility is a planner-owned internal decision

### 4. Reduce Prepared / Session To Contract Layers

Prepared/session should retain only:

* parameter binding
* contract mapping
* execution wiring

Remove:

* template re-matching that recreates route shape
* local projection or route re-derivation
* flow branching that behaves like second-stage planning

### 5. Unify Access / Explain Traversal

Enforce one canonical access traversal contract.

Planner, explain, and diagnostics should consume that contract rather than
building separate structural walks.

### 6. Add Pipeline Invariants

Tests and guards should enforce:

* no execution plan created outside planner
* no duplicate entrypoints for the same shipped query family
* no second planning walk over the same planner structure
* no prepared/session re-planning

---

## Recommended Work Order

### Phase 1

* eliminate mutation bypass
* unify load entrypoints

This is the highest-leverage first step because it removes the clearest hard
and semi-hard bypasses.

### Phase 2

* collapse prepared/session shadow planning

This should happen only after the canonical route topology is unambiguous.

### Phase 3

* unify access/explain traversal

This should follow once execution reachability is already singular.

---

## Exit Condition

`0.120` is complete when:

* exactly one meaningful execution pipeline topology exists
* no direct execution-plan construction remains outside planner
* no duplicated entrypoints remain for the same shipped query family
* prepared/session are contract-only
* access and explain consume one planner-owned traversal contract

This should be visible as:

* reduced module touch count per slice
* reduced LoC in route, session, and execution-entry clusters
* lower hotspot concentration around route entry and flow ownership
* velocity median blast radius `<= 8`

### Velocity

* fewer modules touched per change
* reduced route-entry gravity wells
* clearer change boundaries

### Complexity

* fewer branching sites at the pipeline level
* fewer duplicated route-construction paths

### DRY

* elimination of pipeline-level duplication rather than helper-level trimming

### Canonical Semantic Authority

---

## Non-Goals

`0.120` should not:

* reopen semantics
* add features
* redesign the executor wholesale
* flatten every boundary in the system
* delete useful adapter branches that are only structural and do not create
  parallel planning topology

---

## Summary

`0.120` should not ask “how many helper paths exist?”

It should ask:

> how many ways can execution be reached for the same logical family

Then it should remove all but one meaningful answer.

The intended end state is simple:

* lowering produces planner-owned intent
* planner owns staged route construction
* execution consumes that staged route
* prepared/session and diagnostics become consumers, not alternate pipelines

If `0.116` through `0.119` make the system semantically correct and physically
smaller, `0.120` should make execution topology singular.
