# 0.43 - BYTES Query Terminal

## Status

In Progress  
Target: `0.43`  
Dependency: `0.42` EXPLAIN surfaces (complete)  
Last updated: 2026-03-06

---

## 1. Objective

Add a scalar load terminal `bytes()` that returns the total row-payload size
for the effective query window.

The goal is query-level size introspection for debugging and tuning:

- estimate payload footprint of filtered windows
- compare count vs payload size on the same predicate
- detect unexpectedly large entity windows before paging/export

`0.43` keeps this as a terminal-surface feature and does not introduce new
execution operators.

---

## 2. Non-Goals

- no global storage accounting (`TABLE_BYTES`, `INDEX_BYTES`, disk/block usage)
- no standalone SQL parser/CLI scope
- no new access-path or planner execution-shape features
- no grouped `aggregate(bytes())` in this line
- no explain-node vocabulary expansion

---

## 3. Scope (`0.43.x`)

`0.43.x` ships scalar fluent terminal support only:

- `FluentLoadQuery::bytes() -> Result<u64, QueryError>`
- same non-paged policy gate as existing scalar terminals
- semantics tied to the same effective scalar response window used by `execute()`

Grouped aggregate syntax support is explicitly deferred.

---

## 4. Semantic Contract

### 4.1 Definition

`bytes()` returns:

`sum(row_payload_len)`

where `row_payload_len` is the persisted raw row payload byte length.

### 4.2 Window Semantics

Rows included in the sum are exactly the rows that survive the canonical scalar
execution window:

- access path + predicate evaluation
- DISTINCT strategy
- ordering strategy
- limit/offset window

No new admission policy is introduced for `bytes()`.

### 4.3 Policy and Consistency

`bytes()` inherits existing scalar load consistency behavior:

- no separate missing-row policy
- no separate predicate semantics
- no grouped execution mode

### 4.4 Output and Overflow

- output type: `u64`
- accumulation: saturating `u64` addition
- empty window returns `0`

Planned API shape:

```rust
let total_bytes = session
    .load::<User>()
```

Note: examples remain fluent-query based (not SQL text), consistent with the
current runtime API boundary.

## 6. Sanity Check Notes

The original draft had several scope/architecture mismatches that are corrected
here:

- SQL-style examples were replaced with fluent API examples.
- Grouped support is no longer assumed; grouped `bytes` is deferred.
- Implementation ownership now references real modules in this repo (for
  example `query/fluent/load/terminals.rs`, `executor/load/terminal.rs`)
  instead of non-existent placeholder paths.
- EXPLAIN impact is scoped to existing execution surfaces; no new explain node
  families are required for scalar `bytes()`.

## 7. Implementation Slices (`0.43.x`)

Sequence labels are patch-agnostic by design.

### 7.1 Slice A - Fluent Terminal Surface

- add `bytes()` to `FluentLoadQuery` terminal API
- gate through `ensure_non_paged_mode_ready()`
- wire terminal dispatch through `execute_load_query_with(...)`

### 7.2 Slice B - Executor Terminal Implementation

- add `LoadExecutor::bytes(plan)` under `executor/load/terminal.rs`
- keep logic terminal-local; do not mutate planner route/access contracts
- return `u64` with saturating accumulation

### 7.3 Slice C - Measurement Path

V1 measurement path:

- execute canonical scalar plan
- sum persisted row payload lengths for rows in the effective result window

This prioritizes semantic correctness over first-pass optimization.

### 7.4 Slice D - Explain and Docs Alignment

- keep existing `explain_execution*()` behavior unchanged
- document that `bytes()` uses canonical scalar execution window semantics
- update `CHANGELOG.md` + `docs/changelog/0.43.md` when shipped

### 7.5 Slice E - Tests

Add deterministic coverage for:

- empty window -> `0`
- full scan window
- index-backed filtered window
- `order_by + limit + offset` window
- DISTINCT interaction
- parity sanity against materialized scalar response window

## 9. Risks and Follow-Ups

Known tradeoff in v1:

- implementation favors correctness and reuse of canonical scalar behavior over
  deep kernel-level byte-accumulator optimization.

Deferred follow-ups:

- grouped `aggregate(bytes())` support
- projection-level byte metrics (`bytes_by(field)` style surfaces)
- optional fast path that accumulates payload size during lower-level row
  materialization boundaries

---

## 10. Exit Criteria

- scalar `bytes()` terminal is available on fluent load queries
- result semantics match canonical scalar execution window behavior
- deterministic tests cover window/predicate/distinct/order interactions
- no execution-vocabulary expansion required for this line
