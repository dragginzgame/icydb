## Title

**0.71 — Aggregate Execution Simplification**

---

## Status

Draft

This is the next minor-line design after `0.70` closed planner-gated index
visibility for the session-backed planner/executor path.

The `0.71` target is not another planner-correctness redesign.
It is the aggregate execution subsystem itself.

---

## Context

`0.70` settled the index-correctness question:

* only `Ready` indexes are planner-visible
* planner visibility owns correctness
* execution trusts planned visible indexes

That means aggregate execution no longer needs its own correctness or
"authority" story.

The next pressure point is different:

* aggregate execution remains one of the densest internal runtime areas
* aggregate domains are mostly partitioned correctly, but not always
  implemented consistently within those partitions
* the typed scalar aggregate lane still shows small but real setup overhead,
  even after recent cleanup

So the `0.71` line should not reopen planner visibility.
It should make aggregate execution internally coherent.

---

## Current Baseline

The current tree already moved partway toward a cleaner aggregate shape:

* `COUNT` / `EXISTS` now share one existing-rows execution family
* order-sensitive terminals now have an explicit prepared family instead of
  living as ad hoc scalar-terminal exceptions
* numeric aggregate prep now separates boundary shape from payload family

That is the right direction.

What remains is to treat those partitions as the design contract, not just as
local cleanup.

---

## Problem Statement

Aggregate execution is no longer one monolith, but it still carries internal
inconsistency:

* duplicate strategy vocabulary for similar work
* partial use of prepared boundaries
* duplicate setup across typed SQL and runtime preparation
* explain/descriptor risk when prepared strategy and rendered meaning drift
* small repeated setup costs in the typed scalar lane

The top-level domain split is mostly correct.
The implementation inside those domains is not fully finished.

---

## Goals

### Primary Goal

Make each aggregate execution domain internally consistent:

* one explicit prepared boundary
* one strategy vocabulary
* one execution path per domain

### Secondary Goals

* reduce duplicate setup in typed scalar aggregate execution
* keep aggregate explain surfaces derived from prepared strategy, not
  reconstructed ad hoc
* improve scanability and maintenance without reopening correctness policy

---

## Non-Goals

`0.71` does not attempt to:

* redesign planner-visible index correctness
* add a new aggregate correctness or authority model
* widen aggregate semantics
* add cost-based aggregate planning
* unify all aggregate families into one generic runtime
* broaden grouped/global distinct behavior beyond current semantics

This is an execution-structure cleanup line, not a new capability line.

---

## Core Design Rule

The aggregate subsystem should be partitioned by execution domain, not by
historical helper shape.

That means:

* keep fundamentally different aggregate domains separate
* make each domain internally consistent
* do not collapse those domains into one "aggregate blob"

---

## Aggregate Domain Partition

The intended partition for `0.71` is:

### 1. Existing-Rows Aggregates

* `COUNT`
* `EXISTS`

Invariant:

* depends only on existing-row traversal
* final result differs only in how the shared scan result is finalized

### 2. Scalar Id / Extrema Aggregates

* `min`
* `max`
* `min_by`
* `max_by`

Invariant:

* chooses one extremal winner
* does not share the same execution structure as numeric accumulation

### 3. Order-Sensitive Terminals

* `first`
* `last`
* `nth_by`
* `median_by`
* `min_max_by`

Invariant:

* ranking or order-sensitive selection is the primary execution concern

### 4. Numeric Aggregates

* `sum`
* `avg`

Invariant:

* numeric accumulation
* numeric coercion and null semantics are primary

### 5. Projection / Distinct Aggregates

* `count_distinct_by`
* projection-style aggregate terminals

Invariant:

* slot extraction, set semantics, and projection ownership dominate

This partition is the design contract for `0.71`.

---

## Design Principles

### 1. Keep Legitimate Domain Splits

Do not unify:

* existing-row aggregates with numeric aggregates
* order-sensitive terminals with numeric aggregates
* projection/distinct behavior with existing-row behavior

Those are different execution problems.

### 2. Remove Accidental Divergence Within A Domain

Inside one domain, do remove:

* duplicated strategy enums
* dead flags
* duplicate setup paths
* half-prepared / half-ad-hoc entrypoints

### 3. Prepare Once, Carry Once

Preparation should resolve:

* slots
* aggregate op shape
* execution family

once, then carry that state into execution and explain.

### 4. Explain Must Project Prepared Strategy

Aggregate explain/descriptor output should describe the chosen prepared
strategy.

It should not re-derive aggregate meaning from scattered route symptoms.

### 5. Performance Claims Must Stay Measured

`0.71` should prefer cleaner code first, but any claimed runtime win must be
validated with the existing query/perf matrix.

---

## Behavioral Invariant

Aggregate preparation is the single behavioral source.

That means:

* preparation decides the aggregate execution domain
* preparation decides the execution family within that domain
* execution consumes prepared strategy
* runtime descriptor surfaces consume prepared strategy
* `EXPLAIN` consumes prepared strategy

What must not happen:

* execution re-classifies aggregate domain or policy locally
* descriptor assembly reconstructs aggregate behavior from route symptoms
* `EXPLAIN` and runtime drift because they each own a separate branch tree

Shared helpers are fine.
Parallel behavioral classification is not.

---

## Proposed Design

### A. Existing-Rows Family

Treat existing-row aggregation as one real family.

The family should have:

* one prepared boundary
* one execution strategy shape
* one shared scan/reducer path

`COUNT` versus `EXISTS` should differ only at finalization.

The important rule is:

* no separate strategy vocabulary for the same traversal dependency
* no dead state such as covering/correctness bits that execution does not use

### B. Order-Sensitive Family

Treat order-sensitive terminals as their own explicit prepared family.

They should not survive as exceptions hidden inside the scalar terminal
matcher.

This family should own:

* response-order-sensitive operations
* rank-target operations
* any per-order selection semantics that do not share the existing-row or
  numeric structure

### C. Numeric Family

Numeric aggregate prep should stay split into:

* boundary: field/op contract
* payload: direct aggregate versus grouped distinct execution family

The grouped global distinct path should remain its own payload family, not a
fake numeric strategy case.

### D. Typed Scalar Setup Path

Typed SQL aggregate execution should stop rebuilding setup that planner/lower
stages already know.

The design target is:

* resolve target slots once
* avoid repeated owned-field-name and boundary reconstruction where possible
* keep typed and fluent aggregate entrypoints consuming the same prepared
  execution shape

This is the most likely place for real setup-cost wins.

This is also the best first coding slice because it is:

* local
* high leverage
* low policy risk
* a direct way to establish the prepare-once/project-everywhere pattern for
  the rest of `0.71`

### E. Explain / Descriptor Parity

Aggregate explain and descriptor assembly should consume prepared aggregate
strategy directly.

The design target is:

* one prepared meaning
* one explain projection of that meaning

not:

* one runtime strategy
* plus a second describe/explain reconstruction path

---

## Rollout Order

### Slice 1. Typed Scalar Preparation Flattening

* flatten duplicated typed-scalar setup sites
* introduce one prepared typed-scalar strategy shape
* make runtime descriptor and `EXPLAIN` consume that prepared strategy
* keep planner visibility and aggregate eligibility unchanged

### Slice 2. Numeric Domain Normalization

* finish accidental divergence cleanup inside the numeric family
* keep `sum` / `avg` under one clear boundary-plus-payload model
* align empty-window, nullability, and reject-path behavior with the prepared
  strategy surface

### Slice 3. Order-Sensitive Lane Isolation

* keep the order-sensitive family explicit and self-contained
* avoid drifting it back into scalar-id/extrema just because some helpers look
  similar

### Slice 4. Projection / Distinct Normalization

* standardize prepared strategy, descriptor, and runtime setup
* keep projection/distinct handling explicit instead of bespoke

### Slice 5. Existing-Rows Cleanup

* finish remaining historical coupling in the existing-rows lane
* keep this later in the line because it is the most likely to hide legacy
  branching that existed only because it shipped first

### Slice 6. Targeted Perf Follow-Up

Only after the structure is clean:

* run the query matrix
* target measured setup overhead in the typed scalar lane
* keep or revert local cleanups based on measured outcome

---

## Success Criteria

`0.71` is successful if:

* aggregate domains remain clearly separated by execution invariant
* each domain has one explicit prepared boundary
* typed scalar aggregate setup is flatter and easier to inspect
* aggregate explain surfaces stay aligned with prepared strategy
* no aggregate correctness or planner-visibility logic is reintroduced into
  execution

---

## What `0.71` Should Not Become

It should not become:

* a new planner-visibility design line
* a broad index-exploitation umbrella
* a generic optimizer rewrite
* a monolithic "all aggregates share one runtime" abstraction

Those would blur the now-clean `0.70` boundary and overreach the next useful
subsystem slice.

---

## Final Framing

`0.70` made index-backed execution safe by planner visibility.

`0.71` should make aggregate execution structurally honest:

* keep distinct execution domains distinct
* remove accidental divergence inside those domains
* simplify setup before chasing new behavior
