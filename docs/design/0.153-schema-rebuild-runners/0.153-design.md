# 0.153 Schema Rebuild Runners

## Purpose

0.152 made schema mutation planning catalog-native and fail-closed. It can name
metadata-only changes, rebuild-required index work, cleanup-required index
drops, unsupported mutations, runner capabilities, and publication preflight,
but it intentionally does not execute physical rebuild work.

0.153 is the next stage: add the first physical schema mutation runners behind
the 0.152 preflight boundary. The goal is not SQL DDL. The goal is to make
selected rebuild-required mutation plans physically true before accepted schema
publication.

The closeout target for 0.153 is:

Schema mutation plans that require supported index rebuild or cleanup work can
run through a schema-owned runner, validate the resulting store state, invalidate
runtime planning state, and publish the accepted snapshot atomically. Unsupported
or incomplete work still fails closed.

## Non-Goals

0.153 should not add:

- broad SQL DDL syntax
- arbitrary type changes
- slot remapping
- primary-key rewrites
- full row rewrites
- unique constraint add/drop policy
- concurrent online index build
- generalized migration scripts
- generated metadata fallback reconstruction

DDL remains a later frontend over the mutation engine. Full data rewrites remain
a later migration system.

## Hard Invariants

- Accepted schema snapshots remain the catalog authority.
- Runner input must be `SchemaMutationExecutionPlan`, not generated
  `EntityModel` or `IndexModel` authority.
- Runner work starts only after `SchemaMutationRunnerContract` preflight says
  the required capabilities are present.
- Rebuilt or cleanup-affected stores must remain staged-only until publication.
- Physical work is not publishable until validation has passed.
- Snapshot publication is atomic with runtime visibility.
- Failed rebuild or cleanup restores the previous visible store state or leaves
  the mutation unpublished.
- Accepted schema publication changes runtime schema identity so stale planner,
  cache, visible-index, and prepared-plan state cannot be reused.
- Missing accepted rebuild metadata is an incompatible accepted-schema error.
- Before `1.0.0`, runner state formats hard-cut to the latest contract.

## Runner Boundary

The runner boundary should consume three things:

```rust
struct SchemaMutationRunnerInput {
    accepted_before: AcceptedSchemaSnapshot,
    accepted_after: PersistedSchemaSnapshot,
    execution_plan: SchemaMutationExecutionPlan,
}

enum SchemaMutationRunnerResult {
    ReadyToPublish(SchemaMutationRunnerReport),
    Rejected(SchemaMutationRunnerRejection),
}
```

The names are illustrative. The important boundary is that the runner consumes
accepted snapshot contracts and execution steps that were already lowered by the
schema mutation layer. It should not reclassify mutation compatibility and
should not reopen generated model authority.

## Execution Phases

Each supported physical mutation should follow the same phase order:

1. Preflight runner capabilities against the execution plan.
2. Reserve or stage affected index stores.
3. Build or clean physical index data from accepted row/index contracts.
4. Validate physical work against accepted contracts.
5. Invalidate planner/cache/runtime schema state.
6. Publish the accepted schema snapshot.
7. Record runner diagnostics.

The runner must not publish between phases 2 and 4. It can only publish after
validation proves that the physical store state satisfies the accepted target
contract.

## Store Visibility

Staging is a first-class storage contract, not a best-effort implementation
detail. Rebuild and cleanup work must encode whether affected index state is
visible to runtime readers.

The initial contract can be small:

```rust
enum SchemaMutationStoreVisibility {
    StagedOnly,
    Published,
}
```

or it can be represented by a narrow staged-store wrapper. Either shape must
make this invariant explicit:

physical rebuild output is not planner-visible, read-visible, or write-visible
until publication flips it from staged to published.

This boundary prevents in-place partially visible rebuilds. It also gives later
online builds, crash recovery, resumable rebuilds, replication, and concurrent
schema changes a durable state model to extend instead of reverse-engineering
visibility from ad hoc store contents.

## Initial Supported Work

### Add Non-Unique Field-Path Index

Supported when the mutation plan carries a `SchemaFieldPathIndexRebuildTarget`.

The runner must:

- scan authoritative rows for the entity
- derive keys from accepted row-layout slots and field-path metadata
- apply the accepted predicate contract
- write forward index entries
- write reverse index entries when required by current index storage semantics
- validate key count and membership parity
- reject corrupt rows without publishing the new snapshot

Unique indexes remain out of scope until uniqueness proof and conflict reporting
are explicit runner contracts.

### Add Deterministic Expression Index

Supported when the mutation plan carries a `SchemaExpressionIndexRebuildTarget`
for the existing deterministic expression subset.

The runner must:

- evaluate expression keys from accepted source slots
- use accepted expression op, input kind, output kind, and canonical text
- apply accepted predicate metadata
- write the same forward/reverse index shape as live writes
- validate rebuilt keys against accepted expression contracts

The runner must not evaluate arbitrary SQL expression trees.

### Drop Non-Required Secondary Index

Supported when the mutation plan carries a
`SchemaSecondaryIndexDropCleanupTarget`.

The runner must:

- remove or retire the accepted secondary index store state
- invalidate planner/cache state before the dropped index can be chosen again
- publish the accepted snapshot only after cleanup has completed

Unique secondary index drops remain out of scope until constraint-removal policy
exists.

## Validation Model

Validation should be a named runner phase, not an incidental assertion inside
the build loop.

Initial validation should prove:

- every rebuilt forward index entry can be traced back to a live row
- every indexed live row admitted by the predicate has a matching index entry
- reverse membership agrees with forward membership where reverse indexes are
  maintained
- expression keys match accepted expression contracts
- dropped index stores are no longer visible to planner/runtime selection
- planner/cache invalidation happened before publication

Validation semantics are authoritative in the first implementation:

- validation is a full-scan check over the entity's visible live rows
- deleted/tombstoned rows do not participate unless the storage layer exposes
  an explicit live-row view that includes them for recovery diagnostics
- duplicate index entries are rejected unless the accepted index contract says
  the duplicate shape is legal for a non-unique key
- ordering is validated only where the store contract promises ordered keys;
  membership correctness is the primary invariant
- no MVCC or concurrent snapshot semantics are assumed in 0.153; if a future
  runner needs concurrency, it must first add an explicit rebuild snapshot
  contract

Validation can be expensive in the first implementation. It is better for 0.153
to be correct and fail-closed than to optimize rebuilds prematurely.

## Failure And Rollback

The runner needs a rollback story before publication can widen.

Acceptable first implementation:

- keep the old accepted snapshot visible until all physical work validates
- write rebuilt state into isolated/staged index storage where possible
- on failure, remove staged state and return a classified runner rejection
- never publish the new snapshot after a failed physical phase

If staging cannot be isolated for one store shape, that mutation remains
unsupported until the storage boundary can provide rollback semantics.

## Snapshot Identity And Runtime Epochs

Accepted schema publication must change runtime schema identity. The concrete
mechanism can be a runtime schema epoch, accepted snapshot fingerprint, or a
combined generation token, but all planner/cache surfaces need one shared
concept:

```rust
struct RuntimeSchemaEpoch { ... }
```

or equivalent.

Publication must invalidate or distinguish at least:

- visible-index sets
- access-choice and route caches
- SQL/prepared-plan caches
- accepted row-layout decode plans
- explain/diagnostic plan snapshots
- cursor or continuation signatures that depend on row/index shape

The epoch should be derived from accepted catalog publication, not generated
model metadata. A rebuild that validates but does not publish must not advance
runtime identity. A failed publication must not leave partial epoch changes
behind.

## Runtime Visibility

Runtime visibility changes only after successful runner validation and snapshot
publication.

The publication boundary should continue to be schema-owned:

```rust
preflight -> physical runner -> validation -> invalidation -> publish
```

No executor, planner, SQL, or generated model code should publish accepted
schema snapshots directly.

## Metrics And Diagnostics

0.153 should add low-cardinality runner diagnostics:

- mutation kind
- rebuild requirement
- runner capability set
- runner phase
- rows scanned
- index keys written
- validation failures
- rollback failures
- publish outcome

Diagnostics should not require matching error strings. Tests should assert
typed outcomes and metrics buckets where practical.

## Patch Sequence

1. Add runner result, rejection, report, and phase contracts.
2. Add a no-op runner adapter that accepts metadata-only plans and rejects
   physical-work plans without capabilities.
3. Add explicit staged-store visibility contracts and guards.
4. Add runtime schema epoch/snapshot-publication identity contracts.
5. Add staged runner wiring at the reconciliation publication boundary without
   changing runtime behavior.
6. Implement non-unique field-path index rebuild execution.
7. Add validation and rollback coverage for field-path rebuilds.
8. Implement deterministic expression-index rebuild execution for the existing
   accepted expression subset.
9. Add validation and rollback coverage for expression rebuilds.
10. Implement non-required secondary index cleanup.
11. Add runtime visibility/cache invalidation checks.
12. Close the minor with audit coverage and keep SQL DDL deferred.

## Success Criteria

0.153 is complete when:

- supported index rebuild and cleanup mutation plans can become publishable only
  after runner preflight, physical execution, validation, and invalidation
- staged store state cannot be observed by runtime planning, reads, or writes
  before accepted snapshot publication
- accepted snapshot publication advances or changes runtime schema identity so
  stale planner/cache/prepared state cannot be reused
- unsupported or incomplete mutation plans still fail closed
- accepted schema/index contracts remain the only runtime authority
- generated metadata remains limited to proposal/model-only/test lanes
- SQL DDL remains a future frontend, not part of runner semantics
