## 1. Release Goal

Finalize and harden the post-load execution architecture by:

- stabilizing executor module boundaries
- formalizing execution layer responsibilities
- consolidating shared contracts
- improving `EXPLAIN` visibility for the new execution layers
- stabilizing runtime observability/introspection surfaces

This line introduces no new query-surface features.

---

## 2. Background

Earlier versions accumulated heterogeneous execution logic under:

- `db::executor::load`

That surface eventually held scan mechanics, pipeline orchestration, aggregate
runtime logic, terminal reducers, and fast-path routing in one area, which
increased coupling and reduced architectural clarity.

`0.48` completed the structural split into dedicated roots:

```text
db::executor/
  scan/
  pipeline/
  aggregate/
  terminal/
```

The transitional `load` compatibility facade has been removed. `0.49` focuses
on stabilizing the resulting architecture and formalizing boundary contracts.

## 3. Execution Architecture

### 3.1 Final Layer Model

The executor is modeled as a deterministic stream pipeline:

`scan -> pipeline -> aggregate -> terminal`

### 3.2 Layer Responsibilities

#### scan

Physical data-source traversal.

Examples:

- primary-key scans
- index-range scans
- fast streaming scans

Responsibilities:

- produce ordered key/value streams
- enforce access-plan boundaries
- expose scan metadata consumed by `EXPLAIN`

#### pipeline

Execution orchestration.

Responsibilities:

- stage orchestration
- execution contracts
- continuation handling
- fast-path routing

Pipeline does not implement scan internals or aggregation engines.

#### aggregate

Aggregation engines.

Responsibilities:

- `GROUP BY`
- `DISTINCT`
- grouped pagination
- aggregate state machines

Aggregation logic must remain independent of scan implementations.

#### terminal

Final reducers and result shaping.

Examples:

- `LIMIT`
- ranking
- page materialization
- final projection shaping

Terminal operators never depend on scan implementations.

### 3.3 Execution Stream Contract

Execution streams must satisfy these invariants:

- ordered streams preserve key ordering emitted by the scan layer
- pipeline stages must not reorder rows unless reordering is explicitly
  required by `ORDER BY` semantics
- aggregation may change ordering only when group semantics require it
- terminal operators preserve input order unless explicitly documented
- identical plan fingerprints must produce deterministic stream behavior

### 3.4 Fast-Path Stability

Fast-path execution (`fast_stream`, `fast_stream_route`, and pipeline fast-path
selection) must satisfy:

- fast-path selection is deterministic for a given plan fingerprint
- fast-path rejection reasons are observable via `EXPLAIN`
- fast-path execution does not change semantic output shape

### 3.5 Continuation Stability

Executor continuation tokens must encode:

- plan fingerprint
- ordering contract
- grouping contract
- projection shape contract

Continuation decoding must reject mismatched plan fingerprints or contract
signatures.

## 4. Module Ownership Rules

### 4.1 Scan Isolation

Pipeline may interact with scans only through explicit execution contracts.

Allowed:

- `pipeline -> scan request contract`
- `scan -> execution stream contract`

Forbidden:

- `pipeline -> direct index/scan implementation access`

### 4.2 Aggregation Independence

Aggregation modules must not depend on scan implementations.

Allowed:

- `aggregate -> pipeline contracts`
- `aggregate -> projection evaluation contracts`

Forbidden:

- `aggregate -> pk_stream internals`
- `aggregate -> secondary_index internals`

### 4.3 Terminal Isolation

Terminal operators:

- may consume pipeline or aggregate outputs
- must not access scan internals
- must not access planner internals

## 5. Shared Contracts

`0.49` consolidates cross-layer execution contracts into explicit owner-named
modules.

Primary contract surfaces:

- `db::executor::pipeline::contracts` (or equivalent pipeline contract root)
- `db::executor::stream`
- `db::executor::projection`
- `db::executor::runtime_context`

Contract categories:

- execution inputs
- stream interfaces
- page materialization contracts
- continuation metadata

These contracts are the only supported cross-layer interfaces.

`shared/` must not become a generic catch-all namespace; cross-layer contracts
should live in owner-named modules (for example `contracts`, `projection`,
`stream`, and `runtime_context`).

### 5.1 Executor Namespace Rule

`db::executor::shared` must not exist.

Cross-layer helpers must live under the module that owns the concept (for
example `pipeline::contracts`, `projection`, `stream`, and `context`).

---

## 6. Visibility Rules

To prevent cross-layer leakage:

- default shared execution types to `pub(in crate::db::executor)`
- prefer `pub(super)` in leaf modules unless wider visibility is required by an
  explicit contract
- avoid widening visibility to satisfy convenience imports

Visibility widening requires an explicit boundary rationale and corresponding
structural guard coverage.

## 7. EXPLAIN Improvements

`0.49` extends `EXPLAIN` with additive execution-layer metadata for the
stabilized architecture.

### 7.1 Layer Visibility

Each execution node reports its owning layer:

- `scan`
- `pipeline`
- `aggregate`
- `terminal`

Example:

```json
{
  "node_id": 3,
  "node": "IndexRangeScan",
  "layer": "scan",
  "index": "users_email_idx",
  "prefix_len": 2
}
```

### 7.2 Fast-Path Visibility

Execution nodes expose:

- `fast_path_selected: true|false`
- `fast_path_reason: <string>`

### 7.3 Pushdown Visibility

Execution nodes expose predicate pushdown status:

- `predicate_pushdown_mode: full | partial | none`

### 7.4 Streaming vs Materialized Visibility

Execution nodes expose:

- `execution_mode: streaming | materialized`
- `streaming: true|false`
- `materialization_reason: <optional>`

### 7.5 Aggregation Strategy Visibility

Grouped execution exposes:

- `group_strategy: streaming | materialized`
- `distinct_budget: <n>`

All additions are observability-only and must not alter planner or runtime
behavior.

## 8. Structural Tests

`0.49` adds or tightens structural tests that enforce execution boundaries.

Coverage goals:

- module ownership checks for forbidden import directions
- explicit guardrails preventing:
  - `aggregate -> scan` coupling
  - `terminal -> scan` coupling
  - `terminal -> planner` coupling
- executor-tree layout checks for the stabilized top-level model:
  - `scan`
  - `pipeline`
  - `aggregate`
  - `terminal`

### 8.1 Structural Guard Scripts

Structural checks must verify:

- aggregate modules do not import scan modules
- terminal modules do not import scan modules
- terminal modules do not import planner modules
- pipeline modules do not import planner internals
- scan modules do not import aggregation engines
- `db::executor::shared` does not exist

These checks should run as CI invariants to prevent architectural drift.

## 10. Observability & Introspection Stabilization

`0.49` consolidates and stabilizes runtime observability surfaces without
changing execution semantics.

Focus areas:

- `EXPLAIN` rendering consistency
- execution-layer introspection
- metrics surface stability
- correlation between execution nodes and runtime telemetry

### 10.1 EXPLAIN Schema Consistency

Execution explain rendering must preserve deterministic field ordering and
consistent property naming across node families.

Goals:

- stable JSON output for equivalent plans
- deterministic property ordering
- consistent field naming conventions
- snapshot coverage for representative execution plans

### 10.2 Execution Node Correlation

Execution nodes expose stable identifiers:

- `node_id`

These identifiers enable correlation between:

- `EXPLAIN` nodes
- execution metrics
- debugging traces

`node_id` assignment must remain stable for a given execution plan shape.

### 10.3 Execution Mode Introspection

Execution nodes expose execution-mode metadata:

- `execution_mode: streaming | materialized`
- `execution_mode_detail`

These fields support diagnosis of pipeline downgrades and materialization
boundaries.

### 10.4 Predicate Pushdown Diagnostics

Execution nodes expose pushdown visibility:

- `predicate_pushdown_mode: full | partial | none`

This metadata makes predicate split points explicit between:

- storage-layer pushdown
- executor-side residual filtering

### 10.5 Metrics Surface Stabilization

Runtime metrics expose row-flow counters:

- `rows_scanned`
- `rows_filtered`
- `rows_aggregated`
- `rows_emitted`

Metrics emission remains additive and must not affect runtime behavior.

## 12. EXPLAIN Subsystem Internal Stabilization

`db::query::explain` is already partially split (`mod.rs`, `plan.rs`,
`execution.rs`, `render.rs`, plus helper modules). `0.49` treats this as
second-stage internal hardening rather than a first-time split.

Goal:

- stabilize explain internals before larger post-`0.49` feature work lands

Target layout:

```text
db/query/explain/
  mod.rs
  plan.rs
  execution.rs
  render.rs
  json.rs
  nodes/
    scan.rs
    pipeline.rs
    aggregate.rs
    terminal.rs
```

Ownership by file:

- `plan.rs`: planner-side explain DTO projection
- `execution.rs`: executor-side explain DTO projection
- `render.rs`: text/JSON rendering entrypoints
- `json.rs`: deterministic JSON helper routines and ordering rules
- `nodes/*`: node-family property extraction and normalization

This work is observability-only and must preserve runtime semantics.

## 14. Pipeline Orchestration Stage-Descriptor Cleanup

`db::executor::pipeline` already includes staged entrypoint boundaries, but
`0.49` can still reduce orchestration noise by separating three concerns:

- stage sequencing/orchestration
- stage policy selection
- stage execution behavior

Goal:

- keep pipeline orchestration minimal and deterministic
- express route decisions as stage descriptors instead of branching trees
- preserve existing execution semantics and planner behavior

Target internal module shape:

```text
db/executor/pipeline/stages/
  mod.rs
  stage.rs
  planner.rs
```

Stage descriptor model (example):

- `PipelineStage::Scan(...)`
- `PipelineStage::Filter(...)`
- `PipelineStage::Aggregate(...)`
- `PipelineStage::Terminal(...)`

Execution loop shape:

- build deterministic stage list from existing routed contracts
- execute stages in-order over execution stream/context contracts

Fast-path and grouped/scalar route differences should be represented as stage
descriptors or stage configuration, not nested orchestration branching in the
entrypoint root.

Constraints:

- no query-surface changes
- no planner semantic changes
- no result-shape changes
- no continuation-envelope semantic changes

This is an internal architecture cleanup and observability-preserving
refactor.

### 14.1 Structural Consolidation Evaluation Queue

The following executor structural refactors are tracked for `0.49` evaluation.
Each item is evaluated independently and may be accepted or rejected on its own
merits. Rejection of one item does not block others.

| R2 | Clarify execute/runtime naming | `pipeline/execute` | `pipeline/runtime` or `pipeline/strategy` (if ownership matches) | Low | Accepted/Completed |

| R4 | Clarify execution context naming | `executor/context` | rename only if it improves ownership clarity (`runtime_context`/equivalent) | Low | Accepted/Completed |
| R5 | Unify continuation authority | split across `executor/continuation`, `pipeline/grouped_runtime/continuation`, `route/contracts/continuation` | one top-level continuation authority with consumed contracts | Medium | Accepted/Completed |
| R6 | Re-home kernel operator modules | `executor/kernel/{distinct,reducer,post_access}` | `pipeline/operators/*` (or equivalent stage-owned placement) | Medium | Accepted/Completed |

### 14.2 Evaluation Rule (One-by-One)

For each candidate (`R1` ... `R7`), `0.49` evaluation must record:

- boundary rationale (why ownership improves)

Only accepted candidates become completion blockers; unaccepted candidates remain
documented as deferred follow-up.

## 17. Completion Criteria

`0.49` is complete when:

- `db::executor::load` no longer exists
- executor module layout matches the stabilized layer model
- `db::executor::shared` no longer exists
- cross-layer contracts live under owner modules
- `EXPLAIN` execution metadata remains stable and deterministic
- `db::query::explain` internals are stabilized with explicit render/JSON/node
  ownership boundaries
- runtime metrics expose row-flow counters
- execution diagnostics provide stable node-correlation metadata and additive
  row-flow counters
- pipeline orchestration is reduced to stage planning + deterministic stage-loop
  execution with stage-owned behavior
- accepted structural-consolidation candidates (`R1`...`R7`) are completed with
  explicit accept/reject decisions recorded
- fast-path execution behavior is deterministic
- execution stream contracts are explicit and enforced
- continuation envelopes remain executor-compatible and fail closed on mismatch
- structural tests enforce executor layer boundaries
- CI invariants reject `executor::shared` references and namespace reintroduction

Rationale:

After a major internal refactor, a stabilization release should lock the
architecture before introducing new features. This preserves clarity and lowers
risk for subsequent capability work in `0.50+`.
