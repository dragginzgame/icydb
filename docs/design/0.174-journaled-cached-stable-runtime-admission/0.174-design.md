# IcyDB 0.174 — Journaled Cached-Stable Runtime Admission

## Status

Active runtime-admission design. 0.174.0 completed the admission-readiness
check after 0.173 closed the journaled protocol foundation. Runtime behavior
remains unchanged until later 0.174 slices land the full admission path.

## Thesis

0.174 should publicly admit the first `storage(journaled(...))` runtime store
only after 0.173 has fixed the protocol contract for commit-marker-bound
journal publication, journal batch ordering, fold watermark semantics, schema
replay, and index materialization.

The runtime model remains:

```text
canonical stable data/index/schema BTrees
+ committed journal tail after fold watermark
+ volatile Rust BTree live cache/read view
```

`storage(heap())` remains volatile. `storage(journaled(...))` is a separate
journaled cached-stable durable storage contract.

## Admission Gate

Public derive admission is the last step, not the first.

0.174 must not accept public `storage(journaled(...))` until the same releasable
branch contains:

- generated journaled wiring;
- journaled runtime capabilities;
- journaled data/index/schema store construction;
- bounded/fallible journal codec;
- commit-marker-bound journal publication;
- recovery from canonical stable BTrees plus committed journal tail;
- fold implementation and interruption tests;
- diagnostics rendering;
- relation policy;
- stable and heap regression proof;
- source-audit guards.

0.174 must implement the 0.173 chosen protocol decisions exactly:

- commit-marker journal publication strategy;
- journal-tail ordering authority;
- fold watermark semantics;
- schema replay ordering;
- index truth/materialization rule.

If implementation discovers that a 0.173 decision is not viable, stop and
revise the 0.173/0.174 designs rather than silently switching to an implicit
hybrid.

## 0.174.0 Readiness Baseline

0.174.0 starts the runtime-admission line without admitting journaled runtime
storage.

Readiness findings:

- 0.173 is closed as the protocol foundation;
- the embedded bounded journal batch payload strategy is binding;
- monotonic `journal_sequence` remains the journal-tail ordering authority;
- the fold watermark remains the durable replay boundary;
- schema replay must use the reconstructed accepted schema timeline;
- index BTree state remains derived materialized state, not independent journal
  truth;
- public derive syntax remains reserved until generated wiring, runtime
  construction, recovery, fold, diagnostics, relation behavior, and source
  audits all pass in the same branch;
- no runtime behavior changes are part of 0.174.0.

0.174 implementation slices must revise the design before changing any 0.173
protocol decision.

## Storage Form

The only admitted form is:

```rust
#[store(
    name = "sessions",
    storage(journaled(
        data_memory_id = 40,
        index_memory_id = 41,
        schema_memory_id = 42,
        journal_memory_id = 43,
    )),
)]
```

Memory roles:

- `data_memory_id`: canonical durable row BTree source of truth;
- `index_memory_id`: canonical durable index BTree materialized from data plus
  accepted schema;
- `schema_memory_id`: canonical durable schema BTree source of truth;
- `journal_memory_id`: durable journal tail for committed mutations not yet
  folded into the canonical stable BTrees.

Journal-only and malformed forms stay rejected. There are no checkpoint
memories.

## Runtime Admission Scope

0.174 should implement:

- schema and derive admission for the full four-ID form;
- generated wiring for journaled stores;
- journaled runtime capabilities and allocation metadata;
- bounded/fallible journal batch codec;
- marker-bound append/read journal-tail storage;
- journaled data/index/schema store wrappers;
- live public write/read/index behavior;
- recovery from canonical stable BTrees plus committed journal tail;
- fold into canonical stable BTrees;
- diagnostics rendering;
- relation policy through capability axes.

## Runtime Contracts

Journaled writes:

- append marker-bound journal batches;
- update live Rust cache projections only after durable publication has a
  recovery path;
- do not apply row/schema changes into canonical stable BTrees during the
  normal write path;
- reach canonical stable BTrees through fold.

Journaled recovery:

- resolves existing commit-marker state first;
- replays only complete committed journal batches after the durable fold
  watermark;
- rebuilds live caches from canonical stable BTrees plus journal tail;
- rebuilds or validates indexes from recovered rows and accepted schema;
- fails closed on malformed, partial, future-version, marker-mismatched,
  duplicate, missing, or out-of-order journal state that cannot be resolved
  deterministically.

Journaled fold:

- applies committed journal records into canonical stable BTrees;
- validates or rebuilds affected indexes;
- advances the fold watermark only after stable mutations are durable;
- advances or clears the journal tail only after the watermark is durable;
- remains correct after interruption at every fold phase.

## Capability Mapping

Journaled stores expose:

- allocation identity: present, journaled four-role allocation;
- durability: durable;
- recovery: stable base plus journal replay;
- commit participation: durable;
- schema metadata persistence: canonical stable schema history plus committed
  journal tail;
- relation source: durable source;
- relation target: durable target;
- live validation: supported.

Policy roots must consume capability axes, not concrete `Journaled` branches,
except at accepted schema validation, generated wiring, store construction,
registry capability derivation, and tests/proof fixtures.

## Diagnostics Contract

Diagnostics render journaled stores distinctly:

- storage: `journaled`;
- allocation kind: journaled cached-stable;
- canonical data/index/schema memory IDs;
- journal memory ID;
- durability: durable;
- recovery: stable base plus journal replay;
- commit participation: durable.

Diagnostics remain projections and must not become capability authority.

## Relation Policy

Journaled stores may participate as durable relation sources and durable
relation targets because recovery reconstructs their rows, indexes, and schema
metadata.

Validation-only reads do not make journaled targets mutation participants.
Heap relation behavior remains unchanged.

## Non-Goals

- No durable behavior for `storage(heap())`.
- No checkpoint memories.
- No journal-only admission.
- No compatibility aliases.
- No new SQL storage syntax.
- No migration tooling.
- No independent durable index journal truth.
- No relation semantic expansion beyond capability-based journaled durability.
- No stable ABI change for existing `storage(stable(...))` stores.

## Patch Slices

### 0.174.0 — Admission Readiness Check

- Verify 0.173 protocol foundation is closed.
- Re-read commit-marker publication strategy, journal ordering, fold watermark,
  schema replay, and index truth decisions.
- Treat those 0.173 decisions as binding. Do not switch strategies during
  implementation without revising the designs.
- Keep runtime behavior unchanged.

### 0.174.1 — Schema, Derive, And Build Wiring

- Admit the full four-ID public form.
- Reject malformed forms.
- Add generated wiring and registry allocation metadata.
- Keep stable and heap wiring unchanged.

### 0.174.2 — Journal Runtime Store And Codec

- Implement bounded/fallible journal batch codec.
- Implement append/read journal-tail storage.
- Bind journal batches to existing commit-marker authority.

### 0.174.3 — Runtime Store Wrappers And Live Behavior

- Add journaled data/index/schema wrappers.
- Route public write/read/index behavior through wrappers.
- Prove live cache projections are not durability authority.

### 0.174.4 — Recovery

- Recover from canonical stable BTrees plus committed journal tail.
- Rebuild or validate indexes.
- Fail closed on malformed journal state.

### 0.174.5 — Fold

- Fold committed journal tail into canonical stable BTrees.
- Prove fold watermark and interruption safety.
- Bound replay growth through fold behavior.

### 0.174.6 — Relations, Diagnostics, And Closeout

- Prove relation policy.
- Render diagnostics.
- Close A1-A19 with source-audit, stable, heap, codec, recovery, fold, and
  relation evidence.

## Closeout Language

0.174 should close only when the line can honestly say:

```text
0.174 admits the first public storage(journaled(...)) runtime store using the
0.173 protocol foundation. Journaled stores use canonical stable
data/index/schema BTrees as durable base, marker-bound journal batches as the
committed tail, and volatile Rust BTree caches as live projections. Recovery
rebuilds from the stable base plus committed journal tail, and fold safely
applies committed tail records into the canonical stable BTrees without loss or
double-apply.
```
