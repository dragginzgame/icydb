# 0.208 Exact Composite Contracts

Status: implementation complete; focused closeout validated on 2026-07-21;
full repository suites remain user-owned push validation

Closeout evidence: [IcyDB 0.208 changelog](../../changelog/0.208.md)

## Decision

0.208 replaces the former generic persisted `Structured { queryable }` field
kind with exact, nominal record, tuple, and newtype contracts owned by accepted
schema.

There is no open-ended persisted field kind. A custom Rust type is persistable
only when its generated proposal provides a complete composite shape that can
be reconciled into the accepted composite catalog. The general runtime `Value`
union remains an internal execution carrier; it is not a schema contract.

This is a pre-1.0 hard cut. The old accepted schema representation, codec
discriminator, generated methods, documentation, and tests are removed rather
than decoded, translated, aliased, or retained beside the current form.

## Problem

The old `Structured { queryable }` kind described only a coarse runtime
property. It could not prove:

- nominal Rust type identity;
- record member names and nullability;
- tuple arity and element contracts;
- newtype payload contract;
- recursive enum, collection, or composite membership; or
- whether two structurally similar values were actually the same declared
  type.

That made generated code an attractive fallback authority and left admission,
storage, query capability, and schema transition code without one exact fact to
consume. A boolean queryability flag also mixed schema identity with derived
planner policy.

## Scope

0.208 owns:

- generated exact composite proposals;
- one store-local accepted composite catalog;
- ID-backed composite references in accepted field kinds;
- exact recursive admission and persisted decoding;
- accepted schema encoding, fingerprinting, reconciliation, and transition
  validation for composites;
- capability derivation for whole composite fields and nested accepted paths;
- hard-cut removal of the generic structured field contract; and
- focused correctness, corruption, and resource-bound evidence.

0.208 does not add:

- an open or heterogeneous persisted field contract;
- schema-less JSON/document storage;
- runtime fallback reconstruction from generated models;
- implicit structural typing between nominal Rust types;
- arbitrary nested query traversal;
- compatibility decoding for the former accepted schema form; or
- a new public query architecture.

Generated `EntityModel` and `FieldKind` values are proposal and
generated-compatible decode evidence only. Once an accepted schema exists,
runtime admission and execution consume accepted IDs and definitions.

## Public And Generated Model

The generated model describes one composite with:

```rust
FieldKind::Composite {
    path,
    codec: CompositeCodec::StructuralV1,
    shape,
}
```

`path` is the nominal identity proposed by generated code. `shape` is exactly
one of:

```rust
CompositeShapeModel::Record(&[CompositeFieldModel])
CompositeShapeModel::Tuple(&[CompositeElementModel])
CompositeShapeModel::Newtype(CompositeElementModel)
```

Record members carry stable persisted names, exact nested kinds, and
nullability. Tuple elements preserve position and arity. Newtypes remain
nominal and do not collapse into their inner scalar or collection kind.

Empty records and empty tuples are distinct shapes. They must not be conflated
through an empty-vector sentinel.

## Accepted Composite Catalog

Each accepted store revision owns a canonical `AcceptedCompositeCatalog`.
Catalog construction:

1. discovers all composite proposals reachable from entity fields, enum
   payloads, collections, and other composites;
2. rejects empty or conflicting nominal paths;
3. assigns deterministic non-zero `CompositeTypeId` values;
4. resolves nested enum and composite references to accepted IDs;
5. canonicalizes record-member identity;
6. rejects incompatible duplicate definitions;
7. rejects recursive contract cycles and excessive depth; and
8. validates the complete catalog before publication.

Accepted fields store only the ID-backed reference:

```rust
AcceptedFieldKind::Composite { type_id }
```

The complete definition belongs to the catalog, not to every field and not to
the executor.

## Required Invariants

1. Every persisted field has one exact accepted recursive contract.
2. Every composite reference resolves in the accepted composite catalog of the
   same accepted schema authority.
3. Nominal paths are unique within a store-local accepted catalog.
4. A repeated nominal path must have one identical codec and shape.
5. Record member names are non-empty and unique.
6. Tuple arity and element order are exact contract facts.
7. Newtypes remain distinct from their inner kind.
8. Nested nullability is validated at the member or element boundary that owns
   it.
9. Enum and composite references are validated against the same accepted
   bundle.
10. Contract graphs are bounded and acyclic.
11. Malformed, unknown, missing, or wrong-shape values fail closed.
12. Runtime code never reconstructs a missing composite definition from a
    generated model.

## Admission

Accepted admission consumes the composite type ID, resolves the definition,
and validates the input recursively.

### Records

- the input is a map of member name to value;
- every accepted member appears exactly once unless its accepted nullability
  permits an explicit null at that position;
- unknown, duplicate, or missing required members reject;
- each member is normalized against its accepted nested kind; and
- canonical output follows the accepted member order.

### Tuples

- the input is a list;
- the list length must equal the accepted arity;
- each position is normalized against the accepted element contract; and
- position is semantic and is never reconstructed by field name.

### Newtypes

- the input carries exactly one payload;
- the payload is normalized against the accepted inner contract; and
- the nominal composite identity remains part of schema authority even when
  the physical payload resembles its inner value.

Admission shares the existing accepted-value depth and byte budget. Invalid
shape does not produce an empty value, partial row, or alternate decode path.

## Persistence

Exact composites use `CompositeCodec::StructuralV1` and the canonical recursive
value-storage lane. `FieldStorageDecode::CatalogValue` means:

1. decode a bounded canonical recursive value;
2. resolve the accepted composite contract;
3. validate the decoded value against that exact contract; and
4. expose it only after accepted admission succeeds.

The value-storage union is an internal codec mechanism. Its ability to carry
different runtime variants does not authorize a field whose accepted kind is
unknown or variable.

The accepted schema bundle persists both enum and composite catalogs. Current
bundle, enum-catalog, and composite-catalog envelopes have one current magic
and version. Prior development forms fail with typed persisted-format or
corruption errors.

## Defaults And Omission

Composite field omission follows the same accepted absence policy as every
other field. Rust `Default` is not database default authority.

- required composite field omitted: reject;
- nullable composite field given explicit `NULL`: admit null;
- database default declared and accepted: materialize that exact accepted
  default;
- generated Rust constructor default only: no database omission right.

Nested record members and tuple/newtype elements do not gain omission semantics
from the outer Rust type. Their accepted shape and nullability are exhaustive.

## Query And SQL Capabilities

Whole composite fields have no implicit scalar comparison, ordering, grouping,
aggregate, key, relation, or index capability. Capability is derived from
accepted schema; it is not stored as a boolean on the field kind.

Nested path support exists only where accepted publication has produced an
explicit path contract and the consuming planner operation supports that path.
The executor must not inspect a runtime value and infer capability from the
value currently present.

SQL and fluent queries consume the same accepted capability facts. Unsupported
whole-composite predicates or order/group terms reject during schema-aware
lowering or validation before execution.

## Schema Reconciliation And Transition

Generated composite proposals may create the initial accepted catalog or be
compared with an existing accepted catalog during reconciliation. They do not
replace accepted authority at runtime.

Transition validation compares nominal identity, codec, shape, member names,
tuple arity, nested kinds, and nullability. An incompatible change must be
expressed through an explicitly supported schema transition; it must not be
accepted because two runtime values happen to decode.

Publication treats the accepted schema snapshot and its enum/composite bundle
as one authority. A field reference may not be published without its referenced
catalog definition.

## Errors

The maintained error boundary distinguishes at least:

- unknown composite type;
- composite shape mismatch;
- composite record-member mismatch;
- accepted contract mismatch;
- bounded decode failure;
- corrupt or unsupported persisted catalog; and
- incompatible schema transition.

Errors retain their typed class and origin through public projection. String
matching is not part of the contract.

## Hard-Cut Deletion Inventory

0.208 deletes:

- `FieldKind::Structured { queryable }`;
- `AcceptedFieldKind::Structured { queryable }`;
- the structured accepted wire discriminator;
- generated schema output that emits the generic kind;
- executor/planner decisions based on the old queryability boolean;
- comments and tests that describe open-ended structured persistence; and
- compatibility decoding, aliases, and transition helpers for that form.

No anti-resurrection test should preserve the spelling. Current-surface tests
must prove exact composite behavior instead.

## Performance And Size Evidence

Focused native microbenchmarks on the 2026-07-21 closeout tree used 20,000
iterations per case. The normalization measurements include construction of an
owned input for each iteration; canonical validation borrows an already
normalized value.

| Shape | Normalize | Canonical validation |
| --- | ---: | ---: |
| Three-member exact record | 299 ns/op | 25 ns/op |
| Comparable three-entry typed map | 290 ns/op | 37 ns/op |
| Two-element exact tuple | 176 ns/op | 15 ns/op |
| Record containing the exact tuple | 275 ns/op | 27 ns/op |

Record admission uses one temporary authored-member vector for deterministic
name ordering and one canonical output vector. Tuple admission allocates only
its canonical output vector, while a newtype adds no container allocation of
its own. Canonical validation walks borrowed catalog and value state without a
temporary collection. Nested composites repeat only the allocation required by
each contained record or tuple.

For one 22-byte persisted exact-record payload, canonical decoding measured
195 ns/op; decoding plus validation against accepted composite authority
measured 258 ns/op.

The corrected exact-only SQL performance canister is 3,933,708 raw
non-gzipped Wasm bytes. That remains 60,286 bytes (1.51%) smaller than the
discarded 3,993,994-byte open-ended experiment. The closeout corrections add
2,950 raw bytes (0.08%) relative to the initial 0.208 exact-only capture because
accepted introspection now renders the complete catalog-owned shape and its
fail-closed error path.

The bounded correction batch touches 30 core files and is approximately
+1,166/-346 lines before documentation. Most additions are invariant and
regression tests. Production structure becomes simpler at SQL projection and
session catalog ownership, while publication validation and exact diagnostic
rendering add necessary invariant logic rather than another execution path.

## Final Architectural Truth

After 0.208, a persisted custom Rust value is an exact nominal composite. Its
accepted type ID, codec, complete shape, nested kinds, and nullability are
schema authority. The old generic `Structured {}` contract is gone, and no
loose replacement exists.
