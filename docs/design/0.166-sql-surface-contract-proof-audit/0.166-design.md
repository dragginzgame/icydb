# IcyDB 0.166 SQL Surface Contract Proof Audit

## Status

Closed.

The SQL surface matrix records direct public-entrypoint evidence for the
supported single-entity SQL families and fail-closed public-entrypoint evidence
for the major explicitly unsupported SQL families. The final matrix has no
supported rows classified as partially proven, unproven, contradicted, or
accidentally accepted.

Future SQL contract changes must update the matrix in the same slice as the
behavior, documentation, or test change that alters the public surface.

## Summary

0.166 is a post-0.165 correctness and confidence audit for IcyDB's public SQL
surface.

0.165 closed the broad naming hard cut and role-alignment pass. The next
production-confidence gap is not whether more names can be cleaned up. It is
whether the SQL behavior IcyDB claims to support is directly proven through the
public APIs users call.

The central question for 0.166 is:

```text
Does the documented SQL contract match the live public SQL surface?
```

0.166 should turn the SQL contract, 1.0 feature contract, and 1.0 TODO list into
an explicit proof matrix:

```text
documented SQL claim
-> public SQL entrypoint behavior
-> direct tests
-> explicit unsupported cases
-> known gaps
```

## Source Documents

The audit starts from the contract, not from implementation convenience.

Primary contract sources:

- `docs/contracts/SQL_SUBSET.md`
- `docs/architecture/SQL_SURFACE_MAPPING.md`
- `docs/1.0-FEATURES.md`
- `docs/1.0-TODO.md`

Primary public SQL entrypoints:

- `execute_sql_query::<E>(...)`
- `execute_sql_update::<E>(...)`
- `execute_sql_ddl::<E>(...)`

`docs/contracts/SQL_SUBSET.md` remains the normative SQL contract. The other
documents provide supporting feature, TODO, and implementation-facing mapping
claims that must not contradict the normative contract.

## Motivation

IcyDB now has a cleaner internal vocabulary after 0.165, but production
confidence depends on stronger proof around the shipped SQL contract.

Missing features are acceptable when they are intentionally excluded and
documented. The higher-risk cases are contract drift and accidental acceptance:

- A documented SQL feature is not actually supported.
- A supported SQL behavior is only tested through private or internal helpers.
- SQL behavior works accidentally but is not part of the contract.
- Unsupported SQL syntax is accepted but only partially implemented.
- `SQL_SUBSET.md`, `SQL_SURFACE_MAPPING.md`, `1.0-FEATURES.md`, `1.0-TODO.md`,
  and live behavior disagree.
- Public SQL response shapes are not directly tested.
- Error behavior for excluded SQL is unstable or ambiguous.

0.166 should convert those risks into visible classifications: proven,
partially proven, unproven, contradicted, intentionally unsupported, and
accidentally accepted.

## Non-Goals

- No broad naming audit.
- No SQL feature expansion campaign.
- No parser generosity because another SQL engine accepts a syntax family.
- No widening of SQL behavior merely to satisfy user expectations beyond the
  documented IcyDB contract.
- No replacement of public-surface tests with parser, planner, executor, or
  storage-only tests.
- No performance campaign before the correctness surface is contract-stable.
- No patch-number sequencing in this design.

## Guiding Principle

The audit starts from documented claims.

For every documented supported behavior, ask:

```text
Is this behavior directly tested through the public SQL entrypoint users call?
```

For every documented exclusion, ask:

```text
Is this SQL rejected deliberately and predictably through the public entrypoint?
```

For every behavior that happens to work, ask:

```text
Is this intentionally part of the SQL contract, or accidental acceptance?
```

Internal parser, lowering, planner, executor, and storage tests can support the
audit, but they do not prove public SQL support by themselves.

## Classification Model

Every audited SQL behavior should land in one of these states.

| State | Meaning |
| --- | --- |
| Proven | Documented, implemented, and directly tested through the relevant public SQL entrypoint. |
| Partially proven | Documented and at least partly implemented, but coverage is incomplete or not public-surface direct. |
| Unproven | Documented or implied, but no direct public-surface test proves it. |
| Contradicted | Documentation and live public behavior disagree. |
| Intentionally unsupported | Explicitly outside the SQL subset and covered by fail-closed public-surface tests. |
| Accidentally accepted | Accepted by parser/lowering/execution without a supported contract and complete semantic proof. |

Accidentally accepted SQL should not remain unresolved. It should be rejected
fail-closed, promoted into the contract with direct tests, or recorded as a
named follow-up decision before 1.0.

## Durable Artifact

The first durable artifact is the SQL surface proof matrix:

```text
docs/design/0.166-sql-surface-contract-proof-audit/sql-surface-matrix.md
```

Each row should map one documented or discovered SQL behavior to its contract
source, public entrypoint, test evidence, classification, and resolution. Where
practical, the row should include the exact SQL statement or statement family
used as test evidence.

Suggested row shape:

```text
Contract item:
  SELECT field projection

Claim source:
  docs/contracts/SQL_SUBSET.md
  docs/1.0-FEATURES.md
  docs/1.0-TODO.md

Claim:
  Simple field projection is supported for entity-backed SELECT queries.

Public entrypoint:
  execute_sql_query::<E>(...)

SQL evidence:
  SELECT name FROM Customer WHERE age >= 18 ORDER BY name LIMIT 10

Expected behavior:
  Query returns projected fields in the public SQL response shape.

Positive public-surface tests:
  Existing:
    - test name, if present
  Missing:
    - test case description, if absent

Negative / fail-closed tests:
  Existing:
    - test name, if present
  Missing:
    - unsupported case description, if absent

Current classification:
  Proven / Partially proven / Unproven / Contradicted /
  Intentionally unsupported / Accidentally accepted

Observed gap:
  What is missing, ambiguous, or inconsistent.

Resolution:
  Add public-surface test / update docs / reject unsupported syntax /
  promote behavior to contract / open follow-up.
```

The matrix should be explicit enough that a reviewer can answer which SQL
claims are supported, rejected, or still unproven without re-running the whole
audit mentally.

## Execution Discipline

Each 0.166 work item should start from a matrix row.

A change should not add, remove, or widen SQL behavior unless the matrix row
already classifies the behavior and names the intended resolution.

Default resolutions:

- documented and untested -> add public-surface tests
- documented and unsupported -> fix behavior or correct docs
- undocumented and accepted -> reject fail-closed unless explicitly promoted
- unsupported and accepted -> reject fail-closed
- unsupported and rejected but untested -> add fail-closed tests

Feature promotion requires an explicit product decision, an updated normative
contract entry, direct public-surface tests, and nearby fail-closed tests.

The matrix should be updated in the same change as any test, documentation, or
behavior change that resolves a row.

## Query Surface

The query audit covers documented read, explain, and introspection behavior
through `execute_sql_query::<E>(...)`.

Candidate families to classify:

- scalar `SELECT`
- scalar `DISTINCT`
- field projection
- wildcard projection
- computed projection
- projection aliases
- entity names and single-table aliases
- `WHERE`
- comparison operators
- boolean operators
- postfix predicates such as `IS NULL`, `IN`, `BETWEEN`, `LIKE`, and `ILIKE`
- bounded text and numeric function forms admitted by the contract
- searched `CASE`
- `ORDER BY`
- `LIMIT` and `OFFSET`
- global aggregates
- grouped aggregates
- grouped/global `HAVING`
- `EXPLAIN`
- `DESCRIBE`
- `SHOW INDEXES`
- `SHOW COLUMNS`
- `SHOW ENTITIES`
- `SHOW TABLES`
- public query response shape
- empty result behavior
- unknown entity errors
- unknown field errors
- type mismatch errors

The audit should not assume all candidate shapes are supported merely because
they are common SQL. Each item is classified from the contract and live public
behavior.

Proof requirements:

- Tests call `execute_sql_query::<E>(...)`.
- Tests verify the public response shape, not only success or row count.
- Tests distinguish empty valid results from failed queries.
- Tests verify ordering only when `ORDER BY` is part of the claim.
- Unsupported query syntax fails before partial execution.

## Update Surface

The mutation audit covers documented state-changing SQL through
`execute_sql_update::<E>(...)`.

Candidate families to classify:

- `INSERT`
- `UPDATE`
- `DELETE`
- admitted narrow `RETURNING`
- field assignment
- ID behavior
- missing required fields
- unknown fields
- type validation
- relation or reference fields, where SQL exposes them
- constraint failures
- affected-row count or equivalent public response field
- mutation response shape

Proof requirements:

- Tests call `execute_sql_update::<E>(...)`.
- Tests verify persisted state after mutation.
- Tests verify the public mutation response.
- Tests validate failure behavior for invalid inputs.
- Rejected mutations do not partially mutate state.

## DDL Surface

The schema audit covers accepted-catalog SQL DDL through
`execute_sql_ddl::<E>(...)`.

Candidate families to classify:

- `CREATE INDEX`
- `CREATE UNIQUE INDEX`
- `CREATE INDEX IF NOT EXISTS`
- `DROP INDEX`
- `DROP INDEX IF EXISTS`
- `ALTER TABLE ... ADD COLUMN`
- `ALTER TABLE ... ALTER COLUMN ... SET DEFAULT`
- `ALTER TABLE ... ALTER COLUMN ... DROP DEFAULT`
- `ALTER TABLE ... ALTER COLUMN ... SET NOT NULL`
- `ALTER TABLE ... ALTER COLUMN ... DROP NOT NULL`
- `ALTER TABLE ... RENAME COLUMN`
- `ALTER TABLE ... DROP COLUMN`
- `ALTER TABLE ... DROP COLUMN IF EXISTS`
- schema publication results
- duplicate names and duplicate index contracts
- generated/model-owned field and index rejection
- invalid field definitions
- unsupported field types
- unknown model or entity references
- DDL response shape

Proof requirements:

- Tests call `execute_sql_ddl::<E>(...)`.
- Tests verify accepted schema state after DDL.
- Tests verify invalid DDL fails closed.
- Failed DDL does not publish partial schema state.
- Duplicate, conflict, and generated-ownership behavior is explicit.

## Fail-Closed SQL

Fail-closed coverage is a first-class 0.166 goal.

`docs/contracts/SQL_SUBSET.md` says anything not stated there is outside the
supported SQL surface and must fail closed. A small SQL subset is acceptable.
Ambiguous acceptance is not.

Major unsupported families to classify include:

- multi-entity statements
- joins
- subqueries
- common table expressions
- `UNION`, `INTERSECT`, and `EXCEPT`
- window functions
- recursive queries
- transactions
- arbitrary functions outside the admitted bounded function set
- casts, unless explicitly admitted elsewhere
- simple `CASE value WHEN ...`
- non-prefix text pattern matching where excluded
- grouped predicate forms excluded by `SQL_SUBSET.md`
- unsupported grouped projection forms
- computed or aggregate `RETURNING` projections
- cursor pagination on scalar SQL
- byte-metric diagnostics through SQL
- DDL forms outside the accepted-catalog subset
- mutation forms outside the public update subset
- vendor-specific SQL extensions

Fail-closed tests should prove:

- Unsupported SQL is rejected through the public entrypoint.
- The rejection category is appropriate.
- Mutating or DDL statements leave state unchanged after rejection.
- Rejection does not depend on matching fragile prose unless exact error text is
  already part of the public contract.

## Documentation Reconciliation

0.166 reconciles the normative contract with supporting documents and live
behavior.

The audit should identify:

- features claimed as supported but not tested
- features claimed as supported but not implemented
- features listed as TODO but already implemented
- features excluded in `SQL_SUBSET.md` but implied elsewhere
- features implemented but undocumented
- features documented ambiguously
- public entrypoint routing claims that do not match live behavior

Acceptable resolutions:

- add missing public-surface tests
- tighten `SQL_SUBSET.md` wording
- adjust `SQL_SURFACE_MAPPING.md` when implementation-facing mapping is stale
- move a feature from TODO to supported only when directly proven
- move a feature from supported to TODO when not implemented
- add explicit unsupported notes
- reject accidentally accepted SQL
- open named follow-up work for deliberate feature implementation

Resolution should prefer contract clarity over feature growth.

## Public-Surface Test Bar

A test counts as public-surface proof only when it uses the relevant public SQL
entrypoint:

- `execute_sql_query::<E>(...)` for read, explain, and introspection behavior
- `execute_sql_update::<E>(...)` for mutation behavior
- `execute_sql_ddl::<E>(...)` for accepted-catalog schema DDL behavior

Parser-only tests do not prove public SQL support.
Planner-only tests do not prove public SQL support.
Executor-only tests do not prove public SQL support.
Storage tests do not prove public SQL support.

Those tests may still be useful supporting evidence, but the matrix should not
mark a contract item as proven unless the public entrypoint is exercised
directly.

## Error Contract Expectations

0.166 should prefer stable error categories over exact string matching unless
exact strings are already treated as public API.

Useful checks include:

- unsupported feature classification
- invalid syntax classification
- unknown entity or model classification
- unknown field classification
- type mismatch classification
- constraint violation classification
- wrong-entrypoint classification
- state unchanged after rejection

The goal is predictable, intentional failure. Perfect diagnostics are secondary
unless an unclear diagnostic hides a contract ambiguity.

## Response Shape Expectations

Tests should verify public SQL response structures, not just success.

For query responses, verify relevant pieces such as:

- row count
- projected field names
- projected field values
- ordering when claimed
- aggregate result fields when claimed
- grouped result fields when claimed
- `next_cursor` only where grouped SQL exposes it
- empty-result representation

For update responses, verify relevant pieces such as:

- affected count or count payload
- returned rows for admitted `RETURNING`
- public mutation status
- persisted state after mutation

For DDL responses, verify relevant pieces such as:

- schema publication status
- no-op status for admitted `IF EXISTS` / `IF NOT EXISTS` shapes
- created, changed, or dropped object identity where exposed
- accepted schema state after DDL

## Accidental Acceptance Policy

If unsupported SQL currently parses, lowers, or executes, 0.166 should classify
it as accidental unless the docs intentionally claim it.

Default resolution is fail-closed rejection.

Promotion into the supported contract requires:

- explicit product decision
- updated `SQL_SUBSET.md`
- direct public-surface tests
- fail-closed tests for nearby unsupported variants
- response-shape proof

Do not let parser breadth silently define the product contract.

## Performance Policy

SQL performance harnesses should not be the first workstream.

Performance results are meaningful only once the contract being measured is
stable. 0.166 should run SQL performance harnesses after:

- the matrix is populated
- public-surface correctness gaps are patched or recorded
- unsupported SQL behavior fails closed
- documentation contradictions are resolved or named as follow-ups

Performance findings should remain separate from contract proof findings.

## Validation

Validation for 0.166 should include:

- formatting
- focused public SQL query tests
- focused public SQL update tests
- focused public SQL DDL tests
- fail-closed unsupported SQL tests
- relevant compile-fail tests if schema or macro behavior is touched
- clippy for touched crates when code changes are introduced
- doc consistency review
- accidental-acceptance scans where practical

Individual audit slices can use focused validation when changes are contained,
but closeout should make clear what remains unproven.

## Closeout Criteria

0.166 is complete because the project can answer these questions directly from
the matrix and tests:

- Which SQL features are supported?
- Which SQL features are intentionally unsupported?
- Which supported features are proven through public entrypoints?
- Which documented claims were corrected?
- Which live behaviors were rejected as accidental acceptance?
- Which missing features remain product decisions rather than hidden bugs?

Concrete closeout requirements:

- Every supported SQL claim has direct public-surface coverage or an explicit
  documented gap.
- Every explicitly unsupported major SQL family has fail-closed coverage.
- Every contradiction among `SQL_SUBSET.md`, `SQL_SURFACE_MAPPING.md`,
  `1.0-FEATURES.md`, `1.0-TODO.md`, and live behavior is resolved or recorded.
- No major unsupported SQL family is accidentally accepted without
  classification.
- Public response shapes are tested for the main query, update, and DDL paths.
- SQL performance harnesses run only after correctness and contract proof are
  stable enough for their results to be meaningful.

## Expected Outcome

The expected outcome is not a larger SQL engine.

The expected outcome is a more trustworthy SQL engine.

After 0.166, IcyDB should have:

- a clear SQL contract
- a visible proof matrix
- direct public-surface tests for claimed behavior
- fail-closed tests for excluded SQL
- fewer accidental parser or executor acceptances
- cleaner 1.0 feature and TODO alignment
- a better basis for production confidence

0.166 should make it possible to say, with evidence:

```text
This is the SQL surface IcyDB supports, these are the public tests that prove
it, and this is the SQL IcyDB intentionally rejects.
```
