# IcyDB 0.180 - Compact Diagnostic Codes

## Status

Draft design for 0.180.

0.180 should move IcyDB diagnostics from text-first public errors toward
compact machine-readable diagnostic codes with optional host-side text
rendering. It is intentionally scoped as a breaking pre-`1.0.0` hard cut.

## Purpose

Reduce live canister Wasm footprint and stabilize machine-readable failures by
moving production diagnostics from text-first errors to compact codes and
structured details.

The CLI and local/debug tooling should own rich human text. Production canister
Wasm should return compact diagnostic identity and should not construct long
diagnostic prose for known errors.

## Background

The 2026-06-06 wasm-footprint audit showed that `one_sql_query` is the largest
current audit fixture:

- `one_sql_query` shrunk Wasm: `2,955,294` bytes;
- `one_fluent_query` shrunk Wasm: `2,389,137` bytes;
- SQL-over-fluent gap: `566,157` bytes;
- SQL data segment: `241,200` bytes;
- fluent data segment: `178,624` bytes;
- SQL-over-fluent data segment gap: `62,576` bytes.

String extraction suggests that compact diagnostics will not erase the whole
SQL cost. The likely recoverable area is tens of KB, not hundreds. The larger
SQL gap also includes parser, planner, result, Candid, and execution code.

Still, diagnostic prose is one of the most tractable live-Wasm data costs. It
also makes the public error contract less precise than it should be.

## Problem

The public facade currently returns an error payload with a text message:

```text
Error {
  kind,
  origin,
  message: String,
}
```

Core errors also store rendered messages and many call sites construct
human-readable strings before the facade converts them for canister callers.

Moving text rendering only in the CLI is not enough if core still links and
constructs long static messages. To reduce live Wasm, the production path must
be code/detail-first before message construction.

## Goals

- Keep machine-readable error identity stable and explicit.
- Remove long human diagnostic prose from production canister error paths.
- Let the CLI render friendly messages, hints, and examples from shared codes.
- Allow local/debug builds to opt back into rich text without making it part of
  the live canister default.
- Avoid string matching in tests by asserting codes and structured details.
- Preserve enough structured detail for automation and support diagnostics.

## Non-Goals

- Do not preserve compatibility with the current text-first error wire before
  `1.0.0`.
- Do not move SQL parser/planner code out of canisters in this design.
- Do not promise a large Wasm reduction from diagnostics alone.
- Do not make generated models or the CLI authoritative for runtime errors.

## Proposed Crates

Use two separable layers.

### `icydb-diagnostic-code`

Small production-safe crate.

Responsibilities:

- stable diagnostic code enums or compact numeric ids;
- error class and origin taxonomy;
- small structured detail payloads;
- Candid/serde support only where needed for public wire contracts;
- no long prose, hints, examples, or formatted paragraphs.

This crate may be depended on by `icydb-core`, `icydb`, generated canisters,
integration fixtures, and CLI code.

### `icydb-diagnostic-text`

Optional renderer crate, or CLI-owned module if a crate proves unnecessary.

Responsibilities:

- map diagnostic codes and structured details to human messages;
- add hints, examples, and remediation text;
- render stable CLI output;
- support docs/tests that need rich text.

This layer must not be part of production canister defaults. It may be enabled
by CLI, host tooling, local development, and explicit debug features.

## Public Wire Shape

Target live canister error payload:

```text
IcydbError {
  code: IcydbErrorCode,
  class: ErrorClass,
  origin: ErrorOrigin,
  detail: Option<IcydbErrorDetail>,
}
```

Possible code shape:

```text
E_QUERY_VALIDATE
E_QUERY_PLAN
E_QUERY_UNORDERED_PAGINATION
E_QUERY_UNSUPPORTED_SQL_FEATURE
E_SCHEMA_DDL_MISSING_EXPECTED_VERSION
E_SCHEMA_DDL_PUBLICATION_RACE_LOST
E_STORE_NOT_FOUND
E_STORE_CORRUPTION
E_RUNTIME_INTERNAL
```

Codes should be stable enough for CLI and external callers. The string spelling
is for documentation and CLI display; the wire may use enum discriminants or
compact variant labels if that proves smaller.

## Structured Details

Details should carry data that callers may act on without parsing text:

```text
UnsupportedSqlFeature { feature: SqlFeatureCode }
SchemaDdlAdmission { reason: SchemaDdlAdmissionCode }
StoreKey { key_hash_or_short_key }
Field { entity, field }
Index { entity, index }
VersionContract { expected, accepted, next }
```

Details should avoid embedding full prose. For live Wasm, details should also
avoid large arbitrary strings unless the data is user input or entity metadata
needed for the caller to recover.

## Text Rendering

CLI/local rendering should be a pure mapping from compact diagnostic payloads
to text:

```text
E_SCHEMA_DDL_PUBLICATION_RACE_LOST: schema publication lost a race
hint: refresh the accepted schema and retry the DDL statement
```

Rules:

- rich text lives in `icydb-diagnostic-text` or CLI-owned renderer code;
- renderer can use long prose, hints, and examples;
- renderer must not be linked by production canister defaults;
- tests for user-facing CLI output may assert text;
- core/canister tests should assert codes/details.

## Feature Gates

Suggested features:

- `diagnostic-text`: enables local/debug human text rendering in `icydb`;
- `sql`: continues to gate SQL support;
- production/default canister builds keep `diagnostic-text` off;
- CLI enables text rendering because it is a host binary, not canister Wasm.

In production canister builds:

- `Error::message()` should be removed, unavailable, or return only a tiny
  fallback code label;
- known internal constructors should not allocate/render human prose;
- `thiserror` display strings should not be relied on for live user output.

## Internal Error Model

Core should move from text-first:

```text
InternalError { class, origin, message: String, detail }
```

to diagnostic-first:

```text
InternalError {
  code,
  class,
  origin,
  detail,
  debug_context,
}
```

`debug_context` must be absent or tiny in production builds. If a dynamic
string is necessary, it should represent user data or compact context, not a
full rendered sentence.

Constructor helpers should choose codes/details. They should not format
sentences unless `diagnostic-text` is enabled.

## SQL Feature Codes

SQL unsupported feature diagnostics are an early high-value target because
parser/session diagnostics currently carry long labels.

Move from:

```text
UnsupportedSqlFeature { feature: &'static str }
```

to:

```text
UnsupportedSqlFeature { feature: SqlFeatureCode }
```

Examples:

```text
OrderByUnsupportedForm
DropStatementBeyondDropIndex
AlterTableUnsupportedOperation
ParameterizedSchemaVersion
MultiStatementSql
WindowFunctions
Join
Union
```

The CLI renderer can map those to actionable messages.

## Result Surface Interaction

Compact errors are necessary but not sufficient. The SQL audit also shows that
wide public result types increase Candid/type surface.

Follow-up work should consider narrow endpoint result types:

- count-only SQL endpoints can return a smaller count result;
- generic SQL endpoints can keep the full SQL result enum;
- debug/admin endpoints can opt into describe/show/explain/DDL result shapes;
- generated endpoint defaults should stay minimal and explicit.

This is separate from diagnostic-code work but should be measured together.

## Implementation Slices

### Slice 1 - Design and Proof Inventory

- Add this design note.
- Add a source guard or audit report listing production `message()` and
  `to_string()` based error conversions.
- Record current wasm string/data baselines.

### Slice 2 - Shared Diagnostic Code Crate

- Add the compact code/detail crate.
- Move public error class/origin/kind enums to that crate or re-export them
  from there.
- Add Candid/serde derives only where wire consumers require them.
- Do not change behavior yet.

### Slice 3 - Public Facade Wire Hard Cut

- Replace public `Error { message: String }` with compact diagnostic payload.
- Add conversion from core diagnostic errors to public compact errors.
- Update canister, integration, and CLI decoding tests to assert codes/details.
- Mark this as breaking in changelog/release notes.

### Slice 4 - Core Constructor Migration

- Move high-frequency core constructors to code/detail-first construction.
- Replace stringly SQL unsupported-feature details with `SqlFeatureCode`.
- Keep dynamic context compact.
- Remove or feature-gate long `thiserror` strings from production paths.

### Slice 5 - CLI Text Renderer

- Add host-side rendering for compact payloads.
- Preserve or improve CLI output quality.
- Add CLI tests for rendered messages and hints.
- Keep canister tests code/detail based.

### Slice 6 - Wasm Measurement and Guards

- Rerun wasm-footprint audit.
- Add or update a guard that fails if live canister defaults link the text
  renderer.
- Add string/data budget tracking for `one_sql_query` and fluent comparison
  fixtures.

## Testing Strategy

Production/core tests:

- assert diagnostic codes and structured details;
- avoid matching human message strings;
- prove SQL parser unsupported-feature cases emit stable feature codes;
- prove DDL admission keeps stable schema DDL codes;
- prove store/runtime classes and origins survive conversion.

CLI tests:

- assert rendered text for representative codes;
- assert hints for common recovery cases;
- assert unknown/future code fallback rendering.

Wasm tests/audits:

- compare `one_sql_query` against `one_fluent_query`;
- track data segment bytes and printable string bytes;
- verify `diagnostic-text` is not linked into default canister builds.

## Migration Notes

This is a breaking pre-`1.0.0` hard cut.

Downstream callers that currently inspect `Error.message` must switch to
`Error.code`, `Error.class`, `Error.origin`, and `Error.detail`.

CLI and local tooling may continue to display rich text. Live canister callers
should treat the error payload as structured data.

## Open Questions

- Should the wire code be a Candid enum, `nat32`, or short text?
- Which details can safely carry entity/index/field names without giving back
  too much data-segment savings?
- Should `diagnostic-text` be a separate crate or remain inside the CLI crate
  until another host consumer needs it?
- Should local canister builds default `diagnostic-text` on, or should it
  always require explicit opt-in?
- Which current tests genuinely need human text, and which can move to
  code/detail assertions?

## 0.180 Closeout Standard

This work can close when:

- production canister public errors no longer carry long message strings;
- CLI renders rich messages from compact codes/details;
- SQL unsupported-feature diagnostics use compact feature codes;
- DDL admission diagnostics still expose stable machine-readable reasons;
- default canister builds do not link the text renderer;
- wasm-footprint audit records the raw `.wasm` and data-segment impact.
