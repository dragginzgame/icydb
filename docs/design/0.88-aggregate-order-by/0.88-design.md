# 0.88 Grouped Aggregate ORDER BY with Bounded Top-K

## Thesis

`0.88` should introduce **bounded execution support for grouped `ORDER BY` over aggregate expressions**, enabling queries like:

```sql
SELECT class_name, AVG(strength)
FROM character
GROUP BY class_name
ORDER BY AVG(strength) DESC
LIMIT 10;
```

without violating IcyDB’s core guarantees:

* bounded execution
* stable semantics
* no implicit full materialization or unbounded sort

The core mechanism is:

> evaluate aggregate-based order expressions using a **bounded top-k selection strategy** instead of canonical grouped-key ordering.

---

## Problem statement

After `0.85–0.87`, the system can:

* compute aggregate expressions
* filter with them (`HAVING`)
* reject them in `ORDER BY`

The current limitation:

```sql
ORDER BY AVG(strength)
```

is rejected—not because it’s not computable, but because:

* it breaks canonical grouped-key order
* it requires sorting grouped results
* sorting is currently unbounded / undefined

This creates a usability gap:

* users expect aggregate ranking queries
* the engine already computes all required values
* only ordering is missing

---

## Architectural position

### Introduce a second grouped ordering mode

Up to now, grouped ordering has been:

```text
Mode A: Canonical grouped-key order (streaming, continuation-safe)
```

`0.88` introduces:

```text
Mode B: Bounded top-k order (aggregate-based, heap-backed)
```

These are fundamentally different execution strategies.

---

### Do not overload canonical ordering

Aggregate ordering must NOT:

* reuse canonical grouped ordering
* pretend to preserve grouped-key order
* reuse continuation semantics

It is a distinct execution mode.

---

## Scope

### In scope

* `ORDER BY` expressions that depend on:

  * aggregate outputs
  * post-aggregate expressions
* only when paired with:

  * `LIMIT`
* bounded top-k execution strategy
* support for:

  * ASC / DESC
  * multi-column ordering (if feasible)

---

### Out of scope

* aggregate ordering without `LIMIT`
* cursor continuation for aggregate ordering
* full sort of grouped output
* streaming execution for aggregate ordering
* mixing canonical and aggregate ordering modes

---

## Execution model

### Canonical grouped ordering (existing)

```text
scan → group → emit in grouped-key order → LIMIT applied
```

### Aggregate ordering (new)

```text
scan
→ group (build full aggregate state)
→ finalize grouped rows
→ maintain top-k heap
→ emit sorted results
```

---

## Core algorithm

### Top-K heap selection

For:

```sql
ORDER BY expr LIMIT K
```

Execution:

1. compute grouped rows (key + aggregate outputs)
2. for each row:

   * evaluate `expr`
   * insert into heap of size K
   * evict worst element if needed
3. after all rows:

   * extract heap
   * sort (small K)
   * emit

---

### Complexity

* Time: `O(N log K)`
* Memory: `O(K)`

Where:

* N = number of groups
* K = LIMIT

This preserves boundedness.

---

## Admission rules

### Required

* `LIMIT` must be present
* `ORDER BY` expression may depend on:

  * aggregate outputs
  * grouped keys
  * scalar functions over those

---

### Rejected

```sql
ORDER BY AVG(strength)
```

(without LIMIT)

Reason:

* requires full sort
* violates boundedness

---

### Allowed

```sql
ORDER BY AVG(strength) DESC LIMIT 10
```

```sql
ORDER BY MAX(level) - MIN(level) LIMIT 5
```

```sql
ORDER BY ROUND(AVG(strength), 2) LIMIT 20
```

---

## Expression support

This slice **reuses 0.85/0.86 expression model**:

Allowed leaves:

* grouped keys
* aggregate outputs

Allowed composition:

* arithmetic
* scalar functions
* nested expressions

---

## Lowering changes

### Detect aggregate-based ordering

During lowering:

* inspect `ORDER BY` expression
* if it contains aggregate references:

```text
→ mark plan as AggregateOrderTopK
```

Else:

```text
→ use canonical grouped ordering
```

---

### Plan representation

Introduce:

```rust
enum GroupedOrderMode {
    Canonical,
    TopK {
        order_expr: CompiledExpr,
        limit: usize,
    },
}
```

---

## Executor changes

### New execution path

Inside grouped executor:

```text
if order_mode == Canonical:
    stream results
else:
    run Top-K pipeline
```

---

### Heap representation

Each entry:

```rust
struct HeapEntry {
    order_value: Value,
    row: GroupedOutputRow,
}
```

Heap comparator:

* respects ASC / DESC
* handles nulls consistently with SQL semantics

---

## Multi-column ORDER BY (optional v1)

Support if simple:

```sql
ORDER BY AVG(x), MAX(y) LIMIT 10
```

Heap key becomes tuple.

If complex:

→ defer to later slice

---

## Null handling

Define explicitly:

* NULLS LAST (default) or consistent with existing engine rules
* must match scalar ORDER BY semantics

---

## Continuation semantics

### Important constraint

Aggregate ORDER BY:

* **does NOT support continuation/cursors**

Reason:

* heap-based selection depends on full scan
* ordering is not prefix-stable

---

### Behavior

* cursor token must be rejected
* or ignored with explicit error

---

## Interaction with HAVING

Execution order:

```text
group → finalize → HAVING → Top-K selection → emit
```

HAVING filters before heap insertion.

---

## Interaction with projection

Projection evaluated:

* after aggregate finalization
* before or during heap insertion (implementation choice)

---

## Parser

No changes needed.

Already supports expressions.

---

## Validation

Add rule:

```text
if ORDER BY uses aggregate expressions:
    require LIMIT
```

Error message:

> "aggregate-based ORDER BY requires LIMIT for bounded execution"

## Examples

### Valid

```sql
SELECT class_name, AVG(strength)
FROM character
GROUP BY class_name
ORDER BY AVG(strength) DESC
LIMIT 10;
```

```sql
SELECT class_name, MAX(level), MIN(level)

GROUP BY class_name

```

---

### Invalid

```sql
ORDER BY AVG(strength)
```

(no LIMIT)

```sql
ORDER BY AVG(strength) LIMIT 10 OFFSET 5
```

(if OFFSET unsupported for this mode)

---

## Guardrails

### 1. No implicit full sort

Never fallback to:

* materialize all rows
* sort fully

---

### 2. No continuation

Top-K mode is not cursor-compatible.

---

### 3. Strict LIMIT requirement

Always enforce.

---

### 4. Separate execution paths

Do not mix canonical and aggregate ordering logic.

---

## Acceptance criteria

### Functional

* aggregate ORDER BY works with LIMIT
* supports scalar expressions over aggregates
* rejects missing LIMIT

---

### Architectural

* canonical ordering unchanged
* Top-K path isolated and explicit

---

### Performance

* O(N log K) behavior confirmed
* no full sort path introduced
