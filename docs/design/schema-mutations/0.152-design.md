# 0.152 Catalog-Native Schema Mutations

## Purpose

0.149 through 0.151 moved runtime schema, row, field-path index, and
expression-index authority onto accepted persisted schema snapshots. 0.152
starts the next stage: schema evolution as explicit catalog mutation semantics.

The key shift is from generated proposal comparison as the main vocabulary to
accepted snapshot mutation reasoning. Generated `EntityModel` and `IndexModel`
remain valid authoring/proposal inputs, but once a schema is accepted the
catalog snapshot is the authority that runtime mutation rules must rewrite,
validate, and publish.

## Hard Invariants

- Accepted schema snapshots are the catalog authority.
- SQL DDL is a frontend onto mutation operations, not the source of mutation
  semantics.
- Unsupported mutations fail closed before snapshot publication.
- Rebuild requirements are explicit even before rebuild orchestration exists.
- Missing accepted metadata is an incompatible accepted-schema error, not a
  reason to fall back to generated metadata.
- Before `1.0.0`, internal mutation formats hard-cut to the latest contract.

## Internal Shape

The first internal contract is deliberately small:

```rust
enum SchemaMutation {
    AddNullableField { ... },
    AddDefaultedField { ... },
    AddNonUniqueIndex { ... },
    AddExpressionIndex { ... },
    DropNonRequiredSecondaryIndex { ... },
    AlterNullability { ... },
}

enum MutationCompatibility {
    MetadataOnlySafe,
    RequiresRebuild,
    UnsupportedPreOne,
    Incompatible,
}

enum RebuildRequirement {
    NoRebuildRequired,
    IndexRebuildRequired,
    FullDataRewriteRequired,
    Unsupported,
}
```

`MutationPlan` is the schema-owned artifact that records ordered mutation
operations, the compatibility bucket, the rebuild requirement, and a
deterministic audit fingerprint.

`SchemaMutationDelta` is the schema-owned classifier for comparing one accepted
snapshot with a proposed replacement. Transition policy consumes this
classification, then decides whether the delta is currently publishable.

## Compatibility Matrix

| Operation | Initial 0.152 classification | Runtime visibility |
| --- | --- | --- |
| Exact accepted snapshot match | metadata-only safe | immediately visible |
| Add nullable field at appended slot | metadata-only safe | immediately visible after snapshot publish |
| Add default-backed field at appended slot | metadata-only safe | immediately visible after default payload validation |
| Add non-unique field-path index | requires index rebuild | not visible until rebuild support lands |
| Add deterministic expression index | requires index rebuild | not visible until rebuild support lands |
| Drop non-required secondary index | requires index/store cleanup | not visible until cleanup support lands |
| Tighten nullability | unsupported pre-1.0 | rejected |
| Widen nullability | unsupported pre-1.0 until policy exists | rejected |
| Change field type/storage codec | incompatible or full rewrite | rejected |
| Slot remap or arbitrary reorder | incompatible or full rewrite | rejected |
| Primary-key rewrite | unsupported pre-1.0 | rejected |
| Foreign-key/relationship constraint mutation | unsupported pre-1.0 | rejected |
| Arbitrary SQL expression index | unsupported pre-1.0 | rejected |

## Rebuild Semantics

`NoRebuildRequired` means old physical rows remain valid under the new accepted
snapshot. Today that is limited to exact match and append-only nullable/defaulted
field additions whose missing-slot absence policy is accepted by schema-owned
decode rules.

`IndexRebuildRequired` means the schema mutation can be described, but runtime
visibility must wait for an index rebuild phase that fills forward/reverse index
stores, proves uniqueness where relevant, invalidates planner/cache state, and
publishes the new index contract atomically.

`FullDataRewriteRequired` is reserved for changes that require rewriting stored
rows before the new schema can be visible.

`Unsupported` is a deliberate fail-closed result for mutations that need a
larger migration planner or product decision.

## 0.152 Scope

0.152 should keep implementation narrow:

- name and centralize the mutation contract types
- route existing append-only field reconciliation through `MutationPlan`
- preserve current runtime behavior
- document rebuild requirements for index mutations without making them visible
- add focused tests for deterministic mutation fingerprints and rebuild buckets

The slice should not add broad SQL DDL, online rebuild orchestration, generated
fallback reconstruction, slot remapping, primary-key rewrites, or arbitrary type
changes.

## Next Slices

1. Add mutation diagnostics to reconciliation errors and schema metrics.
2. Add an internal mutation request API that lowers proposal diffs into
   `MutationPlan`.
3. Add index rebuild planning for non-unique accepted field-path indexes.
4. Add deterministic expression-index rebuild planning for the existing accepted
   expression subset.
5. Add store cleanup planning for dropping non-required secondary indexes.
6. Add SQL DDL as a thin parser/lowering frontend after the mutation engine owns
   compatibility and rebuild semantics.
