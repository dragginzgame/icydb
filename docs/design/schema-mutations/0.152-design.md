# 0.152 Catalog-Native Schema Mutations

## Purpose

0.149 through 0.151 moved runtime schema, row, field-path index, and
expression-index authority onto accepted persisted schema snapshots. 0.152
starts the next stage: schema evolution as explicit catalog mutation semantics.

The key shift is from generated proposal comparison as the main vocabulary to
accepted snapshot mutation reasoning. Generated `EntityModel` and `IndexModel`
remain valid authoring/proposal inputs, but once a schema is accepted the
catalog snapshot is the authority that runtime mutation rules must rewrite,
validate, and publish.

## Hard Invariants

- Accepted schema snapshots are the catalog authority.
- SQL DDL is a frontend onto mutation operations, not the source of mutation
  semantics.
- Unsupported mutations fail closed before snapshot publication.
- Rebuild requirements are explicit even before rebuild orchestration exists.
- Missing accepted metadata is an incompatible accepted-schema error, not a
  reason to fall back to generated metadata.
- Before `1.0.0`, internal mutation formats hard-cut to the latest contract.

## Internal Shape

The first internal contract is deliberately small:

```rust
enum SchemaMutation {
    AddNullableField { ... },
    AddDefaultedField { ... },
    AddNonUniqueIndex { ... },
    AddExpressionIndex { ... },
    DropNonRequiredSecondaryIndex { ... },
    AlterNullability { ... },
}

enum MutationCompatibility {
    MetadataOnlySafe,
    RequiresRebuild,
    UnsupportedPreOne,
    Incompatible,
}

enum RebuildRequirement {
    NoRebuildRequired,
    IndexRebuildRequired,
    FullDataRewriteRequired,
    Unsupported,
}
```

`MutationPlan` is the schema-owned artifact that records ordered mutation
operations, the compatibility bucket, the rebuild requirement, and a
deterministic audit fingerprint.

`SchemaRebuildPlan` is the schema-owned physical-work classification derived
from `MutationPlan`. It currently describes accepted field-path index rebuild
targets, accepted expression-index rebuild targets, accepted secondary-index
drop cleanup targets, full row rewrites, and unsupported mutation actions.
These are planning facts only; publication still fails closed until a later
slice owns physical rebuild/cleanup execution and validation.

`SchemaMutationDelta` is the schema-owned classifier for comparing one accepted
snapshot with a proposed replacement. Transition policy consumes this
classification, then decides whether the delta is currently publishable.

`SchemaMutationRequest` is the internal request-lowering surface. Generated
proposal diffs, accepted index snapshots, and later SQL DDL frontends lower into
this request vocabulary before producing a `MutationPlan`. This keeps
parser/proposal code from constructing mutation plans directly and keeps
compatibility/rebuild semantics inside the schema mutation layer.

`SchemaFieldPathIndexRebuildTarget` is the first accepted index rebuild target.
It is derived from `PersistedIndexSnapshot` rather than generated `IndexModel`
metadata and carries accepted index store identity, predicate display metadata,
and accepted field-path key slots. Unique indexes and unsupported key shapes
remain fail-closed until their validators and rebuild semantics exist.

`SchemaExpressionIndexRebuildTarget` extends the same snapshot-owned model to
deterministic expression indexes. It preserves accepted mixed key-item order,
accepted source slots, expression op, input/output kinds, and canonical
expression text. Unique expression indexes, field-path-only keys, empty keys,
and unsupported key shapes stay fail-closed until physical rebuild validation
can prove them.

`SchemaSecondaryIndexDropCleanupTarget` is the accepted cleanup target for
dropping secondary indexes. It carries accepted index ordinal, store identity,
uniqueness, and predicate display metadata. The initial lowering admits only
non-unique secondary indexes because unique indexes remove constraints and need
explicit constraint-drop policy before publication.

`SchemaMutationExecutionPlan` is the execution-facing contract derived from a
`MutationPlan`. It names whether a plan is publishable immediately, requires a
future physical runner, or remains unsupported, and it orders physical steps
such as index rebuilds, index cleanup, validation, and runtime invalidation.
Its execution gate reduces that plan to ready-to-publish, awaiting-physical-work,
or rejected status for future callers. This contract does not widen
publication: startup reconciliation still asks the publication gate and rejects
every rebuild-required plan until execution exists.

## Compatibility Matrix

| Operation | Initial 0.152 classification | Runtime visibility |
| --- | --- | --- |
| Exact accepted snapshot match | metadata-only safe | immediately visible |
| Add nullable field at appended slot | metadata-only safe | immediately visible after snapshot publish |
| Add default-backed field at appended slot | metadata-only safe | immediately visible after default payload validation |
| Add non-unique field-path index | requires index rebuild | not visible until rebuild support lands |
| Add deterministic expression index | requires index rebuild | not visible until rebuild support lands |
| Drop non-required secondary index | requires index/store cleanup | not visible until cleanup support lands |
| Tighten nullability | unsupported pre-1.0 | rejected |
| Widen nullability | unsupported pre-1.0 until policy exists | rejected |
| Change field type/storage codec | incompatible or full rewrite | rejected |
| Slot remap or arbitrary reorder | incompatible or full rewrite | rejected |
| Primary-key rewrite | unsupported pre-1.0 | rejected |
| Foreign-key/relationship constraint mutation | unsupported pre-1.0 | rejected |
| Arbitrary SQL expression index | unsupported pre-1.0 | rejected |

## Rebuild Semantics

`NoRebuildRequired` means old physical rows remain valid under the new accepted
snapshot. Today that is limited to exact match and append-only nullable/defaulted
field additions whose missing-slot absence policy is accepted by schema-owned
decode rules.

`IndexRebuildRequired` means the schema mutation can be described, but runtime
visibility must wait for an index rebuild phase that fills forward/reverse index
stores, proves uniqueness where relevant, invalidates planner/cache state, and
publishes the new index contract atomically.

`FullDataRewriteRequired` is reserved for changes that require rewriting stored
rows before the new schema can be visible.

`Unsupported` is a deliberate fail-closed result for mutations that need a
larger migration planner or product decision.

## 0.152 Scope

0.152 should keep implementation narrow:

- name and centralize the mutation contract types
- route existing append-only field reconciliation through `MutationPlan`
- preserve current runtime behavior
- document rebuild requirements for index mutations without making them visible
- add focused tests for deterministic mutation fingerprints and rebuild buckets

The slice should not add broad SQL DDL, online rebuild orchestration, generated
fallback reconstruction, slot remapping, primary-key rewrites, or arbitrary type
changes.

## Next Slices

1. Add mutation diagnostics to reconciliation errors and schema metrics.
2. Add an explicit rebuild publication gate owned by `MutationPlan`. Done in
   `0.152.1`; future slices should extend the gate rather than reimplementing
   publishability in reconciliation.
3. Add schema-owned rebuild planning actions derived from `MutationPlan`. Done
   in `0.152.2`; future slices should feed these actions into a physical
   rebuild runner before publication.
4. Add an internal mutation request API that lowers proposal diffs into
   `MutationPlan`. Done in `0.152.2`; future frontends should lower to
   `SchemaMutationRequest` before producing plans.
5. Add index rebuild planning for non-unique accepted field-path indexes. Done
   in `0.152.3`; future slices should feed these accepted rebuild targets into
   a physical rebuild runner before publication.
6. Add deterministic expression-index rebuild planning for the existing accepted
   expression subset. Done in `0.152.4`; future slices should feed these
   accepted rebuild targets into a physical rebuild runner before publication.
7. Add store cleanup planning for dropping non-required secondary indexes. Done
   in `0.152.5`; future slices should feed these accepted cleanup targets into
   a physical cleanup runner before publication.
8. Add the schema-owned mutation execution boundary that future rebuild and
   cleanup runners must satisfy before publication. Done in `0.152.6`; future
   slices should implement physical execution behind this boundary without
   changing startup reconciliation policy.
9. Add SQL DDL as a thin parser/lowering frontend after the mutation engine owns
   compatibility and rebuild semantics.
