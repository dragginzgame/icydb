# 0.204 SQL Correctness Matrix And Generative Oracles

Status: approved in direction; review and performance-evidence revisions
incorporated; begins after 0.203 closeout; implementation not started

## Context

IcyDB already has substantial SQL correctness evidence:

- broad example-based parser, planner, executor, storage, and canister tests;
- a large deterministic performance and route matrix;
- deterministic SQLite comparison scenarios;
- seeded random SELECT comparison scenarios;
- seeded mutation comparison scenarios; and
- a required live-canister SQL lane in CI.

That evidence has found real defects and makes SQL one of the better-tested product
surfaces. It is not yet one coherent correctness system, however. The current
evidence is distributed across ordinary unit and integration tests, a performance
audit harness, optional SQLite/PocketIC comparisons, and historical design ledgers.
The boundaries between product coverage, route coverage, performance observation,
and differential correctness are therefore difficult to state and enforce.

The current comparison harness is also stronger than its default execution policy.
The deterministic matrix contains substantially more scenarios than its default
prefix-limited run executes. Random SELECT coverage is seeded, but it samples SQL
fragments and templates rather than constructing a typed syntax tree. Mutation
coverage is bounded to a small schema and short scenario set. The actual SQLite
comparisons are opt-in, can skip when `sqlite3` is unavailable, and are not required
by the ordinary GitHub workflow. The performance report can record scenario failures
without making those failures a correctness gate.

Finally, the archived SQL surface ledger is no longer an exact description of the
active contract. It predates current introspection statements and still records at
least one retired spelling as proven. That does not mean the active implementation
has regressed. It means the repository lacks a single current, machine-checked
statement connecting supported SQL features to their required evidence.

0.204 converts these useful but partially independent mechanisms into bounded,
reproducible correctness and performance evidence lines with shared typed scenarios
and separate verdicts.

## Dependency On 0.203

0.204 is sequenced after
[`0.203-design.md`](../0.203-legacy-architecture-hard-cut/0.203-design.md).

Implementation must not begin against surfaces that 0.203 is removing or renaming.
In particular, 0.204 assumes all of the following are already true:

- the retained internal SQL helper has its explicit trusted-entry-point name;
- the shadow public-SQL admission path and its duplicate test matrix are gone;
- planner and route diagnostics expose typed facts rather than requiring message or
  SQL-text reclassification;
- fictitious grouped-route state has been removed; and
- active SQL documentation describes only the maintained current surface.

This dependency is architectural, not merely chronological. A new generator or
evidence provider built on the old public admission helper, deleted response aliases,
string diagnostics, or shadow query shells would immediately recreate the ambiguity
0.203 is intended to remove.

0.204 may be designed and reviewed while 0.203 is in progress. Its implementation
starts only after 0.203 closes.

## Problem Statement

The current SQL evidence has nine structural gaps.

### 1. There Is No Current Coverage Authority

[`SQL_SUBSET.md`](../../contracts/SQL_SUBSET.md) is the normative user-facing
contract, but there is no machine-checked manifest saying which evidence is required
for each maintained feature. The historical proof ledger is descriptive and stale.

### 2. Correctness And Performance Are Coupled

The large performance matrix is also used as a source of correctness scenarios and
route assertions. A performance report has different failure and sampling semantics
from a correctness gate. Reusing scenarios is valuable; reusing the permissive
performance verdict is not.

The matrix can expose obvious instruction hotspots, but its default prefix, debug
profile, single sample, small fixtures, partial ranking/delta fields, and weak
baseline identity do not provide comprehensive hotspot evidence. Raw totals alone
cannot distinguish fixed overhead from per-row, per-entry, store, cache, or output
scaling. Performance therefore needs a stronger evidence contract, not merely a
larger scenario count.

### 3. Default Sampling Is Prefix-Biased

A bounded prefix of a large, intentionally ordered matrix does not preserve feature,
type, route, or provider diversity. Adding an earlier family can silently remove a
later family from the default run without changing the configured limit.

### 4. Random SELECT Generation Is Text-Template Sampling

The current seeded random SELECT generator is reproducible, but it combines known
SQL fragments rather than generating a typed expression tree. It therefore has
limited control over nesting, type validity, null propagation, feature interaction,
and minimization.

### 5. SQLite Is A Partial, Optional Oracle

SQLite is intentionally not a model for all IcyDB semantics. Some IcyDB values,
storage routes, metadata statements, generated-canister policies, and diagnostics do
not have a fair SQLite equivalent. Even within the valid overlap, the executable
comparison is optional today and may silently skip when its external binary is
missing.

### 6. Failure Policy Is Too Permissive For A Correctness Gate

The current audit can record failures for inspection, and its delta report can
classify a new failure without necessarily failing because a new failure exists.
That is appropriate for exploratory performance auditing, not for admitted SQL
correctness.

### 7. Generated Failures Are Not Minimized Or Promoted

A seed can reproduce a random failure, but the harness has no typed shrinker for the
query, fixture, or mutation sequence. There is no compact checked-in regression
corpus produced from minimized failures.

### 8. Mutation Coverage Is Narrow

Current differential mutation scenarios provide useful smoke coverage but do not
form a schema-aware state machine over the maintained INSERT, UPDATE, DELETE,
RETURNING, ordering, limiting, nullability, and atomic-failure surface.

### 9. Harness Structure Obscures Test Intent

One very large audit file owns scenario construction, execution, route
classification, SQLite adaptation, reporting, and performance concerns. Some route
classification still derives meaning from SQL text. This makes review difficult and
encourages additions that are easy to count but hard to classify.

## Design Goal

0.204 establishes one current and machine-checked SQL evidence model in which:

1. every stable SQL contract feature identifier names its required evidence
   obligations and minimum strengths;
2. deterministic selection is stratified rather than prefix-based;
3. generated SQL is constructed from a typed, schema-aware representation;
4. generated data deliberately exercises null, duplicate, boundary, and ordering
   behavior;
5. each scenario declares its required evidence strength and eligible providers;
6. required SQLite overlap fails closed when the pinned reference is unavailable;
7. every admitted-case failure and required provider mismatch fails the correctness
   lane without delta gating;
8. generated failures shrink to a replayable query, fixture, snapshot, generator
   version, and seed identity;
9. mutation testing compares state transitions and atomicity, not only response
   strings;
10. performance evidence fully scans the maintained query matrix in `wasm-release`,
    confirms ranked candidates, exercises scale sentinels, reports normalized costs,
    and compares only reproducible baselines; and
11. CI has explicit pull-request, scheduled correctness, scheduled performance, and
    closeout budgets.

The line improves evidence for the current SQL product. It does not broaden the SQL
language and it does not turn SQLite compatibility into a product requirement.

## Fuzzing Definition

In 0.204, “fuzzing” means bounded, seeded, schema-aware generative testing with
semantic properties, appropriately classified evidence providers, replay, and
shrinking. It has three required forms:

- valid typed SELECT generation;
- one-violation invalid SELECT mutation; and
- sequential DML state-machine generation.

This is distinct from coverage-guided arbitrary-byte fuzzing. A cheap lexical or
parser robustness target may be retained or added, but libFuzzer-style
infrastructure, long-running sanitizer campaigns, and randomized schema evolution
are not 0.204 closeout requirements. They can follow after the typed scenario
representation is a stable reusable input boundary.

## Authority Model

Authority is divided by concern rather than treated as a precedence list:

- accepted schema snapshots are authority for entity, field, type, constraint, and
  index shape;
- [`SQL_SUBSET.md`](../../contracts/SQL_SUBSET.md) is authority for intended accepted
  and rejected SQL behavior;
- the production parser, lowering, planner, executor, and storage paths are the
  system under test;
- the coverage manifest states evidence obligations and does not define product
  behavior;
- reference adapters state comparability rules and do not define IcyDB behavior; and
- generator models and checked-in corpus entries are bounded test inputs only.

An implementation disagreement with `SQL_SUBSET.md` is a product defect unless the
contract is intentionally amended. Existing parser, planner, or executor behavior
must not become self-authorizing merely because it is observable. An intentional
contract change updates the contract feature identifier, manifest obligation, and
current deterministic evidence in the same focused slice.

Generated `EntityModel` or `IndexModel` values may support test proposals and
fixtures, but they must not become runtime schema authority or a fallback for an
accepted snapshot.

The coverage manifest must not duplicate the SQL grammar or become a second product
specification. Provider eligibility must not silently amend accepted semantics. Any
contract, manifest, or adapter disagreement is a failing repository-consistency
check until the correct authority is deliberately updated.

## Evidence Architecture

The target evidence flow is:

```text
SQL_SUBSET.md ───────┐
                    ├──> coverage manifest ──> required evidence cells
accepted snapshot ──┘                              │
                                                   v
                                        typed scenario generator
                                                   │
                                 typed query + fixture + replay data
                                                   │
                                      current SQL renderer and runner
                                                   │
             ┌───────────────────────────────┬─────┴─────────────────────────┐
             v                               v                               v
        IcyDB result                 reference oracle               metamorphic path
             │                    / contract assertion                / boundary
             └───────────────────────────────┼───────────────────────────────┘
                                             v
                                normalized typed comparison
                                             │
                         pass or bounded minimized replayable failure
```

The generator owns a test-only typed model. The production parser remains the only
authority for accepting rendered SQL. The generator must not bypass parsing by
injecting production AST nodes into the executor for its main comparisons.

Internal direct execution can provide a consistency property when it represents a
maintained path, but it is not automatically an independent reference oracle. At
least one side of every public-SQL scenario must traverse the actual current SQL
frontend.

## Coverage Manifest

### Purpose

The manifest replaces the archived proof ledger as the current coverage index. It
answers four questions for each finitely enumerated contract feature:

- Is the feature accepted or explicitly rejected?
- Which evidence obligations and evidence strengths are required?
- Which stable tests or generated families provide those obligations?
- Which reference, metamorphic, contract, or boundary checks are eligible?

### Manifest Universe

`SQL_SUBSET.md` receives stable, unobtrusive feature identifiers as HTML comment
metadata immediately before the paragraph, list item, or table block that defines
the cell. Multiple canonical comments may precede one shared table block:

```html
<!-- icydb-sql-feature id="select.projection.scalar" kind="semantic" status="accepted" -->
<!-- icydb-sql-feature id="query.join" kind="syntax" status="rejected" -->
```

Identifiers are lowercase dot-separated segments matching
`[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)+`. `kind` is exactly `syntax`, `semantic`,
`policy`, or `interaction`; `status` is exactly `accepted` or `rejected`. The comment
format and attribute order are canonical so the checker does not need a permissive
metadata parser.

Each identifier is unique, machine-readable, and independent of heading text or test
names. The manifest must have exactly one current entry for every identifier and
must not contain identifiers absent from the contract.

Feature identifiers classify one of four finite kinds:

- `Syntax`: an accepted or explicitly rejected statement, clause, or expression
  family;
- `Semantic`: a value-type or clause behavior whose result semantics differ;
- `Policy`: an entrypoint, resource, admission, or fail-closed boundary; or
- `Interaction`: an explicitly supported composition of otherwise isolated
  features.

A type-specific combination gets its own semantic identifier only when its IcyDB
semantics, error behavior, normalization, or provider eligibility differ. Interaction
coverage has separate identifiers and cannot be inferred from isolated feature
coverage.

Every accepted syntax, semantic, and policy identifier requires at least one
isolated deterministic provider. Every interaction identifier requires at least one
deterministic composition provider. Generated coverage supplements these providers;
it does not replace them.

Every explicitly rejected identifier requires a deterministic typed rejection
provider at the maintained current boundary. It does not require tests for retired
aliases whose only purpose would be anti-resurrection.

Explicitly rejected features belong to the manifest only when `SQL_SUBSET.md`
enumerates their current boundary. The infinite set of other SQL inputs is not an
`outside contract` manifest category. It is covered by general fail-closed and
robustness properties without a false claim of exhaustive enumeration.

### Required Fields

Each manifest cell contains at least:

- a stable contract feature identifier;
- feature kind and corresponding contract location;
- acceptance status;
- statement, clause, expression, value-type, policy, or interaction category;
- required evidence obligations and minimum evidence strengths;
- an explicit performance-observation disposition of `none` or one or more
  performance obligations;
- permitted reference or invariant providers;
- one or more stable deterministic providers;
- generated-family membership when applicable; and
- a required rationale for intentionally unavailable reference oracles.

The feature identifier describes the maintained contract cell, not a test function.
Test renames do not churn the feature namespace. A current spelling can be part of a
syntax identifier, but a retired spelling is removed under the pre-1.0 hard cut.

### Evidence Classes

The initial evidence classes are:

- `Parse`: valid syntax accepts or invalid syntax rejects with a typed class;
- `Lower`: parsed syntax lowers to the maintained semantic representation;
- `Execute`: result values and row sets are correct;
- `Route`: typed planner/executor route facts satisfy the scenario;
- `Boundary`: the generated-canister/public policy is enforced;
- `State`: mutation state transition and atomicity are correct;
- `ReferenceDifferential`: a true reference oracle agrees within a declared
  normalization; and
- `Regression`: a historically minimized defect remains covered.

Performance is not a correctness evidence class. It is a separate evidence line
with its own required obligations, runner, baseline, and verdict. A manifest cell
declares either no performance obligation or one or more of `BroadScan`,
`ScaleSentinel`, `RegressionSentinel`, and `FocusedHotspot`. A performance
observation never satisfies a correctness obligation, and a correctness pass never
satisfies a required performance obligation.

Not every syntax cell needs a performance measurement. The checked-in performance
profile must, however, cover every maintained execution-route family, statement
family, storage surface, aggregate mode, projection mode, cache mode, and declared
scale stratum. A `none` disposition is valid for syntax-only and rejection cells;
the manifest consistency gate rejects a performance profile that leaves a required
performance stratum without a stable scenario.

### Evidence Strength

Evidence providers are classified by what they can prove:

- `ReferenceOracle`: expected truth comes from SQLite within its pinned overlap or
  from a deliberately independent state model;
- `MetamorphicInvariant`: two executions or transformations that should preserve
  semantics agree, while potentially sharing production logic;
- `ContractAssertion`: a hand-authored expected value, row set, typed rejection, or
  invariant is derived directly from the active contract; and
- `BoundaryAssertion`: the generated-canister entrypoint, serialization, admission,
  or resource boundary behaves as specified.

`FrontendEquivalent` and `ExecutionModeEquivalent` are metamorphic invariants, not
reference oracles. They may share parsing, planning, execution, or storage logic.
Two correlated execution paths cannot satisfy a manifest requirement for
`ReferenceDifferential`, even when they agree.

### Consistency Gate

A required manifest check fails when:

- a contract feature identifier has no manifest cell or a manifest identifier is
  absent from the contract;
- an accepted cell lacks its required deterministic evidence;
- an accepted interaction lacks its own deterministic composition provider;
- a provider names a missing scenario or generated family;
- a rejected cell is exercised only through a retired alias or other legacy path;
- a reference provider is required for a type or feature that its adapter declares
  incomparable;
- correlated metamorphic paths are claimed as independent reference evidence;
- two cells claim contradictory acceptance status; or
- duplicate or malformed feature identifiers exist.

This check is a bijection over explicit contract metadata and manifest entries. It
does not infer completeness from headings, test names, source-text searches, or the
manifest's own contents.

## Typed Scenario Model

Every deterministic or generated correctness scenario is represented by typed
test-only metadata. The representation includes:

- stable scenario or generated-family identity;
- contract feature identifiers;
- evidence obligation, evidence strength, and provider identity;
- generator version, root seed, derived sub-seed, and case index when generated;
- replayable accepted snapshot material;
- fixture fingerprint;
- statement and feature tags;
- typed query model and rendered SQL;
- expected acceptance or rejection class;
- required reference or invariant providers;
- result ordering semantics;
- route expectations expressed through post-0.203 typed diagnostics;
- row, expression-depth, statement-count, shrink, and artifact budgets; and
- replay metadata sufficient to reconstruct the case without an audit artifact.

SQL text is an output and replay aid, not the source of scenario classification.
Classifiers must not use `contains`, prefix tests, or ad hoc clause extraction from
the rendered SQL to determine the expected route or provider.

### Stable Identity

Deterministic scenario IDs are stable across harmless refactors. Generated cases
have an identity composed from:

- generator version;
- canonical snapshot/fixture family;
- generator family identifier;
- root seed;
- case index; and
- evidence-provider identity.

The generator version is hard-cut before 1.0. If its format or semantics change,
checked-in corpus entries are migrated to the new current format or removed. The
harness must not retain compatibility decoders for retired generator formats.

### Deterministic Machinery

Generation uses one test-owned `SplitMix64` implementation. All operations below are
wrapping unsigned 64-bit operations:

```text
state = state + 0x9e3779b97f4a7c15
z = state
z = (z ^ (z >> 30)) * 0xbf58476d1ce4e5b9
z = (z ^ (z >> 27)) * 0x94d049bb133111eb
output = z ^ (z >> 31)
```

For a non-zero unsigned bound, bounded choice computes
`threshold = (0_u64.wrapping_sub(bound)) % bound`, draws until `value >= threshold`,
then returns `value % bound`. Weighted choice uses the same operation over the
checked sum of weights and selects by cumulative half-open ranges. Zero bounds,
weight overflow, and an all-zero weight set are typed harness errors. The
implementation has fixed state-transition, bounded-choice, and weighted-choice
golden vectors.

A family/case sub-seed is the first eight bytes, interpreted as an unsigned
little-endian integer, of:

```text
BLAKE3(
  "icydb-sql-0.204/subseed/v1" ||
  generator_version_u32_be ||
  root_seed_u64_be ||
  len(family_id)_u32_be || family_id_utf8 ||
  case_index_u64_be
)
```

Each family and case therefore owns an independent random stream. Adding or
reordering another family cannot reshuffle existing cases.

Canonical generation order is accepted-snapshot entity identity, then stable field
identity, then stable index identity. Names are used only as bytewise UTF-8
tie-breakers when an accepted representation lacks a stable numeric identity.
Generation must not depend on `HashMap` or `HashSet` iteration, wall time, process
identity, thread scheduling, filesystem enumeration, or locale.

The generator version is bumped when any existing case identity can produce a
different query, fixture, classification, or replay record, including changes to:

- the RNG or sub-seed derivation;
- canonical schema ordering;
- generation choices or weights;
- the SQL renderer;
- fixture construction;
- provider eligibility; or
- canonical serialization.

Adding a new independently seeded family does not require a version bump when all
existing family outputs remain byte-for-byte unchanged. Golden test vectors cover
seed derivation, ordering, representative rendered SQL, fixtures, and serialization.

### Replay Snapshot And Serialization

A snapshot fingerprint alone is insufficient replay material. Every replay record
must either:

- embed the bounded canonical accepted test snapshot; or
- reference a checked-in immutable snapshot fixture by stable path and content hash.

If a referenced fixture changes or disappears, affected corpus entries are migrated
to the current form in the same hard-cut slice. The harness does not search for a
similar current schema or reconstruct one from generated models.

Replay and corpus records use versioned RFC 8785 canonical JSON. Large integers,
decimals, identifiers, and bytes use tagged string encodings rather than lossy JSON
numbers. Floating-point values use tagged IEEE bit encodings so NaN payloads and
signed zero are not normalized accidentally. Schema components, rows, and fields are
arrays in their canonical order. Unknown versions reject; there is no legacy decoder
before 1.0.

### Result Normalization

Comparisons operate on typed cells and explicit row-order semantics wherever the
test boundary permits. They must not treat tab-delimited display text as canonical
result truth.

Normalization is declared per evidence provider and follows these rules:

- ordered rows when SQL ordering is semantically defined;
- when order is undefined, a multiset of canonical typed-row encodings sorted
  bytewise, preserving duplicate counts;
- explicit null representation;
- exact integer comparison within the declared common type range;
- integer overflow or non-representability remains a typed outcome and is never
  normalized away;
- bytewise UTF-8 text comparison with no implicit Unicode normalization;
- collation and Unicode case behavior only when the provider declares an exact
  common rule;
- canonical byte comparison;
- exact decimal coefficient/scale comparison only when the provider has a lossless
  mapping; and
- no generic floating-point tolerance.

The initial SQLite reference excludes floating-point, NaN, infinities, and
signed-zero-sensitive cases. IcyDB contract and metamorphic assertions compare the
typed contract outcome; approximate functions require a feature-specific tolerance
recorded in the manifest, otherwise they are ineligible for differential evidence.

`RETURNING` rows are unordered unless `SQL_SUBSET.md` explicitly guarantees order
for that statement shape. Query order is guaranteed only by the active SQL contract,
not by incidental executor or SQLite output order.

Normalization must not erase a product distinction merely to make SQLite agree.

## Evidence Providers

A scenario may require more than one provider. No single provider covers the whole
SQL surface, and only providers classified `ReferenceOracle` satisfy independent
differential obligations.

### `SqliteReference`

SQLite is a `ReferenceOracle` for result and state semantics with a fair, pinned
mapping. It never decides whether IcyDB syntax should be accepted. Eligible families
include:

- scalar projection over overlapping primitive values;
- predicates and boolean combinations with aligned null behavior;
- ordering, limiting, offset, aliases, and distinctness when normalization is
  equivalent;
- aggregate and grouped queries whose function and coercion semantics align; and
- mutations over overlapping fields and constraints.

SQLite is not automatically authoritative for IcyDB decimals, floating point,
principals, ULIDs, managed fields, storage routes, metadata, generated-canister
policy, textual explain plans, or any feature whose coercion and null rules differ.

### `StateModelReference`

Mutation sequences are compared with a deliberately independent `ReferenceOracle`
over rows, constraints, and statement atomicity. The model is derived from the
contract and uses its own typed values and transition functions. It must not call
production predicate evaluation, assignment, normalization, planner, executor, or
storage helpers.

The state model is the primary DML reference. SQLite is a secondary reference only
for the explicitly enumerated DML overlap.

### `FrontendEquivalent`

The current SQL frontend result is compared with a separately constructed
maintained internal operation where such an operation exists. This is useful for
projection truth, filtering, ordering, and mutation state, but it must not revive
the deleted shadow public SQL path from 0.203. Because the paths may share production
logic, this provider is a `MetamorphicInvariant`, not a reference oracle.

### `ExecutionModeEquivalent`

Equivalent executions compare modes that should not change semantics, such as cold
and warm compiled-query state or typed eligible and fallback routes. Route selection
comes from typed diagnostics. This provider is a `MetamorphicInvariant`.

Route forcing, when required, is injected through a dedicated non-default
test-support capability compiled only into the integration-test dependency graph.
A repository invariant keeps that capability out of default, release, and wasm
builds. It must not be exposed as a production public API, SQL hint, environment
variable, persistent setting, or runtime fallback.

### `RejectionInvariant`

Out-of-contract generated variants must reject with the expected typed error class,
must not panic or return an internal invariant error, and must not mutate state.
Tests cover current boundaries, not removed legacy aliases merely to prove that old
behavior remains absent. This provider is a `ContractAssertion`, or a
`BoundaryAssertion` when the entrypoint boundary is under test.

### `IcyDbContractOnly`

Features with no fair external oracle retain deterministic evidence against the
IcyDB contract. Examples include generated-canister admission policy, introspection,
typed route diagnostics, and storage-specific behavior. These use
`ContractAssertion` or `BoundaryAssertion`, not nominal differential evidence.

### Provider Mapping

| Surface | Required primary evidence | Eligible additional provider | Explicitly not required |
| --- | --- | --- | --- |
| scalar SELECT overlap | contract assertion plus SQLite reference | frontend metamorphic invariant | SQLite for non-overlap types |
| DISTINCT / aggregate / grouped | contract assertion | SQLite reference where semantics align | textual plan equality |
| route-sensitive SELECT | contract assertion plus typed route fact | execution-mode metamorphic invariant | SQL-text route inference |
| INSERT / UPDATE / DELETE | state-model reference plus response/post-state assertions | narrow SQLite reference | display-text-only comparison |
| RETURNING | state-model reference plus typed returned rows/post-state | narrow SQLite reference | implicit row ordering |
| invalid current syntax | typed rejection and no mutation | rejection invariant | legacy anti-resurrection tests |
| DDL and catalog mutation | catalog-native deterministic tests | narrow state invariants | randomized SQLite DDL parity |
| introspection / EXPLAIN | IcyDB contract assertions | typed internal facts | SQLite output or plan parity |
| canister admission policy | live boundary tests | none | direct helper as policy reference |
| blobs / managed identifiers | IcyDB typed evidence | narrow byte/state model | automatic SQLite comparability |

## Deterministic Matrix Design

### Separate Correctness From Performance

The correctness matrix gets its own verdict and bounded runner. The performance audit
reuses stable scenario definitions and fixtures, but it retains separate
measurements, budgets, reports, baseline policy, and artifact policy.

A correctness scenario fails on an unexpected error, result mismatch, state
mismatch, route mismatch, panic, or internal invariant failure. A performance
scenario fails its performance verdict when a required observation is missing or
incomparable, the performance environment is wrong, a required result or route is
unstable, a configured regression budget is exceeded, or the run is incomplete. A
metric without a checked-in threshold remains discovery evidence and cannot by
itself fail a regression gate. The two verdicts must never be conflated.

An admitted scenario error is always a correctness failure. It also prevents the
performance run from passing because the required measurement is missing, but the
report retains both typed verdicts rather than relabeling the product defect as a
performance regression.

### Stratified Selection

Bounded runs select scenarios by declared strata rather than ordered prefix. At a
minimum, strata include:

- statement family;
- scalar versus grouped shape;
- value-type family;
- nullability;
- predicate family;
- ordering/limit/offset behavior;
- route expectation;
- mutation kind; and
- evidence strength and provider class.

Each required lane defines quotas or full enumeration for its strata. Selection is
deterministic for a given manifest and seed. Adding a scenario to one stratum cannot
silently evict the only selected scenario from another required stratum.

### Required Subset

The compact required subset is not a hand-maintained list of favorite SQL strings.
It is selected from the manifest to cover every `pr_required` contract identifier
within the pull-request budget. One scenario may satisfy several explicitly declared
identifiers, but the manifest records that mapping. The selected composition is
printed in machine-readable form and checked for coverage drift.

## Performance Evidence Program

Performance is a first-class 0.204 deliverable. The program must be broad enough to
discover previously unknown instruction hotspots and controlled enough to decide
whether a candidate is repeatable, scale-sensitive, and regressed relative to a
comparable baseline.

The primary production-cost signal is IC local instructions measured on the real SQL
frontend and execution path. Host wall-clock time is recorded for CI budgeting only;
it is not a query-cost oracle. SQLite timing is diagnostic context for the SQLite
adapter and is never an IcyDB performance baseline.

### Performance Profile And Coverage

A checked-in `PerformanceProfile` references stable scenario and contract feature
identifiers. It defines:

- the complete admitted deterministic query set for the broad scan;
- required route, statement, storage, aggregate, projection, cache, value-type, and
  scale strata;
- the exact scale-sentinel scenarios and fixture profiles;
- the reportable raw and normalized metrics;
- the metrics with checked-in regression thresholds;
- focused hotspot scenarios and their sample modes; and
- deterministic candidate-selection and confirmation limits.

The broad scan is full enumeration, not the current first-`N` default. Every
generated scenario ID in the selected deterministic matrix must appear exactly once
as a success or a typed failure. The report carries both the expected and observed
scenario-set hash. A prefix, time budget, missing shard, duplicate ID, or silently
truncated candidate set makes the performance verdict incomplete and failed.

The profile is complete only when every maintained execution route and major costly
semantic path has at least one stable representative. At minimum this includes
primary, secondary, scan, direct-row, kernel-row, pure-covering, hybrid-covering,
scalar aggregate, grouped aggregate, blob projection, heap mirror, journaled mirror,
and every maintained SQL cache mode. Exact route names come from typed diagnostics,
not SQL-text classification.

### Two-Stage Discovery And Confirmation

The scheduled performance lane has two deterministic stages.

`P1 BroadScan` runs the complete performance profile once in `wasm-release`. Its
purpose is discovery and route/phase attribution. Every scenario is measured in the
declared cold mode. Cold eligibility is proven by typed cache counters and fixture
isolation; a runner may not infer cold state from call order.

`P2 Confirm` reruns the deduplicated union of:

- the 20 highest-cost scenarios for every reportable raw instruction phase;
- the 20 highest-cost scenarios for every eligible normalized metric;
- the worst scenario in every required route, family, and scale stratum;
- every scenario that crosses a checked-in baseline threshold; and
- every checked-in `RegressionSentinel` or `FocusedHotspot`.

The union is ordered by stable scenario ID and is capped at 256 scenarios. Required
stratum representatives and threshold crossings are never truncated. If the union
would exceed 256, the lane fails and the profile must be deliberately revised; the
runner does not silently sample it.

Each P2 scenario runs five isolated cold samples and, when the maintained path has a
cacheable mode, five proven warm samples. Warm mode is established through the
test-support boundary and verified by typed cache-hit counters, not assumed from
execution order. A non-cacheable path records warm mode as typed `NotApplicable`
with its eligibility reason. Gating uses the median and reports minimum and maximum.
A sample set is unstable when its instruction range exceeds the greater of 10,000
instructions or one percent of its median. An unstable required candidate remains
failed until the source of variation or the reviewed stability policy changes.

P1 finds candidates cheaply; P2 confirms them without multiplying every scenario by
ten. A candidate discovered in P1 cannot be dismissed from the artifact merely
because it does not cross a regression threshold.

### Scale Sentinels And Normalized Cost

Small fixtures find fixed overhead but cannot establish scaling behavior. The
checked-in scale profile therefore includes exact row-cardinality sentinels at 16,
256, and 2,048 rows. It also includes deterministic zero-match, one-match,
approximately-quarter-match, and all-match predicates where the route family is
eligible, plus result windows of 1, 10, and 50 where the contract permits them.
Blob-bearing sentinels declare their exact payload distribution separately.

The profile uses a reviewed pairwise selection of these dimensions rather than a
full Cartesian product. Every maintained route family appears at all three row
cardinalities. Every eligible predicate route appears at both sparse and dense
selectivity extremes; exact-key, unique, or otherwise ineligible routes record their
typed reason and use their natural match cardinality. Fixture generation must
produce the declared counts exactly; an approximate selectivity records the exact
realized numerator and denominator in the artifact.

Reports rank both absolute and normalized cost. Eligible denominators include:

- fixture row;
- returned row;
- aggregate row ingested;
- data-store get;
- index range scan;
- index entry read; and
- output byte.

A normalized value is emitted only when its denominator is a measured typed counter
or declared fixture fact and is nonzero. The report never invents a denominator from
SQL text. For each scale-sentinel pair it also reports the marginal instruction
slope between adjacent cardinalities and calls out a route change separately from a
cost slope.

Every gated normalized metric and slope has a checked-in absolute and relative
threshold. The initial total-instruction migration threshold is an increase of both
100,000 instructions and 10 percent; it is imported from the maintained matrix
policy. Other phase and normalized thresholds are calibrated and checked in during
the performance slice before they can gate closeout. A metric without both values is
explicitly observation-only, never implicitly passing.

### Required Attribution

The matrix report retains every top-level phase needed to locate a hotspot:

- compile and compile subphases;
- planner and planner subphases;
- store;
- executor invocation;
- executor runtime;
- response finalization;
- grouped stream, fold, and finalize;
- scalar aggregate base-row and reducer work;
- direct-row, kernel-row, pure-covering, and hybrid-covering work;
- store get, range-scan, and entry-read counts;
- cache hits and misses; and
- returned rows and projected output bytes.

Executor-invocation and response-finalization fields already exposed by typed
attribution must not be dropped by the matrix sample. Grouped, scalar, projection,
and storage counters participate in rankings and delta reports rather than appearing
only in raw JSON.

The artifact carries a versioned phase-ownership table. It identifies nested or
overlapping counters and defines which fields reconcile to compile, execute, and
total cost so the report cannot double-count them. It records the attributable
residual for each top-level phase and the ratio of that residual to total
instructions. A new unexplained residual above the checked-in attribution budget is
a performance-instrumentation failure, not evidence that the unmeasured work is
cheap.

0.204 reports the performance dimensions it can measure honestly. Instruction
attribution, storage-operation counts, result cardinality, and output bytes are
required. Peak heap, allocator traffic, and stable-memory byte volume are marked
`NotMeasured` unless reliable typed counters are added; closeout must not claim that
instruction-hotspot coverage proves memory-hotspot coverage.

### Comparable Baselines And Verdicts

Every performance artifact records a `PerfEnvironmentIdentity` containing:

- performance-profile version and scenario-set hash;
- accepted-snapshot hash, fixture-profile hash, exact cardinalities, and data
  distribution identity;
- generator version and seed material, where applicable;
- `CanisterWasmProfile` and the exact `wasm-release` build configuration;
- raw non-gzipped wasm SHA-256 and byte size;
- source revision and dirty-state marker;
- Rust toolchain, wasm target, lockfile hash, and relevant feature set;
- PocketIC binary version and content hash;
- diagnostics/attribution schema version; and
- cache-mode and instruction-counter policy.

Identity hashes use BLAKE3 over domain-separated, length-prefixed canonical fields,
except for the wasm SHA-256 named above. The scenario-set hash is
`BLAKE3("icydb-sql-perf-scenarios/v1" || sorted_len_prefixed_scenario_ids)`. The
fixture-profile hash uses the same construction with domain
`icydb-sql-perf-fixture/v1` and the canonical fixture profile plus realized fixture
facts. Golden vectors pin both encodings.

Baseline comparison refuses to produce a pass/fail delta when profile, scenario,
snapshot, fixture, toolchain, target, PocketIC, attribution, or counter-policy
identity differs. Source revision and wasm identity are recorded as the expected
before/after subjects rather than required to match. Incomparable artifacts may be
shown side by side, but they cannot satisfy the performance verdict.

When a toolchain, PocketIC, profile, fixture, scenario set, or attribution identity
changes deliberately, the migration produces a bridge artifact: the previous
accepted source and the candidate source are both measured under the new identity.
The candidate is compared to that new-identity previous-source artifact. A newly
introduced required scenario that cannot run on the previous source receives an
initial reviewed budget only after three clean current-source confirmation runs and
is marked as having no historical delta. Neither case permits CI to accept an
incomparable old/current pair.

The baseline and current reports must contain the same required scenario IDs and
sample modes. A new failure, resolved failure, missing scenario, result-signature
change, route-invariant change, or cache-mode change is fatal to the performance
comparison and is not hidden behind an instruction-delta threshold. Product and
correctness ownership remains separately typed.

Checked-in thresholds use confirmed P2 medians. The initial migration gates total
instructions at the existing dual threshold of at least 100,000 instructions and 10
percent. Slice 7 must calibrate top-level phase, normalized-cost, slope, attribution
residual, and explicit per-sentinel budgets from clean repeated `wasm-release` runs.
Baseline updates are manual, reviewed changes with a rationale; CI never blesses a
new baseline automatically.

Any confirmed threshold regression and any scenario that remains in a raw or
normalized top-20 set across two scheduled runs is reviewed for promotion into the
focused five-sample performance audit. Promotion creates a stable focused sentinel,
not an allowlisted exception. A fixed or intentionally changed hotspot leaves the
focused set only through a reviewed profile change with replacement evidence.

## Schema-Aware SELECT Generator

### Typed Intermediate Model

The generator constructs a test-owned typed query tree containing:

- source entity and visible fields;
- expression type and nullability;
- literal, field, unary, binary, function, aggregate, and searched-CASE nodes;
- projection aliases;
- WHERE, GROUP BY, HAVING, ORDER BY, LIMIT, and OFFSET clauses; and
- explicit contract-feature, evidence-strength, and provider tags.

The renderer then emits SQL for the current contract. It does not emit alternative
legacy spellings.

### Generation Context

Expression generation is constrained by:

- accepted snapshot field types;
- clause context;
- aggregate versus scalar scope;
- maximum expression depth;
- operator input and output types;
- nullability;
- currently accepted function signatures;
- alias visibility rules; and
- provider comparability.

The first closeout target uses bounded expression depths from one through four.
Deeper exploration belongs in scheduled runs and must remain configurable.

### Query Families

The first generator covers:

- scalar projections and aliases;
- arithmetic, comparison, boolean, and string expressions;
- null predicates and null-producing expressions;
- searched CASE;
- maintained scalar functions;
- WHERE combinations;
- ORDER BY, LIMIT, and OFFSET;
- DISTINCT;
- global aggregates;
- GROUP BY and HAVING; and
- combinations of these features within explicit complexity budgets.

Every generated family must produce both simple isolated cases and interactions.
Coverage counters must demonstrate that the family is reached; a target case count
alone is not evidence of feature diversity.

### Invalid Variants

Invalid generation is separate from valid generation. It starts from a valid typed
case and applies one classified violation, such as:

- wrong operator type;
- illegal aggregate scope;
- unknown current field or alias;
- invalid clause combination;
- out-of-range bound; or
- unsupported function signature.

The expected failure class is attached before rendering. Arbitrary byte or token
fuzzing may remain a cheap parser robustness target, but it is not a substitute for
semantic invalid-case generation.

A case classified valid that the SQL frontend rejects always keeps the lane red.
Triage then determines whether the test renderer emitted invalid SQL or the product
rejected contract-valid SQL. A suspected harness-rendering defect is not converted
into an expected product rejection to make the lane pass.

## Fixture And Data Generation

Generated data is as important as generated SQL. Each family declares the data
properties needed to make its semantics observable.

The bounded fixture generator covers:

- empty, one-row, and multi-row entities;
- nullable and non-null values;
- duplicate projection and grouping values;
- minimum, zero, ordinary, and maximum numeric neighborhoods;
- empty, ASCII, Unicode, and prefix-related text;
- true/false distributions;
- byte values where a reference or invariant provider is fair;
- key uniqueness and deliberate constraint collisions;
- rows that select all, some, or none; and
- ties that expose ordering assumptions.

Fixture generation is snapshot-aware. It does not synthesize a runtime schema from
generated models, and it never writes invalid stored values merely to force an
executor path.

Every comparison reports the fixture seed and a compact canonical fixture. The
fixture size has a hard row bound in required lanes.

## Shrinking And Regression Corpus

### Failure Unit

A generated failure consists of:

- embedded canonical accepted snapshot or immutable snapshot-fixture reference;
- fixture;
- typed query or mutation sequence;
- rendered SQL;
- evidence providers;
- observed typed outcomes; and
- generator version, root seed, sub-seed, and case index.

### Mismatch Signature

Shrinking preserves a structured mismatch signature containing:

- contract feature identifiers;
- execution phase;
- subject and comparison provider identities;
- typed error class when relevant;
- mismatch category; and
- route-invariant or atomicity-violation category when relevant.

Mismatch categories include acceptance, typed error, row shape, value, ordering,
duplicate multiplicity, route, post-state, atomicity, boundary, panic/trap, timeout,
and internal invariant failure. Raw error messages, exact differing values, row
indices, and rendered display text are diagnostic details, not signature identity.

### Shrink Order

The shrinker minimizes while preserving the same mismatch signature and required
provider strength:

1. remove unrelated statements from a mutation sequence;
2. remove fixture rows;
3. remove projections, predicates, grouping keys, or ordering terms;
4. simplify expression nodes;
5. reduce expression depth;
6. reduce literals toward type-specific boundary values; and
7. reduce schema participation without changing accepted-snapshot authority.

Every shrink candidate is revalidated as an in-contract case before its failure is
accepted. A shrink that changes a valid differential mismatch into an unrelated
parse rejection is invalid.

Candidate order is deterministic. The initial required budgets are:

| Budget | Tier A native | Tier C / Tier E closeout |
| --- | ---: | ---: |
| shrink candidates attempted | 256 | 4,096 |
| complete scenario evaluations | 512 | 8,192 |
| generated fixture rows | 16 | 64 |
| mutation sequence statements | 8 | 32 |
| expression depth | 3 | 4 |
| canonical replay artifact | 256 KiB | 1 MiB |

A scenario evaluation includes the subject and every required provider for one
candidate. A wall-clock safety timeout is reported as infrastructure failure and does
not redefine the deterministic candidate or execution budget.

When the shrink budget is exhausted, the lane remains failed. The runner emits the
original replay plus the smallest signature-preserving case found and records
`minimization_complete: false`. Incomplete minimization must never hide, downgrade,
or reclassify the original defect. A generated replay record exceeding its lane's
artifact bound is itself a failing harness-budget violation.

### Corpus Policy

Confirmed failures are minimized and converted into small checked-in regression
cases after review. CI prints or attaches the proposed corpus entry; it does not
modify the repository during a run.

Corpus entries are data, not permanent exceptions. They assert the corrected current
behavior. They must not become a failure allowlist or preserve retired pre-1.0
formats. When the generator format hard-cuts, corpus entries move to the new format
or are replaced by direct deterministic tests.

## Mutation State Machine

The mutation generator operates as a bounded sequence over an accepted test schema
and an independently tracked row state.

### Initial Scope

The 0.204 state machine covers maintained forms of:

- single- and multi-row INSERT;
- INSERT from supported query shapes where a fair model exists;
- UPDATE predicates and assignments;
- DELETE predicates;
- ORDER BY, LIMIT, and OFFSET on mutation forms that support them;
- RETURNING projections;
- null/default behavior represented by the active contract;
- exact-key, range, and compound predicates;
- no-op mutations; and
- constraint or type failures that must be atomic.

Generated sequences begin with small schemas and row counts. Additional entity and
type families are admitted only after their state-model semantics are explicit.

### Per-Step Assertions

After every statement, the runner checks:

- typed response values and affected-row counts;
- returned rows under explicit ordering semantics;
- resulting IcyDB state;
- independent state-model reference;
- SQLite state only when the step belongs to the enumerated SQLite DML overlap;
- absence of partial mutation after a rejected statement; and
- continued validity of subsequent reads.

A failing sequence is replayed from its initial snapshot and fixture for every shrink
attempt. The harness must not continue from partially failed state and present later
symptoms as independent failures.

## DDL And Catalog Boundary

0.204 records deterministic DDL and catalog evidence in the manifest, but a
randomized schema-evolution state machine is not required for closeout.

The following remain in current deterministic coverage:

- accepted and rejected DDL syntax;
- catalog-native schema mutation semantics;
- migration and snapshot acceptance rules;
- metadata/introspection consistency; and
- atomic failure of catalog mutations.

The following are explicitly deferred:

- arbitrary DDL sequence generation;
- cross-version or upgrade fuzzing;
- crash/recovery fuzzing during schema mutation;
- SQLite schema-text or catalog-layout parity; and
- generated runtime fallback reconstruction from models.

If query and DML evidence closes early, a separately reviewed design may add a
bounded DDL state machine. It is not silently absorbed into 0.204.

## Failure Policy

Required correctness lanes fail on any of the following:

- unexpected IcyDB rejection of an admitted case;
- unexpected acceptance of a classified invalid case;
- typed result or row-set mismatch;
- state-transition or atomicity mismatch;
- required reference or invariant mismatch;
- required route mismatch;
- panic, trap, timeout, or internal invariant error;
- missing required provider environment or fixture;
- non-replayable generated failure;
- manifest coverage regression; or
- generator, renderer, adapter, or infrastructure failure that prevents required
  evidence from running.

There is no admitted-query common-failure allowlist. If an apparently valid generated
case is not accepted by the current contract, its generator classification is fixed.
If it is contract-valid, the product defect is fixed or the lane remains red.

Expected rejection is keyed by typed error class or structured diagnostic facts,
never error-message text.

Resolved expected failures are also treated as manifest drift. The current contract
and generator must be updated rather than carrying historical expectation records.

Baselines may report newly introduced, persistent, and resolved failures, but they
never determine the correctness verdict. Every admitted-case failure or required
mismatch is fatal even when it already existed in a previous report.

### Failure Ownership

Reports classify failures without changing their fatality:

- `ProductFailure`: IcyDB disagrees with the contract or a valid required reference;
- `HarnessRenderingFailure`: a typed valid case cannot be rendered and accepted as
  the intended current SQL;
- `ReferenceAdapterFailure`: reference setup, mapping, eligibility, or normalization
  is internally inconsistent;
- `InfrastructureFailure`: a required dependency, fixture, process, timeout, or
  artifact facility is unavailable; and
- `UnresolvedFailure`: evidence is insufficient to assign one of the preceding
  owners without guessing.

The original subject and provider outcomes remain in the bounded replay record.
Classification never turns a red result green.

Product defects discovered by any slice block the affected closeout criteria until
fixed and revalidated, even when the production fix lands in a separate focused
patch.

## SQLite Contract

### Pinned Runtime

Required lanes use a bundled SQLite library resolved to an exact source version and
checksum by `Cargo.lock`; they do not execute an arbitrary system `sqlite3` binary.
The adapter asserts that the runtime version equals the bundled compile-time version
and records both values, the source checksum, and `PRAGMA compile_options` in the run
artifact. Updating the bundled source or compile-option set requires an explicit
adapter-eligibility review and golden-environment update.

Slice 3 checks in one `SqliteEnvironmentContract` containing the exact version
number, source checksum, and sorted compile-option set resolved by the lockfile. A
required run compares its environment byte-for-byte with that record before any
scenario executes. A dependency update therefore cannot silently change the
reference environment.

Loadable extensions and application-defined collations or functions are disabled.
Required evidence must not depend on `SQLITE_ENABLE_UPDATE_DELETE_LIMIT` even if it
appears in a future compile-option set.

Local developer commands may offer an explicit non-SQLite mode, but their output
must clearly state that reference-differential validation was not performed.

An ordinary required command must never report a passing differential suite after
silently skipping all external comparisons.

### Connection Contract

Every SELECT scenario or mutation sequence receives a fresh in-memory database. The
adapter verifies and applies this exact connection policy before fixtures load:

- UTF-8 encoding;
- `foreign_keys = ON`;
- `case_sensitive_like = ON`;
- `journal_mode = MEMORY`;
- `synchronous = OFF`;
- `temp_store = MEMORY`;
- `trusted_schema = OFF`; and
- `recursive_triggers = OFF`.

Fixture setup and each comparison run occur inside an explicit transaction. A DML
sequence uses one transaction plus a savepoint around every statement expected to
reject, allowing atomicity and post-error continuation to be checked. Connections
are never reused across scenario identities.

### Adapter Rules

The SQLite adapter owns explicit mappings for:

- schema setup;
- identifier quoting;
- parameter binding;
- supported value conversion;
- result extraction;
- row-order semantics; and
- failure ownership as product, rendering, adapter, or infrastructure failure.

Adapters are narrow and feature-tagged. An unsupported mapping excludes a scenario
from `SqliteReference`; it does not coerce values until the outputs happen to agree.

The initial exact value overlap is:

- signed integers representable exactly as both IcyDB's declared type and SQLite
  `INTEGER`;
- unsigned integers no greater than signed 64-bit maximum, encoded as checked
  non-negative SQLite `INTEGER` values;
- booleans encoded as checked `INTEGER` values zero and one;
- UTF-8 text under SQLite `BINARY` collation;
- byte strings under SQLite `BLOB`; and
- null where both field and expression semantics align.

Unsigned integers above signed 64-bit range, all decimals, floating-point values,
NaN, infinities, signed-zero distinctions, principals, ULIDs, and managed fields are
excluded initially. Decimal contract assertions retain exact coefficient/scale
semantics; a future SQLite decimal mapping requires an explicit eligibility change.

Text equality and ordering are bytewise under `BINARY`. Case transformation and
case-insensitive matching are comparable only for ASCII inputs whose result is
identical under both engines. Unicode case folding, locale-sensitive behavior, and
implicit normalization are excluded.

The SQLite DML overlap is explicitly limited to standard INSERT, UPDATE, and DELETE
forms without mutation ORDER BY, LIMIT, or OFFSET. RETURNING may compare eligible
typed values, but its rows are treated as unordered. The independent state model is
the primary DML reference for every generated mutation family.

### Independence

The SQLite SQL is rendered from the typed scenario with a small declared adapter. It
must not reuse IcyDB result values, route decisions, or executor transformations to
compute expected output.

## CI And Execution Tiers

### Tier A: Required Pull-Request Core

The fast required lane runs without PocketIC and includes:

- manifest consistency;
- performance-profile consistency and normalization arithmetic checks;
- deterministic scenario selection checks;
- compact stratified SQL correctness cases;
- a compact required SQLite differential set;
- fixed-seed typed SELECT generation;
- generator determinism and bound checks;
- shrinker self-tests; and
- a fixed native mutation state-machine set.

The required profile runs:

- every deterministic provider marked `pr_required`, with the manifest check proving
  that every accepted syntax, semantic, policy, and interaction identifier, plus
  every explicitly rejected identifier, has at least one such provider;
- roots `0x1cdb_0204_0000_0001` and `0x1cdb_0204_0000_0002`;
- eight valid cases per SELECT generator family per root;
- four invalid cases per violation family per root; and
- four mutation sequences per DML statement family per root.

Tier A uses the native limits in the shrink-budget table. The bundled SQLite
environment is mandatory. Runner-only time after compilation is targeted at 180
seconds, and the required SQL runner has a 600-second hard timeout.

### Tier B: Required Live-Canister Boundary

The existing live SQL canister lane retains its boundary and policy assertions and
adds a fixed manifest-derived differential subset. It proves that the generated
canister surface, serialization boundary, and trusted SQL dispatch agree with the
native evidence.

The checked-in Tier B profile must contain:

- at least one boundary assertion for every canister-relevant syntax and policy
  identifier on the query, update, and DDL entrypoints;
- at least two scenarios for every canister-relevant SQLite reference family; and
- every stable scenario designated `live_required` by the manifest.

The full scenario ID list is checked in, capped at 96 cases, and validated against
the manifest. Tier B does not generate random cases. Its added differential portion
targets 300 seconds and has a 600-second hard timeout.

If the required manifest obligations cannot fit within 96 cases, Tier B remains red
until this design and the checked-in cap are deliberately revised. The runner never
truncates or samples the required profile to meet the cap.

### Tier C: Scheduled Multi-Seed Evidence

The scheduled lane runs:

- the full deterministic SQLite-reference matrix;
- multiple independent SELECT seeds;
- multiple independent mutation seeds;
- larger row and expression-depth budgets;
- all checked-in regression corpus entries;
- coverage-distribution reports; and
- failure minimization with replay artifacts.

Scheduled artifacts include exact seeds, generator version, manifest revision,
scenario counts by stratum, provider counts, and minimized failures.

### Tier D: Required Scheduled Performance

Tier D runs nightly on the default branch and is required again for 0.204 closeout.
It uses the checked-in `PerformanceProfile` and a `wasm-release` canister only. It
runs:

- the complete P1 broad scan across eight deterministic scenario-ID shards;
- all required scale sentinels;
- P2 candidate selection only after all P1 shard receipts merge successfully;
- five cold and five eligible warm P2 samples across eight deterministic shards;
- the focused performance baseline audit;
- comparable-baseline and configured-threshold gates; and
- performance, attribution, normalization, route, and raw wasm-size artifacts.

P1 has a 30-minute hard timeout per shard. P2 has a 45-minute hard timeout per shard.
The aggregate runner-time target is 120 minutes across both stages. Slice 7 records
aggregate runner time, wall-clock duration, build time, and the slowest shard from
three clean scheduled runs. A timeout, missing receipt, incomplete scenario set,
incomparable environment, unstable required sample, or confirmed threshold breach
fails Tier D.

Tier D uses the same BLAKE3 scenario-ID sharding algorithm as Tier C, with eight
required receipts per stage. Candidate selection, the 256-case cap, sample count,
profile, scale ladder, thresholds, shard count, and hard timeouts are reviewed
contract values. They never adapt to elapsed CI time.

### Tier E: Closeout

Closeout requires the full deterministic correctness matrix, the Tier C multi-seed
budget, and a clean Tier D performance verdict against a comparable baseline. A
clean performance report cannot override failed correctness, and clean correctness
cannot override missing or regressed required performance evidence.

Release/readiness work records raw non-gzipped wasm size where affected. The harness
itself is test-only and should have no production wasm delta. Any added production
attribution is measured for raw wasm bytes and instruction overhead before it is
accepted.

### Tier C Seeds, Quotas, And Sharding

Tier C uses these eight root seeds for both SELECT and DML, with family-derived
sub-seeds keeping their streams independent:

```text
0x1cdb_0204_0000_0011  0x1cdb_0204_0000_0012
0x1cdb_0204_0000_0013  0x1cdb_0204_0000_0014
0x1cdb_0204_0000_0015  0x1cdb_0204_0000_0016
0x1cdb_0204_0000_0017  0x1cdb_0204_0000_0018
```

For each root Tier C runs:

- 32 valid cases per SELECT generator family;
- eight invalid cases per violation family;
- 16 mutation sequences per DML statement family.

In addition, Tier C runs every deterministic SQLite-reference scenario and every
checked-in regression corpus entry once per scheduled run.

Tier C uses the scheduled limits in the shrink-budget table. Expression depths one
through four must each receive cases in every eligible expression family.

Tier A and Tier B use one shard. Tier C and each Tier D stage use exactly eight
shards. A scenario shard is the first eight bytes of
`BLAKE3("icydb-sql-shard/v1" || len(scenario_id)_be || scenario_id_utf8)`, interpreted
as an unsigned little-endian integer, modulo eight. All eight shard receipts are
required before the lane can pass; sharding distributes the complete case set and
never samples it.

Tier C targets 45 minutes of aggregate runner time across all shards, with a
15-minute hard timeout per shard. Slice 8 records aggregate runner time, wall-clock
duration, and the slowest shard from three clean CI runs. Changing seeds, quotas,
shard count, complexity limits, performance profile, sample counts, or hard timeouts
is a reviewed contract change, not an adaptive runtime decision.

No lane uses ordered `.take(limit)` or elapsed time as its coverage policy. Case
counts supplement, but do not replace, contract-identifier and stratum coverage.

## Reporting

Every run emits a concise summary containing:

- contract feature identifiers and manifest cells covered or missing;
- scenario counts by feature stratum;
- provider counts by evidence strength plus exclusions;
- generator root seeds, derived sub-seeds, and version;
- pinned SQLite environment identity when used;
- pass, expected-rejection, and failure counts;
- failure counts by product, rendering, adapter, infrastructure, or unresolved
  ownership;
- route distribution from typed facts;
- mismatch signature and shrink completeness for failures; and
- artifact locations when applicable.

Performance artifacts additionally contain:

- complete `PerfEnvironmentIdentity`, expected and observed scenario-set hashes, and
  every required shard receipt;
- P1 observation and P2 candidate-inclusion reason for each scenario;
- cold/warm eligibility, cache proof, and minimum/median/maximum samples;
- raw, normalized, and marginal-slope rankings by phase and stratum;
- phase-ownership version, attribution residuals, and instrumentation-calibration
  result;
- baseline comparability decision, per-metric thresholds, and typed verdict causes;
- measured and `NotMeasured` resource dimensions; and
- raw non-gzipped wasm hash and byte size.

Machine-readable artifacts use one current bounded format. Decoding must be fallible
and bounded. Before 1.0, format changes hard-cut to the new form without legacy
fallback decoders.

The ordinary passing output remains small. Full SQL, fixtures, and typed outcomes are
printed only for failures or explicit verbose/audit modes.

## Proposed Harness Shape

The implementation should decompose the current large audit ownership into focused
test modules. Exact filenames may follow Rust integration-test constraints, but the
responsibilities are:

- coverage manifest and consistency checks;
- typed scenario metadata;
- snapshot-aware fixtures;
- SELECT generation and rendering;
- mutation state machine;
- SQLite adapter;
- reference providers and metamorphic invariants;
- normalization and comparison;
- shrinking and corpus replay;
- CI selection and budgets;
- correctness reporting; and
- performance profile and scale fixtures;
- performance discovery and candidate confirmation;
- performance baseline, normalization, and delta policy; and
- performance reporting.

Shared scenario metadata may be reused by correctness and performance runners. Their
execution policies and verdict types stay separate.

No test helper should expose a new production public API merely to make the harness
convenient. Narrow test-only observability may be added where typed post-0.203 facts
are otherwise unavailable.

## Migration Invariant

Each implementation slice leaves exactly one current owner for every responsibility
it migrates. Superseded authority documents, classifiers, selection policies,
adapters, verdict paths, and reports are removed or marked historical in the same
slice that installs their replacement.

The existing monolithic performance audit may temporarily delegate shared scenario
or measurement work to new modules. Once a responsibility has migrated, the old
file must not retain a parallel classifier or correctness verdict and must not
receive new correctness logic. Delegation is transitional implementation structure,
not dual authority.

Focused validation must demonstrate that the current command reaches the new owner
and that no superseded runner remains an alternative passing path. A slice is not
complete when both old and new mechanisms can independently claim correctness.

## Non-Goals

0.204 does not:

- add SQL syntax, functions, types, or compatibility aliases;
- promise SQLite language or storage compatibility;
- make SQLite a runtime dependency;
- retain or recreate pre-0.203 shadow public query paths;
- build a second production parser, planner, or executor in the test harness;
- infer route truth from SQL strings or diagnostic prose;
- optimize production query plans merely to satisfy a route distribution target;
- fuzz arbitrary schema evolution, upgrades, or crash recovery;
- replace focused deterministic regression tests with random cases;
- accept known admitted-query failures behind an allowlist;
- reconstruct accepted schema snapshots from generated models;
- introduce backward-compatible decoders for retired test artifact formats; or
- turn a test-only harness change into an unrelated production refactor;
- benchmark every generated fuzz case or use fuzz-case count as performance proof;
- use host wall-clock or SQLite timing as an IcyDB query-cost oracle;
- auto-bless a performance baseline in CI; or
- claim peak-memory, allocation, or stable-byte hotspot coverage without reliable
  typed measurements.

Production defects discovered by the line are fixed in focused follow-up slices with
their own validation and changelog entries. They are not hidden inside a broad
harness patch.

## Implementation Slices

Each slice is a review and landing unit, not automatically a patch release. Each
slice stays within at most two primary technical domains.

### Slice 1: Current Coverage Authority

Deliver:

- reconcile the archived proof ledger against the post-0.203 SQL contract;
- introduce the machine-readable coverage manifest;
- add stable contract feature metadata and a checked contract/manifest bijection;
- classify syntax, semantic, policy, and interaction identifiers;
- map maintained features to isolated and interaction-specific deterministic
  evidence;
- add manifest consistency checks; and
- mark reference-provider exclusions explicitly.

This slice changes evidence metadata and documentation, not SQL behavior.

Exit criteria:

- every active contract feature identifier has exactly one manifest cell;
- accepted isolated and interaction cells have their required deterministic
  providers;
- explicitly rejected cells have current typed deterministic providers;
- retired spellings are absent;
- current introspection and mutation forms are represented;
- missing required evidence fails a focused check;
- the archived proof ledger is historical, not a second current authority.

### Slice 2: Typed Correctness Harness Core

Deliver:

- typed scenario and result metadata;
- evidence-strength and mismatch-signature types;
- typed route expectations from 0.203 diagnostics;
- correctness-specific verdicts;
- typed normalization;
- stratified deterministic selection; and
- separation of correctness from performance execution policy.

- no correctness classifier derives route or provider truth from SQL text;
- reference and metamorphic evidence cannot be confused in the manifest;
- an admitted scenario failure fails the focused runner;
- bounded selection preserves every required stratum;
- the performance audit can reuse scenarios without owning correctness verdicts;
- the superseded correctness classifier and verdict path are removed.

### Slice 3: Required SQLite Gate

Deliver:

- bundled lockfile-pinned SQLite and connection-environment checks;
- compact manifest-derived native differential coverage;
- compact live-canister differential smoke coverage;

- required CI wiring.

Exit criteria:

- required differential lanes fail when the pinned SQLite environment is absent or
  different;
- no required lane silently skips all comparison work;
- the compact set covers every declared high-level SQLite reference family;
- SQLite exclusions are visible and justified; and
- the optional system-CLI runners no longer own a correctness verdict.

### Slice 4: Typed SELECT Generator And Shrinker

Deliver:

- current-contract SQL renderer;
- fixed-seed valid and invalid generation;
- feature coverage counters;
- fixture generation for SELECT semantics;

Start with scalar SELECT, predicates, aliases, ordering, limiting, nulls, functions,

Exit criteria:

- all initial generated families reach their configured coverage;
- identical seed and generator version reproduce identical cases;
- family insertion and iteration-order changes do not perturb existing cases;
- every generated case respects its configured bounds;
- invalid cases carry a typed expected rejection before rendering;
- injected harness faults shrink to a smaller replayable case in self-tests;
- shrink-budget exhaustion remains red and emits the smallest case found.

### Slice 5: Aggregate And Execution-Mode Evidence

Deliver:

- DISTINCT, global aggregate, GROUP BY, and HAVING generated families;
- SQLite eligibility for semantically aligned cases;
- frontend or execution-mode metamorphic invariants where maintained;
- typed cold/warm or eligible/fallback comparison; and
- route-distribution reporting from typed facts.

Exit criteria:

- grouped and aggregate semantics have contract assertions and a reference provider
  where one is eligible;
- comparison does not depend on textual explain output;
- route forcing exists only behind the compile-time test-support boundary and cannot
  become a runtime fallback; and
- route distribution is evidence, not a product optimization target.

### Slice 6: Mutation State Machine

Deliver:

- accepted-snapshot-aware mutation schema families;
- bounded INSERT, UPDATE, DELETE, and RETURNING sequences;
- deliberately independent row-state reference model;
- explicitly enumerated secondary SQLite overlap;
- per-step state and atomicity checks;
- mutation-sequence shrinking; and
- replayable mutation corpus support.

Exit criteria:

- every maintained DML family has deterministic and generated state evidence;
- the state model calls no production evaluation or mutation helpers;
- rejected mutations leave both observed and modeled state unchanged;
- RETURNING is unordered unless the contract says otherwise and final state is
  checked independently; and
- a failing sequence minimizes from its initial fixture.

### Slice 7: Performance Discovery And Regression Line

Deliver:

- checked-in `PerformanceProfile` coverage and consistency checks;
- typed performance scenarios and route facts shared with, but verdict-independent

- fail-closed `PerfEnvironmentIdentity` comparison;
- complete `wasm-release` P1 broad-scan enumeration with deterministic receipts;
- deterministic P2 candidate selection and five-sample cold/warm confirmation;

- checked-in comparable baselines, threshold policy, and focused-hotspot promotion;
  and
- required nightly Tier D wiring and bounded machine-readable artifacts.

The current performance audit may be decomposed or replaced, but it must not retain
SQL-text route classification, prefix-based full-run semantics, debug-profile

Exit criteria:

- the full profile executes without first-`N`, time-based, or surface-prefix
  sampling;
- every required performance stratum and all three scale cardinalities have stable

- P1 and P2 scenario-set hashes, shard receipts, and sample counts are exact;

- raw, normalized, and slope reports reproduce deterministically from one artifact;
- required attribution fields are neither dropped nor double-counted;
- baseline comparison rejects every declared identity mismatch;
- missing scenarios, route/result/cache-mode drift, unstable samples, and confirmed

- three clean scheduled runs establish the reviewed time budget and calibrated

- any production attribution change has measured instruction overhead and raw
  non-gzipped wasm-byte delta; and

## Validation Plan

### Manifest Validation

- stable contract metadata and manifest entries form an exact bijection;
- syntax, semantic, policy, and interaction granularity is explicit;
- every accepted isolated and interaction cell has its required deterministic
  provider;
- provider references resolve;
- required evidence obligations and minimum strengths are present;
- metamorphic invariants cannot satisfy reference-differential obligations;
- retired aliases and legacy paths are absent; and
- reference-provider exclusions have an explicit reason.

### Harness Validation

- deterministic selection is invariant to scenario insertion order;
- typed normalization preserves meaningful distinctions;
- unordered comparison uses canonical typed-row multisets and preserves duplicate
  counts;
- correctness failures produce a failing verdict;
- performance reporting cannot suppress a correctness failure;
- correctness success cannot satisfy a missing or failed required performance
  obligation;
- route expectations use typed facts;
- route forcing is absent from default, release, and wasm dependency graphs;
- failure ownership remains distinct from verdict; and
- artifacts reject oversized or malformed input fallibly.

### Generator Validation

- fixed golden vectors reproduce sub-seeds, query, fixture, classification, and
  canonical serialization;
- inserting or reordering a family does not perturb existing identities;
- generation does not depend on unordered collections or ambient process state;
- replay embeds a bounded snapshot or resolves an immutable checked-in fixture with
  the expected content hash;
- generated SQL parses only when classified valid;
- type and clause constraints hold before rendering;
- depth, row, statement, shrink, execution, and artifact budgets hold;
- every required family reaches its minimum coverage;
- invalid variants reject in their intended typed class;
- an unexpectedly rejected valid case remains a failing result.

### Evidence Provider Validation

- bundled SQLite version, checksum, compile options, and PRAGMAs match the pinned
  connection contract;
- SQLite adapters round-trip every declared comparable value family;
- excluded types never silently coerce into the overlap;
- text collation, ASCII case eligibility, nulls, duplicates, numeric ranges, and
  byte mappings are explicit;
- floating point, NaN, signed zero, non-lossless decimals, and oversized unsigned
  integers remain excluded from the initial SQLite reference;
- the state model calls no production evaluation, normalization, mutation, planner,
  executor, or storage helper;
- metamorphic providers are never reported as reference oracles; and
- required provider absence fails closed.

### Shrinker Validation

- known synthetic query faults shrink structurally;
- fixture shrinking retains the triggering row relation;
- mutation shrinking replays from initial state;
- a valid mismatch never shrinks into an unrelated parse rejection;
- signature identity excludes raw messages and exact differing values;
- budget exhaustion emits the smallest case found with incomplete status and remains
  failed; and
- minimized records replay from checked-in corpus form.

### Mutation Validation

- response, unordered-by-default RETURNING rows, and final state are checked
  separately;
- no-op mutations preserve state;
- rejected statements are atomic;
- sequence replay is deterministic;
- SQLite and the state model remain independently initialized; and
- subsequent reads remain valid after both successful and rejected steps.

### Performance Validation

- profile checks reject a missing route, semantic-path, cache-mode, storage-surface,
  or scale stratum;
- P1 rejects prefix selection, duplicate IDs, missing IDs, a wrong scenario-set
  hash, or a missing shard receipt;
- only `wasm-release` artifacts can satisfy Tier D;
- `PerfEnvironmentIdentity` accepts a known comparable pair and rejects each
  identity-field mismatch independently;
- golden vectors pin scenario-set and fixture-profile identity hashes;
- a deliberate environment change requires a comparable previous-source bridge
  artifact before a candidate delta can pass;
- P2 candidate selection has golden coverage for every inclusion rule, stable
  ordering, deduplication, and the fail-closed 256-case cap;
- five cold and five eligible warm samples are required and their cache state is
  proven by typed counters;
- an injected sample spread above the fixed stability bound fails;
- 16/256/2,048-row fixtures reproduce exact cardinality and selectivity facts;
- raw, normalized, and adjacent-cardinality slope arithmetic has golden vectors,
  including zero-denominator ineligibility;
- executor-invocation and response-finalization survive sample serialization, while
  grouped, scalar, projection, storage, output, and cache fields participate in
  rankings and deltas;
- the versioned phase-ownership table prevents overlapping fields from being summed
  twice and exposes an injected attribution residual;
- a missing scenario, new scenario failure, result or route drift, cache-mode drift,
  unstable confirmation, and threshold breach each fail the performance verdict;
- a metric without a checked-in threshold is reported as observation-only rather
  than passing a gate;
- baseline updates require an explicit reviewed artifact and cannot occur during a
  comparison run;
- confirmed recurring candidates promote into the focused audit under the checked-in
  policy; and
- production attribution overhead and raw non-gzipped wasm-byte change are measured
  whenever the diagnostics path changes.

### Repository Validation

- focused native SQL checks pass;
- required SQLite differential checks pass;
- focused live-canister SQL checks pass;
- scheduled multi-seed checks pass before closeout;
- fixed seeds, quotas, shard receipts, complexity limits, and hard timeouts are
  enforced;
- only one current owner exists for each migrated correctness responsibility;
- required Tier D P1, P2, scale, focused, and baseline-comparison evidence completes
  against a comparable environment;
- documentation links and current commands are valid.

The full repository `make test` suite remains the user's push-workflow validation and
is not an agent closeout command.

## Risks And Mitigations

### Risk: The Manifest Certifies Its Own Universe

A manifest-only completeness check can prove that every listed cell has evidence
while silently omitting a contract feature.

Mitigation:

- make stable `SQL_SUBSET.md` metadata the finite universe;
- enforce a contract/manifest bijection;
- distinguish isolated and interaction identifiers; and
- require deterministic providers independently of generated counts.

### Risk: Correlated Agreement Is Mistaken For Truth

Two engines can share a bug, and two IcyDB paths can share most production logic.

Mitigation:

- classify reference, metamorphic, contract, and boundary evidence separately;
- forbid metamorphic agreement from satisfying independent differential evidence;
- keep focused hand-authored semantic tests;
- make the state model deliberately small and independent; and
- distinguish contract-only evidence from differential proof.

### Risk: SQLite Differences Create Noise

Coercion, null, collation, decimal, function, or ordering differences can create
false mismatches.

Mitigation:

- opt scenarios into SQLite through typed feature eligibility;
- pin the bundled version, compile options, PRAGMAs, collation, and value mappings;
- normalize only documented equivalent semantics;
- exclude unfair types explicitly; and
- treat adapter failures separately from product failures.

### Risk: The Generator Mostly Produces Trivial Cases

Large counts can hide poor feature interaction.

Mitigation:

- stratified family quotas;
- expression-depth distribution;
- fixture-property coverage;
- interaction counters; and
- review coverage distributions, not only totals.

### Risk: The Generator Mostly Produces Invalid Cases

An unconstrained grammar wastes budget and obscures semantic defects.

Mitigation:

- construct valid typed trees by design;
- isolate one classified violation for negative cases; and
- fail generator self-checks when a valid case rejects before comparison.

### Risk: Shrinking Changes The Failure

Naive text reduction can transform a semantic mismatch into unrelated invalid SQL.

Mitigation:

- shrink the typed tree and fixture;
- revalidate contract classification after every candidate; and
- preserve the structured mismatch signature and provider strength.

### Risk: Corpus Entries Become Compatibility Baggage

Old generated formats can accumulate shims before 1.0.

Mitigation:

- one current generator/corpus version;
- hard-cut migrations;
- promote important cases to direct current tests; and
- never add legacy decoders.

### Risk: Required Differential CI Becomes Slow Or Flaky

Process startup, PocketIC, or excessive seeds can make pull requests unreliable.

Mitigation:

- keep native and live-canister budgets separate;
- enforce fixed seeds, quotas, complexity bounds, and hard timeouts;
- reserve full matrices and many seeds for scheduled runs; and
- change budgets only through reviewed contract updates.

### Risk: A Broad Matrix Still Hides Scaling Hotspots

Thousands of one-sample queries over tiny fixtures can rank fixed overhead while
missing nonlinear scans, dense-result costs, and output amplification.

Mitigation:

- require 16/256/2,048-row route-family sentinels;
- include sparse and dense selectivity plus bounded result windows;
- rank measured per-unit costs as well as raw totals; and
- report adjacent-cardinality slopes and route changes.

### Risk: Performance Instrumentation Perturbs The Path

Fine-grained counters can add wasm bytes and instructions or can double-count nested
phases, making the observer part of the apparent hotspot.

Mitigation:

- use typed attribution on the real production path rather than a duplicate
  executor;
- version phase ownership and expose unattributed residuals;
- calibrate diagnostics-on against the narrowest trustworthy total-only sentinel;
  and
- report raw non-gzipped wasm bytes and instruction overhead for instrumentation
  changes.

### Risk: Baseline Drift Launders Regressions

Different fixtures, profiles, binaries, toolchains, scenario sets, or cache modes can
make an attractive delta meaningless. Automatically accepting a new baseline can
turn a regression into the new normal.

Mitigation:

- compare only artifacts with matching `PerfEnvironmentIdentity` fields;
- require exact scenario and sample-mode sets;
- bridge deliberate environment changes by measuring previous and candidate sources
  under the new identity;
- fail closed on result, route, cache-mode, or required-scenario drift; and
- update baselines only through reviewed changes with rationale.

### Risk: Full Performance Evidence Dominates CI

Repeating every matrix query in every cache and scale mode would be expensive enough
to discourage regular use.

Mitigation:

- keep Tier D scheduled and separate from the pull-request correctness budget;
- use one full P1 discovery sample and repeat only the deterministic P2 candidate
  union;
- cap P2 explicitly and fail rather than silently truncate; and
- shard both stages with all receipts required.

### Risk: Route Evidence Drives Product Distortion

Tests can encourage artificial route forcing or optimization-only changes.

Mitigation:

- assert route only where the contract or invariant requires it;
- otherwise compare execution modes semantically;
- keep performance thresholds separate; and
- use test-only forcing narrowly.

### Risk: 0.204 Recreates 0.203 Legacy Paths

Harness convenience could revive a deleted direct public-SQL helper or string-based
diagnostics.

Mitigation:

- require 0.203 closeout first;
- use only the retained trusted entry point;
- consume typed diagnostics; and
- reject new production public APIs whose only caller is the harness.

### Risk: Transitional Dual Authority Persists

The monolithic audit and typed harness could both remain active and disagree about
classification or verdicts.

Mitigation:

- migrate one responsibility at a time;
- remove the superseded owner in the same slice;
- permit the old audit to delegate but not independently judge correctness; and
- validate that each current command reaches exactly one verdict owner.

### Risk: DDL Scope Consumes The Line

Random schema evolution has a much larger state and recovery surface than query and
DML generation.

Mitigation:

- keep DDL deterministic in the 0.204 manifest;
- defer randomized DDL explicitly; and
- require a separate design before expanding that boundary.

## Cost And Complexity

### Production Runtime

The harness is test-only. The intended direct production runtime, query latency, and
raw wasm-size delta is zero. Reusing already exposed typed attribution has no new
runtime surface. If required attribution needs production diagnostics changes, the
change is accepted only with measured instruction overhead and raw non-gzipped wasm
bytes. Product fixes discovered by the harness are measured and reported in their
own slices.

### Test Code

The initial implementation adds typed generator, evidence-provider, shrinker, and
manifest structure. Raw test line count may grow before existing monolithic
ownership is decomposed. The closeout target is structurally simpler ownership:

- smaller responsibility-focused modules;
- fewer SQL-string classifiers;
- one current coverage authority;
- one shared typed scenario model; and
- separate correctness and performance verdicts;
- one comparable performance-environment identity; and
- one full-scan-to-focused-hotspot promotion path.

Overall harness complexity initially increases to gain independent generation and
shrinking, then should settle below the current conceptual complexity even if the
line count remains larger.

### CI Cost

Pull-request cost is bounded by stratified quotas and fixed seeds. Full deterministic
overlap and multi-seed exploration move to scheduled execution. Tier D adds a nightly
full `wasm-release` discovery scan plus a bounded repeated candidate set; it does not
multiply every scenario by the confirmation sample count. The implementation record
must report wall-clock changes for Tier A and Tier B before making them required and
must record three clean Tier C and Tier D timing runs before closeout.

### Slice Reporting

Every implementation slice reports:

- files touched and approximate line delta;
- whether implementation shape became simpler, stayed neutral, or became more
  complex;
- focused validation status;
- Tier A or Tier B runtime delta when affected;
- Tier C or Tier D runner-time and slowest-shard delta when affected;
- production performance delta when a discovered bug fix affects execution; and
- raw non-gzipped wasm-size delta when production code changes.

## Closeout Decision

The line closes on evidence quality, not scenario count. Completion requires a
contract-bounded coverage authority, appropriately classified evidence strengths,
deterministic replay and shrinking, an independent stateful DML reference, pinned
SQLite overlap, one owner per migrated responsibility, required CI gates that fail
on every correctness regression, and a clean comparable performance verdict that
can discover, confirm, scale, and retain query hotspots.

A larger random case count or a single small-fixture performance ranking without
those properties does not close 0.204.
