# IcyDB 0.168 — Runtime Store Backend Boundary

## Status

Proposed design.

## Summary

0.168 introduces a runtime store-backend boundary without adding a second
backend.

0.167 made store storage explicit at the schema and generated-wiring boundary:

```rust
storage(stable(
    data_memory_id = 20,
    index_memory_id = 21,
    schema_memory_id = 22,
))
```

The only implemented storage form remains stable storage. 0.168 should now
make the runtime store APIs backend-neutral enough that a future
`storage(heap())` implementation can be added without spreading stable-specific
iterator, range, diagnostics, and commit assumptions through executor,
reconciliation, diagnostics, and mutation code.

This is still not the heap-store release.

0.168 is also a hard-cut internal architecture release. It must not preserve
stable iterator APIs as aliases, shims, compatibility helpers, or fallback
routes once their call sites have moved to backend-neutral traversal. If an old
store traversal API no longer owns a real boundary, delete it rather than
keeping it as a convenience wrapper.

## Current State

The runtime store wrappers are thin stable-memory wrappers:

- `DataStore` owns a stable
  `BTreeMap<RawDataStoreKey, RawRow, VirtualMemory<DefaultMemoryImpl>>`.
- `IndexStore` owns a stable
  `BTreeMap<RawIndexStoreKey, IndexEntryValue, VirtualMemory<DefaultMemoryImpl>>`.
- `SchemaStore` owns a stable schema metadata `BTreeMap`.

These wrappers already provide a useful boundary for simple operations:

```rust
get(...)
insert(...)
remove(...)
len()
is_empty()
memory_bytes()
```

At the start of 0.168, data-store runtime surfaces still exposed
stable-structure shapes directly:

```rust
DataStore::entries() -> ic_memory::stable_structures::btreemap::Iter<...>
DataStore::range(...) -> ic_memory::stable_structures::btreemap::Iter<...>
DataStore::range_for_entity(...) -> ic_memory::stable_structures::btreemap::Iter<...>
```

0.168.1 removed these `DataStore` iterator-returning APIs and migrated live
data-store call sites to `visit_*` traversal. `IndexStore::entries()` already
snapshots into a `Vec`, but range/scan code still reaches through
store-backed raw key/value iteration paths and should be audited as part of
the runtime boundary.

Schema reconciliation and mutation staging code also uses store entry
iteration heavily. Before heap storage is admitted, those call sites should
consume backend-neutral traversal methods rather than stable-map iterators.

## Problem

Adding `storage(heap())` before tightening the runtime store boundary would
force one of two bad designs:

1. expose two incompatible iterator families from the store APIs; or
2. thread `if stable` / `if heap` branches through executor, diagnostics,
   reconciliation, schema mutation, cursor continuation, and commit paths.

Both would make heap support look simple initially while increasing long-term
storage-engine coupling.

The main risk is not the heap `BTreeMap` itself. The risk is allowing the
stable-memory implementation type to remain part of the crate-internal
store-wrapper contract between store wrappers and higher layers.

## Goals

- Keep stable stores as the only implemented runtime backend.
- Preserve stable row/index/schema ABI exactly.
- Preserve generated stable wiring exactly.
- Preserve stable memory IDs, stable keys, and allocation identity exactly.
- Replace stable-iterator return types at store boundaries with
  backend-neutral traversal operations.
- Keep executor, diagnostics, reconciliation, and schema mutation code from
  depending on stable-map iterator types.
- Define the minimal backend operation surface needed by current stable stores.
- Add proof tests showing stable behavior and scan ordering are unchanged after
  the boundary refactor.
- Keep `storage(heap())` reserved and rejected.

## Non-Goals

- No `storage(heap())` admission in 0.168.
- No heap-backed `DataStore`, `IndexStore`, or `SchemaStore`.
- No `DataStoreBackend::Heap`, `IndexStoreBackend::Heap`, or
  `SchemaStoreBackend::Heap` live runtime variants unless the stable-only seam
  cannot be landed without an internal enum. If an enum is introduced, it must
  have only a stable variant in 0.168.
- No WAL.
- No stable-memory migration.
- No stable key change.
- No physical memory ID change.
- No allocation metadata semantic change.
- No row codec, index-entry codec, or schema-store codec change.
- No relation semantic change.
- No query planner feature expansion.
- No unrelated SQL DDL publication-gate cleanup.
- No unrelated relation descriptor consolidation.
- No backwards-compatible aliases for removed store traversal APIs.
- No compatibility shims that route new traversal APIs through old public
  stable-iterator surfaces.

## Core Invariant

0.168 is a runtime API boundary refactor, not a storage behavior change.

For every generated stable store:

```text
data/index/schema stable memory IDs unchanged
data/index/schema stable keys unchanged
allocation identity unchanged
row/index/schema bytes unchanged
scan order unchanged
commit/recovery behavior unchanged
```

Stable-map implementation details must move inward. Higher layers should
depend on store operations, not on stable-map iterator types.

## Decisions

0.168 makes these decisions before implementation starts:

- Use visitor / try-visitor traversal as the primary hot-path store boundary.
- Traversal supports early stop without treating pagination or bounded scans as
  errors.
- Keep owned snapshot helpers only for diagnostics or explicitly
  allocation-oriented call sites.
- Do not introduce heap runtime variants in 0.168.
- Introduce internal stable backend wrapper types only if they are needed to
  remove stable iterator leakage cleanly.
- If commit/recovery code is untouched, prove unchanged behavior through
  focused tests plus source audit. If it is touched, require direct
  commit/recovery behavior tests.
- Owner-local cleanup is allowed only when required to remove stable iterator
  leakage. Do not combine this release with unrelated DRY cleanup.
- Removed traversal APIs are hard-cut. Do not keep aliases or fallback helper
  names for compatibility.

## Design Direction

### Data Store Boundary

Current data-store direct iteration surfaces must be removed from the
store-wrapper boundary and replaced with visitor-style traversal methods.

Stable iterator wrappers may exist only inside stable-backend internals. They
must not remain callable by executor, reconciliation, diagnostics, mutation,
cursor, or commit-facing code.

Candidate shape:

```rust
pub enum Visit {
    Continue,
    Stop,
}

impl DataStore {
    pub(in crate::db) fn visit_entries<E>(
        &self,
        visitor: impl FnMut(&RawDataStoreKey, &RawRow) -> Result<Visit, E>,
    ) -> Result<(), E>;

    pub(in crate::db) fn visit_range<E>(
        &self,
        range: impl RangeBounds<RawDataStoreKey>,
        visitor: impl FnMut(&RawDataStoreKey, &RawRow) -> Result<Visit, E>,
    ) -> Result<(), E>;

    pub(in crate::db) fn visit_entity<E>(
        &self,
        entity: EntityTag,
        visitor: impl FnMut(&RawDataStoreKey, &RawRow) -> Result<Visit, E>,
    ) -> Result<(), E>;
}
```

The exact names can change during implementation. `std::ops::ControlFlow` is
also acceptable if it makes call sites clearer. The important point is that
callers no longer receive `ic_memory::stable_structures::btreemap::Iter` and
do not abuse errors as normal early-stop control flow.

Visitor methods should preserve stable-map order for the stable backend. They
should not allocate unless the caller explicitly collects.

### Index Store Boundary

Index-store callers need:

```rust
get(key)
insert(key, value)
remove(key)
clear()
len()
is_empty()
generation()
state()
mark_building()
mark_ready()
mark_dropping()
memory_bytes()
visit_entries(...)
visit_range(...)
```

`IndexStore::entries()` currently returns a `Vec<(RawIndexStoreKey,
IndexEntryValue)>`, which hides the stable iterator type but forces allocation.
0.168 should decide which call sites need allocation and which should use a
visitor path.

Diagnostics may keep a snapshot-producing helper if it is explicitly named as a
snapshot helper:

```rust
snapshot_entries_for_diagnostics()
```

Execution and mutation paths should prefer traversal methods.

### Schema Store Boundary

`SchemaStore` can remain stable-backed in 0.168, but schema reconciliation and
diagnostics should not require stable-map iterator types from its public store
surface.

The expected operation families are:

```rust
load_latest(...)
load_version(...)
insert_snapshot(...)
remove_snapshot(...)
visit_snapshots(...)
footprint(...)
allocation_metadata(...)
```

If schema-store internals already hide the stable iterator fully, 0.168 should
document and prove that instead of adding unnecessary abstractions.

### Backend Naming

Use storage/backend vocabulary only where it owns a real runtime boundary.

Good 0.168 names:

```rust
StableDataStoreBackend
StableIndexStoreBackend
StableSchemaStoreBackend
DataStoreTraversal
IndexStoreTraversal
SchemaStoreTraversal
```

Avoid names that imply heap support is live:

```rust
HeapDataStore
StoreStorage::Heap
DataStoreBackend::Heap
```

If internal backend enums are introduced in 0.168, they should be stable-only:

```rust
enum DataStoreBackend {
    Stable(StableDataStoreBackend),
}
```

Do not add dead heap variants to make a future diff look smaller.

### Future Architecture Axes

The final storage design should keep runtime map backend and durability policy
as separate axes.

The runtime backend answers where rows, indexes, and schemas live right now:

```rust
enum DataStoreBackend {
    Stable(StableDataStoreBackend),
    Heap(RustBTreeDataStoreBackend),
}
```

Equivalent shapes can later exist for `IndexStoreBackend` and
`SchemaStoreBackend`. This is the stable BTree versus Rust-native BTree
distinction.

Durability policy answers what survives upgrade or crash, and how replay or
checkpointing works:

```rust
enum DurabilityMode {
    StableDirect,
    VolatileHeap,
    JournaledHeap,
    CachedStable,
}
```

Future modes need not be one-to-one:

| Storage form | Runtime backend | Durability policy |
| --- | --- | --- |
| `storage(stable(...))` | stable BTree backend | direct durable stable memory |
| `storage(heap())` | Rust-native BTree backend | volatile heap |
| `storage(journaled(...))` | Rust-native BTree backend | stable WAL/checkpoint |
| `storage(cached_stable(...))` | heap/cache front | stable durable backing |

0.168 prepares the backend operation seam. It does not decide the full
durability policy surface.

### Future WAL Readiness

0.168 does not add WAL, but it must avoid making WAL harder.

All current mutations should remain centralized behind `DataStore`,
`IndexStore`, and `SchemaStore` methods. Higher layers should not directly
mutate backend map internals. A future WAL layer must be able to observe or
wrap store-level insert, remove, clear, and schema-publication operations
without rewriting executor, reconciliation, or diagnostics code.

## Runtime Operation Inventory

0.168 should first inventory all direct store traversal call sites.

Known high-value areas:

- data full scans;
- data entity-prefix scans;
- index range scans;
- cursor continuation;
- diagnostics storage reports;
- schema reconciliation startup scans;
- schema mutation staging and publication;
- index rebuild validation;
- commit-window integrity/recovery interaction.

For each call site, classify the required operation:

| Call-site need | Preferred boundary |
| --- | --- |
| stream rows in key order | visitor/range traversal |
| collect diagnostics snapshot | explicitly named snapshot helper |
| count entries | `len()` or traversal count |
| estimate storage bytes | `memory_bytes()` |
| validate staged index contents | visitor over target prefix/range |
| resume from cursor anchor | range traversal from raw-key bound |

The inventory is not a license for broad DRY cleanup. It should change only
the code needed to remove stable iterator leakage or isolate allocation-heavy
snapshot helpers.

### 0.168.0 Inventory

Initial source-audit commands:

```bash
rg -n "stable_structures::btreemap::Iter|ic_memory::stable_structures::btreemap::Iter" \
  crates/icydb-core/src/db \
  --glob '!**/data/store.rs' \
  --glob '!**/index/store.rs' \
  --glob '!**/schema/store.rs'

rg -n "\\.entries\\(\\)|\\.range\\(|range_for_entity\\(|DataStore::entries|DataStore::range|IndexStore::entries|SchemaStore::" \
  crates/icydb-core/src/db \
  --glob '!**/tests/**'
```

0.168.0 findings:

- Stable `btreemap::Iter` type names are already confined to store modules.
- Higher layers still depend on store traversal APIs that are stable-shaped by
  behavior even where the concrete iterator type is hidden.
- `DataStore::entries()` was used by schema reconciliation, startup index
  preflight, diagnostics, integrity scans, and runtime context helpers until
  0.168.1 removed the API.
- `DataStore::range(...)` was used by physical stream access, runtime context
  helpers, and aggregate terminals until 0.168.1 removed the API.
- `DataStore::range_for_entity(...)` was used by aggregate terminals until
  0.168.1 removed the API.
- `IndexStore::entries()` is used by commit rebuild, diagnostics, integrity
  scans, schema reconciliation, startup index preflight, and schema mutation
  staging/publication.
- `index::scan::raw` still ranges directly over the index store's backing map
  and should move behind an index-store traversal method.

Heap-source audit:

```bash
rg -n "StoreStorage::Heap|DataStoreBackend::Heap|IndexStoreBackend::Heap|SchemaStoreBackend::Heap|storage\\(heap" \
  docs crates testing schema canisters
```

Finding: heap appears only in reserved syntax diagnostics, compile-fail tests,
and design/changelog prose. There is no live runtime heap backend.

## Required Tests

### Store Boundary

- Data-store traversal visits entries in the same order as stable BTreeMap
  iteration.
- Data-store range traversal preserves inclusive/exclusive raw-key bounds.
- Data-store entity-prefix traversal preserves compact data-key prefix bounds.
- Index-store traversal preserves raw index-key order.
- Index-store range traversal preserves cursor-resume bounds.
- Schema-store traversal preserves accepted snapshot ordering.

### Cross-Layer Behavior

- Full scan query results remain stable.
- Exact-key lookups remain stable.
- Secondary-index range scans remain stable.
- Cursor continuation remains stable.
- Schema reconciliation still reads accepted snapshots from the schema store.
- Schema index rebuild validation still classifies target and unrelated entries
  correctly.
- Diagnostics storage reports still include data/index/schema stores with
  `storage: stable`.

### Negative / Hard-Cut

- `storage(heap())` still rejects.
- Heap syntax with memory IDs still rejects.
- No generated heap wiring appears in stable generated output.
- No live `StoreStorage::Heap` runtime variant exists.

## Closed Questions

- Visitor / try-visitor traversal is the primary store boundary.
- Traversal must support early stop.
- Snapshot-producing helpers are allowed only where allocation is explicit and
  intentional, preferably in diagnostics-facing code.
- Stable backend wrapper types are optional and should be introduced only when
  needed to remove stable iterator leakage cleanly.
- Commit/recovery proof depends on implementation footprint: source audit plus
  focused smoke tests if untouched; direct behavior tests if touched.

## Proposed Verdict

0.168 should be a runtime boundary release, not a heap-storage release.

The useful closeout state is:

- stable remains the only implemented backend;
- store APIs no longer leak stable-map iterator types to higher layers;
- generated stable wiring and stable ABI are unchanged;
- heap syntax remains reserved and rejected;
- future heap work has a narrow runtime operation surface to implement.
