# 0.205 Grouped Early Materialization

Status: implementation in progress; design cut on 2026-07-14; Slice 1 contract,
exact profile, focused parity authority, Slice 2 route-proof alignment, the
Slices 3-5 ordered transition/page/count kernel, and typed runtime diagnostics
implemented on 2026-07-19; focused ordered/hash PocketIC evidence now passes,
while the broader comparable performance verdict remains pending

## Context

IcyDB already streams grouped input rows from a resolved key stream. The grouped
runtime reads one retained-slot `RowView`, applies the effective filter, and
folds that row directly into aggregate state. It does not first collect the
source rows into one full input vector.

The remaining early-materialization problem is later in the same pipeline.
`GroupedExecutionMode` currently distinguishes `HashMaterialized` from
`OrderedMaterialized`, but both modes retain every observed group in a hash
table/vector bundle until the input stream ends. Only then does grouped page
finalization apply `HAVING`, projection, cursor filtering, ordering, and the
page window.

For a route that already proves rows arrive contiguously in canonical group-key
order, retaining every group is unnecessary. Once the key changes, the prior
group can never receive another row. Its aggregate state can be finalized,
filtered, projected, and passed to bounded page selection immediately.

In this design, “grouped early materialization” names the problem being removed:
ordered grouped execution materializes ownership earlier and for longer than
the semantics require. The target is early **finalization** with materialization
only at the active-group and response-page boundaries.

## Audited Pre-0.205 Baseline

The repository audit on 2026-07-14, rechecked against the completed 0.204 tree
on 2026-07-19, established the following current state:

| Area | Current behavior |
| --- | --- |
| grouped input | `GroupedStreamStage` yields keys and decodes retained row slots one row at a time |
| generic grouped fold | `GenericGroupedFoldRunner` ingests each row into `GroupedAggregateBundle` |
| dedicated grouped count | `GroupedCountState` retains all `(GroupKey, count)` entries plus a bucket index |
| grouped route modes | `HashMaterialized` and `OrderedMaterialized` |
| ordered admission | planner proposes `OrderedGroup`; the currently reachable session route selects `OrderedMaterialized` for a direct equality-bound group-prefix stream without explicit SQL ordering |
| explicit group ordering | an explicit group-key `ORDER BY` can remain planner-ordered while generic load-order route projection reports `requires_materialized_sort` and selects `HashMaterialized` |
| hash fallback | explicit planner fallback reason plus runtime route projection; executor never upgrades |
| fold dispatch | route mode is recorded, but fold dispatch selects global DISTINCT, dedicated count, or generic aggregation without branching on `GroupedExecutionMode`; both physical labels retain all groups |
| page finalization | bounded grouped windows use visitor/selection state; only genuinely unbounded global ordering collects candidate rows |
| resource enforcement | total groups, estimated group bytes, per-group DISTINCT, and total DISTINCT are capped and fail with typed errors |
| grouped DISTINCT eligibility | field-target per-group DISTINCT is rejected by `AggregateKind::supports_grouped_streaming` and remains hash materialized |
| cursor contract | strict opaque grouped continuation over canonical group keys; continuation identity is planner-strategy-bound, so cross-mode parity compares semantic pages rather than cursor bytes |
| observability | EXPLAIN and plan metrics report the selected grouped materialization mode and downgrade reason |

This means 0.205 is not a row-streaming rewrite and must not reopen the source
stream, row decoding, cursor wire, or public response contracts. It replaces
full grouped-state retention only where existing planner and route proof already
establish contiguous group ownership.

## Problem Statement

The current ordered label proves a useful physical fact but does not consume
that fact:

```text
ordered key stream
  -> decode/filter one row
  -> lookup or create group in full hash table
  -> retain every group until end of input
  -> finalize all groups
  -> apply HAVING/projection/page selection
```

This has five costs.

### 1. Ordered And Hash Modes Have The Same Peak Group Retention

An eligible ordered route still allocates group entries and a stable-hash bucket
index proportional to all groups observed. The ordered proof changes the label
and final sort behavior, but not the dominant group-state lifetime.

### 2. Page Limits Cannot Stop Grouped Scans Early

Even when physical group order is exactly the requested output order, execution
does not finalize qualifying groups until the source is exhausted. A small
`LIMIT` therefore cannot stop after the page and lookahead boundary is known.

### 3. Dedicated Count And Generic Aggregates Duplicate Retention Shapes

The dedicated grouped `COUNT(*)` path and generic bundle use different state
containers, but both retain all groups. Adding ordered finalization separately
to both would deepen the fast-path ownership gap already recorded in
[`fast-path-inventory.md`](../../governance/fast-path-inventory.md).

### 4. Runtime Labels Overstate The Ordered Optimization

`OrderedMaterialized` truthfully says the path remains materialized, but it also
shows that the planner's strongest grouped order proof has no streaming grouped
execution mode to select.

### 5. Resource Caps Hide Avoidable Physical Retention

The existing caps correctly prevent unbounded state, but a cap is not a reason
to hold state after its semantic lifetime ends. Eligible ordered grouping
should have peak live aggregate state proportional to one group, not all groups.

## Design Goals

0.205 must:

1. replace `OrderedMaterialized` with one truthful `OrderedStreaming` mode;
2. consume only existing planner proof in its first implementation line and
   make route projection consume that same grouped-order fact;
3. keep at most one active group aggregate state for the canonical ordered
   fold, plus explicitly bounded page/output state;
4. finalize a group exactly once when the next distinct key proves it closed;
5. evaluate `HAVING` only after that group is finalized;
6. preserve projection, ordering, limit, offset, cursor, and lookahead results;
7. stop the input stream when a canonical page plus lookahead is complete and
   no later group can affect it;
8. keep hash grouping as the explicit bounded fallback for every incompatible
   shape;
9. preserve cumulative group/DISTINCT work caps while making the group-byte cap
   measure peak live state rather than already-released ordered groups;
10. expose the selected mode, downgrade reason, rows scanned, groups observed,
    groups finalized, and peak live grouped state through typed diagnostics;
11. cover generic aggregates and the dedicated grouped `COUNT(*)` family under
    one ordered transition owner; and
12. make no public SQL, fluent, cursor-wire, persisted-format, or schema change.

## Non-Goals

0.205 does not:

- add SQL syntax or admit new aggregate functions;
- broaden ordered-group planner semantic eligibility in the first
  implementation line;
- remove hash grouping;
- raise the configured `max_groups`, `max_group_bytes`, or DISTINCT caps;
- add disk or stable-memory spill;
- implement a cost-based optimizer;
- make unordered output order a public promise;
- redesign scalar streaming, DISTINCT projection, or `INSERT ... SELECT`;
- change the grouped cursor token or continuation signature;
- add snapshot isolation across cursor calls;
- stream public response bytes before query success is known; or
- preserve `OrderedMaterialized` as an alias, compatibility shim, or hidden
  fallback.

Broader eligibility for residual filters, grouped field-target DISTINCT, or new
aggregate state machines may follow only after this line closes with evidence.

## Authority Model

Authority remains divided by concern:

- accepted schema snapshots own group-field identities, physical slots, types,
  indexes, and row decoding;
- query planner semantics own `GroupedPlanStrategy` and its fallback reason;
- route planning revalidates physical direction, projection safety, and access
  capability without upgrading the planner proposal;
- the route-owned `GroupedExecutionMode` selects the runtime fold;
- the ordered fold owns group-transition detection and early finalization;
- aggregate state implementations own fold and final value semantics;
- grouped page selection owns `HAVING`, projection, cursor filtering, offset,
  limit, lookahead, and public row construction;
- the resource model owns work and memory caps; and
- EXPLAIN/metrics project typed facts from those owners without re-deriving
  eligibility.

Generated `EntityModel` or `IndexModel` values remain proposal,
reconciliation, model-only convenience, and test inputs. They must not be used
to reconstruct ordered execution authority when the accepted snapshot or route
proof is absent.

## Target Execution Shape

Eligible ordered grouped execution becomes:

```text
planner OrderedGroup proof
  -> route revalidation
  -> OrderedStreaming
  -> ordered key stream
  -> decode/filter one row
  -> derive canonical group key
  -> same key: fold into active group
  -> new key: finalize prior group
       -> HAVING
       -> cursor/offset
       -> projection
       -> bounded page + lookahead
  -> reset active aggregate state
  -> stop when the page is final
```

Incompatible execution remains:

```text
planner hash/top-k fallback or failed route revalidation
  -> HashMaterialized
  -> bounded full group table
  -> existing grouped finalization
```

The target has one semantic fork at the route-owned execution mode. Aggregate
functions, public rows, cursors, and errors must not acquire mode-specific
semantics.

## Execution Mode Hard Cut

The current enum:

```rust
enum GroupedExecutionMode {
    HashMaterialized,
    OrderedMaterialized,
}
```

becomes conceptually:

```rust
enum GroupedExecutionMode {
    HashMaterialized,
    OrderedStreaming,
}
```

Stable observability codes become:

| Mode | Code | Meaning |
| --- | --- | --- |
| `HashMaterialized` | `hash_materialized` | retains a bounded group table because contiguous canonical group ownership is not proven |
| `OrderedStreaming` | `ordered_streaming` | retains one active group and finalizes groups at proven key transitions |

`ordered_materialized` is deleted from current code, snapshots, and active
documentation in the same slice. Pre-1.0 hard-cut rules prohibit an alias or
dual diagnostic spelling.

If any admitted route genuinely needs ordered input plus full group retention,
it must be classified `HashMaterialized` unless a separately named and justified
physical mode is designed. The old name must not survive as a convenience
fallback.

## Ordered Eligibility

The first 0.205 line does not invent new planner semantics. The initially
reachable production cohort has an equality-bound index prefix, an implicit
canonical ascending group-key order, and no explicit SQL `ORDER BY`. It accepts
only a route for which all current semantic gates already hold:

- planner strategy is `OrderedGroup`;
- the access path proves group-key prefix order;
- the requested canonical output order is compatible with that physical order;
- descending traversal is used only when physical reverse traversal is proven;
- the grouped aggregate family is streaming-compatible;
- grouped `HAVING` is streaming-compatible;
- route projection is safe for ordered grouping;
- residual-filter and other current planner exclusions remain exclusions; and
- the group-key tuple is non-empty.

For an already-admitted `OrderedGroup` plan, Slice 2 makes route projection
recognize an explicit canonical group-key `ORDER BY` when the same physical
direction is proven. This removes the competing generic
`requires_materialized_sort` conclusion; it does not permit route planning to
upgrade a planner hash decision. Descending enters only when the planner admits
the grouped shape and route facts prove physical reverse traversal.

Grouped field-target DISTINCT remains `HashMaterialized`: the current
`GroupAggregateSpec::streaming_compatible` authority rejects it. 0.205 does not
change that eligibility decision.

Top-k ordering by aggregate/computed output remains `HashMaterialized`, because
the final rank of a group cannot establish that no later group belongs in the
page. Global/zero-key aggregation stays on its existing scalar or explicit
materialized path.

The executor may downgrade an `OrderedGroup` proposal before scanning if route
revalidation fails. It may never upgrade a hash proposal to ordered streaming,
even if runtime data happens to arrive sorted.

## Ordered Fold State Machine

One owner under `aggregate/runtime/grouped_fold` must implement the ordered
transition protocol. It is shared by the generic aggregate bundle and the
dedicated grouped `COUNT(*)` specialization.

Conceptually, the state is:

```text
OrderedGroupFoldState
  last_finalized_key: Option<GroupKey>
  active: Option<ActiveGroupState>
  groups_observed: u64
  groups_finalized: u64
  page_selection: OrderedGroupedPageSelection
```

`ActiveGroupState` contains:

- the canonical owned group key;
- aggregate states for that one group;
- aggregate completion flags required by current fold semantics.

For every filtered input row:

1. derive its canonical group key from accepted retained slots;
2. if there is no active group, open the first group and record its budget;
3. if the key equals the active key, fold the row into the active states;
4. if the key follows the active key in the proven direction, finalize the
   active group, feed it to ordered page selection, then open the new group;
5. if the key violates monotonic order, fail closed with a typed internal or
   corruption-class error; and
6. if page selection proves the response final, stop without opening or reading
   another group.

At end of input, finalize the active group once. Empty input emits no grouped
row, preserving current semantics.

The comparison must use the same canonical grouped-key ordering authority as
planning, materialized finalization, and cursor construction. No executor-local
tuple comparator may redefine null, numeric, enum, list, or composite-key
ordering.

## Finalization And Page Selection

Early finalization must reuse the maintained aggregate and grouped-output
semantics. It must not create a second implementation of aggregate result
construction or `HAVING` expression evaluation.

The ordered page selector processes one finalized group at a time in final
canonical order:

1. finalize aggregate values;
2. evaluate the compiled grouped `HAVING` expression;
3. reject a non-qualifying group without changing offset/page counts;
4. apply the strict grouped cursor boundary;
5. consume the initial qualifying-group offset;
6. project the grouped output row;
7. retain up to `limit + 1` qualifying rows for page plus lookahead; and
8. stop the source scan once lookahead proves the next cursor and no later group
   can affect this page.

The selector may retain less by storing the last emitted group key separately,
but it must never retain more than the admitted page/lookahead bound.

Unbounded grouped output remains an explicit response materialization boundary:
the public call returns an owned vector, so all emitted rows are retained even
though aggregate group state is released. This is not full group-table
materialization; only final response ownership grows, subject to existing public
admission and response bounds.

## HAVING

`HAVING` remains post-aggregation. Ordered streaming does not evaluate it while
a group is still open.

Only the currently planner-approved streaming-compatible `HAVING` family enters
`OrderedStreaming`. A group that fails `HAVING` consumes group-work budgets but
does not consume cursor offset, page limit, or lookahead. Execution continues
until enough qualifying groups are found or input ends.

0.205 does not broaden the compatibility classifier. Any later broadening must
show that the expression depends only on the finalized current group and cannot
require comparison with future groups or global state.

## DISTINCT

Grouped field-target DISTINCT remains `HashMaterialized` in 0.205 because the
current planner compatibility owner rejects it. Its existing per-group and
total DISTINCT-value caps remain authoritative and cumulative.

Global DISTINCT, SELECT-level DISTINCT, and any shape whose deduplication domain
crosses group boundaries likewise remain materialized. Input order alone never
upgrades a DISTINCT shape. A later design may admit active-group DISTINCT state
only with explicit planner, budget, parity, and performance evidence.

## Dedicated Grouped COUNT(*)

The dedicated count path keeps its optimized row/key extraction and count-state
update, but it must consume the shared ordered transition owner when the route
mode is `OrderedStreaming`.

The ordered count state is one `GroupKey` plus one counter. It must not allocate
the all-group `GroupedCountState` bucket index. `HashMaterialized` continues to
use the existing bucketed count state.

This split preserves the useful count hot path while preventing a second owner
from independently deciding whether group transitions are safe. The associated
structural guard closes the grouped dedicated fast-path ownership gap in the
fast-path inventory.

## Resource Accounting

0.205 separates cumulative work from live memory instead of charging released
ordered groups as though they were still allocated:

- `groups_observed <= max_groups` remains cumulative and bounds group work;
- `distinct_values_observed <= max_distinct_values_total` remains cumulative
  and bounds DISTINCT work;
- `distinct_values_in_group <= max_distinct_values_per_group` remains a
  per-group bound for materialized DISTINCT;
- `current_live_group_bytes <= max_group_bytes` becomes the memory authority;
  and
- `peak_live_group_bytes` records the maximum observed live charge.

The budget owner gains explicit reserve/release operations for canonical group
keys, aggregate states, and hash-container overhead. Hash mode
reserves each group and releases all group state at terminal finalization.
Ordered mode reserves one active group and releases it after its finalized
candidate has crossed into page/output ownership. Response rows and page
lookahead remain separately bounded public-output state; they are not charged
as live aggregate group state.

Configured limits do not increase. An ordered query may nevertheless succeed
under `max_group_bytes` where an equivalent forced hash route returns the
existing typed memory-limit error, because the two physical modes genuinely
have different peak live memory. That difference is deterministic, visible in
EXPLAIN/metrics, and is a resource outcome rather than a query-semantic
difference. Output parity is required whenever both modes complete.

The conservative size model must not reuse full-table `Vec` capacity growth as
the charge for one streaming group. The resource owner defines named charges
for active key/state and mode-specific container overhead so tests can lock the
calculation without executor-local `size_of` formulas.

Diagnostics expose:

- `groups_observed`;
- `groups_finalized`;
- `peak_live_groups` (expected `0` or `1` for ordered streaming);
- `peak_live_aggregate_states`;
- `peak_live_distinct_values`; and
- `early_scan_stop`.

0.205 establishes this work/live-memory split as an explicit resource-contract
change. Any later adjustment to configured limits or size charges requires its
own reviewed resource-model decision.

## Cursor And Live-State Semantics

The grouped cursor remains opaque and unchanged. It continues to encode a
strict boundary over the canonical emitted group key and remains bound to the
same query signature, direction, projection, aggregate, DISTINCT, `HAVING`, and
grouped shape.

0.205 may consume an already validated route-level resume optimization, but it
does not add a new cursor seek protocol. If execution begins before the cursor
boundary, finalized groups at or before that boundary are discarded by the
ordered selector without entering the response window.

Pagination remains best-effort over live state rather than snapshot-isolated.
Early finalization does not change mutation-between-pages behavior.

## Downgrade And Failure Rules

Downgrade is permitted only before the key stream is consumed:

- planner hash/top-k strategy -> `HashMaterialized`;
- planner ordered strategy plus failed route capability ->
  `HashMaterialized` with the existing typed downgrade reason; and
- planner ordered strategy plus successful route capability ->
  `OrderedStreaming`.

After ordered scanning starts, execution must not silently restart as hash
grouping. A restart would duplicate stable-memory reads and instruction work,
complicate diagnostics, and create a hidden strategy switch. An observed order
violation, corrupt row/key, or invariant failure therefore fails the query
closed.

The response is still constructed before the public call returns, so a failure
does not expose a partial page.

## Observability And Explain

EXPLAIN and execution metrics must report the physical mode that actually ran.
The new stable fact is `ordered_streaming`; `ordered_materialized` disappears.

Required typed facts are:

- planner grouped strategy;
- planner fallback reason, when any;
- route grouped execution mode;
- route downgrade reason, when any;
- groups observed and finalized;
- peak live grouped/aggregate/DISTINCT state;
- whether input ended naturally or stopped after page lookahead; and
- scanned, filtered, qualifying-group, and returned-row counts.

Snapshots should assert typed descriptor structure first. Rendering snapshots
may lock the current spelling, but runtime code and tests must not parse EXPLAIN
strings to recover the mode.

0.205 updates the public DTO/version policy only if that policy has already
been decided as part of the 1.0 blocker. It must not casually add public fields
to an unstable DTO merely to complete the optimization.

## Public And Persisted Compatibility

0.205 changes no:

- SQL grammar or admitted query shape;
- fluent query API;
- grouped result DTO semantics;
- cursor wire or signature method;
- accepted schema snapshot;
- row, index, marker, journal, or stable-memory bytes;
- generated canister method; or
- stable-memory allocation identity.

The implementation record must explicitly state “no persisted-format change.”
Any discovered need to change cursor or persisted bytes stops the slice and
requires a separate classified design.

## Proposed Module Shape

The exact filenames may adjust during implementation, but ownership should
converge toward:

```text
db/query/plan/semantics/grouped_strategy.rs
  planner OrderedGroup/Hash/TopK authority

db/executor/planning/route/contracts/execution/mod.rs
  GroupedExecutionMode::{HashMaterialized, OrderedStreaming}

db/executor/aggregate/runtime/grouped_fold/
  ordered/
    mod.rs              mode entrypoint
    state.rs            active group + transition state
    page.rs             ordered finalized-group consumer
  count/                count-specific fold state, not order eligibility
  generic/              generic aggregate state, not order eligibility

db/executor/explain/descriptor/load.rs
db/executor/plan_metrics.rs
  typed observability projection
```

The ordered module owns transition mechanics, not aggregate semantics. Generic
and count paths provide active-state construction/fold/finalize operations
through an internal enum or sealed static dispatch chosen before the row loop.
No trait-object dispatch is required in the hot loop.

## Migration Invariant

Each implementation slice must leave one owner for every migrated decision:

- one route execution-mode projection;
- one ordered key-transition state machine;
- one current observability spelling;
- one grouped page/cursor semantic owner; and
- one hash fallback path.

Once `OrderedStreaming` is wired, the old ordered full-table branch, enum
variant, descriptor node, metrics label, snapshots, and documentation are
deleted in the same coherent slice. Test-only forcing may select
`HashMaterialized` to prove parity, but production must not keep a shadow
ordered-materialized implementation.

## Validation Plan

Focused validation must prove both semantic parity and physical-state change.

### Semantic Parity

- single-key and composite group keys;
- ascending and descending canonical group order;
- nullable keys, repeated nulls, and ordering ties;
- every currently streaming-compatible aggregate family;
- multiple aggregates in one group;
- empty input and one-group input;
- aggregate `FILTER` expressions where currently admitted;
- compatible `HAVING` pass/fail mixtures;
- grouped field-target DISTINCT remains on the typed hash fallback;
- offset, limit, exact-page, short-page, and lookahead pages;
- initial and resumed grouped cursor pages;
- scalar/SQL public response projection and aliases; and
- forced hash vs naturally ordered output parity whenever both modes complete,
  plus typed/observable divergence when only hash exceeds the live-byte cap.

### Failure And Boundaries

- group count at and above `max_groups`;
- ordered active-group bytes and hash full-table bytes at and above
  `max_group_bytes`;
- released ordered groups no longer contribute to current live bytes while
  peak bytes remain observable;
- per-group and total DISTINCT at and above their caps;
- physical direction revalidation failure before scan;
- projection capability downgrade before scan;
- out-of-order key injection fails closed without hash restart;
- corrupt/missing row behavior matches the current consistency contract;
- top-k/aggregate-ordered shapes remain hash materialized; and
- no cursor/persisted-format bytes change.

### Physical Evidence

- `peak_live_groups <= 1` for ordered streaming;
- hash fallback retains the expected bounded group table;
- bounded pages stop after enough qualifying groups plus lookahead;
- large blob fields not referenced by group/filter/aggregate/projection are not
  decoded or retained;
- ordered grouped count no longer allocates the all-group bucket index;
- no new hot-loop dynamic dispatch or per-row plan branching; and
- EXPLAIN/metrics report the actual mode and stop reason.

### Performance Evidence

Use the 0.204 performance provider and comparable environment identity. At
minimum, characterize:

- 16, 256, and 2,048 input rows where current query bounds admit them;
- few-large-groups and many-small-groups distributions;
- page sizes 1, 16, and an unbounded/public-boundary case;
- generic aggregates, grouped `COUNT(*)`, and `HAVING`, with grouped DISTINCT
  retained as a hash control;
- cold and eligible warm execution; and
- hash fallback controls over identical logical results.

No performance claim is accepted from a debug-only or single-sample result.

## Cost And Complexity

### Runtime

Expected eligible ordered-mode effects:

- peak aggregate group state falls from `O(group_count)` to `O(1)`;
- bounded canonical page retention stays `O(page_limit + 1)`;
- group lookup changes from stable hashing/table lookup to active-key compare;
- small pages may scan only through the lookahead group; and
- hash fallback complexity remains unchanged.

The design does not promise lower total work for unbounded output or one very
large group. It must not regress those cases outside the reviewed performance
budget.

### Wasm Size

The implementation adds an ordered transition kernel and removes the ordered
full-table execution branch. Every production slice reports raw non-gzipped
`.wasm` bytes; gzip is secondary context. Any material growth must be explained
by a maintained runtime owner rather than duplicate mode-specific semantics.

### Source Complexity

The initial line adds ordered state/page modules but deletes the
`OrderedMaterialized` branch and prevents generic/count duplication. Closeout
should be conceptually simpler:

- fewer live group states;
- one order-transition owner;
- one explicit hash fallback;
- no shadow ordered materialized mode; and
- observability labels that match physical behavior.

Every slice reports files touched, approximate line delta, and whether the
implementation shape became simpler, stayed neutral, or became more complex.

The current 0.205 working-tree scope touches approximately 66 maintained
source, test, evidence, and active-documentation files at +2,665/-278 lines
(about +2,387 net), excluding the concurrent dependency and design-document
work. The source count grows because the physical route is now proven across
route selection, generic/count parity, diagnostics, and the 0.204 evidence
harness. The runtime shape is nevertheless simpler: one shared ordered
transition owner replaces full-table ordered retention, while hash grouping
remains the sole materialized fallback. Final release preparation must refresh
this approximate count after unrelated worktree changes settle.

## Risks And Mitigations

### Risk: Planner Proof Does Not Match Physical Stream Order

Mitigation: route revalidation occurs before scan, the fold verifies monotonic
canonical keys during execution, and violations fail closed without restart.

### Risk: Early Stop Skips A Later Qualifying Group

Mitigation: early stop is allowed only when physical group order is the final
page order and page plus lookahead is complete. Top-k or globally ranked grouped
output remains materialized.

### Risk: HAVING Changes Offset Or Cursor Semantics

Mitigation: only finalized groups that pass `HAVING` enter cursor/offset/page
accounting, matching current grouped page finalization order.

### Risk: DISTINCT Accidentally Enters The Ordered Lane

Mitigation: keep the current planner compatibility rejection and prove grouped
field-target DISTINCT remains `HashMaterialized`. Any later admission requires
a separate design and evidence line.

### Risk: Count Fast Path Creates A Second Ordered Owner

Mitigation: count supplies specialized active state to the shared transition
kernel; it does not own route eligibility or group-order transitions. A
structural guard locks this boundary.

### Risk: Diagnostics Freeze An Internal Detail Prematurely

Mitigation: use typed internal facts and follow the separately decided public
DTO/version policy. Do not parse or promote rendering strings as authority.

### Risk: Mode-Specific Memory Accounting Looks Like Semantic Drift

Mitigation: keep group and total-DISTINCT work caps cumulative, define
`max_group_bytes` explicitly as peak live aggregate state, expose the actual
mode and peak charge, and require output parity whenever both modes complete.
No configured cap is silently raised.

## Closeout Decision

0.205 closes when ordered grouped execution consumes its proof physically: one
active group is finalized at each canonical key transition, bounded pages may
stop after lookahead, incompatible shapes remain explicitly hash materialized,
and the complete 0.204 evidence line proves identical public semantics.

Renaming the mode without reducing peak live group state, adding an ordered
path that silently restarts as hash grouping, or raising configured resource
caps without a separate contract decision does not close 0.205.
