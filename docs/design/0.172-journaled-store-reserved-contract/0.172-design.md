# IcyDB 0.172 — Journaled Store Reserved Contract

## Status

Proposed for 0.172.0.

## Thesis

0.169 admitted explicit volatile `storage(heap())` stores. 0.170 made stable
and heap behavior explicit through storage capabilities. 0.171 made those
capabilities authoritative for mutation classification, commit participation,
recovery, and proof-facing diagnostics.

0.172 should be the first durability-design line after that groundwork:
reserve the contract for a future journaled cached-stable store without
admitting production journaled runtime behavior or weakening the existing
contracts.

The core question is:

```text
What would make Rust-BTree reads over stable-BTree durable truth safe when
writes are journaled, and where must that durability contract be proven?
```

This line does not make volatile heap stores durable by implication.
`storage(heap())` remains live-only. Any future journaled cached-stable
behavior must use an explicit storage form, explicit capability mapping, and
explicit journal-replay proof before it can be admitted.

## Why This Is 0.172

The previous storage lines intentionally staged the work:

- 0.167 made per-store storage explicit with `storage(stable(...))`;
- 0.168 hid stable-map traversal shapes behind runtime store boundaries;
- 0.169 admitted volatile heap-backed runtime stores;
- 0.170 described stable/heap behavior through capability axes;
- 0.171 proved write, commit, and recovery policy consumes those capabilities.

Only after those boundaries exist is it safe to discuss a journaled cached-stable
store. 0.172 should not start by adding a broad WAL implementation or admitting
production journaled stores. It should first reserve the syntax, define the
journal/fold contract, capability mapping, recovery authority, and proof rows
needed before journaled cached-stable storage can be admitted in a later minor.

## Summary

0.172 is scoped as a journaled-store reserved contract.

The reserved storage form is:

```rust
#[store(
    name = "sessions",
    storage(journaled(
        data_memory_id = 40,
        index_memory_id = 41,
        schema_memory_id = 42,
        journal_memory_id = 43,
    )),
)]
```

This spelling is intentionally separate from `storage(heap())`.

- `storage(stable(...))` remains the current durable stable-map store.
- `storage(heap())` remains volatile and live-only.
- `storage(journaled(...))` would be a future cached-stable durable store.

For journaled stores, the stable memories have distinct roles:

- `data_memory_id`: canonical durable data BTree source of truth;
- `index_memory_id`: canonical durable index BTree source of truth;
- `schema_memory_id`: canonical durable schema BTree source of truth;
- `journal_memory_id`: durable journal tail for accepted writes not yet folded
  into the canonical stable BTrees.

The future Rust `BTreeMap` data/index/schema structures are volatile read
caches/live views. They use no memory IDs and must be rebuildable from stable
data/index/schema BTrees plus committed journal tail.

The intended budget is four stable memory IDs per journaled store, not seven:
there are no separate checkpoint memories in this design.

No aliases, shims, or compatibility fallbacks should be introduced. Before
1.0, storage syntax should hard-cut to the chosen explicit form.

0.172 reserves and proves the contract. A later minor is responsible for
runtime admission.

## Reserved Contract Decisions

These decisions are part of the 0.172 reserved contract. They must be closed
before any later runtime admission work begins.

### Storage Form

Reserved storage form:

```text
storage(journaled(...))
```

Rationale:

- keeps volatile heap and durable journaled storage distinct;
- leaves room for a separate non-journaled `cached_stable(...)` if that is
  ever needed;
- makes cached-stable journaled behavior opt-in and grepable;
- matches the existing design notes that reserved journaled storage is
  a Rust-BTree read path backed by stable source-of-truth BTrees and a stable
  journal tail.

Rejected alternatives:

- `storage(heap(durable = true))`: blurs volatile and durable heap semantics.
- `storage(heap(journaled(...)))`: makes heap the policy owner instead of
  durability being an explicit storage mode.
- treating `storage(heap())` as durable after 0.172: not allowed.

0.172 should reject `storage(journaled(...))` as reserved with exact diagnostics
rather than instantiate journaled runtime stores.

### Reserved Diagnostic Contract

Reserved journaled syntax must produce a diagnostic equivalent to:

```text
storage(journaled(...)) is reserved for a future journaled cached-stable durable store and is not admitted in 0.172
```

Malformed journaled forms are distinct from recognized reserved syntax:

- `storage(journaled(data_memory_id = 40, index_memory_id = 41, schema_memory_id = 42, journal_memory_id = 43))`
  should fail as a reserved feature;
- `storage(journaled(journal_memory_id = 43))` should fail as malformed
  journaled storage because it is missing `data_memory_id`, `index_memory_id`,
  and `schema_memory_id`;
- `storage(journaled(foo = 1))` should fail as an unknown journaled storage
  field, malformed journaled storage form, or similarly precise diagnostic;
- duplicate, empty, or mixed journaled forms should fail as syntax/shape errors
  rather than silently falling through to a generic storage failure.

### Parser And Runtime Boundary

0.172 may add parser or schema nodes only as a reserved form if required for
exact diagnostics. It must not permit:

- accepted schemas with journaled storage;
- generated runtime wiring for journaled stores;
- store registration with journaled storage;
- runtime store handles with journaled storage capabilities;
- journaled data, index, or schema store construction.

Recognizing syntax for diagnostics is not runtime admission.

Implementation preference:

0.172 should prefer parser/derive-level reserved syntax handling over adding an
accepted schema-level `StoreStorage::Journaled` variant. A schema-level
reserved node is allowed only if required for exact diagnostics. In either
case, journaled storage must not become accepted schema metadata, generated
runtime wiring, registry metadata, runtime store handles, or data/index/schema
store construction.

### Durability Mechanism

0.172 defines the journal/fold contract before admitting a journaled
cached-stable runtime store.

The contract should answer:

- how stable data/index/schema BTrees remain the canonical durable source of
  truth;
- what stable memory stores the journal tail;
- how journal records identify logical row and schema mutations;
- how journal replay interacts with existing commit markers;
- whether journal record encoding is new hard-cut format or reuses existing
  commit-marker row operations;
- how committed journal records are folded into stable data/index/schema
  BTrees;
- how the applied journal tail is cleared, truncated, or advanced;
- what happens if journal replay fails.

0.172 must not admit production `storage(journaled(...))` unless journal
growth, fold bounds, replay bounds, and stable-source/journal consistency are
specified and tested. The reserved-contract default is: journaled syntax
remains rejected until those conditions are satisfied.

### Capability Mapping

If `storage(journaled(...))` is admitted in a later minor, its capabilities
should be explicit:

- allocation identity: present;
- durability: durable;
- recovery: stable base plus journal replay;
- commit participation: durable;
- schema metadata persistence: canonical stable schema history plus committed
  journal tail;
- relation source: durable source;
- relation target: durable target;
- live validation: supported.

This mapping must not be inferred from diagnostics DTOs, generated entity
models, Rust type order, or backend implementation details outside store
construction and registry capability derivation.

0.172 defines this intended capability mapping as design evidence only. Runtime
store handles must not return journaled capabilities in 0.172 because no
production journaled store can be accepted, generated, registered, or
constructed.

Durability is not a recovery implementation. Stable stores and future journaled
stores may both classify as durable commit participants, but recovery code must
consume the specific recovery capability:

- stable commit replay;
- stable base plus journal replay;
- none.

Journaled allocation identity must render as a four-role allocation: canonical
data memory, canonical index memory, canonical schema memory, and journal-tail
memory. It must not render as `storage: stable`.

### Existing Commit Marker Relationship

0.171 proved commit and recovery policy through existing capabilities. 0.172
must not admit journaled storage until transaction visibility authority is
chosen. Candidate models are:

- existing commit marker authority: journal records are prepared, commit
  markers publish visibility, and recovery replays only records covered by
  committed markers;
- journal commit-record authority: the journal owns begin/write/commit/abort
  visibility and existing commit markers do not publish journaled visibility;
- explicit hybrid: existing commit markers bound cross-store visibility while
  journal records provide ordered replay integrity.

Any new persisted format must be bounded, fallible, and versioned. There should
be no compatibility fallback decoder unless explicitly accepted as a hard-cut
format rule before 1.0.

0.172 may leave the candidate models documented, but a later runtime-admission
line must choose exactly one transaction visibility authority before
implementation: existing commit marker, journal commit records, or explicit
hybrid.

### Journal Fold Requirement

Decision:

0.172 does not admit production journaled storage without a bounded journal
fold story.

A later admission line must decide when accepted journal records are folded
into the canonical stable data/index/schema BTrees, how replay remains bounded,
and how journal growth is controlled. A pure append-only journal may be an
implementation experiment later, but it is not enough to admit a production
storage mode.

## Journal Invariants

If journaled storage is admitted later, journal replay must be:

- ordered: records replay in a deterministic sequence;
- bounded by transaction visibility: uncommitted records do not become visible;
- idempotent or explicitly protected against double-apply;
- fallible: malformed, partial, or future-version records fail closed;
- schema-aware: records identify the schema version or fingerprint required to
  decode or validate them;
- stable-source-consistent: recovered data, index, and schema state matches the
  canonical stable BTrees plus committed journal tail;
- fold-consistent: applying committed journal records to stable BTrees and then
  clearing/advancing the journal tail has a clear handoff;
- capability-derived: durable classification comes from journaled
  capabilities, not allocation identity or storage-mode checks.

## Read View Invariant

A future journaled store read view must be equivalent to:

```text
canonical stable data/index/schema BTrees
+ committed journal tail not yet folded
+ current live transaction overlay, if any
```

The volatile Rust `BTreeMap` caches are projections of that view. They are not
durable authority and must be rebuildable from the canonical stable BTrees plus
the committed journal tail.

## Journal Fold Invariant

Journal folding must be safe across interruption.

A fold must not advance or clear the durable journal tail until the
corresponding data/index/schema mutations are durably applied to the canonical
stable BTrees.

After interruption, recovery must be able to either:

- replay the still-unfolded committed journal tail; or
- observe that the fold completed and the tail was durably advanced.

Partial fold state must not make committed writes disappear or apply twice.

## Journal Record Taxonomy

0.172 should sketch the record model but not implement it. The preferred first
model is logical mutation records owned by the store capability/recovery
boundary, not direct physical map operations leaked into higher layers.

Tentative record categories:

```rust
enum JournalRecord {
    BeginTxn { txn_id },
    RowPut { txn_id, entity_path, primary_key, row_bytes, schema_fingerprint },
    RowDelete { txn_id, entity_path, primary_key, schema_fingerprint },
    SchemaPut { txn_id, store_path, schema_snapshot_bytes },
    CommitTxn { txn_id },
    AbortTxn { txn_id },
}
```

The later admitted format may use different names, but the first admitted model
should journal logical row/schema mutations. Index state should be rebuilt or
maintained during fold from accepted schema, not treated as independent durable
truth unless a later proof explicitly admits index journal records.

## Affected Store Set

Journaled durability must use the same affected-store definition established in
0.171:

- mutated data stores;
- maintained index stores;
- relation-maintenance stores when represented as prepared mutation
  operations;
- schema stores for schema-changing operations.

Validation-only reads must not make a store a journal participant.

## Hard-Cut Rules

- Do not make `storage(heap())` durable.
- Do not admit production `storage(journaled(...))` in 0.172.
- Do not keep reserved-syntax compatibility aliases after journaled storage is
  admitted.
- Do not infer journaled durability from allocation identity alone.
- Do not treat diagnostics DTOs as capability authority.
- Do not render journaled stores as `storage: stable`, even if they use stable
  memory internally.
- Do not add runtime fallback reconstruction from generated models.
- Do not change stable keys, stable memory IDs, row codecs, index codecs,
  schema codecs, or relation semantics unless explicitly hard-cut in this
  design.
- Do not add WAL or journal-fold code that bypasses store capability
  classification.
- Do not add separate checkpoint memories for journaled stores. The canonical
  stable data/index/schema BTrees are the durable source of truth.

## Non-Goals

- No durable behavior for `storage(heap())`.
- No transparent heap-to-stable flush.
- No production journaled runtime store.
- No journal replay implementation.
- No journal fold implementation.
- No migration tooling.
- No stable ABI change.
- No stable key change.
- No memory-ID reassignment.
- No SQL storage syntax.
- No relation semantic expansion.
- No compatibility shims.
- No fallback recovery path through generated models.

## Tentative Goals

- Keep `storage(journaled(...))` reserved while closing the design/proof
  contract.
- Define the journaled store capability mapping.
- Define journaled stable source-of-truth and journal-tail terminology.
- Define how journaled stores would participate in mutation classification
  after admission.
- Define recovery authority options for journaled data, index, and schema
  state.
- Define that volatile Rust BTrees are read caches/live views with no memory
  IDs.
- Prove stable and volatile heap behavior remain unchanged.
- Prove `storage(heap())` remains volatile while journaled syntax is reserved.
- Prove no higher layer branches on concrete journaled/stable/heap backend
  modes instead of capabilities.

## Relation Policy

Journaled stores should become durable relation sources and durable relation
targets only after stable base plus journal replay can recover the data and
relation-maintenance state needed for validation.

Until runtime admission, relation policy must continue to consume existing
stable/heap source and target capability axes. Reserved journaled syntax must
not add a relation branch in executor, relation, reconciliation, commit, or
recovery roots.

## Diagnostics Contract

Diagnostics may describe the reserved journaled contract and later render
journaled allocation metadata, but they remain projections:

- diagnostics must not derive capability truth;
- diagnostics must not render journaled stores as `storage: stable`;
- journaled allocation identity should render as journaled allocation, not
  stable-map allocation;
- volatile `storage(heap())` diagnostics remain unchanged.

## Source-Audit Expectations

Policy code should consume capabilities, not concrete storage modes:

```bash
rg -n "StoreStorage::Stable|StoreStorage::Heap|StoreStorage::Journaled|storage_mode" \
  crates/icydb-core/src/db \
  --glob '!**/data/**' \
  --glob '!**/index/**' \
  --glob '!**/schema/**' \
  --glob '!**/registry/**'
```

Expected result:

- no executor, relation, reconciliation, cursor, commit, or recovery policy
  root owns concrete storage-mode branching;
- diagnostics may render storage mode but must not derive policy from it;
- registry may derive capabilities from accepted storage metadata.

Allowed journaled hits in 0.172:

- parser/derive reserved-form diagnostics;
- schema validation for reserved forms, if required for exact diagnostics;
- compile-fail/UI tests;
- design/changelog proof text.

Disallowed journaled hits in 0.172:

- runtime store handles;
- accepted store metadata;
- generated runtime registration;
- registry accepted-store snapshots;
- executor, relation, reconciliation, cursor, commit, or recovery policy roots;
- data, index, or schema backend construction.

Persisted-format code should stay isolated:

```bash
rg -n "journal|fold|CommitMarker|StoredCommitMarker|RowOp" \
  crates/icydb-core/src/db/commit crates/icydb-core/src/db/data \
  crates/icydb-core/src/db/index crates/icydb-core/src/db/schema
```

Expected result:

- journal/fold record formats are isolated and bounded;
- existing commit marker codec changes are explicit and tested;
- row/index/schema codecs remain unchanged unless the proof matrix records the
  hard-cut change.
