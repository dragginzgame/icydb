# 0.139 Aggregate Semantic Sealing

## Status

Proposed

## Motivation

`0.138` introduced a clean aggregate lowering pipeline:

```text
SqlGlobalAggregateTerminal
  -> PreparedSqlScalarAggregateStrategy
  -> StructuralAggregateTerminal
```

`0.138.8` removed invalid internal state for aggregate targets by introducing:

```text
AggregateTarget::{Rows, Field, Expr}
```

However, one remaining structural impurity exists:

> Aggregate semantics are still partially encoded as flags and kind
> combinations, rather than as a sealed, canonical representation.

Concretely:

* `distinct_input: bool` is carried independently of aggregate kind.
* Some aggregate kinds, such as `MIN` and `MAX`, silently ignore `DISTINCT`.
* DISTINCT normalization currently occurs outside the semantic owner in
  `terminal.rs`.
* The system can still represent semantically invalid or redundant combinations
  internally.

Example:

```text
MIN(x) DISTINCT
```

is representable in terminal and partially normalized before strategy, rather
than being structurally impossible.

This leads to:

* soft boundary violations, with terminal performing semantic normalization
* reliance on conventions instead of types
* potential future drift if new aggregate kinds or features are added

## Goal

Introduce a sealed semantic aggregate representation such that:

* invalid aggregate combinations are unrepresentable
* DISTINCT semantics are encoded at the correct layer
* terminal becomes purely syntactic
* strategy becomes purely execution-bound

## Non-Goals

Do not change:

* SQL behavior
* output values or ordering
* DISTINCT semantics
* HAVING behavior
* projection remap ordering
* executor terminal kinds
* grouped aggregate semantics
* public APIs or persisted formats

This is an internal semantic representation upgrade only.

## Design

### 1. Introduce `AggregateSemantics`

A new internal representation captures normalized aggregate input meaning:

```rust
enum AggregateSemantics {
    Count {
        target: AggregateTarget,
        distinct: bool,
    },
    Sum {
        target: AggregateTarget,
        distinct: bool,
    },
    Avg {
        target: AggregateTarget,
        distinct: bool,
    },
    Min {
        target: AggregateTarget,
    },
    Max {
        target: AggregateTarget,
    },
}
```

Properties:

* `MIN` and `MAX` do not carry `distinct`.
* `DISTINCT` is only representable where it has semantic effect.
* Invalid combinations are structurally impossible.
* Normalization is performed once at construction.

`AggregateSemantics` deliberately does not include `FILTER`. The filter is an
execution predicate attached to the aggregate terminal, while this enum seals
the aggregate kind, input target, and DISTINCT semantics. The complete
executable aggregate identity therefore includes both `AggregateSemantics` and
the optional filter expression.

### 2. Move Semantic Normalization Into One Constructor

Add a single canonical constructor:

```rust
impl AggregateSemantics {
    fn from_terminal(terminal: SqlGlobalAggregateTerminal) -> Self
}
```

Responsibilities:

* map `SqlAggregateKind` to semantic variant
* normalize DISTINCT behavior:
  * drop DISTINCT for `MIN` and `MAX`
  * preserve DISTINCT for `COUNT`, `SUM`, and `AVG`
* preserve `AggregateTarget`
* return the filter expression separately, unchanged

After this step:

```text
Terminal is raw.
Semantics is canonical.
```

### 3. Refactor Strategy to Consume Semantics

Change:

```rust
PreparedSqlScalarAggregateStrategy {
    target: AggregateTarget,
    distinct_input: bool,
    descriptor_shape: ...
}
```

to:

```rust
PreparedSqlScalarAggregateStrategy {
    semantics: AggregateSemantics,
    filter_expr: Option<Expr>,
    descriptor_shape: ...
}
```

Strategy responsibilities become:

* map `AggregateSemantics` to descriptor family
* construct executor terminal
* expose projection metadata

Strategy no longer interprets raw `distinct_input`.

The strategy may keep `filter_expr` as a sibling field or wrap the two pieces in
a small internal terminal structure:

```rust
struct PreparedAggregateTerminal {
    semantics: AggregateSemantics,
    filter_expr: Option<Expr>,
}
```

The important invariant is that code never carries a raw `distinct_input` flag
after the semantic constructor has run.

### 4. Terminal Becomes Purely Syntactic

Remove all semantic normalization from:

```text
terminal.rs
```

Specifically:

* remove `preserve_distinct_for_strategy()`
* retain raw DISTINCT flag exactly as parsed/lowered

Terminal now only represents:

* aggregate kind
* raw input shape
* raw DISTINCT
* filter expression

### 5. Deduplication Uses Executable Semantic Identity

Current dedup relies on terminal equality after normalization.

After this change:

* dedup must use executable semantic identity
* the identity must include the filter expression

Either use the semantic terminal shape directly if it derives `Eq`, or introduce
a narrow key type:

```rust
struct AggregateTerminalIdentity {
    semantics: AggregateSemantics,
    filter_expr: Option<Expr>,
}
```

This ensures:

* `MIN(x)` and `MIN(DISTINCT x)` dedupe correctly
* `SUM(x)` and `SUM(DISTINCT x)` do not dedupe
* `SUM(x)` and `SUM(x) FILTER (...)` do not dedupe
* ordering and remap behavior remain unchanged

Projection layer must switch from raw terminal equality to executable semantic
identity equality before output remap indices are finalized.

## Ownership Rules

### `terminal.rs`

Owns:

* raw SQL aggregate facts
* no normalization
* no semantic interpretation

### `semantics.rs`

Owns:

* aggregate meaning
* DISTINCT normalization
* canonical representation
* executable identity construction, including filter-aware dedup keys

### `strategy.rs`

Owns:

* execution mapping
* descriptor selection
* executor terminal construction

### `projection/`

Owns:

* terminal collection
* semantic dedup/remap using semantics-owned identity

## Migration Plan

1. Introduce `AggregateSemantics`.
2. Implement `from_terminal(...)`.
3. Update strategy constructors to consume semantics.
4. Move DISTINCT normalization into semantics.
5. Remove normalization from terminal.
6. Update projection dedup to use filter-aware executable semantic identity.
7. Validate remap stability with existing tests.

## Validation

Run:

```bash
cargo fmt --all
cargo test -p icydb-core sql_aggregate --features sql -- --nocapture
cargo test -p icydb-core db::sql::lowering::tests --features sql -- --nocapture
cargo check -p icydb-core --tests --features sql
cargo clippy -p icydb-core --all-targets --features sql -- -D warnings
git diff --check
```

Focus validation on:

* DISTINCT behavior unchanged
* `MIN` and `MAX` DISTINCT correctly ignored
* filtered and unfiltered aggregates remain distinct terminals
* projection remap indices unchanged
* HAVING correctness preserved

## Expected Outcome

After this change:

* terminal is purely syntactic
* semantic normalization is centralized and sealed
* strategy is purely execution-bound
* invalid aggregate combinations are unrepresentable
* DISTINCT semantics cannot drift across layers

This completes the aggregate lowering model:

```text
Terminal (syntax)
  -> Semantics (meaning)
  -> Strategy (execution)
  -> Executor
```
