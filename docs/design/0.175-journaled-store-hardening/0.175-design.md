# IcyDB 0.175 — Journaled Store Audit, Hardening, And DRY Cleanup

## Status

Active design. This is a post-admission hardening line for the journaled
cached-stable store work admitted in 0.174.

0.175 should not add a new storage mode or alter the journaled durability
contract. It should audit the admitted implementation, remove duplicated
store-wrapper flow, harden recovery/fold edge cases, and add regression guards
so journaled stores keep the intended read/write shape.

## Thesis

0.174 admitted `storage(journaled(...))` with the 0.173 protocol foundation:

```text
canonical stable data/index/schema BTrees
+ marker-bound committed journal tail
+ volatile Rust BTree live projections
```

0.175 should now make that implementation boring and auditable:

- normal reads use store-wrapper projections and indexes, not recovery/fold
  machinery;
- normal writes append marker-bound journal batches and update live
  projections, not canonical stable BTrees;
- recovery/fold is explicit, idempotent, and isolated from query execution;
- data/index/schema store wrappers share common traversal and overlay
  mechanics where they have the same shape;
- diagnostics and capability rendering remain projections, not runtime
  authority;
- stable and heap behavior remain unchanged.

This is a cleanup and proof-strengthening release, not a feature-expansion
release.

## Why This Is 0.175

0.169 admitted volatile heap stores. 0.170 made storage behavior capability
based. 0.171 consumed durability classification at write/recovery boundaries.
0.172 and 0.173 reserved and fixed the journaled protocol. 0.174 admitted the
first runtime journaled cached-stable store.

After admission, the highest risk is drift:

- duplicated data/index traversal logic can diverge;
- read paths can accidentally reconstruct overlays or replay journal tails;
- fold/recovery helpers can leak into hot query paths;
- diagnostics or capability surfaces can become policy authority;
- performance can regress because journaled stores have more moving parts than
  stable or heap stores.

0.175 is the line to reduce that drift before adding more journaled features.

## Non-Goals

0.175 must not add:

- a new storage mode;
- new `storage(...)` syntax;
- durable heap semantics;
- checkpoint memories;
- a different journal publication strategy;
- a different journal-tail ordering authority;
- stable key changes;
- memory ID changes;
- row/index/schema codec changes;
- a commit-marker format change unless a hard audit proves the existing
  journaled marker payload is unsound;
- planner feature expansion;
- relation semantic expansion.

0.175 must not make diagnostics DTOs, generated `EntityModel`/`IndexModel`, or
Rust type order runtime authority.

## Hard Invariants

### Runtime Read Shape

Normal journaled reads must not:

- replay the journal tail;
- rebuild live caches;
- fold committed journal batches;
- validate full index divergence;
- construct storage diagnostics;
- recompute capabilities through schema reconstruction;
- materialize a full canonical-plus-live snapshot before bounded scans.

Expected read shape:

```text
query/session
  -> planner/executor route
  -> DataStore/IndexStore/SchemaStore wrapper
  -> live projection / canonical base / ordered overlay traversal
```

### Runtime Write Shape

Normal journaled writes must:

- validate through accepted schema/runtime metadata;
- build bounded marker-bound journal batches;
- publish journal tail through the commit-marker-bound protocol;
- update volatile live projections only after durable recovery has a path;
- keep canonical stable data/index/schema BTrees untouched until fold.

### Recovery/Fold Shape

Recovery and fold must remain explicit maintenance paths:

- recovery resolves commit markers before replaying journal-tail batches;
- recovery uses the fold watermark as the replay boundary;
- fold applies committed journal-tail records into canonical stable BTrees;
- fold advances the durable watermark before clearing folded tail batches;
- interrupted cleanup after watermark advance must remain recoverable;
- already-folded batches must not replay.

### Capability Boundary

Policy code should consume capability axes and mutation durability
classification, not branch directly on storage mode.

Allowed concrete storage/backend branching remains limited to:

- schema/derive/build parsing and validation;
- generated wiring;
- store construction;
- store-wrapper internals;
- registry capability derivation;
- diagnostics rendering;
- tests and source-audit fixtures.

## Work Areas

### 1. Closeout Audit Baseline

Start 0.175 with a read-only audit of the admitted 0.174 implementation.

The audit should verify:

- journaled public admission uses the full four-ID form only;
- generated journaled wiring registers data, index, schema, and journal roles;
- stable stores still use stable allocation identity and stable commit replay;
- heap stores remain volatile with absent allocation identity;
- journaled stores render distinct journaled diagnostics, never `stable`;
- journaled normal writes do not mutate canonical stable BTrees;
- recovery repairs missing marker-bound journal-tail publication;
- recovery/fold updates canonical data/schema and derived index state;
- source guards cover concrete backend policy branching;
- hot query paths do not call recovery, fold, diagnostics, or full-snapshot
  overlay reconstruction.

### 2. DRY Store Overlay Traversal

Data and index store wrappers have the same logical ordered-overlay problem:

```text
canonical stable ordered map
+ live heap ordered map
- tombstones
```

0.175 should keep one internal helper for:

- live-over-canonical key precedence;
- tombstone visibility;
- ascending and descending traversal;
- range bounds;
- early stop;
- store-specific visitor adaptation.

The ordered overlay helper contract must be concrete and testable. It must be
streaming: it must not allocate a merged `BTreeMap`, `Vec`, or full snapshot
before invoking the visitor.

Required semantics:

- canonical-only key is visible unless tombstoned;
- live-only key is visible unless tombstoned;
- live value wins over canonical value for duplicate keys;
- tombstoned key is hidden;
- raw key order is preserved;
- range bounds are respected;
- ascending and descending traversal are supported if store wrappers expose both;
- visitor early-stop stops traversal immediately;
- helper does not call recovery, fold, diagnostics, or schema reconciliation.

Data and index wrappers should own only:

- concrete key/value types;
- stable-map iterator adaptation;
- live-map iterator adaptation;
- store-specific visitor return type.

Schema store traversal joins this boundary when accepted-schema lookup is on a
query hot path. Its ordered overlay use must preserve latest-version semantics:
bounded entity ranges, descending early stop, live-over-canonical duplicate
precedence, tombstone fallback to prior visible versions, and no full snapshot
materialization for `latest_*`, `len`, `is_empty`, or raw snapshot visits.

### 3. Snapshot Boundary Audit

Allocation-heavy snapshot helpers are allowed only when explicitly named and
owned by diagnostics, fold, or tests.

Any remaining snapshot/materialization helper must encode owner and purpose in
its name, for example:

- `*_snapshot_for_fold`;
- `*_snapshot_for_diagnostics`;
- `*_snapshot_for_test`.

Generic production names such as `journaled_entries_snapshot` are not allowed
in store-wrapper code because they are too easy to call from hot paths.

0.175 should prove:

- data reads do not use a journaled snapshot materializer;
- index scans do not use a journaled snapshot materializer;
- fold-only index materialization is clearly named and isolated;
- diagnostics snapshot construction does not feed runtime policy;
- `is_empty` stops after the first visible entry;
- `len` may stream/count visible entries but must not allocate or materialize
  a merged overlay snapshot.

### 4. Recovery And Fold Hardening

0.175 should add focused recovery/fold edge-case evidence without redesigning
the protocol.

Preferred tests:

- missing marker-bound journal batch is repaired from commit-marker payload;
- duplicate matching journal batch is reused without duplicate replay;
- mismatched journal batch fails closed;
- journal sequence gap after fold watermark fails closed;
- folded-visible batch is skipped or cleaned according to watermark rules;
- cleanup interruption after watermark advance does not lose committed rows;
- recovery rebuilds derived indexes from recovered row/schema truth;
- schema fingerprint mismatch fails closed before row replay publishes state.

If a failure mode is not safely injectable through existing test seams, record
the gap rather than adding broad production injection infrastructure.

0.175 should prefer store/protocol-level test seams over new production
failure-injection machinery. Any new injection point must be test-only or
demonstrably inert in production.

### 5. Hot-Path Performance Guards

0.175 should keep lightweight regression guards around the flows that exposed
the 0.174 performance issue.

Required proof:

- stable and journaled `ORDER BY id ASC LIMIT 1` scans visit one row;
- journaled data and index range traversal honors early stop;
- journaled indexed queries keep the index route;
- journaled read-only queries do not call recovery/fold;
- journaled read-only queries do not construct storage diagnostics;
- journaled read-only queries do not decode/replay the full journal tail;
- query cache hits do not reconstruct accepted schema authority, diagnostics,
  or capability DTOs before cache lookup.
- journaled schema latest-snapshot and range reads stream the canonical/live
  overlay with tombstones and descending early stop;
- cold SQL query calls load the accepted runtime schema directly when present
  instead of rerunning generated-model reconciliation on every cache-key build;
- immediate SQL compile-to-execute paths carry accepted schema, schema
  fingerprint, and accepted authority through plan lookup;
- cold query-call behavior is cheap by construction and does not rely on query
  cache writes being persisted.

Optional ignored native timing reports may compare stable, heap, and journaled
stores, but correctness tests should not rely on fragile absolute instruction
thresholds.

Current canister evidence, observed 2026-06-01:

- `select name from character order by id asc limit 1` on `demo_rpg` dropped
  from roughly `58.3Mi` to `38.3Mi` after the schema-store streaming and cold
  SQL context reuse changes.
- The latest reported shape is:
  `1 row, 38.3Mi [ccccccccccccppppeeee] {er=8.2Mi} {r=9us}`.
- This is a material improvement, but still too expensive for the target cold
  query-call path. Executor cost remains around `8.2Mi`, so the remaining work
  is primarily compile/planner-side reduction.

  read traversal;
- cold SQL query-call cache-key and compile/plan paths do not rebuild accepted
  schema snapshots when a runtime accepted snapshot already exists;
- fold/diagnostics/test materialization is purpose-named.

### 7. Documentation And Demo Cleanup

0.175 may clarify documentation around:

- stable vs heap vs journaled tradeoffs;
- journaled memory ID roles;
- fold/recovery behavior;
- why journaled is not the same as `storage(stable(...))`;
- why `storage(heap())` remains volatile.

Demo canister changes are allowed only if they improve validation or
documentation. Do not mix demo UX work into journal protocol hardening unless
the test surface needs it.

Documentation and demo updates should not precede the technical hardening
evidence in H2-H17 and H19-H22.

## Patch Slices

### 0.175.0 — Audit Baseline And Plan

- Land this design.
- Run a read-only 0.174 journaled closeout audit.
- Inventory duplicated overlay/recovery/fold/source-guard flows.
- Identify allowed snapshot helpers and hot-path forbidden helpers.
- No runtime behavior change.

### 0.175.1 — Ordered Overlay DRY Boundary

- Centralize data/index canonical-plus-live overlay traversal.
- Preserve early stop, ordering, bounds, live override, and tombstone semantics.
- Remove duplicated data/index merge loops.
- Prove range scans do not materialize full snapshots.
- Add mixed canonical/live/tombstone early-stop tests for data and index stores.
- Prove empty-side fast paths are not the only streaming path.
- Apply the same ordered overlay helper to schema-store raw snapshot traversal
  when accepted-schema lookup hits query hot paths.
- Prove descending traversal interleaves canonical/live versions correctly and
  gives live duplicate keys precedence.

### 0.175.2 — Snapshot And Hot-Path Guard Cleanup

- Remove or isolate remaining query-facing snapshot materialization.
- Add guards for read-only queries not calling recovery/fold/diagnostics.
- Add source guards for generic snapshot/materialization names in read paths.
- Keep fold-only materialization explicitly named.
- Prove query cache hits avoid pre-cache schema, diagnostics, and capability
  reconstruction.
- Make cold SQL query calls avoid cache-write dependence by carrying accepted
  schema, schema fingerprint, and accepted authority from compile through plan
  lookup.
- Prove SQL cache-key construction reads the accepted runtime schema directly
  when present and does not force generated-model reconciliation.
- Preserve stable and heap behavior.

### 0.175.3 — Recovery/Fold Edge Hardening

- Add focused fail-closed tests for marker-bound journal repair, mismatch,
  duplicate, sequence-gap, watermark, and fold cleanup behavior.
- Prefer test-only protocol/store seams over broad production failure-injection
  infrastructure.
- Record explicit gaps for non-injectable interruption points.

### 0.175.4 — Source-Audit And Diagnostics Authority

- Tighten source-audit guards for concrete journaled branching.
- Prove diagnostics remain projection-only.
- Prove capabilities are derived from accepted registry/store metadata, not
  diagnostic DTOs or generated model fallback.

### 0.175.5 — Closeout

- Close proof matrix H1-H24.
- Record stable/heap regression evidence.
- Record journaled hot-path performance evidence.
- Include before/after supporting evidence for:
  `select name from character order by id asc limit 1`.
- Record codec/no-ABI-change evidence.
- Decide whether any remaining hardening belongs in 0.176.

## Closeout Standard

0.175 can close only if:

- H1-H24 are closed or have explicit non-blocking caveats;
- no normal journaled query path performs recovery, fold, diagnostics, or full
  overlay snapshot materialization before it can honor route bounds and visitor
  early-stop;
- no cold SQL query-call path depends on persisted query cache writes to avoid
  accepted schema reconstruction or full overlay materialization;
- source guards prove concrete backend policy branching is contained;
- stable and heap regressions pass;
- codec, commit-marker, memory ID, and stable-key compatibility are either
  untouched or directly tested if touched;
- any remaining recovery/fold interruption gaps are documented as future work.
