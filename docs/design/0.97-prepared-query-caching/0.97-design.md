# 0.97 Prepared Query Caching & Execution Efficiency

## Thesis

`0.97` should introduce **prepared query caching and execution reuse**, reducing the fixed cost of query execution by avoiding repeated parse, lowering, validation, and planning work for equivalent queries.

> **0.97 shifts the system from “compile every time” to “compile once, execute many”, leveraging the now-stable semantic and execution model established in 0.96.**

This slice focuses on **performance and execution efficiency**, not SQL feature expansion.

---

## Problem statement

After `0.96`, the system has:

* unified semantic model (`Expr`)
* unified compiled execution model
* no interpreter paths
* stable planner + executor contracts

However, each query still pays the full cost of:

```text
parse → lower → validate → plan → compile → execute
```

Even for identical queries.

This creates a **high fixed-cost floor**:

* simple queries still incur heavy setup cost
* repeated queries do not benefit from reuse
* overall system feels slower than necessary despite efficient execution

---

## Design goal

Introduce **prepared query caching** so that:

* repeated queries reuse compiled execution plans
* parsing and planning are skipped when possible
* execution becomes the dominant cost, not compilation

The goal is:

> **reuse compiled execution artifacts safely and consistently across identical query intents.**

---

## Non-goals

This slice does **not** aim to:

* add new SQL features
* introduce JOIN, subqueries, or DISTINCT improvements
* redesign the planner or execution engine
* introduce a generic VM or execution IR
* optimize every allocation or micro-path
* implement cost-based optimization

This is a **reuse and caching slice**, not a feature or architecture slice.

---

## Core design

## 1. Prepared execution plan caching

### Concept

Cache the result of full query compilation:

```text
(query identity, schema fingerprint) → SharedPreparedExecutionPlan
```

Where:

* **query identity** = normalized SQL or structural query model
* **schema fingerprint** = version/hash of schema relevant to the query

---

## 2. Cache key

The cache key must include:

### Required

* normalized SQL string OR structural query fingerprint
* schema fingerprint

### Optional (if applicable)

* query mode (read/write)
* parameter shape (future)

### Explicit rule

> Two queries must only share a cached plan if they are semantically identical under the current schema.

---

## 3. Cache value

Cache stores:

* `SharedPreparedExecutionPlan`

This already encapsulates:

* compiled expression programs
* resolved access paths
* projection execution
* ordering logic

### Important

Do NOT cache:

* raw parser output
* partially lowered state

Only cache **fully prepared execution plans**.

---

## 4. Compile vs execute split

Internally formalize:

```text
compile(query) → SharedPreparedExecutionPlan
execute(plan) → result
```

Even if not exposed publicly yet.

### Explicit rule

> Execution must not trigger re-compilation of the query.

---

## 5. Cache lookup flow

### Before

```text
parse → lower → validate → plan → compile → execute
```

### After

```text
lookup cache
  → hit: execute cached plan
  → miss: compile → store → execute
```

---

## 6. Cache invalidation

Cache must be invalidated when:

* schema changes
* index changes affecting plan shape

### Strategy

Use schema fingerprint:

* if fingerprint mismatches → cache miss

### Explicit rule

> Cached plans must never be reused across incompatible schema states.

---

## 7. Expression reuse (implicit benefit)

Because compiled expressions are already part of the prepared plan:

* WHERE → PredicateProgram reused
* projection → ScalarProjectionExpr reused
* ORDER BY → reused compiled expressions

No additional work required.

---

## 8. Execution improvements (bounded)

### 8.1 LIMIT short-circuiting

Where possible:

* stop execution once LIMIT reached
* avoid scanning entire dataset

### 8.2 ORDER BY + LIMIT

Allow partial sorting where applicable:

```text
ORDER BY ... LIMIT N → top-N optimization
```

Only if trivial to integrate with current model.

---

## 9. Memory and allocation (light pass)

Perform small improvements:

* avoid unnecessary cloning in execution hot paths
* reuse buffers where obvious

Do not perform deep allocation redesign.

---

# Part II — Contracts

## Contract 1 — Cache correctness

> Cached plans must be semantically equivalent to freshly compiled plans.

---

## Contract 2 — No runtime compilation

> Execution must never re-run lowering or planning logic.

---

## Contract 3 — Schema safety

> Cached plans must be invalidated when schema changes affect correctness.

---

## Contract 4 — Transparent behavior

> Caching must not change query results, ordering, or error behavior.

---

# Part III — Acceptance criteria

## Functional

* repeated queries return identical results whether cached or not
* cache hit avoids parse/lower/plan steps
* schema changes invalidate cache correctly

---

## Performance

* repeated identical queries show measurable reduction in instruction count
* simple queries fall below previous fixed-cost floor

---

## Architectural

* `SharedPreparedExecutionPlan` is the cache value
* no new execution abstraction introduced
* existing compiled expression model reused

---

## Testing

Add tests for:

* cache hit vs miss behavior
* schema change invalidation
* identical SQL producing same cache key
* slightly different SQL producing different cache entries

# Part V — Risks

## Risk 1 — incorrect cache key

Mitigation:

* rely on structural fingerprinting where possible

---

## Risk 2 — stale plans

Mitigation:

* schema fingerprint gating

---

## Risk 3 — premature optimization

Mitigation:

* keep scope limited to plan reuse
* avoid deep executor changes

---

# Part VI — Out of scope

* JOIN
* DISTINCT redesign
* subqueries
* window functions
* generic execution VM
* cost-based planning

---

# Part VII — Follow-on

After `0.97`, the system will have:

* unified semantics (0.96)
* unified execution (0.96)
* efficient reuse of compiled plans (0.97)

This enables:

* feature expansion (DISTINCT, functions)
* larger SQL features (JOIN)
* deeper performance tuning later

---

## Thesis sentence

> **0.97 introduces prepared query caching and execution reuse, eliminating repeated compilation cost and shifting query performance from compile-bound to execution-bound.**
