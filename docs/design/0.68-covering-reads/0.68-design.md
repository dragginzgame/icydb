# First-Class Covering Reads for Scalar SQL and Fluent Loads

## Status

Draft

## Goal

Introduce a planner-owned, first-class covering-read route for scalar reads so eligible queries can return results directly from index components without resolving `DataKey` and reading row storage.

The design extends the existing planner-owned covering metadata used by aggregate and field-projection terminals into a generalized execution contract for load/query surfaces.

## Motivation

At the current one-row filtered-select baseline, further meaningful wins are unlikely to come from small materialization cleanups alone. The remaining large fixed costs come from work the engine still performs even when an index already contains the required data:

* resolve index entries to `DataKey`
* read row storage
* decode row payloads into slot/value form
* materialize structural rows only to project fields already present in the index

That work is unnecessary for a subset of scalar reads where:

* the predicate is satisfied by the chosen access path
* the requested projection is fully coverable from index fields and/or constants
* the required order is already satisfied by traversal order or a planner-known reorder contract
* the query does not require row-only semantics

The repo already has partial covering/index-only support for COUNT/EXISTS and field-target aggregate projection. This proposal generalizes that capability into a planner-selected load/read route.

## Existing Foundation

The current engine already contains several building blocks that should be reused rather than bypassed:

1. **Planner-owned covering metadata exists today for aggregate projection.**
   `CoveringProjectionContext` and `CoveringProjectionOrder` are planner-owned and derived in `db/query/plan/covering.rs`.

2. **Index-covering existing-row terminal eligibility is already planner-owned.**
   COUNT/EXISTS fast-path eligibility was moved out of terminal-local heuristics and into planner-owned logic.

3. **Access-window contracts are already planner-owned.**
   `ScalarAccessWindowPlan` and `AccessWindow` centralize offset/limit/fetch interpretation for pushdown and bounded access decisions.

4. **The audit still identifies general covering projection as only partial.**
   Current support is concentrated in aggregate/field terminals rather than generalized scalar read execution.

This means the missing capability is not “invent covering from scratch.” The missing capability is: make general covering reads a first-class planner/runtime contract instead of a set of terminal-local heuristics.

## Non-Goals

This design does not attempt to:

* add full SQL projection parity for arbitrary expressions
* add cost-based plan selection
* widen query semantics by silently changing stale-row or missing-row behavior
* replace all row-materialized scalar reads
* solve streaming-access handoff in the same slice
* introduce broad index-only support for grouped execution

## User-Facing Outcome

Eligible reads such as the following should avoid row storage entirely:

* `SELECT name FROM User WHERE name = 'alice' ORDER BY id LIMIT 1`
* `SELECT name FROM User ORDER BY name ASC LIMIT 1`
* fluent single-field loads whose projected field is present in the chosen index
* selected multi-field reads where all requested output fields are present in index components plus the primary key

Ineligible shapes continue to use the existing row-materialized path with unchanged semantics.

## Design Principles

### 1. Planner owns eligibility

Whether a read is covering must be decided in planner-owned code, not rediscovered in executor terminals.

### 2. Covering is a route shape, not a shortcut

The executor should not “notice” late that a query happens to be coverable. The planner should emit an explicit execution contract that the executor can honor directly.

### 3. Semantics must stay explicit

If a query needs row storage to preserve current correctness or stale-row parity, it must not be admitted into the covering route.

### 4. Existing order/window contracts should be reused

Covering reads should consume existing `CoveringProjectionOrder`, `CoveringProjectionContext`, and `AccessWindow` semantics instead of inventing parallel runtime logic.

## Proposed Design

### A. Add a planner-owned covering read contract

Introduce a new planner-owned contract in `db/query/plan/covering.rs` for scalar read projection, conceptually:

* `CoveringReadPlan`
* `CoveringReadFieldSource`
* `CoveringReadOrderContract`

Suggested shape:

```rust
pub(in crate::db) enum CoveringReadFieldSource {
    IndexComponent { component_index: usize },
    PrimaryKey,
    Constant(Value),
}

pub(in crate::db) struct CoveringReadField {
    output_slot: PlannedFieldSlot,
    source: CoveringReadFieldSource,
}

pub(in crate::db) enum CoveringReadOrderContract {
    TraversalOrder(Direction),
    PrimaryKeyOrder(Direction),
}

pub(in crate::db) struct CoveringReadPlan {
    fields: Vec<CoveringReadField>,
    prefix_len: usize,
    order_contract: CoveringReadOrderContract,
}
```

This should be derived only for access shapes that are already index-backed and whose requested projection can be satisfied without row access.

### B. Planner derives covering-read eligibility during scalar planning

Add a planner entrypoint that derives a covering-read plan from:

* `AccessPlan`
* selected projection layout
* scalar order contract
* page/window contract
* residual predicate compatibility
* missing-row/stale-row semantics

Suggested helper:

```rust
pub(in crate::db) fn covering_read_plan<K>(
    plan: &AccessPlannedQuery<K>,
    projection: &PlannedProjectionLayout,
    primary_key_name: &'static str,
    strict_predicate_compatible: bool,
) -> Option<CoveringReadPlan>
```

This function should return `None` unless all requested output fields are satisfiable from index state or constants and the order contract is safe.

### C. Extend execution routing with a distinct covering-read route

Add a route/terminal selection path that makes the covering-read shape explicit.

Conceptually:

* `ExecutionNodeType::CoveringRead`
* `LoadTerminalFastPathContract::CoveringRead`
* route descriptor / EXPLAIN support for `covering_projection=true` on scalar reads

This should sit alongside existing row-materialized load execution rather than hide inside it.

### D. Add a covering-read terminal runtime

Add runtime support that consumes index traversal output directly and materializes projected values without row fetch.

Main responsibilities:

* consume index entry stream
* decode only index components already present in traversal output
* project requested values into scalar/structural output rows
* apply page/window semantics through existing `AccessWindow`
* preserve output order according to planner-provided order contract

The runtime should not reinterpret covering eligibility.

### E. Preserve a strict fallback boundary

If any part of the covering contract becomes unsafe or unsupported, planning must fall back to the current row-materialized path.

Examples of fallback triggers:

* projection references a non-indexed field
* order shape is unsupported for index traversal output
* predicate requires row evaluation not covered by strict index compatibility
* semantics require row presence or row payload inspection
* projection contains unsupported computed expressions

## Eligibility Rules

The first slice should be intentionally narrow.

### Eligible in phase 1

* scalar reads only
* index-prefix and index-range access paths
* unordered reads
* reads ordered by:

  * suffix index order already recognized by planner-owned covering contracts
  * primary key order already recognized by planner-owned covering contracts
* direct field projection where every field is either:

  * primary key
  * indexed component
  * planner-known constant under fully bound prefix semantics
* `LIMIT` / `OFFSET` through existing access-window contracts
* strict residual predicates only when current planner/runtime contracts already mark them compatible

### Explicitly ineligible in phase 1

* grouped queries
* arbitrary computed projections
* projections requiring row-only fields
* joins
* DISTINCT over generalized scalar covering reads
* queries whose ordering requires executor-side re-sort beyond current planner-owned contracts
* cases where stale-row parity depends on row fetch

## Semantics and Correctness

The main correctness hazard is accidentally changing “existing row” semantics.

Current covering COUNT/EXISTS logic is conservative: it only stays index-only when the planner/runtime contracts say the shape is safe. General covering reads must keep that same posture.

The planner should only produce `CoveringReadPlan` when:

1. traversal values are sufficient for the requested output
2. row fetch is not required to preserve accepted semantics
3. output ordering is fully determined by the planner contract
4. page/window behavior is preserved under the existing `AccessWindow` rules

If any of those are uncertain, the query remains on the current path.

## Explainability

Add explicit EXPLAIN/reporting descriptors for scalar covering reads.

Suggested descriptor additions:

* execution node type: `CoveringRead`
* `covering_projection=true`
* covering order source:

  * `index_order`
  * `primary_key_order`
* projection field sources:

  * `index_component`
  * `primary_key`
  * `constant`

This keeps the route explainable and testable and follows the project’s move toward planner-owned explainable contracts.

## Rollout Plan

### Slice 1: planner contract only

* add `CoveringReadPlan` and derivation helpers
* keep executor fallback-only initially
* add unit tests proving eligible vs ineligible shape derivation

### Slice 2: route selection and explain

* add explicit route/terminal fast-path contract
* add EXPLAIN descriptors
* add route-matrix coverage

### Slice 3: runtime for single-field scalar covering reads

* support single-field projection from index-backed access
* preserve existing fallback for everything else
* add perf harness cases for one-row filtered selects and ordered top-1 reads

### Slice 4: multi-field covering reads

* support multi-field projections when every field is coverable
* keep expression/computed fields out of scope

### Slice 5: constant-covering and selective widening

* admit safe constant-covering projection cases
* only widen once perf and complexity deltas are understood

## Validation

### Correctness tests

* planner derivation matrix for eligible/ineligible shapes
* route-matrix tests proving explicit covering-read selection
* SQL/fluent parity tests for identical results
* order stability tests for index-order vs primary-key-order contracts
* page/window tests for offset/limit correctness
* stale-row parity tests against the row-materialized fallback

### Explain tests

* EXPLAIN reports `covering_projection=true` only for eligible scalar shapes
* EXPLAIN reports correct order source and field-source metadata

### Performance tests

Add perf cases that directly target the current pain points:

* one-row equality lookup on indexed field, projected indexed field
* one-row equality lookup on indexed field, projected PK + indexed field
* ordered secondary-index top-1 scalar read
* ordered PK top-1 scalar read where projection is coverable
* compare covering-read route vs current row-materialized baseline

## Risks

### 1. Semantic drift

If covering-read eligibility is too permissive, scalar reads may stop matching current row-materialized behavior.

### 2. Contract duplication

If executor/runtime re-derives order or projection contracts, the project will recreate the same planner/executor duplication that 0.46.2 intentionally removed.

### 3. Complexity spread

If this is implemented as many terminal-local fast paths, it will increase route/runtime fanout instead of reducing it.

### 4. Premature widening

Trying to include grouped, computed, or row-sensitive shapes in the first slice will likely collapse the clarity of the design.

## Why this is the right 0.x+1 target

This work is big enough to justify a dedicated minor-version design because it changes a real planner-to-runtime contract rather than performing local cleanup. It also aligns directly with the repo’s existing direction:

* planner-owned covering eligibility already exists
* planner-owned access-window contracts already exist
* executor-side covering heuristics have already been pulled back toward planner ownership
* the remaining gap is general scalar covering reads

That makes this a continuation of the current architecture, not a side-path.

## Open Questions

1. Should phase 1 admit multi-field projections immediately, or require single-field-only first to minimize route/runtime fanout?
2. Should primary-key reorder shapes be admitted in phase 1, or should the first slice only admit pure traversal-order reads?
3. Can the current physical access layer expose decoded index components cleanly enough, or does it need an intermediate representation first?
4. Do we want one dedicated covering-read terminal, or one more general terminal family parameterized by row-vs-covering materialization input?

## Recommendation

Start with a deliberately narrow first slice:

* planner-owned `CoveringReadPlan`
* explicit scalar covering-read route
* runtime for single-field index-covered scalar reads
* EXPLAIN + route/perf coverage

If that lands cleanly and shows real movement on the ~418k/~416k cohort, then widen to multi-field coverable projections in the next slice.
