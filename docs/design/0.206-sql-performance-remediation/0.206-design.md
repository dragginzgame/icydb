# 0.206 SQL Performance Remediation

Status: active; 0.205 closeout accepted and Slice 1 retained-candidate
measurement pending as of 2026-07-19

## Decision Summary

0.204 established reproducible SQL performance discovery, confirmation, scale,
attribution, and regression evidence. It did not turn the expensive current
query families it found into a production-remediation program. Its final
same-source ordinary comparison proved that the selected baseline reproduced;
it did not prove that the baseline was acceptably fast.

0.205 is the first focused optimization consumer of that evidence. It removes
full grouped-state retention for eligible ordered grouping. 0.206 follows it by
reranking the complete post-0.205 profile and remediating the largest remaining
avoidable query-cost family, plus at most one directly coupled family, before
schema, row-format, bulk-mutation, constraint, integrity, and identity work
expands the runtime surface.

This is not a cost-based optimizer or an open-ended executor rewrite. The line
keeps deterministic planning and current SQL semantics. Its candidate audit
covers general owner-level work that the accepted evidence shows is repeated
unnecessarily:

- materialized ordering and bounded pages that still decode or retain more than
  the final page requires;
- residual-filter ordered scans whose candidate work remains disproportionate
  to the returned window;
- scalar aggregate paths whose per-input-row cost is materially above the
  maintained streaming alternatives; and
- large membership predicates whose cold compile/planner work dominates their
  execution cost.

0.206 cannot close by promoting more scenario IDs into regression sentinels.
It must publish a checked-in human review, assign every deterministically
selected leading family an owner and disposition, freeze exactly one primary
`OptimizeNow` family plus at most one directly coupled family, land every
selected improvement, record exact before/after evidence, and select a reviewed
post-remediation baseline.

## Evidence That Opens The Line

The accepted 0.204 ordinary performance workflow is run `29657817214`. Its
Tier D current artifact records:

- 1,777 complete P1 scenarios;
- 383 five-sample P2 candidates;
- 45 scale totals;
- 141 normalized scale costs;
- 30 adjacent-cardinality slopes; and
- 56,462 comparison deltas.

All 56,462 ordinary comparison deltas, all scale totals, all normalized costs,
and all slopes were numerically unchanged because the ordinary run compared the
newly selected current-source baseline with the same production behavior. That
is valid reproducibility evidence, but it supplies no production speedup.

The accepted artifact exposes these leading scale families:

The P1 report also shows:

- the highest 512-row materialized-order cases cost about 23.2 through 24.4
  million instructions and perform 512 data-store gets even for page limits 1,
  3, or 10;
- the 250-member sparse-`IN` page scenario costs 15,359,856 instructions,
  including 7,466,903 planner instructions; and
- confirmed warm cache state materially reduces several cold costs, including
  the sparse-`IN` page scenario from 15,359,856 to 6,050,110 instructions and
  a small indexed count from 1,242,252 to 72,323 instructions.

These numbers rank IC local instructions, not host latency. Peak heap,
allocator traffic, and stable-memory byte volume remained explicitly
unmeasured in 0.204. 0.206 must not describe an instruction improvement as a
memory improvement without adding reliable typed evidence.

0.205 changes the maintained grouped profile and production implementation.
The opening 0.206 report therefore consumes the exact accepted 0.205 closeout
bundle, uses the 0.204 artifact only as historical before-context, and follows
the 0.204 bridge procedure whenever profile or environment identity changes.

## Problem Statement

### 1. Ranking Was Mistaken For Remediation

0.204 selected the top 20 observations for every raw and normalized metric.
Every maintained metric therefore has ranked candidates even when its absolute
cost is unavoidable. Calling all recurring candidates hotspots did not answer:

- which cost is semantically necessary;
- which phase owns avoidable work;
- which general implementation change should remove that work;
- which query needs a better schema/index rather than executor code; or
- which candidate is only high because it returns or decodes more data.

### 2. The Human Engineering Result Is Not Durable

The complete P1 Markdown report and large P1/P2/scale JSON bundles are retained
as GitHub Actions artifacts. The repository records counts and hashes but has
no 0.204 performance investigation report under `docs/reports/`. A 90-day
artifact must not be the only place where the optimization priorities can be
understood.

### 3. Regression Gates Do Not Judge The Initial Baseline

The checked-in thresholds correctly reject future regressions. They do not
declare the accepted initial cost good. Comparing one source revision with
itself can produce a perfect verdict while bounded queries still perform tens
of millions of instructions.

### 4. Query Shape And Engine Waste Need Separate Dispositions

An unindexed expression order must inspect all eligible rows. A database cannot
provide ordered early stop without a compatible access path. That does not
authorize unnecessary field decoding, duplicated row construction, unbounded
candidate retention, repeated plan work, or a query-specific special case.

0.206 must distinguish:

- **contract-required work**: required by current semantics and available
  indexes;
- **schema-dependent work**: removable only when the application declares a
  compatible index or changes its query;
- **avoidable engine work**: repeated decode, materialization, planning,
  copying, or traversal not required for the result; and
- **unmeasured work**: a suspected cost without sufficient typed evidence.

## Goals

0.206 must:

1. preserve the accepted 0.204 and 0.205 artifacts through a checked-in concise
   investigation report and structured findings ledger;
2. rerank the exact post-0.205 P1, P2, and scale evidence;
3. derive the leading set through the deterministic rule below;
4. classify each leading family from typed route, phase, store-operation,
   row-count, and cache facts before owner-module source inspection influences
   the conclusion;
5. assign every leading family one explicit owner and disposition;
6. freeze exactly one primary `OptimizeNow` family and at most one directly
   coupled `OptimizeNow` family;
7. optimize the selected general execution owners rather than scenario IDs;
8. retain deterministic rule-based planning with no cost model or plan search;
9. preserve SQL results, ordering, cursors, route safety, cache semantics, and
   accepted-schema authority;
10. retain complete scan/work budgets and fail-closed resource behavior;
11. measure cold and eligible proven-warm modes separately;
12. record exact total, phase, normalized, slope, store-operation, output, and
    raw non-gzipped wasm deltas;
13. fail the line when a selected `OptimizeNow` finding remains unimplemented
    or misses its predeclared eliminated-work budget without an explicit design
    revision; and
14. select a reviewed post-remediation baseline only after all required
    correctness and performance evidence is clean.

## Non-Goals

0.206 does not:

- add a cost-based optimizer, statistics catalog, cardinality estimator, or
  plan enumeration;
- add automatic indexes, query hints, route forcing, or an old/new runtime
  fallback for the selected owner;
- change SQL result, null, ordering, cursor, or error semantics;
- treat adding an index to a test fixture as an engine optimization;
- special-case a benchmark scenario ID, SQL string, entity name, or literal;
- optimize the grouped ordered fold owned by 0.205 a second time;
- require one implementation rewrite from every candidate family below;
- add peak-memory claims without typed peak-state evidence;
- weaken read, group, DISTINCT, output, instruction, or artifact bounds;
- auto-bless a faster or slower baseline in CI;
- delete expensive scenarios after improving them; or
- close with only measurement, sentinel promotion, or deferred follow-up prose.

## Hard-Cut Implementation Rule

0.206 replaces the selected production owner directly. Once the replacement is
proved, the superseded algorithm, dispatch branch, helper surface, and tests
whose only purpose was the old path are deleted in the same slice. Before 1.0,
the before-state lives in immutable evidence, not retained production code.

There is no feature flag, tuning toggle, dual execution, shadow comparison,
compatibility wrapper, legacy alias, or runtime fallback to the superseded
strategy. Route safety may still choose among maintained semantic strategies
that already have distinct contracts; it may not select old-versus-new 0.206
implementations.

0.206 does not change a persisted row, cursor, public DTO, or canonical
fingerprint format. If the selected optimization requires such a format change,
the design is revised and the format moves directly to one current form with a
typed failure or cache miss for the old form. No compatibility decoder, dual
fingerprint, or old-cache lookup is added.

Measurement additions stay internal and minimal. Use existing typed
attribution first; add only counters required by a selected budget. A
slice-only counter is removed at closeout unless a maintained regression guard
still consumes it. 0.206 does not widen a public metrics DTO or persisted
format merely to prove the optimization.

## Evidence And Disposition Authority

The machine-readable P1/P2/scale artifacts remain measurement authority. The
checked-in investigation report is the engineering decision record. It must
live at:

`docs/reports/investigations/2026/07/19/sql-performance-remediation/0.206/`

The investigation owns:

- `report.md`: ranked families, phase localization, before/after summaries,
  implementation decisions, remaining risks, and one immutable run-reference
  table whose rows record role, run ID, source revision, environment/profile
  identity, and artifact hash;
- `findings.json`: stable finding IDs, scenario families, owner modules,
  evidence references, selection state, budgets, and dispositions; and
- only the smallest unique supporting artifacts needed to preserve conclusions.

The raw multi-megabyte P1/P2 bundles remain reproducible workflow artifacts and
are not copied into report history merely for retention.

The investigation report and findings ledger may evolve through the line. A
referenced benchmark run does not: its identity, role, revision, environment,
and artifact hash are append-only facts. An opening run never owns results from
a later implementation or final-comparison run.

The report and ledger are deliberately simple checked-in evidence, not a new
reporting framework. Slice 1 may use one-off audit extraction over the existing
0.204 artifacts; it does not add a generic ranking service, findings database,
or public evidence API.

## Leading-Set And Selection Authority

The term **leading family** means the deduplicated union of the top five typed
families in the accepted post-0.205 opening evidence by each of:

1. total instructions at the maintained 2,048-row scale point;
2. 256-to-2,048 marginal instructions per input row;
3. the maintained normalized instruction cost; and
4. absolute cold compile-plus-planner instructions.

Observations are grouped by stable typed scenario-family ID before ranking. A
family's score is its largest admitted observation for that metric, and equal
scores are ordered by family ID. A missing or inapplicable metric does not
silently score as zero. The report records its absence, and insufficient typed
evidence produces `NeedsTypedMeasurement` where a conclusion depends on it.

The six families in the opening table are historical context, not an expanding
scope definition. Source inspection cannot add a family to the leading set.
The leading set freezes with the comparable opening run and does not expand
merely because later remediation changes the ranking. A newly introduced
regression still fails the ordinary maintained regression policy; it does not
silently become another 0.206 implementation family.

Slice 1 selects exactly one primary family as `OptimizeNow`. A selectable
family's avoidable-cost score is its largest admitted typed avoidable-phase
instruction count in the opening evidence; the greatest score wins, with ties
ordered by family ID. It may select at most one additional directly coupled
family. A family is directly coupled only when it shares the same production
owner and invariant and the same bounded implementation change necessarily
removes measured work from both. Sharing an executor area, benchmark, or
convenient patch is not sufficient.

Both selections and their budgets freeze at the end of Slice 1. Every selected
family must land before closeout. If no leading family has a safe general
optimization supported by typed evidence, 0.206 is revised rather than closed
without a production improvement. Every unselected leading family still
receives a complete disposition; an avoidable but unselected family requires a
concrete `NamedFollowUp`, not relabeling as contract-required work.

Each finding has exactly one current disposition:

- `OptimizeNow`: one of the one or two families selected and frozen by Slice 1;
- `ContractRequired`: the work is necessary under current semantics and the
  report identifies the proof;
- `SchemaDependent`: a compatible declared index or query change is required;
- `OwnedBy205`: a historical pre-0.205 grouped observation that no longer
  reproduces as a leading family in the accepted post-0.205 opening run;
- `NeedsTypedMeasurement`: no optimization claim is allowed until the missing
  counter exists; or
- `NamedFollowUp`: work is outside the bounded owner scope and has a concrete
  accepted follow-up design, owner, and trigger.

`NamedFollowUp` is not a generic backlog bucket. An owner-less or trigger-less
entry remains unresolved. A regression sentinel is evidence membership, not a
disposition.

`OwnedBy205` is not valid for grouped work that remains leading after 0.205.
Such a family receives a fresh 0.206 disposition and is eligible for primary or
directly coupled selection under the same rules as every other family.

## Selected-Finding Budgets

Before selection freezes, every selected family has one machine-checkable
eliminated-work budget containing:

- the exact typed counter or metric that proves the targeted work;
- its opening value and required target;
- the scenario IDs, family ID, and cardinalities that exercise it;
- cold, proven-warm, or both measurement modes;
- the allowed sampling variance;
- total and relevant phase instruction guardrails; and
- semantic, route, cache, resource, and ordinary regression guards.

The primary target is structural, such as maximum retained candidates, typed
slots materialized per structurally valid row, row constructions per qualifying
row, canonicalization passes per membership item, or aggregate input
materializations. An instruction percentage alone is not an eliminated-work
budget. If no typed counter can prove the claimed removal at selection freeze,
the finding remains `NeedsTypedMeasurement` and cannot be selected as
`OptimizeNow`.

Any retention-changing selection must expose `peak_retained_candidates` and
budget it explicitly. Byte or heap improvements require corresponding typed
byte or peak-heap evidence; instruction reductions do not imply either.

## Candidate Finding Families

The families below define audited owner boundaries and eligible Slice 1
selections. They do not each require a production rewrite in 0.206.

### Ordered Materialization And Bounded Pages

Audit the non-grouped order/window owners for:

- materializing typed values for fields not needed for filter, order, cursor,
  or projection after the complete row has been structurally validated;
- constructing full rows before bounded top-k/page admission;
- retaining more candidate state than `offset + limit + lookahead` requires on
  an initial offset page, or more than `limit + lookahead` after a cursor
  boundary has been safely applied;
- repeating key/row conversion across scan, order, and response stages; and
- failing to consume an already-proven physical order/limit stop.

Full scans remain legal where no compatible order exists. The target is lower
per-row and retained-candidate cost, not a false early stop.

Selective materialization must preserve corrupt-row failure behavior. Every
encountered row still receives bounded structural validation of its complete
current-form accepted envelope, slot count, and every slot's canonical encoding
in the same deterministic order. Only typed value construction for unreferenced
slots may be skipped. This must remain one current-form slot walker and decoder
owner, not a new validator beside the old decoder. If the current failure
classification and facts cannot be preserved without fully materializing a
slot, that slot remains materialized and the suspected saving is not eligible.

Any bounded ordering heap uses the complete final deterministic ordering key,
including requested direction, null-order semantics, all declared order terms,
and the canonical primary-key tie-breaker.

### Residual Filter And Ordered Scan

Audit candidate/qualifying-row accounting, residual expression evaluation,
selective row decode, and stop proof. Any new stop rule must be derived from
existing deterministic route proof and remain correct for cursors, lookahead,
offsets, null ordering, and sparse matches. The input stream must already be
proven to be in final total output order, with no later materialization,
projection, or ordering stage capable of reordering it. An ordered access path
alone is not sufficient.

### Scalar Aggregate Input And Reducer Work

Separate unavoidable row/index traversal from row materialization, expression
evaluation, aggregate input decode, reducer fold, and finalization. Reuse a
covering or cardinality owner only when the accepted plan already proves it;
there is no runtime route upgrade. Optimizations preserve the existing input
fold order whenever checked numeric overflow, floating-point accumulation, or
another admitted aggregate can observe reordering.

### Large Membership Compile And Planning

Audit canonicalization, hashing, range extraction, capability evaluation,
planner schema resolution, cache-key construction, and repeated allocation for
large `IN`/`NOT IN` sets. Cold work may be reduced without weakening canonical
query identity. Warm-cache wins do not excuse avoidable cold work because every
distinct admitted statement must compile at least once.

0.206 preserves the exact existing canonical fingerprint bytes, not merely an
informal notion of equivalent cache identity. Opening oracles freeze member
order, duplicates, `NULL` members, parameter positions, and the distinction
between `IN` and `NOT IN`. An implementation may reuse already-computed
canonical facts but may not deduplicate, reorder, or renumber inputs when doing
so changes those bytes or `NOT IN` three-valued semantics. A fingerprint-format
change requires a separate design.

### Response And Blob Output

Response finalization and blob hex/output bytes remain ranked context. Optimize
only duplicated conversion or copying; the instructions required to return the
declared bytes are contract-required output work.

Duplicated response or blob work may be selected as the primary or directly
coupled family when it enters the deterministic leading set and satisfies the
same typed-evidence and eliminated-work-budget rules.

## Implementation Slices

### Slice 1: Durable Review And Post-0.205 Rerank

- capture the concise investigation report and structured findings;
- resolve investigation, run, artifact, and profile identity and any required
  profile bridges;
- rerank totals, normalized costs, slopes, cold/warm savings, and route classes;
- freeze the deterministic leading set and its complete findings ledger;
- classify each leading family from typed route/runtime evidence before source
  inspection, then trace it to its owner module;
- assign every leading family a complete disposition;
- select exactly one primary `OptimizeNow` family and at most one directly
  coupled family; and
- predeclare the complete machine-checkable eliminated-work budget for each
  selected family.

Only typed measurement needed to make the opening evidence complete may be
added before selection freezes. It must not change query semantics or dispatch,
and it participates in the opening environment and wasm evidence. A family
still lacking proof at the freeze remains `NeedsTypedMeasurement` and is not
selectable.

### Slice 2: Primary Remediation

- implement only the frozen primary family's general owner-level change;
- meet its structural eliminated-work target and all instruction guardrails;
- preserve every declared semantic, route, cache, corruption, and resource
  guard; and
- record exact before/after typed and instruction evidence.

### Slice 3: Directly Coupled Remediation, If Selected

- implement only the frozen directly coupled family through the shared bounded
  owner/invariant change;
- meet its independent structural target and guardrails; and
- record evidence independently from the primary family so one passing budget
  cannot hide the other's miss.

If Slice 1 selects no directly coupled family, Slice 3 is absent. A later
convenient opportunity does not reopen selection.

### Slice 4: Full Rerank, Baseline, And Closeout

- rerun the complete required 0.204 correctness and performance tiers;
- publish the final ranked report and close every finding disposition;
- compare all maintained scenarios, scale totals, normalized costs, and slopes;
- select the reviewed post-remediation baseline manually;
- record raw non-gzipped wasm; include files touched and approximate line delta
  as non-gating review context, with a short qualitative explanation of why the
  implementation became simpler, stayed neutral, or became more complex; and
- update active performance guidance and 1.0 readiness.

If a selected production slice proves there is no general safe optimization at
its owner boundary, the design is revised before that slice closes. The finding
is not silently relabeled and another family is not substituted after the
Slice 1 freeze merely to make the checklist pass.

## Validation Plan

Every production slice must run focused semantic and performance validation.
Final closeout uses the complete 0.204 authority.

Focused validation is proportional to the selected owner. It covers the changed
family, its directly shared owner, and the smallest maintained guard set that
can detect route or semantic drift. A slice does not manufacture irrelevant
aggregate, membership, output, or route cases merely to fill the matrix below;
inapplicable rows are recorded as such. The final existing 0.204 run supplies
the broad cross-family check.

Semantic evidence includes:

- heap and journaled mirrors;
- indexed, unindexed, covering, hybrid, and residual routes;
- limits 1, 3, 10, and the maintained public boundary;
- offset, initial cursor, resumed cursor, and lookahead pages;
- ascending and descending order with nulls and ties;
- sparse, quarter, dense, and zero-match predicates;
- scalar aggregate families, DISTINCT, FILTER, empty input, and unchanged input
  fold order wherever reordering is observable;
- exact existing membership fingerprints for reordered-looking, duplicate,
  `NULL`, literal/parameter, `IN`, and `NOT IN` cases;
- cold misses and proven warm cache hits; and
- malformed/corrupt row failure parity, including corruption confined to an
  unreferenced slot when selective materialization is selected.

Performance evidence includes:

- exact before/current environment comparability;
- five stable P2 samples for changed and guard scenarios;
- 16, 256, and 2,048-row scale points where admitted;
- total, compile, planner, store, executor, and finalization instructions;
- typed storage operations, rows scanned/decoded/returned, and output bytes;
- the exact structural counter named by each selected budget;
- `peak_retained_candidates` for every retention-changing selection;
- normalized cost and adjacent-cardinality slopes;
- raw non-gzipped wasm bytes; and
- typed byte or peak-heap evidence for any byte or memory claim.

Files touched and approximate line delta are review context, not acceptance
criteria. The final report also gives a short qualitative explanation of the
complexity direction instead of treating a line count as a quality verdict.

A slice improvement is accepted only when its predeclared eliminated-work
budget is met and no maintained semantic, route, cache, resource, or regression
gate fails. Moving cost into an unmeasured phase does not count.

## Risks And Mitigations

### Risk: Optimizing An Intentionally Bad Schema

Mitigation: classify the required full scan separately from per-row engine
overhead. Do not add hidden indexes or route forcing.

### Risk: Benchmark-Specific Fast Paths

Mitigation: production changes are owned by typed route/plan/runtime facts.
Structural tests reject scenario IDs, entity names, and SQL text as dispatch
inputs.

### Risk: Lower Instructions Increase Peak State

Mitigation: retain existing resource caps and require
`peak_retained_candidates` in every retention-changing selected budget. No
memory claim follows from instruction evidence alone; byte or heap claims need
their own typed counters.

### Risk: The Baseline Is Rewritten To Hide A Miss

Mitigation: every slice preserves before/current artifacts. Baseline selection
occurs only after the final comparison and requires a reviewed rationale.

### Risk: Future Row-Format Work Erases The Gain

Mitigation: 0.209 must run the complete query-performance comparison after its
row-envelope hard cut. A regression is fixed there or remains a release blocker;
the 0.206 baseline is not silently replaced.

## Closeout Decision

durable, and measurably faster production result after 0.205. A reproducible
baseline, a longer sentinel inventory, or an expiring workflow artifact is not
performance remediation.
