IcyDB 0.60 Design — Remove Views

Status: Planned (0.60 slice)
Theme: Eliminate generated view types and collapse entity I/O to core entity + structural representations

1. Objective

Validate whether generated view types are a primary WASM size driver and remove them if they are not structurally necessary.

Specifically remove:

EntityView

EntityCreate

EntityUpdate

CoreAsView

CoreUpdateView

CoreCreateView

2. Problem Statement
Current cost profile

Each entity generates:

3 additional structs (View, Create, Update)

deep generic instantiations:

CoreAsView<T> per field

CoreUpdateView<T> per field

conversion impls:

From<Entity> for View

From<View> for Entity

merge logic:

per-field Option<T> patch handling

list patch types:

Vec<ListPatch<T>>

Structural issue

These are:

not part of execution kernel

not needed for schema

not required for storage

They exist purely for:

API shaping + partial update ergonomics

Key observation

The system already has:

structural schema

entity struct

sanitization

validation

👉 Views are redundant projection layers

3. Hypothesis

Removing views will:

Reduce:

monomorphisation (major)

symbol count

binary size

compile time

With minimal impact on:

execution engine

schema correctness

4. Design Decision
4.1 Remove all generated view types

Delete generation of:

complex_entity_views::ComplexEntityView
complex_entity_views::ComplexEntityCreate
complex_entity_views::ComplexEntityUpdate
4.2 Remove conversion traits

Delete:

impl CoreAsView for Entity
impl CoreUpdateView for Entity
impl CoreCreateView for Entity

impl From<Entity> for View
impl From<View> for Entity
4.3 Remove patch merge system

Delete:

fn merge(&mut self, patch: UpdateViewType)

and:

ListPatch

field-level patch application logic

5. Replacement Strategy

You have three viable replacement tiers — pick one for 0.60:

Option A (Recommended): Direct Entity API
Create
fn create(entity: Entity)
Update
fn update(id, entity: Entity)
Partial update

👉 Not supported in 0.60

Rationale

simplest

isolates WASM impact cleanly

avoids introducing new abstractions prematurely

Option B: Structural Patch (lightweight)

Introduce:

struct Patch {
    fields: Vec<(FieldId, Value)>
}

Runtime applies patch using schema.

Option C: Value-based DTO (future direction)
type Row = ValueMap
type Patch = ValuePatch

But this overlaps with 0.61 direction — defer.

6. API Surface Changes
Before
create(EntityCreate)
update(EntityUpdate)
get() -> EntityView
After (0.60)
create(Entity)
update(Entity)
get() -> Entity
7. Execution Impact
None (critical)

Execution engine already operates on:

structural plans

decoded values

entity boundaries

Views are not part of kernel path

8. Sanitization & Validation
No change

Already implemented on:

impl SanitizeAuto for Entity
impl ValidateAuto for Entity
9. Expected Wins
WASM size
Component	Expected Reduction
View structs	5–15%
CoreAsView monomorphisation	10–25%
CoreUpdateView + patch logic	10–30%
TOTAL	20–40% entity-related reduction
Compile time

significant drop in trait instantiations

fewer generics → faster builds

Code size per entity

From:

~1500–3000 LOC

To:

~600–1200 LOC

10. Risks
1. Loss of partial updates

Mitigation:

accept for 0.60

reintroduce structurally in 0.61

2. API breakage

Mitigation:

versioned change (0.60)

migration guide:

Create → Entity

Update → Entity

3. Frontend friction

Mitigation:

frontends send full objects instead of patches

acceptable short-term

11. Migration Plan
Step 1 — Remove generation

delete view module generation

Step 2 — Remove trait bounds

Search/remove:

CoreUpdateView

CoreCreateView

Step 3 — Update API layer

Replace:

create(view)

Step 4 — Delete patch system

symbol counts

top offenders

12. Measurement Plan

Before/after:

twiggy top

twiggy dominators

Focus on:

CoreAsView

13. Success Criteria

0.60 slice is successful if:

≥20% WASM reduction (entity-heavy workloads)

significant drop in mono count

no execution regressions

14. Non-Goals (important)

Do NOT in 0.60:

redesign entity macro

introduce descriptor system

rewrite execution engine

This slice is:

pure removal + measurement

15. Forward Link (0.61)

After this:

if size drops significantly → views were a major culprit

if not → macro duplication is dominant → proceed with descriptor system

16. One-line Summary

0.60 removes view types entirely to eliminate a large source of monomorphisation and measure their real impact on WASM size before deeper architectural changes.
