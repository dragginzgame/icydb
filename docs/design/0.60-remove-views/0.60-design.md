IcyDB 0.60 Design

Status: Planned

Theme: Remove Generated View Types

Supersedes:

- The current entity macro pattern that generates `View`, `Create`, and
  `Update` companion types for every entity

## 0. Summary

`0.60` removes generated entity view types from IcyDB and makes entity structs
the direct public DTO surface.

The target shape is:

- entity structs derive `CandidType` directly
- entity structs remain the persisted row type
- generated `EntityView`, `EntityCreate`, and `EntityUpdate` types disappear
- `CoreAsView`, `CoreCreateView`, and `CoreUpdateView` disappear from the
  entity contract
- patch merge infrastructure tied to generated update views disappears

This is a deliberate breaking change.

It is not an execution-engine slice.

It is an API and macro simplification slice intended to answer one concrete
question:

- how much per-entity wasm and macro complexity is caused by generated view
  surfaces rather than executor code

## 1. Problem Statement

### 1.1 Current State

Today each entity macro expansion generates more than the entity itself.

For one entity, the system currently carries:

- the entity struct
- one generated `View` struct
- one generated `Create` struct
- one generated `Update` struct
- conversion logic between entity and view
- create-from-view logic
- update merge logic
- nested patch types for list/set/map fields

This has three costs:

- more generated code per entity
- more trait instantiations per entity
- more public surface area than the runtime actually needs

### 1.2 Architectural Observation

The execution engine no longer depends on view types.

Views are now primarily an API-shaping layer for:

- canister-facing DTO ergonomics
- partial update ergonomics
- response projection ergonomics

That makes them a reasonable next size target.

### 1.3 Important Sanity Check

The draft idea is directionally correct, but there are three constraints that
must be stated explicitly:

1. Direct entity write APIs already exist.

`insert`, `replace`, `update`, and batch variants already accept `Entity`
today. `0.60` is therefore not inventing direct-entity writes; it is removing
the parallel view-based APIs and generated DTO families.

2. Removing views does **not** automatically remove serde from entities.

Persisted row encode/decode still uses entity serialization boundaries today.
`0.60` should not promise serde removal unless persistence contracts are
redesigned in a separate slice.

3. Direct entity `CandidType` is only valid if entity field types remain
directly Candid-compatible.

That is likely true for the current core primitive stack, but it is a hard
migration requirement and should be treated as part of the implementation
checklist.

## 2. Goals

### 2.1 Primary Goal

Remove generated entity `View` / `Create` / `Update` companion types and make
entity structs the only generated public DTO form.

### 2.2 Secondary Goals

- reduce per-entity generated code
- reduce monomorphization and symbol fanout tied to view conversion and patch
  logic
- simplify the macro output model
- make entity API shape easier to explain and reason about

### 2.3 Non-Goals

- execution-engine redesign
- persistence codec redesign
- mutation semantics redesign
- descriptor-system redesign
- preserving backwards compatibility for view-based APIs
- introducing a new structural patch format in the same slice

## 3. Design Decision

`0.60` takes the simple removal path.

### 3.1 Remove Generated Entity DTO Families

Stop generating:

- `EntityView`
- `EntityCreate`
- `EntityUpdate`

Stop generating their module wrappers and re-export aliases.

### 3.2 Make Entities the Direct Public DTO Surface

Entity structs should derive `CandidType` directly and become the surface type
for:

- canister-facing create/update payloads
- write responses
- entity-returning response helpers

### 3.3 Remove Entity View Traits

For entity macro output, remove:

- `CoreAsView`
- `CoreCreateView`
- `CoreUpdateView`

This also means removing the facade aliases built on top of them for entity
use:

- `View<T>`
- `Create<T>`
- `Update<T>`

### 3.4 Remove Patch-Merge Infrastructure Tied to Entity Update Views

Remove:

- generated entity `merge` implementations
- entity `UpdateViewType`
- `ListPatch` / `SetPatch` / `MapPatch` usage that exists only to support
  generated entity update views

If any patch helpers still have non-entity consumers after the slice, they
should be re-evaluated explicitly rather than left behind by accident.

## 4. Replacement Model

`0.60` should use the simplest replacement model.

### 4.1 Entity-In, Entity-Out

The intended public shape after the slice is:

```rust
create(entity: Entity)
replace(entity: Entity)
update(entity: Entity)
get(...) -> Entity
list(...) -> Vec<Entity>
```

This is already close to the current direct session/write surface.

The real change is removing the parallel `*_view` path and view-return helpers.

### 4.2 No Partial Update Surface in 0.60

Partial update support should be removed from this slice rather than replaced
immediately.

That means:

- no generated `EntityUpdate`
- no entity patch merge DTO
- no lightweight structural patch introduced just to preserve compatibility

If partial update returns later, it should return in a new structural design
rather than as a compatibility shim.

## 5. Concrete Surface Changes

### 5.1 Session and Write APIs

Remove view-based write helpers such as:

- `insert_view`
- `replace_view`
- `update_view`

Keep direct entity helpers:

- `insert`
- `replace`
- `update`
- batch variants

### 5.2 Response Helpers

Remove view-oriented response helpers such as:

- `WriteResponse::view()`
- `WriteBatchResponse::views()`
- any equivalent entity-response `views()` helpers

The remaining surface should return entities or ids directly.

### 5.3 Macro Output

Entity macro generation must stop emitting:

- companion `*_views` modules for entities
- entity view conversion impls
- entity create/update impl glue

### 5.4 Trait Graph

This slice likely requires narrowing the current trait graph, because today
`TypeKind` still depends on `AsView`.

`0.60` should treat this as first-class work, not as fallout:

- remove entity dependence on `AsView`
- remove entity dependence on `CreateView`
- remove entity dependence on `UpdateView`
- keep any remaining non-entity trait contracts only if they still serve an
  active non-entity purpose

## 6. Expected Impact

### 6.1 Execution Impact

None by design.

The executor should not depend on generated view types after `0.59`.

### 6.2 API Impact

High and intentional.

This is a breaking API simplification:

- code using generated `View`, `Create`, or `Update` types must migrate
- code using partial update merge semantics must migrate
- code using `.view()` / `.views()` helpers must migrate

### 6.3 Persistence Impact

None intended.

Entities should remain the persisted row type.

This slice should not change row encoding format unless that becomes strictly
necessary during implementation, in which case it should be split out and
treated as a separate design decision.

## 7. Risks

### 7.1 Overstating the Size Win

Removing views is likely a real wasm win, but the current draft numbers are too
speculative to treat as planning truth.

The doc should assume:

- meaningful reduction is plausible
- exact percentage is unknown until measured

Do not anchor the slice on a promised `20–40%` reduction.

### 7.2 Hidden Trait Coupling

Views are threaded through more than codegen.

They currently touch:

- trait aliases
- session helpers
- response helpers
- patch merge logic
- macro tests

If the slice is treated as “just delete generated structs,” it will stall on
trait-graph fallout.

### 7.3 Field-Type Candid Compatibility

Direct entity `CandidType` requires all field representations used in entities
to remain valid Candid surface types.

That needs explicit compile validation across:

- primitive wrappers
- collection wrappers
- relation key fields
- nested record/enum/newtype fields

### 7.4 Loss of Partial Update Ergonomics

This is a real product/API regression.

It is acceptable only if the slice is framed honestly:

- `0.60` trades ergonomics for simplification and measurement
- later structured patch design is optional follow-up work, not part of this
  slice

## 8. Migration Plan

### 8.1 Macro Removal

- stop generating entity view modules
- stop generating entity create/update companion structs
- stop generating entity view conversion impls

### 8.2 Trait Cleanup

- remove entity use of `CoreAsView`
- remove entity use of `CoreCreateView`
- remove entity use of `CoreUpdateView`
- narrow `TypeKind` and related trait bounds so entities no longer inherit view
  requirements

### 8.3 Public API Cleanup

- remove `insert_view`, `replace_view`, `update_view`
- remove response `.view()` / `.views()` helpers
- remove public aliases that only exist for generated entity DTOs

### 8.4 Patch Cleanup

- remove entity update-merge generation
- remove entity-facing patch helpers that become dead
- delete dead tests that only validate generated view/patch behavior

### 8.5 Validation

At minimum:

- `cargo test --workspace`
- `cargo clippy --workspace --all-targets -- -D warnings`
- wasm-size reruns on `minimal` and `twenty`
- Twiggy comparison before/after on the `twenty` canister

## 9. Measurement Plan

The point of the slice is not only simplification.

It is also to measure whether generated view families are still a major
per-entity wasm driver.

Measure:

- `minimal` wasm size
- `twenty` wasm size
- per-entity delta from the controlled pair
- Twiggy `top`
- Twiggy `dominators`
- Twiggy `monos`

Primary comparison questions:

- how much does per-entity wasm drop after view removal
- whether project-owned duplication shifts away from macro/view surfaces
- whether remaining duplication is now mostly serde / candid / session glue

## 10. Success Criteria

`0.60` is successful if:

- generated entity view families are fully removed
- entity structs become the direct public DTO surface
- no compatibility shims survive
- workspace validation passes
- wasm measurements show a clear per-entity size change, regardless of whether
  the win is large or modest

The most important outcome is a clean measurement, not a guaranteed percentage.

## 11. Forward Link

If `0.60` shows a large per-entity win, then view generation was a major macro
footprint contributor.

If `0.60` shows only a modest win, then the next likely source is broader macro
duplication and descriptor/codegen shape rather than entity DTO layering.

That follow-up belongs in `0.61`, not in this slice.

## 12. One-Line Summary

`0.60` removes generated entity view/create/update types, makes entity structs
the direct DTO surface, and measures how much macro and wasm overhead those
generated layers were really adding.
