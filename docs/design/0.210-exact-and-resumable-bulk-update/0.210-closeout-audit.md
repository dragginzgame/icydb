### Remediated contract answers

- A continuation cannot substitute another application operation id: the expected id is
  required and compared before target-store recovery or row access.
- A journaled mutation cannot consume the terminal sequence and strand Verify: admission
  reserves the next representable revision before marker publication.
- The trusted UPDATE lanes are now only Exact, intentional Prefix, Resumable, single-row,
  and caller-supplied atomic batch; no unbounded test exception remains.
- Engine policy drift across every currently owned resumable compatibility input changes
  the continuation identity.
- Current generated endpoints cannot expose raw resumable state without failing a
  syntax-aware semantic guard.
- The final-source raw Wasm size claim is reproducible, while hash nondeterminism is
  reported rather than hidden.

## 1. Original v0.210.1 executive decision

0.210 has a coherent architecture and most of its difficult ordinary-path behavior is
implemented correctly: Exact, Prefix, and Resumable are distinct execution contracts;
Exact freezes a completely exhausted key set before staging; Forward commits at most one
marker-owned atomic batch; deferred candidates are not checkpointed past; and Verify can
complete only after a clean full primary-key sweep at one unchanged durable revision.

It is nevertheless **not ready to close**. Two P1 defects violate explicit 0.210
contracts:

1. A continuation carries an operation id, but resume accepts no expected operation id
   and never compares the encoded value. A token can therefore be substituted between
   two otherwise identical authorized operations.
2. The durable mutation revision can become unrepresentable immediately after a
   successful commit at journal sequence `u64::MAX`. The final Forward batch can commit
   and then fail while capturing the Verify revision, leaving replay unable to complete.

Five additional closeout defects are material: the repository retains a test-only
unbounded pre-hard-cut UPDATE lane; batch-policy compatibility is a hand-maintained
constant without a drift guard; the checked closeout evidence materially overstates
boundary/failure coverage; the generated-surface guard is spelling-based and bypassable;
and the recorded Wasm evidence is not evidence for the final tagged source. Active audit
documents also retain one retired UPDATE ingress name.

No evidence of row/index corruption, false Verify completion under an ordinary revision
change, Exact-to-Prefix fallback, public raw continuation endpoints, a hidden job
registry, or a legacy token decoder was found.

## 4. Owner-readable architectural explanation

### The pre-0.210 problem

An update cap has two incompatible meanings. It can be a **safety assertion**—“the
complete target has at most N rows; reject otherwise”—or an **intentional selector**—
“mutate only the first N rows in this deterministic order.” A bounded collector cannot
distinguish those meanings on its own. If a complete-set operation stops when the
collector reaches N, it can falsely report success after mutating only a prefix of a
larger target. Raising N only moves the ambiguity and increases resource risk; it does
not prove the selector was exhausted.

Before 0.210, trusted/application update flows could route broad UPDATE meaning through
bounded machinery without an explicit completion contract. 0.210 makes the caller
choose one of three meanings. A multi-message operation cannot be one transaction:
each canister message has its own atomic commit boundary, so the long operation is a
convergence protocol built from independently atomic batches.

### Exact lane

`require_affected_at_most` is a positive assertion, not a selection `LIMIT`. The engine
probes up to the assertion plus one matching row, under a separate authoritative-key
scan ceiling. Both initial ceilings are 4,096, but they protect different resources.
The primary-only selector must exhaust before success; the complete key set is frozen,
then every after-image, sanitizer, managed field, constraint, index/relation effect,
journal payload, marker payload, receipt, and requested `RETURNING` shape is staged or
preflighted. Only then may the existing marker-owned commit boundary publish. The marker
turns subsequent apply into recoverable mechanical work. Exact success means the whole
selector was proved exhausted and the whole frozen set committed atomically. Assertion,
scan, marker, journal, response, or semantic rejection means no row/index mutation.

The caller assertion and engine ceiling are independent: an assertion below 4,096 can
reject earlier; an assertion above 4,096 is rejected at validation; sparse predicates
can hit the scanned-key ceiling without hitting the affected-row assertion. `RETURNING`
is permitted only within separately preflighted response bounds.

### Prefix lane

Public SQL `UPDATE ... ORDER BY <canonical primary key> ASC LIMIT n` intentionally
selects one deterministic bounded window. Its success says only that the selected
window was processed. It neither asserts nor proves complete-set exhaustion. It must
not lower into Exact, and Exact must not lower into it. This is maintained public
semantics, not a compatibility fallback.

### Resumable lane

Prepare validates one narrow stable convergence plan without scanning target rows. It
resolves any accepted `DEFAULT` into a fixed accepted slot payload immediately, binds
schema/scope/patch/target/policy into an opaque current-format continuation, and returns
`Forward(start)`. The application must durably custody that token outside the target
store before asking for the first resume.

Forward traverses only canonical accepted primary keys in ascending order. Each call
examines at most 256 authoritative keys, identifies at most 64 rows whose caller-authored
target fields do not already equal the frozen patch, and trial-stages complete row plus
derived-state mutations. Exact production marker/journal length rules select the largest
fitting prefix. The call either commits one complete marker-owned atomic batch or no
batch. Clean rows may advance the checkpoint without a marker. A candidate that does
not fit is left strictly after the returned checkpoint.

This makes response loss safe: replaying an older continuation re-examines committed
rows; `needs_patch` sees their authored fields already converged and does not re-run a
managed-only mutation. Forward keyspace exhaustion does **not** mean complete. It
captures the store-wide durable mutation revision and enters Verify. Verify performs a
bounded full primary-key sweep using the same frozen scope and patch. A revision change
or residual row resets to `Forward(start)`. Only clean keyspace exhaustion followed by
a final unchanged-revision check may return `complete = true`. This proves stable
convergence during one verification sweep, not a start-of-job snapshot, exactly-once
processing, permanent enforcement, or operation-wide atomicity.

## 7. End-to-end flow traces

### Exact

`DbSession::execute_trusted_sql_exact_update` / facade wrapper
→ SQL surface classification
→ accepted catalog and row-layout descriptor
→ `SqlExactUpdatePolicy` validation
→ structural selector and patch
→ primary-only projection with 4,097-key scan probe
→ assertion-plus-one matching candidate probe
→ overflow/exhaustion decision
→ frozen target keys
→ common accepted structural mutation resolver
→ authoritative current-row revalidation
→ sanitizers/managed fields/constraints/indexes/relations
→ canonical journal and marker preflight
→ bounded `RETURNING` preflight
→ marker publication
→ mechanical row/derived/journal apply
→ marker clear/recovery
→ receipt and deterministic `RETURNING` shaping.

### Prefix

Generated/public SQL bounded update endpoint
→ public SQL policy classification
→ require canonical accepted primary-key ascending order, positive bounded `LIMIT`, no
offset
→ validated prefix bounds
→ select exactly that window
→ common accepted mutation resolver and marker-owned atomic apply
→ bounded result. No exhaustion claim is made or consumed.

### Resumable

`prepare_trusted_sql_resumable_update(operation_id, sql)`
→ mandatory store recovery gate
→ journaled-store check
→ accepted descriptor
→ plan/selector/patch normalization
→ accepted-default resolution into fixed slot payload
→ eligibility and dependency closure
→ target/schema/scope/patch fingerprints
→ encode `Forward(start)`
→ application durable external custody.

Later `resume_trusted_sql_resumable_update(sql, continuation)`
→ strict current token decode
→ mandatory recovery gate
→ journaled/accepted authority
→ rederive plan/fixed patch/eligibility
→ compare target/schema/scope/patch/policy (operation id is missing: F1)
→ validate checkpoint
→ Forward or Verify.

Forward
→ exclusive PK scan
→ accepted-row decode and stable scope
→ authored-field `needs_patch`
→ at most 64 candidates/256 classified keys
→ complete common-resolver staging
→ exact canonical pack trial
→ at most one marker publication/apply
→ checkpoint after last classified-or-committed key
→ on authoritative exhaustion, capture durable revision and enter Verify.

Verify
→ compare captured revision before page evidence
→ bounded exclusive PK scan with same scope/patch
→ revision drift or residual work resets `Forward(start)`
→ otherwise retain captured revision across pages
→ on authoritative exhaustion, final revision compare
→ return no continuation and `complete=true` only if still equal.

## 9. Exact-lane audit

### Entry points and policy

The public/trusted Rust facade can request Exact explicitly through the trusted SQL
Exact method and `SqlExactUpdatePolicy`; generated primary-key methods remain single-row
and generated bounded methods remain Prefix. SQL text alone does not infer Exact. The
policy stores a positive `NonZeroU32`, rejects assertions above 4,096, and derives a
checked assertion-plus-one selection limit. `StructuralProjectionScanBudget` separately
derives a checked 4,097-key probe from the 4,096 authoritative scanned-key ceiling.

The two probes answer different questions:

- matching candidate N+1 proves the caller's affected-row assertion false;
- authoritative key 4,097 proves the engine cannot establish exhaustion within its
  scan budget, even if few rows matched.

Covering/secondary routing is disabled for the scan-budget path. Candidate discovery is
primary-only and bounded; a page end is not treated as entity-keyspace exhaustion. The
executor can stop once either overflow is proved; it need not count the suffix. If both
limits would be exceeded, the scan overflow may surface first, but no success or partial
mutation is possible.

### Frozen selection and accepted after-images

The selector runs to an accepted terminal result before mutation staging. Candidate keys
are materialized in deterministic order and passed as expected current identities to the
same accepted structural resolver used by maintained mutation paths. Each authoritative
current row is re-read/revalidated, historical layouts are decoded under the accepted
0.209 layout contract, the update patch preserves unspecified fields, and the emitted
row is a complete current-layout envelope. The key set is not reselected during staging.

Sanitizers, built-in managed fields, uniqueness/exclusion, row constraints, ordinary
indexes, and relations are evaluated inside the common preflight. Journal and marker
payloads are built before publication. Requested `RETURNING` rows are shaped from
sanitized staged after-images and row/count/byte limits are checked before the marker;
postcommit shaping repeats deterministic conversion and has no new data-dependent
failure branch under the same accepted schema.

### Atomicity and rejection analysis

Before marker publication, rows, indexes, relations, journal tail, and receipts remain
unpublished. Projection scan metrics/caches may observe a read, but those are not row or
index state and do not change Exact semantics. After publication, apply is mechanical
and the 0.207 recovery gate completes all-after. Heap stores omit journal replay but use
the same marker-owned row/derived atomic apply; journaled stores include the journal
append in the marker.

No cap reduction, truncation, retry-as-Prefix, or Exact-to-Resumable fallback was found.
Zero matches correctly produces a successful empty exact result without a marker.
Already-satisfied rows are still part of Exact's frozen target; normal update semantics
decide their resulting managed behavior rather than silently changing contract lanes.

Repository-wide symbol and semantic searches found no production use of

## 11. Resumable eligibility and prepare

### Eligibility result

The current MVP accepts one generic accepted entity in one store only. Prepare and
resume require `StoreRuntimeStorageMode::Journaled`. The SQL policy rejects windows,
`RETURNING`, subquery/cross-entity and row-relative forms. `AcceptedFixedUpdatePatch`
accepts literal values and accepted `DEFAULT`; it rejects primary-key, generated,
managed/internal ownership, and unsupported assignment shapes. Accepted descriptor
metadata rejects unique, exclusion, relation-owned, or other global constraints on an
assigned field. Callback profiles reject application sanitizer/validator semantics that
could make convergence replay unsafe. Built-in managed behavior remains permitted only
where it is not globally constrained.

The accepted selector's normalized roots feed dependency closure. Assigned roots must
be disjoint. Unknown roots, unknown expression forms, incomplete compile evidence, and
aggregate/cross-entity meaning reject rather than becoming an empty permissive set.
Current accepted schemas have no row-dependent computed fields, so the closure is often
just normalized predicate roots; this is truthful current authority, not reconstruction
from generated models. Future row-dependent accepted fields would require this owner to
grow before eligibility could remain sound.

Ordinary non-unique index maintenance is permitted because each candidate is staged as
a complete atomic row/index mutation. There is no caller batch size, `OFFSET`, secondary
route, arithmetic/append assignment, nondeterministic value, PK move, resumable row
payload, or heap generation counter accepted as durable authority.

### Accepted defaults and frozen meaning

`DEFAULT` is resolved by `AcceptedFixedUpdatePatch::from_update_intent` against the
accepted descriptor during prepare, producing a concrete accepted slot payload. Its
fingerprint binds those bytes. Resume rederives the current accepted fixed patch and
compares the fingerprint before row scanning. A later accepted default/schema change
therefore rejects the old token; it cannot reinterpret the operation. Historical fill
remains a row-decoding authority only and is never used as update policy. Every emitted
after-image uses the current stamped row layout.

### Read-only prepare and custody ordering

The operation-specific prepare path performs policy/catalog/fingerprint work and token
encoding only. It does not invoke the primary-key scanner, build a target marker, append
the target journal, advance the target revision, or create an engine job row. It returns
`Forward` with no checkpoint. This gives the application a real interval in which to
persist the token outside the target store before any call to resume.

One subtle qualification is correct under the governing ATOMICITY contract: prepare
starts through the mandatory recovered-store gate. If an unrelated earlier marker was
already durably published, recovery may mechanically finish that committed mutation.
That is not target discovery or a mutation authored by prepare; forbidding it would
violate the repository's recovery-before-access rule.

Custody durability, operation-id uniqueness, collision handling, and per-resume
authorization are necessarily application responsibilities. The repository's
Toko-shaped test demonstrates sequencing but holds control state in an in-memory test
variable; it does not prove a durable external application write or restart. More
importantly, engine resume does not validate the application operation id at all (F1),
so the intended custody binding is incomplete even if an application persists it.

## 12. Continuation codec, identity, and trust boundary

### Current wire form

The deterministic encoding order is:

1. magic `ICYU` and format version 1;
2. 16-byte operation id;
3. entity tag and 32-byte target/store allocation identity;
4. accepted schema fingerprint method and 16-byte fingerprint;
5. 32-byte normalized scope hash;
6. 32-byte fixed accepted-patch hash;
7. phase;
8. length-prefixed raw accepted primary-key checkpoint;
9. optional Verify revision;
10. 32-bit batch-policy identity;
11. CRC32C over the preceding payload.

The decoder enforces 156 bytes minimum, 2,048 maximum, exact magic/current version,
bounded checked length reads, valid entity-tag checkpoint decoding, valid phase/revision
pairing, checksum, and full input exhaustion. Unknown phase/version, truncation,
oversize, malformed key, invalid revision presence, or trailing bytes fail as the typed
malformed-continuation boundary. There is no map ordering ambiguity, executable SQL,
row after-image, authorization, cumulative count, stable-memory address, job pointer,
or unbounded field. The representative initial token is 156 bytes.

CRC is correctly described as corruption detection, not authentication. The application
must keep the bytes trusted and reauthorize resume. The public Rust `try_from_bytes` is
therefore appropriate for application custody and is not itself a public canister
endpoint.

### Binding validation

Target/entity, schema fingerprint+method, scope, patch, and batch policy are rederived
and compared before the Forward/Verify row scanner. The checkpoint is then decoded as
the expected accepted entity key type. Format failure occurs even earlier. No current
legacy decoder, downgrade, translator, alias, or fallback exists.

The operation id is the exception. It is encoded at prepare but the resume method takes
only `(sql, continuation)`, and `validate_resumable_update_bindings` has no expected
operation-id parameter or comparison. The diagnostic registry has no operation-id

## 13. Forward scanning, `needs_patch`, and checkpoints

Forward constructs an accepted entity primary-key range with `Bound::Excluded` for a
present checkpoint. It uses the store's authoritative data-key iteration order and
decodes each row under the accepted row contract. It compiles and evaluates the frozen
scope, then calls fixed-patch satisfaction only over caller-authored target slots.
Historical stamped layouts are decoded through the 0.209 accepted-fill model; malformed
current or historical rows fail rather than gaining update/default values.

The scanner classifies at most 256 keys and collects at most 64 needs-patch candidates.
An extra iterator callback sets `has_more`; therefore exactly filling a clean page or
candidate page is not confused with keyspace exhaustion. Clean, out-of-scope, or
already-converged rows can advance the checkpoint without a marker and without managed
field writes. A needs-patch row is passed to the common accepted structural resolver,
which constructs a complete after-image and all derived effects.

`needs_patch` deliberately excludes managed timestamps. On response-loss replay, a row
whose authored fields were committed is no-work, so a timestamp is not changed again
solely because the old token was retried. Application sanitizers/validators are
ineligible, preventing a sanitizer from making the stored target perpetually differ
from the frozen patch. Accepted equality is slot-payload/value-canonical, including
NULL, structured values, and resolved defaults.

### Checkpoint proof

Each candidate records both the checkpoint and `keys_scanned` count immediately before
it. After trial packing, if candidate k is the first deferred row, the returned progress
rolls back to k's `checkpoint_before` and `keys_scanned_before`. Thus every key covered
by the returned checkpoint was either fully classified no-work or included in the
committed batch. A rejected/deferred candidate remains strictly after the token.

Insertions ahead of a Forward checkpoint may be seen later; insertions behind it may be
missed during that Forward pass, but the final Verify starts at the beginning and the
journal revision invalidates an unstable sweep. Deletes and scope/value changes behave
similarly. This is convergence, not snapshot isolation. Moving a primary key is
ineligible as a resumable assignment; independent deletes/inserts still invalidate
Verify.

Tests cover empty and multi-batch ordinary progression, clean pages, response-loss
replay, and residual work. They do not directly pin all requested boundaries: first/last
key, exactly 256/257, needs-patch at 257, candidate 64/65, maximum encoded PK, malformed
checkpoint, or duplicate/reordered iterator fault. The production iterator authority
makes duplicate/reordered output structurally unlikely, but the claimed adversarial
evidence is incomplete.

## 14. Incremental packing, marker atomicity, and replay

### Packing

For each ordered candidate, the common resolver produces the complete new row and
ordinary index/relation deltas. Trial packing uses canonical production helpers for
journal row put/delete payloads, batch framing, marker payload framing, and control-slot
length. It accounts for old/new effects through the same prepared row operation used by
final commit, rather than a separate approximate byte heuristic. Length conversion and
framing bounds are checked; no unordered map controls candidate order.

If the next complete row fits, it remains in the prefix. At the first nonfit row, the
packer stops and removes it. A nonempty prefix commits exactly once. If the first row
cannot fit alone, the explicit `ResumableUpdateSingleRowResourceExceeded` boundary is
returned without marker or checkpoint advancement. A clean page creates no empty
marker. No loop can publish a second batch in the same call.

The real 64-row fixture reproduces 10,869 encoded journal-batch bytes and 10,907 durable
marker/control-slot bytes. The test prints rather than asserts those exact numbers, so
codec drift could silently make the prose stale even though the structural bound remains
tested. The design's exact values are current in this checkout.

No quadratic unbounded route was found: staging is bounded at 64. Some prefix trial
accounting is repeated, but its structural cap prevents the claimed resource model from
becoming unbounded.

### Marker boundary and failpoints

The canonical 0.207 flow remains:

full after-image and derived-state preflight
→ journal/marker encode and capacity proof
→ marker publication (the commit boundary)
→ mechanical rows/indexes/relations/journal apply
→ marker clear.

Recovery invokes the marker's already-resolved bytes; it does not re-run SQL, accepted
defaults, scope, sanitizers, managers, or plan logic. Before publication, retry simply
reruns bounded planning/staging against current authority. After publication, guarded
recovery completes all-after. No pending-reset rollback path or compatibility rollback
was introduced.

The generic commit suite covers failures before marker, immediately after marker,
during row/index/journal application, and marker clear/recovery. Resumable tests cover a
representative before/after-marker failure and old-token response-loss replay. They do
not independently exercise every listed real resumable boundary—marker construction,
encoding, each derived owner, after clear/before response, and custody-write loss—as
claimed by the design.

Earlier committed batches are never rolled back by later failures. Receipts are
per-call, not replay-stable cumulative authority. The implementation does not promise
exactly-once invocation, operation-wide atomicity, cumulative exact updated count, or
start-of-job snapshot semantics; searches found no production API making those claims.

## 15. Durable mutation revision authority

The revision is not a new counter. It is the next contiguous `JournalSequence` after the
highest of the persisted fold watermark and journal-tail entries. `JournalSequence` is
a big-endian persisted `u64`; sequence zero is the fold-watermark control key, normal
append starts after it, and `next()` uses checked addition. Folding copies the highest
folded sequence into the watermark and removes represented tail entries without
changing the next sequence. Marker recovery appends the already-staged sequence
idempotently; reuse for different bytes fails closed.

### Mutation-ingress revision matrix

| Mutation ingress | Changes accepted row state? | Participates in revision? | Atomic with mutation? | Evidence |
|---|---:|---:|---:|---|
| Typed/generated single-row insert/update/delete | yes | yes on journaled store | yes, same marker batch | common commit builder + journal tests |
| Fluent mutation | yes | yes | yes | same accepted structural resolver/commit owner |
| SQL INSERT/UPDATE/DELETE | yes | yes | yes | common commit builder; correctness tests |
| Exact UPDATE | yes | yes | yes | frozen batch feeds common builder |
| Intentional Prefix UPDATE | yes | yes | yes | same builder |
| `update_many_atomic` | yes | yes | yes | one caller-supplied marker batch |
| Resumable Forward | yes when nonempty | yes | yes | one common commit then receipt |
| Marker recovery | completes already-committed change | reuses staged sequence idempotently | marker is existing boundary | recovery tests |
| Accepted schema publication/default/layout metadata | can change accepted meaning | conservatively advances journal sequence | marker-owned publication | schema publication path/tests |
| Journal fold | no new accepted mutation | preserves, does not advance meaning | watermark/tail atomic owner | fold-without-restart test |
| Index repair/publication only | not accepted row bytes; may change derived authority | schema/publication paths conservatively journal where applicable | marker/publication owner | code trace/guards |
| Test raw-row helper | test-only accepted bytes | may bypass | not production | `cfg(test)` fixture only |
| Production low-level `with_data_mut` | recovery projection reset/replay only | consumes marker/journal authority | recovery-owned | repository-wide owner search |
| Heap mutation | yes, but heap is ineligible for resumable Verify | no durable journal revision required | heap marker model | explicit resumable rejection |

No reachable production direct write bypass was found. Every normal journaled row
mutation obtains a sequence during common commit-window construction, and accepted
schema publication conservatively invalidates Verify. Custody writes are outside this
store and do not alter the target revision, as required. Unrelated target-store writes
therefore restart Verify conservatively.

### Exhaustion defect

`next_append_sequence()` returns `last_sequence.next()`. If the highest existing
sequence is `u64::MAX - 1`, it validly returns `u64::MAX`; the common commit builder
stages and commits that batch. Afterward `durable_store_revision()` asks again for the
next sequence, which overflows and reports journal corruption. If this was Forward's
exhausting final batch, rows and derived state are already committed before revision
capture fails. Replaying the old token sees converged rows, but revision remains
unrepresentable, so the operation can never enter/complete Verify. This violates
fail-closed-before-mutation exhaustion semantics and creates an unknown-outcome call
(F2). No boundary test exists.

## 16. Verify and completion

Forward enters Verify only when the authoritative iterator reports no additional key,
after any nonempty batch is committed and its marker cleared. A 256-key page, 64-row
candidate cap, marker/journal resource edge, or instruction/response condition cannot
set the exhaustion flag. It then captures the durable revision and creates
`Verify(revision,start)`.

Each Verify call:

1. compares current durable revision with the captured revision before accepting page
   evidence;
2. on mismatch, clears phase checkpoint/revision to `Forward(start)` and returns exactly
   `revision_changed`;
3. scans the same accepted PK order, scope, and fixed authored patch, at most 256 keys;
4. on any scoped needs-patch row, resets to `Forward(start)` and returns exactly
   `residual_work`;
5. on a clean nonexhausted page, preserves the original revision and returns the last
   fully examined PK as the next Verify checkpoint;
6. on clean authoritative exhaustion, compares the revision a second time;
7. returns `complete=true` with no continuation only if the revision still matches.

The reset to start is essential and implemented correctly; no stale Forward or Verify
checkpoint survives. A behind-checkpoint insertion/change increments revision and
invalidates a later page, while residual work already encountered also restarts from
the beginning. A journal fold without a new mutation preserves revision and does not
spuriously restart. A stale Verify or Forward token can be replayed, but its current
schema/policy bindings are rechecked and the state machine cannot convert stale partial
evidence directly into completion.

The covered scenarios include empty/ordinary completion, multi-page Verify, unrelated
write, behind-checkpoint residual work, journal fold, stale replay, and final stable
completion. Exact 256/257 boundaries, revision corruption/exhaustion, every scope
transition direction, and full engine-upgrade restart are not all separately evidenced.
F2 means Verify is not available after the terminal journal sequence has been consumed.

The precise completion claim upheld by ordinary paths is:

> The complete authoritative entity keyspace was scanned while the target store's
> durable mutation revision remained unchanged, and no accepted row matching the scope
> needed an authored target value.

It is not a transaction spanning calls, a snapshot from prepare, exactly-once work,
future enforcement, or an application business fence.

## 17. Receipts, diagnostics, and failure propagation

The receipt has private fields and public getters for phase, `keys_scanned`,
`rows_updated`, restart reason, continuation, and complete. Only the internal progress
and completion constructors create it. Progress always has exactly one continuation and
`complete=false`; completion is Verify, has no continuation/restart, zero updated rows,
and `complete=true`. This prevents callers from constructing impossible combinations
despite the representation using independent fields.

Forward `keys_scanned` counts only fully classified keys, rolling back before a deferred
candidate. `rows_updated` is exactly the committed prefix in this call. Clean Forward
and all Verify calls report zero updates. There is no `batch_examined`, scope-match
count, needs-patch count, cumulative advisory total, row payload, or resumable
`RETURNING`. Recovery/replay does not invent cumulative rows.

Diagnostics retain distinctions for invalid/missing/high Exact assertion, affected
overflow, scan overflow, Exact window, resumable window/returning, journaled-store
requirement, global/scope dependency, malformed/target/schema/scope/patch/policy
continuation, oversized single row, callbacks, and managed global constraint. Core,
facade, CLI, and registry tests compare typed codes rather than prose. Restarts are
successful progress states, not errors or completion.

Two diagnostic gaps follow the substantive findings:

- there is no operation-id mismatch variant because no comparison exists (F1);
- journal revision exhaustion is collapsed into journal corruption and can occur after
  a successful commit (F2), rather than a typed pre-marker resource/exhaustion boundary.

Searches of affected production code found no new `panic!`, `assert!`, `unwrap`,
`expect`, or string-matched error substitution in executor paths. The metric conversion
uses `unwrap_or(u64::MAX)` only to saturate an observational scan metric, not to resolve
state or control completion. Test-only unwraps/assertions are conventional. Match arms
for phase/version/eligibility fail closed rather than defaulting unknown cases into a
supported lane.

## 18. Persistence, schema, layouts, and temporal defaults

Exact and Resumable both obtain `AcceptedSchemaCatalogContext` and an accepted row-layout
runtime descriptor before resolving selector/patch or decoding rows. `EntityModel` and
`IndexModel` participate in proposal compatibility, generated convenience, and tests;
neither lane reconstructs accepted runtime authority from a generated model. A generated
proposal that differs from accepted schema fails descriptor/fingerprint checks.

Historical rows are decoded using their exact stamped 0.209 layout. Historical fills
remain frozen decode values for provably absent old slots. Current-layout omission or a
malformed row is corruption; it does not fall through to the insert policy or a default
value. Exact/Resumable update construction preserves decoded existing values for
unassigned fields and emits one complete current-layout row envelope. Accepted insert
defaults are not reused as historical fill or as implicit update values.

For resumable `SET field = DEFAULT`, accepted policy is resolved to a concrete payload at
prepare and rederived/fingerprinted on resume. A later default, layout, constraint,
index, relation, sanitizer, managed-field, or accepted schema change changes the accepted
schema and/or patch binding, rejecting before scanning. The schema fingerprint therefore
covers the accepted authority consumed by the descriptor, while scope and patch hashes
bind normalized operation-specific meaning separately.

The baseline-to-HEAD byte/semantic inventory found no change to:

- persisted row envelope or row-layout-version encoding;
- historical-fill or insert-default catalog encoding;
- accepted schema stable snapshot;
- ordinary index/relation key encoding;
- journal batch record format or fold watermark;
- marker/control-slot envelope;
- SQL cursor, Candid, or JSON public representation.

Journal and marker files changed to expose exact length calculations; encode/decode
format remains current-only and unchanged. The new continuation is not engine stable
state: it is a current hard-cut application-custodied blob. Version 1 is the only reader,
and schema/format/policy mismatch requires application restart rather than translation.

## 19. Upgrade and engine lifecycle

There is no durable engine job registry. Active jobs exist only as application control
state plus current continuation bytes. Accepted-schema change causes schema/fixed-patch
binding rejection; token-format and explicit policy-id change reject; target allocation
or entity change rejects. Earlier committed row batches remain ordinary accepted state.
Starting a new operation under current authority safely skips authored rows already at
the frozen target through `needs_patch`.

No token migration, compatibility decoder, version downgrade, hidden active-job lookup,
or fallback policy was found. Schema DDL does not pretend to enumerate active operations.
The application must mark or clear its control state as `restart_required` across an
engine upgrade before accepting resume requests; this is documented but not enforced by
an engine registry, by design. The Toko-shaped evidence models restart intent but does
not persist and reopen real application control state.

The batch-policy identity is present and compared, but it is the literal constant `1`.
No derivation or governance test couples a required increment to the scan bound, staged
row bound, packing/accounting rules, checkpoint interpretation, codec semantics,
`needs_patch` equality, or revision interpretation. Thus current old-policy mismatch
works only when a developer remembers to bump the constant. That is insufficient proof
of the claimed hard engine-upgrade boundary (F4).

## 25. Ranked findings

### F1 — P1: encoded operation id is never validated on resume

- **Category:** continuation identity / trusted custody.
- **Violated contract:** a continuation cannot change operation id; every binding is
  compared before row access.
- **Production evidence:** the id is encoded in
  [`resumable_update.rs:133`](../../../crates/icydb-core/src/db/session/sql/resumable_update.rs#L133)
  and decoded into the continuation at
  [`resumable_update.rs:216`](../../../crates/icydb-core/src/db/session/sql/resumable_update.rs#L216),
  but resume accepts only SQL and a token at
  [`resumable_update.rs:486`](../../../crates/icydb-core/src/db/session/sql/resumable_update.rs#L486).
  Binding validation at
  [`resumable_update.rs:978`](../../../crates/icydb-core/src/db/session/sql/resumable_update.rs#L978)
  compares target, schema, scope, patch, and policy only. The diagnostic list at
  [`icydb-diagnostic-code/src/lib.rs:685`](../../../crates/icydb-diagnostic-code/src/lib.rs#L685)
  has no operation-id mismatch.
- **Test/document evidence:**
  [`sql_resumable_update.rs:147`](../../../crates/icydb-core/src/db/session/tests/sql_resumable_update.rs#L147)
  proves different operation ids produce different bytes, but never attempts to resume
  operation A with operation B's token. The design says the id is bound and checked.
- **Reachable scenario:** an authorized application handler loads a token under control
  row A but accidentally/substitutively supplies token B. If target, entity, schema,
  SQL scope, patch, and policy are the same, IcyDB accepts and advances B's checkpoint.
- **Observable consequence:** custody/progress and authorization attribution cross
  operation identities. The row meaning may happen to be identical, but the caller's
  selected operation identity and lifecycle state are not protected.
- **Why tests missed it:** they asserted byte inequality at preparation, not boundary
  validation at resume; the API exposes no expected id to test.
- **Smallest hard-cut correction:** require the expected application operation id at
  resume (or an equivalently validated operation-bound custody value), compare it before
  recovered target-row access/scanning, and add a typed diagnostic. Do not add aliases or
  accept both old/new forms.
- **Closure evidence:** substitute and mutate operation ids through the real facade;
  assert typed code, zero scanned keys, unchanged rows/indexes/journal/marker/receipt;
  exercise the application custody fixture.
- **Confidence:** high.

### F2 — P1: terminal journal sequence can commit before revision exhaustion is detected

- **Category:** durable revision / postcommit failure / convergence liveness.
- **Violated contract:** revision exhaustion fails closed; one call returns a complete
  atomic batch with valid continuation or no batch; Verify must remain reachable after
  Forward exhaustion.
- **Production evidence:** `JournalSequence::next` checks addition at
  [`journal/codec.rs:39`](../../../crates/icydb-core/src/db/journal/codec.rs#L39).
  `next_append_sequence` returns highest+1 at
  [`journal/store.rs:194`](../../../crates/icydb-core/src/db/journal/store.rs#L194),
  allowing `u64::MAX` when highest is `MAX-1`. Common commit construction consumes that
  sequence at
  [`commit_window.rs:1117`](../../../crates/icydb-core/src/db/executor/mutation/commit_window.rs#L1117).
  After a Forward commit, exhaustion capture calls
  [`durable_store_revision`](../../../crates/icydb-core/src/db/session/sql/resumable_update.rs#L967),
  which asks for the *next* sequence and then fails.
- **Test/document evidence:** no revision corruption or terminal-exhaustion test exists;
  ordinary revision/fold tests use low sequence values. The design claims exhaustion
  fails closed.
- **Reachable scenario:** highest journal sequence is `u64::MAX-1`; an exhausting
  Forward batch is assigned `u64::MAX`, marker-published, applied, and cleared. Capturing
  `Verify` revision then errors. Replaying the old Forward token sees authored rows
  converged, but every revision read still fails.
- **Observable consequence:** the call reports failure after committing its batch and
  the operation can no longer prove completion. This is an unknown-outcome API result
  and permanent resumable-lane stall for that store.
- **Why tests missed it:** natural tests cannot approach a 64-bit terminal value and no
  injected sequence fixture targets the boundary.
- **Smallest hard-cut correction:** reserve an always-representable post-append revision
  before marker publication—for example, reject a mutation before staging/publication
  when consuming the candidate sequence would leave no next revision. Surface a typed
  exhaustion error; do not wrap, reset, or reuse sequence ids.
- **Closure evidence:** inject `MAX-1`, attempt ordinary and exhausting resumable writes,
  prove typed pre-marker rejection with byte-identical rows/indexes/journal/marker, then
  prove recovery and Verify at the highest admitted sequence.
- **Confidence:** high.

## 26. Challenged but upheld decisions

- **Prepare through recovery:** `recovered_store` can finish a previously published
  unrelated marker. This does not violate read-only operation preparation; it enforces
  the repository's mandatory recovery-before-access contract.
- **CRC without authentication:** correct. The continuation is trusted application state,
  not a bearer credential. Authentication belongs to application authorization.
- **Public Rust byte reconstruction:** correct for trusted custody. The prohibited surface
  is generated/public canister input, which is currently absent.
- **Affected versus scan overflow order:** correct. If 4,097 matching primary keys also
  exceed the scanned-key probe, either typed rejection establishes “not exact”; neither
  may mutate or report success.
- **Two extra-probe scanners:** correct. Both Exact and Forward/Verify require iterator
  evidence beyond a full bounded page to distinguish budget from keyspace exhaustion.
- **Deferred candidate checkpoint rollback:** correct. The saved `checkpoint_before` and
  count exclude the first nonfitting candidate.
- **Managed fields excluded from `needs_patch`:** correct. This prevents response replay
  from producing managed-only updates; application sanitizers/validators are ineligible.
- **Receipt fields rather than a public state enum:** acceptable. Fields and constructors
  are private, and only two constructors create valid continuation/complete combinations.
- **Current dependency closure often equals predicate roots:** acceptable under current
  accepted schema facts; unsupported/unknown expression evidence rejects. It must evolve
  before row-dependent computed accepted fields are admitted.
- **Journal fold does not advance revision:** correct. It changes representation, not
  accepted row state, and preserves the highest sequence in the watermark.
- **No persisted row/schema migration:** correct. Length helper refactoring does not alter
  marker/journal bytes; the only new format is external trusted continuation state.
- **Generated model checks:** acceptable as proposal compatibility evidence. Runtime row
  decode, constraints, layout, scope, and patch use accepted catalog authority.
- **Public SQL Prefix retained:** correct. Deterministic ordered `LIMIT` is an intentional
  public contract, not legacy fallback or complete-set proof.
- **No engine job registry:** correct MVP ownership. Durable application custody plus
  restart is simpler and avoids a second scheduling/authorization authority.

## 27. Remaining uncertainty

ordering in a Toko-shaped in-memory fixture but does not persist application control
state, lose a response/custody update, restart the process/canister, reauthorize, and
resume under current authority end to end. That gap matters because operation identity
validation is currently defective.

the first-row and 64/65 packing/resource boundary; managed-field replay with historical
rows and fixed defaults; revision corruption/exhaustion after correction; every real
failpoint through each public lane; semantic generated-surface enforcement; MSRV and the
human full push suite; and canonical final Wasm reproducibility.

### Direct answers

- **Can Exact ever report success without proving complete selector exhaustion?** No
  reachable current path was found; Exact success consumes distinct exhaustion evidence.
- **Can Exact overflow or any pre-marker rejection change state?** Not row, index,
  relation, journal, or marker state by production trace; the exhaustive byte-level test
  claim is incomplete.
- **Can Prefix ever be mistaken for complete-set success?** No current consumer makes
  that inference.
- **Can prepare access or mutate target rows?** It does not scan or author target rows.
  Its mandatory recovery gate may mechanically finish an already-committed older marker.
- **Can the first mutation occur before durable external custody?** Yes if an application
  violates the required ordering; the split API makes custody possible but cannot attest
  external durability, and the repository fixture does not prove a real durable write.
- **Can a continuation change operation meaning?** Yes: operation-id substitution is not
  validated (F1). Schema, target, scope, patch, format, and current policy are bound.
- **Can any unsupported resumable plan reach row access?** No operation-specific row scan
  was found before eligibility/binding; mandatory recovery is the noted exception for an
  already-published marker.
- **Can Forward skip a row deferred by packing?** No; it rolls the checkpoint and scanned
  count back before that candidate.
- **Can one resume call commit more than one batch?** No.
- **Can response loss cause a committed row to be mutated again solely because of
  replay?** Not when its authored target slots remain converged; `needs_patch` skips it.
- **Can scan-page exhaustion be mistaken for keyspace exhaustion?** No; the extra callback
  separates a full page from iterator exhaustion.
- **Can Forward exhaustion directly report complete?** No; it enters Verify.
- **Can any journaled row mutation bypass revision invalidation?** No reachable production
  bypass was found. Terminal revision representability is defective (F2).
- **Can Verify complete across a revision change?** No under representable revision state;
  it checks before pages and again at final exhaustion.
- **Can behind-checkpoint work evade the final proof?** Not if mutation revision remains
  sound; Verify restarts at start on drift/residual work.
- **Can heap stores enter the resumable lane?** No.
- **Can an old token survive schema, engine, format, or policy change?** Schema, explicit
  format, target, and changed policy ids reject. An engine semantic change can
  accidentally accept it if the hand-maintained policy id is not bumped (F4).
- **Can a public/generated endpoint accept a raw continuation?** None currently does.
  The guard against future aliases/wrappers is incomplete (F6).
- **Does any ambiguous or unbounded trusted update route remain?** No shipped production
  route; a prohibited test-only unbounded lane remains (F3).
- **Is the recorded closeout and performance evidence reproducible?** Runtime instruction,
  marker, and continuation figures are. The final Wasm size/hash and several claimed
  adversarial test lines are not (F5, F7).
- **Is 0.210 genuinely ready to close?** **No.** Correct F1 and F2, close the bounded P2
  evidence/hard-cut issues, then run the human-owned full push suite from a clean
  worktree.
