# **0.136 StructuralPatch Boundary Closeout**

## Thesis

`0.136` finishes the structural write cleanup left open by the earlier
`StructuralPatch`, create-input, and persisted-row slot-contract work.

The goal is not to add another mutation mode. The goal is to make the current
write surfaces explicit:

* typed full-entity saves
* typed authored creates
* structural field patches
* SQL writes lowered into structural patches

All of those should converge on one prepared mutation handoff before commit
planning, while preserving their distinct public ergonomics and validation
contracts.

---

## Current State Audit

### Public Facade

* `icydb::Create<E>` aliases the generated create-input type through
  `EntityCreateType`.
* `DbSession::create(...)` accepts authored create inputs.
* `DbSession::insert(...)`, `update(...)`, and `replace(...)` accept complete
  typed entity after-images.
* `DbSession::mutate_structural(...)` accepts:

```rust
key + StructuralPatch + MutationMode
```

`StructuralPatch` is field-name-driven at the public facade and wraps the core
patch type.

### Core Patch Model

Core persisted-row currently has two patch shapes:

```rust
StructuralPatch {
    entries: Vec<StructuralFieldUpdate>,
}

SerializedStructuralPatch {
    entries: Vec<SerializedStructuralFieldUpdate>,
}
```

The split is useful:

* `StructuralPatch` is still semantic/runtime-value-shaped.
* `SerializedStructuralPatch` is the mechanical slot-payload handoff.

That boundary should survive.

### Create Model

Generated create inputs materialize:

```rust
EntityCreateMaterialization {
    entity,
    authored_slots,
}
```

This lets typed create preserve the distinction between:

* field omitted by caller
* field explicitly authored
* generated or managed field materialized by preflight

That provenance is still necessary and should not be collapsed into
`StructuralPatch`.

### SQL Write Model

SQL `INSERT` and `UPDATE` lower into core `StructuralPatch` internally.

This is the right broad shape: SQL writes are dynamic field-name writes, so the
structural patch pipeline is their natural handoff. SQL should not grow a
separate row-write representation.

### Executor Model

Executor mutation currently converges on:

```rust
MutationInput {
    data_key,
    serialized_slots,
}
```

This is close to the desired handoff. It already means later mutation stages do
not need to keep typed entities or public patch builders alive.

## Ownership Map

| Area | Actual Responsibility | Should Own | Should NOT Own | Verdict |
| --- | --- | --- | --- | --- |
| Public `icydb::StructuralPatch` | field-name-driven public patch builder over `InputValue` | API ergonomics and error mapping | slot serialization or save semantics | Clean wrapper, but docs should call it public-only |
| Core `db::data::StructuralPatch` | ordered field-slot + `Value` patch | semantic structural patch before encoding | commit staging or row-existence policy | Clean |
| `SerializedStructuralPatch` | ordered field-slot + encoded payload patch | mechanical persisted-row handoff | public API or SQL lowering | Clean |
| Generated `Create<E>` | authored create DTO excluding generated/managed fields | typed create ergonomics and authored-slot provenance | sparse update semantics | Clean |
| `EntityCreateMaterialization` | typed after-image plus authored slots | create authorship validation input | general patch representation | Clean |
| SQL write execution | lowers SQL write statements into structural patches | SQL-specific syntax and generated/managed field rejection | persisted-row payload ownership | Mostly clean |
| Save executor structural lane | mode-specific patch preflight and commit handoff | row-existence policy, final typed validation, normalization | public patch construction | Slightly broad but acceptable |
| Persisted-row patch module | patch serialization, replay, materialization | slot-payload mechanics | mutation mode semantics | Clean |

## Main Findings

### 1. The names are doing too much work

`StructuralPatch` is used for:

* sparse update over an existing row
* sparse insert after-image materialization
* sparse replace after-image materialization
* SQL `INSERT` row construction
* SQL `UPDATE` assignment replay

The behavior is controlled by `MutationMode`, not by the patch type.

This is correct mechanically, but the doc/API story should make clear that
`StructuralPatch` is a structural field patch, not update-only semantics.

### 2. Create and patch are intentionally different

`Create<E>` must keep authored-slot provenance. `StructuralPatch` intentionally
does not. Collapsing create into patch would lose the ability to distinguish
omission from authorship before generated/managed-field preflight.

### 3. Typed full-entity saves bypass `SerializedStructuralPatch`

Typed `insert/update/replace` now prepare a canonical row directly from the
typed entity. Structural writes go through `SerializedStructuralPatch`, then
normalize back to a typed entity before commit.

This is acceptable because typed saves already own a complete after-image.
`0.136` should not force typed saves through a dense patch unless there is
clear duplication to delete.

### 4. Structural insert/replace and update have different preflight shapes

Structural update overlays an existing raw row before typed validation.
Structural insert/replace materializes from the sparse patch first, then
normalizes to a complete typed entity.

This difference is semantically required. The cleanup opportunity is naming
and bundling, not making every mode share the same first step.

### 5. SQL write lowering is a valid structural-patch producer

SQL should keep lowering assignments/insert columns into `StructuralPatch`. The
important invariant is that SQL, facade structural writes, and internal
structural writes all pass through the same model-owned field serialization and
save preflight after lowering.

---

## Target Model

The intended write model is:

```text
typed full entity
  -> typed save preflight
  -> canonical row op
  -> commit

typed create input
  -> create materialization + authored slots
  -> typed save preflight
  -> canonical row op
  -> commit

public / SQL structural write
  -> StructuralPatch
  -> SerializedStructuralPatch
  -> mode-specific after-image materialization
  -> typed save preflight
  -> canonical row op
  -> commit
```

`StructuralPatch` stays above persisted-row encoding. `SerializedStructuralPatch` stays
below it. `MutationInput` remains the executor handoff.

## Recommended 0.136 Design

### 1. Rename by documentation first, code second

Before changing type names, document the semantic contract:

* `StructuralPatch` = ordered structural field patch
* `MutationMode::Update` = update semantics
* `MutationMode::Insert` / `Replace` = after-image materialization semantics

If a future breaking rename is worth doing, the better name is likely:

```rust
StructuralPatch
```

with the old `UpdatePatch` name removed rather than aliased.

### 2. Preserve the two patch phases

Keep:

```rust
StructuralPatch -> SerializedStructuralPatch
```

Do not merge them. The first is caller/value-shaped; the second is
persisted-row-payload-shaped.

### 3. Make the mutation handoff explicit

`MutationInput` should be treated as the executor-owned structural mutation
handoff:

```rust
key + SerializedStructuralPatch
```

It should not grow public API behavior. It should only carry already-lowered
write material needed by save execution.

### 4. Keep create authored-slot provenance separate

Generated create inputs should not lower directly to `StructuralPatch`.

They should continue to produce:

```rust
typed entity + authored_slots
```

because create authorship is a typed-create invariant, not a structural patch
invariant.

### 5. Centralize write-boundary field invariants

The same save preflight should own:

* primary-key consistency
* generated/managed write rules
* decimal scale
* bounded text once `0.135` lands
* deterministic set/map encoding
* relation validation

Patch serialization should only encode values into slot payloads. It should
not become a second business-rule validator.

## Safe Refactor Plan

### Step 1 — Document and narrow names

* Update public and core docs to call `StructuralPatch` a structural field patch.
* Make clear that mutation semantics live in `MutationMode`.
* No behavior change.

### Step 2 — Bundle structural mutation arguments internally

Introduce an internal input bundle if it reduces argument churn:

```rust
StructuralMutationRequest {
    mode,
    key,
    patch,
    write_context,
    authored_public_patch,
}
```

This is internal only. It should not replace the public facade API.

### Step 3 — Make `MutationInput` the only serialized handoff

Audit structural save paths and keep all post-lowering structural execution on:

```rust
MutationInput::from_structural_patch(...)
```

or an equivalent constructor. Avoid ad hoc `SerializedStructuralPatch`
construction outside persisted-row tests and row helpers.

### Step 4 — Split public-authored versus internal-authored structural writes

Keep the existing distinction explicit:

* public structural patches reject explicit generated-field writes
* internal SQL-generated patches may include generated values

If needed, replace `Option<&StructuralPatch>` with a small enum:

```rust
StructuralPatchOrigin {
    PublicAuthored,
    InternalLowered,
}
```

This avoids using `Option` as policy.

### Step 5 — Add missing parity tests

Add or confirm tests for:

* public `StructuralPatch` cannot write generated fields
* SQL `INSERT` can include internally generated values
* structural update overlays existing row
* structural insert/replace do not inherit omitted old fields
* duplicate patch entries remain last-write-wins
* typed create missing authored field fails through authored-slot validation
* bounded text validation, once `0.135` lands, is shared by typed and structural writes

### Step 6 — Hard rename

After the boundary is stable, apply the hard rename:

```rust
UpdatePatch -> StructuralPatch
FieldUpdate -> StructuralFieldUpdate
SerializedUpdatePatch -> SerializedStructuralPatch
```

Because backwards compatibility is not a goal, this should be a hard cut with
no aliases.

---

## Do Not Do

* Do not collapse `Create<E>` into `StructuralPatch`.
* Do not collapse `StructuralPatch` and `SerializedStructuralPatch`.
* Do not make SQL writes own their own patch representation.
* Do not move mode semantics into the patch type.
* Do not add storage-format differences for insert/update/replace.
* Do not make patch serialization own generated/managed-field policy.
* Do not force typed full-entity saves through dense serialized patches unless
  a concrete duplication removal justifies it.

---

## Bottom Line

`0.136` should close the write-boundary story around `StructuralPatch`.

The current architecture is mostly right:

* create keeps typed authorship provenance
* structural writes use `StructuralPatch`
* SQL writes lower into the structural patch lane
* persisted-row owns the serialized slot-payload form
* executor converges on `MutationInput`

The cleanup is to make the names, origin policy, and handoff boundaries explicit
so future work does not revive generated `Update` DTOs or fork SQL/fluent write
validation again.
