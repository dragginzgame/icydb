# 0.96 Execution Convergence

## Thesis

`0.96` should complete the transition from **planner-owned expression semantics** to **compiled execution across all remaining expression-bearing runtime paths**, removing the last interpreter-heavy execution lane and aligning global aggregate execution with the compiled model already used elsewhere.

> **0.96 finishes execution convergence by ensuring planner-owned `Expr` is never evaluated directly at runtime, but is always compiled into the appropriate execution form before evaluation.**

This slice is **not** about adding SQL surface area. It is the execution follow-through required after the expression, filter, HAVING, and ordering unification work landed in `0.85–0.95`.

---

## Problem statement

After `0.95`, the system is now largely converged on:

* `SqlExpr` as parser-owned syntax
* `Expr` as planner-owned semantics
* shared lowering onto planner `Expr`
* compiled execution in most major runtime lanes

The system already follows the right pattern in several places:

* `WHERE` compiles to `PredicateProgram`
* scalar projection compiles to `ScalarProjectionExpr`
* expression `ORDER BY` compiles through the scalar projection seam
* grouped projection compiles to `GroupedProjectionExpr`
* grouped aggregate input/filter expressions already compile onto shared seams

However, one meaningful runtime lane still lags behind:

* the dedicated **global aggregate SQL execution path** still recursively interprets planner `Expr` for:

  * post-aggregate global `HAVING`
  * post-aggregate global output projection

This means the architecture is now **semantically unified**, but not yet **fully execution-unified**.

That leaves:

* one remaining interpreter-heavy path
* duplicated execution shape
* avoidable runtime inconsistency
* a risk that future features widen the uncompiled lane before it is removed

---

## Design goal

Complete execution convergence by:

* removing remaining runtime evaluation of raw planner `Expr`
* compiling global post-aggregate projection expressions before execution
* compiling global post-aggregate `HAVING` through the same post-aggregate compiled seam
* reusing existing compiled-expression patterns rather than inventing a new general VM

The goal is:

> **one semantic expression owner (`Expr`), many specialized compiled execution forms, and no remaining raw-`Expr` runtime evaluation path.**

---

## Non-goals

This slice does **not** aim to:

* add new SQL features
* widen `HAVING` alias support
* add `DISTINCT`
* add `JOIN`
* add subqueries
* add window functions
* introduce a generic bytecode VM
* redesign the planner IR
* refactor the full compiled-expression ecosystem into one abstraction layer

This is an **execution convergence** slice, not a feature slice.

---

## Why now

`0.85–0.95` established the semantic model:

* planner-owned `Expr`
* phase-based meaning
* clause convergence onto shared expression lowering
* boolean/filter convergence
* HAVING convergence
* removal of public predicate-first filtering
* expression-owned ordering and projection

At this point, adding more SQL surface first would likely cause repeated rework:

* new features would either have to land on interpreter paths temporarily
* or would require ad hoc compiled treatment before the core execution story is finished

So `0.96` is the correct stabilization slice:

> **finish execution convergence first, then expand SQL surface on top of the stable compiled core.**

---

# Part I — Current model

## Semantics

The system now converges on planner `Expr` for user-facing expression surfaces:

* `WHERE`
* `HAVING`
* scalar projection
* expression `ORDER BY`
* fluent filter input

That semantic convergence is already the right end state.

## Execution

The execution model is mostly compiled already:

* `WHERE` -> `PredicateProgram`
* scalar projection -> `ScalarProjectionExpr`
* expression ordering -> compiled scalar/order seam
* grouped projection/HAVING -> grouped compiled forms
* grouped aggregate input/filter -> compiled expression seams

## Remaining gap

The remaining gap is:

* **global post-aggregate projection**
* **global post-aggregate HAVING**

These still evaluate planner `Expr` more directly than the rest of the system.

---

# Part II — Target model

## Before

```text
SqlExpr / Fluent filter / HAVING / Projection / ORDER BY
  -> lower to Expr
  -> some paths compile to specialized execution form
  -> global aggregate SQL lane still interprets Expr directly
```

## After

```text
SqlExpr / Fluent filter / HAVING / Projection / ORDER BY
  -> lower to Expr
  -> compile to specialized execution form
  -> runtime evaluates only compiled forms
```

No remaining runtime lane should recursively interpret raw planner `Expr`.

---

# Part III — Core design

## 1. `Expr` remains the semantic IR, not the runtime IR

`Expr` continues to own:

* semantic meaning
* clause/phase legality
* planner-visible structure
* fingerprint/caching identity

But `Expr` should **not** be executed directly at runtime.

### Explicit rule

> Planner `Expr` is a semantic/planning representation only. Runtime execution must consume compiled execution forms.

---

## 2. Global post-aggregate projection gets a compiled execution form

Global aggregate output expressions should compile into a dedicated **post-aggregate compiled projection form** rather than being evaluated recursively from raw `Expr`.

That compiled form should:

* operate over the global aggregate output context
* support the same post-aggregate expression surface currently admitted
* follow the same compile-once/evaluate-once-per-result pattern as existing compiled projection seams

This can be:

* a dedicated global post-aggregate compiled expression type
* or a reuse/widening of an existing post-aggregate-compatible compiled projection seam

The important part is the contract, not the exact name.

### Contract

> Global post-aggregate projection must be compiled before execution and must not recursively evaluate planner `Expr` at runtime.

---

## 3. Global `HAVING` reuses the same compiled post-aggregate seam

Global aggregate `HAVING` should compile through the same post-aggregate compiled seam as global projection, rather than owning a separate evaluator.

That means:

* same expression ownership
* same compiled representation family
* same post-aggregate evaluation context
* same boolean boundary semantics as elsewhere

### Execution shape

```text
global aggregate result
  -> evaluate compiled post-aggregate HAVING
  -> if TRUE, evaluate compiled post-aggregate projection
  -> else, emit no row
```

Or, if projection and `HAVING` are both needed, both should still be evaluated from the same compiled seam family.

### Explicit rule

> Global `HAVING` must not introduce a dedicated clause-specific evaluator.

---

## 4. Reuse the existing compiled-expression pattern

`0.96` should follow the pattern already established in the codebase, not replace it.

Existing compiled forms already show the intended architecture:

* `PredicateProgram`
* `ScalarProjectionExpr`
* grouped compiled projection forms
* compiled expression-owned ordering seams

So `0.96` should extend that model, not invent a new one.

### Explicit rule

> `0.96` must complete specialized compiled execution reuse, not introduce a generic expression VM or bytecode layer.

---

## 5. No raw-`Expr` recursion in executor paths

After `0.96`, no runtime executor path should recursively walk planner `Expr` to produce runtime values or boolean decisions.

This is the key invariant.

### Explicit rule

> Runtime expression evaluation from raw planner `Expr` is no longer allowed after compilation/validation completes.

That includes:

* boolean admission decisions
* output projection shaping
* post-aggregate filtering

---

# Part IV — Evaluation context contracts

## 1. Pre-aggregate boolean evaluation

Already compiled through predicate runtime. No design change here.

## 2. Scalar row projection evaluation

Already compiled through scalar projection seams. No design change here.

## 3. Grouped post-aggregate evaluation

Already compiled through grouped seams. No design change here.

## 4. Global post-aggregate evaluation

This is the `0.96` target.

Global post-aggregate execution must evaluate against a finalized aggregate result context containing the aggregate outputs needed by:

* projection
* `HAVING`
* any post-aggregate expression references admitted by the planner

### Explicit rule

> Global post-aggregate compiled expressions operate over the finalized global aggregate output context, not raw source rows.

---

# Part V — Validation and planning boundaries

`0.96` does **not** move validation later.

Validation remains where it belongs:

* parser/lowering validation for SQL syntax and shape
* planner/type validation for expression legality
* planning-time compilation into runtime forms

The key point is:

> After validation and planning complete, runtime execution should not still be interpreting semantic structure.

### Explicit rule

> Compilation is total for admitted global post-aggregate expressions: no runtime semantic interpretation remains.

---

# Part VI — Explain / fingerprint / cache identity

This slice should **not** change semantic identity.

`Expr` remains the owner of:

* explain structure
* query fingerprint structure
* cache identity

Compiled forms remain execution artifacts only.

That means:

* no new query identity model
* no compiled-form-owned fingerprint
* no feature-visible explain changes beyond possible implementation details

### Explicit rule

> `0.96` changes execution form, not semantic query identity.

---

# Part VII — Acceptance criteria

## Functional

Existing admitted queries must continue to work, including:

* global aggregate projection
* global aggregate `HAVING`
* mixed post-aggregate expression shapes already supported
* parity between equivalent SQL and fluent semantic intent where applicable

## Architectural

* no remaining executor path recursively evaluates raw planner `Expr`
* global post-aggregate projection is compiled before runtime
* global post-aggregate `HAVING` is compiled before runtime
* no clause-specific global `HAVING` evaluator exists
* no new generic bytecode/VM layer is introduced

## Behavioral

* boolean boundary semantics remain unchanged:

  * `TRUE` admits
  * `FALSE` rejects
  * `NULL` rejects
* post-aggregate projection semantics remain unchanged
* explain/fingerprint/cache identity remain planner-`Expr` owned

## Testing

Add or strengthen tests covering:

* global aggregate `HAVING` parity before/after compilation
* global aggregate projection with post-aggregate expressions
* SQL/fluent semantic parity for equivalent admitted shapes
* no-regression coverage for null behavior and numeric coercion in post-aggregate execution
* fail-closed behavior for unsupported shapes

# Part IX — Risks and boundaries

## Risk 1 — over-generalizing too early

The biggest design risk is introducing a generic VM/bytecode layer before the existing compiled seams have actually reached diminishing returns.

This slice should avoid that.

## Risk 2 — clause-specific reimplementation

It would be easy to fix global `HAVING` with a dedicated mini evaluator. That would be the wrong move.

The compiled seam must be shared with the corresponding post-aggregate projection family.

## Risk 3 — changing semantics accidentally

This slice must preserve:

* null boundary behavior
* numeric coercion behavior
* admitted expression surface
* fingerprint/explain identity

`0.96` is about **execution convergence**, not feature widening.

---

# Part X — Out of scope

The following should stay out of `0.96`:

* alias support widening in `HAVING`
* new scalar functions beyond incidental plumbing
* `DISTINCT`
* `JOIN`
* subqueries
* windows
* query result caching redesign
* major planner IR redesign
* generic bytecode VM

---

# Part XI — Follow-on after 0.96

Once `0.96` lands, the system should have:

* one semantic expression owner
* one consistent “compile before execute” model
* no remaining raw semantic-expression interpreter path at runtime

That unlocks safer future work in:

* scalar function expansion
* baseline query performance work
* additional single-table SQL features
* later reconsideration of whether compiled seams are repetitive enough to justify a more generic execution abstraction

But that future abstraction question should be asked **after** execution convergence is complete, not before.

---

## Thesis sentence

> **0.96 completes execution convergence by removing the last interpreter-heavy runtime expression path, compiling global post-aggregate projection and `HAVING` onto the existing compiled expression model, and ensuring planner-owned `Expr` is never evaluated directly at runtime.**
