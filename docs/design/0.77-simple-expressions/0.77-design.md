# 📄 0.77 Design: Simple SQL Expressions (Predicate Promotion)

## Preamble

* **scope**: introduce simple expression support in SQL predicates (WHERE)
* **goal**: allow field-to-field and arithmetic comparisons in filtering
* **theme**: promote existing projection expression capability into filtering
* **non-goals**:

  * no JOIN support
  * no subqueries
  * no full SQL expression completeness
  * no planner algebra rewrites
  * no new execution paths

---

## Problem Statement

The system already supports expression evaluation in projection:

* field references
* arithmetic (`field + 1`)
* binary expressions (`Expr::Binary`)

However, filtering (WHERE) is restricted to:

```text
field op literal
```

This creates an inconsistent surface:

| Capability        | Projection | WHERE |
| ----------------- | ---------- | ----- |
| field + 1         | ✅          | ❌     |
| field_a > field_b | ✅ (expr)   | ❌     |

This is unintuitive and breaks surface coherence.

---

## Design Goals

1. **Surface coherence**

   * expressions valid in SELECT should be usable in WHERE

2. **Reuse existing engine**

   * no new expression evaluator
   * reuse projection expression runtime

3. **Single execution path**

   * no duplicate predicate flows
   * no new loops

4. **Planner honesty**

   * do not fake index usage for complex expressions

---

# Part 1: Feature Surface

## 1.1 Supported predicate expressions

### Field vs field

```sql
WHERE field_a > field_b
WHERE field_a = field_b
```

### Arithmetic vs field/literal

```sql
WHERE field + 1 > other_field
WHERE field + 1 > 10
```

### Expression vs expression

```sql
WHERE field_a + 1 > field_b - 2
```

---

## 1.2 Supported operators

* `=`
* `!=`
* `<`
* `<=`
* `>`
* `>=`

---

## 1.3 Supported expression types

* field references
* literals
* binary arithmetic:

  * `+`
  * `-`
  * `*`
  * `/` (only if already supported)
* parentheses (if parser already supports)

---

## 1.4 Explicit exclusions

* functions
* aggregates in WHERE
* string operations
* casts
* subqueries
* boolean expression trees beyond existing AND/OR

---

# Part 2: Internal Representation

## 2.1 New predicate form

Replace:

```text
Compare(field, op, value)
```

With:

```text
CompareExpr(left_expr, op, right_expr)
```

---

## 2.2 Unification rule

All predicates become expressions:

```text
field > 10
→ CompareExpr(Field(field), >, Literal(10))
```

---

## 2.3 Single predicate runtime

* remove separate field-vs-value path
* use expression evaluation for all predicates

---

# Part 3: Execution Model

## 3.1 Row evaluation

For each row:

```text
left = eval(left_expr)
right = eval(right_expr)
result = compare(left, right)
```

---

## 3.2 Execution constraints

Must preserve:

* single-pass row loop
* no second filtering pass
* no new kernel

---

## 3.3 Integration point

Expression predicates execute inside:

```text
execute_kernel_row_scan(...)
→ row filter stage
```

---

# Part 4: Type Semantics

## 4.1 Allowed comparisons

* numeric vs numeric
* compatible field vs field
* field vs literal (existing rules)

---

## 4.2 Rejections

* incompatible types (e.g. string vs number)
* invalid arithmetic combinations

---

## 4.3 Consistency rule

> predicate expressions follow the same type rules as projection expressions

---

# Part 5: Planner Behavior

## 5.1 Predicate classification

### Index-usable

```sql
field > 10
field = 5
```

→ existing planner paths

---

### Row-evaluated

```sql
field_a > field_b
field + 1 > 10
```

→ fallback to row filtering

---

## 5.2 No algebraic rewriting

Do NOT transform:

```sql
field + 1 > 10
```

into:

```sql
field > 9
```

That is a future optimization.

---

## 5.3 Planner invariants

* do not degrade existing index plans
* detect simple predicates early
* fallback cleanly for expressions

---

# Part 6: SQL Surface

## 6.1 Supported queries

```sql
SELECT * FROM users WHERE score > rank;
SELECT * FROM users WHERE score + 1 > rank;
SELECT * FROM users WHERE score + bonus > 100;
```

---

## 6.2 Explain behavior

Explain should indicate:

* index predicate vs
* row-evaluated predicate

---

# Part 7: Fluent API Alignment

## 7.1 Add expression predicates

Examples:

* `field("a").gt(field("b"))`
* `field("a").add(1).gt(field("b"))`

---

## 7.2 Shared lowering

Both SQL and fluent APIs must produce:

```text
CompareExpr(...)
```

# Part 10: Risks

## 10.1 Performance regression

If simple predicates go through slower path

→ **mitigation**: detect simple cases early

---

## 10.2 Duplicate predicate logic

→ **mitigation**: unify all predicate handling

---

## 10.3 Type divergence

→ **mitigation**: reuse projection evaluator
