# IcyDB 0.178 - Schema Migration / DDL Transition Admission

## Status

Active design.

0.178 defines how schema-changing DDL enters the accepted-schema transition
pipeline after 0.176 catalog authority and 0.177 explicit schema-version
contracts.

This line is about transition admission. It should make DDL state the expected
accepted schema version, the next declared schema version, and the intended
shape change before schema-owned mutation planning or physical work can run.

## Thesis

DDL should publish explicit schema transitions against accepted schema
authority.

DDL is not schema authority. DDL is an authoring frontend that must bind to the
current accepted catalog, declare the version contract it expects to advance,
lower into catalog-native mutation requests, and publish only through
schema-owned admission, runner, validation, invalidation, and snapshot
publication.

Expected shape:

```text
DDL text
  -> parsed transition intent
  -> accepted-before catalog binding
  -> EXPECT SCHEMA VERSION check
  -> SET SCHEMA VERSION check
  -> derive accepted-after candidate
  -> version/fingerprint admission
  -> mutation compatibility and physical-work admission
  -> execute / publish / fail closed
```

## Why This Is 0.178

0.176 says where accepted schema authority lives and how query paths consume
it cheaply.

0.177 says what accepted schema identity means:

```text
entity_tag / entity_path / store_path
schema_version
schema_fingerprint
```

0.178 is where DDL uses that identity. It should not ride along with the first
schema-authority cleanup because migration complexity expands quickly:

- nullable columns;
- defaults;
- generated fields;
- managed fields;
- index backfill;
- relation validation;
- journaled schema replay;
- query-plan invalidation;
- old row decode;
- new row write;
- rollback and fail-closed behavior.

Those concerns need one explicit admission boundary rather than scattered
special cases in parser, planner, executor, or storage code.

## Non-Goals

0.178 must not add:

- a broad migration framework;
- online row rewrites;
- multi-statement migration transactions;
- automatic schema-version increments;
- version-gap imports;
- generated-model fallback authority;
- DDL-specific schema-store publication paths;
- DDL-specific index rebuild paths;
- compatibility readers for obsolete pre-1.0 schema formats;
- query-planner ownership of schema transition policy.

0.178 may reserve syntax for later transition classes, but unsupported classes
must fail before physical work, schema-store writes, runtime epoch changes, or
visible accepted-schema changes.

## Proposed Syntax Direction

Exact syntax can adapt to IcyDB parser conventions, but mutating DDL should
carry both expected and next schema versions.

Example:

```sql
ALTER TABLE Character
  EXPECT SCHEMA VERSION 3
  SET SCHEMA VERSION 4
  ADD COLUMN nickname text;
```

Equivalent parser shapes are acceptable if they preserve the contract:

```sql
ALTER TABLE Character
  ADD COLUMN nickname text
  EXPECT SCHEMA VERSION 3
  SET SCHEMA VERSION 4;
```

or:

```sql
ALTER TABLE Character
  FROM SCHEMA VERSION 3
  TO SCHEMA VERSION 4
  ADD COLUMN nickname text;
```

The contract matters more than keyword choice:

- expected current version is explicit;
- next version is explicit;
- shape transition is explicit;
- parser preserves these fields as transition intent, not diagnostics.

## Admission Packet

DDL lowering should produce a schema-owned admission packet before mutation
planning:

```text
SchemaDdlTransitionAdmission
  statement_kind
  target_entity
  target_store_path
  expected_schema_version
  next_schema_version
  accepted_before_identity
  accepted_after_candidate
  candidate_fingerprint
  shape_delta
  source_span / statement text for diagnostics
```

The packet is not a mutation plan. It is the evidence needed to decide whether
the DDL statement is eligible to become one.

Parser code should not construct accepted snapshots directly. The accepted
after-candidate must be derived by schema-owned lowering against the
accepted-before snapshot.

## Admission Rules

DDL should fail before mutation planning or physical work if:

- the accepted schema version is not the expected version;
- the next version is not valid;
- the next version is not exactly `expected + 1` for a shape-changing
  operation;
- the operation changes shape without a version bump;
- the operation is a no-op but bumps version;
- the candidate fingerprint is unchanged when a shape-changing transition is
  claimed;
- the candidate fingerprint changes while the version is unchanged;
- entity identity does not match the accepted catalog binding;
- the DDL operation cannot lower to a catalog-native mutation request;
- the mutation requires unsupported physical work;
- validation cannot prove old rows remain decodable and new writes are
  well-defined.

For mutating DDL, the default version rule is:

```text
accepted version = N
next version = N + 1
candidate fingerprint != accepted fingerprint
```

For no-op DDL, such as `IF EXISTS` / `IF NOT EXISTS` cases that truly do not
change accepted shape:

```text
accepted version = N
no accepted-after publication
no version bump
candidate fingerprint = accepted fingerprint
```

A no-op should report no publication. It must not mint a new schema version.

## Version/Fingerprint Matrix

0.178 consumes the 0.177 matrix at the DDL boundary:

| Accepted Version | Accepted Fingerprint | Expected Version | Next Version | Candidate Fingerprint | Result |
| --- | --- | --- | --- | --- | --- |
| `N` | `F` | `N` | absent | `F` | no-op allowed only if command is no-op |
| `N` | `F` | not `N` | any | any | reject stale DDL expectation |
| `N` | `F` | `N` | `N` | different | reject shape change without version bump |
| `N` | `F` | `N` | `N + 1` | `F` | reject empty bump |
| `N` | `F` | `N` | `N + 1` | different | candidate transition |
| `N` | `F` | `N` | `> N + 1` | different | reject version gap |
| `N` | `F` | `N` | `< N` | any | reject rollback |

After this matrix admits a candidate transition, schema mutation compatibility
rules still decide whether the transition can publish, requires physical work,
or remains unsupported.

## DDL Responsibilities

DDL owns:

- source syntax for transition intent;
- author-declared expected and next schema versions;
- accepted catalog binding diagnostics;
- source-span-aware parse and bind errors;
- reporting the mutation/admission/publish result to the caller.

DDL does not own:

- accepted schema authority;
- field ID or slot allocation policy;
- persisted row decode policy;
- default encoding semantics;
- index rebuild semantics;
- relation validation semantics;
- runtime invalidation semantics;
- schema-store publication;
- query cache invalidation;
- generated schema reconciliation.

## Schema-Owned Responsibilities

Schema-owned mutation code owns:

- deriving accepted-after snapshots from accepted-before snapshots;
- assigning durable field IDs and physical slots;
- preserving retired field IDs and slots;
- validating default payloads through accepted field codecs;
- classifying mutation deltas;
- computing candidate fingerprints;
- enforcing version/fingerprint admission;
- producing `SchemaMutationRequest`, mutation plans, and execution plans;
- admitting or rejecting required physical work;
- publishing accepted-after snapshots only after validation succeeds.

The DDL frontend must use these surfaces. It must not special-case its own
schema mutation semantics.

## Transition Classes

### Metadata-Only

Examples:

- rename DDL-owned column;
- change nullable/default metadata where old-row materialization remains
  valid;
- drop DDL-owned secondary index when no physical cleanup risk remains;
- add metadata-only accepted facts that do not affect row bytes.

Requirements:

- explicit expected and next schema versions;
- changed accepted fingerprint;
- no old-row decode ambiguity;
- no unsupported generated/DDL ownership collision;
- cache/cursor invalidation through accepted schema identity.

### Decode-Safe Additive Field

Examples:

- nullable no-default column;
- required column with one accepted encoded default payload.

Requirements:

- explicit expected and next schema versions;
- durable field ID and slot allocation from schema-owned policy;
- accepted absence policy for old rows;
- accepted write policy for new rows;
- default payload validation through accepted field codecs;
- generated/managed field ownership rules preserved.

### Physical-Work Required

Examples:

- secondary index add requiring backfill;
- unique index add requiring duplicate validation;
- relation validation;
- `SET NOT NULL` requiring accepted-row scan;
- field drop requiring dependent index cleanup;
- expression-index rebuild.

Requirements:

- explicit expected and next schema versions;
- mutation execution plan;
- runner preflight;
- physical store validation before publication;
- runtime invalidation;
- fail-closed rollback behavior if validation fails.

### Unsupported

Examples:

- type change without rewrite policy;
- primary-key mutation;
- generated field mutation through SQL DDL;
- managed field mutation without explicit management policy;
- relation target migration;
- full row rewrite before a runner exists;
- multi-entity transition without transaction semantics.

Unsupported transitions reject before accepted-after publication and before any
partial physical work that cannot be rolled back or ignored safely.

## Generated And Managed Fields

DDL must respect field ownership.

Generated fields are Rust-schema owned. SQL DDL must not mutate them unless a
future explicit adoption or ownership-transfer command is designed.

Managed fields, such as generated IDs or timestamp-managed fields, require
write-policy preservation:

- DDL cannot silently create generated/managed fields;
- DDL cannot remove generated/managed behavior from generated fields;
- DDL-owned fields must persist explicit write-policy metadata if they ever
  support managed behavior;
- schema fingerprints must change when write-policy facts change.

## Defaults And Nullability

Defaults are database defaults encoded into accepted schema metadata. They are
not Rust struct defaults.

DDL admission must validate:

- default literal type against accepted field kind;
- deterministic payload encoding;
- non-null payload for required fields;
- old-row materialization behavior;
- new-row write behavior;
- default changes on DDL-owned fields only, unless a later generated-field
  adoption policy exists.

`SET NOT NULL` is not metadata-only unless accepted-row scanning proves no old
row materializes `NULL`.

## Index Backfill

DDL-created indexes are accepted index contracts. They become planning-visible
only after physical work is complete and validated.

Admission must distinguish:

- metadata candidate;
- rebuild-required candidate;
- runner-ready candidate;
- runner-missing-capability rejection;
- validation failure;
- publication success.

The planner must never see the accepted-after index as visible before the
mutation runner has validated physical index state and schema publication has
completed.

## Relations

Relation-changing DDL must remain unsupported until relation validation and
mutation semantics are explicit.

Before admission can publish relation changes, the design must answer:

- source and target accepted identity;
- strong vs weak relation enforcement;
- existing row validation;
- reverse index/backlink requirements;
- cross-entity publication atomicity;
- rollback behavior.

0.178 should keep relation DDL fail-closed unless the transition is explicitly
covered by schema-owned mutation rules and validation.

## Journaled Schema Replay

Journaled stores add one more admission constraint: schema transitions must be
replayable from accepted schema authority and marker-bound journal records.

DDL admission must not require query/recovery paths to reconstruct authority
from generated models or diagnostics. Journaled schema publication should
preserve:

- accepted-before identity;
- accepted-after identity;
- marker-bound schema mutation publication;
- recovery-time schema fingerprint validation;
- live projection invalidation;
- fold/recovery isolation from query hot paths.

If a DDL transition cannot be replayed or validated after interruption, it must
reject before publication.

## Query And Cursor Invalidation

Successful DDL publication changes accepted schema identity. Therefore it must
invalidate:

- shared query-plan cache entries through accepted schema fingerprint;
- compiled SQL cache entries through accepted schema fingerprint;
- continuation/cursor signatures that include accepted schema identity;
- visible-index projections;
- executor authority that carries accepted row/decode contracts.

Invalidation is an effect of accepted snapshot publication, not a DDL-specific
side channel.

## Execution And Rollback

DDL execution must be fail-closed:

- parse/bind/admission failure leaves accepted schema unchanged;
- unsupported mutation leaves accepted schema unchanged;
- failed physical work leaves accepted schema unchanged;
- failed validation leaves accepted schema unchanged;
- runtime invalidation failure rejects before publication;
- publication failure reports failure without pretending the transition is
  visible.

Physical work that occurs before publication must either be:

- validated and then published;
- ignored because accepted schema never points at it;
- cleaned up by an explicit cleanup path;
- recoverable through journaled replay/fold rules.

DDL must not publish accepted-after metadata first and then hope physical work
eventually catches up.

## Diagnostics

DDL transition diagnostics should include:

- statement kind;
- target entity name/path/tag;
- store path;
- expected schema version;
- accepted schema version;
- requested next schema version;
- accepted-before fingerprint;
- accepted-after candidate fingerprint;
- version/fingerprint admission result;
- mutation compatibility result;
- physical-work requirement;
- runner admission result;
- validation status;
- runtime invalidation status;
- publication status;
- no-op status when applicable.

Diagnostics must distinguish:

- stale expected version;
- invalid next version;
- missing version bump;
- empty version bump;
- version gap;
- unsupported transition;
- physical runner missing;
- validation failure.

Diagnostics render decisions. They do not provide schema authority back to DDL,
query planning, or execution.

## Relationship To Existing Lines

### 0.176

0.176 defines accepted catalog authority and cheap query consumption. 0.178
must consume that catalog identity for DDL binding, but it must not move DDL
policy into query hot paths.

### 0.177

0.177 defines explicit schema version contracts. 0.178 requires mutating DDL
to state those contracts.

### 0.152 / 0.153

0.152 and 0.153 define catalog-native mutation planning and physical runner
boundaries. 0.178 should lower DDL into those surfaces rather than creating a
parallel migration engine.

### 0.155 / 0.157 / 0.158

Earlier DDL lines proved concrete index and field DDL slices. 0.178 tightens
the admission contract around explicit expected/next schema versions. Once
implemented, versionless mutating DDL should reject or be migrated through a
deliberate pre-1.0 hard cut, not silently keep old admission semantics.

## Source-Audit Candidates

DDL must not publish schema directly:

```bash
rg -n "publish_accepted_snapshot|insert_persisted_snapshot|latest_persisted_snapshot|SchemaStore" \
  crates/icydb-core/src/db/session/sql \
  crates/icydb-core/src/db/sql
```

Expected:

- SQL DDL binds accepted catalog state;
- schema-owned mutation/publication code performs snapshot publication;
- parser and SQL session code do not write schema store state directly.

Versionless mutating DDL should be contained:

```bash
rg -n "ALTER TABLE|CREATE INDEX|DROP INDEX|SET SCHEMA VERSION|EXPECT SCHEMA VERSION|schema_version" \
  crates/icydb-core/src/db/session/sql \
  crates/icydb-core/src/db/sql
```

Expected:

- mutating DDL AST carries expected and next version fields;
- no mutating DDL path silently supplies the next version;
- no-op DDL does not bump version.

Physical work must remain schema-owned:

```bash
rg -n "SchemaMutationRunnerInput|SchemaMutationExecutionPlan|SchemaMutationRequest|SchemaMutationRunnerContract" \
  crates/icydb-core/src/db/session/sql \
  crates/icydb-core/src/db/schema
```

Expected:

- DDL lowers into schema mutation requests/plans;
- runner input is built from accepted before/after snapshots;
- SQL code does not invent physical rebuild or validation semantics.

## Proof Targets

0.178 should prove:

- mutating DDL requires expected current schema version;
- mutating DDL requires next schema version;
- stale expected version rejects before mutation planning;
- next version must be exactly expected `+ 1` for shape changes;
- shape-changing DDL without version bump rejects;
- no-op DDL with version bump rejects;
- unchanged fingerprint with bumped version rejects by default;
- changed fingerprint with unchanged version rejects;
- supported DDL lowers into catalog-native mutation requests;
- unsupported DDL rejects before physical work and publication;
- physical-work DDL publishes only after validation and runtime invalidation;
- successful DDL invalidates query/SQL/cursor identity through accepted schema
  fingerprint;
- generated fields, managed fields, relation changes, and unsupported rewrites
  fail closed unless explicitly admitted by schema-owned rules;
- journaled schema transitions are replayable or reject before publication.

## Closeout Standard

0.178 can close when:

- the DDL AST/binder carries expected and next schema versions for mutating
  transitions;
- schema-owned admission enforces the version/fingerprint matrix before
  mutation planning;
- no-op DDL cannot bump schema version;
- supported DDL paths lower through catalog-native mutation requests and
  runner admission;
- unsupported transitions fail before physical work or accepted snapshot
  publication;
- successful publication updates accepted schema identity and invalidates
  stale plans/cursors;
- source guards prove SQL DDL does not own schema-store writes, physical runner
  semantics, or generated-model fallback authority.

## Deferred Work

These remain outside 0.178 unless explicitly pulled into a later transition
class:

- multi-statement migration transactions;
- version-gap imports;
- online row rewrites;
- broad type-change migration;
- relation migration;
- generated-field adoption;
- managed-field DDL;
- public migration script files;
- rollback tooling beyond fail-closed admission/publication semantics.
