# IcyDB 0.178 - Schema Migration / DDL Transition Admission

## Status

Active design.

The design is good to proceed, but implementation should land in narrow slices.
The first slice is `0.178.1`: single-statement DDL transition admission over
existing accepted entities and already-supported schema-owned DDL transition
classes. Transition drafts, relation DDL, new index-backfill capability, online
row rewrites, journal replay work, entity create/drop/rename, and
generated/managed-field adoption remain later work unless a later slice
explicitly scopes them.

0.178 defines how schema-changing DDL enters the accepted-schema transition
pipeline after 0.176 catalog authority and 0.177 explicit schema-version
contracts.

This line is about transition admission. It should make DDL state the expected
accepted schema version, the next declared schema version, and the intended
accepted-schema-affecting transition before schema-owned mutation planning or
physical work can run.

## Thesis

DDL should publish explicit schema transitions against accepted schema
authority.

DDL is not schema authority. DDL is an authoring frontend that must bind to the
current accepted catalog, declare the version contract it expects to advance,
lower into catalog-native mutation requests, and publish only through
schema-owned admission, runner, validation, invalidation, and snapshot
publication.

Expected flow:

```text
DDL text
  -> parsed transition intent
  -> accepted-before catalog binding
  -> EXPECT SCHEMA VERSION check
  -> SET SCHEMA VERSION check
  -> derive accepted-after candidate
  -> version/fingerprint admission
  -> mutation compatibility and physical-work admission
  -> execute / publish / fail closed
```

## Schema-First IC Rationale

IcyDB is schema-first because an Internet Computer canister packages program,
storage, ABI, and upgrade lifecycle into one deployed Wasm unit. There is no
separate database service with an external catalog owner. The canister's Rust
schema macros can therefore declare entity shape, primary keys, indexes,
relations, validators, storage roles, and generated endpoints before deploy,
while accepted schema snapshots remain the runtime authority after deploy.

This model fits the IC environment because:

- stable-memory layouts and schema reconciliation need deterministic identity
  across upgrades;
- generated Rust and Candid-facing surfaces should stay typed instead of
  degrading into generic row bags;
- cycle cost, Wasm footprint, and instruction determinism reward compile-time
  schema knowledge;
- upgrade admission can compare generated schema intent or DDL transition
  intent against the accepted schema stored in stable memory;
- query planning, row decoding, defaults, and index visibility can consume one
  accepted catalog authority instead of reconstructing truth from generated
  models or SQL text.

Most modern databases make a different tradeoff. PostgreSQL, MySQL, and similar
systems are long-lived shared services whose catalogs outlive individual
application builds and serve many clients, languages, admin tools, replication
flows, and analytics jobs. There, making one Rust macro expansion the schema
authority would couple the database catalog to one application build and fight
normal operational workflows such as online DDL, DBA-managed migrations,
cross-service schema evolution, and external observability.

For IcyDB, macros are a proposal and convenience surface, DDL is an authoring
frontend, and accepted schema snapshots are runtime authority. 0.178 preserves
that split: DDL can author schema transitions, but it must publish only through
schema-owned admission and accepted snapshot publication.

## Why This Is 0.178

0.176 says where accepted schema authority lives and how query paths consume
it cheaply.

0.177 says what accepted schema identity means:

```text
entity_tag / entity_path / store_path
schema_version
schema_fingerprint
```

0.178 is where DDL uses that identity. It should not ride along with the first
schema-authority cleanup because migration complexity expands quickly:

- nullable columns;
- defaults;
- generated fields;
- managed fields;
- index backfill;
- relation validation;
- journaled schema replay;
- query-plan invalidation;
- old row decode;
- new row write;
- rollback and fail-closed behavior.

Those concerns need one explicit admission/publication boundary rather than
scattered special cases in parser, planner, executor, or storage code. The
authoring surface may still be multi-command as long as intermediate commands
only build an unpublished transition draft and do not mutate the accepted
schema, run physical work, or invalidate runtime state.

## Implementation Slicing

`0.178.1` should implement only single-statement mutating DDL admission and
publication with explicit:

```text
EXPECT SCHEMA VERSION N
SET SCHEMA VERSION N + 1
```

In scope:

- parse and bind expected and next schema versions;
- bind the accepted-before identity/header;
- derive the accepted-after candidate through schema-owned lowering;
- carry accepted and candidate fingerprint method versions as part of typed
  identity;
- compute candidate fingerprints only inside schema-owned admission;
- enforce the 0.177 version/fingerprint matrix before mutation compatibility;
- reject no-op version bumps;
- reject versionless mutating DDL;
- reject unsupported physical-work transitions before publication;
- recheck the live accepted identity at publication time before publishing;
- publish only through the schema-owned publication path;
- invalidate only through accepted snapshot publication.

`0.178.1` does not introduce new concrete transition classes. It tightens
admission for existing-entity SQL DDL transitions that already lower through
schema-owned mutation and publication paths from earlier field/index DDL lines:

- DDL-owned field addition, default, nullability, rename, and retained-slot drop
  transitions when schema-owned admission can prove old-row decode, new-write
  semantics, and ownership safety;
- supported DDL-owned secondary index add/drop transitions only through the
  existing schema-owned runner/admission/publication surfaces;
- true `IF EXISTS` / `IF NOT EXISTS` no-ops, which publish nothing and do not
  bump schema version.

All other catalog-affecting DDL must either bind as a true no-op and publish
nothing, or pass/fail the version gate and then reject before physical work and
accepted snapshot publication as an unsupported transition class.

Out of scope for `0.178.1`:

- transition drafts;
- relation DDL;
- new index-backfill capability beyond existing schema-owned runner support;
- online row rewrites;
- `CREATE TABLE`, `DROP TABLE`, entity rename, and other entity birth/death
  transitions;
- multi-entity catalog transitions;
- generated-field adoption;
- managed-field DDL;
- version-gap import;
- DDL-specific schema-store writes.

Entity birth/death does not fit the per-entity version matrix because the
accepted-before or accepted-after identity may be absent. Those transitions
need a separate catalog-level version/fingerprint design before they can enter
DDL admission.

## Non-Goals

0.178 must not add:

- a broad migration framework;
- online row rewrites;
- multi-statement physical migration transactions;
- automatic schema-version increments;
- version-gap imports;
- generated-model fallback authority;
- DDL-specific schema-store publication paths;
- DDL-specific index rebuild paths;
- compatibility readers for obsolete pre-1.0 schema formats;
- query-planner ownership of schema transition policy.

0.178 may reserve syntax for later transition classes, but unsupported classes
must fail before physical work, schema-store writes, runtime epoch changes, or
visible accepted-schema changes.

## Proposed Syntax Direction

Exact syntax can adapt to IcyDB parser conventions, but publication of mutating
DDL should carry both expected and next schema versions.

Single-statement publication remains the most compact form:

```sql
ALTER TABLE Character
  EXPECT SCHEMA VERSION 3
  SET SCHEMA VERSION 4
  ADD COLUMN nickname text;
```

Equivalent parser shapes are acceptable if they preserve the contract:

```sql
ALTER TABLE Character
  ADD COLUMN nickname text
  EXPECT SCHEMA VERSION 3
  SET SCHEMA VERSION 4;
```

or:

```sql
ALTER TABLE Character
  FROM SCHEMA VERSION 3
  TO SCHEMA VERSION 4
  ADD COLUMN nickname text;
```

Later `0.178.x` work may support an explicit unpublished draft flow so several
DDL edits can be authored independently and published as one schema transition:

```sql
ALTER TABLE Character
  BEGIN SCHEMA TRANSITION
  EXPECT SCHEMA VERSION 3;

ALTER TABLE Character
  ADD COLUMN nickname text;

ALTER TABLE Character
  ADD COLUMN title text DEFAULT 'wanderer';

ALTER TABLE Character
  ADD COLUMN reputation nat64 DEFAULT 0;

ALTER TABLE Character
  COMMIT SCHEMA TRANSITION
  EXPECT SCHEMA VERSION 3
  SET SCHEMA VERSION 4;
```

The CLI may eventually expose shorter sugar for that flow, but the
parser/binder should still model it as:

```text
begin draft at accepted schema version 3
append ADD COLUMN nickname text
append ADD COLUMN title text DEFAULT 'wanderer'
append ADD COLUMN reputation nat64 DEFAULT 0
commit draft as schema version 4, expecting accepted version 3
```

Intermediate draft commands must not publish a visible accepted-after snapshot.
`DESCRIBE`, query planning, writes, and `SHOW INDEXES` continue to observe the
accepted-before schema until the final commit publishes successfully.

For `0.178.1`, draft syntax should not be parsed as an executable DDL form. If
reserved draft keywords are recognized, they must produce an unsupported-feature
error before catalog binding, candidate derivation, mutation planning, physical
work, publication, or invalidation.

`EXPECT SCHEMA VERSION` and `SET SCHEMA VERSION` accept only unsigned integer
schema-version literals in `0.178.1`. They do not accept expressions,
parameters, identifiers, or runtime-evaluated values. Duplicate `EXPECT`
clauses reject. Duplicate `SET` clauses reject. Clause order may be flexible
only if the AST preserves at most one expected version, at most one next
version, and source spans or equivalent statement context for both.

The contract matters more than keyword choice:

- expected current version is explicit;
- next version is explicit;
- accepted-schema-affecting transition is explicit;
- multi-command drafts accumulate one accepted-schema-affecting transition
  rather than publishing partial accepted schemas;
- parser preserves these fields as transition intent, not diagnostics.

`SET SCHEMA VERSION 3` by itself should not mean "publish version 3". If a
short syntax with `SET 3` is admitted for the start of a draft, it must be an
alias for "begin a draft expecting accepted version 3", not a schema-store
write.

## Transition Terminology

0.178 uses these terms:

- **accepted-schema-affecting transition**: any transition whose
  accepted-after fingerprint differs from the accepted-before fingerprint.
  Version/fingerprint admission is about this category.
- **storage-shape transition**: a transition that changes durable row layout,
  field slots, index state, or physical validation/cleanup requirements.
- **metadata-only transition**: an accepted-schema-affecting transition that
  requires no physical row/index work before publication.
- **true no-op**: a syntactically DDL statement that binds against the accepted
  catalog and produces the same accepted fingerprint with no publication.

Avoid using "shape-changing" as admission shorthand. A rename, default change,
or index visibility change may not change row bytes, but it can still change
the accepted fingerprint and therefore require explicit version admission.

## Transition Drafts Appendix

Transition drafts are deferred from `0.178.1`.

When implemented, a transition draft is an authoring artifact, not schema
authority. The default lifecycle should be ephemeral, session-scoped, one active
draft per entity per session, not persisted across restart or upgrade, and
never visible to query, write, `DESCRIBE`, `SHOW INDEXES`, or generated
reconciliation paths.

Draft state should carry:

- target entity and store identity;
- expected accepted schema version;
- accepted-before identity, fingerprint, and fingerprint method version captured
  at draft start;
- ordered draft DDL steps;
- current accepted-after candidate derived by schema-owned lowering;
- accumulated accepted-schema delta;
- source spans or statement text for diagnostics.

Draft commands are admitted only when they bind to the same entity and the live
accepted schema still matches the draft's expected accepted-before identity.
If another schema publication advances the accepted schema before the draft
commits, the draft is stale and must reject before mutation planning or
physical work.

Draft edits may update only the candidate transition. They must not:

- write the schema store;
- allocate accepted-visible field IDs or slots outside schema-owned candidate
  derivation;
- run index backfill, row scans, or physical cleanup;
- invalidate runtime caches or cursors;
- make the candidate visible to query planning, writes, introspection, or
  generated schema reconciliation.

Commit is the first command that may lower the accumulated candidate into a
mutation request, run physical-work admission, validate rows/indexes, publish an
accepted-after snapshot, and invalidate runtime state. Abort or replacement of
an unpublished draft should discard only draft metadata.

A draft commit with no effective accepted-schema change is an empty bump and
must reject. This includes an empty draft, a draft containing only commands that
bind as true no-ops, or a draft whose accumulated steps cancel out to the same
accepted fingerprint.

## Admission Packet

DDL lowering should produce a schema-owned admission packet before mutation
planning:

```text
SchemaDdlTransitionAdmission
  statement_kind
  target_entity
  target_store_path
  expected_schema_version
  next_schema_version
  accepted_before_identity
  accepted_before_fingerprint_method_version
  accepted_after_candidate
  candidate_fingerprint_method_version
  candidate_fingerprint
  accepted_schema_delta
  transition_steps
  source_span / statement text(s) for diagnostics
```

The packet is not a mutation plan. It is the evidence needed to decide whether
the DDL statement is eligible to become one.

The fingerprint method version fields may be embedded in typed identity or
fingerprint wrapper types. The invariant is that DDL and query paths must never
compare raw fingerprint values without method-version awareness. A method
version mismatch in `0.178.1` rejects before the normal DDL matrix unless
schema-owned code provides an explicit same-domain normalization function.
DDL/query paths must not choose between hard-cut, recompute, or reject behavior
and must not bridge method versions themselves.

Parser code should not construct accepted snapshots directly. The accepted
after-candidate must be derived by schema-owned lowering against the
accepted-before snapshot.

For transition drafts, this packet is produced at commit time from the ordered
draft steps. Draft edit commands may ask schema-owned code to refresh the
candidate for diagnostics, but they must not become mutation plans.

## Admission Rules

DDL should fail before mutation planning or physical work if:

- a DDL form that can mutate omits `EXPECT SCHEMA VERSION`;
- an accepted-schema-affecting DDL transition omits `SET SCHEMA VERSION`;
- the accepted schema version is not the expected version;
- accepted and candidate fingerprints use incompatible method versions;
- the next version is not valid;
- the next version is not exactly `expected + 1` for an accepted-schema-affecting
  operation;
- the operation changes accepted schema without a version bump;
- the operation is a no-op but bumps version;
- the candidate fingerprint is unchanged when an accepted-schema-affecting
  transition is claimed;
- the candidate fingerprint changes while the version is unchanged;
- entity identity does not match the accepted catalog binding;
- the DDL operation cannot lower to a catalog-native mutation request;
- the mutation requires unsupported physical work;
- validation cannot prove old rows remain decodable and new writes are
  well-defined.

For mutating DDL, the default version rule is:

```text
accepted version = N
next version = N + 1
candidate fingerprint != accepted fingerprint
```

For a transition draft, that rule applies to the final committed candidate, not
to each draft edit. Three `ADD COLUMN` draft edits from version `N` can publish
as one transition to `N + 1` if the final candidate fingerprint differs from
the accepted-before fingerprint and the whole accumulated transition is
admitted.

If the final draft candidate fingerprint still equals the accepted-before
fingerprint, `SET SCHEMA VERSION N + 1` rejects as an empty version bump.

For no-op DDL, such as `IF EXISTS` / `IF NOT EXISTS` cases that truly do not
change accepted schema:

```text
accepted version = N
no accepted-after publication
no version bump
candidate fingerprint = accepted fingerprint
```

A no-op should report no publication. It must not mint a new schema version.
If a no-op statement supplies `SET SCHEMA VERSION` and the candidate
fingerprint is unchanged, the statement rejects as an empty bump even when the
syntax was explicitly tolerant, such as `IF EXISTS` or `IF NOT EXISTS`.

Any DDL form that can mutate must still declare `EXPECT SCHEMA VERSION` even
when the specific execution binds as a true no-op. True no-ops do not require
`SET SCHEMA VERSION`.

## Version/Fingerprint Matrix

0.178 consumes the 0.177 matrix at the DDL boundary. The table assumes accepted
and candidate fingerprints have already been proven to use the same fingerprint
method version or equivalent typed identity domain.

| Accepted Version | Accepted Fingerprint | Expected Version | Next Version | Candidate Fingerprint | Result |
| --- | --- | --- | --- | --- | --- |
| `N` | `F` | absent | any | any | reject missing expected schema version for any DDL form that can mutate |
| `N` | `F` | not `N` | any | any | reject stale DDL expectation |
| `N` | `F` | `N` | absent | `F` | no-op allowed only if command is true no-op |
| `N` | `F` | `N` | absent | different | reject missing next schema version |
| `N` | `F` | `N` | `< N` | any | reject rollback |
| `N` | `F` | `N` | `N` | different | reject accepted-schema change without version bump |
| `N` | `F` | `N` | `N` | `F` | no-op allowed only if command is true no-op and no publication occurs |
| `N` | `F` | `N` | `N + 1` | `F` | reject empty bump |
| `N` | `F` | `N` | `N + 1` | different | candidate transition |
| `N` | `F` | `N` | `> N + 1` | any | reject version gap |

After this matrix admits a candidate transition, schema mutation compatibility
rules still decide whether the transition can publish, requires physical work,
or remains unsupported.

For transition drafts, the matrix is evaluated at commit. Intermediate draft
edits are not no-ops or version bumps from the accepted catalog's perspective
because they do not publish anything.
Only the committed candidate can justify the next version, and unchanged
candidate fingerprints never justify a new schema version.

## Publication-Time Identity Recheck

Single-statement DDL publication is conditional on the live accepted schema
identity still matching the accepted-before identity captured during binding.
Schema-owned publication must compare the live accepted identity against that
captured identity inside the publication critical section. The identity includes
entity identity, store path, accepted schema version, fingerprint method
version, and accepted fingerprint.

If another statement publishes first and changes the live identity, publication
rejects as stale DDL / publication race lost and leaves accepted schema
unchanged. This rule applies to single-statement DDL in `0.178.1`; later draft
commit logic must use the same recheck before publication.

## DDL Responsibilities

DDL owns:

- source syntax for transition intent;
- source syntax for beginning, editing, committing, and aborting unpublished
  transition drafts if a later slice implements them;
- author-declared expected and next schema versions;
- accepted catalog binding diagnostics;
- source-span-aware parse and bind errors;
- reporting the mutation/admission/publish result to the caller.

DDL does not own:

- accepted schema authority;
- field ID or slot allocation policy;
- persisted row decode policy;
- default encoding semantics;
- index rebuild semantics;
- relation validation semantics;
- runtime invalidation semantics;
- schema-store publication;
- query cache invalidation;
- generated schema reconciliation.

## Schema-Owned Responsibilities

Schema-owned mutation code owns:

- deriving accepted-after snapshots from accepted-before snapshots;
- assigning durable field IDs and physical slots;
- preserving retired field IDs and slots;
- validating default payloads through accepted field codecs;
- classifying mutation deltas;
- computing candidate fingerprints;
- proving candidate fingerprint method compatibility;
- enforcing version/fingerprint admission;
- producing `SchemaMutationRequest`, mutation plans, and execution plans;
- admitting or rejecting required physical work;
- publishing accepted-after snapshots only after validation succeeds.
- rechecking captured accepted-before identity against live accepted identity
  inside publication before the accepted-after snapshot becomes visible.

The DDL frontend must use these surfaces. It must not special-case its own
schema mutation semantics.

## Transition Classes

### Metadata-Only

Examples:

- rename DDL-owned column;
- change nullable/default metadata where old-row materialization remains
  valid;
- drop DDL-owned secondary index when no physical cleanup risk remains;
- add metadata-only accepted facts that do not affect row bytes.

Requirements:

- explicit expected and next schema versions;
- changed accepted fingerprint;
- no old-row decode ambiguity;
- no unsupported generated/DDL ownership collision;
- cache/cursor invalidation through accepted schema identity.

### Decode-Safe Additive Field

Examples:

- nullable no-default column;
- required column with one accepted encoded default payload.

Requirements:

- explicit expected and next schema versions;
- durable field ID and slot allocation from schema-owned policy;
- accepted absence policy for old rows;
- accepted write policy for new rows;
- default payload validation through accepted field codecs;
- generated/managed field ownership rules preserved.

### Physical-Work Required

Examples:

- secondary index add requiring backfill;
- unique index add requiring duplicate validation;
- relation validation;
- `SET NOT NULL` requiring accepted-row scan;
- field drop requiring dependent index cleanup;
- expression-index rebuild.

Requirements:

- explicit expected and next schema versions;
- mutation execution plan;
- runner preflight;
- physical store validation before publication;
- runtime invalidation;
- fail-closed rollback behavior if validation fails.

### Unsupported

Examples:

- type change without rewrite policy;
- primary-key mutation;
- generated field mutation through SQL DDL;
- managed field mutation without explicit management policy;
- relation target migration;
- full row rewrite before a runner exists;
- multi-entity transition without transaction semantics.

Unsupported transitions reject before accepted-after publication and before any
partial physical work that cannot be rolled back or ignored safely.

## Generated And Managed Fields

DDL must respect accepted schema fact ownership.

Accepted schema facts should carry ownership metadata where the fact can be
authored by more than one surface:

- generated-owned;
- DDL-owned;
- managed/generated-policy-owned when applicable.

Generated reconciliation must not erase, reinterpret, or reject DDL-owned
accepted fields or indexes merely because they are absent from the Rust
generated model. Conversely, DDL must not mutate generated-owned facts without
a future explicit adoption or ownership-transfer design.

Generated fields remain Rust-schema owned unless a future explicit adoption or
ownership-transfer command is designed.

If accepted schema facts lack ownership metadata required to classify a DDL
mutation, DDL must treat those facts as not DDL-owned and reject unless a
separate adoption or hard-cut policy has assigned ownership explicitly.

Managed fields, such as generated IDs or timestamp-managed fields, require
write-policy preservation:

- DDL cannot silently create generated/managed fields;
- DDL cannot remove generated/managed behavior from generated fields;
- DDL-owned fields must persist explicit write-policy metadata if they ever
  support managed behavior;
- schema fingerprints must change when write-policy facts change.

## Defaults And Nullability

Defaults are database defaults encoded into accepted schema metadata. They are
not Rust struct defaults.

DDL admission must validate:

- default literal type against accepted field kind;
- deterministic payload encoding;
- non-null payload for required fields;
- old-row materialization behavior;
- new-row write behavior;
- default changes on DDL-owned fields only, unless a later generated-field
  adoption policy exists.

`SET NOT NULL` is not metadata-only unless accepted-row scanning proves no old
row materializes `NULL`.

## Index Backfill

DDL-created indexes are accepted index contracts. They become planning-visible
only after physical work is complete and validated.

Admission must distinguish:

- metadata candidate;
- rebuild-required candidate;
- runner-ready candidate;
- runner-missing-capability rejection;
- validation failure;
- publication success.

The planner must never see the accepted-after index as visible before the
mutation runner has validated physical index state and schema publication has
completed.

## Relations

Relation-changing DDL must remain unsupported until relation validation and
mutation semantics are explicit.

Before admission can publish relation changes, the design must answer:

- source and target accepted identity;
- strong vs weak relation enforcement;
- existing row validation;
- reverse index/backlink requirements;
- cross-entity publication atomicity;
- rollback behavior.

0.178 should keep relation DDL fail-closed unless the transition is explicitly
covered by schema-owned mutation rules and validation.

## Journaled Schema Replay

Journaled stores add one more admission constraint: schema transitions must be
replayable from accepted schema authority and marker-bound journal records.

DDL admission must not require query/recovery paths to reconstruct authority
from generated models or diagnostics. Journaled schema publication should
preserve:

- accepted-before identity;
- accepted-after identity;
- marker-bound schema mutation publication;
- recovery-time schema fingerprint validation;
- live projection invalidation;
- fold/recovery isolation from query hot paths.

If a DDL transition cannot be replayed or validated after interruption, it must
reject before publication.

## Query And Cursor Invalidation

Successful DDL publication changes accepted schema identity. Therefore it must
invalidate:

- shared query-plan cache entries through accepted schema fingerprint;
- compiled SQL cache entries through accepted schema fingerprint;
- continuation/cursor signatures that include accepted schema identity;
- visible-index projections;
- executor authority that carries accepted row/decode contracts.

Invalidation is an effect of accepted snapshot publication, not a DDL-specific
side channel.

Continuation/cursor signatures should include accepted schema identity.
Publication does not need to find and mutate every outstanding cursor. Stale
cursors reject on resume because their embedded identity no longer matches the
accepted schema.

Publication must be atomic with respect to accepted identity and runtime
invalidation. There must be no observable window where the accepted-after
schema is visible while old query plans, compiled SQL entries, cursors,
visible-index projections, or executor contracts remain valid. If invalidation
requires fallible preparation, preflight it before publication; the visible
identity swap and invalidation must occur as one publication step or fail
closed.

## Execution And Rollback

DDL execution must be fail-closed:

- parse/bind/admission failure leaves accepted schema unchanged;
- unsupported mutation leaves accepted schema unchanged;
- failed physical work leaves accepted schema unchanged;
- failed validation leaves accepted schema unchanged;
- runtime invalidation failure rejects before publication;
- publication failure reports failure without pretending the transition is
  visible.

Physical work that occurs before publication must either be:

- validated and then published;
- ignored because accepted schema never points at it;
- cleaned up by an explicit cleanup path;
- recoverable through journaled replay/fold rules.

DDL must not publish accepted-after metadata first and then hope physical work
eventually catches up.

## Diagnostics

DDL transition diagnostics should carry stable machine-readable variants. Human
messages may include entity names, store paths, versions, source spans, and
abbreviated fingerprints, but tests should assert variants rather than matching
rendered strings.

Minimum `0.178.1` variants:

```rust
enum SchemaDdlAdmissionError {
    MissingExpectedSchemaVersion,
    MissingNextSchemaVersion,
    StaleExpectedSchemaVersion,
    InvalidNextSchemaVersion,
    AcceptedSchemaChangeWithoutVersionBump,
    EmptyVersionBump,
    VersionGap,
    VersionRollback,
    FingerprintMethodMismatch,
    UnsupportedTransitionClass,
    PhysicalRunnerMissing,
    ValidationFailed,
    PublicationRaceLost,
}
```

Rendered DDL transition diagnostics should include:

- statement kind;
- target entity name/path/tag;
- store path;
- expected schema version;
- accepted schema version;
- requested next schema version;
- accepted-before fingerprint;
- accepted-before fingerprint method version;
- accepted-after candidate fingerprint;
- accepted-after candidate fingerprint method version;
- version/fingerprint admission result;
- mutation compatibility result;
- physical-work requirement;
- runner admission result;
- validation status;
- runtime invalidation status;
- publication status;
- no-op status when applicable.

Diagnostics must distinguish:

- missing expected schema version;
- missing next schema version;
- stale expected version;
- invalid next version;
- missing version bump;
- empty version bump;
- version gap;
- version rollback;
- fingerprint method mismatch;
- unsupported transition;
- physical runner missing;
- validation failure;
- publication race lost.

Diagnostics render decisions. They do not provide schema authority back to DDL,
query planning, or execution.

## Relationship To Existing Lines

### 0.176

0.176 defines accepted catalog authority and cheap query consumption. 0.178
must consume that catalog identity for DDL binding, but it must not move DDL
policy into query hot paths.

### 0.177

0.177 defines explicit schema version contracts. 0.178 requires mutating DDL
to state those contracts.

### 0.152 / 0.153

0.152 and 0.153 define catalog-native mutation planning and physical runner
boundaries. 0.178 should lower DDL into those surfaces rather than creating a
parallel migration engine.

### 0.155 / 0.157 / 0.158

Earlier DDL lines proved concrete index and field DDL slices. 0.178 tightens
the admission contract around explicit expected/next schema versions. Once
implemented, versionless mutating DDL should reject or be migrated through a
deliberate pre-1.0 hard cut, not silently keep old admission semantics.

## Source-Audit Candidates

DDL must not publish schema directly:

```bash
rg -n "publish_accepted_snapshot|insert_persisted_snapshot|latest_persisted_snapshot|SchemaStore" \
  crates/icydb-core/src/db/session/sql \
  crates/icydb-core/src/db/sql
```

Expected:

- SQL DDL binds accepted catalog state;
- schema-owned mutation/publication code performs snapshot publication;
- parser and SQL session code do not write schema store state directly.

Versionless mutating DDL should be contained:

```bash
rg -n "ALTER TABLE|CREATE INDEX|DROP INDEX|SET SCHEMA VERSION|EXPECT SCHEMA VERSION|schema_version" \
  crates/icydb-core/src/db/session/sql \
  crates/icydb-core/src/db/sql
```

Expected:

- mutating DDL publication AST carries expected and next version fields;
- draft-edit AST, if present, carries draft/entity binding but cannot silently
  publish;
- no mutating DDL path silently supplies the next version;
- no-op DDL does not bump version.

Physical work must remain schema-owned:

```bash
rg -n "SchemaMutationRunnerInput|SchemaMutationExecutionPlan|SchemaMutationRequest|SchemaMutationRunnerContract" \
  crates/icydb-core/src/db/session/sql \
  crates/icydb-core/src/db/schema
```

Expected:

- DDL lowers into schema mutation requests/plans;
- runner input is built from accepted before/after snapshots;
- SQL code does not invent physical rebuild or validation semantics.

## Proof Targets

`0.178.1` should prove:

Parser and binder coverage:

- mutating DDL without `EXPECT SCHEMA VERSION` rejects;
- mutating DDL without `SET SCHEMA VERSION` rejects;
- mutating DDL with `SET` but no `EXPECT` rejects;
- duplicate `EXPECT SCHEMA VERSION` rejects;
- duplicate `SET SCHEMA VERSION` rejects;
- non-literal schema-version syntax rejects if it reaches the parser;
- true no-op DDL forms still require `EXPECT SCHEMA VERSION`;

Admission matrix coverage:

- stale `EXPECT SCHEMA VERSION` rejects before mutation planning;
- `SET` to the same version with changed accepted schema rejects;
- `SET N + 1` with unchanged fingerprint rejects as an empty bump;
- `SET N + 2` with changed fingerprint rejects as a version gap;
- `SET N + 1` with changed fingerprint reaches mutation compatibility instead
  of auto-publishing;
- rollback rejects;
- raw fingerprint comparison is method-version-aware and rejects method-version
  mismatch before the DDL matrix unless schema-owned same-domain normalization
  is explicitly present;

Publication path coverage:

- publication rechecks the captured accepted-before identity against the live
  accepted identity inside the schema-owned publication boundary;
- publication race loss leaves accepted schema unchanged;
- unsupported transitions after a valid version gate reject before
  publication;
- SQL DDL code does not write the schema store directly;
- supported transitions publish only through schema-owned admission and
  publication;
- successful supported transitions invalidate caches through accepted schema
  identity;
- query paths do not compute DDL candidate fingerprints;
- publication and invalidation occur as one schema-owned operation;
- failed validation leaves accepted schema unchanged;

No-op coverage:

- no-op `IF EXISTS` / `IF NOT EXISTS` DDL with only `EXPECT` does not publish
  and does not bump;
- no-op `IF EXISTS` / `IF NOT EXISTS` DDL with `SET N + 1` rejects as an empty
  bump;
- `ADD COLUMN IF NOT EXISTS` or similar idempotent DDL, if supported, binds as
  a true no-op only when the accepted catalog already matches exactly;

Ownership and unsupported-transition coverage:

- generated fields, managed fields, relation changes, and unsupported rewrites
  fail closed unless explicitly admitted by schema-owned rules;
- missing ownership metadata required for DDL classification rejects as
  not DDL-owned unless an explicit adoption/hard-cut policy exists.

Later draft implementation should prove:

- draft edits do not publish accepted schema, run physical work, or invalidate
  runtime state;
- draft commits evaluate the version/fingerprint matrix once;
- stale drafts reject when accepted schema identity no longer matches the draft
  start identity;
- empty draft commits reject;
- cancelled drafts leave accepted schema unchanged.

## Closeout Standard

`0.178.1` can close when:

- the DDL AST/binder carries expected and next schema versions for mutating
  single-statement transitions;
- `0.178.1` applies only to existing accepted entities and explicitly defers
  entity birth/death, entity rename, and multi-entity catalog transitions;
- schema-owned admission enforces the version/fingerprint matrix before
  mutation compatibility;
- accepted and candidate fingerprint method versions are carried in typed
  identity and raw fingerprints are not compared without method awareness;
- fingerprint method mismatch rejects before the DDL matrix unless schema-owned
  same-domain normalization exists;
- no-op DDL cannot bump schema version;
- true no-op DDL forms require `EXPECT SCHEMA VERSION` but not `SET SCHEMA
  VERSION`;
- publication rechecks captured accepted-before identity against live accepted
  identity inside the schema-owned publication critical section;
- supported DDL paths lower through catalog-native mutation requests and
  runner admission;
- unsupported transitions fail before physical work or accepted snapshot
  publication;
- successful publication updates accepted schema identity and invalidates
  stale plans/cursors atomically, with cursor staleness enforced by embedded
  accepted identity on resume;
- diagnostics expose stable machine-readable error variants;
- source guards prove SQL DDL does not own schema-store writes, physical runner
  semantics, query identity, or generated-model fallback authority.

Draft-enabled closeout additionally requires:

- transition drafts can accumulate multiple edit commands and publish once
  through a commit command with explicit expected and next schema versions;
- draft commits cannot bump schema version unless the accumulated candidate
  changes the accepted schema fingerprint;
- draft lifecycle is explicit and session-scoped or intentionally more durable
  with restart/upgrade semantics.

- online row rewrites;
