# 0.207 Exact And Resumable Bulk UPDATE

Status: proposed; design cut on 2026-07-15; implementation starts after 0.206 closeout

## Context

IcyDB has three useful but currently separate write/read concepts:

- update_many_atomic stages a caller-provided single-entity batch and commits it
  all-or-nothing;
- public bounded SQL UPDATE admits an explicitly ordered LIMIT window and does
  not claim that every matching row was updated;
- partial_window and admin_batch provide bounded read windows, with ordinary
  cursor continuation defined over live state rather than a pinned snapshot.

An application operation such as Toko's reset_token_tiers falls between those
surfaces. Its semantic promise is complete:

> Every token in one collection must converge to the requested tier values.

Using partial_window(MAX_COLLECTION_EFFECT_ROWS) does not satisfy that promise.
It updates at most a prefix and has neither overflow rejection nor completion
continuation. Raising the constant only moves the correctness boundary.

Canister update calls are serialized, which makes complete target selection and
staging safe inside one message. Serialization does not remove:

- the update-call instruction limit;
- transient heap limits;
- commit marker and journal bounds;
- per-row index and relation work;
- RETURNING response limits;
- the need to recover an interrupted marker-owned batch.

Once work spans multiple update calls, other calls can run between batches.
Therefore IcyDB needs two explicit contracts instead of one cap with ambiguous
meaning:

1. bounded exact UPDATE: complete atomically or reject before mutation; and
2. resumable UPDATE: commit bounded atomic batches until a bounded verification
   sweep can prove convergence against an unchanged durable store revision.

No API may silently turn the first contract into the second.

## Problem Statement

A row cap can mean two incompatible things:

- safety assertion: fail if more than N rows would be affected; or
- selection window: update the first N rows and leave the rest.

The first is exact. The second is partial. Reusing a read window for an exact
mutation loses the distinction and permits successful partial completion.

Unbounded trusted UPDATE is not a sufficient answer. Although one serialized
call can safely freeze all matching keys, an arbitrary collection can exceed
heap, instruction, marker, or journal budgets before a complete atomic batch is
formed.

Manual repeated LIMIT updates are also incomplete as an engine contract. They
do not by themselves define:

- canonical continuation;
- retry behavior after a committed response is lost;
- schema changes between calls;
- completion under rows inserted behind a checkpoint;
- patch idempotency;
- progress and response semantics;
- whether LIMIT is caller policy or engine resource policy.

## Design Decision

0.207 introduces two distinct single-entity UPDATE lanes.

### Lane A: Bounded Exact UPDATE

The caller supplies a max_affected safety assertion. IcyDB selects at most
max_affected plus one matching primary keys in canonical order.

- If the extra row exists, IcyDB returns a typed overflow error before mutation.
- If selection reaches exhaustion within the cap, IcyDB freezes the complete
  key set, stages every patch and index effect, and commits one atomic batch.
- A successful result means every row matching the statement snapshot was
  included.
- A failure before the marker means zero rows changed.

max_affected is not SQL LIMIT and never selects a successful prefix.

### Lane B: Trusted Resumable Convergence UPDATE

The application starts or resumes a trusted/admin bulk update. IcyDB owns the
batch resource budget and advances through canonical primary-key keysets. Each
call commits at most one atomic row batch and returns an opaque continuation to
trusted application code.

The v1 resumable patch must be fixed and idempotent. Assigned fields must be
provably disjoint from:

- primary-key fields;
- fields referenced by the scope predicate;
- generated-owned and managed/internal fields.

IcyDB implicitly skips rows whose authored assigned fields already equal the
target patch. Replaying a prior continuation therefore does not apply the same
semantic change twice.

After forward keyset traversal reaches the end, IcyDB starts a bounded,
resumable verification sweep from the beginning. The sweep records the target
store's durable mutation revision and may return complete = true only after it
has examined the complete entity keyspace without that revision changing. A
changed revision conservatively restarts forward convergence, so inserts or
updates behind an old checkpoint cannot be missed.

The entire job is not atomic. Each batch is atomic.

V1 admits the resumable lane only for a store whose existing journal authority
provides a durable monotonic mutation revision. Exact UPDATE remains available
for heap and journaled stores. A transient heap generation is not sufficient
authority for a continuation that may outlive runtime reinitialization.

V1 also requires application-custodied continuation provenance. A multi-page
verification checkpoint is proof state, and opacity or an unkeyed checksum does
not prevent a client from forging it. Generated/public endpoints expose an
operation id, not raw continuation bytes. Toko stores the latest engine-issued
continuation in trusted application state outside the target store's mutation
revision. An externally supplied bearer-token protocol requires a separate
authenticated-token or durable-job-state design.

## Why Serialization Helps

Within one update call, canister serialization and the no-await invariant mean:

- accepted schema cannot change during selection and staging;
- matching rows cannot be concurrently rewritten by another IcyDB call;
- the candidate set can be frozen without an MVCC snapshot identifier;
- cap overflow can reject before mutation;
- one batch can be fully prevalidated before its marker.

This is sufficient for Lane A whenever the complete set fits resource bounds.

## What Serialization Does Not Solve

Serialization does not provide an unlimited message. A single exact update can
still be rejected because:

- scanning cannot finish within the read/instruction budget;
- candidate keys or patches exceed transient heap;
- prepared row and index records exceed marker or journal bounds;
- one row's encoded mutation cannot fit;
- bounded RETURNING cannot fit its response budget.

Lane B spans messages. Messages remain serialized individually, but unrelated
or conflicting writes may occur between two resume calls. The resumable
contract therefore uses convergence verification and requires the application
to own any stronger business fence.

## Goals

- Make exact collection UPDATE succeed completely or reject with zero changes.
- Make max_affected a safety assertion rather than a hidden selector.
- Provide bounded, resumable UPDATE for collections that cannot fit one atomic
  message.
- Keep every resumable batch marker-bound and atomic.
- Make fixed-patch retries idempotent.
- Give complete = true a server-proved convergence meaning.
- Keep verification itself bounded for arbitrarily large collections.
- Keep continuation independent of the selected physical query route.
- Bind continuations to accepted schema, target entity, scope, and patch.
- Keep proof-bearing continuation bytes under trusted application custody.
- Preserve accepted schema as runtime authority.
- Preserve SQL as a frontend to catalog-native structural mutation.
- Keep generated/public broad UPDATE closed by default.
- Avoid a new engine-owned durable job registry in v1.
- Preserve current row, index, marker, journal, and accepted-schema formats.

## Non-Goals

0.207 does not:

- provide a multi-entity transaction;
- make the complete resumable job atomic;
- add async, await, yield, timers, or a background scheduler to IcyDB;
- provide a pinned MVCC snapshot across update calls;
- guarantee completion while conflicting writes continue indefinitely;
- support resumable arithmetic, increment, append, or other non-idempotent
  patches in v1;
- support resumable patches that change scope-predicate fields;
- mutate primary-key, generated-owned, or managed/internal fields;
- add bulk INSERT, DELETE, UPSERT, or schema mutation;
- reuse an ordinary read cursor as a mutation token;
- expose the trusted resumable lane through generated public endpoints by
  default;
- treat a continuation token as authorization;
- accept client-authored verification checkpoints or raw bearer continuations;
- promise an exact cumulative changed-row count without durable job state;
- introduce an engine-owned stable job table or new stable-memory allocation;
- make transient heap generations authoritative across resumable calls;
- use OFFSET continuation;
- preserve unbounded trusted UPDATE as a compatibility fallback.

DELETE may adopt the exact/resumable split later, but it is not part of this
slice.

## Terminology

### Scope

The accepted structural predicate identifying rows eligible for the operation,
for example collection_id equals a specific collection.

### Fixed Patch

A bounded set of accepted field assignments whose target values are stable for
the operation. The operation timestamp used by sanitizer/managed behavior is
also fixed in the continuation even though managed fields are not caller
assigned.

### Needs Patch

A row matches the scope and at least one caller-assigned field differs
structurally from its fixed target value.

### Exact Selection

A selector that has reached exhaustion and proved the complete matching key set
within the caller and engine budgets.

### Forward Checkpoint

The last canonical primary key examined by a resumable forward scan page. It
advances even when the page finds no row needing the patch. Page-budget
exhaustion is not entity-keyspace exhaustion.

### Store Mutation Revision

A durable monotonic value that changes whenever a committed row batch changes
the target store. For a journaled store, the current authority is the next
append sequence derived from the journal tail and its persisted fold watermark.
Journal folding preserves this value's monotonic meaning.

### Verification Sweep

A bounded, canonical primary-key scan from the beginning after forward
traversal reaches the end. Its continuation carries the captured store mutation
revision and its last examined primary key. It is the sole authority for
complete = true.

### Operation Atomicity

Lane A is atomic as one operation. In Lane B, one batch call is atomic; the
multi-call job is a resumable convergence process.

## Authority Model

The authority order is:

1. accepted schema owns field layout, field ownership, indexes, relations, and
   value semantics;
2. the validated structural UPDATE plan owns scope and patch meaning;
3. authoritative rows own current values;
4. the exact frozen key set or resumable keyset checkpoint owns one call's
   target progression;
5. the commit marker owns one atomic row batch after the boundary;
6. the continuation carries resumable progress but is not authorization or
   runtime schema authority.

Generated EntityModel values remain proposal/model convenience and must not
reconstruct accepted runtime meaning.

## Lane Selection Matrix

| Caller intent | Required lane | Success meaning |
|---|---|---|
| Update one exact primary key | Existing primary-key UPDATE | That row was updated or did not match |
| Update a known-small complete collection | Bounded exact | Every matching row in the statement snapshot committed |
| Update one deliberate ordered prefix | Existing public bounded UPDATE | Exactly the selected LIMIT window committed |
| Converge an arbitrarily large durable collection | Trusted resumable | This batch committed; complete only after a stable-revision verification sweep |
| Return a partial read prefix | partial_window | Only the requested read prefix |

No implicit fallback moves between rows of this matrix.

## Lane A: Exact Execution Flow

### 1. Validate Intent

The exact request requires:

- accepted entity and store authority;
- an admitted UPDATE scope;
- a positive max_affected;
- a max_affected value no greater than the engine exact-row ceiling;
- accepted assignments;
- an explicit all-rows intent if no WHERE predicate exists;
- bounded RETURNING policy when requested.

The caller cap and engine budgets are separate. A caller cap within the row
ceiling does not guarantee that unusually wide rows or index effects fit byte
and instruction budgets.

### 2. Select Cap Plus One

IcyDB selects matching primary keys in canonical accepted primary-key order.
Selection stops when:

- the selector is exhausted; or
- max_affected plus one keys have been found; or
- a scan/instruction budget rejects.

The candidate cap limits affected rows. A separate scan budget protects a
selective residual predicate that may inspect many rows before finding a small
candidate set.

No data, index, or schema mutation occurs.

### 3. Reject Overflow

If max_affected plus one matching keys are found, IcyDB returns a typed
TooManyAffectedRows-style error containing bounded diagnostic facts:

- caller cap;
- proof that at least cap plus one rows matched;
- target entity and operation class.

It does not need to count the entire collection. The physical store remains
byte-identical.

### 4. Freeze And Stage

If the selector reaches exhaustion within the cap, IcyDB:

- freezes the complete canonical primary-key list;
- reads each current accepted row;
- constructs its structural after-image;
- runs sanitization, validation, relations, uniqueness, and index preparation;
- validates duplicate and cross-row conflicts;
- stages bounded RETURNING output;
- encodes and validates marker/journal resource usage.

Every failure remains precommit and changes zero rows.

### 5. Commit One Atomic Batch

IcyDB persists the normal marker and mechanically applies the prepared batch.
Guarded recovery completes an interrupted marker-owned batch.

The success receipt reports exact matched and updated counts. It may report zero
updated when the exact matching rows already contain the assigned values,
according to the maintained SQL UPDATE count contract selected for the surface.

## Lane A API Shape

The exact semantic must be carried by a typed policy object rather than inferred
from missing LIMIT.

Conceptually:

~~~text
ExactUpdateRequest
  max_affected
  returning bounds

execute_exact_update(validated_plan, request)
  -> ExactUpdateReceipt

ExactUpdateReceipt
  matched_rows
  updated_rows
  returning
~~~

The implementation may expose this through a fluent structural facade, a
validated SQL plan adapter, or both. Both must lower into the same structural
exact owner.

SQL LIMIT remains selection semantics and cannot serve as max_affected.

## Lane B: Resumable Eligibility

A resumable plan is admitted only when all of the following are proven:

- one target entity and store;
- a journaled store with a durable monotonic append revision;
- accepted schema and row layout;
- a deterministic scope predicate;
- canonical accepted primary-key ASC continuation;
- fixed literal/default structural assignment values;
- no primary-key assignment;
- no generated-owned or managed/internal assignment;
- assigned fields are disjoint from every field dependency of the scope;
- no OFFSET;
- no caller-selected batch size;
- no subquery or cross-entity dependency;
- no assignment whose result depends on the current row value;
- bounded mutation token and response encoding.

If predicate dependency analysis cannot prove disjointness, the plan rejects.
There is no permissive fallback.

Exact Lane A does not need the scope/assignment disjointness restriction because
it freezes the complete statement target set before mutation.

## Lane B: Resumable Execution Flow

### 1. Start

Starting a job:

- validates trusted/admin execution intent;
- requires a trusted continuation-custody owner;
- derives the accepted structural scope and fixed patch;
- records the accepted schema fingerprint;
- fixes the operation timestamp used by write sanitization;
- generates a bounded opaque operation id for custody lookup and diagnostics;
- begins in Forward phase with no checkpoint.

IcyDB creates no server-side job row. The application may own a job record and
is responsible for storing only continuations returned by IcyDB. That record
must not be written through the target store while a Verify sweep is in
progress, because such a write would correctly invalidate the store-wide
revision proof. Application custody rejects an operation-id collision rather
than replacing an existing job.

### 2. Resume And Revalidate

Each resume call:

- reruns endpoint authorization before entering IcyDB;
- loads the prior continuation from trusted application custody rather than
  accepting client-authored bytes;
- decodes the bounded current token;
- validates target entity and store;
- re-derives the submitted scope and patch;
- compares their fingerprints with the token;
- validates accepted schema fingerprint;
- validates phase, phase-specific checkpoint, batch policy id, and operation
  timestamp;
- validates that the target store still provides the required durable mutation
  revision authority.

A mismatch rejects before row selection or mutation.

The application resubmits the intended update alongside the token. The token is
not an executable serialized plan.

### 3. Select A Forward Batch

Forward selection finds rows that:

- are strictly after the canonical primary-key checkpoint;
- match the stable scope;
- still need the fixed patch.

The engine scans and selects within its row, byte, scan, instruction, and
response budgets. The physical access route may use an accepted index, but
continuation meaning remains the canonical primary key. The scan records both:

- the last canonical primary key examined; and
- whether the authoritative entity keyspace was exhausted.

The selected primary keys are frozen before staging.

### 4. Stage And Commit The Batch

The selected rows use the same structural preflight and atomic marker-bound
commit discipline as Lane A.

After the page:

- selected rows commit as one batch and the next continuation advances to the
  last examined primary key;
- a clean page that did not exhaust the entity keyspace returns an advanced
  Forward continuation without opening a marker;
- only proven entity-keyspace exhaustion transitions to Verify.

If a committed page also reached entity-keyspace exhaustion, the resulting
continuation may enter Verify with a mutation revision captured after that
batch's marker is cleared. It does not manufacture successful completion from
suffix exhaustion.

### 5. Start A Verification Sweep

Suffix exhaustion captures the journaled store's current mutation revision and
returns, or immediately begins, a Verify phase with no primary-key checkpoint.
The verification phase is read-only; the resumable operation itself therefore
does not advance the captured revision.

### 6. Resume A Bounded Verification Page

Before every verification page, IcyDB compares the current durable store
mutation revision with the revision captured by the token.

- If they differ, the verification proof is invalidated and IcyDB returns an
  in-progress continuation reset to Forward phase from the beginning.
- If they match, IcyDB scans a bounded canonical primary-key page strictly
  after the verification checkpoint.

The page searches for any row that:

- matches the stable scope; and
- still needs the patch.

If a needs-patch row exists, IcyDB returns an in-progress continuation reset to
Forward phase from the beginning. A later call resumes convergence.

If the bounded page contains no needs-patch row but does not exhaust the entity
keyspace, IcyDB returns a Verify continuation carrying:

- the same captured store mutation revision; and
- the last canonical primary key examined.

If the page exhausts the entity keyspace, IcyDB rechecks the store mutation
revision and returns complete = true only if it is still equal to the captured
value. The final recheck is explicit even though normal canister execution is
serialized; it keeps the proof boundary local and fail-closed.

This bounded stable-revision sweep catches:

- rows inserted behind the prior checkpoint;
- rows changed behind the checkpoint by another call;
- a lost response followed by replay of an older continuation;
- route changes that do not alter semantic primary-key order.

Any committed row batch in the same store invalidates a verification sweep,
including an unrelated entity mutation. That conservative false restart is
acceptable in v1; a narrower revision would require new durable authority.

Journal folding does not invalidate a sweep because the persisted fold
watermark preserves the highest folded sequence and therefore the next append
sequence. Revision exhaustion or corruption fails closed with a typed internal
or resource error; it never certifies completion.

### 7. Completion

complete = true means:

> A complete bounded verification sweep examined the entity keyspace while the
> target store's durable mutation revision remained unchanged, and found no
> accepted row matching the scope that required a caller-assigned target value.

It does not mean:

- a historical start-of-job snapshot was updated;
- no future call can change a row;
- the whole job was one transaction;
- application writes were globally fenced.

## Resumable Token

The current-only token contains:

~~~text
format version
operation id
target store/entity identity
accepted schema fingerprint
normalized scope fingerprint
fixed patch fingerprint
fixed operation timestamp
phase
phase-specific canonical primary-key checkpoint
captured store mutation revision when phase is Verify
engine batch-policy identity
~~~

The token does not contain:

- row after-images;
- a generated EntityModel;
- authorization;
- an exact cumulative updated count;
- arbitrary executable SQL;
- a stable-memory address;
- a server-side job pointer.

The codec is:

- bounded;
- fallible;
- canonical;
- current-form only before 1.0;
- distinct from ordinary read cursor semantics;
- validated before any store access driven by token contents.

The token may use the existing cursor envelope owner only if doing so does not
conflate read and mutation semantic payloads. It must have a distinct mutation
kind and validation contract.

### Integrity And Custody

The phase, checkpoint, and captured revision are proof-bearing. A SHA-256
fingerprint, CRC, format magic, or opaque encoding can detect some mistakes but
cannot authenticate them against a caller who can recompute the bytes. IcyDB
does not currently own a stable secret suitable for a synchronous MAC, and
0.207 does not add one.

Therefore the v1 token codec and constructors remain inside the trusted engine
boundary. Application code may persist the opaque value, but public/generated
endpoints must not accept raw token bytes from clients. They accept an
application operation id, reauthorize it, and load the latest engine-issued
continuation from trusted application state. The state may be volatile if the
application is willing to restart the idempotent job after upgrade.

Application custody must be outside the target StoreHandle's journal revision,
or at least avoid writes between Verify pages. Otherwise saving the next token
would itself invalidate every verification sweep. Toko should use its existing
operation/control state or a distinct store, not a row in the target token
store.

The custody update order is load prior token, execute one engine call, then
store the returned continuation or completed state. If execution commits but
the custody update is interrupted, the prior token remains and its documented
effect-idempotent retry converges safely. The application must never publish a
next token before the corresponding engine call succeeds.

Token opacity is an API boundary, not authorization. V1 remains trusted/admin
only, and every call re-runs endpoint authorization. If a future surface must
round-trip proof tokens through untrusted clients, implementation stops for a
design choosing either authenticated tokens with managed key lifecycle or an
engine-owned durable job registry.

## Resumable Receipt

Conceptually, inside the trusted application boundary:

~~~text
ResumableUpdateReceipt
  operation_id
  batch_examined
  batch_updated
  continuation
  complete
  phase
~~~

The receipt intentionally omits an exact cumulative changed count. Without
durable job state, a committed response can be lost and the prior token can be
retried. Effects remain safe, but a stateless cumulative counter could
undercount or double-count.

An application may maintain advisory progress, but it must not treat that value
as an engine-certified exact total.

## Retry And Idempotency Semantics

### Failure Before Marker

The call returns an error, changes no target row, and the same continuation may
be retried.

### Interruption After Marker

The marker owns the atomic batch. Guarded recovery completes it before the next
operation. The caller may not have received the next continuation.

Retrying the prior token is safe because already-converged rows no longer need
the fixed patch. The retry either advances over later rows or verifies
completion.

### Lost Successful Response

The prior token can be submitted again. The server does not promise to reproduce
the byte-identical lost receipt, but it promises forward-safe convergence and no
duplicate non-idempotent effect.

### Repeated Complete Request

A completed receipt carries no continuation. Starting the same fixed update
again is a new operation and normally reaches completion through a read-only
verification sweep with zero rows needing change.

## Managed Fields And Operation Time

Although resumable callers cannot assign managed/internal fields, sanitization
may maintain fields such as updated timestamps.

The operation timestamp is fixed at job start and carried in the token. Every
batch uses that same timestamp where the maintained write policy permits. This
prevents retry timing from changing the semantic after-image solely because a
response was lost.

If an existing managed policy requires per-batch current time, that policy must
be classified explicitly. It cannot silently weaken idempotency.

## Concurrency And Live-State Semantics

### Within One Call

No await or yield is allowed. Target selection, preflight, and marker
persistence are serialized.

### Between Calls

Other messages may insert, delete, or update target rows.

- A new row after the checkpoint that already has target values needs no work.
- A committed write between verification pages changes the durable store
  revision and restarts forward convergence.
- A new or conflicting row behind the checkpoint is found by the restarted
  forward scan or the next stable verification sweep.
- A deleted row disappears from the convergence set.
- A row changed after complete is a later application mutation, not a failure of
  the completed job.
- Continuous conflicting writes may prevent completion indefinitely.

### Business Fencing

IcyDB convergence cannot by itself guarantee a higher-level invariant such as
"all future tokens in this collection use the reset tier."

Toko must choose one application policy:

- reject conflicting tier writes while reset is in progress;
- make new tokens inherit the target tier;
- attach a collection reset generation to writes;
- define an explicit last-writer-wins rule.

The engine receipt reports database convergence over the stable interval proved
by the final verification sweep. It does not substitute for application policy.

## Toko Reference Shape

Toko's reset has:

- stable scope: collection_id equals the target collection;
- fixed patch: every tier field equals its reset value;
- canonical continuation: Token accepted primary key ASC;
- idempotent convergence: rows already reset are skipped.

Conceptually:

~~~text
reset_token_tiers_exact(collection_id)
  -> Complete { updated_rows }
  | TooManyForExact { exact_cap }

start_reset_token_tiers_resumable(collection_id)
  -> InProgress { operation_id, batch_updated }

continue_reset_token_tiers(operation_id)
  -> InProgress { operation_id, batch_updated }
  | Complete { operation_id }
~~~

The client-visible result deliberately omits the engine continuation. Toko maps
operation_id to the latest engine-issued continuation in trusted control state
outside the token row store.

The recommended endpoint behavior is:

1. attempt Lane A when the collection is expected to fit the audited exact cap;
2. if exact selection proves cap plus one, return TooManyForExact without
   changing rows or silently starting Lane B;
3. require an explicit resumable-start request, then on each follow-up call
   authorize operation_id, load its trusted
   continuation, and execute at most one Lane B page or batch;
4. report success-as-complete only after IcyDB's stable-revision verification
   sweep;
5. enforce the chosen application write fence while in progress.

An endpoint may choose resumable mode immediately and skip the exact attempt.

The engine must not automatically mutate the first exact-cap rows while
returning TooManyForExact. Overflow changes zero rows.

## SQL And Structural Ownership

SQL remains a frontend. The maintained authority is one accepted structural
bulk-UPDATE plan containing:

- entity/store authority;
- structural scope;
- scope field dependencies;
- structural fixed patch;
- accepted schema fingerprint;
- exact or resumable execution intent;
- returning/resource bounds.

Exact and resumable adapters consume that plan. They do not each reparse or
reinterpret SQL.

Public bounded SQL UPDATE remains:

~~~sql
UPDATE Token
SET tier = 'default'
WHERE collection_id = ?
ORDER BY id ASC
LIMIT 100
~~~

Its success means one deliberate prefix, not completion.

Exact policy uses a separate max_affected request value, not SQL LIMIT.
Resumable policy uses an engine-owned batch size, not caller LIMIT.

## Trusted Surface Boundary

V1 exact UPDATE may be exposed to trusted/session callers and narrowly validated
generated endpoints when their policy owns a safe max_affected.

V1 resumable UPDATE is trusted/admin-only:

- generated public endpoints do not expose it automatically;
- generated/public endpoints do not decode client-supplied raw continuations;
- trusted mode does not bypass schema, validation, relation, index, or resource
  checks;
- endpoint authorization remains application-owned;
- the token grants no authority.

Future public resumable exposure requires a separate authorization and abuse
budget review.

|---|---|---|---|
| Invalid/stale token | Absent | Prior batches remain | Reject this call |

| Batch selection budget failure | Absent | Prior batches remain | Reject this call |

## Resource Ownership

### Exact Row Cap

max_affected remains the exact overflow assertion. Admission rejects a request
whose value exceeds either the engine exact-row ceiling or a narrower
surface-specific policy cap; it does not silently substitute a lower cap and
misreport that as the caller's assertion.

The engine ceiling is internal resource policy selected from measurement. It is
not copied from a Toko constant and does not guarantee that every within-ceiling
batch fits its byte, marker, or instruction budgets.

### Exact Scan Budget

Selection also owns a scan/instruction budget. A residual predicate cannot scan
an arbitrary store merely because it matches fewer than max_affected rows.

### Marker And Journal Bytes

Prepared row records, index effects, and marker batches are encoded and bounded
before marker persistence. An exact or resumable batch never opens a commit
window that its staged payload cannot represent.

### Resumable Batch Budget

The engine owns:

- maximum rows selected;
- maximum staged mutation bytes;
- maximum scan work;
- maximum RETURNING rows and bytes;
- maximum token bytes.

The batch may contain fewer than the nominal row count when byte or instruction
estimation requires it. Progress means either committing at least one selected
row, advancing the last-examined Forward checkpoint over a clean scan page, or
proving entity-keyspace exhaustion.

If one valid row cannot fit, the call returns a typed single-row/resource error
without mutation. If even one candidate key cannot be examined within the scan
budget, the call returns a typed no-progress resource error rather than
reissuing an unchanged continuation.

### Verification Page Budget And Revision

Verification uses its own bounded scan/instruction budget and must return a
continuation after making bounded keyspace progress. It never attempts to scan
an arbitrarily large collection in one message.

V1 reuses the existing journal-tail sequence as the durable per-store mutation
revision:

- a committed marker-owned row batch advances the append sequence;
- the persisted fold watermark retains the highest folded sequence;
- clearing folded tail batches therefore cannot move the revision backward;
- overflow or corrupt/noncontiguous journal state rejects fail-closed.

The revision is store-wide rather than entity-wide. Unrelated writes may cause
extra verification restarts but cannot produce a false completion.

The transient DataStore generation is deliberately not used as continuation
authority: it resets with runtime reinitialization and currently saturates at
u64::MAX. Heap stores therefore support Lane A but reject Lane B in v1.

### No Unbounded Collection

Candidate keys, patches, prepared operations, RETURNING values, and token
payloads are all bounded collections with incremental checks.

## Persisted And Wire Compatibility

0.207 expects no change to:

- row encoding;
- data or index keys;
- index values;
- accepted schema snapshots or roots;
- commit markers or journal record codecs;
- stable allocation identities;
- ordinary read cursor semantics.

The verification revision is derived from the existing journal tail and fold
watermark. It adds no stable field, allocation, journal record, or persisted
codec.

0.207 introduces one current-form mutation continuation codec inside the
trusted engine/application boundary. IcyDB does not store it, public endpoints
do not decode it, and ordinary read cursors do not accept it. If an application
persists the opaque bytes in its own control state, that state remains
application-owned rather than an IcyDB stable-memory allocation.

Before 1.0:

- only the current token version decodes;
- removed token forms receive no aliases or fallback;
- no anti-resurrection tests preserve retired token shapes.

If implementation requires an engine-owned durable job store, new stable memory
allocation, or marker record for job progress, work stops for a persisted-format
design amendment. It must not be smuggled into the token slice.

## Proposed Module Shape

Production ownership should converge on:

- one accepted structural BulkUpdatePlan;
- one ExactUpdateExecutor;
- one ResumableUpdateExecutor;
- one bounded mutation-token codec/validator;
- existing atomic row-batch commit machinery;
- one convergence predicate owner for fixed patches;
- one bounded verification-sweep owner backed by the existing journal
  revision;
- one receipt model.

The design rejects:

- a generic transaction framework;
- a background job scheduler;
- a callback/capability hierarchy for mutation modes;
- separate SQL and fluent bulk semantics;
- using partial_window as a mutation selector;
- using read cursor state as mutation authority;
- an unbounded trusted Vec collection fallback;
- per-application engine branches.

Toko is a reference fixture, not a production dependency or special executor
path.

## Hard-Cut Cleanup

After both lanes are maintained:

- broad trusted UPDATE call sites must select explicit exact, bounded-prefix, or
  resumable intent;
- no externally reachable trusted path may stage unbounded candidate rows;
- helpers that interpret a row cap ambiguously are deleted or renamed;
- examples claiming complete mutation through partial_window are removed;
- examples round-tripping raw mutation proof tokens through public clients are
  rejected;
- tests that preserve an unbounded or silently partial legacy route are deleted;
- active docs teach max_affected as a safety assertion and continuation as
  explicit partial progress.

Existing public deterministic LIMIT UPDATE remains because it is a maintained
explicit-prefix surface, not a legacy fallback.

## Implementation Slices

### Slice 1: Baseline And Contract Lock

- Re-audit exact, bounded-prefix, trusted broad, and typed atomic UPDATE paths.
- Record row/byte/instruction/marker/RETURNING baselines.
- Add 0.207 cases to the 0.204 evidence authority.
- Add structural guards proving partial_window is not used by write execution.
- Select engine exact and resumable resource-budget owners.
- Confirm the mutation token codec owner and size limit.
- Confirm the application-custody boundary and ensure its writes do not share
  the target store revision during Verify.
- Confirm that every maintained journaled row mutation advances the same
  durable append revision and that fold preserves it.

Exit condition: every current broad UPDATE ingress and resource boundary is
enumerated with no production behavior change.

### Slice 2: Bounded Exact UPDATE

- Add explicit exact execution intent and max_affected.
- Implement canonical cap-plus-one selection.
- Freeze complete keys only after selector exhaustion.
- Route complete staging through existing atomic row-batch commit.
- Enforce RETURNING and marker budgets before commit.
- Add exact overflow and interruption evidence.

Exit condition: exact success proves complete inclusion, and every overflow or
precommit failure leaves byte-identical before state.

### Slice 3: Fixed-Patch Resumable Plan And Token

- Add accepted scope dependency extraction.
- Admit only fixed idempotent patches disjoint from scope and primary key.
- Add a current-form bounded mutation token codec and opaque token type without
  a public/generated raw-byte decoder or public field constructors.
- Bind token to entity/store, accepted schema, scope, patch, operation time, and
  batch policy.
- Admit resumable execution only when the store exposes the durable journal
  revision required by verification.
- Keep resumable surface trusted/admin-only.

Exit condition: malformed, stale, cross-plan, and ineligible resumable requests
reject before row access or mutation.

### Slice 4: Forward Atomic Batches

- Select needs-patch rows by canonical primary-key checkpoint.
- Distinguish bounded scan-page exhaustion from entity-keyspace exhaustion.
- Advance over clean scan pages without opening a marker.
- Freeze one engine-bounded key batch.
- Stage and commit through atomic row-batch machinery.
- Return per-call receipt and continuation.
- Prove old-token retry after committed response loss is effect-idempotent.

Exit condition: every successful in-progress call commits one complete atomic
batch or makes bounded read-only keyspace progress, and can be retried safely.

### Slice 5: Verification And Live-State Convergence

- Add bounded from-the-beginning Verify pages after suffix exhaustion.
- Capture and carry the durable store mutation revision across Verify pages.
- Restart Forward when the revision changes or work appears.
- Certify complete only after entity-keyspace exhaustion at the unchanged
  revision.
- Cover inserts, deletes, and conflicting writes between calls.
- Freeze managed operation time across tokens.
- Add the Toko collection-tier reference fixture.

Exit condition: complete is returned only when an independent full-scope oracle
finds no row needing the fixed patch.

### Slice 6: Surface Cleanup And Closeout

- Require explicit intent at broad trusted UPDATE ingresses.
- Delete unbounded/silent-partial fallbacks and preservation-only tests.
- Update active contracts, facade guidance, diagnostics, and examples.
- Complete 0.204 matrix evidence.
- Record final instruction, heap, marker, source-complexity, and raw wasm deltas.

Exit condition: every maintained broad UPDATE is exact, explicit-prefix, or
resumable, with no ambiguous cap.

## Validation Plan

### Structural Audit

Focused searches must prove:

- write execution does not call partial_window;
- every production mutation of a journaled row store either appends a
  marker-owned journal batch or is recovery replay of an already represented
  sequence;
- exact max_affected is not lowered to SQL LIMIT;
- resumable batch size is not caller controlled;
- no OFFSET continuation exists;
- no external trusted UPDATE stages an unbounded candidate collection;
- token decode is bounded and fallible;
- no public/generated endpoint decodes a raw resumable token;
- proof-token custody writes do not advance the target store revision between
  Verify pages;
- Verify tokens bind a phase-specific checkpoint to a durable store revision;
- heap stores cannot enter the v1 resumable lane;
- generated models do not supply runtime row/schema authority;
- no server-side durable job store or allocation was introduced;
- no new marker, journal, row, or index codec was introduced.

### Exact Matrix

Cover:

- zero matches;
- one match;
- cap minus one;
- exactly cap;
- cap plus one;
- far more than cap with early overflow proof;
- indexed and full-scan scope;

## Performance And Size Budget

### Exact Complexity

For E matching rows within cap and S scanned rows:

- selector work is O(S), stopping at cap plus one or exhaustion;
- staged row work is O(E);
- index/relation work is O(E multiplied by affected derived contracts);
- transient memory is O(E row/index/RETURNING payload);
- marker/journal bytes are O(E encoded mutations).

Exact mode deliberately pays complete staging cost for atomicity.

### Resumable Complexity

For engine batch B:

- one call stages O(B) rows and derived effects;
- continuation memory is O(1) plus bounded primary-key/token bytes;
- one verification page scans bounded work and uses O(1) continuation memory;
- a successful quiet-period verification sweep is O(total entity rows);
- total job work is O(forward convergence plus verification sweeps);
- any store-wide row write during Verify can add a conservative restart.

No query hot-path cost is added outside bulk-update entrypoints.

### Measurement Matrix

Measure:

- matches: 0, 1, exact cap, cap plus one, and large multi-batch;
- row widths: small, representative, and near maintained row limit;
- indexes: 0, 1, 4, and 8 where fixtures permit;
- reverse relations: none and representative;
- heap and journaled stores for exact, and journaled stores for resumable;
- indexed scope and residual/full scan;
- no RETURNING and bounded RETURNING;
- normal batch, response-loss retry, multi-page verification, revision restart,
  and journal fold during verification.

Record:

- selection instructions and rows scanned;
- preflight/staging instructions;
- apply/recovery instructions;
- peak transient heap;
- encoded marker and journal bytes;
- data and index writes;
- token and receipt bytes;
- number of calls to convergence.

### Wasm

Raw non-gzipped wasm bytes are primary. Gzip is secondary context.

The implementation should avoid duplicating SQL/fluent or exact/resumable
staging pipelines. A material wasm increase requires owner-level attribution.

### Complexity

Closeout reports:

- files touched;
- approximate production and test line delta;
- execution owners before and after;
- whether the implementation became simpler, neutral, or more complex.

The intended conceptual result is neutral-to-simpler despite adding one lane:

- one structural bulk-update plan;
- one atomic row-batch apply owner;
- explicit exact and resumable policies;
- zero ambiguous cap interpretations;
- zero unbounded external trusted collectors.

## Risks And Mitigations

### Exact Cap Is Mistaken For A Guarantee Of Fit

Mitigation: document row count as one bound among scan, byte, marker, index, and
response budgets. Every resource rejection remains precommit.

### Resumable Completion Lies About Rows Behind The Checkpoint

Mitigation: suffix exhaustion never completes directly. Only exhaustion of a
bounded from-the-beginning verification sweep at one unchanged durable store
mutation revision may return complete.

### Verification Is Accidentally Unbounded

Mitigation: Verify has the same progress discipline as Forward: an
engine-bounded page, a canonical primary-key checkpoint, and a typed no-progress
resource error. Completion may require many read-only calls.

### Runtime Generation Is Mistaken For Durable Revision

Mitigation: v1 admits resumable mode only on journaled stores and derives the
proof revision from the durable journal tail plus fold watermark. The
saturating, runtime-only DataStore generation is never encoded into a token.

### A Row Write Bypasses The Durable Revision

Mitigation: make journaled row mutation ingress a Slice 1 structural audit and
closeout gate. Normal writes must append a marker-owned batch; recovery may only
replay effects already represented by that sequence. Any other production
bypass blocks the verification design until it is removed or incorporated into
the same durable revision authority.

### Unrelated Writes Starve Verification

Mitigation: store-wide revision invalidation is deliberately conservative.
Applications should quiesce or fence high-write collections while completing a
bulk job. Entity-local durable revisions are deferred unless measurements show
the false-restart rate justifies a persisted-format design.

### Patch Changes Its Own Scope

Mitigation: extract accepted field dependencies and require disjoint assigned
fields. Unknown proof rejects.

### Lost Response Reapplies Managed Changes

Mitigation: fixed authored patch, implicit needs-patch filter, and one operation
timestamp carried across continuations.

### Concurrent Writers Prevent Completion

Mitigation: report in-progress honestly and require Toko/application fencing for
stronger business invariants. Do not add hidden locks across messages.

### Token Becomes An Authorization Capability

Mitigation: trusted/admin-only v1, explicit reauthorization every call, and
documentation that token possession grants nothing.

### Client Forges Verification Progress

Mitigation: do not expose the proof token as a bearer protocol. Keep decode and
construction inside the trusted boundary; public endpoints accept an operation
id and load the latest engine-issued token from application custody. An unkeyed
hash or checksum is explicitly insufficient. Authenticated public tokens or an
engine-owned job registry require a separate design.

### Application Custody Invalidates Its Own Verification

Mitigation: do not save proof progress through the same target StoreHandle
between Verify pages. Toko's operation record lives in separate control state.
Tests cover that the chosen custody write leaves the captured target revision
unchanged.

### Token Becomes A Persisted Job Protocol Accidentally

Mitigation: no stable job store or pointer in 0.207. If exact totals, server-side
cancellation, or exactly-once receipts require persistence, stop for a separate
format design.

### Public Bounded UPDATE Is Accidentally Reclassified Exact

Mitigation: keep LIMIT window semantics and exact max_affected policy in
different typed plans and receipts.

### Full-Scan Scope Consumes The Message Before Finding A Batch

Mitigation: separate scan budget, typed precommit rejection, and performance
evidence for indexed Toko collection_id scope.

### One Row Exceeds A Resumable Batch Budget

Mitigation: return a typed no-progress resource error before marker persistence.
Do not loop or skip the row silently.

### Old Unbounded Trusted UPDATE Remains Reachable

Mitigation: inventory every call site in Slice 1 and require explicit intent by
closeout. Pre-1.0 hard cut; no compatibility wrapper remains.

## Closeout Decision

0.207 establishes one rule for complete collection updates:

> If the complete target fits, freeze it and commit it atomically. If it does
> not fit, change nothing until the caller explicitly enters a resumable,
> idempotent convergence protocol.

Single-threaded execution makes the first lane safe. Explicit continuation, a
bounded stable-revision verification sweep, and application fencing make the
second lane honest. A partial read window is never a substitute for either.
