# 0.80 Single Compiled SQL Cache Design

## Status

Proposed

## Objective

Reduce the fixed instruction-cost floor for normal SQL queries by introducing one durable, future-safe cache boundary for SQL compilation, without creating parallel execution semantics or stage-fragmented cache ownership.

This slice is explicitly aimed at the shared baseline cost seen across ordinary SQL queries, where parse, lower, validate, plan, execute setup, and result shaping dominate more than access-path work.

## Problem Statement

Current SQL execution pays a large fixed cost even for small indexed reads and tiny projections. Audit evidence suggests that the common floor is not primarily access-path work, but the repeated end-to-end SQL compile pipeline:

* canister SQL entrypoint
* parse
* lower
* validate
* plan
* execute setup
* result shaping / encoding

That means improvements like additional indexes, predicate special cases, or surface-level syntax extensions are unlikely to materially reduce the common baseline. They improve selected scenarios, not the repeated shared cost borne by nearly every non-metadata SQL query.

The architectural risk is choosing an optimization shape that later has to be undone when query families widen. A fragmented strategy that caches parse output, lowering output, validation output, planning output, and executor-prepared state independently would create multiple invalidation domains, ambiguous ownership, and pressure toward duplicated semantics.

The design goal for 0.80 is therefore not “cache every boundary,” but to define exactly one cacheable artifact that remains valid as future query shapes expand.

## Design Goal

Introduce one canonical SQL compile artifact and one cache that owns it.

The cacheable unit is the query-owned compiled SQL command.

Execution remains a derived step that prepares executor/runtime state from that compiled command and then runs it. Executor preparation may later gain targeted memoization, but it is not the primary cache contract in this slice.

The central implementation risk is artifact incompleteness. If a cache hit still rebuilds too much semantic-ish executor setup, the system will gain cache complexity without materially lowering the shared SQL floor.

## Non-Goals

This slice does not:

* introduce multiple independent caches for parser, lowering, validation, planning, or executor preparation
* add SQL surface features
* add query-family-specific fast paths
* attempt global cross-session cache sharing
* canonicalize or parameterize SQL literals
* guarantee zero re-planning across schema evolution
* redesign result encoding beyond what is needed to preserve clean compile/execute boundaries
* treat metadata commands as the primary optimization target unless they fit the same semantic cache contract naturally

## Architectural Decision

### Chosen boundary

Cache exactly one artifact:

`CompiledSqlCommand`

Conceptually:

`(raw SQL text, schema compatibility fingerprint, execution-affecting flags) -> CompiledSqlCommand`

This artifact is query-owned semantic output. It represents the compiled meaning of one SQL statement for one compatible schema/runtime feature context.

### Rejected boundary shapes

#### Rejected: per-stage caches

Do not create separate caches for:

* parsed SQL AST
* lowered SQL command
* validated query intent
* planned query shape
* executor-prepared state

Reason:

* each stage would require its own cache key and invalidation rules
* partial hits create awkward ownership questions
* future query-shape widening would increase coupling across stage-specific cache contracts
* this makes performance work harder to reason about, not easier

#### Rejected: executor-prepared state as the main cached artifact

Reason:

* executor-prepared structures are runtime-derived
* executor internals are more likely than semantic query shape to change over time
* making executor-prepared bundles the primary cache value would couple cache stability to runtime implementation details

Executor preparation should remain derived from `CompiledSqlCommand` in this slice.

## Existing Architectural Alignment

This design aligns with existing IcyDB direction rather than introducing a new parallel model.

Relevant codebase history already moved toward:

* query-owned compile handoff decoupled from executor runtime shape
* executor-owned prepared state for repeated fixed SQL projection/setup work

0.80 extends that line by making the compile handoff explicitly cacheable and session-reusable.

## Proposed Model

### Internal pipeline

The internal SQL path should become structurally explicit:

1. `compile_sql(...) -> CompiledSqlCommand`
2. `execute_compiled_sql(...) -> SqlResponse`

Even when called back-to-back, the seam must be first-class.

### Cache ownership

Add one session-local cache:

`SqlCompiledCommandCache`

The cache owns only compiled semantic commands.

### Lifecycle

For each SQL request:

1. build cache key from raw SQL text + schema compatibility fingerprint + execution-affecting flags
2. look up `CompiledSqlCommand`
3. on hit, skip parse/lower/validate/plan and continue to executor preparation
4. on miss, compile end-to-end, store result, then continue to executor preparation
5. execute

## `CompiledSqlCommand`

### Requirements

`CompiledSqlCommand` must:

* fully represent one validated/planned SQL statement
* be stable across executor implementation refactors
* distinguish query family cleanly
* be sufficient input for executor preparation
* avoid embedding transient execution scratch or row buffers

This completeness requirement is the make-or-break property for 0.80. A cache hit must already own enough query semantics that the executor can prepare runtime state without reconstructing planner intent.

### Recommended shape

`CompiledSqlCommand` should be an enum over semantic command families rather than a scalar-only structure.

Illustrative shape:

```rust
pub enum CompiledSqlCommand {
    ScalarSelect(CompiledScalarSelect),
    GroupedSelect(CompiledGroupedSelect),
    GlobalAggregate(CompiledGlobalAggregate),
    Delete(CompiledDelete),
    Update(CompiledUpdate),
    Insert(CompiledInsert),
}
```

Exact variants can follow current supported SQL families. The key point is that future query shapes extend this artifact rather than requiring new cache models.

### Ownership rule

`CompiledSqlCommand` is query-owned semantic state.

It may include:

* lowered SQL command identity
* resolved entity/field/projection/grouping/order semantics
* validated predicate shape and compare/coercion semantics
* finalized planner output, including access path, residual predicate ownership, grouping route, and execution family selection
* semantic metadata needed to derive executor preparation without re-running query planning decisions

It must not include:

* mutable row buffers
* reusable temporary sort/projection vectors
* request-local scratch allocations
* response-encoding accumulators

Those remain execution-owned.

### Executor-derived state

The executor may still derive runtime-owned structures from `CompiledSqlCommand`, but only if that derivation does not redo semantic compile work.

Acceptable executor-derived work includes:

* allocating request-local scratch buffers
* constructing cursor/stream state from already-decided access and route specifications
* preparing projection/materialization helpers from already-resolved semantic projection definitions
* building response encoders and row-shaping accumulators

Not acceptable on a cache hit:

* re-lowering SQL forms
* re-validating predicate or projection semantics
* re-deciding access-path or grouped/scalar route choice
* reconstructing semantic planner ownership that should already live in `CompiledSqlCommand`

## Cache Key

### Required fields

The initial cache key should include:

* raw SQL string
* schema compatibility fingerprint
* relevant execution-affecting feature flags / SQL mode bits

### Why raw SQL text first

Use raw SQL string identity in 0.80 rather than normalized parameterized identity.

Reason:

* lowest-risk first slice
* simple and auditable semantics
* avoids early entanglement with literal normalization and parameter-sensitive planning concerns
* preserves a clean path for future prepared/parameterized expansion without requiring it now

Current implementation note:

* the first shipped cache key uses the exact raw SQL string as submitted
* no whitespace, casing, or literal normalization is applied before lookup
* two semantically equivalent strings with different surface text still map to different cache entries in this slice

### Schema compatibility fingerprint

The fingerprint must change whenever semantic compilation would differ.

Candidate inputs:

* entity/relation model versioning relevant to SQL-visible schema
* index layout / visibility changes relevant to plan validity
* SQL feature capability changes that alter compile outcomes

The key requirement is not the exact mechanism, but that a cache hit must imply semantic compatibility.

### Explicit exclusion

Do not include transient runtime state in the key.

Examples to exclude unless proven compile-affecting:

* current scratch capacity
* request-specific allocation history
* result page size unless it alters semantic planning

## Cache Value

### Stored value

Store:

* `CompiledSqlCommand`
* minimal metadata for observability/debugging

Optional metadata:

* hit count
* compile-time instruction count sample
* statement family
* schema fingerprint used for compile
* insertion / last-hit timestamps

### Do not store by default

Do not store deep executor-prepared state in 0.80.

If profiling later proves a large remaining fixed floor after compile-cache hits, that can be evaluated as a follow-up slice, but only as a derivative optimization under the same semantic cache contract.

## Cache Scope

### Session-local in 0.80

The cache should be session-local.

Reason:

* lowest invalidation complexity
* clean ownership
* avoids global memory-management and cross-session staleness policy in the first slice
* naturally matches “compile once, execute many” inside repeated SQL use in one logical session

### Capacity policy

Start with a bounded LRU.

Desired properties:

* fixed upper bound on memory growth
* deterministic eviction behavior
* straightforward observability

The exact capacity can be tuned later. 0.80 only needs a bounded, simple policy.

## Invalidation

### Primary invalidation rule

A compiled command is reusable only if its schema compatibility fingerprint still matches.

### Invalidation mechanisms

Two acceptable implementations:

1. fingerprint mismatch on lookup
2. explicit cache flush on schema-changing operations

0.80 may choose either or both. The core requirement is correctness, not maximal reuse.

### Fail-closed rule

On any doubt about compatibility, miss the cache and recompile.

This slice must favor correctness over cache-hit rate.

## Execution Boundary

### Compile phase

`compile_sql(...)` owns:

* parse
* lower
* validate
* semantic plan build
* `CompiledSqlCommand` construction

### Execute phase

`execute_compiled_sql(...)` owns:

* executor/runtime preparation derived from compiled command
* row access / stream execution
* projection materialization
* result shaping / encoding

### Rule

Execution must not silently redo semantic compilation work on cache hits.

Minor runtime derivation is expected. Semantic re-lowering/re-validation/re-planning is not.

This is a hard invariant, not just a performance preference.

## Observability

0.80 should ship with explicit visibility into the compile/execute split.

Minimum metrics:

* cache hit count
* cache miss count
* compile instruction count / timing
* execute instruction count / timing
* evictions
* invalidations / fingerprint mismatches
* hit rate by command family

This is necessary to ensure the slice actually reduces the shared floor rather than merely moving work around.

Metadata commands may use the same seam if they fit naturally, but they are not the optimization target that justifies this slice. The main target is the common floor on ordinary scalar, grouped, aggregate, and mutation-capable SQL families.

## Rollout Plan

### Slice A: explicit compile/execute seam

Introduce:

* `compile_sql(...) -> CompiledSqlCommand`
* `execute_compiled_sql(...) -> SqlResponse`

No cache required yet.

This slice should define `CompiledSqlCommand` precisely enough that implementers can inspect a future cache-hit path and prove semantic planner work is already captured there rather than rebuilt later.

Exit criteria:

* existing SQL behavior preserved
* compile and execute costs measured separately
* no duplicated semantic path introduced
* `CompiledSqlCommand` defined concretely enough that a cache-hit path can be audited for semantic completeness

### Slice B: session-local compiled-command cache

Add:

* `SqlCompiledCommandCache`
* key construction
* lookup/store/evict flow
* observability counters

Exit criteria:

* cache hits skip parse/lower/validate/plan
* cache hits do not rebuild semantic planner state indirectly inside executor preparation
* correctness preserved under schema changes
* bounded memory behavior

### Slice C: targeted derivative reuse only if profiling justifies it

Possible follow-up work:

* memoize selected executor-prepared bundles derived from `CompiledSqlCommand`
* improve result-shaping reuse
* evaluate literal normalization / prepared parameter API only if repeat-query patterns justify it

This is explicitly not part of 0.80 core scope.

## Correctness Invariants

1. There is one canonical cacheable semantic artifact for SQL compilation.
2. A cache hit must be semantically equivalent to recompiling under the same compatible schema/feature context.
3. Executor preparation remains derived from compiled semantic state and must not reconstruct semantic planner ownership on cache hits.
4. Query-family growth extends `CompiledSqlCommand`; it does not introduce new top-level cache models.
5. On compatibility uncertainty, the system fails closed by recompiling.

## Why This Survives Future Query Shapes

This design is future-safe because it scales by widening the semantic command artifact, not by multiplying caches.

When a new query family lands:

* add a new `CompiledSqlCommand` variant or extend existing semantic payloads
* keep the same cache contract
* keep executor preparation derived

That means grouped SQL, aggregate SQL, richer projection expressions, or future supported statement types all fit the same model.

The solution survives because the reusable boundary is semantic, not mechanical.

## Alternatives Considered

### 1. No cache, continue fixed-cost cleanup only

Pros:

* minimal new moving parts

Cons:

* likely insufficient to materially lower the common SQL floor
* repeats compile work on every statement

Decision: rejected as sole strategy

### 2. Cache each stage independently

Pros:

* potentially finer-grained reuse

Cons:

* fragmented ownership
* harder invalidation
* greater future recode risk
* more parallel semantics pressure

Decision: rejected

### 3. Cache executor-prepared state as primary artifact

Pros:

* may reduce more fixed runtime setup in some cases

Cons:

* couples cache stability to executor internals
* weaker long-term architectural boundary

Decision: rejected for 0.80 primary model

### 4. Global cache shared across sessions

Pros:

* potentially higher reuse

Cons:

* higher invalidation and memory-policy complexity
* larger blast radius for mistakes

Decision: defer

## Risks

### Risk: schema fingerprint too weak

If the compatibility token misses a semantic change, stale compiled commands may be reused incorrectly.

Mitigation:

* keep fingerprint conservative
* fail closed on uncertainty
* prefer false misses over false hits

### Risk: execution still redoes too much work

If executor preparation silently reintroduces semantic work, cache hits may not materially lower the floor.

Mitigation:

* instrument compile vs execute separately
* audit hit-path behavior explicitly

### Risk: enum growth becomes unwieldy

As SQL families expand, `CompiledSqlCommand` may widen.

Mitigation:

* keep variant payloads family-local
* maintain query-owned semantic ownership boundaries
* avoid executor leakage into variant definitions

## Testing Strategy

0.80 should add tests for:

* repeated identical SQL in one session hits cache
* schema-changing operation invalidates or bypasses stale entry
* distinct feature-flag contexts do not alias incorrectly
* cache eviction preserves correctness
* grouped/aggregate/scalar families all compile through the same top-level cache contract
* cache hit does not re-run semantic compile stages
* cache hit does not rebuild semantic planner state inside executor preparation

## Recommended Acceptance Criteria

* one new canonical `CompiledSqlCommand` cache contract exists
* SQL execution has an explicit `compile -> execute` seam
* session-local cache reuses compiled semantic commands safely
* schema compatibility is enforced fail-closed
* no secondary stage-specific cache model is introduced
* instrumentation proves cache hits skip compile stages
* the hit path is auditable and proves semantic planner work is not being rebuilt under executor preparation

## Final Recommendation

0.80 should implement one session-local cache of query-owned compiled SQL commands, keyed by raw SQL text plus conservative schema compatibility and execution-affecting flags.

The durable architectural rule is:

**cache semantic compile output once, derive runtime preparation from it, and let future query families extend the same compiled-command contract rather than creating new caches.**
