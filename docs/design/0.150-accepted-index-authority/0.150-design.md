# IcyDB 0.150 - Accepted Index Authority

## Status

Design draft.

0.149 moved runtime row decode, row emission, predicate/projection planning,
save validation, session execution, and SQL execution toward accepted schema
contracts. The remaining generated runtime authority is concentrated around
index metadata and standalone model-only planning/explain convenience surfaces.

0.150 should make accepted schema metadata authoritative for indexes in the
same way 0.149 made accepted row contracts authoritative for persisted row
bytes.

## Problem

Generated `EntityModel` and `IndexModel` still own several runtime index facts:

* which fields are indexed
* which indexes are visible to planning
* which key slots or expression inputs define an index
* whether an index is unique
* which index metadata participates in cache fingerprints
* which index metadata is used to build forward/reverse index keys

That bridge was acceptable while accepted snapshots did not persist full index
definitions. It should not remain a silent runtime authority after row and
field contracts are accepted-schema-backed.

## Goals

* Persist accepted index definitions in schema snapshots.
* Make `SchemaInfo` own index membership and visible index lookup from accepted
  metadata.
* Make planner-visible indexes come from accepted schema snapshots, not
  generated `E::MODEL.indexes()`.
* Make cache fingerprints include accepted index contracts.
* Make index key construction consume accepted index metadata plus accepted row
  contracts.
* Keep generated `IndexModel` as an authoring/proposal bridge used during
  startup reconciliation.
* Hard-cut internal formats before `1.0.0`; do not add production fallback
  paths that silently reconstruct missing accepted index metadata from
  generated models.

## Non-Goals

* Do not add SQL DDL.
* Do not support arbitrary index migration policy before the accepted index
  contract is represented.
* Do not support field rename/remap through indexes.
* Do not infer runtime index visibility directly from generated models after
  acceptance.
* Do not remove generated models as macro-authored schema proposals.

## Accepted Index Snapshot

Accepted schema snapshots should store a normalized index contract. Exact type
names can differ, but the metadata must be sufficient for planning and key
construction without reopening generated `IndexModel`:

```rust
struct AcceptedIndexContract {
    name: String,
    unique: bool,
    key: AcceptedIndexKeyContract,
    visibility: IndexVisibility,
    fingerprint: SchemaFingerprintBytes,
}

enum AcceptedIndexKeyContract {
    FieldPath {
        field_id: FieldId,
        slot: usize,
        path: Vec<String>,
        kind: PersistedFieldKind,
    },
    Expression {
        expression_id: String,
        input_slots: Vec<usize>,
        output_kind: PersistedFieldKind,
        expression_fingerprint: SchemaFingerprintBytes,
    },
}
```

The snapshot should prefer stable accepted field identifiers where available
and retain generated-compatible slot numbers only as a validated projection of
the accepted row contract.

## Runtime Boundaries

`SchemaInfo` should expose accepted index facts:

```text
SchemaInfo
  -> accepted row fields
  -> accepted relation facts
  -> accepted index contracts
  -> visible index set
```

Planning and execution should consume schema-owned index views:

* access choice and covering decisions use `SchemaInfo` index contracts
* `VisibleIndexes` is constructed from accepted schema metadata
* primary-key and secondary-index key builders receive accepted index contracts
* index predicates and field-path index reads resolve through accepted slots
* explain/trace metadata renders accepted index names and key shape

Generated indexes remain proposal input:

```text
generated EntityModel/IndexModel
  -> compiled schema proposal
  -> reconciliation
  -> accepted schema snapshot
  -> SchemaInfo
  -> planner/executor/index key runtime
```

## Phases

1. Add accepted index contract types to schema snapshots and proposals.
2. Reconcile generated index proposals against persisted accepted index
   contracts and reject unsupported drift.
3. Route `SchemaInfo` index membership, visible index lookup, and cache
   fingerprints through accepted index contracts.
4. Route planner access choice and query cache keys through accepted visible
   indexes.
5. Route forward/reverse index key construction through accepted index
   contracts plus accepted row contracts.
6. Rename or cfg-gate remaining generated index helpers as proposal-only,
   model-only, or test-only.
7. Add guards proving runtime planning and index writes cannot reopen generated
   index metadata after schema acceptance.

## Validation

Focused validation should cover:

* schema proposal and reconciliation tests for accepted index contracts
* cache fingerprint drift tests for index definition changes
* planner access-choice tests proving accepted visible indexes are used
* forward/reverse index key tests for accepted field slots and expression keys
* schema-evolution rejection tests for unsupported index drift
* write-boundary guards for generated `E::MODEL.indexes()` and `IndexModel`
  usage in runtime paths
* full `icydb-core` SQL and mutation-save suites after index key routing

## Open Questions

* What is the first stable persisted representation for expression index
  contracts?
* Should accepted snapshots store expression input slots, expression field IDs,
  or both?
* When an index contract changes before general migration support exists, is
  the only allowed behavior fail-closed reconciliation?
* Should standalone fluent `.plan()` and direct structural explain gain
  accepted-schema entrypoints, or remain explicitly model-only convenience
  APIs?
* Which describe/show surfaces should render generated proposal metadata versus
  accepted runtime metadata when they diverge during reconciliation failure?
