# 0.202 Explicit Rust `Default` Generation

Status: proposed post-0.200 hard-cut implementation line.

0.202 removes the historical assumption that every schema-authored Rust type
should implement `Default`. Domain types implement Rust `Default` only when the
schema author explicitly requests it with `traits(add(Default))` and the type
has a valid Rust construction strategy.

This is a generated Rust API policy change. It does not change database
defaults, accepted-schema absence policy, persisted row decoding, or runtime
schema authority.

## Context

IcyDB originally treated `Default` as a baseline capability for every generated
type. The derive layer still reflects that history: `Default` is included in
the shared type-trait set, and individual node implementations then derive it,
generate it, suppress it, or require extra syntax.

The current result is mixed:

- entities and records receive `Default` only when every field has a Rust
  construction value;
- newtypes receive it only when a newtype default expression exists;
- enums normally require one variant marked `default`;
- tuples derive it and therefore require every element to implement it;
- lists, sets, and maps derive their natural empty default;
- generated create inputs derive it because every authored field is optional;
- validator and sanitizer marker types derive it as internal construction
  convenience.

The public entity, entity-key, and general schema type contracts no longer
require `Default`. `Principal`, `Account`, and `Ulid` also no longer implement
it merely to provide identity-like zero values. Accepted schema metadata, not
Rust construction, is already authoritative for database defaults and missing
persisted fields.

The shared default-on trait policy is therefore no longer an invariant. It is a
source of implicit API surface and confusing exceptions.

## Problem

Automatic `Default` generation makes a claim about a domain value: that there
is one unsurprising, generally useful value that can be constructed without
input. That claim is not true for every entity, record, enum, newtype, or tuple.

The current policy has several problems:

1. `Default` begins enabled globally even though several node kinds silently
   decline to implement it.
2. Enum authors must choose a default variant even when the domain has no
   natural default.
3. Tuple generation inherits transitive `Default` bounds from its elements
   without an explicit author decision.
4. Entity and record `Default` availability changes when field construction
   metadata changes, creating accidental public API churn.
5. Rust construction defaults can be mistaken for database defaults even
   though accepted schema metadata owns database behavior.
6. `traits(remove(Default))` expresses the intended no-default state as an
   exception to an obsolete global rule.
7. An explicit `traits(add(Default))` request must not be silently ignored when
   the type cannot be constructed.

## Design Goal

Rust `Default` for schema-authored domain types is explicit, local, and
fallible at macro expansion when its structural requirements are not met.

The schema author opts in with:

```rust
traits(add(Default))
```

The derive layer then either emits one valid implementation or reports a clear
compile-time error. It never infers the trait merely because the fields happen
to be constructible.

## Authority Separation

0.202 keeps three concepts separate.

### Rust construction default

`impl Default for T` is typed Rust construction convenience. It exists only
when explicitly requested for a domain type or intrinsically owned by an
internal/canonical-empty generated type.

### Schema-authored field default

`field(..., default = ...)` is schema authoring metadata. It may provide a Rust
construction expression, but it also proposes a database default. Its presence
does not automatically authorize `impl Default` for the enclosing entity or
record.

### Accepted database default

The accepted schema snapshot owns database default behavior. Missing-slot
materialization and sparse-write absence policy consume accepted encoded
metadata. They never consult Rust `Default` implementations.

The dependency direction is one-way:

```text
explicit traits(add(Default))
        +
available Rust construction expressions
        |
        v
generated impl Default

accepted schema metadata -----------------> runtime database defaulting
```

There is no arrow from generated Rust `Default` to accepted runtime behavior.

## Core Rule

Remove `TraitKind::Default` from the shared `TYPE_TRAITS` list.

Domain nodes do not add it automatically:

- entity;
- record;
- newtype;
- enum;
- tuple.

For those nodes, `DefaultTrait::strategy(...)` is considered only when the
author explicitly supplies `traits(add(Default))`.

Constructibility does not imply intent. A type with all-optional fields does
not receive `Default` unless the author requests it.

## Type Policy Matrix

| Generated type | 0.202 policy | Reason |
| --- | --- | --- |
| Entity | Explicit `traits(add(Default))` | An entity generally has no canonical placeholder value. |
| Record | Explicit `traits(add(Default))` | A record default is a domain decision. |
| Newtype | Explicit `traits(add(Default))` plus a construction expression | The wrapped primitive's default must not leak through implicitly. |
| Enum | Explicit `traits(add(Default))` plus one marked variant | Variant choice is a domain decision. |
| Tuple | Explicit `traits(add(Default))` | Element defaults must not create an accidental tuple default. |
| List | Intrinsic empty `Default` | Empty is the canonical collection identity. |
| Set | Intrinsic empty `Default` | Empty is the canonical collection identity. |
| Map | Intrinsic empty `Default` | Empty is the canonical collection identity. |
| Generated entity create input | Intrinsic `Default` | It means that no authored fields were supplied; all fields are `Option`. |
| Validator/sanitizer marker | Internal `Default` when required | It is construction plumbing, not a persisted domain value. |
| Core primitive/value type | Unchanged by 0.202 | Primitive semantics are owned by their defining modules, not schema derive policy. |

Collection nodes should add `TraitKind::Default` locally rather than inheriting
it from the shared type-trait set. This makes their policy visible at the node
that owns the empty-value invariant.

## Domain-Type Rules

### Entities

An entity may opt in only when every generated field has a Rust construction
expression.

Existing construction sources remain valid:

- explicit field `default = ...`;
- insert generation;
- managed-write construction;
- `None` for optional fields;
- empty vectors for many-valued fields.

Example:

```rust
#[entity(
    store = "AppStore",
    version = 1,
    pk(fields = ["id"]),
    fields(
        field(
            ident = "id",
            value(item(prim = "Ulid")),
            generated(insert = "Ulid::generate")
        ),
        field(
            ident = "name",
            value(item(prim = "Text", unbounded)),
            default = "String::new"
        )
    ),
    traits(add(Default))
)]
pub struct User {}
```

If any required field lacks a construction expression, macro expansion rejects
the explicit request and names the field or fields that prevent construction.
It must not emit no implementation and allow the failure to appear later at an
unrelated call site.

Insert generation used by an entity `Default` retains its current generated
construction behavior. 0.202 does not redesign generator determinism or turn a
generated entity default into database authority.

### Records

Records follow the same all-fields-constructible rule as entities.

```rust
#[record(
    fields(
        field(
            ident = "label",
            value(item(prim = "Text", unbounded)),
            default = "String::new"
        ),
        field(ident = "note", value(opt, item(prim = "Text", unbounded)))
    ),
    traits(add(Default))
)]
pub struct Profile {}
```

A record with a required field and no construction expression remains valid;
it simply cannot opt into `Default` until the author supplies a construction
policy.

### Newtypes

Newtypes require both explicit trait intent and an existing newtype construction
default:

```rust
#[newtype(
    primitive = "Ulid",
    item(prim = "Ulid"),
    default = "Ulid::nil",
    traits(add(Default))
)]
pub struct OptionalCorrelationId {}
```

The `default = ...` node metadata remains independently representable. It does
not imply the Rust trait. An explicit trait request without a construction
expression is rejected with a newtype-owned diagnostic.

0.202 does not restore `Default` to identity-like core types. A newtype around
`Principal`, `Account`, or `Ulid` needs an intentional constructor if it opts
in.

### Enums

Enums no longer require a default variant unless `Default` is explicitly
enabled.

No-default enum:

```rust
#[enum_(
    variant(ident = "Pending"),
    variant(ident = "Active"),
    variant(ident = "Closed")
)]
pub struct Status {}
```

Explicit default enum:

```rust
#[enum_(
    variant(ident = "Pending", default),
    variant(ident = "Active"),
    variant(ident = "Closed"),
    traits(add(Default))
)]
pub struct Status {}
```

When `Default` is enabled:

- exactly one variant must be marked `default`;
- a unit variant constructs directly;
- a payload variant requires its payload Rust type to implement `Default`;
- the generated implementation remains ordinary Rust construction and has no
  accepted-catalog authority.

The macro can validate variant count and shape directly. Payload trait
satisfaction is enforced by a generated, enum-local compile-time assertion so
the diagnostic points at the requested enum default rather than an unrelated
downstream use.

A marked default variant without `traits(add(Default))` is rejected as unused
Rust-default configuration. Field-level enum database defaults continue to use
`field(default = "Type::Variant")`; they do not depend on the enum's Rust
default variant.

### Tuples

Tuples no longer derive `Default` through the shared trait list.

An explicit request derives or implements `Default` only when every element
type satisfies the Rust trait:

```rust
#[tuple(
    value(item(prim = "Nat64")),
    value(item(prim = "Text", unbounded)),
    traits(add(Default))
)]
pub struct CursorSeed;
```

Trait satisfaction should be asserted at the tuple generation boundary so a
failure identifies the tuple and offending element type.

## Intrinsic Defaults

### Collections

Generated list, set, and map wrappers keep `Default` automatically. Their
default does not require their item, key, or value types to implement
`Default`; it constructs the empty backing collection.

This is not a domain placeholder. Empty is the collection identity and is
already required by normal collection ergonomics.

The ownership is node-local:

```text
List::traits() -> add Default
Set::traits()  -> add Default
Map::traits()  -> add Default
```

An explicit `traits(add(Default))` on these types is redundant but valid. It
does not produce a second implementation.

### Generated create inputs

Generated `<Entity>_Create` types keep `#[derive(Default)]`. Every authorable
field is represented as `Option<T>`, so the default means "no field authored."
Materialization still returns a typed missing-authored-field error when a
required value has neither authorship nor an allowed construction expression.

Entity create input defaults do not imply that the resulting entity implements
`Default`.

### Validator and sanitizer markers

Generated validator and sanitizer marker defaults remain internal macro
plumbing. 0.202 may remove them if inspection proves they are unused, but it
must not broaden the slice to redesign validation or sanitization construction.

## Trait Configuration Semantics

After `Default` leaves `TYPE_TRAITS`:

- `traits(add(Default))` is the only domain-type opt-in;
- `traits(remove(Default))` on a domain type is invalid because there is no
  inherited trait to remove;
- no alias, deprecated spelling, implicit fallback, or compatibility shim is
  added;
- existing explicit removals, if any are found during implementation, are
  deleted rather than translated into a compatibility path.

The current trait builder's duplicate and invalid-removal validation remains
the syntax authority. Diagnostics should describe the current rule rather than
mentioning pre-0.202 behavior.

## Generated Strategy Contract

Trait resolution must distinguish three outcomes:

1. `Default` not selected: emit nothing and perform no default-constructibility
   validation.
2. `Default` selected and constructible: emit exactly one derive or explicit
   implementation.
3. `Default` selected and not constructible: emit a targeted macro error.

The third outcome must never collapse into the first.

Entity and record strategy helpers currently return an empty strategy when a
required field lacks a construction expression. Under 0.202 that is incorrect
for an explicit request and must become validation failure.

Implementations should continue preferring Rust's derived `Default` when every
construction expression exactly matches the implicit field default. Custom
constructors continue to use one explicit implementation.

This derive-versus-manual optimization is an implementation detail. It must not
change whether the public trait exists.

## Migration Policy

0.202 is a pre-1.0 hard cut.

For each active generated domain type that currently exposes `Default`, choose
one current-form migration:

1. Add `traits(add(Default))` when the domain genuinely has an intentional
   default value.
2. Replace call sites with explicit struct/variant construction when the value
   being built is specific to the test or application flow.
3. Use the generated create input when the call site means "no authored create
   fields" rather than "default entity."
4. Remove arbitrary enum `default` markers when no Rust default is intended.

Do not preserve old behavior through aliases, blanket implementations,
fallback derives, or generated compatibility wrappers.

Fixtures and examples must demonstrate the maintained explicit policy. They
should not add `traits(add(Default))` mechanically merely to keep old
`Type::default()` call sites compiling.

## Diagnostics

Diagnostics should be local and actionable.

Representative messages:

```text
Default was requested for entity User, but required field `email` has no Rust construction value
```

```text
Default was requested for record Profile, but required fields `name` and `region` have no Rust construction values
```

```text
Default was requested for newtype CorrelationId, but no `default = ...` constructor is configured
```

```text
Default was requested for enum Status, but no variant is marked `default`
```

```text
enum Status marks a Rust default variant but does not enable `traits(add(Default))`
```

Diagnostics must not suggest that Rust `Default` supplies a database default.

## Persisted Format And Runtime Impact

0.202 does not change persisted format.

It does not change:

- accepted schema snapshots;
- accepted field default encoding;
- row slot encoding or decoding;
- sparse write absence policy;
- enum catalog identity or values;
- schema mutation semantics;
- query planning or execution;
- cursor or index-key formats.

Removing an unused generated Rust implementation cannot alter runtime database
meaning. Any implementation attempt that changes accepted schema content or
missing-slot behavior is out of scope.

## Public API Impact

This is an intentional generated Rust API break.

Downstream code may lose `Type::default()` for domain types that did not
explicitly opt in. The compiler identifies those call sites. The migration is
to state domain intent explicitly, not to restore blanket behavior.

Collection defaults and generated create-input defaults remain available, so
ordinary empty collection construction and sparse create-input building do not
regress.

## Non-Goals

0.202 does not include:

- database-default redesign;
- accepted-schema absence-policy changes;
- persisted-format changes;
- automatic inference of `traits(add(Default))` from constructible fields;
- restoring `Default` to `Principal`, `Account`, or `Ulid`;
- removing ordinary defaults from numeric, temporal, collection, or other core
  types owned outside schema derive;
- redesigning insert generators or managed-write construction;
- changing generated create-input omission semantics;
- broad validator or sanitizer refactoring;
- a compatibility period for implicit domain defaults;
- aliases for the former behavior;
- opportunistic schema macro syntax redesign.

## Implementation Slices

### Slice 1: Trait Selection And Validation

- Remove `Default` from `TYPE_TRAITS`.
- Add it locally for list, set, and map nodes.
- Preserve generated create-input defaults.
- Make entity, record, newtype, enum, and tuple strategies respond only to
  explicit trait selection.
- Convert impossible explicit requests into targeted macro errors.
- Allow enums with no default variant when `Default` is not selected.
- Reject unused enum default markers.

This slice owns derive behavior and focused derive tests. It must not touch
runtime default materialization.

### Slice 2: Active Schema And Call-Site Migration

- Inventory active generated domain types and `Type::default()` uses.
- Add explicit trait intent only where the domain meaning is defensible.
- Replace fixture-specific defaults with explicit values.
- Remove arbitrary enum default markers.
- Update current examples and macro fixtures.

This slice should remain schema/test-facing. Unrelated uses of `Default` on
runtime DTOs, metrics, planner state, or ordinary Rust containers are not part
of the migration.

### Slice 3: Closeout

- Confirm accepted database-default and missing-slot tests remain unchanged in
  meaning.
- Confirm no runtime generic entity/type bound gained `Default`.
- Update active schema authoring documentation.
- Add the root `CHANGELOG.md` `Unreleased` note under changelog governance.
- Record code-size and complexity deltas.

Slices may land together when the resulting change remains within the normal
review limits, but their ownership and validation evidence should remain
separable.

## Validation Plan

### Trait selection

- A required-field entity without `Default` compiles and implements all normal
  entity contracts.
- A required-field record without `Default` compiles and remains usable as a
  structured field value.
- An enum with no marked default variant compiles and participates in the
  current enum catalog/runtime contracts.
- A tuple containing a non-defaultable element compiles when `Default` is not
  requested.
- Lists, sets, and maps still construct empty values through `Default` without
  requiring defaultable contained types.

### Explicit success

- Constructible entities and records with `traits(add(Default))` receive the
  intended derived or manual implementation.
- Newtypes with an explicit constructor and trait opt-in construct the intended
  value.
- Unit and payload enums with valid explicit defaults construct the intended
  variant.
- Fully defaultable tuples opt in successfully.
- Generated create inputs remain default-constructible independently of their
  entity.

### Explicit rejection

- Entity and record requests identify missing construction fields.
- Newtype requests without a constructor fail locally.
- Enum requests with zero or multiple default variants fail locally.
- Enum default markers without trait intent fail locally.
- Payload enum and tuple element trait failures point to the generated type
  boundary.
- `traits(remove(Default))` fails as an invalid removal on domain types.

Coverage should prove the maintained explicit surface and normal use of
non-defaultable types. It should not retain tests whose sole purpose is
memorializing obsolete implicit behavior.

### Runtime separation

- Existing accepted field-default proposal, publication, decode, and sparse
  write tests continue to pass without consulting Rust `Default`.
- Required missing persisted fields without accepted defaults still fail
  closed.
- Nullable absence and accepted encoded defaults retain their current behavior.
- Generated create materialization still distinguishes authored slots from
  fallback construction.

### Broader checks

- Run formatting after implementation edits.
- Run focused schema-derive and macro compile suites first.
- Run schema fixture and facade checks once the migration is complete.
- Run broader workspace checks only when the slice is otherwise ready.

## Cost And Complexity Expectations

Runtime performance should remain neutral. The change affects generated trait
surface and compile-time validation, not query or persistence execution.

Raw non-gzipped wasm size is expected to remain neutral or decrease slightly
where unused generated implementations disappear. Any measured report should
use raw `.wasm` bytes as the primary signal and gzip only as secondary context.

Implementation complexity should stay neutral or improve:

- one global implicit rule is removed;
- node-local intrinsic defaults become explicit;
- silent strategy suppression becomes direct validation;
- domain intent becomes visible in schema source.

The closeout report should include files touched, approximate line delta, raw
wasm delta where measured, and whether the final implementation shape became
simpler, stayed neutral, or became more complex.

## Risks

### Mechanical opt-in everywhere

The migration could preserve the old policy by adding `traits(add(Default))` to
every schema type. Review must require a domain reason for each opt-in.

### Database-default confusion

Developers may assume removal of Rust `Default` changes field omission or
database behavior. Documentation and diagnostics must consistently name Rust
construction versus accepted database defaults.

### Hidden downstream API use

Downstream crates may call generated `Type::default()` even when this repository
does not. This is an intended pre-1.0 break. Release notes must provide the
explicit-construction migration.

### Poor trait-bound diagnostics

Payload enums and tuples may otherwise fail with remote derive errors. Emit
type-local assertions or validation context so failures identify the requested
`Default` boundary.

### Accidental runtime coupling

Implementation work in shared default helpers could drift into persisted-row
fallback behavior. Runtime absence policy is out of scope and must remain
accepted-schema-owned.

## Closeout Standard

0.202 is complete only when:

1. `Default` is absent from the shared type-trait list.
2. Entities, records, newtypes, enums, and tuples receive it only through
   explicit `traits(add(Default))`.
3. Lists, sets, maps, generated create inputs, and any retained internal marker
   defaults have documented node-local ownership.
4. Impossible explicit requests fail at the owning macro boundary.
5. Enums without Rust defaults require no arbitrary default variant.
6. Field and newtype default metadata do not implicitly add the Rust trait.
7. Accepted database-default and missing-slot behavior is unchanged.
8. Active fixtures and examples express current intent rather than mechanically
   restoring old behavior.
9. No compatibility alias, fallback, blanket implementation, or dual policy is
   retained.
10. Focused macro, schema, and runtime-separation validation passes.
11. Complexity and raw wasm-size effects are reported with the implementation.
