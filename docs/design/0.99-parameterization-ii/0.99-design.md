# 0.99 Parameterization II: Symbolic Prepared Templates

## Thesis

`0.99` should replace sentinel-value substitution with symbolic parameter slots inside the internal prepared-template lane so fixed-route prepared execution no longer depends on collision-resistant concrete values.

> **0.99 upgrades prepared SQL template execution from collision-resistant sentinel substitution to symbolic parameter slots, preserving the fixed-route prepared model while removing small-domain binding limits and widening the admitted template lane to compare-family shapes such as bool.**

This is an internal representation upgrade, not a restart of the feature.

`0.99` does **not** introduce planner-wide symbolic parameter nodes. It upgrades only the internal prepared-template representation used by the fixed-route prepared execution lane.

The current `0.98.1` shape already stores:

* prepared statement
* frozen parameter contracts
* optional execution template

So `0.99` should upgrade the template representation without redoing the prepare/execute split.

---

## Problem statement

`0.98.1` uses collision-resistant non-null sentinel values to freeze internal prepared templates.

That works for large-domain compare-family bindings, but it creates two avoidable problems:

* it artificially excludes small-domain and collision-prone kinds such as `bool`
* it forces execute-time sentinel replacement through frozen template payloads

The current design therefore proves the prepared-template lane is useful, but it encodes parameter identity indirectly through concrete stand-in `Value`s.

`0.99` removes that limitation by preserving parameter identity symbolically inside the prepared-template lane.

---

## Contracts

The slice should preserve the existing prepared-execution contract while upgrading the internal representation:

* fixed-route prepared planning remains fixed-route
* `prepare` still freezes one template shape
* `execute` binds values without replanning
* unsupported and value-sensitive families still fall back
* slot identity is preserved symbolically, not encoded through concrete stand-in values

---

## Target model

### Before (`0.98.1`)

```text
prepare
  -> freeze template using sentinel Values
execute
  -> replace sentinels with runtime bindings
```

### After (`0.99`)

```text
prepare
  -> freeze template using symbolic Slot references
execute
  -> read runtime bindings through slot environment
```

### External behavior stays the same

* fixed-route compare-family only
* unsupported and value-sensitive families still defer to fallback behavior
* the fallback path still exists for anything out of scope

---

## Core design changes

### 1. Replace `template_binding: Option<Value>` with symbolic slot metadata

Today the contract carries an optional template binding specifically to enable the collision-resistant template lane.

In `0.99`, the frozen contract should instead carry:

* slot index
* type family
* null rule
* compare or HAVING role when needed for validation

It should **not** carry a concrete stand-in `Value`.

---

### 2. Add symbolic slot nodes to the internal template representation

This does **not** need to become planner-wide `Expr::Param` everywhere. Keep the slice smaller.

Only the prepared execution template representation becomes slot-aware. That means:

* access-path payloads can reference `Slot(0)`, `Slot(1)`, and so on
* compare leaves can reference symbolic slots
* grouped `HAVING` compare leaves can reference symbolic slots
* aggregate compare thresholds can reference symbolic slots

This is the main upgrade. It removes the need for functions such as `bind_prepared_template_value(...)` to search and replace concrete sentinel values throughout:

* key ranges
* index ranges
* prefix payloads
* grouped predicates
* similar frozen template payloads

That replacement machinery exists today because the template is frozen as concrete `Value`s.

---

### 3. Keep fixed-route planning exactly as-is

`0.99` should not mix this representation upgrade with value-sensitive route templates.

The slice should preserve:

* the same admitted fixed-route subset as `0.98.1`
* the same prepare-time planning
* the same fallback for value-sensitive families

That means no `LIKE ?` template specialization yet. The existing `0.98` scope already keeps that out, and `0.99` should keep it out too.

---

## Feature payoff

The representation change is the main point of the slice. The feature widening follows from that representation change.

Once the template is symbolic instead of sentinel-based:

* sentinel generation is no longer required for admitted template shapes
* small-domain bindings no longer need collision-resistant stand-ins
* `bool` becomes a natural fixed-route compare-family admission target

That means the immediate payoff for `0.99` is not "bool as a feature in isolation." The payoff is that the prepared-template lane stops depending on a workaround representation, and `bool` is the clearest proof that the workaround has been removed.

### Bool and small-domain admission
Once the template is symbolic instead of sentinel-based:

* `Bool` no longer needs a unique non-null stand-in
* multi-slot bool compares stop being collision-prone
* other small-domain kinds become plausible on the fixed-route template lane where the existing fixed-route compare-family rules still hold

Today, cases like `FieldKind::Bool | ... => None` are limitations of the sentinel scheme, not of prepared execution itself.

---

## Non-goals

`0.99` should **not** combine this slice with:

* value-sensitive parameterization
* `LIKE ?` or prefix-family route templates
* broad SQL surface widening
* full planner-wide parameter nodes
* a generic bytecode or VM layer

Combining those concerns would turn one bounded upgrade into several overlapping slices.

---

## Practical scope

### In scope

* internal prepared-template v2
* symbolic slot references in fixed-route template plans
* execution-time binding environment for template evaluation
* bool compare-family support
* continued support for grouped mixed-slot queries on the fixed-route lane
* removal or sharp reduction of sentinel replacement machinery from the template lane

### Out of scope

* value-sensitive prepared templates
* non-compare-family widening
* `ORDER BY` or projection parameter widening unless those shapes are already in the admitted fixed-route subset
* public API redesign

---

## Likely code seams

The exact file split may move, but the implementation will likely touch these boundaries:

* `db/session/sql/parameter.rs`
* `db/sql/lowering/prepare.rs`
* `PreparedSqlParameterContract`
* `PreparedSqlExecutionTemplate`
* template bind or rewrite helpers that currently perform sentinel substitution
* runtime compare and grouped `HAVING` template readers

---

## Why this is the right size

`0.98.1` already isolated the prepared-template fast path as a distinct lane that bypasses raw SQL recompilation and the shared structural cache.

`0.99` upgrades that lane's internal representation instead of changing the whole engine. This is therefore not a rewrite. It is a contained v2 of the template mechanism.

---

## Risks

The main risks in this slice are boundary failures rather than algorithmic novelty:

* widening scope into value-sensitive templates
* accidentally introducing planner-wide symbolic parameter machinery
* preserving fallback behavior incorrectly
* creating subtle drift between fixed-route template execution and fallback execution

---

## Recommended acceptance criteria

* repeated prepared executions still bypass raw SQL recompilation in the template lane, as `0.98.1` already does for supported shapes
* the shared structural query-plan cache remains untouched by repeated fixed-route template executions, matching the current template fast-path behavior
* bool compare-family prepared queries are admitted and reusable on the template lane
* grouped `HAVING` mixed-slot queries still work with symbolic slots just as they do today with mixed compare contracts on the fixed-route lane
* sentinel-specific binding generation and rewrite are no longer part of execution for admitted symbolic-slot templates

---

## Summary

This is a medium slice, but it is bounded and plausible as `0.99`.

It is the cleanest parameterization v2 because it upgrades the prepared-template lane directly without dragging in value-sensitive planning or broader planner architecture changes.
