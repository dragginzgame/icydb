# 0.67 Reopen Packets

This file records the bounded `0.67` evaluation seams as reopen-only proposal
packets.

Its job is simple:

* define the exact decision question
* define what is allowed to reopen as a later `0.67` candidate
* define what evidence is required before a candidate can close as `keep`,
  `merge`, or `reject`

This file does not authorize implementation by itself.

Any code slice opened from here must still satisfy the main constraints in
[0.67-design.md](/home/adam/projects/icydb/docs/design/0.67-feature-coherence/0.67-design.md),
especially:

* no new lanes
* bounded reopen-only scope
* wasm and perf remain keep/reject gates

All opening packets are now closed at the current `0.67` checkpoint.
Anything opened from here is a reopen request, not unfinished opening work.
At this stage these packets belong to Phase 5 reopen-only follow-up work, not
to the already-completed opening phases.
The shipped `0.67.1` and `0.67.2` write-path slices did not reopen any packet,
because they stayed inside the existing lane and contract model.

## Packet A — Aggregate Lane Evaluation

Current `0.67` resolution: this packet is closed as `keep`.

That means the dedicated aggregate lane remains the accepted baseline for the
current checkpoint. This packet only reopens if one bounded `COUNT(*)`-first
merge candidate later earns one bounded Phase 5 reopen slot.

### Decision Question

If this packet is reopened later in `0.67`, the decision question is:

Should constrained global aggregate SQL remain on the dedicated
`execute_sql_aggregate(...)` lane, or should one bounded subset merge into the
typed dispatch SQL surface?

### Current Boundary

Today:

* constrained global aggregate SQL executes through
  `execute_sql_aggregate(...)`
* non-aggregate dispatch SQL executes through `execute_sql_dispatch(...)`
* explain can already describe constrained aggregate SQL without merging the
  execution result contract

This means the current split is not a parser capability gap. It is a public
result-contract choice.

### What Counts As A Valid Candidate

A valid reopened `0.67` candidate must:

* extend one existing lane
* keep the grouped lane unchanged
* define the replacement public result contract before implementation
* identify exactly which aggregate shape is being evaluated

Allowed candidate scope:

* one constrained global aggregate shape only
* one replacement typed-dispatch result contract
* one explicit explain-behavior story

Disallowed candidate scope:

* broad aggregate-lane collapse
* grouped and global aggregate unification in the same slice
* new sidecar aggregate lane or adapter lane
* widening `query_from_sql(...)` in the same patch

### Preferred First Candidate

If `0.67` reopens one aggregate merge experiment, it should start with
`COUNT(*)`.

Why:

* it is the clearest aggregate-shape decision point
* it is already called out in the main design doc as the current ambiguity
  seam
* it is easier to explain and measure than a broader terminal set

### Required Proposal Contents

Any reopened aggregate-lane proposal must answer:

1. What concrete result contract replaces the current scalar `Value` contract?
2. How does `COUNT(*)` render on the widened surface?
3. Does `EXPLAIN` stay clearer, or get harder to understand?
4. What exact user-facing coherence problem does this solve?
5. Why is that coherence gain worth the contract widening and retained-cost
   risk?

### Acceptance Gates

An aggregate candidate may only close as `merge` if all of the following are
true:

* the replacement result contract is explicit and stable enough to test
* `COUNT(*)` no longer depends on accidental lane ownership
* explain output remains at least as honest as the current split
* raw wasm remains acceptable against current `minimal` and `one_simple`
  baselines
* perf remains acceptable against the current quickstart matrix
* the user-facing coherence gain is clearer than the added contract weight

### Keep Closure

This packet closes as `keep` if:

* the dedicated aggregate lane remains easier to explain than a widened
  dispatch result contract
* `COUNT(*)` and the other supported terminals are explicitly recorded as
  dedicated aggregate-lane behavior
* no bounded merge candidate clears the acceptance gates above

### Reject Closure

This packet closes as `reject` if:

* the best bounded merge candidate still widens the contract without enough
  coherence value
* or the candidate fails wasm/perf gates
* or the candidate makes explain semantics less honest or less readable

## Packet B — Planner `STARTS_WITH(...)` Evaluation

Current `0.67` resolution: this packet is closed as `merge`.

That means the bounded `LIKE '<prefix>%'` family remains in place and one
bounded direct `STARTS_WITH(...)` family is now also accepted.
This packet only reopens if broader text-predicate surface changes later earn
one bounded Phase 5 reopen slot.

### Decision Question

If this packet is reopened later in `0.67`, the decision question is:

Should reduced SQL keep the current bounded planner-bearing prefix spellings as
they are, or should broader direct text-predicate surface be added?

### Current Boundary

Today:

* planner-bearing prefix semantics already exist through:
  * `<field> LIKE '<prefix>%'`
  * `LOWER(<field>) LIKE '<prefix>%'`
  * `UPPER(<field>) LIKE '<prefix>%'`
* runtime text-function semantics already exist for `STARTS_WITH(...)`
* bounded direct predicate spellings are now also accepted:
  * `WHERE STARTS_WITH(<field>, '<prefix>')`
  * `WHERE STARTS_WITH(LOWER(<field>), '<prefix>')`
  * `WHERE STARTS_WITH(UPPER(<field>), '<prefix>')`
* those accepted direct spellings are now locked consistently across
  executable `SELECT`, executable `DELETE`, `EXPLAIN DELETE`, and
  `EXPLAIN JSON DELETE`

So the open question is no longer capability. It is whether any broader
predicate surface would add enough coherence value to justify itself.

### What Counts As A Valid Candidate

A valid reopened `0.67` candidate must:

* preserve the existing exact direct predicate form
* preserve existing capability checks
* preserve honest explain output
* avoid opening broader text-predicate or expression work

Allowed candidate scope:

* one bounded broadening step beyond the accepted direct spelling
* one explicit explain story for the recognized path

Disallowed candidate scope:

* generic text-function predicates
* case-insensitive expansion beyond the existing bounded `LIKE` family
* broader expression-engine work
* widening other SQL function spellings in the same slice

### Required Proposal Contents

Any reopened planner-`STARTS_WITH(...)` proposal must answer:

1. What exact predicate syntax is being added?
2. How does it differ from the existing bounded `LIKE` path?
3. What exact planner capability check proves it is semantically honest?
4. How does explain present the new spelling without overstating capability?
5. What user-facing coherence problem does this solve that the current `LIKE`
   path does not?

### Acceptance Gates

A planner-`STARTS_WITH(...)` candidate may only close as `merge` if all of the
following are true:

* the recognized form is exact, not heuristic
* explain remains honest about access-path selection
* the new spelling solves a real user-facing coherence problem
* raw wasm cost is acceptable against current release baselines
* the new spelling does not force broader text-expression widening

### Keep Closure

This packet closes as `keep` if:

* the current bounded spelling set remains unchanged:
  * the `LIKE '<prefix>%'` family
  * direct `STARTS_WITH(...)` for plain fields and LOWER/UPPER field wrappers
  * the same accepted direct family across `DELETE`, `EXPLAIN DELETE`, and
    `EXPLAIN JSON DELETE`
* no broader candidate beats the current bounded set on exactness and wasm
  cost

### Reject Closure

This packet closes as `reject` if:

* the measured candidate does not materially improve user-facing coherence
* or it duplicates the existing `LIKE` path without enough value
* or it regresses explain honesty or raw wasm cost

## How To Use These Packets

When a `0.67` candidate is proposed:

1. Name the packet it belongs to.
2. State whether it is trying to close as `keep`, `merge`, or `reject`.
3. Fill in the required proposal contents before opening implementation work.
4. Reject the slice immediately if it needs a new lane, a sidecar result
   contract, or a second unrelated surface change.

If a candidate cannot satisfy those steps, it is not yet a valid `0.67`
coherence slice.
