# 0.67 SQL Surface Matrix

This file is the public SQL surface inventory for the current `0.67.x` line.

Its job is simple:

- record what each public SQL-facing surface actually supports today
- make the deliberate surface splits explicit
- keep the `0.67` decisions grounded in shipped behavior instead of older
  assumptions
- make it obvious when a later `0.67` patch changed behavior versus when it
  only changed internal cost or implementation shape

## Public Surface Inventory

| Surface | Result contract | `SELECT` / `DELETE` execution | Computed text projection | Grouped SQL execution | Global aggregate execution | `EXPLAIN` | `DESCRIBE` / `SHOW *` | Intentional boundary |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| `query_from_sql(...)` | `Query<E>` | Yes, as lowered structural intent | No | Yes, as lowered structural intent | No | No | No | Structural-only lowered query builder |
| `execute_sql(...)` | `Response<E>` | Yes | No | No | No | No | No | Entity-row execution only |
| `execute_sql_dispatch(...)` | `SqlQueryResult` | Yes | Yes | No | No | Yes | Yes | Main typed unified SQL surface |
| `execute_sql_grouped(...)` | `PagedGroupedResponse` | Grouped `SELECT` only | No | Yes | No | No | No | Dedicated grouped execution lane |
| `execute_sql_aggregate(...)` | `Value` | No | No | No | Yes | No | No | Dedicated global aggregate lane |
| generated `sql_dispatch::query(...)` | `SqlQueryResult` | Yes | Yes | No | No | Yes | Yes | Generated canister SQL facade |

## Interpretation

- `query_from_sql(...)` is intentionally narrower than the dispatch surfaces.
  It builds typed structural intent and stops there.
- Computed text projection is currently dispatch-owned. It is available on
  `execute_sql_dispatch(...)` and generated `sql_dispatch::query(...)`, but not
  on `query_from_sql(...)` or `execute_sql(...)`.
- Grouped and global aggregate execution still travel through dedicated lanes:
  `execute_sql_grouped(...)` and `execute_sql_aggregate(...)`.
- `EXPLAIN` is broader than execution alone. Dispatch-style surfaces can render
  explain output for reduced grouped and constrained global aggregate SQL even
  though those execution shapes stay on dedicated lanes.
- The generated canister helper is broader than its internal query/explain core.
  The outer public generated facade also owns `DESCRIBE` and `SHOW *` routing.
- The generated surface is a constrained projection of the typed dispatch lane,
  not an independent SQL feature system.
- `0.67.1` did not change this matrix. It reduced one-row mutation fixed cost
  while leaving the public SQL surface unchanged.
- `0.67.2` also did not change this matrix. It reduced typed-write and batch
  insert cost while leaving the public SQL surface unchanged.

    predicate spellings outside the current subset

## 0.67 Decision Hooks

This matrix is the baseline for the carried-forward `0.66` deferrals and the
shipped `0.67` bounded merge.

The explicit decision items are now closed baseline records plus reopen
conditions:

1. Keep the structural-only `query_from_sql(...)` boundary unless one bounded
   reopen candidate later clears the gates.
2. Reopen the dedicated global aggregate lane only if one bounded
   `COUNT(*)`-first merge candidate later clears the `0.67` gates.
3. Keep the current bounded planner-bearing prefix spellings unless one later
   reopen candidate clearly beats them:
   - `LIKE '<prefix>%'`
   - direct `STARTS_WITH(...)` for plain fields and LOWER/UPPER field wrappers

Until one of those decisions changes, this matrix should stay aligned with the
shipped code and with [SQL_SUBSET.md](/home/adam/projects/icydb/docs/contracts/SQL_SUBSET.md).
