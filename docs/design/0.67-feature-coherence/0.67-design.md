## Motivation

`0.66` shipped more than the opening design note originally promised:

* the staged text-function surface is live
* generated `sql_dispatch` now supports computed projection parity
* generated `sql_dispatch` now supports real `DELETE` and `EXPLAIN DELETE`
* the perf harness and release wasm audit lines are both in place

That means the main open problem is no longer “missing text functions.”
The open problem is **surface coherence**:

* some shipped behavior is not reflected cleanly in the docs
* some intentionally deferred boundaries from `0.66` still need an explicit
  `0.67` decision
* some user-visible SQL capabilities still travel through different public
  surfaces with different contracts

So `0.67` should begin from the actual shipped `0.66` state, not from the older
opening assumptions.

## Carried-Forward Deferrals From 0.66

These are the real `0.66` deferrals that now exist as explicit `0.67`
decision records:

### 1. Planner-recognized `STARTS_WITH(...)`

Opening `0.67` resolution: accept one bounded direct planner-bearing spelling
alongside the existing `LIKE` path.

The `0.66` design explicitly left planner recognition optional unless it proved
cheap, exact, and semantically honest. That remains true.

### 2. `query_from_sql(...)` stays structural-only

Opening `0.67` resolution: keep the structural-only boundary.

The current boundary is:

* `query_from_sql(...)` rejects computed text projection
* callers are directed to `execute_sql_dispatch(...)`

This is now treated as an intentional boundary between planner-owned structural
intent and dispatch-owned session semantics. The only reason to reopen it later
would be one bounded widening slice with clear user-facing value.

### 3. Aggregate SQL remains on a dedicated lane

Opening `0.67` resolution: keep the dedicated lane.

The current surface is:

* scalar SQL through `execute_sql(...)`
* projection/explain/metadata SQL through `execute_sql_dispatch(...)` and the
  generated canister `sql_dispatch::query(...)` facade
* constrained scalar aggregate SQL through `execute_sql_aggregate(...)`

This is now treated as an intentional public result-contract boundary, not as a
missing-capability gap. The only reason to reopen it later would be one bounded
`COUNT(*)`-first merge candidate that clearly beats the current split on
coherence, wasm, and perf.

### 4. SQL contract/docs drift

This is already real.

The code now supports generated computed projection parity and generated
`DELETE` / `EXPLAIN DELETE`, but older contract text still describes a narrower
generated query surface.

`0.67` should treat that drift as work, not as a documentation footnote.

## Goal

`0.67` should make the shipped SQL surface **explicit, coherent, and auditable**
without reopening broad new SQL syntax work.

Success means:

1. the documented SQL subset matches the current shipped implementation
2. every carried-forward `0.66` deferral has an explicit `keep`,
   `documented`, `merge`, or `reject` decision
3. no new parallel lane is introduced just to patch over ambiguity
4. any actual unification slice is justified by user-facing coherence, not by
   abstract architectural neatness
5. wasm and perf remain first-class keep/reject gates for every slice

## Working Definitions

### Coherence

For `0.67`, coherence is a mechanical property, not an aesthetic one.

It means:

* no SQL feature is exposed through multiple public APIs with different
  semantics unless that difference is:
  * intentional
  * documented
  * test-covered
* every shipped SQL feature appears in the surface inventory exactly once
* every shipped SQL construct maps to one lowering path per surface

### Lane

A lane is a distinct lowering plus execution path with:

* its own entrypoint
* its own result contract
* its own explain semantics

Examples in the current line:

* structural query lane: `query_from_sql(...) -> Query<E>`
* typed dispatch SQL lane: `execute_sql_dispatch(...) -> SqlQueryResult`
* aggregate SQL lane: `execute_sql_aggregate(...) -> Value`
* generated canister SQL lane: `sql_dispatch::query(...) -> SqlQueryResult`

---

## Non-Goals

* Full SQL parity with PostgreSQL or SQLite
* New broad syntax such as joins, CTEs, unions, or generic expression engines
* Large up-front IR rewrites without a concrete paying surface
* Lane unification purely for aesthetic consistency

## Real Problems To Solve

### 1. Shipped behavior is ahead of the written contract

The repo already ships features that older `0.66` text does not describe
accurately.

Examples:

* generated `sql_dispatch` now supports computed projection
* generated `sql_dispatch` now supports real `DELETE`
* generated `sql_dispatch` now supports `EXPLAIN DELETE`

This means the first `0.67` deliverable is not new code. It is a reliable
surface inventory.

### 2. A few public SQL surfaces still have deliberate contract splits

The important ones are:

* `query_from_sql(...)` vs `execute_sql_dispatch(...)`
* aggregate SQL vs non-aggregate SQL result lanes
* bounded planner-bearing prefix spellings vs any broader text-predicate
  surface

These are narrow enough to evaluate directly.

### 3. The current design draft should not overcommit to one answer

The earlier `0.67` draft assumed:

* one normalized SQL IR
* lane collapse around aggregate handling
* aggregate unification as the primary direction

That is too strong as an opening premise.

For `0.67`, those are **possible solutions**, not the starting contract.

## Coherence Invariants

These are the operational `0.67` invariants.

1. No feature is exposed through multiple public APIs with different semantics
   unless the difference is explicitly documented.
2. Every supported SQL feature appears in exactly one row of the surface
   matrix.
3. Every supported SQL construct maps to one lowering path per surface.
4. Differences between surfaces must be:
   * intentional
   * documented
   * test-covered

If a proposed slice cannot be explained in terms of these invariants, it is not
yet a `0.67` coherence slice.

## Design Principles

### 1. Document actual shipped behavior first

No `0.67` implementation work should begin from stale `0.66` assumptions.

### 2. Treat each carried-forward deferral as a decision record

For each deferred item, `0.67` should explicitly answer one of:

* keep as-is
* widen
* merge
* reject

### 3. Keep subset boundaries explicit

If two surfaces intentionally differ, that difference must be:

* documented
* tested
* justified

### 4. Do not introduce generic scaffolding without a paying slice

New IR or shared abstraction should only land if it solves a concrete surface
coherence problem and survives wasm/perf measurement.

### 5. Result-shape contracts matter

The aggregate split cannot be evaluated purely as “fewer lanes.”

It changes public result-shape expectations:

* aggregate surface returns one scalar-style value contract
* non-aggregate SQL returns row/explain/metadata payloads

So any merge must justify that contract cost explicitly.

## 0.67 Constraint — No New Lanes

No new execution lane may be introduced in `0.67`.

Any `0.67` feature or coherence slice must:

* extend an existing lane
* clarify and document an existing lane
* or be rejected

This is a hard constraint for the line-opening work. `0.67` is not allowed to
create a new lane just to avoid making a keep/merge decision on an existing
surface split.

### Decision State Meanings

The decision register uses a small fixed vocabulary.

* `documented`
  * the shipped behavior is already accepted
  * the line only needed contract and matrix alignment
  * no implementation slice is implied
* `keep`
  * the current boundary is intentionally preserved
  * closure requires docs plus regression coverage
  * no widening work should open unless this record is explicitly replaced
* `evaluate`
  * the current boundary remains the default
  * the line still owes an explicit keep/merge/reject closure
  * no code slice may open unless one concrete proposal clears the entry
* `merge`
  * only valid once a bounded slice is approved
  * requires a user-facing coherence gain, not just structural neatness
  * requires wasm and perf gates before acceptance
* `reject`
  * the proposed widening or merge was evaluated and declined
  * the current boundary remains in force
  * the rejection itself counts as a closed `0.67` outcome

### Opening Decision Register

| Item | Opening `0.67` status | Default action | What would justify a change |
| --- | --- | --- | --- |
| `query_from_sql(...)` computed projection boundary | `keep` | Keep structural-only | Strong product need for `Query<E>` to own session-style computed transforms without muddying planner-owned intent |
| Aggregate SQL dedicated lane | `keep` | Keep the dedicated lane | One bounded `COUNT(*)`-first merge candidate that clearly beats the current split on result-contract clarity, wasm, and perf |
| Planner-recognized `STARTS_WITH(...)` family | `merge` | Keep the accepted bounded direct spelling alongside the existing `LIKE` planner path | Only if broader text-predicate surface changes later |
| Generated-query surface contract drift | `documented` | Keep shipped generated subset and document it accurately | Only changes if the generated public subset itself changes |

This means `0.67` does **not** open by assuming lane collapse.

It opens with:

* two explicit `keep` decisions
* one explicit bounded `merge` decision
* one contract-rebaseline item that is already satisfied by the opening docs

### Decision Closure Rule

An item is only closed for `0.67` when all of the following are true:

* its register status is one of:
  * `documented`
  * `keep`
  * `merge`
  * `reject`
* the design doc rationale matches the current status
* the surface matrix and subset contract reflect the same decision
* the relevant regression coverage exists for the kept or widened boundary

Until then, `evaluate` means “still open,” not “softly accepted.”

### A. `query_from_sql(...)` and computed text projection

Current state:

* `query_from_sql(...)` is structural-only
* computed projection is available through dispatch/session SQL

Decision options:

1. Keep structural-only
2. Widen to support the current computed text subset

Current recommendation:

* default to **keep structural-only** unless there is strong product value in
  making `Query<E>` carry session-owned computed transforms

Opening `0.67` status:

* `keep` unless one concrete user-facing slice proves this split is harmful

Why:

* it keeps `Query<E>` closer to planner-owned structural intent
* it avoids silently widening a core generic contract
* the project already has an actionable alternative surface

Opening `0.67` keep record:

* this boundary is already regression-covered in both:
  * core session SQL tests
  * public facade session tests
* `0.67` should treat that existing coverage as the default lock on the
  structural-only contract
* any widening proposal should begin by replacing this `keep` record with one
  concrete user-facing rationale, not by arguing from aesthetic lane symmetry

### B. Aggregate SQL split

Current state:

* constrained global aggregate SQL uses `execute_sql_aggregate(...)`
* non-aggregate SQL uses row/explain/metadata-oriented paths

Decision options:

1. Keep dedicated aggregate lane
2. Merge aggregate lowering and execution into the broader SQL dispatch surface

Current recommendation:

* keep the dedicated aggregate lane in the opening `0.67` line

Opening `0.67` status:

* `keep`, unless one bounded `COUNT(*)`-first merge candidate later proves
  materially better

Why:

* the current split is already clearer than any unproven widened result
  contract
* the shipped surface already has a working aggregate story
* the burden of proof is still on a merge proposal, not on the existing lane

Opening `0.67` keep record:

* the current split is already regression-covered at:
  * core execution surfaces
  * public facade execution surfaces
  * integration/perf sampling for grouped and aggregate lanes
* `0.67` should treat that existing coverage as the default lock on the
  dedicated aggregate contract
* reopening this lane now would widen a public result contract without enough
  proven user-facing gain

Example coherence target:

* `COUNT(*)` is the clearest aggregate-shape decision point in the current line
* `0.67` should ensure that `COUNT(*)` is either:
  * explicitly frozen as part of the dedicated aggregate lane contract
  * or normalized into a later unified aggregate result contract
* what `0.67` should not do is leave `COUNT(*)` as an accidental special case
  with unclear lane ownership

Closure paths:

* close as `keep` if:
  * the dedicated aggregate lane remains clearer for users than any widened
    row/explain/metadata result contract
  * `COUNT(*)` and the other supported terminals are explicitly recorded as
    dedicated aggregate-lane behavior
  * no measured merge candidate clears wasm and perf gates
* close as `merge` only if:
  * one bounded slice defines the replacement public result contract up front
  * `COUNT(*)` no longer depends on special lane ambiguity
  * explain output stays at least as honest and readable as the current split
  * raw wasm and perf deltas are acceptable against the current baselines
* close as `reject` if:
  * the best measured merge candidate fails the result-contract test
  * or fails raw wasm/perf gates without enough user-facing coherence value

Opening `0.67` closure:

* this packet is closed as `keep` for the line opening
* any future merge attempt should be treated as a reopen request against this
  keep record, not as unfinished baseline work

### C. Planner-recognized `STARTS_WITH(...)`

Current state:

* runtime semantics exist
* one bounded direct planner-bearing spelling now exists

Decision options:

1. Keep the accepted bounded direct spelling as-is
2. Broaden beyond the accepted direct spelling

Current recommendation:

* keep the accepted slice narrow and do not broaden it into generic
  text-function predicates

Opening `0.67` status:

* `merge`, with one bounded direct spelling now accepted on the reduced SQL
  lane

Opening `0.67` merge record:

* current SQL already has one canonical planner-bearing prefix path:
  * `<field> LIKE '<prefix>%'`
  * `LOWER(<field>) LIKE '<prefix>%'`
  * `UPPER(<field>) LIKE '<prefix>%'`
  * `STARTS_WITH(...)` for plain fields and LOWER/UPPER field wrappers
  * `STARTS_WITH(LOWER(<field>), '<prefix>')`
  * `STARTS_WITH(UPPER(<field>), '<prefix>')`
* those shapes are already covered by core lowering tests, core session tests,
  and public facade tests that prove they lower to the existing strict or
  casefold `StartsWith` intent
* the accepted direct spelling is still intentionally narrow:
  * plain-field and LOWER/UPPER field-wrapper forms only
  * strict text intent only
  * no generic text-function predicate widening
* the existing bounded `LIKE` family remains in place alongside the direct
  spelling

Closure paths:

* close as `merge` because:
  * the accepted direct spelling is exact, not heuristic
  * explain remains honest about the resulting access path
  * the new spelling improves SQL coherence without opening a new lane
  * broader text-function predicate forms remain fail-closed
* close as `reject` if:
  * any later broadening candidate cannot justify itself against the current
    bounded spelling set

Opening `0.67` closure:

* this packet is closed as `merge` for the current line
* any future broadening attempt should be treated as a reopen request against
  this bounded merge record, not as unfinished baseline work

### D. Generated-query surface contract

Current state:

* generated surface is narrower than typed SQL overall
* but broader than older docs currently say

Role:

* the generated SQL surface is a constrained projection of the typed dispatch
  SQL lane, not an independent feature system

Required outcome:

* `0.67` must publish the actual generated subset explicitly

This is not optional.

Opening `0.67` status:

* `documented` once [SQL_SUBSET.md](/home/adam/projects/icydb/docs/contracts/SQL_SUBSET.md)
  and
  [0.67-surface-matrix.md](/home/adam/projects/icydb/docs/design/0.67-feature-coherence/0.67-surface-matrix.md)
  match the shipped code

Generated-surface constraints:

* reuse the same lowering and execution semantics as the typed dispatch lane
  where possible
* enforce restrictions at the boundary instead of inventing alternate
  semantics
* never introduce unique user-visible SQL behavior that only exists on the
  generated canister helper

## Rejection Is Success

`0.67` does not require a code change to succeed.

If all evaluated items resolve to:

* `keep`
* `document`
* or `reject`

then `0.67` is still successful.

That is a valid outcome for this line because the primary goal is decision
clarity and surface governance, not forced implementation churn.

## Validation

Required for any `0.67` keep/reject decision:

* `cargo test -p icydb-core db::session::tests -- --nocapture`
* `cargo test -p canister_quickstart --features sql -- --nocapture`
* `cargo test -p icydb-testing-integration --test sql_canister -- --nocapture`
* `WASM_CANISTER_NAME=minimal WASM_PROFILE=wasm-release WASM_SQL_VARIANTS=sql-on bash scripts/ci/wasm-size-report.sh`
* `WASM_CANISTER_NAME=one_simple WASM_PROFILE=wasm-release WASM_SQL_VARIANTS=sql-on bash scripts/ci/wasm-size-report.sh`

If a slice claims coherence improvement but regresses raw wasm without enough
value, it should be restaged or rejected.

## Success Criteria

`0.67` is successful when:

1. every `0.66` deferral has a recorded `0.67` decision
2. docs reflect the real shipped SQL subset
3. generated and typed SQL differences are explicit and tested
4. any lane merge that lands is justified by user-facing coherence, not just
   deduplication aesthetics
5. wasm/perf gates remain part of every keep/reject decision

---

## Summary

`0.67` should not start by assuming that all SQL lanes must collapse.

It should start by:

* rebaselining the real shipped surface
* carrying forward the actual `0.66` deferrals
* making explicit keep/merge decisions on those few remaining incoherent
  boundaries

That gives the project a cleaner foundation for any later unification work
without pretending that every visible split is automatically a bug.
