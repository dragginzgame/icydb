# 0.73 — Grouped Aggregate Planning

## 1. Overview

0.73 introduces planner-owned, deterministic route selection for **single-entity grouped aggregates**. The planner produces a **prepared grouped strategy** which becomes the *single behavioral source* for both execution and EXPLAIN.

This continues the execution-shape normalization work of:

* 0.71 — Aggregate execution normalization
* 0.72 — Single-entity deterministic planning

0.73 closes the remaining gap: **grouped route planning quality**.

---

## 2. Core Invariant

> The prepared grouped strategy is the single behavioral source.

* Execution MUST consume the prepared grouped strategy directly
* EXPLAIN MUST project from the prepared grouped strategy
* No downstream re-derivation of grouped behavior is allowed

This prevents divergence between runtime behavior and explain output.

---

## 3. Scope

### In Scope

Single-entity grouped queries:

```sql
SELECT tier, COUNT(*) FROM Customer GROUP BY tier
SELECT tier, COUNT(age) FROM Customer GROUP BY tier
SELECT tier, SUM(age) FROM Customer GROUP BY tier
SELECT tier, AVG(age) FROM Customer GROUP BY tier
```

Progressive widening:

* simple WHERE filters (contract-preserving)
* cases where access order matches group key
* planner-authored fallback/rejection reasons

### Out of Scope

* joins / relation planning
* grouped DISTINCT redesign
* HAVING redesign
* aggregate cost model
* multi-route speculative optimization

---

## 4. Execution Shape Families

Grouped planning is organized around **execution-shape families**, not optimization heuristics.

### 4.1 Streaming Grouped Fold

* Input is already ordered by group key
* Groups can be folded incrementally
* No full materialization required

Characteristics:

* O(1) or bounded memory per group
* stable streaming output

### 4.2 Materialized Grouped Fold

* Input not ordered by group key
* Requires accumulation (hash / map)

Characteristics:

* O(N) memory
* post-pass emission

### 4.3 Grouped DISTINCT Fallback (Deferred)

* Explicitly modeled as a separate family
* Not admitted in 0.73

---

## 5. Planner Responsibilities

The planner becomes the **sole authority** for grouped route selection.

### 5.1 Admission

Planner determines whether grouped execution is admitted:

* single entity
* supported aggregates
* supported grouping keys (initially single key)

### 5.2 Route Selection

Planner chooses exactly one:

* StreamingGroupedFold
* MaterializedGroupedFold

Decision is deterministic and rule-based (no cost model).

### 5.3 Strategy Construction

Planner produces:

```rust
struct GroupedStrategy {
    shape: GroupedShape,
    group_key: KeySpec,
    aggregates: Vec<AggregateSpec>,
    input_order: InputOrderSpec,
    filter: Option<FilterSpec>,
    rejection: Option<GroupedRejection>,
}
```

This object is **fully executable**.

---

## 6. Execution Model

Execution consumes `GroupedStrategy` directly.

### 6.1 No Re-Planning

* No recomputation of grouping logic
* No shape switching at runtime

### 6.2 Shape-Driven Execution

Execution branches only on `shape`:

```rust
match strategy.shape {
    StreamingGroupedFold => exec_streaming(strategy),
    MaterializedGroupedFold => exec_materialized(strategy),
}
```

### 6.3 Stability Guarantee

Given identical input + strategy → identical output behavior.

---

## 7. EXPLAIN Model

EXPLAIN is a **projection of the strategy**, not a reconstruction.

### 7.1 Example

```
GROUPED STRATEGY
  shape: streaming_grouped_fold
  group_key: tier
  aggregates:
    - count(*)
  input_order: index(tier)
```

### 7.2 Rejection Example (0.73.1)

```
GROUPED STRATEGY
  rejected: group_key_order_unavailable
  fallback: materialized_grouped_fold
```

---

## 8. Admission Rules (0.73.0)

Initial deterministic rules:

### StreamingGroupedFold admitted if:

* access path provides ordering on group key
* no operator breaks ordering before grouping

### Otherwise:

* fallback to MaterializedGroupedFold

No cost comparison is performed.

---

## 9. Rejection & Fallback (0.73.1)

Planner attaches explicit reasons:

```rust
enum GroupedRejection {
    RequiresMaterialization,
    GroupKeyOrderUnavailable,
    DistinctGroupingNotAdmitted,
}
```

These are:

* surfaced in EXPLAIN
* stable and testable

---

## 10. Filtered Grouped Cohorts (0.73.2)

Extend admission to:

```sql
SELECT tier, COUNT(*)
FROM Customer
WHERE active = true
GROUP BY tier
```

Constraint:

* filter must not break ordering contract for streaming

---

## 11. Key Design Decisions

### 11.1 Strategy as Behavioral Source

Avoids:

* execution/explain drift
* duplicated logic paths

### 11.2 Shape-First Design

Focus on:

* execution correctness
* deterministic behavior

Not:

* optimizer sophistication

### 11.3 Deterministic Planning

No:

* cost model
* adaptive switching
* speculative scoring

---

## 12. Non-Goals

* No attempt to optimize across multiple candidate plans
* No join-aware grouped planning
* No DISTINCT rearchitecture

---

## 13. Milestone Breakdown

### 0.73.0

* EXPLAIN projection

## 14. Success Criteria

* execution uses no grouped-specific logic outside strategy
* EXPLAIN matches execution behavior exactly
* deterministic planner decisions
* clear rejection diagnostics

---

## 15. Future Extensions (Not 0.73)

* multi-key GROUP BY
* grouped DISTINCT family
* cost-based selection
* join-aware grouping
* ordered grouped output guarantees

---

## 16. One-Line Definition

> 0.73 makes grouped route choice deterministic and planner-owned for single-entity grouped aggregates, with execution and explain projected from one prepared grouped strategy.
