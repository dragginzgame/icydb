# 0.141.5 Retained Slot Borrowing Design

## Status

Implemented in 0.141.5.

0.141.4 made `RowView` the borrowed row surface for structural data-row
projection materialization. Identity projection, direct data-row field
projection, and scalar data-row fallback projection now visit borrowed
`RowView` instances and copy into owned `MaterializedProjectionRows` only at
the structural result boundary.

Two projection paths intentionally remain owned:

- SQL `DISTINCT`, because accepted rows may be retained after one candidate
  callback and canonical-key storage still owns values.
- Retained slot rows, because `StructuralCursorPage::SlotRows` currently carries
  `Vec<RetainedSlotRow>`.

This document scopes the retained-slot follow-up for 0.141.5.

---

## Current Code Shape

`RowView` is deliberately small:

```rust
enum RowView<'a> {
    Borrowed(&'a [Value]),
    Owned(Vec<Value>),
}
```

It exposes positional output access through `get`, `values`, and `into_owned`.
It does not currently expose storage-slot access, required-slot diagnostics, or
`Cow<Value>` reads. Those contracts still live on structural row readers,
retained slot rows, and scalar projection evaluation helpers.

`RetainedSlotRow` is also not a plain `Vec<Value>`. It has two storage modes:

```text
Indexed { layout: RetainedSlotLayout, values: Vec<Option<Value>> }
Sparse { slot_count, entries: Vec<RetainedSlotEntry> }
```

Retained rows support:

- `slot_ref(slot)` for borrowed slot reads.
- `take_slot(slot)` for owned direct projection.
- `into_dense_slots()` for callers that need a full slot-indexed image.

The important consequence is that retained slot rows are already detached from
the original raw row by the time projection materialization sees them. True
borrowed retained-slot execution therefore cannot be implemented only by adding
a new materialization wrapper around `RetainedSlotRow`.

---

## Problem

Before 0.141.5, retained-slot projection allocated owned output rows through
`project_slot_rows` and `project_slot_row`.

That was correct for the payload shape, but it kept the retained-slot path
behind the 0.141 borrowed-row goal:

```text
data-row page -> RowView::Borrowed -> MaterializedProjectionRows
slot-row page -> RetainedSlotRow -> RowView::Owned -> MaterializedProjectionRows
```

The remaining question is not whether projection can borrow from
`RetainedSlotRow`; it already can borrow individual slot values through
`slot_ref`. The question is where the executor can preserve a borrowed row
buffer long enough to expose a contiguous projected `RowView::Borrowed`.

---

## Key Constraint

Borrowed retained-slot views are only sound when the backing storage outlives
the entire projected-row callback.

That is easy for 0.141.4 data-row visitors because the visitor owns a reusable
`Vec<Value>` buffer:

```text
decode/project into scratch Vec<Value>
visit RowView::Borrowed(&scratch)
clear scratch for next row
```

It is harder for retained-slot rows because the current structural payload is:

```text
StructuralCursorPage::SlotRows(Vec<RetainedSlotRow>)
```

Each `RetainedSlotRow` owns compact optional slot storage, not one contiguous
projection-order value slice.

---

## 0.141.5 Plan

0.141.5 borrows during retained-slot materialization.

Keep `RetainedSlotRow` owned, but add a retained-slot visitor in projection
materialization:

```text
RetainedSlotRow -> reusable projection Vec<Value> -> RowView::Borrowed
```

The visitor mirrors the 0.141.4 data-row visitor:

- direct retained-slot projection fills one reusable `Vec<Value>` by consuming
  each retained row with `take_slot`, preserving duplicate-slot missing-field
  behavior
- scalar retained-slot projection evaluates against borrowed slot references
  and fills the same reusable output buffer
- structural non-DISTINCT materialization copies into
  `MaterializedProjectionRows` at the final boundary

This does not remove the earlier allocation that produced `RetainedSlotRow`.
It does remove the per-output-row allocation currently performed between
`RetainedSlotRow` and `MaterializedProjectionRows`.

This is the correct next patch because it advances the borrowed materialization
boundary without pretending retained-slot scan output is already borrowed. It
also keeps the ownership boundary unchanged:

```text
RetainedSlotRow -> RowView::Borrowed callback -> MaterializedProjectionRows
```

### Why This Slice

- Smallest safe 0.141.5 slice.
- Keeps `StructuralCursorPage` and scan kernels unchanged.
- Preserves all retained-slot validation and missing-field behavior.
- Extends the same visitor pattern introduced in 0.141.4.
- Acknowledges the remaining owned input: `RetainedSlotRow` storage remains
  owned in this patch.
- Keeps direct retained-slot projection split between borrowed non-DISTINCT
  materialization and owned DISTINCT projection, because DISTINCT still retains
  accepted rows.

---

## Implementation Plan

### 1. Retained-slot row visitors

Projection materialization now has helpers analogous to the data-row visitors:

- `visit_slot_row_views(...)`
- `visit_direct_slot_row_views(...)`
- `visit_scalar_slot_row_views(...)`

These helpers borrow each projected output row from one reusable
`Vec<Value>` buffer and call:

```rust
visit(RowView::Borrowed(shaped.as_slice()))
```

### 2. Preserve owned single-row projectors

Keep `project_slot_row(...) -> RowView<'static>` for DISTINCT and any other
callers that may retain accepted rows beyond a callback.

The non-DISTINCT structural path uses the borrowed visitor. DISTINCT continues
using the owned projector unless its accumulator changes to consume
rows synchronously without retaining `RowView<'static>`.

### 3. Keep retained-slot semantics unchanged

Direct retained-slot projection currently uses `take_slot` and reports missing
fields through `ProjectionEvalError::MissingFieldValue`.

The borrowed visitor preserves that behavior by still consuming direct retained
rows with `take_slot`. Missing slots produce the same internal error
classification and message.

Scalar retained-slot projection continues evaluating through the existing
compiled scalar projection evaluator and borrowed value-reader path.

### 4. Do not change scan or page payloads

Do not change:

- `StructuralCursorPage::SlotRows(Vec<RetainedSlotRow>)`
- `RetainedSlotRow` storage variants
- `RetainedSlotLayout`
- DISTINCT row retention

Those are larger follow-ups.

## Validation Plan

Add focused projection materialization tests for:

- direct retained-slot projection visits borrowed `RowView` instances
- wide retained-slot scalar fallback projection visits borrowed `RowView`
  instances and preserves output order
- missing retained slot errors remain unchanged
- DISTINCT retained-slot projection remains on the owned projector path

Run:

- `cargo fmt --all`
- `cargo check`
- `cargo clippy -- -D warnings`
- focused projection materialization tests

---

## Non-Goals

- Changing `MaterializedProjectionRows`.
- Changing SQL `DISTINCT`.
- Changing `StructuralCursorPage` payload types.
- Replacing `RetainedSlotRow` storage.
- Adding a separate retained-slot view abstraction.
- Expanding `RowView` into a slot-addressable row abstraction.

---

## Future Work

After 0.141.5 lands, the next meaningful retained-slot borrowing step is below
the projection materializer:

- teach scan/page construction to expose borrowed retained-slot backing storage
- evaluate whether `RetainedSlotRow::Indexed` can use a reusable row buffer in
  cursorless streaming contexts
- keep DISTINCT owned until the accumulator no longer needs retained row views

---

## Summary

0.141.5 does not introduce a broad retained-slot abstraction before the
executor has a borrowed retained-slot producer.

The pragmatic next slice is to make retained-slot **materialization** borrow
projected output rows from a reusable buffer, matching the 0.141.4 data-row
visitor model while preserving the existing scan/page/retained-row ownership
contracts.
