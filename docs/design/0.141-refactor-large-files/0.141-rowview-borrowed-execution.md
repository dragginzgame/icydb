# 0.141 Borrowed Row Execution

## Status

- RowView abstraction introduced
- Vec<Vec<Value>> removed from executor hot paths
- DISTINCT operates on RowView
- Structural layer is the only allocation boundary

Current limitation: most RowView instances are still `Owned`.

---

## Thesis

**0.141 should convert RowView from a structural abstraction into a true zero-copy execution primitive by introducing borrowed row production paths.**

> The current system correctly defers allocation, but still *performs* allocation.
> 0.141 eliminates those allocations for common execution paths.

---

## Current State

Execution flow:


execute.rs → VecRowView::Owned
 → distinct.rs → structural → Vec<Vec<Value>>


This preserves architecture, but not performance:

- Per-row allocation still occurs
- Per-field allocation still occurs
- Cache locality is unchanged

RowView is present but not yet *utilized*.

---

## Target Model

Execution flow after 0.141:


execute.rs → VecRowView::Borrowed
 → distinct.rs → structural → Vec<Vec<Value>>


Key invariant:

> Rows remain borrowed until the final structural boundary.

---

## Goals

### 1. Introduce Borrowed Row Production

At least one full execution path must produce:

```rust
RowView::Borrowed(&[Value])

without intermediate allocation.

2. Preserve Existing Semantics

Must not change:

SQL behavior
DISTINCT correctness
NULL handling
Map canonicalization
Decimal normalization
Output shape (Vec<Vec<Value>> at boundary)
3. Maintain Layer Boundaries
execute.rs does not know structural layer
distinct.rs does not know DataRow / RowLayout
structural/* remains the only allocation boundary

0.141 makes it real by eliminating allocation in row production.

Rows should stay borrowed as long as possible, and only become owned at the final boundary.
