# 0.127 Direct Structured-Field Codec Cut

## Thesis

`0.127` should remove runtime `Value` as the mandatory adapter layer for
structured custom field persistence and generated structured field codecs.

This is not a public API boundary slice.

It is not a planner/runtime scalar-value deletion slice.

It is not a storage-format rewrite.

It is a structured-field codec ownership slice whose job is to replace:

* `typed field -> ValueCodec -> Value -> persisted slot bytes`

with a direct persisted structured-field codec contract.

`0.127` removes `Value` as the mandatory adapter for derive-owned structured
custom field persistence, but it does not remove the broader persisted-row
slot contract's `Value`-based decode, project, update, and canonicalization
surfaces.

The target end state is:

* structured custom fields do not have to round-trip through `Value`
* persisted-row custom field encode/decode uses a direct codec owner
* schema derives emit direct structured codec implementations for generated
  record, tuple, enum, newtype, list, set, and map wrappers
* `Value` remains available where it still has a real runtime job
* `ValueCodec` no longer owns structured-field persistence

---

## Why `0.127` Exists

`0.126` cleaned the public boundary problem:

* public inputs moved onto `InputValue` / `FilterValue`
* public outputs moved onto `OutputValue`
* `FieldValue` was deleted
* key-only callers moved onto `KeyValueCodec`
* the remaining derive-owned structured wrapper seam was reduced to
  `ValueSurfaceMeta` plus `ValueCodec`, while the broader persisted-row slot
  and patch surfaces remained intentionally `Value`-based

That left one historical adapter seam isolated enough to address directly.

Today, structured custom fields still persist through runtime `Value`:

* generated schema value surfaces lower typed structured fields into `Value`
* persisted-row helpers encode those `Value` instances into slot payloads
* typed reconstruction reads slot payloads back into `Value`
* generated codecs rebuild the typed field from that `Value`

That design predates the stronger slot/binary ownership work.

The codebase now has much stronger storage and slot boundaries than when
`Value` first became the generic adapter for structured fields. Keeping
structured persistence on `Value` has three costs.

### 1. Structured Persistence Still Depends On A Generic Runtime Union

Generated record, tuple, enum, list, set, and map wrappers still need
`ValueCodec::{to_value, from_value}` even though the storage lane already has
custom binary and slot-specific structure.

### 2. Structured Field Codecs Carry Runtime Semantics They Do Not Need

For persisted custom fields, the real need is:

* stable structural encode
* stable structural decode
* metadata about queryability/addressability

That is narrower than the full runtime `Value` surface.

### 3. Future `Value` Simplification Is Still Blocked By Structured Persistence

As long as structured persistence depends on `ValueCodec`, the runtime `Value`
layer remains coupled to:

* generated structured wrappers
* persisted-row custom field helpers
* structural storage compatibility paths

That prevents later cleanup from shrinking `Value` to the places where it is
still genuinely useful.

---

## Goal

Introduce a direct structured-field storage codec contract and move
persisted-row custom field encode/decode onto it.

`0.127` should deliver:

* one direct codec owner for structured custom fields
* persisted-row helpers consuming that codec instead of `ValueCodec`
* schema derives emitting direct structured codec impls for generated wrapper
  types
* `ValueCodec` no longer required for structured custom field persistence
* a clear boundary between:
  * runtime `Value` use
  * structured persisted-field codec use

---

## Core Invariant

Structured custom field persistence must not require runtime `Value` as an
intermediate representation.

Corollaries:

* generated structured field wrappers may still implement metadata traits
* generated structured field wrappers should not need `to_value()` /
  `from_value()` for persisted-row storage
* persisted-row custom field helpers should ask the field owner for bytes or a
  structural storage view directly
* runtime `Value` remains valid only where the runtime layer actually needs a
  generic semantic value

---

## Scope

### In Scope

* direct structured-field codec trait design
* persisted-row structured custom field encode/decode
* schema derive output for structured wrapper types
* list/set/map/record/tuple/enum/newtype structured storage ownership
* narrowing `ValueCodec` by removing structured persistence from it

### Out Of Scope

* removing runtime `Value` from planner/predicate/executor generic logic
* removing `Value` from aggregate/group/runtime state
* public `InputValue` / `OutputValue` changes
* key codec redesign
* storage-key encode redesign
* changing the external persisted binary format unless strictly necessary
* changing `UpdatePatch`, `FieldUpdate`, `SerializedUpdatePatch`, or the
  slot-reader `get_value` / `required_value_by_contract` seams, which remain
  intentionally `Value`-based in this slice

## Current Problem Shape

The structured persistence seam currently sits across these owners:

* [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs)
* [/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/persisted_row.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/persisted_row.rs)
* [/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs)
* [/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs)

The current flow for a structured custom field is effectively:

1. generated wrapper implements `ValueCodec`
2. derive-generated `PersistedRow::write_slots` reaches the persisted custom
   payload helpers
3. persisted-row encode calls `value.to_value()`
4. slot/storage helpers encode the generic `Value`
5. decode reconstructs a generic `Value`
6. derive-generated `PersistedRow::materialize_from_slots` rebuilds typed data
   with `from_value()`

That is the seam `0.127` should remove.

The broader persisted-row slot contract is still intentionally `Value`-shaped:

* `decode_slot_value_from_bytes(...) -> Value`
* `encode_slot_value_from_value(...) -> Vec<u8>`
* canonical row rebuilding and patch application still move through slot-level
  `Value` decode/re-encode
* slot-reader projection and required-slot read surfaces still expose
  `Result<Option<Value>, _>` / `Result<Value, _>`-shaped contracts
* `UpdatePatch`, `FieldUpdate`, and `SerializedUpdatePatch` remain
  `Value`-carrying write/update surfaces

`0.127` should not try to remove those generic slot-level `Value` surfaces.
It should only cut the custom structured field adapter layer that currently
depends on them.

## Ownership Model

### Runtime Semantic Layer: `Value`

`Value` remains the internal generic semantic union for:

* planner literal representation
* predicate/runtime generic comparison
* aggregate/group runtime outputs
* fallback scalar/projection glue
* storage-key bridging where still needed

`0.127` should not try to remove these uses.

### Structured Persistence Layer: Direct Codec Trait

Structured custom fields should instead own a direct storage codec.

Working name for the design doc:

* `PersistedStructuredFieldCodec`

The exact final name can still change if the implementation reveals a better
ownership line.

The trait’s job is:

* encode one typed custom structured field directly to the existing persisted
  custom slot payload bytes
* decode one typed custom structured field directly from those persisted
  custom slot payload bytes

It does not own:

* queryability metadata
* key-only conversion
* generic runtime semantic value conversion
* arbitrary by-kind structural encoding for all `FieldKind`

### Metadata Layer: `ValueSurfaceMeta`

`ValueSurfaceMeta` continues to own:

* `ValueSurfaceKind`
* structured/queryable metadata

This stays separate from the direct codec so metadata-only callers do not
depend on storage conversion.

---

## Trait Direction

The exact final signature can still move during implementation, but `0.127`
should be built around one of these two shapes.

### Preferred Direction: Direct Bytes Contract

```rust
trait PersistedStructuredFieldCodec: Sized {
    fn encode_persisted_structured_payload(&self)
        -> Result<Vec<u8>, InternalError>;

    fn decode_persisted_structured_payload(bytes: &[u8])
        -> Result<Self, InternalError>;
}
```

Why this is preferable:

* it keeps the codec type-owned rather than row-owned
* it stops reintroducing another generic value adapter
* it matches the actual need: bytes in, bytes out
* field-name context stays on persisted-row call sites, where those errors are
  already contextualized today

### Acceptable Fallback: Structural Storage View Contract

```rust
trait PersistedStructuredFieldCodec {
    fn encode_structured_value(&self) -> Result<StructuralValue, InternalError>;
    fn decode_structured_value(value: &StructuralValue) -> Result<Self, InternalError>;
}
```

`0.127` should prefer the direct-bytes contract. A structural intermediate is
acceptable only if it is a private helper representation for codec
implementation and does not become a new cross-module authority, trait
boundary, or generated public or semipublic type.

The design should not accept:

* `Value` under a new name
* another wide runtime union that recreates the same coupling
* a new shared structural authority type that sits beside `Value` and
  accumulates the same responsibilities under a different label

---

## Compatibility Invariants

`0.127` does not change the external payload shape for existing structured
custom fields unless the implementation proves a compatibility-preserving
rewrite.

The new codec must preserve:

* stable canonical payload shape
* null and optional semantics
* fail-closed malformed payload rejection
* consistent field-name-contextual error mapping

More concretely:

* equivalent typed values must continue to emit the same canonical payload
  bytes
* malformed or non-canonical persisted custom payloads must continue to be
  rejected rather than repaired opportunistically
* required versus optional structured slot behavior must remain identical at
  the persisted-row contract boundary

This is the main safety constraint for the slice. The point is to change the
ownership of the codec path, not to fork the persisted custom payload format.

---

## What Generated Code Should Still Own

Even after the cut, some generated code remains legitimate.

### Record / Tuple / Enum / Newtype Shape Logic

Generated code should still own:

* record field names and ordering
* tuple position order
* enum variant arm mapping and payload shape
* newtype inner forwarding

That is real schema shape, not duplication.

### Shared Collection Logic

List/set/map wrappers should continue to lean on shared helpers whenever
possible.

`0.127` should preserve or improve:

* one shared list collection encode/decode helper
* one shared set collection encode/decode helper
* one shared map encode/decode helper

The goal is not to make every generated type bespoke. It is to make generated
types use the right owner contract.

## Non-Goals

`0.127` should not:

* remove `Value` from planner literals
* remove `Value` from predicate normalization
* remove `Value` from grouped runtime
* redesign aggregate output representation
* change public value-boundary types again
* mix in naming cleanup unrelated to the codec cut

---

## Success Criteria

`0.127` is complete when:

* persisted structured custom field encode/decode no longer requires
  `ValueCodec`
* generated structured wrappers implement the new direct codec owner
* `ValueSurfaceMeta` remains only metadata
* `ValueCodec` is no longer the storage bridge for structured custom fields
* `Value` remains only where runtime generic semantics still justify it

---

## Expected Follow-Through

If `0.127` succeeds, the next cleanup becomes much clearer:

* audit the remaining `ValueCodec` uses
* decide whether projection/enum payload glue should stay on `ValueCodec`
  or split again
* shrink runtime `Value` only where there is now clear evidence it is still
  overextended

That is the correct order.

`0.127` should cut the historical structured-persistence adapter first.
Only after that should the codebase ask whether more of `Value` can disappear.
