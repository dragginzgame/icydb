# 0.212 Bounded And Resumable Integrity Check

Status: proposed

Design cut: 2026-07-16

Implementation order: after 0.211 closeout

## Decision Summary

IcyDB will replace the current aggregate whole-database integrity report with
two accepted-schema-native inspection lanes:

- `Quick` performs bounded metadata, control-state, codec, fingerprint, and
  ownership checks in one call; and
- `Deep` scans one entity through bounded deterministic phases, returns bounded
  machine-readable findings, and resumes through a schema-owned inspection job.

Deep completion requires canonical phase exhaustion followed by a final check
that the accepted schema identity and every participating store revision remain
unchanged. A concurrent change invalidates the scan; it is never reported as a
clean result.

Inspection is read-only with respect to the inspected database. Deep writes
only isolated bounded job/receipt metadata. It never repairs, deletes,
rebuilds, coerces, folds, or republishes inspected state implicitly. Repair is a
separate future design.

SQL and typed entrypoints lower to the same scanner:

~~~sql
CHECK INTEGRITY Token
CHECK INTEGRITY Token QUICK
CHECK INTEGRITY Token DEEP
CHECK INTEGRITY Token DEEP CONTINUE 'job_id' AFTER page_sequence
CHECK INTEGRITY Token DEEP ABORT 'job_id'
~~~

Bare `CHECK INTEGRITY entity` starts `Deep`. `QUICK` is the explicit bounded
metadata lane. Public callers never supply raw phase/checkpoint bytes; the
engine resolves an authorization-bound job ID to trusted bounded progress.

### Non-Negotiable Invariants

1. Quick and Deep consume accepted schema as runtime authority and never
   reconstruct missing contracts from generated models.
2. Every call has hard row, key, byte, finding, decode, heap, and instruction
   bounds.
3. Page fullness is never keyspace or phase exhaustion.
4. Deep traversal uses canonical primary-key and physical derived-store order,
   never an offset.
5. Deep completion requires a full phase sweep at one unchanged accepted schema
   identity and participating-store revision vector.
6. A stale job or concurrent change returns `Invalidated`, not `Clean` and not
   corruption.
7. Every finding is bounded and machine-readable; callers never parse display
   text for classification.
8. A persisted row violating a validated accepted contract is an integrity or
   corruption finding.
9. A historical row violating a pending 0.211 activation is migration
   validation, not accepted-state corruption.
10. Inspection never repairs implicitly.
11. Persisted decoding is bounded, fallible, and current-form only.
12. Recovery does not run the resumable global Deep scanner as a hidden
    synchronous postcondition.

## Normative Integrity Contract

The maintained contract is:

> Quick verifies bounded catalog and storage-control structure. Deep verifies
> authoritative rows and every accepted derived invariant through resumable
> pages. A clean Deep result means the complete declared phase set was clean at
> one unchanged accepted schema identity and participating-store revision
> vector.

The negative contract is:

> Incomplete, resource-limited, stale, or concurrently invalidated inspection
> is never presented as clean. Inspection never mutates the inspected state.

## Audited Baseline

The source audit on 2026-07-16 found an existing integrity report, not an empty
feature area:

| Concern | Current owner | Current behavior |
|---|---|---|
| Public typed entry | `DbSession::integrity_report` | Returns one aggregate report; no entity/mode/job input |
| Scan owner | `crates/icydb-core/src/db/diagnostics/integrity.rs` | Scans every registered store in one call |
| Global target lookup | `collect_global_live_keys_by_entity` | Builds `BTreeMap<EntityTag, BTreeSet<PrimaryKeyValue>>` for all live rows in heap memory |
| Forward scan | `scan_store_forward_integrity` | Decodes data keys/rows, prepares expected index effects, and counts missing/divergent entries |
| Reverse scan | `scan_store_reverse_integrity` | Decodes index entries and counts references whose row key is absent from the global key map |
| Output | `IntegrityReport`, `IntegrityStoreSnapshot`, `IntegrityTotals` | Aggregate saturating counters; no bounded finding records or row identities |
| Error classification | `classify_scan_error` | Buckets corruption and misuse; some invariant/internal failures abort the report |
| Recovery dependency | `validate_recovery_integrity` | Runs the full report after recovery and fails on missing/divergent/orphan index counters |
| SQL surface | SQL parser/executor | No `CHECK INTEGRITY` command |
| Schema metadata integrity | `db/schema/integrity.rs` and submodules | Validates accepted schema structure, indexes, and relations independently of row scan |
| Revision proof | journaled store mutation sequence/fold watermark | Used by 0.210 as durable store revision authority |

The existing report provides useful forward/reverse index checks, but its
complete live-key materialization and whole-store traversal are not
message-bounded. Aggregate counters also cannot identify which row, index,
relation, or constraint failed.

The opening implementation audit must additionally inventory:

- every data, index, reverse-relation, schema, marker, journal, fold-watermark,
  allocation, and ownership keyspace;
- point-lookup and ordered-visit capabilities for each keyspace;
- every accepted row decoder and exact-kind invariant after 0.208;
- every accepted constraint verifier after 0.211;
- current mutation revision coverage for journaled stores;
- recovery's bounded touched-effect set and current post-recovery report call;
- current endpoint authorization and SQL introspection gates;
- current raw non-gzipped wasm bytes; and
- current integrity-report runtime and peak heap on representative stores.

## Problem Statement

### 1. The Current Report Is Not Bounded By One Message

It visits all stores and materializes every live primary key globally. A large
database can exceed instruction or heap limits before returning any result.

### 2. Aggregate Counters Are Not Operational Diagnostics

A count of missing index entries does not identify the entity, row, logical
index, expected key, observed witness, or stable diagnostic code. Operators
cannot distinguish one repeated defect from many independent defects.

### 3. The Current Report Does Not Cover The Full Accepted Contract

It does not provide a single accepted-native proof of exact field kinds,
required/null policy, enum catalog membership, validated checks, relation target
existence, or reverse relation agreement.

### 4. A Moving Multi-Message Scan Can Overclaim

A resumable job without accepted identity and revision stability can miss rows
around its checkpoint or combine observations from incompatible schema/store
states.

### 5. Recovery And Operator Inspection Have Different Boundaries

Recovery must verify the bounded effects it just replayed before admitting
runtime. It must not depend on a hidden whole-database scan that can exceed one
message. An operator Deep scan is intentionally resumable and cannot be a
synchronous recovery postcondition.

## Goals

0.212 must:

1. add one bounded Quick lane;
2. add one bounded resumable Deep lane for a named entity;
3. use accepted schema only for row, index, relation, enum, and constraint
   semantics;
4. carry accepted schema identity, phase, canonical checkpoint, and store
   revision vector in one bounded inspection job;
5. return bounded typed findings with stable diagnostic codes;
6. distinguish clean, findings, incomplete, resource-limited, and invalidated
   outcomes;
7. verify row-key/row-value identity and recursive exact accepted kinds;
8. verify forward and reverse index agreement without a global live-key map;
9. verify validated check constraints and immediate relation integrity;
10. verify accepted schema/control/journal/ownership structure at the right
    lane and phase;
11. keep pending constraint violations in migration-validation context;
12. replace recovery's global integrity-report dependency with bounded
    recovered-effect verification;
13. expose typed and SQL surfaces over the same scanner;
14. never repair implicitly;
15. hard-cut the aggregate-only public report and old current-form DTOs; and
16. close with correctness, corruption, concurrency, resource, performance,
    wasm-size, and source-complexity evidence.

## Non-Goals

0.212 does not:

- repair rows, indexes, relations, schema, markers, journals, or ownership;
- rebuild an index or reverse relation generation;
- resume or validate a pending 0.211 constraint activation;
- infer accepted schema from generated `EntityModel` or callbacks;
- provide a planner hint from a partial scan;
- promise a database-wide atomic snapshot across unrelated stores;
- add cross-entity business assertions;
- validate arbitrary application validators or sanitizers;
- add schema-less or open-ended content inspection;
- support caller-defined regex, callbacks, scripts, or inspection functions;
- add a mutable runtime journal/durability mode or a session-level equivalent
  of `PRAGMA journal_mode`;
- sleep, wait for a lock, or hide retry loops inside one message; bounded
  progress, resource, and invalidation outcomes remain explicit;
- retain the old aggregate report as an alias or compatibility wrapper; or
- design a repair API.

## Terminology

### Quick

A one-call bounded check of accepted metadata and storage control structure. A
clean Quick result does not claim that every row or derived entry was scanned.

### Deep

A multi-phase authoritative row and derived-state scan. Deep may claim entity
integrity only after stable completion.

### Finding

A bounded machine-readable observation of a definite invariant failure in one
examined key/value or accepted metadata object.

### Invalidated

The accepted schema identity or participating store revision changed during a
Deep sweep. Invalidation is concurrency/progress state, not corruption.

### Clean

Every required phase exhausted without findings at one unchanged proof
identity. Only Deep can make the row-level clean claim.

## Authority Model

1. accepted schema owns row layout, exact kinds, null policy, enum identity,
   primary key, indexes, relations, and validated constraints;
2. authoritative data stores own rows;
3. accepted index definitions plus authoritative rows define expected index
   entries;
4. physical index stores are derived state to compare against expectation;
5. accepted relations plus source rows define expected reverse entries;
6. marker, journal, fold watermark, and allocation registries own durability
   control facts;
7. store mutation revisions own multi-message stability;
8. the isolated inspection-job registry owns bounded traversal position but no
   inspected row/index/relation authority; and
9. generated models, application validators, and sanitizers are not integrity
   authority.

## Public Surface

### SQL

The opening grammar is:

~~~sql
CHECK INTEGRITY entity
CHECK INTEGRITY entity QUICK
CHECK INTEGRITY entity DEEP
CHECK INTEGRITY entity DEEP CONTINUE 'job_id' AFTER page_sequence
CHECK INTEGRITY entity DEEP ABORT 'job_id'
~~~

Rules:

- bare form is equivalent to `DEEP` start;
- `QUICK` rejects a job ID;
- Deep start creates the bounded authorized job and returns empty page sequence
  zero; the first `CONTINUE ... AFTER 0` performs inspection work;
- Deep start first performs the minimum bounded accepted-identity admission
  needed to bind the job; undecodable authority returns typed corruption and
  creates no job;
- `DEEP CONTINUE` requires exactly one engine-issued job ID owned by the
  authorized caller/capability and the last acknowledged page sequence;
- `DEEP ABORT` removes an owned non-running job and its cached receipt without
  touching inspected state;
- unknown modes, extra entities, predicates, projection, ordering, limits, and
  user-selected internal phase/checkpoint values fail closed;
- Quick may use the read-only introspection lane;
- Deep start/continue uses an authorized update/admin lane because it advances
  isolated inspection-job progress; and
- deployment policy exposes a distinct integrity capability rather than
  silently inheriting ordinary row-query access.

### Typed

The typed surface accepts entity identity, mode, and optional job ID and returns
the same `IntegrityCheckPage` as SQL. SQL does not own an alternate report
shape. Raw checkpoint constructors/decoders remain private to the job owner.

### Result Shape

Conceptually:

~~~text
IntegrityCheckPage {
    entity: EntityIdentity,
    mode: Quick | Deep,
    accepted_schema_version: u32,
    accepted_schema_fingerprint: Fingerprint,
    status: IntegrityCheckStatus,
    phase: IntegrityPhase,
    progress: IntegrityProgress,
    page_sequence: u64,
    findings: Vec<IntegrityFinding>,
    job_id: Option<IntegrityJobId>,
}

IntegrityCheckStatus
  = QuickCompleteClean
  | QuickCompleteWithFindings
  | DeepInProgress
  | DeepCompleteClean
  | DeepCompleteWithFindings
  | Invalidated
  | ResourceLimited
~~~

`ResourceLimited` means the engine could not classify even one next unit within
hard safety limits. Ordinary page-boundary progress is `DeepInProgress`.

## Quick Check

Quick performs only work bounded by accepted catalog and control-format limits:

- accepted schema root, bundle, entity snapshot, constraint/activation, enum,
  and job codec decoding;
- accepted root/bundle/entity fingerprints and schema version agreement;
- field ID/slot uniqueness, row-layout closure, exact-kind capability, primary
  key metadata, nullability, defaults, and bounds;
- index logical identity, physical placement, key expression, uniqueness,
  lifecycle, and constraint pairing;
- relation identity, primary-key target binding, cardinality, reverse owner,
  activation state, and restrictive-action closure;
- marker envelope/state and recovery readiness;
- journal control metadata, sequence bounds, chunk-envelope structure, and fold
  watermark ordering that can be established without scanning an unbounded
  tail;
- stable-memory allocation identity and ownership uniqueness;
- current storage mode/capability agreement; and
- current-form format versions for every inspected control family.

Quick verifies storage mode/capability agreement from the same typed facts
already exposed by store and entity diagnostics; it does not duplicate those
facts in an integrity-only wire shape. A well-formed heap store is not
corruption, but operator renderers must label it `volatile`, and
production-oriented tooling that composes integrity and storage diagnostics
should emit a volatility advisory. A journaled store receives no such advisory
when its declared and runtime capabilities agree. The advisory does not change
an otherwise clean integrity status. Inspection never changes storage mode;
there is no mutable PRAGMA-like durability setting.

If a control family contains an unbounded number of entries, Quick verifies its
bounded root/summary only and reports that entry-level verification belongs to
Deep. Quick never silently truncates and then says it verified the full family.

Quick normally runs after recovery admission. A corrupt/nonrecoverable marker
or journal envelope fails through typed corruption. A separate future forensic
surface may inspect raw pre-recovery state; Quick does not bypass recovery
gating.

## Deep Phase Machine

The semantic phase order is:

~~~text
QuickMetadata
    -> Rows(primary_key_checkpoint)
    -> IndexEntries(schema_index_id, physical_key_checkpoint)
    -> ReverseRelations(relation_id, physical_key_checkpoint)
    -> JournalTail(sequence_checkpoint)
    -> FinalRevisionCheck
    -> DeepCompleteClean | DeepCompleteWithFindings
~~~

Phases with no applicable entries advance without fabricating a checkpoint.
One call advances within one phase or crosses a bounded number of empty phase
boundaries; it never performs unbounded nested traversal.

### Row Phase

For every authoritative row, Deep verifies:

- data-store key decodes in the current format;
- key entity identity matches the requested entity;
- row envelope decodes with bounded length/count/depth;
- storage key equals the row's accepted primary-key fields exactly;
- every accepted top-level slot belongs to the row layout;
- required fields are present and nullable policy is respected;
- every scalar and composite value recursively matches its exact accepted kind;
- integer bounds, decimal scale/precision, text/blob bounds, collection bounds,
  ordering/uniqueness encoding, and enum catalog identity are valid;
- every validated `CHECK` evaluates to `True` or `Unknown`;
- every expected index entry exists and exactly matches the derived row witness;
- every accepted relation target exists; and
- every expected reverse relation entry exists and exactly identifies the
  source row/relation.

Pending 0.211 activation gates are metadata-checked by Quick. Deep does not
label a historical candidate violation as corruption. Activation validation is
performed by `VALIDATE CONSTRAINT` and uses migration diagnostics.

### Index Entry Phase

For every physical index entry belonging to the accepted entity, Deep verifies:

- key framing, entity tag, physical index placement, and value/witness codec;
- logical `SchemaIndexId` resolves through the current accepted index owner;
- referenced authoritative row exists;
- the row still derives this exact index key and value;
- conditional membership still includes the row;
- no entry belongs to a dropped, pending, or different index generation; and
- unique-index logical keys do not map to conflicting row witnesses.

Forward row verification plus exact witness comparison detects duplicate
logical unique keys without retaining every logical key: if two authoritative
rows derive the same unique key, at least one cannot match the single
authoritative physical witness. Focused evidence must prove this property for
all supported unique encodings.

### Reverse Relation Phase

For every reverse relation entry, Deep verifies:

- current-form key/value framing and relation identity;
- target identity belongs to the accepted relation target entity;
- source row exists;
- source row still contains the target in the declared relation field and
  cardinality;
- source/target primary-key kinds match accepted relation identity; and
- the entry belongs to the active accepted reverse generation.

Entries in isolated 0.211 activation staging are checked as migration-storage
closure, not as accepted relation authority. Missing staged entries are a
migration job failure; missing accepted entries are corruption.

### Journal Tail Phase

For the owning journaled store, Deep verifies in bounded sequence pages:

- complete current-form batch/chunk envelopes;
- strictly ordered sequence identity above the fold watermark;
- no unexplained gaps, overlaps, duplicate chunks, or batch identity changes;
- fold watermark and canonical/tail boundary agreement;
- schema/row/index/relation records reference known current-format families;
- marker/tail state is compatible with completed recovery.

Deep inspection does not replay or fold the journal.

## Inspection Job And Stable Completion

Deep progress lives in one bounded catalog-native inspection job:

~~~text
IntegrityJobV1 {
    id: IntegrityJobId,
    owner: IntegrityJobOwner,
    entity: EntityIdentity,
    accepted_schema_version: u32,
    accepted_schema_fingerprint: Fingerprint,
    phase: IntegrityPhase,
    checkpoint: IntegrityCheckpoint,
    captured_store_revisions: StoreRevisionVector,
    findings_seen: u64,
    pages_completed: u64,
    last_receipt: Option<IntegrityPageReceipt>,
}
~~~

The registry is isolated from inspected data-store revisions so saving progress
does not invalidate its own proof. Job creation/advancement is marker-safe,
bounded, authorized, and current-form only. A job contains no generated Rust
path, user callback, row after-image, or unbounded collection.

`IntegrityJobId` is a lookup identity, not trusted progress supplied by the
caller. The registry resolves it, verifies the current caller/capability owns
the job, and loads the private checkpoint. Public SQL and typed APIs never
accept raw continuation bytes. Unkeyed hashes, CRCs, and opaque encoding are not
treated as authentication.

Every job step:

1. resolves and authorizes the job and accepted entity/store authority;
2. verifies job format, entity, schema version/fingerprint, phase, checkpoint,
   and bounded receipt state;
3. compares every current participating revision with the captured vector;
4. returns and records `Invalidated` without continuing if any identity changed;
5. scans strictly after the last fully classified canonical key;
6. atomically saves the next checkpoint and bounded receipt; and
7. on final exhaustion, rechecks schema identity and every revision before
   saving a complete terminal receipt.

Receipts have monotonically increasing page sequence. The caller acknowledges
the prior sequence when requesting the next page. A retry with the same
acknowledged sequence returns the cached bounded receipt instead of advancing
again. This prevents response loss from silently dropping findings. Receipt
storage is capped to one page per job.

The SQL/typed execution envelope therefore carries `job_id` and
`acknowledged_page_sequence`; callers do not author phase/checkpoint/revision
facts. The shell may render this as:

~~~sql
CHECK INTEGRITY Token DEEP CONTINUE 'job_id' AFTER 7
~~~

`DeepCompleteClean` means the entire phase machine completed with zero findings
at one stable proof identity. `DeepCompleteWithFindings` makes the same
stable-sweep claim but records at least one finding across the job.

Completed, invalidated, and abandoned jobs are removed explicitly or by one
bounded retention policy. Job count/bytes are capped per owner and globally;
capacity exhaustion rejects a new job without touching inspected state.

Journaled V1 uses the durable mutation revision established for 0.210. Quick is
available to heap stores. The Deep proof vector also carries a journal-control
generation or equivalent fold/tail identity: data mutation revision alone is
insufficient because journal folding may change physical traversal while
preserving logical row revision. A fold, cleanup, marker transition, or journal
topology change invalidates the relevant phase.

Deep heap support requires a trustworthy runtime epoch plus monotonic mutation
revision that invalidates jobs across every write and runtime reinitialization;
until that authority exists, heap Deep rejects explicitly.

## Bounded Findings

Conceptually:

~~~text
IntegrityFinding {
    diagnostic_code: IntegrityDiagnosticCode,
    error_class: ErrorClass,
    severity: IntegritySeverity,
    entity: EntityIdentity,
    store: StoreIdentity,
    phase: IntegrityPhase,
    primary_key: Option<PrimaryKeyValue>,
    field_paths: Vec<FieldPath>,
    constraint_id: Option<ConstraintId>,
    constraint_name: Option<String>,
    schema_index_id: Option<SchemaIndexId>,
    relation_id: Option<RelationId>,
    expected: Option<BoundedIntegrityValue>,
    observed: Option<BoundedIntegrityValue>,
}
~~~

Finding count, string bytes, path count/depth, primary-key bytes, and
expected/observed rendering are independently bounded. Raw row payloads,
unbounded collection contents, secrets, and arbitrary application messages are
not echoed.

Stable diagnostic families include:

- schema/control codec or fingerprint mismatch;
- primary-key key/value mismatch;
- missing/extra/undecodable row slot;
- exact-kind, nullability, bounds, decimal, enum, or composite violation;
- validated check constraint violation;
- missing/divergent/orphan index entry;
- duplicate logical unique key;
- missing relation target;
- missing/divergent/orphan reverse relation entry;
- marker/journal/fold discontinuity; and
- allocation/ownership conflict.

A finding observed in one serialized call is definite for the bytes examined in
that call. If the job is later invalidated, the overall sweep loses its stable
completeness claim; the invalidation itself is not recast as corruption.

## Error And Outcome Classification

The scanner distinguishes:

- malformed request, job ID, or acknowledgement: typed validation/misuse error;
- unsupported storage capability: typed unsupported error;
- resource inability to classify one next unit: `ResourceLimited` with bounded
  context;
- schema/revision change: `Invalidated`;
- persisted current accepted invariant failure: corruption/integrity finding;
- pending activation historical incompatibility: migration validation, outside
  Deep accepted-integrity findings; and
- impossible scanner/catalog state: internal or corruption error.

Display text is never classification authority.

## Recovery Boundary

The current recovery path calls the whole-database aggregate integrity report
after replay. 0.212 removes that unbounded dependency before removing the old
report.

Recovery instead verifies the bounded effect set owned by the recovered marker
or replay batch:

- each recovered row put/delete matches its final authoritative row state;
- each staged index/reverse effect matches that row operation;
- schema publication identity matches the recovered schema record;
- marker, journal sequence, and fold boundary reach the documented terminal
  state; and
- every checked effect fits the existing commit caps.

This is recovered-effect verification, not a claim that the entire database was
scanned. Quick may run after recovery for bounded metadata/control closure.
Deep remains an explicit operator action.

Recovery interruption tests must prove that the bounded verifier sees the same
all-before/all-after states as normal marker recovery. It must not clear a
marker merely because a global scan could not finish.

## No Implicit Repair

Quick and Deep are read-only over inspected state. Deep may advance only its
isolated bounded job/receipt record. They never:

- insert, delete, or rewrite a row;
- create, delete, or rewrite an index/reverse entry;
- rebuild a staged or accepted generation;
- clear a marker;
- replay, truncate, or fold a journal;
- advance a fold watermark;
- publish schema;
- coerce a value; or
- suppress a finding by mutating evidence.

A later repair design must name repair authority, dry-run planning, bounds,
atomicity, recovery, authorization, and post-repair verification explicitly.
It must not be hidden behind `CHECK INTEGRITY`.

## Persisted And Wire Formats

Deep job records and public result DTOs are current-form V1 values with:

- magic/version or equivalent discriminant;
- explicit total byte bound;
- bounded owner, phase, checkpoint, and last-receipt variants;
- canonical entity/store/schema identity;
- canonical revision ordering;
- bounded counts and strings; and
- fallible decode with typed incompatible-format/corruption errors.

Job records are isolated diagnostic progress state, not accepted schema or
inspected data authority. Their proof validity derives from current accepted
identity and store revisions. Job creation, page receipt, acknowledgement,
terminal state, retention, and removal are bounded and recoverable.

The old `IntegrityReport`, `IntegrityStoreSnapshot`, and `IntegrityTotals`
aggregate-only public shapes are removed when the new surface lands. There is
no alias, wrapper, dual endpoint, or decoder for their retired current form.

## Pre-1.0 Hard Cut

0.212 directly replaces the old integrity report execution and public DTO
surface. It does not keep:

- the global live-key materialization path;
- an unbounded all-store convenience wrapper;
- aggregate-only results beside finding pages;
- generated-model fallback scanning;
- recovery's global report call;
- old job/receipt spellings or format versions; or
- anti-resurrection tests for removed surfaces.

Tests cover the maintained Quick/Deep current surface instead.

## Focused Validation

### Quick

Evidence must cover valid and corrupt:

- schema root/bundle/entity codecs and fingerprints;
- field/slot/primary-key closure;
- index/relation/constraint/activation pairing;
- enum catalogs and exact-kind capabilities;
- marker state and recovery readiness;
- journal control metadata and fold ordering;
- allocation identity and ownership uniqueness; and
- every control-format version/boundary.

Evidence also covers operator composition rendering heap storage as a
non-corruption `volatile` advisory, journaled storage as durable without that
advisory, Quick retaining its actual integrity status, and no inspection path
mutating either mode.

### Row Deep Scan

Evidence must cover:

- empty, one-row, multi-page, and maximum admitted rows;
- malformed data keys/envelopes;
- scalar and every composite-kind mismatch;
- missing/extra slots, required/null violations, bounds, decimal scales, text,
  blobs, collections, and enum catalog values;
- scalar and composite primary-key key/value mismatch;
- accepted check `False`, `True`, and `Unknown` behavior;
- pending activation violations excluded from accepted corruption; and
- first/middle/last finding with bounded page progress.

### Index And Relation Phases

Evidence must cover:

- missing, divergent, and orphan index entries;
- conditional index membership drift;
- duplicate logical unique keys across page boundaries;
- dropped/pending/wrong physical index generation;
- missing relation targets;
- missing/divergent/orphan reverse entries;
- relation list/set cardinality and duplicate encoding;
- source and target entities in the same and different stores; and
- activation staging classified separately from accepted authority.

### Jobs, Receipts, And Concurrency

Evidence must cover:

- canonical checkpoint exclusivity with no skip/duplicate;
- exact page-boundary versus phase-exhaustion distinction;
- phase transitions over empty keyspaces;
- accepted schema version/fingerprint change;
- source, target, index, relation, and journal mutation revision change;
- journal fold/cleanup or marker-control change without a logical row mutation;
- change immediately before final revision recheck;
- stale, malformed, oversized, cross-entity, cross-owner, and replayed job IDs;
- job/receipt codec corruption;
- response loss returning the same cached receipt for the same acknowledgement;
- acknowledgement advancing exactly once;
- lost/expired job restart;
- `DeepCompleteClean` only after zero-finding stable sweep;
- `DeepCompleteWithFindings` only after finding-bearing stable sweep; and
- invalidation never reported as corruption or clean.

### Bounds

Evidence must cover exact and one-over limits for:

- rows/entries per page;
- decoded and response bytes;
- finding count and finding payload;
- primary-key/path/value rendering;
- job/receipt bytes and checkpoint variants;
- expression/check operations;
- heap and instruction budget; and
- one next unit too large to classify.

### Recovery

Evidence must cover every current marker/replay/fold failpoint and prove:

- recovered touched rows and derived effects agree;
- schema publication effects agree;
- global Deep is not invoked;
- Quick does not overclaim row integrity;
- recovery remains all-before/all-after; and
- corruption in untouched data remains discoverable by explicit Deep without
  making recovery unbounded.

### SQL And Typed Parity

Evidence must cover:

- bare, Quick, Deep start, and Deep continue forms;
- unsupported syntax failing closed;
- authorization on local and IC targets;
- identical typed/SQL page payloads;
- stable diagnostic codes and bounded display rendering; and
- no public raw checkpoint parsing outside the job owner.

## Performance And Complexity Budget

### Runtime And Heap

Each call must be proportional to its admitted page plus bounded point lookups.
No call constructs a live-key set proportional to database size.

Measure:

- Quick by catalog/control size;
- row pages with no indexes, many indexes, checks, and relations;
- index and reverse phases;
- journal-tail pages;
- finding-free and finding-saturated pages;
- job/receipt encode/decode; and
- stable revision invalidation.

Report peak temporary bytes and instruction/host time, not only wall time.

### Persisted And Response Bytes

Report job/receipt bytes, maximum finding-page bytes, registry growth,
schema/control metadata delta, and whether any row/index/relation/journal format
changed. Job storage must remain isolated from inspected store revisions.

### Wasm Size

Report raw non-gzipped wasm bytes for every slice. Gzip is secondary context.
Reuse accepted decoders and derived-key functions without pulling generated
application validation into the scanner.

### Source Complexity

Each slice reports files touched, approximate line delta, owner count, and
whether the implementation became simpler, stayed neutral, or became more
complex.

The target ownership shape is:

~~~text
accepted schema       -> inspection semantics
Quick                 -> bounded metadata/control closure
Deep phase machine    -> bounded rows + derived state + journal
inspection job owner  -> traversal identity + receipt replay + stable proof
recovery verifier     -> bounded recovered-effect postcondition
repair                -> absent
~~~

## Risks And Mitigations

### Risk: Quick Is Mistaken For Full Integrity

Mitigation: mode and claim scope are explicit in every payload. Quick uses
Quick-specific terminal statuses and never emits a Deep completion claim.

### Risk: Deep Combines Different Database States

Mitigation: jobs bind accepted identity and participating revisions. Any change
invalidates the sweep, including the final boundary recheck.

### Risk: Findings Exhaust Response Memory

Mitigation: independent count/byte bounds stop only after a fully classified
unit and save one bounded receipt. Raw payloads are never echoed.

### Risk: Point Lookups Miss A Unique Duplicate

Mitigation: each authoritative row must match the single exact physical unique
witness; focused proofs cover duplicate pairs across all scan/page positions.

### Risk: Integrity Calls Generated Application Code

Mitigation: accepted row/check/index/relation owners provide every semantic
primitive. Source audits reject generated validator/sanitizer fallback.

### Risk: Recovery Becomes Less Safe When Global Scan Is Removed

Mitigation: recovery verifies its complete bounded effect set and control-state
terminal conditions. Explicit Deep owns unrelated historical inspection.

### Risk: Operators Treat Invalidation As Corruption

Mitigation: `Invalidated` is a distinct status with revision-change diagnostics,
not an integrity finding or error-string convention.

### Risk: Inspection Becomes Repair By Convenience

Mitigation: scanner modules receive read-only capabilities. Mutation requires a
separate design and API.

## Final Rule

An integrity claim is valid only for the scope actually scanned. Quick proves
bounded metadata/control structure. Deep proves accepted row and derived-state
invariants only after canonical exhaustive traversal at one unchanged accepted
schema identity and participating-store revision vector. Invalidation is not
corruption, and inspection never repairs.
