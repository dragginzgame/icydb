# 0.212 Bounded And Resumable Integrity Check

Status: implemented; closeout evidence recorded in `0.212-status.md`

Design cut: 2026-07-16

Protocol review cut: 2026-07-23

Implementation order: after 0.211 closeout

## Decision Summary

IcyDB will replace the current aggregate whole-database integrity report with
two accepted-schema-native inspection lanes:

- `Quick` performs bounded metadata, control-state, codec, fingerprint, and
  ownership checks in one call; and
- `Deep` scans one entity through bounded deterministic phases, returns bounded
  machine-readable findings, and resumes through a separately persisted
  inspection-progress job.

Deep completion requires canonical phase exhaustion followed by a final check
that the accepted inspection plan and complete typed proof vector remain
unchanged. Every page performs the same before/after proof-vector handshake
around temporary findings. A concurrent change invalidates the scan; it is
never reported as a clean result.

Inspection is read-only with respect to the inspected database. Deep writes
only to a physically separate bounded inspection-progress store whose writes,
recovery, and retention do not participate in any inspected revision, marker,
or journal. It never repairs, deletes, rebuilds, coerces, folds, or republishes
inspected state implicitly. Repair is a separate future design.

SQL and typed entrypoints lower to the same scanner:

~~~sql
CHECK INTEGRITY Token QUICK
CHECK INTEGRITY Token DEEP START 'submission_key'
CHECK INTEGRITY DEEP CONTINUE 'job_id' AFTER page_sequence
CHECK INTEGRITY DEEP ABORT 'job_id'
~~~

Deep start is idempotent by an owner-bound submission key so a lost start
response cannot orphan an unknowable job. Public callers never supply raw
phase/checkpoint bytes; the engine resolves an authorization-bound job ID to
trusted bounded progress.

### Non-Negotiable Invariants

1. Quick and Deep consume accepted schema as runtime authority and never
   reconstruct missing contracts from generated models.
2. Every call has hard row, key, byte, finding, decode, heap, and instruction
   bounds.
3. Page fullness is never keyspace or phase exhaustion.
4. Deep traversal uses canonical physical key order and bounded sub-unit
   verifier checkpoints, never an offset or a semantic primary key as physical
   progress.
5. Deep completion requires a full phase sweep at one unchanged accepted
   inspection plan and complete typed proof vector.
6. A stale job or concurrent change returns `Invalidated`, not `Clean` and not
   corruption.
7. Every finding is bounded and machine-readable; callers never parse display
   text for classification.
8. A persisted row violating a validated accepted contract is an integrity or
   corruption finding.
9. A historical row violating a pending 0.211 activation is migration
   validation, not accepted-state corruption.
10. Inspection never repairs implicitly.
11. Inspection accepts only maintained current envelope and physical-key
    codecs while resolving every accepted historical row-layout identity,
    historical enum/type identity, and frozen fill defined by accepted schema.
12. Recovery does not run the resumable global Deep scanner as a hidden
    synchronous postcondition.
13. Every page checks the complete proof vector both before scanning and
    immediately before atomically persisting its checkpoint/receipt; findings
    from a changed page are discarded.
14. Inspection progress persists only in the dedicated progress store and can
    never append to, mark, fold, or revise an inspected store.
15. One malformed or oversized physical unit cannot wedge progress: framing
    failures advance through a bounded finding, and multi-finding units resume
    by verifier family and ordinal.
16. Deep inspects only active accepted index/reverse generations. Pending
    staging belongs to 0.211 validation, and retired bytes follow their
    explicit cleanup lifecycle.
17. Every Deep proof vector contains one durable database-incarnation identity;
    restore, import, replacement, or reinitialization cannot make an old job
    authoritative merely by reusing the same visible revisions.
18. Expiry and abort never replace an unacknowledged page receipt. One persisted
    pending-terminal state preserves that receipt until exact acknowledgement,
    and terminal acknowledgement is itself persisted before retention may
    delete the job.

## Normative Integrity Contract

The maintained contract is:

> Quick verifies bounded catalog and storage-control structure. Deep verifies
> authoritative rows and every accepted derived invariant through resumable
> pages from one bounded, fingerprinted `AcceptedInspectionPlan`. A clean Deep
> result means the complete declared phase set was evaluable and clean at one
> unchanged plan identity and proof vector.

The negative contract is:

> Incomplete, resource-limited, stale, or concurrently invalidated inspection
> is never presented as clean. Inspection never mutates the inspected state.
> `DeepCompleteWithFindings` proves a stable exhaustive physical sweep with
> definite failures; it does not imply that invariants dependent on corrupt
> units were all evaluable.

## Problem Statement

### 1. The Current Report Is Not Bounded By One Message

It visits all stores and materializes every live primary key globally. A large
database can exceed instruction or heap limits before returning any result.

### 2. Aggregate Counters Are Not Operational Diagnostics

A count of missing index entries does not identify the entity, row, logical
index, expected key, observed witness, or stable diagnostic code. Operators
cannot distinguish one repeated defect from many independent defects.

### 3. The Current Report Does Not Cover The Full Accepted Contract

It does not provide a single accepted-native proof of exact field kinds,
required/null policy, enum catalog membership, validated checks, relation target
existence, or reverse relation agreement.

### 4. A Moving Multi-Message Scan Can Overclaim

A resumable job without accepted identity and revision stability can miss rows
around its checkpoint or combine observations from incompatible schema/store
states.

### 5. Recovery And Operator Inspection Have Different Boundaries

Recovery must verify the bounded effects it just replayed before admitting
runtime. It must not depend on a hidden whole-database scan that can exceed one
message. An operator Deep scan is intentionally resumable and cannot be a
synchronous recovery postcondition.

## Goals

0.212 must:

1. add one bounded Quick lane;
2. add one bounded resumable Deep lane for a named entity;
3. build one bounded fingerprinted `AcceptedInspectionPlan` from the same
   accepted artifacts used by write admission and use it for both lanes;
4. carry plan identity, phase, canonical physical/sub-unit checkpoint, and the
   complete typed proof vector in one bounded inspection job;
5. return bounded typed findings with stable diagnostic codes;
6. distinguish clean, findings, incomplete, resource-limited, invalidated,
   uninspectable-authority, and uninspectable-storage outcomes;
7. verify row-key/row-value identity and recursive exact accepted kinds;
8. verify forward and reverse index agreement without a global live-key map;
9. verify validated check constraints and immediate relation integrity;
10. verify accepted schema/control/journal/ownership structure at the right
    lane and phase;
11. keep pending constraint violations in migration-validation context;
12. replace recovery's global integrity-report dependency with bounded
    recovered-effect verification;
13. expose typed and SQL surfaces over the same scanner;
14. never repair implicitly;
15. hard-cut the aggregate-only public report and old current-form DTOs; and
16. persist jobs/receipts in a physically separate non-journaled progress
    store;
17. bind every job/proof to a durable database incarnation and preserve
    unacknowledged receipts across expiry/abort; and
18. close with correctness, corruption, concurrency, resource, performance,
    wasm-size, and source-complexity evidence.

## Non-Goals

0.212 does not:

- repair rows, indexes, relations, schema, markers, journals, or ownership;
- rebuild an index or reverse relation generation;
- resume or validate a pending 0.211 constraint activation;
- inspect pending activation staging or declare retired-generation bytes
  corrupt merely because cleanup has not finished;
- infer accepted schema from generated `EntityModel` or callbacks;
- build a second integrity-specific entity/relation model tree;
- provide a planner hint from a partial scan;
- promise a database-wide atomic snapshot across unrelated stores;
- add cross-entity business assertions;
- validate arbitrary application validators or sanitizers;
- add schema-less or open-ended content inspection;
- support caller-defined regex, callbacks, scripts, or inspection functions;
- add a mutable runtime journal/durability mode or a session-level equivalent
  of `PRAGMA journal_mode`;
- sleep, wait for a lock, or hide retry loops inside one message; bounded
  progress, resource, and invalidation outcomes remain explicit;
- retain the old aggregate report as an alias or compatibility wrapper; or
- design a repair API.

## Terminology

### Quick

A one-call bounded check of accepted metadata and storage control structure. A
clean Quick result does not claim that every row or derived entry was scanned.

### Deep

A multi-phase authoritative row and derived-state scan. Deep may claim entity
integrity only after stable completion. Its entity scope is source-owned: it
checks rows, indexes, and relations owned by the selected accepted entity,
including direct target lookups and that entity's reverse projections. Inbound
relations owned by other source entities belong to those entities' Deep scans.

### Finding

A bounded machine-readable observation of a definite invariant failure in one
examined key/value or accepted metadata object.

### Invalidated

The accepted inspection plan or any typed proof component changed during a Deep
sweep. Invalidation is terminal concurrency/progress state, not corruption.

### Uninspectable

A load-bearing accepted-authority artifact cannot be decoded, fingerprinted,
or resolved well enough to define inspection semantics. This is terminal
accepted-authority corruption, not a successful sweep with ordinary findings.

### Uninspectable Storage

An authoritative ordered physical container cannot advance to another key or
prove exhaustion because its traversal structure is corrupt. This is a typed
terminal incomplete outcome, distinct from an accepted-authority failure and
from a malformed individual key/value that can be classified and passed. It
never permits a clean or complete-with-findings claim.

### Clean

Every required phase was evaluable and exhausted without findings at one
unchanged proof identity. Only Deep can make the row-level clean claim.

## Authority Model

1. accepted schema owns row layout, exact kinds, null policy, enum identity,
   primary key, indexes, relations, and validated constraints;
2. authoritative data stores own rows;
3. accepted index definitions plus authoritative rows define expected index
   entries;
4. physical index stores are derived state to compare against expectation;
5. accepted relations plus source rows define expected reverse entries;
6. marker, journal, fold watermark, and allocation registries own durability
   control facts;
7. the database lifecycle owner owns one durable nonzero
   `DatabaseIncarnationId`;
8. store mutation revisions own multi-message stability;
9. one accepted, bounded, fingerprinted `AcceptedInspectionPlan` projects the
   maintained artifacts above into inspection traversal/checker inputs without
   becoming a second schema authority;
10. the physically separate inspection-progress store owns bounded traversal
   position and receipt replay but no inspected row/index/relation authority;
   and
11. generated models, application validators, and sanitizers are not integrity
    authority.

### Accepted Inspection Plan And Target Boundary

Accepted-schema selection builds one `AcceptedInspectionPlan` from the same
accepted row layouts, historical layout/type catalogs, compiled constraints,
active index/relation generations, store bindings, and physical codecs used by
write admission. The plan is bounded, canonical, fingerprinted, and tied to the
accepted schema version/fingerprint. The existing accepted entity fingerprint
already commits to the canonical snapshot payload; the plan fingerprint binds
that fingerprint, the accepted store-root fingerprint, entity/store identity,
and one inspection-semantics version. The implementation does not persist a
second copy of the snapshot or a redundant plan-payload digest. Quick and Deep
load this artifact from the accepted selection; diagnostics do not walk the
authored graph again or compile a parallel interpretation.

The plan has no separately mutable publication state. Initial schema
acceptance, later catalog mutation, and 0.211 activation promotion atomically
publish the accepted snapshot/root inputs from which it is built, and the
accepted-selection cache refuses a plan whose identity or root authority no
longer matches. A validated constraint or active generation therefore cannot
change without a new accepted entity/root identity and plan fingerprint, so an
existing Deep job invalidates before using mixed semantics.

The plan contains, at minimum:

- the accepted entity and its one IcyDB store binding;
- the canonical raw data-key interval for that entity;
- maintained current envelope/key codecs;
- every accepted current and historical row layout plus the historical
  type/enum identities and frozen fills needed to reconstruct a current
  logical row image;
- current validated constraint programs;
- active accepted forward-index and source-owned relation/reverse generations;
- direct relation-target store bindings;
- canonical phase order, verifier-family order, and physical key bounds; and
- the exact proof-component read set described below.

0.212 targets only an entity present in this accepted, IcyDB store-bound
projection. An `ic-model` declaration that has no IcyDB store binding is not an
inspection target. Typed integration rejects it as
`UnsupportedModelOnlyTarget`; SQL/name lookup reports that it is not a
registered accepted IcyDB entity. Neither case is a missing-row or corruption
finding.

Relation meaning comes from the one centrally resolved `rel = "..."` graph
that schema acceptance projected into accepted relation artifacts. Integrity
adds no relation macro, registry, or model tree. IcyDB remains independent of
`ic-model`: the accepted plan contains only IcyDB-owned accepted identities,
programs, codecs, and bindings.

Plan admission distinguishes:

- **load-bearing authority failure**: the accepted root/entity/plan,
  row-layout history, active index/relation identity, compiled constraint
  program, store binding, key domain, or proof-component handle cannot be
  decoded, fingerprinted, or resolved. Quick returns `QuickUninspectable`;
  Deep start creates no job, and a discovered failure in an existing job
  terminates as `Uninspectable`.
- **inspectable invariant failure**: a physical row, derived entry, target
  lookup, or traversable control/journal unit definitely violates a trusted
  plan. Deep emits a finding and continues. If that failure prevents dependent
  verifier families from running for the unit, the result records those
  families as blocked.
- **physical-container traversal failure**: a Deep phase's trusted ordered
  container cannot identify the next physical key or prove exhaustion. The
  maintained `ic-stable-structures` iterator has no fallible traversal API and
  the Wasm profile uses aborting panics, so a load-bearing B-tree failure traps
  the call before IcyDB can persist a typed receipt. The outstanding receipt
  and checkpoint remain unchanged, so the job cannot falsely complete or skip
  work and the owner may abort it through the progress store. Returned
  authority or unit failures remain typed `Uninspectable` or findings.

`DeepCompleteWithFindings` therefore means a stable exhaustive traversal of
every physical unit in the declared scope with at least one definite failure.
It does not claim every dependent invariant was evaluable.
`DeepCompleteClean` is available only when all required verifier families were
evaluable and clean. Neither completion status is available after physical
container traversal failure.

## Public Surface

### SQL

The opening grammar is:

~~~sql
CHECK INTEGRITY entity QUICK
CHECK INTEGRITY entity DEEP START 'submission_key'
CHECK INTEGRITY DEEP CONTINUE 'job_id' AFTER page_sequence
CHECK INTEGRITY DEEP ABORT 'job_id'
~~~

Rules:

- Quick has no job/acknowledgement syntax;
- Deep start requires one bounded owner-scoped idempotency/submission key,
  performs the complete accepted-plan/proof-vector admission handshake,
  atomically creates the job, and returns page sequence zero;
- retrying the same owner/key/entity start at sequence zero returns the same
  job and cached start receipt; if that job was already advanced by another
  authorized actor, start replay returns typed `SubmissionAlreadyAdvanced`
  with the job/current sequence rather than pretending earlier pages were
  delivered; reuse with a different target fails with a typed conflict;
- undecodable load-bearing authority returns `Uninspectable` and creates no
  job;
- `DEEP CONTINUE` requires exactly one engine-issued job ID owned by the
  authorized caller/capability and the last acknowledged page sequence;
- Continue and abort do not accept an entity: the authorized job already owns
  the accepted entity identity, and lowering resolves and validates that
  identity from the job record;
- `DEEP ABORT` records an owned pending-abort intent without replacing an
  unacknowledged page. Exact acknowledgement of that page atomically produces
  the terminal `IntegrityAbortReceipt`; abort then replays that receipt and
  never immediately deletes the record;
- unknown modes, extra entities, predicates, projection, ordering, limits, and
  user-selected internal phase/checkpoint values fail closed;
- Quick may use the read-only introspection lane;
- Deep start/continue uses an authorized update/admin lane because it advances
  physically separate inspection-progress state; and
- deployment policy exposes a distinct integrity capability rather than
  silently inheriting ordinary row-query access.

### Typed

The typed surface makes invalid combinations unrepresentable:

~~~text
IntegrityCheckRequest
  = Quick {
      entity: AcceptedEntityIdentity,
    }
  | DeepStart {
      entity: AcceptedEntityIdentity,
      submission_key: IntegritySubmissionKey,
    }
  | DeepContinue {
      job_id: IntegrityJobId,
      acknowledged_sequence: u64,
    }
  | DeepAbort {
      job_id: IntegrityJobId,
    }
~~~

There is no mode plus optional job/acknowledgement tuple. SQL lowers into this
enum and does not own an alternate report shape. Raw checkpoint
constructors/decoders remain private to the job owner.

### Result Shape

Conceptually:

~~~text
IntegrityCheckResponse
  = Quick(QuickIntegrityResult)
  | Deep(IntegrityJobReceipt)

QuickIntegrityResult {
    entity: EntityIdentity,
    database_incarnation_id: DatabaseIncarnationId,
    accepted_schema_version: u32,
    accepted_schema_fingerprint: Fingerprint,
    status:
      QuickCompleteClean
      | QuickCompleteWithFindings
      | QuickUninspectable(AcceptedAuthorityError)
      | QuickResourceLimited(ResourceDiagnostic),
    total_findings: u64,
    omitted_findings: u64,
    findings: Vec<IntegrityFinding>,
}

DeepIntegrityPage {
    job_id: IntegrityJobId,
    page_sequence: u64,
    phase: IntegrityPhase,
    status:
        DeepInProgress
      | DeepCompleteClean
      | DeepCompleteWithFindings
      | Invalidated(ProofVectorChange)
      | Uninspectable(AcceptedAuthorityError)
      | ResourceLimited(ResourceDiagnostic)
      | Expired(LeaseDiagnostic),
    pages_completed: u64,
    findings_seen: u64,
    findings: Vec<IntegrityFinding>,
    blocked_verifier_families: Vec<IntegrityVerifierFamily>,
}

IntegrityAbortReceipt {
    job_id: IntegrityJobId,
    page_sequence: u64,
    status:
        TerminationPending(Aborted | Expired)
      | Terminal(Aborted | Expired),
}

~~~

The durable job, not every receipt, owns entity, incarnation, accepted-schema,
inspection-plan, and proof-vector identity. The authorized opaque job ID
selects that record for continuation, replay, acknowledgement, and abort.
Copying those facts into every page would create a second persisted receipt
representation without adding an authorization or proof invariant.

`Invalidated`, `Uninspectable`, `ResourceLimited`, `Expired`, `Aborted`, and
both Deep completion outcomes are terminal and cannot advance. Their exact
terminal acknowledgement/replay request remains valid. A terminal receipt
remains replayable until its acknowledgement is persisted and retention later
removes the record. Ordinary page-boundary progress is `DeepInProgress`.

Sub-unit checkpoints and framing classification guarantee that admitted
physical corruption cannot itself produce `ResourceLimited`.
`ResourceLimited` is reserved for a typed engine/platform inability to perform
the minimum bounded verification atom or checked counter exhaustion. It leaves
the inspected database unchanged, persists a terminal receipt without
advancing past an unclassified atom, and requires a new job after the resource
condition is corrected.

## Quick Check

Quick performs only claims with one explicit bounded proof owner:

| Quick family | Bound/proof | Quick claim |
| --- | --- | --- |
| Database lifecycle control | One bounded current-form incarnation record | Complete nonzero current incarnation identity |
| Accepted root, target entity and `AcceptedInspectionPlan` | Hard accepted byte/count/depth bounds plus exact fingerprint agreement | Complete target-plan decode and identity agreement |
| Current and historical row-layout/type/enum catalogs reachable by the plan | Hard plan byte/count/depth bounds | Complete reachable layout/fill closure |
| Fields, primary key, constraints, active indexes and source-owned relations | Hard per-entity accepted maxima encoded in the plan | Complete identity, slot, program and active-generation pairing |
| Pending activation lifecycle descriptors | Hard accepted job/count/record bounds | Complete descriptor/state pairing only; staging-entry correctness remains exclusively with 0.211 validation |
| Marker/control state for each participating store | One bounded current-form record per store in the plan | Complete envelope/state and recovery-readiness check |
| Journal topology control | One bounded control record per participating journaled store | Tail/fold bounds and root agreement only; every tail entry is deferred to `JournalTails` |
| Stable-memory allocation registry | Complete 256-value `u8` memory-ID domain plus bounded stable-key owner records | Complete identity/ownership uniqueness, not a sampled summary |
| Storage capabilities | One bounded descriptor per participating store | Complete declared/runtime capability agreement |
| Data rows | Unbounded family | Deferred to `Rows` |
| Active forward-index entries | Unbounded family | Deferred to `IndexEntries` |
| Active reverse entries for source-owned relations | Unbounded family | Deferred to `ReverseRelations` |

The opening inventory must freeze the numeric accepted maxima used by every
“hard” row above. In particular, allocation uniqueness is a Quick claim only
because all 256 memory IDs and their bounded stable-key owner records are
visited in one bounded call. If implementation evidence disproves that current
owner bound, 0.212
must add a named `AllocationEntries` Deep phase before implementation; it may
not substitute an unauthenticated summary.

The frozen opening maxima are:

- 512 KiB for one canonical accepted entity snapshot, whose byte bound is also
  the single cardinality bound for its contained metadata vectors;
- depth 64 for accepted recursive field/value structure;
- depth 32, 256 nodes, 64 children, 64 membership items, 4 KiB of literal
  payloads, and 16 KiB of canonical bytes for one accepted check expression;
- four components for primary and index keys;
- 256 UTF-8 bytes for one constraint name;
- 512 KiB each for accepted enum and composite catalogs, inside a 16 MiB
  accepted bundle with 4 KiB store paths;
- 64 KiB for one pending validation job, with 16 store revisions, 64 findings
  per receipt, and 32 fields per finding;
- 16 MiB for one current commit marker or journal batch/control record; and
- the complete 256-value `u8` stable-memory ID domain.

These are existing accepted/current-form owners, not integrity-only duplicate
admission policies. If any of them changes, the owner and this inventory change
together.

An authoritative maintained root may replace entry traversal only when its
codec, transactional maintenance, and authenticated proof of the exact claimed
property are part of accepted authority. A count, watermark, hash, or cached
summary without that contract proves only its own bounded well-formedness.

Quick verifies storage mode/capability agreement from the same typed facts
already exposed by store and entity diagnostics; it does not duplicate those
facts in an integrity-only wire shape. A well-formed heap store is not
corruption, but operator renderers must label it `volatile`, and
production-oriented tooling that composes integrity and storage diagnostics
should emit a volatility advisory. A journaled store receives no such advisory
when its declared and runtime capabilities agree. The advisory does not change
an otherwise clean integrity status. Inspection never changes storage mode;
there is no mutable PRAGMA-like durability setting.

Quick always performs the complete bounded check even when more diagnostics are
found than fit in one response. It counts every finding with checked arithmetic,
retains only the canonical bounded prefix, and returns the exact
`total_findings` plus `omitted_findings`. The invariant is
`total_findings == findings.len() + omitted_findings`; response rendering may
derive `findings_truncated` from `omitted_findings != 0` rather than persist a
second boolean authority. `QuickCompleteClean` requires `total_findings == 0`,
and `QuickCompleteWithFindings` requires it to be nonzero. A count or response
bound failure returns typed `QuickResourceLimited` and never a complete status.
For an incomplete Quick status, the counters describe only findings observed
before the typed stop and make no full-scope total claim.

Quick therefore never silently truncates and then says it returned every
diagnostic. Every deferred checking claim names the Deep phase that owns it.

Quick normally runs after recovery admission. A corrupt/nonrecoverable marker
that prevents recovery fails through recovery's typed corruption boundary
before Quick. After admission, a readable but inconsistent marker/journal
control unit is an inspectable Quick finding; an accepted-plan failure remains
`QuickUninspectable`. Quick does not traverse an ordered physical row, index,
relation, or journal-entry container; those resumable traversal failures belong
to the stable-structure trap boundary described above. A separate future
forensic surface may inspect raw pre-recovery state; Quick does not bypass
recovery gating.

## Deep Phase Machine

The semantic phase order is:

~~~text
QuickMetadata
    -> Rows(physical_data_key, verifier_family, ordinal)
    -> IndexEntries(active_schema_index_id, physical_key, verifier_family, ordinal)
    -> ReverseRelations(active_relation_id, physical_key, verifier_family, ordinal)
    -> JournalTails(store_id, sequence_or_chunk_key, verifier_family, ordinal)
    -> FinalProofVectorCheck
    -> DeepCompleteClean | DeepCompleteWithFindings
~~~

Phases with no applicable entries advance without fabricating a checkpoint.
One call advances within one phase or crosses a bounded number of empty phase
boundaries; it never performs unbounded nested traversal.

All physical phases use one conceptual bounded cursor:

~~~text
PhysicalUnitCheckpoint<K>
  = BeforeFirst
  | Within {
      physical_key: K,
      verifier_family: IntegrityVerifierFamily,
      ordinal: u32,
    }
  | After {
      physical_key: K,
    }
~~~

`Within` means every deterministic verification atom through that
family/ordinal has been classified, but the physical unit is not complete.
`After` means the unit is fully classified and the next ordered visit is
strictly greater than that raw key. Family and ordinal order is frozen in the
accepted inspection plan. Page, ordinal, and cumulative-finding counters use
checked arithmetic; exhaustion never wraps.

The smallest verification atom and one maximum bounded finding must fit in one
maximum receipt. A row, index entry, reverse entry, or journal chunk can
therefore span pages without being skipped or requiring all of its findings in
one response.

### Row Phase

The accepted plan defines one canonical lexicographic raw data-key interval
`[entity_start, entity_end)` from the entity's accepted physical tag/prefix.
Membership in that interval is decidable from bounded raw key bytes without
decoding the semantic primary key. `Rows` visits every physical key in the
interval and checkpoints the exact raw key. `PrimaryKeyValue` is optional
diagnostic context produced only after successful key decoding.

The underlying data-store key type has one frozen maximum encoded length, and
the job checkpoint bound is at least that maximum. Schema acceptance proves
the entity intervals are canonical and non-overlapping. A raw key inside the
selected interval is attributed to that entity even when its suffix or
semantic primary key is malformed.

An entity scan makes no claim about raw keys outside every recognized accepted
entity interval in a shared physical store. Such unowned physical garbage
requires a future/store-level physical integrity target unless the opening
inventory adds a bounded complete store-domain phase; it is not silently
included in the selected entity's clean claim.

For every physical row unit, Deep verifies:

- data-store key decodes in the current format;
- key entity identity matches the requested entity;
- the maintained current row envelope decodes with bounded
  length/count/depth;
- its stamped layout identity resolves to either the current or an accepted
  historical row layout;
- physical slot closure is exact for that stamped historical/current layout;
- historical field/type/enum identities decode under that layout, and frozen
  historical fills supply fields introduced after it;
- the reconstructed logical row image conforms to the current accepted field
  contract;
- storage key equals the row's accepted primary-key fields exactly;
- required fields are present and nullable policy is respected;
- every scalar and composite value recursively matches its exact accepted kind;
- integer bounds, decimal scale/precision, text/blob bounds, collection bounds,
  ordering/uniqueness encoding, and enum catalog identity are valid;
- every validated `CHECK` evaluates to `True` or `Unknown`;
- every expected index entry exists and exactly matches the derived row witness;
- every accepted relation target exists; and
- every expected reverse relation entry exists and exactly identifies the
  source row/relation.

An envelope whose declared bytes/count/depth exceeds accepted limits produces
one bounded `OversizedRow`/`OverdeepRow` finding from framing and advances
without materializing the payload. A malformed layout stamp or unit can block
dependent logical/index/relation verifier families for that row; those
families are recorded as blocked, and physical progress continues by the
sub-unit cursor.

Pending 0.211 activation gates are metadata-checked only by the standalone
bounded Quick lifecycle check. Deep's `QuickMetadata` phase consumes the
accepted plan and active accepted facts; it does not inspect staging or bind
pending progress into its claim. Deep does not label a historical candidate
violation as corruption. Activation validation is performed by
`VALIDATE CONSTRAINT` and uses migration diagnostics.

### Index Entry Phase

For every physical entry in each active accepted index-generation domain owned
by the selected entity, Deep verifies:

- key framing, entity tag, physical index placement, and value/witness codec;
- logical `SchemaIndexId` resolves through the current accepted index owner;
- referenced authoritative row exists;
- the row still derives this exact index key and value;
- conditional membership still includes the row;
- an entry encoded inside the active generation's physical domain names that
  active generation; and
- unique-index logical keys do not map to conflicting row witnesses.

Forward row verification plus exact witness comparison detects duplicate
logical unique keys without retaining every logical key: if two authoritative
rows derive the same unique key, at least one cannot match the single
authoritative physical witness. Focused evidence must prove this property for
all supported unique encodings.

Pending activation staging belongs exclusively to the 0.211 activation
validator. Retired generations belong to their explicit cleanup lifecycle and
are not corruption merely because bytes still exist. Deep does not traverse
either domain. A pending, retired, unknown, or otherwise wrong generation
encoded *inside an active generation's physical domain* is corruption.

### Reverse Relation Phase

For every physical entry in each active reverse generation owned by a relation
whose source is the selected entity, Deep verifies:

- current-form key/value framing and relation identity;
- target identity belongs to the accepted relation target entity;
- source row exists;
- source row still contains the target in the declared relation field and
  cardinality;
- source/target primary-key kinds match accepted relation identity; and
- the entry belongs to the active accepted reverse generation.

Deep for entity `E` proves outgoing relations owned by `E`: its row phase
point-checks every direct target and expected reverse entry, and its reverse
phase scans those active physical reverse domains for extra/divergent entries.
It does not prove inbound relations owned by other source entities, even if
their reverse bytes are physically stored beside `E`; those are covered by the
source entities' Deep scans.

As with forward indexes, pending staging is owned only by 0.211 validation and
retired generations by cleanup. A wrong generation encoded inside the active
reverse domain is corruption; bytes in a separate pending/retired domain are
outside the 0.212 claim.

### Journal Tail Phases

For every participating journaled store in the accepted plan's dependency
closure, in canonical store order, Deep verifies the physical tail in bounded
sequence/chunk pages:

- complete current-form batch/chunk envelopes;
- strictly ordered sequence identity above the fold watermark;
- no unexplained gaps, overlaps, duplicate chunks, or batch identity changes;
- fold watermark and canonical/tail boundary agreement;
- schema/row/index/relation records reference known current-format families;
- marker/tail state is compatible with completed recovery.

Deep inspection does not replay or fold the journal.

## Inspection Job And Stable Completion

### Physically Separate Inspection-Progress Store

Deep progress lives in a dedicated `InspectionProgressStore`, not in an
inspected data, index, schema, marker, journal, or database-control store. Its
contract is:

- it owns one explicit stable allocation identity and key namespace reserved
  by IcyDB configuration; that allocation cannot alias any database role;
- job creation, advancement, terminalization, acknowledgement, abort, and
  eligible retention deletion each replace or delete one bounded job record
  atomically;
- the record key is a domain-separated canonical digest of database
  incarnation, owner, and submission key; the record stores and rechecks the
  full bounded triple, so idempotent start and conflict detection require no
  second mutable index or multi-record transaction;
- it does not use the database commit marker, journal, fold machinery, store
  mutation revision, allocation-generation mutation, or schema publication
  path;
- its record writes never appear in any journal traversed by Deep and its own
  revision/generation is deliberately absent from the proof vector;
- database recovery neither replays nor depends on inspection progress;
  progress-store opening validates its own current-form header and bounded
  records independently;
- a corrupt job record fails that job with a typed progress-record error but
  cannot block database recovery or become an inspected-database finding;
- ordinary reopening of exactly the same persistent database retains its
  incarnation and may retain a valid job. Restore, import, database replacement,
  or reinitialization that could reuse any earlier proof counter must mint a new
  incarnation before serving inspection or mutation requests, so an old job
  invalidates even when every other visible proof component is identical; and
- retention never deletes `InProgress`, `TerminalPending`, or an unacknowledged
  `Terminal`. After a frozen inactivity lease expires, it records
  `TerminalPending(Expired)` without replacing the cached page. Exact
  acknowledgement converts that pending intent into the terminal receipt
  without touching inspected state.

The atomic single-record guarantee is part of the progress-store primitive,
not an assertion that a multi-record journal transaction happens to be
atomic. The opening allocation inventory must reserve and collision-check this
memory explicitly. Progress-store capacity is bounded per owner and globally;
capacity failure rejects Deep start without changing inspected state.

### Database Incarnation

`DatabaseIncarnationId` is a nonzero, bounded, durable database-lifecycle
identity, independent of schema fingerprints, mutation revisions, index or
relation generations, journal sequence/fold values, and allocation generation.
It is not restored from logical import payloads and is not derived from any of
those counters. The lifecycle owner must mint a never-reused value from
authority outside the replaced/restored payload. If the platform cannot prove
freshness, restore/import/replacement/reinitialization fails closed before the
database becomes available.

A byte-for-byte ordinary reopen of the same persistent state retains the
identity. A restore, clone into another database, import, database replacement,
or reinitialization that can make earlier revision values visible must publish
a fresh identity atomically with making that state current. Incarnation
exhaustion or freshness uncertainty is a typed lifecycle failure; values never
wrap or saturate. Changing the identity invalidates retained jobs lazily through
the ordinary proof-vector comparison and requires no scan of the progress
store.

The incarnation is current-form persisted control metadata and a mandatory
component of Quick control closure, every Deep start `A/B` capture, every page
proof comparison, every job record, and job/submission-key domain separation.
It is not schema meaning and therefore does not alter the accepted schema
fingerprint.

### Exact Proof-Vector Read Set

The accepted inspection plan resolves the complete direct entity dependency
closure before job creation. It includes the selected source store, every
direct outgoing relation target store, stores holding active forward/reverse
generations for the selected entity, and each participating store's durability
control. It is not a transitive business-relation closure.

Every Deep read has an exact typed proof component:

| Deep read | Required proof component |
| --- | --- |
| Database lifecycle identity | Database incarnation identity |
| Accepted inspection plan | Accepted schema version/fingerprint and plan fingerprint |
| Selected entity source rows | Source data-store mutation revision |
| Direct outgoing relation target lookups | Data-store mutation revision for every direct target store |
| Active forward-index lookup/scan | Revision/generation for every active physical index generation |
| Active source-owned reverse lookup/scan | Revision/generation for every participating active reverse generation |
| Marker/control state | Marker/control generation for every participating store |
| Journal traversal | Journal topology, tail, and fold generation for every participating journaled store |
| Physical allocation facts | Allocation-registry generation |

The vector begins with exactly one database-incarnation component and
canonicalizes the remaining components by proof-component kind, store identity,
accepted object identity, and generation. A generic “store changed” value
cannot replace a required generation-specific component. The
inspection-progress store and its allocation contents are not proof components;
its allocation *ownership* is covered by the database allocation-registry
generation, which does not change when a job record changes.

Deep start performs a capture handshake:

1. load and validate the accepted plan;
2. resolve its complete read set;
3. capture proof vector `A`;
4. re-resolve/verify the plan identity and capture proof vector `B`;
5. if `A != B`, return typed start invalidation and create no job; and
6. otherwise atomically create the job with `B` and its sequence-zero receipt
   in the progress store.

### Job Record

The bounded current job shape is:

~~~text
IntegrityJobV1 {
    id: IntegrityJobId,
    database_incarnation_id: DatabaseIncarnationId,
    owner: IntegrityJobOwner,
    submission_key: IntegritySubmissionKey,
    entity: EntityIdentity,
    accepted_schema_version: u32,
    accepted_schema_fingerprint: Fingerprint,
    inspection_plan_fingerprint: Fingerprint,
    phase: IntegrityPhase,
    checkpoint: IntegrityCheckpoint,
    captured_proof_vector: IntegrityProofVector,
    state: IntegrityJobState,
    lease_deadline: InspectionLeaseDeadline,
    findings_seen: u64,
    pages_completed: u64,
    blocked_verifier_families: BoundedVerifierFamilySet,
    last_receipt: IntegrityReceiptEnvelope,
}

IntegrityJobState
  = InProgress
  | TerminalPending(IntegrityPendingTerminal)
  | Terminal {
      outcome: IntegrityTerminalOutcome,
      receipt_acknowledged: bool,
    }

IntegrityPendingTerminal
  = Expired(LeaseDiagnostic)
  | Aborted

IntegrityTerminalOutcome
  = DeepCompleteClean
  | DeepCompleteWithFindings
  | Invalidated
  | Uninspectable
  | ResourceLimited
  | Expired
  | Aborted

IntegrityReceiptEnvelope {
    replay_key: Start | Continue { acknowledged_sequence: u64 },
    receipt: IntegrityJobReceipt,
}

IntegrityJobReceipt
  = Page(DeepIntegrityPage)
  | Abort(IntegrityAbortReceipt)
~~~

A job contains no generated Rust path, user callback, row after-image,
accepted artifact copy, or unbounded collection.

The state/receipt codec rejects impossible pairs. `InProgress` and
`TerminalPending` must retain a nonterminal page receipt. `Terminal` must retain
the matching terminal page or abort receipt. `receipt_acknowledged` exists only
for `Terminal`, is persisted by atomic record replacement, and is the sole
retention authority. `TerminalPending` is not a terminal result: it freezes
advancement while preserving the one outstanding receipt and its replay key.

`IntegrityJobId` is a lookup identity, not trusted progress supplied by the
caller. The registry resolves it, verifies the current caller/capability owns
the job, and loads the private checkpoint. Public SQL and typed APIs never
accept raw continuation bytes. Unkeyed hashes, CRCs, and opaque encoding are not
treated as authentication; the record-key digest above is only lookup and
collision-domain identity, and authorization plus full-key comparison still
run.

Every job lookup, including retry, abort, terminal acknowledgement, and
terminal replay, first compares the job's database incarnation with the current
lifecycle identity. A mismatch returns typed job invalidation and never replays
an old clean, findings, expiry, or abort receipt as current-database authority.
Within one unchanged incarnation, terminal receipts remain historical
point-in-time results and do not re-run the full proof vector merely because a
later ordinary mutation occurred.

### Page Advancement Handshake

Every in-progress job step:

1. resolves and authorizes the job and accepted entity/store authority;
2. verifies job format, state, entity, plan fingerprint, phase, physical/sub-unit
   checkpoint, checked counters, and bounded receipt state;
3. captures the complete current proof vector and compares it with the job's
   vector;
4. atomically persists/replays terminal `Invalidated` without scanning if it
   differs;
5. scans from the private checkpoint and constructs findings, blocked-family
   facts, next checkpoint, counters, and candidate status only in bounded
   temporary memory;
6. recaptures the *entire* proof vector after scanning;
7. if it differs, discards the candidate page and atomically persists an
   `Invalidated` receipt with no candidate findings/checkpoint advancement;
   otherwise
8. atomically replaces the one job record with its next checkpoint and receipt.

The same second comparison governs a candidate terminal page; there is no
weaker final-only check. Pages execute without `await` and use the same
mutation serialization boundary as writes, but the end-of-page comparison
remains a required invariant and test oracle.

Receipts have monotonically increasing page sequence. The caller acknowledges
the prior sequence when requesting the next page. The receipt envelope freezes
the request acknowledgement that produced it. A retry with that same
acknowledgement returns the cached bounded receipt instead of advancing again.
This prevents response loss from silently dropping findings. Receipt storage is
capped to one page per job.

Sequence and cumulative finding increments use checked arithmetic. The frozen
maximum in-progress page count is at most `u64::MAX - 1`, reserving one
representable sequence for any terminal receipt, including pending expiry or
abort. If the next page or cumulative finding total would exceed its maximum,
candidate findings are discarded and the reserved next sequence persists
replayable terminal
`ResourceLimited(CounterExhausted)` with the prior exact cumulative count and
without advancing the physical checkpoint. No counter saturates or wraps.

The acknowledgement transition table is:

| Job/receipt state | Request | Result |
| --- | --- | --- |
| `InProgress`, last receipt sequence `n` | Continue acknowledging `n` | Advance once and write sequence `n + 1` |
| Any state, last receipt produced from acknowledgement `n` | Retry Continue acknowledging `n` | Replay the cached receipt without advancing |
| `TerminalPending(reason)`, last receipt produced from acknowledgement `n` | Retry Continue acknowledging `n` | Replay the cached receipt; preserve the pending reason |
| `TerminalPending(reason)`, last receipt sequence `n` | Continue acknowledging `n` | Persist terminal reason receipt at sequence `n + 1`; do not scan |
| `Terminal`, last receipt sequence `n`, acknowledgement false | Continue acknowledging `n` | Persist acknowledgement true and replay the terminal receipt; no new sequence |
| `Terminal`, last receipt sequence `n`, acknowledgement true | Retry Continue acknowledging `n` | Replay the terminal receipt; no state change |
| `Terminal` | Continue with any other sequence | Typed stale-ack error |
| `InProgress` | Abort | Persist `TerminalPending(Aborted)` and return/replay `TerminationPending` with the unchanged outstanding receipt |
| `TerminalPending(Aborted)` | Abort | Replay `TerminationPending`; no state change |
| `TerminalPending(Expired)` | Abort | Replay `TerminationPending(Expired)`; no state change |
| Terminal | Abort | Replay the existing terminal/abort receipt; never reopen |
| `InProgress` past its inactivity lease | Bounded retention pass | Persist `TerminalPending(Expired)`; preserve the outstanding receipt |
| `TerminalPending` | Bounded retention pass | No state change |
| `Terminal` with acknowledgement false | Bounded retention pass | No state change |
| `Terminal` with acknowledgement true and minimum retention elapsed | Bounded retention pass | Delete the one progress record |

Each controller request is synchronous and does not yield between progress
load, bounded advancement, and record replacement. The supported boundary
therefore cannot observe a concurrent in-flight advancement and owns no
separate busy state or advancement-lease protocol. A future yielding controller
would require an explicit new durable protocol rather than reusing this one.
The first pending abort/expiry intent wins and cannot be overwritten by another
terminal cause. `Invalidated`, `Uninspectable`, and `ResourceLimited` cannot
advance; only their exact terminal acknowledgement/replay request remains valid.
`Expired` likewise requires a new submission key/job after its terminal receipt
is acknowledged; it is abandonment state, not corruption or proof
invalidation.

Database incarnation plus owner plus `IntegritySubmissionKey` is unique for the
retained job/idempotency window. A lost Deep-start response is retried with the
same key and returns the same sequence-zero job receipt. Different payload
reuse is a typed conflict. Because no Continue can be formed without the
returned job ID, a genuinely lost start response cannot have advanced beyond
sequence zero through that request. After acknowledged-terminal retention and
record deletion, reusing the key creates a new job and never resurrects old
progress.

The SQL/typed execution envelope therefore carries `job_id` and
`acknowledged_page_sequence`; callers do not author phase/checkpoint/revision
facts. The shell may render this as:

~~~sql
CHECK INTEGRITY DEEP CONTINUE 'job_id' AFTER 7
~~~

`DeepCompleteClean` means the entire phase machine completed with zero findings
and zero blocked verifier families at one stable proof identity.
`DeepCompleteWithFindings` means every physical unit in scope was exhausted at
one stable proof identity and at least one definite finding was recorded; its
blocked-family set states which dependent invariants were not evaluable.

Only terminal jobs whose `receipt_acknowledged` bit is durably true are removed
explicitly or by one bounded retention policy after a frozen minimum window.
Unacknowledged terminal receipts and `TerminalPending` records are never
deleted by time-based retention; they count against frozen per-owner/global
capacity, so abandoned clients can exhaust only that bounded capacity and
cannot cause unbounded progress storage. Expiry first records a pending intent,
exact page acknowledgement writes its terminal receipt, exact terminal
acknowledgement persists the acknowledgement bit, and only a later pass may
delete it. Deletion touches only the progress store and cannot change or
invalidate an active database proof.

Journaled V1 uses the durable mutation revision established for 0.210. Quick is
available to heap stores. Data mutation revision alone is insufficient because
journal folding may change physical traversal while preserving logical row
revision. The exact vector therefore carries every participating store's
journal topology/tail/fold and marker/control generations. A fold, cleanup,
marker transition, or journal topology change invalidates the sweep.

Deep heap support requires a trustworthy runtime epoch plus monotonic mutation
revision that invalidates jobs across every write and runtime reinitialization;
until that authority exists, heap Deep rejects explicitly.

## Bounded Findings

Conceptually:

~~~text
IntegrityFinding {
    diagnostic_code: IntegrityDiagnosticCode,
    error_class: ErrorClass,
    severity: IntegritySeverity,
    entity: EntityIdentity,
    store: StoreIdentity,
    phase: IntegrityPhase,
    physical_key: BoundedPhysicalKey,
    primary_key: Option<PrimaryKeyValue>,
    field_paths: Vec<FieldPath>,
    constraint_id: Option<ConstraintId>,
    constraint_name: Option<String>,
    schema_index_id: Option<SchemaIndexId>,
    relation_id: Option<RelationId>,
    expected: Option<BoundedIntegrityValue>,
    observed: Option<BoundedIntegrityValue>,
}
~~~

Finding count, string bytes, path count/depth, physical/primary-key bytes, and
expected/observed rendering are independently bounded. Raw row payloads,
unbounded collection contents, secrets, and arbitrary application messages are
not echoed.

Stable finding families include:

- inspectable marker/journal control mismatch;
- malformed physical data key and unowned-domain caveat;
- oversized, overdeep, or malformed row framing;
- unknown/unusable stamped historical layout;
- primary-key key/value mismatch;
- missing/extra/undecodable row slot;
- exact-kind, nullability, bounds, decimal, enum, or composite violation;
- validated check constraint violation;
- missing/divergent/orphan index entry;
- duplicate logical unique key;
- missing relation target;
- missing/divergent/orphan reverse relation entry;
- marker/journal/fold discontinuity; and
- allocation/ownership conflict.

A load-bearing accepted root/plan/layout-program failure uses the terminal
`Uninspectable` authority diagnostic family instead of entering this finding
list.

A finding observed in one serialized call is definite for the bytes examined in
that call. If the job is later invalidated, the overall sweep loses its stable
completeness claim; the invalidation itself is not recast as corruption.

## Error And Outcome Classification

The scanner distinguishes:

- malformed request, job ID, or acknowledgement: typed validation/misuse error;
- model-only or non-store-bound target: typed unsupported/not-registered error,
  never a finding;
- unsupported storage capability: typed unsupported error;
- load-bearing accepted authority failure: terminal `Uninspectable` with a
  typed authority-corruption diagnostic;
- ordered physical-container failure below the fallible IcyDB boundary: the
  platform traps before receipt replacement; the retained checkpoint and
  outstanding receipt remain authoritative and no completion claim is made;
- engine/platform inability to execute the minimum verification atom or
  checked counter exhaustion: terminal `ResourceLimited` with bounded context;
- plan/proof-vector change: terminal `Invalidated`;
- inactivity lease expiry: `TerminalPending(Expired)` until the outstanding
  receipt is acknowledged, then terminal `Expired`; neither state is corruption
  or a clean/dirty claim;
- persisted current accepted invariant failure: corruption/integrity finding;
- corrupt physical unit blocking dependent checks: definite finding plus
  bounded blocked-verifier-family coverage;
- pending activation historical incompatibility: migration validation, outside
  Deep accepted-integrity findings; and
- impossible scanner state with trusted authority intact: internal error.

Malformed individual bounded keys/values remain progressable findings.
Container traversal failure is never converted to an empty iterator,
`ResourceLimited`, an internal error, or `DeepCompleteWithFindings`. Display
text is never classification authority.

## Recovery Boundary

The pre-0.212 recovery path called the whole-database aggregate integrity
report after replay. Slice 6 removes that unbounded dependency and the old
report together.

Recovery instead verifies the bounded effect set owned by the recovered marker
or replay batch:

- each recovered row put/delete matches its final authoritative row state;
- each staged index/reverse effect matches that row operation;
- schema publication identity matches the recovered schema record;
- marker, journal sequence, and fold boundary reach the documented terminal
  state; and
- every checked effect fits the existing commit caps.

This is recovered-effect verification, not a claim that the entire database was
scanned. Quick may run after recovery for bounded metadata/control closure.
Deep remains an explicit operator action.

Recovery interruption tests must prove that the bounded verifier sees the same
all-before/all-after states as normal marker recovery. It must not clear a
marker merely because a global scan could not finish.

## No Implicit Repair

Quick and Deep are read-only over inspected state. Deep may atomically replace
only its one bounded record in the physically separate progress store. They
never:

- insert, delete, or rewrite a row;
- create, delete, or rewrite an index/reverse entry;
- rebuild a staged or accepted generation;
- clear a marker;
- replay, truncate, or fold a journal;
- advance a fold watermark;
- publish schema;
- coerce a value; or
- suppress a finding by mutating evidence.

A later repair design must name repair authority, dry-run planning, bounds,
atomicity, recovery, authorization, and post-repair verification explicitly.
It must not be hidden behind `CHECK INTEGRITY`.

## Persisted And Wire Formats

Deep job records and public result DTOs are current-form V1 values with:

- magic/version or equivalent discriminant;
- explicit total byte bound;
- bounded owner/submission key, job state, phase, physical/sub-unit checkpoint,
  proof vector, blocked-family set, pending-terminal/terminal-ack state, receipt
  replay key, and last-receipt variants;
- the exact `DatabaseIncarnationId` captured by the job;
- canonical entity/store/schema identity;
- canonical revision ordering;
- bounded counts and strings; and
- fallible decode with typed incompatible-format/corruption errors.

The progress store has its own current-form header/allocation record and
single-record atomic replacement codec. It has no database journal record,
marker variant, or fold state. Job records are diagnostic progress state, not
accepted schema or inspected data authority. Their proof validity derives from
the accepted inspection-plan identity and complete typed proof vector. Job
creation, page receipt, acknowledgement, terminal state, retention, and
removal are bounded and independently recoverable.

The database-control persisted-format inventory adds the current
`DatabaseIncarnationId`. There is no decoder that synthesizes it from schema,
store, journal, allocation, or generated-model state. Pre-0.212 control bytes
without the identity follow the project hard cut and fail current-form open;
operators recreate or explicitly import through the lifecycle path that mints a
fresh identity.

“Current-form only” for persisted inspection means the maintained current
progress/job/result format and the maintained current physical key/row
envelopes. It does not reject an accepted historical row layout stamped inside
the current row envelope; those layouts and their frozen fills resolve through
the accepted inspection plan.

The old `IntegrityReport`, `IntegrityStoreSnapshot`, and `IntegrityTotals`
aggregate-only public shapes are removed when the new surface lands. There is
no alias, wrapper, dual endpoint, or decoder for their retired current form.

## Pre-1.0 Hard Cut

0.212 directly replaces the old integrity report execution and public DTO
surface. It does not keep:

- the global live-key materialization path;
- an unbounded all-store convenience wrapper;
- aggregate-only results beside finding pages;
- generated-model fallback scanning;
- recovery's global report call;
- mode plus optional job/acknowledgement request shapes;
- semantic-primary-key continuation checkpoints;
- inspection jobs persisted through database marker/journal machinery;
- Deep traversal of pending activation or retired-generation domains;
- old job/receipt spellings or format versions; or
- anti-resurrection tests for removed surfaces.

Tests cover the maintained Quick/Deep current surface instead.

## Risks And Mitigations

### Risk: Quick Is Mistaken For Full Integrity

Mitigation: mode and claim scope are explicit in every payload. Quick uses
Quick-specific terminal statuses and never emits a Deep completion claim.

### Risk: Deep Combines Different Database States

Mitigation: jobs bind the accepted inspection plan and complete typed proof
vector. Any change invalidates the sweep. Every page compares the full vector
before and after scanning and discards temporary findings/checkpoint state on
change.

### Risk: Restore Reuses The Same Proof Counters

Mitigation: the proof vector and job identity include the durable
`DatabaseIncarnationId`. Any restore, import, replacement, clone, or
reinitialization capable of reusing counter values must mint a fresh
incarnation from lifecycle authority outside the restored payload. Identical
schema, revision, journal, generation, and allocation values therefore cannot
make an old job authoritative.

### Risk: Progress Writes Invalidate Or Extend The Scan

Mitigation: one separately allocated non-journaled progress store performs
atomic single-record replacement and is absent from the proof vector and all
inspected journals. Its recovery and retention are independent.

### Risk: Findings Exhaust Response Memory

Mitigation: Deep's independent count/byte bounds stop after a deterministic
verifier-family/ordinal atom, not necessarily a whole physical unit. One
maximum finding fits a receipt; raw payloads are never echoed. Quick continues
its entirely bounded check after its returned finding prefix is full and
reports exact total and omitted counts.

### Risk: Expiry Or Abort Drops A Lost Page

Mitigation: expiry and abort first persist `TerminalPending` while leaving the
single cached receipt unchanged. Only exact acknowledgement of that receipt
produces the terminal receipt, and only exact acknowledgement of the terminal
receipt sets the durable retention-eligibility bit. Time-based retention never
deletes either pending state or an unacknowledged terminal receipt.

### Risk: A Malformed Or Oversized Unit Wedges Its Phase

Mitigation: phases checkpoint raw bounded physical keys, classify oversized
framing without materialization, and resume large units by verifier
family/ordinal. Semantic primary keys are diagnostic context only.

### Risk: Corrupt Container Traversal Looks Exhausted

Mitigation: IcyDB advances only after an iterator returns a classified key and
the complete proof remains unchanged. A stable-structure traversal trap rolls
back the call before progress-record replacement, so it cannot become empty
iteration, checkpoint advancement, or either Deep completion status. The
current storage dependency cannot convert that platform trap into a typed
receipt.

### Risk: Historical Rows Are Misclassified As Current-Layout Corruption

Mitigation: current envelopes resolve accepted historical layout/type/enum
identities and frozen fills before current logical constraints and derived
effects are evaluated.

### Risk: Corrupt Authority Produces False Exhaustive Findings

Mitigation: load-bearing plan/catalog/program failures terminate
`Uninspectable`; only trusted-plan physical failures become findings. Clean
requires every verifier family evaluable.

### Risk: Pending Or Retired Generations Look Corrupt

Mitigation: Deep traverses active domains only. 0.211 owns staging and cleanup
owns retired domains; only a wrong generation encoded inside an active domain
is an integrity finding.

### Risk: Quick Overclaims An Unbounded Family

Mitigation: every Quick family has a frozen hard bound, an authoritative
authenticated root, or a named Deep phase. Unauthenticated summaries prove
only their own well-formedness.

### Risk: Point Lookups Miss A Unique Duplicate

Mitigation: each authoritative row must match the single exact physical unique
witness; focused proofs cover duplicate pairs across all scan/page positions.

### Risk: Integrity Calls Generated Application Code

Mitigation: accepted row/check/index/relation owners provide every semantic
primitive. Source audits reject generated validator/sanitizer fallback.

### Risk: Recovery Becomes Less Safe When Global Scan Is Removed

Mitigation: recovery verifies its complete bounded effect set and control-state
terminal conditions. Explicit Deep owns unrelated historical inspection.

### Risk: Operators Treat Invalidation As Corruption

Mitigation: `Invalidated` is a distinct status with revision-change diagnostics,
not an integrity finding or error-string convention.

### Risk: Inspection Becomes Repair By Convenience

Mitigation: scanner modules receive read-only capabilities. Mutation requires a
separate design and API.

## Final Rule

An integrity claim is valid only for the scope actually scanned. Quick proves
only its explicitly bounded metadata/control families. Deep proves the
selected accepted store-bound entity's source-owned row/derived-state scope
only after canonical physical exhaustion at one unchanged accepted inspection
plan and complete typed proof vector. `DeepCompleteWithFindings` may record
blocked dependent verifier families; only `DeepCompleteClean` proves the full
accepted invariant set was evaluable and clean. Invalidation is not corruption,
physical-container traversal failure is typed incomplete rather than false
exhaustion, progress persistence is physically separate, unacknowledged
receipts survive expiry/abort, and inspection never repairs.
