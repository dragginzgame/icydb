## Patch 1 Result

`AcceptedInspectionPlan` now compiles from one verified accepted catalog
selection and owns:

- the accepted entity/store identity;
- the exact accepted entity snapshot;
- the store-local enum/composite catalog authority;
- the ordinary final-after-image write program;
- a deterministic fingerprint binding entity/store identity, accepted schema
  revision/version/fingerprint, accepted store-root fingerprint, and the

The session accepted-schema cache now carries this one artifact rather than

The plan deliberately reuses the existing accepted entity fingerprint as the
commitment to canonical snapshot meaning. It does not persist a duplicate
snapshot or redundant plan-payload digest.

## Patch 2 Result

The current version-1 database-control envelope now carries one nonzero
`DatabaseIncarnationId` beside the transient commit-marker payload:

- virgin database initialization mints it from lifecycle authority outside the
  persisted payload;
- ordinary reopen, marker publication, marker clear, and recovery preserve it;
- zero or malformed identities fail closed;
- the accepted schema fingerprint does not include it; and
- the sole maintained control envelope is current-form only. Pre-0.212 control
  bytes have no decoder, synthesis path, or upgrade-on-open behavior.

Native initialization uses platform entropy. Wasm initialization binds the
canister identity, IC time, and a checked process-local sequence under a
domain-separated hash. The repository has no supported raw restore/import
surface; any future lane capable of reusing prior proof counters must mint and
atomically publish a new incarnation before serving the restored database.

The Quick engine now:

- resolves a registered entity through runtime routing;
- consumes the canonical accepted inspection plan rather than generated schema
  meaning;
- runs after recovery and revalidates the current control envelope;
- resolves every source-owned accepted relation through its existing active
  projection and includes the direct target store in bounded control checks;
- validates canonical heap/journaled capability and allocation shapes without
  treating well-formed volatile storage as corruption;
- consumes the committed `ic-memory` allocation capability and independently
  checks its bounded declaration closure for duplicate keys or slots;
- reads each participating journaled store's existing bounded tail/fold proof
  identity, reporting readable control drift as a typed finding and corrupt
  control bytes as `Uninspectable`;
- returns the accepted entity identity, durable incarnation, accepted schema
  version/fingerprint, typed completion state, exact finding count, bounded
  finding prefix, and exact omitted count;
- distinguishes a selected load-bearing plan failure as typed
  `Uninspectable` rather than clean completion; and
- retains only 64 findings while continuing exact checked counting.

Patch 7 now exposes the final request enum only through the trusted typed,
direct SQL, and generated controller-gated frontends. All three call this same
Quick route.

The former aggregate `IntegrityReport`, `IntegrityStoreSnapshot`, and
`IntegrityTotals` DTOs are no longer part of the public core re-export.
At this patch boundary recovery's historical scanner remained private; Patch 6
later deleted it after bounded recovered-effect verification became

## Patch 3 Result

The accepted inspection plan now also compiles the exact current/historical
row contract and active forward-index projection once per accepted identity.
Its one fingerprint-bound accepted row program serves both write admission and
integrity: ordinary writes evaluate accepted checks plus pending gates, while
the integrity sub-unit view selects only validated accepted checks in stable
constraint-ID order. No accepted expression is compiled twice. Integrity
loading is accepted-native: it no longer rejects a valid DDL-evolved accepted
schema merely because it differs from the current generated proposal. Typed
query and mutation routes retain their separate generated-compatibility
checks.

The internal row engine now:

- traverses only the canonical raw data-key interval for the selected accepted
  entity;
- resumes from an exact private `BeforeFirst`, `Within`, or `After` physical
  checkpoint rather than a semantic primary key;
- validates row envelopes through the accepted row-layout authority, including
  authenticated historical layouts and frozen fills;
- classifies the physical key, each active field, primary-key agreement, each
  validated accepted check, expected active forward-index entries, relation
  targets, and expected reverse-relation entries in deterministic sub-unit
  order;
- excludes pending check activation from accepted-corruption evaluation;
- records malformed prerequisites once and explicitly blocks dependent
  verifier families instead of reporting silent clean success;
- fails closed if an exact within-row resume key disappears or falls outside
  the entity interval; and
- enforces independent row, verifier-atom, finding, and loaded-row-byte bounds.

A full page is not treated as exhaustion. Exhaustion is reported only after
the canonical physical interval itself ends. The data-store visitor supplies
one bounded `RawRow` at a time; the engine does not retain the entity row set,
all keys, or all findings.

Initial accepted relations legitimately use physical generation zero, while
activation candidates require a nonzero isolated generation. The shared
relation projector now exposes distinct active and candidate constructors so
row inspection can verify both without weakening candidate admission.

### Focused Evidence

- tiny atom pages resume repeatedly within rows and eventually classify the
  exact entity interval;
- malformed row envelopes produce one typed finding and blocked dependent
  families;
- disappeared exact resume rows fail closed as corruption;
- field-path and expression-index rows report missing expected forward
  entries by stable logical index ID;
- accepted relation rows report missing expected reverse entries by stable
  relation ID; and
- the existing accepted structural reader remains the sole decoder for current
  and authenticated historical row layouts.

The row core is crate-private and has no typed or SQL application surface.
Until Patch 5 binds it to durable authorized Deep jobs, narrowly documented

## Patch 5 Result

Deep now has one durable internal controller over the accepted inspection
plan. Its canonical order is Quick metadata, rows, active forward-index
domains, active source-owned reverse-relation domains, every participating
journal tail, and the final unchanged-proof check. The controller:

- captures and persists one typed proof vector containing database
  incarnation, accepted-plan identity, database control, allocation
  generation, participating store generations/journal identities, and active
  index/relation generations;
- requires exact A/B proof equality before idempotent start and exact pre/post
  equality around every bounded page;
- discards candidate findings and checkpoint advancement on proof drift before
  persisting terminal `Invalidated`;
- keeps private raw checkpoints, cumulative counters, blocked-family state,
  the exact replay key, and one bounded receipt inside one current-form job;
- replays lost start/page/terminal responses without scanning or advancing
  twice;
- preserves the outstanding page through abort or inactivity expiry by first
  persisting `TerminalPending`, then emits the terminal receipt only after the
  exact page acknowledgement;
- persists terminal acknowledgement and deletes only acknowledged terminal
  records after the frozen retention window; and
- rejects heap Deep explicitly until a trustworthy heap runtime epoch exists.

The journal phase validates current envelopes, sequence continuity, fold
watermark agreement, and non-reused batch identity. Duplicate identity proof
is constant-memory and resumable: one candidate batch is compared against
earlier live sequences under two-batch/decoded-byte page bounds rather than
retaining an unbounded ID set. A malformed batch, sequence gap, or repeated
batch identity is a bounded typed finding and traversal continues from an
exact checkpoint; malformed prior batches also block the batch-identity
verifier family. Corrupt ordered-container topology that prevents identifying
the next physical batch remains terminal `UninspectableStorage`.

`InspectionProgressStore` occupies a separately declared stable-memory
allocation and never participates in an inspected store's marker, journal,
mutation revision, allocation generation, or proof vector. It has one
current-form version-1 header and checksummed job envelope, a 512 KiB record
bound, 64-job global capacity, eight-job owner capacity, bounded 16-record
retention pages, and fail-closed job/state/proof/checkpoint validation.
Decoded checkpoints must name a verifier family owned by their phase, a
captured index/relation/store ordinal, and a journal sequence inside the
captured fold/append interval. Corrupt individual job records remain isolated
from database recovery.

The schema/build contract now reserves and collision-checks the progress
memory ID and stable key. Maintained canister fixtures allocate one new memory
slot; no existing row, index, relation, accepted-schema, journal, cursor, or
commit-marker payload changed. The progress header/job envelope is a new
diagnostic-progress persisted surface, not a compatibility reader for any
older form.

The Patch 3/4 row and derived cores are now reachable only through the durable
controller in production. Their direct page helpers are test-only, their
temporary `dead_code` allowances were removed, and Patch 7 now exposes the
controller only through its authorized typed and SQL frontends.

### Focused Evidence

- exact bounded journal traversal covers empty, chunked, resumed,
  duplicate-identity, gap, malformed, and fold-boundary cases;
- a journal envelope corrupted without changing proof counters produces a
  typed finding, blocks batch-identity proof, and reaches stable
  `DeepCompleteWithFindings`;
- proof capture is stable without mutation, changes after a durable write, and
  rejects heap storage without a runtime epoch;
- start and page response loss replay exact cached receipts;
- the empty accepted fixture traverses every phase and acknowledges one clean
  terminal receipt;
- proof drift after start terminalizes as `Invalidated` with no candidate
  findings;
- abort and inactivity expiry preserve the outstanding receipt until exact
  acknowledgement;
- acknowledged terminal retention deletes the job while unacknowledged
  terminal state remains;
- fair maintained retention reaches jobs beyond the frozen 16-record page
  bound and wraps only after exhausting the progress keyspace;
- wrong owners, corrupt current job bytes, and ninth retained owner jobs
  reject with typed failures; and
- decoded phase-family and journal-interval checkpoints cannot skip captured
  work; and
- focused all-feature `icydb-core` checking and Deep/journal tests pass.

Repository-wide tests, all feature combinations, Wasm builds, raw Wasm size,
and instruction measurements remain human-owned closeout evidence under
`AGENTS.md`.

## Patch 6 Result

Recovery no longer invokes a whole-database integrity scan after replay.
Before marker clearance it now verifies only the bounded final effect set
already owned by the recovered marker:

- every final marker-owned row put matches exact authoritative row bytes;
- every final row delete is absent from authoritative row storage;
- rebuilt forward-index and reverse-relation effects for recovered row puts
  match the accepted rebuild projection exactly;
- persisted schema snapshots, accepted-schema publication identity, and
  constraint-validation job changes match their recovered records;
- every marker batch sequence is covered by the durable fold watermark; and
- every journaled store has no physical tail batch remaining after fold.

The verifier walks at most the existing 16 MiB marker payload and retains one
bounded identity per final logical effect. Journal closure is one ordered-map
lookup per registered journaled store. Row deletes require no old-row
reconstruction: the preceding canonical rebuild cleared derived storage and
rebuilt it solely from rows that remain authoritative.

The former aggregate integrity scanner, report DTOs, and tests that preserved
that superseded surface were deleted outright. Explicit Quick and Deep
inspection remain the sole integrity workflows; recovery neither invokes nor
aliases them.

### Focused Evidence

- recovered marker effects pass after normal row/index rebuild;
- deleting a recovered authoritative row makes verification fail closed;
- deleting one recovered derived index entry makes verification fail closed;
- existing marker/fold/index-rebuild interruption tests still cross the
  verification boundary before marker clearance; and
- focused all-feature `icydb-core` checking and recovery tests pass.

No public API, SQL behavior, dependency, package version, accepted-schema,
row, index, relation, journal, cursor, or marker encoding changed. The old

## Patch 7 Typed Surface Result

The trusted typed frontend now exposes one `IntegrityCheckRequest` enum with
exact `Quick`, `DeepStart`, `DeepContinue`, and `DeepAbort` variants over the
existing durable controller. Quick and Deep start pin the generated entity
selector's tag, path, and store path, then require an exact match with current
accepted authority. Continue and abort accept only the engine-issued job ID;
private phase and checkpoint state remain inside the progress record.

`DbSession::execute_admin_integrity` requires the caller to supply a bounded
stable owner identity after enforcing controller or equivalent
integrity-specific authorization. The same owner is checked for start replay,
continuation, terminal acknowledgement, and abort. Candid-decoded entity,
owner, submission-key, and job-ID inputs are revalidated at execution instead
of trusting Rust constructors.

The public facade preserves job-protocol failures as typed
`IntegrityCheckError::Job` variants and maps accepted-authority or physical
engine failures through the normal typed database error payload. Existing
Quick and Deep lifecycle tests now exercise every public request variant. No
second integrity runner, raw checkpoint input, SQL behavior, persisted format,
or package version was added.

## Patch 7 Direct SQL Result

The direct administrative SQL frontend accepts exactly the four design forms:
entity-bearing Quick and Deep start, plus entity-free Deep continue and abort.
It uses the existing reduced-SQL lexer and one dedicated grammar, then lowers
directly into `IntegrityCheckRequest`. SQL returns the same
`IntegrityCheckResult` receipts and owns no report DTO or execution runner.

Quick/start entity names resolve only to registered runtime selectors. The
typed controller then performs its existing exact tag, entity-path, store-path,
and accepted-plan validation before inspection. Continue and abort accept a
64-character hexadecimal engine-issued `IntegrityJobId`; malformed, zero,
unknown, wrong-owner, and stale identities retain typed protocol failures.
SQL cannot provide a phase, physical checkpoint, proof vector, predicate,
projection, ordering, limit, or extra continuation entity.

Focused parity coverage starts through one frontend and replays or advances
through the other, proving exact Quick results and Deep receipts rather than
comparing independently reconstructed output. Parser coverage rejects missing,
extra, negative, and unsupported forms. Generated endpoint and config exposure
were kept outside this direct-API slice instead of temporarily routing
integrity through query or DDL capability.

## Patch 7 Bounded Retention Result

Every well-formed authorized typed or direct SQL integrity attempt now runs one
progress-only maintenance page after the requested operation. Each page
visits at most 16 retained job records, expires idle jobs by persisting
`TerminalPending(Expired)`, and removes only terminal records whose receipt
acknowledgement and retention deadline are already durable.

The maintained scan uses one heap-only cursor per progress allocation so
records after the first page cannot starve. Exhaustion wraps the cursor to the
first key; ordinary restart safely loses only that scheduling hint and resumes
from the first key. The cursor is not persisted authority, is absent from job
or proof formats, and cannot change inspection results. Requested operations
run before maintenance so an abort or continuation result does not depend on
the advisory cursor position. A maintenance failure remains typed and
fail-closed.

Focused evidence covers 17 retained jobs across the 16-record page boundary,
wraparound without repeated expiry, and a real typed Quick request freezing an
expired job before a later abort observes the existing expiry reason.
