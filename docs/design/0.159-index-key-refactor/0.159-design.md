# IcyDB 0.159 â€” Compact Key Encoding, Key-Owned Index Entries, and Key Taxonomy Cleanup

## Status

Active

Implemented slices:

* `0.159.0` introduces the DB-internal key taxonomy wrappers, compact
  canonical primary-key encoders, and the live primary data-store hard cut from
  fixed `EntityTag + StorageKey(64)` bytes to variable-width
  `EntityTag + EncodedPrimaryKey` bytes. The same slice hard-cuts secondary
  index primary-key suffixes from 64-byte `StorageKey` segments to compact
  `EncodedPrimaryKey` segments, adds decode/validation boundaries for
  `RawDataStoreKey` and `RawIndexStoreKey`, moves full data-store scans to
  entity-prefix ranges, and updates commit-marker row-key sizing/decode for
  compact keys. Index entry values are now key-owned as well: `RawIndexKey`
  owns row identity and `RawIndexEntry` stores only the storage-owned
  presence/existence witness.

## Summary

IcyDB currently uses a universal fixed-width `StorageKey` frame for primary-key payloads. The frame is 64 bytes regardless of the natural size of the logical key. This keeps the existing storage protocol simple, but it permanently amplifies key size across the primary data store and secondary indexes.

A recent repository audit found that this cost is larger than just primary-row key width. Secondary indexes currently store the primary key in both the `RawIndexKey` suffix and the `RawIndexEntry` membership payload, producing roughly 135 bytes of primary-key-related storage per indexed row before index components and map overhead. The audit also found that secondary index keys are already variable-length stable-memory data, which weakens the argument that all persisted key material must be fixed 64-byte frames.

0.159 proposes a hard-cut redesign of the key model:

1. Replace universal 64-byte primary-key frames with compact canonical primary-key encodings.
2. Persist primary-key kind tags; use bounded key classes as implementation groupings.
3. Keep `Principal` and `Subaccount` as first-class key/index types.
4. Keep `Account` as a `Key64` class only; it must not force all keys to 64 bytes.
5. Remove duplicated primary-key membership from secondary index values.
6. Clean up key naming and conceptual ownership so each key representation has exactly one responsibility.

This is a hard storage-format cut. No compatibility with the old 64-byte key format is required.

# Motivation

The current key stack has become difficult to reason about:

```text
StorageKey
DataKey
RawDataKey
RawIndexKey
RawIndexEntry
primary-key suffix
index component bytes
commit marker key bytes
```

Some layering is valid, but the current terminology obscures the distinction between:

```text
logical key value
canonical encoded key bytes
persisted data-store key
persisted index-store key
index component encoding
index row identity
```

The main technical issue is that `StorageKey` is too central and too vague. It sounds like â€śthe database storage key,â€ť but it actually represents an encoded primary-key payload. That payload is then embedded into data-store keys, index-store keys, index-entry values, commit markers, cursors, and recovery structures.

The 0.159 redesign should move IcyDB from a `StorageKey`-centered model to a **canonical key encoding model**.

# Goals

* Reduce primary-key and secondary-index storage amplification.
* Preserve deterministic stable-memory ordering.
* Preserve range-scan correctness.
* Preserve composite secondary index support.
* Preserve `Principal` and `Subaccount` key/index support.
* Keep `Account` support if practical, without allowing it to inflate all other keys.
* Remove duplicated primary-key membership from secondary index values.
* Use shared primitive ordered encoders wherever primary keys and secondary index components overlap.
* Own physical descending secondary-index key ordering for SQL DDL and generated
  index contracts, rather than treating `DESC` as a parser-only DDL feature.
* Clarify key naming and representation ownership.
* Make each key type correspond to exactly one conceptual layer.
* Keep public schema syntax unchanged where possible.
* Prefer a clean hard cut over compatibility scaffolding.

---

# Non-Goals

* No backwards-compatible migration from the old 64-byte format.
* No removal of `Principal` or `Subaccount` support.
* No general move to arbitrary unbounded key payloads.
* No user-facing requirement to manually select physical index formats.
* No change to logical query semantics.
* No full optimizer redesign.

# Current State

The audit established the current primary data key shape as:

```text
RawDataKey = EntityTag(u64 big-endian) + StorageKey(64 bytes)
```

for a total fixed key size of 72 bytes. It also found that `RawDataKey` is represented as a fixed byte array and is stored in the primary data BTree as `BTreeMap<RawDataKey, RawRow>`. Entity-local scans are implemented as range scans over `RawDataKey`.

The current secondary index key shape is already variable-length:

```text
RawIndexKey =
  key_kind(1)
  + index_id(entity_tag u64 + ordinal u16)
  + component_count(1)
  + repeated len(u16) + component
  + len(u16) + primary_key
```

The primary-key suffix currently contains the full 64-byte `StorageKey`. Separately, `RawIndexEntry` also stores primary-key membership, creating duplication.

Current natural payload sizes are much smaller than 64 bytes for most key types:

| Type            |                 Natural payload | Current frame |
| --------------- | ------------------------------: | ------------: |
| `Nat` / `Nat64` |                         8 bytes |      64 bytes |
| `Int` / `Int64` |                         8 bytes |      64 bytes |
| `Timestamp`     |                         8 bytes |      64 bytes |
| `Ulid`          |                        16 bytes |      64 bytes |
| `Principal`     | max 29 bytes, or 30 with length |      64 bytes |
| `Subaccount`    |                        32 bytes |      64 bytes |
| `Account`       |                        62 bytes |      64 bytes |
| `Unit`          |                         0 bytes |      64 bytes |

The audit found that `Principal` and `Subaccount` do not justify universal 64-byte keys. `Account` is the only currently admitted primary-key type close to 64 bytes.

# Design Decision

IcyDB should hard-cut from:

```text
universal fixed 64-byte StorageKey
```

to:

```text
kind-tagged compact canonical primary-key bytes with bounded classes
```

The persisted primary-key discriminator should be a logical kind tag, not merely a width class tag. The class model remains useful for sizing and implementation grouping:

| Class   | Types                                          |
| ------- | ---------------------------------------------- |
| `Key8`  | `Int`, `Nat`, `Timestamp`                      |
| `Key16` | `Ulid`, future `Int128` / `Nat128` if admitted |
| `Key32` | `Principal`, `Subaccount`                      |
| `Key64` | `Account` only                                 |

The key point is that `Account` can remain supported without forcing every primary key and every secondary index suffix to consume 64 bytes.

The persisted kind tag determines the logical type, width class, decoder, validation rules, and ordering semantics. A class alone is not enough because `Key8` can mean `Int`, `Nat`, or `Timestamp`, and those are different logical domains.

# Conceptual Model

0.159 should explicitly separate five concepts.

## 1. Logical primary-key value

The typed user/domain value that identifies an entity row.

Examples:

```rust
Nat
Int
Timestamp
Ulid
Principal
Subaccount
Account
Unit
```

Proposed conceptual name:

```rust
PrimaryKeyValue
```

This is not persisted directly.

---

## 2. Canonical encoded primary key

The deterministic, ordered byte encoding of a logical primary-key value.

Proposed name:

```rust
EncodedPrimaryKey
```

Properties:

* compact
* canonical
* ordered
* validated
* bounded by class
* suitable as a suffix inside data and index store keys

This replaces the conceptual role currently occupied by `StorageKey`.

---

## 3. Encoded index component

The deterministic, ordered byte encoding of one secondary-index field.

Proposed name:

```rust
EncodedIndexComponent
```

This is distinct from `EncodedPrimaryKey` because secondary indexes support more orderable value types than primary keys.

For example, secondary index components may support:

```text
Bool
Text
Decimal
Float
Int128
Nat128
Principal
Subaccount
Account
```

while primary keys may support a smaller keyable set.

---

## 4. Data-store key

The complete persisted key used in the primary row store.

Proposed structured name:

```rust
DataStoreKey
```

Proposed raw persisted name:

```rust
RawDataStoreKey
```

Shape:

```text
DataStoreKey =
  EntityTag
  + EncodedPrimaryKey
```

`DataStoreKey` is the logical/structured representation.

`RawDataStoreKey` is the serialized stable-memory BTree key representation.

---

## 5. Index-store key

The complete persisted key used in the secondary index store.

Proposed structured name:

```rust
IndexStoreKey
```

Proposed raw persisted name:

```rust
RawIndexStoreKey
```

Shape:

```text
IndexStoreKey =
  IndexId
  + EncodedIndexComponent[]
  + EncodedPrimaryKey
```

The final `EncodedPrimaryKey` suffix is the row-identity tiebreaker.

This suffix should be the authoritative row identity for the index entry.

# Naming Cleanup

## Retire or demote `StorageKey`

`StorageKey` should not remain the central term.

The name is too broad and has become misleading. It sounds like a full database storage key, but it is actually an encoded primary-key payload.

Preferred replacement:

```rust
EncodedPrimaryKey
```

or, if the project wants to emphasize byte ownership:

```rust
PrimaryKeyBytes
```

Recommended name:

```rust
EncodedPrimaryKey
```

because the key property is not merely that it is bytes, but that it is canonical and order-preserving.

---

## Rename data key types

Current-style names:

```rust
DataKey
RawDataKey
```

Recommended names:

```rust
DataStoreKey
RawDataStoreKey
```

Rationale:

* `DataStoreKey` means â€śthe key used by the primary row store.â€ť
* `RawDataStoreKey` means â€śthe serialized persisted key used by the stable-memory BTree.â€ť

---

## Rename index key types

Current-style names:

```rust
RawIndexKey
```

Recommended names:

```rust
IndexStoreKey
RawIndexStoreKey
```

Rationale:

* `IndexStoreKey` is the structured key.
* `RawIndexStoreKey` is the persisted BTree key.
* The name makes it clear that this is the full secondary index store key, not just a logical index value.

---

## Rename index value type

Current-style name:

```rust

```

Recommended names, depending on final shape:

```rust
IndexEntryValue
```

```rust

```

or:

```rust

```

Preferred design:


The key architectural rule is:

```text

```

The target key vocabulary should be:

| Layer                | Proposed Type           | Responsibility                                              |
| -------------------- | ----------------------- | ----------------------------------------------------------- |
| Logical row identity | `PrimaryKeyValue`       | Typed entity primary-key value                              |
| Encoded row identity | `EncodedPrimaryKey`     | Canonical compact ordered primary-key bytes                 |
| Encoded index field  | `EncodedIndexComponent` | Canonical ordered secondary-index component bytes           |
| Structured data key  | `DataStoreKey`          | `EntityTag + EncodedPrimaryKey`                             |
| Persisted data key   | `RawDataStoreKey`       | Stable-memory BTree key bytes for primary rows              |
| Structured index key | `IndexStoreKey`         | `IndexId + components + primary suffix`                     |
| Persisted index key  | `RawIndexStoreKey`      | Stable-memory BTree key bytes for secondary indexes         |
| Index value          | `IndexEntryValue`       | Minimal presence/witness payload, no duplicated primary key |

# Encoding Design

## EncodedPrimaryKey

Proposed shape:

```text
EncodedPrimaryKey =
  key_kind_tag
  + canonical_payload
```

Primary-key kind tags should be persisted unless the implementation deliberately refactors every generic decode path to become schema-known.

Persisting the kind tag is recommended because:

* commit markers may need schema-independent decode/diagnostics
* corruption handling is easier
* malformed key rejection is clearer
* debug tooling remains simpler
* index suffix decoding remains self-describing

The extra byte is acceptable because the design still eliminates the 64-byte universal frame.

The kind tag determines:

* logical type
* width class
* decoder
* malformed-key validation
* ordering semantics

The recommended initial mapping is:

| Kind tag | Logical type  | Class   |
| -------- | ------------- | ------- |
| `0x01`   | `Nat`         | `Key8`  |
| `0x02`   | `Int`         | `Key8`  |
| `0x03`   | `Timestamp`   | `Key8`  |
| `0x04`   | `Ulid`        | `Key16` |
| `0x05`   | `Principal`   | `Key32` |
| `0x06`   | `Subaccount`  | `Key32` |
| `0x07`   | `Account`     | `Key64` |
| `0x08`   | `Unit`        | kind-only singleton |

The exact numeric values may change during implementation, but the persisted format must be kind-aware. A width class tag alone is not sufficient.

---

## Shared Primitive Ordered Encoders

Primary-key encoding and secondary-index component encoding must share primitive payload encoders for overlapping logical types, or prove identical logical ordering with tests.

This applies at least to:

* `Account`
* `Int`
* `Principal`
* `Subaccount`
* `Timestamp`
* `Nat`
* `Ulid`
* `Unit`

The target implementation should have one primitive ordered encoding authority, such as:

```rust
PrimitiveOrderedEncoder<T>
```

or an equivalent module-level API consumed by both:

```text
EncodedPrimaryKey payload
EncodedIndexComponent payload
```

This avoids two independently correct-looking encoders diverging for the same logical primitive.

---

## Key Classes

```text
Key8:
  kind tag + 8-byte canonical payload

Key16:
  kind tag + 16-byte canonical payload

Key32:
  kind tag + canonical payload up to 32 bytes

Key64:
  kind tag + canonical payload up to 64 bytes
```

For fixed-width payloads, the payload length is implied by the class.

`Unit` is a kind-only singleton and has no payload.

For `Principal`, the encoding must preserve exact logical ordering and validity. Principal ordering is length-first and then bytes, matching the current logical `Ord` implementation unless IcyDB deliberately changes principal ordering everywhere.

The recommended `Principal` payload is:

```text
len + principal bytes
```

This fits the `Key32` class because IcyDB principals max at 29 bytes, so `len + bytes` is at most 30 bytes. A padded representation may be used only if it preserves the same length-first logical ordering.

The required invariant is:

```text
logical_compare(a, b) == encoded_bytes(a).cmp(encoded_bytes(b))
```

The same invariant must hold for secondary index `Principal` components. Primary and secondary principal ordering must not diverge.

For `Timestamp`, raw signed big-endian encoding is invalid. Timestamp payloads must use a signed-order-preserving biased `i64` transform:

```text
biased = timestamp_millis ^ i64::MIN
payload = biased.to_be_bytes()
```

or an equivalent transform where bytewise lexicographic ordering matches signed `i64` ordering.

# Data Store Key Design

Current:

```text
RawDataKey =
  entity_tag(8)
  + storage_key(64)
```

Proposed:

```text
RawDataStoreKey =
  entity_tag(8)
  + encoded_primary_key(variable bounded)
```

Important requirements:

* entity tag remains first
* entity-local scans remain prefix/range scans
* primary-key range scans must use the same canonical key encoder
* full entity scans use prefix range bounds, not fake primary-key sentinels
* malformed raw keys must be rejected at decode boundaries
* ordering must remain stable across upgrades after the hard cut

This is the highest-risk part of the design because the old `RawDataKey` was fixed-size. The correctness hinge is range-bound construction for variable-size `RawDataStoreKey`.

---

# Secondary Index Key Design

Current secondary index keys already use variable-length stable-memory data. The redesign should retain that general envelope while replacing the 64-byte primary-key suffix.

Proposed:

```text
RawIndexStoreKey =
  key_kind
  + index_id
  + component_count
  + encoded_index_component[]
  + encoded_primary_key_suffix
```

The primary-key suffix should use `EncodedPrimaryKey`, not the old 64-byte frame.

The suffix remains necessary because it:

* makes index entries unique
* provides deterministic tiebreaker ordering
* identifies the referenced row
* supports duplicate index component values

---

# Secondary Index Value Design

Current design duplicates primary-key membership in the value. The audit found that each indexed row currently pays both:

```text
RawIndexKey suffix:
  len + 64-byte primary key

RawIndexEntry value:
  count + 64-byte primary key + witness
```

for roughly 135 bytes of primary-key-related storage per secondary index row.

Proposed rule:

```text
Index row identity lives in RawIndexStoreKey, not IndexEntryValue.
```

Therefore, `IndexEntryValue` should become a minimal witness/version payload.

The implementation should decode the referenced primary key from the `RawIndexStoreKey` suffix.

This is a separate but complementary hard-cut improvement. It may produce the largest immediate storage win.

The target is not zero bytes at all costs. The target is no second copy of row identity. `IndexEntryValue` may retain:

* one witness byte
* a format/version byte
* a stale-entry repair marker
* other small recovery metadata required by existing invariants

It must not duplicate primary-key membership.

---

# Ordering Requirements

For every admitted primary-key type:

```text
logical_compare(a, b) == encoded_primary_key(a).cmp(encoded_primary_key(b))
```

For every admitted secondary-index component type:

```text
logical_compare(a, b) == encoded_index_component(a).cmp(encoded_index_component(b))
```

Required ordering coverage:

* signed integers
* unsigned integers
* timestamp
* ULID
* principal
* subaccount
* account
* composite index fields
* primary-key suffix tiebreakers
* ascending and descending secondary-index key contracts

The audit specifically warns that current timestamp bytes may use signed big-endian encoding, which is not lexicographically correct for negative timestamps. 0.159 must fix this. `Timestamp` is an `i64` domain and negative timestamps are valid, so timestamp encoding must use signed-order-preserving biased bytes. Raw signed big-endian timestamp payloads are forbidden.

For every primitive admitted in both primary keys and secondary index components, tests must also prove:

```text
primary_payload_cmp(a, b) == secondary_component_payload_cmp(a, b)
```

for the same logical type.

---

# Descending Secondary-Index Ordering

`CREATE INDEX ... (field DESC)` should move out of the 0.157 DDL line and into
0.159 because it is a physical key-ordering feature, not just SQL syntax.

0.157 may parse `DESC` and reject it with an explicit unsupported diagnostic.
0.159 must decide and implement the storage contract:

```text
IndexStoreKey =
  IndexId
  + direction-aware EncodedIndexComponent[]
  + EncodedPrimaryKey
```

The direction contract must be catalog-native. Accepted index metadata should
record each key component's direction explicitly, and generated and DDL-created
indexes must use the same representation.

Descending support must specify:

* how descending component bytes are encoded or inverted
* whether the primary-key suffix remains ascending as a deterministic
  tiebreaker or is also direction-adjusted for full tuple reversal
* how mixed `ASC` / `DESC` composite indexes compare lexicographically
* how range bounds are derived for descending and mixed-direction components
* how planner route eligibility distinguishes ascending, descending, and mixed
  direction indexes
* how cursors encode direction-aware raw index boundaries

The implementation must not fake `DESC` by scanning an ascending index and
reversing results unless that is an explicitly documented fallback outside the
physical index contract. The durable accepted schema contract should describe
the physical index order that was actually built.

# Range Bound Design

This must be designed before broad implementation.

## Entity full scan

For an entity tag `E`, entity-local scan bounds need to include all keys whose prefix begins with `E`.

Use prefix bounds, not fake minimum/maximum primary-key sentinels:

```text
lower = Included(entity_tag_bytes(E))
upper = Excluded(entity_tag_bytes(E + 1))
```

When `E == u64::MAX`, the upper bound is `Unbounded`.

These bounds are range-only raw byte boundaries. They are not persisted row keys and must not be decoded as `DataStoreKey` values.

## Primary-key range scan

For:

```text
E where primary_key >= A and primary_key < B
```

bounds should be:

```text
lower = entity_tag(E) + encode_primary_key(A)
upper = entity_tag(E) + encode_primary_key(B)
```

Correctness depends on encoded primary-key ordering.

## Class-aware range scans

If a single entity can only have one primary-key type, then all primary-key comparisons inside that entity occur within one known key class.

That simplifies ordering.

If heterogeneous primary-key kinds can appear inside the same entity namespace, kind tags must be ordered deliberately and documented. Prefer avoiding heterogeneous primary-key kinds per entity.

# Malformed Key Handling

Decode boundaries must reject:

* unknown key kind tags
* invalid class length
* invalid principal length
* invalid account encoding
* invalid subaccount length
* trailing bytes
* non-canonical padded forms
* non-canonical integer encodings
* invalid data-store key prefixes
* invalid index-store key envelopes
* malformed primary-key suffixes

Malformed persisted keys should not silently compare as valid logical keys.

# Commit Marker and Recovery Impact

The audit identifies fixed-size assumptions in commit marker capacity and decode logic.

0.159 must update commit/recovery encoding to accept compact variable-sized data-store keys.

Required policy:

```text
Commit markers store RawDataStoreKey bytes using length-delimited encoding.
```

or another explicit bounded variable-size representation.

Hard-cut means old marker compatibility is not required.

# Risks

## 1. Variable RawDataStoreKey range bounds

This is the highest correctness risk.

The existing fixed 72-byte data key made lower/upper range construction straightforward. Variable bounded keys require precise prefix-bound design for full entity scans.

Mitigation:

* design prefix range-bound helpers before changing call sites
* centralize all data-store range construction
* property-test scan inclusion/exclusion boundaries

---

## 2. Ordering bugs

Compact encodings must preserve logical ordering exactly.

Mitigation:

* property tests per type
* explicit signed integer normalization
* signed-order-preserving timestamp normalization
* shared primitive encoder tests for primary/index overlap

---

## 3. Index ownership transition

Removing primary-key membership from the value changes where row identity is recovered.

Mitigation:

* create a single `RawIndexStoreKey::primary_suffix()` decoder
* forbid ad hoc suffix parsing
* update delete/rebuild paths carefully

---

## 4. Cursor compatibility

Hard cut removes old compatibility concerns, but internal cursor correctness still matters.

Mitigation:

* regenerate cursor format around raw store keys
* test continuation heavily

---

## 5. Naming churn

Renaming key types may produce a large diff.

Mitigation:

* separate mechanical rename from semantic changes where possible
* do not mix with unrelated refactors

---

# Open Questions

1. Should `Unit` remain a primary-key type?
2. Should `Account` remain primary-key admissible, or only index-component admissible?
3. Should `RawDataStoreKey` expose raw byte ordering only, or decode eagerly for validation?
4. Should heterogeneous primary-key kinds be impossible within a single entity primary-key namespace?

# Final Verdict

0.159 should hard-cut the key model.

The current universal 64-byte `StorageKey` is correct but over-normalized. It pays a permanent storage and traversal cost across the data store and every secondary index. The audit shows that the strongest justification for fixed-width persisted keys does not fully hold, because secondary index keys are already variable-length stable-memory data.

The best future design is:

```text
kind-tagged compact canonical primary keys
+
shared primitive ordered encoders
+
prefix-bounded entity scans
+
key-owned secondary index entries with minimal witness values
+
clear key taxonomy
```

This keeps the correctness properties IcyDB needs while improving:

* storage density
* secondary index footprint
* BTree fanout
* cache locality
* conceptual clarity
* long-term maintainability

The hard cut is justified now because this storage protocol will become much more expensive to change after compatibility freezes.
