# 0.199 Read-Admission Owner Split

Status: structural cleanup with focused validation. This slice preserves public
read admission, SQL/fluent execution behavior, EXPLAIN text/JSON facts,
diagnostic codes, cursor formats, persisted formats, and public API shape.

## Question

Can the read-admission module stop mixing public vocabulary, policy/budget
evaluation, planner-summary extraction, verbose EXPLAIN rendering, and tests in
one large production file?

## Evidence

Before this slice, `crates/icydb-core/src/db/query/admission.rs` was a
2,411-line module that owned:

- public read-admission vocabulary and stable rejection diagnostics;
- default bounded-read constants;
- grouped/public/admin/diagnostic policy construction;
- policy evaluation and fail-closed rejection ordering;
- planner access-path projection into admission summaries;
- grouped/limit/offset/materialization admission summary extraction;
- verbose EXPLAIN text rendering;
- admission-focused unit tests.

Those responsibilities are now split while preserving the same exported
admission surface:

- `admission/policy.rs` owns bounded-read constants, grouped budgets, public
  and diagnostic policies, and policy evaluation.
- `admission/plan_summary.rs` owns planner-to-admission projection for access
  kind, selected index, exact-key bounds, grouped facts, residual filters,
  ordering, and plan shape.
- `admission/render.rs` owns stable verbose EXPLAIN text rendering for
  admission summaries.
- `admission/tests.rs` owns admission-focused unit tests.
- `admission.rs` remains the public vocabulary/coordinator for lanes, bound
  kinds, decisions, access classes, grouped/materialization facts, stable
  rejection diagnostics, and summary accessors.

## Non-Goals

- No read-admission behavior change.
- No rejection-code or diagnostic-code change.
- No SQL/fluent planning, execution, or cache behavior change.
- No EXPLAIN field-name or payload shape change.
- No cursor, persisted-format, storage, or public API change.

## Complexity Delta

- Public API complexity: neutral.
- Runtime behavior: neutral.
- Review complexity: lower. Policy evaluation, planner projection, and text
  rendering now have separate owner files, and the production vocabulary file
  no longer carries the full test body inline.
- File shape:
  - `admission.rs`: 2,411 lines before, 743-line vocabulary/coordinator
    after.
  - `admission/policy.rs`: 587 lines.
  - `admission/plan_summary.rs`: 311 lines.
  - `admission/render.rs`: 147 lines.
  - `admission/tests.rs`: 685 lines.

The total line count grows modestly because moved responsibilities now have
module headers and local imports.

## Validation

- `cargo check -p icydb-core --all-features`
- `cargo test -p icydb-core --lib --all-features admission -- --nocapture`
- `cargo clippy -p icydb-core --all-features --all-targets -- -D warnings`
- `cargo check -p icydb --no-default-features --features sql`
- `cargo fmt --all --check`
- `git diff --check`

## Residual Risk

Low. This is a module ownership split. The remaining risk is visibility or lint
drift around policy re-exports and test-only grouped policy helpers, covered by
focused admission tests, all-feature core clippy, and the facade SQL feature
check.
