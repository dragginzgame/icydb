# 0.199.0 Direct Query Boundary Trace

Status: 0.199.0 audit and cleanup artifact.

Verdict: direct `Query<E>` execution remains necessary for generated, SQL,
diagnostic, and performance harness code, but it should not appear as a normal
public endpoint recipe. The 0.199.0 cleanup hides fluent `.query()` extraction
methods from public docs, keeps direct execution doc-hidden, and updates the
public facade reference to teach semantic terminals first. The 0.199.1 cleanup
adds hidden fluent attribution terminals so the SQL perf audit canister no
longer needs raw `Query<E>` extraction for fluent scenario measurement.

## What Passed

- Direct execution still runs default read admission through
  `ensure_default_query_read_admission(query)` before execution.
- Public docs now teach `load::<E>()` semantic terminals instead of direct
  `Query<E>` construction.
- The `Query<E>` re-export and direct execution methods are doc-hidden.
- Existing perf harness fluent call sites now use hidden fluent attribution
  terminals instead of raw `Query<E>` extraction.

## Finding

### 199-DQ-001: Direct Query Is Internal, With Fluent Attribution On The Fluent Boundary

Category: public API hygiene / generated boundary

Severity: Low after cleanup

Priority: P3 after 0.199.1 cleanup

Confidence: High

Evidence:

- `crates/icydb/src/db/query/mod.rs` re-exports `Query` as `#[doc(hidden)]`.
- `crates/icydb/src/db/session/mod.rs` marks `execute_query(...)`,
  `trace_query(...)`, and diagnostics attribution direct-query helpers as
  hidden.
- Before 0.199.1, `canisters/audit/sql_perf/src/lib.rs` intentionally used
  `.query()` and direct attribution helpers for performance measurement.
- After 0.199.1, fluent perf scenarios use hidden fluent
  `execute_with_attribution()` terminals.
- Before this cleanup, public/core fluent `.query()` methods were not hidden.

Recommended policy:

- Keep direct query surfaces callable but doc-hidden until/unless generated
  infrastructure can avoid them entirely.
- Do not add compatibility aliases or a public policy-builder layer around
  direct queries.
- Do not teach direct query construction in README, public facade docs, or
  read-intent docs.
- If a future generated path needs direct query execution, keep that path
  generated/internal and covered by focused compile checks.
- Prefer hidden fluent attribution terminals for fluent perf/diagnostic code
  when the caller starts from a fluent wrapper.
