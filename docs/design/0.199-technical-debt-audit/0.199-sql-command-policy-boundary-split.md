# 0.199 SQL Command Policy Boundary Split

Status: structural cleanup with focused validation. This slice preserves SQL
UPDATE/DELETE policy behavior, generated/public write admission, returning
bounds, SQL execution, cache behavior, cursor formats, persisted formats, and
public API shape.

## Question

Can command-specific SQL UPDATE and DELETE policy owners stop mixing public DTO
exports, policy model/proof state, parser-shape planning, and owner-local tests
in one broad file per command?

## Evidence

Before this slice:

- `crates/icydb-core/src/db/session/sql/update_policy.rs` was about 1,205
  lines and owned UPDATE public DTO exports, assignment ownership checks,
  context/bounds model, policy rejection mapping, validated-plan construction,
  parser-shape classification, and tests.
- `crates/icydb-core/src/db/session/sql/delete_policy.rs` was about 952 lines
  and owned DELETE public DTO exports, context/bounds model, policy rejection
  mapping, validated-plan construction, parser-shape classification, and tests.
- The shared SQL write policy split had already separated common
  `model`/`bounds`/`shape` ownership, but the command-specific UPDATE/DELETE
  policy files still kept their local model and planning responsibilities
  together.

After this slice, each command policy has the same owner shape:

- `update_policy.rs` and `delete_policy.rs` are narrow module indexes that
  preserve the previous exported surface.
- `update_policy/model.rs` and `delete_policy/model.rs` own public/hidden DTOs,
  contexts, proof-carrying plan wrappers, policy reports, and rejection enums.
- `update_policy/planning.rs` and `delete_policy/planning.rs` own SQL parsing,
  command-shape classification, policy rejection selection, and validated-plan
  construction.
- `update_policy/tests.rs` and `delete_policy/tests.rs` keep the existing
  owner-local policy tests beside the command policy owner without inflating the
  production parent files.
- The structural guard now checks the split parent/model/planning ownership for
  both UPDATE and DELETE.

## Non-Goals

- No SQL UPDATE or DELETE policy behavior change.
- No generated/public write admission change.
- No returning projection, SQL write execution, cache, cursor, storage, or
  persisted-format change.
- No public DTO rename, compatibility shim, or API hard cut.
- No merge of UPDATE and DELETE into a generic command-policy abstraction.

## Complexity Delta

- Public API complexity: neutral.
- Runtime complexity: neutral.
- Review complexity: lower. Command-specific DTO/proof state and
  command-specific planning/admission logic can now be reviewed independently.
- File shape:
  - `delete_policy.rs`: about 952 lines before, 21-line module index after.
  - `delete_policy/model.rs`: 327 lines.
  - `delete_policy/planning.rs`: 180 lines.
  - `delete_policy/tests.rs`: 451 lines.
  - `update_policy.rs`: about 1,205 lines before, 21-line module index after.
  - `update_policy/model.rs`: 371 lines.
  - `update_policy/planning.rs`: 241 lines.
  - `update_policy/tests.rs`: 599 lines.

The total line count is approximately neutral because tests and module headers
move with their owners.

## Validation

- `cargo test -p icydb-core --lib --all-features sql_write -- --nocapture`

## Residual Risk

Low. This is a module ownership split. Remaining risk is all-target visibility
drift, covered by core check/clippy before release. UPDATE and DELETE behavior
is still covered through the existing focused SQL write/delete/update test
filters.
