# 0.199 Session Catalog/Write Facade Split

Status: structural cleanup with focused validation. This slice preserves public
`DbSession` catalog, storage-report, typed write, write-returning, and
structural mutation method names. It does not change public read intent,
read admission, SQL lowering, generated SQL authority, cursor tokens, persisted
formats, or executor strategy.

## Question

Can the remaining catalog/show/storage-report and high-level write helper
ownership move out of the central session module without changing public facade
behavior?

## Evidence

After the SQL facade split, `crates/icydb/src/db/session/mod.rs` still mixed:

- session construction and fluent load/delete entrypoints;
- catalog/show/schema-description methods;
- storage-report facade methods;
- write-returning projection conversion;
- typed insert/create/replace/update helpers;
- batch write helpers;
- public structural mutation wrapper types and methods;
- hidden query trace/attribution helpers.

The catalog methods are pass-through accepted-schema/catalog facade wrappers.
The write helpers own public row-returning projection conversion and structural
patch facade types above the core save pipeline. Both groups have clear owner
boundaries and do not need to live beside read/query session setup.

## Change

Move catalog and storage-report facade ownership into
`crates/icydb/src/db/session/catalog.rs`:

- `show_indexes`;
- `show_columns`;
- `show_entities`;
- `try_show_entities`;
- `show_stores`;
- `show_memory`;
- `describe_entity`;
- `try_describe_entity`;
- `storage_report`.

Move public write facade ownership into `crates/icydb/src/db/session/write.rs`:

- `MutationMode`;
- `StructuralPatch`;
- write-returning projection selection and row output conversion;
- `insert`, `insert_returning_all`, and `insert_returning`;
- `create`, `create_returning_all`, and `create_returning`;
- insert/replace/update batch helpers;
- `replace`;
- `update`, `update_returning_all`, and `update_returning`;
- `mutate_structural`;
- `structural_patch`.

`db::session` re-exports `MutationMode` and `StructuralPatch` so existing public
paths remain stable. Delete returning continues to use the shared
`row_projection_output_from_entities` helper instead of introducing a second
projection implementation.

## Result

`crates/icydb/src/db/session/mod.rs` drops from about 650 lines to about 160
lines. The central session module now mainly owns session construction,
configuration, read/delete entrypoints, and hidden query tracing/attribution
handoffs. Catalog/show/storage and write-returning/structural mutation behavior
now have explicit owner files.

## Validation

- `cargo fmt --all`
- `cargo check -p icydb --all-features`
- `cargo test -p icydb --lib --all-features`
- `cargo clippy -p icydb --all-features --all-targets -- -D warnings`

## Non-Goals

- No public API changes.
- No read-intent change.
- No read-admission change.
- No SQL lowering or generated SQL authority change.
- No write semantic change.
- No row projection payload shape change.
- No cursor or persisted-format change.
- No executor strategy change.
