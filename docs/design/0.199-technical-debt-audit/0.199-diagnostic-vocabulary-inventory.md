# 0.199 Diagnostic Vocabulary Inventory

Status: code-reading-backed 0.199.0 audit artifact. This is an inventory only;
it does not change diagnostic codes, rendered messages, public APIs, cursor
formats, persisted formats, or admission behavior.

## Question

Where is read-admission diagnostic vocabulary owned, and which parts are stable
machine-readable facts versus human guidance?

## Ownership Map

| Concern | Owner | Evidence | Stability |
| --- | --- | --- | --- |
| Broad compact diagnostic class | `DiagnosticCode::QueryReadAdmission` | `crates/icydb-diagnostic-code/src/lib.rs` | Stable public code family. |
| Public read-admission reason enum | `QueryReadAdmissionCode` | `crates/icydb-diagnostic-code/src/lib.rs` | Wire-order significant. |
| Numeric registry entries | `QUERY_READ_*` codes `188..204` | `crates/icydb-diagnostic-code/src/registry.rs` | Stable compact public error codes. |
| Internal admission rejection enum | `QueryAdmissionRejection` | `crates/icydb-core/src/db/query/admission.rs` | Internal source of decision reasons. |
| Internal-to-public mapping | `QueryAdmissionRejection::code()` | `crates/icydb-core/src/db/query/admission.rs` | Must stay one-to-one with public enum. |
| Structured error detail | `QueryErrorDetail::QueryReadAdmission { reason }` | `crates/icydb-core/src/error/mod.rs` | Structured detail; not rich prose. |
| Human guidance | `READ_ADMISSION.md` / `read-intent.md` | `docs/contracts/READ_ADMISSION.md`, `docs/guides/read-intent.md` | Explanatory, should not be exact-phrase authority. |
| Guard script | `check-read-admission-invariants.sh` | `scripts/ci/check-read-admission-invariants.sh` | Should enforce codes, anchors, concepts, not prose. |

## Reason Families

| Family | Public reasons | Developer guidance |
| --- | --- | --- |
| Missing endpoint intent | `PublicQueryRequiresLimit` | Choose `page`, `next_page`, `collect_complete`, exact aggregate, exact-key access, grouped limits, or deliberate `partial_window`. |
| Missing route proof | `PublicQueryRequiresIndex`, `UnboundedFullScanRejected`, `ScanBoundUnavailable`, `EstimatedOnlyBoundRejected` | Add/choose a proven access path or move broad work to trusted/admin code. |
| Route or materialization unsafe | `SortRequiresMaterialization`, `MaterializationExceedsBudget` | Align order with the selected route, add a suitable index, or keep broad reports trusted/admin-only. |
| Policy budget exceeded | `ScanBoundExceedsPolicy`, `ProjectionResponseMayExceedLimit`, `ReturnedRowBoundExceedsPolicy`, `PrimaryKeyInputExceedsPolicy` | Narrow the predicate/page/key input/response shape or use trusted/admin work. |
| Grouped budgets missing/exceeded | `GroupedQueryRequiresLimits`, `GroupedQueryExceedsBudget` | Add grouped query-owned limits and distinct-entry budgets where needed. |
| Lane mismatch | `DiagnosticLaneDoesNotExecute`, `IntrospectionDisabledForLane`, `UnsupportedStatementForQueryLane` | Keep diagnostics and SQL introspection out of arbitrary public row execution. |
| Offset pagination | `PublicQueryOffsetRejected` | Use cursor paging through `page(limit)` / `next_page(limit, cursor)`. |

## Guardrail

The stable source of truth is the structured detail code, not rendered prose.
Tests and CI should prefer:

- enum parity;
- registry entries;
- section anchors;
- API tokens;
- broad concepts;
- forbidden public-policy or legacy-surface terms.

They should avoid exact explanatory sentences unless the exact string is itself
a public API contract.

## Remaining Risk

Read-admission vocabulary still appears in docs, tests, EXPLAIN, and CLI-like
rendering paths. The next cleanup should only centralize wording if it can do so
without increasing wasm size or turning compact diagnostic codes into rich
runtime prose.
