# 0.199 Terminal Module Split

Status: structural cleanup with focused validation. This slice preserves public
load terminal APIs, read-admission behavior, diagnostic DTOs, cursor tokens,
persisted formats, generated SQL authority, and executor strategy.

## Question

Can the large fluent load `terminals.rs` file be split without changing the
terminal ownership model?

## Evidence

Before this slice, `crates/icydb-core/src/db/query/fluent/load/terminals.rs`
contained:

- public terminal methods;
- terminal descriptor driver implementations;
- runtime-to-public output conversion helpers;
- shared readiness/admission/read-intent support helpers.

That made the public terminal list harder to scan and mixed API behavior with
private wiring.

## Change

Split private implementation support into:

- `terminals/driver.rs` for descriptor-to-session driver implementations;
- `terminals/output.rs` for value output conversion helpers;
- `terminals/read_intent.rs` for public existence and complete-small-set
  semantic terminals;
- `terminals/support.rs` for shared terminal readiness, admission, slot, exact
  aggregate, existence, count attribution, and complete-small-set handoffs.

The remaining `terminals.rs` keeps the public terminal methods and their
direct terminal descriptors that are still large enough to deserve their own
future aggregate/projection split.

## Result

`terminals.rs` is reduced from about 1,580 lines to under 950 lines. The
terminal module is easier to scan without introducing a terminal enum, broad
dispatch object, compatibility layer, or execution-route rewrite.

## Validation

- `cargo check -p icydb-core --all-features`
- `cargo test -p icydb-core --all-features read_admission`
- `cargo test -p icydb-core --all-features aggregate_terminals`
- `cargo test -p icydb-core --all-features collect_complete`
- `cargo clippy -p icydb-core --all-features --all-targets -- -D warnings`
- `cargo fmt --all --check`
- `git diff --check`

## Non-Goals

- No public fluent API changes.
- No terminal rename.
- No read-admission change.
- No EXPLAIN or diagnostic DTO change.
- No query planner or executor strategy change.
- No cursor or persisted-format change.
