# 0.199.0 Exact-Key Proof Ownership Trace

Status: 0.199.0 audit artifact. This file records code-reading evidence for
the exact primary-key proof boundary after the 0.197 and 0.198 lines.

Verdict: exact-key behavior is not obviously unsafe. Admission and execution
consume the selected access path rather than reparsing the original query. The
first cleanup slice introduced `PrimaryKeyAccessProof` as a named planner-local
projection for selected primary-key access. The 0.199.2 cleanup keeps the proof
decision single-use inside the planner pipeline by returning one predicate-strip
result that also owns the "filter expression is redundant" fact. A broader
follow-up remains possible if admission/cache/explain should consume that same
named projection rather than their own selected-access projections.

## 0.199.2 Cleanup

The planner pipeline previously asked the selected primary-key proof the same
question twice for redundant exact-key predicates:

1. once to decide whether the scalar filter expression could be removed; and
2. once to decide whether the normalized predicate could be stripped.

0.199.2 collapses that into `PrimaryKeyPredicateStripResult`, a private
pipeline-local result carrying both:

- the remaining normalized predicate; and
- whether the primary-key predicate was stripped.

This does not change selected access, read admission, executor routing, cache
identity, SQL behavior, cursor format, or persisted format. It only removes one
duplicated proof decision inside the planner owner boundary.

## Confirmed Strengths

1. Typed key access and input-resource accounting are projected in one pass.
   Evidence: `KeyAccessPlanningProjection` in `query/intent/key_access.rs`
   keeps the normalized access path and raw input resource facts together.

2. Admission uses selected access, not original query strings or SQL fragments.
   Evidence: `QueryAdmissionSummary::from_plan(...)` calls
   `summarize_access_plan(plan)` and derives exact row bounds through
   `AdmissionAccessProjection`.

3. Executor routing consumes access payload shape.
   Evidence: `resolve_physical_key_stream(...)` dispatches `ByKey` to
   `runtime.resolve_by_key(...)` and `ByKeys` to `runtime.resolve_by_keys(...)`.

4. Cache identity projects from the selected access path.
   Evidence: `AccessPathCacheKey::from_access_path(...)` maps `ByKey` and
   `ByKeys` from the access path, not from rendered diagnostics.

5. Tests cover the high-risk behavior:
   - public exact-key filter admission without limits;
   - exact-key filters with residual predicates;
   - invalid residual fields for existing and missing primary-key values;
   - primary-key `IN` deduped row bounds;
   - raw primary-key input term and payload caps;
   - one-key `by_ids(...)` matching `by_id(...)`;
   - redundant primary-key predicate stripping;
   - planner and typed-key canonicalization agreement for one-key sets.

## Main Finding

### 199-EK-001: Exact-Key Proof Consumers Are Still Coordinated Projections

Category: proof ownership / maintainability

Severity: Medium

Priority: P2 after first cleanup

Confidence: High

Evidence:

- `KeyAccessPlanningProjection` exists for typed key access in
  `query/intent/key_access.rs`.
- `PrimaryKeyAccessProof` now exists for planner-local selected primary-key
  access proof in `query/plan/primary_key_access_proof.rs`.
- Predicate-derived raw input facts are collected separately by
  `primary_key_input_resource_from_predicate(...)` in `query/plan/pipeline.rs`.
- Predicate redundancy is detected by `PrimaryKeyAccessProof`.
- Admission creates a separate `AdmissionAccessProjection` in
  `query/admission.rs`.
- Cache identity creates a separate `AccessPathCacheKey` projection in
  `query/intent/cache_key.rs`.
- EXPLAIN/access choice stores separate selected reasons such as
  `IntentKeyAccessOverride`, `PlannerPrimaryKeyLookup`, `PlannerKeySetAccess`,
  `ByKeyAccess`, and `ByKeysAccess`.

What passed:

- The first 0.199 cleanup removed the local `ExactPrimaryKeyAccess` helper from
  `pipeline.rs` and replaced it with a named planner-private
  `PrimaryKeyAccessProof` module.
- 0.199.2 removed the duplicate pipeline-local proof check by making redundant
  predicate stripping return the predicate plus the filter-expression stripping
  decision together.
- Admission/execution are selected-access driven.
- Invalid residuals are validated before admission can turn a missing key into
  a not-found result.
- Raw input caps are tested for typed `by_ids(...)` and primary-key `IN`.
- One-key typed and planner canonicalization have focused tests.

What is missing:

- There is still no cross-consumer object equivalent to
  `ExactPrimaryKeyProof { access, origin, raw_input_resource, residual_status }`
  that typed-key, predicate, admission, explain, and cache all consume.
- `PrimaryKeyAccessProof` includes half-open primary-key ranges for existing
  predicate stripping, even though the 0.197 exact-key slice was intentionally
  scalar/finite-set focused. This is not necessarily a bug, but it should stay
  clearly documented as selected primary-key access proof rather than a 0.197
  public exact-key API promise.
- Reason-code vocabulary distinguishes many origins and projections, but the
  docs/tests do not identify one owner boundary for "this query is exact key
  bounded."

Recommended fix:

Do not rewrite the planner. The first patch has already added the
planner-local projection. Only continue if a follow-up audit proves that
admission/cache/explain drift is still costly.

Possible follow-up patch:

```text
Thread the private selected primary-key proof projection into exactly one more
consumer boundary, if evidence shows that consumer is drifting. Prefer admission
first only if this can reduce duplicated row-bound projection without moving
admission policy into planning.

Do not change public API, read admission policy, executor routing, cursor
format, persisted format, SQL behavior, or selected access semantics.
```

Acceptance criteria:

- `filter(pk = value)` and `by_id(value)` still select `ByKey`.
- `filter(pk IN (...))` and `by_ids(...)` still select `ByKeys` or canonical
  one-key `ByKey`.
- Raw input term/payload caps remain preserved for duplicate-heavy `IN` and
  `by_ids(...)` inputs.
- Invalid residual fields still fail before execution, for both existing and
  missing primary-key values.
- Existing exact-key admission tests pass.
- Existing cache-key same-shape/different-parameter tests pass.
- No public read intent, SQL, cursor, or persisted format changes.
