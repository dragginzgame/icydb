# 0.199 Existence Terminal Boundary Cleanup

Status: small code cleanup with focused validation. This slice preserves public
API names, read-admission behavior, diagnostic DTOs, cursor formats, persisted
formats, and existence semantics.

## Question

Did 0.198's semantic existence helpers leave repeated terminal wiring that can
drift from the read-intent/admission contract?

## Evidence

`crates/icydb-core/src/db/query/fluent/load/terminals.rs` already routes
aggregate-style terminals through concrete terminal descriptors and
`TerminalStrategyDriver`.

The repeated part was the semantic existence terminal boundary:

- `exists()` rejected prior row-window caps and then executed the existence
  terminal;
- `exists_with_attribution()` repeated the same guard and attached
  `ReadIntentKind::ExistenceCheck`;
- `explain_exists()` repeated the same guard and attached
  `ReadIntentKind::ExistenceCheck` to the explain plan.

`not_exists()` and `explain_not_exists()` intentionally remain aliases over the
same existence terminal behavior with negated result semantics.

## Change

Added private owner-local helpers:

- `execute_existence_terminal(...)`
- `execute_existence_terminal_with_attribution(...)`
- `explain_existence_terminal(...)`

These helpers own the prior-row-window rejection and existence read-intent
metadata handoff once. Public methods now delegate to the helpers.

## Result

This is a narrow DRY cleanup. It makes existence execution, attribution, and
EXPLAIN wiring less fragile without changing terminal descriptors, route
selection, public API, or result semantics.

## Non-Goals

- No public existence API rename.
- No admission-policy change.
- No existence execution strategy change.
- No cursor-token or persisted-format change.
- No broad terminal dispatch enum.
