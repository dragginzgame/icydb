# 0.199 Read-Admission Invariant Hardening

Status: small 0.199.0 cleanup slice. This changes only the CI invariant script
for read-admission documentation checks. It does not change admission behavior,
diagnostic codes, public APIs, cursor formats, persisted formats, or generated
SQL behavior.

## Question

Can the read-admission invariant check keep the useful guards without failing
on ordinary documentation phrasing edits?

## Evidence

`scripts/ci/check-read-admission-invariants.sh` already contains high-value
checks:

- required read-admission and read-intent document section anchors;
- required public API tokens and links;
- `QueryReadAdmissionCode` coverage in `READ_ADMISSION.md`;
- internal `QueryAdmissionRejection` and public `QueryReadAdmissionCode` enum
  parity;
- default admission budget constants;
- forbidden public custom-policy facade terms;
- forbidden high raw `.limit(...)` examples outside the explicit contract.

The weaker part was guide-level concept matching that encoded explanatory
sentence shape. That made docs edits fragile without improving API or admission
safety.

## Change

The read-intent guide concept checks now use broader concepts:

- `partial_window(...)` must still be framed as partial row-window intent;
- the guide must still include the low-level partial execution token;
- migration guidance must still warn against mechanical rewrites;
- generated SQL boundary guidance must still name public endpoint risk and
  `icydb_query`.

The script still verifies section anchors, API tokens, diagnostic enum parity,
read-admission budgets, and forbidden public policy surfaces.

## Result

This keeps the invariant command useful as a semantic guard while reducing
phrase-level brittleness. Future 0.199 diagnostic-vocabulary cleanup should
prefer structured codes, anchors, and broad concepts over exact prose.
