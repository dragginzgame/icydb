# 0.199 Fluent Attribution Boundary

Status: 0.199.1 cleanup slice, extended by 0.199.3 terminal-attribution
cleanup. This changes hidden diagnostics/testing surfaces and the SQL perf
audit canister. It does not change public read behavior, public read admission,
cursor formats, persisted formats, generated SQL authority, or normal endpoint
APIs.

## Question

Can fluent perf attribution stop depending on raw `Query<E>` extraction from
normal fluent wrappers?

## Evidence

Before this slice, the SQL perf audit canister measured fluent scenarios by:

1. building a fluent query;
2. extracting the hidden `Query<E>` with `.query()`;
3. passing it to `DbSession::execute_query_result_with_attribution(...)`.

That path was correct but it kept the direct-query boundary visible in audit
code and made the public facade guide list direct-query methods for audit
completeness.

## Change

- Added hidden diagnostics-only `execute_with_attribution()` terminals to the
  facade and core fluent load wrappers.
- Updated the SQL perf audit canister to measure fluent scenarios through those
  hidden fluent terminals instead of `query()` plus direct session execution.
- Removed copyable direct-query examples from the public facade guide.

## Result

The direct-query surface remains available and hidden for generated/internal
tooling, but the active fluent perf path no longer needs raw query extraction.
This narrows the normal mental model without changing runtime behavior.

## 0.199.3 Count Attribution Follow-Up

`exists_with_attribution()`, `count_with_attribution()`, and
`count_exact_with_attribution()` now share one private read-intent attachment
helper:

- `with_fluent_terminal_read_intent(...)`

The count methods also share one private terminal adapter:

- `execute_count_terminal_with_attribution(...)`

The helper applies the caller-supplied `ReadIntentKind` to hidden fluent
terminal attribution results. The count adapter owns the session call to the
fluent count terminal. The public methods still differ where semantics differ:

- `exists_with_attribution()` rejects prior partial windows and reports
  `ExistenceCheck`;
- `count_with_attribution()` reports `BoundedRowWindow`;
- `count_exact_with_attribution()` first rejects prior partial windows and then
  reports `ExactAggregate`.

This removes duplicated hidden diagnostics wiring without changing existence,
count, exact-count, admission, or result semantics.
