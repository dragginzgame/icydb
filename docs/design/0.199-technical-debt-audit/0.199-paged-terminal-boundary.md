# 0.199 Paged Terminal Boundary

Status: 0.199 cleanup slice. This changes only private fluent pagination
wiring. It does not change public API names, read-admission policy, cursor
formats, persisted formats, generated SQL behavior, or pagination semantics.

## Question

Can public page and trusted admin batch terminals share the same paged-query
construction path without weakening their different admission and trust
requirements?

## Evidence

`page(limit)`, `next_page(limit, cursor)`, and `admin_batch(...)` all create a
`PagedLoadQuery` by:

- rejecting prior partial row-window caps;
- rejecting a pre-existing cursor on the fluent builder;
- applying an engine-owned load limit;
- optionally installing a continuation cursor; and
- validating paged-mode readiness.

The terminals differ in two places:

- public page terminals use the caller-supplied page limit and public read
  admission;
- admin batch uses the engine-owned `ADMIN_BATCH_ROWS` limit and requires the
  explicit trusted-read lane.

## Change

Added one private constructor:

- `into_paged_query(...)`

Added one private terminal contract:

- `PagedTerminal`

The helper owns the shared validation and query shaping. The terminal contract
owns:

- the raw-limit conflict diagnostic;
- the read-intent label;
- any terminal-specific validation.

That lets `admin_batch(...)` check the trusted lane after raw-limit and cursor
ownership errors, preserving the existing diagnostic order, without making
public page construction pass a no-op validation closure.

`PagedLoadQuery` also carries the terminal read intent selected by the creator:

- public page and continuation terminals set `PublicPage`;
- trusted admin batches set `TrustedAdminBatch`.

That keeps read-intent attribution on the paged wrapper boundary instead of
labeling every paged execution as public and then overriding admin batches
after execution.

The move made the plain `PagedLoadExecution::with_read_intent(...)` setter
dead, so the cleanup removes that internal method. Traced paged execution still
keeps its read-intent setter because `PagedLoadQuery` attaches read intent
before dropping trace metadata into the public response payload.

The focused test coverage now also pins `page(0)` normalization to one row and
`PublicPage` intent. This covers the lower-bound counterpart to the existing
oversized-page clamp test.

## Result

Paged terminal construction now has one owner-local handoff into
`PagedLoadQuery`. Public page and trusted admin batch semantics remain
separate, and `admin_batch(...)` reports trusted-batch intent directly from
the paged wrapper. The stale plain-execution relabeling path is gone.
