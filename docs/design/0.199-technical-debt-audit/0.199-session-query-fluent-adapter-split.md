# 0.199 Session Query Fluent Adapter Split

Status: structural cleanup with focused validation. This slice preserves fluent
terminal behavior, read admission, executor strategy, cache identity, cursor
formats, persisted formats, and public API shape.

## Question

Can the core session fluent adapter stop owning every scalar, projection, bytes,
row-window, and ranking terminal handoff in one module without introducing a
new dispatch layer or changing terminal behavior?

## Evidence

Before this slice, `crates/icydb-core/src/db/session/query/fluent.rs` owned:

- scalar count/existence terminal handoffs and attribution;
- scalar id terminal decoding;
- numeric field aggregate terminal handoffs;
- projection/value/count-distinct terminal decoding;
- bytes/take/top/bottom materialized terminal handoffs.

That made the module an accidental coordinator for several already-distinct
executor boundaries. The split keeps the same `DbSession` inherent methods and
visibility while moving those method families into owner modules:

- `fluent/scalar.rs` owns scalar aggregate, existence, id, and numeric field
  terminal session adapters.
- `fluent/projection.rs` owns scalar projection/value/count-distinct terminal
  session adapters.
- `fluent/materialized.rs` owns bytes, take, and top/bottom materialized
  terminal session adapters.

The parent `fluent.rs` is now a module index only.

## Non-Goals

- No public fluent API rename.
- No read-intent, read-admission, planner, cache, cursor, storage, SQL, or
  executor behavior change.
- No new terminal enum or dynamic dispatch layer.
- No movement of terminal construction from `db::query::fluent::load`.

## Complexity Delta

- Public API complexity: neutral.
- Runtime complexity: neutral.
- Review complexity: lower. Session fluent adapter ownership now matches the
  executor boundary being called.
- File shape:
  - `fluent.rs`: 764 lines before, 8 lines after.
  - `fluent/scalar.rs`: 424 lines.
  - `fluent/projection.rs`: 233 lines.
  - `fluent/materialized.rs`: 144 lines.

The total line count grows slightly from module headers/import separation, but
the behavioral owner boundaries are clearer.

## Validation

- `cargo fmt --all`
- `cargo check -p icydb-core --all-features`
- `cargo clippy -p icydb-core --all-features --all-targets -- -D warnings`
- `cargo test -p icydb-core --lib --all-features aggregate_terminals -- --nocapture`
- `cargo test -p icydb-core --lib --all-features read_admission -- --nocapture`

## Residual Risk

Low. This is a file ownership split. Remaining risk is import/privacy drift in
feature combinations, covered by the all-target core clippy pass and focused
terminal/read-admission tests.
