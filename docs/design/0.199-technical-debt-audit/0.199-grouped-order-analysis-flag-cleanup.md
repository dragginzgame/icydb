# 0.199 Grouped Order Analysis Flag Cleanup

Status: validation-blocker cleanup with focused validation. This slice
preserves SQL projection/grouped ordering behavior and only changes the
planner-local representation used while analyzing grouped `ORDER BY`
expressions.

## Question

Can grouped `ORDER BY` expression analysis keep one recursive owner without
carrying a wide boolean struct that fails current all-target clippy?

## Evidence

After the 0.199.20 and 0.199.21 grouped/scalar `ORDER BY CASE` work,
`GroupedOrderExprAnalysis` carried:

- canonical grouped-order shape;
- whether the expression references only grouped fields;
- whether the expression contains aggregate terms;
- whether the expression contains `CASE`;
- whether the expression contains non-aggregate wrapper functions.

The four booleans were semantically useful, but all-target clippy rejects that
shape with `struct_excessive_bools`. This cleanup keeps the same recursive
analysis owner and replaces those booleans with a small internal flag value plus
named predicate methods.

## Non-Goals

- No SQL `ORDER BY` behavior change.
- No grouped Top-K admission change.
- No CASE ordering semantic change.
- No public API, SQL surface, cache, cursor, storage, or persisted-format
  change.

## Complexity Delta

- Public API complexity: neutral.
- Runtime behavior: neutral.
- Review complexity: neutral to slightly lower. The analysis still has one
  owner, but related facts now live behind named flag methods instead of four
  public-looking fields.

## Validation

- `cargo clippy -p icydb-core --all-features --all-targets -- -D warnings`
- `cargo test -p icydb-core --lib --all-features sql_projection -- --nocapture`

## Residual Risk

Low. This is a planner-local representation cleanup. Existing SQL projection
and grouped/order tests exercise the affected admission and execution paths.
