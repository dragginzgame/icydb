# 0.191 Durability Productization And Format Policy

Status: closed through 0.191.1.

Theme: turn 0.189 and 0.190 reliability evidence into user-facing durability
contracts, format policy, backup/restore posture, checksum decisions, and
operator documentation.

## Entry Criteria

Start 0.191 only after 0.190 has produced:

- expanded recovery failpoint results;
- persisted-format corpus or fuzz results;
- structural decode safety results;
- large-index recovery characterization;
- evidence notes for backup/restore/import and checksums.

## Scope

0.191 owns:

- backup/restore/import threat model;
- checksum decision;
- persisted-format compatibility policy;
- operator durability guide;
- documented recovery and scale limits;
- streaming index fold/rebuild design if 0.190 proves it necessary.

## Non-Goals

- Do not reopen the 0.189 marker/journal proof unless evidence invalidates it.
- Do not require SQLite/POSIX file-database behavior.
- Do not add checksums without a format classification.
- Do not claim backup/import support without hostile-input and compatibility
  requirements.

## Work Track A: Backup, Restore, And Import Threat Model

Answer:

- Are stable-memory snapshots ever imported from outside the current canister?
- Is backup/restore an IcyDB-supported operation before 1.0.0?
- Are corrupted or malicious stable-memory images in scope?
- What does recovery guarantee for internally produced but corrupted state?
- What does recovery explicitly not repair?

Outcomes:

- if import/restore remains out of scope, document that explicitly;
- if import/restore is in scope, define corruption detection and compatibility
  requirements.

## Work Track B: Checksum Decision

Checksums should be decided after the threat model is clear.

Questions:

- should checksums cover marker/journal only?
- should they cover row/schema/index envelopes too?
- are checksums intended to detect accidental corruption, hostile imports, or
  both?
- is the change a pre-1.0 hard cut, backward-compatible extension, or migration?

Outcomes:

- explicit no-checksum decision with rationale; or
- checksum design with persisted-format classification.

## Work Track C: Operator Durability Guide

Required content:

- durable `journaled` versus volatile `heap` stores;
- single-message mutation atomicity;
- no `await` during IcyDB mutation execution;
- what happens if application code returns `Err` after a successful write;
- guarded recovery before reads/writes;
- startup recovery window and unsafe direct-store access;
- stable-memory memory ID partition expectations;
- what recovery can and cannot repair;
- supported backup/restore/import assumptions, or explicit out-of-scope status;
- thread/process/multi-registry support boundaries for host tests and non-IC
  environments;
- documented recovery-size or large-index limits from 0.190.

This doc must not imply POSIX file-database behavior.

## Acceptance Criteria

0.191 is complete when:

- backup/restore/import scope is explicit;
- checksum decision is documented and any format change is classified;
- operator durability guide reflects the tested IC-native contract;
- known recovery/index scale limits are documented;
- any required streaming index fold/rebuild design is tracked separately.

## Slice 0.191.0 Baseline

The first slice promotes the 0.191 work from evidence notes into a normative
operator contract without changing runtime behavior or persisted bytes.

Deliverables:

- add `docs/contracts/DURABILITY.md`;
- add `docs/contracts/PERSISTED_FORMAT_POLICY.md`;
- add `docs/operations/DURABILITY_GUIDE.md`;
- link it from the foundations, roadmap, atomicity, transaction, and SQL
  storage-mode contracts;
- explicitly state that raw stable-memory backup/restore/import is not yet a
  supported product surface;
- make the current no-checksum decision explicit and classify it as no
  persisted-format change;
- make future persisted-format change classification explicit for marker,
  journal, row, schema, index, cursor, and structural-value bytes;
- record a 0.191 decision ledger for backup/restore/import scope, checksum
  posture, and persisted-format impact;
- record the 0.190 256-row host recovery characterization as a regression
  floor, not a production IC budget guarantee;
- add a 0.191 1,024-row host recovery characterization floor while preserving
  the no-production-budget-guarantee language;
- add a mixed ordinary, conditional, and expression index recovery
  characterization floor while preserving the same no-production-budget
  boundary;
- add a PocketIC same-WASM upgrade/reentry instruction probe over a 32-row
  journaled `sql_perf` fixture while preserving the same no-production-budget
  boundary;
- give operators a checklist for storage choice, memory IDs, guarded access,
  batch atomicity, backup/import scope, checksums, and recovery-size limits;
- track streaming index fold/rebuild separately in
  `streaming-recovery-followup.md` for any future production-scale recovery
  guarantee.

Non-goals for this slice:

- do not add checksum bytes;
- do not implement backup/restore/import commands;
- do not change recovery runtime semantics;
- do not set a production recovery-size guarantee;
- do not implement streaming recovery.

Non-goals for this slice:

- do not change runtime behavior;
- do not change persisted bytes;
- do not add backup/import tooling;
- do not add checksum bytes;
- do not broaden recovery-size claims.

## Validation Plan

Routine validation:

- documentation lint or link check where available;
- format check;
- all-features workspace tests if implementation work is included;
- supported no-default checks if implementation work is included.

Evidence validation:

- 0.191 decisions cite 0.189/0.190 evidence rather than restating assumptions;
- any persisted-format change cites the 0.189 format classification rule;
- any backup/import claim names the hostile-input and compatibility boundary.

## Relationship To 0.190

0.191 depends on 0.190 evidence notes. If 0.190 does not produce enough evidence
to decide checksums, backup/restore/import posture, or operator limits, 0.191
should first close that evidence gap rather than invent a product contract.
