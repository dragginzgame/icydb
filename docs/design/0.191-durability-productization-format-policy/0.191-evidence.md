# 0.191 Durability Productization Evidence

This file records evidence added after the 0.190 reliability proof closeout.
It supports the 0.191 durability, persisted-format, and operator guidance
contracts without changing runtime persisted formats.

## Decision Ledger

### Backup, Restore, And Import

Decision: raw stable-memory backup, restore, and import are not supported
product surfaces in the current line.

Supported boundary:

- same-canister stable memory under normal IC execution and upgrade
  preservation;
- internally produced interrupted commit or recovery states;
- guarded recovery before IcyDB read/write entrypoints.

Unsupported boundary:

- importing raw stable-memory bytes from another canister;
- accepting untrusted backup images;
- version-gap restore from older internal formats;
- treating malformed external bytes as recoverable production data.

Rationale: 0.189 and 0.190 added fail-closed recovery and malformed persisted
byte coverage for IcyDB-produced states, but they did not add an import
compatibility contract, hostile-input validation pass, checksum envelope, or
operator restore workflow.

Persisted-format classification: no persisted-format change. This decision is
documentation-only and does not alter marker, journal, row, schema, index,
cursor, or structural-value bytes.

### Checksum Posture

Decision: the current line does not add persisted checksums.

Rationale: the supported threat model is same-canister stable memory plus
bounded, fallible decoding and guarded recovery of internally produced states.
The 0.189/0.190 malformed corpus proves fail-closed behavior for known
malformed envelopes, but it does not claim end-to-end detection of every
well-formed-but-wrong value or hostile imported image. Adding checksums should
be designed together with the import/restore threat model.

Persisted-format classification: no persisted-format change. Future checksum
bytes are persisted format and must be classified under
`docs/contracts/PERSISTED_FORMAT_POLICY.md` before implementation.

Classification for this slice:

## Recovery Size Characterization

### Status

0.191 raises the checked host recovery characterization floor for startup
secondary-index rebuild from 256 rows to 1,024 rows, and adds a mixed-shape
index rebuild floor over ordinary, conditional, and expression indexes.

This is still a host regression floor, not a production IC recovery-size
ceiling or instruction-budget guarantee.

### Simple Secondary-Index Dataset Shape

- Rows: 1,024 authoritative `RecoveryIndexedEntity` rows.
- Primary key: one `Ulid` key per row.
- Secondary indexes: one non-unique generated index over `group`.
- Stale index state: one stale secondary-index entry with no authoritative row.
- Journal tail: empty; recovery authority is the marker plus canonical row
  store.
- Recovery operation: marker-authorized startup recovery clears stale secondary
  index state and rebuilds derived secondary entries from authoritative rows.

### Mixed Index-Shape Dataset Shape

- Rows: 384 authoritative rows, split into 128 ordinary indexed rows, 128
  conditional-index rows, and 128 expression-index rows.
- Primary key: one `Ulid` key per row.
- Secondary indexes:
  - one non-unique generated index over `group`;
  - one non-unique generated conditional index over active `group` rows;
  - one non-unique generated expression index over `UPPER(email)`.
- Stale index state: one stale index-only entry per shape.
- Journal tail: empty; recovery authority is the marker plus canonical row
  store.
- Recovery operation: marker-authorized startup recovery clears stale secondary
  index state, rebuilds ordinary and expression index entries for every
  matching row, and rebuilds conditional index entries only for predicate-true
  rows.

### Executable Claim

The simple secondary-index regression test proves that guarded startup
recovery:

- rebuilds exactly 1,024 row-derived secondary-index entries;
- drops stale index-only state;
- leaves the canonical row store unchanged;
- clears the marker after successful recovery;
- decodes sampled rebuilt entries back to their original primary keys.

The mixed index-shape regression test proves that guarded startup recovery:

- rebuilds exactly the ordinary, conditional, and expression index keys derived
  from canonical rows;
- drops stale index-only state for each index shape;
- leaves the canonical row store unchanged;
- clears the marker after successful recovery;
- keeps conditional-index predicate-false rows absent;
- decodes sampled conditional and expression entries back to their original
  primary keys.

The PocketIC same-WASM upgrade/reentry probe proves that a small journaled
`PerfAuditJournaledUser` fixture can survive an upgrade, enter the normal
guarded update path, return one row from a journaled fluent LIMIT 1 query, and
stay under the checked regression instruction budget.

### Evidence

Executable test:

- `recovery_startup_rebuilds_secondary_index_large_host_floor`
- `recovery_startup_rebuilds_mixed_index_shapes_host_floor`
- `sql_perf_journaled_upgrade_guarded_reentry_stays_bounded`

The previous 256-row characterization remains as a smaller window:

- `recovery_startup_rebuilds_secondary_index_characterization_window`

### PocketIC Upgrade/Reentry Dataset Shape

- Rows: 32 deterministic `PerfAuditJournaledUser` rows loaded by
  `load_journaled_reentry_probe_fixture`.
- Storage mode: journaled stable-memory store in the audit canister.
- Upgrade operation: rebuild and install the same local `sql_perf` WASM over
  the already loaded fixture, preserving stable memory.
- Reentry operation: one update call executes a journaled fluent
  `ORDER BY id ASC LIMIT 1` query and measures
  `ic_cdk::api::performance_counter(1)` inside the canister.
- Regression budget: `JOURNALED_UPGRADE_REENTRY_BUDGET` in
  `testing/integration/tests/sql_perf_audit.rs`.
- Representative validation sample: first guarded reentry after upgrade
  measured `2,152,129,178` instructions; the next warm guarded call measured
  `318,340` instructions.

### Non-Claim

This does not claim:

- a production IC instruction-budget bound;
- a maximum supported production index size;
- streaming rebuild support;
- backup/import validation support;
- that same-WASM upgrade/reentry over the audit fixture is a production-size
  ceiling for arbitrary schemas or indexes;
- that the full `sql_perf` corpus is within a single-message guarded reentry
  budget after upgrade.

Those remain separate productization decisions.

### Follow-Up

Before broad production-scale recovery claims, IcyDB should add one of:

- broader PocketIC or instruction-budget measurements for recovery work;
- a documented production recovery-size bound;
- a streaming index fold/rebuild design.

The streaming follow-up is tracked in
`docs/design/0.191-durability-productization-format-policy/streaming-recovery-followup.md`.
