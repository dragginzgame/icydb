# IcyDB 0.168 — Store Storage Engine Boundary

## Status

Proposed.

## Summary

0.168 introduces an explicit per-store storage-engine boundary without changing
runtime storage behavior.

Today, every IcyDB `Store` is implicitly stable-memory-backed. A store owns
three stable memory roles:

```text
data
index
schema
```

Generated wiring initializes each role through `ic_memory_key!(...)` using the
configured memory IDs and stable keys. This release makes that implicit
assumption explicit by adding a nested store storage configuration, with
`storage(stable(...))` as the only implemented storage form in 0.168.

The long-term goal is to support multiple store engines:

| Mode | Meaning |
| --- | --- |
| `storage(stable(...))` | Durable stable-memory store, current behavior. |
| `storage(heap())` | Volatile Rust `BTreeMap` store, future. |
| `storage(cached_stable(...))` | Stable store with heap acceleration, future. |
| `storage(journaled(...))` | Heap-fronted durable store with stable WAL/checkpointing, future. |

0.168 is the first minor patch toward that goal. It adds the
schema/build/runtime vocabulary needed for per-store storage configuration while
preserving the current stable-memory ABI.

## Current State

The schema `Store` node currently assumes every store has three memory IDs:

```rust
data_memory_id: u8
index_memory_id: u8
schema_memory_id: u8
```

These correspond to the data, index, and schema stable-memory roles.

Store validation currently validates all three memory IDs, validates stable
keys for data/index/schema roles, and requires those three memory IDs to be
distinct.

Generated store wiring currently emits one thread-local `DataStore`,
`IndexStore`, and `SchemaStore` per schema store. Each is initialized through
`ic_memory_key!(key = ..., id = ...)`, and the generated store registry records
stable allocation identities for the three roles.

The runtime store wrappers are already thin persistence boundaries:

- `DataStore` wraps a stable
  `BTreeMap<RawDataStoreKey, RawRow, VirtualMemory<DefaultMemoryImpl>>`.
- `IndexStore` wraps a stable
  `BTreeMap<RawIndexStoreKey, IndexEntryValue, VirtualMemory<DefaultMemoryImpl>>`.
- `SchemaStore` wraps a stable schema metadata `BTreeMap`.

The store registry already supports absent allocation identities through
`StoreAllocationIdentities::absent()`, which is useful for future heap stores.

## Problem

IcyDB currently has no explicit language for "what kind of storage backs this
store."

That makes future heap or cached storage modes harder to introduce safely
because stable-memory assumptions are spread across:

- schema validation
- generated store wiring
- allocation identity reporting
- diagnostics
- schema-store metadata
- runtime store handles
- relation and commit expectations

If heap stores are added directly without first making the storage engine
explicit, the implementation risks becoming a set of one-off exceptions:

- if heap, skip memory IDs here
- if heap, skip allocation metadata there
- if heap, special-case diagnostics elsewhere

0.168 should avoid that by introducing one explicit storage boundary first.

## Goals

- Add an explicit store storage configuration.
- Preserve existing stable-store behavior exactly.
- Make the macro storage form explicit with `storage(stable(...))`.
- Keep memory IDs required inside `storage(stable(...))`.
- Keep stable-key generation unchanged.
- Keep stable allocation identity unchanged.
- Prepare schema/build/runtime code for a future heap engine.
- Make generated store wiring branch through an explicit stable-engine path,
  even though only stable is implemented in 0.168.
- Add tests proving that `storage(stable(...))` produces the same stable wiring
  as the current implicit stable store declaration.
- Ensure the new storage vocabulary does not imply heap support yet.

## Non-Goals

- No volatile heap store implementation in 0.168.
- No Rust `std::collections::BTreeMap` backend in 0.168.
- No stable-memory migration.
- No physical memory ID change.
- No stable-key format change.
- No allocation metadata semantic change.
- No WAL.
- No heap-fronted stable store.
- No background flush.
- No pre-upgrade heap serialization.
- No relation semantic changes.
- No storage-engine mixing within one store.

## Core Invariant

0.168 must preserve the existing stable-store ABI:

```text
stable store identity = data/index/schema memory IDs + stable keys
```

Adding a storage configuration must not change:

- `memory_id`
- `stable_key`
- data/index/schema allocation roles
- stable-memory initialization
- schema-store persistence
- commit/recovery behavior

For 0.168, all live stores are still stable stores.

The storage configuration is an explicit model boundary, not a behavior change.

## User-Facing Syntax

Use an explicit nested stable storage form:

```rust
#[store(
    name = "sessions",
    storage(stable(
        data_memory_id = 20,
        index_memory_id = 21,
        schema_memory_id = 22,
    )),
)]
pub struct SessionStore;
```

Reserved future form:

```rust
#[store(
    name = "sessions",
    storage(heap()),
)]
pub struct SessionStore;
```

In 0.168, `storage(heap())` must reject with a clear diagnostic:

```text
heap store storage is reserved but not implemented in 0.168;
use storage(stable(...)) for current stable-memory behavior
```

## Naming

Use user-facing syntax:

```rust
storage(stable(data_memory_id = 20, index_memory_id = 21, schema_memory_id = 22))
```

rather than:

```rust
storage = "stable", data_memory_id = 20, index_memory_id = 21, schema_memory_id = 22
```

Rationale:

- `stable(...)` owns the fields that only make sense for stable memory.
- `heap(...)` can later own heap-specific fields without accepting irrelevant
  stable memory IDs.
- `storage(...)` names the configuration axis, while `stable(...)`, `heap()`,
  and future variants own their variant-specific config.
- `engine` is an implementation term; `stable(...)` and future storage forms
  describe the user-visible storage contract.

Internal enum names may still use `StoreEngine` or `StoreStorage`.

Recommended schema/derive naming:

```rust
pub enum StoreStorage {
    Stable(StoreStableMemoryConfig),
}
```

The derive parser may use a parse-only reserved variant for targeted
diagnostics:

```rust
enum ParsedStoreStorage {
    Stable(StoreStableMemoryConfig),
    HeapReserved,
}
```

`storage(stable(...))` requires stable memory IDs. `storage(heap())` must not
parse or require stable memory IDs before the heap-reserved diagnostic can be
emitted.

## Schema Model

Replace the implicit memory-ID fields:

```rust
pub struct Store {
    def: Def,
    ident: &'static str,
    name: &'static str,
    canister: &'static str,
    data_memory_id: u8,
    index_memory_id: u8,
    schema_memory_id: u8,
}
```

with an explicit storage config:

```rust
pub struct Store {
    def: Def,
    ident: &'static str,
    name: &'static str,
    canister: &'static str,
    storage: StoreStorage,
}

pub enum StoreStorage {
    Stable(StoreStableMemoryConfig),
}

pub struct StoreStableMemoryConfig {
    data_memory_id: u8,
    index_memory_id: u8,
    schema_memory_id: u8,
}
```

0.168 can hard-cut constructor usage to `new_stable(...)`:

```rust
impl Store {
    pub const fn new_stable(
        def: Def,
        ident: &'static str,
        store_name: &'static str,
        canister: &'static str,
        stable: StoreStableMemoryConfig,
    ) -> Self {
        Self {
            def,
            ident,
            name: store_name,
            canister,
            storage: StoreStorage::Stable(stable),
        }
    }
}
```

Do not add a heap constructor in 0.168 unless it returns an unsupported
validation error path. The first slice should not create a heap-backed runtime
store.

## Accessors

Existing stable accessors may remain for compatibility:

```rust
data_memory_id() -> u8
index_memory_id() -> u8
schema_memory_id() -> u8
```

but internally they should route through stable storage config.

Add explicit storage accessors:

```rust
storage() -> &StoreStorage
is_stable_storage() -> bool
stable_memory_config() -> Option<&StoreStableMemoryConfig>
```

Future heap work should be able to avoid calling stable-only accessors.

## Validation Rules

### Stable Stores

For `storage(stable(...))`:

- `data_memory_id` is required.
- `index_memory_id` is required.
- `schema_memory_id` is required.
- memory IDs must be inside the canister memory range.
- memory IDs must not be reserved.
- data/index/schema memory IDs must be distinct.
- stable keys must validate exactly as today.

### Heap Stores

In 0.168:

- `storage(heap())` rejects as reserved / not implemented.
- heap syntax must not be silently accepted.
- memory IDs on heap syntax must not be silently ignored.

If the parser accepts heap syntax in order to provide a targeted error, then
validation should reject all of these:

```rust
#[store(storage(heap()))]
#[store(storage(heap(data_memory_id = 1)))]
#[store(storage(heap(index_memory_id = 2)))]
#[store(storage(heap(schema_memory_id = 3)))]
```

with diagnostics explaining that heap storage is not implemented in 0.168.

Derive parsing should therefore parse storage as a nested enum form. Missing
stable IDs should produce stable-specific diagnostics only inside
`storage(stable(...))`; they should not preempt the reserved-heap diagnostic.

## Generated Wiring

Generated stable wiring should be moved behind an explicit stable-storage
rendering path.

Current behavior:

```rust
DataStore::init(ic_memory_key!(key = data_stable_key, id = data_memory_id))
IndexStore::init(ic_memory_key!(key = index_stable_key, id = index_memory_id))
SchemaStore::init(ic_memory_key!(key = schema_stable_key, id = schema_memory_id))
```

must remain the generated output for stable stores.

Implementation shape:

```rust
fn store_registry_entry_tokens(...) -> (...) {
    match store.storage() {
        StoreStorage::Stable(config) => {
            stable_store_registry_entry_tokens(..., config)
        }
    }
}
```

Do not add heap generated wiring in 0.168.

The stable branch must continue to register:

```rust
StoreAllocationIdentities::new(
    data_identity,
    index_identity,
    schema_identity,
)
```

Heap stores in the future should register:

```rust
StoreAllocationIdentities::absent()
```

but that is not active in 0.168.

## Diagnostics

0.168 may add storage-mode reporting if there is a natural existing diagnostics
surface.

Recommended diagnostic shape:

```yaml
store: sessions
storage: stable
data_allocation:
  memory_id: 20
  stable_key: icydb.<namespace>.sessions.data.v1
index_allocation:
  memory_id: 21
  stable_key: icydb.<namespace>.sessions.index.v1
schema_allocation:
  memory_id: 22
  stable_key: icydb.<namespace>.sessions.schema.v1
```

If adding this requires broad diagnostics churn, defer it.

Minimum 0.168 diagnostic requirement:

- existing stable allocation diagnostics must not change meaning.

## Migration / Compatibility

0.168 is intended to be runtime-behavior-preserving.

0.168 hard-cuts the macro syntax so stores must declare
`storage(stable(...))` explicitly. This is a source-level macro change only; no
stable-memory migration is required.

No stable-key generation changes.

No physical memory ID changes.

No schema-store data format changes unless the schema model serialization must
record `storage = stable`. If it does, the change must be treated as schema
metadata shape evolution only, not stable row/index ABI evolution.

## Relation Policy

0.168 does not add heap stores, so no relation behavior changes.

However, the design should reserve the future policy:

- stable stores must not require strong referential integrity against volatile
  heap stores.

Future direction:

- heap -> stable relation: likely allowed.
- stable -> heap strong relation: reject.
- stable -> heap weak/optional relation: deferred.

Do not implement this policy in 0.168 unless the schema model needs a
placeholder diagnostic for reserved heap syntax.

## Future Heap Store Semantics

The next slice may add:

```rust
#[store(storage(heap()))]
```

with semantics:

- data rows live in Rust heap `BTreeMap`
- index rows live in Rust heap `BTreeMap`
- schema snapshots live in Rust heap `BTreeMap`
- no stable memory IDs
- no stable allocations
- no upgrade durability
- no WAL
- no flush

That future work should require a backend abstraction inside:

- `DataStore`
- `IndexStore`
- `SchemaStore`

The likely implementation direction is:

```rust
enum DataStoreBackend {
    Stable(StableDataMap),
    Heap(std::collections::BTreeMap<RawDataStoreKey, RawRow>),
}
```

and equivalent backend enums for `IndexStore` and `SchemaStore`.

0.168 should not add those backend enums unless they are needed to land the
storage-mode model cleanly.

## Required Tests

### Schema / Validation

- Store with `storage(stable(...))` validates as stable.
- Store without a storage form rejects after the hard cut.
- Stable store still requires data/index/schema memory IDs.
- Stable store still rejects duplicate data/index/schema memory IDs.
- Stable store still validates stable key shape.
- `storage(heap())` rejects as reserved/not implemented.
- `storage(heap(...))` with stable memory IDs rejects with a diagnostic that
  heap storage is unsupported in 0.168.
- Unknown storage form rejects:

  ```rust
  storage(durable())
  storage(memory())
  storage(cached())
  ```

### Generated Wiring

- `storage(stable(...))` emits the same stable wiring as the current implicit
  stable store declaration.
- Stable generated wiring still uses `ic_memory_key!`.
- Stable generated wiring still registers all three allocation identities.
- No generated heap `DataStore` / `IndexStore` / `SchemaStore` path exists in
  0.168.

### Registry / Allocation Identity

- Stable stores still register data/index/schema allocation identities.
- Allocation identity values are unchanged for an existing fixture.
- `StoreAllocationIdentities::absent()` remains available for future heap
  wiring but is not used by stable generated stores.

### Diagnostics

If diagnostics are touched:

- Stable stores still show data/index/schema allocations.
- Storage mode, if shown, is `stable`.
- Stable allocation metadata semantics from 0.163 are unchanged.

## Open Questions

### Should the public syntax be flat `storage = "stable"` or nested `storage(stable(...))`?

Tentative answer: use nested `storage(stable(...))`.

### Should `storage(heap())` be parsed and rejected with a targeted diagnostic?

Alternative: treat it as unknown until heap lands.

Tentative answer: parse and reject with a targeted reserved-feature diagnostic.

### Should runtime `StoreStorage::Heap` exist as a real enum variant in 0.168?

Tentative answer: avoid a live heap runtime variant; allow parser-level
recognition for diagnostics only.

### Should diagnostics show `storage: stable` immediately?

Tentative answer: only if the change is local and low-noise.

## Recommended Changelog Wording

- Added an explicit store storage-mode boundary and made the existing
  stable-memory store engine explicit while preserving current stable store
  behavior.
- Stores now declare stable storage with a nested `storage(stable(...))`
  configuration while keeping the same data/index/schema memory IDs, stable
  keys, generated wiring, and allocation identity semantics.
- Reserved heap storage syntax for a future volatile store engine without
  adding heap-backed runtime storage in this release.

## Verdict

0.168 should be a low-risk boundary-setting release.

It should make this true:

- IcyDB stores now have an explicit nested storage configuration.
- The only implemented mode is stable.
- Stable behavior is unchanged.
- Future heap stores have a clean place to attach.

Do not implement heap storage in 0.168. The value of this release is making the
storage-engine seam explicit before adding a second engine.
