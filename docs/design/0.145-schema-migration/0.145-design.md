# 📄 IcyDB 0.145 — Schema Identity & Layout Authority

## Summary

0.145 introduces **persistent schema identity and layout authority** for entity rows, enabling safe schema evolution without abandoning the slot-framed storage model.

The release formalizes the separation between:

* **logical schema identity** (field meaning)
* **physical row layout** (slot position)

This resolves the current limitation where schema changes are tightly coupled to slot order, making non-trivial migrations unsafe.

The persisted schema record becomes the live runtime schema authority.
Macro-generated `EntityModel` and `FieldModel` values are treated as a
compiled proposal for the next schema version, not as unconditional runtime
truth. On startup or upgrade, IcyDB diffs the stored live schema against the
compiled proposal and opens the database only after every difference is either
automatically safe or explicitly confirmed by a migration policy. Once
accepted, the stored schema is advanced and the live runtime schema and written
schema are synced.

---

# 🎯 Goals

* Enable **schema evolution** without breaking existing data
* Preserve **slot-based row performance**
* Maintain **fail-closed structural decode guarantees**
* Introduce **minimal persistent metadata**, without impacting hot paths
* Avoid multi-version runtime complexity (for now)

---

# ❌ Non-Goals

* No support for multi-version row decoding at runtime
* No per-row schema versioning
* No indexing changes
* No planner or query semantics changes
* No row format change in this release

---

# 📦 Release Shape

This work should land as one large minor-version slice, not as independently
shippable partial schema-authority patches.

The schema BTree, startup reconciliation, live-schema authority switch,
retired-slot handling, and SQL capability gate form one correctness boundary.
Introducing only some of them creates a split-brain model where generated
metadata and persisted metadata can disagree without one clear authority.

Mergeable intermediate patches inside the branch are fine, but the release
boundary should not expose:

* persisted schema metadata that is not the live runtime authority
* derive-generated metadata that can silently override the stored live schema
* logical field removal without retained physical slot metadata or rewrite
* SQL planning paths that bypass `sql_capabilities(field_kind)`
* writes opening before accepted schema proposals are persisted

The minor version may still stage implementation internally, but the first
published version containing `__icydb_schema` must also contain the startup
reconciliation and fail-closed authority rules that make it safe.

# 🧠 Core Design

## 1. Stable Field Identity (FieldId)

Each field is assigned a **stable, persistent identifier**:

```rust
struct FieldId(u32);
```

Properties:

* Assigned once at schema creation
* Never reused
* Stable across renames, reorders, and migrations

---

## 2. Row Layout Mapping

Logical fields are mapped to physical slots via a layout:

```rust
struct RowLayout {
    version: u32,
    field_to_slot: Vec<(FieldId, u16)>,
}
```

This introduces:

```text
FieldId → SlotIndex
```

instead of:

```text
SlotIndex = meaning (old model)
```

---

## 3. Persisted Schema Store

A new internal BTree is introduced:

```text
__icydb_schema
```

### Key

```text
(entity_name, version)
```

### Value

```rust
struct PersistedSchema {
    version: u32,
    fields: Vec<PersistedField>,
}

struct PersistedField {
    id: FieldId,
    name: String,
    slot: u16,
    field_kind: PersistedFieldKind,
    nullable: bool,
    default: Option<Value>,
    storage_decode: FieldStorageDecode,
    leaf_codec: LeafCodec,
}
```

`PersistedField` records the live decode/layout contract for a field, not just
a name-to-slot registry. Startup reconciliation compares the compiled
`EntityModel`/`FieldModel` proposal against persisted records to decide whether
the canister may continue, must run a controlled migration, or must fail
closed. `field_kind` records the expected IcyDB schema/runtime shape for the
slot payload, including non-SQL concepts such as relations, sets, maps,
generated enums, blobs, and structured records. `nullable` and `default`
define absence semantics for existing rows. `storage_decode` and `leaf_codec`
preserve the row-level decode contract so storage does not infer payload
meaning from bytes alone.

`PersistedFieldKind` must be the full IcyDB field shape, not a traditional SQL
type and not the reduced predicate-validation `FieldType` currently used by
the planner. Its persisted form should mirror the generated `FieldKind`
contract closely enough to distinguish scalar, blob, list, set, map,
relation, enum, and structured payloads:

```rust
enum PersistedFieldKind {
    Scalar(PersistedScalarKind),
    Blob,
    List(Box<PersistedFieldKind>),
    Set(Box<PersistedFieldKind>),
    Map {
        key: Box<PersistedFieldKind>,
        value: Box<PersistedFieldKind>,
    },
    Relation(PersistedRelationKind),
    Enum(PersistedEnumKind),
    Structured(PersistedStructuredKind),
}
```

For example, `Vec<u8>` is not inferred from a blob-like slot payload. The
schema must persist either a blob field shape or a list-of-unsigned-byte field
shape, depending on the declared schema-level type, and decode rejects payloads
that do not match that persisted shape.

### SQL Compatibility Projection

The persisted schema store is not a SQL catalog. It is the IcyDB storage and
runtime schema authority used to validate and decode existing stable rows.

SQL support is derived as a capability projection from `field_kind`:

* SQL predicates and ordering are allowed only for field kinds with explicit
  comparison, coercion, and literal-normalization semantics.
* Collections and structured fields remain persistable even when they are not
  valid SQL predicate or index keys.
* Blob/bytes fields may be selectable and transportable through SQL result
  rows, while still having limited SQL comparison semantics.
* Any field kind without a defined SQL mapping is rejected at SQL planning or
  lowering time, not weakened in the persisted schema.

This keeps non-SQL IcyDB types safe: storage decode validates the full IcyDB
field shape, and the SQL layer only exposes the subset of operations that are
well-defined for that shape.

The SQL capability projection is derived from the accepted live schema after
the compiled proposal has been reconciled with persisted layout metadata.
Persisted schema records authorize SQL access only for fields that remain in
the live logical schema; retired physical slots stay decode-visible but
SQL-inaccessible.

The projection should be centralized behind one planner-visible helper instead
of spread across ad hoc `FieldKind` matches:

```rust
struct SqlCapabilities {
    selectable: bool,
    comparable: bool,
    orderable: bool,
    groupable: bool,
    aggregate_input: AggregateInputCapabilities,
    literal_coercion: Option<SqlLiteralCoercion>,
    predicate_family: Option<SqlPredicateFamily>,
}

fn sql_capabilities(kind: &PersistedFieldKind) -> SqlCapabilities;
```

`sql_capabilities` is the admission gate for SQL operations. It should derive
from the full IcyDB field kind and classify what SQL may do with that field; it
must not erase the field kind into a generic SQL type. For example:

* `Blob` may be selectable but not generally orderable or comparable.
* `List`, `Set`, `Map`, and non-queryable `Structured` fields may be
  selectable only when result transport supports their `Value` shape, and they
  are rejected for scalar predicates, ordering, grouping, and index-key use.
* Numeric scalar kinds may admit numeric comparisons, arithmetic, `SUM`, and
  `AVG` according to their existing coercion rules.
* Text-like scalar kinds may admit text comparison, `LIKE`, prefix functions,
  and case-folding functions according to their existing normalization rules.
* Relation fields inherit SQL predicate capability from their key kind, but the
  relation identity remains part of the persisted field kind.

Planner and SQL lowering rejection paths should call this helper before
building executable plan fragments:

* `WHERE` and `HAVING` reject fields whose capabilities do not admit the
  requested predicate family.
* `ORDER BY` rejects fields whose capabilities are not orderable.
* `GROUP BY` and `DISTINCT` reject fields whose capabilities are not groupable
  under the current result-shaping contract.
* aggregate lowering rejects aggregate inputs that the capability table does
  not admit.
* index planning rejects candidate index keys that are not comparable/keyable
  under the same capability projection.

## 4. Schema Versioning (Per Entity)

Schema versions are:

* **scoped per entity**
* monotonic
* tied to layout changes
* produced by reconciling the compiled schema proposal with the persisted live
  schema

Example:

```text
character → v1, v2, v3
guild     → v1
```

---

## 5. Single-Version Runtime Model

0.145 enforces:

> **one active schema version per entity at runtime**

Implications:

* No mixed-layout reads
* No per-row version tracking
* No branching in hot paths

---

## 6. Startup Reconciliation

On canister startup or upgrade, IcyDB compares the compiled `EntityModel`
proposal against the latest persisted live schema record for that entity.

Allowed outcomes:

* No persisted schema exists: initialize the stored live schema from the
  compiled generated metadata.
* Proposal field IDs and persisted field IDs match with compatible field kind,
  storage-decode, and slot metadata: keep the stored live schema current.
* Proposal adds a field with an allowed default/nullability contract: allocate a
  new field ID/slot and persist a new schema version before opening the store.
* Proposal omits a previously persisted field: require an explicit confirmation
  policy, then mark that field as logically removed while retaining its retired
  physical slot metadata.
* Proposal changes a field kind, storage-decode contract, or slot mapping:
  require an explicit controlled migration or fail closed before opening the
  store.

The persisted schema decides the current logical API and physical layout. The
compiled schema is a candidate next version. Generated Rust code may expose a
typed view over the accepted live schema, but it does not silently override the
stored schema.

This means persisted rows must not be decoded directly into the current
derive-generated Rust struct at the storage boundary. If the stored live schema
contains retired slots or fields that are absent from the current compiled
proposal, the current struct cannot deserialize the full row shape. Storage
decode must first produce a schema-indexed dynamic row representation from the
persisted live schema. Generated structs are then projections over that dynamic
row after reconciliation has proven the projection is valid.

# 🔁 Schema Evolution Model

## Supported Changes

### Add field

* New FieldId assigned
* Appended slot (or mapped)
* Field kind, nullability, default, and decode semantics persisted with the
  new field
* Existing rows resolve missing values through the persisted default or
  nullable semantics; otherwise decode fails closed

### Rename field

* FieldId unchanged
* Name updated in schema metadata

### Remove field

* FieldId removed from the logical query/write surface
* Physical slot remains in the active layout as a retired slot until rows are
  rewritten
* Retired slot metadata remains persisted so structural decode can still
  validate existing rows fail-closed
* Slot reclamation requires a controlled rewrite that emits rows matching a new
  compacted layout; it must not happen by silently dropping the slot from the
  layout

---

## Unsupported (without migration)

* Slot reordering without rewrite
* Type changes incompatible with decode
* Arbitrary layout reshaping
* Slot reclamation without rewriting existing rows into the reclaimed layout

---

# 🔄 Migration Strategy

Schema changes follow:

1. New schema version written to `__icydb_schema`
2. New rows written using updated layout
3. Existing rows:

   * either remain valid under compatibility rules
   * or are rewritten (controlled migration)

0.145 does **not** introduce background compaction or multi-version coexistence.

## Removed Prototype Migration Modules

Removed before 0.145:

* `crates/icydb-core/src/db/schema_evolution`
* `crates/icydb-core/src/db/migration`

The abandoned `db/schema_evolution` prototype and the old row-op
`db/migration` engine were removed before 0.145 so they cannot bias the
slot-schema design.

0.145 must define a fresh supported schema-layout migration path instead of
reviving stale compatibility surfaces. The design may reuse existing commit
marker ownership if useful, but it must not rely on the removed migration-state
side channel in the commit control slot.

No partially exposed, unused migration compatibility infrastructure should be
reintroduced after this work lands.

# 🧾 Read Path Changes

Old:

```text
row[slot] → value
```

New:

```text
FieldId → slot (via RowLayout) → row[slot]
```

Storage decode becomes:

```text
stored bytes
  → DynamicRow keyed by persisted FieldId / retired slot metadata
  → typed generated projection only when the accepted live schema permits it
```

Notes:

* Mapping is resolved **once per query / execution context**
* No additional per-row overhead introduced
* Hot path remains slot-based
* Typed Rust structs are adapters over accepted live-schema fields, not the
  primary storage decode target
* Retired slots remain decode-visible to `DynamicRow` but are omitted from
  generated typed projections

---

# ⚡ Performance Considerations

* Slot-based reads remain unchanged
* Layout indirection is amortized outside the row loop
* No additional allocations introduced
* No per-row branching

---

# 🔒 Safety & Invariants

* Structural decode remains **fail-closed**
* Slot count mismatches still error
* Layout must fully describe all physical slots expected in stored rows,
  including retired slots left behind by logical field removal
* Persisted field metadata must define the expected field kind for every active
  or retired physical slot
* The expected field kind must include cardinality and container shape, so
  scalar, blob, list, set, and map payloads cannot be confused during decode
* Missing slots are accepted only when the persisted field metadata explicitly
  allows null or provides a default value
* Slot payloads whose persisted encoding or decoded value does not match the
  persisted field kind are rejected before they enter dynamic row
  materialization
* Typed generated projections may read only fields present in the accepted live
  logical schema and must fail if a required projected field cannot be supplied
* Missing fields resolve via defaults or explicit error

# 🧩 Implementation Plan

## Step 1 — Introduce FieldId

* Add `FieldId` to schema model
* Assign deterministically from current schema

## Step 2 — Persist Schema

* Add `__icydb_schema` BTree
* Write initial live schema snapshot on first startup/init from compiled
  generated metadata

## Step 3 — Startup Reconciliation

* Compare compiled `EntityModel`/`FieldModel` proposal metadata with persisted
  live schema records
* Admit only compatible proposal changes automatically
* Require explicit confirmation, controlled migration, or fail closed for
  destructive or ambiguous changes

## Step 4 — Introduce RowLayout

* Build layout from persisted schema
* Keep slot order identical to current system

## Step 5 — Reader Indirection

* Replace direct slot access with:

  ```rust
  slot = layout[field_id]
  ```

* Ensure resolution happens outside hot loops
* Decode persisted bytes into a dynamic row keyed by accepted live schema
  metadata before projecting into generated Rust structs
* Make generated entity structs typed projections/adapters instead of the first
  storage deserialization target

## Step 6 — Schema Updates

* On schema change:

  * assign new version
  * persist updated live schema and layout before opening write paths

## Step 7 — SQL Capability Gate

* Define `sql_capabilities(field_kind)` as the single SQL admission boundary
* Replace scattered SQL/planner `FieldKind` checks with capability lookups
* Route unsupported SQL operations through typed planner/lowering errors before
  execution

# 🧠 Design Rationale

## Why not name-keyed rows?

* Higher storage cost
* Slower reads
* Breaks slot-based optimization

## Why not per-row schema version?

* Adds overhead per row
* Introduces branching in hot path
* Unnecessary for current system scope

## Why not multi-version support now?

* Adds complexity across decode, planner, executor
* Not needed for current scale
* Can be layered later if required

---

# 🚀 Future Extensions (Post-0.145)

* Segment-level schema versioning
* Background compaction / rewrite
* Multi-version read support (if needed)
* Slot reuse / compaction strategies
* Path-based indexing using FieldId

# 🧾 TL;DR

0.145 introduces:

* **FieldId (stable identity)**
* **RowLayout (FieldId → slot mapping)**
* **Persisted live schema metadata (`__icydb_schema`)**
* **Startup reconciliation for compiled schema proposals**
* **SQL capability admission derived from the accepted live schema**

Result:

* Keep slot performance
* Enable safe schema evolution
* Avoid schema-authority split brain
* Avoid multi-version complexity
