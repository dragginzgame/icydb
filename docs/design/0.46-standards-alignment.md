# 0.46 - Standards Alignment

## Status

Draft  
Target: `0.46`  
Dependency: `0.45` cleanup baseline  
Last updated: 2026-03-08

Current implementation slices (branch-local):

- planner lowering now folds `TRUE` predicates and routes `LIMIT 0` / constant
  `FALSE` windows to empty by-keys access
- aggregate terminals now short-circuit provably empty windows to canonical
  zero outputs without scan-budget use
- predicate normalization now performs conservative AND simplification for
  equality/bound contradictions and bound tightening before planner pushdown
- planner tests now pin plan-stability behavior for equivalent and
  contradictory canonical predicate shapes
- scalar DISTINCT planning now lowers duplicate-safe single-path access shapes
  to no-op runtime dedup, while composite shapes retain explicit streaming or
  materialized dedup strategies
- branch-surface containment note: DISTINCT dedup strategy branching is now
  owned by one planner helper (`distinct_runtime_dedup_strategy`) instead of
  being re-decided at multiple executor call sites
- temporal view boundaries now return semantic `Timestamp`/`Duration` types
  instead of raw `u64` views, preserving millisecond storage encoding
- session now exposes a constant-row terminal (`select_one`) that returns
  `Value::Int(1)` without planner/executor routing, with regression coverage
  for zero-metrics/no-access behavior
- aggregate EXISTS regression coverage now pins full-scan `offset + 1`
  short-circuit scan budgeting
- boolean AND-constraint simplification is now owned by
  `db/predicate/simplify.rs`, with `normalize` reduced to orchestration +
  canonical ordering/dedup responsibilities
- route planning now projects one explicit `AccessWindow` contract
  (`lower_bound`, `upper_bound`, `fetch_limit`) consumed by bounded
  index-range/load/aggregate hint derivation
- continuation-to-route handoff now exposes `access_window_for(...)` so route
  feasibility and hint derivation consume one preprojected window contract
  instead of re-reading raw route-window fields
- scalar continuation projection and route window contracts now carry one
  keep-window `AccessWindow` authority, with lookahead fetch budgets derived
  mechanically instead of storing separate keep/fetch counts
- scalar route window derivation now consumes one planner-owned
  `ScalarAccessWindowPlan` directly, removing duplicate executor-side
  reconstruction of offset/limit window semantics
- bounded aggregate probe-hint safety now keys DISTINCT+offset suppression to
  logical DISTINCT intent (`scalar_plan().distinct`) so offset windows remain
  unbounded even when runtime DISTINCT dedup strategy is a no-op
- `count_distinct_by` now routes through the existing distinct-projection
  terminal path, enabling covering index and constant-covering fast paths for
  eligible field-target DISTINCT counts with existing stale-row parity guards
- covering existing-row terminal eligibility is now planner-owned
  (`query/plan/covering.rs`) and consumed by both route terminal selection and
  EXPLAIN covering-projection reporting, removing duplicated COUNT/EXISTS
  eligibility logic
- covering projection order/context derivation is now planner-owned
  (`query/plan/covering.rs`) and consumed by executor field-projection
  terminals, removing duplicate index-order vs primary-key-order contract
  interpretation in executor runtime modules

## 1. Objective

`0.46` is a standards-alignment line focused on API return-shape correctness
and behavior-preserving execution improvements.

Primary goal:

- align query/runtime return types with common database expectations where
  those expectations improve API clarity and reduce storage-type leakage.

This document starts with the temporal return-type note and is intentionally
open for additional `0.46` items.

## 4. Track A - Temporal Return-Type Normalization

### 4.1 Problem

Current behavior can leak temporal storage representation at API boundaries
(for example raw integer values such as `u64`).

That shape is storage-friendly but not API-friendly. It diverges from common
database expectations where temporal outputs are typed domain values.

### 4.2 Standards Target

Align temporal result surfaces with standard typed returns used by SQL systems
such as MySQL and PostgreSQL:

- date-like values return a date type
- timestamp-like values return a timestamp type
- duration/interval-like values return a duration type

### 4.3 Required Contract

- public query/runtime result boundaries must return semantic temporal types:
  - `Date`
  - `Timestamp`
  - `Duration`
- raw numeric temporal storage values must not be the default API return shape
  for temporal fields.
- internal storage units (for example millis/micros or other integer encodings)
  remain an internal concern.
- storage representation may stay integer-backed as long as boundary conversion
  is explicit and tested.

### 4.4 Scope in `0.46`

- define one canonical temporal return contract for query/runtime boundaries.
- remove remaining raw temporal numeric leakage from API-facing result paths.
- lock behavior with regression tests at decode/projection/response boundaries.
- document boundary guarantees for users and contributors.

### 4.5 Acceptance Criteria

- temporal query fields no longer return raw `u64` as the primary API type.
- typed temporal values are stable across:
  - scalar projections
  - grouped projections (where applicable)
  - serialized response DTO surfaces
- tests explicitly cover storage-vs-return separation:
  - integer-backed storage remains unchanged
  - returned type is semantic (`Date`/`Timestamp`/`Duration`)

## 5. Track B - DISTINCT Pushdown and Streaming Dedup

### 5.1 Problem

DISTINCT currently tends to resolve as a post-access executor operation, which
can increase memory pressure on large windows.

### 5.2 Goal

Push DISTINCT down to planner/access/executor streaming paths when semantics
are preserved, so the runtime can avoid materializing large duplicate sets.

### 5.3 Pushdown Opportunities

| Case | Strategy |
|---|---|
| primary-key scan DISTINCT | planner marks DISTINCT as no-op where PK uniqueness already guarantees dedup |
| unique secondary index DISTINCT | planner marks DISTINCT as no-op for covered unique key projections |
| prefix index DISTINCT | planner routes to early dedup on index-prefix stream |
| ordered DISTINCT | executor uses streaming dedup over ordered access stream |

Example target shape:

`SELECT DISTINCT user_id FROM events ORDER BY user_id`

Planner/access/executor target flow:

`IndexPrefixScan(user_id) -> streaming dedup`

### 5.4 Ownership and Module Plan

Planner ownership:

- add `query/plan/distinct_pushdown.rs` for DISTINCT eligibility and pushdown
  contract derivation.

Executor ownership:

- add `executor/load/distinct_stream.rs` for streaming DISTINCT execution over
  planner-approved ordered/index-backed windows.

### 5.5 Required Safety Gates

Pushdown is allowed only when all gates hold:

- projection shape is dedup-compatible with pushdown source columns
- ORDER BY contract remains satisfied without reordering drift
- predicate semantics do not require post-materialization entity evaluation
- NULL and collation/equality behavior matches canonical DISTINCT semantics
- continuation/cursor resume invariants remain stable

### 5.6 Acceptance Criteria

- eligible DISTINCT queries avoid full post-access dedup materialization.
- planner emits explicit pushdown/no-op/fallback reason paths.
- executor streaming dedup preserves exact query results vs fallback.
- explain/trace/test surfaces make DISTINCT pushdown decisions observable.

## 6. Track C - LIMIT Pushdown Depth Improvements

### 6.1 Problem

Current execution often behaves like:

`scan -> filter -> limit`

even when planner/access already knows an upper bound window can be enforced
earlier.

### 6.2 Goal

Push bounded-window contracts deeper into the access layer so eligible scans can
stop earlier.

### 6.3 Planned Contract

Introduce an access-bound contract shape:

`AccessWindow { lower_bound, upper_bound, fetch_limit }`

Planner sets this once; access/executor consume it mechanically.

### 6.4 Ownership and Module Plan

- planner contract derivation under `query/plan` (bounded window projection)
- access consumption for early-stop behavior on eligible path kinds

### 6.5 Acceptance Criteria

- eligible LIMIT queries convert from unbounded scan windows to bounded access
  windows.
- row/ordering/cursor semantics remain identical to canonical fallback.
- explain/trace surfaces expose when bounded access-window pushdown is active.

## 7. Track D - Covering Index Projection Expansion

### 7.1 Problem

Covering/index-only fast paths exist but do not yet cover enough aggregate and
dedup shapes.

### 7.2 Goal

Expand index-covering execution for:

- `COUNT`
- `DISTINCT`
- eligible `GROUP BY` windows where projected fields are index-covered

### 7.3 Example Target

`SELECT COUNT(*) FROM table WHERE email = ?`

Target flow:

`IndexPrefixScan -> count fold (no row loads)`

### 7.4 Ownership and Module Plan

- planner capability derivation: `query/plan/covering.rs`
- aggregate index-only count path: `executor/aggregate/index_count.rs`

### 7.5 Acceptance Criteria

- eligible covered shapes avoid primary row materialization.
- strict-missing and stale-key semantics remain parity-safe vs fallback.
- explain/trace/test surfaces explicitly mark covering/index-only activation.
- covering fast paths preserve `MissingRowPolicy` semantics under stale index
  entries (for example index key exists but row was deleted), including count
  parity vs fallback.

## 8. Track E - Predicate Pushdown Expansion

### 8.1 Goal

Expand predicate pushdown coverage now that predicate canonicalization is
deterministic.

### 8.2 Pushdown Targets

| Operator Shape | Pushdown Target |
|---|---|
| `IN (...)` | index multi-point scan |
| `BETWEEN` | index range scan |
| `IS NULL` | prefix or nullable-bound scan |
| `LIKE 'prefix%'` | prefix range scan |

### 8.3 Ownership and Module Plan

- planner pushdown derivation: `query/plan/predicate_pushdown.rs`
- access routing consumes canonical planner-derived pushdown contracts

### 8.4 Acceptance Criteria

- planner emits explicit pushdown vs fallback reason paths.
- equivalent canonical predicates map to stable pushdown decisions.
- runtime parity holds for pushed vs non-pushed execution on all guarded cases.

## 9. Track F - Multi-Key `IN` Index Routing

### 9.1 Problem

Multi-key equality windows can degrade into repeated single-key probes.

### 9.2 Goal

Route `IN` equality sets to one batched access plan.

Example:

`SELECT * FROM users WHERE id IN (1,2,3,4)`

Planner target:

`ByKeysAccessPlan`

Execution target:

`index/by_keys_stream.rs`

### 9.3 Acceptance Criteria

- planner emits one canonical by-keys access shape for eligible `IN` windows.
- executor processes batched key lookups without repeated per-key replanning.
- duplicate-key and ordering semantics remain deterministic and tested.

## 10. Track G - Ordered Partial Index Scan Fast Path

### 10.1 Goal

Improve planner detection where `ORDER BY` already matches index order so sort
materialization can be bypassed.

Example:

`SELECT * FROM events ORDER BY created_at LIMIT 10`

Target flow:

`IndexRangeScan(created_at) -> early stop`

### 10.2 Acceptance Criteria

- eligible ordered-limit windows avoid redundant sort stages.
- mixed-direction and tie-break semantics remain stable.
- cursor continuation boundaries remain parity-safe under early-stop windows.
- planner verifies deterministic tie-break stability before sort bypass (for
  example `ORDER BY created_at` is only sort-bypass eligible when index order
  guarantees deterministic tie handling, such as `(created_at, id)` with stable
  PK tie-break projection).

## 11. Track H - EXISTS / NOT EXISTS Early Termination

### 11.1 Goal

Avoid full-scan behavior for existence terminals when first-hit (or confirmed
empty) semantics allow early stop.

### 11.2 Ownership and Module Plan

- planner terminal routing: `query/plan/exists.rs`
- executor terminal mechanics: `executor/terminal/exists.rs`

### 11.3 Acceptance Criteria

- `EXISTS` stops after first matching row on eligible shapes.
- `NOT EXISTS` uses safe anti-existence routing with deterministic fallback.
- terminal parity and scan-budget behavior are explicitly regression-tested.
- projection evaluation must not occur before early termination in
  `SELECT EXISTS(...)` paths.

## 12. Track I - Boolean Predicate Simplifier

### 12.1 Goal

Add a canonical boolean simplifier before planning to reduce contradictions and
redundant guards.

Examples:

- `a = 1 AND a = 1 -> a = 1`
- `a = 1 AND a = 2 -> FALSE`
- `a > 5 AND a > 3 -> a > 5`

### 12.2 Ownership and Module Plan

- simplification authority: `predicate/simplify.rs`
- planner consumes simplified form for routing and pushdown decisions

### 12.3 Acceptance Criteria

- simplification is semantics-preserving and deterministic.
- contradiction folding and redundancy elimination are covered by regression
  tests.
- pushdown eligibility improves for simplified equivalent predicates.

## 13. Track J - LIMIT 0 Fast Path

### 13.1 Problem

Queries with `LIMIT 0` can still traverse planning/execution paths that should
be unnecessary.

Examples:

- `SELECT * FROM users LIMIT 0`
- `SELECT COUNT(*) FROM users LIMIT 0`

### 13.2 Goal

Emit a zero-window terminal plan before access routing and return immediately.

### 13.3 Ownership and Module Plan

Planner:

- `query/plan/limit_zero.rs`

Executor:

- `executor/terminal/empty.rs`

Candidate contract forms:

- `ExecutionPlan::EmptyResult`
- or `AccessWindow { lower_bound: None, upper_bound: None, fetch_limit: 0 }`

### 13.4 Acceptance Criteria

- planner detects `LIMIT 0` before access routing.
- executor performs no scan for zero-window plans.
- cursor semantics remain deterministic.
- EXPLAIN clearly reports an empty terminal node (for example `EmptyResult`).

### 13.5 Estimated Cost

- approximately `40-60 LOC`

## 14. Track K - Constant Predicate Folding

### 14.1 Problem

Constant predicates can still flow through normal pushdown and access planning
despite deterministic outcomes.

Examples:

- `WHERE 1 = 1`
- `WHERE 1 = 0`
- `WHERE FALSE`

### 14.2 Goal

Fold constants early:

- constant `TRUE` -> predicate removed
- constant `FALSE` -> empty-result terminal

### 14.3 Ownership and Module Plan

- planner folding pass: `query/plan/constant_predicate.rs`
- hook at predicate canonicalization stage before pushdown analysis

### 14.4 Acceptance Criteria

- constant predicates fold before pushdown analysis.
- provable-false predicates bypass access planning and produce empty plans.
- EXPLAIN shows explicit constant-false folding (for example
  `PredicateConstantFalse`).

### 14.5 Estimated Cost

- approximately `50-80 LOC`

### 16.4 Acceptance Criteria
