# 0.114 Prepared Semantic Authority Collapse

## Thesis

`0.114` should remove prepared fallback as a parallel semantic authority and
collapse prepared execution back onto planner-owned semantic outputs.

This is not a prepared SQL widening slice.

It is not a template-lane expansion slice.

It is not a route/planner redesign slice.

It is a structural contraction slice whose job is to make the prepared path
stop deciding what expressions mean and reduce it to what prepared execution
should actually own:

* parameter binding
* value substitution
* execution wiring
* binding-compatibility checks against already-owned semantic contracts

The architectural target is:

* lowering owns expression structure
* planner owns semantic typing and canonical expression/predicate shape
* prepared execution consumes those outputs without re-deriving them

If [prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs:21)
and [parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs:306)
continue to act like a second semantic engine, prepared SQL will remain the
clearest cross-cutting seam in the system even if public behavior stays
correct.

---

## Why `0.114` Exists

The post-`0.111` audit set now converges on the same seam.

### 1. Completeness Audit

The completeness audit in
[completeness.md](/home/adam/projects/icydb/docs/audits/reports/2026-04/2026-04-21/completeness.md:1)
still classifies prepared SQL as `Partial`.

That is no longer mainly about missing product surface.

It is about architectural duplication:

* prepared SQL is behaviorally strong
* fallback typing and contract reasoning remain too local to the prepared lane

That means the limiting factor is ownership shape, not external feature reach.

### 2. Canonical Semantic Authority Audit

The canonical semantic authority audit in
[canonical-semantic-authority.md](/home/adam/projects/icydb/docs/audits/reports/2026-04/2026-04-21/canonical-semantic-authority.md:1)
reported:

* no missing canonical typed models
* no raw-string semantic reparse authority
* one active owner-drift seam: prepared fallback typing

`0.114` is therefore not a “fix broken semantics” slice.

The semantics are already canonical and converged.

The issue is that prepared fallback remains large enough to become an
expensive drift seam if it keeps growing.

### 3. Complexity Accretion Audit

The complexity audit in
[complexity-accretion.md](/home/adam/projects/icydb/docs/audits/reports/2026-04/2026-04-21/complexity-accretion.md:1)
shows branch pressure worsening and reconcentrating around the expression
pipeline cluster, including:

* `db::session::sql::parameter`
* `db::sql::lowering::prepare`
* `db::query::plan::expr::ast`
* `db::query::plan::expr::type_inference`
* `db::predicate::bool_expr`

This is not fanout-led architecture sprawl.

It is branch-density growth in adjacent semantic owners.

That makes `0.114` not just a semantic-owner cleanup. It is also a structural
contraction target.

### 4. DRY Consolidation Audit

The DRY audit in
[dry-consolidation.md](/home/adam/projects/icydb/docs/audits/reports/2026-04/2026-04-21/dry-consolidation.md:1)
identifies one dominant authority-contraction target:

* [prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs:21)
* [parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs:306)

The duplication type is not cosmetic boilerplate.

It is evolution-drift duplication: semantic-adjacent reasoning duplicated
across lowering and session binding.

That makes `0.114` the correct next slice because all four audits point to the
same owner boundary.

## Current Problem

Prepared fallback has improved behaviorally, but it still carries too much
local reasoning.

Recent work already fixed the important prepared `WHERE` / prepared `HAVING`
behavioral split:

* expression-owned `WHERE` and `HAVING` now both admit fallback contract
  collection across the admitted expression family
* grouped symbolic template lanes still remain explicitly restricted to the
  compare-family grouped path
* expression-owned grouped `HAVING` shapes stay on fallback rather than
  silently widening template admission

That was the correct first move.

What remains now is the structural seam underneath that behavior:

* [prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs:21)
  still owns too much contract-collection and family-sensitive reasoning
* [parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs:306)
  still owns too much rebinding and expression/predicate walking that is
  semantically adjacent to planner-owned truth
* both files are now large enough that the prepared lane risks becoming a
  parallel expression subsystem

The bad shape is no longer a correctness split between `WHERE` and `HAVING`.

The bad shape is:

* semantic truth is planner-owned in principle
* prepared fallback still locally reenacts too much of it in practice

---

## Core Invariant

Prepared execution must not introduce or own any semantic truth that is not
already produced by lowering or planner stages.

If a semantic decision exists both in planner-owned expression typing /
canonicalization and inside the prepared path, planner-owned logic must be the
single authority.

Prepared traversal is allowed.

Prepared semantic reinterpretation is not.

---

## Goal

Collapse prepared fallback onto planner-owned semantic outputs so the prepared
path becomes visibly structural-only.

Concretely:

* [prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs:21)
  must stop acting like a semantic decision owner
* [parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs:306)
  must stop acting like a semantic reconstruction owner
* prepared fallback must consume planner-owned structure and family truth
* parameter binding must preserve canonical semantic form rather than
  reshaping or rediscovering it

The target is not “less code” as such.

The target is:

* no owner-local semantic truth in the prepared lane

---

## Ownership Boundary

`0.114` should make this split explicit in code shape, not just in comments.

### Lowering Owns

* SQL parse/lower structure
* admitted expression shape
* lane selection inputs that are syntactic or structural

### Planner Owns

* semantic typing
* compare-family truth
* result-family truth
* canonical predicate / expression shape
* the semantic contracts that prepared execution consumes

### Prepared Layer Owns

* runtime parameter binding
* value substitution
* contract-conformance checks against already-defined semantic contracts
* template instantiation where already admitted
* execution wiring

Prepared-layer checks may validate whether runtime values satisfy an existing
parameter contract.

They must not independently derive:

* expression meaning
* predicate meaning
* compare-family semantics
* CASE-family semantics
* dynamic function-family semantics

Prepared execution may traverse planner-owned structures to attach runtime
values.

This traversal must stay structural only and must not alter or reinterpret
semantic meaning.

## Concrete Actions

### 1. Map prepared semantic-adjacent ownership

Inventory [prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs:21)
and [parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs:306)
for logic that still performs or implies semantic reasoning, especially:

* family inference
* expression-tree meaning walks
* predicate-like interpretation
* compare-family reconciliation
* CASE-family reconciliation
* function-family reconciliation
* duplicated `WHERE` / `HAVING` collector structure that now shares one
  behavioral boundary

For each helper, classify it as:

* planner-owned semantic truth
* prepared-boundary policy
* mixed logic that must be split

### 2. Push semantic truth back behind planner-owned outputs

Where prepared fallback still depends on owner-local family or meaning
decisions, move that truth to planner-owned outputs or planner-owned helpers.

The prepared lane should consume:

* already-typed structure
* already-owned family truth
* already-owned canonical shape

Not rediscover them.

### 3. Thin `prepare.rs`

After refactoring, [prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs:21)
should look primarily like:

* contract collection from planner-owned shapes
* lane gating
* binding preparation
* fallback/template routing
* execution handoff

It should not read like a second type-inference or semantic reconciliation
engine.

### 4. Thin `parameter.rs`

After refactoring, [parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs:306)
should primarily own:

* binding validation
* value substitution
* structural instantiation of already-admitted contracts

It should not carry its own meaning layer for expressions or predicates.

### 5. Collapse aligned collector structure

Now that prepared `WHERE` and prepared `HAVING` are aligned behaviorally on the
fallback contract boundary, reduce the remaining duplicated collector structure
where safe.

This does not mean erasing the grouped template guard.

The distinction that must remain visible is:

* compare-family grouped `HAVING` may still use the symbolic grouped template
  lane
* general expression-owned grouped `HAVING` must remain fallback-only

So the correct contraction is:

* shared fallback collection model
* distinct lane-admission guard where intentionally required

### 6. Freeze the boundary with tests

Tests are part of the slice, not optional follow-up.

The required proof targets are:

1. prepared and non-prepared equivalent queries preserve the same canonical
   semantic plan shape
2. prepared fallback does not reinterpret semantic meaning locally
3. parameter binding does not alter canonical form
4. expression-owned grouped `HAVING` fallback stays off template lanes
5. grouped compare-family `HAVING` retains the explicit template-lane guard
6. structural unification of `WHERE` / `HAVING` fallback collection does not
   widen admitted template behavior

## Recommended Work Order

### 1. Stabilize the new behavioral boundary

Start from the already-landed `WHERE` / `HAVING` alignment:

* both admit expression-owned fallback contract inference
* grouped template lanes remain explicitly compare-family-only

Treat that as the fixed external contract.

### 2. Identify remaining duplicated semantic reasoning

Map the helpers in:

* [prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs:21)
* [parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs:306)

that still reenact semantic meaning rather than consume planner-owned outputs.

### 3. Move truth upward, not sideways

Do not solve this by creating a new generic “prepared shared utils” semantic
layer.

The correct direction is:

* semantic truth moves upward into planner-owned outputs/helpers
* prepared code becomes thinner

### 4. Unify aligned fallback collection

Reduce duplicated `WHERE` / `HAVING` collection structure now that the
behavior is aligned, while keeping grouped template gating explicit.

### 5. Freeze with invariants and parity tests

Treat tests as the actual contract that prevents the seam from reappearing.

## Exit Condition

`0.114` is done when:

* prepared fallback no longer depends on owner-local semantic reasoning in
  [prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs:21)
  or [parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs:306)
* prepared execution consumes planner-owned semantic outputs without
  reinterpretation
* [prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs:21)
  is visibly reduced to contract collection, lane gating, binding setup, and
  execution handoff
* [parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs:306)
  is visibly reduced to structural binding/substitution and compatibility
  validation against existing contracts
* aligned prepared `WHERE` / `HAVING` fallback collection no longer appears as
  duplicated semantic-adjacent ladders
* prepared and non-prepared equivalent queries preserve the same canonical
  semantic plan shape
* grouped template admission remains explicitly bounded and does not widen as
  a side effect

Architecturally, this should move prepared SQL from:

* behaviorally strong but architecturally `Partial`

toward:

* behaviorally strong and structurally `Bounded`

in the completeness model.

It should also reduce:

* the prepared fallback drift seam in CSA
* the prepared/lowering branch-pressure hotspot cluster in complexity
  accretion
* the highest-value evolution-drift duplication pair in DRY consolidation

## Non-Goals

`0.114` should not:

* widen prepared template admission
* widen prepared fallback admission beyond the shipped expression family
* add new parameter-family product surface
* redesign prepared execution from scratch
* broaden grouped semantic families
* redesign route/ranking
* turn into a general parser/lowering contraction slice
* use “DRY” as justification for flattening boundary-protective duplication

---

## Summary

`0.114` is the first architectural contraction slice after the recent
semantic expansion lines.

Its job is not to add features.

Its job is to remove the only cross-cutting seam that now shows up
consistently across the audits:

* completeness: prepared SQL still reads `Partial`
* canonical semantic authority: prepared fallback typing is the active owner
  seam
* complexity accretion: the prepared/expression cluster is a branch hotspot
* DRY consolidation: prepared fallback is the highest-value authority
  contraction target

The intended end state is simple:

* planner owns semantic truth
* prepared execution consumes it
* the prepared lane stops behaving like a second semantic engine
