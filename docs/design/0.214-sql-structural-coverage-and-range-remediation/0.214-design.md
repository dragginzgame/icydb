# 0.214 SQL Structural Coverage And Range Remediation

Status: proposed

Design cut: 2026-07-21

Implementation order: after 0.213 closeout

## Decision Summary

0.214 is a bounded successor to the completed 0.204 SQL correctness line and
the completed 0.206 SQL performance-remediation line. It does not reopen,
rewrite, or amend either design.

The line has two connected purposes:

1. expand SQL evidence by adding genuinely new typed query structures, schema
   strata, and feature interactions rather than multiplying seeds around the
   existing fixed templates; and
2. establish a fresh current-source performance baseline, close the missing
   attribution around compound range and membership planning, and hard-cut one
   proven duplicate compound-range traversal.

The coverage work is test-owned. It does not add SQL syntax or product
semantics. The production work is deliberately limited to one planner-owned
range-conjunction remediation. Large-membership and branch-prefix planning are
measured and classified in this line but are not automatically a second
production optimization.

0.214 starts only after 0.213 because the near-term roadmap is serial. Its
opening evidence must therefore represent the current accepted schema, row
envelope, constraint, integrity, and identity behavior after 0.209 through
0.213 rather than projecting the 0.206 runtime forward.

### Non-Negotiable Invariants

1. The 0.204 SQL contract manifest remains the sole current coverage
   authority. 0.214 extends its evidence; it does not create a second feature
   ledger.
2. `SQL_SUBSET.md` remains the authority for maintained accepted and rejected
   SQL behavior. Generated cases do not define product semantics.
3. Accepted schema snapshots are the sole runtime and generator input
   authority for entity, field, leaf, type, nullability, index, and write-policy
   facts.
4. Generated Rust models may construct test proposals but may not reconstruct
   runtime schema facts or fill gaps in an accepted snapshot.
5. An accepted generated case derives its exact feature set and structural
   signature from its validated typed AST, accepted snapshot, bound-value
   relationship class, and typed route/materialization facts where the frozen
   obligation requires them. A rejected case derives them from one validated
   base AST plus one typed invalid mutation and expected error kind.
   Family-wide labels are not copied onto every case.
6. More seeds, roots, fixtures, or rendered literals do not count as broader
   structural coverage when the typed query structure is unchanged.
7. Coverage obligations enumerate only supported, type-valid interactions.
   0.214 does not require an invalid Cartesian product of all 98 contract
   identifiers.
8. Required reference evidence fails closed. SQLite exclusions remain typed
   and explicit; an unavailable or ineligible reference never silently becomes
   passing evidence.
9. SQL/fluent parity is useful ingress evidence but is not presented as an
   independent semantic oracle when both paths consume the same planned
   execution artifact.
10. Generator-owned values remain exact, bounded value kinds. 0.214 does not
    recreate a universal runtime `AnyValue` or introduce a generic value
    transport at the write boundary.
11. Nested field-path evidence is derived from the accepted stored-leaf
    contract, including exact selectable/queryable facts. It is not inferred
    from a generated Rust nesting shape.
12. Mutation generation preserves authored value, omitted insert, explicit
    insert `DEFAULT`, explicit update `DEFAULT`, and preserved-existing intent
    as distinct facts until the accepted write-policy owner resolves them.
13. Rust `Default`, derived defaults, `Option::default()`, and deserialization
    defaults are never persisted-field default authorities in generated or
    production mutation evidence.
14. 0.214 does not generate randomized DDL, schema transitions, recovery
    schedules, or historical row layouts. Those remain separate state spaces.
15. The current performance profile is hard-cut to one new identity before
    production optimization. Old 0.204 and 0.206 bundles remain historical
    evidence and are not decoded through compatibility paths.
16. Cold compilation, warm prepared execution, planner work, store work,
    executor work, response finalization, output bytes, and exact structural
    counters remain distinct performance dimensions.
17. Peak retained items are not peak heap bytes. No allocator, heap-byte, or
    stable-memory-byte claim is made without a reliable typed measurement.
18. Compound-range extraction may combine bounds only when compatibility is
    proved from canonical planned facts. It never factors through `OR`, `NOT`,
    incompatible expressions, collations, prefixes, or null semantics.
19. Every non-extracted conjunct remains an exact residual predicate. A field
    comparison such as `name = name` is not discarded as a tautology because
    SQL null semantics can make it unknown.
20. An eligible lower-and-upper range conjunction produces one physical
    bounded range traversal. Scalar and grouped consumers do not retain
    separate implementations of that fusion rule.
21. Results, ordering, ties, null behavior, DISTINCT behavior, grouping,
    HAVING, cursors, lookahead, offsets, limits, and error semantics remain
    unchanged.
22. Bounded public-read admission, scan budgets, group limits, DISTINCT limits,
    output limits, and fail-closed corruption behavior remain unchanged.
23. The selected production owner is replaced directly. No feature flag,
    scenario dispatch, route hint, old/new fallback, or compatibility wrapper
    retains the superseded qualifying traversal.
24. A cost reduction cannot be accepted by moving work into an unmeasured
    phase or by deleting an expensive maintained scenario.
25. Every format or algorithm cut lands atomically with deletion of its
    superseded current owner. No slice leaves dual generator versions, replay
    decoders, profile selectors, or qualifying range algorithms.
26. 0.214 extends the existing SQL generator, coverage manifest, scheduler,
    replay/shrinker, and performance harness. It does not add another crate,
    harness, SQL AST, provider registry, plugin interface, or configuration
    layer.
27. The derived structural signature is the only new cross-cutting coverage
    abstraction. It is computed from the existing typed AST and is not stored
    beside that AST as independently authored state. Obligation IDs and
    required-witness declarations extend existing manifest/scheduler records;
    they do not form another coverage model layer.
28. The interaction ledger is one static extension of the existing manifest,
    not a graph engine, database, solver, or runtime registry.
29. 0.214 adds exactly one SELECT schema profile and one mutation schema
    profile beyond the maintained profiles. Further profiles require an
    explicit design revision, not an implementation-time convenience choice.
30. Production changes are limited to owner-local range planning/access and
    diagnostic plumbing. Test coverage work must not require production DTOs,
    public APIs, executor wrappers, or broad module reorganization.

## Normative Contract

The coverage contract is:

> A scheduled generated declaration is evidence for its exact typed structure,
> accepted schema profile, fixture properties, provider contract, and observed
> execution facts. Repeating the declaration with different seed material
> increases data variation but does not create a new structural obligation.

The interaction contract is:

> Maintained SQL interactions are reviewed as type-valid obligations. Every
> required interaction has generated evidence, deterministic evidence, or one
> explicit inapplicability/exclusion reason. Absence is never inferred to be
> covered because both features appear somewhere independently.

The performance contract is:

> 0.214 establishes an opening baseline on the exact post-0.213 source and
> profile before changing production planning. Candidate and final evidence
> use that same profile identity, fixture identity, toolchain, feature set, and
> measurement contract.

The range-planning contract is:

> A conjunction containing compatible lower and upper constraints for one
> index domain may lower to one intersected physical range plus the complete
> residual conjunction. Each qualifying index entry is traversed at most once
> by that access range. Ineligible conjunctions continue through the canonical
> maintained planner without an unsafe approximation.

## Audited Baseline

### Correctness Evidence

The completed 0.204 line provides a strong evidence foundation:

- 98 stable SQL contract feature identifiers;
- 93 reachable deterministic providers;
- exact provider/evidence obligations and reference exclusions;
- typed valid and singly-invalid SELECT generation;
- independent mutation state modeling;
- deterministic replay and bounded shrinking;
- fail-closed scheduled correctness receipts; and
- one current checked-in regression corpus format.

The scheduled Tier C catalog contains 2,505 top-level declarations:

| Declaration family | Count |
| --- | ---: |
| Valid generated SELECT | 2,048 |
| Invalid generated SELECT | 320 |
| Generated mutation sequence | 128 |
| Deterministic SQLite-reference scenario | 8 |
| Reviewed regression corpus entry | 1 |

Those counts are not 2,505 independent SQL structures. The 0.204 breadth audit
records:

- eight valid SELECT families selecting from eight handwritten structural
  slots, for 64 valid template slots;
- five invalid SELECT violation templates;
- four structural mutation-sequence variants, each containing eight
  statements;
- independent roots and sub-seeds varying fixtures, values, literals, and some
  eligible choices around those structures;
- one generated SELECT snapshot containing text, two signed integers, boolean,
  nullable text, and a generated ULID primary key;
- no secondary indexes in that generated snapshot; and
- deterministic rather than generated ownership for stored defaults, managed
  fields, blobs, composite keys, secondary-index families, and policy behavior.

This is good contract evidence but incomplete interaction evidence. The
generator can prove that one feature appears in an exact template. It does not
yet systematically prove that every reviewed supported interaction has been
exercised with the relevant schema, nullability, route, and materialization
facts.

### Performance Evidence

The accepted 0.206 profile contains:

- 1,787 fully enumerated P1 scenarios;
- 72 scale scenarios at reviewed cardinalities;
- 424 P2 confirmations in the accepted final cohort;
- 15 focused-hotspot scenario identities;
- 351 regression sentinels;
- ten grouped contract sentinels;
- separate cold and eligible warm samples; and
- typed compile, planner, store, executor, response, route, row, output, and
  selected peak-state facts.

The 0.206 leading set contained fourteen owned findings:

| Final disposition | Count | Meaning |
| --- | ---: | --- |
| Completed `OptimizeNow` | 1 | Bounded expression-order retention was hard-cut |
| `NeedsTypedMeasurement` | 7 | Evidence identifies an expensive family but not exact repeated work |
| `ContractRequired` | 4 | Current work is required for exact semantics/resource state |
| `SchemaDependent` | 2 | Removing the scan requires an application-declared compatible index |

The completed optimization reduced retained expression-order candidates from
2,048 to 11 at the largest scale point. It demonstrated that a general
owner-level change can materially improve one family without changing SQL
semantics, but it did not make the entire executor cheap.

The strongest unresolved execution candidate is shared by the first and
second ranked scale families. A conjunction containing lower range, upper
range, and a non-range residual is currently capable of becoming two one-sided
range children. The grouped hash consumer then observes 4,096 index entries to
obtain 2,048 rows. Existing evidence does not isolate the instruction cost of
each child traversal.

Five cold compile/planner findings are also unresolved:

- sparse membership page construction;
- sparse membership count lowering;
- over-cap exclusion pruning;
- wide non-covering branch construction; and
- wide covering branch construction.

They share SQL binding, predicate normalization, or prefix-planner ownership,
but existing metrics do not count canonicalization passes, members revisited,
branches built, or exclusions reconsidered. They remain candidates, not proven
duplicate work.

Peak heap bytes, allocator traffic, and stable-memory byte volume remain
explicitly unmeasured. This design does not reinterpret instruction or retained
item counters as those dimensions.

### Executor Architecture

The 0.206 addendum concludes that the executor is fundamentally coherent but
locally overcomplicated. Its focused consolidation slices are implemented,
while comparable instruction evidence for the consolidated current shape is
still required through the ordinary performance workflow.

0.214 therefore assumes:

- SQL and fluent entrypoints should continue converging after planning;
- ordered and hash grouping remain genuinely different physical strategies;
- covering, hybrid, and full materialization remain meaningful paths;
- bounded heaps and `PendingOrderRows` remain invariant-bearing; and
- future performance work should target one demonstrated owner rather than
  introduce a new executor framework.

## Problem Statement

### 1. Scheduled Volume Masks Structural Repetition

Tier C executes thousands of declarations, but most scheduled SELECTs revisit
64 structural slots and most mutation sequences revisit four structural
variants. Seed diversity is useful for values and fixtures; it is not a
substitute for new typed query composition.

### 2. Feature Presence Does Not Prove Feature Interaction

The manifest proves that every contract cell has evidence. It does not yet
provide one reviewed ledger for required interactions such as:

- nullable ordering combined with computed projection and bounds;
- DISTINCT combined with computed projection and post-projection order;
- indexed filtering combined with incompatible expression ordering;
- aggregate `FILTER` combined with aggregate `DISTINCT` and empty input;
- grouped aggregation combined with HAVING, ordering, and pagination; or
- accepted nested leaves combined with filtering, ordering, projection, and
  mutation `RETURNING`.

Some of these combinations already have deterministic tests. The gap is that
their obligation and exact evidence are not uniformly derived and reviewed.

### 3. The Generated Schema Universe Is Too Narrow

The initial generated SELECT snapshot deliberately stayed inside a small
SQLite overlap and contains no secondary index. It cannot independently vary
schema shape, route eligibility, exact stored-leaf selection, or materialized
versus covering behavior.

The mutation generator likewise cannot demonstrate broad write-intent,
default, structured-leaf, composite-key, and derived-state interactions by
repeating its current authored key/text/numeric sequence.

### 4. Invalid Generation Covers Only Five Violation Shapes

The current singly-invalid lane is precise and shrinkable, but it samples five
failure classes. It does not systematically cover the reviewed binder,
aggregate-legality, grouping, alias, accepted-leaf, or mutation-authorship
rejections present in the maintained contract.

### 5. The Performance Baseline Will Be Stale At 0.214 Start

0.209 changes the row envelope and decode contract. Later lines change current
accepted constraints, integrity surfaces, and generated identity behavior.
The executor audit addendum also records structural simplifications after the
selected 0.206 baseline. A production optimization cannot be judged against a
mixture of those source shapes.

### 6. Compound Range Planning Has Suspicious Duplicate Traversal

The leading grouped hash range families perform two one-sided index traversals
for a conjunction whose lower and upper bounds target the same index domain.
Hash grouping is required; visiting the bounded range twice is not known to be
required.

The planner must retain the non-range residual and SQL three-valued logic. The
optimization opportunity is one access-range intersection, not deletion of
semantic predicate work.

### 7. Membership And Prefix Planning Lack Eliminated-Work Counters

The cold costs are large, but total instructions alone cannot distinguish
necessary member handling from repeated normalization, branch construction, or
pruning. Optimizing these paths before measuring the work would repeat the
mistake 0.206 was designed to prevent.

## Goals

0.214 must:

1. audit exact structural and interaction coverage against the post-0.213
   contract manifest, which is expected to contain 98 identifiers; any count
   drift requires design review before Slice 1;
2. introduce one derived structural-query signature that ignores seeds,
   literals, fixture values, and rendered names;
3. classify every reviewed supported interaction as generated-required,
   deterministic-required, or inapplicable with an exact reason;
4. replace fixed-slot valid SELECT generation with bounded typed composition;
5. expand generated SELECT evidence across reviewed schema, nullability,
   index, route, and fixture strata, while binding stored-leaf interactions to
   exact deterministic providers;
6. expand singly-invalid generation across reviewed current-contract rejection
   families without creating multi-fault ambiguity;
7. replace four fixed mutation sequence structures with bounded,
   provenance-aware state-machine composition over the maintained authored
   profile and one default-aware profile;
8. preserve independent oracles, exact provider eligibility, replay, shrinking,
   and deterministic budgets;
9. hard-cut old generator/replay/profile formats rather than add version
   compatibility;
10. establish one fresh performance profile and opening baseline on exact
    post-0.213 source before changing production planning;
11. add the minimum typed attribution needed to prove or reject duplicate
    compound-range traversal;
12. freeze an eliminated-work and instruction budget before implementing the
    range change;
13. replace eligible duplicate one-sided traversal with one canonical bounded
    range and exact residual evaluation;
14. measure and classify the five membership/prefix cold-planning families
    without silently turning them all into implementation work;
15. preserve every public semantic, resource, corruption, cache, diagnostic,
    and persistence contract;
16. close with current correctness, performance, raw non-gzipped wasm, and
    source-complexity evidence; and
17. leave one explicit findings ledger for measured work not selected in this
    line.

- become a broad executor, planner, or test-framework rewrite.

## Hard-Cut And Minimal-Shape Rules

The implementation shape is fixed before Slice 0 begins:

- extend `testing/sql-generator` and the existing integration SQL harness in
  place;
- derive one structural signature directly from the current typed AST;
- add one static interaction table under the existing coverage-manifest
  authority;
- add one indexed/nullable SELECT snapshot beside the maintained reference
  snapshot;
- add one default-aware mutation snapshot beside the maintained authored
  scalar snapshot;
- use the existing provider, scheduler, replay, shrink, receipt, P1, P2, scale,
  and comparison owners;
- add only diagnostics counters consumed by the range or membership finding;
  and
- change exactly one production algorithm: compatible compound-range
  extraction/traversal.

The following shapes are forbidden:

- a generic query-combinator framework separate from the existing typed AST;
- a second manifest, feature graph, or coverage persistence format;
- a transport DTO that copies the AST only to rename its fields;
- a stored signature plus validation needed only because the signature was
  separated from its AST;
- optional booleans or empty collections encoding generator modes;
- a provider trait with one real implementation;
- dynamic registration or feature discovery;
- configuration flags selecting old/new generator or planner behavior;
- a test-only public production API;
- a scan- or grouped-specific wrapper around the executor-wide range plan; and
- abstractions for joins, distributed execution, cost planning, or other
  unsupported futures.

The hard cuts are exact:

1. the compositional SELECT generator replaces the fixed 64-slot current
   profile in one landing unit; the old dispatch, current generator version,
   superseded replay decoder, stale golden fixtures, and slot-only helpers are
   deleted or regenerated in that unit;
2. the current eight-family SELECT taxonomy is not retained as a scheduler,
   quota, seed, replay, report, or feature authority: `SelectGeneratorFamily`,
   `ALL_SELECT_GENERATOR_FAMILIES`, the per-family Tier A/Tier C quotas, and
   family-level `contract_features()` are deleted with the fixed profile;
   semantic statement/result shapes remain facts derived from the typed AST,
   and scheduled witness identity supplies deterministic partitioning;
3. the compositional mutation generator replaces the four fixed current
   sequence variants in one landing unit; `MUTATION_STRUCTURAL_VARIANT_COUNT`
   and fixed-family seed/replay identity are deleted rather than translated
   into optional legacy fields;
4. the new performance profile replaces the previous current profile in code;
   old evidence remains immutable documentation, not a selectable profile or
   decoder; and
5. the fused compound-range owner replaces the qualifying two-child branch in
   one landing unit; the old branch cannot remain as a fallback or diagnostic
   comparison path, and no separate literal, parameter, cold-cache,
   warm-cache, or access-cursor finalizer survives.

Current-format tests prove the maintained surface. Source inspection proves
the old implementation is absent. 0.214 does not retain tests whose sole
purpose is preventing resurrection of a removed pre-1.0 path.

## Coverage Authority

### One Derived Structural Signature

Every generated declaration has one canonical, lossless, comparable structural
signature derived from its validated test-owned declaration. The full closed
representation is coverage authority; a digest may identify a report or
artifact but cannot replace the full signature.

The signature owns only facts that distinguish semantic query structure:

- accepted or singly-invalid declaration kind;
- schema profile identity;
- statement and result-shape family;
- projection expression shapes and result kinds;
- aggregate functions and `DISTINCT`/`FILTER` modifiers;
- predicate tree shape, operator families, operand roles, and maximum depth;
- grouping-key shapes;
- HAVING shape;
- order target shapes, directions, null-sensitive participation, and
  deterministic tie-break ownership;
- limit/offset form; and
- exact singly-invalid violation and expected typed error kind where present.

Every referenced field or stored leaf carries abstract accepted-schema roles,
not its authored name. The closed role description includes, where applicable:

- sole or composite primary-key position;
- single-column secondary-index participation;
- composite-index identity and prefix position;
- indexed versus non-indexed access role;
- nullable versus non-nullable storage;
- stored-leaf identity, selectability, and queryability;
- projection availability from the chosen index; and
- pure-covering, hybrid-covering, or non-covering participation.

One field may own several roles. The representation keeps the closed role set
together; it does not split it into parallel booleans or infer it later from a
field name.

Schema roles and route roles have separate derivation points. Field/index/
nullability/leaf capabilities derive before execution from the AST and accepted
snapshot. Required access and covering participation come from the frozen
structural obligation, while the observed access and covering roles derive
from the compiled plan and execution receipt. The two use the same closed role
vocabulary and must match; neither is inferred from SQL text.

Literal and bound values are abstracted only through an exact semantic class.
The initial closed representation retains:

- literal versus prepared-parameter source and parameter-slot identity;
- lower-only, upper-only, point, nonempty bounded, equal-open, crossed, and
  null-bound range classes;
- inclusive/exclusive bound relationships;
- membership authored count, distinct count, duplicate presence, null
  presence, and empty/nonempty class; and
- empty, ordinary, minimum, maximum, and boundary-adjacent prefix classes.

Therefore `x > 10 AND x < 5` and `x > 5 AND x < 10` cannot share one
signature. Payloads may be omitted only after their complete planner-relevant
relationship class has been retained.

It deliberately excludes:

- root seed and sub-seed;
- case index;
- literal payloads when their semantic class is unchanged;
- fixture row values;
- current entity and field spellings; and
- rendered SQL whitespace.

The signature is derived, not authored beside the AST. Coverage reports group
by it so reviewers can distinguish structural breadth from repeated data
variation. The implementation should use one invariant-bearing type, not a
tuple plus parallel labels or optional fields. Replay artifacts retain the AST
and accepted snapshot as authority and do not serialize a redundant signature;
reports may serialize the signature derived during validation.

Canonical encoding is lossless and has deterministic equality and ordering.
Validation proves canonical round-trip and expected ordering; it does not use
probabilistic collision tests. If an artifact includes a digest, it also
retains or resolves to the full signature and verifies the digest against it.

### Reviewed Interaction Ledger

0.214 extends the existing manifest owner with a bounded interaction ledger.
It is one static table using the existing manifest vocabulary, not a second
contract document or generic relation engine. Each entry names:

- the participating current contract identifiers;
- required accepted schema facts;
- required fixture properties;
- minimum evidence strength;
- provider eligibility;
- required route or materialization facts when those are contractual;
- one of `GeneratedRequired`, `DeterministicRequired`, or `Inapplicable`; and
- an exact reason for deterministic-only or inapplicable classification.

The frozen catalog covers these interaction groups:

| Group | Required dimensions |
| --- | --- |
| Scalar composition | projection, computed expression, predicate, DISTINCT, order, limit, offset |
| Null semantics | stored/computed null, comparison, membership, DISTINCT, aggregate, ordering |
| Indexed scalar execution | primary/secondary access, residual, compatible/incompatible order, covering mode |
| Global aggregation | predicate, empty input, `DISTINCT`, `FILTER`, HAVING, multiple projections |
| Grouped aggregation | ordered/hash mode, multiple groups, HAVING, aggregate order, window |
| Field paths | accepted stored leaf, selectability, predicate, projection, order, RETURNING |
| Mutation | insert/update/delete, intent provenance, RETURNING, bounded order, rejection atomicity |
| Cache/entry parity | cold/warm, SQL/fluent, compiled/direct where eligible |

The ledger does not contain all mathematical pairs. It contains the complete
finite catalog of reviewed interactions the current product claims and can
meaningfully exercise at the Slice 0 freeze.

### Frozen Obligation Universe

Slice 0 writes one checked-in current-form catalog at:

`docs/design/0.214-sql-structural-coverage-and-range-remediation/0.214-coverage-obligations.json`

The file is the canonical lossless evidence projection of the one static
manifest extension, not a separately authored second ledger. The manifest
entries remain the code owner; generation of the JSON and exact comparison
with the checked-in bytes fail on drift. The scheduler consumes those same
entries and records the projected catalog hash. No runtime, replay, or public
format depends on this evidence artifact.

The catalog contains a finite, sorted set of stable
`InteractionObligationId` values and a finite, sorted set of
`RequiredStructuralObligation` declarations. Names are conceptual; the
implementation may reuse existing manifest/scheduler structs rather than add
transport wrappers.

The candidate universe is derived before generation from two bounded sources:

1. every post-0.213 manifest identifier whose contract kind or evidence
   requirement participates in an interaction; and
2. the exact closed axis-member sets for each interaction group in the table
   above.

Every manifest identifier receives either one or more obligation references or
an explicit `NoStructuralInteractionRequired` reason. Every type-valid tuple in
each closed group receives an obligation or an exact `Inapplicable` reason.
Static validation enumerates those frozen axis sets and fails on an omitted
tuple or manifest participation decision. It does not ask the generator which
tuples exist and does not implement a generic feature graph or solver.

Each interaction obligation records its complete disposition and every
contract identifier it references. Each required structural obligation names:

- one stable obligation ID;
- the interaction obligations it witnesses;
- one expected full structural signature;
- required fixture-property classes;
- required provider and evidence strength;
- required route/materialization facts when contractual; and
- its deterministic scheduled witness identity.

`RequiredStructuralObligation` and `ObservedStructuralSignature` are distinct
roles even when they use the same lossless signature representation. Required
obligations are frozen inputs. Observed signatures are evidence produced by
executing scheduled witnesses.

The required closure chain is:

~~~text
InteractionObligationId
    -> RequiredStructuralObligation
    -> scheduled witness
    -> ObservedStructuralSignature
    -> eligible provider
    -> passing evidence receipt
~~~

Unexpected observed signatures are discovery evidence only. They neither add
to nor satisfy the frozen required set. A required signature that was never
generated is a missing-evidence failure.

The canonical catalog hash is included in every scheduled correctness receipt.
Slice 1 cannot begin until the catalog is reviewed and checked in. Adding,

### Required SELECT Schema Profiles

The new generated lane has exactly two accepted-snapshot-derived profiles:

1. **Reference scalar** retains the current boolean, integer, text, nullable
   text, and generated primary-key overlap for stable SQLite comparison.
2. **Indexed nullable reference** adds nullable accepted fields, one
   single-field secondary index, and one compatible composite index while
   keeping result values inside the exact SQLite overlap. SQLite proves values;
   IcyDB diagnostics prove the typed route.

Accepted nested leaves, blobs, composite keys, and other non-reference scalar
strata remain `DeterministicRequired` under their exact existing providers.
0.214 strengthens and links those interaction obligations; it does not build a
second structured-value evaluator. A third generated profile requires an
explicit design revision. Profile identity is part of the structural signature
and replay artifact.

### Required SELECT Composition

The generator no longer selects one complete query from `case_index % 8`.
Instead, independent versioned sub-streams choose compatible typed axes under
bounded quotas:

- scalar, distinct, global aggregate, or grouped shape;
- projection expressions;
- predicate structure;
- grouping and aggregate modifiers;
- HAVING where legal;
- ordering and deterministic tie-break;
- limit and offset; and
- fixture property package.

Construction remains validity-first. It must not generate arbitrary fragments
and retry until parsing succeeds. A bounded compatibility table or typed
constructor prevents illegal combinations before rendering.

The scheduler consumes the frozen required structural obligations; it does not
derive its required universe from whatever the generator happened to emit. It
chooses the declared deterministic witness for every obligation and compares
the observed lossless signature with the required signature. Independent seeds
then vary literals and fixtures within those already-required structures.

### Invalid SELECT Generation

The test-owned declaration has two exact forms:

~~~rust
enum GeneratedDeclaration {
    Accepted(ValidatedSqlAst),
    Rejected(SinglyInvalidProposal),
}

struct SinglyInvalidProposal {
    valid_base: ValidatedSqlAst,
    violation: InvalidSqlViolation,
    expected: SqlErrorKind,
}
~~~

The spelling is conceptual and should reuse the existing generator AST and
typed error taxonomy. It does not authorize a second general SQL AST.

An accepted signature is derived from `ValidatedSqlAst`. A rejected signature
is derived from the validated base plus the one violation and expected typed
error. Grammar-invalid text is therefore never presented as a validated AST.
The invalid renderer owns only the exact typed mutation from the valid base.

Validation proves before rendering that:

- the base is a valid current-contract query;
- the mutation family is applicable to that base;
- exactly one classified violation is introduced;
- the expected error kind belongs to that violation; and
- no second mutation or free-form invalid text participates.

The invalid evidence ledger contains exact current rejection families from:

- grammar/clause ordering and bounded literal overflow;
- missing or non-selectable accepted fields/leaves;
- operator and function type mismatch;
- ambiguous or invalid alias binding;
- aggregate/non-aggregate projection legality;
- invalid grouping and HAVING references; and
- invalid order targets.

Every generated invalid case carries its expected typed rejection before SQL
is rendered. Rejections that require accepted-leaf or other deterministic
profiles stay with their exact deterministic provider rather than forcing a
third generated schema. Tests compare typed causes, never display strings.

### Mutation Profiles And State Composition

Mutation generation retains one independent state model and uses exactly two
profiles:

1. **Authored scalar**, preserving the current key/text/numeric profile;
2. **Accepted default**, containing nullable and non-nullable fields with
   constant accepted defaults and exact omitted/insert-`DEFAULT`/update-
   `DEFAULT` provenance.

The default-aware snapshot is frozen as:

| Field role | Accepted kind | Nullability | Write/default policy | Derived participation |
| --- | --- | --- | --- | --- |
| `id` | `Nat64` | non-null | caller-authored, required, no default | sole primary key |
| `name` | `Text` | non-null | caller-authored, required, no default | none |
| `tier` | `Text` | non-null | accepted constant default `"bronze"` | one non-unique secondary index |
| `score` | `Nat64` | non-null | accepted constant default `7` | none |
| `note` | `Text` | nullable | accepted null omission/default policy | none |

The snapshot has no relation, sanitizer, validator, generation policy,
management policy, structured field, or composite key. Duplicate-rich `tier`
values are a fixture property, not a schema fact.

For `tier`, `score`, and `note`, omitted insert, explicit insert `DEFAULT`, and
explicit update `DEFAULT` resolve to the table's accepted payload or null
policy. An absent update assignment means preserve existing and never invokes
that policy. Omitted `name` and `DEFAULT` for `id` reject with their exact
accepted write-intent errors.

Composite-key, structured-leaf, managed, and identity mutation interactions
remain with exact deterministic providers. 0.214 records them in the
interaction ledger but does not generalize the independent mutation model to
those domains. A third mutation profile requires an explicit design revision.

The operation and ingress matrix is frozen as:

| Obligation | Profile | Intent/operation | Required ingress |
| --- | --- | --- | --- |
| `mutation.authored.insert` | authored scalar | authored single/multi insert | SQL and typed |
| `mutation.authored.insert_query` | authored scalar | INSERT FROM QUERY with maintained key/number sources | SQL |

generation, and management callbacks. Those semantics stay with deterministic
providers unless a future independent model is designed. Identity generation,
historical fill, schema transitions, and resumable bulk execution likewise
retain their dedicated 0.209 through 0.213 evidence rather than being
approximated inside this state machine.

Sequences are composed from bounded typed operations and coverage obligations,
not selected from four fixed complete vectors. Required operations include:

- single and multi-row insert;
- authored-profile INSERT FROM QUERY with the maintained key/number sources;
- exact and compound update;
- bounded ordered update/delete;
- no-op mutation;
- RETURNING;
- explicit authored/default intent distinctions; and
- atomic rejection with unchanged state.

The state model derives affected counts, returned rows, and complete post-state

### Canonical Range Shape

Compilation and execution have distinct authority.

Compilation proves shape compatibility and produces one prepared conjunction
artifact containing:

- one canonical index identity and equality-prefix contract;
- every compatible lower and upper bound source as either a typed literal or a
  typed prepared-parameter slot, with exact inclusivity;
- the order direction and cursor-domain contract;
- the prepared residual program; and
- the complete residual predicate containing every non-extracted conjunct.

Compatibility requires the same indexed expression, prefix, value domain,
comparison semantics, and collation/casefold contract. Field-to-field
comparisons, computed or value-unstable expressions, incompatible prefixes,
and boolean shapes that cannot be represented as conjunction intersection
remain residual or use the canonical existing plan. Typed literal and prepared
parameter sources are eligible; arbitrary expression evaluation is not.

Prepared-cache identity contains index/prefix/order shape, literal-versus-slot
sources, parameter slot identities and kinds, comparator inclusivity, residual
program identity, and cursor-domain shape. It does not contain bound parameter
values. Reusing a warm prepared artifact therefore never reuses a prior
execution's finalized bounds.

After parameter binding and canonical cursor validation, one executor-owned
range finalizer:

1. reads each literal or parameter source exactly once;
2. validates the bound value against the prepared accepted kind;
3. chooses the strongest lower and strongest upper bound using exact canonical
   comparison, with strictness winning when equal values compete;
4. intersects an eligible cursor continuation bound with those query bounds;
5. detects null, equal-open, crossed, and otherwise empty ranges; and
6. emits exactly one finalized access state: `Empty` or `Bounded`.

Every eligible literal-only, parameterized, cold-cache, warm-cache, initial,
and same-access-domain continuation execution uses this finalizer. Compilation
may validate constant kinds and preserve literal sources, but it does not own
an independent literal-only strongest-bound shortcut. A warm prepared cache
reuses only the prepared source shape. Cursor routing owns cursor identity and
route validation; the finalizer is the only owner that intersects a validated
same-domain continuation bound with the prepared query bounds.

The names are conceptual, but the finalized state must be one
invariant-bearing enum rather than optional bounds plus an empty flag. `Empty`
performs zero physical range traversals, zero data gets, and zero residual
evaluations.

A null parameter used by a comparison bound follows current SQL three-valued
WHERE semantics and finalizes the conjunction to `Empty`. A value with an
invalid accepted kind fails at binding before access. Equal inclusive bounds
form a point range; equal bounds with either exclusive side and crossed bounds
are empty.

For a cursor whose canonical route uses the same index domain:

- ascending continuation contributes an exclusive lower bound after the last
  complete ordered key;
- descending continuation contributes an exclusive upper bound before the last
  complete ordered key;
- composite index components and the primary-key tie-break remain part of the
  complete continuation key; and
- equal query/cursor values retain the stricter bound.

A cursor owned by a different terminal/output domain is not converted into an
access bound; its current terminal semantics remain unchanged. Cursor identity,
schema, order, and direction validation still occurs before range finalization.

Finalization does not relower predicates, re-enter the logical planner, or
recompute index eligibility. The finalized range travels directly to physical
access, and the already-prepared residual program evaluates only rows admitted
by that range.

Scalar pages, scalar aggregates, and grouped execution consume the same
prepared/finalized range artifacts. No grouped-only fusion wrapper is added.

- aggregate `DISTINCT`, `FILTER`, empty input, and HAVING rejection;
- insert, insert-from-query, update, delete, RETURNING, no-op, bounded order,
  and atomic rejection;
- omission and exact `DEFAULT` provenance;
- heap and journaled stores where the production route differs;
- SQL/fluent and direct/compiled parity; and
- diagnostic/non-diagnostic semantic parity.

### Performance

The final evidence includes:

- exact opening, candidate, and final environment identities;
- full current P1 and scale execution;
- stable P2 cold and eligible warm samples;
- compile, planner, store, executor, response, and total instructions;
- grouped stream/fold/finalize and scalar aggregate attribution;
- exact index scans, entries, data gets, rows decoded, rows returned, and
  output bytes;
- compound-range and membership/prefix structural counters;
- normalized costs and adjacent-cardinality slopes;
- instrumentation overhead and phase reconciliation;
- selected-family eliminated-work budgets; and
- raw non-gzipped wasm bytes, with gzip only as secondary context.

### Complexity

The implementation has these hard complexity limits:

- no new workspace package or second SQL harness;
- one new cross-cutting derived coverage representation at most: the
  structural signature; owner-local closed types remain justified only when
  they make an invalid state unrepresentable;
- interaction entries reuse existing manifest/provider/evidence types;
- no new production type exists solely to transport diagnostic counters;
- no generic abstraction is accepted with only one meaningful implementation;
- no broad executor/planner re-export or module-layout cleanup is bundled;
- the range slice deletes the qualifying duplicate branch and does not add a
  second semantic dispatcher; and
- needing a third schema profile, a second production optimization, a generic
  framework, or a compatibility path is a stop condition requiring design
  revision.

Every slice also reports:

- files touched;
- approximate production and test line delta;
- types/helpers/branches removed and added;
- whether authority handoffs decreased, stayed neutral, or increased;
- whether the implementation shape became simpler, neutral, or more complex;
  and
- why any added invariant-bearing type justifies itself.

Test growth is expected because this is a coverage line. Production complexity
must not increase merely to accommodate the harness. Net line reduction is not
an acceptance criterion, but the number of policy owners and representable
invalid states must decrease or remain unchanged.

## Risks And Mitigations

### Risk: Replacing Templates With A Generator Framework

Mitigation: use the existing typed AST, replay, shrinking, and scheduler. Add
only derived structural identity and bounded compatibility construction. Do
not create a generic SQL algebra or plugin system.

### Risk: Counting Impossible Feature Pairs As Gaps

Mitigation: every interaction is reviewed against current grammar, typing, and
semantic contracts and is explicitly classified. There is no blind 98-by-98
coverage threshold.

### Risk: Generated Output Defines Its Own Coverage Universe

Mitigation: Slice 0 freezes manifest participation, every type-valid group
tuple, stable obligation IDs, required signatures, witnesses, and a catalog
hash before generator changes. Observed signatures can satisfy only that
required set; unexpected signatures are discovery evidence.

### Risk: A Signature Digest Becomes Coverage Authority

Mitigation: the full lossless closed signature owns equality and ordering.
Digests identify artifacts only and are always verified against the full
representation.

### Risk: SQLite Becomes Accidental Product Authority

Mitigation: reference eligibility remains exact and typed. Accepted-native
semantics use independent models or deterministic invariant providers, with
explicit exclusions.

### Risk: The Test Model Copies Production Bugs

Mitigation: the model cannot call production evaluation, normalization,
default, planning, executor, mutation, or storage helpers. SQL/fluent parity is
reported as parity, not independent truth.

### Risk: Current Row-Format Work Invalidates Performance Comparison

Mitigation: no production optimization begins until the post-0.213 opening
profile and cohort are frozen. Old runs are context, not baseline authority.

### Risk: Range Fusion Drops A Residual Or Changes Null Semantics

Mitigation: extraction produces access bounds and an explicit residual in one
planned artifact. Null, field-to-field, incompatible expression, and
contradictory-bound cases are required validation.

### Risk: Prepared Values Or Cursors Reintroduce Planning

Mitigation: compilation stores typed literal/parameter sources and cursor
domain shape once. The executor binds each source once, intersects a validated
directional continuation, and emits `Empty` or `Bounded` without relowering or
planner re-entry. Warm cache identity never contains prior bound values.

### Risk: A Faster Traversal Increases Other Work

Mitigation: data gets, residual evaluations, decoded rows, retained state,
output, all phase instructions, slopes, and raw wasm remain gated. Work cannot
move into an unmeasured phase.

### Risk: Membership Measurement Expands Into A Second Remediation

Mitigation: 0.214 requires classification only. A production change needs its
own frozen eliminated-work contract and explicit design revision or successor
line.

### Risk: Scenario Growth Makes Scheduled Evidence Unusable

Mitigation: structural obligations drive a bounded set, not arbitrary
Cartesian expansion. Tier A/Tier C wall-clock, artifact, fixture, expression,
and shrink budgets remain hard gates and are revised only with reviewed
evidence.

## Closeout Decision Rule

0.214 does not close because it executes more SQL strings. It closes when the
maintained contract has materially broader typed structural and interaction
evidence, one fresh comparable current performance authority exists, and the
selected duplicate compound-range traversal has been removed without semantic
or resource regression.

The line is intentionally not the final possible SQL optimization effort. It
must leave measured membership/prefix and any newly discovered families with
honest dispositions so a later bounded slice can select them without reopening
0.204, 0.206, or this design.
