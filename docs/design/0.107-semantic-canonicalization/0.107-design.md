0.107 Bounded Semantic Canonicalization for Boolean `WHERE`

Thesis

`0.107` should eliminate representation-level divergence for a small class of
already-admitted boolean `WHERE` expressions by introducing bounded semantic
canonicalization, starting with searched `CASE`.

`0.107` ensures that semantically equivalent boolean `WHERE` expressions
produce identical predicate extraction, residual plans, and explain output by
introducing a tightly scoped canonicalization rule for searched `CASE`,
without opening general-purpose SQL rewrite policy.

This is a targeted equivalence-class unification slice, not a general
algebraic optimizer.

Problem statement

After `0.103` and `0.105`:

* scalar `WHERE` is expression-complete
* residual filtering is explicit plan state
* predicate extraction is structurally normalized and stable

However, semantically equivalent expressions can still diverge when they enter
the system through different syntactic forms.

Concrete example:

* `CASE WHEN a THEN TRUE ELSE b END`
* `(a AND TRUE) OR (NOT a AND b)`

These are semantically equivalent, but today may produce:

* different normalized forms
* different predicate extraction results
* different residual expressions
* different explain output

This breaks:

* predictability
* explain parity
* long-term normalization guarantees

The current issue manifests primarily with searched `CASE`, but the underlying
category is broader:

* multiple front-door representations of the same boolean semantics

Design goal

Introduce a minimal, safe canonicalization rule that:

* collapses a known equivalence class into one canonical boolean form
* operates only on already-admitted planner-owned expressions
* improves extraction and explain consistency
* does not expand SQL surface or rewrite policy broadly

The goal is to make equivalent boolean `WHERE` expressions behave identically
without introducing general algebraic rewriting.

Non-goals

`0.107` does not aim to:

* introduce general boolean algebra simplification
* apply distributive laws such as `A AND (B OR C)` rewriting
* canonicalize arithmetic expressions
* rewrite `BETWEEN`, `IN`, or other syntactic sugar
* change clause admission or validation rules
* affect grouped semantics or `HAVING`
* widen predicate expressiveness
* introduce cost-based rewrite logic

This is not a general optimizer slice.

Normative model

Rule 1 - canonicalization is post-validation, pre-extraction

Semantic canonicalization operates in this phase order:

* parse
* validate
* canonicalize (bounded)
* normalize (`0.105`)
* extract

It must not:

* participate in admission
* bypass validator rules
* introduce new expression shapes that were not already valid

Rule 2 - bounded equivalence class: searched `CASE` (boolean-valued)

Canonicalization applies only to searched `CASE` expressions that:

* appear in scalar `WHERE`
* evaluate to boolean
* have only boolean-valid `WHEN` predicates and `THEN`/`ELSE` arms under the
  existing scalar `WHERE` boolean-valid contract

Rule 3 - canonical form

The canonicalization rule is:

```sql
CASE
  WHEN p1 THEN r1
  WHEN p2 THEN r2
  ...
  ELSE rN
END
```

becomes:

```sql
(p1 AND r1)
OR (NOT p1 AND p2 AND r2)
OR (NOT p1 AND NOT p2 AND r3)
...
OR (NOT p1 AND ... AND rN)
```

This is the first-match semantics expansion.

If `ELSE` is absent, the searched `CASE` is treated as `ELSE NULL` for
canonicalization purposes only when that implicit null arm remains admissible
and semantically identical under the engine's existing scalar boolean rules.
If that condition is not met, canonicalization is skipped.

Constraints:

* preserves evaluation-order semantics
* must be structurally correct under the existing boolean expression model
* must reuse existing `Expr` operators (`AND`, `OR`, `NOT`)

Rule 4 - safety constraints

Canonicalization is applied only when safe:

* all arms are boolean-valued
* no aggregate or grouped-only expressions are introduced
* no change in null/boolean collapse semantics under the existing three-valued
  scalar boolean runtime rules
* no duplication of side-effecting expressions

The side-effect rule is forward-looking. Side effects are not applicable today,
but the rule must remain part of the contract.

The core semantic invariant is:

* canonicalization is allowed only when the transformed form is provably
  identical under the engine's existing three-valued boolean semantics

If any constraint is not satisfied:

* canonicalization is skipped
* the original expression is preserved

Rule 5 - residual ownership remains explicit

After canonicalization:

* the resulting expression becomes the new `filter_expr`
* predicate extraction operates on this canonical form
* residual is explicitly derived as `residual_filter_expr`

No implicit residual behavior is reintroduced.

Rule 6 - explain contract

Explain must reflect the canonicalized form.

That means semantically equivalent expressions produce identical:

* predicate
* `residual_filter_expr`
* explain output

Explain does not need to preserve original syntactic spelling.

Identity surfaces that are derived from semantic filter shape should also use
the canonical form so explain, fingerprint, and nearby query-shape surfaces do
not drift from one another.

Rule 7 - no interaction with template admission

Canonicalization must not:

* widen prepared template eligibility
* introduce new predicate shapes into template lanes

Prepared boundary remains unchanged.

Examples

Example 1 - simple `CASE` equivalence

Input:

```sql
WHERE CASE WHEN a THEN TRUE ELSE b END
```

Canonical:

```sql
WHERE (a AND TRUE) OR (NOT a AND b)
```

Expected result:

* identical predicate extraction
* identical residual behavior
* identical explain output regardless of input form

Example 2 - multi-arm `CASE`

Input:

```sql
WHERE CASE
  WHEN a THEN x
  WHEN b THEN y
  ELSE z
END
```

Canonical:

```sql
WHERE (a AND x)
   OR (NOT a AND b AND y)
   OR (NOT a AND NOT b AND z)
```

Example 3 - non-canonicalized shape

```sql
WHERE CASE WHEN a THEN COUNT(*) ELSE b END
```

This is not canonicalized because it:

* introduces aggregate semantics
* violates the scalar `WHERE` boundary

Likely code seams

* planner expression canonicalization pass
* integration point before `0.105` normalization
* boolean expression constructors
* explain rendering

Risks

Risk 1 - semantic drift

Incorrect `CASE` expansion could change evaluation semantics.

Mitigation:

* strictly enforce first-match expansion rules
* reuse existing boolean evaluation semantics

Risk 2 - over-expansion

Expression size may grow significantly for large `CASE` trees.

Mitigation:

* this slice remains bounded
* large-scale optimization is out of scope
* if expansion exceeds one explicit bounded implementation threshold,
  canonicalization is skipped and the original expression is preserved

Risk 3 - scope creep

Contributors may attempt to add more rewrite rules.

Mitigation:

* explicitly forbid additional equivalence classes in this slice

Verification matrix

Area | Required proof
--- | ---
Canonicalization | searched `CASE` expressions expand correctly into boolean form
Equivalence | canonical and original forms produce identical results
Extraction | equivalent forms produce identical predicate extraction
Residual | equivalent forms produce identical `residual_filter_expr`
Explain | equivalent forms produce identical explain output
Safety | invalid or unsafe `CASE` shapes are not rewritten
Prepared | no change in template admission

Acceptance criteria

`0.107` is complete when:

* searched `CASE` expressions in scalar `WHERE` are canonicalized when safe
* semantically equivalent `CASE` and boolean-expanded forms produce identical:
  * predicate extraction
  * residual plans
  * explain output
* no new rewrite classes are introduced
* prepared execution behavior remains unchanged

Thesis sentence

`0.107` introduces bounded semantic canonicalization for boolean searched
`CASE` in scalar `WHERE`, ensuring semantically equivalent expressions produce
identical predicate extraction, residual execution, and explain output without
expanding general rewrite policy.
