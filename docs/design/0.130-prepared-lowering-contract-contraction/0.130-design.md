# 0.130 Prepared Lowering Contract Contraction

## Thesis

`0.130` should contract the prepared-query and lowering handoff after the
`0.129` access-contract line is closed.

This is not an access execution-contract continuation.

It is not a new SQL feature line.

It is not a persisted-row or storage-format line.

It is a prepared/lowering boundary cleanup whose job is to make prepared
statement templates, parameter rebinding, lowered query artifacts, and fallback
error reporting flow through fewer adjacent contracts.

The intended end state for `0.130` is:

* prepared SQL compilation emits one clearly-owned reusable semantic artifact
* parameter rebinding applies through one prepared-input boundary
* lowered select/update/delete execution paths consume the same prepared
  contract shape where behavior is shared
* fallback and unsupported-shape errors are classified once at the owning
  boundary
* access execution contracts from `0.129` remain stable and are consumed as
  inputs, not reopened as a redesign target

---

## Why `0.130` Exists

The April 2026 audits repeatedly identify prepared/lowering pressure as real,
but `0.129` intentionally deferred it while access execution contracts were
still in flight.

After `0.129`, the access side should be stable enough that the next structural
front can move up one layer:

* prepared statement compilation
* parameter binding and rebinding
* lowered query ownership
* session SQL execution entrypoints
* fallback and error wording around unsupported prepared shapes

Those areas are close enough that changes in one can force defensive wrappers
or duplicated contract checks in another. `0.130` should reduce that drift
without mixing in new planner capabilities.

---

## Problem

Prepared execution still carries too many neighboring shapes:

* source SQL and parsed statement shape
* prepared template state
* parameter binding state
* lowered select/query artifacts
* execution-ready route/access artifacts
* fallback and unsupported-shape diagnostics

Each surface is legitimate, but the handoff can still read as a sequence of
small adapters rather than one intentional prepared contract.

That creates three risks:

* parameter rebinding behavior can drift from initial compilation behavior
* fallback errors can be constructed away from the owner that knows the failure
  taxonomy
* lowered query artifacts can become broader than the execution path actually
  needs

---

## Goal

Make prepared lowering a single explicit handoff from reusable semantic query
shape to execution-ready artifacts.

`0.130` should deliver:

* one prepared-input contract for bound parameters
* one prepared semantic artifact that is reused by execution instead of
  rebuilding equivalent state
* fewer session SQL bridge adapters around select/update/delete execution
* typed error constructors at the owner boundaries for unsupported prepared
  shapes
* clear separation between prepared semantic reuse and access execution
  mechanics

This should be done without:

* changing SQL semantics
* changing public SQL syntax
* changing persisted data
* changing cursor token format
* reworking access execution contracts from `0.129`
* broad parser cleanup

---

## Core Invariant

Prepared compilation owns reusable semantic meaning; execution lowering owns
runtime mechanics.

Corollaries:

* prepared templates may cache validated semantic structure
* parameter rebinding must go through the same input contract as initial
  binding
* lowered execution artifacts should not retain source-level scaffolding after
  lowering is complete
* access contracts should be treated as an already-stabilized downstream
  execution input
* unsupported prepared shapes should fail at the boundary that owns the
  taxonomy, not through generic string checks

---

## Scope

### In Scope

* prepared statement template structure
* prepared parameter binding and rebinding
* lowered select/query artifact ownership
* session SQL execution bridge cleanup
* prepared fallback and unsupported-shape error construction
* focused regression tests for prepared parameter behavior and lowered
  execution parity

### Out Of Scope

* access execution-contract redesign
* secondary order-pushdown eligibility changes
* new SQL grammar
* new aggregate semantics
* persisted-row format changes
* storage/index physical rewrites
* broad parser module split

## Proposed Shape

### Prepared Input Contract

Prepared parameter values should enter through one prepared-owned input
contract.

That contract should own:

* parameter count validation
* parameter type conversion
* missing parameter errors
* rebinding behavior for cached templates

It should not own:

* SQL parsing
* access route selection
* executor terminal behavior

### Prepared Semantic Artifact

Prepared compilation should produce one reusable semantic artifact that can be
bound and lowered repeatedly.

That artifact should preserve:

* statement kind
* validated expression structure
* stable parameter slots
* enough planning context to lower deterministically

It should not preserve source-level scaffolding after semantic validation
unless a downstream surface still needs it for diagnostics.

### Lowered Execution Artifact

Lowering should produce the execution-ready artifact consumed by session and
executor entrypoints.

That artifact should include:

* logical execution plan
* access plan or lowered access contract
* execution mode and route inputs
* mutation/write inputs where relevant

It should not rebuild prepared parameter semantics.

## Patch Sequence

### Opening Patch: Prepared Boundary Inventory

* map current prepared template, parameter binding, and lowered artifact owners
* remove dead forwarding helpers that only pass through to the same owner
* add focused tests where removal would otherwise be hard to review

### Subsequent Patch: Parameter Rebinding Contract

* consolidate parameter-count and missing-parameter checks
* ensure initial binding and rebinding share the same owner
* keep error classes stable and typed

### Subsequent Patch: Lowered Artifact Narrowing

* reduce fields carried after lowering when they are source-level scaffolding
* split select/update/delete only where behavior genuinely diverges
* keep route/access contracts as downstream inputs

### Closeout Patch: Error and Documentation Pass

* move repeated fallback/unsupported error construction onto owning error types
* update changelog
* record intentionally retained boundaries
* stop before parser or access redesign begins

Patch numbers should be assigned only when the actual slices are created.

---

## Acceptance Criteria

`0.130` is complete when:

* prepared parameter binding and rebinding share one contract
* lowered execution artifacts no longer carry source-level state without a
  clear consumer
* session SQL entrypoints have fewer pass-through adapters around prepared
  execution
* unsupported prepared shapes use typed owner-local error construction
* access execution contracts remain unchanged except for call-site adaptation
* prepared SQL tests preserve current behavior

---

## Non-Goals for This Line

`0.130` should not become a parser rewrite or a new SQL feature train.

Later lines can target:

* parser module shape
* broader expression normalization
* additional SQL syntax
* physical index-store rewrites

The point of `0.130` is to make prepared execution easier to reason about
after access execution contracts have been stabilized.
