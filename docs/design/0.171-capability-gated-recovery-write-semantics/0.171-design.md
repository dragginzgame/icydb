# IcyDB 0.171 — Capability-Gated Recovery And Write Semantics

## Status

In progress.

## Thesis

0.170 made storage behavior explicit through capabilities and introduced the
first durable/live/mixed mutation classification. 0.171 should make that
capability-derived classification authoritative at the write/recovery boundary:
mutation, commit, recovery, and diagnostic/proof surfaces must consistently
consume affected-store classification as durable-only, live-only, or mixed
durable/live.

This is not WAL. This is not durable heap. This is the line that proves the
engine's write path understands mixed durability before new persistence
machinery is added.

## Why This Is 0.171

0.169 admitted volatile `storage(heap())` stores. 0.170 described stable and
heap behavior through capability axes and introduced internal durable/live/mixed
commit classification. The next unresolved seam is whether the write, commit,
recovery, and diagnostic paths consistently apply those capabilities when
mutation outcomes cross storage-durability boundaries.

The key question is:

```text
When a write touches stable stores, heap stores, or both, what durability
contract did the operation actually have?
```

Answering that is more urgent than adding journaled storage. WAL,
checkpointing, or durable heap would be dangerous before mixed durability is
mechanically classified, tested, and visible at the correct internal boundary.

## Summary

0.171 hardens the operational use of the 0.170 capability model and existing
mutation classification across mutation planning, commit participation,
recovery admission, failure ordering, and diagnostic/proof surfaces.

0.171 should not reintroduce direct stable/heap checks in higher layers. Code
that needs durability policy must consume runtime store capabilities:

- `commit_participation`;
- `recovery`;
- `durability`;
- `allocation_identity`;
- relation source/target capability where relation validation is involved.

## Current Baseline

0.170 already closed these foundations:

- stable and heap stores expose runtime storage capabilities;
- public writes can emit durable-only, live-only, or mixed durable/live
  classifications;
- relation admission consumes source/target relation capabilities;
- source-audit guards block concrete stable/heap branching in executor,
  relation, commit, diagnostics, and reconciliation roots;
- heap volatility and stable compatibility are covered by focused regression
  tests.

0.171 should build on that baseline. It should not duplicate the 0.170
capability model or create a parallel mutation-classification vocabulary.

## 0.171.0 Boundary Inventory

0.171.0 records the current write/recovery boundary before changing behavior.
This is the baseline later 0.171 patches must harden.

### Existing Classification Owners

The current durable/live/mixed classification is owned by:

- `MutationCommitClass` in `crates/icydb-core/src/metrics/sink.rs`;
- `affected_store_handles_for_prepared_row_ops(...)` in
  `crates/icydb-core/src/db/executor/mutation/commit_window.rs`;
- `classify_mutation_commit_plan(...)` in
  `crates/icydb-core/src/db/executor/mutation/commit_window.rs`;
- `MetricsEvent::MutationCommitPlan`, which records the classification at the
  write boundary.

The classifier already consumes
`handle.storage_capabilities().commit_participation()` instead of matching on
concrete stable/heap storage mode.

Focused proof already exists for stable-only, heap-only, and mixed
classification in
`crates/icydb-core/src/db/session/tests/heap_runtime.rs`, including
`public_writes_emit_durable_live_and_mixed_commit_classifications`.

### Current Affected-Store Behavior

The current affected-store collector includes store handles whose data store or
index store pointers are present in prepared row operations.

That means ordinary row mutation classification currently includes:

- data stores receiving row insert/update/delete operations;
- index stores receiving maintained index insert/delete/update operations;
- relation-maintenance stores when those mutations are represented as prepared
  row/index operations.

Read-only validation targets are not included merely because they were read for
relation validation, lookup, planning, or diagnostics. This matches the 0.171
contract that validation-only reads do not make a target store a commit
participant.

Schema stores are not included for ordinary row mutations. Schema-changing
operations need their own proof if they later share this durability
classification boundary.

### Write Stage Mapping

The current implementation maps to the conceptual write stages as follows:

1. Validate inputs, relation constraints, and structural preflight through the
   save/mutation preflight paths and `preflight_prepare_row_op_batch*`.
2. Stage live row/index/relation mutations as prepared row operations and
   preflight overlays before mutating real stores.
3. Collect affected store handles and classify the mutation commit plan in
   `open_commit_window*` and single-row save/delete paths.
4. Prepare durable commit participation through `CommitMarker::new(...)` and
   the commit guard entrypoints.
5. Persist durable marker state through `begin_commit(...)` or
   `begin_single_row_commit(...)`.
6. Apply staged live mutations and finish the commit through
   `apply_prepared_row_ops(...)`, `apply_prepared_single_row_op(...)`, and
   `finish_commit(...)`.
7. Expose proof-facing classification through `MetricsEvent::MutationCommitPlan`.

The current implementation records classification before apply in the owning
write paths. Later 0.171 hardening should preserve that ordering or document any
intentional change.

### Failure-Injection Inventory

The following failure boundaries already have focused test seams:

- validation/preflight failure before durable marker publication, including
  `strong_relation_missing_fails_preflight`,
  `insert_many_atomic_rejects_duplicate_unique_index_values_in_request_before_marker`,
  and `failed_heap_source_to_stable_target_write_leaves_no_heap_side_effects`;
- preflight purity before real store mutation, covered by
  `commit_window_preflight_does_not_mutate_real_stores_before_apply`;
- generation drift between preflight and apply, covered by
  `commit_window_rejects_apply_when_index_store_generation_changes`;
- finish/apply failure after marker state, covered by commit guard regression
  tests such as `finish_commit_error_keeps_marker_for_recovery` and
  `finish_commit_mixed_state_failure_rolls_back_index_prefix_without_row_visibility`.

The following cases are not yet directly injectable without either a narrower
test seam or an explicit gap record:

- physical marker-persistence failure inside `begin_commit(...)`;
- stable-side apply failure after heap-side preparation in a mixed public write;
- simulated recovery after a single successful mixed operation proving that only
  the durable stable portion survives.

0.171.3 should add narrow tests for injectable cases and record explicit gaps
for cases that would require broad production failure-injection machinery.

### Source-Audit Baseline

`crates/icydb-core/tests/write_boundary_guards.rs` already contains source-audit
guards for the storage capability boundary:

- `storage_capability_policy_roots_do_not_branch_on_concrete_storage_modes`;
- `relation_and_commit_policy_consume_storage_capability_axes`.

0.171 should extend or tighten these guards only where the recovery/write
boundary needs more specific coverage.

### 0.171.0 Outcome

0.171.0 is documentation and inventory only. It does not change runtime write,
commit, recovery, storage, relation, diagnostics, or codec behavior.

## Goals

- Make affected-store mutation classification the authoritative internal
  contract for write, commit, and recovery policy.
- Prove stable-only writes use the durable commit/recovery path.
- Prove heap-only writes remain live-only and make no durable recovery claim.
- Prove mixed writes distinguish stable durable participation from heap
  live-only side effects.
- Gate commit and recovery participation through capability axes, not storage
  mode.
- Prove recovery never attempts to replay heap rows, heap indexes, or heap
  schema metadata as durable state.
- Document and test mixed-write failure ordering.
- Expose mutation durability classification through proof-facing tests first,
  and through an existing trace, diagnostic, or metric surface only if that
  surface already owns write durability reporting.
- Strengthen source-audit guards around mutation, commit, and recovery roots.
- Preserve stable ABI, stable keys, memory IDs, and row/index/schema codecs.

## Non-Goals

- No WAL.
- No checkpointing.
- No heap persistence.
- No cached-stable store.
- No journaled store.
- No migration tooling.
- No stable ABI change.
- No stable key change.
- No memory-ID change.
- No row, index, schema, or commit-marker codec change unless explicitly
  justified and hard-cut.
- No SQL storage syntax.
- No relation semantic expansion beyond consuming existing capabilities.
- No compatibility shims, aliases, or fallback recovery paths.

## Hard-Cut Rules

- Do not keep direct stable/heap recovery or write policy as a compatibility
  path once a capability gate owns that decision.
- Do not classify heap as absent allocation identity and then infer durability
  from that absence; consume `commit_participation` and `recovery` explicitly.
- Do not treat diagnostics DTOs, generated models, or trace output as runtime
  capability authority.
- Do not persist heap rows, heap indexes, or heap schema snapshots into stable
  recovery machinery.
- Do not imply atomic durable recovery across stable and heap stores.
- A successful mixed write may have live all-or-nothing behavior during the
  current process lifetime, but only the durable-participating portion may be
  claimed after recovery. Any heap-side success is live-only state.

## Mutation Durability Contract

0.171 uses the existing 0.170 mutation durability classes, or the final
equivalent names:

```rust
enum MutationDurabilityClass {
    DurableOnly,
    LiveOnly,
    MixedDurableAndLive,
}
```

Classification must be derived from affected store handles and their
capabilities, not from concrete storage modes:

```text
all affected stores durable commit participants => DurableOnly
all affected stores live-only                  => LiveOnly
mixed durable and live-only participants       => MixedDurableAndLive
```

The classification is an internal durability contract. It may be projected into
metrics, traces, diagnostics, or proof fixtures, but it does not widen the
public write API by default.

## Affected Store Set

For mutation durability classification, the affected store set includes stores
whose live contents may be mutated by the operation:

- data stores receiving row insert/update/delete;
- index stores receiving maintained index insert/delete/update;
- relation-maintenance stores or reverse-index stores if the operation mutates
  them;
- schema stores only for schema-changing operations.

The affected store set does not include stores that are only read for
validation, planning, lookup, or diagnostics.

Relation validation reads may consume source/target relation capabilities, but
they do not make a read-only target store a commit participant. If relation
maintenance mutates a reverse index, that mutated store is part of the affected
store set.

## Commit Participation Contract

Stable stores participate in durable commit/recovery. Heap stores do not.

0.171 should prove:

```text
stable-only mutation -> durable commit path
heap-only mutation   -> live-only path, no durable recovery claim
mixed mutation       -> stable durable portion + heap live-only portion
```

Commit preparation and recovery-facing code should consume:

```text
capabilities.commit_participation
capabilities.recovery
```

not:

```text
StoreStorage::Stable
StoreStorage::Heap
DataStoreBackend::Stable
DataStoreBackend::Heap
```

## Recovery Contract

Recovery remains stable-store recovery infrastructure in 0.171.

Heap stores:

- are not replayed from commit markers;
- do not recover heap data rows;
- do not recover heap index entries;
- do not recover heap schema snapshots as durable accepted history;
- may rebuild live schema metadata through the normal heap startup path.

Stable stores:

- keep existing commit-marker and recovery behavior;
- keep existing stable allocation identity;
- keep existing row/index/schema codec behavior.

If recovery code sees a mutation or marker shape that would imply heap replay,
it must fail closed or remain unreachable by construction. It must not silently
promote heap state to durable state.

## Mixed-Write Failure Semantics

0.171 must document the live execution guarantee separately from durable
recovery semantics.

## Write Boundary Stages

0.171 should describe failures against the existing write pipeline stages. If
the implementation uses different names, the design closeout should map the
real implementation stages to this conceptual sequence:

1. plan/classify affected stores;
2. validate inputs and relations;
3. stage live row/index/relation mutations;
4. prepare durable commit participation;
5. persist commit marker or durable publication state;
6. apply/finish live mutations;
7. expose success/result/diagnostic classification.

Expected contract:

- Validation failures before mutation must leave no heap or stable side effects.
- Failed mixed writes must not emit a success-like durable classification.
- If the existing write/session path is live-transactional, prove heap and
  stable side effects are rolled back together on failure.
- If any narrower guarantee exists, document it precisely and test it.
- Successful mixed writes may be live-atomic during execution, but recovery
  after upgrade may observe only the durable stable portion.

Failure-ordering tests should cover:

- validation fails before mutation;
- heap-side staging/apply fails before stable durable publication;
- stable-side staging/apply fails after heap-side work is prepared;
- commit marker persistence fails;
- commit finish/apply fails after durable marker persistence.

0.171 should first inventory which failure modes are already injectable through
existing test seams. Add narrow injection hooks only where they do not broaden
production API or destabilize commit code. For non-injectable failure modes,
record a precise gap and prove the nearest existing boundary.

## Diagnostics And Trace Contract

0.171 prefers proof-facing or internal trace evidence over public DTO
expansion. Public diagnostics should change only if an existing diagnostics
surface already represents write/commit classification.

Diagnostics and traces may render mutation durability classification, but they
must remain consumers of capability-derived facts.

Acceptable outputs include:

```text
mutation_durability=durable_only
mutation_durability=live_only
mutation_durability=mixed_durable_and_live
```

or equivalent structured fields on an existing metrics/trace event.

Diagnostics must not imply:

- heap durability;
- heap recovery replay;
- atomic durable recovery across stable and heap stores;
- allocation identity for heap stores.

## Source-Audit Contract

Mutation, commit, recovery, diagnostics, and relation write-boundary code should
not match directly on stable/heap storage modes outside allowed construction and
store-wrapper zones.

Policy audit command shape:

```text
rg -n "StoreStorage::Heap|StoreStorage::Stable|StoreRuntimeStorageMode::Heap|StoreRuntimeStorageMode::Stable|DataStoreBackend::Heap|IndexStoreBackend::Heap|SchemaStoreBackend::Heap|\\.storage_mode\\(\\)" \
  crates/icydb-core/src/db/executor \
  crates/icydb-core/src/db/commit \
  crates/icydb-core/src/db/relation \
  crates/icydb-core/src/db/schema/reconcile.rs
```

Expected result: no production policy roots branch on concrete stable/heap
storage mode.

Rendering/projection audit command shape:

```text
rg -n "StoreStorage::Heap|StoreStorage::Stable|StoreRuntimeStorageMode::Heap|StoreRuntimeStorageMode::Stable|\\.storage_mode\\(\\)" \
  crates/icydb-core/src/db/diagnostics \
  crates/icydb-cli/src
```

Expected result: rendering roots may project storage mode, but they must not use
it as policy authority. Diagnostics DTOs and CLI output remain consumers.

Capability consumption audit:

```text
rg -n "commit_participation\\(\\)|recovery\\(\\)|durability\\(\\)|relation_source\\(\\)|relation_target\\(\\)|storage_capabilities\\(\\)" \
  crates/icydb-core/src/db/executor \
  crates/icydb-core/src/db/commit \
  crates/icydb-core/src/db/relation \
  crates/icydb-core/src/db/diagnostics
```

Expected result: durability and recovery policy is visible through capability
axis consumption at the owning boundary.

## Patch Slices

### 0.171.0 — Design And Boundary Inventory

- Land this design doc.
- Inventory current mutation durability classification and existing proof tests.
- Define the affected-store set against current write/relation/index behavior.
- Inventory write stages and map them to the current implementation.
- Inventory which failure modes are injectable today.
- Identify commit/recovery call sites that already consume capabilities.
- Add or update source-audit commands if the current 0.170 guard is too broad
  or too narrow for recovery-specific coverage.
- Do not change runtime behavior.

### 0.171.1 — Classification Consumption Hardening

- Make affected-store classification the authoritative write-boundary contract.
- Keep classification derived from store-handle capabilities.
- Prove stable-only, heap-only, and mixed public writes produce the expected
  internal classification.
- Prove validation-only reads do not affect commit participation.
- Keep the classification internal/proof-facing unless an existing diagnostic
  surface naturally exposes it.

### 0.171.2 — Commit And Recovery Capability Gate

- Ensure commit/recovery code consumes `commit_participation` and `recovery`
  capability axes.
- Prove heap stores never enter durable replay expectations.
- Prove stable behavior remains unchanged.
- Avoid codec or marker-format changes.

### 0.171.3 — Mixed Failure Semantics

- Add focused failure-ordering tests for already-injectable failure modes.
- Prove validation failures leave no heap side effects.
- Add narrow injection hooks only when they stay test-local or preserve the
  production API boundary.
- Document any failure mode that cannot be injected safely in this line and
  prove the nearest existing boundary.

### 0.171.4 — Diagnostics And Audit Surface

- Add or tighten trace/diagnostic rendering for mutation durability
  classification only if an existing surface already owns write durability
  reporting.
- Otherwise keep proof evidence test-only.
- Strengthen source-audit guards for mutation, commit, recovery, diagnostics,
  and relation write-boundary code.

### 0.171.5 — Closeout

- Close W1-W16.
- Record stable ABI/no-codec-change evidence.
- Record heap volatility/recovery evidence.
- Record mixed-write semantics evidence.
- Record skipped checks explicitly.

## Release-Line Recommendation

0.171 should be "Capability-Gated Recovery And Write Semantics." It is the
right bridge between "storage capabilities exist" and later "new durable
storage machinery exists." The release should make mixed durability hard to
misrepresent before WAL, checkpointing, journaled storage, cached stable, or
heap persistence are designed.
