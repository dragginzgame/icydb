# 0.124 Scalar Function Semantics & Coverage Completion

## Thesis

`0.124` should formalize scalar function semantics, typing, capability, and
coverage before join work expands expression complexity across multiple tables
and planning surfaces.

This is not a feature-expansion-first slice.

It is not a join slice.

It is not a lowering-only cleanup slice.

It is a scalar-function centralization slice whose job is to make planner-owned
function semantics explicit and complete enough that later join work consumes a
stable function model instead of multiplying ad hoc function behavior across
new query shapes.

The target end state is:

* one canonical scalar-function taxonomy
* one canonical function classification / specification owner
* planner-owned typing and capability answers for scalar functions
* executor consuming planner-frozen function semantics instead of reclassifying
  function families
* explicit null behavior and surface eligibility per function family
* no scalar-function semantics expressed outside the canonical specification
  owner except pure executor dispatch

---

## Why `0.124` Exists

Recent slices centralized expression meaning, collapsed flow, unified pipeline
topology, and started collapsing broad structural classification seams.

What remains under-formalized is scalar function behavior.

The current tree already has real scalar-function coverage, but it is still
distributed across:

* parser acceptance and function-name lookup
* lowering-time `SqlScalarFunction -> Function` mapping
* planner type inference and function-family grouping
* executor function evaluation dispatch
* builder / preview / aggregate folding paths

That is workable while the query surface is still relatively narrow.

It becomes a real architectural risk once joins widen:

* predicate complexity
* projection expressions
* grouped semantics
* ordering
* null propagation
* function evaluation over multiple input tables

Without one explicit scalar-function model, joins will amplify ambiguity rather
than just add row combination.

## Audit Alignment

### 1. Function Taxonomy Is Present but Not Yet Canonically Owned

The current code already has bounded scalar function enums:

* `db/sql/parser/model.rs` (`SqlScalarFunction`)
* `db/query/plan/expr/ast.rs` (`Function`)

But the transition from parsed function name to planner-owned semantic function
still relies on phase-local mapping in:

* `db/sql/lowering/expr.rs`

This means taxonomy exists, but the canonical semantic contract is not yet the
primary thing other layers consume.

### 2. Function Typing Is Concentrated but Still Feels Ad Hoc

Planner type inference already carries meaningful function grouping in:

* `db/query/plan/expr/type_inference/mod.rs`

That is the strongest current semantic owner, but it still behaves more like an
internal typing table than a full function specification surface.

Important missing explicitness:

* null behavior
* deterministic / non-deterministic classification
* surface eligibility by lane (`WHERE`, `HAVING`, grouped projection,
  `ORDER BY`)
* one named capability model for consumers outside type inference

### 3. Evaluation and Folding Still Consume Function Families Indirectly

Executor evaluation and preview / fold paths currently carry their own function
dispatch groupings in:

* `db/executor/projection/eval/scalar_function.rs`
* `db/query/plan/expr/preview.rs`
* `db/query/builder/aggregate.rs`

Those are legitimate consumers, but they still expose the absence of one
planner-owned capability registry that clearly says:

* what the function is
* how it types
* how null behaves
* where it is admissible

---

## Problem

Scalar function support is real, but the concept is still only partially
formalized.

The repeated pattern is:

* parser owns the accepted function names
* lowering owns function mapping
* type inference owns function grouping for typing
* executor owns function evaluation buckets
* other consumers derive local assumptions from those same families

This creates four classes of risk:

### 1. Capability Drift

Adding or changing one function can require touching parser, lowering, typing,
evaluation, and tests without one central contract that says what must change.

### 2. Null-Semantics Drift

Function null behavior is partly implicit in evaluation logic and partly
implicit in typing / validation expectations.

That is too loose for joins, where null propagation becomes much more visible.

### 3. Surface Drift

The codebase does not yet make it explicit which function families are allowed
in:

* `WHERE`
* `HAVING`
* grouped projection
* grouped ordering
* scalar ordering

Some of that is enforced today, but not through one named function capability
model.

### 4. Coverage Drift

The current supported set is meaningful, but there is no single completion
standard that says when scalar function coverage is "enough" for pre-join SQL.

---

## Goal

Replace distributed scalar-function interpretation with one planner-owned
function semantics contract.

`0.124` should deliver:

* one canonical function taxonomy
* one canonical function specification owner
* one explicit typing / capability / null-behavior contract per function
* one clear minimal-complete scalar function set for pre-join SQL

This is not about turning every consumer into the same implementation.

It is about making every consumer rely on the same named function semantics.

Hard constraint:

`0.124` is not complete if the registry exists but parallel semantic match
ladders still remain in type inference, lowering, builder/fold logic, or other
consumers because "it is convenient."

---

## Core Invariant

For each scalar function family, there is exactly one canonical planner-owned
semantic specification.

Other layers may still:

* parse names
* lower syntax
* evaluate runtime values
* render explain / labels

But they must do so by consuming the canonical function specification rather
than re-deriving function meaning from scattered local groupings.

Corollary:

`0.124` must centralize function semantics, not just collect helper methods.

The canonical output of this slice is a function specification surface, not a
bag of free-floating `is_*` predicates.

Critical enforcement rule:

No scalar function semantics may be expressed outside the canonical function
specification.

Concretely:

* no semantic `match Function::...` ladders outside the canonical
  specification owner
* the only allowed non-spec `match Function::...` site is executor evaluation
  dispatch, and that dispatch must be execution-only

That means:

* type validity must not be rechecked in executor
* capability must not be reclassified in executor
* null-behavior class must not be rediscovered in executor

All non-executor consumers must go through:

* `function_spec(function)`

rather than keeping local convenience ladders.

## Scope

### In Scope

* scalar function taxonomy definition
* planner-owned function classification / specification
* function typing rules
* function capability rules by query surface
* function null behavior
* function determinism classification where relevant
* minimal complete pre-join scalar function coverage
* removal of ad hoc function-family ladders where the canonical spec can serve
  instead

### Out Of Scope

* joins
* aggregate-function redesign
* user-defined functions
* executor algorithm redesign
* broad parser redesign outside scalar function coverage
* widening into full cast / type-system redesign unless strictly needed for the
  canonical function contract

---

## Ownership Boundary

### Planner / Query Owns

* scalar function taxonomy
* scalar function classification / specification
* type rules
* capability / surface eligibility
* null-behavior contract
* determinism classification

Preferred location:

* `db/query/plan/expr/function_semantics.rs`

or another clearly planner-owned expression-semantics module.

### Parser Owns

* surface syntax acceptance
* mapping text names onto parser function tokens

Parser must not become the semantic owner of function behavior beyond name /
shape acceptance.

### Lowering Owns

* syntax-to-planner adaptation
* argument-shape normalization where the syntax requires it

Lowering must not keep the long-term semantic contract for typing or capability.
Lowering must canonicalize function identity before planner semantics attach.

### Executor Owns

* runtime evaluation of already-classified planner functions

Executor must not reclassify function families or decide admissibility.
Executor must not re-check type validity, capability, or null-behavior class.
Executor may only dispatch execution over the already-canonical planner
function identity.

## Function Taxonomy

`0.124` should define one explicit scalar function taxonomy sufficient for
pre-join SQL.

Recommended minimal taxonomy:

### 1. Boolean / Predicate

* `IS NULL`
* `IS NOT NULL`
* `IS MISSING`
* `IS EMPTY`
* `IS NOT EMPTY`

### 2. Null Handling

* `COALESCE`
* `NULLIF`

### 3. Text

* `TRIM`
* `LTRIM`
* `RTRIM`
* `LOWER`
* `UPPER`
* `LENGTH`
* `LEFT`
* `RIGHT`
* `STARTS_WITH`
* `ENDS_WITH`
* `CONTAINS`
* `POSITION`
* `REPLACE`
* `SUBSTRING`

### 4. Numeric

* `ABS`
* `ROUND`
* `CEIL`
* `CEILING`
* `FLOOR`

### 5. Collection / Structural

* `COLLECTION_CONTAINS`

This taxonomy must be explicit in the canonical specification owner rather than
being implied by repeated match families in parser, type inference, and
evaluator dispatch.

Maintenance rule:

Whenever scalar functions are listed in source, they must stay in alphabetical
order.

For enum declarations that define scalar-function identity or closely related
function-taxonomy surfaces, use `#[remain::sorted]` so ordering drift is caught
mechanically rather than by review alone.

---

## Canonical Function Specification

`0.124` should introduce a planner-owned function specification model.

Recommended shape:

```rust
enum FunctionCategory {
    BooleanPredicate,
    NullHandling,
    Text,
    Numeric,
    Collection,
}

struct FunctionSpec {
    category: FunctionCategory,

    // typing
    signature: FunctionSignature,

    // runtime semantics
    null_behavior: FunctionNullBehavior,
    determinism: FunctionDeterminism,

    // planner-level classification
    result_type: FunctionResultType,

    // surface eligibility
    capabilities: FunctionCapabilities,
}
```

Important constraint:

Do not flatten everything into unrelated booleans.

The specification must be directly consumable by lowering, type inference,
planner validation, and executor dispatch without those consumers
reinterpreting function meaning through local ladders.

### Function Signature

Typing must be first-class in the specification surface.

Recommended shape:

```rust
enum FunctionSignature {
    Fixed {
        args: Vec<TypeFamily>,
        result: TypeFamily,
    },
    Variadic {
        min_arity: usize,
        arg_family: TypeFamily,
        result: TypeFamily,
    },
    Dependent {
        // COALESCE / NULLIF style
    },
}
```

Without a real signature model, type inference will keep reinventing rules
locally.

### Surface Capabilities

Avoid boolean explosion like:

* `where_allowed: bool`
* `having_allowed: bool`
* ...

Prefer a named surface model:

```rust
enum FunctionSurface {
    Where,
    Having,
    Projection,
    GroupedProjection,
    OrderBy,
}

struct FunctionCapabilities {
    allowed_surfaces: SmallVec<[FunctionSurface; N]>,
}
```

This reduces capability drift and prevents another family of nearly-identical
flags from growing beside the canonical function registry.

## Null Semantics

`0.124` must make null behavior explicit.

At minimum, every scalar function should be classified into one of:

* strict null propagation
* null-observing predicate behavior
* null-special handling

Recommended shape:

```rust
enum FunctionNullBehavior {
    Strict,
    NullIgnoring,
    NullObserving,
}
```

Examples:

* `LOWER`, `UPPER`, `TRIM`, `ABS`, `ROUND`, `SUBSTRING`: usually strict
* `COALESCE`: null-special
* `NULLIF`: null-special
* `IS NULL` / `IS NOT NULL`: null-observing predicate behavior

This is important before joins because null propagation across multi-row /
multi-table expression surfaces becomes much harder to reason about if the
rules stay implicit.

The design should therefore treat null semantics as executable contract data,
not explanatory commentary.

---

## Typing Contract

Planner-owned function semantics must answer:

* arity
* argument type families
* coercion allowance
* result type family
* dynamic-result behavior for functions like `COALESCE` / `NULLIF`

Current type inference is a good starting point, but `0.124` should convert it
from "where function grouping happens today" into "the place that consumes one
canonical function specification."

The type inference module should stop being the hidden registry.

It should consume the registry.

---

## Surface Eligibility

`0.124` should explicitly classify where functions are allowed.

This must include at least:

* `WHERE`
* `HAVING`
* scalar projection
* grouped projection
* scalar `ORDER BY`
* grouped `ORDER BY`

Important nuance:

This slice does not require every function to be valid in every surface.

It requires every surface restriction to be derived from one named capability
contract rather than from repeated local match ladders.

---

## Operator Alignment

Scalar operators and scalar functions are not the same syntax family, but they
are the same semantic class of problem.

Examples:

* `+`, `-`, `*`, `/`
* `ABS`, `ROUND`

`0.124` does not need to fully unify operators and functions into one runtime
implementation surface.

But it should add this constraint:

scalar operators must follow the same style of semantic contract as scalar
functions.

That means later work must not leave operators on one implicit model and
functions on a formalized one.

If full unification is too large for `0.124`, the slice should still make the
compatibility expectation explicit.

---

## Function Identity Stability

Function identity must be canonical before planner semantics attach.

This matters for:

* prepared execution
* caching
* fingerprinting
* canonicalization

Important rule:

lowering must canonicalize function identity before planner ownership begins.

Example:

* `CEIL` and `CEILING` may both be parser-accepted names
* they must not survive as separate semantic identities once lowered

The planner-owned function taxonomy must operate on canonical function identity,
not on syntax aliases.

---

## Coverage Completion Standard

`0.124` should not aim for PostgreSQL-level scalar coverage.

It should aim for "complete enough to support realistic pre-join SQL without
reopening function semantics during join work."

That means:

* arithmetic and boolean expression surface already supported by operators
* null-handling coverage complete and explicit
* basic text manipulation complete
* basic numeric transformation complete
* conditional / `CASE` semantics explicit and stable

This slice should prefer semantic completeness over breadth.

It is better to have one well-specified bounded function set than a larger but
informally governed one.

## Concrete Actions

### Step 1. Inventory Existing Scalar Function Surfaces

Map current function handling across:

* parser name acceptance
* parser function-call shape rules
* lowering-time function mapping
* planner type inference
* executor evaluation
* preview / fold helpers
* explain / label rendering

Mark each site as:

* canonical semantic owner candidate
* syntax adapter
* runtime consumer
* duplicated classification consumer

### Step 2. Introduce One Planner-Owned Function Specification Module

Create:

* canonical function taxonomy
* canonical function specification surface
* named capability / null / determinism facets

Recommended path:

* `db/query/plan/expr/function_semantics.rs`

### Step 3. Move Typing onto the Canonical Function Spec

Refactor type inference so it consumes the function specification instead of
owning implicit function grouping itself.

This is the first place the no-duplicate-semantics invariant must be enforced.

### Step 4. Move Lowering onto the Canonical Function Spec

Lowering should map parsed function tokens onto planner-owned function
classification without maintaining a second semantic table.

Lowering must canonicalize identity, not redefine semantics.

### Step 5. Move Evaluation onto Planner-Frozen Function Semantics

Executor evaluation should dispatch on the planner-owned function taxonomy while
avoiding local reclassification of capability or type families.

Allowed executor shape:

```rust
match function {
    Function::Lower => eval_lower(...),
}
```

Disallowed executor shape:

* any re-check of typing validity
* any capability decision
* any null-behavior categorization

### Step 6. Fill Coverage Gaps Only After the Registry Exists

If core scalar functions are still missing from the minimal complete set, add
them only after the semantic registry is in place.

That prevents the slice from degenerating into "add functions one by one."

### Step 7. Add Guardrails

Add structural tests that prevent:

* new scalar functions from landing without a canonical spec entry
* local function-capability ladders from regrowing outside the canonical owner
* parser / lowering / evaluator drift on the shipped scalar function set
* semantic `match Function::...` ladders outside the canonical specification
  owner and pure executor dispatch
* function identity enums or canonical function lists drifting out of
  alphabetical order

## Recommended Work Order

### `0.124.0` — Function Semantics Registry

Start here.

Deliver:

* canonical taxonomy
* function specification surface
* null / determinism / capability facets

Do not widen coverage yet unless necessary to establish the model.

### `0.124.1` — Typing and Lowering Centralization

Refactor:

* planner type inference
* lowering function mapping

so both consume the same canonical registry.

### `0.124.2` — Evaluation and Coverage Completion

Refactor:

* executor scalar function evaluation
* preview / fold helpers

to consume the planner-owned function taxonomy, then fill any remaining
minimal-complete scalar function gaps.

## Exit Condition

`0.124` is complete when:

* scalar functions have one canonical planner-owned specification surface
* function typing comes from that specification surface
* function capability and surface eligibility come from that specification
  surface
* null behavior is explicit per function family
* current scalar-function coverage reaches the declared minimal-complete pre-join
  set
* adding or changing one scalar function no longer requires hunting through
  parser, lowering, typing, and evaluator match ladders to rediscover its
  contract

Visible signs:

* fewer duplicated scalar-function match ladders
* clearer function coverage / capability auditability
* smaller blast radius for scalar-function changes
* join preparation no longer blocked on informal function behavior

## Non-Goals

`0.124` should not:

* become a join slice
* add broad SQL feature surface opportunistically
* turn into a user-defined function framework
* let lowering or executor reclaim scalar function semantic ownership
* add functions one at a time without the canonical registry

---

## Summary

`0.124` should formalize scalar function semantics before joins widen the
expression problem.

The main point is not "support more functions."

The main point is:

* one taxonomy
* one specification owner
* one typing / capability / null-semantics contract
* one minimal-complete scalar function surface for pre-join SQL

If `0.116` through `0.123` centralized meaning, flow, topology, and broad
structural classification, `0.124` should centralize the last major
under-formalized concept in the expression layer: scalar function semantics.

Success should be visible as this before/after shift:

Before:

* parser knows names
* lowering maps names
* type inference guesses types
* executor decides behavior

After:

* parser owns names only
* lowering owns identity only
* planner owns the full function contract
* executor performs execution only
