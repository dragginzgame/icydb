# 0.110 Grouped Semantic Alignment

## Thesis

`0.110` should bring grouped `HAVING` onto the same semantic ownership line that scalar `WHERE` now uses, instead of leaving grouped boolean handling as a structurally normalized but semantically second-tier path.

> **`0.110` aligns grouped `HAVING` with scalar `WHERE` by introducing bounded semantic canonicalization, canonical identity, and explain parity for already-admitted grouped boolean expression forms, without widening grouped surface area or reopening grouped execution ownership.**

This is a **semantic-alignment slice**, not a grouped-surface expansion slice.

---

## Problem statement

After `0.103` through `0.109.1`, scalar filtering is materially ahead of grouped filtering.

Scalar `WHERE` now has:

* planner-owned semantic `filter_expr`
* derived pushdown predicate where applicable
* explicit residual ownership
* bounded semantic canonicalization for searched `CASE`
* canonical query identity
* explain surfaces that reflect canonical semantic form

Grouped `HAVING` does not yet match that line.

Today grouped `HAVING` is closer to:

* structurally normalized boolean form
* planner-owned grouped expression semantics
* grouped validation and execution ownership
* but no scalar-style bounded semantic canonicalization
* no equivalent grouped identity follow-through
* no guarantee that semantically equivalent grouped boolean forms explain the same way

That leaves an architectural asymmetry:

* scalar and grouped filters are both planner-owned
* but only scalar filters get canonical semantic collapse for equivalent admitted forms

This creates three problems:

* grouped identity remains weaker than scalar identity
* grouped explain remains more representation-sensitive than scalar explain
* future grouped feature work risks repeating solved scalar problems in a second tier

---

## Design goal

Bring grouped `HAVING` up to the same semantic alignment standard as scalar `WHERE`, within the grouped surface that already exists.

That means:

* bounded semantic canonicalization for already-admitted grouped boolean forms
* grouped identity surfaces derive from canonical grouped semantic form
* grouped explain reflects canonical semantic form rather than spelling-sensitive structure
* no grouped execution or clause-admission widening occurs

The goal is:

> **make grouped `HAVING` semantically first-class on the same bounded canonicalization/identity/explain line as scalar `WHERE`, without widening grouped semantics beyond already-admitted forms.**

---

## Non-goals

`0.110` does **not** aim to:

* widen grouped expression admission
* add new grouped function families
* reopen generic grouped projection-expression widening
* change grouped execution order or ownership
* introduce grouped residual execution analogous to scalar row residual filtering
* widen prepared template admission
* introduce general boolean algebra simplification
* canonicalize arithmetic expressions
* add cost-based grouped planning
* change scalar semantics

This is not a grouped optimizer slice and not a grouped surface-completion slice.

---

## Explicit current boundary

Before `0.110`:

* grouped `HAVING` is planner-owned and validated
* grouped boolean normalization exists
* append-order normalization and grouped identity are better than before
* but semantic equivalence classes like scalar searched `CASE` do not yet collapse in grouped `HAVING`

So two grouped boolean filters may both be admitted and semantically equivalent, yet still differ in:

* canonical grouped expression shape
* grouped identity
* explain rendering

`0.110` exists to close that asymmetry for a bounded set of already-admitted grouped boolean forms.

---

## Normative shipped scope

`0.110` ships exactly these grouped-semantic alignment moves:

### 1. Bounded semantic canonicalization for grouped searched `CASE`

Grouped `HAVING` may canonicalize searched `CASE` into canonical boolean form when:

* the grouped expression is already admitted in grouped `HAVING`
* all `WHEN` predicates are already valid grouped boolean expressions
* all `THEN` / `ELSE` arms are already valid grouped boolean result expressions
* the transformation is semantically identical under the existing grouped boolean runtime rules

### 2. Canonical grouped semantic identity

When grouped `HAVING` canonicalizes to a planner-owned canonical grouped boolean form, grouped semantic identity surfaces must derive from that canonical form rather than pre-canonical spelling.

### 3. Grouped explain parity

Grouped explain surfaces that claim to describe semantic grouped filter shape must reflect the canonical grouped semantic form, not just the original structural spelling.

---

## Out of scope

Everything not listed above remains out of scope in `0.110`, including:

* grouped canonicalization for non-boolean result expressions
* grouped canonicalization for `BETWEEN`, `IN`, or other sugar
* grouped residual row filtering
* grouped surface widening by analogy with scalar WHERE
* broader grouped function-family expansion
* prepared template widening for grouped canonicalized forms

---

## Identity and template boundary

Canonical grouped semantic form in `0.110` changes only:

* grouped semantic fingerprint surfaces
* grouped shared query-plan identity surfaces
* grouped explain semantic-shape surfaces

It does not change:

* prepared template identity
* prepared template admission
* syntax-preserving debug surfaces
* raw SQL text caches

The explicit boundary for `0.110` is therefore:

* grouped semantic identity is canonicalized
* grouped structural/cache identity is canonicalized
* prepared/template identity is **not** canonicalized and remains syntax-bound

This line is normative across the implementation:

* grouped fingerprint / explain semantic-shape surfaces consume canonical grouped form
* grouped structural shared-cache identity consumes canonical grouped form
* prepared SQL/template caches keep syntax-bound statement identity and must not inherit grouped semantic canonicalization

So `0.110` may align grouped semantic identity where grouped semantic shape is already modeled, but it must not indirectly widen prepared admission by changing prepared-template identity or prepared-template legality.

---

## Normative model

### Rule 1 - grouped canonicalization is post-validation, pre-identity

The required grouped phase order is:

* parse
* grouped validation / legality checks
* bounded grouped semantic canonicalization
* grouped structural normalization
* grouped semantic identity projection
* grouped explain / grouped plan reuse surfaces

Grouped canonicalization must not:

* participate in clause admission
* bypass grouped validation
* introduce grouped expression forms not already valid in grouped `HAVING`

### Rule 2 - canonicalization applies only to already-admitted grouped forms

`0.110` does not create new grouped equivalence classes beyond the bounded class it explicitly ships.

It applies only to grouped searched `CASE` forms that are already admitted and already executable in grouped `HAVING`.

### Rule 3 - grouped searched `CASE` canonical form

For grouped `HAVING`, the canonicalization rule mirrors the scalar first-match form, but only inside the already-admitted grouped boolean space.

For `0.110`, searched `CASE` canonicalization applies only when `ELSE` is explicit.

Forms with omitted `ELSE` are out of scope unless the implementation models the omitted arm as `ELSE NULL` and proves semantic identity under grouped boolean semantics.

Given:

```sql
CASE
  WHEN p1 THEN r1
  WHEN p2 THEN r2
  ...
  ELSE rN
END
```

The grouped canonical form is:

```sql
(p1 AND r1)
OR (NOT p1 AND p2 AND r2)
OR (NOT p1 AND NOT p2 AND r3)
...
OR (NOT p1 AND ... AND rN)
```

Where:

* each `pᵢ` is already a valid grouped boolean predicate
* each `rᵢ` is already a valid grouped boolean result expression
* the transformed form remains within the admitted grouped boolean expression model

### Rule 4 - grouped three-valued boolean semantics must be preserved

Canonicalization is allowed only when the transformed grouped form is provably identical under the engine’s grouped boolean semantics.

That includes:

* true-only branch matching
* null behavior matching the existing grouped boolean rules
* no change in first-match semantics
* no aggregate-phase ownership widening

If these conditions are not met:

* canonicalization is skipped
* original grouped expression is preserved

This includes explicit `ELSE NULL`: it may canonicalize only when the expanded grouped form is provably identical under grouped boolean semantics.

### Rule 5 - grouped semantic identity derives from canonical grouped form

Where grouped semantic identity surfaces exist and claim to represent grouped filter semantics, they must derive from the canonical grouped `HAVING` form when one exists.

This applies to:

* grouped fingerprint / grouped plan-shape identity surfaces
* grouped reuse/cache identity surfaces that model grouped semantic shape
* grouped explain semantic surfaces

This does not apply to:

* syntax-preserving debug surfaces
* raw SQL spelling
* route-choice identity

Prepared-template identity and prepared-template admission remain outside this canonical grouped identity boundary.

### Rule 6 - grouped explain must not drift from grouped semantic identity

If grouped explain says it is describing grouped semantic filter shape, it must derive from the same canonical grouped expression tree that grouped identity uses.

Equivalent grouped forms should therefore produce identical:

* grouped semantic explain shape
* grouped identity projection
* grouped reuse identity, where applicable

Syntax-preserving grouped debug surfaces remain spelling-preserving and are not required to derive from canonical grouped semantic form.

### Rule 7 - no grouped execution widening

`0.110` is about grouped semantic alignment, not grouped execution redesign.

It must not:

* introduce scalar-style grouped residual row filtering
* add new grouped runtime lanes
* widen grouped route or planner access capabilities
* change prepared-template boundaries

### Rule 8 - post-canonicalization normalization is structural-only

Grouped semantic canonicalization produces the canonical semantic tree.

Later grouped structural normalization may apply only existing structural boolean rewrites, including:

* associative flattening
* redundant nesting removal
* deterministic child ordering
* duplicate collapse where already part of grouped normalization

Later passes must not introduce new semantic rewrites, additional simplification, or a second canonical semantic form.

### Rule 9 - no further simplification beyond the shipped rewrite

`0.110` does not introduce general boolean simplification.

For example, `CASE WHEN p THEN TRUE ELSE FALSE END` may canonicalize to the shipped first-match expansion, but it must not further collapse to `p`.

---

## Examples

### Example 1 - grouped searched `CASE` equivalence

Input A:

```sql
HAVING CASE WHEN COUNT(*) > 5 THEN TRUE ELSE SUM(score) > 100 END
```

Input B:

```sql
HAVING (COUNT(*) > 5 AND TRUE)
    OR (NOT (COUNT(*) > 5) AND SUM(score) > 100)
```

If both forms are already admitted in grouped `HAVING`, then after `0.110` they should produce:

* identical canonical grouped filter shape
* identical grouped explain semantic form
* identical grouped identity surfaces

### Example 2 - grouped form remains distinct when canonical forms differ

These remain distinct:

```sql
HAVING COUNT(*) > 5 OR SUM(score) > 100
```

```sql
HAVING COUNT(*) > 5 AND SUM(score) > 100
```

`0.110` does not merge non-equivalent grouped boolean forms.

### Example 3 - unsafe grouped canonicalization is skipped

If a grouped searched `CASE` form cannot be expanded without violating grouped boolean semantics or grouped legality rules, canonicalization is skipped and the original grouped expression remains.

---

## Identity contract

For grouped `HAVING`, canonical grouped semantic form owns the grouped-filter component of semantic identity when that canonical form exists.

Grouped semantic identity still includes the full grouped query shape, not just grouped filter form. So grouped identity must remain distinct across differing:

* grouped projection shape
* grouping keys
* grouped ordering shape
* limit/offset semantics where grouped query identity includes them

`0.110` aligns the grouped filter component of identity. It does not collapse full grouped-query identity to filter alone.

---

## Explain contract

Grouped explain should render grouped semantic filter shape from the canonical grouped form when one exists.

Grouped explain does not need to preserve original syntactic spelling.

It must not:

* silently mix original grouped spelling with canonical grouped identity
* show a grouped predicate-only rendering when grouped semantic form is richer
* drift from grouped identity surfaces

Syntax-preserving debug explain surfaces may preserve original grouped spelling, but they must remain clearly separate from canonical grouped semantic explain surfaces.

## Risks

### Risk 1 - grouped/scalar policy confusion

A scalar canonicalization rule could accidentally be widened into grouped clause-admission policy.

Mitigation:

* keep grouped canonicalization post-validation only
* apply only to already-admitted grouped forms

### Risk 2 - grouped semantic drift

Incorrect grouped searched `CASE` expansion could change grouped boolean semantics.

Mitigation:

* preserve first-match semantics exactly
* require identical grouped three-valued boolean behavior
* skip canonicalization when proof is not clean

### Risk 3 - scope creep into grouped surface widening

Contributors may use `0.110` to justify admitting more grouped forms.

Mitigation:

* state explicitly that `0.110` does not widen grouped admission

### Risk 4 - partial alignment

If grouped explain canonicalizes but grouped identity does not, or vice versa, grouped drift remains.

Mitigation:

* require one grouped canonical semantic authority for both surfaces

| ------------------------ | ---------------------------------------------------------------------------------------------- |
| Grouped canonicalization | already-admitted grouped searched `CASE` expands correctly into grouped canonical boolean form |
| Grouped equivalence      | canonical and original grouped forms produce identical grouped results                         |
| Grouped identity         | equivalent grouped canonicalized forms produce identical grouped semantic identity surfaces    |
| Grouped explain parity   | grouped explain derives from the same canonical grouped form as grouped identity               |
| Distinctness             | semantically distinct grouped forms remain distinct                                            |
| Boundary                 | no grouped admission widening or prepared-template widening occurs                             |

## Acceptance criteria

`0.110` is complete when:

* grouped searched `CASE` in grouped `HAVING` is canonicalized when safe
* semantically equivalent grouped canonical forms produce identical:

  * grouped semantic identity
  * grouped explain surfaces
* semantically distinct grouped forms remain distinct
* no grouped execution or prepared-boundary widening occurs
* grouped `HAVING` is no longer a structural-only second tier for the shipped bounded equivalence class

---

## Thesis sentence

`0.110` aligns grouped `HAVING` with scalar `WHERE` by introducing bounded semantic canonicalization, grouped identity alignment, and explain parity for already-admitted grouped boolean forms, without widening grouped surface area or grouped execution ownership.
