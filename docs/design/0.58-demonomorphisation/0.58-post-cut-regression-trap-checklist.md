# IcyDB 0.58 Post-Cut Regression Trap Checklist

## Purpose

Prevent reintroduction of:

- runtime generics (`<E: EntityKind>`)
- string-based identity (`EntityName`)
- hidden monomorphization via traits/helpers

This checklist should be enforced during PR review and CI.

## 1. HARD FAIL CHECKS (must never pass CI)

### 1.1 EntityKind in runtime modules

Forbidden in:

- `db/index/*`
- `db/cursor/*`
- `db/executor/*`
- `db/access/*`

Patterns to block:

```rust
<E: EntityKind>
impl<E: EntityKind>
where E: EntityKind
```

CI check:

```bash
rg -n "EntityKind" crates/icydb-core/src/db/{index,cursor,executor,access} && exit 1
```

### 1.2 EntityName in runtime

Forbidden:

```rust
EntityName
&str entity_name
```

CI check:

```bash
rg -n "EntityName" crates/icydb-core/src/db/{index,cursor,executor,access} && exit 1
```

### 1.3 Runtime hashing of identity

Forbidden:

```rust
hash(entity_name)
```

Identity must be:

- constant
- compile-time
- passed as `EntityTag`

### 1.4 No Generic Call Graph Reachability

No function reachable from runtime modules may be generic over `EntityKind`.

This includes transitive helper calls, not only direct function signatures in
`db/index`, `db/cursor`, `db/executor`, or `db/access`.

Common hidden failure mode:

```rust
fn process(row: &Row) {
    helper::<E>(row)
}
```

Even if `helper` is declared outside runtime modules, inlining reintroduces
monomorphized logic into runtime loops.

Review check:

- if runtime code can reach `fn ...<E: EntityKind>(...)`, it is a failure
- if runtime code can reach `impl<E: EntityKind> ...`, it is a failure
- if runtime code can reach helper traits generic over `EntityKind`, it is a
  failure
- this applies regardless of helper module location

### 1.5 Stable Tag Assignment Guarantee

`EntityTag` values must be stable across builds, machines, and compiler
versions.

Rule:

- tags are assigned from one authoritative identity registry
- changing existing tag values is storage-breaking
- adding new tags is safe
- renaming entities is safe only when tag values remain unchanged
- allocation must be monotonic and historical values must never be reused
- no reuse is allowed across schema resets or test environments in the same
  codebase
- persisted tag values are part of on-disk format; changes require explicit
  migration tooling
- collisions must fail build/test immediately
- ad-hoc reassignment in refactors is forbidden

### 1.6 Coarse CI Guard for Transitive Generic Leaks

If full call-graph analysis is not available, add a coarse CI guard over
runtime and runtime-adjacent helper modules.

Example baseline:

```bash
rg -n "<\\s*[A-Za-z_][A-Za-z0-9_]*\\s*:\\s*EntityKind|where\\s+[A-Za-z_][A-Za-z0-9_]*\\s*:\\s*EntityKind|impl\\s*<\\s*[A-Za-z_][A-Za-z0-9_]*\\s*:\\s*EntityKind" \
  crates/icydb-core/src/db/{index,cursor,executor,access} && exit 1
```

Expand scanned modules when runtime imports additional helper trees.

Bootstrap constraint:

- initial registry generation must be deterministic and must not depend on
  source-file discovery or macro expansion order

## 2. HIGH-RISK REINTRODUCTION PATTERNS

### 2.1 “Helper” generics sneaking into runtime

These are the most common regressions.

Forbidden example:

```rust
fn decode<T: EntityKind>(...)
```

or worse:

```rust
fn process<T: FieldValue>(...)
```

Why dangerous:

- looks harmless
- silently monomorphizes entire call chains

Fix:

- convert to `Value`
- or pass function pointer / enum

### 2.2 Trait-based polymorphism in hot loops

Forbidden example:

```rust
fn scan<T: Comparator>(...)
```

Replace with:

```rust
enum ComparatorKind { Asc, Desc }
```

### 2.3 Generic closures capturing type params

Forbidden example:

```rust
resolve(..., |x| decode::<E>(x))
```

Even if `resolve` is non-generic, this:

- forces monomorphization

Fix:

- closure must not reference generic type
- move decode outside loop

### 2.4 No Generic Closures Crossing Runtime Boundary

Planner/API layers must not pass generic closures or monomorphized functions
into runtime execution paths.

Forbidden pattern:

```rust
runtime_execute(..., |row| decode::<E>(row))
```

Required pattern:

- pass data (`EntityTag`, precomputed descriptors, enum modes)
- pass only monomorphic function pointers or operation descriptors
- keep typed decode/materialization at explicit outer boundary adapters

## 3. BOUNDARY VIOLATIONS

### 3.1 Typed decode inside scan loop

Forbidden example:

```rust
for entry in scan {
    let entity = decode::<E>(entry);
}
```

Correct:

```rust
let raw = scan(...);
let decoded = decode_batch::<E>(raw);
```

### 3.2 Passing type instead of identity

Forbidden example:

```rust
fn execute<E: EntityKind>(...)
```

Correct:

```rust
fn execute(entity: EntityTag, ...)
```

### 3.3 Reconstructing EntityName from tag in hot path

Forbidden example:

```rust
let name = registry.lookup(tag);
```

Rule:

- allowed only in:
  - logging
  - debug assertions
- never in scan loops

### 3.4 No Per-Row EntityTag Branching

Runtime scan loops must not branch on `EntityTag` per row.

Forbidden pattern:

```rust
for row in rows {
    match entity_tag {
        EntityTag(1) => ...,
        EntityTag(2) => ...,
        _ => ...,
    }
}
```

Rule:

- resolve entity-specific behavior before row iteration starts
- use precomputed execution descriptors/ops chosen once per operation

### 3.5 Resolve Entity Behavior Once Per Operation

`EntityTag` carries identity, but behavior dispatch must be resolved once per
operation, not per row.

Preferred shape:

```rust
let runtime = entity_runtime_registry.resolve(entity_tag)?;
scan_loop(rows, runtime.ops);
```

Avoid large per-row `match entity_tag` dispatch tables in hot loops.

Constraint:

- descriptor resolution must occur once per query/operation
- descriptor resolution must not allocate or perform map lookups inside scan
  loops
- descriptor lookup must be O(1) via static table/direct indexable structure
  (not hash maps)
- descriptor tables should be directly indexed by `EntityTag` (or one dense
  mapping), not multi-hop indirection chains
- resolved descriptors must remain immutable and reusable for full query
  execution

### 3.6 No Identity Re-Derivation

Runtime and planner code must not derive or infer entity identity from names,
schemas, or descriptors.

Rule:

- `EntityTag` is the only authoritative identity in execution paths
- name lookups are for debug/log/introspection only
- continuation/cursor identity envelopes must include `EntityTag`

## 4. TRAIT SYSTEM TRAPS

### 4.1 FieldValue recursion explosion

Risky:

```rust
impl<T: FieldValue> FieldValue for Option<T>
impl<T: FieldValue> FieldValue for Vec<Box<T>>
```

Rule:

- `FieldValue` must not be used in:
  - scan
  - executor loops
- only in:
  - parsing
  - planning

### 4.2 Blanket impls causing fan-out

Example:

```rust
impl<T> Something for T where ...
```

These multiply code paths invisibly.

## 5. DATA STRUCTURE REGRESSIONS

### 5.1 Reintroducing generics into keys

Forbidden example:

```rust
struct DataKey<E>
```

Must remain:

```rust
struct DataKey {
    entity: EntityTag,
    key: StorageKey,
}
```

### 5.2 Reintroducing string identity

Forbidden example:

```rust
struct DataKey {
    entity: String
}
```

### 5.3 Key Encoding Rule

`EntityTag` must be encoded as the leading component of runtime keys.

Why:

- guarantees partition isolation
- prevents mixed-entity range and prefix overlap

Encoding invariant:

- `EntityTag` key encoding must be fixed-width and big-endian for stable
  lexicographic partition ordering

## 6. CODEGEN / MACRO RISKS

### 6.1 Macros emitting generics into runtime

Macros often reintroduce:

```rust
fn foo<E: EntityKind>()
```

Rule:

- macros must emit:
  - `EntityTag`
  - not generics
- except at outer API layer

### 6.2 Tag generation drift

Risk:

- missing/unstable identity registry assignment
- accidental tag reassignment during refactors

Rule:

- tags must be:
  - stable
  - registry-assigned
- no derivation from entity names is allowed
- assignment must not depend on:
  - file ordering
  - macro expansion ordering
- collisions must fail build

Tag mutation semantics:

- changing existing tag values is storage-breaking
- adding new tags is safe
- renaming entities is safe only when tag values remain unchanged

### 6.3 Identity Registry Is Mandatory

One authoritative identity registry must govern tag assignment.

Registry invariants:

- one tag per entity
- no duplicate tags
- no missing entities
- renaming entities keeps tag values unchanged
- centralized across the workspace
- no crate-local independent tag assignment
- exactly one registry source of truth is allowed at build time (no layered
  generation)
- registry entries remain in deterministic stable order
- all runtime-visible entities participating in storage/execution are
  registered

## 7. PERFORMANCE-SIZE CROSSOVER TRAPS

### 7.1 Over-inlining large functions

After de-monomorphization, functions may still bloat via inlining.

Watch for:

- large decode functions
- scan loops

Fix (if needed):

```rust
#[inline(never)]
```

### 7.2 Enum explosion instead of generics

Enums are good, but this is bad:

```rust
enum BigEnum {
    Variant1(...),
    Variant2(...),
    ...
    Variant20(...)
}
```

Keep enums:

- small
- control-flow focused

## 8. VALIDATION CHECKLIST (run per PR)

Required commands:

```bash
cargo check --workspace
cargo test --workspace
```

Runtime-identity invariant checks:

```bash
rg -n "EntityKind|EntityName" crates/icydb-core/src/db/{index,cursor,executor,access} && exit 1
```

Transitive reachability invariant:

- any function reachable from runtime modules must not be generic over
  `EntityKind`, regardless of module location

Required partition-boundary regression:

- include at least one cross-entity range-scan test proving adjacent
  `EntityTag` partitions do not overlap or bleed

Size regression:

```bash
twiggy monos <wasm>
```

Look for:

- duplicate symbol patterns
- multiple instantiations of same logic

Index/cursor identity audit points:

- range-bound framing remains tag-partition-safe
- prefix scans do not rely on lexical `EntityName` ordering
- continuation anchors remain stable under `EntityTag` identity

## 9. QUICK MENTAL MODEL

When reviewing code, ask:

> Is this using a *type* where it should use *data*?

If yes, likely regression.

## 10. GOLDEN RULE

> Runtime must be uniform. Types belong at the edges.

If runtime behavior changes based on generic type, `0.58` is broken.
