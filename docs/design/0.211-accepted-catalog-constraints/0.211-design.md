# 0.211 Accepted-Catalog Constraints

Status: active

Design cut: 2026-07-16

Implementation order: after 0.210 closeout

## Decision Summary

IcyDB will add a first-class accepted constraint catalog.

The catalog gives every database constraint stable identity, a stable name,
accepted-schema ownership, deterministic current-form encoding, explicit
generated-versus-SQL-DDL origin, typed diagnostics, and one unified
introspection surface.

This does not replace the accepted structures that already enforce primary
keys, nullability, unique indexes, and relations. Those structures remain the
execution authorities for their existing semantics. The constraint catalog
references them and owns their shared identity and introspection contract.
Accepted snapshot integrity requires the references and structural facts to
agree exactly.

`CHECK` is the new semantic family. It is persisted as a canonical, versioned,
row-local expression and evaluated by a catalog-native engine over the final
accepted after-image. It contains no Rust function pointer, trait object,
callback, generated decoder dependency, or application hook.

Every declaration begins as an unpublished candidate. An exact bounded clean
proof may publish it directly as a validated constraint. Work that cannot or
must not complete in that call uses bounded resumable activation: it publishes
an accepted-schema-owned record that gates new writes, validates historical
rows through bounded primary-key pages, and becomes a fully accepted constraint
only after one complete clean sweep at an unchanged participating-store
revision vector.

The activation record is not a weak constraint and is not planner authority for
historical rows. It is temporary schema-migration authority with explicit
`EnforcingNewWrites` or `Validating` state. The accepted constraint collection
continues to contain only `Validated` constraints. Introspection merges both
collections into one lifecycle view.

### Non-Negotiable Invariants

1. Every accepted constraint has one nonzero stable ID and one nonempty name
   unique within its entity constraint namespace.
2. Accepted constraint IDs are never reused after drop.
3. Every accepted constraint is fully validated; incomplete work exists only
   as an explicit accepted-schema activation record.
4. Primary-key, not-null, unique-index, and relation execution remains owned by
   their current accepted structural contracts.
5. The constraint catalog references those contracts without duplicating their
   field lists, target definitions, key definitions, or delete behavior.
6. Every accepted `CHECK` expression is deterministic, row-local, bounded,
   canonical, type-checked against accepted field kinds, and executable without
   generated Rust code.
7. Accepted checks evaluate the final value intended for persistence, after
   sanitization, defaults, generated values, and managed-write synthesis.
8. Every supported mutation ingress evaluates applicable accepted checks and
   activation write gates before its commit marker.
9. `CHECK` rejects only `FALSE`; `TRUE` and SQL-style `UNKNOWN` satisfy the
   check. Null prohibition remains a separate not-null constraint.
10. Accepted schema, not generated validators, is runtime constraint authority.
11. Persisted constraint decoding is bounded, fallible, current-form only, and
    participates in accepted schema fingerprints.
12. No physical index ordinal is treated as stable schema identity.
13. Only `Validated` constraints may support planner assumptions or integrity
    claims about historical rows.
14. Constraint validation pages, findings, job receipts/checkpoints, and
    physical staging are bounded; no page exhaustion is mistaken for entity
    exhaustion.

## Normative Constraint Contract

The maintained contract is:

> Every database constraint is represented by accepted schema and has stable
> identity, name, origin, deterministic semantics, and typed diagnostics.
> Accepted constraints are fully validated and apply uniformly. A constraint
> being activated is represented separately, gates every relevant new write,
> and cannot be used as a historical-row assumption until validation completes.

For structural constraints:

> Primary-key, nullability, uniqueness, and relation structures remain their
> execution authorities. The accepted constraint catalog provides their stable
> constraint identity and unified introspection, and snapshot integrity proves
> exact agreement.

For checks:

> A `CHECK` constraint is a canonical accepted expression over one row. It is
> evaluated without application callbacks against the final canonical
> after-image before commit planning proceeds.

For application validation:

> Rust validators and sanitizers remain application behavior unless the schema
> explicitly declares an accepted check. Arbitrary Rust code is never inferred
> into persisted database semantics.

The central gap is therefore not that primary-key, not-null, unique, and
relation enforcement are absent. They already exist as accepted facts. The gap
is that there is no stable named constraint registry spanning those facts and
no accepted-native `CHECK` expression/evaluator.

The opening implementation audit must refresh this table and record:

- every accepted field, index, relation, schema codec, fingerprint, marker,

- every place dense index ordinals or relation names are treated as logical
  identity;
- every generated validator and sanitizer family and whether it can be
  represented by the proposed check language;
- every mutation ingress and schema row-rewrite path that produces a final
  accepted after-image;
- current DDL full-row validation and resource failure boundaries;
- current unique, relation, and primary-key integrity diagnostics;
- raw non-gzipped wasm size; and
- representative save and DDL validation costs before constraint evaluation.

## Problem Statement

### 1. Generated Validation Is Not Accepted Database Semantics

The derive layer knows validator declarations and emits visitor calls. Save
preflight invokes those calls for generated typed entities. The accepted schema
does not persist their identity, arguments, semantics, dependencies, or
diagnostic names.

Consequences include:

- SQL and tooling cannot introspect them as database constraints;
- a DDL-created field cannot receive the same rule through accepted metadata;
- schema migration cannot validate a candidate using accepted authority alone;
- an integrity checker cannot independently evaluate the rule;
- recovery and structural tools cannot render the rule without generated code;
- a generated validator implementation may change across application upgrades
  without an accepted schema identity change; and
- arbitrary custom callbacks cannot be made durable or deterministic safely.

### 2. Existing Constraints Have No Unified Stable Identity

Primary keys, nullability, uniqueness, and relations are discoverable through
different accepted structures. Introspection must understand each structure
independently, and not-null constraints have no independent name or ID.

An accepted constraint registry is needed for stable names, diagnostics, DDL
ownership, schema diffing, and a single tooling surface.

### 3. Physical Index Ordinals Are Not Constraint Identity

The current runtime `IndexId` embeds a dense per-entity ordinal used in physical
keys. Index drop/compaction may reassign surviving ordinals. A unique constraint
cannot use that physical identifier as its durable logical identity.

0.211 must separate stable schema-level index definition identity from physical
index placement.

### 4. Query Predicates Are Not Persisted Checks

The query predicate and SQL expression trees contain useful parser,
normalization, and scalar-semantics components, but their complete surfaces are
not valid accepted constraint languages. Query expressions may contain names,
parameters, aggregates, projection-only functions, membership operations,
missing-field semantics, or execution coercions that a durable row constraint
must not inherit accidentally.

A dedicated canonical constraint AST is required. It may reuse proven
low-level literal admission and scalar comparison helpers, but it owns a
smaller persisted semantic contract.

### 5. Exact One-Message Activation Does Not Scale

Adding a check, making a field non-nullable, building a unique index, or adding
a relation to a populated entity may require a complete authoritative scan and
derived-state build. Entity size cannot be assumed to fit one canister message.

Publishing the final structural contract before that work completes would let
planners and integrity checks assume an unproved fact. Keeping the candidate
entirely outside accepted schema would let mutation ingress bypass its new-write
gate. The safe split is an accepted-schema-owned activation record with narrow
write authority, followed by bounded validation and atomic promotion to a
fully accepted constraint.

## Goals

0.211 must:

1. add stable `ConstraintId` identity and a non-reusing per-entity allocation
   policy;
2. add one unique constraint-name namespace per accepted entity;
3. record generated-versus-`SqlDdl` origin for every accepted constraint;
4. present primary-key, not-null, unique-index, relation, and check constraints
   through one accepted catalog and introspection surface;
5. keep existing structural constraint owners authoritative for execution;
6. add stable schema-level index and relation identities suitable for
   constraint references;
7. add one bounded, deterministic, canonical `CHECK` expression language;
8. bind check field references to accepted `FieldId` values and exact kinds;
9. derive field dependencies from the canonical expression rather than persist
   a second mutable list;
10. evaluate accepted checks against final canonical after-images on every
    supported write ingress;
11. support generated and SQL-DDL-created checks without application callbacks;
12. activate constraints through bounded new-write enforcement and resumable
    historical validation;
13. keep accepted constraints fully validated while representing temporary
    activation state explicitly in accepted schema;
14. reuse 0.210's canonical-page and stable-revision proof principles without
    reusing its continuation wire blindly;
15. add typed constraint diagnostics and `SHOW CONSTRAINTS` introspection;
16. make constraint-aware field/index/relation mutation fail closed;
17. provide accepted-native constraint verification hooks for the bounded
    integrity checker designed in 0.212;
18. hard-cut old persisted schema forms without aliases or reconstruction; and
19. close with correctness, corruption, resource, performance, wasm-size, and
    source-complexity evidence.

## Non-Goals

0.211 does not:

- persist arbitrary Rust validators, sanitizers, function pointers, closures,
  trait objects, paths to callbacks, or application hooks;
- infer a database constraint from every existing `validator(...)` use;
- treat sanitization as a constraint;
- add weak, unchecked, permanently new-writes-only, or planner-visible
  partially validated constraints;
- treat page exhaustion, a stale checkpoint, or a changing revision as proof
  of complete validation;
- add cascades, `SET NULL`, `SET DEFAULT`, deferred relations, or update actions;
- reintroduce relation strength terminology;
- add DDL-created primary keys, relations, or unique constraints independent of
  the existing unique-index surface;
- add subqueries, aggregates, windows, joins, parameters, nondeterministic
  functions, application functions, or cross-row checks;
- add assertions spanning multiple entities or stores;
- admit whole composites or nested composite paths into check expressions;
- expose generated validator messages as stable database diagnostics;
- reuse the full query predicate AST as the persisted constraint wire;
- treat dense physical index ordinal as durable schema identity;
- automatically migrate pre-0.211 accepted snapshots; or
- keep legacy constraint-free current-form decoding beside the new catalog.

## Vocabulary

### Database Constraint

A database constraint is accepted-schema-owned, deterministic, introspectable,
independently evaluable, and enforced uniformly across supported write
ingresses.

### Application Validator

An application validator is generated or handwritten Rust behavior invoked by
the typed visitor framework. It may use application logic and custom messages.
It is not automatically a database constraint.

### Sanitizer

A sanitizer transforms authored values before final persistence admission. Its
output is subject to database constraints. The sanitizer itself is not
persisted as a constraint.

### Structural Constraint

Primary-key, not-null, unique-index, and relation constraints are structural.
Their execution semantics already live in accepted row layout, fields, indexes,
and relation contracts.

### Check Constraint

A check constraint owns one canonical row-local boolean expression. It has no
physical index or relation owner and is evaluated directly against an accepted
row after-image.

### Accepted Constraint

An accepted constraint has completed existing-row validation and belongs to the
current accepted schema fingerprint. Only accepted constraints may support
planner assumptions or historical integrity claims.

### Constraint Activation

A constraint activation is temporary accepted-schema migration authority. It
reserves the final constraint identity and semantics, gates relevant new writes,
owns or references bounded validation work, and is introspectable. It is not a
fully accepted constraint until promotion succeeds.

## Authority Model

Authority is divided by concern:

1. accepted schema owns constraint identity, names, origin, references, check
   expressions, and active migration write gates;
2. accepted row layout and primary-key fields own row identity;
3. accepted field contracts own nullability;
4. accepted index contracts own uniqueness and physical index derivation;
5. accepted relation contracts own target existence and restrictive delete
   safety;
6. accepted check expressions own row-local check semantics;
7. constraint catalog integrity proves structural constraint references match
   their accepted execution owners;
8. the schema-owned activation runner owns bounded historical proof and staged
   derived state without becoming planner authority;
9. compiled accepted checks and activation gates own fingerprint-bound runtime
   evaluation plans;
10. generated models may propose generated constraints but never reconstruct
   missing accepted constraints at runtime; and
11. SQL DDL is a frontend to the same catalog-native mutation semantics.

The catalog is not a second implementation of uniqueness, nullability, or
referential integrity. It is their identity and introspection registry. A
snapshot with a not-null constraint referencing a nullable field, a unique
constraint referencing a non-unique index, or a relation constraint referencing
the wrong relation is invalid accepted metadata.

## Accepted Constraint Model

The target semantic shape is:

~~~text
ConstraintId(u32)
SchemaIndexId(u32)
RelationId(u32)

AcceptedConstraint {
    id: ConstraintId,
    name: String,
    origin: ConstraintOrigin,
    kind: AcceptedConstraintKind,
}

ConstraintActivation {
    id: ConstraintId,
    name: String,
    origin: ConstraintOrigin,
    kind: ConstraintActivationKind,
    state: ConstraintActivationState,
    base_schema_fingerprint: AcceptedSchemaFingerprint,
    activation_epoch: u64,
}

ConstraintActivationState
  = EnforcingNewWrites
  | Validating

ConstraintOrigin
  = Generated
  | SqlDdl

AcceptedConstraintKind
  = PrimaryKey
  | NotNull {
        field_id: FieldId,
    }
  | Unique {
        index_id: SchemaIndexId,
    }
  | Relation {
        relation_id: RelationId,
    }
  | Check {
        expression: CanonicalConstraintExprV1,
    }

ConstraintActivationKind
  = NotNull {
        field_id: FieldId,
    }
  | Unique {
        index_id: SchemaIndexId,
    }
  | Relation {
        relation_id: RelationId,
    }
  | Check {
        expression: CanonicalConstraintExprV1,
    }
~~~

The exact Rust module names may follow local conventions, but these semantic
facts are fixed.

There is no `StrongRelation` variant. `Relation` means the complete current
relation contract. Introspection renders current `ON DELETE RESTRICT` behavior
from the relation contract; the constraint does not persist a general action
enum with unsupported values.

`Declared` is a request/draft state outside the accepted snapshot. Publication
of `ConstraintActivation` is the atomic point at which its new-write gate
becomes runtime authority. A direct exact clean publication instead inserts the
validated constraint and final structural owner atomically without exposing an
activation. `Validated` is derived for every value in `constraints`; it is not
redundantly persisted. Promotion atomically removes the activation and inserts
the accepted constraint with the same ID, name, origin, and canonical meaning.

The two collections are deliberate. Pending not-null activation must not make
historical nulls undecodable by prematurely flipping the accepted field's
global nullability. Pending uniqueness must not expose an incomplete index to
the planner. Pending relation activation must not expose an incomplete reverse
relation index to delete enforcement. Final structural owners publish only with
promotion.

Constraint IDs and names are unique across accepted constraints and activation
records. At most one activation exists for a given ID, name, structural owner,
or candidate owner.

There is no free-form `semantics_version: u32`. The check AST and its codec are
explicitly `V1`. Before 1.0, changing semantics requires a current-form format
cut, not simultaneous execution of arbitrary old versions.

## Stable Identity

### Constraint IDs

`ConstraintId` is nonzero and stable within one entity's accepted history.

Initial generated snapshots allocate IDs deterministically from canonical
constraint order. Later schema mutations preserve IDs for surviving constraints
and allocate new IDs from a persisted high-water mark. Drop does not compact IDs
and a later constraint never reuses a retired ID.

Activation reserves the final `ConstraintId` before its write gate publishes.
Promotion preserves that ID. Abort/drop retires it; a later retry receives a
new ID so stale validation jobs cannot attach to a recreated constraint.

Direct exact publication allocates the final ID in the same atomic accepted
schema publication as the validated constraint. A candidate that fails or
exceeds a bound while still unpublished neither advances the persisted
allocator nor retires an ID.

The accepted entity snapshot therefore carries the next constraint allocation
value or equivalent non-reusing allocator state.

Constraint identity survives:

- field rename;
- index physical ordinal compaction;
- unrelated field, index, relation, and constraint additions or drops; and
- accepted schema version changes that do not replace the constraint.

Dropping and recreating a logically similar constraint creates a new ID.

### Constraint Names

Constraint names are nonempty and unique across all constraint kinds within one
accepted entity. They use the accepted SQL/schema identifier normalization
policy.

Generated and SQL-authored checks require an explicit stable name. Structural
constraints without authored names receive deterministic system names at first
catalog creation; those names are then persisted and do not silently change on
field rename.

The initial naming policy must cover:

- one primary-key constraint;
- every non-null field;
- every unique index; and
- every relation.

Constraint diagnostics always include the accepted name and ID. They may also
render current field/index/relation names for context.

### Constraint Origin

Origin belongs only to the constraint registry; structural owners do not
repeat it.

- the primary-key constraint is `Generated`;
- a not-null constraint proposed by generated field metadata is `Generated`;
- a not-null constraint added through SQL DDL is `SqlDdl`;
- a unique constraint inherits the creation origin of its paired accepted
  index;
- relation constraints are `Generated` in 0.211; and
- checks use the origin of their explicit generated declaration or SQL DDL
  creation.

Origin controls reconciliation and supported drop surfaces. It does not change
enforcement strength.

### Schema-Level Index Identity

0.211 introduces `SchemaIndexId`, stored in `PersistedIndexSnapshot`. It is
stable logical accepted identity. The existing runtime `db::index::key::IndexId`
continues to carry entity tag plus dense physical ordinal for index-key framing.

When index drop compacts ordinals:

- surviving `SchemaIndexId` values do not change;
- physical index ordinals may change under existing publication rules;
- unique constraints continue to reference `SchemaIndexId`; and
- accepted snapshot integrity resolves each logical ID to exactly one index.

This split prevents physical layout from becoming constraint identity.

### Relation Identity

0.211 introduces stable `RelationId` on accepted relation snapshots. Generated
relations allocate IDs deterministically and preserve them across unrelated
schema changes. Origin remains on the paired constraint rather than being
duplicated on the relation owner.

The relation constraint references only `RelationId`. Target entity path,
local fields, cardinality, target primary-key resolution, and restrictive
delete semantics remain owned by the accepted relation contract.

## Structural Constraint Closure

Every accepted snapshot must satisfy these one-to-one rules:

- exactly one `PrimaryKey` constraint exists and corresponds to the snapshot's
  ordered primary-key field IDs;
- every field with `nullable == false` has exactly one `NotNull` constraint;
- no nullable field has a `NotNull` constraint;
- every unique index has exactly one `Unique` constraint;
- no non-unique index has a `Unique` constraint;
- every accepted relation has exactly one `Relation` constraint;
- every structural constraint reference resolves to exactly one accepted
  owner; and
- every constraint name and ID is unique.

Activation closure is separate and state-aware:

- a pending not-null activation references one currently nullable accepted
  field and has no accepted not-null twin;
- a pending unique activation references one reserved candidate
  `SchemaIndexId` and staged index definition that are not planner-visible;
- a pending relation activation references one reserved candidate `RelationId`
  and candidate relation definition whose reverse projection is not yet
  delete-authoritative;
- a pending check owns its canonical expression directly;
- no accepted constraint and activation may claim the same semantic owner;
- the activation's base fingerprint and epoch match its schema-owned validation
  job; and
- unrelated schema mutation for the entity is blocked until the activation is
  promoted or aborted, avoiding implicit candidate rebasing.

The validated catalog intentionally does not repeat primary-key fields, index
key items, relation local/target fields, cardinality, or referential actions.
An activation may reference an unpublished candidate owner only for the life of
the migration; promotion replaces that candidate with the final structural
owner atomically.

Snapshot construction, codec decode, reconciliation, mutation staging, and
accepted publication all run the same closure validation.

## Generated Proposal Model

Generated entity proposals gain constraint proposal metadata.

Structural proposals are derived from existing generated facts:

- primary-key metadata proposes the primary-key constraint;
- non-null fields propose not-null constraints;
- unique index proposals propose unique constraints; and
- relation proposals propose relation constraints.

Initial bootstrap may place generated constraints directly in the accepted
collection only when authoritative emptiness is proved in the same bounded
publication context. Adding or tightening any constraint after rows may exist
uses activation; generated origin is not a validation bypass.

Generated checks use an explicit schema declaration with a required name and a
row-local expression. The intended style follows generated filtered-index
predicate precedent:

~~~text
constraint(
    name = "balance_nonnegative",
    check = "balance >= 0",
)

constraint(
    name = "active_tier",
    check = "tier IN ('bronze', 'silver', 'gold')",
)
~~~

The derive parses and type-checks the expression at build time against the
generated field proposal, then emits a structured check proposal. Runtime
reconciliation resolves field names to accepted IDs and independently verifies
the complete accepted expression.

The SQL text is proposal input and diagnostic source, not persisted semantic
authority. Accepted schema persists the canonical bound AST and renders
canonical SQL using current accepted field names.

Existing `validator(...)` annotations remain application validators. They are
not silently promoted. Authors who require database semantics must declare a
named `constraint(..., check = "...")`.

A later ergonomic feature may translate a closed set of built-in validators
into explicit check proposal syntax, but only if the schema declaration names
the constraint and the lowering is exact. Arbitrary validator paths never
become persisted callbacks.

## Canonical Check Expression V1

`CanonicalConstraintExprV1` is a dedicated accepted AST. It binds fields by
`FieldId`, embeds admitted typed literals, and contains only deterministic
row-local operations.

Conceptually:

~~~text
CanonicalConstraintExprV1
  = True
  | False
  | Not(expr)
  | And([expr, ...])
  | Or([expr, ...])
  | Compare {
        left: ConstraintValueExprV1,
        op: Eq | Ne | Lt | Lte | Gt | Gte,
        right: ConstraintValueExprV1,
    }
  | IsNull(value)
  | IsNotNull(value)
  | MultipleOf {
        value: ConstraintValueExprV1,
        factor: AcceptedLiteral,
    }

ConstraintValueExprV1
  = Field(FieldId)
  | Literal(AcceptedLiteral)
  | CharLength(FieldId)
  | OctetLength(FieldId)
  | Cardinality(FieldId)
~~~

`BETWEEN` and inclusive generated range validators lower to `Gte AND Lte`.
Length equality, minimum, maximum, and range validators lower to comparisons
over the appropriate length expression. `MultipleOf` remains explicit because
the initial query predicate AST has no durable modulus contract.

Bounded enum membership is frontend syntax, not a second persisted node. An
expression of the form `enum_field IN (variant, ...)` lowers to an `Or` of
same-enum equality comparisons before canonical encoding. The list must be
nonempty, contain only admitted non-null literals from the field's exact enum
catalog, and fit the dedicated membership-item bound. Canonical `Or` sorting
and deduplication then make generated `IN`, SQL `IN`, and an explicit equality
disjunction converge to identical accepted bytes and fingerprints.

A null enum field evaluates the lowered comparisons under the ordinary
three-valued rules and therefore produces `Unknown`; enum membership does not
imply not-null. Canonical introspection renders the accepted lowered expression
and does not preserve or reconstruct the source `IN` spelling.

V1 deliberately excludes general arithmetic. It also excludes collection
membership other than the bounded enum sugar above, text patterns,
missing-field tests, arbitrary scalar functions, case expressions, and nested
paths.

### Supported Kinds

V1 comparison support is capability-driven from exact accepted field kinds:

- boolean and exact scalar equality where accepted equality is defined;
- ordering only where accepted ordering is defined;
- field-to-field comparison only for the same exact accepted kind in the
  opening line;
- field-to-literal comparison after the literal is admitted to the field's
  exact kind at bind time;
- enum equality only within the same accepted enum type;
- bounded `IN` membership for one enum field and literals admitted to that
  field's exact enum catalog, lowered to equality disjunctions;
- `CharLength` for text, counting Unicode scalar values consistently with
  accepted text bounds;
- `OctetLength` for blobs;
- `Cardinality` for declared list, set, and map fields; and
- `MultipleOf` for exact integer and decimal kinds with a nonzero admitted
  factor.

Floating-point `MultipleOf`, cross-kind numeric comparison, relations,
composites as whole values, and nested composite paths are rejected in the
current form.

### Literal Contract

Constraint literals are admitted during proposal/DDL binding against one exact
expected accepted kind. Persisted expressions carry canonical literal values
and any enum catalog identity required by that kind.

Runtime evaluation performs no text parsing, enum-name resolution, numeric
narrowing/widening, JSON conversion, or kind inference.

### Canonicalization

Canonicalization must:

- bind every field reference to `FieldId`;
- flatten nested `And` and `Or` nodes;
- sort commutative children by canonical encoded bytes;
- remove duplicate identical children;
- normalize `BETWEEN` and range sugar to comparisons;
- lower bounded enum `IN` syntax to same-enum equality comparisons joined by
  canonical `Or`;
- enforce the enum membership-item bound before lowering;
- normalize literal representation through accepted value admission;
- validate operator capabilities and exact operand kinds;
- validate nonzero `MultipleOf` factors;
- enforce maximum expression depth, node count, child count, literal bytes, and
  final encoded bytes; and
- derive a sorted unique dependency set by traversing field references.

The dependency set is not persisted beside the AST. Runtime compilation may
cache it under accepted fingerprint authority.

### Null Semantics

Constraint evaluation returns `True`, `False`, or `Unknown`.

- `CHECK` rejects only `False`;
- `True` and `Unknown` satisfy the check;
- comparison with null produces `Unknown`;
- boolean composition uses SQL three-valued logic;
- `IS NULL` and `IS NOT NULL` always produce `True` or `False`; and
- a not-null guarantee is represented separately by `NotNull`.

This matches SQL `CHECK` behavior and the existing convention that optional
application validators skip absent values. Authors requiring a value must pair
the check with accepted non-nullability.

An impossible type mismatch, invalid accepted literal, corrupt opcode, or
unreadable row slot is an internal/corruption error. It is not converted to
`Unknown` or a user constraint violation.

## Bounded Constraint Activation

### Lifecycle

The logical lifecycle is:

~~~text
Declared (unpublished request or draft)
    -> Validated (exact bounded clean publication)

Declared (unpublished request or draft)
    -> EnforcingNewWrites (accepted activation record)
    -> Validating (accepted activation record + validation job)
    -> Validated (accepted constraint + final structural owner)

EnforcingNewWrites or Validating
    -> Aborted (activation removed, reserved IDs retired, staging unreachable)
~~~

`Declared` is not live runtime authority and does not appear in the accepted
constraint collection. Direct publication is legal only after exact bounded
clean exhaustion in the same message. The transition to `EnforcingNewWrites`
is marker-bound accepted-schema publication. No write can observe the
activation metadata without also observing its gate.

`EnforcingNewWrites` is temporary migration state, not a supported steady-state
integrity level. The constraint is visible and diagnosable, but callers and the
planner are told explicitly that historical rows are unproved.

The transition to `Validating` creates or resets one schema-owned validation
job. A concurrent data revision change resets proof progress within
`Validating`; it does not weaken or remove the new-write gate. Only a complete
clean read-only Verify sweep may promote to `Validated`.

### Validation Job Authority

Validation progress is stored separately from the semantic constraint record
so page advancement does not create a new constraint meaning or planner
fingerprint on every call.

Conceptually:

~~~text
ConstraintValidationJob {
    entity: EntityIdentity,
    constraint_id: ConstraintId,
    activation_epoch: u64,
    activation_fingerprint: Fingerprint,
    base_schema_fingerprint: AcceptedSchemaFingerprint,
    phase: ConstraintValidationPhase,
    checkpoint: Option<PrimaryKeyValue>,
    captured_store_revisions: Option<StoreRevisionVector>,
    staged_generation: Option<StagedGenerationId>,
    rows_scanned: u64,
    findings_seen: u64,
    restarts: u64,
    last_receipt: Option<ConstraintValidationReceipt>,
}

ConstraintValidationPhase
  = Forward
  | Verify
~~~

The job is catalog-native durable migration state, not a generated model,
session pointer, or application callback. Job publication and advancement use
typed marker/recovery semantics. Job storage does not advance any data revision
whose stability the job is trying to prove.

The job does not persist an unbounded finding list. It retains at most one
bounded finding receipt with a monotonic page sequence. A retry without
acknowledging that sequence returns the same receipt; an acknowledged request
may advance. This prevents response loss from silently dropping migration
findings. Aggregate counters are saturating and diagnostic; promotion still
requires zero findings in the current stable Verify sweep.

The activation fingerprint covers ID, name, origin, candidate kind, canonical
check expression or candidate structural owner, write-gate policy, and
activation epoch. A dropped and recreated constraint cannot resume an old job.

### Bounded Page Contract

Validation reuses 0.210's Forward/Verify proof shape.

Forward pages:

1. verify entity identity, activation ID/epoch/fingerprint, base schema
   identity, job phase, and staging identity;
2. scan rows strictly after the canonical primary-key checkpoint;
3. evaluate historical row-local facts and build/converge staged unique or
   reverse state where applicable;
4. stop before exceeding row, decoded-byte, finding-count, expression-work,
   staged-write, heap, or instruction bounds;
5. return the last fully classified primary key as the next checkpoint;
6. never treat a full page as keyspace exhaustion; and
7. after clean authoritative suffix exhaustion and marker clearance, capture
   every participating revision and enter `Verify` from the beginning.

Forward may mutate only isolated staging and job state. It does not publish a
structural owner. If Forward found historical violations, the job remains
unpromotable until explicit repairs change the source revision and a later
Forward pass is clean.

Verify pages are read-only over rows and staging:

1. compare current participating revisions with the captured vector before
   scanning;
2. return `Forward(start)` with explicit `revision_changed` if any differ;
3. scan strictly after the Verify primary-key checkpoint;
4. if a row violates the candidate or staged derived state is incomplete or
   divergent, return `Forward(start)` with a bounded finding/residual-work
   result;
5. advance only past fully classified rows within the same hard page bounds;
6. on authoritative exhaustion, recheck every captured revision; and
7. promote only if the complete Verify sweep recorded zero findings and the
   final vector is unchanged.

The captured vector includes every data store whose mutations can change the
proof. A row-local check or not-null activation normally captures the source
store. Relation activation also captures target stores. Candidate staging uses
its own generation identity and revision; Verify never writes it.

The opening implementation uses the existing journaled-store durable mutation
revision principle from 0.210. A storage mode without a trustworthy
multi-message revision may use a bounded exact fast path only when exhaustion
is proved in one call; otherwise activation rejects with a typed unsupported
resource error rather than overclaiming completion.

Revision invalidation is deliberately conservative and may restart Verify
because of an unrelated write. Entity-local revisions or mutation-log catch-up
are future optimizations, not weaker V1 proof rules.

### Per-Kind Write Gates

New-write enforcement is derived by constraint kind:

- `Check`: every final after-image evaluates the canonical expression;
- `NotNull`: every final after-image must carry a non-null value for the target
  field even though historical decoding still uses the nullable accepted-before
  field contract;
- `Unique`: the candidate index is incomplete and cannot prove absence against
  unscanned history, so inserts and updates that author or change candidate key
  or predicate-dependency fields are blocked during activation; unrelated
  updates may proceed, and deletes update staging where an entry exists;
- `Relation`: new source inserts and relation-field updates must resolve every
  target and dual-write the staged reverse projection; source deletes remove
  staged entries where present; and target deletes are conservatively blocked
  until reverse authority is complete.

The unique and relation barriers are explicit migration conflicts with stable
diagnostic codes. They are not silent partial enforcement. A later design may
replace them with a sound bounded mutation-log protocol, but it may not accept
a write using an incomplete index.

Every typed, structural, SQL, batch, bulk, import, and schema-rewrite ingress
must consult activation gates alongside accepted constraints. An ingress that
cannot do so rejects while an applicable activation exists.

### Findings And Repair Boundary

Historical violations discovered during activation are migration validation
findings, not evidence that previously accepted state was corrupt. Each finding
contains stable constraint identity, entity, canonical primary key, bounded
field paths, error class, and diagnostic code.

Validation may continue to collect bounded pages of findings. Where the gate
can admit the corrected after-image, repair is an explicit normal mutation and
must pass that gate. Any repair changes a captured revision and forces the later
proof sweep to restart. The activation runner never repairs, coerces, deletes,
or rewrites a row implicitly.

Unique activation is the conservative exception to in-place repair: because
candidate key/dependency changes are blocked while the index is incomplete, an
operator must abort the activation, repair duplicate/key-membership data under
accepted-before, and start a new activation. Source-row deletion may remain
available when the documented gate can stage it safely. No privileged repair
bypass writes through an incomplete unique index.

### Atomic Promotion

After a clean stable sweep, one marker-bound publication performs the
kind-specific final transition:

- `Check` moves the canonical expression from activation to accepted
  constraint;
- `NotNull` flips accepted field nullability and installs its accepted
  constraint;
- `Unique` promotes the fully verified staged index definition and physical
  generation with its accepted constraint;
- `Relation` promotes the candidate relation and complete reverse generation,
  removes the broad target-delete migration barrier, and enables selective
  immediate `RESTRICT` from authoritative reverse state.

The same publication removes the activation and validation job. Interruption
recovers to either the complete before-state or complete promoted state. There
is no interval where planner-visible metadata outruns physical authority.

The direct exact clean path performs the same final accepted publication
without first exposing an activation or validation job. Failure before that
publication leaves accepted schema and allocator state unchanged.

### Abort And Staleness

Aborting removes the activation gate and job atomically, retires the reserved
constraint/index/relation identities, and makes staged generations unreachable.
Cleanup may be resumable only when unreachable bytes cannot affect runtime
authority.

Schema changes for the same entity are blocked while activation is live except
for validation, explicit abort/drop, and the final promotion. Accepted schema
replacement, activation epoch mismatch, missing staging, corrupt job state, or
unsupported store revision authority fails closed.

## Constraint Compilation And Evaluation

Accepted checks and pending check gates compile once per accepted schema
fingerprint into bounded runtime programs:

~~~text
accepted FieldId references
    -> accepted slots and exact codecs
    -> bounded constraint program
    -> final canonical after-image evaluation
~~~

The compiled program retains:

- constraint ID and name;
- pre-resolved field slots;
- exact accepted operand kinds and codecs;
- admitted canonical literals;
- bounded operation sequence or tree; and
- derived field dependencies.

Cache identity includes accepted entity identity, accepted schema version,
accepted fingerprint, and activation epoch when applicable. A schema or
activation change cannot reuse a stale check program.

Evaluation should borrow already decoded canonical slot values where the write
pipeline has them. It must not materialize the generated entity a second time
or decode the complete row once per constraint.

Accepted checks and activation gates run in stable `ConstraintId` order. The
first failure returns a typed constraint violation containing entity,
constraint ID, constraint name, canonical primary key when available, bounded
field paths, error class, and diagnostic code. Diagnostics do not depend on
generated validator text.

## Write Admission Order

The normative order is:

1. resolve accepted schema and complete authored-value/default/generated-value
   handling;
2. run applicable sanitizers and managed-write synthesis;
3. run application validators where the ingress has generated application
   behavior;
4. construct and admit the complete final canonical after-image;
5. evaluate accepted `CHECK` constraints and pending row-local activation gates
   against that after-image;
6. validate primary-key identity, uniqueness, relations, and other accepted
   cross-row/index facts in the existing preflight owners;
7. apply pending unique/relation migration barriers and stage permitted
   activation-derived effects;
8. finish row, index, reverse-relation, marker, journal, and response staging;
9. publish the commit marker only after all fallible work succeeds.

Implementations may combine adjacent passes or validate existing structural
facts earlier when semantics are unchanged. The fixed guarantee is that checks
observe final persisted values and all constraints finish before marker
publication.

An application validator may reject before a database check. That does not
make it accepted authority. An ingress without generated application behavior
still runs accepted checks and activation gates.

## Mutation Boundary Matrix

| Boundary | 0.211 constraint policy |
|---|---|
| Typed create/save | Application validation where declared, then accepted constraints and activation gates |
| Structural create/save | Accepted constraints and activation gates independent of generated fallback; current typed preflight may remain defense in depth |
| SQL INSERT/UPDATE | Same accepted constraints and gates over the final SQL after-image |
| Atomic batch | Every row satisfies every applicable constraint/gate before one batch marker; a migration barrier rejects the whole batch |
| Non-atomic batch | Each row satisfies constraints/gates before its individual commit; prior committed prefix remains per existing contract |
| Exact bulk update | Every frozen after-image satisfies constraints/gates before the atomic marker |
| Resumable bulk update | Each batch satisfies constraints/gates under pinned/revalidated accepted fingerprint; migration barriers may make the plan ineligible |
| Field default/generation | Constraints and gates see the materialized default/generated final value |
| Schema row rewrite/backfill | Every accepted-after row satisfies accepted-after constraints/gates before schema publication |
| DDL `ADD CONSTRAINT CHECK` | Plain `ADD` publishes a validated constraint only after exact bounded full validation; `NOT VALID` publishes an `EnforcingNewWrites` activation for resumable validation |
| Recovery replay | Does not rerun checks; replays only marker-owned bytes under matching accepted fingerprint |
| Future import/restore | Must evaluate accepted constraints/gates as fresh row ingress unless a separately trusted admitted-image contract proves equivalence |
| Index projection | Continues only after row admission, including constraints/gates |
| Relation projection | Continues only after row admission, including constraints/gates |

Recovery remains replay, not fresh application validation. It verifies bounded
current-form envelopes and matching accepted authority, then mechanically
completes the already admitted commit.

## SQL DDL Surface

0.211 adds the narrow opening surface:

~~~sql
ALTER TABLE entity
ADD CONSTRAINT constraint_name CHECK (expression)

ALTER TABLE entity
ADD CONSTRAINT constraint_name CHECK (expression) NOT VALID

ALTER TABLE entity
DROP CONSTRAINT constraint_name

ALTER TABLE entity
DROP CONSTRAINT IF EXISTS constraint_name

ALTER TABLE entity
VALIDATE CONSTRAINT constraint_name

ALTER TABLE entity
VALIDATE CONSTRAINT constraint_name AFTER page_sequence
~~~

`ADD CONSTRAINT CHECK` accepts only the V1 expression subset. It does not accept
parameters. Bounded enum `IN` is accepted only as the canonical lowering
defined above; general SQL `IN`, subquery `IN`, and collection membership are
rejected. Binding resolves field names against accepted-before and produces a
canonical candidate.

Plain `ADD` has validating semantics. It may publish the new constraint
directly as `Validated` only when one bounded call proves authoritative entity
exhaustion and every historical row satisfies the candidate. If that exact
proof exceeds a row, byte, expression, heap, or instruction bound, the call
returns a typed resource outcome and changes nothing: it publishes no
constraint, activation, validation job, staging state, or row effect.

`ADD ... NOT VALID` is the explicit resumable form. It atomically publishes the
candidate as an `EnforcingNewWrites` activation and does not claim that
historical rows have been scanned. The write gate is live immediately; the
candidate remains unavailable for planner or historical-integrity assumptions
until `VALIDATE CONSTRAINT` completes and promotion succeeds. `NOT VALID` never
creates a permanent new-writes-only constraint.

`VALIDATE CONSTRAINT` starts the job or returns its current cached page.
`AFTER page_sequence` acknowledges that page and advances at most one bounded
validation page. Each response contains constraint/job identity, page sequence,
state, revision status, rows scanned, bounded findings, and completion status.
Callers continue until it reports `Validated`, repair findings explicitly, or
abort the activation. Page sizing is engine-owned within hard caps rather than
an untrusted promise of completion.

DDL-created checks may reference generated- or SQL-DDL-owned accepted fields.
The check itself is `SqlDdl`-owned. Removing a referenced field is blocked until
the dependent DDL constraint is dropped. Generated reconciliation preserves
DDL-owned constraints that remain valid.

`DROP CONSTRAINT` in the opening line drops or aborts only SQL-DDL-owned check
constraints. Generated checks must be removed from generated schema and
reconciled. Structural constraints continue to use their existing mutation
surfaces:

- primary key is not DDL-mutable;
- `SET NOT NULL` starts not-null activation and `DROP NOT NULL` aborts a pending
  activation or removes the accepted constraint under existing ownership
  rules;
- `CREATE UNIQUE INDEX` starts unique activation and `DROP INDEX` aborts its
  pending activation or removes the accepted pair; and
- relations remain generated-schema-owned.

`DROP CONSTRAINT` targeting a structural or generated-owned constraint rejects
with a typed ownership/unsupported-operation error. No alternate drop path is
silently selected.

### Relation DDL Boundary

0.211 does not add `FOREIGN KEY` syntax. The future surface is reserved for the
catalog-level design in [Entity Lifecycle and Relation
DDL](../ideas/entity-lifecycle-and-relation-ddl.md). It must lower into the same
accepted relation owner, activation state machine, reverse-generation staging,
target-existence proof, and atomic promotion defined here; it may not create a
second SQL-only relation implementation.

The reserved semantics remain narrow: accepted primary-key target identity,
immediate checking, and `ON DELETE RESTRICT`. There is no unchecked relation,
alternate-key target, deferred check, cascade, `SET NULL`, or `SET DEFAULT`
meaning hidden behind future syntax. Until that complete catalog transition is
implemented, `FOREIGN KEY` DDL fails closed.

## Generated Reconciliation During Activation

Generated reconciliation that adds a generated check, relation, or not-null
fact publishes the same activation record and uses the same runner. A matching
activation counts as reconciliation in progress; repeated startup contact must
preserve its ID, epoch, job, and staging rather than restarting it.

If the generated proposal changes while activation is live, reconciliation
rejects as a stale migration until the operator aborts the old activation or
the exact same canonical proposal is restored. Generated models never execute
the historical scan and never reconstruct missing activation state.

## Schema Mutation Dependencies

Constraint dependencies participate in every schema mutation:

- field rename preserves field ID and constraint semantics; canonical SQL and
  diagnostics render the current field name;
- field drop rejects while any check references the field;
- dense field-ID reassignment during supported row-layout mutation rewrites
  every bound check expression through one validated ID map;
- field kind change rejects unless every dependent expression can be rebound
  with identical admitted semantics and existing rows validate;
- nullability tightening creates an activation and flips the field contract
  only on promotion; loosening removes the accepted constraint atomically;
- unique-index drop aborts its activation or removes its paired accepted
  constraint under the existing unique-index ownership surface;
- index ordinal compaction preserves `SchemaIndexId` and constraint identity;
- relation removal removes its paired generated relation constraint through
  generated reconciliation; and
- constraint-name conflicts reject regardless of constraint kind or origin.

Unknown dependencies or failed remapping reject before publication. Mutation
does not keep stale display SQL as semantic fallback.

## Unique-Index Pairing

In 0.211, uniqueness remains an index property enforced by existing index
owners. The constraint registry makes that property visible as a database
constraint.

Rules:

- `CREATE UNIQUE INDEX name ...` reserves one `SchemaIndexId`, creates a staged
  candidate index, and publishes one SQL-DDL-origin unique activation;
- generated unique index proposals create generated-origin unique activations;
- the initial constraint name matches the index name;
- only promotion makes the index planner-visible and installs the accepted
  unique constraint;
- `DROP INDEX` atomically aborts the activation or removes the accepted pair;
- non-unique indexes have no unique constraint; and
- there is no separate `ADD CONSTRAINT UNIQUE` surface in 0.211.

This preserves current semantics while leaving a clean future path for
constraint-owned backing indexes if that feature is designed explicitly.

## Relation Constraint Pairing

Every validated declared relation has one generated-origin relation constraint.
Adding a relation to a populated entity first creates its generated-origin
activation, candidate relation definition, and isolated reverse generation.

The constraint references `RelationId`; the relation owner continues to define:

- local accepted fields;
- target entity identity;
- target primary-key compatibility;
- cardinality;
- target-existence validation;
- reverse-relation projection; and
- delete blocking.

Introspection reports current `ON DELETE RESTRICT`. No action enum is persisted
because other actions are unsupported. Ordinary key fields remain outside the
relation constraint family.

Relations remain primary-key-only. A relation target is the target entity's
accepted typed primary-key value, never an arbitrary unique field. Alternate
mutable unique keys are not entity identity and do not enter 0.211 relation
binding.

Checking is immediate. `DEFERRABLE`, `INITIALLY DEFERRED`, and equivalent forms
fail explicitly; they are never ignored or simulated. Final delete behavior is
`RESTRICT`.

`CASCADE`, `SET NULL`, and `SET DEFAULT` are not admitted. A future cascade
requires a separately designed bounded-exact closure planner that detects
cycles, caps rows/bytes/instructions/depth, stages every row and derived effect,
and commits the complete closure atomically or changes nothing. It may never
silently cascade a prefix or degrade into resumable partial deletion.

## Introspection

Accepted schema description gains a deterministic constraint collection. SQL
adds:

~~~sql
SHOW CONSTRAINTS FROM entity
SHOW CONSTRAINTS IN entity
~~~

The typed and SQL surfaces return the same payload ordered by `ConstraintId`.
They merge validated constraints and live activation records. Each item
includes:

- constraint ID;
- accepted name;
- kind;
- origin (`generated` or `sql_ddl`);
- validation state (`enforcing_new_writes`, `validating`, or `validated`);
- current bounded validation phase and counters where applicable;
- referenced field, schema index, or relation identity where applicable;
- current canonical field paths;
- target entity for relations;
- derived action (`restrict`) for relations;
- derived semantics version (`check_expr_v1`, `relation_pk_restrict_v1`, or the
  corresponding structural V1 contract), never a free persisted integer;
- current field/index/relation display names;
- canonical rendered check SQL for checks;
- current derived field dependency names for checks; and
- derived referential behavior for relations.

Introspection never exposes Rust callback paths or generated validator messages.
It renders accepted authority only.

Primary-key authority remains structurally separate in the snapshot but appears
as a primary-key constraint beside the other kinds.

`DESCRIBE entity` embeds the same constraint rows. It does not maintain a
second shape or derive pending state from generated models.

There is no relation-strength column. Every promoted relation has the complete
relation guarantee; `validation_state` describes temporary migration progress,
not a permanent weak/strong mode.

## Typed Constraint Diagnostics

Constraint failures and validation findings use one bounded machine-readable
payload:

~~~text
ConstraintDiagnostic {
    constraint_id: ConstraintId,
    constraint_name: String,
    constraint_kind: ConstraintKind,
    entity: EntityIdentity,
    primary_key: Option<PrimaryKeyValue>,
    field_paths: Vec<FieldPath>,
    context: ConstraintDiagnosticContext,
    error_class: ErrorClass,
    diagnostic_code: ConstraintDiagnosticCode,
}

ConstraintDiagnosticContext
  = WriteAdmission
  | MigrationValidation
  | Integrity
~~~

The payload and each nested value have explicit count and byte bounds. Text is
supplementary and callers never parse it for identity or classification.

The same semantic failure is classified by where it is discovered:

- an incoming write that violates an accepted constraint or activation gate is
  `WriteAdmission` / `InvariantViolation`;
- a historical row incompatible with an activating constraint is
  `MigrationValidation` / a dedicated migration-validation error class; and
- an already persisted row violating a `Validated` accepted constraint is
  `Integrity` / `Corruption`.

Revision invalidation, page exhaustion, and stale job identity are progress or
concurrency outcomes, not constraint violations. A pending constraint's
historical violation is never mislabeled as accepted-state corruption.

## Constraint Verification Hooks And 0.212 Boundary

Every accepted constraint kind must have a catalog-native verification path:

- primary key verifies row-key/row-value identity and non-null key components;
- not null scans the accepted field slot for explicit null;
- unique rebuilds or independently checks uniqueness from authoritative rows
  and compares derived index state where appropriate;
- relation verifies every source target exists and verifies reverse-relation
  projection consistency; and
- check evaluates the canonical expression for every authoritative row.

The integrity verifier consumes accepted schema and authoritative stores. It
does not call generated entity validators or sanitizers.

0.211 owns these verification primitives because activation needs them. The
database-wide quick/deep scanner, resumable multi-phase continuation, bounded
finding pages, `CHECK INTEGRITY`, and hard-cut replacement of the existing
aggregate integrity report belong to [0.212 bounded and resumable integrity
check](../0.212-bounded-resumable-integrity-check/0.212-design.md).

`SHOW CONSTRAINTS` reports catalog and activation metadata; it never implies a
fresh live full-table scan.

## Persisted Encoding And Fingerprints

The accepted schema snapshot gains:

- constraint allocator high-water state;
- stable schema-level index IDs;
- stable relation IDs;
- ordered accepted constraints;
- ordered constraint activation records;
- canonical `CheckExprV1` payloads.

Schema migration storage additionally gains bounded validation job records and
staged-generation identity. Job checkpoints and counters are not constraint
semantics and do not alter the accepted constraint meaning fingerprint on every
page, but every job is structurally bound to the current activation fingerprint
and epoch.

The codec must bound:

- total constraints per entity;
- total activations per entity;
- constraint name bytes;
- expression depth;
- expression node count;
- `And`/`Or` child count;
- embedded literal bytes;
- total expression bytes; and
- total constraint catalog bytes.

Validation-job codecs separately bound phase payloads, checkpoint bytes,
participating revisions, counters, one cached receipt, and staged-generation
identity.

Decode validates canonical order, IDs/names unique across constraints and
activations, allocator monotonicity, activation epoch/fingerprint closure,
reference closure, exact kind capabilities, literal admission, validation-job
pairing, staged-owner pairing, and structural constraint pairing before
producing runtime authority.

Accepted cache, admission, commit, marker, journal, describe, and reconciliation
fingerprints include the complete current constraint representation.

Generated check source SQL and generated Rust paths are not fingerprint inputs.
The canonical bound expression is.

## Pre-1.0 Hard Cut

0.211 is a current-form persisted schema cut.

Old snapshots without stable schema index/relation identities, constraint
allocator state, accepted constraints, and the activation representation do not
decode as current 0.211 snapshots. There is no:

- synthesized constraint collection during decode;
- generated-model reconstruction fallback;
- alias from physical index ordinal to `SchemaIndexId`;
- relation-name-to-ID compatibility shim;
- dual fingerprint method;
- legacy validator-path constraint encoding;
- ad hoc decoding of orphan validation jobs or staged generations; or
- anti-resurrection test suite for removed forms.

Development stores using pre-0.211 accepted metadata must be recreated under
the new format unless a separately requested external migration tool is
designed. Runtime does not silently translate them.

Physical row, data-key, index-key, reverse-relation, and journal bytes should
remain unchanged where the new logical identities do not alter their current
owners. Implementation must inventory and report every persisted family that
actually changes.

## Error Model

New paths use typed errors for:

- duplicate, empty, invalid, or overlong constraint names;
- zero, duplicate, retired, reused, or out-of-range constraint IDs;
- invalid constraint allocator state;
- unresolved field, schema-index, or relation identity;
- structural constraint/owner mismatch;
- unsupported expression node, field kind, or operator;
- constraint expression too deep, wide, large, or literal-heavy;
- invalid, mismatched, or noncanonical literal;
- empty, oversized, null-containing, cross-enum, or non-enum membership;
- zero or unsupported `MultipleOf` factor;
- check constraint violation;
- bounded migration validation findings with canonical row identity;
- validation page, finding, row, byte, expression, staging, heap, or instruction
  budget exhaustion;
- stale activation epoch, fingerprint, base schema, revision vector, phase, or
  checkpoint;
- corrupt, missing, duplicate, or orphan validation job;
- missing, divergent, or planner-visible staged generation;
- explicit unique-key or target-delete migration barrier conflict;
- generated-versus-SQL-DDL ownership violation;
- field/index/relation mutation blocked by a dependent constraint;
- stale accepted fingerprint during compiled check execution;
- corrupt persisted constraint catalog or expression; and
- integrity verification failure by constraint ID and name.

Incoming constraint violation is an authored-data invariant failure. Historical
candidate incompatibility is migration validation. A `Validated` constraint
violated by persisted state is integrity/corruption. Corrupt accepted metadata,
impossible compiled-program mismatch, malformed persisted rows, and invalid job
closure are internal/corruption failures. Production code does not match error
strings or panic to classify them.

## Alternatives Considered

### Persist Every Structural Detail Inside AcceptedConstraint

Rejected. Repeating primary-key fields, unique key definitions, relation target
fields, and delete behavior would create a second semantic authority. The
catalog references existing structural owners and validates closure instead.

### Derive All Constraints Only At Introspection Time

Rejected. Derived display alone cannot provide stable constraint IDs, stable
names, origin, DDL ownership, check expressions, or durable schema diffing.

### Persist Only Check Constraints

Rejected. It solves evaluation but leaves primary key, not null, uniqueness,
and relations fragmented across tooling and diagnostics. The unified registry
is part of the feature.

### Persist Generated Validator Paths And Arguments

Rejected. A Rust path is not stable executable database semantics. It cannot be
used safely by SQL DDL, migration validation, independent integrity tools, or a
runtime without the callback.

### Automatically Promote Every Built-In Validator

Rejected. Existing validator declarations lack stable constraint names and may
have application-specific construction or diagnostic behavior. Database checks
must be explicit.

### Reuse The Complete Query Predicate AST

Rejected. It is broader, contains frontend/runtime concerns, and evolves under
different compatibility pressures. Constraint V1 is a smaller persisted AST
that may reuse proven low-level semantics.

### Store Field Dependencies Beside The Expression

Rejected as independent authority. Dependencies are deterministically derived
from bound field IDs. Runtime may cache them and introspection may render them.

### Store Validation State Inside AcceptedConstraint

Rejected. Accepted constraints remain validated structural truth. Temporary
state belongs to a separate accepted-schema activation record whose authority
is limited to migration write gates and validation. Introspection merges the
two without making planners understand partially valid structural owners.

### Require Exact One-Message Validation

Rejected as the only activation path. Entity size is not bounded by one
canister message. Plain SQL `ADD CONSTRAINT CHECK` may use an exact bounded fast
path and change nothing if it cannot prove exhaustion; `NOT VALID`, generated
reconciliation, and structural activation use bounded pages and
stable-revision completion. Storage without either proof capability fails
explicitly rather than overclaiming validation.

### Enforce Unique Writes Through An Incomplete Index

Rejected. Absence from a partially built index does not prove absence from
historical rows. V1 installs an explicit insert and key/predicate-dependency
write barrier until the candidate index is complete and validated.

### Store A Free-Form Semantics Version Per Constraint

Rejected before 1.0. `CheckExprV1` and the snapshot codec own current semantics.
An arbitrary integer would imply multi-version runtime execution without a
defined need.

After 1.0, compatibility may justify one schema-wide accepted
`schema_semantics_edition`, not independently mutable per-constraint integers.
Such an edition would be immutable during ordinary execution, change only
through explicit schema migration, and participate in accepted-schema,
plan/cache, token, activation, and inspection fingerprints. It could retain
older frontend-coercion, comparison, null, collation, numeric-expression, and
migration semantics while a database is deliberately upgraded. New databases
would use the newest edition; an existing database would retain its accepted
edition until an explicit migration succeeds. An edition must never make exact
write admission, declared-relation enforcement, or accepted-schema authority
optional. 0.211 neither persists nor executes multiple editions.

### Use Physical IndexId For Unique Constraints

Rejected. It contains a dense ordinal whose placement can change. 0.211 adds a
separate stable schema-level identity.

### Add ReferentialAction Now

Rejected. Current relations are enforced with restrictive delete behavior.
Persisting unsupported variants would imply semantics that do not exist.

### Add Schema-Native Domains Now

Deferred until accepted checks are implemented and stable. A future domain is
a named accepted scalar contract: one exact base kind/codec/bound plus
deterministic accepted constraints shared by every field that declares the
domain. It is not an alias for a Rust validator path and cannot weaken the
field's exact-kind or absence policy.

Domain identity, canonical encoding, generated-versus-DDL ownership,
constraint dependencies, migration validation, and introspection need their
own design. 0.211 supplies the constraint engine they would reuse but does not
add domain syntax or persisted domain records.

## Focused Validation

### Identity And Catalog Closure

Evidence must cover:

- deterministic initial IDs and names;
- monotonic non-reusing constraint allocation;
- stable constraint IDs through field rename and unrelated mutations;
- stable `SchemaIndexId` through physical ordinal compaction;
- stable `RelationId` through unrelated changes;
- duplicate/zero IDs, duplicate/empty names, bad allocator state, missing
  references, and wrong origins;
- exactly one primary-key constraint;
- exact not-null/nullability pairing;
- exact unique/index pairing;
- exact relation pairing; and
- bounded current-form codec round trips and corrupt decode rejection.

### Activation Identity And Progress

Evidence must cover:

- `Declared` remaining unpublished and non-authoritative;
- atomic publication of activation metadata and its write gate;
- ID/name reservation across activation and promotion;
- abort retiring every reserved identity;
- state transitions only along the documented lifecycle;
- activation epoch/fingerprint and base-schema matching;
- one live job per activation and rejection of orphan/duplicate jobs;
- response loss replaying the same cached bounded finding receipt until its page
  sequence is acknowledged;
- canonical checkpoint advancement only after fully classified rows;
- Forward staging work completing before revision capture;
- Verify remaining read-only over rows and staging;
- row, byte, finding, expression, staged-write, heap, and instruction bounds;
- page fullness distinct from authoritative exhaustion;
- source, target, or staging revision invalidation returning Verify to Forward;
- a clean full sweep plus final unchanged-revision recheck before promotion;
- stale replay and interruption at every job/promotion marker failpoint; and
- storage modes without durable revision authority rejecting unsafe resumable
  completion.

### Check Binding And Canonicalization

Evidence must cover:

- generated and SQL source expressions lowering to identical ASTs;
- name-to-`FieldId` binding;
- exact accepted literal admission;
- canonical `And`/`Or` flattening, ordering, and deduplication;
- range and `BETWEEN` lowering;
- bounded enum `IN` lowering to the same canonical equality disjunction from
  generated and SQL inputs;
- dependency derivation;
- fingerprint changes for every semantic change;
- rejection of parameters, aggregates, subqueries, volatile functions,
  unsupported kinds, nested paths, and relations; and
- depth, node, child, literal, and total-byte boundaries.

### Check Semantics

Evidence must cover:

- `TRUE`, `FALSE`, and `UNKNOWN` truth tables;
- null comparison and `IS NULL`/`IS NOT NULL` behavior;
- exact equality and ordering by supported kind;
- same-kind field-to-field comparisons;
- text Unicode-scalar length, blob octet length, and collection cardinality;
- inclusive range checks;
- signed, unsigned, wide-integer, and decimal multiple-of inside the admitted
  subset;
- zero-factor rejection;
- enum equality and bounded membership within one accepted type;
- empty, null-containing, oversized, cross-enum, and non-enum membership
  rejection;
- no runtime coercion or parsing; and
- typed violations containing stable ID and name.

### Mutation Parity

The same accepted and rejected after-images must converge across:

- typed create/save;
- structural create/save;
- SQL INSERT/UPDATE;
- atomic and non-atomic batches under their documented commit scopes;
- exact bulk update where supported;
- heap and journaled stores;
- defaulted, generated, managed, and explicitly authored values; and
- schema row rewrites/backfills.

Evidence must prove checks and activation gates observe the final
sanitized/materialized value and finish before marker publication.

### SQLite And Independent Oracle Matrix

0.211 extends the provider model and replay infrastructure owned by 0.204; it
does not create a separate constraint differential harness. Focused evidence
must include:

- wrong-kind inserts and updates, plus every deliberately admitted frontend
  conversion and every forbidden or lossy conversion;
- null into required fields and optional relation nulls;
- duplicate unique values;
- rejected relation insertion when the target is absent;
- blocked target deletion while referenced;
- scalar and composite primary-key relation identity;
- `CHECK` success, failure, and `UNKNOWN` behavior;
- a constraint conflict in the final row of an atomic batch;
- marker interruption around a constraint-bearing write;
- exact before-state after every rejected mutation; and
- activation over empty, valid, invalid, and concurrently changing historical
  rows.

Lockfile-pinned SQLite `STRICT` is a reference oracle only for cases whose type,
null, uniqueness, comparison, and check semantics are explicitly mapped as
equivalent. IcyDB typed identity, exact composite kinds, accepted catalogs,
relation/reverse projections, marker recovery, bounded activation, and
revision invalidation use the independent state model, contract assertions,
boundary assertions, and fault injection instead. SQLite agreement may never
satisfy those IcyDB-specific obligations, and incidental SQLite disagreement
may never redefine IcyDB semantics.

### Activation Gates And Existing Rows

Evidence must cover:

- empty and populated entities;
- plain `ADD CONSTRAINT CHECK` proving a complete clean bounded scan before
  atomic validated publication;
- plain `ADD` exceeding a bound with no constraint, activation, job, staging,

- `ADD ... NOT VALID` atomically publishing the write gate and visible
  activation without claiming historical validation;
- all rows valid;
- first, middle, and last-page historical violation;

- revision change after suffix exhaustion forcing restart;

- generated-field references from SQL-DDL-owned checks;
- generated constraint drop rejection;

- `IF EXISTS` only no-oping for absent names;
- abort/drop ownership behavior; and

## Performance And Complexity Budget

### Write Runtime

Entities without accepted checks must not pay per-row expression evaluation.
Structural constraint registry metadata must not add a second uniqueness,
not-null, relation, or primary-key enforcement pass.

Entities with checks should:

- compile once per accepted fingerprint;
- pre-resolve `FieldId` values to slots;
- share decoded slot values across checks;
- short-circuit on the first false constraint;
- avoid generated entity rematerialization for catalog-native evaluation; and
- account for expression node and literal work through bounded budgets.

Benchmark no-check saves, one simple comparison, several independent checks,
wide boolean checks, bounded enum membership, length checks, and near-bound
expressions across typed and structural/SQL paths.

### Activation Validation

Measure rows scanned, row decode work, expression operations, peak temporary
bytes, staged writes, revision restarts, and instruction/host time for empty,
small, and largest admitted validation targets. One page must not retain all
rows or all primary keys in memory.

### Catalog And Persisted Bytes

Report:

- accepted snapshot byte delta from IDs, allocator, and structural constraints;
- per-check expression bytes;
- marker/journal schema payload delta;
- schema describe/introspection response size; and
- whether physical row, index-key, and relation-key bytes changed.

### Wasm Size

Report raw non-gzipped wasm bytes for each implementation slice. Gzip is
secondary context. The dedicated AST/evaluator and SQL DDL surface should reuse
proven scalar helpers where semantics are identical without importing the full
query engine into mutation preflight.

### Source Complexity

Each slice reports files touched, approximate line delta, owner count, and
whether implementation shape became simpler, stayed neutral, or became more
complex.

The target ownership shape is:

~~~text
accepted structural owner <-> accepted constraint identity/reference
accepted CheckExprV1       -> accepted check evaluator
constraint activation      -> write gate + bounded validation job + staging
application validator      -> generated/application visitor only
sanitizer                  -> authored-value transformation only
~~~

There is no callback-backed persisted constraint lane and no hidden partially
accepted lane. Incomplete work is always an explicit activation with narrower
authority than a validated constraint.

## Risks And Mitigations

### Risk: Constraint Catalog Becomes A Second Structural Authority

Mitigation: structural kinds reference stable owner IDs and do not repeat owner
details. One closure validator rejects every mismatch during decode,
reconciliation, mutation, and publication.

### Risk: Stable Identity Is Confused With Physical Placement

Mitigation: add `SchemaIndexId` beside the existing physical ordinal and keep
constraint references on logical identity.

### Risk: Existing Validators Are Assumed To Be Database Constraints

Mitigation: require explicit named check declarations. Keep `validator(...)`
and sanitizers application-only and document the distinction.

### Risk: Query Expression Evolution Changes Constraint Semantics

Mitigation: persist a dedicated bounded `CheckExprV1`; reuse only low-level
helpers whose semantics are explicitly adopted by the check contract.

### Risk: Checks Observe Pre-Sanitized Or Pre-Default Values

Mitigation: evaluate only the final canonical after-image and add parity tests
for sanitizers, defaults, generated values, managed fields, SQL, and batches.

### Risk: Null Semantics Surprise Authors

Mitigation: document that `UNKNOWN` satisfies `CHECK`, render not-null
separately, and test optional fields exhaustively.

### Risk: Validation Never Finds A Quiescent Revision

Mitigation: every page remains bounded and safe, revision changes produce an
explicit restart outcome, and the gate remains active. V1 favors proof over
liveness. Entity-local revisions or mutation-log catch-up require later
evidence and cannot weaken stable-sweep completion.

### Risk: Pending State Is Mistaken For Historical Truth

Mitigation: activations are structurally separate from accepted constraints,
carry explicit states, and are excluded from planner assumptions. Introspection
and diagnostics never render them as `validated`.

### Risk: An Incomplete Unique Or Reverse Index Authorizes A Write

Mitigation: candidate generations remain planner-invisible. Unique inserts,
key/predicate-dependency changes, and target deletes use explicit migration
barriers until atomic promotion makes the corresponding index authoritative.

### Risk: DDL Checks Block Generated Schema Evolution

Mitigation: make dependencies introspectable and fail reconciliation with the
blocking constraint ID/name. The operator drops or changes the DDL-owned check
explicitly before removing its field.

### Risk: Constraint Evaluation Adds Hot-Path Decode Work

Mitigation: cache fingerprint-bound programs, share slot decoding, skip the
owner entirely when no checks exist, and measure each slice.

### Risk: Validation Job Progress Becomes Semantic Authority

Mitigation: checkpoint/counters live in a bounded job record bound to activation
identity. They never make a constraint planner-visible; only stable-sweep
promotion changes validated semantics.

## Final Rule

If a rule must be a database guarantee, it must exist in accepted schema and be
independently executable without application code. Existing structural
constraints keep their proven enforcement owners. `CHECK` uses a bounded
canonical accepted expression. Every accepted constraint is named, stable,
introspectable, and fully validated. Incomplete historical proof is a temporary
accepted-schema activation with a mandatory write gate, bounded resumable
validation, no planner authority, and atomic promotion or abort.
