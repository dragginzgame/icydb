# IcyDB 0.211 State-of-the-Minor Reconstruction Audit

Date: 2026-07-23  
Audit kind: bounded, read-only state reconstruction and triage  
Decision: **Coherent but requires specified corrections**

This document is not a final release certification. It reconstructs what the
0.211 line was meant to contain, what `v0.211.0` released, what the current
local commit adds, and which bounded corrections and evidence remain before a
separate closeout audit is justified.

## 1. Executive Decision

The current 0.211 implementation is substantially one coherent realization of
the original accepted-catalog constraint design. It is not ten hours of
unrelated runtime invention disguised as one minor version.

`v0.211.0` deliberately released only Slice 1: stable logical constraint,
index, and relation identity plus the first accepted-schema format hard cut.
The single local commit after that tag implements the design's remaining
Slices 2–8: the accepted registry, bounded CHECK language, final-after-image
enforcement, bounded activation, not-null/unique/relation staging, SQL
administration and introspection, generated reconciliation, recovery, and
focused performance work.

The implementation has one accepted constraint catalog, one durable
Forward/Verify job model, one main activation runner, shared active/candidate
key projections, and planner-invisible candidate generations. No P0 or P1
corruption, false-promotion, replay, visibility, or lost-live-maintenance defect
was proven in this bounded audit.

Four material closeout issues remain:

1. runtime constraint failures discard the accepted name and most of the
   machine-readable diagnostic contract;
2. the stable generated-proposal drift diagnostic is implemented for CHECK but
   not uniformly for other live generated activation kinds;
3. generated CHECK reconciliation requires exact equality with DDL-extended
   accepted fields/indexes/relations, so a valid generated change can reject
   instead of carrying unrelated DDL-owned accepted state forward;
4. all post-release runtime work is compressed into one 190-file commit, so
   slice provenance, full-suite evidence, and exact performance provenance are
   weaker than the status document implies.

These are P2 closeout issues. The first three fail closed rather than silently
publishing invalid state. They should be corrected in narrow slices before a
final closeout audit. Further feature development should wait until then.

The roughly 23,000 added lines are large but mostly concrete state, codecs,
SQL surfaces, behavior tests, and bounded recovery machinery required by the
original design. The audit did not find a speculative generic migration
framework, broad trait hierarchy, compatibility layer, or second activation
engine. The code is overgrown as a review unit, not proven over-engineered as
an architecture.

## 8. Owner-Readable Current Architecture

### 8.1 Declaration to accepted identity

Generated macros produce proposal metadata, including generated CHECK
declarations. SQL DDL produces catalog-native mutation requests. Neither is
runtime authority.

Startup reconciliation compiles the generated `EntityModel` to a proposal and
passes it with the accepted snapshot to
`derive_generated_accepted_candidate` in
`crates/icydb-core/src/db/schema/mutation/generated_candidate.rs:48`.
The accepted snapshot's `AcceptedConstraintCatalog` in
`crates/icydb-core/src/db/schema/constraint.rs:367` allocates and retains
accepted IDs, names, origins, kinds, and activation records.

Primary-key, field, index, and relation records retain physical and structural
facts. Their stable IDs link them to the registry. This split is intentional:
the registry owns accepted constraint identity and lifecycle; the structural
record owns the data required to enforce or materialize that structure.

### 8.2 Accepted constraint to live write

Accepted schema selection compiles row-local CHECK and not-null contracts into
`CompiledAcceptedRowConstraints`
(`schema/check/compile.rs:142`) bound to the accepted fingerprint.

`SaveExecutor::preflight_resolved_entity_with_provenance`
(`executor/mutation/save_validation.rs:107`) owns the ordered mutation
preflight:

1. resolve authored/default/generated values;
2. run application sanitization and validation where the ingress has generated
   application behavior;
3. construct the canonical final after-image;
4. enforce provenance and activation barriers;
5. evaluate accepted row-local constraints;
6. validate accepted relation semantics;
7. prepare physical commit effects.

Typed, structural/fluent, SQL, batch, prepared, and replayed logical writes
reach this save/commit authority rather than each owning an accepted CHECK
evaluator.

Unique and relation constraints consume the same final canonical row but use
their specialized key/edge projection and physical stores. They are not
reimplemented in the row-expression evaluator.

### 8.3 Pending activation

An accepted `ConstraintActivationSnapshot` owns identity, origin, kind,
base-schema fingerprint, activation fingerprint, and epoch. A durable
`ConstraintValidationJob`
(`schema/constraint_validation.rs:157`) owns phase, checkpoint, participating
revisions, staged generation, counters, receipt sequence, and last receipt.

`constraint_activation_runner.rs` is the one main activation controller.
Forward pages build evidence or staged state. Verify pages prove a stable
revision vector. Findings are emitted in bounded receipts that must be
acknowledged exactly before progress advances. Revision drift restarts proof
rather than promoting stale evidence.

Promotion is a schema mutation under the existing marker publication boundary.
Abort removes only the pending activation/job/candidate state owned by that
activation.

### 8.4 Candidate unique and relation state

`PersistedSchemaSnapshot` separately holds accepted indexes/relations and
candidate indexes/relations. Accessors at
`schema/snapshot.rs:807` and `schema/snapshot.rs:813` are schema-internal.
Query/access planning consumes accepted indexes only.

`UniqueConstraintProjection`
(`schema/mutation/user_index_domain.rs:576`) is reused by activation staging,
live commit preparation, and recovery rebuild. `RelationConstraintProjection`
(`relation/reverse_index.rs:246`) serves the same shared role for candidate and
accepted relation edges.

Candidate unique writes use an explicit barrier in
`commit/prepare.rs:714-749`: source deletion is representable, while an update
that changes candidate membership/key rejects. Candidate relation writes
dual-maintain the candidate generation and target deletion checks accepted
pending relation authority.

### 8.5 SQL administration and display

SQL parser/lowering identifies administrative constraint operations, but
schema mutation and the activation runner own their semantics. SQL execution
returns typed `SqlConstraintValidationOutput` and
`SqlConstraintValidationFindingOutput`. `DESCRIBE` and
`SHOW CONSTRAINTS FROM/IN` use the same `EntityConstraintDescription`
projection (`schema/describe.rs:197`).

SQL has no separate accepted registry, scanner, candidate generation, or
promotion state machine.

### 8.6 Recovery

Accepted schema and validation job closure is decoded and validated by
`SchemaStore`. Marker recovery uses accepted activation identity, durable job
state, accepted rows, and the shared unique/relation projections to reconstruct
candidate physical state. Candidate keys/edges are derived state, not a second
persisted semantic authority.

Recovery does not consult generated models to reconstruct accepted meaning and
does not translate obsolete schema formats.

## 10. Activation State-Machine Comparison

| Kind | Immediate live gate | Forward work | Verify proof | Durable candidate | Revisions | Promotion | Abort |
| --- | --- | --- | --- | --- | --- | --- | --- |
| CHECK | Final after-image must satisfy pending expression | Scan rows and emit bounded failures | Stable entity revision and clean full scan | None | Source entity | Move activation to validated constraint | Remove pending activation/job |
| Not-null | New final after-image cannot be null | Scan historical rows for null | Stable entity revision and clean full scan | None | Source entity | Change field nullability and accept structural constraint | Remove pending tightening/job |
| Unique | Key-changing/member-adding live writes block; source deletion maintained | Build isolated candidate keys and find duplicates | Stable source revision and duplicate-free candidate | Candidate index generation | Source entity | Move candidate to accepted indexes and validate registry entry | Delete candidate generation and pending state |
| Relation | Source writes dual-maintain candidate edges; target deletion blocks while proof depends on it | Build candidate edges and report missing targets | Stable source and target revisions | Candidate relation generation | Source and target | Move candidate to accepted relation and validate registry entry | Delete candidate generation and pending state |

All four use the same activation identity, job, Forward/Verify vocabulary,
receipt acknowledgement, restart policy, and marker-owned schema transition.
The differences are necessary evidence and physical staging mechanics, not
parallel lifecycle engines.

The job currently represents phase-specific fields as one phase plus several
validated `Option` fields (`constraint_validation.rs:164-173`). Decode and
transition validation reject impossible combinations. A phase-payload enum
could encode those combinations more tightly, but the present representation
is not a proven closeout defect and replacing it now would add risk without
changing authority.

## 12. Persistence and Hard-Cut Analysis

`crates/icydb-core/src/db/schema/codec.rs:48-49` defines exactly:

- schema snapshot codec version `1`;
- contract profile `ICYZ`.

`PersistedSchemaSnapshotWire::into_snapshot`
(`codec.rs:532-537`) rejects any different version or profile with a typed
incompatible-persisted-format error. There is no previous-profile branch,
translation, dual decoder, inferred registry, or generated-model fallback.

The pre-release `ICYU` profile and intermediate post-release profiles survive
only in historical status/changelog prose. They do not survive as production
constants. The unrelated literal `ICYU` in resumable SQL update continuation
encoding names a different current protocol and is not accepted-schema
compatibility.

`schema/transition/compatibility.rs` is unfortunately easy to misread by name,
but it classifies current generated-proposal versus accepted-schema transitions.
It is not a persisted compatibility decoder.

`RawSchemaSnapshot::from_bytes` recognizes current header-bearing entity
snapshots and current schema-store control records because they share a stable
namespace. Entity admission still requires the exact current accepted schema
profile and fingerprint. This is current record dispatch, not an old raw schema
fallback.

The hard cut is explicit in root `CHANGELOG.md:52-54`: stores created by
`0.211.0` must be recreated. The current status also records `ICYZ` as the sole
accepted entity-snapshot profile. This is an intentional pre-1.0 hard cut, not
an accidental inability to upgrade.

Malformed codec tests cover future/wrong versions and incompatible profiles.
No compatibility decoder or anti-resurrection compatibility implementation
was found.

  structures until promotion.

Focused interruption tests for unique and relation activation passed. A focused

## 14. Rust Complexity and Over-Engineering Assessment

### What the added machinery guarantees

The largest additions are concrete:

- a bounded CHECK AST, canonicalizer, binder, compiler, and evaluator;
- accepted identity/catalog codec and closure checks;
- one durable two-pass validation job with receipt replay;
- one activation runner with four kind-specific proof/staging branches;
- candidate key/edge projection shared by runtime and recovery;
- SQL grammar, lowering, execution, display, and typed result DTOs;
- behavior and malformed-state tests.

Those guarantees—boundedness, accepted authority, revision proof, atomic
promotion, candidate invisibility, and crash recovery—are explicitly required
by 0.211. Removing them to reduce line count would weaken the design.

### Suspected over-engineering that was not found

The post-release range does not introduce:

- a general-purpose migration plugin framework;
- broad substitution traits with one implementation;
- deeply nested generic state machines;
- public constructors for raw candidate/persisted state;
- multiple activation phase enums;
- a second marker or recovery protocol;
- a SQL-owned constraint scanner;
- compatibility translation layers;
- an unused framework for future constraint families.

### Complexity worth revisiting later

`ConstraintValidationJob` uses a phase plus related optional fields, and its
validation method enforces their legal combinations. A phase-specific payload
enum would make more impossible states unrepresentable. The existing decoder
already fails closed, tests cover malformed closure, and a representation
rewrite would touch persistence and recovery. This is a P3 design observation,
not mandatory 0.211 remediation.

`AcceptedSaveContract` and several commit contracts use tuples or concrete
transport structs. Most predate 0.211 or acquired one additional accepted
constraint component. No field-for-field DTO ladder was proven to own duplicate
semantics.

The one 1,500-line activation runner is large, but splitting it without moving
authority would improve navigation rather than architecture. Its kind-specific
branches share one job and transition owner. This is safely deferrable unless
future changes demonstrate divergent policy.

### Complexity verdict

The Rust code went materially further in size, but not beyond the guarantees
the original design demanded. The main process failure was allowing seven
slices plus cleanup to accumulate into one local commit. The architecture
itself is not proven to be an unjustified generic system.

## 18. Ranked Findings

No P0 or P1 finding was proven.

### F-01 — P2 — Accepted constraint diagnostics lose required identity and context

**Category:** implementation defect; missing propagation; public diagnostic
contract.

**Violated contract:** The design requires one bounded machine-readable
`ConstraintDiagnostic` for write admission, migration validation, and
integrity. It includes entity, accepted ID and name, kind, primary key where
available, field paths, context, error class, and diagnostic code
(`0.211-design.md:1041-1045`, `1322-1356`). The root changelog says accepted ID
and name are retained at the runtime boundary (`CHANGELOG.md:16-18`).

**Production evidence:**

- `InternalError::mutation_constraint_violation` accepts
  `_constraint_name` and discards it
  (`crates/icydb-core/src/error/mod.rs:421-430`);
- `mutation_constraint_activation_write_blocked` likewise discards
  `_constraint_name` (`error/mod.rs:446-455`);
- `ExecutorErrorDetail::ConstraintViolation` and
  `ConstraintActivationWriteBlocked` retain only `constraint_id`
  (`error/mod.rs:1562-1572`);
- diagnostic projection reduces those variants to generic runtime-boundary
  codes (`error/mod.rs:1899-1934`);
- bounded validation findings retain row key, field IDs, and a numeric finding
  code, but not the designed accepted name/kind/entity/context/class payload.

**Test/document evidence:** Mutation tests at
`executor/tests/mutation_save.rs:3835-3905` and SQL tests at
`session/tests/sql_write.rs:231` assert only the broad detail variant/ID.
They cannot detect loss of name or context. The status claims stable typed
diagnostics are complete (`0.211-status.md:37`).

**Reachable scenario:** A SQL or typed write violates a named accepted CHECK,
or an operator pages a failed activation. The database knows the accepted name,
kind, entity, and context at evaluation time, but the caller receives only a
numeric ID plus a generic code.

**Consequence:** No corruption or false acceptance occurs, but operators and
programmatic callers cannot reliably identify or classify the accepted
constraint using the promised stable payload. Context is reduced prematurely
to prose or absent data.

**Why tests missed it:** They match the broad enum variant and ID rather than
the designed public/runtime diagnostic structure.

**Smallest hard-cut correction:** Define one bounded typed diagnostic payload
at the core error boundary and carry it directly through write failures,
activation findings, facade SQL/Candid output, and CLI projection. Remove the
underscore/discarded name parameters; do not add a parallel compatibility
variant.

**Required closing evidence:** Real typed and SQL write violations plus a
migration finding must assert exact ID, name, kind, entity, context, class, and
stable diagnostic code through the public/facade boundary. Corrupt compiled
program coverage must remain a distinct typed corruption case.

**Confidence:** High.

### F-02 — P2 — Stable proposal-drift rejection is CHECK-only

**Category:** incomplete hard cut; missing propagation; documentation
overclaim.

**Violated contract:** Generated CHECK, relation, and not-null additions use the
same activation lifecycle, and any generated proposal change during a live
activation must reject as a stable stale migration
(`0.211-design.md:1187-1197`). The root changelog generalizes this to live
generated activation (`CHANGELOG.md:45-48`).

**Production evidence:**

- `GeneratedAcceptedCandidateError::StaleConstraintActivation` exists at
  `schema/mutation/generated_candidate.rs:30-39`;
- CHECK derivation explicitly compares the live activation with the current
  proposal and returns that error
  (`generated_candidate.rs:595-625`);
- an existing relation candidate mismatch returns `Ok(None)` instead
  (`generated_candidate.rs:280-294`);
- an existing unique candidate mismatch returns `Ok(None)`
  (`generated_candidate.rs:445-459`);
- not-null derivation returns `Ok(None)` for a missing/changed tightening
  (`generated_candidate.rs:511-568`).

The later transition remains fail closed, but the dedicated stable conflict
classification is lost outside CHECK.

**Test/document evidence:** The only exact drift test is
`schema/reconcile/tests.rs:753-815`, which removes a generated CHECK and asserts
`GeneratedConstraintActivationStale`. There are no corresponding changed or
removed proposal tests for generated not-null, unique, and relation
activations. The status accurately describes only CHECK at
`0.211-status.md:467-473`, while the design and root changelog are broader.

**Reachable scenario:** Startup has a live generated relation, unique, or
not-null activation. The generated declaration is removed or semantically
changed before validation completes. Reconciliation rejects, but through a
generic transition mismatch rather than the promised stale-activation
diagnostic.

**Consequence:** Accepted/job/candidate state remains safe, so this is not a
false-promotion defect. Operators cannot distinguish stale live migration from
an unrelated schema incompatibility, and behavior differs by constraint kind.

**Why tests missed it:** Only CHECK proposal drift is exercised.

**Smallest hard-cut correction:** Centralize one live-generated-activation
proposal-match check before per-kind candidate derivation. Exact matching should
resume; any changed or missing generated declaration should return the single
stable stale-activation error. Do not add per-kind aliases.

**Required closing evidence:** For not-null, unique, and relation activation,
test removed and changed proposals, exact proposal resume, unchanged accepted
activation/job/candidate bytes after rejection, and the exact stable conflict
code through startup reconciliation.

**Confidence:** High.

## 19. Challenged but Upheld

### Structural details outside the registry

The accepted catalog does not copy every not-null/index/relation implementation
detail. Stable IDs link registry identity to field/index/relation records that
own necessary physical facts. This is one registry plus structural owners, not
several identity registries synchronized by convention.

### Row-local gate versus unique/relation enforcement

CHECK and pending/accepted not-null share one compiled row-local evaluator.
Unique and relation enforcement uses shared final after-images but specialized
key/edge projections and stores. Forcing all four into one evaluator would
erase real physical and equality-domain differences.

### Plain atomic ADD CHECK

A bounded plain `ADD CHECK` performs exact proof and publication in one
operation. It is not a second unbounded activation scanner. Work exceeding the
bound rejects or must use `NOT VALID` and the canonical runner.

### Candidate unique write behavior

The unique candidate path does not naïvely dual-maintain every live change.
It permits safe source deletion, blocks key/membership-changing updates, and
relies on bounded proof. This conservative barrier preserves proof validity and
is deliberate, not missing maintenance.

### Relation revision vector

Relation proof captures source and target revisions while row-local/unique proof
uses the source entity revision. These are different proof domains, not
competing global revision authorities.

### `schema/transition/compatibility.rs`

Despite its historical-looking name, it classifies current generated proposal
transitions. It does not decode or preserve obsolete formats.

### Current raw schema dispatch

Shared schema-store record dispatch accepts current entity and control record
envelopes, then exact entity decoding applies `ICYZ`. It is not a legacy raw
fallback.

### Integrity work

0.211 supplies accepted-native compilation and projection hooks and deliberately
does not add another scanner. The actual resumable integrity sweep belongs to
0.212. This boundary matches the original design.

### Activation job option fields

The representation permits construction of invalid combinations internally,
but constructor/decode/transition validation rejects them. A phase-payload enum
would be cleaner, yet no reachable fail-open behavior was demonstrated.
