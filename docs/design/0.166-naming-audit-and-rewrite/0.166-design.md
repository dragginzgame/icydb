# IcyDB 0.166 — Naming Audit And Rewrite

## Status

Planned.

## Summary

0.166 is a complete naming audit and rewrite pass for live IcyDB code and
active documentation.

The pass standardizes architectural role names, helper verbs, and stale
historical vocabulary without changing behavior. It uses
`docs/architecture/NAMING.md` as the naming policy and checks every proposed
rename against idiomatic database and query-engine vocabulary before it lands.

This is not a style polish pass. It is a codebase-wide architecture readability
pass whose output should make ownership boundaries, planner/executor handoffs,
and public database concepts easier to read.

## Thesis

Names in IcyDB are part of the architecture.

They tell readers whether a value is:

- persisted schema/catalog authority
- query intent
- a planner-selected payload
- an admission proof
- a route decision
- a frozen fact snapshot
- a local traversal context
- an execution-ready payload
- an identity/cache-key surface
- runtime behavior semantics

Recent structural cleanup improved ownership, but some names still preserve
older layering. 0.166 closes that gap by aligning live names with current
architecture.

## Non-Negotiables

- No behavior changes.
- No compatibility shims for old internal names before 1.0.
- No public rename unless the new name removes real ambiguity or likely misuse.
- No replacement of standard database vocabulary with local architecture jargon.
- No broad test-only renames unless they support a production concept rename.
- No archive-doc rewrite for historical prose; archive hits are classified.

## Database Idiom Alignment

0.166 must stay idiomatic for database and query-engine readers.

The rewrite should preserve conventional terms when they are the right domain
word:

- `Catalog`, `Schema`, `Relation`, `Field`, `Index`, `Key`, `Constraint`
- `Predicate`, `Projection`, `Aggregate`, `Grouping`, `Ordering`, `Limit`,
  `Offset`
- `LogicalPlan`, `AccessPath`, `ExecutionPlan`, `Executor`
- `Cursor`, `Continuation`, `Page`, `Window`
- `Transaction`, `Commit`, `Snapshot`, `Replay`, `Recovery`

IcyDB role families are added only where they clarify ownership or phase:

- `*Contract` for admission/proof surfaces, not persisted SQL constraints
- `*Decision` for selected route or mode plus reason
- `*Facts` for reusable derived snapshots
- `*Context` for owner-local short-lived traversal inputs
- `*Identity` for equivalence, hashing, cache, and dedup authority
- `*Semantics` only for behavior that changes results

If a proposed rename makes the code less recognizable to someone familiar with
database internals, reject it unless the local architecture benefit is stronger
and documented.

## Scope

### In Scope

- all live Rust crates
- active architecture, contract, governance, and design docs
- active changelog entries that describe the current slice
- public facade names, but only under a higher rename bar
- module/file names that own renamed concepts
- test names that lock renamed production concepts

### Archive Handling

Archived design docs and old changelog entries are scanned but not rewritten by
default. Remaining hits are classified as historical vocabulary unless they
point to active instructions or copied policy that should move forward.

### Out Of Scope

- semantic or behavioral changes
- release/version number edits
- unrelated module restructuring
- low-value leaf helper churn
- renaming conventional database terms into generic local role names

## Method

0.166 proceeds by family, not by random symbol.

For each family:

1. Inventory live names and old stems with `rg`.
2. Classify each candidate by database concept and IcyDB role family.
3. Record a role proof before renaming:
   - current name
   - owning module
   - fields or return payload
   - main consumers
   - chosen name family
   - rejected alternatives
   - public-surface impact
4. Rename the whole concept through types, modules, helpers, imports, tests,
   comments, examples, and active docs.
5. Scan old vocabulary and classify remaining hits.
6. Run focused checks for the touched family.

## Audit Families

### 1. Naming Policy

Bring the durable policy into `docs/architecture/NAMING.md` and keep
`docs/architecture/TERMINOLOGY.md` focused on domain vocabulary.

This family is complete when the architecture docs clearly separate:

- database/domain terms
- IcyDB role families
- helper verb meanings
- public rename discipline

### 2. Route And Access Selection

Audit names around route facts, capabilities, selected modes, reasons, and
execution admission.

Primary questions:

- Is this a route decision or an admission contract?
- Is this a conventional access-path concept?
- Are boolean capability bundles really facts, capabilities, or decision
  outputs?

Likely candidates to prove before renaming:

- `LoadOrderRouteContract`
- `LoadOrderRouteDecision`
- `RouteCapabilities`
- `LoadRouteCapabilityFacts`
- `GroupedExecutionModeProjection`

### 3. Covering Reads

Audit covering projection, covering read, existing-row, and index-component
names.

Primary questions:

- Is the value a selected covering read plan?
- Is it a reusable fact snapshot?
- Is it a local derivation context?
- Does `Projection` remain the right database term?

Likely candidates to prove before renaming:

- `CoveringProjectionContext`
- `CoveringReadPlan`
- `CoveringReadExecutionPlan`
- `CoveringProjectionFieldSourcePolicy`
- covering helper verbs such as `prepare_*` versus `derive_*`

### 4. Prepared Execution

Audit names around prepared plans, shared monomorphic shells, cached residents,
and runtime handoff bundles.

Primary questions:

- Is this a plan, core payload, shared shell, resident cache, or runtime bundle?
- Does `Shared*` mean generic-free shell, cloned resident state, or shared
  ownership?
- Is `Core` still meaningful or just historical layering?

Likely candidates to prove before renaming:

- `PreparedExecutionPlanCoreShared`
- `PreparedExecutionPlanCore`
- `SharedPreparedExecutionPlan`
- `PreparedScalarRuntimeParts`
- `PreparedGroupedRuntimeParts`
- `PreparedAccessPlanParts`
- `SharedPreparedProjectionRuntimeParts`

### 5. Continuation, Cursor, Page, And Window

Audit pagination and resumable execution names.

Primary questions:

- Is the concept a cursor token, continuation contract, page window, or access
  window?
- Does `Plan` imply too much execution selection?
- Are grouped and scalar names parallel where they represent the same role?

Likely candidates to prove before renaming:

- `PlannedContinuationContract`
- `ScalarAccessWindowPlan`
- `GroupedContinuationWindow`
- `GroupedWindowProjection`
- `PlannedCursor`
- `GroupedPlannedCursor`

### 6. SQL Lowering And Query Shape

Audit lowered SQL payloads, expression analysis, aggregate descriptors, and
prepared SQL shapes.

Primary questions:

- Is this lowered query intent, a structural shape, a selected plan, or an
  execution descriptor?
- Does `Shape` mean a compact structural family, or is it hiding a phase
  payload?
- Does `Descriptor` describe observability/rendering, or should it be a plan or
  kind?

Likely candidates to prove before renaming:

- `LoweredSelectShape`
- `LoweredBaseQueryShape`
- `LoweredExprAnalysis`
- `LoweredSqlAggregateShape`
- `PreparedSqlScalarAggregateDescriptorShape`

### 7. Identity Versus Semantics

Audit every live surface where `semantic`, `semantics`, `identity`,
`fingerprint`, `hash`, `cache`, `dedup`, or `canonical` appear near each other.

Primary rule:

- `Identity` means equivalence, hashing, cache-key, or dedup authority.
- `Semantics` means behavior that changes runtime or SQL-visible results.

Likely zones:

- query-plan identity
- prepared-query cache keys
- aggregate planning and lowering
- projection deduplication
- tests for equality, hashing, or reuse

### 8. Helper Verbs

Audit `classify_*`, `analyze_*`, `derive_*`, `resolve_*`, `prepare_*`, and
`canonicalize_*` across live code.

Primary rules:

- `classify_*` returns compact categories.
- `analyze_*` returns richer descriptive results.
- `derive_*` computes new facts or outputs.
- `resolve_*` maps to an already-existing symbol, slot, path, or component.
- `prepare_*` freezes reusable state for a later phase.
- `canonicalize_*` normalizes to an equivalent stable form.

The goal is not to rename every helper. The goal is to remove misleading verbs
on shared architectural surfaces.

### 9. Public Surface Audit

Audit public names last and under a higher bar.

Public names should remain stable unless the current name is materially wrong,
misleading, or inconsistent with conventional database vocabulary.

Names to audit carefully:

- `QueryResponse`
- `ProjectionResponse`
- `MutationResult`
- `MutationMode`
- `SqlProjectionRows`
- `SqlQueryRowsOutput`
- `SqlGroupedRowsOutput`
- `SqlQueryResult`
- `QueryExecutionAttribution`
- `SqlQueryExecutionAttribution`
- `ExplainExecutionDescriptor`
- `ExplainPropertyMap`

Candidate public renames are not accepted by default. Each one needs a public
impact note and a reason the new name is better for database users, not only
for internal consistency.

### 10. Residual Sweep

After family rewrites, run a whole-repo scan for stale vocabulary in live code
and active docs.

Minimum scans:

```bash
rg -n "OldName|old_name|old-name|old concept phrase" crates docs CHANGELOG.md
rg -n "Plan|Contract|Decision|Facts|Context|Shape|Identity|Semantics|Analysis" crates docs
rg -n "classify_|analyze_|derive_|resolve_|prepare_|canonicalize_" crates
```

Remaining hits must be classified as:

- intentional current vocabulary
- conventional database vocabulary
- archive/history
- generated artifact text
- stale drift to fix

## Completion Criteria

0.166 is complete when:

- `docs/architecture/NAMING.md` is the active naming authority
- live code has been audited across every family above
- accepted renames are applied through modules, imports, helpers, tests, and
  active docs
- public-surface names have explicit keep/rename decisions
- helper verbs are no longer misleading on shared architecture surfaces
- `Identity` and `Semantics` are separated according to role
- conventional database terms are preserved where they are the better name
- old vocabulary scans have no unclassified live hits
- focused checks pass for all touched families

## Expected Payoff

The pass should leave the codebase easier to review:

- database concepts stay recognizable
- IcyDB-specific phase artifacts advertise their ownership role
- route, covering, prepared execution, continuation, lowering, and public
  response names stop mixing historical near-synonyms
- future feature work has one naming policy to extend

The result should feel less clever, not more: standard database terms where
they fit, precise IcyDB role names where the architecture needs them.
