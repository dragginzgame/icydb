# 0.62 Status Tracker

Current state: in progress after `0.62.4`; architecture phase is complete and
the current work is explicit-decoder organization plus measured tightening.

## Scope

This status tracks the `0.62` persisted-row structuralization line and the
follow-up wasm-surface tightening that remained after the runtime serde graph
was removed.

Design reference:

- `docs/design/0.62-structural-persisted-row-decode/0.62-design.md`

---

## Landed Through `0.62.4`

- [x] Persisted rows use one slot-based structural format only; legacy row
  versions were hard-cut.
- [x] `PersistedRow` derive/codegen is the canonical typed boundary for row
  write/materialization.
- [x] Commit/index planning no longer decodes `E` from row bytes.
- [x] Index key extraction, predicate evaluation, projection execution, and
  relation validation are structural/slot-driven in runtime paths.
- [x] Scalar leaf codec v1 is in place for the scalar field families that were
  moved off CBOR.
- [x] Scalar-only predicate compilation exists and dispatches once at the
  execution boundary instead of branching per node.
- [x] Scalar-only projection compilation exists for structural SQL projection
  execution.
- [x] Reverse-index and strong-relation runtime paths decode storage keys
  directly instead of round-tripping through `Value`.
- [x] Commit control envelopes and commit-marker envelopes are binary.
- [x] Commit-marker payload decode is binary and structural.
- [x] Runtime `serde_cbor::from_slice` is gone from persistence/runtime decode
  paths.
- [x] `structural_field` is no longer a monolith; it is split into directory
  modules with explicit ownership boundaries.
- [x] Projection materialization/eval is no longer concentrated in two large
  files; structural SQL row loops and compiled scalar projection eval now have
  their own owner modules.

## Current Architecture Boundary

Persistence:

- row container: slot-based structural format
- commit frame: binary
- commit payload: binary

Execution:

- hot scalar paths: `SlotReader -> scalar/native execution`
- generic fallback: explicit and isolated

Decode:

- scalar leaves: direct decode
- complex leaves: explicit structural decode in
  `crates/icydb-core/src/db/data/structural_field/`

Typed boundaries:

- entity reconstruction
- response/materialization boundaries that truly need typed values

## Current `structural_field` Module Layout

- `crates/icydb-core/src/db/data/structural_field/mod.rs`
  - public boundary and top-level by-kind dispatch
- `crates/icydb-core/src/db/data/structural_field/cbor.rs`
  - low-level bounded CBOR parsing and raw-slice walkers
- `crates/icydb-core/src/db/data/structural_field/scalar.rs`
  - scalar primitive fast path
- `crates/icydb-core/src/db/data/structural_field/kind.rs`
  - composite `ByKind` list/map/enum decode
- `crates/icydb-core/src/db/data/structural_field/leaf.rs`
  - typed wrapper and fixed-shape leaf payloads
- `crates/icydb-core/src/db/data/structural_field/storage_key.rs`
  - direct `StorageKey` and strong-relation decode
- `crates/icydb-core/src/db/data/structural_field/value_storage.rs`
  - externally tagged `Value` payload decode and shallow untyped fallback

This split is now the intended baseline. Follow-up work should preserve these
ownership lines unless there is a strong reason to change them.

## Current Projection Module Layout

- `crates/icydb-core/src/db/executor/projection/materialize/mod.rs`
  - shared materialization helpers, validation, grouped projection output, and
    typed test-facing row projection
- `crates/icydb-core/src/db/executor/projection/materialize/structural.rs`
  - structural SQL projection row loop and scalar-vs-generic structural
    dispatch
- `crates/icydb-core/src/db/executor/projection/eval/mod.rs`
  - generic slot-reader and grouped expression evaluation plus shared error
    taxonomy
- `crates/icydb-core/src/db/executor/projection/eval/scalar.rs`
  - compiled scalar projection expressions and scalar slot-reader execution

This split is also now part of the intended baseline. New work should tighten
these modules in place instead of letting `materialize/mod.rs` or
`eval/mod.rs` grow back into mixed-owner sinks.

## Recent Kept Wins

- Shared CBOR array/map walkers replaced repeated container loops in the
  structural-field decoder.
- Tagged variant parsing was unified for schema enums and `Value` payloads.
- `ValueEnum` payload decode stopped building temporary owned strings before
  constructing `ValueEnum`.
- Untyped enum/shallow fallback dispatch was flattened by branching on CBOR
  major type directly.
- Projection/commit/unique row-key validation now shares one slot-owned
  storage-key validator instead of duplicating `Value -> StorageKey` round
  trips.
- `structural_field` was split into `cbor` / `scalar` / `kind` / `leaf` /
  `storage_key` / `value_storage`, with sibling modules importing from the
  owning boundary instead of reaching through a monolith.
- projection materialization was split into a `materialize/` directory module,
  with the structural SQL row loop moved into `materialize/structural.rs`
- compiled scalar projection evaluation was split into `eval/scalar.rs`, so
  `eval/mod.rs` now owns only the generic/grouped lane plus the shared error
  taxonomy

Measured local wins that were kept:

- `decode_value_enum_payload_bytes`: `4207 -> 3677`
- `decode_untyped_enum_payload_bytes`: `1017 -> 630`
- `decode_untyped_shallow_bytes`: `658 -> 304`
- `project_data_rows_from_projection_structural`: `7018 -> 6684`

## Reverted / Non-Paying Directions

- Broad map-helper unification in `structural_field.rs` regressed retained wasm
  and was reverted.
- Direct-loop rewrites for `Value::Map` payload decode regressed retained wasm
  and were reverted.
- Small decode-surface cleanups that did not delete a real retained path were
  not kept.

Rule kept from these attempts:

- accept or reject by measured retained/runtime wasm movement, not by
  architectural neatness alone

Refinement after the `0.62.4` organization pass:

- do not re-enter symbol-golf mode if it cuts across the new module ownership
  boundaries
- prefer coherent local ownership first, then measure within that cleaned
  structure

   - keep reducing projection-side retained surface without reintroducing
     generic `Value` work on scalar-only rows

   - tighten scalar projection evaluation only inside the cleaned scalar lane
5. `crates/icydb-core/src/db/executor/projection/materialize/mod.rs`
   - keep the root as shared validation/materialization boundary, not another
     implementation sink

Do not restart another representation migration unless measurement proves the
current explicit decoder surface has plateaued.
