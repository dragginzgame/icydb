# IcyDB 0.62 Design

Status: **Proposed**
Theme: **Canonical Structural Persisted-Row Decode (Serde Graph Collapse)**

---

# 0. Executive Summary

`0.62` represents a shift from **control-flow demonomorphization** (0.58–0.61) to **data-shape normalization**.

Previous work successfully removed large monomorphic execution paths. Remaining wasm size is now dominated by:

* `serde_cbor` deserialization graphs
* per-entity `Deserialize` instantiations

The most promising remaining non-Candid structural lever is:

> **Decode persisted rows into a single canonical structural representation, and eliminate entity-typed persisted-row deserialization from non-boundary runtime paths.**

---

# 1. Problem Statement

## 1.1 Current State

Persisted rows are decoded via:

* `deserialize_row::<E>`
* `try_decode::<E>`

Each entity `E` (including nested sub-records) introduces a distinct `Deserialize` graph.

Result:

* Large `serde_cbor` footprint
* Code duplication proportional to number of entity shapes

## 1.2 Observed Limits (0.61)

* Structuralizing execution paths → **high payoff**
* Structuralizing tight compute (forward index, predicates) → **regression**
* Removing decode call sites → **no effect unless type diversity reduced**

Conclusion:

> **The remaining cost is driven by type graph diversity, not call-site count.**

---

# 2. Design Goal

## Primary Goal

> **Collapse all persisted-row deserialization onto a single structural representation.**

## Secondary Goals

* Preserve full typed entity hierarchy (including sub-records)
* Avoid runtime abstraction regressions
* Minimize trait surface expansion
* Maintain current validation and error semantics

---

# 3. Core Design

## 3.1 Canonical Structural Decode

Introduce a single decode path:

```
RawRow → StructuralRow
```

Where:

```
StructuralRow = slot-indexed structural row view
```

### Properties

* No `<E>`
* No per-entity specialization
* Stable decode graph
* Lazy slot access
* `Value` materialization only when a caller actually needs it

---

## 3.2 Typed Reconstruction (Boundary Only)

Typed entities are reconstructed only when required:

```
StructuralRow → E
```

Used only at:

* API boundaries
* response construction
* external interfaces

Not used in:

* commit preparation
* relation validation
* index planning

---

## 3.3 Elimination of Typed Decode in Runtime

Remove non-boundary runtime usages of:

* `deserialize_row::<E>`
* `try_decode::<E>`

Except:

* explicit typed boundary reconstruction

## 3.4 Decode Ownership Rule

At any given call site:

* either structural decode or typed decode may occur
* never both in the same execution path

Implications:

* no fallback paths that reintroduce `try_decode::<E>` after structural decode
* no mixed structural and typed interpretation of the same persisted row
* structural decode is authoritative for any path that adopts it

Violating this rule:

* reintroduces `Deserialize` graphs
* weakens or eliminates the expected wasm reduction

---

# 4. Sub-Records and Type Hierarchy

## 4.1 Preservation Guarantee

All existing entity types remain unchanged.

Example:

```rust
struct User {
    id: u64,
    profile: Profile,
}

struct Profile {
    name: String,
    age: u32,
}
```

## 4.2 Structural Representation

Nested structures are represented as:

```
Value::Record / Value::Map / nested Value
```

## 4.3 Reconstruction

Typed reconstruction builds sub-records explicitly from structural values.

---

# 5. System Architecture After Change

## 5.1 Data Flow

### Before

```
RawRow → deserialize::<E> → execution
```

### After

```
RawRow → StructuralRow → execution
                           ↓
                       (optional)
                           ↓
                           E
```

---

## 5.2 Execution Layer

All execution paths operate on:

* slots
* `Value`
* structural readers

No entity typing inside execution kernels.

---

## 5.3 Commit Preparation

Already partially aligned:

* structural orchestration
* typed forward-index leaf

Next step:

* feed structural rows directly
* minimize typed leaf scope further if possible (without regression)

---

## 5.4 Relation and Index Layers

Remain structural:

* slot-based access
* key decoding
* predicate evaluation

Known hard boundary for now:

* forward-index planning in commit preparation remains a typed leaf unless a
  later experiment proves structuralization is a measured win

## 5.5 True Typed Boundaries

Typed persisted-row reconstruction remains correct at true boundaries:

* entity/response reconstruction
* external-facing typed DTO assembly
* any final diagnostic path that genuinely requires typed identity formatting

---

# 6. Migration Plan (Phased)

## Phase 1 — Canonical Structural Decode

* Introduce one canonical `decode_structural_row`
* Make the representation slot-indexed and lazy by default
* Ensure structural consumers can read slots without forcing `Value`
  materialization

## Phase 2 — Replace Runtime Decode Sites

Replace:

* non-boundary commit/runtime persisted-row decode
* non-boundary relation/runtime persisted-row decode
* any remaining helper decode that only proves structure or extracts slots

With structural equivalents

## Phase 3 — Isolate Typed Reconstruction

* centralize `StructuralRow → E`
* keep typed reconstruction only at explicit boundaries
* remove scattered non-boundary decode usage

## Phase 4 — Cleanup

* remove dead decode helpers
* eliminate redundant persisted-row `Deserialize` graphs

---

# 7. Expected Impact

## 7.1 Wasm Size

* Significant reduction in `serde_cbor`
* Elimination of per-entity decode duplication

## 7.2 Compile Time

* Reduced monomorphization
* Fewer LLVM codegen units

## 7.3 Runtime Performance

* Slight regression in decode-heavy paths possible
* Neutral or improved overall due to reduced branching elsewhere

---

# 8. Risks

## 8.1 Partial Adoption Risk

If structural decode is not universal:

* no size reduction
* possible regression

## 8.2 Reconstruction Bugs

* incorrect slot mapping
* nested record handling errors

## 8.3 Performance Trade-offs

* structural `Value` interpretation vs typed field access

## 8.4 Known Regression Trap

Structuralizing tight compute paths is not automatically a win.

Prior experiments showed regressions when structuralization pushed eager
materialization into paths like forward-index planning and unique validation.
`0.62` therefore targets persisted-row payload decode ownership, not tight
compute-path rewriting.

---

# 9. Non-Goals

* Removing entity types
* Flattening schema
* Rewriting domain model
* Optimizing serde internals
* Structuralizing tight compute paths that have already measured as regressions
* Protocol payload decode changes
* Commit-marker or commit-control payload decode changes
* Candid changes
* Persistence format replacement

---

# 10. Success Criteria

* No non-boundary `deserialize_row::<E>` / `try_decode::<E>` in runtime paths
* Single structural persisted-row decode entrypoint
* Reduced wasm size (measured via slope)
* No regression in correctness tests
* Preserved key-mismatch and decode-failure diagnostics
* Preserved corruption / unsupported / internal error classification
* Preserved persisted-row compatibility-window behavior

---

# 11. Strategic Position

After `0.62`:

* Execution fully structural
* Decode unified
* Remaining size dominated by:

  * Candid
  * Value representation

This enables future work:

* custom persistence codec
* schema-aware binary format

---

# 12. Summary

`0.62` is not an incremental cleanup.

It is:

> **A shift from entity-typed persistence to structural persistence with typed projection.**

This is the only remaining path to materially reduce wasm size beyond the 0.61 plateau.
