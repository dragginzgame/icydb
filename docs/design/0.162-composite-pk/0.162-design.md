# IcyDB 0.162 — Composite Primary Keys

## Status

In progress.

Implemented slices so far:

- formal `pk(fields = [...])` and `index(fields = [...])` parsing
- ordered `PrimaryKeyModel` metadata
- generated composite key structs and key codec impls
- fixed-capacity composite primary-key value construction
- compact composite `EncodedPrimaryKey` encoding/decoding
- raw data/index key taxonomy support for composite primary-key suffix bytes
- `PrimaryKeyCodec` / `PrimaryKeyDecode` type signatures moved to
  `PrimaryKeyValue` and primary-key-owned encode errors
- `DecodedDataStoreKey` and index primary-key suffix decode moved to
  scalar-or-composite `PrimaryKeyValue`
- composite primary-key declarations parse and generate typed `E::Key` structs
- accepted schema snapshots, schema descriptors, schema fingerprints, and DDL
  admission use ordered primary-key field metadata
- index mutation planning, unique-index preflight, commit index deltas,
  ordered key streams, and several materialized executor read paths use
  scalar-or-composite row identity
- exact fluent `by_id` / `by_ids` load paths and `delete().by_id(...)` paths
  accept composite primary-key identity through the scalar-or-composite
  `PrimaryKeyValue` boundary
- id-returning aggregate and projection terminals, including `min`, `max`,
  `min_by`, `max_by`, `nth_by`, `median_by`, `min_max_by`,
  `values_by_with_ids`, `top_k_by_with_ids`, and `bottom_k_by_with_ids`, carry
  scalar-or-composite identity through aggregate reducer/output boundaries
- grouped row decode, materialized numeric aggregates, and materialized bytes
  terminals use data-key-aware row decoding instead of scalar-only storage-key
  accessors
- index-range cursor anchor derivation and covering-row presence probes accept
  scalar-or-composite row identity instead of forcing scalar storage-key
  accessors
- index-anchor primary-key equivalence checks compare decoded compact
  `PrimaryKeyValue` anchors against scalar-or-composite boundary values
- deterministic `ORDER BY` normalization and cursor boundary validation use
  the ordered primary-key field list, so composite primary keys remain a full
  row-identity tie-break during cursor pagination
- secondary-order compatibility contracts recognize ordered composite
  primary-key suffixes instead of requiring a single terminal primary-key field
- covering projection primary-key sources carry ordered primary-key component
  indexes, so primary-store and hybrid covering routes can project composite
  primary-key fields without returning the whole key tuple
- primary-key null planning treats every ordered primary-key component as
  non-nullable row identity and lowers impossible `IS NULL` predicates to
  explicit empty access
- storage snapshot diagnostics accept compact composite data keys when counting
  per-entity storage rows, keeping `icydb snapshot` key-shape agnostic while
  proving composite rows do not inflate corrupted-key counts
- model-only EXPLAIN covering-route descriptors use the ordered generated
  primary-key field list, so composite primary-key projections map to component
  sources instead of the scalar primary-key bridge
- covering-index projection and bytes/distinct projection strategies compare
  deterministic secondary-order contracts against the full ordered primary-key
  suffix, so composite suffixes remain eligible for index-order fast paths

## Summary

0.162 adds first-class composite primary keys.

An entity primary key becomes an ordered, non-empty list of admitted scalar key
components. A scalar primary key is the one-component case and uses the same
field-list declaration syntax as composite primary keys.

Example:

```rust
#[entity(
    store = BattleStore,
    pk(fields = ["battle_id", "entity_id"]),
    index(fields = ["battle_id", "cell_id"])
)]
pub struct EntityPlacement {
    pub battle_id: BattleId,
    pub entity_id: EntityId,
    pub cell_id: u32,
    pub layer: u8,
    pub sort_key: u16,
}
```

The persisted row-key shape stays:

```text
EntityTag || EncodedPrimaryKey
```

but `EncodedPrimaryKey` becomes capable of encoding either:

- one scalar component
- an ordered composite list of scalar components

This release builds on the 0.159 compact key taxonomy. It must not reintroduce
64-byte `StorageKey` vocabulary as the primary-key model.

The long-term model is not “everything becomes composite.” Scalar and composite
primary keys remain distinct identity variants because they have different
typed API shapes:

- scalar entities use the scalar component as `E::Key`;
- composite entities use a generated named key struct as `E::Key`.

The unification point is the primary-key identity boundary:
`PrimaryKeyValue::Scalar` or `PrimaryKeyValue::Composite`. Cleanup work in
0.162 should move runtime, schema, index, commit, cursor, and executor paths to
that boundary. Any remaining `StorageKey` use is a transitional scalar bridge
only for paths whose contract is still genuinely scalar; those bridges must
fail closed for composite values and should be removed or narrowed as each path
becomes primary-key-value aware.

## Hard-Cut Policy

0.162 is a hard-cut release for primary-key behavior.

Do not add compatibility shims, aliases, fallback parsers, or old-name
re-exports for removed primary-key syntax or runtime behavior.

Specifically:

- `pk(field = "id")` is removed and must reject.
- `pk(fields = "id")` and comma-string primary-key parsing are not supported.
- scalar primary keys use `pk(fields = ["id"])`; there is no second scalar
  declaration lane.
- composite keys use generated named `<EntityIdent>Key` structs; do not add
  tuple aliases or alternate generated key names.
- generated key structs use their deterministic public name or fail on
  collision; do not silently choose a replacement name.
- accepted schema metadata must use the latest 0.162 shape; do not add
  multi-version persisted schema decode fallbacks for this pre-1.0 line.
- scalar-only helpers that become semantically wrong must be removed or made
  explicitly scalar-only and fail closed for composite keys.
- generated `EntityModel` must not become a runtime fallback authority for
  composite primary-key shape.

## Goals

- Support natural row identity such as:
  - `(battle_id, entity_id)`
  - `(canister_id, local_id)`
  - `(battle_id, turn_id, event_id)`
  - `(principal, subaccount)`
- Keep `Id<E>` and `EntityKey::Key` as the typed identity boundary.
- Preserve entity-prefix scans.
- Preserve secondary-index shape:
  - index components + primary-key suffix
- Make accepted schema snapshots the runtime authority for primary-key shape.
- Keep primary-key encoding allocation-free on ordinary encode/decode paths.
- Avoid surrogate IDs where the true identity is composite.
- Avoid string concatenation, JSON keys, blob keys, or ad hoc tuple encoding.

## Non-Goals

- No partial primary-key range planner in the first slice.
- No mutable primary-key updates.
- No nested composite key components.
- No `Text`, `Blob`, `many`, optional, indirect, map, list, set, or structured
  primary-key components in this release.
- No `Unit` component inside a composite primary key.
- No internal/generated composite primary keys.
- No legacy multi-version schema fallback.
- No redesign of secondary index storage.
- No runtime migration from an existing scalar primary key to a composite
  primary key.
- No relaxation of `EntityKey::Key` from `Copy` to `Clone`.
- No compatibility aliases for generated key types or removed primary-key
  syntax.

## Core Invariants

```text
An entity primary key is an ordered, non-empty list of admitted scalar key components.
A scalar primary key is exactly one component and is declared with pk(fields = ["..."]).
A composite primary key is declared with pk(fields = [...]) and has 2..=MAX_PRIMARY_KEY_FIELDS components.
Every row for one accepted entity uses the same primary-key arity, field order, durable field IDs, row slots, and component kinds.
```

Accepted schema snapshots are runtime authority for:

- primary-key arity
- primary-key field order
- primary-key field IDs
- primary-key row slots
- primary-key component kinds
- primary-key source

Generated models may propose or reconcile primary-key shape, but runtime
planning, decoding, DDL checks, and persistence decisions must use accepted
schema authority.

Storage ordering authority:

```text
EncodedPrimaryKey byte order is the storage order.
```

Any semantic `Ord` implementation must match encoded byte order exactly for all
values admitted by one accepted entity primary-key contract. Tests must compare
semantic ordering against encoded byte ordering, especially for signed values
and variable-width components such as `Principal`.

## Surface Syntax

Use one syntax for all primary keys:

```rust
#[entity(store = Store, pk(fields = ["id"]))]
```

Composite primary keys use the same syntax with multiple fields:

```rust
#[entity(store = Store, pk(fields = ["battle_id", "entity_id"]))]
```

This is a 0.162 hard cut. The old scalar-only spelling:

```rust
pk(field = "id")
```

must reject with guidance to use:

```rust
pk(fields = ["id"])
```

Validation:

- `fields` is required.
- `field` is not supported.
- `pk(fields = ["id"])` declares a scalar primary key.
- `pk(fields = [...])` declares a composite primary key when it contains
  `2..=MAX_PRIMARY_KEY_FIELDS` fields.
- `fields` must be a Rust array literal of string literals, not a comma-string.
- `pk(fields = "")` rejects.
- `pk(fields = [])` rejects.
- `pk(fields = ["", "id"])` rejects.
- duplicate fields reject.
- every component field must be:
  - present on the entity
  - cardinality `One`
  - non-optional
  - non-indirect
  - primary-key encodable
  - `Copy`
- relation primary-key components must declare an explicit primitive key type.

`Unit` remains admitted for scalar singleton primary keys. It is rejected as a
composite primary-key component in 0.162.

## Index Attribute Syntax

0.162 also hard-cuts generated index declarations to the same formal field-list
shape.

Use:

```rust
index(fields = ["battle_id", "cell_id"])
```

Do not use:

```rust
index(fields = "battle_id, cell_id")
```

Generated field-path indexes should not parse comma-strings. Each index key
item must be one explicit string literal in the array.

Expression index items may remain string literals inside the same array:

```rust
index(fields = ["tenant_id", "LOWER(email)"])
```

Expression strings inside `index(fields = [...])` are index-only generated
expression-index input. They must not influence primary-key syntax or
primary-key parsing, and they are not a compatibility lane for primary keys.

Primary keys must remain field-only and must not support expression terms.

## Primary-Key Source

Scalar primary keys keep the existing `internal` / `external` semantics.
Omitting `source` on a scalar primary key keeps the existing scalar default.

Composite primary keys are external-only in 0.162. Omitting `source` on a
composite primary key resolves to `external`. A declaration that combines
`pk(fields = [...])` with explicit internal/generated primary-key source
semantics must reject.

Reason: internal composite generation requires component-level generation rules.
IcyDB would need to define whether it generates all components, one component,
or some subset. That is a create-materialization and write-policy problem, and
it is deferred.

## Runtime Model

Replace scalar-only primary-key model fields:

```rust
primary_key: &'static FieldModel,
primary_key_slot: usize,
```

with a primary-key model:

```rust
pub struct PrimaryKeyModel {
    fields: &'static [&'static FieldModel],
    slots: &'static [usize],
}
```

`EntityModel` owns the model:

```rust
pub struct EntityModel {
    path: &'static str,
    entity_name: &'static str,
    primary_key: PrimaryKeyModel,
    fields: &'static [FieldModel],
    indexes: &'static [&'static IndexModel],
}
```

Scalar convenience accessors, if needed, must be explicit:

```rust
scalar_primary_key() -> Option<&'static FieldModel>
scalar_primary_key_slot() -> Option<usize>
```

Do not keep scalar-only `primary_key()` as the normal authority.

## Generated Key Type

For scalar entities:

```rust
type Key = Ulid;
```

For composite entities, derive generates a named key struct:

```rust
#[derive(
    ::icydb::__reexports::candid::CandidType,
    Clone,
    Copy,
    Debug,
    ::icydb::__reexports::serde::Deserialize,
    Eq,
    PartialEq,
    Ord,
    PartialOrd,
    Hash,
)]
#[candid_path("::icydb::__reexports::candid")]
#[serde(crate = "::icydb::__reexports::serde")]
pub struct EntityPlacementKey {
    pub battle_id: BattleId,
    pub entity_id: EntityId,
}
```

Generated composite key structs are public user-facing types because they are
the `E::Key` type used by load/delete/relation APIs.

0.162 requires generated composite key structs to be `Copy`. All admitted
composite primary-key component types must therefore be `Copy`. Relaxing
`EntityKey::Key` from `Copy` to `Clone` is deferred until non-copy primary-key
components, such as `Text`, are admitted.

Generated composite key structs use the deterministic name `<EntityIdent>Key`.
If that name collides with a user-defined item in the same module, derive
generation fails with a diagnostic. 0.162 must not silently choose an alternate
name. Explicit custom key type naming is deferred until there is a concrete
need.

Generated entity impls use the named key:

```rust
impl EntityKey for EntityPlacement {
    type Key = EntityPlacementKey;
}

impl EntityValue for EntityPlacement {
    fn id(&self) -> Id<Self> {
        Id::from_key(EntityPlacementKey {
            battle_id: self.battle_id,
            entity_id: self.entity_id,
        })
    }
}
```

Prefer generated named key structs over tuples for diagnostics, schema display,
Candid, and future SQL binding.

## Primary-Key Value Model

Do not add `StorageKey::Composite`.

0.159 already introduced a compact primary-key value and encoder taxonomy. 0.162
extends that taxonomy. Composite identity must never route through
`StorageKey`; scalar-only `StorageKey` paths are retained only where the code
is explicitly scalar and must fail closed for composite keys.

This split is permanent at the identity model level and temporary only at the
old scalar bridge level. The target executor/storage API should ask for
`PrimaryKeyValue` whenever it means “row identity.” It should ask for
`StorageKey` only when the operation is intentionally limited to scalar
runtime values, scalar SQL projection values, or an older row-slot decode
contract that has not yet been converted.

`PrimaryKeyValue` and `PrimaryKeyComponent` are internal runtime/storage
taxonomy. Normal users continue to interact with typed `E::Key` and `Id<E>`.

Target model:

```rust
pub const MAX_PRIMARY_KEY_FIELDS: usize = 4;

#[derive(Clone, Copy, Debug, Eq, PartialEq, Hash)]
pub enum PrimaryKeyComponent {
    Nat(u64),
    Int(i64),
    Timestamp(Timestamp),
    Ulid(Ulid),
    Principal(Principal),
    Subaccount(Subaccount),
    Account(Account),
    Unit,
}

#[derive(Clone, Copy, Debug, Eq, PartialEq, Hash)]
pub struct CompositePrimaryKeyValue {
    len: u8,
    components: [PrimaryKeyComponent; MAX_PRIMARY_KEY_FIELDS],
}

#[derive(Clone, Copy, Debug, Eq, PartialEq, Hash)]
pub enum PrimaryKeyValue {
    Scalar(PrimaryKeyComponent),
    Composite(CompositePrimaryKeyValue),
}
```

`PrimaryKeyComponent::Ord` must not rely on accidental enum declaration order.
It must compare through the same stable component kind tags used by
`EncodedPrimaryKey`. Implement `Ord` / `PartialOrd` manually through those tags
and the canonical component payload ordering.

`CompositePrimaryKeyValue` must expose safe constructors and iteration:

```rust
impl CompositePrimaryKeyValue {
    pub fn try_from_components(
        components: &[PrimaryKeyComponent],
    ) -> Result<Self, PrimaryKeyEncodeError>;

    pub fn components(&self) -> &[PrimaryKeyComponent];
}
```

The fixed-size storage may use an internal filler value, but filler state must
not be visible through `components()` or ordering/equality logic.

`PrimaryKeyValue::Composite` must not allocate on ordinary encode/decode paths.
If any implementation slice temporarily uses `Vec`, it needs explicit
wasm/instruction proof under the MSH hot-path gate.

## Primary-Key Errors

Move typed primary-key admission away from `StorageKeyEncodeError` naming.

Target shape:

```rust
pub enum PrimaryKeyEncodeError {
    UnsupportedComponentKind { kind: &'static str },
    ComponentTooLarge { component: usize, len: usize, max: usize },
    TooManyComponents { count: usize, max: usize },
    TooFewComponents { count: usize, min: usize },
}
```

Then:

```rust
trait PrimaryKeyCodec {
    fn to_primary_key_value(&self) -> Result<PrimaryKeyValue, PrimaryKeyEncodeError>;
}

trait PrimaryKeyDecode {
    fn from_primary_key_value(key: &PrimaryKeyValue) -> Result<Self, InternalError>;
}
```

Scalar impls return `PrimaryKeyValue::Scalar`.

Generated composite key impls return `PrimaryKeyValue::Composite`.

Primary-key traits use `PrimaryKeyValue` directly. Do not add compatibility
aliases from `StorageKeyEncodeError` or alternate old-name error wrappers.
Scalar-only callers may explicitly convert scalar values, but they must reject
composite values rather than adapting them through a compatibility path.

Decode should also use a primary-key-specific error taxonomy before higher
layers map corruption into store/internal errors:

```rust
pub enum PrimaryKeyDecodeError {
    Empty,
    UnknownKind { tag: u8 },
    InvalidLength { len: usize, expected: usize },
    InvalidCompositeCount { count: usize },
    UnitCompositeComponent,
    NestedComposite,
    CompositeNotScalar,
    TrailingBytes { len: usize },
}
```

## EncodedPrimaryKey Format

Current scalar format remains:

```text
scalar_kind_tag || scalar_payload
```

`pk(fields = ["id"])` is a scalar primary key and uses the existing scalar
`EncodedPrimaryKey` format. It must not encode as `COMPOSITE_TAG, count = 1`.

New composite format:

```text
COMPOSITE_TAG || component_count:u8 || component_1 || component_2 || ... || component_n
```

Each component is encoded as:

```text
scalar_kind_tag || scalar_payload
```

Rules:

- component count must be `2..=MAX_PRIMARY_KEY_FIELDS`.
- a composite cannot contain another composite.
- `Unit` cannot appear inside a composite.
- decoded component order must match accepted schema primary-key order.
- encoded byte order must match semantic lexicographic ordering for all values
  admitted by one entity.
- scalar component payloads must reuse the same primitive ordering as 0.159
  `EncodedPrimaryKey`.
- malformed tags, counts, lengths, or trailing bytes fail closed.

Ordering note:

All rows of one entity have the same accepted primary-key arity and component
kinds. The implementation only needs stable ordering inside that entity-local
namespace. Cross-entity or mixed scalar/composite ordering is not semantic API.

## Encoded Size Bounds

0.162 needs explicit encoded-size bounds for stable structures and commit
marker validation.

Target constants:

```rust
pub const MAX_PRIMARY_KEY_FIELDS: usize = 4;
pub const MAX_PRIMARY_KEY_COMPONENT_ENCODED_SIZE: usize = 1 + MAX_PRIMARY_KEY_COMPONENT_PAYLOAD_SIZE;
pub const MAX_COMPOSITE_PRIMARY_KEY_ENCODED_SIZE: usize =
    1 + 1 + MAX_PRIMARY_KEY_FIELDS * MAX_PRIMARY_KEY_COMPONENT_ENCODED_SIZE;
```

The exact constant names can match the existing key taxonomy, but the bound must
be explicit and conservative. It should cover the largest admitted scalar
component payload, currently the account payload.

## Data Store

Keep:

```text
RawDataStoreKey = EntityTag || EncodedPrimaryKey
```

Change `DecodedDataStoreKey` from scalar identity:

```rust
key: StorageKey
```

to composite-aware identity:

```rust
key: PrimaryKeyValue
```

All typed save/load/delete paths continue to accept `E::Key`.

Full entity scans must continue to use entity-prefix ranges, not fake minimum or
maximum primary-key sentinels.

## Secondary Indexes

Current index shape remains:

```text
IndexId || components[] || primary_key_suffix
```

but the primary-key suffix becomes composite-aware `EncodedPrimaryKey`.

This allows:

```text
index components: (battle_id, cell_id)
suffix:           (battle_id, entity_id)
```

so multiple occupants of one cell can coexist under the same cell index.

`IndexEntryValue` must remain presence/witness only. Row identity stays owned by
`RawIndexStoreKey`.

### Index Range Sentinels

Index range bounds may continue to use raw low/high primary-key suffix sentinel
segments to scan all rows for an index component prefix.

Those sentinels are range-bound artifacts only. They must never be decoded as
`EncodedPrimaryKey` row identities.

Cursor and row-identity decode paths must decode only real index entries, not
wildcard bounds.

## Accepted Schema Snapshot

Replace scalar-only accepted metadata:

```rust
primary_key_field_id: FieldId
```

with:

```rust
PersistedPrimaryKeySnapshot {
    fields: Vec<PersistedPrimaryKeyFieldSnapshot>,
    source: PrimaryKeySource,
}
```

Each field snapshot should include:

```rust
field_id: FieldId,
slot: SchemaFieldSlot,
name: String,
kind: PersistedFieldKind,
```

If primary-key field snapshots duplicate field facts such as slot, name, or
kind, schema snapshot integrity validation must prove they match the
corresponding accepted `PersistedFieldSnapshot` for the same durable field ID.

Bump the schema codec version.

Accepted schema, not generated `EntityModel`, must be runtime authority for
primary-key shape. This follows MSH’s rule that accepted schema snapshots are
runtime authority and generated models are limited to proposal, reconciliation,
model convenience, and tests.

## Schema Reconciliation

Reconciliation must reject any primary-key shape drift that would change
existing row-key encoding.

Reject:

- scalar primary key to composite primary key
- composite primary key to scalar primary key
- composite arity change
- composite field order change
- component field ID change
- component row slot change
- component kind change
- component nullability change to nullable
- component cardinality change away from `One`
- component storage codec change
- dropping any primary-key component field

Field rename may update display/name metadata only if durable primary-key field
IDs, row slots, arity, order, and component kinds remain unchanged.

## SQL / Structural Rules

Initial behavior:

- `INSERT` must provide all composite primary-key fields.
- `UPDATE` cannot mutate primary-key fields.
- `DELETE` by key requires the full composite key.
- exact lookup requires all primary-key components.
- partial primary-key prefix scans are deferred.

Structural mutations must validate agreement between the explicit `E::Key` and
any primary-key fields present in a structural patch or complete after-image.

- `INSERT` / replacement after-images must contain every primary-key component
  and must match the explicit key.
- Sparse `UPDATE` patches may omit primary-key fields.
- If a sparse patch includes a primary-key field, it must equal the existing
  explicit key component.
- Mismatches reject during mutation admission, before commit planning.

Later work may add primary-key prefix access planning.

## Relation Components

A relation used as a composite primary-key component must lower to exactly one
admitted scalar component.

A relation field may be a primary-key component only when its target entity has
a scalar primary key and the relation field lowers to that single admitted
scalar key component.

Relation fields targeting composite-key entities are not admitted as
primary-key components in 0.162. That avoids recursive composite relation shape
in the first release.

## Implementation Order

1. Add fixed-capacity `CompositePrimaryKeyValue` and composite
   `EncodedPrimaryKey` tests.
2. Move `PrimaryKeyCodec` / `PrimaryKeyDecode` onto `PrimaryKeyValue`.
3. Convert `DecodedDataStoreKey` to store `PrimaryKeyValue`.
4. Convert index primary-key suffix handling to composite-aware encoded keys.
5. Add explicit encoded-size bounds.
6. Add `PrimaryKeyModel`.
7. Add unified derive syntax: `pk(fields = [...])`.
8. Generate named composite key structs and key codec impls.
9. Update accepted schema snapshot and codec.
10. Update schema proposal/reconciliation.
11. Update structural and SQL mutation admission.
12. Add exact composite-key load/delete/query tests.
13. Remove or narrow scalar-only helpers.

## Audit Addendum: Composite-Key Coverage

Composite primary keys change the shape of the hottest row-identity paths. Any
audit that claims query performance, instruction cost, generated-code size, raw
wasm size, storage-key size, commit/recovery behavior, or index-entry footprint
must include at least one entity with a composite primary key once the audited
surface can construct such an entity.

Observability audits must also include composite primary-key entities. Schema
observability must expose ordered primary-key fields so CLI schema rendering and
schema-check drift detection can compare composite identity structurally.
Metrics and snapshot reports are key-shape agnostic today because they report
entity counters and storage inventory rather than row identity; if they start
reporting key identity or key-size details, they must add composite-key fixtures
at that point.

Scalar-only fixtures remain useful as baselines, but they are no longer
sufficient proof for 0.162 key-path work. Audit harnesses should include both:

- a scalar primary-key entity, to preserve the existing baseline;
- a composite primary-key entity with at least two admitted scalar components.

For size and wasm audits, generated composite key structs and their codec impls
must be represented in the measured canister when the audit scope includes
generated schema/model code. For runtime performance and instruction audits,
measured paths should cover at minimum:

- insert/save by composite key;
- exact load/delete by composite key;
- full entity scan over composite-key rows;
- secondary index scan where the index key suffix is a composite primary key;
- commit marker/recovery decode for rows with composite primary keys.

Any audit result that still uses only scalar-primary-key entities must say so
explicitly and must not be used as complete evidence for composite-key release
readiness.

## Deferred Work

- primary-key prefix scans.
- range queries over composite primary-key components.
- SQL planner selection for partial primary-key predicates.
- migrations from scalar primary key to composite primary key.
- `Text` primary-key components.
- relaxing `EntityKey::Key` from `Copy` to `Clone`.
- internal/generated composite primary keys.
- relation fields targeting composite-key entities as primary-key components.
- richer CLI display polish beyond ordered primary-key field reporting.

## Verdict

0.162 should make composite primary keys a first-class identity feature.

Do not implement this as:

- hidden surrogate ID + unique composite index
- string-concatenated keys
- JSON/blob keys
- `Vec<Value>` runtime key comparisons
- `Vec<PrimaryKeyComponent>` on ordinary hot key paths
- generated-model runtime fallback

The clean shape is:

```text
PrimaryKeyModel
PrimaryKeyValue
CompositePrimaryKeyValue
PrimaryKeyEncodeError
EncodedPrimaryKey
PersistedPrimaryKeySnapshot
Generated composite key structs
Composite-aware DecodedDataStoreKey
Composite-aware index suffixes
```
