# IcyDB 0.162 — Composite Primary Keys

## Status

Proposed

## Summary

0.162 adds first-class composite primary keys.

A primary key becomes an ordered, non-empty list of scalar key components. A
scalar primary key is treated as the one-component case.

Example:

```rust
#[entity(
    store = BattleStore,
    pk(fields = ["battle_id", "entity_id"]),
    index(fields = ["battle_id", "cell_id"])
)]
pub struct EntityPlacement {
    pub battle_id: BattleId,
    pub entity_id: EntityId,
    pub cell_id: u32,
    pub layer: u8,
    pub sort_key: u16,
}
````

The persisted row key remains:

```text
EntityTag || EncodedPrimaryKey
```

but `EncodedPrimaryKey` becomes capable of encoding either a scalar key or a
composite key.

## Goals

* Support natural row identity such as:

  * `(battle_id, entity_id)`
  * `(tenant_id, slug)`
  * `(canister_id, local_id)`
  * `(battle_id, turn_id, event_id)`
* Keep `Id<E>` and `EntityKey::Key` as the typed identity boundary.
* Preserve entity-prefix scans.
* Preserve secondary-index shape:

  * index components + primary-key suffix.
* Make accepted schema snapshots the runtime authority for primary-key shape.
* Avoid surrogate IDs where the true identity is composite.
* Avoid string concatenation, JSON keys, or ad hoc tuple encoding.

## Non-Goals

* No partial primary-key range planner in the first slice.
* No mutable primary-key updates.
* No nested composite key components.
* No composite keys containing `many`, `optional`, indirect, blob, text, map,
  list, set, or structured values.
* No legacy multi-version schema fallback unless explicitly required by migration
  policy.
* No redesign of secondary index storage.

## Core Invariant

```text
An entity primary key is an ordered, non-empty list of scalar key components.
A scalar primary key is exactly a one-component primary key.
```

## Surface Syntax

Keep existing scalar syntax:

```rust
#[entity(store = Store, pk(field = "id"))]
```

Add composite syntax:

```rust
#[entity(store = Store, pk(fields = ["tenant_id", "slug"]))]
```

Validation:

* `field` and `fields` are mutually exclusive.
* `fields` must be non-empty.
* no duplicate fields.
* max arity: `MAX_PRIMARY_KEY_FIELDS`, initially `4`.
* every component must be:

  * present on the entity
  * cardinality `One`
  * non-optional
  * non-indirect
  * storage-key encodable
* relation PK components must declare explicit primitive key type.

## Runtime Model

Replace scalar primary-key model fields:

```rust
primary_key: &'static FieldModel,
primary_key_slot: usize,
```

with:

```rust
pub struct PrimaryKeyModel {
    fields: &'static [&'static FieldModel],
    slots: &'static [usize],
}
```

`EntityModel` owns:

```rust
pub struct EntityModel {
    path: &'static str,
    entity_name: &'static str,
    primary_key: PrimaryKeyModel,
    fields: &'static [FieldModel],
    indexes: &'static [&'static IndexModel],
}
```

Compatibility accessors, if needed, must be explicit:

```rust
scalar_primary_key() -> Option<&'static FieldModel>
scalar_primary_key_slot() -> Option<usize>
```

Do not keep scalar-only `primary_key()` as the normal authority.

## Generated Key Type

For scalar entities:

```rust
type Key = Ulid;
```

For composite entities, derive generates:

```rust
#[derive(Clone, Copy, Debug, Eq, PartialEq, Ord, PartialOrd, Hash)]
pub struct EntityPlacementKey {
    pub battle_id: BattleId,
    pub entity_id: EntityId,
}
```

Then:

```rust
impl EntityKey for EntityPlacement {
    type Key = EntityPlacementKey;
}

impl EntityValue for EntityPlacement {
    fn id(&self) -> Id<Self> {
        Id::from_key(EntityPlacementKey {
            battle_id: self.battle_id,
            entity_id: self.entity_id,
        })
    }
}
```

Prefer generated named key structs over tuples for diagnostics, schema display,
Candid, and future SQL binding.

## Primary-Key Value Model

Do not add `StorageKey::Composite`.

Introduce:

```rust
pub enum PrimaryKeyComponent {
    Nat(u64),
    Int(i64),
    Timestamp(Timestamp),
    Ulid(Ulid),
    Principal(Principal),
    Subaccount(Subaccount),
    Account(Account),
    Unit,
}

pub enum PrimaryKeyValue {
    Scalar(PrimaryKeyComponent),
    Composite(Vec<PrimaryKeyComponent>),
}
```

Update:

```rust
trait PrimaryKeyCodec {
    fn to_primary_key_value(&self) -> Result<PrimaryKeyValue, StorageKeyEncodeError>;
}

trait PrimaryKeyDecode {
    fn from_primary_key_value(key: PrimaryKeyValue) -> Result<Self, InternalError>;
}
```

Scalar impls return `PrimaryKeyValue::Scalar`.

Generated composite key impls return `PrimaryKeyValue::Composite`.

## EncodedPrimaryKey Format

Current scalar format remains:

```text
scalar_kind_tag || scalar_payload
```

New composite format:

```text
COMPOSITE_TAG || component_count:u8 || component_1 || component_2 || ... || component_n
```

Each component is:

```text
scalar_kind_tag || scalar_payload
```

Rules:

* component count must be `1..=MAX_PRIMARY_KEY_FIELDS`.
* composite cannot contain composite.
* decoded order must match semantic order.
* raw byte ordering must match `PrimaryKeyValue::Ord`.

## Data Store

Keep:

```text
RawDataStoreKey = EntityTag || EncodedPrimaryKey
```

Change `DecodedDataStoreKey` from:

```rust
key: StorageKey
```

to:

```rust
key: PrimaryKeyValue
```

All typed save/load/delete paths continue to accept `E::Key`.

## Secondary Indexes

Current index shape remains:

```text
IndexId || components[] || primary_key_suffix
```

But the primary-key suffix becomes composite-aware `EncodedPrimaryKey`.

This allows:

```text
index components: (battle_id, cell_id)
suffix:           (battle_id, entity_id)
```

so multiple occupants of one cell can coexist under the same cell index.

## Accepted Schema Snapshot

Replace:

```rust
primary_key_field_id: FieldId
```

with:

```rust
PersistedPrimaryKeySnapshot {
    fields: Vec<PersistedPrimaryKeyFieldSnapshot>,
    source: PrimaryKeySource,
}
```

Each field snapshot should include:

```rust
field_id: FieldId,
slot: SchemaFieldSlot,
name: String,
kind: PersistedFieldKind,
```

Bump schema codec version.

Accepted schema, not generated `EntityModel`, must be runtime authority for
primary-key shape. This follows MSH’s rule that accepted schema snapshots are
runtime authority and generated models are limited to proposal, reconciliation,
model convenience, and tests.

## SQL / Structural Rules

Initial behavior:

* `INSERT` must provide all composite PK fields.
* `UPDATE` cannot mutate primary-key fields.
* `DELETE` by key requires the full composite key.
* exact lookup requires all PK components.
* partial primary-key prefix scans are deferred.

Later work may add primary-key prefix access planning.

## Implementation Order

1. Add `PrimaryKeyComponent`, `PrimaryKeyValue`, and composite
   `EncodedPrimaryKey` tests.
2. Convert `DecodedDataStoreKey` to store `PrimaryKeyValue`.
3. Convert index primary-key suffix handling to composite-aware encoded keys.
4. Add `PrimaryKeyModel`.
5. Add derive syntax: `pk(fields = [...])`.
6. Generate named composite key structs.
7. Update accepted schema snapshot and codec.
8. Update schema proposal/reconciliation.
9. Update structural and SQL mutation admission.
10. Add exact composite-key load/delete/query tests.
11. Remove or narrow scalar-only helpers.

## Required Tests

### Compile / derive

* scalar `pk(field = "id")` still works.
* composite `pk(fields = ["a", "b"])` works.
* duplicate PK field rejects.
* missing PK field rejects.
* optional PK field rejects.
* many-cardinality PK field rejects.
* indirect PK field rejects.
* too many PK fields rejects.

### Encoding

* scalar golden encodings remain stable.
* composite roundtrip.
* composite ordering:

  * `(1, 1) < (1, 2) < (2, 0)`
* malformed component count rejects.
* nested composite rejects.

### Data store

* insert/load/update/delete by composite key.
* duplicate composite insert rejects.
* entity-prefix scan still works.
* batch duplicate detection catches duplicate composite keys.

### Indexes

* secondary index entries use composite suffix.
* same index components with different composite PKs coexist.
* unique index collision detection still works.
* cursor/index continuation roundtrips composite suffix.

### Schema

* schema snapshot encodes/decodes composite PK.
* accepted schema exposes PK field IDs and slots.
* field drop rejects PK fields.
* field rename preserves durable PK field IDs.

## Validation

Required:

```text
cargo fmt --all
cargo check -p icydb-core
cargo test -p icydb-core primary_key
cargo test -p icydb-core index
cargo test -p icydb-core schema
cargo clippy -p icydb-core --all-targets -- -D warnings
git diff --check
```

If generated code or key-path code changes raw wasm shape, run the minimal wasm
audit and mark the result comparable only if a baseline artifact exists. MSH
requires proof for hot or wasm-sensitive cleanup and treats raw `.wasm` bytes as
the primary size signal.

## Deferred Work

* primary-key prefix scans.
* range queries over composite PK components.
* SQL planner selection for partial PK predicates.
* migrations from scalar PK to composite PK.
* public schema introspection display for composite PKs.
* optional Candid helpers for generated composite key structs.

## Verdict

0.162 should make composite primary keys a first-class identity feature.

Do not implement this as:

* hidden surrogate ID + unique composite index
* string-concatenated keys
* JSON/blob keys
* `Vec<Value>` runtime key comparisons
* generated-model runtime fallback

The clean shape is:

```text
PrimaryKeyModel
PrimaryKeyValue
EncodedPrimaryKey
PersistedPrimaryKeySnapshot
Generated composite key structs
Composite-aware DecodedDataStoreKey
Composite-aware index suffixes
```
