# 0.69 Secondary Row Authority Addendum

## Purpose

This addendum captures the next design question after the first secondary
covering execution optimizations plateaued:

> how can secondary covering reads become authoritative enough to avoid a
> separate `data.contains(...)` probe, without weakening correctness or moving
> authority into the wrong layer?

This is intentionally narrower than the main `0.69` design doc. It does not
try to redesign all covering execution. It focuses on the remaining step
between:

- a clean, fail-closed `row_check_required` secondary covering route
- a future secondary cohort that can deliver a much larger execute-phase drop

## What The Last Local Pass Proved

The current local optimization line is no longer dominated by accidental
materialization overhead.

The representative SQL cohort was originally framed as:

```sql
SELECT id, name FROM Customer ORDER BY name ASC, id ASC LIMIT 2
SELECT id, name FROM Customer ORDER BY name DESC, id DESC LIMIT 2
```

Across the staged secondary-covering cleanups, the total movement from the
original baseline is now:

- ascending generated: `675,563 -> 658,545` (`-2.52%`)
- ascending typed: `672,342 -> 655,047` (`-2.57%`)
- descending generated: `680,802 -> 662,659` (`-2.66%`)
- descending typed: `677,387 -> 659,407` (`-2.65%`)

The most recent partial-decode slice only bought another small step:

- ascending generated: `662,192 -> 658,545` (`-0.55%`)
- ascending typed: `659,878 -> 655,047` (`-0.73%`)
- descending generated: `666,439 -> 662,659` (`-0.57%`)
- descending typed: `664,659 -> 659,407` (`-0.79%`)

That is the key signal. The local path is getting cleaner and cheaper, but the
route is now close to its honest ceiling while it still pays for a separate
authoritative row-existence check.

## Current Constraint

Today the secondary covering route is still explicit:

- planner/runtime contract: `existing_row_mode = row_check_required`
- executor behavior: secondary index hit is not trusted on its own
- authority check: `data.contains(...)` remains the authoritative row-presence
  check

That is correct because stale secondary entries are still semantically real.

The current implementation should therefore be understood as:

- good execution-shape cleanup
- not yet a row-authority breakthrough

## What The Current Witness Actually Proves

The current `secondary_covering_authoritative` state is narrower than it may
look from the route name alone.

It is not a per-entry existence proof. It is a synchronized store-pair fact:

- `DataStore` and `IndexStore` each carry one
  `secondary_covering_authoritative` boolean
- any direct row-store mutation invalidates the row-side boolean immediately
- any direct index-store mutation invalidates the index-side boolean
  immediately
- `StoreHandle::secondary_covering_authoritative()` is only true when both
  sides still advertise the synchronized state

The restore path is equally explicit:

- commit apply restores the witness only after the shared commit window closes
  successfully and the touched row/index store pairs are marked back to
  synchronized state
- startup recovery restores the witness only after row-op replay, full
  secondary-index rebuild, post-recovery integrity validation, and marker
  clearing all succeed

That makes the current witness good for one narrow claim:

> this row store and this secondary index store were last brought back into a
> known synchronized state by authoritative commit or recovery machinery

That is enough for honest `witness_validated` route admission on the current
shipped cohorts.

It is not enough, on its own, to prove that one specific secondary entry still
corresponds to one currently visible base row without more candidate-level
authority.

## Problem Statement

The remaining large fixed cost on secondary covering reads is not generic
projection materialization anymore. It is the existence of a second
authoritative lookup boundary at all.

If the engine must always do:

1. traverse secondary index
2. decode index output
3. probe authoritative row store for existence
4. only then emit output

then the route can improve incrementally, but it will not deliver a
step-function execute drop.

So the next `0.69` question is not:

> how do we make the current `row_check_required` path a little cheaper?

It is:

> when, if ever, can secondary covering become authoritative enough that the
> extra existence probe is not required as a separate row-store lookup?

## Success Criterion

The success criterion for this design line should stay concrete.

Target query:

```sql
SELECT id, name FROM Customer ORDER BY name ASC, id ASC LIMIT 2
```

Success means:

- a visible step-function drop, not another `~1%` improvement
- unchanged stale-row semantics
- planner-owned admission, not executor-local upgrading
- explain surfaces remain explicit about the authority mode in use

## Acceptable Directions

### 1. Planner-Proven Secondary Authority Cohorts

Admit a secondary cohort only if the planner can prove that the chosen route is
authoritative for row existence under current semantics.

This would require an explicit route contract, not an inferred optimization.

Conceptually:

```rust
enum SecondaryRowAuthorityMode {
    PlannerProven,
    WitnessValidated,
    SeparatePresenceProbe,
}
```

This is the cleanest answer if it exists, but it is also the hardest one to
justify. The burden of proof is high:

- stale secondary entries must be impossible for the admitted cohort
- the proof must come from planner-owned semantics or structural invariants
- executor must consume the result, not rediscover it

If that proof cannot be written clearly, this direction should fail closed.

### 2. Storage-Level Existence Witness

Introduce a storage-owned witness that can validate row existence without a
second row-store lookup.

Examples could include:

- tombstone-aware witness bits
- version or epoch witnesses
- co-maintained existence markers whose semantics are explicit and durable

The important rule is that this remains a storage/runtime contract, not a
heuristic.

Good shape:

- one explicit witness contract
- one explicit validation step
- explainable route semantics

Bad shape:

- “usually fresh” assumptions
- recency-based shortcuts
- implicit trust in index maintenance timing

### 3. Fused Authority Probe

Allow the storage/runtime layer to answer row existence while the secondary key
is already hot, without crossing a second logical lookup boundary.

This is only acceptable if it stays semantically equivalent to the current
authoritative check.

The goal would be:

- preserve fail-closed semantics
- reduce the cost of the check
- avoid a planner lie about secondary authority

This is less ambitious than a fully planner-proven cohort, but it may be the
most realistic next step if the storage engine can support it cleanly.

## What The Fused Probe Proved

That narrow fused-authority prototype has now been implemented on the stale
single-component covering collector and measured on the real stale fallback
cohort:


stale.

- the collector no longer builds one `DataKey` and then calls the generic

- stale fallback semantics
- row-check counters
- the amount of authoritative work per candidate

## What The Storage-Level Existence Witness Proved

The next prototype did change what authoritative work happens.

It stays deliberately narrow:

- cohort:
  `SELECT name FROM Customer ORDER BY name ASC LIMIT 2`
- stale setup:
  remove the leading base row while keeping the secondary `name` entry stale
- route rule:
  only the stale single-component secondary order-only covering cohort may
  consume the storage-owned existence witness

The boundary ownership stays explicit:

- storage owns one per-membership existence witness on the raw secondary entry
- planner/runtime expose that through
  `existing_row_mode = storage_existence_witness`
- executor consumes it while keeping row-visibility policy and stale fallback
  semantics executor-owned

What changed:

- the stale one-column route no longer performs a separate borrowed-data-store
  authoritative probe per candidate
- the stale one-column route no longer re-encodes a primary-row raw key per
  candidate
- generic `query_from_sql(...).explain_execution()` still stays conservative
  and reports `row_check_required`
- store-backed SQL explain/runtime promote only this narrow stale cohort to
  `storage_existence_witness`
- broader stale bounded-range and extra-component covering projections still
  stay on `row_check_required`

noise:

  `481,193 -> 454,594` (`-26,599`, `-5.53%`)

The fused helper only changed seam ownership.

## Current Steering Rule

The current `0.69` authority rule should now stay explicit:

- keep the landed stale storage-witness family
- keep adjacent widenings only if they remove authoritative work **and**
  produce a clearly non-noise same-query runtime delta
- reject adjacent widenings by analogy when they only reduce some internal
  probe counters without moving end-to-end totals enough to matter

That means the current storage witness design now has seven kept stale-path
wins and one measured adjacent rejection.

That is a good stopping point, not a prompt to keep widening the same
contract opportunistically.

## Stronger Stored Witness Versus Storage-Level Existence Witness

The next design choice is now clearer because the current synchronized witness
has real code behind it.

### Stronger Explicit Stored Witness

This path means enriching or hardening the current synchronized store-pair
witness story.

Examples:

- making the synchronized state more explicit or durable
- storing richer witness metadata than one boolean
- recording stronger commit/recovery provenance for one row/index store pair

This can still be valuable. It may widen truthful `witness_validated`
admission, reduce ambiguity in explain surfaces, or make recovery semantics
clearer.

But it is still mostly a stronger version of today’s proof:

- the row store and index store are synchronized as a pair
- the route may rely on that pair-level witness for admitted cohorts

By itself, that does **not** obviously remove the per-candidate authoritative
existence step on stale-prone `row_check_required` routes. It strengthens route
admission semantics more than it changes the amount of work the executor must
still do for each candidate.

### Storage-Level Existence Witness

This path is different. It aims to change what authoritative work happens for
each candidate.

The core idea is:

- keep row visibility policy executor-owned
- keep planner admission explicit and fail closed
- let storage supply one explicit existence witness that can be consumed during
  covering validation without a separate logical row-store lookup boundary

This does **not** mean trusting the index blindly or moving row authority into
`db::index`.

It means designing one storage/runtime contract that can answer:

> does this candidate still have authoritative row existence under the current
> visibility rules?

without forcing the current separate `key -> raw key -> data.contains(...)`
style boundary per candidate.

If a step-function secondary covering win still exists under the current
semantic posture, this is the most credible remaining place for it to come
from.

## Current Cohort Classification

Under the current storage and runtime contract, the cohort classification is
already fairly clear.

### `planner_proven` Under Current Rules

None of the currently admitted secondary cohorts look honestly
`planner_proven`.

Why:

- stale secondary entries are already treated as semantically real runtime
  state
- the executor test suite explicitly requires secondary reads to skip or error
  on stale keys by consulting primary-row authority
- the planner can see access shape and predicate semantics, but it cannot prove
  that a secondary entry is current at query time under the existing storage
  contract

So the current answer is:

> under today’s model, there is no credible secondary-index cohort that should
> be upgraded directly from `row_check_required` to `planner_proven`

That includes the seemingly simple cohort:

```sql
SELECT id, name FROM Customer ORDER BY name ASC, id ASC LIMIT 2
```

Its shape is narrow and hot, but it is still secondary-index backed and still
inherits stale-entry risk.

### Best Candidates For A New Authority Contract

These are the best candidates for a witness-based or fused-authority follow-on,
ordered by expected leverage and complexity.

1. Pure order-only single-component secondary covering reads
   - Example:
     ```sql
     SELECT id, name FROM Customer ORDER BY name ASC, id ASC LIMIT 2
     ```
   - Why first:
     - narrowest projection
     - one secondary component
     - no residual predicate
     - already measured in the perf harness
   - Expected mode:
     - likely `WitnessValidated` or fused-authority, not `planner_proven`

2. Bounded single-component secondary ranges
   - Example:
```sql
SELECT id, name
FROM Customer
WHERE name >= 'a' AND name < 'b'
ORDER BY name ASC, id ASC
LIMIT 2
     ```
   - Why second:
     - same basic authority problem as order-only
     - slightly more runtime complexity from range bounds

3. Filtered or composite secondary covering reads
   - Example:
```sql
SELECT id, tier, handle
FROM CustomerAccount
WHERE active = true
ORDER BY handle ASC, id ASC
LIMIT 2
```
   - Why later:
     - more moving parts
     - greater risk of accidentally mixing row-authority and guard semantics

4. Expression-index covering reads
   - Example:
```sql
SELECT id, tier, handle
FROM CustomerAccount
WHERE active = true AND tier = 'gold'
ORDER BY LOWER(handle) ASC, id ASC
LIMIT 2
     ```
   - Why last:
     - same row-authority problem plus extra decode/canonicalization complexity
     - not the place to start a new authority contract

### Practical Conclusion

The first implementation cohort for secondary row authority should stay as
close as possible to:

```sql
SELECT id, name FROM Customer ORDER BY name ASC, id ASC LIMIT 2
```

That query is good because:

- it is already measured
- it has already received the local cleanup passes
- its remaining cost is dominated by the authority boundary itself
- it is simple enough that a new witness or fused-authority contract will be
  easy to evaluate honestly

## What Must Not Happen

### Do Not Weaken `row_check_required`

No:

- probabilistic skipping
- “recent index writes are safe” assumptions
- silent fallback from missing checks to optimistic reads

If the route still depends on row visibility, the contract must continue to say
so explicitly.

### Do Not Push Authority Into `db::index`

The index layer may emit entries or expose a narrow witness surface, but it
must not become the owner of:

- row-presence semantics
- projection policy
- fallback eligibility

The boundary should remain:

- index/storage emits or validates under a narrow contract
- planner declares the route mode
- executor consumes the contract

### Do Not Invent A Fake Secondary `planner_proven` Cohort

The measured gap is not a license to widen semantics dishonestly.

If a future patch cannot explain, in one sentence, why a secondary route is
authoritative, it should not ship as `planner_proven`.

## Recommended Next Slice

The best next slice is to design one explicit authority contract before more
runtime tuning.

Recommended order:

1. treat current secondary `planner_proven` widening as closed for now
   - no current cohort qualifies honestly under the existing storage contract
2. start with the narrow order-only `Customer ORDER BY name` cohort as the first
   witness/fused-authority candidate
3. design one explicit witness or fused-authority contract for
   `row_check_required`
4. only then implement the first narrow runtime cohort against that contract
5. widen to bounded, filtered, composite, and expression-index cohorts only if
   the same authority contract still explains the behavior cleanly

That first candidate step is now complete, and the result narrows the next
phase:

1. keep the fused authority helper as explicit executor-owned contract cleanup
2. stop spending time on `row_check_required` seam repackaging
3. only pursue another secondary-authority slice if it changes what
   authoritative work happens, not just where the helper lives
4. otherwise the next honest big win requires one stronger explicit stored
   witness or storage-level existence witness design

Given the current implementation seams, those two options are not equally
promising:

- a stronger explicit stored witness is the cleaner next documentation and
  contract direction if the goal is to make the current synchronized-store
  story more durable and explicit
- a storage-level existence witness is the more credible next headline if the
  goal is to remove one per-candidate authoritative step on stale-prone
  secondary covering reads

So the practical priority is:

1. stop widening and stop stale-seam repackaging
2. treat the current synchronized witness as good enough for truthful
   `witness_validated` admission, but not as the likely breakthrough lever
3. design one explicit storage-level existence witness contract next
4. only fall back to “stronger stored witness” work first if the storage-level
   contract cannot be stated cleanly without weakening semantics

```sql
SELECT name FROM Customer ORDER BY name ASC LIMIT 2
```

with the leading base row removed while the secondary `name` entry remains
stale.

That is still the best first probe because:

- it is already instrumented and runtime-verified
- the stale-path counters are already known
- it isolates the exact authority boundary that still remains

The prototype should aim to replace:

- one decoded secondary candidate
- plus one separate logical
  `storage_key -> raw key -> data.contains(...)` style step

with:

- one executor-consumed storage witness check that answers authoritative row
  existence under the same fail-closed semantics

The ownership split should stay strict:

- planner still declares the authority mode
- executor still owns row visibility policy and stale fallback semantics
- storage provides one explicit witness surface
- `db::index` still does not become the owner of row-authority policy

The first candidate API should therefore look more like:

```rust
enum SecondaryExistenceWitnessDecision {
    Present,
    Missing,
    Unavailable,
}
```

consumed through one explicit store/runtime boundary, and not like a new
planner-local boolean or an index-owned shortcut.

The required proof for that prototype is also explicit:

- the stale cohort must still emit only `["bob"]`
- stale fallback semantics must stay exact and fail closed
- `row_presence_probe_count` must drop because the authoritative step itself
  changed, not because metrics were relabeled
- same-query PocketIC movement must be clearly above noise

If a proposal cannot satisfy those points, it is not the next headline. It is
just another repackaging of the current stale-authority seam.

That keeps the next step centered on:

- one route-level authority decision
- one explicit runtime validation mode
- one measurable SQL cohort

## Working Rule

Use this rule for the next `0.69` slice:

> If secondary covering is going to get a breakthrough-class perf win, it must
> come from changing what authoritative work the route performs, not from
> polishing the current `row_check_required` path a little more.

If a proposed optimization does not change that authority story, it is probably
not the next major `0.69` lever.
