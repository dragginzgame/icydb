# 0.69 Read Boundary And Materialization Addendum

## Purpose

This addendum captures the next executor-adjacent design line after the first
`0.69` covering-read wins:

> how do we reduce encode/decode and materialization cost on read-heavy paths
> without reopening the structural-row integrity hole that `0.65.1` closed?

This is not a generic “make reads lazy” note. The current repo history already
rejected that direction once.

The design goal is narrower and stricter:

- preserve the canonical-row hard-cut introduced in `0.65.1`
- stop paying full semantic materialization cost when callers only need a
  subset of fields or cells

The right shape is therefore:

```text
raw row
  -> parse envelope
  -> validate every declared slot against its storage contract
  -> materialize only requested slots or cells
  -> consumer
```

not:

```text
raw row
  -> parse envelope
  -> decode only touched fields
  -> consumer
```

## Existing Hard Constraint

`0.65.1` made the structural read boundary itself authoritative.

The changelog in `docs/changelog/0.65.md` and the current
`StructuralSlotReader::from_raw_row(...)` comments in
`crates/icydb-core/src/db/data/persisted_row/mod.rs` both state the same rule:

- every declared slot must be validated before the reader is returned
- malformed unprojected fields must fail when the row is opened
- later consumers must not be able to observe a row that passed shape checks
  but still hides bad unused slot payloads

That contract is also locked by regression coverage, including:

- malformed unprojected scalar payload rejection in
  `crates/icydb-core/src/db/executor/projection/tests.rs`
- slot-count and slot-span corruption rejection in
  `crates/icydb-core/src/db/data/persisted_row/mod.rs`

So the design line here is not “undo `0.65.1`.” It is:

> keep row-open validation fail-closed, but stop forcing full runtime object
> construction for untouched fields.

## Audit Summary

The current codebase has many encode/decode call sites, but the live read-path
cost collapses to a few important boundaries.

### 1. Structural Row Boundary

The hottest shared read boundary is:

- `crates/icydb-core/src/db/data/structural_row.rs`
- `crates/icydb-core/src/db/data/persisted_row/mod.rs`

Today `StructuralSlotReader::from_raw_row(...)` does:

1. parse the structural row envelope and slot spans
2. allocate one slot cache
3. call `decode_all_declared_slots()`
4. decode every declared slot into cached runtime values
5. return the fully materialized reader

That is structurally correct under `0.65.1`, but it means row-backed reads pay
full semantic decode even when the caller only needs a narrow projection.

### 2. Secondary Membership And Index-Key Decode

The next hot boundary is:

- `crates/icydb-core/src/db/index/entry.rs`
- `crates/icydb-core/src/db/index/scan.rs`
- `crates/icydb-core/src/db/index/key/codec/mod.rs`

Today `RawIndexEntry::decode_keys()` validates the frame and allocates
`Vec<StorageKey>` before scan callers consume those keys. Some scan paths then
also decode raw index-key bytes into runtime components or `Value`.

This boundary is lower risk than the structural row reader and is a good first
optimization target because it does not require changing canonical-row
semantics.

### 3. Covering Projection Decode

The current covering path still turns admitted covering component bytes into
runtime `Value` before terminal SQL rendering:

- `crates/icydb-core/src/db/executor/covering.rs`
- `crates/icydb-core/src/db/executor/pipeline/operators/terminal/runtime.rs`

That is appropriate for generic executor use, but it is more work than a
SQL-only terminal needs when the output is a small direct scalar cell.

### 4. Cold Or Lower-Leverage Boundaries

These still matter, but they are not the first read-side targets:

- cursor token encode/decode
- migration-state encode/decode
- commit control-slot encode/decode
- write-side `ByKind` CBOR AST staging
- generic `FieldStorageDecode::Value` rewrites

Those should stay behind the hotter read-side work unless profiling proves
otherwise.

## Working Design Rule

Use this rule for the next read-side work:

> split validation from semantic materialization, not validation from access.

That means:

- keep `StructuralSlotReader::from_raw_row(...)` as the strict shared boundary
- validate every declared slot before returning the reader
- defer expensive runtime object construction until a caller actually requests
  that slot or cell

The important consequence is that “validated” becomes a first-class state, not
an implicit synonym for “already materialized into `Value`.”

## Reader Refactor Shape

### Phase A: Validate All Declared Slots

At construction time, the reader should still enforce:

- slot-table bounds
- declared-slot completeness
- scalar envelope correctness
- payload shape validity for each slot
- primary-key slot availability and storage-key encodability where that is part
  of the shared invariant

This phase preserves the `0.65.1` fail-closed behavior.

### Phase B: Materialize On Demand

After construction, slot access should only pay semantic decode for the slots
the caller actually requests.

This is the key cost-structure change:

- construction still validates all declared slots
- later projection, predicate, relation, and index consumers only materialize
  what they touch

### Slot Cache State

The current cache effectively has:

- `Pending`
- `Decoded(Value)`

The next honest shape should be more explicit. Conceptually:

- `Pending`
- `ValidatedScalar(...)`
- `ValidatedBytes(...)`
- `Decoded(Value)` or another materialized scalar/object state
- `Null`

The exact Rust types can differ, but the semantics matter:

- construction can finish with every slot validated
- later access can materialize without re-validating raw bytes
- malformed unused payloads still fail at row-open time

- retained-slot narrowing changed what work happens

- cached required-slot layout changed where the slot set is owned

  packaged

So the decision rule for the next executor slice should stay blunt:

- keep changes that remove work
- keep cleanup that makes one shared contract more explicit, even if it does
  not move perf much
- reject representation churn when it does not change the measured work mix

## Required New Seams

The reader split depends on field-level “validate only” APIs.

Today many helpers combine:

- payload/envelope correctness checking
- runtime object construction

The refactor should introduce parallel seams for non-trivial storage lanes:

- validate `ByKind` field bytes without building the final runtime object
- materialize `ByKind` field bytes from validated metadata
- validate value-storage bytes without recursively rebuilding `Value`
- materialize value-storage bytes from validated metadata

The exact names may differ, but the split must be explicit in ownership.

Without that split, the constructor cannot keep the `0.65.1` hard-cut while
also avoiding unconditional recursive decode.

## Lower-Risk Wins First

Before structural-reader surgery, there are two narrower wins that fit the same
design line and have smaller blast radii.

### 1. Stream Raw Index Entry Membership

First slice:

- add visitor or iterator-style membership decode on `RawIndexEntry`
- keep frame validation explicit
- let scan callers stream `StorageKey` values directly instead of first
  allocating `Vec<StorageKey>`

Good shape:

- `visit_keys(...)`
- or `iter_keys(...)`
- keep `decode_single_key()` as the common one-key fast path

Why first:

- narrow blast radius
- no canonical-row contract change
- immediately useful for secondary scan hot loops

### 2. Add SQL-Only Covering Cell Rendering

Second slice:

- keep the existing `Value` covering decode for generic executor consumers
- add a SQL-only direct cell path for terminal projection
- use it only when admitted covering component kinds can be rendered directly

This should stay executor-owned and fail closed. It is not a planner change.

## Recommended Rollout Order

The current best sequence is:

1. Index entry streaming decode
   - borrowed or visitor-style membership decode
   - switch secondary scan hot loops to stream keys

2. Covering direct cell path
   - keep the generic `Value` path
   - add SQL-only direct scalar rendering in terminal projection

3. Structural reader split
   - keep `from_raw_row(...)` strict
   - validate every declared slot at construction
   - materialize slot values lazily on demand
   - preserve and extend the `0.65.1` regressions

4. Value-storage and writer follow-up
   - reduce `FieldStorageDecode::Value` usage where possible
   - replace known-shape write-side CBOR AST staging only after the read-side
     work is stable

## Success Metrics

For this line, the important measurements are not just total instructions.

Track:

- rows opened
- slots validated per row
- slots materially decoded per row
- `FieldStorageDecode::Value` materializations per query
- index-entry keys decoded per scan
- allocated membership vectors count and bytes
- covering components decoded to `Value` versus direct cell render
- row-presence checks and key re-encode counts

The decisive metric for the structural-reader split is:

> average materialized slots per row after validation

If that remains close to “all declared fields,” the complexity is not paying
for itself.

## Guardrails

### Keep

- the `0.65.1` row-open fail-closed contract
- one strict shared structural read boundary
- executor-owned covering output shaping
- narrow, explicit fast paths with fail-closed fallback

### Do Not Do

- do not reintroduce latent malformed unused payloads
- do not weaken `StructuralSlotReader::from_raw_row(...)` into a best-effort
  accessor
- do not blur validation and materialization again under a new cache type
- do not start with deep `FieldStorageDecode::Value` rewrites unless the
  earlier slices prove that is the remaining dominant cost
- do not treat write-side `ByKind` staging cleanup as the first read-side win

## Current Recommendation

Given the current tree, the highest-confidence next move is:

1. implement index-entry streaming decode
2. then add the SQL-only covering direct render path
3. then start the structural reader split under the explicit rule:
   strict boundary validation plus lazy semantic materialization

That preserves the canonical-row hard-cut from `0.65.1` while targeting the
largest remaining materialization tax on real read paths.
