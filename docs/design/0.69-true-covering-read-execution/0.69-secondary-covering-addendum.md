# 0.69 Secondary Covering Addendum

## Purpose

This addendum captures the first secondary-covering execution cleanup after
`0.69.0` landed. The goal is to preserve the architectural lessons from that
slice so later widening work does not trade a small perf gain for long-term
boundary debt.

It is intentionally narrower than the main `0.69` design doc. This note is
about one specific question:

> how should `row_check_required` secondary covering execution be optimized
> without weakening correctness or moving execution policy into the index
> layer?

## What Improved

The current secondary covering fast path is materially better than the first
attempt because it now keeps the ownership boundary explicit:

- the index layer streams `(storage_key, component)` pairs only
- the executor owns row-presence checks
- the executor owns decoded value mapping
- the SQL terminal still owns output-shape specifics

That means the runtime is still fused for performance, but policy authority has
not drifted downward into `db::index`.

The important local optimization is also now structurally honest:

- no intermediate `(DataKey, component)` vector for the narrow
  single-component secondary covering cohort
- no fake `planner_proven` widening
- no hidden weakening of `row_check_required`
- no raw-key plumbing leaking upward as a new cross-layer contract

## Current Architectural Assessment

### Good

- boundary discipline is correct again:
  - index scan emits
  - executor validates
  - terminal renders
- the narrow `SingleComponentCoveringCollector` contract is the right level of
  abstraction for this cohort
- `row_check_required` remains visible and explicit
- the optimization is localized, measurable, and fail-closed

### Not Yet Changed

- secondary covering is still not `planner_proven`
- stale secondary-key risk still forces row-visibility checks
- the dominant cost is still inside the execute bucket, especially:
  - row presence checks
  - secondary decode
  - per-row runtime plumbing

## Performance Readout

The exact measured SQL cohort for this slice was:

```sql
SELECT id, name FROM User ORDER BY name ASC, id ASC LIMIT 2
SELECT id, name FROM User ORDER BY name DESC, id DESC LIMIT 2
```

Net movement from the original secondary-covering baseline:

- ascending generated: `675,563 -> 662,192` (`-1.98%`)
- ascending typed: `672,342 -> 659,878` (`-1.85%`)
- descending generated: `680,802 -> 666,439` (`-2.11%`)
- descending typed: `677,387 -> 664,659` (`-1.88%`)

That is not a breakthrough-class win, but it is a legitimate execute-path
improvement with stable generated-vs-typed and asc-vs-desc behavior.

The right conclusion is:

- the intermediate materialization boundary was worth removing
- the remaining large cost is not in that boundary anymore
- larger future gains require a better answer to secondary row-presence cost,
  not more polishing of the same staging path

## Expression Key-Only Investigation

The later `0.69` expression-backed key-only covering work answered one
important negative question:

> the slower `SELECT id, tier ... STARTS_WITH(LOWER(handle), 'br') ...`
> witness-backed route is not slower because it lost route-level bounded access
> or fast-path eligibility.

The owner-local session tests now lock that the key-only prefix cohort and the
older fuller sibling:

```sql
SELECT tier, handle
FROM FilteredIndexedSessionSqlEntity
WHERE active = true AND tier = 'gold'
  AND STARTS_WITH(LOWER(handle), 'BR')
ORDER BY LOWER(handle) ASC, id ASC
LIMIT 2
```

and

```sql
SELECT id, tier
FROM FilteredIndexedSessionSqlEntity
WHERE active = true AND tier = 'gold'
  AND STARTS_WITH(LOWER(handle), 'BR')
ORDER BY LOWER(handle) ASC, id ASC
LIMIT 2
```

keep parity on:

- `IndexRangeScan` root shape
- equality-prefix metadata
- fast-path label and reason
- `IndexRangeLimitPushdown` / `TopNSeek` presence
- `OrderByAccessSatisfied` presence
- traced optimization label
- traced scanned-key count

That means the remaining gap on the expression key-only cohort is not a route
planner/hint issue. It is deeper in the runtime path that materializes the
key-only covering result after traversal is already fixed.

One attempted local optimization specialized the `pk + constant` projected-row
path for this cohort. It moved PocketIC generated dispatch from roughly
`1,326,224` to `1,325,601` local instructions on:

```sql
SELECT id, tier
FROM ActiveUser
WHERE active = true AND tier = 'gold'
  AND STARTS_WITH(LOWER(handle), 'br')
ORDER BY LOWER(handle) ASC, id ASC
LIMIT 2
```

That is effectively noise, so it was not kept as a shipped optimization.

The working conclusion is:

- route parity is already good enough
- the next real expression-key-only gain will not come from more route/hint
  tweaking
- it will need a deeper expression-covering execution shape, likely below the
  current `pk + constant` projected-row materializer

One later experiment tried the next obvious deeper lever: a storage-key-only
raw-range scan path for the constant-projection expression cohort. It was kept
fully centralized under the existing index scan + executor covering
boundaries, measured on the exact PocketIC SQL probe below, and then backed
out:

```sql
SELECT id, tier
FROM ActiveUser
WHERE active = true AND tier = 'gold'
  AND STARTS_WITH(LOWER(handle), 'br')
ORDER BY LOWER(handle) ASC, id ASC
LIMIT 2
```

Measured outcome:

- the generated probe stayed effectively flat: `1,326,224 -> 1,326,219`
- the order-only key-only sibling moved into a worse band during the
  experiment, so the branch reverted to the prior baseline before continuing

That result matters because it rules out one tempting next move:

- a storage-key-only scan bypass is not, by itself, the missing execute-path
  win for expression key-only prefix/range cohorts
- the next deeper attempt should target the remaining runtime materialization
  cost directly, not another alternative traversal wrapper around the same
  bounded scan

## Guardrails For The Next Slice

### Keep

- keep `row_check_required` fail-closed until planner-owned row-authority proof
  exists
- keep the optimization explicitly narrow:
  - single-component
  - covering
  - order-aligned
- keep executor-owned presence checks outside the index layer
- keep capability or fallback signaling explicit in types, not implicit in
  `Option` control flow

### Do Not Do

- do not weaken stale-row safety
- do not move data-store visibility checks into generic index helpers
- do not generalize this path into a broad programmable scan engine
- do not invent a fake secondary `planner_proven` cohort just to force a bigger
  perf result

## Best Next Directions

If `0.69.1+` keeps attacking secondary covering, the best candidates are:

1. Reduce the cost of `row_check_required` itself without weakening semantics.
   - Examples:
   - batch or amortized presence checks
   - co-located existence probes if the storage contract can support them
   - planner-owned hints that remain fail-closed

2. Reduce decode cost further.
   - Examples:
   - partial component extraction when full decoded-key materialization is not
     required
   - narrower decode paths for known single-component layouts

3. Tighten continuation and scan-budget behavior only when semantics remain
   identical.
   - This slice already showed how easy it is to get stale-row budget semantics
     wrong.

## Working Rule

Use this rule when evaluating future secondary covering changes:

> Specialize execution at the executor boundary, not inside the storage/index
> layer, unless the new index-side contract stays narrow enough that the index
> layer still only emits entries and never owns row-visibility or projection
> policy.

If a future optimization breaks that rule, it should not ship as part of
`0.69`.
