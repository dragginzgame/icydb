## Shipped In `0.69.0`

- [x] S0. Execution-grade covering route contract
  - Introduced planner-owned `CoveringReadExecutionPlan`.
  - Added explicit `CoveringExistingRowMode`.
  - Route and runtime now consume execution-grade covering contracts instead of
    projection-only covering metadata.

- [x] S1. Explain visibility
  - `EXPLAIN EXECUTION` and `EXPLAIN JSON` now surface
    `existing_row_mode`.
  - The shipped route is now observable as an execution contract instead of a
    hidden terminal shortcut.

- [x] S2. First planner-proven cohort
  - PK-only projection over authoritative primary-store `FullScan`.
  - PK-only projection over strict half-open primary-key `KeyRange`.
  - These shapes may skip row-existence checks because the planner can already
    prove row authority from the chosen access path.

- [x] S3. Widened exact PK lookup cohort
  - PK-only `ByKey`.
  - Canonical ascending PK-only `ByKeys`.
  - These shapes now compile through first-class `CoveringRead`, but they stay
    explicit `row_check_required`.

- [x] S4. Outer-surface coverage
  - Core planner, route, session, and executor snapshot coverage are in place.
  - Demo RPG typed/generated coverage is in place.
  - PocketIC query/explain/perf coverage is in place for the first PK-only
    cohort.

- [x] S5. First explicit secondary witness cohorts
  - Added a route-visible `witness_validated` authority mode for narrow
    secondary covering reads.
  - Store/index witness state is now synchronized and invalidated explicitly
    during commit and recovery.
  - The first shipped cohorts are single-field `Customer.name` covering reads
    and the first filtered composite `CustomerAccount(tier, handle)`
    order-only read.

## Current Admitted Cohorts

### `planner_proven`

- PK-only projection over primary-store `FullScan`
- PK-only projection over strict half-open primary-key `KeyRange`

### `witness_validated`

- `SELECT id, name FROM User ORDER BY name ASC, id ASC LIMIT 2`
- `SELECT id, name FROM User ORDER BY name DESC, id DESC LIMIT 2`
- `SELECT id, name FROM User WHERE name = ... ORDER BY id LIMIT 1`
- `SELECT id, name FROM User WHERE name = ... ORDER BY id DESC LIMIT 1`
- `SELECT id, name FROM User WHERE name >= ... AND name < ... ORDER BY name ASC, id ASC LIMIT 2`
- `SELECT id, name FROM User WHERE name >= ... AND name < ... ORDER BY name DESC, id DESC LIMIT 2`
- `SELECT id, tier, handle FROM ActiveUser WHERE active = true AND tier = ... ORDER BY handle ASC, id ASC LIMIT 2`
  - current repo names:
    - `User` => `Customer`
    - `ActiveUser` => `CustomerAccount`

### `storage_existence_witness`

- stale single-field secondary order-only covering:
  - `SELECT name FROM Customer ORDER BY name ASC LIMIT 2`
- stale first-component-plus-PK secondary order-only covering:
  - `SELECT id, name FROM Customer ORDER BY name ASC, id ASC LIMIT 2`
- stale two-component secondary order-only covering:
  - `SELECT id, priority, status FROM CustomerOrder ORDER BY priority ASC, status ASC, id ASC LIMIT 2`
- stale descending two-component secondary order-only covering:
  - `SELECT id, priority, status FROM CustomerOrder ORDER BY priority DESC, status DESC, id DESC LIMIT 2`
- stale composite first-component-plus-PK secondary order-only covering:
  - `SELECT id, priority FROM CustomerOrder ORDER BY priority ASC, status ASC, id ASC LIMIT 2`
- stale descending composite first-component-plus-PK secondary order-only covering:
  - `SELECT id, priority FROM CustomerOrder ORDER BY priority DESC, status DESC, id DESC LIMIT 2`
- stale equality-prefix suffix-order composite covering:
  - `SELECT id, priority, status FROM CustomerOrder WHERE priority = 20 ORDER BY status ASC, id ASC LIMIT 2`
- all are admitted only on the explicit stale secondary witness path after
  the leading base row is removed while the secondary `name` entry or
  `(priority, status)` entry remains stale

### `row_check_required`

- broader secondary/index-backed covering routes carried forward from `0.68`
- PK-only `ByKey`
- canonical ascending PK-only `ByKeys`

## Explicit Fail-Closed Boundaries

- `ByKeys` plus descending PK order is still not admitted
- broader secondary/index covering routes are not planner-proven yet
- filtered composite descending `ORDER BY handle DESC, id DESC` still fails
  closed to the materialized route
- non-PK primary-store projections are outside the first honest cohort
- grouped, aggregate, and broad DISTINCT execution are still outside `0.69`

## What `0.69.0` Proved

- covering-read admission can now be a real execution route instead of planner
  metadata only
- row-presence semantics can stay explicit at the planner boundary instead of
  being implied by runtime shortcuts
- the first planner-proven scalar cohort is narrow but already measurable
- explicit secondary authority can now be shipped without silently reinterpreting
  all secondary covering reads as planner-proven
- the same witness contract can widen across adjacent cohorts while staying
  centralized in one route classifier
- widening can keep following the same rule: admit only what the planner can
  state explicitly, fail closed otherwise
