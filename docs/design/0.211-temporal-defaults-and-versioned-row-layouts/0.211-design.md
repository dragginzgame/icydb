# 0.211 Temporal Defaults And Versioned Row Layouts

Status: proposed

Design cut: 2026-07-17

Implementation order: after 0.210 closeout

## Decision Summary

IcyDB will split the current field `default` fact into two accepted-schema
facts with different temporal authority:

- an **insert default** determines the value of an omitted field in a new
  logical after-image; and
- a **historical fill** determines the frozen logical value of a field that
  did not exist in the physical layout of an older row.

Changing or dropping an insert default affects future writes only. It never
changes the logical value of an already persisted row. A field added over
existing rows freezes its historical fill at publication, even if its insert
default later changes or is removed.

The split alone cannot prove whether a short row is legitimately old or has
been truncated. Therefore every canonical row will also carry a bounded
accepted row-layout version. Accepted schema records the current row-layout
version, the oldest admitted layout version, and the layout version in which
each field was introduced. A missing trailing slot is legal only when the row
truthfully predates that field and the accepted field owns a historical fill.
A row whose layout version says the field should be present but whose payload
omits it is corruption.

Every current write emits a complete current-layout row. Typed creates, SQL
inserts, structural writes, replacements, batches, imports, and future SQL
`DEFAULT` syntax all lower to one accepted-schema-owned insertion policy.
Generated Rust expressions may propose an insert default and may implement an
explicit Rust `Default`, but runtime database writes never evaluate generated
Rust defaults as authority.

The initial SQL surface added by this line is:

~~~sql
INSERT INTO Token (id, tier) VALUES (DEFAULT, DEFAULT);
INSERT INTO Token DEFAULT VALUES;
UPDATE Token SET tier = DEFAULT WHERE id = '...';

ALTER TABLE Token ALTER COLUMN tier SET DEFAULT 'bronze';
ALTER TABLE Token ALTER COLUMN tier DROP DEFAULT;
~~~

`DEFAULT` is an explicit request to apply the accepted insertion policy. It is
not an untyped value, a callback, or a request to reevaluate historical rows.

### Non-Negotiable Invariants

1. Accepted schema is the sole runtime authority for insert defaults,
   historical fills, layout identity, nullability, generation, and management.
2. Insert omission and historical physical absence are different operations
   with different owner types and helpers.
3. `SET DEFAULT` and `DROP DEFAULT` affect future logical after-images only.
4. A historical fill is frozen when an additive field is published over
   existing rows and cannot follow later insert-default changes.
5. Every row carries one bounded row-layout version admitted by the current
   accepted schema.
6. A row has exactly the physical slot count declared by its stamped layout
   version; shorter or longer payloads are corruption.
7. Historical fill is legal only for fields introduced after the row's layout
   version.
8. Every current insert, replacement, update result, and rewrite emits a full
   row stamped with the current row-layout version.
9. Explicit `NULL`, omission, `DEFAULT`, insert generation, and engine-managed
   values remain distinct write intents.
10. Insert-default and historical-fill payloads are bounded canonical slot
    payloads admitted by the exact field contract and accepted enum catalog.
11. Defaults are deterministic data. Current time, randomness, callbacks,
    canister calls, environment reads, and arbitrary Rust function execution
    are not database defaults.
12. Recovery replays already admitted bytes and never reevaluates a default,
    generator, or historical fill.
13. Imports and restores cannot obtain values merely by presenting a malformed
    short current-layout row.
14. Persisted row and schema decoding is bounded, fallible, and current-form
    only.
15. Before `1.0.0`, the old single-default schema form and unstamped row
    envelope are hard-cut without aliases, inference, or compatibility decode.

## Normative Default Contract

The maintained contract is:

> Every field has one exact accepted kind, one current insertion policy, and
> one explicit historical-absence policy. Omission in a new after-image is
> resolved from current accepted schema. Physical absence in an older row is
> resolved only from the row's admitted layout identity and the field's frozen
> historical fill. Changing the former never changes the latter.

The temporal rule is:

> A database default is a rule for constructing a future after-image, not a
> live expression over stored rows. When an additive schema change must give
> preexisting rows a logical value without rewriting them, that value becomes
> separate immutable historical schema data.

The negative contract is:

> IcyDB does not infer historical values from the current insert default, Rust
> `Default`, generated model code, physical slot count alone, or application
> callbacks. A row that cannot prove a legal historical layout fails closed.

## Audited Baseline

The source audit on 2026-07-17 found a strong exact-default foundation with one
important temporal conflation:

| Concern | Current owner | Current behavior |
|---|---|---|
| Generated default proposal | `FieldDatabaseDefault` in `crates/icydb-core/src/model/field.rs` | Distinguishes no default, canonical slot payload, and proposal-only authored enum unit |
| Accepted field metadata | `PersistedFieldSnapshot::default` and `SchemaFieldDefault` | Stores one `None` or canonical slot payload fact |
| Accepted wire | `SchemaFieldDefaultWire` in `db/schema/codec.rs` | Persists the same single default fact |
| Insert omission | `AcceptedFieldAbsencePolicy` | Derives `Required`, `NullIfMissing`, or `DefaultIfMissing` from nullability and the single default |
| Historical short row | `StructuralRowContract::missing_slot_value` | Uses the same absence policy and current default to synthesize every missing trailing accepted slot |
| Structural insert/replace | persisted-row patch serialization | Uses `missing_slot_payload` to complete omitted slots |
| Typed create | generated `<Entity>_Create` plus save preflight | Materializes a Rust value from generated expressions, then separately proves omitted slots are accepted |
| SQL insert | SQL write executor | Leaves omitted accepted slots sparse; complete structural serialization later applies the same missing-slot helper |
| Explicit SQL `DEFAULT` | SQL parser/executor | Not part of the maintained DML subset |
| DDL default change | schema field mutation | Mutates the single default, scans no rows, rejects some indexed fields, and currently rejects dropping the only default from a non-nullable field |
| Add-column history | additive DDL | Physically short old rows are read through nullable or default-based missing-slot materialization |
| Row envelope | `db/codec/mod.rs` | Carries magic, format version, and payload length, but no accepted row-layout identity |

## Problem Statement

### 1. One Default Currently Owns Two Different Times

The current metadata means both:

1. what a future insert receives when it omits the field; and
2. what an old physical row receives when it predates the field.

Those facts may begin with the same bytes during `ADD COLUMN ... DEFAULT`, but
they do not remain the same semantic fact. The first is mutable DDL policy. The
second is historical data authority.

### 2. Changing A Default Can Change Untouched Historical Rows

Consider:

~~~sql
ALTER TABLE Token ADD COLUMN tier TEXT NOT NULL DEFAULT 'bronze';
ALTER TABLE Token ALTER COLUMN tier SET DEFAULT 'silver';
~~~

If old rows remain physically short and reads consult the current single
default, an untouched old row changes logically from `bronze` to `silver`.
Its indexes, checks, relation projections, query results, and integrity claim
can also change without a row mutation. That violates the normal database
meaning of `SET DEFAULT` and makes metadata mutation rewrite data implicitly.

### 3. Physical Slot Count Alone Cannot Prove Legitimate History

An old row and a truncated current row can have the same short payload. The
current slot count says how many values are present, not which accepted layout
created the row. A strict integrity checker cannot classify the difference
without persisted layout identity.

### 4. Initial-Field Absence Can Be Misclassified As Omission

A nullable or defaulted initial field may currently materialize when its slot
is physically absent, even though every row layout that ever existed required
that slot. That can hide truncation behind valid null/default semantics.
Insert omission and persisted-row corruption must not share an escape hatch.

### 5. Typed Create Materializes Before Accepted Authority

Generated create builders currently use generated Rust expressions to build a
concrete entity and then use accepted schema to decide whether omitted fields
were legal. Reconciliation protects much of this boundary, but it is still a
dual materialization shape. The accepted payload should construct the database
after-image directly; generated Rust code should not be a parallel runtime
default engine.

### 6. Explicit Default Intent Is Missing From SQL

Omitting an insert column can use a default, but SQL cannot currently say
`DEFAULT` for one value, use `DEFAULT VALUES`, or reset one field with
`UPDATE ... SET field = DEFAULT`. Adding these forms without a shared intent
owner would create another default implementation.

### 7. Current DDL Restrictions Follow From The Conflation

Changing a default used by an indexed historical field can change the logical
index key of short rows, so the current implementation conservatively blocks
some changes. Dropping a default from a non-nullable field is also blocked even
though the coherent future behavior is simply to require that field on future
inserts. Splitting the roles removes both artificial restrictions.

## Goals

0.211 must:

1. split accepted insert defaults from frozen historical fills;
2. persist exact row-layout identity in every canonical row;
3. separate `SchemaVersion` from physical `RowLayoutVersion`;
4. make row physical slot count exact for its stamped layout version;
5. make initial-field physical absence fail as corruption;
6. freeze the logical value of additive fields for preexisting rows;
7. make `SET DEFAULT` and `DROP DEFAULT` future-only metadata changes;
8. permit a non-nullable field to have no insert default, making omission a
   typed admission error;
9. retain exact explicit-`NULL` semantics;
10. route every insertion/replacement ingress through one accepted insertion
    policy;
11. make typed creation accepted-first rather than generated-expression-first;
12. add explicit SQL `DEFAULT` forms over the same policy;
13. preserve separate insert-generation and engine-management owners;
14. define DDL, rewrite, import, restore, recovery, constraint, index,
    relation, and integrity interactions;
15. provide bounded machine-readable introspection for both temporal facts;
16. hard-cut old accepted-schema and row-envelope forms; and
17. close with correctness, corruption, DDL atomicity, recovery, performance,
    row/catalog bytes, raw wasm, and source-complexity evidence.

## Non-Goals

0.211 does not:

- add arbitrary default expressions, callbacks, Rust function paths, current
  time, randomness, canister calls, subqueries, or cross-row reads;
- merge database defaults with insert generation or engine-managed fields;
- make Rust's `Default` trait implicit on domain entities;
- add a per-table strictness mode;
- add a general schema backfill language;
- make physical lazy fill planner-visible as a stored value;
- silently repair old or malformed rows;
- add general computed columns;
- add update-time generation for ordinary fields;
- allow a default to bypass accepted kind, bounds, enum, check, unique, or
  relation admission;
- preserve old row-envelope or single-default schema decoders;
- add a legacy migration shim for development databases;
- introduce a mutable SQL pragma for default semantics; or
- design post-`1.0.0` edition compatibility.

## Vocabulary

### Authored Value

A value explicitly supplied by a caller, including an explicitly supplied
`NULL`. It is validated and canonicalized under the accepted field contract.

### Omission

The absence of a field assignment while constructing a new logical
after-image. Omission is an instruction to consult current insertion policy;
it is not a persisted value.

### Explicit Default Intent

An explicit request such as SQL `DEFAULT` to use current insertion policy.
For insert and replacement it is semantically equivalent to omission while
remaining distinguishable in parsing and diagnostics.

### Insert Default

A deterministic canonical accepted slot payload used for an omitted ordinary
field in a future after-image. It may be changed or removed by admitted schema
DDL.

### Insert Generation

An engine operation such as ULID or insertion timestamp generation. It is not
an insert default because it produces a new value per admitted write.

### Engine-Managed Value

A value whose authorship and update timing belong to the engine, such as a
managed timestamp. It is not a default.

### Historical Physical Absence

A slot absent from a row because the row was emitted under an admitted layout
that predates the field's introduction.

### Historical Fill

The accepted frozen logical value used for historical physical absence. It is
either `NULL` or one canonical slot payload. `Reject` means absence is not
legal.

### Row Layout Version

A monotonically increasing entity-local identity for physical row slot shape.
It changes only when the admitted physical layout changes, not for every schema
mutation.

### Schema Version

The existing monotonically increasing identity for the complete accepted
schema snapshot. Default changes bump schema version but do not by themselves
bump row-layout version.

### Complete Current Row

A canonical row containing every physical slot required by the current
accepted layout and stamped with the current row-layout version.

## Temporal Semantics

The required behavior is easiest to see as one history:

| Event | Existing row `A` | New omitted row `B` | Accepted insert policy | Frozen historical fill |
|---|---|---|---|---|
| Before field exists | no field | no field | none | none |
| `ADD tier NOT NULL DEFAULT 'bronze'` | logically `bronze`, physically absent | `bronze`, physically present | `bronze` | `bronze` |
| `SET DEFAULT 'silver'` | still `bronze` | `silver` | `silver` | `bronze` |
| `DROP DEFAULT` | still `bronze` | omission rejects | required | `bronze` |
| Explicit insert `tier='gold'` | still `bronze` | `gold` | required | `bronze` |
| Update row `A` | after-image contains physical `bronze` | unchanged | required | still retained until proven unnecessary |

For a nullable additive field without a default:

| Event | Existing old row | New omitted row | Explicit `NULL` |
|---|---|---|---|
| Add nullable field | historical fill is `NULL` | insertion policy produces `NULL` | stored as `NULL` |
| Later set default `x` | old row remains `NULL` | omission produces `x` | remains explicit `NULL` |
| Later drop default | old row remains `NULL` | omission produces `NULL` | remains explicit `NULL` |

No metadata-only default operation changes an already admitted logical row.

## Accepted Authority Model

Conceptually, the accepted field and row layout become:

~~~text
struct PersistedFieldSnapshot {
    id: FieldId,
    name: String,
    slot: SchemaFieldSlot,
    kind: AcceptedFieldKind,
    nested_leaves: Vec<...>,
    nullable: bool,
    introduced_in_layout: RowLayoutVersion,
    insert_default: SchemaInsertDefault,
    historical_fill: SchemaHistoricalFill,
    write_policy: SchemaFieldWritePolicy,
    origin: PersistedFieldOrigin,
    storage_decode: ...,
    leaf_codec: ...,
}

enum SchemaInsertDefault {
    None,
    SlotPayload(Vec<u8>),
}

enum SchemaHistoricalFill {
    Reject,
    Null,
    SlotPayload(Vec<u8>),
}

struct SchemaRowLayout {
    current_version: RowLayoutVersion,
    history_floor: RowLayoutVersion,
    field_to_slot: Vec<(FieldId, SchemaFieldSlot)>,
}
~~~

The names are normative in meaning, not necessarily final Rust spelling.
`SchemaFieldDefault` must not remain as an ambiguous shared owner.

### Why Historical Fill Is A Separate Payload

An additive field with a default initially stores the same canonical bytes in
both roles. They intentionally become independent facts. Direct duplication is
preferred over a semantic reference from historical fill to mutable insert
default. An encoding-level deduplication may be added only if catalog byte
measurements justify it and the decoded accepted model still exposes two
independent immutable values.

### Field Closure Rules

Accepted-schema construction rejects a field unless all of these hold:

1. `introduced_in_layout` is between the row-layout history floor and current
   version, inclusive, or is an older retained introduction version that is
   already below a collapsed history floor;
2. field slots are dense and introduction versions are nondecreasing by slot
   while historical layouts remain admitted;
3. an insert-default payload decodes canonically under the exact field kind;
4. a historical-fill payload decodes canonically under the exact field kind;
5. `SchemaHistoricalFill::Null` is legal only for a nullable field;
6. a payload cannot encode `NULL`; nullable historical null uses the explicit
   `Null` variant;
7. historical `Reject` is mandatory when every admitted layout includes the
   field;
8. generation and engine management do not masquerade as either payload;
9. primary-key fields have historical `Reject` and cannot be nullable; and
10. enum, text, blob, decimal, collection, composite, and recursive bounds are
    checked through accepted catalog authority.

If a field kind or bound changes in a way that would invalidate either
payload, the mutation rejects unless an exact marker-owned rewrite removes the
old dependency and publishes a valid current contract atomically.

### Row-Layout Closure Rules

The accepted layout enforces:

- nonzero bounded `current_version` and `history_floor`;
- `history_floor <= current_version`;
- one dense active slot mapping;
- append-only slots while `history_floor < current_version` and older layouts
  remain admitted;
- an exact expected physical slot count for every admitted version; and
- no reuse of an old row-layout version for a different shape.

For the V1 additive history model, the expected slot count for layout version
`v` is the number of current fields whose introduction version is at or before
`v`. Initial fields share the initial layout version. A full exact rewrite may
advance `history_floor` to the new current version, after which older layout
versions fail closed and dense slot reassignment is legal.

## Persisted Row Envelope

The current row envelope carries only:

~~~text
magic | row_format_version | payload_len | slot_container
~~~

0.211 hard-cuts it to:

~~~text
magic
| row_format_version
| row_layout_version: u32
| payload_len: u32
| slot_container
~~~

The row format version changes to the new current form. The exact endianness
and bounds remain codec-owned, but layout version is fixed-width and decoded
before slot allocation or traversal.

The additional four bytes per row buy a proof unavailable from slot count:

- which accepted physical shape emitted the row;
- whether a missing trailing field is legitimate history;
- whether a supposedly current row was truncated;
- whether an import is trying to exploit historical fill; and
- whether a rewrite actually advanced every touched row.

A full schema fingerprint is deliberately not stored per row. It would add far
more bytes and would make rows appear incompatible after metadata-only changes
such as `SET DEFAULT`. Row layout identity is the minimal temporal fact needed
for physical interpretation.

### Row Decode Algorithm

For one row and current accepted schema:

1. validate the bounded row envelope and current row-format version;
2. decode the row-layout version without allocation;
3. require `history_floor <= row_layout_version <= current_version`;
4. derive the exact physical slot count for that admitted layout version;
5. require the slot container to contain exactly that count;
6. decode every physically present slot through its exact accepted contract;
7. for each current field introduced after the row-layout version, use only
   that field's accepted historical fill;
8. reject if the historical fill is `Reject` or invalid;
9. evaluate keys, indexes, relations, checks, and projection over the resulting
   logical row; and
10. never mutate or repair the row as a side effect of reading it.

There is no branch that consults the current insert default during row decode.

### Row Encode Algorithm

Every current row emitter:

1. constructs a complete logical after-image under accepted insertion/update
   semantics;
2. validates the final after-image;
3. serializes every current physical slot canonically;
4. writes the exact current slot count; and
5. stamps the current accepted row-layout version.

An update of an old row therefore performs a natural physical backfill for
that row. The unchanged historical logical values are decoded from frozen fill
before the complete current after-image is emitted.

## Insertion Policy

The schema runtime owner derives a current insertion policy per field:

~~~text
enum AcceptedInsertOmissionPolicy {
    Required,
    Null,
    DefaultPayload,
    Generate(InsertGeneration),
    Managed(WriteManagement),
}
~~~

The exact internal shape may keep generation and management in their existing
owners. The important rule is one resolver with this precedence:

1. an authored value is validated as authored;
2. an explicitly authored `NULL` is accepted only when nullable;
3. engine-managed fields reject caller authorship and use management policy;
4. omitted insert-generated fields invoke the accepted generator;
5. an ordinary omitted field with an insert default uses its canonical
   payload;
6. an ordinary omitted nullable field without a default becomes `NULL`; and
7. every other omission rejects as a missing required field.

Generation cannot be represented as `SlotPayload` because it is evaluated per
write. Historical fill cannot invoke generation because historical rows must
have a stable logical value.

### Write Intent Model

All frontends lower to an internal intent equivalent to:

~~~text
enum FieldWriteIntent {
    Authored(Value),
    UseInsertPolicy,
    PreserveExisting,
}
~~~

`Value::Null` remains an authored value. Insert omission and insert SQL
`DEFAULT` lower to `UseInsertPolicy`. Update omission lowers to
`PreserveExisting`. `UPDATE ... SET field = DEFAULT` lowers to
`UseInsertPolicy` only for ordinary authorable fields.

This intent is transient. It is never persisted in a row.

### Insert, Replace, And Update

- **Insert** resolves every absent field through current insertion policy.
- **Replace** constructs a new complete row and resolves every absent field
  through current insertion policy.
- **Update** preserves fields without assignments. An explicit default intent
  resolves only the assigned field through current insertion policy.
- **Upsert** applies insert semantics on its insert branch and update semantics
  on its conflict-update branch.
- **Batch and bulk operations** use the same branch semantics for every
  operation before the shared commit marker.

Historical fill is never a write intent. It participates only while decoding
an already persisted older row into the before-image.

## Typed Creation Becomes Accepted-First

Generated `<Entity>_Create` remains an ergonomic authored-input carrier, but
it must stop being a second database-default evaluator.

The target flow is:

~~~text
generated create input
    -> authored sparse field intents + provenance
    -> accepted insertion-policy resolution
    -> complete canonical accepted slot image
    -> typed entity materialization where required
    -> sanitizer / application validator
    -> canonical accepted after-image
    -> accepted checks / indexes / relations / commit plan
~~~

Generated `default = ...` syntax may serve two explicit compile-time roles:

- propose one database insert default that reconciliation lowers to canonical
  accepted bytes; and
- help implement a whole-entity Rust `Default` only when the entity explicitly
  opts into `traits(add(Default))`.

Those roles share source syntax but not runtime authority. Database creation
reads accepted bytes. Calling `Entity::default()` executes Rust semantics and
produces an ordinary fully authored Rust value if later inserted.

The generated create DTO may still derive Rust `Default`; that means “no input
fields authored,” not “materialize the database row from Rust defaults.” Its
omitted fields remain unresolved until the accepted insertion owner runs.

### Sanitizer Boundary

Sanitizers continue to run on the constructed typed after-image according to
the maintained write contract. If sanitization changes a defaulted value, the
stored value and every derived effect use the sanitized canonical result. The
accepted insert-default payload itself is not mutated by sanitization.

Application validators remain application code, not persisted database
constraints. Accepted 0.209 checks evaluate the final canonical after-image
after default resolution, generation, management, and sanitization.

## SQL DML Surface

0.211 adds these forms to the fail-closed SQL subset:

~~~sql
INSERT INTO Token (id, tier) VALUES (DEFAULT, DEFAULT);
INSERT INTO Token DEFAULT VALUES;
UPDATE Token SET tier = DEFAULT WHERE id = '...';
~~~

### Insert `DEFAULT`

For insert and replacement, `DEFAULT` means `UseInsertPolicy`:

- an ordinary insert default is used when present;
- a nullable ordinary field without a default becomes `NULL`;
- an insert-generated field invokes its accepted generator;
- an engine-managed field invokes its accepted insertion management; and
- a required ordinary field without a default rejects.

The parser does not lower `DEFAULT` to `NULL` or a literal. It remains an
explicit AST/write-intent variant until accepted insertion resolution.

### `DEFAULT VALUES`

`DEFAULT VALUES` applies `UseInsertPolicy` to every field. It succeeds only if
every field can be completed through a default, nullability, generation, or
management. A missing required ordinary field rejects the whole statement
before commit planning.

### Update `SET field = DEFAULT`

For an ordinary authorable field:

- use its current insert default if present;
- otherwise use `NULL` if nullable; and
- otherwise reject as missing required default semantics.

Insert-only generators do not rerun during update. Engine-managed fields remain
non-authorable. Therefore `SET generated_or_managed_field = DEFAULT` rejects
unless a later design explicitly gives that write policy update-default
semantics.

SQL expressions cannot embed `DEFAULT` inside arithmetic, function arguments,
predicates, collections, composites, or checks. It is legal only in the
explicit assignment/value positions above.

## SQL DDL Semantics

### `ADD COLUMN`

The additive matrix is:

| Field declaration | Existing rows present | Insert default | Historical fill | Future omission |
|---|---:|---|---|---|
| nullable, no default | yes | none | `Null` | `NULL` |
| nullable, default `v` | yes | `v` | frozen `v` | current default |
| not null, default `v` | yes | `v` | frozen `v` | current default |
| not null, no default | yes | none | impossible | reject DDL |
| any exact field | entity proven empty | declared/default policy | `Reject` after advancing history floor | normal current policy |

An empty-entity proof may be a bounded exact first-key/length proof owned by the
store. If emptiness cannot be proven within the DDL boundary, the conservative
nonempty row applies.

Publishing an additive field over existing rows atomically:

1. increments row-layout version;
2. appends one dense physical slot;
3. records `introduced_in_layout`;
4. freezes the historical fill if required;
5. records the independently mutable insert default;
6. validates every payload under the candidate accepted field contract;
7. stages any required accepted index/relation/constraint activation; and
8. publishes the candidate schema or changes nothing.

### `SET DEFAULT`

`SET DEFAULT value`:

- encodes one deterministic exact canonical insert-default payload;
- changes only `insert_default`;
- does not change `historical_fill`;
- does not bump row-layout version;
- does not scan or rewrite rows;
- does not rebuild indexes merely because historical rows are physically
  short; and
- bumps accepted schema version/fingerprint through the normal DDL contract.

Every future row still validates its resolved default through checks, unique
indexes, strong relations, and other final after-image constraints. A default
that is individually well typed but conflicts with another authored field is
rejected when that concrete write is attempted.

`SET DEFAULT NULL` canonicalizes to no insert default only for a nullable
field. It is equivalent to `DROP DEFAULT`, not a persisted null payload. It
rejects on a non-nullable field because such a declaration cannot construct a
valid omitted value.

### `DROP DEFAULT`

`DROP DEFAULT` clears only `insert_default`:

- nullable future omission becomes `NULL`;
- non-nullable future omission becomes a typed missing-required-field error;
- historical rows retain their frozen fill;
- existing physical values remain unchanged;
- row-layout version is unchanged; and
- no index rebuild or row scan is required.

Dropping a default from a non-nullable field is therefore legal. `NOT NULL`
governs accepted values; it does not require omission to be legal.

### Nullability Changes

`DROP NOT NULL` changes future omission to `NULL` only when no insert default
exists. It does not rewrite or replace a frozen historical payload.

`SET NOT NULL` must validate the complete logical row set, including historical
fills. A field with historical `Null` cannot become non-nullable until an exact
bounded rewrite/backfill removes every old null and atomically advances the
layout history floor. A non-null historical payload may satisfy the field-level
condition, but all 0.209 checks and other constraints still apply.

### Rename

A field rename changes stable accepted naming, not physical row layout,
introduction version, insert-default bytes, or historical-fill bytes.

### Drop Or Type Rewrite

A drop, slot reassignment, or incompatible kind/codec change cannot reinterpret
older stamped rows under a new shape. It requires a marker-owned exact rewrite
that:

1. decodes each row under the accepted before-layout;
2. constructs and validates the complete after-image;
3. stages every row and derived effect;
4. emits the new complete layout version;
5. proves no before-layout row remains;
6. advances `history_floor` atomically; and
7. only then publishes slot reassignment or incompatible field metadata.

If the complete rewrite cannot fit the admitted atomic boundary, the DDL form
remains unsupported until a separate bounded migration design provides an
equivalent no-partial-publication proof.

## Generated Ownership And Reconciliation

Field origin continues to control who may propose a default change:

- generated-owned fields receive insert-default proposals from generated
  schema and reconcile them through the catalog-native transition owner;
- DDL-owned fields receive insert-default proposals from admitted SQL DDL;
- SQL may inspect but cannot change or remove a generated-owned default; and
- an exact SQL no-op may return no mutation without transferring ownership.

A generated proposal never supplies runtime historical authority. Reconciliation
handles the two cases explicitly:

1. for a newly added generated field, the catalog-native additive transition
   derives and freezes historical fill from the accepted candidate default or
   nullability while assigning the new layout version; and
2. for an existing generated field whose declared default changed, only the
   accepted insert default changes and the stored historical fill is retained.

Generated `EntityModel` therefore proposes current intent. It does not carry a
copy of accepted layout history, does not reconstruct a missing historical
fill, and does not become read/write runtime authority. Startup reconciliation
must fail with a typed schema-transition diagnostic if the generated proposal
cannot be admitted against stored accepted history.

## Layout History Lifecycle

### Additive History

Layout versions are entity-local and monotonic. Metadata-only schema changes do
not consume a layout version. An additive physical slot does.

Example:

~~~text
layout 1: id, name
layout 2: id, name, tier
layout 3: id, name, tier, note

row stamped 1 -> exactly 2 slots; tier/note use their frozen fills
row stamped 2 -> exactly 3 slots; note uses its frozen fill
row stamped 3 -> exactly 4 slots; no historical fill is used
~~~

A row stamped `2` with only two slots is corruption. A row stamped `1` with
three slots is also corruption. Slot count does not override layout identity.

### Natural Promotion On Write

Any successful update or replacement of a historical row emits a complete row
at the current layout version. This gradually removes lazy historical absence
without changing logical values.

### History-Floor Advancement

Historical fills remain authoritative until an exact proof establishes that no
row below a chosen layout version remains. An admitted full rewrite may then:

- advance `history_floor`;
- change now-unreachable historical fills to `Reject`;
- permit dense slot reorganization; and
- reduce accepted historical metadata.

An ordinary read, partial scan, suffix exhaustion assumption, or count estimate
cannot advance the floor. A future standalone compaction command is outside
0.211; 0.210 Deep may provide verification evidence but does not repair or
mutate metadata itself.

## Defaults And Accepted Constraints

0.209 constraint evaluation observes logical values, not physical omission.

For incoming writes:

~~~text
authored intent
-> insert default / null / generation / management
-> sanitization
-> exact canonical after-image
-> accepted checks, uniqueness, relations, and final admission
~~~

For historical validation and 0.210 Deep inspection:

~~~text
row envelope + stamped layout
-> exact physical slot decode
-> frozen historical fill for legitimately future fields
-> logical row
-> accepted constraints and derived-state verification
~~~

Changing an insert default does not require revalidating existing rows because
their logical values do not change. It also does not prove that every possible
future combination will satisfy a multi-field check; each actual after-image
must pass.

Schema dependencies include both default roles:

- enum variant removal must reject while either payload references it;
- bound or codec tightening must admit both payloads;
- a historical relation fill must remain a valid typed identity and is subject
  to 0.209 relation activation/projection rules; and
- index construction over old rows uses frozen historical values.

No accepted check, relation, index, or planner path may consult the current
insert default to reinterpret an existing row.

## Imports, Restores, Backfills, And Recovery

### Imports

An import has two explicit lanes:

- **logical current import** supplies field intents and constructs a complete
  current row through ordinary accepted insertion policy; or
- **trusted physical restore** supplies current-form row bytes whose row-layout
  version, exact slot count, key identity, codecs, and accepted schema identity
  are validated.

A current logical import cannot submit a short slot container. A trusted
physical restore cannot claim an old layout version unless that version remains
inside the accepted history window and its exact slot shape matches.

### Schema Backfills

A backfill is an explicit row mutation. It decodes frozen historical values,
authors the intended replacement, validates the complete after-image, and emits
the current row layout. It cannot mutate historical fill metadata piecemeal.

### Restore Across Schema Identity

Restoring rows against a different accepted schema is not a normal decode
operation. It requires an explicit migration that binds source schema/layout
identity to target accepted schema and either produces current complete rows or
fails. No generated-model reconstruction or best-effort slot matching exists.

### Recovery

Recovery replays canonical row bytes and derived effects already admitted by
the commit marker. It preserves the row-layout stamp and does not:

- rerun an insert default;
- regenerate an ID or timestamp;
- apply current management time;
- replace a historical fill;
- sanitize again; or
- reinterpret old bytes through generated models.

Recovered-effect verification checks the replayed row under the marker-bound
accepted schema identity.

## Introspection

Typed schema description, `DESCRIBE`, and the maintained column-introspection
surface expose at least:

| Column fact | Meaning |
|---|---|
| `insert_omission` | `required`, `null`, `default`, `generated`, or `managed` |
| `insert_default` | bounded canonical rendering when present |
| `insert_default_bytes` | payload byte count |
| `insert_default_hash` | stable bounded diagnostic hash |
| `introduced_in_layout` | entity-local row-layout version |
| `historical_fill` | `reject`, `null`, or `payload` |
| `historical_fill_bytes` | payload byte count |
| `historical_fill_hash` | stable bounded diagnostic hash |
| `row_layout_current` | current accepted physical layout version |
| `row_layout_history_floor` | oldest admitted row layout version |

Canonical value rendering is computed by decoding accepted payload bytes under
the accepted field contract. A display string is not persisted as second
authority. Large text/blob/collection/composite values are bounded and may be
rendered as kind, byte length, and hash rather than echoed.

Introspection must make this difference obvious:

~~~text
tier:
  insert_default = 'silver'
  historical_fill = 'bronze' for rows before layout 2
~~~

It must never collapse both facts into one `default` label when they differ.

## Fingerprints, Plans, And Tokens

Accepted schema identity includes:

- current and floor row-layout versions;
- every field introduction version;
- insert-default discriminants and payload bytes;
- historical-fill discriminants and payload bytes; and
- existing exact kind, nullability, write policy, origin, index, relation, and
  constraint facts.

Changing an insert default invalidates cached write plans and any prepared
statement whose explicit `DEFAULT` semantics depend on it. It need not
invalidate a read plan merely because existing logical rows are unchanged, but
simple V1 cache invalidation may conservatively bind all plans to the complete
accepted fingerprint.

Row-layout changes invalidate every row-codec projection and layout-dependent
cursor/token fingerprint. A token cannot resume against a different accepted
layout contract unless its owning design explicitly proves independence.

The row itself stores layout version, not an insert-default fingerprint.

## Error Model

Machine-readable errors distinguish at least:

| Error class | Example |
|---|---|
| Write validation | omitted non-null field has no insert default |
| Write validation | explicit `NULL` supplied to non-null field |
| Write validation | `UPDATE ... SET generated = DEFAULT` is unsupported |
| DDL validation | default literal has wrong exact kind or exceeds bounds |
| DDL validation | `SET DEFAULT NULL` on non-null field |
| DDL validation | additive required field has existing rows and no historical fill |
| Schema integrity | insert-default or historical-fill payload is noncanonical |
| Schema integrity | field introduction version is outside layout closure |
| Persisted-row corruption | row layout version is outside accepted history window |
| Persisted-row corruption | physical slot count differs from stamped layout |
| Persisted-row corruption | historical fill is `Reject` for a legitimately absent field |
| Migration validation | existing logical rows prevent `SET NOT NULL` or rewrite |
| Restore validation | source layout/schema identity is not admitted |

Diagnostics expose entity, field ID/name, slot, row primary key when safely
decoded, stamped row-layout version, current/floor versions, expected/actual
slot count, and a stable diagnostic code. Callers do not parse display text for
classification.

A malformed stored row is not reported as an incoming missing-field validation
error. It is integrity/corruption. An incompatible historical row discovered
during a candidate rewrite is migration validation. This distinction matches
the 0.209 and 0.210 operational model.

## Persisted Encoding And Hard Cut

0.211 changes at least these durable surfaces:

- accepted field snapshot and wire encoding;
- accepted row-layout encoding;
- accepted schema fingerprints and revisions;
- canonical row envelope;
- row encoder/decoder contracts;
- schema description DTOs;
- SQL prepared/default intent fingerprints; and
- any durable migration or integrity job that embeds row-layout identity.

The pre-`1.0.0` rule is direct:

1. delete `SchemaFieldDefault` and its single-role wire form;
2. add current-form insert-default and historical-fill facts;
3. add current-form `RowLayoutVersion`, introduction version, and history
   floor;
4. change the canonical row envelope to the stamped current format;
5. reject old schema snapshots and unstamped rows with typed incompatible-
   format errors;
6. do not infer historical fill from nullability or current insert default;
7. do not infer row layout from physical slot count;
8. do not reconstruct missing accepted facts from generated models;
9. recreate pre-cut development stores or migrate them with a separately
   authorized one-off tool outside runtime fallback; and
10. delete tests whose only purpose is preserving the old form while retaining
    tests for the maintained current semantics.

The implementation updates
`docs/contracts/PERSISTED_FORMAT_INVENTORY.md` before closeout. There is one
active row and schema format after the cut.

## Alternatives Considered

### Keep One Default And Rewrite Rows On Every Default Change

Rejected. It turns a small metadata operation into an unbounded migration,
creates unnecessary index/relation churn, and still obscures the semantic
difference between future policy and historical data.

### Keep One Default And Document That Old Rows Change

Rejected. Untouched rows would change query results and derived invariants
without a row mutation. This is surprising and operationally unsafe.

### Copy The Old Default Into Every Existing Row During `ADD COLUMN`

Useful only when an exact atomic rewrite is already affordable. It cannot be
the general behavior for large entities because one canister message may not
fit. Frozen historical fill provides stable semantics without assuming a full
rewrite.

### Split The Defaults But Trust Physical Slot Count

Rejected as insufficiently strict. A legitimate old short row and truncated
current row remain indistinguishable.

### Stamp Full Schema Version In Every Row

Rejected. Metadata-only changes would create new row identities despite an
unchanged physical shape, and historical interpretation would be coupled to
unrelated schema evolution.

### Stamp Full Schema Fingerprint In Every Row

Rejected. It adds excessive per-row bytes and makes defaults, constraints,
names, and other non-layout facts part of physical row decoding.

### Treat Rust `Default` As Database Authority

Rejected. Application code is not stable accepted metadata, is not available
to SQL/restore/integrity uniformly, and may change across upgrades.

### Store Default Expressions Instead Of Canonical Values

Rejected for V1. Expression timing, nondeterminism, semantics versioning, and
upgrade stability add risk with little benefit. Deterministic canonical data
is sufficient. Insert generation remains the explicit dynamic mechanism.

### Make A Default Mandatory For Every Field

Rejected. Required authorship is a valuable exact contract. A non-nullable
field without a default should reject omission.

### Make `DROP DEFAULT` Rewrite Historical Fill To Null Or Reject

Rejected. That would mutate existing logical rows or make valid historical
rows unreadable. Historical fill is independent.

### Persist An Encoding-Level Reference From Historical Fill To Insert Default

Rejected as the semantic model because later default changes require copy-on-
write and invite accidental aliasing. Direct independent values are clearer.
Encoding-level deduplication remains a measurement-driven implementation
option if it is invisible after decode.

### Allow `DEFAULT` Everywhere In SQL Expressions

Rejected. `DEFAULT` is contextual write intent, not a scalar with a standalone
type. V1 admits it only in direct insert/update value positions.

## Focused Validation

### Accepted Catalog And Codec

- insert-default none/payload round trips canonically;
- historical reject/null/payload round trips canonically;
- introduction/current/floor layout versions round trip canonically;
- invalid version ranges, slot ordering, and payloads reject;
- payloads with wrong kind, noncanonical representation, forbidden null,
  decimal scale, text/blob bound, enum variant, or collection shape reject;
- accepted fingerprints differ for every semantic fact; and
- unknown/old current-form variants fail closed within decode bounds.

### Row Layout And Corruption

- initial-layout rows require every initial slot regardless of nullability or
  insert default;
- each admitted historical layout accepts exactly its declared slot count;
- short, long, forged-version, zero-version, future-version, and below-floor
  rows reject;
- a current-version row missing one trailing slot is corruption;
- a historical row receives only fills for fields introduced later;
- sparse direct reads and full reads classify the same malformed row;
- malformed span lengths/offsets remain bounded and fail closed; and
- old unstamped row envelopes reject after the hard cut.

### Temporal Default Matrix

For nullable and non-nullable primitive, enum, decimal, text/blob, collection,
composite, and relation-capable fields:

- add with default, read old row, change default, read old row unchanged;
- drop default, read old row unchanged;
- add nullable without default, later set default, old row remains null;
- explicit null remains null across default changes;
- new omitted rows use only the current insert default;
- new authored rows preserve their authored value;
- update of an old row emits its frozen logical value physically;
- unrelated update preserves explicit null and historical fill semantics;
- indexed historical keys remain unchanged after default metadata changes; and
- relation/check/unique behavior observes final logical values.

### Mutation Ingress Parity

Run the same accepted scenarios through:

- generated typed create;
- full typed insert/save;
- structural insert and replacement;
- SQL insert and SQL `DEFAULT`;
- SQL update `SET DEFAULT`;
- atomic batches, including a failure in the final operation;
- exact and resumable bulk updates;
- DDL backfill/rewrite helpers;
- logical import;
- trusted physical restore; and
- recovery replay.

Every successful lane produces the same canonical logical after-image and
current row-layout stamp. Every rejected lane leaves exact before-state.

### Generated, Managed, And Rust Defaults

- insert-generated ULIDs/timestamps run once per admitted insert intent;
- historical fill never invokes generation;
- replay never regenerates;
- engine-managed fields remain non-authorable;
- `UPDATE ... SET generated_or_managed = DEFAULT` rejects;
- a create DTO Rust `Default` means no authored inputs;
- an entity lacks Rust `Default` unless explicitly requested;
- explicit entity Rust `Default` insertion is treated as an authored full
  value; and
- changing generated Rust code without accepted reconciliation cannot change
  database default behavior.

### DDL Atomicity

- additive publication freezes both facts or changes nothing;
- empty-entity required addition advances the floor safely;
- populated required addition without a fill rejects before publication;
- default set/drop changes schema version but not layout version;
- metadata interruption leaves before or after snapshot, never mixed facts;
- nullable `SET DEFAULT NULL` equals drop; non-null rejects;
- `SET NOT NULL` sees historical nulls;
- index/relation/constraint failure leaves row/schema/derived before-state;
- exact rewrite stamps every row before advancing floor; and
- failpoints around final publication never admit a row version without its
  schema or vice versa.

### SQL Surface

- omitted column, explicit `DEFAULT`, and the corresponding typed omission
  converge;
- `DEFAULT VALUES` succeeds and fails field-by-field under the same resolver;
- update omission preserves while `SET field = DEFAULT` assigns;
- default in predicates, nested expressions, functions, checks, and unsupported
  positions fails closed;
- wrong literal types fail rather than coerce lossily;
- prepared plans invalidate on accepted default changes; and
- `DESCRIBE` renders differing insert/historical values distinctly.

### Constraint And Integrity Integration

- 0.209 checks evaluate defaulted new rows and frozen old rows identically to
  equivalent physical values;
- unique/index projections match logical historical values;
- strong relation targets and reverse entries match historical relation fills;
- Quick validates catalog/layout/fill closure;
- Deep reports definite row-stamp/slot corruption with stable facts;
- Deep does not classify valid older layout rows as corruption;
- an invalidated scan never advances history floor; and
- inspection never repairs or materializes a row.

### Recovery And Restore

- interruption before/after marker preserves exact row layout/version effects;
- replayed rows retain their stamped version and values;
- default changes between original execution and replay do not affect bytes;
- current logical imports emit complete rows;
- forged old-layout physical restores reject on exact shape or schema identity;
  and
- cross-schema restore requires explicit migration rather than fallback.

### Differential And Independent Oracles

Extend the 0.204 comparison matrix where semantics intentionally overlap:

- insert omission versus explicit null;
- `SET/DROP DEFAULT` future-only behavior;
- add-column constant default preserving old logical values;
- non-null fields without a default rejecting omission;
- insert `DEFAULT`, `DEFAULT VALUES`, and update `SET DEFAULT`; and
- exact before-state after rejected mutations.

SQLite or PostgreSQL may be an oracle only for overlapping SQL semantics.
Versioned row envelopes, typed identity, accepted schema, bounded execution,
recovery, and frozen historical fill require independent IcyDB oracles.

## Performance And Complexity Budget

### Write Runtime

The hot insert path may perform one accepted insertion-policy resolution per
omitted field and one canonical payload copy/decode. It must not materialize a
generated Rust default and then decode the accepted default again.

Measure:

- no-default complete typed insert;
- sparse typed/SQL insert with many defaults;
- generated and managed fields;
- current-row update;
- historical-row update requiring frozen fill;
- indexed/check/relation-bearing defaulted rows; and
- batch/bulk default-heavy operations.

Accepted-first typed creation should remove duplicate semantic work even if
its internal dataflow changes.

### Read Runtime

Current-layout reads pay one fixed-width layout-version decode and exact slot-
count comparison. Historical reads additionally materialize only missing
future fields. Sparse direct reads must retain their bounded selective-decode
shape and must not decode every physical field merely to validate layout.

Report current and historical read deltas separately.

### Persisted Bytes

The expected row-envelope delta is four raw bytes per row. Report actual raw
row, stable-memory node, journal, marker, and fixture impact.

Accepted schema may temporarily carry two equal default payload copies for an
additive field. Report:

- snapshot bytes with no defaults;
- many small defaults;
- maximum bounded text/blob/composite defaults;
- divergent insert/historical payloads; and
- schema fingerprint/cache footprint.

Do not add semantic aliasing merely to optimize unmeasured catalog bytes.

### Wasm Size

Report raw non-gzipped `.wasm` bytes for every slice. Gzip is secondary
context. Prefer one shared canonical payload validator and one row-envelope
codec rather than frontend-specific default implementations.

### Source Complexity

Each slice reports files touched, approximate line delta, owner count, and
whether the implementation became simpler, stayed neutral, or became more
complex.

The target ownership shape is:

~~~text
accepted field insert default -> future omission / explicit DEFAULT
accepted field historical fill -> older stamped row only
insert generation              -> per-insert dynamic values
engine management              -> engine-authored lifecycle values
row layout version             -> exact physical row shape
accepted constraint catalog    -> final logical after-image invariants
generated Rust defaults        -> proposal / explicit Rust trait only
~~~

The design intentionally adds one physical identity owner while removing the
ambiguous shared missing-slot owner and generated runtime default authority.

## Risks And Mitigations

### Risk: Row-Layout Version Adds Per-Row Overhead

Mitigation: store a fixed-width entity-local `u32`, not a full fingerprint.
Measure raw stable-memory and runtime impact; four bytes is the explicit
correctness budget.

### Risk: Two Default Payloads Drift Accidentally

Mitigation: use separate types, accessors, mutation functions, and source
guards. Only additive publication initializes both; later default DDL has no
historical-fill capability.

### Risk: Layout Version Is Bumped For Metadata-Only DDL

Mitigation: separate `RowLayoutVersion` from `SchemaVersion`. Tests require
default/nullability/rename metadata cases to preserve physical layout identity
unless a rewrite actually changes shape.

### Risk: A Forged Old Version Obtains Historical Values

Mitigation: require exact slot count for the claimed admitted version and
restrict trusted physical restores. Runtime writes always stamp current
layout. Store corruption remains detectable even though a sufficiently
coordinated malicious byte rewrite is outside the storage threat model.

### Risk: Layout History Grows Without Bound

Mitigation: bound versions and accepted fields, store introduction facts rather
than arbitrary full layouts, and allow only exact rewrite proof to advance the
history floor. Measure catalog growth across maximum additive DDL.

### Risk: Default Changes Still Affect Derived State

Mitigation: all existing-row reads use frozen fill or physical values. Future
rows stage indexes, relations, and checks from their actual final values.
Focused tests compare derived bytes before and after metadata-only DDL.

### Risk: Accepted-First Typed Creation Changes Sanitizer Behavior

Mitigation: preserve the documented sanitizer/validator order over the same
complete typed after-image and add parity tests for defaulted, generated,
managed, optional, collection, and composite fields.

### Risk: `DEFAULT` Becomes A Coercion Escape Hatch

Mitigation: keep it as contextual write intent and resolve only through exact
accepted policy. It never becomes `Value::Any`, a null alias in non-null
fields, or a general expression.

### Risk: Historical Fill Outlives Its Need

Mitigation: retaining it is semantically safe and bounded. Remove it only after
an exact rewrite/history-floor advancement; never from a heuristic scan.

### Risk: Large Default Payloads Duplicate Catalog Bytes

Mitigation: enforce existing per-field/schema bounds, measure real snapshots,
and consider encoding-only deduplication later without changing the decoded
two-fact authority model.

### Risk: Full Rewrite DDL Exceeds One Message

Mitigation: reject without changes unless the exact atomic boundary fits. A
future bounded migration may stage a complete candidate and publish only after
stable proof; 0.211 does not permit partial visible layout change.

### Risk: Recovery Recomputes Under New Defaults

Mitigation: recovery consumes marker-bound canonical bytes and accepted schema
identity. Tests change default metadata around interruption and prove byte-
exact replay.

- [ ] explicit Rust entity `Default` remains opt-in and separate;
- [ ] insert generation and engine management remain separate owners;
- [ ] generated schema proposes insert defaults only, while catalog-native
      reconciliation derives additions and preserves accepted history;
- [ ] SQL cannot redefine or remove generated-owned defaults;
- [ ] `SET DEFAULT` changes only future insertion policy;
- [ ] `DROP DEFAULT` is legal on non-null fields and makes future omission
      required;
- [ ] nullable `SET DEFAULT NULL` canonicalizes to no default and non-null
      usage rejects;
- [ ] default metadata changes do not bump row-layout version, scan rows, or
      rebuild indexes solely because of old short rows;
- [ ] additive DDL freezes a historical fill or proves the entity empty;
- [ ] required additive fields without a fill reject over existing rows;
- [ ] nullability, drop, type, and slot changes obey exact rewrite/history-
      floor rules;
- [ ] SQL insert `DEFAULT`, `DEFAULT VALUES`, and update `SET DEFAULT` lower to
      the shared accepted policy;
- [ ] unsupported SQL `DEFAULT` expression positions fail closed;
- [ ] explicit `NULL` remains distinct from omission and `DEFAULT`;
- [ ] 0.209 constraints, indexes, and relations observe final logical values;
- [ ] 0.210 Quick/Deep verifies layout/default closure and never repairs;
- [ ] imports/restores cannot exploit a forged short current row;
- [ ] recovery never reevaluates defaults, generation, management, or fills;
- [ ] introspection exposes both temporal facts and layout history distinctly;
- [ ] schema/plan/token fingerprints include the required semantic facts;
- [ ] old single-default accepted snapshots and unstamped row envelopes fail
      current-form decode without aliases, inference, or generated fallback;
- [ ] focused correctness, corruption, DDL atomicity, recovery, differential,
      and boundary-parity evidence passes;
- [ ] runtime, peak heap, row/catalog bytes, raw non-gzipped wasm, files
      touched, approximate line delta, and complexity direction are reported;
      and
- [ ] no legacy anti-resurrection tests remain.

## Final Rule

A database default constructs a future after-image; it is not a live value for
old rows. Historical physical absence is admitted only by a stamped row layout
and a separately frozen accepted fill. Every current write emits a complete
current-layout row, every frontend uses the same accepted insertion policy, and
no generated Rust expression, physical slot-count guess, or compatibility
fallback can silently supply persisted data.
