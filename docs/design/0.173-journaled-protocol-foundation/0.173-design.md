# IcyDB 0.173 — Journaled Protocol Foundation

## Status

Closed protocol-foundation design. Runtime journaled storage remains
unadmitted; public runtime admission is reserved for 0.174 or later.

## Thesis

0.172 reserved `storage(journaled(...))` as a future journaled cached-stable
store. 0.173 should not publicly admit that storage form. It should close the
protocol foundation required before public runtime admission:

- commit-marker-bound journal publication;
- journal batch identity, ordering, and visibility;
- physical journal-tail ordering;
- fold watermark semantics;
- schema replay ordering;
- index truth versus materialized index state;
- source-audit guards proving no public journaled runtime admission exists.

The hard release rule is:

```text
No released 0.173 tag may accept public storage(journaled(...)).
```

0.173 may add internal protocol scaffolding and tests, but accepted schemas,
generated runtime wiring, registry handles, diagnostics, relation policy, and
store construction must remain stable/heap only.

## Why This Is 0.173

0.172 answered the storage-shape question:

```text
canonical stable BTrees + committed journal tail + volatile Rust BTree caches
```

The next risk is not public syntax. The next risk is protocol ambiguity. Before
IcyDB admits a journaled runtime store, it must define exactly when journal
bytes become committed state, how recovery orders them, and how fold prevents
loss or double-apply.

0.173 should therefore be a protocol foundation line. Public runtime admission
belongs in 0.174 after this foundation is closed.

## Reserved Storage Form

The future full four-ID form remains:

```rust
#[store(
    name = "sessions",
    storage(journaled(
        data_memory_id = 40,
        index_memory_id = 41,
        schema_memory_id = 42,
        journal_memory_id = 43,
    )),
)]
```

Memory roles remain design-only in 0.173:

- `data_memory_id`: canonical durable row BTree source of truth;
- `index_memory_id`: canonical durable index BTree materialized from data plus
  accepted schema;
- `schema_memory_id`: canonical durable schema BTree source of truth;
- `journal_memory_id`: durable journal tail for committed mutations not yet
  folded into the canonical stable BTrees.

`storage(journaled(...))` stays parser-reserved in public derive syntax until a
later admission line provides generated wiring, runtime construction, recovery,
fold, diagnostics, relation behavior, and closeout proof in one releasable
branch.

## Non-Admission Boundary

0.173 must not permit:

- accepted schemas with journaled storage;
- generated runtime wiring for journaled stores;
- store registration with journaled storage;
- runtime store handles with journaled storage capabilities;
- journaled data, index, or schema store construction;
- diagnostics that render a live journaled store;
- relation policy branches for concrete journaled stores.

Allowed 0.173 journaled references:

- design/changelog proof text;
- parser-reserved diagnostics inherited from 0.172;
- internal protocol model tests, if they cannot instantiate runtime stores;
- source-audit tests proving journaled runtime admission is absent.

## Commit-Marker Journal Publication Strategy

0.173.1 chooses the marker-bound journal publication strategy required before
runtime admission work can start.

Chosen strategy:

```text
embedded bounded journal batch payload in the existing commit marker
```

Rationale:

- the existing commit marker remains the in-flight recovery authority;
- recovery can complete a missing journal append from marker authority;
- no separate prepared-batch store is introduced before runtime admission;
- bytes in journal memory are never committed only because they exist there.

Rejected alternatives:

- durable prepared-batch locator plus digest: adds a second durable protocol
  surface before the first journaled runtime store is admitted;
- verify-only digest with fail-closed recovery: can detect inconsistency but
  cannot complete a missing journal append, making first admission harder to
  reason about.

Recovery behavior:

- missing journal batch: recovery reconstructs and appends the complete bounded
  batch payload embedded in the persisted commit marker;
- partial journal batch: partial raw tail bytes are not committed state;
  recovery either repairs from the marker payload when safe or fails closed;
- duplicate journal batch: recovery verifies the existing complete batch
  matches the marker payload and treats the append as already complete;
- mismatched journal batch: recovery fails closed if an existing batch identity
  maps to bytes that differ from the marker payload;
- already-present marker-bound journal batch: recovery reuses the complete
  verified batch and must not append a duplicate;
- bytes-present-but-not-committed batch: recovery must not replay raw journal
  bytes unless they form a complete, ordered, marker-bound batch visible under
  the existing commit-marker publication protocol.

0.174 may implement this by hard-cutting the commit-marker payload shape to
carry bounded journal batch payloads. If that proves impractical, the design
must be revised before runtime admission rather than silently switching to a
locator, digest-only, or hybrid strategy.

A journal batch must not become committed merely because bytes exist in journal
memory. Visibility remains bound to the existing commit-marker authority.

## Journaled Write Publication State Machine

0.173 should specify the exact write publication sequence without admitting
runtime journaled stores.

The intended first protocol is:

1. validate inputs, relations, and accepted schema contracts;
2. prepare the complete mutation batch, including stable row operations and
   journaled logical row/schema records;
3. persist an existing commit marker with the durable mutation authority needed
   to recover the batch, including the chosen journal publication strategy;
4. append complete journal batches for journaled participants, each bound to
   the commit marker identity;
5. apply ordinary stable-store participants through the existing stable durable
   mutation path;
6. update journaled live cache projections only after journal publication has a
   recovery path;
7. finish the existing commit marker only after all durable publication work is
   complete;
8. report success only after the write is fully published or fully recoverable
   through the existing marker.

For journaled participants, normal write publication appends marker-bound
journal batches and updates live cache projections. It does not fold row/schema
mutations into the canonical stable BTrees during the write path. Canonical
stable BTrees receive journaled mutations during fold.

Recovery must resolve any existing commit marker before exposing journaled
state. Partial raw journal bytes are not committed state.

## Journal Batch Envelope

0.173 should define a batch envelope, even if the persisted runtime format is
not yet admitted.

Required conceptual shape:

```rust
struct JournalBatch {
    version: JournalFormatVersion,
    batch_id: JournalBatchId,
    commit_marker_id: CommitMarkerId,
    journal_sequence: JournalSequence,
    records: BoundedVec<JournalRecord>,
}
```

The envelope must let recovery answer:

- is this batch complete?
- is this batch bound to the existing commit-marker authority?
- where does this batch sit in deterministic replay order?
- was this batch already folded?
- can replay or fold apply this batch idempotently?

Logical records should stay row/schema oriented:

```rust
enum JournalRecord {
    RowPut { entity_path, primary_key, row_bytes, schema_fingerprint },
    RowDelete { entity_path, primary_key, schema_fingerprint },
    SchemaPut { store_path, schema_snapshot_bytes },
}
```

Index entries are not independent durable journal truth in the first model.

## Journal Tail Ordering And Fold Watermark

0.173.2 chooses a single durable ordering authority:

```text
monotonic journal_sequence assigned to each complete journal batch
```

Physical journal-tail shape for the first admitted model:

```text
journal key   = journal_sequence
journal value = encoded JournalBatch
```

Replay order is ascending `journal_sequence` after the durable fold watermark.
`batch_id` is idempotence identity, not replay order. `commit_marker_id` binds
visibility to the existing commit-marker authority, not replay order. A
separate byte-offset ordering authority is not used in the first model.

Journal sequence allocation rules:

- each complete journal batch receives one monotonically increasing sequence;
- a retry of the same marker-bound batch must reuse the same sequence and
  batch identity;
- a different batch must not reuse an existing sequence;
- a batch whose encoded `journal_sequence` disagrees with the physical journal
  key fails closed;
- sequence allocation state must be durable before the batch can be treated as
  published.

Recovery must handle the following deterministically:

- duplicate batch IDs;
- sequence gaps;
- out-of-order batches;
- commit-marker binding mismatches;
- folded batches still visible past the fold watermark.

Allowed outcomes are successful idempotent recovery or fail-closed recovery.
Silent best-effort replay is not allowed.

Recovery handling:

- duplicate batch ID at the same sequence with identical bytes: treat as the
  same idempotent batch;
- duplicate batch ID at a different sequence: fail closed;
- duplicate sequence with different bytes or identity: fail closed;
- sequence gap after the fold watermark: fail closed unless the gap is covered
  by a durable fold watermark;
- out-of-order physical iteration: fail closed if sorted journal keys do not
  match encoded sequence order;
- commit-marker binding mismatch: fail closed;
- batch sequence at or below the fold watermark: skip and do not replay;
- batch sequence after the fold watermark: replay only if complete,
  marker-bound, and ordered.

## Fold Watermark And Replay Boundary

0.173.3 defines the fold watermark as the durable replay boundary:

```text
all complete committed journal batches covered by the watermark
have been durably applied to canonical stable BTrees
and must not be replayed again
```

Watermark shape for the first admitted model:

```rust
struct FoldWatermark {
    highest_folded_journal_sequence: JournalSequence,
    fold_epoch: u64,
}
```

Replay rules:

- recovery must replay only complete committed journal batches with
  `journal_sequence > highest_folded_journal_sequence`;
- batches with `journal_sequence <= highest_folded_journal_sequence` must be
  skipped and must not be applied again;
- a journal tail containing only folded batches is valid and should recover as
  an empty replay tail;
- a journal tail containing a folded batch with bytes that disagree with the
  recorded fold watermark fails closed;
- a missing batch with sequence above the watermark fails closed unless a
  marker-bound recovery path can reconstruct it.

Durable fold phases:

1. select a contiguous committed range above the current fold watermark;
2. persist fold intent containing the selected range and next `fold_epoch`;
3. apply row and schema records in the range to canonical stable BTrees;
4. rebuild or validate affected index materialized state;
5. persist the new fold watermark;
6. advance or clear journal-tail bytes covered by the watermark;
7. clear fold intent or make completion derivable from the watermark.

Restart behavior:

- after persisted fold intent but before stable-BTree apply: recover by
  replaying the still-unfolded range from the journal tail;
- after partial stable-BTree apply but before watermark persist: reapply the
  selected range idempotently or fail closed if idempotence cannot be proven;
- after watermark persist but before tail cleanup: skip covered batches and
  retry cleanup later;
- after partial tail cleanup: use the watermark as authority and replay only
  batches after it.

Fold completion is authoritative only after the watermark is durable. Tail
cleanup is space reclamation, not durability authority.

## Index Truth

0.173.4 defines index truth for future journaled stores.

For future journaled stores, canonical data and schema BTrees are logical
durable truth. The canonical index BTree is durable materialized state derived
from data plus accepted schema.

The journal must not make index entries an independent durable truth in the
first model. Recovery may validate or rebuild index state from recovered rows
and accepted schema. If validation detects divergence that cannot be safely
repaired, recovery must fail closed.

Index recovery rules:

- journal records contain no durable index records in the first model;
- recovered rows plus the reconstructed accepted schema are the input to index
  rebuild or validation;
- index materialization must be deterministic from row bytes, primary keys,
  accepted index definitions, relation-maintenance definitions, and accepted
  field codecs;
- if the durable index BTree contains entries missing from the recovered
  row/schema view, recovery must delete/rebuild them or fail closed;
- if the durable index BTree is missing entries required by the recovered
  row/schema view, recovery must insert/rebuild them or fail closed;
- if expression, relation, or field-path index derivation cannot be evaluated
  from accepted schema authority, recovery must fail closed;
- source audits must continue to prove index codecs are unchanged unless a
  later runtime-admission design explicitly hard-cuts them.

A later design may add explicit index journal records only with divergence
proof tests showing row/index journal truth cannot disagree.

## Schema Replay Invariant

0.173.4 defines schema replay for future journaled stores.

Schema replay must be deterministic and fail closed.

During recovery:

- schema journal records are applied in journal order before later row records
  that depend on their schema fingerprint;
- row records carry the schema fingerprint they require for decode or
  validation;
- row bytes may remain opaque until the matching accepted schema snapshot is
  reconstructed;
- an unknown, missing, future, or mismatched schema fingerprint fails closed;
- generated `EntityModel` or `IndexModel` must not be used as fallback schema
  authority.

Schema replay rules:

- canonical stable schema BTree snapshots are loaded first;
- committed journal `SchemaPut` records above the fold watermark are applied in
  journal order;
- the reconstructed schema timeline is the only authority for resolving row
  `schema_fingerprint` values;
- a `RowPut` or `RowDelete` whose fingerprint is older than the current replay
  point may use that matching accepted snapshot only if it exists in the
  reconstructed timeline;
- a row record that references a schema fingerprint introduced by a later
  `SchemaPut` fails closed as out-of-order;
- schema snapshot bytes in journal records must decode through the accepted
  schema snapshot codec and bounds, not through generated model fallback;
- schema-store materialized state after replay must match the reconstructed
  accepted schema timeline.

## Fold Phases

0.173.3 defines fold phases before implementation:

1. select a complete committed journal batch range;
2. mark fold started or derive an idempotent fold range from durable state;
3. apply row and schema records to the canonical stable BTrees;
4. rebuild or validate affected indexes from accepted schema and recovered
   rows;
5. persist a fold watermark;
6. advance or clear the journal tail through the completed range;
7. mark fold complete or make completion derivable from the durable watermark
   and journal tail.

Recovery must be correct after interruption at every phase.

## Source-Audit Expectations

0.173 should add or preserve guards proving journaled runtime admission is
absent outside allowed zones.

Disallowed production hits in 0.173:

- `StoreStorage::Journaled`;
- `StoreRuntimeStorageMode::Journaled`;
- `StoreRuntimeStorageCapabilities::journaled`;
- `DataStoreBackend::Journaled`;
- `IndexStoreBackend::Journaled`;
- `SchemaStoreBackend::Journaled`;
- generated `new_journaled` store wiring;
- registry accepted-store journaled handles;
- executor, relation, reconciliation, cursor, commit, or recovery concrete
  journaled policy branches.

Internal protocol-model names are allowed only if they cannot instantiate
runtime stores and are covered by source-audit tests.

Protocol tests may model bounded/fallible journal batches, but they must not
create production journal store codecs reachable from runtime store handles.

## Non-Goals

- No public `storage(journaled(...))` admission.
- No generated journaled runtime wiring.
- No runtime journaled store handles.
- No journaled data/index/schema backend construction.
- No durable behavior for `storage(heap())`.
- No checkpoint memories.
- No transparent heap-to-stable flush.
- No new SQL storage syntax.
- No migration tooling.
- No relation semantic expansion.
- No compatibility shims.

## 0.173.5 Closeout

0.173.5 closes the protocol foundation without admitting journaled runtime
storage.

Closed evidence:

- public `storage(journaled(...))` remains reserved through compile-fail UI
  fixtures;
- accepted schemas, generated wiring, registry handles, diagnostics rows,
  relation branches, runtime store handles, and store constructors remain
  stable/heap only;
- the embedded bounded journal batch payload strategy is recorded directly;
- monotonic `journal_sequence` is the chosen journal-tail ordering authority;
- the fold watermark is the durable replay boundary;
- schema replay and index materialization rules are specified before runtime
  admission;
- existing stable and heap capability, allocation, diagnostics, volatility,
  commit, recovery, and codec behavior remain covered by existing focused
  validation;
- 0.174 public admission gates are explicit and must not be bypassed.

0.173 does not introduce journaled row/index/schema codecs, a journal-tail
runtime store, commit-marker format changes, generated journaled wiring, or
runtime journaled capabilities.
