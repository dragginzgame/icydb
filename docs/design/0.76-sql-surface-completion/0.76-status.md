# 0.76 Status Tracker

Current state: active work in progress. The `0.76` line is focused on closing
the remaining non-JOIN SQL surface gaps that still fit the normalized
single-path execution model.

## Scope

This status tracks the active `0.76` SQL surface-completion line.

## Completed In The Current `0.76` Line

- [x] S0. Scalar `SELECT DISTINCT` without PK projection
  - Removed the SQL lowering gate that required primary-key projection for
    scalar `SELECT DISTINCT`.
  - Moved immediate SQL DISTINCT semantics to the projected-row boundary.
  - Cursorless SQL `LIMIT/OFFSET` now apply after projected-row dedup for that
    surface.

- [x] S1. Global aggregate DISTINCT qualifiers for `COUNT/SUM/AVG`
  - `COUNT(DISTINCT field)`, `SUM(DISTINCT field)`, and `AVG(DISTINCT field)`
    now parse, lower, explain, and execute on the global aggregate SQL lane.
  - The implementation reuses the existing aggregate runtime and applies
    distinctness at the SQL aggregate input boundary instead of widening the
    core aggregate kernel.

- [x] S2. Grouped `MIN(field)` / `MAX(field)`
  - Grouped SQL and planner admission now allow `MIN(field)` and `MAX(field)`
    in the admitted grouped cohorts.
  - Grouped runtime state stores and compares actual field values for grouped
    field-target extrema instead of rejecting them as unsupported.

- [x] S3. Ordered `DELETE ... OFFSET`
  - `DELETE ... ORDER BY ... LIMIT/OFFSET` now parses, lowers, and executes as
    one ordered delete window.
  - Delete planning now carries ordered delete window state instead of a
    limit-only delete bound.
  - Delete `LIMIT/OFFSET` continues to require explicit ordering.

- [x] S4. `MIN/MAX(DISTINCT field)` normalization
  - `MIN(DISTINCT field)` and `MAX(DISTINCT field)` now lower to the existing
    `MIN/MAX(field)` path instead of failing closed.
  - This keeps SQL behavior closer to standard expectations without creating a
    new aggregate runtime lane, because the DISTINCT qualifier is redundant for
    extrema.

- [x] S5. Grouped aggregate DISTINCT qualifiers for `COUNT/SUM/AVG`
  - `COUNT(DISTINCT field)`, `SUM(DISTINCT field)`, and `AVG(DISTINCT field)`
    now parse, lower, explain, and execute on the grouped aggregate SQL lane.
  - Grouped runtime distinctness for those functions now deduplicates aggregate
    input values per group instead of deduplicating by row identity.
  - The implementation keeps the existing grouped aggregate runtime and narrows
    the fix to grouped DISTINCT value tracking for the admitted field-target
    aggregate kinds.

- [x] S7. Grouped top-level `SELECT DISTINCT ... GROUP BY ...` normalization
  - Top-level grouped `DISTINCT` now parses, lowers, explains, and executes on
    the admitted grouped SQL lane.
  - SQL lowering normalizes the redundant top-level `DISTINCT` away for grouped
    projection shapes instead of routing through grouped DISTINCT adjacency
    policy or widening grouped runtime.

- [x] S8. Session-owned grouped computed text projection
  - `execute_sql_grouped(...)`, grouped `EXPLAIN`, and unified SQL dispatch now
    admit grouped text-function projection items when those functions wrap
    grouped fields before aggregate outputs.
  - The implementation rewrites those queries to the canonical grouped base
    select and transforms grouped key values only at the session boundary after
    the base grouped query runs.
  - Generic grouped lowering still stays fail-closed for this shape, so the
    change does not widen grouped planner/runtime expression ownership.

- [x] S9. Reduced SQL projection aliases at the session/output boundary
  - Reduced SQL now accepts `AS alias` and bare trailing aliases for projection
    items.
  - Alias capture stays parser-owned and only affects outward SQL column labels
    on dispatch/computed/grouped/global-aggregate surfaces.
  - Lowering, planning, execution, and `EXPLAIN` remain alias-neutral, so the
    change does not widen expression ownership or alter execution semantics.

- [x] S10. Narrow `ORDER BY` alias normalization
  - Reduced SQL now accepts `ORDER BY <alias>` when the alias resolves to one
    already-supported order target.
  - Field aliases normalize during shared SQL prepare/lowering, so scalar,
    grouped, and `EXPLAIN` surfaces stay planner/runtime-neutral.
  - Session-owned computed projection also rewrites `ORDER BY` aliases for
    `LOWER(field)` and `UPPER(field)` before the base-query rewrite, while
    broader computed or aggregate alias targets stay fail-closed.

- [x] S11. Reduced SQL writes on the typed dispatch lane
  - Reduced SQL now parses `INSERT INTO entity (field, ...) VALUES (...)` and
    `UPDATE entity SET field = literal [, ...] WHERE ...`.
  - SQL write execution stays on the typed `execute_sql_dispatch::<E>(...)`
    lane and reuses the existing structural mutation path instead of widening
    lowered query/runtime ownership.
  - The current admitted write shape stays intentionally bounded:
    - `INSERT` supports explicit-column single-row and multi-row tuples
    - `INSERT INTO entity VALUES (...)` positional forms are admitted on the
      typed dispatch lane
    - `INSERT` now also admits single-table aliases on the narrowed write lane
    - `INSERT ... SELECT` is now admitted on the typed dispatch lane when the
      source stays on the same scalar entity lane with:
      - field-only projection, or one admitted scalar computed text projection
      - deterministic primary-key-backed ordering
      - grouped, aggregate, and broader unsupported computed source shapes
        still stay fail-closed
    - schema insert-generated fields can now be omitted on:
      - explicit-column insert
      - positional insert whose tuple width matches the non-generated field set
    - current schema-owned generated insert support is intentionally narrow:
      - `generated(insert = "...")` currently admits single-value primitive `Ulid` and `Timestamp` fields
      - ordinary `default = ...` values do not grant SQL omission rights
      - conflicting `default = ...` and `generated(insert = "...")` contracts now fail at derive time
      - `generated(insert = ...)` cannot be combined with derive-managed write fields
      - typed-dispatch SQL now rejects explicit generated-field writes on both `INSERT` and `UPDATE`
      - public structural `Insert` / `Update` / `Replace` now reject explicit generated-field writes too
      - typed entity saves still do not enforce the same authored-value policy because typed-save provenance is not solved yet
    - typed authored create now has its own generated companion surface:
      - macro-generated `EntityCreate` is distinct from the readable entity shape
      - `create(...)` carries authored omission to save preflight explicitly
      - generated and managed fields are structurally absent from the create-authored type
      - the older full-entity `insert(...)` path still exists as the complete after-image write surface
    - `UPDATE` now supports reduced SQL predicate-selected rows and ordered
      windows on the typed dispatch lane, but still requires `WHERE ...`
    - single-table aliases are now admitted on the narrowed `UPDATE` surface
    - primary-key mutation remains fail-closed
    - `RETURNING` remains fail-closed across reduced SQL write lanes
  - The SQL write lane now also mirrors the derive-owned timestamp contract
    for schema entities by synthesizing `created_at` / `updated_at` on insert
    and refreshing `updated_at` on update.

## What `0.76` Has Proved So Far

- the normalized executor/runtime can absorb more SQL surface without
  reintroducing duplicate execution paths
- projected-row DISTINCT semantics can stay local to the SQL boundary instead
  of reopening planner-wide distinct execution complexity
- redundant top-level grouped `DISTINCT` can also stay local to SQL lowering
  instead of reopening grouped DISTINCT adjacency policy for unchanged grouped
  row shapes
- grouped computed text projection can also stay local to the session SQL
  boundary by rewriting onto the existing grouped base query and transforming
  only grouped key cells after execution
- projection aliases can stay local to parser/session output shaping without
  reopening planner/runtime alias ownership or changing execution plans
- `ORDER BY` aliases can also stay local to SQL prepare/session rewrite
  boundaries when they only normalize onto already-admitted order targets
- grouped field-target extrema fit the existing grouped runtime with a narrow
  state fix rather than a new grouping algorithm
- grouped DISTINCT field aggregates can stay inside the current grouped runtime
  when distinctness is tracked at the per-group aggregate-input boundary
- ordered SQL delete windows can be expressed as a delete-specific window
  contract instead of a parser-only exception
- reduced SQL writes can also stay local to the typed dispatch/session write
  boundary when the admitted shape is one explicit-column insert or one
  predicate-selected update over the existing structural mutation contract
- schema-owned field metadata can carry reduced SQL insert-generation policy
  without turning ordinary typed `default = ...` values into generic SQL
  omission semantics, and with one explicit generator allowlist (`Ulid::generate`
  and `Timestamp::now`) instead of a generic default fallback
- insert-generated fields can stay system-owned on the authored write surfaces
  that can already prove caller intent, while typed-save provenance remains a
  separate later design problem instead of being solved with heuristics

## Explicit Fail-Closed Boundaries

- generic projection-expression widening remains outside `0.76`
- no JOIN support

These boundaries are now both documented and covered:

- generic grouped projection-expression widening remains lowering-gated even
  though the session-owned grouped computed text lane is now admitted
- broader computed or aggregate `ORDER BY` aliases remain fail-closed even
  though field aliases and `LOWER/UPPER` aliases are now admitted on their
  narrower lanes
- JOIN remains parser-gated with stable unsupported-feature labels
