# 0.76 Status Tracker

Current state: active work in progress. The `0.76` line is focused on closing
the remaining non-JOIN SQL surface gaps that still fit the normalized
single-path execution model.

## Scope

This status tracks the active `0.76` SQL surface-completion line.

## Completed In The Current `0.76` Line

- [x] S0. Scalar `SELECT DISTINCT` without PK projection
  - Removed the SQL lowering gate that required primary-key projection for
    scalar `SELECT DISTINCT`.
  - Moved immediate SQL DISTINCT semantics to the projected-row boundary.
  - Cursorless SQL `LIMIT/OFFSET` now apply after projected-row dedup for that
    surface.

- [x] S1. Global aggregate DISTINCT qualifiers for `COUNT/SUM/AVG`
  - `COUNT(DISTINCT field)`, `SUM(DISTINCT field)`, and `AVG(DISTINCT field)`
    now parse, lower, explain, and execute on the global aggregate SQL lane.
  - The implementation reuses the existing aggregate runtime and applies
    distinctness at the SQL aggregate input boundary instead of widening the
    core aggregate kernel.

- [x] S2. Grouped `MIN(field)` / `MAX(field)`
  - Grouped SQL and planner admission now allow `MIN(field)` and `MAX(field)`
    in the admitted grouped cohorts.
  - Grouped runtime state stores and compares actual field values for grouped
    field-target extrema instead of rejecting them as unsupported.

- [x] S3. Ordered `DELETE ... OFFSET`
  - `DELETE ... ORDER BY ... LIMIT/OFFSET` now parses, lowers, and executes as
    one ordered delete window.
  - Delete planning now carries ordered delete window state instead of a
    limit-only delete bound.
  - Delete `LIMIT/OFFSET` continues to require explicit ordering.

- [x] S4. `MIN/MAX(DISTINCT field)` normalization
  - `MIN(DISTINCT field)` and `MAX(DISTINCT field)` now lower to the existing
    `MIN/MAX(field)` path instead of failing closed.
  - This keeps SQL behavior closer to standard expectations without creating a
    new aggregate runtime lane, because the DISTINCT qualifier is redundant for
    extrema.

- [x] S5. Grouped aggregate DISTINCT qualifiers for `COUNT/SUM/AVG`
  - `COUNT(DISTINCT field)`, `SUM(DISTINCT field)`, and `AVG(DISTINCT field)`
    now parse, lower, explain, and execute on the grouped aggregate SQL lane.
  - Grouped runtime distinctness for those functions now deduplicates aggregate
    input values per group instead of deduplicating by row identity.
  - The implementation keeps the existing grouped aggregate runtime and narrows
    the fix to grouped DISTINCT value tracking for the admitted field-target
    aggregate kinds.

## What `0.76` Has Proved So Far

- the normalized executor/runtime can absorb more SQL surface without
  reintroducing duplicate execution paths
- projected-row DISTINCT semantics can stay local to the SQL boundary instead
  of reopening planner-wide distinct execution complexity
- grouped field-target extrema fit the existing grouped runtime with a narrow
  state fix rather than a new grouping algorithm
- grouped DISTINCT field aggregates can stay inside the current grouped runtime
  when distinctness is tracked at the per-group aggregate-input boundary
- ordered SQL delete windows can be expressed as a delete-specific window
  contract instead of a parser-only exception

## Explicit Fail-Closed Boundaries

- top-level grouped `SELECT DISTINCT ... GROUP BY ...` remains outside the
  current `0.76` line
- generic projection-expression widening remains outside `0.76`
- no JOIN support

These boundaries are now both documented and covered:

- grouped top-level DISTINCT remains lowering-gated
- grouped projection-expression widening remains lowering-gated
- JOIN remains parser-gated with stable unsupported-feature labels
