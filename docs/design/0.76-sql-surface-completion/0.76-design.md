# 📄 0.76 Design: SQL Surface Completion (Pre-JOIN Coherence)

## Preamble

* **scope**: complete and normalize the SQL surface for all non-JOIN functionality
* **goal**: remove inconsistencies between SQL and fluent/query APIs, and eliminate unintuitive restrictions in DISTINCT and grouped aggregation
* **non-goals**:

  * no JOIN support
  * no broad projection-expression widening
  * no expansion of execution engine capabilities beyond existing structural primitives
  * no regression in execution-kernel simplicity or single-pass guarantees

---

## Problem Statement

The execution engine is now structurally normalized and coherent. However, the SQL surface remains **incomplete and inconsistent** in several areas:

### High-impact inconsistencies

* SQL does not support `DISTINCT` semantics uniformly
* SQL does not support aggregate `DISTINCT` qualifiers (e.g. `COUNT(DISTINCT x)`)
* grouped `MIN(field)` and `MAX(field)` are unsupported despite similar aggregates being supported

### Medium-impact limitations

* grouped projection rules are stricter than expected SQL semantics
* `DISTINCT + GROUP BY` combinations are heavily restricted

### Low-impact omissions

* no SQL `INSERT`
* no `DELETE ... OFFSET`

---

## Design Goals

1. **Surface coherence**

   * SQL and fluent/query APIs must expose the same logical capabilities

2. **Predictable semantics**

   * SQL behavior should match standard expectations where feasible

3. **No duplication of execution paths**

   * all new functionality must reuse:

     * existing kernel row scan
     * existing projection/materialization
     * existing grouped runtime

4. **Preserve structural simplicity**

   * no reintroduction of parallel execution flows
   * maintain single-pass execution

---

# Projection Expression Boundary

## Current Boundary

The current product surface keeps a deliberate split between:

* structural planner-owned projection
* one bounded session-owned computed projection lane

That means:

* `query_from_sql(...)` remains structural-only
* dispatch/session SQL may continue to expose the already-bounded computed text
  projection family
* this line does **not** reopen generic planner-owned arithmetic, boolean, or
  computed select-list expression support

## 0.76 Decision

`0.76` completes and normalizes the non-JOIN SQL surface that already fits the
current structural executor model. It does **not** widen the planner/query
surface to admit general projection expressions.

If generic projection expressions are reconsidered later, that must be a
separate design slice with its own planner, fingerprint, validation, and
execution authority review. It is not implicit follow-up work inside `0.76`.

# Part 1: SQL DISTINCT Normalization

## Current Issue

* `SELECT DISTINCT` is restricted to projections that include the primary key
* behavior is unintuitive and diverges from standard SQL

## Design

### 1.1 Remove PK projection requirement

Allow:

```sql
SELECT DISTINCT age FROM users;
```

### 1.2 Execution model

* DISTINCT is implemented as:

  * projection → deduplication over projected rows
* reuse existing:

  * slot-row projection
  * row equality semantics

### 1.3 Implementation approach

* introduce a **post-projection dedup stage**
* operate on:

  * projected slot rows (preferred)
  * or materialized values (fallback)

### 1.4 Constraints

* no requirement for PK presence
* ordering must still be explicit:

  * DISTINCT + ORDER BY must obey existing ordering constraints

---

# Part 2: SQL Aggregate DISTINCT

## Current Issue

* SQL rejects:

  ```sql
  SELECT COUNT(DISTINCT age) FROM users;
  ```
* fluent/query APIs support equivalent behavior

## Design

### 2.1 Add DISTINCT qualifier to aggregates

Supported:

```sql
COUNT(DISTINCT x)
SUM(DISTINCT x)
AVG(DISTINCT x)
```

### 2.2 Execution model

* reuse existing aggregate runtime
* add a **distinct-input adapter** per aggregate

### 2.3 Implementation approach

* per-aggregate:

  * wrap input stream with a dedup set
* no change to:

  * grouping logic
  * fold structure

### 2.4 Constraints

* DISTINCT applies **per aggregate input**
* not a global DISTINCT

---

# Part 3: Grouped MIN/MAX(field)

## Current Issue

* grouped `MIN(field)` / `MAX(field)` explicitly unsupported

## Design

### 3.1 Enable grouped MIN/MAX(field)

Allow:

```sql
SELECT age, MIN(score) FROM users GROUP BY age;
```

### 3.2 Execution model

* reuse grouped runtime
* extend reducer set with:

  * field-target min reducer
  * field-target max reducer

### 3.3 Implementation approach

* same structure as SUM/AVG:

  * per-group state
  * update on each row

### 3.4 Constraints

* no new grouping algorithm
* no change to candidate selection logic

---

# Part 4: DISTINCT + GROUP BY

## Current Issue

* combinations like:

  * grouped DISTINCT
  * DISTINCT aggregates in grouped context
    are heavily restricted

## Design

### 4.1 Allow DISTINCT inside grouped aggregates

```sql
SELECT age, COUNT(DISTINCT score) FROM users GROUP BY age;
```

### 4.2 Execution model

* DISTINCT applies **within each group**
* each group maintains its own dedup set

### 4.3 Constraints

* no global DISTINCT + GROUP BY mixing
* no cross-group deduplication

---

# Part 5: Grouped Projection Rules

## Current Issue

* SQL requires:

  * group keys first
  * exact projection match

Example rejected:

```sql
SELECT COUNT(*) FROM users GROUP BY age;
```

## Design

### 5.1 Relax projection constraints

Allow:

* projections that include:

  * aggregates
  * optional group keys

### 5.2 Execution model

* grouping remains unchanged
* projection stage determines output shape

### 5.3 Constraints

* non-aggregated fields must still:

  * be part of GROUP BY

---

# Part 6: Lower-Priority Additions

## 6.1 SQL INSERT (optional)

* map directly to existing typed save API
* minimal SQL syntax support

## 6.2 DELETE ... OFFSET (optional)

* extend delete planner
* reuse pagination window logic

---

# Architectural Constraints

All additions must:

* reuse:

  * `execute_kernel_row_scan`
  * projection/materialize/structural.rs
  * grouped runtime
* maintain:

  * single-pass execution
  * no per-row branching
  * no duplicated loops
