# 0.146 Schema Identity & Layout Authority Status

## Status

In progress. The foundational identity and generated-memory boundary are now
implemented, the reserved schema memory has a concrete raw metadata map plus a
typed persisted-schema snapshot codec, and startup now writes or verifies the
initial generated schema snapshot for each registered entity.

## Current Implementation State

0.146 starts from the cleaned storage boundary left by 0.144 and the streaming
cleanup completed in 0.145.

The first implementation work established schema identity as an explicit
runtime owner before introducing a durable `__icydb_schema` map:

* separate durable field identity from generated field slot order
* keep generated `EntityModel` as the compiled proposal, not the live authority
* make database-level defaults explicit instead of inferring them from Rust
  `Default`
* prepare a persisted schema snapshot shape that can later become the
  `__icydb_schema` row
* reserve a third generated stable memory for each store so schema metadata has
  a dedicated persistence boundary before reconciliation is implemented
* back that memory with a raw schema metadata map keyed by entity tag and
  schema version
* define the typed persisted-schema snapshot codec stored inside raw schema
  map values
* reconcile registered runtime hooks against those stored snapshots at the
  recovery boundary, accepting first-create or exact initial-schema matches

## Completed Foundation

The current code has:

* `FieldId` as the durable logical field identity.
* `SchemaVersion`, `SchemaFieldSlot`, and `SchemaRowLayout` as schema-owned
  layout artifacts.
* `CompiledSchemaProposal` and `CompiledFieldProposal` as the generated
  `EntityModel` / `FieldModel` bridge.
* `PersistedSchemaSnapshot` and `PersistedFieldSnapshot` as the initial owned
  shape for future persisted schema metadata.
* `SchemaFieldDefault::None` as an explicit placeholder that prevents Rust
  struct defaults from being treated as database defaults.
* `FieldDatabaseDefault` as the runtime-model source for database-level
  defaults, currently `None` only, so future default support has an explicit
  schema contract instead of reusing Rust construction defaults.
* Derive-generated `FieldModel` construction now emits a database-default
  contract explicitly. The emitted value is still `FieldDatabaseDefault::None`,
  so Rust construction defaults remain separate from database persistence
  defaults.
* `SchemaStore` as the per-store stable-memory boundary for future schema
  metadata.
* `RawSchemaKey`, `RawSchemaSnapshot`, and a concrete `SchemaStore` BTreeMap as
  the raw `__icydb_schema` persistence boundary.
* A schema-owned persisted snapshot codec that maps
  `PersistedSchemaSnapshot` to/from `RawSchemaSnapshot` without making the
  internal DTOs public wire types.
* Typed `SchemaStore` insert/load entrypoints so callers do not need to handle
  raw schema payload encoding directly.
* A schema-owned startup reconciliation pass that writes missing initial
  snapshots and fails closed when an existing snapshot no longer matches the
  compiled proposal.
* Startup reconciliation loads the latest stored schema snapshot for each
  entity instead of hard-coding the initial version lookup, so future-version
  metadata cannot be ignored under the current exact-match policy.
* Latest-version schema lookup fails closed when the newest payload is corrupt
  instead of falling back to an older decodable snapshot.
* Schema-store writes reject typed snapshots whose snapshot version and
  row-layout version diverge, keeping persisted metadata internally coherent
  before transition support exists.
* Schema codec decoding rejects raw persisted snapshots whose snapshot version
  and row-layout version diverge, so hostile or stale raw metadata cannot enter
  the accepted snapshot path.
* Schema-store writes and schema-codec decoding reject snapshots whose
  persisted field slot metadata diverges from the row-layout mapping, keeping
  the row-layout authority coherent with the duplicated field metadata still
  present in the snapshot format.
* Schema-store writes and schema-codec decoding reject snapshots whose
  primary-key field identity is absent from the row-layout or persisted field
  metadata.
* Schema metadata integrity checks reject ambiguous accepted-schema inputs,
  including duplicate row-layout field IDs, duplicate row-layout slots,
  duplicate persisted field IDs, and duplicate persisted field names.
* Nested leaf metadata integrity checks reject empty nested paths and duplicate
  nested paths under the same top-level field before accepted field-path
  inference can become order-dependent.
* `db::schema::transition` as the owner of schema transition policy, with
  `SchemaTransitionDecision` as the explicit reconciliation result. The only
  accepted decision is still exact initial-schema equality, but future
  migration work now has one decision boundary to extend.
* `AcceptedSchemaSnapshot` as the typed handoff for snapshots that have passed
  reconciliation, separating decoded schema payloads from accepted live
  authority at call sites.
* Accepted live schema construction now uses a fallible production constructor
  that runs schema metadata integrity checks before wrapping a persisted
  snapshot as accepted.
* Accepted-schema entity and field lookup helpers, so metadata consumers can
  source entity headers plus top-level slot, field-kind, and queryability facts
  from accepted persisted snapshot fields.
* Accepted-schema grouped field fact lookup, so `SchemaInfo` and accepted
  DESCRIBE formatting can receive persisted kind, row-layout slot, and
  nested-leaf descriptors through one schema-owned handoff instead of
  reassembling accepted field facts from separate lookups.
* Accepted-schema field fact accessors for primary-key kind, top-level field
  kind, top-level slot, and nested leaf metadata, with raw persisted snapshot
  access narrowed to the schema module.
* Accepted-schema top-level field slot lookup now resolves through
  `SchemaRowLayout`, so the persisted row-layout mapping is the named authority
  for slot answers even though exact-match reconciliation still keeps
  field-snapshot slot values identical today.
* Accepted-schema `SchemaInfo` carries top-level slot indexes sourced from that
  accepted row-layout mapping, and grouped structural validation now checks
  grouped field slots through `SchemaInfo` instead of direct generated-model
  field-table lookup. The grouped structural validation sub-boundary no longer
  accepts a generated model parameter for that slot-validity check.
* Prepared execution metadata now resolves direct field slots and field-path
  root slots through the selected `SchemaInfo`, so accepted row-layout
  authority feeds scalar projection, projection reachability, direct retained
  projection slots, duplicate-preserving data-row direct projection slots,
  residual-filter, order-expression, grouped aggregate expression, and prepared
  index predicate execution handoffs.
* Executor-core prepared-plan lowering now preserves an already-finalized
  accepted-schema static execution shape instead of re-finalizing through the
  generated-model fallback.
* Reconciliation diagnostics that report the first rejected schema difference
  while keeping the exact-match policy unchanged.
* A plain Candid `slot: u16` field on `EntityFieldDescription`, with the Rust
  `slot()` accessor preserving `Option<u16>` semantics for nested fields that
  are not top-level persisted row slots.
* Fallible SQL metadata paths (`DESCRIBE` / `SHOW COLUMNS`) now load the
  accepted initial persisted schema snapshot and use it for entity headers plus
  top-level slot, kind, and queryability values.
* A schema-owned SQL capability projection over `PersistedFieldKind`, covering
  selectable, comparable, orderable, groupable, and aggregate-input facts.
  Top-level SQL projection, grouping, direct ordering, and global aggregate
  field-target admission now consume this accepted-schema projection.
* A schema-owned `PersistedFieldKind` to planner `FieldType` projection, used
  by accepted metadata formatting and available for the later live-schema
  validation switch.
* A schema-owned strict SQL literal canonicalization path for
  `PersistedFieldKind`, now used by SQL INSERT/UPDATE literal validation.
* Session-cached SQL/fluent query planning now prepares its top-level
  `SchemaInfo` from the accepted schema snapshot, so cached plan validation can
  consume accepted persisted field types instead of only the generated model
  cache.
* Shared SQL/fluent prepared-plan cache keys and SQL compiled-command cache
  keys now use a cache-only accepted schema fingerprint derived from the
  accepted persisted schema snapshot plus the generated index contract.
* Top-level SQL/fluent expression type inference now reads the accepted-schema
  `FieldType` from `SchemaInfo`; nested field-path inference reads accepted
  nested leaf descriptors when snapshots carry them.
* Session SQL SELECT/DELETE compilation, SQL UPDATE target selectors, and
  `INSERT ... SELECT` source selectors now canonicalize top-level
  predicate/filter literals through accepted-schema-derived `SchemaInfo`
  instead of generated `FieldKind` lookup.
* SQL SELECT projection, `GROUP BY`, SELECT/DELETE/UPDATE direct `ORDER BY`,
  and global aggregate field-target admission now consume schema-owned SQL
  capabilities from the accepted-schema-derived `SchemaInfo`.
* Persisted schema snapshots now carry nested leaf descriptors for generated
  structured-record fields, and field-path expression type inference reads
  those descriptors from accepted schema metadata when present.
* SQL SELECT projection admission now checks accepted nested leaf SQL
  capabilities when projection expressions reference field paths.
* Generated `#[store(...)]` declarations now require `schema_memory_id`.
* Generated canister wiring now creates `DataStore`, `IndexStore`, and
  `SchemaStore` statics and registers them as one store triplet.
* Schema validation now checks each store's schema memory ID for range,
  reserved-ID, within-store distinctness, and canister-wide uniqueness.
* Commit schema fingerprints now flow through the compiled schema proposal,
  preserving the current fingerprint contract while moving ownership toward
  the schema layer.

## Explicitly Not Live Yet

The third stable memory is now backed by a raw schema metadata BTreeMap, and
raw values carry typed persisted schema snapshots. Startup reconciliation
currently supports only first-create and exact generated initial-schema
matches. It does not yet support schema evolution or live runtime schema
authority.

The current runtime still reads generated static row-layout metadata, and
nested field-path execution still uses generated nested traversal contracts.
That is intentional until the accepted persisted schema can become live
row-layout authority for read/write decoding and runtime path contracts.

Infallible Rust metadata helpers remain generated-model views by design.
Fallible `try_*` metadata helpers are the accepted-schema counterparts for SQL
and diagnostics surfaces that can return reconciliation errors. This keeps the
Rust metadata API non-failing until live row-layout authority has a complete,
non-fallible access story.

## Next Correctness Boundary

The next implementation boundary is safe schema transition policy and live
layout authority:

1. Define explicit safe transitions beyond exact initial-schema equality.
2. Add fail-closed diagnostics that explain rejected changes.
3. Expose the accepted live schema as the authority for later read/write work.
4. Move row decoding, runtime nested path contracts, and any remaining
   expression-result capability checks through that accepted live schema instead
   of generated static metadata.

No row-format compatibility branch, runtime read remapping, or SQL capability
switch should be introduced before this reconciliation boundary exists.

## Guardrails

Do not add schema transitions that bypass the schema-owned reconciliation
boundary.

Do not add row-format compatibility branches or per-row schema versions in this
minor version.

Do not treat struct defaults emitted by derive code as database defaults.

Do not let generated `EntityModel` silently override a persisted live schema
once the schema map exists.

Do not route runtime row-layout decisions through persisted schema data until
that persisted schema has become the accepted live row-layout authority.
Top-level SQL/query type checks may use accepted schema metadata after startup
reconciliation has accepted the snapshot.

Commit/replay schema fingerprints remain tied to the generated compiled schema
proposal until migration policy exists. Runtime cache fingerprints may follow
the accepted schema snapshot because they only control in-memory invalidation.

Infallible Rust metadata helpers may expose generated physical slot numbers
until accepted live row layout authority has a non-fallible API. Fallible SQL
metadata paths must source top-level slot numbers from the accepted schema
snapshot.
