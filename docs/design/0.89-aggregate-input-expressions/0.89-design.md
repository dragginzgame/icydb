# 0.89 Aggregate Input Expression Widening

## Thesis

`0.89` should allow aggregate functions to consume admitted scalar
expressions as inputs, so queries like:

```sql
SELECT class_name, AVG(strength + agility)
FROM character
GROUP BY class_name
ORDER BY AVG(strength + agility) DESC
LIMIT 10;
```

work through the existing scalar expression model instead of requiring field-only
aggregate inputs or awkward rewrites.

The core mechanism is:

> evaluate aggregate input expressions per input row using the shared scalar
> evaluator, then feed the resulting `Value` into the existing aggregate
> accumulators.

This must not introduce a second input-expression runtime, a grouped-only
expression family, or aggregate-specific expression semantics.

---

## Problem statement

After `0.85–0.88`, the system can:

* compute post-aggregate expressions in grouped projection
* filter on post-aggregate expressions in grouped `HAVING`
* order grouped output by aggregate-based expressions with bounded Top-K

But aggregate inputs are still too narrow.

Users expect query shapes like:

```sql
AVG(strength + agility)
SUM(level * 2)
MAX(ROUND(strength / 2, 0))
COUNT(strength + 1)
```

These are all semantically ordinary single-entity SQL queries, but today they
are either rejected or require projection-layer rewrites that do not match
normal SQL expectations.

This leaves a major remaining gap in the single-entity SQL surface even after
aggregate ordering lands.

---

## Design goal

Allow aggregate functions to consume the same scalar expression family already
admitted by the shared scalar evaluator.

That means:

* planner-owned aggregate input expressions
* one executor-prepared compiled scalar program for aggregate inputs
* per-row scalar evaluation before aggregation
* existing aggregate accumulators stay the only owners of aggregation semantics

The goal is to widen SQL expressiveness without creating a special aggregate
expression subsystem.

---

## Non-goals

This slice does not aim to:

* support nested aggregates
* support post-aggregate expressions as aggregate inputs
* support subqueries or window functions inside aggregate inputs
* widen `DISTINCT aggregate(expr)` in the same patch
* add aggregate `FILTER (...)` clauses
* add a second scalar evaluator path for aggregate inputs

This is aggregate input widening, not aggregate feature generalization.

---

## Query families enabled by this slice

### Global aggregates

```sql
SELECT AVG(strength + agility)
FROM character;
```

```sql
SELECT MAX(ROUND(level / 2, 0))
FROM character;
```

### Grouped aggregates

```sql
SELECT class_name, AVG(strength + agility)
FROM character
GROUP BY class_name;
```

```sql
SELECT class_name, SUM(level * 2)
FROM character
GROUP BY class_name
ORDER BY SUM(level * 2) DESC
LIMIT 10;
```

```sql
SELECT class_name, COUNT(strength + 1)
FROM character
GROUP BY class_name
HAVING COUNT(strength + 1) > 5;
```

---

## Expression admissibility

Aggregate input expressions are only allowed when they are expressible over the
existing scalar expression family already admitted by the shared scalar
evaluator.

Allowed leaves:

* model fields
* literals

Allowed composition:

* arithmetic
* scalar functions already supported by the shared scalar evaluator
* nested scalar expressions built from those same admitted parts

Rejected:

* aggregate calls inside aggregate input expressions
* grouped/post-aggregate expressions
* subqueries
* window expressions
* any scalar function family not already admitted by the shared scalar
  expression model

This keeps aggregate input widening inside the existing scalar-expression
admissibility boundary instead of opening a second, broader expression policy.

---

## Aggregate-family semantics

This slice applies to:

* `SUM(expr)`
* `AVG(expr)`
* `MIN(expr)`
* `MAX(expr)`
* `COUNT(expr)`

The evaluation rule is:

```text
row -> evaluate scalar expr -> feed result into aggregate accumulator
```

Specific notes:

* `COUNT(expr)` counts only rows where the evaluated expression result is
  non-null
* `SUM` / `AVG` / `MIN` / `MAX` reuse existing accumulator semantics after
  expression evaluation
* aggregate input expressions do not introduce new coercion rules; they reuse
  the existing scalar expression typing/coercion behavior before the aggregate
  sees the resulting value

Out of scope for `0.89`:

* `DISTINCT aggregate(expr)`

That should remain explicitly rejected or unchanged until a separate slice owns
its semantics and planner/runtime cost.

---

## Planner model

### Aggregate input becomes expression-owned

The planner should no longer treat aggregate input as field-only.

Conceptually:

```rust
enum AggregateInputExpr {
    Field(...),
    ScalarExpr(...),
}
```

or the equivalent existing planner-owned scalar expression type.

The important rule is ownership:

* the logical plan stores planner-owned aggregate input expressions
* the executor prepares compiled scalar programs from that planner-owned form
* logical planning must not store executor-owned compiled scalar IR directly

---

## Executor model

### Compile once, evaluate per row

Aggregate input expressions should be compiled once per query execution onto the
existing shared scalar evaluator path.

Execution model:

```text
scan
→ evaluate aggregate input expr per row
→ feed result into aggregate accumulator
→ finalize
```

This is true for:

* grouped aggregates
* global aggregates

Important boundary rule:

* do not introduce a special aggregate-input evaluator
* aggregate input execution must reuse the same scalar compiled expression path
  already used elsewhere in the engine

---

## Interaction with existing slices

### Projection (`0.85`)

Projection widening stays post-aggregate.

`0.89` is a different phase:

* projection expressions: post-aggregate, grouped-row phase
* aggregate input expressions: pre-aggregate, per-row scalar phase

### HAVING (`0.86`)

No semantic change.

Grouped `HAVING` continues to run after aggregation.

`0.89` only changes what aggregate outputs can be computed in the first place.

### Aggregate ORDER BY (`0.88`)

This slice composes directly with `0.88`.

Example:

```sql
SELECT class_name, AVG(strength + agility)
FROM character
GROUP BY class_name
ORDER BY AVG(strength + agility) DESC
LIMIT 10;
```

works as:

* `0.89`: aggregate input expression widening
* `0.88`: bounded Top-K aggregate ordering

---

## Validation rules

Add explicit planner checks:

```text
aggregate input expression must not contain:
  - aggregate functions
  - grouped/post-aggregate expressions
  - subqueries
  - unsupported scalar functions
```

Error behavior should stay specific.

Examples:

* nested aggregate input should be rejected as nested aggregate misuse
* unsupported scalar function should be rejected as unsupported expression shape
* post-aggregate dependency should fail closed as invalid aggregate input phase

---

## Explain / fingerprint / cache follow-through

This slice changes logical query shape.

So `0.89` must update:

* explain output for aggregate expressions
* fingerprint hashing for aggregate-input expression structure
* structural cache identity for aggregate-input expression structure

This is not optional follow-up.

Without it, the feature lands semantically but leaves stale identity and
observability surfaces behind.

---

## Guardrails

### 1. No new evaluator path

Aggregate input widening must reuse the shared scalar evaluator.

### 2. No grouped-phase evaluation

Aggregate input expressions must be evaluated:

* per input row
* before aggregation

Never on finalized grouped rows.

### 3. No DISTINCT widening in the same slice

Keep `DISTINCT aggregate(expr)` out of `0.89`.

### 4. Fail closed on unsupported constructs

No fallback rewrites.

If an aggregate input expression is not admitted, reject it explicitly.

---

## Acceptance criteria

### Functional

* global aggregates accept admitted scalar input expressions
* grouped aggregates accept admitted scalar input expressions
* `COUNT(expr)` uses evaluated non-null result semantics
* nested aggregates are rejected
* unsupported scalar-expression shapes remain rejected

---

### Architectural

* planner stores planner-owned aggregate input expressions
* executor compiles aggregate input expressions once per execution
* aggregate input evaluation reuses the shared scalar evaluator
* no duplicate evaluator path is introduced
* explain, fingerprint, and cache identity reflect aggregate-input expression
  shape

---

### Performance

* aggregate input expression compilation happens once per execution
* per-row evaluation cost is limited to the shared scalar evaluator path
* no extra grouped-only evaluation layer is introduced

---

## Follow-on

Potential `0.90` closeout candidates after `0.89`:

* remaining single-entity SQL polish and consistency gaps
* final explain / error-surface parity cleanup
* decision on whether `DISTINCT aggregate(expr)` deserves its own slice or
  remains intentionally unsupported

---

## Thesis sentence

> **0.89 enables aggregate input expressions by allowing aggregates to consume admitted scalar expressions, evaluated per row through the shared scalar evaluator, without introducing new execution paths or widening aggregate semantics beyond safe single-entity bounds.**
