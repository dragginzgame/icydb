# 0.85 Grouped Projection Expression Widening

## Thesis

`0.85` should widen grouped SQL projection from the current **group-key plus aggregate leaf** model to a **post-aggregate expression model**.

The goal is not to “add `ROUND(AVG(...))`” as a one-off syntax patch.

The real target is:

> allow bounded scalar expression evaluation over already-computed grouped outputs, while preserving the current grouped execution invariants.

That means:

* keep grouping and aggregate accumulation unchanged
* keep grouped `ORDER BY` boundedness rules unchanged
* keep aggregate ownership in the grouped executor
* add a new **post-aggregate expression layer** for grouped projection

The first motivating examples are:

```sql
SELECT class_name, ROUND(AVG(strength), 4)
FROM character
GROUP BY class_name
ORDER BY class_name
LIMIT 100;
```

```sql
SELECT class_name, AVG(strength) * 2
FROM character
GROUP BY class_name
ORDER BY class_name
LIMIT 100;
```

```sql
SELECT class_name, COUNT(*) + 1
FROM character
GROUP BY class_name
ORDER BY class_name
LIMIT 100;
```

---

## Problem statement

Current grouped SQL support is intentionally narrow.

Grouped projection today is effectively limited to:

* grouped key fields
* aggregate leaves

That boundary is correct for `0.79`-style grouped execution stabilization, but it leaves a visible gap:

* aggregate results can be projected
* aggregate results can be filtered in `HAVING`
* aggregate results cannot be composed into scalar expressions in projection

So queries like:

```sql
SELECT class_name, AVG(strength)
FROM character
GROUP BY class_name
HAVING AVG(strength) >= 10
ORDER BY class_name
LIMIT 100;
```

work, while queries like:

```sql
SELECT class_name, ROUND(AVG(strength), 4)
FROM character
GROUP BY class_name
ORDER BY class_name
LIMIT 100;
```

fail at parse/lowering because the engine still treats grouped projection as a special-case aggregate surface rather than a general post-aggregate expression surface.

This is now the main expressiveness gap in grouped SQL.

---

## Architectural position

### Keep aggregate execution separate from projection expression evaluation

The grouped executor should continue to own:

* grouping
* aggregate state accumulation
* aggregate finalization
* grouped row production

`0.85` should not move aggregate semantics into the generic scalar evaluator.

Instead, the grouped executor should produce a **group output row**, and only then should scalar expression evaluation run over that output.

The execution shape becomes:

```text
scan -> group -> finalize aggregate outputs -> evaluate grouped projection expressions -> emit
```

Not:

```text
scan -> generic expression eval -> group -> generic expression eval
```

This is the central design constraint.

---

### Do not widen grouped execution into a full arbitrary SQL expression engine

`0.85` is not a general “SQL expressions everywhere” slice.

It is specifically:

* post-aggregate grouped projection expression widening
* over already-valid grouped keys and aggregate outputs

Out of scope for this slice:

* aggregate arguments that are themselves arbitrary nested expressions beyond current support
* window functions
* joins
* subqueries
* generic scalar function support in every SQL clause
* relaxing grouped boundedness rules

---

### Preserve grouped boundedness invariants

The current grouped rules stay intact:

* grouped `ORDER BY` still requires `LIMIT`, unless separately relaxed in a later slice
* grouped `ORDER BY` must still respect grouped-key prefix rules
* grouped execution should not gain new implicit full-sort or full-materialization behavior

`0.85` is an expression-layer slice, not an execution-strategy slice.

---

## Scope

### In scope

* grouped projection expressions over:

  * grouped key references
  * finalized aggregate outputs
  * scalar functions already supported on scalar projection lanes where semantically valid
  * arithmetic over finalized aggregate outputs where semantically valid
* grouped `ORDER BY` / `HAVING` reuse of those widened grouped output expressions if the implementation can support it cleanly
* grouped validation and lowering updates needed to model these expressions explicitly
* executor support for evaluating projection expressions against finalized grouped output rows

### Out of scope

* changing aggregate accumulation semantics
* changing grouped execution boundedness guarantees
* changing storage or binary serialization layers
* joins
* window functions
* subqueries
* full general expression grammar redesign across all SQL surfaces
* unbounded grouped sort support without `LIMIT`

---

## Current state

Today grouped projection is intentionally fail-closed around a narrow expression set:

* grouped keys
* aggregate leaves

The grouped path does not yet support:

* function-over-aggregate
* arithmetic-over-aggregate
* general nested grouped projection expressions

That is why a query like:

```sql
SELECT class_name, ROUND(AVG(strength), 4)
FROM character
GROUP BY class_name
ORDER BY class_name
LIMIT 100;
```

fails today even though:

```sql
SELECT class_name, AVG(strength)
FROM character
GROUP BY class_name
HAVING AVG(strength) >= 10
ORDER BY class_name
LIMIT 100;
```

works.

The current grouped execution model is already correct and should remain the base.

---

## Design goal

### Introduce a grouped post-aggregate expression model

The engine should gain an explicit grouped expression layer with these properties:

1. grouped keys remain addressable as projection inputs
2. finalized aggregate outputs become addressable as projection inputs
3. scalar expression evaluation can run over those inputs after aggregation is complete
4. grouped execution itself remains unchanged below that boundary

Conceptually:

```text
GroupedSourceRow {
  group_key_slots...
  aggregate_output_slots...
}
```

Then grouped projection expressions are evaluated against that row.

This allows:

* `ROUND(AVG(strength), 4)`
* `AVG(strength) * 2`
* `COUNT(*) + 1`
* `class_name || '-' || level` if/when grouped string composition is supported

without changing grouping mechanics.

---

## Proposed model

### New grouped expression input classes

Grouped expressions should resolve only against two authorities:

1. **Grouped key inputs**

   * fields present in the `GROUP BY` key
   * directly projected from the finalized grouped row

2. **Aggregate output inputs**

   * finalized aggregate results
   * e.g. `COUNT(*)`, `AVG(strength)`, `MAX(hit_points)`

No grouped expression in this slice should resolve directly back to raw source row fields outside the grouped key.

That means this remains invalid:

```sql
SELECT class_name, name, COUNT(*)
FROM character
GROUP BY class_name;
```

and this also remains invalid:

```sql
SELECT class_name, ROUND(name, 2)
FROM character
GROUP BY class_name;
```

because `name` is not a grouped key and not an aggregate output.

---

### Grouped expression evaluation boundary

Evaluation should occur only after grouped aggregation is finalized.

So:

* `AVG(strength)` is computed by grouped aggregation
* `ROUND(..., 4)` is applied after finalization
* arithmetic or scalar functions over aggregates are post-aggregate operations

This preserves the current authority split:

* grouped executor owns aggregate math
* scalar evaluator owns scalar function application over finalized values

---

### Expression support in v1

`0.85 v1` should support:

#### Projection

* grouped key reference
* aggregate leaf reference
* scalar function over grouped key or aggregate output
* arithmetic over grouped key or aggregate output
* nested scalar expressions whose leaves are grouped keys or aggregate outputs

Examples intended to work:

```sql
SELECT class_name, ROUND(AVG(strength), 4)
FROM character
GROUP BY class_name
ORDER BY class_name
LIMIT 100;
```

```sql
SELECT class_name, AVG(strength) * 2
FROM character
GROUP BY class_name
ORDER BY class_name
LIMIT 100;
```

```sql
SELECT class_name, COUNT(*) + MAX(level)
FROM character
GROUP BY class_name
ORDER BY class_name
LIMIT 100;
```

#### HAVING

For v1, keep `HAVING` conservative unless reuse falls out naturally from the same expression model.

Two acceptable choices for v1:

* **Option A, narrower:** keep `HAVING` aggregate/key predicates only
* **Option B, broader:** allow the same widened grouped expressions inside `HAVING`

Recommendation: start with **Option A** unless Option B comes nearly for free from the lowering model.

#### ORDER BY

For v1, preserve current grouped `ORDER BY` constraints.

If widened grouped expressions in `ORDER BY` complicate boundedness analysis, defer them.

Recommendation: grouped `ORDER BY` continues to operate on the current grouped ordering rules first; widened projection support is the main goal.

## Parser changes

### Current problem

The current parser shape for some functions is bounded around scalar forms and expects identifiers or limited arithmetic forms, not aggregate expressions as arguments.

### Target

Introduce a grouped-aware SQL expression shape that can represent:

* aggregate leaves
* function calls wrapping aggregate leaves
* arithmetic wrapping aggregate leaves
* nested scalar expression trees over grouped inputs

This does not require a full SQL grammar redesign if the engine already has a reusable SQL expression AST.

The key parser rule is:

> grouped projection items must be parsed as expressions, not as a fixed menu of “field or aggregate” surface items.

---

## Lowering changes

### Current problem

Grouped lowering likely treats grouped projection as a special-case shape where aggregate nodes are terminal.

### Target

Lower grouped projection into:

* grouped key expression nodes
* aggregate expression nodes
* scalar expression nodes over those grouped inputs

A useful internal split is:

1. **aggregate plan phase**

   * identify grouped keys
   * identify aggregates that must be computed

2. **post-aggregate expression phase**

   * build expression trees referencing grouped key slots and aggregate output slots

This lets the grouped executor compute all required grouped outputs first, then evaluate the final projection expression tree.

---

## Validation rules

### Allowed leaves in grouped projection expressions

A grouped projection expression leaf must be one of:

* grouped key field
* aggregate output

Anything else remains invalid.

### Still invalid in v1

These should fail closed:

```sql
SELECT class_name, name + 1, COUNT(*)
FROM character
GROUP BY class_name;
```

```sql
SELECT class_name, ROUND(name, 2)
FROM character
GROUP BY class_name;
```

```sql
SELECT class_name, AVG(ROUND(strength, 2))
FROM character
GROUP BY class_name;
```

The last example is intentionally out of scope because it changes aggregate input semantics rather than post-aggregate projection semantics.

### Type validation

Grouped expression validation should continue to enforce:

* function signatures
* arithmetic compatibility
* aggregate output type compatibility
* nullability behavior consistent with existing scalar evaluation rules

---

## Executor changes

### Execution model

Grouped execution should remain:

1. scan source rows
2. build/accumulate grouped state
3. finalize aggregate outputs
4. build grouped output row
5. evaluate post-aggregate projection expressions
6. apply existing grouped output pagination/order constraints
7. emit rows

### Projection evaluator

The simplest implementation path is likely:

* introduce a grouped-output row representation
* adapt the existing scalar expression evaluator to read from grouped key slots and aggregate output slots

This should avoid creating a separate grouped-only function semantics path.

That is important.

The grouped projection layer should reuse the shared expression evaluator where possible, not fork into a second evaluator tree.

---

## Ordering and boundedness

`0.85` should not change the current grouped boundedness contract.

So:

* grouped `ORDER BY` still requires `LIMIT` unless separately relaxed later
* grouped `ORDER BY` must still start with grouped key prefix
* expression widening must not imply full-sort support

If expression-valued grouped `ORDER BY` is desirable later, that should be a separate boundedness review.

## Phased execution plan

### Phase A: grouped projection expression model

* define grouped post-aggregate expression representation
* distinguish grouped key leaves vs aggregate output leaves
* keep validation fail-closed

### Phase B: parser and lowering widening

* parse grouped select items as expressions
* lower aggregate leaves separately from post-aggregate scalar expression trees

### Phase C: executor projection evaluation

* finalize grouped output row
* evaluate shared scalar expressions over grouped output slots
* reuse shared evaluator where possible

### Phase D: targeted function enablement

* enable `ROUND(aggregate, scale)` and arithmetic-over-aggregate first
* keep support set narrow and explicit

### Phase E: optional HAVING / ORDER BY widening

* only if it falls out cleanly from the same model
* otherwise defer to later slice

---

## Guardrails

### Guardrail 1: no grouped-only evaluator fork

Do not introduce a second ad hoc grouped function evaluator.

The design should reuse shared scalar expression evaluation over finalized grouped outputs.

### Guardrail 2: no aggregate-input widening in this slice

This slice is about post-aggregate expression evaluation, not arbitrary aggregate argument expressions.

Do not let:

```sql
AVG(ROUND(strength, 2))
```

slip in under the same banner.

### Guardrail 3: no boundedness regression

Do not relax grouped `ORDER BY` execution guarantees in this slice.

### Guardrail 4: grouped keys and aggregate outputs remain the only leaves

No raw non-grouped source fields may enter grouped projection expressions.

---

## Acceptance criteria

### Functional

* grouped projection accepts scalar functions over aggregate outputs
* grouped projection accepts arithmetic over aggregate outputs
* grouped projection still accepts grouped keys and aggregate leaves
* invalid non-grouped raw field references still fail closed

### Architectural

* grouped executor remains the sole owner of aggregate accumulation/finalization
* shared scalar evaluator is reused for post-aggregate expression evaluation
* no new grouped-only expression semantics path is introduced

### Validation

* unsupported grouped expressions fail cleanly
* function signatures remain type-checked
* grouped key authority remains preserved

### Performance

* no change to grouped accumulation strategy
* expression evaluation cost is paid only after aggregate finalization
* no new implicit full-sort or full-materialization path is introduced

---

## Thesis sentence

> **`0.85` should widen grouped SQL projection to support post-aggregate scalar expressions over grouped keys and finalized aggregate outputs, while preserving current grouped execution ownership and boundedness guarantees.**
