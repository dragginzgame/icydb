IcyDB 0.61 Design

Status: In Progress
Theme: Entity Macro Output Compression

Patch numbering note: active `0.61` release notes track the next patch after
the workspace version in `Cargo.toml`. With the workspace currently at
`0.61.0`, the shipped work in this line is recorded under `0.61.1`.

## 0. Summary

`0.61` is the follow-up line after `0.60` removed generated view-style DTOs.

The goal is not to remove views again. That work is already done.

The real remaining problem is that entity macro expansion still emits too much
per-entity code and metadata:

- one full schema AST constant
- one full runtime `EntityModel`
- one per-field `FieldModel` constant set
- one per-field visitor walk
- one per-field sanitize walk
- one per-field validate walk
- one per-type `FieldValue` conversion layer
- one set of relation accessors and query field constants

`0.61` should therefore be a macro-compression line, not a second DTO-removal
line and not an immediate unsafe runtime rewrite.

## 1. Sanity Check

### 1.1 What The Current Draft Gets Right

The current draft is correct about three important things:

1. There is still meaningful structural duplication in macro output.
2. Visitor, sanitize, and validate generation are still major per-entity code
   emitters.
3. Schema/model duplication is now a more credible wasm target than executor
   generic cleanup.

### 1.2 What Is Out Of Date

The current draft still assumes the old view/update system exists.

That is no longer true after `0.60`:

- generated `EntityView`, `EntityCreate`, and `EntityUpdate` are gone
- `CoreAsView`, `CoreCreateView`, and `CoreUpdateView` are gone
- the internal patch/view layer is gone

So sections centered on “collapse view types”, “Value-backed views”, or
“replace `CoreAsView` / `CoreUpdateView`” are no longer the real `0.61`
problem.

### 1.3 What Is Too Aggressive For The Current Codebase

The draft jumps straight to a descriptor-and-offset runtime:

- one universal `EntityDescriptor`
- pointer arithmetic for field access
- a fully runtime-driven visitor/sanitize/patch engine

That is not impossible, but it is not the immediate shape suggested by the
current code:

- `Visitable`, `SanitizeAuto`, and `ValidateAuto` are still typed trait-based
  contracts in `icydb-core`
- `FieldValue` still converts concrete Rust types to and from `Value`
- entity relation helpers still intentionally return typed `Id<T>` wrappers
- schema registration and runtime model ownership are still split across
  `schema::node::*` and `model::*`

So a full offset-driven runtime engine would be a larger architectural rewrite
than this draft currently acknowledges.

## 2. Actual Current Macro Hotspots

These are the real remaining emitters in the current macro code.

### 2.1 Schema AST Emission

`crates/icydb-schema-derive/src/node/entity.rs`

The macro still emits a full nested schema AST through
`::icydb::schema::node::Entity::new(...)` and companion `Field` / `Value` /
`Item` construction.

This is real output weight.

### 2.2 Runtime Model Emission

`crates/icydb-schema-derive/src/imp/inherent/entity.rs`

The macro also emits:

- one query-facing `FieldRef` constant per field
- one `FieldModel` constant per field
- one `__MODEL_FIELDS` array
- one `__ENTITY_MODEL`
- one relation accessor set

This means schema and runtime model facts are still materialized twice.

### 2.3 Traversal Emission

`crates/icydb-schema-derive/src/imp/visitable.rs`

Entity and record traversal are still emitted as explicit field-by-field
`perform_visit(...)` / `perform_visit_mut(...)` sequences.

### 2.4 Sanitization And Validation Emission

`crates/icydb-schema-derive/src/imp/sanitize.rs`
`crates/icydb-schema-derive/src/imp/validate.rs`

These still emit per-field and per-container traversal logic directly into each
generated type.

### 2.5 Value Conversion Emission

`crates/icydb-schema-derive/src/imp/field_value.rs`

Structured schema types still generate explicit `FieldValue::to_value` /
`from_value` logic, especially for enums, lists, maps, records, and newtypes.

## 3. Design Goal

The `0.61` goal should be:

- reduce per-entity generated code size
- remove duplicated schema/model emission where one authority is enough
- shrink field-walking boilerplate across visit/sanitize/validate
- keep typed API surfaces where they are still materially useful

It should **not** assume that `0.61` must land a universal unsafe descriptor
runtime in one patch.

## 4. Corrected Strategy

### 4.1 Treat `0.61` As Macro Compression, Not DTO Replacement

`0.60` already removed the generated DTO families.

So `0.61` should focus on the current remaining macro surface:

- schema constants
- runtime model constants
- field-walk emission
- value-conversion emission

### 4.2 Collapse Duplicate Structural Authorities First

The biggest concrete duplication now is between:

- schema-node emission in `node/entity.rs`
- runtime model emission in `imp/inherent/entity.rs`

The first serious `0.61` move should be to make those two surfaces derive from
one lower-level field metadata builder instead of recomputing the same entity
facts separately.

### 4.3 Reduce Boilerplate Before Replacing The Runtime

The field-walk code in:

- `imp/visitable.rs`
- `imp/sanitize.rs`
- `imp/validate.rs`

should first be collapsed behind shared derive-side helpers so the macro emits
less repeated structure even before any runtime erasure is attempted.

### 4.4 Defer Unsafe Offset-Based Access Until After Measurement

Offset-based field access may still be a good later direction, but it should be
an explicit later phase.

It should not be the assumed first patch because:

- it changes the trust boundary
- it changes traversal contracts
- it needs careful handling for nested structured fields
- it is much larger than the current draft implies

## 5. Concrete Plan

### Phase 1: Establish The Real Baseline

Use the current macro code and the existing generated sample in this design
directory as the baseline.

Tasks:

- capture one representative generated entity expansion before `0.61` changes
- measure where tokens are coming from:
  - schema AST emission
  - runtime model emission
  - visitor emission
  - sanitize/validate emission
  - `FieldValue` emission
- record fresh wasm numbers after `0.60` so `0.61` has a clean starting point

Exit criteria:

- we know which remaining emitter family is largest
- we stop guessing that the old view system is still the main cause

### Phase 2: Collapse Schema/Model Builder Duplication

Introduce one derive-internal field metadata lowering layer and make both
schema-node emission and runtime-model emission use it.

Concretely:

- add one shared derive-side helper that lowers `Field` / `Value` / `Item`
  metadata once
- stop having `node/entity.rs` and `imp/inherent/entity.rs` each derive field
  kind metadata independently
- remove per-field `__MODEL_FIELD_*` constants if one direct `__MODEL_FIELDS`
  array is enough

This is the most grounded immediate size win because it attacks duplication
that visibly exists in the current emitted code.

### Phase 3: Compress Field-Walk Emission

Refactor derive generation so entity/record field traversal is described once
and reused across:

- `Visitable`
- `SanitizeAuto`
- `ValidateAuto`

Concretely:

- introduce one derive-side field-walk helper for entity/record fields
- make visit/sanitize/validate generation consume that shared lowering path
- keep typed field access for now; do not jump straight to raw offsets

This will not eliminate all monomorphization, but it should cut emitted token
duplication and make later runtime consolidation easier.

### Phase 4: Re-evaluate `FieldValue`

After Phases 2 and 3, remeasure.

If the line still shows large per-entity growth, the next likely target is
`FieldValue` emission for structured schema types.

Concretely:

- inspect enum/list/map/newtype/record `FieldValue` impl size
- determine whether some conversions can move to shared helpers without losing
  boundary typing
- keep typed relation and entity surfaces unless measurement proves they are a
  dominant cost

### Phase 5: Optional Runtime Descriptor Follow-Up

Only if the earlier phases are not enough:

- prototype descriptor-driven field walking for entity/record traversal
- keep that prototype scoped to traversal/sanitize/validate first
- do not combine it with a second broad API redesign in the same patch

This is where offset-based access belongs if it remains necessary after the
lower-risk compression work.

## 6. Non-Goals

`0.61` should explicitly not promise all of the following in one slice:

- a full unsafe pointer/offset runtime
- replacing all typed field access with `Value`
- removing typed relation accessors
- removing `FieldRef` query constants
- another DTO/view-system redesign

## 7. Expected Outcome

If `0.61` succeeds, the macro line should end with:

- less duplicated emitted metadata
- less repeated field-walk boilerplate
- a clearer separation between:
  - schema authority
  - runtime model authority
  - boundary value conversion

The likely result is a meaningful but not magical wasm win.

The correct success criterion is:

- measurable per-entity reduction after `0.60`
- lower generated expansion size
- a cleaner path for any deeper descriptor-runtime work later

## 8. One-Line Summary

`0.61` should compress the current macro output that still exists after `0.60`
by collapsing schema/model duplication first and reducing repeated
visit/sanitize/validate emission before attempting a larger descriptor-runtime
rewrite.
