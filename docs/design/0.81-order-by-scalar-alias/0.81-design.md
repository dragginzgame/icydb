# 0.81 Design: Bounded Scalar Computed `ORDER BY` Alias

## Preamble

* scope: admit `ORDER BY <alias>` for bounded scalar computed projection aliases
* goal: allow scalar arithmetic and `ROUND(...)` projection aliases to participate in ordering without widening direct expression ordering or grouped execution semantics
* theme: extend the existing canonical scalar expression path through one narrow alias-normalization seam
* non-goals:
  * no direct `ORDER BY` arithmetic expressions
  * no direct `ORDER BY ROUND(...)`
  * no grouped arithmetic projection
  * no grouped `ROUND(...)`
  * no aggregate/computed grouped alias ordering
  * no generic nested SQL expression grammar
  * no new planner/runtime side path

---

## Problem Statement

The reduced SQL surface still has one obvious bounded coherence gap.

Already admitted today:

* scalar projection aliases
* `ORDER BY <alias>` when the alias resolves to an already-supported order target
* bounded scalar arithmetic projection
* bounded scalar `ROUND(expr, scale)` projection

Still blocked today:

```sql
SELECT age + 1 AS next_age FROM Customer ORDER BY next_age;
SELECT ROUND(age / 3, 2) AS rounded_age FROM Customer ORDER BY rounded_age;
SELECT dexterity + charisma AS total FROM Character ORDER BY total;
```

That gap is more visible now because the scalar expression family is already
real and canonical. The missing piece is not a new runtime expression family.
It is one narrow alias-ordering extension:

**allow `ORDER BY <alias>` when that alias resolves to one already-admitted bounded scalar computed projection expression.**

---

## Why This Should Be The Slice

### 1. It closes the most obvious bounded scalar coherence gap

Projection already admits bounded computed scalar expressions. Alias ordering
already works for narrower target families. The missing piece is alias ordering
for the same bounded scalar computed expressions the projection lane already
owns.

### 2. It stays on the canonical expression path

The current direction of the SQL/query line is clear:

* parser owns syntax branches
* lowering owns semantic/admission branches
* execution owns routing/materialization

This feature should stay inside that split. It should be a normalization and
admission extension, not a new order-only execution family.

### 3. It is still small enough to measure honestly

This slice touches the shared compile/setup floor:

* parser alias capture
* lowering alias resolution
* canonical order-target normalization
* plan construction
* execute setup
* row shaping

That makes it a good fit for the existing perf harness without widening into a
generic expression-ordering project.

---

## Current Boundary

### Already admitted

The reduced SQL surface already admits:

* `ORDER BY field`
* `ORDER BY LOWER(field)`
* `ORDER BY UPPER(field)`
* `ORDER BY <alias>` when the alias resolves to one of those targets
* scalar arithmetic projection
* scalar `ROUND(field, scale)`
* scalar `ROUND(arithmetic_expr, scale)`
* bounded field-to-field arithmetic projection in scalar projection position

### Still intentionally excluded

The surface still intentionally excludes:

* direct `ORDER BY` arithmetic expressions
* direct `ORDER BY ROUND(...)`
* grouped arithmetic projection
* grouped `ROUND(...)`
* grouped computed alias ordering
* aggregate-derived alias ordering
* generic nested SQL expression parsing

That boundary remains correct for this slice.

---

## Proposed Surface

### Admitted in this slice

Allow `ORDER BY <alias>` when the alias resolves to exactly one admitted
bounded scalar computed projection expression on the scalar query lane.

Required admitted forms:

```sql
SELECT age + 1 AS next_age FROM Customer ORDER BY next_age;
SELECT age - 1 AS prev_age FROM Customer ORDER BY prev_age DESC;
SELECT age * 2 AS doubled_age FROM Customer ORDER BY doubled_age;
SELECT age / 3 AS age_thirds FROM Customer ORDER BY age_thirds;
SELECT ROUND(age, 2) AS rounded_age FROM Customer ORDER BY rounded_age;
SELECT ROUND(age / 3, 2) AS rounded_age FROM Customer ORDER BY rounded_age DESC;
SELECT dexterity + charisma AS total FROM Character ORDER BY total;
SELECT ROUND(dexterity + charisma, 3) AS dextrisma FROM Character ORDER BY dextrisma;
```

Multi-term ordering remains admitted when each term resolves to an already
admitted target:

```sql
SELECT age + 1 AS next_age, id
FROM Customer
ORDER BY next_age ASC, id ASC;
```

### Not admitted in this slice

Do not admit:

```sql
SELECT age + 1 FROM Customer ORDER BY age + 1;
SELECT ROUND(age / 3, 2) FROM Customer ORDER BY ROUND(age / 3, 2);
SELECT age, ROUND(age / 3, 2), COUNT(*) FROM Customer GROUP BY age ORDER BY rounded_age;
SELECT SUM(age) AS total_age FROM Customer ORDER BY total_age;
```

The allowed form is alias ordering onto already-admitted scalar computed
projection expressions, not direct expression ordering and not grouped or
aggregate computed ordering.

---

## Design Goal

Keep this slice boring.

This feature should not widen the planner/runtime contract beyond what already
exists for bounded scalar expressions. The change should remain an alias
resolution and order-target normalization extension layered over the existing
canonical scalar expression path.

The target is:

**projection expression stays canonical and query-owned; `ORDER BY <alias>` resolves onto that existing canonical expression without introducing a second order-expression engine.**

This also needs to fit the current cache line cleanly:

* the admitted order target must extend the same query-owned semantic artifact
* it must slot into `CompiledSqlCommand`, not create a second cache model
* it must not create grouped-only or order-only side caches
* admitted computed `ORDER BY` aliases must canonicalize onto the same
  structural order/query representation as the equivalent non-aliased internal
  form before compiled-command cache lookup or insertion

---

## Architectural Approach

### Chosen approach

Resolve eligible computed projection aliases during SQL prepare/lowering onto
the same canonical scalar expression already owned by the projection item.

Conceptually:

1. parse projection items and aliases
2. parse `ORDER BY <alias>` terms
3. determine whether the alias resolves to one admitted bounded scalar computed projection expression
4. normalize the admitted alias onto the same canonical semantic expression and
   structural order/query representation already owned by the projection path
5. continue through normal planning and execution

### Rejected approach: new session-owned computed ordering lane

Do not introduce a session-owned rewrite lane or order-only semantic family for
numeric computed ordering.

Reason:

* it would regress the ownership cleanup already done
* it would split projection semantics from order semantics again
* it would complicate explain, planner parity, and cache ownership

### Rejected approach: admit direct arithmetic order expressions

Do not use alias support as a backdoor to general `ORDER BY` expression
parsing.

Reason:

* that is a larger SQL surface decision
* it widens parser and order-target contracts more than necessary
* it makes current fail-closed boundaries ambiguous

---

## Ownership Rules

### Parser owns

* projection alias capture
* bounded scalar arithmetic projection parsing
* bounded `ROUND(...)` projection parsing
* `ORDER BY <alias>` surface acceptance

### Lowering owns

* alias resolution
* determining whether an alias resolves to one admitted bounded scalar computed projection expression
* normalization of admitted aliases onto canonical scalar expression semantics
* fail-closed rejection for non-admitted alias targets

### Planning owns

* normal order specification over the canonical semantic order target
* no special planner branch for “computed alias ordering”

### Execution owns

* evaluation of the canonical scalar expression already admitted by the bounded scalar expression family
* no alias-specific execution semantics

---

## Exact Admitted Expression Family

This slice should only admit aliases that resolve to scalar projection
expressions already admitted on the scalar projection lane.

### Arithmetic aliases

Admit aliases whose projection expression is exactly one bounded arithmetic
expression of the current scalar family:

* `field + literal`
* `field - literal`
* `field * literal`
* `field / literal`
* `field + field`
* `field - field`
* `field * field`
* `field / field`

### `ROUND` aliases

Admit aliases whose projection expression is exactly one bounded `ROUND(...)`
expression where the input is:

* one field, or
* one already-admitted bounded arithmetic expression

### Explicit exclusion

Keep out:

* chained arithmetic beyond the current bounded scalar family
* grouped computed projections
* grouped `ROUND(...)`
* aggregate-derived aliases
* direct expression order terms

---

## Order Semantics

When the alias resolves to an admitted computed scalar projection, ordering
should behave exactly as if the canonical expression had been the order target
all along.

That implies:

* `ASC` / `DESC` semantics stay unchanged
* multi-term ordering stays unchanged
* outward projection labels remain the chosen aliases
* explain output remains semantically equivalent to the canonical internal
  order target representation

The design should not create an order-only expression contract. It should reuse
the projection’s canonical expression meaning.

---

## Explain and Cache Parity

### Explain parity

Aliased computed-order queries should explain the same semantic order target as
their canonical internal form. The visible SQL still preserves the outward
alias in the original statement surface, but plan semantics remain canonical.

### Cache parity

This slice must fit the current compiled-command/cache model:

* admitted computed alias ordering should extend the same semantic compiled
  artifact used for other SQL query families
* admitted alias ordering must canonicalize before compiled-command cache
  lookup and before cache insertion
* the cache must not assume that current scalar-only admitted order targets are
  permanent
* future grouped widening, if it happens later, must extend the same
  query-owned semantic artifact rather than introduce grouped-only cache paths

---

## Perf Strategy

This slice should be developed with the perf harness active from day one.

### Required perf scenarios

Add targeted scenarios for at least:

* bounded arithmetic alias ordering
  * `SELECT age + 1 AS next_age, id FROM Customer ORDER BY next_age ASC, id ASC LIMIT 10`
* bounded arithmetic alias ordering descending
  * `SELECT age - 1 AS prev_age, id FROM Customer ORDER BY prev_age DESC, id ASC LIMIT 10`
* bounded field-to-field arithmetic alias ordering
  * `SELECT dexterity + charisma AS total, id FROM Character ORDER BY total ASC, id ASC LIMIT 10`
* bounded `ROUND(...)` alias ordering
  * `SELECT ROUND(age / 3, 2) AS rounded_age, id FROM Customer ORDER BY rounded_age ASC, id ASC LIMIT 10`
* bounded `ROUND(field + field, scale)` alias ordering
  * `SELECT ROUND(dexterity + charisma, 3) AS dextrisma, id FROM Character ORDER BY dextrisma ASC, id ASC LIMIT 10`
* companion baseline scenarios where ordering uses the existing admitted target
  instead of the alias, so attribution stays honest

### Perf acceptance goal

The feature should ride the shared compile/setup floor instead of creating a
new outlier lane. It does not need to prove zero overhead; it needs to show no
suspicious step-function relative to comparable scalar computed projection
scenarios.

---

## Testing Strategy

### Parser tests

Add coverage for:

* arithmetic projection alias followed by `ORDER BY <alias>`
* field-to-field arithmetic projection alias followed by `ORDER BY <alias>`
* `ROUND(...)` projection alias followed by `ORDER BY <alias>`
* multi-term `ORDER BY` with one computed alias and one field term
* fail-closed direct expression ordering remains rejected

### Lowering tests

Add coverage proving:

* admitted computed aliases normalize onto the canonical scalar expression
  order target
* projection and ordering semantics share the same expression family
* admitted aliased and canonical equivalent forms normalize onto the same
  structural order/query representation before cache identity is consulted
* grouped computed alias ordering remains rejected
* aggregate alias ordering remains rejected in this slice

### Session and execution tests

Add coverage proving:

* result rows are ordered correctly for arithmetic alias ordering
* result rows are ordered correctly for field-to-field arithmetic alias ordering
* result rows are ordered correctly for `ROUND(...)` alias ordering
* outward projection labels remain the chosen aliases
* ordering parity matches the canonical underlying expression semantics

### Explain tests

Add coverage proving:

* aliased computed-order queries explain with canonical semantic order targets
* no planner-shape drift appears between equivalent aliased and canonical forms

### Cache-compatibility tests

Add one hard acceptance test proving that equivalent admitted forms produce the
same compiled semantic artifact or equivalent compiled plan identity.

At minimum, prove parity between:

* the admitted aliased form
* the equivalent canonical internal form

The important rule is not the exact assertion helper. The important rule is
that cache compatibility is testable:

* same structural order/query representation
* same semantic compiled identity
* no duplicate cache shape caused solely by alias spelling

### Perf tests

Add targeted PocketIC perf scenarios mirroring the admitted forms.

---

## Explicit Fail-Closed Boundaries

These must remain true after this slice:

* direct `ORDER BY` arithmetic expression remains out
* direct `ORDER BY ROUND(...)` remains out
* grouped arithmetic projection remains out
* grouped `ROUND(...)` remains out
* grouped computed alias ordering remains out
* aggregate-derived alias ordering remains out
* generic nested expression grammar remains out

This is a bounded scalar alias-ordering slice, not the start of generic
expression ordering.

---

## Risks

### Risk: alias normalization reintroduces a side path

If implementation resolves computed aliases through a session-only rewrite lane
or order-only expression contract, it will regress the canonical-expression
cleanup already done.

Mitigation:

* require lowering tests to prove canonical expression parity
* require explain parity tests
* reject designs that add order-only evaluation logic

### Risk: admitted family widens accidentally

Once arithmetic alias ordering works, it will be tempting to admit direct
arithmetic ordering or grouped computed ordering in the same slice.

Mitigation:

* keep admitted and rejected forms explicit
* keep fail-closed tests explicit
* keep grouped widening out of scope

### Risk: cache shape drifts

If implementation adds SQL-only or grouped-only side caches to make alias
ordering work, it will be fighting the current cache architecture instead of
extending it.

Mitigation:

* require the admitted order target to slot into the same query-owned semantic
  artifact
* reject cache changes that introduce a second semantic ownership model

---

## Acceptance Criteria

This slice is complete when:

* `ORDER BY <alias>` works for admitted bounded scalar arithmetic projection aliases
* `ORDER BY <alias>` works for admitted bounded scalar `ROUND(...)` projection aliases
* `ORDER BY <alias>` works for admitted bounded field-to-field arithmetic projection aliases
* implementation reuses the canonical scalar expression path rather than introducing a side path
* admitted aliased and canonical equivalent forms share the same compiled
  semantic identity or equivalent compiled plan identity
* grouped and aggregate computed alias ordering remain fail-closed
* explain remains semantically aligned with canonical order targets
* targeted perf scenarios exist for the admitted forms
* no new planner/runtime/cache ownership drift is introduced

---

## Final Recommendation

Ship one narrow feature:

**bounded scalar computed projection alias ordering for arithmetic and `ROUND(...)`, scalar lane only, with the current compiled-command ownership and perf harness active from day one.**

That closes the most obvious bounded scalar SQL coherence gap while preserving
the current parser/lowering/execution ownership split and the current
query-owned cache model. It is a real SQL usability win, but it is still small
enough to measure and reason about cleanly.
