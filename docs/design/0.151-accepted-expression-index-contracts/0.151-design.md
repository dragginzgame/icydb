# IcyDB 0.151 - Accepted Expression Index Contracts

## Status

Design draft.

## Context

0.149 made accepted persisted row contracts authoritative for runtime writes and response
materialization. 0.150 moved runtime field-path index planning, key construction,
visibility, and explain surfaces onto accepted index contracts.

The remaining generated-runtime authority is expression indexes. 0.150 deliberately
wrapped that lane as a compatibility bridge, but expression index key shape is still
ultimately derived from generated index metadata. 0.151 closes that gap by making
accepted expression index contracts the only runtime source of expression index key
shape, visibility, uniqueness, and fingerprint identity after schema acceptance.

## Problem

Expression indexes currently need generated metadata for at least some of these
runtime decisions:

- Which expression operation derives each key component.
- Which accepted field or field path feeds the expression.
- The input and output value kind used by planner eligibility and key encoding.
- Filter predicate membership for partial expression indexes.
- Unique index validation and index store identity.
- Query plan, explain, and cache fingerprint identity.
- Write-time key construction and recovery parity.

That is the wrong authority boundary after schema acceptance. Generated models can
propose and compare schema shape, but accepted schema snapshots must decide runtime
behavior.

## Goals

- Persist accepted expression index contracts in schema snapshots.
- Keep expression indexes limited to the existing deterministic expression-index
  subset; this is not a new arbitrary SQL expression engine.
- Expose accepted expression index contracts through `SchemaInfo` and
  `VisibleIndexes`.
- Route planner choice, explain output, plan fingerprints, writes, uniqueness
  validation, and recovery through accepted expression contracts.
- Remove production use of generated expression candidate wrappers.
- Fail closed when accepted expression metadata is missing, unsupported, or drifts
  from the generated proposal.

## Non-Goals

- Adding new expression operations.
- Adding SQL DDL for expression indexes.
- Preserving compatibility fallbacks for older internal accepted-schema formats.
- Reworking the general scalar expression evaluator.
- Changing field-path index authority completed by 0.150.

## Hard Invariant

After accepted schema reconciliation, runtime code must not call
`E::MODEL.indexes()`, `EntityModel::indexes()`, or consume raw `IndexModel` for
expression index planning, explain, cache identity, write key construction,
uniqueness validation, or recovery. Those generated surfaces remain valid only in
proposal, model-only, compatibility-comparison, and test-only lanes.

## Contract Shape

The accepted snapshot needs a stable, canonical contract for every expression index.
The exact Rust names can follow existing schema modules, but the runtime shape should
carry these concepts:

```rust
struct AcceptedExpressionIndexContract {
    ordinal: u16,
    name: AcceptedIndexName,
    store: AcceptedIndexStoreId,
    unique: bool,
    key_items: Vec<AcceptedIndexKeyItemContract>,
    predicate: Option<AcceptedIndexPredicateContract>,
    fingerprint: AcceptedIndexFingerprint,
}

enum AcceptedIndexKeyItemContract {
    FieldPath(AcceptedFieldPathIndexKeyContract),
    Expression(AcceptedExpressionIndexKeyContract),
}

struct AcceptedExpressionIndexKeyContract {
    op: AcceptedExpressionIndexOp,
    source: AcceptedExpressionSourceContract,
    input_kind: AcceptedValueKind,
    output_kind: AcceptedValueKind,
    canonical_text: String,
    fingerprint: AcceptedExpressionFingerprint,
}

struct AcceptedExpressionSourceContract {
    field_id: AcceptedFieldId,
    row_slot: AcceptedRowSlot,
    path: AcceptedFieldPathContract,
    nullable: bool,
}
```

`AcceptedExpressionIndexOp` should mirror the current supported generated
`IndexExpression` subset:

- `Lower`
- `Upper`
- `Trim`
- `LowerTrim`
- `Date`
- `Year`
- `Month`
- `Day`

The contract must be canonical enough that two semantically identical proposals
produce the same accepted fingerprint and two materially different proposals do not.

## Authority Flow

Generated model metadata remains the proposal input:

```text
generated IndexModel
  -> schema proposal
  -> accepted expression index contract
  -> accepted schema snapshot
  -> SchemaInfo / VisibleIndexes
  -> planner, executor, explain, writes, recovery
```

Runtime code consumes only the accepted half of that flow.

## Reconciliation Rules

Schema acceptance should derive expression contracts deterministically from generated
proposals, then persist the accepted result. Later reconciliation should compare the
generated proposal to the accepted contract and fail closed on meaningful drift:

- The index name, store identity, uniqueness, and key item count must match.
- Field-path key components must resolve to accepted field ids and accepted row slots.
- Expression key components must resolve to accepted source field ids, accepted slots,
  canonical field paths, supported operations, and stable value-kind conversions.
- Partial-index predicates must resolve to accepted predicate contracts.
- Fingerprints must include expression op, source identity, source path, output kind,
  predicate identity, uniqueness, and store identity.

Unsupported expression forms should be rejected at acceptance time. Missing accepted
expression metadata should make the index invisible to runtime planning and writes,
or fail the operation when the schema claims the index is required.

## Planner And Explain

`VisibleIndexes` should expose accepted expression index contracts alongside accepted
field-path contracts. Planner and explain code should not receive generated expression
candidate slices.

Planner eligibility should read from accepted expression contracts:

- Key order and prefix structure.
- Expression op and output kind.
- Predicate membership and guarded filtered-index eligibility.
- Unique/non-unique route metadata.
- Store and fingerprint identity.

Explain output should render from accepted contracts so that plan text, JSON explain,
and trace summaries reflect the same authority used by execution.

## Writes And Recovery

Write-time index maintenance should derive expression keys from accepted row contracts
and accepted expression contracts:

- Read source values through accepted row-layout slots.
- Apply the accepted expression operation.
- Encode key parts using the accepted output kind.
- Apply accepted partial-index membership predicates.
- Validate unique indexes using accepted uniqueness and store identity.

Recovery and rebuild paths must use the same accepted contracts as normal writes. A
recovered expression index must produce the same keys and fingerprints as live write
maintenance for the same accepted snapshot.

## Removal Plan

0.150 introduced generated expression wrappers to narrow the remaining runtime bridge.
0.151 should delete that bridge once accepted expression contracts are live:

- Remove production uses of `GeneratedExpressionCandidateIndex`.
- Remove production uses of `GeneratedExpressionIndex`.
- Move any remaining `SemanticIndexAccessContract::from_generated_index` style helper
  into proposal/model-only/test-only code, or replace it with accepted constructors.
- Keep shared expression evaluation helpers only if they accept accepted expression
  operations or are clearly authority-neutral.

## Phase Plan

1. Add accepted expression index snapshot and proposal contract types.
2. Derive canonical expression contracts from generated proposals during schema
   acceptance.
3. Persist and reload expression contracts in accepted schema snapshots.
4. Expose accepted expression indexes through `SchemaInfo` and `VisibleIndexes`.
5. Route planner, explain, cache fingerprints, and access-choice reranking through
   accepted expression contracts.
6. Route write key construction, partial membership, uniqueness validation, and
   recovery through accepted expression contracts.
7. Remove generated expression runtime wrappers and strengthen guards.

## Validation

The 0.151 slice should finish with targeted and broad validation:

- Contract derivation tests for every supported expression op.
- Reconciliation drift tests for op, source field, source path, output kind,
  predicate, uniqueness, and store identity.
- SQL expression-index planning and execution matrix.
- Filtered expression-index matrix.
- Covering/key-only expression-index read tests.
- Write, update, delete, uniqueness, and recovery parity tests.
- Explain text and JSON snapshot coverage.
- Guard tests proving non-test runtime modules do not call generated index surfaces
  for expression index planning or key construction after accepted schema info exists.
