# 0.154 Developer Stability

## Purpose

0.153 builds the schema-owned field-path index rebuild runner boundary: accepted
runner input, staged row/key materialization, isolated store writes, validation,
rollback diagnostics, runtime invalidation, snapshot publication handoff, and
final physical store promotion.

0.154 should turn that runner machinery into a stable database surface that
developers can test against. The goal is not broad DDL. The goal is to make
accepted schema changes that are already supported by the engine safe,
observable, and deterministic enough to ship to developers.

The closeout target for 0.154 is:

Accepted schema publication can run supported physical field-path index rebuild
work end to end through the schema-owned runner, keep unsupported schema changes
fail-closed, discard failed unpublished staged work when safe, fail startup
closed otherwise, and expose enough diagnostics for developers to trust a test
database without SQL DDL.

## Non-Goals

0.154 should not add:

- broad SQL DDL syntax
- arbitrary type changes
- slot remapping
- primary-key rewrites
- full row rewrites
- expression-index rebuild execution
- secondary-index drop cleanup execution
- multiple mutation plans or multi-index rebuild batches
- unique-index rebuilds or uniqueness conflict reporting
- concurrent online index build
- generated metadata fallback reconstruction
- generated index metadata as accepted runtime authority

DDL remains a later frontend over accepted schema mutation semantics. 0.154 can
ship a stable developer test surface without making SQL syntax the migration
authority.

## Hard Invariants

- Accepted persisted schema snapshots remain the catalog authority.
- Generated model/index metadata is allowed only in proposal, reconciliation,
  model-only, and test lanes.
- Runner work starts from `SchemaMutationRunnerInput` and
  `SchemaMutationExecutionPlan`; it must not reconstruct work from generated
  `EntityModel` or `IndexModel` metadata.
- Physical index data is not runtime-visible while staged or building.
- Snapshot publication is allowed only after physical validation and runtime
  invalidation have completed.
- Generic runner publication readiness must stay false until the physical
  `IndexStore` is promoted to ready.
- Failure before publication must leave the old accepted snapshot visible.
- Failure after staged writes must roll back staged physical work or return a
  typed failure that proves publication did not happen.
- Unsupported expression-index rebuild, secondary-index cleanup, full row
  rewrite, and DDL paths remain fail-closed until their own runner contracts are
  implemented.
- 0.154 supports one physical mutation shape: one non-unique field-path
  secondary index add for one entity.
- Before `1.0.0`, internal runner and recovery formats hard-cut to the latest
  contract.

## 0.154 Work Surface

### Supported Path

0.154 should wire exactly one physical schema mutation path:

single non-unique field-path secondary index add for one entity.

The accepted execution plan must contain exactly:

1. `BuildFieldPathIndex`
2. `ValidatePhysicalWork`
3. `InvalidateRuntimeState`

The runner input must name one accepted field-path rebuild target, one accepted
before snapshot, and one accepted after snapshot for the same entity. The target
index must be non-unique, secondary, field-path-only, and backed by accepted
row-layout slots.

Everything else remains fail-closed:

- expression-index rebuilds
- secondary-index drop cleanup
- unique index add/drop
- multiple index additions in one accepted transition
- mixed metadata-plus-physical batches beyond the one accepted index add shape
- full row rewrites
- DDL-sourced mutation execution

This narrow shape is the 0.154 ship target. Additional runner paths should not
be partially wired into reconciliation.

### Reconciliation Integration

Wire accepted schema reconciliation to the field-path runner only for the
supported plan shape:

1. accepted-before and accepted-after snapshots describe the same entity
2. the execution plan contains exactly one non-unique field-path index rebuild,
   physical validation, and runtime invalidation
3. the runner contract advertises field-path build, physical validation, and
   runtime invalidation
4. row scanning can produce accepted `SchemaFieldPathIndexRebuildRow` values
5. the target `IndexStore` is isolated from runtime readers until final publish

Every other rebuild-required plan remains blocked. Expression indexes and
secondary-index drops should not be partially wired just because they share
planning vocabulary with field-path indexes.

### Exclusive Rebuild Gate

The initial runner can be synchronous and full-scan. 0.154 may take an
exclusive schema/write gate for the affected entity for the full rebuild
transaction:

scan accepted live rows -> stage keys -> validate physical state -> invalidate
runtime state -> publish.

No writes or concurrent schema mutations for that entity may interleave between
the row scan, validation, and publication boundary. Reads may continue only if
they are guaranteed to observe the old accepted schema and old visible index
set until publication completes. If that guarantee is not available at the
integration point, reads should also wait behind the same gate for 0.154.

### Row Scan Adapter

Add a production adapter that scans authoritative live rows and emits
`SchemaFieldPathIndexRebuildRow` values using accepted row-layout contracts.

The adapter must:

- use accepted field slots and persisted row decode contracts
- skip rows only through typed, counted skip reasons
- reject malformed rows before publication
- avoid generated field order and generated field metadata
- produce deterministic row order for deterministic index-key output

The first version can be full-scan and synchronous. Correctness is more
important than rebuild throughput for the developer-testing release.

### Publication Transaction

0.154 production publication should make the externally visible order
unambiguous:

1. Build the staged field-path index store from accepted row contracts.
2. Validate staged physical contents against the accepted rebuild target.
3. Invalidate runtime epoch/cache state for the affected entity.
4. Promote the physical `IndexStore` to ready while the old accepted snapshot is
   still the visible schema authority.
5. Publish the accepted-after snapshot.
6. Expose the new `SchemaInfo` / `VisibleIndexes` derived from the accepted-after
   snapshot.

The important runtime rule is that no caller can observe the new accepted index
metadata until the physical store is ready. If implementation code prepares a
snapshot publication handoff before physical-store promotion, that handoff must
remain non-visible: the schema-store write and runtime `SchemaInfo` exposure are
still the final step.

Failure implications:

- failure before step 4 removes staged physical work and leaves the old accepted
  snapshot visible
- failure during physical-store promotion leaves the old accepted snapshot
  visible and must remove or quarantine the unreferenced physical store
- failure during accepted snapshot publication leaves the old accepted snapshot
  visible; any ready-but-unreferenced physical store must be discarded on the
  same attempt or detected and discarded on startup
- once step 5 succeeds, startup must be able to prove the accepted snapshot and
  ready physical store agree before exposing the database

### Physical Store Binding

Bind runner output to the real accepted index-store identity without exposing
building state to runtime readers.

The binding must:

- start from the accepted store path in `SchemaFieldPathIndexRebuildTarget`
- create or reserve an isolated building `IndexStore`
- validate raw key/entry membership against the staged write batch
- publish the physical store before the accepted-after snapshot becomes visible
- leave stale or mismatched store state rejected with typed diagnostics

The existing mockable sink boundaries should remain in place until the real
schema-store write path is narrow enough to test directly.

### Runtime Invalidation

Connect the invalidation sink to the runtime state that can observe schema or
index shape:

- visible-index selection
- planner/access-choice caches
- prepared SQL cache entries that depend on accepted schema identity
- accepted row-layout decode plans
- explain diagnostics that include index choice
- cursor or continuation signatures that include schema/index identity

Invalidation must use accepted snapshot runtime epochs. A staged rebuild that
does not publish must not advance the visible runtime epoch.

### Failure And Recovery

0.154 needs a clear restart story for developer databases.

At minimum:

- accepted snapshot remains old until supported physical work validates
- staged physical work has a typed rollback path
- runner failures report phase, classified error, and rollback action counts
- startup discards failed/unpublished staged work when it can prove that work is
  not referenced by the accepted published schema
- startup fails closed with a typed diagnostic when staged/building/ready store
  state cannot be proven safe to discard or accepted to expose
- no recovery path rehydrates accepted runtime authority from generated index
  metadata

0.154 should not imply generalized recovery or resumable rebuild. Resume can
come later. The developer-stability requirement is simpler: safe discard for
unpublished staged work, otherwise fail closed before exposure.

### Developer Diagnostics

Developer-visible schema mutation diagnostics should include:

- phase
- mutation kind
- target entity and index name/store
- rows scanned
- rows skipped
- index keys written
- validation status
- rollback status and action counts when rollback runs
- runtime invalidation status
- physical-store publish status
- accepted snapshot publish status

The internal reports can stay typed and narrow, but developer testing needs a
stable way to inspect what happened without matching error strings.

## Developer Test Surface

The 154 developer-facing surface should prioritize stable database behavior over
syntax breadth:

- existing schema bootstrapping and proposal paths continue to work
- supported accepted field-path index additions can rebuild physical index state
- unsupported schema changes fail with typed, actionable diagnostics
- reads, writes, and explains after publication observe the accepted schema and
  ready index store consistently
- startup after a successful publish sees the same accepted schema/index state
- startup after a failed staged rebuild does not expose staged physical state
- schema mutation diagnostics explain successful and failed runner attempts

SQL DDL can wait. Developers need a database that behaves predictably under the
accepted schema paths they already use.

## Test Plan

0.154 should close with tests that cover:

- production row-scan adapter emits deterministic field-path rebuild rows
- malformed persisted rows fail the runner before publication
- reconciliation invokes the field-path runner for the supported accepted plan
- unsupported expression-index rebuild and secondary-index cleanup plans remain
  blocked
- runtime cache/planner invalidation observes accepted epoch changes
- post-publication explain/planning can select the rebuilt field-path index
- staged validation failure rolls back and leaves no accepted snapshot publish
- exclusive entity schema/write gating prevents row mutation during rebuild
- developer-visible mutation reports include phase, target, row/key counts,
  validation, rollback, invalidation, and publish status
- restart after successful publication sees ready index state
- restart after failed staged work discards safely unreferenced staged state or
  fails closed with typed diagnostics
- source guards keep generated index metadata out of accepted runner authority

## Ship Criteria

0.154.0 is ready for developer testing when:

- supported non-unique field-path index rebuilds run from accepted snapshots to
  ready physical stores through reconciliation
- the supported production path is limited to one non-unique field-path
  secondary index add for one entity
- unsupported physical schema work remains fail-closed
- developer-visible runner diagnostics are typed enough to debug failed
  developer databases
- runtime invalidation prevents stale planner/cache reuse after schema publish
- recovery cannot expose staged or building index state as accepted runtime state
- no write or schema mutation can interleave with the accepted row scan,
  validation, and publication boundary for the entity
- validation covers unit, integration, write-boundary guard, and restart paths
- changelogs document the developer-visible stable database behavior

## Deferred Work

After 0.154:

1. Add SQL DDL as a frontend that lowers into accepted mutation requests.
2. Add deterministic expression-index rebuild once expression validation and
   recovery semantics are explicit.
3. Add secondary-index drop cleanup once removed-store visibility and recovery
   are explicit.
4. Add unique-index rebuilds only after uniqueness proof and conflict reporting
   are schema-owned runner contracts.
5. Add online/concurrent rebuild support only after rebuild snapshot semantics
   are explicit.
