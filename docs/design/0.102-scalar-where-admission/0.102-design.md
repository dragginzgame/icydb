# 0.102 Expression-Complete Scalar WHERE Admission

## Thesis

`0.102` should stop treating the current scalar `WHERE` operand whitelist as the
main feature boundary.

After `0.100`, scalar `WHERE` already owns planner expression semantics and an
explicit residual-filter path. The next step is:

> **admit scalar `WHERE` expression trees whenever the engine can already lower,
> type-check, and evaluate them correctly, while keeping pushdown extraction
> conservative and explicitly derived.**

This is the slice that turns scalar `WHERE` from "expression-owned, but still
heavily whitelist-gated" into "semantically admitted unless it crosses a real
scope boundary."

---

## Current baseline

This document starts after the `0.100` ownership split and the `0.101` surface
follow-through.

Already true before `0.102`:

* scalar `WHERE` lowers through planner-owned `Expr`
* planning can keep a residual scalar filter expression
* pushdown predicate extraction is separate from semantic filter ownership
* searched `CASE` in scalar `WHERE` is admitted
* a narrow affine arithmetic family can be normalized back onto the direct
  predicate lane
* common scalar function families already admitted elsewhere can appear in some
  `WHERE` positions

Still visibly incomplete before `0.102`:

* many recursive scalar expression trees still fail closed in `WHERE`
* nested arithmetic trees are still more restricted than the parser and runtime
  suggest
* the practical admission line is still partly defined by a local validator
  whitelist rather than by semantic/runtime capability

---

## Problem statement

After `0.100`, the parser and planner already understand recursive scalar
expression trees in `WHERE`.

However, scalar `WHERE` admission is still narrower than that model:

* some shapes parse and lower, but still fail at `WHERE` admission
* one small arithmetic family works only because it is rewritten into the older
  field-vs-literal predicate lane
* the validator still rejects many nested compare operands even when the engine
  could, in principle, evaluate them as residual expressions

That creates two problems:

* the SQL surface still feels arbitrary: one expression tree works, a slightly
  larger one fails
* performance and feature admission are still too entangled: shapes that are
  semantically valid are rejected instead of being planned honestly as residual
  filters

So the main remaining issue is not recursive parsing.

The issue is that scalar `WHERE` admission is still too whitelist-shaped for
the now-expression-owned model.

---

## Design goal

Make scalar `WHERE` admission primarily semantic and runtime-driven.

That means:

* if a scalar boolean `Expr` shape can be lowered, type-checked, and evaluated
  correctly on the scalar row path, it should normally be admitted
* pushdown extraction should continue to derive only the safe access subset
* performance differences should show up as route shape and residual filtering,
  not as feature rejection

The goal is:

> **let scalar `WHERE` accept the recursive scalar expression trees the engine
> already knows how to execute, and reserve fail-closed behavior for real
> semantic or scope boundaries.**

---

## Shared expression invariant

`0.102` must not create a clause-local `WHERE`-only expression family.

The normative rule is:

* scalar `WHERE` admits exactly the scalar expression node/operator/function
  families that already have defined scalar row-path runtime semantics in the
  shared planner/runtime expression model
* "already admitted scalar expression constructs" means exactly the node kinds
  whose planner `Expr` forms already have shared scalar row-path lowering,
  type-check, and runtime evaluation support
* clause-local parser admission elsewhere does **not** by itself imply `WHERE`
  admission; `WHERE` admission still depends on that shared scalar row-path
  support existing for the same planner/runtime expression family
* scalar `WHERE` may still reject those families when they cross a real clause
  boundary such as aggregate, grouped-only, subquery, or window ownership
* scalar `WHERE` does **not** invent new `WHERE`-local expression kinds just to
  widen admission ahead of the shared scalar runtime

This keeps `WHERE` as one consumer of the shared scalar expression model rather
than as a parallel feature surface.

---

## Non-goals

`0.102` does **not** aim to:

* add JOIN
* add subqueries
* add window functions
* admit aggregates in scalar `WHERE`
* admit grouped-only semantics in scalar `WHERE`
* redesign grouped `HAVING`
* widen `GROUP BY` expression ownership
* replace conservative pushdown extraction with arbitrary expression pushdown
* widen fixed-route prepared templates to arbitrary scalar expression filters
* promise that every admitted scalar `WHERE` shape is cheap or indexable

This is a scalar admission line, not a planner-wide execution rewrite.

---

## Explicit current boundary

Today the practical scalar `WHERE` line still behaves like this:

```text
parse recursive WHERE expression
  -> lower to planner Expr
  -> normalize a few special cases
  -> admit only one bounded validator subset
  -> extract pushdown predicate / residual filter from that admitted subset
```

That means a query like:

```sql
WHERE CASE WHEN level >= 30 THEN TRUE ELSE level = 20 END
```

can succeed, while:

```sql
WHERE (level + 1) * 2 >= 20
```

may still fail closed even though it is also just a recursive scalar
expression tree.

`0.102` exists to move that boundary from "current validator whitelist" to
"semantic/runtime support."

---

## Normative shipped scope

`0.102` ships **exactly** these changes:

### 1. Scalar `WHERE` admission becomes expression-complete for admitted scalar families

Scalar `WHERE` should admit recursive scalar boolean expression trees whenever:

* all leaf references are valid scalar row inputs
* all operators/functions in the tree are already admitted scalar expression
  constructs
* the tree type-checks to scalar boolean semantics under the existing runtime
  rules
* the tree can be evaluated on the scalar residual-expression path

### 2. Compare-operand admission stops being field/literal-centric

Nested compare operands are no longer limited to the current narrow family of:

* direct fields
* literals
* one bounded casefold wrapper family
* one bounded affine rewrite family

Instead, compare operands are admitted whenever the operand expression is itself
an admitted scalar expression subtree.

### 3. Pushdown extraction remains derived and conservative

The planner may still extract only the safe predicate subset.

General recursive expression admission does **not** imply:

* broader index pushdown
* broader direct routing
* broader access-path inference

Extraction in `0.102` must be **sound, not maximal**.

That means:

* extracting one pushdown-safe subexpression is allowed
* failing to extract another still-pushdown-safe subexpression is a planning
  quality issue, not a semantic correctness failure
* the release contract requires conservative correctness, not optimizer-grade
  decomposition

### 4. Residual expression evaluation becomes the normal fallback for non-pushdown scalar shapes

If a scalar `WHERE` subtree is semantically admitted but not pushdown-safe, it
must remain executable through the residual scalar expression path instead of
failing closed.

Normative fallback rule:

* if semantic admission succeeds and conservative predicate extraction cannot
  represent the full boolean tree, planning must preserve correctness on
  planner-owned `filter_expr` plus residual execution
* extraction limits are not themselves an admission failure
* the engine may still extract any sound predicate subset, but whatever is not
  extracted must remain executable on the residual filter path

---

## Admission contract

The following rule is normative for `0.102`:

### Admit

A scalar `WHERE` expression is admitted iff:

* it is a scalar boolean `Expr`
* every child subtree is an admitted scalar expression subtree from the shared
  scalar runtime model
* every function/operator in the tree already has defined scalar row-path
  runtime semantics in that shared model
* it requires no grouped, aggregate, subquery, or window ownership

### Reject

A scalar `WHERE` expression still rejects fail-closed if it:

* introduces aggregates
* depends on grouped-only values
* introduces subqueries
* introduces window expressions
* depends on a scalar function/operator whose type/null semantics are still
  undefined in the runtime

This is the line between "not yet implemented semantics" and "implemented but
possibly residual."

## Concrete query families

The following families are examples of the intended `0.102` widening:

### Newly admitted recursive arithmetic compares

```sql
SELECT name
FROM Character
WHERE (level + 1) * 2 >= 20;
```

```sql
SELECT name
FROM Character
WHERE strength + dexterity >= 24;
```

```sql
SELECT name
FROM Character
WHERE ROUND((level + 1) / 3, 2) >= 7;
```

### Newly admitted mixed boolean trees

```sql
SELECT name
FROM Character
WHERE (CASE WHEN is_npc THEN level ELSE 0 END >= 10)
  AND (strength + dexterity >= 24);
```

```sql
SELECT name
FROM Character
WHERE NOT ((level + 1) * 2 < 20 OR LOWER(class_name) = 'wizard');
```

### Still out of scope

```sql
SELECT name
FROM Character
WHERE COUNT(*) > 1;
```

```sql
SELECT name
FROM Character
WHERE EXISTS (SELECT 1);
```

```sql
SELECT name
FROM Character
WHERE ROW_NUMBER() OVER (...) > 1;
```

## Pushdown extraction contract

`0.102` does not change the conservative extraction rule.

For any admitted scalar `WHERE` expression:

* extract only the subexpressions that are semantically equivalent and
  representable in the pushdown `Predicate` model
* keep everything else on the residual scalar expression path

Example:

```sql
WHERE class_name = 'Wizard' AND (strength + dexterity) >= 24
```

should plan as:

* pushdown predicate:
  * `class_name = 'Wizard'`
* residual expression:
  * `(strength + dexterity) >= 24`

The key point is:

> semantic admission widens more than pushdown admission.

That is expected and correct.

It is also acceptable for the planner to extract less than the full safe
predicate subset, as long as whatever it does extract remains semantically
sound and the rest stays on the residual expression path.

---

## Performance contract

Performance is **not** the main admission boundary for `0.102`.

If a scalar `WHERE` expression is semantically supported, it should normally:

* execute
* expose its pushdown/residual split honestly in explain and diagnostics
* take whatever route quality the planner can legitimately provide

That means some newly admitted queries may:

* materialize more rows
* lose direct streaming
* lose some order-preserving routes
* rely entirely on residual post-access filtering

Those are planning outcomes, not reasons to reject the query.

---

## Prepared-execution boundary

`0.102` should keep the prepared boundary explicit:

| Family | `0.102` status | Notes |
| --- | --- | --- |
| existing compare-family fixed-route templates | Unchanged | The current symbolic-slot lane remains |
| newly admitted general scalar `WHERE` expression trees | Fallback prepared only | No fixed-route template widening is implied |

So `0.102` widens semantic query admission and fallback-prepared execution, but
it does **not** promise arbitrary scalar expression fixed-route templates.

Grouped pre-filter parity does **not** imply grouped symbolic-template
widening.

When grouped pre-aggregate `WHERE` shares the scalar pre-filter admission
family:

* newly admitted residual-heavy grouped shapes remain fallback-prepared by
  default
* they do **not** become symbolic grouped templates merely because they are now
  semantically admitted
* grouped fixed-route template admission must still satisfy the independently
  existing grouped symbolic access contract

This keeps `0.102` aligned with the tightened `0.99` grouped prepared boundary
instead of reopening it by implication.

Grouped pre-filter parity also does **not** widen unrelated grouped policy
surfaces. In particular, `0.102` does not by implication change:

* grouped computed-projection policy
* grouped `ORDER BY` policy
* grouped `HAVING`

## Risks

The main risks are:

* widening admission faster than runtime semantics are specified
* drift between scalar projection semantics and scalar `WHERE` semantics for the
  same expression tree
* accidentally admitting expression trees that should remain grouped/aggregate
  scoped
* planner diagnostics becoming unclear if residual-heavy plans are not surfaced
  explicitly
* accidentally widening fixed-route prepared templates by implication instead
  of by explicit design
* accidentally widening grouped computed-projection or grouped `ORDER BY`
  policy just because scalar `WHERE` semantic admission became broader

---

## Explain contract

`0.102` should make the filter split explicit without blurring filter identity.

Explain must distinguish:

* the planner-owned semantic scalar filter expression
* the extracted pushdown predicate, when present
* the residual execution filter, when present

The semantic filter shown in explain should be the planner-owned semantic
filter identity, not an accidentally collapsed presentation of only the derived
predicate contract.

When scalar-plan shape identity depends on `filter_expr`, explain, cache, and
fingerprint surfaces must remain aligned enough that semantically distinct
scalar filters do not accidentally read as one query shape just because the
same pushdown predicate subset was extracted from them.

---

## Verification matrix

`0.102` should be verified across these boundaries:

| Area | Required proof |
| --- | --- |
| SQL lowering | recursive scalar `WHERE` trees lower without clause-shape rejection when they are semantically admitted |
| Type/null semantics | admitted `WHERE` expression trees preserve the same scalar runtime truth behavior as equivalent projection evaluation |
| Pushdown extraction | admitted-but-non-pushdown subtrees remain residual instead of being over-extracted |
| Execution | nested arithmetic / function / boolean / `CASE` trees execute correctly on scalar row paths |
| Grouped pre-filter parity | grouped pre-aggregate `WHERE` keeps the same admitted scalar expression family where the grouped route shares the scalar pre-filter seam |
| Explain | semantic filter, extracted pushdown predicate, and residual execution filter remain distinguishable |
| Prepared boundary | newly admitted scalar expression trees stay fallback-prepared unless explicitly widened later |
| Negative coverage | aggregate, grouped-only, subquery, and window `WHERE` shapes still fail closed |

The projection-parity bucket is mandatory for each newly admitted family:

* arithmetic compare operands
* function-wrapped compare operands
* nested `CASE` boolean trees
* mixed pushdown/residual conjunctions

For each family, tests should prove that scalar `WHERE` truth behavior matches
the equivalent scalar projection/runtime semantics instead of drifting into a
clause-local interpretation.

---

## Acceptance criteria

`0.102` is done when:

* scalar `WHERE` no longer relies on the current narrow compare-operand
  whitelist as the primary admission boundary
* nested scalar expression trees that the runtime already supports can execute
  in `WHERE`
* residual execution remains the normal fallback for admitted-but-non-pushdown
  shapes
* explain and diagnostics make the pushdown/residual split explicit
* prepared execution remains explicit about fallback vs fixed-route ownership
* out-of-scope semantic families still fail closed cleanly
