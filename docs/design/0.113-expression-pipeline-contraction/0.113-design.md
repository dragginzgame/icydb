# 0.113 Expression Pipeline Contraction

## Thesis

`0.113` should reduce branch density and duplicated semantic reasoning across
the expression pipeline by enforcing single-authority ownership per concern and
removing intra-layer semantic duplication.

This is not a feature slice.

It is a structural contraction slice that follows directly from the current
complexity and semantic-ownership audits.

The expression pipeline should read as one linear ownership flow:

* parse
* lower
* canonicalize
* type
* plan
* bind
* execute

Not:

* parse
* lower
* infer
* reinterpret
* re-infer
* bind
* adjust
* execute

---

## Why `0.113` Is Next

`0.112` is the narrow prepared fallback consolidation slice.

That is the right first cut because it removes one contained duplication seam
and proves that the ownership model works on a real boundary.

The next structural step should not be more prepared-only cleanup.

It should be the broader expression-pipeline contraction pass across the exact
cluster that the complexity audit identified as the current branch-pressure
tier:

* `db::query::plan::expr::ast`
* `db::query::plan::expr::type_inference`
* `db::predicate::bool_expr`
* `db::session::sql::parameter`
* `db::sql::lowering::prepare`

That cluster now combines:

* high branch density
* adjacent semantic ownership
* real risk of duplicated decision trees

So `0.113` should apply the same owner-discipline that `0.112` proves, but
across the whole expression pipeline instead of only the prepared fallback
lane.

## Core Rule

One concern must have one owner.

The owner split for this slice is:

* structure -> `expr::ast`
* typing -> `type_inference`
* canonicalization -> `bool_expr`
* binding -> `prepare` / `parameter`

No overlap.

If a concern appears to have two active owners, `0.113` should treat that as a
contraction target, not as acceptable layering.

---

## Goals

### 1. Reduce branch density in the hotspot cluster

The current top branch tier is too concentrated in adjacent expression owners.

`0.113` should reduce:

* `branch_sites_total`
* duplicated local decision ladders
* repeated family-by-family branching across neighboring modules

This does not require every file to become small.

It does require the hotspot cluster to stop growing by stacking more semantic
branches on top of each other.

### 2. Remove duplicated semantic reasoning

The slice should eliminate intra-layer duplication where neighboring owners are
making the same kind of semantic decision twice.

The target is not abstract DRY.

The target is:

* one owner for structure
* one owner for typing
* one owner for canonicalization
* one owner for binding

### 3. Make the expression pipeline visibly linear

After `0.113`, a reader should be able to trace one straightforward path:

* parse -> lower -> canonicalize -> type -> plan -> bind -> execute

Without encountering adjacent modules that appear to:

* reinterpret the same expression semantics
* repeat family reconciliation
* branch on the same concern under different names

### 4. Preserve the ownership model proven in `0.112`

`0.112` should prove that prepared fallback can stop re-deriving semantic
truth.

`0.113` should extend that same owner discipline to the surrounding expression
pipeline instead of letting the rest of the cluster stay branch-heavy and
overlapping.

---

## What `0.113` Is Not

`0.113` should not be:

* new expression features
* grouped widening
* planner redesign
* prepared template widening
* a generic cleanup pass with no ownership thesis

This is a contraction slice, not an expansion slice.

---

## Recommended Shape

### 1. Map the current concern overlap

For each of the target files, classify the dominant logic as:

* structure ownership
* typing ownership
* canonicalization ownership
* binding ownership
* overlap that should be removed

The immediate question for each hotspot is:

* does this branch tree belong here at all

### 2. Collapse adjacent duplicate decision trees

Where adjacent owners are doing near-parallel family reasoning, contract that
logic back to the one correct owner.

Examples of the shape to remove:

* typing logic duplicated outside `type_inference`
* canonical semantic rewrites duplicated outside `bool_expr`
* binding-layer reasoning that starts to behave like typing or
  canonicalization

### 3. Reduce hotspot concentration, not just file count

Do not treat module splitting by itself as success.

The success condition is reduced pressure in the hotspot cluster:

* lower branch density
* fewer overlapping decision ladders
* clearer ownership boundaries

If a split only moves branches around without reducing overlap, it should not
count as a contraction win.

### 4. Keep the pipeline linear in tests and docs

Tests for this slice should reinforce the ownership flow, not only behavior.

The useful questions are:

* does the pipeline still have one clear owner per concern
* did a branch family move to the right owner
* do adjacent modules still look like they are sharing one semantic job

---

## Exit Condition

`0.113` is done when:

* the hotspot expression cluster shows lower branch-pressure concentration than
  the pre-slice baseline
* duplicated semantic decision trees are removed across the target files
* the expression pipeline reads as a linear owner flow
* structure, typing, canonicalization, and binding each have one obvious owner
* `0.112`'s prepared-fallback ownership model still holds and is not
  re-fragmented by surrounding cleanup

---

## Sequence

The clean sequence after this decision is:

* `0.112`
  * prepared fallback consolidation
* `0.113`
  * expression pipeline contraction
* `0.114`
  * prepared semantic authority collapse
* optional later line
  * route selection unification or next semantic widening

That sequence matters because it keeps expansion work behind structural
contraction instead of mixing them together.

## Summary

`0.113` should be the expression-pipeline contraction slice.

`0.112` removes one clear duplication seam.

`0.113` should then apply that owner model across the full expression hotspot
cluster:

* `expr::ast` owns structure
* `type_inference` owns typing
* `bool_expr` owns canonicalization
* `prepare` / `parameter` own binding

No overlap.
