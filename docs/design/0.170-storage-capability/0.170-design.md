# IcyDB 0.170 — Storage Capability And Mixed-Durability Contracts

## Status

In progress.

## Thesis

0.169 introduced the first non-stable backend and proved heap storage can live
behind the runtime store-wrapper boundary. 0.170 should turn the implicit
stable/heap rules into a general storage capability model, so future backends
such as `journaled(...)`, `cached_stable(...)`, WAL-backed stores,
checkpointing, and migration tooling can be added without reintroducing
backend-specific policy into executor, relation, diagnostics, reconciliation,
or commit code.

0.169 explicitly keeps higher layers backend-neutral, requires heap volatility
to be visible, keeps the stable ABI unchanged, and defers WAL, checkpointing,
migration, and SQL storage syntax. The next safe line is not "make heap
durable"; it is "make durability, recovery, relation eligibility, diagnostics,
and commit participation first-class storage capabilities."

## Why This Is 0.170

The first unresolved seam after 0.169 is mixed-store behavior. Mixed canisters
may mutate both stable and heap stores, but only the stable portion participates
in durable commit/recovery. Heap mutations are live-only side effects and must
not be advertised as atomically durable. 0.170 should make that policy
mechanically visible and testable.

The second unresolved seam is cross-backend relation policy. 0.169 rejects
stable-source strong relations into heap targets because stable rows must not
own durable integrity against volatile targets, while heap-source relations can
keep live validation semantics. That rule should become capability-driven
instead of heap-specific, because future volatile, cached, journaled, or
partially durable backends will need the same admission machinery.

The third seam is future-proofing. 0.169 deferred WAL, checkpointing,
heap-to-stable flush, cached stable stores, journaled stores, migration tooling,
SQL storage syntax, and relation semantic expansion. 0.170 should lay the
capability layer that lets those later minors be additive instead of invasive.

## Summary

0.170 introduces a storage capability descriptor for every store storage mode
and routes durability, recovery, diagnostics, relation admission, and mixed
write planning through capabilities instead of direct stable/heap checks.

This is a behavior-preserving architecture line. It makes the 0.169 stable/heap
contracts explicit and mechanically enforced; it does not broaden storage
semantics.

## Capability Ownership

Storage capabilities are derived from accepted store storage metadata and then
carried through runtime store registry/store-handle surfaces.

Allowed capability authorities:

- accepted schema/catalog store declarations;
- generated store wiring only as proposal/bootstrap input;
- runtime store registry handles after accepted registration.

Disallowed capability authorities:

- generated `EntityModel`, `IndexModel`, or `SchemaInfo` as runtime fallback
  authority;
- current Rust type order;
- diagnostics DTOs;
- inferred allocation identity;
- backend implementation types outside store-wrapper internals.

Capabilities describe the contract for a store role. They are not allocation
identity. Stable allocation identity remains `memory_id + stable_key`; heap
allocation identity remains absent.

## Capability Model

Initial storage capability surface:

```rust
pub struct StoreStorageCapabilities {
    pub storage_mode: StoreStorageMode,
    pub allocation_identity: AllocationIdentityCapability,
    pub durability: StoreDurability,
    pub recovery: StoreRecoveryCapability,
    pub commit_participation: CommitParticipation,
    pub schema_metadata: SchemaMetadataCapability,
    pub relation_source: RelationSourceCapability,
    pub relation_target: RelationTargetCapability,
    pub live_validation: LiveValidationCapability,
}
```

`storage_mode` may be carried for diagnostics/rendering, but policy code must
branch only on capability axes. Source audits should treat policy matches on
`capabilities.storage_mode` the same as direct matches on `StoreStorage::Heap`
or `StoreStorage::Stable`.

Role-specific enums should own the meaning of each axis:

```rust
pub enum AllocationIdentityCapability {
    Present,
    Absent,
}

pub enum StoreDurability {
    Durable,
    Volatile,
}

pub enum StoreRecoveryCapability {
    StableCommitReplay,
    None,
}

pub enum CommitParticipation {
    Durable,
    LiveOnly,
}

pub enum SchemaMetadataCapability {
    DurableAcceptedHistory,
    LiveRebuiltMetadata,
}
```

The exact names can change, but the implementation should prefer narrow enums
over loose boolean flags such as `is_heap`, `is_durable`, or
`participates_in_commit`.

Initial capability values:

| Axis | Stable | Heap |
| --- | --- | --- |
| Allocation identity | present | absent |
| Durability | durable | volatile |
| Recovery | stable commit replay | none |
| Commit participation | durable | live-only |
| Schema metadata | durable accepted history | live rebuilt metadata |
| Relation source | durable source | live source |
| Relation target | durable target | volatile target |
| Live validation | supported | supported |

The capability axes are intentionally small. Adding `Journaled`, `CachedStable`,
or WAL-backed storage later should add capability enum values and derivation
logic, not a new set of direct backend checks in executor, relation, diagnostics,
or commit code.

## Goals

- Add a storage capability model.
- Derive stable and heap capabilities from the existing schema-level
  `StoreStorage::Stable` and `StoreStorage::Heap` forms.
- Expose one runtime-facing capability descriptor through store construction,
  registry, or store handles.
- Make mixed-write durability explicit.
- Move relation admission and runtime fail-closed checks onto source/target
  capabilities.
- Strengthen diagnostics so storage mode, durability, recovery, commit
  participation, allocation identity, and schema-metadata semantics are visible.
- Add source-audit tests that block storage-policy leakage into higher layers.
- Preserve 0.169 stable and heap behavior.

## Non-Goals

- No WAL.
- No checkpointing.
- No heap-to-stable flush.
- No journaled backend.
- No cached stable backend.
- No SQL storage syntax expansion unless a narrow diagnostics rendering test
  needs it.
- No stable ABI change.
- No stable key generation change.
- No physical stable-memory ID change.
- No relation semantic expansion.
- No compatibility shims, aliases, or fallbacks.

## Hard-Cut Rules

- Do not keep heap-specific relation or commit policy as a parallel path once a
  capability gate owns that decision.
- Do not retain old helper names that imply stable/heap policy is still decided
  by backend kind.
- Do not add compatibility aliases for capability DTO fields during this
  pre-1.0 line.
- Do not make capability derivation fall back to generated model reconstruction
  when accepted runtime store metadata is unavailable.
- Do not treat capability mismatches as allocation identity mismatches.

## Mixed-Write Contract

Public write execution can classify affected stores as:

```text
durable_only
live_only
mixed_durable_and_live
```

Mixed writes remain allowed. Any durable result, trace, diagnostic, or internal
commit classification must distinguish stable commit participation from heap
live side effects. No user-visible success path should imply atomic durable
commit across stable and heap stores.

The classification may remain internal/proof-facing in 0.170. It should become
public only where an existing diagnostic or execution trace surface already
needs to explain durability semantics.

0.170 separates live execution atomicity from durable recovery atomicity.

A mixed write may remain live-transactional if the existing write/session model
already stages or rolls back all live mutations on failure. However, successful
mixed writes still have only per-store durability semantics:

- stable mutations participate in durable commit/recovery;
- heap mutations are live side effects only;
- recovery after upgrade may observe the durable stable portion without the heap
  portion.

Tests must cover failure ordering so a failed mixed write does not accidentally
leave heap side effects behind unless that is already the documented public
write/session behavior.

## Relation Contract

Relation admission and runtime validation should consume source/target storage
capabilities instead of hard-coding stable/heap checks.

0.170 must preserve 0.169 behavior:

- stable source -> heap target strong relation rejects;
- stale accepted stable -> heap metadata fails closed at runtime;
- heap source -> heap target live validation remains supported;
- heap source -> stable target live validation remains supported if already
  admitted by the existing relation model.

Non-strong relation behavior across stable/heap boundaries must also remain
explicit. If weak or diagnostic relation paths continue to be non-enforcing,
tests and docs should say so directly rather than relying on the strong-relation
proof rows.

## Diagnostics Contract

Diagnostics should expose capability contracts without moving or redefining
existing stable allocation fields.

Minimum rendering concepts:

```text
storage: stable | heap
durability: durable | volatile
commit: durable | live-only
recovery: stable-replay | none
allocation: present | absent
schema_metadata: durable-accepted-history | live-rebuilt-metadata
```

Stable diagnostics remain semantically unchanged. Heap diagnostics remain
explicit about absent allocation identity and live-only schema metadata.

Capability diagnostics must not imply that a heap store has a stable memory
allocation, and must not imply that stable allocation fingerprints or schema
metadata are allocation identity.

## Source-Audit Contract

Executor, relation, reconciliation, diagnostics, and commit code should not
match directly on stable/heap except through capability APIs.

Allowed storage-mode branching:

- schema model and validation;
- derive parsing;
- generated wiring;
- store construction;
- store-wrapper internals;
- tests and proof fixtures.

Disallowed storage-mode branching:

- executor policy;
- relation admission/runtime policy outside the capability gate;
- diagnostics semantics outside capability projection/rendering;
- commit/recovery classification outside capability projection;
- schema reconciliation policy outside accepted schema authority.

Concrete source-audit command shape:

```text
rg -n "StoreStorage::Heap|StoreStorage::Stable|DataStoreBackend::Heap|IndexStoreBackend::Heap|SchemaStoreBackend::Heap" \
  crates/icydb-core/src/db \
  --glob '!**/data/**' \
  --glob '!**/index/**' \
  --glob '!**/schema/**' \
  --glob '!**/registry/**'
```

Expected result: no executor, relation, reconciliation, diagnostics, cursor, or
commit policy code branches directly on backend/storage mode after the
capability gate lands.

Capability consumers may match on capability enums. Store construction and
store-wrapper internals may still match on backend/storage mode.

Registry-specific source-audit command shape:

```text
rg -n "StoreStorage::Heap|StoreStorage::Stable|DataStoreBackend::Heap|IndexStoreBackend::Heap|SchemaStoreBackend::Heap" \
  crates/icydb-core/src/db/registry
```

Expected registry hits are limited to capability derivation/projection. The
registry must not own executor, relation, commit, or diagnostics policy beyond
capability projection.

## Patch Slices

### 0.170.0 — Design And Capability Model

- Land this design doc.
- Add `StoreStorageCapabilities` or the final equivalent.
- Define stable and heap capability derivation.
- Add focused model tests for stable, heap, and one test-only future capability
  value.
- Do not change runtime behavior yet.

### 0.170.1 — Registry And Diagnostics Integration

- Attach capabilities to store registry snapshots or store handles.
- Extend diagnostics with durability/recovery/commit capability fields.
- Preserve existing stable diagnostic fields.

### 0.170.2 — Test Fixture Macro Cleanup

- Record the approved test-support diversion that simplified the schema
  fixture macros.
- Keep this separate from the storage capability proof rows.

### 0.170.3 — Mixed Mutation Plan Classification

- Classify public write paths by affected store capabilities.
- Assert stable-only, heap-only, and mixed classifications.
- Prove mixed writes do not advertise atomic durable commit semantics.
- Prove failed mixed writes do not accidentally leave heap side effects behind,
  or document the existing live-write semantics if they intentionally do.
- Keep classification internal/proof-facing unless an existing diagnostic
  surface needs to expose it.

### 0.170.4 — Capability-Based Relation Admission

- Replace heap-specific relation checks with source/target capability checks.
- Preserve 0.169 relation behavior exactly.
- Keep runtime stale-metadata fail-closed behavior.
- Add explicit weak/non-strong relation coverage so only strong-relation
  behavior changes are inferred from strong-relation tests.

### 0.170.5 — Runtime Boundary Audit

- Add source-audit commands that ensure executor, relation, commit,
  diagnostics, and reconciliation code use capability APIs instead of direct
  stable/heap matching.
- Keep storage-mode branching limited to the allowed zones.

### 0.170.6 — Closeout

- Close the C1-C15 proof matrix.
- Record regression evidence against 0.169 heap behavior.
- Record stable ABI/no-codec-change evidence.

- `cargo clippy -p icydb-core --all-targets -- -D warnings`;

Do not claim full workspace test coverage unless `cargo test --workspace`
actually runs.

## What Not To Make 0.170

0.170 should not be "Durable Heap WAL." WAL is too easy to get wrong before the
system has a generic vocabulary for durable, recoverable,
commit-participating, and volatile-target storage.

0.170 should not become "SQL syntax for storage modes" unless the implementation
needs a tiny SQL-facing rendering proof. SQL syntax is useful, but it should
consume the capability model instead of embedding stable/heap policy.
