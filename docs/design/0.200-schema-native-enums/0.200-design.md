# 0.200 Schema-Native Enums

Status: proposed design line after the 0.199 technical-debt audit.

0.200 makes enum definitions schema-native so enum values are admitted,
stored, queried, indexed, decoded, and evolved through accepted schema
authority. Generated Rust enums remain the typed authoring and API surface, but
generated metadata must not be the runtime source of truth after a schema
snapshot has been accepted.

This is a schema-evolution design, not a general type-system rewrite.

## Purpose

IcyDB already models enums in several places:

- schema macros generate Rust enum types and variant metadata;
- `FieldKind::Enum` carries enum path and variant payload metadata;
- accepted schema snapshots persist enum path and variant metadata inside
  `PersistedFieldKind::Enum`;
- query predicate normalization resolves loose enum literals to the accepted
  enum path;
- structural enum payload decode can use variant payload metadata.

Those pieces are useful, but they are not yet one schema-native enum authority.
The current shape can still drift because some boundaries validate only
`Value::Enum` shape or enum path, not accepted variant membership and payload
shape. Enum definition changes also do not have explicit mutation vocabulary,
so transition policy sees them as ordinary field-kind differences.

0.200 should close that gap.

## Hard Invariants

- Accepted schema snapshots are the runtime enum authority.
- Every accepted schema snapshot exposes one snapshot-wide enum registry view.
- No same-path conflicting enum definition can publish.
- Generated enum metadata is proposal input only after initial compilation.
- Every enum write, default, query literal, decoded payload, structural patch,
  index key, rebuilt index key, and grouped key must validate against the same
  accepted-value admission boundary.
- Loose enum values are construction-time only and must canonicalize to strict
  accepted values before planning, plan hashing, cache identity construction,
  `EXPLAIN` evidence, index route selection, storage, or row decode.
- Defaults are canonicalized into strict accepted values during schema snapshot
  acceptance. Runtime default materialization must not consult generated enum
  metadata as fallback authority.
- Unsupported enum definition changes fail closed before snapshot publication.
- SQL DDL, if added, is a frontend onto schema-owned enum mutations.
- SQL enum DDL exists only after an accepted enum catalog exists.
- Typed Rust decode behavior for accepted variants absent from the current
  generated Rust enum must be explicit and tested.
- Before `1.0.0`, do not add compatibility aliases, legacy spellings, or
  fallback decoding for retired enum schema shapes.
- Initial enum ordering is equality-only unless a byte-exact ordering policy is
  introduced in accepted schema and implemented consistently in query and index
  code.

## Current Drift Points

The current code has enough enum metadata to support schema-native behavior,
but enforcement is spread out:

- `FieldKind::Enum { path, variants }` carries generated enum identity and
  per-variant payload metadata.
- `PersistedFieldKind::Enum { path, variants }` stores an owned accepted
  version of that metadata.
- Predicate normalization checks strict enum path and resolves loose enum path,
  but it does not own full variant membership validation.
- SQL/write literal typing currently reduces enum fields to a coarse enum
  scalar class.
- `FieldStorageDecode::Value` stores a generic runtime `Value` envelope; enum
  membership must be checked separately against the accepted field contract.
- `SchemaTransitionPlanKind` and `SchemaMutationDelta` do not name enum
  mutations, so enum additions, removals, renames, and payload-shape changes are
  generic incompatible field contract changes.

That split leaves too many places where a valid-looking `Value::Enum` can be
accepted without proving it is a valid value for the accepted enum type.

## Engine Comparison

PostgreSQL and CockroachDB are the best precedent: enums are catalog-native
types created with `CREATE TYPE ... AS ENUM`, referenced by columns, and altered
with `ALTER TYPE`. They treat each enum type as distinct and require explicit
casts for cross-type comparison. PostgreSQL enum ordering is declaration-order
based. PostgreSQL supports adding and renaming enum values, but does not remove
existing values or change existing sort order except by dropping and recreating
the type.

CockroachDB is also catalog-native. It supports enum type creation, value
addition, rename operations, and value drops. Its drop-value behavior requires
searching rows that could contain the value, which is the right precedent for
IcyDB's proof-or-rewrite stance.

MySQL is a weaker fit. Its `ENUM` is column-local and ordinal-backed. Sort
order and numeric behavior come from declaration position, which is exactly the
kind of hidden drift IcyDB should avoid.

SQLite is also a weaker fit. It has no native enum and typically models enum
domains with text plus `CHECK` constraints or lookup tables. That is useful as
a SQL compatibility pattern, but it is not enough for IcyDB's typed
schema/runtime authority model.

References:

- PostgreSQL enumerated types: <https://www.postgresql.org/docs/current/datatype-enum.html>
- PostgreSQL `ALTER TYPE`: <https://www.postgresql.org/docs/current/sql-altertype.html>
- CockroachDB `ENUM`: <https://www.cockroachlabs.com/docs/stable/enum>
- CockroachDB `DROP VALUE`: <https://www.cockroachlabs.com/docs/stable/drop-value>
- MySQL `ENUM`: <https://dev.mysql.com/doc/refman/8.4/en/enum.html>
- SQLite datatypes and `CHECK`: <https://www.sqlite.org/datatype3.html>

## Target Model

0.200 introduces one schema-owned enum contract:

```rust
struct AcceptedSchemaContracts {
    enums: AcceptedEnumRegistry,
}

struct AcceptedEnumRegistry {
    by_path: BTreeMap<String, AcceptedEnumType>,
}

struct AcceptedEnumType {
    id: Option<EnumTypeId>,
    path: EnumPath,
    variants: BTreeMap<EnumVariantName, AcceptedEnumVariant>,
    ordering: EnumOrderingPolicy,
    source: EnumDefinitionSource,
}

struct AcceptedEnumVariant {
    name: EnumVariantName,
    payload_kind: Option<PersistedFieldKind>,
    payload_storage_decode: FieldStorageDecode,
}

enum EnumDefinitionSource {
    GeneratedProposal { generated_path: String },
    SqlCatalog,
    Internal,
}

enum EnumOrderingPolicy {
    EqualityOnly,
}
```

This is a design shape, not a required exact type layout. The important point is
that enum path, variant membership, payload kind, payload storage decode,
ordering policy, and source are one accepted-schema contract.

For 0.200 embedded-registry mode, accepted enum identity is accepted
schema/catalog-snapshot scoped `EnumPath`. All enum definitions visible to the
publication decision share this namespace. Current per-entity persisted schema
artifacts may be aggregated into that registry view during implementation, but
a per-entity-only check is an implementation stepping stone, not the 0.200
closeout contract. If path is the identity, then an enum path or Rust type path
rename is a schema type rename and must be rejected until a rewrite or stable-ID
migration exists.

The first implementation slice must build `AcceptedEnumRegistry` from the
current embedded `PersistedFieldKind::Enum { path, variants }` shape:

```rust
impl AcceptedEnumRegistry {
    fn insert_or_verify_equivalent(
        &mut self,
        source_field: FieldPath,
        enum_type: AcceptedEnumType,
    ) -> Result<(), EnumDefinitionConflict>;
}
```

Before snapshot publication, every accepted enum identity visible in the
accepted schema must resolve to exactly one accepted definition. Registry
construction must recursively traverse enum payload kinds and collect any
nested enum definitions before publishing the snapshot.

`EnumDefinitionSource` is diagnostic and policy metadata. It is not part of
semantic enum equivalence in embedded-registry mode unless a source-specific
policy explicitly opts in. It is retained for diagnostics, migration reporting,
and future catalog policy.

## Canonical Enum Equivalence

Duplicate enum identities are accepted only when their semantic canonical enum
definitions are equal:

```rust
struct CanonicalEnumDefinition {
    path: EnumPath,
    ordering: EnumOrderingPolicy,
    variants: BTreeMap<EnumVariantName, CanonicalEnumVariant>,
}

struct CanonicalEnumVariant {
    name: EnumVariantName,
    payload_kind: Option<CanonicalPersistedFieldKind>,
    payload_storage_decode: FieldStorageDecode,
}
```

`CanonicalPersistedFieldKind` means a recursively canonicalized
`PersistedFieldKind`, not a required new persisted type.

The canonical comparison includes path, ordering policy, variant names, payload
presence, canonical payload kind, and payload storage decode. It excludes source
field path and diagnostic source metadata unless a source-specific policy
explicitly opts in.

Conflict diagnostics should identify both fields and the exact reason:

```rust
struct EnumDefinitionConflict {
    path: EnumPath,
    first_source_field: FieldPath,
    second_source_field: FieldPath,
    reason: EnumDefinitionConflictReason,
}

enum EnumDefinitionConflictReason {
    DifferentOrderingPolicy,
    MissingVariant { variant: EnumVariantName },
    ExtraVariant { variant: EnumVariantName },
    DifferentPayloadPresence { variant: EnumVariantName },
    DifferentPayloadKind { variant: EnumVariantName },
    DifferentPayloadStorageDecode { variant: EnumVariantName },
}
```

## Embedded Registry Mode vs Catalog Mode

0.200 has two distinct modes. They must not be blurred.

Embedded-registry mode:

- enum definitions are discovered from accepted fields;
- unused enum types do not exist;
- `CREATE TYPE`, `DROP TYPE`, and `ALTER TYPE` are unavailable;
- `AddEnumType` is unavailable except as part of accepting a field proposal
  that mentions the enum;
- the accepted registry rejects same-path conflicting definitions before
  publication.

Catalog mode:

- enum definitions live in a separate accepted enum catalog;
- fields reference enum entries by accepted type ID or path;
- unused enum types can exist;
- `CREATE TYPE`, `DROP TYPE`, `ALTER TYPE`, and standalone `AddEnumType`
  become meaningful schema mutations;
- `DROP TYPE` must reject while accepted fields reference the enum.

The first 0.200 implementation should start with embedded-registry mode.
SQL enum DDL waits for catalog mode.

## Accepted Value Admission

Enum admission should be part of a general accepted-value admission boundary:

```rust
fn admit_value_against_accepted_kind(
    contracts: &AcceptedSchemaContracts,
    kind: &PersistedFieldKind,
    storage_decode: FieldStorageDecode,
    value: &Value,
    boundary: AdmissionBoundary,
) -> Result<AcceptedValue<'_>, ValueAdmissionError>;
```

Enum-specific logic is one branch:

```rust
fn admit_enum_value(
    registry: &AcceptedEnumRegistry,
    contract: &AcceptedEnumType,
    value: &ValueEnum,
    boundary: AdmissionBoundary,
) -> Result<AcceptedEnumValue<'_>, EnumValueAdmissionError>;
```

Admission rules:

- A strict enum value must have a path matching the accepted enum path.
- A loose enum value may be resolved only when exactly one expected enum
  contract is known at a construction boundary.
- Persisted values, decoded values, index keys, and grouped keys must be
  strict.
- The variant name must exist in the accepted variant table.
- A unit variant must not carry a payload.
- A payload variant must carry a payload.
- The payload must recursively pass `admit_value_against_accepted_kind` using
  the accepted payload kind and storage decode contract.
- Unknown variants are user/data errors at write/query boundaries and
  corruption/invariant errors at accepted decode boundaries.

Boundary policy:

| Boundary | Loose path allowed? | Unknown variant classification | Payload mismatch classification |
| --- | --- | --- | --- |
| Structural write | Only with expected field contract | user/data error | user/data error |
| SQL write | Only through target column context | SQL typing error | SQL typing error |
| Field default | No after acceptance | schema acceptance error | schema acceptance error |
| Query literal | Only with exactly one expected enum contract | query planning error | query planning error |
| Structural patch | Only with expected field contract | user/data error | user/data error |
| Accepted row decode | No | corruption/invariant error | corruption/invariant error |
| `ByKind` decode | No | corruption/invariant error | corruption/invariant error |
| Index key build | No | corruption/invariant error | corruption/invariant error |
| Index rebuild | No | corruption/rebuild error | corruption/rebuild error |
| Group key build | No | internal invariant error | internal invariant error |

Loose values must canonicalize to strict accepted enum values before planning,
plan hashing, cache identity construction, `EXPLAIN` evidence, index route
selection, storage, or row decode. The same logical query must not have
different cache identities because one path used a loose enum spelling.

## Runtime Boundaries

The accepted-value admission boundary must be consumed by these paths:

- structural writes and SQL writes before slot encoding;
- accepted row decode after generic `FieldStorageDecode::Value` decode;
- accepted `ByKind` enum decode and validation;
- field defaults and generated/default-backed additive fields;
- predicate literal normalization and validation;
- structural patches;
- index key construction and index rebuild validation for enum fields;
- grouped/aggregate equality paths when enum values appear in grouped keys.

Index keys for enum fields must never encode variant `Vec` position. Equality
keys must include accepted enum identity and variant label. Payload variants
must include the canonical admitted payload representation.

Under `EnumOrderingPolicy::EqualityOnly`, enum range operators and ordering
operations are unavailable. Reject `<`, `<=`, `>`, `>=`, enum `ORDER BY`,
`MIN(enum)`, `MAX(enum)`, and range index routing until a byte-exact ordering
policy is introduced and implemented across comparisons, indexes, and SQL.
Deterministic registry iteration order is an implementation detail and does
not define enum comparison, SQL ordering, aggregate ordering, or range index
semantics.

`FieldKind::accepts_value` should remain a coarse generated-model helper or
become stricter only if it can consume accepted enum metadata. Runtime accepted
schema paths should not rely on a coarse `Value::Enum` match.

## Default Materialization

Generated defaults are proposal input, not runtime authority.

At schema snapshot acceptance:

1. resolve generated/default proposals against the proposed accepted enum
   contract;
2. convert them to strict `Value::Enum` values with the accepted path;
3. validate unit/payload shape and recursive payload contracts;
4. persist canonical defaults inside the accepted schema snapshot.

Runtime row materialization uses the accepted default value from the snapshot.
It must not call generated Rust default metadata as a fallback.

## Mutation Vocabulary

0.200 adds enum-aware mutation requests and transition buckets. The exact names
can change, but the semantics should be explicit:

```rust
enum SchemaMutation {
    AddEnumType { ... },
    DropEnumType { ... },
    RenameEnumType { ... },
    AddEnumVariant { ... },
    RenameEnumVariant { ... },
    DropEnumVariant { ... },
    ChangeEnumVariantPayload { ... },
    ChangeEnumVariantStorageDecode { ... },
    ChangeEnumOrderingPolicy { ... },
}

enum SchemaMutationDelta<'a> {
    EnumDefinitionUnchanged { path: &'a str },
    AddEnumType { path: &'a str },
    DropEnumType { path: &'a str, references: Vec<FieldPath> },
    AddEnumVariant {
        path: &'a str,
        variant: &'a str,
        payload_kind: Option<&'a PersistedFieldKind>,
        safety: EnumAddVariantSafety,
    },
    RenameEnumVariant {
        path: &'a str,
        from: &'a str,
        to: &'a str,
        required: RewriteRequirement,
    },
    DropEnumVariant {
        path: &'a str,
        variant: &'a str,
        required: ProofOrRewriteRequirement,
    },
    ChangeEnumVariantPayload {
        path: &'a str,
        variant: &'a str,
        required: RewriteRequirement,
    },
    ChangeEnumVariantStorageDecode {
        path: &'a str,
        variant: &'a str,
        required: RewriteRequirement,
    },
    ChangeEnumOrderingPolicy {
        path: &'a str,
        from: EnumOrderingPolicy,
        to: EnumOrderingPolicy,
        supported: bool,
    },
    IncompatibleEnumChange {
        path: &'a str,
        reason: EnumIncompatibilityReason,
    },
}
```

`AddEnumType` and `DropEnumType` are catalog-mode mutations. In embedded-registry
mode, a new enum definition can appear only as part of accepting a field
proposal that mentions it. The transition layer must not treat enum changes as
generic field-kind churn once enum migration support exists.

`AddEnumVariant` safety must be split by surface:

- storage safety;
- index safety;
- query safety;
- typed Rust decode safety.

The label "metadata-only safe" is not enough for enum variants.

## Compatibility Matrix

| Operation | Initial 0.200 classification | Conditions |
| --- | --- | --- |
| Exact enum definition match | metadata-only safe | same path, variants, payload kinds, storage decode, and ordering |
| Add unused enum type | catalog-only safe | unavailable in embedded-registry mode |
| Add unit variant | data-safe, not automatically API-safe | key/storage encoding must be path/label based; generated enum fields require current generated Rust proposal to contain the variant unless dynamic-only behavior exists |
| Add payload variant | data-safe after centralized admission | SQL cannot author it initially; typed decode caveat applies |
| Rename variant | rewrite required | reject until rewrite runner exists |
| Drop variant | proof or rewrite required | proof must scan every row/storage/index location where value can appear |
| Change payload kind | rewrite required | reject until rewrite runner exists |
| Change payload storage decode | rewrite required | reject until rewrite runner exists |
| Rename enum type/path | rewrite or stable-ID migration required | reject under path-as-identity model |
| Reorder variants | unsupported | no declaration-order or ordinal behavior |
| Add alias for an old variant name | unsupported pre-1.0 | reject |
| Change ordering policy | unsupported initially | reject while query/index ordering semantics are equality-only |

Under `EqualityOnly`, enum ordering is not available. Do not import MySQL-style
ordinal behavior, PostgreSQL declaration-order behavior, or SQL collation-aware
behavior by accident. If a future `Utf8ByteLabelOrder` policy is added, it must
define comparison bytes, SQL collation interaction, index encoding, and
aggregate behavior mechanically.

## SQL Surface

SQL enum DDL is not part of embedded-registry mode.

Allowed before an accepted enum catalog exists:

- SQL reads and writes against generated enum fields using accepted field
  contracts;
- bare string enum literals only when a target enum column gives exactly one
  expected enum contract;
- explicit enum casts only if the enum type already exists through accepted
  field metadata and resolves to exactly one accepted enum identity.

Forbidden before an accepted enum catalog exists:

- `CREATE TYPE`;
- `DROP TYPE`;
- `ALTER TYPE`;
- unused enum types.

Once catalog mode exists, SQL DDL may lower to schema mutation requests:

```sql
CREATE TYPE status AS ENUM ('Draft', 'Active');
ALTER TYPE status ADD VALUE 'Archived';
DROP TYPE status;
```

Initial catalog-mode SQL limitations:

- SQL-authored enums are label-only.
- Payload-bearing Rust enums remain generated/proposal-authored.
- `DROP TYPE` rejects while any accepted field references the enum.
- `ALTER TYPE ... ADD VALUE ... BEFORE/AFTER ...` rejects under
  equality-only ordering.
- `RENAME VALUE`, `DROP VALUE`, type rename, and value reorder reject until
  physical rewrite or proof semantics exist.
- SQL DDL does not define enum mutation semantics; schema mutation planning
  does.

SQL-authored labels need one accepted grammar. Initial recommendation:

- labels are non-empty UTF-8 strings;
- labels are case-sensitive;
- whitespace is significant;
- NUL and control characters are rejected;
- maximum label length is bounded;
- Rust-generated labels remain Rust identifier-derived, but catalog-authored
  SQL labels do not have to be valid Rust identifiers.

## Generated Rust Surface

The Rust macro surface should continue to generate:

- the Rust enum type;
- `EnumValue` and runtime value bridges;
- variant name constants;
- generated proposal metadata for enum path, variants, payload kinds, and
  payload storage decode.

Generated metadata becomes a proposal for the accepted enum contract. After
acceptance, runtime paths use accepted schema.

Payload-bearing variants are an IcyDB-specific extension beyond ordinary SQL
enums. Their migration support must be stricter than label-only engines:

- payload absence/presence is part of the variant contract;
- payload kind changes are data rewrites;
- wrapper/indirect payloads must not be reconstructed from generated metadata
  at runtime.

### Typed Decode Compatibility

Accepted schema can evolve independently of the current generated Rust enum
type. A dynamic value surface can represent a new accepted variant before Rust
code knows about it, but a generated Rust enum field cannot decode that value
exhaustively.

Pre-1.0 policy:

- generated enum fields reject snapshot publication if the accepted variant set
  is not representable by the current generated enum proposal;
- dynamic enum fields may later opt into accepted variants absent from generated
  Rust types, but that is not part of the first 0.200 slice;
- typed Rust decode must not use generated metadata as fallback authority;
- read-time typed decode failures, if introduced later, must be explicit
  accepted-schema/newer-code diagnostics.

Therefore, adding a variant to a generated enum field is publishable only when
the current generated proposal includes that variant. SQL-only variants for
generated enum fields are rejected until a dynamic-field policy exists.

If multiple generated fields share one accepted enum identity, every generated
field proposal using that identity must represent the same accepted variant
set. Otherwise, publication fails before the snapshot becomes accepted.

## Implementation Slices

1. Accepted enum registry over the current persisted shape.

   Build one snapshot-wide `AcceptedEnumRegistry` from embedded
   `PersistedFieldKind::Enum` definitions. Accept duplicate paths only when
   their semantic canonical enum definitions are equal. Reject conflicting
   variant sets, payload kinds, storage decode, or ordering before snapshot
   publication. Do not add SQL DDL or persisted format rewrites.

2. Shared accepted-value admission.

   Implement `admit_value_against_accepted_kind` and the enum-specific
   `admit_enum_value` branch. Cover strict path validation, loose resolution
   only with one expected contract, variant membership, unit/payload shape,
   recursive payload checks, and boundary-specific error classification. Do not
   add compatibility fallback.

3. Route writes, defaults, patches, and predicates through admission.

   Close the normal user input gaps for structural writes, SQL writes,
   generated/default-backed additive fields, predicate literal normalization,
   and structural patches. Loose literals must canonicalize before planning,
   cache identity, `EXPLAIN`, and index route selection.

4. Decode, index, and grouped-key validation.

   Validate accepted row decode after generic `Value` decode, accepted `ByKind`
   enum decode, index key construction, index rebuild, and grouped-key equality.
   Do not add repair tooling or fallback decode. Same-label values from
   different enum identities must not collide.

5. Enum-aware transition deltas.

   Name enum changes explicitly: exact match, add variant, rename/drop variant,
   payload kind change, storage decode change, type/path rename, ordering
   change, and unsupported alias/reorder cases. Unsupported changes fail closed
   before publication with enum-specific diagnostics.

6. Catalog mode, only if needed for SQL DDL.

   Promote enum definitions to a separate accepted catalog. Allow unused enum
   types, field references to catalog enum entries, standalone `AddEnumType`,
   and `DROP TYPE` reference checks. Do not add rename/drop/reorder without
   proof or rewrite support.

7. SQL DDL frontend.

   Lower label-only `CREATE TYPE`, `ALTER TYPE ... ADD VALUE`, and unreferenced
   `DROP TYPE` to the same schema mutation path. Reject payload-bearing SQL
   enums, `RENAME VALUE`, `DROP VALUE`, type rename, and `BEFORE/AFTER`
   positioning in the initial SQL subset.

## Validation Plan

Snapshot registry tests:

- same enum path with identical definitions across two fields is accepted;
- same enum definition from two fields with variants declared in different
  order is accepted under `EqualityOnly` when canonical variant maps match;
- same enum path with different variants is rejected;
- same enum definition with different `EnumDefinitionSource` is accepted
  because source is diagnostic-only in embedded-registry mode;
- same enum path with same variants but different payload kind is rejected;
- same enum path with same payload kind but different storage decode is
  rejected;
- nested enum payload definitions participate in registry validation;
- enum inside payload kind is collected and conflict-checked even when no
  top-level field directly uses that nested enum;
- conflict diagnostics include both field paths and exact reason;
- unused enum type is impossible in embedded mode and possible only in catalog
  mode.

Admission tests:

- strict path match is accepted;
- strict path mismatch is rejected;
- loose path is accepted only under exactly one expected enum field;
- loose path rejects when the expected field is not enum;
- loose path rejects when expression has no expected enum type;
- unknown unit and payload variants reject;
- unit variant with payload rejects;
- payload variant without payload rejects;
- payload variant with wrong payload kind rejects;
- payload variant with wrong storage decode rejects;
- nested enum payload unknown variant rejects.

Defaults tests:

- accepted default uses strict accepted enum path;
- generated default with unknown variant rejects at snapshot acceptance;
- generated default with wrong payload shape rejects at snapshot acceptance;
- additive field default materialization does not call generated metadata at
  runtime;
- old generated default cannot resurrect a retired variant.

Predicate and query tests:

- loose query literal canonicalizes to strict enum before planning;
- unknown loose query literal rejects before planning;
- strict wrong-path query literal rejects before planning;
- enum literal in `IN (...)` validates every element;
- mixed enum paths in `IN (...)` reject;
- field-to-field enum comparison rejects different enum paths;
- query plan hash/cache identity includes canonical enum identity, not loose
  spelling;
- enum ordering and range queries reject under `EqualityOnly`.

SQL tests:

- SQL write to enum column accepts valid label under target column context;
- SQL write to enum column rejects unknown label;
- SQL bare string enum literal rejects without target enum context;
- SQL explicit cast rejects if accepted registry resolution is ambiguous;
- SQL explicit cast to wrong enum type rejects;
- SQL cross-enum comparison rejects;
- SQL `CREATE TYPE`, `DROP TYPE`, and `ALTER TYPE` are unavailable before
  catalog mode;
- catalog-mode `DROP TYPE` rejects while any accepted field references the
  enum;
- `ALTER TYPE ... ADD VALUE BEFORE/AFTER ...` rejects under equality-only
  ordering;
- SQL payload-bearing enum authoring rejects in the first SQL subset.

Rust typed API tests:

- generated Rust enum variant writes are admitted through accepted schema;
- generated Rust enum variant absent from accepted schema rejects;
- accepted schema variant absent from generated Rust type rejects publication
  for generated enum fields;
- two generated fields with the same enum path but different generated variant
  sets reject publication;
- SQL-only `AddEnumVariant` against a generated enum field rejects with typed
  decode incompatibility reason;
- generated proposal metadata cannot be used as runtime fallback after accepted
  schema differs.

Decode and corruption tests:

- accepted `FieldStorageDecode::Value` decode rejects unknown enum variant;
- accepted decode rejects unit variant carrying payload;
- accepted decode rejects payload variant missing payload;
- accepted decode rejects nested invalid enum payload;
- public read aborts on invalid stored enum;
- diagnostic/admin path reports corruption without coercing;
- index rebuild fails on invalid stored enum value.

Index and group tests:

- enum index key construction includes enum identity;
- same label under different enum paths does not collide;
- enum payload variant index key includes canonical payload encoding;
- `AddEnumVariant` does not require index rebuild only if key encoding is
  path/label based;
- grouped equality treats enum identity as part of the key;
- enum `ORDER BY`, `MIN`, `MAX`, and range index routing reject even though
  internal registry storage uses `BTreeMap`;
- equality lookup still works under `EqualityOnly`.

Transition tests:

- exact definition match produces metadata-only/no-op classification;
- add unit variant is named `AddEnumVariant`, not generic field mismatch;
- add payload variant is named and gated on payload admission availability;
- rename variant is enum-specific incompatible/rewrite-required;
- drop variant is enum-specific proof-or-rewrite-required;
- payload kind change is enum-specific rewrite-required;
- payload storage decode change is enum-specific rewrite-required;
- enum path rename is enum-specific rewrite-required or rejected;
- reorder is rejected;
- alias is rejected pre-1.0.

Broader checks should include existing schema, SQL, and persisted-row focused
tests. No wasm/perf delta is expected from the design doc itself;
implementation slices should report raw non-gzipped `.wasm` deltas when they
land.

## Non-Goals

- No legacy enum aliases.
- No compatibility wrappers for old variant spellings.
- No accepted-but-unknown generated Rust variant publication for generated enum
  fields unless a dynamic-field policy exists.
- No MySQL-style numeric/ordinal enum behavior.
- No PostgreSQL-style declaration-order enum comparison in the initial policy.
- No `BEFORE`/`AFTER` enum insertion under equality-only ordering.
- No SQL enum DDL before catalog mode.
- No SQL payload-bearing enum authoring in the first SQL subset.
- No enum drop or rename without proof/rewrite semantics.
- No runtime fallback reconstruction from generated enum metadata.
- No broad schema draft system; drafts remain a separate design track.

## Closeout Standard

0.200 is complete when:

1. every accepted schema snapshot has a single registry view of enum
   definitions;
2. no same-path enum definition conflict can publish;
3. every enum write, SQL write, structural patch, default, query literal,
   accepted decode, index key, index rebuild, and group key goes through the
   same accepted-value admission path;
4. loose enum values are construction-time only and never reach storage,
   indexes, cache identity, or row decode;
5. defaults are strict accepted values after snapshot acceptance;
6. enum payload validation recursively uses accepted field-kind/storage-decode
   contracts;
7. enum mutation deltas are named and no longer appear as generic field-kind
   churn;
8. unsupported rename/drop/reorder/payload/decode/path changes fail closed
   before publication;
9. SQL enum DDL exists only after catalog semantics exist;
10. generated Rust metadata remains proposal/API input and never becomes
    fallback runtime authority;
11. typed Rust decode behavior for accepted-but-unknown variants is explicitly
    defined and tested;
12. enum ordering is equality-only or mechanically defined in a way that
    matches index and query behavior.
