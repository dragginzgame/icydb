# 0.197 Deterministic Optimizer Canonicalization

Status: implementation in progress; see
[`implementation-results.md`](implementation-results.md) for the current
evidence ledger and remaining closeout gaps.

0.197 should improve IcyDB's optimizer behavior without changing the optimizer
model. IcyDB should remain a deterministic rule-based planner, not a general
cost-based database optimizer. The first target is to make semantically
equivalent primary-key lookup spellings reach the same admitted plan.

The motivating case is an auth mapping table whose principal field is the
external primary key:

```rust
db!()
    .load::<PrincipalToUser>()
    .filter(FilterExpr::eq("pid", pid))
    .try_one()
```

Semantically, this is the same lookup as:

```rust
db!()
    .load::<PrincipalToUser>()
    .by_id(Id::from_key(pid))
    .try_one()
```

IcyDB should be able to prove that equivalence for strict scalar primary-key
equality. Until that proof is reliable across public read admission, auth paths
can use explicit `by_id(...)` lookups as a local workaround. The design goal for
0.197 is to remove the need for that workaround.

## Why Defer From 0.196

0.196 is an ordered-read pushdown and performance-evidence line. It needs clean
before/after matrix evidence for bounded ordered queries, route classification,
and read/materialization counters.

Primary-key predicate canonicalization should be deferred because it changes a
different boundary:

- public read admission may stop rejecting missing `LIMIT` for exact key
  predicates;
- filtered lookup plans may switch from full-scan or index-family rejection to
  `ByKey`;
- `EXPLAIN` and diagnostics will show different selected access paths;
- some failure order may intentionally change, such as `E188` disappearing for
  exact primary-key lookups.

Mixing that into 0.196 would make the ordered-read pushdown evidence harder to
interpret. 0.197 should start from a closed 0.196 baseline.

## Problem

IcyDB has a real deterministic planner and read-admission layer, but not all
semantically equivalent query spellings are guaranteed to reach the same access
proof.

For users, this is surprising:

```text
WHERE primary_key = ?
```

is a primary-key lookup in PostgreSQL, MySQL, SQLite, and most ORM mental
models. In IcyDB, the explicit key API is currently the safer way to state that
proof, especially on public read paths.

That creates two problems:

1. Ergonomics: users must know when to use key APIs instead of natural filters.
2. Safety UX: public-read errors can imply the query is unbounded even when the
   schema proves the predicate is exact-key bounded.

The underlying policy is still correct: public reads must fail closed unless
the planner proves a bounded admitted shape. The gap is that the proof should
exist for strict scalar primary-key equality predicates.

## Design Goal

Add a small deterministic optimizer-canonicalization layer for primary-key
lookup shapes:

1. Recognize strict equality on the scalar primary-key field.
2. Canonicalize it to the same logical access proof as `by_id(...)`.
3. Recognize finite `IN` on the scalar primary-key field as `ByKeys` when the
   list is bounded and type-compatible.
4. Preserve residual predicates without widening cardinality.
5. Make public read admission consume the canonical `ByKey` / `ByKeys` proof
   before applying generic missing-limit or full-scan rejection.
6. Surface the canonicalization in explain/diagnostics so operators can see
   that a filter spelling planned as a key lookup.

The first implementation should be narrow and boring. It should make the common
correct thing work, not introduce a broad rewrite framework.

0.197's ergonomics win is exact-key only: exact-key public reads should not
need fake `.limit(1)` / `.limit(N)` ceremony after canonicalization. General
non-primary-key reads still need bounded page, aggregate, or admin semantics,
and arbitrary high `.limit(...)` calls are not endorsed by this line.

## Canonicalization Result

Primary-key canonicalization should produce one planner-owned result. This
result is the artifact consumed by admission, explain, and execution.

```text
PrimaryKeyCanonicalization
  None { reason }
  ByKey { key_expr, residual_filter, proof }
  ByKeys { key_exprs, raw_count, deduped_count, residual_filter, proof }
  Empty { reason }
```

`None { reason }` means the query is valid, but no exact primary-key proof was
found. The query continues through the existing generic path and may reject as
it does today.

`ByKey { ... }` means exactly one scalar primary-key value is proven. The
returned-row bound is at most one.

`ByKeys { ... }` means a finite set of scalar primary-key values is proven. The
returned-row bound is at most the deterministic deduplicated key count.

`Empty { reason }` means the predicate is proven contradictory without scanning,
for example `pk = a AND pk = b` where `a != b`, or `pk IN ()`. Execution returns
an empty result through the normal response path.

Do not implement this as a boolean "is primary-key equality" flag. The result
must carry enough information for read admission, explain, cache identity, and
execution to agree on the same proof.

## Canonicalization Failure Classes

Canonicalization outcomes must distinguish valid non-application from hard
failure.

`None { reason }` is allowed only for valid query shapes where the primary-key
canonicalization rule does not apply, for example:

- non-primary-key field;
- expression-wrapped key;
- coercive/casefold equality;
- unsupported `OR` shape;
- partial composite primary key.

These are valid shapes that may continue through the existing generic planner.

The following are not `None` fallback cases:

- wrong primary-key value type;
- malformed `IN` list;
- missing required SQL parameter;
- over-budget raw `IN` input;
- over-budget deduplicated key count;
- invalid residual predicate;
- unknown residual field.

Those must fail through validation/admission and must not fall back to scanning.

## Exact Key Authority

The canonical key value must be produced by the same accepted-schema key encoder
used by explicit `by_id(...)` / `by_ids(...)`.

The rule must not infer key identity from generated model fields, Rust struct
field names, frontend aliases, or secondary index metadata alone. Accepted schema
authority owns the scalar primary-key field identity, field type, and key
encoding.

For external primary keys, `filter(pk_field = value)` must canonicalize through
the same key-construction path as `Id::from_key(value)`.

Tests must prove that these shapes use the same logical key and return the same
result or error class:

```rust
db!().load::<E>().filter(FieldRef::new(pk).eq(value)).try_one()
db!().load::<E>().by_id(Id::from_key(value)).try_one()
```

Secondary unique-index equality may become a later exact-access feature, but it
is not primary-key canonicalization unless explicitly designed.

## Preferred First Slice

### Covered Shapes

Admit these scalar single-entity shapes:

```text
pk = literal_or_bound_value
pk IN finite_value_list
pk = value AND residual_filter
residual_filter AND pk = value
duplicate or contradictory primary-key constraints
```

Where:

- `pk` is the entity's scalar primary-key field from accepted schema authority;
- the comparison is strict equality, not text-casefold or numeric widening;
- every value is compatible with the primary-key field type;
- `IN` input is finite and within existing resource, row-budget, and payload
  policies;
- residual predicates are evaluated after the key lookup and cannot increase
  cardinality.

For `pk = value AND residual_filter`, the returned row bound is still at most
one. For `pk IN (...) AND residual_filter`, the returned row bound is at most
the finite key count after deduplication.

If commuted equality (`value = pk`) is already normalized by the shared semantic
predicate path, it may share this rule. If not, it should remain a disabled
reason until a small follow-up adds it deliberately.

### Duplicate And Contradictory Key Constraints

The first slice should handle simple primary-key constraint intersections,
because they are small, deterministic, and prevent unsafe fallback behavior.

Expected behavior:

| Shape | Result |
| --- | --- |
| `pk = a AND pk = a` | collapse to `ByKey(a)` |
| `pk = a AND pk = b`, `a != b` | `Empty` |
| `pk = a AND pk IN (a, b, c)` | narrow to `ByKey(a)` |
| `pk = a AND pk IN (b, c)` | `Empty` |
| `pk IN ()` | `Empty` |
| `pk IN (a, a, a)` | deterministic `ByKeys([a])` or `ByKey(a)` after canonicalization |

Malformed key terms fail validation and must not fall back to scanning.

### Empty Result Terminal Semantics

`Empty` is a proven empty result set, not a scan and not a validation error.

Expected terminal behavior:

| Terminal | `Empty` behavior |
| --- | --- |
| optional zero/one lookup | returns no row / `None` |
| required one-row lookup | uses the same not-found/cardinality helper as an empty `ByKey` result |
| collection/page lookup | returns an empty page/result set |
| count | returns `0` |
| public read admission | admitted as bounded zero-row access unless the contradiction involved an invalid predicate |

`Empty` must perform zero data-store gets and zero index range scans.

### `IN` Budgets And Deduplication

For `pk IN (...)`, admission must enforce all of:

1. raw input term count or parameter-list count;
2. deduplicated key count after strict accepted-schema key encoding.
3. existing request/query payload-size budgets for the encoded key list.

A huge raw list with many duplicates must not be accepted merely because the
deduplicated result count is small. Parsing, lowering, encoding, and
deduplication work must remain bounded by existing endpoint/resource policy.

For variable-width primary keys, raw term count alone is insufficient. A small
number of very large key values must not create excessive encoding, cache-key,
explain, or diagnostic output work.

Deduplication must be deterministic. Prefer encoded-key sort/dedup or a
deterministic ordered set. Do not expose nondeterministic hash iteration order
through result ordering, explain output, cache identity, or tests.

### `ByKeys` Ordering Semantics

`ByKeys` proves a finite row bound. It does not automatically prove output
order.

For 0.197.0, `ByKeys` without explicit `ORDER BY` returns rows in deterministic
encoded primary-key order after deduplication. It does not preserve input-list
order.

Tests must prove:

- `pk IN (b, a, b)` and `pk IN (a, b)` return the same row order;
- deduplication order does not depend on hash iteration;
- SQL and fluent `IN` produce the same deterministic result signature.

If `ORDER BY pk` is requested, the planner may satisfy it from sorted key order
only when the key encoder order matches the requested primary-key order.

If `ORDER BY non_pk` is requested, the query may still be admitted because the
input set is finite and bounded, but the order must be reported as materialized
over the finite `ByKeys` result set.

Cursor pagination over `ByKeys` is out of scope unless the same deterministic
order and cursor contract are explicitly proven.

### Explicitly Deferred Shapes

Do not include these in the first slice:

- composite primary-key partial predicates;
- primary-key range predicates beyond already-supported route families;
- `OR` normalization except simple all-primary-key disjunctions if already
  naturally handled by existing `IN` lowering;
- expression-wrapped primary keys, such as `LOWER(pk) = ?`;
- coercive equality, text casefold, lossy numeric conversion, or string-parsed
  primary-key values;
- join-like, subquery-like, or multi-entity planning;
- changing cursor token format;
- changing persisted formats.

Composite primary-key support can follow later, but only as full-key equality
over all ordered primary-key components. Partial primary-key prefix scans are a
separate planner feature.

## Admission Contract

Read admission should treat canonicalized key access as bounded proof.

For `ByKey`:

- selected access: `ByKey`;
- returned-row bound: `1`;
- scan bound: exact or conservative one-key bound;
- missing explicit `LIMIT` should not reject;
- ordering is irrelevant for a single-row lookup unless the caller explicitly
  requests cursor pagination through a surface that still requires an order.

For `ByKeys`:

- selected access: `ByKeys`;
- returned-row bound: finite key count after deduplication;
- public policy must reject if raw input count exceeds the resource policy;
- public policy must reject if deduplicated key count exceeds the endpoint row
  budget;
- public policy must reject if encoded key-list payload work exceeds the
  request/query payload-size policy;
- no full-scan fallback is allowed when the key list is malformed or over
  budget.

For key access plus residual predicates:

- residual filters may reduce results but never expand them;
- admission may use the key-derived upper bound;
- execution must still evaluate the residual predicate before returning rows;
- explain must show both the key access and the residual filter.

For `ByKeys + residual`, admission bound is the deduplicated key count. Runtime
may perform at most one logical primary-key lookup per deduplicated key before
residual evaluation. Residual evaluation must not trigger fallback scan, index
range scan, or secondary-index probing in 0.197.0.

If the planner cannot prove these facts, the query stays on the existing
generic path and may reject as it does today.

### Validation And Rejection Order

Canonicalization must not hide invalid predicates or invalid values.

| Query shape | Expected behavior |
| --- | --- |
| `pk = valid_value` without `LIMIT` on public read | admitted as `ByKey` |
| `pk IN bounded_values` without `LIMIT` on public read | admitted as `ByKeys` |
| `pk IN over_budget_values` | rejected by read admission before scan |
| `pk = valid_value AND valid_residual` | admitted as `ByKey` plus residual |
| `pk = valid_value AND unknown_field = x` | validation error, not admitted |
| `unknown_field = x AND pk = valid_value` | validation error, not admitted |
| `pk = wrong_type` | type error, not scan fallback |
| coercive or casefold primary-key equality | no canonicalization |
| composite primary-key partial equality | no scalar `ByKey` proof |

The intent is not simply to admit more queries. It is to admit only queries with
a valid exact bound and to keep invalid shapes fail-closed.

Canonicalization may extract a key proof early, but execution/admission must not
short-circuit validation of the full predicate.

Examples:

- `pk = missing_value AND unknown_field = x` must fail validation.
- It must not return not-found merely because the primary-key lookup misses.
- `pk = existing_value AND unknown_field = x` and
  `pk = missing_value AND unknown_field = x` must have the same validation error
  class.

## Parameterized SQL And Plan Cache Contract

For parameterized SQL, canonicalization may cache the shape:

- primary-key field identity;
- parameter slot or literal slot;
- strict expected key type;
- residual predicate shape;
- selected access family.

It must not cache a concrete key value unless the bound value is included in the
cache key.

At execution/admission time, the bound value must be strictly encoded through
the accepted-schema primary-key encoder. Wrong-type or missing parameters fail
through the existing validation path; they do not fall back to scanning.

If plan-cache identity, SQL-cache identity, method version, or admission
semantics change, the relevant cache or method version must be hard-cut. Old
cache entries must miss. No compatibility cache read path is allowed.

Parameterized and literal SQL must converge onto the same canonical access proof
after values are bound and encoded.

## Hard-Cut And Cache Stance

0.197 introduces no persisted-format change and no cursor-format change.

If canonicalization changes plan-cache identity, SQL-cache identity, admission
method versions, EXPLAIN schema, or selected-access serialization, the relevant
method/cache version must be bumped.

Old cached generic-scan or rejected plans must not be reused for
canonicalizable primary-key predicates.

No compatibility flag should preserve the old behavior where exact primary-key
filters are rejected as missing `LIMIT`.

If a query was previously rejected but is now admitted because of exact key
proof, that is an intentional hard-cut behavior change and must be documented in
the changelog/release notes for the implementation slice.

## Optimizer Options

### Option A: API-Only Discipline

Keep telling users to use `by_id(...)` for primary-key access.

Pros:

- minimal engine work;
- no admission changes;
- explicit access proof in code.

Cons:

- diverges from database expectations;
- makes natural SQL/fluent filters less useful;
- pushes planner knowledge into application code;
- does not help generated or SQL surfaces where `WHERE pk = ?` is the natural
  spelling.

This remains a useful workaround, but it should not be the long-term design.

### Option B: Rule-Based Canonicalization

Canonicalize strict scalar primary-key equality and finite primary-key `IN` into
key access before admission.

Pros:

- matches database expectations for primary-key lookup;
- deterministic and explainable;
- reuses existing `ByKey` / `ByKeys` executor behavior;
- does not require a cost model;
- fixes both fluent and SQL shapes if they share the same semantic plan layer.

Cons:

- changes some public-read rejection order;
- needs careful tests across external primary keys and generated surfaces;
- must avoid accidentally treating coercive or partial predicates as exact key
  proofs.

This is the preferred 0.197 first slice.

### Option C: Exact Cardinality-Aware Rule Ranking

Use exact index prefix cardinality and existing route facts to improve route
choice, admission summaries, and explain output.

Pros:

- aligns with 0.196 audit findings;
- improves diagnostics and route quality beyond primary-key equality;
- remains deterministic if cardinality facts are exact and generation-aware.

Cons:

- broader than the auth lookup problem;
- more interaction with 0.196 performance evidence;
- can become a cost-model substitute if not scoped carefully.

This is a good later 0.197 slice after primary-key canonicalization lands.

### Option D: Cost-Based Optimizer

Introduce statistics/selectivity estimates and choose plans by estimated cost.

Pros:

- closer to PostgreSQL/SQLite optimizer architecture;
- can handle richer tradeoffs in the long term.

Cons:

- conflicts with the current 1.0 deterministic planner contract;
- needs statistics maintenance, invalidation, and explain policy;
- introduces estimate quality problems on canister workloads;
- much larger risk than the current issue warrants.

This is explicitly out of scope for 0.197.

## Implementation Shape

The exact owner should be chosen during implementation, but the rule should live
where SQL and fluent predicates share semantic planning, not in one frontend
surface.

Preferred flow:

```text
query intent / lowered SQL
  -> semantic predicate binding against accepted schema authority
  -> primary-key canonicalization result
  -> access plan: ByKey / ByKeys / Empty / generic path
  -> read admission consumes bounded key proof
  -> executor evaluates any residual filter
  -> explain reports original filter plus canonical selected access
```

Do not implement this as a late executor trick. The executor must not silently
upgrade a scan into a key lookup. The planner/admission boundary owns the proof.

No full-scan fallback is allowed after a malformed, wrong-type, or over-budget
key proof. The engine must fail closed through the existing validation/admission
diagnostics.

## Explain And Diagnostics Contract

EXPLAIN and verbose diagnostics should expose whether canonicalization was
attempted, whether it succeeded, and why it was disabled.

Suggested stable fields:

```text
canonicalization_attempted=true
canonicalization_applied=true|false
canonicalization_rule=primary_key_equality|primary_key_in|none
canonicalization_disabled_reason=<enum>
canonicalization_failure_class=<enum|null>
primary_key_field=pid
primary_key_source=accepted_schema
selected_access=ByKey|ByKeys|...
access_reason=primary_key_equality_canonicalized
admission_basis=exact_primary_key|finite_primary_keys|...
raw_key_count=<n>
deduped_key_count=<n>
key_value_redacted=true
residual_filter_present=true|false
```

Possible disabled reasons:

- `not_primary_key_field`;
- `non_scalar_primary_key`;
- `partial_composite_key`;
- `non_strict_equality`;
- `coercive_equality`;
- `casefold_or_collation_equality`;
- `commuted_equality_not_normalized`;
- `expression_wrapped_key`;
- `unsupported_or_shape`;
- `no_primary_key_constraint`.

Possible failure classes:

- `wrong_value_type`;
- `unbound_parameter`;
- `malformed_in_list`;
- `raw_in_list_over_budget`;
- `deduped_in_list_over_budget`;
- `encoded_key_payload_over_budget`;
- `invalid_residual`;
- `unknown_residual_field`.

EXPLAIN should expose counts, selected access, rule names, and stable reason
enums. It should not include full raw key values by default. For privacy and
bounded-output safety, show `primary_key_field`, `raw_key_count`,
`deduped_key_count`, and `key_value_redacted=true` or a short deterministic
debug hash if needed. Do not allocate large debug strings on the success path.

Names are tentative; final names should match existing route reason vocabulary.

- wrong-type and missing SQL parameters fail validation and do not scan;

## Implementation Sequencing

Do not implement all optimizer roadmap items together. Keep 0.197.0 to this
order:

| Step | Patch | Closeout |
| --- | --- | --- |
| 1 | Add canonicalization result model and EXPLAIN fields | Diagnostic-only unless admission and execution already consume the selected artifact; disabled reasons visible |
| 2 | Fluent scalar `pk = value` -> `ByKey` | `filter(pk=value)` equals `by_id(value)` for external and generated PK |
| 3 | SQL scalar literal/parameter `WHERE pk = ...` -> `ByKey` | SQL/fluent parity and parameter cache tests pass |
| 4 | `pk IN (...)` -> `ByKeys` or `Empty` | deterministic dedup, budget rejection, no scan fallback |
| 5 | Residual predicates | `ByKey`/`ByKeys` plus residual validation/evaluation |
| 6 | Full matrix/behavior delta | expected newly admitted queries classified; no broad regressions |

Option C cardinality-aware rule ranking is not part of 0.197.0. Keep that as a
later slice after exact key canonicalization is stable.

## Boundary With 0.198 Read-Intent Ergonomics

0.197 solves exact-key read ergonomics only. It should remove fake `.limit(1)` /
`.limit(N)` ceremony for proven exact primary-key reads by making public read
admission consume the planner-owned `ByKey` / `ByKeys` / `Empty` proof. It
should not introduce general read-intent terminals or policies. 0.198 should
own the broader developer-facing answer to arbitrary query caps.

In Scope For 0.197:

- `by_id(id).try_one()` remains the explicit exact-key API.
- `filter(pk = value).try_one()` canonicalizes to `ByKey` when accepted schema
  proves strict scalar primary-key equality.
- finite `pk IN (...)` canonicalizes to `ByKeys` when accepted schema proves a
  bounded exact primary-key set.
- contradictory exact-key predicates canonicalize to `Empty`.
- exact-key public reads do not require fake `.limit(1)` / `.limit(N)`
  ceremony.
- diagnostics may explain why an exact-key-looking predicate did or did not
  canonicalize.

Out Of Scope For 0.197:

- general public page terminal, such as `page(PageRequest)`;
- complete-small-set terminal, such as `collect_all_or_too_many(...)`;
- exact aggregate terminals, such as `count_exact()` / `sum_exact(...)`;
- admin batch iteration API, such as `for_each_batch(...)`;
- named read-policy API, such as `PublicPage`, `CompleteSmallSet`,
  `ExactAggregate`, or `AdminBatch`;
- linting or banning arbitrary `.limit(1000)` / `.limit(10_000)` call sites;
- generated endpoint read-intent redesign;
- broad read-admission diagnostic tree for page/aggregate/admin alternatives.

0.197 changes planner-owned exact-key proof, public read admission, selected
access paths, EXPLAIN, cache identity, and execution route selection. It must
stay narrow so SQL, fluent, admission, cache, and executor all consume the same
`ByKey` / `ByKeys` / `Empty` artifact. General read-intent APIs change public
terminal ergonomics and endpoint design, so they need a separate 0.198
closeout.

### 0.198 Follow-Up Shape

0.198 should address the broader developer ergonomics problem:

```text
Developers should choose semantic read intent, not raw row caps.

Candidate 0.198 scope:

public page terminal with clamped page size and deterministic order;
complete-small-set terminal that returns complete results or typed TooManyRows;
exact aggregate terminals or explicit unsupported/too-expensive diagnostics;
admin batch terminal with explicit trusted/admin gating;
named read policies;
read-admission diagnostics that suggest intent-specific APIs;
migration guide away from raw high .limit(...) calls.

Candidate 0.198 non-goals:

no cost-based optimizer;
no weakening public read admission;
no silent clamping of complete reads;
no count/sum implemented by loading capped rows;
no persisted-format change;
no replacement for 0.197 exact-key canonicalization.
```

## Implementation Prompt Addendum

Important: 0.197 is a deterministic primary-key canonicalization slice, not a
general optimizer rewrite.

Do not implement read-intent ergonomics in this slice. Do not add page,
complete-small-set, aggregate, admin-batch, named policy, or raw-limit linting
APIs as part of 0.197. This line only implements exact primary-key
canonicalization and the diagnostics directly required for that proof.

Implement planner-owned canonicalization for strict scalar primary-key equality
and finite primary-key `IN` only.

Do not implement this as an executor shortcut. Admission, EXPLAIN, and execution
must consume the same selected access artifact.

Use accepted schema authority and the same key encoder as explicit
`by_id(...)`/`by_ids(...)`. Do not infer primary-key identity from generated
model fields, aliases, or secondary index metadata alone.

Handle or explicitly test:

- `pk = value`;
- `value = pk` if commuted equality is already normalized or easy to support;
- `pk = value AND residual`;
- `residual AND pk = value`;
- `pk IN finite list`;
- duplicate `IN` values;
- empty `IN`;
- contradictory primary-key predicates;
- wrong-type key value;
- unknown residual field;
- partial composite primary key;
- coercive/casefold/expression-wrapped equality.

For SQL parameters, cache the canonicalizable shape and parameter slot, not the
bound key value, unless the value is part of the cache key.

For `ByKeys`, enforce both raw input count/resource limits and deduplicated key
count/row-budget limits. Enforce encoded key-list payload budgets. Deduplicate
deterministically and return un-ordered `ByKeys` results in encoded primary-key
order.

No full-scan fallback is allowed after a malformed, wrong-type, or over-budget
key proof. Fail closed through the existing validation/admission diagnostics.

Add before/after focused performance and behavior evidence comparing filter
spellings against explicit `by_id(...)`/`by_ids(...)`. Performance closeout
requires fresh before/after focused artifacts and a fresh full deterministic SQL
matrix pair from the same machine/build/matrix configuration. Classify all
changed statuses/access routes.

Historical matrix artifacts are context only.

## Final Safety Clarifications

### Valid Non-Application Versus Failure

`None { reason }` is only for valid query shapes where the primary-key
canonicalization rule does not apply. Invalid key terms, wrong types, missing
parameters, invalid residual predicates, malformed `IN` lists, and over-budget
inputs must fail through validation/admission. They must not continue through
the generic planner as scan candidates.

### Full Predicate Validation

Canonicalization must not hide invalid residual predicates. The complete
predicate must be semantically validated before execution can return a result. A
missing primary-key row must not mask an unknown residual field or wrong-type
residual value.

### Empty Terminal Semantics

`Empty` is a proven zero-row result. It is not a validation error and not a scan.
Optional terminals return no row, required one-row terminals use the same
not-found/cardinality helper as an empty key lookup, collection terminals return
an empty result, and count returns zero. `Empty` must perform zero row-store and
index-range reads.

### ByKeys Deterministic Order

For 0.197.0, `ByKeys` without explicit `ORDER BY` returns rows in deterministic
encoded primary-key order after deduplication. Do not expose hash iteration or
raw input-list order.

### Closeout Evidence

Historical matrix artifacts are context only. Performance closeout requires a
fresh before/after focused primary-key canonicalization matrix. A full
deterministic SQL matrix must also be produced for release closeout. If it
cannot be run, the patch may close as correctness-only but must not claim broad
performance closeout.

## Risks

### Rejection-Order Churn

Some queries that currently fail with `E188`, `E5`, or `E190` may become
admitted exact-key reads. That is intended when the proof is exact, but tests
must pin the new behavior so future admission changes are deliberate.

### Over-Wide Equality Recognition

The dangerous bug is claiming `ByKey` for a predicate whose value is not exactly
the primary-key value under accepted schema semantics. The first slice must
require strict type compatibility and avoid textual parsing or lossy coercion.

### Composite-Key Leakage

Composite keys need all ordered components to prove exact lookup. A predicate on
one component is not a full primary-key lookup. Keep composite support deferred
unless implementation can prove full-key equality cleanly.

### Planner/Executor Drift

If executor behavior notices primary-key equality independently, explain,
admission, and execution can diverge. Keep the canonicalization planner-owned
and make executor consume the selected access artifact.

### Cache Staleness

The same query spelling may now select a different access path. If cached plan or
SQL artifacts do not include the new canonicalization shape/version, old
generic-scan or rejected plans could be reused incorrectly. Treat cache identity
changes as hard cuts.

### Resource Amplification Through `IN`

Deduplicated cardinality alone is not enough. Raw `IN` inputs can be large even
when all values are duplicates, and variable-width key values can be large even
when the raw term count is small. Enforce raw input budgets, encoded payload
budgets, and deterministic deduplication before using the deduplicated row bound
for admission.

## Success Criteria

0.197.0 is complete when:

- strict scalar primary-key equality and finite primary-key `IN` share the same
  plan family as explicit `by_id(...)` / `by_ids(...)`;
- exact key values are encoded through accepted schema authority and the same
  key encoder as explicit key APIs;
- public read admission accepts exact key lookup filters without fake `LIMIT`
  or `ORDER BY` ceremony;
- general non-primary-key reads remain outside this slice and still require
  bounded page, aggregate, or admin semantics;
- public read admission rejects malformed, wrong-type, and over-budget key
  proofs without scanning;
- explain clearly shows the key route, canonicalization reason, disabled
  reason, raw key count, deduplicated key count, and residual presence;
- SQL parameterized shapes cache parameter/key-extractor shape, not stale key
  values;
- focused before/after behavior and performance evidence exists;
- full deterministic matrix deltas are produced for performance/release closeout
  and all changed access/status outcomes are classified;
- unsupported adjacent shapes still fail closed;
- no cost-based planner, persisted-format change, cursor-format change, or
  generated-model runtime fallback is introduced.

## Follow-Up Optimizer Roadmap

After the first slice, useful deterministic optimizer improvements include:

1. Exact cardinality surfacing in admission summaries and explain.
2. Deterministic route ranking improvements using exact generation-compatible
   cardinality facts, not estimates.
3. More semantic canonicalization for already-admitted boolean forms.
4. Better public diagnostics that explain what index or order proof is missing.
5. Optional full composite-key equality canonicalization.

These should remain separate slices. The first 0.197 deliverable should be the
small primary-key lookup equivalence that users already expect from a database.
