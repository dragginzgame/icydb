# 0.128 Persisted-Row Slot Contract Narrowing

## Thesis

`0.128` should narrow the persisted-row slot contract without reopening the
larger `0.127` ownership split.

This is not another `Value` deletion slice.

It is not a persisted format rewrite.

It is not a public API redesign.

It is not a planner/runtime semantic slice.

It is a persisted-row slot-contract slice whose job is to reduce the breadth
of the current `Value`-based row boundary by splitting one narrow ownership
lane at a time.

The intended end state for `0.128` is:

* the broader slot contract is still explicit and local to `persisted_row`
* canonical row re-emission has a narrower owner than the full slot contract
* helper duplication is reduced without changing persisted bytes
* `reader.rs`, `patch.rs`, and `contract.rs` stop feeling like one merged
  responsibility even when they still cooperate closely

---

## Why `0.128` Exists

`0.127` finished the important persistence-ownership cleanup:

* `ValueCodec` is gone
* runtime conversion now lives on `RuntimeValueEncode` /
  `RuntimeValueDecode`
* typed persisted structured fields now use `PersistedStructuredFieldCodec`
* typed persisted `ByKind` fields now use `PersistedByKindCodec`
* typed key encode/decode now uses `StorageKeyCodec` / `StorageKeyDecode`
* the normal `icydb::db` facade no longer exposes the persisted-row helper
  toolbox
* obvious persisted-row codec and reader helper duplication was removed

That leaves one real remaining seam:

* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/contract.rs`
* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/reader.rs`
* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/patch.rs`

Those files still share a broad slot-level contract shaped around:

* `Value <-> bytes`
* slot materialization
* patch/update replay
* canonical row re-emission
* per-slot structural validation

That seam is not leftover `0.127` residue in the same sense as the deleted
`ValueCodec` bridge or the deleted by-kind router.

It still has a real job.

The problem is that it is too broad.

So `0.128` exists to narrow that slot-contract boundary without pretending the
whole thing should disappear.

---

## Problem

Today the persisted-row slot contract still mixes several jobs:

* slot-level decode through field contracts
* slot-level encode from runtime `Value`
* dense canonical slot-image rebuilding
* patch replay and serialized patch staging
* row-reader materialization helpers

Those jobs are related, but they are not the same ownership problem.

The main risk in the current shape is not incorrect behavior.

The main risk is that future cleanup keeps bouncing across all three owners at
once:

* `contract.rs`
* `reader.rs`
* `patch.rs`

That makes every local cleanup feel larger than it is.

The first `0.128` slice should therefore reduce breadth before it tries to
reduce `Value`.

---

## Goal

Start narrowing the persisted-row slot contract by splitting off canonical row
re-emission from the broader slot-materialization and patch/update lanes.

`0.128` should deliver:

* one smaller owner for dense canonical slot-image rebuild and raw-row
  re-emission
* fewer helper loops duplicated between payload-source and value-source row
  rebuild paths
* clearer separation between:
  * slot decode/materialization
  * canonical row emission
  * patch/update replay

This should be done without:

* changing persisted bytes
* changing error taxonomy
* changing the reader cache model
* changing patch/update semantics

---

## Core Invariant

The broader persisted-row slot contract may stay `Value`-based where it still
owns true row-level structural work.

Corollaries:

* `Value` may remain the slot-contract currency for now
* canonicalization/re-emission can still be narrowed even if patch/update is
  left alone
* `reader.rs` does not need to be redesigned in the opening `0.128` patch
* `patch.rs` does not need to change in the opening `0.128` patch

---

## Scope

### In Scope

* narrowing canonical row emission and slot-image rebuild ownership
* removing helper duplication inside the current slot-contract lane
* introducing one clearer owner boundary for canonical slot-image work
* local helper collapse in `persisted_row/contract.rs` that does not change
  bytes or behavior

### Out Of Scope

* deleting `Value` from the persisted-row slot contract wholesale
* redesigning `UpdatePatch`, `FieldUpdate`, or `SerializedUpdatePatch`
* redesigning `SlotReader::get_value` or `required_value_by_contract`
* changing storage-key binary layout
* changing structured field binary layout
* moving planner/runtime generic `Value` logic
* another broad trait split

## Opening Slice

The opening `0.128` patch should target:

* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/contract.rs`

and specifically these responsibilities:

* dense canonical slot-image rebuilding
* row re-emission from canonical slot payloads
* helper compression around payload-source versus value-source canonical loops

The opening patch should **not** try to change:

* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/reader.rs`
  beyond call-site retargets if needed
* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/patch.rs`
* the slot-level `decode_slot_value_from_bytes(...) -> Value` boundary
* the slot-level `encode_slot_value_from_value(...) -> Vec<u8>` boundary

That means the first patch is a contraction patch, not a semantic one.

## Proposed Shape

### Current Broad Contract

Today one file still acts as the owner for:

* field-contract decode from bytes
* field-contract encode from `Value`
* canonical slot-image normalization
* raw row re-emission

That is workable, but broader than necessary.

### Narrower Shape

The narrower shape for `0.128` is:

* `contract.rs` keeps slot-level encode/decode and field-contract validation
* canonical slot-image rebuild and raw-row re-emission move behind one tighter
  internal owner boundary
* the public or crate-visible surface does not grow

This can be done either by:

* introducing a small internal helper module under
  `persisted_row/contract.rs`, or
* keeping the same file while making the canonical row-emission lane visibly
  separate and locally shared

The key point is ownership clarity, not file count.

---

## First Patch Plan

1. Identify the canonical row-emission cluster inside `contract.rs`.
   That currently includes:
   * `dense_canonical_slot_image_from_payload_source(...)`
   * `dense_canonical_slot_image_from_value_source(...)`
   * `encode_slot_payload_from_parts(...)`
   * `emit_raw_row_from_slot_payloads(...)`

2. Collapse the remaining same-owner helper duplication there.
   That includes:
   * shared loop structure
   * shared payload staging
   * any forwarding-only encode helpers that do not add policy

3. Leave the slot-level `Value` encode/decode functions in place.
   Specifically:
   * `decode_slot_value_from_bytes(...)`
   * `decode_slot_value_for_field(...)`
   * `encode_slot_value_from_value(...)`

4. Do not touch patch replay in the opening patch.
   `patch.rs` remains a later `0.128.x` step.

5. Do not touch reader cache/materialization in the opening patch.
   `reader.rs` remains a later `0.128.x` step unless a tiny retarget is needed.

---

## Expected Outcome

After the opening `0.128` patch:

* the persisted-row slot contract is still intentionally `Value`-shaped where
  that remains the real row-level owner
* canonical row re-emission becomes easier to reason about as its own lane
* future `0.128.x` work can choose one next seam cleanly:
  * patch/update currency
  * reader materialization

That is the right next move after `0.127`.

`0.127` removed obsolete ownership.

`0.128` should narrow the remaining real owner instead of pretending it is all
the same cleanup.
