# 0.125 Value Boundary Separation

## Thesis

`0.125` should separate internal runtime value semantics from public API value
contracts by introducing explicit input and output boundary types while
retaining `db::Value` as the internal runtime representation.

This is not an expression traversal slice.

It is not a query semantics redesign.

It is not a storage or binary format slice.

It is a value-ownership and API-boundary slice whose job is to:

* keep `db::Value` as the internal execution value
* define explicit boundary types for input and output
* prevent internal execution semantics from leaking into public APIs
* remove serialization constraints from the runtime value layer

The target end state is:

* `db::Value` remains the internal runtime value type
* `OutputValue` defines all public result surfaces
* `InputValue` defines all external input/query value surfaces
* no public API exposes `db::Value`
* all boundary crossings are explicit and centralized

---

## Why `0.125` Exists

The current system exposes `db::Value` across multiple public boundaries:

* SQL result rows (`Vec<Vec<Value>>`)
* session/load helper APIs (`values_by`, `project_values`, and similar)
* public re-exports via prelude
* response and projection surfaces

That creates three problems.

### 1. Internal Representation Is Frozen

Because `Value` is public:

* changes to numeric handling, null semantics, or collections become breaking
  changes
* internal evolution is constrained by API compatibility

### 2. Execution Semantics Leak Into API

`Value` currently represents:

* execution-layer semantics
* potentially richer internal variants than needed externally
* evaluation-oriented structure

Those are not appropriate as a public contract.

### 3. Serialization Concerns Leak Into Core Types

Because `Value` is exposed:

* it must derive `CandidType`
* internal design is shaped by serialization rather than execution

---

## Goal

Introduce a strict three-layer value model:

`InputValue -> Value -> OutputValue`

Where:

* `Value` is internal-only
* `InputValue` is external input
* `OutputValue` is external output

---

## Core Invariant

`db::Value` must not cross public API boundaries.

`OutputValue` is the only type returned externally.

`InputValue` is the only value type accepted externally.

All conversions are explicit and centralized.

Corollary:

No public API may expose or depend on `db::Value`.

---

## Scope

### In Scope

* retain `db::Value` as the runtime value
* introduce `OutputValue`
* formalize `InputValue`
* replace all public `Value` usage with `OutputValue`
* remove `CandidType` from `Value`
* remove public re-exports of `Value`
* audit and remove or reshape value-first APIs
* introduce a centralized conversion boundary

### Out Of Scope

* expression traversal
* query planning or execution redesign
* storage or binary format changes
* type-system redesign
* unifying input and output value types

---

## Ownership Model

### Runtime Layer: `db::Value`

`Value` is:

* the execution/runtime semantic value
* used by evaluator, planner, and executor

It is:

* not public
* not serialized
* free to evolve

### Input Boundary: `InputValue`

`InputValue` represents:

* user-supplied query values
* filter literals
* external input data

It may include:

* coercion-friendly representations
* query-specific flexibility

It does not need to match `OutputValue`.

### Output Boundary: `OutputValue`

`OutputValue` represents:

* fully evaluated query results
* the public API contract
* the serialized value type

It must be:

* minimal
* deterministic
* stable
* Candid-compatible

---

## `OutputValue` Design

`OutputValue` should be a subset contract, not a mirror of `Value`.

Initial shape:

```rust
enum OutputValue {
    Null,
    Bool(bool),
    Int(i64),
    Float(f64),
    Text(String),
}
```

Optional, only if required:

* `List(Vec<OutputValue>)`
* `Map(Vec<(String, OutputValue)>)`
* `Blob(Vec<u8>)`

Constraint:

`OutputValue` must only contain value forms that are explicitly supported by
the public API.

---

## Conversion Boundary

All conversions occur at a single boundary:

`execution -> Value -> OutputValue -> API`

Requirements:

* conversion must be total
* conversion must be deterministic
* no scattered conversions across modules
* no partial or lazy conversions

Implementation can use:

* `impl From<Value> for OutputValue`
* or an equivalent centralized mapping surface

---

## Public API Changes

### SQL Results

Replace:

* `Vec<Vec<Value>>`

With:

* `Vec<Vec<OutputValue>>`

### Session / Load APIs

Audit and update:

* `values_by`
* `project_values`
* `distinct_values_by`
* `first_value_by`
* `last_value_by`

These should be:

* removed if redundant
* or rewritten to return structured types or `OutputValue`

### Public Re-exports

Remove:

* `Value` from prelude
* any public exposure of internal value modules

### Endpoint Audit

Classify APIs into three groups.

#### A. Entity APIs

These remain unchanged:

* `load::<T>() -> Vec<T>`

#### B. Query APIs

These convert to output boundary values:

* `Vec<Row<OutputValue>>`
* or equivalent public result structures that carry `OutputValue`

#### C. Value-First APIs

These are reviewed or removed.

APIs returning raw values should:

* be removed if redundant
* or be rewritten using `OutputValue`

---

## What `0.125` Should Not Do

`0.125` should not:

* rename or remove `Value`
* expose `Value` through wrappers
* partially migrate APIs
* unify `InputValue` and `OutputValue`
* redesign query results beyond value substitution

This slice must stay boundary-focused.

---

## Migration Strategy

1. Introduce `OutputValue`.
2. Introduce or confirm `InputValue`.
3. Implement centralized `Value -> OutputValue` conversion.
4. Update all public APIs to use `OutputValue`.
5. Remove public re-exports of `Value`.
6. Remove `CandidType` from `Value`.
7. Audit and remove or reshape redundant value-first APIs.

---

## Completion Criteria

`0.125` is complete when:

* `db::Value` is not publicly exposed
* all public APIs use `OutputValue`
* SQL results no longer return `Value`
* no public re-exports expose `Value`
* `Value` does not derive `CandidType`
* conversion is centralized
* value-first APIs are removed or reshaped

---

## Expected Payoff

If `0.125` succeeds, the system gains:

* full internal control over runtime value semantics
* a stable, minimal API value contract
* freedom to evolve execution and numeric models
* elimination of serialization constraints from core types
* reduced API surface complexity

This establishes a clean and durable separation between:

* query input
* execution semantics
* query output

while preserving the ergonomics of `db::Value` internally.
