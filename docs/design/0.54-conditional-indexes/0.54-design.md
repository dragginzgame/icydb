# 0.54 — Filtered / Conditional Indexes

Status: **Implemented in 0.54.0**

Implementation tracking lives in:

`docs/design/0.54-conditional-indexes/0.54-status.md`

This milestone introduces **filtered (conditional) secondary indexes**.

Filtered indexes store entries **only for rows satisfying a predicate**, allowing smaller indexes and faster scans for sparse or state-partitioned datasets.

This feature extends the existing secondary-index model without changing:

* index storage format
* cursor semantics
* recovery model
* commit marker system

Filtered indexes are implemented as **schema-level predicates evaluated during mutation and validated during planning**.

---

# Goals

Filtered indexes allow indexes to apply only to a subset of rows.

Example:

```sql
CREATE INDEX idx_active_users
ON users(email)
WHERE active = true;
```

Only rows satisfying:

```
active = true
```

are stored in the index.

Benefits:

* smaller index structures
* reduced mutation overhead
* faster scans
* enables partial uniqueness constraints

Example use cases:

```
soft-delete tables
status partitions
sparse columns
stateful entity indexes
```

---

# Non-Goals

This milestone does **not** introduce:

* expression indexes
* multi-predicate implication solvers
* cost-based predicate reasoning
* partial index rewriting

Predicate implication is intentionally limited to **deterministic structural checks**.

---

# Index Model Extension

Current index definition:

```
IndexDefinition {
    fields
    unique
}
```

New structure:

```
IndexDefinition {
    fields
    unique
    predicate: Option<Predicate>
}
```

Predicate is stored in **schema metadata only**.

Index entries remain unchanged:

```
index_key + row_reference
```

No predicate value is stored in the index entry itself.

---

# Predicate Representation

The predicate uses the existing **predicate AST and canonicalization system**.

Constraints:

* must be deterministic
* must reference only columns of the indexed entity
* must not contain aggregates
* must not contain subqueries

Examples allowed:

```
active = true
status = 'ACTIVE'
age > 18
deleted = false
```

Examples rejected:

```
NOW() > created_at
age > other_table.age
SUM(balance) > 0
```

Validation occurs at schema creation.

---

# Planner Integration

Filtered indexes are eligible only when the query predicate **implies the index predicate**.

Example:

Index predicate:

```
active = true
```

Query predicate:

```
active = true AND age > 20
```

Index is usable.

Query:

```
age > 20
```

Index is not usable.

Planner rule:

```
query_predicate ⇒ index_predicate
```

Initial implication rules:

| Index Predicate | Query Predicate        | Allowed |
| --------------- | ---------------------- | ------- |
| active=true     | active=true            | ✓       |
| active=true     | active=true AND age>10 | ✓       |
| age>10          | age>20                 | ✓       |
| age>10          | age>5                  | ✗       |

Implication checks rely on **predicate canonicalization** and simple operator comparisons.

Complex boolean reasoning is intentionally out of scope.

---

# Planner Changes

During index candidate evaluation:

```
if index.predicate exists:
    if not implies(query_predicate, index.predicate):
        skip index
```

Filtered index eligibility is evaluated **before route shape selection**.

This preserves existing access planning semantics.

---

# Mutation Pipeline Changes

Current mutation path:

```
row mutation
→ compute index keys
→ insert / remove index entry
```

Filtered mutation logic:

```
row mutation
→ evaluate predicate(row)

if predicate true:
    ensure index entry exists

if predicate false:
    ensure index entry absent
```

Handling updates:

```
old row predicate true
new row predicate false
→ remove index entry
```

```
old false
new true
→ insert index entry
```

This evaluation occurs inside the **existing index mutation pipeline**.
