# 0.190 IC Reliability Proof Expansion

Status: active.

Theme: expand the 0.189 IC-native reliability proof from the minimal
marker/journal boundary to recovery, persisted-format, resource-budget, and
large-index failure modes.

0.189 proves the minimum durability boundary:

- supported cfg matrix is explicit and green;
- marker/journal/marker-clear interruption has an oracle-driven host harness;
- oversized journal values cannot bypass the configured allocation bound;
- marker/journal corrupt bytes fail safely;
- obvious API data-loss footguns are documented.

0.190 expands that proof. It is an executable reliability milestone, not a
backup/restore/checksum productization milestone.

## Entry Criteria

Start 0.190 only after 0.189 has:

- a working host failpoint harness;
- recovery oracle vocabulary for pre-marker and marker-authorized post-commit
  states;
- marker/journal deterministic corruption tests;
- a resolved raw journal batch oversized-allocation proof or fix;
- a persisted-format change classification rule.

0.190 must build on those tools, not create a parallel reliability framework.

## Progress

- `0.190.0` starts Work Track A by extending the 0.189 host failpoint harness
  into journal-tail folding. The first landed sites cover interruption before
  a journal batch folds into canonical rows and interruption after the fold
  watermark is durably persisted but before tail cleanup.
- The journal-tail fold test proves both structured returned errors and host
  unwind interruptions, guarded retry convergence, repeated pre-fold
  interruption, marker visibility while recovery is incomplete, journal-tail
  visibility, fold-watermark state, and row/index oracle state.
- A failpoint after canonical row fold but before fold-watermark persistence is
  intentionally not admitted in the first slice. An exploratory test showed
  that retrying from that exact gap is not currently idempotent for model
  batches, so it must be treated as a separate semantic-hardening target before
  the harness can claim that window.
- The current Work Track A slice also adds a secondary-index rebuild failpoint
  after all derived indexes have been marked building and cleared. The test
  records the intended mode split: structured returned errors restore the
  pre-rebuild index snapshot through the existing rollback path, while host
  unwind leaves the cleared/building derived state for guarded retry. Both modes
  must converge back to the marker-authorized row/index oracle through reentry.
- The journaled index materialized-view fold site now has matching coverage
  after rebuilt index entries have folded into canonical index storage but
  before marker clear and index readiness. The oracle requires marker authority
  to remain visible, rows/indexes to match the marker-authorized post-state,
  and index readiness to stay `Building` until guarded retry completes.
- The existing marker-clear failpoint now explicitly proves the
  marker-cleared/readiness-not-restored reentry case: interruption after clear
  leaves no marker but still leaves indexes `Building`, and guarded retry must
  run recovery to completion before marking indexes `Ready`. This retry is
  keyed by an interrupted recovery-domain hint, not by treating every
  intentionally non-ready index as recoverable derived state.
- A durable row-op replay failpoint is not part of the current 0.190.0 claim.
  Durable journaled stores recover from marker-bound journal batches; obsolete
  durable marker row-ops fail closed, while heap row-ops are live-only and have
  no IC durability contract. If a durable row-op recovery path is reintroduced,
  it must get its own failpoint and oracle before it is considered supported.
- `0.190.1` starts Work Track B by proving repeated guarded recovery
  interruption across multiple recovery phases. The first .1 test interrupts
  after fold-watermark persistence, re-enters through the guarded recovery path,
  interrupts again after journaled index materialization fold, then verifies
  final rows, indexes, marker state, journal-tail state, watermark, and
  readiness converge to the marker-authorized oracle.
- The .1 repeated-interruption coverage also includes the
  marker-cleared/readiness-not-restored path: recovery is interrupted after
  marker clear, re-enters with marker authority absent but recovery-domain
  completion still pending, interrupts again after journaled index fold, and
  then converges to `Ready` without resurrecting marker or journal-tail state.
- The next Work Track C slice adds an upgrade/reentry-shaped host proof:
  after an `AfterMarkerClear` unwind leaves stable marker bytes absent and
  indexes still `Building`, the test drops only volatile recovery-domain and
  schema-readiness bookkeeping for that database domain and proves guarded
  startup recovery restores `Ready` from stable row/index/journal state.
- Work Track D now starts with a deterministic malformed-byte corpus at the
  DB boundary. The first corpus covers row envelopes, persisted schema snapshot
  Candid bytes, raw index-key/index-entry bytes, structural value-storage
  envelopes, and by-kind structural fields, asserting that each malformed case
  returns a decode/validation error instead of panicking or silently producing
  a value.
- Work Track F now has a first checked-in evidence note:
  [0.190-evidence.md](0.190-evidence.md). The executable characterization
  rebuilds a 256-row secondary index from authoritative rows, drops stale
  index-only state, and documents that this is a host regression floor rather
  than a production IC recovery-size bound.
- The evidence note also records 0.191 productization inputs for checksum,
  backup/restore/import, production recovery-size, and operator durability
  guide decisions.
- The current closeout slice adds source-audit guards for the IC update-message
  model. Commit, recovery, DELETE, mutation, and write-execution runtime paths
  are checked for async/await, thread, spawn, and canister-call tokens outside
  tests so the 0.190 reliability proof does not quietly gain a reentry or
  scheduling surface. The same guard freezes commit failpoint hooks as test-only
  machinery and checks that `_for_tests` helpers do not survive in production
  write/recovery runtime code.

## Scope

0.190 owns:

- expanded recovery failpoints beyond marker write, journal append, and marker
  clear;
- repeated interruption during recovery itself;
- one host or PocketIC trap-like or instruction-budget scenario after durable
  authority exists;
- persisted-format fuzzing or deterministic corpus tests beyond marker/journal;
- structural decode and unsafe-code verification where feasible;
- stable-memory resource and instruction-budget characterization;
- large-index recovery and fold scale limits.

0.190 may produce notes that inform backup/restore/import, checksum, and
operator-doc decisions, but those decisions are not hard 0.190 closeout items.

## Minimum Viable 0.190 Closeout

0.190 is complete enough to ship when it produces executable evidence for the
broader IC-native reliability risks left after 0.189:

1. Expanded failpoint coverage reaches at least one fold path, one
   derived-index rebuild/fold path, and one repeated recovery-interruption path.
2. Every expanded failpoint test declares durable authority and validates
   against an explicit row/index oracle.
3. At least one host or PocketIC trap-like or instruction-budget scenario runs
   after durable authority exists.
4. Persisted-format corpus or fuzz tests cover raw row, schema, index, and
   structural decode beyond 0.189 marker/journal coverage.
5. Structural decode has either Miri coverage or a documented malformed-input
   fallback suite.
6. Large-index recovery produces measured evidence, a documented supported
   bound, a failing threshold test, or a tracked streaming design.
7. Evidence notes exist for 0.191 productization decisions.

## Fail-Closed Definition

For 0.190, "fails closed" means guarded reads and writes refuse to proceed while
recovery is incomplete or unsafe.

A fail-closed path must not:

- expose user rows from a partially recovered store;
- enter write planning or commit preflight;
- clear recovery authority;
- mark derived readiness;
- silently continue after corruption, incompatible format, or resource
  exhaustion.

A later guarded access must be able to retry recovery unless the state has been
classified as unrecoverable corruption or incompatible format.

## Non-Goals

- Do not implement SQLite-style transactions.
- Do not introduce multi-message transactions.
- Do not make pagination snapshot-isolated.
- Do not add POSIX file-system durability machinery.
- Do not change stable-memory formats without classifying the change under the
  0.189 persisted-format rule.
- Do not use broad fuzzing as a substitute for oracle-driven failpoint tests.
- Do not make checksum implementation or backup/restore support a 0.190
  blocker.

## Work Track A: Expanded Recovery Failpoints

0.190 extends the 0.189 harness to recovery phases that were too broad for the
minimum closeout.

Candidate failpoint sites:

- durable row-op replay, only if a durable row-op recovery path exists;
- journal tail fold before watermark persist;
- after fold watermark persist before tail cleanup;
- live projection rebuild;
- secondary-index rebuild;
- journaled index materialized-view fold;
- marker clear followed by repeated interruption before volatile readiness is
  restored.

Required oracle:

- every test must reuse the 0.189 durable phase vocabulary;
- every test must declare whether durable authority is the marker, folded
  canonical state, journal watermark, or derived-data rebuild;
- every test must compare against explicit expected row and index state;
- no test may accept an intermediate state merely because it is internally
  consistent.

Durable authority vocabulary:

| Durable authority | Meaning | Expected oracle |
| --- | --- | --- |
| Marker | Commit marker is still authoritative for the whole mutation. | Recovered state equals full marker-authorized post-commit state. |
| Journal tail | Marker-bound or persisted journal records describe unapplied work. | Replay or fold exactly once; no duplicate row/index effects. |
| Fold watermark | Canonical state has incorporated records up to a persisted watermark. | Records at or below watermark must not replay again. |
| Canonical row store | Row state is authoritative but derived readiness may be missing. | Rebuild derived indexes/readiness before normal reads. |
| Derived rebuild | Row state is valid; derived structures are being reconstructed. | Reads/writes fail closed until derived structures are ready. |

Minimum required expanded failpoint coverage:

- one journal tail fold interruption before watermark persist;
- one interruption after watermark persist before tail cleanup;
- one secondary-index or live-projection rebuild interruption;
- one journaled index materialized-view fold interruption;
- one marker-cleared/readiness-not-restored guarded reentry case.

Durable row-op replay coverage is required only if a durable row-op replay path
exists in the active persisted format. The current journaled durable format uses
journal batches for recovery authority; marker row-ops for durable journaled
stores fail closed and heap row-ops are live-only.

Additional failpoint sites may be added as stretch coverage.

## Work Track B: Repeated Recovery Interruption

Recovery itself must be safe to interrupt and re-enter.

Test shape:

1. Create a pre-state and expected post-state.
2. Reach a named durable phase.
3. Interrupt recovery at a named recovery failpoint.
4. Construct a fresh guarded access path.
5. Re-enter recovery.
6. Repeat at least once for selected phases.
7. Assert final rows, indexes, marker state, journal visibility, and readiness
   state match the declared oracle.

Acceptance:

- recovery is idempotent across repeated interruption;
- journal tail records below a persisted fold watermark do not replay twice;
- marker-authorized effects are applied exactly once;
- reads/writes do not proceed while recovery is incomplete.

## Work Track C: Trap And Instruction-Budget Coverage

0.189 covers returned errors and panic/unwind in host tests. 0.190 adds at least
one IC-style interruption scenario that is harder to simulate.

Targets:

- abort-like trap where host-feasible;
- instruction-budget exhaustion where host or PocketIC can reproduce it;
- canister upgrade/reentry scenario where stable memory persists and heap
  readiness does not.

Minimum acceptance:

- at least one trap-like or instruction-budget test occurs after durable
  authority exists;
- guarded reentry either completes recovery or fails closed;
- normal reads/writes cannot proceed on partial recovery.
- write and recovery runtime paths remain synchronous, and commit failpoint or
  `_for_tests` helpers remain test-only; any future async, thread, spawn,
  canister-call, production failpoint, or production test-helper introduction
  must be made explicit and re-audited against the IC update-message model.

A returned `Result::Err` from a test failpoint does not satisfy this track. The
closeout requires a mode that approximates abrupt interruption, such as
panic/unwind, abort-like trap, PocketIC trap, instruction-budget exhaustion, or
upgrade/reentry with heap readiness lost.

Full budget modeling may remain follow-up work if the first test proves the
harness and contract.

## Work Track D: Persisted-Format Corpus Or Fuzzing

0.189 covers deterministic marker/journal corruption. 0.190 broadens coverage to
the rest of IcyDB-owned persisted envelopes.

Targets:

- raw row decode;
- accepted schema snapshot decode;
- index key decode;
- index envelope decode;
- structural field decode callbacks.

Required invariant:

- malformed bytes produce structured corruption or incompatible-format errors;
- malformed bytes do not panic;
- malformed bytes do not allocate unbounded memory;
- malformed bytes do not loop indefinitely;
- malformed bytes do not silently produce a valid but wrong logical state.

For decoders that can produce apparently valid domain objects, corpus tests
should include semantic validation against an expected model or invariant. A
test that only checks "decode returned Ok" or "decode returned Err" is not
enough for row, schema, index, or structural envelopes where wrong-but-valid
values are possible.

Implementation options:

- `cargo fuzz` targets if the tooling cost is acceptable;
- deterministic corpus runner if fuzz tooling is too heavy for the main
  workspace;
- minimized corpus committed under an audit or test fixture path.

0.190 is acceptable with a deterministic corpus runner if it is repeatable in
CI.

## Work Track E: Unsafe And Miri Coverage

Unsafe code is not automatically wrong, but structural decode and persisted
input plumbing should have mechanical coverage.

Actions:

- try Miri on the smallest package/suite that covers structural field decode;
- if Miri is unavailable or impractical on the active project toolchain,
  document that and add deterministic malformed structural-field tests instead;
- route structural decode fuzzing through the same corpus strategy as Work
  Track D;
- keep metrics-sink unsafe coverage separate from persisted-input safety unless
  a concrete aliasing bug is found.

Acceptance:

- Miri coverage exists for the smallest relevant suite, or a documented
  malformed-input fallback suite exists;
- structural decode malformed input cannot panic, loop indefinitely, or
  silently produce the wrong logical state.

## Work Track F: Resource Budget And Large Index Recovery

SQLite's disk-full concern maps to stable-memory growth, cycles, instruction
limits, and recovery work that can exceed an IC message budget.

0.190 must distinguish:

- structured resource failures that can return `Result::Err`;
- trap-like resource failures such as grow traps, OOM, and instruction limits;
- scalability budget failures where work is correct but too large for practical
  IC execution.

Large index fold/rebuild must produce one of:

- measured evidence that current fold/rebuild fits the intended IC budget;
- a documented maximum supported index size for recovery;
- a failing threshold test that marks the current scale cliff;
- a tracked streaming fold/rebuild design.

The threshold-test option is acceptable. It is better to know the scale cliff
than to imply 0.190 fixed a streaming problem it only discovered.

Large-index characterization must produce a checked-in evidence note or test
summary that records:

- dataset shape;
- number of rows;
- number and type of indexes;
- journal tail size;
- recovery or fold operation measured;
- host or PocketIC environment;
- measured time, instruction proxy, or memory behavior;
- resulting supported bound, threshold failure, or streaming follow-up.

## Evidence Notes For 0.191

0.190 should produce notes for follow-up productization, but should not block on
final policy decisions.

The notes should answer, as far as evidence allows:

- did fuzz/corpus testing reveal a need for checksums?
- did resource characterization reveal a recovery-size limit?
- are imported stable-memory images plausibly in scope?
- does backup/restore require a separate persisted-format threat model?
- is streaming index fold/rebuild required before production durability claims?

These notes feed 0.191.

## Acceptance Criteria

0.190 is complete when:

- expanded failpoint coverage includes fold, index rebuild, index fold, and
  repeated recovery interruption;
- every expanded failpoint test declares durable authority and validates against
  an explicit row/index oracle;
- at least one host or PocketIC test covers trap-like or instruction-budget
  interruption after durable authority exists;
- persisted-format fuzzing or corpus tests cover row, schema, index, and
  structural decode in addition to 0.189 marker/journal coverage;
- structural decode has Miri coverage or a documented malformed-input fallback
  suite;
- large-index recovery has measured evidence, a documented supported bound, a
  failing threshold test, or a streaming follow-up design;
- 0.191 evidence notes exist for backup/restore/import, checksums, and operator
  documentation.

## Validation Plan

Routine validation:

- format check;
- all-features workspace tests;
- supported no-default tests or no-default smoke tests;
- clippy on supported feature combinations;
- invariant checks.

Reliability validation:

- expanded failpoint matrix;
- repeated recovery interruption tests;
- persisted-format fuzz/corpus runner;
- structural decode malformed-input tests;
- large-index recovery characterization;
- PocketIC or host instruction-budget scenario where feasible.

## Relationship To 0.189

0.190 depends on 0.189's harness and oracle. If 0.189 does not produce those,
0.190 should not start by broadening scope. It should first finish the minimum
marker/journal proof.

## Relationship To 0.191

0.191 owns productization decisions informed by 0.190 evidence:

- backup/restore/import threat model;
- checksum decision and any format change;
- operator durability guide;
- streaming index fold/rebuild implementation design if required;
- documented production limits.

0.190 may create follow-up implementation designs, but those should be separate
once the evidence identifies the real constraint.
