# 0.190 IC Reliability Evidence Notes

## IC Update-Message Source Guard

### Status

0.190 adds a source-audit regression test for the synchronous update-message
assumption used by the recovery proof.

### Executable Claim

Commit, recovery, DELETE, mutation, and write-execution runtime paths are
scanned outside test-only code for async/await, host thread, spawn, and
canister-call tokens. The guard is intentionally source-shaped: it does not
prove every possible scheduling property, but it prevents the core write and
recovery surfaces from quietly gaining obvious yield, reentry, or host
concurrency APIs.

The same source-audit module also checks that commit failpoint symbols and
generic `_for_tests` helper symbols disappear after test-only code is stripped
from production runtime sources. That keeps the 0.189/0.190 interruption
harness and supporting fixtures from becoming production control surfaces.

### Non-Claim

This is not a PocketIC instruction-budget measurement and does not replace the
oracle-driven interruption tests. It only backs the IC-native assumption that
normal mutation/recovery execution is synchronous unless a future change
explicitly widens that model.

## Large-Index Recovery Characterization

### Status

0.190 adds a deterministic host regression test for startup secondary-index
rebuild over a bounded multi-row window. This is a characterization floor, not
a production IC recovery-size ceiling.

### Dataset Shape

- Rows: 256 authoritative `RecoveryIndexedEntity` rows.
- Primary key: one `Ulid` key per row.
- Secondary indexes: one non-unique generated index over `group`.
- Stale index state: one stale secondary-index entry with no authoritative row.
- Journal tail: empty; recovery authority is the marker plus canonical row
  store.
- Recovery operation: marker-authorized startup recovery clears stale secondary
  index state and rebuilds derived secondary entries from authoritative rows.

### Executable Claim

The regression test proves that guarded startup recovery:

- rebuilds exactly 256 row-derived secondary-index entries;
- drops stale index-only state;
- leaves the canonical row store unchanged;
- clears the marker after successful recovery;
- decodes sampled rebuilt entries back to their original primary keys.

### Non-Claim

This does not claim a production IC instruction-budget bound, a maximum
supported production index size, or streaming rebuild support. It only creates a
checked-in floor for deterministic host recovery behavior.

### Follow-Up

This satisfies the minimum 0.190 evidence-floor requirement. 0.191 should add a
larger measured host/PocketIC threshold, publish an explicit production
recovery-size bound, or design streaming fold/rebuild before the project makes
production-scale recovery guarantees.

## 0.191 Productization Notes

### Checksum Decision

0.189 and 0.190 now exercise deterministic corruption cases across
marker/journal bytes and a first row/schema/index/structural decode corpus.
Those tests prove fail-closed decode behavior for known malformed envelopes.
They do not prove detection of every random bit flip or adversarially
well-formed-but-wrong persisted value.

0.191 should decide whether the supported threat model includes accidental
stable-memory corruption, operator-supplied backups, or imported images from
outside the canister. If yes, checksums or authenticated envelope metadata
should be evaluated as a persisted-format feature. If no, the operator guide
must say that current protection is structural validation, not end-to-end
payload integrity.

### Backup, Restore, And Import Threat Model

Imported stable-memory images are plausibly in scope for real operators, but
0.190 does not claim that arbitrary imported bytes are safe beyond the tested
decode and recovery boundaries. 0.191 should define whether backup/restore
inputs are trusted operator artifacts, untrusted external artifacts, or
unsupported.

That decision should cover:

- marker, journal, row, schema, and index byte envelopes;
- whether unknown future versions hard-cut before `1.0.0`;
- whether checksum failures are corruption, incompatible format, or operator
  restore failures;
- whether restore should run a full integrity scan before serving reads.

### Recovery Size And Streaming Decision

The 256-row host characterization is a floor for deterministic recovery
behavior, not a production IC limit. 0.191 should choose one of:

- publish a conservative supported recovery-size bound;
- add PocketIC or instruction-budget measurements for a larger threshold;
- design streaming fold/rebuild before broad production durability claims.

Until then, large-index recovery remains correct-by-construction for the tested
window but not budget-certified for arbitrary production scale.

### Operator Durability Guide

The operator-facing guide should be explicit that:

- heap stores are volatile and not durable;
- journaled stores rely on guarded recovery before normal access;
- non-atomic helpers may leave prefix commits visible;
- SQL-style transaction blocks and snapshot pagination are not provided;
- production-scale recovery bounds are still being characterized.
