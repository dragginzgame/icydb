# 0.157 DDL Continuation

Initial design boundary and closeout summary for the 0.157 DDL line.

## Status

Completed.

## Purpose

0.155 established the first supported SQL DDL path around `CREATE INDEX` for
accepted field-path indexes. 0.156 then moved endpoint generation and developer
tooling behind explicit `icydb.toml` config.

0.157 should continue DDL by expanding the supported catalog mutation surface
without weakening the existing authority model.

The target is:

IcyDB can expose more useful DDL workflows while keeping catalog-native schema
mutation as the implementation authority, keeping unsupported DDL fail-closed,
and making the resulting schema state visible through the existing schema,
describe, explain, and query planning surfaces.

## Why A Design Doc Is Worth It

Small parser or CLI additions do not need a design document. DDL shape changes
do need one when they affect:

- accepted schema snapshot contents
- catalog mutation admission rules
- physical rebuild or cleanup work
- index visibility in `DESCRIBE`, schema endpoint payloads, or planner metadata
- cache, cursor, or explain identity after schema publication
- failure semantics for partially prepared or partially rebuilt catalog changes

That is the expected 0.157 work area, so this design doc should be the active
place to keep decisions before code lands.

## Non-Goals

0.157 should not:

- treat SQL text as schema authority
- add compatibility readers for older pre-1.0 DDL formats
- add broad migration scripting
- implement physical descending index key ordering; `DESC` belongs with the
  0.159 compact key/index encoding work because it changes byte ordering,
  index scan direction semantics, planner eligibility, and cursor identity
- add runtime TOML/config parsing
- weaken controller-gated generated endpoint boundaries
- make unsupported DDL silently no-op
- introduce dynamic typing or runtime-only entity lookup for normal typed APIs

## Hard Invariants

- Accepted schema snapshots remain runtime authority.
- SQL DDL is a frontend over catalog-native mutation semantics.
- Unsupported DDL fails before physical work or schema publication.
- A successful DDL publication must be visible through accepted-schema
  introspection surfaces.
- Query-plan cache and continuation cursor identity must include enough schema
  authority to reject stale post-DDL plans.
- Physical index work must be validated before accepted metadata is published.
- Generated models remain proposal/convenience inputs, not runtime authority.

## Completed DDL Work

0.157 completed the index-DDL candidates:

- hardened `CREATE INDEX` naming and duplicate-contract diagnostics
- added `DROP INDEX` for DDL-published indexes only
- exposed generated-vs-DDL index origin metadata through accepted schema,
  `SHOW INDEXES`, `DESCRIBE`, and schema CLI surfaces
- added DDL execution reports with `published` and `no_op` statuses
- added conservative `CREATE INDEX IF NOT EXISTS` and `DROP INDEX IF EXISTS`
- added unique, composite, filtered, explicit-`ASC`, expression, and unique
  expression index support

`CREATE INDEX ... (field DESC)` is deliberately not a 0.157 candidate. 0.157
may continue to parse and reject `DESC` clearly, but support should land in
0.159 with the physical key ordering redesign.

Future DDL candidates should state whether they are metadata-only,
rebuild-required, or cleanup-required before implementation begins.

## Open Decisions

- Further table/field DDL belongs to 0.158.
- Physical descending index key ordering belongs to 0.159.
- Planner transparency over the resulting access paths belongs to 0.160.

## Validation Expectations

DDL slices should normally include:

- core parser and lowering coverage for accepted and rejected SQL shapes
- accepted-catalog mutation coverage
- generated canister endpoint coverage when endpoint behavior changes
- schema/describe/explain visibility coverage after publication
- stale cache/cursor coverage when schema identity changes
- clippy and formatting checks for touched crates
