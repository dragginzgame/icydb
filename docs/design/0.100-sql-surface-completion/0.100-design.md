# 0.100 Single-Table SQL Surface Completion

## Thesis

`0.100` should complete the **single-table SQL surface** on top of the now-stable semantic, execution, DISTINCT, and parameterization foundations, so the engine feels feature-complete within its current non-join scope.

> **0.100 expands the admitted SQL expression and scalar-function surface without changing the core execution architecture, making single-table SQL broadly usable before the engine turns to aggressive optimization and later multi-table work.**

This slice is about **surface breadth**, not architectural reinvention.

---

## Problem statement

After `0.99`, the engine has:

* unified planner-owned expression semantics
* compiled execution across production paths
* correct DISTINCT / ORDER / paging behavior
* fixed-route parameterized prepared execution
* symbolic-slot prepared templates in the internal parameterized lane

So the core is now strong.

However, the engine can still feel **feature-thin** at the SQL surface:

* common scalar functions may still be missing or incomplete
* expression support may still vary by clause
* some standard boolean/null conveniences may still be absent or uneven
* users may still hit “simple SQL” gaps despite the engine being architecturally mature

At this point, the limiting factor is no longer execution design. It is **surface completeness**.

---

## Design goal

Complete the single-table SQL surface so that:

* common scalar functions are available
* common expression forms behave consistently across clauses
* standard boolean/null convenience forms are supported where semantically appropriate
* the engine feels complete within its current single-table boundary

The goal is:

> **finish breadth before deep optimization or JOIN, so the engine’s visible SQL surface matches the maturity of its internal architecture.**

---

## Non-goals

This slice does **not** aim to:

* add JOIN
* add subqueries
* add window functions
* redesign planner or executor architecture
* introduce a new execution IR or VM
* widen value-sensitive parameterization
* rethink DISTINCT / ORDER / paging contracts
* do a broad performance-only pass

This is a **surface completion** slice, not a structural or multi-table slice.

---

## Why now

The engine now has the right prerequisites:

* semantic convergence is done
* execution convergence is done
* DISTINCT/paging correctness is done
* parameterization is in place

That means adding surface features now is much safer than it would have been earlier. New scalar functions and expression forms can land on top of a stable core rather than forcing repeated execution-path rework.

So `0.100` is the right moment to:

> **finish the single-table SQL language before turning to optimization or JOIN.**

---

# Part I — Current boundary

## What is already strong

The engine already has:

* planner-owned `Expr`
* compiled predicate/projection/grouped execution
* correct post-projection DISTINCT
* stable ORDER BY and paging contracts
* prepared execution for the admitted parameterized subset

## What still feels incomplete

Depending on current admitted surface, likely gaps still include some combination of:

* missing common scalar functions
* incomplete CASE support or clause parity
* uneven admission of expression forms across SELECT / WHERE / HAVING / ORDER BY
* incomplete boolean convenience forms
* incomplete null-handling function surface
* missing polish around `IN`, `BETWEEN`, or similar common syntax

The architecture is mature enough that these are now **surface gaps**, not core-engine gaps.

---

# Part II — Core design

## 1. Complete a baseline scalar-function set

`0.100` should add a bounded but useful set of scalar functions that fit naturally into the existing `Expr` and compiled-execution model.

### 1.1 Text functions

Admit a practical baseline such as:

* `LOWER(x)`
* `UPPER(x)`
* `LENGTH(x)`
* `TRIM(x)`
* `LTRIM(x)`
* `RTRIM(x)`

These are high-value, deterministic, and naturally single-row.

### 1.2 Numeric functions

Admit a practical baseline such as:

* `ABS(x)`
* `ROUND(x [, scale])`
* `CEIL(x)` / `CEILING(x)`
* `FLOOR(x)`

These complete the basic numeric shaping surface.

### 1.3 Null/value-selection functions

Admit a practical baseline such as:

* `COALESCE(a, b, ...)`
* `NULLIF(a, b)`

These are important for real-world query shaping and help round out expression completeness.

### Explicit rule

> Scalar functions in `0.100` must lower through the shared expression model and execute through the existing compiled-expression seams. No function-specific executor lane may be introduced.

---

## 2. Finish the generic scalar-function model

If any bounded parser-era special handling remains for individual functions, `0.100` should continue the move toward a generic scalar-function admission model:

```text
SQL function call
  -> planner-owned function expression
  -> shared type inference / validation
  -> shared compiled execution
```

The parser should own syntax only. Function semantics should remain planner/runtime owned.

### Explicit rule

> Scalar functions are expressions, not clause-specific or parser-owned semantic islands.

---

## 3. Complete expression parity across clauses

The same admitted scalar-expression forms should behave consistently wherever the current non-join SQL surface already admits expressions.

That includes, as appropriate:

* `SELECT`
* `WHERE`
* `HAVING`
* `ORDER BY`

This does not mean every clause must admit every shape blindly. It means:

* phase rules remain intact
* type rules remain intact
* but there should be no arbitrary clause-specific divergence for ordinary scalar expressions

### Explicit rule

> Clause meaning is still phase-defined, but admitted ordinary scalar expression forms should not diverge unnecessarily across SQL clauses.

---

## 4. Complete CASE support cleanly

`CASE` is one of the highest-value expression surfaces for real queries and should be treated as first-class within the admitted single-table SQL boundary.

`0.100` should support the currently intended `CASE` forms consistently across the clauses where expressions are already allowed and semantically meaningful.

### Target examples

```sql
SELECT CASE WHEN strength > 10 THEN 'high' ELSE 'low' END
FROM Character;
```

```sql
SELECT name
FROM Character
WHERE CASE WHEN active THEN 1 ELSE 0 END = 1;
```

```sql
SELECT COUNT(*)
FROM Character
HAVING CASE WHEN COUNT(*) > 5 THEN TRUE ELSE FALSE END;
```

where those shapes fit the already-established phase/type rules.

### Explicit rule

> CASE remains an ordinary planner-owned expression and must not introduce a special evaluator path.

---

## 5. Finish common boolean convenience forms

Where semantically appropriate and already aligned with the engine’s boolean model, `0.100` should complete support for common boolean convenience syntax such as:

* `IS TRUE`
* `IS FALSE`

and any already-intended related forms, provided they preserve the current unified boolean boundary:

* `TRUE` admits
* `FALSE` rejects
* `NULL` rejects

### Explicit rule

> Boolean convenience forms must lower into the same unified expression/boolean pipeline and must not introduce clause-specific boolean semantics.

---

## 6. Finish common predicate sugar within the current model

`0.100` should also tighten common SQL predicate sugar where it fits naturally into the single-table expression system and existing route/normalization model.

Candidates include:

* `BETWEEN`
* `IN (...)`

This does not require inventing new planner architecture. It means ensuring these common surfaces are admitted cleanly, normalized consistently, and behave predictably with:

* literals
* prepared parameters where already admissible
* existing boolean/null contracts

### Explicit rule

> Predicate sugar should normalize into the existing shared semantic model rather than introducing isolated execution paths.

---

## 7. Keep parameterization aligned with the widened surface, but do not over-widen it

`0.100` should not turn into a second parameterization slice.

When adding new surface forms, the parameterization contract should remain explicit:

* if a new function/predicate shape is safely value-insensitive and already within the admitted prepared-template lane, it may be parameterizable
* if not, it should remain unsupported for prepared templates or continue to use fallback behavior

This keeps `0.100` focused on surface completion rather than reopening the parameterization boundary.

---

# Part III — Contracts

## Contract 1 — No new execution architecture

All new surface forms must reuse:

* shared lowering onto planner `Expr`
* shared type inference / validation
* existing compiled execution seams

No parser-owned or clause-owned executor islands should be added.

---

## Contract 2 — Phase rules still define clause legality

Expression breadth increases, but existing phase ownership remains intact:

* pre-aggregate forms stay pre-aggregate
* post-aggregate forms stay post-aggregate
* grouped/global HAVING rules stay governed by the same planner constraints

`0.100` is not allowed to blur these boundaries in the name of surface completion.

---

## Contract 3 — No special boolean model

All boolean-capable new forms must reuse the existing unified boolean collapse boundary and null behavior.

---

## Contract 4 — Parameterization remains bounded

New surface admission must not silently widen prepared execution beyond the explicit parameterization contracts already established in `0.98` and `0.99`.

---

## Contract 5 — No value-sensitive planning widening

`0.100` must not pull in `LIKE ?`-style route-template specialization or other value-sensitive prepared planning work. That remains a separate concern.

---

## Numeric

## Null/value

This is enough to make single-table SQL feel much more complete without overreaching into multi-table or analytical features.

---

# Part V — Explain and observability

Because `0.100` widens admitted expression surface, explain output should remain consistent with the existing planner-owned model.

## 1. Explain should show the new forms as ordinary expressions

New scalar functions, CASE forms, and sugar-normalized predicates should appear in explain output as part of the same semantic expression model rather than as bespoke front-door artifacts.

## 2. Explain JSON should stay aligned

Structured explain should represent these new forms consistently with the existing expression schema or node family conventions already in use.

## 3. Explain execution should not diverge semantically

If execution explain is supported for these forms, it should reflect the same compiled semantic structure, not an alternative clause-specific interpretation.

---

# Part VI — Validation and type behavior

`0.100` should be explicit about function and expression validation.

## 1. Arity checking

Every admitted scalar function must have a fixed arity contract or a clearly defined variadic contract.

Examples:

* `LOWER(x)` -> arity 1
* `COALESCE(a, b, ...)` -> variadic, minimum arity 2
* `ROUND(x [, scale])` -> arity 1 or 2

## 2. Type checking

Each function must define:

* admitted argument families
* result type behavior
* null propagation behavior

## 3. Fail closed on ambiguous or unsupported shapes

If a function or expression form would require unclear coercion, phase widening, or special-case runtime treatment, it should be rejected rather than admitted partially.

---

# Part VII — Acceptance criteria

## Functional

The engine should support a meaningful baseline of common scalar and expression forms in single-table SQL, including representative examples such as:

```sql
SELECT LOWER(name) FROM Character;
SELECT UPPER(name) FROM Character;
SELECT LENGTH(name) FROM Character;
SELECT ABS(score_delta) FROM Character;
SELECT ROUND(score, 2) FROM Character;
SELECT COALESCE(alias, name) FROM Character;
SELECT NULLIF(alias, name) FROM Character;
SELECT CASE WHEN strength > 10 THEN 'high' ELSE 'low' END FROM Character;
SELECT name FROM Character WHERE active IS TRUE;
SELECT name FROM Character WHERE strength BETWEEN 5 AND 10;
SELECT name FROM Character WHERE name IN ('Alice', 'Bob');
SELECT COUNT(*) FROM Character HAVING COUNT(*) > 5;
```

with clause-appropriate variants where admitted.

## Architectural

* all new forms lower through shared planner `Expr`
* all new forms execute through existing compiled seams
* no new parser/executor semantic islands are introduced
* no value-sensitive parameterization widening occurs implicitly

## Behavioral

* boolean/null semantics remain consistent
* DISTINCT / ORDER / paging contracts remain unchanged
* explain surfaces remain aligned

## Prepared execution

Where a new form is inside the already-admitted fixed-route prepared subset, prepared execution should continue to work through the existing parameterized/template machinery. Unsupported parameterized placements should fail closed or continue to use the documented fallback behavior.

# Part IX — Risks

## Risk 1 — accidental architectural drift

It would be easy to add one-off function or clause-specific helpers. `0.100` must avoid that and keep all new surface forms on the shared expression path.

## Risk 2 — over-widening scope

If too many exotic functions or predicate families are attempted at once, this slice will become messy. Keep the set practical and common.

## Risk 3 — parameterization drift

New forms may tempt implicit widening of the prepared-template lane. That should remain explicit and bounded.

## Risk 4 — inconsistent clause admission

If surface forms are added piecemeal without a parity rule, the engine may accumulate confusing “works here but not there” behavior again.

---

# Part X — Out of scope

The following stay out of `0.100`:

* JOIN
* subqueries
* window functions
* value-sensitive prepared route templates
* full optimizer redesign
* broad performance-only work
* public API redesign around parameterization

---

# Part XI — Follow-on

Once `0.100` lands, the engine should have:

* a mature single-table semantic model
* compiled execution throughout
* correct result pipeline behavior
* bounded prepared execution
* a broadly complete single-table SQL surface

That sets up two clean next directions:

* **optimization pass** over a now-stable surface
* later **JOIN** work on top of a more feature-complete base

---

## Thesis sentence

> **0.100 completes the practical single-table SQL surface by adding a baseline scalar-function set and finishing common expression forms on top of the existing shared expression and compiled execution model, without widening into multi-table or architectural redesign work.**
