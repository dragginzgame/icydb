# 0.100 Single-Table SQL Surface Completion

## Thesis

`0.100` should complete a **bounded practical subset** of the remaining
single-table SQL surface on top of the now-stable semantic, execution,
DISTINCT, and parameterization foundations.

> **0.100 ships one explicit, auditable set of common scalar functions and
> expression forms on top of the existing shared expression and compiled
> execution model, without widening into multi-table work, planner redesign, or
> implicit prepared-execution expansion.**

This slice is about **surface completion through a locked inventory**, not
surface discovery during implementation.

---

## Problem statement

After `0.99`, the engine has:

* unified planner-owned expression semantics
* compiled execution across production paths
* correct DISTINCT / ORDER / paging behavior
* bounded parameterized prepared execution
* symbolic-slot prepared templates on the fixed-route template lane

So the core is now strong.

However, the SQL surface can still feel incomplete in practical single-table
use:

* common scalar functions are still missing or uneven
* some expression forms remain absent or clause-fragmented
* some standard boolean/null conveniences remain incomplete
* users can still hit "simple SQL" gaps even though the engine architecture is
  already mature

The main risk for `0.100` is no longer architectural uncertainty. It is
**implementation-led scope drift**.

So this document must behave as a release contract:

* it must state exactly what `0.100` ships
* it must state exactly what remains fail-closed
* it must make clause admission and prepared-execution status explicit per
  feature family

---

## Design goal

Complete one bounded practical subset of the single-table SQL surface so that:

* common scalar functions are available
* common expression forms have explicit clause contracts
* boolean/null convenience forms are available where they fit the existing
  model
* the engine feels materially more complete within its current single-table
  boundary

The goal is:

> **finish one practical audited subset before optimization or JOIN, and do it
> with explicit boundaries rather than open-ended surface expansion.**

---

## Non-goals

`0.100` does **not** aim to:

* add JOIN
* add subqueries
* add window functions
* redesign planner or executor architecture
* introduce planner-wide parameter nodes
* widen value-sensitive prepared planning
* widen the fixed-route prepared-template lane implicitly
* ship a generic "all remaining scalar SQL" completion pass
* do a broad optimization-only pass

This is a **bounded shipped subset** slice, not a "finish every remaining SQL
surface gap" slice.

---

## Explicit current boundary

`0.100` builds on the current single-table expression and execution boundary.
That means:

* all new surface forms must lower through the shared planner expression model
* all new surface forms must execute through existing compiled-expression seams
* parser-owned semantics remain prohibited
* clause legality remains phase-defined
* grouped/global planner boundaries remain intact
* prepared execution stays bounded by the explicit `0.98` and `0.99`
  contracts

`0.100` does **not** reopen:

* generic grouped projection-expression widening
* generic function admission in `GROUP BY`
* planner-wide parameterization as part of normal expression ownership
* `LIKE ?` or other value-sensitive route-template specialization
* general `IN` expression-list coercion work
* broad coercion redesign

If a form is not listed in the shipped inventory below, it stays out of scope
and fail-closed for `0.100`.

---

## Normative shipped inventory

`0.100` ships **exactly** these new SQL surface families:

### Text scalar functions

* `LOWER(x)`
* `UPPER(x)`
* `LENGTH(x)`
* `TRIM(x)`
* `LTRIM(x)`
* `RTRIM(x)`

### Numeric scalar functions

* `ABS(x)`
* `ROUND(x)`
* `ROUND(x, scale)`
* `CEIL(x)`
* `CEILING(x)`
* `FLOOR(x)`

### Null/value-selection scalar functions

* `COALESCE(a, b, ...)`
* `NULLIF(a, b)`

### Expression forms and predicate sugar

* searched `CASE`
  * `CASE WHEN <pred> THEN <expr> [WHEN ... THEN ...] [ELSE <expr>] END`
* `BETWEEN`
* `IN (...)` with:
  * literal lists
  * bounded parameter lists
* `IS TRUE`
* `IS FALSE`

### Explicitly out of scope in `0.100`

Everything not listed above remains out of scope and fail-closed, including:

* simple `CASE <expr> WHEN ...`
* custom-charset `TRIM`
* `IS NULL` / `IS NOT NULL` widening if not already shipped elsewhere
* broad function-catalog expansion beyond the listed families
* general expression lists inside `IN (...)`
* implicit widening of prepared-template support for newly admitted syntax
* new clause families or projection semantics beyond the matrix below

## Clause admission matrix

The matrix below is normative for `0.100`.

Legend:

* `Admitted`: shipped directly in this slice
* `Narrowed`: admitted only under the notes-column restriction
* `Normalized`: accepted only because lowering rewrites it to an already-owned
  semantic shape
* `Rejected`: fail-closed in this slice

| Feature family | SELECT | WHERE | GROUP BY | HAVING | ORDER BY | Notes |
| --- | --- | --- | --- | --- | --- | --- |
| Text scalar functions | Admitted | Admitted | Rejected | Narrowed | Admitted | `HAVING` only when arguments are already-admitted grouped/HAVING scalar inputs; no new grouped ownership |
| Numeric scalar functions | Admitted | Admitted | Rejected | Narrowed | Admitted | Same boundary as text functions |
| `COALESCE` / `NULLIF` | Admitted | Admitted | Rejected | Narrowed | Admitted | Same boundary as other scalar functions |
| searched `CASE` | Admitted | Admitted | Rejected | Narrowed | Admitted | `WHEN` arms remain phase-correct; no grouped widening by analogy |
| `BETWEEN` | Normalized | Normalized | Rejected | Narrowed | Normalized | Lowers to existing compare-family bounds; `HAVING` only where the lowered compare shape is already legal |
| `IN (...)` literal lists | Normalized | Normalized | Rejected | Narrowed | Rejected | Lowers to existing equality-membership semantics; no direct `ORDER BY IN (...)` admission |
| `IN (...)` bounded parameter lists | Normalized | Normalized | Rejected | Narrowed | Rejected | Same semantic boundary as literal-list `IN`, with explicit prepared status below |
| `IS TRUE` / `IS FALSE` | Normalized | Normalized | Rejected | Narrowed | Rejected | Lowers to the existing unified boolean boundary; no standalone ordering form |

`GROUP BY` remains intentionally narrow in `0.100`. This slice does **not**
reopen generic expression admission there.

## Prepared-execution status matrix

Each shipped family must carry one explicit prepared status.

Legend:

* `Fixed-route template`: admitted on the `0.99` symbolic-slot fixed-route
  template lane
* `Fallback prepared only`: admitted to prepared execution only through the
  non-template fallback path
* `Not prepared`: not admitted for prepared execution in `0.100`

| Feature family | Prepared status | Notes |
| --- | --- | --- |
| Text scalar functions | Fallback prepared only | No new fixed-route template widening is promised for function-owned shapes |
| Numeric scalar functions | Fallback prepared only | Same as text functions |
| `COALESCE` / `NULLIF` | Fallback prepared only | New null/value-selection forms do not widen the template lane |
| searched `CASE` | Fallback prepared only | Remains outside the fixed-route template lane in `0.100` |
| `BETWEEN` on field-target compare families | Fixed-route template | Only when lowering produces the already-admitted compare-family range shape |
| `BETWEEN` outside that narrowed shape | Not prepared | Fail-closed rather than silently widening template coverage |
| `IN (...)` literal lists | Not prepared | No parameter slots are involved |
| `IN (...)` bounded parameter lists | Fallback prepared only | `0.100` does not promise fixed-route template admission for list membership |
| `IS TRUE` / `IS FALSE` on admitted boolean compare-family shapes | Fixed-route template | This relies on the `0.99` bool-safe symbolic-slot model |
| `IS TRUE` / `IS FALSE` outside that narrowed shape | Fallback prepared only | No broader template widening is implied |

If a parameterized placement is not explicitly covered above, it remains out of
scope for prepared execution in `0.100`.

## Per-family semantic contracts

This section is normative. It is not an implementation reminder.

### Text scalar functions

#### `LOWER(x)` / `UPPER(x)`

* arity: exactly 1
* input: text
* result: text
* null behavior: null propagates
* failure mode: non-text input rejects fail-closed

#### `LENGTH(x)`

* arity: exactly 1
* input: text
* result: integer length
* semantic rule: length is measured over text characters, not raw byte length
* null behavior: null propagates
* failure mode: non-text input rejects fail-closed

#### `TRIM(x)` / `LTRIM(x)` / `RTRIM(x)`

* arity: exactly 1
* input: text
* result: text
* semantic rule: trim leading/trailing whitespace only
* custom trim-character syntax is out of scope
* null behavior: null propagates
* failure mode: non-text input rejects fail-closed

### Numeric scalar functions

#### `ABS(x)`

* arity: exactly 1
* input: numeric
* result: same numeric family where supported by the existing numeric model
* null behavior: null propagates
* failure mode: non-numeric input rejects fail-closed

#### `ROUND(x)`

* arity: exactly 1
* input: numeric
* result: rounded numeric value in the existing numeric family model
* semantic rule: rounds to integral scale
* null behavior: null propagates
* failure mode: non-numeric input rejects fail-closed

#### `ROUND(x, scale)`

* arity: exactly 2
* inputs:
  * first argument: numeric
  * second argument: integer scale
* result: rounded numeric value using the supplied integer scale
* semantic rule: ambiguous coercions are rejected rather than inferred loosely
* null behavior: null propagates if either argument is null
* failure mode: non-numeric first argument or non-integer scale rejects
  fail-closed

#### `CEIL(x)` / `CEILING(x)` / `FLOOR(x)`

* arity: exactly 1
* input: numeric
* result: rounded numeric value in the existing numeric family model
* semantic rule: `CEIL` and `CEILING` are synonyms
* null behavior: null propagates
* failure mode: non-numeric input rejects fail-closed

### Null/value-selection functions

#### `COALESCE(a, b, ...)`

* arity: variadic, minimum 2
* inputs: arguments must share one admitted common result family under the
  existing scalar coercion rules
* result: first non-null argument, or null if all arguments are null
* result type: the admitted common family
* null behavior: returns first non-null argument, not blanket null propagation
* failure mode: incompatible argument families reject fail-closed

#### `NULLIF(a, b)`

* arity: exactly 2
* inputs: arguments must be comparable under the existing equality family rules
* result: null when `a = b`, otherwise the first argument
* result type: type family of the first argument
* null behavior: follows the comparison and first-argument contract above
* failure mode: incomparable argument families reject fail-closed

### searched `CASE`

* admitted form:
  * `CASE WHEN <predicate> THEN <expr> [WHEN ... THEN ...] [ELSE <expr>] END`
* simple `CASE <expr> WHEN ...` is out of scope
* `WHEN` arms:
  * must be boolean-valid under the existing unified boolean boundary
* `THEN` / `ELSE` arms:
  * must share one admitted common result family under existing scalar coercion
    rules
* result:
  * first matching `THEN` branch
  * else `ELSE`
  * else null if `ELSE` is omitted
* phase rule:
  * `CASE` does not widen clause phase ownership
  * aggregate-bearing arms are legal only where those aggregate-bearing
    expressions are already legal
* failure mode:
  * invalid boolean `WHEN` arms or incompatible result arms reject fail-closed

### `BETWEEN`

* admitted form:
  * `<expr> BETWEEN <lo> AND <hi>`
* semantic rule:
  * lowers to the existing compare/range semantic model
  * no bespoke runtime path is introduced
* parameterization rule:
  * fixed-route template admission is promised only for already-admitted
    compare-family field-target range shapes
* failure mode:
  * shapes that require new compare families or phase widening reject
    fail-closed

### `IN (...)`

#### Literal-list `IN`

* admitted form:
  * `<expr> IN (<literal>, <literal>, ...)`
* semantic rule:
  * lowers to existing equality-membership semantics
* input rule:
  * list members must be coercible to one admitted comparison family with the
    left-hand side
* failure mode:
  * incompatible or ambiguous list coercion rejects fail-closed

#### Bounded parameter-list `IN`

* admitted form:
  * `<expr> IN (?, ?, ...)`
* semantic rule:
  * fixed list arity is part of the query shape
  * `0.100` admits this form only on the fallback prepared lane
* failure mode:
  * broader expression-list membership or implicit template widening rejects
    fail-closed

### `IS TRUE` / `IS FALSE`

* admitted forms:
  * `<expr> IS TRUE`
  * `<expr> IS FALSE`
* semantic rule:
  * lower into the existing unified boolean boundary rather than adding a new
    boolean evaluator path
* boolean boundary:
  * true admits
  * false rejects
  * null rejects
* failure mode:
  * non-boolean-valid shapes reject fail-closed

## Core implementation contracts

### Contract 1 - No new execution architecture

All shipped families must reuse:

* shared lowering onto planner-owned expressions
* shared type inference / validation
* existing compiled execution seams

No parser-owned or clause-owned semantic islands may be introduced.

### Contract 2 - Phase rules still define legality

Surface breadth widens, but existing phase ownership remains intact:

* pre-aggregate forms stay pre-aggregate
* post-aggregate forms stay post-aggregate
* grouped/global `HAVING` rules stay planner-owned

`0.100` is not allowed to blur these boundaries in the name of convenience.

### Contract 3 - Normalization is preferred to new runtime lanes

Where `BETWEEN`, `IN`, or boolean convenience can lower to an already-owned
semantic shape, `0.100` should normalize them there instead of inventing new
runtime behavior.

### Contract 4 - Prepared widening must be explicit

No shipped family may silently widen the fixed-route prepared-template lane.
Prepared status is exactly what the matrix above says.

## Verification matrix

Each shipped family requires explicit coverage across the ownership boundaries
that `0.100` touches.

| Feature family | Parser | Lowering / normalization | Execution | Explain | Prepared coverage | Negative coverage |
| --- | --- | --- | --- | --- | --- | --- |
| Text scalar functions | Parse call syntax and arity | Lower to builtin function expressions | Execute through shared compiled scalar evaluation | Render as ordinary expressions | Fallback prepared execution only | Reject wrong arity, wrong type, `GROUP BY` use |
| Numeric scalar functions | Parse call syntax and optional `ROUND` scale | Lower to builtin function expressions | Execute through shared compiled scalar evaluation | Render as ordinary expressions | Fallback prepared execution only | Reject wrong arity, wrong type, ambiguous coercion |
| `COALESCE` / `NULLIF` | Parse variadic/binary forms | Lower to shared expression nodes with common-type validation | Execute through shared scalar evaluation | Render as ordinary expressions | Fallback prepared execution only | Reject incompatible argument families |
| searched `CASE` | Parse searched `CASE` only | Lower to planner-owned `CASE` expression | Execute through shared compiled expression evaluation | Render as ordinary expression form | Fallback prepared execution only | Reject simple `CASE`, invalid `WHEN`, incompatible result arms |
| `BETWEEN` | Parse syntax | Normalize to compare/range semantics | Execute on existing compare/range paths | Explain the normalized shape coherently | Fixed-route template only for the narrowed compare-family case | Reject widened compare families or illegal clause placements |
| `IN (...)` literal lists | Parse syntax | Normalize to equality-membership semantics | Execute on existing equality-membership path | Explain normalized membership shape | Not prepared | Reject incompatible list coercion, reject widened expression lists |
| `IN (...)` bounded parameter lists | Parse syntax | Normalize to the same equality-membership semantics | Execute on existing equality-membership path | Explain normalized membership shape | Fallback prepared execution only | Reject implicit fixed-route template widening |
| `IS TRUE` / `IS FALSE` | Parse syntax | Normalize to the unified boolean boundary | Execute through the existing boolean collapse path | Explain normalized boolean semantics | Fixed-route template only on the narrowed admitted boolean shape | Reject non-boolean-valid inputs and illegal clause placements |

`0.100` will likely touch these boundaries:

* SQL parser / model

  * `BETWEEN`
  * `IN (...)`
  * `IS TRUE` / `IS FALSE`
* SQL lowering / normalization
  * builtin function lowering
  * searched `CASE` lowering
  * `BETWEEN` normalization
  * `IN (...)` normalization
  * boolean convenience normalization
* planner expression model / validation
  * function identity
  * arity checking
  * common-type inference
  * null behavior contracts
  * boolean validity checks

  * shared scalar function evaluation
  * shared `CASE` evaluation
  * normalized compare / membership / boolean paths

  * textual explain rendering
  * structured explain rendering

  * fallback prepared coverage for function, `CASE`, and parameter-list `IN`
  * fixed-route template coverage only for the narrowed `BETWEEN` and boolean
    cases promised above

## Risks

### Risk 1 - implementation-led scope growth

If unlisted scalar functions or broader `CASE` / `IN` variants are admitted
during implementation, the slice loses its boundary and becomes another
open-ended SQL line.

### Risk 2 - grouped boundary drift

It would be easy to let "expression parity" reopen `GROUP BY` or grouped
projection ownership by accident. The matrix above forbids that.

### Risk 3 - implicit prepared widening

New syntax is likely to pressure the prepared-template lane. `0.100` must keep
that pressure fail-closed unless the prepared-status matrix explicitly admits a
shape.

### Risk 4 - semantic under-specification

If type coercion, null behavior, or clause legality stay informal, two
implementations can both claim to satisfy `0.100` while diverging. The
per-family contracts above are intended to prevent that.

---

## Out of scope

The following stay out of `0.100`:

* any unlisted scalar function family
* simple `CASE`
* custom `TRIM` character syntax
* general `GROUP BY` expression widening
* general expression-list `IN (...)`
* JOIN
* subqueries
* window functions
* value-sensitive prepared route templates
* optimizer redesign
* public API redesign around parameterization

---

## Follow-on

Once `0.100` lands, the engine should have:

* a mature single-table semantic model
* compiled execution throughout
* correct result-pipeline behavior
* bounded prepared execution
* one explicitly completed practical SQL surface subset

That sets up two clean next directions:

* optimization over a now-stable shipped surface
* later multi-table work on top of a more feature-complete single-table base

> **0.100 completes one explicit, testable subset of the remaining practical
> single-table SQL surface by shipping a locked inventory of common scalar

> discovery.**
