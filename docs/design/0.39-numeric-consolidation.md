# 0.39 — Numeric Consolidation

## Status

In Progress  
Target: `0.39.x`  
Scope: numeric semantics unification across planner typing, expression evaluation, aggregate arithmetic, and continuation/fingerprint stability.

---

## 1. Objective

`0.39` consolidates numeric behavior into one explicit semantic contract so
planner and executor cannot drift on:

- numeric type admissibility
- numeric coercion/comparison behavior
- arithmetic null/invalid handling
- aggregate numeric eligibility

The release line is correctness-first and fail-closed: when semantic intent
cannot be proven deterministic, planning must reject shape construction.

---

## 2. Baseline at 0.38.3

The projection/expression spine is active and grouped projection evaluation is
projection-driven. Numeric behavior is still partially distributed across
multiple modules with overlapping predicates and evaluators.

Current state:

- canonical expression and projection semantics exist (`Expr`, `ProjectionSpec`)
- grouped and scalar projection evaluation execute through expression paths
- continuation/fingerprint projection hashing is semantic and alias-insensitive
- grouped continuation identity uses explicit grouped-shape hashing

Remaining gap:

- numeric domain and coercion rules are not yet defined as one shared,
  planner-owned contract used uniformly by all execution paths.

---

## 3. Non-Goals

- No joins.
- No window functions.
- No cost-based planning.
- No expression pushdown into storage or index engines.
- No dynamic function/operator registration.
- No relational expansion beyond single-entity algebra.

---

## 4. Architectural Invariants

### N1. Single Numeric Contract

Numeric admissibility and coercion rules must be declared once and consumed by
planner and executor through shared contracts.

### N2. Planner Owns Semantic Validity

Invalid numeric expression shapes must fail at planning boundaries. Executor
enforces only invariant checks for bypassed invalid shapes.

### N3. Deterministic Runtime Arithmetic

Expression arithmetic and aggregate arithmetic must be deterministic,
null-stable, and independent of source ordering beyond declared operation
semantics.

### N4. Hash/Continuation Alignment

Numeric expression semantics used for execution must match the semantics used
for continuation/fingerprint identity.

### N5. No Storage-Layer Numeric Semantics

Numeric expression/coercion logic must not move into storage/index subsystems.

---

## 5. Execution Plan

## 5.1 0.39.0 — Numeric Domain Contract Extraction

Scope:

- centralize numeric field-kind capability predicates in one shared internal
  module
- remove duplicate hard-coded numeric field-kind lists where possible
- keep runtime behavior unchanged

Required outcomes:

- planner and executor import shared numeric domain helpers
- no duplicate numeric-kind tables in touched modules
- no API surface changes

## 5.2 0.39.1 — Planner Numeric Typing Closure

Scope:

- tighten expression type inference for numeric operations
- remove permissive `Unknown` pass-through for numeric operators where shape is
  statically decidable
- define explicit comparable-type compatibility for equality/order operators

Required outcomes:

- invalid numeric binary/unary shapes fail at planning
- type compatibility matrix is explicit and tested
- planner numeric typing is deterministic across equivalent shapes

## 5.3 0.39.2 — Runtime Numeric Evaluator Convergence

Scope:

- align scalar and grouped expression evaluators on one numeric coercion
  contract
- eliminate divergence between comparison and arithmetic coercion paths
- keep evaluator pure and post-materialization

Required outcomes:

- one canonical numeric arithmetic/comparison contract at runtime
- grouped/scalar parity tests for numeric expressions
- no trait-object/dynamic evaluator dispatch

## 5.4 0.39.2 — Predicate + Range Numeric Convergence

Scope:

- align `WHERE` and grouped `HAVING` numeric comparison semantics on the shared
  numeric comparison authority
- remove planner range-bound numeric variant ad hoc checks and route numeric
  bound comparisons through shared numeric compare helpers
- keep index pushdown boundaries explicit (strict-only compile path remains
  authoritative)

Required outcomes:

- numeric comparisons in predicate runtime and grouped HAVING are behaviorally
  aligned with shared numeric compare semantics
- planner range-bound compatibility checks do not rely on discriminant-based
  numeric variant gating
- index predicate compilation and range pushdown strict-only boundaries are
  explicit and regression-locked

## 5.5 0.39.2 — Aggregate Numeric Semantics Alignment

Scope:

- align field-target aggregate numeric eligibility and arithmetic coercion with
  planner numeric contracts
- preserve deterministic decimal behavior for `sum/avg` and distinct variants

Required outcomes:

- aggregate numeric field eligibility and expression numeric typing are
  intentionally aligned or intentionally documented where they differ
- regression matrix covers non-numeric rejection, overflow/error behavior, and
  distinct numeric stability

## 5.6 0.39.3 — Fingerprint/Continuation Numeric Drift Guards

Scope:

- add regression guards proving numeric semantic changes alter shape identity
  only when semantics change
- lock alias/no-op presentation changes as identity-stable

Required outcomes:

- continuation/fingerprint contracts stay stable under non-semantic numeric
  presentation changes
- semantic numeric changes invalidate continuation signatures

## 5.7 0.39.6+ — Guard Consolidation and Closure

Scope:

- add structural/signature guards at numeric contract boundaries
- remove stale or duplicate numeric semantic checks in executor paths

Required outcomes:

- no duplicated numeric semantic validation branches remain in production paths
- structural guards prevent new duplicate numeric contract definitions

---

## 6. Testing Requirements

`0.39` closure requires:

- planner numeric typing matrix tests:
  - unary/binary numeric valid/invalid combinations
  - comparison compatibility matrix
  - grouped projection numeric compatibility checks
- runtime numeric evaluation parity tests:
  - scalar vs grouped arithmetic/comparison parity
  - null propagation consistency
  - invalid-shape invariant failures when planner is bypassed
- aggregate numeric regression tests:
  - numeric target acceptance/rejection
  - `sum/avg` deterministic decimal behavior
  - distinct numeric stability under insertion-order variation
- continuation/fingerprint numeric drift tests:
  - semantic numeric expression changes invalidate continuation
  - alias/presentation-only changes do not invalidate

---

## 7. Definition of Done

`0.39` is done only when all conditions hold:

- shared numeric contract exists and is used by planner and executor callsites
  that own numeric semantics
- planner rejects invalid numeric expression shapes deterministically
- scalar/grouped expression evaluators use aligned numeric coercion rules
- aggregate numeric semantics are aligned with planner numeric contracts
- continuation/fingerprint behavior remains semantically aligned with runtime
  numeric semantics
- structural guards prevent numeric semantic drift and duplicate rule ownership
- workspace quality gates pass (`fmt`, `clippy`, `check`, `test`)

---

## 8. Release Safety Risks

Primary risks:

- numeric domain mismatch between planner and runtime evaluator
- grouped/scalar arithmetic divergence
- aggregate eligibility divergence from expression typing
- continuation reuse on numeric semantic drift
- accidental executor-side reintroduction of semantic numeric validation

Mitigations:

- shared contract extraction
- planner-first validation ownership
- parity and drift regression matrices
- structural signature/visibility guards
