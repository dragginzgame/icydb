# 0.213 Model And Schema Split: Additional Thoughts

Status: preserved exploratory discussion, not an approved design.

This companion note records the crate and product-boundary reasoning that led
to the broader 0.213 authority-separation topic. It is retained separately so
the main thoughts document can synthesize the direction without erasing the
original discussion.

Yes. After looking at the actual crates, I think this is the most honest architectural description of what you already have.

icydb-schema is not really “the database schema.” It is a build-time entity-model AST and validator. icydb-core::db::schema is the actual accepted database-schema authority. The confusing naming makes those two roles look more unified than they really are.

The existing macros reinforce this. #[entity], #[record], #[enum_], #[list], #[map], #[newtype], #[set] and #[tuple] constitute a fairly broad Rust data-model compiler. The RPG schema, for example, declares an empty Character marker and the macro generates the real Rust type and its behaviour from the definition. That is much more than an ordinary database derive. (example, macro entrypoints)

The conceptual product

I would describe it as:

A Rust entity-model system that generates Rust data types, derives and explicit application validation/normalization support, with an IcyDB integration that compiles the database-compatible portion into an IcyDB schema proposal.

The crucial authority rule remains:

The definition is the authoring source.
Generated Rust types are an application projection.
Application validation and normalization are another projection, invoked explicitly.
The IcyDB schema proposal is a database projection.
Once accepted, the accepted schema snapshot—not the definition or generated Rust model—is IcyDB’s runtime authority.

The source therefore produces both outputs. The Rust type should not produce the database schema through reflection, and IcyDB should not execute the Rust validation projection.

What the current code tells us

The desired seam already partly exists:

icydb-schema owns a build-time graph, nodes and validation.
icydb-core does not depend on icydb-schema.
Generated EntityModel metadata is explicitly described as proposal/reconciliation input; accepted schema remains runtime authority. (entity contract)
icydb-build reads the validated authoring graph and generates actor integration.

But icydb-schema-derive is not currently engine-independent. It directly depends on icydb-core and generates:

EntityModel, FieldModel and IndexModel proposals;
PersistedRow materialization;
runtime-value codecs;
field projections;
entity placement and keys;
database defaults and generated fields;
index predicates and relation metadata.

Its entity parser even imports core-owned EntityName and IndexName. (manifest, entity generator, persisted-row generator)

So I would not simply rename the existing two crates and declare them generic. They need an internal separation.

The package split I would aim for

Names are placeholders:

Package	Responsibility
model-core	Type/model IR, records, enums, collections, paths, issues, application validation and normalization traits
model-macros	Parse entity definitions and generate ordinary Rust types and derives
icydb-model	IcyDB-specific directives and lowering into schema proposals and runtime bridge implementations
icydb-core	Accepted catalog, layouts, persistence, constraints, planning and execution
icydb	Public database facade and convenient integration re-exports
icydb-build / icydb-config	Canister actor generation and IcyDB configuration

The generic/model side would own:

records, enums, newtypes, lists, maps, sets and tuples;
Rust struct/enum generation;
Clone, Eq, Ord, Serde and optional Candid derives;
application validation traversal;
application normalization traversal;
documentation and model metadata;
Rust construction defaults.

The IcyDB integration would own:

stores and canisters;
stable-memory allocation;
primary keys;
indexes and filtered-index expressions;
database relations;
accepted/database defaults;
generated and managed fields;
schema versions and fingerprints;
row layouts and persisted codecs;
EntityModel, FieldModel and IndexModel proposals;
PersistedRow, database projection and query traits.

One particularly important split is defaults:

rust_default       → model/Rust construction
icydb_insert_default → IcyDB schema proposal
historical_fill      → accepted IcyDB layout

Those must never collapse back into one ambiguous “default.”

Validation and normalization can live there

The companion system can absolutely provide:

user.normalize();
user.validate()?;

or generated visitor implementations.

That gives you the “all entity knowledge is discoverable together” experience. But IcyDB should not automatically invoke those application behaviours as database admission.

IcyDB independently enforces whatever was completely lowered into accepted semantics:

field types;
nullability;
checks;
unique constraints;
relations;
database defaults;
generated/managed-field policy.

That means a trim, phone-number parser or contextual validator can be declared or implemented alongside the model without becoming part of SQL or structural write semantics.

Should it remain in the IcyDB repository?

Yes—definitely for now.

A repository boundary and an architectural boundary are different things. Keeping the companion system in the IcyDB monorepo gives you:

atomic changes across the model compiler and IcyDB backend;
one CI run for generated-code compatibility;
no cross-repository release choreography;
easy hard cuts while everything remains pre-1.0;
independent Cargo packages that can still be published separately.

I would only consider a separate repository once all of these are true:

The generic model packages compile without depending on icydb, icydb-core or IcyDB-specific primitives.
The model-to-IcyDB projection has a stable intermediate contract.
There is a real second consumer besides IcyDB.
The companion system needs an independent release cadence.
Cross-repository compatibility would be cheaper than monorepo coordination.

Until then, I would use this rule:

Separate the product and dependency boundaries now; separate the repository only when independent usage proves that it is genuinely a separate project.

So yes: keep it all in the IcyDB repository, but reinterpret the workspace as containing both a companion Rust model system and an IcyDB backend/engine. That is a meaningful architectural correction, not merely branding.

Yes—I agree. I no longer think it needs to become a separate generic “companion product.” It will always be IcyDB’s model-authoring frontend; the real problem is that schema, authoring model, and accepted runtime schema currently share muddled names.

I’d use this vocabulary:

Current concept	Better name
Source definitions and their AST	Entity model
Validation of those definitions	Model validation
Generated Rust types and implementations	Rust model projection
Generated database description	Schema proposal
Activated runtime authority	Accepted schema
Persisted historical representation	Row layout

Then the crates could become:

icydb-model
    authored entity-model AST, graph and validation
    currently largely icydb-schema

icydb-model-macros
    #[entity], #[record], #[enum_], etc.
    currently icydb-schema-derive

icydb-build
    workspace/config discovery and generation orchestration

icydb-core
    accepted schema, layouts, storage, indexes and execution

icydb
    public database facade and model-macro re-exports

I prefer icydb-model-macros over icydb-model-derive because these macros do much more than derive traits from an existing Rust type—they often generate the type itself and several projections.

The overall relationship becomes:

IcyDB entity model
    ├── generates Rust types and implementations
    └── compiles an IcyDB schema proposal
                         ↓ acceptance
                  Accepted schema
                         ↓
                    IcyDB runtime

The main authority rules should be:

icydb-model owns the authoring language and its AST.
The macros own code generation, not runtime truth.
A schema proposal is untrusted input to reconciliation/acceptance.
icydb-core owns the accepted schema and enforces it.
Generated Rust types never reconstruct runtime schema authority.
Application normalization and contextual validation remain explicit Rust.
Anything completely representable by IcyDB may be lowered into the schema proposal.

I would keep everything in the IcyDB repository and under the icydb-* namespace. There is no benefit in pretending this is a database-neutral framework when its purpose is to provide IcyDB’s developer experience. The separation should be architectural, not artificial product independence.

One possible further naming improvement inside core is:

icydb_core::schema::accepted
icydb_core::schema::proposal
icydb_core::schema::reconcile
icydb_core::schema::layout

Or even rename the runtime module to accepted_schema if that makes authority clearer. The important thing is that bare schema must stop ambiguously meaning both:

the model AST developers author; and
the accepted database contract the engine executes.

So I think your conclusion is right: keep it integral to IcyDB, rename the authoring side around “model,” reserve “schema” for the database projection and accepted runtime authority, and untangle the crate dependencies without trying to create a generic external framework.
