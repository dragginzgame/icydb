# 0.213 Standalone IcyDB And External IC Model

Status: proposed

Design cut: 2026-07-23

Implementation order: after 0.212 closeout and explicit user direction to
start 0.213; before 0.214

This document defines the proposed product, repository, dependency, schema,
runtime, validation, and migration boundary between standalone IcyDB and the
external `ic-model` authoring system.

Implementation does not begin until this proposed design is reviewed and
accepted. Generic continuation within 0.212 does not authorize 0.213, and
generic continuation within 0.213 never authorizes 0.214.

The validator/sanitizer review exposed the immediate problem, but the broader
issue is that `model` and `schema` still name overlapping authoring, generated,
accepted, and runtime concepts. That makes proposal data look authoritative
and makes application callbacks look like database semantics when shared
execution happens to invoke them.

## Problem Statement

IcyDB already states the intended database authority correctly:

- source definitions author an entity model;
- generated metadata proposes database schema;
- reconciliation accepts one durable database schema; and
- the accepted schema snapshot is runtime authority.

The type and module vocabulary does not yet make that separation inevitable.
`EntityModel` is still described as a macro-generated runtime schema snapshot,
and some of its fields are described as authoritative for runtime planning.
`SchemaInfo` can be constructed from either trusted generated metadata or an
accepted snapshot. Runtime and model-only planning helpers therefore share
types and names even where their authority is different.

The desired seam exists in part:

- `db::schema::proposal` projects trusted `EntityModel` data into a typed
  proposal;
- accepted snapshots persist field IDs, layouts, constraints, indexes,
  relations, defaults, and write policies; and
- accepted read/write contracts already forbid runtime fallback
  reconstruction from generated models.

0.213 makes that seam structural rather than conventional.

The product boundary is:

- IcyDB is a standalone database for the Internet Computer that accepts
  a schema and can be used through SQL and dynamic structural values without
  generated Rust entity types;
- `ic-model` is a separate model-authoring and code-generation project;
- one set of `ic-model` declarations generates Rust application types,
  canister/store build output, and IcyDB schema fragments for database-scoped
  proposal composition; and
- IcyDB must not depend on `ic-model` or execute its application callbacks.

The current public surface does not yet meet that goal. Structural mutation and
SQL entrypoints still commonly require an entity type parameter, while
`EntityRuntimeHooks` combines generated model metadata with executable
callbacks. 0.213 removes that requirement from database authority rather
than merely rename the generated types.

The 0.211 constraint work establishes an important authority boundary:

- accepted constraints are durable database semantics;
- Rust validators and sanitizers are application behavior; and
- accepted checks evaluate the final canonical after-image.

The current implementation nevertheless runs Rust sanitizers and validators in
the shared typed/structural save preflight. Ordinary SQL writes therefore
usually observe those callbacks too, because the structural after-image is
materialized as the generated Rust entity before save validation.

That can make the two layers look unified even though they are not. Accepted
constraint activation, integrity verification, recovery replay, and any ingress
that cannot execute generated application behavior must remain independent of
Rust callbacks. Resumable SQL update already rejects entity graphs requiring
application callbacks rather than silently skipping them.

The result is a useful but potentially confusing combination:

1. resolve accepted defaults, generated values, managed values, and preserved
   values;
2. run applicable Rust sanitizers;
3. run Rust application validators;
4. construct the final canonical row;
5. evaluate accepted checks;
6. validate the remaining structural facts and stage the commit.

The ordering means accepted checks see sanitized values, which is desirable.
It also means that a Rust validator duplicating an accepted check can reject
first and hide the stable database-constraint diagnostic on ordinary writes.

## Contract Vocabulary

| Concept | Working meaning |
| --- | --- |
| Authoring schema | The host-only `ic-model` graph assembled by the current-style macros; it contains canister, store, entity, relation, type, validator, and normalizer declarations and is not accepted database schema |
| Entity model | The entity/type portion of the authored `ic-model` graph: entities, records, enums, collections, Rust-facing behavior, and explicit IcyDB directives |
| Model validation | Validation of the authored graph and macro declaration itself |
| Rust model projection | Generated Rust types, derives, visitors, application validation, and application normalization |
| Dynamic model descriptor | Runtime-inspectable application-model metadata generated by `ic-model`; not database authority |
| IcyDB schema contract | Small public, versioned proposal vocabulary owned by IcyDB and usable without `ic-model` |
| Canonical scalar atom | Engine-neutral Rust representation of one IcyDB scalar value, owned by the public protocol leaf and re-exported by both products; it contains no storage, clock, generation, query, or accepted-schema authority |
| Schema fragment | Reusable, store-free entity/type-scoped declaration keyed by immutable source identities; never submitted directly |
| Target store identity | Opaque IcyDB-issued routing identity for one configured store inside the target database; it is not a store definition or physical memory configuration |
| Schema proposal | Composed, database-scoped, multi-store-capable, non-authoritative change intent submitted atomically to reconciliation |
| Source key | Immutable author identity used for reconciliation; distinct from editable SQL and display names |
| Accepted schema | Persisted database authority selected by reconciliation and DDL |
| Accepted runtime contract | Runtime projection derived only from accepted schema for planning, decoding, admission, integrity, and recovery |
| Row layout | Versioned persisted representation owned by accepted schema |
| Application preparation | Optional Rust validation/normalization outside database authority |
| Typed row projection | Complete decoded application value; not insert or patch intent |
| Typed write input | Operation-specific authored intent that preserves omission, `DEFAULT`, `NULL`, and value provenance |

The intended authority graph is:

~~~text
ic-model repository                              IcyDB repository
-------------------                              ----------------
current-style declarations
        |
        v
host-only authoring schema
    |---> Rust types, impls, descriptors
    |---> canister/store build output
    |
    `---> entity/type SchemaFragment --\
                                        | select complete database closure
other fragments / removals ------------/
                                        `---> SchemaProposal ---\
SQL DDL / direct schema API --------------------> reconciliation
                                                     |
                                                     v
                                               accepted schema
                                                     |
                                                     v
                                          accepted runtime contracts
                                                     |
                                                     v
                                          planning / admission / storage
~~~

The authored definition is the developer source, but it is not runtime
database authority. Generated Rust types do not reconstruct accepted schema.
The schema proposal is admitted input, not trusted state. Accepted runtime
contracts do not reopen the proposal to fill missing facts. SQL DDL,
programmatic schema proposals, and `ic-model` proposals converge at the same
catalog-native reconciliation boundary.

## Product And Package Boundary

There are two products in separate repositories.

### IcyDB Repository

> IcyDB is a standalone database for Internet Computer canisters. It accepts a
> public schema proposal or SQL DDL, reconciles that input into an accepted
> schema, and executes SQL or structural operations without requiring generated
> Rust entity types.

Package ownership:

| Package | Responsibility |
| --- | --- |
| `icydb-schema` | Public, bounded, versioned schema-proposal contract, canonical scalar atoms, and builders; no accepted runtime or storage implementation |
| `icydb-core` | Reconciliation, accepted catalog, row layouts, storage, planning, execution, integrity, and recovery |
| `icydb` | Public database, SQL, dynamic value, structural mutation, and optional typed-adapter surfaces |
| `icydb-build` / `icydb-config` | Canister integration, database instances, stable-memory placement, endpoint policy, and deployment configuration |

The current `icydb-schema` name is reclaimed for this public proposal contract
as soon as its current authoring AST moves to the temporary unpublished
`icydb-model-legacy` extraction package. The complete current host-side
authoring graph—including canister, store, entity, record, type, validator,
and sanitizer nodes—later moves to `ic-model`; old and new meanings never
coexist in `icydb-schema`.

Canister/store placement and stable-memory IDs remain IcyDB deployment
semantics, but their authored nodes stay in the same `ic-model` in-memory graph
as entities. That graph is the single build-time composition source. Those
deployment nodes do not enter an IcyDB `SchemaProposal` or become accepted
database schema.

### IC Model Repository

> `ic-model` is an Internet Computer model-authoring system. Annotated
> declarations preserve the current type-generating macro experience and
> assemble one host-only in-memory authoring schema containing canisters,
> stores, entities, records, enums, collections, validators, and normalizers.
> That graph produces Rust application types, canister/store build output, and
> reusable IcyDB `SchemaFragment` values that are composed over a complete
> database-scoped closure.

Public package ownership:

| Package | Responsibility |
| --- | --- |
| `ic-model` | Public facade, current-style host-only authoring schema and registry, graph validation, reusable base model/type library, application validation/normalization traits, database-closure proposal-fragment lowering, build-time code generation, and macro re-exports |
| `ic-model-macros` | One published proc-macro package based on the current `icydb-schema-derive` compiler: `canister`, `store`, `entity`, `record`, `enum_`, `list`, `map`, `newtype`, `set`, `tuple`, `validator`, and `normalizer`, plus internal helper derives |

The external repository exposes exactly these two public packages. A published
proc-macro package cannot rely on an unpublished normal path dependency:
registry publication would require that dependency to be published too. Using
one proc-macro package therefore keeps the current parser, node validation,
type generation, and schema-node emission together without a dependency cycle
or a fourth public package. If implementation experience later requires a
separate code-generation package, it must be a fourth published package with
an exact-version dependency; that is not the 0.213 design.

### Required Dependency Direction

~~~text
schema/model crate ----------> ic-model
                                  |   |
                                  |   `-------> ic-model-macros
                                  |                 |
                                  |                 `-> icydb-schema
                                  |
                                  `-----------> icydb-schema

consumer build.rs -----------> schema/model crate
       |
       `---------------------> ic-model build entrypoint
                                  |
                                  `--emits consumer code references-->
                                             public icydb build/runtime paths

runtime application ---------> schema/model crate
       |
       `---------------------> icydb
                                  |   |
                                  |   `-------> icydb-schema
                                  |
                                  `-----------> icydb-core
                                                   |
                                                   `-> icydb-schema

ic-model generated output --emits consumer-side references--> public icydb paths

forbidden:
icydb or icydb-core ----------> ic-model
ic-model ---------------------> icydb-core
ic-model ---------------------> icydb
~~~

`ic-model` depends on IcyDB's small public schema contract. It does not depend
on the `icydb` runtime facade or engine internals. It re-exports the proposal
vocabulary through `ic_model::schema` and canonical scalar atoms through
`ic_model::types`, so ordinary generated code does not require a direct
`icydb-schema` dependency either. `icydb-schema` is the only IcyDB package in
the `ic-model` dependency closure. These namespaces are exact re-exports of
contract-owned types, not `ic-model` mirrors or translations.

A schema/model crate depends on `ic-model` and compiles its declarations into
the same host-only graph when used as a build dependency. Loading it in the
consumer build script populates that graph; the `ic-model` build entrypoint
validates it and emits actor/integration code against public IcyDB paths. The
runtime application therefore depends directly on both the schema/model crate
and IcyDB. Generated tokens may refer to the consumer's public `icydb` path
without creating a Cargo dependency from `ic-model` or `ic-model-macros` to
`icydb`.

IcyDB builds, tests, and operates with no `ic-model` package present. Direct
IcyDB users can construct the same public proposal contract without the model
compiler.

The public proposal contract must contain owned, bounded, fallible data. It
must not expose engine-private DTOs, accepted field IDs, row-layout slots,
persisted codecs, generated function pointers, Rust callback paths, or
`'static` references. Proposal-format compatibility must be explicit and
typed across repository versions.

### Ownership Matrix

| Concern | Owner |
| --- | --- |
| Current-style `canister`, `store`, `entity`, `record`, enum, newtype, collection, validator, and normalizer declarations | `ic-model` |
| Host-only process-global authoring schema, constructor registration, sealing/digest, graph validation, and graph inspection | `ic-model` |
| Rust records, enums, newtypes, lists, maps, sets, tuples, helper derives, visitors, and model documentation | `ic-model` |
| Current `icydb::base::types` reusable domain wrappers and records, except retired audit-timestamp wrappers | `ic-model::base::types` |
| Current `icydb::base::validator` and application-safe sanitizer behavior | `ic-model::base::validator` and `ic-model::base::normalizer` after classification |
| Dynamic `ModelDescriptor` and application field traversal | `ic-model` |
| Explicit application validation and normalization | `ic-model` |
| Rust construction defaults | `ic-model` |
| Reusable fragment lowering and complete multi-store database-closure selection from the sealed authoring graph | `ic-model` |
| Logical scalar kinds, canonical scalar atoms, bounded proposal literals, immutable source keys, source-level constraint expressions, fragments, opaque target database/store identities, removals, and proposal envelopes | `icydb-schema` |
| IcyDB entities, fields, primary keys, indexes, relations, accepted defaults, generated/managed policies, and representable constraints | `icydb-schema` proposal vocabulary, authored through `ic-model` or directly |
| Accepted IDs, canonical compiled constraint programs, fingerprints, evaluators, catalogs, scalar generation and database operations, row layouts, persisted codecs, indexes, journals, integrity, and recovery | `icydb-core` |
| Runtime `InputValue` / `OutputValue`, database APIs, opaque accepted bindings, `TypedRowAdapter`, `TypedWriteAdapter`, and `WriteCell` | `icydb` |
| Domain-specific application wrappers such as URL, phone, money, ICRC records, or title-cased values | `ic-model` or user code |
| `created_at` / `updated_at` field lifecycle | Accepted IcyDB write-management policy; not a wrapper type, callback, validator, or normalizer |
| Proposal validation and reconciliation | IcyDB |
| Authored canister/store placement and stable-memory declarations in the unified graph | `ic-model` |
| Meaning and enforcement of database instances, stores, canister endpoints, stable-memory IDs, and allocation policy | `icydb-build` / `icydb-config` / IcyDB |
| Typed conversion between application types and public IcyDB values | Explicitly selected generated implementations of IcyDB-owned adapter traits in the consumer crate |

`ic_model::types::Timestamp` and `icydb::types::Timestamp`, for example, are
re-exports of the same canonical scalar atom rather than parallel types.
Application wrappers may contain these atoms, but they do not redefine scalar
or database semantics. `icydb-core` supplies storage codecs, query coercion,
generation, clock access, and accepted-schema behavior around the atoms.
Conversely, IcyDB does not depend on application wrappers to decode or enforce
an accepted field.

### Layer-Qualified Entity And Record Names

`entity` and `record` are legitimate shared domain words. The design does not
invent unrelated vocabulary or rename the current compiler merely to hide
their correspondence. Concrete public Rust paths must nevertheless disclose
their layer and authority rather than making every package export an
unqualified `Entity`, `Record`, or `EntityModel`.

The normative naming families are:

| Layer | Entity concept | Record concept |
| --- | --- | --- |
| `ic-model` authoring graph | `ic_model::node::Entity` | `ic_model::node::Record` |
| `ic-model` generated inspection | `EntityDescriptor` | `RecordDescriptor` |
| `icydb-schema` proposal contract | `EntityFragment` / `EntitySourceKey` | `RecordTypeFragment` / `TypeSourceKey` |
| `icydb-core` accepted catalog | `AcceptedEntity` or `EntityCatalogEntry` | `AcceptedRecordType` |
| `icydb` public runtime | `EntityName`, `EntityBinding`, or `TypedEntityBinding` | `InputValue::Record` / `OutputValue::Record` |
| Generated application code | domain names such as `User` | domain names such as `Address` |

Exact secondary names remain API-reviewable, but the authority suffixes are
not. In particular:

- the ergonomic authoring syntax remains `#[entity]` and `#[record]`;
- the current module-qualified compiler node names can remain
  `ic_model::node::Entity` and `ic_model::node::Record`;
- the old concrete `EntityModel` name hard-cuts to the appropriately scoped
  descriptor, fragment, or accepted type rather than moving intact;
- no `icydb::Entity` trait remains necessary for SQL, dynamic, or structural
  database operations;
- proposal types use `Fragment`, `Proposal`, or `SourceKey`;
- accepted and runtime types use `Accepted`, `Catalog`, `Binding`, `Input`, or
  `Output`; and
- shared enum-variant words such as `InputValue::Record` are acceptable
  because their owning type already identifies the layer.

Glob imports must not be required to use these APIs. Examples and diagnostics
use qualified names where two layers appear together.

### Preserved Current Authoring Surface

0.213 preserves the current `icydb-schema-derive` type-generating authoring
style as the primary `ic-model` API. It does not require users to rewrite the
schema as ordinary structs plus a new derive system.

The proposed spelling is the current spelling with crate-path and immutable
source-key adjustments:

~~~rust
use ic_model::prelude::*;

#[canister(
    memory_namespace = "app",
    memory_min = 100,
    memory_max = 104,
    commit_memory_id = 104
)]
pub struct AppCanister {}

#[store(
    ident = "APP_STORE",
    store_name = "main",
    canister = "AppCanister",
    storage(journaled(
        data_memory_id = 100,
        index_memory_id = 101,
        schema_memory_id = 102,
        journal_memory_id = 103,
    ))
)]
pub struct AppStore {}

#[entity(
    source_key = "app.user",
    store = "AppStore",
    version = 1,
    name = "users",
    pk(fields = ["id"]),
    audit_timestamps(
        created_at(
            source_key = "app.user.created_at",
            ident = "created_at"
        ),
        updated_at(
            source_key = "app.user.updated_at",
            ident = "updated_at"
        )
    ),
    fields(
        field(
            source_key = "app.user.id",
            ident = "id",
            value(item(prim = "Nat64"))
        ),
        field(
            source_key = "app.user.email",
            ident = "email",
            value(item(prim = "Text", unbounded))
        )
    )
)]
pub struct User {}
~~~

`record`, `enum_`, `list`, `map`, `newtype`, `set`, `tuple`, `validator`, and
the hard-cut `normalizer` replacement for `sanitizer` retain the same
declarative family and parser/code-generation architecture.

The existing placement and relation language is also preserved normatively:

- `#[entity(...)]` declares a persisted root and must name exactly one
  `store = "..."`; the store reference is required even when the complete
  graph contains only one store, and there is no “use the only store”
  inference;
- `#[record]`, `#[enum_]`, `#[newtype]`, and collection declarations remain
  reusable and store-independent until a persisted entity field makes them
  reachable;
- existing scalar relations continue to be authored on the field item with
  `rel = "Target"` alongside the explicit primitive shape;
- existing composite relation declarations remain nested `relation(...)`
  items inside `#[entity(...)]` and continue to name their target with
  `rel = "Target"`; and
- 0.213 introduces no standalone `#[relation]` attribute macro or second
  relation declaration system.

Every field item containing `rel = "..."`, and every existing nested composite
relation declaration, contributes a relation edge to the one complete host
graph. Target paths are unresolved references while the graph is collecting;
they resolve only after all declarations have registered and the graph seals.
The current relation sanity checks remain a compatibility floor, including
target existence and kind, local-field existence, component arity and order,
primitive shape and bounds, cardinality consistency, generated-component
rejection, admissible target-primary-key components, and source/target storage
capabilities. Extraction may improve diagnostics, but it may not skip or
weaken any existing relation check.

The `audit_timestamps(...)` spelling is proposed and remains API-reviewable.
Its explicitness is normative: omitting it means the entity has no managed
audit timestamp fields. The current compiler behavior that silently appends
`created_at` and `updated_at` to every entity is hard-cut. Direct
`icydb-schema` authors can express the same two accepted field-management
policies without using `ic-model`.

### One In-Memory Authoring Schema

The current compiler's registration mechanism is intentionally preserved:

1. each macro parses and validates its local declaration;
2. it emits the generated Rust type and application implementations;
3. it emits one constant authoring-schema node; and
4. on non-Wasm targets, a constructor inserts that node into one
   process-global `ic-model` authoring schema.

The schema/model crate remains both a normal dependency and a build dependency
of a generated canister. Loading it in `build.rs` runs the host constructors.
`ic_model::build::get_schema()` seals and validates the complete graph once
before lowering or actor generation. Constructor registration remains
disabled on Wasm, so this build graph is not shipped as runtime database
authority.

Canister, store, entity, relation, and type nodes must remain in this one graph:

- each persisted entity explicitly names exactly one store;
- each store names its canister and storage/memory configuration;
- relation validation can inspect source and target store capabilities;
- canister validation can detect cross-store memory collisions; and
- build-time actor generation and database-scoped proposals consume the same
  validated declarations.

Splitting `canister` and `store` into a separate macro/registry would create
two configuration authorities and is rejected.

One IcyDB database can span multiple stores. In the current model-driven
canister, the database scope is the one IcyDB database instance and store
registry generated for a canister declaration; a store is a physical,
durability, and capability partition inside that database, not a synonym for
the database. If a future canister owns multiple database instances, their
store membership must be declared explicitly before this selection rule can be
extended; store count or declaration order is never used to infer it.

Proposal selection therefore gathers the complete source/target/type closure
across every store in the selected database. A database-enforced
`rel = "..."` may cross stores within that database when the current
source/target capability checks admit it. A target outside the selected
database, a missing target, or an inadmissible cross-store capability pairing
rejects graph validation or composition. A relation is never dropped during
fragment extraction, converted into application-only behavior, or emitted as
an unenforceable external reference.

The one graph has three outputs:

~~~text
host-only ic-model authoring schema
    |---> generated Rust application types and behavior
    |---> canister/store actor and deployment build output
    `---> reusable entity/type IcyDB SchemaFragment values
              |
              `---> complete database-scoped SchemaProposal
~~~

Canister/store nodes participate in authoring validation and build output.
They are not copied into `SchemaFragment`, `SchemaProposal`, accepted schema,
or runtime row contracts. Database-closure lowering includes only logical
database facts representable by `icydb-schema`. Store membership selects and
validates the closure and maps each persisted entity to an opaque
target-database store identity in the proposal envelope; it does not serialize
the store declaration, stable-memory placement, or durability configuration.

The process-global graph has an explicit lifecycle:

~~~text
Collecting --seal and validate--> Sealed(
    Valid(immutable snapshot, semantic digest)
    | Invalid(immutable deterministic errors, input digest)
)
~~~

Registration while `Collecting` records declarations only. Sealing
canonicalizes by immutable declaration identity, resolves all references,
runs complete-graph validation, and terminally stores either one immutable
snapshot or one deterministic error set. A failed validation does not reopen
collection. After sealing:

- late registration fails with a typed error and cannot mutate the sealed
  result;
- every call to `get_schema()` and every generator receives the same immutable
  snapshot/digest or the same cached error set/input digest;
- generators receive that snapshot explicitly and do not re-read mutable
  global state;
- constructor arrival order, hash-map iteration, wall-clock metadata, and
  generator invocation order cannot change output, digest, or diagnostics; and
- duplicate declaration identity is a deterministic error even when payloads
  match; registration never silently overwrites a prior node.

Unit tests that need different graphs use explicit local
`AuthoringSchemaBuilder`/session instances. Tests of constructor registration
or the real process-global graph run in isolated fixture processes; the global
registry has no parallel-test reset API.

Before sealing, model-generation tooling receives or derives the consumer's
resolved Cargo host normal/build dependency graph and requires exactly one
package identity named `ic-model`. Multiple linked versions or sources fail
with a typed build diagnostic before graph output is accepted. This check is
deliberately outside the registry: two versions would otherwise own two
valid-looking process globals and either could expose an incomplete graph.
Whether build orchestration supplies precomputed Cargo metadata or the
entrypoint obtains an equivalent resolver view remains API-reviewable; the
non-optional single-version failure rule is normative.

### Validator Declaration Projection

One `#[validator]` declaration has two generated outputs:

1. a host metadata node containing immutable declaration identity, target and
   use-site references, constructor arguments, signature facts, and authored
   ordering for whole-graph target, duplicate, signature, and ordering
   validation; and
2. ordinary application Rust dispatch compiled into the generated model,
   which directly constructs/calls the validator from `Model::validate()` on
   native or Wasm targets without consulting the host registry.

The host node and reference edges disappear with the rest of the authoring
graph on Wasm; the statically generated Rust dispatch remains. A validator
callback, callback pointer, constructor registry entry, or executable wrapper
never enters `SchemaFragment`, `SchemaProposal`, accepted schema, an IcyDB
runtime registry, or typed-adapter conversion. `Model::validate()` invokes the
application dispatch only when the application explicitly asks it to.
Normalizers follow the same host-metadata/generated-application-dispatch
split, while remaining a separately invoked application transformation.

The build flow preserves the current registration model:

~~~text
consumer build.rs loads schema/model crate
    -> host constructors populate the ic-model graph
    -> ic-model validates the complete graph once
    -> graph-specific generation emits actor/integration code
    -> emitted code targets the consumer's public IcyDB build/runtime paths
~~~

This replaces the current graph-reading portion of
`icydb_build::generate_with_options()`; it does not create a second IcyDB
configuration. Exact function names and option types remain API-reviewable.
IcyDB can retain standalone build/config primitives, but the model-driven
canister uses the `canister` and `store` nodes from this one graph.

Generated IcyDB adapter code remains separable from the authoring graph.
Entity/record/type declarations alone can compile through `ic-model` without
the `icydb` runtime package. When the current entity macro is asked to emit
typed runtime adapters, that selection remains an option on `#[entity(...)]`
rather than a replacement derive API. The exact option spelling is
API-reviewable. The consuming schema/application crate then depends directly
on both `ic-model` and `icydb`.

### Base Model Library And Canonical Scalars

The current `icydb::base` surface is primarily a first-party model library,
not database runtime. Its 44 macro-authored newtypes, records, enum, and map
move with the compiler into `ic-model`:

~~~text
icydb::base::types       -> ic_model::base::types
icydb::base::validator   -> ic_model::base::validator
icydb::base::sanitizer   -> ic_model::base::normalizer
~~~

This includes reusable application types such as money, percentages, ranges,
country and language codes, URLs, colours, hashes, geographic strings, memo
values, and ICRC records. They continue to use the current `newtype`,
`record`, `enum_`, and collection declarations and register their nodes in the
one host authoring graph. Database-closure proposal lowering includes only the
shared types reachable from entities in the selected database; merely linking
the base library does not add every base type to every proposal.

The move is semantic, not a source copy. It does not introduce the
schema-native domain system deferred by 0.211. Reusable type declarations and
each consuming persisted field lower as follows:

- type-shape facts, such as exact record/enum/newtype/collection structure,
  lower into the reusable type fragment;
- field-kind facts, such as the concrete scalar kind, scale, and bounded
  length/byte shape required to encode the use, lower into each consuming
  field definition;
- a representable value restriction, such as non-negativity or an accepted
  expression-language length bound, is instantiated as an ordinary 0.211
  field/entity constraint for every consuming persisted field;
- each instantiated constraint has a deterministic `ConstraintSourceKey`
  derived from the consuming `FieldSourceKey` and the immutable base-rule
  source key, so two uses of one base newtype remain distinct accepted
  constraints; and
- a rule outside the accepted expression language remains an explicit
  application validator and makes no database-wide enforcement claim.

Consequently, the reusable type fragment does not carry a shared accepted
domain object or executable restriction. Constraint instantiation substitutes
the consuming accepted field reference into the source expression, and the
ordinary 0.211 compiler/evaluator remains the only database evaluator. The
compiler generates neither a callback evaluator nor a local Rust mirror of
the accepted constraint. Direct `icydb-schema` authors can produce the same
per-use field definitions and constraints without linking `ic-model`.

After that lowering:

- genuinely application-only checks remain explicit `ic-model` validators;
- application transformations move under `normalizer` and never run
  implicitly in IcyDB;
- `CreatedAt` and `UpdatedAt` wrapper types and their clock-reading sanitizers
  are retired rather than transferred;
- the ICRC-3 model enum and map move, but its handwritten
  `RuntimeValueEncode` and `PersistedStructuralValueCodec` implementations do
  not; typed conversion uses the public adapter lane and persisted structural
  encoding remains accepted-schema-driven inside IcyDB; and
- no base type owns an accepted slot, persisted codec, database clock, value
  generator, or runtime callback.

The compiler also cannot continue expanding primitives to
`::icydb::types::*`, because that would make `ic-model` source depend on the
IcyDB runtime. `icydb-schema` therefore owns the engine-neutral scalar atoms
needed by both proposal literals and generated application types. Both
products re-export the same Rust types:

~~~text
icydb_schema::scalar::Timestamp
        | exact re-export             | exact re-export
        v                             v
ic_model::types::Timestamp       icydb::types::Timestamp
~~~

The exact internal module spelling is API-reviewable. One Rust type identity,
one canonical parse/format/value representation, and the absence of an
`ic-model -> icydb` dependency are normative. Standard primitives remain
standard Rust types.

The scalar atom API budget is deliberately small. The leaf may own canonical
in-memory representation, equality/hash/order facts intrinsic to that
representation, bounded fallible construction, canonical bounded parsing and
formatting, and bounded transport traits needed by proposal values. All input
limits are part of the fallible contract; it does not add unbounded
convenience parsing.

The scalar atom owner does not provide clocks or randomness, database
defaults or generation, SQL coercion or comparison policy, storage codecs,
row decoding, schema-dependent validation, application normalization, or
persistence behavior. Those are IcyDB runtime or application
responsibilities. Any current `icydb-core::types` surface must be classified
accordingly: only canonical representation and bounded protocol operations
move to the leaf; engine operations remain in `icydb-core`. The existing
`icydb-primitives` capability table is folded into this same leaf rather than
becoming another public dependency in the `ic-model` closure.

The IcyDB-only experience does not mention `User`:

~~~rust
let receipt = database.apply_schema(proposal)?;
database.execute_sql("SELECT * FROM users")?;
database.mutate("users", key, patch)?;
~~~

The names are proposed API spelling. Their ownership and authority boundaries
are normative even if API review later refines the exact method names.

One split is already non-negotiable:

~~~text
Rust construction default  -> Rust model projection
IcyDB insert default        -> schema proposal and accepted schema
historical fill             -> accepted row-layout authority
~~~

Those concepts must never collapse back into one ambiguous `default`.

## Decision Summary

The separate repository is part of the architecture, not merely release
organization. It makes IcyDB's standalone contract testable and makes the
dependency direction visible to Cargo.

The system remains unified in one host-only `ic-model` authoring schema. That
graph has three outputs: Rust application types and behavior, canister/store
build output, and reusable fragments selected over one complete database
closure. It is separated at every IcyDB authority boundary.

In particular:

- reserve `model` for authored and generated application concepts, while
  retaining the qualified term `ic-model authoring schema` for the existing
  host-only composition graph;
- reserve unqualified IcyDB `schema` types for proposal, reconciliation,
  accepted authority, layout, and runtime contracts;
- retain shared domain words such as entity and record while requiring every
  concrete cross-layer type to disclose descriptor, fragment, accepted,
  catalog, binding, input, or output authority;
- make IcyDB's SQL and structural database surfaces entity-name/schema driven,
  without an `E: Entity` requirement;
- keep canister/store declarations in the same `ic-model` host graph as
  entities, while IcyDB build/config owners define and enforce their
  deployment meaning;
- treat one IcyDB database as a scope that can span multiple stores; model
  composition gathers its complete relation/type closure and uses only opaque
  IcyDB-issued store routing identities;
- require explicit entity-to-store placement, preserve the existing
  `rel = "..."` relation language and checks, and infer neither an only store
  nor a standalone `#[relation]` surface;
- seal the host registry once into an immutable, order-independent
  snapshot/digest and reject late, duplicate, or multi-version registration;
- lower only logical entity/type facts into reusable `SchemaFragment` values,
  then compose the complete multi-store database closure with opaque target
  store routing; never lower canister/store deployment definitions into
  accepted database schema;
- move the current reusable `icydb::base` model/type library into
  `ic-model::base`, after classifying durable facts and application behavior;
- instantiate representable reusable value restrictions as ordinary 0.211
  constraints at each consuming field, without accepted domains or local
  evaluator mirrors;
- give canonical scalar atoms one protocol-leaf owner and re-export the same
  Rust types through both `ic_model::types` and `icydb::types`;
- keep persisted-row encoding and row-slot selection in IcyDB rather than
  generated application types;
- replace implicit audit fields and clock-reading sanitizers with explicit
  accepted `CreatedAt` / `UpdatedAt` field-management policies using one
  durable operation timestamp;
- compile application validator dispatch directly into generated Rust/Wasm
  while keeping its host metadata and callbacks out of every IcyDB contract
  and runtime registry;
- let generated typed adapters convert between application types and IcyDB
  public dynamic values without becoming runtime authority;
- emit runtime adapter code only through an explicit option on the current
  entity authoring surface;
- prove ordinary model/schema generation compiles without `icydb`;
- forbid IcyDB packages from depending on `ic-model`;
- forbid `ic-model` packages from depending on `icydb` or `icydb-core`;
- do not introduce an `icydb-ic-model` bridge product; the consuming
  application owns the explicit runtime integration;
- do not introduce a separate canister/store macro crate, registry, or
  `icydb-macros` bridge;
- prevent runtime owners from accepting proposal-only types where an accepted
  contract is required;
- replace dual-authority helper types with types whose construction proves
  whether they are proposal-only or accepted;
- authors do not duplicate a representable rule as both a Rust validator and
  an accepted constraint;
- the accepted schema must remain the authority for durable invariants;
- arbitrary contextual Rust validation must remain application-only;
- arbitrary Rust sanitization must not become hidden database semantics; and
- any future database-owned transformation requires its own bounded,
  deterministic, persisted representation.

Validation and transformation then become one important subproblem inside the
larger model/schema separation.

## Validation And Transformation Contract

### Database Constraints

An accepted constraint rejects an invalid final value or row. It is named,
persisted, deterministic, fingerprinted, introspectable, independently
evaluable, and enforced by every IcyDB write ingress.

An `ic-model` entity declaration can express a database-compatible rule once:

~~~rust
#[entity(
    source_key = "app.account",
    store = "AppStore",
    version = 1,
    name = "accounts",
    pk(fields = ["id"]),
    constraint(
        source_key = "app.account.balance_nonnegative",
        name = "balance_nonnegative",
        check = "balance >= 0"
    ),
    fields(
        field(
            source_key = "app.account.id",
            ident = "id",
            value(item(prim = "Nat64"))
        ),
        field(
            source_key = "app.account.balance",
            ident = "balance",
            value(item(prim = "Int64"))
        )
    )
)]
pub struct Account {}
~~~

The immutable source-key spelling is new and remains API-reviewable; the
type-generating declaration style is not new. The validated host graph lowers
that declaration into a public schema fragment. It does not generate a second
Rust evaluator for that constraint. Early proof against database rules uses
an IcyDB API that evaluates the live accepted contract and therefore preserves
its null behavior, coercions, identities, and diagnostics.

Authors do not repeat the same rule as an independent Rust validator. A
separate application validator is retained only when it has genuinely
application-specific semantics.

### No Database Normalizer In 0.213

0.213 does not add an accepted sanitizer, normalizer, mutation rewrite,
trigger, or transformation language. IcyDB admits authored values, applies its
existing accepted defaults/generation/management policies, constructs the
canonical after-image, and evaluates accepted constraints. It does not execute
`ic-model` Rust code.

A future database rewrite design would require a bounded persisted form,
stable and versioned semantics, exact fingerprint participation, deterministic
evaluation, uniform ingress coverage, typed diagnostics, migration rules, and
independent execution without generated Rust. That future work is outside
0.213.

### Managed Created And Updated Timestamps

`created_at` and `updated_at` are not normalizers and are not ordinary
generated columns. They are bounded accepted write-management policies
evaluated by IcyDB while it constructs the canonical after-image. They remain
uniform across typed, dynamic, SQL, batch, relation-driven, and resumable
mutation lanes without introducing a generic trigger system.

The authoring shorthand lowers into ordinary timestamp fields plus accepted
management metadata:

~~~rust
enum FieldWriteManagement {
    CreatedAt,
    UpdatedAt,
}
~~~

`CreatedAt` and `UpdatedAt` are policy variant names, not generated Rust value
types. A decoded `User` exposes canonical `Timestamp` fields; insert, patch,
and replace inputs omit them.

The enum spelling may remain as today. Its accepted semantics are normative:

- each entity can have at most one field with each policy;
- a managed audit field is a required, non-null, one-cardinality canonical
  `Timestamp` field;
- it cannot simultaneously have an authored default, insert generator,
  application normalizer, or another management policy;
- SQL/display names remain editable and have no semantic significance;
  accepted field identity and immutable source key carry the policy;
- ordinary insert, patch, update, and replace inputs cannot author, null, or
  reset the field, and generated operation types expose it only on decoded
  rows;
- direct SQL or structural attempts to target it fail with a typed
  managed-field-authorship diagnostic; and
- recovery and restore preserve stored values and never call the clock.

IcyDB captures one canonical operation timestamp at the logical operation
boundary. “Logical” is intentional: the boundary also covers non-atomic
batches and resumable jobs, so it is not defined by one commit window.

| Ingress | Logical operation timestamp |
| --- | --- |
| Single typed, structural, or SQL mutation statement | Capture once at statement/request admission |
| Explicit atomic transaction | Capture once at transaction admission and use the final net effects |
| Atomic batch | Capture once at batch admission |
| Explicit non-atomic batch | Capture once at request admission and reuse it for each independently committed item and any committed prefix |
| Resumable or bulk mutation job | Capture once, durably bind it to the submission/job identity before the first chunk can commit, and reuse it for every chunk |
| Commit retry or recovery replay | Read the timestamp already stored in prepared commit/job state; never resample |

A retry that resumes the same durable submission identity reuses its timestamp.
A new non-idempotent request is a new logical operation and may capture a new
timestamp. Every row affected within one listed operation observes the
specified timestamp even when its commits are independent. No generated type,
scalar method, visitor, or callback calls `now()` during admission.

The row behavior is:

| Operation | `CreatedAt` | `UpdatedAt` |
| --- | --- | --- |
| Insert | Set to the operation timestamp | Set to the same operation timestamp |
| Patch/update with a logical row change | Preserve | Set to the operation timestamp |
| Replace with a logical row change | Preserve | Set as for patch/update |
| Semantic no-op | Preserve | Preserve; the timestamp alone must not manufacture a row change |
| Schema activation, reindex, compaction, integrity scan, or recovery replay | Preserve | Preserve |

“Logical row change” is decided from the accepted candidate after-image before
refreshing `UpdatedAt` and excludes storage-only rewrites. For an atomic
transaction or batch that touches the same row more than once, the comparison
is between the accepted-before row at the operation boundary and the final net
after-image after all authored changes; an intermediate change that is later
restored does not refresh `UpdatedAt`. A non-atomic batch has no cross-item
netting: each independently committed item compares against the row accepted
before that item, although every item shares the request timestamp.

0.211 relations enforce immediate `RESTRICT` integrity and do not cascade
rewrites into related rows. Updating a source row's relation fields is an
ordinary row update under the rules above; relation validation itself never
refreshes an unrelated row's audit fields. A bulk or resumable operation
therefore records one meaningful logical operation time rather than a
different time per chunk. The IC clock is monotonic but not strictly
monotonic, so `UpdatedAt` is non-decreasing and consecutive logical changes
may share one value. Callers needing a concurrency token or total order use an
accepted row/schema revision rather than wall-clock time.

If both policies exist, integrity requires the value of the `CreatedAt`-managed
field to be no later than the value of the `UpdatedAt`-managed field,
regardless of their editable names. Accepted-schema validation enforces the
field shape and policy cardinality; normal write admission enforces the
lifecycle. If the captured operation timestamp is earlier than an existing
audit value, admission fails with a typed clock/integrity error rather than
clamping one row to a different timestamp or moving time backward. A future
privileged import API may define explicit preservation rules, but ordinary DML
has no override escape hatch.

Adding management metadata to an existing canonical `Timestamp` field does
not rewrite rows and requires that every admitted row contract already
provides a canonical logical value through its physical slot or an existing
frozen 0.209 historical fill. Adding a new managed field to a non-empty
accepted entity is an ordinary 0.209 additive row-layout transition: because
the field is required, the proposal must provide a bounded canonical
`Timestamp` slot payload that 0.209 can freeze as
`SchemaHistoricalFill::SlotPayload`, and that payload must pass the candidate
accepted constraints. The management policy governs only future DML. If the
required history cannot be represented by the existing 0.209 historical-fill
contract, reconciliation rejects with a typed unsupported/missing-fill error.
0.213 adds no row-dependent fill plan, migration callback, or new backfill
language, and it never assumes the activation clock. An empty entity needs no
historical fill.

Identity-preserving recovery in the same canister preserves stored timestamps
and does not consult the clock. Cross-canister snapshot restore or logical
import cannot assume comparable canister clocks: it preserves accepted audit
values and never rebases them. If the target canister's later operation
timestamp is earlier than a restored `UpdatedAt` or violates
`CreatedAt <= UpdatedAt`, the first DML that would move time backward fails
with the same typed clock/integrity diagnostic until the target clock catches
up. 0.213 provides no clamping or timestamp-translation escape hatch; this
operational consequence is part of restore testing and documentation.

Existing declarations migrate deliberately:

- the compiler stops silently appending audit fields;
- declarations that want the current behavior add the explicit
  `audit_timestamps(...)` option with immutable field source keys;
- accepted existing timestamp values and their scalar encoding are preserved;
- `base::types::time::{CreatedAt, UpdatedAt}` and
  `base::sanitizer::time::{CreatedAt, UpdatedAt}` are hard-cut; and
- malformed historical audit metadata fails migration with a typed error
  rather than synthesizing timestamps.

This follows the useful database practice of keeping one stable time value for
a logical operation and making current-time mutation database-owned. It does
not model the fields as immutable generated expressions: current time is
operation context, and the bounded accepted management policy is sufficient
without exposing generic triggers.

### Application Validation And Normalization

Application validation and normalization belong to `ic-model`. Macro-generated
types expose explicit application operations. With the runtime adapter
explicitly selected, an operation-specific write can be prepared as:

~~~rust
let authored = UserInsert::new()
    .email(WriteCell::Value(email));
let authored = authored.normalize()?;
authored.validate()?;
database.insert_typed(&binding, authored)?;
~~~

These operations may provide richer errors, inspect request context through
explicit arguments, or apply transformations before a database call. They are
not persisted, fingerprinted, used by IcyDB integrity checking, invoked by
SQL, assumed by recovery, or registered as database runtime hooks.

The current `sanitizer` vocabulary hard-cuts to `normalizer` in `ic-model`.
The current automatic callback invocation from IcyDB save preflight is
removed. Typed adapters perform conversion only; they do not implicitly
normalize or validate unless a separately named convenience API explicitly
composes those application steps.

`Model::validate()` executes application validators only. Proposal builders
validate proposal structure only. Neither locally interprets accepted
constraints. The only early database-rule validation API evaluates the current
accepted schema through IcyDB and never substitutes for write admission.

## Classification Of Existing IcyDB-Style Rules

0.213 inventories every built-in validator and sanitizer and applies this
ownership classification:

| Existing behavior | Likely durable owner | Notes |
| --- | --- | --- |
| Numeric minimum/maximum/range | Accepted constraint | Exact comparisons are directly representable |
| Text/blob/collection length | Accepted constraint | Requires frozen `CHAR_LENGTH`, octet-length, or cardinality semantics |
| Enum membership | Accepted type or constraint | Prefer the exact accepted enum catalog |
| Decimal scale | Accepted field kind where possible | Rejecting excess scale may be safer than silently rounding |
| Numeric multiple-of | Accepted constraint | Exact integer/decimal semantics only |
| Not-null | Accepted structural constraint | Must remain separate from SQL `CHECK` unknown semantics |
| Trim or fixed case conversion | `ic-model` application normalizer | IcyDB gains no rewrite language in 0.213 |
| Managed timestamps | Accepted write policy | Explicit fields, one durable operation timestamp, no wrapper or clock-reading callback |
| URL, MIME, phone, ISO, hash format | `ic-model` application validator unless already exactly representable | Promotion needs a separate frozen database constraint contract |
| Locale-sensitive title casing | `ic-model` application normalizer | Version and locale behavior make durable semantics unsafe |
| Decimal or money rounding | Explicit application conversion | Lossy silent database mutation is excluded |
| Caller authorization or external lookup | Application validator or IcyDB access policy | Not a row-local deterministic constraint |
| Arbitrary custom Rust callback | `ic-model` application-only | Never persisted by path or inferred as accepted semantics |

## Runtime And Adapter Flow

The standalone IcyDB write pipeline is:

~~~text
resolve accepted schema and authored intent
    -> capture or restore the durable operation timestamp
    -> resolve defaults, generation, CreatedAt, and preserved values
    -> construct the logical candidate excluding UpdatedAt refresh
    -> detect semantic no-op versus logical row change
    -> apply UpdatedAt only for insert or logical row change
    -> construct the canonical after-image
    -> evaluate accepted checks
    -> validate structural and cross-row constraints
    -> stage and publish the commit
~~~

Standalone SQL and structural callers enter the database without an
application model:

~~~text
SQL or dynamic values
    -> entity/table resolution through accepted schema
    -> accepted database pipeline
~~~

Explicitly selected `ic-model` typed-adapter preparation sits outside that
authority:

~~~text
request
    -> generated operation-specific insert / patch / replace intent
    -> optional ic-model application normalization
    -> optional ic-model application validation
    -> generated implementation of an IcyDB-owned adapter trait
    -> opaque accepted-schema binding
    -> IcyDB database mutation ingress
    -> accepted database pipeline
~~~

The high-level adapter can still offer both phases conveniently. It must not
generate persisted row bytes or choose accepted row-layout slots. The core
database contract remains complete when `ic-model` is not linked.

## Normative Boundary Details

### Existing Type Disposition

Every production use of the old `EntityModel`, `E::MODEL`, `SchemaInfo`,
generated model constants, proposal lowering, and accepted runtime projections
is classified before it moves:

- authored-model or macro validation;
- generated Rust convenience;
- schema proposal or reconciliation;
- accepted runtime planning or execution; or
- test-only model construction.

The final boundary is:

- the old `EntityModel` concrete type is retired rather than transferred;
- `EntityDescriptor` and `RecordDescriptor` belong to `ic-model`;
- IcyDB accepts `SchemaProposal`, never an application descriptor;
- the shared `SchemaInfo` shape splits into accepted-runtime types and
  model/proposal-only types rather than retaining dual constructors;
- accepted planning, admission, integrity, recovery, relations, and commit
  preparation accept only types constructed from accepted schema;
- `E::MODEL` is removed from IcyDB database traits and runtime dispatch;
- model-only planner helpers move to `ic-model`, test support, or consume an
  explicit public proposal contract; and
- explicitly selected generated typed adapters use public IcyDB values and
  accepted binding APIs, not engine model structs.

### Public IcyDB Schema Contract

`icydb-schema` exposes:

- canonical engine-neutral scalar atoms used by proposal literals and
  generated application fields;
- owned `SchemaFragment`, `EntityFragment`, `RecordTypeFragment`,
  `SchemaProposal`, explicit removal, and other layer-qualified proposal DTOs;
- opaque target database/store identity tokens used only to bind and route a
  composed proposal, with construction/verification owned by IcyDB;
- builders that provide ergonomic construction and local validation;
- an explicit proposal contract version;
- an explicit required-capability set;
- deterministic canonical ordering;
- bounded, fallible serialization and decoding for transported proposals; and
- typed errors for malformed, oversized, obsolete, or unsupported input.

The package is a pure leaf. It owns logical scalar kinds, canonical scalar
atoms, bounded proposal literals, immutable source-key types, the source-level
constraint expression AST, fragments, removals, composition errors, and the
proposal envelope. The atom surface owns only canonical representation,
intrinsic equality/order facts, bounded fallible
construction, bounded canonical parse/format behavior, and bounded transport
support. It does not own clocks or randomness, database defaults or
generation, SQL coercion or comparison policy, accepted or schema-dependent
validation, application normalization, storage codecs, row decoding,
persisted row representation, or unbounded convenience parsing.
The package does not depend on `icydb`, `icydb-core`, the authoring compiler,
macros, a temporary legacy AST, or another IcyDB-owned primitives package.

`icydb-core` owns accepted IDs, canonical compiled constraint programs,
accepted fingerprints, evaluators, scalar generation and database operations,
physical layouts, and persisted runtime forms. `icydb` owns runtime
`InputValue` / `OutputValue` and database APIs; those public values contain the
same canonical atoms rather than redefining them. A source-level proposal
expression never contains accepted bytecode, an accepted constraint ID, a
physical slot, or a runtime fingerprint. The proposal envelope may carry an
expected accepted head solely as an optimistic concurrency condition.

The source identity vocabulary is distinct from editable naming:

~~~rust
struct EntitySourceKey(/* bounded immutable key */);
struct FieldSourceKey(/* bounded immutable key */);
struct TypeSourceKey(/* bounded immutable key */);
struct ConstraintSourceKey(/* bounded immutable key */);
struct IndexSourceKey(/* bounded immutable key */);
struct RelationSourceKey(/* bounded immutable key */);

enum SchemaRemoval {
    Entity(EntitySourceKey),
    Field {
        entity: EntitySourceKey,
        field: FieldSourceKey,
    },
    Type(TypeSourceKey),
    Constraint(ConstraintSourceKey),
    Index(IndexSourceKey),
    Relation(RelationSourceKey),
}
~~~

SQL/display names remain editable attributes. Renaming `users` or `email`
preserves its source key and reconciles as a rename, not delete-plus-add.
Source keys are not accepted IDs: IcyDB resolves them through the accepted
catalog and continues to own accepted identity allocation. Direct proposal
authors must persist their keys with the source declaration. SQL DDL assigns a
source key when an object is first accepted and preserves it through rename.

The top-level contract contains the logical entity/type graph, immutable source
keys, editable names, fields, nullability, primary keys, indexes, relations,
accepted defaults, generated/managed write policies, representable
constraints, and proposal-envelope routing of persisted entities to opaque
IcyDB-issued store identities belonging to the target database. It does not
define stores, canisters, stable-memory IDs, durability settings, accepted
entity/field IDs, physical index ordinals, row-layout slots, persisted codecs,
journal records, runtime hooks, or callbacks.

### Proposal Composition

The validated `ic-model` authoring graph lowers each entity and shared type
into reusable, store-free `SchemaFragment` values. A fragment has
entity/type/module-local authorship scope and can contain definitions plus
references to immutable source keys. It is not accepted by
`Database::apply_schema`, and a macro-generated node does not bypass
complete-graph validation by submitting itself directly.

Composition is database-scoped, not store-scoped. For the selected canister's
one database, it gathers every persisted entity in all of that database's
stores plus the complete reachable shared-type and relation source/target
closure. Each persisted entity fragment is assigned to an opaque
IcyDB-issued target-store identity. The assignment is proposal-envelope
routing, not a store definition: the corresponding `#[store]` node remains
the single source of model-driven deployment configuration, and IcyDB verifies
that every routed store belongs to the target database.

`SchemaProposal::compose` creates the only submit-capable form:

~~~rust
let authoring_schema = ic_model::build::get_schema()?;
let assignments = authoring_schema.icydb_assignments_for_database(
    "AppCanister",
    &target_database,
)?;

let proposal = SchemaProposal::compose(
    target_database,
    submission_key,
    expected_accepted_head,
    assignments,
    explicit_removals,
)?;
~~~

The exact graph-query spelling is API-reviewable. The required behavior is
that sealing and canister/store/entity/relation validation complete first,
every store belonging to one explicitly selected database contributes its
persisted entity closure, and canister/store deployment nodes themselves are
not included in the proposal. Store count, “only store” inference, and
constructor order cannot affect selection.

The proposal carries an opaque target-database token and all included changes
are admitted against that database's one expected accepted head in one atomic
submission. Applying it to any other database fails before mutation. Reusable
schema authorship lives in fragments; a composed submission is deliberately
bound to its target. It is a delta, not an assertion that the fragments
describe every entity in the database. In 0.213:

- absence means unchanged;
- deletion never follows from absence;
- every removal is an explicit, typed operation naming an immutable source
  key;
- a fragment cannot request physical placement, accepted identity reuse, or a
  partially authoritative scope; and
- there is no implicit prune, replace-all, or module-owned deletion mode.

Composition constructs one source-key symbol table. Duplicate definition keys
are errors even when their payloads happen to match; one fragment must own
each definition. A reference is not a duplicate definition. Differing
definitions, editable-name collisions, duplicate removals, definition/removal
collisions, and incompatible capability requirements return typed composition
errors.

Relations and named type references target immutable source keys. Model-driven
composition resolves the complete relation source/target closure across all
stores in the selected database before lowering. Direct proposal composition
may reference a definition already present at the expected accepted head, but
that reference must resolve inside the same target database. Application
fails atomically if a reference is outside the target database, absent,
removed by the same proposal, ambiguous, routed to a store not owned by that
database, of the wrong kind, or incompatible with source/target store
capabilities. Cross-fragment and cross-store declaration order does not affect
the result. No relation is omitted or weakened because its target lives in a
different store.

Canonicalization sorts unordered definitions and operations by their immutable
source keys. Semantically ordered members—composite key fields, index columns,
tuple members, and explicit sort directions—retain authored order. The
canonical proposal digest therefore does not depend on fragment iteration
order.

Builders and composition validate their own structural invariants, but IcyDB
treats every proposal as untrusted and validates it again. SQL DDL and
`SchemaProposal` input call the same catalog-native mutation owners. SQL
remains a frontend and does not become an alternate source of schema
semantics.

### Application, Concurrency, And Activation

Every proposal carries an IcyDB-issued opaque target-database token, a
caller-generated immutable submission key, and a required expected accepted
head:

~~~rust
struct ExpectedDatabaseIdentity(/* opaque bounded token */);

enum ExpectedAcceptedHead {
    Empty,
    Exact {
        revision: ExpectedSchemaRevision,
        fingerprint: ExpectedSchemaFingerprint,
    },
}
~~~

The target and exact-head fields are bounded, opaque proposal transport tokens
obtained from IcyDB. `icydb-core` still owns computing and interpreting
database identities, accepted revisions, and fingerprints; proposal code
cannot derive accepted runtime semantics from the tokens.

There is no blind or cross-database application API in 0.213. A target
identity mismatch or stale revision/fingerprint fails before mutation with a
typed conflict. The canonical proposal digest, target identity, expected head,
and submission key are durably associated. Repeating the same key with the
same digest, target, and expected head returns the existing receipt or job.
Reusing that key with any different member is a typed idempotency conflict.

`Database::apply_schema` returns a durable, queryable receipt:

~~~rust
struct SchemaChangeReceipt {
    database_identity: DatabaseIdentity,
    submission_key: SchemaSubmissionKey,
    proposal_digest: SchemaProposalDigest,
    prior_head: AcceptedSchemaHead,
    outcome: SchemaChangeOutcome,
}

enum SchemaChangeOutcome {
    NoOp { accepted_head: AcceptedSchemaHead },
    Applied { accepted_head: AcceptedSchemaHead },
    Pending {
        job: SchemaChangeJob,
        candidate_head: CandidateSchemaHead,
    },
    Failed { error: SchemaChangeFailure },
}
~~~

Admission of the proposal is atomic: no subset is silently accepted if
composition, reconciliation, or durable job creation fails. `NoOp` and
`Applied` are successful terminal states. `Failed` is a durable terminal state
for work that failed after job admission. `Pending` is not
success-by-assumption; the durable job and receipt can be queried by submission
key, resumed, and observed through a typed terminal outcome. A failed
candidate never becomes the accepted head; its prior accepted head remains
runtime authority.

Reconciliation must use the activation machinery established by 0.211.
Creating or validating a constraint, index, or relation that cannot finish
within the bounded call produces `Pending`; `apply_schema` cannot skip scans,
mark an unvalidated object active, or implement a second activation protocol.
Visibility of the old head, candidate facts, and newly active head follows the
0.211 catalog-native state machine. The current accepted head remains the
runtime authority until that state machine says otherwise.

Supported SQL DDL lowers into the same catalog-native change intent. The
database/transaction supplies its target identity and expected head; the SQL
execution context supplies a durable statement submission key. DDL that starts
resumable activation returns the same receipt/job semantics rather than
reporting the candidate as fully active. SQL is not a second mutation or
activation implementation.

Before 1.0, IcyDB accepts the current proposal contract version or fails with a
typed version/capability error. It does not translate obsolete proposal forms.

The public proposal cannot simply expose current `EntityModel` or
`CompiledSchemaProposal`. It needs owned data and a public contract shaped for
untrusted admission rather than generated `&'static` metadata.

### IC Model Package Split

The two public packages have these responsibilities:

- `ic-model` owns the migrated authoring node graph, its non-Wasm global
  registry and complete-graph validation, application descriptors and
  validation/normalization traits, the migrated reusable base model library,
  exact schema/scalar re-exports, complete database-closure lowering into
  `icydb_schema::SchemaFragment`, build-time traversal of the graph, and the
  public macro re-exports; and
- `ic-model-macros` owns the current type-generating macro compiler:
  `canister`, `store`, `entity`, `record`, `enum_`, `list`, `map`, `newtype`,
  `set`, `tuple`, `validator`, and the hard-cut `normalizer` replacement for
  `sanitizer`, plus any private helper derives needed by those expansions.

`ic-model` always depends on the public `icydb-schema` package because producing
IcyDB fragments is part of its product contract. It never depends on `icydb`
or `icydb-core`, normally, optionally, or for package-local development tests.
`ic-model-macros` may also depend on the same pure protocol leaf for canonical
scalar kinds and proposal vocabulary; it never depends on `icydb` or
`icydb-core`. Cross-product validation belongs to a separate consumer fixture.

The current attribute/type-generating surface is the normative authoring
style. 0.213 does not require an additional `#[derive(Model)]` system or
semantic parity between two authoring languages. Each macro emits its
application type or marker and one `ic-model` authoring node. Ordinary
`ic-model` code validates the assembled graph, lowers its logical entity/type
facts into `SchemaFragment`, composes the complete database closure, and
consumes its canister/store facts for build output. Proc macros do not
construct engine-private DTOs or accepted runtime contracts. Parser and
code-generation reuse remain private modules in the one published proc-macro
package.

Ordinary macro expansion resolves the consumer's `ic-model` dependency and
uses its `schema` re-export. Entity/type expansion emits no literal or
resolved path to `icydb` or `icydb-schema`. Two explicit integration outputs
may emit consumer-side IcyDB paths: complete-graph actor generation requested
by the build script, and a runtime-adapter option on the current entity
surface. The consumer owns the direct `icydb` dependency in both cases.

Generated paths are never assumed to be literally `::ic_model` or `::icydb`.
Macro expansion resolves renamed dependencies from the consumer manifest and
supports explicit crate-path overrides where manifest-aware resolution is
unavailable or ambiguous. Compile tests cover schema-only use without
`icydb`, default names, renamed dependencies, explicit overrides, and opted-in
adapter use with both direct dependencies.

### Existing Package And Library Disposition

The extraction is a transfer and dependency cleanup, not a wholesale compiler
rewrite:

| Current owner | 0.213 disposition |
| --- | --- |
| `icydb-schema` authoring nodes, global build registry, and graph validation | Move to `ic-model`; `icydb-schema` is reclaimed as the pure proposal protocol |
| `icydb-primitives` scalar capability metadata | Fold into the reclaimed `icydb-schema` leaf so `ic-model` has one IcyDB protocol dependency rather than a second scalar package |
| Pure value-representation portions of `icydb-core::types` | Move to canonical scalar atoms in `icydb-schema`; retain clocks, generation, coercion, SQL, storage, codecs, and other engine operations in `icydb-core` |
| `icydb-schema-derive` | Move its cohesive parser and generators into `ic-model-macros`, including `canister` and `store`; replace `icydb-core` parsing/types and generated `::icydb` authority with public `ic-model`/`icydb-schema` boundaries |
| `icydb-derive` application conveniences such as arithmetic, display, inner-value, and field-projection helpers | Fold required private helper generation into `ic-model-macros`; do not preserve a third public derive package |
| `icydb-derive::PersistedRow` and any generated physical-slot/layout codec | Delete; persisted row layout and decoding remain IcyDB accepted-runtime responsibilities |
| `icydb::base::types` | Move reusable model declarations to `ic_model::base::types`; retire `CreatedAt` / `UpdatedAt` wrapper types |
| `icydb::base::validator` and `icydb::base::sanitizer` | Move application-only behavior to `ic_model::base::{validator, normalizer}` after durable-rule classification; delete clock-reading audit sanitizers and do not copy accepted-constraint mirrors |
| ICRC-3 handwritten runtime/persisted codec implementations in the base library | Delete from the model library; public typed adapters and accepted IcyDB structural codecs own conversion and persistence respectively |
| `icydb-build` traversal of the authoring graph | Move the graph-reading traversal and actor-code emission to the `ic-model` build entrypoint; emitted code targets public IcyDB build/runtime primitives, whose deployment meaning remains in IcyDB |
| IcyDB facade re-exports of the current schema macros | Hard-cut after downstream users import `ic-model` directly |

No `icydb-macros`, `ic-model-derive`, or unpublished code-generation package
is introduced by 0.213. Graph-specific generation may emit public IcyDB paths
as tokens without linking `ic-model` to IcyDB. If a remaining IcyDB-owned
build primitive must be invoked during generation, the consumer build script
is the composition point and depends on both public products directly. It
must not create a second registry or duplicate canister/store configuration.

### Local Rust Validation

`ic-model::Model::validate()` runs application validators only. Structural
proposal validation checks source declarations and composition only. Neither
generates or runs a local evaluator for accepted constraints.

IcyDB separately exposes accepted-schema validation through its public API
when callers need early proof against the live database contract. That API
accepts operation-specific structural or typed intent, resolves its candidate
after-image through the live binding, and uses the same accepted
expression/evaluator path as admission. It returns stable
accepted-constraint diagnostics but performs no write and invokes no
application callback. Final write admission remains authoritative because
accepted schema and row state can change between validation and mutation.

### Diagnostic Ownership

Explicit application preparation returns `ic-model` diagnostics before the
caller invokes IcyDB. A direct IcyDB write returns IcyDB admission diagnostics
with stable accepted constraint identity. There is no combined hidden
callback/database pipeline whose incidental ordering chooses the error.

A convenience API may collect both classes outside the write path, but it does
not replace database admission and does not change write-error precedence.

### SQL And Application Callbacks

No. The separate product boundary makes this a decision rather than an open
option. SQL and dynamic structural writes use only accepted database semantics.
IcyDB does not link, locate, or execute `ic-model` callbacks.

A caller may explicitly run `ic-model` application preparation before sending
values to IcyDB, but that is an application-side phase. SQL issued directly
against IcyDB does not acquire extra behavior merely because generated
application types happen to exist in the same canister.

### Typed Adapter And Accepted-Schema Drift

The ordinary `ic-model` projection describes the fragment produced from one
source definition and emits no runtime adapter. When the author explicitly
selects adapter generation, the live database may still have accepted
compatible DDL changes or may reject a stale proposal. That adapter therefore
cannot assume generated field order equals accepted row-layout order.

The adapter traits belong to `icydb`, not `ic-model`:

~~~rust
pub trait TypedRowAdapter {
    type Row;

    fn decode_row(
        binding: &TypedEntityBinding,
        row: OutputRow,
    ) -> Result<Self::Row, TypedAdapterError>;
}

pub trait TypedWriteAdapter {
    fn encode_write(
        self,
        binding: &TypedEntityBinding,
    ) -> Result<TypedWrite, TypedAdapterError>;
}
~~~

The exact signatures remain API-reviewable, but ownership is normative. Only
the explicit runtime-adapter option on the current entity authoring surface
implements these IcyDB-owned traits for consumer-owned generated types. The
application therefore depends directly on both `ic-model` and `icydb`; neither
product package depends on the other.

IcyDB issues an opaque `TypedEntityBinding` after resolving immutable entity
and field source keys against an accepted snapshot. The binding is tied to:

- database identity;
- accepted entity identity;
- accepted schema revision and fingerprint;
- the relevant entity generation; and
- the requested row/write projection.

Callers and generated code cannot inspect accepted field IDs, physical slots,
compiled programs, or persisted codecs through the binding. They can only ask
IcyDB to encode/decode public values through it. Using a binding with another
database, entity, revision, fingerprint, or generation returns a typed stale
or mismatched-binding error. Cached bindings are invalidated when the accepted
fingerprint or relevant generation changes.

Compatible accepted additions may be ignored by an older adapter when it
requests an explicit known-field row projection. A missing known source key,
incompatible field kind, incompatible nullability, or unavailable required
projection fails with a typed binding error. Editable renames do not break a
binding request because generated adapters use immutable source keys.

### Typed Write Intent And Provenance

A complete `User` is a decoded row projection. It is not automatically an
insert, replace, or patch input. Reusing it as write intent would collapse the
authored/defaulted/generated/managed/preserved distinctions established by
0.209.

The opted-in adapter expansion generates operation types that preserve intent
explicitly:

~~~rust
enum WriteCell<T> {
    Omitted,
    Default,
    Null,
    Value(T),
}

struct UserInsert { /* authorable insert fields */ }
struct UserPatch { /* authorable patch fields */ }
struct UserReplace { /* authorable replacement fields */ }
struct User { /* complete decoded row projection */ }
~~~

`WriteCell` is owned by `icydb`. `Default` is included in addition to
`Omitted`, `Null`, and `Value` because an explicit SQL-style `DEFAULT` request
is authored intent and must not be conflated with absence.

The lowering rules are operation-specific:

| Input | Insert | Patch | Replace |
| --- | --- | --- | --- |
| `Omitted` | no authored value; accepted insert default/generation policy may supply it | preserve the current accepted value | no authored value; apply accepted replacement omission rules, never preserve implicitly |
| `Default` | explicitly request the accepted default | explicitly reset through the accepted default policy | explicitly request the accepted default |
| `Null` | explicitly authored `NULL` | explicitly set `NULL` | explicitly authored `NULL` |
| `Value(v)` | explicitly authored value | explicitly replace with `v` | explicitly authored value |

Nullability and default availability are still accepted-schema decisions.
Generated, managed, or otherwise database-owned fields are omitted from
authorable setters/types unless the accepted public contract explicitly
permits authorship. The adapter lowers intent into IcyDB's structural mutation
form; it does not resolve defaults, generation, management, preservation, or
row slots itself.

Audit-managed fields are never such an exception: decoded rows expose their
canonical `Timestamp` values, while `UserInsert`, `UserPatch`, and
`UserReplace` expose no `WriteCell` or setter for them.

Application normalization and validation operate on the operation-specific
input before encoding. They may transform or reject authored `Value` cells but
must not turn `Omitted` into authored data, synthesize database defaults, or
claim database provenance. Database admission retains the authoritative
provenance for the final after-image.

Typed reads may return a complete `User` plus separate IcyDB-owned result
metadata when provenance is requested. The row projection itself does not
become writable by implication. A convenience “replace from row” operation
must be explicitly named and lower through `UserReplace`; there is no
ambiguous `database.insert_typed(user)` API.

All binding, conversion, and intent errors are typed. The adapter must never
reconstruct, inspect, or silently downgrade the accepted schema.

## Migration And Release Contract

Repository separation cannot be one source move. The authority boundary must
exist inside IcyDB before the code is transferred:

1. move the existing authoring AST into a temporary unpublished
   `icydb-model-legacy` package without splitting its canister/store/entity
   graph, make `icydb-schema` the pure proposal/scalar protocol leaf
   immediately, fold the current scalar capability owner into it, and route
   the new contract through the existing catalog-native reconciliation owner;
2. make IcyDB schema acceptance, SQL, dynamic reads, and structural writes
   operate without generated entity types;
3. split accepted runtime contracts from model-only helpers and remove
   generated model/callback requirements from database recovery, integrity,
   relations, and commit preparation;
4. rebase the current embedded compiler and its host-only constructor registry
   onto `ic-model` paths while preserving its type-generating declarations,
   complete graph validation, canister/store build output, and complete
   database-closure fragment lowering; primitive expansion uses the canonical
   scalar re-exports, implicit audit fields hard-cut to explicit accepted
   management, and an explicit option on the existing entity surface emits
   operation-specific write inputs and implementations of public IcyDB-owned
   adapter traits;
5. establish the external `ic-model` workspace and move the unified authoring
   graph, current macro compiler, reusable base model library, classified
   application validators/normalizers, required helper derives, build
   traversal, and application behavior as the two-package product;
6. prove one-way released-package interoperability before deleting the
   embedded packages; and
7. hard-cut old `icydb::schema`, `icydb::base`, macro re-exports, generated
   persistence implementations, audit-timestamp wrappers/sanitizers, and
   callback-coupled database paths.

This staging does not authorize dual runtime dispatch or compatibility shims.
`icydb-model-legacy` is an explicitly temporary extraction package with
`publish = false`; it must not gain new API or authority. During staging it
retains one complete host graph so canister/store and entity declarations
cannot drift. The temporarily embedded compiler must use the same new
proposal/adapter path that the external repository will use. The final cut
removes both temporary owners completely.

No existing validator or sanitizer is silently promoted during extraction.
Each declaration must still be classified as an accepted constraint, accepted
type fact, accepted write policy, `ic-model` application validator/normalizer,
or application-only behavior. Automatic promotion from a Rust callback path
would recreate the authority problem identified by 0.211.

Existing accepted `created_at` and `updated_at` identities, values, and scalar
encoding are preserved. Source declarations gain explicit audit policy and
source keys, but migration does not rewrite historical timestamps or sample
the clock. Existing base-library users hard-cut their imports from
`icydb::base` to `ic_model::base`; no IcyDB facade compatibility re-export is
left behind.

Cross-repository release order must be explicit:

1. IcyDB publishes or otherwise freezes the required public schema/scalar
   contract revision;
2. `ic-model` targets that revision and passes golden proposal plus typed
   adapter integration tests;
3. downstream fixtures switch to the external packages; and
4. IcyDB removes the embedded compiler and old re-exports.

Path or Git dependencies can support local integration but are not the
released compatibility contract.

## Risks

- A separate repository can hide rather than solve coupling if `ic-model`
  imports `icydb-core` or IcyDB retains generated-model runtime requirements.
- An oversized public schema contract can become a second engine API and make
  internal refactoring harder.
- Canonical scalar atoms can leak engine behavior into the protocol leaf if
  clocks, generation, query coercion, or persistence methods move with their
  value representation.
- Parallel `ic-model` and IcyDB scalar structs can drift in representation and
  force unnecessary conversions if exact shared type identity is not enforced.
- Cross-repository version skew can produce proposals newer than the running
  database or adapters older than the accepted schema.
- Renamed consumer dependencies can break generated paths unless manifest
  resolution and explicit overrides are tested.
- A normal model macro can accidentally make schema-only users depend on
  `icydb` if runtime adapter tokens are not strictly opt-in.
- Bare or reused `Entity`, `Record`, and `EntityModel` type names can recreate
  authority ambiguity even when their Cargo dependencies are correct.
- Reusing or changing an immutable source key can misidentify an authored
  object; tooling must make keys visible and stable without presenting them as
  editable names.
- Proposal composition can become a second reconciliation engine if it tries
  to assign accepted identities or evaluate database compatibility locally.
- Treating a store as a database can omit a cross-store relation target and
  produce a valid-looking but incomplete proposal; database closure and opaque
  store routing must be explicit.
- An unsealed or silently overwriting constructor registry can make output
  depend on link/constructor order, while multiple linked `ic-model` versions
  can create disjoint incomplete graphs.
- Lowering a base-newtype restriction as shared accepted metadata can
  accidentally introduce domains or a second evaluator instead of ordinary
  per-use 0.211 constraints.
- Durable schema jobs can report misleading success if `Pending` receipts are
  treated as fully active rather than queried to a terminal state.
- Generated adapters can accidentally retain persisted-row codecs, accepted
  slots, or layout assumptions and recreate runtime authority outside IcyDB.
- Ambiguous typed write inputs can erase omission/default/preservation
  provenance even when the structural database lane is correct.
- Sampling the clock per row, retry, or resumable chunk can give one logical
  operation inconsistent audit timestamps.
- Cross-canister restore can preserve an audit value ahead of the target
  canister's unrelated clock and cause typed write rejection until it catches
  up.
- An `UpdatedAt` policy that runs before semantic no-op detection can create
  timestamp-only writes and make every no-op appear to change the row.
- Retaining implicit audit fields or the old clock-reading sanitizers can
  recreate hidden schema and dual timestamp authority.
- Splitting canister/store declarations from entity declarations can create
  two registries, two configuration sources, and undetectable drift.
- Rewriting the current macro surface into a derive-first API can add
  migration cost without improving the authority boundary.
- Shipping the host authoring graph or constructor registry in Wasm can
  accidentally recreate generated schema as runtime authority.
- Dividing actor generation between `ic-model` and `icydb-build` without one
  explicit composition boundary can duplicate canister/store interpretation.
- A multi-repository move can leave two apparent owners unless the final
  hard-cut and release order are explicit.
- Application normalization can still conceal bad input if convenience APIs
  make the explicit preparation phase unclear.
- A new dynamic standalone surface can duplicate typed planning/execution
  unless both adapters lower into the same accepted structural owner.
- Cross-repository integration support can increase native build time and
  combined downstream Wasm size even if standalone IcyDB becomes smaller.

## Non-Goals

0.213 excludes:

- turning `ic-model` into a database-neutral multi-backend ORM; its IcyDB
  projection deliberately targets IcyDB's public contract;
- making IcyDB, `icydb-core`, or IcyDB runtime correctness depend on
  `ic-model`;
- allowing `ic-model` to depend on `icydb-core` or engine-private modules;
- allowing `ic-model` to depend on the `icydb` runtime facade;
- introducing an `icydb-ic-model` bridge package for code that can be emitted
  into the explicitly opted-in consumer;
- requiring schema-only `ic-model` consumers to depend directly on `icydb` or
  `icydb-schema`;
- keeping the current embedded model/schema packages as compatibility
  wrappers after the external cut;
- re-exporting `ic-model` from `icydb` in a way that reverses the dependency;
- splitting canister/store declarations into a separate macro crate, registry,
  or configuration from the entity/type authoring graph;
- lowering canister/store definitions, stable-memory allocation, durability
  configuration, or endpoint policy into reusable `SchemaFragment` or
  accepted schema; opaque target-store routing remains proposal-envelope
  composition metadata;
- replacing the current attribute/type-generating macros with a required
  derive-first authoring API;
- introducing `icydb-macros`, `ic-model-derive`, or a third unpublished
  code-generation package;
- introducing a third public base-model package for the current
  `icydb::base` library;
- defining duplicate application-side and database-side canonical scalar
  representations;
- shipping the host-only authoring graph or its constructor registry as Wasm
  runtime database authority;
- generating accepted row-layout slots, persisted row bytes, recovery state,
  or runtime hook registries in `ic-model`;
- renaming crates without first proving the dependency and authority split;
- changing a durable format merely to improve terminology;
- persisting Rust callback paths, function pointers, closures, or trait
  objects;
- arbitrary triggers or external calls;
- nondeterministic generic normalizers;
- locale-dependent transformations;
- automatic promotion of every validator or sanitizer;
- silent historical rewrites;
- compatibility shims for pre-1.0 callback semantics;
- expanding 0.211 to implement normalization retrospectively;
- adding an accepted normalizer, rewrite, or trigger language;
- introducing accepted shared-domain objects in 0.213; reusable value
  restrictions instantiate ordinary 0.211 constraints at each field use;
- implementing managed audit timestamps as generic triggers, callbacks,
  generated expressions, or authorable wrapper values;
- treating `updated_at` as a strict ordering or optimistic-concurrency token;
- adding implicit delete-by-absence, prune, or replace-all proposal semantics;
- exact unsigned identity generation, which belongs to 0.214; and
- SQL structural coverage, range remediation, and performance work, which
  belong to 0.215.

## Appendix A: Platform And Comparable-System Evidence

The common pattern is not to eliminate validation or transformation. It is to
make the authority and execution boundary explicit.

### Internet Computer Time

The Internet Computer exposes nanosecond system time that is monotonic but not
strictly monotonic, and time is constant within one message execution. This
supports one captured operation timestamp but not using `updated_at` as a
unique revision. Resumable work crosses message executions, so IcyDB must
persist the original logical operation timestamp rather than sample the clock
again.

Sources:

- <https://docs.internetcomputer.org/guides/backends/timers/>
- <https://docs.internetcomputer.org/guides/security/miscellaneous/>

### PostgreSQL

PostgreSQL keeps database invariants in schema-owned `CHECK`, domain, unique,
primary-key, and foreign-key constraints. Reusable scalar restrictions can be
expressed as domains. Write-time computation or transformation uses explicit
features such as generated columns and triggers rather than application
validators being treated as constraints.

PostgreSQL also documents an important durability issue: check behavior is
assumed to be immutable. Changing a user function used by a check can leave
stored rows inconsistent until the constraint is dropped and re-added. That is
strong support for IcyDB persisting canonical semantics rather than callback
paths.

For audit time, PostgreSQL distinguishes insert defaults from generated
columns, disallows volatile current-time functions in generated expressions,
allows a `BEFORE` trigger to set the current timestamp, and provides stable
transaction/statement timestamp concepts. IcyDB adopts the useful ownership
and stable-operation-time properties but uses a bounded accepted field policy
rather than adding general triggers.

Sources:

- <https://www.postgresql.org/docs/current/ddl-constraints.html>
- <https://www.postgresql.org/docs/current/sql-createdomain.html>
- <https://www.postgresql.org/docs/current/ddl-generated-columns.html>
- <https://www.postgresql.org/docs/current/sql-createtrigger.html>
- <https://www.postgresql.org/docs/current/trigger-definition.html>
- <https://www.postgresql.org/docs/current/functions-datetime.html>

### SurrealDB

SurrealDB places validation and transformation beside each other but keeps
them as separate schema clauses. `ASSERT` validates a value. `VALUE` can
transform an authored value before it is stored, including a documented
lowercase-email example. Both are database-visible schema behavior.

This demonstrates that database transformation must be an explicit
schema-owned feature. 0.213 does not import that feature into IcyDB.

Sources:

- <https://surrealdb.com/docs/reference/query-language/statements/define/field>
- <https://surrealdb.com/docs/learn/schema-management/tables-and-fields/fields-and-validation>

### Gel

Gel exposes schema-defined mutation rewrites for insert and update. Rewrites
are explicit schema objects, can distinguish authored from preserved values,
and apply to DML through its SQL adapter. They are complementary to constraints
and triggers rather than disguised application hooks. This supports deferring
rewrites to a separate future database design instead of preserving current
Rust callback execution.

Sources:

- <https://docs.geldata.com/reference/datamodel/mutation_rewrites>
- <https://docs.geldata.com/reference/using/sql_adapter>

### MongoDB And Convex

MongoDB collection validators reject invalid inserted or updated documents but
do not turn arbitrary application cleanup into database validation. Convex
offers a useful authoring precedent: its declarative validator builder is used
for both function arguments and table schemas, while a pushed schema validates
existing documents and future writes.

Convex therefore demonstrates declaration reuse without making arbitrary
application functions the database authority.

Sources:

- <https://www.mongodb.com/docs/manual/core/schema-validation/>
- <https://docs.convex.dev/database/schemas>

### Ecto, Rails, And SeaORM

Application data layers retain validation and lifecycle preparation because
they are useful. Ecto explicitly distinguishes validations, which can run
without the database and provide early feedback, from database constraints,
which provide safe integrity enforcement. Rails and SeaORM expose lifecycle
callbacks before validation or save.

These systems also demonstrate the bypass risk: an application callback is
only as universal as the application write path that invokes it. IcyDB must
not describe such a callback as database authority merely because its normal
typed save path calls it.

Sources:

- <https://hexdocs.pm/ecto/Ecto.Changeset.html>
- <https://guides.rubyonrails.org/active_record_callbacks.html>
- <https://docs.rs/sea-orm/latest/sea_orm/entity/trait.ActiveModelBehavior.html>

The combined lesson is:

- durable invariants belong to database schema;
- application preparation remains useful but must be explicit; and
- database transformations, if added later, require their own schema-owned
  contract rather than callback reuse.
