# 0.213 Standalone IcyDB And External IC Model

Status: proposed

Design cut: 2026-07-23

Implementation order: after 0.212 closeout and explicit user direction to
start 0.213; before 0.214

This document defines the proposed product, repository, dependency, schema,
runtime, validation, and migration boundary between standalone IcyDB and the
external `ic-model` authoring system.

Implementation does not begin until this proposed design is reviewed and
accepted. Generic continuation within 0.212 does not authorize 0.213, and
generic continuation within 0.213 never authorizes 0.214.

The validator/sanitizer review exposed the immediate problem, but the broader
issue is that `model` and `schema` still name overlapping authoring, generated,
accepted, and runtime concepts. That makes proposal data look authoritative
and makes application callbacks look like database semantics when shared
execution happens to invoke them.

## Problem Statement

IcyDB already states the intended database authority correctly:

- source definitions author an entity model;
- generated metadata proposes database schema;
- reconciliation accepts one durable database schema; and
- the accepted schema snapshot is runtime authority.

The type and module vocabulary does not yet make that separation inevitable.
`EntityModel` is still described as a macro-generated runtime schema snapshot,
and some of its fields are described as authoritative for runtime planning.
`SchemaInfo` can be constructed from either trusted generated metadata or an
accepted snapshot. Runtime and model-only planning helpers therefore share
types and names even where their authority is different.

The desired seam exists in part:

- `db::schema::proposal` projects trusted `EntityModel` data into a typed
  proposal;
- accepted snapshots persist field IDs, layouts, constraints, indexes,
  relations, defaults, and write policies; and
- accepted read/write contracts already forbid runtime fallback
  reconstruction from generated models.

0.213 makes that seam structural rather than conventional.

The product boundary is:

- IcyDB is a standalone database for the Internet Computer that accepts
  a schema and can be used through SQL and dynamic structural values without
  generated Rust entity types;
- `ic-model` is a separate model-authoring and code-generation project;
- one `ic-model` declaration generates an application-model projection and
  an IcyDB schema fragment for database-scoped proposal composition; and
- IcyDB must not depend on `ic-model` or execute its application callbacks.

The current public surface does not yet meet that goal. Structural mutation and
SQL entrypoints still commonly require an entity type parameter, while
`EntityRuntimeHooks` combines generated model metadata with executable
callbacks. 0.213 removes that requirement from database authority rather
than merely rename the generated types.

The 0.211 constraint work establishes an important authority boundary:

- accepted constraints are durable database semantics;
- Rust validators and sanitizers are application behavior; and
- accepted checks evaluate the final canonical after-image.

The current implementation nevertheless runs Rust sanitizers and validators in
the shared typed/structural save preflight. Ordinary SQL writes therefore
usually observe those callbacks too, because the structural after-image is
materialized as the generated Rust entity before save validation.

That can make the two layers look unified even though they are not. Accepted
constraint activation, integrity verification, recovery replay, and any ingress
that cannot execute generated application behavior must remain independent of
Rust callbacks. Resumable SQL update already rejects entity graphs requiring
application callbacks rather than silently skipping them.

The result is a useful but potentially confusing combination:

1. resolve accepted defaults, generated values, managed values, and preserved
   values;
2. run applicable Rust sanitizers;
3. run Rust application validators;
4. construct the final canonical row;
5. evaluate accepted checks;
6. validate the remaining structural facts and stage the commit.

The ordering means accepted checks see sanitized values, which is desirable.
It also means that a Rust validator duplicating an accepted check can reject
first and hide the stable database-constraint diagnostic on ordinary writes.

## Contract Vocabulary

| Concept | Working meaning |
| --- | --- |
| Entity model | The authored `ic-model` graph: entities, records, enums, collections, Rust-facing behavior, and explicit IcyDB directives |
| Model validation | Validation of the authored graph and macro declaration itself |
| Rust model projection | Generated Rust types, derives, visitors, application validation, and application normalization |
| Dynamic model descriptor | Runtime-inspectable application-model metadata generated by `ic-model`; not database authority |
| IcyDB schema contract | Small public, versioned proposal vocabulary owned by IcyDB and usable without `ic-model` |
| Schema fragment | Reusable entity/type-scoped declaration keyed by immutable source identities; never submitted directly |
| Schema proposal | Composed, database-scoped, non-authoritative change intent submitted atomically to reconciliation |
| Source key | Immutable author identity used for reconciliation; distinct from editable SQL and display names |
| Accepted schema | Persisted database authority selected by reconciliation and DDL |
| Accepted runtime contract | Runtime projection derived only from accepted schema for planning, decoding, admission, integrity, and recovery |
| Row layout | Versioned persisted representation owned by accepted schema |
| Application preparation | Optional Rust validation/normalization outside database authority |
| Typed row projection | Complete decoded application value; not insert or patch intent |
| Typed write input | Operation-specific authored intent that preserves omission, `DEFAULT`, `NULL`, and value provenance |

The intended authority graph is:

~~~text
ic-model repository                         IcyDB repository
-------------------                         ----------------
authored entity model
    |---> Rust types, impls,
    |     dynamic descriptor
    |
    `---> IcyDB SchemaFragment --\
                                  | compose
other fragments / removals ------/
                                  `----------> SchemaProposal ---\
SQL DDL / direct schema API -----------------> reconciliation
                                                  |
                                                  v
                                            accepted schema
                                                  |
                                                  v
                                       accepted runtime contracts
                                                  |
                                                  v
                                       planning / admission / storage
~~~

The authored definition is the developer source, but it is not runtime
database authority. Generated Rust types do not reconstruct accepted schema.
The schema proposal is admitted input, not trusted state. Accepted runtime
contracts do not reopen the proposal to fill missing facts. SQL DDL,
programmatic schema proposals, and `ic-model` proposals converge at the same
catalog-native reconciliation boundary.

## Product And Package Boundary

There are two products in separate repositories.

### IcyDB Repository

> IcyDB is a standalone database for Internet Computer canisters. It accepts a
> public schema proposal or SQL DDL, reconciles that input into an accepted
> schema, and executes SQL or structural operations without requiring generated
> Rust entity types.

Package ownership:

| Package | Responsibility |
| --- | --- |
| `icydb-schema` | Public, bounded, versioned schema-proposal contract and builders; no accepted runtime or storage implementation |
| `icydb-core` | Reconciliation, accepted catalog, row layouts, storage, planning, execution, integrity, and recovery |
| `icydb` | Public database, SQL, dynamic value, structural mutation, and optional typed-adapter surfaces |
| `icydb-build` / `icydb-config` | Canister integration, database instances, stable-memory placement, endpoint policy, and deployment configuration |

The current `icydb-schema` name is reclaimed for this public proposal contract
as soon as its current authoring AST moves to the temporary unpublished
`icydb-model-legacy` extraction package. That AST later moves to `ic-model`;
old and new meanings never coexist in `icydb-schema`. Canister/store
definitions and stable-memory IDs remain IcyDB deployment concerns rather than
application-model facts.

### IC Model Repository

> `ic-model` is an Internet Computer model-authoring system. Annotated
> declarations produce one application projection—generated or derived Rust
> types, implementations, and a dynamic descriptor—and one database
> projection, an IcyDB `SchemaFragment` that composes into a database-scoped
> `SchemaProposal`.

Public package ownership:

| Package | Responsibility |
| --- | --- |
| `ic-model` | Public facade, model descriptor, model validation, application validation/normalization traits, proposal-fragment lowering, and macro re-exports |
| `ic-model-macros` | One published proc-macro package containing both derives and attribute/type-generating macros; parser and code-generation IR remain private modules in this package |

The external repository exposes exactly these two public packages. A published
proc-macro package cannot rely on an unpublished normal path dependency:
registry publication would require that dependency to be published too. Using
one proc-macro package therefore shares parser and IR code without a dependency
cycle or a fourth public package. If implementation experience later requires
a separate code-generation package, it must be a fourth published package with
an exact-version dependency; that is not the 0.213 design.

### Required Dependency Direction

~~~text
schema-only application -----> ic-model
                                  |   |
                                  |   `-------> ic-model-macros
                                  |
                                  `-----------> icydb-schema

runtime-adapter application -> ic-model
            |
            `----------------> icydb
                                  |   |
                                  |   `-------> icydb-schema
                                  |
                                  `-----------> icydb-core
                                                   |
                                                   `-> icydb-schema

explicit adapter expansion --emits consumer-side impls--> public icydb paths

forbidden:
icydb or icydb-core ----------> ic-model
ic-model ---------------------> icydb-core
ic-model ---------------------> icydb
~~~

`ic-model` depends on IcyDB's small public schema contract. It does not depend
on the `icydb` runtime facade or engine internals. It re-exports the proposal
vocabulary through an `ic_model::schema` namespace so ordinary generated code
does not require a direct `icydb-schema` dependency either. `icydb-schema` is
the only IcyDB package in the `ic-model` dependency closure. The namespace is
an exact re-export of contract types, not an `ic-model` mirror or translation
layer.

A schema-only application depends on `ic-model` and compiles with no `icydb`
package present. An application explicitly selecting typed database adapters
depends directly on both `ic-model` and `icydb`; only that opted-in expansion
implements IcyDB-owned adapter traits in the consumer crate. Emitting those
tokens does not create a package dependency from `ic-model` or
`ic-model-macros` to `icydb`. IcyDB builds, tests, and operates with no
`ic-model` package present.

The public proposal contract must contain owned, bounded, fallible data. It
must not expose engine-private DTOs, accepted field IDs, row-layout slots,
persisted codecs, generated function pointers, Rust callback paths, or
`'static` references. Proposal-format compatibility must be explicit and
typed across repository versions.

### Ownership Matrix

| Concern | Owner |
| --- | --- |
| Rust records, enums, newtypes, lists, maps, sets, tuples, derives, visitors, and model documentation | `ic-model` |
| Dynamic `ModelDescriptor` and application field traversal | `ic-model` |
| Explicit application validation and normalization | `ic-model` |
| Rust construction defaults | `ic-model` |
| Logical scalar kinds, bounded proposal literals, immutable source keys, source-level constraint expressions, fragments, removals, and proposal envelopes | `icydb-schema` |
| IcyDB entities, fields, primary keys, indexes, relations, accepted defaults, generated/managed policies, and representable constraints | `icydb-schema` proposal vocabulary, authored through `ic-model` or directly |
| Accepted IDs, canonical compiled constraint programs, fingerprints, evaluators, catalogs, row layouts, persisted codecs, indexes, journals, integrity, and recovery | `icydb-core` |
| Runtime `InputValue` / `OutputValue`, database APIs, opaque accepted bindings, `TypedRowAdapter`, `TypedWriteAdapter`, and `WriteCell` | `icydb` |
| Domain-specific application wrappers such as URL, phone, money, or title-cased values | `ic-model` or user code |
| Proposal validation and reconciliation | IcyDB |
| Database instances, stores, canister endpoints, stable-memory IDs, and allocation policy | `icydb-build` / `icydb-config` |
| Typed conversion between application types and public IcyDB values | Explicitly selected generated implementations of IcyDB-owned adapter traits in the consumer crate |

An application wrapper may use an IcyDB public scalar type, but it does not
redefine the database scalar contract. Conversely, IcyDB does not depend on
application wrappers to decode or enforce an accepted field.

### Layer-Qualified Entity And Record Names

`entity` and `record` are legitimate shared domain words. The design does not
invent unrelated vocabulary merely to hide their correspondence. Concrete
public Rust types must nevertheless disclose their layer and authority rather
than making every package export a bare `Entity`, `Record`, or `EntityModel`.

The normative naming families are:

| Layer | Entity concept | Record concept |
| --- | --- | --- |
| `ic-model` authoring and inspection | `EntityDescriptor` | `RecordDescriptor` |
| `icydb-schema` proposal contract | `EntityFragment` / `EntitySourceKey` | `RecordTypeFragment` / `TypeSourceKey` |
| `icydb-core` accepted catalog | `AcceptedEntity` or `EntityCatalogEntry` | `AcceptedRecordType` |
| `icydb` public runtime | `EntityName`, `EntityBinding`, or `TypedEntityBinding` | `InputValue::Record` / `OutputValue::Record` |
| Generated application code | domain names such as `User` | domain names such as `Address` |

Exact secondary names remain API-reviewable, but the authority suffixes are
not. In particular:

- the ergonomic authoring syntax remains `#[entity]` and `#[record]`;
- the old concrete `EntityModel` name hard-cuts to the appropriately scoped
  descriptor, fragment, or accepted type rather than moving intact;
- no `icydb::Entity` trait remains necessary for SQL, dynamic, or structural
  database operations;
- proposal types use `Fragment`, `Proposal`, or `SourceKey`;
- accepted and runtime types use `Accepted`, `Catalog`, `Binding`, `Input`, or
  `Output`; and
- shared enum-variant words such as `InputValue::Record` are acceptable
  because their owning type already identifies the layer.

Glob imports must not be required to use these APIs. Examples and diagnostics
use qualified names where two layers appear together.

### Two Projections From One Declaration

Deriving an existing type has this proposed spelling:

~~~rust
#[derive(ic_model::Model)]
#[model(icydb(
    source_key = "app.user",
    table = "users",
    primary_key = "id"
))]
pub struct User {
    #[model(icydb(source_key = "app.user.id"))]
    pub id: u64,
    #[model(icydb(source_key = "app.user.email"))]
    pub email: String,
}
~~~

The type-generating macro style remains available:

~~~rust
#[ic_model::entity(
    fields(
        field(source_key = "app.user.id", name = "id", ty = u64),
        field(source_key = "app.user.email", name = "email", ty = String)
    ),
    icydb(source_key = "app.user", primary_key = "id")
)]
pub struct User;
~~~

Both forms expose the same two outputs:

~~~rust
let model = User::model_descriptor();
let fragment = User::icydb_schema_fragment();

let proposal = ic_model::schema::SchemaProposal::compose(
    target_database,
    submission_key,
    expected_accepted_head,
    [
        User::icydb_schema_fragment(),
        Account::icydb_schema_fragment(),
    ],
    [],
)?;
~~~

The application projection contains:

- the generated concrete Rust type or derived implementations for an existing
  type;
- a dynamic `ModelDescriptor`;
- explicit validation and normalization;
- field and visitor functionality.

The database projection is an owned `SchemaFragment`, exposed through
`ic_model::schema`, containing only IcyDB schema facts representable by the
public contract. Fragments compose into one database-scoped `SchemaProposal`;
only that composed proposal may be submitted to IcyDB. Neither projection is
an accepted schema or a persisted row layout.

Ordinary `Model` expansion is schema-only with respect to IcyDB runtime
integration. It emits no path to `icydb`, no adapter implementation, and no
database-specific insert, patch, or replace type. Because proposal types are
re-exported through `ic_model::schema`, a schema-only consumer needs no direct
dependency on either `icydb` or `icydb-schema`.

### Explicit Runtime Adapter Opt-In

Runtime adapter generation is a separate, explicit selection. Proposed
spelling for an existing type is:

~~~rust
#[derive(ic_model::Model, ic_model::IcyDbAdapter)]
#[model(icydb(
    source_key = "app.user",
    table = "users",
    primary_key = "id"
))]
pub struct User {
    #[model(icydb(source_key = "app.user.id"))]
    pub id: u64,
    #[model(icydb(source_key = "app.user.email"))]
    pub email: String,
}
~~~

The type-generating form uses an equally explicit adapter option:

~~~rust
#[ic_model::entity(
    fields(
        field(source_key = "app.user.id", name = "id", ty = u64),
        field(source_key = "app.user.email", name = "email", ty = String)
    ),
    icydb(source_key = "app.user", primary_key = "id"),
    adapter(icydb)
)]
pub struct User;
~~~

The exact opt-in spelling remains API-reviewable. Its separation from ordinary
`Model` generation is normative. Only the opted-in expansion:

- mentions the consumer's resolved `icydb` crate path;
- treats the complete application type as a typed decoded-row projection;
- emits operation-specific `UserInsert`, `UserPatch`, and `UserReplace`
  inputs; and
- implements IcyDB-owned row/write adapter traits in the consumer crate.

The consuming application must directly depend on both `ic-model` and
`icydb`. Neither `ic-model` package acquires a normal, optional, or development
dependency on `icydb` merely to generate these tokens. Cross-product tests live
in a separate integration fixture whose application manifest owns both
dependencies.

The IcyDB-only experience does not mention `User`:

~~~rust
let receipt = database.apply_schema(proposal)?;
database.execute_sql("SELECT * FROM users")?;
database.mutate("users", key, patch)?;
~~~

The names are proposed API spelling. Their ownership and authority boundaries
are normative even if API review later refines the exact method names.

One split is already non-negotiable:

~~~text
Rust construction default  -> Rust model projection
IcyDB insert default        -> schema proposal and accepted schema
historical fill             -> accepted row-layout authority
~~~

Those concepts must never collapse back into one ambiguous `default`.

## Decision Summary

The separate repository is part of the architecture, not merely release
organization. It makes IcyDB's standalone contract testable and makes the
dependency direction visible to Cargo.

The system remains unified at the `ic-model` authoring level where one
declaration legitimately has two projections. It is separated at every IcyDB
authority boundary.

In particular:

- reserve `model` for authored and generated application concepts;
- reserve `schema` for database proposal, reconciliation, accepted authority,
  layout, and runtime contracts;
- retain shared domain words such as entity and record while requiring every
  concrete cross-layer type to disclose descriptor, fragment, accepted,
  catalog, binding, input, or output authority;
- make IcyDB's SQL and structural database surfaces entity-name/schema driven,
  without an `E: Entity` requirement;
- keep canister/store placement and stable-memory configuration in IcyDB;
- keep persisted-row encoding and row-slot selection in IcyDB rather than
  generated application types;
- let generated typed adapters convert between application types and IcyDB
  public dynamic values without becoming runtime authority;
- emit runtime adapter code only through an explicit application-side opt-in;
- prove ordinary model/schema generation compiles without `icydb`;
- forbid IcyDB packages from depending on `ic-model`;
- forbid `ic-model` packages from depending on `icydb` or `icydb-core`;
- do not introduce an `icydb-ic-model` bridge product; the consuming
  application owns the explicit runtime integration;
- prevent runtime owners from accepting proposal-only types where an accepted
  contract is required;
- replace dual-authority helper types with types whose construction proves
  whether they are proposal-only or accepted;
- authors do not duplicate a representable rule as both a Rust validator and
  an accepted constraint;
- the accepted schema must remain the authority for durable invariants;
- arbitrary contextual Rust validation must remain application-only;
- arbitrary Rust sanitization must not become hidden database semantics; and
- any future database-owned transformation requires its own bounded,
  deterministic, persisted representation.

Validation and transformation then become one important subproblem inside the
larger model/schema separation.

## Validation And Transformation Contract

### Database Constraints

An accepted constraint rejects an invalid final value or row. It is named,
persisted, deterministic, fingerprinted, introspectable, independently
evaluable, and enforced by every IcyDB write ingress.

An `ic-model` declaration can express a database-compatible rule once:

~~~rust
#[model(
    constraint(
        source_key = "app.account.balance_nonnegative",
        name = "balance_nonnegative",
        check = "balance >= 0"
    )
)]
pub struct Account {
    pub balance: i64,
}
~~~

The IcyDB projection lowers that declaration into a public schema fragment.
It does not generate a second Rust evaluator for that constraint. Early proof
against database rules uses an IcyDB API that evaluates the live accepted
contract and therefore preserves its null behavior, coercions, identities, and
diagnostics.

Authors do not repeat the same rule as an independent Rust validator. A
separate application validator is retained only when it has genuinely
application-specific semantics.

### No Database Normalizer In 0.213

0.213 does not add an accepted sanitizer, normalizer, mutation rewrite,
trigger, or transformation language. IcyDB admits authored values, applies its
existing accepted defaults/generation/management policies, constructs the
canonical after-image, and evaluates accepted constraints. It does not execute
`ic-model` Rust code.

A future database rewrite design would require a bounded persisted form,
stable and versioned semantics, exact fingerprint participation, deterministic
evaluation, uniform ingress coverage, typed diagnostics, migration rules, and
independent execution without generated Rust. That future work is outside
0.213.

### Application Validation And Normalization

Application validation and normalization belong to `ic-model`. Generated or
derived types expose explicit application operations. With the runtime adapter
explicitly selected, an operation-specific write can be prepared as:

~~~rust
let authored = UserInsert::new()
    .email(WriteCell::Value(email));
let authored = authored.normalize()?;
authored.validate()?;
database.insert_typed(&binding, authored)?;
~~~

These operations may provide richer errors, inspect request context through
explicit arguments, or apply transformations before a database call. They are
not persisted, fingerprinted, used by IcyDB integrity checking, invoked by
SQL, assumed by recovery, or registered as database runtime hooks.

The current `sanitizer` vocabulary hard-cuts to `normalizer` in `ic-model`.
The current automatic callback invocation from IcyDB save preflight is
removed. Typed adapters perform conversion only; they do not implicitly
normalize or validate unless a separately named convenience API explicitly
composes those application steps.

`Model::validate()` executes application validators only. Proposal builders
validate proposal structure only. Neither locally interprets accepted
constraints. The only early database-rule validation API evaluates the current
accepted schema through IcyDB and never substitutes for write admission.

## Classification Of Existing IcyDB-Style Rules

0.213 inventories every built-in validator and sanitizer and applies this
ownership classification:

| Existing behavior | Likely durable owner | Notes |
| --- | --- | --- |
| Numeric minimum/maximum/range | Accepted constraint | Exact comparisons are directly representable |
| Text/blob/collection length | Accepted constraint | Requires frozen `CHAR_LENGTH`, octet-length, or cardinality semantics |
| Enum membership | Accepted type or constraint | Prefer the exact accepted enum catalog |
| Decimal scale | Accepted field kind where possible | Rejecting excess scale may be safer than silently rounding |
| Numeric multiple-of | Accepted constraint | Exact integer/decimal semantics only |
| Not-null | Accepted structural constraint | Must remain separate from SQL `CHECK` unknown semantics |
| Trim or fixed case conversion | `ic-model` application normalizer | IcyDB gains no rewrite language in 0.213 |
| Managed timestamps | Accepted write policy | Already closer to database-owned generation than sanitization |
| URL, MIME, phone, ISO, hash format | `ic-model` application validator unless already exactly representable | Promotion needs a separate frozen database constraint contract |
| Locale-sensitive title casing | `ic-model` application normalizer | Version and locale behavior make durable semantics unsafe |
| Decimal or money rounding | Explicit application conversion | Lossy silent database mutation is excluded |
| Caller authorization or external lookup | Application validator or IcyDB access policy | Not a row-local deterministic constraint |
| Arbitrary custom Rust callback | `ic-model` application-only | Never persisted by path or inferred as accepted semantics |

## Runtime And Adapter Flow

The standalone IcyDB write pipeline is:

~~~text
resolve accepted schema and authored intent
    -> resolve defaults, generation, management, and preserved values
    -> construct the canonical after-image
    -> evaluate accepted checks
    -> validate structural and cross-row constraints
    -> stage and publish the commit
~~~

Standalone SQL and structural callers enter the database without an
application model:

~~~text
SQL or dynamic values
    -> entity/table resolution through accepted schema
    -> accepted database pipeline
~~~

Explicitly selected `ic-model` typed-adapter preparation sits outside that
authority:

~~~text
request
    -> generated operation-specific insert / patch / replace intent
    -> optional ic-model application normalization
    -> optional ic-model application validation
    -> generated implementation of an IcyDB-owned adapter trait
    -> opaque accepted-schema binding
    -> IcyDB database mutation ingress
    -> accepted database pipeline
~~~

The high-level adapter can still offer both phases conveniently. It must not
generate persisted row bytes or choose accepted row-layout slots. The core
database contract remains complete when `ic-model` is not linked.

## Normative Boundary Details

### Existing Type Disposition

Every production use of the old `EntityModel`, `E::MODEL`, `SchemaInfo`,
generated model constants, proposal lowering, and accepted runtime projections
is classified before it moves:

- authored-model or macro validation;
- generated Rust convenience;
- schema proposal or reconciliation;
- accepted runtime planning or execution; or
- test-only model construction.

The final boundary is:

- the old `EntityModel` concrete type is retired rather than transferred;
- `EntityDescriptor` and `RecordDescriptor` belong to `ic-model`;
- IcyDB accepts `SchemaProposal`, never an application descriptor;
- the shared `SchemaInfo` shape splits into accepted-runtime types and
  model/proposal-only types rather than retaining dual constructors;
- accepted planning, admission, integrity, recovery, relations, and commit
  preparation accept only types constructed from accepted schema;
- `E::MODEL` is removed from IcyDB database traits and runtime dispatch;
- model-only planner helpers move to `ic-model`, test support, or consume an
  explicit public proposal contract; and
- explicitly selected generated typed adapters use public IcyDB values and
  accepted binding APIs, not engine model structs.

### Public IcyDB Schema Contract

`icydb-schema` exposes:

- owned `SchemaFragment`, `EntityFragment`, `RecordTypeFragment`,
  `SchemaProposal`, explicit removal, and other layer-qualified proposal DTOs;
- builders that provide ergonomic construction and local validation;
- an explicit proposal contract version;
- an explicit required-capability set;
- deterministic canonical ordering;
- bounded, fallible serialization and decoding for transported proposals; and
- typed errors for malformed, oversized, obsolete, or unsupported input.

The package is a pure leaf. It owns logical scalar kinds, bounded proposal
literals, immutable source-key types, the source-level constraint expression
AST, fragments, removals, composition errors, and the proposal envelope. It
does not depend on `icydb`, `icydb-core`, the authoring compiler, macros, or a
temporary legacy AST.

`icydb-core` owns accepted IDs, canonical compiled constraint programs,
accepted fingerprints, evaluators, physical layouts, and persisted runtime
forms. `icydb` owns runtime `InputValue` / `OutputValue` and database APIs.
A source-level proposal expression never contains accepted bytecode, an
accepted constraint ID, a physical slot, or a runtime fingerprint. The
proposal envelope may carry an expected accepted head solely as an optimistic
concurrency condition.

The source identity vocabulary is distinct from editable naming:

~~~rust
struct EntitySourceKey(/* bounded immutable key */);
struct FieldSourceKey(/* bounded immutable key */);
struct TypeSourceKey(/* bounded immutable key */);
struct ConstraintSourceKey(/* bounded immutable key */);
struct IndexSourceKey(/* bounded immutable key */);
struct RelationSourceKey(/* bounded immutable key */);

enum SchemaRemoval {
    Entity(EntitySourceKey),
    Field {
        entity: EntitySourceKey,
        field: FieldSourceKey,
    },
    Type(TypeSourceKey),
    Constraint(ConstraintSourceKey),
    Index(IndexSourceKey),
    Relation(RelationSourceKey),
}
~~~

SQL/display names remain editable attributes. Renaming `users` or `email`
preserves its source key and reconciles as a rename, not delete-plus-add.
Source keys are not accepted IDs: IcyDB resolves them through the accepted
catalog and continues to own accepted identity allocation. Direct proposal
authors must persist their keys with the source declaration. SQL DDL assigns a
source key when an object is first accepted and preserves it through rename.

The top-level contract contains the logical entity/type graph, immutable source
keys, editable names, fields, nullability, primary keys, indexes, relations,
accepted defaults, generated/managed write policies, and representable
constraints. It does not contain database-instance placement, accepted IDs,
physical index ordinals, row-layout slots, persisted codecs, journal records,
runtime hooks, or callbacks.

### Proposal Composition

An entity or shared-type declaration produces a `SchemaFragment`. A fragment
has entity/type/module-local authorship scope and can contain definitions plus
references to immutable source keys. It is not accepted by
`Database::apply_schema`.

`SchemaProposal::compose` creates the only submit-capable form:

~~~rust
let proposal = SchemaProposal::compose(
    target_database,
    submission_key,
    expected_accepted_head,
    [
        User::icydb_schema_fragment(),
        Account::icydb_schema_fragment(),
    ],
    explicit_removals,
)?;
~~~

The proposal carries an opaque target-database token and all included changes
are admitted against that database's one expected accepted head in one atomic
submission. Applying it to any other database fails before mutation. Reusable
schema authorship lives in fragments; a composed submission is deliberately
bound to its target. It is a delta, not an assertion that the fragments
describe every entity in the database. In 0.213:

- absence means unchanged;
- deletion never follows from absence;
- every removal is an explicit, typed operation naming an immutable source
  key;
- a fragment cannot request physical placement, accepted identity reuse, or a
  partially authoritative scope; and
- there is no implicit prune, replace-all, or module-owned deletion mode.

Composition constructs one source-key symbol table. Duplicate definition keys
are errors even when their payloads happen to match; one fragment must own
each definition. A reference is not a duplicate definition. Differing
definitions, editable-name collisions, duplicate removals, definition/removal
collisions, and incompatible capability requirements return typed composition
errors.

Relations and named type references target immutable source keys. Composition
resolves references to definitions present in the submission and carries
external references for reconciliation against the expected accepted head.
Application fails atomically if a reference is absent, removed by the same
proposal, ambiguous, or of the wrong kind. Cross-fragment declaration order
does not affect the result.

Canonicalization sorts unordered definitions and operations by their immutable
source keys. Semantically ordered members—composite key fields, index columns,
tuple members, and explicit sort directions—retain authored order. The
canonical proposal digest therefore does not depend on fragment iteration
order.

Builders and composition validate their own structural invariants, but IcyDB
treats every proposal as untrusted and validates it again. SQL DDL and
`SchemaProposal` input call the same catalog-native mutation owners. SQL
remains a frontend and does not become an alternate source of schema
semantics.

### Application, Concurrency, And Activation

Every proposal carries an IcyDB-issued opaque target-database token, a
caller-generated immutable submission key, and a required expected accepted
head:

~~~rust
struct ExpectedDatabaseIdentity(/* opaque bounded token */);

enum ExpectedAcceptedHead {
    Empty,
    Exact {
        revision: ExpectedSchemaRevision,
        fingerprint: ExpectedSchemaFingerprint,
    },
}
~~~

The target and exact-head fields are bounded, opaque proposal transport tokens
obtained from IcyDB. `icydb-core` still owns computing and interpreting
database identities, accepted revisions, and fingerprints; proposal code
cannot derive accepted runtime semantics from the tokens.

There is no blind or cross-database application API in 0.213. A target
identity mismatch or stale revision/fingerprint fails before mutation with a
typed conflict. The canonical proposal digest, target identity, expected head,
and submission key are durably associated. Repeating the same key with the
same digest, target, and expected head returns the existing receipt or job.
Reusing that key with any different member is a typed idempotency conflict.

`Database::apply_schema` returns a durable, queryable receipt:

~~~rust
struct SchemaChangeReceipt {
    database_identity: DatabaseIdentity,
    submission_key: SchemaSubmissionKey,
    proposal_digest: SchemaProposalDigest,
    prior_head: AcceptedSchemaHead,
    outcome: SchemaChangeOutcome,
}

enum SchemaChangeOutcome {
    NoOp { accepted_head: AcceptedSchemaHead },
    Applied { accepted_head: AcceptedSchemaHead },
    Pending {
        job: SchemaChangeJob,
        candidate_head: CandidateSchemaHead,
    },
    Failed { error: SchemaChangeFailure },
}
~~~

Admission of the proposal is atomic: no subset is silently accepted if
composition, reconciliation, or durable job creation fails. `NoOp` and
`Applied` are successful terminal states. `Failed` is a durable terminal state
for work that failed after job admission. `Pending` is not
success-by-assumption; the durable job and receipt can be queried by submission
key, resumed, and observed through a typed terminal outcome. A failed
candidate never becomes the accepted head; its prior accepted head remains
runtime authority.

Reconciliation must use the activation machinery established by 0.211.
Creating or validating a constraint, index, or relation that cannot finish
within the bounded call produces `Pending`; `apply_schema` cannot skip scans,
mark an unvalidated object active, or implement a second activation protocol.
Visibility of the old head, candidate facts, and newly active head follows the
0.211 catalog-native state machine. The current accepted head remains the
runtime authority until that state machine says otherwise.

Supported SQL DDL lowers into the same catalog-native change intent. The
database/transaction supplies its target identity and expected head; the SQL
execution context supplies a durable statement submission key. DDL that starts
resumable activation returns the same receipt/job semantics rather than
reporting the candidate as fully active. SQL is not a second mutation or
activation implementation.

Before 1.0, IcyDB accepts the current proposal contract version or fails with a
typed version/capability error. It does not translate obsolete proposal forms.

The public proposal cannot simply expose current `EntityModel` or
`CompiledSchemaProposal`. It needs owned data and a public contract shaped for
untrusted admission rather than generated `&'static` metadata.

### IC Model Package Split

The two public packages have these responsibilities:

- `ic-model` owns the application runtime facade, `ModelDescriptor`,
  `EntityDescriptor`, `RecordDescriptor`, validation/normalization traits, and
  the `ic_model::schema` re-export;
- `ic-model-macros` contains both derives that enrich ordinary authored Rust
  types and attributes/functions that create complete types from declarative
  definitions.

`ic-model` always depends on the public `icydb-schema` package because producing
IcyDB fragments is part of its product contract. It never depends on `icydb`
or `icydb-core`, normally, optionally, or for package-local development tests.
Cross-product validation belongs to a separate consumer fixture.

Both authoring styles produce one equivalent `ModelDescriptor`. Ordinary
`ic-model` code lowers that descriptor into `SchemaFragment`; proc macros do
not construct engine DTOs. Parser and IR reuse remain private modules in the
single published proc-macro package.

Ordinary model expansion resolves the consumer's `ic-model` dependency and
uses its `schema` re-export. It emits no literal or resolved path to `icydb` or
`icydb-schema`. Only explicit adapter expansion resolves the consumer's direct
`icydb` dependency and emits runtime adapter paths.

Generated paths are never assumed to be literally `::ic_model` or `::icydb`.
Macro expansion resolves renamed dependencies from the consumer manifest and
supports explicit crate-path overrides where manifest-aware resolution is
unavailable or ambiguous. Compile tests cover schema-only use without
`icydb`, default names, renamed dependencies, explicit overrides, and opted-in
adapter use with both direct dependencies.

### Local Rust Validation

`ic-model::Model::validate()` runs application validators only. Structural
proposal validation checks source declarations and composition only. Neither
generates or runs a local evaluator for accepted constraints.

IcyDB separately exposes accepted-schema validation through its public API
when callers need early proof against the live database contract. That API
accepts operation-specific structural or typed intent, resolves its candidate
after-image through the live binding, and uses the same accepted
expression/evaluator path as admission. It returns stable
accepted-constraint diagnostics but performs no write and invokes no
application callback. Final write admission remains authoritative because
accepted schema and row state can change between validation and mutation.

### Diagnostic Ownership

Explicit application preparation returns `ic-model` diagnostics before the
caller invokes IcyDB. A direct IcyDB write returns IcyDB admission diagnostics
with stable accepted constraint identity. There is no combined hidden
callback/database pipeline whose incidental ordering chooses the error.

A convenience API may collect both classes outside the write path, but it does
not replace database admission and does not change write-error precedence.

### SQL And Application Callbacks

No. The separate product boundary makes this a decision rather than an open
option. SQL and dynamic structural writes use only accepted database semantics.
IcyDB does not link, locate, or execute `ic-model` callbacks.

A caller may explicitly run `ic-model` application preparation before sending
values to IcyDB, but that is an application-side phase. SQL issued directly
against IcyDB does not acquire extra behavior merely because generated
application types happen to exist in the same canister.

### Typed Adapter And Accepted-Schema Drift

The ordinary `ic-model` projection describes the fragment produced from one
source definition and emits no runtime adapter. When the author explicitly
selects adapter generation, the live database may still have accepted
compatible DDL changes or may reject a stale proposal. That adapter therefore
cannot assume generated field order equals accepted row-layout order.

The adapter traits belong to `icydb`, not `ic-model`:

~~~rust
pub trait TypedRowAdapter {
    type Row;

    fn decode_row(
        binding: &TypedEntityBinding,
        row: OutputRow,
    ) -> Result<Self::Row, TypedAdapterError>;
}

pub trait TypedWriteAdapter {
    fn encode_write(
        self,
        binding: &TypedEntityBinding,
    ) -> Result<TypedWrite, TypedAdapterError>;
}
~~~

The exact signatures remain API-reviewable, but ownership is normative.
Only explicit adapter expansion in the consumer crate implements these
IcyDB-owned traits for consumer-owned generated types. The application
therefore depends directly on both `ic-model` and `icydb`; neither product
package depends on the other.

IcyDB issues an opaque `TypedEntityBinding` after resolving immutable entity
and field source keys against an accepted snapshot. The binding is tied to:

- database identity;
- accepted entity identity;
- accepted schema revision and fingerprint;
- the relevant entity generation; and
- the requested row/write projection.

Callers and generated code cannot inspect accepted field IDs, physical slots,
compiled programs, or persisted codecs through the binding. They can only ask
IcyDB to encode/decode public values through it. Using a binding with another
database, entity, revision, fingerprint, or generation returns a typed stale
or mismatched-binding error. Cached bindings are invalidated when the accepted
fingerprint or relevant generation changes.

Compatible accepted additions may be ignored by an older adapter when it
requests an explicit known-field row projection. A missing known source key,
incompatible field kind, incompatible nullability, or unavailable required
projection fails with a typed binding error. Editable renames do not break a
binding request because generated adapters use immutable source keys.

### Typed Write Intent And Provenance

A complete `User` is a decoded row projection. It is not automatically an
insert, replace, or patch input. Reusing it as write intent would collapse the
authored/defaulted/generated/managed/preserved distinctions established by
0.209.

The opted-in adapter expansion generates operation types that preserve intent
explicitly:

~~~rust
enum WriteCell<T> {
    Omitted,
    Default,
    Null,
    Value(T),
}

struct UserInsert { /* authorable insert fields */ }
struct UserPatch { /* authorable patch fields */ }
struct UserReplace { /* authorable replacement fields */ }
struct User { /* complete decoded row projection */ }
~~~

`WriteCell` is owned by `icydb`. `Default` is included in addition to
`Omitted`, `Null`, and `Value` because an explicit SQL-style `DEFAULT` request
is authored intent and must not be conflated with absence.

The lowering rules are operation-specific:

| Input | Insert | Patch | Replace |
| --- | --- | --- | --- |
| `Omitted` | no authored value; accepted insert default/generation policy may supply it | preserve the current accepted value | no authored value; apply accepted replacement omission rules, never preserve implicitly |
| `Default` | explicitly request the accepted default | explicitly reset through the accepted default policy | explicitly request the accepted default |
| `Null` | explicitly authored `NULL` | explicitly set `NULL` | explicitly authored `NULL` |
| `Value(v)` | explicitly authored value | explicitly replace with `v` | explicitly authored value |

Nullability and default availability are still accepted-schema decisions.
Generated, managed, or otherwise database-owned fields are omitted from
authorable setters/types unless the accepted public contract explicitly
permits authorship. The adapter lowers intent into IcyDB's structural mutation
form; it does not resolve defaults, generation, management, preservation, or
row slots itself.

Application normalization and validation operate on the operation-specific
input before encoding. They may transform or reject authored `Value` cells but
must not turn `Omitted` into authored data, synthesize database defaults, or
claim database provenance. Database admission retains the authoritative
provenance for the final after-image.

Typed reads may return a complete `User` plus separate IcyDB-owned result
metadata when provenance is requested. The row projection itself does not
become writable by implication. A convenience “replace from row” operation
must be explicitly named and lower through `UserReplace`; there is no
ambiguous `database.insert_typed(user)` API.

All binding, conversion, and intent errors are typed. The adapter must never
reconstruct, inspect, or silently downgrade the accepted schema.

## Migration And Release Contract

Repository separation cannot be one source move. The authority boundary must
exist inside IcyDB before the code is transferred:

1. move the existing authoring AST into a temporary unpublished
   `icydb-model-legacy` package, make `icydb-schema` a pure proposal-contract
   leaf immediately, and route the new contract through the existing
   catalog-native reconciliation owner;
2. make IcyDB schema acceptance, SQL, dynamic reads, and structural writes
   operate without generated entity types;
3. split accepted runtime contracts from model-only helpers and remove
   generated model/callback requirements from database recovery, integrity,
   relations, and commit preparation;
4. adapt the current embedded model compiler so ordinary model generation
   emits layer-qualified descriptors and public schema fragments with no
   `icydb` path, while a separate explicit adapter selection emits
   operation-specific write inputs and implementations of public IcyDB-owned
   adapter traits;
5. establish the external `ic-model` workspace and move the authoring AST,
   derives, type-generating macros, and application behavior;
6. prove one-way released-package interoperability before deleting the
   embedded packages; and
7. hard-cut old `icydb::schema`, macro re-exports, generated persistence
   implementations, and callback-coupled database paths.

This staging does not authorize dual runtime dispatch or compatibility shims.
`icydb-model-legacy` is an explicitly temporary extraction package with
`publish = false`; it must not gain new API or authority. The temporarily
embedded compiler must use the same new proposal/adapter path that the
external repository will use. The final cut removes both temporary owners
completely.

No existing validator or sanitizer is silently promoted during extraction.
Each declaration must still be classified as an accepted constraint, accepted
type fact, accepted write policy, `ic-model` application validator/normalizer,
or application-only behavior. Automatic promotion from a Rust callback path
would recreate the authority problem identified by 0.211.

Cross-repository release order must be explicit:

1. IcyDB publishes or otherwise freezes the required public schema-contract
   revision;
2. `ic-model` targets that revision and passes golden proposal plus typed
   adapter integration tests;
3. downstream fixtures switch to the external packages; and
4. IcyDB removes the embedded compiler and old re-exports.

Path or Git dependencies can support local integration but are not the
released compatibility contract.

## Risks

- A separate repository can hide rather than solve coupling if `ic-model`
  imports `icydb-core` or IcyDB retains generated-model runtime requirements.
- An oversized public schema contract can become a second engine API and make
  internal refactoring harder.
- Cross-repository version skew can produce proposals newer than the running
  database or adapters older than the accepted schema.
- Renamed consumer dependencies can break generated paths unless manifest
  resolution and explicit overrides are tested.
- A normal model macro can accidentally make schema-only users depend on
  `icydb` if runtime adapter tokens are not strictly opt-in.
- Bare or reused `Entity`, `Record`, and `EntityModel` type names can recreate
  authority ambiguity even when their Cargo dependencies are correct.
- Reusing or changing an immutable source key can misidentify an authored
  object; tooling must make keys visible and stable without presenting them as
  editable names.
- Proposal composition can become a second reconciliation engine if it tries
  to assign accepted identities or evaluate database compatibility locally.
- Durable schema jobs can report misleading success if `Pending` receipts are
  treated as fully active rather than queried to a terminal state.
- Generated adapters can accidentally retain persisted-row codecs, accepted
  slots, or layout assumptions and recreate runtime authority outside IcyDB.
- Ambiguous typed write inputs can erase omission/default/preservation
  provenance even when the structural database lane is correct.
- Moving canister/store placement into `ic-model` would couple reusable
  application types to one database instance and deployment topology.
- A multi-repository move can leave two apparent owners unless the final
  hard-cut and release order are explicit.
- Application normalization can still conceal bad input if convenience APIs
  make the explicit preparation phase unclear.
- A new dynamic standalone surface can duplicate typed planning/execution
  unless both adapters lower into the same accepted structural owner.
- Cross-repository integration support can increase native build time and
  combined downstream Wasm size even if standalone IcyDB becomes smaller.

## Non-Goals

0.213 excludes:

- turning `ic-model` into a database-neutral multi-backend ORM; its IcyDB
  projection deliberately targets IcyDB's public contract;
- making IcyDB, `icydb-core`, or IcyDB runtime correctness depend on
  `ic-model`;
- allowing `ic-model` to depend on `icydb-core` or engine-private modules;
- allowing `ic-model` to depend on the `icydb` runtime facade;
- introducing an `icydb-ic-model` bridge package for code that can be emitted
  into the explicitly opted-in consumer;
- requiring schema-only `ic-model` consumers to depend directly on `icydb` or
  `icydb-schema`;
- keeping the current embedded model/schema packages as compatibility
  wrappers after the external cut;
- re-exporting `ic-model` from `icydb` in a way that reverses the dependency;
- placing canister/store memory allocation or endpoint policy in the reusable
  application model;
- generating accepted row-layout slots, persisted row bytes, recovery state,
  or runtime hook registries in `ic-model`;
- renaming crates without first proving the dependency and authority split;
- changing a durable format merely to improve terminology;
- persisting Rust callback paths, function pointers, closures, or trait
  objects;
- arbitrary triggers or external calls;
- nondeterministic generic normalizers;
- locale-dependent transformations;
- automatic promotion of every validator or sanitizer;
- silent historical rewrites;
- compatibility shims for pre-1.0 callback semantics;
- expanding 0.211 to implement normalization retrospectively;
- adding an accepted normalizer, rewrite, or trigger language;
- adding implicit delete-by-absence, prune, or replace-all proposal semantics;
- exact unsigned identity generation, which belongs to 0.214; and
- SQL structural coverage, range remediation, and performance work, which
  belong to 0.215.

## Implementation Patches

0.213 is planned as eight meaningful landing patches with a specific closeout
audit before 0.214. A landing patch is a review/worktree handoff, not an
automatically numbered release:

### Patch 1: Public Proposal Contract

Outcome: inventory every authority crossing, freeze the vocabulary, introduce
immutable source keys plus bounded fragments/proposal DTOs inside
`icydb-schema`, and add dependency/layer guards. The old authoring AST moves
immediately to `icydb-model-legacy` with `publish = false`; old and new
contracts do not share the reclaimed leaf package. Freeze the layer-qualified
entity/record naming matrix before new public types land.

Focused evidence: proposal builder/codec bounds, malformed/version/capability
errors, source-key rename behavior, leaf-package dependency graph, temporary
package publication guard, public-name inventory, opening performance/Wasm
baselines, and explicit numeric closeout budgets recorded in the status
tracker before later production edits.

### Patch 2: Proposal Composition And Schema Application

Outcome: compose fragments and explicit removals into one database-scoped
proposal, then apply it through optimistic, idempotent, catalog-native schema
mutation. Return durable terminal or pending receipts and route resumable work
through the 0.211 activation state machine.

Focused evidence: conflict and ordering matrices, empty database
initialization, wrong-database and stale-head rejection, exact replay,
submission-key conflict, atomic failure, pending/resumed activation, SQL DDL
convergence, accepted snapshot authority, and a standalone fixture with no
`ic-model` dependency.

### Patch 3: Standalone Read And SQL Query Surface

Outcome: make dynamic reads, SQL query/introspection, planning, and output rows
entity-name/schema driven. Generated typed reads become an outer projection
over this maintained structural lane.

Focused evidence: SQL and dynamic read parity, accepted field/index routing,
projection shapes, prepared-query behavior, schema-fingerprint invalidation,
and absence of `E::MODEL` from the maintained read path.

### Patch 4: Standalone Structural And SQL Write Surface

Focused evidence: write-ingress parity, accepted defaults/generated/managed
values, preserved values, all four `WriteCell` states, opaque-binding
fingerprint/generation mismatch, constraints, relations, atomicity, returned
rows, and proof that SQL and direct dynamic writes never invoke validators or
normalizers.

### Patch 5: Accepted Runtime Authority Cleanup

Outcome: split dual-authority `SchemaInfo`-style helpers and remove generated
models, generated persistence codecs, and callback hooks from commit,

Focused evidence: construction-level authority guards, recovery and integrity
without model registration, accepted relation enforcement, typed failure for
missing/corrupt accepted state, and a complete production-use inventory for

### Patch 6: Embedded Compiler Boundary

Outcome: make ordinary embedded authoring generate only application types,
layer-qualified descriptors, application validators/normalizers, and

Add a separate explicit adapter expansion that generates complete row
projections, operation-specific insert/patch/replace inputs, and

accepted-constraint mirrors, hard-cut automatic save callbacks and the

Focused evidence: derive and type-generating semantic parity, one-declaration
constraint lowering without a second evaluator, explicit application

### Patch 7: External IC Model Repository

Outcome: establish the external repository with the two public packages
`ic-model` and `ic-model-macros`, transfer the model AST and single proc-macro

Focused evidence: independent package builds, no `icydb-core` dependency,

### Patch 8: Consumer Cutover And Hard Cut

Outcome: switch maintained IcyDB examples and fixtures to external `ic-model`,
delete `icydb-model-legacy`, the embedded compiler crates, and IcyDB macro

Slice boundaries change only through a reviewed design/status update before

## Appendix A: Comparable Systems

The common pattern is not to eliminate validation or transformation. It is to
make the authority and execution boundary explicit.

### PostgreSQL

PostgreSQL keeps database invariants in schema-owned `CHECK`, domain, unique,
primary-key, and foreign-key constraints. Reusable scalar restrictions can be
expressed as domains. Write-time computation or transformation uses explicit
features such as generated columns and triggers rather than application
validators being treated as constraints.

PostgreSQL also documents an important durability issue: check behavior is
assumed to be immutable. Changing a user function used by a check can leave
stored rows inconsistent until the constraint is dropped and re-added. That is
strong support for IcyDB persisting canonical semantics rather than callback
paths.

Sources:

- <https://www.postgresql.org/docs/current/ddl-constraints.html>
- <https://www.postgresql.org/docs/current/sql-createdomain.html>
- <https://www.postgresql.org/docs/current/ddl-generated-columns.html>
- <https://www.postgresql.org/docs/current/sql-createtrigger.html>

### SurrealDB

SurrealDB places validation and transformation beside each other but keeps
them as separate schema clauses. `ASSERT` validates a value. `VALUE` can
transform an authored value before it is stored, including a documented
lowercase-email example. Both are database-visible schema behavior.

This demonstrates that database transformation must be an explicit
schema-owned feature. 0.213 does not import that feature into IcyDB.

Sources:

- <https://surrealdb.com/docs/reference/query-language/statements/define/field>
- <https://surrealdb.com/docs/learn/schema-management/tables-and-fields/fields-and-validation>

### Gel

Gel exposes schema-defined mutation rewrites for insert and update. Rewrites
are explicit schema objects, can distinguish authored from preserved values,
and apply to DML through its SQL adapter. They are complementary to constraints
and triggers rather than disguised application hooks. This supports deferring
rewrites to a separate future database design instead of preserving current
Rust callback execution.

Sources:

- <https://docs.geldata.com/reference/datamodel/mutation_rewrites>
- <https://docs.geldata.com/reference/using/sql_adapter>

### MongoDB And Convex

MongoDB collection validators reject invalid inserted or updated documents but
do not turn arbitrary application cleanup into database validation. Convex
offers a useful authoring precedent: its declarative validator builder is used
for both function arguments and table schemas, while a pushed schema validates
existing documents and future writes.

Convex therefore demonstrates declaration reuse without making arbitrary
application functions the database authority.

Sources:

- <https://www.mongodb.com/docs/manual/core/schema-validation/>
- <https://docs.convex.dev/database/schemas>

### Ecto, Rails, And SeaORM

Application data layers retain validation and lifecycle preparation because
they are useful. Ecto explicitly distinguishes validations, which can run
without the database and provide early feedback, from database constraints,
which provide safe integrity enforcement. Rails and SeaORM expose lifecycle
callbacks before validation or save.

These systems also demonstrate the bypass risk: an application callback is
only as universal as the application write path that invokes it. IcyDB must
not describe such a callback as database authority merely because its normal
typed save path calls it.

Sources:

- <https://hexdocs.pm/ecto/Ecto.Changeset.html>
- <https://guides.rubyonrails.org/active_record_callbacks.html>
- <https://docs.rs/sea-orm/latest/sea_orm/entity/trait.ActiveModelBehavior.html>

The combined lesson is:

- durable invariants belong to database schema;
- application preparation remains useful but must be explicit; and
- database transformations, if added later, require their own schema-owned
  contract rather than callback reuse.
