# 0.213 Schema Authority And Application Model Separation: Thoughts

Status: exploratory notes only.

Roadmap position: after 0.212 closeout and an explicit user direction to begin
0.213; before 0.214

This is not an approved design, implementation plan, release commitment, or
normative contract. It records questions and working conclusions about the
boundary between IcyDB's authored entity model, generated Rust projection,
schema proposal, accepted database schema, runtime contracts, validators, and
normalizers. A later design may keep, revise, or reject any part of these
notes.

No implementation starts from these notes. They must first be promoted into a
reviewed and approved design with repository evidence and bounded landing
patches.

The validator/sanitizer review exposed the problem, but it is not the whole
problem. The broader issue is that `model` and `schema` still name overlapping
authoring, generated, accepted, and runtime concepts. That makes proposal data
look authoritative and makes application callbacks look like database
semantics when shared execution happens to invoke them.

[A companion exploratory note](0.213-model-schema-split-notes.md) preserves the
more detailed crate/package discussion that led to this synthesis.

## Why These Notes Exist

IcyDB already states the intended database authority correctly:

- source definitions author an entity model;
- generated metadata proposes database schema;
- reconciliation accepts one durable database schema; and
- the accepted schema snapshot is runtime authority.

The type and module vocabulary does not yet make that separation inevitable.
`EntityModel` is still described as a macro-generated runtime schema snapshot,
and some of its fields are described as authoritative for runtime planning.
`SchemaInfo` can be constructed from either trusted generated metadata or an
accepted snapshot. Runtime and model-only planning helpers therefore share
types and names even where their authority is different.

The desired seam exists in part:

- `db::schema::proposal` projects trusted `EntityModel` data into a typed
  proposal;
- accepted snapshots persist field IDs, layouts, constraints, indexes,
  relations, defaults, and write policies; and
- accepted read/write contracts already forbid runtime fallback
  reconstruction from generated models.

0.213 should make that seam structural rather than conventional.

The 0.211 constraint work establishes an important authority boundary:

- accepted constraints are durable database semantics;
- Rust validators and sanitizers are application behavior; and
- accepted checks evaluate the final canonical after-image.

The current implementation nevertheless runs Rust sanitizers and validators in
the shared typed/structural save preflight. Ordinary SQL writes therefore
usually observe those callbacks too, because the structural after-image is
materialized as the generated Rust entity before save validation.

That can make the two layers look unified even though they are not. Accepted
constraint activation, integrity verification, recovery replay, and any ingress
that cannot execute generated application behavior must remain independent of
Rust callbacks. Resumable SQL update already rejects entity graphs requiring
application callbacks rather than silently skipping them.

The result is a useful but potentially confusing combination:

1. resolve accepted defaults, generated values, managed values, and preserved
   values;
2. run applicable Rust sanitizers;
3. run Rust application validators;
4. construct the final canonical row;
5. evaluate accepted checks;
6. validate the remaining structural facts and stage the commit.

The ordering means accepted checks see sanitized values, which is desirable.
It also means that a Rust validator duplicating an accepted check can reject
first and hide the stable database-constraint diagnostic on ordinary writes.

## Working Vocabulary

| Concept | Working meaning |
| --- | --- |
| Entity model | The authored build-time graph: entities, records, enums, collections, Rust-facing behavior, and IcyDB directives |
| Model validation | Validation of the authored graph and macro declaration itself |
| Rust model projection | Generated Rust types, derives, visitors, application validation, and application normalization |
| Schema proposal | Generated, non-authoritative database description submitted to reconciliation |
| Accepted schema | Persisted database authority selected by reconciliation and DDL |
| Accepted runtime contract | Runtime projection derived only from accepted schema for planning, decoding, admission, integrity, and recovery |
| Row layout | Versioned persisted representation owned by accepted schema |
| Application preparation | Optional Rust validation/normalization outside database authority |

The intended authority graph is:

~~~text
authored entity model
    |---> generated Rust model projection
    |
    `---> IcyDB schema proposal
                |
                v
          reconciliation / DDL
                |
                v
          accepted schema
                |
                v
       accepted runtime contracts
                |
                v
      planning / admission / storage
~~~

The authored definition is the developer source, but it is not runtime
database authority. Generated Rust types do not reconstruct accepted schema.
The schema proposal is admitted input, not trusted state. Accepted runtime
contracts do not reopen the proposal to fill missing facts.

## Tentative Product And Package Boundary

The likely product description is:

> IcyDB includes an entity-model authoring system that generates Rust
> application types and compiles the database-compatible portion into an IcyDB
> schema proposal. Once accepted, the accepted schema—not the authored model or
> generated Rust projection—is database runtime authority.

Everything can remain in the IcyDB repository and under the `icydb-*`
namespace. Repository separation is not required to create a real dependency
and authority boundary.

Possible package names, not decisions:

| Package | Tentative responsibility |
| --- | --- |
| `icydb-model` | Authored entity-model AST, graph, declaration validation, and model-level metadata |
| `icydb-model-macros` | Type generation and Rust/application projections |
| IcyDB-specific proposal lowering | Stores, keys, indexes, relations, defaults, write policies, layouts, and schema proposals |
| `icydb-core` | Accepted catalog, runtime contracts, storage, planning, execution, integrity, and recovery |
| `icydb` | Public database facade and convenient model/macro re-exports |

The exact crate split remains open. Renaming alone is insufficient while the
model macro crate depends directly on engine-owned types and emits persistence,
query, and proposal implementations. The design must first inventory which
facts are generic model facts and which are IcyDB database facts.

One split is already non-negotiable:

~~~text
Rust construction default  -> Rust model projection
IcyDB insert default        -> schema proposal and accepted schema
historical fill             -> accepted row-layout authority
~~~

Those concepts must never collapse back into one ambiguous `default`.

## Initial Conclusion

The system should be unified at the authoring and developer-model level where
one declaration legitimately has multiple projections. It should be separated
at every authority boundary. In particular, it should not unify arbitrary Rust
execution or generated proposal metadata with durable database authority.

In particular:

- reserve `model` for authored and generated application concepts;
- reserve `schema` for database proposal, reconciliation, accepted authority,
  layout, and runtime contracts;
- prevent runtime owners from accepting proposal-only types where an accepted
  contract is required;
- replace dual-authority helper types with types whose construction proves
  whether they are proposal-only or accepted;
- authors should not need to declare the same representable rule once as a
  Rust validator and again as an accepted constraint;
- the accepted schema must remain the authority for durable invariants;
- arbitrary contextual Rust validation must remain application-only;
- arbitrary Rust sanitization must not become hidden database semantics; and
- database-owned transformation, if IcyDB wants it, needs its own bounded,
  deterministic, persisted representation.

Validation and transformation then become one important subproblem inside the
larger model/schema separation.

## Validation And Transformation Within The Separation

### 1. Accepted Constraint

An accepted constraint rejects an invalid final value or row. It is named,
persisted, deterministic, fingerprinted, introspectable, independently
evaluable, and enforced by every supported write ingress.

Conceptual generated declaration:

~~~rust
constraint(
    name = "balance_nonnegative",
    check = "balance >= 0",
)
~~~

The one declaration may also generate a convenient local Rust checker, but the
generated checker is only a mirror. The accepted expression remains database
authority.

An author should not normally need this duplicate:

~~~rust
validator(path = "base::validator::num::Gte", args(0))
~~~

If both are deliberately retained, documentation must explain that the
application validator can reject before the accepted constraint and therefore
win diagnostic precedence on callback-capable writes.

### 2. Accepted Normalizer Or Mutation Rewrite

An accepted normalizer transforms a value as database semantics. It would need
the same authority properties as an accepted check: bounded persisted form,
stable semantics, exact fingerprint participation, deterministic evaluation,
uniform ingress coverage, typed diagnostics, migration rules, and independent
execution without generated Rust code.

Possible vocabulary is `normalize`, `rewrite`, or `transform`. `Sanitizer` is
probably the wrong database term because it implies that a value becomes
generally safe rather than that one precise transformation was applied.

Conceptual syntax only:

~~~rust
normalize(
    field = "email",
    using = "lowercase(trim($value))",
)
~~~

Equivalent future SQL might be possible, but no syntax is proposed here.

The expression language should be much smaller than arbitrary Rust and may be
smaller than the query expression language. Every operation would need frozen
semantics. Unicode trimming, case conversion, decimal rounding, time access,
and locale-sensitive behavior cannot be admitted casually.

### 3. Application Validation And Preparation

Application validators and input preparation remain ordinary Rust behavior.
They may provide richer errors, inspect request context, consult application
state, or perform transformations that should happen before a database call.
They are not persisted, fingerprinted, used by integrity checking, or assumed
by recovery.

Clearer vocabulary may be useful while pre-1.0 hard cuts remain available:

~~~rust
application_validator(path = "can_caller_change_balance")
application_normalizer(path = "prepare_display_name")
~~~

Whether existing `validator(...)` and `sanitizer(...)` spellings should be
renamed is an open question. The important requirement is that their authority
boundary be unmistakable.

## What Comparable Systems Suggest

The common pattern is not to eliminate validation or transformation. It is to
make the authority and execution boundary explicit.

### PostgreSQL

PostgreSQL keeps database invariants in schema-owned `CHECK`, domain, unique,
primary-key, and foreign-key constraints. Reusable scalar restrictions can be
expressed as domains. Write-time computation or transformation uses explicit
features such as generated columns and triggers rather than application
validators being treated as constraints.

PostgreSQL also documents an important durability issue: check behavior is
assumed to be immutable. Changing a user function used by a check can leave
stored rows inconsistent until the constraint is dropped and re-added. That is
strong support for IcyDB persisting canonical semantics rather than callback
paths.

Sources:

- <https://www.postgresql.org/docs/current/ddl-constraints.html>
- <https://www.postgresql.org/docs/current/sql-createdomain.html>
- <https://www.postgresql.org/docs/current/ddl-generated-columns.html>
- <https://www.postgresql.org/docs/current/sql-createtrigger.html>

### SurrealDB

SurrealDB places validation and transformation beside each other but keeps
them as separate schema clauses. `ASSERT` validates a value. `VALUE` can
transform an authored value before it is stored, including a documented
lowercase-email example. Both are database-visible schema behavior.

This is close to the possible accepted-constraint plus accepted-normalizer
model, although IcyDB would still need its own stricter bounded and durable
semantics.

Sources:

- <https://surrealdb.com/docs/reference/query-language/statements/define/field>
- <https://surrealdb.com/docs/learn/schema-management/tables-and-fields/fields-and-validation>

### Gel

Gel exposes schema-defined mutation rewrites for insert and update. Rewrites
are explicit schema objects, can distinguish authored from preserved values,
and apply to DML through its SQL adapter. They are complementary to constraints
and triggers rather than disguised application hooks.

Sources:

- <https://docs.geldata.com/reference/datamodel/mutation_rewrites>
- <https://docs.geldata.com/reference/using/sql_adapter>

### MongoDB And Convex

MongoDB collection validators reject invalid inserted or updated documents but
do not turn arbitrary application cleanup into database validation. Convex
offers a useful authoring precedent: its declarative validator builder is used
for both function arguments and table schemas, while a pushed schema validates
existing documents and future writes.

Convex therefore demonstrates declaration reuse without making arbitrary
application functions the database authority.

Sources:

- <https://www.mongodb.com/docs/manual/core/schema-validation/>
- <https://docs.convex.dev/database/schemas>

### Ecto, Rails, And SeaORM

Application data layers retain validation and lifecycle preparation because
they are useful. Ecto explicitly distinguishes validations, which can run
without the database and provide early feedback, from database constraints,
which provide safe integrity enforcement. Rails and SeaORM expose lifecycle
callbacks before validation or save.

These systems also demonstrate the bypass risk: an application callback is
only as universal as the application write path that invokes it. IcyDB should
not describe such a callback as database authority merely because its normal
typed save path calls it.

Sources:

- <https://hexdocs.pm/ecto/Ecto.Changeset.html>
- <https://guides.rubyonrails.org/active_record_callbacks.html>
- <https://docs.rs/sea-orm/latest/sea_orm/entity/trait.ActiveModelBehavior.html>

## Classification Of Existing IcyDB-Style Rules

The eventual design should inventory every built-in validator and sanitizer.
The following is only an initial classification.

| Existing behavior | Likely durable owner | Notes |
| --- | --- | --- |
| Numeric minimum/maximum/range | Accepted constraint | Exact comparisons are directly representable |
| Text/blob/collection length | Accepted constraint | Requires frozen `CHAR_LENGTH`, octet-length, or cardinality semantics |
| Enum membership | Accepted type or constraint | Prefer the exact accepted enum catalog |
| Decimal scale | Accepted field kind where possible | Rejecting excess scale may be safer than silently rounding |
| Numeric multiple-of | Accepted constraint | Exact integer/decimal semantics only |
| Not-null | Accepted structural constraint | Must remain separate from SQL `CHECK` unknown semantics |
| Trim or fixed case conversion | Possible accepted normalizer | Only with exact frozen Unicode/ASCII semantics |
| Managed timestamps | Accepted write policy | Already closer to database-owned generation than sanitization |
| URL, MIME, phone, ISO, hash format | Undecided | Could require a larger frozen constraint library; otherwise application-only |
| Locale-sensitive title casing | Application preparation | Version and locale behavior make durable semantics risky |
| Decimal or money rounding | Prefer explicit application conversion | Lossy silent mutation needs exceptional justification |
| Caller authorization or external lookup | Application validator or access policy | Not a row-local deterministic constraint |
| Arbitrary custom Rust callback | Application-only | Must never be persisted by path or inferred as accepted semantics |

## Possible Database Write Shape

If accepted normalizers are pursued, a clean conceptual database pipeline could
be:

~~~text
resolve accepted schema and authored intent
    -> resolve defaults, generation, management, and preserved values
    -> apply accepted normalizers/rewrites
    -> construct the canonical after-image
    -> evaluate accepted checks
    -> validate structural and cross-row constraints
    -> stage and publish the commit
~~~

Application preparation would sit outside that authority:

~~~text
request or typed value
    -> optional Rust application normalization
    -> optional Rust application validation
    -> database mutation ingress
    -> accepted database pipeline
~~~

The high-level typed API could still offer both phases conveniently. The core
database contract must remain correct when only the accepted phase is
available.

## Questions That Need Answers Before A Design

### Which Existing Types Cross The Authority Boundary?

The design needs an explicit inventory of production uses of `EntityModel`,
`SchemaInfo`, generated model constants, proposal lowering, and accepted
runtime projections. Each use should be classified as:

- authored-model or macro validation;
- generated Rust convenience;
- schema proposal or reconciliation;
- accepted runtime planning or execution; or
- test-only model construction.

`SchemaInfo` is a particular warning sign because accepted and model-only
constructors feed one shared shape. The likely direction is to split it into
types whose constructors prove their authority instead of relying on callers
to remember which fields are trustworthy.

The same test applies to production uses of `E::MODEL`: proposal generation
and typed application adaptation may legitimately need it; accepted planning,
admission, integrity, and recovery must not.

### Where Should The Package Boundary Fall?

The dependency graph must be inspected before choosing crate names. The useful
boundary is not “generic” versus “database” by branding; it is authored model
facts versus IcyDB proposal facts versus accepted runtime authority.

Questions include:

- Can the authored-model AST exist without importing engine-owned schema and
  query types?
- Which macro outputs are ordinary Rust projection, and which are
  IcyDB-specific proposal adapters?
- Should proposal lowering live with the model macros, in a dedicated bridge,
  or behind an IcyDB-owned trait?
- Which public re-exports preserve a coherent IcyDB product without restoring
  the dependency cycle?

The answer may be a type split before a crate split. Moving the same ambiguous
types between packages would not solve the authority problem.

### Do We Need Accepted Normalization At All?

The smallest option is to retain accepted constraints and move Rust sanitizers
clearly outside the database executor. Applications would normalize input
explicitly before calling the database.

The larger option is a bounded accepted rewrite language. This improves parity
for conveniences such as trimming and lowercasing, but adds durable expression,
migration, activation, introspection, performance, and security obligations.

This is the primary decision. It should not be answered merely by preserving
the current callback behavior.

### What Is The Target Of A Normalizer?

- authored fields only;
- defaulted or generated fields;
- preserved fields on partial update;
- the whole final row;
- one field or multiple fields; and
- insert, update, replace, or an explicit subset?

Transforming an unassigned preserved value during an unrelated update is
surprising and should probably be forbidden unless a schema rewrite explicitly
says otherwise.

### Must Normalizers Be Idempotent?

Recovery should replay admitted bytes rather than rerun a normalizer. Even so,
idempotence may simplify retries, import, schema backfill, and reasoning. If it
is required, the system needs a way to prove or structurally guarantee it
rather than trusting documentation.

### How Are Semantics Frozen?

- Which string operations are ASCII-only or Unicode-aware?
- Can Unicode library upgrades change results?
- Is locale ever permitted?
- Are time, randomness, caller identity, or external state permitted?
- How are function identities encoded and fingerprinted?
- Can an operation be removed while accepted snapshots still reference it?

The default answer should be deterministic, row-local, bounded operations with
no external state. Time and identity belong in dedicated accepted write-policy
features rather than a generic function language.

### How Does Activation Work?

Adding a check can validate historical rows without rewriting them. Adding a
normalizer may imply that historical rows should be rewritten to a new
canonical value. That is a materially different migration operation involving
indexes, relations, uniqueness, journals, resumability, and conflict policy.

An accepted normalizer must not automatically become an unbounded hidden
backfill.

### What Does Local Rust Validation Mean?

A generated local checker derived from `constraint(...)` can improve early
feedback, but it evaluates the generated proposal rather than necessarily the
currently accepted runtime snapshot. It must be described as convenience, not
proof of database admission.

Potential choices include:

- generate no local checker and rely on database admission;
- generate a proposal-local checker for explicit use;
- let a session validate a value against the resolved accepted snapshot; or
- provide an application changeset/preparation API that reports both
  application and accepted failures without attempting a write.

### Which Diagnostic Wins?

If an application validator and accepted constraint reject the same value, the
current pipeline normally reports the application failure first. Alternatives
include:

- preserve application-first behavior;
- evaluate accepted constraints first once a canonical row exists;
- skip a duplicate generated application validator when it came from the same
  constraint declaration; or
- collect both classes in a bounded diagnostic result outside the write path.

The database write error should retain stable constraint identity whenever the
accepted constraint is the rule the caller needs to understand.

### Should SQL Invoke Application Callbacks?

Ordinary SQL currently reaches shared typed preflight and therefore invokes
callbacks where a generated entity is available. This needs an explicit
contract rather than remaining an implementation side effect.

Possible directions:

- SQL uses only accepted database semantics;
- SQL continues to use application callbacks but documents them as additional,
  non-authoritative behavior; or
- the public SQL facade optionally provides application preparation while the
  structural database ingress remains accepted-only.

Uniform behavior across SQL, structural, bulk, resumable, import, migration,
and future tooling favors keeping the core structural ingress accepted-only.

## Migration And Compatibility Thoughts

No existing validator or sanitizer should be silently promoted.

A possible migration process is:

1. inventory every built-in and project validator/sanitizer declaration;
2. classify it as an accepted constraint, accepted type fact, accepted write
   policy, possible accepted normalizer, or application-only behavior;
3. require an explicit stable name for every promoted database rule;
4. compare exact old and proposed behavior, including null and partial-update
   semantics;
5. remove duplicate application validators unless standalone validation still
   needs them;
6. activate or validate accepted rules using the normal accepted-schema
   lifecycle; and
7. hard-cut misleading callback spellings if a rename is approved before 1.0.

Automatic promotion based only on a Rust path and arguments would recreate the
authority problem identified by 0.211.

## Risks

- A general rewrite language could expand into triggers and duplicate much of
  the query executor.
- Invisible normalization can conceal bad input and make auditing harder.
- Lossy normalization can corrupt financial or identity data while appearing
  successful.
- Unicode or library-version drift can change persisted results across
  upgrades.
- Normalizing preserved fields can turn a narrow partial update into a broad
  mutation.
- Duplicate local and accepted evaluation can add Wasm size and write cost.
- Application-first errors can hide stable accepted constraint identities.
- Treating callback-capable SQL as proof of universal behavior can leave
  resumable, migration, integrity, or future import paths inconsistent.

## Likely Non-Goals

If this becomes a design, it should probably exclude:

- turning the entity-model authoring layer into a database-neutral product or
  moving it to a separate repository;
- renaming crates without first proving the dependency and authority split;
- changing a durable format merely to improve terminology;
- persisting Rust callback paths, function pointers, closures, or trait
  objects;
- arbitrary triggers or external calls;
- nondeterministic generic normalizers;
- locale-dependent transformations;
- automatic promotion of every validator or sanitizer;
- silent historical rewrites;
- compatibility shims for pre-1.0 callback semantics; and
- expanding 0.211 to implement normalization retrospectively;
- exact unsigned identity generation, which belongs to 0.214; and
- SQL structural coverage, range remediation, and performance work, which
  belong to 0.215.

## Evidence Needed Before Promotion To Design

- a production-use inventory for `EntityModel`, `E::MODEL`, `SchemaInfo`,
  proposal lowering, model-only planning, and accepted runtime projections;
- a crate dependency graph separating authored-model, generated Rust,
  proposal, accepted-schema, and executor ownership;
- a duplicate-fact matrix showing where defaults, constraints, layouts,
  indexes, relations, and write policies currently appear;
- compile-time or construction-level guards proving that accepted runtime
  owners cannot consume proposal-only data;
- an invocation map for Rust validators and sanitizers across every write
  ingress;
- a repository-wide validator and sanitizer inventory;
- a capability matrix against accepted checks and any candidate normalizer
  language;
- examples covering typed, SQL, structural, batch, resumable, migration,
  integrity, recovery, and import-like ingress;
- explicit authored/defaulted/generated/managed/preserved provenance rules;
- null, ordering, error-precedence, and idempotence decisions;
- a migration story for existing declarations;
- performance and raw non-gzipped Wasm budgets; and
- a source-complexity comparison between removing callbacks from database
  authority and adding accepted rewrites.

If the work is promoted into a minor-version design, implementation should be
planned as roughly six to eight meaningful landing patches, with a specific
closeout audit before any following minor begins. These notes intentionally do
not approve patches, but a plausible seven-slice review shape is:

1. inventory the authority crossings, settle vocabulary, and add boundary
   guards;
2. separate authored entity-model facts from IcyDB schema-proposal facts;
3. split dual-authority helpers such as accepted and model-only `SchemaInfo`;
4. harden accepted-only planning, admission, integrity, and recovery inputs;
5. separate structural database writes from typed Rust application adapters;
6. classify validators and sanitizers, remove duplicate declarations where the
   accepted rule is authoritative, and make any outer callbacks explicit; and
7. hard-cut obsolete surfaces, update examples and diagnostics, measure
   complexity/performance/Wasm impact, and perform the 0.213 closeout audit.

The slice boundaries should change if repository evidence shows a cleaner
dependency order. “Please continue” within one of these slices or within 0.213
does not authorize starting 0.214. Advancing requires an explicit closeout and
an explicit instruction to begin the next minor.

## Current Lean

The current lean is:

1. finish 0.211 with accepted constraints as the sole database validation
   authority;
2. complete 0.212 bounded resumable integrity checking against that accepted
   authority;
3. use 0.213 to make the entity-model, proposal, accepted-schema, runtime
   contract, and application-adapter boundaries structural in types and
   dependencies;
4. stop encouraging duplicate validator-plus-constraint declarations and make
   application callback and SQL behavior explicit;
5. do not add an accepted rewrite language unless the inventory demonstrates
   that explicit application preparation is insufficient; and
6. hand a clean accepted-runtime boundary to 0.214 identity generation and
   0.215 SQL structural coverage and range remediation.

0.213 is therefore an authority and data-model cleanup, not an accepted
normalization feature. Its most conservative successful result may include no
database rewrite language at all: clarify the boundary, move Rust sanitizers
out of core database authority, unify representable validation declarations
around accepted constraints, and retain application preparation as an
explicit outer layer.
