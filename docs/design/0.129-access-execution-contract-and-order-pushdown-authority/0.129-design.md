# 0.129 Access Execution Contract and Order-Pushdown Authority

## Thesis

`0.129` should stabilize the access execution-contract split that is now in
flight and make secondary-index `ORDER BY` pushdown eligibility a single
access-owned authority.

This is not a persisted-row follow-up.

It is not a broad query-planner cleanup.

It is not a prepared/lowering contraction line.

It is not a new SQL feature line.

It is an access-contract closeout line whose job is to make the current
access, route, cursor, explain, and executor consumers agree on one normalized
execution contract without re-deriving access policy in multiple layers.

The intended end state for `0.129` is:

* executable access contracts are the only runtime access mechanics consumed by
  cursor and executor code
* access capabilities are projected from executable access contracts, not
  recomputed by route or terminal consumers
* secondary-index `ORDER BY` pushdown eligibility is computed once and then
  projected to explain, trace, and runtime surfaces
* protective executor assertions remain separate from planner eligibility
  decisions
* stale transitional modules and dead compatibility helpers are gone

### Audit Alignment

The April 2026 audits point in two directions:

* prepared/lowering has real structural pressure and should become a future
  contraction target
* route capability derivation and planner/hint consumption are
  boundary-protected duplication and should not be blindly merged

That means `0.129` should not open the prepared/lowering contraction yet.
Doing that while the access-contract split is active would mix two structural
fronts.

Instead, `0.129` should finish the access work with a narrow rule:

> planner/access code owns eligibility and normalized execution mechanics;
> executor code owns fail-closed runtime assertions and physical execution.

The goal is to reduce drift without deleting protective boundaries.

## Problem

The access path from planning to execution still has too many adjacent shapes:

* logical access plans selected by query planning
* executable access contracts consumed by cursor and executor code
* route capabilities derived for load and aggregate planning
* secondary-order pushdown decisions consumed by explain and runtime routing
* executor assertions that protect physical stream assumptions

Those surfaces are all legitimate.

The problem is that they are not yet cleanly tiered.

If this line is left incomplete, future work can easily reintroduce:

* route consumers that infer access facts from raw path shape
* explain or trace code that derives pushdown status independently
* executor fast paths that inspect planner-level policy instead of executable
  mechanics
* stale execution-contract modules that look public enough to become new
  dependency targets

That is a velocity and correctness risk even when all current tests pass.

---

## Goal

Make executable access contracts the shared runtime access mechanics boundary,
and make secondary-order pushdown eligibility one access-owned decision.

`0.129` should deliver:

* a small, final-looking `db::access::execution_contract` export surface
* one route-facing capability projection over executable access contracts
* one order-pushdown eligibility model that distinguishes:
  * not applicable
  * applicable and eligible
  * applicable and rejected with a deterministic reason
* route hints and executor planning consuming capabilities rather than
  rebuilding access reasoning
* cursor and pagination behavior unchanged
* explain and trace surfaces reading from the same eligibility result as
  runtime planning

This should be done without:

* changing SQL semantics
* changing persisted data
* changing cursor token format
* merging planner policy with executor fail-closed assertions
* broad prepared/lowering cleanup
* broad aggregate execution redesign

---

## Core Invariant

Access policy is decided before execution, but execution still verifies the
physical assumptions it depends on.

Corollaries:

* access lowering may normalize logical access paths into executable contracts
* access capabilities may project executable-contract facts for route planning
* route and terminal code may branch on capabilities
* route and terminal code must not re-derive planner eligibility from raw
  access-path internals when a capability already exists
* executor code may still fail closed when a physical path receives a shape it
  cannot safely execute
* explain and trace code must project existing decisions instead of computing
  separate policy answers

This preserves the protective duplication identified by the audits while
removing accidental re-derivation.

## Proposed Shape

### Access Execution Contract

`db::access::execution_contract` should own normalized mechanics:

* executable access plan shape
* executable access node/path shape
* execution bounds
* execution path payload
* execution strategy metadata

It should not own:

* logical access-path selection
* route admission policy
* physical stream execution
* explain formatting

The module root should export only the items needed across the `db` boundary.
Deep module imports from consumers should be treated as transitional debt.

### Access Capabilities

`db::access::capabilities` should own capability projection from executable
contracts.

That includes questions like:

* what scan family this executable path represents
* whether it supports primary-key stream access
* whether it can use reverse traversal
* whether it can support count pushdown
* whether it can support bytes terminal fast paths
* which index-shape metadata route planning may consume

Route and terminal consumers should ask this owner for capability facts instead
of matching directly on executable path payloads when a capability already
exists.

### Secondary Order Pushdown

`db::access::order_pushdown` should own the shared eligibility DTOs:

* `PushdownApplicability`
* `SecondaryOrderPushdownEligibility`
* `SecondaryOrderPushdownRejection`
* `PushdownSurfaceEligibility`

The key shape is that "not applicable" is distinct from "applicable but
rejected".

That distinction prevents explain/runtime surfaces from losing information and
prevents `Option<...>` from becoming a policy channel.

### Route and Executor Consumers

Route planning may compute eligibility and attach capability-informed facts.

Executor and terminal code may consume those facts.

They should not reinterpret the original planner access path to answer the
same question again.

Runtime fail-closed checks still belong in executor code because they protect
physical assumptions at the boundary where unsafe execution would happen.

## Patch Sequence

### Opening Patch: Contract Stabilization

* finish the executable access-contract module split
* delete stale module references
* resolve transitional warnings
* keep cursor and executor behavior unchanged
* add or update owner-local tests only where the split changed observable
  access-contract construction

### Subsequent Patch: Order-Pushdown Authority

* make `order_pushdown.rs` the canonical DTO owner
* compute secondary-order pushdown applicability once
* project that result into explain and trace surfaces
* keep runtime route decisions aligned with the same result
* add regression coverage for eligible and rejected pushdown cases

### Subsequent Patch: Route Consumer Cleanup

* retarget load and aggregate route hints onto access capabilities
* remove small same-owner helper repetition only after the capability boundary
  is clear
* keep executor runtime assertions separate from route eligibility decisions

### Closeout Patch: Audit and Documentation

* update changelog entries
* record final validation
* document any intentionally retained protective duplication
* stop before prepared/lowering contraction begins

Patch numbers should be assigned only when the actual slices are created.

## Acceptance Criteria

`0.129` is complete when:

* no stale execution-contract modules or imports remain
* `db::access::execution_contract` has a small intentional export surface
* route/hint code consumes access capabilities instead of duplicating access
  shape reasoning where a capability exists
* secondary-order pushdown has one shared eligibility result consumed by
  runtime and surface projections
* explain/trace output and route behavior agree on eligible versus rejected
  pushdown state
* pagination, cursor resume, covering-read, and aggregate route tests remain
  unchanged in behavior
* any retained planner/executor duplication is explicitly classified as
  protective fail-closed separation

## Non-Goals for the Next Line

The prepared/lowering audits are real, but they should not be mixed into
`0.129`.

A later design line can target:

* prepared fallback contract inference
* session parameter rebinding
* duplicated prepared-template failure wording
* planner-owned prepared semantic outputs

order-pushdown authority are stable.

The point of `0.129` is to finish the access front before opening the prepared
front.
