# IcyDB 0.165 — Naming Audit And Role Alignment

## Status

Complete.

## Summary

0.165 is a complete naming audit and accepted-rename pass: only names that pass
role proof are changed.

The pass aligns architectural role names, helper verbs, and stale historical
vocabulary without changing behavior. It uses
`docs/architecture/NAMING.md` as the naming policy and checks every proposed
rename against idiomatic database and query-engine vocabulary before it lands.

This is not a style polish pass. It is a codebase-wide architecture readability
pass whose output should make ownership boundaries, planner/executor handoffs,
and public database concepts easier to read.

## Thesis

Names in IcyDB are part of the architecture.

They tell readers whether a value is:

- persisted schema/catalog authority
- query intent
- a planner-selected payload
- an admission proof
- a route decision
- a frozen fact snapshot
- a local traversal context
- an execution-ready payload
- an identity/cache-key surface
- runtime behavior semantics

Recent structural cleanup improved ownership, but some names still preserve
older layering. 0.165 closes that gap by aligning live names with current
architecture.

## Non-Negotiables

- No behavior changes.
- Complete hard cut for accepted internal renames: no compatibility shims,
  aliases, forwarding modules, duplicate helper names, or parallel old/new
  vocabulary before 1.0.
- No public rename unless the new name removes real ambiguity or likely misuse.
- No replacement of standard database vocabulary with local architecture jargon.
- No broad test-only renames unless they support a production concept rename.
- No archive-doc rewrite for historical prose; archive hits are classified.

## Hard-Cut Policy

0.165 is a naming hard cut, not a compatibility migration.

When a rename is accepted, the old internal name is removed from live code and
active docs in the same family-scoped slice. Do not leave:

- deprecated aliases
- compatibility wrappers
- forwarding modules
- duplicate old/new helper functions
- test fixtures that preserve the old vocabulary as if it were current
- active docs that teach both names

Remaining old-name hits after a family lands must be classified as archive
history, generated artifact text, conventional domain vocabulary, or stale
drift to fix. Unclassified live hits block that family from closing.

Public renames still require a higher bar and migration wording, but accepted
public renames are also hard cuts before 1.0 unless a separate design explicitly
approves compatibility.

## Database Idiom Alignment

0.165 must stay idiomatic for database and query-engine readers.

The pass should preserve conventional terms when they are the right domain word:

- `Catalog`, `Schema`, `Relation`, `Field`, `Index`, `Key`, `Constraint`
- `Predicate`, `Projection`, `Aggregate`, `Grouping`, `Ordering`, `Limit`,
  `Offset`
- `LogicalPlan`, `AccessPath`, `ExecutionPlan`, `Executor`
- `Cursor`, `Continuation`, `Page`, `Window`
- `Transaction`, `Commit`, `Snapshot`, `Replay`, `Recovery`

IcyDB role families are added only where they clarify ownership or phase:

- `*Contract` for admission/proof surfaces with invariants attached, not
  persisted SQL constraints and not plain parameter bundles
- `*Decision` for selected route or mode plus reason, not a boolean capability
- `*Facts` for reusable derived read-only snapshots consumed by multiple
  decisions
- `*Context` for owner-local short-lived traversal or input bundles that are not
  stored, cached, or returned as architectural outputs
- `*Identity` for equivalence, hashing, cache, and dedup authority
- `*Semantics` only for behavior that changes runtime or SQL-visible results

If a proposed rename makes the code less recognizable to someone familiar with
database internals, reject it unless the local architecture benefit is stronger
and documented.

## Ambiguous Stem Policy

Some stems are useful only when their meaning is narrow. They must not become
general-purpose escape hatches.

| Stem | Keep Only When It Means | Reject When It Hides |
| --- | --- | --- |
| `Shape` | Compact structural family, usually identity/cache relevant. | Full lowered payload, executable plan, or runtime bundle. |
| `Descriptor` | Renderable or observable description. | Execution instruction or planner-selected object. |
| `Parts` | Temporary construction or handoff decomposition. | Stable architectural type. |
| `Core` | Genuine invariant payload shared by wrappers. | Historical "stuff" bucket. |
| `Shared` | Shared ownership, generic-free shared shell, or cache-resident state, with the intended meaning stated by context. | Vague reuse or cloned payload with no ownership role. |

## Scope

### In Scope

- all live Rust crates
- active architecture, contract, governance, and design docs
- active changelog entries that describe the current slice
- public facade names, but only under a higher rename bar
- module/file names that own renamed concepts
- test names that lock renamed production concepts

### Archive Handling

Archived design docs and old changelog entries are scanned but not rewritten by
default. Remaining hits are classified as historical vocabulary unless they
point to active instructions or copied policy that should move forward.

Old changelog entries remain historical. The root `CHANGELOG.md` unreleased
section and any copied policy text that still acts as current guidance are
active surfaces and must use current vocabulary.

### Out Of Scope

- semantic or behavioral changes
- release/version number edits
- unrelated module restructuring
- low-value leaf helper churn
- renaming conventional database terms into generic local role names

## Method

0.165 proceeds by family, not by random symbol.

For each family:

1. Inventory live names and old stems with `rg`.
2. Classify each candidate by database concept and IcyDB role family.
3. Record a role proof before keeping or renaming:
   - current name
   - owning module
   - fields or return payload
   - main consumers
   - chosen name family
   - rejected alternatives
   - public-surface impact
4. For accepted renames, rename the whole concept through types, modules,
   helpers, imports, tests, comments, examples, and active docs.
5. Scan old vocabulary and classify remaining hits.
6. Run focused checks for the touched family.

## Deliverables Per Family

Each audit family produces:

- accepted renames, with role proof
- kept names, with rejected alternatives where relevant
- deferred candidates, with reason
- old-vocabulary scan terms
- residual-hit classification
- focused check command output

A family may complete with no renames if the audit proves the current names are
correct.

## Review Discipline

Renames land in small, family-scoped changes.

Cross-family opportunistic renames are deferred unless they unblock the current
family. Pure mechanical renames should not be mixed with behavior edits,
unrelated module restructuring, or opportunistic cleanup. If a rename requires
file movement or public facade impact, the role proof must explain why that
movement is part of the concept rename.

Each accepted rename should preserve bisectability: the repository should
compile and focused checks should pass after each family lands.

Hard cut does not mean giant patch. It means each accepted rename slice removes
its own old vocabulary completely before moving to the next family.

- database/domain terms
- IcyDB role families
- helper verb meanings
- public rename discipline

- Is this a conventional access-path concept?

### 6. SQL Lowering And Query Shape

Audit lowered SQL payloads, expression analysis, aggregate descriptors, and

Current family notes: `sql-lowering-query-shape.md`.

Primary questions:

- Is this lowered query intent, a structural shape, a selected plan, or an
  execution descriptor?
- Does `Shape` mean a compact structural family, or is it hiding a phase
  payload?
- Does `Descriptor` describe observability/rendering, or should it be a plan or
  kind?

Likely high-value names to prove before keeping or renaming:

- `LoweredSelectShape`
- `LoweredBaseQueryShape`
- `LoweredExprAnalysis`
- `LoweredSqlAggregateShape`
- `PreparedSqlScalarAggregatePlanFragment`

### 7. Identity Versus Semantics

Audit every live surface where `semantic`, `semantics`, `identity`,
`fingerprint`, `hash`, `cache`, `dedup`, or `canonical` appear near each other.

Current family notes: `identity-versus-semantics.md`.

Primary rule:

- `Identity` means equivalence, hashing, cache-key, or dedup authority.
- `Semantics` means behavior that changes runtime or SQL-visible results.

Likely zones:

- query-plan identity
- prepared-query cache keys
- aggregate planning and lowering
- projection deduplication
- tests for equality, hashing, or reuse

### 8. Helper Verbs

Audit `classify_*`, `analyze_*`, `derive_*`, `resolve_*`, `prepare_*`, and
`canonicalize_*` across live code.

Current family notes: `helper-verbs.md`.

Primary rules:

- `classify_*` returns compact categories.
- `analyze_*` returns richer descriptive results.
- `derive_*` computes new facts or outputs.
- `resolve_*` maps to an already-existing symbol, slot, path, or component.
- `prepare_*` freezes reusable state for a later phase.
- `canonicalize_*` normalizes to an equivalent stable form.

The goal is not to rename every helper. The goal is to remove misleading verbs
on shared architectural surfaces.

### 9. Public Surface Audit

Audit public names last and under a higher bar.

Public names should remain stable unless the current name is materially wrong,
misleading, or inconsistent with conventional database vocabulary.

Names to audit carefully:

- `QueryResponse`
- `ProjectionResponse`
- `MutationResult`
- `MutationMode`
- `SqlProjectionRows`
- `SqlQueryRowsOutput`
- `SqlGroupedRowsOutput`
- `SqlQueryResult`
- `QueryExecutionAttribution`
- `SqlQueryExecutionAttribution`
- `ExplainExecutionDescriptor`
- `ExplainPropertyMap`

Candidate public renames are not accepted by default. Each one needs a public
impact note and a reason the new name is better for database users, not only
for internal consistency.

Any accepted public rename must include a changelog note and migration wording,
even before 1.0.

Current family notes:

- [public-surface.md](public-surface.md)

### 10. Residual Sweep

After family passes, run a whole-repo scan for stale vocabulary in live code
and active docs.

Minimum scans:

```bash
rg -n "OldName|old_name|old-name|old concept phrase" crates docs CHANGELOG.md
rg -n "FamilyOldName|FamilyOldStem|FamilyOldPhrase" crates docs CHANGELOG.md
rg -n "Plan|Contract|Decision|Facts|Context|Shape|Identity|Semantics|Analysis" crates docs
rg -n "classify_|analyze_|derive_|resolve_|prepare_|canonicalize_" crates
```

Each family must maintain its own concrete old-vocabulary scan list. Generic
suffix and helper-verb scans are secondary quality checks, not substitutes for
family-specific stale-name sweeps.

Remaining hits must be classified as:

- intentional current vocabulary
- conventional database vocabulary
- archive/history
- generated artifact text
- stale drift to fix

Current family notes:

- [residual-sweep.md](residual-sweep.md)

## Completion Criteria

0.165 is complete when:

- `docs/architecture/NAMING.md` is the active naming authority
- live code has been audited across every family above
- accepted renames are applied through modules, imports, helpers, tests, and
  active docs
- accepted internal renames leave no live aliases, shims, forwarding modules,
  or parallel old/new helper vocabulary
- kept names have explicit reasons for high-value candidates
- deferred names have explicit triggers or owner milestones
- public-surface names have explicit keep/rename decisions
- helper verbs are no longer misleading on shared architecture surfaces
- `Identity` and `Semantics` are separated according to role
- conventional database terms are preserved where they are the better name
- old vocabulary scans have no unclassified live hits
- focused checks pass for all touched families
- final whole-workspace checks pass after all accepted renames land

## Expected Payoff

The pass should leave the codebase easier to review:

- database concepts stay recognizable
- IcyDB-specific phase artifacts advertise their ownership role
- route, covering, prepared execution, continuation, lowering, and public
  response names stop mixing historical near-synonyms
- future feature work has one naming policy to extend

The result should feel less clever, not more: standard database terms where
they fit, precise IcyDB role names where the architecture needs them.
