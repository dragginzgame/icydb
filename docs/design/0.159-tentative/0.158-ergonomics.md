# IcyDB Addendum — Planner Transparency, Index Guarantees, and Access-Path Contracts

## Status

Proposed

## Summary

IcyDB’s current query model is architecturally correct:

* query APIs express logical predicates
* the planner selects physical access paths
* composite secondary indexes are matched against canonicalized predicate sets
* hot paths can already plan to `Index` / `ByKey` rather than `FullScan`

However, recent developer feedback exposed an ergonomics and observability gap:

> Developers visually associate `.filter(...)` with post-scan filtering, even when the planner is performing indexed access.

This addendum proposes improvements focused on:

* planner transparency
* explicit access-path guarantees
* regression safety
* index observability
* developer confidence
* index quality analysis

The goal is **not** to expose physical storage internals directly in normal query APIs. The goal is to make planner behavior inspectable, enforceable, and testable.

---

# Problem Statement

Today, this query:

```rust
.load::<MovementIntent>()
.filter(session_id.eq(id))
.filter(turn_number.eq(turn))
.filter(status.eq(Open))
```

may correctly plan to a bounded composite index scan.

But visually, many developers interpret:

```rust
.filter(...)
```

as:

```text
scan everything, then filter in memory
```

even when that is not true.

This creates several problems:

* reduced trust in planner behavior
* uncertainty about hot-path safety
* reliance on verbal/documentation explanations
* difficulty detecting planner regressions
* inability to enforce “must use index” contracts
* lack of visibility into why a plan was selected

---

# Design Goals

## Goals

* Make planner-selected access paths highly visible.
* Allow hot paths to assert index usage contracts.
* Improve explain-plan diagnostics.
* Detect planner regressions early.
* Surface index-prefix matching behavior explicitly.
* Provide tooling for index quality and redundancy analysis.
* Preserve logical query APIs (`filter`, predicates, expression trees).
* Preserve planner ownership of physical execution decisions.

## Non-Goals

* No manual index hints as the primary API model.
* No mandatory index selection by application code.
* No exposure of low-level storage engine internals in normal query flow.
* No abandonment of predicate-first planning.
* No optimizer destabilization for cosmetic API changes.

---

# Current Architecture

Current behavior is already structurally correct:

```text
Logical Predicates
    ↓
Canonical Predicate Set
    ↓
Planner
    ↓
Index Prefix Matching
    ↓
Physical Access Path
```

The planner already supports:

* composite secondary indexes
* index-prefix matching
* keyed lookups
* bounded range scans
* fallback to `FullScan`

Hot paths are already validated through `explain()`-based tests.

The missing piece is explicit developer-facing visibility and enforceability.

# Proposed Improvements

---

# 1. Expanded Explain Plans

## Problem

Current explain output is insufficiently explicit about:

* why an index was chosen
* which predicates bound the index
* which prefix matched
* whether the scan is exact or ranged
* whether fallback occurred

## Proposal

Expand explain output to expose planner reasoning.

Example:

```text
Plan:
  Access Path: SecondaryIndexScan
  Entity: MovementIntent

  Index:
    movement_intent(session_id, turn_number, status)

  Bound Prefix:
    session_id
    turn_number
    status

  Predicate Shape:
    Equality
    Equality
    Equality

  Estimated Rows:
    4

  Residual Filters:
    None
```

Range example:

```text
Bound Prefix:
  session_id
  turn_number

Range Predicate:
  turn_number > 100
```

Fallback example:

```text
Access Path: FullScan

Reason:
  No usable index prefix matched predicates
```

## Benefits

* makes planner behavior self-explanatory
* reduces developer confusion
* improves debugging
* improves performance auditing
* enables regression assertions

---

# 2. Explicit Index Guarantees

## Problem

Hot paths currently rely on planner correctness implicitly.

A planner regression may silently degrade:

```text
IndexScan → FullScan
```

without a hard failure.

## Proposal

Introduce optional access-path contracts.

---

## `require_index()`

```rust
.load::<MovementIntent>()
.filter(...)
.require_index()
```

Behavior:

* planning fails if no usable index path exists
* query execution aborts before runtime scan fallback

---

## `require_index_named()`

```rust
.require_index_named("movement_intent_session_turn_status")
```

Behavior:

* planner must use the specified index
* fail if planner selects another path

Useful for:

* deterministic hot paths
* perf-sensitive canister operations
* regression testing

---

## `require_access_path()`

```rust
.require_access_path(AccessPath::ByKey)
```

Allows asserting:

* `ByKey`
* `IndexScan`
* `RangeScan`
* etc.

---

# 3. Planner Regression Assertions

## Problem

Tests currently only assert broad planner outcomes.

Example:

```text
not FullScan
```

This may still permit degraded plans.

## Proposal

Add structured planner assertions.

Example:

```rust
assert_plan(query)
    .uses_index("movement_intent_session_turn_status")
    .bound_prefix_len(3)
    .access_path(IndexScan)
    .has_no_residual_filters();
```

---

# 4. Index Prefix Diagnostics

## Problem

Composite index usability is often misunderstood.

Example:

```text
(session_id, turn_number, status)
```

supports:

```text
session_id
session_id + turn_number
session_id + turn_number + status
```

but not arbitrary non-prefix combinations.

This behavior should be surfaced explicitly.

## Proposal

Expose planner diagnostics such as:

```text
Rejected Index:
  movement_intent(session_id, turn_number, status)

Reason:
  Missing leading prefix field: session_id
```

or:

```text
Trailing fields unused after range predicate:
  status
```

---

# 5. Residual Filter Visibility

## Problem

Some predicates may execute partially through index narrowing and partially as residual evaluation.

This is currently opaque.

## Proposal

Explain plans should distinguish:

```text
Index-Bound Predicates
Residual Predicates
```

Example:

```text
Index-Bound:
  session_id = ?
  turn_number > ?

Residual:
  expensive_expression(foo)
```

This improves:

* planner understanding
* perf debugging
* optimizer auditing

---

# 6. Index Analysis Tooling

## Problem

Schemas may accumulate:

* redundant indexes
* low-selectivity indexes
* overly wide indexes
* unused indexes
* overlapping prefixes

without visibility.

## Proposal

Add schema analysis tooling.

---

## Redundancy Analysis

Detect:

```text
(session_id)
(session_id, status)
```

where the narrower index may be redundant.

---

## Prefix Coverage Analysis

Show:

```text
Query Shapes Covered:
  session_id
  session_id + status

Missing Coverage:
  status + turn_number
```

---

## Usage Statistics

Potential future runtime metrics:

```text
Index Usage Count
Average Rows Scanned
Average Selectivity
Fallback Frequency
```

---

# 7. Planner Warning System

## Proposal

Allow optional warnings for suspicious plans.

Examples:

```text
Warning:
  Composite index matched only first field
```

```text
Warning:
  FullScan on entity with 1.2M rows
```

```text
Warning:
  Range predicate prevents trailing field narrowing
```

Warnings should be:

* optional
* non-fatal
* testable

---

# 8. Range Predicate Awareness

## Problem

Composite index behavior changes substantially once range predicates appear.

Example:

```rust
.filter(session_id.eq(...))
.filter(turn_number.gt(...))
.filter(status.eq(...))
```

Trailing fields may become non-selective after the range boundary.

## Proposal

Make planner handling explicit in explain output.

Example:

```text
Index Prefix:
  session_id

Range Boundary:
  turn_number > 100

Trailing Fields Not Used:
  status
```

This improves developer understanding without exposing engine internals unnecessarily.

---

# 9. Preserve Predicate-First Architecture

## Important Architectural Constraint

These improvements must NOT invert architectural ownership.

The correct layering remains:

```text
logical predicates
    →
planner
    →
physical access path
```

Application code should continue expressing:

* logical intent
* not storage traversal instructions

The planner remains responsible for:

* access-path selection
* index matching
* fallback policy
* optimization

The new APIs are:

* diagnostics
* guarantees
* assertions
* enforcement contracts

—not primary query construction primitives.

These changes should provide:

* stronger developer trust
* easier planner debugging
* safer hot-path guarantees
* better performance regression detection
* clearer index semantics
* reduced confusion around `.filter(...)`
* improved schema/index hygiene

without compromising IcyDB’s existing architectural separation between:

* logical query expression
* and physical execution planning.
