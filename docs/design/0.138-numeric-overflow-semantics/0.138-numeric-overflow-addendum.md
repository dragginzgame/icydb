# 0.138 Design Addendum: Numeric Overflow Semantics

## Status

Proposed post-`0.138` follow-up.

## Motivation

Current numeric behavior in IcyDB is inconsistent with standard SQL semantics:

* Decimal arithmetic saturates, for example to `i128::MAX`.
* Integer overflow behavior depends on underlying Rust primitives.
* Expression evaluation may silently produce incorrect values instead of
  failing.

This creates correctness risks:

* Different large values can collapse to the same saturated value.
* `ORDER BY`, `GROUP BY`, and comparisons can become incorrect.
* Aggregate results may be corrupted.
* Constant folding may diverge from runtime behavior.

Industry-standard databases, including PostgreSQL, SQL Server, and MySQL in
strict mode, treat overflow as a query error for exact numeric types.

## Goal

Introduce a SQL-consistent numeric overflow model without destabilizing
existing low-level numeric implementations.

Key objective:

> Unify overflow handling at the SQL evaluation layer, not the primitive type
> layer.

## Design Overview

Introduce a three-layer model:

```text
Types Layer (existing)
    - Decimal, integers
    - may remain saturating or wrapping

Numeric Evaluation Layer (new/extended)
    - checked operations
    - returns NumericEvalError

SQL Execution Layer
    - maps NumericEvalError -> QueryError
```

This preserves performance and stability while enforcing correct SQL
semantics.

## NumericEvalError

Introduce a small internal error type:

```rust
enum NumericEvalError {
    Overflow,
    NotRepresentable,
}
```

Usage:

* Returned by checked numeric operations.
* Never exposed directly to users.
* Mapped to `QueryError` at the SQL boundary.

## Checked Numeric Operations

Add checked variants for numeric operations used in SQL evaluation.

### Decimal

* Add `checked_*` operations, starting with `checked_powu`.
* Do not remove existing saturating behavior yet.

Example:

```rust
fn checked_powu(...) -> Result<Decimal, NumericEvalError>
```

### Integer

Use Rust checked operations:

* `checked_add`
* `checked_sub`
* `checked_mul`
* `checked_pow`, if applicable

Map `None` to `NumericEvalError::Overflow`.

## SQL Semantics

For all exact numeric types, including integer and decimal:

| Condition         | Result                    |
| ----------------- | ------------------------- |
| Valid computation | Value                     |
| Overflow          | `ERROR: numeric overflow` |
| Not representable | Error                     |

No silent saturation. No wrapping. No implicit truncation.

Floating-point behavior is future work:

* It may return `inf` or `NaN`.
* It must be explicitly defined before becoming the SQL contract.

## Integration Points

### 1. SQL Numeric Functions

Example:

* `decimal_power()`

Change from returning `Option<Decimal>` to returning
`Result<Decimal, NumericEvalError>`.

### 2. Projection Evaluation

In `projection_eval.rs`, map:

```rust
NumericEvalError::Overflow
    -> QueryError::numeric_overflow
```

### 3. Aggregates

All aggregate accumulation paths must use checked numeric operations:

* `SUM`
* `AVG`
* `MIN` and `MAX`, if coercion is involved

Overflow during aggregation immediately returns `QueryError` and does not
silently clamp.

### 4. Constant Folding

Critical invariant:

> Compile-time folding must match runtime behavior.

Rules:

* Overflow during folding must not be treated as "cannot fold".
* Either propagate an error during planning or disable folding for that
  expression with identical runtime behavior.

Never allow silent skip of folding plus runtime error mismatch.

## COUNT Semantics

`COUNT` is already unified via `reducer_core::finalize_count`.

Overflow considerations:

* Internal storage may remain `u32` or `u64`.
* Finalization always uses the shared helper.
* If future overflow becomes possible, treat it as
  `NumericEvalError::Overflow`.

## DISTINCT

DISTINCT is out of scope for this addendum.

DISTINCT currently has divergent implementations:

* Scalar: `Vec<Value>` plus linear scan.
* Grouped: canonical key sets plus budgets.

Future work must define:

* canonical equality semantics
* memory model
* budget behavior

## Non-Goals

* Do not rewrite `Decimal` to be globally checked.
* Do not introduce a trait-based numeric abstraction.
* Do not unify scalar and grouped execution paths.
* Do not change performance characteristics of hot loops.

## Migration Strategy

1. Introduce checked operations alongside existing operations.
2. Switch SQL evaluation paths to checked variants.
3. Keep low-level saturating behavior unchanged.
4. Gradually expand coverage across arithmetic, functions, and aggregates.

## Expected Outcome

After implementation:

* Numeric overflow is explicit and deterministic.
* There is no silent data corruption from saturation or wrapping.
* SQL behavior aligns with industry standards.
* Execution and performance architecture remains stable.

## Summary

This addendum formalizes a clean separation:

* Types remain fast and permissive.
* Numeric evaluation becomes safe and explicit.
* SQL layer enforces correctness.

Result:

> Correct-by-construction numeric semantics without destabilizing the engine.
