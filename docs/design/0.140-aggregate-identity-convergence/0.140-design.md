# 0.140 Aggregate Identity Convergence

## Status

Proposed

## Motivation

`0.139` introduced a sealed semantic model for SQL global aggregates:

```text
Terminal (syntax)
  -> AggregateSemantics (meaning)
  -> Strategy (execution)
  -> Executor
```

This established:

* a single semantic authority for SQL global aggregates
* DISTINCT behavior encoded in types
* a clean separation between syntax, meaning, and execution

However, grouped aggregates still operate on a separate semantic surface:

* grouped specs carry raw flags (for example, `distinct: bool`)
* normalization is applied through helper methods
* DISTINCT behavior is partially duplicated across planner and execution code

At the same time, the system already contains a shared semantic core:

```text
AggregateIdentity
```

which:

* normalizes aggregate kind and input identity
* encodes observable DISTINCT behavior
* ensures `MIN(DISTINCT x)` is equivalent to `MIN(x)`

The current system is correct, but:

> The shared semantic core exists implicitly rather than being consistently enforced across aggregate systems.

---

## Goal

Make `AggregateIdentity` the **explicit shared semantic core** for aggregate
identity across:

* SQL global aggregates
* grouped aggregate planning and execution

while preserving:

* the separation between SQL lowering and grouped planning
* existing execution behavior and planner rules

---

## Non-Goals

Do not:

* unify grouped aggregates under SQL global aggregate semantics
* change SQL results or behavior
* change grouped execution behavior
* modify DISTINCT eligibility rules in grouped planning
* change executor reducer behavior
* alter projection remap or ordering
* refactor query planning architecture

This is a **semantic convergence**, not a system rewrite.

---

## Design

### 1. Promote `AggregateIdentity` as the Shared Core

`AggregateIdentity` becomes the canonical representation of aggregate meaning:

* aggregate kind
* input identity
* observable DISTINCT behavior

### Properties

* DISTINCT normalization is owned **only by `AggregateIdentity`**
* `MIN` / `MAX` do not carry DISTINCT
* `COUNT` / `SUM` / `AVG` retain DISTINCT
* identity reflects observable results, not syntax

---

### 2. Define Aggregate Semantic Key

Aggregate equivalence for deduplication and comparison must include `FILTER`:

```text
AggregateSemanticKey = AggregateIdentity + filter_expr
```

### Rules

* `AggregateIdentity` excludes `FILTER`
* `FILTER` remains a separate semantic dimension
* aggregates with different filters must not dedupe

Example:

```sql
SUM(x) FILTER (WHERE active)
SUM(x) FILTER (WHERE archived)
```

These must remain distinct.

---

### 3. Position SQL Global Semantics as a Wrapper

SQL global aggregate semantics becomes a SQL-lowering projection over the shared
identity core:

```text
SqlGlobalAggregateTerminal
  -> AggregateTerminalSemantics {
        identity: AggregateIdentity,
        filter_expr,
    }
  -> PreparedAggregateSemantics
```

Responsibilities:

* convert terminal syntax to `AggregateIdentity`
* preserve filter expression
* bind model-specific target information in `PreparedAggregateSemantics`
* perform no independent DISTINCT normalization

### Important

> SQL global aggregate semantics does NOT own DISTINCT normalization.
> It delegates entirely to `AggregateIdentity`.

---

### 4. Align Grouped Aggregates with Identity

Grouped aggregate specs should expose:

```text
GroupAggregateSpec / GroupedAggregateExecutionSpec {
    identity: AggregateIdentity,
    filter_expr,
    execution attachments...
}
```

### Rules

* semantic decisions must use `AggregateIdentity`
* raw DISTINCT flags may exist only before identity construction
* grouped specs must not rely on raw flags for semantic meaning

---

### 5. Preserve Grouped Execution Ownership

Grouped aggregates continue to own:

* planner validation rules
* DISTINCT eligibility constraints
* compiled expressions
* per-group execution configuration
* runtime reducer behavior

These are not part of semantic identity and remain unchanged.

---

### 6. Maintain Separation of Concerns

Final structure:

```text
                AggregateIdentity
                 (shared core)
                  /       \
                 /         \
   SQL Global Semantics   GroupedAggregateSpec
      (SQL global)         (planner/execution)
              \               /
               \             /
                Strategy / Execution
```

---

## Ownership Rules

### `query/plan/semantics/identity.rs`

Owns:

* aggregate identity
* DISTINCT normalization
* equality semantics

### `sql/lowering/aggregate/semantics.rs`

Owns:

* SQL terminal to identity conversion
* SQL global prepared semantic wrappers
* no semantic normalization authority

### `query/plan/group.rs` and Related Modules

Own:

* grouped execution specs
* planner rules
* runtime configuration

Must not:

* duplicate DISTINCT normalization logic
* reinterpret aggregate meaning independently

---

## Migration Plan

1. Audit grouped specs for raw DISTINCT plus normalization patterns.
2. Ensure `AggregateIdentity` is constructed and exposed early.
3. Replace normalization helpers with identity-based logic.
4. Update grouped dedup/equality to use:

   ```text
   AggregateIdentity + filter_expr
   ```

5. Remove redundant normalization paths where safe.

## Expected Outcome

After this slice:

* aggregate identity is defined in exactly one place
* DISTINCT normalization cannot drift across systems
* SQL global and grouped aggregates share identity rules
* `FILTER` remains correctly scoped as part of semantic equivalence
* grouped specs expose identity instead of raw flags for semantics

The system retains:

* clean layering
* domain separation
* stable behavior

while improving:

* consistency
* correctness guarantees
* maintainability

---

## Scope Clarification

This slice introduces:

> **shared semantic identity with domain-specific projections**

It does not unify aggregate systems.

---

## Future Work

Out of scope:

* deeper type sealing in grouped specs
* executor-level identity reuse
* unified aggregate planning interfaces
* cross-system semantic abstraction layers

---

## Summary

`0.140` completes the semantic alignment started in `0.139` by:

* promoting `AggregateIdentity` to the shared core
* removing duplicated normalization logic
* preserving domain-specific behavior

This results in:

```text
One identity -> multiple projections -> consistent execution
```
