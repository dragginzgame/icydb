# 0.208 Exact Composite Contracts And Explicit AnyValue

Status: proposed; design cut on 2026-07-16; implementation starts after 0.207
closeout

## Decision Summary

IcyDB remains always strict. Every persisted top-level slot belongs to the
accepted row layout, and every accepted field has one exact persisted kind and
one explicit absence policy. Non-queryable fields are allowed. Undeclared
slots and shape-changing typed fields are not.

The current `Structured { queryable }` kind violates that rule. It tells the
planner whether a value is queryable, but it does not tell accepted-schema
admission whether the value is a record, tuple, or newtype, which fields or
elements it contains, or which nested kinds are legal. Admission therefore
uses the same open-ended recursive value walk for ordinary typed composites
that an explicit heterogeneous field would need.

0.208 makes two sequenced changes:

1. replace generic `Structured` with nominal, exact composite contracts owned
   by accepted schema; and
2. add `AnyValue` as the only explicit heterogeneous persisted field kind.

`AnyValue` is not a non-strict mode. It has a fixed schema identity, a fixed
versioned codec, a required encoded-byte bound, a fixed depth bound, explicit
nullability, and no implicit query capabilities. Its contents may vary in
kind, but its outer storage contract may not.

The order is mandatory. The exact-composite slice closes the implicit escape
hatch before the explicit one is opened. There is never a release state in
which both generic `Structured` and `AnyValue` are accepted persisted kinds.

### Non-Negotiable Invariants

1. Every physical user slot is either an accepted field slot or an explicit
   accepted retired-slot placeholder.
2. Every non-`AnyValue` field is admitted against one exact recursive accepted
   kind.
3. A record accepts exactly its declared field names; a tuple accepts exactly
   its declared arity; a newtype accepts exactly its declared inner kind.
4. Generated Rust materialization may confirm an accepted contract, but it may
   not supply missing runtime schema authority.
5. Heterogeneous persisted content is legal only in an explicitly declared
   `AnyValue` field.
6. `AnyValue` preserves admitted value tags without target-type coercion.
7. `AnyValue` cannot be indexed, compared, ordered, grouped, aggregated,
   related, or addressed by nested path in 0.208.
8. Every logical write ingress either performs accepted-schema admission or
   rejects the operation before the commit marker.
9. Persisted decoding is bounded, fallible, current-form only, and never
   reconstructs accepted shape from generated models.

## Normative Strictness Contract

The maintained contract is:

> Every accepted persisted field has one exact field kind, one physical slot,
> one explicit absence policy, and one accepted codec identity. Every logical
> write path must produce a complete value admitted by that contract or reject
> before the commit marker. IcyDB has no non-strict entity or table mode.

For composites:

> The accepted schema owns the complete nominal composite type and recursive
> shape. Queryable nested leaves are derived views of that shape, not a
> substitute for it.

For heterogeneous data:

> Heterogeneous content is permitted only inside an explicitly declared
> `AnyValue` field. The field still has a fixed versioned codec, encoded-byte
> bound, depth bound, schema identity, absence policy, and capability set.

For layout:

> Non-queryable persisted fields are allowed. Undeclared persisted slots and
> shape-changing typed fields are not.

Ephemeral planner values, aggregate results, projections, and other runtime
values are outside the persisted-field rule. They do not authorize a database
row to carry an undeclared slot or an untyped value in an exact field.

## Problem Statement

### 1. Ordinary Composites Currently Use An Implicit ANY Contract

`Structured { queryable: false }` accepts any canonical scalar, enum, list, or
map shape that passes the global recursive budget. The accepted kind does not
state that one field is specifically `example::Address`, a record with exactly
`street`, `city`, and `postal_code`, or a tuple with exactly three elements.

That makes an ordinary typed record depend on a later generated decoder for a
guarantee the accepted schema should own.

### 2. Queryability Is Not Type Identity

The `queryable` boolean answers whether predicates may target a field. It does
not answer what bytes are legal in that field. Combining capability and shape
in `Structured { queryable }` makes a non-queryable exact record look
equivalent to an intentionally heterogeneous payload.

The concepts must be separate:

- accepted kind and codec determine which values may persist;
- capability metadata determines which operations may consume those values.

### 3. Nested Leaves Are An Incomplete Structural Contract

Nested leaf metadata is intentionally projection-oriented. It can describe
queryable paths without assigning each leaf a row slot, but it does not
necessarily enumerate non-queryable members, tuple positions, nominal
newtypes, empty composites, or the complete recursive type graph.

Promoting nested leaves into shape authority would overload a useful derived
view and still leave structural gaps. The accepted composite definition must
be complete first; nested leaves are then derived from it.

### 4. Global Bounds Do Not Replace A Field Bound

The existing 4 MiB value budget and depth limit prevent unbounded recursive
admission work. They do not state the storage contract of an individual field.
A deliberately heterogeneous field needs its own positive encoded-byte bound
so schema review, row planning, defaults, imports, and resource accounting can
reason about it.

### 5. Current Documentation Permits Undeclared Persisted Values

`docs/FOUNDATIONS.md` currently says entities may carry values that are not
part of the schema. That wording conflicts with accepted row-layout authority
and the normative write-admission contract. Ephemeral runtime values may exist
outside persisted schema. Persisted top-level slots may not.

## Goals

0.208 must:

1. make accepted schema own exact record, tuple, and newtype identity and
   recursive shape;
2. remove generic `Structured` as an accepted and generated field kind;
3. remove open-ended untyped admission from every ordinary composite path;
4. retain generated structured codecs as typed encode/decode owners without
   making them runtime schema fallbacks;
5. retain nested field paths as derived query and index metadata;
6. add one explicit public `AnyValue` field type for intentionally
   heterogeneous content;
7. require every `AnyValue` declaration to choose a positive encoded-byte
   bound no larger than the engine global cap;
8. use one fixed, versioned, canonical, bounded `AnyValue` codec;
9. preserve value tags without implicit target-kind coercion;
10. fence all comparison, ordering, grouping, aggregate, index, unique,
    primary-key, relation, and nested-path capabilities for `AnyValue`;
11. route every logical write ingress through accepted-schema admission or an
    explicit unsupported-operation rejection;
12. hard-cut old schema/catalog representations without aliases, dual
    decoders, or generated fallback reconstruction;
13. update active contracts, foundations, generated examples, diagnostics,
    and schema description surfaces to the exact terminology; and
14. close with correctness, corruption, resource, performance, wasm-size, and
    source-complexity evidence.

## Non-Goals

0.208 does not:

- add a per-entity, per-table, per-session, or global non-strict mode;
- allow undeclared top-level row fields;
- turn ordinary `Blob` into an `AnyValue` substitute;
- add JSON text as a second heterogeneous persisted representation;
- add schema inference from stored payloads;
- expose internal `Value` as a public field type;
- add predicate, equality, ordering, grouping, aggregate, index, uniqueness,
  primary-key, relation, or nested-path semantics for `AnyValue`;
- silently index or compare an inner `AnyValue` scalar based on its current
  tag;
- add arbitrary recursive/cyclic composite type graphs in the opening line;
- add database defaults or generated values for `AnyValue` in the opening
  line;
- add a SQL text literal grammar for heterogeneous values;
- preserve `Structured`, `queryable: false`, `FieldStorageDecode::Value`, or
  `StructuralFallback` as compatibility spellings where they represented the
  removed implicit escape;
- auto-migrate pre-0.208 accepted schema snapshots; or
- retain tests whose only purpose is proving the removed generic structured
  path stays absent.

## Vocabulary And Naming

### Composite

A composite is an exact schema-native record, tuple, or nominal newtype. Its
accepted definition includes stable type identity, complete recursive shape,
nullability at each nested member, and codec identity.

Collections are not collapsed into the word composite at accepted field-kind
sites. `List`, `Set`, and `Map` remain explicit recursive accepted kinds and
may contain composite type references.

### AnyValue

`AnyValue` is the public and schema-facing name for a value whose inner kind
may vary between writes. The name is deliberate:

- `Opaque` would imply raw uninterpreted bytes, which bounded `Blob` already
  provides;
- `Json` would exclude IcyDB-native values such as principals, ULIDs,
  decimals, enums, and wide integers or would imply lossy JSON coercion; and
- `VersionedBlob` is better represented as an ordinary exact record containing
  a version and bounded blob.

`AnyValue` is not an alias for internal `Value`. It is a public boundary type
with a closed, documented variant set and explicit conversions into admitted
canonical values.

### Runtime Value

The internal `Value` and boundary `InputValue` / `OutputValue` families remain
generic execution carriers. Their existence does not make a field
heterogeneous. Accepted field kind determines whether a runtime value may be
persisted.

### Non-Queryable

Non-queryable means the planner exposes no predicate, ordering, grouping,
aggregate, index, or nested-path capability for that field. It does not mean
untyped, undeclared, unvalidated, or omitted from accepted schema.

## Authority Model

The authority order is:

1. the accepted schema catalog owns row layout, field kind, absence policy,
   codec identity, composite definitions, enum definitions, bounds, indexes,
   and relations;
2. the accepted composite catalog owns nominal composite identity and complete
   recursive shape;
3. the accepted enum catalog owns enum identity and variants;
4. the admitted canonical after-image owns the value that may enter commit
   planning;
5. generated types and codecs encode/decode Rust values and propose schema,
   but do not reconstruct missing accepted authority;
6. nested leaves, planner capabilities, indexes, and relation projections are
   derived from accepted authority; and
7. raw persisted rows, journals, imports, and restores are untrusted bytes
   until bounded current-form decoding proves the accepted contract applicable
   to their boundary.

SQL remains a frontend. It may lower a supported typed `AnyValue` parameter or
exact composite operation into the same admission owner, but it does not own a
different dynamic typing rule.

## Accepted Type Model

The exact names may be adjusted to local module conventions, but the semantic
shape is fixed:

~~~text
AcceptedFieldKind
  = existing exact scalar and collection kinds
  | Enum { type_id: EnumTypeId }
  | Composite { type_id: CompositeTypeId }
  | AnyValue {
        max_bytes: u32,
        codec: AnyValueCodec::CanonicalV1,
    }

AcceptedCompositeType {
    id: CompositeTypeId,
    path: String,
    codec: CompositeCodec::StructuralV1,
    shape: Record | Tuple | Newtype,
}

Record {
    fields: [
        CompositeField {
            name,
            kind: AcceptedFieldKind,
            nullable,
        },
        ...
    ]
}

Tuple {
    elements: [
        CompositeElement {
            kind: AcceptedFieldKind,
            nullable,
        },
        ...
    ]
}

Newtype {
    inner: AcceptedFieldKind,
    nullable: bool,
}
~~~

`CompositeTypeId` is store-local accepted identity, following the enum-catalog
pattern. Two Rust types with the same structural members but different stable
type paths remain different nominal composite types. Shape equality does not
erase nominal identity.

The accepted composite catalog is a sibling authority to the accepted enum
catalog inside the existing accepted schema catalog context. The opening slice
must not rename or generalize the complete enum subsystem merely to obtain a
more abstract catalog name. A later owner-consolidation slice may do that only
with independent evidence.

The accepted definition graph must be canonical:

- type paths are unique within one store-local catalog;
- IDs are allocated deterministically by the accepted publication owner;
- record field names are unique and canonically ordered for identity;
- tuple position is identity;
- member kinds and nullability are complete;
- referenced enum and composite IDs exist in the same accepted catalog
  authority;
- the opening implementation rejects cyclic composite definitions; and
- the full graph participates in the accepted schema fingerprint.

Empty records and empty tuples remain distinct exact shapes. Unit and nominal
zero-field types are not silently interchanged.

## Generated Proposal And Reconciliation

Schema derives must propose complete composite definitions rather than only
`Structured { queryable }` plus optional nested leaves.

A generated proposal includes:

- stable Rust type path;
- record, tuple, or newtype discriminator;
- complete member names or tuple positions;
- exact recursive member kinds;
- nested nullability;
- generated composite codec identity; and
- derived queryable nested leaves, if any.

Reconciliation resolves proposal type paths and definitions into accepted
store-local IDs. Existing accepted identity is reused only when the nominal
path, shape, nested kinds, nullability, and codec match exactly under the
maintained schema-mutation rules. Unknown, incomplete, cyclic, or mismatched
definitions reject publication.

The runtime may compare a generated codec proposal with accepted shape before
typed materialization. It may not synthesize an accepted composite definition
from generated metadata on a cache miss, restore, replay, query, or write path.

## Exact Composite Admission

Accepted admission replaces the `Structured -> normalize_untyped` branch with
shape-directed admission:

### Records

A record value must:

- use the canonical record/map envelope owned by `StructuralV1`;
- contain every declared field exactly once;
- contain no undeclared field;
- use canonical field-name encoding and ordering;
- admit every member against its exact accepted kind;
- encode explicit nested null only where that member is nullable; and
- consume the shared recursive depth and byte budgets.

Nested record fields do not gain omission defaults in 0.208. A nullable member
is present with an explicit null value; it is not silently omitted. Database
defaults and generated write policies remain top-level accepted field
policies.

### Tuples

A tuple value must:

- use the canonical tuple/list envelope owned by `StructuralV1`;
- contain exactly the accepted number of elements;
- admit each position against its exact accepted kind and nullability; and
- consume the shared recursive depth and byte budgets.

### Newtypes

A newtype remains nominal. It admits exactly its declared inner kind and does
not become interchangeable with that inner kind merely because payload bytes
are compatible.

### Collections And Enums

Lists, sets, and maps recursively admit their declared element kinds.
Composite members use accepted composite IDs; enum members use accepted enum
IDs. Set uniqueness, map key validity, canonical ordering, and all existing
collection bounds remain enforced.

### One Walk

The implementation should validate exact shape, canonical ordering, nullability,
and resource consumption during the same traversal used to decode or
canonicalize the payload. It must not decode an entire recursive value once,
then walk it a second time solely to rediscover exact shape, unless measurement
shows no viable one-walk implementation.

Generated typed materialization remains a final consistency check on typed
paths. It is no longer the first owner capable of detecting an extra record
field or wrong tuple arity.

## Codec Split

The current labels `FieldStorageDecode::Value` and
`LeafCodec::StructuralFallback` are too broad where they hide the difference
between exact composite bytes and intentionally heterogeneous bytes.

The maintained semantic lanes are:

| Lane | Accepted owner | Codec | Shape authority |
|---|---|---|---|
| Exact scalar/collection | `AcceptedFieldKind` | existing exact by-kind codecs | recursive accepted kind |
| Exact composite | accepted composite catalog | `CompositeCodec::StructuralV1` | accepted composite definition |
| Heterogeneous | `AcceptedFieldKind::AnyValue` | `AnyValueCodec::CanonicalV1` | closed AnyValue domain plus bounds |

Implementation may keep a narrow internal decode strategy enum if it still
removes ambiguity, but persisted and schema-facing representations must encode
the distinct lanes. There is no generic current-form `Value` lane.

`CompositeCodec::StructuralV1` should preserve existing generated composite
payload bytes where those bytes already have one deterministic grammar and can
be validated from accepted metadata. Byte preservation is an implementation
optimization, not permission to retain generic `Structured` authority.

`AnyValueCodec::CanonicalV1` is a separately named codec even if it reuses
low-level canonical-value tag and length helpers. Exact composites never select
it implicitly.

## AnyValue Schema Surface

The intended Rust field type is `AnyValue`. The intended explicit schema
spelling follows the existing built-in item style:

~~~text
field(
    ident = "payload",
    value(item(prim = "AnyValue", max_bytes = 65_536)),
)
~~~

The exact derive parser location is implementation detail; these semantics are
not:

- `max_bytes` is mandatory and positive;
- `unbounded` is rejected;
- the bound includes the complete persisted `CanonicalV1` field payload,
  including its version/tag/length envelope;
- the bound may not exceed the engine-wide accepted-value cap;
- outer `Option` / accepted nullability remains separate from the inner value
  domain; and
- ordinary custom structs do not become `AnyValue` merely because they
  implement a runtime conversion trait.

No legacy `Structured`, `Value`, `opaque`, `json`, or `any` spelling aliases the
new type.

## AnyValue Value Domain

`AnyValue` has a closed public variant set corresponding to the stable values
supported by `CanonicalV1`. The implementation must document that set beside
the type. Adding or changing a variant is a public and persisted-format design
change, not an incidental expansion of internal `Value`.

The domain may include the stable IcyDB scalar, enum, list, and map families
that already have deterministic canonical encodings. Admission still enforces:

- valid principals, accounts, subaccounts, decimals, ULIDs, numeric values,
  enum payloads, map keys, and collection forms;
- canonical map ordering and unique canonical map keys;
- accepted enum-catalog identity for enum values;
- maximum recursive depth of 64;
- the field's declared encoded-byte bound; and
- the engine-wide accepted-value cap.

An identifier stored inside `AnyValue` is only its scalar value. It does not
become a relation and receives no target-existence or delete-safety semantics.

`AnyValue` admission preserves the caller's explicit variant tag. It does not:

- convert text to numbers, dates, principals, enums, or ULIDs;
- convert signed numeric variants to unsigned variants;
- narrow or widen a numeric tag based on magnitude;
- parse JSON text into a structured value;
- interpret blob bytes as another type; or
- infer a record type from map keys.

Canonical ordering and encoding are normalization of one already-declared
variant, not cross-kind coercion.

Top-level explicit null follows accepted field nullability. There is no hidden
"present null" state distinct from the field's explicit null value. Nulls
inside an admitted list or map remain inner values governed by the canonical
container contract.

## AnyValue Capability Fence

`AnyValue` begins with the smallest honest capability set:

| Capability | 0.208 policy |
|---|---|
| Typed create/save | Allowed through explicit `AnyValue` admission |
| Typed load/whole-value projection | Allowed |
| SQL bound-parameter write | Allowed only through an explicitly tagged `AnyValue` boundary value |
| SQL text literal write | Rejected; no heterogeneous literal grammar |
| SQL whole-value projection | Allowed only through an explicitly tagged lossless output value |
| Database default | Rejected in the opening line |
| Generated value | Rejected |
| Equality / inequality predicate | Rejected |
| Range / membership / pattern predicate | Rejected |
| Ordering | Rejected |
| Grouping / DISTINCT | Rejected |
| Aggregate input | Rejected |
| Primary key | Rejected |
| Secondary or expression index | Rejected |
| Unique or exclusion constraint | Rejected |
| Relation key or target | Rejected |
| Nested field path | Rejected |
| Field-to-field comparison | Rejected |

Whole-value projection means transport and retrieval, not comparison. Boundary
types must retain the explicit `AnyValue` wrapper so a projected inner scalar
cannot accidentally acquire the capabilities of an ordinary typed field.

If a frontend lacks the tagged lossless input or output representation, it
rejects the operation. It does not stringify, emit JSON, expose internal enum
tags accidentally, or degrade to blob bytes.

Future capabilities require separate designs that define stable semantics,
accepted fingerprints, index encodings, cursor behavior, and evidence. The
runtime never dispatches on the current inner tag to grant an undeclared
capability.

## Write And Recovery Boundary Matrix

Every boundary must have an explicit policy:

| Boundary | Exact composite rule | AnyValue rule |
|---|---|---|
| Typed create/save | Admit complete generated value against accepted composite ID | Admit explicit wrapper against `CanonicalV1` and field bound |
| SQL INSERT/UPDATE | Lower supported exact inputs to the same accepted admission | Require tagged parameter; reject text literals and untagged values |
| Atomic batch | Admit every after-image before batch marker | Same; one failure rejects the complete atomic batch |
| Exact bulk update | Freeze target set, then admit every after-image before marker | Only explicit fixed tagged assignment if the executor supports the field; otherwise reject |
| Resumable bulk update | Apply accepted eligibility rules per 0.207 | Ineligible in the opening line |
| Default materialization | Use accepted exact typed default | No AnyValue defaults in the opening line |
| Schema backfill | Admit every authored exact value before publication | Require explicit tagged values and normal bounds, or reject unsupported backfill |
| Import | Bounded current-form decode and exact accepted admission | Bounded current-form decode, tag preservation, and field-bound admission |
| Restore | Verify accepted fingerprint, layout, codec, and exact payload shape | Verify accepted fingerprint, codec version, depth, tags, and bytes |
| Recovery replay | Replay only marker-owned current-form canonical bytes under matching authority | Same; no renormalization or legacy decoding after marker |
| Index projection | Derive only declared exact leaves/capabilities | Reject; no index projection |
| Relation projection | Derive only declared relation members | Reject; inner identifiers are not relations |

Mechanical replay after a durable marker does not rerun user validation or
invent a new value. It must still use bounded fallible current-form decoding
and matching accepted authority; corrupt or mismatched bytes produce the
existing typed corruption/recovery failure rather than entering row state.

## Read And Projection Rules

Exact composite reads use accepted composite shape for row validation, nested
path resolution, and output projection. Generated entity decoding may then
materialize the Rust type.

Nested queryable leaves are derived only when the accepted composite definition
and leaf codec prove a lossless exact path. A leaf cannot exist merely because
generated metadata mentions it. Conversely, non-queryable members remain part
of exact composite validation even though they produce no planner path.

`AnyValue` has no nested planner paths. A map inside `AnyValue` is content, not
schema. Whole-value retrieval preserves the explicit wrapper and canonical
variant tags.

No read path may respond to an invalid exact composite by returning a partial
record, dropping unknown fields, filling missing fields from Rust `Default`, or
falling back to an untyped map.

## Top-Level Layout Rule

The current strict row-layout behavior is retained and made explicit:

- every live user field has one accepted slot;
- every retired slot is an accepted layout placeholder, not user payload;
- a raw row with more physical slots than the accepted layout permits is
  corrupt and rejects;
- a shorter historical row is materialized only through accepted absence and
  schema-migration policy;
- imports and restores cannot append undeclared slots; and
- query-invisible does not mean layout-invisible.

0.208 does not replace retired-slot or schema-migration machinery. It prevents
that machinery from being interpreted as permission for arbitrary payload
fields.

## Schema Identity And Fingerprints

The accepted schema fingerprint must include:

- every `CompositeTypeId` reference;
- canonical composite path and shape;
- record field names and nested member order;
- tuple arity and element order;
- newtype identity and inner kind;
- nested nullability;
- referenced enum and composite identities;
- composite codec identity;
- `AnyValue` kind identity;
- `AnyValue` codec version and `max_bytes` bound; and
- the capability classification derived for each field.

Changing any of those facts is a schema change. It cannot reuse a prepared
plan, cursor, marker, replay payload, or accepted-value authority from the old
fingerprint.

Schema description and diagnostics must render exact names such as
`composite(example::Address)` and `any_value(max_bytes=65536,
codec=canonical_v1)`. They must not render both as `structured`.

## Persisted Format And Hard Cut

0.208 is a pre-1.0 current-form format cut.

Implementation must version every persisted representation whose meaning
changes, including as applicable:

- accepted field-kind wire;
- accepted composite catalog/root wire;
- accepted schema fingerprint input;
- schema marker and journal payloads that embed accepted kinds;
- field codec discriminator; and
- public tagged `AnyValue` input/output wire.

Old accepted `Structured { queryable }` snapshots do not decode as composite
or `AnyValue`. There is no alias, discriminator translation, dual decode,
generated reconstruction, or "best effort" shape inference.

Development stores carrying old accepted metadata must be recreated under the
current format. A separately designed external migration tool may be added if
there is a real deployment need, but it is not an in-engine fallback and is
outside 0.208.

Existing exact composite payload bytes may remain byte-compatible when
`StructuralV1` can validate them entirely from the new accepted composite
definition. That does not make the old catalog representation supported.
Implementation must record whether row payload bytes changed separately from
whether catalog and schema-root bytes changed.

`AnyValue` has no pre-0.208 persisted form. Open-ended `Structured` payloads are
not automatically reclassified as `AnyValue`.

## Error Model

All new rejection paths use typed errors. Required classes include:

- unknown or mismatched composite type identity;
- missing, duplicate, or undeclared record field;
- wrong tuple arity;
- nested kind or nullability mismatch;
- unsupported cyclic composite definition;
- invalid composite or AnyValue codec version;
- AnyValue field bound missing, zero, or above the engine cap;
- AnyValue payload exceeding the declared bound;
- recursive depth exhaustion;
- unsupported AnyValue capability;
- untagged or lossy AnyValue boundary input;
- accepted fingerprint or catalog authority mismatch; and
- corrupt persisted composite or AnyValue payload.

Production code must not match error strings or use panics to classify these
conditions. Persisted decode remains bounded and fallible.

## Documentation Contract

The implementation line must update active documentation together with code:

- `docs/FOUNDATIONS.md` must remove the claim that persisted entity values may
  exist outside schema and distinguish ephemeral runtime values from persisted
  slots;
- `docs/contracts/WRITE_ADMISSION.md` must name exact composite and explicit
  `AnyValue` admission;
- `docs/contracts/SQL_SUBSET.md` must record tagged parameter/projection support
  and all unsupported AnyValue expression families;
- `docs/contracts/PERSISTED_FORMAT_INVENTORY.md` must list the composite catalog
  and AnyValue codec/version/bounds;
- schema DSL and generated-code documentation must use exact composites for
  ordinary records/tuples/newtypes and explicit bounded `AnyValue` only for
  heterogeneous fields; and
- diagnostics and examples must not use "opaque", "structured", or
  "non-queryable" as synonyms for heterogeneous.

The short normative sentence should appear consistently:

> Non-queryable persisted fields are allowed. Undeclared persisted slots and
> shape-changing typed fields are not.

## Alternatives Considered

### Keep Structured And Add AnyValue Beside It

Rejected. That leaves two heterogeneous admission paths, one implicit and one
explicit, and preserves the accepted-schema authority gap for normal records
and tuples.

### Rename Structured To AnyValue Without Exact Composite Types

Rejected. It would honestly name the escape hatch but would reclassify every
ordinary composite as heterogeneous and leave generated materialization as the
shape authority.

### Treat Nested Leaves As The Complete Composite Schema

Rejected. Nested leaves are deliberately query-oriented and do not necessarily
represent all non-queryable members, tuple structure, nominal identity, or
codec ownership.

### Store Heterogeneous Values As Blob

Available to applications, but not an equivalent database type. Blob preserves
bytes and delegates all structure to the application. `AnyValue` preserves a
bounded canonical IcyDB value with typed transport and validation while still
denying implicit query semantics.

### Call The Type Json

Rejected. JSON does not faithfully cover IcyDB's value domain and would invite
text parsing, numeric coercion, and ordering assumptions that 0.208 explicitly
forbids.

### Make AnyValue Comparable By Canonical Bytes

Rejected for 0.208. Byte ordering would become user-visible semantic ordering,
affect indexes, uniqueness, cursors, grouping, and future codec evolution.
Transport canonicalization alone is not enough evidence for those semantics.

### Permit Unbounded AnyValue Under The Global Cap

Rejected. The engine cap is a final safety ceiling, not a substitute for a
field's resource contract. Every heterogeneous field must state its own
positive bound.

### Rely On Generated Rust Decode For Exact Shape

Rejected. It violates accepted-schema runtime authority and leaves imports,
restores, replay, structural SQL, and non-materializing paths dependent on an
unwritten fallback assumption.

## Implementation Slices

### Slice 1: Inventory, Format Gate, And Exact Composite Model

- refresh the baseline owner/format/performance inventory;
- introduce store-local composite type identity and accepted composite
  definitions;
- make generated proposals include complete record, tuple, and newtype shape;
- include composite definitions in accepted fingerprint and schema codecs;
- hard-cut old accepted schema/catalog versions; and
- add catalog canonicalization, corruption, definition, and reconciliation
  evidence.

This slice may land without public `AnyValue`, but it may not retain an
open-ended current-form `Structured` field.

### Slice 2: Exact Composite Admission And Codec Split

- replace `Structured -> normalize_untyped` with accepted shape-directed
  composite admission;
- split exact composite codec identity from heterogeneous canonical value
  identity;
- retarget row encode/decode, canonicalization, structural saves, imports,
  restore, replay, nested leaf projection, and schema backfill;
- remove generic current-form `FieldStorageDecode::Value` and
  `StructuralFallback` meanings affected by the split;
- preserve exact existing composite payload bytes only where accepted-native
  validation proves them; and
- delete generated-model runtime fallbacks and obsolete generic branches.

Slice 2 closes the implicit escape hatch and is independently releasable.

### Slice 3: Explicit Bounded AnyValue

- add the public closed `AnyValue` type;
- add the mandatory schema `max_bytes` declaration;
- add accepted `AnyValue` kind and `CanonicalV1` codec identity;
- add tagged input/output boundary forms;
- route typed saves, supported SQL bound parameters, loads, projection,
  batches, import, restore, and replay through one admission owner;
- reject defaults, generated values, resumable updates, untagged SQL literals,
  and every fenced query/index/relation capability; and
- add byte, depth, canonicality, tag-preservation, corruption, and capability
  evidence.

### Slice 4: Contract Closure And Evidence

- update foundations, write admission, SQL subset, persisted-format inventory,
  examples, schema descriptions, and diagnostics;
- audit every logical mutation ingress and persisted decode boundary;
- run focused cross-frontend and cross-storage parity tests;
- measure representative exact-composite and AnyValue admission;
- report raw non-gzipped wasm delta for each code slice;
- report files touched, approximate line delta, and complexity direction; and
- complete the closeout checklist without legacy anti-resurrection tests.

## Focused Validation

### Accepted Composite Catalog

Evidence must cover:

- deterministic ID allocation and canonical definition ordering;
- nominally distinct types with structurally identical shapes;
- exact record field names and nested kinds;
- exact tuple arity and element kinds;
- nominal newtype identity;
- nested list, set, map, enum, and composite references;
- unknown IDs, duplicate paths, duplicate fields, missing definitions, cycles,
  and malformed codec discriminators;
- bounded current-form decode; and
- schema fingerprint changes for every contract fact.

### Exact Admission

Evidence must show rejection of:

- missing, extra, and duplicate record fields;
- wrong record field kind or nested nullability;
- short and long tuples;
- wrong tuple element kind;
- inner values that fit the broad runtime union but not the exact composite;
- malformed sets and maps;
- mismatched enum or composite catalog identity;
- wrong composite codec; and
- over-depth or over-budget payloads.

Evidence must show successful current-form round trips for representative
records, tuples, newtypes, nested composites, empty composites, collections,
and enum payloads.

### AnyValue

Evidence must show:

- two rows may intentionally hold different inner variants in the same
  `AnyValue` field;
- each variant tag survives encode, persist, decode, and projection;
- no numeric, text, JSON, enum, principal, ULID, or blob coercion occurs;
- the field bound counts final encoded bytes and rejects at bound plus one;
- zero, omitted, unbounded, and engine-cap-exceeding declarations reject;
- depth 64 is accepted where otherwise valid and depth 65 rejects according to
  the existing precise depth convention;
- maps canonicalize key order only within the explicit map variant, while
  lists preserve order and duplicates;
- invalid enums, duplicate map keys, corrupt tags, corrupt lengths, trailing
  bytes, and unknown codec versions reject;
- outer nullability is enforced independently; and
- every capability in the fence table fails closed.

### Boundary Parity

Representative accepted and rejected cases must converge across:

- typed create and save;
- supported SQL bound-parameter mutation;
- atomic batch mutation;
- exact bulk update eligibility and execution;
- schema backfill where supported;
- heap and durable stores;
- import and restore;
- marker replay and recovery; and
- whole-value typed and SQL projection where supported.

Unsupported boundaries need typed rejection evidence, not a permissive no-op or
fallback.

### Strictness Oracle Boundary

0.208 extends the 0.204 evidence providers rather than creating a separate
strictness harness. The focused matrix must cover wrong-kind inserts and
updates, null into required fields, every deliberately admitted frontend
conversion, every forbidden or lossy conversion, defaulted/generated values,
a failure in the final row of an atomic batch, and exact unchanged before-state
after rejection.

Lockfile-pinned SQLite `STRICT` is an independent reference only for explicitly
mapped scalar and nullability semantics. Exact composite identity and shape,
accepted slot layout, nominal enums, `AnyValue` tags and bounds, import/restore,
marker replay, and recovery use the independent IcyDB state model, boundary
assertions, corruption fixtures, and fault injection. SQLite agreement does not
satisfy those IcyDB-specific obligations, and the adapter must not normalize
away an IcyDB product distinction merely to create agreement.

### Layout And Corruption

Evidence must retain strict rejection for extra top-level slots and prove that
non-queryable exact composite and `AnyValue` slots still participate in layout,
row bounds, canonical re-emission, journal, restore, and corruption checks.

## Performance And Complexity Budget

### Runtime

The main exact-composite path should replace one open-ended recursive walk with
one shape-directed recursive walk. It must not add a second full payload
traversal to every write. Generated materialization remains necessary for typed
paths, but accepted admission should validate while decoding/canonicalizing
rather than materializing an additional generic tree solely for validation.

The opening benchmark set must include:

- shallow and deeply nested records;
- wide records;
- tuples of representative arity;
- nested collections and enum payloads;
- `AnyValue` scalar, list, and map payloads near several byte bounds;
- typed create/save and structural SQL-bound mutation; and
- restore or replay validation of representative batches.

Report instruction or host-time deltas, allocations, peak temporary bytes, and
encoded row bytes against the pre-slice baseline. A material regression pauses
the slice for owner-level profiling; it does not justify restoring generic
admission.

### Wasm Size

Accepted composite definitions and exact validators will add code and metadata.
Implementation must report raw non-gzipped wasm bytes per slice; gzip is
secondary context. Type definitions should be deduplicated by accepted
store-local ID rather than repeated in every field contract.

### Persisted Bytes

Report separately:

- accepted schema/catalog byte delta;
- representative row payload byte delta;
- marker/journal byte delta; and
- `AnyValue` version/tag envelope overhead.

Exact composite rows should remain byte-neutral when current deterministic
payloads can be validated under `StructuralV1`. Any deviation must be justified
by a stronger current-form invariant, not accidental re-encoding.

### Source Complexity

0.208 adds one accepted composite catalog and one explicit heterogeneous kind,
but deletes a generic accepted kind and implicit admission branches. Slice
reports must include files touched, approximate line delta, and whether owner
count and branching became simpler, stayed neutral, or became more complex.

The intended end state is conceptually simpler:

~~~text
exact field      -> exact accepted admission
explicit AnyValue -> bounded canonical heterogeneous admission
everything else  -> reject
~~~

There is no third fallback lane.

## Risks And Mitigations

### Risk: Composite Catalog Duplicates Enum Catalog Machinery

Mitigation: reuse the established identity, canonicalization, authority, and
fingerprint patterns, but keep the new owner narrow. Do not generalize the full
type catalog until real duplication is measured after both are correct.

### Risk: Accepted Validation Adds A Second Decode

Mitigation: validate exact shape during the existing decode/canonicalization
walk and retain borrowed slices where practical. Benchmark before and after
each admission slice.

### Risk: Byte-Compatible Payloads Are Mistaken For Compatible Schemas

Mitigation: keep nominal type ID, complete shape, codec version, and accepted
fingerprint authoritative even if row payload bytes do not change.

### Risk: AnyValue Becomes A Convenience Default

Mitigation: require explicit type and positive field bound, provide no implicit
conversion from arbitrary custom types, reject query/index capabilities, and
make documentation use exact composites by default.

### Risk: Canonical Bytes Accidentally Define Comparison

Mitigation: capability checks reject all AnyValue comparison, ordering,
grouping, uniqueness, and index paths before execution. Canonical encoding is
transport identity only.

### Risk: SQL Or JSON Coerces Tags

Mitigation: require an explicitly tagged lossless boundary value. Frontends
without that representation reject. No stringification or JSON fallback is
permitted.

### Risk: Generated Models Regain Runtime Authority

Mitigation: accepted-native tests cover admission, raw-row validation, import,
restore, replay, and projection without generated reconstruction. Runtime
source audit rejects model fallback branches.

### Risk: The Hard Cut Leaves Existing Open-Ended Data Unclassified

Mitigation: make the break explicit. Users must remodel each field as an exact
composite, bounded blob, or explicit bounded `AnyValue`. No automatic
classification can know domain intent safely.

## Closeout Checklist

0.208 is complete only when all of the following are true:

- [ ] accepted schema persists complete nominal record, tuple, and newtype
      definitions;
- [ ] generated proposals reconcile exact composite definitions into
      store-local accepted IDs;
- [ ] ordinary composites never route through untyped admission;
- [ ] accepted-native validation rejects wrong record shape and tuple arity
      without relying on generated materialization;
- [ ] nested leaves are derived from complete accepted composite authority;
- [ ] the generic current-form `Structured` accepted kind is gone;
- [ ] ambiguous persisted decode/leaf codec labels are split or removed;
- [ ] `AnyValue` is the only explicit heterogeneous persisted field kind;
- [ ] every `AnyValue` field has a positive final-encoded-byte bound and fixed
      `CanonicalV1` codec;
- [ ] `AnyValue` tags round-trip without cross-kind coercion;
- [ ] every fenced query, index, key, unique, relation, and nested-path use
      rejects before execution;
- [ ] typed, SQL-supported, batch, bulk, backfill, import, restore, replay,
      index, and relation boundaries have an explicit admission or rejection
      policy;
- [ ] old accepted schema/catalog forms fail current-form decode without
      aliases, shims, reconstruction, or dual dispatch;
- [ ] active foundations and contracts state that every persisted slot is
      declared;
- [ ] focused correctness, corruption, resource, and cross-boundary evidence
      passes;
- [ ] raw non-gzipped wasm, runtime cost, persisted bytes, files touched, line
      delta, and complexity direction are reported; and
- [ ] no legacy anti-resurrection tests remain.

## Final Rule

IcyDB is always strict. Exact values use exact accepted kinds. Intentionally
heterogeneous values use an explicit bounded `AnyValue`. A non-queryable field
is still declared, typed, encoded, bounded, and validated. There is no implicit
ANY lane.
