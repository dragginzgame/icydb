## Pre-0.55 Foundation Slice (Prerequisite)

Goal:

- Remove string-authoritative predicate and SQL side paths that would otherwise
  compound drift when expression indexes are introduced.

Boundary:

- This is prerequisite hardening for `0.55` expression indexes.
- This is not a full architecture rewrite.

### Mapped raw-string semantic authority sites

- Conditional-index predicate metadata is still string-authoritative:
  - `crates/icydb-core/src/model/index.rs`
  - `crates/icydb-schema/src/node/index.rs`
  - `crates/icydb-schema-derive/src/node/index.rs`
- The same predicate semantics are reparsed across multiple runtime layers:
  - `crates/icydb-core/src/db/schema/info.rs`
  - `crates/icydb-core/src/db/query/plan/planner/index_select.rs`
  - `crates/icydb-core/src/db/index/plan/mod.rs`
  - `crates/icydb-core/src/db/commit/prepare.rs`
- Commit schema fingerprinting does not include index predicate semantics:
  - `crates/icydb-core/src/db/schema/fingerprint.rs`
- SQL dispatch and EXPLAIN wrappers perform semantic string scanning outside the
  canonical reduced SQL parser:
  - `crates/icydb/src/db/sql/mod.rs`
  - `crates/icydb-build/src/db.rs`
- Identifier normalization logic is duplicated in separate string helpers:
  - `crates/icydb-core/src/db/sql/lowering.rs`
  - `crates/icydb/src/db/sql/mod.rs`
- Projection column labels are reparsed from SQL text in the facade:
  - `crates/icydb/src/db/sql/mod.rs`
- Diagnostics derive entity identity by splitting index name text:
  - `crates/icydb-core/src/db/diagnostics/mod.rs`

### Ranked cleanup plan

- P0 (now): canonical index predicate authority and single lowering boundary.
- P0 (now): include predicate semantics in commit schema fingerprinting.
- P0 (now): route SQL command/explain dispatch through canonical parser tokens.
- P1 (soon): unify SQL identifier normalization into one shared utility/boundary.
- P1 (soon): stop projection-label SQL reparsing; use canonical projection metadata.
- P1 (soon): define typed index key-item metadata (`Field` / `Expression`) for
  expression-index rollout.
- P2 (later): remove `index_name.split('|')` semantic dependency in diagnostics.

### Concrete foundation slices

- [x] F0. Canonical index predicate metadata ownership
  - Introduce canonical predicate authority in index metadata path.
  - Lower once at schema/derive boundary; downstream consumers use canonical form.
  - Implemented:
    - `db::index::predicate::canonical_index_predicate(...)` parse/cache boundary
    - schema validation, planner implication gating, and index-mutation membership
      compile now consume the canonical boundary (no per-layer ad hoc parsing)
    - schema/derive/model raw predicate SQL is explicitly documented as input-only
  - Touch:
    - `crates/icydb-schema/src/node/index.rs`
    - `crates/icydb-schema-derive/src/node/index.rs`
    - `crates/icydb-core/src/model/index.rs`
    - `crates/icydb-core/src/db/schema/info.rs`

- [x] F1. Predicate semantics in schema fingerprinting
  - Add canonical predicate fingerprinting to commit schema hash contract.
  - Implemented:
    - commit schema fingerprint now includes canonical predicate semantics for
      filtered indexes
    - equivalent predicate SQL text with identical semantics hashes identically
    - semantic predicate changes now invalidate commit schema fingerprint
  - Touch:
    - `crates/icydb-core/src/db/schema/fingerprint.rs`
    - predicate fingerprint/hash helpers as needed

- [x] F2. Canonical SQL command routing boundary

    - facade manual `statement_entity_name(...)` / `is_explain_sql(...)` side

  - Touch:

- [x] F4. Canonical projection label ownership
  - Remove facade SQL text reparsing for projection labels.
  - Lower labels from canonical projection metadata.
  - Implemented:

### Exit criteria before expression-index enablement

- No planner/runtime path treats raw predicate SQL text as semantic authority.
- Schema fingerprint changes when filtered-index predicate semantics change.
- SQL route/explain dispatch does not depend on manual keyword scanning helpers.
- Index metadata path has a canonical key-item model ready for expression
  key materialization.
