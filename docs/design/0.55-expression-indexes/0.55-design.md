# 0.55 — Expression Indexes

Status: **In Progress**

Theme: **Deterministic derived-key secondary indexes.**

Implementation tracking should live in:

`docs/design/0.55-expression-indexes/0.55-status.md`

## Intent

`0.55` introduces expression-backed secondary indexes.

Expression indexes allow IcyDB to index a deterministic value derived from one row of one entity rather than only indexing raw stored field values.

Examples:

```sql
CREATE UNIQUE INDEX idx_users_email_casefold
ON users (LOWER(email));

CREATE INDEX idx_orders_created_day
ON orders (DATE(created_at));

CREATE INDEX idx_users_username_norm
ON users (LOWER(TRIM(username)));
```

This line extends the existing secondary-index model after `0.54` filtered / conditional index work.

Expression indexes are implemented as schema-declared deterministic key expressions that:

- validate at schema creation time
- compile to canonical index-key derivation plans
- execute inside the existing mutation/index maintenance pipeline
- participate in planning only when the query predicate or order shape can be lowered to the same canonical expression semantics

## Core Principle

Expression indexes must remain a deterministic extension of existing index contracts, not a new optimizer subsystem.

This line must preserve:

- existing index storage authority
- deterministic planner behavior
- existing cursor semantics
- existing recovery / replay model
- existing commit marker protocol

Expression indexes must not introduce planner-side guesswork, best-effort rewrites, or frontend-only semantics.

Canonical authority remains the normalized engine intent and canonical expression/key representation, not SQL syntax or builder call shape.

## Goals

Expression indexes allow IcyDB to index derived keys such as:

- case-normalized text
- trim-normalized text
- date / time buckets
- deterministic scalar transforms on one row
- composite keys mixing raw fields and derived fields

Examples:

```sql
CREATE UNIQUE INDEX idx_users_email_casefold
ON users (LOWER(email));

CREATE INDEX idx_orders_customer_day
ON orders (customer_id, DATE(created_at));

CREATE INDEX idx_users_username_norm
ON users (LOWER(TRIM(username)));
```

Benefits:

- enables canonical identity keys without forcing global text-collation changes
- supports indexed lookup on normalized user-facing identifiers
- supports bucketed lookup and ordering on derived scalar values
- keeps normalization/query semantics explicit instead of hidden in storage order
- provides a principled future path for expression-aware SQL/index planning

## Non-Goals

This milestone does not introduce:

- arbitrary SQL expression evaluation everywhere
- nondeterministic functions in index definitions
- cross-row or aggregate expressions
- subqueries inside index expressions
- multi-entity references
- cost-based planner rewriting
- general-purpose expression simplification engine
- locale-sensitive or user-configurable collations
- covering / `INCLUDE` index payloads
- bitmap or multi-index intersection planning

This line also does not change global text comparison semantics.

Strict text semantics remain the engine default. Expression indexes provide an opt-in derived-key path for fields like email and username without changing canonical database-wide ordering/comparison contracts.

## Relationship to 0.54 Filtered Indexes

`0.54` adds optional index predicate metadata and filtered-index planning / mutation gates while explicitly excluding expression indexes.

`0.55` builds on the same model direction:

- `0.54` answers which rows belong in the index
- `0.55` answers what derived key is stored for each indexed row

These features are orthogonal and composable.

End-state model direction:

```text
IndexDefinition {
    key_items: [Field | Expression]
    unique
    predicate: Option<Predicate>
}
```

Filtered predicates and key expressions must remain independently validated and must compose without changing storage/recovery fundamentals.

## Pre-0.55 Foundation Prerequisite

Before expression-index enablement, the filtered-index line needs boundary
hardening captured in:

`docs/design/0.55-expression-indexes/0.55-status.md`

Required preconditions:

- current implementation still has string-authoritative predicate side paths
- canonical predicate lowering must become the sole semantic authority
- predicate semantics must be included in schema fingerprinting
- this is a prerequisite/foundation item for `0.55` expression indexes

This prerequisite records and resolves semantic-authority drift without
expanding `0.54` scope into a full tokenization redesign.

## Design Constraints

Expression indexes must satisfy all of the following:

- deterministic
- row-local
- entity-local
- canonicalizable
- replay-stable
- storage-format-stable under declared schema fingerprinting rules

A valid expression index must:

- reference only fields of the indexed entity
- produce one scalar key component per declared expression item
- avoid hidden dependence on execution time, environment, or external state
- lower to one canonical expression representation used by schema, planner, fingerprinting, and mutation paths

## Expression Subset (Initial)

Initial supported expression items should remain intentionally small.

### Allowed expression forms

- direct field reference
- `LOWER(text_field)`
- `UPPER(text_field)` if supported symmetrically by canonical text rules
- `TRIM(text_field)`
- `LOWER(TRIM(text_field))`
- deterministic date/time bucket extraction from one field, for example:
  - `DATE(timestamp_field)`
  - `YEAR(timestamp_field)`
  - `MONTH(timestamp_field)`
  - `DAY(timestamp_field)`
- explicit cast / normalization forms only if existing value coercion rules can express them canonically without widening runtime ambiguity

### Explicitly rejected in initial line

- arithmetic expressions
- concatenation expressions
- boolean expressions as index keys
- nested arbitrary function trees
- user-defined functions
- relation traversals
- aggregate calls
- subqueries
- nondeterministic functions (`NOW()`, random, environment access)
- locale-aware text transforms

The purpose of the initial subset is not expressiveness. The purpose is to ship one small, canonical, engine-owned expression family that is easy to audit across planner, mutation, and recovery paths.

## Canonical Expression Representation

Expression indexes require one canonical representation shared across:

- schema metadata
- runtime index model
- derive/codegen surfaces
- planner candidate evaluation
- predicate/order matching
- index-key derivation at mutation time
- fingerprint / `EXPLAIN` surfaces where relevant

Initial shape:

```text
IndexKeyItem {
    Field(field_name)
    Expression(IndexExpression)
}

IndexExpression {
    Lower(Field)
    Trim(Field)
    LowerTrim(Field)
    Date(Field)
    Year(Field)
    Month(Field)
    Day(Field)
}
```

This representation must be:

- serializable in schema/runtime metadata
- displayable in deterministic canonical text form
- comparable for equality and fingerprint stability
- independent of SQL/frontend syntax details

No planner/runtime path may inspect raw user text for semantics once the expression is normalized.

## Index Model Extension

Current post-`0.54` direction:

```text
IndexDefinition {
    fields
    unique
    predicate: Option<Predicate>
}
```

New model:

```text
IndexDefinition {
    key_items: [IndexKeyItem]
    unique
    predicate: Option<Predicate>
}
```

Where:

- `Field(name)` preserves current secondary-index behavior
- `Expression(expr)` adds derived-key behavior

Legacy field-only indexes remain valid and unchanged in behavior.

Index entries still store:

```text
index_key + row_reference
```

The storage format remains conceptually the same: expression indexes change how `index_key` is derived, not the authority model of index entries.

## Schema Surface

Expression indexes should be schema-declarable without introducing ad hoc string-only semantics as the long-term authority.

There are two acceptable staged paths.

### Stage A — textual metadata, canonicalized immediately

Schema declarations may temporarily accept canonical expression text that is validated and lowered into `IndexExpression` at schema creation / derive time.

### Stage B — typed schema builder surface

Longer-term preferred shape:

```rust
Index::new_with_expression(
    &[IndexKeyItem::Expression(IndexExpression::Lower("email"))],
    true,
    None,
)
```

The important contract is:

- schema authority must become canonical expression metadata
- raw text is an input format only
- runtime/planner may not depend on unparsed expression text

## Validation Rules

Validation occurs at schema creation / derive time.

Expression indexes must reject definitions that are:

- nondeterministic
- non-row-local
- type-invalid
- aggregate-bearing
- subquery-bearing
- relation-dependent
- unsupported by the canonical expression subset

Examples allowed:

```text
LOWER(email)
LOWER(TRIM(username))
DATE(created_at)
customer_id, DATE(created_at)
```

Examples rejected:

```text
NOW()
LOWER(other_table.email)
SUM(balance)
email || '@' || domain
CASE WHEN active THEN email ELSE backup_email END
```

Validation outcomes must be explicit and classified as schema-definition misuse, not silently coerced.

## Planner Integration

Expression indexes are eligible only when the query/order intent can be lowered into the same canonical key semantics.

Planner rule:

```text
query_key_shape == index_key_shape
```

or, for compatible prefix/order use:

```text
query_key_prefix == index_key_prefix
```

Examples:

Index:

```text
LOWER(email)
```

Query intent lowered canonically to:

```text
LOWER(email) = 'ann@example.com'
```

Index is usable.

Query:

```text
email = 'Ann@example.com'
```

Index is not automatically usable unless the planner/query intent already carries the same explicit casefold semantics.

This preserves the current engine principle that frontends do not widen semantics beyond canonical planner intent.

### Order planning

Expression indexes may also support order routes when the order key matches the same canonical expression key shape.

Example:

Index:

```text
LOWER(username)
```

Order:

```sql
ORDER BY LOWER(username) ASC
```

Potentially eligible.

Order:

```sql
ORDER BY username ASC
```

Not expression-index eligible under strict semantics.

### Planner boundary rule

The planner may match:

- canonical expression equality predicates
- canonical expression prefix/range bounds where supported by the expression family
- canonical expression ordering

The planner may not invent expression rewrites on behalf of strict raw-field queries.

## Query / Frontend Surface Implications

This line introduces new engine semantics, so frontend surfaces may expand only where they can map directly to canonical expression intent.

Acceptable future reduced-SQL additions after engine support exists:

```sql
SELECT * FROM users WHERE LOWER(email) = 'ann@example.com';
SELECT * FROM users ORDER BY LOWER(username) ASC LIMIT 10;
```

But this line should not require broad SQL expression support.

A narrow path is preferred:

- planner/runtime gain canonical expression-aware predicate/order intent
- SQL/frontends may later admit only the expression subset already supported by engine intent
- no frontend may introduce broader expression semantics than the engine owns

## Mutation Pipeline Changes

Current secondary-index mutation path computes index keys from raw field lists.

Expression index mutation path becomes:

```text
row mutation
→ derive canonical key items from row
→ encode index key
→ insert / remove index entry
```

For mixed raw + expression keys:

```text
(customer_id, DATE(created_at))
```

Mutation derives:

```text
row.customer_id
bucket_date(row.created_at)
```

Then encodes the composite index key through the existing ordered-key contract.

This derivation must occur inside the existing index mutation pipeline so that:

- save / replace / delete use the same key derivation authority
- replay / recovery rebuild uses the same prepared row-op path
- diagnostics / integrity checks can re-derive expected expression keys from typed rows without new alternate logic

## Recovery, Replay, and Integrity

Expression indexes must preserve `0.53` survivability expectations:

- replay remains deterministic
- recovery rebuild can re-derive expected index entries from persisted rows
- incompatibility is explicit and classified
- integrity tooling can detect missing/divergent/orphan expression-index entries

This line must not introduce a storage model that requires hidden best-effort reconstruction.

If expression metadata changes schema fingerprints or runtime hook behavior, those changes must remain explicit and recovery-safe under existing persistence contracts.

## Uniqueness Semantics

Unique expression indexes enforce uniqueness on the derived canonical key, not on the raw stored source fields.

Example:

```text
UNIQUE LOWER(email)
```

Rows with:

```text
Ann@Example.com
ann@example.com
```

Conflict because both derive:

```text
ann@example.com
```

This is a primary reason to ship expression indexes: they provide explicit, index-level normalized identity semantics without changing global text comparison rules.

## Text Semantics Boundary

This milestone must not turn IcyDB into a collation-driven engine.

Current engine direction is:

- strict text semantics by default
- explicit casefold/coercion behavior only where declared

Expression indexes preserve that contract by keeping normalization local and opt-in.

This is preferred over implicit global case-insensitive semantics because it:

- avoids hidden ordering changes
- avoids cursor/fingerprint drift from ambient collation policy
- keeps identity normalization explicit at schema/query boundaries
- preserves deterministic storage/planner contracts

## Filtered + Expression Composition

Filtered and expression indexes must compose cleanly.

Example:

```sql
CREATE UNIQUE INDEX idx_active_email_ci
ON users (LOWER(email))
WHERE active = true;
```

Composition rule:

- key expression validation and predicate validation remain separate
- planner eligibility requires both:
  - canonical key-shape match
  - predicate implication for the filtered portion
- mutation maintenance applies both:
  - evaluate `predicate(row)`
  - if true, derive expression key and maintain entry

This keeps `0.54` and `0.55` orthogonal rather than creating one blended feature with ambiguous ownership.

## Error Surface

Expression-index failures should remain classified and fail-closed.

Required error families:

- unsupported expression definition at schema time
- invalid field/type use in expression definition
- unsupported planner/query expression shape
- deterministic mismatch between query intent and available expression index
- derived-key uniqueness conflicts on unique expression indexes
- integrity divergence between persisted rows and expression-index entries

No path should silently downgrade an unsupported expression-index query into a semantically different raw-field route.

## EXPLAIN Surface

`EXPLAIN` should make expression-index use visible in a deterministic way.

Where an expression index is selected, plan/execution output should expose:

- selected index name
- canonical key expression shape
- whether eligibility came from equality, prefix/range, or ordering match
- whether a filtered predicate also gated the index

This line does not require elaborate optimizer output, but it should make expression-index selection auditable.

## Rollout Plan

### Slice A — Canonical expression model + metadata foundation

Goal:

- add canonical `IndexKeyItem` / `IndexExpression` metadata to schema and runtime index models
- preserve legacy raw-field index behavior
- keep storage/recovery semantics unchanged at this stage

Exit outcomes:

- schema/runtime index models can represent field and expression key items
- derive/codegen can emit canonical expression metadata
- display/accessor tests lock field vs expression index shape

### Slice B — Schema-time validation boundaries

Goal:

- validate supported expression subset and reject unsupported/nondeterministic forms at schema creation / derive time

Exit outcomes:

- unsupported expressions fail closed
- type/field misuse is explicitly classified
- canonical expression parsing/lowering is deterministic

### Slice C — Mutation-path expression key derivation

Goal:

- extend existing index mutation pipeline to derive expression keys from rows for save / replace / delete / replay preparation

Exit outcomes:

- normal writes and replay derive identical expression index ops
- unique expression conflicts enforce derived-key uniqueness
- mixed raw + expression composite keys behave deterministically

### Slice D — Planner eligibility for expression-aware predicates / ordering

Goal:

- allow planner candidate evaluation to match canonical expression key shapes for supported equality/range/order patterns

Exit outcomes:

- canonical expression predicates can use expression indexes
- canonical expression ordering can use expression indexes where shape permits
- no implicit semantic widening from strict raw-field predicates

### Slice E — Filtered + expression composition

Goal:

- support combined filtered + expression indexes without changing ownership boundaries

Exit outcomes:

- predicate implication and expression-key matching compose deterministically
- mutation pipeline applies predicate gate + derived key maintenance together

### Slice F — Diagnostics, EXPLAIN, and end-to-end parity tests

Goal:

- expand tests and diagnostics so expression indexes are visible and recovery / integrity behavior is locked

Exit outcomes:

- `EXPLAIN` exposes selected expression index route
- integrity scan can detect missing/divergent/orphan expression index entries
- matrix tests cover save/load/delete/replay/query parity

## Validation Matrix (Required)

At minimum, tests should cover:

### Schema / metadata

- field-only legacy indexes still serialize/display identically
- expression index metadata is canonical and stable
- unsupported expressions fail schema validation

### Mutation / uniqueness

- `LOWER(email)` unique conflict on case-only differences
- `LOWER(TRIM(username))` normalizes both trim and case
- composite `(customer_id, DATE(created_at))` encodes deterministically
- delete/remove path removes derived-key entry correctly
- update old->new expression value removes stale key and inserts new key

### Planner / route selection

- canonical expression equality uses matching expression index
- raw strict field equality does not spuriously use expression index
- canonical expression ordering uses expression index when supported
- filtered + expression composition obeys both gates

### Recovery / integrity

- replay rebuild re-derives identical expression-index entries
- integrity report detects missing forward expression-index entries
- integrity report detects orphan expression-index references
- incompatible schema/expression metadata remains fail-closed and classified

### EXPLAIN / diagnostics

- plan/execution output names expression index when selected
- non-eligible strict query surfaces clear fallback / non-use behavior

## Explicit Exclusions for This Line

The following remain out of scope for `0.55` unless promoted deliberately in a later design update:

- arithmetic expression indexes
- concatenation or general string-building expression indexes
- arbitrary boolean expression indexes
- JSON-path / document expression indexes
- collation families
- covering / `INCLUDE` payload indexes
- optimizer-style equivalence rewrites between raw-field queries and expression index semantics

## Summary

`0.55` is the Expression Indexes line.

It introduces deterministic derived-key secondary indexes so IcyDB can support normalized identity keys and bucketed/indexable derived values without changing its global strict text semantics or weakening planner/recovery determinism.

This line formalizes:

- canonical expression-key metadata in schema/runtime index models
- fail-closed schema validation for a small deterministic expression subset
- mutation-time derived-key maintenance inside the existing index pipeline
- planner eligibility based on canonical expression-shape matching
- filtered + expression index composition without boundary collapse
- recovery/integrity/`EXPLAIN` parity for derived-key index behavior

This line is not about a general SQL expression engine.

It is about adding one small, explicit, deterministic expression-key capability to the existing index architecture.
