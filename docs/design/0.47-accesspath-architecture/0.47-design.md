## 1. Purpose

Define `AccessPath` as a physical storage traversal contract, not a planning
policy surface.

`AccessPath` answers:

- how storage is traversed
- what traversal guarantees the executor can assume
- what continuation semantics apply to that traversal

`AccessPath` does not answer:

- why a path was selected
- whether the path is legal for a query shape
- which path should be preferred

Those decisions remain planner-owned.

---

## 2. Ownership Model

Owner boundary:

- owner subsystem: `db/access/*`
- authoritative types:
  - `db/access/path.rs` (`AccessPath`)
  - `db/access/plan.rs` (`AccessPlan`)
  - `db/access/dispatch.rs` (canonical path dispatch)

Primary consumers:

- `db/query/plan/*` constructs/lower paths
- `db/executor/*` executes lowered contracts
- `db/query/explain/*` renders path semantics

Non-owners:

- `cursor`
- `index`
- `codec`
- `query/intent`

Rule:

- only planner-side boundaries construct semantic `AccessPath` contracts for
  production query flow
- non-owner layers treat `AccessPath` as immutable and consume through
  dispatch/capability boundaries

---

## 3. Decision Authority Rule

Target budget:

- `AccessPath` decision owners `<= 3`

Allowed decision-owner boundaries:

- `query/plan/route` (path selection policy)
- `query/plan/feasibility` (shape legality/pushdown feasibility)
- `executor/dispatch` (execution dispatch only, no planner policy)

Forbidden locations for new `AccessPath` decisions:

- `executor/load`
- `cursor`
- `index`
- `storage/commit`

Those layers may validate and execute contracts, but should not re-derive
path-selection policy.

Current baseline (2026-03-09):

- `AccessPath decision owners: 5` (above target)

Required correction for `0.47`:

- converge current ownership to the three allowed boundaries above
- all other layers must treat `AccessPath` as immutable data contracts

## 4. Variant Contract

Current semantic variants (`db/access/path.rs`):

- `ByKey`
- `ByKeys`
- `KeyRange`
- `IndexPrefix`
- `IndexMultiLookup`
- `IndexRange`
- `FullScan`

### 4.1 Variant Invariants

| Variant | Required Invariant | Ordering Guarantees | Cursor Compatibility | Resume Semantics |
|---|---|---|---|---|
| `ByKey` | one canonical primary-key payload | deterministic singleton | compatible | resume is trivial (terminal/single-hit) |
| `ByKeys` | key set semantics (dedupe/canonical order) | planner-defined deterministic key-set order | compatible | resume by stable key-set progression |
| `KeyRange` | bounded primary-key range | primary-key order | compatible | strict monotonic continuation over key bounds |
| `IndexPrefix` | prefix arity and strict value coercion validity | index-defined ordered traversal | compatible | strict continuation within prefix envelope |
| `IndexMultiLookup` | canonicalized lookup set over leading index slot | per-lookup ordered traversal + union contract | compatible with planner constraints | resume must preserve per-lookup canonical order contract |
| `IndexRange` | equality prefix + bounded range component contract | index-ordered traversal | compatible | strict continuation monotonicity with bounded envelope |
| `FullScan` | no index-assisted narrowing | storage/default traversal order contract | compatible only where planner policy permits | resume semantics depend on planner+route paging contract |

### 4.2 `IndexRange` Explicit Contract

`IndexRange` requires:

- ordered index traversal source
- explicit semantic envelope bounds
- valid index id/arity/prefix contract

`IndexRange` guarantees:

- deterministic ordered traversal
- strict continuation monotonicity when continuation is present
- bounded envelope revalidation at runtime

## 5. Execution Contract

Executor may assume:

- planner legality checks already passed
- lowered access envelope/bounds are shape-valid
- index identity contracts are already validated at planner/lowering boundaries

Executor must still:

- fail closed on runtime invariant violations
- execute traversal exactly as declared by the contract
- apply projection/filter/paging semantics without mutating access semantics

This keeps planner policy and executor runtime responsibilities cleanly split.

---

## 6. Routing Model

Adopt/maintain a two-step mapping:

`AccessPath` -> route/access classification -> executor route shape dispatch

Current building blocks:

- `AccessRouteClass` (`db/access/execution_contract/route.rs`)
- `RouteShapeKind` / `ExecutionRouteShape`
  (`db/executor/route/contracts.rs`)

Goal:

- keep direct `AccessPath` branching centralized (`db/access/dispatch.rs`)
- route/executor consumers operate on route/capability contracts instead of
  variant-specific branching spread

Containment rule:

- route planner hub follows `route planner -> RouteShape -> executor dispatch`
- route planner should not directly depend on access internals beyond exposed
  access-route contract surfaces
- route planner cross-layer dependency pressure should trend toward a single
  cross-layer contract boundary (`<=1` tracked cross-layer family)

### 6.1 AccessPath -> RouteShape Mapping Contract

Mapping must remain explicit and centralized.

Current-direction mapping examples:

| AccessPath Variant | Route/Dispatch Shape Intent |
|---|---|
| `ByKey` | single-lookup load/aggregate route shape |
| `ByKeys` | key-set lookup route shape |
| `KeyRange` | ordered primary-key scan route shape |
| `IndexPrefix` | ordered secondary-index scan route shape |
| `IndexMultiLookup` | multi-lookup secondary route shape |
| `IndexRange` | ordered bounded secondary-index scan route shape |
| `FullScan` | full-scan route shape |

Rule:

- mapping tables are planner/route-authority data, not ad-hoc executor/load
  branching logic

Expected outcomes:

- fewer branch sites
- cleaner EXPLAIN coupling
- lower fan-out in runtime routing modules

## 7. Extension Rules

Before adding a new `AccessPath` variant, answer:

1. Is this a new physical traversal mode?
2. Does it change ordering guarantees?
3. Does it change continuation/resume behavior?

If all answers are `no`, do not add a variant.

Should not become new variants:

- predicate pushdown preference
- covering-detection policy
- cost-based preference/candidate ranking

May justify a new variant:

- new traversal primitive over storage/index substrate
- materially different ordering contract
- materially different continuation contract

## 8. Capability API Direction

Preferred consumption pattern:

- consumers call capability methods
- consumers avoid scattered `match AccessPath::...` branching

Capability examples (target API direction):

- `guarantees_order()`
- `supports_strict_resume()`
- `is_index_backed()`
- `supports_reverse_traversal()`

Example contract direction:

```rust
impl<K> AccessPath<K> {
    fn guarantees_order(&self) -> bool;
    fn supports_strict_resume(&self) -> bool;
    fn is_index_backed(&self) -> bool;
}
```

Current baseline:

- `AccessRouteClass` already carries route-facing capabilities; continue moving
  runtime consumers toward this contract surface.

## 9. Explain Integration

Rule:

- `AccessPath` must remain explainable through stable contract surfaces without
  leaking planner internals.

Direction:

- preserve one explain-description boundary for access traversal semantics
- avoid EXPLAIN call sites reaching into variant internals outside designated
  rendering adapters

Target API direction:

```rust
pub(crate) struct AccessPathDescription {
    // stable explain-facing traversal description fields
}

impl<K> AccessPath<K> {
    fn describe(&self) -> AccessPathDescription;
}
```

## 10. Metrics and Guardrails

Track in recurring audit/CI:

- `AccessPath` variant count
- `AccessPath` decision-owner count
- `AccessPath` reference-site count
- `AccessPath` direct match/branch site count
- route-planner hub import pressure (HIP)
- feature-slice touch-size pressure tied to access/route changes

Suggested thresholds:

- variants `<= 10`
- decision owners `<= 3`
- branch-site growth must be justified when increasing
- module size guidance:
  - soft cap: `600 LOC`
  - hard cap: `900 LOC`
  - modules above soft cap require decomposition review
- route-planner cross-layer count `<=1` in HIP table
- slice-size containment guardrails:
  - median files touched per slice `<=8`
  - p95 files touched per slice `<=15`

- decision owners: `5`

  `17` (method-sensitive proxy)

Trigger:

## 11. Anti-Patterns (Prohibited)

- mutable `AccessPath` rewriting in non-planner runtime flow
- planner-policy checks reimplemented in executor runtime modules
- variant-specific logic scattered across unrelated layers
- new decision owners in non-authorized boundaries

Bad pattern:

```rust
if path.is_index_range() && predicate_has_x {
    // planner policy leakage in runtime
}
```

Good pattern:

```rust
// planner emits access + route contracts
if route_shape.is_streaming() {
    // executor dispatches only
}
```

Prefer:

- planner encodes semantics once
- access/route contracts expose capabilities
- executor dispatch executes declared route shape
