# 0.119 Structural Simplification & Flow Deletion

## Thesis

`0.119` should remove redundant expression flows, intermediate
representations, and adapter scaffolding that remain after semantic and flow
centralization.

This is not a continuation of `0.118` in the sense of adding more semantic or
routing work.

It is not a semantic slice.

It is not a feature slice.

It is not a routing slice.

It is a subtractive slice whose purpose is to:

* reduce LoC
* eliminate duplicate paths
* delete obsolete adapters and wrappers
* make the physical expression pipeline match the logical model that the
  earlier slices already established

`0.116` centralizes truth-condition semantics.

`0.117` centralizes expression-family semantics.

`0.118` collapses the remaining dominant expression-flow path.

`0.119` should then remove the structural remnants that no longer justify
their existence.

---

## Why `0.119` Exists

After:

* `0.116` truth semantics centralization
* `0.117` expression-family semantics centralization
* `0.118` expression-flow centralization

the system should already have:

* one semantic authority
* one dominant execution path

But the tree can still contain:

* legacy adapter flows
* redundant intermediate shapes
* thin wrapper layers
* duplicated branching scaffolding
* conversion helpers that exist only because an older path once needed them

Those remnants are no longer correctness assets.

They are now pure cost:

* higher LoC
* higher cognitive load
* higher blast radius
* misleading architecture signals

This is why `0.119` exists.

It is the first slice in the line whose purpose is primarily subtraction.

## Problem

The logical expression model can already be correct while the physical code
still reflects older architectures.

That creates a mismatch:

### Logical Model

```text
lowering -> planner Expr -> execution
```

### Physical Code

```text
lowering
  -> predicate helpers
    -> planner Expr
      -> prepared/session wrappers
        -> executor
```

Even when some of those hops are now thin, they still:

* exist
* must be maintained
* appear in grep and audits
* inflate the number of places touched by routine changes
* make the architecture look broader than it really needs to be

The problem is no longer meaning.

The problem is leftover structure.

---

## Goal

Make the physical code structure match the logical model.

That means:

* no redundant expression flow paths
* no intermediate representations that exist only to be converted again
* no adapter layers that add no real logic
* no duplicated branching outside planner

---

## Core Invariant

For any shipped expression form, there is exactly one structural flow path
from lowering to planner to execution.

And:

* planner owns all semantic branching
* all other layers are either thin linear adapters or deleted

If two physical paths exist for the same expression handling and only one is
needed after `0.116` through `0.118`, the non-canonical path should be
removed.

---

## Ownership Boundary

`0.119` should make the subtractive boundary visible in code shape.

### Parser / Lowering Owns

* syntax-specific structure building
* direct construction of planner-compatible inputs

Parser / lowering must not keep:

* intermediate expression carriers that only survive one hop
* adapter scaffolding whose only purpose is forwarding into planner

### Predicate Owns

* runtime predicate transport where a predicate surface still exists
* minimal boundary-local adaptation from planner-owned forms

Predicate must not keep:

* expression rebuilding that duplicates planner flow
* redundant compile entrypoints
* planner-equivalent branch ladders

### Planner Owns

* canonical `Expr`
* semantic branching
* canonical planner-owned contracts used by downstream consumers

### Prepared / Session Owns

* binding
* contract transport
* execution wiring

Prepared / session must not keep:

* duplicate bind/plan helpers
* wrapper layers that only rename or forward planner contracts
* parallel template handling that no longer carries distinct behavior

### Executor Owns

* execution of planner-frozen contracts

Executor must not keep:

* expression-family rerouting
* shadow expression compilation paths

## What `0.119` Targets

### 1. Delete Redundant Flow Paths

Identify places where:

* two physical paths exist for the same expression handling
* one is planner-driven
* the other is legacy or adapter-driven

Action:

* delete the non-canonical path entirely

### 2. Collapse Intermediate Representations

Find structures that:

* exist only to be converted again
* are not used outside a single hop

Example pattern:

```text
LoweringStruct -> PredicateStruct -> Expr
```

Action:

* remove the intermediate representation
* lower directly into `Expr`

### 3. Eliminate Adapter-Only Layers

Find functions or modules that:

* only forward calls
* only rename parameters
* only wrap planner calls without adding real logic

Action:

* inline or delete them

### 4. Remove Duplicate Branching

Search for:

* `match` / `if` ladders on expression kinds
* outside planner expression modules

Action:

* replace them with planner calls
* delete the local branch ladders

### 5. Shrink `bool_expr.rs` To A True Adapter

Current role can still include:

* structure transport
* planner-result adaptation

Target role:

* convert planner-owned truth forms into runtime predicate form
* perform only minimal boundary-local adaptation

Remove:

* expression rebuilding
* compare-family logic that planner already owns
* structural routing that survives only for historical reasons

### 6. Simplify Prepared / Session Path

After the earlier centralization slices:

* most distinct semantic paths should already be gone

Remaining work:

* delete leftover wrappers
* delete parallel helper paths that no longer represent distinct behavior
* ensure one path from bound statement to planned query/execution wiring

### 7. Inline Thin Helpers Across Boundaries

Target helpers that:

* exist only for layer purity
* no longer provide real clarity or isolation

Guideline:

* if removing a helper does not reduce clarity, remove it

## Concrete Actions

### 1. Flow Graph Audit

Map expression flow across:

* lowering
* predicate
* planner
* prepared/session

Mark:

* duplicated paths
* unused paths
* pass-through nodes

Do not start with mechanical inlining alone.

Start by proving which nodes are structurally unnecessary.

### 2. Predicate Reduction Pass

In [bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs:1):

* identify functions that rebuild expression trees
* identify functions that convert expression families or simply transport
  planner-owned results through redundant shells
* remove or inline them where the surviving path stays readable

Target outcome:

* the file shrinks materially
* entrypoint count decreases
* its role becomes obviously adapter-only

### 3. Lowering Simplification

* remove predicate-dependent expression shaping
* remove intermediate carriers that only exist before planner ingestion
* ensure lowering emits planner-compatible `Expr` directly wherever the shipped
  surface already allows it

### 4. Prepared Path Pruning

* unify any remaining parallel helper paths
* delete legacy/fallback divergence that no longer carries distinct logic
* delete duplicate bind/plan helpers that only preserve old layering shape

### 5. Branch Elimination Sweep

Across the expression path:

* grep for expression-family branching outside planner
* replace it with planner calls or planner-owned contracts
* delete the local branching sites

### 6. Dead Code Removal

After the main flow deletions:

* remove unused functions
* remove unused enums or variants
* remove unused conversion helpers

Delete, do not deprecate.

## New Invariants

`0.119` is not complete without deletion-oriented guards.

Add tests or structural guards proving:

* no duplicate expression compile entrypoints remain
* no predicate -> planner -> predicate roundtrip path remains where one direct
  path is sufficient
* no branching on expression kind remains outside planner modules for shipped
  expression handling
* one path exists from lowering -> planner per shipped expression form

These are structural simplification invariants, not new semantic invariants.

---

## Recommended Work Order

### 1. Identify Pass-Through And Duplicate Nodes

Build the flow graph first.

Do not start by deleting helpers opportunistically.

The first job is to prove which nodes add no unique logic.

### 2. Reduce Predicate First

The best first reduction target remains:

* [bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs:1)

Why first:

* it is still the clearest structural remnant of earlier expression flow
* it is named directly in the velocity read
* deleting there is most likely to reduce both LoC and touched-module count

### 3. Remove Single-Hop Lowering Intermediates

Once predicate is thinner:

* remove intermediate lowering shapes that only survive until planner
* collapse directly into planner-compatible `Expr`

### 4. Prune Prepared / Session Wrappers

After the predicate and lowering deletions:

* remove thin prepared/session wrappers and duplicate helpers
* keep only the binding and transport surfaces that still provide real value

### 5. Run A Dead-Code Sweep

Only after the main path is simplified:

* remove dead conversion helpers
* remove dead branches
* remove now-unused internal structures

### 6. Freeze The Smaller Shape With Guards

Use structural guards to stop duplicate paths and dead adapters from returning.

## Exit Condition

`0.119` is done when:

* LoC decreases materially across predicate, lowering, and prepared/session
  expression-path code
* number of expression flow entrypoints is reduced
* no duplicate flow paths remain for shipped expression handling
* [bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs:1)
  is materially smaller and simpler
* routine expression changes touch fewer modules than the pre-slice baseline

This should be visible as:

* fewer hotspot modules in the expression cluster
* lower branch pressure outside planner
* a smaller set of physical files participating in the same logical change

The explicit velocity targets remain:

* median blast radius `<= 8`
* visible reduction in expression-cluster hotspot size

## Non-Goals

`0.119` should not:

* introduce new abstractions without necessity
* refactor for style alone
* preserve unused code “just in case”
* widen feature surface
* reopen planner semantics
* turn into an executor or route-planner rewrite

## Feedback

What looks strong:

* the slice is correctly subtractive rather than additive
* it follows naturally after `0.116` through `0.118`
* it is grounded in velocity and structural cost rather than speculative new
  architecture
* it gives a clear deletion rule: remove what no longer adds distinct logic

What I would keep slightly guarded:

* I would treat `0.119` as contingent on `0.118` landing cleanly enough that
  the canonical flow path is already stable
* I would avoid promising broad dead-code removal all at once unless the first
  flow-graph audit shows the deletions are genuinely local
* I would keep “remove adapter-only layers” subordinate to readability
  rather than turning it into blanket inlining

The main thing I agree with is the sequencing:

`0.119` should not invent anything.

If `0.116` through `0.118` succeed, `0.119` should mostly delete.

---

## Summary

`0.119` is the first subtractive slice in this sequence.

* `0.116` through `0.118` make the system correct and centralized
* `0.119` should make the system physically smaller and more honest

The goal is simple:

* remove everything that no longer needs to exist now that the correct
  architecture is already in place
