# IcyDB 0.169 — Heap Store Runtime Backend

## Status

Closed.

## Summary

0.169 admits the first non-stable store backend: volatile heap storage.

0.167 made store storage explicit at the schema and generated-wiring boundary.
0.168 moved stable iterator shapes behind runtime store-wrapper operations.
0.169 should use that boundary to add `storage(heap())` without spreading
heap/stable conditionals through executor, diagnostics, reconciliation,
mutation, or commit code.

This is a hard-cut feature line. Do not keep reserved-syntax fallbacks, aliases,
or compatibility shims once heap storage is admitted.

## Closed State

The live storage forms are:

```rust
storage(stable(
    data_memory_id = 20,
    index_memory_id = 21,
    schema_memory_id = 22,
))

storage(heap())
```

`storage(stable(...))` remains the durable stable-memory form. Generated stable
stores still initialize `DataStore`, `IndexStore`, and `SchemaStore` through
`ic_memory_key!(...)` and register stable allocation identities.

`storage(heap())` is the canonical volatile heap form. Generated heap stores
initialize heap-backed `DataStore`, `IndexStore`, and `SchemaStore` wrappers and
register explicitly absent allocation identities.

Runtime callers now depend mostly on store-wrapper operations:

```text
get
contains
insert
remove
visit_entries
visit_range
clear
len
memory_bytes
allocation_metadata
```

The 0.168 closeout proved those callers no longer need stable-map iterator
types.

## Goals

- Admit `storage(heap())` in derive/schema declarations.
- Add heap-backed runtime implementations for data, index, and schema store
  roles.
- Preserve stable-store behavior, ABI, memory IDs, stable keys, and allocation
  identities exactly.
- Keep heap store allocation identities explicitly absent.
- Keep heap stores volatile: no stable memory, no stable key, no schema-store
  persistence across canister upgrades, and no durability claim.
- Make diagnostics report `storage: heap` for heap-backed data/index/schema
  store rows.
- Keep executor, reconciliation, diagnostics, schema mutation, relation, and
  commit code backend-neutral at the store-wrapper boundary.
- Prove heap and stable stores share the same logical row/index/schema
  semantics where volatility does not matter.
- Keep heap collection ordering deterministic and identical to raw key order.

## Non-Goals

- No WAL.
- No checkpointing.
- No heap-to-stable flush.
- No `cached_stable(...)` or `journaled(...)`.
- No stable-memory migration.
- No stable-store ABI change.
- No stable key generation change.
- No physical stable-memory ID change.
- No query planner feature expansion.
- No relation semantic change.
- No compatibility alias for the old heap-reserved diagnostic.
- No mixed backend for one logical store; a store is stable or heap.

## Core Invariants

### Stable Stores

Stable stores must remain byte-for-byte compatible:

```text
storage(stable(...))
-> same generated memory IDs
-> same stable keys
-> same allocation identities
-> same row/index/schema codecs
-> same scan ordering
-> same commit/recovery behavior
```

### Heap Stores

Heap stores are explicit volatile stores:

```text
storage(heap())
-> no memory IDs
-> no stable keys
-> no stable allocation identities
-> no upgrade persistence
-> deterministic in-memory raw key ordering while live
```

Heap volatility must be visible in diagnostics and docs. It must not be hidden
behind stable-looking metadata.

### Runtime Boundary

Higher layers must not branch directly on stable vs heap. Backend choice belongs
inside store construction and store-wrapper internals.

Allowed:

```text
StoreStorage -> generated store wiring -> StoreHandle
StoreHandle -> DataStore/IndexStore/SchemaStore operations
```

Rejected:

```text
executor/reconciliation/diagnostics/commit code checking heap vs stable
executor/reconciliation/diagnostics/commit code touching backend map internals
```

## Design

### Schema Model

Extend store storage from the stable-only model:

```rust
pub enum StoreStorage {
    Stable(StoreStableMemoryConfig),
}
```

to:

```rust
pub enum StoreStorage {
    Stable(StoreStableMemoryConfig),
    Heap(StoreHeapConfig),
}
```

`StoreHeapConfig` should be intentionally small. If no heap fields are needed,
use an empty config type rather than encoding fake stable fields.

```rust
pub struct StoreHeapConfig;
```

Heap storage must reject stable-only fields:

```rust
storage(heap(data_memory_id = 1))
storage(heap(index_memory_id = 2))
storage(heap(schema_memory_id = 3))
```

Unknown heap fields should reject rather than being ignored.

### Derive Syntax

Accepted:

```rust
#[store(
    name = "scratch",
    storage(heap()),
)]
```

Still accepted:

```rust
#[store(
    name = "durable",
    storage(stable(
        data_memory_id = 20,
        index_memory_id = 21,
        schema_memory_id = 22,
    )),
)]
```

Rejected:

```rust
storage(heap)
storage(heap(data_memory_id = 1))
storage(heap(unknown = true))
storage(stable())
storage(stable(data_memory_id = 1))
storage(stable(...), heap())
storage = "heap"
```

There should be no transitional acceptance of the old reserved heap diagnostic.
Once `storage(heap())` is admitted, it is admitted as the canonical heap form.

### Runtime Store Types

0.169 should introduce backend variants inside store wrappers or behind small
private backend structs. The public runtime shape should stay store-wrapper
oriented.

Acceptable internal shape:

```rust
enum DataStoreBackend {
    Stable(StableDataStoreBackend),
    Heap(HeapDataStoreBackend),
}

pub struct DataStore {
    backend: DataStoreBackend,
}
```

Equivalent private backend wrappers are acceptable if they keep callers on the
same `DataStore` API.

Heap implementations should use ordered in-memory maps:

```rust
std::collections::BTreeMap<RawDataStoreKey, RawRow>
std::collections::BTreeMap<RawIndexStoreKey, IndexEntryValue>
std::collections::BTreeMap<RawSchemaKey, RawSchemaSnapshot>
```

The heap backend must preserve canonical raw key order for visitor traversal.

### Generated Wiring

Generated stable wiring remains unchanged:

```text
Store::new_stable(...)
-> ic_memory_key!(...)
-> StoreAllocationIdentities::new(...)
```

Generated heap wiring should not use `ic_memory_key!(...)` and should register
explicit absent allocation identities:

```text
Store::new_heap(...)
-> DataStore::init_heap()
-> IndexStore::init_heap()
-> SchemaStore::init_heap()
-> StoreAllocationIdentities::absent()
```

The absence must be explicit in generated output; heap wiring must not share the
stable allocation identity path.

### Diagnostics

Diagnostics should separate absent heap allocation identity from live heap schema
metadata. Heap stores should report:

```text
storage: heap
allocation:
  memory_id: -
  stable_key: -
schema:
  version: <live version | ->
  fingerprint: <live fingerprint | ->
  durability: volatile
```

Heap schema metadata may be shown while the process is live, but it must be
labelled as volatile/live metadata. It must not appear as stable allocation
identity.

Stable diagnostics remain unchanged: stable stores continue to report stable
allocation identity and accepted schema metadata according to the 0.163 and
0.167 contracts.

Diagnostics must not imply heap stores are durable.

### Heap Schema Store Lifecycle

For `storage(heap())`, schema snapshots in `SchemaStore` are live-only.

On initialization, heap schema metadata may be built from the generated schema
declaration and stored in the heap schema store for runtime validation,
diagnostics, and reconciliation while the process is live.

On upgrade/restart, heap schema snapshots are not recovered from stable memory.
They are reconstructed from generated declarations or start empty according to
the heap-store startup path. No heap schema snapshot is treated as durable
accepted history.

### Commit And Recovery

Commit markers are stable-memory-backed today. 0.169 should fail closed if a
heap store reaches a durable commit/recovery path that would imply upgrade
replay support.

Decision:

```text
heap writes are allowed while live;
heap rows/indexes/schemas are volatile;
commit markers remain stable-store recovery infrastructure;
upgrade recovery does not restore heap store contents.
```

A mixed canister may commit stable-store mutations through the existing durable
commit path while heap-store mutations remain live-only side effects. Any code
path that would imply heap replay, heap recovery, or heap durability must fail
closed in 0.169.

### Mixed Store Write Semantics

0.169 permits a canister to contain both stable and heap stores.

The durability contract is per store:

- stable-store mutations participate in the existing stable commit/recovery
  contract;
- heap-store mutations are live-only and are lost on upgrade/restart;
- no 0.169 operation may advertise atomic durable commit across stable and heap
  stores.

If one public write path mutates both stable and heap stores, only the stable
portion has upgrade recovery semantics. The heap portion is volatile. Tests and
diagnostics must not imply otherwise.

### Relation Policy

0.169 does not expand relation semantics.

Until a dedicated cross-backend relation policy is designed:

- heap -> heap relations may use existing live validation semantics;
- heap -> stable relations may use existing validation semantics while live;
- stable -> heap relation declarations reject at schema admission because a
  stable store must not own referential integrity against a volatile target;
- stale accepted strong-relation metadata that reaches runtime with a stable
  source and heap target must also fail closed before target lookup or
  reverse-index mutation;
- any relation edge that would require durable referential integrity from a
  stable store into a volatile heap store must fail closed.

## Source-Audit Commands

Heap/stable branching outside store construction should be rare and justified:

```bash
rg -n "StoreStorage::Heap|DataStoreBackend::Heap|IndexStoreBackend::Heap|SchemaStoreBackend::Heap" \
  crates/icydb-core/src crates/icydb-build/src crates/icydb-schema/src crates/icydb-schema-derive/src
```

Expected result after implementation:

- schema/build/derive code may branch on `StoreStorage`;
- store-wrapper internals may branch on backend variants;
- executor, query, relation, reconciliation, diagnostics, and commit code should
  not own storage-mode policy.

Heap/stable runtime branching outside store internals:

```bash
rg -n "match .*StoreStorage|StoreStorage::Heap|DataStoreBackend::Heap|IndexStoreBackend::Heap|SchemaStoreBackend::Heap" \
  crates/icydb-core/src/db \
  --glob '!**/data/**' \
  --glob '!**/index/**' \
  --glob '!**/schema/**' \
  --glob '!**/registry/**'
```

Expected result: no executor, relation, reconciliation, diagnostics, cursor, or
commit module owns heap-vs-stable policy.

Direct backend-map mutation:

```bash
rg -n "BTreeMap<|\\.insert\\(|\\.remove\\(|\\.clear\\(" crates/icydb-core/src/db \
  --glob '!**/data/**' \
  --glob '!**/index/**' \
  --glob '!**/schema/**'
```

Expected result: higher layers call store-wrapper methods; they do not mutate
backend map internals directly. Local overlays, tests, and ordinary in-memory
collections must be classified separately.

Stable wiring should remain stable:

```bash
rg -n "ic_memory_key|StoreAllocationIdentities::new|StoreAllocationIdentities::absent" \
  crates/icydb-build/src crates/icydb-core/src/db/registry
```

Heap diagnostics should be explicit:

```bash
rg -n "storage: heap|StoreSnapshotStorageMode::Heap|storage\\(heap" \
  docs crates testing
```

## Closeout Evidence

0.169 closes the heap store runtime backend matrix with source-audit evidence
plus focused validation.

### Codec Preservation

The ABI-preservation audit compared the 0.169 line against the 0.168.6
runtime-boundary baseline:

```bash
git diff --name-only c73511eb8..HEAD -- \
  crates/icydb-core/src/db/data/persisted_row \
  crates/icydb-core/src/db/index/entry.rs \
  crates/icydb-core/src/db/index/key \
  crates/icydb-core/src/db/schema/codec.rs \
  crates/icydb-core/src/db/schema/snapshot.rs \
  crates/icydb-core/src/db/codec
```

Result: no files. Heap store admission changed store wrappers and traversal
owners, not row payload codecs, index-entry/key codecs, schema-snapshot codecs,
or shared DB codec helpers.

Focused validation:

```bash
cargo test -p icydb-core index
cargo test -p icydb-core schema
cargo test -p icydb-core commit
```

### Backend-Neutral Runtime Boundary

Heap/stable branching outside store construction and store-wrapper internals was
audited with:

```bash
rg -n "match .*StoreStorage|StoreStorage::Heap|DataStoreBackend::Heap|IndexStoreBackend::Heap|SchemaStoreBackend::Heap" \
  crates/icydb-core/src/db \
  --glob '!**/data/**' \
  --glob '!**/index/**' \
  --glob '!**/schema/**' \
  --glob '!**/registry/**'
```

Result: no hits. Executor, query, relation, reconciliation, diagnostics,
cursor, and commit modules do not own heap-vs-stable policy.

The broader heap/backend scan is intentionally limited to schema/build/derive
storage admission and store-wrapper internals:

```bash
rg -n "StoreStorage::Heap|DataStoreBackend::Heap|IndexStoreBackend::Heap|SchemaStoreBackend::Heap" \
  crates/icydb-core/src crates/icydb-build/src crates/icydb-schema/src crates/icydb-schema-derive/src
```

Result: hits are schema/build/derive storage-mode admission or
`DataStore`/`IndexStore`/`SchemaStore` backend internals.

Direct mutation auditing confirms higher layers call store-wrapper methods
rather than mutating backend maps directly:

```bash
rg -n "BTreeMap<|\\.insert\\(|\\.remove\\(|\\.clear\\(" crates/icydb-core/src/db \
  --glob '!**/data/**' \
  --glob '!**/index/**' \
  --glob '!**/schema/**'
```

Result: hits are ordinary in-memory collections, tests, diagnostics maps,
commit-window overlays, or store-wrapper method calls such as
`store.insert(...)`, `store.remove(...)`, and `store.clear(...)`. Backend map
mutation remains owned by store modules.

Focused validation:

```bash
cargo test -p icydb-core diagnostics
cargo test -p icydb-core commit
cargo test -p icydb-core schema
cargo test -p icydb-core index
```

### 0.169.3 — Commit / Recovery / Volatility Decision

- Enforce the live-only heap write contract.
- Keep stable commit/recovery behavior unchanged.

  durability.

- Reject stable-source relation declarations that target heap-backed entities.
- Reject stale accepted strong-relation metadata that would validate a stable
  source row against a heap target at save time.
