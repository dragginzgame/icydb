# **0.135 Bounded Text (Max Length Only)**

## Thesis

`0.135` introduces **optional maximum length constraints for text fields**, enforced at the schema write boundary, without changing runtime or storage representations.

> **0.135 adds `max_len` as a schema-level invariant for text fields, enforced centrally at write time, while leaving all other validation (e.g. min length) to the existing `#[validator]` system.**

This preserves a single text representation and avoids introducing parallel validation or storage paths.

---

## Goals

* Support `VARCHAR(n)`-style **maximum length constraint** for text
* Enforce constraint **once at write boundary**
* Maintain:

  * single `Value::Text(String)` runtime type
  * single UTF-8 storage encoding
* Avoid duplication with `#[validator]`

---

## Non-Goals

* No `min_len` at schema level
* No changes to:

  * storage encoding
  * index encoding
  * planner/executor behavior
  * query semantics
* No new runtime types (`Value::Varchar`)

---

## Current Validation Model

IcyDB currently has **two distinct validation systems**:

### 1. Schema / slot validation (core)

* Owned by `FieldKind`, `FieldModel`
* Enforced:

  * during write preparation
  * before persistence
* Examples:

  * type compatibility
  * decimal scale

### 2. `#[validator]` (derive, user-defined)

* Defined in `icydb-schema-derive`
* Arbitrary user logic
* Used for:

  * business rules
  * formatting constraints
  * cross-field validation

---

## Design Principle

> **Schema defines storage invariants.
> `#[validator]` defines application-level rules.**

---

## Schema Representation

Extend `FieldKind::Text`:

```rust
Text {
    /// Maximum allowed text length, counted as Unicode scalar values.
    max_len: Option<u32>,
}
```

Alternative:

```rust
Text(TextConstraint)

struct TextConstraint {
    /// Maximum allowed text length, counted as Unicode scalar values.
    max_len: Option<u32>,
}
```

### Properties

* Stored only in schema/model
* Not embedded in:

  * `Value`
  * storage encoding
  * index encoding

### Schema visibility

Because `max_len` is schema metadata, it must be visible anywhere schema shape
is reported or compared:

* schema fingerprinting
* schema description / debugging output
* relation or field compatibility checks where text field identity matters

---

## Validation Model

### Single enforcement point

Extend existing write validation:

```rust
fn validate_write_value(field: &FieldModel, value: &Value) -> Result<()>
```

Add:

```rust
match (field.kind(), value) {
    (FieldKind::Text { max_len: Some(max) }, Value::Text(s)) => {
        let len = s.chars().count();

        if len > max {
            return Err(...);
        }
    }
    _ => {}
}
```

Implementation note:

* do not hand-write `chars().count()` at multiple callsites
* keep scalar-count validation behind one schema-owned helper
* classify over-limit input as user input rejection, not corruption or internal error

---

## Length Semantics

* Length = **Unicode scalar count** (`chars().count()`)
* Not byte length

Rationale:

* Matches SQL expectations
* Avoids UTF-8 ambiguity

---

## Enforcement Scope

### Enforced

* SQL INSERT / UPDATE
* Fluent writes
* Generated write paths
* Internal mutation paths

### Not enforced

* SELECT queries
* predicates
* expression evaluation
* index encoding
* persisted-row decoding

Long text literals are still valid in read/query contexts. For example,
`WHERE name = 'very long literal'` may parse and plan normally; rejection only
occurs if that value is written into a bounded text field.

---

## Storage & Indexing

No changes:

* UTF-8 encoding remains unchanged
* Index ordering remains unchanged
* No branching on `max_len`

---

## Interaction with `#[validator]`

### Clear separation

| System             | Purpose           |
| ------------------ | ----------------- |
| Schema (`max_len`) | storage invariant |
| `#[validator]`     | application logic |

### Rule

> **Any constraint that must always hold for stored data MUST be expressed in the schema.**

### Implications

* `max_len` belongs in schema
* `min_len` belongs in `#[validator]`

---

### Example

```rust
field(ident = "name", value(item(prim = "Text", max_len = 255)))
```

With application-level validation, bounded text remains a schema directive on
the item contract:

```rust
field(
    ident = "name",
    value(item(prim = "Text", max_len = 255, validator(path = "base::validator::len::Min", args(1))))
)
```

Conceptually:

```rust
Text(max_len = 255)
#[validator(min_len = 1)]
name: String
```

* `max_len` → always enforced
* `min_len` → optional/user-defined

---

### Execution boundary

```text
input → optional user validation → schema validation → persist
```

User validators may run before write preparation, but schema validation is the
final required gate before persistence.

---

### Non-goal

* Do not duplicate `max_len` in `#[validator]`
* Schema is the canonical source for storage constraints

---

## Schema Macro

Support the item directive, mirroring decimal `scale`:

```rust
field(ident = "name", value(item(prim = "Text", max_len = 255)))
```

### Behavior

* Parsed by `icydb-schema-derive` as item metadata
* Emitted into `FieldKind::Text`
* Compile-time validation:

  * must be positive
  * only valid on text fields
  * rejected for custom `is = "..."`, relation fields, and non-text primitives

---

## Invariants

1. Single runtime representation:

   * `Value::Text(String)`

2. Single storage representation:

   * UTF-8 bytes

3. Single validation point:

   * write boundary

4. No duplication:

   * no scattered `.len()` checks

5. No execution impact:

   * executor unaware of constraints

6. Optional / nested values:

   * optional text validates only present values
   * recursive container support must use the same schema-owned validator
   * no separate list/set/map text-length checks

---

## Implementation Checklist

* Schema derive parses `item(prim = "Text", max_len = N)`
* `FieldKind::Text { max_len }` carries the invariant
* Schema fingerprinting and schema description include `max_len`
* One write-boundary validator enforces Unicode scalar-count length
* Storage, index, `Value`, predicates, and expression evaluation remain unchanged
* SQL and fluent writes share the same validation gate

---

## Comparison to Decimal(scale)

| Feature             | Decimal(scale) | Text(max_len) |
| ------------------- | -------------- | ------------- |
| Schema-owned        | yes            | yes           |
| Write-time enforced | yes            | yes           |
| Storage impact      | yes            | no            |
| Runtime type change | partial        | none          |

* schema-owned `max_len`
* enforced once at write boundary
* no runtime or storage changes

while preserving a clean separation from the `#[validator]` system.
