# 0.77 SQL Completeness Status

## Purpose

This status note tracks whether IcyDB SQL is complete within the intended
single-entity boundary.

The boundary is strict:

* no joins
* no subqueries
* no window functions

This note is not a roadmap for widening SQL beyond that boundary.
It is a completeness audit for the surface we already intend to ship.

---

## Canonical Boundary

The current public SQL surface is:

* `execute_sql_query::<E>(...)`
* `execute_sql_update::<E>(...)`

The current single-entity SQL contract is defined in:

* `docs/contracts/SQL_SUBSET.md`

The implementation-facing surface mapping is defined in:

* `docs/architecture/sql-surface-mapping.md`

The completion bar for `0.77` is stricter than “implemented somewhere.”
The admitted single-entity SQL surface is only complete when it is:

* represented in SQL terms
* represented in one canonical fluent or typed form
* frozen in the contract docs
* directly tested on the live surfaces

## Section 2: Partial / Inconsistent Areas

These areas work, but they are not fully complete yet.

### Projection

Current state:

* field projection works
* alias labeling works
* narrow computed text projection works

Why still partial:

* alias reuse in ordering is intentionally narrow and not exhaustively covered
* narrow computed text projection now has one shared fluent representation, but
  only through projection terminals rather than one broader row-projection
  query surface
* grouped/computed/projection interactions are covered, but not with the same
  breadth as scalar projection

Primary references:

* `crates/icydb-core/src/db/session/tests/sql_projection.rs`
* `crates/icydb-core/src/db/session/tests/sql_grouped.rs`

### Grouped `SELECT`

Current state:

* grouped key + aggregate queries are implemented
* grouped `HAVING` exists
* grouped explain exists

Why still partial:

* grouped ordering restrictions are subtle
* grouped projection ordering rules are strict and easy to drift
* grouped edge breadth is weaker than scalar read coverage

Primary references:

* `crates/icydb-core/src/db/session/tests/sql_grouped.rs`

### Global aggregate `SELECT`

Current state:

* implemented in the SQL executor
* directly covered in core

Why still partial:

* aggregate terminals are now the clearest canonical fluent representation, but
  projection-shaped aggregate parity is still not the same thing as one
  row-producing fluent aggregate query surface

Primary references:

* `crates/icydb-core/src/db/session/tests/sql_aggregate.rs`
* `crates/icydb-core/src/db/session/tests/query_lowering.rs`

### `RETURNING`

Current state:

* `DELETE RETURNING`
* `INSERT RETURNING`
* `UPDATE RETURNING`

all exist for the admitted shapes.

Why still partial:

* core runtime coverage is stronger than public facade coverage
* unsupported returning-projection edge coverage exists, but can still miss
  alias/projection breadth questions

Primary references:

* `crates/icydb-core/src/db/session/tests/sql_delete.rs`
* `crates/icydb-core/src/db/session/tests/sql_write.rs`
* `crates/icydb-core/src/db/session/tests/sql_surface.rs`

### `EXPLAIN`

Current state:

* `EXPLAIN SELECT`
* `EXPLAIN DELETE`
* `EXPLAIN JSON SELECT`
* `EXPLAIN JSON DELETE`
* grouped/global aggregate explain

Why still partial:

* execution explain still uses the shared execution-tree contract rather than a
  delete-specific outward shape, so the stable proof needs to stay explicit

Primary references:

* `crates/icydb-core/src/db/session/tests/sql_explain.rs`
* `crates/icydb-core/src/db/session/tests/explain_execution.rs`

### Introspection

Current state:

* `DESCRIBE`
* `SHOW INDEXES`
* `SHOW COLUMNS`
* `SHOW ENTITIES`

are all covered in core.

Why still partial:

* `SHOW TABLES` is admitted only as an alias for `SHOW ENTITIES`, so that alias
  contract should keep one explicit direct proof instead of relying on parser
  behavior alone

Primary references:

* `crates/icydb-core/src/db/session/tests/sql_surface.rs`

## Section 3: Missing But In-Scope Features

These are not “new SQL features.”
They are missing completeness work within the admitted boundary.

### 1. Keep direct `SHOW TABLES` alias proof

`SHOW TABLES` is admitted only as an alias for `SHOW ENTITIES` and should have
one explicit test proving the alias contract directly.

### 2. Keep direct `EXPLAIN EXECUTION DELETE` proof

This is admitted in the contract and now has one explicit test.
That proof should remain direct rather than becoming transitive.

### 3. Keep the dedicated SQL canister smoke suite small and explicit

The dedicated SQL test-canister surface exists:

* `canisters/test/sql/src/lib.rs`
* `testing/pocket-ic/tests/sql_canister.rs`

There is now one current PocketIC SQL suite proving:

* `query(sql)`
* `update(sql)`
* result packaging
* error mapping

That suite should stay small and explicit rather than regrowing the deleted
parity/perf matrix.

### 4. Keep direct public SQL payload proof

The public crate now has direct outward SQL packaging proof in:

* `crates/icydb/src/db/sql/mod.rs`

That proof is still narrower than full direct session-facade execution tests,
but it locks the outward payload contract and is reinforced by the dedicated
canister-edge suite.

## Section 4: Parity Gaps (SQL vs Fluent)

### SQL supports something fluent does not

#### Introspection

SQL supports:

* `DESCRIBE`
* `SHOW INDEXES`
* `SHOW COLUMNS`
* `SHOW ENTITIES`
* `SHOW TABLES` aliasing `SHOW ENTITIES`

Fluent does not have equivalent query-builder forms.
This is acceptable because these are operational SQL surfaces, not ordinary
query semantics.

#### Text explain surface

SQL supports:

* `EXPLAIN ...`
* `EXPLAIN EXECUTION ...`
* `EXPLAIN JSON ...`

Fluent has explain helpers, but not one text-SQL explain surface.
This is also acceptable as an operational SQL boundary.

### Fluent supports something SQL does not

#### Cursor pagination surface

Fluent has:

* `page()`
* `execute_paged()`
* direct cursor-token continuation APIs

SQL does not expose the same first-class typed pagination API.
This is acceptable because SQL remains text-oriented and result-shaped rather
than typed-builder-driven.

#### Terminal helpers

Fluent has typed terminal helpers such as:

* `exists()`
* `not_exists()`
* `min()`
* `max()`
* `bytes()`
* `bytes_by(...)`

SQL does not expose the same helper vocabulary; it expresses some of those
semantics through admitted SQL itself and leaves others fluent-only.

#### Typed write helpers

Fluent/typed APIs still own:

* `create(...)`
* `insert(...)`
* `update(...)`
* `replace(...)`
* `delete::<E>()`

plus the typed/fluent `...returning...` family.

That is intentional and acceptable.

## Section 5: Final Checklist To Reach Completion

### Freeze bar to satisfy

Before `0.77` is considered complete, the admitted single-entity SQL surface
must be:

* represented in SQL terms
* represented in one canonical fluent or typed form
* frozen in the contract and surface-mapping docs
* directly tested on the live public surfaces

This means “complete” is not just parser or executor support.
It means contract, representation, and proof all line up.

### Contract holes to close

* keep one direct `SHOW TABLES` alias test
* add one direct `EXPLAIN EXECUTION DELETE` test

### Surface proofs to retain

* keep the small PocketIC SQL suite for the current canister `query(sql)` /
  `update(sql)` surfaces
* keep the direct public SQL payload-packaging tests in:
  * `crates/icydb/src/db/sql/mod.rs`

### Consistency holes to review

* review grouped edge breadth:
  * grouped ordering restrictions
  * grouped `HAVING` boolean breadth
  * grouped/distinct interaction breadth
* review write alias breadth if aliases remain admitted on all write forms

### Completion rule

SQL within the single-entity model should be treated as “complete enough” when:

* every admitted statement family has one direct proof
* public facade query/update SQL surfaces have direct proof
* canister SQL query/update endpoints have one small end-to-end smoke suite
* remaining SQL-vs-fluent differences are intentional operational differences,
  not accidental capability drift
