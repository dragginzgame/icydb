# 0.77 Addendum: SQL Coverage Gaps

## Purpose

This addendum tracks the remaining SQL coverage gaps after the `0.77`
surface cleanup.

The goal is not to restate the SQL contract.
The goal is to answer:

* which admitted SQL shapes are already covered well
* which ones are only covered in core unit tests
* which ones are still weak or missing
* which gaps matter most for the current product shape

This note is intentionally biased toward deletion-era cleanup discipline:

* if a surface is live, it should have direct coverage
* if a contract is admitted, it should have explicit proof
* if only one layer is tested, that is not full coverage

For the broader completeness readout across implementation, testing, and
fluent parity, see:

* `docs/design/0.77-sql-freeze/0.77-completeness-status.md`

---

## Current Product Shape

The live SQL product surface after `0.77` is:

* `execute_sql_query::<E>(...)`
* `execute_sql_update::<E>(...)`
* canister `query(sql)` forwarding to `execute_sql_query::<E>(...)`
* canister `update(sql)` forwarding to `execute_sql_update::<E>(...)`

The canonical fluent/query surface is now flatter too:

* fluent grouped queries execute through `execute()`
* typed `execute_query(&Query<E>)` no longer branches in the public facade

So SQL coverage should now be evaluated against:

* parser/lowering coverage
* core session/runtime coverage
* public `icydb` facade coverage
* canister endpoint coverage

For `0.77`, coverage is not considered sufficient until the admitted
single-entity SQL surface is:

* represented in SQL terms
* represented in one canonical fluent or typed form
* frozen in the docs
* directly proved on the live surfaces

## Coverage Grid

| area | current state | main remaining gap |
|---|---|---|
| `SELECT` | mostly covered | alias and facade/canister edge coverage |
| `WHERE` | mostly covered | boolean-composition breadth is weaker than basic predicate coverage |
| `ORDER BY` | mostly covered | grouped/alias/expression interaction edges |
| `LIMIT/OFFSET` | mostly covered | grouped interaction and end-to-end canister coverage |
| `DISTINCT` | partial | non-primary-key and grouped interaction edge breadth |
| `GROUP BY` | partial | projection ordering rules and grouped ordering/window edges |
| `HAVING` | partial | boolean composition breadth and aggregate-reference breadth |
| global aggregates | strong in core | facade/canister coverage still lags direct core/fluent parity |
| projection | partial | alias reuse and computed/grouped edge breadth |
| mutations | mostly covered | write alias breadth is still lighter than the main mutation/update surface |
| `RETURNING` | mostly covered | breadth can still improve, but the live canister update surface now has direct proof |
| `EXPLAIN` | strong | keep direct `EXPLAIN EXECUTION DELETE` proof explicit |
| introspection | partial | `SHOW TABLES` alias coverage now exists and should stay explicit |
| surface ownership / rejection | medium-high | core is strong and the dedicated canister query/update surface now has one direct proof |
| integration / end-to-end | medium-high | a small PocketIC SQL integration suite now covers the live dedicated SQL canister |

## What Is Already Strong

Core session SQL coverage is concentrated and broad in:

* `crates/icydb-core/src/db/session/tests/sql_surface.rs`
* `crates/icydb-core/src/db/session/tests/sql_scalar.rs`
* `crates/icydb-core/src/db/session/tests/sql_projection.rs`
* `crates/icydb-core/src/db/session/tests/sql_grouped.rs`
* `crates/icydb-core/src/db/session/tests/sql_aggregate.rs`
* `crates/icydb-core/src/db/session/tests/sql_delete.rs`
* `crates/icydb-core/src/db/session/tests/sql_write.rs`
* `crates/icydb-core/src/db/session/tests/sql_explain.rs`
* `crates/icydb-core/src/db/session/tests/query_lowering.rs`

These suites already prove most of the semantic contract for:

* scalar reads
* projections
* grouped reads
* global aggregates
* explain surfaces
* writes and `RETURNING`
* query-lowering fail-closed boundaries
* metadata/introspection ownership

The main remaining problem is not “no core SQL tests.”
The main problem is that some admitted or shipped surfaces are only covered at
the core-session layer.

---

## Confirmed Gaps

### 1. Keep the dedicated PocketIC SQL integration suite small and explicit

The live SQL canister endpoints exist in:

* `canisters/test/sql/src/lib.rs`
* `testing/pocket-ic/tests/sql_canister.rs`

There is now one small `testing/pocket-ic/tests/sql_canister.rs` suite proving:

* canister `query(sql)` -> `execute_sql_query::<E>(...)`
* canister `update(sql)` -> `execute_sql_update::<E>(...)`
* canister error mapping
* canister result-shape packaging

That suite should stay intentionally small instead of restoring the deleted
large parity/perf matrix.

### 2. `SHOW TABLES` alias needs explicit coverage to stay locked

`SHOW TABLES` is part of the admitted contract, but current core session SQL
tests cover:

* `SHOW ENTITIES`
* `SHOW INDEXES`
* `SHOW COLUMNS`
* `DESCRIBE`

and now include one direct `SHOW TABLES` alias proof.

That alias proof should remain explicit because `SHOW TABLES` is an admitted
alias of `SHOW ENTITIES`, not just parser trivia.

### 3. `EXPLAIN EXECUTION DELETE` needs explicit coverage to stay locked

Current explain coverage includes:

* `EXPLAIN SELECT`
* `EXPLAIN DELETE`
* `EXPLAIN JSON SELECT`
* `EXPLAIN JSON DELETE`
* grouped/global aggregate explain forms

This admitted shape now has one current direct proof for:

* `EXPLAIN EXECUTION DELETE ...`

That proof should remain direct because the execution-tree contract is shared
rather than delete-specific.

### 4. Public `icydb` facade SQL coverage is still lighter than core coverage

The public outward SQL payload layer now has direct tests in:

* `crates/icydb/src/db/sql/mod.rs`

The live canister suite also exercises the public session SQL methods
transitively.

Coverage is still weighted toward the core session SQL runtime rather than
direct in-process session-facade execution tests.

That means:

* outward payload packaging is directly locked
* outward error mapping is covered at the canister edge
* direct facade execution breadth is still lighter than core breadth

So this is no longer a missing edge surface, but it is still a lighter proof
area than the core SQL runtime.

### 5. Write alias coverage is uneven

There is direct update-alias coverage:

* `crates/icydb-core/src/db/session/tests/sql_write.rs`

But direct insert/delete alias coverage is weaker or absent in the current core
session SQL suites.

This is a lower-priority gap than the higher-priority items above, but still worth tracking if
those write alias forms remain admitted.

---

## What Is Not A Gap

These areas are no longer the main concern:

* old SQL dispatch parity
* grouped-vs-scalar public wrapper parity
* generated canister SQL routing parity
* route/lane helper coverage for removed public surfaces

Those were the `0.77` cleanup target and should not drive new coverage work.

Coverage work after `0.77` should follow the current product surface, not the
deleted one.

---

## Priority Order

### Priority 1: Preserve the live canister SQL suite

Keep the small PocketIC SQL suite against the dedicated SQL test canister that
proves:

* one `query(sql)` smoke path
* one `update(sql)` smoke path
* one grouped read
* one `RETURNING`
* one introspection command
* one deterministic rejection

This should stay small. The goal is not to restore the old broad parity
harness. The goal is to prove the live canister surface exists and is wired
correctly.

### Priority 2: Explicit admitted-shape holes

Add direct core tests for:

* `SHOW TABLES` aliasing `SHOW ENTITIES`
* `EXPLAIN EXECUTION DELETE`

These are cheap and should be closed quickly.

### Priority 3: Broaden direct facade proof only if needed

Current direct public-crate payload proof already exists in:

* `crates/icydb/src/db/sql/mod.rs`

Only add more direct session-facade execution tests if coverage drift shows up
there in practice.

### Priority 4: Secondary breadth gaps

After the higher-value gaps are closed:

* insert/delete alias breadth
* `HAVING` boolean composition breadth
* grouped ordering / offset edge breadth
* distinct edge breadth

---

## Tracking Rule

For the remainder of `0.77.x`:

* admitted SQL shapes should have at least one direct proof
* public facade surfaces should have at least one direct proof per statement family
* canister SQL endpoints should have one small end-to-end smoke suite

If a shape is admitted in `docs/contracts/SQL_SUBSET.md` but only exercised
through lower-level helpers, it should still be treated as an open coverage gap
until one direct surface-level test exists.
