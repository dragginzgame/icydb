# 0.77 Design: SQL Surface Consolidation and Contract Freeze

## Preamble

* **scope**: consolidate and stabilize the SQL surface completed across `0.76`
* **goal**: freeze the admitted single-entity SQL contract, harden boundary behavior, and unify row-returning semantics without reopening large semantic expansion
* **theme**: finish and prove the surface we already have
* **non-goals**:

  * no `JOIN`
  * no subqueries
  * no window functions
  * no generic predicate-expression promotion in `WHERE`
  * no broad projection-expression widening
  * no new executor architecture
  * no cost-based or algebra-rewrite planner work

---

## Problem Statement

`0.76` landed most of the intended non-`JOIN` single-entity SQL surface:

* scalar projection and aliases
* grouped and global aggregate execution
* grouped/distinct policy slices
* narrow computed text projection
* narrow SQL writes
* typed dispatch write convergence
* managed timestamp ownership convergence
* sparse structural materialization fixes
* typed authored insert surface

The remaining risk is no longer “major missing SQL capability.”

The remaining risk is that the surface is still slightly too easy to misunderstand:

* contract wording has lagged implementation
* some surfaces expose narrower subsets than the admitted language boundary
* some new `0.76` boundaries are only lightly covered outside happy paths
* row-returning behavior is still more split than it should be
* canister/query-only vs write-capable surfaces need firmer statement

So `0.77` should not be another expansion release.
It should be the release where the SQL surface becomes **explicit, coherent, and stable**.

---

## Design Goals

### 1. Freeze the admitted SQL boundary

Make the supported single-entity SQL subset explicit, normative, and stable.

### 2. Separate language contract from surface mapping

The main contract should answer:

> Is this SQL shape part of IcyDB SQL?

A separate mapping document should answer:

> Which public surfaces expose that shape today?

### 3. Harden boundary behavior

Strengthen tests around newly landed `0.76` boundaries, especially negative and fail-closed cases.

### 4. Unify row-returning semantics

Where a statement returns rows, it should converge on one row/projection result contract, regardless of whether those rows came from:

* `SELECT`
* mutation `... RETURNING`
* future fluent `.returning(...)`

### 5. Avoid reopening core semantic complexity

Do not widen into generic expression predicates, broader SQL function semantics, or planner rewrites in this release.

---

# Part 1: Release Definition

## 1.1 What 0.77 is

`0.77` is:

* SQL contract freeze
* SQL surface mapping cleanup
* boundary-condition test hardening
* returning/result-shape convergence
* write-surface coherence cleanup where needed
* public-surface clarification

## 1.2 What 0.77 is not

`0.77` is not:

* a new expression system
* a large planner/runtime rewrite
* a broader SQL grammar push
* a “halfway to general SQL” release

---

# Part 2: Contract Freeze

## 2.1 Normative contract

`docs/contracts/SQL_SUBSET.md` becomes the single normative statement of:

* supported SQL statement families
* supported clauses and semantics
* supported expression subset
* explicit fail-closed boundaries
* error behavior
* stability requirements

It should describe the **admitted SQL language boundary**, not implementation-lane history.

## 2.2 Surface mapping

`docs/architecture/sql-surface-mapping.md` becomes the implementation-facing companion note describing:

* `query_from_sql(...)`
* `execute_sql(...)`
* `execute_sql_grouped(...)`
* `execute_sql_aggregate(...)`
* `execute_sql_dispatch(...)`
* generated canister SQL surface

It should explain:

* which surfaces expose which admitted statement families
* where result-shape differs
* where convergence is strong
* where the surface is still intentionally split

## 2.3 Freeze rule

After `0.77`, if an admitted SQL shape changes, the contract document must change in the same patch.

---

# Part 3: Supported Surface After 0.77

## 3.1 Admitted SQL boundary

The admitted single-entity SQL subset after `0.77` should include:

* `SELECT`

  * scalar rows
  * scalar projection
  * scalar `DISTINCT`
  * grouped aggregate queries
  * constrained global aggregate queries
  * admitted computed text projection
* `INSERT`

  * including admitted typed-dispatch `INSERT ... SELECT` source shapes
* `UPDATE`
* `DELETE`
* `EXPLAIN`
* `DESCRIBE`
* `SHOW INDEXES`
* `SHOW COLUMNS`
* `SHOW ENTITIES`
* `SHOW TABLES`

## 3.2 Explicitly rejected boundary

The contract should continue to fail closed for:

* joins
* subqueries
* CTEs
* set operations
* window functions
* quoted identifiers
* multi-statement input
* generic expression widening outside the admitted subset
* broad function semantics outside the admitted subset
* multi-entity semantics

## 3.3 Policy clarification areas

`0.77` should explicitly freeze:

* grouped `ORDER BY` restrictions
* grouped `HAVING` restrictions
* grouped `DISTINCT` restrictions
* alias limits
* canister query-only write policy if retained
* admitted SQL write boundaries
* admitted `INSERT ... SELECT` source limits

# Part 4: Row-Returning Convergence

## 4.1 Problem

Right now, the surface still over-distinguishes by provenance:

* `SELECT` returns row/query payloads
* writes return mutation outcomes
* future mutation `... RETURNING` wants to behave like query rows
* fluent/typed `.returning(...)` should mirror SQL, not invent another result family

The correct split is not:

* query vs mutation

The correct split is:

* row-returning statements vs non-row-returning mutation outcomes

## 4.2 Target rule

If a statement returns rows, it should use the same row/projection result contract.

This should apply to:

* `SELECT`
* `INSERT ... RETURNING`
* `UPDATE ... RETURNING`
* `DELETE ... RETURNING`
* fluent `.returning(...)`

## 4.3 Non-goal

This does **not** require unifying all executor internals or forcing a giant umbrella runtime result type.

The convergence target is the **public row-returning contract**, not deep executor payload identity.

## 4.4 0.77 outcome

`0.77` should at least define and, if feasible, implement this rule:

> row-returning statements share one row/result family; non-returning writes remain mutation-result families

If full implementation is too large for one slice, `0.77` should at minimum:

* define the rule
* align SQL `RETURNING` design with it
* align fluent `.returning(...)` design with it
* avoid landing conflicting result-shape work

# Part 5: Testing and Proof Work

## 5.1 Highest-priority gaps

The highest-value testing work is around `0.76` boundary conditions, not more example expansion.

### A. Sparse persisted-row bridge

Add direct tests for:

* sparse missing-slot materialization
* derive-default-backed omission behavior
* malformed sparse serialized patches
* duplicate-slot cases
* out-of-order slot cases
* fail-closed behavior on invalid patch forms

### B. Sanitize write-context lifecycle edges

Add focused tests for:

* `Insert`
* `Replace`
* `Update`

Especially:

* `CreatedAt` preserve-on-update behavior
* `UpdatedAt` refresh behavior
* replace-path lifecycle semantics
* mode-aware sanitizer behavior under explicit write context

### C. SQL write coercion and messaging

Add focused tests for:

* PK literal widening
* field coercion widening
* unsupported entity messaging
* generated/managed field authored-write rejection
* admitted omission vs rejection boundaries

### D. Public facade parity

Add a smaller but meaningful facade-level pass for:

* insert-typed behavior
* SQL dispatch/write surface behavior
* grouped/explain parity where under-sampled
* row-returning surface expectations if started in `0.77`

## 5.2 Testing philosophy

`0.77` should prioritize:

* invariant tests
* boundary tests
* fail-closed tests
* convergence tests

It should not primarily spend effort on more one-off sibling examples.

# Part 6: Write-Surface Consolidation

## 6.1 What is already settled

The following should be treated as settled after `0.76`:

* managed timestamps are declared by derive
* sanitizers own lifecycle policy
* save preflight executes lifecycle policy
* SQL dispatch is not a second timestamp owner
* sparse structural materialization is the real path before preflight
* typed authored insert exists as a narrow explicit companion type
* generated and managed fields are structurally un-authorable on that insert path

## 6.2 What 0.77 should do here

`0.77` should mostly:

* document this clearly
* add missing tests
* avoid reopening ownership questions that are already clean

## 6.3 Optional forward-compatible work

If the release includes a write-surface feature beyond testing, the most coherent next step is:

* typed/fluent `returning(...)` alignment

Not:

* broad new insert helpers
* full update companion-type expansion unless clearly needed
* new lifecycle ownership refactors

---

# Part 7: Public Surface Decisions

## 7.1 Canister SQL write policy

`0.77` should explicitly state one of:

* generated canister SQL remains query-only
* or generated canister SQL admits writes

This should no longer be discoverable only through tests or dispatch code.

## 7.2 Authored write types

`0.77` should treat the current typed insert companion as:

* a narrow authored-write contract
* not a revival of the old `CreateView` / `UpdateView` family

That distinction should be explicit in docs.

## 7.3 Surface availability wording

The docs should clearly distinguish:

* admitted SQL language boundary
* per-surface availability
* per-surface result shape

This prevents “supported everywhere” from being implied when the real contract is “supported by the admitted language, exposed variably by surface.”

---

# Part 8: Non-Goals for 0.77

The following should remain out of scope:

## 8.1 Predicate-expression promotion

Do not replace the specialized predicate model with general `CompareExpr(...)` in this release.

## 8.2 Generic projection-expression widening

Do not broaden the admitted computed-expression surface beyond the current narrow subset.

## 8.3 Planner algebra rewrites

Do not add expression rewrites like:

* `field + 1 > 10` → `field > 9`

## 8.4 Join-adjacent work

Do not drift into:

* joins
* relation planning
* multi-entity routing
* SQL completeness work beyond the admitted single-entity boundary

---

# Part 10: Rollout

## 10.1 Suggested order

1. finalize SQL contract docs
2. finalize surface mapping docs
3. add missing boundary-condition tests
4. make explicit public-surface policy decisions
5. only then land any row-returning convergence slice if still in budget

## 10.2 Success criteria

After `0.77`, we should be able to say:

* the single-entity SQL boundary is explicit and frozen
* surface splits are documented instead of implicit
* recent `0.76` write and structural paths are proven at their real failure boundaries
* row-returning convergence has a clear rule, even if only partially implemented
* the next release can choose between deeper convergence or genuine new semantics from a stable base

---

# Risks

## 10.1 Mistaking implementation convenience for contract

Mitigation:

* keep the normative contract separate from surface mapping

## 10.2 Over-scoping into new semantics

Mitigation:

* keep predicate-expression widening and planner rewrites out of scope

## 10.3 Under-testing newly load-bearing `0.76` seams

Mitigation:

* prioritize sparse patch, sanitize-context, coercion, and facade-parity tests

## 10.4 Premature deep result-model unification

Mitigation:

* unify row-returning public behavior first
* preserve specialized internal result families unless a stronger reason appears

---

# Final Statement

After `0.77`:

> IcyDB should have a frozen, explicit, single-entity SQL contract;
> a separate and honest public-surface mapping;
> hardened proof around the new write/materialization/lifecycle boundaries landed in `0.76`;
> and a clear convergence rule that row-returning statements should share one result contract, without reopening large new SQL semantics.
