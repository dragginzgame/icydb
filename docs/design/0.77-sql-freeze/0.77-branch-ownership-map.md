# 0.77 SQL Branch Ownership Map

This addendum freezes branch ownership for the single-entity SQL stack.

It is not a risk audit.
It is an ownership contract intended to stop branch rebound across parser,
lowering, and execution.

## 1. Allowed Layers

Every SQL `match` or `if` in the live runtime must belong to exactly one layer:

1. `parser`
2. `lowering`
3. `execution`

No branch site is allowed to be ‚Äúshared‚Äù across those roles.

## 2. Ownership Rule

### Parser

Parser branches own only:

- statement-family detection
- clause presence and clause ordering
- token-shape and delimiter handling
- admitted syntax and fail-closed syntax rejection
- alias spelling capture
- projection-item syntactic form

Parser branches must not own:

- entity/schema compatibility
- semantic grouping rules
- aggregate/query shape validation
- runtime result packaging
- execution-mode selection

### Lowering

Lowering branches own only:

- entity binding and identifier normalization
- semantic projection validation
- grouped/global aggregate shape validation
- HAVING/ORDER/projection semantic checks
- admitted text-function validation
- conversion from parsed SQL shapes into canonical query/update intent

Lowering branches must not own:

- tokenization or clause-order parsing
- runtime execution routing
- result rendering/payload packaging
- canister/public-surface admission messages

### Execution

Execution branches own only:

- query vs update surface admission
- explain/introspection dispatch
- runtime statement execution selection
- projection payload materialization
- grouped/scalar/runtime output packaging
- user-surface error guidance tied to execution ownership

Execution branches must not own:

- SQL grammar recognition
- semantic revalidation already settled in lowering
- entity-name normalization rules that belong to lowering

## 5. Rebound Prevention Rules

When adding SQL capability:

1. Add syntax branches only in `parser`.
2. Add semantic/admission branches only in `lowering`.
3. Add runtime-selection/payload branches only in `execution`.

The following are explicit regressions:

- parser branches that inspect execution/result concerns
- lowering branches that reparse or reinterpret token order
- execution branches that revalidate grouped/aggregate semantic rules already
  owned by lowering
- new public SQL entrypoints that duplicate execution-family routing
- moving one semantic decision into both lowering and execution ‚Äúfor safety‚Äù

Review rule for every new `if` or `match`:

1. Ask which layer owns it.
2. If the answer is `both`, reject it.
3. If the answer is `temporary`, reject it.
4. If the answer is `just in case`, reject it.
