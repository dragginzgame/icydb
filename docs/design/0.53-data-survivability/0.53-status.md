## Slice Tracker

### Slice A - Stable Persisted Row Format & Schema Evolution

Goal:

- Introduce explicit persisted-row format versioning and fail-closed
  compatibility-aware decoding.

Exit gates:

- [x] persisted row format versioning contract is implemented and documented
- [x] N-1 decode compatibility coverage exists for row payloads
- [x] decode incompatibility emits classified compatibility errors
- [x] structural corruption emits corruption-classified errors

### Slice B - Commit Marker Compatibility & Replay Semantics

Goal:

- Version commit markers and lock replay semantic compatibility boundaries.

Exit gates:

- [x] commit-marker format versioning is implemented
- [x] recovery decode path is compatibility-aware and fail-closed
- [x] replay semantic invariants are documented and regression-tested

### Slice C - Explicit Migration Engine (No Data Loss)

Goal:

- Execute migrations as explicit row-op commit streams with resume-safe
  recovery behavior.

Exit gates:

- [x] migration plan contract is implemented
- [x] migration execution is deterministic and replay-compatible
- [x] interrupted migrations are idempotent or explicitly resume-safe
- [x] migration failure paths are classified and observable
- [x] migration cursor state is persisted durably for upgrade-safe resume

### Slice D - Corruption Detection & Integrity Tooling

Goal:

- Expand integrity detection and observability while preserving taxonomy
  boundaries.

Exit gates:

- [x] integrity checks cover data/index and forward/reverse consistency
- [x] corruption vs compatibility vs misuse taxonomy is enforced
- [x] operator-facing integrity scan/reporting surfaces are documented

### Slice E - Upgrade, Replay & Recovery Hardening

Goal:

- Lock upgrade/replay survivability guarantees under the compatibility
  contract.

Exit gates:

- [x] upgrade replay retains readable persisted rows within contract bounds
- [x] crash-mid-migration recovery scenarios are regression-tested
- [x] no silent row orphaning/discard paths remain in upgrade/recovery flows

### Evidence (2026-03-13)

- Operator-facing integrity reporting surfaces are available at:
  `Db::integrity_report` and `DbSession::integrity_report`, backed by
  deterministic `IntegrityReport` snapshots.
- Upgrade/replay compatibility is regression-tested for persisted rows:
  - `recovery_startup_rebuild_decodes_previous_row_format_within_window`
  - `recovery_startup_rebuild_rejects_future_row_format_fail_closed`
- Crash-mid-migration recovery is regression-tested with marker-bound durable
  progress:
  - `migration_recovery_does_not_reexecute_step_when_marker_already_bound_progress`
  - `migration_recovery_resumes_remaining_steps_after_marker_bound_crash`
- Recovery now fails closed before marker clear if post-recovery integrity finds
  missing/divergent/orphan findings, preventing silent row orphaning/discard
  paths in upgrade/recovery flows.
