# IcyDB 0.53 - Data Survivability & Upgrade Safety

Status: **Planned for 0.53**.

Theme: **Data survivability, compatibility-aware upgrades, and explicit recovery
safety**.

## Intent

`0.53` establishes IcyDB's persisted-data survivability contract.

This line defines guarantees for persisted row readability, commit replay
compatibility, explicit migration execution, corruption classification, and
upgrade safety.

The goal of `0.53` is not new query capability. The goal is to ensure that
persisted data remains readable, recoverable, and explicitly evolvable across
upgrades without silent loss or orphaning.

## Core Guarantee

No upgrade, migration, or format evolution may silently make persisted data
unreadable.

All structural change must be explicit, compatibility-aware, recoverable, and
observable.

## Compatibility Contract

`0.53` introduces the persistence compatibility contract for IcyDB.

This contract guarantees:

* persisted data is not silently discarded due to format evolution
* format-breaking changes require explicit migration planning
* decode incompatibility surfaces as explicit classified error, never silent
  omission
* recovery and replay remain compatibility-aware and deterministic

Compatibility window for this line: **N-1 persisted-format compatibility**.

This means each release in the supported window must read persisted bytes from
its immediately previous persisted format version, with explicit errors for
anything outside that window.

Implementation progress and slice execution tracking live in:

`docs/design/0.53-data-survivability/0.53-status.md`

# 1. Stable Persisted Row Format & Schema Evolution

This milestone introduces explicit persisted-row format versioning and
compatibility-aware decode rules.

## Goals

* Add explicit version tagging to persisted row format
* Guarantee backward-compatible decode behavior within the compatibility window
* Prevent decode panics from rendering data unreachable
* Formalize allowed schema-evolution rules
* Ensure index derivation changes do not orphan data

## Outcomes

* Persisted rows include explicit format version
* Persisted-row decoding is version-aware and fail-closed
* Structural corruption is detected, not silently ignored
* Schema-evolution rules are documented and enforced
* Format-breaking changes require explicit migration plan
* Compatibility window is explicit and tested

## Non-Goals

* Automatic field-level migrations
* Implicit schema rewrite on upgrade
* Transparent structural coercion without version awareness

---

# 2. Commit Marker Compatibility & Replay Semantics

This milestone formalizes commit-marker wire compatibility and replay semantic
stability across upgrades.

## Goals

* Version the `CommitMarker` format explicitly
* Guarantee marker-format decode compatibility within contract bounds
* Guarantee replay semantic stability across minor upgrades
* Prevent marker decode failure from bricking recovery
* Freeze core commit semantics

## Outcomes

* Commit markers include explicit protocol version
* Recovery path supports version-aware decoding
* Commit replay semantics are formally documented
* Replay of previously valid commit markers preserves prior durability semantics
  within the compatibility contract
* Marker-format changes require explicit compatibility strategy

## Non-Goals

* Changing commit atomicity semantics
* Rewriting recovery model
* Introducing distributed transaction semantics

---

# 3. Explicit Migration Engine (No Data Loss)

This milestone introduces an explicit migration execution model built on the
existing row-op commit protocol.

Migrations must be:

* deterministic
* recoverable
* replayable
* non-lossy

## Goals

* Define a `MigrationPlan` abstraction
* Execute migrations as row-op streams
* Use existing commit-marker + recovery path
* Prevent partial migration states from becoming permanent
* Ensure migration interruption is recoverable without data loss

## Outcomes

* Migrations are structurally equivalent to normal row mutations
* Migration execution is crash-safe
* Migrations are idempotent or explicitly resume-safe under replay/recovery
  rules
* Data transformation is explicit and reviewable
* Upgrade failures cannot silently destroy data

## Non-Goals

* Automatic destructive migrations
* In-place schema rewrite without explicit recovery path
* Best-effort migration without recovery guarantees

---

# 4. Corruption Detection & Integrity Tooling

This milestone strengthens detection and observability of structural persisted
state failures.

Silent corruption is unacceptable.

## Goals

* Detect invalid row format during decode
* Detect index/data divergence
* Detect reverse-index inconsistencies
* Provide integrity scan utilities
* Expand error taxonomy across corruption and compatibility classes

## Outcomes

* Structural corruption results in explicit `Corruption`-classified errors
* Compatibility/version failures produce explicit compatibility-classified
  errors
* Integrity scans validate:
  * Data <-> Index consistency
  * Forward <-> Reverse relation consistency
* Snapshot tools expose structural counts
* Operators can distinguish corruption, incompatibility, and misuse

## Non-Goals

* Silent repair without operator visibility
* Automatic best-guess repair of corrupted data
* Masking corruption as user error

---

# 5. Upgrade, Replay & Recovery Hardening

This milestone formalizes no-data-loss guarantees across canister upgrades and
recovery.

## Goals

* Guarantee upgrades do not invalidate existing readable rows
* Guarantee recovery replay is version-stable within the compatibility contract
* Ensure recovery never discards valid persisted rows silently
* Add stress tests for crash-mid-migration scenarios
* Ensure upgrades do not require implicit persisted-data rewrite

## Outcomes

* Upgrade safety becomes a tested invariant
* Crash during migration is recoverable
* Replay semantics remain deterministic
* Marker + row version compatibility is validated

## Non-Goals

* Snapshot isolation across upgrades
* Cross-canister migration coordination
* Rolling format upgrades without a declared compatibility window

---

# Survivability Invariants Introduced in This Milestone

The following become explicit structural guarantees:

* Persisted rows are versioned
* Commit markers are versioned
* Persisted-format compatibility window is explicit
* Decode failure is explicit and classified
* Incompatibility is classified, never silent
* Migrations are commit-protocol-driven
* Migration execution is deterministic and resume-safe
* Valid persisted rows remain discoverable unless explicitly migrated or
  explicitly classified unreadable
* No upgrade may silently orphan persisted data
* No recovery path may discard valid rows without classification

---

# Explicit Exclusions (Not Guaranteed)

The following remain out of scope:

* indefinite backward compatibility across all historical formats
* cross-canister transactional upgrades
* distributed schema coordination
* automatic cascade repair of corrupted relations
* automatic index rebuild on structural drift (manual tooling only)
* implicit destructive schema evolution
* semantic reinterpretation of historical data beyond declared migration rules

---

# Summary

`0.53` is the **Data Survivability & Upgrade Safety** line.

It formalizes:

* persisted-row format stability and compatibility windowing
* commit-marker compatibility and replay semantic stability
* explicit migration semantics under commit/recovery rules
* corruption and compatibility error taxonomy boundaries
* upgrade/replay safety guarantees

This line is not about new query features.

It is about making IcyDB resilient to persisted-format evolution and upgrade
operations without silent data loss.
