# IcyDB 0.164 — Composite Relation Targets

## Status

In progress.

The first implementation slice moved reverse-relation index target identity
onto complete `PrimaryKeyValue` encoding. Composite relation targets are still
not admitted, but reverse-index storage and delete validation no longer require
the target identity to be projected through one scalar component.

Accepted strong-relation metadata also now carries target primary-key metadata
as an ordered component list. Existing scalar relation declarations populate a
one-component list; composite admission will extend that same descriptor rather
than introducing a parallel relation target model.

Relation target extraction from accepted source rows now returns
`PrimaryKeyValue` identities directly. Current scalar relation fields still
lower to `PrimaryKeyValue::Scalar`, but the mutation/delete validation
boundary no longer exposes a scalar-component-only target-key API.

A follow-up slice moved accepted strong-relation source metadata onto ordered
local component descriptors. Existing scalar relation declarations still
populate one local component, but relation validation now checks local
component count against accepted target primary-key component count before
using scalar target-key lanes.

Accepted strong-relation target identity construction also now accepts ordered
target primary-key component metadata directly. Existing scalar relation
fields pass a one-component target list; composite target admission can reuse
the same identity descriptor instead of adding another scalar-specific target
constructor.

Relation target extraction now returns an explicit none/one/many target-key
domain. Current scalar relations still produce none or one target key, while
set/list relation payloads can preserve many targets without encoding
cardinality as raw vector length.

Accepted relation target metadata extraction and target identity validation
now have one shared relation-domain owner. Save validation and reverse-index
preparation consume that named descriptor instead of duplicating tuple-shaped
target parsing or runtime target identity checks.

The shared descriptor also carries relation cardinality (`single`, `list`, or
`set`) explicitly, so collection relation semantics are no longer inferred by
re-matching accepted field kinds in each runtime relation path.

## Summary

0.164 extends relation targets from scalar primary keys to composite primary
keys.

0.162 made composite primary keys first-class for row identity, data keys,
index suffixes, cursors, and relation source identity. It deliberately left
relation targets scalar-only because the relation model still treated the
target key as one field value. 0.164 closes that gap by making relation target
identity an ordered primary-key tuple:

```text
accepted relation metadata
    + accepted target PrimaryKeyModel
    + source row local component fields
        -> PrimaryKeyValue::Composite(...)
        -> EncodedPrimaryKey / raw target data key
        -> strong validation and reverse-index identity
```

This is not a new primary-key encoding. It reuses the 0.162
`PrimaryKeyValue` / `EncodedPrimaryKey` machinery and removes the remaining
scalar-only relation target bridge.

## Background

After 0.162, these cases are valid:

| Source primary key | Target primary key | Relation behavior |
| --- | --- | --- |
| Scalar | Scalar | Works |
| Composite | Scalar | Works; source reverse identity is composite-aware |
| Scalar | Composite | Rejected / fail-closed |
| Composite | Composite | Rejected / fail-closed |

The unsupported rows are not a storage limitation in the data key layer. The
data key layer already accepts `PrimaryKeyValue::Composite`. The gap is the
relation metadata and execution path:

- schema relation admission compares one relation field against one target
  primary-key field
- save-time strong relation validation builds a target raw key from one runtime
  scalar value
- reverse relation indexes key by one target component plus the source
  primary-key value
- diagnostics name one target primary-key field even when the target has a
  tuple primary key

0.164 makes relation target identity tuple-aware across those boundaries.

The reverse-index target component now stores the complete target
`EncodedPrimaryKey` as one system index component. This keeps scalar and
composite target identity on the same key-owned storage path while the
relation-admission model is widened.

## Goals

- Support strong relations targeting entities with composite primary keys.
- Model a composite target relation as an ordered mapping from source local
  component fields to the accepted target primary-key fields.
- Validate relation tuple arity, order, component kinds, and nullability before
  the relation can become accepted schema state.
- Build target row identity as a full `PrimaryKeyValue`, never as the first
  target component.
- Keep scalar target relations working through the same canonical target-key
  path used for composite targets.
- Preserve composite source identity in reverse relation indexes.
- Use accepted schema/catalog state as the only runtime authority.
- Keep generated models as proposal/convenience input only.
- Fail closed for malformed, partial, mixed-null, stale, or unsupported
  relation tuples.
- Remove or narrow scalar-only relation target helpers instead of adding
  compatibility shims.

## Non-Goals

- No new primary-key wire or storage encoding.
- No change to `PrimaryKeyValue`, `PrimaryKeyComponent`, or
  `EncodedPrimaryKey` semantics.
- No support for `int_big` / `nat_big` primary-key components.
- No generated-model fallback reconstruction at runtime.
- No automatic migration from scalar relation fields to composite relation
  tuples.
- No relation field as a composite primary-key component in this slice.
- No relation target inference from Rust struct field order.
- No broad SQL foreign-key syntax redesign.
- No ergonomic generated composite relation key API unless it naturally falls
  out of the accepted schema model.

## Core Invariant

A relation target identifies one target row by the full ordered target primary
key.

```text
target_key = PrimaryKeyValue::Composite([
    component_0,
    component_1,
    ...
])
```

The relation layer must not validate, store, delete-check, reverse-index, or
query a composite target using only `target_key[0]`.

For scalar targets, the same invariant degenerates to:

```text
target_key = PrimaryKeyValue::Scalar(component_0)
```

## Relation Metadata Model

0.164 should promote relation target metadata from "one relation field has one
target key kind" to "one accepted relation edge owns an ordered target-key
mapping".

Conceptual accepted shape:

```text
RelationEdge {
    source_entity
    relation_name
    strength
    cardinality
    target_entity
    target_store
    local_components: [
        { source_field_id, source_slot, source_field_name },
        ...
    ]
    target_components: [
        { target_pk_field_id, target_slot, target_field_name, kind },
        ...
    ]
}
```

The durable authority should be field IDs/slots from the accepted schema
snapshot. Field names are diagnostic labels. Rust declaration order and
generated model reconstruction are not runtime authority.

For scalar target relations, `local_components.len() == 1`. Existing scalar
relation declarations can lower into this same accepted shape.

For composite target relations, `local_components.len() == target primary-key
arity`. The local fields remain ordinary persisted fields in the source row;
the relation edge is metadata over those fields.

## Declaration Strategy

The first slice needs a schema declaration that can express an ordered local
field list for the relation. The exact frontend syntax can reuse the current
schema vocabulary or introduce a small relation-edge declaration, but the
accepted model must contain explicit ordered component metadata.

Conceptual shape:

```text
relation author -> User {
    fields = [author_tenant_id, author_user_id]
}
```

The implied target field order is the target entity's accepted primary-key
order:

```text
User primary key = [tenant_id, id]
local fields    = [author_tenant_id, author_user_id]
```

An explicit target-field list should be needed only if IcyDB later supports
relations to non-primary unique keys. 0.164 should not add that feature.

The important boundary is that declaration parsing may come from generated
Rust or SQL frontend metadata, but runtime execution consumes only the accepted
schema/catalog relation edge.

## Type Compatibility

Admission compares each local component against the corresponding accepted
target primary-key component.

Accept:

- target scalar primary key with exactly one compatible local component
- target composite primary key with exactly one compatible local component per
  target primary-key field
- local field component kind matches the target component kind
- local field bounds match target component bounds where the kind carries
  bounds, such as decimal scale, text length, or blob byte length
- `i128` / `u128` components if they are admitted primary-key components
- source entities with scalar or composite primary keys

Reject:

- relation to a composite target with one scalar local field
- wrong tuple arity
- wrong local component order
- mismatched primitive kind or primitive bounds
- missing local field
- relation target with a non-admissible primary-key component kind
- relation target using `int_big` / `nat_big`
- local component field that is computed, generated-only, retired, or absent
  from the accepted source row contract
- any fallback path that projects `target primary key fields[0]`

This comparison should live at schema admission/reconciliation time, with
runtime validation rechecking the accepted contract enough to fail closed on
corrupt or stale metadata.

## Nullability

Composite relation nullability must be all-or-none in the first slice.

For optional single-target relations:

| Local tuple state | Meaning |
| --- | --- |
| all components `NULL` | no target |
| all components non-`NULL` | validate target exists |
| mixed `NULL` / non-`NULL` | reject row |

For required single-target relations, every component must be non-`NULL`.

For collection relations, every element must be a complete target tuple. If the
existing persisted collection shape cannot represent tuple elements without
ambiguity, collection relations to composite targets should remain rejected in
0.164. Do not encode a collection of composite targets as parallel local
arrays.

Partial tuples are never silently ignored because that would create one-way
relation edges and incorrect reverse-index entries.

## Runtime Flow

Save-time strong relation validation:

```text
accepted relation edge
    -> read local component slots from source row in declared relation order
    -> apply all-or-none nullability policy
    -> lower non-null values to PrimaryKeyComponent values
    -> build PrimaryKeyValue::Scalar or PrimaryKeyValue::Composite
    -> encode target raw data key with target entity tag
    -> check target data store membership
```

Delete-time strong relation blocking:

```text
candidate target PrimaryKeyValue
    -> scan reverse relation index by full target identity
    -> decode source row identity from reverse entry
    -> re-read source row if needed
    -> compare full target PrimaryKeyValue, not a scalar prefix
```

Reverse-index mutation preparation:

```text
old source row relation target keys
new source row relation target keys
    -> canonical sorted unique target PrimaryKeyValue set per side
    -> produce add/remove edge mutations keyed by target key + source key
```

The extraction helper should return a relation-target key domain, not a scalar
component domain:

```rust
enum RelationTargetKeys {
    None,
    One(PrimaryKeyValue),
    Many(Vec<PrimaryKeyValue>),
}
```

The exact type can differ, but the API must make scalar-only projection hard to
express.

## Reverse Index Identity

Reverse relation indexes must key an edge by full target identity and full
source identity.

Conceptually:

```text
reverse_index_id(source_entity, relation_ordinal)
    + target EncodedPrimaryKey
    + source EncodedPrimaryKey
```

The current index storage shape may encode the target identity as index
components and the source identity as the row suffix. 0.164 may keep that
physical layout only if a composite target is encoded as a single canonical
target identity segment or an otherwise unambiguous ordered segment list.

Required property:

```text
(tenant = A, user = 7) and (tenant = B, user = 7)
```

must produce distinct reverse-index target prefixes. Any target-prefix helper
that accepts only `PrimaryKeyComponent` should be removed, renamed, or narrowed
to scalar-only callers.

## Source Of Truth

Runtime relation metadata must come from accepted schema/catalog state.

Allowed sources:

- accepted source row decode contract
- accepted relation edge metadata
- accepted target entity primary-key metadata
- accepted target entity tag/name/store metadata
- persisted schema/catalog records

Disallowed sources:

- generated `EntityModel` / `IndexModel` fallback reconstruction
- generated `SchemaInfo` fallback views
- current Rust field order
- first-field scalar helpers
- stale schema snapshots used as reconstruction authority
- compatibility bridges that reinterpret a composite key as one scalar key

If accepted authority is unavailable for a relation role, the relation must
fail closed. Do not reconstruct relation target metadata from generated models.

## Diagnostics

Diagnostics should make tuple identity visible.

They should distinguish:

- source entity and relation name
- relation cardinality and strength
- local source fields in relation order
- target entity primary-key fields in primary-key order
- target key value, when available
- source key value, when reverse-index state is involved

Example:

```text
relation UserPost.author targets User primary key (tenant_id, id)
local fields: (author_tenant_id, author_user_id)
target key: (tenant_id = "acme", id = 42)
```

Avoid scalar wording for composite targets, such as:

```text
target primary key field: tenant_id
```

When reporting corrupt reverse-index state, include whether the mismatch was
in target identity, source identity, or relation metadata.

## Storage And Performance

0.164 should not introduce a second relation storage format. It should reuse
the compact primary-key encoding already used for data keys and index suffixes.

Expected storage cost grows with target primary-key arity:

```text
scalar target:
    one encoded target component

two-component target:
    two encoded target components plus tuple framing/order metadata already
    required by EncodedPrimaryKey
```

Validation cost also grows linearly with component count because each local
component must be read, lowered, and encoded. That cost is inherent to full
target identity.

Add a small measurement or fixture comparison for:

- scalar strong relation target
- two-component composite strong relation target
- composite source primary key plus composite target primary key reverse index

## Implementation Plan

1. Audit all relation code paths that carry `PrimaryKeyComponent`,
   `key_kind`, or scalar relation target assumptions.
2. Add accepted relation-edge metadata that records ordered local component
   fields and ordered target primary-key components.
3. Lower existing scalar relation declarations into the new relation-edge
   shape.
4. Extend schema admission/reconciliation to validate tuple arity, order,
   component kinds, bounds, and nullability policy.
5. Keep generated model relation metadata as proposal input only; runtime
   builders must require accepted schema/catalog authority.
6. Add a helper that extracts `Option<PrimaryKeyValue>` or
   `Vec<PrimaryKeyValue>` from source row slots according to accepted relation
   metadata.
7. Update save-time strong relation validation to encode target raw data keys
   from `PrimaryKeyValue`.
8. Update reverse-index construction and delete validation to use full target
   `EncodedPrimaryKey` identity.
9. Rename, isolate, or delete scalar-only helper APIs so composite callers
   cannot accidentally consume `target_pk.fields[0]`.
10. Update diagnostics and schema/table describe output to render relation
    field lists and target primary-key lists.
11. Add focused tests before broadening frontend syntax or generated helper
    ergonomics.
12. Update the changelog once behavior is implemented.

## Validation Plan

Schema admission tests:

- scalar relation target still works
- composite relation target with matching local tuple is accepted
- composite relation target with one local field is rejected
- wrong arity is rejected
- wrong component order is rejected
- mismatched component kind or bounds are rejected
- missing local component field is rejected
- generated-model fallback metadata cannot populate runtime relation-edge
  authority
- relation target containing `int_big` / `nat_big` remains rejected

Runtime validation tests:

- strong relation validates existing composite target row
- strong relation rejects missing composite target row
- optional composite relation with all components `NULL` is accepted as no
  target
- optional composite relation with mixed null components rejects
- required composite relation with any null component rejects
- scalar target relation behavior does not regress
- target `i128` / `u128` components work where admitted by primary-key taxonomy

Reverse-index tests:

- reverse index preserves full composite target identity
- two target rows sharing the second component but differing in the first
  component do not collide
- source scalar primary key plus target composite primary key works
- source composite primary key plus target composite primary key works
- delete blocking finds source rows through full target tuple identity
- stale/corrupt reverse entries fail closed with target/source identity
  diagnostics

Guard tests:

- no relation admission path can use only the first target primary-key field
- scalar-only target helper APIs are unavailable to composite relation code
- collection relations to composite targets reject unless tuple elements are
  explicitly represented

## Deferred Work

- Relation fields targeting composite-key entities as primary-key components.
- Generated ergonomic key structs or accessors for composite relation targets.
- Relations to non-primary unique keys.
- SQL foreign-key DDL syntax for composite relations.
- Partial primary-key prefix planning for relation reverse-index scans.
- Collection relation syntax for composite target tuples if not included in the
  first slice.

## Open Questions

- Should the accepted relation-edge descriptor live beside field metadata, or
  should it become a separate entity-level relation catalog?
- Should collection relations to composite targets be in 0.164, or should they
  remain scalar-only until tuple element storage is explicit?
- Should diagnostics print decoded component values for all supported primary
  key types, or abbreviate large blob/text components?
- Do we need a dedicated `RelationTargetKey` wrapper around `PrimaryKeyValue`
  to prevent scalar helper leakage?

## Recommended Changelog Wording

- Added support for strong relations targeting composite-primary-key entities
  by validating ordered local relation field tuples against accepted target
  primary-key metadata and routing target lookup through `PrimaryKeyValue` /
  `EncodedPrimaryKey`.
- Reverse relation indexes now preserve full target tuple identity for
  composite relation targets. Partial nullable composite relation tuples fail
  closed instead of projecting one primary-key component.
