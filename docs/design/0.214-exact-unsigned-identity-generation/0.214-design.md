# 0.214 Exact Unsigned Identity Generation

Status: proposed

Design cut: 2026-07-17

Implementation order: after 0.213 closeout; Toko adoption follows IcyDB closeout

## Current Project Situation

At the 2026-07-17 design cut, 0.204 SQL correctness evidence was the active
implementation line and was only partway through its planned work. Identity
generation was explored during that line after a discussion about ULID
alternatives and Toko entity IDs. The exploration crossed from a small scalar
helper into accepted schema, mutation admission, commit-marker, journal,
recovery, integrity, SQL, and generated-schema ownership before a design had
been accepted.

That was the wrong delivery boundary even though the experiment produced
useful evidence. Continuing it in the active worktree would mix two persisted-
format projects, make 0.204 harder to review and validate, and allow an
experimental Nat64 shape to become accidental authority for 0.214.

At the point it was stopped, the implementation spike comprised 38 code and
test files at approximately 1,550 inserted and 50 removed lines. The working
estimate was:

- roughly 70 percent of the original Nat64-only experiment;
- roughly 50 percent of the generalized contract selected by this design;
- accepted-schema, typed-create, SQL-insert, marker, journal, and recovery
  paths present in experimental form;
- focused schema, journal, transition, package compilation, and one end-to-end
  identity atomicity/recovery test passing;
- the complete recovery interruption matrix, generalized width matrix,
  restore/integrity closeout, performance evidence, and raw wasm measurement
  still incomplete; and
- no Toko audit or Toko source change started.

The experimental implementation remains deliberately ringfenced:

- every 0.214 implementation and test change is stored only in the local Git
  stash named <code>wip/0.212 identity implementation spike (design
  active)</code>; the stash name is historical and is not renamed with the
  design line;
- this design document is the only active 0.214 worktree artifact;
- the then-active 0.204 source, evidence, and changelog work was not placed in
  that stash; and
- the spike's Sequence naming and persisted shapes have no compatibility or
  release standing.

The stash is a local implementation aid, not durable project authority. It is
not transferred by push or clone and may eventually be unavailable. This
document must therefore be sufficient to implement 0.214 from a clean tree.

Implementation resumes only when all of the following are true:

1. the active delivery line has reached 0.214 after 0.213 closeout in the
   agreed design order;
2. this proposed contract has been reviewed and accepted;
3. the then-current accepted-schema, commit, journal, recovery, and integrity
   owners have been re-audited for drift;
4. the code-only stash is applied as reference rather than presumed-correct
   implementation; and
5. every experimental Sequence/u64/absent-as-zero/per-row-advance shape is
   hard-cut to the maintained Identity/u128/explicit-state/range-owned design.

No 0.214 implementation should be opportunistically pulled into an earlier
design line. Existing tests or fixtures that need a numeric key must use the
currently maintained caller-authored Nat semantics rather than depending on the
stashed generator.

## Decision Summary

IcyDB will support database-owned monotonically increasing identities on its
existing exact unsigned integer field kinds:

- Nat8 / u8;
- Nat16 / u16;
- Nat32 / u32;
- Nat64 / u64; and
- Nat128 / u128.

This is an insert-generation policy, not a new scalar type. IcyDB will not add
Serial8, Serial16, Serial32, Serial64, or Serial128 field kinds. A field remains
an exact Nat field in accepted schema and remains the corresponding Rust
integer in generated APIs. The accepted write policy separately states that
the database owns identity generation.

The proposed generated-schema spelling is:

~~~rust
field(
    ident = "id",
    value(item(prim = "Nat64")),
    generated(insert = "Identity::next")
)
~~~

The accepted representation remains an enum:

~~~rust
enum FieldInsertGeneration {
    Ulid,
    Timestamp,
    Identity,
}
~~~

There will not be an auto_increment boolean. IcyDB already has multiple
mutually exclusive insert-generation semantics, so an enum is the exact model
and keeps schema, codec, diagnostics, and execution exhaustive.

Version 1 identity generation is deliberately narrow:

- the field is the sole primary-key field;
- the field is one non-null scalar Nat8, Nat16, Nat32, Nat64, or Nat128;
- generation starts at one and advances by one;
- the generator is GENERATED ALWAYS: ordinary callers cannot author or
  override the value;
- committed values are never reused after deletion;
- a rejected operation does not consume an identity;
- one atomic statement allocates deterministic consecutive values per entity;
- overflow fails before the commit marker with a typed exhaustion error; and
- identity state advances atomically with the rows that receive the values.

Nat64 is the recommended default. Nat8 and Nat16 are appropriate only for
deliberately bounded catalogs whose lifetime allocation limit is understood.
Nat32 is a compact domain choice, Nat64 is the normal local database identity,
and Nat128 is available when the domain contract genuinely requires it.

Narrower accepted kinds do not currently imply narrower physical storage.
Nat8, Nat16, Nat32, and Nat64 all use the Nat64 runtime value and scalar codec,
with accepted-schema range checks providing their exact semantics. Any future
compact integer codec is a separate persisted-format design applying to Nat
fields generally, not a side effect of identity generation.

### Non-Negotiable Invariants

1. Accepted schema is the sole runtime authority for whether a field is an
   identity.
2. Identity is an insert-generation enum variant, never inferred from a field
   name, Rust integer type, primary-key position, or missing value.
3. Identity generation does not introduce a Serial scalar kind or wrapper
   codec.
4. The accepted field kind remains exact and determines the maximum admitted
   identity.
5. Version 1 permits identity only on the required sole primary key.
6. The first generated identity is one, the step is one, and cycling is never
   allowed.
7. Ordinary create, insert, save, update, replace, and structural write lanes
   cannot author an identity value.
8. Identity generation occurs only for a new logical row. Updates,
   replacements of an existing row, deletes, index rebuilds, integrity scans,
   recovery, and reads never allocate.
9. Rejected preflight, resource admission, validation, relation, index, or
   commit preparation does not consume an identity.
10. A durable marker makes row publication and identity-state publication one
    atomic recovery fact.
11. A committed identity is never made available again by row deletion,
    rollback of an uncommitted attempt, schema reconciliation, or index
    rebuild.
12. One statement reads identity state once per participating entity and
    commits at most one contiguous range advance per entity.
13. Allocation order is deterministic and follows the admitted row-intent
    order.
14. Durable high-water state uses one canonical u128 representation for every
    supported unsigned field width.
15. Converting the canonical high-water value to the accepted field kind is
    checked before row construction.
16. A published identity declaration has an explicit zero-state control
    record. Missing state after publication is corruption, not an implicit
    zero.
17. Recovery replays admitted row bytes and sequence advances; it never calls
    the generator again or derives state from MAX(id).
18. Imports and restores cannot infer the high-water mark from live rows,
    because deleted committed identities are still unavailable.
19. Accepted policy drift to or from identity is rejected unless an explicit
    catalog-native migration is designed and executed.
20. Persisted state and journal decoding are bounded, fallible, checksummed
    where required, and current-form only.
21. No compatibility alias for the experimental Sequence::next spelling or
    Sequence enum variant is retained before 1.0.0.
22. SQL and typed mutation lanes lower to the same accepted identity owner.

## Normative Contract

The maintained contract is:

> An unsigned identity field is an exact accepted Nat field whose accepted
> insert-generation policy assigns the next store-local, entity-local value
> when and only when a new logical row is admitted. The generator's operational
> high-water state commits atomically with those rows and is never reconstructed
> from generated Rust models or current row contents.

The authorship rule is:

> Identity means GENERATED ALWAYS. Omission or explicit SQL DEFAULT requests
> the accepted generator. Supplying a numeric value through an ordinary write
> lane is an authorship violation even when that value equals the next
> available identity.

The durability rule is:

> A rejected operation consumes no identity. Once an operation's durable
> commit marker is published, recovery must complete both its rows and its
> identity range. A later deletion does not rewind the range.

The negative contract is:

> IcyDB does not promise dense visible IDs, global uniqueness across entities
> or databases, secrecy, smaller physical encoding for narrower Nat kinds,
> caller-selected sequence values, or compatibility with SQLite rowid
> behavior.

## Problem Statement

### 1. Serial Types Conflate Value Kind With Generation Policy

A u64 field and a database-generated u64 identity have the same value domain,
comparison, index, relation, and wire semantics. Their difference is
authorship at insertion and the operational state needed to choose a value.

Adding Serial64 as a scalar kind would duplicate the Nat64 codec and leak a
write policy into every type consumer. It would also require five Serial kinds
for width flexibility.

### 2. A Boolean Cannot Represent Existing Generator Semantics

IcyDB already supports distinct insert generators for ULID and Timestamp. An
auto_increment boolean would either sit beside that enum with invalid
combinations or become a second source of generator authority.

One exhaustive FieldInsertGeneration enum is simpler and makes unsupported
combinations unrepresentable.

### 3. Auto-Increment Is Durable State, Not A Number Helper

Choosing current + 1 is the smallest part of the feature. The high-water mark
must agree with row commits through interruption, journal publication, replay,
fold, batch rejection, restore, and schema transition.

Updating the row and sequence independently can either expose a row without
advancing state or advance state without the row. Deriving from live rows can
reuse deleted IDs. Both violate the contract.

### 4. Width Generality Needs One Canonical State Model

Separate u8, u16, u32, u64, and u128 state codecs would multiply persisted
formats and branch recovery by schema width. A canonical u128 high-water value
can represent every admitted domain, while accepted field kind supplies the
exact upper bound and runtime value conversion.

### 5. Narrower Domains Are Not Automatically Smaller Encodings

Today Nat8, Nat16, Nat32, and Nat64 share a fixed-width Nat64 scalar encoding.
Allowing identity generation for those exact kinds is useful for domain
contracts, API types, overflow testing, and deliberate capacity limits. It is
not evidence of a persisted-byte saving.

### 6. Insert-Shaped Work Is Not Always A Logical Insert

Recovery and integrity code may project an existing row into empty derived
index state using a before-absent/after-present shape. That shape must not
trigger identity allocation or demand a sequence advance.

The commit path needs explicit logical mutation intent. Identity correctness
must never depend only on optional before/after bytes.

## Goals

1. Support exact generated identities for all bounded unsigned primitive
   integer kinds already owned by IcyDB.
2. Preserve one scalar-kind system and one insert-generation enum.
3. Make Nat64 the documented default without artificially forbidding other
   exact domains.
4. Bind identity allocation to accepted schema, final after-image validation,
   and the durable commit marker.
5. Keep rejected writes free of sequence consumption.
6. Aggregate batch allocation into one range advance per entity.
7. Make recovery, integrity checking, import, and restore semantics explicit.
8. Provide stable typed diagnostics for authorship, exhaustion, state
   corruption, and migration incompatibility.
9. Keep the implementation bounded in rows, bytes, instructions, and stable
   control-state growth.
10. Define a clean boundary for a later Toko adoption audit.

## Non-Goals

0.214 does not add:

- new Serial scalar types;
- signed or arbitrary-precision identities;
- identities on non-primary-key fields;
- identities on composite primary keys;
- multiple identity fields per entity;
- caller-selected start values or increments;
- cycling, caching, reseeding, setval, or session-local sequence functions;
- GENERATED BY DEFAULT or an ordinary override escape hatch;
- a last_insert_rowid-style session global;
- global uniqueness across stores, canisters, replicas, or applications;
- gapless visible row numbering;
- implicit conversion of existing Nat or ULID primary keys;
- MAX(id) reconstruction;
- compact Nat codecs;
- an automatic Toko schema migration;
- general SQL CREATE TABLE support; or
- a compatibility alias for the experimental Sequence name.

## Vocabulary

### Identity Field

The sole non-null unsigned primary-key field whose accepted insert-generation
policy is Identity.

### Identity Domain

The exact accepted Nat8, Nat16, Nat32, Nat64, or Nat128 range of an identity
field.

### High-Water Mark

The greatest identity whose allocation is durably committed, including values
whose rows were later deleted.

### Identity Range

One contiguous statement allocation represented as exclusive previous
high-water and inclusive new high-water values.

### Logical Insert

A mutation that creates a row for a previously absent primary key under
ordinary accepted write semantics.

### Derived Projection

Index, relation, diagnostic, or recovery work computed from an already
authoritative row. It may look insert-shaped to a derived store but is not a
logical row insertion.

### GENERATED ALWAYS

The identity field is database-authored on logical insert and cannot be
assigned through ordinary caller write intent.

## Exact Domain Guidance

| Rust / accepted kind | Maximum committed allocations | Guidance |
|---|---:|---|
| u8 / Nat8 | 255 | Tiny immutable or deliberately bounded catalogs; exhaustion tests |
| u16 / Nat16 | 65,535 | Small bounded catalogs with a proven lifetime ceiling |
| u32 / Nat32 | 4,294,967,295 | Explicit 32-bit domain or interoperability requirement |
| u64 / Nat64 | 18,446,744,073,709,551,615 | Recommended general-purpose IcyDB local identity |
| u128 / Nat128 | 2^128 - 1 | Exceptional domains needing a 128-bit sequential key |

The capacity is lifetime committed allocation, not current row count. Deleting
rows does not restore capacity.

Nat8 and Nat16 are supported because exact field kinds should compose
consistently, not because they are safe defaults. Documentation and
introspection must expose the exact domain and exhaustion state. IcyDB should
not emit a noisy runtime warning on every insert.

## Public Schema Surface

The generated DSL uses one spelling for every width:

~~~rust
field(
    ident = "id",
    value(item(prim = "Nat32")),
    generated(insert = "Identity::next")
)
~~~

Identity::next is a well-known declarative generator token. Runtime database
authority is the accepted FieldInsertGeneration::Identity enum value, not a
function pointer or an arbitrary Rust callback.

Schema validation requires:

1. exactly one primary-key field;
2. the identity field is that field;
3. the field is non-null and single-valued;
4. the primitive is Nat8, Nat16, Nat32, Nat64, or Nat128;
5. no database default;
6. no other insert generator;
7. no engine-managed write policy; and
8. no caller-authored create field for the identity.

An ordinary unsigned primary key without the generator remains caller-authored:

~~~rust
field(ident = "external_id", value(item(prim = "Nat64")))
~~~

That is the correct model for imported protocol numbers, externally allocated
IDs, or values whose exact numeric identity belongs to the application.

## Accepted Authority Model

Accepted field write policy owns:

~~~rust
enum FieldInsertGeneration {
    Ulid,
    Timestamp,
    Identity,
}
~~~

The Identity variant carries no width payload. Duplicating the field kind
inside the generator would permit drift such as Identity(Nat32) on a Nat64
field. The accepted field kind is the sole width authority.

Accepted schema integrity verifies the full eligibility rule whenever a
snapshot is decoded, published, selected, or checked. Generated EntityModel
may propose the policy during reconciliation, but runtime allocation never
falls back to generated metadata.

The accepted schema fingerprint includes the Identity variant. Operational
high-water state does not enter that fingerprint because it changes on every
committed insertion.

## Operational State

Each identity owns one stable control record keyed by:

~~~text
(store identity, entity tag, durable field id)
~~~

The store is implicit in the owning SchemaStore. EntityTag and durable FieldId
prevent an unrelated accepted field from accidentally inheriting a record.

Conceptually:

~~~rust
struct IdentitySequenceState {
    high_water: u128,
}
~~~

The persisted record owns:

- a dedicated key namespace;
- fixed magic;
- one current format version;
- a fixed-width big-endian u128 high-water value; and
- a checksum over the current-form payload.

The state is operational control data, not accepted catalog data. It can live
in SchemaStore's control namespace without becoming part of accepted schema
fingerprints or snapshot replacement.

### Publication And Absence

Publishing a new identity declaration atomically creates a zero high-water
record. Once accepted schema declares Identity, a missing record is corruption.
The runtime must not use unwrap_or(0) for an already published declaration.

This distinction matters after all rows have been deleted: live rows cannot
prove whether identities were previously committed. The control record is the
only no-reuse authority.

Identity control records survive ordinary accepted-snapshot replacement.
Version 1 does not garbage-collect a retired record. An explicit future
catalog migration may define safe retirement, but silent deletion is
forbidden.

### Canonical Width

All states persist u128. Nat8 through Nat64 still emit Value::Nat64 after a
checked conversion; Nat128 emits Value::Nat128. One helper owns:

~~~text
u128 high-water
    -> checked next
    -> accepted-kind bounds check
    -> exact runtime Value
    -> primary-key and row-slot equality proof
~~~

The extra state bytes over a u64-only record are per identity field, not per
row, and avoid a second persisted state format when Nat128 is used.

## Allocation Semantics

### Start, Step, And Exhaustion

The initial high-water mark is zero. The first value is one. Each generated
value is the previous value plus one.

Before constructing a row, allocation checks:

1. u128 addition does not overflow;
2. the result is at most the accepted field kind's maximum;
3. conversion to Value::Nat64 or Value::Nat128 is exact; and
4. the same value is present in the row primary key and accepted field slot.

Failure returns IdentityExhausted before the marker. The high-water state and
all rows remain unchanged.

### Statement-Local Cursor

One statement-local cursor is created lazily per participating identity
entity. It reads the live recovered high-water mark once and allocates in
admitted intent order.

The cursor mutates only statement-local memory. Durable state changes only
through the successful commit marker and its journal publication.

### Deterministic Order

Within one entity:

- SQL multi-row INSERT follows VALUES order;
- typed batch creation follows input order;
- catalog-native atomic batches follow their stable admitted intent order; and
- retries against unchanged durable state produce the same candidate values.

No ordering is promised between independent entities because each has its own
sequence.

### Range Aggregation

Commit preparation aggregates all generated rows for one entity into:

~~~rust
struct IdentityRangeAdvance {
    entity_tag: EntityTag,
    field_id: FieldId,
    expected_high_water: u128,
    new_high_water: u128,
    allocation_count: u32,
}
~~~

The exact owner type may differ, but its invariants do not:

- new_high_water - expected_high_water equals allocation_count;
- allocation_count is nonzero and within the admitted row cap;
- every allocated row has exactly one value in that range;
- values are contiguous and appear in deterministic intent order; and
- at most one range exists per entity in one commit window.

This is both smaller and easier to validate than one journal record per row.

### Gap Semantics

Rejected operations do not consume values, but IcyDB does not advertise
gapless visible IDs:

- deleted rows leave gaps;
- a restore may contain a high-water mark greater than every live row;
- future explicit migrations may reserve a range; and
- current row count says nothing about remaining capacity.

Applications must not use identity arithmetic as a row-count or completeness
proof.

## Mutation Ingress

Every mutation ingress lowers to an explicit logical row intent before
identity handling.

### Typed Create

The generated create input omits the identity field. Accepted insertion policy
allocates the value, constructs the complete after-image, then runs normal
exact-kind, sanitizer, validator, constraint, relation, index, and resource
preflight.

The created entity returned to the caller contains the concrete integer ID.

### SQL INSERT

Both omission and explicit DEFAULT request identity generation:

~~~sql
INSERT INTO Event (payload) VALUES ('created') RETURNING id;
INSERT INTO Event (id, payload) VALUES (DEFAULT, 'created') RETURNING id;
~~~

An explicit numeric identity rejects:

~~~sql
INSERT INTO Event (id, payload) VALUES (42, 'created');
~~~

RETURNING is the supported way to obtain generated IDs. IcyDB does not add
last_insert_rowid(), a mutable connection register, or a session-global current
sequence value.

### Save, Replace, And Update

An existing entity carries its identity as row identity. Saving or replacing
that existing row preserves the value and allocates nothing.

A caller-authored full entity whose key is absent from storage cannot use save
as a backdoor insertion with a chosen identity. It rejects and directs the
caller to the create surface.

UPDATE cannot assign, clear, default, or regenerate the identity field.

### Structural Writes

Ordinary structural insert and replace lanes obey the same authorship rules as
typed and SQL writes. A raw or structurally valid numeric value is not enough
to bypass accepted ownership.

Catalog migrations, imports, and restores use separate explicit authorities
described below. There is no boolean bypass on ordinary structural mutation.

### Atomic Batches

The complete batch is preflighted before marker publication. If the final row
fails, no row and no identity range commits.

Successful batches commit all row effects and one range advance per identity
entity in the same marker. Returned rows preserve the input order and expose
their allocated IDs.

### Derived Work

Commit preparation must carry an explicit intent such as:

~~~rust
enum RowPreparationIntent {
    LogicalMutation(RowMutationKind),
    DerivedProjection,
}
~~~

The exact type can integrate with existing commit owners, but a private
derived_index_projection boolean is not the desired final abstraction.
Recovery index rebuild, integrity projection, and relation reconstruction use
DerivedProjection and cannot allocate or require identity advances.

## Commit, Journal, And Recovery

### Commit Marker

The marker owns:

- admitted row before/after bytes;
- derived index and relation effects;
- accepted schema fingerprint binding; and
- the aggregated identity range advances.

Preparation proves all range and row associations before the marker is
written. Applying a marker with an identity mismatch is an internal invariant
or corruption failure, not a recoverable caller validation error.

### Journal Record

Journaled stores encode one current-form identity range record per entity per
commit window. The record carries stable entity and field identity plus
expected and new u128 high-water values.

Heap stores use the same semantic owner without claiming durability beyond the
heap store's declared lifecycle.

### Apply Order

The marker is the atomic authority, so interruption may occur around physical
row, index, relation, and sequence writes without changing the logical
outcome. Recovery either completes the exact committed marker or observes no
committed operation.

No public contract depends on whether the live sequence projection is
physically updated immediately before or after the row projection.

### Replay And Fold

Replay requires:

- current live high-water equals expected_high_water before first apply; or
- the exact new_high_water is already present during idempotent replay of the
  same durable batch.

Any skip, rewind, field mismatch, divergent range, or malformed width is store
corruption.

Fold applies the same expected/new proof to canonical state and may accept the
exact already-folded value when interruption occurred after the canonical
write but before fold-watermark publication.

Recovery never:

- calls Identity::next;
- scans for MAX(id);
- treats missing state as zero;
- allocates for index rebuild; or
- changes an admitted row's identity.

## Schema Transitions

### New Entity

A newly accepted entity may declare an identity. Catalog publication creates
its zero-state control record atomically with the declaration.

### Existing Entity

Version 1 rejects transitions that:

- add Identity to an existing field;
- remove Identity from an existing field;
- move Identity to a different field;
- change the identity field's exact Nat width;
- change the primary key between scalar and composite; or
- replace generated ownership with DDL ownership or the reverse.

These are migration operations, not metadata edits.

### Why MAX(id) Is Not A Migration

Seeding from the greatest live ID can reuse a larger ID whose row was deleted.
Even if no deletion is known, proving that fact requires durable history not
present in the live row set.

A future explicit adoption migration must require a caller-supplied authority
or a source control-state proof, validate every existing key, and publish the
chosen high-water mark with the accepted policy. 0.214 does not invent that
surface.

### Drop And Recreate

Dropping rows or an entity does not silently recycle its identity state.
Reusing the same accepted entity and field identities therefore cannot restart
at one. A genuinely new identity domain requires new durable schema identity
or full store recreation through an explicit destructive operation.

## Imports, Restores, And Backfills

### Ordinary Import

An ordinary import is a write ingress and cannot author identity fields. It
either requests generation for new logical rows or uses a non-generated Nat
primary key designed for externally allocated values.

### Exact Restore

An exact current-form restore includes:

- accepted schema identity;
- every canonical row;
- the exact identity control record, including zero; and
- journal/marker state required by the restore contract.

Restore validates:

- the state record belongs to the accepted entity and field;
- high-water is within the exact Nat domain;
- every live identity is between one and high-water;
- row key and identity slot agree; and
- no current-form control record is missing or duplicated.

High-water may be greater than the maximum live row because deleted IDs remain
consumed. Restore must preserve that fact rather than normalize it.

### Backfills

Schema backfills do not regenerate existing identities. A backfill that creates
new logical rows uses the ordinary bounded insertion policy and commits its
range with each atomic page. Resumable partial row creation must have an
explicit design and idempotency key; it cannot reserve unbounded ranges in
advance.

## Introspection And Diagnostics

DESCRIBE and schema tooling should expose:

| Fact | Example |
|---|---|
| insertion policy | identity |
| field kind | Nat64 |
| ownership | generated or DDL |
| start | 1 |
| increment | 1 |
| cycle | false |
| override policy | always |
| state status | ready, exhausted, or corrupt |

Operational diagnostics may expose high_water and remaining_capacity where
authorization and response bounds permit. They must not suggest that
high_water equals live row count.

Stable typed failures include at least:

| Condition | Error class |
|---|---|
| Unsupported field kind or key shape | Schema validation |
| Caller authors an identity | Write authorship / validation |
| Exact domain is exhausted | Resource or identity exhaustion |
| Missing or malformed state | Integrity / corruption |
| Marker range does not match rows | Internal invariant or corruption |
| Schema transition changes identity policy | Migration validation |
| Restore state is absent or incompatible | Restore validation |

Machine-readable facts should include entity, durable field ID, field name,
accepted kind, and relevant expected/actual high-water values. Code and tests
must not parse diagnostic text.

## Integrity Checking

### Quick Check

The bounded quick check verifies:

- every accepted identity declaration has exactly one current-form state
  record;
- state magic, version, length, and checksum;
- entity and durable field identity;
- u128 high-water within the accepted field domain;
- identity eligibility remains structurally valid; and
- no active marker or journal shape contradicts the declared state owner.

### Deep Resumable Check

For each identity row, the deep scan verifies:

- storage key matches the row primary-key field;
- the value decodes under the exact accepted Nat kind;
- value is at least one;
- value is no greater than committed high-water;
- no duplicate primary key exists; and
- all ordinary row, index, relation, and constraint checks still pass.

At clean completion, the scanner proves maximum live identity is no greater
than high-water. It does not require equality.

The scan carries accepted schema identity, captured store revision, bounded
findings, and canonical primary-key continuation as defined by 0.212. A
concurrent revision change invalidates the proof rather than silently blessing
a mixed snapshot.

The checker never repairs, rewinds, or derives sequence state.

## SQL DDL

The catalog-native concept maps to standard identity terminology:

~~~sql
id BIGINT GENERATED ALWAYS AS IDENTITY
~~~

IcyDB need not add this parser form in the first implementation slice if its
current DDL surface cannot create a sole primary key safely. When admitted,
SQL is only a frontend to the same accepted Identity policy and operational
state publication.

Unsupported forms fail explicitly:

- GENERATED BY DEFAULT;
- custom START WITH;
- custom INCREMENT BY;
- MINVALUE or MAXVALUE overrides;
- CYCLE;
- CACHE;
- ALTER IDENTITY;
- RESTART or RESEED; and
- identity on an added non-primary-key column.

Generated-owned identity policy is introspectable through SQL but cannot be
silently removed or redefined by DDL.

## ULID Versus Unsigned Identity

Use an unsigned identity when:

- one IcyDB entity is the sole allocator;
- values are created through the database;
- small ordered numeric keys are useful;
- IDs need not be created offline or merged across independent databases; and
- enumeration is not a security concern.

Use ULID when:

- callers or multiple systems must allocate before reaching IcyDB;
- data may be merged across stores or canisters;
- a 128-bit globally unique value is part of an external contract;
- time-sortable decentralized allocation is valuable; or
- exposing a simple row count approximation through sequential public IDs is
  undesirable.

ULIDs are not authorization secrets. Numeric identities are not promised to
be physically smaller until the actual key and row codecs are measured.

## Toko Adoption Boundary

Toko changes are a second project step after the 0.214 IcyDB implementation is
accepted, formatted, focused-tested, and measured.

The Toko audit will classify each entity:

1. keep ULID for externally visible, cross-system, preallocated, or mergeable
   identity;
2. consider Nat64 Identity for purely local database-owned rows;
3. use Nat32 or narrower only with a documented lifetime allocation ceiling;
4. retain caller-authored Nat fields for external protocol numbers; and
5. avoid converting an existing primary key merely to save bytes without an
   end-to-end storage measurement.

The audit must include public Candid/API shape, relations, fixtures, imports,
stable data, migration cost, and client assumptions. It does not automatically
rewrite existing ULID tables.

## Alternatives Considered

### Add Serial8 Through Serial128 Types

Rejected. These types duplicate Nat value semantics and couple authorship to
scalar kind. Generation belongs in accepted write policy.

### Add An auto_increment Boolean

Rejected. IcyDB already has ULID and Timestamp insert generation. A boolean
creates invalid combinations and provides no useful exhaustive match point.

### Support Only Nat64

Rejected as the final contract. Nat64 remains the recommendation, but the
engine already owns exact bounded unsigned kinds and one u128 operational state
can serve them consistently. A Nat64-only persisted state would create needless
format churn when Nat128 or a narrower exact domain is later requested.

### Support Nat32 And Nat64 Only

Rejected. It saves little implementation complexity once conversion and bounds
are centralized, while making otherwise composable exact kinds inconsistent.
The danger of small domains is handled by explicit opt-in and clear capacity
documentation.

### Include Signed Integers

Rejected. Negative identity space, start rules, and overflow semantics add no
current domain value. Applications needing signed external numbers should use
ordinary caller-authored Int fields.

### Include NatBig

Rejected. Identity state and exhaustion must remain fixed-width and bounded.
Nat128 already supplies a practically unbounded sequential domain.

### Generate MAX(id) Plus One

Rejected. It requires a scan or index lookup, loses deleted allocation history,
couples correctness to derived index state, and cannot provide the same
recovery proof as explicit control state.

### Consume IDs As Soon As They Are Requested

Rejected for IcyDB's initial semantics. It creates gaps for rejected writes and
requires a separate durable allocation transaction. Marker-bound allocation
keeps the guarantee stronger without concurrent connection contention.

### Permit Explicit Values With GENERATED BY DEFAULT

Rejected. It complicates high-water synchronization, imports, conflict
semantics, and caller expectations. Applications needing authored numbers can
use an ordinary Nat primary key.

### Expose last_insert_rowid

Rejected. It introduces mutable session state and ambiguity for batches. The
existing INSERT RETURNING surface returns exact generated values directly.

### Compress Nat8, Nat16, And Nat32 During This Work

Rejected as scope coupling. Compact codecs would change every persisted Nat
field and key, require independent ordering and format evidence, and should be
justified by measurements.

## Implementation Slices

### Slice 1: Freeze The Accepted Surface

- hard-cut Sequence naming to Identity;
- extend derive validation across all five exact unsigned kinds;
- keep one FieldInsertGeneration enum;
- add schema codec, fingerprint, integrity, and transition tests; and
- reject every ineligible shape with typed or stable compile diagnostics.

### Slice 2: Current-Form Operational State

- key state by accepted entity tag and durable field ID;
- persist fixed-width u128 high-water records;
- create explicit zero state with schema publication;
- make missing active state corruption;
- retain state across snapshot replacement; and
- add bounded codec, checksum, corruption, and width-bound tests.

### Slice 3: Accepted-First Allocation

- centralize checked u128-to-accepted-value conversion;
- create one statement cursor per identity entity;
- integrate typed create, SQL INSERT, DEFAULT, RETURNING, and structural intent;
- reject authored full-entity insertion and identity UPDATE; and
- prove final key/slot equality before commit preparation completes.

### Slice 4: Range-Owned Atomic Commit

- aggregate per-row candidates into one range per entity;
- bind ranges to marker authority;
- encode current-form u128 journal range records;
- apply and fold idempotently;
- preserve rejected-write no-consumption semantics; and
- replace insert-shape inference with explicit logical mutation intent.

### Slice 5: Recovery, Restore, And Integrity

- complete interruption coverage around row and range publication;
- ensure derived index/relation rebuild never allocates;
- extend quick and deep integrity checks;
- require exact state in restore;
- classify divergent range state as corruption; and
- remove all temporary prototype instrumentation.

### Slice 6: Introspection, Docs, And Measurement

- expose identity policy and domain through schema description;
- document SQL and typed usage;
- update persisted-format inventory and contracts;
- record focused performance and raw wasm-size deltas;
- close the root Unreleased changelog for the actual code slice; and
- remove stale Nat64-only or Sequence examples.

### Slice 7: Toko Audit

- inventory current Toko primary keys and allocation boundaries;
- classify candidates using the ULID-versus-identity rules;
- estimate API, relation, migration, and stored-byte impact;
- propose a separate Toko change plan; and
- make no automatic schema conversion.

Each slice is a review and landing unit. No slice may introduce a generated
model fallback, an implicit identity, a second mutation owner, or a
compatibility alias.

## Focused Validation

### Schema Matrix

- Identity accepts Nat8, Nat16, Nat32, Nat64, and Nat128 sole primary keys;
- nullable, collection, relation, signed, NatBig, non-primary, and composite
  forms reject;
- default, management, and second-generator combinations reject;
- codec and fingerprint round trips preserve Identity exactly; and
- policy or width drift rejects during reconciliation.

### Allocation Matrix

- first identity is one for every width;
- consecutive insertions advance exactly once;
- Nat8 and Nat16 reach their exact maximum and then reject;
- Nat32, Nat64, and Nat128 exercise boundary conversion cases;
- deletion never permits reuse;
- rejected validation, uniqueness, relation, resource, and final-row failures
  consume nothing; and
- multiple entities in one batch advance independently.

### Ingress Parity

- typed create;
- SQL INSERT omission;
- SQL INSERT DEFAULT;
- SQL multi-row INSERT RETURNING;
- atomic typed batch;
- structural insert;
- save of an existing row;
- attempted save-as-insert;
- UPDATE and replace;
- import and exact restore; and
- direct test helpers that construct commit operations.

Every ingress must reach the same accepted owner or reject before marker
publication.

### Atomicity And Recovery

Interrupt around:

- local allocation before marker;
- marker persistence;
- row journal publication;
- identity range journal publication;
- live row apply;
- live identity apply;
- canonical row fold;
- canonical identity fold;
- fold watermark update;
- marker clearing; and
- derived index rebuild.

After recovery, the outcome is either the complete committed row/range or the
unchanged pre-marker state. No duplicate, rewind, reuse, skipped row, or
generator reevaluation is admitted.

### Integrity And Restore

- absent active state;
- bad magic, version, length, checksum, and u128 payload;
- state beyond accepted kind maximum;
- row zero or row above high-water;
- row key/slot mismatch;
- high-water greater than live maximum after deletion;
- exact restore with consumed deleted IDs;
- restore missing a zero-state record; and
- concurrent deep-scan invalidation.

### Performance Evidence

Measure at least:

- generated versus caller-authored Nat64 insert;
- generated Nat128 insert;
- one-row and maximum admitted multi-row batches;
- journal bytes per entity range;
- recovery with identity-bearing commits;
- stable control bytes per identity field;
- raw non-gzipped wasm delta; and
- source line and owner-count complexity.

## Performance And Complexity Budget

### Write Runtime

Identity allocation adds one state lookup per participating entity per
statement, one checked increment per generated row, and one range-state write
per entity per successful commit. It must not scan rows or indexes.

Batch preparation may store allocated values already required for the rows,
but must not build an unbounded secondary allocation structure beyond the
existing admitted row cap.

### Persisted Bytes

Operational state is one small fixed-width control record per identity field.
Journal overhead is one range record per identity entity per commit, not one
record per inserted row.

Row and primary-key encodings remain the existing exact Nat encodings. The
design claims no Nat8/Nat16/Nat32 space saving.

### Wasm

The implementation adds no dependency. Closeout reports raw non-gzipped wasm
bytes for representative generated and SQL canisters. Gzip is secondary
context.

### Source Complexity

The desired final shape adds:

- one accepted enum variant;
- one exact eligibility helper;
- one canonical conversion helper;
- one operational state owner;
- one statement cursor;
- one marker/journal range owner; and
- integrity and introspection integration.

It should remove Nat64-specific branches, per-row advance records, absent-as-
zero fallback, and shape-based derived projection exceptions from the spike.

## Risks And Mitigations

### Risk: Small Domains Exhaust Unexpectedly

Mitigation: generation is explicit, capacities are documented and
introspectable, exhaustion is typed, and Nat64 is the recommended default.

### Risk: A Missing Record Restarts At One

Mitigation: zero state is explicitly published and missing active state is
corruption.

### Risk: Batch Failure Burns A Range

Mitigation: cursors are statement-local and range state is marker-bound only
after all preflight succeeds.

### Risk: Recovery Applies A Range Twice

Mitigation: expected/new range validation plus journal identity and fold
watermark permit only exact idempotent replay.

### Risk: Derived Index Rebuild Looks Like An Insert

Mitigation: explicit row preparation intent separates logical mutation from
derived projection.

### Risk: A Restore Reuses Deleted IDs

Mitigation: exact restore requires operational state and never derives it from
live maximum.

### Risk: Width Is Duplicated In State

Mitigation: state is canonical u128; accepted field kind is the sole domain
width.

### Risk: Narrow Types Are Mistaken For Compact Storage

Mitigation: documentation, introspection, and measurement explicitly separate
semantic width from current physical codec width.

### Risk: Public Numeric IDs Leak Volume

Mitigation: identity selection guidance treats enumerability as a domain
choice; ULID remains available and authorization never relies on ID opacity.

### Risk: The Spike Becomes Accidental Format Authority

Mitigation: status remains proposed, all experimental names and formats are
hard-cut, and no compatibility tests or aliases are retained before 1.0.0.

- Identity is a generator policy on an exact Nat field, not a Serial type;
- the eligible widths and lifetime capacities;
- Nat64 is the normal default;
- start one, step one, no cycle, GENERATED ALWAYS;

- deletion does not permit reuse;
