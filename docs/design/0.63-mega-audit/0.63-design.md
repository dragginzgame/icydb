IcyDB 0.63 Design

Status: Planned for 0.63

Theme: Post–de-monomorphization consolidation audit (hygiene, authority, and redundancy)

1. Intent

0.63 is not an architectural rewrite.

It is a systematic consolidation pass over the codebase after the wasm-size-driven restructuring in 0.58–0.62.

The goal is to:

remove accidental complexity introduced during rapid refactors
collapse duplicate logic and drifted flows
tighten authority boundaries
simplify generated surfaces
preserve or improve wasm output characteristics

This is a discipline release, not a feature release.

2. Context (Post 0.62)

After 0.62.x:

runtime is largely structuralized
E is removed from execution kernels
generator surface has been reduced (relations, metadata, helpers)
metadata moved to trait-based authority
emitted convenience APIs significantly reduced

However:

duplication exists from incremental refactors
some flows exist in parallel or drifted forms
generator/build/test surfaces still contain residual convenience emission
module boundaries reflect history, not final intent
3. Core Objectives
3.1 Code Hygiene
eliminate dead code, stale paths, and unused abstractions
remove historical scaffolding left from earlier phases
normalize naming, module structure, and ownership boundaries
3.2 Redundancy Elimination
identify and remove:
duplicate helpers
parallel implementations of the same logic
thin forwarding layers
repeated invariant enforcement
3.3 Authority Clarification
ensure each responsibility has exactly one owner
remove:
duplicated authority surfaces
shadow metadata
alternative construction paths
3.4 Generated Surface Minimization
continue reducing:
per-entity emitted methods
per-type helper duplication
prefer:
shared traits
structural runtime contracts
3.5 Measured Size Discipline
maintain or improve:
one_simple baseline
ten_simple slope
reject changes that:
increase emitted output without strong justification
4. Non-Goals
no new architecture phase
no broad reintroduction of generics into runtime
no type-erasure experiments beyond current boundaries
no cleanup that is purely aesthetic without measurable benefit
5. Audit Framework (Mandatory)

0.63 is driven by structured audits, not ad hoc cleanup.

Each audit produces:

classification
candidate actions
risk assessment
verification plan
5.1 DRY / Consolidation Audit

Classify duplication into:

protective duplication
validation vs execution
boundary vs runtime
accidental duplication
same-layer repeated logic
evolution drift
similar but diverged implementations

Actions:

consolidate accidental duplication
preserve protective duplication explicitly
collapse drift into canonical paths
5.2 Complexity Accretion Audit

Identify:

unnecessary indirection
nested abstractions with no payoff
over-splitting or under-splitting

Actions:

flatten trivial layers
remove unused abstraction points
simplify control flow
5.3 Module Structure Audit

Check:

module boundaries reflect responsibility
no mixed concerns inside modules
naming aligns with ownership

Actions:

move code to correct owners
split modules where needed
merge fragmented ownership
5.4 Layer Violation Audit

Ensure:

execution does not depend on:
schema
derive
high-level APIs
structural runtime is cleanly separated from typed boundaries

Actions:

remove cross-layer leakage
re-establish clean boundaries
5.5 Velocity Preservation Audit

Ensure:

changes do not degrade:
compile time
iteration speed
readability of common paths
6. Workstreams
6.1 Runtime Consolidation

Targets:

aggregate kernels
projection / materialization
scan / stream helpers

Focus:

remove duplicate traversal/fold logic
unify execution paths where safe
keep kernel monomorphic and minimal
6.2 Generator / Derive Cleanup

Continue removal of:

per-type convenience methods
redundant metadata surfaces
duplicate impl emission

Replace with:

shared traits
minimal metadata contracts
6.3 Test & Fixture Surface Cleanup
remove reliance on convenience APIs
align tests with real contract
eliminate duplicated fixture helpers
6.4 API Surface Tightening
remove:
unnecessary public methods
duplicate constructors
ensure:
public API reflects true contract, not convenience
6.5 Error & Validation Consolidation
unify error construction paths
remove duplicated validation logic
ensure consistent error origin classification
7. Acceptance Criteria
7.1 Hygiene
no duplicate authority surfaces
no unused abstractions
no parallel implementations of same logic
7.2 Structural Integrity
clear ownership boundaries
no layer violations
7.3 Generated Output
reduced or equal emitted surface
no reintroduction of per-entity duplication
7.4 Wasm Metrics
one_simple: stable or improved
ten_simple: flat or improved slope
7.5 Tests
all schema, relation, validation, and canister tests pass
tests reflect real contract, not removed conveniences
8. Measurement Strategy

Every meaningful change should be evaluated against:

wasm size reports:
one_simple
ten_simple
compile success across:
core
derive
build
canisters
focused contract tests:
schema
relation
validation
SQL dispatch
9. Guiding Principles
one authority per concern
no duplicate flows without explicit justification
prefer removal over abstraction
prefer shared traits over generated methods
explicit over implicit (no hidden behavior)
measure impact, don’t assume it
10. Release Framing

0.63 = consolidation and stabilization after size-driven rewrite

clean the system
remove redundancy
clarify ownership
prepare for future work on a stable base
