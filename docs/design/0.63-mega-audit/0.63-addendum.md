# 0.63 Addendum

**Status:** Follow-up notes attached to the `0.63` consolidation line
**Theme:** Late-line wasm audit findings and post-audit action queue

---

## Purpose

This addendum preserves the late `0.63` follow-through findings that surfaced
after the main consolidation seams were largely drained.

It is not a new design phase.
It does not reopen the `0.63` runtime cleanup line.

Its purpose is narrower:

- preserve the current wasm audit readout in one stable place
- record the highest-value owned wasm suspects found in the debug audit
- keep the next reduction targets explicit instead of losing them in daily
  audit bundles

---

## Context

By the end of the `0.63` line:

- the broad same-layer ownership cleanup is largely complete
- generated/test harness duplication has been trimmed materially
- predicate execution now has one canonical execution tree and one capability
  classifier
- continuation ownership is hardened around one explicit contract boundary
- the recurring audit suite is green again

The next useful work is no longer broad consolidation.
It is measured, hotspot-driven follow-through.

---

## Late Audit Findings

The late debug wasm inspection showed a stable pattern across the audited
canisters (`minimal`, `one_simple`, `one_complex`, `ten_simple`,
`ten_complex`):

- the biggest debug-only shallow offender is the wasm `"function names"`
  subsection
- the biggest non-symbol shallow offender is the `.rodata` data segment
- the biggest retained runtime root is the `canister_query query` export tree
- the next meaningful retained buckets are:
  - SQL dispatch / session SQL execution
  - projection/scalar execution
  - explain assembly
  - cursor / continuation revalidation
  - commit / recovery / storage reporting

Important caveat:

- debug symbol names are useful for attribution, but they are not a release
  optimization target by themselves
- raw non-gzipped wasm is the primary optimization metric; gzip remains useful
  as transport context, but it does not decide whether a size cut is worth
  keeping

## Action Queue

These are ordered by likely return on effort.

### 1. SQL Query Export Tree

Highest priority.

The dominant retained tree in debug is the SQL query export path:

- `export "canister_query query"`
- `ic_cdk_executor::machinery::in_tracking_query_executor_context`
- `icydb::db::sql::execute_generated_sql_dispatch`
- `DbSession::execute_lowered_sql_dispatch_query_for_authority`
- core SQL session lowering/execution entrypoints

Primary files:

- `crates/icydb/src/db/sql.rs`
- `crates/icydb-core/src/db/session/sql.rs`
- `crates/icydb-core/src/db/executor/projection/materialize/structural/mod.rs`
- `crates/icydb-core/src/db/executor/pipeline/entrypoints/scalar/mod.rs`

Goal:

- reduce retained size in the query-lane dispatch and execution path without
  introducing a new abstraction layer

### 2. `.rodata` Pressure

Second priority.

The `.rodata` data segment is the largest non-symbol shallow block in every
audited canister.

Likely owners to inspect first:

- diagnostics text
- explain descriptor labels
- static error/invariant strings
- generated actor-surface / interface metadata that is retained in query-capable
  canisters

Primary files:

- `crates/icydb-core/src/db/diagnostics/`
- `crates/icydb-core/src/db/executor/explain/`
- `crates/icydb-core/src/error/mod.rs`

Goal:

- reduce stored strings and static descriptors only where the bytes are
  actually retained and measurable

### 3. Explain Path Retention

Third priority.

Explain-related code still shows up high enough in debug retained size to be a
real follow-up candidate.

Primary files:

- `crates/icydb-core/src/db/executor/explain/descriptor.rs`
- `crates/icydb-core/src/db/sql/lowering/mod.rs`
- `crates/icydb-core/src/db/query/intent/query.rs`

Goal:

- keep explain additive and useful, but avoid letting descriptor assembly and
  lowered-plan rendering become a silent retained-size sink

### 4. Cursor / Continuation Validation

Fourth priority.

The cursor and continuation revalidation family remains a meaningful retained
subtree inside the query execution path.

Primary files:

- `crates/icydb-core/src/db/executor/continuation/engine.rs`
- `crates/icydb-core/src/db/executor/executable_plan.rs`
- `crates/icydb-core/src/db/query/plan/continuation.rs`
- `crates/icydb-core/src/db/cursor/spine.rs`
- `crates/icydb-core/src/db/cursor/anchor.rs`

Goal:

- keep the explicit contract and fail-closed semantics, but watch for repeated
  heavy validation paths inside the runtime query lane

### 5. Commit / Recovery / Storage Reporting

Fifth priority.

This family remains visible in retained size after the broader `0.63`
consolidation work.

Primary files:

- `crates/icydb-core/src/db/commit/recovery.rs`
- `crates/icydb-core/src/db/commit/store.rs`
- `crates/icydb-core/src/db/commit/prepare.rs`
- `crates/icydb-core/src/db/diagnostics/storage_report.rs`

Goal:

- reduce retained cost without blurring the storage-safety or recovery
  boundaries

## Do Not Chase First

These surfaced in the audit, but should not drive the first optimization pass:

- wasm `"function names"` subsection
  - debug-only attribution aid, not a release target
- `table[0]` / `elem[0]`
  - large retained buckets, but usually downstream of function reachability
    rather than the first root cause
- third-party library hotspots by themselves
  - `candid`
  - `num_bigint`
  - `sha2`
  - `ic_stable_structures`

These should only be targeted through owned call-path reduction, not by chasing
dependencies in isolation.

---

## Working Rule

The next work after `0.63` should follow this rule:

- use the audit to pick one owned retained subtree
- measure it
- reduce it without widening architecture
- stop when the next move becomes speculative instead of measured
- judge success primarily by built/shrunk non-gzipped wasm, not by gzip deltas

This preserves the spirit of `0.63`:

- consolidation first
- ownership clarity first
- size work only when it is measured and attributable
