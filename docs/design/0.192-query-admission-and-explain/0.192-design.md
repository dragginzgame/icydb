# 0.192 Query Admission And EXPLAIN

Status: proposed.

Theme: plan-aware read admission for bounded public canister query surfaces.

0.192 makes read execution product-safe by defining explicit query lanes,
conservative public admission policy, stable rejection diagnostics, and EXPLAIN
output that reports both the selected plan and whether that plan is admitted for
a given canister surface.

## Context

0.190 established the reliability proof baseline for IC-native marker, journal,
recovery, failpoint, persisted-format corpus, and source-audit behavior. 0.191
productizes that baseline into durability, backup/import posture, checksum
policy, and operator contracts.

0.192 should productize the read side of the query surface. IcyDB already has a
mature single-entity SQL and fluent query substrate:

- scalar and grouped `SELECT`;
- SQL `INSERT`, `UPDATE`, and `DELETE`;
- bounded public write policies;
- typed SQL output values;
- blob output and blob comparison boundaries;
- aggregate and grouped execution;
- EXPLAIN text and JSON descriptors;
- stable compact diagnostic codes;
- compound indexes and unique-index integrity;
- accepted schema snapshots as runtime authority.

The gap is not missing SQL power. The gap is admission: the same query engine
that is useful for administrators and diagnostics can become a public canister
footgun if generated/public read endpoints admit unbounded scans, large
responses, materialized sorts, or ambiguous ad-hoc SQL without an explicit
policy.

## Entry Criteria

0.192 starts only after:

- 0.190 closeout evidence remains green;
- 0.191 durability productization has either closed or explicitly declared that
  query admission can proceed without depending on unresolved checksum/import
  decisions;
- accepted schema snapshots remain the runtime authority for query planning;
- existing SQL write policies and generated admin/controller gates are green;
- EXPLAIN JSON/text baseline behavior is covered by tests or snapshots;
- no persisted-format change is required for the first 0.192 slice.

This sequencing is intentional. Query admission was identified as the next
product-safety gap after the post-0.190 audit, but 0.191 is already assigned to
durability productization and format policy. 0.192 therefore owns query
admission while preserving 0.191 as the durability contract line.

## Core Problem

Today, IcyDB has stronger policy discipline for SQL writes than for SQL reads.
`UPDATE` and `DELETE` can be classified through primary-key-only and bounded
deterministic lanes, with limits and `RETURNING` bounds. Fluent queries can
assert access requirements after planning, and EXPLAIN can describe selected
access paths. Generated SQL endpoints are controller-gated.

Those are strong primitives, but they do not yet form a product-level public
read contract. 0.192 defines that contract.

The central question is:

Which read queries are safe for public endpoints, which are admin-only, which
are diagnostics-only, and what exact diagnostic or EXPLAIN output explains a
rejection?

## Read Execution Scope

For 0.192, "read admission" applies to IcyDB read execution surfaces, not only
to IC query methods. Any generated or public-facing method that executes a
read-only SQL or fluent query must choose a lane.

This includes:

- IC query methods;
- controller/admin query helpers;
- diagnostic EXPLAIN endpoints;
- update methods that expose read-like query behavior without using the write
  policy layer.

This rule prevents update-method wrappers from bypassing read admission merely
because the outer IC method is not an IC query.

## Goals

- Define explicit read execution lanes:
  - `PublicRead`;
  - `AdminAdHoc`;
  - `DiagnosticExplain`;
  - `DevTest`.
- Add a read-side admission policy equivalent in authority to the existing SQL
  write policy layer.
- Require public read surfaces to choose a policy explicitly.
- Make unsafe public full scans reject by default.
- Support policy controls for:
  - required `LIMIT`;
  - maximum returned rows;
  - maximum scanned rows when a bounded scan can be proven or mechanically
    enforced;
  - maximum encoded response bytes;
  - index-required mode;
  - full-scan rejection;
  - materialized sort rejection or budget;
  - projection width or response budget;
  - grouped-query group count and group-byte budgets;
  - cursor or continuation behavior where the surface supports it.
- Add stable diagnostic codes for read-admission rejection reasons.
- Extend EXPLAIN text and JSON output with admission status for the requested
  lane.
- Keep SQL and fluent query admission behavior aligned where they express the
  same logical query.
- Preserve the 0.190 and 0.191 durability assumptions: no recovery, marker,
  journal, stable-memory, or persisted-format changes are required for this
  line.

## Non-Goals

- Do not pursue SQLite compatibility.
- Do not add joins.
- Do not embed SQLite in canisters.
- Do not add general relational transaction semantics.
- Do not make EXPLAIN a promise of exact row counts or exact instruction costs.
- Do not let estimated planner costs authorize public query execution.
- Do not add async, reentrant, threaded, or canister-call execution to query
  admission.
- Do not change persisted row, index, schema, journal, marker, cursor, or fold
  watermark formats.
- Do not weaken controller gates for current admin SQL helpers.
- Do not expose admin ad-hoc query power as the default public endpoint shape.

## Existing Baseline

### SQL Contract

`docs/contracts/SQL_SUBSET.md` defines IcyDB SQL as a constrained
single-entity SQL frontend. It explicitly rejects a general-purpose relational
SQL contract and keeps typed/fluent APIs as canonical public surfaces.

Current documented read shapes include:

- scalar row loads;
- scalar `DISTINCT` loads;
- global aggregate loads;
- grouped aggregate loads;
- aggregate `DISTINCT` and `FILTER`;
- admitted computed projections;
- `ORDER BY`;
- `LIMIT` and `OFFSET`;
- EXPLAIN;
- introspection commands.

That contract is strong enough to serve as the semantic SQL boundary. 0.192
should add the operational admission boundary.

### Write Admission

`crates/icydb-core/src/db/session/sql/write_policy.rs` already contains the
shape of a useful policy vocabulary:

- primary-key `WHERE` proof;
- canonical primary-key `ORDER BY` proof;
- positive `LIMIT` requirement for bounded writes;
- offset rejection;
- staged-row bounds;
- `RETURNING` row and response-byte bounds.

`update_policy.rs` and `delete_policy.rs` apply that vocabulary to generated
and public write surfaces. Read admission should mirror this style: classify the
selected query shape, carry execution bounds, and reject with precise stable
diagnostics.

### Fluent Access Requirements

`crates/icydb-core/src/db/query/intent/access_requirement.rs` already supports
post-planning fail-closed assertions:

- require a secondary index;
- require a specific index name;
- require a specific access path;
- forbid residual filters.

This is the correct model for public read admission: evaluate the selected plan
after planning rather than treating policy as a planner hint.

### EXPLAIN

`crates/icydb-core/src/db/query/explain/json.rs` renders canonical EXPLAIN JSON
with fields for:

- execution node type;
- access strategy;
- predicate pushdown;
- residual filter expression;
- projection;
- ordering source;
- limit;
- cursor;
- covering scan;
- expected rows when available;
- node properties.

0.192 should add admission fields beside these plan fields. EXPLAIN should tell
the developer both what the engine selected and why a public/admin/diagnostic
lane would admit or reject it.

### Diagnostics

`crates/icydb-diagnostic-code/src/lib.rs` provides stable compact public error
identity. Existing categories include query validation, query plan, access
requirements, unsupported SQL features, SQL surface mismatch, and SQL write
boundary errors.

0.192 should add a read-admission diagnostic family or extend the existing
query access boundary with stable leaf reasons.

## Query Lanes

### PublicRead

`PublicRead` is the default safe lane for public canister read endpoints.

Required properties:

- no full scan unless the policy explicitly allows a formally bounded full
  scan;
- a positive `LIMIT` when the result can return rows;
- returned-row count bounded by policy;
- response bytes bounded conservatively;
- materialized sort rejected unless bounded by policy;
- grouped execution admitted only with group count and group-byte budgets;
- residual filters allowed only when the selected access path and scan bound
  remain within policy;
- introspection commands rejected unless a surface explicitly opts in;
- EXPLAIN-only execution can be admitted without running the query.

The initial default should be conservative:

- require an index-backed access path for non-primary-key reads;
- reject unbounded full scans;
- require `LIMIT` for scalar row-producing SQL;
- reject `OFFSET` unless skipped rows are included in a proven or enforced scan
  bound;
- cap returned rows and encoded response bytes;
- reject materialized sort unless the limit and scan bound are both admitted;
- require grouped limits for grouped queries.

`PublicRead` policies must be finite by construction. The implementation should
prefer checked constructors over direct struct literals for public policies.
Critical public caps such as `max_returned_rows` and `max_response_bytes` must
not be absent. If `max_scanned_rows` is absent, the policy must reject any plan
that requires a scan-bound proof.

### AdminAdHoc

`AdminAdHoc` is for controller-gated operations, local shell use, and trusted
maintenance workflows.

It may allow:

- full scans;
- wider projections;
- introspection;
- larger result limits;
- diagnostic-heavy EXPLAIN;
- DDL-adjacent schema inspection.

It must still carry explicit bounds. Admin power should be opt-in and visible
in generated code and configuration. The important distinction is not that
admin queries are unbounded forever; it is that they are not confused with
public endpoint safety.

Admin EXPLAIN may display estimates, broader plan details, index names, schema
descriptors, and configured policy limits. Those estimates are diagnostics, not
public admission authority.

### DiagnosticExplain

`DiagnosticExplain` is explain-only. It should parse, lower, plan, and run
admission, but it should not execute data reads beyond what planning requires.

It should report:

- selected access path;
- selected index name when present;
- compound component usage when available;
- pushed-down predicate;
- residual predicate;
- ordering source;
- whether sort/materialization is required;
- limit and cursor behavior;
- admission lane;
- admission decision;
- rejection reason when rejected;
- relevant policy limit values.

`DiagnosticExplain` is not automatically public-safe. Generated public EXPLAIN
surfaces must either be disabled by default or use a redacted explain policy.
Admin/controller EXPLAIN may include full index names, schema descriptors, and
policy limits. Public EXPLAIN, if enabled, must not reveal raw values, hidden
fields, private schema metadata, or admin-only introspection.

### DevTest

`DevTest` is for unit tests, integration tests, and local development helpers.
It may bypass some product defaults when the test names that intent explicitly.
It must not leak into production generated endpoints.

Source-audit guards should preserve the existing `_for_tests` and test-helper
boundaries.

## Admission Model

Admission happens after logical lowering and physical access planning.

Planning answers: "What would IcyDB do?"

Admission answers: "Is that selected plan allowed for this surface?"

This distinction is important. Public policy must not be a hint that silently
reranks plans. If the selected plan violates policy, the query fails closed with
the selected plan attached to diagnostics and EXPLAIN.

Public admission is based on hard bounds, conservative static upper bounds, or
mechanically enforced runtime caps. Estimates may be displayed in EXPLAIN, but
they are never sufficient to admit a `PublicRead` query.

### Bound Taxonomy

Admission should classify every policy-relevant bound with a compact taxonomy:

```rust
pub enum QueryBoundKind {
    Exact,
    ConservativeUpperBound,
    EnforcedRuntimeCap,
    EstimateOnly,
    Unavailable,
}
```

`PublicRead` admits `Exact`, `ConservativeUpperBound`, and
`EnforcedRuntimeCap` bounds. It rejects `EstimateOnly` and `Unavailable` when a
proof is required. `AdminAdHoc` may display estimate-only information, but it
should still carry explicit execution caps.

### Proposed Core Types

Names are provisional, but the implementation should converge on a small
vocabulary like:

```rust
pub enum QueryAdmissionLane {
    PublicRead,
    AdminAdHoc,
    DiagnosticExplain,
    DevTest,
}

pub struct QueryAdmissionPolicy {
    pub lane: QueryAdmissionLane,
    pub require_limit: bool,
    pub max_returned_rows: Option<u32>,
    pub max_scanned_rows: Option<u64>,
    pub max_response_bytes: Option<u32>,
    pub require_index: bool,
    pub allow_full_scan: bool,
    pub allow_materialized_sort: bool,
    pub max_materialized_rows: Option<u32>,
    pub max_projection_columns: Option<u16>,
    pub grouped: GroupedAdmissionPolicy,
}

pub struct QueryAdmissionSummary {
    pub lane: QueryAdmissionLane,
    pub decision: QueryAdmissionDecision,
    pub selected_access: String,
    pub selected_index: Option<String>,
    pub limit: Option<u32>,
    pub scan_bound: Option<u64>,
    pub scan_bound_kind: QueryBoundKind,
    pub returned_row_bound: Option<u32>,
    pub returned_row_bound_kind: QueryBoundKind,
    pub response_byte_bound: Option<u32>,
    pub response_byte_bound_kind: QueryBoundKind,
    pub materialization: QueryMaterializationSummary,
    pub rejection: Option<QueryAdmissionRejection>,
}
```

The actual code should follow local module boundaries. The design intent is the
important part: one policy object, one summary, one stable rejection family.
Public policies should be created through checked constructors so unbounded
public caps cannot be created accidentally.

### Admission Order

Recommended admission order:

1. Confirm the statement family is allowed for the lane.
2. Confirm recovery/readiness has already been enforced by the session/runtime
   path.
3. Confirm the selected access path satisfies index/full-scan policy.
4. Confirm the query has a positive limit when required.
5. Compute returned-row bound from limit, cursor/windowing, aggregate shape, or
   grouped policy.
6. Compute scan bound and scan-bound kind when the selected access path can
   prove or enforce one.
7. Apply scan budget.
8. Apply materialized sort/materialized projection policy.
9. Apply grouped-query budgets.
10. Apply response-byte policy using conservative bounds or execute-time
    streaming checks where static proof is not possible.
11. Produce `Admitted` or a stable rejection reason.

If a bound cannot be proven and the lane requires proof, reject.

## Public Policy Defaults

Initial public defaults should favor safety over convenience:

- `require_limit = true` for row-producing scalar SQL;
- `require_index = true` for non-primary-key filters;
- `allow_full_scan = false`;
- `allow_materialized_sort = false` unless the selected route and limit provide
  a small bounded materialization;
- `max_returned_rows` should be finite;
- `max_response_bytes` should be finite;
- grouped queries require explicit group and group-byte limits;
- introspection is disabled for public surfaces;
- EXPLAIN can be exposed only through an explain-only lane.

Admin defaults may be broader, but they must still be named and configured.

## Bounded Full Scan Policy

A bounded full scan is admitted only when the engine can prove or enforce a
physical scan cap before execution. A `LIMIT` on returned rows is not by itself
a scan bound unless the selected access path can stop after that limit without
examining additional rows.

Examples:

- admitted: store cardinality is known and no greater than
  `policy.max_scanned_rows`;
- admitted: the selected plan has an enforceable physical scan cap and stops at
  that cap;
- rejected: `LIMIT 10` over an unordered full scan when matching rows may
  require scanning the whole store;
- rejected: `COUNT(*)` over an unbounded store, even though it returns one row.

## Residual Filter Policy

Residual filters are admitted for `PublicRead` only when the selected access
path and scan bound remain within policy.

Examples:

- admitted: `WHERE status = 'open' AND description LIKE 'abc%'` using
  `by_status` with `scan_bound <= policy.max_scanned_rows`;
- rejected: `WHERE description LIKE '%abc%'` with no index and no scan bound;
- rejected: `WHERE status = 'open'` with `by_status` selected when status
  cardinality is unavailable and the policy requires `max_scanned_rows`.

An index-selected plan is not automatically safe. The residual work still has
to fit the lane.

## Aggregate Admission Policy

Global aggregates are row-producing only in the response sense. They may still
require unbounded input scans.

`PublicRead` admits aggregate queries only when:

- the selected access path has a proven or enforced scan bound;
- aggregate state has a bounded memory footprint;
- `DISTINCT` aggregate state has a hard entry or byte budget;
- response bytes are bounded;
- materialized aggregate state cannot exceed policy.

`COUNT(*)` over an unbounded full scan is rejected for `PublicRead` unless the
store cardinality is known and within policy. `MIN` and `MAX` may be admitted
when the selected index order proves bounded access; they are not admitted
merely because they return one row.

## Offset And Cursor Policy

`PublicRead` rejects `OFFSET` by default unless skipped rows are included in
the proven or enforced scan bound. Deep pagination should use cursor or
continuation semantics rather than `OFFSET`.

Cursor resume must preserve the original lane, policy-relevant bounds, selected
index and schema fingerprint where available, and response/scan budgets. A
cursor must not allow a query rejected at initial admission to resume through a
weaker path.

This is a policy over existing cursor behavior. It does not require a cursor
persisted-format change for the first 0.192 slice.

## Diagnostic Codes

0.192 should add stable diagnostics equivalent to the following conceptual
reasons:

- `PublicQueryRequiresLimit`;
- `PublicQueryRequiresIndex`;
- `UnboundedFullScanRejected`;
- `ScanBoundUnavailable`;
- `ScanBoundExceedsPolicy`;
- `EstimatedOnlyBoundRejected`;
- `SortRequiresMaterialization`;
- `MaterializationExceedsBudget`;
- `ProjectionResponseMayExceedLimit`;
- `GroupedQueryRequiresLimits`;
- `GroupedQueryExceedsBudget`;
- `DiagnosticLaneDoesNotExecute`;
- `IntrospectionDisabledForLane`;
- `UnsupportedStatementForQueryLane`.

Exact names can follow the existing diagnostic-code style, but each rejection
must be stable and specific enough for developers to fix the query.

## EXPLAIN Contract

EXPLAIN should report admission separately from planning.

Text EXPLAIN should include a compact admission block, for example:

```text
admission lane=PublicRead decision=rejected reason=PublicQueryRequiresLimit
policy require_limit=true max_returned_rows=100 max_response_bytes=1048576
selected access=IndexRange index=by_status residual_filter=false
```

JSON EXPLAIN should add a stable object, for example:

```json
{
  "admission": {
    "lane": "PublicRead",
    "decision": "rejected",
    "reason": "PublicQueryRequiresLimit",
    "require_limit": true,
    "max_returned_rows": 100,
    "max_scanned_rows": 1000,
    "max_response_bytes": 1048576,
    "selected_access": "IndexRange",
    "selected_index": "by_status",
    "scan_bound": 250,
    "scan_bound_kind": "ConservativeUpperBound",
    "returned_row_bound": null,
    "returned_row_bound_kind": "Unavailable",
    "materialized_sort": false,
    "residual_filter": false
  }
}
```

Do not make the JSON shape depend on rich debug prose. It should use compact,
stable strings or numeric diagnostic codes consistent with existing diagnostic
policy.

Public runtime errors should include stable, actionable diagnostic identity and
a compact selected-plan summary only where safe for the lane. Full plan details
belong in EXPLAIN, and public EXPLAIN may be redacted.

## SQL And Fluent Parity

SQL and fluent paths do not need identical APIs, but they must agree on the
same admission facts when they lower to the same logical query:

- selected access path;
- index-required success or failure;
- full-scan rejection;
- limit requirement;
- offset rejection;
- scan-bound kind;
- response-row bound;
- aggregate admission;
- grouped limits;
- residual-filter policy;
- materialized-sort policy.

Fluent APIs may continue to expose explicit access assertions such as
`require_index()` and `require_no_residual_filter()`. Public generated surfaces
should apply policy even when a caller does not opt into those assertions.

## Generated Endpoint Policy

Generated canister helpers should not infer public safety from the presence of
SQL support.

Required generated-surface decisions:

- admin/controller SQL query endpoint remains explicitly admin-lane;
- public read endpoint, if generated, must name a `PublicRead` policy;
- EXPLAIN endpoint, if generated, must name a `DiagnosticExplain` policy;
- introspection remains disabled for public IC surfaces by default;
- update/write policy remains separate and unchanged except for shared
  diagnostic vocabulary where useful.

Configuration should make the lane visible in generated code or build metadata.
An omitted policy should fail closed rather than defaulting to broad ad-hoc
power.

0.192 closeout must include a checked-in inventory of read execution surfaces
and the lane each surface uses. Any production read surface without an explicit
lane is a closeout blocker.

## Response-Size Policy

Write `RETURNING` already has response-byte checks. Read admission should reuse
that lesson without overpromising static precision.

Acceptable implementation strategy:

- statically reject projections that are obviously wider than policy;
- use row limits and known fixed-size fields to compute conservative upper
  bounds where possible;
- require execute-time response-byte checks for variable-width text/blob/list
  outputs;
- stop finalization and return a stable admission/boundary diagnostic before
  crossing the response limit;
- keep blob output supported but bounded.

If a public lane cannot prove or enforce the byte bound, it must reject.

Response-byte enforcement must not return partial successful result sets. If
the encoded response would exceed policy, execution returns a stable boundary
error. For read-only execution this is safe because no mutation has occurred.
For any surface that combines reads with writes, the write policy boundary
remains authoritative and read admission must not justify post-write partial
output.

## Materialization Policy

Materialization is not inherently forbidden. Unbounded materialization is.

Public policy should classify:

- no materialization;
- bounded top-N or index-range limit pushdown;
- bounded materialized sort;
- unbounded materialized sort;
- grouped/hash materialization;
- projection materialization.

The first implementation may conservatively reject materialized sorts in the
public lane unless an existing top-N or limit-pushdown path proves the bound.

## Grouped Query Policy

Grouped execution already has `GroupedExecutionConfig` with group count and
group-byte limits. 0.192 should make those limits mandatory for public grouped
queries.

Public grouped admission should reject when:

- no group-count limit is present;
- no group-byte limit is present;
- distinct aggregate state has no bounded policy;
- fallback to hash grouping would exceed policy;
- HAVING or aggregate filters require unbounded materialization.

Admin and diagnostic lanes may allow larger grouped budgets, but they should
still surface the configured limits in EXPLAIN.

## Recovery And Readiness Behavior

0.192 must not bypass recovery authority.

Admission and EXPLAIN run after the same recovery/readiness checks as normal
read execution. If recovery is incomplete, marker-bound state is unsafe,
accepted schema snapshots cannot be loaded, or persisted bytes fail closed,
query admission does not get a special path around that failure.

The admission layer must not:

- read raw stable stores directly;
- reconstruct runtime authority from generated models;
- ignore index readiness;
- clear recovery markers;
- fold journal state;
- publish schema snapshots;
- treat EXPLAIN as a recovery inspection bypass.

## Failure Modes

Expected failure modes are query-boundary failures, not recovery failures:

- statement family not allowed for the lane;
- public read missing `LIMIT`;
- public read using `OFFSET` without a proven or enforced skipped-row bound;
- full scan selected under no-full-scan policy;
- secondary index required but not selected;
- selected named index does not match policy;
- residual filter forbidden by policy;
- scan bound unavailable;
- scan bound exceeds policy;
- estimate-only bound supplied where public policy requires proof;
- materialized sort required but disallowed;
- materialized result exceeds policy;
- aggregate query has no proven or enforced input scan bound;
- aggregate or `DISTINCT` state has no hard memory/entry/byte budget;
- grouped query missing hard limits;
- grouped query exceeds hard limits;
- projected response may exceed byte policy;
- response byte limit exceeded during finalization;
- introspection disabled for lane.

Each failure should include a compact selected-plan summary where safe for the
lane. Full plan details belong in EXPLAIN, not necessarily in ordinary public
runtime errors.

## API And Schema Changes

Expected API changes:

- add read admission policy types in the query/session layer;
- add read admission summaries to EXPLAIN diagnostics;
- add generated build configuration for public/admin/read/explain policies;
- add stable diagnostic codes and registry entries;
- add optional host/CLI rendering for admission decisions.

Expected schema and persisted-format changes:

- none.

If a later slice persists query policy in schema snapshots, that is a separate
persisted-format decision and must follow the 0.191 persisted-format policy.

## Feature-Gate Policy

Admission core types must compile in the supported no-default matrix if fluent
queries depend on them. SQL-specific admission integration must remain behind
SQL feature gates. EXPLAIN-specific rendering must remain behind the existing
EXPLAIN feature boundary where applicable.

This prevents read admission from reintroducing helper leakage or feature-matrix
regressions.

## Backward Compatibility And Rollout

0.192 is allowed to make unsafe public read surfaces fail closed. That is a
product-safety correction, not a SQLite-compatibility regression.

Rollout rules:

- admin/controller SQL endpoints keep existing broad behavior only under the
  `AdminAdHoc` lane and controller guard;
- public generated read endpoints without an explicit policy fail closed;
- existing generated code needs a migration note explaining how to select a
  `PublicRead` policy or keep read SQL admin-only;
- public EXPLAIN, if enabled, must choose a redaction policy explicitly.

## Validation Plan

Unit tests:

- public read rejects missing `LIMIT`;
- public read rejects full scan;
- public read admits indexed bounded query;
- public read rejects estimate-only scan bounds;
- public read rejects `OFFSET` unless skipped rows are included in the scan
  bound;
- public read rejects `COUNT(*)` over an unbounded full scan;
- public read rejects materialized sort without bounded proof;
- public read admits bounded top-N or index-limit-pushdown shape when supported;
- public read rejects grouped query without hard limits;
- public read rejects `DISTINCT` aggregate state without hard entry or byte
  budget;
- response-byte budget rejects large text/blob projection;
- response-byte overflow returns a boundary error, not a partial successful
  result;
- EXPLAIN-only reports rejection without executing rows, proven with a counter,
  failpoint, or mock executor;
- fluent and SQL paths agree for equivalent access requirements.

Integration tests:

- generated admin endpoint admits broader ad-hoc query under controller guard;
- generated public endpoint applies public policy;
- generated introspection remains disabled for public IC target by default;
- generated EXPLAIN endpoint reports admission status;
- public EXPLAIN, if enabled, redacts lane-sensitive plan details.

Regression tests:

- existing SQL write policies stay unchanged;
- existing EXPLAIN plan fields remain stable or intentionally versioned;
- existing 0.190 IC update-model source guards remain green;
- no new async/await/spawn/canister-call tokens in write or recovery paths;
- no `_for_tests` helper leakage into production paths;
- admission core types compile in the supported no-default matrix when fluent
  reads depend on them;
- SQL-specific admission code remains behind SQL features;
- EXPLAIN-specific rendering remains behind the EXPLAIN feature boundary.

Documentation:

- update `docs/contracts/SQL_SUBSET.md` with the read-admission boundary;
- document generated endpoint lane defaults;
- document EXPLAIN admission output;
- add the read-surface inventory and lane map;
- add a migration note for generated/public read endpoints;
- mention that this line has no persisted-format impact.

## Acceptance Criteria

0.192 is complete when:

- a public read policy type exists;
- generated endpoints wire public, admin, and explain policies explicitly;
- unsafe public full scans are rejected by default;
- public row-producing reads require bounded output;
- `PublicRead` admission uses only proven bounds, conservative upper bounds, or
  enforced runtime caps;
- estimate-only bounds cannot admit a public query;
- `PublicRead` rejects `OFFSET` by default unless skipped rows are included in
  the proven or enforced scan bound;
- `PublicRead` aggregate queries require an input scan bound even when output
  row count is one;
- `PublicRead` rejects `COUNT(*)` over an unbounded full scan;
- `PublicRead` policies are finite by construction for returned rows and
  response bytes;
- scan, response, sort/materialization, projection, and grouped budgets are
  enforced or conservatively rejected;
- grouped and `DISTINCT` aggregate state has hard entry or byte budgets;
- EXPLAIN JSON includes admission status for a requested lane;
- text EXPLAIN includes a compact admission summary;
- public EXPLAIN, if enabled, has a redaction/confidentiality policy;
- stable diagnostic codes exist for all public read rejection reasons;
- SQL and fluent paths have parity tests for shared admission facts;
- a checked-in read-surface inventory proves every production read execution
  surface has an explicit lane;
- SQL-specific admission code remains correctly feature-gated;
- fluent/core admission works in the supported non-SQL matrix if fluent reads
  depend on it;
- EXPLAIN-only tests prove row execution is not invoked;
- docs define the intentionally bounded public SQL/read surface;
- no marker, journal, recovery, stable-memory, or persisted-format behavior is
  changed.

## Relationship To 0.190 And 0.191

0.192 depends on the 0.190 reliability baseline and must not weaken it. Query
admission is a pre-execution boundary layered above accepted schema authority
and guarded recovery. It does not alter marker/journal semantics, failpoint
coverage, persisted-format decoding, or stable-memory ownership.

0.192 follows the 0.191 durability productization line. 0.191 answers what
durable storage and persisted bytes mean to operators. 0.192 answers which read
queries may safely execute through public, admin, and diagnostic canister
surfaces.

## Suggested Implementation Slices

### Slice 1: Core Admission Vocabulary

Add lane, policy, decision, rejection, summary, bound-kind, and diagnostic-code
types. Do not change SQL behavior yet.

Closeout:

- constructors enforce safe `PublicRead` defaults;
- diagnostic registry entries exist;
- no persisted-format change.

### Slice 2: Plan Summary Extraction

Teach the planner/executor boundary to produce a compact admission-facing
summary:

- selected access;
- selected index;
- full scan versus key lookup versus index range;
- residual filter;
- ordering source;
- limit;
- offset;
- materialization;
- aggregate/grouped shape;
- scan-bound kind;
- returned-row bound.

Closeout:

- unit tests cover representative plans;
- no runtime policy enforcement yet.

### Slice 3: PublicRead Enforcement For Scalar SELECT

Start with scalar row-producing `SELECT`.

Closeout:

- missing `LIMIT` rejected;
- full scan rejected;
- indexed bounded query admitted;
- offset policy enforced;
- response row cap enforced.

### Slice 4: Materialization And Response Bytes

Add sort, materialization, projection, and byte caps.

Closeout:

- large text/blob/list projection trips byte policy;
- materialized sort is rejected unless bounded;
- byte overflow does not return a partial successful result.

### Slice 5: Aggregates And Grouped Queries

Add global aggregate and grouped policy.

Closeout:

- `COUNT(*)` over unbounded full scan rejected;
- grouped query requires group count and group-byte caps;
- distinct aggregate state is bounded or rejected.

### Slice 6: EXPLAIN Admission Output

Add text and JSON admission blocks.

Closeout:

- JSON snapshot tests cover admission;
- public/admin/diagnostic lane differences are visible;
- public EXPLAIN redaction policy is tested;
- explain-only execution does not execute row reads.

### Slice 7: Generated Endpoints And Integration

Wire generated public, admin, and explain policies.

Closeout:

- omitted public policy fails closed;
- admin ad-hoc remains controller-gated;
- public introspection is disabled by default;
- read-surface inventory is checked in;
- SQL and fluent parity tests pass.
