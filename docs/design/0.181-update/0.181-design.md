# IcyDB 0.181 - Safe and Bounded SQL UPDATE Roadmap

## Status

In progress for 0.181.

0.181.0 starts with Phase 0 / Slice A: document the current SQL `UPDATE`
boundary and preserve existing generated endpoint rejection behavior without
changing execution semantics.

## Summary

IcyDB already supports SQL `UPDATE` in the session/library SQL write lane. The
current implementation is not a parser stub: it parses `UPDATE`, lowers it,
selects target primary keys, builds structural patches, rejects unsafe field
writes, supports narrow write `RETURNING`, and routes mutation through the
accepted-schema structural save/commit pipeline.

The remaining work is not "add `UPDATE`." The remaining work is to make the
`UPDATE` boundary explicit by surface and to avoid accidental exposure of the
full internal write-lane semantics through generated canister SQL endpoints or
future public mutation helpers.

This design proposes a staged roadmap:

- Capture the current `UPDATE` contract, including narrow `RETURNING`.
- Introduce explicit `UPDATE` exposure policy gates.
- Define a conservative primary-key-only public/write policy.
- Later admit bounded deterministic non-primary-key `UPDATE` with explicit
  `ORDER BY primary_key LIMIT n`.
- Move completion, resumability, and retry semantics to cursor/token or admin
  job phases.
- Keep unbounded non-primary-key `UPDATE` out of public/generated mutation
  surfaces.

The central safety rule is:

> Public/generated SQL mutation must be stricter than broad session SQL
> mutation unless a surface explicitly opts into broader admin semantics.

## Current Source Reality

The source audit found these current facts.

- Parser support:
  - `UPDATE` is parsed by
    `crates/icydb-core/src/db/sql/parser/statement/update.rs`.
  - `SqlUpdateStatement` carries entity, optional alias, assignments,
    predicate, `ORDER BY`, `LIMIT`, `OFFSET`, and optional `RETURNING` in
    `crates/icydb-core/src/db/sql/parser/model.rs`.
  - Parser tests include `UPDATE ... RETURNING *`.
- Lowering support:
  - `lower_update_selector_shape` requires `WHERE`.
  - `ORDER BY` terms must be direct fields.
  - Core/session lowering appends missing primary-key order terms as an
    internal deterministic fallback.
  - `LIMIT` and `OFFSET` are passed into the lowered selector.
- Executor support:
  - `execute_sql_update_statement` validates `RETURNING` fields before mutation.
  - It rejects primary-key, generated, and managed/internal field assignment.
  - It selects target primary keys, applies structural update patches, and
    shapes either count output or `RETURNING` projection output.
- Narrow write `RETURNING` support:
  - `RETURNING *` and `RETURNING field, ...` are admitted on the SQL write lane.
  - Computed and aggregate `RETURNING` projections are rejected before mutation.
  - Unknown or duplicate returning fields are rejected before mutation.
- Public library/session write lane:
  - `DbSession::execute_sql_update::<E>(...)` exists in core and facade APIs.
  - It admits `INSERT`, `UPDATE`, and `DELETE` mutation SQL for one entity type.
  - It returns SQL-shaped count or projection results.
- Generated canister SQL endpoints:
  - Generated SQL currently exports `__icydb_query` and `__icydb_ddl` when the
    relevant features are enabled.
  - `__icydb_query` dispatches to `execute_sql_query_with_perf_attribution`.
  - `__icydb_ddl` dispatches to `execute_sql_ddl`.
  - No generated `__icydb_update` SQL endpoint was found in the inspected
    macro/build code.
  - Generated query integration tests prove the query endpoint rejects mutation
    SQL for `DELETE` and `UPDATE`.
- CLI routing:
  - The CLI SQL shell routes SQL to configured query or DDL endpoints.
  - The CLI route classifier currently exposes only `Query` and `Ddl` call
    kinds.
  - Row mutation SQL is not routed to a generated SQL write endpoint because no
    such configured endpoint exists today.
- Existing docs:
  - `docs/contracts/SQL_SUBSET.md` already lists `INSERT`, `UPDATE`, `DELETE`,
    and narrow write `RETURNING` on the public SQL write lane.
  - `docs/architecture/SQL_SURFACE_MAPPING.md` already lists
    `execute_sql_update::<E>` as the mutation SQL owner.

The design must therefore distinguish session/library write support from
generated canister SQL endpoint exposure. It must also treat `UPDATE RETURNING`
as current narrow support that needs policy formalization, not as a missing
feature.

## Problem

The current `execute_sql_update::<E>` write lane is broader than the safest
generated/public canister mutation policy should be.

Core/session SQL currently admits non-primary-key multi-row `UPDATE` shapes such
as:

```sql
UPDATE Character
SET active = false
WHERE age = 21
```

It also supports bounded ordered forms:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10
```

And internally it may append primary-key fallback ordering where no explicit
ordering is supplied.

That behavior can be useful for session/library or admin-style execution, but it
is risky as an exported generated canister mutation surface because public
canister methods must account for:

- accidental mass updates;
- unbounded target selection;
- target primary-key materialization costs;
- canister instruction limits;
- trap/retry behavior;
- deterministic ordering;
- durable receipts;
- replay behavior;
- resumability;
- stable-memory mutation windows;
- index, reverse-index, relation, and uniqueness maintenance;
- user-facing documentation guarantees.

The dangerous failure mode is not that `UPDATE` is absent or inherently
incorrect. The dangerous failure mode is that a future generated SQL write
endpoint or public helper wires directly into broad session `UPDATE` semantics
and accidentally exposes unbounded bulk mutation as a public API.

## Goals

1. Preserve current session/library `UPDATE` behavior unless a concrete
   correctness issue is found.
2. Make the current `UPDATE` and `UPDATE RETURNING` boundary explicit in code,
   tests, and docs.
3. Ensure generated query and DDL endpoints continue to reject row-mutation SQL.
4. Define an explicit policy gate for any public/generated write surface.
5. Define a conservative public/write primary-key `UPDATE` proof.
6. Define strict bounded non-primary-key `UPDATE` ordering and limit rules.
7. Defer completion, replay, and resumability guarantees to cursor/token or
   admin job phases.

## Non-Goals

- Do not remove current session/library `UPDATE`.
- Do not expose unrestricted bulk `UPDATE` through generated canister SQL
  endpoints.
- Do not widen `RETURNING` beyond the current narrow `*` and field-list forms in
  the first policy slices.
- Do not introduce `UPDATE ... JOIN`, subquery `UPDATE`, CTE `UPDATE`, broad
  computed assignment expressions, or expression `RETURNING`.
- Do not implement a durable migration/job engine in this slice.
- Do not change parser grammar unless policy enforcement needs better shape
  inspection.

## Terminology

Core/session SQL means SQL executed through IcyDB internal session APIs and
generic core paths. It may expose broad primitives because callers are closer to
database authority.

Public library write lane means the existing
`execute_sql_update::<E>(...)` API exposed by the session facade. Today it owns
`INSERT`, `UPDATE`, and `DELETE` SQL mutation for one concrete entity type.

Generated query endpoint means the generated `__icydb_query` canister method.
It is a read/query endpoint and must reject row mutation SQL.

Generated DDL endpoint means the generated `__icydb_ddl` canister method. It is
an update method for accepted schema/catalog DDL and must reject row mutation
SQL.

Generated SQL write endpoint means a possible future generated canister method
for row mutation SQL. No generated `__icydb_update` SQL endpoint was found in
the inspected source. If introduced, it must use an explicit `UPDATE` policy and
must not inherit broad session semantics by default.

Generated typed/fluent write helpers means generated or facade helpers for
typed row creation, update, replace, or delete. They are not general SQL
mutation endpoints, but they are part of the broader public row mutation model.

CLI/dev shell means the CLI SQL shell that currently routes SQL to generated
query or DDL endpoints. Because no generated SQL write endpoint exists today,
the CLI does not currently provide generated-canister SQL `UPDATE` execution.

Primary-key `UPDATE` means an `UPDATE` whose `WHERE` clause is proven by a
conservative policy to identify at most one row by complete primary-key equality.

Bounded deterministic `UPDATE` means an `UPDATE` with a predicate, an explicit
canonical primary-key `ORDER BY`, and an explicit positive `LIMIT` at or below a
configured maximum.

Unbounded non-primary-key `UPDATE` means any non-primary-key `UPDATE` that may
affect an arbitrary number of rows and has no public-surface bound.

## Current UPDATE Boundary

The current implementation appears to have the following behavior.

| Capability | Current session/library status |
| --- | --- |
| Parser admits `UPDATE` | Yes |
| `WHERE` required for executable `UPDATE` | Yes |
| Normal stored field assignment | Yes |
| Primary-key field mutation | Rejected |
| Generated field mutation | Rejected |
| Managed/internal field mutation | Rejected |
| Non-primary-key predicates | Yes |
| Multi-row `UPDATE` | Yes |
| `ORDER BY` | Yes |
| Expression `ORDER BY` | Rejected |
| `LIMIT` | Yes |
| `OFFSET` | Yes |
| Implicit primary-key fallback ordering | Yes, in lowering |
| `RETURNING *` | Yes, on SQL write lane |
| `RETURNING field, ...` | Yes, on SQL write lane |
| Computed/aggregate `RETURNING` | Rejected before mutation |
| Uses accepted schema authority | Yes |
| Uses structural mutation path | Yes |
| Generated `__icydb_query` accepts `UPDATE` | No |
| Generated `__icydb_ddl` accepts row `UPDATE` | No; targeted integration coverage exists |
| Generated SQL write endpoint | Not currently found |
| Durable cursor/chunked bulk `UPDATE` | No |

This boundary is acceptable as the current session/library contract. It must not
be treated as the generated canister SQL contract.

## Normative Rules

- Public/generated SQL `UPDATE` MUST NOT inherit full core/session semantics by
  default.
- Generated query endpoints MUST reject `UPDATE`.
- Generated DDL endpoints MUST reject `UPDATE`.
- Any generated SQL write endpoint MUST choose an explicit `UPDATE` exposure
  policy before execution.
- Public primary-key `UPDATE` MUST prove complete primary-key equality.
- Public primary-key proof v1 MUST reject `OR`, `IN`, casts, coercions without
  an obviously safe canonical representation, expression equality, column
  equality, partial composite keys, duplicate key predicates, and extra non-PK
  predicates.
- Public bounded non-PK `UPDATE` MUST specify explicit primary-key `ORDER BY`
  and explicit `LIMIT`.
- Public bounded `UPDATE` MUST NOT rely on implicit primary-key fallback
  ordering.
- Public bounded `UPDATE` MUST reject `OFFSET`.
- Public bounded `UPDATE` MUST reject non-primary-key `ORDER BY`.
- Public bounded `UPDATE` MUST reject `DESC` in v1.
- Unbounded non-PK `UPDATE` MUST NOT be exposed through public/generated
  mutation surfaces.
- Bulk `UPDATE` MUST use cursor/token or admin job semantics before becoming
  broadly public.
- Narrow `RETURNING *` and `RETURNING field, ...` MAY remain supported where the
  write surface already admits them, but public bounded `RETURNING` MUST remain
  bounded by the same target-selection policy as the update itself.

Current and target policies should distinguish the surfaces explicitly.

| SQL shape | Core/session SQL | Generated query endpoint / `__icydb_query` | Generated DDL endpoint / `__icydb_ddl` | Public write lane / `execute_sql_update::<E>` | CLI/dev shell | Future admin/migration tooling |
| --- | --- | --- | --- | --- | --- | --- |
| `SELECT` | Yes | Yes | No | No | Yes, via `__icydb_query` | Maybe, read-only |
| DDL | Yes, via DDL lane | No | Yes | No | Yes, via `__icydb_ddl` | Yes |
| `INSERT` | Yes | No | No | Yes | No generated SQL write route today | Maybe, policy-gated |
| `UPDATE` by primary key | Yes | No | No | Yes today; target policy is explicit PK proof | No generated SQL write route today | Yes |
| `UPDATE` bounded non-PK | Yes | No | No | Yes today; target public policy requires explicit pk order and limit | No generated SQL write route today | Yes |
| `UPDATE` unbounded non-PK | Yes today | No | No | Yes today; must not be inherited by generated public surfaces | No generated SQL write route today | Yes, under job/admin policy |
| `UPDATE RETURNING` | Narrow `*` and field-list today | No | No | Narrow `*` and field-list today; policy must bound public use | No generated SQL write route today | Yes, bounded by job/admin policy |
| `DELETE` | Yes | No | No | Yes | No generated SQL write route today | Yes, under separate policy |

Notes:

- `execute_sql_update::<E>` is a public library/session write lane, not a
  generated canister SQL endpoint in the inspected source.
- `__icydb_query` currently rejects mutation SQL at the query execution surface.
- `__icydb_ddl` is an update method for schema/catalog DDL, not row mutation
  SQL.
- If a generated SQL write endpoint is added, it should not simply expose the
  current broad `execute_sql_update::<E>` behavior without a policy gate.
- The target-policy cells do not imply an immediate narrowing of the existing
  session/library `execute_sql_update::<E>` API. They define the policy that any
  generated/public canister SQL write exposure must select before using the
  underlying write machinery.

## Accepted and Rejected Shapes

### Current session/library SQL

The existing write lane may continue to support the current broad set:

```sql
UPDATE Character
SET age = 22
WHERE id = 1

UPDATE Character
SET active = false
WHERE age = 21

UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10

UPDATE Character
SET active = false
WHERE age = 21
LIMIT 10

UPDATE Character
SET age = 22
WHERE id = 1
RETURNING id, age
```

The `LIMIT` without explicit `ORDER BY` form is acceptable internally only
because core/session lowering supplies deterministic primary-key fallback
ordering. Public bounded policy must not rely on that fallback.

### Public/write policy, phase 1

Accept only primary-key `UPDATE` shapes that pass the v1 primary-key proof:

```sql
UPDATE Character
SET age = 22
WHERE id = 1
```

Reject non-primary-key `UPDATE`:

```sql
UPDATE Character
SET active = false
WHERE age = 21
```

Reject bounded non-primary-key `UPDATE` in phase 1:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10
```

That shape waits for the bounded deterministic phase.

### Public/write policy, bounded phase

Accept:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10
```

Reject implicit fallback:

```sql
UPDATE Character
SET active = false
WHERE age = 21
LIMIT 10
```

Reject non-primary-key ordering:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY name
LIMIT 10
```

Reject unbounded public non-PK update:

```sql
UPDATE Character
SET active = false
WHERE age = 21
```

### Always rejected without redesign

These should be rejected across all mutation surfaces unless explicitly
redesigned:

```sql
UPDATE Character
SET id = 2
WHERE id = 1

UPDATE Character
SET generated_field = 'x'
WHERE id = 1

UPDATE Character
SET managed_internal_field = 'x'
WHERE id = 1

UPDATE Character
SET active = false
```

The last form lacks `WHERE`.

## Primary-Key UPDATE Proof V1

Primary-key `UPDATE` is the safest first public/write policy because it targets
at most one row.

The v1 proof algorithm should be intentionally conservative:

- `WHERE` must normalize to a conjunction of equality predicates.
- Every primary-key component must appear exactly once.
- Predicate order does not matter.
- Parentheses are allowed only where they preserve a simple conjunction.
- Column references may be unqualified or qualified by the target table alias.
- Equality values may be simple literals or bind parameters if the SQL surface
  supports bind parameters.
- For a single-column primary key, `WHERE id = ?` or
  `WHERE target_alias.id = ?` is acceptable.
- For a composite primary key, every component must be constrained by equality
  in the `WHERE` clause.
- `OR` is rejected.
- `IN` is rejected.
- Range predicates are rejected.
- Casts/coercions are rejected unless existing typed lowering has an obviously
  safe canonical representation.
- Function calls and expression equality are rejected.
- Equality against another column is rejected.
- Missing composite-key parts are rejected.
- Duplicate or conflicting predicates on the same primary-key component are
  rejected.
- Extra non-primary-key predicates are rejected in v1, even when the full
  primary key is present.

Allowing additional guard predicates such as `AND active = true` can be a future
relaxation once the proof helper has a precise at-most-one contract. It should
not be part of the first public-safe policy.

### Accepted PK-Proof Examples

```sql
UPDATE Character
SET age = 22
WHERE id = 1

UPDATE Character AS c
SET age = 22
WHERE c.id = 1

UPDATE Membership
SET role = 'admin'
WHERE org_id = 10 AND user_id = 20
```

### Rejected PK-Proof Examples

```sql
UPDATE Character
SET age = 22
WHERE id IN (1)

UPDATE Character
SET age = 22
WHERE id = 1 OR id = 2

UPDATE Character
SET age = 22
WHERE id + 0 = 1

UPDATE Character
SET age = 22
WHERE CAST(id AS int) = 1

UPDATE Character
SET age = 22
WHERE id = other_id

UPDATE Membership
SET role = 'admin'
WHERE org_id = 10

UPDATE Character
SET age = 22
WHERE id = 1 AND active = true
```

The last example is intentionally rejected in v1 to keep the public proof
simple.

## Bounded Deterministic UPDATE Ordering V1

Bounded deterministic `UPDATE` is the second public/write policy. It admits
small multi-row updates only when the target set is deterministic and bounded.

The accepted shape is:

```sql
UPDATE Entity
SET field = value
WHERE predicate
ORDER BY primary_key
LIMIT n
```

The v1 ordering and limit rules are:

- `WHERE` is required.
- `ORDER BY` is required.
- `ORDER BY` must contain exactly the full primary key.
- Primary-key columns must appear in canonical schema primary-key order.
- Target-table alias qualification is allowed.
- Default direction or explicit `ASC` is accepted.
- `DESC` is rejected.
- Mixed directions are rejected.
- Partial primary-key ordering is rejected.
- Extra ordering columns are rejected.
- Non-primary-key ordering is rejected.
- Expression ordering is rejected.
- Positional ordering such as `ORDER BY 1` is rejected.
- Collation and null-order modifiers are rejected unless the existing SQL subset
  has a safe canonical representation.
- Public bounded `UPDATE` must not rely on implicit primary-key fallback
  ordering.
- `LIMIT` must be explicit.
- `LIMIT` must be positive.
- `LIMIT` must be at or below the configured public maximum.
- `OFFSET` is rejected.

### Accepted Bounded Examples

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10

UPDATE Membership
SET active = false
WHERE role = 'member'
ORDER BY org_id, user_id
LIMIT 10
```

### Rejected Bounded Examples

```sql
UPDATE Character
SET active = false
WHERE age = 21
LIMIT 10

UPDATE Character
SET active = false
WHERE age = 21
ORDER BY name
LIMIT 10

UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id DESC
LIMIT 10

UPDATE Membership
SET active = false
WHERE role = 'member'
ORDER BY user_id, org_id
LIMIT 10

UPDATE Membership
SET active = false
WHERE role = 'member'
ORDER BY org_id
LIMIT 10

UPDATE Membership
SET active = false
WHERE role = 'member'
ORDER BY org_id, user_id, created_at
LIMIT 10
```

## RETURNING Policy

`UPDATE RETURNING` is not a missing feature in the current write lane. The
current narrow support is:

```sql
UPDATE Character
SET age = 22
WHERE id = 1
RETURNING *

UPDATE Character
SET age = 22
WHERE id = 1
RETURNING id, age
```

The current narrow projection rules are:

- `RETURNING *` is admitted.
- `RETURNING field, ...` is admitted.
- Unknown returning fields are rejected before mutation.
- Duplicate returning fields are rejected before mutation.
- Computed returning expressions are rejected before mutation.
- Aggregate returning expressions are rejected before mutation.

Public/write policy should preserve those narrow forms only when the underlying
`UPDATE` shape is admitted by the same surface policy:

- primary-key `UPDATE RETURNING` may return at most one row;
- bounded non-PK `UPDATE RETURNING` must be bounded by the same explicit limit;
- unbounded non-PK `UPDATE RETURNING` must not be exposed through
  public/generated mutation surfaces.

Future `RETURNING` work should formalize result-size caps, field visibility,
ordering, and canister response-size implications. It should not widen
`RETURNING` until those constraints are explicit.

## Batch Completion Semantics

Phase 3 bounded `UPDATE` should return only conservative affected-row
information by default:

```text
rows_affected: u64
```

It should not expose `complete: bool` unless the executor can determine
completion without performing an unbounded scan and without overpromising retry
or continuation semantics.

Completion belongs naturally to the cursor/token phase, where a durable receipt
can define:

```text
update_token: Option<Token>
rows_matched: u64
rows_updated: u64
last_primary_key: Option<PrimaryKey>
complete: bool
```

In that phase, `complete` means the cursor protocol has no remaining work under
its documented predicate, assignment fingerprint, schema fingerprint, ordering,
and continuation rules.

## Roadmap

### Phase 0: Baseline Current Contract and Docs

#### Objective

Capture current `UPDATE` behavior without changing semantics.

#### Work

- Document that session/library SQL supports `UPDATE`.
- Document that `execute_sql_update::<E>` is the existing public library write
  lane for `INSERT`, `UPDATE`, and `DELETE`.
- Document current narrow `UPDATE ... RETURNING` support.
- Document that generated `__icydb_query` rejects row mutation SQL.
- Document that generated `__icydb_ddl` is schema/catalog DDL, not row mutation.
- Document that no generated SQL write endpoint was found in the inspected
  source.
- Avoid saying simply "IcyDB supports `UPDATE`" without naming the surface.

#### Tests

Preserve or add focused tests proving:

```sql
UPDATE Character SET age = 22 WHERE id = 1
```

works through `execute_sql_update::<E>`.

```sql
UPDATE Character SET active = false WHERE age = 21
```

works through `execute_sql_update::<E>` if this is already supported.

```sql
UPDATE Character SET age = 22 WHERE id = 1 RETURNING id, age
```

works through the current write lane.

```sql
UPDATE Character SET active = false
```

fails because `WHERE` is missing.

Generated query endpoint rejects:

```sql
UPDATE Character SET age = 22 WHERE id = 1
```

Generated DDL endpoint rejects row mutation SQL:

```sql
UPDATE Character SET age = 22 WHERE id = 1
```

#### Deliverable

A documented baseline that is accurate about current behavior and current
generated endpoint exposure.

#### Exit Criteria

A reader can determine exactly where `UPDATE` works, where `UPDATE RETURNING`
works, and where row mutation SQL is rejected.

### Phase 1: Explicit Statement Classification and Policy Gate

#### Objective

Prevent accidental exposure of broad session/library `UPDATE` through generated
public endpoints or future public write helpers.

#### Work

Introduce or clarify statement classification.

Suggested conceptual surfaces:

- `SqlSurface::SessionWrite`
- `SqlSurface::GeneratedQuery`
- `SqlSurface::GeneratedDdl`
- `SqlSurface::GeneratedWrite`
- `SqlSurface::CliDev`
- `SqlSurface::AdminMigration`

Suggested conceptual `UPDATE` policies:

- `UpdateExposurePolicy::SessionWriteCurrent`
- `UpdateExposurePolicy::PublicPrimaryKeyOnly`
- `UpdateExposurePolicy::PublicBoundedDeterministic`
- `UpdateExposurePolicy::AdminBulk`

The exact names should follow IcyDB conventions. The key requirement is that
generated/public mutation must have a separate gate from broad session
execution.

#### Policy Helper

Add a helper that can answer:

- Is this statement an `UPDATE`?
- What entity does it target?
- Which fields does it assign?
- Does it mutate primary-key/generated/managed fields?
- Does it have `WHERE`?
- Does the `WHERE` clause pass primary-key proof v1?
- Is it bounded?
- Does it have explicit canonical primary-key `ORDER BY`?
- Does it use `OFFSET`?
- Does it request `RETURNING`?
- Is `RETURNING` one of the narrow admitted forms?

This helper should be usable by generated endpoint routing before execution.

#### Tests

Add policy classification tests independent of execution.

Accepted by `SessionWriteCurrent`:

```sql
UPDATE Character SET active = false WHERE age = 21
```

Rejected by `GeneratedQuery`:

```sql
UPDATE Character SET active = false WHERE age = 21
```

Rejected by `GeneratedDdl`:

```sql
UPDATE Character SET active = false WHERE age = 21
```

Accepted by `PublicPrimaryKeyOnly`:

```sql
UPDATE Character SET age = 22 WHERE id = 1
```

Rejected by `PublicPrimaryKeyOnly`:

```sql
UPDATE Character SET active = false WHERE age = 21
```

Rejected by `PublicPrimaryKeyOnly` because v1 rejects extra guards:

```sql
UPDATE Character SET age = 22 WHERE id = 1 AND active = true
```

#### Deliverable

A fail-closed policy gate that must be consciously configured by each SQL
surface.

#### Exit Criteria

There is no route by which a generated public endpoint can inherit full
session/library `UPDATE` semantics by accident.

### Phase 2: Public/Write-Lane Primary-Key UPDATE Boundary

#### Objective

Define the smallest public-safe `UPDATE` policy: one row by complete primary-key
equality.

#### Accepted Shape

```sql
UPDATE Entity
SET field = value
WHERE primary_key = value
```

Composite keys require every primary-key component:

```sql
UPDATE Membership
SET role = 'admin'
WHERE org_id = 10 AND user_id = 20
```

#### Rejected Shapes

```sql
UPDATE Entity
SET field = value
WHERE non_pk_field = value

UPDATE Entity
SET field = value
WHERE primary_key IN (value)

UPDATE Entity
SET field = value
WHERE primary_key = value OR primary_key = other_value

UPDATE Entity
SET field = value
WHERE primary_key = value AND non_pk_field = value

UPDATE Entity
SET primary_key = value
WHERE primary_key = value

UPDATE Entity
SET generated_field = value
WHERE primary_key = value

UPDATE Entity
SET managed_field = value
WHERE primary_key = value
```

#### Result Contract

The result contract should be minimal:

```text
rows_affected: u64
```

Narrow `RETURNING` may be admitted if the same primary-key proof passes:

```sql
UPDATE Entity
SET field = value
WHERE primary_key = value
RETURNING primary_key, field
```

Do not widen `RETURNING` in this phase.

#### Safety Properties

- Target cardinality is at most one.
- No batching is required.
- No cursor is required.
- No `ORDER BY` is required.
- Instruction-limit surprise is limited to normal single-entity mutation costs.
- The structural mutation pipeline handles validation and commit behavior.

#### Tests

Generated or public write policy accepts:

```sql
UPDATE Character SET age = 22 WHERE id = 1
```

It rejects:

```sql
UPDATE Character SET active = false WHERE age = 21
```

Generated query endpoint still rejects all `UPDATE`.

Generated DDL endpoint still rejects all row `UPDATE`.

#### Deliverable

A public/generated write policy that admits only primary-key `UPDATE` unless a
surface explicitly opts into a broader policy.

#### Exit Criteria

Users can update one entity by primary key through the public-safe write policy,
and nothing broader is exposed through generated public SQL mutation.

### Phase 3: Explicit Bounded Deterministic UPDATE

#### Objective

Admit small multi-row public updates only when deterministic and bounded.

#### Accepted Shape

```sql
UPDATE Entity
SET field = value
WHERE predicate
ORDER BY primary_key
LIMIT n
```

#### Requirements

- `WHERE` is required.
- Explicit canonical primary-key `ORDER BY` is required.
- Explicit `LIMIT n` is required.
- `n` must be positive.
- `n` must not exceed a configured maximum.
- `OFFSET` is rejected.
- Non-primary-key `ORDER BY` is rejected.
- Expression `ORDER BY` is rejected.
- `DESC` is rejected in v1.
- Narrow `RETURNING` is allowed only if bounded by the same limit.
- No implicit primary-key fallback at the public boundary.

#### Limit Policy

Introduce a maximum public `UPDATE` limit.

Example:

```text
MAX_PUBLIC_UPDATE_LIMIT = 100
```

The exact value should be chosen by benchmarking and canister instruction budget
testing. It should be conservative by default and configurable only through
explicit generated-code or schema-level policy.

#### Result Contract

Return conservative affected-row information:

```text
rows_affected: u64
```

Do not expose `complete: bool` in Phase 3 by default. If completion cannot be
determined without an unbounded scan, it must wait for the cursor/token phase.

#### Why Explicit Ordering Matters

Core/session lowering may append primary-key fallback ordering. Public policy
should not rely on that.

The public caller should write:

```sql
ORDER BY id
LIMIT 100
```

because this makes the bounded-batch nature obvious in user code, generated
logs, docs, and reviews.

#### Tests

Accepted:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10
```

Rejected:

```sql
UPDATE Character
SET active = false
WHERE age = 21
LIMIT 10

UPDATE Character
SET active = false
WHERE age = 21
ORDER BY name
LIMIT 10

UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id DESC
LIMIT 10

UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id

UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 1000000
```

#### Deliverable

A public bounded batch `UPDATE` contract.

#### Exit Criteria

Public multi-row `UPDATE` exists only as an explicit, deterministic, bounded
batch, and it does not claim completion semantics.

### Phase 4: Cursor/Token Batch UPDATE

#### Objective

Support resumable bulk mutation without unbounded public execution.

#### Motivation

Users will eventually need to update many rows. Requiring them to manually
repeat:

```sql
UPDATE Entity
SET field = value
WHERE predicate
ORDER BY id
LIMIT 100
```

can work, but it does not provide a durable contract for continuation, receipts,
retries, completion, or progress reporting.

#### Proposed Shape

Introduce an explicit batch protocol.

Possible API-level shape:

```text
prepare_update(sql) -> update_token
execute_update_batch(update_token, limit) -> update_batch_receipt
```

Or SQL-like shape:

```sql
UPDATE Character
SET archived = true
WHERE last_seen_at < 1700000000
ORDER BY id
LIMIT 100
CONTINUE WITH token
```

The exact syntax is future work. Cursor-based bulk `UPDATE` should be a separate
design, not an accidental consequence of unbounded SQL.

#### Cursor Token Contents

A durable update cursor should include or reference:

- target entity;
- normalized predicate fingerprint;
- assignment patch fingerprint;
- canonical primary-key ordering;
- last processed primary key;
- accepted schema version or schema fingerprint;
- batch size policy;
- created timestamp or logical epoch;
- cumulative rows affected;
- completion flag;
- token version;
- authorization context if applicable.

The token should be opaque to clients.

#### Receipt Contents

Each batch should return:

```text
update_token: Option<Token>
rows_matched: u64
rows_updated: u64
last_primary_key: Option<PrimaryKey>
complete: bool
```

Possibly:

```text
batch_id: u64
mutation_fingerprint: Hash
schema_fingerprint: Hash
```

Here, `complete` is meaningful because the cursor protocol owns continuation
state and can define what remains.

#### Idempotency

The design must define what happens when a client retries a batch call.

Preferred behavior:

- repeated execution with the same completed batch token should not duplicate
  effects;
- each receipt should either commit atomically or report failure without partial
  visible mutation;
- continuation should be based on durable primary-key progress, not offset.

#### Tests

- Cursor resumes after a partial series of batches.
- Cursor rejects use after schema-incompatible changes.
- Cursor rejects changed assignment/predicate fingerprint.
- Cursor remains deterministic across retries.
- Cursor handles rows that no longer match the predicate.
- Cursor handles rows inserted before, during, and after the cursor range
  according to a documented policy.

#### Deliverable

A durable bounded bulk `UPDATE` mechanism.

#### Exit Criteria

IcyDB can safely support large logical updates through repeated bounded mutation
calls with explicit completion and retry semantics.

### Phase 5: Administrative Bulk UPDATE and Migration Jobs

#### Objective

Support administrator or migration-grade bulk `UPDATE` with explicit operational
semantics.

#### Motivation

Some updates are not user-facing API operations. They are schema migrations,
backfills, denormalization repairs, or operational jobs.

These should not be confused with ordinary public SQL mutation.

#### Possible Direction

Introduce a migration/admin job layer:

```text
start_update_job(sql, batch_size) -> job_id
run_update_job_batch(job_id) -> job_receipt
inspect_update_job(job_id) -> job_status
cancel_update_job(job_id)
```

#### Requirements

- explicit admin authority;
- durable job state;
- bounded batch execution;
- resumable progress;
- per-batch receipts;
- final summary;
- schema-version compatibility;
- clear trap/retry semantics;
- observability.

#### Deliverable

A separate admin/migration path for bulk data maintenance.

#### Exit Criteria

Large data rewrites are handled as jobs, not as unbounded public SQL calls.

### Phase 6: Formalize or Expand Bounded RETURNING

#### Objective

Formalize current narrow `UPDATE RETURNING` behavior by surface and, only if
safe, expand bounded result contracts.

#### Current Baseline

The parser, executor, docs, and tests already admit:

```sql
UPDATE Character
SET age = 22
WHERE id = 1
RETURNING *

UPDATE Character
SET age = 22
WHERE id = 1
RETURNING id, age
```

Unsupported forms remain fail-closed:

```sql
UPDATE Character
SET age = 22
WHERE id = 1
RETURNING LOWER(name)

UPDATE Character
SET age = 22
WHERE id = 1
RETURNING COUNT(*)
```

#### Formalization Requirements

- Returned rows must be bounded by the same update policy as target selection.
- Returned fields must respect projection and visibility rules.
- Result ordering must match mutation ordering.
- Generated/managed field exposure must follow existing read/write policy.
- Large result materialization must be capped.
- Unknown and duplicate returning fields must continue to fail before mutation.
- Computed and aggregate returning expressions must remain rejected unless a
  later design admits them explicitly.

#### Deliverable

A precise, surface-specific bounded `RETURNING` contract.

#### Exit Criteria

`UPDATE` result sets remain useful without creating an unbounded response-size or
mutation-result hazard.

## Safety Invariants

### Invariant 1: No mutation without WHERE

This must always fail:

```sql
UPDATE Entity SET x = 1
```

### Invariant 2: No primary-key mutation

This must always fail:

```sql
UPDATE Entity SET id = 2 WHERE id = 1
```

### Invariant 3: No generated-field mutation

This must always fail unless generated-field semantics are redesigned:

```sql
UPDATE Entity SET generated_value = 10 WHERE id = 1
```

### Invariant 4: No managed/internal-field mutation

Managed fields remain owned by the system.

### Invariant 5: Public non-PK UPDATE must be bounded

This must fail publicly:

```sql
UPDATE Entity SET x = 1 WHERE y = 2
```

### Invariant 6: Public bounded UPDATE must be explicitly ordered

This must fail publicly:

```sql
UPDATE Entity SET x = 1 WHERE y = 2 LIMIT 10
```

### Invariant 7: Public bounded UPDATE must order by primary key

This must fail publicly:

```sql
UPDATE Entity SET x = 1 WHERE y = 2 ORDER BY name LIMIT 10
```

### Invariant 8: Core/session behavior must not imply generated behavior

A shape accepted by session/library SQL is not automatically accepted by
generated public endpoints.

### Invariant 9: All UPDATE paths must use accepted schema authority

No `UPDATE` path should bypass schema validation, relation checks, index
maintenance, reverse-index maintenance, or uniqueness enforcement.

### Invariant 10: Bulk mutation requires a batch contract

Unbounded bulk `UPDATE` should not become a public API until cursor/token,
bounded receipts, and replay semantics exist.

## Error Message Policy

Errors should distinguish endpoint rejection from SQL shape rejection.

For generated query endpoint:

```text
UPDATE is a mutation statement and is not accepted by the generated query SQL endpoint.
```

For generated DDL endpoint:

```text
UPDATE is a row mutation statement and is not accepted by the generated DDL SQL endpoint.
```

For primary-key proof rejection:

```text
Public primary-key UPDATE requires complete primary-key equality and no extra predicates.
```

For public/generated write surface with unsupported broad `UPDATE`:

```text
Non-primary-key UPDATE requires explicit ORDER BY primary_key and LIMIT on this endpoint.
```

For unbounded public `UPDATE`:

```text
Unbounded UPDATE is not accepted on this endpoint.
```

For implicit fallback rejection:

```text
Public bounded UPDATE must specify ORDER BY primary_key explicitly; implicit
primary-key fallback is only available in session SQL.
```

For invalid field mutation:

```text
UPDATE cannot assign to primary-key, generated, or managed fields.
```

For unsupported `RETURNING`:

```text
UPDATE RETURNING supports only * or plain field lists on this surface.
```

## Open Questions

### Should session/library unbounded UPDATE remain after 1.0?

**Recommendation:** yes, if clearly documented as session/library or admin-level
behavior. Generated public endpoints must not expose it by default.

### Should a generated SQL write endpoint be added?

**Recommendation:** only after Phase 1 policy gating exists. The endpoint should
not call broad `execute_sql_update::<E>` without selecting a stricter policy.

### Should CLI expose broad UPDATE?

**Recommendation:** not through generated canister endpoints by default. The CLI
may expose broad session/admin `UPDATE` only in local/dev/admin contexts where
the authority model is explicit.

### Should public primary-key UPDATE allow extra guard predicates?

**Recommendation:** not in v1. Reject extra non-PK predicates until the proof
helper has a precise at-most-one contract and tests for guard semantics.

### Should public bounded UPDATE allow OFFSET?

**Recommendation:** no. `OFFSET` is fragile for mutation because the result set
changes as rows are updated. Use primary-key cursors or durable tokens instead.

### Should public bounded UPDATE allow implicit primary-key fallback?

**Recommendation:** no. Session/library SQL may keep implicit fallback;
public/generated SQL should require explicit `ORDER BY primary_key`.

### Should public bounded UPDATE allow DESC?

**Recommendation:** reject `DESC` in v1. Admit only canonical primary-key order
with default direction or explicit `ASC`.

### Should public bounded UPDATE allow RETURNING?

**Recommendation:** yes only for the already-supported narrow `*` and field-list
forms, and only when the update shape is otherwise admitted and bounded.

### What should the default public LIMIT cap be?

**Recommendation:** choose after benchmarking. Start conservative. A placeholder
such as `MAX_PUBLIC_UPDATE_LIMIT = 100` is reasonable for design discussion, but
the implementation should validate against canister instruction budgets.

## Recommended Roadmap Summary

| Phase | Name | Public capability | Safety boundary |
| --- | --- | --- | --- |
| 0 | Baseline contract | None new | Document current write lane, generated endpoints, and `RETURNING` |
| 1 | Policy gate | None new | Prevent accidental exposure |
| 2 | PK `UPDATE` | Single-row public-safe `UPDATE` | Complete primary-key equality proof |
| 3 | Bounded `UPDATE` | Multi-row public-safe `UPDATE` | Explicit canonical `ORDER BY pk LIMIT n` |
| 4 | Cursor `UPDATE` | Resumable bulk `UPDATE` | Durable cursor/token and completion receipt |
| 5 | Admin jobs | Migration/backfill `UPDATE` | Explicit admin job model |
| 6 | Bounded `RETURNING` | Formalized result rows | Same target bound as mutation |

## Final Recommendation

IcyDB should not treat `UPDATE` as missing. It should treat `UPDATE` as an
already-working session/library write capability that needs explicit
surface-specific policy before generated/public canister exposure expands.

The next implementation should be narrow:

- document current `execute_sql_update::<E>` behavior and `UPDATE RETURNING`;
- lock generated query and DDL rejection with tests;
- introduce an explicit `UPDATE` policy gate;
- define public-safe write policy as primary-key-only first.

Only after that should IcyDB expose bounded non-primary-key `UPDATE`, and only
in this explicit form:

```sql
UPDATE Entity
SET field = value
WHERE predicate
ORDER BY primary_key
LIMIT n
```

Unbounded public/generated `UPDATE` should remain rejected until IcyDB has
durable cursor/token, bounded receipt, replay, and resumability semantics.
