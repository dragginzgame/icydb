# IcyDB 0.181 - Safe and Bounded SQL UPDATE Roadmap

## Status

Draft.

## Summary

IcyDB already supports SQL `UPDATE` in core/session SQL. The current
implementation is not a parser stub: it parses `UPDATE`, lowers it, selects
target primary keys, builds structural patches, rejects unsafe field writes, and
routes mutation through the accepted-schema structural save/commit pipeline.

The remaining work is not "add `UPDATE`." The remaining work is to make the
`UPDATE` boundary explicit and improve it in stages so that public/generated
canister surfaces do not accidentally inherit the full breadth of internal
session SQL mutation semantics.

This design proposes a staged roadmap:

- Document and freeze the current core/session `UPDATE` contract.
- Introduce explicit `UPDATE` exposure policy gates.
- Expose only primary-key `UPDATE` first on public/generated mutation surfaces.
- Later admit bounded deterministic non-primary-key `UPDATE` with explicit
  `ORDER BY primary_key LIMIT n`.
- Defer true bulk `UPDATE` until cursor/token, receipt, replay, and resumability
  semantics exist.
- Keep unbounded `UPDATE ... WHERE ...` internal/session-only or reject it
  before public `1.0` exposure.

The central safety rule is:

> Public/canister SQL mutation must be stricter than core/session SQL mutation.

## Background

A read-only audit found the following current state.

Core/session SQL supports `UPDATE`.

Known references from the audit:

- Parser admits `UPDATE`:
  - `crates/icydb-core/src/db/sql/parser/statement/mod.rs`
  - `crates/icydb-core/src/db/sql/parser/statement/update.rs`
  - `crates/icydb-core/src/db/sql/parser/model.rs`
- Parser tests cover assignments, aliases, `ORDER BY`, `LIMIT`, `OFFSET`, and
  `RETURNING` syntax:
  - `crates/icydb-core/src/db/sql/parser/tests/mod.rs`
- Lowering requires `WHERE`, rejects expression `ORDER BY`, and appends
  primary-key ordering as fallback:
  - `crates/icydb-core/src/db/sql/lowering/select/mod.rs`
- Execution builds a structural patch, rejects primary-key/generated/managed
  field writes, selects target primary keys, and applies an internal structural
  update batch:
  - `crates/icydb-core/src/db/session/sql/execute/write/update.rs`
  - `crates/icydb-core/src/db/executor/mutation/save/structural.rs`
- Public session facade exposes SQL execution:
  - `crates/icydb/src/db/session/mod.rs`
  - `crates/icydb-core/src/db/session/sql/mod.rs`
- Docs currently list `UPDATE` as admitted mutation SQL:
  - `docs/contracts/SQL_SUBSET.md`
  - `docs/architecture/SQL_SURFACE_MAPPING.md`
- Generated canister SQL endpoints expose query and DDL surfaces, not a public
  row-mutation SQL endpoint:
  - `crates/icydb-build/src/db/sql.rs`
  - `testing/integration/tests/sql_canister.rs`

The perceived absence of `UPDATE` is therefore likely an exposure-layer issue,
not an engine capability issue.

## Problem

The current core/session `UPDATE` implementation is broader than the safest
public canister policy should be.

Core/session SQL currently admits non-primary-key multi-row `UPDATE` shapes such
as:

```sql
UPDATE Character
SET active = false
WHERE age = 21
```

It also supports bounded ordered forms:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10
```

And internally it may use primary-key fallback ordering where no explicit
ordering is supplied.

This is useful inside core/session SQL, but risky as a public generated canister
surface because public mutation endpoints must account for:

- accidental mass updates;
- unbounded target selection;
- target primary-key materialization costs;
- canister instruction limits;
- trap/retry behavior;
- deterministic ordering;
- durable receipts;
- replay behavior;
- resumability;
- stable-memory mutation windows;
- index, reverse-index, relation, and uniqueness maintenance;
- user-facing documentation guarantees.

The dangerous failure mode is not that `UPDATE` is incorrect. The dangerous
failure mode is that a future generated mutation endpoint wires directly into
the existing broad session `UPDATE` path and accidentally exposes unbounded bulk
mutation as a public API.

## Goals

This design has five goals.

1. Preserve existing core/session `UPDATE` behavior unless a concrete
   correctness issue is found.
2. Make the current `UPDATE` boundary explicit in code, tests, and
   documentation.
3. Ensure generated query and DDL endpoints continue to reject row-mutation SQL.
4. Define a safe staged public mutation policy for `UPDATE`.
5. Defer true bulk `UPDATE` until IcyDB has a durable batch/cursor contract with
   bounded receipts and replay semantics.

## Non-Goals

- This design does not propose removing core/session `UPDATE`.
- This design does not propose exposing unrestricted bulk `UPDATE` through
  generated canister endpoints.
- This design does not propose `UPDATE ... JOIN`, subquery `UPDATE`, CTE
  `UPDATE`, computed assignment expressions beyond what is already supported, or
  unlimited `RETURNING`.
- This design does not propose a durable migration/job engine in the first
  slice.
- This design does not require changing the parser grammar unless policy
  enforcement requires better shape inspection.

## Terminology

Core/session SQL means SQL executed through IcyDB's internal or library session
APIs. It is allowed to expose more powerful primitives because callers are
closer to the database authority boundary.

Generated query endpoint means a generated canister endpoint intended for
query/read SQL. It must reject mutation statements.

Generated DDL endpoint means a generated canister endpoint intended for
schema/DDL SQL. It must reject row-mutation statements.

Generated public mutation endpoint means a future generated canister endpoint
intended to execute row mutation SQL. This surface must be stricter than
core/session SQL.

Primary-key `UPDATE` means an `UPDATE` whose `WHERE` clause is statically or
semantically provable to target at most one entity by primary key equality.

Example:

```sql
UPDATE Character
SET age = 22
WHERE id = 1
```

For composite primary keys, this means equality constraints over all primary-key
components in canonical key order.

Bounded deterministic `UPDATE` means an `UPDATE` with a predicate, explicit
primary-key ordering, and an explicit limit.

Example:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10
```

Unbounded `UPDATE` means any `UPDATE` that may affect an arbitrary number of
rows and has no explicit public-surface bound.

Example:

```sql
UPDATE Character
SET active = false
WHERE age = 21
```

## Current UPDATE Boundary

The current implementation appears to have the following behavior.

| Capability | Current core/session status |
| --- | --- |
| Parser admits `UPDATE` | Yes |
| `WHERE` required | Yes |
| Normal stored field assignment | Yes |
| Primary-key field mutation | Rejected |
| Generated field mutation | Rejected |
| Managed/internal field mutation | Rejected |
| Non-primary-key predicates | Yes |
| Multi-row `UPDATE` | Yes |
| `ORDER BY` | Yes |
| Expression `ORDER BY` | Rejected |
| `LIMIT` | Yes |
| `OFFSET` | Yes |
| Implicit primary-key fallback ordering | Yes |
| Uses accepted schema authority | Yes |
| Uses structural mutation path | Yes |
| Index/reverse-index maintenance | Through shared save/commit pipeline |
| Relation checks | Through shared mutation machinery |
| Unique conflict behavior | Atomicity coverage exists |
| Generated query endpoint accepts `UPDATE` | No |
| Generated DDL endpoint accepts `UPDATE` | No |
| Generated public mutation endpoint | Not currently exposed |
| Durable cursor/chunked bulk `UPDATE` | No |

This boundary is acceptable for internal/session use, but it should not be
treated as the public generated canister contract.

## Design Principle

The `UPDATE` roadmap should follow one principle:

> The lower-level engine may support broad mutation, but every higher-level
> exposure surface must explicitly choose which subset it admits.

No public endpoint should admit `UPDATE` merely because the parser and executor
support it.

## Exposure Policy Matrix

Recommended steady-state policy:

| Surface | `UPDATE` policy |
| --- | --- |
| Core/session SQL | Keep existing support |
| Generated query endpoint | Reject all `UPDATE` |
| Generated DDL endpoint | Reject all `UPDATE` |
| Generated public mutation endpoint, first slice | Admit primary-key `UPDATE` only |
| Generated public mutation endpoint, later bounded slice | Admit explicit `ORDER BY primary_key LIMIT n` non-PK `UPDATE` |
| CLI/dev shell | May expose core/session semantics if clearly documented |
| Migration/admin tooling | May support broader staged/batched `UPDATE` under separate safety model |
| Public unbounded `UPDATE` | Reject |

## Accepted and Rejected Shapes

### Core/session SQL

Core/session SQL may continue to support the current broad set:

```sql
UPDATE Character
SET age = 22
WHERE id = 1

UPDATE Character
SET active = false
WHERE age = 21

UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10

UPDATE Character
SET active = false
WHERE age = 21
LIMIT 10
```

The last form is acceptable internally if core lowering supplies deterministic
primary-key fallback ordering.

### Generated public mutation SQL, phase 1

Accept only primary-key `UPDATE`:

```sql
UPDATE Character
SET age = 22
WHERE id = 1
```

Reject non-primary-key `UPDATE`:

```sql
UPDATE Character
SET active = false
WHERE age = 21
```

Reject bounded non-primary-key `UPDATE` in phase 1:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10
```

That shape should wait until the bounded deterministic phase.

### Generated public mutation SQL, bounded phase

Accept:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10
```

Reject:

```sql
UPDATE Character
SET active = false
WHERE age = 21
LIMIT 10
```

Even though core/session SQL may provide implicit primary-key fallback ordering,
public/canister SQL should require the user to say the ordering explicitly.

Reject:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY name
LIMIT 10
```

Public bounded `UPDATE` should order by the canonical primary key, not an
arbitrary field.

Reject:

```sql
UPDATE Character
SET active = false
WHERE age = 21
```

Unbounded public `UPDATE` should remain unavailable.

### Always rejected

These should be rejected across all mutation surfaces unless explicitly
redesigned:

```sql
UPDATE Character
SET id = 2
WHERE id = 1

UPDATE Character
SET generated_field = 'x'
WHERE id = 1

UPDATE Character
SET managed_internal_field = 'x'
WHERE id = 1

UPDATE Character
SET active = false
```

Unsupported `RETURNING` should also fail closed until IcyDB defines a result
contract for `UPDATE` receipts.

## Primary-Key UPDATE Semantics

Primary-key `UPDATE` is the safest first public slice because it targets at most
one row.

A public primary-key `UPDATE` should require:

- exactly one target entity;
- a valid assignment list;
- no writes to primary-key, generated, or managed fields;
- a `WHERE` clause that proves primary-key identity;
- all schema, relation, index, reverse-index, and uniqueness checks already
  enforced by the structural mutation path.

For a single primary key:

```sql
UPDATE Character
SET age = 22
WHERE id = 1
```

For a composite primary key:

```sql
UPDATE Membership
SET role = 'admin'
WHERE org_id = 10 AND user_id = 20
```

For composite keys, public policy should require equality constraints for every
primary-key component.

Optional additional guard predicates may be allowed only if the planner can
still prove the target set is at most one row:

```sql
UPDATE Character
SET age = 22
WHERE id = 1 AND active = true
```

This is still at-most-one because `id = 1` identifies the row.

However, implementation may choose to reject additional predicates in the first
public slice to keep the policy simple.

## Bounded Deterministic UPDATE Semantics

Bounded deterministic `UPDATE` is the second public slice.

The shape should be:

```sql
UPDATE Entity
SET field = value
WHERE predicate
ORDER BY primary_key
LIMIT n
```

The requirements are:

- `WHERE` is required.
- `ORDER BY` is required.
- `ORDER BY` must exactly match the canonical primary-key ordering.
- `LIMIT` is required.
- `LIMIT` must be positive.
- `LIMIT` must be at or below a configured public maximum.
- `OFFSET` should be rejected on public/canister mutation endpoints.
- The result should include at least affected row count.
- The mutation should remain atomic within the selected batch.
- The public endpoint must not silently rely on implicit primary-key fallback
  ordering.

`OFFSET` should be avoided publicly because mutation changes the result set as
it runs. Cursor-based batching is a better long-term model.

## Unbounded UPDATE Policy

Public unbounded `UPDATE` should be rejected.

Rejected:

```sql
UPDATE Character
SET archived = true
WHERE last_seen_at < 1700000000
```

The problem is not only mass mutation. It is that a public canister method has
no durable contract for:

- how many rows were selected;
- how many rows were actually mutated;
- whether the mutation can be resumed;
- whether retry repeats already-mutated rows;
- how instruction limits are handled;
- whether the client can safely page through the remaining target set.

Unbounded `UPDATE` can remain available in core/session SQL if that is
considered an internal/admin-level capability.

## Roadmap

### Phase 0: Baseline Audit and Contract Capture

#### Objective

Capture the current `UPDATE` behavior without changing semantics.

#### Work

Add or update docs that state:

- core/session SQL supports `UPDATE`;
- generated query endpoints reject `UPDATE`;
- generated DDL endpoints reject `UPDATE`;
- no public generated row-mutation SQL endpoint currently exposes broad
  `UPDATE`;
- current core/session `UPDATE` is not the same thing as public canister
  `UPDATE` policy.

#### Tests

Preserve or add tests proving:

```sql
UPDATE Character SET age = 22 WHERE id = 1
```

works in core/session SQL.

```sql
UPDATE Character SET active = false WHERE age = 21
```

works in core/session SQL if this is already supported.

```sql
UPDATE Character SET active = false
```

fails because `WHERE` is missing.

Generated query endpoint rejects:

```sql
UPDATE Character SET age = 22 WHERE id = 1
```

Generated DDL endpoint rejects row mutation SQL.

#### Deliverable

A clearly documented baseline.

#### Exit Criteria

A reader can determine exactly where `UPDATE` works and where it does not.

### Phase 1: Explicit Statement Classification and Policy Gate

#### Objective

Prevent accidental exposure of broad core/session `UPDATE` through future
public endpoints.

#### Work

Introduce or clarify statement classification.

Suggested conceptual categories:

- `SqlSurface::CoreSession`
- `SqlSurface::GeneratedQuery`
- `SqlSurface::GeneratedDdl`
- `SqlSurface::GeneratedMutation`
- `SqlSurface::CliDev`

Suggested conceptual `UPDATE` policy:

- `UpdateExposurePolicy::CoreSession`
- `UpdateExposurePolicy::PublicPrimaryKeyOnly`
- `UpdateExposurePolicy::PublicBoundedDeterministic`
- `UpdateExposurePolicy::InternalBulk`

The exact names should follow IcyDB conventions. The key requirement is that
public/canister mutation must have a separate gate from core/session execution.

#### Policy Helper

Add a helper that can answer:

- Is this statement an `UPDATE`?
- What entity does it target?
- Which fields does it assign?
- Does it mutate primary-key/generated/managed fields?
- Does it have `WHERE`?
- Is the `WHERE` clause primary-key-exact?
- Is it bounded?
- Does it have explicit primary-key `ORDER BY`?
- Does it use `OFFSET`?
- Does it request unsupported `RETURNING`?

This helper should be usable by generated endpoint routing before execution.

#### Tests

Add unit tests around classification.

Accepted by `CoreSession`:

```sql
UPDATE Character SET active = false WHERE age = 21
```

Rejected by `GeneratedQuery`:

```sql
UPDATE Character SET active = false WHERE age = 21
```

Rejected by `GeneratedDdl`:

```sql
UPDATE Character SET active = false WHERE age = 21
```

Accepted by `PublicPrimaryKeyOnly`:

```sql
UPDATE Character SET age = 22 WHERE id = 1
```

Rejected by `PublicPrimaryKeyOnly`:

```sql
UPDATE Character SET active = false WHERE age = 21
```

#### Deliverable

A fail-closed policy gate that must be consciously configured by each SQL
surface.

#### Exit Criteria

There is no route by which a generated public endpoint can inherit full
core/session `UPDATE` semantics by accident.

### Phase 2: Public Primary-Key UPDATE

#### Objective

Expose the smallest useful public generated mutation SQL shape.

#### Accepted Shape

```sql
UPDATE Entity
SET field = value
WHERE primary_key = value
```

#### Rejected Shapes

```sql
UPDATE Entity
SET field = value
WHERE non_pk_field = value

UPDATE Entity
SET field = value
WHERE non_pk_field = value
ORDER BY primary_key
LIMIT 10

UPDATE Entity
SET primary_key = value
WHERE primary_key = value

UPDATE Entity
SET generated_field = value
WHERE primary_key = value

UPDATE Entity
SET managed_field = value
WHERE primary_key = value
```

#### Result Contract

The first public result contract should be minimal:

```text
rows_affected: u64
```

Optionally:

```text
matched: bool
updated: bool
```

Do not introduce broad `RETURNING` in this phase.

#### Safety Properties

This phase is safe because:

- target cardinality is at most one;
- no batching is required;
- no cursor is required;
- no `ORDER BY` is required;
- no instruction-limit surprise should occur beyond normal single-entity
  mutation costs;
- the existing structural mutation pipeline handles validation and commit
  behavior.

#### Tests

Generated mutation endpoint accepts:

```sql
UPDATE Character SET age = 22 WHERE id = 1
```

Generated mutation endpoint rejects:

```sql
UPDATE Character SET active = false WHERE age = 21
```

Generated query endpoint still rejects all `UPDATE`.

Generated DDL endpoint still rejects all `UPDATE`.

#### Deliverable

A public generated mutation SQL endpoint, if introduced, that only admits
primary-key `UPDATE`.

#### Exit Criteria

Users can update one entity by primary key through public generated mutation
SQL, and nothing broader.

### Phase 3: Explicit Bounded Deterministic UPDATE

#### Objective

Admit small multi-row public updates only when deterministic and bounded.

#### Accepted Shape

```sql
UPDATE Entity
SET field = value
WHERE predicate
ORDER BY primary_key
LIMIT n
```

#### Requirements

- `WHERE` is required.
- Explicit `ORDER BY primary_key` is required.
- Explicit `LIMIT n` is required.
- `n` must be positive.
- `n` must not exceed a configured maximum.
- `OFFSET` is rejected.
- Non-primary-key `ORDER BY` is rejected.
- Expression `ORDER BY` is rejected.
- Unsupported `RETURNING` is rejected.
- No implicit primary-key fallback at the public boundary.

#### Limit Policy

Introduce a maximum public `UPDATE` limit.

Example:

```text
MAX_PUBLIC_UPDATE_LIMIT = 100
```

The exact value should be chosen by benchmarking and canister instruction budget
testing. It should be conservative by default and configurable only through
explicit generated-code or schema-level policy.

#### Result Contract

Return:

```text
rows_affected: u64
limit: u64
complete: bool
```

Where `complete` means the executor believes no additional rows matching the
predicate remain after this batch.

If `complete` is expensive or ambiguous, omit it in this phase. Do not
overpromise.

#### Why Explicit Ordering Matters

Core/session lowering may append primary-key fallback ordering. Public policy
should not rely on that.

The public caller should write:

```sql
ORDER BY id
LIMIT 100
```

because this makes the bounded-batch nature obvious in user code, generated
logs, docs, and reviews.

#### Tests

Accepted:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10
```

Rejected:

```sql
UPDATE Character
SET active = false
WHERE age = 21
LIMIT 10
```

Rejected:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY name
LIMIT 10
```

Rejected:

```sql
UPDATE Character
SET active = false
WHERE age = 21
```

Rejected:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
```

Rejected:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 1000000
```

#### Deliverable

A public bounded batch `UPDATE` contract.

#### Exit Criteria

Public multi-row `UPDATE` exists only as an explicit, deterministic, bounded
batch.

### Phase 4: Cursor-Based Batch UPDATE

#### Objective

Support resumable bulk mutation without unbounded public execution.

#### Motivation

Users will eventually need to update many rows. Requiring them to manually
repeat:

```sql
UPDATE Entity
SET field = value
WHERE predicate
ORDER BY id
LIMIT 100
```

can work, but it does not provide a durable contract for continuation, receipts,
retries, or progress reporting.

#### Proposed Shape

Introduce an explicit batch protocol.

Possible API-level shape:

```text
prepare_update(sql) -> update_token
execute_update_batch(update_token, limit) -> update_batch_receipt
```

Or SQL-like shape:

```sql
UPDATE Character
SET archived = true
WHERE last_seen_at < 1700000000
ORDER BY id
LIMIT 100
CONTINUE WITH token
```

The exact syntax is future work. The important point is that cursor-based bulk
`UPDATE` should be a separate design, not an accidental consequence of unbounded
SQL.

#### Cursor Token Contents

A durable update cursor should include or reference:

- target entity;
- normalized predicate fingerprint;
- assignment patch fingerprint;
- canonical primary-key ordering;
- last processed primary key;
- accepted schema version or schema fingerprint;
- batch size policy;
- created timestamp or logical epoch;
- cumulative rows affected;
- completion flag;
- token version;
- authorization context if applicable.

The token should be opaque to clients.

#### Receipt Contents

Each batch should return:

```text
update_token: Option<Token>
rows_matched: u64
rows_updated: u64
last_primary_key: Option<PrimaryKey>
complete: bool
```

Possibly:

```text
batch_id: u64
mutation_fingerprint: Hash
schema_fingerprint: Hash
```

#### Idempotency

The design must define what happens when a client retries a batch call.

Preferred behavior:

- repeated execution with the same completed batch token should not duplicate
  effects;
- each receipt should either commit atomically or report failure without partial
  visible mutation;
- continuation should be based on durable primary-key progress, not offset.

#### Tests

- Cursor resumes after a partial series of batches.
- Cursor rejects use after schema-incompatible changes.
- Cursor rejects changed assignment/predicate fingerprint.
- Cursor remains deterministic across retries.
- Cursor handles rows that no longer match the predicate.
- Cursor handles rows inserted before, during, and after the cursor range
  according to a documented policy.

#### Deliverable

A durable bounded bulk `UPDATE` mechanism.

#### Exit Criteria

IcyDB can safely support large logical updates through repeated bounded mutation
calls.

### Phase 5: Administrative Bulk UPDATE and Migration Jobs

#### Objective

Support administrator or migration-grade bulk `UPDATE` with explicit operational
semantics.

#### Motivation

Some updates are not user-facing API operations. They are schema migrations,
backfills, denormalization repairs, or operational jobs.

These should not be confused with ordinary public SQL mutation.

#### Possible Direction

Introduce a migration/admin job layer:

```text
start_update_job(sql, batch_size) -> job_id
run_update_job_batch(job_id) -> job_receipt
inspect_update_job(job_id) -> job_status
cancel_update_job(job_id)
```

#### Requirements

- explicit admin authority;
- durable job state;
- bounded batch execution;
- resumable progress;
- per-batch receipts;
- final summary;
- schema-version compatibility;
- clear trap/retry semantics;
- observability.

#### Deliverable

A separate admin/migration path for bulk data maintenance.

#### Exit Criteria

Large data rewrites are handled as jobs, not as unbounded public SQL calls.

### Phase 6: Optional RETURNING Support

#### Objective

Add a bounded result contract for `UPDATE` output rows.

#### Current Concern

The parser may admit `RETURNING`, but unsupported executor semantics should fail
closed.

Do not expose broad `RETURNING` until the result contract is clear.

#### Safe First Slice

For primary-key `UPDATE`:

```sql
UPDATE Character
SET age = 22
WHERE id = 1
RETURNING id, age
```

For bounded `UPDATE`:

```sql
UPDATE Character
SET active = false
WHERE age = 21
ORDER BY id
LIMIT 10
RETURNING id
```

#### Requirements

- returned rows must be bounded by the same update limit;
- returned fields must respect projection/visibility rules;
- result ordering must match mutation ordering;
- generated/managed field exposure must follow existing read policy;
- large result materialization must be capped.

#### Deliverable

Bounded `RETURNING`.

#### Exit Criteria

`UPDATE` result sets are useful without creating an unbounded response-size
hazard.

## Safety Invariants

The following invariants should hold across all phases.

### Invariant 1: No mutation without WHERE

This must always fail:

```sql
UPDATE Entity SET x = 1
```

### Invariant 2: No primary-key mutation

This must always fail:

```sql
UPDATE Entity SET id = 2 WHERE id = 1
```

### Invariant 3: No generated-field mutation

This must always fail unless generated-field semantics are redesigned:

```sql
UPDATE Entity SET generated_value = 10 WHERE id = 1
```

### Invariant 4: No managed/internal-field mutation

Managed fields remain owned by the system.

### Invariant 5: Public non-PK UPDATE must be bounded

This must fail publicly:

```sql
UPDATE Entity SET x = 1 WHERE y = 2
```

### Invariant 6: Public bounded UPDATE must be explicitly ordered

This must fail publicly:

```sql
UPDATE Entity SET x = 1 WHERE y = 2 LIMIT 10
```

### Invariant 7: Public bounded UPDATE must order by primary key

This must fail publicly:

```sql
UPDATE Entity SET x = 1 WHERE y = 2 ORDER BY name LIMIT 10
```

### Invariant 8: Core/session behavior must not imply public behavior

A shape accepted by core/session SQL is not automatically accepted by generated
public endpoints.

### Invariant 9: All UPDATE paths must use accepted schema authority

No `UPDATE` path should bypass schema validation, relation checks, index
maintenance, reverse-index maintenance, or uniqueness enforcement.

### Invariant 10: Bulk mutation requires a batch contract

Unbounded bulk `UPDATE` should not become a public API until cursor/token,
bounded receipts, and replay semantics exist.

## Error Message Policy

Errors should distinguish endpoint rejection from SQL shape rejection.

Examples:

For generated query endpoint:

```text
UPDATE is a mutation statement and is not accepted by the generated query SQL endpoint.
```

For generated DDL endpoint:

```text
UPDATE is a row mutation statement and is not accepted by the generated DDL SQL endpoint.
```

For public mutation endpoint with unsupported broad `UPDATE`:

```text
Non-primary-key UPDATE requires explicit ORDER BY primary_key and LIMIT on this endpoint.
```

For unbounded public `UPDATE`:

```text
Unbounded UPDATE is not accepted on this endpoint.
```

For implicit fallback rejection:

```text
Public bounded UPDATE must specify ORDER BY primary_key explicitly; implicit
primary-key fallback is only available in core/session SQL.
```

For invalid field mutation:

```text
UPDATE cannot assign to primary-key, generated, or managed fields.
```

## Open Questions

### Should core/session unbounded UPDATE remain after 1.0?

**Recommendation:** yes, if clearly documented as core/session/admin-level
behavior. However, public generated endpoints must not expose it.

### Should CLI expose broad UPDATE?

**Recommendation:** CLI may expose broad session `UPDATE` in local/dev/admin
contexts, but it must be documented as direct session SQL. If CLI targets
deployed canisters through generated endpoints, it should inherit the public
endpoint policy.

### Should public bounded UPDATE allow OFFSET?

**Recommendation:** no. `OFFSET` is fragile for mutation because the result set
changes as rows are updated. Use primary-key cursors or durable tokens instead.

### Should public bounded UPDATE allow implicit primary-key fallback?

**Recommendation:** no. Core/session may keep implicit fallback; public/canister
SQL should require explicit `ORDER BY primary_key`.

### Should public bounded UPDATE allow RETURNING?

**Recommendation:** not in the first bounded phase. Add bounded `RETURNING` later
with explicit result-size rules.

### What should the default public LIMIT cap be?

**Recommendation:** choose after benchmarking. Start conservative. A placeholder
such as `MAX_PUBLIC_UPDATE_LIMIT = 100` is reasonable for design discussion, but
the implementation should validate against canister instruction budgets.

## Recommended Roadmap Summary

| Phase | Name | Public capability | Safety boundary |
| --- | --- | --- | --- |
| 0 | Baseline contract | None new | Document where `UPDATE` works |
| 1 | Policy gate | None new | Prevent accidental exposure |
| 2 | PK `UPDATE` | Single-row public `UPDATE` | At-most-one target |
| 3 | Bounded `UPDATE` | Multi-row public `UPDATE` | Explicit `ORDER BY pk LIMIT n` |
| 4 | Cursor `UPDATE` | Resumable bulk `UPDATE` | Durable cursor/token |
| 5 | Admin jobs | Migration/backfill `UPDATE` | Explicit admin job model |
| 6 | `RETURNING` | Bounded result rows | Result-size cap |

## Final Recommendation

IcyDB should not treat `UPDATE` as missing. It should treat `UPDATE` as an
already-working core/session capability that needs a staged public exposure
roadmap.

The next implementation should be narrow:

- document current core/session `UPDATE`;
- lock generated query/DDL rejection with tests;
- introduce an explicit `UPDATE` policy gate;
- prepare public mutation SQL to admit only primary-key `UPDATE` first.

Only after that should IcyDB expose bounded non-primary-key `UPDATE`, and only
in this explicit form:

```sql
UPDATE Entity
SET field = value
WHERE predicate
ORDER BY primary_key
LIMIT n
```

Unbounded public `UPDATE` should remain rejected until IcyDB has durable
cursor/token, bounded receipt, replay, and resumability semantics.
