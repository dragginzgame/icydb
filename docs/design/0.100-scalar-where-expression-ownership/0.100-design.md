# 0.100 Scalar WHERE Expression Ownership

## Thesis

`0.100` should move scalar `WHERE` onto the same planner-owned boolean
expression model already used elsewhere, instead of requiring every scalar
filter to collapse fully into the older `Predicate` model before planning and
execution.

> **0.100 makes scalar `WHERE` expression-first, splitting one scalar filter
> into two explicit contracts:**
>
> * **one planner-owned boolean expression that preserves full scalar filter
>   semantics**
> * **one derived predicate-style pushdown contract used only where access
>   planning and residual predicate fast paths can exploit it**

This is the architectural slice that makes scalar `WHERE` feel like the rest
of the expression system.

It is also the slice that turns `Predicate` from the universal scalar filter
contract into a narrower derived IR.

---

## Problem statement

After `0.99`, the engine already has:

* shared planner-owned expression lowering
* compiled scalar projection expression execution
* grouped `HAVING` expression ownership
* aggregate `FILTER` expression ownership
* bounded prepared execution and symbolic-slot template groundwork

So the expression model itself is no longer the weak point.

However, scalar `WHERE` still sits on an older contract:

* SQL `WHERE` lowers to planner `Expr`
* then SQL `WHERE` validation narrows that expression to the subset that can be
  compiled back into `Predicate`
* then scalar planning, access selection, residual filtering, explain,
  fingerprinting, and runtime filtering all still consume `Predicate`

That creates three practical problems:

* normal scalar SQL can still fail on unsurprising boolean-expression shapes
  such as arithmetic compares even though projection and grouped expression
  execution already support the same subexpressions
* scalar filter semantics are artificially constrained by the pushdown IR
  instead of the other way around
* the current `PredicateExecutionModel = Predicate` alias prevents a clean
  split between:
  * semantic filter ownership
  * access extraction ownership
  * residual runtime filtering ownership

So the main issue is no longer "missing parser support."

The issue is that scalar `WHERE` still treats the older predicate model as the
universal execution contract.

---

## Design goal

Make scalar `WHERE` expression-first so that:

* already-supported scalar boolean expressions can remain intact through scalar
  planning
* access planning can extract only the pushdown-safe predicate subset
* residual scalar filtering can evaluate the remaining boolean expression per
  row
* `Predicate` stops being the mandatory front-door scalar filter model

The goal is:

> **one scalar filter keeps its full boolean expression semantics, while a
> narrower predicate IR is derived only for pushdown and fast-path planning.**

---

## Non-goals

`0.100` does **not** aim to:

* add JOIN
* add subqueries
* add window functions
* widen grouped `GROUP BY` expression admission
* redesign grouped `HAVING` ownership, which already sits on planner `Expr`
* remove the public `Predicate` type from the repository in one step
* widen the fixed-route prepared-template lane to arbitrary scalar boolean
  expression filters
* collapse all index selection, predicate execution, and filter semantics into
  one new generic IR

This slice is about **scalar filter ownership**, not general SQL surface
completion and not a planner-wide IR rewrite.

---

## Explicit current boundary

Today, scalar filtering still looks like this:

```text
SQL WHERE / fluent filter
  -> planner Expr
  -> validate to predicate-compilable subset
  -> compile back into Predicate
  -> access planning / runtime filtering consume Predicate
```

That means scalar `WHERE` currently inherits the limitations of the predicate
compiler even when:

* parsing is fine
* planner expression lowering is fine
* runtime scalar expression evaluation is already available elsewhere

Examples:

* `SELECT (level + 1) * 2 AS score ...` can work through scalar projection
  expression evaluation
* `WHERE (level + 1) * 2 >= 10` fails because scalar `WHERE` does not yet own a
  residual expression filter lane

This document exists to change that boundary.

---

## Contracts

`0.100` must make these contracts explicit:

* scalar filter semantics are owned by planner boolean `Expr`, not by
  `Predicate`
* access planning may use only a derived pushdown-safe subset
* scalar runtime filtering may evaluate a residual boolean expression after
  access
* pushed-down predicate extraction must never change the truth semantics of the
  original scalar filter
* `Predicate` becomes one derived IR for pushdown/extraction/runtime fast paths,
  not the universal scalar filter contract

---

## Target model

### Before (`0.99`)

```text
SQL WHERE / fluent filter
  -> planner Expr
  -> collapse to Predicate
  -> access planning uses Predicate
  -> residual filtering uses PredicateProgram
```

### After (`0.100`)

```text
SQL WHERE / fluent filter
  -> planner Expr
  -> validate full scalar boolean expression
  -> extract pushdown Predicate subset when possible
  -> keep residual boolean Expr when needed
  -> access planning uses extracted Predicate subset
  -> residual filtering evaluates compiled boolean expression
```

### Important consequence

The scalar filter contract becomes a split model:

* **semantic scalar filter**: boolean `Expr`
* **pushdown filter**: extracted `Predicate`

That is the central design change.

---

## Why this is the right size

This is a whole-minor-version architectural slice because it crosses:

* query intent
* logical planning
* access planning
* runtime filtering
* explain / cache-key / fingerprint ownership
* SQL and fluent entrypoints

But it is still bounded:

* it does not reopen grouped expression architecture
* it does not require JOIN or subquery work
* it does not require planner-wide parameter redesign
* it does not require deleting the public `Predicate` type immediately

This makes `0.100` the right release line to do the scalar filter split before
`0.101` continues broader SQL surface completion.

---

## Normative shipped scope

`0.100` ships **exactly** these architectural changes:

### 1. Scalar filter planning becomes expression-first

Scalar queries carry one planner-owned boolean expression filter instead of
requiring full early collapse into `Predicate`.

This applies to:

* reduced SQL `WHERE`
* fluent `FilterExpr`
* internal scalar filter seams that already lower through planner `Expr`

### 2. Pushdown extraction becomes explicit and derived

Planning must derive one pushdown-safe `Predicate` subset from the scalar
boolean expression when possible.

This extracted predicate is used for:

* access-path selection
* filtered index implication checks
* exact/range/index-prefix pushdown where supported
* retained-slot residual predicate fast paths that still benefit from compiled
  predicate programs

### 3. Residual scalar filtering gains a real expression path

When the scalar filter cannot be fully represented as a pushdown predicate, the
plan keeps the remaining boolean expression for residual execution.

This residual expression path is the contract that makes ordinary scalar SQL
`WHERE` widening possible without lying to access planning.

### 4. The `PredicateExecutionModel = Predicate` alias is removed

The alias must stop existing once scalar planning can distinguish:

* semantic scalar filter expression
* derived predicate execution model

The new contracts do **not** need to use these names, but the alias itself
must be removed so the split remains explicit.

### 5. `Predicate` remains as a narrower IR

`Predicate` survives `0.100`, but in a reduced role:

* pushdown extraction IR
* predicate runtime fast path IR
* index predicate compilation source
* explain / cache / fingerprint shape for the extracted pushdown subset where
  relevant

It is no longer the universal scalar filter ownership boundary.

---

## Clause boundary

`0.100` changes **scalar `WHERE` ownership only**.

It does **not** reopen other clause boundaries:

| Clause | `0.100` status | Notes |
| --- | --- | --- |
| scalar `WHERE` | Changed | Moves to expression-first ownership with derived pushdown extraction |
| grouped `HAVING` | Unchanged | Already expression-owned; do not redesign in this slice |
| projection expressions | Unchanged | Already expression-owned |
| `ORDER BY` expressions | Unchanged | Already expression-owned |
| `GROUP BY` | Unchanged | Remains intentionally narrow |
| write-statement `WHERE` | Narrowly aligned | May reuse the same scalar expression split if the same scalar query path is shared, but broad write-surface widening is not the point of the slice |

---

## Admitted scalar `WHERE` expression family

`0.100` does **not** promise "all possible SQL boolean expressions."

It promises that scalar `WHERE` can now keep the already-supported planner
boolean expression family intact instead of forcing full early collapse into
`Predicate`.

That means:

* compare expressions over admitted scalar operands
* boolean `AND` / `OR` / `NOT`
* searched `CASE` in scalar boolean position
* admitted scalar function wrappers in boolean position
* arithmetic and other already-supported scalar expression subtrees where the
  planner boolean expression model and scalar-expression runtime already support
  them

Still out of scope:

* aggregates in scalar `WHERE`
* grouped-only semantics in scalar `WHERE`
* subqueries
* window expressions
* planner-unowned parser special cases

---

## Pushdown extraction contract

Pushdown extraction must be explicit and conservative.

### Extraction rule

The extractor may derive `Predicate` fragments only when the fragment is:

* semantically equivalent to the original boolean subexpression
* representable in the existing predicate model
* safe for access planning / residual predicate runtime use

### Non-rule

The extractor must **not** widen access planning by guessing.

If a subexpression is not pushdown-safe, it stays in the residual boolean
expression path.

### Example

For:

```sql
WHERE (level + 1) * 2 >= 10 AND class_name = 'Wizard'
```

the clean target shape is:

* pushdown predicate:
  * `class_name = 'Wizard'`
* residual expression:
  * `(level + 1) * 2 >= 10`

If some arithmetic form can be algebraically rewritten safely into a direct
predicate, that is an optimization, not the defining contract.

---

## Residual execution contract

Scalar execution must gain one explicit residual boolean-expression filter path.

That path must:

* compile the residual boolean `Expr` onto the same scalar expression runtime
  used by the admitted expression family
* evaluate on the row view visible after access planning and retained-slot
  materialization
* preserve SQL three-valued boolean semantics consistently with the planner
  boolean model

The important point is:

> `0.100` should make residual scalar filtering expression-aware, not just
> predicate-aware.

---

## Fluent and SQL alignment

`0.100` must align both front doors.

### SQL

Reduced SQL `WHERE` must stop validating only the predicate-compilable subset.
It must validate the admitted scalar boolean expression family.

### Fluent

`FilterExpr` already lowers to planner boolean `Expr`, so it should now feed
the same scalar expression-first filter path instead of immediately compiling
back into `Predicate`.

### Compatibility

`filter_predicate(...)` may remain as a compatibility seam, but after `0.100`
it should route through the split model rather than defining the entire scalar
filter architecture.

---

## Prepared-execution boundary

`0.100` must not silently widen the fixed-route prepared-template lane.

Prepared status for this slice:

| Family | `0.100` status | Notes |
| --- | --- | --- |
| existing compare-family fixed-route prepared templates | Unchanged | `0.99` symbolic-slot lane remains |
| scalar `WHERE` expressions that still reduce fully to the existing compare-family template shapes | Unchanged | Keep existing template ownership |
| new general scalar residual expression filters | Fallback prepared only | No promise of fixed-route template support in `0.100` |

This preserves the existing prepared-template boundary while scalar `WHERE`
ownership is being upgraded.

## Risks

The main risks are architectural drift and semantic mismatch:

* residual boolean expression evaluation can diverge from extracted predicate
  semantics
* explain output can become ambiguous if it does not distinguish:
  * semantic scalar filter
  * extracted pushdown predicate
  * residual execution filter
* cache-key / fingerprint ownership can drift if one layer keys on the semantic
  expression while another keys on only the extracted predicate
* access planning can accidentally become unsound if extraction admits
  non-equivalent rewrites
* prepared execution can widen accidentally if general scalar filter
  expressions leak into the fixed-route template lane

---

## Verification matrix

`0.100` should be verified across these boundaries:

| Area | Required proof |
| --- | --- |
| SQL lowering | admitted scalar `WHERE` expression shapes lower successfully |
| Fluent parity | SQL and fluent expression filters build equivalent scalar semantics |
| Pushdown extraction | extracted predicate subset is explicit and does not exceed safe pushdown shapes |
| Residual execution | residual boolean expressions execute correctly on scalar row paths |
| Explain | output distinguishes semantic filter vs extracted pushdown vs residual execution when relevant |
| Cache / fingerprint | semantically different scalar residual expressions do not alias incorrectly |
| Prepared boundary | general scalar expression filters do not silently enter the fixed-route prepared template lane |
| Negative coverage | grouped-only, aggregate, subquery, and other out-of-scope scalar `WHERE` shapes still fail closed |

---

## Acceptance criteria

`0.100` is done when:

* scalar `WHERE` no longer requires full early collapse into `Predicate`
* the repository no longer defines `PredicateExecutionModel = Predicate`
* scalar planning can carry both:
  * semantic scalar boolean expression ownership
  * derived pushdown predicate ownership
* scalar execution can evaluate residual boolean expressions after access
* SQL shapes such as arithmetic compare expressions in scalar `WHERE` work
  through the general residual-expression path instead of ad-hoc rewrite-only
  exceptions
* explain and planning surfaces show the split clearly enough that maintainers
  can audit pushdown vs residual ownership

---

## Summary

`0.100` is the scalar filter architecture slice.

It does not try to finish all remaining SQL surface work. It makes scalar
`WHERE` stop depending on the older predicate model as the universal contract,
recasts `Predicate` as one narrower derived IR, and puts scalar SQL filtering
on the same expression-owned trajectory already established elsewhere in the
engine.
