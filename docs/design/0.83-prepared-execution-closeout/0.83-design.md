**0.83 Prepared Execution Closeout**

## Thesis

The architecture already has the right major reusable boundary:

* `CompiledQuery`
* `PreparedExecutionPlan`
* `SharedPreparedExecutionPlan`
* shared structural query-plan/prepared-plan cache
* SQL-local raw-SQL compile cache

The remaining optimization opportunity is not the absence of a reusable execution artifact. It is that **some executor-owned runtime prep is still rebuilt on repeated execution**, especially on scalar SQL paths.

0.83 should therefore **tighten the existing prepared execution contract** so that semantics-free, repeatable prep is attached to or absorbed into the existing prepared plan boundary.

---

## Problem statement

The remaining fixed-cost floor is increasingly concentrated in **residual execution prep**, not just in parse/lower/validate/plan.

A concrete example is SQL projection runtime still preparing executor-owned projection/runtime shape on demand before execution/materialization, rather than consuming a fully frozen reusable artifact.

That means the current system already has the correct *kind* of architecture, but the prepared boundary is still slightly too soft in a few hot seams.

So the question for 0.83 is:

> What execution prep is still rebuilt every time, even though it is deterministic from the already-prepared plan and safe to reuse?

---

## Scope

### In scope

* repeated scalar SQL execution hot paths
* executor-owned prep that is:

  * deterministic from the prepared/shared plan
  * semantics-free
  * safe to reuse across repeated executions
* tightening `PreparedExecutionPlan` / `SharedPreparedExecutionPlan` or adding a narrowly attached reusable adjunct
* measurement against named repeated-run scenarios

### Out of scope

* replacing existing artifact hierarchy with a new one
* changing cache identity model
* making raw SQL text the main shared cache key
* broad grouped/scalar runtime unification
* new SQL features
* canister/bootstrap reshaping

---

# Architectural position

## Canonical reusable execution contract remains:

**`SharedPreparedExecutionPlan`**

That should remain the execution-facing reusable authority.

If something new is introduced, it should be one of these only:

### Option A: extend the existing plan

Attach more frozen prep directly into:

* `PreparedExecutionPlan`
* `SharedPreparedExecutionPlan`

### Option B: attach a narrow adjunct

If ownership hygiene requires it, add something like:

```rust
pub struct PreparedExecutionRuntime {
    pub projection_runtime: PreparedProjectionRuntime,
    // future narrowly scoped reusable runtime prep
}
```

But this must be:

* subordinate to prepared execution
* not a new peer “program” abstraction
* not a third cache identity tier

---

# Cache model

## Keep existing cache split

### SQL-local cache

Raw SQL keyed caching remains:

* conservative
* surface-local
* useful for avoiding repeated SQL-specific front-end work

### Shared cross-surface cache

Structural identity remains the main reusable cross-surface contract:

* anchored on the existing structural/shared plan cache
* not raw SQL text

So 0.83 should say explicitly:

> This slice does not introduce a new cache identity model. It increases the reusable value stored behind the existing prepared/shared structural boundary.

---

# Primary target seam

The strongest evidence-based seam is:

* SQL projection runtime still builds executor-owned projection shape on demand
* repeated executions pay this prep again
* this is narrow, measurable, and plausibly reusable

So the first 0.83 target should be:

## Freeze reusable projection runtime prep for repeated scalar SQL execution

### Desired state

Repeated execution of the same already-prepared scalar query should:

* not rebuild projection runtime shape
* not reinterpret projection materialization shape from plan on each execution
* instead consume a frozen reusable projection-runtime artifact derived once at prepare time

---

# Design shape

## 1. Inventory residual execution prep

First enumerate all execution-time prep that still occurs after prepared/shared plan retrieval.

Classify each seam:

### A. Reusable and semantics-free

Move into prepared boundary.

### B. Execution-context dependent

Keep runtime-local.

### C. Mixed

Split so only reusable subpart moves.

This avoids umbrella claims and keeps the slice evidence-driven.

---

## 2. Default implementation stance

Default to extending the existing prepared boundary.

The preferred order is:

### A. First choice

Attach additional frozen reusable prep directly to:

* `PreparedExecutionPlan`
* `SharedPreparedExecutionPlan`

### B. Only if ownership hygiene requires it

Introduce a narrow subordinate adjunct such as:

```rust
pub struct PreparedScalarExecutionRuntime {
    pub projection: PreparedProjectionRuntime,
}
```

Any such adjunct must remain:

* subordinate to prepared execution
* built from the prepared/shared plan
* non-authoritative for cache identity
* a storage location for frozen runtime prep only

The artifact should contain only data that is:

* deterministic from prepared/shared plan
* independent of one specific execution invocation
* not tied to cursor state, page state, or live result buffers

For `0.83`, projection/runtime prep is the first resident.

---

## 3. Phase 1 implementation target

Phase 1 is explicit: freeze reusable projection runtime prep for repeated scalar SQL execution.

Repeated execution of the same already-prepared scalar SQL query should:

* not rebuild projection runtime shape
* not reinterpret projection materialization shape from plan on each execution
* instead consume one frozen reusable projection-runtime artifact derived once at prepare time

This is the first implementation target because it is:

* already known to rebuild on repeated scalar SQL execution
* narrow enough to measure cleanly
* semantics-free once the prepared plan is fixed

No second seam should be added to `0.83` until the first seam is implemented and measured.

---

## 4. Shift construction to prepare/cache fill path

The artifact should be built:

* when producing the prepared/shared execution plan
* once per structural identity entry
* then reused by repeated execution

Concretely, the intended build point is the existing shared prepared-plan
construction/cache-fill path, not SQL runtime entrypoints:

* build during prepared/shared plan creation on cache miss
* store behind the existing structural shared cache boundary
* reuse on subsequent executions that recover the same prepared/shared entry

Not:

* rebuilt inside SQL runtime entrypoints on every run

---

## 5. Change execution to consume, not prepare

Execution path goal:

Before:

* load prepared plan
* derive projection runtime shape again
* materialize

After:

* load prepared plan plus frozen reusable projection/runtime prep
* execute directly

That is the real fixed-cost reduction story.

---

# Grouped/scalar stance

Do **not** state:

> grouped + scalar share evaluation core

as a 0.83 commitment.

That is too broad for the evidence. Grouped carries additional concerns:

* cursor/continuation
* pagination
* output shaping
* grouped-specific execution contracts

So the correct wording is:

> 0.83 is scoped to repeated scalar SQL execution. Grouped execution is explicitly deferred except where a narrowly isolated duplicate-prep seam is independently identified and proven reusable.

That keeps the slice honest.

---

# Acceptance criteria

## Functional

* repeated scalar SQL execution no longer rebuilds identified reusable projection/runtime prep on each run
* no new peer execution artifact is introduced above or beside `SharedPreparedExecutionPlan`
* no new cache identity tier is introduced
* structural shared caching remains the cross-surface reuse mechanism
* raw SQL caching remains SQL-local and conservative

## Performance

Named repeated-run scenarios show measurable reduction in fixed execution cost, especially on:

* simple indexed scalar reads
* simple projection queries
* explainable repeated SQL runs that previously rebuilt projection runtime prep

The verification set should name the concrete scenarios up front and keep them
stable for the slice. At minimum:

* one simple indexed scalar SQL projection query
* one simple covering or hybrid projection query
* one repeated `EXPLAIN EXECUTION` or equivalent repeated SQL path that exercises the same prepared projection/runtime seam

For those scenarios, verification should prove both:

* total repeated-run cost decreases
* repeated execution no longer rebuilds the identified projection/runtime prep on the hot path

## Boundary hygiene

* moved prep is semantics-free and deterministic
* execution-context-dependent state remains runtime-local
* grouped work is either deferred or explicitly isolated to a proven seam

---
# Verification

Verification should cover three things:

## 1. Reuse behavior

* repeated scalar SQL executions recover the same prepared/shared entry
* no new cache identity tier appears in the design or implementation
* SQL-local raw-SQL caching remains local and conservative

## 2. Hot-path behavior

* repeated execution consumes frozen projection/runtime prep
* repeated execution does not rebuild plan-derived projection runtime shape on the targeted path

## 3. Semantics

* output rows, labels, cursor behavior, and projection semantics remain unchanged
* grouped behavior is unchanged unless one isolated grouped seam is explicitly admitted and proven

---

# Thesis sentence

> **0.83 should make `SharedPreparedExecutionPlan` more complete, not replace it.**
