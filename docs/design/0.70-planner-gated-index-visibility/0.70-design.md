## Title

**0.70 — Planner-Gated Index Visibility**

---

## Status

In Progress

---

## Context

The earlier `0.69` and `0.70` work proved that the read engine benefits from
making index-backed execution paths more explicit.

It also exposed a recurring problem:

* the system kept reasoning about index correctness inside execution
* different paths carried their own correctness checks or fallback rules
* the resulting model was harder to explain than it needed to be

The design decision for the next `0.70` phase is now fixed:

* index correctness is enforced by planner visibility
* execution trusts planned indexes

This collapses the older "authority" line into one simpler rule:

* visible index == authoritative index

---

## Index Visibility Invariant (New)

An index must not be visible to the planner unless it is:

* fully built
* fully populated
* fully correct

Visible index == authoritative index.

### Consequences

* no partially-built index may participate in planning or execution
* execution paths must not perform correctness checks for index validity
* planner visibility gating is the single enforcement point

This is the new anchor for the whole `0.70` line.

---

## Authority Model (Revised)

Index correctness is enforced at planning time via visibility gating, not at
execution time.

That means:

* planner decides whether an index is visible
* execution assumes a visible index is correct and complete
* execution does not perform runtime correctness resolution for index-backed
  reads

The word "authority" is therefore no longer a separate execution concept in
this design.

It is just planner visibility.

---

## Index Lifecycle And Visibility

The design-level lifecycle contract is:

```rust
enum IndexState {
    Building,
    Ready,
    Dropping,
}
```

The runtime code now uses `Ready` as the lifecycle name for:

* planner-visible
* complete
* trustworthy

### Rules

* only `Ready` indexes are visible to the planner
* `Building` indexes are not visible
* `Dropping` indexes are not visible
* `Building -> Ready` must be atomic

That transition may happen only after:

* full backfill
* full correctness validation, if validation is required for that index type

The planner must never see a partially ready index.

---

## Planner Responsibilities (Updated)

The planner must:

* filter indexes by `state == Ready`
* treat every visible index as authoritative
* choose fallback paths when no visible index is suitable
* keep index lifecycle correctness out of execution-time branching

The planner must not:

* consider `Building` indexes
* consider `Dropping` indexes
* defer correctness decisions to execution
* introduce runtime checks for whether an index is complete enough to trust

If a path depends on an index being correct, the planner must enforce that by
visibility, not by pushing a correctness question downstream.

---

## Aggregate Existing-Row Shortcuts (Revised)

`COUNT` and `EXISTS` existing-row shortcuts now follow the same visibility
rule.

They:

* rely on planner visibility only
* do not require a dedicated runtime correctness model
* are correct when planned against a visible index

Fallback paths are selected by the planner, not by execution-time checks.

This applies to the existing-row shortcut family only:

* `CountExistingRows { direction, covering }`
* `ExistsExistingRows { direction }`

It does not mean every aggregate optimization problem is solved.

It means index correctness for those shortcuts is no longer a runtime concern.

Questions such as:

* whether a shortcut is semantically valid for a given aggregate
* whether a grouped or extrema path should optimize at all

remain planner and strategy questions, not execution-time visibility checks.

---

## Execution Model (Simplified)

Execution assumes:

* every planned index access is correct
* every planned index access is complete

Execution must not:

* inspect index lifecycle state for correctness gating
* perform correctness validation for planned index-backed reads
* branch based on whether an index is trustworthy

Execution may still branch for ordinary strategy mechanics:

* covering versus non-covering work
* materialized versus fused execution
* aggregate strategy selection

But it must not branch on index correctness.

---

## Explain And Debug Surfaces

`EXPLAIN` should describe planner visibility choices, not execution-time
correctness resolution.

That means the long-term explain story should answer questions like:

* which visible index was chosen?
* which fallback path was chosen when no visible index qualified?
* whether an index-backed shortcut was admitted by planner-visible readiness

It should not depend on execution rebuilding a separate correctness decision.

The earlier execution-side validity labels were removed with the Option A
cleanup.

---

## Removed Transitional Designs

The earlier `0.70` exploration included executor-owned correctness mechanisms
such as:

* runtime snapshots
* runtime resolvers
* execution-time correctness classification for index-backed reads

Those notes are now superseded by the planner-gated visibility invariant.

They may still be useful as historical records of the discarded approach, but
they are no longer the target design for `0.70`.

---

## Non-Goals (Current Design)

The system does not currently support:

* partially visible indexes
* progressive index rollout
* execution-time correctness resolution for index-backed reads

If future requirements ever introduce partially-built index visibility, that
would require a separate design.

It should not be smuggled back in under the current model.

---

## Current Stop Point

The intended `0.70` direction is now simple:

* indexes are either invisible or fully usable
* planner enforces that boundary
* execution trusts planned index accesses

This is a stricter and easier-to-audit model than the earlier execution-owned
correctness line.

It also gives later work a cleaner constraint:

* optimize planner admission and execution strategy freely
* but keep index correctness out of execution

---

## Summary

The revised `0.70` design is:

* planner-gated index visibility
* visible index == authoritative index
* no execution-time correctness resolution for index-backed reads

That is the new design truth the rest of the `0.70` line should follow.
