# 0.91 CASE Expressions

## Thesis

`0.91` should introduce `CASE` expressions into the shared scalar expression model so that conditional query shapes like:

```sql
SELECT
  class_name,
  SUM(CASE WHEN strength > 10 THEN 1 ELSE 0 END)
FROM character
GROUP BY class_name
ORDER BY SUM(CASE WHEN strength > 10 THEN 1 ELSE 0 END) DESC
LIMIT 10;
```

work through the existing scalar evaluator and clause-phase rules, without introducing new execution paths, new aggregate semantics, or a second conditional-expression subsystem.

The core mechanism is:

> compile `CASE` as a normal scalar expression node, evaluate conditions in order, and return the selected branch value under the same phase and typing rules that already govern other scalar expressions.

This slice should increase SQL expressiveness while staying strictly inside the existing single-entity execution model.

---

## Problem statement

After `0.85–0.90`, the engine has:

* shared scalar expression evaluation
* grouped projection widening
* grouped `HAVING`
* bounded aggregate `ORDER BY`
* aggregate input expressions
* a much clearer SQL surface contract

But one large usability gap remains: **conditional scalar expressions**.

Users naturally expect query shapes like:

```sql
SUM(CASE WHEN strength > 10 THEN 1 ELSE 0 END)
AVG(CASE WHEN class_name = 'mage' THEN strength ELSE NULL END)
CASE WHEN strength > agility THEN strength ELSE agility END
ORDER BY CASE WHEN class_name = 'mage' THEN 0 ELSE 1 END, name
```

Without `CASE`, users are forced into:

* missing-feature dead ends
* awkward rewrites
* inability to express common conditional aggregation patterns

This is now one of the most visible remaining SQL surface gaps that can still be closed **without changing the execution model**.

---

## Design goal

Add `CASE` as part of the already-admitted scalar expression family.

That means:

* planner-owned `CASE` expression nodes
* executor-prepared compiled scalar programs
* no grouped-only `CASE`
* no aggregate-only `CASE`
* no special runtime beyond the shared scalar evaluator

The goal is to introduce **conditional scalar selection**, not to introduce SQL control flow as a new subsystem.

---

## Non-goals

This slice does not aim to:

* add a new execution phase
* add aggregate `FILTER (...)`
* add window semantics
* add subqueries inside `CASE`
* add a second type-inference or coercion system
* widen unsupported function families
* introduce `CASE`-specific runtime evaluation outside the shared scalar evaluator

This is a scalar-expression widening slice, not a semantics reset.

---

## Supported CASE form

`0.91` should support **searched CASE** only.

Supported form:

```sql
CASE
  WHEN <condition_1> THEN <expr_1>
  WHEN <condition_2> THEN <expr_2>
  ...
  ELSE <expr_n>
END
```

Example:

```sql
SELECT
  name,
  CASE
    WHEN strength > agility THEN strength
    ELSE agility
  END
FROM character;
```

### Out of scope for 0.91

Do not add simple `CASE` in the same slice:

```sql
CASE class_name
  WHEN 'mage' THEN 1
  ELSE 0
END
```

That can be added later as parser/lowering sugar over searched `CASE` if it is still wanted.

Keeping `0.91` to searched `CASE` avoids unnecessary surface branching.

---

## Query families enabled by this slice

### Scalar projection

```sql
SELECT
  name,
  CASE
    WHEN strength > agility THEN strength
    ELSE agility
  END AS max_stat
FROM character;
```

### Aggregate inputs

```sql
SELECT
  AVG(CASE WHEN class_name = 'mage' THEN strength ELSE NULL END)
FROM character;
```

```sql
SELECT
  class_name,
  SUM(CASE WHEN strength > 10 THEN 1 ELSE 0 END)
FROM character
GROUP BY class_name;
```

### HAVING

```sql
SELECT
  class_name,
  SUM(CASE WHEN strength > 10 THEN 1 ELSE 0 END) AS strong_count
FROM character
GROUP BY class_name
HAVING strong_count > 5;
```

### ORDER BY

```sql
SELECT
  name,
  class_name
FROM character
ORDER BY
  CASE WHEN class_name = 'mage' THEN 0 ELSE 1 END,
  name;
```

### Grouped aggregate ordering

```sql
SELECT
  class_name,
  SUM(CASE WHEN strength > 10 THEN 1 ELSE 0 END) AS strong_count
FROM character
GROUP BY class_name
ORDER BY strong_count DESC
LIMIT 10;
```

## Expression admissibility

`CASE` belongs to the existing scalar expression family and must obey the same clause/phase rules as every other scalar expression.

### Allowed condition leaves

Conditions may use only the already-admitted scalar boolean expression family for the current phase, including:

* fields
* literals
* comparisons
* boolean composition
* admitted scalar functions
* nested admitted scalar expressions

### Allowed result expressions

Each `THEN` and `ELSE` branch may contain only the already-admitted scalar expression family for the current phase.

### Rejected inside CASE

* subqueries
* window expressions
* unsupported scalar function families
* cross-phase references
* aggregate expressions in phases where aggregates are not admitted
* grouped/post-aggregate expressions in pre-aggregate phases

This keeps `CASE` inside the existing expression boundary instead of turning it into a widening loophole.

---

## Phase behavior

`CASE` is admitted only where scalar expressions are already admitted for that phase.

### Pre-aggregate / per-row phases

Allowed in:

* `WHERE`
* aggregate input expressions
* scalar projection on non-grouped queries
* scalar `ORDER BY` on non-grouped queries

In these contexts, `CASE` may only depend on:

* row fields
* literals
* admitted scalar functions
* nested admitted scalar expressions

### Post-aggregate / grouped-row phases

Allowed in:

* grouped projection
* grouped `HAVING`
* grouped `ORDER BY`

In these contexts, `CASE` may only depend on:

* grouped keys
* aggregate outputs
* admitted post-aggregate scalar expression forms

### Key rule

`CASE` does not relax phase boundaries.

It only composes expressions that were already legal in that clause and phase.

---

## Semantics

Evaluation rule:

```text
evaluate WHEN conditions in order
→ first condition that is true selects its THEN expression
→ if no WHEN condition is true, evaluate ELSE expression
→ if ELSE is absent, return NULL
```

### Important notes

* conditions are evaluated in source order
* the first true branch wins
* missing `ELSE` behaves as `ELSE NULL`
* `CASE` returns one scalar value

This should be the only semantic model used by planner, explain, tests, and executor.

---

## Type behavior

`CASE` must not introduce a new type-coercion subsystem.

### Rule

The result type of `CASE` is derived from the set of `THEN` and `ELSE` branch result expressions using the same scalar type-compatibility/coercion rules already used elsewhere in the engine.

That means:

* if branch result types already have an admitted common type, use it
* if they do not, reject the expression
* do not invent `CASE`-specific coercion rules

Examples:

```sql
CASE WHEN strength > 10 THEN 1 ELSE 0 END
```

valid: both numeric

```sql
CASE WHEN strength > 10 THEN name ELSE 0 END
```

rejected unless the existing scalar type system already admits a shared type

### Condition typing

Each `WHEN` condition must type-check as a boolean expression under the current scalar typing rules.

---

## Null behavior

`CASE` must reuse existing scalar null semantics.

That means:

* `WHEN` conditions evaluate under the engine’s existing boolean/null rules
* branch result nulls propagate as normal scalar values
* omitted `ELSE` is equivalent to `NULL`

Do not add `CASE`-specific null semantics in this slice.

---

## Error behavior

Expression evaluation or typing failures inside `CASE` should follow the same ownership model as other scalar expressions.

### Ownership

* parser rejects invalid `CASE` syntax
* lowering / planning rejects invalid phase usage, unsupported nested constructs, and type incompatibilities
* runtime should evaluate only already-admitted compiled `CASE` expressions

### Examples

Reject clearly for:

* `CASE` containing subqueries
* `CASE` branch expressions crossing phase boundaries
* non-boolean `WHEN` conditions
* incompatible branch result types
* unsupported function families inside conditions or branches

No generic late runtime “invalid query” fallback.

---

## Planner model

### Add CASE as a scalar expression node

The logical/planner-owned scalar expression model should gain a `CASE` form.

Conceptually:

```rust
enum ScalarExpr {
    ...
    Case {
        when_then_arms: Vec<(ScalarExpr, ScalarExpr)>,
        else_expr: Option<Box<ScalarExpr>>,
    },
}
```

The important ownership rule is unchanged:

* logical planning stores planner-owned expression structure
* executor preparation compiles that structure into executor-owned IR
* logical planning must not store executor-owned compiled IR directly

### Validation

Planner validation must enforce:

* all `WHEN` conditions are boolean-typed
* all `THEN` / `ELSE` branches belong to the admitted expression family for the clause/phase
* all branch result types are compatible under existing scalar typing rules
* no forbidden nested expression families appear inside `CASE`

---

## Executor model

### Compile once, evaluate through shared scalar evaluator

`CASE` should compile into the same shared scalar evaluator path already used for other scalar expressions.

Execution model:

```text
compile CASE once per prepared execution
→ evaluate conditions in order at runtime
→ evaluate selected branch result
→ return scalar Value
```

Important boundary rules:

* do not introduce a dedicated `CASE` evaluator subsystem
* do not introduce grouped-only `CASE` runtime
* do not duplicate branch logic separately for aggregate inputs / projection / HAVING / ORDER BY

`CASE` is just another compiled scalar expression node.

---

## Interaction with existing slices

### Projection (`0.85`)

`CASE` composes directly with scalar and grouped projection.

Examples:

```sql
SELECT
  class_name,
  CASE WHEN AVG(strength) > 10 THEN 'strong' ELSE 'normal' END
FROM character
GROUP BY class_name;
```

This is valid only if the post-aggregate projection phase already admits the referenced aggregate output shape through the existing grouped expression model.

### HAVING (`0.86`)

`CASE` may be used in grouped `HAVING` so long as its inner expressions obey post-aggregate phase rules.

### Aggregate ORDER BY (`0.88`)

`CASE` composes with grouped `ORDER BY` only under the same grouped ordering admission rules already defined by `0.88`.

For example:

```sql
ORDER BY
  CASE WHEN AVG(strength) > 10 THEN 0 ELSE 1 END,
  class_name
LIMIT 10
```

should run only through the bounded grouped Top-K path when required.

### Aggregate input expressions (`0.89`)

This is the most important composition point.

`0.89` allows aggregates to consume admitted scalar expressions. `0.91` makes `CASE` one of those scalar expressions.

Example:

```sql
SUM(CASE WHEN strength > 10 THEN 1 ELSE 0 END)
```

This must behave as:

```text
per row
→ evaluate CASE using shared scalar evaluator
→ feed resulting Value into SUM accumulator
```

No special aggregate-CASE semantics should be added.

### SQL surface closure (`0.90`)

`0.91` is an intentional, explicit extension of the closed scalar expression family defined by `0.90`.

It should update the 0.90 contract rather than bypass it.

---

## Explain / fingerprint / cache follow-through

This slice changes logical query shape and must update all relevant identity and observability surfaces.

`0.91` must update:

* explain output for `CASE` expression structure
* structural fingerprint hashing for `CASE`
* cache identity for `CASE` expression shape

This is required so that:

* identical `CASE` structures produce identical fingerprints
* semantically different `CASE` structures produce different fingerprints
* explain reflects actual planner-owned conditional structure

### Normalization note

If the engine later introduces scalar normalization or simplification for `CASE`, fingerprint/cache behavior must reflect that canonicalization consistently.

For `0.91`, it is acceptable to preserve source-structure identity if no broader scalar canonicalization is already in place.

---

## Guardrails

### 1. No new execution path

`CASE` must reuse the shared scalar evaluator.

### 2. No phase widening

`CASE` must not allow expressions that were previously illegal in that clause/phase.

### 3. No aggregate semantic widening

`CASE` does not add:

* aggregate FILTER
* nested aggregates
* grouped-only aggregate semantics

### 4. No second type system

`CASE` must reuse existing scalar type compatibility/coercion rules.

### 5. Fail closed on unsupported constructs

Unsupported `CASE` shapes must be rejected explicitly during planning.

---

## Acceptance criteria

### Functional

* searched `CASE` works in scalar projection
* searched `CASE` works in aggregate input expressions
* searched `CASE` works in grouped projection / `HAVING` / `ORDER BY` where phase-admitted
* omitted `ELSE` behaves as `NULL`
* conditions are evaluated in order
* first true branch wins

### Architectural

* planner stores planner-owned `CASE` expression structure
* executor compiles `CASE` through the shared scalar evaluator path
* no duplicate `CASE` runtime path is introduced
* clause/phase validation remains centralized
* explain, fingerprint, and cache identity reflect `CASE` structure

### Error-surface

* invalid `CASE` syntax is parser-rejected
* non-boolean `WHEN` conditions are planner-rejected
* incompatible branch result types are planner-rejected
* unsupported nested constructs are planner-rejected
* runtime is not discovering unsupported SQL-family usage late

### Performance

* `CASE` compilation happens once per prepared execution
* runtime cost is limited to ordered condition evaluation plus selected-branch evaluation
* no grouped-only or aggregate-only conditional runtime layer is introduced

---

## Out of scope

Explicitly not part of `0.91`:

* simple `CASE value WHEN ...`
* aggregate `FILTER (...)`
* subqueries inside `CASE`
* window expressions inside `CASE`
* joins, CTEs, set operations
* `CASE`-specific coercion rules
* `CASE` as a loophole for unsupported scalar function families

---

## Follow-on

Potential later follow-on slices after `0.91`:

* simple `CASE` as sugar over searched `CASE`
* aggregate `FILTER (...)`
* broader scalar normalization / simplification
* additional scalar function families

None of those should be coupled into the initial `CASE` slice.

---

## Thesis sentence

> **0.91 introduces searched CASE expressions into the shared scalar expression model, enabling conditional scalar logic across admitted query phases while reusing the existing evaluator, type system, and phase boundaries without introducing new execution paths or aggregate semantics.**
