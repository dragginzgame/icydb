# 0.139 Aggregate Semantic Sealing

## Status

Implemented

## Motivation

`0.138` introduced a clean SQL global aggregate lowering pipeline:

```text
SqlGlobalAggregateTerminal
  → PreparedSqlScalarAggregateStrategy
  → StructuralAggregateTerminal
```

`0.138.x` (including target sealing) removed invalid internal state by introducing:

```rust
AggregateTarget::{Rows, Field, Expr}
```

However, one structural gap remained:

> Aggregate **semantic meaning** (especially DISTINCT behavior) was still represented indirectly through flags and partial normalization across layers.

Examples of the problem:

* `distinct: bool` existed independently of aggregate kind
* `MIN(DISTINCT x)` and `MAX(DISTINCT x)` were representable and normalized outside a single semantic authority
* DISTINCT handling leaked into `terminal.rs` and partially into lowering

This meant:

* semantic normalization was not owned by one layer
* some semantic rules were enforced by convention instead of types
* future drift risk existed if new aggregate forms were introduced

## Goal

Introduce a **sealed semantic representation of SQL global aggregates** such that:

* invalid or redundant aggregate combinations are unrepresentable
* DISTINCT behavior is encoded in the type system
* terminal becomes purely syntactic
* strategy becomes purely execution-bound
* SQL global semantic key construction occurs exactly once

## Non-Goals

Do not change:

* SQL aggregate results or behavior
* projection ordering or remap semantics
* HAVING behavior
* executor terminal kinds
* grouped aggregate semantics
* public APIs or persisted formats

This is an internal representation and ownership correction only.

---

## Design

### 1. Use One Semantic Aggregate Key

`0.139` keeps planner `AggregateIdentity` as the lower-level semantic key and
adds SQL-local wrappers in `aggregate/semantics.rs` for the global aggregate
lane:

```rust
struct AggregateTerminalSemantics {
    semantic_identity: AggregateIdentity,
    filter_expr: Option<Expr>,
}

enum PreparedAggregateSemantics {
    Count {
        target: PreparedAggregateTarget,
        distinct: bool,
    },
    Sum {
        target: PreparedAggregateTarget,
        distinct: bool,
    },
    Avg {
        target: PreparedAggregateTarget,
        distinct: bool,
    },
    Min {
        target: PreparedAggregateTarget,
    },
    Max {
        target: PreparedAggregateTarget,
    },
}
```

### Properties

* `MIN` / `MAX` cannot carry `distinct`
* `COUNT` / `SUM` / `AVG` explicitly encode `distinct`
* invalid combinations are structurally impossible
* aggregate meaning is fully captured before strategy construction
* filter-aware terminal dedup uses `AggregateTerminalSemantics`
* model-bound execution consumes `PreparedAggregateSemantics`

---

### 2. Centralize SQL Semantic Construction

Build semantic aggregate keys in one place:

```rust
impl AggregateTerminalSemantics {
    fn from_terminal(terminal: &SqlGlobalAggregateTerminal) -> Self
    fn from_owned_terminal(terminal: SqlGlobalAggregateTerminal) -> Self
}
```

Responsibilities:

* convert raw SQL aggregate terminal facts into `AggregateIdentity`
* delegate observable DISTINCT normalization to `AggregateIdentity`:

  * drop DISTINCT for `MIN` / `MAX`
  * preserve for `COUNT` / `SUM` / `AVG`

* map normalized aggregate identity into SQL-local prepared semantic variants
* preserve `AggregateTarget`
* preserve filter expression separately (unchanged)

After this step:

```text
terminal = raw syntax
semantics = canonical meaning
```

---

### 3. Refactor Strategy to Consume Semantics

Replace:

```rust
PreparedSqlScalarAggregateStrategy {
    target: AggregateTarget,
    distinct_input: bool,
    descriptor_shape: ...
}
```

with:

```rust
PreparedSqlScalarAggregateStrategy {
    semantics: PreparedAggregateSemantics,
    filter_expr: Option<Expr>,
    descriptor_shape: ...
}
```

Strategy responsibilities become:

* map semantic variant → descriptor family
* construct executor terminal
* expose projection metadata

Strategy no longer interprets raw DISTINCT flags.

---

### 4. Make Terminal Purely Syntactic

Remove semantic normalization from:

```text
terminal.rs
```

Specifically:

* remove DISTINCT erasure or preservation logic
* retain raw DISTINCT flag exactly as parsed/lowered

Terminal now owns only:

* aggregate kind
* input shape (rows / field / expr)
* raw DISTINCT
* filter expression

---

### 5. Use Semantic Identity for Deduplication

Global aggregate projection dedup must operate on **semantic identity**, not raw terminal shape.

Use the semantic terminal key:

```rust
AggregateTerminalSemantics::from_terminal(&terminal)
```

This ensures:

* `MIN(x)` and `MIN(DISTINCT x)` dedupe
* `COUNT(x)` and `COUNT(DISTINCT x)` do not
* filter expressions remain part of identity

Projection remap logic switches from terminal equality → semantic identity equality.

---

## Ownership Rules

### `terminal.rs`

Owns:

* raw SQL aggregate facts
* no normalization
* no semantic interpretation

### `semantics.rs`

Owns:

* SQL global aggregate semantic wrappers
* terminal → `AggregateIdentity` conversion
* prepared semantic type sealing

Does not own:

* lower-level DISTINCT normalization rules, which remain owned by
  `AggregateIdentity`

### `strategy.rs`

Owns:

* execution mapping
* descriptor selection
* executor terminal construction

### `projection/`

Owns:

* terminal collection
* semantic dedup/remap

### `lowering/`

Owns:

* SQL → aggregate terminal lowering
* admission rules (e.g. DISTINCT + FILTER rejection)

## Expected Outcome

After this slice:

* terminal is purely syntactic
* `aggregate/semantics.rs` is the SQL global wrapper over the shared aggregate
  identity core
* strategy is purely execution-bound
* DISTINCT semantics cannot drift across layers
* invalid aggregate combinations are unrepresentable

Final pipeline:

```text
Terminal (syntax)
  → Semantics (meaning)
  → Strategy (execution)
  → Executor
```

---

## Scope Clarification

`aggregate/semantics.rs` is the **single SQL global owner for terminal
semantic wrapping and prepared aggregate type sealing**.

It is not the single source of truth for:

* shared aggregate identity and DISTINCT normalization, which are owned by
  `AggregateIdentity`
* grouped aggregate semantics
* planner-level aggregate policy
* runtime reducer behavior (e.g. NULL handling)

Those remain owned by their respective layers.

---

## Slice 2 Follow-Through

The remaining field-target compatibility path has been removed:

* field-target aggregate lowering now preserves raw `DISTINCT` for `MIN` and
  `MAX`
* `AggregateTerminalSemantics` and `PreparedAggregateSemantics` remain the only
  SQL global aggregate wrappers that expose no runtime `MIN` / `MAX` DISTINCT
* lowering is now syntax-to-aggregate-expression only, with no SQL global
  aggregate meaning rewrite
