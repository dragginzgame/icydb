# 0.115 Expression Pipeline Contraction

## Thesis

`0.115` should contract the expression pipeline so semantic ownership is fully
centralized in planner-owned expression typing and canonicalization, and all
adjacent layers consume that single authority.

This is not a feature-expansion slice.

It is not a prepared SQL slice.

It is not a grouped semantics slice.

It is a cross-cutting contraction slice that:

* reduces branch pressure in the expression pipeline hotspot cluster
* removes remaining semantic-adjacent duplication outside the prepared path
* ensures future expression work lands in one place instead of multiple layers

`0.114` removes prepared fallback as a semantic owner.

`0.115` should finish the broader contraction so expression meaning is owned in
exactly one place across the system.

## Problem

The system currently has one conceptual semantic authority, but multiple
adjacent modules still perform:

* normalization
* coercion
* interpretation
* structural rewriting

That leads to:

* branch duplication
* hotspot clustering
* increased blast radius per slice
* risk of future drift between expression families

The current shape still reads like:

* parser/lowering -> partial normalization
* predicate layer -> boolean canonicalization
* planner -> type inference + AST
* prepared -> already collapsed in `0.114`
* executor -> evaluation contracts

Even when correct, that creates multiple places where expression meaning is
still “touched”.

---

## Goal

Make planner-owned expression typing plus canonicalization the only semantic
authority for expression meaning.

All other layers must:

* adapt inputs into planner-owned forms
* consume planner-owned outputs

They must not:

* reinterpret expression semantics
* perform independent canonicalization
* encode family-specific meaning locally

---

## Core Invariant

Expression meaning is defined exactly once, in planner-owned expression typing
and canonicalization.

Everything else is:

* translation before planner
* consumption after planner

If a semantic rule exists both in planner-owned expression logic and in an
adjacent layer, planner-owned logic must be the single authority.

Structural adaptation is allowed.

Independent semantic reinterpretation is not.

---

## Ownership Boundary

`0.115` should make the owner split explicit in code shape.

### Parser / Lowering Owns

* syntactic structure
* SQL-specific surface forms
* construction of raw expression trees

Parser / lowering must not:

* canonicalize meaning
* perform semantic simplification beyond trivial normalization

### Predicate Layer Owns

* structural adaptation of predicate forms

Predicate must not:

* own independent boolean semantics
* diverge from planner canonicalization

### Planner Owns

* expression AST (`Expr`)
* type inference
* family classification
* canonicalization
* semantic equivalence

This is the single semantic authority.

### Executor Owns

* evaluation of already-canonical expressions

Executor must not:

* reinterpret expression structure
* reclassify expression families

### Session / Prepared Owns

* binding
* substitution
* execution wiring

This should already be enforced by `0.114`.

## Concrete Actions

### 1. Identify duplicate semantic operations

Audit for:

* boolean normalization in predicate vs planner
* type-family inference outside planner
* expression rewrites outside planner
* `CASE` / compare / function-family logic outside planner

Classify each operation as:

* planner-owned truth
* structural adaptation
* mixed logic that must be split

Planner-owned truth moves to planner.

Structural adaptation stays where it is.

Mixed logic must be separated.

### 2. Collapse canonicalization into planner

Ensure there is:

* one canonicalization path
* one type inference path
* one family classification system

Reduce or remove:

* parallel boolean canonicalization paths
* duplicated simplification logic

### 3. Simplify the predicate layer

Refactor [bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs:1)
and related modules so they reuse planner canonicalization instead of acting as
semi-independent semantic owners.

Predicate should become:

* adapter
* structural helper

Not a semantic authority.

### 4. Reduce lowering semantic work

In SQL lowering:

* keep parsing and structural mapping
* remove deeper semantic decisions
* defer meaning to planner

### 5. Reduce branch pressure in hotspots

Target:

* `expr::ast`
* `type_inference`
* `bool_expr`

Goals:

* fewer parallel match trees
* fewer duplicated family checks
* more centralized logic

### 6. Add invariant tests

Tests should prove:

1. all canonical forms originate from planner-owned expression logic
2. predicate and lowering do not diverge from planner canonicalization
3. expression equivalence is consistent across surfaces
4. expression-related slices do not need to re-open semantic owners outside the
   planner boundary for routine follow-through

---

## Recommended Work Order

### 1. Map expression-related semantic operations across layers

Start with an ownership inventory of expression operations in:

* planner expression modules
* predicate normalization modules
* expression-related lowering modules

### 2. Move semantic truth to planner

Where adjacent layers still own meaning, move that truth upward into
planner-owned expression logic.

### 3. Reduce predicate / lowering duplication

After the planner boundary is clearer, simplify predicate and lowering code so
they adapt and consume rather than decide.

### 4. Simplify the hotspots

Use the ownership cleanup to reduce the branch-heavy match trees in the main
expression cluster.

### 5. Freeze the boundary with tests

Treat the invariant tests as the contract that prevents the old multi-owner
shape from growing back.

## Exit Condition

`0.115` is done when:

* planner is the only semantic authority for expressions
* predicate and lowering layers no longer duplicate canonicalization
* branch pressure in the expression hotspot cluster decreases
* expression-related changes touch fewer modules than the pre-slice baseline
* semantic equivalence is enforced consistently across surfaces

This should be visible as:

* lower hotspot pressure in complexity accretion
* fewer remaining semantic-adjacent duplication families in DRY
* tighter expression-family ownership in CSA
* lower median blast radius for expression-related slices in velocity

The explicit velocity target for follow-through slices is:

* median blast radius `<= 8`

## Summary

`0.115` completes the contraction arc started in `0.114`.

Where `0.114` removes prepared fallback as a semantic owner, `0.115` should
ensure that expression semantics are owned in exactly one place across the
entire system.

That reduces:

* complexity
* duplication
* drift risk
* velocity tax

And it sets up the system for:

* smaller slices
* clearer ownership
* more predictable feature delivery
