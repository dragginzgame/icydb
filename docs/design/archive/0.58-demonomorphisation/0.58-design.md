## Closeout

### Status Summary

The `0.58` size-reduction effort is complete and meets the bar for release.

The major size reduction source was structural de-monomorphization that is
already landed in this line.

The remaining size is now dominated by typed execution bodies, not by removable
orchestration or adapter duplication.

### Key Conclusions

- structural de-monomorphization is sufficiently complete
- planner and orchestration layers have been flattened where that work paid
- redundant generic instantiations at the system boundary have been eliminated
- no further high-confidence local size wins remain

The important nuance is practical, not theoretical:

- this does not claim total size exhaustion
- it does claim that practical, safe, local optimizations are exhausted for
  `0.58`

The residual size is now structural, not incidental.

Dominant remaining contributors are:

- execution kernels
- typed `<E>`-driven specialization

These are architectural issues, not cleanup-level issues.

### What Was Attempted and Resolved

The `0.58` audit line already explored and resolved the remaining obvious local
size candidates.

Measured wins were kept.

Measured non-wins were reverted.

This includes:

- additional wasm-level reductions that did not pay
- further adapter and orchestration deduplication that did not reduce release
  wasm
- experimental reductions that only moved code without collapsing it

### Release Decision

Verdict: `GO` for `0.58`.

Rationale:

- the release meets the size goal directionally and practically
- the remaining size requires architectural change, not incremental tuning
- the risk-adjusted return on further local optimization work is negative

### Transition Note

`0.58` closes the local optimization phase.

`0.59` is not a continuation of planner and runtime cleanup. It opens the
execution-kernel redesign phase.

### Practical Implication for `0.58`

`0.58` is a continuation and hardening line, not a first-pass redesign.

The remaining gap is to complete the hard cut so runtime identity and hot-path
execution are consistently non-generic and `EntityTag`-driven everywhere in
scope, without fallback islands.

### Structural Plan Pivot

The next hard cut after the `EntityTag` and runtime-loop work is the plan model
itself.

`0.58` now treats the planner's existing `AccessPlan<Value>` representation as
the canonical plan identity. The remaining typed `AccessPlan<K>` /
`AccessPlannedQuery<K>` executor handoff is no longer authoritative.

This means:

- `AccessPlannedQuery` becomes structural and stores `AccessPlan<Value>`
- `ContinuationContract` becomes structural and stores `AccessPlan<Value>`
- `ExecutablePlan<E>` becomes the boundary that owns:
  - one structural plan
  - one typed access sidecar derived once from `E`

The sidecar is execution-only support. It must not become a second plan model.
It may contain typed key conversion or typed traversal helpers, but it must not
carry planner semantics, predicate semantics, or route policy.

## Intent

`0.58` is a hard-cut runtime simplification line.

The primary objective is to remove runtime generic specialization pressure and
string-based entity identity from execution-heavy paths.

This line replaces runtime `EntityName` and runtime `E: EntityKind` execution
coupling with one compact runtime identity:

- `EntityTag(u64)`

This is an intentional breaking line with no compatibility shims.

## Hard Rules

1. No backward compatibility layer.
2. No dual identity representation in runtime flows.
3. No runtime hashing for entity identity.
4. Runtime identity must be constant, copyable, and O(1) comparable.
5. Runtime scan/execution paths must not depend on `EntityName`.
6. Runtime scan/execution paths must not be generic over `E: EntityKind`.
7. No function reachable from runtime modules may be generic over
   `EntityKind`.
8. Runtime scan loops must not branch on `EntityTag` per row; entity-specific
   behavior must be resolved before row iteration.
9. Planner/API boundaries must not pass generic closures or monomorphized
   functions into runtime execution paths.
10. `EntityTag` is authoritative storage identity and must not be derived from
    `EntityName`.
11. Changing an existing `EntityTag` value is storage-breaking; adding new tags
    is safe; renaming is safe only when tag values remain unchanged.
12. Runtime behavior dispatch must resolve once per operation, not per row.
13. Index/cursor ordering and partition semantics must be `EntityTag`-based, not
    `EntityName` lexical behavior.
14. Runtime/planner code must not re-derive identity from names, schemas, or
    descriptors; `EntityTag` is the only execution identity.

## Runtime Identity Contract

Introduce:

```rust
#[derive(Clone, Copy, Debug, Eq, PartialEq, Hash, Ord, PartialOrd)]
pub struct EntityTag(pub u64);
```

Extend `EntityKind`:

```rust
pub trait EntityKind: EntitySchema + EntityPlacement + Kind + TypeKind {
    const ENTITY_TAG: EntityTag;
}
```

Every entity must declare one fixed compile-time tag constant.

## EntityTag Assignment Contract

### Authoritative Identity Model

- `EntityTag` is assigned, persistent identity, not derived from entity name.
- entity name is a mutable label.
- `EntityTag` is canonical storage identity.
- final runtime/planner system must not derive identity from names.

### Assignment Rules (Revised)

1. tags are assigned via one global registry (codegen or manifest owned).
2. assignment is deterministic and reproducible via that registry.
3. tags are emitted as compile-time constants in entity impls.
4. a build validation step must:
   - ensure all entities have a registered tag
   - verify global uniqueness
   - fail on collision or missing entries
5. tag values must remain stable across builds, machines, and compiler
   versions.
6. renaming an entity must not change its `EntityTag`.
7. tag values must be allocated monotonically and must never be reused.
8. exactly one registry source of truth must exist at build time; multiple
   sources or layered generation is prohibited.
9. initial registry bootstrap must be deterministic and independent of source
   file discovery order and macro expansion order.

### Tag Mutation Semantics

- changing an existing tag value is storage-breaking.
- adding new tags is safe.
- renaming entities is safe if the tag remains unchanged.
- once persisted, tag values are part of on-disk format and require explicit
  migration tooling when changed.

### Explicitly Prohibited

- tag derivation from entity name (hash or otherwise)
- any assignment that changes when an entity is renamed or moved
- implicit tag generation based on macro expansion or file ordering
- runtime tag generation
- runtime identity re-derivation from names or schema descriptors

## Entity Identity Registry

`EntityTag` assignment is governed by one authoritative registry.

Properties:

- global across the workspace
- stable across builds
- independent of module structure and naming
- source of truth for all entity identity
- centralized and shared by all crates contributing runtime entities
- no crate-local independent tag assignment
- registry entries are stored in deterministic stable order to preserve
  reproducibility and minimize diff churn

Allowed implementations:

- generated Rust module (preferred for `0.58`)
- JSON/manifest consumed at build time
- `build.rs` generated constants

Example:

```rust
pub mod entity_tags {
    pub const USER: EntityTag = EntityTag(1);
    pub const ORDER: EntityTag = EntityTag(2);
}
```

Registry invariants:

- one tag per entity
- no duplicate tags
- no missing entities
- tag values must never be reassigned
- tag values must be monotonically allocated and never reused
- all runtime-visible entities participating in storage or execution are
  registered; unregistered entities must not reach runtime execution paths

### Tag Allocation Strategy (`0.58`)

- selected strategy: monotonic increasing integer tags in registry order of
  assignment (for example `1`, `2`, `3`, ...)
- allocation must use the next unused value
- historical values must never be reused, even after entity removal
- historical values must never be reused, including schema resets or test
  environments within the same codebase

### Registry Mutation Rules

- adding a new entity:
  - assign a new unused `EntityTag`
  - must not reuse historical values
- removing an entity:
  - retire tag; do not reuse
- renaming an entity:
  - update registry key only
  - tag remains unchanged
- modifying an existing tag value:
  - prohibited (storage-breaking)

Rename behavior:

- entity rename updates registry key only
- tag value remains unchanged
- persisted data remains valid

## Debug Mapping Contract

A reverse mapping must exist:

```rust
fn entity_tag_name(tag: EntityTag) -> &'static str;
```

Used for:

- `EXPLAIN`
- logs
- debugging corrupted ranges

This mapping should be generated from the identity registry.

Debug mapping requirements:

- must cover all registered tags
- must be generated from registry source of truth
- must not rely on runtime reflection

## Scope

In-scope runtime domains for de-monomorphization:

- `db/index/*`
- `db/cursor/*`
- `db/executor/*`
- `db/access/*`

Identity replacement target:

- remove runtime `EntityName` usage from key/runtime execution flow
- move runtime dispatch identity to `EntityTag`

Out of scope:

- compatibility migration adapters
- preserving old persisted `EntityName` key wire format
- incremental dual-mode fallbacks

## DataKey Cut

`DataKey` runtime identity must move from:

```rust
entity: EntityName
```

to:

```rust
entity: EntityTag
```

Generic constructors such as `DataKey::from_key::<E>` / `try_new::<E>` should
be removed from runtime-facing paths and replaced by explicit identity-bearing
constructors:

```rust
DataKey::new(entity_tag, storage_key)
```

`EntityTag` key encoding contract:

- fixed-width `u64` encoding
- big-endian byte order
- byte-order/width stability across versions

## Persistence Format Rule (Hard Cut)

`0.58` selects one explicit persistence contract:

- Option A (selected): persist `EntityTag` only.
- pre-`0.58` persisted `EntityName`-keyed data is invalid for this line and
  requires reset/rebuild.

Option B is explicitly rejected for this line:

- no mixed-format decode support
- no dual write/read identity format
- no compatibility branch in runtime decode paths

Forward-compatibility note:

- future versions should support tag-stable migrations without requiring full
  data reset.
- any tag-value change is an on-disk format change and must ship with explicit
  migration tooling.

## De-Monomorphization Boundary Rule

Runtime flow functions in scan/index/cursor/executor/access must not carry
`<E: EntityKind>` parameters.

Typed entity boundaries remain only at outer API/planner/decode adapters.

Boundary conversion pattern:

- outer typed layer supplies `E::ENTITY_TAG`
- runtime layer executes on `EntityTag + raw key/value primitives`
- final typed decode/materialization happens at adapter boundary

Typed decode placement rule (strict):

- typed decode must happen after scan completion or at batch boundary
- typed decode must not occur per row inside scan loops

Disallowed pattern:

```rust
for row in scan_rows(...) {
    let entity = decode::<E>(row)?;
    // ...
}
```

Required pattern:

```rust
let raw_rows = scan_rows(...)?;
let entities = decode_batch::<E>(raw_rows)?;
```

## Dispatch Model Revision (Post Slice C Pivot)

Original goal: eliminate monomorphization via descriptor + erased fn pointers.

Revised goal: eliminate monomorphization via dynamic dispatch (`dyn Trait`).

Rationale:

- primary objective is WASM size reduction, not maximal hot-path optimization
- trait objects remove monomorphization with stronger safety guarantees
- this reduces unsafe surface area and maintenance complexity

Updated invariant:

- runtime contains no `<E: EntityKind>`
- runtime may use `dyn Trait` for behavior dispatch
- identity must still not be derived from names or schemas
- execution dispatch no longer requires a descriptor + erased-fn-pointer model

## Runtime Behavior Dispatch Contract

`EntityTag` carries identity; it must not drive per-row branch trees in hot
loops.

Dispatch rule:

- resolve entity runtime behavior once per operation/query
- store resolved behavior in descriptor/table/trait-object form
- run scan loops against resolved operations without tag branching
- descriptor resolution must not allocate or perform map lookups inside scan
  loops
- descriptor lookup must be O(1) via static table/direct indexable structure,
  not runtime hash maps
- descriptor tables or trait-object registries should be directly indexed by
  `EntityTag` (or one dense tag-to-index mapping) without indirection chains

One acceptable boundary shape:

```rust
trait EntityRuntime {
    fn key_codec(&self, bytes: &[u8]) -> Result<KeyView, InternalError>;
    fn value_codec(&self, bytes: &[u8]) -> Result<Value, InternalError>;
}
```

```rust
let runtime = runtime_registry.resolve(entity_tag)?;
scan_rows_with(runtime, ...);
```

Descriptor implementation constraint:

- dynamic dispatch objects should reference shared monomorphic implementations
  where possible
- per-entity duplication is allowed only when behavior is materially different
- resolved dispatch objects are immutable and reusable for the full execution
  of a query/operation

## Planner/Runtime Handoff Contract

Planner/API layers must pass runtime data, not generic behavior.

Allowed runtime handoff payloads:

- `EntityTag`
- raw execution descriptors/enums
- `dyn Trait` dispatch handles

Disallowed runtime handoff payloads:

- generic closures capturing `E`
- typed callbacks that invoke `decode::<E>` in runtime loops
- monomorphized helper trait adapters crossing into runtime entrypoints

## Index/Cursor Identity Semantics Contract

Replacing `EntityName` with `EntityTag` is not only a field swap; it is also an
ordering and partition contract.

Required invariants:

- key partitioning and namespace separation are defined by `EntityTag`
- `EntityTag` must be encoded as the leading component of runtime key framing
- range bounds and prefix scans must not rely on string lexical ordering
- continuation anchors must remain stable under `EntityTag` key framing
- continuation/cursor identity envelopes must include `EntityTag` to prevent
  cross-entity resume ambiguity

## Runtime Guardrails (CI Hard Fail)

In runtime-heavy modules:

- `db/index/*`
- `db/cursor/*`
- `db/executor/*`
- `db/access/*`

any usage of `EntityKind` or `EntityName` must fail CI.

Additional CI/review guardrails:

- any function reachable from runtime modules must not be generic over
  `EntityKind`, regardless of module location
- planner/API layers must not pass generic closures/typed decode closures into
  runtime execution entrypoints
- runtime scan loops must not branch on `EntityTag` per row
- index/cursor code must not assume lexical `EntityName` ordering semantics
- runtime/planner code must not derive or infer identity from names, schemas,
  or descriptors

Enforcement mechanism:

- add one CI invariant script (grep-based guard is acceptable)
- add a coarse transitive-generic guard in runtime and runtime-adjacent helper
  trees for `EntityKind` generic signatures
- add a targeted index/cursor invariant test set for range/prefix/continuation
  behavior under `EntityTag` identity
- add one invariant check/test for no identity re-derivation from names
- fail CI when disallowed runtime-type identity dependencies are detected
- run this guard in the standard checks pipeline

Baseline CI guard commands:

```bash
rg -n "EntityKind|EntityName" crates/icydb-core/src/db/{index,cursor,executor,access} && exit 1
rg -n "<\\s*[A-Za-z_][A-Za-z0-9_]*\\s*:\\s*EntityKind|where\\s+[A-Za-z_][A-Za-z0-9_]*\\s*:\\s*EntityKind|impl\\s*<\\s*[A-Za-z_][A-Za-z0-9_]*\\s*:\\s*EntityKind" crates/icydb-core/src/db/{index,cursor,executor,access} && exit 1
```

## Acceptance Criteria

1. Runtime paths in `db/index`, `db/cursor`, `db/executor`, `db/access` no
   longer use `EntityName` identity.
2. Runtime paths in those modules no longer expose `<E: EntityKind>` in hot
   execution signatures.
3. Entity identity in runtime flow is `EntityTag` only.
4. All entities have compile-time tag constants and uniqueness checks.
5. Persistence uses `EntityTag` only with no mixed-format decode path.
6. CI fails on any `EntityKind`/`EntityName` usage in targeted runtime modules.
7. Workspace build/tests pass under the new contract.
8. Wasm analysis shows reduced monomorphized duplication in runtime paths.
9. Runtime scan loops perform no per-row `EntityTag` branching.
10. Runtime-executable call graph contains no `EntityKind`-generic helper
    reachability from runtime modules.
11. Planner/runtime handoff uses data descriptors or monomorphic function
    pointers only; no generic closures cross the boundary.
12. Index/cursor range-bound, prefix-scan, and continuation-anchor semantics
    are verified to be `EntityTag`-partition-correct.
13. One centralized workspace identity registry owns all `EntityTag`
    assignments.
14. Registry allocation is monotonic and never reuses historical tag values.
15. Runtime key encodings use `EntityTag` as the leading identity component.
16. Runtime descriptor resolution occurs once per operation and never inside
    scan loops.
17. Debug reverse mapping (`entity_tag_name`) covers all registered tags and is
    generated from the identity registry without runtime reflection.

## Risks

1. Over-broad signature churn across non-runtime layers.
2. Hidden `EntityKind` couplings reintroduced via helper modules.
3. Identity collisions if tag uniqueness is not enforced.
4. Runtime regressions if typed decode boundary drifts into runtime loops.
5. Generic reintroduction via helper traits (for example `FieldValue`,
   projection helpers, decode helpers) re-infects runtime loops.
6. Per-row `EntityTag` branch tables in hot loops negate size/perf gains.
7. Cursor/range semantics drift if legacy lexical `EntityName` assumptions
   survive in index/cursor framing.

Mitigations:

- enforce module-boundary checks
- add explicit no-generic/no-`EntityName` guards in targeted runtime modules
- fail build/test on duplicate `EntityTag` assignments
- keep typed decode at explicit adapter boundary only
- audit helper traits/functions for hidden generic runtime re-entry points
- resolve dispatch descriptors once per operation before scan loops
- add range/prefix/continuation regression coverage under `EntityTag` framing

## Exit Criteria

`0.58` is complete when runtime execution flow is identity-driven by
`EntityTag` and no longer monomorphized on entity type across hot runtime
paths.
