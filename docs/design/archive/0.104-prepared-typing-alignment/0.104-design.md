0.104 Prepared Typing Alignment

Thesis

0.104 should eliminate, or at least tightly constrain, the second type-family
system that prepared fallback execution now carries for expression-owned
parameterized `WHERE` shapes.

0.104 eliminates the parallel prepared fallback typing system by deriving
parameter contracts from planner-owned expression typing, ensuring prepared and
non-prepared execution agree on bind-family decisions without widening the
template boundary.

Problem statement

After `0.103`:

* residual filtering is explicit plan state
* explain can describe semantic vs pushdown vs residual clearly
* prepared templates remain explicitly predicate/access-owned

However, prepared fallback now has its own recursive type-family inferer in
`prepare.rs` so it can collect contracts for expression-owned parameterized
`WHERE` queries.

That means the system now has two nearby owners for expression typing:

* planner expression typing
* prepared fallback parameter-family inference

This is risky because:

* newly admitted scalar functions can widen one path without widening the other
* prepared fallback can reject or misclassify bindings that non-prepared
  execution would accept
* parameter-contract drift becomes easier as the SQL surface expands

Design goal

Make prepared parameter typing derive from the same expression contract as the
planner.

That means:

* prepared fallback parameter families are derived from planner expression
  typing
* bind validation stays coarse and stable
* prepared templates remain predicate/access-only
* no new template widening happens in this slice

Non-goals

0.104 does not aim to:

* widen prepared symbolic templates
* introduce expression-owned prepared templates
* make prepared execution use the full planner type lattice at runtime
* change scalar `WHERE` admission
* change residual execution semantics
* optimize prepared fallback performance

This is a contract-alignment slice, not a template-expansion slice.

Normative model

Rule 1 - prepared templates remain unchanged

Prepared fixed-route templates continue to operate only on predicate/access
shapes.

General expression-owned `WHERE` semantics remain fallback-only in prepared
execution.

Rule 2 - fallback parameter typing derives from planner-owned expression typing

When prepared fallback needs parameter contracts for expression-owned `WHERE`
shapes, those contracts must derive from the same planner-owned expression
typing contract used by normal expression planning.

Prepared fallback typing must either:

* reuse planner expression type inference directly, or
* derive from a shared helper that is also used by planner typing

It must not reimplement expression typing logic independently in `prepare.rs`.

The required flow is:

* expression subtree
* planner-owned expression type
* projected coarse bind family

Not:

* expression subtree
* direct handwritten bind-family guess

Prepared fallback may keep a coarse outward contract family such as:

* `Bool`
* `Numeric`
* `Text`

But the decision for which family applies must come from planner-aligned type
reasoning rather than a parallel handwritten rule set.

Rule 3 - bind validation stays coarse

Prepared bind validation does not need the full planner type lattice.

It may still validate against coarse families, but the mapping from expression
shape to coarse family must be planner-aligned and centrally defined.

Bind validation operates only on the projected coarse family and must not
introduce additional semantic rules beyond planner typing.

If planner-aligned typing cannot determine a valid type for a parameter
position, prepared fallback must fail closed during contract collection.

Rule 4 - clause policy stays separate from typing

This slice does not change clause legality.

It only changes how already-admitted prepared fallback shapes derive parameter
contracts.

So:

* clause admission remains where it is
* template admission remains where it is
* parameter typing alignment changes underneath those boundaries

Rule 5 - no template leakage

Any expression-owned parameter shape that requires fallback typing alignment
must still keep:

* `template_binding = None`
* no symbolic template route
* no legacy fixed-route template route

Typing alignment must not become an accidental template admission path.

Examples

Prepared fallback expression-owned compare

```sql
SELECT name
FROM character
WHERE COALESCE(NULLIF(age, ?), 99) = ?
```

This should:

* prepare successfully
* derive numeric parameter families through planner-aligned typing
* stay off all template lanes
* execute through prepared fallback only

Prepared fallback unary text expression

```sql
SELECT name
FROM character
WHERE LOWER(TRIM(COALESCE(nickname, ?))) = ?
```

This should:

* derive text-family contracts through planner-aligned typing
* remain fallback-only
* not require a second handwritten prepared-only function-family table

Mixed predicate-owned and expression-owned parameters

```sql
SELECT name
FROM character
WHERE id = ? AND COALESCE(age, ?) > 10
```

This should:

* collect both parameter contracts successfully
* keep `id = ?` on the predicate-owned/template-eligible side
* keep `COALESCE(age, ?)` on the expression-owned fallback-only side
* preserve the rule that typing alignment does not widen template admission

Likely code seams

* prepared parameter-contract collection
* planner expression typing helpers
* fallback bind-family projection from planner types
* prepared-query tests for expression-owned `WHERE`

Risks

Risk 1 - accidental template widening

If typing alignment is implemented inside template admission instead of
fallback typing, prepared templates will grow implicitly.

Risk 2 - over-coupling bind validation to planner internals

Prepared bind validation should stay coarse and stable. This slice should
derive from the source contract, not expose the whole planner type lattice at
the session boundary.

Risk 3 - incomplete family coverage

If planner-aligned typing only covers some expression families, prepared
fallback will remain inconsistent for the remaining admitted shapes.

Verification matrix

Area | Required proof
--- | ---
Typing alignment | prepared fallback and planner typing agree on coarse family for admitted expression-owned `WHERE` shapes
Fallback behavior | aligned parameter contracts do not widen any template lane
Binding validation | valid bindings accepted and invalid bindings rejected consistently with planner-owned typing
Template boundary | predicate/access templates remain unchanged
Explain/runtime parity | prepared fallback still executes the same semantic filter as non-prepared execution

Acceptance criteria

0.104 is complete when:

* prepared fallback no longer depends on an independent handwritten expression
  family system
* coarse prepared bind families derive from planner-aligned typing
* template admission remains unchanged
* expression-owned prepared `WHERE` fallback stays correct across admitted
  scalar expression families

Thesis sentence

0.104 aligns prepared fallback parameter typing with the planner-owned
expression model so expression-owned prepared `WHERE` shapes stay correct and
predictable without widening the prepared template boundary.
