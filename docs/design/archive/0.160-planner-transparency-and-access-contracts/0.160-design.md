# 0.160 Planner Transparency and Access Contracts

## Status

Implemented.

## Purpose

IcyDB’s query API is intentionally predicate-first: application code expresses
logical filters, and the planner chooses the physical access path. That remains
the right architecture.

The gap is not planner ownership. The gap is that hot-path developers cannot
always see or enforce what the planner chose. A fluent query such as:

```rust
load::<MovementIntent>()
    .filter(session_id.eq(id))
    .filter(turn_number.eq(turn))
    .filter(status.eq(Open))
```

may already lower to a bounded composite index access path, but the call site
still reads like post-scan filtering. 0.160 should make selected access paths,
index-prefix matching, residual work, and planner fallback decisions explicit,
testable, and optionally fail-closed.

## Current Repo Facts

- `ExplainPlan` already exposes the logical query surface and the selected
  `ExplainAccessPath` in `db::query::explain::plan`.
- `ExplainAccessPath` already has the planner vocabulary needed for this work:
  `ByKey`, `ByKeys`, `KeyRange`, `IndexPrefix`, `IndexMultiLookup`,
  `IndexRange`, `FullScan`, `Union`, and `Intersection`.
- `AccessChoiceExplainSnapshot` already records planner-owned candidate,
  alternative, rejection, and selected-reason metadata.
- Residual expression and predicate metadata already exist on execution explain
  descriptors.
- Planner ranking already centralizes prefix length, exactness, filtered-index
  preference, range bound count, and order compatibility in `AccessCandidateScore`.

The design should use those existing surfaces. It should not introduce a second
planner explanation model beside them.

## Goals

- Make selected access paths and planner-choice reasons visible in stable text
  and JSON explain output.
- Distinguish predicates used for index access from predicates left as residual
  evaluation.
- Surface composite-index prefix behavior, including range anchors and trailing
  fields that cannot narrow the scan.
- Add fail-closed access requirements for hot paths and regression tests.
- Keep physical selection planner-owned; requirements assert the result after
  planning rather than acting as optimizer hints.
- Drive index metadata from accepted runtime schema where available.

## Non-Goals

- No manual index hints as the normal query-construction API.
- No storage-key or BTree internals in public query builders.
- No generated-model fallback reconstruction in runtime planning paths.
- No cost model redesign in this slice.
- No index-intersection or bitmap-merge implementation in this slice.

## Recommended Design

## Implementation Status

The first 0.160 slice is implemented:

- `ExplainPlan` now carries `ExplainAccessDecisionV1`, projected from the
  planner-owned `AccessChoiceExplainSnapshot`.
- Logical JSON explain exposes the structured selected access path, selected
  reason, candidates, alternatives, rejections, and residual-work summary.
- Logical text explain renders one compact access-decision summary and leaves
  detailed candidate lists to JSON.
- `IndexPrefix` JSON explain now reports bound fields, bound values, and
  unbound trailing fields.
- `IndexRange` JSON explain now reports equality prefix fields/values, the
  range anchor field, bound inclusivity, and trailing fields.
- Fluent query builders now support `require_index()`,
  `require_index_named(...)`, `require_access_path(...)`, and
  `require_no_residual_filter()`.
- Requirement failures preserve the selected access decision in
  `AccessRequirementError`.
- Internal test helpers can assert selected index, access kind, bound prefix
  length, and no-residual state without parsing rendered explain text.

The implemented residual DTO is intentionally summary-level for 0.160. It
exposes residual burden class, scalar-residual presence, predicate-residual
presence, access-bound constraint count, and residual predicate count. It does
not yet expose a stable public list of every access-bound or residual predicate
term.

Still deferred:

- broader runtime/session tests for accepted-schema named-index requirements
- representative migration of older string-based planner tests to the new
  structured helper

### 1. Canonical Access Decision Model

Promote the planner-owned access decision into a stable explain payload instead
of rendering only ad hoc strings.

The access decision should include:

- selected access kind
- selected index name, when applicable
- selected reason from `AccessChoiceSelectedReason`
- candidate summaries from `AccessChoiceCandidateExplainSummary`
- rejected index reasons
- alternative eligible index names
- residual burden class

This should be projected from the existing `AccessChoiceExplainSnapshot`; the
snapshot remains the planner-owned source of truth.

### 2. Prefix and Range Detail

For `IndexPrefix`, explain should show:

- index name
- full index key fields/items
- bound prefix length
- bound fields/items
- bound values
- unbound trailing fields/items

For `IndexRange`, explain should show:

- equality prefix fields/items and values
- range field/item
- lower and upper bounds
- whether the lower/upper bound is inclusive, exclusive, or unbounded
- trailing fields/items not usable after the range anchor

This turns composite-index behavior into inspectable planner output without
changing how queries are authored.

### 3. Residual Predicate Visibility

Explain output should separately report:

- access-bound predicate/count summary
- residual scalar filter presence
- residual predicate model presence/count
- residual burden classification

This distinction matters because an index access path can still leave work for
post-access evaluation. The selected path is only fully exact when the access
contract and residual metadata prove no remaining predicate work.

### 4. Access Requirements

Add optional fail-closed access requirements to fluent query builders.

Initial API shape:

```rust
load::<MovementIntent>()
    .filter(session_id.eq(id))
    .filter(turn_number.eq(turn))
    .filter(status.eq(Open))
    .require_index();
```

Named index requirement:

```rust
load::<MovementIntent>()
    .filter(session_id.eq(id))
    .require_index_named("movement_intent_session_turn_status");
```

Access-kind requirement:

```rust
load::<MovementIntent>()
    .filter(session_id.eq(id))
    .require_access_path(RequiredAccessPath::IndexPrefix);
```

Requirement evaluation happens after planning and before execution. A failed
requirement is a plan-domain query error, not a runtime fallback.

Requirements are assertions, not hints:

- `require_index()` accepts `IndexPrefix`, `IndexMultiLookup`, and `IndexRange`.
- `require_index_named(name)` additionally checks the selected semantic index
  contract name.
- `require_access_path(kind)` checks the selected `AccessPathKind`.
- requirements must not change candidate ranking.
- requirements must not force a less-preferred index.
- `require_no_residual_filter()` fails when the selected plan still carries
  residual scalar or predicate work after access selection.

### 5. Structured Plan Assertions

Tests should not need to match rendered explain strings. Add test/support
helpers that inspect `ExplainPlan` and access-decision DTOs directly:

```rust
assert_plan(plan)
    .uses_index("movement_intent_session_turn_status")
    .access_kind(RequiredAccessPath::IndexPrefix)
    .bound_prefix_len(3)
    .has_no_residual_filter();
```

These helpers should live in test support first. Public assertion APIs can wait
until the shape stabilizes.

### 6. Index Analysis Is Separate

Index-quality analysis is useful, but it should not be mixed into the first
access-contract slice.

Future tooling can analyze accepted schema indexes for:

- redundant prefixes
- overlapping composite indexes
- unused or low-selectivity indexes, once metrics exist
- query-shape coverage gaps

That tooling should read accepted schema/index metadata, not generated model
fallbacks.

## Error Semantics

Access requirement failures should be explicit and structured:

- no selected index path
- selected index name mismatch
- selected access kind mismatch
- selected path still has residual work when an exact contract is requested

The error should preserve the selected access decision so callers and tests can
diagnose why the requirement failed.

## Explain Output Shape

Text explain should remain compact but include enough access detail to answer:

- What path was selected?
- Which index was selected?
- Which prefix was bound?
- Was there a range anchor?
- Whether scalar or predicate residual work remains, and how much predicate
  work remains.
- Why were other indexes rejected?

JSON explain should carry the complete structured access-decision model shipped
in this release so downstream tests and tools do not parse text.

## Rollout Plan

### Phase 1: Explain Decision Projection

Goal: expose the existing access-choice snapshot through stable explain text and
JSON.

Likely modules:

- `db::query::explain`
- `db::query::plan::access_choice`
- executor explain descriptor assembly

Validation:

- explain determinism tests
- JSON/text explain snapshot tests
- existing planner tests

### Phase 2: Prefix, Range, and Residual Detail

Goal: enrich `ExplainAccessPath` rendering with bound prefix, range, trailing
field, and residual metadata.

Likely modules:

- `db::query::explain::access_projection`
- `db::query::explain::render`
- `db::query::explain::json`
- `db::query::plan::planner::{prefix,range}`

Validation:

- composite equality-prefix tests
- composite range-anchor tests
- residual predicate tests
- filtered/expression index tests

### Phase 3: Access Requirements

Goal: add fail-closed planner-result requirements to fluent builders.

Likely modules:

- fluent query builder state
- session query execution path
- `AccessPathKind` / selected-index inspection helpers
- query plan error model

Validation:

- `require_index()` passes for `IndexPrefix`, `IndexMultiLookup`, and
  `IndexRange`
- `require_index()` fails for `FullScan`, `ByKey`, `ByKeys`, and `KeyRange`
- named requirement fails on selected-index mismatch
- requirements do not change chosen plan

### Phase 4: Test Assertion Helpers

Goal: make planner regression tests direct and structured.

Likely modules:

- core test support
- planner/explain tests

Validation:

- migrate representative string-based or broad `not FullScan` assertions to
  structured checks
- keep public explain rendering tests focused on rendering only

### Phase 5: Index Analysis Tooling

Goal: add accepted-schema index hygiene diagnostics after the contract surface is
stable.

Likely modules:

- schema/index metadata readers
- CLI or developer diagnostics surface

Validation:

- redundant-prefix fixtures
- overlapping composite-index fixtures
- accepted-schema-only authority tests

## Risks

- Explain output can become noisy if text rendering tries to show every
  candidate by default. Keep text concise and JSON complete.
- Requirement APIs can be mistaken for hints. Name and docs must emphasize that
  they assert planner output after selection.
- Named-index requirements can make application code brittle if used broadly.
  They should be recommended only for hot paths and regression tests.
- Residual-free claims must be based on planner-owned residual metadata, not
  string labels.
- Runtime accepted schema must remain the index metadata authority; generated
  model bridges are only valid for proposal/model-only/test contexts.

## Final Target

0.160 should ship a planner-observability and enforcement layer:

- richer, structured access-decision explain output
- explicit composite prefix/range/residual diagnostics
- optional fail-closed access requirements
- structured test assertions

The result should let developers keep writing predicate-first queries while
being able to prove when those queries use the intended indexed access path.
