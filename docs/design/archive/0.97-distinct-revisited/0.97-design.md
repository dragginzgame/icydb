# 0.97 DISTINCT and Result Pipeline Completion

## Thesis

`0.97` should formalize and tighten the SQL `SELECT DISTINCT` contract as full, correct, and expression-aware deduplication **after projection**, completing the SQL-side single-table result pipeline without redefining every existing non-SQL `DISTINCT` path in the repo.

> **0.97 formalizes SQL `SELECT DISTINCT` as post-projection deduplication over compiled expression outputs, preserves existing non-SQL distinct semantics outside this slice, rejects ambiguous non-projected `ORDER BY` under `DISTINCT`, and makes pagination plus grouped-normalization behavior explicit.**

This slice builds directly on the stable semantic and execution model established in `0.96`.

---

## Problem statement

After `0.96`, the system has:

* unified semantic expression model (`Expr`)
* fully compiled execution paths (no interpreter)
* consistent projection and ordering execution
* stable query planning and execution contracts

However, `DISTINCT` semantics are still split across domains:

* SQL scalar `SELECT DISTINCT` already performs projection-level deduplication in admitted cases
* fluent/kernel `distinct()` paths still include row-identity-sensitive behavior in some execution flows
* planning and execution already expose `DISTINCT` strategy choices
* the repo therefore has shipped `DISTINCT` behavior, but not one fully explicit contract for SQL projection `DISTINCT`

`0.97` is therefore not introducing `DISTINCT` from scratch.

It is defining and tightening the SQL `SELECT DISTINCT` contract, especially for projection-level deduplication over compiled expression outputs, while making planner/executor behavior explicit and coherent.

---

## Design goal

Introduce `DISTINCT` as a **first-class result pipeline stage** that:

* operates on fully evaluated projection output
* supports arbitrary expressions
* integrates cleanly with ORDER BY and LIMIT
* preserves existing non-SQL distinct semantics outside this slice

The goal is:

> **complete the single-table query pipeline by adding deterministic, expression-aware row deduplication without introducing new execution abstractions.**

---

## Non-goals

This slice does **not** aim to:

* introduce JOIN, subqueries, or window functions
* support `DISTINCT ON`
* support DISTINCT inside aggregate functions
* optimize DISTINCT via index pushdown or advanced strategies
* redefine every existing fluent/kernel `distinct()` path
* redesign execution engine or planner IR
* introduce new expression abstractions

This is a **correctness and SQL-pipeline-completion slice**, not a repo-wide `DISTINCT` redesign, performance slice, or architecture slice.

---

# Part I — Current model

## Existing SQL result pipeline

```text
scan
→ WHERE
→ GROUP BY / aggregation
→ projection (compiled expressions)
→ ORDER BY
→ LIMIT / OFFSET
```

Current SQL projection `DISTINCT` behavior is admitted in some scalar paths, but the contract is not explicit enough for ordering, pagination, and cross-domain semantics.

---

## Current limitation

* SQL projection output already supports duplicate elimination in some shipped cases
* but the exact SQL `SELECT DISTINCT` contract is still underspecified relative to planner strategy selection, continuation semantics, and non-projected ordering
* existing fluent/kernel `distinct()` behavior is not the same contract and should not be silently reinterpreted as part of this slice

---

# Part II — Target model

## Final SQL `SELECT DISTINCT` pipeline

```text
scan
→ WHERE
→ GROUP BY / aggregation
→ projection (compiled expressions)
→ DISTINCT
→ ORDER BY
→ LIMIT / OFFSET
```

### Key invariant

> DISTINCT operates on the fully evaluated projected row, not on source rows or partial expressions.

---

# Part III — Core design

## 1. DISTINCT operates on projected rows

DISTINCT must evaluate uniqueness over:

```text
Vec<Value> (projected row representation)
```

Not:

* raw storage rows
* individual columns independently
* pre-projection expressions

### Explicit rule

> DISTINCT deduplicates rows based on the full projected output vector.

### Domain split

`0.97` is SQL-first.

It defines the SQL `SELECT DISTINCT` contract as projection-distinct over the final projected tuple.

It does **not**, by itself, redefine every existing fluent/kernel `distinct()` path that currently operates with row-identity-sensitive semantics.

### Explicit rule

> `0.97` defines SQL projection `DISTINCT`, not a repo-wide semantic replacement for every existing `distinct()` call site.

---

## 2. Expression compatibility

DISTINCT must support:

```sql
SELECT DISTINCT strength * 2 FROM Character;
SELECT DISTINCT LOWER(name) FROM Character;
SELECT DISTINCT strength, LOWER(name) FROM Character;
```

Because projection is already compiled:

* DISTINCT works on compiled expression outputs
* no new semantic expression type is required

---

## 3. Execution strategy

### Initial implementation

Use hash-based deduplication for the SQL projection-distinct lane:

```text
HashSet<ProjectedRow>
```

Where:

* `ProjectedRow` = vector of evaluated values
* hashing and equality use existing `Value` semantics

### Properties

* correct
* deterministic
* simple
* consistent with the SQL projection pipeline

### Explicit rule

> DISTINCT correctness takes priority over optimization in 0.97.

---

## 4. Planner integration

`DISTINCT` already participates in planning in this codebase.

`0.97` does not introduce `DISTINCT` planning from scratch.

Instead, it clarifies which existing distinct execution strategies remain valid, which are SQL projection-distinct specific, and how those strategies interact with explain, cache identity, and route selection.

### Explicit rule

> Preserve the existing planner-owned `DISTINCT` strategy surface and repurpose or extend it for SQL projection-distinct where needed; do not collapse this slice to a new bool-only planning story.

---

## 5. DISTINCT + ORDER BY

Supported:

```sql
SELECT DISTINCT name FROM Character ORDER BY name;
```

Rejected in `0.97`:

```sql
SELECT DISTINCT name FROM Character ORDER BY age;
```

because `age` is not derivable from the projected distinct tuple.

### Explicit rule

> Under SQL `SELECT DISTINCT`, `ORDER BY` expressions must be derivable from the projected distinct tuple. Non-projected ordering terms are rejected fail-closed.

This avoids hidden ordering payloads, representative-row ambiguity, and unstable continuation behavior.

---

## 6. DISTINCT + LIMIT

Supported:

```sql
SELECT DISTINCT name FROM Character LIMIT 10;
```

Execution order:

```text
project → distinct → order → limit / offset
```

No early-exit optimization required in 0.97.

### Continuation invariant

> For SQL `SELECT DISTINCT`, ordering, page windows, and continuation tokens are computed over the ordered deduplicated stream, not the pre-deduped input stream.

### Explicit rule

> Continuation state for SQL `SELECT DISTINCT` queries must be defined in terms of the post-dedup ordered row stream.

### Pagination invariant

> All `LIMIT`, `OFFSET`, and continuation semantics for SQL `SELECT DISTINCT` operate over the ordered deduplicated stream, never over raw scanned rows or pre-dedup page windows.

### Current scalar SQL surface boundary

The current top-level scalar SQL statement payload does not expose continuation
tokens. In `0.97`, the shipped SQL `SELECT DISTINCT` proof therefore covers
`LIMIT` / `OFFSET` semantics directly, while grouped SQL remains the only SQL
statement surface that emits `next_cursor`.

---

## 7. DISTINCT vs GROUP BY

DISTINCT is **not** GROUP BY.

### Distinction

| Feature      | DISTINCT           | GROUP BY         |
| ------------ | ------------------ | ---------------- |
| Aggregation  | No                 | Yes              |
| Semantics    | dedupe rows        | group rows       |
| Output shape | projection-defined | grouping-defined |

### Explicit rule

> DISTINCT must not introduce aggregation semantics.

### Grouped top-level DISTINCT normalization

Top-level `SELECT DISTINCT ... GROUP BY ...` remains normalized as a no-op in `0.97` where grouping already guarantees uniqueness of the projected grouped row shape under the admitted SQL surface.

### Explicit rule

> Preserve the current grouped top-level `SELECT DISTINCT` normalization instead of inventing a second grouped post-projection distinct phase.

---

## 8. DISTINCT + HAVING

Invalid:

```sql
SELECT DISTINCT name FROM Character HAVING ...
```

Unless explicitly supported in grammar (out of scope).

---

## 9. NULL semantics

DISTINCT must treat NULL values according to row equality rules:

* two rows are equal if all fields compare equal under Value equality
* NULL = NULL for DISTINCT purposes (standard SQL behavior)

### Explicit rule

> DISTINCT uses value equality, not SQL comparison semantics, for deduplication.

---

# Part IV — Execution details

## 1. Data structure

Introduce:

```text
HashSet<ProjectedRow>
```

or equivalent.

Requirements:

* deterministic equality
* stable hashing
* supports Value types

---

## 2. Integration point

SQL projection `DISTINCT` should be implemented:

* after projection evaluation
* before final page windows
* with ordering and continuation defined over the deduplicated stream

Executor pipeline:

```text
evaluate projection → produce row
→ check hash set
  → if new: emit
  → if duplicate: skip
→ order deduped rows
→ apply limit / offset / continuation window
```

---

## 3. Memory considerations

* DISTINCT requires storing seen rows
* acceptable for now (single-table scope)

No streaming optimization required.

---

# Part V — Acceptance criteria

## Functional

* SQL `SELECT DISTINCT` removes duplicate projected rows correctly
* supports:

  * single column
  * multiple columns
  * arbitrary expressions
* integrates with `ORDER BY`, `LIMIT`, `OFFSET`, and continuation semantics

---

## Behavioral

* SQL projection `DISTINCT` operates on projected values
* non-SQL `distinct()` semantics are preserved outside this slice
* non-projected `ORDER BY` under `SELECT DISTINCT` rejects fail-closed
* grouped top-level `SELECT DISTINCT` remains a no-op normalization
* does not affect aggregation semantics
* does not introduce new expression evaluation paths

---

## Architectural

* no new execution abstraction introduced
* uses existing compiled projection output
* preserves the existing planner strategy surface for distinct execution instead of pretending the implementation is bool-only

---

## Testing

Add tests for:

```sql
SELECT DISTINCT strength;
SELECT DISTINCT strength, name;
SELECT DISTINCT strength * 2;
SELECT DISTINCT LOWER(name);
SELECT DISTINCT strength ORDER BY strength;
SELECT DISTINCT strength LIMIT 5;
```

Also include:

* duplicate elimination correctness
* NULL equality behavior
* fail-closed rejection for `SELECT DISTINCT name ORDER BY age`
* pagination / continuation behavior over deduped ordered rows
* grouped top-level `SELECT DISTINCT ... GROUP BY ...` normalization parity

# Part VII — Risks

## Risk 1 — wrong placement

Mitigation:

* ensure SQL projection `DISTINCT` runs after projection

---

## Risk 2 — incorrect equality semantics

Mitigation:

* reuse Value equality semantics
* add explicit NULL tests

---

## Risk 3 — confusion with GROUP BY

Mitigation:

* keep logic separate
* avoid sharing aggregation paths

---

## Risk 4 — ambiguous ORDER BY semantics

Mitigation:

* reject non-projected `ORDER BY` terms under SQL `SELECT DISTINCT`
* do not carry hidden representative-row ordering payloads in `0.97`

---

## Risk 5 — continuation drift

Mitigation:

* define pagination and continuation over the ordered deduped stream
* add direct continuation and offset coverage for SQL `SELECT DISTINCT`

---

# Part VIII — Out of scope

* DISTINCT pushdown optimization
* memory-efficient streaming dedupe
