# 0.86 Grouped HAVING Expression Widening

## Thesis

`0.86` should widen grouped SQL `HAVING` from the current **symbol-based predicate model** to a **post-aggregate boolean expression model**.

The goal is not to extend the existing `GroupHavingClause` shape.

The real target is:

> allow bounded boolean expression evaluation in `HAVING` over finalized grouped outputs, by reusing the post-aggregate expression layer introduced in `0.85`.

That means:

* reuse the grouped output row (`group keys + aggregate outputs`)
* reuse the shared scalar expression evaluator
* keep aggregate accumulation and finalization unchanged
* keep grouped ordering and boundedness unchanged
* replace symbol-only `HAVING` with expression-based `HAVING`

---

## Problem statement

After `0.85`, the system supports:

* post-aggregate projection expressions:

  * `ROUND(AVG(strength), 2)`
  * `COUNT(*) + 1`
  * `MAX(level) - MIN(level)`

But `HAVING` is still limited to:

* grouped key symbols
* aggregate symbols
* compare operators
* `AND` composition

So this works:

```sql
HAVING AVG(strength) >= 10
```

But this does not:

```sql
HAVING ROUND(AVG(strength), 2) >= 10
HAVING COUNT(*) + 1 > 5
HAVING MAX(level) - MIN(level) >= 3
```

This creates a clear asymmetry:

| Surface    | Expression support |
| ---------- | ------------------ |
| Projection | widened (0.85)     |
| HAVING     | narrow             |

`0.86` removes that asymmetry.

---

## Architectural position

### Reuse the 0.85 post-aggregate expression model

Do not introduce a second grouped expression system.

`0.85` already establishes:

```text
GroupedOutputRow {
  group_key_slots...
  aggregate_output_slots...
}
```

`0.86` should evaluate `HAVING` as:

```text
boolean_expr(GroupedOutputRow) -> bool
```

---

### Preserve execution layering

Execution remains:

```text
scan
-> group
-> finalize aggregates
-> evaluate HAVING expression
-> evaluate projection expressions
-> apply ordering / limit
-> emit
```

Key constraint:

* `HAVING` runs **after aggregate finalization**
* `HAVING` does **not influence grouping mechanics**

---

### Do not widen execution semantics

`0.86` is strictly:

* expression widening

It must NOT:

* change grouping strategy
* change aggregate accumulation
* change ordering or continuation
* introduce new execution paths

---

## Scope

### In scope

* expression-based `HAVING` over:

  * grouped keys
  * finalized aggregate outputs
* scalar functions over grouped outputs
* arithmetic over grouped outputs
* comparison operators
* `AND` composition (minimum)

### Out of scope

* aggregate input expressions:

  * `AVG(ROUND(strength, 2))`
* raw non-grouped fields
* `OR` / `NOT` (optional future)
* grouped `ORDER BY` widening
* grouped DISTINCT policy changes
* joins, subqueries, window functions

---

## Current state

Grouped `HAVING` is currently:

* represented as `GroupHavingClause`
* built from `GroupHavingSymbol`

  * `GroupField`
  * `AggregateIndex`
* evaluated as simple compare predicates

Validation ensures:

* grouped fields are in `GROUP BY`
* aggregate indices are valid

This model cannot represent:

* nested expressions
* function calls
* arithmetic composition

---

## Design goal

### Introduce grouped HAVING as a boolean expression tree

Replace:

```text
GroupHavingClause {
  symbol,
  op,
  value
}
```

With:

```text
Expr::Boolean
```

Where leaves resolve to:

* grouped key slots
* aggregate output slots

---

## Proposed model

### Expression leaf types

Grouped `HAVING` expressions may only reference:

1. **Grouped key slots**
2. **Aggregate output slots**

No other sources are allowed.

---

### Expression evaluation boundary

Evaluation occurs only after:

```text
aggregate finalization
```

So:

```sql
ROUND(AVG(strength), 2) >= 10
```

becomes:

```text
slot_avg -> ROUND -> compare
```

---

## Expression support (v1)

### Supported

* grouped key references
* aggregate references
* scalar functions over them
* arithmetic over them
* comparison operators
* `AND`

### Examples

```sql
HAVING ROUND(AVG(strength), 2) >= 10
```

```sql
HAVING COUNT(*) + 1 > 5
```

```sql
HAVING MAX(level) - MIN(level) >= 3
```

```sql
HAVING class_name = 'Mage' AND AVG(strength) > 10
```

---

### Not supported (v1)

```sql
HAVING name = 'Ada'
```

```sql
HAVING ROUND(name, 2) > 0
```

```sql
HAVING AVG(ROUND(strength, 2)) > 10
```

```sql
HAVING COUNT(*) > 1 OR class_name = 'Mage'
```

---

## Parser changes

### Current

* HAVING parsed into symbol + compare form

### Target

* parse HAVING as full expression
* reuse existing SQL expression AST

Constraint:

> HAVING must parse as an expression, but validation restricts allowed leaves

---

## Lowering changes

### Current

* HAVING lowered into `GroupHavingClause`

### Target

Two-phase lowering:

#### Phase 1: aggregate extraction

* walk expression tree
* extract aggregate leaves
* assign aggregate indices

#### Phase 2: expression rewriting

* replace aggregate nodes with `AggregateSlot(idx)`
* replace grouped fields with `GroupKeySlot(idx)`
* produce expression tree over slots

---

## Validation rules

### Allowed leaves

* grouped key
* aggregate output

### Disallowed

* raw fields outside group key
* nested aggregates
* aggregate inside scalar inside aggregate

---

### Type rules

* enforce function signatures
* enforce arithmetic compatibility
* enforce boolean result type for HAVING

---

## Executor changes

### Execution flow

```text
GroupedOutputRow built
-> evaluate HAVING expression
-> if true: continue
-> else: drop row
```

---

### Evaluator reuse

* reuse shared scalar expression evaluator
* expression reads from grouped slots

No new evaluator should be introduced.

---

## DISTINCT interaction

Current rule:

* grouped DISTINCT + HAVING is rejected

`0.86` keeps this unchanged.

---

## Ordering and boundedness

No changes.

Grouped `ORDER BY`:

* still requires LIMIT
* still uses grouped-key prefix rules
* still uses narrow proof model

---

## Examples

### Valid after 0.86

```sql
SELECT class_name, AVG(strength)
FROM character
GROUP BY class_name
HAVING ROUND(AVG(strength), 2) >= 10
ORDER BY class_name
LIMIT 100;
```

```sql
SELECT class_name, COUNT(*)
FROM character
GROUP BY class_name
HAVING COUNT(*) + 1 > 5
ORDER BY class_name
LIMIT 100;
```

---

### Still invalid

```sql
HAVING name = 'Ada'
```

```sql
HAVING AVG(ROUND(strength, 2)) > 10
```

```sql
HAVING COUNT(*) > 1 OR class_name = 'Mage'
```

---

## Phased plan

### Phase A: expression model

* introduce HAVING as Expr
* restrict leaves

### Phase B: lowering

* extract aggregates
* rewrite to slot-based expression

### Phase C: executor

* evaluate HAVING over grouped output row

### Phase D: cleanup

* remove or wrap `GroupHavingClause`
* unify validation paths

---

## Guardrails

### 1. No new evaluator

Reuse shared expression evaluator.

---

### 2. No aggregate input widening

Do not allow:

```sql
AVG(ROUND(...))
```

---

### 3. No boundedness changes

Do not touch ORDER BY semantics.

---

### 4. Strict leaf authority

Only:

* grouped keys
* aggregate outputs

---

## Acceptance criteria

### Functional

* HAVING supports scalar expressions over aggregates
* HAVING supports arithmetic over aggregates
* invalid raw field references fail

---

### Architectural

* grouped executor unchanged
* shared evaluator reused
* no duplicate logic paths

---

### Performance

* no change to grouping cost
* HAVING evaluation happens post-finalization only

---

## Thesis sentence

> **`0.86` widens grouped SQL HAVING to support post-aggregate boolean expressions over grouped keys and finalized aggregate outputs, reusing the 0.85 expression layer while preserving grouped execution and boundedness guarantees.**
