# IcyDB 0.72 Design Doc

## Deterministic Planning & Index Exploitation

**Status:** In Progress
**Version:** 0.72.x
**Date:** 2026-04-08

---

## 1. Title

**0.72 — Deterministic Planning & Index Exploitation**

---

## 2. Context

`0.70` fixed the read-correctness boundary:

* non-ready indexes are hidden from planning
* planner-visible indexes are the only indexes execution may trust
* execution no longer owns index-correctness classification

`0.71` then normalized aggregate execution so prepared strategy became the behavioral source for runtime and explain across the aggregate families. That leaves the next major improvement surface above execution shape:

> **planning quality**

This is also consistent with the explicit `0.70` aggregate boundary:

> aggregate optimization is a planner/strategy problem, not a runtime correctness problem.

---

## 3. Problem

The system now has a cleaner correctness and execution model, but planning is still the main place where query quality can drift.

Current problems are likely to include:

* implicit or heuristic index choice
* incomplete or uneven composite-index exploitation
* ORDER BY / LIMIT plans that do not consistently exploit already-visible indexes
* EXPLAIN surfaces that describe the chosen route, but not always the planner’s decision process as explicitly as they could

The post-`0.70` design already says the long-term explain story should answer planner questions like:

* which visible index was chosen?
* which fallback path was chosen when no visible index qualified?
* whether an index-backed shortcut was admitted by planner-visible readiness

`0.72` should make that concrete.

---

## 4. Core Goal

Make planning:

* **deterministic**
* **more exploitative of visible indexes**
* **more explainable**

without changing the `0.70` correctness boundary.

---

## 5. Non-Goals

`0.72` is **not**:

* a new correctness model
* a reintroduction of execution-time index validation
* progressive or partial index visibility
* a cost-based optimizer with statistics
* another aggregate execution normalization line

Those are outside the current architectural stop point. The current shipped invariant remains:

> planner visibility owns index correctness, and execution trusts the resulting plan.

---

## 6. Design Principles

### 6.1 Planner owns admissibility

If an index is not planner-visible, it does not exist for route selection.
If it is planner-visible, execution may trust it.

### 6.2 Determinism over ad hoc heuristics

Given the same:

* model
* visible index set
* query shape

the planner should choose the same route every time.

That means route selection should use one canonical comparison key, not a
sequence of loosely ordered local preferences.

Suggested first comparison key:

1. predicate-match class
2. order-satisfaction class
3. projection/covering class
4. residual-work class
5. stable structural tie-breaker

Where the tie-breaker should be something planner-owned and deterministic,
such as `(entity_name, index_name, access_family)`, not discovery order or
incidental iteration order.

### 6.3 Explain should project planner choice

EXPLAIN should not recreate a second planning story downstream. It should render planner-owned route choice and fallback reasoning, consistent with the direction already documented in `0.70`.

### 6.4 Strategy exploitation, not policy widening

`0.72` should exploit the already-valid visible indexes better. It should not widen which indexes are considered valid or visible.

---

## 7. Scope

`0.72` should focus on four concrete planning improvements.

First-slice scope boundary:

* single-entity read routes only
* no grouped aggregate planning changes
* no relation/join planning changes
* no distinct-planner redesign

The goal of `0.72` is to make the existing visible-index planner more
deterministic and more exploitative within that boundary, not to broaden the
query classes that planner policy owns.

### 7.1 Deterministic index selection

Introduce a stable planner ordering for competing visible indexes.

Suggested comparison key:

1. exact predicate alignment
2. order-compatible route
3. covering / projection-compatible route
4. narrower residual work
5. stable tie-breaker

This is not a statistics-based cost model. It is a **rule-ordered planner policy**.

---

### 7.2 Composite index exploitation

Formalize planner admission for common composite cohorts instead of leaving them as uneven special cases.

Primary target cohorts:

* equality prefix:

  * `WHERE a = ?`
  * `WHERE a = ? AND b = ?`
* bounded range after equality prefix
* order-compatible suffix scans:

  * `WHERE a = ? ORDER BY b`
* simple order-only composite scans where predicate is absent and planner-visible index order is already sufficient

This remains consistent with earlier `0.70` wording that later work should optimize planner admission and execution strategy freely, while keeping correctness out of execution.

---

### 7.3 ORDER BY / LIMIT exploitation

Make planner routing more intentional for bounded ordered reads.

Representative queries:

```sql
SELECT * FROM Customer ORDER BY age LIMIT 10;
SELECT name FROM Customer WHERE age >= 30 ORDER BY age DESC LIMIT 5;
```

Planner should prefer:

* visible index order when it already satisfies the requested order/window
* fallback materialized sort only when no visible index qualifies

This matches the current `0.70.5` behavior where non-visible building indexes fall back to ordinary full-scan/materialized routes; `0.72` should improve the positive case, not alter the fallback rule.

---

### 7.4 Planner-authored explain choice surfaces

Add or tighten explicit planner-facing explain vocabulary around:

* chosen visible index
* planner-authored rejected route families
* chosen fallback when no index qualifies
* whether a route is:

  * covering
  * order-compatible
  * materialized
  * bounded by planner-visible window/seek rules

The key point is that EXPLAIN should answer planner-choice questions, not execution-correctness questions. That direction is already explicit in the `0.70` notes.

Important constraint:

* EXPLAIN must not infer or recreate rejected routes downstream
* if rejected route families or fallback reasons are shown, they must come
  from planner-emitted metadata captured during route selection

That keeps explain as a projection of planner behavior instead of turning it
into a second planner.

---

## 8. What 0.72 should not reopen

### 8.1 No runtime index correctness branching

Do not reintroduce:

* authority snapshots
* runtime index trust decisions
* “soft downgrade” execution checks

Those were explicitly removed from the `0.70` target model.

### 8.2 No aggregate-family redesign

`0.71` is the execution-shape cleanup line. `0.72` should consume that cleaner execution surface, not reopen it.

### 8.3 No full cost model

No statistics, cardinality estimates, or adaptive planning in this line.

### 8.4 No grouped, relation, or distinct-planner expansion

Do not use `0.72` to broaden planner ownership into:

* grouped aggregate route selection
* relation or join planning redesign
* distinct-specific planner policy

Those may become later planner work, but they are not part of the first
deterministic-planning line.

---

## 9. Proposed Architecture Boundary

```text
query intent / lowered SQL
    -> planner-visible indexes
    -> deterministic route selection
    -> prepared route contract
    -> execution
    -> explain projection
```

Where:

* **planner** owns:

  * visible-index admission
  * route choice
  * fallback choice
  * deterministic tie-breaking

* **execution** owns:

  * running the prepared route
  * no correctness branching

* **explain** owns:

  * projection of planner route choice and execution shape
  * no second planner

This follows the same “single source, downstream projection” direction that recent aggregate work has been following, and it preserves the `0.70` correctness boundary.

---

## 10. Rollout Plan

### Slice 1 — Deterministic planner ordering

* centralize route/index ranking
* define one canonical comparison key
* keep the first pass on single-entity read routes only
* no behavior widening
* add stable explain assertions

Current status:

* prefix-family deterministic choice is implemented and locked in planner plus verbose explain
* range-family deterministic choice is implemented and locked in planner plus verbose explain
* order-only fallback choice is implemented and locked in planner plus verbose explain
* the current canonical score is:

  * `prefix_len`
  * family-specific `exact`
  * `order_compatible`
  * lexicographic index name tie-break

### Slice 2 — Composite equality-prefix and suffix-order cohorts

* admit the simplest useful composite families
* keep explicit rejection for unsupported shapes

### Slice 3 — ORDER BY / LIMIT exploitation

* prefer visible index order when it already satisfies the query
* lock fallback behavior when no visible index qualifies

### Slice 4 — Explain planner-choice surfaces

* chosen index
* planner-emitted fallback reason
* planner-emitted rejected route families
* route class
* no execution-correctness labels

---

## 11. Validation

### Functional

* core tests
* generated SQL parity
* PocketIC

This mirrors the `0.70.5` style of validating both route behavior and explain behavior across core, parity, and PocketIC.

### Performance

Focus on:

* ORDER BY / LIMIT cohorts
* equality-prefix and range queries over visible secondary indexes
* explain surfaces for those same planner-visible paths

### Invariant locks

Add tests that prove:

* non-visible indexes remain absent from planning
* deterministic planner choice stays stable
* explain shows planner fallback rather than execution downgrade

---

## 12. Expected Impact

### Immediate

* more predictable route selection
* better exploitation of already-visible indexes
* clearer planner-facing explain output

### Medium-term

* easier future extension to additional visible-index cohorts
* less route drift from local heuristics
* better foundation for later planner work without reopening execution correctness

---

## 13. Summary

`0.72` should be the first line that fully takes advantage of the post-`0.70` and post-`0.71` architecture:

* `0.70` made planner visibility the correctness boundary
* `0.71` normalized aggregate execution around prepared strategy
* `0.72` should make planner route choice deterministic, more exploitative, and more explainable

> **0.72 is a planner-quality line, not a correctness line and not another aggregate cleanup line.**
