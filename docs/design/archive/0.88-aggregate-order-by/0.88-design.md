# 0.88 Grouped Aggregate ORDER BY with Bounded Top-K

## Thesis

`0.88` should introduce **bounded execution support for grouped `ORDER BY` over aggregate expressions**, enabling queries like:

```sql
SELECT class_name, AVG(strength)
FROM character
GROUP BY class_name
ORDER BY AVG(strength) DESC
LIMIT 10;
```

without violating IcyDB’s core guarantees:

* bounded execution
* stable semantics
* no implicit full materialization or unbounded sort

The core mechanism is:

> evaluate aggregate-based order expressions using a **bounded top-k selection strategy** instead of canonical grouped-key ordering.

---

## Problem statement

After `0.85–0.87`, the system can:

* compute aggregate expressions
* filter with them (`HAVING`)
* reject them in `ORDER BY`

The current limitation:

```sql
ORDER BY AVG(strength)
```

is rejected—not because it’s not computable, but because:

* it breaks canonical grouped-key order
* it requires sorting grouped results
* sorting is currently unbounded / undefined

This creates a usability gap:

* users expect aggregate ranking queries
* the engine already computes all required values
* only ordering is missing

---

## Architectural position

### Introduce a second grouped ordering mode

Up to now, grouped ordering has been:

```text
Mode A: Canonical grouped-key order (streaming, continuation-safe)
```

`0.88` introduces:

```text
Mode B: Bounded top-k order (aggregate-based, heap-backed)
```

These are fundamentally different execution strategies.

If any grouped `ORDER BY` term depends on aggregate output or other
post-aggregate expression evaluation, the entire grouped order list runs in
Top-K mode.

Canonical grouped-key terms may still appear in that same term list as
tie-breakers.

---

### Do not overload canonical ordering

Aggregate ordering must NOT:

* reuse canonical grouped ordering
* pretend to preserve grouped-key order
* reuse continuation semantics

It is a distinct execution mode.

---

## Scope

### In scope

* `ORDER BY` expressions that depend on:

  * aggregate outputs
  * post-aggregate expressions
  * grouped keys as mixed order terms or tie-break terms
* only when paired with:

  * `LIMIT`
* bounded top-k execution strategy
* support for:

  * ASC / DESC
  * multi-column ordering
  * `ORDER BY <alias>` when the alias resolves to an admitted grouped
    post-aggregate order expression

---

### Out of scope

* aggregate ordering without `LIMIT`
* cursor continuation for aggregate ordering
* full sort of grouped output
* streaming execution for aggregate ordering
* wrapper/function ordering that does not reuse admitted grouped-key or
  post-aggregate expression families

---

## Execution model

### Canonical grouped ordering (existing)

```text
scan → group → emit in grouped-key order → LIMIT applied
```

### Aggregate ordering (new)

```text
scan
→ group (build full aggregate state)
→ finalize grouped rows
→ evaluate grouped ORDER BY key tuple
→ maintain top-k heap
→ emit sorted results
```

---

## Core algorithm

### Top-K heap selection

For:

```sql
ORDER BY expr LIMIT K
```

Execution:

1. compute grouped rows (key + aggregate outputs)
2. for each row:

   * evaluate the full grouped order key tuple
   * insert into heap of size `K`
   * evict worst element if needed
3. after all rows:

   * extract heap
   * sort the retained rows deterministically
   * emit the final `K`

---

### Complexity

* Time: `O(N log K)`
* Incremental ordering memory: `O(K)`

Where:

* N = number of groups
* K = LIMIT

This is the incremental ordering cost only.

Grouped aggregate execution still owns aggregate state for all live groups;
Top-K adds bounded retained-order state on top of that existing grouped fold
cost.

---

## Admission rules

### Required

* `LIMIT` must be present
* `ORDER BY` term list may depend on:

  * aggregate outputs
  * grouped keys
  * scalar composition permitted by the grouped post-aggregate expression model
* if any term requires post-aggregate expression evaluation, the full grouped
  order list runs in Top-K mode
* every Top-K grouped order must have one deterministic final tie-break:

  * user-specified later terms, if present
  * otherwise canonical grouped key order as the final fallback

---

### Rejected

```sql
ORDER BY AVG(strength)
```

(without LIMIT)

Reason:

* requires full sort
* violates boundedness

---

### Allowed

```sql
ORDER BY AVG(strength) DESC LIMIT 10
```

```sql
ORDER BY MAX(level) - MIN(level) LIMIT 5
```

```sql
ORDER BY ROUND(AVG(strength), 2) LIMIT 20
```

```sql
ORDER BY AVG(strength) DESC, class_name ASC LIMIT 10
```

```sql
ORDER BY avg_strength DESC LIMIT 10
```

where `avg_strength` is a projection alias for an admitted grouped
post-aggregate expression.

---

This slice **reuses 0.85/0.86 expression model**:

Allowed leaves:

* grouped keys
* aggregate outputs

Allowed composition:

* arithmetic
* scalar functions
* nested expressions

## Lowering changes

### Detect aggregate-based ordering

During lowering:

* inspect the full `ORDER BY` term list
* if any term contains aggregate references or other post-aggregate
  expression structure:

```text
→ mark plan as AggregateOrderTopK
```

Else:

```text
→ use canonical grouped ordering
```

---

### Plan representation

Introduce:

```rust
enum GroupedOrderMode {
    Canonical,
    TopK {
        order_terms: Vec<GroupedOrderExpr>,
        retention_bound: usize,
    },
}
```

Where:

* `GroupedOrderExpr` is planner-owned
* executor preparation compiles those order terms onto the grouped evaluator
  boundary
* logical planning must not store executor-owned compiled IR directly
* each term must carry:

  * direction
  * null-ordering policy, or an explicit inherited global null-ordering rule

---

## Executor changes

### New execution path

Inside grouped executor:

```text
if order_mode == Canonical:
    stream results
else:
    run Top-K pipeline
```

---

### Heap representation

Each entry:

```rust
struct HeapEntry {
    order_key: Vec<Value>,
    group_key: Vec<Value>,
    row: GroupedOutputRow,
}
```

Heap comparator:

* respects ASC / DESC per term
* handles nulls consistently with SQL semantics
* falls back to canonical grouped key for deterministic ties

---

## Multi-column ORDER BY

Required in this slice:

```sql
ORDER BY AVG(x), MAX(y) LIMIT 10
```

Heap key becomes tuple, with canonical grouped key as the final deterministic
fallback after user-visible terms.

---

## Null handling

Define explicitly:

* NULLS LAST (default) or consistent with existing engine rules
* must match scalar ORDER BY semantics

---

## Continuation semantics

### Important constraint

Aggregate ORDER BY:

* **does NOT support continuation/cursors**

Reason:

* heap-based selection depends on full scan
* ordering is not prefix-stable

---

### Behavior

* cursor token must be rejected

---

## Interaction with HAVING

Execution order:

```text
group → finalize → HAVING → Top-K selection → emit
```

HAVING filters before heap insertion.

---

## Interaction with projection

Projection evaluated:

* after aggregate finalization
* after retention when feasible

Implementation rule:

* do not force full projection work for every finalized grouped row before
  heap insertion
* retain minimal order payload through the heap path and project retained rows
  after extraction when possible

Aggregate Top-K introduces a distinct ordering strategy, but it must reuse the
shared grouped post-aggregate evaluator and grouped fold/finalize pipeline.

No changes needed.

Already supports expressions.

Add rule:

if ORDER BY uses aggregate expressions:
    require LIMIT

> "aggregate-based ORDER BY requires LIMIT for bounded execution"

## Examples

### Valid

```sql
SELECT class_name, AVG(strength)
FROM character
GROUP BY class_name
ORDER BY AVG(strength) DESC
LIMIT 10;
```

```sql
SELECT class_name, MAX(level), MIN(level)
FROM character
GROUP BY class_name
ORDER BY MAX(level) - MIN(level)
LIMIT 5;
```

```sql
SELECT class_name, AVG(strength) AS avg_strength
FROM character
GROUP BY class_name
ORDER BY avg_strength DESC, class_name ASC
LIMIT 10;
```

---

### Invalid

```sql
ORDER BY AVG(strength)
```

(no LIMIT)

```sql
ORDER BY AVG(strength) LIMIT 10 OFFSET 5
```

(`OFFSET` remains out of scope for `0.88`)

---

## Guardrails

### 1. No implicit full sort

Never fallback to:

* add a full grouped-output sort stage after finalization

Top-K must operate on finalized grouped rows without introducing an unbounded
post-finalize sort stage.

---

### 2. No continuation

Top-K mode is not cursor-compatible.

---

### 3. Strict LIMIT requirement

Always enforce.

---

### 4. Separate execution paths

Do not mix canonical and aggregate ordering logic.

---

## Acceptance criteria

### Functional

* aggregate ORDER BY works with LIMIT
* supports scalar expressions over aggregates
* supports mixed grouped-key and aggregate term lists
* supports alias-based grouped aggregate ordering when the alias resolves to
  an admitted grouped order expression
* rejects missing LIMIT
* rejects `OFFSET`

---

### Architectural

* canonical ordering unchanged
* Top-K path isolated and explicit
* planner stores planner-owned grouped order expressions; executor compiles
  them during grouped runtime preparation
* deterministic tie-break behavior is explicit and tested
* Top-K reuses the shared grouped post-aggregate evaluator and grouped
  fold/finalize pipeline instead of forking a new evaluator path

---

### Performance

* `O(N log K)` Top-K behavior confirmed
* no full sort path introduced
