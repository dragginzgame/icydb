# 0.210 Exact And Resumable Bulk UPDATE

Status: active; MVP hard cut on 2026-07-15; exact, resumable Forward, and
stable-revision Verify contracts and focused closeout evidence completed on
2026-07-22

## Decision Summary

A bulk-update row limit can be either a safety assertion or a deliberate
selection window. Confusing the two creates false success: an operation meant
to update a complete collection may update only a bounded prefix and still
appear complete. Raising the limit only moves the collection size at which the
bug occurs.

0.210 keeps three explicit contracts:

| Contract | Caller meaning | Success meaning |
|---|---|---|
| Exact | Complete target must fit one call | Complete atomic commit |
| Prefix | One ordered window is enough | Only that window was processed |
| Resumable | Bounded progress is acceptable | Atomic batches, then stable full verification |

The caller selects the contract. The engine never falls back from exact to
prefix or resumable, and never reports suffix exhaustion as collection-wide
completion.

The MVP is intentionally narrow. Resumable execution uses the authoritative
primary-key scan only, has no row `RETURNING`, supports only fixed authored
assignments with independently batch-safe constraints, and restarts rather
than migrates across schema or engine upgrades.

### Non-Negotiable Invariants

1. Exact overflow changes zero bytes of row and index state.
2. One resumable mutating call commits one complete atomic batch or no batch.
3. A continuation cannot change the operation id, schema, scope, patch, or
   engine batch policy.
4. The initial continuation is durably custodied before the first target-row
   mutation.
5. `complete = true` requires a full clean verification sweep at one unchanged
   durable store revision.

~~~text
                  batch committed / scan advanced
             +---------------------------------------+
             |                                       |
             v                                       |
Prepare --> Forward -----------------------------> Forward
             |
             | authoritative keyspace exhaustion
             v
         Verify(revision, start)
             |
             | clean bounded page
             v
         Verify(revision, next_pk)
             |
             +-- revision changed ----------> Forward(start)
             |
             +-- needs-patch row found -----> Forward(start)
             |
             +-- clean exhaustion,
                 same revision -------------> Complete
~~~

## Context And Dependencies

IcyDB currently has:

- `update_many_atomic`, which atomically commits a caller-supplied
  single-entity batch;
- public bounded SQL `UPDATE`, whose deterministic `LIMIT` selects an
  intentional prefix;
- `partial_window` and admin pagination, which provide bounded live-state read
  windows.

An application promise such as "every token in this collection has the reset
tier" is not satisfied by updating
`partial_window(MAX_COLLECTION_EFFECT_ROWS)`. A collection with one more row
would retain stale state while the operation appeared successful.

0.210 builds on:

- [ATOMICITY](../../contracts/ATOMICITY.md), including full preflight before a
  commit marker and mechanical apply after it;
- [TRANSACTION_SEMANTICS](../../contracts/TRANSACTION_SEMANTICS.md), including
  single-entity atomic batches;
- [READ_ADMISSION](../../contracts/READ_ADMISSION.md), including the explicit
  partial-window contract;
- [0.204 SQL correctness
  evidence](../archive/0.204-sql-correctness-matrix-generative-oracles/0.204-design.md);
- [0.207 redo-only schema index
  publication](../0.207-redo-only-schema-index-publication/0.207-design.md);
- [0.208 exact composite
  contracts](../0.208-exact-composite-contracts/0.208-design.md), including exact
  recursive admission; and
- [0.209 temporal defaults and versioned row
  layouts](../0.209-temporal-defaults-and-versioned-row-layouts/0.209-design.md),
  including the accepted-first insertion policy and final row envelope.

Implementation remains sequenced after 0.209 so the new mutation ingress emits
only the final current row format and does not require a later default/layout
retrofit. The 0.207 publication model remains its atomic commit and recovery
foundation. The archived
[0.181 UPDATE design](../archive/0.181-update/0.181-design.md) is design input,
not a compatibility contract.

Canister serialization makes exact selection and staging sound inside one
no-await update call. It does not remove instruction, heap, marker, journal,
index, relation, or response limits. Work spanning messages is therefore a
live-state convergence protocol, not one transaction.

## Scope

0.210 adds:

- a bounded exact, single-entity update policy;
- a trusted, single-entity resumable convergence policy for journaled stores;
- current-form internal continuation state;
- stable-revision verification before resumable completion;
- a pre-1.0 hard cut of ambiguous and unbounded trusted bulk-update paths.

0.210 does not add:

- multi-entity transactions;
- resumable `INSERT`, `DELETE`, `UPSERT`, or schema mutation;
- background jobs, timers, async execution, or a durable engine job registry;
- caller-selected resumable batch sizes;
- public bearer continuations;
- resumable row `RETURNING`;
- arithmetic, append, row-relative, or nondeterministic patches;
- resumable changes to primary-key, scope-dependent, generated, managed,
  unique, exclusion-constrained, or relation-owned fields;
- compatibility decoders, aliases, or migration for old resumable tokens;
- a lasting application business fence.

## Authority

The authority order is:

1. accepted schema owns layout, field ownership, constraints, indexes,
   relations, and value semantics;
2. the validated structural update plan owns scope and patch meaning;
3. authoritative rows own current values;
4. the exact frozen key set or resumable primary-key checkpoint owns one call's
   progression;
5. the commit marker owns a staged atomic batch after the boundary;
6. the durable journal revision owns verification stability.

Generated `EntityModel` and `IndexModel` values remain proposal and test
convenience. Runtime execution never reconstructs accepted authority from
them. SQL is a frontend to the same catalog-native structural plan used by
non-SQL callers.

## Semantic Lanes

### Exact Update

The exact request carries `require_affected_at_most`. The name is intentionally
an assertion, not `max_rows` or `limit`.

Conceptually:

~~~text
execute_exact_update(
    plan,
    ExactPolicy {
        require_affected_at_most,
        returning_policy,
    },
) -> ExactUpdateReceipt
~~~

Execution is:

1. Validate the accepted entity, scope, assignments, positive caller
   assertion, engine exact-row ceiling, scan budget, and bounded `RETURNING`
   policy.
2. Scan in accepted primary-key order until the selector exhausts, finds
   `require_affected_at_most + 1` matches, or reaches a resource bound.
3. If cap plus one exists, return typed overflow before mutation. The engine
   need not count the remaining target.
4. Only after selector exhaustion proves the complete key set, freeze it and
   stage all row, sanitizer, constraint, index, relation, marker, journal, and
   `RETURNING` effects.
5. Commit the fully prepared batch through existing marker-owned atomic apply.

Caller row count and engine resource admission remain separate. A target
within the asserted row count may still reject before mutation because its
scan, encoded rows, indexes, relations, marker, journal, heap, instructions, or
response do not fit one call.

The initial exact implementation sets both the affected-row ceiling and the
independent authoritative scanned-key ceiling to 4,096. Each uses one extra
probe element to distinguish exhaustion from overflow without exposing a
partial result.

Exact success means the complete matching set for the serialized statement
call was included. Every overflow or other pre-marker failure changes nothing.

### Intentional Prefix Update

The maintained public bounded SQL update remains a distinct contract:

~~~sql
UPDATE Token
SET tier = 'default'
WHERE collection_id = ?
ORDER BY id ASC
LIMIT 100
~~~

Its `LIMIT` selects one deliberate ordered prefix. It makes no complete-set
claim and cannot lower into `require_affected_at_most`. No new prefix executor
is required by 0.210.

### Resumable Convergence Update

Resumable execution converges authored field values over several calls. Each
mutating call is atomic, but the operation as a whole is not. Other messages
may change the target between calls.

Lane B is not resumable SQL statement replay. It has a narrower meaning:

> For every current row matching a stable scope, converge a fixed set of
> authored fields to fixed target values, then prove that no row needs that
> patch during one stable full verification sweep.

## Resumable MVP Eligibility

A plan is admitted only when the engine proves all of the following:

- exactly one accepted entity in one journaled store;
- a durable monotonic store mutation revision;
- a deterministic structural scope predicate;
- a fixed patch of accepted literal or accepted-default values;
- canonical accepted primary-key ascending progression;
- no `OFFSET`, subquery, cross-entity dependency, or caller batch size;
- no assignment depending on a current row value;
- no assignment to primary-key, generated, managed, or internal fields;
- no assignment to a field participating in uniqueness, exclusion, relation
  ownership, or another constraint whose validity can depend on a different
  batch;
- assigned fields are disjoint from the transitive dependency closure of the
  scope;
- ordinary non-unique index maintenance is independently valid per batch;
- bounded continuation and receipt encoding;
- no required row-level callback, audit emission, or other observable side
  effect beyond the normal converged row and its maintained derived storage.

Scope dependency closure is accepted-schema-owned and transitive through
generated fields, computed expressions, normalized field paths, and any other
accepted structural predicate component. Unknown or incomplete evidence
rejects the plan. The executor does not guess and does not retain a permissive
fallback.

Rejecting unique, relation-owned, and global-constraint assignments is an MVP
eligibility rule, not a claim that they are impossible forever. It prevents an
operation from committing early batches before discovering a permanent
cross-batch conflict.

Lane A does not need these resumable restrictions because it freezes and
preflights the complete target before mutation.

## Prepare Before Mutation

The first-batch custody order is mandatory:

1. The application authenticates the request and allocates its operation id.
2. `prepare_resumable_update` validates the current plan and returns an initial
   `Forward(start)` continuation. It does not scan or mutate target rows.
3. The application durably stores the operation id, immutable request meaning,
   and initial continuation outside the target store's mutation revision.
4. Only a later `resume_resumable_update` call may perform the first target-row
   batch.

There is therefore always a previously persisted continuation before any
target mutation. If prepare or custody is interrupted, target state is
unchanged. If a later batch commits but the custody update is interrupted, the
application retains the prior continuation and can replay it safely.

The engine creates no job row. Application custody must reject operation-id
collisions and reauthorize every resume call. Public callers receive only the
application operation id, never raw proof-bearing continuation bytes.

## Forward Execution

The MVP always scans authoritative entity keys in canonical accepted
primary-key order. It does not use a secondary-index access route. This removes
route-order proofs, unbounded sort risk, and route-dependent omissions from the
first implementation. A future optimized route requires a separate measured
design showing globally correct primary-key progression.

For each key strictly after the checkpoint, the executor:

1. reads and validates the authoritative accepted row;
2. evaluates the stable scope;
3. if in scope, evaluates `needs_patch` over caller-authored target fields;
4. if no patch is needed, records the key as fully examined;
5. if a patch is needed, stages the complete after-image and all ordinary
   derived effects into the pending batch;
6. advances past that row only when its mutation is included in the batch that
   commits.

Packing is incremental and deterministic under the engine batch-policy
identity carried by the token:

- add the next complete row effect only if the batch remains inside every
  encoded row, index, relation, marker, journal, and response budget;
- stop before the first needs-patch row that would exceed a budget;
- commit the non-empty staged prefix;
- if the first needs-patch row alone cannot fit, return a typed single-row
  resource error without mutation or checkpoint advancement;
- bound clean scanning separately so a page can advance through nonmatching or
  already-converged rows without opening a marker.

The returned checkpoint is the last key whose status is unambiguous: it was
fully classified as no-work, or its complete mutation belongs to the committed
batch. A row deferred by packing remains after the returned checkpoint.

One Forward call therefore does exactly one of the following:

- commits one non-empty atomic batch and returns progress;
- advances a read-only checkpoint over a bounded clean page;
- proves authoritative entity-keyspace exhaustion and enters Verify;
- returns a typed no-progress error.

It never commits two batches in one call. It never treats page-budget
exhaustion as entity-keyspace exhaustion.

## Verification And Completion

Forward suffix exhaustion captures the current durable store mutation revision
after any batch marker has cleared and enters `Verify(revision, start)`.

Every Verify call:

1. compares the current revision with the captured revision;
2. if changed, returns `Forward(start)` with `revision_changed` as the restart
   reason;
3. otherwise scans one bounded primary-key page from its verification
   checkpoint;
4. if any scoped row needs the patch, returns `Forward(start)` with
   `residual_work` as the restart reason;
5. if the page is clean but not exhausted, returns the next Verify checkpoint
   with the same revision;
6. if the entity keyspace is cleanly exhausted, rechecks the revision and
   returns complete only when it is unchanged.

The final recheck keeps the proof boundary local and fail-closed even though a
normal canister call is serialized.

`complete = true` means:

> The complete authoritative entity keyspace was scanned while the target
> store's durable mutation revision remained unchanged, and no accepted row
> matching the scope needed an authored target value.

It does not mean that the whole operation was one transaction, that it updated
a start-of-job snapshot, or that a future write cannot invalidate the business
condition.

The MVP uses the existing journal append sequence plus persisted fold watermark
as the store-wide revision authority. Every committed row mutation in that
store must participate in that sequence. Journal folding must preserve its
monotonic meaning. Corruption, exhaustion, or any mutation bypass fails closed
and blocks completion. Heap stores reject the resumable MVP because their runtime
generation is not durable authority.

Store-wide revision invalidation is conservative. An unrelated write may
restart verification. Entity-local revisions and new persisted authority are
out of scope.

## Retry Semantics

The authored patch is effect-idempotent because rows already at all target
values are skipped.

- Before marker persistence, an error changes no target state and the same
  continuation is retryable.
- After marker persistence, guarded recovery completes the atomic batch.
- If the response or custody update is lost after commit, replaying the prior
  continuation encounters the committed rows as already converged and safely
  continues.
- Earlier completed batches are never rolled back by a later failure.

Exactly-once invocation and replay-stable row receipts are not promised.
The resumable MVP therefore has no SQL-style row `RETURNING` and no exact cumulative
changed-row count.

## Managed Fields

`needs_patch` compares only caller-authored target fields. A row already at
those values is skipped entirely; it does not receive a new `updated_at` value
or replay per-row write behavior merely because it matches the scope.

Rows that actually need the patch run normal sanitizer and managed-field logic
once in the batch that commits them. Managed time is batch-local, not fixed for
the whole job. This avoids an operation-wide timestamp in the token while
remaining response-loss safe: replay skips a row whose authored patch already
committed. If a later writer changes an authored field back, applying the patch
again is new convergence work and may legitimately update managed fields
again.

This is an explicit convergence contract, not an attempt to reproduce every
side effect of one ordinary SQL `UPDATE` over a historical snapshot.

## Continuation And Custody

The bounded current-form continuation contains:

~~~text
format version
operation id
target store and entity identity
accepted schema fingerprint
normalized scope fingerprint
fixed authored-patch fingerprint
phase
phase-specific primary-key checkpoint
captured store revision in Verify
engine batch-policy identity
~~~

It contains no row after-images, executable SQL, authorization, cumulative
count, stable-memory address, or server job pointer.

The codec is bounded, canonical, fallible, current-form only, and semantically
distinct from read cursors. Constructors and raw decode remain inside the
trusted boundary. Opaque encoding, CRC, or an unkeyed hash is not
authentication; public/generated endpoints must not accept the raw bytes.

Application custody follows:

1. load the prior token by authorized operation id;
2. call the engine once;
3. after success, replace it with the returned continuation or completed
   state.

Custody state must be outside the target store revision. Saving a Verify token
through that store would invalidate its own proof.

## Upgrade And Schema Lifecycle

Active resumable MVP operations do not migrate across an accepted-schema change
or an IcyDB engine upgrade. The application marks them `restart_required`,
abandons their old continuation, and prepares a new operation under current
authority. Earlier committed row batches remain and the new fixed convergence
operation safely skips them.

This is a deliberate pre-1.0 hard cut:

- only the current token version decodes;
- no legacy token aliases, translation, fallback, or anti-resurrection tests;
- schema fingerprint, token-format, or batch-policy mismatch rejects before
  row access or mutation;
- application upgrade handling clears or marks active operation control state
  before accepting resume requests.

Because the engine has no durable job registry, it cannot block schema changes
based on hidden active jobs. Restart is the only current lifecycle contract.

## API And Receipt Shape

Conceptually:

~~~text
prepare_resumable_update(operation_id, validated_plan)
  -> InitialContinuation

resume_resumable_update(operation_id, validated_plan, trusted_continuation)
  -> ResumableUpdateReceipt

ResumableUpdateReceipt
  phase
  keys_scanned
  rows_updated
  restart_reason
  continuation
  complete
~~~

Receipt fields are per-call:

- `keys_scanned` counts authoritative entity keys fully examined in this call;
- `rows_updated` counts rows committed by this call;
- `restart_reason` is absent unless Verify returned to Forward because of
  `revision_changed` or `residual_work`;
- `continuation` is present only while in progress and remains inside trusted
  custody;
- `complete` is true only under the verification rule above.

The MVP does not expose ambiguous `batch_examined`, scope-match counts,
rows-needing-patch counts, advisory cumulative totals, or row payloads.

## Application Business Fence

Database convergence does not enforce a lasting application invariant. Toko
must enforce its tier policy at ordinary write ingress.

The preferred Toko shape is a collection-owned desired tier generation:

- starting a reset establishes the desired values and current generation;
- new tokens are created at that generation;
- tier writes must conform to or explicitly supersede it;
- the bulk operation converges existing tokens to it;
- completion proves database convergence during the final stable interval.

The exact schema and endpoint policy are application-owned and outside 0.210.
If Toko chooses a temporary write rejection fence instead, that is also an
application decision. IcyDB must not imply that its completion receipt replaces
either fence.

## Hard-Cut Cleanup

Before closeout:

- every broad trusted update ingress selects exact, intentional-prefix, or
  resumable intent explicitly;
- ambiguous cap names and helpers are renamed or deleted;
- no external trusted path collects an unbounded update target;
- no write path uses `partial_window` as proof of complete mutation;
- no raw resumable continuation crosses a generated/public endpoint;
- no unbounded, silently partial, or compatibility fallback remains;
- tests whose only purpose is preserving removed behavior are deleted;
- active docs and diagnostics use the current three-contract terminology.

The maintained public deterministic `LIMIT` update stays because it is an
explicit prefix contract, not a legacy fallback.

## Implementation Plan

### Slice 1: Exact Contract And Ingress Hard Cut

- Re-audit broad update ingresses and focused row, byte, scan, marker, journal,
  response, and raw wasm baselines.
- Add `require_affected_at_most` and cap-plus-one exact selection.
- Route fully staged exact work through existing atomic batch apply.
- Require explicit intent at broad trusted ingresses and remove ambiguous or
  unbounded fallbacks.

Exit: exact success proves complete selection; overflow and every precommit
failure change nothing.

### Slice 2: Resumable Prepare And Eligibility

- Add the narrow eligibility proof and transitive scope dependency closure.
- Add the bounded current-form trusted continuation.
- Add read-only prepare and application-custody ordering.
- Reject heap stores, row `RETURNING`, global constraints, and unsupported
  lifecycle state.

Exit: an initial continuation can be persisted before any target mutation, and
every unsupported plan rejects before row access or mutation.

Implementation status: complete. Preparation reuses the canonical SQL update
normalization and accepted structural selector/patch lowering, then freezes
accepted patch payloads and emits one bounded current-format trusted token.
Accepted schema owns scope-field resolution and the global-write-constraint
fact. The current schema model has no row-dependent computed or generated
fields, so its transitive dependency graph contains only terminal stored-field
roots; unknown roots and unsupported expression variants still reject rather
than becoming an empty dependency set. Application operation-id uniqueness and
durable custody remain application-owned. Preparation remains a separate
read-only call, so the application can persist the initial continuation before
invoking resume.

### Slice 3: Forward Batches

- Add primary-key-only bounded scanning and `needs_patch`.
- Add deterministic incremental batch packing.
- Commit at most one atomic batch per call.
- Prove clean-page progress, oversized-single-row failure, interruption
  recovery, and old-token replay.

Exit: every successful Forward call commits one complete atomic batch or makes
bounded read-only progress.

Implementation status: complete. Resume rebinds target, schema, scope, fixed
patch, checkpoint key type, and batch policy before row access. It scans at
most 256 authoritative primary keys, stages at most 64 needs-patch rows, and
uses the canonical journal/marker codecs' exact length rules to commit the
largest fitting non-empty prefix through the existing atomic structural batch
owner. Deferred rows remain after the returned checkpoint. Marker failpoint
coverage proves same-token retry before marker publication and redo recovery
after marker publication.

### Slice 4: Verification And Closeout

- Add bounded Verify pages using the durable journal revision.
- Restart on revision change or residual work.
- Add Toko-shaped reference coverage and hard-cut cleanup.
- Record instruction, heap, marker, raw non-gzipped wasm, and complexity deltas.
- Add maintained SQL cases to the 0.204 evidence authority.

Exit: only a full clean stable-revision sweep returns complete, and no ambiguous
or unbounded legacy route remains.

Implementation status: complete. Forward exhaustion captures the durable
journal sequence and enters bounded Verify. Verify uses the same
scope-plus-fixed-patch predicate as Forward, restarts from Forward start on
revision change or residual work, and returns complete only after clean
exhaustion plus a final revision recheck. Accepted-schema changes and token or
batch-policy drift reject before row access. Generated callback profiles reject
application sanitizers/validators, and accepted schema rejects globally
constrained authored or update-managed fields so later batches cannot discover
a cross-batch conflict. The mutation-atomicity guard also rejects any generated
surface that exposes the raw continuation or its prepare/resume methods;
applications retain operation-id authorization and continuation custody.

### 0.210.2 Closeout Corrections

The independent `0.210.1` audit found two contract defects and six evidence or
hard-cut defects. `0.210.2` closes them without compatibility paths:

- resume requires the application operation id and compares it with the token
  before store recovery or row access;
- every journaled mutation reserves a representable post-commit revision and
  returns a typed exhaustion diagnostic before marker publication;
- the batch-policy identity is derived from all current compatibility inputs;
- the test-only unbounded update executor is deleted, and maintained update
  tests select Exact or canonical primary-key-ordered Prefix explicitly;
- generated-surface protection parses the actual generated Rust exports and
  asserts their complete signatures, rather than relying only on spelling
  searches;
- recurring guidance and recorded evidence distinguish direct tests,
  structurally inherited marker authority, and application-owned custody.

Direct focused evidence now includes operation-id mismatch before row access,
revision exhaustion with unchanged row/index/journal/marker state, the 64th and
65th candidate checkpoint boundary, effect-safe old-token replay, clean 256-key
pages, marker retry/recovery, multi-page Verify, and per-input batch-policy
identity drift. Exact marker/journal oversize at the public boundary, a first
resumable row too large to pack, and a real application process restart that
loses the response before updating external custody remain distinct unexercised
integration cases; this document does not claim that those cases have direct
tests. External durable custody is an application obligation enabled by the
prepare/resume split, not a fact the target-store engine can attest by itself.

## Focused Validation

Implementation uses focused package, target, and named-test selections only.
The user owns full workspace validation during the push workflow.

The closeout evidence target covers:

- exact: zero, at-cap, cap-plus-one, scan-bound, marker-bound, unique/relation
  failure, bounded `RETURNING`, heap, and journaled stores;
- prepare: interruption before custody, custody before first mutation,
  operation-id binding, documented application-owned collision policy, and

Direct state snapshots cover affected-row overflow, scan overflow, unique and
relation rejection, and revision exhaustion. Other Exact resource boundaries
reuse the same pre-marker staging owner but are not described as direct
public-boundary state-oracle tests. Resumable failpoint tests directly cover
retry before marker publication and redo recovery after publication; the wider
marker failpoint matrix is inherited structural evidence. Tests compare typed
errors or diagnostic codes, never strings.

## Performance And Complexity Budget

Exact work is `O(S + E)` for `S` scanned keys and `E` fully staged matches, plus
derived index and relation work. Its transient memory and marker size are
bounded by the complete admitted batch.

One resumable call scans a bounded primary-key page and stages a bounded batch.
Continuation memory is bounded by one primary key and token metadata. One quiet
verification sweep is `O(total entity keys)`; conflicting store writes may
cause conservative restarts.

Closeout records:

- scanned keys, updated rows, staging/apply instructions, transient heap, and
  encoded marker/journal bytes;
- representative no-index and indexed authored-field cases;
- exact cap, cap plus one, large multi-batch, response-loss retry, and
  multi-page verification;
- raw non-gzipped wasm bytes as primary size evidence, with gzip secondary;
- files touched, approximate production/test line delta, and whether owner
  shape became simpler, neutral, or more complex.

No query hot-path cost is added outside bulk-update entrypoints. The intended
owner shape is one accepted structural update plan, one atomic batch apply
owner, and two explicit completion policies. The MVP deliberately avoids a
generic transaction framework, route optimizer, scheduler, job registry, or
public token protocol.

### Closeout Measurements

The focused PocketIC audit converged 512 journaled rows through one prepare,
eight 64-row Forward batches, and two 256-key Verify pages. Prepare consumed
300,660 local instructions. Forward batches ranged from 46,286,272 to
48,005,155 instructions (47,101,864 average); Verify pages averaged 5,471,511
instructions. The operation scanned 512 keys in Forward, scanned 512 keys in
Verify, and updated exactly 512 rows. Every multi-message step remained well
below the maintained two-billion-instruction test ceiling. There is no previous
resumable contract with which to claim an improvement; these are absolute
bounded-cost evidence.

The full 64-row Forward marker-interruption fixture records 10,869 encoded
journal-batch bytes and 10,907 durable marker/control-slot bytes. The prepared
fixture continuation is 156 bytes, below its 2 KiB hard bound. The repository
does not expose a reliable peak-heap counter for this call shape, so no numeric
heap claim is made. The structural heap bound is instead explicit: at most 256
keys are scanned, at most 64 complete candidates are staged, and marker packing
retains only the largest exact fitting prefix.

## Final Rule

> If the complete target can be selected, fully staged, and proven to fit
> within one-call budgets, freeze it and commit it atomically. Otherwise,
> change nothing unless the caller explicitly chooses bounded, idempotent
> convergence.

business contract. It is never an implementation fallback for either form of
