**0.83 Serialization Boundary Cleanup**

## Thesis

`0.83` should not stop at deleting small CBOR seams.

The real remaining runtime owner is the CBOR-shaped structural field contract
under:

* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/cbor.rs`
* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/composite.rs`
* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key.rs`
* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs`

So the real `0.83` target is:

> define an IcyDB-owned structural binary v1 below `Value`, preserve bounded
> traversal, and port the structural field runtime one owner at a time.

That means:

* do **not** start at `Value`
* do **not** invent one universal `Value -> bytes` path
* do start at the walker layer under `Value`

---

## Problem statement

The runtime has already converged on the right model for one lane:

* scalar slot payloads are custom
* narrow
* cheap
* schema-aware

That lane is the success case and should remain the model.

The remaining problem is the structural lane.
Today it is no longer generic serde in the hot path, but it is still
fundamentally a hand-rolled CBOR-derivative structural grammar:

* bounded head parsing
* skip
* array walking
* map walking
* tagged-variant payload splitting
* wrapper payload encode/decode

That contract is consumed by:

* `storage_key.rs` for direct relation/key decoding
* `composite.rs` for schema-driven list/map/enum recursion
* `value_storage.rs` for externally tagged `Value` decode and shallow fallback

So if “CBOR is gone” is to mean anything real in `0.83`, the structural field
contract itself has to be replaced.

---

## Architectural position

### Keep `Value` as semantic IR

`Value` is still the correct runtime semantic representation.

It should remain the thing higher layers manipulate for:

* projections
* generic structural values
* mixed-type field materialization
* untyped fallback behavior where that still exists

But `Value` should **not** become the universal storage codec.

That would blur three separate concerns that the runtime currently keeps
separate for good reasons:

* schema-driven `ByKind` decode
* direct storage-key decode
* `FieldStorageDecode::Value`

### Keep the scalar binary lane as-is in principle

The scalar slot lane is already the model:

* custom binary
* explicit null/value envelope
* no generic `Value` routing
* no generic CBOR routing

Do not widen that lane back through generic `Value`.

### Replace the structural lane under `Value`

The new target is:

* scalar binary lane: unchanged in principle
* structural binary lane: new IcyDB-owned binary contract
* value envelope lane: still semantically `Value`, but encoded in the new
  structural binary rather than in a CBOR-shaped tagged envelope

---

## Scope

### In scope

* design and implementation of Structural Binary v1
* a new bounded walker layer parallel to the current structural CBOR walker
* owner-by-owner migration of:
  * `storage_key.rs`
  * `composite.rs`
  * `value_storage.rs`
* explicit read/write/version cutover plan for the structural field format

### Out of scope

* changing public `Candid` canister boundaries
* rewriting the scalar slot lane into a generic value codec
* inventing one universal `Value -> bytes` format for the whole runtime
* changing SQL semantics or query planning semantics
* broad row-storage redesign such as field-addressable stable storage

## Primary design goal

### Structural Binary v1

`0.83` should define an IcyDB-owned Structural Binary v1 that replaces the
current CBOR-shaped structural contract.

The format must preserve the walker capabilities that the current runtime
already depends on:

* parse one value head
* recover bounded lengths
* skip one value without rebuilding a generic tree
* walk arrays/maps in place
* split externally tagged variant payloads

Those are not optional conveniences.
They are the reason the current hand-rolled structural decoder exists at all.

---

## Format goals

Structural Binary v1 should support explicit forms for:

* null
* scalar wrapper payloads where needed
* list
* map
* enum
* relation/reference-compatible forms
* value envelope forms used by `FieldStorageDecode::Value`

The exact byte grammar can evolve during design, but it must satisfy explicit
structural constraints so the first implementation does not quietly recreate
CBOR.

Structural Binary v1 must not:

* use major-type plus additional-info encoding in the CBOR style
* allow indefinite-length containers
* allow multiple encodings for the same semantic value

Structural Binary v1 should:

* use a single-byte tag, or similarly small fixed tag space, per variant
* use fixed-width or explicitly length-prefixed containers
* guarantee canonical encoding, meaning one encoding per semantic value

Beyond those format constraints, it must satisfy these runtime properties:

1. bounded traversal
* no unbounded recursive allocation requirement
* no need to materialize a generic tree to inspect one child
* partial traversal must stay `O(depth)`, not `O(total structure size)`

2. fail-closed decode
* malformed bytes remain corruption, not best-effort recovery

3. schema-driven recursion
* `ByKind` composite decode stays keyed off declared `FieldKind`, not generic
  runtime guessing

4. direct key decode
* relation and storage-key paths must still decode directly to canonical key
  forms without rebuilding a generic `Value`

## Execution plan

### Phase A: format plus walker

First define the new structural binary grammar and add a parallel walker layer.

That walker should live beside the current CBOR owner and mirror its capability
class, for example under:

* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/binary.rs`

The walker is the real API surface of the format and should be treated as a
contract, not just a helper module.

It should expose a minimal primitive set equivalent to the current low-level
operations:

* `parse_head(bytes, offset) -> (tag, len, payload_offset)`
* `skip_value(bytes, offset) -> next_offset`
* `enter_list(...) -> iterator`
* `enter_map(...) -> iterator`
* `split_variant(...) -> (tag, payload_slice)`

All higher-level structural decoding must be expressible only through these
primitives.

That rule exists to prevent:

* ad hoc byte peeking in upper layers
* duplicated parsing logic
* format leakage upward into `storage_key`, `composite`, or `value_storage`

The goal here is not to port callers yet.
The goal is to establish the new traversal seam first.

### Phase B: narrow runtime lanes first

Port:

* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key.rs`

first.

Why first:

* narrowest owner
* highest-value direct decode path
* least generic surface
* proves that the new walker can support direct key extraction without
  materializing `Value`

This phase should also cover relation singleton/list handling that depends on
that direct key lane.

### Phase C: schema-driven composites

Port:

* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/composite.rs`

second.

That owner is the schema-driven recursive `ByKind` lane for:

* list
* map
* enum

This is the place where the design must preserve:

* declared `FieldKind` authority
* validate-only traversal
* fail-closed recursion

### Phase D: `Value` envelope lane

Port:

* `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs`

last.

This is intentionally last because it is the broadest and most compatibility
sensitive structural owner.

The design target here is:

* keep `Value` as semantic IR
* replace the current CBOR-shaped externally tagged envelope with an
  IcyDB-owned structural binary envelope

This should remain isolated as the `FieldStorageDecode::Value` owner, not
become the universal route for every field kind.

The critical constraint is:

* the `Value` envelope must not become a self-describing generic format

Instead:

* it must be structurally encoded using the same binary primitives as the rest
  of the structural lane
* it must not embed dynamic field names or arbitrary tag/value nesting outside
  the defined forms

In other words, `Value` must be reconstructed by interpretation over the owned
binary grammar, not by embedding a generic schema in bytes.

### Phase E: write path and version cut

Only after parallel read/write support exists should the runtime cut over the
structural field format.

That phase should include:

* explicit version/cutover story
* bounded migration behavior
* removal of the old structural CBOR reader after the cut

Write-path discipline during migration must be strict:

* new writes emit only Structural Binary v1
* old CBOR reads remain supported temporarily during the cut
* no dual-write path
* no runtime “choose either format” write logic

That keeps the transition one-way instead of creating a forked system.

Before `1.0`, this should still end in one active structural format, not a
permanent multi-version compatibility branch.

---

## Ordering rule

The migration order is:

1. walker
2. `storage_key`
3. `composite`
4. `value_storage`
5. write/version cut

This is the correct order because it follows the existing ownership boundaries
and keeps the broadest semantic surface for last.

Starting at `value_storage` first would be the wrong move:

* too broad
* too compatibility-sensitive
* too easy to accidentally invent a universal `Value` storage path

---

## Guardrails

### Guardrail 1: preserve partial traversal

The replacement format must preserve bounded skip/walk behavior as a
first-class requirement.

If the new structural binary requires full materialization just to inspect
children, then the rewrite has failed even if the bytes are “custom.”

### Guardrail 2: keep semantic and byte compatibility separate

`Value` compatibility and persisted-byte compatibility are not the same thing.

The runtime can preserve semantic behavior while changing the structural field
byte contract, but only through an explicit versioned cutover.

So the design must not be:

* “change `Value` and the bytes will follow”

It must be:

* “change the structural binary contract explicitly and preserve semantic
  behavior above it”

### Guardrail 3: do not widen scalar lanes

The scalar lane is already the narrow custom success case.

Do not regress it by routing more fields through generic `Value` serialization
just because the structural binary work is happening nearby.

---

## Acceptance criteria

### Functional

* the structural field runtime has a parallel non-CBOR walker
* `storage_key.rs` can decode through that walker without generic `Value`
* `composite.rs` can recurse through that walker while preserving declared
  `FieldKind` behavior
* `value_storage.rs` can eventually move to the new structural binary without
  becoming the universal storage path

### Architectural

* the remaining DB/runtime CBOR dependency is no longer the structural field
  contract
* `Value` remains semantic IR, not the single storage/wire authority
* scalar and structural lanes remain separate on purpose

### Verification

* bounded traversal behavior is covered by dedicated tests
* malformed structural bytes remain fail-closed corruption
* relation/storage-key paths still decode directly to canonical key forms
* by-kind list/map/enum recursion still validates without unnecessary
  materialization
* structural decode does not allocate proportional to full container size when
  only accessing a subset

### Size and performance

* raw wasm bytes should improve or remain flat
* no regression on structural read paths that depend on partial traversal
* any new binary walker must stay comparable or better than the current
  hand-rolled CBOR-shaped walker on targeted decode paths

> that order.**
