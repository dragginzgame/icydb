# 0.117 Expression Family Semantics Centralization

## Thesis

`0.117` should make planner-owned expression typing and classification the sole
authority for expression family semantics, just as `0.116` makes planner-owned
logic the sole authority for truth-condition semantics.

This is not a feature-expansion slice.

It is not a grouped-semantic widening slice.

It is not a prepared widening slice.

It is not an executor redesign slice.

It is a semantic-authority contraction slice whose job is to ensure that
expression family meaning is defined exactly once and then consumed
consistently across lowering, predicate adaptation, planner, prepared
execution, runtime, and explain.

`0.114` removes prepared fallback as a semantic owner.

`0.115` starts the broader expression-pipeline contraction.

`0.116` centralizes truth-condition semantics inside that pipeline.

`0.117` should continue the same line by centralizing the remaining
expression-family semantics that are not specifically “truth”:

* `CASE` result typing and propagation
* function-family classification
* planner-owned result-family classification and propagation for admitted
  non-truth expression families
* remaining expression rewriting outside planner

Inside `0.117`, both the `CASE` lane and the bounded function-family lane
belong to the same design line.

The first bounded contraction should be:

* `CASE` result typing and propagation

Function-family classification remains part of `0.117`, but only as a
follow-on contraction inside the same line if the initial `CASE` slice stays
local and audit signals remain favorable.

---

## Why `0.117` Exists

After `0.116`, the truth-condition lane should be planner-owned, prepared
semantics should still remain bounded and structurally cleaner, and the next
remaining semantic distribution pressure should sit in expression family
meaning more broadly.

That gives the system this shape:

| Layer / Semantic Family | Status |
| ---- | ---- |
| prepared semantics | bounded and structurally cleaner |
| truth-condition semantics | centralized by `0.116` |
| expression family semantics | still partially distributed |

That makes `0.117` the natural continuation of the expression-pipeline cleanup
line.

It is not a pivot into a new subsystem.

It is the next contraction step in the same semantic-owner sequence.

## Problem

Even after truth semantics centralization, the system can still have one
remaining distributed semantic lane:

* function-family identity
* result expression family identity
* `CASE` result typing and propagation
* expression rewriting or semantic shaping outside planner

That means the broad shape can still read like:

* lowering builds structure and still carries some expression-family decisions
* predicate helpers still touch expression families adjacent to planner
* planner owns canonical `Expr`, typing, and family classification
* executor consumes canonical expressions, but may still reflect a broader
  expression-family surface than ideal

Even if this remains correct, it creates:

* duplicated family checks
* repeated pattern matches across planner/predicate/lowering
* hotspot concentration in `type_inference`, `ast`, and adjacent helpers
* unnecessary blast radius for routine expression follow-through

The problem is not truth semantics anymore.

The problem is the broader lane of expression family meaning outside truth:

* what kind of expression this is
* what result family it yields
* how `CASE` results propagate through the canonical model
* whether non-truth expression rewrites still happen outside planner

---

## What “Expression Family Semantics” Means

For `0.117`, expression family semantics means the semantic rules that define
what kind of expression one canonical `Expr` node represents and what kind of
result it yields.

That includes:

### Included Semantic Families

* `CASE` result typing and propagation
* planner-owned result type-family classification
* planner-owned classification of admitted function result families
* semantic family identity used for planning and canonicalization
* function-family classification
* expression family identity used by planner, runtime, and explain
* remaining expression rewriting outside planner where the rewrite implies
  semantic family meaning

### Not The Focus Of This Slice

* truth-condition semantics already targeted by `0.116`
* output label shaping
* session/result rendering
* executor value formatting
* grouped semantic widening
* prepared template widening
* new SQL feature families
* executor redesign
* route planner redesign
* joins, subqueries, or relational surface expansion

So `0.117` is broader than the truth lane, but it is still narrower than
“all expression semantics at once.”

---

## Goal

Make planner-owned expression typing and classification the single semantic
authority for expression family semantics.

All adjacent layers must either:

* adapt expression-family inputs into planner-owned forms
* consume planner-owned expression-family outputs

They must not:

* reinterpret expression family meaning locally
* perform independent result-type or family classification
* encode `CASE` result propagation semantics outside planner-owned logic
* keep parallel expression rewrite ladders that imply semantic classification

---

## Core Invariant

Expression family identity and result-family propagation are defined exactly
once, in planner-owned expression typing and classification.

Everything else is:

* structural translation before planner
* structural consumption after planner

If an expression-family rule exists both in planner-owned expression logic and
in an adjacent layer, planner-owned logic must be the single authority.

Adjacent layers may construct or consume expression nodes, but they must not
independently classify function families, propagate `CASE` result families, or
encode result-type meaning locally.

Structural adaptation is allowed.

Independent expression-family reinterpretation is not.

---

## Ownership Boundary

`0.117` should make the owner split visible in code shape.

### Parser / Lowering Owns

* SQL-specific surface syntax
* raw structural expression construction
* trivial syntactic normalization needed to build planner inputs

Parser / lowering must not own:

* function-family semantic classification
* result-type semantics beyond structural lowering
* `CASE` result propagation meaning
* independent expression rewrite systems that imply semantic family meaning

### Predicate Layer Owns

* structural adaptation between predicate-shaped APIs and planner-owned
  expression families
* helper conversions that consume planner-owned expression results

Predicate must not own:

* independent expression family meaning
* local `CASE` result propagation semantics
* independent function-family classification
* result-type meaning for non-truth expression families

### Planner Owns

* expression AST
* type inference
* expression family classification
* result typing
* `CASE` result propagation semantics
* semantic equivalence of admitted non-truth expression families

This is the single semantic authority for expression family meaning.

### Executor Owns

* evaluation of already-canonical expression-family contracts

Executor must not own:

* expression-family reinterpretation
* local reclassification of function families

### Session / Prepared Owns

* binding
* substitution
* execution wiring

`0.114` and `0.116` should already keep these lanes structural-only.

`0.117` must preserve that.

## Concrete Actions

### 1. Map Expression Family Operations Across Layers

Inventory expression-family operations across:

* planner expression modules
* predicate modules
* expression-related lowering modules

For each operation, classify it as:

* planner-owned semantic truth
* structural adaptation
* mixed logic that must be split

Examples to inventory:

* function-family classification
* result-type propagation
* `CASE` result typing
* expression family checks duplicated outside planner
* expression rewrites outside planner that imply semantic family meaning

### 2. Centralize Function-Family Classification In Planner

Ensure there is:

* one planner-owned function-family classification path
* one result-type/family propagation path for admitted function families
* no parallel function-family meaning in predicate or lowering

### 3. Centralize `CASE` Result Typing And Propagation

Move all admitted `CASE` result typing and propagation semantics behind the
planner-owned expression model.

`CASE` truth behavior belongs to `0.116`.

`CASE` result-family typing and propagation belongs to `0.117`.

This is the first bounded contraction target inside the `0.117` line.

### 4. Simplify Predicate Ownership After Truth Centralization

After `0.116`, inspect
[/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs:1)
for what semantic work remains.

The target is:

* adapter/helper behavior remains
* expression-family semantic work leaves

### 5. Reduce Lowering Semantic Work In Expression Families

In expression-related lowering:

* keep parse/build structure
* keep trivial syntax adaptation
* remove deeper expression-family decisions where planner can own them instead

Lowering should produce planner inputs, not a second expression-family
semantic system.

### 6. Reduce Hotspot Branch Pressure

Target the remaining expression-family hotspot cluster:

* `type_inference/mod.rs`
* `expr/ast.rs`
* any remaining semantic work in `bool_expr.rs`
* expression-family helpers in lowering

Use the ownership cleanup to reduce:

* duplicate family match ladders
* duplicated `CASE` propagation logic
* repeated function-family checks outside planner

### 7. Add Invariant Tests

`0.117` is not complete without tests that prove:

* expression family identity originates from planner-owned logic
* predicate and lowering do not diverge from planner-owned expression family
  classification
* `CASE` result typing and propagation do not regain owner-local semantics
  outside planner
* prepared and non-prepared execution preserve the same canonical expression
  family plan shape
* function-family semantics do not fork across planner, lowering, and runtime

## Recommended Work Order

### 1. Inventory The Remaining Expression-Family Surface

Map expression-family operations across:

* planner expression modules
* predicate modules
* lowering modules

Do not start by moving code blindly.

### 2. Start With `CASE` Result Typing And Propagation

`0.117` should start with `CASE` result typing and propagation as the first
bounded expression-family contraction.

That first step is the right place to prove:

* planner-owned result-family propagation
* narrower predicate/lowering participation
* lower blast radius than the broader function-family surface

Function-family classification remains inside `0.117`, but only as a follow-on
contraction if the initial `CASE` slice stays local and audit signals remain
favorable.

### 3. Narrow Predicate-Layer Responsibility Further

Once one family is planner-owned, simplify predicate-layer helpers so they
consume planner outputs instead of deciding expression-family meaning.

### 4. Only Then Centralize One Bounded Function-Family Seam

Once `CASE` result propagation is planner-owned and the corresponding
predicate/lowering duplication is removed, centralize one bounded
function-family classification seam if it still fits inside the same locality
budget.

This keeps `0.117` from turning into two large semantic slices at once.

### 5. Remove Remaining Lowering-Side Semantic Classification

After predicate ownership narrows further, move any remaining expression-family
classification out of lowering.

### 6. Freeze The Boundary With Invariant Tests

Treat the invariant tests as the contract that keeps distributed
expression-family semantics from returning.

## Exit Condition

`0.117` is done when:

* planner is the only semantic authority for expression family meaning
* predicate and lowering no longer duplicate expression family classification
  or result typing
* `CASE` result typing and propagation are planner-owned
* the remaining expression-family hotspot cluster shows lower branch pressure
  than the pre-slice baseline
* expression-family follow-through touches fewer modules than the pre-slice
  baseline
* semantic equivalence of admitted expression-family forms is enforced
  consistently across surfaces

This should be visible as:

* lower hotspot pressure in complexity accretion
* fewer semantic-adjacent duplication families in DRY
* tighter expression-family owner counts in CSA
* lower median blast radius for expression follow-through slices in velocity
* a narrower and more clearly bounded expression-pipeline seam in completeness

## Summary

`0.117` is the natural continuation of the expression-pipeline contraction line
after `0.116`.

It does not target truth semantics again.

It targets the remaining broader lane of expression family meaning:

* function-family classification
* result expression semantics
* `CASE` result typing and propagation
* remaining expression rewriting outside planner

The intended end state is simple:

* planner owns expression family meaning
* predicate and lowering adapt into that authority
* executor and prepared/session consume it
* the expression pipeline stops distributing semantic family meaning across
  multiple adjacent layers
