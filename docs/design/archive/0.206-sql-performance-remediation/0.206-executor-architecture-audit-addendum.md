## Purpose And Relationship To The Main Design

This addendum records the executor-architecture audit performed after the
bounded expression-order work exposed and removed the old cached/plain row
split. It exists so the audit can guide later executor consolidation without
rewriting the active
[0.206 design](0.206-design.md), its frozen performance-family selection, or
its completed investigation and closeout evidence.

The main 0.206 design remains authority for:

- the selected SQL performance family;
- its eliminated-work and instruction budgets;
- comparable workflow evidence;
- raw non-gzipped wasm evidence; and
- final baseline selection and closeout.

This addendum is authority only for the architectural findings and bounded
follow-up sequence documented here. It does not:

- reopen the 0.206 leading-set or `OptimizeNow` selection;
- relabel existing findings or benchmark evidence;
- require the current design, report, or findings ledger to be restated;
- convert every audit observation into implementation work;
- invent a new patch-number plan; or
- broaden 0.206 into an executor rewrite.

When one of these architectural slices is implemented, its status and evidence
may be appended here. The closed 0.206 documents change only if later evidence
proves that a claim they already own is inaccurate.

### Current Bounded-Order Baseline

The current `PendingOrderRows<R>` design is accepted as the architectural
baseline for this addendum.

Its plain and cached variants own a real invariant:

- plain rows are valid only when no cached expression-order values are needed;
- cached rows retain the exact resolved order, keep count, rows, and evaluated
  order values needed by final ordering; and
- consumers fail closed when they request plain rows from cached state or apply
  a different order contract.

This is a successful simplification. It removes optional auxiliary caches,
parallel row/key collections, and the previous discard-then-reevaluate flow.
None of the slices below should flatten it back into booleans, empty vectors,
or optional side data.

### Structural Scale

The audited working tree contained:

- 318 executor Rust files and 103,517 lines including tests;
- 256 production Rust files and approximately 61,282 production lines;
- 422 production struct or enum declarations;
- 2,087 production functions;
- six executor-local traits; and
- 61 root executor re-export declarations.

Large modules are pressure indicators, not automatic findings. In particular,
physical access, mutation commit, covering reads, save validation, and ordering
own real correctness or resource invariants. Consolidation is recommended only
where the call-site trace proves duplicated authority or invalid intermediate
state.

## Executive Assessment

The executor is **fundamentally coherent but locally overcomplicated**.

It is not carrying several independent execution engines and does not need a
broad authority redesign. SQL and fluent/typed entry paths converge after
planning on the same prepared plans, route planner, ordered key streams, row
materialization kernels, grouped reducers, aggregate terminals, and mutation
commit window. Fast paths generally remain subordinate to a canonical
fallback.

The remaining complexity is concentrated in local authority leaks:

- prepared access facts are derived again downstream;
- deterministic lowering failure is encoded as empty collections plus flags;
- one residual-retry policy has two controllers;
- scalar route policy is represented by combinable enum, option, and boolean
  fields;
- scalar/grouped mode is carried beside enums that already encode the mode;
- a wrong structural row representation can become a successful empty result;
- grouped diagnostics duplicate the semantic execution spine; and
- several small handoff types split related data and reconstruct it immediately.

The right response is a focused consolidation line made of small hard cuts.
The audit does not support a rewrite, a new executor framework, or a generic
operator architecture.

## Ranked Findings

### A1. Prepared Access Lowering Is Not Authoritative

Severity: high

Confidence: high

Disposition: implementation complete; comparable instruction evidence pending

Evidence:

- `prepared_execution_plan/core.rs::build_prepared_execution_plan_core_with_schema_fingerprint`
  lowers access and keeps encoded prefix/range specifications;
- `aggregate/execution.rs::PreparedAggregateStreamingInputs::lowered_access`
  performs the same schema-sensitive lowering again;
- aggregate callers include `aggregate/mod.rs`, `aggregate/terminals.rs`,
  `aggregate/count_terminal.rs`, and `aggregate/numeric/mod.rs`;
- `session/sql/execute/direct_count.rs` lowers a cached prepared logical plan
  again to derive direct-count cardinality specifications; and
- prepared residents carry `index_prefix_spec_invalid` and
  `index_range_spec_invalid` beside empty arrays, then map both failures through
  cursor-invariant errors.

A COUNT journey can lower the same access plan during prepared construction,
terminal strategy/window preparation, and final fast or fallback execution.
The repeated work is deterministic and does not belong to the terminal.

Required invariant:

- the encoded prefix/range specs correspond exactly to the accepted schema,
  entity tag, and immutable logical access plan; and
- invalid lowering fails before execution with its typed cause preserved.

Canonical owner and simpler shape:

- prepared-plan construction lowers once and returns an error immediately;
- consumers borrow `plan.access.executable_contract()` and use the retained
  encoded specifications;
- cached SQL COUNT derives its cardinality artifact from prepared residents;
- delete the invalid flags, empty sentinels, downstream checks, and aggregate
  `lowered_access()` helper.

Risk and proof:

- semantic risk is low, but the failure point and error classification change;
- prove PK, range, composite prefix, multi-lookup, COUNT, EXISTS, numeric,
  extrema, cached SQL, and fast/fallback parity;
- compare focused instruction evidence to confirm lowering work was removed;
  and
- record raw non-gzipped wasm with the implementing slice.

### A2. Residual Retry Has Two Semantic Controllers

Severity: high

Confidence: high

Disposition: implementation complete; comparable instruction evidence pending

Evidence:

- `kernel/mod.rs::ResidualRetrySession` owns structural-page probe, widening,
  unbounded fallback, and metric merging;
- `kernel/mod.rs::KernelRowsResidualRetrySession` repeats that loop for SQL
  aggregate row sinks; and
- the kernel-row controller constructs a fake empty `StructuralCursorPage` to
  call the page-oriented retry decision.

Required invariant:

- page and kernel-row terminals choose the same fetch sequence and accumulate
  rows-scanned, rejection, DISTINCT, and optimization metrics identically;
- only the latest attempt's semantic output survives.

Canonical owner and simpler shape:

- one kernel-local retry controller operates on an attempt-stat view;
- output-specific closures materialize and replace page or row payloads;
- delete `KernelRowsResidualRetrySession` and the fake page.

This must remain a local policy contraction, not a reusable retry framework.

### A3. Structural Page Extraction Fails Open

Severity: high

Confidence: high

Disposition: implementation complete

Evidence:

- `pipeline/contracts/execution/inputs.rs::StructuralCursorPage::data_rows`
  returns an empty slice for retained slot rows; and
- `into_data_rows_and_cursor` returns an empty vector for the same mismatch.

Typed loads, aggregate materialization, bytes terminals, and ranking terminals
call these accessors. A representation-routing defect can therefore appear as a
successful empty result.

Required invariant:

- data-row-only terminals never consume retained-slot payloads; and
- SQL projection consumers explicitly dispatch over either representation.

Canonical owner and simpler shape:

- keep the payload enum;
- replace silent accessors with fallible `require_data_rows`-style extraction;
- retain `consume_projection_rows` for representation-polymorphic SQL
  projection.

### A4. Scalar Route Preparation Encodes A Hidden State Machine

Severity: medium

Confidence: high

Disposition: implementation complete; focused instruction evidence pending

Evidence:

- `ScalarPreparedRuntimeOptions` combines `ScalarRoutePlanFamily`, an optional
  prebuilt route, `ScalarPlanValidationMode`, cursor emission, unpaged mode,
  projection mode, and hint suppression;
- initial and resumed constructors currently create valid combinations, but
  the type admits combinations no caller intends;
- route scan hints are cleared during runtime preparation and again before
  execution; and
- the materialized path can later add an index-set fetch hint.

Canonical owner and simpler shape:

- use variant-owned initial/resumed route input;
- validate authority at one prepared boundary;
- apply all terminal-specific hint changes in one final route-normalization
  step immediately before `ExecutionInputs` construction;
- delete the validation mode and route-family/optional-route pairing.

The slice must preserve cursor bounds, top-k lookahead, residual retry safety,
and any intentionally safe index-set hint.

### A5. Grouped Diagnostics Duplicate Normal Execution

Severity: medium

Confidence: high

Disposition: implementation complete

Evidence:

- normal grouped execution lives in
  `pipeline/entrypoints/grouped.rs::execute_grouped_route_path`;
- the diagnostics branch in
  `execute_prepared_grouped_route_runtime_internal` repeats stream build, fold,
  aggregation timing, metric application, trace attachment, and finalization;
- the phase-attributed outer entrypoint also repeats cursor, handoff, route,
  and runtime preparation.

Canonical owner and simpler shape:

- the normal grouped spine emits cfg-elided phase observations;
- diagnostic entrypoints invoke the same preparation and execution once;
- delete the attributed semantic branch and repeated outer setup.

Non-diagnostic builds must retain no meaningful observation overhead.

### A6. Load And Materialization DTO Ladders Re-encode Existing State

Severity: medium

Confidence: high

Disposition: implementation complete

Evidence:

- `PreparedLoadRouteRuntime` already owns the scalar/grouped distinction;
- execution then produces `LoadExecutionPayload`, pairs it with a separate
  `LoadSurfaceMode` in `LoadPayloadState`, validates the pair twice, and
  rebuilds `LoadExecutionSurface`;
- `RuntimePageMaterializationRequest` is copied field-for-field into
  `KernelPageMaterializationRequest` after authority lookup; and
- the row-collector request omits the already-derived consistency contract and
  recomputes it from the plan.

Canonical owner and simpler shape:

- prepared runtime variants construct `LoadExecutionSurface` directly;
- delete `LoadExecutionPayload`, `LoadPayloadState`, `apply_paging`, and repeat
  mode/payload validators;
- have the runtime adapter construct the terminal request directly; and
- pass the canonical consistency contract to the row collector.

`ScalarMaterializationCapabilities` and terminal scan requests remain useful
because they freeze real materialization and row-runtime policy.

### A7. Cursorless Paging Splits And Reconstructs Window State

Severity: medium

Confidence: high

Disposition: implementation complete

Evidence:

- `CursorlessShortPathPlan` stores keep cap, skip count, and a post-scan window
  strategy;
- it converts `row_skip_count != 0` into
  `page_window_already_applied`; and
- the post-scan tail infers the answer again from the strategy, retained-row
  mode, page, offset, limit, and current row count.

Canonical owner and simpler shape:

- the cursorless scan owns the complete limit/offset window;
- the post-scan tail validates projection and shapes the payload only;
- delete `CursorlessRetainedWindow`, the boolean, the redundancy heuristic,
  and the second page-window application.

Limit zero, offset-only, offset-beyond-input, missing-row, and retained-slot
cases require focused proof.

### A8. Grouped Prepared State And Route Facts Can Have Mixed Provenance

Severity: medium

Confidence: high

Disposition: implementation complete

Evidence:

- `PreparedGroupedRuntimeHandoff` carries execution preparation and grouped
  slot layout as two independent options;
- producers emit both or neither, while the consumer independently recomputes
  either missing value;
- `GroupedRoutePayload` wraps only an `ExecutionRoutePlan`; and
- `GroupedRouteStage` separately copies direction and grouped execution mode
  from that route.

Canonical owner and simpler shape:

- carry `Option<PreparedGroupedRuntimeResidents>` or a `Prepared | Compile`
  variant so preparation and layout move or rebuild together;
- store the route plan directly and read direction/mode from it;
- delete the two independent options, single-field route wrapper, and copied
  route facts.

### A9. Two Small Split-And-Rejoin Shapes Should Be Removed

Severity: medium

Confidence: high

Disposition: implementation complete

`ResolvedExecutionKeyStream::into_stream_resolution_fields` produces a
six-element tuple that DISTINCT decoration immediately uses to reconstruct the
same type. Replace it with an owner-local stream-decoration method that
preserves optimization and counter metadata.

`GroupedPageCandidateRanking::TopK` stores evaluated order values and
directions in parallel vectors, even though compiled order terms already pair
expression and direction. Replace them with one vector of paired resolved
terms. Preserve canonical group-key tie-breaking.

Implementation evidence (2026-07-20):

- `ResolvedExecutionKeyStream::decorate_key_stream` now keeps optimization,
  scan, predicate, and DISTINCT-counter metadata inside the canonical stream
  owner while the stream itself is decorated;
- grouped top-k candidates now carry one vector of resolved value/direction
  terms, so values and comparison directions cannot be separated or have
  different lengths;
- focused DISTINCT counter, bounded grouped top-k, mixed multi-term order, and
  grouped-key tie-break checks pass;
- all-target/all-feature Clippy, no-default native and wasm checks,
  all-feature wasm checks, and repository architecture invariants pass;
- the slice footprint is three files, 54 additions, and 65 deletions (net -11
  lines), and both affected representations are simpler; and
- the raw non-gzipped SQL performance Wasm is 3,907,647 bytes, 181 bytes below
  the Slice F subject and 13,390 bytes below the accepted 3,921,037-byte 0.206
  subject, with SHA-256
  `f9ee21eb09a00ec02ff28d0ece0fc3ed77f188e06533c3bcf698c726c825e373`.
  Its secondary gzip size is 1,243,335 bytes. Comparable focused instruction
  evidence remains required from the normal pushed performance workflow.

### A10. SQL Direct COUNT Is A Second Aggregate-Preparation Path

Severity: medium

Confidence: high that policy is duplicated; medium that removal is beneficial

Disposition: retain deliberately in 0.206

SQL builds and caches `DirectCountCardinalityTarget` before the ordinary global
aggregate plan. If the direct attempt cannot answer, the prepared aggregate
terminal independently derives a cardinality, existing-row, or kernel
strategy.

The early path may be justified because it can avoid full planning for hot
COUNT statements. It must not be removed merely for architectural symmetry.

If retained after measurement:

- one executor-owned count-cardinality artifact and eligibility function
  serves early SQL and prepared aggregate execution;
- SQL caches the artifact but does not own a second semantic dispatcher; and
- cold/hot planning and execution evidence proves the early path's value.

Retention evidence (2026-07-20):

- the accepted three-run calibration records warm exact-cardinality SQL counts
  at 725-741 planner instructions and about 17,700 executor-invocation
  instructions;
- the maintained non-indexed `COUNT(*) WHERE active = true` fallback records
  192,008 warm planner instructions and 494,326 executor-invocation
  instructions, so forcing every count through prepared planning would erase a
  material shortcut;
- the SQL path owns only the pre-planning candidate/cache decision and response
  adaptation; physical prefix-cardinality execution and page-window semantics
  already converge in the executor count terminal; and
- moving the SQL cache DTO into the executor would not remove a semantic path:
  it would move a compiled-command cache concern across the layer boundary.

The original consolidation suggestion is therefore rejected for 0.206. The
early path is a measured planning boundary, not historical duplication. Any
future change must beat both cold and warm exact-cardinality sentinels through
the ordinary pushed performance workflow.

### A11. Hash COUNT Duplicates Generic Grouped Page Finalization

Severity: medium

Confidence: high that mechanics overlap; medium that consolidation will stay
cheap

Disposition: retain deliberately in 0.206

The dedicated hash COUNT reducer is justified by borrowed-key hashing,
allocation behavior, and focused metrics. Its HAVING, continuation, bounded
candidate selection, order, offset/limit/lookahead, projection, and cursor tail
overlaps generic grouped finalization.

Share only a lightweight candidate-selection/finalization tail if benchmarks
show no regression. Do not replace the COUNT reducer with the generic reducer,
box hot iterators, or erase the ordered/hash resource distinction.

Retention evidence (2026-07-20):

- existing 0.206 evidence proves the dedicated COUNT fold keeps ordered
  execution to one live group/state while consuming the rows required for
  exact results;
- the hash COUNT reducer's borrowed-key and allocation behavior remains a
  distinct hot-path invariant; and
- no checked-in comparison demonstrates that introducing a shared
  finalization abstraction reduces instructions, allocations, or Wasm.

The overlap remains documented, but it is not implementation work for 0.206.
Adding an unproven shared tail would increase abstraction precisely where the
audit requires performance proof.

Confidence: high

Disposition: duplicate DELETE count check complete; remaining constraints are

## Ordered Design Slices

### Slice A: Single Prepared Lowering Authority

Theme: make one prepared artifact travel forward.

Ownership transfer:

- prepared-plan construction owns schema-sensitive lowering and failure;
- aggregate and cached SQL consumers use the retained specifications and the
  borrow-based executable access contract.

Hard-cut deletion:

- invalid flags and empty sentinels;
- aggregate `lowered_access()`;
- downstream invalid checks and cursor-error conversion; and
- re-lowering of an already-prepared cached SQL plan.

Preserved semantics:

- all access shapes, accepted-schema authority, fast/fallback parity, cursor
  formats, and aggregate results.

Explicit exclusions:

- do not redesign or remove the early pre-plan SQL COUNT shortcut here.

Validation:

- focused aggregate and access-shape matrices;
- invalid prefix/range failure classification;
- cached SQL COUNT parity;
- instruction and raw non-gzipped wasm deltas.

Stop condition:

- no schema-sensitive access lowering remains downstream of an already-
  prepared plan.

Implementation evidence (2026-07-20):

- prepared-plan construction now returns the index-owned typed failure when
  prefix or range lowering fails, so no invalid flags or empty-spec sentinels
  can enter execution;
- aggregate terminals consume the prepared logical executable contract and
  retained specifications, while cached SQL `COUNT` derives its cardinality
  proof from the cached shared prepared plan;
- the early accepted-authority SQL `COUNT` shortcut remains deliberately
  unchanged;
- source guards confirm that the only executor call to
  `lower_access_with_schema_info` is in prepared-plan construction;
- focused access-failure, prefix-cardinality authority, aggregate-optimization,
  cached SQL/fluent `COUNT`, and direct/fallback parity checks pass;
- all-target/all-feature Clippy, no-default native and wasm checks,
  all-feature wasm checks, and repository architecture invariants pass;
- the production-and-focused-test footprint, excluding the pre-existing
  bounded-order test edit, is 25 files, 152 additions, and 359 deletions
  (net -207 lines), so the implementation shape is materially simpler; and
- the raw non-gzipped SQL performance Wasm is 3,912,824 bytes, 8,213 bytes
  below the accepted 3,921,037-byte subject, with SHA-256
  `46b9a99850dd4092771086b210185f1de13ace8aed407a4194a78464df2d9291`.
  Its secondary gzip size is 1,245,191 bytes. A comparable focused instruction
  measurement remains required before this slice's performance evidence is
  closed; the implementation does not infer an instruction delta from the
  source or size reduction.

### Slice B: One Residual Retry Controller

Theme: one semantic policy with output-specific mechanics.

Ownership transfer:

- the kernel owns one probe/widen/fallback loop and one decision function.

Hard-cut deletion:

- `KernelRowsResidualRetrySession` and the fake page projection.

Preserved semantics:

- lookahead, bounded caps, scan accounting, optimization labels, rejection and
  DISTINCT counters, and latest-attempt output.

Explicit exclusions:

- no residual-expression, route-planner, or adaptive execution redesign.

Validation:

- page/kernel-row decision parity for index-range and top-n routes at zero,
  bounded widening, and unbounded fallback cases.

Stop condition:

- one retry loop serves both terminal payload families.

Implementation evidence (2026-07-20):

- `ResidualRetrySession::materialize` is now the only probe, widen, stop, and
  unbounded-fallback loop for structural pages and aggregate kernel rows;
- the payload families supply only their attempt materializer and payload
  split/rebuild mechanics through a kernel-private trait;
- both attempts carry the same `ExecutionOutcomeMetrics` contract, whose
  retry merge keeps only the latest semantic output and terminal state while
  saturating rows-scanned, rejected-key, and DISTINCT counters across attempts;
- `KernelRowsResidualRetrySession`, its duplicated loop and merge policy, its
  unbounded-fallback wrapper, and the fake empty `StructuralCursorPage` are
  deleted;
- focused index-range and top-n widening, fallback parity, aggregate

### Deliberately Retained Performance Boundaries

0.206 retains:

- the early SQL exact-cardinality path because accepted cold/warm evidence
  proves a material planning benefit; and
- the specialized hash COUNT finalization path because no evidence justifies
  adding a shared abstraction to that hot boundary.

Neither is an outstanding 0.206 implementation task. A future release may
reopen either only with a bounded candidate implementation and ordinary
performance comparison.

## Things Not To Change

- Do not undo or flatten `PendingOrderRows`.
- Do not merge row representations merely to reduce type count.
- Do not replace scan-strategy specialization with per-row generic dispatch.
- Do not merge ordered streaming and hash-materialized grouping.
- Do not remove dedicated grouped COUNT ingestion without performance proof.
- Do not collapse access-key and projected-row DISTINCT.
- Do not remove covering, hybrid-covering, or fully materialized projection
  paths where payload availability genuinely differs.
- Do not weaken bounded heaps, read admission, scan budgets, lookahead, group
  limits, or output limits.
- Do not simplify mutation commit markers, overlay preflight, generation
  guards, journaling, rollback, or the single-row commit optimization merely
  because `commit_window.rs` is large.
- Do not change persisted rows, schema snapshots, cursors, public responses, or
  cache fingerprints in these slices.
- Do not fold explain rendering into execution; explain already uses the
  canonical route planner.
- Do not perform a broad executor re-export or module-layout cleanup.

## Cross-Cutting Validation And Acceptance

Every implemented slice must preserve:

- SQL/fluent and prepared/non-prepared result parity;
- fast/fallback parity;
- ordering, null, tie-breaker, cursor, offset, and lookahead semantics;
- accepted-schema and stable-row authority;
- statement atomicity and fail-closed mutation behavior;
- bounded public read admission and all resource limits;
- typed corruption and invariant failures; and
- diagnostic/non-diagnostic semantic parity.

Validation stays focused per slice. The user-owned full repository suite remains
part of the push workflow and is not run by implementation agents. Each slice
records:

- focused semantic checks;
- relevant instruction/performance evidence;
- raw non-gzipped wasm delta, with gzip only as secondary context;
- files touched and approximate line delta; and
- whether the implementation became simpler, stayed neutral, or became more
  complex.

No slice is accepted merely for reducing line or type count. It must delete
duplicate authority, remove an invalid state, or preserve a demonstrated hot-
path invariant more directly.

## Recommendation

Continue the focused executor consolidation line without reopening the closed
0.206 performance design or beginning a broad rewrite.

**Single Prepared Lowering Authority**, **One Residual Retry Controller**,
**Fail-Closed Materialization Contracts**, **One Final Scalar Route Owner**,
**Collapse Duplicate Load And Grouped Handoffs**, **Diagnostics Around The
Canonical Grouped Spine**, and the two A9 representation hard cuts are
implemented. The owner-local duplicate structural DELETE row-count checks are
also removed.

A10 and A11 are intentionally retained rather than left as ambiguous cleanup.
The accepted evidence proves that early SQL direct `COUNT` materially saves
planning work, while no evidence supports adding a shared grouped COUNT
finalization abstraction. Reopen either only in a later measured slice; there
is no remaining addendum implementation work required to close 0.206.
