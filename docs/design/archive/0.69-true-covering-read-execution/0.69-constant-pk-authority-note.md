# 0.69 Constant-PK Authority Note

## Purpose

This note answers one narrow question before any more stale widening work:

> can constant-plus-primary-key SQL short paths carry membership-level
> `storage_existence_witness` authority honestly, or should that family stay
> fail closed under the current kernel contract?

## Where The Witness Is Lost

The loss is not in the route classifier.

The route can still describe a covering contract through
`LoadTerminalFastPathContract::CoveringRead`, and the SQL terminal context
already retains the lowered index-bounds state in
`CoveringComponentScanState`.

The witness is lost at the generic ordered-key boundary:

- `runtime.rs` constant-plus-primary-key short paths
  - `try_materialize_sql_route_constant_projected_text_rows(...)`
  - `try_materialize_sql_route_constant_projected_rows(...)`
  - `sql_constant_covering_slot_row_template_from_route_contract(...)`
  consume only `&mut dyn OrderedKeyStream`
- `OrderedKeyStream` in
  `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/stream/key/contracts.rs`
  exposes only:
  - `next_key() -> Result<Option<DataKey>, InternalError>`
- the generic secondary scan path that feeds that stream comes from the plain
  key resolver in
  `/home/adam/projects/icydb/crates/icydb-core/src/db/index/scan.rs`
  - `resolve_data_values_in_raw_range_limited(...)`
  - `decode_index_entry_and_push(...)`
  which collapses one membership entry to `DataKey` and discards the
  per-membership `IndexEntryExistenceWitness`

By contrast, the covering paths that already handle stale witnesses honestly do
not go through `OrderedKeyStream`. They use witness-preserving contracts:

- `resolve_data_values_with_components_in_raw_range_limited(...)`
- `scan_single_component_covering_values_in_raw_range_limited(...)`
- `map_covering_projection_pairs(...)`

Those routes carry either:

- `(DataKey, IndexEntryExistenceWitness, components)`
- or `(StorageKey, IndexEntryExistenceWitness, single_component)`

all the way to the covering mapper before any authority-sensitive skip happens.

## Can The Current Constant-PK Path Carry Membership Witness End To End?

Not through the existing `OrderedKeyStream` contract.

That trait is a shared executor-wide key-stream boundary. It feeds:

- budget wrappers
- merge/intersect/distinct stream combinators
- row collectors
- terminal short paths that are not covering-authority-aware

Changing `OrderedKeyStream` itself from `DataKey` to
`(DataKey, IndexEntryExistenceWitness)` would leak covering-only stale authority
into broad generic load execution and force a much wider kernel rewrite.

So the answer is:

- `no` for the current `OrderedKeyStream` lane
- `yes` only if the kernel grows a sibling covering-specific witness-carrying
  stream contract

That still respects shared-kernel boundaries because the terminal SQL covering
lane already has the lowered index-bounds state needed to build such a stream
without route-local cheating.

## Narrowest Credible Kernel Contract Change

If this family is ever widened honestly, the narrowest credible change is:

1. Add one new shared covering stream contract parallel to `OrderedKeyStream`.
   Suggested shape:
   - `OrderedCoveringMembershipStream`
   - `next_membership() -> Result<Option<(DataKey, IndexEntryExistenceWitness)>, InternalError>`
   - optional exact-count hint, matching the existing budget contract style
2. Add one index-scan helper that streams membership witness without decoding
   covering components.
   Suggested shape:
   - `resolve_data_values_with_witness_in_raw_range_limited(...)`
   - or one direct streaming variant over `RawIndexEntry::iter_memberships()`
3. Let the SQL covering terminal lane consume that stream only for the
   constant-plus-primary-key covering family under
   `existing_row_mode = storage_existence_witness`.
4. Leave the existing `OrderedKeyStream` lane untouched for:
   - `row_check_required`
   - materialized row loads
   - generic non-covering execution

That is the smallest end-to-end authority path that does not infer witness from
ordered keys alone and does not add another route-local shortcut.

## Follow-Up Outcome

That shared-kernel witness path has now been implemented for the narrow stale
equality-prefix leading-component family:

```sql
SELECT id, priority
FROM CustomerOrder
WHERE priority = 20
ORDER BY status ASC, id ASC
LIMIT 2
```

The important boundary rule did not change:

- the generic `OrderedKeyStream` lane still cannot consume
  `storage_existence_witness` honestly
- constant-plus-primary-key ordered-key short paths still fail closed under
  `storage_existence_witness`
- the kept widening uses a separate shared covering-membership path that
  carries `(DataKey, IndexEntryExistenceWitness)` pairs end to end before any
  authority-sensitive skip happens

So the design note conclusion is now:

- do not widen this family on the plain ordered-key lane
- do widen only when the shared kernel can keep membership-level witness data
  alive end to end
- the same rule now covers both:
  - `SELECT id, priority FROM CustomerOrder WHERE priority = 20 ORDER BY status ASC, id ASC LIMIT 2`
  - `SELECT id, priority FROM CustomerOrder WHERE priority = 20 ORDER BY status DESC, id DESC LIMIT 2`
