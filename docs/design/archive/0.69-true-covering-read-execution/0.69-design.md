# True Covering Read Execution Mode

## Status

Implemented (Historical Baseline)

The core `0.69` design goal shipped in the `0.69.x` line:

* covering reads became a first-class execution route
* `existing_row_mode` became an explicit route/explain contract
* the first planner-proven and witness-backed cohorts shipped

This file remains the historical design record for that execution-mode work.

It should not be read as the active correctness model for index visibility.
That later question was closed in `0.70` under planner-gated index
visibility.

## Goal

Make planner-approved covering reads execute as a distinct index-only read mode so eligible scalar SQL and fluent load shapes can return results directly from index traversal output without resolving `DataKey`, reading row storage, or rebuilding generic row payloads.

`0.68` established the planner-owned covering-read contract, widened bounded index-backed ordered read coverage, aligned text-prefix and casefold range forms, and reduced some executor materialization overhead. `0.69` should finish the next architectural step: turn that planner contract into a dedicated execution mode instead of continuing to optimize the row-materialized path around it.

## Why `0.69` is the right next step

The remaining large fixed cost on the representative scalar read cohort is still inside the shared execute bucket, not in parse/lower/dispatch polish. The repo now already has the key ingredients needed to attack that bucket directly:

* a planner-owned `CoveringReadPlan`
* planner-owned order and access-window contracts
* bounded and explainable ordered index-backed read paths
* expression-index parity for accepted text prefix/range families
* narrower cursorless retained-slot projection lanes

What is still missing is a real execution boundary that can consume those contracts without falling back to row storage and generic row-shaping.

## Problem Statement

Today, many hot read shapes are already planner-recognized as coverable or almost-coverable, but the engine still pays for work that should become optional when the chosen index already contains the required values.

That remaining work includes:

* resolving traversal output to `DataKey`
* checking row storage for existence through the normal row-backed path
* decoding row payloads into generic slot/value structures
* rebuilding structural rows only to project values that already exist in index components or the primary key
* carrying generic materialization/page-window machinery through paths that only need projected scalar output

As a result, `0.68` improved route quality and reduced some materialization overhead, but it did not yet deliver a distinct index-only execution mode.

## User-Facing Outcome

Eligible reads such as the following should execute without row storage reads:

* `SELECT name FROM User WHERE name = 'alice' ORDER BY id LIMIT 1`
* `SELECT id, name FROM User WHERE name = 'alice' ORDER BY id LIMIT 1`
* `SELECT name FROM User ORDER BY name ASC LIMIT 1`
* `SELECT LOWER(name) FROM User WHERE LOWER(name) >= 'br' AND LOWER(name) < 'bs' ORDER BY LOWER(name) ASC LIMIT 2`
* fluent scalar or narrow structural loads whose projected outputs are fully covered by the chosen index plus primary key

Ineligible shapes continue to use the existing row-materialized path with unchanged semantics.

## Non-Goals

This design does not attempt to:

* add cost-based planning
* admit arbitrary computed SQL projections
* widen grouped, aggregate, or join execution into index-only mode
* change stale-row, missing-row, or existing-row semantics silently
* solve broad DISTINCT index-only execution in the same slice
* replace all retained-slot or structural row materialization surfaces
* introduce terminal-local covering heuristics outside planner-owned contracts

## Design Principles

### 1. Planner owns admission

Whether a query is eligible for index-only covering execution must be decided in planner-owned code. Executor/runtime must consume that decision, not rediscover it.

### 2. Covering execution is a route shape, not a late optimization

The route should say explicitly that the query is executing in covering mode. The executor should not infer that ad hoc from slot layouts or traversal output.

### 3. Semantics fail closed

If preserving current semantics requires row presence checks, row payload inspection, or unsupported reorder logic, the planner must keep the query on the row-materialized path.

### 4. Existing order/window contracts stay authoritative

`0.69` should reuse the current planner-owned order and access-window contracts rather than inventing parallel executor rules.

### 5. The first shipped slice should be narrow and measurable

The goal is to remove whole classes of work from a narrow hot cohort, prove correctness and explainability, and only then widen the route.

## What `0.68` finished, and what it did not

`0.68` did the right preparatory work:

* established planner-owned covering-read contract data
* kept direct-field projection shape under one shared authority
* widened more ordered and bounded index-backed reads
* aligned accepted text-prefix and casefold range families
* fixed important ordering correctness gaps
* reduced clone/payload churn in the cursorless retained-slot lane

But `0.68` still leaves the central gap open:

* no dedicated `ExecutionNodeType::CoveringRead`
* no fully distinct runtime path that stays off row storage
* no direct index-output-to-projection execution mode for scalar reads
* no planner-owned proof boundary for when row existence checks may be skipped versus preserved

So the next step should not be more local executor cleanups. It should be making the planner-owned covering contract executable as its own route.

## Proposed Design

### A. Promote covering reads to a first-class execution node

Extend route selection so eligible scalar reads can compile to an explicit covering-read node.

Conceptually:

```rust
pub(in crate::db) enum ExecutionNodeType {
    // existing variants...
    CoveringRead,
}

pub(in crate::db) enum LoadTerminalFastPathContract {
    // existing variants...
    CoveringRead(CoveringReadExecutionPlan),
}
```

The important change is not the exact type shape; it is that covering execution becomes explicit at the route boundary.

### B. Split planner-owned eligibility into execution-grade admission

`0.68` introduced planner-owned `CoveringReadPlan` contract data. `0.69` should add a second planner helper that answers a stricter question:

> Can this query execute in covering mode with current semantics, current order contract, and current page/window behavior, without row storage?

Suggested conceptual helper:

```rust
pub(in crate::db) fn covering_read_execution_plan<K>(
    plan: &AccessPlannedQuery<K>,
    projection: &PlannedProjectionLayout,
    access_window: &ScalarAccessWindowPlan,
    order_contract: &ScalarOrderContract,
    primary_key_name: &'static str,
    strict_predicate_compatible: bool,
) -> Option<CoveringReadExecutionPlan>
```

This should only return `Some(...)` when:

* every projected output is sourced from an index component, primary key, or planner-known constant
* traversal order or a planner-owned reorder contract is sufficient for the requested output order
* page/window semantics can be preserved without row materialization
* no row-only predicate or stale-row semantics force row access
* the route does not need executor-side “best effort” reinterpretation

### C. Add explicit field-source and row-presence semantics

The current `CoveringReadPlan` field-source contract is a good starting point, but `0.69` needs one more explicit dimension: row-presence semantics.

Conceptually:

```rust
pub(in crate::db) enum CoveringExistingRowMode {
    ProvenByPlanner,
    RequiresRowPresenceCheck,
}
```

The point is to make this decision explicit instead of implicit.

Phase 1 should admit only shapes where row-presence behavior is already safe under planner-known semantics. If that proof is not available, the query remains on the row-materialized path.

### D. Add a dedicated covering-read runtime

Implement a runtime path that:

* consumes index traversal output directly
* decodes only the required index components
* emits projected scalar or narrow structural outputs directly
* uses existing planner-owned order/window contracts
* never falls back into generic row payload decode or generic slot-row assembly unless the route itself is row-materialized

The runtime should not:

* derive covering eligibility
* reinterpret Top-N or ordered-route contracts
* widen route safety based on local observations

### E. Keep fallback strict and local

If any part of the contract is unsupported or uncertain, planning falls back before execution starts.

Fallback triggers should include:

* non-coverable projected fields
* unsupported computed projection expressions
* row-sensitive residual predicates
* DISTINCT shapes not explicitly admitted into covering execution
* order shapes whose safety depends on executor-local heuristics
* offset-sensitive descending secondary-order shapes outside an explicit planner-owned contract
* stale-row or existing-row semantics that still require row storage visibility

## Proposed Contract Shape

A reasonable conceptual contract is:

```rust
pub(in crate::db) enum CoveringReadFieldSource {
    IndexComponent { component_index: usize },
    PrimaryKey,
    Constant(Value),
}

pub(in crate::db) enum CoveringReadOrderContract {
    TraversalOrder(Direction),
    PrimaryKeyOrder(Direction),
}

pub(in crate::db) enum CoveringExistingRowMode {
    ProvenByPlanner,
    RequiresRowPresenceCheck,
}

pub(in crate::db) struct CoveringReadField {
    output_slot: PlannedFieldSlot,
    source: CoveringReadFieldSource,
}

pub(in crate::db) struct CoveringReadExecutionPlan {
    fields: Vec<CoveringReadField>,
    prefix_len: usize,
    order_contract: CoveringReadOrderContract,
    existing_row_mode: CoveringExistingRowMode,
}
```

The details can shift, but the important architectural move is:

* planner-owned field sources
* planner-owned order contract
* planner-owned existing-row semantics
* executor/runtime only consumes that plan

## Eligibility Rules

### Eligible in phase 1

* scalar SQL reads and fluent scalar/narrow-structural loads only
* index-prefix and index-range access paths already admitted by planner-owned route logic
* direct field projections only
* projected outputs fully satisfiable from:

  * index components
  * primary key
  * planner-known prefix-bound constants
* traversal-order or planner-owned reorder contracts already proven safe
* bounded `LIMIT` / `OFFSET` behavior through existing `AccessWindow` contracts
* accepted raw-field and casefold text prefix/range families already normalized by planner/lowering
* only shapes whose existing-row semantics are planner-proven safe

### Explicitly ineligible in phase 1

* grouped queries
* joins
* arbitrary computed projection expressions
* broad DISTINCT shapes
* shapes requiring row payload decode for residual evaluation
* unsupported DESC secondary order families
* route shapes that still need executor-local order reinterpretation
* shapes requiring generic structural row materialization only to preserve legacy behavior

## Execution Outline

For an admitted phase-1 query:

1. planner derives `CoveringReadExecutionPlan`
2. route selection emits explicit covering-read node / fast-path contract
3. physical access produces index traversal output
4. covering runtime decodes only required components
5. runtime projects outputs directly from field sources
6. runtime applies existing access-window semantics
7. result is emitted without `DataKey` resolution, row fetch, row payload decode, or generic structural row rebuild

## Explainability

`EXPLAIN` should make this route obvious.

Suggested additions:

* execution node type: `CoveringRead`
* `covering_projection=true`
* order source:

  * `index_order`
  * `primary_key_order`
* field source per output:

  * `index_component`
  * `primary_key`
  * `constant`
* existing-row mode:

  * `planner_proven`
  * `row_check_required` (only if a later widened slice admits such a mode)

This is important both for debugging and for keeping the architecture honest.

## Rollout Plan

### Slice 1: execution-grade planner admission

* add `CoveringReadExecutionPlan`
* derive planner-owned existing-row semantics explicitly
* keep runtime fallback-only initially
* add planner tests for admitted vs rejected shapes

### Slice 2: explicit route + EXPLAIN

* add explicit covering-read execution node / fast-path contract
* add EXPLAIN/reporting coverage
* add route-matrix tests that prove the route is explicit and fail-closed

### Slice 3: runtime for single-field scalar covering reads

* direct index-backed scalar projection only
* no row storage reads
* perf harness on the one-row filtered-select and top-1 ordered cohorts

### Slice 4: multi-field narrow structural covering reads

* admit `PK + covered field` and similarly simple multi-field projections
* keep computed expressions and broad structural widening out of scope

### Slice 5: selective widening

* admit additional planner-proven safe existing-row modes only if correctness and perf justify the complexity
* revisit broader DESC secondary or DISTINCT families only under one explicit planner-owned contract

## Validation

### Correctness

* planner derivation matrix for eligible/ineligible covering execution shapes
* route-matrix tests showing explicit `CoveringRead` selection
* SQL/fluent parity tests versus row-materialized fallback
* page/window parity tests for `LIMIT` / `OFFSET`
* order stability tests across:

  * traversal order
  * planner-owned PK reorder
  * accepted casefold range/prefix shapes
* stale-row / existing-row parity tests for every admitted shape

### Explainability

* `EXPLAIN` shows `covering_projection=true` only for admitted shapes
* `EXPLAIN` shows correct field-source metadata and order source
* route snapshots stay stable for admitted and rejected families

### Performance

Required perf cases should include:

* one-row equality lookup on indexed field, projected indexed field
* one-row equality lookup on indexed field, projected PK + indexed field
* ordered secondary-index top-1 scalar read
* ordered PK top-1 scalar read where projection is coverable
* casefold expression-index top-1 and narrow range reads
* before/after attribution proving execute-bucket movement, not just front-end noise

## Success Criteria

`0.69` is successful if it:

* introduces an explicit planner-owned covering execution route
* removes row fetch from a narrow admitted hot cohort
* keeps semantics fail-closed and explainable
* reduces the representative scalar execute bucket by a clearly visible amount
* does so without adding terminal-local contract duplication

## Risks

### 1. Semantic drift

If planner admission is too permissive, covering execution will silently stop matching row-materialized behavior.

### 2. Contract duplication

If runtime re-derives order, covering eligibility, or existing-row semantics, `0.69` will repeat the same planner/executor split that earlier cleanup removed.

### 3. Complexity spread

If covering mode appears as many terminal-specific shortcuts instead of one route shape, the codebase will get faster in spots but less organized overall.

### 4. Premature widening

Trying to solve DISTINCT, broad DESC secondary, grouped execution, or computed projection support in the first slice will blur the design and likely reintroduce unsafe fallback rules.

## Open Questions

1. Should phase 1 require planner-proven row existence and admit only those shapes, or is there a clean explicit row-check mode worth keeping in the same minor line?
2. Should phase 1 allow `PrimaryKeyOrder(...)` reorder shapes immediately, or should the first runtime slice stay on pure traversal-order reads only?
3. Does physical access need a dedicated decoded traversal record type so runtime no longer depends on row-oriented materialization helpers?
4. Is it cleaner to build one dedicated covering-read terminal/runtime, or one shared runtime family parameterized by row-vs-covering materialization input?

## Recommendation

Use `0.69` to finish the architectural move that `0.68` prepared but did not complete:

* planner-owned execution-grade covering admission
* explicit `CoveringRead` route / explain surface
* direct index-output-to-projection runtime for a narrow scalar cohort
* strict fallback for anything row-sensitive or contract-ambiguous

That is the first next step that plausibly changes the order of magnitude for the hot indexed scalar read cohort without giving back the architectural cleanup that `0.68` established.
