# 0.123 Structural Classification & Routing Matrix Centralization

## Thesis

`0.123` should centralize the remaining broad structural-classification and
routing matrices that are still duplicated across planner, lowering, executor,
session, and diagnostics surfaces.

This is not a feature-expansion slice.

It is not a semantics-widening slice.

It is not a route-heuristics redesign slice.

It is a classification and routing centralization slice whose job is to remove
the largest remaining "same enum, many ladders" duplication seams:

* `FieldKind` classification ladders
* grouped/global `AggregateKind` support matrices
* SQL statement / lane routing ladders

The target end state is:

* one canonical owner for `FieldKind` classification
* one canonical owner for aggregate-family support classification
* one canonical owner for SQL statement / lane routing classification
* all other layers consume those classifications instead of rebuilding them
  locally

---

## Why `0.123` Exists

After the recent semantic centralization, flow collapse, and pipeline
unification work, the remaining highest-risk duplication is no longer dead flow
or split-brain semantic ownership.

It is repeated classification logic.

The codebase still has several broad seams where the same family is interpreted
in multiple places with slightly different ladders, usually for legitimate
phase-local reasons, but without one explicit canonical classification source.

That is now the main source of:

* drift risk
* blast-radius inflation
* "touch 5 places for one new variant" maintenance cost
* LoC growth through repeated allowlists / exclusions

### Audit Alignment

#### 1. `FieldKind` Classification Ladders

This is the strongest remaining duplication hotspot.

The same `FieldKind` family is interpreted repeatedly across:

* planner expression type inference
* numeric support
* predicate normalization
* SQL binding
* persisted-row contract handling
* mutation validation
* aggregate field handling

Examples:

* `db/query/plan/expr/type_inference/mod.rs`
* `db/numeric/mod.rs`
* `db/predicate/normalize.rs`
* `db/sql/lowering/select/binding.rs`
* `db/data/persisted_row/contract.rs`
* `db/executor/mutation/save_validation.rs`
* `db/executor/aggregate/field/mod.rs`

This is the most dangerous seam because one new `FieldKind` can easily be added
in several places and missed in another.

#### 2. Grouped / Global Aggregate Family Support Matrices

This is the densest remaining "similar but not identical" family seam.

The same `AggregateKind` subsets are still expressed across:

* planner policy
* grouped strategy selection
* grouped runtime construction
* route hints
* SQL reduction
* session SQL aggregate execution

Examples:

* `db/query/plan/model.rs`
* `db/query/plan/semantics/group_model.rs`
* `db/query/plan/semantics/grouped_strategy.rs`
* `db/query/plan/group.rs`
* `db/query/plan/semantics/group_distinct.rs`
* `db/sql/lowering/aggregate.rs`
* `db/session/sql/execute/mod.rs`
* `db/executor/planning/route/hints/aggregate.rs`

Some differences are intentional, but the current shape still requires multiple
local ladders.

#### 3. SQL Statement / Lane Routing Ladders

Statement-family and lane-family routing is still spread across:

* parser
* lowering
* prepare
* session compile
* session execute

Examples:

* `db/sql/parser/statement/mod.rs`
* `db/sql/lowering/prepare.rs`
* `db/sql/lowering/mod.rs`
* `db/session/sql/mod.rs`
* `db/session/sql/execute/mod.rs`

This is now the main "same family routed many times" seam at the SQL surface.

---

## Problem

The codebase still has multiple places where a broad enum or family is routed
through local match ladders.

The repeated pattern is:

* one canonical family exists
* several subsystems need phase-local answers about that family
* instead of consuming one canonical classification, each subsystem partially
  re-derives its own local interpretation

This produces three classes of risk:

### 1. Drift Risk

A new enum variant or family change can be applied in some ladders and missed
in another.

### 2. Locality Risk

One feature addition requires touching many owners because the classification is
distributed.

### 3. LoC Growth

Even when behavior stays correct, repeated allowlists / exclusions inflate the
codebase.

---

## Goal

Replace repeated family-routing ladders with canonical classification
contracts.

For each hotspot family, `0.123` should create:

* one authoritative classifier
* explicit phase-local consumption of that classifier
* fewer local raw enum matches outside the owning classifier

This is not about making every phase behave identically.

It is about making every phase consume the same named classification source.

---

## Core Invariant

For each broad structural family:

* `FieldKind`
* `AggregateKind`
* SQL statement / lane families

there is exactly one canonical classification owner.

Other layers may still make phase-local decisions, but they must do so by
consuming the canonical classification result rather than recreating raw match
ladders over the root enum.

Corollary:

`0.123` must centralize classification, not flatten all policy into one
identical behavior table. Intentional phase differences are allowed, but they
must be named and derived from the canonical owner instead of being implied by
scattered raw ladders.

---

## Scope

### In Scope

* `FieldKind` classification
* grouped/global aggregate-family support classification
* SQL statement / lane routing classification
* guard tests that prevent new local raw ladders from regrowing

### Out Of Scope

* new field kinds
* new aggregate kinds
* new SQL statements or lanes
* feature widening
* route-planner heuristic redesign
* executor algorithm redesign
* broad API redesign

---

## Ownership Boundary

### Planner / Query Owns

* canonical `FieldKind` classification
* canonical aggregate-family support classification
* canonical statement/lane family classification where the classification is
  about planning and structural routing rather than syntax tokenization

Planner/query is the preferred owner because this slice is about semantic and
structural family interpretation, not just local syntax handling.

### Lowering / Session / Executor Own

* phase-local consumption of canonical classifications
* local policy decisions that are explicitly derived from the canonical result
* local transport or boundary adaptation that does not re-own root-family
  classification

Lowering, session, and executor must not keep broad raw ladders over the root
families when the same answer can come from the canonical classifier.

---

## Family 1: `FieldKind` Classification Centralization

### Problem

`FieldKind` is currently interpreted in several places with slightly different
ladders:

* numeric-capable
* predicate-capable
* binding-compatible
* persisted-row-compatible
* aggregate-field-compatible
* mutation-save-compatible

Even where differences are legitimate, the current system does not make those
differences explicit through one shared classification layer.

### Goal

Create one planner-owned `FieldKind` classification contract.

### Target Shape

Introduce one central module, for example:

* `db/query/plan/expr/field_kind_semantics.rs`

with a single classifier over `FieldKind`.

The classifier should expose:

* canonical family / category
* explicit capability facets
* named phase-local predicates when truly necessary

Not dozens of free-floating `is_*` helpers scattered across the tree.

### Consumption Model

All current raw `FieldKind` ladders in the audited hotspots should migrate to
consume the shared classifier.

### Expected Effect

* fewer duplicated `FieldKind` ladders
* easier addition of new `FieldKind` variants
* lower drift risk across planner/lowering/executor/data paths

---

## Family 2: `AggregateKind` Support Matrix Centralization

### Problem

`AggregateKind` support is still encoded across multiple phase-local ladders.

The problem is not that every phase must support identical aggregate subsets.

The problem is that the codebase still lacks one explicit support matrix owner
that names those differences.

### Goal

Create one canonical aggregate-family support classification layer.

### Target Shape

Introduce one central support matrix for `AggregateKind` that can answer
questions like:

* scalar/global supported?
* grouped supported?
* grouped distinct supported?
* planner strategy family?
* route-hint family?
* session SQL reduction family?

The exact shape can be:

* one central descriptor struct
* one aggregate support matrix module
* one planner-owned aggregate-family classification contract

### Consumption Model

Current ladders in planner/grouped/runtime/route/session/lowering should consume
this support matrix instead of matching directly on `AggregateKind` where
possible.

### Expected Effect

* fewer "similar but not identical" ladders
* explicit naming of intentional phase differences
* lower risk when adding or changing aggregate families

---

## Family 3: SQL Statement / Lane Routing Centralization

### Problem

Statement/lane routing still appears in multiple surfaces:

* parser statement family
* lowering lane kind
* prepare routing
* session compile routing
* session execute routing

Some of these are valid phase-local boundaries, but the same statement/lane
families are still re-routed repeatedly.

### Goal

Create one canonical SQL statement / lane classification layer.

### Target Shape

Introduce a shared routing classification that answers:

* statement family
* lowering lane family
* execution surface family
* whether prepare/explain/describe/show-style handling applies

This should not eliminate legitimate phase boundaries.

It should eliminate repeated raw routing over the same root statement family.

### Consumption Model

Parser/lowering/session should consume shared statement/lane classification
results instead of each maintaining their own parallel "what family is this?"
ladders.

### Expected Effect

* fewer repeated statement routing ladders
* lower blast radius when adding or changing SQL surfaces
* clearer ownership of statement-family classification

---

## Concrete Actions

### Step 1. Inventory All Raw Classification Ladders

For each hotspot family:

* `FieldKind`
* `AggregateKind`
* SQL statement / lane families

map every raw match ladder and classify each one as:

* canonical owner candidate
* legitimate phase-local consumer
* duplicate ladder that should be removed

### Step 2. Build Canonical Classification Owner Per Family

Introduce:

* one `FieldKind` classifier
* one `AggregateKind` support matrix
* one SQL statement/lane classifier

### Step 3. Migrate Highest-Risk Consumers First

Recommended order:

* `FieldKind`
* `AggregateKind`
* SQL statement / lane routing

because `FieldKind` is the broadest and most repetitive drift-risk seam.

### Step 4. Remove Local Raw Enum Ladders

After migration, delete local matches that are now redundant.

### Step 5. Add Guardrails

Add structural guards that fail when new local raw ladders are introduced
outside the canonical classifiers.

---

## Recommended Work Order

### `0.123.0` - `FieldKind` Classification Collapse

Start here.

This is the highest-value first cut because it is:

* broad
* repetitive
* cross-layer
* high drift risk
* likely to reduce the most duplicated match ladders

This first cut should stay explicitly bounded to `FieldKind`.

### `0.123.1` - `AggregateKind` Support Matrix Collapse

Once `FieldKind` is centralized, do the same for aggregate-family support
matrices.

This is the next best target because it still has many near-miss ladders, but
some of those differences are intentionally phase-specific and should be named
rather than flattened.

### `0.123.2` - SQL Statement / Lane Routing Collapse

Finish with statement/lane routing after the first two classifiers establish the
pattern.

This should come last because it has the highest risk of overreaching into
phase-local parser/lowering/session control flow if tackled too broadly at the
start.

---

## Exit Condition

`0.123` is complete when:

* `FieldKind` interpretation comes from one canonical classifier
* `AggregateKind` support subsets come from one canonical support matrix
* SQL statement / lane routing comes from one canonical classifier
* audited hot modules no longer carry their own raw ladders over those root
  families unless clearly justified and named as local consumption
* adding a new root enum variant no longer requires hunting through many
  similar ladders

This should be visible as:

* fewer repeated `match` ladders across the hotspot modules
* lower touched-file count for new family variants
* clearer audit interpretation of those families as centrally classified rather
  than broadly re-derived

## Expected Audit Impact

### Velocity

* lower blast radius when adding or adjusting enum variants
* fewer modules touched per family change

### Complexity

* fewer repeated branch ladders
* lower hotspot density in affected modules

### DRY

* major reduction in structural duplication across the three broadest family
  seams

### Canonical Semantic Authority

* clearer read that family interpretation is centralized
* lower risk of phase-local classification drift

---

## Non-Goals

`0.123` should not:

* widen features
* redesign planner ownership
* change semantic meaning of field kinds, aggregate kinds, or statement
  families
* collapse all phase-local behavior into one identical policy
* turn into a broad refactor of unrelated enums

---

## Summary

`0.123` centralizes the three broadest remaining structural-classification seams
in the codebase:

* `FieldKind`
* `AggregateKind`
* SQL statement / lane routing

The goal is not to erase legitimate phase-local differences.

The goal is to make those differences consume one canonical classification
source instead of rebuilding raw enum ladders all over the system.

If `0.116` through `0.122` cleaned up ownership, flow, and structural
traversal, `0.123` should reduce the next major source of LoC growth: repeated
family interpretation.
