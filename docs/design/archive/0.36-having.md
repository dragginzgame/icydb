# 0.36 - HAVING, DISTINCT Aggregates & Streaming Hardening

## Status

Draft - Architectural Design  
Target: 0.36  
Prerequisite: `0.35` GROUP BY is stable, deterministic, and fully test-covered

---

# 1. Objective

Extend grouped query capability with:

* HAVING clause support
* DISTINCT aggregates (`COUNT(DISTINCT x)`, etc.)
* Hardened streaming grouping (formal eligibility + fallback)
* Explicit aggregate projection contracts

While preserving:

* Strict `query -> plan -> executor -> storage` layering
* Deterministic ordering
* Streaming-first execution where possible
* No storage-layer semantic leakage

---

# 2. Non-Goals (0.36)

Explicitly out of scope:

* Window functions
* Nested `GROUP BY` pipelines
* Multi-stage aggregation plans
* Storage-level grouped pushdown
* Parallel/distributed grouping
* Custom aggregate plugins

---

# 3. Feature 1 - HAVING Clause

## 3.1 Logical Model

Extend `LogicalPlan`:

```rust
pub struct HavingPlan {
    pub predicate: Predicate,
}
```

Extend:

```rust
pub(crate) struct LogicalPlan<K> {
    pub(crate) aggregate: Option<AggregatePlan>,
    pub(crate) group_by: Option<GroupByPlan>,
    pub(crate) having: Option<HavingPlan>,
}
```

Constraints:

* `having` is valid only when `group_by` is present.
* HAVING predicates may reference group fields.
* HAVING predicates may reference aggregate terminals.
* HAVING predicates cannot reference raw non-grouped fields.

## 3.2 Execution Pipeline

New pipeline:

```text
Scan
-> Filter (WHERE)
-> Group
-> Aggregate
-> Having
-> Order
-> Page
```

HAVING operates on grouped rows, not raw rows.

## 3.3 Determinism Rule

HAVING filtering must not affect the group ordering contract.

Filtering removes groups but must not reorder remaining groups.

## 3.4 Streaming Implications

Streaming grouping with HAVING is allowed when:

* HAVING predicates reference only group key values.
* HAVING predicates reference only aggregate values computed in-stream.

Streaming fold driver must:

* Finalize aggregates for the group.
* Evaluate HAVING.
* Conditionally emit the group.

No buffering of multiple groups is required.

# 4. Feature 2 - DISTINCT Aggregates

## 4.1 Logical Model

Extend `AggregateSpec`:

```rust
pub struct AggregateSpec {
    kind: AggregateKind,
    target_field: Option<String>,
    distinct: bool,
}
```

DISTINCT allowed only for:

* COUNT
* SUM (optional)
* MIN/MAX (no-op semantics if distinct)

Not allowed for:

* FIRST/LAST

## 4.2 Execution Model

### Streaming Mode

DISTINCT aggregates are streaming-compatible only if:

* Ordered access guarantees distinct collapse per group.
* Or distinct set size is bounded (hard limit).

Otherwise fallback to materialized grouping.

### Materialized Mode

Group state becomes:

```text
HashMap<group_key, AggregateState>
```

Where `AggregateState` may contain:

```text
HashSet<encoded_value_bytes>
```

Memory control is required.

## 4.3 Budget Enforcement

DISTINCT aggregates require new guardrails:

* Max distinct values per group
* Max distinct memory per query

Failure must be explicit and typed.

# 5. Feature 3 - Streaming Eligibility Formalization

`0.35` introduced streaming grouping informally.

`0.36` formalizes that contract.

## 5.1 Streaming Eligibility Matrix

| Condition | Required |
| --- | --- |
| Ordered access path | Yes |
| Forward direction | Yes |
| No residual predicate | Yes |
| Aggregate streaming-compatible | Yes |
| HAVING streaming-compatible | Yes |
| DISTINCT streaming-compatible | Yes |

Planner may mark eligibility.

Executor must validate eligibility before execution.

## 5.2 Streaming Driver Refactor

Extract:

```text
aggregate/core.rs     // fold primitives
aggregate/scalar.rs   // scalar driver
aggregate/grouped.rs  // grouped driver
```

No flag-based driver unification.

Shared fold core only.

---

# 6. Feature 4 - Aggregate Projection Contract

`0.36` introduces explicit projection contracts.

## 6.1 Explicit Output Schema

Define:

```rust
pub struct GroupedOutputSchema {
    pub group_fields: Vec<String>,
    pub aggregate_aliases: Vec<String>,
}
```

Alias rules:

* Each aggregate may have an alias.
* If no alias is provided, derive a canonical alias.

Examples:

* `count -> count`
* `max_rank -> max(rank)`

This is groundwork for HAVING and future projection.

---

# 7. Index Interaction (Expanded)

Still no storage pushdown.

Allow:

* Index-only grouping eligibility detection.
* Partial covering index grouping (scan without row fetch if no non-index aggregate fields).

This remains an executor-level optimization only.

---

# 8. Continuation Contract (Expanded)

Grouped continuation tokens must now encode:

* Group key boundary
* Aggregate shape version
* HAVING presence (shape signature already covers this)

DISTINCT does not change continuation semantics.

---

# 9. Validation Additions

Validation must ensure:

* HAVING references only valid group/aggregate symbols.
* DISTINCT is only allowed for supported aggregate kinds.
* No duplicate aggregate aliases.
* No aggregate inside HAVING.

---

# 10. Performance Expectations

Streaming HAVING:

* `O(n)`
* `O(1)` per-group memory

Materialized DISTINCT:

* `O(n)`
* `O(g + distinct_values)` memory
* `O(g log g)` final sort

---

# 11. Refactor Requirements Before 0.36

Must verify:

* Grouped streaming driver is isolated.
* Comparator alignment is guaranteed.
* Grouped fold core is extracted cleanly.
* No plan -> executor leakage.

If not true, refactor first.

---

# 12. Success Criteria

`0.36` is complete when:

* HAVING works in both streaming and materialized modes.
* DISTINCT aggregates are deterministic.
* Streaming eligibility is formally enforced.
* No layering violations are introduced.
* All continuation semantics are preserved.
* Scalar paths are unaffected.

# 13. Architecture Score Target

After `0.36`:

* Layering Integrity >= 9.2 / 10
* Streaming Contract Formalization >= 9 / 10

# 14. Why This Is the Right 0.36

`0.35` gave:

* Structural grouping
* Deterministic ordering
* Fluent surface

`0.36` builds:

* Post-group filtering (HAVING)
* Aggregate cardinality expansion (DISTINCT)
* Streaming hardening
* Projection contracts

These are natural, coherent expansions, not feature sprawl.
