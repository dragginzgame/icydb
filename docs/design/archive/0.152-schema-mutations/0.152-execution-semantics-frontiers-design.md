# IcyDB 0.152 - Execution Semantics Frontiers

## Status

Deferred design note.

0.152 records the remaining execution-boundary work after the streaming,
projection, predicate, route-hint, and grouped-finalization cleanup passes.
This document is not an implementation commitment for the current branch. It
defines the next architectural frontiers so they can be approached deliberately
instead of as local cleanup.

## Summary

The execution spine no longer has an obvious hidden projection or streaming
boundary mistake.

Recent cleanup moved projection and row shaping closer to executor-owned
boundaries, removed avoidable staging vectors, and clarified where
materialization is semantically required. The remaining opportunities are no
longer broad refactors. They require either new execution interfaces or
carefully scoped correctness rules.

The two real fronts are:

* streaming mutation input for `INSERT ... SELECT`
* early-admissible strict covering projection validation

Everything else should be treated as already architecturally sound unless a
future audit finds a concrete regression.

## Confirmed Settled Areas

| Area | Status | Reason |
| --- | --- | --- |
| SQL `SELECT` projection | Settled | Projection enters executor-owned structural projection; session code shapes SQL response payloads. |
| Fluent projection terminals | Settled | Scalar value projection is pushed into aggregate/projection execution instead of terminal-layer re-projection. |
| Grouped aggregate projection | Settled | Grouped output pages project during grouped finalize after route-owned selection. |
| `DISTINCT` projection outputs | Settled | Count-only paths count admissions directly; value-returning paths retain values because they are public output. |
| Predicate runtime | Settled | Predicate programs evaluate per row through slot readers and short-circuit in the hot loop. |
| Cursorless pagination | Settled | Cursorless retained/direct paths can push skip/keep windows into collection when route order is already final. |
| Cursor-based pagination | Intentionally materialized | Cursor boundaries require ordered retained rows before cursor construction. |
| `ORDER BY` | Intentionally materialized or bounded | Unbounded order needs full comparison; bounded top-k paths use retained heaps where safe. |

## Frontier 1: Streaming Mutation Input

### Current Shape

`INSERT ... SELECT` still has a materialized handoff between the SELECT source
and the mutation batch:

```text
execute SELECT projection
  -> owned projected rows
  -> mutation patch batch
  -> commit window
```

The important limitation is not SQL projection placement. The SELECT side is
already routed through executor-owned structural projection. The limitation is
that the write path expects an owned mutation batch before validation and
commit-window preparation proceed.

### Why This Is Not Local Cleanup

A local change cannot remove the intermediate owned rows safely because the
mutation pipeline currently owns validation, patch construction, batching, and
commit-window preparation as batch-shaped contracts.

Streaming `INSERT ... SELECT` would require a new input contract, such as:

```rust
trait MutationInput {
    type Row<'a>
    where
        Self: 'a;

    fn next_row(&mut self) -> Result<Option<Self::Row<'_>>, QueryError>;
}
```

or a narrower iterator-like internal adapter:

```rust
trait StructuralPatchStream {
    fn next_patch(&mut self) -> Result<Option<StructuralPatch>, QueryError>;
}
```

The exact shape should be chosen with the mutation boundary in view. The key
property is that write execution can consume one validated source row at a time
without first building one full `Vec` of projected source rows.

### Target Shape

```text
SELECT source stream
  -> validate/project one source row
  -> build one mutation patch
  -> feed mutation input stream
  -> commit-window preparation
```

### What This Unlocks

* no intermediate source-row vector for `INSERT ... SELECT`
* early failure on invalid source rows
* lower peak memory for bulk insert-select workloads
* a future path toward chunked commit preparation
* a future path toward backpressure-aware write execution

### Design Risks

This is a feature-sized boundary change.

It touches:

* SQL `INSERT ... SELECT` execution
* structural mutation patch construction
* mutation validation
* commit-window preparation
* failure atomicity and rollback assumptions
* diagnostics and row-count accounting

The first implementation should be narrow and internal. It should not expose a
public streaming write API until the commit-window and rollback contracts are
explicitly proven for streamed input.

### Non-Goals

* Do not change public SQL semantics.
* Do not change commit atomicity.
* Do not introduce partial writes.
* Do not add chunked commits in the first slice.
* Do not generalize this into a public ingestion API until the internal
  mutation stream is proven.

## Frontier 2: Early-Admissible Strict Covering Projection

### Current Shape

Some strict covering projection paths still favor full candidate validation and
accounting before final page selection.

Conceptually:

```text
covering candidates
  -> validate row presence / covering payload contract
  -> decode or retain candidate payload
  -> apply page window
```

This is deliberate for paths where correctness depends on complete candidate
knowledge, ordering, row-presence accounting, `DISTINCT`, or fallback safety.
It is not a structural flaw.

### Opportunity

For a subset of covering paths, the executor may be able to validate and admit
rows in the same pass, then stop once the effective page window is satisfied:

```text
covering candidates
  -> validate candidate
  -> admit into page window
  -> stop when the page is proven complete
```

This should be called early-admissible covering execution. It is a precision
optimization, not a replacement for the existing strict paths.

### Required Safety Rules

An early-stop covering mode is valid only when all of these are true:

* output order is exactly the covering scan order
* no later global ordering or primary-key reordering is required
* no `DISTINCT` semantics need unseen candidates
* no grouped aggregation semantics need full group knowledge
* row-presence checks can be performed before admission
* missing-row policy is enforced identically to the strict path
* the page can prove whether a next cursor or lookahead row is needed
* diagnostics can distinguish candidates seen, rows present, and rows emitted

If any condition is not provable from prepared route state, the executor must
use the existing strict materialized path.

### Candidate Contract

The route planner should not expose a loose boolean. It should derive an
explicit prepared mode:

```rust
enum CoveringProjectionAdmissionMode {
    Strict,
    EarlyAdmissible {
        required_lookahead: usize,
    },
}
```

The executor would then select the early-admissible scan only from that
prepared mode. The mode should be derived near existing covering projection
order and row-presence policy, not re-derived inside the scan loop.

### Design Risks

This is correctness-sensitive.

The risk is stopping early before observing a candidate that would affect:

* final ordering
* `DISTINCT` admission
* cursor continuation
* strict missing-row behavior
* row-count or diagnostics accounting

The first implementation should target only the simplest route-ordered,
non-`DISTINCT`, non-grouped, index-order covering projection case. Broader
admission modes should be added only after tests pin the safety matrix.

### Non-Goals

* Do not weaken row-presence checks.
* Do not early-stop reordered covering projections.
* Do not early-stop `DISTINCT` covering projections unless a separate proof
  shows unseen candidates cannot affect output.
* Do not change cursor format.
* Do not let execution infer safety from ad hoc local checks.

## Implementation Direction

### Phase 1: Record Current Boundary

Keep the current execution model unchanged.

Add tests or audit checks only if they protect existing settled boundaries from
regression:

* SQL projection remains executor-owned.
* Grouped projection remains grouped-finalize-owned.
* Cursorless pagination remains scan-time bounded where route order allows it.
* Cursor-based pagination remains materialized unless a prepared streaming
  proof exists.

### Phase 2: Internal Streaming Mutation Prototype

Introduce an internal mutation input stream behind the existing SQL write
surface.

The prototype should support only `INSERT ... SELECT` and should preserve the
same final mutation batch semantics. A successful prototype may still build the
same commit-window structures internally, but it should avoid a separate
projected source-row vector.

### Phase 3: Early-Admissible Covering Matrix

Define a safety matrix for covering projection early stop.

The first admissible route should be deliberately narrow:

* index-order output
* no `DISTINCT`
* no grouping
* no post-access reorder
* row-presence check available before admission
* page lookahead explicitly represented

Only after that matrix is tested should executor code add a new covering
admission mode.

## Validation Expectations

Any future implementation must include tests for:

* `INSERT ... SELECT` width mismatch and early failure
* `INSERT ... SELECT` rollback or no-partial-write behavior
* source row order preservation
* commit-window accounting
* covering projection row-presence accounting
* covering `LIMIT/OFFSET` with and without lookahead
* covering cursor emission
* strict fallback when early-admissible safety is not proven

## Final Position

Further gains now require new capabilities, not broad refactors.

The execution architecture is streaming-aligned for the known read paths. The
remaining work should be pursued only when the project is ready to add either a
new mutation-stream interface or a formally guarded early-stop covering mode.
