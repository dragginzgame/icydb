# IcyDB 0.148 - Schema Evolution Runtime Cutover

## Status

Implementation closeout.

0.148 is the schema minor after the 0.147 live-schema authority closeout. It
establishes the first narrow, working schema-evolution runtime:

* append-only nullable field additions are accepted
* old shorter rows decode through accepted row contracts
* missing appended nullable fields materialize as `NULL`
* new writes emit the current accepted row layout
* unsupported transitions fail before row decode or write staging

The guiding boundary is:

```text
accepted schema transition plan
  -> accepted row-layout decode/write contracts
  -> old-row read + new-row write behavior
  -> generated Rust type bridge only at typed materialization edges
```

## Summary

0.147 made accepted schema metadata authoritative and pushed runtime row decode
toward accepted field contracts. 0.148 turns that authority into a concrete
runtime feature for the smallest useful transition shape.

0.148 enables:

* append-only nullable fields
* no row rewrite during upgrade
* old rows may have fewer physical slots than the accepted live layout
* missing added nullable fields materialize only through accepted schema
  absence policy
* new writes emit rows in the current accepted layout

Generated metadata remains the compiled proposal and Rust struct
materialization surface. It does not decide live persisted row layout,
missing-slot behavior, or write target slots. Remaining generated uses are
explicitly named as generated-compatible proposal, typed materialization,
complete-image, or value-codec bridges.

## Goals

* Support the first non-exact accepted schema transition: append-only nullable
  fields with a schema-owned missing-slot policy.
* Keep schema evolution fail-closed for removals, reorders, type changes,
  primary-key changes, nested leaf changes, relation changes, and ambiguous
  defaults.
* Make old-row reads work without rewriting stored rows.
* Make new writes produce rows in the accepted current layout.
* Preserve the existing slot-framed row format.
* Keep SQL/fluent planning and cache identity based on accepted schema.
* Retire generated runtime bridges where they decide row shape, missing-slot
  behavior, or target-slot admission.
* Keep typed Rust materialization as a boundary adapter, not schema authority.

## Non-Goals

* Do not support field removal.
* Do not support field rename as a semantic migration.
* Do not support field reorder or slot remapping.
* Do not support field kind/type changes.
* Do not support nested record/list/map/enum shape changes.
* Do not support primary-key changes.
* Do not add per-row schema versions.
* Do not introduce a multi-version row decoder.
* Do not add broad SQL `ALTER TABLE` support in this slice.
* Do not infer database defaults from Rust struct `Default`.
* Do not support required appended fields through database defaults.
* Do not replace the generated-compatible value-codec bridge.
* Do not rewrite all existing rows during startup.

## Release Rule

0.148 should be one correctness boundary, even if it lands as multiple internal
patches.

Do not publish a state where:

* transition policy accepts additive fields but row decode still rejects old
  rows with shorter slot tables
* row decode synthesizes missing fields but write paths still target generated
  slots
* SQL metadata sees the new field but typed or structural execution cannot read
  old rows
* runtime code accepts a missing slot by inference rather than through an
  accepted-schema absence policy

## Transition Policy

### Accepted Plan Kinds

Extend the schema-owned transition artifact:

```rust
enum SchemaTransitionPlanKind {
    ExactMatch,
    AppendOnlyAdditiveFields,
}
```

`SchemaTransitionPlan` should carry the accepted additive field facts needed by
runtime code and diagnostics. Do not expose raw booleans such as
`has_added_fields` or `field_count_changed` as execution authority.

Suggested shape:

```rust
struct AddedFieldTransition {
    field_id: FieldId,
    field_name: String,
    slot: SchemaFieldSlot,
    kind: PersistedFieldKind,
    storage_decode: FieldStorageDecode,
    leaf_codec: LeafCodec,
    nullable: bool,
    default: SchemaFieldDefault,
    absence_policy: AcceptedFieldAbsencePolicy,
}

enum SchemaTransitionPlanKind {
    ExactMatch,
    AppendOnlyAdditiveFields {
        added_fields: Vec<AddedFieldTransition>,
    },
}
```

The exact representation may differ, but the plan must be explicit and
schema-owned.

### Accepted Additive Fields

An additive transition is accepted only when all stored fields and row-layout
entries are an exact prefix of the generated proposal.

The added field must be nullable and must use:

* `NullIfMissing`: field is nullable and has no database default.

Required fields without an explicit database default stay rejected, even if the
Rust struct has a `Default` value. Database-owned defaults are the 0.149
follow-up, not part of the 0.148 runtime boundary.

### Rejected Transitions

Reject:

* stored field removed from generated proposal
* stored or generated field inserted in the middle of the layout
* field ID changed
* field name changed without a migration policy
* field kind changed
* field slot changed for an existing field
* field nullability changed for an existing field
* field default changed for an existing field
* storage decode changed
* leaf codec changed
* nested leaf metadata changed
* generated proposal adds a non-nullable field with no database default

## Row Decode Semantics

### Slot Table Rule

0.147 requires the physical row slot count to equal the accepted contract slot
count. 0.148 must relax that only for accepted additive transitions:

```text
physical_row_slot_count <= accepted_required_slot_count
```

Extra physical slots remain corruption or unsupported transition evidence.

For each accepted slot:

* if the slot exists in the row, decode it through the accepted field contract
* if the slot is missing, materialize through `AcceptedFieldAbsencePolicy`
* if the slot is missing and policy is `Required`, fail closed

Primary-key slots must always exist. A missing primary-key slot is corruption,
not an additive-field absence case.

### Missing Field Materialization

The row reader must distinguish:

```text
missing physical slot
present slot containing NULL
present slot containing value
```

These are not equivalent at the row-envelope boundary.

Runtime value materialization may collapse a missing nullable added field into
`Value::Null`, but diagnostics and invariants should still know that the slot
was absent.

### Direct Dense Decode

Dense decode should produce one logical value per accepted live slot.

For old rows:

* existing prefix slots decode normally
* added nullable slots materialize `NULL`
* required missing slots fail closed

### Sparse Required-Slot Decode

Sparse one-slot decode must be able to answer a request for an added slot that
is physically absent in an old row.

It should not require scanning or allocating a full slot span vector just to
discover that the requested slot is beyond the physical row count. The sparse
scanner can validate the row envelope, validate the primary-key span, and then
return the schema-owned absence value for the requested missing slot.

### Lazy Slot Reader

`StructuralSlotReader` should expose missing added slots as declared slots with
absence-backed materialization.

`has(slot)` may continue to mean physical payload presence if existing callers
need that distinction. Callers that require logical field availability should
use the canonical required-value methods.

## Write Semantics

### New Rows

New rows after an accepted additive transition must be emitted in the current
accepted layout, not the old shorter layout.

For each live accepted field:

* authored value -> encode into accepted slot
* omitted nullable field -> encode explicit `NULL`
* omitted required field -> reject before persistence

New writes emit complete current-layout rows. Missing physical slots remain an
old-row compatibility feature, not a normal new-row representation.

### Updates and Replaces

Updates against old rows must preserve existing fields and materialize added
fields according to accepted absence policy when a full canonical row is
re-emitted.

`REPLACE` should follow insert semantics for all live fields. It must not copy
old missing-slot state forward unless the replace operation is explicitly
defined as a patch over the existing row.

### Index Rebuild

Startup index rebuild must read old rows through the accepted row layout. If an
added field participates in an index, the missing-slot value must be the value
used for index key construction. For nullable additive fields this means the
index sees `NULL` for old rows.

If an added field cannot produce a deterministic index key from its absence
policy, the transition must be rejected while that index exists.

## Generated Bridge Retirement

0.148 should not remove every use of generated metadata. It should remove the
uses that conflict with accepted additive layouts.

### Must Move To Accepted Contracts

These paths must be accepted-contract owned before additive transitions are
enabled:

* raw row slot-count validation
* missing-slot materialization
* non-primary field payload decode
* primary-key field contract validation
* structural projection and retained-slot reads
* SQL write target slot selection
* structural patch replay over old rows
* index rebuild row decode
* reverse relation source-row validation

### May Remain Generated For Now

These are typed compatibility boundaries and may remain generated-backed in
0.148:

* Rust struct materialization after row values have been produced from
  accepted contracts
* generated proposal construction
* generated compile-time validation in derive/macro tests
* typed API method signatures
* typed index model declarations
* value-to-bytes encoding through explicitly named generated-compatible codec
  bridges

The important rule is that generated code may consume accepted-decoded values,
but it must not decide accepted row layout or missing-slot semantics.

## SQL And CLI Surface

Schema evolution in 0.148 is proposal-driven, not SQL-DDL-driven.

The normal flow remains:

```text
deploy/upgrade code with generated schema proposal
  -> startup reconciliation
  -> accepted transition plan
  -> accepted schema snapshot update
```

SQL should observe the accepted result:

* `DESCRIBE <entity>` shows the added field and physical slot
* `SHOW COLUMNS FROM <entity>` shows the added field
* `SELECT added_nullable_field FROM entity` returns `NULL` for old rows
* predicates against added nullable fields follow normal `NULL` semantics
* writes may author the new field after the transition is accepted

No broad `ALTER TABLE` command is required for 0.148. A future schema DDL
slice can lower SQL DDL into the same schema transition machinery once the
runtime semantics are proven.

## Observability

Add explicit transition metrics for:

* exact-match accepted
* additive-fields accepted
* additive-fields rejected because absence policy is unsafe
* removals/reorders/type changes rejected
* old rows decoded with missing additive fields
* new rows written with current accepted layout

Diagnostics should stay concise:

```text
accepted additive schema transition: entity='Character' added_fields=1 first_added='nickname' slot=32 absence=null
```

Rejections should identify the first unsafe field:

```text
schema evolution is not yet supported for entity 'Character': additive field requires explicit database default or nullable field: field='rank' slot=32 kind=Uint
```

## Validation

### Required Tests

Add tests for:

* exact-match transition remains accepted
* nullable append-only field transition is accepted
* required append-only field without database default is rejected
* field removal is rejected
* field reorder is rejected
* field kind change is rejected
* nested leaf change is rejected
* old row read after nullable additive transition returns `NULL`
* new row write after nullable additive transition emits current slot count
* update old row after nullable additive transition preserves old fields and
  produces valid current-layout output
* `DESCRIBE` / `SHOW COLUMNS` show added fields from accepted schema
* SQL `SELECT added_field` over old rows returns `NULL`
* startup index rebuild over old rows decodes through accepted row contracts
* generated-compatible exact-match behavior remains unchanged

## Implementation Order

1. Extend schema transition plan shape with additive-field acceptance.
2. Keep required/no-default additive fields rejected.
3. Teach accepted row contracts to distinguish physical slot count from live
   accepted slot count.
4. Add missing-slot materialization through `AcceptedFieldAbsencePolicy`.
5. Update dense, sparse, one-slot, retained-slot, and lazy row readers.
6. Route structural patch replay and index rebuild through the accepted
   missing-slot-aware readers.
7. Ensure new writes emit current accepted layout.
8. Add upgrade tests for old-row read and new-row write.
9. Retire any generated bridge that still decides slot count, slot lookup, or
   missing-slot behavior.
10. Leave typed materialization and value-codec bridges explicitly named and
    documented.

## Stop Condition

0.148 is complete when:

* nullable append-only fields work across upgrade without row rewrite
* old rows read correctly
* new rows write current layout
* unsupported transitions still fail before row decode/write staging
* generated metadata no longer decides live slot count, missing-slot behavior,
  or write target slots
* remaining generated uses are typed compatibility boundaries only
