# 0.111 Grouped Semantic Follow-Through

## Thesis

`0.111` should admit omitted-`ELSE` grouped searched `CASE` for the
already-shipped grouped boolean searched-`CASE` family when its `ELSE NULL`
expansion is provably semantically identical under the existing grouped
boolean rules, and then carry that admission consistently through grouped
identity and explain.

That means `0.111` is not a general grouped-semantics expansion. It is a
bounded, proof-gated omitted-`ELSE` admission slice inside the grouped
searched-`CASE` family.

This is a grouped semantic follow-through slice, not a grouped surface
expansion, optimizer, routing, or prepared-template slice.

---

## Why `0.111` exists

`0.110.0` shipped a coherent first grouped semantic-alignment cut:

* explicit-`ELSE` grouped searched `CASE` canonicalization
* grouped explain parity for that family
* grouped identity/cache follow-through for that family
* explicit omitted-`ELSE` fail-closed boundary

That was the right stop point for `0.110.0`, but it leaves one clear contract
question unresolved:

* what should happen to omitted-`ELSE` grouped searched `CASE` now that the
  explicit-`ELSE` family is canonicalized and identity-owned?

`0.111` exists to answer that question cleanly and then apply the answer
consistently across grouped semantic identity surfaces.

---

## Chosen policy

`0.111` admits omitted-`ELSE` grouped searched `CASE` and canonicalizes it
when the parser-lowered `ELSE NULL` form is semantically identical under the
existing grouped boolean runtime rules.

Concretely:

* omitted-`ELSE` grouped searched `CASE` remains legal grouped `HAVING`
* the omitted arm is treated as grouped `ELSE NULL`
* if the expanded grouped boolean form is provably identical under grouped
  boolean semantics, the omitted-`ELSE` spelling joins the canonical grouped
  searched-`CASE` family
* identity and explain surfaces then consume that canonical grouped form
* if semantic identity is not proven cleanly for a candidate omitted-`ELSE`
  grouped searched-`CASE` shape, that shape remains out of the admitted
  canonicalized family and is rejected in `0.111`

So `0.111` is not a general rejection slice. It is a proof-gated omitted-`ELSE`
admission and canonicalization slice inside the already-bounded grouped searched-`CASE`
family.

### Proof contract

The proof obligation in `0.111` is about full grouped boolean expression value
semantics inside the bounded grouped searched-`CASE` family already admitted by
`0.110`, not just about final group-retention behavior at the `HAVING`
boundary.

That means an omitted-`ELSE` form may join the canonical family only when its
`ELSE NULL` expansion is provably identical as a grouped boolean expression
under the engine's existing grouped boolean rules for `TRUE`, `FALSE`, and
`NULL`.

`0.111` does not establish a general grouped NULL equivalence framework
outside this bounded searched-`CASE` family.

---

## Must ship scope

### 1. Omitted-`ELSE` grouped searched `CASE` canonicalization

`0.111` must make omitted-`ELSE` behavior explicit and canonical where safe.

That means:

* model omitted `ELSE` as grouped `ELSE NULL`
* prove full grouped boolean expression-value identity under the existing
  grouped boolean rules
* align grouped explain, semantic fingerprint, and structural/cache identity to
  the same canonical grouped form
* reject candidate omitted-`ELSE` shapes whose semantic identity proof is not
  clean enough to admit them into the canonical family

### 2. Identity and explain follow-through for omitted-`ELSE`

Once omitted-`ELSE` canonicalization is admitted, every grouped identity
surface must match it:

* grouped semantic identity
* grouped structural/cache identity
* grouped explain semantic shape

Prepared/template identity remains out of scope here and stays syntax-bound.

---

## Optional additions

### 1. Internal normalization tightening inside the shipped searched-`CASE` family

`0.110.0` already canonicalizes the bounded explicit-`ELSE` grouped searched
`CASE` family and now also collapses small redundant truth wrappers such as
`(predicate) = TRUE` inside grouped boolean branch conditions.

`0.111` may continue that same line, but only as internal normalization inside
the already-admitted family. Examples:

* normalize nested boolean wrappers in `WHEN` predicates
* collapse redundant boolean truth wrappers that are already semantically
  redundant in grouped boolean context
* keep `WHEN` arm order unchanged
* do not introduce new grouped equivalence classes beyond the shipped
  searched-`CASE` family

This is acceptable for `0.111` only when it reuses the existing
planner-owned/shared boolean canonicalization seam rather than inventing a new
grouped-only rewrite engine.

### 2. Canonical emitted-tree tightening

`0.110.0` intentionally pinned identity/explain to the exact emitted grouped
canonical tree instead of adding a second simplification layer. For example,
the shipped explicit-`ELSE` boolean family currently proves against the actual
emitted grouped shape rather than against a more aggressively minimized tree.

`0.111` may tighten that emitted canonical tree only if:

* the stronger simplified tree is provably semantically identical
* explain, fingerprint, cache identity, and grouped SQL/fluent surfaces all
  move together
* the change is treated as a canonical-form change, not a local rendering tweak

This is a lower-priority follow-up inside the same grouped searched-`CASE`
family only. Any further grouped boolean family widening is deferred to a
later design line.

---

## Explicit non-goals for `0.111`

`0.111` should still not:

* widen grouped clause admission
* change grouped execution ownership
* introduce grouped residual row filtering
* widen prepared template admission
* add cost/routing behavior
* introduce broad boolean algebra simplification
* establish a general grouped NULL equivalence framework outside the bounded
  searched-`CASE` family
* add new grouped boolean families beyond searched `CASE`

---

## Recommended shape

The cleanest `0.111` shape is:

1. admit omitted-`ELSE` only when full grouped boolean semantic identity is
   proven cleanly for the bounded searched-`CASE` family
2. make identity / explain / diagnostics match that policy everywhere
3. only then consider small internal tightening inside the already-shipped
   explicit-`ELSE` family

If omitted-`ELSE` canonicalization is not proven cleanly for a candidate
shape, `0.111` should reject that shape rather than admit a noncanonical middle
state.

---

## Boundary with `0.110`

The key boundary is:

* `0.110.0` is complete as shipped
* `0.111` starts from the first explicit-`ELSE` grouped searched-`CASE` slice
  and extends that slice to admitted omitted-`ELSE` forms when grouped boolean
  semantics prove the `ELSE NULL` expansion safe
* any broader grouped equivalence work is deferred until after this omitted-`ELSE`
  proof-gated admission slice

Nothing in `0.111` should rewrite the historical scope of `0.110.0`.
