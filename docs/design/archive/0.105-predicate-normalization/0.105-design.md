0.105 Predicate Extraction Normalization

Thesis

0.105 should make predicate extraction structurally consistent and predictable
across already-admitted scalar `WHERE` expressions.

0.105 defines a normalized, rule-driven extraction model over planner-owned
boolean `Expr` so that equivalent scalar `WHERE` expressions produce
consistent derived pushdown predicates, explicit residual remainders, and
public explain output, independent of syntactic shape.

This slice operates after clause legality is decided.

It does not broaden SQL `WHERE` admission, and it does not move semantic
legality out of `validate.rs`.

Problem statement

After `0.102` and `0.103`:

* scalar `WHERE` admission is expression-complete
* residual filtering is explicit and correct

However:

* predicate extraction may still be shape-dependent
* equivalent expressions can produce different pushdown behavior
* extraction rules are implicit rather than contractually defined

Example risk:

* `a = 1 AND b = 2`
* `b = 2 AND a = 1`
* `(a = 1 AND (b = 2))`

These should behave identically, but may not if extraction is not normalized.

Design goal

Make predicate extraction:

* structurally stable
* shape-independent for equivalent expressions
* explicitly rule-driven rather than heuristic

Non-goals

0.105 does not aim to:

* improve pushdown aggressiveness
* introduce cost-based decisions
* widen predicate expressiveness
* change scalar expression semantics
* optimize residual execution
* broaden SQL clause admission

This is about consistency, not power.

Normative extraction model

Rule 1 - normalization precedes extraction

Extraction must operate on a normalized planner-owned boolean form:

* flatten associative operators (`AND`, `OR`)
* eliminate redundant nesting
* canonicalize bounded reorderings where already semantically safe

This normalization is extraction-oriented.

It must not become a second clause-admission policy layer.

The ownership contract is:

* `0.105` introduces one planner-owned extraction-normalization pass over
  already-admitted boolean `Expr`
* predicate extraction, residual derivation, and explain all consume that same
  normalized form
* SQL lowering does not gain a second clause-local legality or semantics pass

Rule 2 - extraction is sound, not maximal

The extractor must:

* only extract predicates that are semantically exact
* never extract a predicate that changes meaning

It is not required to extract all possible predicates.

Rule 3 - `AND` normalization is required; new mixed extraction power is not

For `AND` trees:

* normalization must expose a stable conjunct order
* equivalent `AND` shapes must normalize to the same planner expression shape
* `0.105` does not require new partial-conjunct extraction power

If the current extracted subset can represent the normalized `AND` subtree,
extraction must be stable across equivalent shapes.

The required normalization contract is:

* flatten nested `AND` nodes recursively
* normalize each child before parent-level ordering
* sort normalized conjunct children by one stable canonical label or
  fingerprint
* remove duplicate normalized conjunct children

So these normalize identically:

* `a = 1 AND b = 2`
* `b = 2 AND a = 1`
* `(a = 1 AND (b = 2))`
* `a = 1 AND a = 1 AND b = 2`

Rule 4 - `OR` handling remains conservative

For `OR` trees:

* extraction only occurs when the full normalized `OR` subtree is representable
* partial extraction across `OR` boundaries is not required

This conservatism applies to derived predicate extraction, not to semantic
expression normalization itself.

The required normalization contract is:

* flatten nested `OR` nodes recursively
* normalize each child before parent-level ordering
* sort normalized disjunct children by the same stable canonical label or
  fingerprint used for `AND`
* remove duplicate normalized disjunct children

Equivalent `OR` shapes must therefore produce the same normalized residual and
explain surfaces even when extraction remains conservative.

Rule 5 - operand normalization stays bounded

Equivalent operand forms should produce the same extraction result:

* commutative reordering does not change extraction when the current
  extraction surface already supports the normalized subtree
* nested forms equivalent under normalization behave identically

Examples that remain out of scope:

* distributive rewrites
* algebraic value-changing rewrites
* partial `OR` extraction

Rule 6 - stable residual definition

Residual expression must be:

* the normalized remainder after extraction
* stable across equivalent input shapes

After `0.103`, that normalized remainder becomes explicit residual plan state.

Extraction must not leave residual ownership implicit again.

The residual contract is:

* extraction runs against the normalized boolean tree
* residual equals the normalized tree with exactly the extracted normalized
  subtrees removed
* if all normalized children are extracted, residual is `None`, not synthetic
  `TRUE`
* if no subtree is extracted, residual is the full normalized expression
* equivalent input shapes must yield identical residual serialization

Examples

Equivalent extractable `AND` shapes

All of these:

* `a = 1 AND b = 2`
* `b = 2 AND a = 1`
* `(a = 1 AND (b = 2))`

must produce:

* the same derived predicate subset
* the same normalized residual remainder
* the same explain surface

Equivalent residual-only shapes

* `STARTS_WITH(REPLACE(name, 'a', 'A'), TRIM('Al')) AND age = 20`

and its swapped / nested spellings must produce:

* the same derived predicate presence or absence
* the same explicit residual filter expression
* the same explain surface

Prepared execution boundary

No change:

* extraction normalization does not widen template eligibility
* template lane still depends on predicate shape only

Likely code seams

* boolean expression normalization helpers
* predicate extraction logic
* SQL-side extraction orchestration
* explicit residual derivation
* explain consistency

Risks

Risk 1 - over-normalization

Unsafe rewrites could change semantics.

Risk 2 - hidden complexity

Normalization must remain simple and bounded.

Verification matrix

Area | Required proof
--- | ---
Normalization | equivalent expressions normalize identically
Extraction | equivalent expressions produce identical derived predicates when the current extraction surface supports them
Residual | explicit residual remainders stay stable across equivalent forms
Explain | consistent output across equivalent queries
Negative | `OR` handling remains conservative
Idempotence | duplicate boolean children collapse to the same normalized shape as the single-child form
Mixed extracted/residual | equivalent mixed `AND` shapes produce the same extracted subset and same residual remainder
Pure extractable | fully extracted normalized shapes produce no synthetic residual artifact

Acceptance criteria

0.105 is complete when:

* predicate extraction is stable across equivalent shapes
* normalization rules are explicit and bounded
* no semantic drift occurs under normalization
* equivalent admitted `WHERE` shapes yield the same derived predicate subset
* equivalent admitted `WHERE` shapes yield the same explicit residual remainder
* equivalent admitted `WHERE` shapes yield the same public explain surface

Thesis sentence

0.105 makes predicate extraction predictable by normalizing already-admitted
boolean expressions and ensuring that semantically equivalent `WHERE` clauses
produce consistent derived predicates, residual remainders, and explain output.
