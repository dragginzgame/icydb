# 0.146 Live Row-Layout Boundary Audit

## Status

Complete for the 0.146 schema-identity slice. This is a boundary audit only;
it does not change runtime decoding, encoding, or query behavior.

## Decision

Generated row layout remains the runtime read/write authority in 0.146.
Accepted persisted schema metadata is now safe for top-level metadata,
top-level SQL/query type checks, top-level SQL capability admission, and
in-memory cache identity. It is not yet safe as the row decode or write-layout
authority.

That split is intentional. The accepted snapshot currently proves that the
stored initial schema matches the compiled proposal, but it does not yet define
safe schema transitions, row remapping, per-row format compatibility, or nested
leaf descriptors.

## Current Accepted-Schema Consumers

The accepted persisted schema snapshot may currently own:

* `DESCRIBE` / `SHOW COLUMNS` entity headers and top-level field metadata.
* Top-level planner `FieldType` lookup through `SchemaInfo`.
* Top-level SQL literal canonicalization for INSERT, UPDATE, and read filters.
* Top-level SQL capability admission for projection, grouping, direct ordering,
  and global aggregate field targets.
* Nested field-path expression type inference when accepted nested leaf
  descriptors are available.
* Session shared query-plan and SQL compiled-command cache identity.

These consumers use accepted schema metadata as validation and planning facts.
They do not reinterpret persisted row bytes.

## Runtime Boundaries That Must Stay Generated

The following runtime boundaries must keep using generated `EntityModel` /
`RowLayout` metadata until migration policy and live layout authority exist:

* Stable row decoding and terminal row decoding.
* Save, replace, update, and mutation patch slot mapping.
* Delete rollback row reconstruction.
* Executor row readers and projection row views.
* Route planning that needs physical slot positions.
* Nested field-path runtime compilation and evaluation contracts.
* Any codec path that interprets the stored row payload shape.

These paths read or write physical row bytes. Switching them to accepted schema
metadata before a transition policy exists would create an implicit migration
without a fail-closed remapping contract.

## Required Later Boundary

Before accepted schema can become the live row-layout authority, the system
needs all of the following:

1. A schema-owned transition policy beyond exact initial-schema equality.
2. Fail-closed diagnostics for rejected layout-changing transitions.
3. A live layout artifact that maps accepted field identity to current physical
   row slots.
4. A clear policy for row decode/write behavior when accepted schema differs
   from the generated proposal.
5. Accepted runtime path contracts so field-path execution no longer depends on
   generated nested `FieldModel` traversal.

Until those exist, generated row layout is the only safe authority for
persisted bytes.

## Guardrail

Do not route decode, encode, mutation patching, or runtime nested field-path
execution through accepted persisted schema data just because top-level SQL
admission and field-path type inference now use that schema. Those are
different trust boundaries.
