# 0.146 Nested Leaf Metadata Design

## Status

Implemented foundation for the 0.146 schema-authority boundary. Runtime decode
and nested path execution still stay generated-layout based.

## Problem

Accepted persisted schema snapshots originally described top-level fields only.
That was enough for top-level SQL projection, grouping, ordering, aggregate
input admission, literal canonicalization, and cache identity. It was not
enough for nested field paths such as `record_field.child`.

Snapshots now carry nested leaf descriptors. Expression type inference can read
those accepted nested leaf types, and SQL SELECT projection admission can read
their accepted SQL capabilities. Runtime path compilation and evaluation still
depend on generated nested traversal contracts until live row-layout authority
exists.

## Goal

Persist enough nested leaf metadata for accepted schema to answer field-path
planning questions without consulting generated nested field descriptions.

The accepted schema should be able to answer:

* Does this nested path exist under a top-level field?
* What is the nested leaf type?
* Is the nested leaf selectable, comparable, orderable, groupable, or usable as
  an aggregate input?
* Which top-level physical slot roots the nested path?

## Proposed Persisted Shape

Each top-level `PersistedFieldSnapshot` now carries a nested leaf description
list. Each nested leaf descriptor includes:

* The path relative to the top-level field.
* The persisted leaf kind.
* SQL/query capability facts derived from that persisted kind.
* Enough identity to distinguish leaves stably within the owning top-level
  field.

The top-level field still owns the physical row slot. Nested leaves describe
the value shape inside that slot; they do not get independent row slots.

## Ownership

Schema snapshot code owns nested metadata persistence and reconciliation.

Query planning consumes nested metadata through accepted-schema lookup helpers,
not by reading snapshot internals directly.

Executor field-path evaluation remains mechanical: it follows a compiled path
against a decoded value. It must not decide whether a nested path is valid or
queryable.

## Migration Steps

1. Extend the compiled schema proposal to emit nested leaf proposals from
   generated nested field metadata. Done.
2. Extend persisted field snapshots with nested leaf descriptors. Done.
3. Reconcile generated nested proposals against stored nested descriptors under
   the current exact-match policy. Done.
4. Add accepted-schema nested lookup helpers to `SchemaInfo`. Done.
5. Move field-path type inference to those accepted-schema lookup helpers. Done.
6. Move SQL SELECT field-path projection admission to accepted nested
   capabilities. Done.
7. Move remaining runtime path contracts after live row-layout authority exists.

## Non-Goals

Do not introduce row remapping, nested indexes, field-level database defaults,
or schema evolution in this step. Nested metadata is only the missing accepted
schema input needed for field-path planning.
