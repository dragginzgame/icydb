# 0.79 Design: Bounded Predicate Expressions

## Preamble

* **scope**: freeze one bounded predicate-expression slice
* **goal**: admit field-to-field comparison in predicate position without widening into generic expression predicates
* **theme**: extend the existing predicate authority, not the scalar projection expression tree
* **non-goals**:
  * no predicate arithmetic
  * no generic `Expr`-in-`WHERE` promotion
  * no grouped predicate-expression support
  * no predicate pushdown or new index-planning rules
  * no planner rewrite or algebra work
  * no new evaluator or second predicate runtime

---

## Problem Statement

`0.78` introduced one bounded scalar projection-expression seam:

* `field + literal`
* `field - literal`
* `field * literal`
* `field / literal`
* `ROUND(expr, n)`

That work was intentionally projection-only.
The shipped predicate surface still assumes the older compare model:

* one field on the left
* one literal or literal-list on the right

So these remain unavailable today:

```sql
SELECT * FROM Customer WHERE age > rank;
SELECT * FROM Customer WHERE name = display_name;
```

and there is no matching fluent field-to-field compare builder either.

The important reality is that predicates and projection expressions do **not**
share one unified AST today.
Projection expressions are planner-owned `Expr`.
Predicates are owned by the predicate subsystem and flow through:

* `Predicate`
* `ComparePredicate`
* planner/index capability analysis
* `PredicateProgram`

So `0.79` should not pretend this is a trivial `Expr::Predicate` extension.
It should extend the **predicate** model in a bounded way that matches the code
we actually have.

---

## Current Predicate Authority

Today the canonical predicate shape is:

* boolean wrapper nodes:
  * `And`
  * `Or`
  * `Not`
* literal compare leaves:
  * `field op literal`
  * `field IN literal_list`
  * `field NOT IN literal_list`
* structural/text leaves:
  * `IS NULL`
  * `IS NOT NULL`
  * `IS MISSING`
  * `IS EMPTY`
  * `TEXT CONTAINS`
  * prefix/suffix variants

That ownership lives in:

* `/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/model.rs`
* `/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/runtime.rs`
* `/home/adam/projects/icydb/crates/icydb-core/src/db/query/builder/field.rs`

The key planner constraint is also clear today:

* index planning only understands field-vs-literal compare leaves
* field-to-field compare cannot be treated as index-sargable in this slice

That means the realistic `0.79` slice is:

* widen predicate leaf shapes
* keep index planning literal-only
* evaluate the new shape as a residual row filter

---

## Design Goals

### 1. Extend predicate leaves, not projection expressions

Add one bounded field-to-field comparison leaf to the predicate model instead
of trying to route predicates through the `0.78` projection-expression AST.

### 2. Keep index planning explicit

Field-to-field compare is admitted semantically, but it is **not** an
index-backed predicate in `0.79`.

### 3. Keep execution on the existing predicate runtime

The runtime should still compile one `PredicateProgram` and evaluate one
predicate tree.
`0.79` must not create:

* a second predicate evaluator
* a projection-expression detour for predicates
* a parallel grouped predicate lane

### 4. Preserve SQL/fluent parity

If SQL admits field-to-field compare, fluent must expose the same bounded
predicate shape in one canonical builder form.

---

## Admitted `0.79` Surface

### SQL

The admitted SQL surface is:

```sql
SELECT * FROM Customer WHERE age > rank;
SELECT * FROM Customer WHERE age >= minimum_age;
SELECT * FROM Customer WHERE score = target_score;
SELECT * FROM Customer WHERE name = display_name;
SELECT * FROM Customer WHERE active = archived;
```

Only one new family is admitted:

* `field compare_op field`

Supported operators:

* `=`
* `!=`
* `<`
* `<=`
* `>`
* `>=`

### Fluent

The matching fluent surface should be one bounded field-to-field builder family,
for example:

```rust
field("age").gt_field("rank")
field("name").eq_field("display_name")
field("active").eq_field("archived")
```

The exact naming can be finalized during implementation, but the contract is:

* one field-to-field compare builder family
* no generic expression builder
* no separate SQL-only predicate lane
* both SQL and fluent must lower to the same predicate-owned compare leaf
  before planning

---

## Canonical Representation

`0.79` should stay predicate-owned.

The realistic canonical model is one new predicate-owned field-to-field compare
leaf, for example:

```text
Predicate::CompareFields {
    left_field,
    op,
    right_field,
}
```

or one equivalent dedicated predicate-owned type carrying:

* left field
* operator
* right field
* whatever normalized compare/coercion metadata is already required by the
  current compare pipeline

`0.79` should **not** do this:

```text
Expr::Predicate { left: Expr, op, right: Expr }
```

Reason:

* the current planner/runtime/index pipeline is predicate-owned, not `Expr`-owned
* compare planning, coercion, normalization, and runtime compilation already
  center on `Predicate`
* forcing predicates through `Expr` in this slice would be more architectural
  widening than feature delivery

So the freeze rule is:

* extend the predicate model directly
* reuse existing compare semantics where possible
* do not unify all predicates and projection expressions in `0.79`
* normalize `field op field` through the same compare
  operator/type-normalization path already used for `field op literal`, except
  that the right operand remains a field reference instead of a literal payload

## Type And Comparison Rules

### Admitted comparisons

The bounded `0.79` surface should admit:

* numeric ↔ numeric:
  * `=`
  * `!=`
  * `<`
  * `<=`
  * `>`
  * `>=`
* text ↔ text:
  * `=`
  * `!=`
  * `<`
  * `<=`
  * `>`
  * `>=`
* bool ↔ bool:
  * `=`
  * `!=`

### Rejection rules

Fail closed for:

* incompatible cross-domain compare:
  * `age > name`
* bool ordered compare:
  * `active > archived`
* collection/map/list compare
* relation-valued compare
* aggregate/function-valued compare

### Coercion rule

`0.79` should reuse the existing predicate coercion model:

* numeric widening where numeric compare already uses it
* strict equality for ordinary strict compares
* no new cross-domain coercion policy

---

## Planning And Execution Invariants

### Planner

The planner behavior for `field op field` must be explicit:

* this shape is predicate-valid
* this shape is **not** index-sargable in `0.79`
* access planning may still choose an index for other predicate components
* the field-to-field compare is evaluated as a residual row filter

So `0.79` does not widen:

* prefix planning
* range planning
* ordered compare planning
* access-choice evaluator capabilities
* planner rewrite heuristics for compare expressions

Field-to-field compare must never be rewritten into an index-eligible predicate
by:

* assuming one side is constant
* deriving pseudo-ranges
* splitting into planner heuristics in this slice

### Runtime

The runtime must stay on one predicate program path.

That means:

* compile both referenced fields through the existing predicate/runtime
  field-access path once per prepared predicate program
* evaluate the two resolved row values through the existing compare semantic
  helpers at residual-filter time
* reuse the existing compare semantic helpers
* return one boolean result

No second predicate evaluator should appear.

---

## Explicitly Admitted In `0.79`

* scalar predicate position only
* field-to-field compare leaves
* operators:
  * `=`
  * `!=`
  * `<`
  * `<=`
  * `>`
  * `>=`
* participation in existing boolean wrapper structure:
  * `AND`
  * `OR`
  * `NOT`

Examples:

```sql
SELECT * FROM Customer WHERE age > rank;
SELECT * FROM Customer WHERE age > rank AND active = archived;
SELECT * FROM Customer WHERE name = display_name OR email = backup_email;
```

The boolean wrappers are not new in `0.79`; they are only allowed to contain
the new bounded compare leaf.

---

## Explicitly Rejected In `0.79`

These remain fail-closed:

* predicate arithmetic:
  * `WHERE age + 1 > rank`
  * `WHERE age > rank + 1`
* predicate functions:
  * `WHERE LOWER(name) = LOWER(display_name)`
  * `WHERE ROUND(score, 2) = target_score`
* grouped predicate expressions:
  * grouped `HAVING age > rank`
  * grouped `HAVING SUM(score) > max_score`
* field-to-literal expression predicates:
  * `WHERE age + 1 > 10`
* membership with fields:
  * `WHERE age IN (min_age, max_age)`
* structural text-function compare widening:
  * `WHERE STARTS_WITH(name, alias_name)`
* projection boolean expressions:
  * `SELECT age > rank FROM Customer`

This slice is predicate-only and field-only.
It is not a general expression-in-predicate release.

Grouped query lanes must reject the new field-to-field compare leaf before
execution.
`0.79` must not introduce any grouped-only interpretation of predicate field
references.

---

## SQL / Fluent Parity Rule

`0.79` is only correct if SQL and fluent expose the same bounded predicate
shape:

* SQL must not admit `field op field` without a matching fluent builder
* fluent must not invent a richer field-to-field predicate model than SQL admits
* both surfaces must lower to the same predicate-owned canonical representation
* there must be no SQL-only special case in parser/planner glue after lowering

Operational differences are not relevant here.
This is ordinary query semantics and therefore requires parity.

---

## Error Ownership

The failure boundary should remain owner-local:

* parser:
  * rejects syntax outside `field op field`
* lowering / predicate validation:
  * rejects unsupported field kinds
  * rejects unsupported operator/type combinations
* planning:
  * must not silently pretend field-to-field compare is index-sargable
* runtime:
  * must not introduce fallback string-matching of error text or a second
    comparison path

Representative failure messages:

* incompatible field comparison types
* ordered comparison not supported for bool fields
* grouped predicate expressions are not supported in this release

---

## Tests Required For Freeze

The `0.79` freeze requires direct proof for:

* parser acceptance of `field op field`
* lowering to one predicate-owned canonical field-to-field compare leaf
* fluent lowering parity for the same shape
* runtime correctness for:
  * numeric compare
  * text compare
  * bool equality / inequality
* planner behavior proving:
  * no accidental index-sargable classification for field-to-field compare
  * residual row filtering still works correctly when combined with other
    index-eligible predicate components
  * representative mixed case such as:

    ```sql
    SELECT * FROM Customer
    WHERE age > 18 AND score > target_score;
    ```

    where:
    * `age > 18` may still participate in ordinary index/access planning if
      otherwise eligible
    * `score > target_score` remains residual-only
    * the final result remains correct with both predicates combined
* fail-closed rejection of:
  * predicate arithmetic
  * predicate functions
  * grouped predicate expressions
  * invalid type/operator combinations

---

## Follow-Up Boundary

The next possible widenings, if chosen later, are still outside this slice:

```sql
SELECT * FROM Customer WHERE age + 1 > rank;
SELECT * FROM Customer WHERE LOWER(name) = LOWER(display_name);
SELECT age > rank FROM Customer;
SELECT * FROM Customer GROUP BY age HAVING age > rank;
```

Those are **not** part of `0.79`.

If a later release wants broader predicate expressions, it should extend this
bounded predicate-owned compare leaf deliberately rather than bypassing it
through a second `Expr` or evaluator path.
