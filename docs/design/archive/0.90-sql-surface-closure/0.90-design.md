# **0.90 SQL Surface Closure**

## Thesis

> **0.90 completes the supported SQL surface for single-entity and grouped queries by turning all admitted syntax into an explicit, consistent, and fully validated contract, with no partial support, no ambiguous behavior, and no new execution models.**

---

## Design goal

0.90 is a **closure slice**, not a feature slice.

It guarantees that:

* every supported query shape:

  * parses
  * validates
  * executes
  * explains
  * fingerprints

consistently.

And that:

* every unsupported shape:

  * fails early
  * fails clearly
  * fails deterministically

---

# **Execution model boundary (non-negotiable)**

All supported queries must map to exactly this model:

```text
scan
→ per-row scalar evaluation (pre-aggregate)
→ grouping + aggregate accumulation (optional)
→ aggregate finalization
→ post-aggregate evaluation (projection / HAVING / ORDER BY)
→ bounded output (LIMIT / Top-K)
```

Anything that cannot be expressed in this pipeline is **out of scope**.

---

# **Supported SQL surface (explicit)**

## 1. Query shape

### Supported

```sql
SELECT <projection>
FROM <single_table>
[WHERE <scalar_predicate>]
[GROUP BY <group_keys>]
[HAVING <post-aggregate predicate>]
[ORDER BY <order_terms>]
[LIMIT <n> [OFFSET <m>]]
```

---

### Not supported

* multiple tables in FROM
* joins of any kind
* subqueries (in any clause)
* CTEs
* set operations (UNION / INTERSECT / EXCEPT)
* window functions

---

## 2. FROM

### Supported

* exactly one base table

### Rejected

```sql
FROM a, b
FROM a JOIN b
```

---

## 3. WHERE (pre-aggregate)

### Supported

* scalar expressions over:

  * fields
  * literals
  * the already-admitted bounded scalar function family
* boolean composition:

  * AND / OR / NOT
* comparisons:

  * =, !=, <, <=, >, >=
* NULL checks:

  * IS NULL
  * IS NOT NULL
* basic boolean evaluation (`IS TRUE`, `IS FALSE` if already admitted)

---

### Rejected

* aggregates in WHERE
* grouped/post-aggregate expressions
* subqueries

---

## 4. GROUP BY

### Supported

* grouping by:

  * fields

Example:

```sql
GROUP BY class_name
```

---

### Rejected

* grouping by aggregates
* grouping by scalar expressions
* grouping by post-aggregate expressions

---

## 5. Aggregate functions

### Supported

* `SUM(expr)`
* `AVG(expr)`
* `MIN(expr)`
* `MAX(expr)`
* `COUNT(expr)`
* `COUNT(*)`

Where `expr` follows 0.89 rules.

---

### Semantics

* input expression evaluated per row
* NULL handling follows scalar semantics
* `COUNT(expr)` counts non-null evaluated values

---

### Rejected

* nested aggregates
* aggregate in scalar expression outside aggregate position
* `DISTINCT aggregate(expr)` as a new widening target
* aggregate FILTER

---

## 6. Projection (SELECT list)

### Supported

* scalar expressions
* aggregate expressions
* mixing grouped keys and aggregates (grouped queries)

Example:

```sql
SELECT class_name, AVG(strength + agility)
```

---

### Alias support

```sql
SELECT AVG(strength) AS avg_strength
```

---

### Rejected

* referencing undefined aliases
* recursive alias definitions

---

## 7. HAVING (post-aggregate)

### Supported

* scalar expressions over:

  * grouped keys
  * aggregate outputs
* the already-admitted bounded post-aggregate expression family

---

### Rejected

* pre-aggregate-only expressions that depend on raw row context
* subqueries
* nested aggregates

---

## 8. ORDER BY

## Two modes (from 0.87–0.88)

---

### A. Canonical grouped-key ordering

#### Supported

* ordering by grouped key prefix
* grouped key ± constant

```sql
ORDER BY class_name
ORDER BY level + 1
```

---

#### Properties

* streaming
* continuation-safe
* prefix-proof-based

---

### B. Top-K aggregate ordering

#### Supported

* ordering by:

  * aggregate outputs
  * post-aggregate expressions
  * mixed with grouped keys as tie-breakers

```sql
ORDER BY AVG(strength) DESC
ORDER BY AVG(strength), class_name
```

---

#### Requirements

* **LIMIT is required**
* grouped aggregate Top-K `OFFSET` is rejected

---

#### Properties

* heap-based Top-K
* not continuation-safe
* deterministic via explicit user order terms plus canonical grouped-key fallback

---

### ORDER BY alias support

```sql
SELECT AVG(strength) AS avg_strength
ORDER BY avg_strength
```

---

### Rejected

* aggregate ORDER BY without LIMIT
* aggregate ORDER BY with OFFSET
* unsupported expression families
* ambiguous ordering without deterministic tie-break

---

## 9. LIMIT / OFFSET

### Supported

* LIMIT in all queries
* OFFSET on the already-admitted scalar query surface

---

### Rejected

* unbounded output requirements
* grouped aggregate Top-K `OFFSET`
* OFFSET without LIMIT where the current scalar surface does not already support it

---

## 10. Scalar expression surface

### Allowed only on already-admitted clause-specific expression seams

* literals
* fields
* arithmetic
* the already-admitted bounded scalar function family
* nested expressions

---

### Phase constraints

| Phase           | Allowed inputs            |
| --------------- | ------------------------- |
| WHERE           | admitted pre-aggregate scalar expressions |
| Aggregate input | admitted per-row scalar expressions |
| HAVING          | admitted post-aggregate expressions over grouped keys and aggregates |
| ORDER BY        | admitted canonical grouped-key or bounded Top-K post-aggregate expressions |
| Projection      | admitted scalar or post-aggregate expressions for the query mode |

---

### Rejected

* cross-phase references
* unsupported function families

---

## 11. Alias resolution

### Rules

* aliases defined in SELECT:

  * usable in ORDER BY
  * not introduced as new behavior in HAVING by `0.90`
* resolution is:

  * single-pass
  * deterministic
  * non-recursive

---

### Rejected

```sql
SELECT a AS b, b AS c
```

(if ambiguous / recursive)

---

## 12. Determinism guarantees

All supported queries must produce:

* deterministic ordering when ORDER BY present
* deterministic Top-K results via:

  * user-specified `ORDER BY` tuple first
  * canonical grouped key as the final deterministic fallback
* stable results across identical executions

---

## 13. Explain / fingerprint / cache identity

### Must include:

* aggregate input expressions (0.89)
* ORDER BY mode (canonical vs Top-K)
* normalized expression structure

---

### Must guarantee:

* identical logical queries → identical fingerprints
* semantically different queries → different fingerprints

---

## 14. Error taxonomy

All unsupported shapes must:

* fail at the correct ownership boundary
* have specific error categories:

Ownership rule:

* parser rejects syntax-family exclusions
* lowering/planner reject semantic out-of-scope SQL shapes
* runtime must not discover unsupported SQL families late

Examples:

* invalid aggregate input phase
* grouped ORDER BY without LIMIT
* unsupported scalar expression
* alias resolution failure
* cross-phase dependency violation

No generic:

```text
"invalid query"
```

---

# **Explicitly rejected SQL surface**

## Not supported (hard boundary)

* joins
* subqueries
* window functions
* CTEs
* set operations
* correlated expressions
* lateral references

---

## Not supported (within current model)

* nested aggregates
* aggregate FILTER
* DISTINCT aggregates as a new widening target
* arbitrary function composition beyond the already-admitted bounded scalar and post-aggregate sets

---

# **Guardrails**

## 1. No new execution models

Everything must map to:

* scan → group → finalize → evaluate → emit

---

## 2. No partial support

Either:

* fully supported
* or fully rejected

---

## 3. No silent rewrites

No hidden transformations like:

```sql
ORDER BY aggregate → implicit full sort
```

---

## 4. Fail closed everywhere

* invalid shapes never execute
* no best-effort behavior

---

# **Acceptance criteria**

## Functional

* all listed supported queries execute correctly
* all rejected queries fail early with correct error

---

## Architectural

* no duplicate evaluator paths
* single scalar expression model
* unified validation across clauses

---

## Quality

* explain reflects real execution
* fingerprint matches logical structure
* errors are precise and stable

---

# **Final thesis**

> **0.90 defines and locks the supported SQL surface by making every admitted query shape explicit, consistent, and deterministic, ensuring that all supported queries fully align with the execution model and all unsupported shapes fail clearly and early.**
