# 0.121 Access / Explain Projection Collapse

## Thesis

`0.121` should make planner-derived access classification the sole structural
source for explain and diagnostics, so transport DTOs stop acting like
recursive structural inputs.

This is not a semantic slice.

It is not an access-path selection slice.

It is not an executor redesign slice.

It is a structural-walk contraction slice whose job is to remove repeated
projection walks across:

* explain label and node-type derivation
* diagnostics-facing access classification
* transport DTOs that currently behave like recursive structural inputs

The target end state is simple:

* planner owns the canonical access traversal
* one planner-derived frozen access projection artifact becomes the shared
  consumer surface for classification work
* explain and diagnostics consume that artifact
* transport DTOs stop behaving like a second structural owner

---

## Why `0.121` Exists

`0.120` already identified access / explain projection as a real next target:

* one canonical access traversal contract should exist
* planner, explain, and diagnostics should consume that contract
* they should stop building separate structural walks

That seam is now concrete in the tree.

Planner already owns a shared visitor-style access traversal in
[/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/access_projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/access_projection.rs:1).

But the current shape still reads roughly like:

```text
AccessPlan
  -> planner-owned projection visitor
    -> ExplainAccessPath DTO
      -> explain label walk
      -> explain node-type classification walk
      -> fingerprint / diagnostics walk
```

Even where each downstream walk is locally correct, the system still pays for:

* repeated recursion over the same topology
* extra DTO-to-label or DTO-to-classification adaptation ladders
* duplicated union / intersection child traversal
* a misleading read that explain transport owns part of access structure

That is why `0.121` exists.

The problem is no longer access semantics.

It is that too many adjacent consumers still walk the same access tree in
their own shape.

### Audit Alignment

#### 1. Velocity Preservation Audit

The current velocity line is still driven by structural follow-through and
module touch count, not missing semantics.

Access / explain projection is a good next locality target because it is:

* bounded
* cross-layer

#### 2. Complexity Accretion Audit

The complexity issue here is not one giant hotspot in isolation.

It is branch repetition across consumers:

* access-plan visitor

`0.121` should reduce that repeated structural branching rather than merely
moving it around.

This seam is a DRY problem in the structural sense:

The right next move is deletion or collapse of extra walks, not more wrapper
coordination.

Canonical semantic authority is already in good shape here:

* access-path meaning belongs to planner / access planning

That means `0.121` should not redefine access semantics.

It should make the structural ownership boundary match that semantic reality.

#### 5. `0.120` Follow-Through

`0.121` is the focused follow-through slice for that exact target.

## Problem

The system still carries more than one structural walk over access topology.

Today the broad shape reads like:

* planner walks `AccessPlan<K>` through `AccessPlanProjection`
* explain projects `AccessPlan<K>` into `ExplainAccessPath`
* explain and diagnostics then walk `ExplainAccessPath` again for:
  * labels
  * node types
  * fingerprinting / classification

That creates two closely-related problems.

### 1. Transport DTOs Behave Like Structural Owners

`ExplainAccessPath` is supposed to be a transport-facing explain surface.

But repeated re-projection from `ExplainAccessPath` makes it look like a
second canonical structural representation of access topology.

That is the wrong ownership shape.

### 2. Equivalent Access Topology Is Walked More Than Once

The same union / intersection / leaf path structure is repeatedly traversed to
derive different outputs.

That causes:

* unnecessary recursion
* repeated match ladders
* more places to update when access variants evolve
* extra blast radius for access / explain changes

The problem is not that DTO rendering exists.

The problem is:

> structural access topology is still being re-walked after planner already
> provided a canonical traversal contract

---

## Goal

Make planner-derived access classification the only structural source for
explain and diagnostics.

The preferred shared shape for `0.121` is:

* one planner-derived frozen access projection artifact

Explain and diagnostics should consume that artifact for classification work
rather than each consumer calling planner traversal separately.

They must not independently recreate recursive access topology walks over a
second owner-local representation.

---

## Core Invariant

Equivalent access topology has exactly one canonical recursive structural
traversal owner per projection family.

Corollaries:

* planner owns canonical access traversal
* one planner-derived frozen access projection artifact is the preferred
  consumer surface for classification work
* `ExplainAccessPath` is transport, not a second structural authority
* explain labels, node types, and diagnostics helpers do not re-own recursive
  access topology
* adding a new access variant should require one structural-walk update, not a
  planner walk plus multiple explain re-walks

Downstream formatting over a frozen projected artifact is allowed.

Independent recursive ownership is not.

## Ownership Boundary

### Planner Owns

* canonical access topology
* recursive traversal over access plans
* structural projection contracts for access families

Planner is the only place that should own recursive access structure.

### Explain Owns

* transport payload shaping
* JSON / text formatting
* presentation-only field naming

Explain must not own:

* a second recursive access traversal model
* local reclassification ladders that mirror planner structure
* transport-driven structural recursion when the planner-derived frozen access
  artifact is already available

### Fingerprint / Diagnostics Own

* hashing or reporting over canonical access projection
* stable output encoding

Fingerprint / diagnostics must not own:

* alternate structural access recursion
* alternate union / intersection traversal contracts

## What `0.121` Targets

### 1. Stop Treating `ExplainAccessPath` As A Classification Input

Current shape:

* `AccessPlan<K>` projects into `ExplainAccessPath`
* downstream helpers recurse over `ExplainAccessPath` again for classification

Target:

* classification comes from one planner-derived frozen access artifact
* `ExplainAccessPath` remains transport-only

### 2. Prefer One Frozen Planner-Derived Access Artifact

Current shape:

* consumers can be read as choosing between direct planner traversal and local
  DTO-driven walks

Target:

* planner derives one frozen access projection artifact
* explain labels, execution-node classification, and diagnostics consume that
  shared artifact

### 3. Collapse Child Traversal Ownership

Union / intersection child recursion should have one owner.

Target:

* one child traversal contract
* one place to update when access topology evolves

### 4. Reduce Parallel Label / Kind / Fingerprint Walks

Current shape:

* label projection
* kind projection
* node-type projection
* fingerprint projection

all operate over overlapping recursive access structure.

Target:

* fewer structural visitors
* thinner output adapters
* less transport-driven recursion

### 5. Keep JSON Rendering Presentation-Only

JSON output may still format a transport shape, but it should not be the
place where access topology is effectively re-derived for multiple adjacent
consumers.

## Concrete Actions

### Step 1. Inventory Every Access Topology Walk

Start with a concrete map of all recursive access walks across:

* planner projection
* explain DTO projection
* explain label projection
* explain node-type classification
* JSON access rendering
* fingerprint hashing

Classify each as:

* canonical structural walk
* presentation-only formatting
* duplicate structural walk that should be removed

### Step 2. Introduce One Canonical Frozen Access Artifact

Prefer one planner-derived frozen access projection artifact as the shared
consumer surface for explain labels, execution-node classification, and
diagnostics.

Do not leave "direct planner traversal vs frozen artifact" as a permanent
parallel ownership choice.

### Step 3. Remove DTO-Driven Classification

Move access-kind and execution-node classification off recursive
`ExplainAccessPath` walks where planner-owned projection already supplies the
same structure.

### Step 4. Collapse Fingerprint Access Walking

Where fingerprinting still walks transport access DTOs separately, refactor it
to consume the canonical access projection path instead of maintaining a second
structural traversal family.

### Step 5. Leave Only Formatting At The Explain Boundary

After structural ownership is collapsed, explain should retain only:

* transport assembly
* JSON/text formatting
* presentation naming

not recursive structural interpretation.

### Step 6. Add Structural Ownership Guards

Tests and guardrails should prove:

* no second recursive access walk is reintroduced in explain helpers
* explain DTOs do not become classification owners
* new access variants fail one canonical guard if they are not wired through
  the single structural projection path

## Recommended Work Order

1. Inventory current access / explain structural walks.
2. Choose the single canonical projected access shape.
3. Collapse access-kind and node-type classification first.
4. Collapse fingerprint transport walking second.
5. Leave JSON formatting last, after ownership is already narrowed.
6. Freeze the boundary with invariant tests.

The first bounded contraction should be:

* access-kind label + execution-node classification

That is the safest first cut because it is:

* clearly duplicated
* structurally central
* smaller than JSON transport reshaping
* easy to prove with parity tests

`0.121.0` should be limited to access-kind label and execution-node
classification.

JSON rendering and fingerprint transport should remain follow-on work inside
the same line only if the canonical frozen access artifact is already in place
and the first cut stays local.

* planner-owned access traversal is the only recursive structural owner
