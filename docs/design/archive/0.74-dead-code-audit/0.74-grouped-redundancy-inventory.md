# 0.74 — Grouped Redundancy Inventory

Historical note: this is a `0.74.0` cleanup inventory, not a current product
contract. References below to grouped guardrails and unsupported grouped
field-target kinds describe audit-time cleanup candidates, not the current SQL
surface. For the current grouped/query boundary, see:

- `docs/contracts/GROUP-BY.md`
- `docs/contracts/SQL_SUBSET.md`

## 1. Purpose

This note is the `0.74.0` working inventory for the grouped subsystem.
It does not delete code by itself.
It classifies grouped files by ownership role so later cleanup slices can remove residue without guessing.

The deletion filter is:

> Planner owns semantics. Runtime consumes. Explain, fingerprint, and continuation project. Anything else is a cleanup candidate.

## 3. Current Cleanup Candidates

### 3.1 Executor Eligibility Wrapper Residue

* `0.74` status:
  * fixed
  * grouped route capability projection now flows directly through `GroupedExecutionModeProjection`
  * the grouped eligibility wrapper and its dedicated helper module were deleted

### 3.2 Aggregate-Family Taxonomy Watch Item

* File: `crates/icydb-core/src/db/query/plan/semantics/grouped_strategy.rs`
* Candidate:
  * `GroupedPlanAggregateFamily::GenericRows`
* Why it is suspect:
  * `StorageKeyTerminals` and `Mixed` have already been collapsed into `GenericRows`.
  * Runtime still only makes active decisions on `CountRowsOnly` and `FieldTargetRows`.
  * `GenericRows` is the right coarse fallback bucket for now, but it should not regrow into a descriptive taxonomy without a real runtime consumer.
* Expected `0.74` direction:
  * keep `GenericRows` coarse unless a later grouped runtime path truly consumes a more specific family

### 3.3 Runtime Family Guard Residue

* Files:
  * `crates/icydb-core/src/db/executor/aggregate/runtime/grouped_fold/mod.rs`
  * `crates/icydb-core/src/db/executor/aggregate/runtime/grouped_fold/engine_init.rs`
  * `crates/icydb-core/src/db/executor/pipeline/contracts/grouped/route_stage/projection.rs`
* Candidates:
  * `is_single_count_rows()` dispatch split
  * any remaining grouped runtime guard that rejects `GenericRows` just because one aggregate has a target field
* Why they are suspect:
  * These are valid guardrails today, but they still reflect a world where grouped aggregate-family routing is being proven at runtime.
  * Once grouped engine routing fully consumes the planner-owned aggregate family, some of these invariants should disappear.
* Expected `0.74` direction:
  * keep only the guardrails that still defend a distinct runtime contract
  * delete the ones that only restate planner-owned family separation
* `0.74` status:
  * fixed
  * the unsupported grouped field-target kind check no longer lives in grouped fold engine init
  * grouped state construction now owns that fail-closed invariant, so grouped engine initialization stays mechanical
  * the dedicated `CountRowsOnly` split is still live on purpose because it routes to a genuinely different grouped fold/finalize contract rather than re-validating planner semantics
  * grouped fold runtime now consumes one planner-carried `GroupedFoldPath` contract instead of consulting `GroupedPlanStrategy` directly
  * grouped route-stage payloads no longer carry planner strategy after route projection, so runtime sees only execution-mechanical grouped artifacts

### 3.4 Fingerprint Grouped Projection Split

* File: `crates/icydb-core/src/db/query/fingerprint/hash_parts.rs`
* Candidates:
  * `hash_grouping_shape_v1(...)`
  * `hash_grouping_shape_v1_from_plan(...)`
* Why they are suspect:
  * Both hash grouped shape from planner-owned strategy, but one starts from explain grouping and one from logical plan.
  * This may be necessary, but it is also a likely place for grouped projection drift.
* Expected `0.74` direction:
  * keep both only if the two entry surfaces truly need separate projection inputs
  * otherwise converge onto one grouped strategy projection seam
* `0.74` status:
  * reduced
  * grouped strategy tag hashing and grouped HAVING hashing now share lower-level helpers
  * plan-owned and explain-owned grouped hashing now both project through one canonical grouped fingerprint shape before hashing
  * the top-level plan/explain entrypoints still both exist because one starts from `ExplainGrouping` and one starts from `AccessPlannedQuery`, but they no longer own separate grouped semantic projection rules

### 3.5 Grouped Projection Carriage Split

* Files:
  * `crates/icydb-core/src/db/query/plan/group.rs`
  * `crates/icydb-core/src/db/executor/pipeline/contracts/grouped/route_stage/payload.rs`
  * `crates/icydb-core/src/db/executor/aggregate/runtime/grouped_output/projection.rs`
  * `crates/icydb-core/src/db/executor/projection/grouped.rs`
* Candidates:
  * raw grouped `AggregateExpr` carriage in grouped handoff/runtime payloads
  * grouped projection matching by full builder expression equality
* Why it was suspect:
  * grouped runtime only needed aggregate semantic identity for projection lookup
  * carrying full builder-layer `AggregateExpr` values through grouped route/runtime was a parallel semantics channel
* `0.74` status:
  * fixed
  * grouped runtime now carries `GroupedAggregateProjectionSpec` instead of raw `AggregateExpr`
  * grouped projection/eval now match aggregate outputs through planner-owned semantic projection specs
* Result:
  * planner still owns grouped aggregate semantic identity
  * runtime projection consumes that lighter structural artifact instead of builder expressions

### 3.6 Metrics-Surface Projection Watch Item

* File: `crates/icydb-core/src/db/executor/plan_metrics.rs`
* Candidate:
  * direct use of the metrics sink grouped execution-mode label
* Why it is only a watch item:
  * The executor-local `GroupedPlanMetricsStrategy` shadow enum has already been removed.
  * Metrics projection now carries the sink execution-mode label directly.
  * This seam is now projection-only, but it should stay that way and not regrow executor-local grouped strategy wrappers.
* Expected `0.74` direction:
  * keep direct projection from grouped route observability/runtime execution mode without reintroducing a local duplicate enum
* `0.74` status:
  * reduced
  * grouped route observability and metrics now share `grouped_execution_mode` vocabulary
  * the route-owned materialization label no longer reuses the older `GroupedPlanStrategy` name on the metrics boundary

## 4. First Kill List

These are the first grouped cleanup targets once deletion starts:

1. Delete runtime grouped-family guardrails that no longer protect a distinct runtime path after family routing is fully planner-owned.
2. Reduce grouped fingerprint projection duplication if one grouped strategy projection seam can serve both plan and explain callsites.

One small cleanup already landed on top of this inventory:

* `GroupedPlanStrategy::streaming_admitted()` was renamed to `ordered_group_admitted()` so the grouped planner artifact no longer implies a streaming runtime path that does not exist.
* `GroupedPlanAggregateFamily::{StorageKeyTerminals, Mixed}` was collapsed into `GenericRows` because grouped runtime never distinguished those buckets.
* The executor-local grouped metrics shadow enum was removed, and grouped runtime now carries the metrics sink execution-mode label directly.
* Grouped route observability, grouped execution descriptors, and grouped metrics now use `grouped_execution_mode` naming for the route-owned materialization label, so planner-owned grouped strategy and route-owned grouped execution mode no longer share the word `strategy`.
* Grouped route observability no longer silently defaults a missing grouped execution mode to `HashMaterialized`; grouped routes now require the route-projected execution-mode label to be present.
* Grouped route feasibility now uses `LoadOrderRouteContract::allows_ordered_group_projection()` at the grouped capability gate instead of borrowing the load-only `allows_streaming_load()` wording.
* Grouped route execution-mode projection is now centralized on `GroupedExecutionMode::from_planner_strategy(...)` plus the route-owned `GroupedExecutionModeProjection` contract, and the route-local grouped feasibility wrapper module was deleted.
* Grouped route assembly and grouped route observability now both assert that the carried grouped execution mode matches the canonical planner-strategy-plus-route-capabilities projection, so grouped route code can no longer drift silently from the centralized projection seam.
* Grouped fold runtime now consumes `GroupedFoldPath` instead of reading `GroupedPlanStrategy` directly, and grouped route-stage payloads no longer carry planner strategy once grouped route projection is complete.
* Grouped fingerprinting now projects both plan-owned and explain-owned grouped identity through one canonical grouped fingerprint shape, and explain-side grouped continuation hashing reuses the planner aggregate-family classifier instead of owning a second aggregate-family rule.
* The dead grouped `COUNT(*)` engine-init invariant was removed because generic grouped engine initialization has only one caller and that caller already diverts the `CountRowsOnly` family before engine construction.
* The executor-side field-target guard was narrowed so it no longer rejects `GenericRows` just because one aggregate has a target field, while still failing closed for historically bypassed grouped field-target kinds at that audit point.
* Raw grouped `AggregateExpr` carriage was removed from grouped handoff/runtime payloads in favor of planner-owned `GroupedAggregateProjectionSpec`.

## 5. What Is Not a Cleanup Target

These grouped files should survive, even if their names or helper shape change:

* `query/plan/semantics/grouped_strategy.rs` as the grouped semantic source
* `query/plan/group.rs` as the grouped planner-to-executor carriage boundary
* `query/explain/plan.rs` as logical grouped explain projection
* `executor/explain/descriptor/load.rs` as execution explain projection
* `executor/route/contracts/execution/observability.rs` as grouped route observability projection

The goal is not to minimize file count. The goal is to minimize places where grouped behavior can disagree.
