# 0.101 Remaining Single-Table SQL Surface Completion

## Thesis

`0.101` should complete one **narrow remaining subset** of the practical
single-table SQL surface on top of the now-stable:

* expression-owned `WHERE`
* compiled execution
* DISTINCT / ORDER / paging contracts
* bounded prepared execution

> **`0.101` ships the remaining common scalar-function and null/value-selection
> families that still make the single-table SQL surface feel incomplete,
> without reopening planner architecture, grouped ownership, or broad prepared
> template widening.**

This slice is not discovery work. It is a **locked remaining-inventory**
release contract.

---

## Current baseline

This document starts from the current tree, not from the older `0.99`
baseline.

Already shipped before `0.101`:

* expression-owned scalar `WHERE`
* grouped pre-aggregate `WHERE` on the same residual-filter seam
* searched `CASE`
* `BETWEEN` / `NOT BETWEEN`
* literal-list `IN (...)`
* `IS TRUE` / `IS FALSE`
* text scalar functions already in the function catalog:
  * `LOWER`
  * `UPPER`
  * `LENGTH`
  * `TRIM`
  * `LTRIM`
  * `RTRIM`

So `0.101` must not restate those as new work.

---

## Problem statement

After `0.100`, the core SQL engine is strong, but the practical single-table
surface still has a few visible gaps:

* common numeric scalar functions are still incomplete
* common null/value-selection functions are still missing
* one bounded prepared-parameter membership shape may still be worth admitting
* users can still hit “simple SQL” gaps even though the execution architecture
  is already mature

The main risk is now scope drift, not architecture uncertainty.

So `0.101` must behave as a release contract:

* it states exactly what new surface is shipped
* it states exactly what remains fail-closed
* it keeps clause legality explicit
* it keeps prepared status explicit per admitted family

---

## Design goal

Complete the remaining high-value practical subset of the single-table SQL
surface so that:

* common numeric scalar transforms are available
* common null/value-selection forms are available
* one bounded membership-parameter form can be admitted if explicitly chosen
* the engine feels materially more complete without widening architecture

The goal is:

> **finish the remaining practical single-table surface subset that fits the
> existing expression, execution, and prepared-query model, and stop there.**

---

## Non-goals

`0.101` does **not** aim to:

* add JOIN
* add subqueries
* add window functions
* redesign planner or executor architecture
* widen grouped/global phase ownership
* replace pushdown planning with raw semantic `Expr`
* introduce planner-wide parameter nodes
* widen value-sensitive prepared planning
* broaden fixed-route prepared templates beyond explicitly admitted cases
* reopen already-shipped `0.100` surface families as “new” work

This remains a bounded single-table surface slice.

---

## Explicit current boundary

`0.101` builds on the current single-table expression-first boundary.

That means:

* all new syntax lowers through existing planner-owned expression kinds
* all new runtime behavior reuses existing compiled seams
* scalar and grouped `WHERE` stay on the current residual-filter path
* `HAVING` stays on its own post-aggregate expression seam
* aggregate `FILTER (WHERE ...)` stays on its own aggregate-local seam
* access planning and pushdown remain predicate-derived, not expression-owned
* prepared execution stays bounded by the explicit `0.98` / `0.99` contracts

`0.101` does **not** reopen:

* generic `GROUP BY` expression admission
* new execution architecture
* broad coercion redesign
* broad function-catalog expansion beyond the inventory below

If a form is not listed below, it remains out of scope and fail-closed.

---

## Normative shipped inventory

`0.101` ships **exactly** these new SQL surface families:

### Numeric scalar functions

* `ABS(x)`
* `CEIL(x)`
* `CEILING(x)`
* `FLOOR(x)`

### Null/value-selection scalar functions

* `COALESCE(a, b, ...)`
* `NULLIF(a, b)`

### Optional bounded membership extension

If `0.101` includes membership-parameter follow-through, it ships exactly:

* `IN (?, ?, ...)` with fixed arity and all-parameter lists only

If this family is not taken during implementation, it remains fail-closed and
must be removed from the release notes and status matrix before cut.

---

## Explicitly out of scope in `0.101`

Everything not listed above remains out of scope and fail-closed, including:

* simple `CASE <expr> WHEN ...`
* custom-charset `TRIM`
* new text scalar function families beyond the already-shipped set
* `NULLIF` / `COALESCE`-driven coercion widening beyond existing scalar-family
  rules
* mixed literal/parameter `IN` lists
* expression-list `IN (...)`
* broad prepared-template widening for function-owned shapes
* new clause families

## Clause admission matrix

The matrix below is normative for `0.101`.

Legend:

* `Admitted`: shipped directly in this slice
* `Narrowed`: admitted only under the notes-column restriction
* `Rejected`: fail-closed in this slice

| Feature family | SELECT | WHERE | GROUP BY | HAVING | ORDER BY | Notes |
| --- | --- | --- | --- | --- | --- | --- |
| `ABS` / `CEIL` / `CEILING` / `FLOOR` | Admitted | Admitted | Rejected | Narrowed | Admitted | `HAVING` only when all leaf inputs are aggregate outputs or grouped keys, and the expression introduces no new aggregate calls |
| `COALESCE` / `NULLIF` | Admitted | Admitted | Rejected | Narrowed | Admitted | Same `HAVING` rule as other scalar functions |
| `IN (?, ?, ...)` fixed-arity parameter lists | Rejected by default | Rejected by default | Rejected | Rejected | Rejected | Only becomes `Normalized` / `Fallback prepared only` if explicitly taken in this slice |

`GROUP BY` remains intentionally narrow in `0.101`.

For `0.101`, a scalar expression is admitted in `HAVING` iff:

* all of its leaf inputs are aggregate outputs or grouped keys, and
* the expression introduces no new aggregate calls

Anything else remains rejected fail-closed.

## Prepared-execution status matrix

Each shipped family must carry one explicit prepared status.

Legend:

* `Fixed-route template`: admitted on the existing symbolic-slot fixed-route
  template lane
* `Fallback prepared only`: admitted to prepared execution only through the
  non-template fallback path
* `Not prepared`: not admitted for prepared execution in `0.101`

| Feature family | Prepared status | Notes |
| --- | --- | --- |
| `ABS` / `CEIL` / `CEILING` / `FLOOR` | Fallback prepared only | Function-owned shapes do not widen the fixed-route template lane in `0.101` |
| `COALESCE` / `NULLIF` | Fallback prepared only | Null/value-selection shapes do not widen the template lane in `0.101` |
| `IN (?, ?, ...)` fixed-arity parameter lists | Fallback prepared only if explicitly shipped | No fixed-route template widening is implied |

If a parameterized placement is not explicitly covered above, it remains out of
scope for prepared execution in `0.101`.

## Per-family semantic contracts

This section is normative.

### Common-type resolution (`0.101`)

Where this document requires one admitted common type or result family, a
common type exists only if all participating arguments belong to the same
existing scalar family:

* numeric family
* text family
* boolean family

`0.101` introduces no new cross-family coercion rules.

Mixed-family inputs reject fail-closed.

### Numeric scalar functions

#### `ABS(x)`

* arity: exactly 1
* input: numeric
* result: same numeric family where supported by the existing numeric model
* null behavior: null propagates
* failure mode: non-numeric input rejects fail-closed

#### `CEIL(x)` / `CEILING(x)` / `FLOOR(x)`

* arity: exactly 1
* input: numeric
* result: rounded numeric value in the existing numeric family model
* semantic rule:
  * `CEIL` and `CEILING` are synonyms
  * no new numeric-family widening is introduced
* null behavior: null propagates
* failure mode: non-numeric input rejects fail-closed

### Null/value-selection functions

#### `COALESCE(a, b, ...)`

* arity: variadic, minimum 2
* inputs: arguments must share one admitted common result family under the
  common-type resolution rule above
* result: first non-null argument, or null if all arguments are null
* result type: the admitted common family
* failure mode: incompatible argument families reject fail-closed

#### `NULLIF(a, b)`

* arity: exactly 2
* inputs: arguments must be comparable under the existing equality family rules
* result: null when `a = b`, otherwise the first argument
* result type: type family of the first argument
* failure mode: incomparable argument families reject fail-closed

### Parameter-list `IN (...)` if shipped

* admitted form:
  * `<expr> IN (?, ?, ...)`
* semantic rule:
  * fixed list arity is part of the query shape
  * all list members must be parameter slots
  * the left-hand side must still be legal under the existing equality-family
    semantics
* prepared rule:
  * fallback prepared only
  * no fixed-route template widening is implied
* failure mode:
  * mixed literal/parameter lists reject fail-closed
  * broader expression-list membership rejects fail-closed

## Core implementation contracts

### Contract 1 - No new execution architecture

All shipped families must reuse:

* shared lowering onto planner-owned expressions
* shared type inference / validation
* existing compiled execution seams

No parser-owned or clause-owned semantic islands may be introduced.

### Contract 2 - Phase rules still define legality

Surface breadth widens, but existing phase ownership remains intact:

* pre-aggregate forms stay pre-aggregate
* post-aggregate forms stay post-aggregate
* grouped/global `HAVING` rules stay planner-owned

### Contract 3 - No implicit prepared widening

No shipped family may silently widen the fixed-route prepared-template lane.

Prepared status is exactly what the matrix above says.

### Contract 4 - No new expression node categories

`0.101` must not introduce new planner expression node categories.

All new surface forms must lower into existing expression kinds such as:

* function call
* comparison
* boolean
* `CASE`

### Contract 5 - Keep the `WHERE` / `HAVING` split intact

`0.101` must not try to route `HAVING` through the new `WHERE` residual-filter
path.

The right reuse target is:

* shared expression compilation utilities
* shared boolean/null collapse behavior
* shared explain wording where appropriate

The wrong target is a unified runtime path across phases.

## Verification matrix

Each shipped family requires explicit coverage across the ownership boundaries
that `0.101` touches.

| Feature family | Parser | Lowering / normalization | Execution | Explain | Prepared coverage | Negative coverage |
| --- | --- | --- | --- | --- | --- | --- |
| `ABS` / `CEIL` / `CEILING` / `FLOOR` | Parse call syntax and arity | Lower to builtin function expressions | Execute through shared compiled scalar evaluation | Render as ordinary expressions | Fallback prepared execution only | Reject wrong arity, wrong type, `GROUP BY` use; verify clause legality matrix |
| `COALESCE` / `NULLIF` | Parse variadic/binary forms | Lower to shared expression nodes with common-type validation | Execute through shared scalar evaluation | Render as ordinary expressions | Fallback prepared execution only | Reject incompatible argument families; verify clause legality matrix |
| `IN (?, ?, ...)` fixed-arity parameter lists if shipped | Parse syntax | Normalize to existing equality-membership semantics | Execute on existing membership path | Explain normalized membership shape | Fallback prepared execution only | Reject mixed literal/parameter lists, widened expression lists, and illegal clause placements |

## Risks

### Risk 1 - stale baseline thinking

If implementation treats already-shipped `0.100` surface as new `0.101` work,
the slice will become incoherent and duplicate prior release scope.

### Risk 2 - implementation-led scope growth

If unlisted scalar functions or broader `IN` variants are admitted during
implementation, the slice loses its boundary.

### Risk 3 - grouped boundary drift

It would be easy to let “expression parity” reopen `GROUP BY` or grouped
projection ownership by accident. This document forbids that.

### Risk 4 - implicit prepared widening

New syntax is likely to pressure the prepared-template lane. `0.101` must keep
that pressure fail-closed unless the prepared-status matrix explicitly admits a
shape.

---

## Out of scope

The following stay out of `0.101`:

* any unlisted scalar function family
* simple `CASE`
* custom `TRIM` character syntax
* broad membership-syntax widening beyond optional all-parameter fixed-arity
  lists
* general `GROUP BY` expression widening
* JOIN
* subqueries
* window functions
* value-sensitive prepared route templates
* optimizer redesign
* public API redesign around parameterization

---

## Follow-on

Once `0.101` lands, the engine should have:

* an expression-first single-table filter model
* compiled execution throughout production paths
* a nearly complete practical single-table scalar SQL surface
* bounded prepared behavior with explicit fallback/template rules

That sets up cleaner next directions:

* optimization over a more complete shipped surface
* later multi-table work on top of a less obviously incomplete single-table SQL
  base

---

## Thesis sentence

> **`0.101` completes the remaining practical single-table SQL surface subset
> that still fits the current planner, execution, and prepared-query model by
> shipping a locked inventory of missing numeric and null/value-selection
> function families, with clause legality and prepared status stated explicitly
> per family.**
