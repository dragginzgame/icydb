# 0.196 Checkpoint After 0.196.10

Date: 2026-07-05

## Purpose

This checkpoint maps the 0.196 implementation line back to the original
SQLite-comparison audit and chooses the next high-ROI 196 slice. It is meant to
prevent the line from drifting into narrow instruction-count work while leaving
the broader audit backlog untracked.

## Current State

0.196 has not become "just perf", but the last slices have been concentrated in
the ordered-read planner/executor/perf lane.

That is still inside the original 196 scope because the SQLite comparison
identified bounded ordered reads, route evidence, result-signature proof, and
full-matrix deltas as the first high-value implementation target. The risk now
is diminishing returns: further tiny instruction-count optimizations are lower
ROI than correctness guards and invariant hardening around the routes that were
just changed.

## Current Evidence

The latest 0.196.10 full matrix closeout recorded:

- 1,757 generated deterministic scenarios;
- 1,676 executed scenarios;
- 81 existing expected/known failures;
- 0 new failures;
- 0 resolved failures;
- 0 result-signature changes;
- 0 cursor-signature changes;
- 0 route fact changes;
- 0 regressions at or above the closeout gate threshold of `>=10%` and
  `>=100k` instructions.

The same delta also showed:

- 1,345 common-success scenarios improved;
- 331 common-success scenarios regressed;
- aggregate common-success total instructions increased by 5,347,464, mostly in
  materialized storage-mirror scenarios rather than route or result changes.

Conclusion: 0.196.10 is semantically clean and has focused compile/bind wins,
but it is not a broad aggregate performance win. The next 196 slice should not
be another small compile micro-optimization.

## Drift Assessment

We have not drifted from 196's SQLite-comparison scope because the implementation
work has been about:

- proving bounded ordered read behavior;
- making route choice observable;
- preventing cherry-picked performance claims;
- using deterministic matrix evidence;
- fixing a cursor continuation correctness bug found in the same route family;
- reducing compile overhead where matrix evidence showed it.

We would drift if the next patches continued to optimize hand-picked scenarios
without improving:

- route correctness confidence;
- invariant coverage;
- diagnostic trustworthiness;
- or a clearly measured full-matrix hotspot.

## Recommended Next Slice: Route-Invariant Tripwires

The next 196 patch should be a correctness/testing slice, not a perf slice.

Recommended title:

`0.196.11 Route-Invariant Tripwires`

Current worktree status:

- the 0.196.11 slice adds a stream fast-path precedence source guard;
- it adds primary-order pushed-vs-materialized oracle parity;
- it adds equality-prefix ordered-suffix pushed-vs-forced-full-scan parity;
- it extends branch-set forced-fallback parity to duplicate branch literals;
- it adds a branch-set cursor mutation regression for deleted boundary and
  unseen rows between pages;
- it updates the fast-path inventory so stream precedence is no longer listed
  as unguarded.

Primary audit findings addressed:

- `SQLITE-COMP-006`: fast-path ownership still has documented unguarded areas.
- `SQLITE-COMP-001`: cursor-boundary pushdown needs no duplicate/no skip
  guardrails around changed route families.
- `SQLITE-COMP-005`: materialized ordered-limit diagnostics should remain
  explicit when a route cannot push.

Why this is the best ROI now:

- It is small compared with differential testing or storage-format work.
- It protects the code just changed in 0.196.7-0.196.10.
- It moves 196 back toward hardening instead of chasing diminishing perf.
- It can land without cursor-format, persisted-format, cache-key, public API, or
  storage changes.

### Scope

Add tests/guards for:

1. Stream fast-path precedence helpers.
   - Ensure pushed primary/secondary/equality-prefix routes are selected only
     when route facts prove order compatibility.
   - Ensure materialized/residual routes remain explicitly classified when
     order compatibility is not proven.

2. Pushed-vs-fallback semantic parity.
   - For at least one changed primary-order route, one secondary branch route,
     and one equality-prefix route, compare the pushed result order/row ids
     against an intentionally materialized or scan-style oracle.
   - Include duplicate ordered values and primary-key tie-breakers.

3. Cursor/page boundary invariants for changed route families.
   - No duplicate rows across pages.
   - No skipped rows across pages.
   - Boundary row deleted before next page.
   - Equal ordered key inserted between pages.
   - ASC and DESC where the current route supports both.

4. Fast-path inventory linkage.
   - Update `docs/governance/fast-path-inventory.md` so every guarded area links
     to the new test or explicitly remains unguarded with a reason.

### Non-Goals

- No new route optimization.
- No new public API.
- No cursor-token shape change.
- No persisted-format change.
- No benchmark headline.
- No SQLite harness in this slice.
- No schema migration work in this slice.

### Acceptance Criteria

- Focused tests pass for the new guard/parity cases.
- Existing cursor/order tests still pass.
- `docs/governance/fast-path-inventory.md` no longer lists stream fast-path
  precedence as unguarded unless the doc names a concrete blocker.
- `cargo fmt --all` passes.
- `cargo clippy -p icydb-core --all-features --all-targets -- -D warnings`
  passes if production/test Rust code is touched.
- No full deterministic matrix is required unless production route/executor
  behavior changes.

### Suggested Patch Prompt

Implement `0.196.11 Route-Invariant Tripwires`.

Do not add a new optimizer path. Add focused tests and source guards that lock
the route families changed during 0.196.7-0.196.10:

- pushed primary-order route parity against a materialized/scan oracle;
- pushed secondary branch-route parity, including equal ordered values and
  primary-key tie-breakers;
- equality-prefix ordered-suffix parity;
- cursor/page boundary no-duplicate/no-skip cases for the changed route
  families;
- stream fast-path precedence ownership guard.

Update `docs/governance/fast-path-inventory.md` to link the new guards. Do not
change cursor token format, persisted format, public API, cache identity, or
read-admission semantics. Run focused tests, `cargo fmt --all`, and clippy for
`icydb-core`.
