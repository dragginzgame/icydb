# IcyDB 0.180 - Compact Diagnostic Codes

## Status

Closed design for 0.180.

0.180 should move IcyDB diagnostics from text-first public errors toward
compact machine-readable diagnostic codes with optional host-side text
rendering. It is intentionally scoped as a breaking pre-`1.0.0` hard cut.

Closeout uses a compact numeric public `ErrorCode` rather than a wide public
`detail` field. The code space carries stable leaf identities for known detail
families, and host/CLI code reconstructs `DiagnosticDetail` through
`Error::diagnostic()` when rich rendering is needed.

## Purpose

Reduce live canister Wasm footprint and stabilize machine-readable failures by
moving production diagnostics from text-first errors to compact codes and
structured details.

The CLI and local/debug tooling should own rich human text. Production canister
Wasm should return compact diagnostic identity and should not construct long
diagnostic prose for known errors.

## Background

The 2026-06-06 wasm-footprint audit showed that `one_sql_query` is the largest
current audit fixture:

- `one_sql_query` shrunk Wasm: `2,955,294` bytes;
- `one_fluent_query` shrunk Wasm: `2,389,137` bytes;
- SQL-over-fluent gap: `566,157` bytes;
- SQL data segment: `241,200` bytes;
- fluent data segment: `178,624` bytes;
- SQL-over-fluent data segment gap: `62,576` bytes.

String extraction suggests that compact diagnostics will not erase the whole
SQL cost. The likely recoverable area is tens of KB, not hundreds. The larger
SQL gap also includes parser, planner, result, Candid, and execution code.

Still, diagnostic prose is one of the most tractable live-Wasm data costs. It
also makes the public error contract less precise than it should be.

## Problem

The public facade currently returns an error payload with a text message:

```text
Error {
  kind,
  origin,
  message: String,
}
```

Core errors also store rendered messages and many call sites construct
human-readable strings before the facade converts them for canister callers.

Moving text rendering only in the CLI is not enough if core still links and
constructs long static messages. To reduce live Wasm, the production path must
be code/detail-first before message construction.

## Goals

- Keep machine-readable error identity stable and explicit.
- Remove long human diagnostic prose from production canister error paths.
- Let the CLI render friendly messages, hints, and examples from shared codes.
- Allow local/debug builds to opt back into rich text without making it part of
  the live canister default.
- Avoid string matching in tests by asserting codes and structured details.
- Preserve enough structured detail for automation and support diagnostics.

## Non-Goals

- Do not preserve compatibility with the current text-first error wire before
  `1.0.0`.
- Do not move SQL parser/planner code out of canisters in this design.
- Do not promise a large Wasm reduction from diagnostics alone.
- Do not make generated models or the CLI authoritative for runtime errors.

## Proposed Crates

Use two separable layers.

### `icydb-diagnostic-code`

Small production-safe crate.

Responsibilities:

- stable diagnostic code enums or compact numeric ids;
- error class and origin taxonomy;
- small structured detail payloads;
- Candid/serde support only where needed for public wire contracts;
- no long prose, hints, examples, or formatted paragraphs.

This crate may be depended on by `icydb-core`, `icydb`, generated canisters,
integration fixtures, and CLI code.

### Text Renderer

Optional renderer crate, or CLI-owned module if a crate proves unnecessary.
0.180 chose the CLI-owned module.

Responsibilities:

- map diagnostic codes and structured details to human messages;
- add hints, examples, and remediation text;
- render stable CLI output;
- support docs/tests that need rich text.

This layer must not be part of production canister defaults. It may be enabled
by CLI, host tooling, local development, and explicit debug features.

## Public Wire Shape

Initial target live canister error payload:

```text
IcydbError {
  code: IcydbErrorCode,
  class: ErrorClass,
  origin: ErrorOrigin,
  detail: Option<IcydbErrorDetail>,
}
```

Final 0.180 implementation keeps the same semantic payload but packs the
stable detail identity into a compact numeric wire code:

```text
Error {
  code: u16,
  class: u8,
  origin: u8,
}
```

`code` is an `ErrorCode` leaf. It preserves the broad `DiagnosticCode` and,
for known compact families, enough detail to reconstruct a `Diagnostic` for
host-side rendering. This keeps the production canister wire small while
retaining machine-readable diagnostic identity.

Possible code shape:

```text
E_QUERY_VALIDATE
E_QUERY_PLAN
E_QUERY_UNORDERED_PAGINATION
E_QUERY_UNSUPPORTED_SQL_FEATURE
E_SCHEMA_DDL_MISSING_EXPECTED_VERSION
E_SCHEMA_DDL_PUBLICATION_RACE_LOST
E_STORE_NOT_FOUND
E_STORE_CORRUPTION
E_RUNTIME_INTERNAL
```

Codes should be stable enough for CLI and external callers. The string spelling
is for documentation and CLI display; the wire may use enum discriminants or
compact variant labels if that proves smaller.

## Structured Details

Details should carry data that callers may act on without parsing text:

```text
UnsupportedSqlFeature { feature: SqlFeatureCode }
SchemaDdlAdmission { reason: SchemaDdlAdmissionCode }
StoreKey { key_hash_or_short_key }
Field { entity, field }
Index { entity, index }
VersionContract { expected, accepted, next }
```

Details should avoid embedding full prose. For live Wasm, details should also
avoid large arbitrary strings unless the data is user input or entity metadata
needed for the caller to recover.

## Minimum Useful Context Plan

Compact diagnostics must not mean context-free diagnostics. The target is to
replace rendered prose with the smallest useful typed context that lets host
tooling, the CLI, tests, and support workflows reconstruct an actionable
message from the error code plus the original query/schema/runtime shape.

Rules:

- Do not store rendered sentences in core diagnostic payloads.
- Do preserve actionable context when it identifies what the caller should fix.
- Prefer indexes into already-known structures over names or rendered labels.
- The main size and design win is typed context instead of strings. Do not
  over-optimize integer width before the owning shape has a real bound.
- Use the natural non-lossy type for the boundary:
  - `usize` is fine for core-internal indexes into `Vec`/slice-backed query,
    plan, projection, cursor, aggregate, and primary-key component structures.
  - `u8`/`u16` are preferred only when the owning shape already has an
    architectural bound or when a public wire/detail type needs a stable small
    representation.
  - `u16` remains a good fit for schema field slots, row slots, index key
    positions, and compact counts when those domains are already bounded that
    way.
  - `u32` remains a good fit for schema versions, offsets, row counts, byte
    lengths, and existing runtime counters whose range is already wider.
  - compact enums with `#[repr(u8)]`-compatible intent for operator, function,
    aggregate-kind, value-kind, component-kind, field-kind-family, and
    subsystem/component codes.
- Never truncate context to fit a smaller type. If a value has no hard bound,
  use `usize` internally or the next safe public width. Omit it only after
  documenting why it is not actionable.
- Public/canister diagnostics may expose only stable compact context. Private
  core errors may carry richer typed context if it is still small and does not
  force text into live Wasm.
- Context should be enough to point at the failing part, not necessarily enough
  to reproduce the whole rendered old message.
- A diagnostic may remain code-only only when additional context is genuinely
  not recoverable or not actionable.

Recovery workflow:

1. Freeze further "string removal" work that collapses useful context to unit
   variants.
2. Inventory each diagnostic family with:
   - current code;
   - current context;
   - context removed during 0.180;
   - minimum useful context;
   - natural typed representation;
   - whether the context is public wire, core-only, or renderer-only.
3. Add or extend compact context/detail types before changing more call sites.
4. Retroactively repair 0.180 diagnostics that lost useful context by restoring
   typed indexes, counts, enum codes, or bounded ids instead of strings.
5. Update tests to assert compact context values, not rendered text.
6. Rerun Wasm audits after each repair slice to confirm the context restoration
   does not bring text payloads back into live canisters.

### 0.180 Context Recovery Checklist

The following families need review before 0.180 closes.

#### Query Planner And SQL Frontend

- `OrderPlanError`
  - Restore: order term index, first/duplicate term index, missing primary-key
    component index.
  - Suggested types: `usize` for core-internal term/component indexes unless
    order specs gain an architectural bound or the context moves onto public
    wire.
  - Do not restore: rendered field names.
- `ExprPlanError`
  - Restore: aggregate kind, function code, argument index, unary/binary
    operator code, CASE arm index, branch index, and compact inferred type or
    value-kind family where useful.
  - Suggested types: compact enums for aggregate/function/operator/type-family,
    `usize` for core-internal argument/arm/branch indexes.
  - Do not restore: debug-formatted `ExprType`, rendered expressions, or field
    names unless the field identity is already part of the accepted query
    surface.
- `SqlLoweringError` and SQL write boundary errors
  - Keep: existing feature/boundary/reason codes.
  - Review for: width mismatch expected/found counts, returning/projection
    indexes, parameter index, and clause/source position where this helps CLI
    rendering.
  - Suggested types: `usize` for internal counts and projection indexes,
    narrower fixed widths only for stable public details, `u16`/`u32` for
    source spans if spans are retained.
- `GroupPlanError`
  - Restore where compacted later: group key index, aggregate index, duplicate
    key index pair, unsupported aggregate kind, HAVING compare index, and
    compact field/slot identity.
  - Suggested types: `usize` for core-internal group/aggregate/HAVING indexes,
    compact enum for aggregate kind and compare operator, existing slot/id
    types for field identity.
  - Do not restore: rendered grouped order terms or field names unless needed
    as user-authored SQL identifiers and no compact slot/index exists.

#### Cursor And Continuation Diagnostics

- `CursorPlanError`
  - Restore: payload subreason code, signature comparison category,
    boundary expected/found arity, expected/actual offsets, order term index,
    primary-key component index, expected type-family, and actual value kind.
  - Suggested types: compact subreason enums, `usize` for core-internal indexes
    and arity, `u32` for offsets.
  - Do not restore: full signatures, rendered field names, or formatted values
    in live canister defaults.
- Cursor token encode/decode errors
  - Restore if actionable: token section code and length/offset for malformed
    payloads.
  - Suggested types: compact token-section enum, `u16`/`u32` for lengths and
    offsets.

#### Executor, Aggregation, Projection, And Runtime

- `AggregateFieldValueError`
  - Restore: target field slot/index, aggregate kind where known, expected
    field-kind family, and actual value kind.
  - Suggested types: existing slot/index types or `usize` internally, compact
    enums for aggregate, field-kind family, and value kind.
  - Do not restore: field names, full `Value` debug output, or field-kind
    rendered text.
- Aggregate reducer/state mismatch diagnostics
  - Restore: aggregate kind and reducer/state code.
  - Suggested types: compact enum codes.
- Projection and compiled-expression diagnostics
  - Restore where useful: projection index, group-key offset, aggregate output
    index, function/operator code, and actual value kind.
  - Suggested types: `usize` internally for indexes and compact enums for
    operation/value categories.
- `KeyCanonicalError`
  - Keep code-only if no recoverable cause exists.
  - Restore only compact value-kind or component code if the failure path can
    identify it without formatting.

#### Store, Codec, Commit, Journal, And Index Internals

- Store/codec corruption and persisted-row decode diagnostics
  - Restore: component code, format version byte, slot index, length/limit
    counts, and expected/found shape code where this identifies the corrupt
    segment.
  - Suggested types: compact component enum, `u8` for format versions,
    `u16` for slots/components, `u32` for byte lengths.
  - Do not restore: long decode labels, source-error text, or full key/value
    debug output.
- Commit marker, control-slot, and journal diagnostics
  - Restore: record/component code, entity tag or store id when available,
    row-op kind, payload length/limit, and mismatch category.
  - Suggested types: compact component enum, entity tag/id numeric type,
    `u32` lengths, compact mismatch enum.
  - Do not restore: store paths, entity paths, or formatted key bytes unless
    the path is already part of an explicit public recovery surface.
- Index/key diagnostics
  - Restore: index id, key component index, key namespace code, expected/found
    arity, and compact value-kind.
  - Suggested types: existing compact id types where available, `usize`
    internally for component indexes and arity.

#### Primary Key And Type Conversion

- Primary-key encode/decode errors
  - Restore: component index, expected component kind, actual value kind,
    component count, and range/overflow category.
  - Suggested types: `usize` for internal component indexes and counts unless a
    public wire/detail bound exists, compact enums for component/value/range
    categories.
  - Do not restore: type names or full value debug output in live defaults.
- Numeric/decimal/type helpers
  - Keep stable numeric codes for common errors.
  - Restore only compact operation or scale/range category when it changes how
    the caller should respond.

#### Public Diagnostic Detail Expansion

Existing `DiagnosticDetail` covers categorical reasons well, but it does not
yet model compact parameters. Add parameterized detail shapes only when the
context is stable and useful across the public boundary, for example:

```text
QueryPlanContext {
  reason: QueryPlanCode,
  primary_index: u16,
  secondary_index: Option<u16>,
  expected: Option<u32>,
  actual: Option<u32>,
}

RuntimeContext {
  reason: RuntimeContextCode,
  component: u8,
  index: Option<u16>,
  expected: Option<u32>,
  actual: Option<u32>,
}
```

If a context field is only useful inside core tests, keep it core-only and let
the public error collapse to the stable code.

## Text Rendering

CLI/local rendering should be a pure mapping from compact diagnostic payloads
to text:

```text
E_SCHEMA_DDL_PUBLICATION_RACE_LOST: schema publication lost a race
hint: refresh the accepted schema and retry the DDL statement
```

Rules:

- rich text lives in CLI-owned renderer code;
- renderer can use long prose, hints, and examples;
- renderer must not be linked by production canister defaults;
- tests for user-facing CLI output may assert text;
- core/canister tests should assert codes/details.

## Feature Gates

Original suggested features:

- `diagnostic-text`: enables local/debug human text rendering in `icydb`;
- `sql`: continues to gate SQL support;
- production/default canister builds keep `diagnostic-text` off;
- CLI enables text rendering because it is a host binary, not canister Wasm.

Final 0.180 shape does not add a canister-facing `diagnostic-text` feature.
Rich text lives in the CLI crate, and default generated canister builds link
only the compact diagnostic-code layer.

In production canister builds:

- `Error::message()` should be removed, unavailable, or return only a tiny
  fallback code label;
- known internal constructors should not allocate/render human prose;
- `thiserror` display strings should not be relied on for live user output.

## Internal Error Model

Core should move from text-first:

```text
InternalError { class, origin, message: String, detail }
```

to diagnostic-first:

```text
InternalError {
  code,
  class,
  origin,
  detail,
  debug_context,
}
```

`debug_context` must be absent or tiny in production builds. If a dynamic
string is necessary, it should represent user data or compact context, not a
full rendered sentence.

Constructor helpers should choose codes/details. They should not format
sentences on production canister paths.

## SQL Feature Codes

SQL unsupported feature diagnostics are an early high-value target because
parser/session diagnostics currently carry long labels.

Move from:

```text
UnsupportedSqlFeature { feature: &'static str }
```

to:

```text
UnsupportedSqlFeature { feature: SqlFeatureCode }
```

Examples:

```text
OrderByUnsupportedForm
DropStatementBeyondDropIndex
AlterTableUnsupportedOperation
ParameterizedSchemaVersion
MultiStatementSql
WindowFunctions
Join
Union
```

The CLI renderer can map those to actionable messages.

## Result Surface Interaction

Compact errors are necessary but not sufficient. The SQL audit also shows that
wide public result types increase Candid/type surface.

Follow-up work should consider narrow endpoint result types:

- count-only SQL endpoints can return a smaller count result;
- generic SQL endpoints can keep the full SQL result enum;
- debug/admin endpoints can opt into describe/show/explain/DDL result shapes;
- generated endpoint defaults should stay minimal and explicit.

This is separate from diagnostic-code work but should be measured together.

## Testing Strategy

Production/core tests:

- assert diagnostic codes and structured details;
- avoid matching human message strings;
- prove SQL parser unsupported-feature cases emit stable feature codes;
- prove DDL admission keeps stable schema DDL codes;
- prove store/runtime classes and origins survive conversion.

CLI tests:

- assert rendered text for representative codes;
- assert hints for common recovery cases;
- assert unknown/future code fallback rendering.

Wasm tests/audits:

- compare `one_sql_query` against `one_fluent_query`;
- track data segment bytes and printable string bytes;
- verify CLI-owned diagnostic text rendering is not linked into default
  canister builds.

## Migration Notes

This is a breaking pre-`1.0.0` hard cut.

Downstream callers that currently inspect `Error.message` must switch to
`Error.code`, `Error.class`, and `Error.origin`. Host-side callers that need
the reconstructed compact detail can call `Error::diagnostic()`.

CLI and local tooling may continue to display rich text. Live canister callers
should treat the error payload as structured data.

## Closeout Decisions

- Public canister errors use compact numeric wire fields:
  `code: u16`, `class: u8`, and `origin: u8`.
- `icydb-diagnostic-code` owns stable code/detail taxonomies and the
  code-to-detail reconstruction table.
- Rich text rendering remains CLI-owned. No separate
  `icydb-diagnostic-text` crate was needed for 0.180.
- Default canister builds do not enable a diagnostic text renderer.
- Context recovery favors compact typed detail and natural indexes/counts over
  rendered sentences. Schema validation and user-defined sanitizer/validator
  messages may remain textual where caller-provided feedback is the feature.
- Core/canister tests should assert compact codes and structured detail;
  CLI/user-output tests may assert rendered text.

## 0.180 Closeout Standard

This work can close when:

- production canister public errors no longer carry long message strings;
- CLI renders rich messages from compact codes/details;
- SQL unsupported-feature diagnostics use compact feature codes;
- DDL admission diagnostics still expose stable machine-readable reasons;
- default canister builds do not link the text renderer;
- wasm-footprint audit records the raw `.wasm` and data-segment impact.

0.180 closeout status:

- public `icydb::Error` is compact `code`/`class`/`origin` wire data;
- CLI rendering maps compact diagnostics to host-side text;
- SQL unsupported features use `SqlFeatureCode`;
- DDL admission diagnostics map to `SchemaDdlAdmissionCode`;
- production canister defaults have no diagnostic text feature or renderer
  dependency;
- the 2026-06-11 wasm-footprint audit records raw shrunk `.wasm`, gzipped
  `.wasm`, and data-section bytes for the audited canister set.
