# IcyDB 0.183 - Branch-Aware Query Optimization Roadmap

## Status

Closed historical roadmap for 0.183.

0.183 delivered the first branch-aware composite-prefix route plus related
count, covering, perf-attribution, and SQL/fluent query cleanup. The line is
now closed for further branch-aware expansion. 0.184 owns the mega-audit
cleanup pass; 0.185 owns the branch-aware query revisit.

0.183 turned the 0.182 SQL performance audit line into targeted planner and
executor work for production-shaped generated list endpoints. The immediate
driver was a generated list/page query with:

```text
collection_id = "01KV5N439P0000000000000000"
AND stage IN ("Draft", "Review")
ORDER BY id ASC
LIMIT 50
```

The endpoint shape also needed an explicit count policy. A page request must not
implicitly pay for an exact total count unless the generated surface asks for a
separate count result.

## Summary

The query needed optimization on five axes at once:

- access planning must recognize `fixed equality prefix + IN set + primary-key
  order`;
- execution must merge multiple branch streams without materializing or sorting
  the full match set;
- pagination must stop after the requested page plus continuation lookahead;
- `COUNT(*)` must run through covered index/count paths instead of hydrating
  rows;
- generated endpoints must avoid eager totals and must pass the real backend
  limit/cursor shape.

The target index for the production shape is:

```text
(collection_id, stage, id)
```

With `collection_id` fixed and `stage` restricted to a small set, each branch
is an `id`-ordered stream:

```text
(collection_id = X, stage = Draft, id ASC)
(collection_id = X, stage = Review, id ASC)
```

The executor can then merge those branch streams by `id` and stop after
`limit + 1` rows for `has_more`. That gives the page path selectivity and the
required global primary-key order without scanning the whole collection, sorting
all matches, or evaluating `stage` as a residual row predicate.

## Current Source Reality

The 0.182 line already added the measurement tools needed to prove this work:

- SQL perf attribution now separates `data_store.get`, `index_store.get`, and
  `index_store.entries`.
- The perf audit matrix includes SQL membership, count, filtered-index, and
  ordered-limit scenarios.
- The scalar executor already has pull-based key streams and bounded scan
  hints.
- Secondary index range/prefix streams already exist for order-preserving leaf
  scans.
- Composite access plans already have union/intersection nodes and ordered key
  stream merge/intersection combinators.

The missing planner shape is narrower:

- `IN` currently plans as a leading-slot multi-lookup only.
- `AND` equality-prefix assembly only consumes equality literals.
- A composite index such as `(collection_id, stage, id)` cannot yet become a
  native branch set for `collection_id = X AND stage IN (...)`.
- Without that branch shape, the planner tends to choose between:
  - a route that preserves `id` order but leaves `stage` as residual;
  - a route that is selective on `stage` but does not preserve global `id`
    order;
  - a materialized fallback that sorts after scanning too many candidates.

`COUNT(*)` has a separate issue: even when it avoids row projection, an exact
filtered count still needs to enumerate candidates unless the matching access
route is fully covered and eventually has exact prefix cardinality metadata.

## Problem

The production query blows the instruction limit when the page path, count
path, or both perform work proportional to the whole collection or whole
matching set instead of the requested page.

The dangerous combinations are:

- `LIMIT 50` is applied only after residual filtering or materialized ordering.
- `COUNT(*)` is executed eagerly for pagination totals and scans every match.
- `stage IN (...)` is not part of the composite prefix, so `stage` remains a
  residual predicate under a broad `collection_id` scan.
- The planner chooses a primary-key ordered route and tests
  `collection_id/stage` row-by-row.
- The planner chooses a selective `(collection_id, stage, id)` route but cannot
  prove global `ORDER BY id`, so it materializes/sorts before paging.
- Continuation resumes by rescanning earlier branch entries instead of
  continuing after the last emitted `id`.

## Goals

1. Make the page path for the target query bounded by `limit + lookahead` plus
   a small branch-stream overhead.
2. Make the count path independent from the page path; generated list endpoints
   should not do an eager total unless requested.
3. Plan `Eq(a) AND In(b)` against composite indexes where `a, b` form a leading
   prefix.
4. Preserve global primary-key order across multiple fixed-prefix branch
   streams when the ordered suffix is the primary key.
5. Carry continuation tokens that resume the merged branch set without
   replaying the first page.
6. Strip access-proven predicates so covered branch access does not retain
   redundant row filters.
7. Execute exact covered counts without row-store reads.
8. Add clear EXPLAIN/perf attribution so failures are visible as route-choice,
   branch-merge, residual-filter, row-read, or count issues.

## Non-Goals

- Do not build a broad cost-based SQL optimizer in this minor line.
- Do not implement approximate counts.
- Do not preserve legacy cursor compatibility before `1.0.0`; cursor shape may
  hard-cut if branch continuations need a new signature.
- Do not make generated models authoritative for runtime access decisions.
- Do not make SQL DDL the source of schema mutation semantics.
- Do not optimize arbitrary large `IN` lists before the small branch-set shape
  is correct and bounded.

## Target Runtime Shape

The planner should lower the target predicate/order pair into a branch-aware
access contract:

```text
CompositePrefixBranchSet {
  index: tokens_by_collection_stage_id,
  fixed_prefix: [collection_id = X],
  branch_slot: stage,
  branch_values: [Draft, Review],
  ordered_suffix: [id ASC],
  order: primary_key_asc,
}
```

This may be represented as a new `AccessPath` variant or as a union of
index-prefix/range children plus an explicit order-preservation contract. The
important invariant is that downstream route planning can distinguish this
shape from a generic unordered union.

Execution should be pull-based:

1. Open one dynamic index stream per branch.
2. Merge branch heads by decoded primary key for `ORDER BY id`.
3. Suppress duplicates defensively.
4. Stop after `limit + 1` emitted rows for cursorless first-page calls.
5. Emit a cursor anchored at the last returned `id` when more rows exist.

For page 2 and later, the continuation should apply `id > last_id` to every
branch in ascending mode, or `id < last_id` in descending mode. This avoids
storing one cursor anchor per branch for the primary-key ordered shape.

## Count Policy

Generated list endpoints should expose three distinct policies:

- `page_only`: return rows plus `has_more`; never count.
- `count_only`: return exact count for the same filter.
- `page_with_count`: run page and count explicitly, with separate attribution.

`page_with_count` should be opt-in because exact count can remain expensive
until prefix cardinality metadata lands.

The 0.183 count progression should be:

1. Covered branch count: sum exact index-entry membership across branch ranges
   without row-store reads when the predicate is fully proven by access.
2. Count route diagnostics: report whether count used row scan, index entry
   scan, branch index count, or cardinality metadata.
3. Prefix cardinality metadata: maintain exact counts for accepted index
   prefixes such as `(collection_id)` and `(collection_id, stage)` so the target
   count becomes proportional to branch count rather than matching entry count.

The first step is enough to remove row hydration and materialized sorting from
counts. The metadata step is the larger win when collections become large.

## Index Guidance

For this production query, `(collection_id, stage, id)` is the preferred index.

`(collection_id, id, stage)` preserves `id` order under a collection prefix but
keeps `stage` as residual, so sparse stages can still scan too much.

`(collection_id, stage, id)` makes both filter clauses part of the index prefix.
The apparent order mismatch is solved by branch merging because each fixed
stage branch is internally ordered by `id`.

The engine should also accept generated/API filters expressed as:

```text
And([Eq(collection_id, X), Or([Eq(stage, Draft), Eq(stage, Review)])])
```

and canonicalize that into the same membership shape as:

```sql
collection_id = X AND stage IN ('Draft', 'Review')
```

## Planner Work

Planner changes should stay narrow and explicit:

- Add a normalized equality-set model for conjunctive predicates that can hold
  either one exact value or a small set of exact values per field/key item.
- Extend composite-prefix assembly to consume one or more `IN` slots after
  fixed equality slots.
- Cap branch expansion with a conservative branch-count limit; fall back to
  current behavior for large sets.
- Rank branch-prefix candidates above broad ordered scans when they satisfy the
  requested order through primary-key suffix merge.
- Strip predicates proven by branch access from the residual runtime filter.
- Surface the selected branch shape in EXPLAIN and perf diagnostics.

The first admitted shape should be:

```text
Eq(prefix_0) AND In(prefix_1) ORDER BY primary_key
```

with all directions uniform and the primary key present as the ordered suffix.

## Executor Work

Executor changes should reuse the existing stream machinery:

- Lower each branch to an index prefix/range stream over the same accepted
  index contract.
- Preserve leaf index order when the branch suffix is the final
  caller-visible order.
- Merge branch streams lazily instead of materializing every branch key.
- Make the scan budget apply to emitted rows, not to every branch
  independently.
- Keep row materialization mode selected from projection coverage:
  - key-only/projection-covered rows should avoid `data_store.get`;
  - non-covered projections should read only returned rows plus lookahead;
  - residual-filtered fallbacks remain visible in attribution.

The initial branch merge can be primary-key-order-only. Broader merge keys can
wait until a separate design proves cursor semantics for non-primary order
terms.

## Continuation Work

Branch-aware cursors need a new continuation signature containing:

- selected index identity and schema version;
- fixed prefix fields and canonical values;
- branch slot and branch values;
- ordered suffix direction;
- last emitted primary key.

For the first shape, a single global primary-key anchor is sufficient. On
resume, every branch adds the anchor range to its prefix:

```text
(collection_id = X, stage = Draft, id > last_id)
(collection_id = X, stage = Review, id > last_id)
```

If a future shape needs non-primary ordering or non-unique order keys, it should
use per-branch anchors or an order-key tuple cursor. That is outside the first
0.183 slice.

## Generated Endpoint Work

Generated list endpoints should not hide expensive work behind UI pagination.

Required generated/API behavior:

- The backend limit must be the UI limit plus one lookahead row when returning
  `has_more`.
- Page calls should default to `page_only`.
- Exact totals should be a separate method or an explicit option.
- Debug/perf builds should expose whether the page path, count path, or both
  ran.
- Generated filter lowering should canonicalize `OR` equality chains into
  membership predicates before shared planning.

This is the fastest mitigation path if the current live endpoint eagerly counts.

## Acceptance Gates

For the target page query:

- no eager count in the default page call;
- no full collection scan when `(collection_id, stage, id)` exists;
- no materialized sort of all matching rows;
- residual filter absent for `collection_id` and `stage`;
- first-page work stops after `limit + 1` emitted branch-merged rows;
- `data_store.get` is zero for key-only/covered projection and proportional to
  returned rows for non-covered projection;
- `index_store.entries` is proportional to scanned branch entries, not the
  whole collection;
- EXPLAIN identifies the branch-aware access route.

For the matching count query:

- exact result matches the row query filter;
- no row-store reads when the filter is fully index-covered;
- diagnostics distinguish branch index-entry count from cardinality metadata;
- count is not run unless the generated/API call requested it.

## Test And Audit Plan

0.182 closeout coverage already includes a production-shaped audit row using
the existing account fixture:

```sql
SELECT id, tier
FROM PerfAuditAccount
WHERE active = true AND tier IN ('gold', 'silver')
ORDER BY id ASC
LIMIT 3
```

and the matching count:

```sql
SELECT COUNT(*)
FROM PerfAuditAccount
WHERE active = true AND tier IN ('gold', 'silver')
```

0.183 implementation tests should add:

- planner unit coverage for `Eq + IN` on composite prefix slot 2;
- route/explain tests proving branch-aware order preservation;
- executor correctness tests comparing branch-merged pages to full filter/sort;
- continuation tests across branch boundaries;
- count tests proving zero row-store reads for covered branch counts;
- generated endpoint tests proving `page_only` does not call count.

## Phases

### Slice 0 - 0.182 Closeout

- Add this design.
- Confirm and use the production-shaped manual perf audit scenarios.
- Use 0.182.23 attribution to classify the current cost as row hydration,
  index traversal, residual filtering, materialized ordering, count, or endpoint
  double work.

### Slice A - Generated Endpoint Rescue

- Ensure generated list calls pass `limit + 1`.
- Make exact totals lazy/explicit.
- Add page/count attribution separation for generated debug builds.

This can land before the full planner work because it reduces immediate
instruction pressure even when the underlying plan is not ideal.

### Slice B - Composite Branch Planning

- Add the branch-aware access contract.
- Extend equality-prefix assembly to admit small `IN` sets after fixed equality
  slots.
- Add branch-aware route ranking and EXPLAIN output.

### Slice C - Lazy Ordered Branch Execution

- Lower branch specs to dynamic index streams.
- Merge by primary key.
- Apply page lookahead to emitted rows.
- Preserve projection coverage behavior.

### First Implementation Slice

The first production slice uses a dedicated semantic route:

```text
AccessPath::IndexBranchSet {
  index,
  fixed_values,
  branch_values,
}
```

It admits only `Eq(prefix_0...) AND In(next_prefix)` with a conservative
branch cap, requires `ORDER BY` to be the primary-key order in ascending mode,
and proves order by treating the fixed equality prefix plus branch slot as the
consumed index prefix. The executor lowers the route into one dynamic index
prefix stream per branch and merges those ordered streams with the existing
duplicate-suppressing ordered-key merge.

The initial implementation intentionally does not hard-cut branch-aware cursor
continuation, covered exact branch counts, prefix cardinality metadata, or
`OR(Eq...)` canonicalization beyond any existing SQL predicate normalization.
The core generated SQL surface was checked and does not appear to emit hidden
list endpoints that eagerly count. The IcyDB-side acceptance target is the
optimized SQL query shape itself, not an application-specific wrapper.

### Slice D - Branch Continuation

- Hard-cut cursor signature for branch-aware primary-key ordered pages.
- Resume all branches from the global primary-key anchor.
- Add continuation compatibility tests.

### Slice E - Covered Branch Count

- Sum branch index-entry counts without row-store reads.
- Surface count route attribution.
- Keep exact totals explicit at generated/API surfaces.

### Slice F - Prefix Cardinality Metadata

- Maintain exact prefix counts for accepted indexes.
- Use `(collection_id, stage)` counts for branch count totals.
- Audit storage overhead and update/delete maintenance cost before enabling by
  default.

## Risks

- Branch explosion: cap `IN` expansion and require exact small literal sets.
- Cursor semantics: keep the first branch cursor primary-key-order-only.
- Filtered indexes: require query predicate implication to be proven before
  using filtered index counts.
- Mixed generated/filter surfaces: normalize generated `Or(Eq...)` and SQL
  `IN` into the same planner-owned shape.
- Count expectations: document that exact count is a separate operation and can
  cost more than page retrieval until cardinality metadata lands.

## Open Questions

- Should generated list methods expose a typed `count_policy`, or should count
  always move to a separate generated method?
- Should the SQL covering branch path become fully pull-lazy instead of bounded
  per-branch collection for first pages?
