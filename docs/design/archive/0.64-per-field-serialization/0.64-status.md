# 0.64 Status Tracker

Current state: public mutation API landed; release closeout is complete and
the `0.64` line is ready to close. `0.64` is the structural field-level mutation and
serialization line defined in
`docs/design/0.64-per-field-serialization/0.64-design.md`.

The goal is not a new typed update surface. The goal is to extend the existing
persisted-row authority so field-level read, write, and patch application all
flow through one structural slot boundary.

## Scope

`0.64` is intended to land in ordered slices:

- field-level decode authority over persisted-row slot payloads
- field-level encode authority from `Value` into slot payloads
- ordered patch representation over stable slot indices
- phase-1 full-row rewrite patch application
- public/session update APIs only after the row boundary is stable

This line should preserve the current runtime-boundary direction:

- no per-entity mutation helpers
- no generated per-field APIs
- no duplicate validation pipeline

---

## Landed Slices

- [x] Persisted-row field decode now has one canonical owner in
  `db::data::persisted_row`: direct slot reads and cached `StructuralSlotReader`
  access both route through the same field-codec boundary instead of each
  rebuilding scalar-vs-CBOR dispatch.
- [x] Persisted-row field encode now has an initial canonical owner in
  `db::data::persisted_row` for scalar leaf slots and
  `FieldStorageDecode::Value` slots, with field-contract validation kept at the
  same boundary instead of starting from a typed update/session API.
- [x] Persisted-row field encode now also covers composite `ByKind` field
  contracts for lists, maps, relations, enums, and typed wrapper leaves, so
  field-level structural writes can reuse the same storage shapes the existing
  structural decoders already expect.
- [x] The ordered structural patch representation (`FieldUpdate` /
  `UpdatePatch`) now lives at the persisted-row seam together with a serialized
  patch artifact, so field patches can be lowered once into canonical slot
  payload bytes before later write-path stages consume them.
- [x] Phase-1 patch application now exists at the persisted-row boundary as
  read row -> replay ordered patch bytes over the slot layout -> re-encode row,
  instead of rebuilding field encode dispatch inside row replay.
- [x] `RawRow` now exposes the row-boundary façade for structural patch replay,
  while delegating all slot semantics back to persisted-row. Later mutation
  staging can therefore target the row boundary without bypassing the canonical
  slot codec owner.
- [x] Existing-row save/update staging now starts using that boundary: typed
  entity after-images are lowered once into a dense serialized slot patch and
  then replayed through `RawRow`, instead of always jumping straight from typed
  entity -> full row bytes.
- [x] Insert/missing-row row construction now also goes through the same
  serialized slot-patch replay path, so the old direct full-row encoder is no
  longer a competing row-construction seam.
- [x] Typed save lanes now lower into an explicit structural mutation input
  before commit-row staging. The executor no longer needs to carry the full
  typed entity all the way into marker-row preparation just to build an
  after-image row.
- [x] That structural mutation input now lives at the shared mutation boundary
  instead of only inside the save executor, so later direct patch/update entry
  points can target one staging shape instead of inventing another.
- [x] The executor now has an internal direct structural existing-row update
  seam: key + ordered field patch lowers into the shared structural mutation
  input, replays through the row boundary, decodes back through the typed
  entity, re-runs the existing save preflight, and only then stages the
  normalized after-image for commit. That keeps structural mutation from
  becoming a weaker bypass around typed invariants.
- [x] That structural update seam now has a caller-side owner in the core
  session layer instead of exposing the raw executor directly. `DbSession`
  accepts the structural patch, routes it through the same shared mutation
  input, and still reuses the typed save validation path before commit staging.
- [x] The internal structural session/executor seam now carries explicit write
  mode semantics instead of smuggling them through helper choice. Structural
  mutation mode is spelled out alongside the patch, even though only the update
  mode is exercised publicly today.
- [x] Internal structural insert/replace semantics now exist too. Insert and
  replace both rebuild from an empty row image, so required fields must be
  present in the patch itself; replace no longer risks inheriting omitted
  fields from the previous row implicitly.
- [x] Typed-vs-structural parity is now pinned for insert, update, and replace
  on equivalent rows, so the internal structural mutation algebra is validated
  against the existing typed write semantics instead of only passing its own
  standalone success cases.
- [x] The public `icydb` façade now exposes one mode-driven structural mutation
  surface instead of freezing separate partial helpers first: callers build an
  `UpdatePatch`, choose `MutationMode`, and route through
  `DbSession::mutate_structural(...)` while the existing session/executor
  validation pipeline remains unchanged underneath.

---

## Current Read

The persisted-row seam is the correct `0.64` starting point because it already
owns:

- stable slot-addressed field access
- structural row byte decoding
- typed materialization/writer bridging through `PersistedRow`

The write-side gap is now narrow instead of foundational. Field codecs,
serialized patch lowering, structural patch replay, explicit structural
mutation modes, typed-vs-structural parity, and the public `icydb`
mode-driven façade are all in place. The remaining work was release-shaping
only, and that closeout is now complete: public wording has been cleaned up,
the public example surface is in place, and the `0.64` line is ready to close.

## Likely Next Slices

- [x] Introduce a real structural mutation input that can reuse serialized slot
  patches without requiring a full typed entity after-image.
- [x] Decide how far the current shared mutation input should be generalized
  before any public/session structural update API is exposed.
- [x] Move the caller-side structural update surface through one
  validate-and-normalize boundary above the executor, so field existence, type
  compatibility, required fields, key consistency, and index invariants are
  still enforced before commit staging.
- [x] Decide whether insert/replace structural modes should become real caller
  surfaces now, or remain internal until the external error/ergonomics
  contract is stable enough to freeze.
- [x] Add one short public usage example for `DbSession::mutate_structural(...)`
  so the mode-driven structural API is understandable without reading the
  internal design notes first.
- [x] Audit the external error wording on the structural path to ensure the new
  public API does not leak internal slot or staging terminology.
- [x] Decide whether `0.64` should ship with only the mode-driven core API or
  also with thin convenience wrappers on top of it.
  Decision: ship the single mode-driven API (`UpdatePatch` +
  `DbSession::mutate_structural(...)`) and do not add thin convenience wrapper
  layers in the `0.64` line.
