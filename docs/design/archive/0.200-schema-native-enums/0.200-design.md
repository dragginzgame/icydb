0.200 makes enum definitions catalog-native and makes accepted schema the only
runtime authority for enum identity, values, defaults, query literals, storage,
indexes, grouping, and generated Rust contract checks.

This design ships one 0.200-native persisted representation. It does not ship
an embedded-registry format, compatibility reader, migration runner, or
generated-metadata fallback.

## Hard-Cut Baseline

0.200 is an internal pre-1.0 persisted-format hard cut under
[PERSISTED_FORMAT_POLICY.md](../../contracts/PERSISTED_FORMAT_POLICY.md).

- Pre-0.200 durable stores are rejected before schema, row, journal payload, or
  index-key decoding.
- No pre-0.200 enum value, field-kind, schema snapshot, index key, default, or
  generated representation is read or converted.
- No legacy aliases, old variant spellings, fallback decoders, dual-read paths,
  dual-write paths, read repair, or generated-metadata reconstruction exist.
- Development and test stores created by earlier formats must be deleted and
  recreated.
- Old fixtures are deleted or regenerated. They are not migration inputs.
- Rollback from a 0.200-native store to an older binary is unsupported.
- Runtime code reads and writes exactly one canonical 0.200 enum
  representation.
- A format mismatch is a typed fatal incompatibility at the recovery boundary.
  It is not an invitation to migrate.

The active format identifier is an internal store-format version, not the
crate's semantic version:

```rust
struct DatabaseFormatVersion(u16);

enum DatabaseFormatAdmissionError {
    UnsupportedFormatVersion {
        found: Option<DatabaseFormatVersion>,
        required: DatabaseFormatVersion,
    },
    MalformedFormatMarker {
        reason: FormatMarkerError,
    },
}
```

The error maps onto the existing incompatible-persisted-format classification
with recovery origin. The exact public/internal error type may differ, but
unsupported format must remain distinguishable from malformed current-format
bytes. A parseable non-current version is unsupported. Invalid boot-record
magic, state, length, or checksum is current storage corruption.

## Store Format Gate

IcyDB currently has separate versioned row, schema, commit, journal, and cursor
surfaces rather than one database-wide header. 0.200 turns the existing
canister-wide commit-control allocation into one `DatabaseControlStore` for the
whole `Db<C>` recovery domain. It does not add a memory allocation or a
`format_memory_id` to each store.

Allocation cardinality remains:

```text
4 * journaled stores + 1 database control allocation
```

Each journaled store retains its complete data, index, schema, and journal
allocations. The existing canister `commit_memory_id` identifies the database
control allocation.

`DatabaseControlStore` is one bounded, fallibly decoded control structure over
raw stable memory:

```rust
struct DatabaseControlStore<M: Memory> {
    memory: M,
}

struct DatabaseBootRecord {
    magic: [u8; 8],
    format_version: DatabaseFormatVersion,
    state: DatabaseBootState,
    checksum: u32,
}

enum DatabaseBootState {
    Initialized,
}

struct CommitControlSlot {
    marker: Option<RawCommitMarker>,
}
```

The control store contains no tree, map, catalog, or generic `Value`. Its fixed
boot prefix is followed by a bounded database-control slot frame and the
existing bounded commit-control slot. These layers have separate codecs and
authority:

- `DatabaseBootRecord` is permanent for the lifetime of the database;
- `CommitControlSlot` is transient and contains at most one in-flight marker;
- one marker may include row operations and marker-bound journal batches for
  multiple registered stores, preserving database-wide atomic recovery;
- clearing a completed commit empties only the marker slot and never removes
  or rewrites the database's format authority.

The 15-byte boot encoding is `ICYDBBOT`, followed by a big-endian `u16` format
version, state byte `0x01` for `Initialized`, and a big-endian CRC-32C
(Castagnoli) over the preceding 11 bytes.

The following 13-byte database-control slot frame is `IDCS`, frame version
`0x01`, a big-endian `u32` logical commit-slot length, and a big-endian CRC-32C
of the exact commit-slot bytes. The existing `CMCS` commit-control slot follows
that frame. The outer frame preserves the exact logical length over page-sized
raw memory, detects truncated or stale bytes, and is written after the slot
payload. Clearing a marker writes a checksummed frame containing the canonical
empty `CMCS` slot; it does not erase the boot prefix.

`DatabaseControlStore` owns a direct fallible codec over `Memory`. It does not
call an infallible generic `Storable::from_bytes`, does not use
`StableCell::init` semantics that may reset unknown bytes, and does not decode
Candid, schema values, rows, journals, or commit-marker payloads while admitting
the boot prefix.

### Control Lifetime And Authority

The boot record is written during virgin-database initialization, survives
canister upgrades and schema publications, and is not deleted or regenerated
during ordinary operation. `Initialized` means the database has been claimed
by the named durable format; it does not assert that every registered store
structure has already allocated pages. Those structures remain idempotently
initializable after a crash.

The permanent boot prefix stores only its magic, database format version,
initialized state, and checksum. It does not store enum definitions, accepted
schema revisions, semantic fingerprints, journal positions, schema-mutation
candidates, migration progress, runtime caches, or plans. The adjacent commit
slot stores only current bounded commit-recovery state.

Future boot metadata may be added only through an explicit versioned control
codec. The control store must not become an unstructured metadata bag.

`ensure_recovered(...)` must validate the database boot record before:

- schema reconciliation;
- commit-marker payload decoding;
- journal record decoding or replay;
- persisted schema snapshot decoding;
- row decoding;
- index-key decoding or rebuild.

Boot admission and initialization are normative:

```text
missing boot record + physically virgin control and store allocations
    -> write and durably flush the current Initialized boot record
    -> initialize the empty commit slot
    -> initialize current-format data/index/schema/journal structures

current valid boot record
    -> idempotently initialize any absent current-format structures
    -> proceed with current-format recovery

missing boot record + any non-virgin control or registered store allocation
    -> UnsupportedFormatVersion { found: None, required: current }

valid boot record + non-current version
    -> UnsupportedFormatVersion { found: Some(found), required: current }

malformed boot record
    -> MalformedFormatMarker
```

Physical virginity requires an absent or zero-initialized database control
allocation and empty data, index, schema, and journal allocations for every
registered journaled store. It is proved with bounded raw allocation facts
before initializing any BTree, journal, schema, index, or commit metadata. It
does not scan rows or decode legacy values. The boot record is the first
database-owned durable write. A crash after that write but before other
structure initialization therefore reopens as a valid current-format empty
database, and initialization resumes idempotently. A crash before that write
leaves the whole recovery domain physically virgin.

The scope is one `Db<C>`/canister recovery domain, not one marker per store or
entity. A newly declared journaled store joins the current database format only
when all of its allocations are physically empty before first use. A newly
declared store pointing at non-empty allocations is rejected before those
allocations are decoded.

The hard cut also requires:

- bumping the row-envelope format version for the new canonical enum value
  representation;
- bumping the strict raw schema-snapshot envelope and schema codec version for
  the catalog-backed field-kind representation;
- assigning one current index-key format tag for enum ID encoding;
- deleting the raw schema snapshot's headerless compatibility path;
- rejecting unsupported envelope versions before decoding their payloads;
- bumping every outer container whose bytes embed the changed enum `Value` or
  field-kind representation, unless a nested independently versioned payload
  rejects old bytes before decoding.

The closeout persisted-format inventory is transitive and includes:

- row values and structured field values;
- accepted defaults and schema snapshots;
- journal row/schema records and commit-marker payloads;
- secondary, unique, and system index keys;
- cursor and continuation-token payloads;
- persisted prepared-query or command artifacts, if any exist;
- backup, export, import, and restore representations;
- replication payloads, if any exist.

Each inventory entry records its current outer or nested version gate. An
absent surface is recorded as absent rather than silently omitted. Cursor
formats that can contain enum values or enum-backed index components receive a
current format version; old path/label enum cursor bytes are not decoded.

Scalar and grouped continuation tokens use token wire version `2` in 0.200.
Version `1` rejects at the token envelope before any boundary or recursive
`Value` decode. The frozen version-2 vectors include the mandatory type ID,
variant ID, and explicit unit body used by a grouped enum key.

The token envelope version and continuation-signature profile are distinct.
The current signature profile hashes the logical execution-shape signature,
entity snapshot fingerprint, accepted root revision, and accepted root
fingerprint. A catalog publication therefore invalidates scalar and grouped
tokens even when the entity snapshot itself is unchanged. The process-local
store-scope token is deliberately excluded because continuation tokens may be
used after a clean reopen of the same durable store. No older signature profile
is accepted as a fallback.

The boot record is the first gate. Per-surface versions remain defense in
depth for malformed or mixed current-format state.

### 0.200 Persisted-Format Inventory

| Surface | Current gate | 0.200 disposition |
| --- | --- | --- |
| Database recovery domain | `ICYDBBOT`, database format version 1 | Admitted before any store recovery decoder. Missing or non-current format rejects unless the complete domain is physically virgin. |
| Row values | `IR`, row envelope version 2 | Version is checked before row payload decode. Canonical enum slots use the `0x84` mandatory-ID envelope inside the row. |
| Structural field/value bytes | Enclosing row version 2, plus fallible structural tags and lengths | No standalone durable root exists. Unknown tags, malformed framing, and non-zero trailing bytes reject. |
| Entity schema snapshots and accepted defaults | `ICYDBSCH`, raw snapshot version 1, then schema codec version 6 | The raw gate precedes schema decode. Accepted enum fields contain only type IDs; defaults carry current canonical value bytes. |
| Accepted enum catalog | `ICYDBENC`, catalog codec version 1 | Bounded decode validates IDs, maps, definitions, references, cycles, and canonical order. |
| Accepted revision bundle and root | `ICYDBASB` version 1 and `ICYDBASR` version 1 | Bundle decode validates catalog, snapshots, defaults, and indexes before the checksummed root can select it. |
| Journal batches | `IJBT`, journal batch version 2 | Version is checked before row or accepted-schema publication record decode. Embedded rows retain row envelope version 2. |
| Commit marker | Commit marker envelope version 2 inside the checksummed database-control slot | Version is checked before marker payload or marker-bound journal batch decode. One marker may cover multiple stores. |
| Secondary and unique index keys | Database format version 1; enum component version 1 (`ValueTag::Enum`, version, type ID, variant ID, unit body) | The generic ordered codec rejects enums. Only catalog-proved unit enums emit the versioned 11-byte component; payload enums reject. |
| Reverse/system indexes | Database format version 1 plus their existing bounded structural key validation | No independent enum representation is introduced. Any enum-bearing accepted component must use the same catalog-proved component encoder. |
| Scalar and grouped continuation tokens | Token wire version 2 | Version 1 rejects before boundary or recursive `Value` decode. Version-2 enum vectors carry mandatory IDs and explicit body state. |
| Prepared query/command artifacts | Absent | Plans and compiled SQL commands are heap-only caches scoped to guarded accepted revision/fingerprint authority. No persisted decoder exists. |
| Backup/export/import/restore format | Absent | Output DTOs are not a durable restore format. Cross-store enum import remains unsupported. |
| Replication payload format | Absent | No replication decoder or compatibility surface exists. |

The schema allocation remains one stable BTree rather than adding another
memory. Its key namespace distinguishes entity snapshots, immutable accepted
revision bundles, and accepted-root slots. The shared stable value wrapper is
opaque for the two control-record namespaces because `ICYDBASB` and `ICYDBASR`
provide their own bounded versioned envelopes. An entity-snapshot key is
different: its reader requires a valid `ICYDBSCH` identity header before the
schema codec receives the payload. Missing, non-current, or malformed entity
snapshot framing cannot fall through as a headerless schema payload.

This inventory is exhaustive for 0.200. Adding a new durable or externally
resumable container requires adding its rejecting version boundary here and in
the persisted-format corpus before it may carry accepted enum bytes.

## Objectives

0.200 must establish:

1. One accepted enum catalog per accepted store catalog revision.
2. Mandatory stable enum type and variant identity.
3. Fields that reference catalog definitions instead of embedding copies.
4. Separate unresolved input literals and canonical runtime enum values.
5. Exact generated proposal equality with accepted canonical definitions.
6. One bundled accepted value contract and two explicit admission operations.
7. Equality-only enum semantics with declaration order erased.
8. A mechanically defined unit-enum equality-key format.
9. Explicit prohibition of payload-enum indexing/grouping until canonical
   payload equality encoding exists.
10. A crash-safe database-wide boot gate before recovery decoding.
11. One current-schema root compare-and-swap as the publication linearization
    point.
12. Live execution guarded by the current accepted store revision.
13. Store-local canonical ID provenance and non-authorable runtime IDs.
14. An acyclic accepted enum type graph and bounded recursive values.
15. Categorical rejection of existing enum field type rebinding.

## Non-Goals

- Pre-0.200 migration, conversion, activation scans, repair, or rewrite runners.
- Compatibility with old snapshots, rows, indexes, defaults, fixtures, or
  binaries.
- Alias or legacy spelling support.
- Embedded enum definitions as a released persisted mode.
- Generated model fallback after accepted catalog publication.
- SQL enum DDL in the initial 0.200 implementation.
- Enum rename, drop, payload-contract mutation, or ordering-policy mutation.
- Dynamic accepted variants absent from current generated Rust contracts.
- A universal recursive equality-key framework as a prerequisite for storing
  payload enum values.
- Recursive accepted enum type graphs in the initial 0.200 contract.
- General query, planner, or type-system redesign.

## Catalog And Identity Model

Accepted enum definitions live once in the store's accepted schema catalog.
The catalog is one canonical schema-store value, not a copy embedded in every
entity snapshot. Entity snapshots record the accepted catalog
revision/fingerprint and reference enum types by ID.

```rust
struct AcceptedSchemaContracts {
    format_version: DatabaseFormatVersion,
    revision: AcceptedSchemaRevision,
    fingerprint: AcceptedSchemaFingerprint,
    enums: AcceptedEnumCatalog,
}

struct AcceptedEnumCatalog {
    by_id: BTreeMap<EnumTypeId, AcceptedEnumType>,
    // Derived after decode; not a second persisted definition source.
    id_by_path: BTreeMap<EnumPath, EnumTypeId>,
}

struct AcceptedEnumType {
    path: EnumPath,
    variants_by_id: BTreeMap<EnumVariantId, AcceptedEnumVariant>,
    // Derived after decode; not a second persisted definition source.
    variant_id_by_name: BTreeMap<EnumVariantName, EnumVariantId>,
    ordering: EnumOrderingPolicy,
}

struct AcceptedEnumVariant {
    name: EnumVariantName,
    body: AcceptedEnumVariantBody,
}

enum AcceptedEnumVariantBody {
    Unit,
    Payload {
        contract: AcceptedValueContract,
    },
}

struct AcceptedValueContract {
    kind: AcceptedFieldKind,
    storage_decode: FieldStorageDecode,
}

enum AcceptedFieldKind {
    // ...
    Enum {
        type_id: EnumTypeId,
    },
}

enum EnumOrderingPolicy {
    EqualityOnly,
}
```

This is a semantic shape, not a required exact in-memory layout. The required
properties are:

- `EnumTypeId` and `EnumVariantId` are mandatory.
- IDs are non-zero fixed-width `u32` values scoped to one accepted store
  catalog. Numeric equality has no meaning across independent stores.
- No accepted type is path-only, and no accepted variant is label-only.
- IDs are persisted, never inferred from optional names, and never reused.
- Paths and labels are schema-visible names, not physical equality identity.
- Type and variant IDs live once as canonical catalog map keys rather than
  being duplicated inside their map values.
- Field kinds store only `EnumTypeId`, not repeated variant definitions.
- Unit variants cannot carry meaningless payload decode metadata.
- Payload kind and storage decode are one inseparable accepted contract.
- Name/ID lookup maps are derived from canonical definitions after decode and
  validated as bijections before use.
- Diagnostic proposal origins are retained separately from semantic
  definitions and do not affect canonical equality.

Current implementation sequencing constructs and validates one complete
store-local catalog candidate from every registered generated entity before
reconciling any entity snapshot. Candidate type IDs follow canonical path byte
order, candidate variant IDs follow canonical name byte order, and complete
same-path definitions must match after ID resolution. Duplicate names,
conflicting definitions, recursive contracts, zero IDs, and checked ID
overflow fail candidate construction.

After every entity snapshot in the store has been accepted, reconciliation
builds one immutable store-local revision bundle containing the catalog and
the accepted snapshots in entity-tag order. Catalog codec version 1 remains
bounded to 512 KiB, persists definitions in ascending type-ID and variant-ID
order, omits derived name lookup maps, and reconstructs those maps only after
validating non-zero IDs, strict ordering, unique names, complete references,
acyclic type graphs, and the depth-64 contract bound. A non-current codec
version is an incompatible persisted format; malformed current bytes are
corruption.

Catalog codec version 1 is an explicit binary format, not Candid or a generic
`Storable` representation. It starts with `ICYDBENC`, a big-endian `u16`
codec version, and a big-endian `u32` type count. Each type stores a big-endian
`u32` type ID, a `u32`-length-prefixed UTF-8 path, an ordering-policy byte, and
a `u32` variant count. Each variant stores a big-endian `u32` variant ID, a
length-prefixed UTF-8 name, and a unit/payload body tag. Payload bodies store
the decode-policy byte followed by the recursively tagged accepted value kind.
All integer payloads and lengths are big-endian; option and boolean values use
only canonical `0`/`1` tags. Trailing bytes, unknown tags, overlong input, and
noncanonical ID order reject. The complete value-kind tag table is frozen by
the codec implementation and its wire-vector tests.

Accepted schema bundle codec version 1 is bounded to 16 MiB and starts with
`ICYDBASB`, a big-endian `u16` codec version, a big-endian `u64` revision, a
length-prefixed store path, length-prefixed canonical catalog bytes, and
entity-tag/snapshot pairs in strictly ascending tag order. The semantic
fingerprint hashes the store path, catalog bytes, entity tags, and accepted
snapshot bytes, but excludes the monotonic revision. The immutable bundle hash
includes the revision and complete bundle envelope.

The schema store now uses one frozen 16-byte namespaced key format in its
existing schema BTree. Namespace `0` stores entity-version snapshots,
namespace `1` stores immutable revision bundles by revision, and namespace `2`
stores the two current-root slots. No reserved entity tag, secondary stable
structure, page offset, or additional memory allocation is used.

Entity snapshots persist `AcceptedFieldKind` recursively. An enum node contains
only its non-zero store-local `EnumTypeId`; enum paths, variant names, bodies,
and payload contracts exist once in the accepted catalog. Generated proposal
objects retain `FieldKind` only until the complete candidate catalog has been
constructed, then resolve every top-level, nested-leaf, relation-key, and index
kind to `AcceptedFieldKind` before snapshot construction. No owned embedded
definition intermediate or embedded-definition snapshot codec exists.

Runtime entity-snapshot reads resolve only through the immutable bundle
selected by the current root. The highest independently stored entity snapshot
is explicitly a staged candidate view and is unavailable to live execution.
The schema store may retain one volatile decoded bundle keyed by the complete
selected checksummed root. Every lookup still selects and validates the root
slots first; a different root, a cleared root, or recovery projection reset
drops or bypasses the cache and requires full bundle verification. The cache is
never persisted and is not format authority.

Catalog construction merges generated definitions and requires exact path,
variant membership, unit/payload shape, payload kind, and payload
storage-decode equality before allocating or retaining IDs. Bundle construction
then recursively validates that every accepted enum field reference resolves
to an existing catalog type. Unknown type IDs or malformed current-format
snapshot bytes reject as corruption before the immutable bundle or root can be
published.

A verified entity selection retains one shared immutable handle to the catalog
from the selected bundle revision. Accepted session-cache entries, query/SQL
catalog contexts, and live `SchemaInfo` projections clone only that handle;
they do not clone definitions or reconstruct a catalog from generated models.
The handle never crosses a revision check and is not serialized separately.
This is the authority source that type-ID field interpretation, canonical row
decode, and catalog-aware planner capability checks must consume.

Catalog-backed row-decode contracts retain the accepted revision and shared
catalog handle as one bundled authority, so a decoder cannot independently mix
those facts. Accepted query authority and the central typed/structural session
save paths construct that bundled authority from the selected catalog context.
SQL write descriptor validation, selector authority, mutation save contracts,
and returning projection now reuse the same context; compiled SQL writes retain
the already revision-checked context rather than loading a bare snapshot or
carrying a second optional `SchemaInfo`. Every live row-scanning reconciliation
and maintenance path now loads the current root selection before decode.
Startup field-path and expression-index rebuilds and SQL field-metadata
validation additionally require that the selected snapshot exactly matches the
pre-mutation snapshot. Bare snapshot-only row-contract construction is
test-only; production scanners may not rebuild catalog authority from
generated models.

Commit preflight and strong-relation source-row validation now use accepted
authority built from the current root. A commit batch resolves one immutable
`CommitPrepareContext`, decoding the entity snapshot and proving generated
compatibility once, then borrows its row contract, index schema, entity
identity, and schema fingerprint while validating every row operation. Runtime
hooks carry only structural entity metadata needed to construct that context;
they do not retain a typed per-row commit-preparation callback. Each row still
checks its entity path and schema fingerprint before committed bytes are
decoded, so batching removes repeated authority construction without weakening
the corruption boundary.

Initial startup bootstrap persists accepted entity snapshots, the immutable
bundle, and the first root directly in the schema allocation. Later entity
snapshot/bundle revisions, including SQL DDL publications, publish through one
marker-bound journal record; replay restores the live snapshots, bundle, and
root, while fold applies the same record to the canonical schema BTree. Exact
bundle matches remain no-ops. Startup catalog reconciliation preserves
accepted type and variant IDs, retains unused definitions, allocates new IDs
from the checked accepted maximum in canonical path/name order, and rejects
variant removal or contract drift before publication. Existing-field variant
additions update the catalog definition while the entity field retains its
existing type ID; they require no entity-snapshot rewrite or row work.
Reconciliation never republishes IDs rebuilt from initial path/name ordering.
Generated reconstruction is never runtime authority.

### ID Allocation

The catalog owns monotonic, append-only ID allocation.

- Existing 0.200-native types and variants retain their IDs across accepted
  publications.
- ID `0` is reserved and never admitted or encoded.
- The first type and variant ID is `1`.
- The allocation frontier is derived with checked `max(existing_id) + 1`; it is
  not a separately persisted counter and does not participate in the semantic
  fingerprint.
- New enum types in one candidate are allocated in canonical `EnumPath` raw
  UTF-8 byte order.
- New variants within each existing or new type are allocated in canonical
  `EnumVariantName` raw UTF-8 byte order.
- Entity registration order, source field order, macro token order, hash-map
  iteration, proposal collection order, thread scheduling, and source variant
  declaration order do not participate.
- IDs are allocated in the candidate catalog and become visible only through
  atomic publication.
- Rejected or stale candidates do not advance the accepted allocation
  frontier. A rebuilt candidate may reuse the same never-published numbers.
- An exact no-op allocates no IDs, writes no candidate artifacts, publishes no
  revision, and leaves the semantic fingerprint unchanged.
- Checked overflow rejects the whole candidate with typed
  `EnumTypeIdExhausted` or `EnumVariantIdExhausted`; allocation never wraps.

IDs are not path or label hashes. Hash-derived identity would make rename
physical again and would require collision policy in every consumer.

Type and variant rename remain rejected in initial 0.200 even though stable IDs
make a future catalog-only rename possible. That future operation needs its own
API, diagnostics, generated contract, and output-rendering design.

### Type Graph Policy

Initial 0.200 rejects direct and indirect cycles through enum payload
contracts. The cycle check follows enum references through options, lists,
sets, maps, records, and nested payload contracts and returns a typed
`RecursiveEnumContract` schema rejection with the canonical cycle path.

Catalog construction is still two-pass so acyclic forward references are
deterministic:

```text
collect and canonicalize every enum path
    -> bind existing fields and existing paths to accepted IDs
    -> allocate IDs for genuinely new paths in canonical byte order
    -> resolve every payload enum reference to an ID
    -> detect direct or indirect cycles
    -> allocate new variant IDs in canonical name byte order
    -> construct and validate complete definitions
```

Canonical definitions, generated-contract matching, and fingerprinting encode
enum references by ID; they never recursively expand a referenced definition.
Supporting recursive accepted type graphs later requires a separate graph and
bounded-decoding design.

## Proposal Canonicalization

Generated Rust metadata is proposal input only.

Generated proposals may repeat an enum definition across fields. Before
catalog construction, proposals are grouped by enum path and canonicalized.

```rust
struct CanonicalEnumProposal {
    path: EnumPath,
    variants: BTreeMap<EnumVariantName, CanonicalEnumVariantBody>,
    ordering: EnumOrderingPolicy,
    origins: Vec<EnumProposalOrigin>,
}

enum CanonicalEnumVariantBody {
    Unit,
    Payload {
        contract: CanonicalValueContract,
    },
}
```

Canonical proposal equality includes:

- enum path binding;
- variant membership;
- unit versus payload status;
- recursively canonical payload kind;
- payload storage decode;
- ordering policy.

It excludes:

- source declaration order;
- field/source origin ordering;
- generated token order;
- diagnostics-only source locations.

Duplicate names inside one proposal reject before map construction.
Definitions with the same path merge only when their complete canonical forms
are equal. Conflict diagnostics retain every origin and report the exact
contract difference.

Under `EqualityOnly`, source declaration reorder is a semantic no-op. Two
proposals with the same names and contracts in different declaration order
produce the same canonical definition and accepted schema fingerprint.

## Exact Generated Contract

Every generated enum field must exactly match the accepted canonical enum
definition bound to its `EnumTypeId`.

Exact matching includes:

- path-to-ID binding;
- complete variant membership;
- name-to-variant-ID binding after catalog resolution;
- unit versus payload status;
- recursively canonical payload kind;
- payload storage decode;
- ordering policy.

These mismatches all reject candidate publication or generated model
registration:

```text
Generated: Result::Ok(Text)
Accepted:  Result::Ok(Blob)

Generated: State::Active
Accepted:  State::Active(Nat64)
```

There is no accepted-but-unknown generated variant path, dynamic fallback,
unknown-variant arm, read-time coercion, or generated metadata reconstruction.
A current generated model paired with the wrong 0.200-native accepted catalog
fails before serving reads or writes.

Exact matching is evaluated against the fully constructed candidate catalog.
It does not block an allowed native variant addition when every current
generated proposal already describes that same candidate definition.

## Input And Runtime Value Domains

Unresolved enum literals are construction values. Runtime enum values are
canonical.

The `InputValue` / `Value` split is authoritative:

```rust
enum InputValueEnum {
    Unqualified {
        variant: EnumVariantName,
        payload: Option<Box<InputValue>>,
    },
    Qualified {
        enum_path: EnumPath,
        variant: EnumVariantName,
        payload: Option<Box<InputValue>>,
    },
}

struct ValueEnum {
    type_id: EnumTypeId,
    variant_id: EnumVariantId,
    body: CanonicalEnumValueBody,
}

enum CanonicalEnumValueBody {
    Unit,
    Payload(Box<Value>),
}
```

The implementation keeps the identity/body shape DRY through one private
value-layer owner, parameterized only by its recursive value domain:

```rust
struct CanonicalEnumValue<V> {
    type_id: EnumTypeId,
    variant_id: EnumVariantId,
    body: CanonicalEnumBody<V>,
}

enum CanonicalEnumBody<V> {
    Unit,
    Payload(Box<V>),
}
```

Accepted admission and runtime `Value::Enum` instantiate this same owner. The
legacy path/label representation and the parallel admission-only value mirror
are removed.

### Sequencing Type Disposal

The cutover removes transitional duplication rather than preserving it:

- the current name/path/payload `ValueEnum` is obsolete runtime state and is
  replaced by the mandatory-ID form; it is not wrapped or retained as a legacy
  variant;
- admission's temporary `CanonicalValue` mirror is folded into runtime
  `Value` once `Value::Enum` is canonical;
- `CanonicalEnumValue<V>` becomes the single private identity/body owner used
  by runtime `ValueEnum`; a second parallel canonical enum struct is prohibited;
- `InputValueEnum` remains the only unresolved authoring representation;
- `OutputValueEnum` remains a name-oriented boundary representation, populated
  by resolving canonical IDs through the execution revision's catalog;
- `AcceptedCatalogSnapshotSelection`, `AcceptedStructuralRowAuthority`,
  `StructuralRowContract`, and `AcceptedRowLayoutRuntimeContract` remain
  required authority/layout types and are not sequencing artifacts;
- bare snapshot-only row-contract constructors remain test-only and must be
  deleted when their low-level fixtures move to published catalog selections.

After 0.200:

- `InputValueEnum` may be unresolved.
- SQL bare strings remain SQL expressions until target typing.
- Parser literals remain parser values until accepted-contract admission.
- `Value::Enum` always contains type ID, variant ID, and explicit unit/payload
  shape.
- Runtime and persisted enum values cannot represent a missing path or missing
  identity.
- Infallible `InputValue -> Value` conversion cannot bypass enum admission.
- Cache identity, predicates after normalization, storage, indexes, grouping,
  cursors, and output materialization consume canonical values only.

Output rendering resolves IDs through the immutable accepted catalog handle
retained by the guarded plan used for execution. Value-producing fluent
terminals return an internal result carrier containing both the runtime value
and that exact handle; the facade consumes the carrier during rendering rather
than reloading the session's current catalog after row access. Grouped and SQL
projection result carriers follow the same ownership rule. Output does not
store labels as runtime identity.

### Catalog Scope And Value Provenance

`EnumTypeId` and `EnumVariantId` are local to one accepted store catalog.
Canonical `Value::Enum` values are not portable between independent catalogs,
even when the stores happen to allocate the same numeric IDs.

The authority boundary is structural:

- public structural, SQL, RPC, and import authoring APIs accept `InputValue`,
  not trusted canonical `Value`;
- canonical `ValueEnum` constructors are crate-private and available only to
  accepted normalization and current-format decode paths;
- persistence, index, hash-key, and cursor construction accept
  `AcceptedValueRef`, not arbitrary `Value`;
- journal replay consumes IDs only because the journal and catalog share one
  admitted store recovery domain;
- cross-store bulk import uses names and accepted `InputValue` contracts, or a
  separate explicit source-catalog-to-target-catalog remapping operation;
- output values expose names and payloads, not authorable trusted IDs.

An admitted value that outlives its borrowed accepted contract carries opaque
store authority as well as revision identity:

```rust
struct AcceptedSchemaAuthority {
    store_scope: AcceptedStoreCatalogScope, // opaque and non-serializable
    revision: AcceptedSchemaRevision,
    fingerprint: AcceptedSchemaFingerprint,
}

struct AdmittedOwnedValue {
    authority: AcceptedSchemaAuthority,
    value: Value,
}
```

`AcceptedStoreCatalogScope` is issued by the registered `Store` handle and
cannot be constructed from a path or integer. Reuse of an
`AdmittedOwnedValue` requires an exact authority match. Persisted rows do not
serialize this wrapper per field; their store allocation and accepted snapshot
provide the provenance binding during strict decode.

### Canonical Enum Wire Encoding

The current structural/generic value codec assigns enum tag `0x84`. The 0.200
enum payload after that tag is exactly:

```text
0x84
TypeId:       u32 big-endian, non-zero
VariantId:    u32 big-endian, non-zero
BodyTag:      u8; 0x00 = Unit, 0x01 = Payload
PayloadLen:   u32 big-endian
PayloadBytes: exactly PayloadLen bytes
```

For `Unit`, `PayloadLen` must be zero. For `Payload`, the bytes contain exactly
one current-format recursively encoded `Value` and must be non-empty. Unknown
body tags, zero IDs, length mismatch, trailing bytes, over-limit payloads, and
non-current nested value encodings reject. The enum envelope, bounded payload
callback, unit/payload vectors, and recursive canonical value-storage codec are
implemented and live for accepted generated typed-save `Value` fields. Unknown
non-zero IDs decode only to the strict-validation boundary, where they classify
as current-format corruption.

The enclosing 0.200 row/value envelope version introduces this representation.
No persisted enum codec writes or accepts paths, labels, optional IDs, or an
alternate tuple shape.

## Accepted Value Admission

Kind and storage decode must travel together:

```rust
struct AcceptedValueContract {
    kind: AcceptedFieldKind,
    storage_decode: FieldStorageDecode,
}

const MAX_ACCEPTED_VALUE_DEPTH: usize = 64;
const MAX_ACCEPTED_VALUE_BYTES: u32 = 4 * 1024 * 1024;
```

0.200 defines separate construction and strict-validation operations:

```rust
fn normalize_and_admit_value(
    contracts: &AcceptedSchemaContracts,
    contract: &AcceptedValueContract,
    input: InputValue,
    budget: &mut ValueAdmissionBudget,
) -> Result<AdmittedOwnedValue, ValueAdmissionError>;

fn validate_canonical_value<'a>(
    contracts: &'a AcceptedSchemaContracts,
    contract: &'a AcceptedValueContract,
    value: &'a Value,
    budget: &mut ValueAdmissionBudget,
) -> Result<AcceptedValueRef<'a>, ValueAdmissionError>;
```

The operation split and input types define whether a caller is constructing or
strictly validating a value. The admission kernel does not accept an inert
boundary-classification argument. Each owning write, query, schema, decode, or
key boundary maps `ValueAdmissionError` into its own public/internal error
classification.

The implemented admission kernel uses runtime `Value` with mandatory enum
type/variant IDs and explicit unit/payload bodies. It resolves `InputValueEnum`
paths and labels, recursively admits
payload enums, normalizes decimal scales and deterministic set/map order,
rejects duplicate set items and map keys, pins admitted values to the exact
opaque store scope, accepted revision, and bundle semantic fingerprint issued
with the immutable catalog handle, and shares the depth-64/4 MiB budget with
strict validation. Admission no longer accepts a free revision argument that
could be paired with the wrong catalog.
Candidate construction validates the catalog's path/ID bijections, variant
contracts, recursive graph, canonical codec round-trip, and exact persisted
field references. It does not synthesize enum values during publication;
normalization and strict value validation run at their actual construction and
decode boundaries.

`InputValue` no longer implements an infallible conversion into runtime
`Value`, and `InputValueEnum` no longer converts directly into `ValueEnum`.
Generated enum, list, map, set, newtype, record, and tuple authoring conversions
now construct recursive `InputValue` directly; enum names and paths remain
unresolved until accepted admission, including nested and indirect payloads.
Internal call sites that provably accept only non-enum literals use an explicit
fallible lowering that rejects direct or nested enum input. Public filter DTO
construction lowers its deliberately string-backed surface directly. Grouped
and structural input reject unresolved enums until their accepted-catalog
admission routes land; neither reconstructs a path/label runtime enum as a
fallback.

The existing `FieldProjection` trait remains a context-free runtime-value
projection for fields that cannot contain catalog enum identity; generated
custom fields return no context-free projection. Generated entities derive a
separate `AuthoredFieldProjection` that preserves the same stable field-slot
and one/optional/many cardinality mapping while returning `InputValue`.
Hand-authored runtime projection implementations therefore do not acquire an
unrelated authoring requirement. An accepted-schema adapter now binds authored
slots to the accepted row contract and its exact catalog handle, rejects
missing or non-generated field slots, and normalizes the authored `InputValue`
into an `AdmittedOwnedValue` under the accepted field's nullable,
storage-decode, and value constraints. Generated typed-save row emission now
uses that adapter for every accepted `Value`-storage and scalar field. DDL-owned
fields continue to materialize through accepted absence/default policy because
they have no generated authored slot. The adapter does not normalize and then
discard a second representation through the legacy runtime `Value` path.

Every `AcceptedRowDecodeContract` owns the exact immutable
`AcceptedValueCatalogHandle` that admitted its field facts. Its constructor
requires that handle, and its catalog/revision accessors are total. A bare
accepted snapshot may describe a row layout, but it cannot become executable
row-decode authority without the matching catalog handle. Low-level
model-proposal tests may build a complete accepted catalog and row contract as
one fixture; they do not create a catalogless runtime contract or reconstruct
metadata after publication.

`CanonicalValue` is an internal alias for runtime `Value`, not a second value
model. The accepted value-storage codec writes it directly and has no
path/label enum tuple codec. Accepted row decode strictly validates IDs and
payload contracts; query execution retains canonical IDs, generated typed
materialization resolves them through `RuntimeEnumContext`, and names appear
only in `OutputValue`.

Generated schema enums never implement context-free `RuntimeValueEncode` or
`PersistedStructuralValueCodec`. Explicit non-database protocol enums may own a
separate structural codec only when their wire shape never emits
`Value::Enum`, carries no catalog identity, and cannot be selected as the
schema-native enum persistence path. ICRC-3 generic protocol values use this
explicit exception.

Every construction, default, decode, and validation path starts with at most
64 recursive value levels and a 4 MiB encoded-value budget. Existing field,
collection, row, schema-snapshot, and payload bounds may be tighter. Each
recursive descent and length-framed payload consumes the shared budget before
allocation or recursion. Exhaustion returns a typed depth/size error; malformed
input cannot overflow the call stack or allocate past its enclosing format
limit.

`normalize_and_admit_value` may allocate and resolve:

- unqualified enum label under exactly one expected enum contract;
- qualified enum path to mandatory type ID;
- variant name to mandatory variant ID;
- recursive payload input to canonical runtime values.

`validate_canonical_value` never resolves or repairs:

- decoded values must already carry current catalog IDs;
- the type ID must match the field contract;
- the variant ID must exist in that type;
- unit/payload shape must match the accepted variant body;
- payload values must recursively validate against the bundled contract.

`AcceptedValueRef` is the proof consumed by persistence and key-building
code. It borrows the accepted contract and authority that proved it and cannot
outlive either. Raw `InputValue` or arbitrary `Value` must not cross those
boundaries.

Runtime field admission and byte emission receive an
`AcceptedFieldPersistenceContract` constructed by schema authority. That
borrowed contract pairs one accepted field's kind, storage decode, leaf codec,
and nullability with the immutable catalog handle that admitted the row layout.
Construction also resolves and retains the recursive `AcceptedValueContract`;
an unknown enum ID or otherwise invalid field/catalog combination cannot
produce the bundle. Field normalization, strict validation, and byte emission
accept only this complete contract, so persistence cannot supply its parts
independently or combine facts from different store scopes or accepted
revisions. Candidate DDL constructs the same bundle inside the schema mutation
owner before default admission.

Runtime row decode also selects this persistence contract before decoding the
slot. The field codec produces an owned canonical value, and the embedded
admission contract strictly validates it before assigning accepted store scope
and revision provenance. No runtime decoded-value admission API accepts the
catalog, kind, storage decode, or nullability rule as separate arguments.
Candidate-schema default validation remains schema-private and catalog-based
because it runs before a published runtime revision exists.

The semantic core is `AcceptedValueAdmissionContract`: immutable catalog
authority, one recursive `AcceptedValueContract`, and the top-level nullability
rule. Schema-info and predicate paths borrow this core directly from their
accepted snapshot. Persistence wraps the same core with field name, storage
decode, and leaf-codec facts. Derived collection-element contracts own only
their derived recursive contract while retaining the same catalog authority.
No normalization or strict-validation API accepts catalog, value contract, and
nullability as independently selectable arguments outside the schema module.

Accepted index projections precompute the recursive value contract for every
field-path component. The owning index binds only its own component metadata to
its immutable catalog before equality-key validation. Stored-row keys and
admitted query literals therefore consume one `AcceptedValueAdmissionContract`;
index code does not reconstruct contracts from independently supplied kinds and
catalogs. Non-enum components keep their existing canonical ordered codec, unit
enums use the catalog-proved equality key, and payload enums remain rejected.

`AcceptedValueRef` does not repeat field nullability. Nullability is checked at
proof construction, after which the proof carries only the catalog, recursive
value contract, and canonical value needed by downstream persistence and key
boundaries.

## Runtime Boundary Routing

The accepted admission boundary must be the only enum gateway for:

- generated Rust writes;
- structural writes and patches;
- SQL writes and bound parameters;
- generated and database defaults;
- predicate and `IN (...)` literals;
- field-to-field comparison typing;
- generic `FieldStorageDecode::Value` row decode;
- accepted `ByKind` decode;
- bulk import and batch write paths;
- index insertion and rebuild;
- grouped and distinct key construction;
- hash join, aggregation, set, deduplication, and partition key selection;
- plan hashing and cache identity;
- output rendering through the plan's accepted revision.

No boundary may validate only that a value is syntactically
`Value::Enum(...)`.

## Defaults

Database defaults are accepted schema content. SQL DDL unit-enum labels remain
unresolved until they are normalized against the live catalog, then persist
only as canonical type/variant IDs. Bundle construction and persisted bundle
decode validate every default payload against the exact field and catalog
contract before publication or use. Missing-slot materialization decodes the
payload through the accepted row contract and never consults generated
metadata.

Generated unit-enum defaults use `default = "Type::Variant"`. The derive layer
stores the enum path and variant name as
`FieldDatabaseDefault::AuthoredEnumUnit`; it does not encode bytes or allocate
IDs. Startup builds or reconciles the full store-local catalog first, then
uses the catalog's unit-default admission path to resolve the type and variant
IDs and delegates framing to the canonical enum wire codec. An authored enum
default cannot construct a persisted snapshot without catalog authority.
Unknown variants and payload variants without an authored payload reject
startup rather than using a fallback codec. The narrow unit path avoids making
the generic recursive input normalizer reachable in canisters that otherwise
do not need it.

During candidate publication:

1. Resolve the field's `EnumTypeId`.
2. Normalize the generated/default input.
3. Resolve and validate the variant ID.
4. Recursively admit any payload.
5. Persist only the canonical accepted default.

Runtime default materialization consumes the canonical accepted default stored
in the accepted snapshot. It never consults generated metadata.

## Equality And Ordering

Initial enum query semantics are equality-only.

Allowed:

- equality and inequality between values of the same enum type;
- `IN (...)` after every input value is admitted;
- equality filtering through supported exact-key encodings;
- deterministic internal catalog iteration that is not exposed as query order.

Rejected:

- cross-type equality, even when labels match;
- `<`, `<=`, `>`, and `>=`;
- enum `ORDER BY`;
- `MIN(enum)` and `MAX(enum)`;
- range index routing;
- declaration-order or ordinal semantics;
- `BEFORE` / `AFTER` variant positioning.

Internal `BTreeMap` order, type IDs, and variant IDs do not define user-visible
ordering.

### Unit Enum Equality Keys

An enum type whose variants are all unit variants has canonical stable-key
capability. Its canonical index-component bytes are exactly 11 bytes:

```text
0x07                         ValueTag::Enum
0x01                         enum equality-key format version
TypeId:    u32 big-endian    non-zero
VariantId: u32 big-endian    non-zero
0x00                         Unit body tag
```

The existing index-key component envelope supplies the component length. No
path, label, ordinal, payload length, or terminator is present. Any byte layout
change requires a new enum equality-key format version and the applicable
outer index/cursor format gate.

Required properties:

- type-tagged and length/framing safe;
- deterministic;
- same enum value produces identical bytes;
- different type or variant IDs produce different bytes;
- independent of source declaration order;
- independent of unrelated variant additions;
- not derived from vector position, path, label, or map iteration order.

Numeric ID byte order is an internal BTree routing fact only. The planner may
use a unit enum component for exact lookup, but may not expose its byte order as
enum ordering.

### Equality Capability

Planning represents the distinction explicitly:

```rust
enum EqualityCapability {
    PairwiseOnly,
    CanonicalStableKey,
}
```

An all-unit enum type has `CanonicalStableKey`. A type containing any payload
variant has at most `PairwiseOnly`, and payload equality delegates exactly to
the recursively accepted payload value's existing semantic equality.

The catalog-local capability classifier and exact unit-key encoder are
implemented. Coarse generated and accepted field semantics classify enums as
equality-only and do not grant grouping from the field kind alone. Live
`SchemaInfo` resolves each enum field or nested leaf through its retained
accepted catalog and records `PairwiseOnly` or `CanonicalStableKey`. SQL
grouping and SQL DDL index creation continue to reject enums. Accepted schema
publication permits field-path index components only for
`CanonicalStableKey`, pins the governing catalog revision into the accepted
index view, and routes runtime key construction through strict accepted-value
proof plus the frozen ID encoder. `PairwiseOnly` index proposals reject before
publication. No route falls back to generated definitions or the generic
ordered `Value` codec.

`PairwiseOnly` permits direct same-type equality, inequality, and admitted
`IN (...)` evaluation by pairwise comparison. It does not authorize any
operation that turns the value into an execution or persistence key, including:

- hash joins or hash partitioning;
- hash aggregation or hash-map-backed deduplication;
- `GROUP BY`, `DISTINCT`, or set keys;
- unique constraints or equality indexes;
- cursor serialization requiring a canonical component;
- plan evidence that embeds a stable encoded equality key.

### Payload Enum Restriction

0.200 may store and directly compare admitted payload enum values, but initial
0.200 does not permit payload-bearing enum fields or values in:

- secondary or unique index components;
- grouped keys;
- distinct keys;
- cursor ordering boundaries;
- any route that requires canonical persisted equality-key bytes.

Schema/index admission rejects those shapes before publication.

Composite indexes may contain an all-unit enum as an exact
equality-constrained component. For example, an index on
`(status, created_at)` may serve:

```sql
WHERE status = 'Active'
ORDER BY created_at
```

The enum prefix must be fixed to one admitted equality value before the
planner uses ordering or range semantics from later components. A continuation
cursor may retain that fixed encoded prefix while resuming within the
`created_at`/primary-key order. `ORDER BY status, created_at`, an enum range,
or an unconstrained scan that exposes enum ID order remains rejected.

Enabling them later requires one canonical recursive equality-key encoder:

```rust
fn encode_equality_key(
    contracts: &AcceptedSchemaContracts,
    contract: &AcceptedValueContract,
    value: &AcceptedValueRef<'_>,
    output: &mut EqualityKeyWriter,
) -> Result<(), EqualityKeyError>;
```

That encoder must define framing, integer widths, text bytes, float zero/NaN
policy, maps, lists, options, nested enums, recursion bounds, and format
versioning. It is a separate implementation slice and must reuse or replace
existing canonical key machinery explicitly. 0.200 does not hide this work
behind the phrase "canonical admitted payload representation."

## Atomic Publication

Each accepted store catalog has one immutable revision bundle and one durable
current-root control record:

```rust
struct AcceptedSchemaRoot {
    revision: AcceptedSchemaRevision,
    fingerprint: AcceptedSchemaFingerprint,
    bundle_key: AcceptedSchemaBundleKey,
    bundle_hash: [u8; 32],
}

struct CandidateSchemaRevision {
    expected_revision: AcceptedSchemaRevision,
    root: AcceptedSchemaRoot,
    bundle: AcceptedSchemaRevisionBundle,
}
```

The logical bundle contains the catalog plus every affected accepted entity
snapshot. It may be encoded as one bounded value or as an immutable manifest
of bounded chunks, but readers reach every part through one root and verify the
bundle hash before use.

Candidate construction is:

```text
generated proposals
    -> canonical proposal set
    -> current catalog ID binding/allocation
    -> exact generated contract checks
    -> canonical default admission
    -> field reference validation
    -> index/group capability validation
    -> entity snapshot construction
    -> semantic fingerprint and immutable revision bundle
    -> persist and verify unreferenced candidate bundle
    -> compare-and-swap current root from expected R to candidate R+1
```

The root compare-and-swap is the only publication linearization point. Root
storage uses two fixed-size slots with magic, monotonically increasing revision,
bundle identity, and CRC-32C over the slot bytes. Publication writes the
inactive slot only after the complete immutable bundle is durable; the highest
valid monotonic root is authoritative. A torn root write leaves the prior valid
slot authoritative. A crash after bundle persistence but before root
publication leaves only an unreferenced candidate, which readers ignore and
later cleanup may remove.

The implemented root codec is a fixed 94-byte `ICYDBASR` record with a
big-endian `u16` codec version, `u64` revision, 32-byte semantic fingerprint,
`u64` bundle key, 32-byte complete bundle hash, and big-endian CRC-32C. The two
slots are logical values under schema-key namespace `2` in the existing schema
BTree. Selection chooses the highest valid revision, tolerates one torn or
checksum-invalid inactive slot when the other is valid, rejects conflicting
roots at the same revision, and rejects a parseable non-current root version
instead of falling back. Publication persists and verifies the immutable
bundle, rechecks the expected root revision, writes the inactive slot, and
then verifies that root selection returns the candidate.

The compare-and-swap runs under the store's schema-mutation serialization and
has this semantic API:

```rust
fn publish_candidate(
    expected_revision: AcceptedSchemaRevision,
    candidate: CandidateSchemaRevision,
) -> Result<(), PublishError>;
```

If the current root is not `expected_revision`, publication returns typed
`StaleSchemaRevision`. The stale candidate must be rebuilt from the new root.
It cannot publish IDs allocated from its obsolete base and does not advance
the accepted ID frontier. Two candidates built from revision `R` cannot both
publish.

No request may observe:

- a field referencing a missing enum type;
- a value referencing a missing variant;
- different fields using partially updated definitions;
- defaults admitted under a different catalog revision;
- indexes built against a different enum key contract;
- a candidate ID allocation without its catalog definition.

Journal batch format version 2 adds one `AcceptedSchemaPublish` record carrying
the store path, expected revision, complete immutable bundle bytes, and root
bytes. The shared commit marker embeds that batch before the store journal is
appended. Apply writes accepted entity snapshots and the bundle to the live
schema overlay before writing the inactive live root; only then is the marker
cleared. Recovery repairs a missing marker-bound tail append, validates every
bundle entity tag/path against the registered store hooks, replays the live
projection, and folds snapshots/bundle/root into the canonical schema BTree.
Replay and fold are idempotent when the candidate root is already current.

The journal/commit machinery does not replace the root rule. The journal makes
the complete candidate durable before its failure-atomic root update. Catalog
and entity snapshot keys are never independent visibility points. Failure
before root publication leaves the prior accepted revision authoritative;
failure afterward retains marker/tail authority for idempotent recovery.

## Revision And Cache Authority

Every admitted query value and compiled plan is associated with the immutable
accepted schema revision used for admission.

Revision and fingerprint have separate meanings:

- `AcceptedSchemaRevision` is the monotonic publication identity for one store
  catalog.
- `AcceptedSchemaFingerprint` is a hash of canonical accepted semantic content
  and interpretation-relevant format versions.

The semantic fingerprint includes:

- store, schema, row/value, index-key, and cursor format identifiers relevant
  to interpretation;
- canonical enum definition bytes;
- type and variant IDs;
- enum paths, names, bodies, and ordering policies;
- field-to-type references;
- payload accepted value contracts;
- equality-key format version where applicable.

It excludes the monotonic revision, derived lookup maps, proposal origins,
source/declaration order, cache state, persisted allocation frontiers, and the
fingerprint field itself. An exact semantic no-op changes neither revision nor
fingerprint.

Accepted session catalog selections now come from the current root-selected
bundle. Shared query-plan cache identities constructed from that catalog carry
the store-local accepted revision in addition to the entity schema version and
fingerprint, so a successful SQL DDL root publication cannot reuse the prior
shared plan entry. The accepted row layout retains the complete immutable
`AcceptedSchemaAuthority` from the catalog handle that admitted it; prepared
plans do not construct a parallel source of runtime authority. Continuation
contracts project only the durable signature components they require: the
entity snapshot fingerprint plus the accepted root revision and fingerprint.
Logical plan rewrites carry that immutable continuation identity unchanged.
Immediately before row access, both SQL execution contexts and fluent
prepared-plan paths compare the full retained authority with the current store
root: the process-local store scope, accepted revision, and semantic root
fingerprint must all match. Typed shared-plan conversion additionally requires
the entity tag, entity path, and store path to match the requested entity type.
A mismatch returns the typed `StaleSchemaRevision` conflict before row access.
Compiled SQL cache identities carry the same accepted revision, and the context
retains its query/update surface so the authority guard does not weaken public
update write bounds. SQL parameter placeholders remain unsupported in 0.200
and reject before lowering, so no unadmitted parameter value can enter a plan
or execution context. A future parameter surface must bind only after this
authority guard.

The current-root comparison reads only the bounded checksummed root record. It
does not decode the catalog bundle, schema snapshots, or rows merely to prove
that a plan is current. Store scope is compared through the catalog's opaque
process-local provenance token; revision and fingerprint are compared against
the selected durable root. Accepted-schema context cache hits use that same
complete authority check, so recreating a heap or recovered store at the same
numeric revision and semantic fingerprint reloads its catalog instead of
reusing a handle from the prior process-local store scope.

Retaining the old immutable catalog is useful for canonical literals,
diagnostics, and explaining how a plan was built. It does not authorize live
execution against rows governed by a newer accepted revision.

After the execution-time guard succeeds, value-bearing output retains the
plan's catalog handle until enum IDs have been rendered to names. Cursor
emission uses the same plan's accepted continuation identity. Neither boundary
may reacquire a possibly newer catalog or recompute schema identity from
mutable session state after execution.

Planner field slots encode field authority as one disjoint state:

```rust
enum FieldSlotAuthority {
    Unresolved,
    ModelOnly(FieldKind),
    Accepted(AcceptedFieldKind),
}
```

A schema-backed accepted slot therefore cannot also retain generated kind
metadata or make a fallback choice later. Schema-backed slot construction
selects `Accepted` only from the accepted `SchemaInfo` contract and fails
closed when that contract is absent. Explicit generated authoring, lowering,
and parity-test paths select `ModelOnly`; expression placeholders with no field
contract select `Unresolved`. Exact generated/accepted contract equality is
enforced during proposal reconciliation and publication, not re-decided by
each planner or executor consumer.

Aggregate setup projects the accepted kind once into a compact, copyable
execution contract containing only the declared diagnostic kind, decoded
runtime representation, and comparison strategy. Per-group reducer states
reuse that projection rather than cloning recursive `AcceptedFieldKind` values
or resolving field semantics per row. Full scalar constraints, recursive
collection shape, and enum identity are validated by accepted row decode before
aggregate execution; the compact aggregate contract guards only the decoded
top-level representation required by its hot path.

Schema-backed predicate normalization likewise dispatches through
`AcceptedFieldKind` for recursive collection shaping, equality-like numeric
canonicalization, and field-to-field coercion selection. The generated
`FieldKind` dispatcher is available only when `SchemaInfo` explicitly has no
accepted catalog authority, for model-only authoring/lowering and parity tests.
An accepted view never falls back when a field contract is missing; the value
is left unchanged for the subsequent schema validator to reject.

Grouped `HAVING` literal canonicalization uses the same planner-slot authority
split. A slot carrying an accepted kind recurses and selects scalar conversion
from that accepted kind. A slot without one is explicitly model-only and may
use its generated proposal kind during authoring/lowering tests; runtime
aggregate execution continues to reject such a slot before row access.

Grouped-key identity canonicalization, borrowed-probe eligibility, and direct
single-field hashing also derive exclusively from accepted slot kinds. An
explicitly model-only or unresolved slot cannot select those runtime fast
paths; model-only planner and test evaluation takes the generic canonical key
path, while live grouped execution rejects the slot before row access.
Accepted enum group-key fast paths remain disabled under the initial enum
grouping prohibition.

Live plan execution is normative:

1. Read the store's current accepted root.
2. Require the plan's store scope, revision, and fingerprint to match it.
3. When parameter support exists, normalize prepared parameters against that
   current accepted bundle. Initial 0.200 rejects placeholders before lowering.
4. Acquire or revalidate a schema execution guard for the same root before any
   row access.
5. Execute and render output through that guarded bundle.

A publication between parameter binding and the execution guard returns typed
`StaleSchemaRevision`; the cache misses and the query recompiles/replans. The
publisher cannot replace the current root during a guarded synchronous
execution. If a future execution model permits such interleaving, it must pin a
complete historical data-and-schema snapshot before allowing old-revision
execution. Catalog retention alone is never sufficient.

## 0.200-Native Enum Mutation Policy

This policy applies only to stores already created in the 0.200 format. It is
not pre-0.200 migration policy.

| Operation | Initial 0.200 policy |
| --- | --- |
| Exact canonical definition | accept as no-op; no revision, fingerprint, or ID change |
| Source declaration reorder | canonical no-op under `EqualityOnly` |
| Add unused enum type | catalog-safe when an internal proposal surface exists |
| Add unit variant | accept only when every generated contract agrees and existing indexes use ID encoding |
| Add payload variant | accept only when payload admission exists and no prohibited index/group/distinct surface references the type |
| Add multiple types/variants in one candidate | allocate in canonical path/name byte order |
| Existing field rebind from enum type A to B | reject |
| Existing enum field changed to non-enum | reject under existing field rewrite policy |
| Existing non-enum field changed to enum | reject under existing field rewrite policy |
| Remove final field reference to enum type | field operation may proceed; retain the unused catalog type |
| Rename variant | reject |
| Drop variant | reject |
| Change payload kind | reject |
| Change payload storage decode | reject |
| Rename type/path | reject initially |
| Add alias | reject categorically |
| Change ordering policy | reject |
| Direct or mutually recursive enum contract | reject |
| Type or variant ID exhaustion | reject without wrapping |

Initial mutation vocabulary needs only current semantics:

```rust
enum EnumCatalogMutation {
    AddEnumType { proposal: CanonicalEnumProposal },
    AddEnumVariant {
        type_id: EnumTypeId,
        proposal: CanonicalEnumVariantProposal,
    },
}

enum EnumCatalogDelta {
    DefinitionUnchanged { type_id: EnumTypeId },
    TypeAdded { type_id: EnumTypeId },
    VariantAdded {
        type_id: EnumTypeId,
        variant_id: EnumVariantId,
    },
    Rejected { reason: EnumMutationRejection },
}

enum EnumMutationRejection {
    ExistingFieldTypeRebind {
        field: FieldPath,
        from: EnumTypeId,
        to: EnumTypeId,
    },
    // Other typed current-policy rejections.
}
```

The initial accepted enum catalog is append-only: types and variants may be
added, but accepted definitions are not removed and IDs are never reused. An
enum type remains in the catalog after its final field reference is removed.
Proposal reconciliation must not interpret absence from the current generated
field set as a catalog drop request.

Resolver order prevents type rebinding from masquerading as an addition. Every
existing field is first bound to its accepted kind and existing enum type ID.
For `AcceptedFieldKind::Enum { type_id: A }`, the candidate must retain `A`.
A changed generated Rust enum path or contract for that field rejects against
`A`; it cannot allocate a new type `B` and retarget the field. Only genuinely
new fields may bind to a new or existing enum type.

Do not build rewrite requirements, proof runners, row scanners, or migration
orchestration for rejected operations. Diagnostics may state that a future
physical rewrite design would be required.

## SQL Surface

The catalog is internal schema authority. It does not require SQL enum DDL.

Initial SQL behavior:

- reads and writes resolve enum values through accepted field contracts;
- bare string labels require a target enum field context;
- explicit qualified literals resolve path to type ID before planning;
- cross-type comparison rejects;
- payload enum SQL authoring remains unsupported unless a separate syntax and
  admission design lands;
- enum ordering, range comparisons, and range indexes reject.

Initial SQL DDL rejects:

- `CREATE TYPE`;
- `ALTER TYPE`;
- `DROP TYPE`;
- variant positioning, rename, drop, or alias syntax.

A later SQL DDL design may lower catalog operations onto the same schema-native
mutation boundary. It must not define separate mutation semantics.

## Validation Plan

### Hard-Cut Format

- Current database format opens successfully.
- No `format_memory_id` or new stable-memory allocation is introduced.
- Allocation cardinality remains four memories per journaled store plus the
  existing canister-wide database control memory.
- Data, index, schema, journal, and canister control allocations cannot collide
  by memory ID or stable key.
- The schema BTree uses its complete schema allocation and has no format-page
  offset convention.
- The control store contains no tree, map, generic `Value`, or infallible
  persisted decoder.
- Crash before boot-record write leaves the complete database recovery domain
  physically virgin.
- Crash after boot-record write but before catalog/BTree initialization reopens
  as a valid current-format empty database.
- No journal, control, data, index, or schema metadata can be durably created
  before the boot record.
- Missing boot record initializes only when the control allocation and every
  registered durable store allocation are physically virgin.
- Missing boot record with any durable database state rejects before commit or
  schema decode.
- Malformed boot bytes classify as `MalformedFormatMarker` corruption.
- Unsupported boot-record version rejects before commit, journal, schema, row, or
  index payload decode.
- One commit marker can cover multiple stores and recovery replays it under the
  same admitted database format.
- A newly declared journaled store joins the current format only with physically
  empty allocations; non-empty newly declared allocations reject before decode.
- Unsupported raw schema envelope rejects before Candid field-kind decode.
- Unsupported row envelope rejects before row payload decode.
- Current schema, row, journal, commit, index, cursor, and export encoders emit
  only current outer or nested versions.
- Every transitive persisted-format inventory entry names its rejecting version
  boundary or records that the surface is absent.
- Recovery never attempts to reconstruct accepted authority from generated
  models after format rejection.

Tests use synthetic unsupported version bytes, not retained legacy fixtures or
old feature spellings.

### Catalog And Identity

- Every enum field references an existing catalog type ID.
- Multiple fields sharing one enum use one catalog entry.
- Same label under different type IDs is unequal.
- Same variant label under different type IDs produces different keys.
- Type and variant IDs remain stable across unrelated native publications.
- Candidate ID allocation is not visible after rejection.
- ID zero rejects, checked ID exhaustion does not wrap, and no-op candidates do
  not allocate.
- Type IDs allocate in canonical path byte order and variant IDs allocate in
  canonical name byte order regardless of entity registration, proposal
  collection, source declaration, or scheduling order.
- Catalog name/ID maps validate as bijections.
- Unit variants cannot carry payload contracts.
- Payload variants always carry bundled kind/decode contracts.
- Direct and mutual enum payload cycles reject with typed cycle diagnostics.
- Acyclic forward references resolve through two-pass ID binding.
- Removing a final field reference retains the unused append-only catalog type.

### Proposal And Generated Contract

- Same definitions in different source orders canonicalize equally.
- Source reorder produces the same schema fingerprint.
- Conflicting same-path proposals reject before publication.
- Variant membership mismatch rejects.
- Unit/payload mismatch rejects.
- Payload kind mismatch rejects.
- Payload storage-decode mismatch rejects.
- Ordering-policy mismatch rejects.
- Top-level and nested accepted snapshot enum type IDs must resolve through the
  bundle catalog before publication or decode succeeds.
- Publication accepts only the current exact generated canonical contract.

### Canonical Values

- Unqualified input requires exactly one expected enum contract.
- Qualified input resolves path to the expected type ID.
- Unknown type or variant rejects before planning/storage.
- Runtime enum values always carry mandatory type and variant IDs.
- Canonical ID constructors are not available to public authoring APIs.
- A canonical enum value from another independent store is not accepted as an
  authoring value.
- Cross-store import requires unresolved names/contracts or explicit catalog
  remapping.
- Unit values cannot carry payloads.
- Payload values cannot omit payloads.
- Recursive payload mismatch rejects.
- Values beyond 64 levels or the applicable encoded-size budget reject with a
  typed error without stack overflow or over-allocation.
- Decoded values are strictly validated and never normalized or repaired.
- Infallible input conversion cannot construct a runtime enum.
- `AcceptedValueRef` cannot outlive the authority and contract it proves.

### Boundary Routing

- Generated and structural writes use accepted admission.
- SQL literal writes use accepted admission; parameter placeholders reject
  before lowering until a revision-guarded parameter API exists.
- Defaults are canonical before publication.
- Predicate and `IN (...)` values are canonical before plan hashing.
- Generic and ByKind decode use strict validation.
- Same-store typed and structural batch paths cannot bypass admission;
  cross-store import is not exposed.
- Persistence encoders require `AcceptedValueRef`; key encoders require the
  equivalent admitted proof for their capability.
- Arbitrary canonical `Value` is not accepted by structural, SQL, RPC, or
  import authoring surfaces.
- The enum wire codec freezes the `0x84`, fixed-ID, body-tag, and payload-length
  layout and rejects every noncanonical shape.

### Equality And Ordering

- Unit enum equality keys include type and variant IDs.
- Unit enum equality components match the exact 11-byte frozen vector.
- Equal values produce equal keys.
- Different type or variant IDs produce different keys.
- Declaration reorder does not change keys.
- Unrelated variant addition does not change existing keys.
- Payload enum index/group/distinct/hash/dedup/partition/cursor-key proposals
  reject.
- Payload enum direct equality remains pairwise and cannot silently choose a
  hash-key execution route.
- `(unit_enum, ordered_field)` indexes support an equality-fixed enum prefix and
  later-field ordering/cursors.
- Unconstrained enum-prefix scans and `ORDER BY enum` cannot expose ID byte
  order.
- Cross-type equality rejects.
- Enum order/range operators, `ORDER BY`, `MIN`, `MAX`, and range routing
  reject.

### Atomicity And Cache Lifecycle

- A field cannot publish with a missing type or variant reference.
- Readers observe one complete immutable revision bundle through one root.
- A torn root write preserves the previous valid checksummed root slot.
- A crash after candidate persistence but before root compare-and-swap leaves
  the old root authoritative.
- Catalog and affected entity snapshots become visible only at root
  compare-and-swap.
- Two candidates built from the same base revision cannot both publish.
- The stale candidate does not advance accepted IDs; rebuilding allocates from
  the newly accepted catalog.
- Defaults and index contracts use the candidate catalog revision.
- Accepted schema fingerprint changes when enum semantics change.
- Revision is excluded from semantic fingerprint content, and an exact no-op
  changes neither.
- Cache entries miss after catalog revision/fingerprint change.
- A cached plan retaining an old catalog cannot execute against a newer live
  accepted revision.
- SQL parameter placeholders reject before lowering in initial 0.200. Future
  prepared parameters may be admitted only after the plan/current-root check,
  and a revision change between binding and execution must return typed stale
  revision or replan before row access.
- Execution and output rendering use the same guarded revision bundle.

### Native Mutation

- Exact definition is a no-op.
- Source declaration reorder is a no-op.
- Unit variant addition preserves existing IDs and keys.
- Payload variant addition rejects when prohibited capabilities reference the
  type.
- An existing enum field cannot change from type ID `A` to `B`.
- Changing only the generated enum path for an existing field rejects without
  allocating a substitute type.
- Existing enum/non-enum type changes reject under field rewrite policy.
- Removing the final field reference retains the enum type and its IDs.
- All unsupported mutations return typed current-policy rejection.

Broader implementation checks include focused schema, persisted-row, index,
query, SQL, recovery, and generated-facade suites. Each implementation slice
reports raw non-gzipped wasm bytes when it changes wasm-reachable code.

## Closeout Standard

0.200 is complete only when:

1. Every database recovery domain admits one crash-safe boot record from its
   existing control allocation before any recovery or durable payload decode.
2. Exactly one catalog-backed enum persisted representation exists.
3. Type and variant IDs are mandatory and stable.
4. Fields reference catalog IDs and never embed accepted definitions.
5. Generated proposals exactly match accepted canonical definitions.
6. Runtime enum values cannot represent unresolved identity.
7. Construction admission and strict decode validation are separate.
8. Every write, default, predicate, patch, decode, and bulk path uses the
   accepted admission boundary, and no executable row-decode contract can omit
   its immutable catalog authority.
9. Unit enum equality keys are versioned and ID-based.
10. Canonical enum value bytes and unit enum key bytes have frozen current
    format vectors.
11. Payload enum index/group/distinct/hash/cursor-key shapes reject until
    canonical payload equality encoding exists.
12. Pairwise equality and canonical stable-key capabilities are distinct.
13. Declaration reorder is a semantic no-op under `EqualityOnly`.
14. Enum ordering and range semantics reject, while equality-fixed unit enum
    composite-index prefixes remain usable.
15. One checksummed current-schema root compare-and-swap is the catalog and
    snapshot publication linearization point.
16. Live plan execution requires the current guarded store scope, revision, and
    fingerprint; retaining an old catalog is insufficient.
17. Canonical enum IDs and admitted owned values cannot cross store catalog
    authority without explicit remapping.
18. Recursive enum type graphs reject, and all admission/decode paths enforce
    bounded depth and size.
19. Existing fields cannot rebind from one enum type ID to another, and unused
    catalog definitions remain append-only.
20. Initial native mutation supports only exact/no-op and admitted additions.
21. Every transitive persisted container has a current rejecting version gate
    or is explicitly recorded as absent.
22. No migration runner, compatibility reader, alias, fallback, repair, or
    generated reconstruction remains in the 0.200 contract.
