0.108 Canonical Query Identity

Thesis

`0.108` should make planner and cache identity follow the canonical semantic
query form instead of preserving avoidable representation-level divergence.

After `0.105` and `0.107`, equivalent scalar `WHERE` expressions can already
normalize and canonicalize to the same semantic filter shape. `0.108` makes
the rest of the system honor that fact by aligning fingerprint, cache-key, and
nearby semantic identity surfaces with the canonical form.

This is an identity-alignment slice, not a new optimizer or SQL-surface slice.

Problem statement

After `0.105` and `0.107`:

* predicate extraction is structurally normalized
* bounded semantic canonicalization can collapse some equivalent boolean forms
* explain can present the canonical semantic filter surface

However, nearby identity surfaces may still diverge if they continue to use
pre-canonical or spelling-sensitive structure.

That means semantically equivalent queries may still:

* generate different fingerprints
* miss query-cache reuse
* fragment explain or planning identity
* behave as though they are different shapes even when the planner already
  knows they are the same

Concrete examples include:

* `a = 1 AND b = 2`
* `b = 2 AND a = 1`
* `CASE WHEN a THEN TRUE ELSE b END`
* `(a AND TRUE) OR (NOT a AND b)`

After `0.105` and `0.107`, these should already collapse to the same semantic
filter shape. `0.108` makes that canonical shape the identity owner.

Design goal

Make canonical semantic query shape the authority for identity surfaces that
are supposed to model planner-visible semantics.

That means:

* semantically equivalent canonicalized filters share the same fingerprint
* semantically equivalent canonicalized filters share the same cache identity
* nearby query-shape surfaces stop drifting from canonical semantic form
* no SQL admission, template, or route-selection widening occurs

Non-goals

`0.108` does not aim to:

* broaden SQL admission
* introduce new semantic canonicalization rules
* change predicate extraction policy
* change route scoring or cost heuristics
* widen prepared template eligibility
* introduce a new cache layer
* merge identities that are not already canonical-semantic equivalents

This is not a new rewrite slice. It is the identity follow-through slice for
work already done in `0.105` and `0.107`.

Normative model

Canonical semantic query identity

For the purposes of `0.108`, canonical query identity consists of:

* canonicalized scalar filter expression (`filter_expr`)
* projection shape
* grouping shape, if any
* ordering shape as declared semantically by the user, not planner-added
  tie-break terms
* `LIMIT` / `OFFSET` semantics

Identity alignment in this slice applies only to components that already have
canonical semantic representations, with primary focus on the scalar filter
expression.

Rule 1 - canonical semantic form owns semantic identity

When a query has a planner-owned canonical semantic filter form, the filter
component of semantic identity must derive from that canonical form.

Identity surfaces that are intended to reflect planner-visible semantics must
derive that filter component from the canonical form rather than from
pre-canonical spelling.

That includes:

* fingerprint surfaces
* semantic query cache keys
* nearby explain/query-shape identity surfaces

It does not include:

* user-facing SQL spelling
* syntax-preserving debug surfaces

Rule 2 - identity alignment is post-canonicalization

The required phase order is:

* parse
* validate
* bounded semantic canonicalization
* structural normalization
* extraction / residual derivation
* identity projection

Identity must not be projected from pre-canonicalized boolean form when a
canonical semantic form already exists.

Rule 3 - alignment only applies to already-collapsed equivalence classes

`0.108` does not invent new equivalence classes.

It only aligns identity for shapes that already normalize or canonicalize to
the same planner-owned semantic form.

So if two queries still have different canonical semantic forms after
`0.105`/`0.107`, they remain distinct identities in `0.108`.

Rule 4 - cache reuse follows semantic identity, not spelling

If two admitted queries produce the same canonical semantic plan shape, cache
identity should treat them as the same query shape for semantic planning
purposes.

This means cache reuse may improve across:

* reordered boolean children
* duplicate-boolean-child collapse
* bounded searched `CASE` canonicalization

Cache identity must still incorporate full semantic plan shape.

Canonical filter alignment does not permit reuse across differing:

* projection shape
* grouping shape
* ordering shape
* `LIMIT` / `OFFSET` semantics

This does not authorize reuse across:

* different admitted semantics
* different projection shapes
* different grouping semantics
* different route-affecting but semantically visible non-filter plan features

Rule 5 - prepared boundary remains unchanged

Identity alignment must not:

* widen prepared templates
* create a new prepared template identity surface
* silently merge prepared and non-prepared identity domains

Prepared fallback may reuse canonical semantic identity only where it already
shares the same planning path.

This slice does not:

* introduce new plan reuse guarantees across bindings
* widen prepared caching behavior
* introduce a new prepared contract

Rule 6 - explain and fingerprint should not drift

Where explain and fingerprint both claim to represent semantic query shape,
they should derive from the same canonical semantic authority.

`0.108` should remove cases where:

* explain stabilizes on canonical form
* but fingerprint/cache identity still reflects pre-canonical structure

Both explain rendering and fingerprint encoding must derive from the same
canonical expression tree representation for the filter component so those
surfaces do not silently diverge while both claim to be canonical.

Rule 7 - route choice is not identity

Route choice remains a planner-time decision, not semantic query identity.

So:

* canonical semantic identity may be the same
* while selected access route may still vary due to visibility, cost, or other
  planner inputs

`0.108` must not collapse route choice into semantic identity.

Examples

Equivalent normalized `AND` shapes

These should share identity:

```sql
SELECT name
FROM character
WHERE a = 1 AND b = 2
```

```sql
SELECT name
FROM character
WHERE b = 2 AND a = 1
```

Expected result:

* identical canonical filter shape
* identical fingerprint
* identical semantic cache identity

Bounded searched `CASE` equivalence

These should share identity after `0.107` canonicalization:

```sql
SELECT name
FROM character
WHERE CASE WHEN a THEN TRUE ELSE b END
```

```sql
SELECT name
FROM character
WHERE (a AND TRUE) OR (NOT a AND b)
```

Expected result:

* identical canonical filter shape
* identical fingerprint
* identical semantic cache identity

Distinct canonical forms remain distinct

These should not be merged if they still canonicalize differently:

```sql
SELECT name
FROM character
WHERE a = 1 OR b = 2
```

```sql
SELECT name
FROM character
WHERE a = 1 AND b = 2
```

Likely code seams

* structural query fingerprint projection
* semantic query cache key construction
* canonical filter-shape rendering or encoding helpers
* explain/query-shape identity plumbing

Risks

Risk 1 - accidental over-merging

If identity alignment merges queries that are not truly canonical-semantic
equivalents, cache reuse and planner reasoning will become unsound.

Mitigation:

* only project identity from the existing canonical semantic form
* do not add new equivalence rules in this slice

Risk 2 - partial alignment

If explain, fingerprint, and cache do not all follow the same semantic owner,
drift will remain and the slice will be incomplete.

Mitigation:

* use one canonical semantic identity authority for all targeted surfaces

Risk 3 - route/identity confusion

If route choice leaks into fingerprint or semantic cache identity, equivalent
queries may stop sharing identity for planner-external reasons.

Mitigation:

* keep route choice out of semantic identity
* keep `0.106` route selection and `0.108` identity alignment separate

Relationship to earlier slices

Slice | Role
--- | ---
`0.105` | structural normalization for extraction stability
`0.106` | cost-aware route selection among valid plans
`0.107` | bounded semantic canonicalization for searched `CASE`
`0.108` | semantic identity alignment for fingerprint and cache surfaces

Verification matrix

Area | Required proof
--- | ---
Fingerprint | semantically equivalent canonicalized filters produce identical fingerprints
Cache identity | semantically equivalent canonicalized filters produce identical semantic cache keys
Explain parity | explain semantic shape and identity surfaces derive from the same canonical form
Distinctness | semantically distinct canonical forms remain different identities
Prepared boundary | no template widening or new prepared identity surfaces are introduced
Route separation | route choice remains independent from semantic identity

Acceptance criteria

`0.108` is complete when:

* canonical semantic filter form is the authority for targeted identity surfaces
* semantically equivalent canonicalized filters share fingerprint and cache identity
* semantically distinct canonical forms remain distinct
* explain/fingerprint/cache semantic surfaces no longer drift from one another
* no SQL admission, prepared-template, or route-selection widening occurs

Thesis sentence

`0.108` aligns fingerprint, cache-key, and nearby semantic query-shape
surfaces with the planner-owned canonical filter form so semantically
equivalent queries share identity without widening SQL, rewrite policy, or
prepared boundaries.
