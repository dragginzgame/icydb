# 0.56 — SQL `DESCRIBE`

Status: **Implemented (`0.56.0`, 2026-03-15)**

Target: **`0.56.x` minor line**

Theme: **Reduced SQL introspection and schema visibility**

## Intent

Initial intent for `0.56` was one explicit SQL introspection command:

```sql
DESCRIBE <entity>
```

Implementation outcome in `0.56.0` kept this lane and also shipped
`SHOW INDEXES <entity>` plus generated-helper `SHOW ENTITIES`.

This line does **not** expand query semantics. It exposes existing schema-introspection
authority through the reduced SQL frontend, the same way current SQL entrypoints already
expose scalar/projection/grouped/aggregate/explain session APIs.

`DESCRIBE` should be implemented as a frontend-to-session projection over existing schema
metadata, not as a query-planner feature.

## Why This Fits Current Architecture

The SQL subset contract already requires frontend lowering into existing authority
surfaces, without semantic emulation layers.

Existing typed introspection already exists:

- `DbSession::describe_entity<E>() -> EntitySchemaDescription`

`EntitySchemaDescription` already carries:

- `entity_path`
- `entity_name`
- `primary_key`
- `fields`
- `indexes`
- `relations`

Field entries already carry:

- `name`
- `kind`
- `primary_key`
- `queryable`

Index entries already carry:

- `name`
- `unique`
- `fields`

Relation entries already carry:

- `field`
- `target_path`
- `target_entity_name`
- `target_store_path`
- `strength`
- `cardinality`

Design consequence: SQL `DESCRIBE` should wrap existing schema authority rather than
introducing new schema derivation logic.

## Primary User-Visible Contract

### Supported Syntax

```sql
DESCRIBE <entity>
```

Examples:

```sql
DESCRIBE User
DESCRIBE public.User
DESCRIBE auth.User
```

### Output

Core:

```rust
pub fn describe_sql<E>(&self, sql: &str) -> Result<EntitySchemaDescription, QueryError>
where
    E: EntityKind<Canister = C>;
```

Facade:

```rust
pub fn describe_sql<E>(&self, sql: &str) -> Result<EntitySchemaDescription, Error>
where
    E: EntityKind<Canister = C>;
```

Return type should remain `EntitySchemaDescription` unchanged to avoid introducing a
second schema format.

## Non-Goals

`0.56` should **not** include:

- `DESCRIBE field`
- `DESCRIBE INDEX <name>`
- `SHOW TABLES`
- `SHOW COLUMNS`
- arbitrary catalog browsing
- planner/executor integration for row execution
- query fingerprints for `DESCRIBE`
- `EXPLAIN DESCRIBE ...`

Reason: current schema surface is entity-scoped and typed, and reduced SQL remains
intentionally small and fail-closed.

## Canonical Authority

Authority remains:

- runtime entity model metadata
- `describe_entity_model(...)`
- `EntitySchemaDescription` and sub-DTOs

`DESCRIBE` must **not**:

- invent field-kind formatting separate from canonical schema summarization
- invent index rendering outside schema DTO contracts
- re-derive relation metadata in parser/session layers
- fork schema output shape from existing DTO serialization-shape tests

## Statement Model

Add parser-owned statement contract:

```rust
pub struct SqlDescribeStatement {
    pub entity: String,
}

pub struct SqlShowIndexesStatement {
    pub entity: String,
}

pub enum SqlStatement {
    Select(SqlSelectStatement),
    Delete(SqlDeleteStatement),
    Explain(SqlExplainStatement),
    Describe(SqlDescribeStatement),
    ShowIndexes(SqlShowIndexesStatement),
}
```

This keeps statement-kind ownership at the parser boundary and compiles into a typed
session command, consistent with current reduced parser structure.

### Route Model (`sql_statement_route`)

`DESCRIBE` should also become a first-class route lane.

Extend route metadata to include:

```rust
pub enum SqlStatementRoute {
    Query { entity: String },
    Explain { entity: String },
    Describe { entity: String },
    ShowIndexes { entity: String },
}
```

`sql_statement_route(...)` is already the SQL dispatch authority outside typed-entity
lowering paths, so it should classify `DESCRIBE` directly rather than leaving route
metadata stale.

## Lowering Contract

### Extend Existing Lowered Command Surface

This line should **extend** the current lowered command surface, not replace it.
Current lowering already includes `ExplainGlobalAggregate`; `0.56` should preserve that
lane and add describe introspection alongside it.

```rust
pub(crate) enum SqlCommand<E: EntityKind> {
    Query(Query<E>),
    Explain { mode: SqlExplainMode, query: Query<E> },
    ExplainGlobalAggregate { mode: SqlExplainMode, command: SqlGlobalAggregateCommand<E> },
    DescribeEntity,
    ShowIndexesEntity,
}
```

Optional future-proofing (not required for `0.56`):

```rust
pub(crate) enum SqlIntrospectionCommand {
    DescribeEntity,
    ShowIndexesEntity,
}

pub(crate) enum SqlCommand<E: EntityKind> {
    Query(Query<E>),
    Explain { mode: SqlExplainMode, query: Query<E> },
    ExplainGlobalAggregate { mode: SqlExplainMode, command: SqlGlobalAggregateCommand<E> },
    Introspection(SqlIntrospectionCommand),
}
```

### Lowering Rule

`DESCRIBE <entity>` lowering boundaries:

1. Parser accepts reduced identifier syntax and statement shape (`SqlStatement::Describe`).
2. Lowering validates trailing-segment entity match against `E`.
3. Lowering returns one describe command variant.
4. Session dispatch calls `describe_entity::<E>()` and returns its DTO payload.

### Explicit Lane Rule

- `DESCRIBE` never lowers into `Query<E>`.
- `DESCRIBE` does not participate in planner validation.
- `DESCRIBE` is a session introspection command parallel to query/explain lanes.

### Entity Matching

Reuse existing SQL entity matching rule:

- trailing entity segment must match `E`
- schema-qualified names are accepted when trailing segment matches

## Session API Design

### Core

Add:

```rust
pub fn describe_sql<E>(&self, sql: &str) -> Result<EntitySchemaDescription, QueryError>
where
    E: EntityKind<Canister = C>;
```

Behavior:

- accepts only `DESCRIBE ...`
- rejects `SELECT`, `DELETE`, `EXPLAIN`
- validates entity identity like existing SQL entrypoints
- returns `self.describe_entity::<E>()`
- does not route through `Query<E>` or planner/executor row-shape validation

### Facade

Add forwarding surface in `icydb`:

```rust
pub fn describe_sql<E>(&self, sql: &str) -> Result<EntitySchemaDescription, Error>
where
    E: EntityKind<Canister = C>;
```

## Generated `sql_dispatch` Impact

`0.54.1` introduced generated canister SQL helpers (`sql_dispatch`), so `0.56`
needs an explicit helper-layer decision for `DESCRIBE`.

### `0.56` Decision

For `0.56`, `DESCRIBE` is supported on both session/facade and generated
`sql_dispatch` helper surfaces.

That means:

- `SqlStatementRoute` is still extended with `Describe { entity }` so routing
  metadata remains complete.
- generated `sql_dispatch::query(...)` supports DESCRIBE rendering in addition
  to projection/explain routing.
- generated `sql_dispatch::describe_schema(...)` exposes typed
  `EntitySchemaDescription` payloads for programmatic callers.
- generated `sql_dispatch::describe(...)` exposes shell-friendly describe lines.
- generated `sql_dispatch::query_rows(...)` remains projection-only and rejects
  DESCRIBE with deterministic unsupported-lane errors.

Rationale:

- keeps canister helper behavior aligned with the established session authority
  (`describe_sql::<E>(...)`) instead of duplicating schema semantics.
- preserves projection-only contracts for `query_rows(...)` while still
  allowing shell + programmatic introspection entrypoints.

### Explicitly Deferred Beyond `0.56`

- broadened catalog SQL beyond entity-scoped introspection
  (`SHOW COLUMNS`, `SHOW SCHEMA`, `DESCRIBE INDEX`, etc.).

### Required Helper-Layer Tests for `0.56`

- generated dispatch route classification includes `Describe` lane detection
- `sql_dispatch::query(...)` renders `DESCRIBE ...` with stable shell output
- `sql_dispatch::describe_schema(...)` returns typed describe payloads
- `sql_dispatch::query_rows(...)` rejects `DESCRIBE ...` with stable
  unsupported-lane error
- `sql_dispatch::query(...)` renders `SHOW INDEXES ...` with stable shell output
- `sql_dispatch::show_indexes(...)` returns shell index payloads
- `sql_dispatch::query_rows(...)` rejects `SHOW INDEXES ...` with stable
  unsupported-lane error
- `sql_dispatch::explain(...)` rejects `EXPLAIN DESCRIBE ...` fail-closed

## Output Contract

`DESCRIBE` should return the existing typed payload as-is:

```rust
EntitySchemaDescription {
    entity_path,
    entity_name,
    primary_key,
    fields,
    indexes,
    relations,
}
```

This avoids:

- bespoke text tables in core
- SQL-only schema row formats
- duplicate describe output structs
- planner projection layers for introspection

Pretty CLI rendering, if needed later, should live outside core/facade canonical
session contracts.

## Parser Behavior

### Accept

- `DESCRIBE User`
- `DESCRIBE public.User`

### Reject (Fail-Closed)

- `DESCRIBE`
- `DESCRIBE User extra`
- `DESCRIBE User WHERE id = 1`
- `DESCRIBE INDEXES User`
- `DESCRIBE "User"`
- `DESCRIBE User; SELECT * FROM User`

Normalization should match current reduced SQL parser behavior:

- keyword case-insensitive
- insignificant whitespace ignored
- optional trailing `;` accepted
- multi-statement input rejected

## Error Contract

Follow existing SQL error philosophy:

- invalid `DESCRIBE` syntax -> parser error
- unsupported `DESCRIBE` modifiers/forms -> unsupported-feature parser error
- entity mismatch -> typed lowering error
- wrong session entrypoint usage -> unsupported execution error

Recommended unsupported-feature labels:

- `DESCRIBE modifiers`
- `quoted identifiers`
- `multi-statement input`

Failure boundary clarification:

- parse acceptance: reduced SQL grammar and identifier-shape acceptance
- lowering validation: trailing entity segment match against `E`
- session dispatch: schema DTO projection via `describe_entity::<E>()` (infallible today)

## Why This Should Not Use `execute_sql(...)`

`execute_sql(...)`, `execute_sql_projection(...)`, `execute_sql_grouped(...)`, and
`execute_sql_aggregate(...)` are row-execution surfaces.

`DESCRIBE` is schema introspection, so a dedicated `describe_sql(...)` entrypoint
matches current lane-by-shape API design better than overloading execution entrypoints.

## Interaction with Existing Non-SQL Helpers

No change to:

- `describe_entity<E>()`
- `show_indexes<E>()`

In `0.56`, SQL `DESCRIBE` maps only to `describe_entity<E>()`.

## Scope Pressure Rationale

Keep `0.56` intentionally narrow (`DESCRIBE <entity>` only). As of 2026-03-15,
SQL frontend modules are already high-pressure surfaces:

- `db/sql/parser/mod.rs`: 1345 LOC
- `db/sql/lowering.rs`: 1423 LOC

That existing complexity reinforces deferring catalog DSL expansion (`SHOW TABLES`,
`SHOW COLUMNS`, `DESCRIBE INDEX`, broader browsing) to later lines.

## Sanity Checks (2026-03-16)

Checked against current checkout:

- `DbSession::describe_entity<E>()` and facade `describe_entity<E>()` already exist.
- Existing SQL session entrypoints are lane-specific; a dedicated `describe_sql(...)`
  entrypoint is structurally consistent.
- `SqlStatementRoute` now models `Query`, `Explain`, `Describe`, and `ShowIndexes`.
- Generated `sql_dispatch` now supports helper-level `SHOW ENTITIES`, route-keyed
  `DESCRIBE`/`SHOW INDEXES` rendering in `query(...)`, and dedicated
  `describe_schema(...)`/`describe(...)`/`show_indexes(...)` helpers, while
  `query_rows(...)` remains projection-only and rejects introspection lanes.
- SQL parser/lowering files are at `db/sql/parser/*` and `db/sql/lowering.rs`.
- Contradiction resolved: optional trailing `;` is accepted; single-statement
  `DESCRIBE User;` is not in reject list.
- Path corrected: this design doc lives at
  `docs/design/0.56-describe/0.56-design.md`.

## Final Recommendation

Keep `0.56` deliberately small:

- entity-scoped SQL introspection only (`DESCRIBE <entity>`, `SHOW INDEXES <entity>`)
- return `EntitySchemaDescription`
- expose via new `describe_sql<E>(...)`
- preserve canonical schema authority and fail-closed SQL boundaries
