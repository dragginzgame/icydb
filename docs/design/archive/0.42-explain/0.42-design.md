## Status

Complete

---

## 1. Objective

Ship `EXPLAIN` on top of the stabilized `0.41` execution vocabulary so
debugging output is a stable public contract rather than a moving internal
projection.

`0.42` is an observability/repr release, not an execution-shape expansion.
`EXPLAIN` is a stable debugging interface, not an optimizer diagnostic tool.

---

## 2. Non-Goals

- no new access/route operator additions in `0.42`
- no planner architecture redesign
- no cost model or join/distributed execution work
- no optimizer traces or cost-estimation surfaces
- no behavioral changes to query semantics
- no standalone SQL/CLI command integration work; `0.42` scope is core query API
  explain surfaces (`Query` / fluent load terminals)

## 4. Canonical Explain Contract

`EXPLAIN` in `0.42` is a serialization of execution nodes.

Pipeline model:

`planner -> access plan -> execution nodes -> EXPLAIN output`

### 4.1 ExecutionNodeDescriptor

Define one canonical descriptor used by all explain surfaces.

Core fields:

- `node_type`
- `execution_mode` (`Streaming` / `Materialized`)
- `children`
- `access_strategy` (with index details when present)
- `predicate_pushdown` (index prefilter details)
- `residual_predicate`
- `projection`
- `ordering_source`
- `limit`
- `cursor`
- `covering_scan`
- `rows_expected` (optional; nullable/omitted when unknown)
- `node_properties` (`Map<String, Value>` for node-local additive metadata)

### 4.2 Node Vocabulary (Frozen in 0.42)

- Access: `ByKeyLookup`, `ByKeysLookup`, `PrimaryKeyRangeScan`,
  `IndexPrefixScan`, `IndexRangeScan`, `FullScan`, `Union`, `Intersection`
- Predicate: `IndexPredicatePrefilter`, `ResidualPredicateFilter`
- Ordering: `OrderByAccessSatisfied`, `OrderByMaterializedSort`
- Distinct: `DistinctPreOrdered`, `DistinctMaterialized`
- Projection: `ProjectionMaterialized`, `ProjectionIndexOnly`
- Seek: `TopNSeek`, `AggregateSeekFirst`, `AggregateSeekLast`
- Cursor/Limit: `LimitOffset`, `CursorResume`, `IndexRangeLimitPushdown`
- Aggregates: `AggregateCount`, `AggregateExists`, `AggregateMin`,
  `AggregateMax`, `AggregateFirst`, `AggregateLast`, `AggregateSum`
- Grouped: `GroupedAggregateHashMaterialized`,
  `GroupedAggregateOrderedMaterialized`
- Pushdown: `SecondaryOrderPushdown`

### 4.3 Predicate Execution Model

Predicate execution is modeled as two explicit stages:

1. `IndexPredicatePrefilter`
   - predicate components compiled into index traversal/filtering
2. `ResidualPredicateFilter`
   - remaining predicate components evaluated after row materialization

When predicate execution splits across stages, both nodes must be emitted.

### 4.4 Canonical Representation

Canonical explain representation is:

`ExecutionNodeDescriptor -> JSON`

Text and verbose explain surfaces are renderers over the canonical descriptor
representation so all surfaces stay semantically aligned.

## 5. Surface Plan

### 5.1 `EXPLAIN` (base)

- human-readable tree over `ExecutionNodeDescriptor`
- minimal stable fields
- deterministic ordering

### 5.2 `EXPLAIN VERBOSE`

- base surface plus rejection reasons and route/hint diagnostics
- explicit pushdown eligibility outcomes

### 5.3 `EXPLAIN JSON`

- machine-consumable serialization of the same descriptor tree
- stable key naming and deterministic child ordering

### 5.4 Deterministic JSON Key Order

Canonical key order:

1. `node_type`
2. `execution_mode`
3. `access_strategy`
4. `predicate_pushdown`
5. `residual_predicate`
6. `projection`
7. `ordering_source`
8. `limit`
9. `cursor`
10. `covering_scan`
11. `rows_expected`
12. `children`
13. `node_properties`

All three surfaces must share one descriptor projection path.

---

## 6. Serialization Rules

- deterministic output for equivalent plans
- no hidden inference in rendering layer
- no surface-specific node renaming
- include explicit booleans/variants for pushdown, seek, and materialization
  decisions
- descriptor children must be emitted in runtime execution order
- preserve forward compatibility by allowing additive optional fields only

## 8. Slice Sequencing (`0.42.x`)

Sequence labels here are intentionally patch-agnostic. Future work is tracked
as ordered slices under `0.42.x` and must not assume concrete patch numbers.

### 8.1 Slice A: Descriptor Foundation (landed)

- canonical execution-node descriptor introduced for explain projection
- executor-owned descriptor assembly path established for terminal/load
  explain contracts
- base canonical text and JSON descriptor renderers introduced

### 8.2 Slice B: Stability Freeze and Drift Guards (landed)

- freeze execution-node vocabulary as a string contract
- add access-root matrix guards for canonical load roots
  (`ByKeyLookup`, `IndexPrefixScan`, `IndexRangeScan`, `IndexMultiLookup`)
- lock predicate-stage and `LIMIT 0` invariants in execution descriptors
- add strict text/JSON snapshots for one canonical index-prefix descriptor tree
- keep this slice contract-only: no new execution-shape features

Acceptance criteria for this slice:

- vocabulary drift fails deterministic tests
- descriptor hierarchy drift fails text and JSON snapshots
- route-root and predicate-stage drift fail matrix tests
- no changes to query semantics or planner shape authority

### 8.3 Slice C: Surface Adapters (landed)

- expose deterministic text/json execution explain from `Query` + fluent load API
- add initial verbose execution explain renderer using canonical descriptor
- lock method-level regression tests for these public surfaces

Acceptance criteria for this slice:

- no surface bypasses canonical descriptor assembly path
- text/json/verbose outputs are deterministic for equivalent plans
- public API surface supports direct execution explain access without manual rendering calls

### 8.4 Slice D: Verbose Diagnostics Completion (landed)

- add explicit route/pushdown diagnostics to verbose output
- include secondary pushdown rejection reasons and active hint metadata
- freeze verbose diagnostics with snapshot/matrix coverage

Acceptance criteria for this slice:

- verbose output reports both positive pushdowns and rejection causes
- route-hint metadata is explicit (seek/limit/cursor/predicate stage)
- diagnostics are additive and deterministic

### 8.5 Slice E: Exit Freeze and Contract Closure (landed)

- finalize unresolved descriptor family coverage
- remove stale checklist debt in status docs
- mark 0.42 exit criteria as complete
- keep scalar execution-descriptor scope explicit: grouped and projection node
  families remain vocabulary-reserved and are deferred until dedicated grouped
  and projection-specific execution explain surfaces are added

Acceptance criteria for this slice:

- no remaining explain-surface TODOs in `0.42` scope docs
- vocabulary + hierarchy + verbose diagnostics all drift-guarded
- clear deferral notes for any intentionally postponed node-family projections
