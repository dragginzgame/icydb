# IcyDB 0.57 Design

Status: **Superseded by `0.58-demonomorphization`**.

Theme: **Missing-capability closure on top of the shipped `0.56` SQL and
introspection baseline**.

Primary evidence for this line is tracked in:

- normative baseline:
  - `docs/contracts/SQL_SUBSET.md`
- release history / shipped status:
  - `docs/changelog/0.56.md`
- `docs/design/0.57-missing-features/icydb-feature-inventory-completeness-audit-2026-03-16.md`
- historical design rationale:
  - `docs/design/0.56-describe/0.56-design.md`

## Intent

`0.57` is a focused closure line for meaningful capability gaps identified by
the feature inventory audit.

The goal is not a broad architecture rewrite. The goal is to ship a small set
of additive, testable capability expansions while preserving current
canonical-authority boundaries and fail-closed behavior.

`0.57` should explicitly target partial or missing capability families that are
already close to current architecture:

- SQL lane-control-flow consolidation lock (as precondition gate)
- reduced-SQL introspection breadth
- constrained grouped SQL `HAVING` breadth
- expression-index eligibility breadth
- selected optimization closure items that have clear ownership boundaries

## Current Baseline (Input to 0.57)

From current checkout and the audit:

- reduced SQL supports:
  - `SELECT` / `DELETE` / `EXPLAIN`
  - grouped execution and constrained grouped `HAVING` (`AND` forms)
  - introspection lanes: `DESCRIBE`, `SHOW INDEXES`, `SHOW COLUMNS`,
    `SHOW ENTITIES`
- generated canister SQL dispatch is unified through `SqlQueryResult`
- expression indexes are shipped with intentionally bounded eligibility
  (`LOWER(field)` casefold `Eq` / `In` + prefix-lowering support)
- grouped optimization and some BYTES closure families remain deferred by design

This baseline means `0.57` should be selective and additive, not foundational.

## Scope Goals

`0.57` prioritizes the following capability outcomes.

1. Ensure SQL lane-control-flow consolidation is landed before new SQL-surface
   expansion; if not landed, execute that consolidation first as a precondition
   gate.
2. Add one bounded introspection command expansion that projects existing
   schema/runtime authority without introducing new catalog semantics.
3. Expand grouped SQL `HAVING` by exactly one additional bounded shape family
   (not broad OR/NOT expansion), with exact accepted/rejected shapes documented
   before implementation.
4. Expand expression-index eligibility on the already-shipped casefold family
   first (before introducing new expression-language families).
5. If an optional optimization slice is taken, prefer BYTES/index-only closure
   over grouped fast-path expansion.

## Non-Goals

The following remain out of scope for `0.57` unless explicitly re-scoped:

- joins
- distributed execution
- cost-based optimization
- broad SQL grammar expansion unrelated to listed goals
- broad execution architecture rewrites
- weakening fail-closed error boundaries
- moving semantic authority into generated helpers or frontend-only layers

Unless explicitly re-scoped, SQL writes (`INSERT`, `UPDATE`) remain outside the
core `0.57` scope because they are a large surface with significant semantic and
recovery implications.

## 0.57 Work Plan (Slices)

Patch labels are intentionally omitted; execute as ordered slices under `0.57`.

### Slice 0 (Precondition Gate) - SQL Lane-Control-Flow Consolidation

If lane-control-flow consolidation is not already landed, `0.57` must begin
with this hardening slice before any new SQL commands are added.

Current checkout note:

- this work is already present via late `0.56` consolidation, so this gate is
  expected to be marked complete at `0.57` line start.

Acceptance criteria:

- lane rejection/gating authority remains centralized
- generated dispatch does not re-introduce parallel lane checks
- core/facade/generated parity tests stay green
- no public API or fail-closed contract regressions

### Slice A - DTO-Backed Introspection Addition (Small, User-Facing)

Target this slice to one command that projects existing schema authority
directly, without broad catalog DSL growth.

Default candidate target for `0.57` line start: `SHOW TABLES` as a bounded
alias over the already-shipped `SHOW ENTITIES` lane and payload.

Acceptance criteria:

- parser/lowering/session/facade/generated dispatch all classify and route the
  new command deterministically
- command output is a projection over existing schema DTO ownership (no new
  catalog metadata authority)
- unsupported variants fail closed with stable unsupported-feature labels
- `SQL_SUBSET` docs and changelog are updated
- generated shell rendering remains deterministic

### Slice B - Grouped HAVING Expansion (Medium, User-Facing + Runtime)

Expand grouped SQL `HAVING` by exactly one additional bounded shape family,
without committing to broad OR/NOT support in this line.

The chosen family must be specified as explicit grammar/shape contracts before
implementation (accepted forms and adjacent rejected forms).

Acceptance criteria:

- SQL-to-fluent parity tests cover the exact new shape family
- unsupported `HAVING` forms remain explicit and fail closed
- grouped execution contracts and continuation behavior remain stable
- explain and validation surfaces stay deterministic

### Slice C - Expression-Index Eligibility Expansion (Medium, Runtime)

Expand expression-index eligibility by widening the already-shipped
`LOWER(field)` casefold family first, before adding new expression-language
families.

Preferred direction for `0.57`: one bounded additional eligibility family on
top of casefold expression keys (for example one prefix or range-oriented
family), without committing to a broader expression-language expansion in this
line.

Acceptance criteria:

- planner eligibility gates are explicit and deterministic
- runtime lookup/mutation/replay behavior is parity-tested
- unsupported expression families remain fail-closed and classified
- docs/changelog clearly state supported vs deferred expression families

### Slice D - Targeted Optimization Closure (Optional if Stable)

If an optimization closure slice is taken in `0.57`, prefer BYTES/index-only
closure work over grouped fast-path expansion.

Acceptance criteria:

- no semantic-authority boundary regression
- stable explain/metrics contracts for any new optimization route
- fallback behavior remains explicit and deterministic

## Architecture and Authority Guardrails

All `0.57` slices must preserve:

- parser as syntax authority only
- lowering/session/planner/runtime as semantic authority
- no generated-helper semantic forks
- canonical entity matching and route behavior
- fail-closed unsupported-lane/unsupported-shape behavior
- introspection lanes remain distinct from row-execution lanes; do not fold
  introspection commands into `execute_sql*` surfaces

Control-flow consolidation from late `0.56` should remain intact; do not
reintroduce parallel lane checks or duplicate rejection trees across surfaces.

## Testing and Validation Strategy

Each `0.57` slice should carry:

- parser/lowering/session/facade tests for newly supported shapes
- parity tests (SQL vs fluent intent/explain where applicable)
- fail-closed matrix tests for unsupported adjacent shapes
- canister/generated-dispatch integration coverage for new SQL surfaces

Recommended validation baseline per slice:

```bash
cargo fmt --all
cargo check --workspace --all-targets --locked
cargo clippy --workspace --all-targets --all-features -- -D warnings
cargo test -p icydb-core --locked
cargo test -p icydb --locked
```

For slices touching generated dispatch:

```bash
cargo test -p canister_demo_rpg generated_sql_dispatch_surface_is_stable -- --nocapture
cargo test -p canister_minimal generated_sql_dispatch_surface_is_stable -- --nocapture
```

## Risks and Mitigations

### Risk: SQL-surface drift between core/facade/generated helpers

Mitigation:

- keep route/lane authority centralized
- require parity tests across core/facade/generated for every new SQL family

### Risk: expression-index scope creep

Mitigation:

- ship one bounded eligibility family at a time
- keep unsupported families explicit in docs and error labels

### Risk: grouped-HAVING expansion destabilizes grouped contracts

Mitigation:

- preserve grouped planner/runtime authority
- gate new shapes behind deterministic validation contracts
- expand continuation and explain parity coverage alongside feature support

### Risk: optimization work increases control-plane complexity

Mitigation:

- treat optimization slice as optional and bounded
- require explicit no-regression checks on branch/control-plane surfaces

## Exit Criteria for 0.57

`0.57` is complete when:

1. at least one meaningful missing capability from each selected scope family
   is shipped (DTO-backed introspection addition, grouped `HAVING` bounded
   expansion, casefold-family expression-index eligibility expansion)
2. unsupported adjacent shapes remain fail closed with deterministic errors
3. core/facade/generated parity is preserved for all new SQL surfaces
4. docs/changelog clearly separate:
   - newly shipped capabilities
   - still-deferred scope
   - hardening-only follow-up work
5. Slice 0 precondition gate is explicitly closed (already landed or shipped
   in-line)

## Deferred Beyond 0.57 (Unless Re-Scoped)

- full SQL write lane (`INSERT`, `UPDATE`) broad support
- broad catalog SQL expansion beyond bounded introspection additions
- broad expression-language support for indexes
- unbounded optimization rewrites that increase control-plane multiplicity
