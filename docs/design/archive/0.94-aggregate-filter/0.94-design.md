# 0.94 Aggregate FILTER and Query-Surface Contract Lockdown

## Thesis

`0.94` should introduce SQL-standard `FILTER (WHERE ...)` support on aggregate functions **and** use that feature slice to formally lock the remaining core IcyDB query contracts that have been widened across `0.85–0.93`.

This is not just a feature add.

It is the point where IcyDB should say, explicitly and contractually:

* what a query expression is
* what each query phase is allowed to see
* what aggregate ownership is
* what normalization is allowed to do
* what cache identity must distinguish
* what execution layers may and may not reinterpret

The core mechanism is:

> add aggregate-owned pre-aggregate row-admission filters, while freezing the syntax → semantics → normalization → execution boundaries that now exist across the query stack.

`0.94` should therefore do two things together:

1. ship `FILTER (WHERE ...)` as a bounded planner-owned aggregate feature in `0.94.0`
2. codify the query-design contracts that this slice must preserve, without treating every contract paragraph as mandatory new code work in the same patch

This slice should increase expressiveness **and** reduce future architectural drift.

### Explicit framing

This document intentionally does two different jobs:

* define the **implementation contract** for the `0.94.0` `FILTER` slice
* define the **policy contracts** that `0.94.0` must preserve

That distinction is important.

The contract sections below are primarily:

> invariants that `0.94.0` must obey

not:

> a mandatory refactor list that `0.94.0` must fully perform

If an existing boundary already satisfies one of those contracts, `0.94.0` should document and preserve it rather than widen the patch with unnecessary architectural cleanup.

---

## Why 0.94 is the right contract-lock point

After the recent line of work, IcyDB already has:

* shared scalar expression ownership
* searched `CASE`
* aggregate input expressions
* unified SQL expression lowering across clauses
* predicate cleanup into validate / normalize / compile stages
* a simplified SQL cache story
* stronger structural cache identity tests

That means the engine is no longer in a “major semantic migration” phase.

It is now close enough to a stable core model that the right move is to:

> add the next small, high-value feature and use that moment to freeze the core contracts around it.

`FILTER` is ideal because it is:

* high value
* small in implementation scope
* entirely inside the current execution model
* a direct test of whether the current boundaries are actually clean

If `FILTER` can land without new seams, then the design is mature enough to formalize.

---

## Design goals

`0.94` has two equal goals.

### Goal A — Feature goal

Support:

```sql
SUM(strength) FILTER (WHERE strength > 10)
COUNT(*) FILTER (WHERE is_npc)
AVG(intelligence) FILTER (WHERE class_name = 'mage')
```

without:

* adding a new execution phase
* rewriting FILTER into CASE
* introducing aggregate-local predicate semantics outside the shared scalar expression model

### Goal B — Contract goal

Freeze the IcyDB query design so that future work must obey these boundaries:

* `SqlExpr` owns syntax only
* planner `Expr` owns semantics
* normalization owns canonical structural shaping only
* compilation owns executable form only
* aggregate filters are pre-aggregate row-admission expressions
* execution does not reinterpret SQL syntax
* cache identity distinguishes all admitted semantic differences
* unsupported shapes fail before execution

This is the real closeout value of `0.94`.

---

## Non-goals

This slice does not aim to:

* add a new execution phase
* widen the grouped runtime contract
* add window semantics
* add joins
* add subqueries
* add aggregate `DISTINCT + FILTER`
* rewrite FILTER into CASE
* create a second boolean/predicate subsystem
* change cache ownership architecture again

This is a contract-and-surface slice, not a subsystem rewrite.

# Part I — 0.94.0 Implementation Contract

## Supported SQL surface

`0.94` should support:

```sql
<aggregate>(<expr>) FILTER (WHERE <predicate>)
COUNT(*) FILTER (WHERE <predicate>)
```

Examples:

```sql
SELECT COUNT(*) FILTER (WHERE is_npc) FROM Character;
```

```sql
SELECT AVG(intelligence) FILTER (WHERE class_name = 'mage') FROM Character;
```

```sql
SELECT
  class_name,
  SUM(strength) FILTER (WHERE strength > 10) AS strong_sum
FROM Character
GROUP BY class_name;
```

---

## Core execution rule

For an aggregate with a filter:

```text
for each input row:
  evaluate filter predicate
  if filter result is TRUE:
    evaluate aggregate input (if any)
    feed result into accumulator
  else:
    skip row for that aggregate
```

### Explicit boolean rule

* only `TRUE` admits the row
* `FALSE` and `NULL` both skip the row

This must reuse the same boolean-boundary rule already used for row filtering, without inventing a second rule set.

### Boundary rule

`FILTER` must reuse the same WHERE-style boolean-admission boundary already used for row admission elsewhere in the engine.

That means:

* boolean meaning remains planner-owned
* clause-boundary collapse remains boundary-owned
* no aggregate-local boolean collapse helper may be introduced

---

## Aggregate-family support

In scope:

* `COUNT(*) FILTER (...)`
* `COUNT(expr) FILTER (...)`
* `SUM(expr) FILTER (...)`
* `AVG(expr) FILTER (...)`
* `MIN(expr) FILTER (...)`
* `MAX(expr) FILTER (...)`

### `COUNT(*) FILTER (...)`

Counts rows where filter is `TRUE`.

### `COUNT(expr) FILTER (...)`

Counts rows where:

* filter is `TRUE`
* evaluated `expr` is non-null

### `SUM / AVG / MIN / MAX FILTER (...)`

If filter is `TRUE`, evaluate the input and pass it into the existing accumulator semantics.

No accumulator should own filter logic beyond “row admitted or skipped.”

---

## Expression admissibility for FILTER predicates

The FILTER predicate must reuse the existing **pre-aggregate boolean expression family**.

Allowed:

* fields
* literals
* comparisons
* boolean composition
* admitted scalar functions
* admitted searched CASE
* nested admitted scalar expressions

Rejected:

* aggregates
* grouped/post-aggregate expressions
* subqueries
* window expressions
* unsupported scalar function families

This is important: FILTER must be a consumer of the existing expression model, not a loophole around it.

---

## Planner model

Aggregate expressions should gain an optional planner-owned filter expression.

Conceptually:

```rust
struct AggregateExpr {
    func: AggregateFunc,
    input_expr: Option<Expr>,
    filter_expr: Option<Expr>,
}
```

The exact structure should match existing aggregate ownership, but the contract is:

* `filter_expr` is planner-owned
* it is not encoded as SQL sugar after lowering
* executor compiles it through the same shared boolean evaluator boundary already used elsewhere

### Explicit storage contract

For `0.94.0`, the intended storage contract is:

* aggregate planner nodes carry `filter_expr: Option<Expr>`
* execution compiles that planner-owned expression to the existing shared executable boolean boundary
* no narrower dedicated aggregate-filter IR should be introduced in `0.94.0` unless an existing compiled aggregate boundary already requires it

---

## Parser / SQL lowering

Parser supports:

```text
AGG(expr) FILTER (WHERE sql_expr)
COUNT(*) FILTER (WHERE sql_expr)
```

Lowering must:

* parse the filter as `SqlExpr`
* lower it via the same pre-aggregate expression seam
* attach it directly to the aggregate node

### Explicit rule

`FILTER` must **not** be lowered by rewriting to CASE.

That would:

* duplicate semantics
* hide the aggregate-owned contract
* complicate cache identity and explain output

---

## Validation

Validation must enforce:

* FILTER predicate is boolean
* FILTER predicate is pre-aggregate
* FILTER predicate contains no aggregates
* FILTER only appears attached to aggregate calls
* `DISTINCT + FILTER` is rejected in `0.94.0`
* aliases inside FILTER predicates are rejected in `0.94.0`

Unsupported shapes must fail at planning/lowering time, not runtime.

---

## Executor

Executor compiles:

* aggregate input expression
* aggregate filter expression

once per prepared execution.

Per row:

* if no filter: existing behavior
* if filter present:

  * evaluate filter
  * if `TRUE`: continue
  * else: skip the row for that aggregate

No new evaluator path is introduced.

---

## Explain / fingerprint / cache identity

`0.94` must update:

* explain output
* structural fingerprinting
* cache identity

so that these remain distinct:

```sql
COUNT(*)
COUNT(*) FILTER (WHERE is_npc)
COUNT(*) FILTER (WHERE NOT is_npc)
SUM(strength) FILTER (WHERE strength > 10)
SUM(strength) FILTER (WHERE strength > 20)
```

Filtered and unfiltered aggregates must never alias structurally.

# Part II — Query-Design Contracts This Slice Must Preserve

This is the main new part of the doc.

These are the invariants `0.94.0` must preserve while adding `FILTER`.

They are not a requirement that `0.94.0` perform a full subsystem-wide rewrite to “re-prove” every contract in code if the current boundary already satisfies it.

## Contract 1 — Syntax ownership

### Rule

`SqlExpr` owns SQL syntax shape only.

It may represent:

* parsed SQL operators
* aggregate/filter syntax
* aliases
* clause-local surface forms

It must not own final engine semantics.

### Meaning

* parser may build `SqlExpr`
* parser may normalize surface shape where explicitly allowed
* parser must not encode runtime meaning or executable contracts

---

## Contract 2 — Semantic ownership

### Rule

Planner `Expr` owns query semantics.

That includes:

* scalar meaning
* boolean meaning
* CASE semantics
* aggregate input semantics
* FILTER predicate semantics
* post-aggregate expression semantics

There must not be a second semantic owner for the same query meaning.

### Meaning

No later layer should reinterpret:

* CASE meaning
* boolean truth rules
* aggregate filter semantics
* alias semantics

Those must already be captured by planner-owned form.

---

## Contract 3 — Phase ownership

### Rule

Every expression is admitted only in one of the existing phase contracts.

#### Pre-aggregate phase

May see:

* row fields
* literals
* pre-aggregate scalar expressions

Used in:

* WHERE
* aggregate inputs
* aggregate FILTER predicates

#### Post-aggregate phase

May see:

* grouped keys
* aggregate outputs
* admitted post-aggregate scalar expressions

Used in:

* grouped projection
* grouped HAVING
* grouped ORDER BY

### Meaning

No feature may cross phases by accident.

FILTER is explicitly pre-aggregate.

HAVING remains post-aggregate.

CASE does not relax phase boundaries.

---

## Contract 4 — Normalization ownership

### Rule

Normalization owns structural shaping only.

It may:

* canonicalize commutative or set-shaped structures where semantics permit
* collapse no-op wrappers
* normalize casefold wrappers
* normalize boolean shape for WHERE compilation
* canonicalize value-set membership lists

It must not invent new semantics.

### Meaning

Normalization may reject unsupported shapes only where that is the explicit contract, but it must not become a shadow semantic layer.

For WHERE specifically:

* validation rejects unsupported shapes
* normalization shapes valid boolean expressions
* compilation consumes normalized form

This must remain the model.

---

## Contract 5 — Compilation ownership

### Rule

Compilation owns executable form only.

It may:

* compile normalized planner expressions to execution structures
* assume validated and normalized input
* be total over its accepted input

It must not:

* reject shapes
* reinterpret SQL syntax
* perform semantic normalization
* invent boolean or aggregate rules

### Meaning

`Predicate` is compiled form, not semantic owner.

Aggregate execution is compiled form, not semantic owner.

FILTER compilation must follow the same rule.

---

## Contract 6 — Aggregate ownership

### Rule

Aggregate accumulators own aggregate semantics only.

They own:

* sum/count/min/max/avg accumulation behavior
* null-handling behavior already defined for each aggregate family

They do not own:

* SQL syntax
* filter parsing
* filter predicate semantics
* CASE semantics
* alias semantics

### Meaning

FILTER changes row admission into the aggregate, not aggregate meaning itself.

---

## Contract 7 — Boolean-boundary ownership

### Rule

Three-valued meaning is preserved in expressions until a clause boundary explicitly collapses it.

For row-filter-style boundaries:

* only `TRUE` passes
* `FALSE` and `NULL` both do not pass

This rule is applied at the boundary, not inside arbitrary subexpressions.

### Meaning

* WHERE keeps only TRUE
* FILTER admits only TRUE
* internal scalar expressions retain three-valued meaning until that point

This must be one explicit boundary rule, not duplicated ad hoc.

---

## Contract 8 — Alias ownership

### Rule

Projection aliases are SQL-surface sugar resolved during SQL normalization/lowering, not planner/runtime semantics.

If HAVING or ORDER BY refer to aliases, those must be rewritten before grouped planner/runtime semantics see them.

### Meaning

Planner/runtime must not gain clause-specific alias interpretation.

This was the correct fix for HAVING alias behavior and should remain the rule.

### `0.94.0` decision

Aliases inside `FILTER` predicates are out of scope for `0.94.0` and must fail closed.

That keeps aggregate filters clearly pre-aggregate and field/expression-owned instead of widening SQL alias normalization into one more clause-specific path.

---

## Contract 9 — Cache identity ownership

### Rule

Structural cache identity must distinguish all admitted semantic differences that can affect execution meaning.

That includes:

* CASE structure
* aggregate input expressions
* FILTER predicates
* DISTINCT vs non-DISTINCT
* grouped ORDER direction
* alias-rewritten semantic structure
* normalized membership-set semantics

It must not rely on weaker or coarser plan fingerprints when correctness requires distinction.

### Meaning

The cache key contract is structural semantic identity, not textual coincidence and not overly coarse planner shape.

---

## Contract 10 — Runtime safety ownership

### Rule

Unsupported query families must fail before execution.

Parser/lowering/planning may reject.

Runtime should not discover new unsupported SQL families late.

### Meaning

By the time execution starts:

* syntax is known
* semantics are known
* phase validity is known
* normalization is complete
* compilation is total over accepted input

This is one of the key audit-friendly invariants 0.94 should freeze.

# Part III — Explicit Test and Rejection Matrix

`0.94.0` should not ship on prose contracts alone. The following tests are the minimum intended ship gates for this slice.

## Positive semantic tests

At minimum:

* `COUNT(*) FILTER (WHERE is_npc)` on the global aggregate lane
* `SUM(strength) FILTER (WHERE strength > 10)` on the global aggregate lane
* grouped filtered aggregate projection such as:
  * `SUM(strength) FILTER (WHERE strength > 10)` grouped by `class_name`
* grouped filtered aggregate plus grouped `HAVING`
* grouped filtered aggregate plus grouped `ORDER BY`

## Negative cache-identity tests

At minimum:

* filtered vs unfiltered same aggregate
* same predicate field, different threshold
* same threshold, different operator
* grouped filtered aggregate vs grouped unfiltered aggregate
* same aggregate/input, different filter predicate structure

## Parser and lowering rejection matrix

At minimum:

* aggregate inside FILTER predicate rejected:
  * `SUM(x) FILTER (WHERE COUNT(*) > 1)`
  * `SUM(x) FILTER (WHERE AVG(y) > 1)`
* `COUNT(DISTINCT x) FILTER (...)` rejected in `0.94.0`
* aliases inside FILTER rejected in `0.94.0`
* window `OVER (...)` still rejected
* subqueries inside FILTER still rejected

## Explain and observability follow-through

At minimum:

* explain output shows filtered aggregate shape
* structural cache identity distinguishes filtered aggregate shapes
* filtered and unfiltered aggregates never alias

# Part IV — What 0.94 should explicitly lock for future work

Once 0.94 lands, future query features must extend one of these existing contracts rather than invent a new one.

## Allowed future extension pattern

A future feature may extend:

* `SqlExpr` syntax
* planner `Expr`
* pre-aggregate phase
* post-aggregate phase
* normalization rules
* compiled execution forms

But it must state explicitly which layer it extends.

## Disallowed future pattern

A future feature must not:

* create a second semantic interpretation lane
* add clause-specific behavior when phase-based behavior would suffice
* hide semantic rewriting inside compilation
* add a private predicate/aggregate/CASE rule set

This is the main reason to write this doc.

# Acceptance criteria

## Functional

* FILTER works for supported aggregate families
* FILTER predicates behave correctly with TRUE / FALSE / NULL
* FILTER composes with CASE, aggregate inputs, HAVING, ORDER BY, and grouping
* aliases inside FILTER predicates fail closed in `0.94.0`

## Architectural

* FILTER is planner-owned, not CASE-rewritten
* the existing WHERE-style boolean boundary is reused for row admission
* no new execution phase exists
* no second aggregate predicate subsystem exists
* all contracts above are reflected in code ownership

## Observability

* explain output shows FILTER shape
* structural cache identity distinguishes filtered aggregate shapes
* no filtered/unfiltered aliasing exists

## Safety

* unsupported FILTER shapes fail before execution
* predicate/compile boundaries remain intact
* boolean NULL collapse remains boundary-owned and singular

# Out of scope

Explicitly not part of `0.94`:

* `DISTINCT + FILTER`
* windowed FILTER
* subqueries inside FILTER
* joins
* subqueries generally
* new execution phases
* new cache architecture

---

# Follow-on after 0.94

After 0.94, the system should be in a good position to:

* reduce the shared SQL compile baseline more aggressively
* simplify remaining clause-specific residuals such as HAVING if desired
* optimize with stronger assumptions because the semantic surface is now contractually frozen

That is a key reason to make 0.94 a contract slice as much as a feature slice.

## Thesis sentence

> **0.94 introduces aggregate FILTER support and uses that feature slice to formally lock IcyDB’s query architecture: SQL syntax in `SqlExpr`, semantics in planner `Expr`, structural shaping in normalization, executable form in compilation, and pre-aggregate row admission for filtered aggregates, so future query work extends explicit contracts rather than creating new semantic seams.**
