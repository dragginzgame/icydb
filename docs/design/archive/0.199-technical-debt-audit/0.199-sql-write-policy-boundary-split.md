# 0.199 SQL Write Policy Boundary Split

Status: structural cleanup with focused validation. This slice preserves SQL
UPDATE/DELETE policy behavior, generated/public write admission, returning
bounds, SQL execution, cache behavior, cursor formats, persisted formats, and
public API shape.

## Question

Can the shared SQL write policy owner stop mixing public DTOs, execution-bound
derivation, and parser-shape classification for UPDATE and DELETE policy gates?

## Evidence

Before this slice, `crates/icydb-core/src/db/session/sql/write_policy.rs`
owned:

- public/hidden write proof DTOs for `WHERE`, `ORDER BY`, `RETURNING`, and
  execution bounds;
- shared validated-plan core wrappers and primary-key/bounded plan proofs;
- generated/public exposure and admission-lane model types;
- bounded write policy rejection and staged-row/returning bound derivation;
- parser-expression classification for primary-key equality, canonical
  primary-key order, aliases, and returning shape;
- unit tests for both bounds and parser-shape behavior.

Those responsibilities are now split while preserving the same exported
symbols used by UPDATE, DELETE, execution, tests, and the public hidden SQL DTO
surface:

- `write_policy/model.rs` owns write proof DTOs, plan-core/proof wrappers, and
  admission/exposure model types.
- `write_policy/bounds.rs` owns staged-row and `RETURNING` bound derivation plus
  bounds-focused tests.
- `write_policy/shape.rs` owns parsed SQL expression/order/returning shape
  classification plus shape-focused tests.
- `write_policy.rs` is now a narrow module index that preserves the previous
  re-export surface.

## Non-Goals

- No SQL UPDATE or DELETE policy behavior change.
- No generated/public write admission change.
- No SQL write execution, returning projection, cache, cursor, storage, or
  persisted-format change.
- No public DTO rename or visibility hard cut.

## Complexity Delta

- Public API complexity: neutral.
- Runtime complexity: neutral.
- Review complexity: lower. Model, bounds, and parser-shape policy logic can
  now be reviewed independently.
- File shape:
  - `write_policy.rs`: 1,022 lines before, 25-line module index after.
  - `write_policy/model.rs`: 422 lines.
  - `write_policy/bounds.rs`: 226 lines.
  - `write_policy/shape.rs`: 352 lines.

The total line count is approximately neutral because tests and module headers
move with their owners.

## Validation

- `cargo check -p icydb-core --all-features`

## Residual Risk

Low. This is a module ownership split. Remaining risk is all-target visibility
drift and statement-family behavior drift, covered by core clippy plus focused
SQL update/delete/write tests before release.
