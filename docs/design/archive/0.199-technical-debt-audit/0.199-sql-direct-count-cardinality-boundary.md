# 0.199 SQL Direct Count Cardinality Boundary

Status: structural cleanup with focused validation. This slice preserves SQL
`COUNT(*)` results, direct cardinality fallback behavior, SQL cache behavior,
query-plan cache identity, read admission, cursor formats, persisted formats,
and public API shape.

## Question

Can SQL direct-count cardinality planning stop living in the shared session
query-plan cache module?

## Evidence

Before this slice, `crates/icydb-core/src/db/session/query/cache.rs` owned two
SQL-only helpers:

- exact count-cardinality prefix spec construction from accepted authority,
  visible indexes, and schema info;
- fallback lowering from a planned access query into executor cardinality
  prefix specs.

Those helpers were only consumed by
`crates/icydb-core/src/db/session/sql/execute/direct_count.rs`. Keeping them in
the query-cache module forced the cache owner to import SQL/count-cardinality
route-planning internals that are not part of cache identity or cache behavior.

The helpers now live beside the direct SQL count target construction that
consumes them:

- `sql/execute/direct_count.rs` owns SQL direct-count target construction,
  cardinality prefix-spec derivation, direct execution, measured execution, and
  fallback result shaping.
- `query/cache.rs` remains focused on shared query-plan cache lookup/insert,
  planner-visible index projection, and accepted-schema/visibility handoff.

## Non-Goals

- No SQL aggregate behavior change.
- No direct-count optimization change.
- No query-plan cache identity, method version, or hit/miss behavior change.
- No read admission, cursor, storage, or persisted-format change.

## Complexity Delta

- Public API complexity: neutral.
- Runtime complexity: neutral.
- Review complexity: lower. SQL direct-count route-planning helpers now live
  with the SQL direct-count execution owner instead of the shared query cache.

## Validation

- `cargo fmt --all`

## Residual Risk

Low. This is a helper ownership move. Remaining risk is SQL feature-target
import drift, covered by SQL-enabled cargo checks and SQL aggregate tests before
release.
