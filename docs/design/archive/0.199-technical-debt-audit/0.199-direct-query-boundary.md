# 0.199.0 Direct Query Boundary Trace

Status: 0.199.0 audit and cleanup artifact, updated by 0.199.10.

Verdict: direct `Query<E>` execution should not appear as a normal public
endpoint recipe. The 0.199.0 cleanup hides fluent `.query()` extraction methods
from public docs and updates the public facade reference to teach semantic
terminals first. The 0.199.1 cleanup adds hidden fluent attribution terminals
so the SQL perf audit canister no longer needs raw `Query<E>` extraction for
fluent scenario measurement. The 0.199.10 cleanup removes the unused hidden
facade `DbSession::execute_query(&Query<E>)` row-execution surface; hidden
facade fluent `.query()` extraction methods were removed in the same slice.
Hidden `Query<E>` uses remain diagnostic/planning-oriented.

## What Passed

- Hidden facade direct row execution has been removed after a non-doc code search
  found no repository callers.
- Public facade `.query()` extraction has been removed from load,
  partial-window, delete, and delete-returning wrappers after a non-doc code
  search found no repository callers.
- Public docs now teach `load::<E>()` semantic terminals instead of direct
  `Query<E>` construction.
- The `Query<E>` re-export and remaining diagnostic direct-query methods are
  doc-hidden.
- Existing perf harness fluent call sites now use hidden fluent attribution
  terminals instead of raw `Query<E>` extraction.

## Finding

### 199-DQ-001: Direct Query Is Internal, With Fluent Attribution On The Fluent Boundary

Category: public API hygiene / generated boundary

Severity: Low after 0.199.10 cleanup

Priority: P3 after 0.199.10 cleanup

Confidence: High

Evidence:

- `crates/icydb/src/db/query/mod.rs` re-exports `Query` as `#[doc(hidden)]`.
- `crates/icydb/src/db/session/mod.rs` removed the unused hidden
  `execute_query(...)` row-execution method in 0.199.10.
- `crates/icydb/src/db/session/mod.rs` still marks `trace_query(...)` and
  diagnostics attribution direct-query helpers as hidden.
- Before 0.199.1, `canisters/audit/sql_perf/src/lib.rs` intentionally used
  `.query()` and direct attribution helpers for performance measurement.
- After 0.199.1, fluent perf scenarios use hidden fluent
  `execute_with_attribution()` terminals.
- Before the 0.199 cleanup line, public/core fluent `.query()` methods were not
  hidden. The public facade extraction methods were removed in 0.199.10; core
  extraction remains internal/test-oriented.

Recommended policy:

- Keep remaining direct query diagnostics callable but doc-hidden until/unless
  generated infrastructure can avoid them entirely.
- Do not add compatibility aliases or a public policy-builder layer around
  direct queries.
- Do not teach direct query construction in README, public facade docs, or
  read-intent docs.
- If a future generated path needs direct query execution, add a narrow
  generated/internal design first instead of restoring the broad facade method.
- Prefer hidden fluent attribution terminals for fluent perf/diagnostic code
  when the caller starts from a fluent wrapper.
- Do not re-add public facade `.query()` extraction just to reach hidden direct
  query diagnostics.
