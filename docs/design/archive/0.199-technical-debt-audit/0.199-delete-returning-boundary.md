# 0.199 Delete Returning Boundary Cleanup

Status: small code cleanup with focused validation. This slice preserves public
delete APIs, mutation semantics, read-admission behavior, diagnostic DTOs,
cursor formats, persisted formats, and generated SQL authority.

## Question

Can fluent delete returning stop using the broad core row materialization
handoff and instead use a delete-owned row materialization boundary?

## Evidence

`crates/icydb-core/src/db/query/fluent/delete.rs` already had distinct fluent
delete terminal intent:

- `execute()` returns affected-row count;
- `execute_rows()` materializes deleted rows for the row-returning surface.

The count terminal already delegated to
`DbSession::execute_delete_count(...)`, which fails closed when called with a
non-delete query. The row-returning terminal delegated to the broader
core session row helper, which also supported load-row execution.
That was correct for normal fluent construction, but it left delete returning
less explicit than delete count.

## Change

Added one session-owned helper:

- `execute_delete_rows(...)`

The helper:

1. rejects non-delete query mode before planning;
2. prepares through the shared cached plan boundary;
3. executes through the shared prepared-plan path with row output;
4. accepts only the delete-row execution outcome.

Fluent delete returning now uses this helper instead of the broad core row
materialization helper.

The public facade returning wrapper also centralizes the returning-selection
projection handoff: `DeleteReturningSelection` now owns conversion into the
optional field slice consumed by the shared row-projection renderer, so the
facade calls that renderer once regardless of `returning_all()` versus
`returning(...)`.

## Result

Delete count and delete row-returning terminals now both use delete-specific
session helpers with explicit mode guards. This narrows the direct-query
boundary without changing public delete behavior or executor strategy. Returning
projection selection is also single-owner at the public facade layer.

## Non-Goals

- No public delete API rename.
- No SQL write policy change.
- No mutation atomicity or persistence change.
- No returning projection redesign.
- No executor route rewrite.
