## Title

**0.71 — Typed Scalar Preparation Note**

---

## Status

Implemented

This note records the first `0.71` coding slice that has now landed:

* flatten typed-scalar aggregate setup
* derive runtime descriptor and `EXPLAIN` surfaces from prepared strategy
* keep planner visibility and aggregate correctness rules unchanged

The typed SQL scalar lane now does this in code, and the same prepare-once /
project-everywhere rule has since been extended into the fluent aggregate
domains.

---

## Original Duplicated Typed-Scalar Setup Sites

The original typed-scalar surface was spread across several owners that each
prepare or reinterpret aggregate behavior differently.

### 1. Typed SQL Aggregate Surface Dispatch

In [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/aggregate.rs),
`DbSession::execute_sql_aggregate(...)` fans typed SQL aggregate requests into
three different boundary families:

* `execute_scalar_terminal_request(...)`
  - `COUNT(*)`
* `execute_scalar_projection_boundary(...)`
  - `COUNT(field)`
* `execute_numeric_field_boundary(...)`
  - `SUM(field)`
  - `AVG(field)`
* `execute_scalar_extrema_value_boundary(...)`
  - `MIN(field)`
  - `MAX(field)`

That fanout is correct at the domain level, but it meant the typed-scalar lane
did not yet have one explicit prepared strategy shape of its own.

### 2. Existing-Rows / Scalar Terminal Preparation

In [terminals.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/terminals.rs),
`execute_scalar_terminal_request(...)` constructs `PreparedScalarTerminalBoundary`
directly from the public request and then chooses:

* `KernelAggregate`
* `CountPrimaryKeyCardinality`
* `ExistingRows { direction }`

This was already much cleaner than the older split, but it was still one
boundary-specific vocabulary rather than one prepared typed-scalar strategy
that the public typed SQL lane could project.

### 3. Field Projection / Extrema Preparation

In [projection/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/projection/mod.rs):

* `execute_scalar_extrema_value_boundary(...)`
* `prepare_scalar_projection_boundary(...)`

both resolve field-slot and prepared aggregate inputs again for typed field
terminals.

That is legitimate domain work, but it meant the typed-scalar lane still
reached the field-extrema and projection families through a second owner-local
setup surface.

### 4. Numeric Preparation

In [numeric/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/numeric/mod.rs),
typed numeric requests still resolve their own prepared numeric boundary and
payload shape:

* `prepare_scalar_numeric_boundary(...)`
* `resolve_prepared_scalar_numeric_boundary(...)`
* `prepare_scalar_numeric_payload(...)`

This is the correct numeric-domain owner, but it was another branch point the
typed-scalar entry lane had to know about directly.

### 5. Aggregate Descriptor / Explain Reconstruction

In [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/aggregate.rs),
`assemble_aggregate_terminal_execution_descriptor_with_model(...)` currently:

* rebuilds `ExecutionPreparation`
* rebuilds aggregate route planning
* derives ordering/execution semantics for descriptor output

That meant aggregate execution descriptor assembly was still a second
behavioral classification path rather than a projection of prepared aggregate
strategy.

### 6. Aggregate Runtime Descriptor State

In [mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/mod.rs),
`prepare_aggregate_execution_state_from_prepared(...)` constructs
`AggregateExecutionDescriptor` from prepared streaming inputs and one
`AggregateExpr`.

That was already closer to the right shape, but the descriptor was still tied
to aggregate runtime orchestration rather than to one typed-scalar prepared
strategy that descriptor and explain surfaces could both project.

## What Landed

The first `0.71` slice now does what this note proposed:

* typed SQL scalar aggregate lowering prepares one explicit scalar strategy in
  [mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs)
* typed SQL aggregate runtime dispatch consumes the prepared runtime
  descriptor in
  [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/aggregate.rs)
* typed SQL aggregate `EXPLAIN EXECUTION` consumes the prepared strategy in
  [explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/explain.rs)
  and
  [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/aggregate.rs)

That rule has also been pushed outward since the first slice:

* fluent existing-row, scalar id/extrema, numeric, order-sensitive, and
  projection/distinct aggregates now prepare family-local strategies before
  runtime dispatch in
  [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/builder/aggregate.rs)
  and
  [terminals.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fluent/load/terminals.rs)
* fluent aggregate explain now projects through one shared prepared explain
  contract for explain-visible domains in
  [query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs)
  and
  [query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs)

## Target Prepared Strategy Shape

The first `0.71` slice introduced one explicit typed-scalar strategy shape
that answers the behavioral questions once.

Conceptually:

```rust
enum PreparedTypedScalarAggregateDomain {
    ExistingRows,
    ProjectionField,
    NumericField,
    ScalarExtremaValue,
}

enum PreparedTypedScalarOrderingRequirement {
    None,
    ResponseOrder,
    FieldOrder,
}

enum PreparedTypedScalarRowSource {
    ExistingRows,
    ProjectedField,
    NumericField,
    ExtremalWinnerField,
}

enum PreparedTypedScalarEmptySetBehavior {
    Zero,
    Null,
}

enum PreparedTypedScalarDescriptorShape {
    CountRows,
    CountField,
    SumField,
    AvgField,
    MinField,
    MaxField,
}

struct PreparedTypedScalarAggregateStrategy {
    domain: PreparedTypedScalarAggregateDomain,
    ordering_requirement: PreparedTypedScalarOrderingRequirement,
    row_source: PreparedTypedScalarRowSource,
    empty_set_behavior: PreparedTypedScalarEmptySetBehavior,
    descriptor_shape: PreparedTypedScalarDescriptorShape,
}
```

The exact type names did change in implementation.
The important rule did not: typed-scalar preparation answers these questions
once, up front, and downstream surfaces only project them.

### Required Properties

The prepared typed-scalar strategy needed to be:

* domain-complete for the typed SQL scalar lane
* non-overlapping
* execution-complete
* descriptor-complete
* `EXPLAIN`-projectable without second guessing

## Exact Surfaces That Must Become Projections

The first slice needed to make these surfaces projections of prepared
strategy, not parallel re-derivations.

### 1. Runtime Execution

Typed SQL aggregate execution in
[aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/aggregate.rs)
should lower once to a prepared typed-scalar strategy and then dispatch on
that prepared strategy, not on a second ad hoc `match` over surface terminals.

### 2. Aggregate Runtime Descriptor

Aggregate execution-state construction in
[mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/mod.rs)
should consume the prepared typed-scalar strategy as its behavioral source.

It should not need a second behavior tree to restate:

* existing-row versus field-extrema versus numeric
* ordering requirement
* empty-set behavior

### 3. Aggregate `EXPLAIN`

Descriptor assembly in
[aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/aggregate.rs)
should render from prepared strategy output instead of rebuilding route-owned
meaning from:

* fresh `ExecutionPreparation`
* fresh aggregate route derivation
* fresh ordering/execution classification

### 4. Runtime Descriptor / Explain Parity

The same prepared strategy should be the source for both:

* runtime descriptor state
* explain descriptor state

That is the critical anti-drift rule for the first `0.71` slice.

---

## Out Of Scope For This Slice

This first slice did not:

* touch planner-visible index policy
* touch `Ready` visibility rules
* widen aggregate eligibility
* introduce a new aggregate correctness model
* unify numeric, projection, and existing-row domains into one runtime

It is a preparation-and-projection cleanup only.

---

## Immediate Keep Rule

The implementation rule for the first `0.71` patch was:

> prepare typed-scalar aggregate behavior once, then project execution,
> descriptor, and explain from that prepared strategy.

If a helper needs to rediscover aggregate behavior from plan shape, route
symptoms, or public terminal kind after preparation already happened, that is
the wrong direction for this slice.
