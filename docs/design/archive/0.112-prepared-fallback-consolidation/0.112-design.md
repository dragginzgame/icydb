# 0.112 Prepared Fallback Consolidation

## Thesis

`0.112` should consolidate prepared fallback execution so
[`/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs`](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs)
stops acting like a second semantic engine and becomes a thin prepared
execution boundary over planner-owned semantic outputs.

This is not a prepared SQL widening slice.

It is a semantic-ownership cleanup slice:

* lowering owns expression structure
* planner owns semantic typing and canonical predicate shape
* prepared execution owns parameter binding, value substitution, and execution
  wiring

The audit result that motivates this is simple:

* prepared SQL is still only `Partial`
* the clearest and most contained seam is `prepare.rs`
* the risk is not raw-string reparsing
* the risk is duplicated semantic-adjacent reasoning inside the prepared lane

---

## Problem

The current prepared fallback lane still duplicates too much logic that is
semantically adjacent to lowering and planning.

The main owner-drift pressure sits in:

* [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs)

Today, the prepared flow still tends to look like:

* parse
* infer
* traverse
* bind
* execute

That is the wrong ownership shape.

Prepared execution should consume planner-owned semantic outputs, not re-derive
them.

The architecture target is:

* lowering:
  * parse
  * produce semantic model
* planner:
  * produce typed and canonical structure
* prepare:
  * bind parameters
  * execute

If `prepare.rs` keeps owning type inference, expression-tree reasoning, or
predicate-like interpretation, prepared and non-prepared paths will keep paying
an unnecessary drift tax.

---

## Goal

Prepared execution should consume planner-owned semantic outputs instead of
re-deriving them locally.

Concretely:

* `prepare.rs` should stop making semantic decisions that lowering or planning
  already own
* prepared fallback should reuse planner-owned coarse type truth
* parameter binding should preserve canonical semantic form rather than
  reshaping or re-interpreting it

---

## Core Invariant

Prepared execution must not introduce or own any semantic truth that is not
already produced by lowering or planner stages.

If a semantic decision exists in both the planner path and `prepare.rs`, the
planner must be the single authority.

This is the binary review rule for the slice:

* allowed: prepared execution consumes planner-owned semantic outputs
* not allowed: prepared execution re-derives semantic meaning locally

---

## Ownership Boundary

`0.112` should make this split explicit and visible in the codebase.

### Lowering / Planner Owns

* expression structure
* semantic typing
* predicate shape
* canonical semantic form

### Prepared Layer Owns

* parameter binding
* value substitution
* execution wiring
* compatibility checks that are strictly about binding contracts, not semantic
  interpretation

Prepared-layer checks must be limited to validating that runtime values conform
to an already-defined parameter contract.

They must not derive or modify:

* type families
* predicate semantics
* expression meaning

It must not:

* re-infer the semantic meaning of an expression tree
* act like a second planner-owned type engine
* quietly reinterpret predicate or compare-family meaning

---

## Concrete Actions

### 1. Identify duplicated reasoning

Audit `prepare.rs` for logic that duplicates semantic-adjacent reasoning,
especially:

* parameter type inference
* traversal of expression trees
* predicate-like logic
* compare-family reconciliation
* CASE-family reconciliation
* dynamic function-family inference

For each helper, classify it as one of:

* prepared-boundary policy
* planner-owned semantic truth
* mixed logic that must be split

Anything that overlaps with these existing owners should be treated as suspect:

* lowering
* semantic model
* planner typing

### 2. Define the ownership boundary in code

Refactor the prepared flow so the semantic boundary is obvious:

* lowering/planner produce the structure and type family answers
* prepared execution consumes those answers
* any remaining prepared-layer checks are about binding compatibility, not
  semantic interpretation

This should be visible from function shape and module boundaries, not just from
comments.

### 3. Refactor the flow

Move from the current effective shape:

```text
prepare.rs:
  parse -> infer -> traverse -> bind -> execute
```

To the target shape:

```text
lowering:
  parse -> semantic model

planner:
  typed + canonical structure

prepare:
  bind parameters -> execute

Constraint:
* prepare must not introduce new semantic transformations between binding and
  execution
```

Prepared execution may traverse planner-owned structures to attach runtime
values.

This traversal must be structural only and must not alter or reinterpret
semantic meaning.

### 4. Add guard tests

Required.

The slice is not complete without tests that enforce:

1. Prepared and non-prepared queries with equivalent semantics produce
   identical canonical plans
2. Parameter binding does not change canonical identity
3. No semantic normalization occurs in the prepared layer
4. Planner-owned typing is reused, not recomputed, by prepared execution

The best tests here are the ones that fail if someone quietly reintroduces
semantic reasoning into `prepare.rs`.

---

## Recommended Work Order

### 1. Map `prepare.rs`

Inventory the helpers that still do:

* family inference
* expression traversal
* predicate-like interpretation
* semantic compatibility decisions

Mark which ones should survive as binding policy and which ones belong back
under planner-owned type/semantic helpers.

### 2. Collapse duplicated reasoning

Push semantic truth behind planner-owned helpers wherever possible.

The key target is not "zero code" in `prepare.rs`.

The key target is:

* no semantic truth lives there independently

### 3. Make the prepared path visibly thin

After the refactor, a reader should be able to tell that `prepare.rs` is
primarily:

* binding
* substitution
* execution wiring

Not:

* semantic typing
* expression meaning
* predicate ownership

### 4. Freeze the boundary with tests

Treat tests as the contract that keeps the seam from regressing.

This slice is not complete until the tests prove the ownership line.

---

## Exit Condition

`0.112` is done when:

* prepared SQL no longer depends on `prepare.rs`-local semantic reasoning
* prepared execution consumes planner-owned semantic outputs without
  reinterpretation
* `prepare.rs` is reduced to binding, substitution, and execution wiring
* prepared and non-prepared equivalent queries produce identical canonical
  plans

The supporting-row classification for prepared SQL should move from `Partial`
to `Bounded` or `Complete`.

---

## Non-Goals

`0.112` should not:

* widen prepared template admission
* widen prepared fallback admission beyond the already-shipped expression
  surface
* redesign prepared execution architecture from scratch
* change planner route/ranking behavior
* reopen grouped semantic boundaries
* turn this into a broad completeness-model or audit-framework slice

This work is about semantic ownership and consolidation, not feature growth.

---

## Summary

`0.112` should close the clearest prepared-SQL semantic seam:

* planner owns semantic outputs
* prepared execution consumes them
* parameter binding preserves canonical form
* `prepare.rs` becomes thin enough that it no longer looks like a second
  semantic engine
