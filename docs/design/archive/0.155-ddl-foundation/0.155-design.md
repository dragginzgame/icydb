# 0.155 DDL Foundation

## Purpose

0.154 made one accepted-schema mutation path developer-supported: adding one
non-unique field-path secondary index for one entity through startup
reconciliation, accepted row contracts, physical index rebuild, runtime
invalidation, final physical-store revalidation, and accepted snapshot
publication.

0.155 should add the foundation for SQL DDL as a frontend over that existing
schema mutation path. The goal is not broad migration syntax. The goal is to
let a developer express the already-supported index-add operation through SQL,
bind it against accepted catalog authority, lower it into the schema mutation
request/planning layer, and execute only when it resolves to the exact supported
0.154 path.

The closeout target for 0.155 is:

`CREATE INDEX name ON entity (field_path);` can run for one non-unique
field-path secondary index add, using accepted catalog snapshots as authority
and the existing schema-owned runner/publication path. Every other DDL shape
fails closed before physical work.

## Non-Goals

0.155 should not add:

- `ALTER TABLE`
- `DROP INDEX`
- `CREATE UNIQUE INDEX`
- expression indexes
- multi-column or mixed key indexes
- multiple DDL statements in one mutation batch
- online or concurrent index build controls
- arbitrary type changes, nullability changes, slot remaps, or row rewrites
- generated `EntityModel` or `IndexModel` reconstruction as DDL authority
- a second schema publication path outside the accepted mutation runner
- generalized migration scripts

The 0.155 parser may reserve obvious future keywords, but unsupported syntax
must not imply partially wired semantics.

## Hard Invariants

- Accepted persisted schema snapshots remain the catalog authority.
- SQL DDL is a frontend onto schema mutation requests, not the source of schema
  mutation semantics.
- A parsed DDL statement must bind against the accepted snapshot before it can
  lower into a mutation request.
- DDL lowering must produce the same mutation request shape that generated
  proposal reconciliation would use for the supported path.
- DDL execution must go through `SchemaMutationRunnerInput`, accepted row
  contracts, runtime invalidation, physical-store validation, and accepted
  snapshot publication.
- Unsupported DDL fails before physical work, schema-store writes, runtime epoch
  changes, or visible `SchemaInfo` changes.
- Generated model/index metadata is allowed only for proposal, reconciliation,
  model-only convenience, and tests.
- Before `1.0.0`, DDL AST/lowering/internal command formats hard-cut to the
  latest version.

## Supported Surface

0.155 should support exactly one DDL statement:

```sql
CREATE INDEX index_name ON entity_name (field_path);
```

where:

- `index_name` is a new non-empty secondary index name for the entity
- `entity_name` resolves to exactly one accepted entity catalog snapshot
- `field_path` resolves to one accepted field-path key
- every path segment is accepted catalog metadata, not generated metadata
- the key is non-unique
- the target entity is the only entity affected
- the resulting accepted snapshot differs only by the new field-path index
- the resulting mutation plan is the single supported non-unique field-path
  index-add path from 0.154

Everything else must fail closed. In particular:

```sql
CREATE UNIQUE INDEX ...
CREATE INDEX ... ON entity (field_a, field_b);
CREATE INDEX ... ON entity (LOWER(field));
CREATE INDEX ... WHERE ...
DROP INDEX ...
ALTER TABLE ...
```

may be parsed later, but 0.155 should reject them before mutation planning or
physical work.

## Layering

The DDL stack should be deliberately thin:

1. Parser produces unresolved SQL DDL intent.
2. Binder resolves entity, field path, and candidate index identity against the
   accepted schema snapshot.
3. Lowerer builds a catalog-native `SchemaMutationRequest`.
4. Mutation planner produces a `MutationPlan` and execution plan.
5. Runner admission proves the plan is the supported field-path index-add path.
6. Existing startup/runtime schema mutation execution performs physical work and
   accepted snapshot publication.
7. Developer diagnostics report parse, bind, plan, runner, and publication
   status without matching error strings.

The parser must not construct `PersistedSchemaSnapshot`,
`PersistedIndexSnapshot`, `MutationPlan`, or runner inputs directly.

## Parser Contract

The first parser slice should add a DDL AST that is separate from SELECT,
INSERT, UPDATE, DELETE, DESCRIBE, SHOW, and EXPLAIN query/update command shapes.

The AST can be narrow:

```rust
SqlDdlStatement::CreateIndex {
    name,
    entity,
    key,
    uniqueness,
}
```

For 0.155:

- `uniqueness` must be non-unique
- `key` must be one field path
- parser output remains unresolved strings/path segments
- parser errors should identify unsupported DDL syntax as unsupported SQL DDL,
  not as query-lane SQL
- statement dispatch should prevent DDL from accidentally entering the read
  query lane

The parser can accept only the exact supported grammar. It does not need a broad
SQL dialect placeholder.

## Binder Contract

The DDL binder owns accepted catalog resolution:

- resolve entity by accepted entity name/path
- reject ambiguous or missing entity references
- resolve field-path segments against accepted field and nested-leaf metadata
- reject paths that are not orderable/indexable under accepted key contracts
- reject fields that depend on generated-only metadata
- reject duplicate index names and duplicate target key/store identity
- allocate or derive the candidate accepted index ordinal through schema-owned
  mutation rules, not parser logic

The binder output should be a typed, accepted-catalog-bound DDL request. It
should carry enough source location or statement text context for developer
diagnostics, but it must not carry generated model metadata.

## Lowering Contract

DDL lowering should produce the same schema mutation request vocabulary as
proposal reconciliation:

```text
SqlDdlCreateIndexIntent
  -> BoundSchemaDdlRequest
  -> SchemaMutationRequest
  -> MutationPlan
  -> SchemaMutationExecutionPlan
```

The lowerer must prove:

- accepted-before snapshot is current
- accepted-after snapshot is derived by appending one accepted non-unique
  field-path index
- mutation fingerprint is deterministic
- execution plan requires only field-path rebuild, physical validation, and
  runtime invalidation
- runner preflight/admission resolves to the supported developer physical path

If any proof fails, the DDL command returns a typed rejection and performs no
physical work.

## Execution Boundary

0.155 should not create a DDL-specific rebuild path.

DDL execution must call the same supported mutation runner path used by 0.154.
The only acceptable difference is the frontend source of the mutation request.
The physical transaction still follows the established order:

1. bind accepted-before and accepted-after snapshots
2. build runner input from accepted mutation execution plan
3. scan rows through accepted row contracts
4. run the field-path index runner
5. validate physical store state
6. invalidate runtime state
7. publish accepted-after snapshot
8. expose new `SchemaInfo` / visible index metadata

The DDL command must report success only after accepted snapshot publication is
visible and physical state is ready. Failed or rejected DDL must leave the
previous accepted schema visible.

## Developer Diagnostics

0.155 should add a developer-visible DDL mutation report that can be surfaced by
the SQL CLI/session path. It should include:

- statement kind
- target entity
- target index
- target field path
- parse status
- bind status
- mutation-plan status
- supported-path admission status
- rows scanned
- index keys written
- validation status
- runtime invalidation status
- accepted snapshot publication status

Diagnostics should be typed internally. Tests should not match full error
strings.

## Command Result Shape

The SQL result surface should distinguish DDL from row-returning query output.
The first result can be compact:

```text
CREATE INDEX index_name ON entity_name (field_path)
status: applied
rows_scanned: N
index_keys_written: N
```

A rejected DDL command should return a typed SQL/DDL error instead of a fake
empty row set.

## Concurrency And Restart

0.155 inherits the 0.154 rebuild assumptions:

- one affected entity
- synchronous full-scan rebuild
- exclusive schema/write gate for the affected entity while rebuilding
- no row writes may interleave between scan, validation, and publication
- startup must fail closed or discard safe unpublished staged work before
  exposing the database

DDL does not weaken these assumptions. If the SQL session path cannot guarantee
the gate, DDL execution should remain rejected until the gate is explicit.

## Test Plan

0.155 should close with tests that cover:

- parser accepts the exact supported `CREATE INDEX name ON entity (field_path)`
  grammar
- parser rejects unsupported DDL shapes without routing them as SELECT/update
  commands
- binder resolves entity and field path from accepted schema snapshots
- binder rejects generated-only, missing, duplicate, expression, unique, and
  multi-key index requests
- DDL lowering produces the same mutation plan and execution plan as the
  accepted proposal path for the supported index add
- DDL execution invokes the existing field-path runner/publication path
- successful DDL makes `SHOW INDEXES`, `DESCRIBE`, `EXPLAIN`, and indexed reads
  observe the accepted-after schema and ready physical index
- rejected DDL leaves accepted schema and physical index state unchanged
- runtime caches and cursor/schema signatures reject stale pre-DDL state
- source guards keep SQL DDL out of direct schema-store writes and generated
  index metadata authority

## Ship Criteria

0.155 is ready when:

- the supported SQL DDL surface is exactly one non-unique field-path secondary
  index add
- DDL binds only against accepted persisted schema snapshots
- DDL lowers through schema mutation requests/plans instead of constructing
  runner inputs or snapshots directly
- successful DDL uses the existing accepted mutation runner/publication path
- unsupported DDL fails before physical work or schema publication
- developer diagnostics explain parse, bind, plan, runner, and publication
  outcomes
- tests cover parser, binder, lowerer, execution, rejection, and source-boundary
  guards

## Deferred Work

After 0.155:

1. Add `DROP INDEX` only after secondary-index cleanup execution and recovery
   semantics are ready.
2. Add expression-index DDL only after expression rebuild execution is
   deterministic and restart-safe.
3. Add `ALTER TABLE ADD COLUMN` only after accepted field-default/nullability
   policy is command-safe and diagnostics are stable.
4. Add unique indexes only after uniqueness conflict reporting and constraint
   publication are schema-owned runner contracts.
5. Add multi-statement or batch DDL only after mutation transaction semantics are
   explicit.
6. Add online/concurrent DDL only after rebuild snapshot semantics are explicit.
