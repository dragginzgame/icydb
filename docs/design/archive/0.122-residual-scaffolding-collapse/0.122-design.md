# 0.122 Residual Scaffolding Collapse

## Thesis

`0.122` should perform sustained structural compression across expression,
predicate, lowering, prepared/session, route-adjacent, and explain-facing
layers by removing residual flow scaffolding that no longer transforms meaning.

This is not a dead-code pass.

It is not a semantic slice.

It is not a route or executor redesign slice.

It is not a small one-patch cleanup.

It is a sustained structural-compression slice whose job is to:

* inline ceremonial single-call helpers
* collapse thin pass-through wrapper chains
* remove DTO-to-DTO or DTO-to-planner-to-DTO loops
* eliminate repeated structural traversal over the same topology
* reduce wrap -> convert -> unwrap ceremony

The intended end state is:

* direct call chains
* fewer structural hops
* fewer adapter-only layers
* less flow ceremony in hot query paths
* no change to semantic ownership or feature behavior

---

## Why `0.122` Exists

The prior line already established the major boundary shifts:

* `0.116` centralizes truth-condition semantics
* `0.117` centralizes expression-family semantics
* `0.118` collapses predicate-owned expression flow
* `0.119` removes obvious obsolete expression-flow scaffolding
* `0.120` contracts pipeline multiplicity
* `0.121` collapses repeated access / explain projection walks

After those slices, the remaining drag is no longer primarily:

* semantic ambiguity
* missing canonical owner
* missing product surface
* obvious dead subsystems

The remaining drag is residual scaffolding:

* single-call helpers in hot paths
* thin pass-through wrappers
* adapter chains that no longer transform meaning
* repeated structural traversal logic
* DTO surfaces that still participate in structural routing after a canonical
  planner-owned form already exists

That is why `0.122` exists.

It is the next subtractive slice, but more sustained and broader than the
earlier explicit dead-lane deletions.

### Audit Alignment

#### 1. Velocity Preservation Audit

The latest velocity line still points at structural follow-through and
module-touch count rather than missing semantics.

The remaining cost pattern is:

* too many hops in routine call chains
* too many boundary-local wrappers
* too many places where one logical change still flows through several thin
  shells before reaching the real owner

That makes `0.122` first and foremost a velocity slice.

#### 2. Complexity Accretion Audit

The complexity issue here is not just branch count.

It is scaffolding count:

* one real transformation wrapped in multiple ceremony layers
* repeated structural traversal helpers
* redundant shell functions that still need maintenance

The right next complexity move is not another abstraction layer.

It is collapse.

#### 3. DRY Consolidation Audit

At this point, DRY pressure is structural:

* duplicate traversal
* duplicate adapter sequencing
* near-identical wrapper ladders
* repeated format-shaping that no longer owns distinct behavior

The correct DRY move is deletion or inlining, not coordination.

#### 4. Canonical Semantic Authority Audit

Canonical semantic authority is already the wrong place to keep working here.

The owner picture is already much healthier.

`0.122` should preserve semantic ownership and simply reduce the amount of
ceremonial structure around that ownership.

#### 5. Completeness Audit

Completeness now reads as bounded and structurally coherent, not feature-starved.

That means `0.122` is justified by compression and locality, not by widening
what the system can do.

## Problem

The system can already have the right owners and the right execution topology
while still carrying too much residual scaffolding.

Today the broad problem shape still includes:

* single-call helpers that only forward arguments
* multi-hop adapter chains where only one hop performs real work
* wrappers that rename values but do not change meaning
* structural DTO loops where one canonical planner-owned artifact already exists
* repeated structural traversal across adjacent consumers
* re-export or helper layers that no longer add a meaningful boundary

That causes:

* higher LoC
* higher cognitive load
* longer grep trails
* broader routine follow-through
* misleading architecture reads

The problem is not “used vs unused code.”

The problem is:

> too much used code exists only as flow ceremony

`0.122` should remove that ceremony.

---

## Goal

Reduce structural indirection until the remaining hot-path flow is mostly
direct and boundary-local.

Concretely:

* direct `A -> B` calls should dominate over `A -> B -> C -> D`
* helpers should remain only when they perform real transformation or protect
  a meaningful boundary
* DTOs should stop driving structural recursion when canonical planner-owned
  structure already exists
* repeated structural traversal should collapse to one canonical owner walk

---

## Core Invariant

Residual scaffolding may be removed aggressively, but semantic ownership and
behavior must remain unchanged.

Corollaries:

* planner ownership does not move
* route-shape behavior does not move
* executor semantics do not move
* public API surface does not widen
* meaningful abstraction boundaries remain

If a helper or layer performs no real transformation beyond forwarding,
renaming, wrapping, or unwrapping, it is a collapse candidate.

If removing it would duplicate real logic or blur an important boundary, it is
not.

2. `db/sql/lowering/*`
3. `db/predicate/*`
4. `db/query/plan/*`
5. `db/executor/planning/*`

Specific target patterns:

* DTO-to-DTO loops
* DTO-to-planner-to-DTO loops
* repeated structural traversal
* wrap -> convert -> unwrap flow ceremony
* redundant re-export layers that no longer add value

### Out Of Scope

* semantic changes
* planner ownership changes
* route/executor redesign
* feature widening
* dead-code hunting as a goal in itself
* removing meaningful abstraction boundaries just because they are small

---

## Ownership Boundary

### Session / Prepared Owns

* surface classification
* binding
* compatibility shells
* statement/result transport

Session / prepared must not keep:

* long forward-only call chains
* duplicate structural routing
* DTO loops that exist only to bounce into planner-owned structure and back

### Lowering Owns

* syntax-bound translation
* structural construction of planner-compatible inputs

Lowering must not keep:

* pass-through structural ladders that no longer change shape
* repeated wrapper layers over one canonical lowering path

### Predicate Owns

* real predicate adaptation where runtime predicate transport still matters

Predicate must not keep:

* planner-equivalent structural ceremony
* redundant wrapper shells around planner-owned truth or expression flow

### Planner Owns

* canonical structural and semantic authority already established by prior
  slices

Planner is not being reworked here.

Planner-facing ceremony around that ownership is being removed.

### Executor / Route-Adjacent Planning Owns

* actual route and execution behavior

Route/executor-adjacent shells that only forward into those owners are fair
collapse targets.

---

## Execution Model

`0.122` should be executed in sustained passes.

It must not stop after one or two patches if more collapse candidates remain.

Each pass should:

1. identify one cluster
2. apply multiple compressions in that cluster
3. remove newly exposed dead code
4. simplify resulting call chains
5. re-run analysis and continue

The slice is complete only when the remaining helpers are mostly meaningful and
the remaining chains are short.

---

## Allowed Transformations

`0.122` explicitly allows:

* inlining functions that:
  * have one call site
  * or only forward arguments
* deleting functions after inlining
* collapsing call chains
* merging adjacent helpers
* removing adapter layers that only rename or forward data
* removing duplicate traversal logic where one canonical path already exists
* removing DTO-driven structural recursion when canonical planner-owned
  structure already exists
* simplifying redundant enum-matching ladders
* removing re-export layers that no longer add real value

This means code may be removed even when it is used, if it is redundant
scaffolding.

---

## Constraints

`0.122` must not:

* change semantics
* change planner expression ownership
* change route-shape behavior
* change executor logic
* widen public API surface
* delete a boundary that still carries meaningful transformation or protection

The guiding rule is:

* remove ceremony
* preserve meaning

## Pass Structure

Each pass should record:

1. cluster targeted
2. proof question for that cluster
2. functions inlined
3. functions deleted
4. call chains simplified (`before -> after`)
5. LoC delta (`removed` vs `added`)
6. any newly exposed dead code

This is not just reporting style.

It is how the slice proves it is sustained rather than opportunistic.

Each pass should have one primary cluster and one proof question.

Examples:

* `db/session/sql/*`
  * can this surface reach the same owner in fewer hops without changing
    query/update/explain behavior?
* `db/sql/lowering/*`
  * are there multiple wrappers around one canonical lowering path?
* `db/predicate/*`
  * does this helper still transform anything, or is it only ceremony around
    planner-owned structure?

This is important because `0.122` should collapse clusters, not just accumulate
small local edits.

---

## Candidate Ranking

Within each pass, collapse candidates should be ranked in this order:

1. multi-hop chains where only one hop performs real work
2. repeated structural traversal over the same topology
3. DTO bounce loops
4. single-call forwarders
5. thin re-export layers

That ordering is deliberate.

If the slice starts with isolated tiny helpers, it can consume time without
removing the real structural drag.

The highest-yield work is:

* whole adapter ladders
* repeated traversal seams
* DTO routing loops

not scattered cosmetic inlining.

---

## Execution Guardrail

`0.122` should be executed as a small number of dense cluster passes, not as a
repo-wide opportunistic sweep.

Preferred shape:

* 3 to 5 dense passes
* each pass centered on one subsystem cluster
* each pass deletes or collapses multiple related scaffolding layers at once

Avoid:

* touching unrelated modules just because a helper is small
* inlining helpers that are small but still meaningful
* broad cleanup momentum that outpaces the cluster thesis

The slice stays healthy only if it remains:

* cluster-driven
* high-yield
* saturation-seeking

and not:

* opportunistic
* cosmetic
* repo-wide by drift

---

## Continuation Rule

Do not stop after small changes.

Continue while any of these remain true:

* single-call helpers remain in hot paths
* multi-hop adapter chains of three or more steps remain
* duplicated structural traversal remains over the same topology
* DTOs still act as structural classification inputs where a canonical planner
  form already exists
* most flow is still `A -> B -> C -> D` instead of direct `A -> B`

If uncertain whether the slice is saturated, do one more pass.

---

## Saturation Condition

`0.122` is complete only when:

* further inlining would duplicate real logic instead of simplify
* remaining helpers are multi-call and meaningful
* remaining adapters perform real transformation
* remaining structural chains longer than two hops are exceptional rather than
  normal

Another way to state the stop condition:

* no remaining single-call helpers in hot paths that only forward
* no routine multi-hop adapter chains
* no duplicated structural traversal over the same topology
* no DTO used as a structural classification input when a canonical artifact
  already exists

---

## Exit Condition

`0.122` is done when:

* the targeted high-density clusters have been compressed to saturation
* call chains are predominantly direct
* most ceremonial wrappers have been removed
* newly exposed dead code from those collapses has been deleted
* further collapse would trade simplification for duplication

Expected visible outcomes:

* lower LoC in session, lowering, predicate, planner-adjacent, and
  explain-facing scaffolding
* fewer modules touched for routine expression / SQL follow-through
* fewer helper hops in the hot path
* clearer owner boundaries because less ceremonial code remains around them

## Non-Goals

`0.122` should not:

* become a semantic redesign
* become a dead-code pass masquerading as architecture work
* widen supported feature families
* re-open route ownership
* re-open executor behavior
* flatten boundaries purely for aesthetic minimalism

---

## Summary

`0.122` is a sustained structural-compression slice.

It is not trying to decide meaning again.

It is not trying to widen product surface.

It is trying to remove the residual scaffolding that still makes the current
system broader, noisier, and more ceremonially indirect than its actual
logical model requires.

The rule is simple:

* keep meaningful boundaries
* remove ceremonial ones
* continue in passes until saturation
