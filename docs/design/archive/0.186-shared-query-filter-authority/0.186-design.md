# 0.186 Shared Query Filter Authority

Status: active baseline.

This line follows the 0.184 query-engine audit cleanup and the 0.185
branch-aware routing revisit. It starts with source mapping and parity proof,
not a broad filter rewrite. Do not change runtime filter semantics, route
choice, cursor format, or public SQL/fluent behavior until a narrow slice has a
clear authority handoff and focused validation.

## Context

0.184 reduced several duplicate query-engine flows. The filter-contract work
made SQL lowering safer by keeping the visible SQL truth expression and the
predicate-pushdown subset together, then moved planner residual expression,
residual predicate, and runtime filter program into one residual-filter
contract.

That was intentionally narrower than the full architecture. SQL and fluent
filters can still enter the query engine through different frontend adapters
before they converge on query intent, planning, and execution.

0.185 stayed focused on branch-aware query routing. It confirmed that SQL and
fluent `IN` / branch-heavy query shapes feed the same planner path where they
should, and it deliberately left full shared-filter authority for this line.

## Problem

IcyDB has multiple representations of one user-visible filtering idea:

- SQL visible boolean expression;
- SQL-derived predicate subset;
- fluent predicate/filter expression;
- access-planning predicate;
- residual expression after access-path satisfaction;
- residual predicate after access-path satisfaction;
- compiled runtime filter program;
- EXPLAIN and cache identity projections.

These are not all redundant, but they should not be independent authorities.
The drift risk is that SQL, fluent, planner, executor, count/cardinality, and
diagnostics can each interpret "the filter" slightly differently.

The concrete failure modes are:

- SQL truth semantics, especially `NULL`, `IN`, and `NOT IN`, diverge from
  predicate pushdown.
- Fluent and SQL build equivalent filters but produce different route,
  residual, cache, or EXPLAIN identity.
- Count/cardinality shortcuts prove a predicate shape that does not match the
  residual runtime filter.
- Branch-aware routing optimizes one frontend but not the other.
- Diagnostics describe the rendered filter string instead of the planner-owned
  proof that execution actually uses.

## Goal

Introduce one frontend-neutral query-filter contract that becomes the shared
authority after frontend-specific lowering.

SQL and fluent should become adapters into this contract:

```text
SQL WHERE
  -> SQL truth/literal/alias lowering
  -> shared query filter contract

Fluent filter
  -> typed/filter builder lowering
  -> shared query filter contract

Shared query filter contract
  -> predicate subset
  -> access route candidates
  -> residual filter contract
  -> count/cardinality eligibility
  -> cache identity
  -> EXPLAIN diagnostics
```

The contract should not encode SQL parser details. SQL-specific work belongs
before the handoff.

## Authority Split

0.186 distinguishes two authorities:

- Pre-access semantic authority: what filter the user asked for, and which
  predicate subset is safely usable before choosing access paths.
- Post-access execution authority: which predicate terms the selected access
  path proved, and which residual filter work must still run.

`NormalizedFilter` is the current candidate for the pre-access semantic
authority. `ResidualFilterContract` remains the post-access execution
authority. 0.184 already improved the post-access side, so 0.186 should avoid
changing residual runtime compilation unless a later slice proves it is
strictly necessary.

## Pre-Access Contract Shape

The exact type names are not final yet, but the shared object likely needs to
carry:

- semantic authority for the user-visible filter, either expression-backed or
  predicate-only;
- canonical visible boolean expression, when the frontend has one;
- canonical predicate subset, when extractable;
- proof of whether the predicate subset covers the user-visible filter
  semantics;
- reason when no subset can be extracted or when coverage is partial;
- source kind for diagnostics only, not for runtime decisions;
- schema-binding facts needed for literal/coercion stability;
- stable identity/fingerprint input.

Coverage is defined against user-visible filter semantics, not only against a
visible expression. That matters because SQL and fluent filters do not need the
same frontend shape.

- SQL: user-visible filter semantics are the bound SQL boolean expression.
  SQL-specific truth, literal, alias, and coercion rules remain before the
  shared handoff.
- Fluent: user-visible filter semantics may be predicate-only. Fluent filters
  do not need a fake SQL-like visible expression just to satisfy the shared
  abstraction.

One possible internal vocabulary is:

```text
FilterSemanticAuthority
  = ExpressionBacked(canonical_bool_expr)
  | PredicateOnly(canonical_predicate)

PredicateCoverage
  = Full
  | Partial(reason)
  | None(reason)
```

Names can change, but this distinction should remain. Do not make SQL
expression semantics the required shape of every shared filter.

## Post-Access Contract Shape

Post-access planning should then derive:

- access-proven predicate terms;
- residual expression;
- residual predicate subset;
- residual runtime program;
- pushdown outcome and reason diagnostics.

0.184 already created part of the post-access side through
`ResidualFilterContract`. 0.186 decides whether the pre-access side also needs
a first-class shared contract.

## Predicate Extraction Ownership

Predicate extraction should move toward the shared query-intent /
planner-preparation boundary.

Frontend adapters own frontend-specific work:

- SQL lowering owns name resolution, alias handling, SQL truth/literal
  semantics, SQL-specific coercion, and construction of the bound visible
  expression.
- Fluent lowering owns typed builder interpretation, API-level operator
  normalization, field/literal binding, and construction of equivalent
  normalized expression or predicate input.

The shared layer should own the authoritative predicate subset, coverage proof,
and extraction failure reason. Frontends may pass already-normalized inputs or
hints during migration, but route selection, count/cardinality, cache identity,
and EXPLAIN must consume the shared extraction result rather than independent
frontend-owned predicate facts.

Do not overload `NormalizedFilter` while proving it as the seam. It should own
pre-access semantic filter facts only. Runtime filter programs, residual shape,
and predicate-pushdown diagnostics stay with `ResidualFilterContract` and
logical planning.

## Non-Goals

- Do not change SQL semantics.
- Do not broaden SQL admission.
- Do not change fluent public API behavior.
- Do not change branch-aware route choice as part of this design.
- Do not make EXPLAIN snapshots the authority for runtime behavior.
- Do not change cache identity in the first slice unless the change is
  intentionally versioned and behavior-neutral.
- Do not turn `NormalizedFilter` into a holder for SQL parser traces, fluent
  builder traces, EXPLAIN formatter state, route hints, or runtime compilation
  details.
- Do not keep backwards-compatibility aliases before 1.0.0 if the accepted
  design hard-cuts an internal contract.
- Do not merge this with cursor redesign, cardinality metadata, or cost-based
  route selection.

## Relationship To 0.184

0.184 remains the query-engine audit cleanup line. It should continue to accept
narrow drift-reduction work, especially where behavior is unchanged and tests
prove existing semantics.

0.184 should not attempt the full frontend-neutral filter contract. That would
turn an audit cleanup line into a broad architecture migration.

The narrow 0.184-compatible work is:

- keep SQL visible expression and predicate subset paired;
- derive predicate facts from the schema-bound SQL expression where safe;
- keep residual-filter facts grouped in planner-owned contracts;
- improve diagnostics to consume planner-owned facts.

## Relationship To 0.185

0.185 stayed branch-aware:

- branch-set route representation;
- branch merge cursors;
- small versus large `IN` route choice;
- SQL and fluent branch-heavy queries feeding the same planner path;
- branch-tree reuse versus special-case `IN` flows.

0.185 exposed pressure that motivates this 0.186 design, but closed without
folding shared filter authority into branch-aware routing.

## Current Baseline

The current source map lives in `source-map.md`. The important existing
authority points are:

- SQL lowering owns `LoweredSqlFilter`, which keeps a visible expression and
  optional predicate subset together until query-intent handoff.
- Fluent query builders lower typed filters into normalized planner
  expressions before query-intent handoff.
- Query intent owns `NormalizedFilter`, which stores one normalized expression,
  one optional predicate subset, and whether the subset covers the expression.
- Finalized planning owns `ResidualFilterContract` and
  `PredicatePushdownDiagnostics` for post-access residual execution and
  diagnostics.

That is already a partial shared authority chain. The 0.186 question is whether
the pre-access handoff should stay as `NormalizedFilter` or become a more
explicit frontend-neutral contract.

## Review Questions

Peer review clarified these answers:

1. Is the current `NormalizedFilter` close enough to become the shared
   pre-access contract, or should a new type own that boundary?
   Probably yes. Prove and extend it before replacing it.
2. Which layer should own predicate extraction: frontend lowering, query
   intent, or planner preparation?
   The shared query-intent / planner-preparation boundary should own the
   authoritative extraction result.
3. Can fluent filters always produce a canonical visible expression, or do
   predicate-only fluent filters remain valid first-class inputs?
   Predicate-only fluent filters remain valid first-class inputs.
4. Should SQL-specific coercion normalization remain a pre-handoff adapter, or
   should the shared contract carry frontend-specific coercion facts?
   SQL-specific mechanisms remain pre-handoff; the shared contract carries only
   bound frontend-neutral facts.
5. Which cache identity surfaces are allowed to change?
   None in the first slice unless explicitly versioned and proven
   behavior-neutral.
6. What EXPLAIN snapshots are semantic contracts versus implementation detail?
   Route, pushdown, access-proven terms, residual presence, shortcut
   eligibility, and pushdown failure reasons are semantic. Rendered expression
   strings, commutative ordering, temporary type names, and diagnostic wording
   are implementation detail.
7. What proof is needed that count/cardinality shortcuts consume the same
   filter authority as page execution?
   Count/cardinality eligibility must be derived from the shared pre-access
   contract plus planner access proof and residual contract, not from a separate
   frontend shortcut.
8. How does this interact with branch-aware `IN` routing in 0.185?
   0.185 remains route-focused. 0.186 should ensure SQL and fluent produce the
   same branch-capable predicate facts before routing, while treating SQL
   `NOT IN` as a residual-sensitive acceptance area.

## First Slice

The first implementation slice stays smaller than the full design:

1. Document the current shared and divergent filter entry points.
2. Add source-audit tests proving SQL and fluent equivalent filters converge at
   the query-intent/planner boundary for representative shapes:
   equality, ranges, `IN`, `NOT IN`, `NULL`, `IS NOT NULL`, text prefix,
   booleans, nested composition, branch-aware `IN`, and count/cardinality
   candidates.
3. For each case, prove agreement on normalized expression shape where one
   exists, predicate subset shape, coverage, route candidate class, chosen
   route class, residual requirement, count/cardinality eligibility, cache
   identity where intended stable, and EXPLAIN semantic facts.
4. Move or name only one frontend-neutral fact in the shared contract.
5. Prove cache, EXPLAIN, route, residual, and count behavior are unchanged.
6. Delete one duplicate derivation path.

No implementation slice should change both the pre-access contract and the
post-access residual contract at the same time.

## Acceptance Gates

Before any code slice changes the filter contract shape, require:

- SQL and fluent parity tests for equivalent filters.
- SQL truth-semantics tests for `NULL`, `IN`, and `NOT IN`.
- Route parity tests for indexed, full-scan, branch-aware, and count/cardinality
  shapes.
- EXPLAIN identity tests that distinguish semantic identity from diagnostics
  presentation.
- Perf attribution showing no regression on common SQL and fluent filters.
- Source invariant tests preventing new frontend-owned predicate shortcuts from
  bypassing the shared contract.
- A hard source invariant: no planner, count/cardinality, route-selection,
  cache-identity, or EXPLAIN consumer may derive new predicate facts directly
  from frontend-owned SQL or fluent representations once the shared contract
  exists.

Allowed consumers read predicate facts from the shared pre-access contract or
from `ResidualFilterContract`. Disallowed consumers reparse SQL expressions,
inspect fluent builder internals, reconstruct branch predicates from rendered
text, perform count-local simplified extraction, or derive semantic EXPLAIN
facts from display strings.

## Current Recommendation

Treat `NormalizedFilter` as the current candidate pre-access authority seam.
Do not rename or replace it first. Add the missing neutral facts one at a time,
starting with either predicate extraction failure/coverage reason or stable
filter identity material, and prove no cache, route, residual, EXPLAIN, or
count/cardinality behavior changes unless explicitly versioned.

The best first deletion target is a duplicate pre-access predicate derivation
or interpretation path: frontend-specific extraction, count/cardinality-local
proof, EXPLAIN-local interpretation, or route-local predicate reconstruction.
Do not start by deleting residual runtime compilation paths.

After 0.185 and this 0.186 decision are complete, run the second query-engine
mega audit parked in
`docs/design/0.187-second-query-engine-audit/second-mega-audit-reminder.md`.
That audit should look for duplicate flows that survive the branch-aware and
shared-filter work, not rediscover work that is already intentionally pending.
