# 0.125 Expression Traversal Contract Unification

## Thesis

`0.125` should unify expression traversal ownership across parser, planner,
and compiled expression representations, and route lowering, validation, and
execution-setup consumers through those owner-local contracts.

This is not an expression-semantics redesign slice.

It is not a generic visitor-framework slice.

It is not an executor evaluation rewrite.

It is an expression-traversal ownership slice whose job is to stop each tree
consumer from reintroducing its own recursive walk whenever it needs one more
structural query.

The target end state is:

* one explicit traversal contract per expression representation
* parser, planner, lowering, and compiled execution consumers asking owner
  trees for structural answers instead of open-coding recursion
* no repeated "walk children, then recurse" ladders for the same tree shape
  across adjacent consumers
* layer-local traversal helpers only where the tree itself is layer-local

---

## Why `0.125` Exists

The recent parser-side work already cleaned one important part of this seam.

`SqlExpr` now owns shared tree queries in
`db/sql/parser/model.rs` through:

* `any_tree_expr(...)`
* `all_tree_expr(...)`
* owner-side structural predicates built on top of those walkers

That solved the parser half of the problem:

* aggregate detection no longer required parallel parser walks
* local-scalar checks no longer required parallel parser walks
* omitted-ELSE detection no longer required parallel parser walks

`0.125` has now extended that ownership model through the first planner and
compiled-expression follow-through too:

* planner `Expr` now owns the main planner-tree traversal contract
* compiled `ScalarProjectionExpr` now owns compiled slot traversal
* frozen `ResolvedOrder` shapes now own their direct-field and referenced-slot
  structural queries
* SQL lowering now routes more lowered-expression and aggregate-leaf scans
  through owner-local traversal instead of reopening local recursion

What remains is the same pattern in other layers.

The current tree still contains several places where planner, lowering, or
compiled execution representations can still accumulate local recursive descent
if new structural queries are added without using the owner contracts that now
exist.

That is now the remaining `0.125` risk: not the absence of a traversal model,
but consumers drifting back out of it.

---

## Audit Alignment

### 1. Parser Already Has the Right Shape

The parser is now the reference model.

`db/sql/parser/model.rs` already demonstrates the intended contract:

* the tree owner defines the recursion
* consumers build structural queries on top
* callers do not keep inventing new recursive helpers

This is the pattern `0.125` should extend to the remaining expression
representations.

### 2. Planner `Expr` Was The Main Hotspot And Is Now The Main Reference

The planner-owned `Expr` tree in `db/query/plan/expr/ast.rs` was the first real
production hotspot, and `0.125` has already moved the repeated structural
queries onto that owner.

The owner-local surface now includes:

* `contains_aggregate()`
* `contains_case()`
* `any_tree_expr(...)`
* `all_tree_expr(...)`
* `try_for_each_tree_expr(...)`
* `try_for_each_tree_aggregate(...)`
* `try_for_each_tree_expr_with_compare_index(...)`
* `references_only_fields(...)`

The important architectural result is no longer the method list itself. It is
that grouped validation, grouped policy, grouped projection aggregate scanning,
logical planning, and SQL lowering now consume the same traversal boundary
instead of rebuilding planner-child recursion inline.

That makes planner `Expr` the reference implementation for the rest of the
slice, not the remaining hotspot.

### 3. Compiled Scalar Trees Were The Natural Follow-On And Are Now In Place

The planner-owned compiled scalar tree in
`db/query/plan/expr/scalar.rs` now owns the compiled slot walk through:

* `for_each_referenced_slot(...)`
* `extend_referenced_slots(...)`
* `mark_referenced_slots(...)`

Execution-input and grouped-stage preparation now consume that owner contract
directly. This is the intended `0.125` pattern:

* the compiled tree owns structural slot traversal
* downstream preparation code only owns slot policy

### 4. Grouped Compiled Trees Likely Need the Same Treatment

`db/executor/projection/grouped.rs` owns `GroupedProjectionExpr`.

That tree already has clear recursive consumers for:

* grouped projection evaluation
* grouped projection compilation from planner `Expr`

Those evaluation and compilation bodies are legitimate.

What `0.125` should still watch for is whether additional structural queries
begin to grow beside them, for example:

* slot collection
* aggregate-leaf discovery
* structural classification
* owner-local `any/all` style queries

The doc should still treat grouped compiled trees as part of the same audit
family, but widening should now be driven by concrete duplicated walkers, not
by the original assumption that planner `Expr` and compiled scalar were still
missing their owner contracts.

---

## Problem

The codebase still has multiple expression representations, and several of
them are missing an explicit traversal contract.

The repeated pattern is:

* a tree representation exists
* one consumer needs a structural answer
* instead of asking the owner tree to walk itself, the consumer redefines
  recursion locally

That creates three classes of risk.

### 1. Structural Drift

Two helpers can ask different questions but still drift on:

* which child variants are visited
* whether alias/test wrappers are transparent
* ordering of visitation
* early-exit behavior

### 2. Maintenance Spread

One new expression variant or child-bearing node forces updates across many
consumer-owned walkers instead of one owner boundary.

### 3. LoC Growth

Even when behavior stays correct, recursive match ladders keep spreading
through planner and lowering modules that should only own leaf policy.

---

## Goal

Replace consumer-owned recursive expression walks with owner-owned traversal
contracts, then hold that boundary as later structural queries appear.

`0.125` should deliver:

* one named traversal contract for planner `Expr`
* one named traversal contract for compiled scalar projection expressions
* parser aggregate-leaf and omitted-ELSE structural scans routed through their
  owner-local contracts instead of new lowering-local helpers
* grouped compiled trees audited under the same rule, and only widened if they
  show the same duplication smell
* parser remaining the reference implementation, not a special-case one-off

This does not require one shared cross-layer visitor type.

It requires one consistent ownership pattern:

* the tree owner defines traversal
* consumers define predicates, folds, and policy on top of that traversal

---

## Core Invariant

For each expression representation:

* `SqlExpr`
* planner `Expr`
* `ScalarProjectionExpr`
* `GroupedProjectionExpr` where needed

there is exactly one owner-local recursion contract.

Other layers may still ask different structural questions, but they must do so
by consuming the owner-local traversal contract instead of rebuilding child
recursion inline.

Corollary:

`0.125` is not complete if parser/planner/lowering code still keeps several
local recursive walkers that differ only in leaf policy for the same tree
shape.

### Traversal Boundary Invariant

For a given expression owner, child visitation order, wrapper transparency, and
early-exit behavior are defined only by the owner-local traversal contract.

Consumers may supply predicates, folds, and leaf policy, but they must not:

* reopen child recursion outside that contract
* selectively unwrap wrapper nodes outside that contract
* redefine early-exit behavior outside that contract

That is the boundary that prevents "shared traversal plus one local special
unwrap" drift.

## Scope

### In Scope

* planner `Expr` traversal contract
* parser-side structural traversal follow-through where SQL lowering is still
  rewalking `SqlExpr` for structural answers
* structural planner/lowering queries over planner `Expr`
* compiled scalar projection traversal contract
* grouped compiled expression audit
* deletion of duplicate consumer-owned recursion where the owner contract can
  serve instead

### Out Of Scope

* replacing all expression enums with one shared enum
* generic visitor trait frameworks spanning parser, planner, and executor
* evaluation algorithm redesign
* type-inference redesign
* broad predicate normalization redesign
* expression rendering redesign

---

## Ownership Model

### Parser Owns Parser Traversal

`SqlExpr` already owns parser-tree traversal in `db/sql/parser/model.rs`.

That should remain parser-local.

### Planner Owns Planner Traversal

`Expr` in `db/query/plan/expr/ast.rs` should own planner-tree traversal.

That contract should become the shared source for:

* aggregate-leaf visitation
* `any/all` structural queries
* grouped validation/policy tree scans
* lowering-time structural analysis where the input is already planner `Expr`

### Compiled Trees Own Compiled-Tree Traversal

`ScalarProjectionExpr` and `GroupedProjectionExpr` should each own their own
compiled-tree traversal contracts when structural consumers exist.

Execution code may still evaluate those trees directly, but structural queries
such as slot collection should not rederive recursion outside the tree owner.

---

## Contract Shape

`0.125` should standardize the pattern, not necessarily the exact function
names.

The expected family is split into two layers.

### Generic Traversal Primitives

The owner should provide the basic recursion contract through helpers such as:

* `any_tree_expr(...)`
* `all_tree_expr(...)`

### Semantically Meaningful Owner Helpers

The owner may also expose a small number of named structural helpers when the
query is common and stable, for example:

* `try_for_each_tree_aggregate(...)`
* referenced-field visitation
* direct-field leakage inspection where that concept is stable enough
* owner-local analysis/fold helpers built on top of the generic traversal
  primitives

For compiled scalar trees, the parallel contract can be narrower:

* referenced-slot visitation
* owner-local `any/all` queries only if real consumers exist

Important constraint:

Do not introduce a generic cross-layer visitor framework just to say "walk the
children."

The right abstraction level is:

* owner-local traversal methods
* explicit consumer policy closures

not:

* one trait hierarchy for every expression family in the system

---

## First Concrete Patch

The first patch should target planner `Expr`.

Reason:

* it has the clearest remaining duplicate recursion
* lowering and grouped validation already consume planner `Expr`
* it will likely delete the largest amount of repeated structural code

That patch should:

1. add owner-local traversal helpers to planner `Expr`
2. move aggregate-leaf visitation onto the planner owner
3. move grouped-HAVING structural and policy scans onto the planner owner or
   onto owner-based traversal consumers
4. route lowering analysis through the owner-local traversal contract instead
   of open-coded recursion

The first patch should not widen into compiled trees unless that naturally
falls out cleanly.

---

## Follow-On Patch

The follow-on patch should target `ScalarProjectionExpr`.

That patch should:

* keep referenced-slot traversal on the compiled tree owner
* make `ScalarProjectionExpr` the owner of any compiled-expression-local
  structural queries that execution setup needs
* generalize that traversal only as far as real compiled consumers need it
* remove any execution-input-local recursion that exists only because the
  compiled tree does not expose the needed walk itself

Execution-input code should consume those owner helpers and should not recurse
over compiled scalar shape directly.

If grouped compiled trees show the same duplication pressure during that audit,
they can follow in the same patch or the next one.

---

## What `0.125` Should Not Do

`0.125` should not:

* unify parser, planner, and compiled trees into one representation
* collapse evaluation, typing, and validation into one mega visitor
* hide meaningful layer boundaries under generic traversal traits
* move leaf policy onto the traversal owner when the policy belongs to the
  consumer

The traversal owner should own recursion.

The consumer should still own the question being asked of that tree.

---

## Completion Criteria

`0.125` is complete when:

* parser `SqlExpr` remains the reference owner-local traversal model
* planner `Expr` exposes the canonical recursion contract for planner/lowering
  structural queries
* grouped validation and lowering no longer keep their own recursive child
  descent over planner `Expr` where the owner contract can answer the same
  question
* compiled scalar slot-reference traversal is owner-local
* no new expression structural query starts by open-coding another recursive
  descent unless the tree owner truly lacks a contract and the missing
  ownership is called out explicitly

---

## Expected Payoff

If `0.125` succeeds, the codebase gets:

* fewer recursive match ladders over expression trees
* lower drift risk when expression variants widen
* clearer ownership boundaries between tree shape and leaf policy
* a repeatable pattern for future expression-tree work as joins and richer
  projection semantics land

That is the right next step after `0.124`.

`0.124` centralized parser-side expression traversal on `SqlExpr`.

`0.125` should extend that ownership model to planner and compiled expression
trees.
