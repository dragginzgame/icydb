# 0.37 - Aggregate Fluent API Consolidation

## Status

Draft - API Design  
Target: `0.37`  
Prerequisite: `0.36.x` grouped contract freeze remains intact

---

# 1. Objective

Consolidate aggregate fluent surface ergonomics by replacing combinatorial
terminal names with composable aggregate builders.

Primary goal:

- reduce method-surface explosion in `load` + grouped aggregate terminals
- keep executor/resource semantics unchanged
- keep deterministic explain/fingerprint/continuation behavior

---

# 2. Problem Statement

Current terminal naming composes concerns into method names:

- context (`group_*`)
- aggregate kind (`count`, `sum`, `min`, `max`, ...)
- modifiers (`distinct`)
- target field (`*_by("field")`)

This scales poorly as new modifiers and aggregate forms are added.

---

# 3. Design Principles

1. API-layer consolidation only.
2. No new operator class.
3. No hidden buffering.
4. Keep `RESOURCE_MODEL.md` enforcement unchanged.
5. Keep planner/executor authority boundaries unchanged.
6. Fail closed for unsupported aggregate expressions.

---

# 4. Proposed Public Shape (`0.37`)

Introduce composable aggregate expressions:

```rust
sum("rank").distinct().alias("unique_rank_sum")
count("id").alias("orders")
```

Introduce aggregate-oriented fluent entrypoints:

```rust
session
    .load::<Order>()
    .group_by("user_id")
    .aggregate(sum("rank").distinct())
    .execute_grouped()?;
```

Global aggregate shape remains available without `GROUP BY`:

```rust
session
    .load::<Order>()
    .aggregate(sum("rank").distinct())
    .execute_grouped()?;
```

Implementation note:

- new builders lower to existing internal aggregate plan/spec structures
- no executor-specific public surface leak

---

# 5. Internal Mapping Contract

Builder expressions must normalize into existing planning contracts:

- aggregate kind
- target field
- distinct flag
- optional alias (projection metadata only)

Equivalent old/new API forms must compile to equivalent logical grouped/scalar
shapes unless the new API enables a new explicitly-scoped feature.

---

# 6. Compatibility Plan

Phase 1 (`0.37.0`):

- add aggregate builder types and fluent entrypoint(s)
- remove combinatorial grouped helper terminals from query/fluent API surface
  (`group_*` aggregate terminals)

Phase 2 (`0.37.x`):

- convert internal callsites/tests to canonical builder entrypoint(s)
- assert builder-shape/fingerprint/continuation regression stability after the
  hard cut

---

# 7. Validation Boundaries

`0.37` builder consolidation must preserve current validation policy:

- global DISTINCT restrictions remain fail-closed (`COUNT`/`SUM` only in
  supported shapes)
- grouped/HAVING restrictions remain unchanged unless explicitly expanded
- unsupported builder combinations return typed validation errors

---

# 8. Non-Goals (`0.37`)

- window aggregate DSL
- arbitrary expression trees for aggregate arguments
- semantic changes to grouped execution strategy eligibility
- continuation token shape/version changes
- storage-level aggregate pushdown

---

# 9. Resource-Audit Follow-Up (`0.37.2`)

The `2026-03-01` resource-model follow-ups tracked here are closed in `0.37.2`:

1. Grouped strategy/runtime alignment:
   - route-level grouped strategy labels are now explicitly materialized
     (`HashMaterialized`, `OrderedMaterialized`)
   - grouped metrics remain explicitly materialized
   - grouped execution mode remains `Materialized` and is runtime-asserted
2. DISTINCT state authority clarification:
   - grouped Class B DISTINCT remains exclusively budget-accounted through
     grouped `ExecutionContext`
   - non-grouped materialized DISTINCT terminals now share one explicit helper
     boundary (`aggregate::materialized_distinct`) and remain documented as
     non-grouped effective-window terminals in `RESOURCE_MODEL.md`

---

# 10. Exit Criteria

`0.37` API consolidation is complete when all are true:

1. composable aggregate builder surface is shipped
2. combinatorial grouped helper terminals are removed from public query/fluent
   surfaces
3. explain/fingerprint/continuation regression tests pass for builder-only
   grouped forms
4. resource/budget behavior remains bounded and unchanged
5. changelog/status/roadmap docs are synchronized
