# 0.103 Residual Execution and Explain Completion

## Thesis

`0.103` should make scalar `WHERE` execution fully explicit by stabilizing the
three filter representations that now exist across planning, execution, and
explain:

* the full semantic filter expression
* the derived pushdown predicate
* the residual post-access filter

> **`0.103` completes the scalar filter model by making semantic filter
> ownership, extracted pushdown ownership, and residual execution ownership
> explicit, consistent, and visible across scalar planning, execution, and
> explain.**

This is the consolidation slice after `0.102`, not another SQL-surface
widening.

---

## Current baseline

After `0.102`, scalar `WHERE` already has:

* planner-owned semantic `filter_expr`
* conservative pushdown predicate extraction
* residual execution for admitted scalar boolean expressions
* explicit prepared fallback behavior for residual-heavy scalar shapes

But the model is still not fully explicit end-to-end:

* the residual filter is not yet a first-class planned artifact everywhere
* execution ordering is still partly implicit
* explain output does not consistently present semantic vs pushdown vs residual
  filtering as one stable contract
* predicate and expression evaluation roles can still read as partly duplicated
  instead of deliberately tiered

---

## Problem statement

After `0.102`, scalar `WHERE` can admit recursive scalar expression trees and
evaluate them through residual filtering.

However, the follow-through is still incomplete:

* the residual filter is not yet represented as one explicit first-class
  planning artifact everywhere
* execution order is not yet stated as a release contract
* explain output does not yet consistently distinguish:
  * the semantic filter
  * the extracted pushdown predicate
  * the residual post-access filter
* the relationship between `filter_expr`, `predicate`, and residual execution
  is still not fully formalized

That leaves four real risks:

* unclear explain output
* subtle correctness drift under mixed pushdown/residual plans
* difficulty reasoning about route quality and performance
* weak foundations for future optimizer work

So the main issue is no longer scalar `WHERE` admission.

The issue is that the execution and explain model for the already-admitted
scalar filter split is still too implicit.

---

## Design goal

Make the scalar filter pipeline explicit, stable, and shared.

That means:

* every scalar plan clearly owns:
  * one semantic filter expression
  * one extracted pushdown predicate
  * one explicit residual execution artifact when residual work exists
* execution order is defined and stable
* explain surfaces render this model directly
* predicate evaluation and residual expression evaluation have clearly
  separated roles

The goal is:

> **finish the scalar `WHERE` split by turning planning-time filter ownership,
> execution-time filter staging, and explain-time filter visibility into one
> shared contract.**

---

## Non-goals

`0.103` does **not** aim to:

* widen SQL surface
* change scalar expression semantics
* widen scalar `WHERE` admission again
* introduce new optimizer rules
* make predicate extraction more aggressive
* widen fixed-route prepared-template coverage
* change grouped `HAVING` semantics
* redesign grouped execution ownership

This is a model-completion slice, not a feature or optimization slice.

---

## Normative model

### The scalar filter triad

Every scalar query plan with a `WHERE` clause must be able to represent these
three filter views:

* `filter_expr`
  * the full planner-owned semantic boolean expression
* `predicate`
  * the extracted pushdown subset representable in the predicate model
* `residual_filter_expr`
  * the post-access boolean expression actually evaluated after access when the
    extracted predicate does not fully discharge the semantic filter

The important ownership rule is:

* `filter_expr` is the semantic owner
* `predicate` is derived
* `residual_filter_expr` is the execution artifact for the remaining
  post-access work

`residual_filter_expr`, when present, must be determined during planning and
carried as explicit plan state.

Execution must consume this explicit artifact directly rather than re-deriving
residual existence or residual shape from `filter_expr`, `predicate`, or
access satisfaction.

### Residual derivation contract

`0.103` must not require a stronger algebraic residual contract than the
planner can actually guarantee.

The normative rule is:

* `predicate` must be a sound extracted prefilter
* `residual_filter_expr`, when present, must preserve full semantic correctness
  after that prefilter

This means:

* if no pushdown predicate is extracted:
  * no derived pushdown predicate is present
  * `residual_filter_expr == filter_expr`
* if the extracted predicate fully covers the semantic filter:
  * no residual filter expression is present
* if the extracted predicate is only a sound subset:
  * `residual_filter_expr` may be either:
    * an exact semantic remainder expression, when the planner can derive one
      cleanly, or
    * a planner-owned semantically equivalent post-access expression that still
      preserves the original filter contract

`0.103` requires soundness and explicitness, not a universal exact remainder
formula.

### Execution order contract

Scalar execution must apply filtering in this order:

1. access execution using `predicate`, when present
2. residual row filtering using `residual_filter_expr`, when present
3. projection and result shaping

Residual filtering:

* must happen before projection
* must use the compiled scalar expression engine
* must be row-wise and deterministic

### Evaluation contract

`0.103` establishes this split:

* predicate evaluation is limited to the extracted pushdown subset and must not
  become a competing semantic owner
* residual evaluation uses the general compiled scalar expression engine

No new evaluation engine may be introduced for this slice.

Longer-term convergence may still reduce duplication further, but `0.103`
already requires the current roles to be explicit and non-overlapping.

## Explain contract

Explain must render the scalar filter triad explicitly when relevant.

Canonical form:

```text
Filter:
  semantic:  <full filter_expr>
  pushdown:  <predicate or none>
  residual:  <residual_filter_expr or none>
```

Rules:

* `semantic` is shown whenever a scalar `WHERE` exists
* `pushdown` is shown only when non-empty
* `residual` is shown only when non-empty
* if `filter_expr` exists and no derived pushdown predicate is present,
  explain must still render both:
  * `semantic: <filter_expr>`
  * `residual: <residual_filter_expr>`
* the semantic filter shown in explain is the planner-owned semantic filter
  identity, not a collapsed rendering of only the extracted predicate
* explain may render a normalized planner-owned form, but that rendering must:
  * preserve the semantic identity of `filter_expr`
  * stay understandable as a user-facing filter explanation rather than an
    implementation-only artifact
* explain must not silently substitute a pushdown-only rendering when the
  semantic filter and extracted predicate differ
* explain must not omit filter staging solely because no derived pushdown
  predicate exists

---

## Identity and stability contract

The semantic filter expression, `filter_expr`, is the identity anchor.

Implications:

* explain output should reflect it directly
* future cache and fingerprint surfaces may depend on it
* `predicate` and `residual_filter_expr` must not become competing semantic
  owners

The split is:

* semantic identity: `filter_expr`
* pushdown identity: `predicate`
* execution-stage identity: `residual_filter_expr`

That separation must remain stable through planning and execution.

---

## Prepared-execution boundary

`0.103` does not widen prepared execution beyond the existing `0.102` line.

Explicitly:

* fixed-route templates continue to cover predicate-owned shapes only
* residual-heavy scalar queries remain fallback-prepared
* residual filters do not enter the fixed-route template lane in this slice

This patch is about making the scalar triad explicit, not about widening the
prepared symbolic-template contract.

---

## Likely code seams

The implementation will likely touch:

* scalar plan model
  * add or stabilize explicit `residual_filter_expr`
* predicate extraction
  * produce both `predicate` and residual execution ownership artifacts
* scalar executor
  * enforce `predicate -> residual -> projection` ordering explicitly
* compiled expression engine
  * ensure shared reuse for residual filtering
* explain surfaces
  * render semantic / pushdown / residual distinctly
* session and prepared seams
  * preserve the triad across planning to execution handoff

---

## Risks

### Risk 1: duplicated evaluation logic

If predicate and expression evaluation drift further apart instead of reading as
deliberately tiered roles, correctness risk grows.

### Risk 2: incorrect residual derivation

Residual execution must be semantically exact after extracted pushdown. A
planner shortcut that treats one sound prefilter as if it were a full semantic
remainder would be incorrect.

### Risk 3: unclear explain output

If explain hides residual work or collapses semantic and extracted filter
ownership back together, users will misread both behavior and performance.

---

## Verification matrix

`0.103` should be verified across these boundaries:

| Area | Required proof |
| --- | --- |
| Planning | `filter_expr`, `predicate`, and `residual_filter_expr` are all present or absent consistently |
| Derivation | extracted `predicate` plus residual execution preserves original scalar filter semantics |
| Execution | residual filters are applied correctly after access and before projection |
| Explain | semantic / pushdown / residual split renders correctly across scalar query shapes |
| Identity | semantic filter identity remains stable across planning, explain, and future cache/fingerprint surfaces |
| Prepared | fallback vs template behavior remains unchanged |
| Negative | pure predicate-owned queries do not invent residual work unnecessarily |

---

## Acceptance criteria

`0.103` is complete when:

* scalar plans always carry the explicit semantic/pushdown/residual triad
* residual filtering is a first-class execution step
* explain clearly shows the semantic/pushdown/residual split
* predicate and expression evaluation roles are clearly separated
* no semantic filter information is lost between planning and execution
* pure predicate-owned scalar queries do not synthesize residual artifacts in
  planning or explain

---

## Thesis sentence

`0.103` completes the scalar `WHERE` execution model by making semantic filter
expression, extracted pushdown predicate, and residual post-access filtering
explicit, consistent, and visible across planning, execution, and explain.
