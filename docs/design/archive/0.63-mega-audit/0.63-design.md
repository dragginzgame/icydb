# 0.63 Design

**Status:** Planned for `0.63`
**Theme:** Post–de-monomorphization consolidation audit

---

## Intent

`0.63` is a **consolidation release**, not a new architectural phase.

It follows the wasm-size-driven runtime rewrite completed across `0.58` → `0.62`. The objective is to **stabilize and simplify** the system now that the major per-entity SQL/delete shells have been removed.

This release focuses on:

* eliminating residual accidental complexity
* collapsing duplicate or drifted logic
* clarifying ownership and authority boundaries
* reducing generated and internal surface area
* preserving (or improving) the current wasm baseline

This is **hygiene and structural integrity**, not feature work.

---

## Current Baseline

From `0.62.11` (simple fixture):

| Metric             | Value     |
| ------------------ | --------- |
| `one_simple` wasm  | 1,388,301 |
| `ten_simple` wasm  | 1,403,325 |
| delta (9 entities) | 15,024    |
| per-entity slope   | ~1,669    |
| `one_simple` gzip  | 506,611   |
| `ten_simple` gzip  | 508,817   |
| gzip delta         | 2,206     |
| gzip slope         | ~245      |

### Interpretation

* Large-scale duplication (typed SQL/delete shells) is already removed
* Remaining costs are:

  * smaller
  * flatter
  * noisier
* Further gains require **measurement-driven local improvements**, not sweeping changes

---

## Post–0.62 State

**What is already true:**

* Runtime execution is largely **structuralized**
* `E`-driven duplication in SQL/delete paths is removed
* Generated SQL dispatch is thinner and authority-driven
* Metadata authority is centralized (models + shared traits)
* Generated per-entity surfaces are significantly reduced

**What remains:**

* Residual duplication from incremental refactors
* Parallel or drifted execution paths
* Historical module boundaries
* Leftover convenience surfaces (generated + tests)
* Local inefficiencies contributing to wasm noise

---

## Core Objectives

### 1. Code Hygiene

* remove dead code and scaffolding
* eliminate stale or unreachable paths
* normalize naming and ownership boundaries

---

### 2. Redundancy Elimination

Target:

* duplicate helpers
* parallel implementations
* no-op forwarding layers
* repeated invariant enforcement

Rules:

* remove accidental duplication
* preserve **intentional (protective) duplication**, but mark it explicitly
* collapse drift into a single canonical implementation

---

### 3. Authority Clarification

Each concern must have **one authoritative owner**.

Eliminate:

* shadow metadata
* duplicated authority surfaces
* alternate construction paths that rebuild identical facts

---

### 4. Generated Surface Minimization

Continue reducing:

* per-entity emitted behavior
* duplicated impl blocks

Prefer:

* shared traits
* shared runtime contracts
* generated data (tables) over generated logic

---

### 5. Measured Size Discipline

All changes must respect:

* `one_simple`
* `ten_simple`
* entity slope

Reject:

* changes that increase emitted output without measurable benefit
* changes that only move symbols without reducing baseline size

---

## Non-Goals

* no new runtime architecture phase
* no reintroduction of generics into hot execution paths
* no expansion of type-erasure beyond current boundaries
* no aesthetic-only cleanup
* no rollback of `0.62.11` structural SQL/delete work

---

## Audit Framework

All work in `0.63` is **audit-driven**, not opportunistic.

Each audit produces:

* classification
* action
* risk
* verification plan

---

### 1. DRY / Consolidation Audit

Classify duplication as:

* **protective duplication** (keep)
* **validation vs execution separation** (keep)
* **boundary vs runtime duplication** (contextual)
* **accidental duplication** (remove)
* **same-layer repetition** (collapse)
* **evolution drift** (unify)
* **diverged implementations** (canonicalize)

---

### 2. Complexity Accretion Audit

Identify:

* unnecessary indirection
* abstraction layers without payoff
* over- or under-factored modules

Actions:

* flatten trivial layers
* remove unused abstraction points
* simplify control flow

---

### 3. Module Structure Audit

Ensure:

* modules map to real ownership boundaries
* no mixed concerns in a single owner
* naming reflects responsibility

Actions:

* relocate code to correct owners
* split mixed modules
* merge artificially fragmented ones

---

### 4. Layer Violation Audit

Ensure:

* execution does not depend on derive/build layers
* shared runtime is isolated from typed boundaries
* generated code does not become hidden authority

Actions:

* remove cross-layer leakage
* reassert subsystem boundaries

---

### 5. Velocity Preservation Audit

Ensure no regression in:

* compile time
* readability of hot paths
* iteration speed

---

## Workstreams

### 1. Runtime Consolidation

Primary targets:

* aggregate kernels
* projection / materialization ownership
* scan / stream helpers
* planner / intent seams

Focus:

* eliminate duplicate traversal and fold logic
* unify structurally identical execution paths
* keep kernels **monomorphic and explicit**

---

### 2. Generator / Derive Cleanup

Remove:

* per-type convenience methods
* duplicate metadata surfaces
* redundant impl emission

Replace with:

* shared traits
* minimal metadata contracts
* generated data instead of behavior

---

### 3. Test & Fixture Cleanup

* remove reliance on deleted conveniences
* align tests with real runtime contracts
* deduplicate fixture helpers and authority definitions

---

### 4. API Surface Tightening

Remove:

* redundant public methods
* duplicate constructors
* stale ergonomic layers

Ensure:

* public API reflects **true authority**, not historical convenience

---

### 5. Error & Validation Consolidation

* unify error construction paths
* deduplicate validation logic
* ensure consistent classification across runtime and API surfaces

---

## Acceptance Criteria

### Hygiene

* no duplicate authority surfaces in targeted areas
* no clearly unused abstractions
* no unjustified parallel implementations

---

### Structural Integrity

* clearer ownership boundaries post-change
* no new layer violations

---

### Generated Output

* emitted surface is reduced or flat
* no new per-entity behavioral duplication

---

### Wasm Metrics

* `one_simple` stable or improved
* `ten_simple` stable or improved
* per-entity slope stable or improved

---

### Tests

* schema, relation, validation, SQL dispatch all pass
* tests reflect **actual kept contracts**

---

## Measurement Strategy

Every change validated against:

* wasm reports:

  * `one_simple`
  * `ten_simple`
* `twiggy` (or equivalent) when slope source is unclear
* compile success across:

  * `icydb-core`
  * `icydb`
  * `icydb-build`
  * canister fixtures
* contract-level tests:

  * schema
  * relation
  * validation
  * SQL dispatch
  * conflict classification parity

---

## Guiding Principles

* one authority per concern
* no duplicate flows without justification
* prefer **removal over abstraction**
* prefer **traits over generated behavior**
* explicit over implicit
* measure, don’t assume
* favor **verified wins** over cleanup optimism

---

## Release Framing

`0.63` is the **post-rewrite consolidation milestone**:

* clean the system
* remove redundancy
* clarify ownership
* protect `0.62.11` gains
* prepare the codebase for the next major phase (features or storage)
