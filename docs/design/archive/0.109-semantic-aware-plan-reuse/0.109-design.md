0.109 Semantic-Aware Plan Reuse

Thesis

`0.109` should make canonical semantic query identity operationally valuable.

After `0.108`, semantically identical admitted queries can already share the
same canonical identity surface. `0.109` follows through by reusing planner
and execution-preparation artifacts when that identity matches exactly,
instead of recomputing the same work for spelling-equivalent queries.

This is a reuse slice, not a new canonicalization, optimizer, or SQL-surface
slice.

Problem statement

After `0.105`, `0.107`, and `0.108`:

* extracted and residual filter structure is stable
* bounded semantic canonicalization can collapse some equivalent boolean forms
* fingerprint and semantic cache identity can align to canonical query shape

However, identical canonical identity is not yet fully exploited unless the
engine actually reuses the artifacts built from that identity.

That means semantically identical admitted queries may still:

* repeat lowering and plan construction work
* rebuild the same access plan
* recompile the same residual execution program
* duplicate execution-preparation work even when identity already matches

Concrete examples include:

* `a = 1 AND b = 2`
* `b = 2 AND a = 1`
* `CASE WHEN a THEN TRUE ELSE b END`
* `(a AND TRUE) OR (NOT a AND b)`

After `0.108`, these can already share semantic identity. `0.109` makes that
identity the authority for actual artifact reuse.

Design goal

Reuse planner-owned artifacts whenever canonical semantic query identity
matches exactly and the same planning path remains valid.

That means:

* identical canonical query identity can reuse the same compiled plan artifact
* reuse is exact-match and semantic-safe, not fuzzy or opportunistic
* prepared and non-prepared paths can share the same semantic reuse base where
  they already share planning behavior
* explain and diagnostics can report reuse clearly enough to verify behavior

Non-goals

`0.109` does not aim to:

* broaden SQL admission
* introduce new semantic canonicalization rules
* change route ranking or cost heuristics
* add partial-plan caching
* add fuzzy plan matching
* add cross-binding prepared reuse guarantees beyond existing planning-path
  equivalence
* add statistics, adaptive planning, or runtime feedback
* widen prepared template eligibility
* change query semantics

This is not a new optimization class. It is the plan-reuse follow-through
slice for canonical identity work already done in `0.108`.

Normative model

Canonical semantic query identity

For the purposes of `0.109`, the reuse key must be the full canonical semantic
query identity already defined by `0.108`:

* canonicalized scalar filter expression (`filter_expr`)
* projection shape
* grouping shape, if any
* ordering shape as declared semantically by the user, not planner-added
  tie-break terms
* `LIMIT` / `OFFSET` semantics

`0.109` primarily improves reuse for filter-driven divergence, but it must not
key reuse on the filter alone.

Rule 1 - reuse is exact-match on canonical semantic identity

Plan reuse is allowed only when canonical semantic query identity matches
exactly.

This means:

* same canonical filter form alone is insufficient
* reuse must not cross differing projection shapes
* reuse must not cross differing grouping shapes
* reuse must not cross differing ordering semantics
* reuse must not cross differing `LIMIT` / `OFFSET` semantics

`0.109` does not authorize:

* partial reuse
* approximate reuse
* close-enough reuse
* reuse across semantically different admitted queries

Rule 2 - reuse artifacts are explicit and bounded

The normative artifact stack for `0.109` is:

* logical plan artifact
* access-plan artifact
* execution-preparation artifact
* compiled residual execution program, if any

This slice should explicitly state which of these ship in `0.109.0` and which
remain deferred follow-ups. The key requirement is that shipped reuse surfaces
are planner-owned and derived from canonical semantic identity.

`0.109` does not introduce:

* row-result caching
* partial subtree caching
* ad hoc executor memoization outside the canonical reuse boundary

Rule 3 - reuse follows the same planning path only

Reuse is valid only when the planning path and planner-visible environment
needed to produce the artifact remain compatible.

That means reuse must not cross:

* schema or index-visibility changes that would alter planning inputs
* planner contract changes that invalidate artifact interpretation
* execution-mode changes that require different compiled preparation

`0.109` is about semantic-aware reuse of the same planning path, not reuse
across planner modes that happen to share some query text.

Rule 4 - invalidation is explicit and fail-closed

Any change that can alter the correctness or interpretation of a reused
artifact must invalidate reuse.

At minimum, `0.109` must invalidate reuse on:

* schema shape changes relevant to the planned entity
* visible index changes relevant to route selection
* planner contract or engine-version changes that alter artifact meaning

If invalidation status is unclear, the engine must miss reuse and rebuild the
artifact rather than reusing speculatively.

Rule 5 - prepared and non-prepared reuse share the same semantic base only
when their planning path is already identical

Prepared and non-prepared queries may benefit from the same canonical semantic
plan-reuse substrate only where they already share the same planning path.

This slice does not:

* introduce new prepared template reuse guarantees
* promise cross-binding plan reuse beyond current planning equivalence
* merge prepared-template identity with fallback identity

Prepared fallback may reuse canonical semantic identity only where it already
lands on the same planner-owned artifact path.

Rule 6 - explain and diagnostics must expose reuse directly

Explain and diagnostics must make reuse visible enough to verify that
canonical identity is being exploited rather than only computed.

At minimum, the engine should be able to surface:

* reuse hit or miss
* reused artifact class
* invalidation or miss reason, when practical

This is especially important for `EXPLAIN EXECUTION VERBOSE`, because `0.109`
should be auditable through the same planner-facing surfaces that `0.103`,
`0.106`, and `0.108` already sharpened.

Rule 7 - reuse is semantic-safe, not opportunistic

The guiding invariant for `0.109` is:

* reuse only when canonical semantic identity and planning-path validity both
  match exactly

The engine must prefer:

* missing reuse and rebuilding

over:

* reusing an artifact across uncertain semantic or planning boundaries

This keeps `0.109` conservative and predictable.

Examples

Equivalent boolean spelling should reuse one plan artifact

These should share canonical identity and reuse the same shipped plan artifact:

```sql
SELECT name
FROM character
WHERE a = 1 AND b = 2
```

```sql
SELECT name
FROM character
WHERE b = 2 AND a = 1
```

Expected result:

* identical canonical semantic identity
* identical fingerprint
* identical semantic reuse key
* same reused shipped artifact, if still valid

Equivalent searched `CASE` should reuse one plan artifact

These should share canonical identity after `0.107` and reuse the same shipped
artifact:

```sql
SELECT name
FROM character
WHERE CASE WHEN a THEN TRUE ELSE b END
```

```sql
SELECT name
FROM character
WHERE (a AND TRUE) OR (NOT a AND b)
```

Expected result:

* identical canonical semantic identity
* identical reuse key
* same reused shipped artifact, if still valid

Different projection must not reuse

These must remain distinct:

```sql
SELECT name
FROM character
WHERE a = 1 AND b = 2
```

```sql
SELECT age
FROM character
WHERE b = 2 AND a = 1
```

Expected result:

* same canonical filter component
* different canonical semantic query identity
* no shared plan reuse

Different ordering must not reuse

These must remain distinct:

```sql
SELECT name
FROM character
WHERE a = 1 AND b = 2
ORDER BY name ASC
LIMIT 10
```

```sql
SELECT name
FROM character
WHERE b = 2 AND a = 1
ORDER BY age ASC
LIMIT 10
```

Expected result:

* same canonical filter component
* different canonical semantic query identity
* no shared plan reuse

Verification matrix

`0.109` should add targeted coverage for:

* equivalent reordered boolean filters hit the same reuse key
* bounded searched `CASE` canonicalization hits the same reuse key
* differing projection shapes miss reuse
* differing grouping shapes miss reuse
* differing ordering shapes miss reuse
* differing `LIMIT` / `OFFSET` semantics miss reuse
* invalidation on schema/index visibility change fails closed
* prepared fallback reuses only when it already shares the same planning path
* explain / verbose diagnostics report hit vs miss consistently
* reused residual program matches the same canonical identity as the reused
  plan artifact

Acceptance criteria

`0.109` is complete when:

* canonical semantic identity can drive reuse of the shipped artifact set
* reuse occurs only on exact canonical semantic identity matches
* invalidation is explicit and fail-closed
* prepared and non-prepared paths do not gain accidental new reuse promises
* explain or diagnostics can demonstrate reuse behavior directly
* no SQL-surface widening or new equivalence-class work is required

Open questions to resolve before implementation starts

The implementation doc should lock these down before coding:

* which artifact classes ship in `0.109.0`
* cache scope and lifetime:
  * per-session
  * per-canister in-memory
  * bounded size / eviction policy
* exact invalidation inputs and versioning strategy
* exact explain / verbose fields used to surface reuse
