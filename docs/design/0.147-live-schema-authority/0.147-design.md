# IcyDB 0.147 - Live Schema Authority

## Status

Proposed.

0.147 continues the accepted-schema authority work from 0.146. The 0.146 line
made accepted persisted schema metadata authoritative for inspection, SQL
admission, planning type checks, cache identity, and diagnostics. It did not
make accepted schema the runtime row decode/write authority, and it did not
introduce schema evolution policy.

0.147 is the next correctness boundary:

```text
accepted schema metadata
  -> explicit transition policy
  -> accepted live row-layout authority
  -> runtime decode/write contracts
```

## Motivation

0.146 deliberately stopped before changing row decode/write authority. Runtime
data rows still decode through generated static layout metadata, even though
the accepted schema snapshot is now the metadata and planning authority. This
was the right split: accepted metadata can be introduced safely without
pretending schema evolution is implemented.

The remaining split-brain risk is clear:

* accepted schema knows durable field IDs, top-level slots, field kinds, and
  nested leaf metadata
* runtime row decode and writes still trust the generated model layout
* schema drift is rejected before row decode, but accepted schema is not yet
  the thing that actually drives row materialization

0.147 closes that gap.

## Goals

* Define the first explicit schema transition policy beyond exact equality.
* Make accepted row-layout metadata the authority for runtime row decode and
  write field-slot contracts where safe.
* Keep fail-closed startup behavior for unsupported schema changes.
* Preserve the current slot-framed row format.
* Keep SQL and fluent planning aligned with accepted schema metadata.
* Keep diagnostics concise enough for local upgrade testing.

## Non-Goals

* Do not add per-row schema versions.
* Do not support multiple live row formats at runtime.
* Do not silently rewrite stored rows during normal reads.
* Do not make Rust struct `Default` values database defaults.
* Do not allow generated metadata to override an accepted persisted schema.
* Do not implement broad migration tooling in the first patch.

## Design Principles

### Accepted Schema Is the Runtime Contract

Generated `EntityModel` remains the compiled proposal. It is used to describe
what the new code wants the schema to be. It must not override persisted schema
metadata after a schema snapshot has been accepted.

Accepted schema owns:

* durable field identity
* live top-level row slots
* persisted field kind
* database-default metadata
* nested leaf metadata
* SQL capability projection

Runtime decode/write code must consume accepted-schema artifacts rather than
re-deriving slot or field-kind facts from generated fields.

### Transition Policy Comes Before Runtime Remapping

Runtime remapping is only safe after the schema transition has been accepted.

The transition owner must answer:

```text
stored accepted snapshot + generated proposal -> transition decision
```

The executor must not decide whether a schema change is safe by looking at row
bytes or generated field tables.

### No Compatibility Branches Before 1.0

The project is still pre-1.0. Internal schema metadata can hard-cut to the
latest format. Do not keep parallel v1/v2 schema decoders or migration
fallbacks for abandoned internal shapes.

## Transition Policy

### Initial Supported Transitions

The first 0.147 transition policy should stay conservative.

Candidate safe transitions:

* no-op exact match
* additive nullable field with explicit database default
* additive optional field whose persisted absence is representable
* metadata-only additions that do not require decoding existing row bytes

Candidate rejected transitions:

* field removal without retained-slot policy
* field type/kind change
* primary-key field change
* slot reorder without explicit remap
* nested leaf type change
* relation target change
* default change that would alter old-row materialization semantics

The first implementation should prefer rejecting too much over accepting a
transition whose runtime semantics are unclear.

### Transition Decision Shape

Extend the existing schema-owned transition artifact rather than adding a new
side channel:

```rust
enum SchemaTransitionDecision {
    Accepted(SchemaTransitionPlan),
    Rejected(SchemaTransitionRejection),
}
```

`SchemaTransitionPlan` should be explicit and closed. Do not expose loose
booleans such as `has_added_fields` or `layout_changed` to runtime code.
The exact-match path is represented as the first accepted plan kind, not as a
separate decision variant, so future accepted transitions extend the same
artifact.

## Live Row Layout Authority

### Decode Boundary

Row decode currently assumes generated field order and generated slot count.
0.147 should introduce a decode contract that can use accepted row layout:

```text
RawRow
  -> accepted row layout
  -> slot reader / row view
  -> typed or structural materialization
```

The accepted layout must provide:

* required physical slot count
* durable field ID to physical slot mapping
* field kind and leaf codec for payload decoding
* absence/default policy for additive fields

### Write Boundary

Writes must use accepted schema for:

* literal normalization
* field admission
* target slot selection
* default synthesis where explicitly authorized
* encoded row shape validation

Generated models may still provide typed Rust construction and proposal
metadata, but accepted schema must decide the persisted write contract.

## Retired and Added Fields

### Added Fields

Added fields require an explicit database absence policy.

Accepted options:

* synthesize a database default
* materialize `NULL` for nullable fields
* reject until a migration is declared

Rust struct defaults do not count.

### Retired Fields

Removing a logical field does not immediately free its physical slot.

Retired slots must remain decode-visible until a rewrite/migration policy
removes them. SQL and public metadata should hide retired logical fields, but
row decode must keep enough information to skip or preserve their slots safely.

## SQL and Fluent Boundaries

SQL and fluent planning already consume accepted schema metadata in 0.146.
0.147 must preserve that direction:

```text
accepted schema -> SchemaInfo -> query planning -> prepared execution
```

Do not reintroduce generated-model lookups for:

* top-level field kind
* top-level slot
* nested leaf metadata
* SQL capability checks
* cache identity

## Diagnostics

Schema-change errors must remain concise and actionable.

Transition rejections should report:

* entity path
* transition category
* first changed field or layout fact
* field name and kind when available
* whether the change is unsupported or malformed

They should not dump whole schema snapshots, row layouts, or raw persisted
payloads.

Runtime metrics should keep transition-policy decisions separate from broader
schema reconciliation outcomes. Reconciliation metrics may include first-create
writes, corrupt stores, and store-write failures; transition metrics should
only count accepted/rejected policy decisions for existing accepted snapshots.

## Implementation Plan

1. Extend transition decisions with accepted transition plans.
2. Define a conservative additive-field policy.
3. Add accepted layout runtime descriptors for row decode/write boundaries.
4. Route decode/write slot selection through accepted layout descriptors.
5. Keep generated-model fallback only where no persisted schema exists.
6. Add tests for rejected drift, accepted additive changes, old-row reads, and
   writes after accepted transition.
7. Remove temporary generated-layout fallbacks once accepted layout authority is
   complete.

## Validation

Required checks:

```bash
cargo fmt --all
cargo test -p icydb-core db::schema --features sql -- --nocapture
cargo test -p icydb-core db::commit --features sql -- --nocapture
cargo test -p icydb-core db::session --features sql -- --nocapture
cargo check -p icydb-core --tests --features sql
cargo clippy -p icydb-core --all-targets --features sql -- -D warnings
make check-invariants
git diff --check
```

Focused behavior tests:

* upgrade with exact schema match
* upgrade with unsupported removed field
* upgrade with unsupported field kind change
* upgrade with additive nullable/defaulted field
* read old rows after accepted additive transition
* write new rows after accepted additive transition
* `DESCRIBE` / `SHOW COLUMNS` after accepted transition
* SQL SELECT/INSERT/UPDATE/DELETE after accepted transition

## Expected Outcome

After 0.147, accepted schema should no longer be only a metadata/planning
authority. It should also be the runtime row-layout authority for supported
transitions.

Unsupported schema changes should still fail closed, but supported additive
changes should have one explicit path from schema reconciliation through
runtime decode/write behavior.
