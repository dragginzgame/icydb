# 0.203 Legacy Architecture Hard Cut

Status: proposed implementation contract; repository-wide audit complete;
implementation not started.

0.203 converts the 2026-07-13 hard-cut legacy architecture audit into the
normative excavation plan for IcyDB's surviving query, admission, planning,
execution, response, diagnostics, metrics, test, and documentation surfaces.

This design does not invent another query architecture. It removes public and
internal concepts that survived earlier pivots after their ownership or
product role disappeared. The target is one legible current architecture,
with one authoritative representation of each semantic fact at each stage.

The evidence baseline is
[the hard-cut legacy architecture audit](../../reports/investigations/2026/07/13/hard-cut-legacy-architecture/01/report.md).
That report remains the archaeological record: it contains the reachability
proof, exact producer/consumer searches, test classifications, deliberate
retention analysis, and the validation readout from the audited worktree.
This document owns the implementation decisions and closeout requirements.

The slices in this document are review and landing units, not independent
compatibility phases and not automatically patch releases. A removed name or
path is removed in the slice that establishes its replacement authority. No
slice may leave an alias, shim, dual dispatch, deprecated wrapper, legacy
fallback, or anti-resurrection test behind.

## Context

IcyDB has already converged substantially:

- typed/fluent and reduced SQL reads lower to `StructuralQuery`;
- the shared query-plan cache produces `AccessPlannedQuery` and
  `SharedPreparedExecutionPlan`;
- `ExecutionRoutePlan` owns executable route selection;
- accepted schema snapshots own runtime schema and row-shape authority;
- ordinary fluent reads use built-in fail-closed admission;
- cursor continuation uses one bounded, version-gated binary contract;
- the executor consumes one effective residual-filter program;
- current persisted formats reject unsupported versions instead of trying
  legacy decoders.

The remaining risk is not a second complete production engine. It is sediment
around that engine: public shells with no current developer role, test-only
architectures that resemble supported products, diagnostics that reconstruct
planner facts, metrics for a forbidden fallback, compatibility names, and
active documentation that describes superseded designs.

The production spine can therefore look singular while repository structure
still teaches contributors that several owners or modes exist. New work can
then attach to the wrong shell, preserve a nonexistent lane, or make
diagnostics disagree with execution.

## Problem Statement

The audited repository has twelve proven hard-cut findings:

1. trusted SQL authority is encoded by documentation and generated caller
   convention rather than the library method boundary;
2. a complete public-SQL admission system survives only under `cfg(test)`;
3. internal admission-policy DTOs are exported through the application
   facade without a public use case;
4. `PlannedQuery` and `CompiledQuery` expose two names for one logical plan;
5. a hidden direct facade `Query` and raw-query session wrappers have no
   remaining repository caller;
6. dead projection responses and explicit SQL compatibility aliases coexist
   with the current row payload;
7. typed access-choice facts are rendered to strings and reparsed for
   structured diagnostics;
8. grouped diagnostics call the only supported grouped route a fallback and
   retain unreachable rejection state;
9. metrics expose a generated-model fallback that production cannot take;
10. the CLI retains an obsolete alternate spelling;
11. compile-fail suites preserve removed syntax and paths as negative
    contracts;
12. active docs and unqualified `v1` terminology describe old cursor, SQL,
    admission, durability, and grouped-planning designs.

None requires a compatibility period. The repository has no external user
contract that justifies preserving these surfaces before 1.0, and no finding
requires a persisted-format migration.

## Design Goal

After 0.203, a contributor can trace a query without knowing IcyDB's prior
pivots:

```text
typed/fluent builder ----> Query intent ----------------+
                                                        |
SQL parse/lowering -----> StructuralQuery -------------+
                                                        |
                                                        v
                                  accepted schema + visible indexes
                                                        |
                                                        v
                              AccessPlannedQuery / shared plan cache
                                                        |
                                                        v
                                  SharedPreparedExecutionPlan
                                                        |
                                                        v
                                  ExecutionRoutePlan + executor
                                                        |
                                                        v
                               one current response/diagnostic projection
```

The end state has:

- one ordinary typed/fluent public construction system;
- one production SQL compile/execute system;
- convergence at `StructuralQuery` and shared prepared planning;
- one owner of evaluated public-read admission;
- an explicit trusted SQL boundary and an independently explicit generated
  controller gate;
- one logical plan type and one executor-ready plan handoff;
- one public projected-row payload;
- typed planner and route truth carried to diagnostics without reparsing;
- no metrics, tests, docs, or aliases advertising removed architectures;
- current durable version gates retained exactly where they protect data.

## Hard-Cut Rules

0.203 is governed by these rules:

1. A removed public name receives no alias, deprecated wrapper, migration
   trait, transitional module, or second spelling.
2. Tests prove the maintained current surface. They do not compile retired
   syntax merely to prove that it remains retired.
3. A typed planner or route fact remains typed until the final rendering
   boundary. Text is not an internal identity protocol.
4. A bypass is visible in the method or lane selected by the caller. Ambient
   convention is not sufficient authority.
5. A test-only product architecture is deleted when production cannot enter
   it. Valid semantic facts are moved to the current owning tests only when
   coverage would otherwise be lost.
6. Accepted schema remains runtime authority. Generated models may support
   proposal, reconciliation, model-only convenience, and tests, but no
   runtime fallback.
7. Current persisted formats remain bounded, versioned, and fail-closed.
   Source/API hard cuts do not authorize speculative durable-format removal.
8. Diagnostics consume the same frozen facts as execution whenever
   practical; they do not independently re-plan or infer route identity.
9. Alternate physical routes remain only when they express real semantics,
   capability, ordering, or performance boundaries.
10. Each implementation slice re-runs reachability against the then-current
    worktree before deletion and stops if a new production consumer is found.

## Non-Goals

0.203 does not include:

- a new query language, query builder, planner, executor, or policy
  framework;
- broad planner or executor refactoring unrelated to a proven authority
  conflict;
- application-configurable read policy;
- public caller-controlled SQL;
- a generalized capability-token framework for trusted calls;
- changes to SQL semantics, fluent query semantics, result cardinality,
  ordering, cursor behavior, or response wire values;
- removal of trusted/admin SQL `OFFSET` support;
- removal of current cursor, row, commit, journal, schema, index, or database
  format versions;
- migration or decoding support for impossible historical formats;
- removal of generated-model proposal, reconciliation, or test utilities;
- cosmetic renaming of durable `v1` identities that still name a current
  wire or persisted contract;
- merging routes solely because their implementations look similar;
- preservation of any audited name for hypothetical future consumers.

## Evidence Classification

The audit classifications become binding 0.203 dispositions:

- **A — Delete now:** unreachable, compatibility-only, obsolete, or contrary
  to the current design;
- **B — Collapse into current authority:** behavior remains valid, but its
  fact or decision has more than one owner;
- **C — Privatise:** the concept remains useful only inside its owning layer;
- **D — Rename or re-document:** behavior is current but its vocabulary
  teaches an obsolete architecture;
- **E — Retain deliberately:** historical-looking code protects a current
  invariant, semantic route, performance boundary, or durable format;
- **F — Requires design decision:** reserved for genuine unresolved product
  choices.

The audit found no Category F blocker. 0.203 can proceed without inventing a
new product decision.

## Executive assessment

Architectural sediment is **high**, but not critical.

The production read spine is substantially more coherent than the repository's
surface area suggests. Typed/fluent and reduced SQL reads both lower to
`StructuralQuery`, use the shared query-plan cache, produce an
`AccessPlannedQuery`/`SharedPreparedExecutionPlan`, and execute through the same
route planner and executor contracts. Accepted schema snapshots are the runtime
authority. Cursor decoding is one bounded, version-gated implementation. The
executor has one effective residual-filter contract and one route-selection
artifact.

The surviving old design families are:

- a public-but-hidden direct `Query<E>` surface beside the fluent facade;
- two inert `PlannedQuery<E>` / `CompiledQuery<E>` stages over the same logical
  plan;
- a test-only public-SQL admission architecture even though production SQL is
  trusted/controller-only;
- public admission DTOs that expose an internal policy vocabulary;
- an obsolete projection-response facade and explicit SQL compatibility aliases;
- diagnostics that stringify typed planner facts and reconstruct them later;
- grouped-route observability that describes the sole supported route as a
  fallback and retains unreachable rejection state;
- metrics for a removed generated-model runtime fallback;
- compatibility spellings and compile-fail tests whose only contract is keeping
  deleted names deleted;
- active documentation and `v1` terminology that describe superseded route,
  cursor, and policy structures.

Most of this sediment is structural or diagnostic. Two areas still affect the
meaning of live runtime surfaces:

1. `DbSession::execute_sql_query` is trusted/admin only by documentation and
   generated-call-site convention; the method itself carries no explicit trusted
   marker, capability, or lane.
2. access-choice and grouped-route diagnostics can present reconstructed or
   fictitious route facts instead of the exact typed facts used by execution.

The highest-risk authority conflict is therefore the SQL trust boundary,
followed by diagnostic route identity. No evidence showed a currently reachable
public-read admission bypass in generated code: generated `icydb_query` calls
`icydb_sql_surface_require_controller("query")` before entering the trusted SQL
helper. The risk is that a library caller can attach a new endpoint directly to
the ambiently trusted helper, and that future work can mistakenly treat the
test-only public-SQL lane as a supported production architecture.

IcyDB is safe to continue extending accepted-schema, storage, and already
canonical typed/fluent internals while these findings are open. It is **not safe
to extend public SQL, admission-lane configuration, direct-query APIs, projection
facades, or route diagnostics before the corresponding authority cleanup**.

No production behavior was changed by this audit.

6. Unit, integration, generated-code, compile-fail, and snapshot tests.
7. Targeted Git history for conflicting or compatibility-shaped surfaces.

The audit began and ended at commit

- constructor/producer and consumer searches;

## Current architecture map

### Surviving read and query spine

The current architecture is one shared semantic spine with two frontends:

```text
typed/fluent builder ----> Query<E> / StructuralQuery ----+
                                                         |
SQL parse + lowering ---> CompiledSqlCommand ------------+--> StructuralQuery
                                                               |
                                                               v
                                      accepted schema + visible indexes
                                                               |
                                                               v
                              shared query-plan cache / AccessPlannedQuery
                                                               |
                                                               v
                                  SharedPreparedExecutionPlan
                                                               |
                                                               v
                             ExecutionRoutePlan + executor contracts
                                                               |
                                                               v
                              terminal/result-specific response shaping
```

The SQL compiled-command cache is not a second semantic planner. It caches SQL
frontend work and may retain the same shared prepared plan. Its identity is
guarded by accepted schema revision/fingerprint and the structural query's
cache identity.

### Frontend traces

#### Typed/fluent read

1. `icydb::db::DbSession::load::<E>()` constructs a facade
   `FluentLoadQuery<'_, E>` over the core fluent builder.
2. Builder operations mutate the typed `Query<E>` / `StructuralQuery` intent;
   semantic terminals such as `page`, `try_one`, `collect_complete`, and exact
   aggregates declare result intent.
3. Ordinary execution applies `QueryAdmissionPolicy::default_bounded_read()`;
   trusted maintenance code must opt into `trusted_read_unchecked()`.
4. `DbSession::cached_shared_query_plan_for_entity` resolves accepted schema and
   visible indexes, builds or reuses the `AccessPlannedQuery`, and returns a
   `SharedPreparedExecutionPlan`.
5. The executor route planner freezes one route result. Terminal code applies the
   executor-owned residual program, order/window/cursor contract, and result
   shaping.

#### SQL read

1. `execute_sql_query::<E>` parses and surface-validates reduced SQL, loads the
   accepted catalog context, and builds a `SqlCompiledCommandExecutionContext`.
2. Scalar/grouped `SELECT` and global aggregates carry or produce a
   `StructuralQuery`; SQL projection labels remain a frontend/result concern.
3. `select_plan` / `aggregate_plan` use the same shared prepared-plan machinery
   and `EntityAuthority` used by fluent execution.
4. Generated `icydb_query` is controller-gated before it calls the trusted SQL
   helper. The library method itself does not encode that authority (HCLA-001).

SQL and fluent therefore converge at `StructuralQuery` and the shared prepared
plan, not at parsing and not only inside the executor.

## Normative Decisions

### Trusted SQL is explicit at the method boundary

The facade and core trusted read entry become
`execute_trusted_sql_query`. The attribution-bearing variant becomes
`execute_trusted_sql_query_with_perf_attribution`. The old
`execute_sql_query` names are deleted in the same slice and receive no
aliases.

Generated `icydb_query` continues to call
`icydb_sql_surface_require_controller("query")` before entering the trusted
helper. The method name communicates trusted execution; the generated gate
continues to own endpoint authorization. 0.203 does not replace either fact
with a generic capability abstraction.

`QueryAdmissionLane::AdminAdHoc`, `QueryAdmissionLane::DevTest`, and
`QueryAdmissionPolicy::admin_ad_hoc` are deleted because production does not
select them. The admission taxonomy describes evaluated decisions, not
ambient bypasses.

SQL parsing, lowering, accepted-schema lookup, shared planning, execution,
response construction, diagnostics, and performance attribution remain
semantically unchanged.

### Public-read admission has one live frontend

The `cfg(test)` public-SQL admission path is deleted in full:

- `execute_sql_query_with_read_admission_policy`;
- `execute_compiled_sql_context_with_read_admission_policy`;
- `execute_global_aggregate_compiled_statement_ref_with_read_admission_policy`;
- their select dispatch and hooks;
- `db/session/sql/execute/select/read_budget.rs`;
- tests whose product contract is executing SQL as `PublicRead`.

Ordinary public-read admission remains an internal policy applied by fluent
semantic terminals. SQL tests continue to prove parsing, lowering, type
checking, shared planning, execution, and error parity, but do not attach a
nonexistent application-facing admission lane.

Any frontend-neutral planner or bound proof found only in the deleted SQL
matrix is moved to the planner/admission owner before the old test is
removed. This is coverage relocation, not preservation of the shadow path.

### Admission DTOs are internal

The facade stops re-exporting:

- `QueryAdmissionAccessKind`;
- `QueryAdmissionDecision`;
- `QueryAdmissionGroupedSummary`;
- `QueryAdmissionLane`;
- `QueryAdmissionOrdering`;
- `QueryAdmissionPlanShape`;
- `QueryAdmissionRejection`;
- `QueryAdmissionResidualFilter`;
- `QueryAdmissionSummary`;
- `QueryBoundKind`;
- `QueryMaterializationSummary`.

Core visibility narrows to the smallest practical admission/diagnostic owner.
Public callers continue to receive current typed domain errors, diagnostic
codes, and rendered EXPLAIN output. They do not construct policy summaries.

### Public query construction is fluent only

`PlannedQuery<E>`, `CompiledQuery<E>`, and their shared private
`QueryPlanHandle` are deleted. So are every facade/core `.planned()` and
`.plan()` terminal, `planned_query_with_visible_indexes`,
`compile_query_with_visible_indexes`, their re-exports, and tests that assert
parity between the two inert shells.

`AccessPlannedQuery` remains the internal logical/access plan.
`SharedPreparedExecutionPlan` remains the executor-ready cache/handoff
contract. Public diagnostic use cases stay on the fluent surface through
direct explain, trace, and plan-hash terminals.

The facade's hidden raw `Query<E>` re-export,
`execute_query_result_with_attribution`, and `trace_query` are deleted.
Core `Query<E>` remains as the implementation type used by fluent builders;
0.203 removes only its obsolete application-facing exposure.

### One projected-row response payload

`RowProjectionOutput` is the sole public projected-row payload for SQL
projection and fluent write-returning results. Its existing `entity`,
`columns`, `rows`, and `row_count` wire fields and value semantics remain
unchanged.

The following are deleted:

- core and facade `ProjectionResponse<E>`;
- `SqlProjectionRows`;
- `SqlQueryRowsOutput`;
- public `ProjectionRows`;
- `ProjectionResponse::from_core` and its dead response methods;
- `RowProjectionOutput::as_projection_rows`;
- `RowProjectionOutput::from_projection` once callers construct the current
  payload directly.

`ProjectionRows` has no validation invariant beyond fields already present in
`RowProjectionOutput`; it is therefore deleted rather than merely hidden.
`SqlQueryResult::Projection` carries `RowProjectionOutput` directly. SQL
rendering consumes that payload without cloning it into an intermediate.

### Access-choice diagnostics carry typed planner truth

`AccessChoiceExplainSnapshot` carries typed candidate identity and typed
`AccessChoiceRejectedReason` values. Eligible and rejected candidates use a
stable planner-owned identity that is also attached to the selected
candidate. Rendered labels remain output, not lookup keys.

`parse_rejected_index_label`, rendered-label candidate matching, and the
one-candidate fallback are deleted. `ExplainRejectedIndexV1` is projected
directly from the typed snapshot. Existing outward reason codes and labels
remain stable unless a test proves that the current reconstruction is
incorrect.

This slice changes diagnostic ownership, not route selection, ranking, tie
breaking, plan hashing, cache identity, or execution.

### Grouped diagnostics report the selected grouped mode

Grouped execution continues to select `HashMaterialized` or
`OrderedMaterialized` according to current planner contracts. The empty
`GROUPED_AGGREGATE_FAST_PATH_ORDER` remains as the explicit route-shape
invariant and planner input, but it is not interpreted as an eligibility
result.

`GroupedRouteDecisionOutcome`, `GroupedRouteRejectionReason`, and the
`eligible`, `outcome`, and `rejection_reason` fields are deleted. Diagnostic
projection consumes the selected `GroupedExecutionMode` and the real
planner-owned `GroupedPlanFallbackReason`.

The properties `grouped_route_eligible`, `grouped_route_outcome`, and
`grouped_route_rejection_reason` disappear. Materialized grouped execution is
reported as the selected current mode, not as fallback from a route that
does not exist.

### Generated runtime fallback disappears from metrics vocabulary

`PreparedShapeFinalizationOutcome::GeneratedFallback` and
`prepared_shape_generated_fallback` are deleted from events, counters,
state, summaries, reports, accessors, snapshots, and tests.

The one remaining real observation is recorded directly through
`record_prepared_shape_already_finalized_for_path` and the existing
`prepared_shape_already_finalized` counter. The two-variant
`PreparedShapeFinalizationOutcome` enum is deleted. No replacement
accepted-vs-generated enum is introduced.

Executor finalization continues to reuse a static accepted-schema-backed
contract when present and otherwise finalizes from accepted `SchemaInfo` or
fails with a typed invariant error.

### Compatibility-only source and tests are removed

The Clap alias `primary_key` is removed from
`ConfigInitUpdatePolicy::PrimaryKey`; `primary-key` remains the sole CLI
spelling. The separate configuration value `primary_key` remains because it
belongs to the TOML/config grammar rather than the CLI compatibility alias.

The following anti-resurrection fixture pairs are deleted:

- `testing/macro-tests/tests/ui/db_default_removed.rs` and `.stderr`;
- `crates/icydb-core/tests/ui/db/contracts_module_absent.rs` and `.stderr`.

The stale `db/contracts_module_private.rs` row is removed from
`crates/icydb-core/tests/ui/ARCHITECTURE_GUARDS.md`. Positive tests for
current default syntax and privacy of live modules remain.

### Active documentation describes only the current architecture

`docs/contracts/CURSOR.md` is rewritten in place as the concise normative
current cursor contract. Its historical checkpoint prose is deleted rather
than retained beside the current contract. It must not claim that current
cursors use CBOR or cannot push continuation into an eligible ordered route.

`docs/architecture/SQL_SURFACE_MAPPING.md` loses the legacy-matrix rationale
and gains a valid archive link. `docs/contracts/READ_ADMISSION.md` loses
`DevTest`, phantom `AdminAdHoc` execution claims, and hidden public-SQL
surface claims. Durability documents point to their actual archived evidence
paths.

Unqualified `GROUP BY v1` and `grouped v1` wording is replaced by the current
invariant it means. Durable identities such as Structural Binary v1,
`expr:v1`, current codec versions, and V1 EXPLAIN DTOs remain because they
name current wire, persisted, cache, or diagnostic contracts.

## Detailed findings

### HCLA-001 — Trusted/admin SQL is ambient rather than explicit

**Files and symbols**

- `crates/icydb/src/db/session/sql.rs` — public
  `DbSession::execute_sql_query<E>` and
  `execute_sql_query_with_perf_attribution<E>`.
- `crates/icydb-core/src/db/session/sql/mod.rs` — core
  `DbSession::execute_sql_query<E>`.
- `crates/icydb-build/src/db/sql.rs` — generated
  `icydb_sql_surface_require_controller` and generated `icydb_query` dispatch.
- `crates/icydb-core/src/db/query/admission.rs` —
  `QueryAdmissionLane::{AdminAdHoc, DevTest}`.
- `crates/icydb-core/src/db/query/admission/policy.rs` — test-only
  `QueryAdmissionPolicy::admin_ad_hoc`.
- `docs/contracts/READ_ADMISSION.md` — surface inventory and lane definitions.

**Current behavior.** A library caller invokes a broadly capable method named
`execute_sql_query`. Its docs say trusted/admin and warn against caller-controlled
SQL, but the type/signature does not require a trusted marker. Generated
`icydb_query` is safe because generated code separately calls
`icydb_sql_surface_require_controller("query")`. Fluent maintenance reads use a
visibly explicit `trusted_read_unchecked()` transition.

The admission vocabulary claims that `execute_sql_query` selects
`AdminAdHoc`, but production SQL never constructs an admin policy or records
that lane. `QueryAdmissionPolicy::admin_ad_hoc` is `cfg(test)`; `DevTest` has no
constructor/producer at all. The actual architecture is therefore:

- production fluent public reads: evaluated by built-in `PublicRead` policy;
- diagnostic explain: evaluated as `DiagnosticExplain` without rows;
- trusted fluent: explicit method marker, policy bypass;
- trusted SQL: ambient method/caller contract, policy bypass;
- generated SQL: controller gate outside the SQL session method.

**Historical design represented.** The lane enum and documentation describe a
multi-lane execution model in which admin and dev/test work are selected and
observable admission lanes. The runtime has instead converged on a public policy
gate plus explicit bypass surfaces.

**Obsolescence proof.** Production reference search finds no
`QueryAdmissionLane::AdminAdHoc` producer and no `DevTest` producer. Generated
tests explicitly assert that generated SQL contains the controller gate and does
not call the hidden public-read-policy method. The public-read contract says
custom application policy is not a current product surface.

**Reachability and compatibility.** The broad SQL helper is reachable public
API and executes rows. The phantom lane variants are reachable only as public
values or tests, not as production decisions. There is no external compatibility
obligation. No persisted format is involved.

**Tests preserving it.** Generator tests correctly preserve the controller
gate and should remain. Admission unit tests manufacture `AdminAdHoc`; those
should go with the unused lane. Public SQL policy tests are addressed separately
by HCLA-002.

**Recommended end state.** Rename the facade helper to
`execute_trusted_sql_query` and update the core/generator handoff in the same
hard-cut slice. Do not keep `execute_sql_query` as an alias. Keep the
generated controller gate. Remove `AdminAdHoc` and `DevTest` from admission
policy taxonomy unless production execution begins to carry those exact typed
facts; do not introduce a general capability framework merely to preserve the
enum.

**Semantic impact.** Intentional source break only. Generated authorization,
SQL semantics, planning, execution, response shape, and performance must remain
unchanged.

**Focused validation.** Facade compile tests, generated SQL surface tests,
controller-gate source assertions, SQL unit/integration tests, and a repository
search proving the old method and phantom lane names are absent.

**Semantic impact.** Intentional source break; no execution or diagnostic

## False positives and deliberate retention

### Current pagination response types

`PagedResponse` and `PagedGroupedResponse` are not the obsolete response-page
facade described by the audit prompt. They contain current items, opaque next
cursor, trace/read-intent metadata, and no page number, total pages, or ambient
offset state. Retain them.

### SQL `OFFSET`

`OFFSET` remains intentional in trusted/admin reduced SQL. Ordinary public
fluent reads do not expose it, and `PublicRead` rejects nonzero offset. This is a
semantic lane distinction, not a compatibility path.

### Cursor wire version and rejection corpus

`TOKEN_WIRE_VERSION = 2` is the sole accepted cursor wire. The decoder checks
equality and fails closed; it has no legacy branch. Tests for old/future tags are
malformed/unsupported-current-boundary tests, not compatibility tests. Retain
the version byte and rejection coverage.

### Database boot legacy-magic check

`LEGACY_STABLE_CELL_MAGIC` in `db/database_format/mod.rs` does not decode an old
database. It detects a non-virgin pre-boot-record stable cell and refuses to
initialize over it. It protects real data from overwrite and should remain,
with its fail-closed purpose documented.

### Durable `v1` names and version fields

Stable-memory allocation names, Structural Binary v1, `expr:v1`, database boot
version, row/commit/journal/schema codec versions, schema `SchemaVersion` values,
and cache method versions are current identity/version contracts. A schema v1
followed by schema v2 in tests represents accepted schema revisions, not dual
execution architecture. Do not rename these cosmetically.

### One current codec per durable surface

The persisted-format inventory and source show one active database boot format,
one raw-row envelope, one commit marker/control format, one journal batch format,
one accepted schema snapshot/catalog format, one index encoding, and one cursor
wire. Unsupported versions reject. No compatibility reader, migration decoder,
or “try old then new” branch was found. Retain version gates and bounded fallible
decoders.

### Generated-model-only schema visibility

`VisibleIndexes::GeneratedModelOnly` and generated `SchemaInfo` constructors are
used for proposal, reconciliation/model-only utilities, and tests. Production
guarded execution selects accepted schema or fails closed. This matches the
architecture rule and is not evidence of runtime fallback. The obsolete metric
in HCLA-009 is the part to remove.

### Predicate and expression representations

`Expr` is the full semantic truth representation. `Predicate` is a deliberately
narrow, validated subset for access selection and efficient residual execution.
`ResidualFilterContract` and `EffectiveRuntimeFilterProgram` preserve which
forms are available without reconstructing missing truth. These are justified
stage contracts, not equivalent peer authorities.

### Prepared execution wrappers

`PreparedExecutionPlan<E>` and `SharedPreparedExecutionPlan` share a core but
serve a real typed vs type-erased cache/handoff boundary. Unlike
`PlannedQuery`/`CompiledQuery`, they are consumed by production execution and
carry executor-ready state. Retain them.

### Grouped materialization algorithms

`HashMaterialized` and `OrderedMaterialized` are distinct physical algorithms
with different ordering/precondition behavior. `GroupedPlanFallbackReason` can
explain a real planner choice between them. Retain these; remove only the outer
fiction that all materialized grouped execution is a route fallback.

### Current fallback terms

Load-order materialization fallback, covering-read fallback, predicate-pushdown
fallback, and scalar aggregate capability fallback remain real route choices
where a preferred optimization can be proven or rejected without changing
semantics. Their producers and consumers are live. Do not delete “fallback” by
keyword alone.

### Explain `V1` DTOs

`ExplainAccessDecisionV1` and related DTOs are one current versioned diagnostic
projection, not a v1/v2 executor split. They remain reachable through public
EXPLAIN. Retain the version tag until a deliberate diagnostic-schema hard cut;
fix HCLA-007 underneath it.

### Old/new row images

“Old row” and “new row” in commit/journal/mutation code are before/after images
for atomic mutation and recovery. They are not legacy/current persisted-format
decoders.

## Risks

### Renaming trust without preserving authorization

An explicit method name does not replace the generated controller gate. Tests
must prove both facts remain present and ordered correctly.

### Deleting tests without relocating current semantic proof

The public-SQL matrix contains some frontend-neutral planner facts. Before
deleting a test, verify that its current invariant is covered at the planner,
admission, SQL-lowering, or fluent terminal owner. Relocate only missing
current coverage.

### Accidental diagnostic-schema break

Typed internal snapshots should preserve current outward reason codes and
labels unless existing output is demonstrably wrong. The design removes
string parsing; it does not casually redesign EXPLAIN V1.

### Conflating grouped algorithms with route sediment

`HashMaterialized` and `OrderedMaterialized` are real physical choices.
Remove fictitious route outcome state without merging the algorithms or
changing planner fallback reasons.

### Over-privatising types used by generated code

Every visibility reduction receives a fresh generator, fixture, and public
compile scan. If generated code has a live requirement, expose the narrow
generated boundary rather than restoring the application-facing re-export.

### Mistaking durable version names for legacy architecture

Version gates and current V1/V2 wire identifiers remain unless a separate
persisted-format design proves they are obsolete. Keyword matches are not
deletion proof.

### Concurrent worktree overlap

The audit observed a moving 0.202 default/persisted-row slice. Each 0.203
landing slice must re-read overlapping modules and preserve unrelated work.
The benchmark-target failure recorded in the audit baseline is not a 0.203
behavior change and should be re-evaluated against the settled worktree.
