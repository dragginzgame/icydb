# IcyDB 0.149 - Schema Defaults And Accepted Write Encoding

## Status

In progress.

0.149 follows the 0.148 nullable-additive runtime cutover. 0.148 proved that
accepted schema contracts can safely read old shorter rows and write current
accepted layouts. 0.149 should extend that model in one direction:

```text
accepted field defaults
  -> required append-only field additions
  -> accepted structural runtime-value encoding
  -> fewer generated write-codec bridges
```

This is still schema evolution by generated proposal and startup
reconciliation. It is not SQL DDL and not a general migration engine.

## Summary

0.148 accepts only append-only nullable field additions. Missing old-row slots
materialize as `NULL`, and new rows emit current accepted slot count. Required
appended fields stay rejected because the database has no schema-owned value to
materialize for old rows.

0.149 introduces database-owned defaults and moves structural runtime-value
encoding onto accepted field contracts so the runtime can safely accept:

```text
append-only required field + explicit database default
```

The central rule is:

```text
schema defaults belong to persisted schema metadata
Rust Default::default() is not database evolution policy
```

## Goals

* Add a persisted schema default representation for supported field kinds.
* Materialize missing old-row slots from accepted database defaults.
* Accept append-only required fields only when they carry an explicit database
  default.
* Keep append-only nullable fields from 0.148 working unchanged.
* Move existing structural runtime-value encoding onto accepted field
  contracts backed by `PersistedFieldKind`.
* Keep generated value-codec bridges only where they are typed compatibility
  surfaces.
* Preserve fail-closed behavior for unsupported defaults and unsupported
  transition shapes.

## Non-Goals

* Do not support SQL `ALTER TABLE`.
* Do not support field removal.
* Do not support field rename.
* Do not support slot remapping.
* Do not support field kind/type changes.
* Do not support nested shape evolution.
* Do not support index definition evolution.
* Do not introduce per-row schema versions.
* Do not backfill all existing rows during startup.
* Do not infer defaults from Rust struct defaults.
* Do not change cursor, row, or persisted storage formats unless a narrow
  default encoding field is required in schema metadata.

## Default Model

### Ownership

Defaults must be stored in accepted schema metadata. They are part of the
database schema contract, not generated Rust type behavior.

The default contract should answer:

* whether a field has a database default
* whether the default is valid for the persisted field kind
* how to materialize the default as runtime `Value`
* how to encode the default into persisted slot bytes

### Suggested Shape

The exact type names may differ, but the model should be explicit:

```rust
enum PersistedFieldDefault {
    None,
    SlotPayload(Vec<u8>),
}
```

Defaults are persisted as already-encoded slot payload bytes, not free-form
runtime `Value` data. That keeps the saved schema tied to field-codec
boundaries and avoids introducing a second persisted value grammar.

### Supported Defaults

Start narrow. Support only field kinds whose defaults can be validated and
encoded without ambiguity.

Supported literal `default = ...` values become database defaults by default.
`db_default = ...` may override the database value when the Rust construction
default needs to differ. Function and path defaults, including generated IDs and
timestamps, remain Rust construction or insert-generation behavior only.

Implemented first candidates:

* bool
* bounded or unbounded text with max length validation
* fixed-width signed integers
* fixed-width unsigned integers
* float32 / float64

Defer or reject if not straightforward:

* blob defaults
* decimal with exact scale normalization
* timestamp/duration/date
* principal/account/subaccount/ulid
* unit defaults until the schema macro has an explicit unit-literal surface
* explicit `NULL` defaults
* relation defaults
* nested structured values
* maps/lists/sets
* enum defaults until enum schema is first-class

Unsupported default values must reject at schema reconciliation time, before any
row decode or write staging.

## Accepted Absence Policy

0.148 has:

```text
NullIfMissing
Required
```

0.149 should add or enable the equivalent of:

```text
DefaultIfMissing(default)
```

Runtime behavior:

* present slot -> decode the stored payload
* missing nullable field with no default -> `NULL`
* missing field with database default -> default value
* missing required field without database default -> fail closed

New writes should encode the default when a required defaulted field is omitted
by a structural write surface that is allowed to omit it.

## Accepted Structural Runtime-Value Encoding

### Current 0.148 Bridge

0.148 deliberately leaves structural runtime-value encoding on
generated-compatible `FieldModel` bridges. That is safe for the
nullable-additive slice because target-slot authority and missing-slot
semantics are accepted-schema-owned, but it is not the final write boundary.

This does not make `Value` a persisted format. `Value` remains an in-memory
boundary type for SQL, structural patches, projections, and decoded row views.
Persisted rows remain slot-framed bytes owned by field codecs.

### Target 0.149 Boundary

Replace generated-compatible structural write encoding with accepted-field
encoding helpers that consume:

```text
AcceptedFieldDecodeContract / PersistedFieldKind
  + FieldStorageDecode
  + LeafCodec
  + Value
```

and produce persisted slot bytes without reopening generated `FieldModel`
validation.

The accepted encoder must share low-level primitives with generated-compatible
encoding. Do not copy codec semantics into a parallel implementation that can
drift.

### Required Parity

For generated-compatible fields:

```text
accepted structural encode(Value) == generated-compatible structural encode(Value)
accepted decode(bytes) == generated decode(bytes)
```

except where accepted schema has a deliberate transition policy, such as
missing-slot default materialization.

## Transition Policy

Extend the 0.148 accepted additive transition:

```text
stored fields are an exact prefix of generated fields
generated appended fields are accepted if:
  - nullable with no default, or
  - explicit database default is valid for the field
```

Reject:

* required appended field without database default
* invalid default for field kind
* default that violates text/blob length, decimal scale, or scalar bounds
* generated Rust default without persisted schema default
* changed default on an existing field unless an explicit policy is designed

The transition plan should carry added-field default facts. Do not let runtime
decode rediscover default policy from raw booleans.

## Write Semantics

### Inserts

For current-layout inserts:

* authored field -> encode authored value through accepted encoder
* omitted nullable field without default -> encode `NULL`
* omitted field with database default -> encode default
* omitted required field without database default -> reject

### Updates

Updates over old rows:

* materialize baseline through accepted row contract
* synthesize missing defaulted fields before applying the patch
* overlay authored patch values
* emit current accepted slot count

### Replaces

Replace follows insert semantics. It does not inherit old missing-slot state.

## SQL And Metadata Surface

0.149 does not require SQL DDL.

Metadata should expose defaults once defaults exist:

* `DESCRIBE <entity>` shows default information for defaulted fields
* `SHOW COLUMNS FROM <entity>` shows default information in the columns section

The exact text should be concise and stable enough for CLI inspection.

## Validation

### Required Tests

Add tests for:

* nullable append-only field still works as in 0.148
* required append-only field without database default is rejected
* required append-only field with database default is accepted
* old rows read defaulted appended fields as the default
* old-row updates persist current-layout rows with defaulted fields
* inserts omitting defaulted required fields encode the default
* replaces omitting defaulted required fields encode the default
* invalid defaults reject at schema reconciliation
* text/blob defaults respect max length
* decimal defaults normalize or reject according to exact scale rules
* accepted structural encode/decode parity against generated-compatible fields
* metadata surfaces display default facts

## Implementation Order

1. Add persisted schema default representation and validation. Completed for
   encoded slot payload representation, macro-side literal validation, and
   accepted-field-kind validation during transition policy.
2. Add accepted default materialization to accepted field/row contracts.
   Completed.
3. Add accepted-field structural runtime-value encoding helpers using shared
   low-level primitives. Completed.
4. Add parity tests between accepted and generated-compatible encoding.
   Completed for the accepted write encoder surface.
5. Extend transition policy to accept required appended fields with explicit
   database defaults. Completed.
6. Route structural write omission/default fill through accepted schema
   defaults. Completed for accepted structural insert/replace staging.
7. Update old-row update/replace handling to synthesize defaults before row
   emission. Completed for old-row update overlay and accepted replace
   after-image staging.
8. Update metadata display for defaults. Completed for generated and accepted
   schema description rows.
9. Retire generated value-codec bridges where accepted encoding now covers the
   path.

## Stop Condition

0.149 is complete when:

* nullable additive fields from 0.148 still work
* required append-only fields with explicit database defaults work
* required append-only fields without defaults still fail closed
* old rows materialize defaults without row rewrite during startup
* new writes emit current accepted layout with defaulted fields populated
* accepted encoding covers structural runtime write paths
* remaining generated write-codec uses are typed compatibility edges only
