# 0.78 Design: Bounded Scalar Projection Expressions

## Preamble

* **scope**: freeze one bounded scalar arithmetic projection slice
* **goal**: admit one small arithmetic projection family without widening routing, grouping, predicate semantics, or generic expression grammar
* **theme**: one canonical projection expression path shared by SQL and fluent
* **non-goals**:
  * no predicate arithmetic
  * no grouped arithmetic
  * no aggregate arithmetic
  * no `ORDER BY` arithmetic
  * no generic expression grammar
  * no planner or routing changes
  * no second evaluator or projection lane

---

## Problem Statement

The frozen `0.77` SQL surface already admitted bounded text projection and
field projection, but it still could not express simple numeric computed
columns such as:

```sql
SELECT age + 1 FROM Customer;
SELECT age - 1 FROM Customer;
SELECT age * 2 FROM Customer;
SELECT age / 2 FROM Customer;
```

The runtime already had the semantic pieces to evaluate a bounded binary
expression, but that capability was not exposed as one shipped scalar
projection surface.

---

## Current `0.78` Surface

### SQL

The shipped SQL surface is:

```sql
SELECT age + 1 FROM Customer;
SELECT age - 1 FROM Customer;
SELECT age * 2 FROM Customer;
SELECT age / 2 FROM Customer;
SELECT name, age + 1 FROM Customer;
```

Only one arithmetic family is admitted:

* `field op numeric_literal`
* operators:
  * `+`
  * `-`
  * `*`
  * `/`

### Fluent

The matching fluent surface is:

```rust
db.load::<Customer>().project_values(&add("age", 1_u64))
db.load::<Customer>().project_values(&sub("age", 1_u64))
db.load::<Customer>().project_values(&mul("age", 2_u64))
db.load::<Customer>().project_values(&div("age", 2_u64))
```

SQL and fluent must both lower to the same canonical expression form.

---

## Canonical Representation

Both frontends lower this feature to:

```text
Expr::Binary {
    op: Add | Sub | Mul | Div,
    left: Expr::Field(...),
    right: Expr::Literal(...),
}
```

This slice must not introduce a second projection-only AST or evaluator.

---

## Execution Invariants

This feature is projection-only.

It must preserve all of the following:

* no planner impact
* no routing impact
* no new execution family
* no grouped/scalar execution split changes
* reuse of the existing scalar projection evaluator

The only new runtime work is one additional bounded expression evaluation per
projected row when this feature is used.

---

## Explicitly Admitted in `0.78`

* scalar `SELECT` projection only
* one numeric field on the left
* one numeric literal on the right
* bounded operators:
  * `+`
  * `-`
  * `*`
  * `/`
* mixed projection lists such as:
  * `SELECT name, age + 1 FROM Customer`
  * `SELECT name, age * 2 FROM Customer`

---

## Explicitly Rejected in `0.78`

These remain fail-closed:

* `field + field`
* chained arithmetic such as `age + 1 + 2`
* grouped arithmetic projection
* `WHERE age + 1 > 10`
* `ORDER BY age + 1`
* aggregate arithmetic such as `SUM(age + 1)`
* generic nested expression parsing

Unsupported forms must fail in the owning layer:

* parser for unsupported syntax outside the admitted shape
* lowering for grouped/projection-shape widening

---

## Parity Rule

This slice is only correct if SQL and fluent share the same bounded projection
shape.

That means:

* SQL must not gain arithmetic projection without a matching fluent form
* fluent must not lower bounded arithmetic through a private non-SQL path
* both frontends must use the same canonical `Expr::Binary` form

---

## Tests Required for Freeze

The freeze requires direct proof for:

* parser acceptance of `field op literal`
* lowering to canonical `Expr::Binary`
* SQL/fluent result parity for the admitted shape
* fail-closed rejection of:
  * `field + field`
  * grouped arithmetic
  * arithmetic outside projection position

---

## Follow-Up Boundary

The next possible widenings, if chosen later, are still outside this bounded
surface:

```sql
SELECT age + bonus FROM Customer;
SELECT age + 1 + 2 FROM Customer;
SELECT * FROM Customer WHERE age + 1 > 10;
```

Those are **not** part of the current `0.78` arithmetic slice. Any later
widening must extend the same canonical binary-expression path rather than
adding a second scalar projection lane.
