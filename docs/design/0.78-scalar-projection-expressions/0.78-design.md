# 📄 0.78 Design: Bounded Scalar Projection Expressions

## Preamble

* **scope**: introduce bounded scalar expression support in SQL projection (SELECT list only)
* **goal**: enable computed output fields without affecting execution planning
* **theme**: extend projection evaluation without expanding query semantics
* **non-goals**:

  * no WHERE expression support
  * no GROUP BY expression support
  * no ORDER BY expressions
  * no planner or routing changes
  * no new execution paths
  * no function support
  * no expression system expansion beyond bounded arithmetic

---

## Problem Statement

The system currently supports only direct field projection:

```sql
SELECT field FROM table;
```

This limits usability and prevents simple computed outputs such as:

```sql
SELECT field + 1 FROM table;
```

Expression evaluation capability exists structurally (`Expr::Binary`), but is not exposed in projection.

---

## Design Goals

1. **Execution containment**

   * expressions must not affect routing, feasibility, or execution stages

2. **Single execution path**

   * reuse one compiled expression evaluator
   * no duplicate evaluation logic

3. **Fail-closed scope**

   * unsupported expressions are rejected at compile time

4. **Zero planner impact**

   * projection expressions must not influence query planning

---

# Part 1: Feature Surface

## 1.1 Supported projection expressions

### Field + literal

```sql
SELECT age + 1 FROM users;
```

### Field + field

```sql
SELECT salary + bonus FROM users;
```

### Chained arithmetic

```sql
SELECT age + bonus + 5 FROM users;
```

### Mixed projection

```sql
SELECT name, age + 1 FROM users;
```

---

## 1.2 Supported operators

* arithmetic:

  * `+`
  * `-`
  * `*`

---

## 1.3 Supported expression forms

* `Field`
* `Literal`
* `Binary(Field/Literal, op, Field/Literal)`
* nested binary expressions (left/right chaining only)

---

## 1.4 Explicit exclusions (fail-closed)

* function calls
* aggregates in projection expressions
* grouped expressions
* string operations
* casts
* subqueries
* ORDER BY expressions
* WHERE expressions

Any unsupported node → **compile-time rejection**

---

# Part 2: Internal Representation

## 2.1 Expression reuse

Projection uses existing:

```text
Expr
```

No new AST types introduced.

---

## 2.2 Canonical form

All projection expressions are represented as:

```text
Expr::Binary(left, op, right)
```

with recursion for chaining.

---

## 2.3 Compilation step

Expressions are compiled once into a runtime form:

```text
CompiledExpr
```

* resolves field references to slot indices
* validates allowed operators
* rejects unsupported forms

---

# Part 3: Execution Model

## 3.1 Row evaluation

For each row:

```text
value = eval(compiled_expr, row)
```

---

## 3.2 Execution invariants

Must preserve:

* single-pass row loop
* no additional execution phases
* no planner involvement
* no branching execution paths

---

## 3.3 Integration point

Executed inside projection stage:

```text
execute_kernel_row_scan(...)
→ project (with expression evaluation)
→ output
```

---

## 3.4 Core invariant

> Projection expressions must use the same compiled evaluator used across all expression evaluation.

No alternative evaluators permitted.

---

# Part 4: Type Semantics

## 4.1 Allowed

* numeric + numeric
* compatible field + field
* field + literal

---

## 4.2 Rejected

* incompatible types (e.g. string + number)
* unsupported operations

---

## 4.3 Consistency rule

> Projection expressions follow consistent type rules across all execution paths.

---

# Part 5: Planner Behavior

## 5.1 No planner impact

Projection expressions:

* do not affect routing
* do not affect feasibility
* do not affect index selection

---

## 5.2 Planner invariants

* identical plans with or without projection expressions
* projection evaluated strictly post-selection

---

# Part 6: SQL Surface

## 6.1 Supported queries

```sql
SELECT age + 1 FROM users;
SELECT name, salary + bonus FROM users;
SELECT age + bonus + 5 FROM users;
```

---

## 6.2 Unsupported queries

```sql
SELECT * FROM users WHERE age + 1 > 10;   -- rejected
SELECT * FROM users ORDER BY age + 1;     -- rejected
SELECT SUM(age + 1) FROM users;           -- rejected
```

---

# Part 7: Fluent API Alignment

## 7.1 Expression construction

```text
field("age").add(1)
field("salary").add(field("bonus"))
```

---

## 7.2 Canonical lowering

All APIs must produce:

```text
Expr::Binary(...)
```

No alternative construction paths.

---

# Part 8: Testing

## 8.1 Core

* field + literal
* field + field
* chained expressions

---

## 8.2 Error handling

* unsupported expressions rejected
* invalid type combinations rejected

---

## 8.3 Execution invariants

* row count unchanged
* ordering unchanged
* results identical across execution paths

## 8.4 Covering parity

* identical results across:

  * covering

# Part 9: Rollout

1. Enable expression parsing in projection
2. Compile expressions into `CompiledExpr`
3. Integrate evaluator into projection stage
4. Add fail-closed validation
5. Expand tests

---

# Part 10: Risks

## 10.1 Execution overhead

* per-row evaluation cost

**Mitigation:**

* compile expressions once
* keep evaluator minimal

---

## 10.2 Scope creep

**Mitigation:**

* enforce projection-only boundary
* fail-closed compilation

---

## 10.3 Duplicate evaluators

**Mitigation:**

* enforce single evaluator invariant

---

# Final Statement

After 0.78:

> SQL projection supports bounded scalar expressions such as:
>
> * `field + 1`
> * `field_a + field_b`
> * `field_a + field_b + 5`
>
> while preserving:
>
> * no changes to routing or planning
> * single execution kernel
> * single-pass evaluation
> * strict fail-closed semantics
> * no new execution paths
