# 0.82 Design: Deterministic Compiled Query Reuse

## Document Status

* status: active
* shipped groundwork: `0.82.0` through `0.82.8`
* current follow-up target: `0.82.9`
* scope of `0.82.9`: keep the `0.82` line on executor/runtime cost by
  investigating how selective direct lanes can avoid full row-store fetches
  without widening semantics or introducing a second storage model
* historical note:
  * earlier references to `0.82.1` in this document describe the first cache
    determinism slice for this design line
  * the next active work under the same design line is now the `0.82.9`
    selective-read / storage-boundary follow-up

---

## Current Follow-Up Target: `0.82.9`

The cache/reuse contract work from the earlier `0.82` slices is already
shipped. The next unresolved cost center under this same design line is no
longer cache identity. It is direct-lane row fetch cost.

The current direct-lane perf audit shows:

* route-ordered direct loads are dominated by `store.get(...)`
* materialized-order direct loads are dominated by `store.get(...)` first and
  direct order extraction second
* widening sparse decode helped the direct-field `ORDER BY` lane modestly, but
  it did not change the larger store-read floor

The important distinction for `0.82.9` is:

* selective lanes can plausibly avoid full `RawRow` fetches
* full fluent entity-row lanes cannot, unless the storage/read boundary itself
  changes

That means `0.82.9` should stay narrow:

1. audit which hot direct lanes are truly selective
2. use covering reads where the current planner/runtime contract already makes
   that honest
3. for the remaining selective lanes, design one narrower slot/view read
   boundary instead of forcing every selective path through `DataStore::get`
4. leave full entity-load lanes on full-row `get()` until there is a stronger
   storage redesign reason

Current conclusions from the audit:

* the hottest fluent audited families are still full-entity lanes, not
  projection-only lanes
* those lanes are not good covering-read candidates under the current
  covering contract
* the next worthwhile design move is one selective-read boundary for:
  * projection-only reads
  * order-key extraction
  * predicate-only selective reads
* the next worthwhile design move is not:
  * another cache tier
  * another broad executor abstraction
  * pretending full entity loads are selective when they are not

The acceptance bar for `0.82.9` is therefore:

* no semantic widening
* no new persistent cache model
* no storage rewrite by default
* one explicit explanation of which lanes can skip full-row reads today
* one explicit boundary proposal for the selective lanes that cannot

## Preamble

* scope: harden the 0.80 compiled-query/cache line into one deterministic, measurable, and security-compatible reuse model
* goal: make repeated SQL and fluent query compilation reuse correct by construction, explicit in ownership, and provable in the perf harness
* theme: one canonical semantic identity, one layered reuse contract, one fail-closed invalidation story
* non-goals:

  * no grouped computed projection widening
  * no grouped computed ordering
  * no aggregate-derived computed ordering
  * no generic expression grammar widening
  * no executor-scratch persistence as a new cache tier
  * no parameterized query-template cache yet
  * no cross-canister durable cache persistence claim

## Problem Statement

The 0.80 line established the right direction:

* explicit compile/execute separation
* one semantic compiled-command artifact
* shared lower-plan reuse across SQL and fluent reads
* canister-lifetime heap reuse for repeated queries
* perf visibility into compile / plan / execute cost splits

The 0.81 line then widened bounded computed ordering while preserving canonical semantic identity and proving alias/direct forms normalize onto the same compiled shape.

`0.82.0` then shipped the first explicit hashing/identity cleanup:

* one canonical `StructuralQueryCacheKey` for the shared lower cache
* explicit `Sha256` vs `xxh3` boundary cleanup
* cache-key construction moved away from hidden manual structural digest walks
* remaining manual hashing cleanup recorded as follow-up instead of being mixed
  into the cache model implicitly

That is enough progress that the next risk is no longer “missing cache architecture.”
The next risk is **cache drift**:

* semantically equivalent queries compiling into different cache identities
* layer duplication between compiled-command reuse and lower-plan reuse
* query/update reuse weakening public surface distinctions
* invalidation becoming implicit or optimistic
* perf wins becoming anecdotal rather than required and measured

So `0.82.1` should not add another SQL feature family first.
It should finish making the cache/reuse line deterministic, explicit, and measurable.

---

## Why This Should Be The Slice

### 1. The cache line exists, but it is not fully closed as a contract

0.80 introduced the compile/execute split and then widened reuse through shared lower-plan and canister-heap repeated-query reuse. `0.82.0` then made the shared lower identity explicit.

That means the project already has multiple reuse layers, not just a single experimental cache.

`0.82.1` is the point where those layers should become a documented, enforced model instead of an emergent one.

### 2. The perf harness is already strong enough to make this accountable

The recurring perf audit and harness already require scenario identity, phase attribution where available, comparability discipline, and PocketIC-backed measurement for public/canister surfaces.

That makes 0.82 unusually well positioned for a real cache slice:

* cold vs warm runs can be measured directly
* compile / plan / execute deltas can be tracked
* surface skew between fluent and SQL can be compared
* rejection-path inflation or cache-layer drift can be caught early

`0.82.1` should use that discipline as part of the feature definition.

### 3. It reduces risk before the next grouped widening line

Grouped computed projections or grouped computed ordering will touch planner ownership, resource policy, continuation compatibility, and explain identity all at once.

Those should land on top of a stable reuse model, not while the cache line is still semantically under-defined.

## Current Boundary

### Already present

The current line already has:

* an explicit compile / execute split for SQL
* one query-owned compiled-command artifact direction
* shared lower-plan reuse across SQL and fluent reads
* canister-lifetime heap reuse for repeated query work
* perf harness support for compile / plan / execute attribution
* security-boundary guidance that treats cache identity and query/update surface isolation as security-sensitive
* one explicit shared lower canonical cache key:
  * `StructuralQueryCacheKey`
* one explicit hashing split:
  * `Sha256` for deterministic semantic identity authorities
  * `xxh3` for in-heap cache hashing/runtime-oriented hashing

### Still incomplete as a contract

What remains under-specified today:

* canonical cache identity as an explicit repo-level invariant
* exact ownership of each cache/reuse layer
* invalidation triggers and fail-closed rules as first-class acceptance criteria
* query vs update reuse guarantees
* required warmed-run perf proof for newly cacheable shapes

## Design Goal

Make repeated-query reuse boring.

`0.82.1` should turn the current cache/reuse line into one deterministic model with:

* one canonical semantic identity before cache lookup
* one explicit ownership table for each reuse layer
* one fail-closed invalidation story
* one measurable cold/warm perf contract
* one security-compatible separation between public surfaces

The target for `0.82.1` is:

**equivalent admitted queries compile to one canonical semantic identity, reuse passes through explicitly owned layers, and every cache hit remains semantically identical to recompiling under the same compatible schema and surface context.**

---

## Core Decisions

### Decision 1: Canonical semantic identity comes before cache lookup

Every admitted query must reduce to one canonical structural identity before any compiled-command or lower-plan cache lookup/insertion occurs.

`0.82.0` already shipped the lower shared-cache-key side of this rule.
`0.82.1` should finish the contract around:

* how that identity composes with higher reuse layers
* where compatibility dimensions are applied
* how parity is proven in tests and perf rows

This includes, where applicable:

* alias-normalized order forms
* direct vs alias forms already proven equivalent by prior slices
* stable canonical ordering representation
* stable canonical predicate/projection/order structure

Rule:

**equivalent admitted forms must not become different cache keys because normalization happened too late.**

### Decision 2: Cache layers must have distinct ownership

`0.82.1` should define and preserve a strict separation between semantic and derived reuse layers.

Illustrative ownership table:

| Layer                   | Owns                                                                | Key basis                                               | Lifetime                                       |
| ----------------------- | ------------------------------------------------------------------- | ------------------------------------------------------- | ---------------------------------------------- |
| compiled-command cache  | semantic query meaning                                              | canonical query identity + schema/surface compatibility | session / canister heap                        |
| shared lower-plan cache | lowered reusable plan form for equivalent SQL/fluent read semantics | canonical semantic identity + lowering compatibility    | local/session or canister heap, as implemented |
| executor/runtime prep   | derived execution setup and scratch                                 | none as canonical semantic key                          | per execution                                  |

Rules:

* no layer may silently duplicate semantic authority owned by an earlier layer
* no layer may widen query meaning during reuse
* executor scratch or materialization buffers do not become a new semantic cache tier in 0.82

### Decision 3: Query and update surfaces must remain semantically distinct

A shared compile/reuse path must not collapse public surface distinctions.

0.82 must preserve:

* distinct cache identity dimensions where `query` vs `update` semantics differ
* no persistence guarantees claimed for standalone IC `query` misses
* no update-warmed reuse that weakens later query-surface semantics

This is both a cache-correctness and security-boundary requirement.

### Decision 4: Invalidation must be explicit and fail-closed

Reuse is only valid under compatible schema and surface conditions.

On any uncertainty, reuse must miss.

`0.82.1` should make explicit triggers for invalidation/miss decisions such as:

* schema/model shape change
* index/layout change relevant to compile or plan choice
* expression capability widening that changes admitted semantics
* cache-method/version mismatch
* query/update surface mismatch where semantics differ

## Architectural Approach

### Step 1: Define canonical query identity explicitly

Introduce or formalize one canonical semantic identity function for admitted reusable queries.

For `0.82.1`, this means building on the already-shipped lower shared key instead
of replacing it again.

It should be based on semantic structure, not surface trivia.

Conceptually:

1. parse / lower / normalize
2. canonicalize admitted equivalent forms
3. produce canonical semantic identity
4. consult compiled-command and lower-plan reuse layers using that identity
5. execute using derived runtime state

This identity must be stable enough to support:

* perf comparisons across equivalent forms
* cache parity tests
* explain/compiled-shape parity checks
* future template reuse without replacing the identity model

### Step 2: Make cache-layer ownership explicit in code and docs

`0.82.1` should document which module owns:

* compiled semantic reuse
* lowered reusable plan reuse
* runtime-derived setup

and should reject implicit ownership drift where the same meaning is reconstructed in multiple layers.

### Step 3: Keep executor/runtime prep derived, not promoted

Executor/runtime preparation may remain optimized, but `0.82.1` should not introduce persistent executor-prepared bundles as a new first-class cache model.

That work can come later if profiling proves it matters after semantic reuse is stable.

## Canonical Identity Rules

`0.82.1` should define these rules explicitly.

### Rule A — Equivalent admitted forms share one identity

Examples:

* admitted alias and direct bounded scalar order forms already proven equivalent
* equivalent canonical order/spec forms after normalization
* equivalent admitted SQL/fluent read semantics where lower-plan sharing is intended

### Rule B — Surface differences that change semantics must remain distinct

Examples:

* `query` vs `update` where public behavior or persistence semantics differ
* schema compatibility changes
* method-tag or cache-version changes

### Rule C — Unsupported forms do not participate in the identity model

Unsupported or fail-closed query shapes must reject before they can claim reuse identity.

## Cache Layer Contract

### Compiled-command cache

Owns:

* canonical semantic compiled query/command artifact

Must guarantee:

* semantic equivalence to recompilation under the same compatible schema/surface context
* no surface-specific weakening
* no duplicate identities for equivalent admitted forms

### Shared lower-plan cache

Owns:

* reusable lower-plan artifacts shared between equivalent SQL and fluent read semantics where explicitly intended

Must guarantee:

* no aliasing between semantically distinct front-door surfaces
* no lower-plan reuse that bypasses planner-owned restrictions
* no independent semantic model distinct from the compiled-command layer

### Runtime/executor prep

Owns:

* per-execution derived setup
* scratch vectors / materialization helpers / temporary execution state

Must guarantee:

* no semantic authority
* no long-lived independent cache identity in this slice

---

## Invalidation and Compatibility

`0.82.1` should make compatibility rules explicit and conservative.

### Required compatibility dimensions

At minimum, the reuse model must account for:

* canonical query identity
* schema/model compatibility fingerprint
* surface compatibility (`query` vs `update` where relevant)
* cache-method/version compatibility

### Required fail-closed behavior

If any required compatibility dimension is unavailable, mismatched, or uncertain:

* skip reuse
* rebuild through the canonical compile path

### Explicit anti-goal

Do not make optimistic “best effort” reuse decisions under ambiguity.

---

## Security Compatibility

`0.82.1` must fit the new security-boundary audit line.

### Required security properties

* no cross-surface cache aliasing between semantically distinct `query` and `update` paths
* no compiled-identity duplication for equivalent admitted forms because normalization happens too late
* no cache-based bypass of validation or planner-owned restrictions
* no claim that standalone public `query` misses create durable persistent cache state on the IC
* no reuse path that widens namespace, cursor, or public SQL boundaries

### Required audit alignment

The `0.82.1` implementation should leave behind:

* one clear cache ownership story for the security audit
* one clear canonical identity rule
* live canister checks for representative query/update reuse boundaries

## Perf Strategy

Perf is part of the `0.82.1` feature, not an optional follow-up.

### Perf questions 0.82 must answer

1. Do cold vs warm runs show real compile-side savings?
2. Are warm hits stable across equivalent admitted forms?
3. Does SQL reuse and fluent reuse stay aligned where lower-plan sharing is intended?
4. Does cache layering create any suspicious execute-path inflation?
5. Do rejection/unsupported paths stay appropriately cheaper or at least not inflate unexpectedly?

### Required perf scenario families

#### A. Canonical identity parity scenarios

Pairs/triples of equivalent admitted forms that should share one semantic identity.

Examples:

* bounded computed alias order vs direct bounded computed order
* equivalent fluent vs SQL read shapes where shared lower-plan reuse is intended

Measure:

* cold compile / plan / execute
* warm compile / plan / execute
* cache layer hit attribution

#### B. Query/update surface isolation scenarios

Representative scenarios where update-warmed reuse may later affect query surfaces.

Measure:

* update cold
* update warm
* subsequent query under equivalent semantics
* explicit hit attribution and surface identity

#### C. High-frequency scalar read scenarios

Representative shared-floor shapes:

* indexed scalar read
* bounded computed projection
* bounded computed ordering
* predicate expression read

Measure:

* cold/warm delta
* stability across repeated runs
* whether compile savings are real relative to execute cost

#### D. Negative/rejection scenarios

Representative fail-closed shapes:

* malformed SQL
* unsupported grouped order shape
* mismatched surface/identity cases if exposed by tests

Measure:

* rejection cost
* no suspicious cache-induced inflation

### Perf acceptance goal

`0.82.1` should demonstrate:

* measurable warmed-run compile-side improvement on representative reusable shapes
* no identity fragmentation across equivalent admitted forms
* no suspicious surface skew introduced by cache reuse
* no method drift hidden as a perf win

## Testing Strategy

### Identity parity tests

Add tests proving:

* equivalent admitted SQL forms produce the same canonical semantic identity
* equivalent admitted direct/alias bounded computed order forms produce the same compiled shape where intended
* equivalent SQL/fluent shapes produce the same lower-plan identity where sharing is intended

### Surface isolation tests

Add tests proving:

* query/update surfaces remain distinct when semantics differ
* update-warmed reuse does not weaken query semantics
* standalone query paths do not rely on persistence claims they do not own

### Invalidation tests

Add tests proving:

* schema-compatible hits reuse
* schema mismatch misses
* cache-version/method changes miss
* uncertain compatibility fails closed

### Explain/cache parity tests

Add tests proving:

* explain identity and compiled identity stay aligned for admitted equivalent forms
* cache reuse does not create explain-shape drift

### Perf harness tests

Add recurring harness rows for:

* cold vs warm representative scalar reads
* direct vs alias bounded computed order parity
* SQL vs fluent lower-plan sharing parity
* query/update reuse boundary cases

## Explicit Fail-Closed Boundaries

These must remain true after 0.82:

* no grouped computed projection widening
* no grouped computed ordering widening
* no aggregate-derived computed ordering widening
* no generic expression grammar widening
* no executor-prepared side cache becoming a second semantic model
* no optimistic reuse under compatibility uncertainty

`0.82.1` is a cache determinism and measurement slice, not a new SQL surface slice.

## Risks

### Risk: canonical identity is too weak

If equivalent admitted forms still compile into different identities, the cache line will fragment and perf wins will be misleading.

Mitigation:

* explicit canonical identity function
* identity parity tests
* required hit attribution in perf rows

### Risk: canonical identity is too broad

If semantically distinct surfaces alias to the same identity, reuse will become a correctness/security problem.

Mitigation:

* surface compatibility dimension
* fail-closed invalidation
* query/update isolation tests

### Risk: reuse layers duplicate authority

If compiled-command and lower-plan reuse both own overlapping semantics, later widening will become brittle.

Mitigation:

* explicit ownership table
* reject new reuse layers that duplicate semantic authority

### Risk: perf claims outrun evidence

If cache wins are asserted without stable cold/warm scenario comparisons, the line will become anecdotal.

Mitigation:

* recurring perf rows are part of acceptance criteria
* method-tag discipline for any harness change
* phase attribution retained where available

---

## Acceptance Criteria

`0.82.1` is complete when:

* one explicit canonical semantic identity rule exists for reusable admitted queries
* equivalent admitted forms prove identity parity where intended
* cache/reuse layer ownership is documented and enforced
* query/update surface isolation is tested explicitly
* invalidation is explicit and fail-closed
* recurring perf scenarios prove cold vs warm reuse benefits on representative shapes
* perf rows show which cache layer handled the query
* no new SQL surface family is widened as part of this slice
* security-boundary guidance for cache identity and surface isolation remains satisfied

---

## Final Recommendation

Ship one cross-cutting cache/perf slice:

**deterministic compiled-query reuse with explicit canonical identity, explicit layer ownership, fail-closed invalidation, and cold/warm perf proof.**

This is the right next step after `0.82.0` because it turns the current reuse architecture into a stable product contract before the next grouped SQL widening line begins.
