# 0.82 Addendum Design Doc

## Structural Query Cache Key Normalization

## Document Status

* status: adopted early
* release: implemented in 0.81.4 as 0.82 groundwork
* scope: shared fluent/lower query cache identity cleanup
* primary area: `crates/icydb-core/src/db/query/intent/model.rs`
* related runtime surfaces:

  * `crates/icydb-core/src/db/session/query.rs`
  * `crates/icydb-core/src/db/session/sql/mod.rs`

## Purpose

This addendum proposes a cleanup of shared query-plan cache identity construction by separating semantic canonicalization from hashing.

Today, the SQL-side cache keys are already structurally simple and use derived hashing on explicit key structs. The remaining complexity is concentrated in the shared structural query fingerprint path, where one manual encoder currently owns three concerns at once:

* semantic normalization
* canonical structural encoding
* byte-level hashing

The goal of this slice is to replace the current implicit canonicalization-through-hash-walk with an explicit canonical key type:

* `StructuralQueryCacheKey`

That key becomes the single normalized identity for shared fluent/lower query plan caching and derives:

* `Hash`
* `Eq`
* `PartialEq`

This is a cleanup and authority-sharpening change. It is not intended to change query semantics, broaden cache scope, or alter SQL cache-key behavior.

---

## Problem Statement

Current cache-key shape is uneven across layers.

### Already clean

SQL upper caches already use explicit derived-hash keys:

* `SqlCompiledCommandCacheKey`
* `SqlSelectPlanCacheKey`

These are straightforward map keys and do not appear to be the main maintenance concern.

### Current complexity hotspot

The shared lower query-plan cache uses a derived map key at the outer level, but the structural query identity fed into that key is produced through a bespoke fingerprinting path in:

* `db/query/intent/model.rs`

That path currently includes:

* one top-level structural fingerprint entrypoint
* many semantic traversal helpers
* several low-level byte-writing helpers
* enum-tag helpers

The issue is not the number of functions by itself. The issue is that the implementation currently fuses:

1. **semantic canonicalization**
2. **wire-format-like encoding**
3. **hash production**

into one authority surface.

That makes the code harder to reason about because the important behavior is not “how do we write bytes,” but “what makes two queries semantically identical for shared cache reuse.”

---

## Design Goal

Introduce a small, explicit, canonical structural query cache key type and make normalization into that key the owned semantic step.

Target shape:

```text
QueryModel
  -> normalize into StructuralQueryCacheKey
  -> derive(Hash, Eq, PartialEq)
  -> use as shared lower cache identity
```

This change should:

* make canonicalization explicit
* remove byte-level hashing helpers from semantic ownership
* reduce helper count and conceptual branching
* make cache identity inspectable and directly testable
* keep SQL upper caches unchanged unless a later cleanup chooses to align naming or structure

---

## Non-Goals

This slice does not aim to:

* redesign SQL cache keys
* change cache invalidation policy
* change schema fingerprint ownership
* broaden the set of query forms that are cacheable
* change query equivalence rules beyond making existing intended equivalence explicit
* introduce serialized on-disk cache-key stability guarantees
* use `derive(Hash)` directly across the entire live `QueryModel` tree

The last point is important. The design does **not** claim that the current raw query model types are suitable as-is for direct derived hashing. The whole point is to normalize first.

---

## Design Overview

### New concept: canonical structural key

Introduce a new internal canonical key family, rooted at:

* `StructuralQueryCacheKey`

This type represents the normalized semantic identity of a query for shared fluent/lower cache reuse.

It should be:

* small
* explicit
* alias-neutral where required
* ordering-stable where required
* free of incidental syntax-only distinctions
* composed only of fields that materially affect shared lower planning identity

Illustrative shape:

```rust
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub(crate) struct StructuralQueryCacheKey {
    pub access: AccessPathCacheKey,
    pub mode: QueryModeCacheKey,
    pub predicate: Option<PredicateCacheKey>,
    pub order: Option<OrderCacheKey>,
    pub distinct: DistinctCacheKey,
    pub projection: ProjectionCacheKey,
    pub grouping: Option<GroupingCacheKey>,
    pub having: Option<PredicateCacheKey>,
    pub consistency: ConsistencyCacheKey,
    pub limit: Option<u32>,
    pub offset: Option<u32>,
}
```

Field names and exact decomposition may vary, but the intended shape is:

* top-level query identity is built from normalized clause identities
* each clause identity is itself a small derived-hash struct or enum
* no byte encoder exists in this layer

---

## Authority Model

### Before

`cache_fingerprint()` implicitly owns:

* semantic normalization
* canonical structural encoding
* final hash output

### After

Authority becomes split cleanly:

#### Query model normalization owns:

* semantic equivalence shaping
* clause normalization
* canonical structural representation

#### Derived hash owns:

* map key hashing only

This creates a better authority boundary:

* semantic decisions are made in normalization code
* hashing becomes mechanical and uninteresting

That is the desired cleanup.

---

## Canonicalization Rules

This section defines what the normalization layer must own.

## 1. Predicate normalization

Normalization must preserve current intended shared-cache equivalence behavior.

Where current fingerprint logic already treats semantically equivalent predicates the same, the new key must do the same.

Expected responsibilities include:

* stable representation of comparison operators
* stable representation of literal values
* canonical field/value ordering where the current logic depends on it
* flattening and ordering of commutative conjunctions if that is already part of intended equivalence
* preserving non-commutative structure where semantics require it

Important constraint:

* this addendum should not silently widen equivalence classes beyond current intended behavior

So if the existing implementation does **not** currently treat two structurally different but logically equivalent predicates as identical, 0.82 should not opportunistically start doing so unless explicitly chosen and tested.

This slice is primarily cleanup, not semantic widening.

---

## 2. Projection normalization

Projection identity should reflect semantic output shape relevant to lower planning identity, while ignoring syntax-only distinctions that do not matter.

Likely responsibilities:

* alias-neutral identity where aliases do not affect lower planning reuse
* stable expression representation
* normalized selection ordering where output order matters
* explicit distinction between projection forms that affect planning shape

If alias handling today is intentionally ignored by the fingerprint, that behavior should move into the normalization layer explicitly.

---

## 3. Order normalization

Order identity must produce one stable representation for one semantic order spec.

Responsibilities:

* stable field/expression representation
* explicit direction representation
* elimination of incidental syntax-only variation
* retention of order item sequence where semantics require sequence sensitivity

The current `order_direction_tag` helper is a hint that this can become a small derived enum instead of manual tag writing.

---

## 4. Grouping normalization

Grouping identity should become an explicit canonical grouping key.

Responsibilities:

* stable grouping item representation
* preserved grouping sequence if that matters to planning identity
* explicit grouped-mode distinction

---

## 5. HAVING normalization

HAVING should reuse the same normalized predicate-key family if possible.

That is a desired cleanup because it prevents grouped predicate identity from drifting into its own parallel shape.

If HAVING has grouped-only representational constraints, they should be enforced at normalization construction time, not by introducing a second ad hoc hash-walk path.

---

## 6. Access path / query mode normalization

The current fingerprint path also encodes access-plan-related structure. That should become explicit structural data.

Examples of intended identity surfaces:

* load vs grouped vs other mode distinctions
* key access shape
* full scan vs indexed access shape
* distinct access-plan-relevant variants

This is where the new key must be careful not to capture volatile execution detail that belongs to a later runtime stage.

Only structural planning identity should be included.

---

## 7. Distinct / consistency / limit / offset

These should become ordinary derived-hash fields, not manual tags or byte writes.

Where they are optional, represent them directly as:

* `Option<u32>`
* small enums
* booleans or dedicated tiny enums

The low-level helper set strongly suggests these are currently being over-mechanized.

---

## Proposed API Shape

### Query model side

Replace the current “compute final fingerprint” framing with “construct canonical key.”

Illustrative API:

```rust
impl QueryModel {
    pub(crate) fn structural_cache_key(&self) -> StructuralQueryCacheKey {
        StructuralQueryCacheKey::from_query_model(self)
    }
}
```

Or equivalent constructor ownership:

```rust
impl StructuralQueryCacheKey {
    pub(crate) fn from_query_model(model: &QueryModel) -> Self { ... }
}
```

Preferred ownership rule:

* `QueryModel` may expose the entrypoint
* the canonical key type owns its own construction details

That makes the normalized form the named concept, not the hash.

---

## Shared session cache integration

Current shared lower cache key usage in `db/session/query.rs` should move from “store fingerprint-like structural identity” to “store canonical structural key.”

Illustrative intent:

```rust
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub(crate) struct QueryPlanCacheKey {
    pub schema_fingerprint: SchemaFingerprint,
    pub structural_query: StructuralQueryCacheKey,
}
```

This keeps the outer cache-key model consistent with existing SQL cache style:

* derived key structs at the map boundary
* explicit subkeys underneath

---

## Migration Plan

## Phase 1 — Introduce canonical key in parallel

Add:

* `StructuralQueryCacheKey`
* clause-level normalized key types
* normalization constructors

Keep the existing fingerprint path temporarily.

Use dual computation during development/tests to verify behavioral parity where feasible.

Examples:

* compare cache hit behavior in targeted tests
* compare equality classes on representative query fixtures
* optionally compare legacy fingerprint equality expectations against new key equality expectations

The main objective of this phase is to prove that the new key is a re-expression of intended identity, not a semantic drift.

---

## Phase 2 — Switch shared lower cache to canonical key

Update shared lower cache lookup/store to use:

* `StructuralQueryCacheKey`

Remove dependence on the old manual fingerprint result from the live cache path.

At this stage, the old fingerprint machinery may still exist behind tests or temporary parity scaffolding.

---

## Phase 3 — Delete manual hash encoder path

Remove:

* manual byte-write helpers
* semantic `hash_*` walkers whose only remaining purpose was the old encoder
* enum-tag helpers that no longer serve any live normalization purpose

The desired end state is:

* semantic helpers are all named as normalization constructors, not hash writers
* hashing is fully derived from canonical key structs

---

## Expected Code Shape Change

### Before

Conceptually:

* `cache_fingerprint()`
* many `hash_*` semantic traversal helpers
* many `xxh3_write_*` low-level output helpers

### After

Conceptually:

* `structural_cache_key()`
* fewer `normalize_*` semantic helpers
* small explicit clause key structs/enums
* no low-level byte writers in this area

That is the main simplification.

---

## Why This Is Better

## 1. The key concept becomes visible

Today the important artifact is implicit. After this change, the important artifact is a real type:

* `StructuralQueryCacheKey`

You can inspect it, debug it, and reason about it directly.

## 2. Semantic ownership is clearer

Normalization code is obviously semantic. Hashing code is not. Right now those are mixed.

## 3. Testing gets better

You can test canonicalization directly:

* two queries should normalize to the same key
* two queries should normalize to different keys

That is more precise than testing opaque final hash behavior.

## 4. Lower maintenance cost

Adding a new query-shape axis becomes:

* extend key type
* extend normalization
* add equivalence tests

instead of:

* thread tags through manual byte encoders
* maintain parallel walker/encoding conventions

## 5. Reduced accidental complexity

The current helper count is inflated because the design has no explicit midpoint between AST/model and final hash. The new canonical key is that midpoint.

---

## Risks

## 1. Semantic drift during migration

The biggest risk is accidentally changing the equivalence class of cached queries.

Mitigation:

* parity-oriented tests
* staged rollout
* conservative normalization rules
* no opportunistic “logical optimization” disguised as cleanup

## 2. Overfitting the canonical key to current internals

If the new key mirrors too much of the current live model shape, it will not actually simplify authority.

Mitigation:

* keep the canonical key small
* exclude incidental syntax-only structure
* include only cache-relevant identity

## 3. Underfitting the key

If the key omits a planning-relevant distinction, cache reuse may become unsound.

Mitigation:

* review each current fingerprint dimension explicitly
* preserve all planning-relevant distinctions from the live path unless deliberately removed with proof

---

## Testing Plan

The tests for this slice should focus on normalized identity, not just final cache behavior.

## Unit tests

Add direct tests for canonicalization:

* identical query model -> identical `StructuralQueryCacheKey`
* alias-only differences -> same key where intended
* order-direction differences -> different key
* grouped vs non-grouped -> different key
* meaningful projection differences -> different key
* relevant predicate structure differences -> different key

## Parity tests

For a curated query fixture set:

* compare old and new equality partitions
* validate that query pairs previously intended to collide still collide
* validate that query pairs intended to differ still differ

## Session cache tests

Retain or add lookup/store tests in shared lower session cache verifying:

* repeated structurally identical query uses same cache entry
* structurally distinct query does not reuse entry

---

## Recommended File Ownership

Suggested ownership shape:

* `db/query/intent/model.rs`

  * entrypoint exposure only, or thin integration only

* new sibling module for canonical key ownership, for example:

  * `db/query/intent/cache_key.rs`
  * or `db/query/intent/model/cache_key.rs`

This is preferable to continuing to grow `model.rs` as a mixed semantic/runtime-utility owner.

Recommended content ownership in the new module:

* `StructuralQueryCacheKey`
* clause-level cache-key structs/enums
* normalization constructors
* direct normalization tests

That gives the new concept a real home.

---

## Naming Guidance

Prefer names that say what the structure is, not what hashing does.

Good:

* `StructuralQueryCacheKey`
* `PredicateCacheKey`
* `ProjectionCacheKey`
* `normalize_predicate`

Avoid:

* `hash_query_predicate`
* `write_projection_tag`
* `encode_query_mode`

The new design should make “hash” mostly disappear from this layer’s vocabulary.

---

## Compatibility Notes

SQL cache keys do not need to change for this addendum.

However, after this lands, the overall cache-key story becomes much cleaner:

* SQL upper cache keys are explicit derived key structs
* shared lower cache key is also an explicit derived key struct
* semantic normalization only exists where it is actually needed

That is the architectural alignment this addendum is aiming for.

---

## Remaining Manual Hashing After This Slice

Even after this addendum lands, a few manual hashing boundaries will still
exist. That is acceptable, but `0.82` should describe them explicitly so future
cleanup work does not blur cache identity, fingerprint authority, and runtime
hashing into one bucket.

### `value/hash.rs` remains the main cleanup candidate

`crates/icydb-core/src/value/hash.rs` is still the largest home-rolled
recursive hasher in the tree.

That is currently justified because:

* value hashing must stay canonical across heterogeneous runtime value shapes
* map ordering and nested value framing are semantic, not incidental
* this path is reused outside one narrow cache table

But if future cleanup wants to simplify one remaining manual hashing surface,
this is the first place to look.

### `query/intent/cache_key.rs` should only grow by reusing canonical shapes

`crates/icydb-core/src/db/query/intent/cache_key.rs` should continue moving
toward reuse of subsystem-owned canonical key/fingerprint authorities where they
already exist.

That means future cleanup may:

* reuse canonical predicate identity instead of reopening predicate digest logic
* reuse other planner-owned canonical key shapes where they are already stable

That does **not** mean:

* hashing `Debug` output
* introducing generic serialization as cache identity
* deriving `Hash` directly on the live query model tree

The right simplification direction is still:

* canonicalize first
* keep cache-key structs explicit
* let derived hashing stay mechanical at the map boundary

### Authority-boundary hashers should remain explicit

`crates/icydb-core/src/db/query/fingerprint/hash_parts.rs` and
`crates/icydb-core/src/db/schema/fingerprint.rs` are still manual hashing
surfaces.

That is expected. Those files are authority boundaries for deterministic plan,
continuation, and schema identity, not just cache-table convenience code.

So `0.82` should not try to erase manual hashing everywhere. It should only
remove manual hashing where semantic normalization and cache identity were
previously mixed together without needing to be.

---

## Decision

For 0.82, the preferred cleanup direction is:

* **do not** derive `Hash` directly on the full live query model tree
* **do** introduce an explicit canonical `StructuralQueryCacheKey`
* **do** normalize query semantics into that key
* **do** derive `Hash` on the normalized key family
* **do** delete the manual byte encoder path after parity is proven

This keeps the semantic contract explicit and materially reduces maintenance complexity in the shared fluent/lower cache identity path.

---

## Implementation Summary

The intended implementation slice is:

1. add canonical structural query cache-key types
2. move semantic identity logic into normalization constructors
3. switch shared lower cache to normalized key usage
4. remove manual encoder and byte-write helpers

That is the cleanest route out of the current mixed-responsibility fingerprint path.

---

## Exit Criteria

0.82 addendum is complete when all of the following are true:

* shared lower query-plan cache uses `StructuralQueryCacheKey`
* canonicalization logic is explicit and test-covered
* no low-level `xxh3_write_*` helpers remain in this path
* old semantic `hash_*` helper naming is removed from live ownership
* cache identity behavior remains equivalent to intended current semantics
