# 0.196 SQLite-Informed Ordered Read Pushdown

Status: implementation started; full-matrix closeout still pending.

0.196 converts the SQLite-comparison audit into an implementation line. The
first implementation target is not "be more SQLite" in a feature-count sense.
It is to apply one SQLite discipline that maps cleanly to IcyDB's IC-native
query engine: do not read, decode, sort, or window more rows than a bounded
ordered query actually needs when the selected access path can safely prove an
earlier stop point.

The audit inputs for this line live beside this document:

- `audit-results.md`
- `findings.json`

## Context

The 0.196 audit found that the next high-value optimization area is bounded
ordered reads and pagination:

- cursor continuation is documented as post-access;
- several `ORDER BY ... LIMIT 1/3/10` matrix rows still read 512 rows;
- materialized order/window work is visible in the performance artifacts;
- exact index cardinality and covering/index-route machinery already exist;
- 0.195 already addressed sparse `IN` liveness and should not be reopened as
  the main target.

Representative current matrix rows from
`testing/integration/target/perf-hotspots/sql_perf_195_1_liveness_full_matrix.md`:

| Scenario | Total | Execute | data_store.get | Rows |
| --- | ---: | ---: | ---: | ---: |
| `journaled_user.select.wide.name_range.age_asc.limit10` | 23,540,542 | 23,281,700 | 512 | 10 |
| `heap_user.select.wide.name_range.age_asc.limit10` | 23,516,234 | 23,264,122 | 512 | 10 |
| `journaled_user.select.narrow.name_range.age_asc.limit1` | 22,690,411 | 22,437,786 | 512 | 1 |

These are not evidence of a corruption bug. They are evidence that bounded
ordered query execution still has an avoidable materialization/read boundary in
some shapes.

## Problem

IcyDB currently has several correct but expensive bounded-read shapes:

```text
access path produces candidates
  -> rows are read/materialized
  -> residual/filter/order/window/cursor is applied
  -> LIMIT returns a small page
```

That shape is safe but leaves performance on the table. SQLite's useful
principle here is not "add a general cost-based optimizer"; it is:

```text
when a B-tree/index can produce the required order or continuation boundary,
seek/stream there directly and stop as soon as the bounded result is proven
complete
```

For IcyDB, the IC-native version must be deterministic, bounded, compatible
with read admission, and explicit about when the engine cannot safely push a
boundary down.

## Design Goal

Introduce an ordered-read pushdown boundary:

1. The planner records whether the selected access path can deliver the
   requested order, cursor boundary, and page limit.
2. Execution pushes cursor/page boundaries into index/primary-key access only
   when the proof is deterministic.
3. Execution stops after `limit + lookahead` rows when the ordered access path
   proves no later row can precede the returned page.
4. Non-pushable shapes remain correct and get explicit explain/perf diagnostics
   instead of silently pretending to be optimized.

The first slice should improve eligible cursor/page shapes and make ineligible
materialized-order shapes visible. It should not change query semantics,
recovery, persistence, or public-read safety.

## Proof Burden

0.196 is deliberately adversarial toward easy implementation shortcuts. A patch
does not close this line by adding a flag, improving one hand-picked query, and
showing a small before/after table.

Closeout must prove all of the following:

- hard-cut behavior for any cursor/cache/protocol change;
- no semantic weakening;
- no public read-admission bypass;
- full deterministic matrix before/after deltas;
- route family and route outcome for every deterministic matrix scenario;
- no cherry-picked performance headline;
- target proof cases that actually exercise pushed/limit-stopped execution.

Focused matrix output is iteration evidence only. The release/closeout
performance claim must be based on the full deterministic matrix delta artifact.

## First Slice Scope

0.196.0 should be narrow:

1. Add explicit plan/runtime facts for:
   - `order_source`;
   - `order_materialized`;
   - `cursor_pushdown`;
   - `limit_stop_after`;
   - `index_entries_visited`;
   - `index_ranges_opened`;
   - `row_presence_checks`;

   - `rows_decoded`;
   - `rows_matched_after_residual`;
   - `rows_returned`;
   - `lookahead_rows_read`;
   - `tombstones_skipped`.
2. Push cursor boundaries into index-backed or primary-key-backed ordered plans
   only when:
   - the selected access path already produces the requested order;
   - the cursor token boundary is for the same entity, index/order contract, and

   - all tie-breaker fields needed for deterministic resume are present;
   - live/tombstone behavior cannot cause duplicate or skipped rows;
   - the executor can stop after the page plus any required lookahead.
3. Keep materialized-order routes correct. For routes that cannot push down,
   report why:
   - `order_not_access_compatible`;

   - `residual_filter_requires_unbounded_candidate_scan`;
   - `missing_tie_breaker`;
   - `unsupported_access_kind`.
4. Run before/after matrix evidence and report the delta.

The first slice does not have to solve every top matrix row. In particular,
`WHERE name >= 'a' ORDER BY age ASC LIMIT N` may require an order-compatible
access path or a later deterministic planner rule. 0.196.0 should still make
that route explainable and measurable.

## Hard-Cut Compatibility Stance

This line follows the repository pre-1.0 hard-cut rule.

Preferred implementation:

- no persisted-format change;
- no cursor-token format change;
- no public API break;
- no compatibility fallback;
- in-memory plan/cache method versions may bump if plan facts change.

If implementation proves that cursor-token shape must change, 0.196 must
hard-cut the cursor protocol:

- replace the current cursor encoding with the latest encoding;
- remove legacy cursor decode fallback;
- fail old cursor tokens clearly with the existing invalid-cursor/read boundary
  diagnostic path;
- update cursor contract docs in the same slice;
- add tests proving old-shape tokens are rejected, not silently reinterpreted;
- add one explicit fixture token from the previous shape and assert that it
  fails through the documented invalid-cursor/read-boundary path. Do not
  regenerate this fixture during the test.

If implementation unexpectedly needs a persisted row/index/schema/journal
format change, stop. That is out of scope for 0.196.0 and requires a separate
persisted-format design update. If such a later storage change is accepted
before 1.0, it should also be a hard cut: latest format only, fail closed on old
bytes, no runtime fallback reconstruction from generated models.

### Fallback Taxonomy

0.196 distinguishes three fallback classes.

1. Semantic execution fallback: allowed.

   If an otherwise valid query cannot prove ordered-read pushdown, execution
   may use the existing materialize/filter/order/cursor/window path. This is
   not compatibility behavior. It must be labeled in EXPLAIN and perf
   attribution.

2. Public read-admission fallback: conditionally allowed.

   If admission accepted the query only because `limit_stop_after` or
   `cursor_pushdown` was proven, runtime must not silently fall back to an
   unbounded post-access route. If runtime cannot honor the pushdown proof, the
   engine must either:

   - prove the fallback route is independently admitted by the same public-read
     policy; or
   - fail closed with the existing read-boundary diagnostic.

   It must not silently execute an unadmitted materialized fallback.

3. Legacy/protocol compatibility fallback: forbidden.

   If the cursor protocol changes, old cursor tokens are rejected. There is no
   legacy cursor decode path, silent reinterpretation, or compatibility shim. If
   any persisted format changes, 0.196.0 stops and a separate persisted-format
   design is required.

### Cache And Version Hard Cut

If plan-cache identity, explain facts, cursor compatibility class, or
access-route selection semantics become part of cached plan behavior, the
relevant cache key or method version must be bumped. The old cache entry must
miss; no dual-read compatibility path is allowed.

Tests must prove a cached plan prepared before the version bump is not reused
after the bump.

## Authority Split

### Planner Authority

The planner may prove and cache stable facts:

- selected access path;
- index/primary-key identity;
- requested order contract;
- tie-breaker contract;
- cursor boundary compatibility class;
- whether order is access-produced or materialized;
- whether covering facts allow avoiding row reads;
- whether a limit stop is semantically possible.

The planner must not cache generation-dependent facts:

- current row counts;
- current liveness;
- current tombstone/live overlay state;
- current index tail shape;
- exact runtime row-read counts.

### Runtime Authority

Execution owns current-state facts:

- current generation/read view;
- live/tombstone checks;
- runtime cursor seek position;
- page lookahead;
- actual rows read and returned;
- runtime fallback reason if a static pushdown proof cannot be honored.

Runtime uncertainty may use the existing correct post-access behavior only when
that fallback is independently admitted for the active read lane. For public
reads admitted because pushdown was proven, uncertainty must fail closed unless
the fallback route is also admitted.

## Definitions

### Access-Compatible Order

An order is access-compatible when the selected index or primary-key traversal
already yields rows in the requested order, including deterministic tie-breaker
fields.

Examples:

- primary-key ascending access for `ORDER BY id ASC`;
- index `(age, id)` for `ORDER BY age ASC, id ASC`;
- exact equality prefix plus remaining index components matching requested
  order.

Non-examples:

- range on `name` with `ORDER BY age` when the selected access path is not age
  ordered;
- order involving an expression unless an expression index/order contract
  proves the same ordering;
- order without a deterministic primary-key tie-breaker.

### Cursor Pushdown

Cursor pushdown means execution uses the cursor boundary to seek the access
stream before reading candidate rows.

It is not enough to apply the cursor after sorting or after materializing a
candidate set. That behavior remains the fallback path and should be labeled as
post-access cursor handling.

### Cursor Token Contract

Before a cursor can be pushed down, the boundary must be checked against:

- entity id;
- accepted schema identity or schema generation;
- query shape or cursor contract fingerprint;
- selected access kind;
- index id or primary-key route id;
- order field list;
- order direction for every field;
- implicit or explicit primary-key tie-breaker;
- last emitted ordered key tuple;
- last emitted primary key;
- page direction;
- cursor protocol version if the protocol changes.

Add rejection tests for each mismatch. If cursor token shape does not change,
still add tests proving:

- unsupported current tokens do not get pushdown accidentally;
- malformed tokens fail through the invalid-cursor/read-boundary path;
- tokens with mismatched order/index/entity/direction are rejected or fall back
  only when public read admission permits fallback.

### Limit Stop

Limit stop means execution can stop scanning once it has enough rows to return
the requested page plus any required lookahead for continuation. It is valid
only when no later unvisited row can sort before an already returned row.

`limit_stop_after` should not be represented as only a boolean. The conceptual
shape is:

```text
LimitStop {
  possible: bool,
  returned_limit: usize,
  lookahead: usize,
  stopped_after_matches: Option<usize>,
  stopped_after_index_entries: Option<usize>,
  disabled_reason: Option<LimitStopDisabledReason>,
}
```

Residual filters and tombstones can make "stop after limit + 1" misleading. An
eligible route may stop after enough live matching rows while still visiting
more index entries. The delta report must make that visible.

### Materialized Order

Materialized order means the selected access path does not produce the requested
order, so the executor must collect candidates and order/window them later.
Materialized order is not a bug, but it is a performance fact that should be
visible in EXPLAIN/perf attribution and read admission.

## Proposed Flow

### Existing High-Level Flow

```text
SQL/fluent input
  -> shared filter contract
  -> planner
  -> prepared plan
  -> access execution
  -> materialize/filter/order/cursor/window
  -> page result
```

### 0.196 Flow

```text
SQL/fluent input
  -> shared filter contract
  -> planner with ordered-read facts
  -> prepared plan
  -> runtime pushdown check
  -> access seek/stream when safe
  -> stop after bounded page when safe
  -> fallback materialize/filter/order/cursor/window when not safe
  -> page result
```

The fallback path is not legacy compatibility. It is the current correct
execution strategy for shapes that cannot prove pushdown. If a protocol/format
changes, that protocol/format is still hard-cut as described above.

## Candidate Route Classes

### Eligible in 0.196.0

- primary-key ordered scans with cursor/page boundaries;
- secondary-index scans where the selected index order exactly matches the
  requested order after equality-prefix stripping;
- existing covering plans where row-presence checks remain explicit;
- limit/page queries where continuation lookahead is already supported or can be
  added without cursor-format churn.

### Diagnostic-only in 0.196.0

- access path filters by one field and orders by another incompatible field;
- materialized sort/window routes;
- residual filters that can force unbounded ordered-index scanning;
- expression order routes without a proven expression-index order contract;
- routes missing deterministic tie-breaker fields.

### Follow-up candidates

- order-compatible index selection when both filter-first and order-first routes
  exist;
- exact-cardinality-assisted admission for order-first residual scans;
- covering-index expansion for projection-only pages;
- offset-addressable row decode from the audit's serialization findings.

Every deterministic matrix scenario must report both a `route_family` and a
`route_outcome`.

`route_family`:

- `primary_order`;
- `secondary_order`;
- `equality_prefix_ordered_suffix`;
- `materialized_order`;
- `post_access_cursor`;
- `residual_filter_ordered_scan`;
- `incompatible_filter_first_order`;
- `unsupported_access_kind`;
- `not_ordered_or_not_paginated`;
- `failed_or_not_executed`.

`route_outcome`:

- `pushed`;
- `eligible_but_runtime_fallback`;
- `eligible_but_not_pushed`;
- `materialized`;
- `post_access`;
- `residual_unbounded`;
- `missing_tie_breaker`;
- `unsupported`;
- `unchanged_or_not_applicable`;
- `failed`.

The delta report must aggregate instruction deltas by:

1. `route_family`;
2. `route_outcome`;
3. `(route_family, route_outcome)`.

For every scenario with outcome `materialized`, `post_access`,
`eligible_but_runtime_fallback`, `residual_unbounded`, `missing_tie_breaker`, or
`unsupported`, the report must include a stable reason enum.

Every material improvement claim must identify whether the target route changed.

Allowed classifications:

- route changed from materialized/post-access to pushed;
- same route but fewer row reads;
- same route but lower planner/executor overhead;
- measurement noise;
- unrelated change.

Do not count a scenario as a 0.196 pushdown win unless either:

1. route facts changed to pushed/limit-stopped execution; or
2. attribution shows the expected lower access/materialization counters.

## Correctness Invariants

0.196 must preserve these invariants:

- index path results equal scan/materialized fallback results;
- cursor pages have no duplicate or skipped rows;
- ordering is deterministic for equal ordered fields;
- tombstoned rows are never returned;
- inserted/deleted rows between pages follow the documented cursor contract;
- pushdown never bypasses residual filters;
- pushdown never weakens read admission;
- fallback remains correct when pushdown proof is unavailable;
- observability/explain fields do not affect behavior;
- cached plans do not embed runtime generation/liveness facts.

- `ICYDB_SQL_PERF_MATRIX_DELTA_FOCUSED_KEYS`;

### Success Threshold

For a targeted pushdown slice:

- eligible focused scenarios should show lower `data_store.get` and lower
  execute instructions;
- full deterministic matrix should have no material regression:
  - no new failures;
  - no failure-key symmetric difference unless intentionally explained;
  - no scenario regression >= 10% and >= 100k instructions unless explicitly
    accepted;
  - aggregate common-scenario total should stay neutral or improve;
- sparse `IN` 0.195 guard scenario should remain neutral;
- every material improvement claim should state whether the route changed,
  access/materialization counters dropped, or the movement is measurement noise.

For a diagnostic-only first slice:

- full matrix should be neutral;
- explain/perf attribution tests must prove new fields are populated;
- no performance win should be claimed.

## Implementation Notes

- Prefer adding plan facts and a narrow runtime pushdown gate over reshaping the
  whole planner.
- Keep the current materialized-order path as the correctness fallback.
- Do not add a general cost-based planner.
- Do not make exact cardinality authoritative unless the generation is proven
  synchronized.
- Do not let read admission accept a query only because a planner estimate says
  it is probably cheap.
- Keep SQL and fluent route facts aligned.
- Keep covering row-presence checks explicit.
- Keep cache keys/versioning explicit if plan facts change.
- Avoid allocating debug strings in the success path merely to populate
  diagnostics.
- Use stable reason enums rather than matching error strings in tests.
- The first implementation must prefer adding one access-seek/stream adapter
  and one planner/runtime proof gate over restructuring the whole executor
  pipeline. If more than one executor route family must be rewritten to land
  the first proof case, stop and split.

## Validation Plan

Minimum validation for implementation slices:

- focused planner/executor tests for cursor pushdown eligibility;
- primary-key forward/reverse cursor tests;
- secondary-index order-compatible cursor tests;
- equality-prefix plus ordered-suffix tests;
- tombstone/live overlay tests;
- residual-filter tests;
- materialized-order fallback tests;
- public-read admission tests proving runtime fallback cannot bypass admission;
- malformed and mismatched cursor-token tests even if token shape does not
  change;
- stale or incompatible cursor token tests if token shape changes;
- cache/method version hard-cut tests if cached plan semantics change;
- cursor mutation edge tests covering:
  - last emitted row deleted before next page;
  - unseen row deleted before next page;
  - new row inserted before the cursor boundary;
  - new row inserted after the cursor boundary;
  - row with equal ordered key inserted between pages;
  - ordered/indexed field updated before next page, if updates can affect that
    index;
  - cursor boundary at first key;
  - cursor boundary at last key;
  - exactly `limit` rows;
  - exactly `limit + 1` rows;
  - empty page after final cursor;
  - ASC and DESC versions of the same boundary cases;

- `cargo fmt --all`;

## Non-Goals

- No embedded SQLite dependency.

- No joins or broad SQL surface expansion.

## Open Questions

- No cursor-token format change was needed for this slice. The hard-cut
  cursor-token branch remains documented for a future protocol-changing slice.
- 0.196 now exposes scalar-load route family/outcome/reason facts on canonical
  execution descriptor roots. A top-level structured public EXPLAIN object for
  the full matrix attribution shape still remains available to the 0.196 audit
  implementation line or any overlapping merged slice.
- The first proof cases are now explicit in the full matrix delta: bucket-range
  `bucket_label_asc` and age-range `age_asc`/`age_desc` scenarios changed from
  eligible to pushed with lower row-store and index work.
- The matrix classifier now marks Blob `ORDER BY bucket, id` scenarios as
  `secondary_order` / `missing_tie_breaker` because the declared Blob ordered
  metadata index is `(bucket, label, id)`. Blob `ORDER BY bucket, label, id`
  remains the pushable route family.
- Matrix samples now carry `limit_stop_after` as a structured object rather than
  an implied boolean, so pushed routes expose returned limit, lookahead, observed
  matches/index entries, and non-pushed routes expose a disabled reason.
- Verbose EXPLAIN now exposes `diag.r.limit_stop_after` from route facts. The
  JSON execution tree exposes bounded fetch proofs as an access-root
  `limit_stop_after` node property plus scalar-load route family/outcome/reason
  properties; it still has no new top-level public object for the structured
  matrix shape.
- Exact cardinality is used for public read admission only when selected
  primary-key access proves a finite key-count bound. Broader cardinality
  ranking remains active in the SQLite-comparison optimizer/canonicalization
  workstream.

## Implementation Sequencing Guard

The first implementation patch should not optimize every route family.

Preferred sequence:

1. Add route facts and full-matrix delta helper.
2. Prove primary-key ordered limit/page pushdown.
3. Prove primary-key cursor continuation.
4. Extend to secondary-index ordered routes.
5. Extend to equality-prefix ordered suffix routes if the same seek/stream
   adapter supports it.

A PR may combine adjacent steps only when the same small adapter and proof gate
cover them without broad executor restructuring.

The first pushed implementation may close 0.196.0 only if:

- at least one deterministic order-compatible route shows pushed or
  limit-stopped execution;
- full deterministic matrix delta artifacts are produced;
- all common successful scenarios have route family and route outcome;
- no public read-admission fallback bypass is possible;
- no persisted-format change lands.

## Codex Implementation Addendum

Important: this is a hard-cut, full-matrix-evidence task.

Do not implement legacy cursor compatibility. If cursor-token shape changes,
replace the encoding, delete old decode fallback, reject old tokens clearly, and
add tests proving old tokens are rejected.

Do not silently fall back to an unadmitted post-access route. If public read
admission accepted the query because pushdown was proven, and runtime cannot
honor pushdown, either prove the fallback route is independently admitted or
fail closed with the existing read-boundary diagnostic.

Before claiming a performance win, produce a full deterministic before/after
matrix comparison. Focused matrix output is not sufficient for closeout.

Add or use a comparison helper that produces:

- full delta JSON for every scenario key in the union of before/after outputs;
- full delta Markdown summary;
- common-success aggregate;
- new/resolved failure symmetric difference;
- top 50 improvements;
- top 50 regressions;
- route family and route outcome for every scenario;
- explanation of whether each target route actually changed.

Do not claim a 0.196 pushdown win unless route facts show pushed/limit-stopped
execution or attribution counters show the expected lower
access/materialization work.
