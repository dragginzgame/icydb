## 1. Objective

Ship `EXPLAIN` on top of the stabilized `0.41` execution vocabulary so
debugging output is a stable public contract rather than a moving internal
projection.

`0.42` is an observability/repr release, not an execution-shape expansion.

---

## 2. Non-Goals

- no new access/route operator additions in `0.42`
- no planner architecture redesign
- no cost model or join/distributed execution work
- no behavioral changes to query semantics

---

- `0.41` explain-blocking execution slices A-E are complete
- execution vocabulary is frozen before `EXPLAIN` surface freeze
- route/executor contracts already expose ordering, pushdown, cursor, and
  bounded-seek signals needed for explain projection

## 4. Canonical Explain Contract

`EXPLAIN` in `0.42` is a serialization of execution nodes.

Pipeline model:

`planner -> access plan -> execution nodes -> EXPLAIN output`

### 4.1 ExecutionNodeDescriptor

Define one canonical descriptor used by all explain surfaces.

Core fields:

- `node_type`
- `children`
- `access_strategy` (with index details when present)
- `predicate_pushdown` (index prefilter details)
- `residual_predicate`
- `projection`
- `ordering_source`
- `limit`
- `cursor`
- `covering_scan`
- `rows_expected` (optional; nullable/omitted when unknown)

### 4.2 Node Vocabulary (Frozen in 0.42)

- Access: `ByKeyLookup`, `ByKeysLookup`, `PrimaryKeyRangeScan`,
  `IndexPrefixScan`, `IndexRangeScan`, `FullScan`, `Union`, `Intersection`
- Predicate: `IndexPredicatePrefilter`, `ResidualPredicateFilter`
- Ordering: `OrderByAccessSatisfied`, `OrderByMaterializedSort`
- Distinct: `DistinctPreOrdered`, `DistinctMaterialized`
- Projection: `ProjectionMaterialized`, `ProjectionIndexOnly`
- Cursor/Limit: `LimitOffset`, `CursorResume`, `TopNSeek`,
  `IndexRangeLimitPushdown`
- Aggregates: `AggregateCount`, `AggregateExists`, `AggregateMin`,
  `AggregateMax`, `AggregateFirst`, `AggregateLast`, `AggregateSum`,
  `AggregateSeekFirst`, `AggregateSeekLast`
- Grouped: `GroupedAggregateHashMaterialized`,
  `GroupedAggregateOrderedMaterialized`
- Pushdown: `SecondaryOrderPushdown`

## 5. Surface Plan

### 5.1 `EXPLAIN` (base)

- human-readable tree over `ExecutionNodeDescriptor`
- minimal stable fields
- deterministic ordering

### 5.2 `EXPLAIN VERBOSE`

- base surface plus rejection reasons and route/hint diagnostics
- explicit pushdown eligibility outcomes

### 5.3 `EXPLAIN JSON`

- machine-consumable serialization of the same descriptor tree
- stable key naming and deterministic child ordering

All three surfaces must share one descriptor projection path.

---

## 6. Serialization Rules

- deterministic output for equivalent plans
- no hidden inference in rendering layer
- no surface-specific node renaming
- include explicit booleans/variants for pushdown, seek, and materialization
  decisions
- preserve forward compatibility by allowing additive optional fields only

---

## 7. Implementation Slices (`0.42`)

1. Descriptor freeze
   - introduce/finalize canonical `ExecutionNodeDescriptor`
   - route existing explain projections through descriptor assembly
2. Base explain rendering
   - text tree output from descriptor
3. Verbose expansion
   - pushdown eligibility and rejection metadata
4. JSON surface
   - stable serialized schema over descriptor
5. Test freeze
   - snapshot and matrix coverage to lock vocabulary and hierarchy

---

## 8. Testing and Freeze Gates

- descriptor projection unit tests for each node family
- route/planner explain parity tests for key shapes:
  - pk access
  - secondary prefix/range access
  - `IndexMultiLookup`
  - ordered + top-N seek
  - aggregate seek terminals
  - grouped materialized strategies
- snapshot tests for text and JSON outputs
- regression tests to prevent node rename/hierarchy drift

---

## 9. Exit Criteria

- one canonical descriptor path powers all explain outputs
- node names and hierarchy are frozen by snapshots
- verbose and JSON are stable and deterministic
- no remaining explain-blocking execution-shape work in `0.42`
