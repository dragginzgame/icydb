# 0.44 - Optimization Closure Pass

## Status

Draft  
Target: `0.44`  
Dependency: `0.43` scalar terminal baseline (complete)  
Last updated: 2026-03-07

---

## 1. Objective

`0.44` closes optimization surfaces introduced across `0.40` to `0.43` so the
runtime does not accumulate partially optimized behavior.

The closure target is consistent across planner, access, executor, and
diagnostics:

- stable pushdown eligibility
- stable fast-path execution
- visible EXPLAIN/trace diagnostics
- complete parity and safety coverage
- controlled complexity growth

`0.44` introduces no new user-facing query semantics.

---

## 2. Non-Goals

- no new query language terminals or planner feature families
- no new access-path variants
- no new EXPLAIN DTO mutation model
- no grouped query surface expansion beyond optimizer closure
- no broad planner architecture redesign

---

## 3. Closure Principles

### 3.1 Close Optimization Surfaces

Each feature/terminal must end in exactly one explicit state:

- pushdown implemented and tested
- executor fast path implemented and tested
- pushdown implemented with no terminal fast path required
- explicitly designated non-optimizable with rationale

This prevents implicit, partial optimization drift.

### 3.2 Stabilize Routing Ownership

Routing policy remains centralized in route/access contracts. Executor terminals
consume route output; they must not re-derive planning policy independently.

### 3.3 Preserve Layer Authority

Authority boundary remains:

`intent -> query/plan -> access -> executor -> index/storage`

- planner owns semantics
- access owns shape/capability
- executor owns runtime mechanics

No layer should recompute another layer's policy decisions.

### 3.4 Require Observability

Every optimization must be visible through:

- EXPLAIN execution descriptors
- `ExecutionTrace` optimization markers

## 5. Feature Closure Work

### 5.1 COUNT Closure

### Current state

- COUNT routing exists, but several shapes still run through generic row-aware
  flows.
- Fast-path behavior is not fully closed across access shapes.
- EXPLAIN/trace signaling is incomplete for COUNT-specific optimization use.

### `0.44` work

Phase 1 (safe):

- optimize `COUNT(*)` and `COUNT(primary_key)` for primary-key 1:1 row shapes:
  - `ExecutionPathKind::FullScan`
  - `ExecutionPathKind::KeyRange`
- bypass row decode/projection/materialization in executor terminal path

Implementation status (2026-03-07):

- complete for non-distinct, no-residual-predicate PK-safe single-path shapes
  (`FullScan`, `KeyRange`) in `aggregate::terminals`, with explicit test-only
  fast-path hit diagnostics for PK-cardinality COUNT routing

Phase 2 (guarded):

- evaluate `IndexPrefix` and `IndexRange` COUNT paths only with explicit
  stale-key/missing-row safety gates
- `COUNT(field)` and `COUNT(nullable_field)` are excluded from fast-path
  closure in `0.44`

Implementation status (2026-03-07):

- complete: route COUNT planning now streams strict-compatible
  index-prefix/index-range shapes through `AggregateFoldMode::ExistingRows`,
  keeps strict-uncertain residual shapes materialized, and projects
  count-specific fold-mode diagnostics through aggregate EXPLAIN descriptor
  metadata (`node_properties.count_fold_mode`)
- complete (post-`0.44.0` follow-up): index-range COUNT routes now inherit
  bounded index-range `LIMIT` pushdown fetch specs under route eligibility
  gates, with parity coverage for stale-leading `MissingRowPolicy::Ignore`
  windows and bounded scan-accounting assertions

### 5.2 BYTES Completion

### Current state

- `bytes()` semantics are correct, but some paths still perform avoidable work.

### `0.44` work

- add executor fast paths where payload lengths are directly available
- avoid decode/materialization where correctness is unchanged
- add explicit EXPLAIN/trace optimization markers for bytes fast-path use

Implementation status (2026-03-07):

- complete: safe PK full-scan/key-range `bytes()` terminal fast path landed
  in `load::terminal` (non-distinct, no residual predicate, PK-order/no-order),
  with explicit eligible-vs-fallback hit-marker regression coverage for both
  full-scan and key-range shapes
- complete (post-`0.44.1` follow-up): unordered secondary scalar `bytes()`
  shapes (`no predicate`, `no distinct`, `no explicit order`) now fold
  persisted payload lengths from ordered key-stream traversal without entity
  materialization, with explicit parity + hit-marker coverage and ordered-shape
  bypass guards
- complete: BYTES fast-path diagnostics counters are test-thread local to keep
  parallel regression runs deterministic
- explicitly non-optimizable in `0.44`: residual-predicate, distinct, and
  order-sensitive BYTES shapes remain on canonical materialized fallback paths

### 5.3 LIMIT Pushdown Closure

### Current state

- LIMIT pushdown is present but not consistently applied to all eligible
  iterator shapes.

### `0.44` work

- enforce limit-aware early stop in eligible PK/index range scans
- LIMIT pushdown is only applied when the access order satisfies the planner's
  required query ordering guarantees
- preserve continuation/cursor semantics under pushdown

Implementation status (2026-03-07):

- complete: route capability gate requires planner logical pushdown
  eligibility for ordered index-range limit-pushdown shapes (while preserving
  no-order behavior), with explicit regression coverage for incompatible
  ordered index-range shapes that disable pushdown
- complete: index-range pushdown residual-cap, underfill-retry, and
  cursor-boundary behavior are covered through route + pagination trace tests

### 5.4 Covering Projection Closure

### Current state

- covering projection exists but routing and diagnostics are inconsistent.

### `0.44` work

- for index-covered projections, bypass row fetch/decode where safe
- preserve correctness for stale index and missing-row modes
- close EXPLAIN/trace visibility gap for covering projection fast paths

Implementation status (2026-03-07):

- complete: aggregate projection terminals now emit explicit fast-path hit
  diagnostics for both index-covered decode paths and constant covering
  projection paths, with eligibility-vs-fallback regression coverage
- complete: aggregate terminal EXPLAIN descriptors project covering-scan
  visibility for eligible index-only `EXISTS`/`COUNT` shapes
- strict-compatible vs strict-uncertain secondary `COUNT` predicate boundaries
  are now explicitly covered in descriptor regression tests

### 5.5 Grouped Aggregate Fast-Path Closure

### Current state

- grouped aggregate flows are mostly materialized.

### `0.44` work

- evaluate bounded grouped fast paths for `COUNT`, `MIN`, `MAX` only when
  ordering and shape guarantees are explicit in route/access contracts

Implementation status (2026-03-07):

- deferred: grouped terminal fast-path expansion is explicitly deferred
  post-`0.44` to preserve current grouped route complexity guardrails
  (no new executor routing branches in this closure pass) while maintaining
  deterministic materialized grouped execution behavior

## 6. Observability Contract

All landed optimizations must project through existing explain/trace surfaces:

- `ExplainExecutionNodeDescriptor`
- `ExplainAggregateTerminalPlan`
- `ExplainExecutionDescriptor`
- `ExecutionTrace` / `ExecutionOptimization`

EXPLAIN DTO construction remains immutable (`ExplainPlan::new(...)`-style
construction; no setter/builder mutation conversion in this line).

---

## 7. Safety Constraints

Optimizations must preserve behavior under:

- stale index entries
- deleted/missing primary rows
- strict vs ignore missing-row policy
- cursor resumption and continuation anchors
- optimizations must not alter result ordering guarantees

Every fast path must keep deterministic fallback:

- `fast path -> canonical fallback path`

---

## 8. Testing Expansion

Minimum closure matrix additions:

- COUNT:
  - pk full scan parity
  - pk range parity
  - index prefix/range parity where enabled
- LIMIT pushdown:
  - limit + pk range
  - limit + index range
  - limit + cursor pagination
- Covering projection:
  - fully index-covered projection parity
  - mixed projection requiring row fetch parity
- Cursor compatibility:
  - cursor + fast path
  - cursor + pushdown
