# 0.92 SQL Cache Revisited

## Thesis

`0.92` should revisit the SQL cache stack now that the compile/execute seam, shared expression pipeline, and attribution surfaces are stable, with the goal of making repeated SQL queries reuse one holistic prepared artifact as early as possible and proving the win against compile/execute attribution.

This is now a better-timed slice than it would have been before `0.91.5`, because the session and pipeline cleanup already removed a large amount of duplicate execution-wrapper noise around the SQL boundary. That means `0.92` can focus on cache ownership and repeated-query reuse instead of mixing cache changes with another runtime-wrapper cleanup pass.

## Problem Statement

The current SQL surface already has explicit compile caching and plan caching:

- `compile_sql_statement_with_cache(...)` reuses a previously compiled SQL artifact by raw SQL text, entity path, schema fingerprint, and surface kind, before parse/lower on a hit.
- `planned_sql_select_with_visibility(...)` may then reuse a session-owned SQL select-plan cache entry that stores `SharedPreparedExecutionPlan` plus projection labels/scales, and falls back to the shared lower query-plan cache when needed.

That means the system is no longer missing “a cache.” The open question is whether the current cache layering is still the best shape for the now-stabilized SQL architecture, or whether it carries unnecessary duplication, attribution noise, or avoidable planner/setup cost on repeat queries.

The key current architectural smell is no longer broad session/runtime duplication. The key smell is that `SELECT` still spans two different SQL-owned reuse stories:

- the normal cached path, where `CompiledSqlCommand::Select` carries `compiled_cache_key: Some(...)`
- the direct-lowered path, where the same command shape carries `compiled_cache_key: None` and `planned_sql_select_with_visibility(...)` falls back straight to the shared query-plan cache

That split is explicit in the code today:

- `CompiledSqlCommand::Select { query, compiled_cache_key: Option<...> }`
- `compile_sql_statement_for_authority(...)` still builds the uncached `None` form
- `planned_sql_select_with_visibility(...)` branches on that option to decide whether SQL owns the prepared-select reuse boundary or whether the shared query-plan cache becomes the top visible reuse story

That is the seam `0.92` should either justify or remove.

## Design Goal

Make repeated SQL execution reuse one prepared SQL artifact at the highest-value seam possible, while preserving:

- schema correctness
- visibility correctness
- projection contract correctness
- no executor-owned scratch-state caching
- clear attribution of which cache boundary actually produced the reuse win
- a sane cache-layer count that humans can still reason about during perf and correctness audits
- a post-`0.91.5` implementation that does not reopen session/runtime wrapper duplication just to move cache ownership

## Preconditions

`0.92` should not begin implementation work until baseline perf checks are captured for the current cache stack.

Before any code changes, the line should record baseline measurements for:

- repeated identical scalar `SELECT` execution
- repeated identical grouped `SELECT` execution
- compile-side instruction attribution
- execute-side planner/setup attribution
- SQL cache attribution showing current hits and misses across:
  - compiled-command cache
  - SQL select-plan cache
  - shared query-plan cache

The purpose of `0.92` is not “change the cache shape and hope it is better.” It is to compare a proposed simplified cache contract against a measured baseline and prove the reuse win.

## Non-Goals

`0.92` should not:

- change SQL semantics
- add new SQL features
- cache executor scratch state or live result pages
- weaken cache keys below current semantic boundaries
- fold grouped continuation state into the SQL compile cache
- reopen the recent session/query/pipeline cleanup with another broad runtime orchestration refactor

The current compile artifact is intentionally session-owned and generic-free, but still distinct from runtime execution state.

## Current Cache Stack

The current design has three relevant reuse layers:

1. `SQL compiled-command cache`
   - keyed by cache method version, surface kind, entity path, schema fingerprint, and raw SQL text
   - on hit, compilation skips parse/lower/semantic construction and returns a cached `CompiledSqlCommand`
2. `SQL select-plan cache`
   - for `SELECT`, keyed by compiled SQL key plus visibility
   - stores a `SharedPreparedExecutionPlan` and outward SQL projection contract (`columns`, `fixed_scales`)
3. `Shared lower query-plan cache`
   - used underneath the SQL layer when the SQL-specific plan cache is absent or misses
   - SQL attribution already tracks this separately from the two upper SQL caches

## Post-0.91.5 Baseline

The current repo state matters for this design:

- scalar and grouped load preparation already converge much more cleanly than they did before `0.91.5`
- grouped/scalar session execution wrappers are no longer the main source of repeated-query complexity
- the remaining ambiguity sits much closer to the SQL cache boundary itself

So `0.92` should be evaluated against the current leaner baseline, not against an older tree that still had duplicated grouped/scalar wrapper stacks. If a proposed `0.92` implementation needs another large runtime-wrapper cleanup to “make the cache work,” that is a sign the cache design is reaching too far.

## Cache Layer Constraint

`0.92` should treat cache-layer count as a first-class design constraint, not only an implementation detail.

The current meaningful reuse stack is already three layers deep:

1. compiled SQL command cache
2. SQL `SELECT` prepared-plan cache
3. shared lower query-plan cache

That is close to the upper sane limit. `0.92` should not add another independently meaningful reuse layer.

The intended end-state should be:

- at most **two first-class SQL reuse layers** that a human needs to reason about during normal repeated `SELECT` execution
- any lower shared query-plan cache that remains should be treated as an internal supporting boundary, not another peer SQL cache story

The design should therefore optimize for:

- one obvious top-level SQL compile cache boundary
- one obvious prepared-`SELECT` reuse boundary
- no fourth cache layer
- no multiple peer cache stories that can all partially explain the same repeated-query win

## Central Design Question

`0.92` should answer:

> Should SQL continue to own two upper cache layers above the shared query-plan cache, or should repeated SQL `SELECT` execution converge on one stronger prepared-SQL artifact that already includes the reusable structural plan and outward projection contract?

The repo is now in a better place to answer that because the code already has:

- a generic-free compiled SQL artifact (`CompiledSqlCommand`)
- a reusable prepared-plan artifact for `SELECT` (`SharedPreparedExecutionPlan`)
- compile/execute attribution explicitly designed to validate whether repeated queries stop paying compile cost and which cache layer hit

## Recommended Direction

The recommended direction is:

- keep the compiled-command cache as the top-level SQL entry cache, because it already sits before parse/lower and is exactly the boundary that removes compile cost on repeat calls
- revisit whether the separate SQL select-plan cache should remain its own layer, or whether its contents should be folded more directly into the prepared SQL artifact for `SELECT`

The desired story is:

- “compiled SQL command” for non-`SELECT` statement families
- “prepared SQL select artifact” for `SELECT`

not:

- “compiled command plus maybe another SQL plan cache plus maybe lower shared query-plan cache”

The current `SqlSelectPlanCacheEntry` already tells you what that artifact wants to be: a prepared plan plus outward SQL projection contract.

The key unresolved point is not whether visibility matters; it already does. The current SQL select-plan cache is visibility-keyed, and any `0.92` simplification must keep that explicit instead of hand-waving it behind a more holistic artifact.

The recommended design pressure is therefore:

- keep `CompiledSqlCommand` generic-free and session-owned
- make the `SELECT` variant carry the cache identity required to resolve prepared-select reuse in the normal path
- stop treating “compiled SQL without compiled cache identity” as an ordinary architectural branch for repeated `SELECT` execution

## Candidate End State

A plausible `0.92` end state is:

- `CompiledSqlCommand::Select` remains the semantic SQL artifact
- `SELECT` execution resolves through one prepared-select artifact owner that is explicitly visibility-sensitive
- repeated query execution does one top-level compiled SQL cache lookup

The prepared-select artifact is keyed by:

- compiled SQL key
- visibility

and stores:

- `SharedPreparedExecutionPlan`
- outward SQL projection contract (`columns`, `fixed_scales`)

That means `SELECT` execution no longer needs to rediscover whether it can reuse a second SQL plan layer in the common repeated-query case, because the visibility-sensitive prepared-select artifact is the SQL-owned reuse boundary for repeated `SELECT` execution.

Lower shared query-plan caching remains an internal reuse boundary, but the SQL surface presents one prepared-SQL reuse contract.

In concrete terms, the normal repeated `SELECT` story should become:

1. lookup compiled SQL command by SQL text / entity / schema / surface kind
2. use that compiled SQL identity plus visibility to resolve one prepared-select artifact
3. execute from that prepared artifact

not:

1. maybe use compiled SQL cache
2. maybe use SQL select-plan cache
3. maybe fall back to shared query-plan cache because the compiled artifact has no cache identity attached

## Ownership Surfaces This Slice Intends To Change

`0.92` should stay explicit about which code boundaries actually own the cache story.

The current primary ownership surfaces are:

- `db/session/sql/mod.rs`
  - `CompiledSqlCommand::Select`
  - `planned_sql_select_with_visibility(...)`
  - SQL compiled-command cache
  - SQL select-plan cache
- `db/session/sql/execute/route.rs`
  - `compile_sql_statement_for_authority(...)`
  - current uncached `CompiledSqlCommand::Select { compiled_cache_key: None }` construction
- `db/session/sql/execute/mod.rs`
  - `execute_compiled_sql_with_cache_attribution(...)`
  - `execute_compiled_sql_with_phase_attribution(...)`
  - the normal repeated-`SELECT` execution path that should become simpler to reason about

The current supporting-but-lower boundary is:

- `db/session/query.rs`
  - shared prepared-plan/query-plan reuse underneath the SQL-owned layer

That means the first `0.92` implementation slice should primarily be a session-SQL ownership change, not a broad executor or planner rewrite.

If a proposed implementation starts needing unrelated changes across grouped runtime ownership, scalar/grouped entrypoint preparation, or projection-runtime orchestration, that is likely scope drift away from the actual cache seam.

## Benchmark Cohort

The baseline and acceptance comparison for `0.92` should use a deliberately small but representative repeated-query cohort:

- repeated scalar `SELECT` with ordinary projection, filter, order, and limit
- repeated grouped `SELECT` with `GROUP BY`, aggregate projection, order, and limit
- repeated scalar SQL projection path that exercises outward SQL column/scale shaping
- repeated `SELECT` with a non-ready/empty visibility context to prove visibility-sensitive reuse still invalidates correctly

The benchmark cohort should not be a random bundle of SQL examples. It should be the minimum set that proves:

- compile reuse works
- prepared-select reuse works
- grouped and scalar shapes both keep correct results
- visibility remains part of the prepared-select identity

If `0.92` cannot show a clearer cache story and a measurable repeated-query win on that small cohort, the slice should not widen before it succeeds there.

## SQL Perf Harness Plan

`0.92` should use the existing PocketIC SQL perf audit harness instead of inventing a one-off benchmark script.

The concrete repeated-query cohort should come from `testing/pocket-ic/tests/sql_perf_audit.rs`:

- scalar primary-key repeat baseline
  - `repeat.user.pk.order_only.asc.limit2.runs10`
  - `repeat.user.pk.order_only.asc.limit2.runs100`
- scalar expression-order repeat baseline
  - `repeat.user.name.lower.order_only.asc.limit3.runs10`
  - `repeat.user.name.lower.order_only.asc.limit3.runs100`
- grouped repeat baseline
  - `repeat.user.grouped.age_count.limit10.runs10`
  - `repeat.user.grouped.age_count.limit10.runs100`

The visibility and warm-cache parity cohort should stay equally explicit:

- cold parity check
  - `user.age.order_only.asc.limit2.cold_query`
- warmed query-after-update parity check
  - `user.age.order_only.asc.limit2.warm_after_update`

That gives `0.92` a narrow but credible proof set:

- simple repeated scalar `SELECT`
- repeated scalar `SELECT` with expression ordering
- repeated grouped `SELECT`
- cold-versus-warmed cache parity on the same outward query shape

## SQL Perf Counters That Matter

The harness already records the counters this design needs. `0.92` should treat these as the primary measurement surface:

- `compile_local_instructions`
- `execute_local_instructions`
- `sql_compiled_command_cache_hits`
- `sql_compiled_command_cache_misses`
- `sql_select_plan_cache_hits`
- `sql_select_plan_cache_misses`
- `shared_query_plan_cache_hits`
- `shared_query_plan_cache_misses`

For grouped scenarios, the grouped subphase counters should remain visible as secondary proof that the cache change did not accidentally move cost deeper into grouped execution:

- `grouped_stream_local_instructions`
- `grouped_fold_local_instructions`
- `grouped_finalize_local_instructions`

The acceptance read should stay simple:

- repeated-query compile cost should drop sharply once the compiled-command cache is warm
- repeated-query SQL prepared-select reuse should show up as SQL-layer select-plan hits or their direct successor if `0.92` renames that owner
- the shared query-plan cache should not become more prominent in the normal repeated `SELECT` story after the simplification
- grouped subphase cost should stay stable unless a measured improvement is explicitly explained

## Measurement Gate

Implementation work should not start until the current branch has a saved baseline for the exact scenarios above.

The first `0.92` coding slice should also stop immediately if the same report shows either of these outcomes:

- the repeated-query line gets structurally simpler but the measured compile or execute cost gets worse on the repeat scenarios
- the repeated-query line looks faster only because hits moved from the SQL-owned boundary down into the shared query-plan cache

## Rejected Directions

`0.92` should reject a few tempting but wrong designs up front.

### 1. Add another cache tier

Rejected.

The current stack is already near the upper sane limit. A new independent SQL cache tier would make attribution and correctness harder to reason about, not easier.

### 2. Hide visibility inside a “smart” prepared artifact without saying so

Rejected.

Visibility is already a real semantic input to prepared `SELECT` reuse. If a new prepared-select owner exists, its key policy must keep visibility explicit.

### 3. Cache executor scratch state

Rejected.

The value of this slice is repeated compile/plan reuse, not persistence of live runtime execution buffers, row pages, or grouped continuation state.

### 4. Solve non-`SELECT` and `SELECT` cache redesign in one pass

Rejected.

Non-`SELECT` compiled SQL already has a much cleaner story: compile artifact reuse without visibility-sensitive prepared-plan ownership. `0.92` should keep that stable and focus on the `SELECT` ambiguity.

## Required Hard Choice

This design implies one important hard choice:

- the current `compiled_cache_key: Option<_>` fallback path for `SELECT` should not survive as a normal architectural branch if `0.92` claims one prepared-SQL reuse contract
- either `SELECT` always carries the compiled SQL cache identity required to resolve the prepared-select artifact, or the design should say explicitly that uncached/direct-lowered `SELECT` execution remains a separate internal-only path for tests or forced bypass

`0.92` should prefer the first answer for the normal session/runtime path. If an uncached path survives, it should be:

- test-only
- diagnostics/debug-only
- or an explicitly named forced-bypass surface

It should not look like a peer runtime path beside ordinary repeated `SELECT` execution.

## What Should Be Deleted If 0.92 Lands

The design should state deletion targets explicitly so this does not turn into another additive cache layer:

- the current “`SELECT` with no compiled cache key” fallback branch in `planned_sql_select_with_visibility(...)` should be removed from the normal session execution path if repeated `SELECT` execution is meant to have one SQL-owned prepared artifact
- the direct `CompiledSqlCommand::Select { compiled_cache_key: None }` construction in `compile_sql_statement_for_authority(...)` should not remain the ordinary session compile shape for `SELECT`
- the SQL surface should not keep two equally first-class reuse stories for repeated `SELECT` execution
- if an uncached path remains for tests or debugging, it should be named as a non-default internal escape hatch rather than looking like part of the intended runtime architecture

The deletion goal is important: if `0.92` lands but all current cache branches remain intact, then the slice probably added a new cache story instead of simplifying one.

There is one more important deletion constraint:

- `0.92` should not leave both the old SQL select-plan cache story and a new prepared-select owner with overlapping responsibility

Either the existing SQL select-plan cache becomes that prepared-select owner in a clearer form, or it is replaced. It should not gain a sidecar sibling.

## Cache Key Policy

`0.92` should preserve the current conservative cache-key stance:

- surface kind
- entity path
- schema fingerprint
- raw SQL text
- and, for prepared `SELECT` reuse, visibility-sensitive distinction where required

The existing keys are intentionally conservative, and the design should preserve that philosophy even if cache layering changes.

If the prepared-select artifact replaces the current separate SQL select-plan cache in name or ownership, visibility must stay part of the artifact key. `0.92` should not simplify the cache story by hiding a real semantic boundary.

## Memory / Cache-Shape Constraint

`0.92` should explicitly constrain cache growth, not only cache hits:

- the new prepared-select artifact must not duplicate prepared plans more than the current SQL select-plan cache does for the same compiled SQL key and visibility set
- the design should not move from “two understandable cache layers” to “one simpler story but a heavier compiled artifact multiplied across visibility contexts”
- if a compiled SQL cache entry owns visibility-sensitive prepared-select children, that child ownership should be the only SQL-owned prepared-plan duplication point

## Validation and Perf Proof

`0.92` should be explicitly attribution-driven. Success is not “cache exists”; success is:

- repeated identical queries show compile-side instruction cost dropping toward cache-hit behavior
- SQL cache attribution clearly indicates which boundary hit
- execute-side planner/setup cost is reduced or at least no worse for the repeated-select path
- grouped and scalar `SELECT` paths both continue to execute correctly through the prepared-plan seam

The current execution attribution and `SqlCacheAttribution` are already built to support this validation.

Because `0.91.5` already reduced a large amount of wrapper noise, `0.92` should also capture one explicit before/after architectural proof:

- how many first-class SQL reuse layers exist in the normal repeated `SELECT` story before the change
- how many remain after the change
- which normal-path branches were deleted to get there

This slice should not be accepted on perf numbers alone if the resulting cache story is still structurally ambiguous.

## Verification Matrix

The first coding slice should prove four things separately.

### 1. Identity

- repeated equivalent `SELECT` statements hit the intended compiled-command cache boundary
- repeated prepared-select resolution uses one visibility-keyed owner
- the normal repeated path no longer depends on the current no-cache-key branch

### 2. Invalidation

- schema fingerprint changes invalidate as they do today
- visibility changes select the correct prepared artifact identity
- query/update surface boundaries remain unchanged

### 3. Attribution

- `SqlCacheAttribution` still makes upper SQL cache hits legible
- compile-side attribution shows the repeated-query compile drop clearly
- execute-side attribution shows whether planner/setup work actually moved

### 4. Behavioral parity

- scalar repeated `SELECT` returns identical rows before and after the cache simplification
- grouped repeated `SELECT` returns identical rows before and after the cache simplification
- outward SQL column labels and fixed scales remain unchanged

## Kill Criteria

`0.92` should be restaged or reverted if any of the following become true:

- the normal repeated `SELECT` story still has more than two first-class SQL reuse layers after the change
- the current `compiled_cache_key: None` branch survives as an ordinary runtime path even though the design claims one prepared-select reuse contract
- compile/execute attribution becomes less legible than it is today
- memory duplication across visibility-sensitive prepared artifacts grows materially without a corresponding repeated-query win
- the implementation starts requiring another broad runtime-wrapper cleanup outside the session-SQL ownership boundary

## First Implementation Slice

The first implementation slice should be deliberately narrow:

1. record baseline attribution and cache-hit behavior for repeated scalar and grouped `SELECT`
2. make the normal compiled `SELECT` path carry the compiled cache identity unconditionally
3. route normal prepared-select reuse through one visibility-keyed prepared-select owner
4. delete the current normal-path `compiled_cache_key: None` fallback branch for ordinary session execution
5. keep non-`SELECT` statement families unchanged

That first slice should not also:

- redesign executor runtime state
- add new diagnostics surfaces
- widen SQL feature support
- change query-plan cache semantics outside the SQL-owned reuse story

If that narrow slice already proves the win, later follow-up cleanup can decide whether the remaining lower shared query-plan cache should stay exactly as-is internally or be further hidden behind the prepared-select owner.

## Acceptance Criteria

### Functional

- identical repeated SQL queries reuse the intended SQL cache boundary
- schema/visibility changes still invalidate correctly
- no change in query results or outward SQL projection contract

### Architectural

- the SQL cache story is simpler than today
- repeated SQL `SELECT` execution has at most two first-class SQL reuse layers to reason about
- compile cache remains above parse/lower
- `SELECT` reuse does not require reasoning about multiple SQL-layer cache boundaries in the common case
- executor-owned runtime scratch state is still not cached
- visibility remains an explicit part of prepared `SELECT` reuse rather than an implicit or hidden sub-rule
- the normal repeated-`SELECT` path no longer depends on the current no-compiled-key fallback branch

### Performance

- baseline perf checks are captured before implementation starts
- compile-local instructions drop sharply on repeated queries relative to that baseline
- select-planner/setup cost is no worse and ideally lower on repeated `SELECT`
- cache attribution clearly shows hits at the intended layer rather than diffuse reuse across multiple boundaries

## Thesis Sentence

`0.92` revisits the SQL cache stack now that the compile/execute seam and prepared-plan boundary are stable, with the goal of turning repeated SQL execution into one clearer prepared-SQL reuse contract, validating the win directly through existing compile/execute and cache attribution surfaces, and proving improvement against recorded baseline perf measurements before implementation began.
