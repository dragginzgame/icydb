# 0.66 — Text Semantics Completion: Functions, Capability Classes, and Planner Contracts

## Status

Implementation in progress

Current checkpoint:

* parser support for the staged text projection surface is in progress
* lowering still rejects generic structural text-expression projection on purpose
* a narrow session-owned computed projection lane is the current implementation shape
* the current computed projection lane is exposed through SQL dispatch/projection payload surfaces and `EXPLAIN`, but not through `query_from_sql`, which remains structural-only and points callers at SQL dispatch instead
* planner-recognized `STARTS_WITH(field, constant)` remains deferred unless it proves cheap and exact

## Problem Statement

`UPPER` and `LOWER` exposed that the missing work in the text surface is not just function count. The larger issue is that text semantics currently risk drifting across parsing, lowering, typing, coercion, runtime evaluation, explain output, and capability classification.

That creates two failure modes:

1. small feature additions land as scattered special cases
2. equivalent text intent can behave differently depending on whether it enters through SQL, fluent/query APIs, projection, predicate runtime, or planner-recognized forms

`0.66` should therefore treat text-function expansion as a semantic-boundary slice, not a grab bag of commands.

## Goals

1. Add a bounded set of high-value missing text functions.
2. Define canonical semantics for text scalar functions in one place.
3. Ensure planner, runtime, and explain surfaces agree on capability class.
4. Treat raw canister wasm growth as a hard release gate, not a secondary note.
5. Keep wasm growth bounded by reusing shared lowering/runtime machinery.
6. Use the slice to flush out remaining text coercion and null-propagation inconsistencies.

## Non-Goals

* Full SQL string-function parity with PostgreSQL or SQLite.
* Regex support.
* Locale-aware or collation-aware text semantics.
* Broad Unicode normalization/casefold machinery beyond the existing `UPPER`/`LOWER` contract.
* New index structures dedicated to substring or fuzzy text search.

## Release Thesis

`0.66` completes the minimum practical text function surface for filtering and projection while tightening the semantic contract so new text operators do not multiply branch logic across planner, runtime, and explain layers.

Just as important, `0.66` must not become a hidden wasm-growth line. Text
surface work is only acceptable if it stays on shared evaluator/lowering paths
and its shipped raw wasm impact remains small, measured, and justified.

## Why these functions

This batch gives strong practical value without forcing heavy new runtime machinery.

* `TRIM`/`LTRIM`/`RTRIM` cover normalization workflows.
* `LENGTH` covers measurement and simple filtering/projection.
* `SUBSTRING` and `POSITION` cover basic extraction/search.
* `REPLACE` covers projection-time cleanup and simple normalization.
* `STARTS_WITH`/`ENDS_WITH` give direct user intent and are cleaner planner targets than forcing expression-shaped rewrites.

This set is also useful for surfacing bugs in:

* null propagation
* argument validation
* indexability classification
* planner/runtime parity
* explain output stability

## Design Principles

### 1. One semantic owner for text function contracts

Text scalar semantics should be specified once and then consumed by:

* SQL lowering
* scalar expression typing
* predicate/runtime evaluation
* planner capability checks
* explain/descriptor rendering

The system should avoid independent per-layer reinterpretation of what a text function means.

This is also the primary wasm-control rule. Every place that re-derives text
function semantics independently is a likely extra symbol family.

### 2. Capability class must be explicit

Each function should be classified as one of:

* **planner-recognized and capability-relevant**
* **runtime-valid but not index/planner-optimizable**
* **invalid for the given operand shape**

This keeps explain output honest and prevents accidental planner drift.

### 3. Null behavior must be uniform

Unless otherwise specified, text scalar functions should be null-propagating:

* any required null input yields null output

Boolean-returning text predicates such as `STARTS_WITH` and `ENDS_WITH` should follow the existing nullable expression rules rather than inventing ad hoc truthiness.

### 4. Coercion must be narrow and explicit

Avoid broad “stringify anything” behavior.

Preferred rule:

* accept text operands where text is semantically required
* reject unsupported operand kinds deterministically
* only permit coercions already consistent with the project’s scalar coercion contract

### 5. Planner optimism must be conservative

Planner recognition should only occur when semantics are exact and stable.

Example:

* `STARTS_WITH(field, constant)` may be planner-recognizable in the future if capability checks support it cleanly
* `ENDS_WITH`, `REPLACE`, and arbitrary `SUBSTRING` should initially be runtime-only

### 6. Wasm cost outranks local elegance

A cleaner-looking split is not automatically acceptable if it grows shipped raw
wasm. `0.66` should prefer slightly denser shared helpers over per-function or
per-layer abstractions when those abstractions duplicate codegen in canister
builds.

This means:

* accept by raw `.wasm` and shrunk `.wasm`, not by debug monos alone
* treat `.wasm.gz` as transport context only
* revert slices that look architecturally nice but measurably bloat release wasm

## Proposed Semantic Contract

## Function Semantics Table

| Function                       |  Arity | Return Type | Null Rule        | Notes                                                               |
| ------------------------------ | -----: | ----------- | ---------------- | ------------------------------------------------------------------- |
| `TRIM(text)`                   |      1 | text        | null-propagating | trims leading and trailing whitespace                               |
| `LTRIM(text)`                  |      1 | text        | null-propagating | trims leading whitespace                                            |
| `RTRIM(text)`                  |      1 | text        | null-propagating | trims trailing whitespace                                           |
| `LENGTH(text)`                 |      1 | integer     | null-propagating | returns text length under the project’s chosen text-length contract |
| `SUBSTRING(text, start, len?)` | 2 or 3 | text        | null-propagating | explicit index rules required                                       |
| `REPLACE(text, from, to)`      |      3 | text        | null-propagating | exact substring replacement                                         |
| `STARTS_WITH(text, prefix)`    |      2 | bool        | null-propagating | direct prefix test                                                  |
| `ENDS_WITH(text, suffix)`      |      2 | bool        | null-propagating | direct suffix test                                                  |
| `POSITION(needle, haystack)`   |      2 | integer     | null-propagating | explicit indexing base required                                     |

## Settled semantic rules

### A. Length semantics

`LENGTH(text)` uses Unicode scalar-count semantics, not UTF-8 byte length.
This matches the current session-owned evaluation path and keeps the SQL-facing
contract aligned with the other character-count helpers in this slice.

### B. Substring indexing base

`SUBSTRING(text, start, len?)` uses 1-based SQL-style indexing.

### C. Out-of-range behavior

The current narrow lane uses deterministic character-based behavior:

* start beyond the end returns the empty string
* zero or negative start returns the empty string
* omitted length returns the suffix from `start`
* zero or negative length returns the empty string

### D. Trimming semantics

`TRIM`, `LTRIM`, and `RTRIM` use Rust standard whitespace semantics on the
current shipped lane.

Initial proposed classification:

| Function / Pattern               | Capability Class               | Notes                                                                         |
| -------------------------------- | ------------------------------ | ----------------------------------------------------------------------------- |

| `ENDS_WITH(...)`                 | runtime-only                   | not prefix-friendly                                                           |
| `TRIM(...)`                      | runtime-only                   | normalization helper                                                          |
| `LTRIM(...)`                     | runtime-only                   | normalization helper                                                          |
| `RTRIM(...)`                     | runtime-only                   | normalization helper                                                          |
| `LENGTH(...)`                    | runtime-only                   | may appear in projection/filter, not planner-optimizable initially            |
| `SUBSTRING(...)`                 | runtime-only                   | arbitrary extraction                                                          |
| `REPLACE(...)`                   | runtime-only                   | projection/filter helper                                                      |

## Architectural Shape

## Required implementation shape

### Parser / surface

* Add the new function names at the SQL/function surface.
* Avoid bespoke parser branches per function where a table- or enum-backed function registry is already appropriate.

### Lowering

* Lower all text functions into a single scalar-expression family or a tightly related family.
* Do not let SQL lowering encode function-specific runtime behavior beyond argument shaping.
* Reject broad projection-expression infrastructure if a smaller surface-specific lowering path can deliver the same user-facing function.

### Typing / coercion

* Type acceptance and coercion rules should live in one owner module.
* Errors should be constructed from that owner boundary instead of per-call-site ad hoc diagnostics.

### Runtime evaluation

* Implement text function execution in a shared evaluator lane.

## Measured Exploration Findings

The first exploratory implementation pass on March 28, 2026 tried two shapes:

1. a shared internal unary text-function expression lane
2. a projection-side SQL frontend slice for `SELECT TRIM(name)`-style expressions

The result was rejected on wasm grounds.

### Rejected public SQL projection slice

Keeping the projection-side SQL function surface increased shrunk wasm from the
current `0.66` baselines:

* `minimal`: `1,137,333 -> 1,142,419` (`+5,086`)
* `one_simple`: `1,268,864 -> 1,273,825` (`+4,961`)

That slice was reverted immediately.

### Rejected internal generic expression lane

Even after removing the public SQL projection slice, the remaining internal
generic text-function expression plumbing still measured above baseline:

* `minimal`: `1,137,333 -> 1,140,370` (`+3,037`)
* `one_simple`: `1,268,864 -> 1,271,776` (`+2,912`)

That internal lane was also reverted.

### Design consequence

The March 28 exploration showed that generic text-expression plumbing carries
meaningful front-loaded wasm cost. `0.66` should therefore avoid merging
standalone preparatory scaffolding. That result should not be interpreted as a
blanket rejection of shared text infrastructure: later batches may amortize the
fixed cost if introduced together with a sufficiently useful first function set
and a clear reuse path.

The right constraint is sequencing, not a permanent ban on shared machinery.

#### Case 1: bad speculative infrastructure

Reject:

* broad generic scaffolding
* no shipped user-visible value yet
* measurable wasm cost now
* unclear near-term reuse path

#### Case 2: amortizable shared infrastructure

Potentially acceptable:

* shared machinery lands together with an immediately shipped function surface
* the next planned function batch is expected to reuse it directly
* the fixed wasm cost is front-loaded but structurally justified

#### Case 3: narrow one-off shortcut

Use cautiously:

* cheaper in the very short term
* but likely to duplicate semantics
* and likely to need replacement once the batch grows

This is only worth doing when the shipped feature is tiny and broader
abstraction would clearly be premature.

### Updated implementation rule

`0.66` should not begin by merging generic projection-expression scaffolding in
isolation. Generic text infrastructure is acceptable only when introduced
together with a concrete shipped surface and a credible follow-on path for
amortizing its wasm cost.

Concretely:

* parser/lowering additions must be tied to one shipped user-facing surface
* shared runtime helpers are acceptable when they support an immediately
  shipped function and are likely to amortize across the planned first batch
  without reopening the semantic surface
* generic projection/text-expression scaffolding should not merge as standalone
  preparatory work; its wasm cost should be evaluated over the expected first
  batch, not over a single exploratory function
* Reuse common helpers for:

  * text operand extraction
  * null propagation
  * integer argument validation
  * substring bounds logic

### Explain / descriptor

* Explain output should render text functions with stable names and capability class.
* Runtime-only functions should not accidentally appear planner-eligible in descriptor output.

## Bug Classes this slice should eliminate

1. **SQL/runtime parity drift**

   * the same text function lowered from SQL behaves differently from direct scalar/runtime construction

2. **Null-propagation mismatch**

   * one surface returns false/empty string while another returns null

3. **Coercion drift**

   * text functions accept non-text values in one layer and reject them in another

4. **Capability drift**

   * planner/explain implies optimization that runtime cannot justify

5. **Function-specific special-case spread**

   * every new function adds branch points independently across parser, lowering, typing, runtime, and explain

## Expected Wasm Impact

Expected wasm impact should be bounded if the implementation reuses shared
machinery, and every slice should be measured against the existing
`wasm-release`, `sql-on` audit canisters before it is kept.

Estimated ranges:

* disciplined shared implementation: roughly `+8 KB` to `+20 KB`
* if planner/explain/runtime add separate bespoke logic per function: risk grows beyond that range

These ranges are not approval targets. They are warning bands. The real rule is:

* keep the smallest implementation shape that satisfies the semantic contract
* reject any slice whose raw wasm growth is larger than expected for the value it adds
* evaluate front-loaded infrastructure cost over the first shipped function
  bundle, not over a preparatory skeleton slice
* prefer staging the function set across multiple shipped cuts rather than
  accepting one bloated batch

Lower-risk functions from a wasm perspective:

* `TRIM`
* `LTRIM`
* `RTRIM`
* `LENGTH`
* `STARTS_WITH`
* `ENDS_WITH`

Higher-risk additions within this slice:

* `SUBSTRING`
* `REPLACE`
* `POSITION`

Control rule for `0.66`:

* prefer shared helpers and one evaluator family over scattered per-function implementations

## Wasm Gate

`0.66` should treat wasm as a first-class ship gate.

### Primary metrics

Decide slices by:

* built `.wasm`
* shrunk `.wasm`

Do not accept or reject a slice primarily by:

* `.wasm.gz`
* debug-symbol/top-mono reductions
* “cleaner architecture” arguments without measured release-wasm support

### Measurement matrix

At minimum, measure:

* `minimal`, `wasm-release`, `sql-on`
* `one_simple`, `wasm-release`, `sql-on`

Current starting baseline for the line:

* `minimal`, shrunk `wasm-release`, `sql-on`: `1,137,333`
* `one_simple`, shrunk `wasm-release`, `sql-on`: `1,268,864`

If a slice changes SQL dispatch or canister-exported text surfaces, also compare:

* `minimal`, `debug`, `sql-on`
* `one_simple`, `debug`, `sql-on`

### Keep / reject rules

Keep a slice when:

* raw/shrunk wasm stays flat or improves, or
* the growth is small, expected, and clearly justified by the shipped feature

Reject or restage a slice when:

* the same behavior can be achieved with less wasm by reusing existing text/runtime helpers
* function-specific error/reporting strings dominate the size delta
* planner/explain support adds more code than the text function itself is worth in the same cut
* the slice hides a per-function branch family in parser, lowering, runtime, and explain simultaneously

### Anti-bloat implementation rules

For `0.66`, prefer:

* one shared text-function enum/registry over many ad hoc parser branches
* one shared text evaluator lane over per-function evaluators
* one owner for type/arity/null validation
* one shared bounds helper for `SUBSTRING`/`POSITION`-style integer arguments
* stable reused explain wording instead of bespoke diagnostics per function

For `0.66`, avoid:

* per-function error enums
* per-function explain descriptor types
* duplicated SQL-lowering and fluent-lowering branches that encode the same semantics twice
* early planner support for cases that can stay runtime-only in the first cut

## Staging Rule

The sequencing rule for `0.66` is:

* do not merge generic projection/text-expression scaffolding as standalone
  preparatory work
* do allow shared infrastructure when it ships together with a concrete
  user-facing bundle and there is a credible near-term amortization story for
  the next batch
* prefer a slightly denser shared lane over a narrow one-off shortcut when the
  next 3–5 planned functions are expected to reuse it directly
* prefer a narrow shortcut only when the shipped feature is tiny and broader
  infrastructure would clearly be premature

## Acceptance Criteria

`0.66` is successful when:

1. the first-batch text functions are available and documented
2. SQL lowering, runtime evaluation, and explain output agree on their semantics
3. null and coercion behavior are deterministic and centrally owned
4. no new fragmented planner/runtime special-case family appears
5. wasm growth remains bounded, measured, and justified by raw/shrunk release-wasm numbers
6. any slice that bloats raw wasm without enough value is restaged or reverted

## Risks

### 1. Surface growth without semantic cleanup

Adding functions without centralizing contracts will increase branch count across multiple owners and create more parity bugs.

### 2. Hidden Unicode/indexing ambiguity

`LENGTH` and `SUBSTRING` can create user-visible semantic confusion unless index base and text-length rules are explicit.

### 3. Planner overreach

Recognizing text predicates too aggressively could create misleading explain output or invalid optimization assumptions.

### 4. Error-surface duplication

Function-specific type/arity/bounds errors can bloat both wasm size and maintenance overhead if each layer emits them independently.

### 5. Slow hidden wasm accretion

Even if each function looks small in isolation, `0.66` can still bloat if every
function adds parser labels, lowering branches, runtime validation, explain
strings, and dedicated tests without shared owners. This is the main release
management risk for the line.

## Alternatives Considered

### Alternative A — Add only a few commands ad hoc

Rejected because `UPPER`/`LOWER` already showed that text semantics have enough cross-layer coupling to justify a contract-first slice.

### Alternative B — Focus only on case-insensitive matching

Deferred. Valuable, but better handled after the base text-function semantic surface is stabilized.

### Alternative C — Pursue full SQL string parity

Rejected as too broad for a bounded release.
