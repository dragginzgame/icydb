# 0.48.1 - Explain Small-Feature Rollout (Tier 1)

## Status

In Progress
Target: `0.48.1`
Dependency: `0.48.0` closeout (`2026-03-10`)
Last updated: 2026-03-10

---

## 1. Objective

Ship low-risk planner/access observability improvements for `EXPLAIN` that
surface already-computed decisions without changing execution behavior or
re-introducing explain-owned policy inference.

Primary goals:

- expose scan-level prefix usage metadata for index routes
- expose explicit sort-elimination metadata in execution nodes
- expose scan-level fetch metadata when limit pushdown is active
- expose pushed-vs-residual predicate reporting in execution nodes
- expose explicit covering-scan status in scalar load execution roots
- expose planner-derived access-choice metadata (`chosen`, `alternatives`,
  `chosen_reason`, `rejections`)
- preserve deterministic explain text/json/verbose output contracts

Architectural guardrail:

- planner/access/executor decisions -> explain projection
- never explain projection -> planner/access/executor decisions

---

## 2. Non-Goals

- no changes to runtime execution semantics
- no new query-language surface
- no join/cost-model subsystem work
- no hash/signature identity contract changes

---

## 3. Scope for `0.48.1`

1. Index prefix length reporting

- root scan nodes now carry `node_properties.prefix_len` for index prefix/range
  routes

2. Sort elimination reporting

- ORDER BY stage nodes now carry `node_properties.order_satisfied_by_index`
  (`true` for access-satisfied order, `false` for materialized sort)

3. Limit pushdown reporting (scan-level)

- root scan nodes now carry `node_properties.fetch` when top-N seek or
  index-range limit pushdown is active

4. Predicate pushdown reporting

- strict prefilter nodes now expose pushed predicate summaries
- residual filter nodes now expose pushed access predicate summaries alongside
  residual predicate payloads

5. Covering index indicator (scalar load surface)

- scalar load execution roots now explicitly report covering status via
  `covering_scan`
- current contract is explicit `false` for scalar load routes until an index-only
  row materialization path exists

6. Access path ranking (explain-only metadata)

- execution root now reports:
  - `access_choice_chosen`
  - `access_choice_chosen_reason`
  - `access_choice_alternatives`
  - `access_choice_rejections`
- alternatives/rejections are deterministic and derived from planner-candidate
  snapshots, not model-only index hints
- rejection entries use concise stable reason codes so chosen-vs-rejected
  ranking is explicit

Out of scope for this patch:

- Tier 2+ planner expansions (`OR` union, `IN` expansion)
