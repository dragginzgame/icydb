# 0.155 Admin SQL Addendum

## Purpose

IcyDB's current local SQL demo path is useful, but it is still demo-shaped:
`icydb sql` defaults to the demo RPG canister and calls a canister-authored SQL
method directly. That does not give developers a standard way to query another
IcyDB-backed canister during local or production-canister testing.

This addendum tracks the v1 replacement: a deliberately small controller-gated
admin SQL query surface that can be enabled by config and used by the CLI.

The closeout target is:

When a canister opts into admin SQL, `icydb sql --canister <name>` calls the
standard controller-gated read-only SQL endpoint. The existing demo SQL
behavior moves onto that same endpoint instead of keeping a special demo-only
query contract.

## V1 Scope

V1 should include only:

- one config switch per canister or build target
- one read-only SQL endpoint
- one authorization rule: caller must be a controller
- one CLI path that uses the standard endpoint
- canister-owned entity routing
- no mutation SQL
- no custom principal lists
- no per-entity permissions
- no row-level policy
- no environment-specific policy language

The endpoint may exist on local, test, or mainnet canisters when the canister
author explicitly enables it. If it is not enabled, the endpoint is absent.

## Config Contract

The v1 config should be a small boolean, not a permission model:

```yaml
admin_sql: true
```

The exact nesting belongs to the ICP/Canic manifest owner. The intended meaning
is:

- `false` or omitted: do not include the admin SQL endpoint
- `true`: include the read-only controller-gated admin SQL endpoint

Config decides whether the surface exists. The canister still enforces caller
authorization at runtime.

## Canister Contract

The standard endpoint should have an explicit admin/debug name, not `query`:

```text
icydb_admin_sql_query(sql: text) -> Result<SqlQueryResult, Error>
```

The endpoint must:

- be read-only
- reject mutation SQL through the existing query-lane rejection path
- require `caller::is_controller()` or the equivalent Canic controller guard
- route SQL to the correct entity through canister-owned routing
- avoid generated-model fallback reconstruction
- return the normal `SqlQueryResult` payload

The endpoint must not become a generated broad public SQL dispatcher hidden
inside `icydb-build`. Generated code may provide helper glue only if the
canister explicitly opts into the admin surface and the routing remains tied to
accepted canister/entity ownership.

## Demo SQL Migration

The current demo SQL behavior should move to this system.

Required migration:

- demo RPG enables `admin_sql: true`
- `icydb sql` defaults may continue to target `demo_rpg` in the `demo`
  environment
- the CLI should call `icydb_admin_sql_query` for ordinary SQL execution
- demo-specific `query(sql)` and `query_with_perf(sql)` should stop being the
  primary shell contract
- perf attribution can remain a separate diagnostics-only endpoint or CLI flag,
  but it must not be the required path for normal SQL execution

The user-facing result should be boring: the same demo commands keep working,
but they now exercise the same admin SQL path that another opted-in canister
would use.

## CLI Behavior

`icydb sql` should:

- call the standard admin SQL endpoint by default
- preserve the current demo defaults when `--canister` and `--environment` are
  omitted
- report a direct error when the endpoint is missing
- report a direct error when the caller is not a controller
- avoid guessing old demo method names unless an explicit compatibility flag is
  added for a short transition

The missing-endpoint message should tell the developer to enable admin SQL for
that canister build rather than suggesting fixture reloads.

## Non-Goals

This addendum does not add:

- SQL mutation admin endpoints
- principal allowlists
- role-based SQL access
- row filtering
- separate local/mainnet policy rules
- runtime discovery that can query every IcyDB database without canister opt-in
- a new SQL execution engine
- a second schema or DDL authority path

## Validation

Implementation should include focused coverage that proves:

- a canister without admin SQL does not export the endpoint
- a canister with admin SQL rejects non-controller callers
- controller callers can run read-only SQL
- mutation SQL sent to the admin query endpoint fails closed
- demo SQL goes through the standard endpoint
- `icydb sql --canister <other opted-in canister>` does not rely on demo method
  names
