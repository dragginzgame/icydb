# 0.193 Public Query Ergonomics And Admission Evidence

Status: proposed.

Theme: make the 0.192 default read-admission hard cut obvious, ergonomic, and
test-proven for ordinary application developers.

0.192 made ordinary typed/fluent reads fail closed by default and kept
trusted/admin bypasses explicit. 0.193 should make that behavior easy to
understand, hard to accidentally bypass, and covered by developer-facing
examples and regression tests.

## Context

0.190 proved the IC-native reliability baseline. 0.191 productized durability
and persisted-format policy. 0.192 added query read-admission lanes, EXPLAIN
diagnostics, and then hard-cut ordinary typed/fluent reads onto the built-in
bounded public-read policy by default.

That final 0.192 hard cut intentionally removed application-facing custom read
policies. The normal path is now:

- application endpoint enforces caller authorization;
- endpoint builds a typed/fluent query;
- ordinary `execute_query`, fluent `execute`, `execute_rows`, cursor-paged
  `execute`, and terminal helpers apply the default bounded policy;
- unsafe full scans, non-zero offsets, materialized sorts, missing row bounds,
  and grouped reads without grouped budgets return read-admission errors before
  row execution;
- controller/admin code uses explicit trusted execution only when it has its
  own caller and resource policy.

The public `icydb` facade `DbSession` is the developer-facing session surface
for this contract. The lower-level `icydb-core` session still exposes
crate-internal execution primitives used by tests, SQL lowering, and facade
wrappers; those core primitives are not the public endpoint contract.

0.193 should not reopen policy customization. Its job is to reduce confusion
and prove the hard cut behaves like a sane fresh-install default.

## Goals

- Document the default typed/fluent read path as the normal public endpoint
  path.
- Provide copyable examples that show admitted and rejected public reads.
- Keep trusted bypasses visibly named and clearly unsuitable for arbitrary
  callers.
- Add tests that prove a fresh application using ordinary execution gets
  read-admission protection without setting any policy.
- Add tests that prove trusted execution bypasses are explicit and do not
  become the default path.
- Improve read-admission error examples so developers can fix rejected queries.
- Keep generated SQL endpoints controller-gated and avoid adding generated
  public read endpoints.
- Keep `DbSession` and fluent APIs on one ordinary execution flow plus one
  explicit trusted/admin flow.
- Preserve 0.190/0.191/0.192 contracts: no persisted-format changes, no
  recovery changes, no cursor-format changes, and no custom-policy revival.

## Non-Goals

- Do not reintroduce application-facing custom `QueryAdmissionPolicy` builders.
- Do not add generated non-controller public SQL or typed/fluent endpoints.
- Do not expose trusted execution as a convenience for public endpoints.
- Do not make SQL ad-hoc reads public-safe by default.
- Do not add SQLite-style transactions, joins, or snapshot pagination.
- Do not weaken generated endpoint controller gates.
- Do not change marker, journal, row, schema, index, cursor, fold watermark, or
  structural-value persisted formats.
- Do not use estimated planner costs as public-read authority.
- Do not widen full-scan admission merely because a query has a small `LIMIT`.

## Developer Contract

The developer-facing rule should be simple:

Use ordinary typed/fluent execution for caller-facing endpoints. Use trusted
execution only for controller/admin paths after enforcing caller authorization
and a separate resource policy.

For a fresh install, the ordinary path must reject risky reads without the
developer specifying paths or policies:

- unindexed full scan: rejected;
- bounded indexed equality/range read: admitted;
- non-zero offset: rejected;
- materialized sort not proven by route: rejected;
- grouped read without grouped limits: rejected;
- grouped read with query-owned grouped limits that fit the built-in policy:
  admitted;
- trusted execution: allowed only through explicit trusted method names.

The documentation should not require new users to understand internal planner
objects or policy structs before they get protection.

## API Shape

0.193 should keep the public facade API vocabulary narrow:

- ordinary:
  - `DbSession::execute_query`;
  - fluent `execute`;
  - fluent `execute_rows`;
  - fluent cursor-paged `execute`;
  - fluent terminal helpers;
- trusted:
  - `DbSession::execute_query_trusted`;
  - fluent `*_trusted` helpers;
  - `trusted_read_unchecked()`.

There should be no third application-facing custom-policy lane. If future
versions need configurable policies, they should be added behind a new design
with explicit user demand and tests proving it does not reintroduce duplicate
flows.

## Which API Should I Use?

The developer-facing contract should include a short table:

| You want to... | Use | Notes |
| --- | --- | --- |
| serve normal users | ordinary typed/fluent execution | Default bounded admission rejects unsafe public read shapes before row execution. |
| run controller diagnostics | trusted/admin execution | Caller authorization and an explicit resource policy are required. |
| explain why a query fails | EXPLAIN or admission diagnostics | Diagnostics do not bypass recovery or authorize execution. |
| paginate public results | cursor-paged ordinary execution | Prefer cursor pagination; avoid `OFFSET`. |
| run a broad maintenance scan | trusted execution | Keep it controller/admin-only and apply a maintenance resource policy. |
| expose arbitrary SQL publicly | do not | Generated SQL remains controller-gated; public SQL is a separate application-owned allowlist problem. |

## Hard-Cut Regression Guard

0.193 must include a guard that fails if the application-facing public facade
reintroduces custom read-policy builders or execute-with-policy helpers.

Forbidden public facade names include:

- `execute_query_with_policy`;
- `execute_with_policy`;
- `with_query_policy`;
- `execute_query_with_read_admission_policy`;
- `execute_sql_query_with_read_admission_policy`;
- `QueryAdmissionPolicy`;
- `GroupedAdmissionPolicy`;
- `QueryAdmissionPolicy::public_custom`;
- public `public_read_policy` builder exposure.

The same guard family must continue to reject a generated `sql.public_read`
configuration key and any generated non-controller public SQL query endpoint.
Future configurable public-read policies need a fresh design and tests; they
must not return as a quiet compatibility helper.

## Error Examples

The developer guide should include representative query and error examples.
The exact wording may evolve, but the diagnostic identity must stay stable.

Examples to document and test where practical:

- query: unindexed fluent filter without `LIMIT`;
  expected result: `QueryReadAdmission` rejection before row execution.
- query: unindexed fluent filter with `LIMIT 1`;
  expected result: full-scan/index-required rejection, not silent admission.
- query: indexed fluent equality with `LIMIT 10`;
  expected result: admitted.
- query: indexed fluent equality with non-zero offset;
  expected result: offset rejection.
- query: indexed fluent equality ordered by a non-route-proven field;
  expected result: materialized-sort rejection.
- query: grouped fluent aggregate without `grouped_limits`;
  expected result: grouped-budget rejection.
- query: grouped fluent aggregate with `grouped_limits` inside defaults;
  expected result: admitted.
- query: same unsafe shape through trusted execution;
  expected result: succeeds only through explicit trusted API.

The error examples should explain the fix in terms a canister developer can
apply: add an index, add a bound, use cursor pagination instead of offset, add
grouped limits, or move the query to a controller/admin trusted path.

## Generated Endpoint Contract

Generated endpoint behavior should remain unchanged:

- `icydb_query` stays controller-gated and trusted/admin;
- generated SQL settings continue to have no `sql.public_read` key;
- no generated public typed/fluent read endpoint is added;
- generated hidden Rust wrappers may use internal names, but exported public
  methods must not silently expose ordinary unguarded reads.

0.193 may add tests or docs that make this easier to audit, but should not
change endpoint authority.

## First Implementation Slice

Slice 1 should be documentation and parity evidence:

- update `READ_ADMISSION.md` with short, copyable public/trusted examples;
- add focused tests proving ordinary fluent execution rejects a full scan on a
  fresh default path without custom policy setup;
- add focused tests proving indexed bounded fluent reads still admit;
- add focused tests proving trusted execution bypass is explicit;
- add source or invariant guard coverage if any removed custom-policy name can
  accidentally return as a public facade helper;
- record the validation matrix in the 0.193 changelog only when preparing a
  patch release.

## Acceptance Matrix

Minimum 0.193 acceptance:

| Scenario | Expected outcome |
| --- | --- |
| ordinary unindexed full scan | read-admission error before row execution |
| ordinary unindexed full scan with small `LIMIT` | read-admission error |
| ordinary indexed bounded read | admitted |
| ordinary non-zero offset | read-admission error |
| ordinary materialized sort | read-admission error |
| ordinary grouped read without query grouped limits | read-admission error |
| ordinary grouped read within query and policy defaults | admitted |
| trusted unsafe read | succeeds only through explicit trusted API |
| generated SQL query endpoint | remains controller-gated trusted/admin |
| public facade | exposes ordinary and trusted lanes, not custom policies |
| core session internals | remain crate-internal execution primitives, not public endpoint authority |

## Validation

Focused validation for the first slices:

- read-admission unit tests;
- fluent execution tests;
- generated endpoint token/config tests;
- public facade no-default checks;
- `make check-invariants`;
- `cargo clippy` on touched packages with all targets/all features;
- `cargo fmt --all`;
- `git diff --check`.

Broader release validation remains governed by the release checklist.

## Closeout Gate

0.193 can close when:

- the default public typed/fluent behavior is documented with examples;
- fresh default execution protection is covered by tests;
- trusted bypass behavior is covered by tests and remains visibly named;
- custom application-facing policy helpers remain absent from the public
  facade;
- generated SQL endpoints remain controller-gated;
- no persisted format, cursor format, response shape, or successful admitted
  query semantics changed unintentionally.
