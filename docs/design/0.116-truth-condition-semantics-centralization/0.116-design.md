# 0.116 Truth-Condition Semantics Centralization

## Thesis

`0.116` should centralize truth-condition semantics so planner-owned
expression typing and canonicalization become the single authority for all
expression meaning that determines filtering truth.

This is broader than raw bool types, but narrower than “all expression
semantics.”

The target lane is:

* `WHERE` predicate meaning
* `HAVING` predicate meaning
* bool-valued expressions used as filter conditions
* compare/null-test/truth-wrapper interpretation
* searched `CASE` and other admitted expression families when their result is
  consumed as a truth condition

This is not a feature-expansion slice.

It is not a grouped-semantic widening slice.

It is not an executor redesign slice.

It is a semantic-authority contraction slice whose job is to ensure that
truth-condition meaning is defined exactly once and then consumed consistently
across lowering, predicate adaptation, planner, prepared execution, runtime,
and explain.

`0.114` removes prepared fallback as a semantic owner.

`0.115` contracts the broader expression pipeline.

`0.116` should make the first major semantic-family collapse inside that
pipeline:

* truth-condition semantics become planner-owned
* adjacent layers stop acting like partial boolean-semantic owners

---

## Why `0.116` Exists

After `0.114.1`, the prepared lane is no longer the clearest architectural
limiter.

The latest completeness rerun shows:

* prepared SQL moved from `Partial` to `Bounded`
* the clearest remaining structural seam is now the broader expression
  pipeline cluster
* the next meaningful completeness movement should come from expression-pipeline
  contraction rather than emergency prepared-lane repair

Within that cluster, the highest-leverage first target is the
truth-condition lane, because it is where the system still has the clearest
overlap between:

* predicate-owned normalization/canonicalization
* planner-owned expression typing/classification
* lowering-owned structural shaping
* session/prepared consumption of already-typed truth conditions

That is the right first semantic-family contraction because it is:

* central to `WHERE` and `HAVING`
* already heavily exercised by the shipped SQL surface
* strongly represented in hotspot and duplication audits
* narrow enough to contract without turning `0.116` into a full expression
  redesign

## Problem

The system currently has one conceptual semantic authority for expression
meaning, but truth-condition logic is still touched in multiple adjacent
layers.

Today the broad shape still reads like:

* parser/lowering performs some truth-shaped structural normalization
* predicate layer performs boolean canonicalization and grouped boolean shaping
* planner performs expression typing and family classification
* prepared/session layers consume truth conditions that are already much better
  owned after `0.114`
* executor evaluates the result

Even when this is correct, it creates too many places where truth-condition
meaning is still handled as if each layer owns part of the rule set.

That leads to:

* branch duplication
* hotspot clustering
* unnecessary blast radius
* future drift risk between:
  * `WHERE`
  * `HAVING`
  * bool-valued expression forms
  * compare/null-test/truth-wrapper families

The problem is not bool types in isolation.

The problem is the full lane where an expression is interpreted as a truth
condition.

---

## What “Truth-Condition Semantics” Means

For `0.116`, truth-condition semantics means all semantic rules that determine
how an expression participates in filtering truth.

That includes:

### Included Semantic Families

* `WHERE` predicate meaning
* grouped/global `HAVING` truth meaning
* bool-valued expressions used directly as filter conditions
* compare-family expressions when used as truth conditions
* null tests when used as truth conditions
* truth wrappers such as `= TRUE`, `IS TRUE`, `IS FALSE`, and equivalent
  admitted forms where they exist in the shipped surface
* searched `CASE` result families when they are used as truth conditions
* canonical equivalence of admitted truth-condition shapes

### Not The Focus Of This Slice

* arbitrary non-boolean projection result shaping
* broad arithmetic simplification
* generalized function-family semantics
* grouped semantic widening beyond the already-shipped families
* all-expression canonicalization at once

So `0.116` is broader than bool columns and broader than raw bool types,
but it is still a bounded semantic-family slice.

---

## Goal

Make planner-owned expression typing and canonicalization the single semantic
authority for truth-condition meaning.

All adjacent layers must either:

* adapt truth-condition inputs into planner-owned forms
* consume planner-owned truth-condition outputs

They must not:

* reinterpret truth-condition meaning locally
* perform independent truth-condition canonicalization
* encode family-specific truth semantics outside planner-owned logic

---

## Core Invariant

Truth-condition meaning is defined exactly once, in planner-owned expression
typing and canonicalization.

Everything else is:

* structural translation before planner
* structural consumption after planner

If a truth-condition rule exists both in planner-owned expression logic and in
an adjacent layer, planner-owned logic must be the single authority.

Structural adaptation is allowed.

Independent truth-condition reinterpretation is not.

---

## Ownership Boundary

`0.116` should make the owner split visible in code shape.

### Parser / Lowering Owns

* SQL-specific surface syntax
* raw structural expression construction
* trivial syntactic normalization needed to build planner inputs

Parser / lowering must not own:

* truth-condition canonicalization
* boolean family meaning
* compare/null-test truth semantics beyond structural lowering

### Predicate Layer Owns

* adaptation between predicate-shaped APIs and planner-owned expression truth
* structural helpers for passing truth-condition forms into planner-owned logic

Predicate must not own:

* independent boolean semantics
* independent truth-condition canonicalization
* a second compare/null-test truth engine

### Planner Owns

* expression AST
* type inference
* family classification
* truth-condition canonicalization
* semantic equivalence of admitted truth-condition forms

This is the single semantic authority for truth-condition meaning.

### Executor Owns

* evaluation of already-canonical truth-condition contracts

Executor must not own:

* truth-condition reinterpretation
* local reclassification of expression families

### Session / Prepared Owns

* binding
* substitution
* execution wiring

`0.114` should already have pushed this lane toward structural-only behavior.

`0.116` must preserve that.

## Concrete Actions

### 1. Map Truth-Condition Operations Across Layers

Start with an ownership inventory of truth-condition operations in:

* planner expression modules
* predicate modules
* truth-condition-related lowering modules

For each operation, classify it as:

* planner-owned truth
* structural adaptation
* mixed logic that must be split

Examples to inventory:

* boolean normalization
* compare-family classification for filter truth
* null-test truth shaping
* truth-wrapper collapse
* searched `CASE` truth-result classification
* grouped/global `HAVING` truth-shape handoff

### 2. Move Truth-Condition Canonicalization Into Planner

Ensure there is:

* one truth-condition canonicalization path
* one family classification path for admitted truth semantics
* one semantic equivalence path for truth-condition shapes

Reduce or remove:

* parallel boolean canonicalization paths
* duplicate compare/null-test truth shaping
* local truth-wrapper meaning outside planner

### 3. Simplify Predicate Ownership

Refactor `db/predicate/*` so it becomes:

* adapter
* structural helper
* planner consumer

rather than a semi-independent truth-condition owner.

This is especially important for
[/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs:1).

The target is not to delete predicate abstractions.

The target is to stop letting them own meaning.

### 4. Reduce Lowering Semantic Work In The Truth Lane

In expression-related lowering:

* keep parse/build structure
* keep trivial syntax adaptation
* remove deeper truth-condition decisions where planner can own them instead

Lowering should produce planner inputs, not a second truth-condition semantic
system.

### 5. Reduce Hotspot Branch Pressure

Target the branch-heavy cluster first:

* `bool_expr.rs`
* `type_inference`
* expression-family helpers in lowering

Use the ownership cleanup to reduce:

* parallel match ladders
* duplicate family checks
* duplicated truth-wrapper handling

### 6. Add Invariant Tests

`0.116` is not complete without tests that prove:

* all canonical truth-condition forms originate from planner-owned logic
* predicate and lowering do not diverge from planner truth-condition
  canonicalization
* `WHERE` and `HAVING` use the same planner-owned truth semantics where the
  admitted family is shared
* prepared and non-prepared execution preserve the same canonical
  truth-condition plan shape
* truth-wrapper and compare/null-test families do not regain owner-local
  semantics outside planner-owned logic

---

## Recommended Work Order

### 1. Inventory Current Truth-Condition Ownership

Map all truth-condition operations across:

* planner expression modules
* predicate modules
* lowering modules

Do not start by moving code blindly.

### 2. Pick One First Semantic-Family Contraction

The best first contraction inside `0.116` is one of:

* truth-wrapper normalization
* compare/null-test truth shaping
* predicate boolean canonicalization handoff

Pick one, make planner the only owner, and prove parity.

### 3. Reduce Predicate-Layer Authority

Once the first family is planner-owned, simplify predicate-layer helpers so
they consume planner outputs instead of deciding meaning.

### 4. Reduce Lowering Truth Semantics

After predicate ownership is narrower, move any remaining truth-condition
meaning out of lowering.

### 5. Freeze The Boundary With Invariant Tests

Treat the invariant tests as the contract that keeps the old multi-owner truth
shape from returning.

## Exit Condition

`0.116` is done when:

* planner is the only semantic authority for truth-condition meaning
* predicate and lowering layers no longer duplicate truth-condition
  canonicalization
* the truth-condition hotspot cluster shows lower branch pressure than the
  pre-slice baseline
* truth-condition-related slices touch fewer modules than the pre-slice
  baseline
* semantic equivalence of admitted truth-condition forms is enforced
  consistently across surfaces

This should be visible as:

* lower hotspot pressure in complexity accretion
* fewer semantic-adjacent duplication families in DRY
* tighter truth-condition ownership in CSA
* lower median blast radius for truth-condition follow-through slices in
  velocity
* continued `Bounded` or better read for prepared SQL in completeness

The explicit velocity follow-through target remains:

* median blast radius `<= 8`

## Summary

`0.116` is the first focused semantic-family contraction inside the broader
expression-pipeline cleanup line.

It does not target raw bool types in isolation.

It targets the entire lane where expressions are interpreted as filtering
truth:

* predicates
* bool-valued filter expressions
* compare/null-test/truth-wrapper interpretation
* admitted searched `CASE` truth-result families

The intended end state is simple:

* planner owns truth-condition meaning
* predicate and lowering adapt into that authority
* executor and prepared/session consume it
* the system stops touching truth-condition semantics in multiple adjacent
  layers
