# 0.118 Expression Pipeline Flow Collapse

## Thesis

`0.118` should collapse expression-related flow into a single planner-owned
execution path so adjacent layers become thin adapters rather than
multi-stage flow participants.

The first and primary contraction target inside that broader flow problem is
predicate-owned expression flow, especially reducing
[bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs:1)
from a major expression-flow stage to an adapter boundary.

This is not a feature-expansion slice.

It is not a grouped-semantic widening slice.

It is not a new semantic-authority slice.

It is not an executor redesign slice.

It is a flow-contraction slice whose job is to eliminate cross-layer
expression routing so routine expression work stops bouncing between lowering,
predicate, planner, prepared/session wiring, and executor-adjacent helpers.

`0.116` centralizes truth-condition semantics.

`0.117` centralizes broader expression-family semantics.

`0.118` should then collapse the remaining mechanical flow:

* lowering should build planner-compatible structure directly
* planner should become the only owner of semantic expression-flow branching
* predicate should become an adapter, not a flow participant
* prepared/session should map planner-owned contracts, not infer structure
* executor should consume planner-frozen expression contracts without
  reclassifying or rerouting expression families

---

## Why `0.118` Exists

Once semantic authority is centralized, the remaining bottleneck is not
correctness or feature coverage.

It is change locality.

The current expression lane can still read like:

* lowering builds expression-shaped structure
* predicate reshapes or re-expresses that structure
* planner types and classifies it
* prepared/session layers map it again
* executor consumes the result

Even if semantics are now unified, that flow still pays too many modules per
change.

That is why `0.118` is the right next move:

* completeness no longer points at a missing product surface
* canonical semantic authority no longer points at missing semantic ownership
* velocity still points at the expression cluster as the live cross-layer drag

This is the first slice in the line whose purpose is not to define meaning
better.

Its purpose is to make that meaning flow through fewer owners.

The first clean way to do that is to remove predicate as the clearest
removable middle hop in the expression path rather than trying to flatten the
entire pipeline at once.

## Problem

The system can still have centralized expression semantics while retaining a
distributed expression flow.

That means the broad execution path can still look like:

* lowering builds one intermediate expression-shaped form
* predicate reshapes or recompiles it into another form
* planner branches on the canonical expression model
* prepared/session layers perform additional expression-family or contract
  routing
* executor consumes the final shape

Even if all of that is semantically correct, it creates:

* unnecessary touched-module count
* hotspot concentration across multiple adjacent layers
* repeated structural branching outside planner
* a routine need to modify predicate, lowering, planner, and prepared/session
  code together for one expression follow-through change

The problem is no longer expression meaning.

The problem is expression flow.

---

## What “Expression Pipeline Flow Collapse” Means

For `0.118`, flow collapse means reducing the number of layers that actively
reshape, branch on, or route expression structures before execution.

That includes:

### Included Flow Targets

* removing lowering -> predicate -> planner chaining where lowering can target
  planner-compatible `Expr` directly
* removing predicate-local expression rebuilding that mirrors planner flow
* reducing prepared/session expression routing to planner-contract consumption
* keeping executor on the consumer side of planner-frozen expression contracts
* making planner the only owner of semantic expression-flow branching
* replacing multi-stage expression routing with a direct
  lowering -> planner -> consumer path

### Not The Focus Of This Slice

* new expression semantics
* new SQL features
* new grouped semantics
* expression-family meaning already targeted by `0.117`
* executor algorithm redesign
* route planner redesign
* storage/index redesign
* generalized all-layer flattening

So `0.118` is not another semantic-authority contraction.

It is the mechanical-flow contraction that follows those semantic contractions.

---

## Goal

Make planner the single execution-path owner for expressions.

All adjacent layers must either:

* build planner-compatible structure before planner
* consume planner-owned expression outputs after planner

They must not:

* rebuild expression families locally
* branch on expression family outside planner-owned contracts
* perform second-stage expression routing that planner can already own
* reinterpret canonical expression shape after planner has frozen it

---

## Core Invariant

Expression flow is defined exactly once around planner-owned canonical `Expr`
and planner-owned typed contracts.

Everything else is:

* structure-only adaptation before planner
* contract-only consumption after planner

If a branching decision about expression flow exists both in planner-owned
expression logic and in an adjacent layer, planner-owned logic must be the
single owner of semantic expression-flow branching.

Adjacent layers may retain thin structural or compatibility branches that do
not change or route expression meaning.

Structural adaptation is allowed.

Mechanical rerouting and expression-family branching outside planner are not.

---

## Ownership Boundary

`0.118` should make the flow split visible in code shape.

### Parser / Lowering Owns

* SQL-specific syntax
* raw structural expression construction
* direct lowering into planner-compatible `Expr`
* trivial syntax adaptation needed to produce planner inputs

Parser / lowering must not own:

* intermediate expression routing stages that predicate or prepared code must
  reinterpret later
* branching on expression family where planner already exposes the answer
* expression flow decisions that require planner-owned classification

### Predicate Owns

* adaptation between planner-owned expression truth/contracts and legacy
  runtime predicate APIs
* final runtime predicate consumption where a predicate surface still exists
* boundary-local shape transport from planner-owned truth/forms into runtime
  predicate structures

Predicate must not own:

* expression rebuilding as a primary flow stage
* compare-family or membership-family routing that mirrors planner flow
* local branching that decides expression family or execution path
* canonical expression-family construction outside planner
* planner-equivalent branch ladders for compare/null/membership routing

### Planner Owns

* canonical `Expr`
* expression typing and classification
* the main branching point for expression execution flow
* canonical planner-owned contracts consumed by prepared/session and executor

This is the single execution-path owner for expressions.

### Prepared / Session Owns

* parameter binding
* contract mapping
* execution wiring
* outward result/projection handling

Prepared / session must not own:

* expression structure inference
* expression-family branching
* independent routing logic that duplicates planner flow

### Executor Owns

* execution of already-frozen planner-owned contracts

Executor must not own:

* expression-shape reinterpretation
* family reclassification
* fallback flow branching that belongs in planner

## Concrete Actions

### 1. Kill Predicate As A Flow Participant

Start by inventorying the expression-routing work in
[bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs:1).

Identify code that still:

* rebuilds expression trees
* converts compare families
* performs structural rewrites that planner could own
* mirrors planner branching in a second lane

This is the primary contraction target for `0.118`.

The target is not to delete predicate abstractions entirely.

The target is to stop using predicate as a major expression-flow stage.

Predicate should:

* call planner normalization/classification
* adapt planner outputs into runtime predicate form
* convert planner-owned truth forms into runtime predicate structures where a
  predicate surface still exists
* perform boundary-local shape transport only

Predicate should not:

* behave like a second expression router
* rebuild canonical expression families
* choose compare/null/membership routing
* own planner-equivalent branch ladders

### 2. Remove Lowering -> Predicate -> Planner Chaining

Inventory the current chain where:

* lowering builds something expression-shaped
* predicate reshapes it
* planner reinterprets it

Then collapse that to:

* lowering outputs planner-compatible `Expr`
* planner owns the first real semantic branch
* predicate consumes planner output only when a runtime predicate shell is
  still required

This is a major locality gain because it removes one whole intermediate hop.
It should be pursued only as far as needed to support the predicate-flow
collapse cleanly inside this slice.

### 3. Collapse Prepared Fallback Flow

Prepared/session code should stop acting like expression-routing participants.

That means:

* prepared should not infer structure
* prepared should not branch on expression family
* prepared should not perform independent expression flow routing

Prepared/session should:

* read planner typing/classification results
* map them to parameter contracts
* wire execution using planner-frozen outputs

### 4. Make Planner The Only Branching Point

Map expression branches across:

* lowering helpers
* predicate helpers
* prepared fallback helpers
* planner
* executor-adjacent expression helpers

Then reduce them to one owner:

* planner owns the material semantic expression branch ladders

Everything else should become:

* linear
* adapter-only
* contract-only
* structurally branched only where syntax or compatibility transport requires
  it

### 5. Add Flow Invariants

`0.118` is not complete without invariants that prove the flow collapse.

Add guards that prove:

* predicate does not construct new expression families as part of normal flow
* lowering does not branch on expression semantics outside planner-owned
  contracts
* prepared/session do not infer expression family or structure independently
* executor does not reinterpret canonical expression shape
* planner remains the only meaningful semantic expression branching owner

These are flow invariants, not semantic invariants.

## Recommended Work Order

### 1. Inventory Remaining Expression Flow Hops

Map the concrete flow across:

* lowering
* predicate
* planner
* prepared/session
* executor-facing consumers

Do not move code blindly.

The first job is to identify where the same expression is still being routed
through multiple structural owners.

### 2. Collapse Predicate Flow First

The best first contraction target inside `0.118` is:

* removing predicate as a major expression-flow stage

Why first:

* the velocity rerun names `bool_expr.rs` as explicit drag
* the predicate hop is currently the clearest removable middle layer
* removing it should reduce both branch density and touched-module count

This is the main body of the slice, not just the first step in a broader
pipeline rewrite.

### 3. Collapse Lowering Directly Into Planner Inputs

Once predicate flow is thinner, remove remaining
lowering -> predicate -> planner chaining.

The target is direct lowering into planner-compatible `Expr`.

### 4. Narrow Prepared / Session To Contract Mapping

After planner owns the main path, strip remaining prepared/session flow logic
down to:

* contract mapping
* binding
* execution wiring

### 5. Freeze The New Shape With Invariants

Treat the flow invariants as the guardrail that stops the old multi-hop
expression path from reappearing in future slices.

## Exit Condition

`0.118` is done when:

* planner is the only owner of semantic expression-flow branching
* predicate no longer acts as a major expression routing stage
* lowering targets planner-compatible `Expr` directly wherever that is
  feasible in the shipped surface
* prepared/session consume planner-owned contracts instead of inferring
  expression structure
* executor consumes planner-frozen expression contracts without
  reclassification or rerouting
* routine expression follow-through touches fewer modules than the pre-slice
  baseline
* routine truth/expression follow-through in the shipped surface no longer
  requires coordinated changes across predicate + lowering + planner for the
  same ownership rule

This should be visible as:

* lower hotspot pressure in the expression cluster
* `bool_expr.rs` becoming materially less central to routine expression
  follow-through
* a lower median blast radius in the next velocity audit
* fewer expression-related slices touching predicate, lowering, planner,
  prepared, and session together

The explicit velocity target remains:

* median blast radius `<= 8`

## Summary

`0.118` is the first flow-contraction slice after semantic centralization.

It does not target meaning.

It does not target features.

It targets the mechanical cost of expression meaning moving through the system.

The intended end state is simple:

* lowering builds planner-compatible structure
* planner owns the single execution path for expressions
* predicate becomes adapter-only
* prepared/session become contract-only consumers
* executor consumes planner-frozen expression contracts

If `0.116` and `0.117` centralize semantic ownership successfully, `0.118`
should remove the remaining velocity tax by collapsing the expression pipeline
itself.
