**0.83 Serialization Boundary Cleanup**

## Thesis

The runtime now has a clear performance direction:

* hot storage paths increasingly use custom row and slot codecs
* hot index and covering paths increasingly use custom ordered codecs
* the remaining generic `crate::serialize` / CBOR surfaces sit mostly at
  protocol edges, row-envelope edges, and complex-field fallback seams

That means the next cleanup line should not be “add one more fast path around
CBOR.”

`0.83` should instead **remove `crate::serialize` and CBOR from core runtime
boundaries wherever the engine already has a better custom ownership model**.

This is primarily a runtime-boundary and decode-hygiene slice.
Wasm shrink is a real supporting outcome, but not the only reason to do it.

---

## Problem statement

IcyDB currently carries three serialization domains:

* `Candid` at the public canister boundary
* `CBOR` through the generic `crate::serialize` helper
* custom codecs for hot row, slot, index, and covering paths

That split is technically defensible, but it has two real costs:

1. conceptual cost
* it is too easy to ask “what format does IcyDB use?” and get a non-answer
* core runtime ownership becomes harder to reason about because generic CBOR
  seams still exist beside custom storage seams

2. implementation cost
* generic CBOR support pulls in code and behavior that hot paths no longer
  want
* duplicate boundary styles make it easier for new generic serialization calls
  to drift back into `db::data`, `db::executor`, or `db::index`

So the question for `0.83` is:

> Which remaining `crate::serialize` / CBOR boundaries are still in core
> runtime ownership, and which of them should be replaced by explicit custom
> engine codecs now?

---

## Scope

### In scope

* serialization boundaries inside:
  * `db::data`
  * `db::executor`
  * `db::index`
  * `db::cursor`
  * `db::codec`
* removal of `crate::serialize` from core runtime hot paths
* replacement of CBOR row-envelope / token boundaries with explicit custom
  codecs where practical
* explicit policy on which remaining complex-value seams may temporarily stay
  on CBOR fallback
* wasm-size impact as a secondary measured result

### Out of scope

* removing `Candid` from public canister surfaces
* a one-patch rewrite of every complex value type in the repo
* changing public query or mutation semantics
* changing cache identity or planner semantics
* broad schema redesign such as per-field-addressable stable storage

---

# Architectural position

## Keep `Candid` at the public edge

Public canister query/update surfaces should remain `Candid`.

That is not the cleanup target.

The target is the internal runtime where generic CBOR is still doing work that
custom engine codecs should own directly.

## Runtime serialization should become purpose-specific

The intended steady state is:

* public canister RPC: `Candid`
* core storage/index/runtime: custom engine codecs
* no generic `crate::serialize` dependency in hot runtime paths

If a generic fallback remains, it should be:

* explicitly scoped
* clearly owned
* not on the dominant read path

---

# Primary design goal

## Shrink the authority of `crate::serialize`

Today:

* `crate::serialize` is the generic CBOR helper
* several DB modules still lean on it for protocol or persisted-payload work

For `0.83`, the design should say explicitly:

> `crate::serialize` is no longer an acceptable default dependency inside core
> runtime modules. New runtime ownership must prefer explicit custom codecs,
> and the remaining generic-CBOR seams must be named rather than assumed.

# Execution order

## Phase 1: cursor tokens off CBOR

Cursor tokens are a good first target because they are:

* self-contained
* bounded
* already versioned/validated by token-specific code
* easier to rewrite than complex field payloads

Desired state:

* scalar and grouped continuation tokens no longer use `crate::serialize`
* token wire format is explicit and owned by token modules
* existing bounded-size checks and corruption behavior remain

This is a clean proof that protocol payloads do not need the generic serializer.

## Phase 2: persisted row outer envelope off CBOR

The inner row payload is already custom.
The outer row envelope should stop pretending to be generic CBOR.

Desired state:

* the row envelope format becomes an explicit engine-owned binary wrapper
* `db::codec` no longer needs CBOR to wrap `(format_version, payload)`
* hot and cold row decode paths agree on one explicit envelope contract

This matters because the engine should not need one generic serializer just to
store a version byte plus one owned payload blob.

## Phase 3: ban new runtime `crate::serialize` use

Once tokens and row envelope have custom owners, the repo should add a policy:

* no new `crate::serialize::*` use in:
  * `db::data`
  * `db::executor`
  * `db::index`
  * `db::cursor`

This can be enforced by code review first and a lightweight CI grep/tripwire
second.

## Phase 4: decide complex-field fallback scope

This is the only part that may or may not fully fit in `0.83`.

Two acceptable `0.83` outcomes exist:

### Option A

Replace at least some complex-field fallback codecs with explicit engine-owned
custom codecs.

### Option B

Leave complex-field fallback temporarily on CBOR, but make that exception
explicit and isolated so the core runtime no longer depends on generic CBOR
outside that fallback seam.

The important part is honesty:

* if complex fallback remains, it should be named as the deliberate residual
  CBOR owner
* not left as an invisible ambient dependency

# Wasm-size position

Yes, `0.83` should expect wasm-size improvement.

Why:

* deleting `serde_cbor`-driven runtime paths removes code
* fewer parallel encode/decode boundaries reduce duplicate logic
* hot-path modules no longer need to carry generic serializer coupling

But the design should still treat this as a **secondary benefit**.

The primary reasons remain:

* cleaner boundary ownership
* fewer accidental generic runtime decode paths
* a more coherent storage/index/runtime model

So `0.83` should say:

> wasm shrink is expected and should be measured, but it is not sufficient by
> itself. The slice succeeds only if runtime ownership becomes clearer and
> generic CBOR authority actually shrinks.

# Acceptance criteria

## Functional

* cursor tokens no longer depend on `crate::serialize`
* persisted row outer envelope no longer depends on `crate::serialize`
* no new runtime `crate::serialize` use appears in `db::data`, `db::executor`,
  `db::index`, or `db::cursor`
* hot query semantics remain unchanged
* public canister `Candid` surfaces remain unchanged

## Architectural

* core runtime serialization boundaries are easier to explain:
  * `Candid` at public edge
  * custom engine codecs in runtime/storage/index paths
  * any remaining CBOR owner explicitly named
* `crate::serialize` is no longer an ambient runtime dependency

## Performance

* no regression on the current hot covering/read paths
* if token or row-envelope decode shows up on measured paths, those paths stay
  flat or improve

## Size

* raw wasm bytes should improve or at least not regress materially
* shrunk `.wasm` remains secondary context, not the primary accept/reject
  metric

# Verification

Verification should cover four things:

## 1. Boundary ownership

* search confirms the intended runtime modules no longer import
  `crate::serialize::*`
* any remaining CBOR fallback owners are explicitly named

## 2. Wire stability

* cursor token round-trips remain correct
* token wire snapshots are either preserved or intentionally rebaselined once
  with explicit note
* persisted row format versioning remains explicit and fail-closed

## 3. Runtime semantics

* persisted rows still decode safely
* continuation tokens still validate size and corruption correctly
* hot covering and hybrid SQL read paths remain correct

## 4. Wasm footprint

* record raw wasm before/after for the representative audit canisters
* mention gzip only as secondary context

---

# Thesis sentence

> **`0.83` should make custom engine codecs the default runtime authority and
> demote CBOR from “ambient helper” to a named, shrinking exception.**
