# 0.148 Schema Evolution Runtime Cutover Status

## Status

In progress.

0.148 has started with the runtime decode foundation and the first additive
startup policy. The current implementation accepts only append-only nullable
field additions, publishes the generated snapshot as the new accepted schema,
and lets accepted row contracts read older shorter physical rows.

## Target Outcome

```text
append-only nullable field added in generated proposal
  -> startup accepts transition
  -> old rows read with NULL for missing field
  -> new rows write current accepted layout
  -> unsupported transitions still fail closed
```

## Completed Additive Runtime Foundation

The first 0.148 implementation slice added absence-policy-aware row decode and
the startup transition policy needed to reach it safely:

* `StructuralRowContract` now validates physical slot count separately from
  accepted logical slot count.
* Accepted contracts can synthesize `Value::Null` for missing trailing nullable
  fields through `AcceptedFieldAbsencePolicy::NullIfMissing`.
* Primary-key validation and primary-key projection use accepted field decode
  contracts before falling back to generated field metadata.
* Generated-only contracts still require exact physical slot counts.
* Dense, sparse indexed, one-slot required, and lazy `StructuralSlotReader`
  paths share the same missing-slot behavior.
* Focused tests open a two-slot old row through three-slot accepted contracts
  and verify cached, dense, sparse, compact, and required-slot reads all return
  `NULL` for the missing appended nullable field while a missing appended
  required field still fails closed.
* Schema transition policy accepts only append-only nullable generated fields.
* Reconciliation writes the generated snapshot as the accepted schema when that
  transition is accepted.
* Schema transition metrics now count accepted append-only nullable field
  transitions separately from exact matches and rejection buckets.
* SQL SELECT coverage installs an old accepted two-field schema, inserts a
  two-slot physical row, accepts the nullable additive transition, and returns
  `NULL` for the missing appended field.
* SQL predicate coverage over the appended nullable field verifies `IS NULL`
  matches the old missing-slot row while `IS NOT NULL` and `= NULL` do not.
* SQL metadata coverage verifies `DESCRIBE` and `SHOW COLUMNS` expose the
  accepted appended nullable field and accepted slot after reconciliation.

Next gate: SQL/session write coverage over actual stored old rows.
