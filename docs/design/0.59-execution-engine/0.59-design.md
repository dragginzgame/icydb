IcyDB 0.59 Design

Status: In Progress

Theme: Execution Engine De-monomorphization (Runtime-Driven Execution)

Supersedes:

- Implicit continuation of the 0.58 structural optimization phase

## 0. Summary

0.59 begins after 0.58 has already removed the high-leverage structural duplication:

- orchestration is structurally de-monomorphized
- runtime boundaries are mostly structural
- wrapper and adapter fanout has been measured and reduced where it paid
- remaining profitable local cuts have been exhausted

The remaining duplication is now concentrated in typed execution bodies.

Examples:

- `grouped_fold::<E>`
- SQL execution paths under `::<E>`
- mutation and validation execution paths under `::<E>`

That changes the problem.

`0.59` is not a continuation of planner and runtime cleanup.

It is a new phase focused on removing monomorphization pressure from execution
kernels themselves.

## 1. Problem Statement

### 1.1 Current State

Execution still duplicates per entity because the execution kernels remain generic.

This means the system is now in a deliberate but size-heavy state:

- orchestration is shared
- execution remains typed
- WASM size is still paying for repeated entity-specialized execution logic
- instruction-cache pressure remains higher than necessary because multiple
  execution variants still exist for the same logical work

### 1.2 What 0.58 Proved

0.58 measured the obvious structural cuts and established the boundary of profitable local optimization.

Proven wins:

- grouped pipeline flattening
- traversal de-monomorphization
- physical access runtime flattening
- write-path consolidation

Proven non-wins:

- SQL dispatch extraction
- SQL generator collapse
- grouped adapter removal

Conclusion:

Local wrapper movement is no longer the problem. Generic execution itself is now the duplication source.

### 1.3 Required Shift

We must move from:

- compile-time specialization in execution kernels

to:

- runtime-driven execution over structural inputs
- shared execution kernels parameterized by compact runtime representations

## 2. Design Goals

### 2.1 Primary Goal

Eliminate `<E>` from execution kernels while preserving behavior and acceptable performance.

### 2.2 Secondary Goals

- keep planner behavior unchanged
- keep parser behavior unchanged
- avoid `dyn Trait` in hot execution loops
- reduce WASM size materially
- preserve runtime performance within acceptable bounds
- keep typed logic confined to thin boundary resolution only
- keep execution data-driven, not entity-variant-driven

### 2.3 Non-Goals

- planner redesign
- parser changes
- schema changes
- query language expansion
- JOIN support

## 3. Design Principle

Control flow must be monomorphic.

Type-specific behavior must be resolved once at the boundary, then represented as structural data or compact runtime enums.

The important constraint is:

- no per-row `match EntityTag` in hot loops

If dispatch is required, it must happen before the loop begins.

Entity-specific behavior must be encoded as structural descriptors or compact
behavioral enums, not as one runtime type per entity.

## 4. Target Architecture

### 4.1 Before

```text
execute::<E>
  -> lowering
  -> planning
  -> execution
  -> projection
```

The execution portion is still duplicated per entity.

### 4.2 After

```text
typed boundary
  -> resolve once
  -> structural runtime descriptors
  -> shared execution loop
  -> typed leaf only if still strictly necessary
```

The shared loop must operate only on:

- `EntityTag`
- `EntityModel`
- `RowView`
- `Value`
- structural slots, ids, and descriptors

## 5. Execution Model

### 5.1 Entity Identity

`EntityTag` remains the runtime identity authority.

Execution entrypoints must accept structural identity, not `<E>`.

Example:

```rust
fn execute_runtime(tag: EntityTag, plan: RuntimePlan) -> Result<RuntimeResult, InternalError>
```

### 5.2 Structural Data Access

Execution loops operate on:

- `RowView`
- `Value`
- precomputed field slots
- precomputed aggregate descriptors
- structural route metadata

Execution loops must not depend on:

- `E::MODEL`
- `E::Field`
- `E::Key`
- typed entity materialization

### 5.3 Runtime Dispatch

0.59 does not pivot to `dyn Trait` in hot paths.

Dispatch must be represented as:

- structural descriptors
- compact enums
- one-time boundary resolution

Preferred pattern:

```rust
struct ExecutionRuntime {
    model: &'static EntityModel,
    route: RuntimeRoute,
    projection: ProjectionDescriptor,
    aggregates: &'static [AggregateDescriptor],
}
```

Then resolve once:

```rust
let runtime = resolve_runtime(tag);
```

and execute without repeated entity branching inside the row loop.

The design preference is:

- structural descriptors
- compact behavior enums
- tagged unions
- table-driven execution where appropriate

The design preference is not:

- per-entity runtime structs
- trait-object dispatch in inner loops

Reasoning:

- `dyn Trait` can reduce code size, but it introduces indirect calls in the
  hottest execution paths
- structural descriptor dispatch preserves predictability and avoids simply
  replacing `<E>` monomorphization with large entity-variant enums

Compact enums are still appropriate for narrow behavioral switches such as:

- aggregate kind
- continuation mode
- projection opcode families

They are not the preferred representation for entire entity execution logic.

## 6. Main Slices

### 6.1 Grouped Fold

This is the primary slice.

Target:

- remove `<E>` from grouped execution kernels
- keep grouped orchestration structural
- replace typed aggregate engine ownership with runtime-owned structural state

Current shape:

- `grouped_fold::<E>(...)`
- typed grouped execution bodies

Target shape:

```rust
fn grouped_fold_runtime(
    runtime: &ExecutionRuntime,
    route: &GroupedRoute,
    rows: impl Iterator<Item = RowView>,
) -> GroupedResult
```

Aggregate engine representation should be structural, for example:

```rust
enum AggregateEngineRuntime {
    Count(CountState),
    Sum(SumState),
    Avg(AvgState),
}
```

Aggregate runtime state must be fully structural and must not carry generic
type parameters.

Good:

```rust
struct CountState {
    value: u32,
}
```

Bad:

```rust
struct CountState<E> {
    value: u32,
    _marker: PhantomData<E>,
}
```

The important rule is that the grouped row loop stays shared.

### 6.2 Current Implementation Checkpoint

`0.59.0` landed the grouped-runtime checkpoint and the next scalar aggregate
checkpoint has now been taken.

What is already structural:

- grouped runtime ownership
- grouped aggregate engine/state ownership
- scalar aggregate reducer state
- scalar aggregate kernel output
- scalar load entrypoint orchestration stage
- scalar row-production/runtime handle
- scalar row decode layout/decoder contract

What changed in the scalar aggregate checkpoint:

- scalar aggregate reducer state no longer owns `Id<E>`
- scalar aggregate kernel output no longer owns `AggregateOutput<E>`
- scalar aggregate kernel now owns structural `StorageKey`-backed identity state
- typed terminal code decodes kernel outputs at the boundary instead of owning
  reducer execution

What changed in the scalar load checkpoint:

- scalar load execution no longer uses `ExecutablePlan<E>` in the shared scalar core
- scalar load execution no longer uses `LoadExecutor<E>` in the shared scalar core
- typed entrypoints now build one structural scalar execution stage and hand it
  to the monomorphic scalar runtime
- route plan, execution preparation, continuation state, and lowered bindings
  now travel through one structural scalar stage contract
- scalar page materialization now runs through one erased row-runtime handle
  instead of typed per-row callbacks
- scalar row-runtime state is now resolved once at the typed boundary and then
  cloned structurally into page materialization
- scalar runtime now assembles one structural page payload
  (`DataRow + cursor`) before typed decode
- typed scalar entrypoints now decode the finished structural page only after
  the shared scalar runtime returns

What changed in the structural row-decode checkpoint:

- scalar row production no longer deserializes `E` inside the scalar loop
- persisted row bytes are now decoded through one structural `RowLayout +
  RowDecoder` contract
- scalar row production now reads:
  `RawRow -> structural slot values -> KernelRow`
- optional/null persisted fields now stay as `Value::Null` in structural row
  state instead of forcing typed inner-field decode
- primary-key consistency is now enforced structurally at row-decode time by
  comparing the decoded primary-key slot with the authoritative `DataKey`

What changed in the aggregate terminal-stage checkpoint:

- scalar aggregate terminals no longer thread `ExecutablePlan<E>` through their
  internal zero-window, fast-path, and dispatch control flow
- typed `LoadExecutor<E>` terminal APIs now prepare one
  `PreparedAggregateTerminalStage` boundary object once, then hand terminal
  routing to shared free aggregate helpers
- index-covering `COUNT` / `EXISTS` terminal fast paths now execute from that
  prepared terminal stage instead of reconstructing terminal runtime state per
  branch
- aggregate reducer selection in `aggregate/mod.rs` now carries prepared
  aggregate execution state instead of handing an `ExecutablePlan<E>` through
  the streaming branch
- field-target extrema aggregate dispatch now consumes the prepared aggregate
  stage directly instead of recovering it from `ExecutablePlan<E>` inside the
  extrema helper

What changed in the aggregate prepared-state checkpoint:

- scalar aggregate execution now has one canonical
  `PreparedAggregateExecutionState` that carries descriptor state plus prepared
  runtime inputs after the typed boundary consumes `ExecutablePlan<E>`
- scalar aggregate materialized fallbacks now execute from prepared stage state
  and the scalar materialized rows path no longer reconstructs
  `ExecutablePlan<E>` internally
- scalar aggregate terminal dispatch no longer rebuilds `ExecutablePlan<E>` to
  re-enter aggregate execution after terminal fast-path checks
- the legacy aggregate rebuild helpers are gone from scalar aggregate
  execution; execution now flows:
  `prepared stage -> reducer selection -> runtime/kernel`

Architecturally, that means scalar aggregate execution has crossed the same
ownership boundary that grouped execution already crossed:

- before: typed execution owned the loop and the reducer state
- after: the structural kernel owns reducer state and typed code only decodes
  final outputs

This is the correct intermediate state, but it is not the end state.

The remaining scalar leak is now narrower:

- `ExecutionRuntimeAdapter<'_, E>` still exists as the typed boundary adapter
- typed boundary setup still resolves store/model authority before entering the
  shared scalar runtime
- aggregate terminal wrappers still exist on `LoadExecutor<E>`, but terminal
  routing beneath them is now stage-driven rather than plan-owned
- scalar materialized aggregate execution still needs to stop borrowing generic
  load semantics where route-budget behavior is aggregate-owned
- final typed page/entity reconstruction still happens at the outer boundary

The system is therefore still effectively:

```text
structural scalar execution stage
  -> structural scalar loop
  -> erased row runtime boundary for structural store-backed row production
  -> structural row decode
  -> structural page assembly / structural scalar kernel where applicable
  -> typed terminal decode
```

The target remains:

```text
structural scalar loop
  -> structural page/runtime materialization
  -> structural scalar kernel
  -> typed terminal decode only
```

### 6.3 Scalar Load Execution

This is now the primary remaining execution slice.

Target:

- remove `<E>` from scalar execution loops
- move scalar iteration/materialization/projection feeding behind one
  structural runtime contract
- keep typed decode only at entrypoint and terminal boundaries

Current hot duplication still exists in:

- typed store/model resolution at scalar boundary construction time
- raw row access still flows through one boundary-resolved store handle instead
  of a fully runtime-owned storage descriptor
- final typed page reconstruction at the outer scalar boundary
- aggregate materialized fallback paths that still inherit load-planner
  shortcut semantics unless route-budget ownership is made explicit

Target shape:

```rust
struct ScalarExecutionPlan {
    route: RuntimeRoute,
    projection: ProjectionDescriptor,
    kernel: ScalarKernelSpec,
}

fn execute_scalar_plan(
    runtime: &ExecutionRuntime,
    plan: &ScalarExecutionPlan,
) -> Result<ScalarAggregateOutput, InternalError>
```

Required ownership split:

- structural runtime layer owns iteration, projection application, and kernel invocation
- thin typed adapter resolves `E`, constructs the runtime plan, and decodes final outputs

This has partially been achieved already:

- typed adapters now construct the scalar execution stage
- shared scalar entrypoint orchestration is monomorphic
- shared scalar page materialization no longer calls typed row callbacks directly
- scalar row production now flows through one erased runtime handle
- scalar row-runtime descriptors are now recovered once at the typed boundary
  instead of being rebuilt during page materialization
- shared scalar runtime now owns structural page assembly before typed decode
- scalar aggregate terminals now prepare one typed boundary stage and dispatch
  through shared terminal helpers instead of passing `ExecutablePlan<E>` through
  each terminal branch
- aggregate reducer streaming selection now preserves prepared aggregate state
  instead of preserving `ExecutablePlan<E>` as the streaming payload

It is not complete until typed boundary setup is reduced to the minimal
descriptor-construction step and materialized aggregate fallbacks stop inheriting
load-specific budget shortcuts.

This row-decode checkpoint closes the larger of those two remaining leaks:

- row decode is now structural and model-driven
- scalar loop semantics no longer depend on `E`-typed entity materialization

The remaining scalar boundary authority is now:

- typed store/model resolution at adapter construction time
- raw row reads through the boundary-resolved entity store handle
- typed terminal wrapper APIs that still receive `ExecutablePlan<E>` at the
  public boundary
- final typed page/entity decode after structural execution completes

This wrapper-collapse checkpoint narrowed the remaining aggregate leak further:

- numeric aggregate materialized fallbacks now consume prepared aggregate
  streaming inputs instead of calling the generic `execute(plan)` load surface
- ranked helper aggregates (`nth`, `median`, `min_max`) now materialize rows
  from prepared aggregate stage state rather than from `ExecutablePlan<E>`
- projection materialized fallbacks now consume prepared aggregate stage inputs
  instead of borrowing the generic load execution surface
- FIRST/LAST field-value projection terminals now prepare aggregate execution
  state explicitly and execute prepared aggregate state directly instead of
  calling the older plan-based aggregate execution entrypoint
- `count_distinct(field)` now prepares aggregate stage inputs once and executes
  distinct projection from prepared state instead of validating the target slot
  and then re-entering a second plan-owned wrapper
- prepared aggregate inputs now own the normalized
  `window_is_provably_empty()` check so numeric and terminal zero-window gates
  share one effective-window authority instead of reading pagination/path flags
  from separate plan-facing helpers
- scalar aggregate public wrappers now share one explicit typed boundary helper:
  `prepare_scalar_aggregate_boundary(plan)`
- terminal, numeric, and projection wrappers now prepare once and immediately
  hand off to family boundary dispatchers plus `*_from_prepared(...)` helpers
  instead of letting wrapper methods own aggregate execution branching
- scalar aggregate now has one typed dispatcher per wrapper family
  (`terminals`, `numeric`, `projection`) so the boundary shape is:
  `public wrapper -> family dispatcher -> prepared aggregate execution`
- obsolete wrapper-era compatibility helpers for terminal value projection were
  removed instead of kept as dead shims
- scalar ranking terminals now follow the same family-boundary shape:
  `public wrapper -> ranking family dispatcher -> shared materialized ranking logic`
- `take`, `top_k_by`, `bottom_k_by`, and ranked value/id projections no longer
  each own their own `execute(plan)` setup path
- scalar byte terminals now also follow one family-boundary dispatcher:
  `public wrapper -> bytes family dispatcher -> shared bytes execution logic`
- `bytes()` and `bytes_by(field)` no longer each own separate
  plan/materialization setup paths
- ranking/bytes family dispatchers now prepare one neutral non-aggregate
  scalar boundary payload (`PreparedScalarMaterializedBoundary`) and execute
  through that shared boundary instead of preparing directly from
  `ExecutablePlan<E>` inside each family
- `bytes_by(field)` covering-constant and covering-index branches now execute
  from that neutral prepared boundary too; unsupported covering component
  decodes fall back to structural materialized execution instead of rebuilding
  typed plan-owned helpers
- grouped DISTINCT aggregate handoff is now split explicitly into:
  `scalar DISTINCT intent -> grouped boundary lowering -> grouped output decode`
  instead of keeping grouped lowering, execution, and scalar decode inline in
  one helper
- grouped DISTINCT no longer rebuilds a grouped `ExecutablePlan<E>` beneath
  that scalar boundary; it now lowers straight to one prepared grouped route
  stage and executes that route directly
- pipeline-layer scalar boundary code no longer imports planner covering
  helpers directly; byte-projection mode classification now lives back in the
  terminal family boundary so the pipeline layer stays structural
- aggregate terminal production callers now invoke the shared scalar terminal
  request boundary directly instead of routing through one method per terminal
  name, and the old terminal helper names are now gone from both production
  code and matrix-test call sites
- numeric aggregate callers now invoke `execute_numeric_field_boundary(...)`
  with `ScalarNumericFieldBoundaryRequest` directly; the old
  `aggregate_{sum,avg}[_distinct]_by_slot(...)` wrapper names are gone
- scalar projection callers now invoke `execute_scalar_projection_boundary(...)`
  with `ScalarProjectionBoundaryRequest` directly; the old
  `values_by_slot(...)`, `distinct_values_by_slot(...)`,
  `values_by_with_ids_slot(...)`, `first_value_by_slot(...)`,
  `last_value_by_slot(...)`, and `aggregate_count_distinct_by_slot(...)`
  wrapper names are gone

This prepared-contract checkpoint moves the next leak out of execution:

- numeric scalar terminals now lower typed requests into one non-generic
  `PreparedScalarNumericBoundary` plus runtime payload before execution
- projection terminals now lower typed requests into one non-generic
  `PreparedScalarProjectionBoundary` plus runtime payload before execution
- those prepared boundary contracts now carry resolved field metadata,
  operation kind, and selected execution strategy without retaining
  `ExecutablePlan<E>`
- numeric execution no longer inspects plan shape during execution; it matches
  only `Streaming`, `Materialized`, or `GlobalDistinctGrouped` strategy from
  the prepared boundary
- projection execution no longer derives covering/distinct/constant strategy
  on the fly; preparation resolves `Materialized`, `CoveringIndex`, or
  `CoveringConstant` up front, including covering DISTINCT dedupe policy and
  effective covering window bounds
- grouped global DISTINCT numeric execution now also has a prepared route
  boundary: typed preparation lowers to one grouped route stage, and execution
  runs that route directly instead of carrying scalar plan ownership through
  the DISTINCT path
- covering-index projection execution still retains one runtime fallback from
  prepared covering strategy to materialized execution when the encoded index
  component cannot be decoded into a runtime `Value`; this fallback is
  execution-data driven, not plan-policy driven

After this checkpoint, the main remaining scalar plan-derived policy is in the
terminal family boundary (`COUNT` / `EXISTS` / id terminals), where fast-path
contracts are still derived directly from `ExecutablePlan<E>` before execution.

This terminal prepared-contract checkpoint removes that remaining leak for the
core scalar terminal family:

- `COUNT`, `EXISTS`, `IdTerminal`, and `IdBySlot` now lower into one
  `PreparedScalarTerminalBoundary` plus prepared runtime payload before
  execution
- terminal execution now matches one non-generic
  `PreparedScalarTerminalStrategy` instead of receiving route fast-path
  contracts derived inline from `ExecutablePlan<E>`
- prepared terminal strategy now owns:
  - PK-cardinality `COUNT`
  - existing-row `COUNT`
  - existing-row `EXISTS`
  - kernel aggregate fallback
- `EXISTS` remains streaming-native in the prepared terminal contract; the
  fast path still executes as one existing-row reducer fold and does not
  materialize rows just to test non-emptiness
- id terminals now keep only resolved aggregate op metadata in the prepared
  boundary and execute through the kernel aggregate path without plan-derived
  branching in the runtime helper

After this checkpoint, the remaining scalar `ExecutablePlan<E>` ownership is
primarily the public typed boundary surface itself plus non-aggregate terminal
families that still accept plans at their outer API edge.

What still remains after that checkpoint:

- typed outer APIs still lower from query/session surfaces into executor
  boundary requests, but the executor-side scalar family fanout has been
  collapsed onto explicit boundary request contracts
- terminal boundary setup still derives some fast-path policy from
  `ExecutablePlan<E>` before consuming it into prepared stage state

One concrete lesson from this checkpoint:

- scalar runtime scan accounting must count scanned keys, not only rows that
  survive typed decode/predicate filtering
- the first erased row-runtime cut accidentally undercounted filtered keys in
  `rows_scanned`
- this surfaced immediately in materialized aggregate tests because aggregate
  fallback depends on load scan accounting even when the aggregate kernel is
  structural

The critical rule for this slice is:

- do not reintroduce typing through generic closures or typed callback tables

Forbidden disguised regressions include:

- `fn execute<F: FnMut(&E)>(...)`
- boxed typed callbacks such as `Box<dyn FnMut(&E)>`
- typed row materialization inside the main scalar loop

Preferred scalar runtime inputs are:

- `RowView` / `KernelRow`
- `Value`
- precomputed field slots
- precomputed row layouts / decode descriptors
- structural projection/kernel descriptors

This slice is done only when:

- scalar execution core no longer depends on `LoadExecutor<E>`
- scalar execution core no longer depends on `ExecutablePlan<E>`
- scalar execution core no longer depends on typed row decode/materialization callbacks
- scalar execution core no longer depends on typed final page reconstruction
- one monomorphic scalar loop owns kernel feeding
- typed code exists only in boundary adapters and terminal decode

The next scalar cut should therefore target the last aggregate/load legacy seams:

- keep typed store resolution at the outer adapter
- promote the boundary-resolved store handle into a more explicit runtime-owned
  storage descriptor so scalar row production stops feeling adapter-owned
- shrink the remaining aggregate/public wrapper surfaces that still accept
  `ExecutablePlan<E>` even though execution beneath them is prepared-stage
  driven
- keep the remaining typed work limited to descriptor construction plus final
  typed page/entity decode

### 6.3.1 Follow-On Storage Roadmap

After scalar row production is fully structural, the next more database-native
storage/runtime optimization should be:

- replace CBOR row payload decode with offset-based row encoding
- avoid field materialization by deserializing the whole entity payload before
  field access
- preserve one stable row layout that runtime kernels can read slot-by-slot

Target direction:

```text
RawRow bytes
  -> row header / slot offsets
  -> direct field decode by offset
  -> structural row/kernel slots
```

This is intentionally not the current `0.59` execution completion criterion.

It is a follow-on optimization after structural scalar ownership is complete.

Why this matters:

- faster field access
- more stable row layout contracts
- better cache behavior during scalar and grouped execution
- less pointer chasing than repeated map/tree decode from one CBOR object

Preferred target form:

- one offset-based row encoding
- stable slot/offset metadata resolved from model/layout descriptors
- direct field reads into `Value` / structural slot payloads

Not preferred:

- nested `Vec<Vec<u8>>` field storage
- ad hoc per-field heap fragments
- reintroducing entity-typed row materialization as the field-access mechanism

This should be treated as a storage-format milestone, not just an executor
refactor, because it changes persisted row layout rather than only runtime
ownership.

### 6.4 SQL Execution

This is the secondary slice.

Target:

- remove `<E>` from SQL execution bodies
- keep parsing and lowering monomorphic
- keep entity resolution at the boundary only

Current hot duplication exists in typed SQL execution/lowering entry chains.

Target shape:

```rust
fn execute_sql_runtime(
    tag: EntityTag,
    prepared: PreparedSqlCommand,
) -> Result<SqlDispatchResult, QueryError>
```

Parsing stays shared.

Lowering must also stay shared where structurally possible.

SQL execution must converge into the same runtime execution descriptors used by
non-SQL paths.

No parallel execution pipeline may exist.

Only the final entity-specific execution leaf may remain typed during
transition, and that leaf must stay limited to boundary concerns rather than
reintroducing entity-wide execution logic.

Important sequencing note:

- SQL execution must not move ahead of scalar load de-monomorphization

If SQL is widened before scalar load is structural, SQL will simply
reintroduce the same typed execution ownership under a different entrypoint.

SQL should become trivial after the scalar runtime contract exists:

- parse/lower once
- build structural scalar execution descriptors
- dispatch through the same scalar runtime

### 6.5 Mutation Execution

This is the tertiary slice.

Target:

- remove `<E>` from mutation orchestration
- preserve typed write semantics
- keep commit and validation behavior unchanged

Representative target:

```rust
fn prepare_row_commit_runtime(
    tag: EntityTag,
    row: RowView,
    model: &'static EntityModel,
) -> Result<RowCommitPlan, InternalError>
```

The same rule applies:

- shared orchestration
- typed work resolved once at the boundary only

Typed leaves are allowed only for:

- decoding
- validation
- final materialization

Typed leaves must not reintroduce entity-wide execution logic under a different
name.

## 7. Performance and Size Tradeoff

The current `0.58` system makes the tradeoff explicit:

- fast execution because kernels are still monomorphized
- larger WASM because kernels are still duplicated

`0.59` intentionally shifts that balance:

- less code duplication
- smaller WASM
- acceptable but carefully measured runtime overhead

This is an explicit architectural trade, not an incidental side effect.

## 8. Allowed and Forbidden Forms

### 8.1 Allowed

- thin typed wrappers that only resolve `E::MODEL`, `E::ENTITY_TAG`, or equivalent metadata
- structural execution loops over `RowView` and `Value`
- structural execution descriptors
- compact enums for narrow behavioral switches
- one-time dispatch before the loop begins

### 8.2 Forbidden

- generic execution loops
- per-row `match EntityTag`
- large enum dispatch over entity variants inside execution loops
- planner changes justified only by execution refactor needs
- trait-object dispatch inside hot row-processing loops
- new duplicated coercion or `match Value::{...}` clusters
- wide `match Value::{...}` branching in hot loops that recreates duplicated
  execution paths
- typed leaves that reintroduce full entity execution logic after structural
  dispatch has already been resolved

## 9. Risks and Mitigations

Primary risks:

- performance regressions if runtime dispatch is chosen poorly
- over-generalization that produces complex control flow and weakens compiler
  optimization
- compatibility shims that accidentally preserve both generic and structural
  execution paths too long

Mitigations:

- benchmark-driven iteration
- resolve dispatch once before hot loops
- isolate hot paths early
- enforce the no-`dyn`-in-hot-loops rule
- remove transitional duplicated paths as soon as the new kernel is accepted

## 10. Migration Strategy

### Phase 1: Grouped Fold Probe

- introduce grouped runtime execution over structural rows
- preserve existing behavior
- measure WASM impact
- measure hot-path performance impact
- status: completed

Decision gate:

- if grouped execution does not reduce size materially, stop and reassess before broader rollout

### Phase 2: Scalar Aggregate Kernel

- remove `Id<E>` and `AggregateOutput<E>` from scalar aggregate kernel state
- keep typed terminal decode at the boundary only
- status: completed

### Phase 3: Scalar Load Execution

- move scalar route materialization and scalar iteration under one structural runtime owner
- remove `LoadExecutor<E>` from scalar execution core
- remove `ExecutablePlan<E>` from scalar execution core
- remove typed row decode/materialization callbacks from scalar runtime core
- move scalar aggregate/public wrapper execution branches onto prepared-stage inputs
- keep typed resolution only at entrypoint and terminal boundaries
- status: in progress

### Phase 4: SQL Execution

- move typed SQL execution under the same runtime-driven model
- keep parser and lowering changes minimal
- do this only after scalar load execution is structural

### Phase 5: Mutation Execution

- convert mutation orchestration and validation entry flow
- keep commit semantics unchanged

### Phase 6: Cleanup

- remove dead generic execution paths
- remove legacy generic execution paths immediately after parity is achieved
- do not maintain dual execution engines beyond the validation window
- collapse transitional compatibility shims
- remeasure and freeze the new baseline

## 11. Success Criteria

0.59 succeeds if:

- execution kernels no longer duplicate per entity in the main hot execution paths
- WASM size drops materially
- no correctness regressions are introduced
- no performance-critical path is replaced with dynamic dispatch in the row loop
- `<E>` is eliminated or nearly eliminated from the execution layer
- scalar runtime no longer depends on typed loop ownership
- scalar runtime no longer depends on typed row materialization ownership

## 12. Release Framing

0.58 established the structural boundary and proved that no further local size wins were worth taking.

0.59 is the explicit execution-engine phase:

- not planner work
- not parser work
- not schema work
- execution de-monomorphization only
