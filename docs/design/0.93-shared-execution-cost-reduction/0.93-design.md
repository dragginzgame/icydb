# 0.93 Shared Execution Cost Reduction

## Thesis

`0.93` should reduce the fixed instruction cost of shared query execution by
tightening the lower prepared-plan boundary that both fluent and SQL now share.

This is a cost-reduction line, not a feature line.

## Post-0.92 Baseline

`0.92` already settled the major cache and ownership questions:

- SQL now has one surface cache plus the shared lower query-plan cache
- fluent goes straight to the shared lower query-plan cache
- `SharedPreparedExecutionPlan` is the only prepared-plan reuse artifact that matters
- SQL no longer owns a separate prepared-plan cache layer

That means `0.93` should not reopen SQL cache ownership or add another reuse
layer. The remaining leverage is the shared execution floor paid by both
frontends.

The live seams that matter now are:

- [`SharedPreparedExecutionPlan`](</home/adam/projects/icydb/crates/icydb-core/src/db/executor/prepared_execution_plan.rs:169>)
- shared cache lookup and plan-key construction in
  [`db/session/query.rs`](</home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs:298>)
- typed wrapper assembly in
  [`compile_query_with_visible_indexes`](</home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs:420>)
  and
  [`planned_query_with_visible_indexes`](</home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs:434>)

## Why This Is The Right Next Move

The engine is coherent enough now that cost work can target one real boundary.

The system already proved:

- one shared lower prepared-plan boundary
- no SQL-specific prepared-plan cache divergence
- measurable repeat-path improvement from deleting redundant setup work

The next useful step is to make the shared execution boundary itself cheaper,
especially for the fluent path that now serves as the production reference.

## In Scope

`0.93` is in scope for:

- shared prepared-plan execution cost reduction
- duplicate lower-boundary pass removal
- lower-boundary key and setup tightening
- typed wrapper and logical-plan handoff cleanup
- grouped and scalar shared-floor reduction

`0.93` is out of scope for:

- new SQL features
- new cache layers
- reintroducing SQL-specific prepared-plan logic
- another broad architecture rewrite outside the shared execution boundary

## Measurement Discipline

`0.93` should start from the existing perf harnesses, not ad-hoc probes.

Primary measurement surfaces:

- [`testing/pocket-ic/tests/fluent_perf_audit.rs`](</home/adam/projects/icydb/testing/pocket-ic/tests/fluent_perf_audit.rs:1>)
- [`testing/pocket-ic/tests/sql_perf_audit.rs`](</home/adam/projects/icydb/testing/pocket-ic/tests/sql_perf_audit.rs:1>)

Representative baseline cohorts should include:

1. repeated scalar fluent load
2. repeated grouped fluent load
3. filtered scalar fluent load
4. grouped fluent load with `HAVING`
5. SQL and fluent equivalents for the same logical query

Record:

- total instructions
- compile attribution
- execution attribution
- shared query-plan cache hit and miss behavior
- scalar versus grouped parity for the same logical floor

The important `0.93` question is not “did one micro-benchmark go down.” It is:

- did the shared floor drop for both fluent and SQL
- did grouped and scalar both improve
- did the change avoid adding new reuse complexity

## Primary Targets

### 1. Remove Duplicate Lower-Boundary Inspection

The shared cache path still inspects the same query shape more than once around:

- predicate preparation
- structural key construction
- plan extraction for typed wrappers

The target is one real pass per phase, not repeated shape inspection just to
feed adjacent helpers.

### 2. Tighten Prepared-Plan Ownership

`SharedPreparedExecutionPlan` should own as much stable execution setup as
possible.

Good candidates:

- anything recomputed on every execution even though it is fixed by the plan
- grouped execution residents that can move fully into the prepared shell
- route/setup descriptors that still behave like a late-bound template

The prepared plan should feel closer to a compiled executable and less like a
partially resolved wrapper.

### 3. Reduce Typed Wrapper Cost

The fluent session boundary still clones the same logical plan into multiple
typed shells:

- `CompiledQuery<E>`
- `PlannedQuery<E>`

That is not a cache problem anymore. It is shared execution-floor overhead.

`0.93` should either:

- make those wrappers thinner, or
- reduce how much logical-plan state has to be cloned into them

without creating another runtime owner.

### 4. Tighten Value And Projection Churn

Look for fixed-cost churn around:

- `Value` creation
- temporary row/value materialization
- projection helpers that rebuild data only to hand it to the next layer

This is especially important on shared scalar paths and on grouped finalization.

### 5. Reduce Grouped Shared-Floor Cost

Grouped execution is still the best place for hidden fixed overhead.

Audit grouped cost around:

- grouped stream setup
- aggregate fold/finalize
- post-aggregate projection
- `HAVING`
- grouped ordering

The key question is whether grouped execution still repeats shaping or
evaluation that can be prepared once.

## Cache Constraint

`0.93` must not add another meaningful runtime reuse layer.

The allowed model remains:

- fluent: shared query-plan cache -> execute
- SQL: compiled SQL cache -> shared query-plan cache -> execute

If a proposed optimization adds another persistent runtime owner or a second
prepared-plan boundary, it is probably the wrong move.

## Success Criteria

`0.93` is successful if it produces a measurable shared-floor reduction across
multiple query families without changing the cache model again.

That should look like:

- visible instruction reduction on fluent repeat cohorts
- visible instruction reduction on SQL equivalents for the same logical plan
- improvement on both scalar and grouped paths
- no new cache-layer complexity
- no reintroduction of SQL-specific prepared-plan divergence

A rough `5%` to `15%` shared-floor reduction is a reasonable target, but the
real bar is clear improvement across representative workloads rather than one
isolated win.

## Suggested Slice Structure

### 0.93.0 Shared Prepared-Plan Tightening

Focus:

- remove duplicated lower-boundary inspection
- reduce typed wrapper overhead
- move more stable setup into the shared prepared plan

### 0.93.1 Grouped And Projection Cost Cleanup

Focus:

- grouped shared-floor reduction
- projection and `Value` churn cleanup

### 0.93.2 Shared-Floor Proof Hardening

Focus:

- fuller fluent/sql parity measurement
- lower-boundary invariant expansion
- final perf proof on the shared execution floor

## Risks

### Cost Work Becomes Another Architecture Pass

This line should optimize the settled shared boundary, not reopen already-solved
cache or SQL ownership questions.

### Micro-Optimizations Hide Larger Duplicate Work

Do not spend the line on tiny helper tweaks if the dominant cost is still
duplicate plan/setup/shaping work.

### SQL-Surface Drift Pulls The Work Back Upward

The shared lower boundary is the reference path now. New SQL-only execution
logic is a regression for this line.

## One-Line Summary

`0.93` should reduce the shared execution floor by tightening the lower
prepared-plan boundary, removing repeated setup work, and trimming fixed scalar
and grouped runtime cost without adding new cache complexity.
