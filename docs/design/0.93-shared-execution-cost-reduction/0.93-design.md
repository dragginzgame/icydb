# 0.93 Shared Execution Cost Reduction

## Thesis

`0.93` should reduce the fixed instruction cost of query execution by eliminating redundant work and tightening the shared prepared-plan execution path.

This is a cost-reduction release, not a surface-expansion release.

## Why This Is The Right Next Move

The system is now in a state where execution cost can be attacked directly:

- there is one shared execution path
- there is no SQL-specific execution divergence
- there are no duplicate first-class SQL cache layers left from the `0.92` cleanup

Recent cache cleanup already shaved a measurable amount of fixed overhead. That is a good signal that there is still structural execution cost left to remove.

The current opportunity is not new query capability. The opportunity is to lower the shared execution floor that every query still pays.

## Preconditions

Before implementation starts, capture a baseline for the current shared execution path.

At minimum, baseline these cases:

1. repeated scalar `SELECT`
2. repeated grouped `SELECT`
3. filtered scalar `SELECT`
4. grouped `SELECT` with `HAVING`
5. SQL and fluent equivalents for the same logical query

For each case, record:

- total local instructions
- compile attribution
- execution/setup attribution
- prepared-plan reuse attribution
- cache-hit versus cache-miss behavior where applicable

This release is explicitly about cost reduction. It should start from measurement, not intuition.

## Scope

`0.93` is in scope for:

- shared execution-path cost reduction
- redundant-pass removal
- prepared-plan tightening
- projection and predicate execution cost cleanup
- grouped execution cost cleanup

`0.93` is out of scope for:

- new SQL features
- new cache layers
- another architecture reshaping pass on already-settled boundaries

## Primary Targets

### 1. Remove Duplicate Tree Walks

Look for repeated expression or predicate traversal across:

- type inference
- normalization
- planning
- predicate compilation
- projection evaluation

The target is one pass per real phase, not two or three partial passes over the same logical tree.

### 2. Tighten Validation And Normalization Boundaries

The current post-`0.91` architecture intentionally separates:

- validation
- normalization
- compilation

That separation should remain conceptually intact. Do not collapse layers just to save a function call.

The real target is narrower:

- remove duplicated checks across validation and normalization
- remove repeated shape inspection that does not change ownership
- keep semantic rejection in validation
- keep structural shaping in normalization

If a combined helper emerges internally, it must still preserve those ownership rules.

### 3. Eliminate Plan-Setup Overhead

Look for repeated per-query work such as:

- rebuilding execution structs
- re-allocating vectors or maps that could be prepared once
- recomputing slots, bindings, or descriptor shapes that already belong in the prepared plan

The prepared plan should own as much stable setup work as possible.

### 4. Tighten Prepared-Plan Execution

`SharedPreparedExecutionPlan` is now the execution boundary that matters.

Execution should minimize:

- re-binding work
- re-resolution work
- runtime branching on already-known plan shape
- allocation caused only by late setup

The prepared plan should feel more like a compiled executable and less like a partially prepared template.

### 5. Reduce `Value` Churn

Look for:

- unnecessary `Value` allocation
- copying where borrowing or slot reuse would work
- boxing or unboxing patterns on hot paths
- temporary row/value materialization that exists only to satisfy an intermediate helper

This is a likely fixed-cost hotspot.

### 6. Tighten The Projection Pipeline

Projection is unified now, so the next question is cost:

- are expressions recomputed unnecessarily
- are full rows built when only a subset is needed
- are Top-K and filtered paths forcing extra materialization

The projection path should not pay for data that the final result never needs.

### 7. Clean Up Grouped Execution Cost

Grouped queries still have the highest chance of hidden fixed overhead.

Audit the grouped path around:

- grouping
- aggregate finalization
- post-aggregate expression evaluation
- `HAVING`
- ordering

Look for:

- repeated evaluation of the same grouped expressions
- materialization before Top-K that can be postponed or avoided
- repeated grouped row shaping when one prepared representation would suffice

### 8. Tighten Predicate Execution Cost

The predicate architecture is now clean enough to optimize safely.

Verify that execution does not reintroduce semantic cost:

- compiled predicates should be minimal
- no normalization logic should survive in execution
- no redundant branching should remain in the hot path

## Cache Layer Constraint

`0.93` must not add another meaningful runtime reuse layer.

The current design should stay sane:

- prepared execution should remain explainable through the existing shared prepared-plan model
- performance work should reduce work inside that model, not create a new peer cache

If a proposed optimization introduces another cache-like boundary, it should be treated as suspect by default.

## Success Criteria

The expected result is a measurable drop in fixed query cost across multiple query families, not a benchmark win on one narrow case.

Success should look like:

- another meaningful reduction in instruction count across repeated query execution
- visible improvement on scalar and grouped paths
- parity between SQL and fluent execution cost for the same logical plan
- no new cache-layer complexity
- no reintroduction of clause-specific or surface-specific execution logic

A rough target of `5%` to `15%` is reasonable, but the real acceptance bar is measured shared-floor reduction across representative workloads.

## Suggested Slice Structure

### 0.93.0 Expression And Predicate Pass Tightening

Focus:

- remove duplicate walks
- remove duplicated validation/normalization inspection
- keep the existing ownership boundaries intact

### 0.93.1 Prepared-Plan Tightening

Focus:

- reduce runtime setup
- move more stable setup work into prepared artifacts
- eliminate redundant execution structs or bindings

### 0.93.2 Execution Micro-Cost Cleanup

Focus:

- `Value` handling
- projection pipeline cost
- grouped execution cost

## Risks To Watch

### Cost Work Accidentally Becomes Architecture Work

This release should not reopen settled ownership boundaries unless a boundary is itself the cost source.

### Micro-Optimizations Hide Real Repeated Work

Do not spend the release on tiny local improvements if duplicate passes or duplicate materialization still dominate the shared floor.

### Benchmark Drift Without Baseline Discipline

If baseline capture is inconsistent, it will be easy to misread noise as progress.

## Key Mindset Shift

`0.92` was about making the system coherent and easy to reason about.

`0.93` should be about making that unified system cheap to execute.

## One-Line Summary

`0.93` should reduce the shared execution floor by removing redundant passes, tightening prepared-plan execution, and trimming fixed runtime cost without adding new surface area or new cache complexity.
