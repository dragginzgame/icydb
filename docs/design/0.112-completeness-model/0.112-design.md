# 0.112 Capability & Completeness Model

## Thesis

`0.112` defines a canonical, repo-owned model for measuring IcyDB's feature
coverage and architectural completeness across the current single-entity SQL
boundary.

The goal is to replace informal progress tracking such as "we support most of
X" with a structured capability matrix that evaluates each feature area across
the full query pipeline:

* surface (parse)
* lowering (semantic representation)
* canonicalization (semantic identity)
* planning (execution strategy)
* execution (runtime correctness)
* explain (diagnostic fidelity)
* proof (test-backed guarantees)

`0.112` does not add new SQL features or execution behavior. It introduces a
measurement model that allows future work to be expressed as movement within a
defined capability space.

---

## Scope

### Included

* definition of feature taxonomy (within current system boundary)
* definition of pipeline stages
* scoring model for completeness
* definition of proof requirements
* baseline capability snapshot of the current system (post-`0.111`)

### Excluded

* new SQL surface features (for example `JOIN`, subqueries, window functions)
* execution changes (planner, routing, cost model)
* semantic widening (grouped or scalar)
* prepared/template redesign
* performance work

---

## Boundary

The capability model is defined for the current single-entity SQL/query
system, which includes:

* `SELECT` (scalar + grouped)
* `WHERE` / `HAVING` predicates
* projection expressions
* aggregates
* `ORDER BY` / `LIMIT` / `OFFSET`
* `DISTINCT`
* mutation (`INSERT` / `UPDATE` / `DELETE`) where applicable
* `EXPLAIN`

Explicitly excluded from this model:

* `JOIN`s
* subqueries (`IN` / `EXISTS` / `ANY` / `ALL`)
* window functions
* multi-entity planning

These remain out of scope and should not be implicitly counted toward coverage.

---

## Core Model

### 1. Feature Taxonomy

Features are grouped into coarse, stable buckets:

#### A. Core Query

* scalar `SELECT`
* grouped `SELECT` (`GROUP BY` + `HAVING`)
* projection expressions
* predicates (`WHERE` / `HAVING`)

#### B. Expression System

* arithmetic expressions
* boolean expressions
* searched `CASE`
* scalar functions (admitted surface)

#### C. Aggregation

* global aggregates
* grouped aggregates
* `DISTINCT`

#### D. Query Modifiers

* `ORDER BY`
* `LIMIT` / `OFFSET`

#### E. Mutation

* `INSERT`
* `UPDATE`
* `DELETE`
* `RETURNING` (if supported)

#### F. Diagnostics

* `EXPLAIN` (all modes)

### 2. Pipeline Stages

Each feature is evaluated across the same pipeline:

`Parse -> Lower -> Canonical -> Plan -> Execute -> Explain -> Proof`

#### Stage definitions

##### Parse

Syntax accepted and parsed into AST.

##### Lower

AST lowered into semantic model (for example `filter_expr`,
`projection_expr`).

##### Canonical

Semantic identity normalized (for example `CASE` canonicalization, boolean
normalization).

##### Plan

Planner produces valid execution strategy (access + residual + projection).

##### Execute

Runtime produces correct results.

##### Explain

Explain reflects actual semantic + execution state (no reconstruction drift).

##### Proof

Behavior is covered by tests on real SQL/fluent surfaces.

### 3. Scoring Model

Each stage is scored per feature:

* `1.0` - complete
* `0.5` - partial / bounded / fail-closed
* `0.0` - not supported
* `N/A` - structurally not applicable

Stages that are not applicable to a feature may be marked as `N/A`.

`N/A` stages are excluded from the feature score calculation:

```text
feature_score = average(stage_scores where stage != N/A)
```

`N/A` should be used only when a stage is structurally not part of the
feature, not when the stage exists but is incomplete.

Feature score:

```text
feature_score = average(stage_scores where stage != N/A)
```

In-scope system completeness score:

```text
in_scope_score = average(feature_scores where feature.in_scope = true)
```

Surface/product completeness score:

```text
surface_score = average(all feature_scores including out_of_scope)
```

Out-of-scope features such as `JOIN`, subqueries, and window functions are
excluded from the in-scope system completeness score. They may be tracked
separately as part of a broader surface completeness score, but do not affect
the reported completeness of the current system boundary.

All stages are equally weighted in `0.112` for simplicity. Future revisions may
introduce weighted scoring.

#### Scoring Guidelines

##### Complete (`1.0`)

* fully supported across the stage
* no known semantic gaps
* consistent across SQL + fluent surfaces
* identity and explain aligned (where applicable)

##### Partial (`0.5`)

* bounded support (for example explicit-`ELSE` `CASE` only)
* fail-closed behavior for unsupported shapes
* known gaps in canonicalization or identity
* limited explain fidelity or test coverage

##### Not Supported (`0.0`)

* feature absent or rejected
* no semantic or execution support

Feature categories are intentionally overlapping and represent different
perspectives on the same execution system. Aggregated scores are heuristic
indicators, not mathematically independent measurements.

---

## Proof Model

A feature/stage can only be scored as `1.0` if:

* it is exercised through real SQL or fluent surfaces
* tests validate:
  * correctness
  * identity (where relevant)
  * explain output (where relevant)

This follows the existing discipline:

> "complete" means present in surface, represented canonically, and directly
> tested on real execution paths.

---

## Normative vs Baseline

The capability model defined in this document is normative.

That includes:

* taxonomy
* pipeline stages
* scoring rules
* proof requirements

The baseline scores provided in this document are provisional and serve as an
initial calibration of the model. They are not yet enforced by CI and may be
refined in subsequent iterations.

Future updates may tighten baseline scores and introduce enforcement once
scoring stabilizes.

## Interpretation

Post-`0.111`, the system shows:

* scalar pipeline: essentially complete (`~95–100%`)
* grouped semantics: strong but still family-scoped (`~80–90%`)
* expression system: strong, with bounded gaps in `CASE` and functions
* execution pipeline: complete for admitted surface
* explain: near-complete and aligned with runtime
* advanced SQL: intentionally out of scope

---

## Intended Use

This model is used to:

* guide prioritization (semantic vs execution vs surface gaps)
* track progress across slices
* prevent accidental scope widening
* communicate system maturity clearly

Future slices should reference this model when claiming improvements, for
example:

> `0.113` increases grouped canonical coverage from `0.8 -> 0.9`

## Future Extensions

Possible follow-ups, not part of `0.112`:

## Summary

`0.112` introduces a structured, enforceable way to answer:

> "How complete is IcyDB?"

Instead of informal answers, the system can now be described in terms of:

* feature coverage
* pipeline completeness
* proof-backed guarantees

This turns architectural progress into something measurable, comparable, and
stable over time.
