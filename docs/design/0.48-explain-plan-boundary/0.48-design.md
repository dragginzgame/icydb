## 1. Objective

Complete the explain-boundary hardening deferred from `0.47` by making
planner-owned contracts the authority for plan identity and continuation
signatures, while keeping `EXPLAIN` as a deterministic presentation surface.

Primary goals:

- decouple plan fingerprint and continuation-signature hashing from
  `query::explain` DTO projection
- keep semantic identity stable while removing explain-path coupling risks
- move continuation envelope/advancement contract ownership to cursor-owned APIs
  consumed by index scanning
- preserve explain/runtime parity behavior and deterministic diagnostics output

### Identity Contract Definition

Plan identity is derived exclusively from planner-owned canonical contracts and
must remain stable across:

- explain projection changes
- DTO or renderer evolution
- diagnostic-only metadata

Identity inputs include:

- canonical logical plan shape
- access path selection
- predicate canonicalization
- grouping and projection semantics

Identity inputs explicitly exclude:

- `EXPLAIN` DTO structures
- renderer formatting
- debug and diagnostic metadata

---

## 2. Non-Goals

- no expansion of public query syntax or new explain node families
- no optimizer-policy redesign outside current planner contracts
- no compatibility layer for legacy hash seeds
- no unrelated subsystem decomposition work

  increasing coupling between observability DTO shape and identity authority

  seams, with part of range-bound reason mapping in `db::access::lowering`

  dependency pressure from hash/signature surfaces to explain DTO assembly

Operating constraints for `0.48`:

- no semantic hash or continuation-signature drift for logically equivalent
  queries
- no regression to cursor continuation safety checks:
  envelope containment and directional advancement remain fail-closed
- no explain output regression for shipped text/json/verbose surfaces

Continuation safety invariants:

- anchor must be contained within the scan envelope
- directional advancement must be monotonic

These guarantees remain cursor-owned and must not be re-derived in index scan
or access-lowering layers.

## 6.1 Explain Small-Feature Track (Patch-Neutral, Subsequent Patches)

Goal: prioritize low-risk, high-observability planner/access improvements that
make `EXPLAIN` materially more useful while preserving the hardened `0.48`
boundary.

Architectural guardrail for every item in this track:

- planner/access/executor decisions -> explain projection
- never explain projection -> planner/access/executor decisions

### Tier 1 - Very Small (`1-2` files, low risk)

1. Covering index indicator

- explain output should expose `covering: true|false` on index-scan nodes when
  covering fast-path eligibility is known
- representative shape:

```text
IndexScan(users_email_idx)
  covering: true
```

- implementation areas:
  - `executor/route`
  - `executor/explain/descriptor`

2. Index prefix length reporting

- explain output should expose matched prefix length for selected index access
  routes
- representative shape:

```text
IndexScan(users_age_city_created_idx)
  prefix_len: 2
```

- implementation areas:
  - `query/plan/access`
  - `executor/explain/descriptor`

3. Limit pushdown reporting

- explain output should expose whether `LIMIT` was pushed into scan fetch bounds
  versus applied as a separate terminal node
- representative shapes:

```text
IndexScan(users_age_idx)
  fetch: 10
```

```text
Limit
  |
  v
Scan
```

- implementation areas:
  - `query/plan/access`
  - `executor/explain/descriptor`

4. Sort elimination reporting

- explain output should expose whether index ordering satisfies the requested
  order
- representative shape:

```text
IndexScan(users_age_idx)
  order_satisfied_by_index: true
```

- implementation areas:
  - `query/plan/access`
  - `executor/explain/descriptor`

### Tier 2 - Small Planner Enhancements

5. Predicate pushdown reporting

- explain output should show pushed predicates versus residual predicates for
  the selected access route
- representative shape:

```text
IndexScan(users_age_idx)
  pushdown: age = 30
  residual: name LIKE 'A%'
```

- implementation areas:
  - `query/plan/predicate`
  - `query/plan/access`
  - `executor/explain/descriptor`

6. Residual filter node

- explain output should show an explicit filter stage whenever residual
  predicates are applied after access
- representative shape:

```text
IndexScan
  |
  v
Filter(name LIKE 'A%')
```

- implementation areas:
  - `executor/explain/descriptor`

7. Cursor resume diagnostics

- explain output should expose continuation anchors and direction for
  paginated/index-continuation flows
- representative shape:

```text
IndexScan(users_id_idx)
  resume_from: id=100
  direction: forward
```

- implementation areas:
  - `cursor/*`
  - `query/explain`

### Tier 3 - Slightly Bigger, Still Safe

8. Fast-path reason codes

- explain output should include explicit reason codes for fast paths (for
  example `index_range`, `projection_subset`)
- representative shapes:

```text
CountFastPath
  reason: index_range
```

```text
CoveringScan
  reason: projection_subset
```

- implementation areas:
  - `executor/route`
  - `executor/explain`

9. Access path ranking (explain-only)

- explain output should show chosen access path and considered alternatives
  without introducing a cost model
- representative shape:

```text
AccessChoice
  chosen: users_age_idx
  alternatives: [users_city_idx]
```

- implementation areas:
  - `query/plan/access`
  - `query/explain`

10. Projection pushdown reporting

- explain output should show projected fields emitted directly from index
  access when applicable
- representative shape:

```text
IndexScan(users_email_idx)
  projected_fields: [email, id]
```

- implementation areas:
  - `query/plan/access`
  - `executor/explain/descriptor`

### Tier 4 - Contained Planner Power Expansions

11. `OR` predicate index union (same-field equality only)

- enable planner expansion for simple disjunctions such as
  `age = 30 OR age = 40` into a bounded union plan
- representative shape:

```text
Union
  |- IndexScan(age=30)
  `- IndexScan(age=40)
```

- implementation areas:
  - `query/plan/predicate`
  - `query/plan/access`
  - `query/explain`

12. `IN` predicate index expansion

- enable planner expansion for `IN` equality sets on one indexed field
- representative shape:

```text
IndexScan(users_age_idx)
  prefix_values: [30, 40, 50]
```

- implementation areas:
  - `query/plan/predicate`
  - `query/plan/access`
  - `query/explain`

Track-level safety constraints:

- explain additions must be deterministic for logically equivalent queries
- explain additions must not feed identity hashing/signature contracts
- parity tests must confirm explain route/decision output still matches runtime
- Tier 4 expansions must start with strict scope guards (same-field equality
  only) and fail closed when outside supported shape
