# IcyDB 0.163 — Stable Allocation Metadata

## Status

Proposed.

## Summary

0.163 should finish the schema-side meaning of stable-memory allocation metadata.

IcyDB already treats stable-memory allocation identity as:

```text
memory_id + stable_key
```

That is the durable ABI. The optional allocation fields:

```text
schema_version
schema_fingerprint
```

exist in the schema model shape but are not currently populated for store
allocations. 0.163 defines what those fields mean, when they are present, and
where they appear in diagnostics.

This is diagnostic/reporting work, not a storage-layout replacement.

## Current State

- `Store::allocation(...)` creates `StableMemoryAllocation` records for data,
  index, and schema roles.
- `Canister::commit_stable_key()` creates the commit-memory key using the same
  stable-key formatter.
- `memory_id` and `stable_key` are active identity fields.
- `schema_version` and `schema_fingerprint` are present on
  `StableMemoryAllocation`, but store allocations currently pass `None`.
- Canic/ic-memory exposes equivalent optional metadata on memory inspection.
- IcyDB accepted schema snapshots already have schema-version and fingerprint
  concepts in the runtime schema/catalog layer.

## Goals

- Define `schema_version` and `schema_fingerprint` for each IcyDB stable
  allocation role.
- Populate allocation metadata where IcyDB has an accepted schema authority.
- Keep allocation identity anchored only on `memory_id` and `stable_key`.
- Make diagnostics clear when an allocation is tied to a specific accepted
  schema shape.
- Ensure schema metadata changes do not imply allocation replacement.
- Keep commit memory metadata explicit and conservative.
- Preserve hard-cut pre-1.0 behavior: no compatibility shims or historical
  fallback reconstruction.

## Non-Goals

- No stable-memory migration format change.
- No change to physical stable-memory IDs.
- No change to stable-key generation.
- No schema-order-derived memory allocation.
- No canister reinstall/upgrade workflow change.
- No primary-key or index-key redesign in this release.
- No requirement that schema fingerprints are bootstrap-fatal allocation
  identity checks.

## Core Invariant

```text
memory_id + stable_key is allocation identity.
schema_version + schema_fingerprint is schema diagnostic metadata.
```

Changing schema metadata must not make an existing stable allocation a different
allocation. Replacing an allocation still requires an intentional stable-key
generation bump such as `.v2`.

## Metadata Semantics

### `schema_version`

`schema_version` identifies the accepted schema version that produced or most
recently reconciled the allocation metadata.

For store role allocations:

- data memory: accepted store/entity layout version
- index memory: accepted index catalog version
- schema memory: accepted schema-store/catalog version

For commit memory:

- either `None`, if commit memory remains schema-neutral
- or the accepted commit-schema version, if the commit schema has a stable
  version authority

The first implementation should prefer `None` for commit memory unless there is
already a single accepted commit schema version boundary.

### `schema_fingerprint`

`schema_fingerprint` is a diagnostic fingerprint of the accepted schema contract
that owns the allocation role.

For store role allocations:

- data memory should use the accepted entity/store row-layout fingerprint
- index memory should use the accepted index catalog fingerprint
- schema memory should use the accepted schema catalog fingerprint

For commit memory:

- use the accepted commit-schema fingerprint only if it already exists as a
  runtime-owned value
- otherwise leave it as `None`

Fingerprints are for diagnostics and drift reporting. They are not allocation
identity and should not be used as a replacement for schema reconciliation.

## Source Of Truth

Runtime allocation metadata must come from accepted schema/catalog state, not
from generated `EntityModel` or `IndexModel` fallback reconstruction.

Allowed sources:

- accepted schema snapshot
- schema store/catalog runtime records
- accepted commit schema fingerprint/version helpers

Disallowed sources:

- iterating generated schema declarations as allocation authority
- deriving metadata from current Rust type order
- reconstructing runtime schema from generated models
- treating missing current schema declarations as retirement or ID reuse

## Allocation Role Model

Every physical IcyDB allocation should continue to expose:

```rust
pub struct StableMemoryAllocation {
    memory_id: u8,
    stable_key: String,
    schema_version: Option<u32>,
    schema_fingerprint: Option<String>,
}
```

Store allocations should be role-specific:

```text
icydb.<memory_namespace>.<store_name>.data.v1
icydb.<memory_namespace>.<store_name>.index.v1
icydb.<memory_namespace>.<store_name>.schema.v1
```

Commit allocation should stay explicit:

```text
icydb.<memory_namespace>.commit.control.v1
```

## Diagnostics Surface

0.163 should expose allocation metadata through existing diagnostics where
natural:

- CLI/schema inspection
- snapshot output
- schema check output
- memory/allocation debug output
- tests around generated canister metadata, if already available

The output should distinguish:

```text
identity: memory_id, stable_key
schema metadata: schema_version, schema_fingerprint
```

Avoid implying that a fingerprint mismatch means the memory was opened under the
wrong key. A mismatch means the allocation metadata and accepted schema need
schema reconciliation/diagnostics, not automatic memory replacement.

## Validation

Add tests proving:

- store allocation identity remains `memory_id + stable_key`
- schema metadata changes do not alter allocation identity
- data/index/schema allocation roles can carry distinct schema metadata
- commit allocation remains explicitly keyed
- generated commit stable key still matches schema-side formatter
- metadata is populated from accepted schema/catalog state, not generated-model
  fallback state
- schema inspection surfaces metadata without treating it as allocation identity

If the first slice cannot safely populate all role metadata, narrow the release
claim and keep missing fields as `None` with tests documenting that behavior.

## Implementation Plan

1. Audit current `StableMemoryAllocation` constructors and consumers.
2. Identify accepted schema version/fingerprint sources for data, index, schema,
   and commit roles.
3. Add explicit allocation metadata builders that accept identity and metadata
   separately.
4. Populate store allocation metadata where accepted schema authority exists.
5. Keep commit metadata `None` unless a commit-schema authority is already
   available.
6. Extend diagnostics and CLI output only where existing surfaces already expose
   allocation information.
7. Add focused tests for identity-vs-metadata behavior.

## Open Questions

- Should schema memory use the whole accepted schema fingerprint or only the
  schema-store catalog fingerprint?
- Does commit memory have a stable schema version, or should it remain
  allocation-only metadata for now?
- Should CLI output hide `None` metadata fields by default or show them
  explicitly for debugging?

## Recommended Changelog Wording

```md
- Defined stable-memory allocation schema metadata semantics and separated
  durable allocation identity (`memory_id` + `stable_key`) from diagnostic
  schema metadata (`schema_version` + `schema_fingerprint`).
- Populated allocation metadata from accepted schema/catalog authority where
  available and surfaced it in diagnostics without treating fingerprint changes
  as memory replacement.
```
