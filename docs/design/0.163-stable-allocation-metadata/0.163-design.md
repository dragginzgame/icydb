# IcyDB 0.163 — Stable Allocation Metadata

## Status

Implemented for storage snapshot diagnostics.

## Summary

0.163 defines the schema-side meaning of stable-memory allocation metadata.

IcyDB already treats stable-memory allocation identity as:

```text
memory_id + stable_key
```

That is the durable ABI. The optional allocation fields:

```text
schema_version
schema_fingerprint
```

exist in the schema model shape and storage snapshot diagnostics. 0.163 defines
what those fields mean, when they are present, and where they appear in
diagnostics.

This is diagnostic/reporting work, not a storage-layout replacement.

## Current State

- `Store::allocation(...)` creates `StableMemoryAllocation` records for data,
  index, and schema roles.
- `Canister::commit_stable_key()` creates the commit-memory key using the same
  stable-key formatter.
- `memory_id` and `stable_key` are active identity fields.
- `schema_version` and `schema_fingerprint` are present on
  `StableMemoryAllocation`; generated store allocation declarations still use
  absent metadata, while runtime storage snapshots derive accepted metadata
  from the schema store.
- Canic/ic-memory exposes equivalent optional metadata on memory inspection.
- IcyDB accepted schema snapshots already have schema-version and fingerprint
  concepts in the runtime schema/catalog layer.

## Goals

- Define `schema_version` and `schema_fingerprint` for each IcyDB stable
  allocation role.
- Populate allocation metadata where IcyDB has an accepted schema authority.
- Keep allocation identity anchored only on `memory_id` and `stable_key`.
- Make diagnostics clear when an allocation is tied to a specific accepted
  schema shape.
- Ensure schema metadata changes do not imply allocation replacement.
- Keep commit memory metadata explicit and conservative.
- Isolate generated-model schema fallback paths so allocation metadata cannot
  be populated from generated `SchemaInfo` or model-only reconstruction.
- Preserve hard-cut pre-1.0 behavior: no compatibility shims or historical
  fallback reconstruction.

## Non-Goals

- No stable-memory migration format change.
- No change to physical stable-memory IDs.
- No change to stable-key generation.
- No schema-order-derived memory allocation.
- No canister reinstall/upgrade workflow change.
- No primary-key or index-key redesign in this release.
- No requirement that schema fingerprints are bootstrap-fatal allocation
  identity checks.

## Core Invariant

```text
memory_id + stable_key is allocation identity.
schema_version + schema_fingerprint is schema diagnostic metadata.
```

Changing schema metadata must not make an existing stable allocation a different
allocation. Replacing an allocation still requires an intentional stable-key
generation bump such as `.v2`.

## Metadata Semantics

### `schema_version`

`schema_version` identifies the accepted schema version that produced or most
recently reconciled the allocation metadata.

For store role allocations:

- data memory: accepted store/entity layout version
- index memory: accepted index catalog version
- schema memory: accepted schema-store/catalog version

For commit memory:

- `None`; 0.163 does not introduce an accepted commit-allocation schema
  version authority

0.163 keeps commit allocation metadata as `None`. Commit memory remains
explicitly keyed by `COMMIT_MEMORY_ID` and `COMMIT_STABLE_KEY`; no accepted
commit-allocation schema version is introduced.

### `schema_fingerprint`

`schema_fingerprint` is a diagnostic fingerprint of the accepted schema contract
that owns the allocation role.

For store role allocations:

- data memory uses the accepted entity/store row-layout fingerprint
- index memory uses the accepted index catalog fingerprint
- schema memory uses the accepted schema catalog fingerprint

For commit memory:

- `None`; commit-memory schema fingerprints are not allocation metadata in
  0.163

Fingerprints are for diagnostics and drift reporting. They are not allocation
identity and should not be used as a replacement for schema reconciliation.

## Source Of Truth

Runtime allocation metadata must come from accepted schema/catalog state, not
from generated `EntityModel` or `IndexModel` fallback reconstruction.

If accepted authority is unavailable for an allocation role, metadata must be
`None`; do not reconstruct it from generated models or stale snapshots.

Allowed sources:

- accepted schema snapshot
- schema store/catalog runtime records
- accepted commit schema fingerprint/version helpers, if a future release adds
  a commit-allocation schema authority

Disallowed sources:

- iterating generated schema declarations as allocation authority
- deriving metadata from current Rust type order
- reconstructing runtime schema from generated models
- treating missing current schema declarations as retirement or ID reuse

## Allocation Role Model

Every physical IcyDB allocation should continue to expose:

```rust
pub struct StableMemoryAllocationMetadata {
    schema_version: Option<u32>,
    schema_fingerprint: Option<String>,
}

pub struct StableMemoryAllocation {
    memory_id: u8,
    stable_key: String,
    schema_metadata: StableMemoryAllocationMetadata,
}
```

The schema node model keeps identity and metadata construction separate. Static
generated store allocation declarations still default to absent metadata, and
runtime storage snapshots decorate data/index/schema rows from accepted
schema-store authority. Schema metadata construction is intentionally split
between absent metadata and `from_accepted_schema_contract(...)`; there is no
generated-model metadata constructor.

Schema-store diagnostics now derive catalog metadata from the latest persisted
schema snapshots in the schema store. Empty schema stores report absent
metadata instead of reconstructing it from generated models.

Store allocations should be role-specific:

```text
icydb.<memory_namespace>.<store_name>.data.v1
icydb.<memory_namespace>.<store_name>.index.v1
icydb.<memory_namespace>.<store_name>.schema.v1
```

Commit allocation should stay explicit:

```text
icydb.<memory_namespace>.commit.control.v1
```

## Diagnostics Surface

0.163 exposes allocation metadata through the storage snapshot diagnostics and
the CLI snapshot renderer:

- snapshot output
- CLI snapshot tables

The output should distinguish:

```text
identity: memory_id, stable_key
schema metadata: schema_version, schema_fingerprint
```

Avoid implying that a fingerprint mismatch means the memory was opened under the
wrong key. A mismatch means the allocation metadata and accepted schema need
schema reconciliation/diagnostics, not automatic memory replacement.

The storage snapshot now includes `schema_storage` rows with:

```text
path
memory_id
stable_key
schema_version
schema_fingerprint
entity_count
```

These rows are schema-store catalog diagnostics. They are not data/index memory
identity checks and they do not change stable-memory opening behavior.

Generated store registry wiring now also carries durable allocation identity
for data, index, and schema memories into storage snapshots. Snapshot identity
fields are diagnostic projections of generated stable-memory declarations; they
do not change memory opening, stable-key generation, or schema reconciliation.

Data-store, index-store, and schema-store snapshot rows now project
role-specific metadata from the same accepted schema-store authority:

- data rows hash the accepted entity row-layout and field contracts with
  accepted index contracts removed
- index rows hash the accepted index catalog contracts
- schema rows hash the full accepted schema-store catalog payloads

All three projections are derived from the latest persisted schema snapshots;
none are reconstructed from generated models. Empty schema stores or failed
metadata derivation leave data/index/schema metadata absent.

## Validation

Implemented tests prove:

- store allocation identity remains `memory_id + stable_key`
- schema metadata changes do not alter allocation identity
- data/index/schema allocation roles can carry distinct schema metadata
- commit allocation remains explicitly keyed
- generated commit stable key still matches schema-side formatter
- metadata is populated from accepted schema/catalog state, not generated-model
  fallback state
- generated-model fallback `SchemaInfo` cannot populate allocation
  `schema_version` or `schema_fingerprint`
- allocation metadata builders require accepted schema/catalog authority, not
  generated model metadata
- storage snapshot diagnostics surface metadata without treating it as
  allocation identity

Metadata remains absent when accepted schema-store authority is unavailable.

## Implementation Summary

1. `StableMemoryAllocation` constructors keep identity and metadata separate.
2. Generated allocation declarations continue to provide `memory_id` and
   `stable_key` identity.
3. Storage snapshots read latest persisted schema snapshots from the schema
   store and derive role-specific data/index/schema metadata.
4. Empty schema stores or metadata derivation failures leave metadata absent.
5. Commit allocation metadata remains `None`; commit memory stays explicitly
   keyed.
6. CLI snapshot tables render identity (`mem`, `stable key`) separately from
   metadata (`version`, `fingerprint`).

## Resolved Questions

- Schema memory uses the full accepted schema-store catalog fingerprint.
- Commit memory remains allocation-only metadata for 0.163.
- CLI snapshot output shows absent metadata as `-`, matching existing table
  rendering style.
