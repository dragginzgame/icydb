## Title

**0.70 — Centralized Read Authority Classification**

---

## Status

Proposed

---

## Context

The `0.69` line introduced **true covering read execution** and established that certain SQL query shapes can execute correctly under staleness **without per-row existence probes**, when backed by appropriate authority guarantees (e.g. `storage_existence_witness`).

Subsequent iterations admitted multiple **specific cohorts** where:

* query semantics remained identical to the fail-closed baseline
* probe and encoding overhead dropped to zero
* execution became measurably cheaper

However, admission of these cohorts is currently:

* **implicit**
* **distributed across multiple modules** (`scan`, `covering`, `terminal`)
* **not centrally defined**

This makes it difficult to:

* predict whether a change applies broadly or narrowly
* reason about correctness boundaries
* extend the system safely

---

## Problem

Authority decisions (probe vs probe-free execution) are currently:

* encoded indirectly through execution paths
* duplicated across multiple layers
* derived from mechanism rather than principle

As a result:

* behavior appears empirical (“this cohort worked”)
* correctness boundaries are hard to audit
* future extensions risk unsound widening

---

## Key Observation (from 0.69)

Empirical results across admitted and rejected cohorts reveal a unifying invariant:

> Some query shapes are **stable under deletion of matching rows**, meaning that missing rows do not change the final TOP-N result.

For these shapes:

* row-existence probes are unnecessary
* covering-membership or storage witness is sufficient

For other shapes:

* missing rows can change the output window
* probes remain required

---

## Core Invariant

A query may execute without row-existence probes iff:

> **Removing any matching row does not change the final TOP-N result produced by the query.**

This property is referred to as:

> **window stability under deletion**

---

## Proposal

Introduce a **centralized authority classifier** that determines whether a query may execute without probes.

### API

```rust
enum AuthorityDecision {
    ProbeRequired,
    CoveringWitnessAllowed,
}

fn authority_for_query(q: &LoweredQuery) -> AuthorityDecision
```

---

## Design

### 1. Centralization

* `authority_for_query` becomes the **single pivot point** for read authority decisions
* all execution paths must respect this classification

### 2. Placement

Introduce a new module:

```text
crates/icydb-core/src/db/executor/authority.rs
```

This module must:

* contain no execution logic
* depend only on lowered query structure
* remain free of storage or scan details

---

### 3. Initial Strategy (Conservative)

The classifier will begin as a **whitelist of proven-safe cohorts**.

#### Admitted cohorts

* equality-prefix + suffix-order + PK tie-break + LIMIT (ASC)
* equality-prefix + suffix-order + PK tie-break + LIMIT (DESC)
* order-only composite cohorts (ASC)
* order-only composite cohorts (DESC)

These cohorts have demonstrated:

* identical result sets under staleness
* probe count reduction to zero
* no window-shift anomalies

#### Rejected / excluded cohorts

* leading-component equality-prefix sibling (window instability observed)
* queries without LIMIT
* queries without total deterministic ordering
* queries with OFFSET / cursor continuation
* multi-predicate cohorts not yet validated

All non-listed shapes default to:

```rust
ProbeRequired
```

---

### 4. Routing Integration

At the routing boundary (e.g. `route/terminal.rs`):

```rust
let authority = authority_for_query(&query);

match authority {
    CoveringWitnessAllowed => {
        // force probe-free covering path
    }
    ProbeRequired => {
        // allow existing probe-based execution
    }
}
```

---

### 5. Rollout Plan

#### Phase 1 — Shadow mode

* call classifier
* log / assert against known cohorts
* no behavior changes

#### Phase 2 — Positive enforcement

* enforce `CoveringWitnessAllowed`
* do not override probe-required paths

#### Phase 3 — Consolidation

* remove duplicated authority logic from:

  * scan
  * covering
  * terminal routing

---

## Non-Goals

This proposal explicitly does NOT:

* redesign the execution engine
* unify scan or covering implementations
* generalize to all SQL shapes
* support OFFSET or cursor-based continuation
* expand grouped or aggregate execution

---

## Testing Strategy

### 1. Cohort assertions

For each admitted cohort:

```rust
assert_eq!(
    authority_for_query(query),
    CoveringWitnessAllowed
);
```

For rejected cohorts:

```rust
assert_eq!(
    authority_for_query(query),
    ProbeRequired
);
```

---

### 2. Parity validation

For all admitted cohorts:

* verify identical results vs fail-closed baseline
* verify probe counts drop to zero
* verify no “window consumption” anomalies

---

### 3. Regression safety

* classifier must be **conservative**
* incorrect allowance is a correctness bug
* over-rejection is acceptable

---

## Expected Impact

### Immediate

* explicit and auditable authority decisions
* reduced cognitive load when evaluating changes
* elimination of “why did this work?” ambiguity

### Medium-term

* simpler extension of new cohorts
* reduced duplication across execution paths
* clearer separation between:

  * **query shape reasoning**
  * **execution mechanics**

---

## Relationship to 0.69

0.69 established:

> “probe-free execution is possible for some covering read shapes”

0.70 formalizes:

> **“which shapes qualify, and why”**

This is a continuation, not a divergence.

---

## Future Work

* generalize classifier from whitelist → rule-based
* extend to additional validated cohorts
* integrate grouped and aggregate lanes where applicable
* unify authority reasoning across read execution paths

---

## Summary

0.70 introduces a **single, centralized authority classifier** based on:

> **window stability under deletion**

This replaces distributed, implicit decisions with a clear and conservative pivot, enabling safe and predictable evolution of probe-free read execution.
