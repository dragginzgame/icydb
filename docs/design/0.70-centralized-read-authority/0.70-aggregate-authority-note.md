# 0.70 Aggregate Shortcut Visibility Note

## Purpose

This note records the revised `0.70` aggregate design after the decision to
collapse index correctness into planner visibility.

The important rule is now simple:

* aggregate index-backed shortcuts do not get their own execution-time
  correctness model

If an index is visible to the planner, execution treats it as complete and
correct.

---

## Current Aggregate Shortcut Inventory

The current route-owned aggregate shortcut entry points are still:

* `crates/icydb-core/src/db/executor/route/terminal.rs`
  * `derive_count_terminal_fast_path_contract_for_model(...)`
  * `derive_exists_terminal_fast_path_contract_for_model(...)`

Those lower into prepared scalar terminal strategies in:

* `crates/icydb-core/src/db/executor/aggregate/terminals.rs`
  * `prepare_scalar_count_terminal_strategy(...)`
  * `prepare_scalar_exists_terminal_strategy(...)`

The current prepared scalar terminal vocabulary remains:

* `CountPrimaryKeyCardinality`
* `CountExistingRows { direction, covering }`
* `ExistsExistingRows { direction }`
* `KernelAggregate`

Field-target extrema still remain on the separate projection lane.

---

## Revised Aggregate Rule

Aggregate shortcuts must follow the same planner visibility contract as the
rest of the system.

That means:

* planner may use only visible indexes
* visible indexes are assumed correct and complete
* execution does not perform a second correctness decision for aggregate
  shortcuts

This applies directly to the existing-row shortcut family:

* `CountExistingRows { direction, covering }`
* `ExistsExistingRows { direction }`

Those shortcuts are either:

* planned against a visible index, in which case execution may trust them, or
* not planned, in which case the planner must choose the fallback path

---

## What This Does Not Mean

This change does **not** mean every aggregate optimization question is solved.

It does **not** answer:

* which aggregate shortcuts the planner should admit
* how grouped aggregates should optimize
* how extrema paths should be simplified
* whether additional aggregate paths deserve optimization later

Those remain planner and strategy questions.

The important change is narrower:

* index correctness is no longer one of those runtime questions

---

## COUNT And EXISTS Existing-Row Shortcuts

`COUNT` and `EXISTS` existing-row shortcuts are now described this way:

* they rely on planner visibility
* they do not require runtime correctness resolution
* they are correct when planned against a visible index

Fallback behavior is therefore planner-owned.

Execution should not decide:

* whether the index is trustworthy
* whether a runtime correctness snapshot is needed
* whether a shortcut must be downgraded because of index lifecycle state

Those are planner visibility decisions.

---

## COUNT Primary-Key Cardinality

`CountPrimaryKeyCardinality` remains a distinct optimization path, but it
follows the same high-level rule:

* correctness comes from planner-visible storage contracts
* not from execution-time correctness branching

So it also stays outside any separate runtime "aggregate authority" model.

---

## MIN/MAX And Other Aggregate Paths

`MIN/MAX`, `SUM/AVG`, grouped aggregates, and distinct aggregate routing still
have their own planning and strategy questions.

But under the revised design, those questions should be framed as:

* planner admissibility
* strategy selection
* execution mechanics

not:

* runtime index correctness resolution

---

## Explain And Debug

Aggregate `EXPLAIN` should eventually surface planner choices, not runtime
correctness decisions.

That means:

* which shortcut was selected
* which fallback was selected when no visible index qualified
* which visible access path the planner trusted

It should not grow a separate execution-time correctness vocabulary.

---

## Removed Designs

The earlier aggregate-specific runtime authority exploration has been removed.

That includes:

* aggregate correctness snapshots
* aggregate correctness resolvers
* execution-time aggregate correctness decisions

Those are no longer part of the target `0.70` design.

The removed snapshot note remains only as a historical record:

* [0.70-aggregate-existing-rows-authority-snapshot.md](/home/adam/projects/icydb/docs/design/0.70-centralized-read-authority/0.70-aggregate-existing-rows-authority-snapshot.md)

---

## Current Stop Point

The aggregate design boundary is now:

* planner visibility owns index correctness
* execution trusts planned aggregate index access
* aggregate optimization remains a planner/strategy problem, not a runtime
  correctness problem

That is the correct place to stop before any further aggregate implementation
work.
