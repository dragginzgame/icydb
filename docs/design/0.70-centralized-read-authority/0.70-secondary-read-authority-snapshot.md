## Title

**0.70 — Secondary Read Authority Snapshot Boundary**

---

## Status

In Progress

---

## Purpose

This note freezes the new `0.70.x` snapshot seam for store-backed secondary
read authority.

The goal of this seam is narrow:

* keep lifecycle and synchronized-witness truth at the registry/store boundary
* let executor authority resolution consume one immutable input
* prevent `authority/read.rs` from drifting back toward live store lookups

This is a boundary note, not a policy expansion note.

---

## Scope

`0.70.x` snapshot scope is **secondary-read authority only**.

It currently exists to support:

* store-backed secondary covering route promotion
* store-backed secondary `EXPLAIN EXECUTION` authority labeling
* resolved-profile derivation for the current secondary read authority line

It is explicitly **not** a general executor snapshot abstraction yet.

---

## Snapshot Contract

The current immutable input is:

* `SecondaryReadAuthoritySnapshot`

It currently carries exactly:

* `index_state`
* `secondary_covering_authoritative`
* `secondary_existence_witness_authoritative`

### Guarantees

The snapshot guarantees:

* it is immutable for the duration of one authority resolution pass
* it is complete for the current secondary-read authority decision
* it is sourced only from registry/store truth
* authority resolution does not need to re-check the live store after snapshot
  creation

### Non-Goals

The snapshot does **not**:

* carry planner input
* perform cost modeling
* permit mutation
* represent cross-query state
* generalize grouped/aggregate authority
* replace the richer resolved profile

---

## Resolver Assumptions

`authority/read.rs` is allowed to assume:

* the snapshot is internally consistent
* lifecycle truth has already been gathered at the registry boundary
* synchronized witness bits in the snapshot are the only store-side authority
  inputs needed for the current secondary-read line

`authority/read.rs` is **not** allowed to:

* inspect `StoreHandle`
* borrow the registry directly
* re-derive lifecycle state heuristically
* re-check store truth after snapshot creation

The ordering stays strict:

1. registry/store builds snapshot
2. authority resolver consumes snapshot plus structural plan inputs
3. resolved profile derives behavior
4. flat labels project from the resolved profile only when admitted

---

## Why This Boundary Exists

Without this seam, `authority/read.rs` remains partly coupled to live store
lookups, which makes it harder to reason about:

* what authority truth is actually consumed
* whether the resolver is deterministic for a given input
* where future policy changes belong

The snapshot makes the authority boundary explicit without changing current
query semantics.

---

## Stop Point

The intended stop point for this slice is:

* snapshot stays secondary-read-specific
* no planner migration
* no aggregate adoption
* no classifier widening

Any later widening should happen only after this seam has had time to settle
as a stable executor boundary.
