# IcyDB 0.52 Design

## Status

- Planned

## Target

- `0.52` minor line

## Theme

- Reduced SQL surface and developer query interface

## Intent

`0.52` introduces a minimal, stable textual query surface and developer query
builder on top of the stabilized engine contracts defined in `0.51`.

`0.52` adds no new query semantics. It only exposes already-supported engine
capabilities through stable textual and fluent frontends suitable for:

- developer usage
- CLI tools
- testing
- scripting
- debugging
- EXPLAIN workflows

This line prepares the system for `1.0` usability without expanding the
underlying execution engine. The SQL surface remains intentionally minimal and
deterministic.

## Canonical Query Representation

SQL parsing and fluent builder construction are frontend entrypoints only.

Canonical authority is the normalized query intent / planner input model.

- SQL -> normalized query intent
- fluent builder -> normalized query intent

Validation happens after normalization, not in frontend-specific semantic paths.
Fingerprinting, EXPLAIN, executable-plan generation, and cursor compatibility
operate on normalized intent, not SQL text shape or builder call shape.

Frontends must not introduce divergent semantics.

## Frontend Parity Invariants

- Equivalent SQL and fluent queries must lower to identical normalized intent.
- Equivalent normalized intent must produce identical fingerprinting, executable
  plan construction, cursor contracts, and EXPLAIN output.
- Validation outcomes must be frontend-independent for equivalent semantics.
- Frontends may differ in syntax-level parsing errors, but not in semantic
  interpretation once normalized.

## Objectives

Primary objectives for `0.52`:

- introduce a reduced SQL query surface
- finalize the fluent query builder API
- provide a stable CLI query interface
- expose EXPLAIN tooling
- maintain deterministic execution guarantees from `0.51`

This line does not introduce new engine semantics.

## Supported Query Surface

The SQL surface is intentionally restricted to match existing engine
capabilities.

Supported statements:

- `SELECT`
- `DELETE`
- `EXPLAIN`

Supported `SELECT` constructs:

- `SELECT *`
- `SELECT field1, field2, ...`
- `SELECT DISTINCT ...` (only where current planner validation permits it)
- aggregate projection items in supported forms:
  - `COUNT(*)`
  - `COUNT(field)`
  - `SUM(field)`
  - `AVG(field)`
  - `MIN(field)`
  - `MAX(field)`
- `WHERE`
- `IN`
- `BETWEEN`
- `IS NULL`
- `IS NOT NULL`
- `AND / OR / NOT`
- `GROUP BY` (within existing planner validation constraints)
- `ORDER BY`
- `LIMIT`
- `OFFSET` for `SELECT` only (policy-constrained by existing
  planner/executor rules)

Supported `DELETE` constructs:

- `DELETE FROM ...`
- `WHERE`
- `ORDER BY`
- `LIMIT`

`DELETE` constraints in this line:

- `DELETE ... OFFSET ...` is out of scope.
- existing `DELETE` validation/policy rules remain authoritative.

Pagination through continuation tokens remains supported via the existing cursor
system.

Aggregate semantics note:

- aggregate semantics, including null behavior and grouped legality, remain
  exactly those of current planner/runtime contracts.
- SQL support does not widen existing grouped/distinct legality.

## Supported Expression Subset

The reduced SQL parser does not accept general SQL expression grammars.

Supported projection expression subset:

- direct field references
- `*`
- supported aggregate calls listed in this document

Out of scope for `0.52` expression parsing:

- arithmetic/computed projection expressions
- arbitrary scalar function-call expressions
- nested expression trees in projection lists
- mixed unsupported scalar-expression + aggregate projection shapes
- aliasing (`AS`) for columns or expressions

## Explicitly Unsupported SQL

The SQL surface intentionally excludes complex features.

Not included:

- `JOIN`
- subqueries
- window functions
- `HAVING`
- complex SQL expressions
- column/expression aliases
- DDL commands
- schema mutation through SQL
- `INSERT` SQL syntax
- `UPDATE` SQL syntax
- quoted identifiers
- SQL function namespace beyond the listed aggregate forms
- multi-statement SQL input

These features may be introduced after `1.0`.

## Existing Validation Constraints Remain Authoritative

`0.52` does not relax existing planner validation or execution policy gates.

This includes current constraints around:

- grouped query shape compatibility
- grouped `DISTINCT` / `ORDER BY` policy gates
- ordering requirements for deterministic pagination and constrained deletes
- existing cursor and continuation compatibility validation

Unsupported combinations must fail validation rather than introduce
frontend-specific behavior.

## Slice A - Reduced SQL Parser

### Goal

Introduce a minimal deterministic SQL parser that maps SQL queries to existing
planner intent structures.

### Requirements

The parser must:

- produce the same canonical query identity as the fluent builder
- enforce the same validation rules
- map directly to existing planner structures
- normalize deterministically and formatting-insensitively

Parser behavior must remain fully deterministic.

### Normalization Rules

For `0.52`, normalization rules are:

- SQL keywords are case-insensitive.
- insignificant whitespace is ignored.
- statement terminator handling is canonicalized consistently.
- equivalent supported query forms lower identically when already supported by
  engine intent normalization.

Parser boundary rule:

- parsing/lowering must not perform optimizer-like semantic rewrites beyond
  normalization required to construct canonical intent.
- multi-statement parsing/execution is out of scope in this line.

### Supported Syntax Examples

```sql
SELECT name FROM users WHERE age > 30 ORDER BY age LIMIT 10;
SELECT city, COUNT(*) FROM users GROUP BY city;
```

### Exit Criteria

- SQL queries map deterministically to query intent
- SQL and fluent builder produce identical executable plans

## Slice B - Fluent Query Builder Completion

Finalize the fluent query builder interface.

Builder style in this line is immutable chained composition.

Example usage:

```rust
Query::select(["name", "age"])
    .where(field("age").gt(30))
    .order_by("age")
    .limit(10);
```

Builder goals:

- deterministic lowering to planner intent
- ergonomic developer usage
- identical behavior to SQL queries
- same validation boundary as SQL for unsupported combinations

Exit criteria:

- fluent API fully mirrors supported SQL surface
- builder output identical to SQL parser output

## Slice C - CLI Query Interface

Introduce a basic CLI for query execution.

Example usage:

```bash
icydb query "SELECT name FROM users WHERE age > 30 LIMIT 10"
```

CLI capabilities:

- run queries
- print results
- run EXPLAIN
- show execution diagnostics already exposed by current engine surfaces

This CLI is primarily a developer tool, not a full production client.
It does not introduce a new tracing subsystem in `0.52`.

CLI output contract in `0.52`:

- human-readable output is developer-facing and may evolve
- machine-stable output, when provided, must use explicit structured modes
  (for example JSON forms)

## Slice D - Explain Tooling

Extend the EXPLAIN interface to work with the new SQL surface.

Supported modes:

- `EXPLAIN`
- `EXPLAIN EXECUTION`
- `EXPLAIN JSON`

Guarantee:

- identical normalized intent produces the same canonical explain model

Explain model rule:

- one canonical explain model with multiple deterministic renderers/modes.
- each renderer deterministically renders that canonical model for its mode.

## Slice E - Query Error Surface

Standardize the user-facing error model.

Error categories:

- syntax errors
- query validation errors
- execution errors
- cursor errors

Errors must map to existing internal error types.

Error handling rules:

- error categorization must remain deterministic
- frontend errors must not obscure underlying planner/runtime cause classes
- equivalent invalid semantics from SQL and fluent entrypoints must map to the
  same validation category

## Non-Goals

`0.52` explicitly avoids expanding engine semantics.

Not included:

- joins
- subqueries
- window functions
- query optimizer heuristics
- new storage features
- schema migration tools

The goal is usability, not capability expansion.

## Success Criteria

`0.52` is complete when:

- SQL queries map deterministically to normalized existing planner structures
- fluent query builder is complete
- CLI query interface works
- EXPLAIN tooling supports SQL queries
- execution behavior remains identical to `0.51`

## Expected Outcome

After `0.52`:

- developers can query IcyDB using SQL or fluent APIs
- CLI tools can execute queries and explain plans
- the engine remains fully deterministic
- the system becomes ready for final `1.0` hardening

## Planned Roadmap

- `0.50` - executor simplification
- `0.51` - engine contract stabilization
- `0.52` - developer query surface
- `0.53` - final hardening
- `1.0` - stable release
