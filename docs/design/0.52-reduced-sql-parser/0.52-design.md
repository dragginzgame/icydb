IcyDB 0.52 Design
Status

Planned

Target

0.52 minor line

Theme

Reduced SQL surface and developer query interface.

Intent

0.52 introduces a minimal, stable textual query surface and developer query builder on top of the stabilized engine contracts defined in 0.51.

The goal of this line is not to expand engine capabilities, but to expose the existing engine features through a simple, deterministic interface suitable for:

developer usage

CLI tools

testing

scripting

debugging

explain workflows

This line prepares the system for 1.0 usability without expanding the underlying execution engine.

The SQL surface remains intentionally minimal and deterministic.

Objectives

Primary objectives for 0.52:

introduce a reduced SQL query surface

finalize the fluent query builder API

provide a stable CLI query interface

expose EXPLAIN tooling

maintain deterministic execution guarantees from 0.51

This line does not introduce new engine semantics.

Supported Query Surface

The SQL surface is intentionally restricted to match the existing engine capabilities.

Supported statements:

SELECT
DELETE
EXPLAIN

Supported query constructs:

SELECT *

SELECT field1, field2

SELECT expression

WHERE

IN

BETWEEN

IS NULL

IS NOT NULL

AND / OR / NOT

GROUP BY

COUNT

SUM

AVG

MIN

MAX

ORDER BY

LIMIT

OFFSET

Pagination through continuation tokens remains supported via the existing cursor system.

Explicitly Unsupported SQL

The SQL surface intentionally excludes complex features.

Not included:

JOIN

subqueries

window functions

HAVING

complex SQL expressions

DDL commands

schema mutation through SQL

These features may be introduced after 1.0.

Slice A — Reduced SQL Parser
Goal

Introduce a minimal deterministic SQL parser that maps SQL queries to the existing planner intent structures.

Requirements

The parser must:

produce the same canonical query identity as the fluent builder

enforce the same validation rules

map directly to existing planner structures

Parser behavior must remain fully deterministic.

Supported syntax

Examples:

SELECT name FROM users WHERE age > 30 ORDER BY age LIMIT 10
SELECT city, COUNT(*) FROM users GROUP BY city
Exit criteria

SQL queries map deterministically to query intent

SQL and fluent builder produce identical executable plans

Slice B — Fluent Query Builder Completion

Finalize the fluent query builder interface.

Example usage:

Query::select(["name", "age"])
    .where(field("age").gt(30))
    .order_by("age")
    .limit(10)

Builder goals:

deterministic lowering to planner intent

ergonomic developer usage

identical behavior to SQL queries

Exit criteria

fluent API fully mirrors supported SQL surface

builder output identical to SQL parser output

Slice C — CLI Query Interface

Introduce a basic CLI for query execution.

Example usage:

icydb query "SELECT name FROM users WHERE age > 30 LIMIT 10"

CLI capabilities:

run queries

print results

run EXPLAIN

show execution traces

This CLI is primarily a developer tool, not a full production client.

Slice D — Explain Tooling

Extend the EXPLAIN interface to work with the new SQL surface.

Supported modes:

EXPLAIN
EXPLAIN EXECUTION
EXPLAIN JSON

Guarantee:

identical query -> identical explain output

The explain tree must remain deterministic.

Slice E — Query Error Surface

Standardize the user-facing error model.

Error categories:

syntax errors

query validation errors

execution errors

cursor errors

Errors must map to existing internal error types.

Non-Goals

0.52 explicitly avoids expanding engine semantics.

Not included:

joins

subqueries

window functions

query optimizer heuristics

new storage features

schema migration tools

The goal is usability, not capability expansion.

Success Criteria

0.52 is complete when:

SQL queries map deterministically to existing planner structures

fluent query builder is complete

CLI query interface works

explain tooling supports SQL queries

execution behavior remains identical to 0.51

Expected Outcome

After 0.52:

developers can query IcyDB using SQL or fluent APIs

CLI tools can execute queries and explain plans

the engine remains fully deterministic

the system becomes ready for final 1.0 hardening
