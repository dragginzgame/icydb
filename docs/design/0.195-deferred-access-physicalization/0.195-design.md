# 0.195 Deferred Access Physicalization

Status: tentative first-pass design.

0.195 is an optimization and code-shape line for access planning/execution. The
goal is to remove avoidable physical index-bound work from queries that plan many
logical access branches but execute only a small live subset.

This is not a new query architecture line. It should preserve the 0.185
branch-aware routing contract, the 0.186 shared filter authority contract, the
0.192 read-admission hard cut, and the 0.193 public-query developer contract.

## Problem

The 0.194 SQL performance matrix exposed a repeated hotspot in large sparse
multi-lookup queries:

```sql
SELECT id
FROM PerfAuditToken
WHERE collection_id IN (
  '01KV5N439P0000000000000000',
  'missing-collection-000',
  ...
  'missing-collection-249'
)
ORDER BY id ASC
LIMIT 50
```

Representative baseline from the 0.194.15 investigation:

| Scenario | Total | Compile | Execute | Planner | Plan build | Store | Executor |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| `token.collection_id.sparse_in.page_only.limit50` | 17,520,748 | 2,247,771 | 15,272,977 | 10,058,403 | 8,107,470 | 3,673,838 | 1,471,811 |

That query had 250 logical `IN` values, but execution only opened 12 index range
scans and read 74 index entries. The route was correct and runtime pruning was
working, but the prepared access path still paid too much up front to build raw
physical bounds for branches that were later proven empty.

The narrow optimization would be "make sparse `IN` faster." The better feature
is broader: keep planner authority logical, then defer generation-sensitive or
prunable physical access work until execution knows which branches are actually
live.

## Design Goal

Introduce a deferred physical access boundary:

1. The planner chooses and proves a logical access route.
2. The prepared plan stores stable, generation-independent access recipes.
3. The executor uses current index/store facts to prune empty work.
4. Raw physical bounds and scan work are materialized only for branches that
   survive runtime pruning.

This should reduce work for sparse branch-heavy queries without changing query
semantics, cache identity, cursor format, or persisted state.

## Authority Split

0.195 must preserve a strict split between logical authority and runtime
physicalization.

Logical access authority:

- chosen route class;
- index identity and index key shape;
- branch structure;
- predicate/residual/order proofs;
- cache-stable recipe inputs;
- EXPLAIN route semantics.

Runtime physicalization authority:

- current-generation branch liveness;
- raw store/index key bounds;
- active branch worklist;
- scan handles opened for this execution;
- runtime materialization metrics.

The prepared plan must not cache generation-dependent facts such as "this prefix
is empty", "this branch is live", current row counts, or current index tail
shape. Those facts can change after insert, update, delete, rebuild, or recovery.

## Definitions

### Logical Access Recipe

A logical access recipe is the prepared, cache-stable description of access work.
It should contain enough information to build the same physical scan later, but
it should avoid eagerly building physical bounds for branches that may be pruned.

Examples of stable recipe facts:

- entity/table authority;
- selected index identity;
- index key kind and arity;
- encoded prefix/range components;
- logical branch id;
- order contract;
- residual filter requirement;
- route diagnostics identity.

### Physical Access Work

Physical access work is execution-local. It is built from a logical recipe plus
the current store/index state.

Examples of physical work:

- raw lower and upper index keys;
- exact range/prefix scan descriptors;
- active branch list;
- branch ordering cursor state;
- runtime scan handles.

### Branch Liveness Proof

A branch liveness proof is an execution-time proof that a branch is either worth
opening or can be skipped. It may use current index/cardinality metadata only
inside one guarded execution.

Absence of a proof must be conservative: materialize and scan the branch rather
than silently dropping it.

The first implementation should encode this as a tri-state result, not as a
boolean:

```text
BranchLiveness
  ProvenEmpty { reason }
  PossiblyLive
  UnknownConservative { reason }
```

Materialization rule:

| Liveness | Action |
| --- | --- |
| `ProvenEmpty` | Skip the branch. |
| `PossiblyLive` | Materialize raw bounds and scan. |
| `UnknownConservative` | Materialize raw bounds and scan. |

Metadata absence, metadata lookup failure, stale metadata, or an incompatible
generation must never be treated as empty.

### Generation Compatibility

A branch may be pruned only if the liveness proof is valid for the same
execution generation/read view as the scan that would otherwise consume the
branch.

If the implementation cannot prove that compatibility, it must downgrade the
branch to `UnknownConservative` and scan it. This is as important as the
prepared-plan cache rule: execution-local liveness is safe only when it belongs
to the same store/index view as the execution itself.

## Non-Goals

- No persisted-format changes.
- No cursor-format changes.
- No recovery, marker, journal, or fold-watermark changes.
- No public API changes.
- No SQL/fluent semantic changes.
- No route-choice changes in the first slice.
- No cost-based planner rewrite.
- No generation-dependent plan-cache identity.
- No caching of empty-prefix decisions in prepared plans.
- No change to count/cardinality shortcut authority in the first slice.
- No broad executor rewrite unless the first slice proves the boundary needs it.

## Proposed Contract

Introduce a small runtime materialization seam between prepared access planning
and physical scan execution.

Tentative names:

- `MultiLookupAccessRecipe`
- `RuntimeMultiLookupWorklist`
- `MaterializedMultiLookupBranch`

The first implementation does not need to rename existing access structs
globally. A conservative path is to add the seam only around the branch-heavy
multi-lookup path, then generalize once the shape proves useful.

Only after the narrow path is proven should broader names such as
`PreparedAccessRecipe`, `RuntimeAccessWorklist`, or `MaterializedAccessWork` be
introduced.

Minimum first-slice contract:

```text
prepared exact-prefix logical branches
  -> execution-generation-compatible liveness probe
  -> proven-empty branches skipped
  -> unknown/possibly-live branches conservatively retained
  -> raw bounds built only for retained branches
  -> existing physical scan/order/residual machinery
```

0.195.0 defers physical bound construction only for
execution-generation-compatible, exact-prefix, indexed multi-lookup page
branches. Proven-empty branches are skipped, unknown branches are
conservatively scanned, and all route/cache/cursor semantics remain unchanged.

The runtime materializer should return metrics so perf tests can prove that the
optimization changed the intended work rather than only changing total
instruction counts.

Suggested metrics:

- planned branch count;
- liveness probe count;
- proven-empty branch count;
- unknown-conservative branch count;
- possibly-live branch count;
- materialized branch count;
- raw lower-bound count;
- raw upper-bound count;
- range scan open count;
- branch prune reason;
- fallback-to-conservative-scan count.

The basic accounting invariant should be:

```text
planned_branch_count =
  proven_empty_branch_count
+ unknown_conservative_branch_count
+ possibly_live_branch_count
```

And, unless branch coalescing is introduced:

```text
materialized_branch_count =
  unknown_conservative_branch_count
+ possibly_live_branch_count
```

If a branch can produce multiple physical ranges, the metrics must keep branch
counts separate from raw-bound counts.

### Liveness Probe Cost

The liveness probe must not require constructing the same raw lower/upper
physical bounds that deferred physicalization is trying to avoid.

It may use compact encoded prefixes, index prefix summaries, or existing
cardinality metadata. If it has to build nearly the same scan descriptor as the
old eager path, it should be treated as a failed design and the branch should
fall back to the existing path until a cheaper proof exists.

## Intended Flow

### Existing High-Level Flow

```text
SQL/fluent input
  -> shared filter contract
  -> planner
  -> prepared plan
  -> executor
  -> physical index/data scans
```

### 0.195 Flow

```text
SQL/fluent input
  -> shared filter contract
  -> planner
  -> prepared logical access recipe
  -> runtime access physicalization
  -> physical index/data scans
```

For sparse `IN`/multi-lookup page execution:

1. The planner still selects the same branch-aware indexed route.
2. The prepared plan stores one logical branch per `IN` value.
3. Execution probes current-generation-compatible index metadata for each exact
   prefix.
4. Branches proven empty are skipped.
5. Unknown or possibly-live branches are retained.
6. Raw index bounds are built only for retained branches.
7. The existing scan/merge/order/residual logic consumes the retained worklist.

Pruning may remove only branches proven empty. It must not reorder surviving
branches, change branch tie-breaking, alter cursor branch identity, or change
the merge comparator input domain.

## Why Not Compile-Time Pruning?

Compile-time pruning would be incorrect for cached plans.

Example:

1. Query is prepared while `collection_id = X` has no rows.
2. The compiled plan is cached.
3. A later insert creates a row with `collection_id = X`.
4. The same prepared plan is reused.

If the prepared plan had permanently removed branch `X`, the later query would
miss a valid row. Therefore empty-prefix pruning must remain an execution-time
decision.

## First Slice: 0.195.0

0.195.0 should be narrow and evidence-driven:

1. Source-map the current eager physical-bound construction path for
   `IndexMultiLookup`/large `IN`.
2. Add runtime materialization counters if the current metrics are not precise
   enough.
3. Implement deferred exact-prefix materialization for the indexed
   multi-lookup page path only.
4. Gate the deferred path to measured large exact-prefix multi-lookup page
   queries. Small branch counts should stay on the existing eager path unless
   measurements show the new seam is neutral.
5. Keep count/cardinality paths unchanged.
6. Keep route choice, cache identity, EXPLAIN semantic output, cursor format, and
   public behavior unchanged.
7. Run the targeted sparse `IN` scenarios and the full SQL perf matrix.

The first slice should be considered successful only if it is both faster on the
targeted sparse branch scenario and neutral or better across the broader matrix.

## Follow-Up Slices

Possible follow-up slices, only after 0.195.0 proves the seam:

- branch-set and `OR` physicalization;
- composite exact-prefix branch pruning;
- mixed exact-prefix/range branch worklists;
- LIMIT-aware branch opening;
- covering-projection scan work deferral;
- runtime materialization metrics in EXPLAIN diagnostics;
- read-admission evidence based on actual materialized work, without making
  planner estimates authoritative.

## Acceptance Matrix

| Scenario | Expected Result |
| --- | --- |
| Sparse indexed large `IN` page query | Same rows/order/cursor behavior; fewer raw bounds built; fewer instructions. |
| Dense indexed large `IN` page query | Same rows/order/cursor behavior; no missed branches; neutral or better performance. |
| Small indexed `IN` query | Same route and result behavior; no unnecessary abstraction overhead. |
| Metadata missing for a branch | Branch is retained and scanned conservatively. |
| Metadata generation mismatch | Branch is retained and scanned conservatively, or the proof is rejected before pruning. |
| Cached plan, missing value later inserted | Later execution returns the new row; no cached empty-branch decision. |
| Cached plan, existing value later deleted | Later execution can prune the branch; result remains correct. |
| Duplicate `IN` values | Existing branch identity/dedup behavior is preserved. |
| Branch requiring residual filter | Residual still runs; pruning never proves user-visible semantics by itself. |
| Ordered/cursor-paged query | Cursor and returned order remain byte/logically stable, including resume after pruned branches. |
| Count/cardinality query | Behavior unchanged in first slice. |
| EXPLAIN route output | Route semantics unchanged; optional runtime counters are non-authority diagnostics. |
| Read admission | Existing 0.192/0.193 public-read decisions unchanged. |
| No-default build | No SQL-only helper leakage from the new seam. |

## Implementation Notes

- Prefer adding a narrow runtime materialization seam over rewriting all access
  lowering.
- Avoid changing cache keys unless explicitly versioned and tested.
- Avoid storing `OnceLock` raw physical bounds in prepared plans for branches
  that runtime pruning can remove.
- Keep all current generation checks inside guarded execution.
- Treat missing current-generation metadata as a conservative scan, not a prune.
- Keep route diagnostics separate from runtime pruning diagnostics.
- Static EXPLAIN must not include branch liveness decisions. `EXPLAIN ANALYZE`,
  perf harness diagnostics, or explicitly runtime-labeled output may include
  them.
- Preserve existing branch-aware tests from 0.185 and shared filter authority
  tests from 0.186.

## Complexity Gate

Every implementation slice in this line should report:

- perf delta for targeted queries;
- full-matrix regression summary;
- wasm-size delta if relevant;
- files touched;
- approximate line delta;
- whether the implementation shape became simpler, neutral, or more complex.

If the first implementation requires a broad executor rewrite or changes more
than the multi-lookup physicalization seam, stop and split the design.

## Validation Plan

Minimum validation for the first implementation slice:

- focused access/executor tests for multi-lookup branch pruning;
- metadata-missing and generation-mismatch conservative fallback tests;
- dense 250-value and small 1/2/3-value `IN` tests;
- duplicate-`IN` tests;
- cached-plan reuse tests across insert/delete;
- residual-filter tests for pruned/retained branches;
- cursor/order stability tests for the affected route, including resume after
  pruned branches;
- targeted sparse `IN` perf matrix before/after;
- full SQL perf matrix before/after;
- `cargo fmt --all`;
- focused `cargo test -p icydb-core ...` tests for access/planning/execution;
- `make clippy` or the current supported clippy matrix before release prep.

## Open Questions

- Should the new seam live under access lowering, executor access, or a new
  access physicalization module?
- Should runtime materialization metrics remain perf-harness-only or become
  operator diagnostics later?
- What is the smallest type name change that makes the logical/physical split
  obvious without churn?
- Can branch liveness checks reuse existing index cardinality/prefix metadata
  directly, or does that need a small wrapper contract?
- What branch-count threshold activates the deferred path for 0.195.0? This
  should be measurement-selected, not guessed permanently.
- Should LIMIT-aware branch opening remain a follow-up until sparse exact-prefix
  pruning is proven?

## Closeout Criteria

0.195 can close when:

- deferred access physicalization exists for at least the sparse indexed
  multi-lookup page path;
- liveness is tri-state and generation-compatible;
- unknown/missing/stale metadata conservatively scans;
- cached-plan reuse is proven safe across data changes;
- targeted sparse branch queries improve materially;
- small and dense branch-heavy queries do not regress materially;
- full matrix shows no broad regression;
- no persisted-format, cursor-format, recovery, or public API changes landed;
- design and changelog wording avoid overclaiming this as a cost-based planner.
