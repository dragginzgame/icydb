# IcyDB 1.0 Readiness Checklist

This is the release-readiness checklist for work that must not be forgotten
before `1.0.0`. The outcome contract lives in
[`1.0-FEATURES.md`](1.0-FEATURES.md); detailed behavior lives in the active
documents under [`contracts/`](contracts/).

Status was reconciled against the repository on 2026-07-19. An unchecked item
in **Open 1.0 Blockers** blocks `1.0.0`. Checked items in **Completed Baseline**
are current shipped capabilities, not promises that their implementation will
never change before the release. Deferred work and explicit non-goals do not
block `1.0.0`.

---

## Open 1.0 Blockers

### Execution And Evidence

- [ ] Complete
  [`0.205 grouped early materialization`](design/0.205-grouped-early-materialization/0.205-design.md):
  ordered grouped routes must finalize completed groups without retaining the
  full group table, while incompatible routes keep an explicit bounded
  materialized fallback.
- [ ] Complete the
  [`0.206 SQL performance remediation line`](design/0.206-sql-performance-remediation/0.206-design.md):
  turn the 0.204/0.205 ranked evidence into one bounded primary production
  improvement plus at most one directly coupled improvement, a checked-in
  human findings report, exact before/after deltas, and a reviewed
  post-remediation baseline before later schema and runtime feature work.
- [ ] Add the remaining ownership tripwire named by
  [`fast-path-inventory.md`](governance/fast-path-inventory.md): the
  bytes-terminal derivation exception. The grouped dedicated path is guarded.
- [ ] Run the final architecture closeout after the implementation lines above:
  prove no known upward import, duplicated semantic authority, hidden strategy
  switch, or generated-model runtime fallback remains.

### Durable Contracts

- [ ] Publish a byte-level persisted-format specification for every durable
  surface named by
  [`PERSISTED_FORMAT_INVENTORY.md`](contracts/PERSISTED_FORMAT_INVENTORY.md),
  including version handling, bounds, malformed-input behavior, and exact codec
  owners.
- [ ] Freeze the `1.0.0` schema-migration boundary. Either implement durable,
  reentrant migration identity/phases/watermarks for every admitted physical
  migration, or explicitly keep broad online migration out of 1.0 and prove
  unsupported transitions fail closed.
- [ ] Publish a production recovery-size policy. This must be either a supported
  row/index/journal size matrix with PocketIC evidence and fail-closed limits,
  or a streaming/reentrant recovery design for shapes above the supported
  bound. The current host and PocketIC probes are regression floors only.

### Public Contract Freeze

- [ ] Audit and freeze the public DTO/version policy for EXPLAIN, SQL endpoint
  results, storage/schema diagnostics, and performance attribution. Every
  exposed structure must be deliberately opaque, accessor-based, or explicitly
  versioned before 1.0.
- [ ] Document the public numeric contract for checked SQL arithmetic and
  aggregates: coercion domain, overflow errors, representability failures,
  decimal division scale, and rounding. The implementation and regression
  tests exist; the user-facing contract is not yet one complete normative
  statement.
- [ ] Reconcile the operations promise in
  [`1.0-FEATURES.md`](1.0-FEATURES.md) with the current product boundary:
  - [ ] decide whether per-logical-index cardinality and median record size are
    required 1.0 metrics or remove those promises;
  - [ ] define what “CLI collection management” means for the accepted
    schema/catalog model; and
  - [ ] either design supported export/import semantics or move CLI
    export/import to explicit post-1.0 scope. Raw stable-memory import is not
    supported by the current
    [`durability contract`](contracts/DURABILITY.md).
- [ ] Refresh [`1.0-FEATURES.md`](1.0-FEATURES.md) and the active contracts after
  the decisions above so the feature contract, non-goals, CLI behavior, and
  release checklist agree exactly.

## Completed Baseline

### Query, Projection, And Predicates

- [x] The
  [`0.204 SQL correctness matrix and generative-oracles line`](design/0.204-sql-correctness-matrix-generative-oracles/0.204-design.md)
  is complete, including its required correctness, performance, replay, and
  closeout evidence.
- [x] The admitted public SQL surface has deterministic proof for supported
  single-entity query, mutation, metadata, projection, grouping, ordering, and
  aggregate forms; unsupported forms fail closed.
- [x] Ordered pagination has one bounded, opaque, fail-closed current cursor
  envelope with query signature, projection/order/group shape, deterministic
  boundary encoding, strict resume behavior, and live-state semantics. See
  [`CURSOR.md`](contracts/CURSOR.md) and
  [`QUERY_CONTRACT.md`](contracts/QUERY_CONTRACT.md).
- [x] Explicit `ORDER BY` owns result order and null ordering. Without
  `ORDER BY`, access-path order is not a public guarantee; execution remains
  deterministic for one plan over one unchanged state.
- [x] Structural field subsets, aliases, wildcard projection, admitted bounded
  computed expressions, grouped projection, and post-aggregate expressions
  have deterministic output shapes. Fluent entity loads remain typed; SQL
  computed projections use the explicit SQL row DTO.
- [x] Typed comparison, range, logical, parenthesized, null, membership, prefix,
  field-to-field, and admitted computed predicates share deterministic
  normalization and evaluation semantics.
- [x] Predicate pushdown rules are explicit. Canonical equality/range shapes may
  use compatible indexes; residual and non-canonical expressions remain above
  storage. Storage does not evaluate query expressions.

### Aggregation, Planning, And Materialization

- [x] Group identity, declared key order, null grouping, aliases, post-aggregate
  `HAVING`, and admitted DISTINCT/statistical aggregates have deterministic
  semantics.
- [x] Group and DISTINCT state is bounded by `max_groups`,
  `max_group_bytes`, `max_distinct_values_per_group`, and
  `max_distinct_values_total`, with typed failure and no spill or truncation.
  See [`RESOURCE_MODEL.md`](contracts/RESOURCE_MODEL.md).
- [x] The planner is deterministic and rule-based: index compatibility,
  ordered-route detection, covering detection, bounded capability scoring, and
  grouped strategy selection are explicit. There is no cost-based optimizer or
  unbounded plan enumeration.
- [x] The planner proposes and the executor revalidates. Execution may downgrade
  with a typed reason, never upgrades beyond planner proof, and reports the
  resulting route through EXPLAIN/metrics.
- [x] Scalar reads use key-first streaming where structurally valid and keep
  materialization boundaries explicit for sorting, DISTINCT retention,
  bounded top-k/page state, response ownership, and atomic mutation preflight.
- [x] The archived 0.145 local staging work is closed: bounded grouped page
  selection, SQL `RETURNING`, aggregate helper staging, page offsets, covering
  projections, delete resolution, scalar aggregate terminals, and local
  DISTINCT projection staging no longer require the old catch-all TODO.
- [x] Grouped execution modes are explicit and route-authoritative. Eligible
  generic and dedicated grouped-count routes report `OrderedStreaming` when
  route proof survives; incompatible routes report `HashMaterialized`.
  Comparable closeout evidence remains the specific 0.205 blocker above.

### Numeric, Transactions, And Durability

- [x] One internal decimal primitive owns numeric storage and evaluation; no
  external decimal dependency remains.
- [x] Declared scale normalization, checked SQL arithmetic, aggregate overflow,
  deterministic division/rounding, and numeric-widen comparison have shared
  implementation and regression coverage.
- [x] Single-entity save/delete is atomic. Non-atomic batches are fail-fast and
  may commit a prefix; atomic batches are scoped to one entity type. No hidden
  cross-entity atomicity, retry, or transaction-block semantics are inferred.
- [x] Commit markers, journal publication, replay, recovery reentry, and
  secondary-index rebuild have fail-closed and idempotence coverage. See
  [`DURABILITY.md`](contracts/DURABILITY.md).
- [x] Accepted schema snapshots are runtime authority for schema identity,
  stable field identity/slots, defaults, enums, indexes, relations, and row
  decoding. Generated models remain proposal/reconciliation/test inputs only.
- [x] Key, row, structural-value, schema, marker, journal, and index decoding is
  bounded and fallible; ordering, prefix matching, and secondary-index rebuild
  are deterministic.
- [x] Read-only integrity tooling detects row/index/relation/schema corruption
  and validates replay/rebuild invariants without moving query semantics into
  storage.

### Observability And CLI

- [x] EXPLAIN exposes access strategy, index selection, predicate pushdown,
  residual filtering, covering state, continuation mode, execution mode, and
  downgrade reason.
- [x] `DESCRIBE` reports accepted physical field slots; schema/catalog metadata
  queries cover entities, fields, stores, and indexes.
- [x] Storage snapshots expose per-entity row counts and bytes plus store-level
  data/index/schema entries and memory footprint.
- [x] Execution metrics expose scan/filter/return counts, aggregate facts, and
  current streaming/materialized mode labels.
- [x] The CLI supports schema show/check, storage snapshots, metrics,
  diagnostics, SQL query/insert/update/delete, EXPLAIN, DDL through the SQL
  surface, and index inspection.

## Explicit 1.0 Non-Goals

These are intentionally outside the 1.0 feature contract and are not release
blockers:

- Relational joins.
- Cost-based optimization.
- Window functions.
- Recursive queries.
- Distributed query execution.
- Snapshot-isolated pagination across calls.
- Implicit cross-entity transactions or PostgreSQL-style transaction blocks.
- Heuristic plan instability or silent runtime strategy upgrades.
- Unbounded projection expressions beyond the admitted bounded scalar family.
- Compatibility aliases, old cursor/schema/protocol decoders, or other legacy
  fallback paths removed before `1.0.0`.

The migration and export/import decisions remain blockers above until they are
placed deliberately in either the 1.0 contract or this non-goal list.

---

## Final Completion Bar

`1.0.0` is ready only when:

- [ ] every item in **Open 1.0 Blockers** is closed;
- [ ] 0.205 and 0.206 closeout evidence is clean;
- [ ] the byte-format, migration, recovery-scale, cursor, numeric, and public
  DTO contracts are frozen and mutually consistent;
- [ ] the final architecture audit finds no known semantic-authority drift or
  layer violation; and
- [ ] [`1.0-FEATURES.md`](1.0-FEATURES.md), active contracts, CLI behavior, and
  this checklist describe the same maintained product.
