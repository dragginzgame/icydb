# IcyDB 1.0 Features

This document describes the high-level feature and engineering contract for
IcyDB 1.0. It is intentionally outcome-focused. Detailed implementation
checklists live in `docs/1.0-TODO.md`.

---

## Query Surface

IcyDB 1.0 supports a bounded, deterministic single-entity query model.

Required behavior:

* Core `SELECT`-style filtering, projection, grouping, ordering, pagination,
  and aggregate flows are complete for admitted single-entity shapes.
* Cross-form interactions work where semantically valid.
* Unsupported SQL or fluent query shapes fail closed with structured errors.
* Result ordering is undefined unless `ORDER BY` is explicit.
* For one plan on one snapshot, unordered results remain deterministic.
* `ORDER BY` null ordering is explicit: `NULL` sorts before non-null values.

---

## Projection

IcyDB 1.0 supports structural projection, aggregate projection, and the
admitted bounded scalar expression projection family.

Required behavior:

* Explicit field subsets return deterministic output shapes.
* Structural field omission semantics are defined.
* Grouped projection validates aggregate outputs against `GROUP BY`.
* `HAVING` fields are not auto-projected.
* Aggregate aliases and output names are deterministic.
* Unsupported broad computed projection forms remain fail-closed until
  explicitly admitted.

---

## Predicate Engine

IcyDB 1.0 has one deterministic predicate model shared by SQL and fluent
queries.

Required behavior:

* Typed comparison, range, logical, parenthesized, and null predicates have
  defined semantics.
* Predicate normalization is deterministic.
* Index pushdown eligibility is explicit and fail-closed.
* Storage never owns expression evaluation or query semantics.
* Field-vs-field, computed, and non-canonical predicates stay post-fetch unless
  proven equivalent to canonical field predicates.

---

## Aggregation

IcyDB 1.0 supports bounded aggregate execution with deterministic identity,
grouping, and fallback behavior.

Required behavior:

* `COUNT DISTINCT` and admitted DISTINCT/statistical aggregate variants are
  bounded by explicit memory contracts.
* Group keys have deterministic identity, ordering, and null behavior.
* `HAVING` evaluates only after aggregation.
* Streaming grouping is used where structurally valid.
* Materialized fallback is explicit where required.
* DISTINCT budget enforcement is part of the aggregate contract.

---

## Planner And Executor

IcyDB 1.0 uses a deterministic rule-based planner and an executor that consumes
explicit plan artifacts.

Required behavior:

* No cost-based optimizer.
* Strategy selection is deterministic.
* Index compatibility, ordered scan, and covering detection rules are explicit.
* Planner proposes execution strategy.
* Executor revalidates and may downgrade.
* Executor never silently upgrades or switches strategy.
* Explain output surfaces index selection, execution mode, and downgrades.

---

## Streaming And Materialization

IcyDB 1.0 is streaming-first where structurally valid, with explicit
materialization boundaries.

Required behavior:

* Rows are not buffered unless a semantic or public boundary requires it.
* `ORDER BY`, DISTINCT retention, cursor construction, public response
  ownership, and unbounded grouped ordering remain valid materialization
  boundaries.
* Streaming execution does not reorder rows except through explicit `ORDER BY`
  semantics.
* Fallback from streaming to materialized execution is visible and deliberate.
* Large-row and blob-heavy paths avoid copying unreturned fields where possible.

---

## Continuation Model

IcyDB 1.0 has a stable, fail-closed continuation and cursor model.

Required behavior:

* Continuation envelopes are stable.
* Plan fingerprint, projection shape, ordering, and grouping are represented in
  cursor validation.
* Cursor boundary encoding is deterministic.
* Invalid or stale cursor state fails closed.

---

## Numeric Core

IcyDB 1.0 has one numeric model for storage, projection, predicates, and
aggregates.

Required behavior:

* One internal decimal primitive.
* Schema-level scale enforcement.
* Authored decimal values normalize to declared field scale before storage.
* Overflow and rounding rules are defined.
* Numeric aggregate behavior is stable and deterministic.

---

## Transaction Model

IcyDB 1.0 keeps transactional guarantees explicit and bounded.

Required behavior:

* Single-entity save/delete is atomic.
* Batch behavior is documented, including that `*_many_non_atomic` helpers are
  fail-fast and may leave a committed prefix.
* Atomic batch behavior is scoped per entity type and exposed through the
  `*_many_atomic` helpers.
* There is no implicit cross-entity atomicity.
* Returning `Err` from a canister update method does not roll back prior
  successful writes in that method.
* Replay and recovery semantics are validated.
* There are no hidden retry semantics.
* There are no implicit Postgres-style transaction blocks or isolation levels.

---

## Storage And Indexing

IcyDB 1.0 has deterministic storage and index contracts.

Required behavior:

* Durable user data examples use `storage(journaled(...))` unless volatility is
  the explicit point of the example.
* `storage(heap())` is documented as live-only volatile state with no recovery
  path, no stable allocation identity, and no durable commit-marker/journal
  participation.
* Live database schemas have persistent identity independent of generated Rust
  field slot order.
* Database-level field defaults are explicit schema contracts, not implicit Rust
  `Default` behavior.
* Key encoding and ordering are deterministic.
* Secondary index compatibility and prefix matching rules are stable.
* Query-layer logic does not live in storage.
* Corruption detection and replay integrity validation are available.
* Secondary index rebuild behavior is deterministic.

---

## Observability

IcyDB 1.0 exposes enough operational information to understand query behavior.

Required behavior:

* `EXPLAIN` reports strategy, index selection, execution mode, and downgrades.
* Predicate pushdown visibility distinguishes pushed predicates from post-fetch
  predicates.
* Engine metrics expose maintained row/work counters and aggregate/streaming
  execution facts; storage snapshots expose current entity/store counts, bytes,
  allocations, and index inventory.
* CLI workflows cover generated configuration, local canister lifecycle, SQL
  query/mutation/DDL/EXPLAIN and index inspection, accepted schema show/check,
  storage snapshots, metrics, and compact diagnostic lookup.

The following 1.0 choices remain open release blockers rather than current
capabilities: per-logical-index cardinality and median-record-size metrics, a
precise collection-management promise, and supported export/import semantics.
They must be deliberately admitted here or moved to explicit post-1.0 scope in
[`1.0-TODO.md`](1.0-TODO.md). Raw stable-memory import is not supported.

---

## Stability And Hardening

IcyDB 1.0 closes structural ownership gaps.

Required behavior:

* No known layering violations.
* No duplicated semantic authority.
* Module ownership is clear.
* Lower layers do not import upward.
* Regression tests cover grouping, HAVING, DISTINCT budgets, continuation
  stability, downgrade determinism, and explain determinism.

---

## Non-Goals

The following remain out of scope for 1.0:

* Relational joins
* Cost-based optimization
* Window functions
* Recursive queries
* Implicit transactional inference
* Heuristic plan instability
* Unbounded projection expressions beyond the admitted bounded scalar family
* Distributed query execution

---

## Completion Bar

IcyDB 1.0 is ready when the feature contract above is implemented, tested, and
documented, and when `docs/1.0-TODO.md` has no unresolved must-fix items.
