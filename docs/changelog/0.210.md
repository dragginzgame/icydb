# IcyDB 0.210 Changelog

## 0.210.2 - 2026-07-22

### 📝 Summary

- Closes the independent 0.210 audit findings with hard cuts: resume now takes
  the authorized application operation id, journal mutation exhaustion fails
  before marker publication, and the last test-only unbounded `UPDATE` route is
  removed.
- Existing `0.210.1` continuations must be abandoned and prepared again. There
  is no old-signature wrapper, token translation, or compatibility decoder.

### 🩹 Fixed

- `resume_trusted_sql_resumable_update::<E>` now accepts `operation_id` and
  rejects a continuation whose bound id differs before store recovery, row
  access, or mutation.
- Journaled row and schema mutations reserve both their append sequence and a
  representable successor revision. Exhaustion returns a typed stable runtime
  diagnostic with unchanged rows, indexes, journal tail, watermark, and marker.
- The continuation batch-policy identity is deterministically derived from the
  token version, size and scan bounds, staged-row bound, packing, checkpoint,
  `needs_patch`, revision, and marker-accounting policy versions.
- Removed the test-only unbounded update execution contract and migrated active
  tests to explicit Exact or canonical primary-key-ordered Prefix entry points.
- Generated SQL surface tests now parse the generated Rust AST and assert the
  complete exported method/type contract contains no resumable state.
- Corrected recurring audit guidance and closeout evidence so it no longer
  teaches retired update ingress or overstates unexercised boundaries.

### 🧪 Testing

- Added real-boundary operation-id mismatch coverage before row access, typed
  journal-revision exhaustion coverage, a full state-neutral resumable
  exhaustion test, and explicit 64th/65th-row checkpoint plus old-token replay
  evidence.
- Added a golden batch-policy identity test that perturbs every compatibility
  input independently, semantic generated-export surface coverage for both
  public update policies, and a structural guard requiring explicit bounded
  candidate collection for every SQL `UPDATE` contract.
- Aligns resumable-update test-only imports with the SQL feature boundary so
  no-default test and Clippy builds remain warning-free.

### 📊 Audit

- Against an exact `v0.210.1` archive built with matched release flags and
  source remapping, raw non-gzipped `sql_perf` Wasm grows from 3,964,124 to
  3,964,390 bytes: +266 bytes (+0.0067%). Two clean final-worktree builds from
  fresh target directories reproduce the raw size.
- The clean builds do not reproduce byte identity, so this release records no
  canonical SHA-256. Raw size remains the primary evidence; gzip and shrink
  measurements are secondary context in the design closeout record.

### 🧭 Migration Notes

- Resume with
  `resume_trusted_sql_resumable_update::<E>(operation_id, sql, continuation)`.
  The application must load the operation id and continuation from the same
  authorized durable custody record.
- Re-prepare any operation created by `0.210.1`; the current batch-policy
  identity intentionally rejects the earlier continuation.

## 0.210.1 - 2026-07-22

### 📝 Summary

- Adds trusted resumable SQL `UPDATE` for fixed convergence work that cannot fit
  safely inside the exact single-call contract.
- Preparation is read-only. Each Forward call commits at most one bounded atomic
  batch, and only a clean full Verify sweep at one stable durable revision may
  report completion.
- The application owns authorization, operation-id uniqueness, and durable
  continuation custody; IcyDB does not add a job registry or bearer-token
  endpoint.

### ➕ Added

- `prepare_trusted_sql_resumable_update::<E>` validates one journaled,
  single-entity fixed update and returns a bounded current-format continuation
  without scanning or mutating rows.
- `resume_trusted_sql_resumable_update::<E>` performs one primary-key-ordered
  Forward or Verify step. Forward scans at most 256 keys, stages at most 64
  candidate rows, and packs the largest exact prefix that fits the canonical
  journal and marker codecs.
- Continuations bind the operation id, target allocation and entity, accepted
  schema, normalized scope, accepted fixed patch, phase checkpoint, captured
  durable revision, and engine batch policy.
- Per-call receipts expose phase, scanned keys, committed rows, restart reason,
  continuation state, and verified completion without claiming exact cumulative
  affected-row counts.
- Adds stable typed diagnostics for malformed or mismatched continuations,
  unsupported storage and SQL shapes, scope/patch dependency conflicts,
  application callbacks, globally constrained authored or managed fields, and
  single-row commit-window exhaustion.

### 🔧 Changed

- Accepted schema is the single eligibility authority for field dependencies,
  global constraints, and update-managed constraints. Generated visitor
  profiles provide only the transitive application-callback proof needed to
  reject unsafe multi-batch execution.
- Resumable rows converge through the existing accepted structural after-image
  resolver and marker-owned atomic batch pipeline. There is no second mutation
  executor, secondary-index progression route, token compatibility decoder, or
  fallback to prefix semantics.
- Direct typed SQL update boundaries now validate the entity named by the SQL
  statement instead of allowing the generic entity route to override it.

### 🧪 Testing

- Focused coverage proves read-only deterministic preparation, current-format
  token bounds, target/schema/scope/patch/batch-policy binding, callback and
  constraint rejection, clean-page progress, exact marker packing, response-loss
  replay, marker recovery, journal-fold stability, conservative revision
  restart, residual-work restart, and stable multi-page completion.
- A Toko-shaped 130-row scoped tier reset converges over multiple calls while
  preserving ten out-of-scope rows. The maintained 512-row PocketIC audit uses
  eight 64-row Forward batches and two 256-key Verify pages.
- Core, facade, CLI, diagnostic, macro, SQL coverage, native/Wasm feature, and
  structural-invariant checks pass under focused validation. Full repository
  suites remain owned by the human push workflow.

### 📊 Audit

- The slice changes approximately 74 files, adding 3,847 and removing 136
  lines: about +2,324/-102 production, +1,195/-3 focused tests, +177/-2
  performance and structural evidence, and +151/-29 active documentation.
- Prepare uses 300,660 local instructions in the maintained fixture. Forward
  batches average 47,101,864 instructions and Verify pages average 5,471,511.
  A full 64-row Forward batch encodes 10,869 journal bytes inside a 10,907-byte
  marker/control slot; the fixture continuation is 156 bytes under a 2 KiB
  bound.
- Raw non-gzipped `sql_perf` Wasm grows from 3,917,856 to 3,963,471 bytes:
  +45,615 bytes (+1.16%). The implementation necessarily adds protocol
  complexity, while retaining one token codec, scanner, batch committer, and
  verification loop.
- Dependencies, Cargo package versions, accepted-schema and persisted-row
  formats, read cursors, generated schemas, and ordinary query semantics are
  unchanged.

### 🧭 Migration Notes

- Use exact update when the complete target is known to fit one call. Use prefix
  update only when one deterministic ordered window is the business operation.
  Use resumable update when bounded multi-call convergence is acceptable.
- Persist the initial continuation outside the target store before the first
  resume, reauthorize every resume by application operation id, and never expose
  raw continuation bytes through a public endpoint.
- Active operations restart after accepted-schema or engine changes; no token
  migration or compatibility path is provided before 1.0.

## 0.210.0 - 2026-07-22

### 📝 Summary

- Adds an exact complete-set trusted SQL `UPDATE` contract whose success proves
  that the entire matching set was selected and committed atomically.
- Retains ordered-prefix `UPDATE` as a separate explicit contract whose success
  applies only to the requested bounded window.
- Removes ambiguous trusted `UPDATE` behavior from the broad mutation ingress.
  Callers must now state whether they require exact completion or an intentional
  prefix.

### ⚠️ Breaking

- `execute_trusted_sql_mutation::<E>` now admits `INSERT` and `DELETE` only.
  Passing `UPDATE` returns the typed explicit-intent diagnostic.
- Trusted `UPDATE` callers must use
  `execute_trusted_sql_exact_update::<E>(sql, require_affected_at_most)` or
  `execute_trusted_sql_prefix_update::<E>(sql)`.
- SQL compilation and cache terminology is hard-cut from the misleading
  update-surface name to the mutation-surface name. No aliases or compatibility
  wrappers preserve the retired internal spelling.

### ➕ Added

- Exact updates require a positive `require_affected_at_most` assertion no
  greater than 4,096 rows. A cap-plus-one match rejects before mutation.
- Exact selection independently enforces a 4,096-key scan budget, also with one
  overflow probe. A target that fits the affected-row assertion can therefore
  still reject safely when proving completeness would scan too much state.
- Exact selection uses accepted-schema authority and canonical primary-key
  traversal rather than allowing a secondary-index shortcut to weaken the
  completion proof.
- Adds typed diagnostics for missing or oversized assertions, affected-row
  overflow, unsupported SQL windows, scan-budget overflow, and ambiguous
  mutation ingress.

Representative exact statement:

```sql
UPDATE User SET state = 'archived' WHERE tenant_id = 7 RETURNING id
```

The caller supplies the positive complete-set assertion separately to
`execute_trusted_sql_exact_update`.

### 🔧 Changed

- Exact and prefix updates converge on the existing accepted structural
  after-image resolver and single-entity atomic mutation pipeline. Exact
  overflow, constraint failure, and pre-marker failure leave rows and derived
  state unchanged.
- Prefix updates retain the maintained public bounded policy: canonical
  ascending primary-key order, a positive limit no greater than 100, and no
  offset. Prefix success makes no claim about later matching rows.
- SQL surface documentation, diagnostics, generated coverage metadata, public
  examples, and performance fixtures now use the explicit contracts.

### 🧪 Testing

- Focused coverage proves empty and complete exact targets, cap-plus-one and
  scan-budget rejection, bounded `RETURNING`, canonical prefix selection,
  unique and relation failure atomicity, and journaled recovery.
- Exact behavior passes with diagnostics enabled and disabled. Public facade
  compilation, diagnostic registry round trips, SQL coverage-manifest checks,
  structural guards, warnings-denied Clippy, formatting, and diff hygiene pass.
- The focused PocketIC SQL-write matrix remains within its instruction bounds.
  A separate 0.209.1 comparison run could not start its isolated PocketIC
  server, so no instruction delta is claimed.

### 📊 Audit

- Before this detailed release-note file was added, the slice changed 57 files,
  adding 1,466 and removing 282 lines. Production code accounts for
  approximately 958 additions and 165 removals; the remaining growth is focused
  tests, performance evidence, and active documentation.
- The implementation is larger because exact completion adds distinct row and
  scan proofs, but ownership remains singular: one exact policy, one
  primary-only planning lane, one executor scan probe, and the existing atomic
  mutation authority.
- Raw non-gzipped `sql_perf` Wasm decreases from 3,983,287 to 3,917,856 bytes,
  a reduction of 65,431 bytes (1.64%).
- Dependencies, Cargo package versions, persisted formats, cursor encoding,
  generated schemas, and ordinary read semantics are unchanged.

### 🧭 Migration Notes

- Use exact update when success must mean that every matching row was changed.
  Choose an assertion that reflects the application invariant, not a desired
  page size.
- Use prefix update only when processing one deterministic ordered window is
  itself the intended result. It is not a fallback for exact completion.
