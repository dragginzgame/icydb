# IcyDB 0.192 Changelog

## 0.192.10

### 📝 Summary

- Continues the duplicate-code cleanup pass with the remaining
  authority-adjacent SQL write-policy mirror.
- Shares common SQL UPDATE/DELETE shape rejection and execution-bound
  derivation while keeping statement-family semantics local.
- Shares diagnostics storage snapshot metadata derivation while keeping public
  report DTO fields and accessors explicit.

### 🧹 Cleanup

- Added a shared SQL write-policy exposure classifier for generated-query,
  generated-DDL, no-filter, broad-filter, and safe-filter shapes.
- Added a shared execution-bound derivation path for `WHERE`, `LIMIT`, and
  `RETURNING` bounded SQL writes.
- Preserved UPDATE-owned assignment checks, DELETE-owned delete-plan checks,
  and family-specific rejection enums.
- Added a private diagnostics storage-role snapshot metadata builder so data,
  index, and schema storage reports derive allocation, durability, recovery,
  commit, and accepted-schema metadata through one path.
- Left public diagnostics Candid record fields, getter names, and report
  layout unchanged.

### 🧪 Testing

- Verified focused SQL write-policy, UPDATE-policy, DELETE-policy, and
  structural guard tests.
- Verified diagnostics report tests, including storage snapshot field-key
  stability coverage.
- Verified invariant checks, `icydb-core` no-default compilation,
  `icydb-core` clippy with all targets and all features, formatting, and
  whitespace validation.

### 🧭 Migration Notes

- No public API, generated endpoint, read-admission, cursor, response-shape,
  cache-identity, diagnostics Candid field, or persisted-format behavior
  changes are introduced.
- SQL UPDATE and DELETE continue to return their statement-family rejection
  behavior; only the shared classification and bound derivation are
  centralized.

## 0.192.8

### 📝 Summary

- Hard-cuts ordinary typed/fluent read execution onto one default bounded
  read-admission path.
- Removes application-facing custom read-policy helpers so public reads have
  one normal execution route and explicit trusted bypasses for admin code.
- Extends typed/fluent read-admission coverage to grouped query shapes.
- Documents the two-budget contract for grouped public reads: query hard
  limits plus policy grouped budgets.

### 🔧 Changed

- `DbSession::execute_query`, normal fluent `execute`, `execute_rows`,
  cursor-paged execution, and fluent terminals now apply the default bounded
  read-admission policy before row execution.
- The default policy rejects unsafe full scans, non-zero offsets, materialized
  sorts, missing row bounds, and grouped reads that do not fit the built-in
  grouped budget contract.
- Trusted maintenance/admin code must use `execute_query_trusted`,
  `*_trusted` fluent methods, or `trusted_read_unchecked()` to bypass the
  default bounded read gate.
- Core policy-bound SQL execution remains available to in-crate regression
  tests only; generated SQL endpoints remain controller-gated trusted/admin
  surfaces.

### 🗑️ Removed

- Removed the public facade custom read-policy execution/evaluation helpers,
  including policy-bound SQL query execution and typed/fluent
  read-admission/ensure helpers.
- Stopped re-exporting the custom `QueryAdmissionPolicy` and
  `GroupedAdmissionPolicy` builder types from the public facade.

### ⚠️ Breaking

- This is a pre-1.0 hard cut. Application code that called the removed custom
  policy helpers must move to the normal bounded read path or an explicit
  trusted/admin bypass.
- Existing typed/fluent queries that relied on full scans, non-zero offset,
  materialized sort, or missing row bounds on the normal execution path now
  return read-admission errors unless routed through trusted execution.

### 🧪 Testing

- Added default fluent execution coverage for rejecting unindexed full scans.
- Added default fluent terminal coverage for rejecting unindexed full scans.
- Added default fluent coverage for admitted indexed bounded reads.
- Preserved internal SQL policy-admission regression coverage behind test-only
  visibility.
- Verified focused read-admission tests, core/facade no-default checks,
  affected-package clippy with all features, and formatting.
- Added fluent grouped admission coverage for rejection without policy grouped
  budgets.
- Added fluent grouped admission coverage for rejection without query-side
  grouped hard limits.
- Added positive coverage for grouped fluent admission with both query hard
  limits and policy grouped budgets.
- Added coverage for grouped query budgets exceeding the policy.
- Added DISTINCT grouped coverage for both missing distinct budget rejection
  and explicit distinct budget admission.

### 📚 Documentation

- Updated the read-admission contract to state that typed/fluent grouped
  public reads must carry `grouped_limits(max_groups, max_group_bytes)` in the
  query shape and a matching `GroupedAdmissionPolicy` in the admission policy.
- Documented that grouped `DISTINCT` aggregates require an explicit
  distinct-entry budget in the policy.

### 🧭 Migration Notes

- Prefer ordinary typed/fluent execution for caller-facing endpoints; it now
  fails closed by default before row execution.
- Use trusted execution only from controller/admin or maintenance paths that
  enforce their own caller authorization and resource policy.
- No generated public read endpoint is added.
- No persisted format, cursor format, response shape, cache identity, or
  successful admitted-query execution behavior change is introduced.

## 0.192.0

### 📝 Summary

- Starts the query-admission and EXPLAIN release line.
- Adds the shared vocabulary needed to classify read execution surfaces before
  running public query workloads.
- Keeps the first slice deliberately narrow: the existing controller-gated SQL
  query endpoint remains admin-style, and public read enforcement is not wired
  into generated endpoints yet.

### ➕ Added

- Added read-admission lanes for public reads, admin ad-hoc reads, diagnostic
  EXPLAIN, and dev/test use.
- Added read-admission policies with finite public-read construction, limit,
  returned-row, scan-bound, full-scan, index, offset, materialized-sort, and
  grouped-query controls.
- Added plan-summary extraction for scalar, grouped, and delete-shaped plans,
  including selected access class, selected index, limit/offset, residual
  filter shape, ordering state, grouped budgets, and materialization facts.
- Added scalar policy evaluation that produces stable rejection reasons instead
  of treating admission as planner prose.
- Added stable `QueryReadAdmission` diagnostic codes and CLI rendering for
  read-admission failures.
- Re-exported the admission vocabulary through the core and facade API layers
  so future public read surfaces and EXPLAIN output can consume the same
  contract.

### 🔧 Changed

- Enabled SQL EXPLAIN support in the CLI facade dependency so shell EXPLAIN
  support matches the core SQL feature graph.
- Added a typed conversion from read-admission diagnostic reasons into query
  and internal error details for future runtime integration.
- Updated the SQL perf matrix audit labeler so the new read-admission
  diagnostic family remains covered by exhaustive diagnostic reporting.

### 🧪 Testing

- Added focused unit coverage for public-read admission defaults, exact
  primary-key admission, indexed bounded scalar admission, missing-limit
  rejection, full-scan rejection, non-zero offset rejection, returned-row cap
  rejection, diagnostic-lane rejection, and stable diagnostic mapping.
- Added diagnostic-code and CLI rendering coverage for the new
  read-admission family.
- Verified the SQL perf matrix audit test target compiles with the new
  diagnostic code.
- Verified focused clippy and no-default checks for the touched packages.

### 🧭 Migration Notes

- No persisted marker, journal, row, schema, index, cursor, fold watermark, or
  structural-value format change is introduced by 0.192.0.
- The existing generated `icydb_query` SQL surface remains controller-gated and
  admin-style.
- 0.192.0 does not add a generated public read endpoint or change SQL/fluent
  query execution semantics.
- The new diagnostic codes are additive.
