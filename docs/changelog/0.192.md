# IcyDB 0.192 Changelog

## 0.192.5

### 📝 Summary

- Extends policy-bound SQL read admission to global aggregate and grouped
  aggregate execution.
- Keeps trusted/admin SQL query behavior unchanged while public read lanes now
  require explicit grouped budgets.

### ➕ Added

- Added policy-bound global aggregate execution so `COUNT(*)` and similar
  aggregate shapes are evaluated through the same read-admission diagnostic
  family as scalar `SELECT` queries.
- Added grouped-query admission checks for group-count caps, per-group byte
  caps, grouped execution cap tightening, and DISTINCT aggregate state.
- Added grouped response-byte probing so grouped result payloads are rejected
  with the existing projection-response admission diagnostic before a
  successful oversized response is returned.
- Added grouped admission facts to text and JSON EXPLAIN output, including
  distinct aggregate count.

### 🧪 Testing

- Added session-level coverage for public SQL rejection of unbounded global
  `COUNT(*)`, grouped queries without budgets, grouped DISTINCT without a
  distinct-state budget, and grouped responses above the configured byte cap.
- Added positive coverage for indexed grouped SQL admitted by explicit group
  and response budgets without requiring a scalar `LIMIT`.
- Verified focused read-admission tests, shared admission unit tests,
  `icydb-core` clippy, `icydb-core` no-default checks, formatting, and
  whitespace validation for the touched surfaces.

### 🧭 Migration Notes

- Existing trusted `execute_sql_query` callers keep their current behavior.
- Policy-bound grouped SQL reads require callers to attach an explicit
  `GroupedAdmissionPolicy`.
- No persisted format, cursor format, cache identity, or successful response
  shape change is introduced.

## 0.192.4

### 📝 Summary

- Extends policy-bound SQL SELECT execution with response-byte cap enforcement.
- Rejects route-proven materialized ORDER BY shapes before row execution on
  public read lanes.

### ➕ Added

- Added SQL read-admission response-size probing that estimates the Candid
  projection payload incrementally and returns the existing
  `ProjectionResponseMayExceedLimit` admission diagnostic when the configured
  byte cap is exceeded.
- Added an executor-owned route helper that projects the initial scalar load
  route’s materialized-sort decision into the shared read-admission summary
  without re-deriving route logic in the SQL session layer.
- Added a summary enrichment hook so route-derived materialization facts can be
  attached before policy evaluation while leaving the generic pre-route plan
  summary neutral.

### 🧪 Testing

- Added session-level coverage proving policy-bound SQL rejects oversized
  projection responses and indexed SELECT shapes that require post-access
  ORDER BY materialization.
- Kept the admitted indexed bounded SELECT regression and trusted SQL
  unbounded admin-path coverage green.
- Verified focused read-admission tests, `icydb-core` clippy, `icydb-core`
  no-default checks, formatting, and whitespace validation for the touched
  surfaces.

### 🧭 Migration Notes

- Existing trusted `execute_sql_query` callers keep their current behavior.
- Response-byte admission is enforced after SELECT execution has produced the
  statement result but before the policy-bound SQL method returns success.
- No persisted format, cursor format, cache identity, or successful response
  shape change is introduced.

## 0.192.3

### 📝 Summary

- Adds an explicit SQL execution seam that applies read-admission policy to
  scalar SELECT plans before row materialization.
- Keeps the existing trusted/admin `execute_sql_query` behavior unchanged.

### ➕ Added

- Added core and facade `execute_sql_query_with_read_admission_policy` methods
  for callers that want SQL SELECT execution governed by a
  `QueryAdmissionPolicy`.
- Added SELECT execution routing that resolves the shared prepared plan,
  evaluates the selected plan through `QueryAdmissionSummary`, and rejects
  before projection/materialization when the policy does not admit the plan.
- Added structured read-admission rejections for unsupported statement shapes
  and introspection statements on the policy-bound SQL lane.

### 🧪 Testing

- Added session-level coverage for missing `LIMIT`, full-scan rejection,
  indexed bounded scalar SELECT admission, non-zero `OFFSET` rejection, and
  returned-row cap rejection.
- Added a regression guard proving the existing trusted SQL query method still
  permits the previous unbounded admin-style query behavior.
- Verified focused read-admission tests, core/facade clippy, all-features
  checks, no-default checks, and whitespace validation for the touched
  surfaces.

### 🧭 Migration Notes

- Existing `execute_sql_query` callers keep the trusted/admin behavior.
- Generated public read endpoints are not automatically wired to the new SQL
  policy-bound method in this slice.
- No persisted format, cursor format, response shape, or cache identity change
  is introduced.

## 0.192.2

### 📝 Summary

- Adds finalized JSON read-admission output to execution EXPLAIN diagnostics.
- Keeps JSON admission facts planner-owned and aligned with the existing
  verbose EXPLAIN admission block.

### ➕ Added

- Added an `admission` object to finalized execution EXPLAIN JSON payloads
  carrying lane, decision, rejection reason, plan shape, selected access,
  selected index, limit/offset, scan and returned-row bounds, response-byte
  bound state, residual filter shape, ordering state, materialization facts,
  and grouped-query facts.
- Added SQL `EXPLAIN EXECUTION JSON ...` support so SQL callers can request the
  same structured execution diagnostics as fluent/session JSON explain
  surfaces.
- Added shared JSON rendering for admission summaries so text EXPLAIN, JSON
  EXPLAIN, and future admission-enforced read surfaces consume the same
  planner-owned facts.

### 🧪 Testing

- Added focused JSON rendering coverage for the admission object.
- Added fluent/session and SQL coverage proving finalized execution JSON
  includes admission facts alongside the execution tree.
- Verified focused all-feature, clippy, no-default, and whitespace checks for
  the touched query-admission and SQL EXPLAIN surfaces.

### 🧭 Migration Notes

- No query execution semantics change is introduced by 0.192.2.
- No generated public read endpoint behavior changes in this slice.
- No persisted format, cursor format, response shape, or cache identity change
  is introduced.

## 0.192.1

### 📝 Summary

- Adds a top-level read-admission block to finalized verbose EXPLAIN

- Keeps admission facts separate from route diagnostics so EXPLAIN can show the
  selected diagnostic lane and rejection reason without changing existing
  `diag.*` route snapshot semantics.

### ➕ Added

- Added stable text labels for admission decisions, bound quality, and
  rejection reasons.
- Added a verbose EXPLAIN admission block that reports lane, decision,
  rejection reason, plan shape, selected access class, selected index,
  limit/offset, scan and returned-row bounds, response-byte bound state,
  residual filter shape, ordering state, materialization facts, and grouped
  query facts.
- Attached diagnostic-lane admission summaries to finalized query diagnostics
  produced by fluent/session verbose EXPLAIN and SQL `EXPLAIN EXECUTION`.

### 🧪 Testing

- Added focused coverage for the stable admission-block renderer.
- Added surface coverage proving `explain_execution_verbose()` includes the
  diagnostic-lane admission block.
- Verified the existing verbose route snapshot stays stable because the new
  block is not emitted as `diag.*` route diagnostics.

### 🧭 Migration Notes

- No query execution semantics change is introduced by 0.192.1.
- No generated public read endpoint behavior changes in this slice.
- No persisted format, cursor format, response shape, or cache identity change
  is introduced.

## 0.192.0

### 📝 Summary

- Starts the query-admission and EXPLAIN release line.
- Adds the shared vocabulary needed to classify read execution surfaces before
  running public query workloads.
- Keeps the first slice deliberately narrow: the existing controller-gated SQL
  query endpoint remains admin-style, and public read enforcement is not wired
  into generated endpoints yet.

### ➕ Added

- Added read-admission lanes for public reads, admin ad-hoc reads, diagnostic
  EXPLAIN, and dev/test use.
- Added read-admission policies with finite public-read construction, limit,
  returned-row, scan-bound, full-scan, index, offset, materialized-sort, and
  grouped-query controls.
- Added plan-summary extraction for scalar, grouped, and delete-shaped plans,
  including selected access class, selected index, limit/offset, residual
  filter shape, ordering state, grouped budgets, and materialization facts.
- Added scalar policy evaluation that produces stable rejection reasons instead
  of treating admission as planner prose.
- Added stable `QueryReadAdmission` diagnostic codes and CLI rendering for
  read-admission failures.
- Re-exported the admission vocabulary through the core and facade API layers
  so future public read surfaces and EXPLAIN output can consume the same
  contract.

### 🔧 Changed

- Enabled SQL EXPLAIN support in the CLI facade dependency so shell EXPLAIN
  support matches the core SQL feature graph.
- Added a typed conversion from read-admission diagnostic reasons into query
  and internal error details for future runtime integration.
- Updated the SQL perf matrix audit labeler so the new read-admission
  diagnostic family remains covered by exhaustive diagnostic reporting.

### 🧪 Testing

- Added focused unit coverage for public-read admission defaults, exact
  primary-key admission, indexed bounded scalar admission, missing-limit
  rejection, full-scan rejection, non-zero offset rejection, returned-row cap
  rejection, diagnostic-lane rejection, and stable diagnostic mapping.
- Added diagnostic-code and CLI rendering coverage for the new
  read-admission family.
- Verified the SQL perf matrix audit test target compiles with the new
  diagnostic code.
- Verified focused clippy and no-default checks for the touched packages.

### 🧭 Migration Notes

- No persisted marker, journal, row, schema, index, cursor, fold watermark, or
  structural-value format change is introduced by 0.192.0.
- The existing generated `icydb_query` SQL surface remains controller-gated and
  admin-style.
- 0.192.0 does not add a generated public read endpoint or change SQL/fluent
  query execution semantics.
- The new diagnostic codes are additive.
