# 0.92 Changelog

## 0.92.1

- `0.92.1` follows the first cache-ownership slice with a tighter session-SQL cleanup pass instead of widening the cache model again.
- the session-owned SQL `SELECT` path now carries fewer internal wrappers while preserving the same top-level ownership split introduced in `0.92.0`:
  - cached and explicit-bypass scalar projection execution now share one `SqlSelectPlanCacheEntry` materialization shell
  - grouped SQL statement packaging now owns one shared grouped-page-to-statement helper instead of repeating grouped cursor hex encoding and outward row shaping across lowered and diagnostics paths
  - parsed SQL compile and lowered SQL execution now share one canonical lowered-`SELECT` to structural-query binder instead of rebinding the same lowered `SELECT` shape in parallel
  - the session SQL cache owner now shares one internal lower query-plan fetch helper before the cached-vs-bypass paths diverge on SQL cache attribution
- the same cleanup line followed through on the remaining local SQL result-shaping seams without changing the cache model again:
  - write-side SQL projection assembly keeps `SqlProjectionPayload` as the owner for `RETURNING` and write-projection rows longer instead of eagerly building and then destructuring `SqlStatementResult::Projection`
  - global aggregate projection/result shaping also routes through the same payload carrier instead of open-coding another projection DTO shell
  - grouped page-to-statement shaping no longer keeps the extra grouped-result wrapper beside the canonical grouped statement builder
- this keeps the intended `0.92` cache boundary intact while reducing more duplicated local flow around it:
  - the visibility-keyed SQL prepared-select owner remains the ordinary repeated-`SELECT` boundary
  - the shared lower query-plan cache still exists only as the internal supporting boundary and the explicit lowered/aggregate bypass source
  - grouped diagnostics and lowered grouped execution still differ only in the grouped runtime they invoke, not in a second SQL packaging shell
- the line also adds the missing cache-identity proof for searched `CASE` and nearby semantic deltas on the normal SQL `SELECT` path:
  - searched `CASE` scalar projection, grouped projection, grouped `HAVING`, and grouped aggregate-input alias ordering now have explicit cache-path tests proving they stay on the normal compiled-`SELECT` and SQL prepared-select reuse story
  - nearby semantic changes now have negative cache-identity tests proving they do not alias on the structural SQL cache key:
    - searched `CASE` condition changes
    - grouped searched-`CASE` `HAVING` threshold changes
    - aggregate `DISTINCT` versus non-`DISTINCT`
    - grouped aggregate `ORDER BY` direction changes
- one important clarification from the new test matrix is now explicit in the line’s maintenance story:
  - structural SQL cache identity is the canonical non-alias boundary for this slice
  - plan fingerprints are intentionally coarser in some scalar projection cases and are not the right negative assertion boundary for every SQL cache-key test
- focused verification for the follow-up cleanup and cache-identity matrix stayed on the shipped SQL session surfaces:
  - `cargo test -p icydb-core db::session::tests::sql_surface -- --nocapture`
  - `cargo test -p icydb-core db::session::tests::sql_grouped -- --nocapture`

## 0.92.0

- `0.92.0` starts the SQL cache revisit as a deletion-oriented session-SQL ownership slice rather than a broad runtime rewrite.
- the normal compiled `SELECT` shape no longer carries an ordinary `compiled_cache_key: None` branch through the session-owned cache path:
  - `CompiledSqlCommand::Select` now carries compiled SQL cache identity unconditionally on the normal query surface
  - normal prepared `SELECT` reuse resolves through the existing visibility-keyed SQL prepared-plan owner instead of treating the shared lower query-plan cache as a peer runtime story
  - uncached execution still exists only as an explicit bypass for lowered or aggregate-only helper paths, rather than as the ordinary compiled `SELECT` model
- this keeps the SQL reuse story aligned with the intended `0.92` seam:
  - compiled SQL command cache remains the top SQL cache boundary
  - one visibility-keyed prepared-select owner remains the SQL-owned repeated-`SELECT` boundary
  - the shared lower query-plan cache stays an internal supporting boundary instead of a second ordinary SQL reuse path
- the line also closes the last small parser/token drift that was blocking verification on the real shipped grouped SQL surface:
  - shared SQL keyword tables now include the searched-`CASE` family tokens
  - grouped post-aggregate order parsing and SQL projection parsing now consume `CASE` / `WHEN` / `THEN` / `ELSE` / `END` through the shared keyword surface instead of relying on identifier fallbacks
  - grouped searched-`CASE` projection, searched-`CASE` `HAVING`, and searched-`CASE` aggregate-input alias ordering all stay admitted on the same shipped grouped SQL surface after the cache cut
- verification for this first `0.92` slice stayed focused on the session-SQL seam plus the repeated-query perf cohort:
  - `cargo test -p icydb-core db::session::tests::sql_surface -- --nocapture`
  - `cargo test -p icydb-core db::session::tests::sql_grouped -- --nocapture`
