# 0.92 Changelog

## 0.92.4

- `0.92.4` keeps the post-`0.92.3` cache model intact and moves the follow-up work down to the shared lower query-plan boundary instead of reopening another SQL-owned cache story.
- the shared lower cache now stores `SharedPreparedExecutionPlan` directly in `db/session/query.rs` instead of routing through the former `QueryPlanCacheEntry` carrier:
  - the logical-plan clone that used to sit beside the prepared plan in the cache entry is gone
  - fluent and SQL now both consume the canonical prepared-plan type directly from the shared cache owner
- the same cleanup follows through on naming and ownership at the lower boundary:
  - `cached_shared_query_plan_for_authority(...)`
  - `cached_shared_query_plan_for_entity(...)`
  - `map_cached_shared_query_plan_for_entity(...)`
  now name the real boundary directly instead of talking like the cache still returns an intermediate wrapper object
- the typed query/session boundary is also a little tighter:
  - `cached_prepared_query_plan_for_entity(...)` now takes `&Query<E>` instead of a raw structural query reference, so the typed/fluent session layer owns the `query.structural()` handoff once instead of repeating it across adjacent helpers
  - the SQL side keeps only `SqlProjectionContract` above that shared prepared plan, which makes the prepared-plan seam line up cleanly across fluent and SQL
- the hot-path micro-optimization in this patch is at shared-cache key construction:
  - cache lookups still normalize one scalar predicate once
  - but they no longer walk that same normalized predicate tree a second time just to derive the predicate portion of `StructuralQueryCacheKey`
  - the session cache boundary now computes one normalized-predicate fingerprint and threads that canonical digest into structural cache-key construction
- that keeps the shared lower cache simpler in two ways:
  - one canonical prepared-plan payload instead of a wrapper carrier
  - one canonical normalized-predicate walk per lookup instead of “normalize then hash the normalized tree again”
- focused verification for the lower-boundary cleanup stayed on the shared cache identity and session SQL surfaces:
  - `cargo test -p icydb-core db::session::tests::sql_surface -- --nocapture`
  - `cargo test -p icydb-core db::query::intent::tests::cache_key -- --nocapture`

## 0.92.3

- `0.92.3` takes the architectural step that the earlier `0.92` slices were preparing for: SQL no longer owns any prepared-plan cache state at all.
- the normal SQL `SELECT` ownership model is now intentionally shallow:
  - raw SQL text still resolves through `SqlCompiledCommandCache`
  - SQL still owns the outward projection contract wrapper used to shape statement results
  - the shared lower query-plan cache is now the only prepared-plan reuse layer for both fluent and SQL `SELECT`
- this removes the last remaining SQL-side prepared-plan/cache split between:
  - compiled-command reuse
  - SQL-owned projection shaping
  and leaves only:
  - compiled-command reuse at the SQL surface
  - shared lower query-plan reuse below it
- the collapse keeps SQL thin without deleting SQL-owned statement shaping:
  - `SqlProjectionContract` is now only the projection contract wrapper (`columns` and `fixed_scales`)
  - `CompiledSqlCommand::Select` no longer owns visibility-keyed prepared-plan children
  - repeated SQL execution now proves compiled-command reuse above shared lower query-plan reuse instead of carrying a SQL-only middle cache layer
- the test seam follows the new ownership model rather than preserving the deleted cache layer as a ghost API:
  - cache-version fail-closed coverage remains on the surviving compiled-command key version boundary and the shared lower query-plan key boundary
- the final full SQL perf audit stayed stable after the collapse, which is the important acceptance criterion for this slice:
  - the focused repeat and warm-path cache-contract tests still pass after moving prepared-plan reuse entirely below SQL
  - the sampled repeat rows now show one consistent visible contract:
    - `sql_compiled_command_cache = 9/1`
    - `shared_query_plan_cache = 9/1`
  - the sampled cold/warm parity rows now show the same new boundary:
    - cold rows: `1/0`, `0/1`
    - warm rows: `0/1`, `1/0`
  - that same attribution now holds across scalar, `DISTINCT`, searched-`CASE`, expression-order, grouped, grouped-`HAVING`, and guarded-account sampled rows
- focused verification for the collapse stayed on the SQL session surfaces plus the explicit PocketIC cache-contract and sampled-audit seams:
  - `cargo test -p icydb-core db::session::tests::sql_surface -- --nocapture`
  - `cargo test -p icydb-core db::session::tests::sql_grouped -- --nocapture`

## 0.92.2

- `0.92.2` follows the cache-cleanup work with a proof-oriented PocketIC slice rather than another runtime or cache-owner rewrite.
- the SQL perf audit harness now carries a direct repeat-path cache contract instead of relying only on the printed report rows:
  - representative scalar, expression-order, and grouped repeated `SELECT` shapes must now miss the SQL compiled-command cache exactly once
  - the shared lower query-plan cache must appear only once as cold-fill support work and must not become the repeat-path owner
  - every later in-call repeat must stay on SQL compiled-command hits plus shared lower query-plan hits
- the repeat-row proof is no longer limited to the original happy-path cohort:
  - grouped `HAVING` plus searched-`CASE` aggregate-input alias ordering
  - grouped execution without `ORDER BY`
  - guarded account expression-order queries
  - scalar `WHERE` plus searched `CASE`
  - scalar expression-order queries
  - `DISTINCT` scalar projection queries
  now all sit inside the repeat-row sampled contract too
- the same proof line now broadens the update-warm cache contract across more of the admitted SQL surface instead of only one simple scalar query:
  - simple scalar ordered `SELECT`
  - expression-ordered scalar `SELECT`
  - grouped `SELECT`
  - guarded account expression-order `SELECT`
- the full sampled report also now carries higher-risk cold/warm parity rows instead of only the original scalar parity pair:
  - grouped `HAVING` plus searched-`CASE` aggregate alias ordering
  - guarded account expression ordering
  both now have explicit cold-query and warm-after-update rows in the sampled audit cohort
- that broader warm-path matrix now proves the same intended story across all four shapes:
  - the update-side warm call populates the compiled-command cache plus the shared lower query-plan cache
  - the later query-side call reuses those warmed artifacts directly
  - the shared lower query-plan cache remains cold-fill support work and does not surface as the later query-side reuse boundary
- this keeps `0.92` aligned with the measurement-first cache discipline established earlier in the cache line:
  - printed audit rows remain useful
  - the normal repeat and warm-after-update SQL cache contract is now asserted directly instead of being left as a report-reading exercise
  - representative named audit report rows are also asserted directly, so the printed repeat and cold/warm parity rows have to keep showing the intended compiled-plus-shared attribution
- the full sampled PocketIC audit now confirms that expanded cohort directly instead of only the focused contract tests:
  - the new repeat rows for grouped `HAVING` plus searched `CASE`, grouped-without-`ORDER BY`, guarded account ordering, scalar `WHERE` plus searched `CASE`, scalar expression ordering, and `DISTINCT` all now report the same final repeat pattern: `sql_compiled_command_cache = 9/1`, `shared_query_plan_cache = 9/1`
  - the higher-risk grouped and guarded cold/warm parity rows also keep the intended transition in the sampled report itself: cold rows report `1/0`, `0/1`, while the warmed rows report `0/1`, `1/0`

## 0.92.1

- `0.92.1` follows the first cache-ownership slice with a tighter session-SQL cleanup pass instead of widening the cache model again.
- the session-owned SQL `SELECT` path now carries fewer internal wrappers while preserving the same top-level ownership split introduced in `0.92.0`:
  - cached and explicit-bypass scalar projection execution now share one `SqlProjectionContract` materialization shell
  - grouped SQL statement packaging now owns one shared grouped-page-to-statement helper instead of repeating grouped cursor hex encoding and outward row shaping across lowered and diagnostics paths
  - parsed SQL compile and lowered SQL execution now share one canonical lowered-`SELECT` to structural-query binder instead of rebinding the same lowered `SELECT` shape in parallel
  - the session SQL cache owner now shares one internal lower query-plan fetch helper before the cached-vs-bypass paths diverge on SQL cache attribution
- the same cleanup line followed through on the remaining local SQL result-shaping seams without changing the cache model again:
  - write-side SQL projection assembly keeps `SqlProjectionPayload` as the owner for `RETURNING` and write-projection rows longer instead of eagerly building and then destructuring `SqlStatementResult::Projection`
  - global aggregate projection/result shaping also routes through the same payload carrier instead of open-coding another projection DTO shell
  - grouped page-to-statement shaping no longer keeps the extra grouped-result wrapper beside the canonical grouped statement builder
- this keeps the intended `0.92` cache boundary intact while reducing more duplicated local flow around it:
  - the compiled SQL cache remains the ordinary repeated-`SELECT` entry boundary
  - the shared lower query-plan cache still exists only as the internal supporting boundary and the explicit lowered/aggregate bypass source
  - grouped diagnostics and lowered grouped execution still differ only in the grouped runtime they invoke, not in a second SQL packaging shell
- the line also adds the missing cache-identity proof for searched `CASE` and nearby semantic deltas on the normal SQL `SELECT` path:
  - searched `CASE` scalar projection, grouped projection, grouped `HAVING`, and grouped aggregate-input alias ordering now have explicit cache-path tests proving they stay on the normal compiled-`SELECT` and shared lower reuse story
  - nearby semantic changes now have negative cache-identity tests proving they do not alias on the structural SQL cache key:
    - searched `CASE` condition changes
    - grouped searched-`CASE` `HAVING` threshold changes
    - aggregate `DISTINCT` versus non-`DISTINCT`
    - grouped aggregate `ORDER BY` direction changes
- one important clarification from the new test matrix is now explicit in the line’s maintenance story:
  - structural SQL cache identity is the canonical non-alias boundary for this slice
  - plan fingerprints are intentionally coarser in some scalar projection cases and are not the right negative assertion boundary for every SQL cache-key test
- focused verification for the follow-up cleanup and cache-identity matrix stayed on the shipped SQL session surfaces:
  - `cargo test -p icydb-core db::session::tests::sql_surface -- --nocapture`
  - `cargo test -p icydb-core db::session::tests::sql_grouped -- --nocapture`

## 0.92.0

- `0.92.0` starts the SQL cache revisit as a deletion-oriented session-SQL ownership slice rather than a broad runtime rewrite.
- the normal compiled `SELECT` shape no longer carries an ordinary `compiled_cache_key: None` branch through the session-owned cache path:
  - `CompiledSqlCommand::Select` now carries compiled SQL cache identity unconditionally on the normal query surface
  - normal repeated `SELECT` reuse stops treating the shared lower query-plan cache as a peer runtime story beside the compiled SQL cache
  - uncached execution still exists only as an explicit bypass for lowered or aggregate-only helper paths, rather than as the ordinary compiled `SELECT` model
- this keeps the SQL reuse story aligned with the intended `0.92` seam:
  - compiled SQL command cache remains the top SQL cache boundary
  - the lower query-plan cache stays the only prepared-plan reuse boundary underneath it
- the line also closes the last small parser/token drift that was blocking verification on the real shipped grouped SQL surface:
  - shared SQL keyword tables now include the searched-`CASE` family tokens
  - grouped post-aggregate order parsing and SQL projection parsing now consume `CASE` / `WHEN` / `THEN` / `ELSE` / `END` through the shared keyword surface instead of relying on identifier fallbacks
  - grouped searched-`CASE` projection, searched-`CASE` `HAVING`, and searched-`CASE` aggregate-input alias ordering all stay admitted on the same shipped grouped SQL surface after the cache cut
- the same pre-cache baseline also now includes the last grouped follow-through fixes that were still missing after the earlier SQL-expression unification line:
  - searched `CASE` in `WHERE` now admits bare bool-field leaves like `CASE WHEN is_npc THEN FALSE ELSE TRUE END` instead of requiring every boolean condition leaf to already look like a compare or function shell
  - grouped `HAVING <aggregate_alias>` now rewrites onto the canonical post-aggregate expression shape instead of leaking the alias name through grouped non-group-field validation
  - grouped aggregate `ORDER BY <alias>` now preserves full filtered aggregate identity, including `FILTER (WHERE ...)`, instead of rendering aliases back to a bare aggregate shell and then failing grouped aggregate matching at execution time
  - grouped retained-slot planning also now keeps source-row slots referenced only by aggregate filter expressions, so grouped filtered aggregates over non-grouped source fields no longer fail at grouped aggregate filter evaluation time
- the same baseline now also closes the remaining ordering transition seams that were still hanging off the old query model:
  - SQL `ORDER BY` now stays on `SqlExpr` through parse, alias normalization, and SQL lowering instead of normalizing onto text, rendering back to SQL labels, and reparsing later
  - the generic planner order contract now carries typed `OrderTerm { label, expr, direction }` instead of string-only order fields, so validation, cursor boundary checks, grouped Top-K handling, explain output, fingerprints, and cache keys all consume one typed order shape
  - the fluent query surface also hard-cuts to typed ordering, deleting `SortExpr`, `order_by(...)`, and `order_by_desc(...)` in favor of `OrderExpr`, `OrderTerm`, `asc(...)`, `desc(...)`, `order_term(...)`, and `order_terms(...)`
- that means the `0.92.0` cache and perf baseline is anchored to the real shipped grouped SQL surface, including these previously broken but now-supported grouped forms:
  - grouped searched-`CASE` `WHERE`
  - grouped aggregate-alias `HAVING`
  - grouped filtered aggregate alias `ORDER BY`
  - grouped filtered aggregates whose `FILTER (WHERE ...)` reads non-grouped source fields
  - grouped aggregate alias ordering over searched `CASE` aggregate inputs
  - grouped aggregate alias ordering over filtered aggregates without losing `FILTER (WHERE ...)`
- and it also means the same baseline no longer carries the last transitional ordering API or internal order round-trip:
  - no SQL-side `ORDER BY` render-and-reparse loop
  - no lower planner `ORDER BY` render-and-reparse loop
  - no string-based fluent ordering compatibility surface beside the typed order model
- verification for this first `0.92` slice stayed focused on the session-SQL seam plus the repeated-query perf cohort:
