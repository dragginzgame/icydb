# IcyDB 0.198 Changelog

## 0.198.13

### 📝 Summary

- Hard-cuts public fluent cursor paging to terminal `page(limit)?` and
  `next_page(limit, cursor)?` calls.
- Keeps `PageRequest` and `PagedLoadQuery` as internal implementation details
  instead of public facade concepts.
- Removes the staged public `page(request)?.execute()` shape from active docs
  and examples.
- Updates CLI read-admission guidance so missing public-read bounds point to
  semantic read-intent terminals instead of raw `LIMIT` or request-object
  ceremony.
- Updates the read-admission invariant script to guard the maintained
  `page(limit)` / `next_page(limit, cursor)` vocabulary.

### ⚠️ Breaking

- Public fluent load paging no longer accepts a public `PageRequest`.
- Public fluent load paging no longer returns a public `PagedLoadQuery` staging
  wrapper.
- Replace public `.page(request)?.execute()` call sites with:

```rust
load.page(limit)?
```

or, for continuation:

```rust
load.next_page(limit, cursor)?
```

### 🔧 Changed

- `icydb` facade paging now returns `PagedResponse<E>` directly from
  `page(limit)?` / `next_page(limit, cursor)?`.
- `icydb-core` fluent paging now returns `PagedLoadExecution<E>` directly from
  the same terminal shape.
- `PageRequest` remains inside the core query module as the internal bridge
  between the public terminal arguments and executor paging machinery.
- Active read-intent docs now teach first-page and continuation-page APIs
  separately instead of optional cursor/request-object construction.
- The public facade reference keeps trusted maintenance batches in the
  trusted/admin tier instead of the normal endpoint cookbook.

### 🗑️ Removed

- Removed public re-exports of `PageRequest`.
- Removed public re-exports of `PagedLoadQuery`.
- Removed active-guide references to `PageRequest::first(...)`,
  `PageRequest::next(...)`, and `page(request)?.execute()`.

### 🧭 Migration Notes

- First public page:

```rust
db.load::<User>()
    .order_desc("created_at")
    .order_asc("id")
    .page(50)?;
```

- Continuation page:

```rust
db.load::<User>()
    .order_desc("created_at")
    .order_asc("id")
    .next_page(50, cursor)?;
```

- Keep `partial_window(n).execute_rows()` only for endpoints that deliberately
  return a partial row window.
- Keep `trusted_read_unchecked().admin_batch(...)` for trusted/admin
  maintenance scans.

  271 insertions, 334 deletions.

## 0.198.12

### 📝 Summary

- Hard-cuts the remaining fluent load row-window escape hatches across the
  facade and core fluent APIs.
- Routes deliberate partial row-window reads through `PartialWindowLoadQuery`
  in both layers.
- Keeps semantic terminals unavailable after selecting a partial row-window
  intent.
- Hides direct prebuilt-query execution from the normal public endpoint API
  documentation.
- Updates read-admission invariants so they enforce the final read-intent
  boundary instead of stale `DbSession::execute_query` public-doc wording.

### ⚠️ Breaking

- Public facade load queries no longer expose fluent `.offset(...)`.
- Production core fluent load queries no longer expose public `.limit(...)` or
  `.offset(...)`; deliberate partial row windows use
  `.partial_window(n).execute_rows()` or `.partial_window(n).execute()`.
- Direct prebuilt-query execution remains available only as a hidden advanced
  surface for generated/internal tooling that intentionally owns low-level
  `Query<E>` construction.

### 🔧 Changed

- `partial_window(...)` now returns a dedicated `PartialWindowLoadQuery`
  wrapper in `icydb-core` as well as in the public facade.
- The public facade wrapper delegates to the core partial-window wrapper
  instead of calling core fluent `.limit(...)`.
- `PartialWindowLoadQuery` exposes materialization and diagnostics, but not
  `page(...)`, `collect_complete()`, existence, exact aggregate, or admin-batch
  terminals.
- `Query<E>` remains the low-level internal/direct-query intent type; facade
  docs no longer teach it as the normal endpoint read API.
- Read-admission docs and invariant checks now describe hidden prebuilt-query
  execution as internal/generated tooling, with ordinary endpoint code using
  semantic fluent read terminals.

### 🗑️ Removed

- Removed public facade fluent load `.offset(...)`.
- Removed production core fluent load `.limit(...)` / `.offset(...)`
  exposure; the remaining helper shape is test-only for core planner/session
  coverage.
- Removed stale read-admission invariant requirements that forced
  `DbSession::execute_query` into public crate docs.

### 🧭 Migration Notes

- Replace public load offset pagination with request-owned cursor paging:
  `load::<E>().order_term(...).page(PageRequest::first(n))?.execute()`.
- Replace public load row caps with
  `load::<E>().partial_window(n).execute_rows()` only when a partial row window
  is the endpoint contract.
- Prefer semantic terminals for semantic answers:
  `try_one()`, `page(...)`, `collect_complete()`, `exists()`, and exact
  aggregate helpers.
- Keep direct `Query<E>` construction for internal/generated diagnostics or
  deliberately low-level direct-query tooling, not ordinary public endpoints.

## 0.198.11

### 📝 Summary

- Hard-cuts the remaining read-intent facade vocabulary before 1.0.
- Renames public delete affected-row caps from `.limit(n)` to
  `.max_affected(n)` so mutation safety bounds do not share read-limit
  vocabulary.
- Renames fieldless exact extrema from `.min_exact()` / `.max_exact()` to
  `.min_id_exact()` / `.max_id_exact()` so aggregate terminals say whether
  they return identifiers or field extrema.
- Renames singleton query shaping from `.only()` to `.singleton()` so singleton
  entity access cannot be confused with generic one-row reads.
- Reorganizes the public facade reference into explicit normal endpoint,
  trusted/admin, diagnostics, internal/generated, and response-helper tiers.
- Fixes current compiler/candid compatibility for principal byte accessors
  while keeping const principal constructors intact.

### ⚠️ Breaking

- Public delete queries no longer expose `.limit(n)` for affected-row caps. Use
  `.max_affected(n)`.
- Public load queries no longer expose fieldless `.min_exact()` or
  `.max_exact()`. Use `.min_id_exact()` / `.max_id_exact()` for primary-key ID
  extrema, and keep `.min_exact_by(field)` / `.max_exact_by(field)` for field
  extrema.
- Singleton load/delete shaping no longer exposes `.only()`. Use
  `.singleton()` on `SingletonEntity` types.

### 🔧 Changed

- Public facade docs now teach Tier 1 normal endpoint APIs first and move
  trusted/admin, diagnostics, internal/generated SQL helpers, and response
  narrowing helpers out of the main endpoint path.
- Read-intent and read-admission docs now use the final
  `.max_affected(...)`, `.min_id_exact()`, `.max_id_exact()`, and
  `.singleton()` vocabulary.
- SQL performance audit code and focused session tests now use the final
  delete affected-row cap spelling.
- Exact aggregate compile coverage now checks the final ID-extrema terminal
  names.

### 🩹 Fixed

- `Principal` byte accessors now compile with the current `candid` API without
  changing the principal value constructors.
- Scalar payload length calculation no longer relies on a `const fn` shape that
  is incompatible with the current borrowed-byte access path.

### 🗑️ Removed

- Removed public fluent delete `.limit(n)` affected-row cap spelling.
- Removed public fieldless `.min_exact()` / `.max_exact()` exact aggregate
  spellings.
- Removed public singleton `.only()` fluent shaping.

### 🧭 Migration Notes

- Replace `delete::<E>().limit(n)` with `delete::<E>().max_affected(n)`.
- Replace `load::<E>().min_exact()` with `load::<E>().min_id_exact()`.
- Replace `load::<E>().max_exact()` with `load::<E>().max_id_exact()`.
- Replace `load::<E>().only().try_one()` with
  `load::<E>().singleton().try_one()` for `SingletonEntity` loads.
- Replace singleton deletes using `.only()` with `.singleton()`.
- Continue using `.min_exact_by(field)` / `.max_exact_by(field)` for field
  extrema.

## 0.198.10

## 0.198.8

### 📝 Summary

- Hard-cuts the public load facade to the 0.198 read-intent vocabulary.
- Replaces the public load `.limit(...)` modifier with
  `bounded_window(...)`, making partial row-window reads explicit instead of
  letting raw limit syntax look like a general public-read intent.
- Removes legacy bounded-window helper aliases from public load queries so
  developers choose semantic terminals such as `page(...)`,
  `collect_complete()`, `count_exact()`, `sum_exact(...)`,
  `min_exact(...)`, `max_exact(...)`, and `avg_exact(...)`.
- Adds public compile guards that reject the removed legacy load APIs.
- Keeps cursor tokens, persisted formats, read-admission policy, and trusted
  delete scoping unchanged.

### ⚠️ Breaking

- Public load queries no longer expose `.limit(...)`. Use
  `.bounded_window(...)` only when the endpoint deliberately returns or
  executes over a partial bounded row window.
- Public load queries no longer expose legacy bounded-window aggregate,
  projection, and convenience aliases such as `count()`, `sum_by(...)`,
  `min_by(...)`, `max_by(...)`, `avg_by(...)`, broad row projection helpers,
  and collection convenience helpers.
- Optional exact row materialization remains available through `try_one()`.
  Low-level explicit execution remains available through `execute()`,
  `execute_rows()`, and trusted execution terminals.
- Delete queries keep their explicit `.limit(...)` scoping API because delete
  limits are mutation safety bounds, not public read-intent terminals.

### 🔧 Changed

- Audit canisters now use `bounded_window(...)` for deliberate partial
  row-window execution.
- The SQL performance audit now uses exact aggregate attribution for exact
  count scenarios instead of the removed bounded-window count alias.
- Public facade docs and examples now describe `bounded_window(...)` as the
  explicit low-level escape hatch for partial windows.
- The 0.198 design/status notes now describe the hard-cut public surface
  without legacy read-limit compatibility.
- The read-admission invariant script now checks stable read-intent guide
  sections, API tokens, and concepts instead of brittle narrative sentence
  matches.

### 🗑️ Removed

- Removed public load `.limit(...)` compatibility.
- Removed public bounded-window helper aliases that overlapped with the
  semantic read-intent terminals introduced in earlier 0.198 patches.
- Removed documentation that taught raw load limits as the preferred public
  read shape.

### 🧭 Migration Notes

- Public list endpoints should use `page(PageRequest)` or
  `execute_paged(PageRequest)` when the caller expects a cursor page.
- Public complete-small-set endpoints should use `collect_complete()`.
- Public exact aggregate endpoints should use the exact aggregate helpers.
- Exact primary-key-style endpoints should use `try_one()`.
- Use `bounded_window(limit)` only when returning or inspecting a deliberately
  partial row window is the API contract.
- Do not add a compatibility wrapper that reintroduces `.limit(...)` on public
  load queries before 1.0. The 0.198 line is intentionally hard-cut.

- Records a supplemental 0.198 status note that treats custom read policies,
  caller-configurable caps, generated public SQL endpoints, persisted-format
  changes, and cursor-token changes as outside the 0.198 architecture rather
  than deferred 0.198 work.

### ➕ Added

- `min_exact()` and `max_exact()` for exact identifier extrema over the admitted
  shape.
- `min_exact_by(field)` and `max_exact_by(field)` for exact field extrema with
  deterministic primary-key tie-breaking.
- `avg_exact(field)` for exact averages over the admitted shape.
- Exact aggregate EXPLAIN helpers for the new min, max, and average terminals.
- Intent-specific raw-limit rejection variants for min, max, and average exact
  aggregate terminals.

  surface and the resolved out-of-scope policy boundary.

### 🔧 Changed

  helpers through the same semantic read-intent limit-ownership boundary.
- `ReadIntentKind::ExactAggregate` now explicitly covers count, sum, min, max,
  and average semantic terminals.
- Low-level aggregate helpers such as `count()`, `sum_by(...)`,
  `min_by(...)`, `max_by(...)`, and `avg_by(...)` remain effective-window
  terminals for APIs that deliberately aggregate a bounded row window.

### 📚 Documentation

- The README now teaches a `PageRequest` public-page example instead of
  leading with a raw row-window limit.
- The read-intent guide now lists the full exact aggregate helper family and
  keeps raw `limit(...)` framed as a bounded row-window primitive.
- The read-admission contract now documents exact aggregate helper behavior,
  EXPLAIN metadata, and raw-limit rejection for the full exact aggregate

- The 0.198 design status now points to the supplemental status note and

  `min_exact_by(field)`, `max_exact()`, `max_exact_by(field)`, or

## 0.198.0

### 📝 Summary

- Starts the read-intent ergonomics line after 0.197 exact-key
  canonicalization.
- Moves public paging to request-owned `PageRequest` input instead of raw
  row-window limits plus separate cursor attachment.
- Adds complete-small-set and exact aggregate terminals so common public read
  promises can be expressed directly.
- Keeps raw row-window execution available for low-level bounded windows
  through `execute_rows()`, `execute()`, `count()`, and `sum_by(...)`.
- Keeps `admin_batch(...)` deferred; it remains a design target, not a public
  API in this slice.

### ⚠️ Breaking

- `page(...)`, `execute_paged(...)`, and `execute_paged_trusted(...)` now take
  a `PageRequest`.
- Public cursor attachment through the fluent load query/page wrapper is
  removed; continuation cursors move through `PageRequest::next(...)` or
  `PageRequest::with_cursor(...)`.
- `exists()`, `not_exists()`, `explain_exists()`, request-owned page
  terminals, `collect_complete()`, `count_exact()`, and `sum_exact(field)`
  reject a prior raw `.limit(...)`.
- The rejection is intentional: these semantic terminals own their own bounds
  and must not silently inherit a low-level row-window cap.

### ➕ Added

- `PageRequest` as the request-owned page shape for public cursor pagination.
- `collect_complete()` for complete small-set reads that return all matching
  rows or fail with a typed too-many-rows intent error.
- `count_exact()` and diagnostics-only `count_exact_with_attribution()` as the
  exact count terminals.
- `sum_exact(field)` as the exact sum terminal.
- Internal read-intent cap authority for public page and complete-small-set
  bounds.
- Intent error variants for raw limits before page, exists, exact aggregate,
  and complete-small-set terminals.
- A read-intent migration guide that maps endpoint promises to the intended
  terminal.

### 🔧 Changed

- Public page size is now supplied through `PageRequest` and clamped by the
  engine-owned public page cap.
- Cursor continuation for page reads is now part of the page request instead of
  an independent fluent modifier.
- `collect_complete()` executes with one lookahead row so it can detect
  overflow instead of silently truncating.
- `count_exact()` and `sum_exact(field)` share the aggregate execution path but
  reject raw limits so exact aggregates cannot mean "aggregate the first N
  rows."
- Read-admission docs now direct callers toward endpoint intent instead of
  arbitrary high caps.
- The read-admission invariant script now checks the read-intent guide,
  cap-authority constants, and absence of public custom policy surfaces.

### 🧭 Migration Notes

- Replace public list reads shaped as `.limit(n).execute_paged()` with
  `.execute_paged(PageRequest::first(n))`.
- Replace continuation reads shaped as `.page()?.cursor(token).execute()` with
  `PageRequest::next(n, token)`.
- Remove `.limit(1)` before existence terminals; use `exists()` or
  `not_exists()` directly.
- Use `collect_complete()` only when the endpoint truly promises a complete
  small set. For larger or user-visible lists, use cursor paging.
- Use `count_exact()` / `sum_exact(field)` for exact aggregate promises. Keep
  `count()` / `sum_by(...)` only when aggregating the effective bounded window
  is the intended contract.
- Do not mechanically replace every raw `limit(...)` call. Classify endpoint
  intent first.

- Adds complete-small-set success, overflow, and raw-limit rejection coverage.
- Adds exact count/sum success and raw-limit rejection coverage.
