# IcyDB 0.198 Changelog

## 0.198.8

### 📝 Summary

- Hard-cuts the public load facade to the 0.198 read-intent vocabulary.
- Replaces the public load `.limit(...)` modifier with
  `bounded_window(...)`, making partial row-window reads explicit instead of
  letting raw limit syntax look like a general public-read intent.
- Removes legacy bounded-window helper aliases from public load queries so
  developers choose semantic terminals such as `page(...)`,
  `collect_complete()`, `count_exact()`, `sum_exact(...)`,
  `min_exact(...)`, `max_exact(...)`, and `avg_exact(...)`.
- Adds public compile guards that reject the removed legacy load APIs.
- Keeps cursor tokens, persisted formats, read-admission policy, and trusted
  delete scoping unchanged.

### ⚠️ Breaking

- Public load queries no longer expose `.limit(...)`. Use
  `.bounded_window(...)` only when the endpoint deliberately returns or
  executes over a partial bounded row window.
- Public load queries no longer expose legacy bounded-window aggregate,
  projection, and convenience aliases such as `count()`, `sum_by(...)`,
  `min_by(...)`, `max_by(...)`, `avg_by(...)`, broad row projection helpers,
  and collection convenience helpers.
- Exact row materialization remains available through `one()` and
  `try_one()`. Low-level explicit execution remains available through
  `execute()`, `execute_rows()`, and trusted execution terminals.
- Delete queries keep their explicit `.limit(...)` scoping API because delete
  limits are mutation safety bounds, not public read-intent terminals.

### 🔧 Changed

- Audit canisters now use `bounded_window(...)` for deliberate partial
  row-window execution.
- The SQL performance audit now uses exact aggregate attribution for exact
  count scenarios instead of the removed bounded-window count alias.
- Public facade docs and examples now describe `bounded_window(...)` as the
  explicit low-level escape hatch for partial windows.
- The 0.198 design/status notes now describe the hard-cut public surface
  without legacy read-limit compatibility.
- The read-admission invariant script now checks stable read-intent guide
  sections, API tokens, and concepts instead of brittle narrative sentence
  matches.

### 🗑️ Removed

- Removed public load `.limit(...)` compatibility.
- Removed public bounded-window helper aliases that overlapped with the
  semantic read-intent terminals introduced in earlier 0.198 patches.
- Removed documentation that taught raw load limits as the preferred public
  read shape.

### 🧭 Migration Notes

- Public list endpoints should use `page(PageRequest)` or
  `execute_paged(PageRequest)` when the caller expects a cursor page.
- Public complete-small-set endpoints should use `collect_complete()`.
- Public exact aggregate endpoints should use the exact aggregate helpers.
- Exact primary-key-style endpoints should use `one()` or `try_one()`.
- Use `bounded_window(limit)` only when returning or inspecting a deliberately
  partial row window is the API contract.
- Do not add a compatibility wrapper that reintroduces `.limit(...)` on public
  load queries before 1.0. The 0.198 line is intentionally hard-cut.

- Intent-specific raw-limit rejection variants for min, max, and average exact
  aggregate terminals.

- `ReadIntentKind::ExactAggregate` now explicitly covers count, sum, min, max,
  and average semantic terminals.
- Low-level aggregate helpers such as `count()`, `sum_by(...)`,
  `min_by(...)`, `max_by(...)`, and `avg_by(...)` remain effective-window
  terminals for APIs that deliberately aggregate a bounded row window.

- The README now teaches a `PageRequest` public-page example instead of
  leading with a raw row-window limit.
- The read-intent guide now lists the full exact aggregate helper family and

- The read-admission contract now documents exact aggregate helper behavior,

  family.
- The 0.198 design status now points to the supplemental status note and

### 🧭 Migration Notes

- Use `count_exact()`, `sum_exact(field)`, `min_exact()`,
  `min_exact_by(field)`, `max_exact()`, `max_exact_by(field)`, or
  `avg_exact(field)` when an endpoint promises an exact aggregate answer.
- Keep `count()`, `sum_by(...)`, `min_by(...)`, `max_by(...)`, or
  `avg_by(...)` only when the endpoint intentionally aggregates the effective
  bounded row window.
- Do not add a raw `limit(...)` before exact aggregate helpers. The semantic
  terminal owns the exact aggregate intent and rejects incompatible raw row
  caps.
- Do not use custom read policies or caller-selected public caps as a 0.198
  migration path. Public endpoints should choose semantic read intents;
  broad maintenance work should stay trusted/admin-gated.

## 0.198.0

### 📝 Summary

- Starts the read-intent ergonomics line after 0.197 exact-key
  canonicalization.
- Moves public paging to request-owned `PageRequest` input instead of raw
  row-window limits plus separate cursor attachment.
- Adds complete-small-set and exact aggregate terminals so common public read
  promises can be expressed directly.
- Keeps raw row-window execution available for low-level bounded windows
  through `execute_rows()`, `execute()`, `count()`, and `sum_by(...)`.
- Keeps `admin_batch(...)` deferred; it remains a design target, not a public
  API in this slice.

### ⚠️ Breaking

- `page(...)`, `execute_paged(...)`, and `execute_paged_trusted(...)` now take
  a `PageRequest`.
- Public cursor attachment through the fluent load query/page wrapper is
  removed; continuation cursors move through `PageRequest::next(...)` or
  `PageRequest::with_cursor(...)`.
- `exists()`, `not_exists()`, `explain_exists()`, request-owned page
  terminals, `collect_complete()`, `count_exact()`, and `sum_exact(field)`
  reject a prior raw `.limit(...)`.
- The rejection is intentional: these semantic terminals own their own bounds
  and must not silently inherit a low-level row-window cap.

### ➕ Added

- `PageRequest` as the request-owned page shape for public cursor pagination.
- `collect_complete()` for complete small-set reads that return all matching
  rows or fail with a typed too-many-rows intent error.
- `count_exact()` and diagnostics-only `count_exact_with_attribution()` as the
  exact count terminals.
- `sum_exact(field)` as the exact sum terminal.
- Internal read-intent cap authority for public page and complete-small-set
  bounds.
- Intent error variants for raw limits before page, exists, exact aggregate,
  and complete-small-set terminals.
- A read-intent migration guide that maps endpoint promises to the intended
  terminal.

### 🔧 Changed

- Public page size is now supplied through `PageRequest` and clamped by the
  engine-owned public page cap.
- Cursor continuation for page reads is now part of the page request instead of
  an independent fluent modifier.
- `collect_complete()` executes with one lookahead row so it can detect
  overflow instead of silently truncating.
- `count_exact()` and `sum_exact(field)` share the aggregate execution path but
  reject raw limits so exact aggregates cannot mean "aggregate the first N
  rows."
- Read-admission docs now direct callers toward endpoint intent instead of
  arbitrary high caps.
- The read-admission invariant script now checks the read-intent guide,
  cap-authority constants, and absence of public custom policy surfaces.

### 🧭 Migration Notes

- Replace public list reads shaped as `.limit(n).execute_paged()` with
  `.execute_paged(PageRequest::first(n))`.
- Replace continuation reads shaped as `.page()?.cursor(token).execute()` with
  `PageRequest::next(n, token)`.
- Remove `.limit(1)` before existence terminals; use `exists()` or
  `not_exists()` directly.
- Use `collect_complete()` only when the endpoint truly promises a complete
  small set. For larger or user-visible lists, use cursor paging.
- Use `count_exact()` / `sum_exact(field)` for exact aggregate promises. Keep
  `count()` / `sum_by(...)` only when aggregating the effective bounded window
  is the intended contract.
- Do not mechanically replace every raw `limit(...)` call. Classify endpoint
  intent first.

### 🧪 Testing

- Adds read-admission coverage for request-owned page limits, page-limit
  clamping, and cursor continuation.
- Adds complete-small-set success, overflow, and raw-limit rejection coverage.
- Adds exact count/sum success and raw-limit rejection coverage.
- Adds existence-terminal raw-limit rejection coverage.
- Updates pagination, branch-set, SQL surface/write, range-choice, and
  execution-convergence tests to use `PageRequest`.
- Preserves focused coverage that raw low-level row-window execution remains
  available through existing low-level terminals.

### 📊 Performance

- No runtime performance improvement is claimed for 0.198.0.
- The slice is an API/admission ergonomics hard cut, not a new optimizer

- Full deterministic perf matrices were not rerun.
- Wasm-size delta was not measured.

- Caller complexity decreases for public endpoints because endpoint promises no

- Documentation and invariant-script complexity increases modestly to keep the
  read-intent contract discoverable and prevent custom policy-surface drift.
- Persisted format, cursor token encoding, recovery, generated-model runtime
  fallback, and cost-based planning remain unchanged.
