## 0.100.1

- `0.100.1` carries the expression-first `WHERE` line into grouped execution instead of leaving grouped pre-aggregate filtering on the older predicate-only runtime seam. Grouped execution preparation now carries the planner-frozen `EffectiveRuntimeFilterProgram`, grouped retained-slot layout compilation marks slots from either a residual predicate program or a compiled scalar boolean-expression program, and grouped fold / grouped distinct runtime evaluate that unified filter program before rows enter grouped aggregation state.
- the prepared grouped runtime resident path was updated too, so the precomputed grouped slot layout used by prepared execution now derives from the effective runtime filter program instead of only the compiled predicate projection. That keeps live grouped execution and prepared grouped execution on the same residual-filter ownership contract.
- grouped observability now matches the shipped runtime behavior. Grouped logical `EXPLAIN` already preserved the semantic `filter_expr`; `0.100.1` locks that in with an explicit grouped searched-`CASE` `WHERE` proof and adds the matching grouped execution-explain proof so grouped residual filter nodes expose:
  - the semantic `filter_expr`
  - the derived residual predicate contract
  instead of collapsing grouped `WHERE` back into one predicate-only story.
- this patch still does not widen grouped `HAVING` architecture or replace access pushdown planning with expression-owned access derivation. `Predicate` remains the narrower derived IR for grouped pushdown / route policy, while grouped pre-aggregate row filtering now shares the same residual expression-first runtime seam as scalar load and delete.

```sql
SELECT age, COUNT(*)
FROM SessionSqlEntity
WHERE CASE WHEN age >= 30 THEN TRUE ELSE age = 20 END
GROUP BY age
ORDER BY age ASC
LIMIT 10;
```

## 0.100.0

- `0.100.0` starts the scalar `WHERE` ownership split described in the `0.100` design line. Scalar plans no longer pretend that the predicate IR is the semantic filter contract. Planning now preserves one planner-owned `filter_expr` alongside the derived predicate-style pushdown contract, so scalar `WHERE` can carry full boolean-expression semantics even when access planning still wants a narrower extracted predicate.
- the scalar explain surfaces now expose that split directly. Logical and execution explain output both carry the semantic `filter_expr` separately from the derived predicate / residual predicate fields, which makes pushed access filtering and residual scalar filtering auditable instead of collapsing them back into one predicate-only story.
- execution now has a real residual scalar expression path instead of forcing all residual scalar filtering back through `PredicateProgram`. The shared scalar page runtime can evaluate either:
  - the older predicate fast path
  - or a compiled scalar boolean-expression program over retained slots / post-access kernel rows
- this expression-first runtime path is now wired through scalar load execution and the delete post-access flow. That means expression-owned scalar `WHERE` no longer stops at planning/explain; searched `CASE`, affine numeric compares, and similar admitted boolean-expression shapes now execute through the planner-frozen residual filter seam instead of failing on the older predicate-only runtime assumption.
- `0.100.0` does not widen grouped query architecture, remove `Predicate` from the repository, or widen prepared-template support to arbitrary boolean expression filters. `Predicate` still exists as a narrower derived IR for pushdown and fast-path planning.
