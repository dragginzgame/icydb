## 0.100.0

- `0.100.0` starts the scalar `WHERE` ownership split described in the `0.100` design line. Scalar plans no longer pretend that the predicate IR is the semantic filter contract. Planning now preserves one planner-owned `filter_expr` alongside the derived predicate-style pushdown contract, so scalar `WHERE` can carry full boolean-expression semantics even when access planning still wants a narrower extracted predicate.
- the scalar explain surfaces now expose that split directly. Logical and execution explain output both carry the semantic `filter_expr` separately from the derived predicate / residual predicate fields, which makes pushed access filtering and residual scalar filtering auditable instead of collapsing them back into one predicate-only story.
- execution now has a real residual scalar expression path instead of forcing all residual scalar filtering back through `PredicateProgram`. The shared scalar page runtime can evaluate either:
  - the older predicate fast path
  - or a compiled scalar boolean-expression program over retained slots / post-access kernel rows
- this expression-first runtime path is now wired through scalar load execution and the delete post-access flow. That means expression-owned scalar `WHERE` no longer stops at planning/explain; searched `CASE`, affine numeric compares, and similar admitted boolean-expression shapes now execute through the planner-frozen residual filter seam instead of failing on the older predicate-only runtime assumption.
- `0.100.0` does not widen grouped query architecture, remove `Predicate` from the repository, or widen prepared-template support to arbitrary boolean expression filters. `Predicate` still exists as a narrower derived IR for pushdown and fast-path planning.
