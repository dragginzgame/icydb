This document aggregates detailed release notes for the `0.40.x` minor line.

## 0.40.24 - 2026-03-05 - Route Shape Dispatch and Access Capability Consolidation

### 📝 Summary

`0.40.24` continues execution-routing simplification by converting scalar route
execution-stage derivation into shape-dispatch form, moving pushdown/index-range
eligibility interpretation behind one access-owned route capability contract, and
keeping continuation-envelope invariants fail-closed in debug builds.
Runtime query behavior remains unchanged.

### 🔧 Changed

- Reduced central router execution branching in
  `crates/icydb-core/src/db/executor/route/planner/execution/mod.rs`:
  - execution stage now derives `RouteShapeKind` and dispatches to shape-local
    builders only.
  - scalar execution-stage dispatch now treats mutation shape as unreachable and
    enforces a scalar-shape debug invariant guard.
  - retired legacy execution-case parity shim after route matrix soak.
- Expanded shape-local execution-stage builders under:
  - `crates/icydb-core/src/db/executor/route/planner/execution/shape_load.rs`
  - `crates/icydb-core/src/db/executor/route/planner/execution/shape_aggregate_count.rs`
  - `crates/icydb-core/src/db/executor/route/planner/execution/shape_aggregate_non_count.rs`
  - `crates/icydb-core/src/db/executor/route/planner/execution/shape_aggregate_grouped.rs`
- Consolidated route pushdown/index-range eligibility interpretation under
  `AccessRouteClass` in
  `crates/icydb-core/src/db/access/execution_contract.rs`:
  - `secondary_order_pushdown_applicability(...)`
  - `index_range_limit_pushdown_shape_eligible_for_order(...)`
  - route callsites in `executor/route/pushdown.rs` and
    `executor/route/capability.rs` now delegate to access-owned capability
    methods instead of re-deriving shape compatibility locally.
- Kept aggregate fast-path policy ownership out of router-stage shape dispatch:
  - route capability derivation consumes aggregate execution policy payloads
    from `crates/icydb-core/src/db/executor/aggregate/capability.rs`.
- Hardened continuation-envelope development invariants in
  `crates/icydb-core/src/db/index/envelope.rs`:
  - `resume_bounds_from_refs(...)` now asserts anchor containment inside the
    original envelope in debug builds.
  - bounded lower/upper key ordering is asserted before continuation rewrite.
- Audited continuation edge behavior `anchor == upper`:
  - continuation rewrite still yields an empty resume envelope and short-circuits
    before storage-range iteration in index scan traversal paths.

### ✅ Verification

- `cargo fmt --all`
- `cargo check -p icydb-core`
- `cargo test -p icydb-core route_feature_budget_shape_kinds_stay_within_soft_delta -- --nocapture`
- `cargo test -p icydb-core route_capabilities_ -- --nocapture`

## 0.40.23 - 2026-03-05 - Fluent/Predicate Module Split and Canonical Fingerprint Hardening

### 📝 Summary

`0.40.23` continues `0.40` structural decomposition by splitting large fluent
and predicate validation files into focused modules, centralizing
cursor/order execution policy behind a planner-owned execution-order contract,
and tightening predicate fingerprint determinism through shared encoding and
canonical normalization boundaries. Runtime query behavior stays stable.

### 🔧 Changed

- Decomposed fluent load API ownership:
  - removed monolithic `crates/icydb-core/src/db/query/fluent/load.rs`.
  - added focused module layout:
    - `crates/icydb-core/src/db/query/fluent/load/mod.rs`
    - `crates/icydb-core/src/db/query/fluent/load/builder.rs`
    - `crates/icydb-core/src/db/query/fluent/load/terminals.rs`
    - `crates/icydb-core/src/db/query/fluent/load/pagination.rs`
    - `crates/icydb-core/src/db/query/fluent/load/validation.rs`
  - preserved external fluent surface via `query::fluent::load::{FluentLoadQuery,
    PagedLoadQuery}` re-exports in `load/mod.rs`.
- Added planner-owned execution ordering contract and routed continuation through
  that authority:
  - added `crates/icydb-core/src/db/query/plan/order_contract.rs` with
    `ExecutionOrdering` and `ExecutionOrderContract`.
  - exported contract from `crates/icydb-core/src/db/query/plan/mod.rs`.
  - `ContinuationContract` now carries `execution_order_contract` for runtime
    consumers (`crates/icydb-core/src/db/query/plan/continuation.rs`).
- Moved cursor compatibility checks behind cursor module ownership:
  - added `crates/icydb-core/src/db/cursor/validation.rs`.
  - re-exported `validate_cursor_compatibility(...)` via
    `crates/icydb-core/src/db/cursor/mod.rs`.
  - cursor validation now consumes `ExecutionOrderContract` instead of
    independent grouped/order/direction policy flags.
- Switched runtime routing to contract-driven strategy/ordering:
  - `ExecutablePlan` now exposes `execution_strategy()` derived from the
    planner contract (`crates/icydb-core/src/db/executor/executable_plan.rs`).
  - session dispatch paths in `crates/icydb-core/src/db/session.rs` now use
    strategy/contract outputs and share cursor decode via
    `decode_optional_cursor_bytes(...)` instead of duplicated decode callsites.
  - traversal and load entrypoints consume planner direction/order contracts
    rather than deriving primary scan direction locally
    (`executor/load/{entrypoints,pk_stream,secondary_index}.rs`,
    `executor/route/mode.rs`, `executor/traversal.rs`).
- Decomposed predicate schema validation ownership:
  - removed monolithic `crates/icydb-core/src/db/predicate/schema.rs`.
  - added directory-owned modules:
    - `crates/icydb-core/src/db/predicate/schema/mod.rs`
    - `crates/icydb-core/src/db/predicate/schema/types.rs`
    - `crates/icydb-core/src/db/predicate/schema/model_checks.rs`
    - `crates/icydb-core/src/db/predicate/schema/errors.rs`
    - `crates/icydb-core/src/db/predicate/schema/validate.rs`
  - updated predicate callsites (`predicate/{fingerprint,normalize}.rs`) to
    consume schema semantics through the new module boundary.
- Consolidated duplicated predicate encoding logic into one shared authority:
  - added `crates/icydb-core/src/db/predicate/encoding.rs`.
  - moved shared predicate/value/coercion sort-key and fingerprint encoders out
    of `predicate/normalize.rs` and `predicate/fingerprint.rs`.
  - updated `crates/icydb-core/src/db/predicate/mod.rs` module wiring.
- Tightened encoder boundary visibility:
  - `encoding.rs` APIs are now `pub(in crate::db::predicate)` to keep encoder
    usage inside predicate authority only.
  - query/executor continue to consume predicate hashing through public
    predicate/fingerprint boundaries rather than direct encoding utilities.
- Enforced canonical predicate normalization through predicate hash API:
  - `crates/icydb-core/src/db/predicate/fingerprint.rs` now canonicalizes via
    `normalize(...)` before structural predicate hashing.
  - `crates/icydb-core/src/db/query/fingerprint/hash_parts.rs` consumes that
    canonical predicate-hash API and no longer performs independent
    normalization policy checks.
  - invariant is now API-owned:
    `fingerprint(normalize(predicate))`.
- Hardened map sort-key determinism in predicate encoding:
  - `encoding.rs` now canonicalizes `Value::Map` entries during sort-key
    encoding using canonical key/value ordering before emitting bytes.
  - this keeps sort-key/fingerprint stability resilient when callers construct
    `Value::Map(...)` directly in non-canonical order.
- Added invariant coverage:
  - `db::predicate::fingerprint::tests` lock canonicalized `AND`/`OR` hash
    order-insensitivity while preserving raw structural order-sensitivity.
  - `db::predicate::encoding::tests::predicate_sort_key_normalizes_map_entry_order`
    locks map-entry order invariance for predicate sort keys.
  - `db::query::fingerprint::fingerprint::tests::fingerprint_and_signature_are_stable_for_reordered_and_non_canonical_map_predicates`
    locks end-to-end fingerprint and continuation-signature stability for
    combined `AND` child reorder + non-canonical map entry order inputs.
- Verification:
  - `cargo check -p icydb-core`
  - `cargo test -p icydb-core --lib`
  - `cargo test -p icydb-core db::predicate::fingerprint::tests -- --nocapture`
  - `cargo test -p icydb-core db::predicate::encoding::tests -- --nocapture`
  - `cargo test -p icydb-core db::query::fingerprint -- --nocapture`
  all pass after decomposition and canonicalization hardening.

## 0.40.20 - 2026-03-04 - Query Intent Approval and Planning DTO Boundaries

### 📝 Summary

`0.40.20` completes the QueryIntent AST migration work and closes the
`query/intent` architecture review as approved. The patch keeps runtime query
behavior stable while tightening intent-to-plan boundaries with explicit
stage DTOs and plan-owned logical assembly contracts.

### 🔧 Changed

- Finalized owned intent state in
  `crates/icydb-core/src/db/query/intent/intent_ast.rs`:
  - `QueryIntent<K>` now owns mode state, key-access conflict tracking, grouped
    declarations, and delete policy flags.
  - `QueryModel` now holds a single `intent: QueryIntent<K>` instead of a
    scalar/grouped state bag.
  - intent mutation and policy checks stay centralized in intent-owned helpers
    (`append_predicate`, order/group/having/key-access setters, and
    `validate_policy_shape`).
- Removed intent-local mode aliases and made mode/spec ownership explicit:
  - `DeleteSpec`, `LoadSpec`, and `QueryMode` are consumed from `query::plan`
    directly (with plan-root re-exports preserved for API stability).
- Moved planning-stage orchestration to explicit plan-owned modules:
  - `crates/icydb-core/src/db/query/plan/access_planner.rs` now owns
    predicate normalization + access planning inputs via
    `AccessPlanningInputs<'_>`.
  - `crates/icydb-core/src/db/query/plan/logical_builder.rs` now owns logical
    planning inputs via `LogicalPlanningInputs`, `LogicalQuery` construction via
    `logical_query_from_logical_inputs(...)`, and final logical plan assembly.
  - consolidated logical planning into `logical_builder.rs` and removed the
    temporary `query/plan/intent_adapter.rs` split.
- Kept `QueryModel::build_plan_model(...)` as a thin 3-phase orchestrator:
  1. schema + intent policy validation,
  2. normalized predicate + access path selection,
  3. logical input projection + logical plan assembly.
- Preserved deterministic order canonicalization ownership at logical assembly:
  - primary-key tie-break normalization remains centralized in
    `query/plan/logical_builder.rs`.
- Added and retained guardrails while hardening this boundary:
  - `cargo check -p icydb-core`
  - `cargo test -p icydb-core db::query::intent::tests -- --nocapture`
  - `cargo test -p icydb-core --test compile_fail -- --nocapture`
  all pass for the `0.40.20` surface.
- Architecture status:
  - `query/intent` review approved.
  - intent/planner boundaries now use stage-specific DTOs rather than
    cross-layer state peeking.
