## 0.73.1

`0.73.1` is the first real grouped route-admission widening patch on top of the
`0.73.0` grouped-strategy foundation. It admits the first grouped field-target
aggregate family on the same planner-owned ordered-group proof model instead of
treating grouped numeric aggregates as a separate planning branch.

What landed in this patch:

- grouped ordered-group proof now accepts both `IndexPrefix` and `IndexRange`
  access when the chosen access path already preserves group-key order in:
  [grouped_strategy.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/grouped_strategy.rs)
- fully indexable grouped predicates no longer force an immediate hash-group
  fallback; grouped planning now fails closed only on real residual predicates
  that break the grouped order contract
- the first admitted grouped field-target family now shares one grouped
  admission framework:
  - `COUNT(field)`
  - `SUM(field)`
  - `AVG(field)`
- grouped reducer state now carries the minimal numeric state needed by the
  admitted field-target family:
  - `SUM(field)` stays on decimal accumulation with null skipping
  - `AVG(field)` carries `sum + row_count` and finalizes through the shared
    decimal-average helper
- grouped planning, validation, and grouped fold initialization now admit those
  field-target aggregates under the same ordered-group proof instead of
  hard-coding `AVG(field)` as a separate blocked family in:
  [group_model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_model.rs),
  [rules.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/grouped/policy/rules.rs), and
  [engine_init.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/grouped_fold/engine_init.rs)
- the admitted grouped public SQL cohort is now locked for both order-only and
  simple equality-prefix filtered shapes in:
  [sql_grouped.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_grouped.rs)
- generated-vs-typed grouped SQL parity is now locked for grouped `COUNT(*)`,
  `COUNT(field)`, `SUM(field)`, and `AVG(field)` explain/execution surfaces in:
  [tests.rs](/home/adam/projects/icydb/canisters/test/sql_parity/src/tests.rs)
- the same grouped public query-lane mirrors are now present on the PocketIC
  surface in:
  [sql_canister.rs](/home/adam/projects/icydb/testing/pocket-ic/tests/sql_canister.rs)

What did not change:

- no grouped cost model
- no grouped `DISTINCT` widening
- no join or relation planning
- no new grouped execution family beyond the current ordered-vs-hash grouped
  split
- no change to the grouped order-proof model itself; aggregate-family widening
  now sits on the same planner-owned admission framework instead of inventing a
  second grouped numeric policy

So `0.73.1` should be read as the first grouped admission patch, not a grouped
numeric side branch:

- planner still owns grouped order proof and grouped fallback reasons
- ordered grouped admission now covers the first field-target aggregate family
- public explain and parity surfaces now lock that same grouped route story

## 0.73.0

`0.73.0` starts the grouped aggregate planning line by making grouped strategy
the single planner-owned behavioral source for grouped planning, grouped route
projection, grouped fingerprints, and grouped `EXPLAIN`, without widening
grouped route admission yet.

What landed in this patch:

- grouped planning now carries one explicit
  `GroupedPlanStrategy` artifact instead of treating grouped shape as a loose
  planner hint in:
  [grouped_strategy.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/grouped_strategy.rs),
  [group.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs), and
  [mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/mod.rs)
- the legacy grouped hint is no longer a peer abstraction on the public
  planner/explain path; grouped explain and grouped fingerprint projection now
  consume `GroupedPlanStrategy` directly in:
  [plan.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/plan.rs) and
  [hash_parts.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/hash_parts.rs)
- grouped planning now carries stable planner-authored fallback reasons instead
  of collapsing every non-ordered grouped shape into an unreasoned hash-group
  fallback:
  - `distinct_grouping_not_admitted`
  - `residual_predicate_blocks_grouped_order`
  - `aggregate_streaming_not_supported`
  - `having_blocks_grouped_order`
  - `group_key_order_unavailable`
- grouped route observability now keeps planner fallback reason and route-gate
  rejection reason separate in:
  [observability.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/route/contracts/execution/observability.rs) and
  [plan.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/route/contracts/execution/plan.rs)
- grouped execution descriptors now project from the grouped planner handoff
  instead of rebuilding a scalar load route for grouped plans:
  [load.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/load.rs)
- grouped `EXPLAIN EXECUTION` now emits explicit grouped aggregate execution
  nodes:
  - `GroupedAggregateHashMaterialized`
  - `GroupedAggregateOrderedMaterialized`
- grouped execution descriptor roots and grouped aggregate nodes now both carry
  the route-owned grouped projection:
  - `grouped_route_outcome`
  - `grouped_route_rejection_reason`
  - `grouped_plan_fallback_reason`
  - `grouped_route_eligible`
  - `grouped_execution_strategy`
- the grouped public explain surface is now locked at both core and facade
  layers in:
  [sql_grouped.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_grouped.rs) and
  [tests.rs](/home/adam/projects/icydb/crates/icydb/src/db/session/tests.rs)
- grouped route and semantics snapshots now freeze the planner-to-route and
  route-to-explain contracts in:
  [tests.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/route/tests.rs) and
  [semantics.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/tests/semantics.rs)

What did not change:

- no grouped route-admission widening beyond the already admitted shapes
- no grouped cost model
- no join or relation planning
- no grouped `DISTINCT` redesign
- no new grouped execution family beyond the current ordered-vs-hash materialized split

So `0.73.0` should be read as the grouped ownership and projection foundation
patch:

- planner derives grouped strategy once
- route/runtime consume that grouped strategy
- `EXPLAIN` projects that grouped strategy and its fallback reason
- downstream surfaces stop re-deriving grouped policy locally
