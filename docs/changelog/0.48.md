# IcyDB 0.48

This document aggregates detailed release notes for the `0.48.x` minor line.

## 0.48.5 - 2026-03-11 - Value/DB Boundary Cleanup and Error Module Split

### Summary

`0.48.5` continues `0.48` boundary tightening by removing one remaining
`value -> db` dependency and restructuring DB-only error construction into
subsystem-scoped modules.

### Changed

- Moved `StorageKey` ownership from `db::data` into `value`:
  - `crates/icydb-core/src/value/storage_key.rs` is now the authority.
  - `value::mod` re-exports `StorageKey`, `StorageKeyDecodeError`, and
    `StorageKeyEncodeError`.
  - `db::data` now consumes/re-exports storage-key contracts from `value`
    instead of owning the implementation.
- Removed DB-specific error constructor coupling from `error::InternalError`
  for executor/query/planner/cursor mapping helpers and moved those helpers to
  `db::error`.
- Split `db::error` into focused subsystem modules while preserving the
  existing `crate::db::error::*` call surface through `mod.rs` re-exports:
  - `db/error/query.rs`
  - `db/error/planner.rs`
  - `db/error/cursor.rs`
  - `db/error/executor.rs`
  - `db/error/access.rs`
- Updated DB runtime callsites to use `crate::db::error::*` constructors and
  mappings, and adjusted non-DB usage (`visitor`) to construct taxonomy via
  `InternalError::classified(...)` directly.

### Testing

- `cargo fmt --all`
- `cargo check -p icydb-core`
- `cargo test -p icydb-core error::tests -- --nocapture`

## 0.48.4 - 2026-03-11 - Explain Module Split, Schema Consolidation, and JSON Writer

### Summary

`0.48.4` continues the `0.48` explain-focused cleanup by reducing EXPLAIN module
surface complexity, consolidating schema-contract ownership under one runtime
module, and standardizing deterministic manual JSON emission through one writer.

### Changed

- Split `db::query::explain` into focused modules with `mod.rs` as a thin
  boundary (`plan`, `execution`, `render`, `access_projection`, `writer`) so
  projection, descriptor, rendering, and adapter responsibilities no longer
  accumulate in one growing file.
- Replaced ad-hoc EXPLAIN JSON object assembly with a minimal deterministic
  `JsonWriter` in `query::explain::writer`, centralizing comma/field sequencing
  and typed field emission while preserving canonical output order.
- Consolidated runtime schema-contract utilities into `db::schema` ownership
  (including describe/format, schema fingerprint authority, schema info/types,
  and validation paths) and removed transitional shim layers so callers import
  directly from the schema boundary.
- Renamed runtime observability module ownership from `obs` to `metrics` and
  collapsed the old nested `obs/metrics/*` layout into one focused
  `metrics/{mod,state,sink,tests}.rs` surface.

### Breaking

- Advanced metrics API paths were renamed:
  - `icydb::obs::EventReport` -> `icydb::metrics::EventReport`
  - `icydb::obs::metrics_report(...)` -> `icydb::metrics::metrics_report(...)`
  - `icydb::obs::metrics_reset_all()` -> `icydb::metrics::metrics_reset_all()`
  - `icydb_core::obs::*` -> `icydb_core::metrics::*`

## 0.48.3 - 2026-03-10 - Remove DB Error Wrapper Layer

### Summary

`0.48.3` removes an internal DB-only error-wrapper module and routes invariant
construction directly through the canonical `InternalError` constructors.

### Changed

- Removed `db::error` wrapper modules:
  - `crates/icydb-core/src/db/error/mod.rs`
  - `crates/icydb-core/src/db/error/invariant.rs`
- Updated DB runtime callsites to construct invariants directly via
  `crate::error::InternalError`:
  - `executor_invariant(msg)` wrappers were replaced with
    `InternalError::query_executor_invariant(msg)`.
  - `cursor_invariant(msg)` wrappers were replaced with
    `InternalError::cursor_invariant(InternalError::executor_invariant_message(msg))`.
  - `planner_invariant(msg)` wrappers were replaced with
    `InternalError::planner_invariant(InternalError::executor_invariant_message(msg))`.
- Removed DB root module wiring for the deleted wrapper layer.
- Preserved invariant message formatting and error taxonomy behavior; this patch
  is an indirection-removal cleanup only.

### Testing

- `cargo fmt --all`
- `cargo check -p icydb-core`
- `cargo test -p icydb-core`

## 0.48.2 - 2026-03-10 - Explain Diagnostics and Codegen Snapshot Fix

### Summary

`0.48.2` continues the explain-focused follow-up with additive execution
diagnostics for scalar index routes and includes a DB codegen bug fix for the
generated storage snapshot endpoint.

### Changed

- Scalar execution explain roots now project planner/runtime-derived diagnostic
  metadata for:
  `covering_scan_reason`, `scan_direction`, `continuation_mode`,
  `resume_from`, `fast_path_selected`, `fast_path_selected_reason`, and
  `fast_path_rejections`.
- Scalar load explain covering eligibility now reflects planner-owned
  index-covering existing-row contracts instead of a constant fallback value.
- Cursor resume nodes now expose the same continuation-mode and resume-source
  metadata projected on access root nodes.
- Strict text/json explain snapshots and matrix assertions were updated to lock
  the new additive metadata surface.

### Fixed

- DB codegen for the generated `icydb_snapshot()` endpoint now calls
  `db().storage_report(&[])` instead of directly invoking
  `::icydb::obs::storage_report(&DB, &[])`, preventing incorrect or unavailable
  global DB handle usage in generated canister code.

## 0.48.1 - 2026-03-10 - Explain Small Features and Access-Choice Diagnostics

### Summary

`0.48.1` extends the post-boundary explain slice with additive execution-node
metadata and planner-derived access-choice diagnostics, while keeping runtime
query behavior unchanged.

### Changed

- Scalar load execution roots now expose additive index-route metadata:
  `prefix_len`, pushed `fetch` budgets, explicit `covering_scan=false`, and
  access-choice diagnostics (`access_choice_chosen`,
  `access_choice_chosen_reason`, `access_choice_alternatives`,
  `access_choice_rejections`).
- ORDER BY execution nodes now report explicit sort-elimination state through
  `order_satisfied_by_index=true|false`.
- Predicate stage descriptors now expose pushed predicate summaries for strict
  prefilter nodes and pushed-access vs residual split visibility for residual
  filter nodes.
- Access-choice alternatives now come from planner-candidate snapshots rather
  than model-only index hints.
- Access-choice reason construction was tightened with planner-owned typed
  reason enums, projected to stable explain reason-code strings.
- Access-choice projection now derives chosen-route shape through
  `ExplainAccessPath` projection so raw `AccessPath` variant references remain
  constrained to `db::access` boundary modules.

### Testing

- `cargo check -p icydb-core`
- `cargo clippy -p icydb-core --lib -- -D warnings`
- `cargo test -p icydb-core session_load_explain_execution_ -- --nocapture`

## 0.48.0 - 2026-03-10 - Explain Boundary and Plan Identity Hardening

### Summary

`0.48.0` finalizes the explain-boundary hardening slice by making planner-owned
contracts the authority for plan identity and continuation signatures, while
keeping `EXPLAIN` as a deterministic presentation surface.

### Changed

- Plan fingerprint and continuation-signature derivation now hash directly from
  planner-owned query/access/grouping contracts instead of requiring explain DTO
  projection in production paths.
- Plan-hash call sites now consume planned-query fingerprint authority directly
  (`Query`, planned-query, compiled-query, and session trace surfaces).
- Cursor continuation guard ownership is consolidated under cursor contracts:
  envelope validation, resume-bound derivation, and directional advancement are
  now exposed through cursor-owned APIs consumed by index scan execution.
- Cursor-anchor range-bound reason mapping moved to cursor ownership; access
  lowering no longer owns cursor-anchor bound-mapping variants.
- `EXPLAIN` behavior remains stable while production identity paths no longer
  depend on explain-only helpers; targeted explain/session parity suites are
  rerun and green.
- Structural ownership language was aligned across docs/tests so plan-hash
  semantics explicitly reference planner contracts.
