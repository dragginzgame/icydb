# IcyDB 0.211: Accepted Constraint Catalog

## 0.211.2

### ⚠️ Breaking

- Hard-cuts the persisted accepted V1 `CHECK` expression language to the
  operations that generated and SQL declarations can actually produce. Any
  development-only accepted snapshot containing the retired database-level
  divisibility node is unsupported; no alias, decoder, or translation remains.
- Caps one canonical accepted entity-schema snapshot at 512 KiB on both encode
  and decode. Oversized development schemas must be reduced and recreated
  rather than loaded through a compatibility path.

### 🔧 Changed

- Moves the schema-snapshot byte limit into the canonical codec so normal
  persistence, unchecked corruption fixtures, and durable decoding enforce one
  bound at the actual serialization boundary.
- Keeps application-level Rust `MultipleOf` validation unchanged. Only the
  unreachable database-owned accepted-expression representation is removed.

### 🩹 Fixed

- Corrects the maintained SQL contract to match runtime behavior:
  `ALTER TABLE ... DROP COLUMN ...` is admitted only for an exactly empty
  entity, publishes the new dense schema without a row rewrite, and rejects
  nonempty entities.

### 🗑️ Removed

- Removes the unused accepted divisibility AST and wire variants, binder,
  compiler, evaluator, renderer, errors, arbitrary-precision helpers, and tests
  that existed only for that unreachable database path.

### 📊 Audit

- Before release-note additions, the bounded closeout correction touches 16
  files at `+149/-315`, a net reduction of 166 lines, without adding an
  authority owner or compatibility machinery.
- The exact current SQL performance Wasm is 4,689,951 raw non-gzipped bytes,
  9,829 bytes smaller than the pre-correction closeout subject. Checked steady
  writes remain within the maintained 5% focused overhead guard.

### 🧪 Testing

- Focused maintained-language, canonical schema-byte-bound, SQL contract,
  native/Wasm feature, warnings-denied Clippy, architecture, and diff-hygiene
  evidence passes. Full release validation remains human-owned.

## 0.211.1

### ⚠️ Breaking

- Hard-cuts development accepted-schema bytes to the sole current version-1
  `ICYZ` profile. Stores created by `0.211.0` must be recreated; no
  compatibility decoder, translation, or migration path remains.
- Hard-cuts secondary-index keys to include the exact physical generation
  beside entity and ordinal identity. Existing development index bytes are not
  decoded or translated, and each materialized index entry grows by eight raw
  key bytes.
- Changes the public Candid `Error` and constraint-validation finding shapes to
  carry `ConstraintDiagnostic` directly. Downstream Candid declarations must
  be regenerated, and the removed field-ID finding DTO is not retained as an
  alias.

### ➕ Added

- Adds one accepted structural constraint registry covering primary-key,
  not-null, unique-index, relation, and bounded canonical `CHECK` identity,
  projected consistently through typed descriptions and SQL introspection.
- Adds fingerprint-bound accepted `CHECK` evaluation over final mutation
  after-images while retaining Rust validators as application-only behavior.
- Adds marker-owned, bounded Forward/Verify activation for `CHECK`, not-null,
  unique-index, and relation constraints, including durable findings,
  revision invalidation, recovery reconstruction, and atomic promotion or
  abort.
- Adds administrative SQL constraint lifecycle operations with paged
  validation and typed findings over the same accepted activation authority.

### 🔧 Changed

- Candidate unique-index and relation generations remain planner-invisible
  while live writes maintain their staged state and explicit write/delete
  barriers protect validation.
- Generated reconciliation preserves unrelated DDL-owned accepted state and
  rejects proposal drift during an active constraint transition.
- Accepted constraint failures retain stable ID, name, kind, entity, primary
  key, field paths, context, class, and code through core, facade, SQL, Candid,
  and CLI boundaries.
- Accepted-native integrity projection reuses existing constraint evaluators
  and key projectors rather than introducing another scanner.
- Restores required `ADD COLUMN` for an exactly empty entity while keeping
  generic reconciliation unable to advance the history floor without the
  operation-specific emptiness proof.

### 🩹 Fixed

- Retains the lazy accepted `SchemaInfo` projection alongside the compiled
  accepted-row constraint program, so immediate SQL aggregate, explain, write,
  update-selector, and DDL execution reuse one revision-checked catalog context
  instead of rebuilding equivalent schema views.

### 🗑️ Removed

- Removes synchronous scan-and-publish constraint creation, planner-visible
  candidate generations, mismatched generated-activation preservation, and
  duplicate public constraint-finding DTOs.

### 🧪 Testing

- Focused activation, recovery, generated-reconciliation, mutation-ingress,
  SQL introspection, diagnostic Candid, and CLI rendering coverage exercises
  the maintained current architecture.
- Recorded performance evidence keeps steady accepted-`CHECK` overhead below
  2% and delete instructions below the released `0.211.0` baseline; full
  release validation and raw Wasm measurement remain part of the human-owned
  push workflow.

## 0.211.0

### ⚠️ Breaking

- Hard-cuts accepted entity snapshots to the sole current version-1 `ICYU`
  profile. Development stores carrying the preceding accepted-schema profile
  must be recreated; no compatibility reader or migration shim remains.

### ➕ Added

- Adds persisted non-reusing constraint-ID high-water state and stable logical
  identities for secondary indexes and relations.
- Separates logical index identity from the dense physical ordinal used by
  index-key storage, allowing physical compaction without changing a surviving
  index's catalog identity.

### 🔧 Changed

- Initial generated schema proposals assign deterministic logical identities,
  while accepted reconciliation and schema mutations preserve stored identity.
- Accepted admission and runtime/cache fingerprints now include constraint-ID
  allocator state plus logical index and relation identities.
- All active internal format and SQL correctness evidence identities remain at
  version 1 under the pre-1.0 hard-cut policy.

### 🧪 Testing

- Covers current schema round trips, malformed identity rejection, physical
  ordinal compaction, generated index rename association, and fingerprint
  sensitivity.
- Raw non-gzipped SQL performance Wasm measured 3,967,488 bytes, an increase of
  3,098 bytes (+0.0782%) over the final 0.210.2 baseline. No row, index-key,
  relation-key, cursor, or journal format changed.
