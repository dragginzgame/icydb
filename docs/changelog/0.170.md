# IcyDB 0.170 Changelog

## 0.170.1

### 🔧 Changed

- Hard-cuts runtime store handles onto explicit
  `StoreRuntimeStorageCapabilities` so allocation identity, durability,
  recovery, commit participation, schema metadata persistence, relation
  source/target eligibility, and live validation travel with the registered
  store handle.
- Updates generated stable and heap store registration to pass the matching
  runtime capability descriptor while keeping stable allocation identity present
  and heap allocation identity absent.
- Extends storage snapshot DTOs and CLI rendering with additive capability
  metadata columns for allocation, durability, commit participation, recovery,
  and schema metadata persistence.

### 🧪 Testing

- Proves stable and heap registration bind the expected runtime capability
  axes.
- Proves the registry rejects mismatched allocation identity and capability
  decisions.
- Updates diagnostics and Candid-shape tests for the new capability metadata.

## 0.170.0

### ➕ Added

- Adds the storage capability design for mixed-durability contracts.
- Adds schema-owned storage capability descriptors derived from
  `StoreStorage::Stable` and `StoreStorage::Heap`.
- Models allocation identity, durability, recovery, commit participation,
  schema metadata persistence, relation source/target eligibility, and live
  validation as explicit capability axes.

### 🔧 Changed

- Marks the 0.170 design as in progress now that implementation has started.
- Clarifies that `storage_mode` is diagnostic/display metadata; policy code is
  expected to consume capability axes instead of matching on stable/heap mode.
- Keeps runtime behavior unchanged for this initial model/projection slice.

### 🧪 Testing

- Adds schema-model coverage for stable and heap capability derivation.
- Proves storage capabilities are separate from stable allocation identity.
- Proves policy-style consumers can read capability axes rather than
  `storage_mode`.
