# IcyDB 0.170 Changelog

## 0.170.3

### 🔧 Changed

- Classifies mutation commit plans from affected store-handle capabilities as
  durable-only, live-only, or mixed durable/live.
- Records commit-plan classifications through the existing mutation metrics
  path so mixed stable+heap writes do not look like atomic durable commits.
- Keeps the classification internal/proof-facing; no new public write response
  contract is added.

### 🧪 Testing

- Proves stable public writes emit durable-only classifications.
- Proves heap public writes emit live-only classifications.
- Proves heap-source writes that validate against stable targets emit mixed
  durable/live classifications.
- Proves failed mixed writes reject before applying heap side effects and emit
  no commit classification.

## 0.170.2

### 🧹 Cleanup

- Replaces the parser-style `__icydb_test_field_model!` macro with explicit
  `TestFieldModelOptions` and a small `test_field_model` helper.
- Removes the hidden `__icydb_test_parse_entity!` layer so `test_entity!`
  forwards directly into the normalized entity emitter.
- Keeps `__icydb_test_emit_entity!` as the single entity emission path for
  marker traits, model storage, placement, runtime surface, `EntityValue`, and
  singleton behavior.
- Keeps test relation declarations field-name based and preserves the existing
  topology split between canister, store, entity, field, and relation helpers.

### 📊 Audit

- Confirms there are no active `test_entity_schema!` Rust call sites.
- Confirms no `crate::__icydb_test_*` helper is used outside
  `crates/icydb-core/src/testing/fixtures.rs`.
- Confirms the old field parser, entity parser, and
  `primary_key(fields = ...)` syntax are gone.
- Reduces `fixtures.rs` from 984 lines to 817 lines while preserving
  schema/model/runtime semantics.

### 🧪 Testing

- Runs focused schema, relation, mutation-save, executor, session, commit,
  diagnostics, registry, workspace check, and clippy validation for the macro
  cleanup.

## 0.170.1

### 🔧 Changed

- Hard-cuts runtime store handles onto explicit
  `StoreRuntimeStorageCapabilities` so allocation identity, durability,
  recovery, commit participation, schema metadata persistence, relation
  source/target eligibility, and live validation travel with the registered
  store handle.
- Updates generated stable and heap store registration to pass the matching
  runtime capability descriptor while keeping stable allocation identity present
  and heap allocation identity absent.
- Extends storage snapshot DTOs and CLI rendering with additive capability
  metadata columns for allocation, durability, commit participation, recovery,
  and schema metadata persistence.

### 🧪 Testing

- Proves stable and heap registration bind the expected runtime capability
  axes.
- Proves the registry rejects mismatched allocation identity and capability
  decisions.
- Updates diagnostics and Candid-shape tests for the new capability metadata.

## 0.170.0

### ➕ Added

- Adds the storage capability design for mixed-durability contracts.
- Adds schema-owned storage capability descriptors derived from
  `StoreStorage::Stable` and `StoreStorage::Heap`.
- Models allocation identity, durability, recovery, commit participation,
  schema metadata persistence, relation source/target eligibility, and live
  validation as explicit capability axes.

### 🔧 Changed

- Marks the 0.170 design as in progress now that implementation has started.
- Clarifies that `storage_mode` is diagnostic/display metadata; policy code is
  expected to consume capability axes instead of matching on stable/heap mode.
- Keeps runtime behavior unchanged for this initial model/projection slice.

### 🧪 Testing

- Adds schema-model coverage for stable and heap capability derivation.
- Proves storage capabilities are separate from stable allocation identity.
- Proves policy-style consumers can read capability axes rather than
  `storage_mode`.
