# 0.162.x

## 0.162.9

### 🔧 Changed

- Moved aggregate, grouped-runtime, cursor-anchor, and retained-row executor
  decode paths behind explicit scalar fail-closed bridges. Scalar
  `StorageKey` access is now limited to intentionally scalar aggregate output
  boundaries and other explicit bridge points.
- Added ordered primary-key fields to schema observability payloads so CLI
  schema rendering can display composite primary keys as ordered field lists.
- Updated schema-check drift analysis to compare generated and accepted
  primary-key fields structurally instead of comparing only a rendered scalar
  label.
- Clarified the 0.162 design audit requirements so observability, performance,
  instruction, wasm-size, storage-size, commit/recovery, and index-footprint
  audits include composite primary-key fixtures once the audited surface can
  construct them.

### 🧪 Testing

- Added focused CLI schema and schema-check coverage for composite primary-key
  reporting, rechecked schema description Candid shape, and reran focused
  core/CLI validation plus clippy and diff hygiene for the observability and
  executor bridge slice.

## 0.162.8

### 🔧 Changed

- Moved forward-index mutation planning and accepted index-key builders onto
  scalar-or-composite `PrimaryKeyValue` row identity, keeping scalar
  `StorageKey` access only at explicit fail-closed bridge points.
- Updated unique-index preflight readers and index plan/read ports to return
  `PrimaryKeyValue`, so duplicate detection and accepted-index publication no
  longer assume scalar row identity.
- Changed commit index-delta materialization to rebuild index row identity from
  the planned `PrimaryKeyValue` instead of re-decoding scalar storage keys.
- Updated ordered key-stream witnesses and stream monotonicity checks to use
  scalar-or-composite row identity consistently.
- Moved covering projection primary-key reads, materialized projection decode,
  bytes terminals, materialized ranking, and kernel row decode through
  composite-aware data-key boundaries. Paths that still require scalar row-slot
  decoding now fail closed through the scalar bridge instead of using direct
  scalar accessors.

### 🧪 Testing

- Rechecked focused data-key, composite-covering, recovery, key-stream,
  workspace/core checking, core clippy, formatting, and diff hygiene coverage
  for the row-identity planning/executor boundary slice.

## 0.162.7

### 🔧 Changed

- Moved accepted schema snapshots and schema-store encoding onto ordered
  primary-key field IDs, preserving composite primary-key arity and field order
  in persisted schema metadata.
- Updated schema runtime descriptors and owned row-decode contracts to carry
  ordered primary-key names, kinds, and slot indices while keeping explicit
  scalar bridge accessors for execution paths that have not yet admitted
  composite row identity.
- Changed schema info and schema description surfaces to mark every accepted
  primary-key component and expose ordered primary-key names from accepted
  schema authority.
- Updated schema reconciliation, transition, and mutation publication paths to
  compare ordered primary-key field identities instead of only the first
  primary-key field.
- Moved schema fingerprints onto ordered generated primary-key fields so
  primary-key arity and component identity affect the commit/schema contract.
- Tightened schema-driven SQL admission so DDL primary-key-field drop checks and
  SQL UPDATE deterministic ordering use the full accepted primary-key field
  list.
- Moved schema index-rebuild staging and startup rebuild adapters onto
  scalar-or-composite `PrimaryKeyValue` row identity so DDL-created indexes can
  materialize composite primary-key suffixes.
- Updated index row witnesses to carry scalar-or-composite primary-key identity
  from the raw index key, leaving scalar `StorageKey` access as a fail-closed
  bridge for paths that have not admitted composite identity yet.
- Updated executor index scans and integrity diagnostics to rebuild/compare row
  identity from `PrimaryKeyValue` instead of assuming every index entry decodes
  to a scalar storage key.

### 🧪 Testing

- Added focused schema codec, schema info, runtime descriptor, DDL, SQL
  lowering, fingerprint, rebuild-staging, and index-entry witness coverage for
  ordered scalar-or-composite primary-key identity.

## 0.162.6

### 🔧 Changed

- Moved decoded data-store keys onto the scalar-or-composite primary-key value
  model, allowing raw data-key decode and bounds to preserve composite primary
  keys while existing scalar-only consumers fail closed.
- Extended index primary-key suffix decoding through the same
  composite-aware boundary so index entries can carry composite row identity
  without duplicating primary-key bytes in values.
- Opened derive support for `pk(fields = ["a", "b"])` composite declarations,
  generating public composite key types with key codec contracts and keeping
  composite keys external-only.
- Hard-cut `PrimaryKeyCodec` onto primary-key-owned encode errors instead of
  the old scalar storage-key encoder error.
- Tightened composite primary-key declaration validation for duplicate fields,
  maximum arity, and disallowed `Unit` components.

### 🧪 Testing

- Added derive and compile-fail coverage for composite primary-key declaration
  validation, generated key contracts, scalar-only fail-closed boundaries, and
  composite-aware raw key decoding.

## 0.162.5

### 🔧 Changed

- Split the compact key taxonomy into scalar `PrimaryKeyComponent` values and
  scalar-or-composite `PrimaryKeyValue` values, matching the 0.162 design shape.
- Tightened `CompositePrimaryKeyValue` construction so composite values require
  `2..=MAX_PRIMARY_KEY_FIELDS` components; `pk(fields = ["id"])` remains the
  scalar case and must not encode as a one-component composite.
- Moved `PrimaryKeyCodec` / `PrimaryKeyDecode` signatures onto
  `PrimaryKeyValue`, with scalar row routing still explicitly unwrapping only
  scalar components until `DecodedDataStoreKey` stores composite-aware identity.
- Added hidden facade re-exports for the primary-key value taxonomy so generated
  composite key structs have a stable codegen target in later 0.162 slices.
- Kept raw data and index store-key wrappers able to preserve composite
  primary-key suffix bytes without yet routing persisted rows through composite
  keys.

### 🧪 Testing

- Rechecked focused key-taxonomy coverage, workspace checking, core clippy, and
  diff hygiene for the trait/taxonomy bridge slice.

## 0.162.0

### 🔧 Changed

- Started the composite-primary-key release line with an implementation design
  that keeps accepted schema snapshots as runtime authority for primary-key
  arity, field order, durable field IDs, row slots, component kinds, and
  primary-key source.
- Defined the unified primary-key declaration direction for 0.162:

  ```rust
  #[entity(store = Store, pk(fields = ["id"]))]
  #[entity(store = Store, pk(fields = ["battle_id", "entity_id"]))]
  ```

- Documented the compact primary-key storage direction: data-store keys remain
  `EntityTag || EncodedPrimaryKey`, scalar keys keep scalar encoding, composite
  keys use a fixed-capacity runtime value model, and ordinary encode/decode
  paths must remain allocation-free.
- Clarified composite-primary-key constraints before implementation:
  composites are external-only, omitted composite `source` resolves to
  `external`, `Unit` is scalar-only, generated key structs are public `E::Key`
  types, and primary-key field facts duplicated in accepted snapshots must
  validate against the accepted field snapshot.

### ⚠️ Breaking

- Hard-cut generated index attributes to formal field lists parsed by Darling
  instead of IcyDB's old comma-string helper:

  ```rust
  index(fields = ["battle_id", "cell_id"])
  index(fields = ["tenant_id", "LOWER(email)"])
  ```

  The old spelling now rejects:

  ```rust
  index(fields = "battle_id, cell_id")
  ```

### 🧪 Testing

- Rechecked schema-derive and macro compile-fail coverage for formal generated
  index field-list parsing and diagnostics, plus workspace checking, derive
  clippy, formatting, and diff hygiene for the parser/design slice.
