# 0.162.x

## 0.162.4

### 🔧 Changed

- Added explicit compact composite primary-key encoding and decoding support to
  the key taxonomy, including the `Composite` kind tag and component-count
  frame.
- Kept composite encoding on explicit side APIs for now, so persisted row
  encode/decode paths remain on the existing scalar primary-key bridge until the
  trait hard cut lands.
- Updated index-key decode error mapping so malformed composite key payloads
  fail closed when encountered at legacy scalar decode boundaries.

### 🧪 Testing

- Added focused composite encoding tests covering roundtrip, invalid component
  counts, `Unit` component rejection, nested composite rejection through scalar
  decode, trailing/error handling, and encoded ordering.

## 0.162.3

### 🔧 Changed

- Changed `icydb-schema-derive` primary-key nodes to retain ordered field-list
  metadata internally while keeping scalar runtime projection as an explicit
  bridge.
- Updated entity validation and generated model storage to walk ordered
  primary-key fields instead of hardcoding a single field assumption.
- Added a fixed-capacity `CompositePrimaryKeyValue` model in the core key
  taxonomy without changing persisted store-key encoding.
- Added generated composite primary-key type scaffolding for ordered
  primary-key metadata while keeping user-facing composite declarations gated.

### 🧪 Testing

- Added focused derive tests for ordered primary-key validation and generated
  key-type scaffolding, plus core key-taxonomy tests for fixed-capacity
  composite primary-key values.

## 0.162.2

### 🔧 Changed

- Added runtime `PrimaryKeyModel` metadata to `EntityModel`, giving generated
  entity models an ordered primary-key field list alongside the existing scalar
  primary-key projection.
- Updated generated entity-model output to emit ordered primary-key field
  references while preserving scalar runtime execution behavior.
- Re-exported the primary-key metadata model through the existing model facade
  so downstream generated code can use the new metadata shape without touching
  storage behavior.

### 🧪 Testing

- Added focused primary-key model tests for scalar and ordered field metadata,
  rechecked schema-derive entity model output, and validated the workspace,
  targeted clippy, formatting, and diff hygiene for the metadata bridge slice.

## 0.162.1

### 🔧 Changed

- Hard-cut the generated scalar primary-key attribute parser to the new formal
  field-list spelling. In this slice, `pk(fields = ["id"])` lowers to the
  existing scalar schema/runtime path while multi-field primary keys reject
  until the composite runtime/schema model lands.
- Moved schema primary-key declaration metadata to an ordered field-list shape,
  while keeping the current scalar runtime projection for existing execution
  code.
- Added compile-fail diagnostics for removed `pk(field = "id")`,
  comma-string `pk(fields = "id, name")`, empty `pk(fields = [])`, and
  composite `pk(fields = ["tenant_id", "local_id"])` declarations.

### 🧪 Testing

- Rechecked schema primary-key metadata tests, schema-derive primary-key parser
  tests, macro compile-fail coverage, workspace checking, schema/derive clippy,
  and diff hygiene for the primary-key syntax slice.

## 0.162.0

### 🔧 Changed

- Started the composite-primary-key release line with an implementation design
  that keeps accepted schema snapshots as runtime authority for primary-key
  arity, field order, durable field IDs, row slots, component kinds, and
  primary-key source.
- Defined the unified primary-key declaration direction for 0.162:

  ```rust
  #[entity(store = Store, pk(fields = ["id"]))]
  #[entity(store = Store, pk(fields = ["battle_id", "entity_id"]))]
  ```

- Documented the compact primary-key storage direction: data-store keys remain
  `EntityTag || EncodedPrimaryKey`, scalar keys keep scalar encoding, composite
  keys use a fixed-capacity runtime value model, and ordinary encode/decode
  paths must remain allocation-free.
- Clarified composite-primary-key constraints before implementation:
  composites are external-only, omitted composite `source` resolves to
  `external`, `Unit` is scalar-only, generated key structs are public `E::Key`
  types, and primary-key field facts duplicated in accepted snapshots must
  validate against the accepted field snapshot.

### ⚠️ Breaking

- Hard-cut generated index attributes to formal field lists parsed by Darling
  instead of IcyDB's old comma-string helper:

  ```rust
  index(fields = ["battle_id", "cell_id"])
  index(fields = ["tenant_id", "LOWER(email)"])
  ```

  The old spelling now rejects:

  ```rust
  index(fields = "battle_id, cell_id")
  ```

### 🧪 Testing

- Rechecked schema-derive and macro compile-fail coverage for formal generated
  index field-list parsing and diagnostics, plus workspace checking, derive
  clippy, formatting, and diff hygiene for the parser/design slice.
