# IcyDB 0.196 Changelog

## 0.196.16

### 📝 Summary

- Fixes the SQL write boundary so `INSERT` can omit schema-managed lifecycle
  timestamps and still persist non-epoch `created_at` and `updated_at` values.
- Extends the SQLite/IcyDB mutation differential harness beyond
  `UPDATE`/`DELETE` exact predicates by adding `INSERT` plus range-predicate
  mutation scenarios on the comparable mutable audit tables.
- Adds generated-model expected signatures to the mutation differential report,
  so the harness can distinguish IcyDB-vs-SQLite engine mismatches from seed or
  generator drift.
- Keeps the slice hard-cut and local: no public API, cursor-token,
  persisted-format, stable-storage, or read-admission contract changes.

### 🩹 Fixed

- SQL `INSERT` now synthesizes omitted `CreatedAt` and `UpdatedAt`
  write-managed fields from the statement write context before structural patch
  construction.
- `INSERT ... VALUES` and supported `INSERT ... SELECT` paths now share the
  same generated/managed omitted-field synthesis path.
- Explicit writes to generated or managed fields still fail closed through the
  existing SQL write policy checks.

### 🔧 Changed

- The SQL insert lowering path now treats generated primary keys and managed
  lifecycle fields as one synthesized-field set when constructing primary-key
  components and structural patches.
- The mutation differential generator now keeps an in-memory expected model for
  each generated mutation and records expected `RETURNING` and post-state
  signatures beside IcyDB and SQLite signatures.
- The harness compares only SQLite-visible shared fields for mutation INSERTs,
  leaving IcyDB-managed timestamps implicit rather than forcing SQLite to model
  IcyDB lifecycle policies.

### 🧪 Testing

- Adds SQL session coverage for named-column and positional INSERTs that omit
  managed timestamps and prove persisted `created_at` and `updated_at` are
  synthesized from one statement timestamp.
- Adds SQL session coverage for `INSERT ... SELECT` with generated primary keys
  plus managed timestamps, proving omitted SQL-owned fields are synthesized and
  persisted.
- Broadens the seeded SQLite mutation scenario generator to require INSERT,
  exact non-primary-key predicates, compound predicates, range predicates,
  indexed-field updates, deterministic post-state reads, and generated expected
  signatures.
- Refactors the mutation generator cleanup helpers so expected signatures are
  derived from the exact rows updated or removed, reducing duplicated harness
  logic.

### 📊 Performance

- No performance win is claimed for this patch; the slice fixes SQL write
  correctness and broadens differential correctness evidence.
- Full deterministic perf deltas were not rerun for this correctness-harness
  slice.
- Wasm-size delta was not measured.

### 📊 Complexity Delta

- Production runtime complexity increases slightly in SQL insert lowering by
  adding managed-field synthesis beside existing generated-field synthesis.
- Test and harness complexity increases through broader mutation-differential
  coverage and generated-model signatures, but the cleanup pass reduces repeated
  scenario-construction logic in the harness.
- Overall implementation shape stays narrow: one production SQL write boundary
  fix plus audit-harness coverage.

## 0.196.10

### 📝 Summary

- Reduces SQL boolean predicate compile work without changing public API,
  cursor tokens, persisted row/index/schema formats, stable storage layout, or
  execution semantics.
- Changes the runtime predicate compiler from eager TRUE/FALSE pair
  construction to demand-driven truth-branch construction.
- Preserves SQL three-valued membership behavior: `IN` / `NOT IN` lists with
  `NULL` still admit only SQL-TRUE rows, and `NOT` still requests the inverse
  branch explicitly.
- Keeps the optimization compile-side only; route facts, result signatures,
  cursor signatures, store counters, and index-entry counts are expected to
  remain unchanged.

### 🔧 Changed

- Boolean predicate compilation now asks for only the branch it needs:
  top-level access predicates request the TRUE branch, while unary `NOT`
  recursively requests the FALSE branch from its child.
- `AND`, `OR`, searched `CASE`, compare leaves, null tests, field predicates,
  text predicates, collection membership, and compact SQL membership all share
  the same `BoolTruth` demand-driven path.
- Compact SQL membership no longer builds the opposite membership predicate
  when the caller only needs the TRUE set.
- `NOT IN (... NULL ...)` still fails closed to `False` for the FALSE branch,
  preserving the existing SQL UNKNOWN behavior.

## 0.196.0

### 📝 Summary

- Closes the SQLite-informed ordered-read pushdown slice with adversarial
  before/after performance evidence instead of focused-only proof.
- Adds full deterministic SQL matrix route classification and union-key delta
  artifacts so optimized, unchanged, failed, and regressed scenarios are visible
  across the whole generated matrix.
- Keeps the production runtime slice narrow: no persisted-format change, no
  cursor-token protocol change, no public API compatibility shim, and no
  recovery or journal semantics change.

### 🔧 Changed

- Selected secondary index-range bounds now count as access-proven predicates
  during residual stripping. Queries such as an age-range filter ordered by the
  same age index no longer retain redundant post-access residual work when the
  chosen range already proves the lower and upper bounds.
- The range-bound proof stays conservative. Stricter sibling predicates remain
  residual filters, and composite ranges strip only the fixed prefix equality
  plus the selected range slot bounds.
- Public read admission can now admit exact primary-key `by_id(...)` and
  bounded `by_ids(...)` reads from the selected key-count upper bound, without
  requiring a redundant `LIMIT`.
- Oversized `by_ids(...)` sets still fail the public returned-row policy.

### 🩹 Fixed

- Fixes the `E188` public read-admission failure for exact `by_id(...)`
  lookup paths that were already route-proven as primary-key access but lacked
  a literal `LIMIT`.

### 🧪 Testing

- Added planner tests for access-proven index-range residual stripping,
  stricter residual predicates, and composite prefix-plus-range stripping.
- Added session read-admission tests proving primary-key `by_id(...)` lookup
  is admitted without a redundant `LIMIT`, while broader unbounded public reads
  remain rejected.
- Added adversarial cursor continuation tests for primary-key and secondary
  ordered paths, including deletion, insertion before and after cursor
  boundaries, duplicate ordered values, primary-key tie-breaks, and indexed
  field updates.
- Added diagnostics coverage for bounded secondary-order and equality-prefix
  ordered-suffix routes.
- Extended the SQL perf matrix harness with route family/outcome labels,
  stable result signatures, cursor signatures, canister wasm profile recording,
  and a saved-report delta helper with closeout gates.
