# 0.112 Changelog

## 0.112.0

- `0.112.0` starts a narrow prepared-SQL cleanup line aimed at one architectural seam: [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs) was still carrying too much duplicated local type and parameter-family reasoning between compare contracts, dynamic fallback functions, and the split between `WHERE` and grouped `HAVING`. This patch keeps the admitted prepared surface the same, but reduces how much of that surface is implemented as separate local policy branches.

- The first part of the consolidation is compare-contract ownership. Field-compare and aggregate-compare parameter contracts now flow through one shared `PreparedSqlCompareTargetContract` helper instead of assembling family and template metadata through parallel local paths. That keeps direct prepared compare slots on one bounded contract path for both scalar `WHERE` and grouped/global `HAVING`.

- The second part is dynamic fallback-family inference. `COALESCE`, `NULLIF`, and searched `CASE` result-family inference now share one bounded metadata path for:
  - missing-message ownership
  - conflict-label ownership
  - result-family inference
  - propagation of inferred result family back into argument contracts

- `0.112.0` then closes the larger behavioral gap between prepared `WHERE` and prepared `HAVING`. Grouped and global `HAVING` no longer stop at compare-only parameter slots during prepared contract collection. Expression-owned grouped `HAVING` can now freeze fallback contracts for the same bounded families that prepared `WHERE` already handled, including:
  - `COALESCE` / `NULLIF`
  - searched `CASE`
  - null tests
  - membership
  - nested compare and arithmetic subtrees inside the admitted post-aggregate expression family

- That alignment does **not** widen grouped template ownership. [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) now keeps an explicit grouped prepared-lane guard:
  - compare-family grouped `HAVING` remains eligible for the symbolic grouped template lane
  - expression-owned grouped/global `HAVING` remains fallback-only

- The final structural pass makes the code match that rule more directly. Shared fallback helpers now cover the same core family-inference work for both `WHERE` and `HAVING`, and the recursive fallback collector itself is now one scoped traversal instead of two near-duplicate trees. The lane-admission distinction still stays explicit at the prepared-template boundary instead of being hidden inside separate collector logic.

- The key behavioral invariant for `0.112.0` is now pinned in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_parameterized.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_parameterized.rs):
  - compare-family grouped `HAVING` still lands on `PreparedSqlExecutionTemplateKind::SymbolicGrouped`
  - expression-owned grouped `HAVING` still reuses the same numeric fallback contract inference model
  - expression-owned grouped `HAVING` still returns `template_kind_for_test() == None`
