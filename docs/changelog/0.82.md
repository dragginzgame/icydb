# 0.82 — Deterministic Compiled Query Reuse

## 0.82.9 - 2026-04-15 - Sparse SQL Projection and Grouped SQL Computed Ordering

This patch ships two narrow but meaningful SQL follow-ups on the same
read-path line.

First, it moves one important SQL projection cohort off the heavier generic
structural projection lane and onto a narrower sparse path:

* index-ordered SQL projections that return the primary key plus one or more
  uncovered row-backed fields
* including expression-backed order routes such as `ORDER BY LOWER(name), id`
  when the planner already selected the matching expression index

Second, it opens the next bounded grouped SQL slice after grouped computed
projection:

* grouped `SELECT` rows may now project scalar expressions that depend only on
  grouped fields
* grouped `ORDER BY` may now admit additive order-preserving expressions over
  grouped keys such as `ORDER BY age + 1`, but only on the bounded ordered
  grouped lane
* grouped runtime pagination and continuation cursors now preserve the actual
  grouped order direction instead of silently falling back to ascending key
  order on descending grouped pages

What changed:

* widened the SQL-side sparse index-backed projection admission in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/covering/mod.rs`
* the sparse SQL projection path no longer requires a projected index
  component in the outward `SELECT` list
* this means SQL rows such as:
  * `SELECT id, rank FROM PerfAuditUser ORDER BY name ASC, id ASC LIMIT 3`
  * `SELECT id, rank FROM PerfAuditUser ORDER BY LOWER(name) ASC, id ASC LIMIT 3`
  * can now stay on the sparse index-backed projection path even though the
    outward row returns only:
    * the primary key
    * one row-backed projected field
* kept the same hybrid sparse-path optimizations under that widened admission:
  * when index order already satisfies the query order, the covering scan now
    caps at `offset + limit`
  * the row-backed side decodes only uncovered fields
  * one-field row-backed reads stay on a direct single-slot decode path

Measured result:

The widened sparse SQL path is a real step-function win on the targeted
index-ordered projection rows.

Representative live PocketIC rows:

* `SELECT id, rank FROM PerfAuditUser ORDER BY name ASC, id ASC LIMIT 3`
  * baseline: `1,104,388`
  * now: `833,957`
  * delta: `-270,431` / `-24.48%`
* `SELECT id, rank FROM PerfAuditUser ORDER BY LOWER(name) ASC, id ASC LIMIT 3`
  * baseline: `1,183,280`
  * now: `892,122`
  * delta: `-291,158` / `-24.60%`
* `SELECT id, score FROM PerfAuditAccount WHERE active = true ORDER BY LOWER(handle) ASC, id ASC LIMIT 3`
  * baseline: `2,046,580`
  * now: `1,044,383`
  * delta: `-1,002,197` / `-48.96%`

Why this matters:

* the sparse SQL projection win and the grouped SQL widening are different
  slices, but both follow the same discipline:
  * widen one already-valid read family
  * keep the runtime contract narrow
  * leave broader expression support fail-closed
* on the sparse SQL side, the missing win was not another cache layer; it was
  one overly narrow sparse-path admission rule
* once the planner/runtime stopped requiring a projected index component in
  the outward row, several real SQL projection families could avoid the
  generic structural projection lane entirely
* that cuts work in two ways:
  * fewer row-backed reads when index order already satisfies the page window
  * much less decode/materialization work per fetched row
* on the grouped SQL side, computed projection over grouped fields does not
  widen grouping semantics; it only allows derived values from the grouped key
  to stay on the grouped lane
* the additive grouped-key ordering slice is intentionally smaller than
  general grouped computed ordering or aggregate ordering, which keeps the
  planner/runtime contract explicit
* the descending grouped-page fix matters because admitting grouped computed
  order without preserving direction through page finalization and cursor
  emission would make grouped ordered pagination lie about its actual order
* the new `db::data` selective row-read boundary also leaves the codebase in a
  better position for the next real storage-side optimization
* the remaining sparse-path floor is still the full owned `RawRow` fetch
  returned by the stable-map `get()` boundary, but that is now isolated behind
  one correct selective-read contract instead of being reimplemented in the
  SQL layer

## 0.82.8 - 2026-04-15 - Typed Write Facade Realignment

This patch realigns the public typed write facade with the fixed-shape write
contract already used by `icydb-core`.

What changed:

* restored the typed public write facade to the same fixed-shape contract used
  by `icydb-core`:
  * `/home/adam/projects/icydb/crates/icydb/src/db/session/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb/src/db/session/delete.rs`
  * single-row typed writes such as `insert`, `create`, `replace`, `update`,
    and `mutate_structural` now return `Result<E, Error>` directly instead of
    `Result<MutationResult<E>, Error>`
  * typed batch writes such as `insert_many_*`, `replace_many_*`, and
    `update_many_*` now return `Result<Vec<E>, Error>`
  * plain typed delete execution now returns `Result<u32, Error>` so count-only
    delete flows match the core delete query contract instead of routing
    through the old SQL-oriented mutation wrapper
* updated the workspace package description in `/home/adam/projects/icydb/Cargo.toml`
  so published crate metadata describes IcyDB as a schema-first typed query
  engine and persistence runtime instead of an ORM

Why this matters:

* the old facade shape leaked SQL `RETURNING` concerns into typed APIs whose
  result shape was already statically known
* that made routine typed writes awkward by forcing callers through
  `.entity()?` even when the write contract could only produce one entity
* the restored shape makes the facade and core agree again on the typed write
  model, which reduces confusion and removes unnecessary wrapper extraction in
  downstream code
* the old package description was lagging behind the actual product shape and
  was signaling a narrower ORM framing than the current planner/executor and
  persistence runtime model

## 0.82.7 - 2026-04-15 - Generic Create Inputs and Sparse Direct-Order Decode

This patch does four small but intentional things.

First, it cleans up the generated create-input naming model without forcing
downstream code to spell the generated concrete type directly.

Second, it keeps the `0.82` executor line on narrow, measurable cuts by making
direct-field materialized ordering over raw `DataRow` payloads decode only the
ordered slots instead of routing through the broader structural row-reader path.

Third, it sharpens the local SQL perf contract by splitting physical
store/index access out of the older umbrella executor bucket.

Fourth, it makes that shell-facing perf contract self-explanatory by adding
local `?` / `help` commands to the Rust CLI.

What changed:

* changed generated create-input concrete type names from `<Entity>Create` to
  `<Entity>_Create`:
  * `/home/adam/projects/icydb/crates/icydb-schema-derive/src/node/entity.rs`
  * generated entities now also implement one new associated-type trait:
    * `EntityCreateType`
  * that trait binds the entity type to its generated authored-create shape
    without making downstream code depend on the generated concrete name
* added the entity-owned create-type association to the core/runtime trait
  boundary:
  * `/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs`
  * `EntityCreateType` now exposes:
    * `type Create: EntityCreateInput<Entity = Self>;`
* exposed one public generic facade alias for downstream use:
  * `/home/adam/projects/icydb/crates/icydb/src/lib.rs`
  * `/home/adam/projects/icydb/crates/icydb/src/traits/mod.rs`
  * downstream crates can now write:
    * `icydb::Create::<MyEntity>`
  * instead of naming the generated concrete type directly
* split the local SQL perf attribution contract from `c/p/e/d` to `c/p/s/e/d`:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/physical_access.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/registry.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`

  * `/home/adam/projects/icydb/canisters/test/sql/src/lib.rs`
  * `/home/adam/projects/icydb/crates/icydb-cli/src/main.rs`
  * `s` now means:
    * physical data/index-store traversal plus physical payload decode at the

  * `e` now means:
    * executor logic above storage such as residual evaluation, ordering,

    * `\\?`
    * `\\help`
  * interactive mode now handles those locally instead of forwarding them to
    the SQL canister
  * one-shot CLI mode also recognizes `--sql "help"` / `--sql "?"` and prints

* updated the macro UI tests to validate the intended downstream surface and

  * `/home/adam/projects/icydb/testing/macro-tests/tests/ui/insert_type_excludes_generated_field.rs`
  * `/home/adam/projects/icydb/testing/macro-tests/tests/ui/insert_type_excludes_managed_field.rs`

  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/order.rs`

    * rows still sort correctly

Why this matters:

* the old generated create type name was workable but awkward as a public
  surface because downstream code either had to:
  * know the generated concrete type name, or
  * thread a separate local alias itself
* the new shape keeps the generated concrete type internal-ish and gives the
  facade one stable generic spelling that reads like a real API type:
  * `Create<Entity>`
* the older `c/p/e/d` footer was mixing physical storage traversal and
  executor logic into one `e` bucket
* that was good enough for a first shell cut, but it was the wrong story once
  the direct-row attribution work had already proved store access was often the
  dominant runtime cost
* the new split is still intentionally coarse:
  * it does not try to expose every executor sub-bucket in the shell footer
  * it only separates the physically meaningful store/index layer from the
    remaining executor logic
* once the footer gained a new `s` phase, relying only on changelog text was
  too indirect for actual shell use
* the right follow-up was to let the shell explain its own legend on demand
  instead of making users remember phase meanings or leave the tool
* on the executor side, the `0.82.5` and `0.82.6` direct-lane attribution work
  already showed that materialized-order direct loads still pay a meaningful
  `direct_order` bucket
* the comparator path was already caching order tuples once per row, so the
  next correct cut was not “more caching”
* the next correct cut was:
  * when `ORDER BY` is direct fields only,
  * decode only those slots,
  * and do not build the more general reader/cache shape needed for
    expression-backed order evaluation

Measured executor read:

* representative warmed direct-field materialized-order fluent load:
  * `user.age.order_only.asc.limit2.warm_after_update`
  * before this cut:
    * `direct_order ≈ 286,154`
    * `total ≈ 779,713`
  * after this cut:
    * `direct_order = 281,192`
    * `total = 774,024`
* that is a modest win, not a step-function shift
* it confirms the cut is worthwhile and clean, but it does not change the
  larger remaining conclusion from the direct-lane audit:
  * route-ordered direct loads are still dominated by `store.get()`
  * materialized-order direct loads are still dominated by `store.get()` first
    and `direct_order` second

widen SQL surface area or add another cache layer.

  without building transient `KernelRow` envelopes

  direct paths and which ones still pay the retained full-row cost
* the local Rust SQL shell is easier to read during repeated interactive work,
  SQL `EXPLAIN EXECUTION` now exposes the current `c/p/e/d` phase meanings and
  projection materialization hints directly in the shell surface, and grouped

  * SQL result rendering is unchanged and still uses the normal `0x...` blob

    * unique terminals:
      * `COUNT(age)`
      * `SUM(age)`

    * `COUNT(age)` once
    * `SUM(age)` once

* this stays aligned with the current ownership split:
  * parser owns surface syntax

  * execution owns routing and materialization only

* it does not attempt semantic equivalence collapse such as:
  * `COUNT(*)`

  * SQL still uses:
    * `SqlCompiledCommandCacheKey`

  * `/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/fingerprint.rs`
  * predicate cache identity now reuses the predicate subsystem's canonical

  * `/home/adam/projects/icydb/crates/icydb-core/src/db/codec/mod.rs`

    fingerprint boundaries that still own them
* kept the current algorithm split explicit:
  * `Sha256` remains the authority for deterministic plan, continuation,
    predicate, and type/schema identity surfaces
  * `xxh3` remains the authority for in-heap cache-table hashing, canonical

  hashing surfaces instead of pretending every manual hasher should disappear:
  * `/home/adam/projects/icydb/docs/design/0.82-deterministic-compiled-query-reuse/0.82-addendum-hashing.md`
  * follow-up guidance now explicitly calls out:
    * `value/hash.rs` as the main remaining cleanup candidate
    * `query/intent/cache_key.rs` as a place to keep reusing subsystem-owned

    * `query/fingerprint/hash_parts.rs` and `schema/fingerprint.rs` as

Why this patch matters:

* the SQL-side cache model was already structurally explicit
* the remaining complexity hotspot was the shared lower fluent/structural cache
  identity path
* `0.82.0` makes that lower identity visible as one real type instead of
  letting semantic identity stay buried inside an implementation-specific hash
  walk
* the patch also makes the repo's hashing split easier to reason about:
  * `Sha256` for deterministic semantic identity authorities
  * `xxh3` for cache/runtime hashing where cryptographic stability is not the
    purpose

What this patch does not do:

* no SQL surface widening
* no new cache tier
* no executor-prepared persistent cache model
* no attempt to replace authority-boundary semantic hashers with `Debug`,
  generic serialization, or blanket `derive(Hash)` on live query model trees
