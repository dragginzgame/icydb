# 0.82 — Deterministic Compiled Query Reuse

## 0.82.1 - 2026-04-15 - Versioned Cache Compatibility and Isolated Warm-Run Proof

This patch turns two previously implicit cache-contract rules into explicit
shipped behavior:

* cache-method/version drift must fail closed
* cold vs warmed repeated-query reuse must be measurable on isolated fixtures,
  not inferred from mixed recurring batches

What changed:

* added one explicit cache-method/version dimension to all current live query
  reuse keys:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * shared lower query-plan cache keys now include one explicit method version
  * SQL compiled-command cache keys now include one explicit method version
  * SQL select-plan cache keys now include one explicit method version
* added fail-closed coverage proving version drift misses instead of aliasing:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs`
  * new tests prove:
    * shared lower cache version mismatch misses
    * SQL compiled/select-plan cache version mismatch misses
    * query/update surface identity remains distinct even when the cache method
      version matches
* added isolated recurring cold-vs-warm parity rows to both PocketIC perf
  harnesses:
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/sql_perf_audit.rs`
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/fluent_perf_audit.rs`
  * `/home/adam/projects/icydb/canisters/audit/sql_perf/src/lib.rs`
  * both harnesses now carry one isolated pair for the same logical
    `ORDER BY age ASC, id ASC LIMIT 2` read:
    * cold query
    * warm-after-update query
  * these rows use isolated fixtures so they represent real cold and real
    warmed behavior instead of inheriting cache state from the broader matrix
* fixed one local Rust SQL shell grouped-query decode regression:
  * `/home/adam/projects/icydb/crates/icydb-cli/src/main.rs`
  * grouped queries were already executing correctly in the canister, but the
    CLI shell failed while decoding the `query_with_perf` JSON envelope before
    it could render grouped rows
  * the failing field was grouped `next_cursor`
  * `dfx --output json` surfaces Candid `Option<String>` as:
    * `[]` for `None`
    * `["cursor-token"]` for `Some("cursor-token")`
  * the CLI had been feeding that raw JSON directly into:
    * `serde_json::from_value::<SqlQueryResult>(...)`
  * grouped shell queries with no continuation cursor therefore failed with:
    * `invalid type: sequence, expected a string`
  * the CLI now normalizes grouped `next_cursor` before deserializing the SQL
    result and carries local regression coverage for both the empty and
    populated grouped cursor JSON forms
* widened the dedicated global aggregate SQL lane from one terminal to one
  bounded aggregate-terminal list:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/aggregate.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/aggregate.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/explain.rs`
  * SQL now admits shapes like:
    * `SELECT MIN(age), MAX(age) FROM ...`
    * `SELECT MIN(age) AS youngest, MAX(age) AS oldest FROM ...`
  * the shape remains intentionally narrow:
    * every projection item must still be a supported aggregate terminal
    * mixed scalar-plus-aggregate projection stays fail-closed
    * grouped aggregate handling remains on the existing grouped lane
  * execution now returns one ordinary projection row carrying one reduced
    value per aggregate terminal instead of rejecting multi-terminal global
    aggregate lists at lowering
  * `EXPLAIN EXECUTION` for that widened shape renders one aggregate execution
    descriptor per terminal rather than pretending the whole query only has
    one aggregate route

Why this patch matters:

* `0.82.0` made the lower shared identity explicit, but method/version drift
  was still only a design rule
* `0.82.1` turns that into one explicit shipped compatibility dimension
* `0.82.1` also fixes one real user-facing shell regression on the same line:
  grouped SQL was correct in the engine but unreadable in the local shell when
  the grouped cursor was absent
* `0.82.1` also closes one obvious reduced-SQL gap without opening the broader
  projection surface: callers can now ask for a small list of global
  aggregates in one `SELECT` instead of issuing one query per aggregate
* the new isolated perf rows also make the cache story easier to read:
  * fluent now shows the clean lower-cache effect directly
  * SQL shows the full layered cache stack effect directly

Measured isolated parity results:

* SQL:
  * cold:
    * compile `107,934`
    * execute `972,257`
    * total `1,080,191`
  * warm-after-update:
    * compile `15,727`
    * execute `735,896`
    * total `751,623`
  * delta:
    * `-328,568`
    * about `-30.4%`
* fluent:
  * cold:
    * compile `290,749`
    * execute `812,315`
    * total `1,103,064`
  * warm-after-update:
    * compile `49,772`
    * execute `809,393`
    * total `859,165`
  * delta:
    * `-243,899`
    * about `-22.1%`

How to read those numbers:

* fluent is the clearest lower-cache proof:
  * compile drops sharply
  * execute stays essentially flat
* SQL shows a stronger total drop because the warmed path is benefiting from
  both:
  * the shared lower-plan cache
  * the SQL-specific upper cache layers
* against the real early pre-cache SQL repeat rows from `0.80.0`, the current
  repeat-query SQL rows are also materially lower on representative families
  such as:
  * primary-key ordered repeat reads
  * lower(name) ordered repeat reads
  * grouped repeat reads

What this patch does not do:

* no new SQL surface widening
* no new cache tier
* no executor-prepared persistent cache promotion
* no fluent “true pre-cache” historical comparison claim; the earliest fluent
  baseline in-repo was already post-cache
* no grouped planner/runtime behavior change
* no grouped cursor protocol change
* no canister payload schema change

## 0.82.0 - 2026-04-15 - Canonical Shared Cache Identity and Hashing Cleanup

This patch turns the shared lower query-plan cache identity into one explicit
canonical structure instead of a hidden structural hash walk, and folds the
hashing addendum into the shipped `0.82.0` line.

What changed:

* introduced one explicit canonical shared lower cache key:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs`
  * new owner type:
    * `StructuralQueryCacheKey`
  * the shared lower cache now keys reuse with:
    * entity path
    * schema fingerprint
    * visibility
    * `StructuralQueryCacheKey`
  * instead of hiding lower-cache identity behind one manual structural digest
* kept SQL upper caches on the same explicit key-story they already had:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * SQL still uses:
    * `SqlCompiledCommandCacheKey`
    * `SqlSelectPlanCacheKey`
  * the lower shared cache now follows the same “normalize first, derive hash at
    the map boundary” model
* collapsed one stale hash-ownership leak at the cache-key layer:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/fingerprint.rs`
  * predicate cache identity now reuses the predicate subsystem's canonical
    fingerprint authority instead of reopening SHA stream construction inside
    the query-intent cache-key layer
* removed the now-dead generic DB re-export of the raw SHA constructor:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/codec/mod.rs`
  * the remaining SHA entrypoints now stay at the explicit deterministic
    fingerprint boundaries that still own them
* kept the current algorithm split explicit:
  * `Sha256` remains the authority for deterministic plan, continuation,
    predicate, and type/schema identity surfaces
  * `xxh3` remains the authority for in-heap cache-table hashing, canonical
    value hashing, and the outer schema commit fingerprint
* updated the `0.82` hashing addendum to record the remaining justified manual
  hashing surfaces instead of pretending every manual hasher should disappear:
  * `/home/adam/projects/icydb/docs/design/0.82-deterministic-compiled-query-reuse/0.82-addendum-hashing.md`
  * follow-up guidance now explicitly calls out:
    * `value/hash.rs` as the main remaining cleanup candidate
    * `query/intent/cache_key.rs` as a place to keep reusing subsystem-owned
      canonical key/fingerprint authorities where they already exist
    * `query/fingerprint/hash_parts.rs` and `schema/fingerprint.rs` as
      authority-boundary manual hashers that should stay explicit

Why this patch matters:

* the SQL-side cache model was already structurally explicit
* the remaining complexity hotspot was the shared lower fluent/structural cache
  identity path
* `0.82.0` makes that lower identity visible as one real type instead of
  letting semantic identity stay buried inside an implementation-specific hash
  walk
* the patch also makes the repo's hashing split easier to reason about:
  * `Sha256` for deterministic semantic identity authorities
  * `xxh3` for cache/runtime hashing where cryptographic stability is not the
    purpose

What this patch does not do:

* no SQL surface widening
* no new cache tier
* no executor-prepared persistent cache model
* no attempt to replace authority-boundary semantic hashers with `Debug`,
  generic serialization, or blanket `derive(Hash)` on live query model trees
