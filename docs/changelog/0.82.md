# 0.82 — Deterministic Compiled Query Reuse

## 0.82.5 - 2026-04-15 - Executor Lane Contracts and Raw-Row Load Proofs

This patch keeps the `0.82` line on executor/runtime ownership. It does not
widen SQL surface area or add another cache layer.

The concrete outcomes are:

* residual predicate handling at the scan boundary is now explicit instead of
  being hidden behind one overloaded boolean
* more cursorless scalar load families can stay on direct raw-row paths
  without building transient `KernelRow` envelopes
* executor lane metrics now prove which fluent load families already use those
  direct paths and which ones still pay the retained full-row cost
* the local Rust SQL shell is easier to read during repeated interactive work,
  SQL `EXPLAIN EXECUTION` now exposes the current `c/p/e/d` phase meanings and
  projection materialization hints directly in the shell surface, and grouped
  SQL field typos now fail with one specific missing-field error instead of one
  misleading generic grouped-shape rejection

## 0.82.4 - 2026-04-15 - Scalar Runtime Materialization Cuts and Direct Data-Row Projection

This patch keeps the `0.82` line on executor/runtime cost instead of widening
SQL surface area again.

The two concrete outcomes are:

* warmed scalar runtime no longer prepares projection-validation and
  retained-slot state on several lanes that do not consume it
* SQL structural projection now has one direct raw-row field-copy path for
  planner-proven direct-field projections, including repeated direct fields on
  the raw `DataRow` path

What changed:

* `PreparedExecutionProjection` now compiles projection state from actual lane
  needs instead of from a broader pre-optimization mode bucket:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs`
  * model-identity projections no longer build shared validation state just to
    no-op inside the validator
  * non-cursor scalar `DataRow` lanes can now opt out of projection
    materialization entirely with `ProjectionMaterializationMode::None`
* shared projection validation now depends on one deliberately narrow row-read
  trait rather than ad hoc slot-reader closures:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/materialize/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/page.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/operators/terminal/runtime/row_collector.rs`
  * the new trait is local to validation only and does not become a generic
    executor-wide row abstraction
* SQL structural projection over raw rows now has a direct field-copy path
  beside the existing retained-slot direct path:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/projection/runtime.rs`
  * when planner metadata proves the outward projection is direct fields only,
    raw-row projection decodes only the declared projected slots and skips the
    generic scalar projection evaluator
* prepared projection shape now freezes two distinct direct-field layouts
  instead of one overloaded layout:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/materialize/mod.rs`
  * retained-slot direct layout stays unique-only because slot rows consume
    values with `take()`
  * raw-row direct layout is duplicate-safe because raw slot reads can borrow
    the same slot multiple times
  * repeated direct projections such as repeated field selections on the raw
    `DataRow` path therefore stay on the direct copy path instead of falling
    back to scalar evaluation

Why this patch matters:

* the remaining visible `0.82` floor was executor/runtime work, especially on
  warmed scalar paths
* the first useful cut was to stop preparing validation/materialization state
  for lanes that never consume it
* the second useful cut was to give simple SQL field-only projections a raw-row
  short path that avoids the generic scalar evaluator
* separating retained-slot and raw-row direct layouts keeps the executor
  boundary honest:
  * retained-slot semantics stay strict
  * raw-row projection can safely handle repeated direct fields

Important boundary:

* this patch does not introduce one generic representation-agnostic row API
  across the executor
* the new validation trait is intentionally narrow and local to the validation
  seam
* the direct raw-row short path applies only to planner-proven direct-field SQL
  projection shapes
* computed expressions still use the existing scalar projection evaluator

Coverage added:

  * direct one-field raw-row copies stay off the scalar fallback path
  * repeated direct raw-row field copies also stay off the scalar fallback path

## 0.82.3 - 2026-04-15 - Duplicate Global Aggregate Terminal Dedupe

This patch keeps the dedicated SQL global aggregate lane narrow but removes one
obvious source of repeated work inside its compiled artifact:

* exact duplicate aggregate projection terminals now compile to one unique
  executable terminal plus one output remap
* execution computes each unique terminal once and then fans the reduced value
  back out into the original SQL projection order
* outward SQL column multiplicity and labels stay unchanged

What changed:

* global aggregate lowering now canonicalizes duplicate aggregate terminals into
  one unique terminal list plus one output remap:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/aggregate.rs`
  * `LoweredSqlGlobalAggregateCommand`
  * `SqlGlobalAggregateCommand`
  * `SqlGlobalAggregateCommandCore`
  now all carry:
    * unique terminals
    * `output_remap`
  * the remap records which original SQL projection slot should read from which
    unique executable terminal
* global aggregate execution now computes only the unique prepared strategies
  and then fans the values back out in original SQL projection order:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/aggregate.rs`
  * duplicate outward columns therefore remain visible, but identical aggregate
    work is no longer rerun per repeated projection slot
* raw `Value::Blob` debug output is now summarized instead of printing the full
  underlying byte array:
  * `/home/adam/projects/icydb/crates/icydb-core/src/value/mod.rs`
  * blob values now render in debug/error contexts as:
    * `Blob(N bytes)`
  * SQL result rendering is unchanged and still uses the normal `0x...` blob
    text form on the outward SQL display path

What this means in practice:

* before:
  * `SELECT COUNT(age), COUNT(age), SUM(age), COUNT(age) ...`
    compiled to four aggregate terminals and ran all four
* now:
  * the same query compiles to:
    * unique terminals:
      * `COUNT(age)`
      * `SUM(age)`
    * output remap:
      * `[0, 0, 1, 0]`
  * execution computes:
    * `COUNT(age)` once
    * `SUM(age)` once
  * final packaging still returns all four SQL output columns in the original
    order

Why this patch matters:

* the previous dedicated global aggregate lane preserved repeated identical
  terminals exactly as written
* that was correct, but it meant repeated aggregate projections like:
  * `COUNT(age), COUNT(age), COUNT(age)`
  paid repeated identical reduction work
* the right place to fix that is the compiled SQL aggregate artifact, not the
  shared lower structural cache and not a runtime side cache
* this stays aligned with the current ownership split:
  * parser owns surface syntax
  * lowering owns canonicalization and semantic compile shape
  * execution owns routing and materialization only

Important boundary:

* this patch dedupes only exact canonical aggregate-terminal duplicates
* it does not attempt semantic equivalence collapse such as:
  * `COUNT(*)`
  * `COUNT(non_null_field)`
* that is a different optimization line because it requires model-aware
  semantic equivalence reasoning, not plain duplicate-terminal collapse
* the blob debug cleanup only changes raw Rust debug/error presentation
* it does not change value serialization, SQL result rendering, or persisted
  row behavior

Coverage added:

* lowering tests:
  * exact duplicate remap
  * mixed duplicate remap
  * distinct-vs-non-distinct separation
  * qualified/unqualified field-target collapse
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs`
* session execution tests:
  * duplicate output columns still render
  * duplicate aliases still render
  * distinct-vs-non-distinct outputs stay separate
  * qualified/unqualified duplicate outputs stay equal
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_aggregate.rs`

## 0.82.2 - 2026-04-15 - Shared COUNT Routes and Fast-Path Inventory

This patch closes one visible consumer-route gap in the aggregate SQL lane and
turns the current fast-path ownership story into an explicit maintained
inventory with a concrete structural tripwire.

What changed:

* rerouted SQL global row-count-equivalent count shapes through the shared
  typed scalar terminal boundary instead of the structural projection-and-count
  detour:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/aggregate.rs`
  * the previous implementation lowered the SQL base query, projected primary
    keys or field values through the structural SQL projection lane,
    materialized every matching row, and then counted those projected rows in
    session code
  * the new implementation rebuilds one typed `Query<E>` from the lowered
    structural base query and reuses:
    * shared query-plan cache resolution
    * `ScalarTerminalBoundaryRequest::Count`
    * the existing executor-owned count fast-path contracts
  * the widened admitted SQL consumer set is still narrow:
    * `COUNT(*)`
    * `COUNT(field)` only when the field is schema-non-nullable and the query
      is not `DISTINCT`
  * this keeps SQL row-count-equivalent count shapes on the same route owner as
    fluent `count()` without widening SQL-local eligibility logic
* exposed the shared query-plan cache attribution on that rerouted SQL count
  path instead of reporting the global aggregate lane as cache-free:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/mod.rs`
  * the SQL aggregate execute branch now returns shared lower cache hits/misses
    when `COUNT(*)` resolves through the typed count terminal path
* added route-parity regression tests proving SQL `COUNT(*)` and
  `COUNT(non-null_field)` both reuse the same shared query-plan cache entry as
  equivalent fluent `count()` windows:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_aggregate.rs`
* expanded the fast-path inventory note so it now records:
  * canonical owners
  * current consumer routes
  * current tripwires
  * remaining unguarded areas
  * `/home/adam/projects/icydb/docs/governance/fast-path-inventory.md`
* added one small structural guard so the SQL count consumer routes are harder
  to regress silently:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/planning/route/tests/fast_path_guards.rs`
  * the guard asserts:
    * SQL aggregate execution still references
      `ScalarTerminalBoundaryRequest::Count`
    * the old `select_fields([authority.primary_key_name()])` projection detour
      does not reappear in that file
    * the `COUNT(field)` widening stays behind one explicit non-nullability
      guard helper

Why this patch matters:

* whole-window `COUNT(*)` should be one of the cheapest aggregate terminals
* `COUNT(field)` on a non-nullable field is semantically the same row-count
  question and should not keep paying field materialization cost
* the executor already had the right count fast-path owner boundary
* the remaining problem was that reduced SQL was bypassing that owner and doing
  extra row materialization work locally
* this patch fixes the surface-routing mistake without adding one more ad hoc
  SQL fast path
* the new inventory/tripwire note also makes later performance work safer by
  naming which boundaries are owners, which surfaces are consumers, and which
  seams are actually guarded today

What this patch does not do:

* no new count eligibility rules
* nullable `COUNT(field)` still uses the value-inspection path
* `COUNT(DISTINCT field)` still uses the value-inspection path
* no `SUM/AVG/MIN/MAX` execution-path change yet
* no planner-visible route-policy widening

## 0.82.0 - 2026-04-15 - Canonical Shared Cache Identity and Hashing Cleanup

This patch turns the shared lower query-plan cache identity into one explicit
canonical structure instead of a hidden structural hash walk, and folds the
hashing addendum into the shipped `0.82.0` line.

What changed:

* introduced one explicit canonical shared lower cache key:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs`
  * new owner type:
    * `StructuralQueryCacheKey`
  * the shared lower cache now keys reuse with:
    * entity path
    * schema fingerprint
    * visibility
    * `StructuralQueryCacheKey`
  * instead of hiding lower-cache identity behind one manual structural digest
* kept SQL upper caches on the same explicit key-story they already had:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * SQL still uses:
    * `SqlCompiledCommandCacheKey`
    * `SqlSelectPlanCacheKey`
  * the lower shared cache now follows the same “normalize first, derive hash at
    the map boundary” model
* collapsed one stale hash-ownership leak at the cache-key layer:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/fingerprint.rs`
  * predicate cache identity now reuses the predicate subsystem's canonical
    fingerprint authority instead of reopening SHA stream construction inside
    the query-intent cache-key layer
* removed the now-dead generic DB re-export of the raw SHA constructor:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/codec/mod.rs`
  * the remaining SHA entrypoints now stay at the explicit deterministic
    fingerprint boundaries that still own them
* kept the current algorithm split explicit:
  * `Sha256` remains the authority for deterministic plan, continuation,
    predicate, and type/schema identity surfaces
  * `xxh3` remains the authority for in-heap cache-table hashing, canonical
    value hashing, and the outer schema commit fingerprint
* updated the `0.82` hashing addendum to record the remaining justified manual
  hashing surfaces instead of pretending every manual hasher should disappear:
  * `/home/adam/projects/icydb/docs/design/0.82-deterministic-compiled-query-reuse/0.82-addendum-hashing.md`
  * follow-up guidance now explicitly calls out:
    * `value/hash.rs` as the main remaining cleanup candidate
    * `query/intent/cache_key.rs` as a place to keep reusing subsystem-owned
      canonical key/fingerprint authorities where they already exist
    * `query/fingerprint/hash_parts.rs` and `schema/fingerprint.rs` as
      authority-boundary manual hashers that should stay explicit

Why this patch matters:

* the SQL-side cache model was already structurally explicit
* the remaining complexity hotspot was the shared lower fluent/structural cache
  identity path
* `0.82.0` makes that lower identity visible as one real type instead of
  letting semantic identity stay buried inside an implementation-specific hash
  walk
* the patch also makes the repo's hashing split easier to reason about:
  * `Sha256` for deterministic semantic identity authorities
  * `xxh3` for cache/runtime hashing where cryptographic stability is not the
    purpose

What this patch does not do:

* no SQL surface widening
* no new cache tier
* no executor-prepared persistent cache model
* no attempt to replace authority-boundary semantic hashers with `Debug`,
  generic serialization, or blanket `derive(Hash)` on live query model trees
