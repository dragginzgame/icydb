# 0.82 — Deterministic Compiled Query Reuse

## 0.82.0 - 2026-04-15 - Canonical Shared Cache Identity and Hashing Cleanup

This patch turns the shared lower query-plan cache identity into one explicit
canonical structure instead of a hidden structural hash walk, and folds the
hashing addendum into the shipped `0.82.0` line.

What changed:

* introduced one explicit canonical shared lower cache key:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs`
  * new owner type:
    * `StructuralQueryCacheKey`
  * the shared lower cache now keys reuse with:
    * entity path
    * schema fingerprint
    * visibility
    * `StructuralQueryCacheKey`
  * instead of hiding lower-cache identity behind one manual structural digest
* kept SQL upper caches on the same explicit key-story they already had:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * SQL still uses:
    * `SqlCompiledCommandCacheKey`
    * `SqlSelectPlanCacheKey`
  * the lower shared cache now follows the same “normalize first, derive hash at
    the map boundary” model
* collapsed one stale hash-ownership leak at the cache-key layer:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/fingerprint.rs`
  * predicate cache identity now reuses the predicate subsystem's canonical
    fingerprint authority instead of reopening SHA stream construction inside
    the query-intent cache-key layer
* removed the now-dead generic DB re-export of the raw SHA constructor:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/codec/mod.rs`
  * the remaining SHA entrypoints now stay at the explicit deterministic
    fingerprint boundaries that still own them
* kept the current algorithm split explicit:
  * `Sha256` remains the authority for deterministic plan, continuation,
    predicate, and type/schema identity surfaces
  * `xxh3` remains the authority for in-heap cache-table hashing, canonical
    value hashing, and the outer schema commit fingerprint
* updated the `0.82` hashing addendum to record the remaining justified manual
  hashing surfaces instead of pretending every manual hasher should disappear:
  * `/home/adam/projects/icydb/docs/design/0.82-deterministic-compiled-query-reuse/0.82-addendum-hashing.md`
  * follow-up guidance now explicitly calls out:
    * `value/hash.rs` as the main remaining cleanup candidate
    * `query/intent/cache_key.rs` as a place to keep reusing subsystem-owned
      canonical key/fingerprint authorities where they already exist
    * `query/fingerprint/hash_parts.rs` and `schema/fingerprint.rs` as
      authority-boundary manual hashers that should stay explicit

Why this patch matters:

* the SQL-side cache model was already structurally explicit
* the remaining complexity hotspot was the shared lower fluent/structural cache
  identity path
* `0.82.0` makes that lower identity visible as one real type instead of
  letting semantic identity stay buried inside an implementation-specific hash
  walk
* the patch also makes the repo's hashing split easier to reason about:
  * `Sha256` for deterministic semantic identity authorities
  * `xxh3` for cache/runtime hashing where cryptographic stability is not the
    purpose

What this patch does not do:

* no SQL surface widening
* no new cache tier
* no executor-prepared persistent cache model
* no attempt to replace authority-boundary semantic hashers with `Debug`,
  generic serialization, or blanket `derive(Hash)` on live query model trees
