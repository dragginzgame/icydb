# 0.121 Changelog

## 0.121.0

- `0.121.0` starts the access/explain projection-collapse line by removing more duplicate structural access walkers instead of adding another shared DTO layer on top of them.

- The first cut lands in fingerprint hashing:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/hash_parts/access.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/hash_parts/access.rs) no longer keeps one visitor for planner-owned `AccessPlan<K>` and another near-identical visitor for `ExplainAccessPath`
  - those two paths now share one generic `AccessFingerprintVisitor`
  - planner access hashing and explain access hashing both read through the same `AccessPlanProjection` contract instead of drifting through parallel structural visitors

- The second cut lands in EXPLAIN access JSON:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/access_projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/access_projection.rs) no longer renders JSON through another hand-written recursive `ExplainAccessPath` match tree
  - `write_access_json(...)` now projects through `project_explain_access_path(...)` via `ExplainAccessJsonProjection`
  - this keeps the DTO as the transport artifact while moving the walk itself back onto the shared projection seam

- The third cut lands in executor EXPLAIN descriptor assembly:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/shared/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/shared/mod.rs) no longer manually recurses `ExplainAccessPath` to build `ExplainExecutionNodeDescriptor` trees
  - access-node descriptor assembly now uses `project_explain_access_path(...)` through `ExplainAccessNodeDescriptorProjection`
  - the old `explain_access_execution_node_type(...)` helper in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/access_projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/access_projection.rs) became redundant after that move and was deleted instead of being left as another compatibility seam

- The fourth cut removes one more duplicated access label ladder:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/access_projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/access_projection.rs) now owns the shared strategy-label projector for both planner-owned `AccessPlan<K>` and `ExplainAccessPath`
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/access_projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/access_projection.rs) no longer keeps a second explain-local strategy-label projection body
  - explain rendering and executor descriptor annotation now consume the planner-owned label helper directly instead of routing back through a query-explain-local re-export:
    - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/render.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/render.rs)
    - [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/shared/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/shared/mod.rs)

- `0.121.0` also collapses one non-access duplicate walk that had the same structural smell:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/capability.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/capability.rs) previously kept two near-identical recursive index-capability classifiers:
    - `classify_index_capability(...)`
    - `classify_index_capability_for_targets(...)`
  - both walked the same `ExecutablePredicate` tree and shared the same branch policy for `True` / `False`, `And`, `Or`, `Not`, and the non-compare leaves
  - the only real difference was the compare-leaf admission rule:
    - raw slot membership via `compare_is_indexable(...)`
    - key-item-aware targets via `classify_index_compare_target(...).is_some()`
  - that recursion now lives behind one shared helper:
    - `classify_index_capability_with_compare(...)`
  - the two public call surfaces remain, but they now only supply the compare-leaf predicate instead of owning parallel tree walks

- This patch is intentionally structural and subtractive:
  - no widened access semantics
  - no widened EXPLAIN feature surface
  - no new access DTO
  - fewer parallel access walkers across fingerprint, EXPLAIN rendering, and executor descriptor assembly

- Validation covered:
  - `cargo test -p icydb-core query::fingerprint -- --nocapture`
  - `cargo test -p icydb-core query::explain::tests::descriptor -- --nocapture`
  - `cargo test -p icydb-core predicate::capability -- --nocapture`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`

## 0.121.1

- `0.121.1` takes the next duplicated tree-query seam after the access/projection work: ad hoc recursive queries over `ExplainExecutionNodeDescriptor`.

- Before this patch, two non-render consumers each owned their own descriptor-tree walk:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs) kept a local `contains_execution_node_type(...)` recursion to answer planner/intent questions about execution shapes
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/diagnostics/node.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/diagnostics/node.rs) kept a separate preorder walk for deterministic node-identity collection
  - both consumers were reading the same descriptor tree, but there was no shared owner for “query this execution descriptor structure”

- That query surface now lives on the descriptor owner itself:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/execution.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/execution.rs) now gives `ExplainExecutionNodeDescriptor` two shared query utilities:
    - `for_each_preorder(...)`
    - `contains_type(...)`
  - the shared traversal is now descriptor-owned rather than being open-coded by adjacent consumers

- Intent no longer owns its own recursive descriptor query:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs) deleted the local `contains_execution_node_type(...)`
  - execution-diagnostics shaping now asks `descriptor.contains_type(...)` directly when checking for shapes such as `TopNSeek`

- Diagnostics no longer owns its own preorder traversal:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/diagnostics/node.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/diagnostics/node.rs) now collects node identities by calling `root.for_each_preorder(...)`
  - deterministic node-identity collection still happens, but the tree walk now comes from the descriptor owner instead of another local helper

- This patch deliberately does not try to merge the legitimate final-output walks:
  - text render still owns its rendering walk
  - JSON render still owns its rendering walk
  - the change is narrower: semantic/query-style descriptor traversals now come from one owner instead of being duplicated in intent and diagnostics

- Validation covered:
  - `cargo test -p icydb-core execution_node_identity -- --nocapture`
  - `cargo test -p icydb-core execution_node_contains_type_scans_preorder_tree -- --nocapture`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`
