# 0.74 — Redundancy and Ownership Audit

## 0.74.8

`0.74.8` continues the read-path cleanup line with two medium-sized materialization wins: the constant-only covering SQL path no longer clones a full dense slot template per key before predicate filtering, and the generic structural SQL projection path now borrows decoded slot values from the canonical structural-row cache instead of cloning them through a local slot cache on every expression read.

What changed:

* replaced the dense per-key constant covering template clone in `db::executor::pipeline::operators::terminal::runtime` with a sparse `SqlConstantCoveringSlotTemplate`, so the constant-only covering path now evaluates residual predicates from sparse slot data and only builds the final retained slot row after the row survives filtering
* added `RetainedSlotRow::from_sparse_entries(...)` in `db::executor::terminal::page`, which lets the constant-covering path materialize the final dense retained row once from sparse entries instead of cloning a full-width `Vec<Option<Value>>` template for every candidate key
* introduced borrowed structural projection evaluators in `db::executor::projection::eval` for both generic expressions and compiled scalar expressions, using `Cow<Value>` so field-only reads can stay borrowed until an operator actually needs ownership
* added borrowed structural projection visitors in `db::executor::projection::materialize`, then switched both retained slot-row projection and structural data-row projection onto those borrowed seams
* removed the generic structural SQL projection local `slot_cache: Vec<Option<Value>>` clone path in `db::executor::projection::materialize::structural`, so structural projection now reads from `RetainedSlotRow::slot_ref(...)` or `StructuralSlotReader::required_value_by_contract_cow(...)` and only materializes owned `Value`s at the final projection-output boundary

What this means architecturally:

* constant-only covering SQL reads now keep the "predicate first, row materialization second" boundary more honestly instead of cloning a full dense slot template for rows that may be rejected immediately
* the structural SQL projection path now aligns better with the canonical structural-row contract: `StructuralSlotReader` owns validated decoded values, and projection materialization borrows from that cache instead of mirroring it in one more local owned slot cache
* the compiled scalar and generic structural projection paths now share the same borrowed reader shape, which reduces the number of parallel "owned slot reader" seams on the SQL projection path
* this is still the safe slice: it trims constant factors without changing SQL projection semantics, planner policy, grouped behavior, or the output row format

What did not change:

* no query planner policy change
* no grouped execution behavior change
* no SQL projection semantic change
* no retained slot-row output shape change
* no typed entity projection API change

Verification:

* `cargo test -p canister_test_sql_parity --lib grouped_ -- --nocapture`
* `cargo test -p icydb-core query_execution_pipeline_snapshot_for_by_key_shape_with_projection_columns_is_stable -- --nocapture`

Notes:

* a broader projection test run was started, but this environment had repeated cargo lock contention while the focused checks were running, so `0.74.8` is currently closed with targeted coverage rather than a full projection-suite pass
* `cargo clippy -p icydb-core --lib --tests -- -D warnings` was still blocked by an unrelated existing lint in `db::executor::kernel::mod`, not by the `0.74.8` changes

## 0.74.7

`0.74.7` shifts from grouped runtime cleanup into benchmark quality and measurement correctness: it widens the SQL perf-audit fixture cohort, adds more grouped SQL scenarios to the PocketIC harness, and removes the fake fixed non-grouped overhead that came from always shaping grouped fold metrics in the perf harness even when the measured query never entered grouped execution.

What changed:

* added larger perf-only fixture builders in `schema::test::sql_parity::fixtures` and a dedicated `fixtures_load_perf_audit()` canister path in `canisters::test::sql_parity`, so the PocketIC SQL perf harness can run on a broader deterministic dataset without changing the ordinary default fixture load used by the rest of the parity suite
* switched the broad PocketIC SQL perf harness in `testing::pocket_ic::tests::sql_canister` to load that perf-audit dataset and widened the grouped benchmark cohort with extra grouped `COUNT(*)` and `SUM(field)` shapes, including filtered and paged grouped windows
* recorded and saved a new current-vs-`HEAD` expanded-cohort perf report under `docs/audits/reports/2026-04/2026-04-09`, including the raw current/head JSON row sets and the full `sql-harness-expanded-delta.tsv` comparison table
* added grouped fold instrumentation to the perf harness surface so grouped SQL perf probes can report the dedicated grouped `COUNT(*)` fold counters directly on the sampled outcome
* fixed the non-grouped perf-harness wrapper tax in `canisters::test::sql_parity::perf` by splitting grouped and non-grouped sample capture: non-grouped surfaces no longer initialize grouped fold metric capture or serialize a zero-valued `grouped_count_fold_metrics` payload, while grouped surfaces still keep the full grouped fold probe
* confirmed through focused size-ladder probes that the earlier non-grouped `~1k` fixed delta was a harness-side observability artifact rather than a row-executor regression, and that the grouped regression story on the expanded fixture set is not a constant setup tax but a workload-shaped finalize/emission cost

What this means architecturally:

* the SQL perf harness now has a clearer split between "benchmark fixture realism" and the ordinary default parity fixture path, so broad PocketIC perf sampling can scale the audited query cohort without forcing the rest of the suite onto heavier baseline data
* non-grouped perf samples no longer pay grouped-observability setup or wire-shaping cost, which makes the harness closer to the real executor work for scalar/typed/load/metadata surfaces
* the grouped instrumentation is now good enough to distinguish fixed wrapper/setup cost from grouped fold/finalize work, which let the audit separate the fake non-grouped tax from the real grouped execution costs on the larger cohort
* the saved expanded-cohort report gives `0.74` a stable baseline for follow-up grouped runtime work without relying on the tiny original fixture set

What did not change:

* no grouped planner policy expansion
* no grouped output ordering change
* no grouped `DISTINCT` semantic change
* no grouped `HAVING` redesign
* no non-grouped row-executor logic change outside the perf harness path

## 0.74.6

`0.74.6` continues the cleanup line in commit, recovery, rebuild, and mutation commit-window orchestration.

What changed:

* removed `CommitGuard` marker-payload retention in `db::commit::guard`, so the in-flight commit handle now carries only persisted marker identity instead of optionally retaining `CommitMarker` row ops for finish-time consumers
* updated migration execution in `db::migration::mod` to apply step-owned row ops directly after `begin_commit_with_migration_state(...)`, and switched the reverse-index recovery test cleanup path to the existing test-only marker clear helper instead of fabricating a marker-backed commit guard
* split unrecovered store lookup from `Db::recovered_store(...)` in `db::mod`, so internal commit preparation can resolve store handles without re-entering `ensure_recovered()` and recursing through replay/rebuild preparation during recovery-backed paths
* removed the pure forwarding `Db` wrappers for `replay_commit_marker_row_ops(...)` and `rebuild_secondary_indexes_from_rows(...)`, and had `db::commit::recovery` call the canonical commit functions directly
* moved rebuild rollback ownership onto `IndexStoreSnapshot` in `db::commit::rebuild`, adding snapshot-owned `capture(...)` and `restore(...)` behavior instead of free helper plumbing, and centralized index mutation application on `PreparedIndexMutation::apply(self)` in `db::commit::apply`
* collapsed the remaining `commit_window` generation-guard helper split by moving capture and verify behavior onto `IndexStoreGenerationGuard`, then removed the free batch snapshot/verify helpers
* removed the duplicate `SingleRowRollbackGuard` from `executor::mutation::commit_window` and extended `CommitApplyGuard` so one guard now owns both the batch rollback closure path and the single-row prepared-op unwind rollback path
* collapsed the shared single-row prepared-op apply sequence in `executor::mutation::commit_window` onto one helper, then shared the multi-row overlay-driven preflight loop between typed and structural commit-window preparation
* centralized the structural delete metric shaping and common zero/delete-only `PreparedRowOpDelta` shapes in `executor::mutation::commit_window`

What this means architecturally:

* `CommitGuard` is now closer to its real ownership boundary: persisted commit identity plus marker-clear responsibility, not retained marker payload carriage
* commit and recovery orchestration now goes through fewer forwarding shells on `Db`, keeping replay/rebuild authority inside the `commit` subsystem instead of mirroring it at the root DB boundary
* rebuild rollback and index-mutation apply behavior now live on the types that actually own those mechanics
* `commit_window` now carries fewer local helper taxonomies: generation guards own generation invariants, `CommitApplyGuard` owns best-effort apply rollback, single-row prepared-op commit sequencing is shared, and the typed/structural preflight paths share one overlay-driven loop

What did not change:

* no mutation planner policy change
* no commit-marker durability model change
* no recovery behavior widening
* no query planner or grouped-planning feature expansion

`0.74.5` finishes the explain-side cleanup and takes one matching route-payload cleanup outside grouped.

What changed:

* replaced the dual grouped hashing entrypoints in `db::query::fingerprint::hash_parts` with one canonical grouped fingerprint projection shape that both plan-owned and explain-owned hashing now project into before any grouped fingerprint bytes are emitted
* removed the logical-explain grouped fallback mirror and the logical-explain grouped strategy mirror in `db::query::explain::plan`, replacing both with planner-owned stable codes on the explain surface
* removed the aggregate-only `ExplainAggregateTerminalRoute` layer from `db::query::explain::execution`, so aggregate explain now relies on the canonical execution ordering source instead of storing a second coarser route enum derived from it
* collapsed repeated grouped/load route-property projection in `db::executor::explain::descriptor::load` onto shared helper seams, so verbose diagnostics and node-property JSON now consume one set of route-owned codes instead of re-encoding the same grouped and ordered-load facts separately
* moved aggregate ordering-source projection into the shared descriptor seam in `db::executor::explain::descriptor::shared`, so aggregate descriptor assembly no longer keeps its own route-to-ordering-source match alongside the shared execution-mode projection
* removed extra orchestrator wrappers in `db::executor::pipeline::orchestrator::strategy::grouping`, including the local `ExecutionMode` and `KernelPayload` enums, so post-projection route/orchestrator payload now carries `PreparedLoadCursor` and `LoadExecutionPayload` directly instead of wrapping and immediately unwrapping the same execution facts again
* updated grouped/session/executor/fingerprint snapshot and parity locks to freeze the new canonical explain codes and slimmer runtime payload shapes

What this means architecturally:

* planner/query fingerprint semantics now have one canonical grouped projection seam
* logical explain now projects planner-owned grouped strategy and fallback identity through stable codes instead of explain-owned shadow enums
* aggregate explain now projects one canonical execution ordering source instead of carrying a second aggregate-only route label derived from the same underlying fact
* load execution descriptors now project grouped and ordered-load route facts through shared helper seams instead of duplicating those encodings across verbose and JSON descriptor paths
* the non-grouped load orchestrator now carries execution payload kinds directly after projection, which is closer to the `0.74` rule that post-projection route/runtime payload should carry execution inputs only

What did not change:

## 0.74.0

`0.74.0` starts the cleanup line with an explicit grouped redundancy inventory instead of immediate broad deletion.

What changed:

* tightened the main `0.74` design around one grouped-first audit method
* added a dedicated grouped redundancy inventory note with file-by-file ownership classification
* recorded the first grouped kill list against concrete files and seams

The grouped inventory now classifies the current grouped subsystem into:

* canonical semantic source
* canonical carriage
* projection
* capability enforcement
* runtime mechanics

That inventory then names the first cleanup targets:

* collapse the grouped route eligibility wrapper in `executor::route::planner::feasibility::grouped_strategy`
* rename `GroupedPlanStrategy::streaming_admitted()` because grouped runtime remains materialized and the current name preserves an older ownership model
* re-evaluate whether `GroupedPlanAggregateFamily::{StorageKeyTerminals, Mixed}` are real runtime families or only descriptive planner buckets
* trim grouped runtime family guardrails once grouped family routing is fully planner-owned
* revisit the split grouped fingerprint projection seams if one grouped-strategy projection path can serve both plan and explain callsites

What did not change:

* no grouped family widening
* no grouped `DISTINCT` redesign
* no grouped `HAVING` redesign
* no planner ambition work
* no broad dead-code deletion by analogy

The main `0.74` rule is now explicit:

> planner owns semantics; runtime consumes; explain, fingerprint, and continuation project

Anything outside that boundary is now a cleanup candidate rather than a tolerated architectural leftover.
