# 0.74 — Redundancy and Ownership Audit

## 0.74.5

`0.74.5` takes the grouped fingerprint/explain seam that was still carrying parallel semantic projection and collapses it onto one canonical grouped hash projection.

What changed:

* replaced the dual grouped hashing entrypoints in `db::query::fingerprint::hash_parts` with one canonical grouped fingerprint projection shape that both plan-owned and explain-owned hashing now project into before any grouped fingerprint bytes are emitted
* switched explain-side grouped continuation hashing to reuse the planner aggregate-family classifier, so grouped continuation signatures no longer lose grouped aggregate-family identity when hashing from `ExplainGrouping`
* added grouped parity locks in `db::query::fingerprint::tests` and `db::query::fingerprint::shape_signature::tests` proving that grouped fingerprint and grouped continuation identity remain stable across the plan-owned and explain-owned projection surfaces
* updated the grouped redundancy inventory to record that grouped fingerprinting still has two top-level entrypoints, but no longer has two grouped semantic projection rules

What this means architecturally:

* planner/query fingerprint semantics now have one canonical grouped projection seam
* explain still projects grouped semantics, but it no longer owns a second grouped aggregate-family interpretation for hashing
* grouped fingerprint identity and grouped continuation identity now agree across plan-owned and explain-owned callsites without changing grouped semantic snapshots

What did not change:

* no grouped family widening
* no grouped `DISTINCT` redesign
* no grouped `HAVING` redesign
* no grouped planner policy expansion

## 0.74.4

`0.74.4` makes the grouped planner-versus-runtime boundary harder to bypass after the earlier naming and projection cleanup.

What changed:

* introduced one explicit `GroupedFoldPath` contract on the grouped planner handoff so the dedicated grouped `COUNT(*)` fold path is now planner-carried instead of being rediscovered from `GroupedPlanStrategy` inside grouped fold runtime
* removed `grouped_plan_strategy` from grouped route-stage payload carriage, so grouped runtime stages now keep only execution-mechanical grouped artifacts after route projection is complete
* switched grouped fold runtime in `executor::aggregate::runtime::grouped_fold` to consume `grouped_fold_path()` directly for the dedicated grouped `COUNT(*)` split
* added grouped handoff locks for the new fold-path projection and source-level tripwires that fail if grouped fold runtime or grouped route-stage payloads start consulting planner strategy directly again
* synced the grouped redundancy inventory so the remaining watch items reflect the new non-bypassable grouped runtime seam

What this means architecturally:

* planner still owns grouped semantics through `GroupedPlanStrategy`
* grouped runtime now sees the planner-projected fold-path contract instead of planner strategy itself for the remaining dedicated grouped fold split
* route-stage payloads no longer preserve a second semantic escape hatch after grouped execution mode and grouped fold path have already been projected
* the grouped boundary is narrower now: planner emits semantics, route projects execution artifacts, runtime consumes those artifacts

What did not change:

* no grouped family widening
* no grouped `DISTINCT` redesign
* no grouped `HAVING` redesign
* no grouped planner policy expansion

* finished the last route/runtime wording cleanup around grouped mode mapping and synced the grouped redundancy inventory to record the new boundary assertions

What this means architecturally:

* grouped route projection is now both centralized and checked
* planner still owns grouped semantics through `GroupedPlanStrategy`
* route now projects grouped execution mode once and asserts that the carried route artifact matches that projection
* grouped explain and grouped metrics stay aligned with the same route-owned execution mode vocabulary without adding another route-local classifier

What did not change:

* no grouped `DISTINCT` redesign
* no grouped `HAVING` redesign

## 0.74.2

`0.74.2` tightens the grouped planner-versus-route boundary after the first cleanup pass.

What changed:

* moved the unsupported grouped field-target invariant out of grouped fold engine init and into grouped aggregate-state construction, so grouped runtime setup no longer re-checks planner-owned family routing while bypassed unsupported field-target kinds still fail closed
* renamed the grouped route-owned materialization label from `GroupedExecutionStrategy` to `GroupedExecutionMode` across route observability, execution explain, runtime carriage, and metrics projection
* removed the silent grouped observability fallback that used to invent `HashMaterialized` when grouped execution mode was missing, so grouped routes now require the route-projected grouped mode artifact to be present
* introduced one route-owned `GroupedExecutionModeProjection` contract and centralized grouped mode selection on `GroupedExecutionMode::from_planner_strategy(...)`
* deleted the last route-local grouped feasibility wrapper module after that projection became a direct route contract instead of one more helper seam
* cleaned the remaining grouped route/runtime comments and metric wording so only planner-owned grouped artifacts still use `strategy` terminology

What this means architecturally:

* planner still owns grouped semantics through `GroupedPlanStrategy`
* route now owns one explicit grouped execution-mode projection contract instead of loose booleans and helper wrappers
* execution explain and metrics consume grouped execution mode directly, while logical explain continues to consume planner-owned grouped strategy
* grouped route observability is stricter now: missing grouped mode is treated as an invariant failure instead of being papered over

What did not change:

* no grouped family widening
* no grouped `DISTINCT` redesign
* no grouped `HAVING` redesign
* no grouped planner policy expansion

## 0.74.1

`0.74.1` is the first real grouped cleanup pass after the inventory cut.

What changed:

* renamed `GroupedPlanStrategy::streaming_admitted()` to `ordered_group_admitted()` so the grouped planner artifact no longer implies a runtime streaming contract that does not exist
* deleted the `GroupedOrderedEligibility` wrapper in `executor::route::planner::feasibility::grouped_strategy` and projected grouped route eligibility directly from the planner-owned grouped strategy plus the remaining route capability gates
* collapsed `GroupedPlanAggregateFamily::{StorageKeyTerminals, Mixed}` into `GenericRows`, leaving only the grouped execution buckets that runtime actually distinguishes
* removed the executor-local grouped metrics shadow enum and carried the metrics sink grouped strategy label directly through grouped route/runtime projection
* removed the dead grouped `COUNT(*)` engine-init invariant and narrowed the remaining field-target guard so `GenericRows` no longer fails just because one aggregate targets a field, while unsupported grouped field-target kinds still fail closed at the executor boundary
* removed raw grouped `AggregateExpr` carriage from grouped runtime projection matching in favor of planner-owned grouped projection/execution specs
* reduced grouped fingerprint duplication by sharing the grouped strategy and grouped `HAVING` hashing helpers across the plan and explain entrypoints instead of keeping two separate encoders

What this means architecturally:

* grouped planner/runtime/explain/fingerprint paths now depend on fewer parallel grouped abstractions
* grouped runtime still keeps its real guardrails, but fewer of them restate planner-owned semantics
* grouped cleanup is now deleting actual compatibility residue instead of only documenting where it lives

What did not change:

* no grouped family widening
* no grouped `DISTINCT` redesign
* no grouped `HAVING` redesign
* no grouped planner policy expansion

## 0.74.0

`0.74.0` starts the cleanup line with an explicit grouped redundancy inventory instead of immediate broad deletion.

What changed:

* tightened the main `0.74` design around one grouped-first audit method
* added a dedicated grouped redundancy inventory note with file-by-file ownership classification
* recorded the first grouped kill list against concrete files and seams

The grouped inventory now classifies the current grouped subsystem into:

* canonical semantic source
* canonical carriage
* projection
* capability enforcement
* runtime mechanics

That inventory then names the first cleanup targets:

* collapse the grouped route eligibility wrapper in `executor::route::planner::feasibility::grouped_strategy`
* rename `GroupedPlanStrategy::streaming_admitted()` because grouped runtime remains materialized and the current name preserves an older ownership model
* re-evaluate whether `GroupedPlanAggregateFamily::{StorageKeyTerminals, Mixed}` are real runtime families or only descriptive planner buckets
* trim grouped runtime family guardrails once grouped family routing is fully planner-owned
* revisit the split grouped fingerprint projection seams if one grouped-strategy projection path can serve both plan and explain callsites

What did not change:

* no grouped family widening
* no grouped `DISTINCT` redesign
* no grouped `HAVING` redesign
* no planner ambition work
* no broad dead-code deletion by analogy

The main `0.74` rule is now explicit:

> planner owns semantics; runtime consumes; explain, fingerprint, and continuation project

Anything outside that boundary is now a cleanup candidate rather than a tolerated architectural leftover.
