# 0.74 — Redundancy and Ownership Audit

## 0.74.12

`0.74.12` finishes the remaining non-path-specific projection/materialization cleanup by consolidating generic prepared projection ownership, deleting the last duplicated structural SQL fallback orchestration, and then tightening that cleanup so the projected-row fast path does not pay structural fallback preparation cost it never uses.

What changed:

* added executor-owned `PreparedProjectionShape` in `db::executor::projection::materialize`, so generic projection preparation now freezes canonical `ProjectionSpec`, compiled projection materialization plan, projection-identity flag, direct projection slot layout, direct field-slot layout, and projected-slot mask in one shared prepared object instead of compiling adjacent local views in multiple executor boundaries
* changed `PreparedSlotProjectionValidation` into that shared prepared projection contract, so slot-row validation and structural SQL fallback materialization now project from the same prepared shape instead of carrying a second validation-only bundle beside the SQL-oriented prepared state
* changed `PreparedExecutionProjection` in `db::executor::pipeline::contracts::execution::inputs` to own one optional generic prepared projection shape plus its validation-enabled flag, and then had the SQL short-path preparation in `db::executor::pipeline::operators::terminal::runtime` consume that shape when it needs already-derived direct projection slots
* removed the old `PreparedStructuralSqlProjection` helper from `db::executor::projection::materialize::structural`, so the structural SQL fallback lane no longer re-prepares its own direct field-slot layout, slot mask, and prepared projection plan after the executor has already prepared the same generic projection semantics elsewhere
* collapsed `execute_sql_projection_rows_for_canister(...)` and `execute_sql_projection_text_rows_for_canister(...)` onto one shared structural SQL page-materialization helper in `db::executor::projection::materialize::structural`, leaving the value-vs-text split only at the final rendered-row boundary instead of duplicating the same projected-row / slot-row / data-row fallback ladder twice
* tightened that consolidation immediately afterward by making the shared structural SQL prepared shape lazy again: the projected-row fast path now returns directly from the scalar page when projected rows are already present, and only prepares the shared structural fallback shape when the page actually falls back to retained slot rows or persisted data rows
* removed the last now-redundant non-path-specific helpers and re-exports that only existed for the old split shape, including the standalone structural `direct_projection_slot_mask(...)` helper and the extra projection-module re-export for direct field-slot lowering

What this means architecturally:

* generic projection preparation now has one real owner instead of three adjacent partial owners; execution-input preparation, slot-row validation, and structural SQL fallback all consume the same prepared projection shape
* structural SQL fallback is cleaner too: value rows and rendered text rows still differ at the public output boundary, but they now share one internal page-to-projection materialization ladder instead of mirroring the same projected-row / slot-row / data-row control flow in parallel entrypoints
* the executor still keeps SQL-specific prepared metadata for route-covering and constant-covering short paths, but the generic projection semantics that sit underneath those paths are no longer duplicated on the structural fallback side
* this patch deliberately stays non-path-specific: it cleans ownership and repeated preparation, but it does not widen planner policy or introduce any new route-specific fast path families

Measured impact:

* before the lazy-follow-up, the generic cleanup refactor itself moved the `SELECT id FROM Customer ORDER BY id ASC LIMIT 1` 5-sample PocketIC average on `TypedDispatchCustomer` up to `321,325`, which made it clear that preparing the shared structural fallback shape eagerly was taxing the projected-row fast path
* after making that shared structural fallback preparation lazy again, the same 5-sample average settled at `313,808`
* against the original `0.74` baseline of `340,561`, that final post-cleanup average is still down `26,753` instructions, about `7.9%`
* this means `0.74.12` is best understood as a structural cleanup patch that preserves a real win over baseline while keeping the generic projection/materialization ownership model cleaner for later work

* `0.74.12` is the stopping point for the non-path-specific cleanup line in this area; the remaining work is no longer generic consolidation

## 0.74.11

`0.74.11` finishes the tiny-query fixed-cost cleanup line by turning SQL projection metadata into prepared execution state instead of execute-time reconstruction. The terminal SQL short path now borrows one prepared projection bundle that already owns direct projection slots, prepared projected-source layouts, and constant-covering slot templates, so the direct covering `SELECT id FROM Customer ORDER BY id ASC LIMIT 1` lane no longer rebuilds those shapes inside execute.

What changed:

* added executor-owned `PreparedExecutionProjection` and `ExecutionInputPreparation` in `db::executor::pipeline::contracts::execution::inputs`, so execution inputs now carry one explicit prepared projection bundle instead of recomputing retained-slot and SQL projection metadata ad hoc at each execution boundary
* added `PreparedSqlExecutionProjection` plus `prepare_sql_execution_projection(...)` in `db::executor::pipeline::operators::terminal::runtime`, so SQL short-path preparation now freezes direct projection slots, route-covering slot layout, single-component projected-source layout, constant projected-source layout, and constant-covering slot templates before execution begins
* threaded that prepared SQL bundle through the executor boundary in `db::executor::pipeline::operators`, `db::executor::pipeline::contracts`, `db::executor::terminal::page`, and the scalar/aggregate execution inputs, so terminal runtime consumes prepared projection metadata through the same normal execution request envelopes rather than reaching back into plan-owned derivation helpers
* moved scalar prepared-state ownership earlier in `db::executor::pipeline::entrypoints::scalar`, so prepared scalar runtime now carries both `ExecutionPreparation` and `PreparedExecutionProjection`, and the shared scalar execution spine no longer rebuilds those fixed-cost inputs once it enters execute
* changed the cursorless SQL covering/projected paths in `db::executor::pipeline::operators::terminal::runtime` to use the prepared bundle directly for direct projection slots, direct projected-source layouts, single-component projected-source layouts, constant projected-source layouts, and constant-covering slot templates
* removed the execute-time self-validation from those SQL projected-row/source-table paths by validating the self-authored layouts once during preparation instead of rechecking them inside the hot row materialization helpers
* kept the public/runtime boundary narrow by re-exporting the prepared SQL contract from the pipeline operator root rather than having downstream executor code reach through a private terminal submodule

What this means architecturally:

* execute now treats SQL projection/source metadata as immutable prepared runtime state instead of rebuilding it from `AccessPlannedQuery` and route contracts during terminal materialization
* the terminal SQL short path is closer to a real hard cut now: it still executes row scans and final materialization, but the projection/source/layout derivation work moved out of the hot path
* the executor boundary is cleaner too, because scalar and aggregate callers now prepare one explicit execution-input bundle instead of widening `ExecutionInputs::new(...)` with more and more loose preparation arguments

Measured impact:

* the original tiny query benchmark, `SELECT id FROM Customer ORDER BY id ASC LIMIT 1` on `TypedDispatchCustomer`, moved from the pre-refactor `312,117` local instructions to `306,514`, a reduction of `5,603` instructions, about `1.8%`

* the attribution probe after this patch reports `305,881` total local instructions with `parse=25,872`, `lower=21,223`, `dispatch=615`, `execute=240,148`, and `wrapper=18,023`, while still staying on the same projected-row short path (`projected_rows_path_hits = 1`)

## 0.74.10

`0.74.10` shifts the `0.74` cleanup line from grouped-only redundancy into tiny-query fixed-cost cleanup, with a hard cut away from runtime schema validation for generated metadata and a series of parser/executor fixed-cost reductions on the direct SQL projection path. Generated `EntityModel`, `IndexModel`, and `FieldModel` metadata is now treated as trusted build-time output, filtered-index predicate parsing moved to derive/build time, and several repeated SQL/frontend and executor setup costs were deleted before the later prepared-state refactor landed.

What changed:

* changed `EntityModel`, `IndexModel`, and `FieldModel` construction into generated/trusted metadata surfaces in `model::entity`, `model::index`, `model::field`, and the matching derive emitters, so runtime planning no longer treats those generated models as ad hoc runtime-built values that must be fully revalidated on every query
* replaced repeated planner-side `SchemaInfo::from_entity_model(...)` validation with a trusted cached schema view in `db::schema::info`, then switched planner, executor validation, mutation-save validation, and cursor boundary consumers onto that trusted cached surface
* moved filtered-index predicate parsing and validation out of runtime and into macro/build code, so `IndexModel` now carries canonical predicate metadata and the runtime `db::index::predicate` path no longer lazily reparses raw SQL predicate text during index-membership checks
* narrowed the legacy compatibility surface by removing invalid-runtime-model test scaffolding and dead runtime validation modules that only existed for the old manually-constructible model story
* trimmed reduced-SQL frontend fixed cost in `db::reduced_sql` by removing the temporary uppercase allocation in keyword matching, removing an extra byte-copy while building identifier and number tokens, and then cutting `SqlTokenCursor` token-kind clone overhead from the parser cursor path
* fixed the `EXPLAIN EXECUTION` lexer regression introduced during that parser cleanup and kept the reduced-SQL keyword matcher on one explicit ASCII-insensitive fast path
* cleaned several generic executor setup costs in `db::executor::pipeline::operators::terminal::runtime` and adjacent projection/runtime helpers before the final prepared-state cut: impossible `Result` wrappers on self-authored constant projected-row helpers were removed, projection-source/constant tables were split more cleanly, redundant reorder/strip passes were skipped when traversal order already matched logical order, and more direct-slot projection work was compiled once instead of rediscovered through broader projection-spec reconstruction
* centralized the remaining covering-read eligibility and explain-reason projection seams under `db::query::plan::covering`, then had executor explain and route/terminal consumers project from that planner-owned helper instead of recomputing local copies
* finished several adjacent `0.74` cleanup tasks while the executor/perf line was open: thinner `SchemaInfo` compatibility residue, shared direct-projection admissibility logic, slimmer error payloads for strict clippy, and a full `cargo clippy --workspace --all-targets -- -D warnings` cleanup pass across the affected runtime surface

What this means architecturally:

* generated schema metadata is now a real trust boundary: build/derive code validates and emits canonical runtime descriptors, and query execution consumes those trusted descriptors instead of redoing the same validation on every query
* filtered-index predicate semantics now follow the same rule, with derive/build time owning predicate parse/validation and runtime consuming canonical predicate metadata only
* parser and terminal/runtime cleanup in this patch were still generic fixed-cost reductions rather than new planner policy or new route families, which kept the optimization line safe while moving the tiny-query floor materially downward before the later prepared-state refactor
* the DB/explain/covering ownership cleanup in this patch also leaves fewer duplicated semantic seams behind, which made the later prepared execution-state cut easier to land cleanly

Measured impact:

## 0.74.9

`0.74.9` finishes the grouped executor cleanup line by replacing the last engine-centric finalize/materialize flow with one shared grouped state bundle and a page-oriented grouped finalize spine. The grouped runtime no longer round-trips through per-engine finalized vectors, sibling key re-alignment, or a separate grouped candidate buffer before paging, and bounded grouped windows now use bounded canonical-key selection instead of sorting every finalized group up front.

What changed:

* added a shared `GroupedAggregateBundle` in `db::executor::aggregate::runtime::grouped_fold::bundle` that stores one group table with all aggregate slot states for that group, replacing the old per-engine grouped finalize shape with one group-centric runtime surface
* moved generic grouped ingest onto that shared bundle in `db::executor::aggregate::runtime::grouped_fold`, so grouped rows now resolve one group entry and update that group’s aggregate slot states in place instead of maintaining separate grouped state per aggregate engine
* restored a dedicated single-aggregate finalize path on top of the shared bundle, so the group-centric runtime shape does not force the common one-aggregate grouped case back through generic aggregate-vector materialization
* fused generic grouped finalize, grouped `HAVING`, continuation resume filtering, offset/limit paging, and grouped row materialization in `db::executor::aggregate::runtime::grouped_fold::page_finalize`, deleting the old grouped candidate-row buffer boundary and moving grouped page ownership onto one pass
* changed bounded generic grouped pages to retain only the smallest canonical `offset + limit + 1` groups in a heap before final page emission, so paged grouped queries no longer pay a full canonical sort across every finalized group when the grouped continuation window is bounded
* changed grouped finalize plumbing to carry owned `GroupKey` values through the fused page stage, which lets bounded grouped selection compare and retain canonical group order without re-cloning grouped key payloads back out of finalized rows
* compiled grouped projection was already in place from the previous patch; `0.74.9` finishes that story by keeping projection downstream of the fused grouped page stage, so grouped projection now runs only on the emitted grouped page rows instead of on intermediate candidate buffers
* added focused grouped SQL regression coverage for multi-aggregate grouped queries that combine `HAVING`, `LIMIT`, `OFFSET`, and cursor resume, locking the hardest fused grouped page-control path onto explicit continuation behavior

What this means architecturally:

* generic grouped execution is now group-centric end to end: ingest updates one shared group table, finalize walks grouped rows instead of aggregate engines, bounded grouped pages select only the needed canonical window, and grouped projection runs after the final page window is known
* the remaining grouped work is mostly constant-factor tuning rather than another execution-model rewrite; the main structural redundancy from `engine finalize -> align -> candidate rows -> page rows` is gone

Validation:

* `cargo fmt --all`
* `cargo clippy -p icydb-core --all-targets -- -D warnings`
* `cargo test -p icydb-core execute_sql_grouped_limit_window_emits_cursor_and_resumes_next_group_page -- --nocapture`
* `cargo test -p icydb-core execute_sql_grouped_multi_aggregate_having_offset_limit_cursor_resumes_consistently -- --nocapture`
* `cargo test -p icydb-core query_from_sql_indexed_grouped_explain_and_execution_project_ordered_group_publicly -- --nocapture`

Notes:

* unbounded grouped pages still sort the full qualifying grouped set, which is intentional because those queries need the full canonical grouped order rather than a bounded page window
* bounded grouped selection now shifts some cost into canonical `GroupKey` comparison inside the heap comparator, so any further grouped hot-path work is likely to be constant-factor tuning around canonical key ordering rather than another large structural deletion

## 0.74.8

`0.74.8` broadens the executor hot-path cleanup line beyond the earlier structural projection slice. This patch removes several repeated per-row and per-group costs across grouped aggregation, ordered structural reads, residual-filter retry, ranking terminals, and SQL row materialization, with the biggest wins coming from replacing owned grouped-key construction on hot grouped paths, deleting redundant grouped sorting work, making retained slot access constant-time, and stopping ranking terminals from cloning whole materialized rows just to rank one field.

What changed:

* replaced grouped aggregate-state lookup in `db::executor::aggregate::contracts::grouped::engine` from a `BTreeMap<stable_hash, bucket>` plus bucket scan to direct `HashMap<GroupKey, ...>` keyed lookup, while keeping grouped output order deterministic through the existing finalized sort boundary
* added borrowed grouped-key probe paths for both dedicated grouped `COUNT(*)` and generic grouped ingest in `db::executor::aggregate::runtime::grouped_fold`, so existing-group hits no longer build `Vec<Value> -> Value::List -> GroupKey` on every row and now only materialize owned grouped keys when a group is first inserted or the structured fallback needs ownership
* replaced the generic grouped short-circuit `done_group_keys` linear scan in `db::executor::aggregate::runtime::grouped_fold::ingest` with hashed membership buckets keyed by the same stable grouped hash contract, which removes the old `rows * engines * completed_groups` scan shape for most completed-group checks
* removed the redundant unbounded grouped candidate sink sort and changed grouped `COUNT(*)` page finalize onto bounded top-k retention in `db::executor::aggregate::runtime::grouped_fold::candidate_rows` and `db::executor::aggregate::runtime::grouped_fold`, so bounded grouped windows no longer flatten and fully sort every surviving group before applying the final page window
* reused one scratch aggregate-value buffer in grouped candidate finalize instead of allocating one `Vec<Value>` per emitted group before sink retention
* compiled grouped projection once in `db::executor::projection::grouped` and switched grouped output materialization onto that compiled seam, removing the old per-row scans over grouped field names and grouped aggregate execution specs during grouped projection evaluation
* replaced sparse retained-slot lookup in `db::executor::terminal::page` with a dense retained row representation, so structural predicates, ordering, and projection now get `O(1)` slot reads instead of repeated linear scans over `(slot, value)` pairs
* precomputed structural order keys in `db::executor::order` before bounded selection and final sort, which removes repeated slot reads and repeated `LOWER(...)` / `UPPER(...)` derivation from the ordering comparator hot loop
* changed residual-filter retry in `db::executor::kernel` and `db::executor::route::hints::load` from one bounded probe plus immediate unbounded fallback to adaptive bounded widening, so selective residual paths can keep expanding a bounded index-backed probe instead of paying an avoidable second full scan
* replaced the ranking terminal’s old incremental sorted insertion algorithm in `db::executor::terminal::ranking::materialized` with bounded nth-style selection, then removed the remaining `(data_key.clone(), raw_row.clone())` ranking-candidate cloning by decoding only the ranked slot from borrowed persisted rows and carrying `(StorageKey, row_index)` plus the ranked value through the bounded ranking window
* replaced the dense per-key constant covering template clone in `db::executor::pipeline::operators::terminal::runtime` with a sparse `SqlConstantCoveringSlotTemplate`, so the constant-only covering path now evaluates residual predicates from sparse slot data and only builds the final retained slot row after the row survives filtering
* added `RetainedSlotRow::from_sparse_entries(...)` in `db::executor::terminal::page`, which lets the constant-covering path materialize the final dense retained row once from sparse entries instead of cloning a full-width `Vec<Option<Value>>` template for every candidate key
* introduced borrowed structural projection evaluators in `db::executor::projection::eval` for both generic expressions and compiled scalar expressions, using `Cow<Value>` so field-only reads can stay borrowed until an operator actually needs ownership
* added borrowed structural projection visitors in `db::executor::projection::materialize`, then switched both retained slot-row projection and structural data-row projection onto those borrowed seams
* removed the generic structural SQL projection local `slot_cache: Vec<Option<Value>>` clone path in `db::executor::projection::materialize::structural`, so structural projection now reads from `RetainedSlotRow::slot_ref(...)` or `StructuralSlotReader::required_value_by_contract_cow(...)` and only materializes owned `Value`s at the final projection-output boundary

What this means architecturally:

* grouped execution now has a more consistent keying story across the dedicated grouped `COUNT(*)` path and the generic grouped aggregate path: both can probe existing groups from borrowed row data before paying the owned grouped-key allocation cost
* grouped page retention now does less unnecessary global work, because the grouped count fast path can retain a bounded window without a full-result sort and the generic candidate sink no longer re-sorts rows that are already finalized in canonical order
* grouped projection is closer to scalar projection now: both compile once into executor-owned runtime forms instead of resolving field/aggregate names dynamically on every emitted grouped row
* the structural read path is now more internally consistent too: retained rows use dense indexed slot access, ordering compares cached sort keys instead of live row reads, and structural SQL projection borrows from the canonical slot cache instead of cloning one more local cache
* residual-filter retry now keeps more selective ordered/index-backed reads on bounded retry loops before paying the older unbounded fallback, which better matches the bounded-window execution model already used elsewhere in the executor
* ranking terminals now use the same general algorithm quality as the shared bounded ordering path and carry far less row payload through that path when the final projection only needs ids or ranked values
* constant-only covering SQL reads now keep the "predicate first, row materialization second" boundary more honestly instead of cloning a full dense slot template for rows that may be rejected immediately
* the structural SQL projection path now aligns better with the canonical structural-row contract: `StructuralSlotReader` owns validated decoded values, and projection materialization borrows from that cache instead of mirroring it in one more local owned slot cache
* the compiled scalar and generic structural projection paths now share the same borrowed reader shape, which reduces the number of parallel "owned slot reader" seams on the SQL projection path
* this is still the safe slice: it trims constant factors and selection algorithms without changing SQL projection semantics, grouped query semantics, planner policy, or public output row formats

What did not change:

* no query planner policy change
* no grouped query semantic change
* no SQL projection semantic change
* no retained slot-row output shape change
* no typed entity projection API change
* no grouped output ordering change
* no grouped `HAVING` redesign
* no grouped family widening
