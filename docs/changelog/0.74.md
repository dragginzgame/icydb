# 0.74 — Redundancy and Ownership Audit

## 0.74.1

`0.74.1` is the first real grouped cleanup pass after the inventory cut.

What changed:

* renamed `GroupedPlanStrategy::streaming_admitted()` to `ordered_group_admitted()` so the grouped planner artifact no longer implies a runtime streaming contract that does not exist
* deleted the `GroupedOrderedEligibility` wrapper in `executor::route::planner::feasibility::grouped_strategy` and projected grouped route eligibility directly from the planner-owned grouped strategy plus the remaining route capability gates
* collapsed `GroupedPlanAggregateFamily::{StorageKeyTerminals, Mixed}` into `GenericRows`, leaving only the grouped execution buckets that runtime actually distinguishes
* removed the executor-local grouped metrics shadow enum and carried the metrics sink grouped strategy label directly through grouped route/runtime projection
* removed the dead grouped `COUNT(*)` engine-init invariant and narrowed the remaining field-target guard so `GenericRows` no longer fails just because one aggregate targets a field, while unsupported grouped field-target kinds still fail closed at the executor boundary
* removed raw grouped `AggregateExpr` carriage from grouped runtime projection matching in favor of planner-owned grouped projection/execution specs
* reduced grouped fingerprint duplication by sharing the grouped strategy and grouped `HAVING` hashing helpers across the plan and explain entrypoints instead of keeping two separate encoders

What this means architecturally:

* grouped planner/runtime/explain/fingerprint paths now depend on fewer parallel grouped abstractions
* grouped runtime still keeps its real guardrails, but fewer of them restate planner-owned semantics
* grouped cleanup is now deleting actual compatibility residue instead of only documenting where it lives

What did not change:

* no grouped family widening
* no grouped `DISTINCT` redesign
* no grouped `HAVING` redesign
* no grouped planner policy expansion

## 0.74.0

`0.74.0` starts the cleanup line with an explicit grouped redundancy inventory instead of immediate broad deletion.

What changed:

* tightened the main `0.74` design around one grouped-first audit method
* added a dedicated grouped redundancy inventory note with file-by-file ownership classification
* recorded the first grouped kill list against concrete files and seams

The grouped inventory now classifies the current grouped subsystem into:

* canonical semantic source
* canonical carriage
* projection
* capability enforcement
* runtime mechanics

That inventory then names the first cleanup targets:

* collapse the grouped route eligibility wrapper in `executor::route::planner::feasibility::grouped_strategy`
* rename `GroupedPlanStrategy::streaming_admitted()` because grouped runtime remains materialized and the current name preserves an older ownership model
* re-evaluate whether `GroupedPlanAggregateFamily::{StorageKeyTerminals, Mixed}` are real runtime families or only descriptive planner buckets
* trim grouped runtime family guardrails once grouped family routing is fully planner-owned
* revisit the split grouped fingerprint projection seams if one grouped-strategy projection path can serve both plan and explain callsites

What did not change:

* no grouped family widening
* no grouped `DISTINCT` redesign
* no grouped `HAVING` redesign
* no planner ambition work
* no broad dead-code deletion by analogy

The main `0.74` rule is now explicit:

> planner owns semantics; runtime consumes; explain, fingerprint, and continuation project

Anything outside that boundary is now a cleanup candidate rather than a tolerated architectural leftover.
