# IcyDB 0.213: Schema Authority And Application Separation

## 0.213.0

### 📝 Summary

- Opens the 0.213 architecture line by separating the pure public schema
  contract from generated application-model metadata and database runtime
  authority.
- Adds bounded exact proposal composition and a read-only target context for
  future optimistic catalog-native application.
- Does not yet add schema mutation through `SchemaProposal`, durable receipts,
  or multi-store schema publication.

### ➕ Added

- Reclaimed `icydb-schema` as the standalone public proposal and canonical
  scalar leaf. It has no dependency on the database runtime, generated model
  graph, storage, recovery, clocks, generators, or application callbacks.
- Added bounded immutable source identities for entities, fields, named types,
  constraints, indexes, and relations.
- Added a current version-1 proposal envelope with explicit capability
  requirements, database/store routing tokens, an immutable submission key,
  an exact expected accepted head, explicit removals, and deterministic
  canonical encoding and digesting.
- Added exact scalar field contracts, source-keyed enum literals, bounded
  accepted-check source expressions, structural record fields, database insert
  policy, generated-value intent, and managed timestamp declarations.
- Added deterministic database-scoped composition with duplicate-definition,
  name, routing, removal, type-graph, enum-literal, relation-closure, and
  capability validation.
- Added `DbSession::schema_application_target`, which issues an opaque database
  identity, canonically ordered store-routing identities, and one exact
  multi-store accepted head from recovered runtime authority.

### 🔧 Changed

- Moved the previous `icydb-schema` authoring graph intact to the unpublished
  temporary `icydb-model-legacy` package so the reclaimed package cannot mix
  the retired graph with the new proposal contract.
- Made `icydb-schema` the single Rust representation owner for canonical
  scalar atoms. `icydb-core` retains runtime operations and re-exports those
  same representations rather than defining parallel scalar types.
- Split exact proposal composition from durable catalog application.
  Composition validates untrusted source-keyed intent but allocates no accepted
  identities, publishes no schema, and owns no marker or recovery behavior.
- Database identities are bound to the durable database incarnation and the
  complete registered store topology. Store identities additionally cover
  their path, capabilities, and stable allocations.
- The optimistic accepted head covers every store-local accepted root in
  canonical path order, including explicit absence, so a topology or accepted
  root change invalidates stale proposal context.
- Kept ULID generation as an engine-owned operation through `GenerateKey`;
  maintained fixtures import that runtime capability explicitly while the
  standalone schema leaf remains free of randomness and clocks.

### ⚠️ Breaking

- Removed the separate `icydb-primitives` package without an alias or
  compatibility wrapper.
- Reclaimed the `icydb-schema` package name for the new proposal/scalar
  contract. The old authoring API is not preserved under that public path.
- Removed permissive and alias `Date` constructors. Strict construction uses
  `Date::try_new`, while textual input uses `Date::parse`.
- The proposal codec accepts only its current version-1 form. No compatibility
  decoder or translation path exists for superseded development shapes.

### 📊 Audit

- The 0.213 opening `sql_perf` subject was 5,157,482 raw non-gzipped Wasm
  bytes. The closing 0.213.0 subject is 5,160,848 bytes, an increase of 3,366
  bytes (+0.0653%) and well below the frozen +2.00% review budget.
- The canonical shrunk artifact is 4,806,575 bytes.
- The affected source seam grew from 176 Rust files and 33,212 lines to
  approximately 194 files and 37,000 lines. Most growth is the bounded public
  contract, exact composition validation, tests, and the temporary package
  bridge. It remains below the frozen +15% source review tripwire.
- The target-authority slice adds no retained bytes to the measured
  `sql_perf` Wasm subject compared with the preceding exact-composition slice.
- No accepted-schema, row, index, relation, cursor, journal, commit-marker, or
  recovery persisted format changes in 0.213.0.

### 🧪 Testing

- Focused proposal tests cover malformed versions and capabilities, collection
  and byte bounds, source-key identity, canonical ordering and digest vectors,
  exact scalar/default compatibility, enum literals, named-type cycles and
  depth, explicit removals, and cross-fragment relation closure.
- Focused runtime tests prove stable target issuance, canonical multi-store
  ordering, distinct store identities, topology-bound database identity, and
  exact accepted-root coverage.
- The schema leaf, core, facade, moved legacy package, build package, and
  derive seams pass their focused compilation and warnings-denied checks.
- No-default-feature checks, documentation tests, formatting, diff hygiene,
  package-boundary guards, memory-allocation guards, and publication guards
  pass.
- Full repository validation remains part of the human-owned push workflow.

### 🧭 Follow-up

- Catalog-native source-to-accepted bindings, proposal lowering, durable
  idempotent receipts, atomic multi-store publication, and reuse of the 0.211
  activation machinery remain subsequent 0.213 work.
- The temporary unpublished model package deliberately blocks workspace
  publication until the final model-package and consumer cutover removes that
  bridge.
