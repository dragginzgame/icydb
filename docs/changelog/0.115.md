# 0.115 Changelog

## 0.115.0

- `0.115.0` starts the next planner-expression ownership cleanup by moving shared boolean normalization and searched-`CASE` canonicalization out of the predicate bridge and onto the planner expression owner in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs). The production semantic entrypoints for:
  - `normalize_bool_expr(...)`
  - `canonicalize_scalar_where_bool_expr(...)`
  - `canonicalize_grouped_having_bool_expr(...)`
  - `is_normalized_bool_expr(...)`
  now live under `query::plan::expr`, while [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs) is reduced back to the predicate bridge/compiler side.

- That first `0.115.0` move matters because planner-owned boolean meaning was still being routed through the predicate subsystem even when the caller was not doing predicate compilation. After this cut, non-predicate consumers now import planner-owned expression semantics directly instead of using the predicate facade. The production call sites moved in:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/normalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/normalize.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/mutation.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/mutation.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/model.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/shared/predicate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/shared/predicate.rs)

- `0.115.0` then moves two more boolean meaning ladders onto that same planner expression owner. Affine numeric compare rewriting and boolean-constant simplification no longer live in the SQL lowering adapter; they now also live in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs). The `WHERE` lowering adapter in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/normalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/normalize.rs) is correspondingly narrower again:
  - it still owns literal-only folding inside scalar `WHERE`
  - it now just calls the planner-owned affine rewrite, boolean simplifier, normalization, and canonicalization steps
  - it no longer owns separate local copies of those rewrite/simplify rules

- `0.115.0` also moves the shared TRUE-only boolean admission rule onto the planner expression owner. [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs) now owns `collapse_true_only_boolean_admission(...)`, and executor/session consumers now import that rule from planner `expr` instead of via an executor projection facade. This keeps the boundary cleaner because grouped `HAVING`, aggregate `FILTER`, scalar residual filtering, and other boolean-result consumers are all collapsing through one planner-owned admission rule rather than an executor-defined helper.

- The last `0.115.0` seam closes the remaining literal-only `WHERE` fold leak into executor preview evaluation. Scalar `WHERE` constant folding in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/normalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/normalize.rs) no longer calls `eval_builder_expr_for_value_preview(...)` from the executor projection layer. Instead, planner `expr` now owns a bounded literal-only evaluator in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/preview.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/preview.rs) through `eval_literal_only_expr_value(...)`.

- That literal-only evaluator is intentionally narrow and fail-closed:
  - it only evaluates literal-owned planner `Expr` trees
  - it rejects field and aggregate leaves
  - it supports the bounded boolean/arithmetic/text-helper surface already exercised by the scalar `WHERE` fold tests
  - it keeps the shared TRUE-only `CASE` condition rule
  - it returns `None` for unsupported shapes instead of widening planner execution or importing the executor runtime stack

- So after `0.115.0`, the authority line is cleaner:
  - planner `expr` owns boolean normalization, canonicalization, affine compare rewriting, boolean simplification, TRUE-only boolean admission, and literal-only expression preview
  - predicate owns predicate compilation/bridge logic
  - SQL lowering owns SQL-specific adaptation, not the underlying boolean meaning ladders
  - executor projection evaluation still owns runtime field-backed expression execution, but lowering no longer reaches into it for constant-only `WHERE` folds

- Structural follow-through for `0.115.0` also includes:
