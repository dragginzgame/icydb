# 0.78 — Simple Scalar Projection Expressions

## 0.78.1 - 2026-04-13 - Widen Bounded Scalar Arithmetic And Add ROUND

This patch keeps the `0.78` slice on the same bounded scalar projection seam,
but finishes the first practical arithmetic family instead of stopping at `+`.

The shipped surface is now:

* SQL:

  ```sql
  SELECT age + 1 FROM Customer;
  SELECT age - 1 FROM Customer;
  SELECT age * 2 FROM Customer;
  SELECT age / 3 FROM Customer;
  SELECT ROUND(age, 2) FROM Customer;
  SELECT ROUND(age / 3, 2) FROM Customer;
  ```

* fluent:

  ```rust
  db.load::<Customer>().project_values(&add("age", 1_u64));
  db.load::<Customer>().project_values(&sub("age", 1_u64));
  db.load::<Customer>().project_values(&mul("age", 2_u64));
  db.load::<Customer>().project_values(&div("age", 3_u64));
  db.load::<Customer>().project_values(&round("age", 2));
  db.load::<Customer>().project_values(&round_expr(&div("age", 3_u64), 2));
  ```

The important boundary did not change:
this is still one bounded scalar projection feature, not a general SQL
expression or function system.

What changed:

* widened the bounded arithmetic operator family from `Add` only to:
  * `Add`
  * `Sub`
  * `Mul`
  * `Div`
* kept those operators on the same canonical `Expr::Binary` path already used
  by the initial `0.78.0` slice instead of introducing a second arithmetic
  projection contract
* added one bounded numeric function, `ROUND(expr, scale)`, as a canonical
  `Expr::FunctionCall { function: Round, ... }`
* kept `ROUND` evaluation entirely in the scalar projection evaluator using the
  existing fixed-point decimal runtime (`Decimal::round_dp(...)`)
* added matching fluent helpers on the same one-field projection contract:
  * `round(field, scale)`
  * `round_expr(&numeric_projection, scale)`
* kept outward SQL projection labels stable and SQL-shaped, so projected column
  names render as:
  * `age - 1`
  * `age * 2`
  * `age / 3`
  * `ROUND(age / 3, 2)`
* updated parser/lowering normalization so `ROUND(...)` stays canonical through
  parse, alias normalization, and lowering without creating a parallel SQL-only
  lane

Still intentionally excluded:

* field-to-field arithmetic
* chained arithmetic
* grouped arithmetic projection
* grouped `ROUND(...)`
* predicate arithmetic
* `ORDER BY` arithmetic
* aggregate arithmetic
* generic nested SQL expression parsing
* broader numeric formatting or casting beyond `ROUND(..., n)`

Result:

* SQL and fluent now share one bounded scalar arithmetic family instead of
  drifting into a SQL-only computed projection feature
* explicit presentation rounding is available without widening planner, route,
  or grouped-execution policy
* the expression/runtime seam stays single-owner:
  parser -> lowering -> canonical `Expr` -> scalar projection evaluator

## 0.78.0 - 2026-04-13 - Ship Bounded Scalar Add Projection

This patch opens the `0.78` line with one deliberately narrow feature slice:
bounded scalar arithmetic projection for the `SELECT` list only.

The shipped surface is:

* SQL:

  ```sql
  SELECT age + 1 FROM Customer;
  SELECT name, age + 1 FROM Customer;
  ```

* fluent:

  ```rust
  db.load::<Customer>().project_values(&add("age", 1_u64))
  ```

The important design point is that SQL and fluent now use the same canonical
expression spine for this shape instead of introducing a new SQL-only computed
projection lane.

What changed:

* added reduced SQL `+` token support and bounded parser support for
  `field + numeric_literal` projection items
* lowered that shape to canonical `Expr::Binary { op: Add, ... }` instead of
  inventing a second projection-specific AST
* promoted the bounded binary scalar-expression path out of `#[cfg(test)]` for
  the shipped `Add` case while keeping broader binary/expression forms
  fail-closed
* added shared fluent numeric projection support through `add(field, literal)`
  and widened the projection terminal helpers to one shared
  `ValueProjectionExpr` contract
* kept outward SQL label rendering stable for the new projection columns, so a
  query such as `SELECT age + 1 ...` reports the column label as `age + 1`
* kept the slice intentionally bounded:
  * no field-to-field arithmetic
  * no chained arithmetic
  * no `-` or `*` on the shipped surface
  * no grouped arithmetic
  * no `WHERE` arithmetic
  * no `ORDER BY` arithmetic
  * no aggregate arithmetic
  * no generic expression grammar

Result:

* SQL and fluent now have one shared scalar arithmetic projection shape
* the feature reuses the canonical scalar projection evaluator instead of
  adding a second execution path
* planning, routing, grouping, and mutation semantics stay unchanged
