# 0.88 Changelog

## 0.88.1 - 2026-04-17 - Bounded Grouped Aggregate ORDER BY

- Widened grouped SQL parsing and lowering so aggregate-driven grouped `ORDER BY` now lowers through the new Top-K lane instead of failing earlier as an unknown field or generic grouped-order mismatch.
- Wired bounded heap-backed grouped finalize for the new `top_k_group` strategy: grouped aggregate order terms are compiled once per execution stage onto the shared grouped post-aggregate evaluator, finalized grouped rows retain only the bounded Top-K candidate heap, and result ties fall back deterministically to canonical grouped-key order.
- Kept the continuation contract fail-closed for this first cut: grouped aggregate Top-K pages do not expose grouped continuation cursors, and grouped aggregate `ORDER BY ... OFFSET ...` is rejected explicitly at planner policy instead of widening rank-window semantics early.
- Updated grouped strategy projection, explain coverage, SQL lowering tests, and session execution coverage so the new `top_k_group` path is visible end to end instead of remaining a planner-only seam.

## 0.88.0 - 2026-04-17 - Grouped Aggregate ORDER BY Groundwork

- Added a planner-owned grouped Top-K strategy lane for grouped `ORDER BY` terms that stay within the grouped post-aggregate expression model but cannot reuse canonical grouped-key ordering.
- Introduced a shared parser/classifier for grouped aggregate and post-aggregate `ORDER BY` terms over grouped keys, aggregate leaves, bounded arithmetic, and `ROUND(...)`, so `0.88` strategy selection no longer has to infer aggregate-order intent from the narrower `0.87` grouped-key proof helper.
- Updated grouped strategy projection and explain coverage so manual grouped plans with aggregate order terms now surface `top_k_group` explicitly instead of collapsing into the narrower `0.87` grouped-key order fallback reasons.
- Followed with a complexity-accretion cleanup pass over the hot reduced-SQL and predicate runtime seams: `db::session::sql::projection::runtime` moved from one mixed root into `runtime/{covering,materialize,render}`, `covering` split again into `pure`, `hybrid`, and `shared`, and the duplicated post-decode assembly path in `pure` was collapsed behind local helpers.
- Split predicate execution and parsing ownership the same way: `db::predicate::runtime` now isolates compare semantics under `runtime/compare/{scalar,value}`, while `db::predicate::parser` now separates boolean-chain expression parsing, operand/special-form parsing, and predicate lowering into `parser/{expression,operand,lowering}`. The parser hotspot moved from one `53`-branch owner to `parser = 3`, `expression = 26`, `operand = 15`, and `lowering = 9`, which makes the remaining pressure explicit instead of hiding it in one file.
- Continued the cleanup pass through shared hashing and SQL tokenization: `db::sql_shared::lexer` is now split into `lexer/{keywords,scan,token_body}`, isolating keyword classification from scanner control flow, and `db::query::fingerprint::hash_parts` now separates access hashing from grouped-shape hashing under `hash_parts/{access,grouping}` instead of keeping those visitor/projection seams in one root module.
- Split structural-field storage-key handling into `storage_key/{decode,encode,scalar}` so the root keeps only the storage-key lane boundary and kind routing, while relation list handling, runtime-value bridging, and fixed-shape scalar codecs live under narrower owners. The old `storage_key = 41` hotspot now measures as `storage_key = 1`, `decode = 12`, `encode = 8`, and `scalar = 21`.
- Split predicate scalar fast-path comparison into `compare/scalar/{direct,text}` so the scalar root keeps only slot-type dispatch while typed/list membership logic and text-mode comparison semantics live under separate owners. The old `compare::scalar = 39` hotspot now measures as `scalar = 1`, `direct = 27`, and `text = 11`.
- Split predicate expression parsing into `expression/{chain,atom}` so boolean-precedence chaining no longer shares a file with field/literal predicate atom handling. The old `parser::expression = 26` hotspot now measures as `expression = 0`, `chain = 1`, and `atom = 25`.
- Added focused coverage around:
  - grouped Top-K order-term parsing/classification
  - explain projection for aggregate grouped order
  - route-mode projection from the new planner strategy
