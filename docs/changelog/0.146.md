# 0.146.x

## 0.146.10

`0.146.10` tightens startup schema reconciliation so the schema store cannot
silently ignore a newer persisted schema snapshot for an entity. Reconciliation
now asks `SchemaStore` for the latest stored version for the entity and
compares that accepted candidate against the generated proposal. The schema
metadata boundary also rejects internally inconsistent typed writes and raw
decoded payloads before they can become accepted metadata. The policy is still
fail-closed and exact-match-only; this change only closes the version-selection
and metadata-integrity boundaries before real schema transitions exist.

### Schema reconciliation

- Added `SchemaStore::latest_persisted_snapshot(...)` to scan schema metadata
  entries for one entity and decode the highest stored version.
- Routed startup reconciliation through that latest-version accessor instead
  of directly loading only `SchemaVersion::initial()`.
- Added schema-store coverage proving latest-version lookup ignores other
  entity tags and selects the highest matching version.
- Added schema-store coverage proving latest-version lookup fails closed on a
  corrupt newest payload instead of falling back to an older decodable
  snapshot.
- Added schema-store coverage proving typed snapshots cannot be persisted when
  their snapshot version and row-layout version diverge.
- Added schema-codec coverage proving raw decoded snapshots fail closed when
  the snapshot version and row-layout version diverge.
- Added typed-write and raw-decode coverage proving persisted field slot
  metadata cannot diverge from the row-layout slot mapping.
- Added typed-write and raw-decode coverage proving schema snapshots cannot
  reference a primary-key field that is absent from the row-layout or field
  metadata.
- Added coverage for ambiguous schema metadata, including duplicate
  row-layout slots and duplicate persisted field names.
- Added coverage for invalid nested leaf metadata, including empty nested
  paths and duplicate nested paths under the same top-level field.
- Added reconciliation coverage proving a newer persisted schema version is
  rejected instead of being ignored or overwritten by a recreated initial
  snapshot.

## 0.146.8

`0.146.8` keeps the next schema-default boundary explicit while leaving the
only current database default as `None`. Derive-generated field metadata now
passes a `FieldDatabaseDefault` contract into `FieldModel`, so Rust construction
defaults cannot accidentally become persistence defaults when database-level
defaults are added later.

Startup schema reconciliation also now routes through an explicit
transition-policy decision. The accepted policy is still exact initial-schema
equality only: first contact writes the generated snapshot, exact matches are
accepted, and every mismatch remains fail-closed with the existing first
difference diagnostic. The change gives later migration work a single
schema-owned decision point to extend instead of letting transition policy hide
inside equality checks or diagnostic formatting. It also tightens the accepted
schema snapshot boundary so metadata, planning, and SQL write paths ask the
accepted wrapper for field facts instead of reaching through to raw field
snapshots.

Runtime metrics now count rejected SQL writes by statement kind and error
class, expose derived Rust helper methods for common read/write efficiency
ratios without changing the Candid report shape, and guard the metrics sink
event/taxonomy ordering with `remain` so future counter families stay aligned
across global and per-entity report updates.

### Schema reconciliation

- Added a dedicated `db::schema::transition` owner for transition decisions,
  with `SchemaTransitionDecision` as the policy result between a persisted
  accepted snapshot and the generated schema proposal.
- Added `SchemaTransitionRejection` so transition diagnostics stay separate
  from the final `InternalError` formatting boundary.
- Narrowed the raw accepted-snapshot accessor to the schema module so
  non-schema callers keep using accepted-schema helper methods instead of
  reopening the persisted payload directly.
- Added accepted-schema field fact accessors for primary-key kind, top-level
  field kind, top-level slot, and nested leaf metadata.
- Routed SQL write literal normalization, accepted-schema `SchemaInfo`
  construction, and accepted-schema `DESCRIBE` projection through those
  accepted-wrapper accessors.
- Renamed the session accepted-schema helper to make its first-contact
  reconciliation side effect explicit: it ensures the initial snapshot before
  returning it, rather than acting as a pure load.
- Kept the only accepted decision as `ExactMatch`; all mismatches still return
  the existing unsupported schema-evolution error with the first rejected
  schema difference.
- Added regression coverage proving the transition policy accepts exact
  snapshots, rejects changed snapshots, and reports row-layout drift before
  runtime row-layout authority work begins.

### Schema authority boundary

- Routed derive-generated field metadata through an explicit
  `FieldDatabaseDefault` contract while keeping the only current database
  default as `None`.
- Exposed `FieldDatabaseDefault` through the public `icydb::model::field`
  generated-code boundary so derive output compiles through the facade crate.
- Added macro-test coverage proving fields with Rust construction defaults still
  emit `FieldDatabaseDefault::None` for the database schema contract.

### Metrics report shape

- Added global and per-entity SQL write error counters by command kind and
  `ErrorClass`, covering unsupported statements, conflicts, not-found errors,
  internal errors, invariant violations, corruption, and incompatible persisted
  formats.
- Added non-persisted helper methods for load selectivity, load filter ratio,
  SQL write mutation ratio, SQL `RETURNING` ratio, index entries per write row,
  reverse-index entries per write row, and relation checks per write row.
- Kept derived ratios out of the Candid report shape so `icydb_metrics()`
  remains a raw-total report and consumers can distinguish missing denominators
  from real zero-valued work.
- Added `remain` ordering guards to the metrics sink event and taxonomy enums,
  plus the single-kind mirror helpers, so future sink counter families stay
  mechanically aligned across global and per-entity report updates.

## 0.146.1

`0.146.1` turns the reserved schema memory into a concrete raw schema metadata
map and starts persisting generated initial schema snapshots at the recovery
boundary. `SchemaStore` now owns a stable BTreeMap keyed by entity tag and
schema version, the raw snapshot payload has a typed persisted-schema codec,
and startup reconciliation writes or verifies each registered entity's initial
snapshot before row/index operations proceed. The current `DESCRIBE` /
`SHOW COLUMNS` surfaces also expose generated physical row slots while live
row-layout authority is still pending.

### Schema store map boundary

- Added `RawSchemaKey` as the durable key for one entity schema version.
- Added `RawSchemaSnapshot` as the raw encoded value boundary for future
  persisted schema snapshots.
- Updated `SchemaStore` from a bare stable-memory holder into a thin
  BTreeMap-backed persistence wrapper with insert, lookup, presence, length,
  and test-clear operations.
- Kept raw schema keys and encoded payloads inside the schema store boundary;
  reconciliation now uses typed snapshot insert/load entrypoints.
- Added schema-store tests for key encoding, raw payload round-tripping, and
  raw snapshot persistence.

### Startup reconciliation

- Added a schema-owned reconciliation pass over registered runtime hooks.
- First contact writes the generated initial persisted schema snapshot for each
  entity into its store's `SchemaStore`.
- Later contacts decode the stored snapshot and require an exact match with the
  compiled proposal until explicit schema-evolution rules exist.
- Rejected snapshots now report the first detected schema difference, such as
  entity identity, primary-key field ID, row layout, field count, field kind,
  nullability, default, storage decode, or leaf codec drift.
- Wired reconciliation through the commit recovery gate, guarded once per store
  registry so normal read/write entrypoints do not repeatedly scan schema
  metadata.
- Added focused tests for first-write, idempotent reread, and changed-snapshot
  rejection.

### Persisted snapshot codec

- Added a schema-owned Candid wire codec for `PersistedSchemaSnapshot` without
  deriving wire traits on the internal schema DTOs.
- Encoded the codec version, schema version, entity identity, primary-key field
  ID, row layout, field metadata, database defaults, storage decode policy,
  leaf codec, and recursive persisted field kind graph.
- Added typed round-trip coverage through `RawSchemaSnapshot` for nested
  `Map<Text, List<Uint>>` metadata.

### Describe slot visibility

- Added slot visibility to `EntityFieldDescription`.
- Top-level generated fields now report their current physical row slot.

## 0.146.0

`0.146.0` starts the schema identity and layout authority work by introducing
schema-owned identity artifacts before adding any durable `__icydb_schema`
payload. Generated `EntityModel` metadata now projects into a compiled schema
proposal, that proposal can build the first persisted-schema snapshot shape for
a new database, and generated stores now reserve a third stable memory for
future schema metadata alongside row and index storage.

### Schema identity foundation

- Added `FieldId` as the durable logical field identity, separate from generated
  field order and executor slot indexes.
- Added schema-owned `SchemaVersion`, `SchemaFieldSlot`, and `SchemaRowLayout`
  types so future reconciliation can map durable field IDs to physical row
  slots without using loose integer pairs.
- Added `CompiledSchemaProposal` and `CompiledFieldProposal` as the bridge from
  generated `EntityModel` / `FieldModel` metadata into schema-identity terms.
- Routed the existing commit-schema fingerprint through the compiled proposal
  without changing the current fingerprint contract.

### Persisted snapshot shape

- Added `PersistedSchemaSnapshot` and `PersistedFieldSnapshot` as the owned
  shape intended for the future `__icydb_schema` payload.
- Added owned `PersistedFieldKind` conversion from generated `FieldKind` so
  live schema snapshots do not depend on generated static metadata for nested,
  relation, enum, collection, or structured field shapes.
- Added `SchemaFieldDefault` with only an explicit `None` state for now, keeping
  database defaults separate from Rust struct `Default` behavior.
- Added schema proposal tests covering initial field IDs, row layout,
  persisted snapshot construction, and storage contracts.

### Store memory boundary

- Added `SchemaStore` as the runtime owner for the per-store schema stable
  memory. It is intentionally only a memory boundary in this slice; startup
  reconciliation and the concrete `__icydb_schema` map remain separate work.
- Extended `StoreHandle` and `StoreRegistry` so generated stores bind row,
  index, and schema stores as one triplet.
- Extended the derive/build path so every `#[store(...)]` declaration now
  supplies `schema_memory_id`, and generated canister wiring emits the matching
  `SchemaStore` static.
- Updated schema validation so `schema_memory_id` is range-checked,
  reserved-ID checked, unique within its store, and unique across all memories
  registered to the same canister.
