# 0.68 Changelog

## 0.68.5 - 2026-04-05 - NumericValue Hard Cut

### Summary

`0.68.5` removes the repo's direct `num-traits` dependency and hard-cuts the
old generic numeric-conversion surface. The public `NumCast`,
`NumFromPrimitive`, and `NumToPrimitive` traits are gone. In their place, the
runtime now uses one narrower local trait, `NumericValue`, only for the
numeric-to-`Decimal` round-trips that validators, sanitizers, and generated
numeric newtypes actually need.

This patch intentionally does not preserve compatibility wrappers. Wrapper
types now expose explicit inherent constructors where the old trait-provided
static methods used to sit, and macro-generated numeric newtypes now implement
`NumericValue` instead of a larger generic cast family.

The practical result is a smaller, more direct numeric model:

- one local contract for numeric validation and sanitization
- explicit conversion entrypoints on wrapper types
- no public `num-traits` surface or hidden macro re-export

### Breaking

- removed the public numeric conversion traits re-exported from:
  - `crates/icydb-core/src/traits/mod.rs`
  - `crates/icydb/src/traits/mod.rs`
  - deleted:
    - `NumCast`
    - `NumFromPrimitive`
    - `NumToPrimitive`
- removed the hidden `num_traits` macro re-export from:
  - `crates/icydb/src/lib.rs`
- removed direct `num-traits` manifest dependencies from:
  - `Cargo.toml`
  - `crates/icydb/Cargo.toml`
  - `crates/icydb-core/Cargo.toml`
- downstream code that previously relied on:
  - `icydb::traits::NumCast`
  - `icydb::traits::NumFromPrimitive`
  - `icydb::traits::NumToPrimitive`
  - generated `Num*` impls on numeric newtypes
  must now move to:
  - `icydb::traits::NumericValue` where generic decimal round-tripping is the
    actual need
  - explicit constructors such as `Float32::try_from_f64(...)`,
    `Timestamp::try_from_i64(...)`, `Date::try_from_u64(...)`, and similar
    inherent methods on wrapper types

### Changed

- added one repo-local numeric round-trip trait in:
  - `crates/icydb-core/src/traits/numeric_value.rs`
  - `NumericValue` is the only remaining shared numeric abstraction and is

## 0.68.4 - 2026-04-05 - Bounded Text Reads and Filtered Ordered Routes

### Summary

`0.68.4` extends the ordered covering-read line from `0.68.3` in two
directions.

First, equivalent bounded text forms now converge onto the same shared ordered
route instead of splitting between bounded and fail-closed planner outcomes.
For the supported strict text family, queries such as:

- `name LIKE 'A%'`
- `STARTS_WITH(name, 'A')`
- `name >= 'A' AND name < 'B'`

now keep the same bounded `IndexRange` + covering-read contract when the query
also carries the canonical `ORDER BY name, id LIMIT n` shape.

Second, guarded filtered indexes now participate in the same ordered fallback
story. When a query predicate fully implies the filtered index guard, the
planner no longer carries that guard forward as a fake residual predicate, so
ordered reads such as `WHERE active = true ORDER BY name, id LIMIT n` can stay
on the explicit covering-read route. The unguarded form still fails closed to a
materialized full scan.

This patch also broadens the outer SQL harnesses so those contracts are no
longer locked only in `icydb-core`. Quickstart and PocketIC now include a
dedicated filtered-index fixture entity, matching query/explain parity tests,
and checked-in perf baselines for the new guarded ordered route.

## 0.68.3 - 2026-04-05 - Expression-Order Reads and Semantic Index Key Order

### Summary

`0.68.3` builds on the ordered covering routes from `0.68.2` by fixing the
remaining expression-order gap. Parser, lowering, executor ordering, and cursor
boundaries now preserve canonical `LOWER(field)` / `UPPER(field)` order terms
end to end, and raw secondary-index traversal now sorts variable-width text
keys by the decoded ordered component bytes instead of by tuple-frame length
prefixes.

The main user-visible result is that expression-backed reads such as
`ORDER BY LOWER(name), id LIMIT n` now stay on the intended ordered route and
return the correct lexical row order. This patch also fixes a lower-level
ordering bug in raw index traversal. The
stable key codec was framing tuple segments with length prefixes, but raw key
ordering was still following that tuple framing instead of the decoded ordered
component bytes. For variable-width text keys, that could let shorter strings
sort ahead of lexicographically earlier longer strings during index-range
scans. `0.68.3` moves `IndexKey` and `RawIndexKey` ordering back onto decoded
component semantics, which closes that gap across planner scans, session SQL,
generated canister SQL, and the PocketIC harness.

### Changed

- extended order-term plumbing for expression-backed order keys in:
  - `crates/icydb-core/src/db/sql/parser/{model.rs,clauses.rs,tests.rs}`
  - `crates/icydb-core/src/db/sql/lowering/{mod.rs,tests.rs}`
  - `crates/icydb-core/src/db/scalar_expr.rs`
  - `crates/icydb-core/src/db/executor/order.rs`
  - `crates/icydb-core/src/db/cursor/boundary.rs`
  - parser, lowering, executor ordering, and cursor boundaries now preserve
    canonical `LOWER(field)` / `UPPER(field)` order terms instead of collapsing
    them back to missing slots
- fixed raw key ordering semantics in:
  - `crates/icydb-core/src/db/index/key/codec/mod.rs`
  - `crates/icydb-core/src/db/index/key/codec/tuple.rs`
  - `IndexKey` and `RawIndexKey` ordering now compare decoded ordered

  - malformed raw keys still fall back to raw byte comparison so diagnostics

shared index-backed route instead of falling back to wider materialized work,

  primary-key or bound-constant output

## 0.68.1 - 2026-04-04 - Cursorless SQL Projection Lane Cleanup

### Summary

`0.68.1` keeps the shipped SQL surface unchanged and does not yet land the
first-class covering-read runtime from the `0.68` design. Instead, it cleans
up the existing cursorless structural SQL projection lane so that it stops
paying more of the generic paged-load machinery than that surface actually
needs. The goals in this patch are:

- make the cursorless retained-slot SQL projection lane explicit instead of
  rediscovering that shape late inside the shared scalar page materializer
- stop carrying full row payloads and outward cursor assembly through the
  immediate SQL projection path when that lane never exposes them
- reduce widened select attribution on the kept projection-heavy shapes without
  reopening the reverted late-bound covering-read runtime experiments

### Changed

- added an explicit cursor-emission contract to the scalar execution inputs in:
  - `crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs`
  - `crates/icydb-core/src/db/executor/pipeline/contracts/execution/mod.rs`
  - `crates/icydb-core/src/db/executor/pipeline/contracts/mod.rs`
  - `CursorEmissionMode` now makes the no-cursor SQL projection lane an owned
    contract instead of an implicit property rediscovered during final page
    assembly
- added an explicit row-payload contract for the same scalar lane in:
  - `crates/icydb-core/src/db/executor/terminal/page.rs`
  - `crates/icydb-core/src/db/executor/pipeline/operators/post_access/contracts/row.rs`
  - `crates/icydb-core/src/db/executor/order.rs`
  - `KernelRowPayloadMode` now distinguishes:
    - full-row structural payloads for cursor/entity lanes
    - slot-only structural payloads for cursorless retained-slot SQL
      projection lanes
  - `KernelRow` now carries `Option<DataRow>` explicitly, so the slot-only lane
    can stay fail-closed instead of relying on accidental full-row assumptions
- reduced cursorless SQL row-collector overhead in:
  - `crates/icydb-core/src/db/executor/pipeline/operators/terminal/runtime.rs`
  - `crates/icydb-core/src/db/executor/pipeline/entrypoints/scalar/mod.rs`
  - `crates/icydb-core/src/db/executor/kernel/mod.rs`
  - the short path now uses one explicit `RowCollectorStreamRequest` envelope
    instead of widening helper signatures
  - cursorless SQL staging now stops after `offset + limit` when that cap is
    valid for the retained-slot lane

## 0.68.0 - 2026-04-04 - Planner-Owned Covering Read Groundwork

### Summary

`0.68.0` starts the `0.68` covering-read line with one narrow internal slice.
It still does not widen the shipped SQL or fluent query surface, and it does
not keep a new runtime fast path yet. This patch only lands the planner-owned
contract and the shared projection-shape helper that later runtime work will
consume.

The goals in this patch are:

- land one planner-owned scalar covering-read contract before broader executor
  routes start consuming it
- keep phase-1 covering eligibility intentionally narrow:
  - direct projected index components
  - direct projected primary key output
  - direct projected constants satisfied by bound index-prefix equality values
- remove duplicated "direct field under optional alias wrappers" logic from the
  executor now that planner and executor both need the same rule

### Changed

- added the first planner-owned covering-read contract in:
  - `crates/icydb-core/src/db/query/plan/covering/mod.rs`
  - new phase-1 covering-read contract types:
    - `CoveringReadFieldSource`
    - `CoveringReadField`
    - `CoveringReadPlan`
  - new planner helper:
    - `covering_read_plan(...)`
  - covering access metadata is now derived once and shared across:
    - covering projection order derivation
    - constant projection recovery from prefix-bound equality values
    - the new per-field covering-read source derivation
- shared the direct-field projection rule in:
  - `crates/icydb-core/src/db/query/plan/expr/projection.rs`
  - `crates/icydb-core/src/db/executor/projection/materialize/structural.rs`
  - planner and executor now both use the same helper when a projection must
    prove that one output stays on one direct field leaf under optional alias
    wrappers
  - this removes the executor-local duplicate helper and keeps later covering
    route work anchored to one projection-shape authority
- no new runtime covering-read consumer ships in this patch
  - the first runtime experiment was intentionally backed out because it was
    mixed on the widened PocketIC select cohort
  - `IndexComponent { .. }`, `PrimaryKey`, and `Constant(...)` sources are
    planner-owned contract data only in `0.68.0`
  - later `0.68` work can now target the runtime side against one explicit
    planner boundary instead of rediscovering covering eligibility ad hoc
- added planner-local regression coverage in:
  - `crates/icydb-core/src/db/query/plan/covering/tests.rs`
  - accepted phase-1 shapes now include:
    - direct index component projection
    - direct primary-key projection
    - direct prefix-bound constant projection
  - rejected phase-1 shapes now include:
    - computed/non-field expressions
    - non-coverable row-field projections
    - residual-predicate shapes without strict predicate compatibility
