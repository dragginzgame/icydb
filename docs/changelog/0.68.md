# 0.68 Changelog

## 0.68.7 - 2026-04-05 - Casefold Expression Ranges and Testkit Cleanup

### Summary

`0.68.7` extends the ordered text-window line from `0.68.4` and `0.68.6`
onto the narrow casefold expression family, then cleans up the outer PocketIC
SQL harness so it relies on the newer public `canic-testkit` setup paths
instead of re-deriving local startup behavior.

Before this patch, the expression-index family had one visible consistency gap:

- `LOWER(field) LIKE 'br%'`
- `STARTS_WITH(LOWER(field), 'BR')`
- `LOWER(field) >= 'br' AND LOWER(field) < 'bs'`

The first two spellings already converged on the bounded `TextCasefold` route.
The explicit ordered-bound form still stayed fenced off even though the
key-item matcher and canonical expression literal lowering already knew how to
map literals onto the same normalized expression-key values.

`0.68.7` closes that specific gap without broadening expression-range support
generally. The new contract is intentionally narrow:

- only canonical ordered text bounds on wrapped fields are accepted
- they reuse the same `TextCasefold` literal-lowering path already used by
  expression `Eq`, `In`, and prefix matching
- unsupported wrapped range forms still fail closed
- strict ordered compares on expression indexes still reject explicitly as
  unsupported range operators

The same patch also removes the last local PocketIC startup glue from the SQL
integration harness. The suite now leans on `canic-testkit`'s fallible
`try_*` setup APIs for startup classification, temp-root handling, and fresh
fixture install, while keeping IcyDB's own demo_rpg wasm build and empty
init-arg contract local.

## 0.68.6 - 2026-04-05 - Filtered Composite Windows and Route Cleanup

### Summary

`0.68.6` extends the filtered ordered-read line from `0.68.4` onto composite
secondary indexes, then tightens the ordered route contract around that same
family so unsafe descending secondary-order shapes fail closed earlier.

Before this patch, a guarded composite query such as:

- `WHERE active = true AND tier = 'gold' AND handle LIKE 'br%'`
- `ORDER BY handle ASC|DESC, id ASC|DESC LIMIT 2`

could prove the filtered guard and the equality prefix on `tier`, but the
planner still stopped at an `IndexPrefix` on the equality portion. The bounded
text window on `handle` was then enforced after access instead of being folded
into the composite scan envelope.

`0.68.6` fixes that gap for the strict raw-field text family. When one query:

- proves the filtered index guard
- pins one leading index slot with equality
- then applies `LIKE 'prefix%'`, `STARTS_WITH(field, 'prefix')`, or the
  equivalent explicit `>= lower AND < upper` bounds on the next indexed field

the planner now lowers that shape to one composite `IndexRange` with:

- one equality-prefix value vector
- one bounded text suffix interval
- covering-read routing when the projection is satisfied by index fields plus
  primary key
- runtime `EXPLAIN EXECUTION` prefix metadata that now includes both
  `prefix_len` and the concrete `prefix_values`

That keeps the filtered composite route in the same bounded access family as
the earlier single-field text windows instead of degrading to an equality-only
prefix scan.

The later follow-up in this same patch restores one cleaner ordered-read
ownership rule:

- planner and route capability decide whether one ordered read is
  direct-streaming, materialized behind one shared boundary, or a full
  materialized fallback
- the shared materialized boundary no longer widens hints locally for ranking
  or pagination
- risky descending secondary-order shapes such as non-unique, residual,
  `DISTINCT`, and offset-sensitive windows no longer rediscover
  `SecondaryOrderTopNSeek` later in terminal-local code

## 0.68.5 - 2026-04-05 - NumericValue Hard Cut

### Summary

`0.68.5` removes the repo's direct `num-traits` dependency and hard-cuts the
old generic numeric-conversion surface. The public `NumCast`,
`NumFromPrimitive`, and `NumToPrimitive` traits are gone. In their place, the
runtime now uses one narrower local trait, `NumericValue`, only for the
numeric-to-`Decimal` round-trips that validators, sanitizers, and generated
numeric newtypes actually need.

This patch intentionally does not preserve compatibility wrappers. Wrapper
types now expose explicit inherent constructors where the old trait-provided
static methods used to sit, and macro-generated numeric newtypes now implement
`NumericValue` instead of a larger generic cast family.

The practical result is a smaller, more direct numeric model:

- one local contract for numeric validation and sanitization
- explicit conversion entrypoints on wrapper types
- no public `num-traits` surface or hidden macro re-export

## 0.68.4 - 2026-04-05 - Bounded Text Reads and Filtered Ordered Routes

### Summary

`0.68.4` extends the ordered covering-read line from `0.68.3` in two
directions.

First, equivalent bounded text forms now converge onto the same shared ordered
route instead of splitting between bounded and fail-closed planner outcomes.
For the supported strict text family, queries such as:

- `name LIKE 'A%'`
- `STARTS_WITH(name, 'A')`
- `name >= 'A' AND name < 'B'`

now keep the same bounded `IndexRange` + covering-read contract when the query
also carries the canonical `ORDER BY name, id LIMIT n` shape.

Second, guarded filtered indexes now participate in the same ordered fallback
story. When a query predicate fully implies the filtered index guard, the
planner no longer carries that guard forward as a fake residual predicate, so
ordered reads such as `WHERE active = true ORDER BY name, id LIMIT n` can stay
on the explicit covering-read route. The unguarded form still fails closed to a
materialized full scan.

That same filtered-index rule now extends through bounded text reads too. When
the query both proves the filtered guard and adds one exact ordered text window
such as `name LIKE 'br%'`, `STARTS_WITH(name, 'br')`, or
`name >= 'br' AND name < 'bs'`, executor preparation now strips only the
already-satisfied filtered guard before compiling index predicate capability.
That keeps the filtered bounded route on the same `covering_read` +
strict-prefilter classification as the unfiltered strict text family instead of
falling back to a fake materialized residual path.

This patch also broadens the outer SQL harnesses so those contracts are no
longer locked only in `icydb-core`. Demo RPG and PocketIC now include a
dedicated filtered-index fixture entity, matching query/explain parity tests,
and checked-in perf baselines for the new guarded ordered route.

### Changed

- extended strict ordered text-range planning and coercion in:
  - `crates/icydb-core/src/db/predicate/parser/{mod.rs,tests.rs}`
  - `crates/icydb-core/src/db/sql/parser/tests.rs`
  - `crates/icydb-core/src/db/query/plan/planner/{compare.rs,range.rs,order_select.rs}`
  - ordered text comparisons on textual literals now stay `Strict` instead of
    widening to numeric coercion
  - bounded strict text routes now cover:
    - `LIKE 'prefix%'`
    - `STARTS_WITH(field, 'prefix')`
    - equivalent explicit ordered bounds such as `>= lower AND < upper`
- widened parity coverage for those equivalent bounded forms in:
  - `crates/icydb-core/src/db/query/intent/tests.rs`
  - `crates/icydb-core/src/db/session/tests/mod.rs`
  - `canisters/demo_rpg/src/lib.rs`
  - `testing/pocket-ic/tests/sql_canister.rs`
  - coverage now locks:
    - row parity across `LIKE`, `STARTS_WITH`, and explicit text-range forms
    - ascending and descending ordered windows on those same bounded routes

## 0.68.3 - 2026-04-05 - Expression-Order Reads and Semantic Index Key Order

### Summary

`0.68.3` builds on the ordered covering routes from `0.68.2` by fixing the
remaining expression-order gap. Parser, lowering, executor ordering, and cursor
boundaries now preserve canonical `LOWER(field)` / `UPPER(field)` order terms
end to end, and raw secondary-index traversal now sorts variable-width text
keys by the decoded ordered component bytes instead of by tuple-frame length
prefixes.

The main user-visible result is that expression-backed reads such as
`ORDER BY LOWER(name), id LIMIT n` now stay on the intended ordered route and
return the correct lexical row order. This patch also fixes a lower-level
ordering bug in raw index traversal. The
stable key codec was framing tuple segments with length prefixes, but raw key
ordering was still following that tuple framing instead of the decoded ordered
component bytes. For variable-width text keys, that could let shorter strings
sort ahead of lexicographically earlier longer strings during index-range
scans. `0.68.3` moves `IndexKey` and `RawIndexKey` ordering back onto decoded
component semantics, which closes that gap across planner scans, session SQL,
generated canister SQL, and the PocketIC harness.
