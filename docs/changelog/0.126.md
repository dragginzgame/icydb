# 0.126.x

## 0.126.4

`0.126.4` is the final pre-`0.127` naming cleanup for the value-boundary line. `0.126.3` already deleted the old `FieldValue` trait and split its real jobs across `KeyValueCodec`, `ValueCodec`, and the remaining metadata owner. What was still inconsistent after that cut was the metadata naming itself: the code no longer had a `FieldValue` trait, but the leftover metadata side was still spelled `FieldValueMeta` / `FieldValueKind`, and the derive implementation file was still `imp/field_value.rs`.

This patch makes that leftover naming match the actual post-split ownership model:
- `FieldValueMeta` is now `ValueSurfaceMeta`
- `FieldValueKind` is now `ValueSurfaceKind`
- the derive implementation module moved from `crates/icydb-schema-derive/src/imp/field_value.rs` to [/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs)

The main cleanup landed in:
- [/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs)
  renamed the remaining metadata owner and its schema affordance enum to `ValueSurfaceMeta` / `ValueSurfaceKind`, which makes the metadata lane line up with the already-renamed derive-side `ValueSurfaceTrait`
- [/home/adam/projects/icydb/crates/icydb-core/src/lib.rs](/home/adam/projects/icydb/crates/icydb-core/src/lib.rs)
  [/home/adam/projects/icydb/crates/icydb/src/lib.rs](/home/adam/projects/icydb/crates/icydb/src/lib.rs)
  hidden `__macro` exports now expose only the `ValueSurface*` names, so generated code no longer has to reach through one last stale `FieldValue*` metadata path
- [/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/mod.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/mod.rs)
  [/home/adam/projects/icydb/crates/icydb-schema-derive/src/trait_kind.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/trait_kind.rs)
  [/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs)
  derive wiring now uses `ValueSurfaceMeta` consistently, and the implementation module name finally matches the value-surface terminology the derive layer has been using since `0.126.3`
- [/home/adam/projects/icydb/crates/icydb-core/src/model/field.rs](/home/adam/projects/icydb/crates/icydb-core/src/model/field.rs)
  [/home/adam/projects/icydb/crates/icydb-core/src/db/schema/types.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/schema/types.rs)
  planner/schema field classification now speaks in terms of `ValueSurfaceKind`, which is a better description of what that enum actually means after the `FieldValue` breakup

This does not widen the `0.126` architecture. It is a naming closeout:
- `KeyValueCodec` still owns typed key conversion
- `ValueCodec` still owns typed structured-runtime conversion
- `ValueSurfaceMeta` now clearly owns only value-surface metadata

That makes `0.126.4` the clean endpoint for the value-boundary line before `0.127` starts the next deeper cut around direct structured persisted-field codecs.

This is the meaningful reduction from `.2` to `.3`:
- `.2` removed `FieldValue` from the query-builder input boundary
- `.3` removes it from the key/cursor/access caller surface as well
- `.3` also stops using the `field_value_*` helper cluster as the generic generated-code conversion owner
- `.3` now deletes the dead `FieldValue` composite trait instead of keeping it around as a compatibility alias

What remains after `0.126.3` is much narrower:
- `ValueSurfaceMeta` owns `kind()` and the structured/queryable metadata that generated field types need
- `ValueSurfaceKind` still exists because the metadata split has not happened yet
- runtime storage and execution still use `Value` internally

So `0.126.3` is the point where `FieldValue` stops being a real abstraction at all. What remains is the actual split surface: `KeyValueCodec`, `ValueCodec`, and `ValueSurfaceMeta`.

## 0.126.2

`0.126.2` takes the next clean step after the public value-boundary split: fluent query-builder literals no longer go through `FieldValue` just to get back onto runtime `Value`. That keeps the query-builder input seam aligned with the `InputValue` / `FilterValue` boundary instead of leaving one older runtime-literal path open in the most user-facing builder layer.

The main owner changes landed in:
- [/home/adam/projects/icydb/crates/icydb-core/src/value/input.rs](/home/adam/projects/icydb/crates/icydb-core/src/value/input.rs)
  added direct `InputValue` conversions for the common fluent literal shapes, including string/text literals, bools, integer primitives, core scalar wrappers such as `Ulid`, `Decimal`, `Timestamp`, `Principal`, and typed `Id<E>` values
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/expr.rs)
  `FilterValue` now derives from `InputValue` instead of from `FieldValue`, so fluent filter construction no longer needs the runtime value-conversion trait to reach the filter wire surface
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/builder/field.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/builder/field.rs)
  field filter builders now take `impl Into<FilterValue>` rather than `impl FieldValue`
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/builder/text_projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/builder/text_projection.rs)
  text projection builders now take `impl Into<InputValue>` for literal arguments instead of `impl FieldValue`
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/builder/numeric_projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/builder/numeric_projection.rs)
  numeric projection builders now take `impl Into<InputValue> + NumericValue`, so bounded numeric projection helpers stop depending on `FieldValue` for literal lowering
- [/home/adam/projects/icydb/crates/icydb-core/src/lib.rs](/home/adam/projects/icydb/crates/icydb-core/src/lib.rs)
  added the hidden core-side `__macro` mirror for the value-carrying macro/runtime seam so derive-generated core test harnesses still compile against the same hidden generated-code boundary after the facade-side hard cut

This is a meaningful trim of `FieldValue` even though it does not remove the trait outright:
- query-builder inputs were the most obviously wrong place for `FieldValue`, because they are user-facing literal boundaries rather than runtime value-codec boundaries
- fluent filters now line up with the same input-side boundary story as grouped `HAVING`, patch values, and the narrowed filter wire contract
- bounded scalar projection helpers now also consume the input boundary directly instead of reopening the runtime `Value` path

That leaves the remaining `FieldValue` uses in a much clearer state for the next slice:
- key/cursor/access codecs
- structured persisted-field codecs
- macro-generated structured field conversion

Those are the places where `FieldValue` still has a real runtime role, which makes the next `0.126.3` audit/split materially cleaner than trying to remove it before the builder-input seam was closed.
