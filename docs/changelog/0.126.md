# 0.126.x

## 0.126.1

`0.126.1` follows the initial output-boundary cut in two directions: it introduces a real input-side boundary type for outward literal-taking APIs, and it finishes the grouped public output split so grouped query and SQL payloads stop reusing the runtime grouped row carrier.

The main owner changes landed in:
- [/home/adam/projects/icydb/crates/icydb-core/src/value/input.rs](/home/adam/projects/icydb/crates/icydb-core/src/value/input.rs)
  added `InputValue` and `InputValueEnum` as the public input-side boundary, with total conversion to and from runtime `Value` / `ValueEnum`
- [/home/adam/projects/icydb/crates/icydb-core/src/value/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/value/mod.rs)
  re-exported the new input-side boundary alongside the already-added output-side boundary
- [/home/adam/projects/icydb/crates/icydb-core/src/lib.rs](/home/adam/projects/icydb/crates/icydb-core/src/lib.rs)
  [/home/adam/projects/icydb/crates/icydb/src/lib.rs](/home/adam/projects/icydb/crates/icydb/src/lib.rs)
  public crate surfaces now expose `InputValue` as part of the value-boundary split
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs)
  grouped `HAVING` builder methods on `Query` now take `InputValue` at the outward API seam and lower it explicitly back onto runtime `Value` before intent storage
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/fluent/load/builder.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fluent/load/builder.rs)
  grouped `HAVING` builder methods on the core fluent load surface now use the same input-side boundary instead of taking runtime `Value` directly
- [/home/adam/projects/icydb/crates/icydb/src/db/session/load.rs](/home/adam/projects/icydb/crates/icydb/src/db/session/load.rs)
  facade grouped `HAVING` builder methods now take `InputValue`, and the old extra facade remapping from `Value` to `OutputValue` was removed because core already returns `OutputValue`
- [/home/adam/projects/icydb/crates/icydb/src/db/session/mod.rs](/home/adam/projects/icydb/crates/icydb/src/db/session/mod.rs)
  `UpdatePatch::set_field(...)` now takes `InputValue` instead of runtime `Value`, so the public structural-mutation patch boundary no longer exposes the internal execution value type
- [/home/adam/projects/icydb/crates/icydb-core/src/db/response/grouped.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/response/grouped.rs)
  split grouped rows into a public `GroupedRow` output DTO that carries `OutputValue` plus an internal `RuntimeGroupedRow` carrier that stays on runtime `Value`
- [/home/adam/projects/icydb/crates/icydb-core/src/db/response/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/response/mod.rs)
  outward grouped response surfaces now re-export the output DTO while keeping the runtime grouped carrier internal to `crate::db`
- [/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs)
  grouped query finalization now converts internal grouped cursor rows onto the public grouped output DTO at the session boundary
- [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/mod.rs)
  grouped SQL statement results now convert grouped runtime rows onto the same public grouped output DTO before leaving the session boundary
- [/home/adam/projects/icydb/crates/icydb/src/db/sql/mod.rs](/home/adam/projects/icydb/crates/icydb/src/db/sql/mod.rs)
  facade grouped SQL rendering now consumes grouped `OutputValue` rows directly instead of re-converting runtime grouped values at render time
- [/home/adam/projects/icydb/crates/icydb-core/src/lib.rs](/home/adam/projects/icydb/crates/icydb-core/src/lib.rs)
  [`icydb_core::prelude`] no longer re-exports runtime `Value`, so internal code that still needs it has to import it explicitly instead of pulling it in through the default prelude
- [/home/adam/projects/icydb/crates/icydb/src/lib.rs](/home/adam/projects/icydb/crates/icydb/src/lib.rs)
  both [`icydb::prelude`] and [`icydb::design::prelude`] now export only `InputValue` and `OutputValue` from the value boundary split instead of handing runtime `Value` to downstream users by default

This leaves the `0.126` boundary split in a materially better state:
- output-side SQL and scalar-return projection APIs are already on `OutputValue`
- grouped query and grouped SQL outward rows are now on `OutputValue` too
- input-side grouped `HAVING` and structural patch APIs are now on `InputValue`
- default actor/design preludes now point callers at the boundary types instead of the runtime value type
- runtime `Value` remains the internal execution/storage representation instead of doubling as the public API contract

The main remaining public `Value` seams after `0.126.1` are the broad crate-level `Value` exports and any other outward APIs that still intentionally expose runtime `Value` by name:
- [/home/adam/projects/icydb/crates/icydb-core/src/lib.rs](/home/adam/projects/icydb/crates/icydb-core/src/lib.rs)
- [/home/adam/projects/icydb/crates/icydb/src/lib.rs](/home/adam/projects/icydb/crates/icydb/src/lib.rs)

## 0.126.0

`0.126.0` starts the value-boundary separation by introducing an explicit output-side value contract and moving the first outward result surfaces onto it instead of continuing to expose runtime `Value` directly.

The initial output-boundary move landed in:
- [/home/adam/projects/icydb/crates/icydb-core/src/value/output.rs](/home/adam/projects/icydb/crates/icydb-core/src/value/output.rs)
  added `OutputValue` and `OutputValueEnum`, with total conversion from runtime `Value` / `ValueEnum`
- [/home/adam/projects/icydb/crates/icydb-core/src/value/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/value/mod.rs)
  re-exported the new output-side boundary from the canonical value module
- [/home/adam/projects/icydb/crates/icydb-core/src/db/response/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/response/mod.rs)
  `ProjectedRow<E>` now carries `Vec<OutputValue>` on the public response boundary instead of `Vec<Value>`
- [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs)
  `SqlStatementResult::Projection.rows` now uses `Vec<Vec<OutputValue>>`
- [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/projection/payload.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/projection/payload.rs)
  kept internal SQL projection payloads on runtime `Value`, with conversion happening only at `into_statement_result()`
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/fluent/load/terminals.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fluent/load/terminals.rs)
  public core load terminals that return projected scalar values now return `OutputValue`
- [/home/adam/projects/icydb/crates/icydb/src/db/session/load.rs](/home/adam/projects/icydb/crates/icydb/src/db/session/load.rs)
  facade load terminals that return projected scalar values now also return `OutputValue`
- [/home/adam/projects/icydb/crates/icydb/src/db/sql/mod.rs](/home/adam/projects/icydb/crates/icydb/src/db/sql/mod.rs)
  facade SQL result rendering now consumes `OutputValue` instead of runtime `Value` for projection rows

This first cut was intentionally bounded:
- runtime execution, planning, and storage still use `Value`
- grouped output was not moved yet
- no storage or binary format changes were mixed into the API boundary move

That leaves `0.126.0` as the clean first separation:
- runtime `Value` stays internal
- outward projection/result surfaces start moving to `OutputValue`
- later `0.126` follow-through can then move the remaining input and grouped-output seams separately instead of forcing one oversized patch
