## 0.72.0

`0.72.0` starts the deterministic-planning line by moving single-entity read
route choice onto one planner-owned comparison key and then locking that same
choice story through planner tests, verbose explain, public SQL `EXPLAIN JSON`,
and PocketIC.

What landed in this patch:

- [ranking.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/ranking.rs)
  now defines the shared deterministic ranking contract used by the current
  single-entity visible-index planner
- prefix, range, compare, and predicate planning now consume that shared score
  instead of relying on local preference order in:
  [prefix.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/prefix.rs),
  [range.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/range.rs),
  [compare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/compare.rs), and
  [predicate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/predicate.rs)
- the first canonical comparison key for this line is now explicit and
  planner-owned:
  - `prefix_len`
  - family-specific `exact`
  - `order_compatible`
  - lexicographic index name as the final stable tie-break
- planner-choice explain now projects from that same resolved score instead of
  telling a second locally reconstructed story in:
  [mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/mod.rs),
  [evaluator.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/evaluator.rs), and
  [model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/model.rs)
- verbose explain now exposes planner-authored `order_compatible_preferred`
  reasoning on both the chosen route and rejected competing routes for the
  first tied cohorts
- core deterministic-choice locks now exist at three layers:
  - planner unit tests in
    [tests.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/tests.rs)
  - intent/verbose explain tests in
    [tests.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests.rs)
  - public session fluent verbose explain tests in
    [mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/mod.rs)
- public SQL coverage now locks the same deterministic choice through generated
  parity `EXPLAIN JSON`, using explicit competing-index fixtures in:
  [schema.rs](/home/adam/projects/icydb/schema/test/sql_parity/src/schema.rs),
  [fixtures.rs](/home/adam/projects/icydb/schema/test/sql_parity/src/fixtures.rs), and
  [tests.rs](/home/adam/projects/icydb/canisters/test/sql_parity/src/tests.rs)
- that parity coverage introduces:
  - `PlannerPrefixChoice` for prefix-family deterministic choice
  - `PlannerChoice` for range-family and order-only deterministic choice
- PocketIC mirrors the same public SQL `EXPLAIN JSON` deterministic-choice
  locks in
  [sql_canister.rs](/home/adam/projects/icydb/testing/pocket-ic/tests/sql_canister.rs)
- the first bounded deterministic cohorts are now frozen end to end:
  - prefix ties
  - range ties
  - order-only fallback ties

What did not change:

- no new ranking dimensions beyond the current canonical key
- no statistics-driven or cost-based optimizer
- no planner-visible index policy widening
- no runtime correctness or authority logic
- no grouped aggregate planning changes
- no relation/join planning changes
- no distinct-planner redesign

This leaves the next `0.72.x` work as narrower deterministic-planning coverage
or packaging follow-up, not a broader optimizer rewrite.

## 0.72.1

`0.72.1` keeps the `0.72.0` ranking key unchanged and extends the deterministic
coverage line to the first admitted composite ordered-read cohorts, including
their real descending behavior.

What landed in this patch:

- the admitted composite cohorts are now locked end to end instead of being
  implied by the current planner behavior:
  - equality-prefix plus suffix-order
  - mixed equality-plus-range ordered scans
  - composite order-only fallback
- planner unit tests now freeze both ascending and descending deterministic
  choice for those cohorts in
  [tests.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/tests.rs)
- verbose explain now mirrors that same ascending/descending deterministic
  choice story in:
  [tests.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests.rs)
  and
  [mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/mod.rs)
- public session `EXPLAIN EXECUTION` now locks the bounded route shapes for the
  admitted composite families in
  [mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/mod.rs)
- generated SQL parity now locks both `EXPLAIN JSON` choice and
  `EXPLAIN EXECUTION` route shape for those same cohorts in
  [tests.rs](/home/adam/projects/icydb/canisters/test/sql_parity/src/tests.rs)
- PocketIC mirrors the same public SQL contracts in
  [sql_canister.rs](/home/adam/projects/icydb/testing/pocket-ic/tests/sql_canister.rs)

The important behavioral split recorded by this patch is:

- ascending equality-prefix plus suffix-order keeps the chosen bounded
  `IndexPrefix` route with `TopNSeek`
- descending equality-prefix plus suffix-order keeps the same deterministic
  index choice, but fails closed to `OrderByMaterializedSort` instead of
  claiming the ascending fast-path contract
- ascending and descending mixed range scans stay on the chosen bounded
  `IndexRange` route with `IndexRangeLimitPushdown`, but do not claim
  `TopNSeek`
- ascending and descending composite order-only fallback stays on the chosen
  bounded `IndexRange` route with both `IndexRangeLimitPushdown` and
  `TopNSeek`

What did not change:

- no new ranking dimensions
- no cost model or planner weights
- no planner-visible index policy widening
- no runtime correctness or authority logic
- no grouped, relation, or distinct planner work

This leaves the next `0.72.x` work as broader ordered-window exploitation or
planner-emitted rejection/fallback detail, not more silent route-admission
drift.
