# 0.150.x

## 0.150.3

`0.150.3` moves runtime field-path cursor-anchor and forward/unique write
index-key construction onto accepted index contracts. Index-range pagination
and write planning still receive temporary generated `IndexModel` DTOs, but
field-path cursor anchors and write keys now look up accepted `SchemaInfo`
index contracts, use accepted index ordinals for `IndexId`, read key
components from accepted row-layout slots, and carry accepted store/uniqueness
facts through preflight read/materialization boundaries. Generated expression
index construction remains on the explicit deferred lane until accepted
expression contracts land.

### Accepted Cursor Anchor Index Keys

- Added an accepted field-path index-key constructor that consumes
  `SchemaIndexInfo` directly.
- Routed runtime `EntityAuthority::index_key_from_slot_ref_reader` through the
  accepted field-path constructor for field-path indexes.
- Kept expression-index cursor-anchor construction on the generated deferred
  lane because accepted expression index contracts are not designed yet.
- Made missing accepted field-path cursor-anchor contracts fail closed instead
  of falling back to generated index field-slot authority.
- Extended write-boundary guards to prove cursor-anchor key construction uses
  accepted index contracts for field-path indexes.

### Accepted Forward Index Write Keys

- Added an accepted field-path index-key constructor for canonical slot
  readers, sharing the accepted component encoder with cursor-anchor key
  construction.
- Routed structural forward-index mutation planning through accepted
  `SchemaInfo` index contracts for field-path indexes.
- Routed unique-index stored-row key rebuilds through accepted field-path index
  contracts.
- Carried accepted field-path index store paths, uniqueness, and diagnostic
  field lists into forward-index delta planning and preflight index readers.
- Narrowed preflight index read ports so field-path write planning no longer
  passes generated `IndexModel` definitions merely to decode index-entry
  membership.
- Preserved generated expression-index write key construction on the explicit
  deferred lane.
- Extended write-boundary guards to prove forward/unique field-path key
  construction and preflight index readers consume accepted contract facts.

## 0.150.2

`0.150.2` starts the planner access-choice authority handoff by adding a
planner-facing accepted field-path index contract to runtime `VisibleIndexes`
and using that accepted contract for runtime order-only field-path access
fallback. The downstream access planner still assembles access paths with the
temporary generated `IndexModel` bridge, but runtime field-path ORDER BY
compatibility now matches against accepted index terms instead of generated
`IndexModel` field order. Generated order fallback is kept on an explicit
model-only lane, and generated expression indexes remain on their explicit
deferred runtime lane until accepted expression contracts are designed.

### Accepted Planner Index Contracts

- Added `AcceptedPlannerFieldPathIndex` and
  `AcceptedPlannerFieldPathIndexField` as the planner-facing accepted
  field-path index DTOs carried by `VisibleIndexes`.
- Built those planner contracts from accepted `SchemaInfo` index metadata when
  runtime visible indexes are selected.
- Kept the generated `IndexModel` bridge inside the accepted planner contract
  only as a temporary access-path DTO bridge until access paths stop storing
  generated index metadata.
- Added runtime debug invariants that prove accepted planner index contracts
  stay consistent with the temporary generated bridge.
- Routed runtime order-only field-path access fallback through accepted
  planner index order terms.
- Split generated order-only fallback into an explicit generated/model-only
  lane so missing accepted field-path index contracts fail closed in runtime
  planning instead of silently reusing generated `IndexModel` field order.
- Preserved generated expression-index order fallback as an explicit deferred
  runtime lane until accepted expression index contracts exist.
- Extended the visible-index boundary guard and ready/non-ready secondary-index
  session test so accepted planner contracts are present and internally
  consistent before downstream planner scoring consumes them.

## 0.150.1

`0.150.1` moves the runtime visible-index boundary onto accepted schema
metadata. Store-ready session planning now builds `VisibleIndexes` by filtering
the temporary generated `IndexModel` planner bridge through accepted
`SchemaInfo` field-path index contracts. Generated expression indexes remain
visible through their explicit deferred lane until accepted expression contracts
are designed. This keeps the downstream planner compatible for now while
preventing runtime field-path visible-index selection from using raw generated
model index declarations as the authority.

### Accepted Visible-Index Filtering

- Changed `VisibleIndexes` so it can carry an owned accepted-schema-filtered
  planner bridge plus an accepted field-path index authority marker.
- Preserved generated expression-index visibility explicitly while field-path
  index visibility is filtered through accepted contracts.
- Routed shared query-plan cache misses and typed query visible-index
  callbacks through accepted `SchemaInfo` before choosing planner-visible
  secondary indexes.
- Routed SQL execution explain visible-index resolution through the accepted
  `SchemaInfo` path.
- Extended the non-ready secondary-index session test so the ready state first
  proves visible indexes are backed by accepted field-path contracts.
- Added a write-boundary guard proving runtime visible-index construction uses
  accepted `SchemaInfo` filtering and no longer exposes the old
  `visible_indexes_for_model` / `visible_indexes_for_store_model` helpers.

## 0.150.0

`0.150.0` starts accepted index authority with the field-path half of the
index contract. Accepted schema snapshots now have a persisted index-contract
surface, generated field-path indexes project into accepted field IDs and row
slots during schema proposal construction, and the persisted schema codec now
hard-cuts to a shape that carries those index contracts. Accepted `SchemaInfo`
also reads top-level index membership and field-path index metadata from the
accepted snapshot instead of generated `EntityModel` index metadata. Expression
indexes are intentionally not accepted yet; they still need a stable canonical
expression contract before runtime can use them as accepted authority.

### Accepted Field-Path Index Contracts

- Added `PersistedIndexSnapshot`, `PersistedIndexKeySnapshot`, and
  `PersistedIndexFieldPathSnapshot` to accepted schema snapshots.
- Added `PersistedSchemaSnapshot::new_with_indexes(...)` and retained the
  field-only constructor for owner-local field-focused fixtures.
- Projected generated field-path indexes from schema proposals into accepted
  top-level `FieldId`, accepted row slot, full field path, persisted field
  kind, nullability, uniqueness, ordinal, store path, and predicate SQL display
  metadata.
- Skipped generated expression index proposals in this first field-path slice
  so expression contracts cannot become accepted runtime authority before their
  stable representation is designed.
- Hard-cut the persisted schema snapshot codec to version `3` and round-tripped
  accepted field-path index contracts through Candid.
- Added accepted snapshot integrity validation for duplicate index identities,
  empty index keys, and index field slots that diverge from the accepted row
  layout.
- Routed accepted `SchemaInfo` field index-membership flags through persisted
  index contracts instead of generated `EntityModel::indexes`.
- Added an internal `SchemaInfo` field-path index metadata surface backed by
  accepted index contracts, including ordinal, name, store, uniqueness, field
  ID, accepted row slot, field path, persisted kind, nullability, and predicate
  SQL display metadata.
- Kept generated `SchemaInfo` views on generated index metadata for
  proposal/model-only callers.
- Extended write-boundary guards so accepted `SchemaInfo` construction cannot
  regain generated index membership lookup or lose its persisted index metadata
  surface.
