# 0.103 Changelog

## 0.103.0

- `0.103.0` starts the residual-explicitness slice by freezing residual filter state directly into finalized planning shape instead of asking executor and explain consumers to keep rediscovering it from `filter_expr`, `predicate`, and access satisfaction.
- Finalized scalar planning state now carries three distinct filter views in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_plan.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_plan.rs) and [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs):
  - `execution_preparation_predicate`
  - `residual_filter_expr`
  - `residual_filter_predicate`
- Planning now derives those artifacts once during static-shape finalization and reuses them for:
  - execution-preparation predicate compilation
  - effective runtime filter program selection
  - route and covering eligibility checks
  - execution explain projection
- This patch also closes the biggest explain-model gap that remained after `0.100` and `0.102`: execution explain no longer assumes that “no predicate object” means “no filter stage worth rendering.” The execution descriptor now carries `residual_filter_expr`, and the residual stage renderer emits a `ResidualFilter` node whenever explicit residual filter state exists, even if there is no derived residual predicate:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/execution.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/execution.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/render.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/render.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/json.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/json.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/shared/predicate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/shared/predicate.rs)
- The executor and route side now consume that explicit residual state more directly across the finalized-plan boundary, including:
  - covering-read compatibility and covering route rejection
  - scalar route capability and ordered streaming fallback
  - grouped ordered-strategy fallback
  - scalar page structural planning
  - prepared execution’s residual-presence checks
- `0.103.0` also includes a naming cleanup where route and scalar page executor terminology now says “residual filter” instead of encoding the older predicate-only worldview. The most visible public-facing route reason is now `residual_filter_blocks_direct_streaming`.
- During this slice, one real regression surfaced: the first residual-explicitness cut introduced a pre-finalization recursion between residual derivation and “predicate fully satisfied” checks. That showed up as a stack overflow in the broader explain matrix test. The shipped fix changes the pre-finalization semantic check so residual-expression survival is decided from planner-owned expression compatibility instead of recursively consulting the residual helper itself. That matters because it keeps the semantic owner on `Expr` even when one derived predicate subset is also present.
- The key new invariants pinned in tests are:
  - expression-only residual filters finalize into explicit residual expression state and compile onto the runtime residual filter lane
  - mixed semantic filters can keep an explicit residual expression even when a derived predicate subset also exists
  - `EXPLAIN EXECUTION` renders residual stages for expression-owned filters instead of silently omitting them
  - the broader session explain matrix no longer overflows the stack after the residual-finalization change
