# IcyDB 0.172 Changelog

## 0.172.5

### 🔧 Changed

- Closes the journaled-store reserved-contract proof matrix.
- Records that production journaled storage remains unadmitted in 0.172:
  runtime storage modes, accepted schema metadata, generated wiring, registry
  handles, diagnostics, relation policy, executor mutation policy, and
  commit/recovery paths remain stable/heap only.
- Locks the 0.173 admission gates: journaled runtime work must choose
  transaction visibility authority, preserve heap volatility, keep stable ABI
  and codecs unchanged unless explicitly hard-cut, and prove journal fold/replay
  bounds before production admission.

### 🧪 Testing

- Ties closeout to the reserved syntax diagnostics, stable/heap capability
  tests, heap volatility/reinit tests, stable recovery tests, diagnostics
  rendering tests, and the 0.172 source-audit guards.

## 0.172.4

### 🔧 Changed

- Defines the future journal persisted-format boundary without introducing
  journal bytes, replay code, or fold runtime.
- Records the required future journal format properties: versioned envelope,
  bounded record and payload sizes, fallible decode, deterministic replay
  ordering, logical row/schema mutation records, transaction-visibility
  handoff, and fold crash-safety.
- Keeps row, index, schema, and commit-marker codec contracts unchanged in
  0.172.

### 🧪 Testing

- Adds a source-audit guard proving production row/index/schema codecs and the
  commit-marker store/marker surfaces do not introduce journal records,
  journal format state, or journal/fold codec paths.

## 0.172.3

### 🔧 Changed

- Records the commit/recovery boundary inventory for the reserved journaled
  contract without implementing journal replay or journal folding.
- Confirms the current durable write authority remains the existing
  commit-marker protocol: `begin_commit` persists row-op authority,
  `finish_commit` clears it only after successful apply, and recovery replays
  recorded row ops rather than planner logic.
- Keeps future journaled transaction visibility as a later admission decision:
  existing commit marker authority, journal commit-record authority, or an
  explicit hybrid must be chosen before runtime admission.

### 🧪 Testing

- Adds a source-audit guard proving commit-marker projection, replay, and
  startup rebuild consume recovery capabilities while no production
  commit/mutation/data/index/schema code introduces journal replay or fold
  surfaces in 0.172.

## 0.172.2

### 🔧 Changed

- Records the capability-readiness audit for the journaled reserved contract:
  future journaled capabilities remain design-only, while runtime capability
  surfaces stay limited to stable and heap storage.
- Keeps journaled storage out of accepted schema metadata, generated wiring,
  runtime registry handles, diagnostics projection, relation policy, executor
  mutation policy, and commit/recovery policy.

### 🧪 Testing

- Adds a source-audit guard proving the only active journaled implementation
  hook is derive-parser reserved diagnostics.
- Reuses the existing stable/heap capability tests as the live runtime
  capability evidence; no runtime journaled capability constructor or test
  convenience path was added.

## 0.172.1

### 🔧 Changed

- Recognizes `storage(journaled(...))` only at the derive parser boundary so
  IcyDB can emit exact reserved-contract diagnostics without admitting
  journaled storage into accepted schemas.
- Treats the full four-ID spelling as recognized-but-reserved:
  `data_memory_id`, `index_memory_id`, `schema_memory_id`, and
  `journal_memory_id`.
- Treats journal-only spelling as malformed because a future journaled
  cached-stable store requires stable data/index/schema source-of-truth
  memories plus a journal tail.

### 🧪 Testing

- Adds derive parser unit coverage for reserved journaled storage,
  journal-only malformed storage, and unknown journaled fields.
- Adds compile-fail UI fixtures for the same reserved and malformed
  journaled-storage forms.

## 0.172.0

### ➕ Added

- Starts the journaled-store reserved-contract line.
- Adds the 0.172 design for reserving `storage(journaled(...))` as a future
  journaled cached-stable storage contract.
- Defines the intended future four-memory shape:
  `data_memory_id`, `index_memory_id`, `schema_memory_id`, and
  `journal_memory_id`.

### 🔧 Changed

- Clarifies that journaled storage is not production-admitted in 0.172.
- Defines stable data/index/schema BTrees as the future canonical durable
  source of truth and `journal_memory_id` as the durable tail for accepted
  writes not yet folded into those stable BTrees.
- Records that volatile Rust BTrees are future read caches/live views with no
  memory IDs and no durability authority.
- Updates older storage-boundary design notes to describe journaled storage as
  cached-stable with a journal tail rather than heap-fronted checkpoint
  storage.

### 🧪 Testing

- Adds planned proof rows for reserved journaled diagnostics, parser/runtime
  separation, stable and heap regression proof, codec preservation, source
  audits, journal fold safety, and future runtime-admission handoff.
- Leaves runtime behavior unchanged.
