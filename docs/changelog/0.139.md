# 0.139.x

## 0.139.0

`0.139.0` completes the aggregate identity convergence follow-through started
after `0.138.9`. SQL global aggregates and grouped aggregates now share the
same `AggregateIdentity` rules for observable aggregate meaning, while
`AggregateSemanticKey` treats aggregate-local `FILTER` expressions as a
separate semantic dimension.

What changed:
- added `AggregateSemanticKey` as the filter-aware aggregate equivalence key:
  `AggregateIdentity` owns aggregate kind, input identity, and observable
  `DISTINCT`; the semantic key adds the aggregate-local filter expression
- changed grouped aggregate equality, grouped execution-spec matching, grouped
  `HAVING` aggregate lookup, and grouped validation to compare semantic keys
  instead of rebuilding parallel raw aggregate comparisons
- changed SQL global aggregate semantics to wrap the shared aggregate identity
  core instead of independently owning DISTINCT normalization
- changed grouped execution specs to expose identity-owned grouped DISTINCT
  decisions to the executor handoff
- removed the leftover `AggregateKind` helper that duplicated grouped
  value-dedup DISTINCT meaning for `COUNT` / `SUM` / `AVG`
- added `GroupedDistinctExecutionMode` so grouped reducer state receives the
  planner-prepared DISTINCT admission mode instead of reinterpreting aggregate
  kind at runtime
- added focused tests proving `MIN(x)` and `MIN(DISTINCT x)` share grouped
  identity, `COUNT(x)` and `COUNT(DISTINCT x)` remain distinct, and filtered
  aggregates do not alias each other

Boundary notes:
- `MIN(DISTINCT ...)` and `MAX(DISTINCT ...)` continue to behave like plain
  `MIN` / `MAX`
- `COUNT(DISTINCT ...)`, `SUM(DISTINCT ...)`, and `AVG(DISTINCT ...)` remain
  distinct-sensitive
- aggregates with different `FILTER` expressions remain separate even when the
  aggregate kind, input, and DISTINCT mode match
- grouped planner eligibility rules and grouped execution routes now consume
  identity-owned facts instead of duplicating identity rules
- raw parsed/builder fields such as aggregate kind, input expression, and raw
  DISTINCT remain allowed at construction boundaries before identity
  construction
