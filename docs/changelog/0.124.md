# 0.124 Changelog

## 0.124.1

- `0.124.1` finishes the scalar-function behavior-family cleanup by moving the remaining shared preview/runtime value transforms onto the canonical function owners instead of leaving those last bodies duplicated between planner preview and executor scalar projection evaluation.

- Shared scalar-evaluation behavior ownership now includes the actual value-level transforms, not just the dispatch family:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/function_semantics.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/function_semantics.rs)
  - added owner-side evaluators on the admitted scalar-function subkinds:
    - `NullTestFunctionKind::eval_value(...)`
    - `UnaryTextFunctionKind::eval_text(...)`
    - `UnaryNumericFunctionKind::eval_decimal(...)`
    - `LeftRightTextFunctionKind::eval_text(...)`
    - `TextPredicateFunctionKind::eval_text(...)`
  - added owner-side evaluators on `Function` for the remaining dynamic/shared bodies:
    - `eval_coalesce_values(...)`
    - `eval_nullif_values(...)`
    - `eval_position_text(...)`
    - `eval_replace_text(...)`
    - `eval_substring_text(...)`
    - `eval_round_numeric(...)`

- Planner literal preview is now a thin layer over the shared owner-side behavior surface:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/preview.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/preview.rs)
  - the preview path still owns:
    - literal-only subtree admission
    - literal argument decoding
    - planner compare-path reuse for `NULLIF`
  - but it no longer owns separate copies of:
    - null-test result shaping
    - unary text transforms
    - unary numeric transforms
    - LEFT/RIGHT text slicing
    - text predicate evaluation
    - `COALESCE`
    - `NULLIF` result shaping
    - `POSITION`
    - `REPLACE`
    - `SUBSTRING`
    - `ROUND`

- Executor scalar projection evaluation is now the same kind of thin wrapper over the shared owner-side behavior surface:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/eval/scalar_function.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/eval/scalar_function.rs)
  - the executor path still owns:
    - runtime argument presence checks
    - runtime type validation
    - executor-facing `QueryError` construction
    - runtime binary compare reuse for `NULLIF`
  - but it no longer owns separate copies of the same value transforms listed above

- This also removes the last local helper-only copies of the shared string and substring mechanics from the preview/runtime seam:
  - `text_position_1_based(...)`
  - `left_chars(...)`
  - `right_chars(...)`
  - `substring_1_based(...)`
  - those mechanics now live on the canonical function-owner side instead of existing twice under preview and runtime

- The result is a cleaner stop condition for the scalar-function formalization line:
  - parser and lowering already read owner-side function identity and admission
  - planner typing and predicate shaping already read owner-side function families and subkinds
  - preview and runtime now also read owner-side value behavior instead of keeping parallel body implementations
  - the remaining layer differences are now only the real boundary differences:
    - argument decoding
    - layer-specific comparison entrypoints
    - layer-specific error construction

- Validation covered:
  - `cargo fmt --all`
  - `cargo test -p icydb-core sql_projection -- --nocapture`
  - `cargo test -p icydb-core semantics -- --nocapture`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`

## 0.124.0

- `0.124.0` starts the scalar-function semantics formalization line by introducing one planner-owned function registry and then making SQL parser and lowering consume that registry-driven ownership surface instead of keeping phase-local function family tables.

- The new planner-owned registry lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/function_semantics.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/function_semantics.rs):
  - added:
    - `FunctionCategory`
    - `FunctionNullBehavior`
    - `FunctionDeterminism`
    - `FunctionTypeInferenceShape`
    - `FunctionSpec`
  - `Function` now owns:
    - `spec()`
    - `type_inference_shape()`

- Planner typing no longer acts as the hidden scalar-function registry owner:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/type_inference/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/type_inference/mod.rs)

  - compare-operand coarse-family checks now also read the shared function registry surface instead of staying local to type inference

- The canonical planner function taxonomy in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/ast.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/ast.rs) is now mechanically ordered and de-duplicated:

- Parser-side SQL expression tree queries now share one recursive walk seam instead of repeating the same structural descent for each question:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/model.rs)

- Supported `ORDER BY` expression reconstruction is now driven by one clause-owned function-call shape surface instead of separate parser and renderer ladders:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/ast.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/ast.rs)

- This keeps the first `0.124.0` cut bounded:
  - one planner-owned scalar-function registry now exists
  - planner typing consumes it instead of owning a parallel hidden ladder
  - SQL parser and SQL lowering now consume enum-owned scalar-function identity, call-shape, surface-admission, casefold, and aggregate-input fold ownership
  - aggregate-input canonicalization no longer exists as two parallel recursive fold/normalize implementations in builder and SQL lowering
  - parser-side `SqlExpr` tree queries and planner-side supported `ORDER BY` expression reconstruction no longer keep separate local recursion and function-shape ladders
  - boolean predicate shaping now reads shared function-family and subkind ownership instead of repeating local function-variant ladders
  - duplicate `CEIL` versus `CEILING` semantic identities are gone from the live planner and parser taxonomies
  - evaluator redesign and broader scalar-function execution coverage remain separate follow-up work
