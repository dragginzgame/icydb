# 0.124 Changelog

## 0.124.3

- `0.124.3` hard-cuts the public fluent filter literal surface away from the runtime `Value` union. `FilterExpr` now carries only frontend-safe `FilterValue` literals (`String`, `Bool`, `Null`, `List`), and the schema-aware query boundary rehydrates typed IDs, enums, numerics, and relation-key literals before the planner consumes the predicate.

**Filter wire cut**

- The public filter expression contract now uses a narrowed wire literal enum instead of embedding runtime `Value` directly:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/expr.rs)
    - added `FilterValue`
    - `FilterExpr::{Eq, Ne, Lt, Lte, Gt, Gte, EqCi, In, NotIn, Contains, TextContains, TextContainsCi, StartsWith, StartsWithCi, EndsWith, EndsWithCi}` now carry `FilterValue`
    - fluent filter constructors still accept ordinary Rust values, but those values are now serialized onto the public filter wire as:
      - `Bool`
      - `Null`
      - recursive `List`
      - canonical string transport for all other scalar values
- This is a hard cut:
  - there is no parallel frontend-only filter type
  - there is no compatibility shim that leaves the old `Value`-shaped filter wire contract live beside the new one

**Schema-aware rehydration**

- Fluent filter lowering no longer pushes raw public filter literals straight into planner expressions:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/expr.rs)
  - query intent now lowers `FilterExpr` through `lower_bool_expr_for_model(...)`, so literal rehydration happens with the authoritative entity schema in hand before planner normalization and validation run
- Planner-owned field-kind literal canonicalization now owns the typed rehydration step for frontend filter literals too:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/field_kind_semantics.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/field_kind_semantics.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/mod.rs)
  - added `canonicalize_filter_literal_for_kind(...)`
  - the schema-aware filter boundary now rehydrates:
    - `Ulid`
    - `Principal`
    - `Account`
    - enums from loose variant-name strings
    - signed/unsigned integer families
    - decimal and float wrappers
    - relation keys through their referenced `key_kind`
    - list/set members recursively
- Text-family operations like `STARTS_WITH`, `ENDS_WITH`, `CONTAINS`, and casefold equality still keep their literal payloads as text-oriented wire values instead of trying to reinterpret them through non-text field kinds

**Facade surface**

- The narrowed public filter literal type is now re-exported through the facade crates:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/mod.rs)
  - [/home/adam/projects/icydb/crates/icydb/src/db/query/expr.rs](/home/adam/projects/icydb/crates/icydb/src/db/query/expr.rs)
  - [/home/adam/projects/icydb/crates/icydb/src/db/query/mod.rs](/home/adam/projects/icydb/crates/icydb/src/db/query/mod.rs)
  - [/home/adam/projects/icydb/crates/icydb/src/lib.rs](/home/adam/projects/icydb/crates/icydb/src/lib.rs)
