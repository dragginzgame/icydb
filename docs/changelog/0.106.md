Instead, `SqlExplainStatement` now carries one explicit `verbose` flag alongside the existing explain mode in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/model.rs), and lowering preserves that flag through the shared SQL explain command path in:

## 0.106.1

- `0.106.1` adds the next bounded planner-quality signal on top of the shipped `0.106.0` structural ranking: residual burden now participates in route choice, but only among already-close valid candidates. This patch does not add statistics, numeric costs, or broad cross-family optimizer behavior.
- The rerank entrypoint is now wired into scalar plan assembly in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs). After the ordinary structural access route is selected, the planner can do one bounded same-score pass over nearby index candidates and keep the route that discharges more of the filter before residual execution.
- The bounded same-score residual comparison lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/mod.rs). The planner-visible residual categories are intentionally coarse:
  - no residual work
  - residual derived predicate only
  - residual scalar expression
  - within one residual-predicate category, fewer remaining predicate terms win
- This patch keeps the `.1` contract narrow:
  - residual burden only breaks ties or near-ties among routes that were already structurally comparable
  - it does not let a weak access path beat a clearly stronger one just because the residual looks smaller
  - it does not invent row-count estimates or selectivity math
- The access-choice explain taxonomy now includes the public reason code `residual_burden_preferred` in:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/model.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/evaluator/ranking.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/evaluator/ranking.rs)
- Verbose execution explain now surfaces the same planner-owned access-choice state in two forms:
  - the existing stable `diag.r.access_choice_*` diagnostics
  - a new human-readable `Access choice` section rendered from the same planner snapshot in [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/load.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/load.rs)
- That new verbose section does not introduce a second scoring path. Candidate labels, residual burden categories, and the final decision all come from the planner-owned access-choice snapshot projected in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/mod.rs).

## 0.106.0

- `0.106.0` starts the bounded cost-aware route-selection line without turning the planner into a full optimizer. The slice only compares routes that were already semantically valid; it does not widen SQL admission, predicate extraction, or residual semantics.
- The shared structural ranking contract lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/ranking.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/ranking.rs). `AccessCandidateScore` now carries more of the planner-visible signals that matter for deterministic route choice:
  - filtered versus unfiltered candidate status
  - bounded-range strength via `range_bound_count`
  - existing order-compatibility status
- That same ranking surface is projected into planner-choice explain in:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/model.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/evaluator/ranking.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/evaluator/ranking.rs)
- The shipped `0.106.0` route-quality signals are:
  - filtered index candidates outrank otherwise tied unfiltered siblings
  - two-sided or otherwise stronger bounded ranges outrank weaker same-family range candidates before lexicographic fallback
  - singleton primary-key child access discovered inside one mixed `AND` outranks a broader secondary range candidate
  - explicit empty child access discovered inside one mixed `AND` outranks a broader secondary route instead of being shadowed by it
  - primary-key half-open ranges remain visible even when sibling conjuncts still exist, instead of requiring the whole `AND` group to be nothing but PK-range clauses
  - when two already-valid routes compete and one preserves the required order while the other does not, the order-preserving route can win at the family-competition boundary
- The primary-key range preservation work is centered in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/range/extract.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/range/extract.rs). `primary_key_range_from_and(...)` now derives its candidate from the primary-key subset of one canonical `AND` tree instead of rejecting the whole candidate as soon as unrelated sibling clauses appear.
- The family-level competition rules live in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/predicate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/predicate.rs). The important boundary here is intentional:
  - this patch does not say “`ORDER BY id` is special”
  - it says one candidate may outrank another when it preserves the required order and the competitor does not
  - the current concrete case is a primary-key range beating one unordered secondary prefix/range when the required order is primary-key order
- The access-choice reason taxonomy now includes the new public explain reason:
  - `stronger_range_bounds_preferred`
  This keeps the shipped route choice auditable without adding numeric costs or a separate optimizer explain surface.

  - planner access-choice unit tests in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/tests/mod.rs)
  - planner route-selection tests in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/tests/mod.rs)
