# 0.109 Changelog

## 0.109.0

- `0.109.0` starts the semantic-aware plan-reuse line by making the canonical identity work from `0.108` operationally visible instead of only hash-visible.
- The shipped artifact class in this first cut is explicit and bounded: shared prepared query-plan reuse. This patch does not add fuzzy matching, partial-plan reuse, new canonicalization rules, or any broader optimizer behavior.
- The outward reuse model is now visible through three surfaces:
  - `trace_query(...)`
  - fluent/session verbose execution explain
  - SQL `EXPLAIN EXECUTION VERBOSE`
- The core trace contract landed in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/trace.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/trace.rs). `TraceReuseArtifactClass` and `TraceReuseEvent` now expose one real shipped reuse class:
  - `SharedPreparedQueryPlan`
- The session-owned reuse path is wired in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs). That file now:
  - reports cache attribution as an explicit trace reuse event
  - emits `diag.s.semantic_reuse_artifact=shared_prepared_query_plan`
  - emits `diag.s.semantic_reuse=hit|miss`
  - keeps this state session-owned instead of pretending cache reuse is planner-owned route state
- SQL verbose explain now projects the same reuse diagnostics from the same cached plan path in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/explain.rs). Plain `EXPLAIN EXECUTION` remains unchanged; the extra `diag.s.*` lines only appear on the verbose path.
- The query-owned verbose renderer gained one bounded extension point in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs), so caller-owned diagnostics can be appended without promoting session cache state into planner-owned explain fields.
- The broader workspace pass also flushed out one real cached-verbose regression: the first `0.109.0` cut was reusing the prepared plan but skipping explain-side access-choice finalization, which flattened chosen reasons to the fail-closed seed `selected_index_not_projected`. The fix landed in:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/explain.rs)
- After that fix, cached fluent and SQL verbose explain surfaces still show the true planner-owned ranking reason, such as:
  - `order_compatible_preferred`
  - `stronger_range_bounds_preferred`
  - `residual_burden_preferred`
  instead of degrading the chosen reason just because the logical plan came from the shared prepared-plan cache.
- This patch also fixed one real outward identity mismatch. Before `0.109.0`, trace-driven reuse could miss correctly on distinct scalar projection shapes while still reporting the same outward `plan_hash()` because scalar projection identity had collapsed to one fixed sentinel. The fix landed in:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/projection.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs)
- After that fix:
  - scalar `SELECT *` keeps the stable default projection sentinel
  - scalar field projections contribute their actual selected-field identity
  - outward plan hashes now stay aligned with the same exact-match reuse key that the shared query-plan cache uses
- The important shipped hit/miss boundaries are now pinned directly:
  - canonical-equivalent scalar filter order can reuse
  - canonical-equivalent grouped fluent `HAVING` order can reuse
  - distinct semantic identity stays a miss for:
    - different ordering / limit shape
    - different projection shape
    - different grouping shape
    - searched-`CASE` forms whose canonical identity still remains distinct on the current surface
