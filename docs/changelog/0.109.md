# 0.109 Changelog

## 0.109.1

- `0.109.1` closes one remaining cache-identity split from the `0.108` / `0.109.0` line: canonical scalar `filter_expr` is now the sole shared structural filter identity owner whenever it exists.
- Before this patch, [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs) still embedded both:
  - the planner-owned canonical scalar `filter_expr`
  - the derived normalized predicate fingerprint
  inside `StructuralQueryCacheKey`.
- That was architecturally wrong for expression-owned scalar filters. It meant the shared query-plan cache key could still drift with predicate-subset extraction policy even when a canonical semantic filter expression already existed and was supposed to own identity by itself.
- The fix is narrow and explicit:
  - when scalar `filter_expr` is present, `StructuralQueryCacheKey` now stores that expression-owned cache key and discards the separate predicate key
  - when scalar `filter_expr` is absent, predicate-only queries still keep predicate identity exactly as before
- The actual behavior change landed in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs). This patch does not change plan semantics, predicate extraction, or route choice. It only removes one over-keying path from shared semantic reuse.
- Because this changes the meaning of in-heap shared query-plan cache entries, [/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs) bumps `SHARED_QUERY_PLAN_CACHE_METHOD_VERSION` so old in-memory entries cannot alias under the new ownership rule.
- Coverage added in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests/cache_key.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests/cache_key.rs) now pins both sides of the boundary:
  - different predicate fingerprints are ignored when one canonical scalar `filter_expr` already owns structural identity
  - predicate-only queries still stay distinct when no scalar `filter_expr` exists
- `0.109.1` also closes one more semantic-cache aliasing hole inside that same expression-owned identity line: unary operators now remain part of `ProjectionExprCacheKey` in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs) instead of being dropped and replaced by their child expression.
- Before that fix, the shared query-plan cache could incorrectly treat `NOT(x)` as identical to `x` for expression-owned scalar filters. In practice that meant positive and negated SQL boolean filters such as `active IS TRUE` and `active IS NOT TRUE` could alias on one cached plan shape even though their semantic filter expressions were different.
- Coverage now pins that boundary directly:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests/cache_key.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests/cache_key.rs) proves unary boolean filters stay distinct at the structural cache-key layer
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs) proves `IS TRUE` and `IS NOT TRUE` build separate shared query-plan entries and do not reuse the same semantic plan at runtime
- Existing reuse proofs still stay green after the ownership change:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs)
  - canonical-equivalent scalar filter forms still reuse one shared query-plan entry
  - trace parity for canonical-equivalent scalar filters still reports one reused prepared query plan
- The same patch also starts the next route-selection cleanup line without changing shipped route policy. [`Predicate::And`](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/predicate.rs) no longer encodes its family-level winner ordering as a chain of early returns for:
  - explicit empty child access
  - singleton primary-key child access
  - required-order-preserving primary-key range
  - secondary range / secondary prefix fallback
- Instead, that winner order now flows through one explicit `AndFamilyCandidateScore` comparison path in:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/ranking.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/ranking.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/predicate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/predicate.rs)
- This is intentionally a consolidation, not a new planner policy. The preserved family order is still:
  - explicit empty child
  - singleton primary-key child
  - required-order-preserving primary-key range over an unordered competing secondary route
  - secondary range
  - secondary prefix
  - intersection fallback when no family candidate outranks the child plans
- `0.109.1` also tightens the outward explain surface for those non-index winners. Before this patch, access-choice explain collapsed all non-index selected routes to the generic chosen-reason code `non_index_access`, even when the chosen root was concretely:
  - `ByKey`
  - `ByKeys`
  - `PrimaryKeyRange`
  - `FullScan`
- That was too lossy for the current route-selection cleanup line. The new bounded projection in:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/mod.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/model.rs)
  now reports:
  - `by_key_access`
  - `by_keys_access`
  - `primary_key_range_access`
  - `full_scan_access`
  while still leaving composite non-index routes on the generic `non_index_access` fallback until a planner-owned non-index family winner reason is stored explicitly.
- Seeded plan construction in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_plan.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_plan.rs) now uses the same bounded non-index chosen-reason projection, so manually assembled test plans and finalized planner-owned plans stay on one explain contract.
- The same file now also keeps that seeded non-index snapshot intact during `finalize_access_choice_for_model_with_indexes(...)`. Non-index plans no longer reproject themselves back through the generic finalize path after the concrete chosen reason has already been seeded, so `ByKey`, `ByKeys`, `PrimaryKeyRange`, and `FullScan` keep the same bounded reason across seeded construction, planner finalization, and outward explain rendering.
- `0.109.1` now pushes that ownership one step earlier too: the actual planner/intent handoff stores an explicit planner-owned non-index winner reason on the selected query plan instead of forcing explain finalization to infer it again from `AccessPath` shape.
- That planner-owned handoff now flows through:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/mod.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/predicate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/predicate.rs)

- The important architectural shift is that non-index access-choice explain for planner-built queries is no longer owned by `access_choice/mod.rs` shape projection. That module now reuses the already-frozen snapshot from the selected plan for non-index routes, while still projecting index-candidate ranking on the indexed path.
- `0.109.1` now closes one more non-index fallback seam on that same ownership line: explicit fluent key-access overrides (`by_id(...)`, `by_ids(...)`, and `only()`) no longer reach explain as bare `AccessPath` shapes with no stored reason.
- The override handoff in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_planner.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_planner.rs) now freezes the bounded planner-owned reason `intent_key_access_override` whenever access planning is short-circuited by explicit key intent. That keeps those routes on the same “planner stores the reason, explain consumes the reason” contract as the rest of the non-index cleanup.
- This is intentionally one generic reason, not separate override reasons for single-key vs many-key vs singleton-only access. That preserves the existing equality/reuse contract where `.by_id(key)` and `.by_ids([key])` still produce one identical planned query shape.
- The same cleanup also makes predicate-driven primary-key lookup and range routes more honest on the outward explain surface:
  - direct predicate-owned primary-key equality now carries `planner_primary_key_lookup`
  - direct predicate-owned primary-key ranges now carry `planner_primary_key_range`
  instead of falling back to the older access-shape labels `by_key_access` and `primary_key_range_access`
- The distinction matters architecturally:
  - `intent_key_access_override` means the route came from fluent key-access intent before normal predicate planning
  - `planner_primary_key_lookup` / `planner_primary_key_range` mean the normal planner derived the same primary-key family from the canonical predicate

## 0.109.0

- `0.109.0` starts the semantic-aware plan-reuse line by making the canonical identity work from `0.108` operationally visible instead of only hash-visible.
- The shipped artifact class in this first cut is explicit and bounded: shared prepared query-plan reuse. This patch does not add fuzzy matching, partial-plan reuse, new canonicalization rules, or any broader optimizer behavior.
- The outward reuse model is now visible through three surfaces:
  - `trace_query(...)`
  - fluent/session verbose execution explain
  - SQL `EXPLAIN EXECUTION VERBOSE`
- The core trace contract landed in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/trace.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/trace.rs). `TraceReuseArtifactClass` and `TraceReuseEvent` now expose one real shipped reuse class:
  - `SharedPreparedQueryPlan`
- The session-owned reuse path is wired in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs). That file now:
  - reports cache attribution as an explicit trace reuse event
  - emits `diag.s.semantic_reuse_artifact=shared_prepared_query_plan`
  - emits `diag.s.semantic_reuse=hit|miss`
  - keeps this state session-owned instead of pretending cache reuse is planner-owned route state
- SQL verbose explain now projects the same reuse diagnostics from the same cached plan path in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/explain.rs). Plain `EXPLAIN EXECUTION` remains unchanged; the extra `diag.s.*` lines only appear on the verbose path.
- The query-owned verbose renderer gained one bounded extension point in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs), so caller-owned diagnostics can be appended without promoting session cache state into planner-owned explain fields.
