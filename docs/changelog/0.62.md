# 0.62 Changelog

## 0.62.10 - 2026-03-23 - Field Type Metadata Cleanup And Collection Ergonomics

### Summary

`0.62.10` continues the per-entity emitted-output line from `0.62.9`, but
focuses on one specific source of repeated generated surface: schema-type
metadata and collection wrapper ergonomics.

Before this patch, generated schema-facing types still carried public inherent
`KIND` and `STORAGE_DECODE` constants even though those constants existed
primarily to feed generated runtime model assembly. That left two awkward facts
in the tree:

- per-type metadata was still exposed as generated public surface instead of a
  single trait-owned authority
- collection wrappers had briefly fallen back to raw `.0` access after helper
  deletion, which was too harsh for real gameplay/data-model code

This patch resolves both sides coherently:

- field kind and storage-decode metadata now live on one shared trait boundary
- list/set/map wrappers keep full container ergonomics through `Deref` and
  `DerefMut` instead of through handwritten generated helper methods

So the emitted surface is smaller and more regular, but the game-design-facing
collection API stays usable.

from generator organization to authority cleanup.

## 0.62.8 - 2026-03-23 - Per-Entity Generator Consistency Sweep

### Summary

`0.62.8` stays on the same per-entity line opened in `0.62.7`, but pushes the
cleanup substantially further across both derive and build generation.

This patch does not change the structural runtime architecture from
`0.62.1` through `0.62.7`. Rows remain slot-based, commit decoding remains
structural, and the cleaned `structural_field`, projection, and scalar/runtime
boundaries remain intact. It also does not reopen the rejected trait-surface
cuts around `PRIMARY_KEY`, `INDEXES`, or SQL descriptor shape.

Instead, `0.62.8` treats generator consistency as the main objective. Several
derive/build owners were still technically correct but unevenly organized:

- default/sanitize/validate/visitable generation still repeated similar

  strategy wiring

  store wiring and SQL dispatch

`0.62.8` keeps those responsibilities in their existing owners, but makes the
owners themselves more regular, so future per-entity reduction work can happen
from a cleaner baseline.

## 0.62.6 - 2026-03-22 - Recursive Boundary Peel And Local Decoder Tightening

`0.62.1` through `0.62.5`: rows remain slot-based, commit decoding stays
structural, and the remaining explicit decoder surface stays split by owner
module.

The point of this patch is narrower. After the ownership cleanup in `0.62.4`
and `0.62.5`, the next hotspot was still the mixed root
`decode_structural_field_by_kind_bytes(...)`. It was simultaneously:

- the scalar fast-path gate
- the non-recursive leaf dispatcher
- the recursive composite re-entry point

`0.62.6` peels those roles apart without introducing a new abstraction family.
The structural-field root is now a thin lane router, and the next measured
decoder reductions stay local to the owning modules that remain on the hot
path.

## 0.62.1 - 2026-03-21 - Slot-Format Hard Cut And Scalar Slot Readers

### Summary

`0.62.1` is the representation-phase-change follow-up to `0.62.0`.

The opening patch proved that structural persisted-row decode could remove a
meaningful amount of per-entity runtime cost, but it also clarified the next
constraint: keeping multiple persisted-row generations alive and continuing to
route important runtime paths through whole-row or generic-value decode would
flatten the wasm win quickly.

This patch therefore makes the hard cut that earlier compatibility work had
been deferring:

- legacy persisted-row formats are gone instead of being carried in parallel
- the slot-based row container is now the only live persisted-row shape
- commit, recovery, and index planning use slot readers as the canonical row
  authority
- common scalar leaf payloads stop riding through CBOR and use one dedicated
  binary leaf codec family instead
- runtime field-projection seams no longer fall back to `SlotReader::get_value`

The important architectural result is that persisted rows now have one live
container contract and one explicit split:

- row container: presence, slot order, field spans, version envelope
- leaf codec: scalar binary codec first, CBOR fallback only where still needed
