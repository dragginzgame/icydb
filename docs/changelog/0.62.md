# 0.62 Changelog

## 0.62.6 - 2026-03-22 - Recursive Boundary Peel And Local Decoder Tightening

### Summary

`0.62.6` keeps the post-serde structural decoder on the same architecture from
`0.62.1` through `0.62.5`: rows remain slot-based, commit decoding stays
structural, and the remaining explicit decoder surface stays split by owner
module.

The point of this patch is narrower. After the ownership cleanup in `0.62.4`
and `0.62.5`, the next hotspot was still the mixed root
`decode_structural_field_by_kind_bytes(...)`. It was simultaneously:

- the scalar fast-path gate
- the non-recursive leaf dispatcher
- the recursive composite re-entry point

`0.62.6` peels those roles apart without introducing a new abstraction family.
The structural-field root is now a thin lane router, and the next measured
decoder reductions stay local to the owning modules that remain on the hot
path.

### Changed

- peeled recursive composite `ByKind` decode into
  [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/composite.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/composite.rs):
  - list/set decode
  - map decode
  - schema-enum payload decode
  - relation key-kind recursive re-entry
- kept non-recursive typed leaves in
  [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/leaf.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/leaf.rs)
  and reduced the root
  [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/mod.rs)
  to one explicit routing contract:
  - scalar fast path first
  - leaf lane second
  - composite lane last
- established the post-split invariant explicitly in code:
  - `mod.rs` chooses the lane
  - leaf decode does not recurse
  - composite decode owns recursive re-entry
- tightened `leaf.rs` locally by collapsing repeated top-level payload framing
  checks for typed wrapper leaves, without touching recursion or the
  `FieldStorageDecode::Value` lane
- tightened
  [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs)
  locally by replacing the repeated `variant` / `path` / `payload` field-name
  comparisons in `decode_value_enum_payload_bytes(...)` with one direct
  encoded-field-name matcher
- rejected and reverted the non-paying follow-up cuts that did not improve
  total wasm:
  - generic field-framing helper extraction in `value_storage.rs`
  - final `ValueEnum` tail-assembly helper extraction in `value_storage.rs`
  - scalar-family helper dedup in `scalar.rs`

### Validation

- `cargo check -p icydb-core -p icydb`
- `cargo test -p icydb-core db::data::structural_field::tests -- --nocapture`
- `cargo build -p canister_one_simple --target wasm32-unknown-unknown`

### Measurement

The kept `0.62.6` stack leaves the debug audit canister at:

- `target/wasm32-unknown-unknown/debug/canister_one_simple.wasm`: `3,516,567`

The structural-field hotspot shape after the kept cuts is:

- `leaf::decode_leaf_field_by_kind_bytes`: `3818`

The important architectural result is that the old mixed root no longer shows
up as one oversized shallow owner. The remaining pressure is now expressed as
separate scalar, leaf, and `Value`-storage owners, which makes subsequent
tightening work measurable per boundary instead of per historical file.

### Next Step

After `0.62.6`, the right follow-up remains narrow and owner-local:

- keep `structural_field/mod.rs` as a lane router rather than re-growing it
  into a mixed decode hub
- only take further `scalar.rs`, `leaf.rs`, or `value_storage.rs` cuts if
  they reduce total wasm, not just local symbol size
- avoid reintroducing generic helpers or reusable decoder frameworks unless a
  later measurement shows a real whole-binary win

## 0.62.5 - 2026-03-22 - Decoder And Projection Ownership Cleanup

### Summary

`0.62.5` is not another representation change and it does not reopen the
serde-removal work. The slot row format, structural commit format, and
explicit decoder/runtime split from `0.62.1` through `0.62.4` remain the same.

This patch is an organization checkpoint on top of that architecture. The
remaining explicit runtime surfaces were technically correct, but two of them
were still drifting toward “large local owner” status:

- the structural field decoder
- projection materialization and projection expression evaluation

`0.62.5` keeps those surfaces explicit but turns them into smaller owner
modules, so future tightening work can happen inside stable internal
boundaries instead of growing more mixed-purpose files.

### Changed

- finalized the `structural_field` directory-module split in
  [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/):
  - `mod.rs` is now primarily the public boundary and top-level by-kind
    dispatch
  - `cbor.rs` owns bounded CBOR parsing and raw walkers
  - `scalar.rs` owns scalar fast-path decode
  - `kind.rs` owns composite `ByKind` list/map/enum decode

  - `ValueVariantTag` and `Value`-storage decode state now live in

    [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/materialize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/materialize.rs)

## 0.62.2 - 2026-03-21 - Structural Runtime Serde Removal And Leaf Tightening

### Summary

`0.62.2` closes the architecture phase opened by `0.62.0` and hard-cut by
`0.62.1`.

After the slot row format, scalar slot codecs, and slot-reader planning work
landed, the remaining size pressure was no longer “generic row decode.” It was
the residual runtime decode and execution surfaces that still retained
`Value`-centric or `serde_cbor`-centric machinery:

- commit-marker payload decode still used one generic serde entrypoint
- scalar predicates, index expressions, and projections still retained generic
  `Value` fallback machinery in important execution paths
- `structural_field.rs` still owned the remaining explicit complex-leaf decode
  surface, with repeated enum/container parsing logic keeping more code alive
  than necessary

This patch removes the last runtime `serde_cbor::from_slice` path, pushes
predicate/index/projection execution onto shared scalar programs where the
query shape allows it, and then tightens the remaining structural field decode
surface with measurement-driven cuts only.

The resulting runtime split is now explicit:

- persisted rows: structural slot container only
- commit frames and payloads: structural binary decode only
- scalar execution: slot/scalar programs, no generic `Value` path
- complex leaf decode: explicit structural field decode only

## 0.62.1 - 2026-03-21 - Slot-Format Hard Cut And Scalar Slot Readers

### Summary

`0.62.1` is the representation-phase-change follow-up to `0.62.0`.

The opening patch proved that structural persisted-row decode could remove a
meaningful amount of per-entity runtime cost, but it also clarified the next
constraint: keeping multiple persisted-row generations alive and continuing to
route important runtime paths through whole-row or generic-value decode would
flatten the wasm win quickly.

This patch therefore makes the hard cut that earlier compatibility work had
been deferring:

- legacy persisted-row formats are gone instead of being carried in parallel
- the slot-based row container is now the only live persisted-row shape
- commit, recovery, and index planning use slot readers as the canonical row
  authority
- common scalar leaf payloads stop riding through CBOR and use one dedicated
  binary leaf codec family instead
- runtime field-projection seams no longer fall back to `SlotReader::get_value`

The important architectural result is that persisted rows now have one live
container contract and one explicit split:

- row container: presence, slot order, field spans, version envelope
- leaf codec: scalar binary codec first, CBOR fallback only where still needed
