# 0.125.x

## 0.125.1

`0.125.1` follows the initial planner `Expr` traversal cut by moving the remaining obvious compiled-expression and lowered-expression structural scans onto their tree owners instead of keeping more adjacent local recursive walkers.

Follow-through owner moves landed in:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/scalar.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/scalar.rs)
  `ScalarProjectionExpr` now owns compiled slot traversal through `for_each_referenced_slot(...)`, `extend_referenced_slots(...)`, and `mark_referenced_slots(...)`, and the older free `extend_scalar_projection_referenced_slots(...)` helper is gone
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_plan.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_plan.rs)
  `ResolvedOrderValueSource` now owns `direct_field_slot()`, while `ResolvedOrder` now owns `referenced_slots()` and `direct_field_slots()` so planner and executor callers no longer split those frozen order-structure queries between separate modules
- [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs)
  retained-slot preparation now marks compiled scalar filter slots through the compiled expression owner instead of keeping an execution-local slot walk over compiled scalar shape
- [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/grouped/stages.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/grouped/stages.rs)
  grouped row-slot preparation now uses the same owner-local compiled scalar traversal for residual filters and compiled aggregate input/filter expressions
- [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/analysis.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/analysis.rs)
  lowered planner-expression analysis no longer redefines recursive descent over `Expr`; it now reads `contains_aggregate()` and `try_for_each_tree_expr(...)` from the planner expression owner
- [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/aggregate.rs)
  the lowering-local aggregate-leaf walker is gone, and global post-aggregate collection now uses `Expr::try_for_each_tree_aggregate(...)` directly
- [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs)
  grouped and global `HAVING` aggregate registration now consumes that same planner-owned aggregate traversal contract instead of a second lowering helper seam
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs)
  grouped projection aggregate scanning no longer keeps its own recursive aggregate walk; it now routes through `contains_aggregate()` and `try_for_each_tree_aggregate(...)`, and the dead `combine(...)` helper fell out afterward
- [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/model.rs)
  `SqlExpr` now owns `for_each_tree_expr(...)` and `for_each_tree_aggregate(...)`, so SQL aggregate-call collection can consume the parser-owned traversal contract instead of reopening one more parser-child recursion in lowering
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/ast.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/ast.rs)
  planner `Expr` now also owns `contains_case()`, so grouped omitted-`ELSE` `CASE` admission no longer keeps a separate raw-`Case` scanner in SQL lowering
- [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs)
  grouped and global omitted-`ELSE` `CASE` admission now reads `canonical.contains_case()` from the planner expression owner instead of keeping a lowering-local recursive helper for that structural question
- [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/mod.rs)
  grouped `HAVING` aggregate collection now calls the parser-owned aggregate collector directly, so the old `extend_grouped_having_aggregate_calls(...)` forwarding helper disappeared instead of surviving as one more fake lowering owner

This leaves `0.125` in a materially cleaner state:
- planner `Expr` owns planner-tree traversal
- parser `SqlExpr` owns parser aggregate-leaf traversal too
- compiled `ScalarProjectionExpr` owns compiled slot traversal
- resolved order shapes own their direct-field and referenced-slot structural queries
- SQL lowering no longer keeps parallel lowered-`Expr` and aggregate-leaf walkers for the same planner tree shape

## 0.125.0

`0.125.0` starts the expression-traversal ownership line by moving the first planner-side tree-query contract onto `db::query::plan::expr::Expr` itself instead of leaving nearby consumers to reopen recursive walks locally.

Planner-owned traversal surface added in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/ast.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/ast.rs):
- `contains_aggregate()`
- `any_tree_expr(...)`
- `all_tree_expr(...)`
- `try_for_each_tree_expr(...)`
- `try_for_each_tree_aggregate(...)`
- `try_for_each_tree_expr_with_compare_index(...)`
- `references_only_fields(...)`

The first consumers moved onto that owner-local contract were:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/projection.rs)
  projection slot marking now walks one owner-local expression traversal instead of keeping a projection-local recursive slot scanner, and the old free `expr_references_only_fields(...)` helper is gone
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs)
  grouped `HAVING` aggregate collection now uses `try_for_each_tree_aggregate(...)`, while resolved-order field validation now uses `try_for_each_tree_expr(...)`
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_having.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_having.rs)
  grouped `HAVING` streaming-compatibility checks now read `all_tree_expr(...)` instead of maintaining another manual recursive admission walk
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/grouped/structure/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/grouped/structure/mod.rs)
  grouped `HAVING` structural symbol validation now walks the expression owner through `try_for_each_tree_expr_with_compare_index(...)` instead of carrying its own recursive compare-index counter and child traversal
- [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/grouped/policy/rules.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/grouped/policy/rules.rs)
  grouped `HAVING` compare-policy validation now uses that same owner-indexed traversal contract instead of a second local recursive walk that reimplemented compare-order numbering
- callers that only needed “does this expression stay within these fields?” now call `expr.references_only_fields(...)` directly:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/grouped/projection_expr/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/grouped/projection_expr/mod.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/aggregate.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/projection.rs)

The main follow-up left after `0.125.0` is no longer planner `Expr` itself. The next likely seam is compiled scalar-expression traversal ownership, where execution-setup code still needs the same kind of owner-local slot/reference walk contract instead of reopening compiled expression shape directly.
