# 0.43 Changelog

## 0.43.4 - 2026-03-07 - DTO Visibility and Accessor Boundary Sweep

### Summary

`0.43.4` continues the `0.43` boundary cleanup by reducing direct field
coupling across internal diagnostics/explain/trace DTOs while keeping EXPLAIN
payloads immutable after construction.

### Changed

- Narrowed internal DTO field visibility and standardized accessor-based reads
  across query/execute/diagnostics call sites and matrix tests.
- Removed direct field reads for `ExplainPlan`, execution descriptor nodes,
  query trace payloads, and storage-report snapshots in favor of getter APIs.
- Kept EXPLAIN DTO construction immutable and constructor-oriented; no
  setter/builder mutation paths were introduced.
- Fixed pagination test call sites that borrowed `explain.access()` by
  reference (`&explain.access()`) after the getter migration.

### Validation

- `cargo fmt --all -- --check`
- `cargo check -p icydb-core`
- `cargo check -p icydb`
- `cargo test -p icydb-core --no-run`

## 0.43.3 - 2026-03-06 - Access/Continuation Surface Cleanup

### Summary

`0.43.3` removes remaining dead or duplicative runtime surfaces from the
`0.43` audit track while preserving query semantics and continuation behavior.

### Changed

- Consolidated `AccessPathKind` ownership under `db::access::dispatch` by
  removing the duplicate enum definition from `db::access::capabilities`.
- Removed always-true access capability gates and related constant plumbing
  (`supports_strict_resume`, ordered-stream capability pass-through fields).
- Removed dead `ExecutablePlan` shape-signature storage
  (`execution_shape_signature`) and the associated accessor; continuation
  compatibility now remains anchored in the immutable continuation contract.
- Removed a pure pass-through constructor layer in access stream descriptor
  construction (`AccessExecutionDescriptor::from_strategy`) and inlined
  executable conversion at call sites.

### Validation

- `cargo check -p icydb-core` passes after each cleanup slice.
- Boundary guard remains green:
  `db::access::tests::runtime_raw_access_path_references_stay_within_access_boundary`.
- Cursor-signature regression coverage remains green with the updated
  stability/sensitivity test path under `executor/tests/cursor_validation.rs`.

## 0.43.2 - 2026-03-06 - Access Capability Surface Consolidation

### Summary

`0.43.2` consolidates executable access capability evaluation under one
access-owned runtime surface to reduce fan-out and branch duplication outside
`db::access`.

### Changed

- `ExecutableAccessPath` now exposes one canonical capability projection method
  (`capabilities()`), and runtime load/route/stream/preparation paths now read
  capability data through that method instead of free helper calls.
- `ExecutableAccessPlan` now exposes canonical plan capability/metrics helpers
  (`capabilities()` and `metrics_kind()`), and route/metrics code now consumes
  those methods directly.
- `AccessRouteClass` projection now derives from the shared access capability
  snapshot, removing duplicated route-shape predicates from
  `execution_contract`.
- Access capability derivation helpers are now internal implementation details
  inside `db::access::capabilities`, reducing export surface and preventing
  re-fanout in executor modules.
- Fixed one boundary-guard regression in `query::intent` by removing a raw
  `AccessPath::` method-pointer reference so runtime raw variant references stay
  access-boundary local.

### Validation

- Added/kept passing boundary guard coverage:
  `db::access::tests::runtime_raw_access_path_references_stay_within_access_boundary`.
- Full `cargo test -p icydb-core` and architecture invariant scripts remain
  green after the consolidation.

## 0.43.1 - 2026-03-06 - BYTES(field) and bytes_by(...)

### Summary

`0.43.1` extends byte-size observability from whole-row totals to field-level
totals via scalar `BYTES(field)` semantics in the fluent API.

```rust
let payload_bytes = session
    .load::<Event>()
    .where_(f("country").eq("US"))?
    .bytes_by("payload")?;
```

### Added

- Added `FluentLoadQuery::bytes_by(field)` as the field-target byte-size
  terminal in the scalar load-query surface.
- Added `LoadExecutor::bytes_by_slot(plan, target_field)` to execute the
  canonical scalar plan and aggregate encoded byte length for the selected
  field across the effective result window.

### Changed

- Added serialized-length helpers that compute encoded byte length without
  allocating output buffers (`serialize::serialized_len` and CBOR counting
  writer support), preserving exact wire-size semantics.
- `bytes_by(...)` resolves field slot ownership through planner metadata and
  fails early on unknown fields before scan-window execution.
- Accumulation remains saturating `u64`, matching existing `bytes()` overflow
  behavior and empty-window semantics.

### Validation

- Added session matrix coverage for `bytes_by("rank")`, `bytes_by("tags")`,
  empty-window zero behavior, and unknown-field failure classification.
- Added access-path parity coverage to lock equivalent results between
  index-prefix and full-scan routes for `bytes_by(...)`.

## 0.43.0 - 2026-03-06 - BYTES() Terminal and Audit Kickoff

### Summary

`0.43.0` introduces a scalar load terminal `bytes()` that reports total
persisted payload bytes for the effective scalar result window.

```rust
let total_bytes = session
    .load::<Event>()
    .where_(f("kind").eq("click"))?
    .bytes()?;
```

### Added

- Added `FluentLoadQuery::bytes()` to the core fluent load terminal surface.
- Added facade session wrapper support (`icydb` crate) for the same terminal.

### Changed

- Added `LoadExecutor::bytes(plan)` under load terminal execution ownership.
- `bytes()` now executes the canonical scalar query plan and sums persisted row
  payload lengths for the resulting ids.
- Accumulation uses saturating `u64` arithmetic, and empty windows return `0`.
- Expanded aggregate parity matrices so `bytes()` is covered across
  full-scan/index-prefix parity, ordered page windows, and DISTINCT windows.
- Added terminal-local saturation unit coverage to lock overflow behavior.

### Documentation

- Aligned `0.43` design/status docs with current module ownership and scope.
- Started the `0.43` crosscutting audit cycle with a dated report set under
  `docs/audits/reports/2026-03/2026-03-06/` (complexity, DRY, layer-violation,
  module-structure, velocity, and summary).
