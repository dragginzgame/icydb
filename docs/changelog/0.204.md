# IcyDB 0.204 Changelog

## 0.204.4 - 2026-07-17

### 📝 Summary

- Corrects the scale-evidence cardinality contract for global aggregates:
  matched input rows remain workload evidence, while the query result contains
  exactly one aggregate row.
- Production SQL behavior is unchanged. This patch repairs evidence validation
  exposed by the first post-0.204.3 calibration attempt.

### 🩹 Fixed

- Global-aggregate scale observations now validate one returned row instead of
  comparing the result row count with predicate selectivity. Predicate matches
  and measured aggregate ingestion remain independently retained.
- Corrects the synthetic scale-report helper that encoded the same mistaken
  cardinality, so local report tests no longer preserve paper truth that live
  canister execution rejects.

### 🧪 Testing

- Focused scale-profile and shard coverage passes for all eight synthetic
  shards, including positive and rejection cases that separate four matched
  rows from one aggregate output row.
- Warning-denied Clippy, formatting, and diff validation pass. Full suites and
  the live scheduled rerun remain user-owned validation after publication.

### 📊 Audit

- The release diff touches four files with 83 insertions and 4 deletions,
  mostly regression coverage and release notes. Production execution complexity
  and WASM-producing source remain unchanged.
- The failed 0.204.3 run completed every P1 shard and four non-aggregate scale
  shards before four aggregate-bearing scale shards rejected the incorrect
  evidence expectation. It is diagnostic input, not calibration evidence.

### 📚 Status

- After publication, begin a fresh three-run calibration cohort and dispatch
  ordinals sequentially. The failed 0.204.3 cohort cannot contribute to the
  initial baseline or budget calibration.

### 📊 Audit

### 📚 Status

## 0.204.2 - 2026-07-17

### 📝 Summary

- Makes comparable performance confirmation fail closed from P1 discovery
  through the final P2 verdict instead of allowing missing baseline provenance
  to silently omit regression candidates.
- Adds an explicit first-baseline calibration lane. Each clean run records one
  canonical cohort and exact ordinal `1`, `2`, or `3`, remains identifiable in
  merged P2 evidence, and cannot satisfy the ordinary comparison verdict.
- This release prepares evidence collection; it does not claim clean Tier D
  measurements, calibrated budgets, or 0.204 closeout.

### ➕ Added

- Adds typed comparable-baseline and initial-calibration basis to deterministic
  P2 selection and merged reports, including the exact reviewed baseline
  environment and retained threshold-crossing count.
- Adds manual workflow inputs for a reviewed baseline run or one explicit
  calibration cohort member. Current Tier D bundles are retained for 90 days so
  all three clean calibration runs can be reviewed together.
- Adds typed clean-source rejection and strict calibration cohort/run parsing.
  Malformed, incomplete, conflicting, dirty, incomparable, or tampered evidence
  fails before it can become a P2 selection or verdict.

### 🔧 Changed

- Normal P1 merge now requires one complete reviewed P1 report, validates exact
  scenario and semantic identity, and derives typed total-instruction threshold
  crossings for P2 confirmation.
- Baseline threshold reasons now carry `P2RawMetric` rather than string metric
  names, and selection validation reconstructs their exact count from retained
  reasons.
- The scheduled performance workflow resolves an explicit reviewed workflow-run
  artifact, downloads its P1 and P2 evidence, and runs confirmed comparison as a
  separate fail-closed gate. Calibration dispatches retain evidence but skip the
  inapplicable ordinary verdict.
- Merged P2 validation remains the sole clean-subject authority. Comparison
  preserves that typed cause and verifies that the current selection names the
  exact baseline report being compared.

### 🧪 Testing

- Focused environment, P1 discovery, P2 selection, shard merge, comparison, and
  command-boundary coverage passes, including malformed calibration identity,
  threshold-count drift, and calibration-as-verdict rejection.
- Focused warning-denied Clippy, formatting, workflow lint, and diff validation
  pass. The full push suite and clean scheduled runs remain user-owned release
  validation.

### 📊 Audit

- The release diff touches eleven evidence, workflow, and documentation files
  with about 1,281 insertions and 75 deletions. The evidence boundary is more
  explicit and therefore larger; no production query or canister runtime path
  changed.
- No Tier D instruction or raw non-gzipped WASM delta was measured. Those values
  require the three clean scheduled calibration runs from the published source
  revision.

### 📚 Status

- After publication, run the same clean revision three times with one cohort and
  ordinals `1`, `2`, and `3`; review their stability, timing, instructions, raw
  WASM, residuals, slopes, and recurring hotspots before selecting the initial
  baseline or checking in calibrated budgets.
- A normal baseline-gated run, remaining scheduled correctness evidence, and the
  final 0.204 closeout audit still follow that review.

## 0.204.1 - 2026-07-17

### 📝 Summary

- Advances the 0.204 evidence line through the completed SQLite, typed SELECT,
  aggregate, execution-mode, and mutation-state slices, and lands the bounded
  machinery needed for scheduled correctness and performance evidence.
- Fixes four production discrepancies found by the new evidence: computed-null
  ordering, descending primary-key mutation routing, atomic duplicate INSERT
  classification, and filtered global `COUNT(*)` residual execution.
- Continues the pre-1.0 hard cut by removing relation modes and query/executor
  scaffolding that no longer represented real variability. This is an
  intermediate release boundary, not 0.204 closeout.

### ➕ Added

- Adds a normative write-admission contract for typed, structural, SQL, batch,
  and schema writes. Recovery remains replay, and accepted-schema validation has
  no trusted bypass.
- Adds a lockfile-pinned bundled SQLite profile and runs the same typed
  differential contract against native IcyDB and the live canister SQL boundary.
- Adds accepted-snapshot-aware typed generation for scalar, `DISTINCT`, global
  aggregate, grouped, and `HAVING` reads, with bounded deterministic replay,
  shrinking, invalid-case ownership, and cold/warm route evidence.
- Adds accepted-snapshot-aware INSERT, INSERT FROM QUERY, UPDATE, and DELETE
  sequences with an independent state model, per-step native and eligible
  SQLite comparison, atomic rejection checks, deterministic replay, and bounded
  shrinking.
- Adds one fixed eight-root Tier C profile, one shard mapping, a strict bounded
  regression-corpus format, content-addressed minimized-failure envelopes,
  exact eight-receipt merge, and typed coverage-distribution evidence. The
  checked-in corpus pins the minimized filtered-global-count regression.
- Adds a strict 1,758-case P1 performance profile, deterministic eight-shard P1
  and P2 artifacts, five isolated cold and five cache-proven warm P2 samples,
  and a 45-case 16/256/2,048-row scale profile with normalized costs and
  adjacent-cardinality slopes.
- Adds complete performance-environment identity, confirmed-median and scale
  baseline formats, attributed-versus-total-only calibration evidence, explicit
  `NotMeasured` resource dimensions, and a nightly fixed-shard workflow. CI
  cannot bless baselines.

### ⚠️ Breaking

- Relations now have one always-enforced contract. Strength and enforcement
  options are removed from the schema DSL, generated model, accepted schema,
  persisted codecs, and describe output. Use an ordinary key-typed field when a
  dangling identifier is intentional, and recreate accepted schema snapshots
  for the current codec.
- Removes the public one-variant `TraceReuseArtifactClass`;
  `TraceReuseEvent::{Hit, Miss}` is the trace contract.
- Removes the public one-variant `UnsupportedQueryFeature`; map predicates now
  fail directly with `ValidateError::MapPredicateUnsupported { field }`.
- Removes the public one-variant `StoreLiveValidationCapability`; every
  registered store supports live validation.

### 🩹 Fixed

- Canonicalizes stored and computed nulls before order comparison and cursor
  boundary construction so ascending null order is consistent.
- Preserves descending primary-key direction through mutation planning and
  DELETE continuation.
- Preserves the executor-owned typed conflict for atomic same-batch duplicate
  INSERT rejection.
- Applies scalar residual filters while raw rows are open, preventing filtered
  global `COUNT(*)` from discarding every row on the data-row-only route.
- Keeps default-feature schema-mutation metrics and SQL-only publication exports
  within their actual feature ownership.
- Aligns recovery and reconciliation fixtures with accepted-schema authority so
  generated metadata cannot roll back a newer accepted snapshot.
- Declares the six real integration-test roots explicitly so SQL performance
  support modules are not also compiled as broken standalone test crates.
- Restores the host-side storage-write performance envelope to the canister's
  canonical averaged-metric field names so Candid decoding cannot drift.
- Aligns atomic duplicate-key assertions with the executor-owned typed conflict.
- Resets the sole current schema, accepted-enum, generator, replay, calibration,
  and allocation-fingerprint method identities to version 1. Role-specific hash
  domains remain distinct, schema codec-version rejection preserves its typed
  incompatible-format cause, and no retired decoder or compatibility path remains.
- Missing resolved DELETE ordering now fails closed, and selected-row caps are
  checked before packaging and commit.

### 🗑️ Removed

- Removes the optional system-CLI comparator, the template-random SELECT path,
  the isolated random mutation comparator, their duplicate verdict/report
  models, compatibility environment controls, and audit-only mutation endpoints.
- Removes the superseded 94-row SQL catalog, blessed JSON baseline and writer,
  first-`N` and other partial P1 controls, duplicate five-repeat sampler, and
  overlapping cache-contract evidence.
- Collapses the generic post-access pipeline into its two real owners:
  scan-time load filtering and DELETE-owned filter/order/window execution. Dead
  plan, stage, stats, tuple-row, and page-terminal artifacts are deleted.
- Removes transport-only wrappers around predicate compilation, aggregate
  semantics, cache order and predicate identity, projection rows, DELETE bounds,
  load routing, EXISTS direction, row decoding, journal traversal, and data-key
  errors. Their canonical planner, executor, cache, persistence, or typed-error
  values now flow directly.
- Removes paper variability from scalar-function determinism, route-feasibility
  rules, index-prefix liveness, branch-set ordered suffixes, trace modes, storage
  capabilities, and DELETE policy state.
- Updates the index-range architecture check to protect the branch-set route's
  sole primary-key-ascending constructor and its real cache/fingerprint inputs,
  instead of requiring the deleted one-variant ordered-suffix proof.
- Retargets the field-projection authority scan from the removed generic
  post-access bridge to the current shared ordering, DELETE, and page owners.
- Renames the EXPLAIN stage to the phase-neutral `residual_runtime`; the removed
  `residual_post_access` name no longer describes execution ownership.

### 🧭 Migration Notes

- Remove relation enforcement or strength arguments from schemas. Relations are
  enforced; unconstrained identifiers are plain key-typed fields.
- Recreate current accepted schema snapshots and regenerate code. No decoder,
  alias, or transitional schema model preserves the removed relation modes.
- Consume trace reuse events and direct typed validation errors instead of the
  removed one-variant public wrappers.

### 🧪 Testing

- New focused evidence covers the bundled-SQLite profile, generated valid and
  invalid reads, mutation state and rollback, replay and shrinking, regression
  corpus, exact shard receipts, P1/P2 selection, cold/warm proof, and scale
  normalization.
- Focused core compilation, warning-denied all-target Clippy with the maintained
  SQL and diagnostic features, and 84 targeted query, DELETE, cache, projection,
  route, and decode tests pass for the final architecture-cleanup slice.
- The user-owned full push suite, live-canister differential, and clean scheduled
  Tier C and Tier D artifacts remain release-workflow validation rather than
  agent-run validation.

### 📊 Audit

- Against `v0.204.0`, the present release-related tree spans approximately 309
  files with 38,100 insertions and 16,900 deletions, a net increase of about
  21,200 lines. This excludes three untracked future-design drafts that do not
  belong to 0.204.1.
- Most growth is explicit test evidence, typed scenario models, strict artifact
  codecs, and deterministic performance profiles. Production query structure is
  simpler: duplicate pipelines, paper stage contracts, one-variant wrappers, and
  parallel classifiers are removed without compatibility replacements.
- A dirty-worktree seam check measured a five-sample median instrumentation
  delta of 22,210 instructions (338 basis points) on a 3,923,308-byte raw
  non-gzipped wasm module. It proves the measurement seam only; it is not a
  clean scheduled baseline or a release performance claim.

### 📚 Status

- Slices 7 and 8 remain open for clean scheduled correctness/performance
  artifacts, a reviewed comparable-baseline bridge, three-run timing and
  threshold calibration, residual budgets, and focused-hotspot promotion
  evidence. Those items gate 0.204 closeout, not this intermediate patch.

## 0.204.0 - 2026-07-15

### 📝 Summary

- Establishes one machine-checked authority for the SQL contract and its
  maintained evidence. Stable feature identifiers now map to explicit evidence
  obligations, provider eligibility, and performance dispositions.
- Adds one shared typed correctness core for deterministic and generated SQL
  scenarios. Correctness and performance retain separate fail-closed verdicts
  while sharing declared scenario facts.
- This is the first 0.204 release boundary, covering design Slices 1 and 2. The
  wider generator, pinned SQLite, mutation-state-machine, performance, corpus,
  and closeout work remains in later slices.

### ➕ Added

- The active SQL contract carries 98 stable syntax, semantic, policy, and
  interaction identifiers. A checked manifest maps them to 93 deterministic
  evidence providers with minimum evidence strength and explicit external
  reference eligibility or exclusion.
- The shared test harness owns typed statement, result-shape, value,
  nullability, predicate, window, mutation, ordering, provider, route,
  acceptance, normalization, mismatch, and failure-ownership facts.
- Focused consistency checks fail closed on missing or duplicate contract
  cells, malformed metadata, unreachable providers, insufficient evidence,
  unused provider declarations, typed rejection drift, route drift, and
  normalization mismatches.

### 🔧 Changed

- Bounded performance-matrix selection is deterministic and stratified across
  ten declared scenario dimensions instead of taking an order-sensitive prefix
  of the matrix.
- Deterministic and generated scenarios declare their route and provider facts
  at construction time. Rendered SQL remains an execution payload and cannot
  become classification authority.
- Performance reports project the shared correctness verdict and retain typed
  failure ownership while keeping performance measurement status independent.
- The active contract now records the already-maintained grouped
  pre-aggregation comparison and shared text-expression predicate behavior.
  This corrects evidence authority without widening the SQL runtime surface.

### 🗑️ Removed

- Removes SQL-text route and provider classifiers, prefix-based default
  selection, reconstructed route fallback for old reports, and the raw SELECT
  signature verdict. No compatibility path preserves those retired test
  authorities.
- Removes current evidence for the retired `SHOW TABLES` spelling. The older
  proof matrix is explicitly historical and no longer competes with the active
  contract and manifest.

### 🧪 Testing

- Focused contract/manifest checks cover the exact feature bijection and
  deterministic provider reachability. Shared harness checks cover typed route
  classification, row-order-aware normalization, duplicate preservation,
  exact typed rejection causes, fail-closed verdicts, and order-independent
  stratified selection.
- The focused correctness target passes all 11 checks. The performance target
  passes 29 local checks with seven live or external comparison cases remaining
  intentionally manual, and both targets are Clippy-clean with warnings denied
  against the release base.
- The user-owned full push suite remains the final release gate.

### 📊 Audit

- Release preparation spans approximately 19 SQL evidence, test, contract,
  design, and changelog files with about 6,400 insertions and 1,300 deletions,
  a net increase of roughly 5,100 lines. Most growth is the explicit coverage
  manifest and typed scenario metadata rather than executable product logic.
- Production complexity remains unchanged. Test-side structure is more explicit
  but has fewer competing authorities: one manifest, one typed scenario model,
  one normalization contract, one correctness verdict, and one deterministic
  selector replace inference and fallback paths.
- No production runtime, persisted format, public API, or generated canister
  behavior changes. Performance and raw non-gzipped wasm deltas are therefore
  not applicable to this test-and-documentation release boundary.

### 📚 Documentation

- Records the Slice 1 and Slice 2 implementation state in the active 0.204
  design while leaving the line explicitly open for Slices 3 through 8.
- Marks the archived 0.166 SQL proof matrix as historical and keeps
  `SQL_SUBSET.md` as product-behavior authority rather than allowing tests or a
  reference engine to define IcyDB semantics.
