# 0.84 — Prepared Execution Residents and Grouped Runtime Attribution

## 0.84.2 - 2026-04-16 - Query Planner Boundary Cleanup

This patch is a maintainability follow-through after the `0.84.0` prepared-plan
resident cuts and the `0.84.1` grouped runtime attribution work. Those patches
made the active compile/prepare and grouped runtime owners easier to see, which
also exposed how much duplicated helper scaffolding still lived around the
planner, validation, expression, and fingerprint boundaries.

It does four concrete things:

* trims repeated query-intent helper scaffolding around plan build, grouped
  mutation guards, paging errors, and intent policy mapping
* removes mirrored planner helper logic across compare/predicate/prefix/range
  access planning and order fallback selection
* narrows planner expression and grouped projection helpers so the projection
  boundary owns the current single projection-field unwrap instead of having it
  reimplemented across adjacent modules
* folds several small validation and fingerprint consumers back onto those
  shared query/planner owners instead of leaving local wrapper duplication in
  place

What changed:

* query-intent planning now owns its access-plan and semantic-validation split
  behind dedicated helpers in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs`
  * `plan_access_from_normalized_predicate(...)`
  * `validate_plan_semantics(...)`
* grouped intent mutation now routes grouped field/aggregate/limit writes
  through one shared grouped-target helper in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/mutation.rs`
  * `grouped_mutation_target(...)`
* intent paging errors and key-access policy mapping are now more local to the
  intent boundary in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/errors/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/policy.rs`
* planner compare/predicate/prefix/range now share more of their local gate and
  traversal logic in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/compare.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/predicate.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/prefix.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/range/extract.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/planner/index_select.rs`
* planner-adjacent order helpers now use smaller shared owners in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/order_term.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/order.rs`
* planner expression and grouped projection consumers now reuse one shared
  projection-field expression helper in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/projection.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/scalar.rs`

  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs`
* the same projection-boundary helper now also feeds the adjacent semantic hash
  surface in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/projection_hash.rs`

Why this matters:

* the earlier `0.84` patches moved more active runtime work onto explicit
  prepared-plan and grouped-runtime owners
* once those hot owners were clearer, the remaining cold-path duplication around
  query planning became much easier to see
* that duplication was not changing query behavior, but it was still making
  planner and validation changes noisier than they needed to be
* this patch keeps the same query behavior and the same feature surface, but it
  leaves fewer parallel helper paths behind in the planner-adjacent code
* that should make the next behavior-changing planning or projection slice
  cheaper to reason about, because more of the shared planner rules now live in
  one owner-local place

## 0.84.1 - 2026-04-16 - Grouped Runtime Phase Attribution

This patch starts the next `0.84` follow-up line after the prepared-plan
resident work landed. The resident cuts proved that repeated grouped reads were
no longer dominated by compile/setup work, so the next useful step was to make
that grouped runtime floor observable instead of guessing at it.

It now does five concrete things:

* adds one grouped execute-phase split at the executor entrypoint:
  * stream build
  * fold/update
  * finalize/output
* threads that grouped phase split through both fluent and SQL perf-attribution
  surfaces instead of keeping grouped queries hidden behind the old scalar-only
  attribution shell
* adds owner-level grouped session tests that lock the new accounting contract
  without assuming host-side instruction counters are non-zero outside wasm
* extends the SQL and fluent PocketIC perf harnesses so grouped reports include
  dedicated grouped `COUNT(*)` fold submetrics:
  * borrowed hash computations
  * bucket candidate checks
  * existing-group hits
  * new-group inserts
* removes one repeated borrowed-probe eligibility check from the grouped fold hot
  path by freezing that decision from planner-visible `FieldKind` metadata once
  per route instead of rediscovering it on every row

What changed:

* grouped execution now has one dedicated perf-attribution carrier in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/entrypoints/grouped.rs`
  * `GroupedExecutePhaseAttribution { stream_local_instructions, fold_local_instructions, finalize_local_instructions }`
* the grouped entrypoint now exposes perf-only execution helpers for:
  * prepared grouped route execution
  * initial grouped canister execution
  * grouped traced load execution on `LoadExecutor<E>`
* grouped perf attribution is now re-exported through the executor and
  pipeline boundaries in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/entrypoints/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/mod.rs`
* typed/fluent query attribution no longer reports grouped queries through an
  empty scalar execute shell; instead it projects grouped runtime onto explicit
  grouped fields in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs`
  * new grouped-facing fields on `QueryExecutionAttribution`:
    * `grouped_stream_local_instructions`
    * `grouped_fold_local_instructions`
    * `grouped_finalize_local_instructions`
  * grouped fluent runtime now reports:
    * `runtime_local_instructions = grouped_stream + grouped_fold`
    * `finalize_local_instructions = grouped_finalize`
* SQL execute attribution now carries the same grouped runtime split in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/mod.rs`
  * new grouped-facing fields on `SqlQueryExecutionAttribution`:
    * `grouped_stream_local_instructions`
    * `grouped_fold_local_instructions`
    * `grouped_finalize_local_instructions`
* the SQL audit canister averages now retain both grouped phase counters and the
  dedicated grouped `COUNT(*)` fold submetrics in:
  * `/home/adam/projects/icydb/canisters/audit/sql_perf/src/lib.rs`
* the public session wrapper keeps the new grouped SQL attribution fields intact
  when it finalizes outward totals in:
  * `/home/adam/projects/icydb/crates/icydb/src/db/session/mod.rs`
* grouped attribution coverage now lives with the grouped session owner tests in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_grouped.rs`
  * one SQL grouped attribution test locks the executor-versus-grouped-phase
    accounting shape
  * one fluent grouped attribution test locks the grouped runtime/finalize rollup
    and proves grouped queries do not populate scalar direct-row counters
* while landing that slice, the current tree also needed a follow-up clippy fix in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/load.rs`

Perf follow-up:

  * `repeat.user.grouped.age_count.limit10.runs100`
    * `grouped_stream_local_instructions = 77,930`
    * `grouped_fold_local_instructions = 328,753`
    * `grouped_finalize_local_instructions = 36,733`
    * `grouped_count_borrowed_hash_computations = 6`
    * `grouped_count_bucket_candidate_checks = 1`
    * `grouped_count_existing_group_hits = 1`
    * `grouped_count_new_group_inserts = 5`
* that confirms the grouped floor is still dominated by fold work, and the first
  planner-frozen borrowed-probe cut moved exactly that slice slightly in the
  right direction without reopening compile-path work

Why this matters:

* the earlier `0.84.0` resident work already showed that repeated grouped rows
  were not moving much when more grouped setup was cached or frozen
* that meant the next useful question was no longer “can we remove more grouped
  preparation?”
* the real question became “which grouped runtime phase still owns the floor?”
* before this patch, the answer was obscured:
  * SQL grouped reads only showed planner/store/executor totals
  * fluent grouped reads returned one fake empty scalar shell for execute-phase
    attribution
* after this patch, grouped queries can be inspected directly at the runtime
  boundary without reopening planner or cache work:
  * stream build cost
  * fold/update cost
  * finalize/output cost
* that keeps the next optimization pass honest, because the grouped line can now
  target the dominant phase instead of widening more cache or resident work by
  intuition

## 0.84.0 - 2026-04-16 - Prepared Execution Residents

This patch starts the `0.84` line by moving repeated SQL and load setup work
onto the shared prepared-plan boundary instead of rebuilding the same execution
preparation and projection/layout contracts on every execute path.

It does three concrete things:

* freezes one shared scalar SQL projection resident on the prepared-plan boundary
* freezes the scalar retained-slot layouts actually used by scalar load entrypoints
* freezes grouped execution preparation and grouped ingest slot layout on the
  prepared-plan boundary for grouped load paths

What changed:

* the shared prepared-plan boundary now owns one frozen scalar SQL projection
  resident in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/prepared_execution_plan.rs`
* the SQL select cache now retains the shared prepared plan instead of throwing
  that resident away and rebuilding projection preparation later in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
* the SQL projection runtime consumes that frozen resident for repeated scalar
  SQL execution in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/projection/runtime.rs`
* the prepared-plan boundary now also freezes the scalar retained-slot layouts
  used by the scalar load families in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/prepared_execution_plan.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/entrypoints/scalar/mod.rs`
* grouped plans now freeze two cursor-independent residents at the same prepared
  boundary:
  * grouped `ExecutionPreparation`
  * grouped ingest `RetainedSlotLayout`
  in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/prepared_execution_plan.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/entrypoints/grouped.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/grouped/stages.rs`
* grouped runtime still keeps local fallback compilation for non-resident callers,
  so the resident cut stays local to the prepared-plan and repeated execution path

Why this matters:

* repeated scalar SQL queries were still paying projection/runtime setup that did
  not depend on the current continuation or page window
* repeated scalar load paths were still recompiling the same retained-slot layout
  families even when the prepared plan was unchanged
* grouped execution was still rebuilding grouped preparation and grouped ingest
  layout even after the grouped plan was already frozen
* moving those pieces onto the prepared-plan boundary keeps the repeated-query
  line focused on one rule:
  * compile or prepare once when the plan is frozen
  * reuse that resident on later executions
* the repeated scalar SQL rows moved materially, which proved the first resident
  cut was paying for itself
* the follow-up scalar retained-slot resident was architecturally correct but
  effectively flat on top of that first win
* the grouped prep/layout resident also cleaned ownership but left
  `repeat.user.grouped.age_count.limit10.runs100` roughly flat, which narrowed
  the next target to grouped runtime rather than grouped compile/setup

Perf read from the line:

* repeated scalar SQL moved materially in the right direction after the first
  resident cut, for example:
  * `repeat.user.pk.order_only.asc.limit2.runs100`
    * `-19.88%`
  * `repeat.user.name.lower.order_only.asc.limit3.runs100`
    * `-50.87%`
  * `user.age.order_only.asc.limit2.warm_after_update`
    * compile `109,076 -> 15,917`
* the follow-up scalar retained-slot resident was architecturally correct but
  effectively flat on top of that first win
* the grouped prep/layout resident also cleaned ownership but left
  `repeat.user.grouped.age_count.limit10.runs100` roughly flat, which narrowed
  the next target to grouped runtime rather than grouped compile/setup
