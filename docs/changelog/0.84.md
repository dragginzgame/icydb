# 0.84 — Prepared Execution Residents and Grouped Runtime Attribution

## 0.84.3 - 2026-04-16 - Grouped Follow-Through and Scalar Execution Seams

This patch is the narrow runtime follow-through after the `0.84.1` grouped
attribution work and the `0.84.2` planner cleanup. By this point the repeated
grouped floor was no longer a broad fold or compile problem. The remaining hot
path was the common single-field grouped `COUNT(*)` row, so this patch stays
local to that owner and trims the row-read and lookup pieces that were still
visible in the new grouped subphase counters.

It does three concrete things:

* tightens the one-slot grouped sparse row-read path so it validates the row
  table once and keeps only the selected grouped slot plus the primary-key slot
  instead of rebuilding the full field-span vector for every row
* adds one direct sparse required-field decode boundary so the one-slot grouped
  path can validate the primary key and decode the selected grouped slot from
  raw field bytes without routing back through the broader structural-row
  wrapper
* adds one direct stable-hash path for single-field identity-canonical grouped
  values, so `GROUP BY age` no longer routes lookup hashing through the generic
  one-element list writer on the hot repeated path

Later in the same patch line, the scalar executor side also stopped treating
retained layout and residual timing as ambient booleans that each late phase
could reinterpret independently. The shared scalar page path now resolves one
explicit materialization contract up front and then executes that resolved
contract through narrower strategy seams.

It does four concrete things on the scalar path:

* moves the runtime page request from a spread of loosely related flags to one
  raw capability bundle, `ScalarMaterializationCapabilities`, so the terminal
  path receives capabilities rather than half-resolved policy
* resolves one `ScalarMaterializationPlan` up front in the shared scalar page
  materializer, so direct-row eligibility, kernel scan behavior, post-access
  ownership, and final payload family are chosen once instead of being
  reconstructed across sibling branches
* resolves one `KernelRowScanStrategy` for the non-direct scalar lane and
  reuses that same strategy from both the main scalar page path and the
  cursorless row-collector short path, removing the hot late dispatch on raw
  `(payload_mode, residual_predicate_scan_mode)` pairs
* resolves one `PostAccessStrategy` plus one final payload strategy so the
  post-access and finalization tails also consume explicit executor-owned
  choices instead of re-reading raw retain-slot and residual-timing state

## 0.84.2 - 2026-04-16 - Query Planner Boundary Cleanup

This patch is a maintainability follow-through after the `0.84.0` prepared-plan
resident cuts and the `0.84.1` grouped runtime attribution work. Those patches
made the active compile/prepare and grouped runtime owners easier to see, which
also exposed how much duplicated helper scaffolding still lived around the
planner, validation, expression, and fingerprint boundaries.

It does four concrete things:

* trims repeated query-intent helper scaffolding around plan build, grouped
  mutation guards, paging errors, and intent policy mapping
* removes mirrored planner helper logic across compare/predicate/prefix/range
  access planning and order fallback selection
* narrows planner expression and grouped projection helpers so the projection
  boundary owns the current single projection-field unwrap instead of having it
  reimplemented across adjacent modules
* folds several small validation and fingerprint consumers back onto those
  shared query/planner owners instead of leaving local wrapper duplication in

## 0.84.1 - 2026-04-16 - Grouped Runtime Attribution and Fold Tightening

This patch started as grouped runtime attribution, but it did not stop there.
Once the repeated grouped floor was visible, the same slice followed that data
through the dominant grouped `COUNT(*)` path and removed a series of small but
real fold and row-read costs without widening grouped query behavior.

It now does seven concrete things:

* adds one grouped execute-phase split at the executor entrypoint:
  * stream build
  * fold/update
  * finalize/output
* threads that grouped phase split through both fluent and SQL perf-attribution
  surfaces instead of keeping grouped queries hidden behind the old scalar-only
  attribution shell
* adds owner-level grouped session tests that lock the new accounting contract
  without assuming host-side instruction counters are non-zero outside wasm
* extends the SQL and fluent PocketIC perf harnesses so grouped reports include
  dedicated grouped `COUNT(*)` fold submetrics:
  * borrowed hash computations
  * bucket candidate checks
  * existing-group hits
  * new-group inserts
* extends that same grouped `COUNT(*)` attribution again so the repeated floor
  can be split inside the dedicated fold path into:
  * row materialization / slot read
  * group lookup / hash-probe work
  * existing-group update
  * new-group insert
* removes repeated grouped `COUNT(*)` fold work by freezing one borrowed-probe
  eligibility decision from planner-visible `FieldKind` metadata, keeping the
  common bucket case inline, reusing precomputed stable hashes, and avoiding one
  intermediate one-element grouped-key vector on the single-field path
* tightens the one-slot grouped read path so the common `GROUP BY age`
  execution shape can decode and fold one direct grouped value without routing
  back through the broader row-view and owned grouped-key materialization path
* follows the new row-read and lookup attribution through the same hot grouped
