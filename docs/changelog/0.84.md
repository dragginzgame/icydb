# 0.84 — Prepared Execution Residents and Grouped Runtime Attribution

## 0.84.2 - 2026-04-16 - Query Planner Boundary Cleanup

This patch is a maintainability follow-through after the `0.84.0` prepared-plan
resident cuts and the `0.84.1` grouped runtime attribution work. Those patches
made the active compile/prepare and grouped runtime owners easier to see, which
also exposed how much duplicated helper scaffolding still lived around the
planner, validation, expression, and fingerprint boundaries.

It does four concrete things:

* trims repeated query-intent helper scaffolding around plan build, grouped
  mutation guards, paging errors, and intent policy mapping
* removes mirrored planner helper logic across compare/predicate/prefix/range
  access planning and order fallback selection
* narrows planner expression and grouped projection helpers so the projection
  boundary owns the current single projection-field unwrap instead of having it
  reimplemented across adjacent modules
* folds several small validation and fingerprint consumers back onto those
  shared query/planner owners instead of leaving local wrapper duplication in
  place

Why this matters:

* the earlier `0.84` patches moved more active runtime work onto explicit
  prepared-plan and grouped-runtime owners
* once those hot owners were clearer, the remaining cold-path duplication around
  query planning became much easier to see
* that duplication was not changing query behavior, but it was still making
  planner and validation changes noisier than they needed to be
* this patch keeps the same query behavior and the same feature surface, but it
  leaves fewer parallel helper paths behind in the planner-adjacent code
* that should make the next behavior-changing planning or projection slice
  cheaper to reason about, because more of the shared planner rules now live in
  one owner-local place

## 0.84.1 - 2026-04-16 - Grouped Runtime Attribution and Fold Tightening

This patch started as grouped runtime attribution, but it did not stop there.
Once the repeated grouped floor was visible, the same slice followed that data
through the dominant grouped `COUNT(*)` path and removed a series of small but
real fold and row-read costs without widening grouped query behavior.

It now does six concrete things:

* adds one grouped execute-phase split at the executor entrypoint:
  * stream build
  * fold/update
  * finalize/output
* threads that grouped phase split through both fluent and SQL perf-attribution
  surfaces instead of keeping grouped queries hidden behind the old scalar-only
  attribution shell
* adds owner-level grouped session tests that lock the new accounting contract
  without assuming host-side instruction counters are non-zero outside wasm
* extends the SQL and fluent PocketIC perf harnesses so grouped reports include
  dedicated grouped `COUNT(*)` fold submetrics:
  * borrowed hash computations
  * bucket candidate checks
  * existing-group hits
  * new-group inserts
* removes repeated grouped `COUNT(*)` fold work by freezing one borrowed-probe
  eligibility decision from planner-visible `FieldKind` metadata, keeping the
  common bucket case inline, reusing precomputed stable hashes, and avoiding one
  intermediate one-element grouped-key vector on the single-field path
* tightens the one-slot grouped read path so the common `GROUP BY age`
  execution shape can decode and fold one direct grouped value without routing
  back through the broader row-view and owned grouped-key materialization path

## 0.84.0 - 2026-04-16 - Prepared Execution Residents

This patch starts the `0.84` line by moving repeated SQL and load setup work
onto the shared prepared-plan boundary instead of rebuilding the same execution
preparation and projection/layout contracts on every execute path.

It does three concrete things:

* freezes one shared scalar SQL projection resident on the prepared-plan boundary
* freezes the scalar retained-slot layouts actually used by scalar load entrypoints
* freezes grouped execution preparation and grouped ingest slot layout on the
  prepared-plan boundary for grouped load paths

What changed:

* the shared prepared-plan boundary now owns one frozen scalar SQL projection
  resident in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/prepared_execution_plan.rs`
* the SQL select cache now retains the shared prepared plan instead of throwing
  that resident away and rebuilding projection preparation later in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
* the SQL projection runtime consumes that frozen resident for repeated scalar
  SQL execution in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/projection/runtime.rs`
* the prepared-plan boundary now also freezes the scalar retained-slot layouts
  used by the scalar load families in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/prepared_execution_plan.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/entrypoints/scalar/mod.rs`
* grouped plans now freeze two cursor-independent residents at the same prepared
  boundary:
  * grouped `ExecutionPreparation`
  * grouped ingest `RetainedSlotLayout`
  in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/prepared_execution_plan.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/entrypoints/grouped.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/grouped/stages.rs`
* grouped runtime still keeps local fallback compilation for non-resident callers,
  so the resident cut stays local to the prepared-plan and repeated execution path

Why this matters:

* repeated scalar SQL queries were still paying projection/runtime setup that did
  not depend on the current continuation or page window
* repeated scalar load paths were still recompiling the same retained-slot layout
  families even when the prepared plan was unchanged
* grouped execution was still rebuilding grouped preparation and grouped ingest
  layout even after the grouped plan was already frozen
* moving those pieces onto the prepared-plan boundary keeps the repeated-query
  line focused on one rule:
  * compile or prepare once when the plan is frozen
  * reuse that resident on later executions
* the repeated scalar SQL rows moved materially, which proved the first resident
  cut was paying for itself
* the follow-up scalar retained-slot resident was architecturally correct but
  effectively flat on top of that first win
* the grouped prep/layout resident also cleaned ownership but left
  `repeat.user.grouped.age_count.limit10.runs100` roughly flat, which narrowed
  the next target to grouped runtime rather than grouped compile/setup
