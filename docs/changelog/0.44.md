# IcyDB 0.44

This document aggregates detailed release notes for the `0.44.x` minor line.

## 0.44.3 - 2026-03-07 - Access Capability Projection Cleanup

### 📝 Summary

`0.44.3` cleans up access capability derivation by collapsing repeated constant
shape checks into one static capability projection per access-path kind, and
extends safe `COUNT` and `bytes()` by-ids fast-path coverage to PK-ordered
windows.

This reduces duplicated interpretation logic across route/access layers while
keeping `COUNT` result semantics stable.

### 🔧 Changed

- Refactored
  `crates/icydb-core/src/db/access/capabilities.rs::derive_access_path_capabilities`
  so static capability flags are derived in one match over `AccessPathKind`
  instead of via multiple per-flag helper functions.
- Removed redundant helper functions that separately re-derived:
  - PK stream access support
  - count-pushdown shape support
  - primary-scan fetch-hint support
  - reverse-traversal support
  - key-direct-access shape
- Removed route shim helpers that only forwarded access capability flags:
  - `supports_pk_stream_access_executable_path`
  - `primary_scan_fetch_hint_for_executable_access_path`
- Removed aggregate-route compatibility shim surface:
  - compatibility accessor `secondary_extrema_probe_fetch_hint(...)`
  - duplicate route field `aggregate_secondary_extrema_probe_fetch_hint`
- Updated route/stream call sites to consume capability snapshots directly:
  - `route::fast_path` now checks path capability flags directly and keeps an
    alignment debug assert against the access-class snapshot
  - `stream::access::physical` now gates primary-scan fetch hints directly via
    `ExecutableAccessPath::capabilities()`
- Expanded primary-key stream COUNT fast-path eligibility in
  `crates/icydb-core/src/db/executor/aggregate/terminals.rs` so
  `ExecutionPathKind::ByKeys` supports:
  - unordered by-ids windows
  - PK-ordered (`ORDER BY id`) by-ids windows
- Made COUNT key-stream fold direction-aware for those shapes so DESC PK-order
  windows use DESC access-stream bindings instead of forcing ASC traversal.
- Added explicit regression coverage for:
  - ordered by-ids dedup/window fast-path behavior
  - ordered DESC by-ids fast-path behavior
  - unordered by-ids fast-path hit diagnostics
- Expanded scalar `bytes()` stream fast-path eligibility in
  `crates/icydb-core/src/db/executor/load/terminal.rs` so PK-ordered by-ids
  windows (`ORDER BY id`) can fold persisted payload bytes from ordered key
  streams without row materialization.
- Made `bytes()` stream fast-path bindings direction-aware so PK DESC by-ids
  windows use DESC access-stream traversal.
- Added regression coverage for ordered by-ids `bytes()` stream-fast-path
  parity and hit diagnostics.
- Added a structural guard test that blocks reintroduction of those route
  capability shim symbols into executor runtime modules.
- Kept payload-derived capability fields (`is_by_keys_empty`,
  index-shape/spec metadata) unchanged.

## 0.44.2 - 2026-03-07 - BYTES Unordered Secondary Stream Fast Path Follow-Up

### 📝 Summary

`0.44.2` extends scalar `bytes()` fast-path coverage for safe unordered
secondary-stream shapes:

- scalar `bytes()` now folds persisted payload sizes directly from ordered
  key-stream traversal for eligible unordered secondary windows
- row materialization/decode is bypassed when route shape guarantees allow it
- ordered or predicate/distinct-sensitive shapes remain on canonical fallback

### 🔧 Changed

- Added ordered key-stream payload-byte folding in
  `crates/icydb-core/src/db/executor/context/mod.rs` via
  `sum_row_payload_bytes_from_ordered_key_stream(...)`.
- Preserved missing-row safety policy behavior during stream folding:
  - `MissingRowPolicy::Error`: strict read, fail closed
  - `MissingRowPolicy::Ignore`: skip stale/missing rows
- Added a BYTES stream fast-path branch in
  `crates/icydb-core/src/db/executor/load/terminal.rs` for unordered scalar
  shapes (`no predicate`, `no distinct`, `no explicit order`).
- Added stream-path hit diagnostics for tests:
  - `take_bytes_stream_fast_path_hits_for_tests`
- Added parity and branch-guard regression coverage in
  `crates/icydb-core/src/db/executor/tests/aggregate/path_parity_matrices.rs`:
  - unordered secondary window parity + hit-marker assertion
  - ordered secondary shape bypass assertion (fast path not used)

## 0.44.1 - 2026-03-07 - COUNT Index-Range Pushdown and EXISTS Covering Follow-Up

### 📝 Summary

`0.44.1` closes two post-`0.44.0` optimization follow-ups:

- COUNT index-range plans can now inherit bounded LIMIT pushdown fetch windows
  when route eligibility gates allow it.
- secondary covering `EXISTS` now supports strict-compatible predicate shapes
  without forcing materialized fallback.

### 🔧 Changed

- Removed the COUNT-specific feasibility gate in
  `crates/icydb-core/src/db/executor/route/planner/feasibility.rs` that
  previously blocked index-range LIMIT spec derivation for COUNT terminals.
- Updated COUNT route execution assembly in
  `crates/icydb-core/src/db/executor/route/planner/execution/shape_aggregate_count.rs`
  so streaming COUNT routes can carry route-derived `index_range_limit_spec`.
- Added route regression coverage in
  `crates/icydb-core/src/db/executor/tests/route/aggregate.rs` to lock bounded
  fetch inheritance for index-range COUNT windows.
- Added stale-leading missing-row parity coverage in
  `crates/icydb-core/src/db/executor/tests/aggregate/secondary_index_matrices.rs`
  to confirm COUNT pushdown parity with canonical `execute().count()` and to
  assert bounded scan-accounting stability.
- Expanded secondary covering `EXISTS` fast-path eligibility in
  `crates/icydb-core/src/db/executor/aggregate/terminals.rs` to allow strict-
  compatible predicate shapes and pass strict index predicate execution through
  the covering stream descriptor path.
- Aligned aggregate EXPLAIN covering eligibility for `EXISTS` in
  `crates/icydb-core/src/db/executor/explain/descriptor.rs` so descriptor
  output matches runtime fast-path eligibility.
- Updated `0.44` optimization design/status trackers to mark these follow-ups
  complete:
  - `docs/design/0.44-optimisation-closure.md`
  - `docs/status/0.44-optimisation-closure-status.md`

## 0.44.0 - 2026-03-07 - COUNT and BYTES Safe PK Fast Paths

### 📝 Summary

`0.44.0` starts optimization closure by landing safe PK-shape fast paths for
both scalar `COUNT(*)` and scalar `bytes()` terminals:

- `ExecutionPathKind::FullScan`
- `ExecutionPathKind::KeyRange`

For those shapes:

- scalar `COUNT(*)` / `COUNT(primary_key)` computes from primary-store
  cardinality windows without row decode/materialization
- scalar `bytes()` folds persisted row payload lengths directly from
  primary-store windows without canonical `execute()` materialization

Both paths preserve canonical pagination window behavior.

### 🔧 Changed

- Added PK-cardinality COUNT fast-path routing in
  `crates/icydb-core/src/db/executor/aggregate/terminals.rs` for single-path,
  non-distinct, no-residual-predicate `FullScan` and `KeyRange` access shapes.
- Added explicit COUNT page-window projection helper so fast-path scan metrics
  preserve existing `offset`/`limit` semantics (`rows_scanned` parity).
- Added PK-cardinality COUNT fast-path hit diagnostics
  (`take_pk_cardinality_count_fast_path_hits_for_tests`) and regression
  coverage for eligible PK full-scan versus ineligible by-id COUNT shapes.
- Kept DISTINCT+offset behavior stable by excluding distinct scalar plans from
  the new cardinality fast path.
- Added `ExecutablePlan::is_distinct()` to expose scalar DISTINCT state through
  the executor contract surface.
- Added safe PK `bytes()` fast-path routing in
  `crates/icydb-core/src/db/executor/load/terminal.rs` for single-path,
  non-distinct, no-residual-predicate `FullScan` and `KeyRange` access shapes.
- Added shared bytes page-window helpers so direct store traversal preserves
  canonical `offset`/`limit` semantics.
- Added BYTES PK fast-path hit diagnostics (`take_bytes_pk_fast_path_hits_for_tests`)
  and regression coverage so eligible PK full-scan windows are explicit in
  tests while ineligible fallback shapes remain non-fast-path.
- Added key-range-specific BYTES fast-path hit-marker regression coverage so
  eligible key-range windows emit fast-path hits while residual-predicate
  key-range shapes remain on fallback.
- Moved BYTES PK window payload traversal behind executor context adapters so
  load terminal modules no longer traverse stores directly, preserving the
  load-module structural boundary guard.
- Converted BYTES/COUNT/covering aggregate fast-path hit diagnostics to
  thread-local test counters so parallel test execution remains deterministic.
- Tightened ordered index-range `LIMIT` pushdown eligibility at route
  capability boundaries so ordered shapes require planner logical pushdown
  eligibility in addition to access shape/order compatibility.
- Added route capability regression coverage that preserves no-order
  index-range behavior while rejecting empty-order planner-bypass shapes.
- Added explicit route regression coverage for incompatible ordered index-range
  shapes to ensure `LIMIT` pushdown is disabled when planner ordering
  guarantees are not satisfied.
- Completed LIMIT pushdown closure coverage for residual-cap, underfill-retry,
  and cursor-boundary behavior on index-range shapes.
- Aligned COUNT secondary routing with covering terminal semantics by enabling
  streaming COUNT pushdown for index-prefix/index-range shapes that satisfy the
  strict prefilter contract, and selecting `AggregateFoldMode::ExistingRows`
  for stale-key-safe fold behavior.
- Kept COUNT shapes with strict prefilter uncertainty materialized at route
  boundaries.
- Updated aggregate EXPLAIN descriptor assembly to project index-only covering
  visibility for eligible scalar `exists()`/`count()` index-prefix/index-range
  shapes, and to keep ordered shapes explicitly non-covering.
- Expanded COUNT covering EXPLAIN regression coverage across strict prefilter
  boundaries so strict-compatible secondary predicates remain covering while
  strict-uncertain predicate shapes report non-covering.
- Added COUNT fold-mode EXPLAIN metadata
  (`node_properties.count_fold_mode`) so descriptor output now distinguishes
  PK-safe `keys_only` routes from secondary stale-key-safe `existing_rows`
  routes.
- Added aggregate descriptor regression coverage so covering-scan explain flags
  track `EXISTS`/`COUNT` fast-path eligibility contracts.
- Added covering projection terminal fast-path diagnostics in
  `aggregate::projection`:
  - `take_covering_index_projection_fast_path_hits_for_tests`
  - `take_covering_constant_projection_fast_path_hits_for_tests`
- Added regression coverage that verifies covering projection hit markers are
  emitted only for eligible constant/index-covered projection shapes and remain
  zero for ineligible fallback shapes.
- Extended `ExecutionTrace` optimization taxonomy with explicit top-N-assisted
  fast-path labels (`PrimaryKeyTopNSeek`, `SecondaryOrderTopNSeek`) and route
  metadata-aware projection from load fast-path outcomes.
- Added pagination trace regressions for PK and secondary-order top-N-assisted
  optimization labels.
- Added non-top-N trace regressions that lock base optimization labels:
  - unpaged PK ordered routes -> `PrimaryKey`
  - unpaged secondary ordered routes -> `SecondaryOrderPushdown`
- Explicitly deferred grouped bounded fast-path expansion
  (`COUNT`/`MIN`/`MAX`) to post-`0.44` with owner/rationale tracking to avoid
  implicit half-finished grouped optimization surfaces.
- Updated workspace toolchain/MSRV references to Rust `1.94.0`.
