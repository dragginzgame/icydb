# 0.87 — Grouped ORDER BY Proof Cleanup

## 0.87.1 - 2026-04-17 - Shared Grouped Row Evaluation Context

This follow-up keeps `0.87` on the grouped post-aggregate cleanup line instead of widening query behavior again. Grouped projection and grouped `HAVING` still do not share one compiled expression IR yet, but they now read one shared finalized grouped-row contract instead of carrying separate runtime assumptions about grouped-key and aggregate payload shape.

The practical effect is internal rather than user-visible: grouped `HAVING` no longer reaches into canonical grouped-key list values directly or threads parallel `group_fields` / `aggregate_values` slices through its runtime evaluator. It now consumes the same `GroupedRowView` adapter that grouped projection already evaluates against, which narrows the remaining evaluator-unification seam down to compiled expression ownership rather than row-shape interpretation.

What changed:

* `GroupedRowView` now always carries grouped field-slot metadata, not just grouped key and aggregate values, so it can serve as the shared post-aggregate row adapter for both grouped projection and grouped `HAVING` in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/grouped.rs`
* grouped `HAVING` runtime evaluation now takes `&GroupedRowView` directly instead of separately accepting grouped fields, raw canonical grouped-key values, and aggregate slices in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/mod.rs`
* grouped page finalization and the grouped count fast path now build one shared grouped-row adapter before running post-aggregate `HAVING` checks, so the grouped runtime has one row-shape owner at those call sites in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/grouped_fold/page_finalize.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/grouped_fold/mod.rs`
* grouped projection tests now read grouped field metadata through the shared adapter API instead of reaching through row-internal test-only state in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/tests/mod.rs`
* the old grouped `HAVING` executor invariant for non-list grouped keys was removed from the plan semantics boundary because grouped `HAVING` no longer resolves grouped fields from raw canonical `Value::List` payloads in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_model.rs`

This does not yet remove the dedicated grouped `HAVING` recursive evaluator. That remaining duplication is now narrower and more mechanical: compiled expression ownership, not grouped row interpretation.

## 0.87.0 - 2026-04-17 - Explicit Grouped ORDER BY Admission

This patch starts the `0.87` line by tightening two grouped-planning seams at
once. First, grouped `ORDER BY` admission now uses one explicit planner-owned
proof model instead of leaving the safe computed cases looking like additive
special handling. Second, grouped `HAVING` is now internally expression-owned
end to end, and the last clause-shaped SQL lowering path has been removed.

The main user-visible widening is that subtractive grouped-key order such as
`ORDER BY level - 2` is now treated as a first-class admitted grouped order
shape, while grouped explain and route observability now distinguish logical
prefix mismatch from unsupported grouped order expressions. Internally, grouped
planning/execution now carries one `GroupHavingExpr` representation instead of
mixing expression-owned and clause-wrapper grouped `HAVING` shapes.

What changed:

* grouped `ORDER BY` proof now classifies grouped order terms explicitly as:
  * canonical grouped key
  * grouped key plus constant
  * grouped key minus constant
  * prefix mismatch
  * unsupported-but-evaluable grouped order expression
* the grouped order classifier now lives in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/projection.rs`
* grouped planner cursor policy now uses that classifier directly instead of a
  boolean helper, so grouped order validation can keep prefix mismatch and
  non-admissible grouped expressions on different policy lanes in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/grouped/cursor/mod.rs`
* grouped planner policy errors now include the explicit grouped-order
  rejection:
  * `GroupPlanError::OrderExpressionNotAdmissible { term }`
  in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/errors.rs`
* grouped strategy/explain/route observability no longer collapse all logical
  grouped-order failures into `group_key_order_unavailable`; they now separate:
  * `group_key_order_prefix_mismatch`
  * `group_key_order_expression_not_admissible`
  * `group_key_order_unavailable`
  in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/grouped_strategy.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/plan.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/tests/grouped.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/planning/route/tests/grouped.rs`
* SQL/grouped validation coverage now pins the newly admitted subtractive
  grouped order shape and the more specific non-admissible grouped-order error
  surface in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/tests/grouped.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/tests/group.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_grouped.rs`
* grouped `HAVING` now uses one internal representation:
  * `GroupPlan` stores only `having_expr`
  * grouped executor handoff/runtime consume `Option<&GroupHavingExpr>`
  * `GroupHavingSpec` has been removed from `icydb-core`
  in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/model.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_plan.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/grouped_runtime/route_stage.rs`
* the last clause-only grouped `HAVING` SQL lowering path is gone:
  * `lower_legacy_having_clause(...)` was removed
  * grouped SQL `HAVING` lowering now always resolves through the expression-owned path
  in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/mod.rs`
* grouped cache-key and explain/fingerprint surfaces were aligned with the
  expression-owned model so grouped `HAVING` no longer carries stale
  clause-wrapper wording or mixed planner/explain fallbacks in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/plan.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/hash_parts/mod.rs`

Example that is now explicitly admitted:

```sql
SELECT level, COUNT(*)
FROM character
GROUP BY level
ORDER BY level - 2
LIMIT 100;
```

Still intentionally out of scope:

* aggregate-valued grouped ordering like `ORDER BY AVG(strength)` remains
  rejected.
* grouped wrapper/function ordering like `ORDER BY LOWER(class_name)` remains
  rejected unless it is separately proven later.
* this patch does not add broader grouped sort semantics; it only formalizes
  safe order-preserving grouped-key transforms.
* `GroupHavingClause` still exists as an ingress/lowering shape in a few query
  builder and lowering boundaries; this patch removes the old stored/runtime
  wrapper model, not every clause-shaped constructor at once.
