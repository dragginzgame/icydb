# 0.108 Changelog

## 0.108.0

- `0.108.0` finishes the first canonical-query-identity cleanup by making canonical scalar `filter_expr` the owner for the nearby query-identity surfaces that were still preserving spelling drift after `0.105` and `0.107`.
- The core hashing work landed in:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/hash_parts/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/hash_parts/mod.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/hash_parts/profile.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/hash_parts/profile.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/projection_hash.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/projection_hash.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/plan.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/plan.rs)
- Before this patch, several identity-adjacent surfaces still drifted between:
  - predicate-owned identity when a derived predicate existed
  - expression-owned identity when only semantic `filter_expr` survived
  - fluent insertion-order identity when repeated `.filter(...)` calls built left-associated raw `AND` trees before plan hashing
- The shipped rule is now:
  - when canonical scalar `filter_expr` exists, it is the semantic identity owner for fingerprint and continuation-signature hashing
  - derived predicate remains a fallback identity input only when no semantic filter expression exists
  - explain-side plan hashing uses the same semantic filter model as planned-query hashing instead of rendered filter text
  - repeated fluent `.filter(...)` accumulation normalizes the combined boolean tree before storing it, so intent-owned `filter_expr` no longer preserves insertion-order drift
- That normalization follow-through landed in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/mutation.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/mutation.rs). The important fix there is not just cleaner shape; it closes a real outward identity bug where canonical-equivalent fluent filters could still produce different `plan_hash_hex()` values and separate shared query-plan cache entries.
- Publicly, the final `0.108.0` outcomes are:
  - equivalent scalar boolean `WHERE` spellings now share one canonical fingerprint and continuation signature when they already reduce to the same planner-owned boolean form
  - semantically distinct expression-owned scalar filters still stay distinct in identity and cache reuse
  - direct query `plan_hash_hex()`, planned-query hash, compiled-query hash, and explain signature now line up on the same canonical scalar filter identity
  - session `trace_query` and the shared query-plan cache now reuse one entry for canonical-equivalent scalar filters instead of splitting by filter spelling or fluent insertion order
- This patch intentionally does not add new rewrite rules, new planner behavior, or new prepared-template widening. It is an identity-alignment slice only: use the canonical semantic filter the planner already owns, and stop letting nearby hash/cache surfaces preserve older spelling-level drift.

```rust
let left = session
    .load::<SessionSqlEntity>()
    .filter(FieldRef::new("age").gte(20_i32))
    .filter(FieldRef::new("age").lt(40_i32));

let right = session
    .load::<SessionSqlEntity>()
    .filter(FieldRef::new("age").lt(40_i32))
    .filter(FieldRef::new("age").gte(20_i32));
```

- After `0.108.0`, those two fluent queries now reuse the same canonical plan identity instead of producing different outward hashes because of filter insertion order.
