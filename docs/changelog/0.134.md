## 0.134.1

`0.134.1` continues delete and key-stream convergence by collapsing the
repeated typed delete entrypoint orchestration into one shared core, then
moving physical primary/index scan leaves onto lazy executor key streams,
making prepared projection/layout residents lazy by execution path, and
starting executor ordering cleanup with a narrow borrowed direct-slot order fast
path for retained rows. It also bounds SQL `DISTINCT` projection once
post-dedup LIMIT/OFFSET has been satisfied. Traced debug execution now also
captures an internal `ExecutionStats` snapshot so executor optimization work
can compare key-stream, predicate, projection, ordering, DISTINCT, and grouped
fold costs without changing public response payloads.

Typed row-returning deletes and count-only deletes now share the same runtime
setup, candidate row resolution, post-access selection, rollback-row packaging,
relation validation, and commit-op preparation path. The public entrypoints
only choose their outward payload shape: deleted response rows or affected-row
count. SQL `DELETE ... RETURNING ...` keeps using the structural projection
path, but now also shares the same typed runtime setup helper and final typed
commit-window bridge.

## 0.134.0

`0.134.0` starts the executor convergence line with behavior-preserving slices
that make duplicated execution paths visible and begin routing SQL/delete
surfaces back through executor-owned runtime seams.

Execution trace construction no longer lets diagnostics interpret planner or
executor structures. The trace DTO now accepts an already projected
`ExecutionAccessPathVariant`, while executor diagnostics owns the
`AccessPlan`/runtime-direction projection used by scalar and grouped execution
entrypoints. The runtime optimization outcome label also moved to
executor-owned diagnostics code, with `db::diagnostics::ExecutionOptimization`
and the db-root re-export retained as compatibility surfaces.

Executor convergence now has an explicit regression suite for SQL/fluent scalar
equivalence, grouped equivalence, delete-target equivalence, SQL `DISTINCT`
equivalence, and light explain/execution route consistency. The delete path now
uses the shared execution key-stream resolver for candidate selection instead
of owning a separate traversal/predicate/key-selection spine; delete-specific
row capture, rollback, commit-op assembly, and response shaping remain
delete-owned. SQL `DISTINCT` projection still materializes the full ordered
stream before dedup/window shaping, but it now bypasses SQL-side covering
projection forks and enters the shared scalar retained-slot execution path
first. SQL global aggregate execution also moved projected-value fold/finalize
behavior into an executor-owned reducer, leaving SQL responsible only for input
projection, optional HAVING, labels, and outward row shaping.

EXPLAIN route reporting now consumes one frozen load execution route-facts
snapshot when building descriptor and verbose diagnostics surfaces. Typed query
EXPLAIN and SQL EXPLAIN both freeze the route facts once, then reuse that
snapshot for descriptor assembly, verbose route diagnostics, and SQL projection
debug annotation instead of recomputing route descriptors locally. The output
shape is unchanged; the new execution convergence tests assert that EXPLAIN
route families match the actual traced execution route for full-scan,
primary-key lookup, and secondary-index range paths.
