# IcyDB 0.167 Changelog

## 0.167.6

### 🔧 Changed

- Tightened store registry registration so every call supplies an explicit
  allocation identity decision. Generated stable stores pass stable allocation
  identities; tests and future non-stable backends pass `absent()` explicitly.

  ```rust
  register_store(..., StoreAllocationIdentities::new(...))
  register_store(..., StoreAllocationIdentities::absent())
  StoreHandle::new(..., StoreAllocationIdentities::absent())
  ```

### 🧪 Testing

- Added registry coverage proving explicit absent allocation identities keep
  data, index, and schema allocation metadata absent, while stable allocation
  identities preserve all three stable allocation descriptors.

## 0.167.5

### 🔧 Changed

- Tightened generated stable-store wiring so the matched
  `StoreStorage::Stable` configuration is the generated memory-ID authority
  for data, index, and schema roles.

  ```rust
  StoreStorage::Stable(config) => stable_store_registry_entry_tokens(..., *config)
  ```

### 🧪 Testing

- Strengthened the generated stable wiring proof so the emitted
  `ic_memory_key!` calls contain the configured data/index/schema memory IDs
  and still register all three stable allocation identities.

  ```text
  id = 10u8
  id = 11u8
  id = 12u8
  ```

### 📚 Documentation

- Updated the 0.167 proof matrix to record that generated stable wiring
  consumes the matched `storage(stable(...))` config directly while preserving
  stable keys, allocation identities, and storage ABI.

## 0.167.4

### ➕ Added

- Added explicit storage-mode metadata to storage snapshot diagnostics for data,
  index, and schema store rows. Current 0.167 stores report `stable`.

  ```text
  storage: stable
  ```

### 🔧 Changed

- The CLI snapshot table now renders a `storage` column for data, index, and
  schema stores.

  ```text
  path  storage  mem  stable key  version  fingerprint
  ```

### 🧪 Testing

- Extended diagnostics tests to prove data, index, and schema snapshot rows
  report `StoreSnapshotStorageMode::Stable`.
- Updated Candid-shape tests so the diagnostic `storage` field is part of the
  public snapshot payload contract.

### 📚 Documentation

- Updated the 0.167 design to record the diagnostics decision as implemented:
  storage-mode reporting is observability metadata only and does not affect
  `memory_id + stable_key` allocation identity.

## 0.167.3

### 🔧 Changed

- Tightened the public `StoreStorage` storage-boundary accessor so stable
  configuration is exposed as optional storage metadata instead of a bare
  stable-only contract on the enum.

  ```rust
  store.storage().stable_memory_config() -> Option<&StoreStableMemoryConfig>
  ```

### 🧪 Testing

- Extended the schema model test for explicit stable storage ownership to prove
  the enum-level storage accessor exposes stable metadata through the
  future-safe optional boundary.

- Added compile-fail coverage for empty `storage()`, flat `storage = "stable"`,
  duplicate `storage(...)` arguments, and a single `storage(...)` declaration
  containing multiple storage modes.

  ```rust
  storage()
  storage = "stable"

  storage(stable(...))
  storage(stable(...))

  storage(stable(...), heap())
  ```

## 0.167.2

### ⚠️ Breaking

- Split the wasm fixture schema helper into separate canister and store
  declarations, and hard-cut the store helper to require the same nested
  `storage(stable(...))` form as `#[store(...)]` declarations.

  ```rust
  define_fixture_canister!(
      ExampleCanister = "ExampleCanister",
      namespace = "example",
      memory_min = 100,
      memory_max = 110,
      commit_memory_id = 103,
  );

  define_fixture_store!(
      ExampleStore = "EXAMPLE_STORE",
      canister = "ExampleCanister",
      storage(stable(
          data_memory_id = 100,
          index_memory_id = 101,
          schema_memory_id = 102,
      )),
  );
  ```

### 🔧 Changed

- Migrated generated wasm fixture schema crates to the nested store-storage
  syntax so repository fixtures no longer preserve the removed flat memory-ID
  vocabulary outside deliberate compile-fail tests.

### 📚 Documentation

- Clarified that `memory_min`, `memory_max`, and `commit_memory_id` remain
  canister-level stable-memory manager configuration in 0.167, while
  data/index/schema stable-memory IDs belong to each store's
  `storage(stable(...))` configuration.

## 0.167.1

### ⚠️ Breaking

- Raised the declared MSRV to Rust `1.96.0`.

  ```toml
  rust-version = "1.96.0"
  ```

### 🧹 Cleanup

- Moved match-shape assertions onto the standard Rust `assert_matches!` and
  `debug_assert_matches!` macros now that the declared MSRV admits them.

  ```rust
  std::assert_matches!(err, ErrorKind::Invalid { .. });
  std::debug_assert_matches!(state, State::Ready);
  ```

### 🧪 Testing

- Added compile-fail and schema-validation coverage proving
  `storage(stable(...))` rejects missing required memory IDs, duplicate
  stable-memory ID arguments, and duplicate data/index/schema role IDs.

  ```rust
  storage(stable(index_memory_id = 101, schema_memory_id = 102))
  storage(stable(data_memory_id = 100, data_memory_id = 101, ...))
  StoreStableMemoryConfig::new(110, 110, 112)
  ```

### 📚 Documentation

- Updated public setup examples to use the 0.167 store-storage syntax instead
  of the removed flat memory-ID form.

  ```rust
  #[store(storage(stable(
      data_memory_id = 100,
      index_memory_id = 101,
      schema_memory_id = 102,
  )))]
  ```

## 0.167.0

### ➕ Added

- Added an explicit per-store storage model with `StoreStorage` and
  `StoreStableMemoryConfig`.

  ```rust
  StoreStorage::Stable(StoreStableMemoryConfig::new(data, index, schema))
  ```

- Added the nested derive syntax for the only implemented 0.167 storage mode:

  ```rust
  #[store(
      ident = "SESSION_STORE",
      store_name = "sessions",
      canister = "AppCanister",
      storage(stable(
          data_memory_id = 20,
          index_memory_id = 21,
          schema_memory_id = 22,
      )),
  )]
  ```

- Added compile-fail coverage for missing store storage, old flat memory IDs,
  reserved heap syntax, heap syntax with stable IDs, and unknown storage modes.

### ⚠️ Breaking

- Store declarations must now place stable memory IDs inside
  `storage(stable(...))`.

  ```rust
  // removed
  #[store(data_memory_id = 20, index_memory_id = 21, schema_memory_id = 22)]

  // required
  #[store(storage(stable(
      data_memory_id = 20,
      index_memory_id = 21,
      schema_memory_id = 22,
  )))]
  ```

- The old flat store memory-ID syntax is rejected rather than aliased or
  shimmed. This is a source-level macro hard cut only.

### 🔧 Changed

- Moved store validation behind the explicit stable-storage branch while
  preserving the existing stable memory range, reserved ID, stable-key, and
  data/index/schema distinctness checks.

- Moved generated store registry wiring behind an explicit stable-storage
  branch. Stable stores still emit `ic_memory_key!` for data, index, and schema
  roles and still register all three stable allocation identities.

- Reserved `storage(heap())` for a future volatile store line without adding a
  runtime heap store variant or generated heap wiring in 0.167.

### 🧪 Testing

- Proved the storage-boundary matrix for schema model ownership, derive macro
  admission/rejection, stable generated wiring, unchanged allocation identity,
  and no generated heap path.

### 📚 Documentation

- Updated the 0.167 storage-boundary design to reflect the implemented hard
  cut, proof matrix status, and deferred heap-store work.
