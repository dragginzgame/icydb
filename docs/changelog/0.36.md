## 0.36.3 - 2026-03-01 - Grouped ORDER BY Boundedness Gate

### 📝 Summary

`0.36.3` closes grouped hardening by enforcing bounded grouped ordering and resolving the remaining grouped resource-audit gaps.

### 🔧 Changed

- Grouped planning now rejects grouped `ORDER BY` without explicit `LIMIT` using typed error `GroupPlanError::OrderRequiresLimit`.
- Grouped DISTINCT admission now routes through one execution-context boundary (`ExecutionContext::admit_distinct_key`) for grouped Class B cap enforcement.
- Grouped metrics strategy labeling is explicitly materialized (`HashMaterialized`, `OrderedMaterialized`) to match grouped runtime mode.
- Grouped continuation cursors now require an explicit grouped `LIMIT`; grouped global DISTINCT continuations remain rejected.
- Added grouped bounded-vector hardening assertions for short-circuit keys, candidate rows, and grouped-page emission limits.
- Added continuation-signature snapshot tests for grouped HAVING, grouped DISTINCT, global DISTINCT SUM, and ordered-hint grouped strategy shapes.
- Added grouped resource-accounting and `SUM(DISTINCT)` determinism tests (large-value accumulation, canonical scale, and traversal-order invariance).
- Reclassified non-grouped scalar DISTINCT helper terminals as effective-window materialized helpers outside grouped Class B budget accounting.
- Refreshed grouped resource-compliance audit reports full pass: [`docs/audits/reports/2026-03/2026-03-01/resource-model-compliance-0.36.3.md`](../audits/reports/2026-03/2026-03-01/resource-model-compliance-0.36.3.md)

```rust

    .load::<Order>()
    .group_by("rank")?
    .group_count()
    .order_by("rank")
    .limit(10)
    .execute_grouped()?;
```

## 0.36.2 - 2026-03-01 - Global DISTINCT Zero-Key Budget Hardening

### 📝 Summary

`0.36.2` hardens zero-key global DISTINCT grouped execution so singleton-group admission and distinct accounting flow through the same grouped budget boundaries as keyed grouped plans.

### 🔧 Changed

- Added explicit implicit-singleton-group admission in grouped execution context for zero-key grouped global DISTINCT execution.
- Global `COUNT(DISTINCT field)` / `SUM(DISTINCT field)` zero-key execution now:
  - records one grouped state through grouped execution context before row fold
  - enforces total distinct cap through grouped budget counters
  - preserves fold order (`distinct insert -> record budget -> SUM fold`)

## 0.36.1 - 2026-03-01 - Grouped DISTINCT Aggregate Contracts

### 📝 Summary

`0.36.1` adds grouped DISTINCT aggregate contracts with conservative v1 eligibility and deterministic grouped fold behavior.

### 🔧 Changed - Grouped DISTINCT Aggregate Semantics

- Extended grouped aggregate plan contracts with an explicit `distinct` flag and added grouped fluent terminals:
  - `group_count_distinct()`
  - `group_min_distinct()`
  - `group_max_distinct()`
- Added grouped plan validation gates for DISTINCT aggregates:
  - reject unsupported kinds (`EXISTS`, `FIRST`, `LAST`)
  - reject DISTINCT field-target grouped terminals in grouped v1
- Wired grouped DISTINCT fold behavior in grouped aggregate state so repeated data-key contributions are deduplicated per group before reducer updates.
- Added grouped DISTINCT budget guardrails with typed runtime failures for:
  - distinct values per group
  - distinct values across one grouped query

### 🔧 Changed - Global DISTINCT Field Aggregates (`COUNT` + `SUM`)

- Added grouped global DISTINCT fluent terminals:
  - `group_count_distinct_by(field)`
  - `group_sum_distinct_by(field)`
- Added grouped validation gates for global DISTINCT (no GROUP BY keys):
  - require exactly one DISTINCT field-target aggregate
  - allow only `COUNT` and `SUM`
  - require numeric field targets for `SUM(DISTINCT field)`
  - reject mixed aggregate shapes without group keys
- Added grouped runtime execution for zero-key global DISTINCT field aggregates:
  - `COUNT(DISTINCT field)` counts first-seen canonical field values
  - `SUM(DISTINCT field)` folds only on first distinct insertion
  - distinct insertions enforce grouped per-group/total distinct budgets
- Added grouped cursor-policy hardening for this shape:
  - global DISTINCT grouped aggregates must not emit continuation cursors
  - continuation cursor inputs are explicitly rejected for this shape

### 🔧 Changed - Explain and Fingerprint Stability

- Added grouped aggregate DISTINCT projection to explain output.
- Extended grouped aggregate structural hashing with DISTINCT so grouped continuation signatures differ when aggregate DISTINCT modifiers change.

### 🔧 Changed - Streaming Eligibility Revalidation

- Added an explicit grouped ordered-streaming eligibility matrix at route feasibility, then revalidated planner ordered-group hints against:
  - access streaming safety
  - direction/reverse-traversal capability
  - grouped aggregate streaming compatibility
  - grouped HAVING operator compatibility
- Added route/explain coverage proving ordered strategy hints deterministically downgrade to hash grouping when eligibility conditions are violated.

### 🔧 Changed - Grouped Continuation Version + Shape Safety

- Updated grouped cursor decode validation to classify unsupported grouped token versions as typed `ContinuationCursorVersionMismatch` errors (parity with scalar cursor validation behavior).
- Added grouped cursor validation coverage proving cross-shape grouped resume is rejected when:
  - HAVING clause shape changes
  - grouped aggregate DISTINCT modifiers change

### 📚 Docs - Resource Contract

- Added and normalized [`docs/contracts/RESOURCE_MODEL.md`](../contracts/RESOURCE_MODEL.md) as the authoritative IC runtime resource model contract for grouped budgets, scan-budget boundaries, streaming eligibility, continuation invariants, and deterministic failure rules.

### 🔒 Contract Freeze - `0.36.x`

- Declared a formal `0.36.x` contract freeze across grouped external/runtime surfaces:
  - grouped projection schema surface
  - explain grouped output shape
  - grouped fingerprint hashing surface used by continuation signatures
  - grouped continuation token wire format/version behavior (`v1` grouped token shape)
  - grouped DISTINCT typed error surface
  - streaming-eligibility downgrade behavior (downgrade-only revalidation)
- `0.36.x` patch releases are constrained to defect fixes without semantic shape drift in these frozen surfaces; intentional surface changes move to `0.37+`.

## 0.36.0 - 2026-03-01 - Ordered Group Strategy Foundations

### 📝 Summary

`0.36.0` starts the grouped hardening line by formalizing grouped execution strategy selection (`HashGroup` vs `OrderedGroup`) across planning, route revalidation, observability, and explain/fingerprint surfaces.

Status tracking: [docs/design/archive/0.36-grouped-hardening/0.36-status.md](../design/archive/0.36-grouped-hardening/0.36-status.md)

### 🔧 Changed - Grouped Strategy Routing

- Added planner grouped strategy hinting so grouped plans explicitly encode ordered vs hash grouping intent.
- Added route-layer grouped strategy revalidation so ordered hints are downgraded to hash when traversal capabilities cannot preserve ordered-group invariants (for example, unsupported reverse traversal).
- Added grouped strategy fields in grouped route contracts and observability so grouped execution mode is explicit at the route boundary.

### 🔧 Changed - Explain/Fingerprint and Metrics Contracts

- Added grouped strategy to grouped explain output so ordered/hash grouped planning decisions are externally visible and deterministic.
- Added grouped strategy to grouped fingerprint hashing to keep semantic shape signatures stable across strategy changes.
- Updated grouped plan metrics mapping to key off grouped execution strategy (`OrderedStreaming` vs `HashMaterialized`) instead of generic execution mode.
- Removed test-only implicit plan deref coercions and switched grouped/scalar test callsites to explicit helper accessors so plan-shape boundaries remain visible in semantics tests.

### 🔧 Changed - HAVING Baseline (Conservative)

- Added explicit grouped HAVING contracts to grouped logical plans and query-to-executor handoff so HAVING shape is not implicit runtime state.
- Added grouped HAVING fluent/query entrypoints with conservative clause semantics:
  - grouped-key symbol references by field
  - grouped-aggregate symbol references by aggregate index
  - compare operators only (`Eq`, `Ne`, `Lt`, `Lte`, `Gt`, `Gte`)
- Added grouped HAVING validation gates:
  - reject HAVING without `GROUP BY`
  - reject grouped HAVING symbols that are not grouped keys or declared grouped aggregates
  - reject grouped `DISTINCT + HAVING` for now
  - reject unsupported HAVING compare operators
- Wired HAVING evaluation in grouped execution at the post-finalize stage (`fold -> finalize group -> HAVING -> emit`), before pagination/cursor emission, without adding a full grouped materialization pass.
- Extended explain and grouped continuation-signature hashing with HAVING shape to prevent cursor-signature collisions across different HAVING predicates.

### 🧪 Testing

- Added route aggregate coverage for ordered grouped strategy selection on index-prefix-aligned grouped shapes.
- Added explain coverage for hash-default vs ordered-eligible grouped strategy projection.
- Added grouped plan validation coverage for grouped HAVING symbol and operator gates.
- Added grouped execution coverage for HAVING filtering behavior and grouped continuation safety.
- Added grouped continuation-signature coverage so HAVING predicate changes produce distinct grouped signatures.
- Re-ran grouped route, explain, fingerprint, grouped plan, grouped intent, and grouped paged-builder suites to confirm parity and deterministic behavior.
