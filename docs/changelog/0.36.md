# IcyDB 0.36

This document aggregates detailed release notes for the `0.36.x` minor line.

---

## 0.36.0 - 2026-03-01 - Ordered Group Strategy Foundations

### 📝 Summary

`0.36.0` starts the grouped hardening line by formalizing grouped execution
strategy selection (`HashGroup` vs `OrderedGroup`) across planning, route
revalidation, observability, and explain/fingerprint surfaces.

Status tracking:
[docs/status/0.36-grouped-hardening-status.md](../status/0.36-grouped-hardening-status.md)

### 🔧 Changed - Grouped Strategy Routing

- Added planner grouped strategy hinting so grouped plans explicitly encode
  ordered vs hash grouping intent.
- Added route-layer grouped strategy revalidation so ordered hints are
  downgraded to hash when traversal capabilities cannot preserve ordered-group
  invariants (for example, unsupported reverse traversal).
- Added grouped strategy fields in grouped route contracts and observability so
  grouped execution mode is explicit at the route boundary.

### 🔧 Changed - Explain/Fingerprint and Metrics Contracts

- Added grouped strategy to grouped explain output so ordered/hash grouped
  planning decisions are externally visible and deterministic.
- Added grouped strategy to grouped fingerprint hashing to keep semantic shape
  signatures stable across strategy changes.
- Updated grouped plan metrics mapping to key off grouped execution strategy
  (`OrderedStreaming` vs `HashMaterialized`) instead of generic execution mode.
- Removed test-only implicit plan deref coercions and switched grouped/scalar
  test callsites to explicit helper accessors so plan-shape boundaries remain
  visible in semantics tests.

### 🔧 Changed - HAVING Baseline (Conservative)

- Added explicit grouped HAVING contracts to grouped logical plans and
  query-to-executor handoff so HAVING shape is not implicit runtime state.
- Added grouped HAVING fluent/query entrypoints with conservative clause
  semantics:
  - grouped-key symbol references by field
  - grouped-aggregate symbol references by aggregate index
  - compare operators only (`Eq`, `Ne`, `Lt`, `Lte`, `Gt`, `Gte`)
- Added grouped HAVING validation gates:
  - reject HAVING without `GROUP BY`
  - reject grouped HAVING symbols that are not grouped keys or declared grouped
    aggregates
  - reject grouped `DISTINCT + HAVING` for now
  - reject unsupported HAVING compare operators
- Wired HAVING evaluation in grouped execution at the post-finalize stage
  (`fold -> finalize group -> HAVING -> emit`), before pagination/cursor
  emission, without adding a full grouped materialization pass.
- Extended explain and grouped continuation-signature hashing with HAVING shape
  to prevent cursor-signature collisions across different HAVING predicates.

### 🧪 Testing

- Added route aggregate coverage for ordered grouped strategy selection on
  index-prefix-aligned grouped shapes.
- Added explain coverage for hash-default vs ordered-eligible grouped strategy
  projection.
- Added grouped plan validation coverage for grouped HAVING symbol and operator
  gates.
- Added grouped execution coverage for HAVING filtering behavior and grouped
  continuation safety.
- Added grouped continuation-signature coverage so HAVING predicate changes
  produce distinct grouped signatures.
- Re-ran grouped route, explain, fingerprint, grouped plan, grouped intent, and
  grouped paged-builder suites to confirm parity and deterministic behavior.
