# 0.39 Changelog

## 0.39.3 - 2026-03-02 - Numeric Identity Drift Guards

### Summary

`0.39.3` hardens numeric identity safety by adding explicit continuation and
fingerprint regression guards for numeric projection semantics. Alias-only
numeric projection changes remain identity-stable, while numeric semantic
changes invalidate identity as required.

### Changed

- Added continuation-signature guards proving numeric projection alias wrapping
  is identity-neutral.
- Added continuation-signature guards proving numeric projection semantic
  changes invalidate continuation identity.
- Added continuation decode/resume stability coverage for alias-only numeric
  projection renames (token decode + grouped resume validation remains stable).
- Added plan-fingerprint guards proving numeric projection alias wrapping is
  identity-neutral.
- Added plan-fingerprint guards proving numeric projection semantic changes
  invalidate fingerprint identity.
- Added literal canonicalization guards:
  - decimal scale-only literal changes are identity-stable
  - top-level literal subtype remains identity-significant when observable
- Added numeric promotion drift guards:
  - `1 + 2`, `1 + 2.0`, and `1.0 + 2` remain identity-stable
- Added aggregate and DISTINCT numeric stability guards:
  - aggregate target-field semantic changes invalidate identity
  - aggregate/DISTINCT numeric no-op promotion path differences remain
    identity-stable
- Locked these guards against regressions in dedicated fingerprint and
  continuation test suites.

## 0.39.2 - 2026-03-02 - Numeric Domain Boundary Hardening

### Summary

`0.39.2` closes numeric semantic drift across runtime surfaces. Arithmetic,
predicate/HAVING/range comparison, and aggregate numeric fold behavior now use
the shared numeric authorities, with explicit strict index-pushdown boundaries
and invariant tests for overflow and rounding behavior.

### Changed

- Added a shared numeric arithmetic contract in `db::numeric` with explicit operator taxonomy and runtime documentation for promotion/coercion boundaries.
- Defined runtime promotion semantics as decimal-domain execution: numeric operands are coerced into decimal first, then arithmetic is applied under decimal rules.
- Routed projection numeric arithmetic through the shared numeric arithmetic contract instead of local per-operator arithmetic logic.
- Updated grouped `HAVING` compare evaluation to use numeric-widen coercion for numeric values, while keeping fail-closed behavior for numeric-vs-non-numeric mismatches.
- Replaced planner range-merge numeric variant gating with shared numeric comparison authority, removing ad hoc discriminant-based numeric checks.
- Tightened planner range bound compatibility to fail closed when bound values are not comparable under shared numeric or strict same-variant ordering semantics.
- Clarified and locked strict index pushdown boundary behavior through existing + refreshed tests: non-strict numeric-widen predicates do not compile into index-only programs or index-range access paths.
- Routed scalar field-target aggregate accumulation (`sum`) and average division (`avg`) through the shared numeric arithmetic contract.
- Routed grouped global `SUM(DISTINCT field)` accumulation through the shared numeric arithmetic contract so grouped distinct numeric fold semantics match scalar aggregate and projection arithmetic semantics.
- Locked key numeric-boundary invariants with focused tests:
  - integer + decimal widening behavior
  - negative division rounding behavior
  - overflow saturation behavior for large values
- Preserved fail-closed behavior when operands cannot enter numeric coercion domain.

## 0.39.1 - 2026-03-02 - Runtime Numeric Convergence Start

### Summary

`0.39.1` begins runtime numeric convergence. Scalar and grouped projection expression evaluation now use the same numeric decimal coercion entrypoint, and aggregate numeric extraction uses that same helper too. This closes a drift risk where numeric arithmetic and numeric comparison could take different coercion paths.

### Changed

- Added a shared numeric decimal coercion helper under `db::numeric` and reused it across runtime numeric evaluation paths.
- Updated scalar and grouped projection arithmetic (`neg`, `add`, `sub`, `mul`, `div`) to use the shared coercion helper.
- Updated runtime numeric equality handling so if either side enters numeric coercion domain, comparison must succeed under numeric rules or fail as an invariant error (instead of silently falling back to plain equality).
- Updated aggregate numeric field extraction to use the same shared decimal coercion helper used by projection evaluation.
- Added runtime regression coverage for scalar/grouped numeric equality widening and scalar/grouped mixed numeric-vs-non-numeric equality rejection.

## 0.39.0 - 2026-03-02 - Numeric Consolidation

### Summary

`0.39.0` starts numeric consolidation by moving numeric capability checks to shared helpers and tightening planner numeric typing. Queries that try numeric-only operations on known non-numeric fields now fail at planning time, while valid mixed numeric expressions continue to work even when the exact numeric subtype is still unresolved.

### Changed

- Added a shared `db::numeric` helper boundary for numeric field-kind eligibility and numeric comparison behavior so planner, predicate semantics, and executor paths use the same core rules.
- Updated planner expression typing to separate numeric eligibility from numeric subtype (`Integer`, `Float`, `Decimal`, `Unknown`) so unresolved subtype does not automatically become permissive behavior.
- Tightened numeric operator inference to reject decidably non-numeric operands (for example `text`, `bool`, and `date` fields) instead of letting them pass through `Unknown`.
- Tightened numeric aggregate inference so `sum(...)` requires an explicit numeric target and fails fast for non-numeric targets; non-sum aggregate semantics remain unchanged for this slice.
- Added focused planner expression tests for fail-closed numeric gating, mixed-subtype numeric acceptance, aggregate target validation, and unknown non-eligible operand rejection.
