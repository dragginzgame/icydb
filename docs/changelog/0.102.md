# 0.102 Changelog

## 0.102.1

- `0.102.1` keeps the `0.102` line on optimizer-quality recovery instead of widening semantic `WHERE` admission again. The main change is a bounded normalization follow-through in the SQL `WHERE` lowering path:
  - literal-only scalar helper subtrees now fold before the conservative derived-predicate gate runs
  - mixed boolean trees then simplify constant boolean branches like `... AND TRUE`, `... AND FALSE`, `... OR FALSE`, and `... OR TRUE`
  - if the surviving expression is back on an already-supported strict or text-casefold compare family, the planner now reclaims the older derived predicate lane instead of leaving the whole shape on residual execution
- The recovered families explicitly pinned in this patch are:
  - strict compare recovery from wrapped constants:
    - `name = TRIM('alpha')`
  - text-casefold compare recovery from wrapped constants:
    - `LOWER(name) = TRIM('ALPHA')`
  - strict text-predicate recovery from wrapped constants:
    - `STARTS_WITH(name, TRIM('Al'))`
  - text-casefold starts-with recovery from wrapped constants:
    - `STARTS_WITH(LOWER(name), TRIM('AL'))`
  - boolean-tree recovery around those same shapes:
    - `... AND NULLIF('alpha', 'alpha') IS NULL`
    - `... AND NULLIF('alpha', 'alpha') IS NOT NULL`
    - `... OR NULLIF('alpha', 'alpha') IS NOT NULL`
    - `... OR NULLIF('alpha', 'alpha') IS NULL`
- The storage behavior stays intentionally explicit rather than pretending every folded boolean becomes a new predicate family:
  - folded `FALSE` continues to store `predicate=False`
  - folded `TRUE` continues to store `predicate=None`
  - recovered strict/casefold compare families store the corresponding `Predicate::Compare(...)`
- This patch also follows the recovered lanes through the other already-shipped `0.100` / `0.102` runtime and diagnostics surfaces so the new behavior is not scalar-lowering-only:
  - scalar `EXPLAIN`
  - grouped pre-aggregate `WHERE` runtime
  - grouped `EXPLAIN`
  - delete runtime
  - delete `EXPLAIN`

```sql
SELECT name
FROM SessionSqlEntity
WHERE STARTS_WITH(LOWER(name), TRIM('AL'))
  OR NULLIF('alpha', 'alpha') IS NOT NULL
ORDER BY age ASC;
```

## 0.102.0

- `0.102.0` starts the expression-complete scalar `WHERE` admission line by replacing more of the older clause-local operand whitelist with a capability-driven admission rule. The boundary is still conservative:
  - semantic `WHERE` admission widens when the shared planner/runtime `Expr` seam can already type-check and execute the subtree
  - derived predicate extraction stays sound, not maximal
  - unsupported extraction falls back to residual execution instead of becoming an admission failure
- The newly admitted families in this patch are the scalar expression trees that already existed elsewhere in projection / grouped / aggregate paths but were still narrower in `WHERE`. Concretely, `0.102.0` widens:
  - wrapped text transform operands such as:
    - `REPLACE(name, 'a', 'A') = 'AlphA'`
  - wrapped text-predicate targets such as:
    - `STARTS_WITH(REPLACE(name, 'a', 'A'), 'Al')`
  - non-literal text-predicate arguments such as:
    - `STARTS_WITH(REPLACE(name, 'a', 'A'), TRIM('Al'))`
  - wrapped `LIKE` / `ILIKE` targets such as:
    - `REPLACE(name, 'a', 'A') LIKE 'Al%'`
    - `REPLACE(name, 'a', 'A') ILIKE 'al%'`
- The runtime / diagnostics follow-through for those widened shapes is part of the shipped contract in `0.102.0`, not a later cleanup:
  - scalar read execution
  - grouped pre-aggregate `WHERE`
  - delete execution
  - scalar / grouped / delete `EXPLAIN`
- This patch does **not** widen pushdown or prepared-template policy. Many newly admitted trees remain `filter_expr`-owned with `predicate=None`, which is the intended result when semantic admission outruns the narrower derived predicate families.

```sql
DELETE FROM SessionSqlEntity
WHERE STARTS_WITH(REPLACE(name, 'a', 'A'), TRIM('Al'))
ORDER BY age ASC
LIMIT 1;
```
