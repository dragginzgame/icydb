# IcyDB 0.195 Changelog

## 0.195.2

### 📝 Summary

- Adds cached-plan reuse evidence for the deferred access physicalization line.
- Proves a large sparse indexed `IN (...)` read can reuse the same shared
  prepared read plan while execution-time branch liveness sees a branch move
  from empty, to live after insert, and back to empty after delete.
- Keeps the slice test-only: no query semantics, public API, cursor format,
  persisted format, diagnostic code, route contract, cache identity, recovery
  behavior, successful result shape, or production runtime changes.

### 🧪 Testing

- Added a session-level large sparse `IN (...)` regression that:
  - warms one shared read-plan cache entry while the late branch is empty;
  - inserts a row into that previously empty branch and proves the cached plan
    returns it;
  - deletes that row through the normal delete executor and proves the same
    cached read plan prunes the branch again;
  - checks raw-bound/range-scan evidence remains below the planned missing
    branch count.
- Re-ran the focused SQL+diagnostics session test.
- Re-ran `icydb-core` all-target clippy for the SQL+diagnostics feature
  surface with warnings denied.

### 📊 Performance

- No runtime performance delta is claimed for this slice. It changes only
  regression evidence around cached-plan correctness.

### 📊 Complexity Delta

- Code footprint: one test file changed (`+163` lines).
- Runtime complexity judgment: unchanged. The production execution path is not
  modified; the added complexity is test-only evidence for the existing
  execution-time liveness contract.

## 0.195.1

### 📝 Summary

- Continues the 0.195 deferred access physicalization line with an executor
  liveness-contract cleanup.
- Replaces implicit boolean empty-prefix pruning with an explicit tri-state
  branch liveness contract: `ProvenEmpty`, `PossiblyLive`, and
  `UnknownConservative`.
- Keeps runtime prefix pruning conservative: only generation-compatible
  `ProvenEmpty` branches are skipped, while missing, stale, or non-user
  metadata scans through the existing fallback path.
- Keeps the slice internal: no query semantics, public API, cursor format,
  persisted format, diagnostic code, route contract, cache identity, recovery
  behavior, or successful result shape changes.

### 🔧 Changed

- `index_prefix_cardinality` now returns an executor-owned
  `IndexBranchLiveness` value instead of exposing a boolean
  "is proven empty" helper.
- Single-prefix and multi-prefix scan paths now consume the same
  `should_scan()` liveness decision.
- Added focused tests for synchronized live prefixes, synchronized empty
  prefixes, generation mismatch, and unsupported/non-user prefix metadata.

### 📊 Performance

- The full deterministic SQL matrix stayed flat against the 0.195.0 matrix:
  1,675 successful scenarios, 81 generated failures, and the same
  success/failure key set.
- Aggregate total moved from 4,850,085,549 to 4,858,143,024 local
  instructions (`+8,057,475`, `+0.17%`), which is treated as noise-level for
  this code-shape cleanup.
- No notable total regressions were found using the `>= +50k` and `>= +5%`
  threshold; no material regressions were found using the `>= +100k` and
  `>= +10%` threshold.
- The sparse indexed `IN (...)` page target remained in the same performance
  band after the liveness cleanup: 14,110,448 to 14,145,853 local instructions
  (`+35,405`, `+0.25%`), with range scans still at `12` and entries read still
  at `74`.

### 📊 Complexity Delta

- Code footprint: five tracked implementation/changelog files changed for this
  slice (`+187/-27`) before release metadata/user-version churn.
- Runtime complexity judgment: clearer and safer. The liveness decision now
  carries why a branch can be skipped or why it must be conservatively scanned,
  instead of overloading `false` to mean both "possibly live" and "unknown."
- The runtime work performed by scan callers stays neutral: callers still make
  one liveness decision per candidate prefix and only skip branches proven
  empty.

### 🧪 Testing

- Added executor liveness tests for `ProvenEmpty`, `PossiblyLive`, and
  `UnknownConservative` outcomes.
- Re-ran the sparse multi-lookup deferred-bound pagination witness.
- Re-ran `icydb-core` compile checks, all-target/all-feature clippy with
  warnings denied, and the supported no-default core check.
- Re-ran the full deterministic PocketIC SQL performance matrix and compared
  it against the 0.195.0 matrix.

## 0.195.0

### 📝 Summary

- Starts the 0.195 deferred access physicalization line with a narrow
  indexed multi-lookup page-query optimization.
- Keeps large exact-prefix multi-lookup branches logical until execution-time
  empty-prefix pruning has selected the retained branches.
- Materializes raw index scan bounds only for retained branches, while small
  multi-lookups stay on the existing eager path.
- Keeps the slice internal: no query semantics, public API, cursor format,
  persisted format, diagnostic code, route contract, cache identity, recovery
  behavior, or successful result shape changes.

### 🔧 Changed

- `LoweredIndexPrefixSpec` can now carry either materialized raw prefix bounds
  or a deferred exact-prefix source that can materialize those bounds on demand.
- Large single-field indexed `IN (...)` / multi-lookup plans defer raw bound
  construction behind a 32-value activation threshold.
- Index prefix cardinality can read the deferred branch identity directly, so
  execution-time liveness pruning does not force the raw bounds it is trying to
  avoid.
- Covering, materialized scan, streaming scan, and snapshot-rendering callers
  now request raw prefix bounds through the fallible access seam.
- Added a focused design note for the 0.195 line at
  `docs/design/0.195-deferred-access-physicalization/0.195-design.md`.

### 📊 Performance

- Sparse indexed `IN (...)` page total dropped from 17,520,748 to 14,110,448
  local instructions (`-3,410,300`, `-19.5%`) against the 0.194.15 full-matrix
  baseline for `token.collection_id.sparse_in.page_only.limit50`.
- The execute bucket dropped from 15,272,977 to 11,875,877 local instructions
  (`-3,397,100`, `-22.2%`).
- The planner bucket dropped from 10,058,403 to 6,638,233 local instructions
  (`-3,420,170`, `-34.0%`).
- The plan-build bucket dropped from 8,107,470 to 4,691,567 local instructions
  (`-3,415,903`, `-42.1%`).
- Physical execution stayed intentionally stable: range scans remained at `12`
  and entries read remained at `74`.
- Store and executor buckets were effectively flat to slightly higher
  (`+1,482` store, `+21,588` executor), which is small compared with the
  planner-side saving and did not change the physical scan shape.

### 📊 Complexity Delta

- Code footprint: ten tracked files changed for the implementation slice
  (`+424/-94`), plus the new 439-line 0.195 design document and this
  changelog.
- Runtime complexity judgment: moderately higher but localized. Access lowering
  now owns a deferred-versus-materialized prefix-bound state, while downstream
  scan callers continue to consume a single `raw_bounds()` seam.
- The slice deliberately keeps small multi-lookups on the eager path to avoid
  adding lazy-bound overhead to common compact queries.
- No persisted bytes, cursor bytes, public API names, or route/cache identity
  surfaces were changed.

### 🧪 Testing

- Added access-lowering tests proving small multi-lookups remain eager and
  large multi-lookups defer raw prefix-bound materialization until requested.
- Extended the sparse multi-lookup pagination witness to prove only the retained
  live branch materializes raw bounds after empty-prefix pruning.
- Re-ran focused access and pagination tests for the deferred-bound path.
- Re-ran `icydb-core` compile checks, all-target/all-feature clippy with
  warnings denied, and the supported no-default core check.
- Re-ran the focused PocketIC SQL performance scenario for sparse indexed
  `IN (...)` page queries. The full SQL performance matrix was not rerun for
  this first slice.
