# 0.93 Changelog

## 0.93.0

- `0.93.0` starts the shared execution cost-reduction line after the `0.92` cache cleanup, with the first concrete cut aimed at redundant prepared-plan rebuilds around the shared lower boundary.
- the first shipped cleanup in this line removes a few obvious “prepared plan -> compiled wrapper -> prepared plan again” hops:
  - typed delete count execution in `db/session/query.rs` now takes the cached `PreparedExecutionPlan<E>` directly instead of rebuilding it through `CompiledQuery<E>`
  - typed SQL delete projection execution in `db/session/sql/execute/write.rs` now does the same
  - fluent explain helpers for `bytes_by(field)` and prepared projection terminals in `db/query/intent/query.rs` now build one `PreparedExecutionPlan<E>` directly for the requested visibility lane instead of routing through `plan_with_visible_indexes(...).into_prepared_execution_plan()`
  - `trace_query(...)` now reads explain/hash/access-family metadata from the cached prepared plan directly instead of combining a compiled-query build with a second prepared-plan lookup
- the same slice also trims the larger typed-wrapper duplication that was still sitting on top of the shared cache:
  - `CompiledQuery<E>` and `PlannedQuery<E>` can now reuse the shared prepared-plan artifact directly through an internal `QueryPlanHandle`
  - session-owned `compile_query_with_visible_indexes(...)` and `planned_query_with_visible_indexes(...)` no longer clone logical-plan state back out of the shared prepared plan just to wrap it as another typed shell
  - direct builder-owned `Query::plan()` / `Query::planned()` behavior stays intact, so the public fluent planning surface still looks the same while the session-owned hot path gets cheaper
- the prepared-plan constructors were also doing one duplicate piece of work on every build:
  - `SharedPreparedExecutionPlan::from_plan(...)` and `PreparedExecutionPlan<E>::build(...)` both froze the planner-owned static execution shape before calling the shared prepared-plan core builder
  - `build_prepared_execution_plan_core(...)` already performs that same freeze internally
  - `0.93.0` removes the outer duplicate pass and keeps one canonical static-shape freeze in the shared builder
- the later `0.93.0` follow-through trims two more shared-floor costs that were still hidden behind otherwise-correct APIs:
  - the session-owned `compile_query_with_visible_indexes(...)` / `planned_query_with_visible_indexes(...)` path was still cloning `SharedPreparedExecutionPlan` twice, once when it came out of the shared cache and again when the typed wrapper constructors borrowed and recloned it; those typed wrapper constructors now take ownership directly on the shared-cache path
  - `PreparedExecutionPlanCore::into_inner()` and aggregate streaming extraction were still cloning the full shared prepared-plan payload even when the core was uniquely owned; that boundary now uses `Arc::try_unwrap(...)` to move the canonical plan/spec payload out directly and falls back to cloning only when another wrapper still holds the shared core
- that means the current `0.93.0` shared-floor model is tighter in three places:
  - fewer rebuilds from shared prepared plan back into typed wrappers
  - fewer duplicate constructor/freeze passes during prepared-plan assembly
  - fewer whole-plan / whole-spec clones when execution consumes a uniquely owned prepared core
- the first benchmark readout for this line is already positive on both surfaces that meet at the shared floor:
  - fluent sampled rows improved again versus their checked-in baselines, including `user.age.order_only.asc.limit3` (`-22.34%`), `repeat.user.age.order_only.asc.limit3.runs10` (`-38.43%`), and `repeat.user.grouped.age_count.limit10.runs100` (`-32.47%`)
  - SQL sampled rows also improved again versus their checked-in baselines, including `repeat.user.pk.order_only.asc.limit2.runs10` (`-22.56%`), `repeat.user.name.lower.order_only.asc.limit3.runs10` (`-51.78%`), and `repeat.user.grouped.age_count.limit10.runs100` (`-15.73%`)
  - the simplified `0.92` cache contract is unchanged during these `0.93.0` cuts: SQL repeat rows still show compiled-command cache hits plus shared-query-plan-cache hits, and fluent repeat rows still reuse only the shared lower prepared-plan cache
- the later `0.93.0` SQL membership follow-through made the remaining compile-side outlier measurable instead of speculative:
  - SQL query attribution now exposes compile subphases for cache-key construction, cache lookup, parsing, aggregate-lane classification, prepared-statement normalization, lowering, structural binding, and cache insertion
  - the focused `IN` / `NOT IN` PocketIC audit showed that the extra compile budget was not a `NOT IN`-specific runtime regression; it was mostly parser + prepare + lower work, plus a fixed aggregate-lane classification cost on plain scalar `SELECT`s
  - that follow-through also adds one cheap parsed-shape fast path before the dedicated global-aggregate lane, so scalar `SELECT`s with no `HAVING` and no aggregate projection items stop paying the heavier aggregate-lane analysis on the compile path
- the next `0.93.0` follow-through hard-cuts the remaining plain membership compile waste on the scalar SQL path:
  - `prepare_select_statement(...)` now short-circuits the already-local scalar `SELECT` family, so field-list projections with local predicates and bare field `ORDER BY` terms no longer rebuild the parsed statement just to discover there is nothing to re-scope or alias-normalize
  - `compile_where_bool_expr_to_predicate(...)` now collapses top-level canonical `OR(eq...)` and `AND(ne...)` chains directly onto one runtime `IN` / `NOT IN` compare instead of expanding those shapes through the generic boolean truth-set compiler and leaving the membership collapse to later predicate normalization
  - together, those two follow-through cuts moved the focused membership compile path materially:
    - `user.age.in.limit3` compile dropped from `253,802` after the aggregate-lane fast path to `199,037`, and total cost dropped to `1,442,231` (`-16.33%` vs the checked-in baseline)
    - `user.age.not_in.limit3` compile dropped from `249,269` after the aggregate-lane fast path to `201,292`, and total cost dropped to `952,839` (`-21.64%` vs the checked-in baseline)
  - the focused subphase probe now shows the remaining membership compile budget concentrated mostly in parser work, not in aggregate-lane classification or SQL statement normalization:
    - `parse` remains about `79k` for `IN` and `85k` for `NOT IN`
    - `prepare` is now about `22.5k`
    - `lower` is now about `38.9k`
    - `bind` is now about `19.2k` for `IN` and `15.8k` for `NOT IN`
- the final `0.93.0` membership follow-through moves the last large SQL-side cost off the parser/lowering front door instead of leaving the fast path to repair it later:
  - the parser now owns a direct `SqlExpr::Membership` node for `IN` / `NOT IN` instead of immediately expanding membership into `OR(eq...)` or `AND(ne...)` boolean chains
  - statement alias normalization, aggregate-call discovery, general SQL-expression lowering, and `WHERE` predicate lowering all understand that narrower membership node directly, so the membership path no longer has to allocate and walk the larger boolean expression tree before collapsing back to one runtime compare
  - that parser-side cut materially reduced the remaining compile buckets on the focused membership probe:
    - `user.age.in.limit3` moved from `compile=199,037`, `parse=79,625`, `prepare=22,508`, `lower=38,864`, `total=1,442,231`
      to `compile=153,371`, `parse=67,907`, `prepare=14,801`, `lower=14,104`, `total=1,400,677`
    - `user.age.not_in.limit3` moved from `compile=201,292`, `parse=85,420`, `prepare=22,664`, `lower=38,864`, `total=952,839`
      to `compile=152,445`, `parse=72,978`, `prepare=13,827`, `lower=14,275`, `total=903,949`
  - relative to the checked-in SQL perf baseline, the final `0.93.0` membership rows now land at:
    - `user.age.in.limit3`: `1,723,754 -> 1,400,677` (`-18.74%`)
    - `user.age.not_in.limit3`: `1,216,013 -> 903,949` (`-25.66%`)
- this is intentionally a narrow first slice:
  - no cache model changes
  - no new SQL-specific logic
  - no new runtime owner
