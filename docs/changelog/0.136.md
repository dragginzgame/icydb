# 0.136.x

## 0.136.1

`0.136.1` tightens the `StructuralPatch` write boundary after the hard rename
by making user-authored structural patches fail through the same executor-owned
validation taxonomy as typed writes.

What changed:
- structural patch values now run the executor-owned decimal-scale and
  bounded-text write-boundary checks before persisted-row serialization
- over-length text in structural writes now reports the same executor
  `Unsupported` error shape as over-length text in typed writes, instead of
  surfacing as an internal persisted-row encode failure
- added regression coverage proving structural bounded-text rejection happens
  before persistence
- added regression coverage proving structural replace materializes sparse
  after-images and does not inherit omitted nullable fields from an existing row
- cleaned up stale `UpdatePatch` wording in the `0.136` design doc after the
  hard rename

Retained behavior:
- `StructuralPatch -> SerializedStructuralPatch -> MutationInput` remains the
  persisted-row handoff
- `MutationMode::Insert`, `Update`, and `Replace` semantics are unchanged
- SQL writes still lower into the same structural patch lane
- persisted-row bytes, cursor behavior, and public response shapes are
  unchanged

## 0.136.0

`0.136.0` starts the `StructuralPatch` boundary closeout by making structural
mutation request ownership explicit inside the save executor and restores typed
generated-enum values as fluent filter literals.

What changed:
- added an internal `StructuralMutationRequest` bundle for structural write
  execution
- replaced the previous `Option<&StructuralPatch>` generated-field policy signal
  with an explicit `StructuralPatchOrigin`
- kept public-authored structural patches rejecting explicit generated fields
  while preserving internally lowered SQL write behavior
- removed the now-unused structural write-context helper that existed only to
  bridge the old loose argument shape
- hard-renamed the public and core structural patch types:
  `UpdatePatch -> StructuralPatch`, `FieldUpdate -> StructuralFieldUpdate`, and
  `SerializedUpdatePatch -> SerializedStructuralPatch`, with no compatibility
  aliases
- generated schema enums now implement `From<Enum>` and `From<&Enum>` for
  `InputValue`, restoring `FilterExpr::eq("stage", Stage::Live)` style
  filters without broadening filter literals to arbitrary `Display` values
- generated schema enums now expose variant-name constants such as
  `Stage::LIVE`, which the generated schema/runtime/persisted-codec paths reuse
  instead of repeating the same variant strings in IcyDB-owned code
- generated enum constants intentionally use the plain upper-snake variant name
  when possible (`TEXT`, `NAT`, `LIVE`), with a collision-safe fallback for
  variants whose names would shadow their enum constructors

Retained behavior:
- `MutationMode` semantics are unchanged; the patch type was renamed to
  `StructuralPatch`
- SQL writes still lower into the structural patch lane
- `StructuralPatch -> SerializedStructuralPatch -> MutationInput` remains the
  persisted-row handoff
- typed full-entity saves and typed create authored-slot provenance are
  unchanged
- enum filter literals still lower through the existing runtime `Value::Enum`
  bridge, preserving variant path and payload semantics
