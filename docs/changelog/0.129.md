## 0.129.1

small duplicated traversal, moving secondary-order pushdown surface labels onto
the access-owned pushdown DTOs, and deleting the thin `AccessStrategy` wrapper
around executable access plans. Runtime route behavior, SQL semantics, cursor
format, and pushdown eligibility are unchanged.

What changed:
- collapsed executable-only access lowering and full prepared access lowering
  onto one shared access-tree traversal
- replaced the collector trait/no-op/raw collector scaffolding with one small
  optional index-spec sink inside lowering
- reduced the touched access/executor/query/session slice by 57 net lines
  after the wrapper and collector cleanup, even with the new pushdown DTO file
  included in that slice
- moved secondary-order pushdown surface projection onto
  `PushdownApplicability` / `SecondaryOrderPushdownEligibility`
- moved stable rejection diagnostic labels onto
  `SecondaryOrderPushdownRejection`
- retargeted executor explain descriptors and verbose route diagnostics to
  consume those access-owned projection helpers instead of matching every
  rejection variant locally
- carried residual-filter presence on `RouteCapabilities` so bounded
  index-range pushdown hinting consumes the already-derived route snapshot
  instead of re-walking the plan for load facts
- tightened the intermediate load capability fact snapshot back to private
  route-capability ownership
- removed `AccessPlan::resolve_strategy()` so `plan.rs` no longer imports the
  execution-contract strategy owner
- deleted `access/execution_contract/strategy.rs`
- retargeted executor and query-planning consumers to use
  `lower_executable_access_plan(...)` plus `ExecutableAccessPlan`'s existing
  `capabilities()`, `as_path()`, and `metrics_kind()` projections directly
- re-exported `summarize_executable_access_plan(...)` for trace/debug surfaces
  that still need the stable human-readable access summary

Retained boundaries:
- `executor/planning/route/pushdown.rs` still computes secondary-order
  eligibility from access capabilities plus the planner-owned deterministic
  order contract; `access/order_pushdown.rs` owns the DTOs and surface labels,
  not route policy derivation
- executor explain still maps pushdown applicability to local fast-path reason
  codes, but those codes are presentation-specific and consume
  `PushdownApplicability` instead of matching rejection variants
- physical executor and scan modules keep fail-closed shape checks separate
  from planner/access eligibility

## 0.129.0

`0.129.0` opens the access execution-contract cleanup line by removing the
remaining weak scaffolding around semantic access plans, executable access
paths, and route-facing capability facts. It also fixes an optional by-kind
decimal null regression uncovered by generated nullable decimal fields. This
patch does not change SQL semantics, cursor formats, order-pushdown
eligibility, raw index bounds, or physical execution behavior. It makes the
access boundary read as a smaller linear handoff: semantic `AccessPlan` /
`AccessPath`, bundled lowering output, one executable path payload, and one
capability snapshot.

What changed:
- removed `AccessPlanKeyAdapter` and `ValueKeyAdapter`, leaving validation
  directly on the only live `AccessPlan<Value>` / `AccessPath<Value>` path
- removed the tuple-shaped `IndexRangePathRef` alias and made index-range
  callers borrow `SemanticIndexRangeSpec` directly
- removed unused cached execution fields from `ExecutableAccessPlan`
- moved route/terminal capability helpers off `AccessPathKind`
- flattened `SinglePathAccessCapabilities` so it owns the route-facing booleans
  directly instead of routing through nested stream/pushdown helper structs
- added `AccessCapabilities::single_path_capabilities(...)` as the consumer
  boundary for single-path facts
- removed the free `single_path_capabilities(...)` helper from the access
  module export surface
- collapsed `ExecutionBounds` into richer `ExecutionPathPayload` variants, so
  one executable access path carries one physical leaf payload instead of
  coordinated `bounds` and `payload` fields
- deleted `AccessPlanDispatch` and `dispatch_access_plan(...)`; access-owned
  traversals now match `AccessPlan` directly while leaf payload projection still
  goes through `AccessPathDispatch`
- retargeted bytes terminal, count terminal, scalar top-N hinting, numeric
  aggregate eligibility, diagnostics, and query cache-key projection onto the
  smaller access contract
- made explicit by-kind `NULL` encode through the canonical structural null
  sentinel before any field-kind leaf encoder runs
- made by-kind decode recognize that null sentinel before dispatching to
  storage-key or structural leaf decoders, so nullable `Decimal` fields decode
  as absent instead of asking the decimal leaf decoder to accept `Value::Null`
- added persisted-row regressions for nullable decimal runtime slots and typed
  `Option<Decimal>` by-kind payloads

That leaves the opening `0.129` boundary clearer:
- `dispatch.rs` owns structural access variant projection only
- `capabilities.rs` owns route-facing capability facts
- `lowering.rs` owns semantic-to-executable traversal and bundled raw index
  spec collection
- `ExecutionPathPayload` owns physical executable leaf shape
- executor consumers read from `AccessStrategy::capabilities()`
- physical executor fail-closed checks remain separate from planner/access
  eligibility decisions
