# 0.107 Changelog

## 0.107.0

- `0.107.0` starts the bounded semantic canonicalization slice by introducing the first planner-owned semantic rewrite in scalar `WHERE`: boolean searched `CASE` now canonicalizes onto the explicit first-match boolean form before the existing `0.105` normalization and extraction flow continues.
- The implementation lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs). `normalize_bool_expr(...)` now routes boolean `Expr::Case` through one bounded first-match expansion instead of preserving the raw `CASE` spelling all the way into residual/explain identity surfaces.
- This remains a planner-`Expr` canonicalization rule, not a SQL parser or lowering rewrite policy expansion. The SQL front door still admits the same searched-`CASE` forms it already admitted; the new behavior starts only after validation and planner-owned expression construction.
- The shipped `0.107.0` rule is intentionally narrow:
  - only boolean searched `CASE` in scalar `WHERE`
  - only the exact first-match expansion
  - bounded by an explicit arm-count threshold
  - no general searched-`CASE` simplification
  - no arithmetic or non-boolean rewrite classes
- The branch builder also collapses the trivial guarded boolean cases that fall straight out of first-match expansion:
  - `guard AND TRUE` becomes `guard`
  - `guard AND FALSE` becomes `FALSE`
  This keeps the canonical form from depending on later cleanup passes for the obvious boolean identities introduced by searched-`CASE` expansion itself.
- The important behavioral outcome of this patch is that the old searched-`CASE` exception is gone. Equivalent searched-`CASE` and explicit boolean first-match forms now line up on the same explain surface in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs) instead of requiring routed-shape-only comparison.
- The lowering-side parity proof in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs) still intentionally ignores `filter_expr` for the fluent-vs-SQL comparison because that older SQL-vs-fluent literal-coercion difference is not a `0.107` semantic-canonicalization issue. This patch removes the searched-`CASE` representation drift without claiming to solve that separate coercion parity seam.

```sql
EXPLAIN
SELECT name
FROM SessionSqlEntity
WHERE CASE WHEN age >= 30 THEN TRUE ELSE age = 20 END
ORDER BY age ASC;
```
