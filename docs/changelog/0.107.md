# 0.107 Changelog

## 0.107.0

- `0.107.0` starts the bounded semantic canonicalization slice by introducing the first planner-owned semantic rewrite for scalar `WHERE`: boolean searched `CASE` now canonicalizes onto an explicit first-match boolean form before the existing `0.105` normalization and extraction flow continues.
- The shipped boundary is narrower than the first implementation cut. The canonicalization now lives on the scalar-`WHERE` seam in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/mod.rs) and [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/normalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/normalize.rs), not in the shared structural boolean normalizer for every planner-owned boolean `Expr`.
- That scope correction matters because the first cut leaked searched-`CASE` rewriting into grouped `WHERE` and other generic boolean-expression surfaces. The final `0.107.0` rule keeps grouped explain, grouped residual filter identity, aggregate filter expressions, and generic non-scalar expression ownership on their original planner `Expr::Case` spelling.
- The canonical boolean expansion itself is now null-safe under the engine's real branch-selection rules:
  - searched `CASE` branch conditions match only on `TRUE`
  - `FALSE` and `NULL` both fall through to the next arm
  - the shipped canonical form models that with `COALESCE(condition, FALSE)` guards instead of raw condition truthiness
  - `ELSE NULL` only collapses to `FALSE` in positive top-level scalar-filter contexts where both outcomes reject the row
  - negative contexts such as `NOT CASE ... END` keep the original three-valued distinction instead of collapsing `NULL` too early
- The shipped `0.107.0` rule is intentionally narrow:
  - only boolean searched `CASE` in scalar `WHERE`
  - only the exact first-match expansion
  - bounded by an explicit arm-count threshold
  - no general searched-`CASE` simplification
  - no arithmetic or non-boolean rewrite classes
- The branch builder still collapses the trivial guarded boolean cases introduced by the expansion itself:
  - `guard AND TRUE` becomes `guard`
  - `guard AND FALSE` becomes `FALSE`
  This keeps the canonical form from depending on later cleanup passes for the obvious boolean identities introduced by searched-`CASE` expansion.
- `0.107.0` also fixes one adjacent correctness issue that the new scalar canonicalization exposed: shared structural query-cache identity in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs) now includes expression-owned scalar `filter_expr`. Without that, different residual-only scalar filters could alias through the shared cache if their derived predicate fragments matched or both fell back to no predicate key.
- The important behavioral outcomes of the final patch are:
  - the old searched-`CASE` explain exception is gone for scalar `WHERE`
  - equivalent searched-`CASE` and explicit boolean first-match forms now line up on the same scalar explain surface
  - grouped searched-`CASE` surfaces keep their semantic `CASE` spelling instead of being silently rewritten
  - null-sensitive searched-`CASE` filters now execute correctly across plain, combined, and negated scalar boolean contexts
  - expression-owned scalar filters no longer reuse the wrong compiled plan through the shared query cache
- The lowering-side bool-field parity proof in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs) now asserts the real `0.107` contract instead of the older non-null predicate-owned expectation: the null-safe searched-`CASE` bool-field form stays expression-owned when no safe derived predicate subset exists.

```sql
EXPLAIN
SELECT name
FROM SessionSqlEntity
WHERE CASE WHEN age >= 30 THEN TRUE ELSE age = 20 END
ORDER BY age ASC;
```
