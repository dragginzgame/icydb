# 0.151.x

## 0.151.6

`0.151.6` makes the remaining standalone generated visible-index constructors
explicitly model-only. Planner and explain code no longer use `schema_owned`
wording for generated index sets now that accepted runtime index authority is
backed by accepted `SchemaInfo` contracts.

### Generated Model-Only Naming

- Renamed `VisibleIndexes::schema_owned` to
  `VisibleIndexes::generated_model_only` for standalone generated query and
  explain surfaces.
- Renamed the test-only `VisibleIndexes::planner_visible` helper to
  `VisibleIndexes::generated_model_only_for_test`.
- Updated planner/explain boundary guards to reject the old ambiguous
  constructor names and require the generated/model-only names.
- Moved the generated index-to-access-contract adapter into
  `access::model_only`, renamed it
  `SemanticIndexAccessContract::model_only_from_generated_index`, and tightened
  guards so it remains isolated to generated/model-only planner projection and
  tests, not accepted session/write/recovery runtime.
- Shifted primary-key compare and primary-key range planner recognition to
  `SchemaInfo::primary_key_name`, reducing another accepted-runtime read of
  generated `EntityModel` field identity.

## 0.151.5

`0.151.5` removes the generated expression-index candidate lane from accepted
runtime planning. Store-ready accepted visible indexes now derive field-path and
expression planner contracts only from accepted `SchemaInfo`, and accepted
runtime schema fingerprint identity now hashes the accepted persisted schema
snapshot directly. Generated index sets remain on explicit model-only planning
and explain surfaces, and generated `EntityModel` index metadata is no longer
appended to accepted commit/cache fingerprint identity.

### Accepted Expression Planner And Fingerprint Authority

- Removed `GeneratedExpressionCandidateIndex` and the generated expression
  candidate collection from accepted `VisibleIndexes`.
- Made accepted runtime visible-index construction schema-only instead of
  accepting generated `model.indexes()` for expression fallback filtering.
- Removed generated model parameters from accepted visible-index helper
  signatures after the helper stopped consuming generated indexes.
- Routed access planning, residual reranking, and explain access-choice
  projection over accepted field-path and accepted expression planner contracts
  only.
- Added a recovery rebuild boundary guard proving startup index rebuild reaches
  expression-aware accepted commit preflight instead of rebuilding generated
  expression keys directly.
- Tightened write-boundary guards so accepted runtime planner/explain entrypoints
  cannot reintroduce generated expression candidate parameters.
- Replaced generated-model accepted fingerprint helpers with
  `accepted_commit_schema_fingerprint` and `accepted_schema_cache_fingerprint`
  APIs that accept only an `AcceptedSchemaSnapshot`.
- Changed accepted runtime fingerprinting to hash the accepted entity path and
  encoded accepted schema snapshot payload as the authoritative contract.
- Removed generated model/index parameters from commit preparation, commit
  rebuild, diagnostics, mutation, delete, session write, SQL write, and cache
  fingerprint call sites.
- Kept generated model commit hashing isolated to test-only fingerprint
  compatibility coverage.
- Added a write-boundary guard proving accepted commit/cache fingerprint APIs
  cannot regain generated `EntityModel`/`IndexModel` runtime authority.

## 0.151.4

`0.151.4` removes the generated expression-index write fallback from
forward-index commit preflight. Accepted expression-index contracts are now the
only runtime source for expression-index write key shape, filtered membership,
store identity, and unique validation after schema acceptance.

### Accepted Expression Write Authority

- Removed the generated expression-index wrapper and generated expression write
  planning loop from forward-index mutation planning.
- Removed commit-preflight gating that reopened generated `EntityModel::indexes`
  to detect deferred expression-index write work.
- Removed generated expression unique validation from write preflight.
- Deleted the now-dead generated structural index-key builder that accepted an
  `IndexModel` beside an accepted row contract.
- Tightened write-boundary guards so forward-index write planning and unique
  validation cannot return to generated expression `IndexModel` authority.

### Validation

- `cargo check -p icydb-core --tests --features sql`
- `cargo test -p icydb-core --test write_boundary_guards -- --nocapture`

## 0.151.3

`0.151.3` moves expression-index write preflight onto accepted expression
contracts. Commit preflight now builds expression-aware accepted `SchemaInfo`
views and uses accepted expression-index metadata for key construction,
filtered membership predicates, and unique validation when that metadata is
available.

### Accepted Expression Write Preflight

- Added accepted expression-index key construction from canonical row slots.
- Routed forward-index mutation planning through
  `SchemaInfo::expression_indexes()` before falling back to generated
  expression indexes.
- Compiled expression-index filtered membership predicates from accepted
  predicate SQL.
- Added accepted expression-index unique validation, including accepted
  expression key rebuilding for conflict checks.
- Updated commit-preflight guards so expression-aware accepted schema info is
  required before forward-index planning can use accepted expression
  contracts.

## 0.151.2

`0.151.2` moves the runtime visible-index boundary onto accepted
expression-index contracts when the accepted schema view includes them. The
generated expression candidate lane remains as an explicit fallback for schema
views that have not opted into expression projection yet.

### Accepted Expression Runtime Visibility

- Added planner-facing accepted expression-index contracts under
  `VisibleIndexes`.
- Added owned accepted expression key items to reduced semantic index access
  contracts so planner, access-choice, explain, and index-key materialization
  can consume expression metadata without reopening `IndexModel`.
- Made accepted runtime query planning build expression-aware `SchemaInfo`
  views at the session boundary.
- Kept generated expression candidates empty when accepted expression contracts
  are available, leaving generated expression planning as a named fallback.
- Extended prefix/range matching, order-term rendering, covering component
  projection, access canonicalization, and index-key construction for accepted
  expression key items.
- Updated write-boundary guards to lock the accepted expression visible-index
  boundary.

## 0.151.1

`0.151.1` adds the first accepted expression-index contract implementation
slice. Persisted schema snapshots can now carry explicit expression key items,
schema proposals derive those contracts from generated `IndexExpression`
metadata, the schema snapshot codec round-trips the accepted expression
contract shape, and `SchemaInfo` can project accepted expression-index
contracts through an opt-in staged path for the next planner/write routing
slices.

### Accepted Expression Snapshot Contracts

- Added accepted expression index operation, key-item, and expression snapshot
  DTOs.
- Added explicit mixed-key `Items` snapshots so composite field/expression
  index keys preserve key-item order after schema acceptance.
- Projected generated expression index key items into accepted source
  field-path contracts, expression operations, input/output persisted kinds,
  and canonical expression text.
- Bumped the persisted schema snapshot codec version and added wire support for
  accepted expression index key items.
- Kept `SchemaInfo::field_path_indexes()` filtered to field-path-only accepted
  indexes until planner/runtime expression routing moves to accepted contracts.
- Added a staged, opt-in `SchemaInfo::expression_indexes()` projection so
  accepted expression-index contracts can be inspected without adding work to
  every hot accepted schema view before planner/write routing consumes them.
- Extended integrity checks to reject invalid accepted expression index
  metadata instead of accepting malformed persisted contracts.
- Kept the long 100-run SQL repeat perf audit rows behind
  `SQL_PERF_AUDIT_LONG_REPEAT` so default PocketIC perf checks stay under the
  single-message instruction cap while retaining the deep-cache audit option.

### Validation

- `cargo test -p icydb-core schema::codec --features sql -- --nocapture`
- `cargo test -p icydb-core schema::snapshot --features sql -- --nocapture`
- `cargo test -p icydb-core schema::info --features sql -- --nocapture`
- `cargo check -p icydb-core --tests --features sql`

## 0.151.0

`0.151.0` starts the accepted expression-index contract phase. The new design
and status docs define the remaining runtime authority cleanup after `0.150`:
expression index planning, explain output, cache fingerprints, write key
construction, uniqueness validation, and recovery should consume accepted
expression index contracts after schema acceptance instead of generated runtime
metadata.

### Accepted Expression Index Contract Design

- Added the `0.151` design doc for accepted expression index contracts.
- Added a small `0.151` status tracker for the patch sequence.
- Named the hard invariant that accepted runtime code must not call
  `E::MODEL.indexes()`, `EntityModel::indexes()`, or consume raw `IndexModel`
  for expression index planning, explain, cache identity, writes, uniqueness
  validation, or recovery after schema reconciliation.
- Split the planned work into accepted snapshot/proposal contracts,
  reconciliation, `SchemaInfo`/`VisibleIndexes`, planner/explain/fingerprint,
  write/recovery, wrapper removal, and guard phases.
