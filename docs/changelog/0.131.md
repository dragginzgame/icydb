# 0.131.x

## 0.131.1

`0.131.1` continues the commit and execution-contract audit with another
boundary-only cleanup. Commit-store byte framing is split into
`commit::store::control_slot` and `commit::store::marker_envelope`, so the
stable-cell owner no longer also carries the full control-slot and marker
envelope codec implementation in one file. The store root still owns durable
marker lifecycle, while control-slot framing and versioned marker-envelope
framing now have separate file-local owners under the commit store boundary.

The runtime hook registry now owns the row-commit preparation and delete-side
strong-relation validation orchestration that depends on registered entity
hooks. `Db` still exposes the same internal recovery/delete callbacks, but it
delegates hook traversal through `db::runtime_hooks`, keeping the database
surface focused on registry/store access instead of hook iteration policy.

Commit-store framing gets one more boundary tightening after the file split.
The shared little-endian byte cursor now lives in `commit::store::bytes`
instead of under the control-slot owner, so marker-envelope decoding no longer
imports through control-slot internals. Direct multi-row and single-row marker
encoders now share one checked control-slot length calculation while preserving
the single-row fast path, and migration-state commit begin now delegates the
marker-empty invariant to the store through `set_with_migration_state_if_empty`.

Prepared index mutation construction is also narrowed. Index planning now uses
`CommitIndexOp` constructors for unchanged, insert, and remove cases; rollback
and reverse-index code use `PreparedIndexMutation` constructors instead of
importing `PreparedIndexDeltaKind` directly. Commit-window metric aggregation
reads delta counters through `PreparedIndexMutation::counter_increments()`,
keeping the delta-kind enum internal to the commit prepared-op/marker owners.

The leftover top-level `db::contracts` module is deleted. Its canonical value
ordering helper now lives in `db::numeric`, next to the numeric-or-strict
comparison primitive it extends into total value ordering. The small
first-violated-rule iterator is localized into the grouped validation policy
and route-feasibility policy owners that use it. The db-level structural guard
test that used to sit under `db::contracts::tests` now lives under
`db::tests`, so invariant coverage remains without keeping a fake contracts
boundary.

Grouped row runtime behavior also moves out of the execution contracts layer.
`RowView`, `StructuralGroupedRowRuntime`, `GroupedStreamStage`,
`GroupedFoldStage`, and `compile_grouped_row_slot_layout_from_parts(...)` now
live under `executor::pipeline::runtime`. The grouped contracts module keeps
the grouped planner/route-stage DTOs, while row decoding, grouped row-view
evaluation, retained-slot layout derivation, and fold-stage runtime counters
sit with runtime ownership.

## 0.131.0

`0.131.0` starts the commit and execution-contract audit line. The commit
store now validates direct multi-row and single-row commit-marker writes before
bytes are persisted, so direct commit-control-slot helpers cannot bypass the
same row-op shape checks used by normal marker decoding. Commit marker clearing
also now has separate verified and raw-test paths: successful commit/recovery
clears through a fallible verified path that preserves migration bytes only
after decoding the existing control slot, while tests that intentionally persist
corruption use an explicit raw clear helper.

The execution-contract audit moves fast-stream, scan-binding, and scalar
materialization request DTOs into the pipeline contracts layer.
`FastStreamRouteKind`, `FastStreamRouteRequest`, `AccessStreamBindings`,
`AccessScanContinuationInput`, `ScalarMaterializationCapabilities`, and
`KernelPageMaterializationRequest` now live in
`db::executor::pipeline::contracts`, and scalar plus aggregate fast paths import
those request shapes from the contracts layer. The scan fast-stream module keeps
`execute_fast_stream_route(...)` and all route dispatch/runtime behavior, the
access stream module keeps traversal cursors, spec-consumption checks, and
physical stream resolution, and the terminal page module keeps row
materialization behavior.

The commit boundary cleanup also narrows `db::commit` root exports so the
module reads as a protocol boundary instead of a general internal toolbox.
Marker byte-codec details such as the current marker format version, maximum
marker byte limit, payload encode/decode helpers, and marker-shape validation
now stay under `commit::marker`; recovery phase helpers for replay and rebuild
stay under their owning recovery submodules. Executor, migration, schema, and
test code still import the protocol-facing commit marker, row-op, guard,
prepared-op, and recovery entrypoint types from `db::commit`.

The execution isolation pass then moves route-attempt behavior out of the
contracts input file. `ExecutionRuntimeAdapter` now lives under
`executor::pipeline::runtime`, where it owns fast-stream dispatch, fallback
stream resolution, and scalar materialization calls. `ExecutionAttemptKernel`
is the route-attempt runtime wrapper that resolves decorated key streams and
materializes one route attempt from an immutable `ExecutionInputs` snapshot.
Retained-slot layout derivation also moved to `executor::pipeline::runtime`,
so `contracts::execution::inputs` keeps the snapshot and request DTOs without
owning slot-requirement compilation. The no-DISTINCT key-stream resolver is now
also on `ExecutionAttemptKernel`, and `ExecutionInputs` freezes row-read
consistency when the snapshot is built instead of re-deriving it during runtime
materialization.

Runtime hook registration now lives under `db::runtime_hooks` instead of
`db::commit::hooks`. `EntityRuntimeHooks` still re-exports from `db` for
generated-code stability, but the registry no longer makes commit appear to own
delete-side strong-relation validation. Commit continues to consume only the
row-commit preparation callback, while delete/relation validation consumes the
relation-delete callback through the db runtime hook registry.
