# 0.131.x

## 0.131.0

`0.131.0` starts the commit and execution-contract audit line. The commit
store now validates direct multi-row and single-row commit-marker writes before
bytes are persisted, so direct commit-control-slot helpers cannot bypass the
same row-op shape checks used by normal marker decoding. Commit marker clearing
also now has separate verified and raw-test paths: successful commit/recovery
clears through a fallible verified path that preserves migration bytes only
after decoding the existing control slot, while tests that intentionally persist
corruption use an explicit raw clear helper.

The execution-contract audit moves fast-stream, scan-binding, and scalar
materialization request DTOs into the pipeline contracts layer.
`FastStreamRouteKind`, `FastStreamRouteRequest`, `AccessStreamBindings`,
`AccessScanContinuationInput`, `ScalarMaterializationCapabilities`, and
`KernelPageMaterializationRequest` now live in
`db::executor::pipeline::contracts`, and scalar plus aggregate fast paths import
those request shapes from the contracts layer. The scan fast-stream module keeps
`execute_fast_stream_route(...)` and all route dispatch/runtime behavior, the
access stream module keeps traversal cursors, spec-consumption checks, and
physical stream resolution, and the terminal page module keeps row
materialization behavior.

The commit boundary cleanup also narrows `db::commit` root exports so the
module reads as a protocol boundary instead of a general internal toolbox.
Marker byte-codec details such as the current marker format version, maximum
marker byte limit, payload encode/decode helpers, and marker-shape validation
now stay under `commit::marker`; recovery phase helpers for replay and rebuild
stay under their owning recovery submodules. Executor, migration, schema, and
test code still import the protocol-facing commit marker, row-op, guard,
prepared-op, and recovery entrypoint types from `db::commit`.

The execution isolation pass then moves route-attempt behavior out of the
contracts input file. `ExecutionRuntimeAdapter` now lives under
`executor::pipeline::runtime`, where it owns fast-stream dispatch, fallback
stream resolution, and scalar materialization calls. `ExecutionAttemptKernel`
is the route-attempt runtime wrapper that resolves decorated key streams and
materializes one route attempt from an immutable `ExecutionInputs` snapshot.
Retained-slot layout derivation also moved to `executor::pipeline::runtime`,
so `contracts::execution::inputs` keeps the snapshot and request DTOs without
owning slot-requirement compilation. The no-DISTINCT key-stream resolver is now
also on `ExecutionAttemptKernel`, and `ExecutionInputs` freezes row-read
consistency when the snapshot is built instead of re-deriving it during runtime
materialization.

Runtime hook registration now lives under `db::runtime_hooks` instead of
`db::commit::hooks`. `EntityRuntimeHooks` still re-exports from `db` for
generated-code stability, but the registry no longer makes commit appear to own
delete-side strong-relation validation. Commit continues to consume only the
row-commit preparation callback, while delete/relation validation consumes the
relation-delete callback through the db runtime hook registry.

What changed:
- added `validate_commit_row_op_shape(...)` as the shared row-op validation
  gate for full markers and direct single-row marker persistence
- routed direct multi-row and single-row control-slot encoders through marker
  shape validation before writing commit-control bytes
- shared commit-control and marker-envelope header writing across direct marker
  encode paths
- replaced infallible marker clearing with `clear_verified(...)`, which can
  report malformed migration-bearing control bytes instead of silently
  discarding them
- added `clear_raw_for_tests(...)` for corruption-oriented tests that need to
  bypass verified marker decode
- added regression coverage for direct marker encoders rejecting row ops with
  neither `before` nor `after` payloads
- added regression coverage for verified marker clearing preserving valid
  migration bytes and rejecting malformed migration-bearing control slots
- removed root-level `db::commit` re-exports for marker byte-codec internals
  and recovery-only replay/rebuild helpers
- retargeted commit-internal imports and tests to the owning marker, replay,
  and rebuild modules for those implementation details
- moved `EntityRuntimeHooks`, runtime hook callback aliases, and hook lookup
  helpers from `commit::hooks` to `db::runtime_hooks`
- kept `db::EntityRuntimeHooks` as the public re-export so generated code keeps
  the same stable import path
- moved `FastStreamRouteKind` and `FastStreamRouteRequest` from
  `executor::scan::fast_stream_route` to
  `executor::pipeline::contracts::fast_stream`
- kept `execute_fast_stream_route(...)` in scan ownership and retargeted scalar
  and aggregate fast-path imports to the contracts-owned DTOs
- moved `AccessStreamBindings` and `AccessScanContinuationInput` from
  `executor::stream::access::bindings` to
  `executor::pipeline::contracts::scan`
- kept `AccessSpecCursor`, `ExecutableAccess`, physical stream resolution, and
  traversal invariant checks in access-stream ownership
- moved `ScalarMaterializationCapabilities` and
  `KernelPageMaterializationRequest` from `executor::terminal::page` to
  `executor::pipeline::contracts::materialization`
- kept scalar materialization plan resolution, direct data-row fast paths,
  page-kernel execution, cursor construction, and post-access materialization
  in terminal page ownership
- moved `ExecutionRuntimeAdapter` and the aligned execution materialization
  contract out of `contracts::execution::inputs` and into
  `executor::pipeline::runtime::adapter`
- added `ExecutionAttemptKernel` under `executor::pipeline::runtime` as the
  route-attempt runtime wrapper for key-stream decoration and materialization
- moved retained-slot layout derivation and `RetainedSlotRequirements` out of
  `contracts::execution::inputs` and into `executor::pipeline::runtime`
- moved the no-DISTINCT key-stream resolver receiver from `ExecutionInputs` to
  `ExecutionAttemptKernel`
- froze row-read consistency inside `ExecutionInputs` at construction time so
  runtime materialization reads a snapshot field instead of re-deriving policy
  from the semantic plan

Retained boundaries:
- commit marker validation remains in commit marker ownership
- commit store owns durable control-slot encoding and verified marker clearing
- raw marker clearing is test-only and explicit
- `db::commit` root exposes the protocol boundary; marker codec details and
  recovery phase helpers stay under their owning child modules
- runtime hook registration is owned by `db::runtime_hooks`; commit consumes
  the row-preparation callback and delete/relation code consumes the relation
  validation callback
- pipeline contracts own fast-stream, scan-binding, and materialization request
  DTOs only
- scan fast-stream route owns route dispatch and physical stream execution
- access stream owns spec cursor consumption, executable stream assembly, and
  physical stream traversal
- terminal page owns row materialization, scalar page policy resolution, and
  output payload construction
- pipeline runtime owns fast-stream dispatch, fallback stream resolution, and
  route-attempt materialization over an already-assembled `ExecutionInputs`
  snapshot
- pipeline runtime owns retained-slot layout derivation; terminal page still
  owns the retained-slot layout and row storage behavior
- `ExecutionInputs` remains the immutable execution snapshot and no longer
  exposes key-stream execution entrypoints
- aggregate and scalar execution paths consume the same fast-stream request
  shape without moving execution behavior into contracts
