# 0.83 — Serialization Boundary Cleanup

## 0.83.1 - 2026-04-15 - Follow-Up CBOR Retreat and Runtime Codec Tightening

This follow-up patch keeps the `0.83` line moving after the first landed
runtime serializer retreat.

It does three concrete things:

* removes another fixed-shape runtime owner from CBOR entirely
* deletes more redundant decode and structural-encoding scaffolding
* aligns one more executor-side byte-accounting path with the owner-local
  runtime `Value` storage encoding instead of CBOR size

What changed:

* replaced the old `ValueWire` mirror enum bridge in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/value/wire.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/value/mod.rs`
* `Value` now deserializes directly through a custom serde enum visitor
  instead of:
  * decoding into a second mirror enum
  * then converting that mirror enum back into `Value`
* map normalization still routes through `Value::from_map`, so the canonical
  map invariant is unchanged
* bigint wrapper handling also stays explicit in the new visitor
* removed structural encoding duplication between `ByKind` storage and
  `Value` storage in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/cbor.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/encode.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs`
* shared CBOR wrapper payload emitters for:
  * account
  * decimal
  * timestamp
  * bigint / biguint
  * subaccount
  * now live under one owner instead of being copied across both structural
    encoding paths
* moved migration state off CBOR entirely in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/migration/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/migration/tests/mod.rs`
* migration progress is now stored as a fixed binary payload with:
  * explicit magic
  * explicit format version
  * exact string / integer / optional-bytes framing
* that makes migration state a true engine-owned protocol instead of another
  generic persisted CBOR payload
* added direct migration-state coverage for:
  * binary round-trip
  * trailing-byte rejection
* moved executor `bytes()` value sizing off CBOR length counting in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/tests.rs`
* `serialized_value_len(...)` now measures the owner-local structural
  value-storage payload instead of running the value back through CBOR just to
  count encoded bytes
* removed the remaining generic persisted-payload decode helper from
  `db::codec` once no production DB caller still needed that shared CBOR
  wrapper
* deleted the now-dead CBOR byte-count helper in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/codec/cbor.rs`
* widened the metadata-free `PersistedRow` derive in:
  * `/home/adam/projects/icydb/crates/icydb-derive/src/lib.rs`
  * `/home/adam/projects/icydb/crates/icydb-derive/src/persisted_row.rs`
* exact-kind inference now also admits:
  * `Account`
  * `Int`
  * `Nat`
  * and the recursive container shapes built from those exact owners
* added explicit decimal scale hints on the metadata-free derive with:
  * `#[icydb(scale = N)]`
* annotated decimal fields now route through:
  * `FieldKind::Decimal { scale: N }`
  * and therefore use the stricter by-kind structural codec instead of the
    CBOR fallback path
* unannotated decimal fields intentionally remain on the compatibility seam,
  because bare `Decimal` types do not carry one schema-authoritative field
  scale on their own
* added a focused persisted-row regression in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs`
* that regression proves the legacy derive decimal hint:
  * emits the same by-kind slot bytes as schema-owned decimal storage
  * decodes the same matching by-kind decimal payload the schema-owned path uses
* persisted decimal scale drift is still covered on the executor/save path in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/tests/mutation_save.rs`

Why this matters:

* `0.83.0` already removed the first obvious generic serializer owners from
  the DB runtime, but several follow-up seams still remained:
  * the `Value` mirror decode bridge
  * duplicated structural wrapper emission
  * migration state as a CBOR payload
  * executor bytes accounting through CBOR length
* the old `ValueWire` mirror enum was defensible when introduced, but it had
  become a second copy of the `Value` variant surface, plus another decode hop
  that could drift from the real `Value` ownership model over time
* migration state is a fixed, engine-owned progress record, so leaving it on
  CBOR after the first `0.83` cut would have kept a generic format at a seam
  that no longer needed one
* moving executor `bytes()` sizing off CBOR closes another small but real
  runtime inconsistency:
  * the terminal now measures the same owner-local structural value payload
    that the rest of the runtime already treats as canonical
  * instead of using a separate “CBOR size” interpretation just for bytes
    accounting
* widening the metadata-free `PersistedRow` derive matters because the
  schema-aware path had already moved further off CBOR, but older derive-owned
  typed rows still needed a practical way to leave the fallback seam without
  pretending ambiguous schema facts could be guessed from Rust types alone
* the decimal hint keeps that line honest:
  * exact kinds are inferred where the byte contract is unambiguous
  * explicit hints are required where schema policy, such as decimal scale,
    must not be guessed
* centralizing the remaining runtime DB CBOR helpers under `db::codec::cbor`
  in `0.83.0` made this follow-up easier to land because the remaining owners
  were already explicit instead of scattered

What remains:

* the main remaining runtime CBOR surface is the structural fallback path in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs`
* that is a materially harder owner than cursor tokens or migration state,
  because it still covers generic structural payload decode for arbitrary
  persisted field shapes
* in practice that fallback is now narrower than it was at the start of
  `0.83.1`, because more metadata-free derive cases can route through exact
  by-kind storage contracts before they ever need the compatibility seam
* the generic serializer module itself still exists for non-DB owners and test
  parity, but production DB/runtime reliance on it is now much smaller than it
  was at the start of the patch

* `cargo test -p icydb-core db::migration:: -- --nocapture`
* `cargo test -p icydb-core db::executor::terminal::tests:: -- --nocapture`

## 0.83.0 - 2026-04-15 - First Runtime Serializer Retreat

This patch starts the `0.83` line by shrinking generic serializer authority in
the DB runtime without trying to solve every remaining CBOR owner in one shot.

The first landed slice focuses on the seams that were already narrow enough to
move safely:

* cursor tokens
* DB-owned row and persisted-value wrappers
* the most obvious generic serializer calls inside `db::*`
* the remaining scattered DB CBOR imports

What changed:

* moved cursor tokens off `crate::serialize` / generic CBOR in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/cursor/token/codec.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/cursor/token/scalar.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/cursor/token/grouped.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/cursor/token/mod.rs`
* scalar and grouped continuation tokens now use explicit token-owned binary
  codecs instead of the generic serializer
* moved the persisted row envelope fully under DB-owned codec policy in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/codec/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/codec/tests/mod.rs`
* the outer row wrapper is now a DB-owned binary envelope instead of a
  generic serializer boundary
* moved `FieldStorageDecode::Value` off generic `crate::serialize` in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/contract.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs`
* the persisted-row `Value` path now encodes through an owner-local
  structural value-storage contract instead of calling
  `crate::serialize::serialize(&Value)`
* the persisted-row `ByKind` write path no longer constructs
  `serde_cbor::Value` trees just to emit structural bytes
* generic persisted structural slot encode/decode now routes through a
  DB-owned boundary instead of `crate::serialize`
* migration state encode and executor bytes sizing both stopped routing
  through `crate::serialize`, even though they were still using DB-owned CBOR
  helpers at this stage
* introduced one DB-owned CBOR helper surface in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/codec/cbor.rs`
* moved the remaining runtime DB CBOR calls behind that helper instead of
  importing `serde_cbor` separately across multiple modules

Why this matters:

* before `0.83.0`, too many DB runtime owners still treated the generic
  serializer as the default answer even when the engine already had a better
  local ownership model
* this first patch deliberately did not promise to delete CBOR outright
* instead it proved the more important thing first:
  * the DB runtime can stop depending on `crate::serialize` as a default
    boundary
  * and the remaining CBOR owners can be made explicit enough to attack one by
    one in follow-ups
* that is the point where the serialization cleanup becomes tractable instead
  of vague
