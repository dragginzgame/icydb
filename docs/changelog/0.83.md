# 0.83 — Serialization Boundary Cleanup

## 0.83.4 - 2026-04-16 - SQL Describe Tables and Executor Contract Trimming

This patch is a small follow-through on the `0.83` cleanup line. It does not
change the structural binary cutover itself, but it keeps reducing leftover
wrapper indirection around the executor/runtime path and makes the local SQL
shell easier to inspect during manual verification.

It does two concrete things:

* changes local SQL `DESCRIBE` rendering from bullet lists to padded ASCII
  tables for fields, indexes, and relations
* removes several more executor-owned wrapper and payload shims that were
  still carrying the same scalar/grouped runtime facts through extra internal
  layers
* keeps reducing duplicated query-intent and session cache/explain assembly
  now that the structural cache-key and prepared execution boundaries are
  more explicit

What changed:

* the local SQL shell now renders `DESCRIBE` field metadata as a table with
  `name`, `type`, `pk`, and `queryable` columns in:
  * `/home/adam/projects/icydb/crates/icydb/src/db/sql/mod.rs`
* `DESCRIBE` index metadata now renders as a matching table with `name`,
  `fields`, and `unique`
* `DESCRIBE` relation metadata now renders as a matching table with `field`,
  `target`, `strength`, and `cardinality`
* the stable shell-output contract tests for `DESCRIBE` were updated to lock
  that new padded table surface in:
  * `/home/adam/projects/icydb/crates/icydb/src/db/sql/mod.rs`
* several executor cleanup passes removed owner-local wrapper shells across:
  * grouped aggregate runtime output/finalization
  * grouped page-finalize candidate shaping
  * grouped pipeline runtime preparation
  * scalar pipeline route preparation and structural finalization
  * load orchestrator kernel/payload forwarding
  * execution contract payload and option wrappers
  * execution input request assembly that had still been rebuilt in the
    kernel layer
* specific examples from that cleanup include:
  * removing the `db::reduced_sql` name in favor of `db::sql_shared`
  * collapsing grouped runtime observability and page-finalize helpers
  * removing single-variant scalar payload wrappers in the execution contracts
  * removing the one-field `ExecutionOutputOptions` shim
  * tightening prepared scalar and grouped runtime construction so shared
    route/runtime bundles are assembled through fewer duplicated paths
  * moving row-collector and runtime-page materialization request assembly
    onto `ExecutionInputs` so `db::executor::kernel` no longer rebuilds the
    same predicate/projection/cursor contract by hand
* the shared lower query cache and query-intent layers also shed several
  more duplicated constructor/forwarding paths:
  * `StructuralQueryCacheKey` now builds its normalized and non-normalized
    predicate forms through one shared predicate-key helper instead of two
    mirrored full-key constructors
  * `QueryPlanCacheKey` now assembles its authority/schema/visibility shell
    through one owner-local helper instead of repeating the same field set in
    multiple constructors
  * `StructuralQuery` now owns its own intent rewrap helpers instead of
    rebuilding `Self { intent }` ad hoc across small infallible and fallible
    transformation methods
  * the load explain path now shares visibility-aware descriptor and verbose
    helpers, and the typed `Query<E>` layer now owns text/json rendering so
    the structural layer no longer carries a second dead rendered explain path

Why this matters:

* the new `DESCRIBE` table output is easier to scan in the shell, especially
  for wide entities like the RPG demo where the old bullet format buried the
  useful comparison columns inside long repeated strings
* the executor/runtime cleanup matters because the post-`0.83.2` binary line
  still had a fair amount of internal wrapper carry-over:
  * single-variant payload enums
  * one-field option structs
  * duplicate prepared-runtime bundle assembly
  * local forwarding structs that only rewrapped the same payload/trace data
* removing those wrappers does not widen public behavior, but it leaves fewer
  hidden seams for later runtime, SQL, and aggregate work to trip over
* the query/cache-side cleanup matters for the same reason:
  * normalized-predicate cache keys, visible-index planning, and explain
    rendering were all already explicit contracts
  * the remaining duplication was mostly owner-local wrappers and repeated
    field assembly
  * collapsing those paths makes later planner and explain changes less
    likely to drift across “same data, different wrapper” code paths

## 0.83.3 - 2026-04-16 - Serde Retrenchment and Binary Follow-Through

remaining ambient serializer policy and tightening the new binary owners now
that the runtime storage boundary is already engine-owned.

  `Deserialize`-only unless a type explicitly owns more

  contracts are stable
* tightens a few remaining encode/decode hot spots and type-local codec

    text
  * `Principal` structural/storage-key/hash lanes now reuse borrowed validated
    bytes instead of allocating `Vec<u8>`
  * `Account` structural/storage-key/hash/entity-key lanes now use fixed-size

Why this matters:

* `0.83.2` already hard-cut the runtime storage boundary to structural binary
  and removed CBOR as a runtime contract
* removing `Serialize` as a schema-wide and core-wide policy matters because
  the old serializer contract had become ambient:
  * types inherited it because the system used to serialize broadly
  * not because those types still owned a real runtime serialization contract
* the follow-up codec cleanup for `Ulid`, `Date`, `Timestamp`, `Principal`,
  and `Account` matters because the structural binary cut would still have
  carried some avoidable text and allocation overhead if those local codecs had
  been left behind unchanged
* the query-path redundancy cleanup matters because it removes work that had
  become easier to see once the broader serializer/CBOR noise was gone

What remains:

* the remaining serde use in core is intentionally narrowed to
  `Deserialize`-owned API and Candid input surfaces
* more codec micro-optimization is still possible, especially around bigint and
  other broader `Value` shapes, but that is a separate performance pass rather
  than unfinished serializer retreat work

## 0.83.2 - 2026-04-16 - Structural Binary Cutover and Final CBOR Removal

This patch is the hard cut that turns the `0.83` line from “CBOR retreat in
progress” into “runtime storage is now engine-owned binary.”

It does four concrete things:

* replaces the last runtime CBOR-shaped structural field contracts with an
  IcyDB-owned structural binary walker and owner-local binary codecs
* removes the generic serializer and `serde_cbor` from `icydb-core`
* removes the final persisted-row CBOR semantics and stale CBOR naming
* deletes the last implicit and explicit legacy `PersistedRow` CBOR escape
  hatches so old-style derives must use explicit storage owners

What changed:

* added the owned Structural Binary v1 walker in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/binary.rs`
* that walker now owns:
  * bounded head parsing
  * bounded skip

  * tagged variant payload splitting
* moved the structural storage-key lane onto that binary owner in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key.rs`
* moved schema-driven composite recursion onto that binary owner in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/composite.rs`
* replaced the old CBOR-shaped `FieldStorageDecode::Value` envelope with an
  owner-local binary `Value` envelope in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs`
* hard-cut scalar and leaf structural owners to binary in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/scalar.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/leaf.rs`

* deleted the last structural-field CBOR owner:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/cbor.rs`
* deleted the older DB-side CBOR helper owner:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/codec/cbor.rs`
* deleted the generic serializer module from `icydb-core`:
  * `/home/adam/projects/icydb/crates/icydb-core/src/serialize/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/serialize/cbor.rs`
* removed `serde_cbor` from:
  * `/home/adam/projects/icydb/crates/icydb-core/Cargo.toml`
* deleted CBOR-specific runtime and test surfaces instead of pretending those
  bytes were still a supported contract:
  * CBOR-specific core tests were removed from `types/*`, `value/tests.rs`,
    persisted-row tests, row-decode tests, and macro fixture tests

  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/contract.rs`

  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/reader.rs`
* `LeafCodec::CborFallback` was renamed to `LeafCodec::StructuralFallback`
  because the remaining fallback is now structural binary, not CBOR
* tightened legacy `PersistedRow` derive routing in:
  * `/home/adam/projects/icydb/crates/icydb-derive/src/persisted_row.rs`
* old-style derived rows now route only through explicit owners:
  * `#[icydb(scale = ...)]`
  * `#[icydb(value)]`
  * `#[icydb(meta)]`
* the old implicit and then explicit CBOR fallback was removed entirely, so
  unsupported legacy field shapes now fail at derive time instead of silently
  reintroducing a generic serializer seam

Why this matters:

* earlier `0.83` patches were still transitional:
  * they narrowed the serializer seams
  * they concentrated remaining CBOR owners
  * they proved more runtime boundaries could move to engine-owned bytes
* `0.83.2` is the point where that line actually closes in the runtime:
  * `structural_field` is no longer “CBOR-shaped with local wrappers”
  * it is now an engine-owned binary boundary with an explicit walker contract
* deleting `serde_cbor` and the old generic serializer matters because it
  removes the temptation to keep testing or routing storage behavior through a
  format the runtime no longer claims to support
* tightening the legacy derive matters because implicit fallbacks are how old
  serializer seams come back:
  * `meta`, `value`, and explicit decimal scale hints are intentional storage
    choices
  * silent generic fallback is not

What remains:

* there is no remaining CBOR runtime owner in `icydb-core`
* there is no remaining generic serializer module in `icydb-core`
* the next cleanup line is no longer “remove CBOR”
* it is tightening policy, tests, and local codec ownership on top of the new
  binary boundary

runtime serializer retreat.

* removes another fixed-shape runtime owner from CBOR entirely

  runtime `Value` storage encoding instead of CBOR size

  * `/home/adam/projects/icydb/crates/icydb-core/src/value/mod.rs`

  map invariant is unchanged
* bigint wrapper handling also stays explicit in the new visitor
* removed structural encoding duplication between `ByKind` storage and
  `Value` storage in:

* shared CBOR wrapper payload emitters for:

* migration progress is now stored as a fixed binary payload with:

* that makes migration state a true engine-owned protocol instead of another

* added direct migration-state coverage for:
  * binary round-trip

  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/tests.rs`

  value-storage payload instead of running the value back through CBOR just to
  count encoded bytes

  `db::codec` once no production DB caller still needed that shared CBOR
  wrapper

  * `/home/adam/projects/icydb/crates/icydb-core/src/db/codec/cbor.rs`

  * `/home/adam/projects/icydb/crates/icydb-derive/src/lib.rs`

  * `Account`
  * `Int`

  * `#[icydb(scale = N)]`
* annotated decimal fields now route through:
  * `FieldKind::Decimal { scale: N }`
  * and therefore use the stricter by-kind structural codec instead of the
    CBOR fallback path
* unannotated decimal fields intentionally remain on the compatibility seam,
  because bare `Decimal` types do not carry one schema-authoritative field
  scale on their own
* added a focused persisted-row regression in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs`
* that regression proves the legacy derive decimal hint:

  * decodes the same matching by-kind decimal payload the schema-owned path uses

  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/tests/mutation_save.rs`

Why this matters:

* `0.83.0` already removed the first obvious generic serializer owners from
  the DB runtime, but several follow-up seams still remained:
  * the `Value` mirror decode bridge
  * duplicated structural wrapper emission
  * migration state as a CBOR payload
  * executor bytes accounting through CBOR length
* the old `ValueWire` mirror enum was defensible when introduced, but it had
  become a second copy of the `Value` variant surface, plus another decode hop
  that could drift from the real `Value` ownership model over time
* migration state is a fixed, engine-owned progress record, so leaving it on
  CBOR after the first `0.83` cut would have kept a generic format at a seam
  that no longer needed one
* moving executor `bytes()` sizing off CBOR closes another small but real
  runtime inconsistency:
  * the terminal now measures the same owner-local structural value payload
    that the rest of the runtime already treats as canonical
  * instead of using a separate “CBOR size” interpretation just for bytes
    accounting
* widening the metadata-free `PersistedRow` derive matters because the
  schema-aware path had already moved further off CBOR, but older derive-owned
  typed rows still needed a practical way to leave the fallback seam without
  pretending ambiguous schema facts could be guessed from Rust types alone
* the decimal hint keeps that line honest:
  * exact kinds are inferred where the byte contract is unambiguous
  * explicit hints are required where schema policy, such as decimal scale,
    must not be guessed
* centralizing the remaining runtime DB CBOR helpers under `db::codec::cbor`
  in `0.83.0` made this follow-up easier to land because the remaining owners
  were already explicit instead of scattered

What remains:

* the main remaining runtime CBOR surface is the structural fallback path in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs`
* that is a materially harder owner than cursor tokens or migration state,
  because it still covers generic structural payload decode for arbitrary
  persisted field shapes
* in practice that fallback is now narrower than it was at the start of
  `0.83.1`, because more metadata-free derive cases can route through exact
  by-kind storage contracts before they ever need the compatibility seam
* the generic serializer module itself still exists for non-DB owners and test
  parity, but production DB/runtime reliance on it is now much smaller than it
  was at the start of the patch
