## 0.71.0

`0.71.0` starts the aggregate execution simplification line by making
prepared aggregate strategy the single behavioral source across the first set
of scalar aggregate domains, without changing planner-visible index policy or
aggregate correctness rules.

What landed in this first patch:

- typed SQL scalar aggregate lowering now prepares one canonical scalar
  strategy that carries aggregate expression, resolved target-slot ownership,
  domain, empty-set behavior, descriptor shape, and runtime descriptor in
  [mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs)
- typed SQL aggregate execution and SQL `EXPLAIN EXECUTION` now project from
  that prepared strategy instead of rediscovering runtime and descriptor shape
  from raw aggregate terminals in
  [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/aggregate.rs),
  [explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/explain.rs), and
  [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/aggregate.rs)
- fluent scalar existing-row and scalar-id/extrema terminals now prepare one
  fluent scalar strategy before execution or explain in
  [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/builder/aggregate.rs),
  [terminals.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fluent/load/terminals.rs),
  [query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs), and
  [query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs)
- fluent numeric `SUM/AVG` terminals now prepare one numeric-field strategy at
  the builder boundary and then project execution from that prepared request
  instead of open-coding numeric boundary selection inline
- fluent order-sensitive terminals now prepare one explicit response-order or
  field-order strategy for `first`, `last`, `nth_by`, `median_by`, and
  `min_max_by` before execution, with `first/last` explain also consuming that
  prepared order-sensitive shape
- fluent projection/distinct terminals now prepare one projection strategy for
  `values_by`, `distinct_values_by`, `count_distinct_by`, `values_by_with_ids`,
  `first_value_by`, and `last_value_by` instead of constructing raw projection
  boundary requests inline

What did not change:

- no planner-visible index policy
- no aggregate eligibility widening
- no grouped/global distinct semantic changes
- no new aggregate correctness or authority model

This leaves `0.71` with explicit prepared-strategy ownership across:

- typed scalar
- fluent scalar
- numeric
- order-sensitive
- projection/distinct

The remaining work for later `0.71.x` patches is packaging and any narrower
follow-up cleanup inside individual aggregate domains, not another planner or
correctness redesign.
