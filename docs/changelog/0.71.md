## 0.71.2

`0.71.2` keeps the `0.71` aggregate-execution normalization line inside one
local structural seam: slot-only decoded rows now stay sparse deeper into the
aggregate runtime instead of repeatedly collapsing back to dense slot vectors
or full `KernelRow` materialization.

What landed in this patch:

- [page.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/page.rs)
  now defines `RetainedSlotRow`, and `KernelRow` now carries either dense
  slots or a retained sparse slot-row contract instead of assuming one dense
  slot vector shape everywhere
- [row_decode.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/row_decode.rs)
  now exposes retained-slot decode helpers, including direct single-slot
  extraction, so slot-only consumers do not need to decode a full row-shaped
  slot vector just to read one field
- [inputs.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs)
  and [runtime.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/operators/terminal/runtime.rs)
  now preserve retained slot rows through the slot-only kernel and structural
  cursor page, including the SQL constant-covering and route-owned covering
  slot-row short paths
- [structural.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/materialize/structural.rs)
  now projects from retained slot rows directly, so SQL structural projection
  materialization no longer forces those rows back into dense vectors first
- aggregate projection, bytes, extrema, helper, and numeric paths now use the
  retained-slot decode pattern where they only need one field value:
  [projection/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/projection/mod.rs),
  [bytes.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/bytes.rs),
  [field_extrema.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/field_extrema.rs),
  [helpers.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/helpers.rs), and
  [numeric/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/numeric/mod.rs)
- this patch was measured against clean `v0.71.1` with the existing PocketIC
  delta harness, and the retained-slot path produced real wins on the
  aggregate surfaces it targeted, including:
  - typed SQL aggregate `COUNT(field)`: about `-1.06%`
  - typed SQL aggregate `SUM(field)` / `AVG(field)`: about `-0.60%`
  - typed SQL aggregate `MIN(field)` / `MAX(field)`: about `-1.62%` to `-1.64%`
  - fluent aggregate `EXPLAIN sum_by(...)` / `avg_distinct_by(...)`: about `-1.07%` to `-1.09%`

What did not change:

- no planner-visible index policy
- no aggregate eligibility widening
- no grouped/global distinct redesign
- no new aggregate behavior or result-shape semantics

This leaves the next `0.71.x` work, if any, as release packaging or another
measured local cleanup, not a broader aggregate redesign.

## 0.71.1

`0.71.1` follows the same aggregate-execution normalization line as
`0.71.0`, but keeps the work on public explain coverage and facade parity
instead of widening aggregate policy or planner behavior.

What landed in this patch:

- fluent aggregate explain now projects through one shared prepared explain
  contract for the explain-visible aggregate families:
  existing-row, scalar id/extrema, numeric, and explain-visible
  order-sensitive terminals
- fluent numeric terminals now expose public explain entrypoints for
  `sum_by`, `sum_distinct_by`, `avg_by`, and `avg_distinct_by`, with parity
  locks proving those methods project from the same prepared numeric
  strategy used by the internal explain helper
- projection/distinct terminals now expose public explain entrypoints for
  `count_distinct_by`, `values_by`, `distinct_values_by`,
  `values_by_with_ids`, `first_value_by`, and `last_value_by`
- projection/distinct explain intentionally uses
  `ExplainExecutionNodeDescriptor` instead of `ExplainAggregateTerminalPlan`,
  because that family is still a projection runtime with terminal metadata,
  not an aggregate-terminal route family
- the prepared projection strategy now owns stable explain labels for
  projection terminal name and output shape, so the public fluent explain
  methods do not rebuild that metadata separately
- the public `icydb` facade now re-exports the aggregate/planning explain
  types it already depends on and wires fluent load/delete wrappers back up
  to `explain()`, `planned()`, and `plan()`, so facade callers no longer lose
  planning introspection that core already exposes
- the facade query helper surface now re-exports `avg(...)` alongside the
  other aggregate builder helpers, closing the last obvious aggregate-builder
  mismatch at the public wrapper boundary
- the public aggregate explain perf harness now measures the new fluent
  explain entrypoints for numeric and projection/distinct terminals, so those
  surfaces are covered in parity and PocketIC instead of being facade-only
  additions without instruction-footprint visibility

What did not change:

- no planner-visible index policy
- no aggregate eligibility widening
- no grouped/global distinct semantic changes
- no new aggregate correctness model

This leaves the next `0.71.x` follow-up space for narrower measured work
inside one aggregate domain at a time, not another structural aggregate
taxonomy pass.

## 0.71.0

`0.71.0` starts the aggregate execution simplification line by making
prepared aggregate strategy the single behavioral source across the first set
of scalar aggregate domains, without changing planner-visible index policy or
aggregate correctness rules.

What landed in this first patch:

- typed SQL scalar aggregate lowering now prepares one canonical scalar
  strategy that carries aggregate expression, resolved target-slot ownership,
  domain, empty-set behavior, descriptor shape, and runtime descriptor in
  [mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs)
- typed SQL aggregate execution and SQL `EXPLAIN EXECUTION` now project from
  that prepared strategy instead of rediscovering runtime and descriptor shape
  from raw aggregate terminals in
  [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/aggregate.rs),
  [explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/explain.rs), and
  [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/aggregate.rs)
- fluent scalar existing-row and scalar-id/extrema terminals now prepare one
  fluent scalar strategy before execution or explain in
  [aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/builder/aggregate.rs),
  [terminals.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fluent/load/terminals.rs),
  [query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs), and
  [query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs)
- fluent numeric `SUM/AVG` terminals now prepare one numeric-field strategy at
  the builder boundary and then project execution from that prepared request
  instead of open-coding numeric boundary selection inline
- fluent order-sensitive terminals now prepare one explicit response-order or
  field-order strategy for `first`, `last`, `nth_by`, `median_by`, and
  `min_max_by` before execution, with `first/last` explain also consuming that
  prepared order-sensitive shape
- fluent projection/distinct terminals now prepare one projection strategy for
  `values_by`, `distinct_values_by`, `count_distinct_by`, `values_by_with_ids`,
  `first_value_by`, and `last_value_by` instead of constructing raw projection
  boundary requests inline

What did not change:

- no planner-visible index policy
- no aggregate eligibility widening
- no grouped/global distinct semantic changes
- no new aggregate correctness or authority model

This leaves `0.71` with explicit prepared-strategy ownership across:

- typed scalar
- fluent scalar
- numeric
- order-sensitive
- projection/distinct

The remaining work for later `0.71.x` patches is packaging and any narrower
follow-up cleanup inside individual aggregate domains, not another planner or
correctness redesign.
