# 0.98 Changelog

## 0.98.2

- `0.98.2` is a bounded follow-through on the internal prepared-SQL template lane. It does not widen the admitted `?` syntax surface; it proves and extends the fixed-route reuse story inside the already-supported compare-family boundary.
- the template-backed grouped aggregate compare lane now covers both field-backed `MIN(field)` and `MAX(field)` `HAVING` contracts across numeric and text-backed aggregate inputs:
  - `HAVING MIN(age) > ?`
  - `HAVING MAX(age) < ?`
  - `HAVING MIN(name) > ?`
  - `HAVING MAX(name) < ?`
  - these shapes now freeze the same safe field-kind template bindings used by ordinary field compares instead of preparing successfully and then needlessly missing the fixed-route lane
- multi-slot prepared queries now have explicit template-backed coverage on both the scalar and grouped paths:
  - scalar:
    - `WHERE age > ? AND name > ?`
  - grouped:
    - `HAVING COUNT(*) > ? AND name > ?`
  - these proofs matter because the current internal template mechanism depends on collision-resistant per-slot sentinel replacement; the new tests lock that repeated executions with different bindings keep producing correct row sets without touching the shared structural query-plan cache
- the grouped `NULL` fallback boundary is now pinned directly:
  - admitted grouped prepared shapes still prepare successfully with `?`
  - `NULL` bindings continue to take the documented fallback path instead of the internal template lane
  - grouped `HAVING COUNT(*) > NULL` now has a live session test proving ordinary SQL three-valued boolean semantics are preserved and the prepared path still bypasses the raw SQL compiled-command cache
- one tempting widening was checked and intentionally not taken:
  - bool compare contracts are admitted at prepare time, but the current template lane still relies on collision-resistant non-null sentinel replacement
  - bool does not offer enough distinct non-null values to make that a safe small patch for multi-slot prepared queries
  - `0.98.2` therefore keeps bool on the admitted-but-non-template side instead of claiming brittle reuse
- that admitted-but-non-template bool boundary is now pinned directly too:
  - `WHERE active = ?` prepares successfully with one `Bool` contract
  - repeated executions with `true` and `false` bindings still return the correct rows
  - the prepared path continues to bypass the raw SQL compiled-command cache even though bool stays outside the fixed-route template lane
- the internal template lane now has one extra safety guard:
  - template build refuses the fixed-route path when one chosen sentinel already appears as an ordinary SQL literal in the prepared statement
  - this prevents the current literal-equality replacement step from accidentally rebinding a user-authored constant that happens to equal one internal template value
  - both text and numeric scalar sentinel-collision regressions are now pinned directly with live prepared-SQL tests
  - the same grouped text-collision boundary is pinned too, so grouped `HAVING` fallback stays correct when one user-authored literal equals the second text-slot sentinel
- focused verification for `0.98.2` covered the newly widened template proofs, grouped `NULL` fallback semantics, and normal hygiene:
  - `cargo fmt --all`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_max_aggregate_compare_contracts -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_text_min_aggregate_compare_contracts -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_text_max_aggregate_compare_contracts -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_mixed_numeric_and_text_compare_contracts -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_grouped_mixed_having_compare_contracts -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_bool_compare_contracts_preserve_fallback_results -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_grouped_null_binding_preserves_having_semantics -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_falls_back_when_template_text_sentinel_collides_with_user_literal -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_falls_back_when_template_numeric_sentinel_collides_with_user_literal -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_grouped_falls_back_when_template_text_sentinel_collides_with_user_literal -- --nocapture`

## 0.98.1

- `0.98.1` is the first real follow-through on the `0.98` prepared SQL line: supported parameterized `SELECT` queries no longer have to route back through bound-literal SQL compilation on every execution.
- prepare now attempts to freeze one internal execution template for the admitted fixed-route subset:
  - the template stores one planner-owned `AccessPlannedQuery` built from collision-resistant non-null placeholder sentinel values
  - the outward SQL projection contract is frozen alongside that plan so repeated executions can stay on the SQL statement surface without rebuilding labels or grouped statement shaping
  - this template is intentionally internal and does not widen the public `DbSession` API yet
- execute now has two explicit parameterized paths instead of one:
  - fast path:
    - validate bindings against the frozen slot contracts
    - patch the prepared template’s predicate, grouped `HAVING`, aggregate filter/input expressions, and access payload literals with the current runtime bindings
    - rebuild the planner-frozen execution residents from that bound plan
    - execute directly from the bound prepared plan without re-entering the raw SQL compiled-command cache or the shared structural query-plan cache
  - fallback path:
    - keep the `0.98.0` literal-rebind behavior for shapes that do not yet expose a safe internal template
    - this keeps the slice fail-closed and avoids over-claiming fixed-route reuse for still-deferred families
- the current `0.98.1` fast path is still deliberately narrower than the full long-term parameterization design:
  - it covers the admitted compare-family shapes where every slot can be assigned one safe non-null template binding
  - `NULL` bindings still take the fallback path
  - unsupported/value-sensitive families remain prepare-time rejections exactly as in `0.98.0`
  - this means the shipped win is real for the current numeric/text compare lane, but it does not yet generalize to every admitted prepared slot contract
- cache behavior is now materially different from `0.98.0` for the supported template lane:
  - repeated prepared executions still bypass the raw SQL compiled-command cache
  - they now also avoid populating or depending on the shared structural query-plan cache for each bound execution
  - the internal reuse story therefore moves from “stable prepare + safe bind validation” to “stable prepare + direct template-backed execution” for the admitted fixed-route subset
- lowering contracts now carry one extra piece of prepare-time metadata per admitted slot:
  - each `PreparedSqlParameterContract` can expose one optional internal template binding
  - this is not part of user-facing SQL semantics; it exists only so the session layer can build one collision-resistant prepared execution shell for the admitted subset
  - value families without a safe template binding stay on the fallback path instead of silently claiming fixed-plan reuse
- focused verification for `0.98.1` covered the new template-backed execution path and the tightened cache expectations:
  - `cargo fmt --all`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`
  - `cargo test -p icydb-core sql_parameterized -- --nocapture`

## 0.98.0

- `0.98.0` does not ship the full parameterized SQL feature yet. It lands the first internal groundwork needed for the `0.98` line while keeping the admitted surface deliberately narrow and fail-closed.
- the parser and shared SQL token layer now understand positional `?` placeholders:
  - `?` is tokenized explicitly instead of failing in the lexer
  - parsed SQL expressions now preserve stable placeholder slot order as `SqlExpr::Param { index }`
  - normalization and aggregate-discovery passes preserve those parameter nodes instead of dropping or rewriting them
- the new session-owned groundwork lives behind an internal SQL boundary:
  - one prepared query shape records the source SQL, normalized prepared statement, and frozen per-slot bind contracts
  - each admitted slot freezes:
    - slot index
    - coarse runtime type family
    - null-admission behavior
  - this keeps prepare-time validation and bind-time validation separate without widening the public `DbSession` surface yet
- the admitted `0.98.0` groundwork surface is intentionally smaller than the final design target:
  - supported:
    - right-hand compare parameters in scalar `WHERE`
    - right-hand compare parameters in compare-style grouped `HAVING`
  - rejected fail-closed:
    - projection expressions
    - aggregate input and aggregate `FILTER`
    - `ORDER BY`
    - bare parameter expressions
    - arithmetic parameter expressions
    - non-field compare shapes such as function-backed `WHERE LOWER(name) = ?`
    - value-sensitive families still outside the current prepare path
- execution currently binds parameter values back into SQL literals before the existing lowering/planning/runtime path:
  - this lets the groundwork reuse the shipped SQL lowering and execution model without introducing planner-wide parameter nodes yet
  - bind validation happens first, so mismatched runtime values fail before execution
  - `NULL` bindings are admitted for the current compare-family surface and then flow through ordinary SQL evaluation semantics
- an important cache-safety boundary is now locked explicitly:
  - the new prepared SQL path bypasses the raw SQL compiled-command cache
  - different bindings for the same prepared source SQL therefore do not alias onto one cached compiled SQL artifact
  - repeated prepared executions still produce correct per-bind results instead of silently reusing an old literal-bound command
- `0.98.0` is still groundwork rather than the final prepared-query slice:
  - it does not yet freeze one structural logical/access plan across different bindings
  - because parameters are rebound to literals before structural planning, the shared structural plan cache can still see different bound values as different structural queries
  - the broader fixed-route prepared-plan contract remains follow-on work on top of this parser/session foundation
- focused verification for `0.98.0` covered parser stability, admitted prepare-time contracts, rejection boundaries, bind validation, null handling, raw-SQL cache safety, and formatting:
  - `cargo fmt --all`
  - `cargo test -p icydb-core parse_sql_preserves_placeholder_slot_order_across_where_and_having -- --nocapture`
  - `cargo test -p icydb-core prepare_sql_query -- --nocapture`
  - `cargo test -p icydb-core sql_parameterized -- --nocapture`
