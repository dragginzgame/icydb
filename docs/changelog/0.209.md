# IcyDB 0.209 Changelog

## 0.209.1 - 2026-07-22

### 🩹 Fixed

- Preserves resolved DDL-owned field values during unrelated typed updates.
  Dropping a required field's insert default now affects future omission only;
  existing logical values remain frozen while old rows promote to the current
  layout.
- Preserves exact layout-window and physical slot-count corruption diagnostics
  through typed response decoding instead of collapsing them into a generic
  persisted-row failure.
- Keeps metadata-only `SET DEFAULT` and `DROP DEFAULT` available when the
  entity-local row-layout version has reached its maximum, while physical
  shape changes retain their typed exhaustion rejection.

### 🧹 Cleanup

- Removes the split/rejoin row-emission flow. Sanitized generated values and
  already-resolved DDL-owned values now form one complete canonical row before
  commit preparation.
- Corrects stale design links, phase-style terminology, import/restore wording,
  and closeout evidence claims without introducing compatibility behavior.

### 🧪 Testing

- Adds one journaled three-layout sequence covering changed and dropped
  defaults, unrelated updates, natural promotion, future omission rejection,
  and byte-exact recovery of rows, indexes, and accepted schema.
- Adds typed-terminal corruption coverage and maximum-layout metadata-DDL
  boundary coverage. The package-scoped all-feature core run passes 4,430
  active tests with nine intentionally ignored tests unchanged.

### 📊 Audit

- Production source is approximately nine lines smaller after deleting the
  duplicate row merger. The complete correction slice adds 558 and removes
  132 lines excluding unrelated user-owned lockfile and 0.210 documentation
  changes; most growth is focused tests and corrected design evidence.
- Public APIs, cursor encoding, accepted-schema encoding, and persisted-row
  format are unchanged. Instruction and raw non-gzipped Wasm deltas were not
  measured for this correctness patch.

## 0.209.0 - 2026-07-22

### 📝 Summary

- Separates database defaults into two independent accepted-schema facts:
  future insert policy and frozen historical fill.
- Gives every persisted row an exact entity-local row-layout stamp, allowing
  legitimate historical absence to be distinguished from current-row
  corruption without inferring history from row length.
- Converges generated create, typed, structural, SQL, and batch writes on one
  accepted-schema after-image resolver that preserves the provenance of
  authored, defaulted, generated, managed, preserved, and historical values.
- Makes SQL default changes future-only while preserving the logical values of
  rows written under older admitted layouts.

### ⚠️ Breaking

- Accepted schemas now persist separate `SchemaInsertDefault` and
  `SchemaHistoricalFill` contracts plus entity-local row-layout history. The
  former single-default accepted representation is removed.
- Persisted rows now use the current stamped row envelope. Pre-0.209 unstamped
  rows and accepted-schema encodings fail closed; there is no compatibility
  decoder, dual-read path, migration shim, or upgrade-on-read behavior.
- Generated create inputs carry authored application fields only. They no
  longer construct a complete Rust entity or evaluate Rust `Default` before
  accepted database policy resolves omission.
- Schema-description output now exposes insert policy, historical fill,
  introduction layout, current layout, history floor, and exact ordered
  primary-key fields directly. The older default-in-type-string and
  scalar-compatible constructor shapes are removed.
- Physical-shape changes such as `DROP COLUMN` are admitted only for an entity
  proven exactly empty. A nonempty rewrite rejects before staging with the
  typed migration-required error; 0.209 contains no partial or resumable
  schema-row rewrite.

### ➕ Added

- Adds checked `RowLayoutVersion` identities independent of general schema
  versions. Layout versions are entity-local, monotonic, nonzero, never reused,
  and fail with a typed exhaustion cause.
- Adds an exact layout stamp to every canonical row and accepted current/floor
  layout facts to every entity snapshot. Each field records the layout in
  which it was introduced.
- Adds explicit write intents for omission, insert `DEFAULT`, update
  `DEFAULT`, authored values, preservation, generation, management, and frozen
  historical fill.
- Adds SQL support for value-position `DEFAULT`, `DEFAULT VALUES`, and update
  assignment `SET field = DEFAULT` over the shared accepted resolver.
- Adds bounded typed diagnostics for required-field omission, layout-window
  violations, physical slot-count mismatch, row-layout exhaustion,
  generated-after-DDL slot collision, transition-budget exhaustion, and
  migration-required schema rewrites.
- Adds accepted-native layout and slot verification facts to the canonical
  structural reader for later integrity and constraint consumers without a
  second row-decoding owner.

### 🔧 Changed - Temporal Semantics

- Changing or dropping an insert default affects future writes only. It never
  reinterprets an untouched historical row.
- Additive DDL records one canonical historical fill for rows whose admitted
  layout predates the field. Nullable additions freeze `NULL`; defaulted
  additions freeze the canonical accepted payload.
- A field can use historical fill only when its introduction layout is newer
  than the row stamp. Missing initial fields and missing slots in a current
  layout are corruption regardless of nullability or default policy.
- Every current insert, replacement, update, batch mutation, and schema rewrite
  emits a complete row stamped with the current layout. Updating an old row
  preserves its frozen logical values while naturally promoting its physical
  representation to the current layout.
- History-floor advancement and the corresponding closure of unreachable
  historical fills occur in the same accepted candidate.

### 🔧 Changed - Mutation Authority

- Generated create, full typed saves, structural mutations, SQL writes, and
  atomic batches all feed the same accepted after-image resolver instead of
  independently materializing omission or defaults.
- Application sanitizers may transform authored values only. Values supplied
  by accepted defaults, historical fill, generation, engine management, or
  preservation retain their provenance across typed materialization.
- Generated primary-key identity remains stable during replacement, including
  pre-marker failure and rollback. Generated replacement no longer risks
  treating a newly generated key as the identity of an existing row.
- Rust `Default`, `Option::default`, derived entity defaults, and Serde defaults
  remain application construction behavior only; none grants database
  omission rights or repairs malformed persisted rows.

### 🔧 Changed - DDL And Derived State

- `SET DEFAULT` and `DROP DEFAULT` are metadata-only, scan-free, and do not
  bump row-layout version. A required field may intentionally have no insert
  default, making future omission reject.
- Nullable `SET DEFAULT NULL` canonicalizes to no insert default; non-nullable
  use rejects instead of storing ambiguous policy.
- `ADD COLUMN` allocates one new physical layout and either freezes historical
  fill or proves the entity empty. Required additions without a valid fill
  reject over existing rows.
- Generated default reconciliation changes future insertion policy without
  rewriting history or advancing the physical layout.
- Candidate unique-index and reverse-relation effects are staged from the
  final candidate catalog and complete logical row view through the existing
  marker-bound publication owner. Missing relation targets, uniqueness
  conflicts, and bounded staging exhaustion reject without publishing partial
  schema or derived state.
- Shared named transition budgets cover row, byte, schema, marker, journal,
  and staged-effect resources before publication.

### 🔧 Changed - Persistence And Recovery

- Row decoding first authenticates the current envelope and layout stamp, then
  requires the exact physical slot count declared by that admitted layout.
  Payload length is validation evidence, never historical authority.
- Accepted schema validation enforces both directions of historical-fill
  closure and rejects invalid layout windows, slot contracts, noncanonical
  payloads, and generated/DDL slot collisions.
- Recovery replays already-resolved marker-bound bytes. It does not reevaluate
  defaults, generation, management, sanitizers, or historical fills under the
  latest policy.
- Logical imports cannot claim historical layouts. No public raw physical
  restore ingress was added; any future restore must authenticate its source
  schema, entity, layout, backup, chunk hash, and key domain.
- Schema, plan, and relevant identity fingerprints now include the temporal
  facts on which execution depends.

### 🔧 Changed - SQL And Introspection

- Direct insert `DEFAULT`, `DEFAULT VALUES`, and update `DEFAULT` preserve
  their distinct intent through parsing, lowering, accepted resolution, and
  execution. Unsupported expression positions reject instead of degrading to
  `NULL`, omission, or another literal.
- Explicit `NULL`, authored zero/false/empty values, omission, and `DEFAULT`
  remain distinct at every write ingress.
- `DESCRIBE` and `SHOW COLUMNS` render bounded canonical insert defaults and
  frozen historical fills separately, together with layout introduction and
  entity history information.
- CLI and facade diagnostic rendering cover every new runtime-boundary code
  exhaustively while preserving the registry-owned broad diagnostic class.

### 🩹 Fixed

- Preserves generated replacement keys and complete secondary-index bytes when
  a replacement fails before publication.
- Rejects generated fields that would collide with DDL-owned accepted slots
  through a typed transition cause rather than a generic internal error.
- Preserves distinct required-field, layout-window, and slot-count causes
  through public error projection and CLI rendering.
- Keeps shared row-fixture helpers available without optional SQL support and
  repairs fixed-boundary tests without weakening generated-key semantics.
- Aligns the live canister DROP COLUMN contract with 0.209: nonempty entities
  return the typed migration-required rejection and leave accepted schema
  visibility unchanged.

### 🗑️ Removed

- Removes the ambiguous single database-default owner and every runtime path
  that used current default metadata to synthesize historical missing slots.
- Removes unstamped-row decoding, old accepted-schema forms, row-length history
  inference, and the retired compatibility corpus breadcrumb.
- Removes the serialized structural-patch DTO/replay layer and its test-only
  materialization path after accepted mutation resolution became authoritative.
- Removes the ordinary public re-export of macro-only generated create
  transport and the scalar-compatible schema-description constructor.
- Removes parallel staged derived-state collections, duplicate transition
  limits, redundant identity transport, discarded production reports, and
  error detail that carried no current invariant.
- Compacts internal error-class and error-origin axes to one byte while
  retaining every typed cause and numeric diagnostic value.

### 🧭 Migration Notes

- Recreate pre-0.209 development stores. This release intentionally does not
  decode or translate the former accepted schema or unstamped row envelope.
- Regenerate schema-derived code so create inputs contain authored fields and
  database omission is resolved by accepted schema after ingress.
- Update schema-description consumers to read explicit insert-default,
  historical-fill, and row-layout fields rather than parsing defaults from a
  rendered type string.
- Application constructors must continue to provide authored required values.
  Add `traits(add(Default))` only when the Rust application type itself should
  implement `Default`; it has no effect on IcyDB insertion policy.
- Plan nonempty physical schema rewrites through a later migration facility.
  Do not retry `DROP COLUMN` through an alias or unchecked path.
- The ordinary query architecture, cursor encoding, and pagination contracts
  are unchanged by 0.209.

### 🧪 Testing

- A combined journaled temporal sequence proves add-default, change-default,
  drop-default, unrelated updates across three admitted historical eras,
  natural row promotion, and byte-exact recovery replay. Separate focused
  coverage proves atomic history-floor advancement.
- Mutation parity covers generated create, typed save/replace, structural
  patch, SQL insert/update/default forms, atomic and non-atomic batches, and
  rejection without partial row, index, relation, schema, or marker effects.
- Persisted-format coverage includes unknown and out-of-window layout stamps,
  wrong slot counts, truncated envelopes, missing initial/current fields,
  noncanonical fills, future versions, exact typed-terminal corruption codes,
  and checked layout exhaustion.
- DDL coverage includes empty physical changes, nonempty migration-required
  rejection, metadata-only default changes at maximum layout version,
  additive unique-index staging, reverse-relation effects, and every named
  transition budget.
- Package-scoped all-feature `icydb-core` validation passes 4,430 active tests;
  nine intentionally ignored tests remain unchanged. Native and Wasm feature
  checks, warnings-denied Clippy, formatting, diff hygiene,
  generated-surface tests,
  diagnostic round trips, and repository invariant guards pass.
- Full workspace and repository suites remain the human-owned push validation
  boundary under repository policy.

### 📊 Audit

- The 0.209 implementation, excluding unrelated design archival and 0.214
  documentation, touches 128 files with approximately 7,549 additions and
  4,473 deletions. The additional code primarily encodes the new persisted
  identity, fail-closed validation, and temporal boundary tests; write-policy
  ownership is simpler because frontend-specific default resolution is gone.
- The stamped row header grows from 7 to 11 bytes: exactly four raw bytes per
  row. A measured one-field `Nat64` accepted snapshot with a three-byte default
  grows from 950 to 996 bytes.
- The SQL performance canister grows from 3,938,804 to 3,983,287 raw
  non-gzipped Wasm bytes (+44,483, 1.13%) for the temporal format and policy
  implementation. Its shrunk raw artifact grows by 41,918 bytes (1.14%).
- Subsequent closeout cleanup reduces the representative `demo_rpg` SQL-on
  raw Wasm from 3,502,786 to 3,390,501 bytes (-112,285), the shrunk raw artifact
  by 112,492 bytes, and the linked function count by 25 without removing typed
  errors or invariants.
- Peak heap bytes, allocator traffic, and stable-memory byte volume were not
  measured. Maintained instruction sentinels, exact row/catalog byte evidence,
  and raw non-gzipped Wasm are the recorded 0.209 performance evidence.
