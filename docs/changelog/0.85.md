# 0.85 — Grouped Post-Aggregate Projection Expressions

## 0.85.3 - 2026-04-16 - Execution Input Ownership Tightening

This patch keeps `0.85` on executor cleanup rather than widening grouped SQL
further. The follow-through here is narrower than `0.85.2`: the scalar and
fast-path execution spine now pushes more ownership onto the prepared
execution-input boundary, and correspondingly leaves `ExecutionKernel` with a
smaller, clearer role.

What changed:

* the scalar route-attempt boundary now lives under prepared execution inputs
  instead of being assembled partly in the kernel:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/kernel/mod.rs`
* `ExecutionInputs` now owns:
  * raw key-stream resolution without DISTINCT decoration
  * DISTINCT-decorated key-stream resolution
  * single-attempt route materialization
  * aligned row-collector versus runtime-page scalar materialization
* the kernel boundary was reduced accordingly:
  * `ExecutionKernel` now mainly owns residual retry policy and retry-attempt
    merge accounting in:
    * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/kernel/mod.rs`
* the fast-path runtime helper now has explicit local ownership seams instead
  of one long mixed policy function:
  * `ResolvedIndexPredicateProgram` owns reuse-versus-on-demand index predicate
    compilation
  * `FastPathResolutionContext` owns fast-path-hit versus fallback stream
    resolution
  * both now live in:
    * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/runtime/fast_path/mod.rs`
* nearby aggregate and grouped-fold callers now consume the
  execution-input-owned stream resolution boundary directly instead of calling
  back into the kernel wrapper in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/field_extrema.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/grouped_fold/mod.rs`

Why this matters:

* the prepared execution-input boundary is now the obvious owner of “resolve
  one stream, decorate it, and materialize one attempt”
* the kernel no longer has to know about row-collector versus canonical page
  materialization or about DISTINCT decoration
* fast-path resolution is easier to audit because compile-mode policy and
  hit-versus-fallback policy are now explicit local helper types instead of
  inline mixed state

Verification:

* `cargo fmt --all`
* `cargo check -p icydb-core`

## 0.85.2 - 2026-04-16 - Grouped and Executor Boundary Cleanup

This patch keeps `0.85` on cleanup rather than widening grouped SQL further.
It follows the grouped post-aggregate projection slice by tightening the two
internal boundaries that were left looking transitional after `0.85.0` and
`0.85.1`: grouped aggregate/projection ownership, and the scalar
route-materialization versus residual-retry split.

What changed:

* grouped planner handoff no longer walks grouped projection expressions twice
  to decide whether aggregates are present and then collect them again; one
  planner-local grouped aggregate scan now owns aggregate-bearing
  classification, first-seen dedupe, and introduced aggregate-slot counting in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs`
* grouped executor projection ownership is tighter and less duplicated:
  * the compiled grouped projection contract now lives under the projection
    boundary instead of the grouped-output runtime boundary in:
    * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/grouped.rs`
  * grouped output materialization and grouped page finalization both consume
    that same contract instead of open-coding the identity short-circuit and
    compile-or-skip path in:
    * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/grouped_output/mod.rs`
    * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/grouped_fold/page_finalize.rs`
* the scalar residual retry split now reads as one explicit route-attempt
  materializer plus one retry session, instead of loosely coupled context and
  retry helper names in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/kernel/mod.rs`
* the shared scalar materialization contract now borrows its row-collector and
  runtime-page request builders instead of consuming itself, which removes the
  move-sensitive contract edge introduced during the residual retry split in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs`

Why this matters:

* grouped post-aggregate projection is now carried by clearer planner and
  executor owners instead of a mix of transitional helpers
* grouped projection compilation authority sits with the projection evaluator,
  which is the correct long-term boundary for later grouped-expression work
* the scalar retry/materialization split is easier to follow and less fragile
  under later executor cleanup because the route-attempt and retry roles are
  now named directly

## 0.85.0 - 2026-04-16 - Grouped SELECT Expression Widening

This patch opens the `0.85` line by widening grouped SQL projection from the
older `group key + aggregate leaf` model to a bounded post-aggregate
expression model. The grouped executor still owns grouping, aggregate state,
and aggregate finalization exactly as before. The new part is that grouped SQL
projection can now evaluate admitted scalar expression trees over grouped key
values and finalized aggregate outputs.

That means grouped SQL now accepts projection shapes like:

```sql
SELECT age, COUNT(*) + MAX(age), ROUND(AVG(age), 2)
FROM Character
GROUP BY age
ORDER BY age ASC
LIMIT 10;
```

What changed:

* grouped SQL parser projection operands now admit aggregate leaves inside the
  bounded arithmetic and `ROUND(...)` projection grammar in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/model.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/projection.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/statement/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/clauses.rs`
* grouped SQL lowering now accepts grouped projection expressions that contain
  aggregate leaves as long as ordinary field references still stay inside the
  declared `GROUP BY` key authority in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/projection.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/aggregate.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/normalize.rs`
* grouped planner handoff now collects aggregate leaves from nested grouped
  projection expressions and carries them in first-seen unique order so the
  grouped executor can resolve repeated aggregate references against one stable
  aggregate output contract in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/grouped_layout.rs`
* grouped fold now sizes reducer state from the frozen grouped aggregate
  execution spec list rather than from the count of aggregate-side projection
  positions, which is the correct contract once one projection item can
  reference more than one aggregate leaf in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/grouped_fold/mod.rs`
* parser fixtures, grouped lowering coverage, planner handoff coverage, and
  grouped execution coverage now lock the new surface in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/tests/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/tests/group.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_grouped.rs`
* the scalar structural page materializer was also split into explicit
  executor-owned child boundaries so the root page module no longer carries
  retained-slot storage, metrics, scan kernels, post-scan shaping, and plan
  resolution in one file:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/page/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/page/retained.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/page/metrics.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/page/scan.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/page/post_scan.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/terminal/page/plan.rs`
* that page split also finishes the scalar seam-extraction line by freezing
  more of the page executor onto explicit resolved contracts instead of late
  flag interpretation:
  * `ScalarMaterializationPlan`
  * `CursorlessShortPathPlan`
  * `KernelRowScanStrategy`
  * `StructuralPostScanTailStrategy`
  * shared cursorless short-path planning and structural payload finalization
* the result is not a behavior change in SQL or fluent reads, but a narrower
  executor ownership model where:
  * request inputs provide capabilities
  * planning resolves one scalar execution contract
  * scan executes one resolved kernel strategy
  * post-scan and final payload shaping run through one shared tail boundary

What is intentionally still out of scope in `0.85.0`:

* grouped `HAVING` remains symbol-based and is not widened to general
  post-aggregate expressions
* grouped `ORDER BY` keeps the existing bounded grouped-key rules and is not
  widened to post-aggregate expression ordering
* aggregate-input widening stays out of scope, so this line does not admit
  shapes like `AVG(ROUND(strength, 2))`

Why this matters:

* grouped SQL can now express the common “format or combine aggregate outputs
  before emitting rows” use case without reopening grouped execution semantics
* the grouped executor remains the single owner of grouping and aggregate
  finalization
* aggregate normalization now happens on the grouped SQL / grouped handoff
  boundary, which leaves later `HAVING` work free to build on a stable
  aggregate-index contract instead of the older one-projection-item to
  one-aggregate assumption
