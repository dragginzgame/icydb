# 0.85 — Grouped Post-Aggregate Projection Expressions

## 0.85.0 - 2026-04-16 - Grouped SELECT Expression Widening

This patch opens the `0.85` line by widening grouped SQL projection from the
older `group key + aggregate leaf` model to a bounded post-aggregate
expression model. The grouped executor still owns grouping, aggregate state,
and aggregate finalization exactly as before. The new part is that grouped SQL
projection can now evaluate admitted scalar expression trees over grouped key
values and finalized aggregate outputs.

That means grouped SQL now accepts projection shapes like:

```sql
SELECT age, COUNT(*) + MAX(age), ROUND(AVG(age), 2)
FROM Character
GROUP BY age
ORDER BY age ASC
LIMIT 10;
```

What is intentionally still out of scope in `0.85.0`:

* grouped `HAVING` remains symbol-based and is not widened to general
  post-aggregate expressions
* grouped `ORDER BY` keeps the existing bounded grouped-key rules and is not
  widened to post-aggregate expression ordering
* aggregate-input widening stays out of scope, so this line does not admit
  shapes like `AVG(ROUND(strength, 2))`

Why this matters:

* grouped SQL can now express the common “format or combine aggregate outputs
  before emitting rows” use case without reopening grouped execution semantics
* the grouped executor remains the single owner of grouping and aggregate
  finalization
* aggregate normalization now happens on the grouped SQL / grouped handoff
  boundary, which leaves later `HAVING` work free to build on a stable
  aggregate-index contract instead of the older one-projection-item to
  one-aggregate assumption
