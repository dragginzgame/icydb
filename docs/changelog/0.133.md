# 0.133.x

## 0.133.1

`0.133.1` closes the diagnostics audit line by tightening the remaining
diagnostics ownership boundaries and removing the last small storage-report
duplication.

Storage reporting now runs both `storage_report_default` and `storage_report`
through one internal builder. The two public entrypoints still preserve their
historical labeling contracts, per-entity accumulation behavior, snapshot
ordering, corruption counters, DTO fields, and Candid field names, but the
shared traversal now owns store sorting, data/index snapshot collection,
corruption counting, and final `StorageReport` assembly in one place.

The final storage-report cleanup deduplicates the repeated
`EntitySnapshot::new` push branch inside the mode-specific entity-stat
emission path without changing iteration order or outward report contents.

The change lands in:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/diagnostics/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/diagnostics/mod.rs)
- [/home/adam/projects/icydb/crates/icydb-core/src/db/diagnostics/storage_report.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/diagnostics/storage_report.rs)

What changed:
- added one internal storage-report builder shared by default and explicit
  report entrypoints
- kept default and explicit entity labeling contracts separate through a small
  internal mode object
- deduplicated index corruption counting and entity snapshot pushing in storage
  reporting

Retained behavior:
- diagnostics report contents, ordering, accessors, constructors, and Candid
  field names are unchanged
- storage corruption counts are unchanged
- integrity scan behavior is unchanged

## 0.133.0

`0.133.0` starts the diagnostics and identity audit line with two boundary
cleanups.

`db::diagnostics` now exposes the full storage-report DTO surface through the
`db` module root. `StorageReport` already returned `DataStoreSnapshot`,
`IndexStoreSnapshot`, and `EntitySnapshot` from public accessors; those row
snapshot DTOs are now re-exported beside `StorageReport`, `IntegrityReport`,
`IntegrityStoreSnapshot`, and `IntegrityTotals`. The diagnostics constructors
used only by report assembly were narrowed to crate-internal visibility, so the
public surface stays focused on reading reports rather than externally
constructing report internals.

The diagnostics implementation was then split without changing behavior.
`diagnostics::model` owns report DTOs and simple accessors, `storage_report`
owns storage footprint snapshot collection, and `integrity` owns read-only
data/index integrity scanning plus the recovery-facing
`integrity_report_after_recovery` entrypoint. The module root now only wires
submodules and re-exports the existing public/crate surfaces.

The identity audit moves generated entity/index name validation onto the shared
`EntityName` and `IndexName` contracts. Entity names and index field segments
now reject the reserved `|` delimiter through the identity owner, index
identities reject empty field lists and empty field segments, and fixed-size
max sentinels use the maximum ASCII byte instead of a lowercase-letter
sentinel. The derive path now calls the same identity constructors used by the
runtime instead of repeating length, ASCII, and generated-name checks locally.
The persisted `IndexName::from_bytes` path is now documented as a canonical
byte-envelope decoder, not a structural segment validator, so future import or
migration tools have a clear trust-boundary rule if they ever accept identity
bytes that were not produced by `IndexName::to_bytes`.

What changed:
- re-exported `DataStoreSnapshot`, `IndexStoreSnapshot`, and `EntitySnapshot`
  from `db`
- narrowed diagnostics DTO constructors that are only used by diagnostics
  assembly and tests
- split `db::diagnostics` into model, storage-report, and integrity-scan owners
- kept `execution_trace` unchanged while moving large report implementation
  bodies out of the module root
- centralized entity and generated-index identity validation on
  `EntityName` / `IndexName`
- rejected delimiter-bearing entity names and index field segments before they
  can alias generated index identities
- rejected generated indexes with no fields or empty field segments at the
  identity boundary
- documented the `IndexName::try_from_parts` versus `IndexName::from_bytes`
  invariant split: structural construction is strict, persisted-byte decode
  validates only the fixed canonical envelope
- added sentinel drift tests proving `EntityName::max_storable()` and
  `IndexName::max_storable()` cover every currently accepted ASCII identity
  byte
- made `IndexId` fields private and exposed the entity-tag component through a
  named accessor
- removed stale schema validation error variants that duplicated identity-owned
  validation errors
- confirmed the removed `InvalidEntityName` / `InvalidIndexName` schema
  variants have no remaining callsites

Retained behavior:
- diagnostics report contents, ordering, accessors, and Candid field names are
  unchanged
- integrity scan behavior is unchanged
- execution trace behavior is unchanged
- persisted identity byte layout remains fixed-size and unchanged
- generated index names still use the same flat `entity|field|...` text form
- `IndexName::from_bytes` still accepts flat canonical bytes that may not be
  reconstructable through `try_from_parts`; callers must keep using it only for
  bytes that originated from validated identity construction
