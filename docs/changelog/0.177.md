# IcyDB 0.177 Changelog

## 0.177.9

### 📝 Summary

- Strengthens the 0.177 publication proof by showing first-contact
  reconciliation persists the generated declared schema version from the
  proposal-produced accepted snapshot, including non-default version `2`.
- Keeps runtime query/cache behavior, public API, DDL behavior, and persisted
  schema format semantics unchanged.

### 🧪 Testing

- Adds a reconciliation guard proving first-contact publication stores the
  declared version snapshot and does not synthesize an initial-version snapshot
  when generated code declares a later version.
- Tightens the source-boundary guard so SQL DDL and first-contact
  reconciliation pass accepted/proposal-produced snapshots into schema storage,
  and schema storage remains keyed by each snapshot's declared version without
  auto-incrementing.

## 0.177.8

### 📝 Summary

- Closes the generated-version runtime-boundary proof for 0.177 by showing
  source-declared `schema_version` values flow through schema proposal and
  admission, not runtime query/cache identity.
- Keeps runtime behavior, query/cache semantics, persisted schema format, and
  public API unchanged.

### 🧪 Testing

- Adds a source-boundary guard proving production session, query, executor,
  fingerprint, and accepted row-contract runtime code do not read generated
  `declared_schema_version()`.
- Pins the remaining generated-model schema proposal bridges for SQL cache-key
  tests, save executor tests, and accepted row-contract tests as explicit
  `#[cfg(test)]` helpers.

## 0.177.7

### 📝 Summary

- Hard-cuts persisted schema-version metadata to the explicit 0.177 contract:
  accepted schema snapshots must carry a positive declared schema version and
  obsolete snapshot codec versions fail closed.
- Keeps query/cache behavior unchanged; this slice only tightens schema
  persistence and accepted-authority validation.

### ⚠️ Breaking

- Persisted schema snapshots with `schema_version = 0` are now rejected at
  decode, store publication, and accepted snapshot construction boundaries.
- Obsolete persisted schema snapshot codec envelopes continue to reject with a
  clear unsupported-codec diagnostic rather than being inferred or upgraded.

### 🧪 Testing

- Adds codec coverage for obsolete schema snapshot codec rejection before
  schema-version inference.
- Adds codec, schema-store, and accepted snapshot tests proving non-positive
  persisted schema versions are rejected.
- Adds a source-boundary guard proving generated and persisted schema-version
  boundaries reject non-positive versions and do not infer missing versions from
  `SchemaVersion::initial()`.

## 0.177.6

### 📝 Summary

- Fixes the 0.177 admission/transition boundary so declared schema version is
  owned by the admission matrix, while schema transition diagnostics describe
  the shape that remains after a valid version/fingerprint gate.
- Keeps SQL DDL publication, query/cache behavior, and public schema semantics
  unchanged.

### 🔧 Changed

- Removes the raw snapshot-version mismatch from generic transition-shape
  diagnostics after admission.
- Preserves rollback/newer-snapshot rejection through the schema-version
  admission diagnostic with stored and candidate version facts.

### 🧪 Testing

- Adds a reconciliation guard proving a valid `N -> N+1` shape change reaches
  compatibility classification and still rejects unsupported field-contract
  transitions after the version gate.
- Updates newer-snapshot rejection coverage to assert the admission-matrix
  rollback diagnostic instead of the old transition fallback string.

## 0.177.5

### 📝 Summary

- Closes more of the 0.177 proof surface with source-boundary guards for
  explicit generated schema versions and method-qualified filterless cache-hit
  identity.
- Keeps runtime schema-version policy, query semantics, cache equivalence, and
  persisted schema format unchanged.

### 🧹 Cleanup

- Converts validated source-local lint suppressions from `allow` to `expect`
  where clippy confirms the lint is reliably present.
- Leaves macro-emitted, feature-dependent, and target-sensitive suppressions as
  `allow` to avoid downstream false positives.

### 🧪 Testing

- Adds a workspace source-boundary guard that every non-negative-test generated
  entity declaration carries explicit `schema_version`.
- Tightens the accepted-catalog source guard so eligible filterless warmed
  query-plan cache hits must use accepted catalog identity/header fingerprint
  method and fingerprint facts before accepted snapshot decode, authority
  construction, or `SchemaInfo` projection.

## 0.177.4

### 📝 Summary

- Aligns schema-node stable-memory allocation metadata with the method-qualified
  accepted identity shape used by core catalog diagnostics.
- Keeps allocation identity unchanged: memory ID and stable key still define
  durable allocation identity, while schema version, fingerprint-method
  version, and fingerprint remain diagnostics only.

### 🔧 Changed

- Adds fingerprint-method version metadata to schema-node stable-memory
  allocation metadata.
- Updates accepted-schema allocation metadata construction so callers must pass
  schema version, fingerprint-method version, and fingerprint together.
- Leaves absent allocation metadata fully absent, without inferring method
  versions for allocations that have no accepted schema/catalog authority.

### 🧪 Testing

- Extends allocation metadata tests to cover data, index, and schema
  role-specific fingerprint method versions.
- Extends stable-memory identity coverage so allocation identity ignores schema
  version, fingerprint-method version, and fingerprint metadata.

## 0.177.3

### 📝 Summary

- Threads accepted schema version through the same query/catalog context that
  already carries accepted fingerprint-method and fingerprint metadata.
- Keeps query and SQL execution identity consumption passive: the contexts
  expose accepted identity after admission without owning schema-version policy.

### 🔧 Changed

- Adds accepted schema-version access to the accepted query catalog context.
- Adds accepted schema-version access to compiled SQL execution context and a
  debug invariant that the context metadata matches the accepted snapshot it
  carries.
- Leaves shared query-plan cache identity, SQL compiled-command cache identity,
  admission policy, planner behavior, and SQL semantics unchanged.

### 🧪 Testing

- Extends accepted catalog identity coverage so the query catalog context
  reports the same version, fingerprint-method version, and fingerprint as the
  selected accepted catalog identity.
- Adds a source-boundary guard proving query cache, SQL cache, and compiled SQL
  context consume accepted identity metadata after admission instead of running
  schema-version admission policy.

## 0.177.2

### 📝 Summary

- Completes the method-qualified diagnostics follow-through for accepted schema
  identity by reporting fingerprint method versions beside schema versions and
  fingerprints.
- Keeps this as observability only: cache identity, admission policy, storage
  identity, query planning, and SQL semantics are unchanged.

### 🔧 Changed

- Adds `schema_fingerprint_method_version` to storage-report data, index, and
  schema store snapshot rows whenever accepted schema metadata is present.
- Carries role-specific fingerprint method versions through schema-store
  catalog/allocation metadata so data, index, and full catalog diagnostics each
  describe which fingerprint method produced their reported bytes.
- Leaves absent metadata rows as fully absent, with no inferred method version
  for empty, heap-only, or journaled-without-accepted-snapshot stores.

### 🧪 Testing

- Extends storage-report assertions so populated data, index, and schema rows
  expose their expected role-specific method versions, while absent rows expose
  none.
- Extends Candid shape guards for store snapshot diagnostics to include the new
  method-version field.
- Adds a source-boundary guard proving catalog diagnostics expose schema
  fingerprint bytes together with their method version.

## 0.177.1

### 📝 Summary

- Tightens the explicit schema-version admission line with clearer rejection
  facts, hot-path source-boundary guards, and storage-passive publication
  proof.
- Keeps query planning passive: normal session, query, and executor paths
  continue to consume accepted identity instead of computing candidate
  admission fingerprints.

### 🔧 Changed

- Extends schema-version admission rejection details to report stored and
  candidate schema versions, fingerprint-method versions, and fingerprints.
- Reports the expected next version on version-gap rejection while preserving
  the existing primary diagnostic classes for missing bumps, empty bumps, gaps,
  rollbacks, and fingerprint-method mismatches.
- Cleans up the reconciliation comment around the 0.177 gate so it reflects
  the now-active matrix enforcement before mutation compatibility.
- Keeps DDL-owned accepted catalog extensions on the existing catalog
  compatibility path instead of rejudging them through generated Rust
  `schema_version`; generated-source shape drift still reaches the 0.177
  version gate before compatibility publication.
- Preserves the first shape-difference detail on schema-version rejection
  diagnostics when reconciliation has one available.
- Adds a storage-publication guard that keeps persisted schema snapshots keyed
  by their declared version instead of a storage-assigned next version.
- Backfills missed demo, SQL-test, and SQL-perf fixture entities with explicit
  initial schema versions so downstream fixture crates compile under the new
  generated-entity macro contract.
- Updates nullable-transition recovery fixtures so the generated candidate is
  version `2` and old accepted prefixes remain version `1`.

### 🧪 Testing

- Adds admission-matrix assertions that rejection diagnostics include the
  compared identity facts.
- Adds a source-boundary guard proving candidate admission fingerprinting,
  identity comparison, and schema-version admission policy stay rooted in
  schema reconciliation rather than session/query/executor hot paths.
- Adds a source-boundary guard proving schema storage publication does not
  synthesize or auto-increment schema versions.
- Adds a source scan confirming every non-negative-test entity declaration in
  the workspace carries an explicit `schema_version`.
- Extends regression coverage across DDL publication, recovery/index rebuilds
  over old nullable rows, and generated-source schema drift metrics under the
  new version gate.

## 0.177.0

### 📝 Summary

- Starts the explicit schema-version contract line with an identity/header
  alignment, generated-declaration, admission-fingerprint input, and
  version/fingerprint matrix enforcement slices.
- Keeps public API, persisted format, schema admission policy, query semantics,
  SQL behavior, and writes unchanged.

### 🔧 Changed

- Makes shared query-plan cache identity compare accepted schema fingerprints
  together with their fingerprint-method version.
- Makes SQL compiled-command cache identity compare accepted schema
  fingerprints together with their fingerprint-method version.
- Makes per-compiled SELECT prepared-plan reuse compare accepted schema
  fingerprints together with their fingerprint-method version.
- Adds explicit `schema_version = N` parsing to generated entity declarations,
  rejecting omission, zero, and non-integer values at the macro boundary.
- Carries the declared generated schema version through `EntityModel`,
  schema-node metadata, and `CompiledSchemaProposal` initial snapshot
  construction without adding query-path candidate fingerprint work or
  admission-policy checks.
- Adds a schema-owned admission identity tuple over stored and candidate
  snapshots so reconciliation can compute version, fingerprint-method version,
  and accepted-shape fingerprint before mutation compatibility classification.
- Computes the admission shape fingerprint from a normalized accepted snapshot
  that removes declared `schema_version` while preserving accepted shape,
  field, index, relation, and layout contract facts.
- Keeps generated index display-name changes out of the admission shape
  fingerprint so existing metadata-only generated index rename behavior stays
  compatible with the version contract.
- Enforces the 0.177 admission matrix before mutation compatibility: unchanged
  version with changed shape, empty bumps, version gaps, rollbacks, and
  fingerprint-method mismatches now reject at the schema-version boundary;
  exactly `N + 1` with changed shape proceeds to existing compatibility checks.
- Extends test-only entity fixtures with explicit schema-version overrides so
  transition tests can distinguish stored accepted version `1` from generated
  candidate version `2`.

### ⚠️ Breaking

- Generated entities must now declare `schema_version = 1` or another positive
  integer in the `#[entity(...)]` attribute.

### 🧪 Testing

- Adds fail-closed cache-key guards proving raw schema fingerprint bytes do not
  alias across fingerprint-method versions.
- Extends source-boundary guards so cache identity must carry the accepted
  schema fingerprint method beside raw fingerprint bytes.
- Adds macro compile-fail coverage for omitted, zero, and non-integer
  `schema_version` declarations.
- Adds proposal coverage proving declared generated versions reach initial
  row-layout and accepted-snapshot construction.
- Adds admission-fingerprint coverage proving declared `schema_version` is
  adjacent identity, not an input to the accepted-shape fingerprint, and that
  accepted field/index shape changes still alter the fingerprint.
- Adds matrix coverage for same identity, single-version changed-shape
  admission, missing bump, empty bump, gap, rollback, and fingerprint-method
  mismatch cases.
- Updates runtime boundary tests so unsupported same-version drift now proves
  rejection at the schema-version gate before select planning, row decode, or
  write/delete staging.
