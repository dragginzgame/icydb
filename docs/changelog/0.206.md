# IcyDB 0.206 Changelog

## 0.206.1 - 2026-07-19

### 📝 Summary

- Bounds the retained working set for cursorless, unindexed,
  expression-ordered SQL pages to `offset + limit + lookahead` candidates.
- Preserves the required complete scan, SQL results, deterministic ordering,
  route selection, cursor behavior, cache identity, and failure semantics.

### 🔧 Changed

- Extends the existing scan-time bounded order collector from direct-field
  orders to every planner-resolved expression order that already qualifies for
  the materialized-slot page path.
- Caches each candidate's complete resolved ordering tuple during bounded
  selection, including direction, null behavior, every declared order term,
  and the planner-appended primary-key tie-breaker.
- Captures one `ResolvedOrder` in the bounded collector so strategy selection
  and every candidate comparison cannot consume different order contracts.
- Keeps final ordering and page projection in the canonical post-access phase.
  Route-ordered, continuation, `DISTINCT`, and non-materialized-slot queries
  retain their existing semantic paths.
- Deletes the expression-only full-retention eligibility restriction instead
  of adding a compatibility branch, feature toggle, route override, hidden
  index, or scenario-specific fast path.

### 🧪 Testing

- Focused executor coverage proves that expression keys are evaluated once per
  candidate during bounded selection and that the complete deterministic key
  retains the correct rows.
- Focused session coverage passes for ascending and descending scalar
  expressions, nullable text expressions, unindexed materialization,
  expression-index covering routes, pagination parity, and exact retained-row
  diagnostics.
- Package-local Clippy passes with warnings denied, the SQL performance matrix
  target compiles, and executor panic, layer-authority, read-admission,
  formatting, JSON, and diff invariants pass.
- Full repository and workspace suites remain user-owned push validation.

### 📊 Audit

- The implementation adds approximately 151 net production lines. Complexity
  increases locally because the maintained borrowed direct-field fast path and
  cached expression-key strategy have distinct storage contracts; no planner,
  route, public API, cursor, or persisted-format layer is added.
- The selected structural budget is a reduction from 2,048 peak retained
  candidates to at most 11 for the 2,048-row `LIMIT 10` sentinel. The complete
  scan and its 2,048 data-store gets remain contract-required.
- Exact post-change instruction and raw non-gzipped Wasm deltas require the
  workflow-built immutable 0.206.1 subject and are not inferred from local
  tests.

### 📚 Status

- The production slice is locally validated and ready for exact performance
  measurement after publication.
- 0.206 remains open until the structural target, instruction guardrails,
  maintained regression matrix, raw Wasm delta, and reviewed post-remediation
  baseline all pass.

## 0.206.0 - 2026-07-19

### 📝 Summary

- Opens the post-0.205 SQL performance-remediation line with a checked-in human
  investigation and machine-readable ledger covering 14 deterministically
  selected leading families.
- Adds the exact retained-candidate fact required to distinguish unavoidable
  full scans from avoidable executor working-set retention without changing
  production query semantics or dispatch.

### ➕ Added

- Records owner modules, typed evidence, dispositions, and selection state for
  every leading P1, P2, scale, normalized-cost, and cold-planning family.
- Adds `kernel_row_peak_retained_candidates` at the canonical kernel scan
  collector and transports it through diagnostics and performance artifacts.
- Hard-cuts diagnostics attribution artifacts to schema version 2 for the new
  exact field; no schema-1 compatibility decoder or dual artifact path exists.

### 🔧 Changed

- Closes the 0.205 grouped-performance evidence line against its exact
  three-run cohort before reranking the maintained post-0.205 profile.
- Freezes the 14-family leading set while requiring typed eliminated-work proof
  before any family can become the primary production selection.
- Keeps application-visible SQL results, admission, planning, route selection,
  persistence, and cursor behavior unchanged.

### 🧪 Testing

- Exact workflow run `29696076149` passes P1, scale, P2, attribution, merge,
  and bundle validation on the published 0.206.0 revision after one
  infrastructure-only PocketIC download retry.
- The run records 2,048 peak retained candidates, 104,312,717 total
  instructions, 35,243,583 order-window instructions, 2,048 data-store gets,
  and 10 returned rows for the selected expression-order sentinel.
- Full repository and workspace suites remain user-owned push validation.

### 📊 Audit

- The published release changes 19 files at approximately +607/-79 lines,
  including version bookkeeping. Production growth is limited to the exact
  diagnostics counter and transport; most growth is the durable investigation
  and evidence contract.
- The exact workflow-built SQL performance Wasm is 3,915,688 raw non-gzipped
  bytes with SHA-256
  `2fd45f0fe3fe2f50e31440c727ef2aa539d9d323f5f9db01f653fef426f3b8e1`,
  228 bytes above the accepted 0.205.2 opening subject.

### 📚 Status

- Exact evidence freezes `scale.user.unsupported_order.all.window10` as the
  sole primary `OptimizeNow` family, with no directly coupled selection.
- The predeclared structural target is at most 11 retained candidates for its
  2,048-row `LIMIT 10` scenario; production remediation begins in 0.206.1.
