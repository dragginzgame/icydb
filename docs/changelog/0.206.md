# IcyDB 0.206 Changelog

## 0.206.4 - 2026-07-20

### 📝 Summary

- Completes the 0.206 performance closeout against the exact hard-cut
  calibration cohort and reviewed post-remediation baseline.
- Removes executor sediment exposed by the bounded-order work so prepared
  access, retry policy, pagination, scalar/grouped routing, materialization,
  and diagnostics each have one canonical owner.

### 🔧 Changed

- Makes prepared-plan construction the sole schema-sensitive access-lowering
  authority. Aggregate terminals and cached SQL `COUNT` consume its encoded
  access facts instead of lowering the same logical plan again.
- Uses one kernel-owned residual retry controller for structural pages and SQL
  aggregate row sinks, including probe widening, unbounded fallback, output
  replacement, and metric accumulation.
- Gives cursorless scanning sole ownership of limit and offset application;
  post-scan processing now validates and shapes the selected payload without
  reconstructing window state.
- Represents initial and resumed scalar routes explicitly and applies every
  runtime route-hint adjustment at one final normalization boundary.
- Keeps grouped preparation and slot layout in one resident bundle, derives
  grouped mode and direction from the route, and has prepared load variants
  construct their final execution surfaces directly.
- Makes grouped diagnostics observe the same preparation, stream, fold, and
  finalization calls used by ordinary execution rather than running an
  attributed semantic copy.
- Retains the early SQL exact-cardinality shortcut deliberately: accepted
  three-run evidence shows a material cold/warm planning benefit. The
  specialized hash `COUNT` finalization boundary also remains because no
  measured evidence supports adding a shared abstraction there.

### 🩹 Fixed

- Wrong structural row representations now fail closed with typed executor
  invariants instead of appearing as successful empty results.
- Resolved stream metadata stays attached while DISTINCT decorates the key
  stream, and every grouped top-k order value stays paired with its comparison
  direction.
- Structural DELETE validates its selected-row cap once before packaging and
  commit preparation while retaining distinct outbound `RETURNING` row and
  byte validation.

### 🗑️ Removed

- Deletes invalid access-lowering flags and empty sentinels, the second
  residual-retry controller and fake page, cursorless post-scan window state,
  scalar route mode/optional-route combinations, load payload-state ladders,
  mixed-provenance grouped handoffs, copied route facts, and the obsolete
  orchestrator state module.
- Deletes the six-field resolved-stream tuple, parallel grouped top-k
  value/direction vectors, duplicate grouped diagnostic execution, and
  post-packaging structural DELETE bound checks.
- Adds no aliases, compatibility wrappers, fallback reconstruction, public API
  compatibility surface, feature mode, cursor format, or persisted-format
  bridge.

### 🧪 Testing

- Focused executor and session coverage passes for prepared access, aggregate
  fast/fallback parity, residual retry, structural materialization failures,
  limit/offset/cursor edges, scalar route normalization, grouped ordered/hash
  parity, SQL/fluent convergence, DISTINCT, grouped attribution, top-k
  ordering, metrics, and bounded DELETE atomicity.
- Strict all-target/all-feature Clippy, native and wasm no-default checks,
  all-feature wasm checks, formatting, diff hygiene, and repository dependency,
  executor-panic, layer-authority, mutation-atomicity, durability,
  read-admission, SQL-branch, generated-build, index-range, and memory-ID
  invariants pass.
- Full repository and workspace suites remain user-owned push validation.

### 📊 Audit

- The accumulated slice changes 71 tracked paths. Rust implementation is
  approximately +1,351/-1,739 lines (net -388), with core production code
  approximately net -673 lines; expanded design and audit evidence accounts
  for the overall documentation growth. The implementation shape is materially
  simpler despite explicit cfg-elided diagnostic observation.
- The final local SQL-performance subject is 3,907,311 raw non-gzipped bytes,
  13,726 bytes below the accepted 3,921,037-byte 0.206 subject, with SHA-256
  `a1cfbfc32c77c612565f8973cdc458683a1a6553bcf718e09c98bcf3ce91beb1`.
  Its secondary gzip size is 1,243,267 bytes.
- No public query semantics, cursor encoding, response shape, persisted format,
  admission limit, dependency version, package version, or generated API is
  intentionally changed.

### 📚 Status

- The executor audit addendum has no remaining 0.206 implementation work.
  Early SQL direct `COUNT` and specialized grouped `COUNT` remain documented,
  measured performance boundaries rather than unresolved cleanup.
- Comparable focused instruction deltas for the consolidated executor slices
  remain owned by the normal pushed performance workflow; this entry claims
  structural and raw-Wasm simplification, not an unmeasured instruction win.

## 0.206.2 - 2026-07-20

### 📝 Summary

- Reuses expression-order tuples evaluated during bounded scan selection when
  canonical post-access ordering produces the final row order.
- Adds an immutable previous-source workflow input so exact historical source
  can be measured under a reviewable current performance environment without
  weakening lockfile or artifact comparison.

### 🩹 Fixed

- Removes the second expression-key evaluation that remained after 0.206.1 and
  made the 16-row guard more expensive despite the bounded retained set.
- Keeps each retained row inseparably paired with its complete resolved order
  tuple until canonical final ordering consumes it.
- Validates the exact resolved-order and bounded keep-count contract at the
  consumption boundary and fails closed with the typed query-executor
  invariant when they differ.

### 🔧 Changed

- Uses one opaque pending-row container for ordinary rows and cached
  expression-order rows. There is no optional cache DTO, parallel row/key
  storage, empty-vector sentinel, second scan-result model, or new execution
  route.
- Keeps the planner as the sole `ResolvedOrder` owner, the kernel window as the
  sole keep-count owner, and canonical post-access ordering as the sole final
  sort owner.
- Allows manual SQL-performance runs to select one exact 40-character source
  revision. Every code-bearing job checks out that revision, and reused
  calibration ordinals verify both source revision and raw Wasm identity.

### 🧪 Testing

- Focused executor tests prove one expression evaluation per candidate,
  complete deterministic ordering, and typed rejection of a mismatched cached
  order contract.
- Focused session tests preserve direct-field, full-entity, and scalar-
  expression unindexed ordering behavior.
- Package-local strict Clippy, formatting, diff validation, machine-readable
  evidence validation, and the locked shared-Wasm build pass. Full repository
  and workspace suites remain user-owned push validation.

### 📊 Audit

- A focused complexity pass reduced the correction from approximately 211 to
  123 net production lines. Runtime complexity remains above 0.206.1 but is
  localized to one invariant-bearing transport type; no public API, planner
  mode, route, feature flag, cursor, or persisted format was added.
- The locked local SQL-performance subject is 3,919,866 raw non-gzipped bytes
  with SHA-256
  `51e3323de6ac2ee4b14098732f43181667b9ce78148ed048e3bb79e91f818d3d`,
  2,922 bytes above tagged 0.206.1 and 1,571 bytes below the earlier correction
  candidate.
- Diagnostic local measurements retain the same result IDs and record total
  instruction deltas of `-0.45%`, `-13.69%`, and `-15.18%` at 16, 256, and
  2,048 rows respectively. They are not final comparable workflow evidence.

### 📚 Status

- The corrected and simplified production slice is locally validated.
- The exact immutable-source bridge and corrected candidate comparison remain
  required before 0.206 performance remediation can be closed.

## 0.206.1 - 2026-07-19

### 📝 Summary

- Bounds the retained working set for cursorless, unindexed,
  expression-ordered SQL pages to `offset + limit + lookahead` candidates.
- Preserves the required complete scan, SQL results, deterministic ordering,
  route selection, cursor behavior, cache identity, and failure semantics.

### 🔧 Changed

- Extends the existing scan-time bounded order collector from direct-field
  orders to every planner-resolved expression order that already qualifies for
  the materialized-slot page path.
- Caches each candidate's complete resolved ordering tuple during bounded
  selection, including direction, null behavior, every declared order term,
  and the planner-appended primary-key tie-breaker.
- Captures one `ResolvedOrder` in the bounded collector so strategy selection
  and every candidate comparison cannot consume different order contracts.
- Keeps final ordering and page projection in the canonical post-access phase.
  Route-ordered, continuation, `DISTINCT`, and non-materialized-slot queries
  retain their existing semantic paths.
- Deletes the expression-only full-retention eligibility restriction instead
  of adding a compatibility branch, feature toggle, route override, hidden
  index, or scenario-specific fast path.

### 🧪 Testing

- Focused executor coverage proves that expression keys are evaluated once per
  candidate during bounded selection and that the complete deterministic key
  retains the correct rows.
- Focused session coverage passes for ascending and descending scalar
  expressions, nullable text expressions, unindexed materialization,
  expression-index covering routes, pagination parity, and exact retained-row
  diagnostics.
- Package-local Clippy passes with warnings denied, the SQL performance matrix
  target compiles, and executor panic, layer-authority, read-admission,
  formatting, JSON, and diff invariants pass.
- Full repository and workspace suites remain user-owned push validation.

### 📊 Audit

- The implementation adds approximately 151 net production lines. Complexity
  increases locally because the maintained borrowed direct-field fast path and
  cached expression-key strategy have distinct storage contracts; no planner,
  route, public API, cursor, or persisted-format layer is added.
- The selected structural budget is a reduction from 2,048 peak retained
  candidates to at most 11 for the 2,048-row `LIMIT 10` sentinel. The complete
  scan and its 2,048 data-store gets remain contract-required.
- Exact post-change instruction and raw non-gzipped Wasm deltas require the
  workflow-built immutable 0.206.1 subject and are not inferred from local
  tests.

### 📚 Status

- The production slice is locally validated and ready for exact performance
  measurement after publication.
- 0.206 remains open until the structural target, instruction guardrails,
  maintained regression matrix, raw Wasm delta, and reviewed post-remediation
  baseline all pass.
