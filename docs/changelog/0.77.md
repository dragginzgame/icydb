# 0.77 — SQL Contract Freeze

## 0.77.2 - 2026-04-13 - Flatten The Remaining Query Surface

This follow-up finishes the remaining public query-surface flattening after the
SQL cleanup landed.

The live query story is now clearer:

- fluent grouped queries execute through the same public `execute()` entrypoint
  as scalar fluent queries
- typed `DbSession::execute_query(&Query<E>)` no longer branches on grouping in
  the public facade
- the last dead grouped text-cursor bridge is gone instead of being kept as
  hidden compatibility residue

What changed:

* introduced one unified fluent load result in core:
  * `crates/icydb-core/src/db/query/fluent/load/result.rs`
  * `LoadQueryResult<E>` now carries either scalar rows or grouped rows
* introduced the matching public facade result:
  * `crates/icydb/src/db/response/query.rs`
  * `QueryResponse<E>` now carries either scalar rows or grouped rows
* changed fluent load execution so grouped and scalar queries now share the
  same public terminal:
  * `crates/icydb-core/src/db/query/fluent/load/terminals.rs`
  * `crates/icydb/src/db/session/macros.rs`
  * public fluent `execute()` now materializes one unified query result instead
    of sending grouped shapes through a separate grouped terminal
* removed the old public grouped fluent terminal:
  * `crates/icydb/src/db/session/load.rs`
  * public grouped execution no longer needs its own `execute_grouped(...)`
    facade story
* updated fluent grouped policy wording so grouped queries are described as
  direct `execute()` queries instead of another special lane:
  * `crates/icydb-core/src/db/query/plan/validate/fluent_policy.rs`
  * `crates/icydb-core/src/db/query/intent/errors/mod.rs`
  * `crates/icydb-core/src/db/query/plan/validate/errors.rs`
* moved the last public typed query grouping branch under one hidden core
  boundary:
  * `crates/icydb-core/src/db/session/query.rs`
  * `crates/icydb/src/db/session/mod.rs`
  * `DbSession::execute_query(&Query<E>)` in `icydb` now forwards through one
    core `execute_query_result(...)` owner instead of branching on
    `query.has_grouping()` in the facade
* deleted the dead grouped text-cursor bridge that no longer had any callers:
  * removed `execute_grouped_text_cursor(...)` from
    `crates/icydb-core/src/db/session/query.rs`
  * removed `GroupedTextCursorPageWithTrace` from
    `crates/icydb-core/src/db/response/grouped.rs`
* narrowed the remaining grouped runtime helper to internal visibility:
  * `crates/icydb-core/src/db/session/query.rs`
  * `execute_grouped(...)` is still a real runtime helper, but it is no longer
    exposed wider than its actual ownership boundary
* updated grouped/scalar test call sites onto the new unified fluent result and
  removed the dead facade helper that the old split had left behind:
  * `crates/icydb-core/src/db/session/tests/...`
  * `crates/icydb-core/src/db/executor/tests/...`
  * `crates/icydb/src/db/session/mod.rs`

Result:

* the public read/query surface is flatter
* fluent grouped execution no longer needs a separate public terminal
* typed `execute_query(...)` no longer leaks grouped branching in the public
  facade
* the remaining grouped/scalar split now lives only in core runtime execution
  families, where it is an implementation detail instead of a public API shape

Verification:

* `cargo fmt --all`
* `cargo clippy -p icydb-core --all-targets -- -D warnings`
* `cargo clippy -p icydb --all-targets -- -D warnings`
* `cargo clippy --workspace --all-targets -- -D warnings`
* `cargo test -p icydb-core --lib --no-run`
* `cargo test -p icydb --lib --no-run`

## 0.77.1 - 2026-04-13 - Delete The Last Lane-Shaped SQL Runtime Wrappers

This follow-up finishes the runtime cleanup that `0.77.0` started.

The live SQL surface is now only:

- `execute_sql_query::<E>(...)`
- `execute_sql_update::<E>(...)`

The older lane-shaped wrapper story no longer exists on the runtime path:

- no live `query_from_sql(...)` session wrapper
- no live `execute_sql(...)` scalar wrapper
- no live `execute_sql_grouped(...)` grouped wrapper
- no runtime aggregate-lane wrapper kept alive just for legacy tests

What changed:

* deleted the remaining lane-shaped SQL test wrappers from the live session SQL
  runtime in:
  * `crates/icydb-core/src/db/session/sql/mod.rs`
* moved the legacy SQL lowering/scalar/grouped compatibility glue into a
  test-only extension trait in:
  * `crates/icydb-core/src/db/session/tests/mod.rs`
  so core tests can still assert old boundary behavior without keeping those
  wrappers on `DbSession` itself
* deleted the now-dead aggregate and grouping surface scaffolding that only
  existed to support the removed wrappers:
  * `crates/icydb-core/src/db/session/sql/aggregate.rs`
  * `crates/icydb-core/src/db/session/sql/execute/mod.rs`
  * `crates/icydb-core/src/db/session/sql/surface/lane.rs`
* cleaned the remaining aggregate lowering/import drift in:
  * `crates/icydb-core/src/db/sql/lowering/aggregate.rs`
* hard-cut the remaining dead SQL cleanup residue instead of leaving
  `allow(dead_code)` escape hatches in the DB codepath:
  * removed the unused scalar projection `CountNonNull` path
  * removed the unused extrema-value helper
  * removed the compile-only planner test scaffolding that no longer proved
    any live runtime contract
* kept the current product boundary unchanged:
  * public SQL still uses `execute_sql_query::<E>(...)` for read/explain/
    introspection SQL
  * public SQL still uses `execute_sql_update::<E>(...)` for mutation SQL
  * canister `query(sql)` and `update(sql)` endpoints stay aligned with that
    same split

Result:

* the runtime SQL story is flatter
* the old wrapper names now exist only in the test harness vocabulary
* grep no longer finds those wrapper owners under the live SQL runtime modules
* the DB codepath no longer carries dead-code suppressions for the removed SQL
  wrapper and aggregate residue

Verification:

* `cargo clippy -p icydb-core --all-targets -- -D warnings`
* `cargo test -p icydb-core --no-run`
* `cargo fmt --all`

## 0.77.0 - 2026-04-12 - Remove SQL Dispatch And Flatten The Remaining SQL Surface

This patch changes the public SQL story materially:

- removes the public typed `execute_sql_dispatch::<E>(...)` facade
- removes generated canister `sql_dispatch::query(...)`
- removes the generated canister `sql_dispatch` module from build output
- keeps typed/fluent APIs as the main Rust-side product surface
- replaces the older mixed SQL entrypoint story with one public SQL query
  executor plus one matching SQL mutation executor, instead of reviving
  route-sharded or dispatch-shaped SQL surfaces

What changed:

* removed the public typed dispatch envelope and helpers from:
  * `crates/icydb/src/db/session/mod.rs`
  * `crates/icydb/src/db/sql/mod.rs`
* removed the generated canister SQL query shim from:
  * `crates/icydb/src/db/session/generated.rs`
  * `crates/icydb-build/src/db.rs`
* removed stale facade re-exports and perf-attribution exports that only made
  sense while public SQL routing still existed
* narrowed the remaining core SQL execution helpers to internal-only visibility
  so `icydb-core` no longer exposes them as part of the public crate surface
* rewrote the current SQL docs so they match the new product direction:
  * typed/fluent APIs remain the primary Rust-side surface
  * `execute_sql_query::<E>(...)` owns read, explain, and introspection SQL
  * `execute_sql_update::<E>(...)` owns mutation SQL
  * structural SQL lowering stays internal/test-owned instead of remaining a
    parallel public API
* updated SQL lane rejection messages to stop pointing users toward removed
  public SQL routers and to point them at surviving typed/fluent helpers instead
* introduced the final single-entity public SQL split in:
  * `crates/icydb-core/src/db/session/sql/mod.rs`
  * `crates/icydb/src/db/session/mod.rs`
  so tiny canister endpoints can stay thin without rebuilding SQL routing in
  local `lib.rs` files
* reduced the demo RPG canister SQL endpoint back to a simple Character-only
  forwarder in:
  * `canisters/demo/rpg/src/lib.rs`
  instead of keeping parser and route-selection glue in the canister itself
* moved `SqlStatementResult -> SqlQueryResult` packaging out of the session
  facade and back into the SQL result module in:
  * `crates/icydb/src/db/sql/mod.rs`
  so `icydb-core` owns execution, `icydb::db::sql` owns result packaging, and
  the facade only forwards
* deleted the remaining dead internal generated-query routing chain in core:
  * generated statement-attempt routing helpers
  * generated authority-table routing helpers
  * generated text-row projection helpers
  * generated canister delete-projection helpers
  * the raw string `execute_sql_statement(...)` wrapper now only exists for
    core tests
  which removed the last internal scaffolding that only existed to support the
  old public/generated SQL routers
* softened planner/explain contract tests so they keep proving deterministic
  semantic choice without freezing exact implementation narration:
  * `crates/icydb-core/src/db/session/tests/verbose_route_choice.rs`
  * `crates/icydb-core/src/db/query/intent/tests/verbose.rs`
  * `crates/icydb-core/src/db/query/plan/planner/tests/mod.rs`
  * `crates/icydb-core/src/db/query/explain/tests/descriptor.rs`
  * `crates/icydb-core/src/db/query/explain/tests/aggregate.rs`
* replaced exact chosen-index names, rejected-candidate inventories, render
  snapshots, and broad execution-descriptor bookkeeping assertions with the
  narrower contract that actually matters:
  * chosen access family such as `IndexPrefix(...)` or `IndexRange(...)`
  * stable `order_compatible_preferred` reasoning where order compatibility is
    the semantic tie-break
  * root route family
  * covering versus materialized behavior
  * fallback reason where it changes grouped semantics
  * ordered-route versus materialized-sort behavior
  * `TopNSeek`, `OrderByAccessSatisfied`, `OrderByMaterializedSort`, and
    grouped aggregate node-family presence
  * `existing_row_mode`, `fetch`, and logical `offset` where those fields carry
    real semantic meaning
* moved one reduced-SQL authority-label helper from canonical JSON snapshots to
  typed execution descriptors:
  * `crates/icydb-core/src/db/session/tests/authority_labels.rs`
  * `crates/icydb-core/src/db/session/tests/mod.rs`
  so the reduced-SQL descriptor tests now prove covering/materialized route
  semantics directly and use rendered JSON only for the actual outward
  “removed authority labels stay gone” surface check
* deleted the stale facade SQL session tests that only existed to exercise the
  removed public SQL router:
  * `crates/icydb/src/db/session/tests.rs`
  * `canisters/demo/rpg/src/perf.rs`
* deleted the broad SQL parity canister and its fixture crate:
  * `canisters/test/sql_parity`
  * `schema/test/sql_parity`
* deleted the broad PocketIC SQL parity target that only existed to exercise
  the removed routed-SQL surface:
  * `testing/pocket-ic/tests/sql_canister.rs`
* deleted the parity-only perf audit normalization helper:
  * `scripts/audit/normalize_perf_audit.sh`
* trimmed the internal SQL execution cleanup further in:
  * `crates/icydb-core/src/db/session/sql/execute/mod.rs`
  * `crates/icydb-core/src/db/session/sql/execute/lowered.rs`
  * `crates/icydb-core/src/db/session/sql/projection/mod.rs`
  * `crates/icydb-core/src/db/session/sql/projection/runtime.rs`
  * `crates/icydb-core/src/db/executor/delete/mod.rs`
  * `crates/icydb-core/src/db/sql/lowering/mod.rs`
  so `icydb-core` no longer carries the dead generated-query path, rendered
  text-row SQL projection branch, or the canister-only delete-projection
  commit bridge that had become unreachable after the public SQL cut
