## 0.153.10

`0.153.10` tightens the field-path runner publication boundary and adds the
0.154 developer-stability design. Accepted snapshot publication handoff now
remains distinct from physical index-store publication in generic runner
diagnostics, so `physical_work_allows_publication()` cannot become true until
the validated `IndexStore` is promoted to ready.

### Publication Readiness

- Added an explicit `PublishPhysicalStore` runner phase for the final
  `IndexStore` ready transition.
- Snapshot publication reports now keep physical store visibility staged, even
  though the accepted snapshot handoff sink has accepted the after snapshot.
- Published-store reports advance runner diagnostics through the physical-store
  publication phase and are now the source for top-level field-path runner
  publication readiness.
- Added a physical-store-not-published readiness blocker so diagnostics expose
  the last boundary before rebuilt index state is runtime-ready.
- Added source-boundary guard coverage locking generic runner publication
  readiness to the physical-store publication phase.
- Added the `0.154` developer-stability design covering reconciliation
  integration, accepted row scanning, real store binding, runtime invalidation,
  recovery, and ship criteria for developer testing without broad SQL DDL.

- Added a runtime invalidation sink contract that records accepted before/after
  schema epochs for one validated staged field-path index store.
- Added a staged field-path runtime invalidation plan that binds isolated
  `IndexStore` validation to `SchemaMutationPublicationIdentity` without
  publishing the accepted-after epoch.
- Added a typed runtime invalidation report that marks the runner diagnostics

  invalidation.
- Publication reports now advance runner diagnostics through `PublishSnapshot`,

  publishable runner state while leaving the real schema-store write behind the
  mockable sink.
- Split the growing schema mutation implementation into a `mutation/` module
  tree, separating runner contracts, field-path staged physical work, and tests

- Added focused coverage proving the invalidation sink receives the accepted

## 0.153.5

`0.153.5` adds staged field-path index-store writer, rollback, isolated
`IndexStore` writer, isolated physical/in-memory validation, and
publication-readiness contracts. Validated staged buffers can now stream
accepted raw write intents, capture prior raw entries for rollback, derive
reverse-order restore/remove rollback plans, exercise staged writes and rollback
in isolated physical and in-memory stores, validate staged store state against
the staged batch, report remaining publication blockers, and report rollback
intent through mockable sinks without mutating runtime-visible `IndexStore`
state, invalidating runtime caches, or publishing snapshots.

### Staged Writer And Rollback

- Added a staged field-path index-store writer sink contract that receives the
  accepted store path plus raw index key and entry pairs from validated staged
  buffers.
- Added a typed staged write report carrying accepted store identity, intended
  entry count, staged-only visibility, and the existing runner diagnostics.
- Added a staged-store read-view contract plus rollback-aware staged write
  batches that capture previous raw index entries before staged writes are
  accepted.
- Added typed rollback snapshots and reverse-order rollback plans that derive
  restore actions for overwritten entries and remove actions for newly inserted
  keys.
- Added a mockable staged rollback writer and typed rollback report for
  restore/remove action counts.
- Added an isolated `IndexStore` writer adapter that applies staged write and
  rollback contracts to a caller-provided building store while preserving
  staged-only visibility and avoiding runtime-visible publication.
- Added fail-closed isolated `IndexStore` validation that checks accepted store
  identity, staged-only visibility, building state, entry counts, raw key
  presence, and raw entry payloads before physical staged state can be treated
  as a matched staged batch.
- Added an isolated in-memory staged-store overlay that implements the staged
  read, write, and rollback contracts without exposing or mutating
  runtime-visible `IndexStore`.
- Added fail-closed overlay validation that checks accepted store identity,
  staged-only visibility, entry counts, raw key presence, and raw entry payloads
  before overlay state can be treated as a matched staged batch.
- Added publication-readiness diagnostics for validated overlays that keep
  store visibility, physical validation, runtime invalidation, and snapshot
  publication barriers explicit before any publishable handoff exists.
- Kept writer and rollback contracts isolated from physical `IndexStore`; this
  patch reports staged intent only and does not publish, invalidate, or widen
  runtime visibility.
- Added focused coverage proving staged write intents, rollback snapshots,
  reverse rollback ordering, restore/remove action derivation, rollback reports,
  isolated `IndexStore` write/rollback behavior, fail-closed physical and
  overlay validation, publication-readiness blockers, and non-publication
  behavior.

## 0.153.0

`0.153.0` starts the schema rebuild runner implementation by making runner
outcomes and diagnostics explicit without changing runtime publication behavior.
The new contracts sit on top of the `0.152` mutation execution preflight
boundary and keep physical rebuild execution deferred.

### Runner Contracts

- Added internal runner phase names for preflight, staged stores, physical
  build, validation, runtime invalidation, snapshot publication, and diagnostics.
- Added staged-store visibility contracts so future rebuild output can be named
  as staged-only before publication.
- Added structured runner rejection kinds and rejection reports for missing
  capabilities and unsupported requirements, with room for validation, rollback,
  and publication failures.
- Added positive runner reports for preflight outcomes, including step count,
  required capabilities, completed phases, and store visibility.
- Added a runner outcome API that classifies plans as no physical work, ready
  for staged physical work, or rejected while keeping `PhysicalWorkReady`
  separate from publication.
- Added a checked runner input contract that binds accepted-before snapshots,
  accepted-after snapshots, and already-derived execution plans together while
  rejecting cross-entity or primary-key-mismatched snapshot pairs.
- Added a no-op runner adapter that consumes checked runner inputs, reports
  metadata-only inputs as no physical work, and rejects physical-work inputs
  through the structured missing-capability outcome until real staged execution
  exists.
- Added focused mutation tests proving metadata-only plans report no physical
  work, supported index rebuild plans report staged physical readiness, and
  missing capabilities or unsupported rewrite requirements reject at preflight.
- Added focused mutation tests proving runner input construction accepts
  same-entity accepted snapshot pairs and fails closed for entity path, entity
  name, and primary-key identity mismatches.
- Added focused mutation tests proving the no-op runner accepts metadata-only
  inputs and fails closed for non-unique field-path index rebuild inputs.
