# 0.153.x

## 0.153.0

`0.153.0` starts the schema rebuild runner implementation by making runner
outcomes and diagnostics explicit without changing runtime publication behavior.
The new contracts sit on top of the `0.152` mutation execution preflight
boundary and keep physical rebuild execution deferred.

### Runner Contracts

- Added internal runner phase names for preflight, staged stores, physical
  build, validation, runtime invalidation, snapshot publication, and diagnostics.
- Added staged-store visibility contracts so future rebuild output can be named
  as staged-only before publication.
- Added structured runner rejection kinds and rejection reports for missing
  capabilities and unsupported requirements, with room for validation, rollback,
  and publication failures.
- Added positive runner reports for preflight outcomes, including step count,
  required capabilities, completed phases, and store visibility.
- Added a runner outcome API that classifies plans as no physical work, ready
  for staged physical work, or rejected while keeping `PhysicalWorkReady`
  separate from publication.
- Added a checked runner input contract that binds accepted-before snapshots,
  accepted-after snapshots, and already-derived execution plans together while
  rejecting cross-entity or primary-key-mismatched snapshot pairs.
- Added a no-op runner adapter that consumes checked runner inputs, reports
  metadata-only inputs as no physical work, and rejects physical-work inputs
  through the structured missing-capability outcome until real staged execution
  exists.
- Added focused mutation tests proving metadata-only plans report no physical
  work, supported index rebuild plans report staged physical readiness, and
  missing capabilities or unsupported rewrite requirements reject at preflight.
- Added focused mutation tests proving runner input construction accepts
  same-entity accepted snapshot pairs and fails closed for entity path, entity
  name, and primary-key identity mismatches.
- Added focused mutation tests proving the no-op runner accepts metadata-only
  inputs and fails closed for non-unique field-path index rebuild inputs.
