# 0.64 Changelog

## 0.64.0 - 2026-03-25 - Public Structural Mutation API

### Summary

`0.64.0` turns the earlier structural field-level serialization work into one
public write surface instead of leaving it as an internal executor experiment.

The new public API is mode-driven rather than helper-fragmented: callers build
an `UpdatePatch`, choose `StructuralMutationMode`, and submit that through
`DbSession::mutate_structural(...)`. The important contract is that this is not
one weaker bypass around typed writes. The structural path still routes through
the shared structural mutation input, rebuilds the persisted row through the
same slot/row boundary, decodes back into `E`, and reuses the existing typed
save preflight before commit staging.

This release also fixes one repeated local tooling problem: normal IcyDB
build/test/check flows now use repo-local cargo state, which keeps sibling
repositories on the same filesystem from blocking each other on the shared
cargo home / target locks.

- expose one complete structural mutation surface instead of freezing
  `update` first and retrofitting `insert` / `replace` later
- keep typed and structural mutation semantics aligned through the same
  validation and commit pipeline
- isolate repo-managed cargo flows into IcyDB-local cargo state so routine
  local builds and hooks stop contending with other repositories

### Public API Note

The intended public entrypoint is the mode-driven surface, not three unrelated
helper APIs:

```rust
let patch = UpdatePatch::new()
    .set_field(User::MODEL, "id", Value::Ulid(user_id))?
    .set_field(User::MODEL, "email", Value::String("alice@example.com".into()))?;

let saved = session.mutate_structural::<User>(
    user_id,
    patch,
    StructuralMutationMode::Update,
)?;
```

This keeps the user model orthogonal:

- one patch representation
- one mutation entrypoint
- explicit write intent via `StructuralMutationMode`
- the same validation and commit semantics as the typed path underneath

### Notes

- `0.64.0` is intentionally one complete public structural mutation surface,
  not an update-only preview that would have forced later insert/replace
  semantics to work around a prematurely frozen contract
- the structural mutation path is a dynamic ingress, not a second weaker write
  engine: it still normalizes back through typed validation before commit
- one exact targeted cargo test re-run under the new repo-local cargo env could
  not complete in the sandbox because `index.crates.io` DNS resolution was
  unavailable; the kept release note only relies on the successful local fmt,
  clippy, existing persisted-row test, and non-network script validation above
