# 0.64 Changelog

## 0.64.4 - 2026-03-26 - Index Entry and Structural Ordering Cleanup

### Summary

`0.64.4` keeps the public SQL and mutation surface unchanged, but trims more of
the retained mutation/runtime fixed cost by tightening structural ordering,
index-entry decode/encode ownership, unique-index validation cleanup, and
commit-preparation row validation.

This patch stayed on the same `0.64` rule as the later cleanup slices: keep
owner-local deduplications that make the code easier to follow, keep direct
tests where behavior moved, and stop before larger abstraction pushes turn into
noise or regressions. The retained result is a smaller index/mutation path
without introducing new feature surface or codegen tricks.

- keep the SQL and mutation feature surface unchanged while shrinking more of
  the retained order and index-entry machinery
- prefer canonical owner-local decode, encode, and comparison paths over
  duplicated loops or ad hoc local string rebuilding
- add or keep direct tests where ordering and index-entry behavior moved, and
  remove dead unique-validation branches that no longer represent live runtime
  states
- reject malformed commit-marker row payloads during commit preparation,
  including non-indexed scalar fields that previously could survive until a
  later query decode touched them

### Changed

- shared more of the structural ordering machinery in:
  - `crates/icydb-core/src/db/executor/order.rs`
  - row-vs-row and row-vs-boundary ordering now use one canonical resolved-slot
    walk instead of carrying separate comparison loops
  - direct order tests continue to pin:
    - full structural ordering
    - bounded top-k ordering
    - descending boundary comparison
- simplified unique-index validation ownership in:
  - `crates/icydb-core/src/db/index/plan/mod.rs`
  - `crates/icydb-core/src/db/index/plan/unique.rs`
  - planner unique validation now reuses precomputed `index_fields` text
    instead of rebuilding field-name lists again at the failure site
- tightened index-entry decode and encode ownership in:
  - `crates/icydb-core/src/db/index/entry.rs`
  - `RawIndexEntry` now has one checked decode path instead of separate
    validate/decode loops over the same payload
  - `RawIndexEntry::try_from_entry(...)` now encodes directly from canonical
    `IndexEntry` membership instead of round-tripping through a temporary
    vector and the more generic key-builder path
- removed dead unique-validation mismatch branches and now-unused constructors
  in:
  - `crates/icydb-core/src/db/index/plan/unique.rs`
  - `crates/icydb-core/src/error/mod.rs`
  - `crates/icydb-core/src/db/index/entry.rs`
  - the old row-key mismatch leg no longer exists in the live unique-validation
    flow, so the extra branch and its error constructors are gone
- hardened commit-marker row validation in:
  - `crates/icydb-core/src/db/commit/prepare.rs`
  - commit preparation now fully decodes every declared slot in marker
    `before`/`after` rows before planning or apply can continue
  - malformed non-indexed fields can no longer bypass commit preparation and
    fail only later when a query endpoint decodes them
- cleaned up runtime-like tests that were still fabricating row bytes through
  direct entity serialization in:
  - `crates/icydb-core/src/db/executor/tests/mutation_save.rs`
  - `crates/icydb-core/src/db/executor/tests/semantics.rs`
  - those tests now build row bytes through the same `RawRow` boundary the live
    runtime uses
- added a regression for malformed non-indexed scalar payload rejection in:
  - `crates/icydb-core/src/db/commit/tests.rs`
  - `crates/icydb-core/src/testing/entity_tags.rs`
  - the new test proves a malformed `name` scalar is rejected during commit
    prepare instead of being accepted into storage and failing later at query
    time

### Wasm Impact

Relative to the prior `0.64.3` baseline:

  - built `.wasm`: `1256424 -> 1255388` (`1,036` bytes)
  - shrunk `.wasm`: `1171353 -> 1170409` (`944` bytes)

  - shrunk `.wasm`: `1253182 -> 1250792` (`2,390` bytes)

This is a smaller patch than `0.64.2` or `0.64.3`, but it still reflects real
retained owner cleanup. The visible win came mostly from pulling duplicated
index-entry decode/encode work onto the type that already owns that contract,
then removing the dead unique-validation mismatch leg that was still keeping
extra machinery live.

- after `0.64.4`, the next likely wasm targets are still in the retained
  mutation/index path, especially around index-key building or the remaining
  commit-preparation ownership seams

query and commit machinery by removing duplicated access canonicalization,
index-planning, and commit-preparation paths.

The important result is that this patch is still reducing retained owner cost,
not removing features. The measured wins came from collapsing repeated
canonicalization and structural reader forwarding onto the types that already
own those contracts, then stopping before the next lower-signal probes turned
into noise or regressions.

- keep the SQL and mutation feature surface unchanged while shrinking more of

- prefer owner-local comparison, range-read, and adapter forwarding helpers

## 0.64.0 - 2026-03-25 - Public Structural Mutation API

### Summary

`0.64.0` turns the earlier structural field-level serialization work into one
public write surface instead of leaving it as an internal executor experiment.

The new public API is mode-driven rather than helper-fragmented: callers build
an `UpdatePatch`, choose `StructuralMutationMode`, and submit that through
`DbSession::mutate_structural(...)`. The important contract is that this is not
one weaker bypass around typed writes. The structural path still routes through
the shared structural mutation input, rebuilds the persisted row through the
same slot/row boundary, decodes back into `E`, and reuses the existing typed
save preflight before commit staging.

This release also fixes one repeated local tooling problem: normal IcyDB
build/test/check flows now use repo-local cargo state, which keeps sibling
repositories on the same filesystem from blocking each other on the shared
cargo home / target locks.

- expose one complete structural mutation surface instead of freezing
  `update` first and retrofitting `insert` / `replace` later
- keep typed and structural mutation semantics aligned through the same
  validation and commit pipeline
- isolate repo-managed cargo flows into IcyDB-local cargo state so routine
  local builds and hooks stop contending with other repositories

### Changed

- exposed the public façade structural mutation contract in `icydb`:
  - `crates/icydb/src/db/session/mod.rs`
  - `crates/icydb/src/db/mod.rs`
  - new public surface:
    - `DbSession::mutate_structural(key, patch, mode)`
    - `UpdatePatch`
    - `StructuralMutationMode`
  - mode semantics are explicit:
    - `Insert`: patch must describe a full after-image and fails if the row already exists
    - `Update`: patch applies over the existing row and fails if the row is missing
    - `Replace`: patch must describe a full after-image, rebuilds from an empty row image, and inserts if the row is missing
- completed the structural write pipeline under one shared mutation core:
  - `crates/icydb-core/src/db/data/persisted_row.rs`
  - `crates/icydb-core/src/db/data/row.rs`
  - `crates/icydb-core/src/db/executor/mutation/mod.rs`
  - `crates/icydb-core/src/db/executor/mutation/save.rs`
  - `crates/icydb-core/src/db/session/write.rs`
  - field-level decode, field-level encode, serialized slot-patch lowering,
    raw-row patch replay, and mutation staging now all run through one
    canonical structural slot/row boundary instead of competing direct row
    encode paths
- locked structural-vs-typed mutation parity before the façade exposure:
  - `crates/icydb-core/src/db/executor/tests/mutation_save.rs`
  - insert / update / replace parity is now pinned on equivalent rows
  - structural unique-index conflicts and key-mismatch handling stay aligned
    with the existing typed path
  - unknown field names are rejected at the patch construction boundary
- tightened the public contract wording around the new mutation surface:
  - `crates/icydb/src/db/session/mod.rs`
  - `docs/design/0.64-per-field-serialization/0.64-design.md`
  - `docs/design/0.64-per-field-serialization/0.64-status.md`
  - the docs now reflect the actual landed API shape instead of the earlier
    proposal-state design
- normalized the most obvious structural-mutation error wording ahead of
  release:
  - `crates/icydb-core/src/error/mod.rs`
  - `crates/icydb-core/src/db/executor/tests/mutation_save.rs`
  - the public-facing messages now say:
    - `unknown field for structural mutation`
    - `structural mutation produced an invalid entity`
  - instead of exposing more implementation-shaped phrases like
    `field not found` or `after-image invalid`
- isolated repo-managed cargo workflows into IcyDB-local cargo state:
  - `Makefile`
  - `.githooks/pre-commit`
  - `.githooks/pre-push`
  - `scripts/env/cargo-local.sh`
  - `scripts/app/build.sh`
  - `scripts/ci/bump-version.sh`
  - `scripts/ci/security-check.sh`
  - `scripts/ci/validate-workflow-packages.sh`

  - normal repo flows now use:
    - `CARGO_HOME=$ROOT/.cache/cargo/icydb`
    - `CARGO_TARGET_DIR=$ROOT/target/icydb`
  - `cargo install` style developer-tool installation remains on the normal
    global cargo home so tool binaries do not get redirected into the repo

### Public API Note

The intended public entrypoint is the mode-driven surface, not three unrelated
helper APIs:

```rust
let patch = UpdatePatch::new()
    .set_field(User::MODEL, "id", Value::Ulid(user_id))?
    .set_field(User::MODEL, "email", Value::String("alice@example.com".into()))?;

let saved = session.mutate_structural::<User>(
    user_id,
    patch,
    StructuralMutationMode::Update,
)?;
```

This keeps the user model orthogonal:

- one patch representation
- one mutation entrypoint
- explicit write intent via `StructuralMutationMode`
- the same validation and commit semantics as the typed path underneath
