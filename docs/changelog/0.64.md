# 0.64 Changelog

## 0.64.1 - 2026-03-25 - SQL Predicate Encoding Wasm Reduction

### Summary

`0.64.1` keeps the full SQL surface but trims more of the fixed query-lane wasm
cost by collapsing repeated predicate canonicalization and encoding logic onto
shared owners.

The important result is that this is not another SQL feature cut. The retained
win comes from reducing duplicated sort-key, compare-list, and map-entry
ordering machinery inside the predicate/query authority path that the SQL query
root already retains.

- keep the SQL feature surface unchanged while shrinking more of the fixed
  predicate/planner support cost
- prefer shared owner-local canonicalization helpers over repeated local sort
  and encoding closures
- stop at the last good measured state and revert the stable-vs-unstable and
  tiny helper extractions that regressed raw wasm

### Changed

- consolidated repeated predicate child canonicalization in:
  - `crates/icydb-core/src/db/predicate/normalize.rs`
  - `normalize_and(...)` and `normalize_or(...)` now share one child
    sort/dedup helper instead of carrying separate cached-sort call shapes
- moved set literal canonicalization onto the shared access owner in:
  - `crates/icydb-core/src/db/predicate/normalize.rs`
  - `crates/icydb-core/src/db/access/canonical.rs`
  - predicate set-literal normalization now reuses the same deterministic
    value-set ordering and dedup boundary as access planning
- shared `IN` / `NOT IN` compare-literal normalization between predicate sort
  keys and predicate fingerprint hashing in:
  - `crates/icydb-core/src/db/predicate/encoding.rs`
  - the compare-list sort/dedup path is now owned once and reused by both
    encoding surfaces
- moved canonical map-entry ordering onto one value-owned helper in:
  - `crates/icydb-core/src/value/mod.rs`
  - `crates/icydb-core/src/value/hash.rs`
  - `crates/icydb-core/src/db/predicate/encoding.rs`
  - canonical map ordering is now owned by `Value` and reused by both
    predicate sort-key encoding and canonical value hashing
- replaced the generic closure-based framed sort-key helper with explicit framed
  writers in:
  - `crates/icydb-core/src/db/predicate/encoding.rs`
  - predicate sort keys, value sort keys, compare-value sort keys, and coercion
    sort keys now use one explicit frame begin/finish path instead of repeated
    closure monomorphizations
- explicitly rejected and reverted the following measured regressions:
  - stable-sort to unstable-sort swaps in planner/access canonicalization
  - tiny field/value branch helper extractions in predicate encoding

### Wasm Impact

Measured on `wasm-release`, `sql-on`, raw non-gzipped wasm only.

Relative to the prior `0.64.0` baseline:

- `minimal`
  - built `.wasm`: `1276874 -> 1270095` (`-6779`)
  - shrunk `.wasm`: `1190526 -> 1184089` (`-6437`)
- `one_simple`
  - built `.wasm`: `1380973 -> 1374322` (`-6651`)
  - shrunk `.wasm`: `1288385 -> 1282070` (`-6315`)

This reduction is almost entirely fixed-cost SQL/query-root trimming, not
per-entity slope movement.

- `WASM_CANISTER_NAME=one_simple WASM_PROFILE=wasm-release WASM_SQL_VARIANTS=sql-on bash scripts/ci/wasm-size-report.sh`

### Notes

- the retained query/planner root is still large, but the easy local predicate
  duplication seam was still worth another measured pass
- the sort-family monos were not solved by switching sort algorithms; the real
  wins came from reducing repeated canonicalization and encoding ownership
- after this patch, the next likely SQL wasm target is a larger retained owner
  seam again, not more tiny helper extraction in predicate encoding

## 0.64.0 - 2026-03-25 - Public Structural Mutation API

### Summary

`0.64.0` turns the earlier structural field-level serialization work into one
public write surface instead of leaving it as an internal executor experiment.

The new public API is mode-driven rather than helper-fragmented: callers build
an `UpdatePatch`, choose `StructuralMutationMode`, and submit that through
`DbSession::mutate_structural(...)`. The important contract is that this is not
one weaker bypass around typed writes. The structural path still routes through
the shared structural mutation input, rebuilds the persisted row through the
same slot/row boundary, decodes back into `E`, and reuses the existing typed
save preflight before commit staging.

This release also fixes one repeated local tooling problem: normal IcyDB
build/test/check flows now use repo-local cargo state, which keeps sibling
repositories on the same filesystem from blocking each other on the shared
cargo home / target locks.

- expose one complete structural mutation surface instead of freezing
  `update` first and retrofitting `insert` / `replace` later
- keep typed and structural mutation semantics aligned through the same
  validation and commit pipeline
- isolate repo-managed cargo flows into IcyDB-local cargo state so routine
  local builds and hooks stop contending with other repositories

### Changed

- exposed the public façade structural mutation contract in `icydb`:
  - `crates/icydb/src/db/session/mod.rs`
  - `crates/icydb/src/db/mod.rs`
  - new public surface:
    - `DbSession::mutate_structural(key, patch, mode)`
    - `UpdatePatch`
    - `StructuralMutationMode`
  - mode semantics are explicit:
    - `Insert`: patch must describe a full after-image and fails if the row already exists
    - `Update`: patch applies over the existing row and fails if the row is missing
    - `Replace`: patch must describe a full after-image, rebuilds from an empty row image, and inserts if the row is missing
- completed the structural write pipeline under one shared mutation core:
  - `crates/icydb-core/src/db/data/persisted_row.rs`
  - `crates/icydb-core/src/db/data/row.rs`
  - `crates/icydb-core/src/db/executor/mutation/mod.rs`
  - `crates/icydb-core/src/db/executor/mutation/save.rs`
  - `crates/icydb-core/src/db/session/write.rs`
  - field-level decode, field-level encode, serialized slot-patch lowering,
    raw-row patch replay, and mutation staging now all run through one
    canonical structural slot/row boundary instead of competing direct row
    encode paths
- locked structural-vs-typed mutation parity before the façade exposure:
  - `crates/icydb-core/src/db/executor/tests/mutation_save.rs`
  - insert / update / replace parity is now pinned on equivalent rows
  - structural unique-index conflicts and key-mismatch handling stay aligned
    with the existing typed path
  - unknown field names are rejected at the patch construction boundary
- tightened the public contract wording around the new mutation surface:
  - `crates/icydb/src/db/session/mod.rs`
  - `docs/design/0.64-per-field-serialization/0.64-design.md`
  - `docs/design/0.64-per-field-serialization/0.64-status.md`
  - the docs now reflect the actual landed API shape instead of the earlier
    proposal-state design
- normalized the most obvious structural-mutation error wording ahead of
  release:
  - `crates/icydb-core/src/error/mod.rs`
  - `crates/icydb-core/src/db/executor/tests/mutation_save.rs`
  - the public-facing messages now say:
    - `unknown field for structural mutation`
    - `structural mutation produced an invalid entity`
  - instead of exposing more implementation-shaped phrases like
    `field not found` or `after-image invalid`
- isolated repo-managed cargo workflows into IcyDB-local cargo state:
  - `Makefile`
  - `.githooks/pre-commit`
  - `.githooks/pre-push`
  - `scripts/env/cargo-local.sh`
  - `scripts/app/build.sh`
  - `scripts/ci/bump-version.sh`
  - `scripts/ci/security-check.sh`
  - `scripts/ci/validate-workflow-packages.sh`
  - `scripts/ci/wasm-size-report.sh`
  - normal repo flows now use:
    - `CARGO_HOME=$ROOT/.cache/cargo/icydb`
    - `CARGO_TARGET_DIR=$ROOT/target/icydb`
  - `cargo install` style developer-tool installation remains on the normal
    global cargo home so tool binaries do not get redirected into the repo

### Public API Note

The intended public entrypoint is the mode-driven surface, not three unrelated
helper APIs:

```rust
let patch = UpdatePatch::new()
    .set_field(User::MODEL, "id", Value::Ulid(user_id))?
    .set_field(User::MODEL, "email", Value::String("alice@example.com".into()))?;

let saved = session.mutate_structural::<User>(
    user_id,
    patch,
    StructuralMutationMode::Update,
)?;
```

This keeps the user model orthogonal:

- one patch representation
- one mutation entrypoint
- explicit write intent via `StructuralMutationMode`
- the same validation and commit semantics as the typed path underneath

### Notes

- `0.64.0` is intentionally one complete public structural mutation surface,
  not an update-only preview that would have forced later insert/replace
  semantics to work around a prematurely frozen contract
- the structural mutation path is a dynamic ingress, not a second weaker write
  engine: it still normalizes back through typed validation before commit
- one exact targeted cargo test re-run under the new repo-local cargo env could
  not complete in the sandbox because `index.crates.io` DNS resolution was
  unavailable; the kept release note only relies on the successful local fmt,
  clippy, existing persisted-row test, and non-network script validation above
