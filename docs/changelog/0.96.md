# 0.96 Changelog

## 0.96.1

- `0.96.1` closes one grouped aggregate `FILTER` validation hole that was still bypassing the normal SQL lowering boundary:
  - global aggregate `FILTER` already rejected alias misuse such as `COUNT(*) FILTER (WHERE total_rows > 0)` with the expected `unknown field` diagnostic
  - grouped aggregate `FILTER` did not
  - grouped shapes like `COUNT(*) FILTER (WHERE ranked_count > 0)` or the same alias misuse nested inside `CASE` were being lowered onto grouped aggregate specs and only failing later when grouped execution tried to compile the invalid filter expression on the scalar seam
- that meant one clearly invalid SQL family was escaping lowering and tripping an internal invariant instead of returning a user-facing error:
  - direct grouped alias leakage inside `FILTER`
  - sibling alias leakage inside `FILTER`
  - grouped alias leakage inside `CASE` used by `FILTER`
  - grouped alias leakage inside aggregate `FILTER` terms referenced from `HAVING`
- the fix is narrow and grouped-lowering owned:
  - grouped aggregate calls are now validated against the entity model before they are attached to the grouped structural query
  - grouped aggregate input expressions keep the existing aggregate-input validation behavior
  - grouped aggregate filter expressions now also validate their first unknown field and scalar-compilation eligibility before grouped execution ever sees them
  - as a result, grouped alias leakage now fails at the normal SQL lowering boundary with `unknown field '...'` instead of reaching `grouped aggregate filter expression must stay on the scalar seam`
- user-visible result:
  - invalid grouped queries such as:

```sql
SELECT
  class_name,
  COUNT(*) FILTER (WHERE ranked_count > 0) AS ranked_count
FROM Character
GROUP BY class_name
ORDER BY class_name ASC
LIMIT 10;
```

  now fail cleanly with an ordinary SQL field-resolution error
  - valid grouped filtered aggregates are unchanged
- focused verification for `0.96.1` covered both lowering and live grouped session execution:
  - `cargo fmt --all`
  - `cargo test -p icydb-core compile_sql_command_rejects_grouped_filter_alias_references_before_execution -- --nocapture`
  - `cargo test -p icydb-core compile_sql_command_rejects_grouped_filter_alias_references_inside_case_before_execution -- --nocapture`
  - `cargo test -p icydb-core execute_sql_statement_grouped_filter_alias_unknown_field_stays_specific -- --nocapture`
  - `cargo test -p icydb-core execute_sql_statement_grouped_filter_alias_unknown_field_inside_case_stays_specific -- --nocapture`
- `0.96.1` also closes one grouped `ORDER BY` alias-normalization gap that showed up immediately after the grouped `FILTER` validation repair:
  - grouped aggregate aliases already worked when the alias itself was the whole order term, such as `ORDER BY total_count DESC`
  - direct post-aggregate arithmetic order expressions also already worked, such as `ORDER BY COUNT(*) + 1 DESC`
  - but the mixed spelling `ORDER BY total_count + 1 DESC` still failed with `unknown field 'total_count'`
  - the bug was narrow: `ORDER BY` alias normalization only rewrote a top-level `Field(alias)` target and did not recurse when the alias appeared as one leaf inside a larger arithmetic, function, null-test, or `CASE` expression
- the fix stays on the SQL normalization boundary:
  - grouped and scalar `ORDER BY` terms now recursively substitute projection aliases before identifier scoping
  - that keeps alias-based and direct spellings on the same canonical SQL expression tree before ordinary lowering begins
  - unsupported leaves still fail closed through the usual field-resolution path; this does not add a new planner-only alias mechanism
- user-visible result:
  - grouped queries such as:

```sql
SELECT
  class_name,
  COUNT(*) AS total_count
FROM Character
GROUP BY class_name
ORDER BY total_count + 1 DESC, class_name ASC
LIMIT 10;
```

  now normalize and execute like the direct canonical spelling `ORDER BY COUNT(*) + 1 DESC, class_name ASC`
- focused verification for the grouped `ORDER BY` alias-expression repair covered both lowering and live grouped execution:
  - `cargo fmt --all`
  - `cargo test -p icydb-core compile_sql_command_normalizes_grouped_count_order_by_alias_inside_expression_with_limit -- --nocapture`
  - `cargo test -p icydb-core compile_sql_command_normalizes_grouped_wrapped_aggregate_order_by_alias_inside_expression_with_limit -- --nocapture`
  - `cargo test -p icydb-core grouped_select_helper_equivalent_row_matrix_matches_canonical_rows -- --nocapture`
- `0.96.1` also closes the small post-convergence tightening items that were worth doing before moving on:
  - global aggregate `HAVING` now has direct null-semantics coverage for `IS NULL`, `IS NOT NULL`, `= NULL`, and `!= NULL`, including the aliased global aggregate form
  - this locks the most fragile shared boolean invariant after `WHERE`, grouped `HAVING`, and global aggregate `HAVING` were unified onto the same expression-owned path
  - the remaining builder-only raw-`Expr` preview helper was renamed to `eval_builder_expr_for_value_preview(...)` and explicitly documented as non-runtime, so the last recursive preview evaluator no longer reads like an executor hot-path seam
  - grouped Top-K order admission now computes one small local proof summary per parsed order expression instead of rescanning the same tree separately for field-only references, aggregate presence, and wrapper-function rejection
  - the lingering clone-heavy read-only `SqlSelectItem -> SqlExpr` aggregate check was reduced by adding `SqlSelectItem::contains_aggregate()` and deleting the now-dead lowering wrapper that only existed to answer that one question
- the important boundary after that closeout is unchanged:
  - production execution still stays on compiled expression forms
  - no new SQL surface was widened
  - the remaining work after `0.96.1` is mostly wording or future feature slices, not another convergence repair
- focused verification for the closeout portion of `0.96.1` covered the newly locked semantic and planner edges:
  - `cargo test -p icydb-core global_aggregate_having_ -- --nocapture`
  - `cargo test -p icydb-core grouped_top_k_classifier_rejects_unsupported_wrapper_functions -- --nocapture`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`

## 0.96.0

- `0.96.0` closes one narrow grouped Top-K follow-through gap in the post-`0.95` SQL surface:
  - grouped filtered aggregates were already executable
  - grouped filtered aggregate aliases were already valid in `HAVING`
  - grouped filtered aggregate alias ordering already worked for simpler boolean filters such as `NOT is_npc`
  - but grouped `ORDER BY` still reparsed canonical rendered order terms through a smaller internal expression parser that did not understand normalized scalar function-call forms such as `IS_NULL(...)` and `IS_NOT_NULL(...)`
- that left one inconsistent boundary:
  - grouped queries such as `COUNT(*) FILTER (WHERE guild_rank IS NOT NULL)` executed correctly in projection and `HAVING`
  - the same aggregate failed only when used as a grouped aggregate order term, because grouped Top-K admission rejected the normalized rendered label before plan construction
- the code change is intentionally local to grouped order admission:
  - the grouped post-aggregate order-expression parser now accepts normalized scalar function-call forms inside aggregate `FILTER` expressions
  - this includes null-test forms such as `IS_NULL(field)` and `IS_NOT_NULL(field)`
  - boolean compositions built from those normalized forms, such as `IS_NOT_NULL(guild_rank) AND level >= 10`, now reconstruct the same planner `Expr` shape instead of failing during grouped order-term parsing
  - grouped Top-K admission therefore sees the same filtered aggregate identity that grouped execution and `HAVING` were already using
- user-visible result:
  - grouped aggregate ordering now works for filtered aggregates whose `FILTER` predicates include null tests and their boolean compositions
  - direct and alias spellings both benefit because the failure was in canonical grouped order-term reparsing, not alias resolution

```sql
SELECT
  class_name,
  COUNT(*) FILTER (WHERE guild_rank IS NOT NULL) AS ranked_count
FROM Character
GROUP BY class_name
ORDER BY ranked_count DESC, class_name ASC
LIMIT 10;
```

- important boundary notes:
  - this patch does not widen the general grouped `ORDER BY` function family
  - it fixes one inconsistency inside the already-admitted grouped filtered aggregate lane by teaching grouped Top-K matching to understand the normalized function-call forms that filtered aggregate rendering already emits
- focused verification for `0.96.0` covered both parser reconstruction and grouped Top-K admission:
  - `cargo fmt --all`
  - `cargo test -p icydb-core grouped_order_parser_preserves_filtered_null_test_aggregate_shape -- --nocapture`
  - `cargo test -p icydb-core grouped_order_parser_preserves_filtered_null_test_boolean_composition_shape -- --nocapture`
  - `cargo test -p icydb-core grouped_top_k_classifier_accepts_filtered_aggregate_null_test_terms -- --nocapture`
  - `cargo test -p icydb-core grouped_top_k_classifier_accepts_filtered_aggregate_null_test_boolean_compositions -- --nocapture`
