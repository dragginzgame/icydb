# 0.75 — Crosscutting Audit Refresh

## 0.75.6 - 2026-04-11 - Executor Planning Boundary and Cleanup

`0.75.6` keeps the `0.75` line on cleanup rather than widening the `0.76` query surface. The point of this patch is to make the executor's pre-runtime planning boundary real instead of nominal, collapse a few more duplicated SQL and pipeline dispatch flows onto shared owners, and remove one remaining fake compatibility story from diagnostics. Query semantics stay the same. The payoff here is architectural honesty: route, continuation, and preparation now live under one planning root; SQL dispatch carries less repeated control flow; and incompatible persisted bytes are classified as corruption instead of an invented compatibility lane.

What changed:

* made `db::executor::planning` a real ownership boundary instead of a light namespace by moving:
  * `db::executor::preparation` -> `db::executor::planning::preparation`
  * `db::executor::continuation` -> `db::executor::planning::continuation`
  * `db::executor::route` -> `db::executor::planning::route`
* kept `db::executor` as the stable subsystem root while re-exporting the route surface there, so runtime callers still consume one executor boundary even though planning now owns the pre-runtime trees physically
* repointed the direct consumers of those planning-owned contracts across:
  * executor runtime and pipeline entrypoints
  * explain descriptors
  * aggregate and delete execution
  * access/execution-contract helpers
  * query/session explain surfaces
* finished the route cleanup that had been left as adjacent classification layers by:
  * removing the redundant executor-side route-shape wrapper around `ExecutionRoutePlan`
  * trimming access-owned shape facts out of `RouteCapabilities`
  * keeping route-only planning outputs and fast-path/runtime contracts under one clearer owner
* collapsed the duplicated typed-vs-generated SQL dispatch tree onto one shared authority-based route in `db::session::sql::dispatch`, so aggregate detection, computed-projection handling, lowered `SELECT` / `DELETE` dispatch, and `EXPLAIN` fallback no longer maintain two near-parallel control flows
* removed one unused lowered SQL projection helper that no longer had live callers after that dispatch cleanup, and deleted the stale audit-doc mention that still referenced it
* simplified the executor pipeline/orchestrator surface by folding several helper-only files back into their true owners:
  * orchestrator contract mode/surface helpers
  * orchestrator state context
  * orchestrator strategy context/grouping
  * grouped runtime context
* removed a few remaining thin forwarding shims in the same runtime area:
  * grouped route resolution now calls the real grouped-route helper directly
  * entrypoint cursor resolution now uses `LoadCursorResolver` directly instead of one extra delegating wrapper
* removed the diagnostics `compatibility_findings` bucket and folded incompatible persisted row formats into ordinary corruption reporting, which matches the repo-wide hard-cut internal format policy instead of pretending there is a supported backwards-compat lane
* updated the affected diagnostics, executor snapshot, and compile-fail expectations so the cleanup lands against the actual current outputs rather than stale snapshots or rustc wording

What this means architecturally:

* executor ownership is clearer now:
  * `db::executor::planning` owns pre-runtime route policy, continuation policy, and execution preparation
  * kernel, stream, pipeline, and terminal runtime remain under the executor runtime boundary
* route planning no longer sits as an ambiguous sibling next to runtime while also acting as a planning subsystem; it is physically under planning where that policy belongs
* SQL dispatch and grouped/pipeline runtime carry fewer local wrapper flows, so there are fewer places where small control-flow changes can drift apart
* diagnostics no longer suggests a supported “compatibility” class for stale on-disk bytes when the actual contract is a hard cut to one current internal format

What did not change:

* no SQL syntax change
* no planner policy widening
* no query result change
* no new cursor, aggregate, or mutation capability
* no new fast path

Verification:

* `cargo fmt --all`
* `cargo check -p icydb-core --all-targets`
* `cargo test -p icydb-core db::executor::route::tests -- --nocapture`
* `cargo test -p icydb-core execute_sql_dispatch -- --nocapture`
* `cargo test -p icydb-core explain_sql_ -- --nocapture`
* `cargo test -p icydb-core --test compile_fail -- --nocapture`

Notes:

* `0.75.6` is a cleanup and ownership patch, not a feature patch
* the only user-visible payload-shape change in this slice is diagnostics: integrity reports no longer expose `compatibility_findings`; incompatible persisted formats are reported under corruption instead

## 0.75.5 - 2026-04-11 - Cursor Hard-Cut and Repo Cleanup

`0.75.5` keeps the `0.75` line on cleanup rather than new feature surface. The point of this patch is to remove the last obvious fake-compatibility cursor protocol residue, collapse a few more inert wrapper/helper seams, and delete helper-only modules that no longer justify independent files after the earlier executor/query normalization work. This is still not planner widening or performance tuning. It is repository simplification: less fake versioning, fewer forwarding layers, fewer tiny one-function files, and clearer active boundaries.

What changed:

* hard-cut the internal cursor token protocol to one current wire shape by removing:
  * token `version` fields
  * `CursorTokenVersion`
  * `GroupedCursorTokenVersion`
  * unsupported-version decode branches
  * cursor-version mismatch errors and tests
  from:
  * `db::cursor::token::wire`
  * `db::cursor::token::scalar`
  * `db::cursor::token::grouped`
  * `db::cursor::token::error`
  * `db::cursor::error`
* removed the remaining cursor-adjacent fake-version naming in continuation-signature hashing by renaming:
  * `new_continuation_signature_hasher_v1()` -> `new_continuation_signature_hasher()`
  * `ExplainHashProfile::ContinuationV1` -> `ExplainHashProfile::Continuation`
  * `contsig:v1` -> `contsig`
  across:
  * `db::query::fingerprint`
  * `db::query::fingerprint::hash_parts`
  * `db::query::fingerprint::shape_signature`
* kept the query/session boundary cleanup on the safe side:
  * added one shared session helper for recovering planner-visible indexes in `db::session::query`
  * removed repeated internal `visible_indexes -> forward` scaffolding there without flattening the real session/query seam
  * cleaned the public `icydb` facade packaging helpers so repeated response/grouped/projection wrapping now routes through shared package/render helpers instead of open-coding the same DTO shaping
* finished the remaining parameterizable aggregate streaming duplication by introducing one shared existing-row traversal loop in `db::executor::aggregate::helpers` and routing:
  * numeric streaming folds
  * scalar projection `COUNT(non-null)` streaming folds
  through that same loop, leaving only field extraction and fold behavior different per callsite
* kept field-extrema separate as a valid specialization, but simplified its implementation by removing the nested closure-shaped loop in `db::executor::aggregate::field_extrema` and replacing it with one explicit per-key fold helper
* cleaned two more low-risk structural seams outside the hot path:
  * relation reverse-index target-key decoding now shares one structural storage-key decoder path in `db::relation::reverse_index` instead of duplicating the same scalar-fast-path vs field-bytes split for “collect all keys” and “contains one key”
  * deleted the inert `*_with_visible_indexes` explain wrappers that ignored the `visible_indexes` argument and immediately forwarded to the same canonical explain descriptor helpers
* continued the repo-wide narration cleanup by replacing the stale “module-local ownership and contracts” boilerplate with concrete ownership/boundary descriptions across the active `session`, `session::sql`, `relation`, `executor::explain`, `query::api`, and query-planning surfaces
* deleted several tiny helper-only modules that no longer justified their own files:
  * `db::query::api::private`
  * `db::query::explain::nodes::aggregate`
  * `db::query::explain::nodes::pipeline`
  * `db::query::explain::nodes::scan`
  * `db::query::explain::nodes::terminal`
  * `db::query::plan::limit_zero`
  * `db::query::intent::order`
  * `db::query::plan::constant_predicate`
* folded those helpers back into their natural owners:
  * sealed response-cardinality marker now lives inline in `db::query::api::result_ext`
  * explain node-layer ownership matchers now live in `db::query::explain::nodes::mod`
  * zero-limit and constant-predicate helpers now live on `db::query::plan::mod`
  * the one tiny order-appending helper now lives directly in `db::query::intent::mutation`

What this means architecturally:

* internal cursor protocol/versioning is now honest again:
  * there is one current cursor token shape
  * old tokens are not treated as a versioned compatibility surface
  * the code no longer suggests a rolling multi-version protocol that never actually existed
* a few more seams now line up with actual ownership:
  * session recovers visibility once, then forwards
  * aggregate streaming traversal is shared where the algorithm is shared
  * relation target decode is shared where the decode contract is shared
  * field-extrema stays separate where the algorithm is genuinely different
* the `db::query` subtree carries less file-level indirection:
  * helper-only leaf files are gone where they were just one trivial function or one tiny ownership matcher
  * module roots keep the real boundaries

What did not change:

* no new SQL or fluent query feature surface
* no new cursor capability
* no new planner policy
* no change to aggregate semantics
* no change to relation semantics
* no new performance fast path

## 0.75.4 - 2026-04-11 - Dead Surface and Boundary Cleanup

`0.75.4` keeps the `0.75` line on cleanup rather than widening SQL surface area. The point of this patch is to finish pruning dead or transitional internal shapes that were still left over after the executor/page/sql normalization work, especially around projection expressions, delete packaging, and a few session/executor boundary wrappers. This patch does not add generic projection expressions. It makes the current bounded surface clearer, removes dead non-test scaffolding, and leaves `icydb-core` passing strict all-target clippy again without dead-code allowances.

What changed:

* removed the dead planner-owned projection-expression selection surface by deleting `ProjectionSelection::Expression` and the related dead helpers in `db::query::plan::expr::projection`, then deleting the covering/planner test scaffolding that only existed to exercise that removed variant
* moved the remaining legacy expression operator surface out of non-test builds by gating the still-useful planner/executor contract scaffolding to tests in:
  * `db::query::plan::expr::ast`
  * `db::query::plan::expr::scalar`
  * `db::query::plan::expr::type_inference`
  * `db::query::fingerprint::projection_hash`
  * `db::executor::projection::eval`
  * `db::executor::projection::grouped`
  * `db::session::sql::projection::labels`
* deleted the dead `StorageExistenceWitness` covering row mode and the explain/runtime branches that only existed for it, so covering-read planning and descriptor output no longer carry a row-mode variant with no live production caller
* narrowed a few stale production surfaces to tests only instead of carrying them as dead public-ish API:
  * the typed `Query<E>::select_fields(...)` hook
  * test-only scalar projection error/eval scaffolding
  * several old diagnostics and planner-expression support branches that no longer participate in the shipped `0.76` surface
* finished the delete cleanup by pushing the three delete output modes onto shared post-access and packaging helpers in `db::executor::delete`, so the typed-row, typed-count, and structural-row delete variants no longer each own their own decode/filter/package ladder
* collapsed the low-impact value-vs-text SQL dispatch duplication at the session boundary by sharing one structural projection preparation helper in `db::session::sql::dispatch` and one lowered-select bridge in `db::session::sql::dispatch::lowered`, leaving only the final payload packaging different
* continued the single-pass/internal-wrapper cleanup inside canonical paths:
  * grouped page finalization now projects directly from candidates instead of staging another filtered candidate vector first
  * `db::executor::terminal::page` and `db::executor::pipeline::operators::terminal::runtime::sql` now choose finalize/payload shape once and keep the selected shaping path explicit
  * `db::executor::projection::materialize::structural` now shares one selected shaper for slot-row/data-row value vs text materialization instead of mirroring adjacent loops
* cleaned the last clippy/test fallout from that pruning by removing dead arithmetic/operator cases, splitting test-vs-non-test helper signatures where clippy was correctly identifying unnecessary wrappers, and deleting stale `dead_code` suppressions rather than masking them
* added the explicit `0.76` design note that broad planner-owned projection expressions remain out of scope for `0.76`, so the now-pruned expression scaffolding is documented as non-goal cleanup instead of looking like a silently missing feature

What this means architecturally:

* the currently shipped SQL/planner/executor surface is more honest about what exists:
  * bounded computed/session projection support remains
  * generic planner-owned projection expressions do not
  * dead legacy operator scaffolding no longer leaks into normal builds as if it were an active product surface
* the delete/session edges now reuse more shared helpers instead of maintaining lightly different packaging ladders
* strict lint cleanliness is meaningful again for this line because dead-code cleanup now uses deletion or `cfg(test)` instead of `allow(dead_code)`

What did not change:

* no new SQL syntax
* no widening of `query_from_sql(...)`
* no new projection-expression feature
* no planner policy change
* no runtime behavior change beyond the deletion of dead internal branches

Verification:

* `cargo check -p icydb-core --all-targets`
* `cargo clippy -p icydb-core --all-targets -- -D warnings`

Notes:

* `0.75.4` is a maintenance and boundary-cleanup patch, not a feature patch
* this patch intentionally treats dead internal expression scaffolding as cleanup debt to delete or fence to tests, not as implied upcoming feature work

## 0.75.3 - 2026-04-10 - Structural Executor and SQL Runtime Cleanup

`0.75.3` keeps the `0.75` line on audit and normalization work rather than another round of query-specific tuning. The point of this patch is to delete parallel execution shapes that had drifted during the earlier executor and route refactors, push SQL planning and projection back onto clearer structural boundaries, and stop perf-only attribution wiring from leaking into the default runtime build. The result is not a new planner feature or a new fast path. It is one simpler executor/story for scalar SQL reads, covering reads, and structural page materialization.

What changed:

* collapsed the duplicated scalar SQL projection entrypoints in `db::executor::pipeline::entrypoints::scalar` onto one canonical structural projection-page executor, then had `db::executor::projection::materialize::structural` do the final value-vs-text shaping from that shared page instead of mirroring the same projected-row fallback ladder twice
* moved visible-index plus structural-plan building for lowered SQL reads into one shared helper in `db::session::sql`, so dispatch and explain no longer rebuild the same visible-index/planner setup in parallel modules
* removed the extra lowered-select forwarding wrapper in `db::session::sql::dispatch::lowered`, so typed SQL dispatch now converts at the boundary instead of paying another internal handoff that only renamed the same structural result
* made the scalar execution boundary structural again by removing executor-emitted `ProjectedRows` and `RenderedProjectedRows` payload variants from `db::executor::pipeline::contracts::execution::inputs`; terminal runtime now returns only `DataRows` or `SlotRows`, and SQL-specific shaping happens in `db::executor::projection::materialize::structural`
* deleted the orphaned single-component covering/index-scan branch by removing `db::index::scan::covering`, collapsing callers onto the canonical multi-component covering path in `db::executor::covering`, and gating the leftover single-component codec helper to tests only
* consolidated the shared kernel row-scan flow in `db::executor::terminal::page` and had the SQL row collector in `db::executor::pipeline::operators::terminal::runtime::sql` reuse it, eliminating a second near-copy of payload-mode selection, scan-budget handling, and row-read loop dispatch
* removed the redundant `FastPathDecision` wrapper from `db::executor::pipeline::runtime::fast_path`, so fast-path resolution now carries `Option<FastPathKeyResult>` directly through the hit-vs-fallback handoff
* normalized the route-covering and constant-covering SQL prep in `db::executor::pipeline::operators::terminal::runtime::sql` by sharing route-contract decoding, predicate-slot admissibility, slot-layout building, constant-covering template prep, and slot-row finishing helpers instead of repeating slightly different versions of that prep in adjacent branches
* added an opt-in `perf-attribution` feature across `icydb-core`, `icydb`, the SQL parity canister, and the PocketIC harness wiring, so deep executor-attribution probes can still be collected when requested without leaving that measurement surface compiled into the default runtime path
* added a combined-pressure session regression in `db::session::tests` that locks a filtered multi-component covering read with projection, composite `ORDER BY`, and pagination against the canonical entity-row result, so the deleted single-component branch and shared row-scan cleanup are both covered under one higher-stress case

What this means architecturally:

* planning, execution, and boundary shaping are cleaner again:
  * `db::session::sql` owns structural plan construction
  * executor returns structural pages
  * typed/rendered shaping happens at the boundary instead of leaking back into the runtime payload model
* the SQL runtime now has fewer parallel helpers to keep in sync:
  * cursorless row collection, route-covering prep, constant-covering prep, and slot-row finishing all reuse shared contracts
  * the old single-component special case is gone where it was only an adapter shape, not a meaningfully different algorithm
* perf work is easier to trust after this patch too:
  * the default build path is no longer forced to carry deep attribution wiring
  * future instruction-count work can measure one cleaner canonical path instead of a mix of legacy wrapper lanes

Verification:

* `cargo fmt --all`
* `cargo check -p icydb-core`
* `cargo test -p icydb-core sql_projection_materialization_prefers_retained_slot_rows -- --nocapture`
* `cargo test -p icydb-core fast_path_spec_arity_accepts_single_spec_shapes -- --nocapture`
* `cargo test -p icydb-core execute_sql_projection_filtered_composite_expression_order_only_pagination_matches_entity_rows -- --nocapture`

Notes:

* `0.75.3` is a structural cleanup patch, not a benchmark claim
* no planner policy changed

## 0.75.1 - 2026-04-10 - Generic SQL Execute Prep Recovery

`0.75.1` follows the structural `0.75.0` cleanup by removing a generic execute-path regression that had crept back into tiny SQL projected reads. The regression was not in parsing, lowering, or the earlier generated `EntityModel` hard-cut. It was in executor-owned projection preparation: non-validating SQL execution was still rebuilding planner-frozen projection shape, and SQL prep was still eagerly compiling mutually exclusive covering layouts that constant projected-row queries could never consume.

What changed:

* fixed a constant projected-row fast-path bug in [sql.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/operators/terminal/runtime/sql.rs) where the constant projected-source layout was incorrectly gated on decoded-component slot-layout state; that bug had knocked `SELECT id ... ORDER BY id LIMIT 1` off the projected-row lane and onto slot rows
* stopped building `PreparedProjectionShape` for non-validating SQL execution attempts in [inputs.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs), so SQL immediate-materialization no longer pays:
  * cloned `ProjectionSpec`
  * cloned scalar projection program vectors
  * rebuilt projected-slot masks
  when the query never uses shared projection validation
* removed the now-dead duplicated direct-slot copy from [mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/projection/materialize/mod.rs), leaving that executor-owned projection bundle only with the pieces still consumed by slot-row validation and structural fallback materialization
* tightened SQL-only prep in [sql.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/operators/terminal/runtime/sql.rs) so PK/constant direct projections without residual predicates no longer eagerly build:
  * decoded-component covering slot layout
  * direct/single-component projected-source layouts
  * constant covering slot-row templates
  that those queries can never use
* kept the projected-row family selection explicit in [sql.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/pipeline/operators/terminal/runtime/sql.rs) by recording when prep has already proven the constant-only projected-row family, then using that to skip dead route-covering work before the actual emitter runs

Measured effect on the canonical tiny query:

* target query: `SELECT id FROM Customer ORDER BY id ASC LIMIT 1`
* surface: `TypedDispatchCustomer`
* regressed post-`0.75.0` line before recovery: `330,725` average local instructions
* after the generic execute-prep recovery in this patch: `319,899` average local instructions
* net recovery in this patch line: `-10,826` instructions, about `-3.3%`

Attribution on that same query moved from:

* total `330,413`
* execute `265,301`

to:

* total `320,129`
* execute `254,354`

That is the important architectural signal: the reclaimed cost is execute-phase fixed overhead, not parser/lowering cost and not row-open cost. The query stays on:

* `projected_rows_path_hits = 1`
* `slot_rows_path_hits = 0`
* `rows_opened = 0`

What this means architecturally:

* the generated `EntityModel` / trusted schema hard-cut from `0.74` remains valid and was not the source of the later regression
* the main `0.75.1` correction is boundary ownership:
  * planner owns frozen projection semantics
  * executor validation bundles are built only when validation is actually requested
  * SQL prep only builds the covering layout family that the current direct projection can really consume
* this is still a generic cleanup, not a new query-specific fast path

What did not change:

* no planner policy widening
* no SQL syntax or result-shape change
* no new route-specific shortcut
* no row format or schema model change
* no further `EntityModel` API change; this patch only reclaims the execute overhead that had hidden part of the earlier model-removal win

Verification:

* `cargo clippy -p icydb-core --all-targets -- -D warnings`
* `cargo test -p icydb-core execute_sql_projection_primary_key_covering_full_scan_returns_ordered_ids -- --nocapture`
* `POCKET_IC_BIN=/tmp/pocket-ic-server-13.0.0/pocket-ic ICYDB_SQL_PERF_PROBE_SURFACE=typed_dispatch_customer ICYDB_SQL_PERF_PROBE_SQL="SELECT id FROM Customer ORDER BY id ASC LIMIT 1" ICYDB_SQL_PERF_PROBE_REPEAT_COUNT=5 cargo test -p icydb-testing-integration --test sql_canister sql_canister_perf_probe_reports_sample_as_json -- --ignored --nocapture`

* a small single-field emitter specialization was tried during this patch line and explicitly discarded because it measured as noise or slight regression

What this means architecturally:

* `0.75.0` is a boundary-cleaning patch, not a semantics patch
* the main `0.75` complexity win here is not fewer total code paths; it is that the highest-pressure DB owners are no longer carrying mixed lexical, planning, explain, scan, persisted-row, and runtime orchestration responsibilities in single files
* the reduced-SQL, lowering, planner-range, explain-shared, index-scan, and persisted-row roots are now real subsystem boundaries again, which makes later authority, DRY, and velocity audits easier to act on without first re-cutting the file layout
* the static `EntityModel` pass is also stricter now: planner owns semantic query lowering, authority owns layout/decode authority, and production executor runtime mostly consumes those frozen contracts instead of re-reading model semantics
* terminal runtime remains a hotspot even after the split, but it is now at least decomposed into a real directory boundary with the SQL-heavy implementation kept under an explicit child instead of continuing to accrete in a single root file

What did not change:

* no planner admission widening
* no SQL syntax or semantics change
* no grouped policy change
* no covering-read policy change
* no persisted-row format change
* no new route-specific fast path
* no new perf claim; this patch is structural first
