# IcyDB 0.183 Changelog

## 0.183.24

### 📝 Summary

- Reduces executor work for indexed lookup shapes that showed up in the Toko
  generator audit: sparse `IN (...)` filters, empty indexed equality probes,
  limited indexed `IN (...)` pages, and covered indexed `count()`/`exists()`
  terminals.
- Adds a range-scan counter to SQL and fluent perf attribution so repeated
  indexed traversals are visible separately from row-store gets and streamed
  index-entry reads.
- Adds Toko-shaped audit rows for two different pressures:
  sparse many-value `IN (...)` filters with mostly missing prefixes, and
  repeated same-indexed full-entity loads that still hydrate the same layer
  over and over.

### 🔧 Changed

- Indexed `IN (...)` count terminals can now use synchronized exact
  index-prefix cardinality for every lowered lookup prefix, then sum the
  unique prefix counts without opening index range streams or hydrating rows.
- Indexed `exists()` terminals share the same prefix-cardinality preflight.
  Positive probes short-circuit once the requested page window is proven to
  contain a candidate row, while empty probes return `false` without opening a
  raw index range.
- Empty indexed equality loads now consult synchronized prefix-cardinality
  metadata before opening the raw index range. When the exact prefix is proven
  empty, the physical stream returns an empty result directly.
- Sparse indexed multi-lookup and branch-set streams now build an active
  prefix list before traversal, skipping prefixes whose exact cardinality is
  synchronized and zero.
- Limited indexed multi-lookup pages use lazy merged prefix streams with a
  page-sized fetch hint instead of materializing every matching entry from
  every `IN (...)` branch before pagination.
- Plan diagnostics now distinguish index range-scan calls from index-entry
  reads. This makes "many empty range opens" visible even when no rows or
  entries are returned.
- The SQL perf audit can now report averaged range-scan calls for SQL and
  fluent query samples, including repeated fluent load loops.

### 📊 Audit

- The sparse Toko-shaped `collection_id IN (...) ORDER BY id ASC LIMIT 50`
  audit row now prunes 250 empty prefixes before traversal. The focused run
  reported one index range scan instead of the old one-range-per-prefix shape.
- In executor unit coverage, an indexed equality lookup for a missing value now
  reports zero raw index range scans. The old shape opened one empty range per
  missing equality lookup, so Toko-style compatibility checks with many missing
  source-part rules avoid that repeated empty traversal cost.
- Covered indexed `IN (...)` `count()` and `exists()` paths now report zero
  rows scanned and zero raw range scans when prefix cardinality is
  synchronized.
- The limited indexed `IN (...)` page test now asserts bounded index-entry
  reads for the requested page instead of reading every matching index entry
  across all branches.
- The repeated same-indexed full-load audit intentionally remains a pressure
  signal rather than a fixed engine path. The current focused PocketIC sample
  still reports 256 row-store gets, 256 index-entry reads, and one range scan
  per averaged load, with shared-plan cache hits after the first pass.

### 🧪 Testing

- Adds focused executor coverage for sparse multi-lookup empty-prefix pruning,
  empty indexed equality range-scan skipping, limited multi-lookup page
  bounding, and prefix-cardinality-backed indexed `count()` and `exists()`.
- Extends SQL perf audit attribution with index range-scan calls and adds
  Toko-shaped scenarios for sparse `IN (...)` range pressure and repeated
  same-indexed full-entity reload pressure.
- Extends the deterministic SQL perf matrix with the sparse `IN (...)`
  hotspot row and includes range-scan hotspots in generated matrix reports.
- Revalidated the focused executor coverage, package checks for SQL and
  diagnostics builds, whitespace checks, repeated-load PocketIC audit, and
  sparse `IN (...)` PocketIC pressure audit.

### 🧭 Migration Notes

- No schema migration is required.
- No cursor format, persistence format, SQL syntax, or query-result semantics
  change.
- Prefix-cardinality shortcuts remain fail-open: if metadata is stale,
  unavailable, or the query has residual work that can change the candidate
  set, execution falls back to the existing scan/materialization path.
- This release reduces executor-side waste for empty and bounded indexed
  traversals, but it does not remove endpoint-level repeated full reloads.
  Toko can still get further wins by batching compatibility-rule lookups or
  caching the layer's existing part names within one mutation.

## 0.183.23

### 📝 Summary

- Reuses the accepted catalog loaded during SQL compilation when executing
  immediate global aggregate statements.
- Computes count-equivalent SQL aggregate terminals once, then fans the shared
  count value out to matching output columns.
- Normalizes non-distinct SQL counts over non-null literal inputs, such as
  `COUNT(1)`, onto the same row-count identity used by `COUNT(*)`.
- Adds fluent compile-phase attribution so cold query-call schema/catalog,
  cache-key, cache-lookup, prepare, plan-build, and cache-insert costs are
  visible separately.
- Reuses warmed accepted query catalog snapshots during filterless shared-plan
  misses, removing the repeated raw schema decode that dominated cold
  filterless fluent query calls.
- Uses a flat sibling merge for wider ordered branch streams while preserving
  the lower-overhead pair merge for two-branch routes.

### 🔧 Changed

- Immediate SQL global aggregate execution now has a catalog-aware route, so
  context-owned compiled SQL does not rebuild accepted schema projection state
  during execution.
- Aggregate identity owns the `COUNT(non-null literal)` to row-count
  normalization, keeping SQL terminal lowering, grouped aggregate identity, and
  cache-key semantics aligned.
- Structural scalar aggregate execution caches the first shared count result
  within a multi-terminal aggregate request and reuses it for later
  count-equivalent terminals before sending remaining terminals to the buffered
  reducer lane.
- Query-plan cache diagnostics now report fluent compile subphases for page and
  terminal attribution, and the audit canister carries those counters through
  averaged fluent perf samples.
- Accepted query catalog selection now reuses an already-cached snapshot when
  the selected identity matches the cached accepted catalog identity, avoiding a
  redundant persisted-schema decode on filterless shared-plan misses.
- Ordered key-stream `merge_all` keeps the existing pair merge for exactly two
  streams and switches to a flat sibling merge for three or more ordered
  streams, reducing nested polling overhead for wider branch sets without
  regressing the common two-branch route.

### 📊 Audit

- Fluent compile attribution showed the old cold filterless query-call hotspot
  was schema/catalog work rather than plan build: the affected samples spent
  about 10.9M instructions in `compile_schema`.
- After accepted catalog snapshot reuse, representative cold filterless fluent
  samples dropped from about 11.5M-11.8M total instructions to about
  0.70M-1.04M total instructions.
- The grouped fluent count sample dropped from about 11.7M total instructions
  to about 0.92M-0.93M total instructions.
- The target two-branch token page stayed stable at about 3.95M total
  instructions with about 1.16M direct key-stream instructions.
- The wider token branch-set page improved from about 7.49M to about 7.13M
  total instructions, with direct key-stream work dropping from about 3.86M to
  about 3.55M instructions.

### 🧪 Testing

- Adds coverage for catalog reuse during immediate SQL global aggregate
  execution.
- Adds diagnostics coverage proving `COUNT(*)`, `COUNT(1)`, and duplicate
  count-equivalent terminals stay off the buffered scalar aggregate path.
- Adds mixed aggregate coverage proving count-equivalent terminals fan out
  around a remaining buffered aggregate terminal.
- Adds lowering and identity coverage proving `COUNT(NULL)` and
  `COUNT(DISTINCT 1)` do not collapse into `COUNT(*)`.
- Adds accepted-query-catalog counter coverage proving filterless shared-plan
  misses reuse the warmed catalog snapshot without decoding raw schema bytes.
- Adds key-stream coverage proving two-stream `merge_all` keeps the pair merge
  route while wider sibling sets use the flat merge route.
- Revalidates focused branch-set fluent page behavior, the fluent perf harness,
  core SQL/diagnostics clippy, focused PocketIC branch-set perf, broad PocketIC
  fluent perf, formatting, and whitespace checks.

### 🧭 Migration Notes

- No schema migration is required.
- No cursor, persistence, public SQL syntax, or query result semantics change.
- Fluent diagnostics/perf attribution payloads include additional compile-phase
  counters; normal query execution and result payloads are unchanged.
- The behavioral result is unchanged; the release only removes duplicated
  compile/catalog work, count-terminal work for equivalent SQL aggregate
  shapes, and nested merge overhead for wider ordered branch streams.

## 0.183.16

### 📝 Summary

- Trims large SQL membership predicate work in the index-only execution path.
- Keeps small encoded `IN` / `NOT IN` lists on the existing linear evaluation
  path while sorting and deduplicating larger membership sets once at compile
  time for binary-search evaluation.
- Removes duplicate predicate preparation walks from SQL base-query binding and
  conservative index-predicate subset compilation.
- Adds a production-shaped 40-literal branch fallback perf audit so large
  over-cap membership filters stay measured alongside the admitted branch-set
  route.

### 🔧 Changed

- Compiled index predicates now distinguish small ordered membership payloads
  from larger sorted membership payloads.
- Large compiled membership predicates evaluate encoded index components with
  binary search instead of scanning every literal candidate for each index
  entry.
- SQL SELECT and DELETE base-query binding now rely on the shared base-shape
  canonicalization pass instead of pre-canonicalizing filters before entering
  that helper.
- Conservative index-predicate subset compilation now attempts full child
  compilation once per `AND` child and drops unsupported children, avoiding a
  separate capability walk for the same large literal payload.

### 📊 Audit

  page shape as the admitted branch route: 50 returned rows, zero row-store
  gets, and 256 streamed index entries in the focused fixture.
- Against a forced-linear membership A/B, the sorted membership path saved
  about 100k executor instructions for the 40-literal fallback case.
- The duplicate preparation cleanup produced a smaller compile-side win of
  about 5.5k compile instructions for that same 40-literal scenario, so deeper
  SQL literal payload cost remains a separate follow-up.

### 🧪 Testing

- Adds focused index-predicate tests proving small membership lists remain
  linear while large membership lists are sorted, deduplicated, and evaluated
  with the same strict `IN` semantics.
- Extends the branch-set perf audit with a large over-cap `IN` fallback and
  verifies it returns the same rows as the admitted branch-set route without
  row hydration or full collection traversal.
- Revalidates focused index-predicate tests, SQL membership lowering tests,
  the branch-set perf audit, clippy for the touched core and integration audit
  targets, the index-range invariant guard, formatting, and whitespace.

### 🧭 Migration Notes

- No schema migration is required.
- No cursor or persistence format changes are introduced.
- The membership payload split is an in-memory compiled-predicate execution
  detail and does not change SQL result semantics.
