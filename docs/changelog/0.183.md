# IcyDB 0.183 Changelog

## 0.183.3

### 📝 Summary

- Closes the branch-set cursor continuation slice for fixed-prefix/small-`IN`
  page queries ordered by primary key.
- Keeps the existing scalar cursor format while teaching branch-aware execution
  to resume each fixed branch from the decoded primary-key boundary.
- Bounds resumed `SELECT *` branch pages at the merged page lookahead window
  instead of scanning the full remaining branch suffix.

### 🩹 Fixed

- Branch-aware composite-prefix streams now derive a per-branch raw index
  anchor from the scalar primary-key cursor boundary, so page 2+ does not
  reopen each fixed branch from the prefix start.
- Branch-set page materialization now caps row hydration at the effective
  page lookahead window when the route proves final primary-key order and no
  residual predicate or distinct phase can shrink rows after the scan.
- Branch-set physical streams receive a matching per-branch fetch hint for the
  page lookahead window, preventing resumed pages from walking the full
  remaining suffix before the merge stops.

### 📊 Audit

- Focused diagnostics now assert resumed `SELECT *` branch pages hydrate only
  returned rows plus lookahead.
- The same diagnostics assert resumed branch index traversal reads at most the
  page lookahead per fixed branch and stays below the full remaining suffix.
- The original branch-route proof suite still covers route admission,
  residual predicate stripping, global primary-key merge ordering, duplicate
  suppression, projection behavior, plan metrics, and cache identity.

### 🧪 Testing

- Revalidated the focused resumed branch-page audit under SQL diagnostics.
- Revalidated the full branch-set SQL diagnostics suite and the non-diagnostics
  SQL branch-set suite.
- Revalidated formatting, whitespace, and SQL diagnostics clippy.

### 🧭 Migration Notes

- No schema migration is required.
- No cursor wire-format change is required; the continuation token remains the
  existing scalar primary-key cursor.
- Prefix cardinality metadata, broader predicate-aware covering, and cheap
  exact counts remain deferred follow-up work.

## 0.183.2

### 📝 Summary

- Fixes covered materialized fallback paging so residual predicates and
  primary-key reordering happen before `LIMIT`.
- Restores scan-time `LIMIT` pushdown for primary-store key-only covering
  reads ordered by primary key.
- Adds a production-shaped `LIMIT 50` branch-set SQL perf audit for the
  `collection_id = X AND stage IN (...) ORDER BY id ASC` query shape.
- Keeps the branch-aware first-page optimization closed while leaving cursor
  continuation, broader predicate-aware covering, and cheap exact counts as
  dedicated follow-up slices.

### 🩹 Fixed

- Covered materialized fallback routes no longer pre-limit index-backed rows
  before residual predicates and primary-key ordering have been applied.
- Sparse over-cap `IN` fallbacks now have focused regression coverage proving
  they retain the unproven stage predicate, materialize-sort before page
  limiting, and return the same globally ordered rows as the admitted branch
  route.
- Pure covering remains available for strict, index-order predicates while
  primary-key reorder fallbacks fail closed to row-materialized execution until
  the covering lane owns predicate evaluation.
- Primary-store key-only covering routes now identify primary-key scan order as
  page-window safe, so `SELECT id ... ORDER BY id LIMIT 1` does not scan the
  full heap or journaled fixture before applying the limit.

### 🔧 Changed

- Covering projection scan-window and page-window decisions now flow through a
  shared helper used by pure, hybrid, and primary-store covering paths.
- Branch covering reads use an adaptive component chunk size: tiny pages keep
  the two-entry branch-head behavior, while larger two-branch pages can fetch a
  wider bounded chunk to reduce repeated index-store reopen overhead.
- The SQL perf audit fixture now seeds enough interleaved matching and
  non-matching token rows to exercise the target `LIMIT 50` page shape under
  pressure.

### 📊 Audit

- The covered `LIMIT 50` token-shaped branch route keeps `data_store.get = 0`
  and reads 100 streamed index entries in the current audit fixture.
- The non-covered `LIMIT 50` branch route hydrates 50 returned rows and reads
  100 streamed index entries.
- The heap `SELECT id ... ORDER BY id ASC LIMIT 1` total-only guard returned to
  the bounded path, dropping from about 5.1M local instructions to about 430k
  in the focused regression run.
- Against the corrected over-cap fallback comparator, the covered branch route
  measured about 3.6x faster in execute instructions and about 5.7x faster in
  total local instructions in the focused audit run.
- The over-cap fallback comparator remains intentionally more expensive:
  it performs row hydration and materialized ordering, proving the branch-aware
  route is not silently degrading to a sorted broad scan.

### 🧪 Testing

- Adds focused branch-set coverage for over-cap fallback filtering before
  primary-key page limiting.
- Adds production-shaped SQL perf coverage for covered, non-covered, and
  over-cap fallback `LIMIT 50` token page queries.
- Revalidates the broader covering matrix so existing explicit primary-key
  suffix and filtered index-order covering routes stay row-store-free where
  they are already proven safe.

### 🧭 Migration Notes

- No schema migration is required.
- Branch-set cursor continuation is not hard-cut in this patch; first-page
  lazy merge remains optimized, while branch-aware resume cursors need a
  separate cursor-format slice.
- Fully predicate-aware pure/hybrid covering is deferred until compiled index
  predicate execution can be passed into the covering lane.
- Cheap exact counts for branch/prefix filters still need prefix cardinality or
  equivalent count metadata; page queries remain separate from exact counts.

## 0.183.1

### 📝 Summary

- Closes the first branch-aware routing cleanup slice after `0.183.0`.
- Makes the branch-set route contract explicit enough for downstream runtime,
  diagnostics, fingerprints, and cache identity to consume without
  reconstructing branch semantics from loose tuple lengths.
- Moves SQL covering branch reads onto lazy branch-head merging so covered and
  hybrid projections do not materialize one page window per branch before
  ordering.

### 🔧 Changed

- `IndexBranchSet` now carries a planner-owned branch-set spec with branch slot,
  branch prefix length, branch field, branch count, concrete branch prefixes,
  branch value cap, and primary-key ordered suffix proof.
- Runtime branch execution consumes the admitted spec instead of deriving the
  branch slot from `fixed_values.len() + 1`.
- Plan fingerprints, SQL cache identity, and EXPLAIN diagnostics include the
  branch-set ordered suffix so branch-aware ordered routes do not collapse onto
  generic prefix or unordered union identities.
- Covering branch reads use resumable index-component chunks and a lazy
  primary-key merge with duplicate suppression, matching the scalar branch
  stream contract instead of sorting and truncating a combined prefix buffer.
- Branch-head covering streams refill in small two-entry chunks, preserving the
  `limit + 1` bounded traversal contract while avoiding the instruction
  overhead of reopening the index store for every single branch head.

### 📊 Audit

- Focused branch-set attribution now proves covered projections keep
  `data_store.get = 0` and pull only the required merged branch heads.
- Non-covered branch projections keep row hydration bounded to returned rows
  plus lookahead while preserving lazy branch index traversal.
- The token-shaped `LIMIT 3` audit row now reads 4 index entries for both
  covered and non-covered page variants; covered `SELECT id` measures 2.436M
  local instructions and non-covered `SELECT id, title` measures 2.505M,
  effectively matching `0.183.0` while avoiding the earlier 6-entry
  per-branch page window.
- The wasm footprint audit for the current default canister set was refreshed
  under `wasm-release`, `sql-on`; the run is recorded as a new baseline because
  the previous comparable report used a different canister set.

### 🧪 Testing

- Adds branch-set spec invariant coverage for canonical branch values,
  over-cap branch rejection, and primary-key suffix route admission.
- Tightens branch-set SQL attribution tests so regressions back to per-branch
  page materialization fail.
- Validated with focused branch-set planner/executor tests, SQL diagnostics
  clippy, the full `icydb-core` SQL+diagnostics lib matrix, formatting, and
  whitespace checks.

## 0.183.0

### 📝 Summary

- Adds branch-aware composite-prefix SQL routing for page queries shaped like
  `fixed_prefix = X AND branch_field IN (...) ORDER BY primary_key ASC LIMIT N`
  against indexes shaped like `(fixed_prefix, branch_field, primary_key)`.
- Targets generated-list style reads such as
  `collection_id = X AND stage IN ('Draft', 'Review') ORDER BY id ASC LIMIT N`
  on `(collection_id, stage, id)`.
- Keeps this first slice narrow: one fixed equality prefix, one small exact
  branch set, and ascending primary-key suffix ordering.

### 🔧 Changed

- The planner can admit the target query shape as an explicit
  `IndexBranchSet` route instead of falling back to a broad ordered scan,
  unordered multi-lookup, full scan, or materialized sort.
- Branch routes open one index prefix stream per branch and merge branch heads
  by decoded primary key, with defensive duplicate suppression.
- Access-proven predicates are stripped from residual filters, so the fixed
  prefix and branch predicates are not rechecked as row predicates once the
  route proves them.
- Covered/key-only projections can stay index-backed with zero row-store
  hydration, while non-covered projections hydrate only returned rows plus
  page lookahead.
- Default page-shaped SQL execution does not run an exact count path; count
  work remains separate from page work.

### 📊 Audit

- For the focused `LIMIT 3` target fixture, covered `SELECT id` stays at
  `data_store.get = 0` and reads only the bounded branch index window.
- The non-covered `SELECT id, title` fixture dropped from hydrating all 12
  matching branch rows to hydrating the returned page rows plus lookahead.
- The SQL perf audit includes a token-shaped surrogate for the production
  query shape and verifies `IndexBranchSet`, no materialized sort, bounded
  index-entry reads, bounded row hydration, and no grouped/count work on the
  default page path.

### 🧪 Testing

- Adds focused planner coverage for branch-route admission, large-`IN`
  fallback, order preservation, and residual predicate stripping.
- Adds executor coverage for globally primary-key-ordered branch merge output,
  duplicate suppression, `limit + 1` lookahead, covered projection behavior,
  non-covered hydration bounds, diagnostics, and plan/cache identity.
- Restores diagnostics-enabled SQL attribution test coverage needed by the
  branch-set proof suite.

### 🧭 Migration Notes

- No schema migration is required. Existing composite indexes with the target
  shape can be selected by the planner.
- Prefix cardinality metadata, arbitrary large `IN` lists, non-primary ordering
  terms, and cursor-format redesign are deferred.
- The SQL covering branch path is bounded by per-branch page/lookahead work in
  this release; making it share the scalar lazy ordered stream abstraction is
  the next cleanup target.
