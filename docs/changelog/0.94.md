# 0.94 Changelog

## 0.94.1

- `0.94.1` stays on the bounded `FILTER` line and closes the biggest remaining contract gaps after `0.94.0` instead of widening the feature surface again.
- the main semantic follow-through in this patch is grouped composition proof:
  - grouped aggregate `FILTER` is now explicitly covered when mixed with unfiltered aggregates in the same projection
  - grouped aggregate `FILTER` is now explicitly covered when the same filtered aggregate meaning is consumed by grouped `HAVING`
  - grouped aggregate `FILTER` is now explicitly covered when grouped `ORDER BY` sorts on that filtered aggregate result
  - this matters because it proves the planner/runtime model is not just â€śfiltered aggregates work in isolationâ€ť; it proves filtered aggregates stay coherent when multiple grouped consumers read the same aggregate meaning
- the grouped semantic matrix now has a direct mixed case:
  - `COUNT(*) FILTER (WHERE age >= 20)`
  - `COUNT(*)`
  - `SUM(age) FILTER (WHERE age >= 20)`
  - grouped by `age`
  - filtered again by `HAVING COUNT(*) FILTER (WHERE age >= 20) > 0`
  - ordered by that same filtered aggregate
  - this locks in that filtered and unfiltered aggregate terminals stay distinct while grouped `HAVING` and grouped `ORDER BY` consume the same filtered aggregate semantics
- the global aggregate side now has matching mixed-projection proof instead of only scalar one-terminal checks:
  - one reduced-row statement now asserts filtered and unfiltered aggregates can share the same global projection payload without losing outward labels, row shape, or reduced values
  - that closes the remaining â€śmixed projectionâ€ť matrix edge from the design doc without opening any new feature family
- observability follow-through is stronger in `0.94.1` too:
  - grouped `EXPLAIN` and `EXPLAIN JSON` now have direct test coverage proving filtered aggregate shape remains visible on the public explain surface
  - global `EXPLAIN EXECUTION` now annotates aggregate terminal descriptors with `filter_expr`, so filtered global aggregates are no longer observability-blind on the execution explain path
  - a focused session-surface test now locks that public execution explain contract in place
- the lower-level planner/observability seam is tighter too:
  - grouped logical explain now has a stable filtered-aggregate snapshot at the planner-owned explain boundary
  - grouped continuation signatures now have a stable filtered-aggregate snapshot too
  - global continuation signatures now have a matching filtered-aggregate snapshot
  - low-level aggregate execution explain now has a direct filtered-terminal proof too
  - that means filtered aggregate shape is no longer only â€ścovered indirectly through session SQLâ€ť; it is frozen at the lower explain/fingerprint layers that SQL ultimately depends on
- this patch does not widen the SQL feature boundary:
  - `DISTINCT + FILTER` is still out
  - aliases inside `FILTER` are still out
  - window `OVER (...)` remains out
  - the work here is contract completion and observability follow-through, not another syntax expansion
- focused verification for `0.94.1` stayed on the new contract edges:

  - `cargo clippy -p icydb-core --all-targets -- -D warnings`

## 0.94.0

- `0.94.0` starts the aggregate `FILTER` line as a bounded slice: it adds parser, lowering, planner, execution, explain, and cache/fingerprint support for aggregate-owned `FILTER (WHERE ...)` without opening a new execution phase or introducing a second boolean boundary.
- the parser now admits aggregate `FILTER` clauses directly on aggregate calls instead of rejecting the syntax up front:
  - `SqlAggregateCall` now carries `filter_expr: Option<Box<SqlExpr>>`
  - aggregate calls like `COUNT(*) FILTER (WHERE age > 1)` stay parser-owned until normal lowering converts that SQL syntax into planner-owned semantics
  - `OVER (...)` remains out of scope and still fails closed
- the lowering boundary now makes the `0.94.0` contract explicit instead of leaving it to executor heuristics:
  - aggregate filters lower onto planner-owned `Expr`
  - lowering reuses the same WHERE-style boolean expression boundary via `lower_sql_where_bool_expr(...)`
  - aggregate-local boolean admission is therefore not a new subsystem; it is the same TRUE-only boundary already used elsewhere
  - `DISTINCT + FILTER` is explicitly rejected in this slice
  - aggregate expressions inside the `FILTER` predicate are also rejected
- aggregate semantic identity now includes `filter_expr` across both global and grouped planning surfaces:
  - `AggregateExpr` carries `filter_expr`
  - grouped planner handoff keeps `filter_expr` on `GroupAggregateSpec` and `GroupedAggregateExecutionSpec`
  - SQL-side global aggregate preparation carries `filter_expr` through `SqlGlobalAggregateTerminal` and `PreparedSqlScalarAggregateStrategy`
- cache identity, fingerprinting, and explain follow that widened aggregate shape instead of silently collapsing filtered and unfiltered aggregates:
  - SQL/global and grouped aggregate cache keys now include filter shape
  - aggregate fingerprints include filter presence and filter expression labels
  - grouped explain output includes aggregate filter identity where applicable
- the real runtime work in this patch is not just parser or lowering support:
  - global aggregate execution now evaluates aggregate filters during SQL aggregate execution instead of treating them as dead lowered metadata
  - grouped aggregate execution now compiles filter expressions into the grouped aggregate runtime state and admits rows only when the compiled filter evaluates to `Value::Bool(true)`
  - `FALSE` and `NULL` both skip the row for that aggregate
  - non-boolean filter results fail closed as invalid logical-plan drift
- that means supported aggregate `FILTER` now has real semantics on both execution families:
  - global examples like `COUNT(*) FILTER (WHERE age >= 30)` and `SUM(age) FILTER (WHERE age >= 30)` now execute through the dedicated global aggregate lane
  - grouped examples like `COUNT(*) FILTER (WHERE age >= 20)` and `SUM(age) FILTER (WHERE age >= 20)` now execute per group while preserving groups whose filtered aggregates admit zero rows
- the grouped runtime slice stays bounded:
  - no new grouped execution phase
  - no rewrite to `CASE`
  - no special aggregate-local predicate IR
  - grouped aggregate filters are just compiled scalar projection expressions evaluated at the grouped terminal boundary
- fail-closed boundary proof was added alongside the positive semantics:
  - lowering tests prove `DISTINCT + FILTER` is rejected in `0.94.0`
  - lowering tests prove aggregate predicates inside `FILTER` are rejected
  - session-level SQL tests prove `DISTINCT + FILTER` stays rejected through the real user surface
  - alias references inside `FILTER` also stay rejected at the SQL surface and currently fail with `unknown expression field '...'`
- semantic and identity proof now covers both positive and negative paths:
  - parser acceptance test for aggregate `FILTER`
  - global aggregate result tests for filtered `COUNT`, `SUM`, and `AVG`
  - grouped aggregate result tests for filtered grouped aggregates
  - cache-identity tests proving nearby filtered aggregate shapes do not alias:
    - filtered vs unfiltered
    - threshold changes
    - grouped operator changes
- focused verification for this slice stayed on the new feature boundary rather than rerunning unrelated lines:
  - `cargo fmt --all`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`
  - `cargo test -p icydb-core parse_sql_accepts_aggregate_filter_clauses -- --nocapture`
  - `cargo test -p icydb-core lower_aggregate_call_attaches_filter_expr_to_aggregate_identity -- --nocapture`
  - `cargo test -p icydb-core lower_aggregate_call_rejects_distinct_filter_pairing_in_0940 -- --nocapture`
  - `cargo test -p icydb-core lower_aggregate_call_rejects_aggregate_predicates_inside_filter -- --nocapture`
  - `cargo test -p icydb-core global_aggregate_filter_value_matrix_matches_expected_values -- --nocapture`
  - `cargo test -p icydb-core grouped_select_helper_filter_aggregate_matrix_matches_expected_grouped_rows -- --nocapture`
  - `cargo test -p icydb-core aggregate_filter_semantic_differences_do_not_alias_sql_cache_identity -- --nocapture`
