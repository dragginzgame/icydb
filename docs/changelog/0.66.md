# 0.66 Changelog

## 0.66.6 - 2026-03-31 - Release Wasm Audit Refresh And Query-Lane Trimming

### Summary

`0.66.6` does not change the shipped SQL, planner, or storage feature surface.
It is a release-size and audit refresh patch over the existing `0.66` line that
keeps the generated SQL query export narrower, trims a few more retained
rendering and dispatch paths from the release canisters, and records a fresh
recurring `wasm-footprint` audit for the standard `minimal`, `one_simple`,
`one_complex`, `ten_simple`, and `ten_complex` SQL audit fixtures.

The main implementation theme in this patch is not new behavior. It is keeping
the generated canister query surface honest about what it actually supports:
`SELECT`, `EXPLAIN SELECT`, `DESCRIBE`, `SHOW INDEXES`, `SHOW COLUMNS`, and
`SHOW ENTITIES`, without retaining as much generic lowerer and dispatch
machinery from the broader typed SQL lane.

### Changed

- tightened the generated query-surface lowerer boundary in:
  - `crates/icydb-core/src/db/sql/lowering/mod.rs`
  - the generated query path now prepares into one dedicated
    `PreparedSqlQuerySurfaceStatement` shape instead of reusing the generic
    `SqlStatement` envelope through the full statement-preparation contract
  - the query-only explain slice now carries only `SqlExplainMode +
    SqlSelectStatement`, so the generated query surface does not retain as much
    delete-capable explain wrapper flow
- tightened the generated query-surface execution boundary in:
  - `crates/icydb-core/src/db/session/sql/dispatch/lowered.rs`
  - `crates/icydb/src/db/sql/mod.rs`
  - the generated query export now asks the core session for projection rows
    directly through one `SELECT`-only helper instead of routing back through
    the broader shared SQL dispatch result wrapper for this narrow path
- kept the generated metadata query lane lightweight in:
  - `crates/icydb/src/db/sql/mod.rs`
  - `SHOW ENTITIES` still resolves directly from the generated authority table
  - supported-entity error rendering still builds its CSV list directly instead
    of retaining an extra `Vec<String> + join(", ")` path for one error surface
- simplified retained query-surface value rendering in:
  - `crates/icydb/src/db/sql/mod.rs`
  - blob, duration, list, and map rendering now build strings directly instead
    of leaning on `format!` and `join()` helper chains for these hot retained
    query-result shapes

## 0.66.1 - 2026-03-29 - Text Semantics Session Lane

### Summary

`0.66.1` turns the previously staged text-semantics work into one
coherent reduced-SQL session lane. It adds bounded computed text projection
support on the unified SQL dispatch surface, extends `EXPLAIN` for that same
lane by rewriting back onto the base field-only `SELECT`, and keeps
`query_from_sql(...)` intentionally structural-only so the typed lowered-query
contract does not reopen generic SQL expression ownership.

This patch also finishes the owner-local SQL session cleanup needed to make
that work land cleanly. The old monolithic `db::session::sql` root is now a
directory module whose branch-heavy responsibilities are explicitly partitioned
by route metadata, projection rendering, explain rendering, aggregate
execution, dispatch orchestration, lowered execution, and computed text
projection evaluation.

It also fixes one real aggregate regression in the existing `count()` fast
path. Full-scan PK-cardinality counts on shared stores were reading the whole
data-store length instead of the current entity-tag range, so one entity's
`count()` could include sibling rows stored under the same backing datastore.

### Changed

- added bounded computed text projection support on the unified SQL dispatch
  lane in:
  - `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  - `crates/icydb-core/src/db/session/sql/dispatch/computed.rs`
  - supported transforms are:
    `TRIM`, `LTRIM`, `RTRIM`, `LOWER`, `UPPER`, `LENGTH`, `LEFT`, `RIGHT`,
    `STARTS_WITH`, `ENDS_WITH`, `CONTAINS`, `POSITION`, `REPLACE`, and
    `SUBSTRING`
  - execution still works by rewriting to a field-only base `SELECT`, loading
    through the existing structural lane, and applying the transform bundle at
    the session boundary instead of reopening generic lowered SQL expressions
- extended `EXPLAIN` support for the computed text projection lane in:
  - `crates/icydb-core/src/db/session/sql/dispatch/computed.rs`
  - `EXPLAIN SELECT TRIM(field) ...` and the other supported text transforms
    now reuse the existing shared explain renderer by rebuilding the base
    field-only `EXPLAIN SELECT ...` statement
- kept `query_from_sql(...)` fail-closed for computed text projection in:
  - `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  - the rejection is now explicit and actionable:
    `query_from_sql does not accept computed text projection; use execute_sql_dispatch(...)`
- decomposed the SQL session owner boundary into narrower directory modules in:

## 0.66.0 - 2026-03-28 - Complexity Hotspot Decomposition

### Summary

`0.66.0` is an internal cleanup release driven by the March 28 DRY and
complexity-accretion audits. It does not change persisted-row encoding, SQL
lowering, route policy, execution semantics, or the public query/session API.

The focus of this release is structural: the largest branch-heavy owner modules
identified by the audit are no longer single growing roots. Instead, the
accumulated logic is split into owner-local directory modules so future work
lands against narrower boundaries with less local accretion pressure.

The main closed audit recommendations in this line are:

- `db::data::persisted_row`: split scalar envelope and `PersistedScalar`
  machinery into a dedicated codec child module
- `db::query::plan::access_choice`: split reason/scoring types from the
  branch-heavy evaluator logic
- `db::executor::explain::descriptor`: split load assembly, aggregate assembly,
  and shared descriptor helpers into owner-local children instead of one large
  mixed root
- route snapshot/capability boilerplate compression across planner and hint
  entrypoints identified by the DRY audit

### Changed

- split the persisted-row owner boundary in:
  - `crates/icydb-core/src/db/data/persisted_row/mod.rs`
  - `crates/icydb-core/src/db/data/persisted_row/codec.rs`
  - scalar envelope encode/decode, slot payload helpers, and `PersistedScalar`
    implementations now live in `codec.rs`
  - dense row patch/rebuild and structural row orchestration remain in the root
    owner module
- split the access-choice planner hotspot in:
  - `crates/icydb-core/src/db/query/plan/access_choice/mod.rs`
  - `crates/icydb-core/src/db/query/plan/access_choice/model.rs`
  - `crates/icydb-core/src/db/query/plan/access_choice/evaluator.rs`
  - reason taxonomy and scoring types moved into `model.rs`
  - prefix/range/multi-lookup candidate evaluation moved into `evaluator.rs`
- split the execution-descriptor explain hotspot in:
  - `crates/icydb-core/src/db/executor/explain/descriptor/mod.rs`
  - `crates/icydb-core/src/db/executor/explain/descriptor/load.rs`
  - `crates/icydb-core/src/db/executor/explain/descriptor/aggregate.rs`
  - `crates/icydb-core/src/db/executor/explain/descriptor/shared.rs`
  - load-node assembly, aggregate-terminal assembly, and shared child
    descriptor helpers are now isolated behind the same owner boundary instead
    of one monolithic file
- compressed the route snapshot/capability boilerplate in:
  - `crates/icydb-core/src/db/executor/route/planner/entrypoints.rs`
  - `crates/icydb-core/src/db/executor/route/hints/load.rs`
  - `crates/icydb-core/src/db/executor/route/hints/aggregate.rs`
  - this only removes same-owner duplication in initial continuation snapshot
    unpacking and access-window plumbing; route policy does not change
- updated the route source audit guard in:
  - `crates/icydb-core/src/db/executor/route/tests.rs`
  - the guard now follows the directory-module path
    `src/db/executor/explain/descriptor/mod.rs` after the descriptor split
- recorded the supporting audit work in:
  - `docs/audits/reports/2026-03/2026-03-28/dry-consolidation.md`
  - `docs/audits/reports/2026-03/2026-03-28/complexity-accretion.md`
