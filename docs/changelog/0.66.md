# 0.66 Changelog

## 0.66.3 - 2026-03-30 - Documentation Fixes And Cleanup

### Summary

`0.66.3` does not expand the shipped SQL or storage surface. It is a cleanup
patch over the `0.66` line that aligns the workspace with the newer Canic
facade, reduces one real parser hotspot that showed up in the recurring
complexity-accretion run, and refreshes the supporting documentation and UI
test fixtures so the release notes and diagnostics match the current tree.

The largest structural change in this patch is inside the reduced SQL parser.
Projection, aggregate-call, and text-function parsing are no longer embedded in
the statement parser root. That branch-heavy cluster now lives under a
dedicated parser child module, which lowers the parser root's local branch
pressure and keeps further SQL-surface growth owner-local instead of
reaccumulating inside `db::sql::parser::mod`.

### Changed

- removed the remaining direct `canic-cdk` dependency inheritance from the
  workspace crates in:
  - `crates/icydb/Cargo.toml`
  - `crates/icydb-core/Cargo.toml`
  - `crates/icydb-schema/Cargo.toml`
- moved direct runtime imports from `canic_cdk::...` onto `canic::cdk::...`
  across the active workspace crates, including:
  - `crates/icydb/src/lib.rs`
  - `crates/icydb/src/db/session/mod.rs`
  - `crates/icydb-build/src/metrics.rs`
  - `crates/icydb-schema/src/node/schema.rs`
  - `crates/icydb-core/src/testing/mod.rs`
  - `crates/icydb-core/src/traits/mod.rs`
  - `crates/icydb-core/src/types/*.rs`
  - `crates/icydb-core/src/db/{data,commit,index,...}`
  - `crates/icydb-core/src/metrics/*`
- updated the hidden generated-code reexport boundary in:
  - `crates/icydb/src/lib.rs`
  - generated code now routes through `::icydb::__reexports::canic::cdk::...`
    instead of `::icydb::__reexports::canic_cdk::...`
- split the reduced SQL parser projection hotspot into a directory child in:
  - `crates/icydb-core/src/db/sql/parser/mod.rs`
  - `crates/icydb-core/src/db/sql/parser/projection.rs`

- replaced the long text-function identifier ladder in:
  - `crates/icydb-core/src/db/sql/parser/mod.rs`
  - `SqlTextFunction::from_identifier(...)` now uses one compact table-driven
    lookup instead of one serial `if` chain per supported function name

## 0.66.2 - 2026-03-29 - Canic 0.18.6 Runtime Bootstrap Alignment

### Summary

`0.66.2` keeps the `0.66.1` reduced-SQL feature surface intact but updates the
canister/runtime integration line to Canic `0.18.6` and aligns generated IcyDB
bootstrap code with Canic's public runtime API.

The generated canister `db()` bootstrap path no longer assembles low-level
`canic-memory` startup manually. Instead it delegates stable-memory registry
initialization to Canic's single public synchronous bootstrap entrypoint,
keeping IcyDB dependent only on the supported facade instead of internal eager
TLS and registry wiring details.

This patch also finishes the corresponding canister-side cleanup: all shipped
SQL harness canisters now use `canic::export_candid` so debug builds pick up
Canic's own assertions, fixture canister memory ranges are rebased above
Canic's reserved range, and the quickstart unit harness documents and works
around Canic's process-global eager-init queue behavior during host-parallel
tests.

### Changed

- updated the workspace Canic line to `0.18.6` in:
  - `Cargo.toml`
  - `Cargo.lock`
- moved generated canister memory bootstrap onto Canic's public runtime API in:
  - `crates/icydb-build/src/db.rs`
  - generated bootstrap now calls
    `::canic::api::runtime::MemoryRuntimeApi::bootstrap_registry()`
  - removed generated low-level bootstrap sequencing around eager TLS,
    registered eager-init flushing, direct registry initialization, and explicit
    one-off local bootstrap state
- switched debug-only Candid export wiring to Canic's facade macro in:
  - `canisters/minimal/src/lib.rs`
  - `canisters/one_simple/src/lib.rs`
  - `canisters/one_complex/src/lib.rs`
  - `canisters/quickstart/src/lib.rs`
  - `canisters/ten_simple/src/lib.rs`
  - `canisters/ten_complex/src/lib.rs`
  - these canisters now import `canic::export_candid` instead of
    `ic_cdk::export_candid`

### Fixed

- rebased fixture canister memory ownership above Canic core's reserved range
  in:
  - `schema/minimal/src/minimal.rs`
  - `schema/one_simple/src/one_simple.rs`
  - `schema/one_complex/src/one_complex.rs`
  - `schema/quickstart/src/schema/relations.rs`
  - `schema/ten_simple/src/ten_simple.rs`
  - `schema/ten_complex/src/ten_complex.rs`
  - this prevents fixture commit/data/index memory IDs from colliding with
    Canic-owned reserved slots during bootstrap validation
- stabilized quickstart host-unit bootstrap under Canic's global eager-init
  queue in:
  - `canisters/quickstart/src/lib.rs`
  - bootstrap-dependent SQL unit helpers now explicitly re-queue the quickstart
    application range on the current host test thread before calling `db()`
  - the workaround is documented as quickstart-specific test harness behavior,
    not a generic SQL test pattern

### Validation

- `cargo check --workspace`
- `cargo test -p canister_quickstart --features sql -- --nocapture`
- `cargo test -p icydb-testing-integration --test sql_canister -- --nocapture`

## 0.66.1 - 2026-03-29 - Text Semantics Session Lane

### Summary

`0.66.1` turns the previously staged text-semantics work into one
coherent reduced-SQL session lane. It adds bounded computed text projection
support on the unified SQL dispatch surface, extends `EXPLAIN` for that same
lane by rewriting back onto the base field-only `SELECT`, and keeps
`query_from_sql(...)` intentionally structural-only so the typed lowered-query
contract does not reopen generic SQL expression ownership.

This patch also finishes the owner-local SQL session cleanup needed to make
that work land cleanly. The old monolithic `db::session::sql` root is now a
directory module whose branch-heavy responsibilities are explicitly partitioned
by route metadata, projection rendering, explain rendering, aggregate
execution, dispatch orchestration, lowered execution, and computed text
projection evaluation.

It also fixes one real aggregate regression in the existing `count()` fast
path. Full-scan PK-cardinality counts on shared stores were reading the whole
data-store length instead of the current entity-tag range, so one entity's
`count()` could include sibling rows stored under the same backing datastore.

### Changed

- added bounded computed text projection support on the unified SQL dispatch
  lane in:
  - `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  - `crates/icydb-core/src/db/session/sql/dispatch/computed.rs`
  - supported transforms are:
    `TRIM`, `LTRIM`, `RTRIM`, `LOWER`, `UPPER`, `LENGTH`, `LEFT`, `RIGHT`,
    `STARTS_WITH`, `ENDS_WITH`, `CONTAINS`, `POSITION`, `REPLACE`, and
    `SUBSTRING`
  - execution still works by rewriting to a field-only base `SELECT`, loading
    through the existing structural lane, and applying the transform bundle at
    the session boundary instead of reopening generic lowered SQL expressions
- extended `EXPLAIN` support for the computed text projection lane in:
  - `crates/icydb-core/src/db/session/sql/dispatch/computed.rs`
  - `EXPLAIN SELECT TRIM(field) ...` and the other supported text transforms
    now reuse the existing shared explain renderer by rebuilding the base
    field-only `EXPLAIN SELECT ...` statement
- kept `query_from_sql(...)` fail-closed for computed text projection in:
  - `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  - the rejection is now explicit and actionable:
    `query_from_sql does not accept computed text projection; use execute_sql_dispatch(...)`
- decomposed the SQL session owner boundary into narrower directory modules in:
  - `crates/icydb-core/src/db/session/sql/mod.rs`
  - `crates/icydb-core/src/db/session/sql/aggregate.rs`
  - `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  - `crates/icydb-core/src/db/session/sql/dispatch/computed.rs`
  - `crates/icydb-core/src/db/session/sql/dispatch/lowered.rs`
  - `crates/icydb-core/src/db/session/sql/surface/mod.rs`

## 0.66.0 - 2026-03-28 - Complexity Hotspot Decomposition

### Summary

`0.66.0` is an internal cleanup release driven by the March 28 DRY and
complexity-accretion audits. It does not change persisted-row encoding, SQL
lowering, route policy, execution semantics, or the public query/session API.

The focus of this release is structural: the largest branch-heavy owner modules
identified by the audit are no longer single growing roots. Instead, the
accumulated logic is split into owner-local directory modules so future work
lands against narrower boundaries with less local accretion pressure.

The main closed audit recommendations in this line are:

- `db::data::persisted_row`: split scalar envelope and `PersistedScalar`
  machinery into a dedicated codec child module
- `db::query::plan::access_choice`: split reason/scoring types from the
  branch-heavy evaluator logic
- `db::executor::explain::descriptor`: split load assembly, aggregate assembly,
  and shared descriptor helpers into owner-local children instead of one large
  mixed root
- route snapshot/capability boilerplate compression across planner and hint
  entrypoints identified by the DRY audit

### Changed

- split the persisted-row owner boundary in:
  - `crates/icydb-core/src/db/data/persisted_row/mod.rs`
  - `crates/icydb-core/src/db/data/persisted_row/codec.rs`
  - scalar envelope encode/decode, slot payload helpers, and `PersistedScalar`
    implementations now live in `codec.rs`
  - dense row patch/rebuild and structural row orchestration remain in the root
    owner module
- split the access-choice planner hotspot in:
  - `crates/icydb-core/src/db/query/plan/access_choice/mod.rs`
  - `crates/icydb-core/src/db/query/plan/access_choice/model.rs`
  - `crates/icydb-core/src/db/query/plan/access_choice/evaluator.rs`
  - reason taxonomy and scoring types moved into `model.rs`
  - prefix/range/multi-lookup candidate evaluation moved into `evaluator.rs`
- split the execution-descriptor explain hotspot in:
  - `crates/icydb-core/src/db/executor/explain/descriptor/mod.rs`
  - `crates/icydb-core/src/db/executor/explain/descriptor/load.rs`
  - `crates/icydb-core/src/db/executor/explain/descriptor/aggregate.rs`
  - `crates/icydb-core/src/db/executor/explain/descriptor/shared.rs`
  - load-node assembly, aggregate-terminal assembly, and shared child
    descriptor helpers are now isolated behind the same owner boundary instead
    of one monolithic file
- compressed the route snapshot/capability boilerplate in:
  - `crates/icydb-core/src/db/executor/route/planner/entrypoints.rs`
  - `crates/icydb-core/src/db/executor/route/hints/load.rs`
  - `crates/icydb-core/src/db/executor/route/hints/aggregate.rs`
  - this only removes same-owner duplication in initial continuation snapshot
    unpacking and access-window plumbing; route policy does not change
- updated the route source audit guard in:
  - `crates/icydb-core/src/db/executor/route/tests.rs`
  - the guard now follows the directory-module path
    `src/db/executor/explain/descriptor/mod.rs` after the descriptor split
- recorded the supporting audit work in:
  - `docs/audits/reports/2026-03/2026-03-28/dry-consolidation.md`
  - `docs/audits/reports/2026-03/2026-03-28/complexity-accretion.md`

### Audit Metrics

- `db::data::persisted_row` root hotspot moved from `1793 LOC / 47 branch sites`
  to `1012 LOC / 32 branch sites`, with the extracted
  `db::data::persisted_row::codec` child carrying the scalar branch cluster
- `db::query::plan::access_choice` root hotspot moved from
  `788 LOC / 66 branch sites` to `86 LOC / 3 branch sites`, with the extracted
  `evaluator` child carrying the decision-heavy branch cluster
- `db::executor::explain::descriptor` root hotspot moved from a single large
  file to:
  - `mod.rs`: `10 LOC / 0 branch sites`
  - `load.rs`: `219 LOC / 8 branch sites`
  - `aggregate.rs`: `67 LOC / 1 branch site`
  - `shared.rs`: `840 LOC / 35 branch sites`

This release deliberately keeps the complexity in the owning subsystem while
stopping further accretion inside one root file. The branch pressure still
exists, but it is now explicitly partitioned by responsibility.
