# IcyDB 0.207 Changelog

## 0.207.0 - 2026-07-21

### 📝 Summary

- Makes user-index schema publication marker-first and redo-only. CREATE INDEX,
  DROP INDEX, startup reconciliation, and interrupted recovery now use one
  bounded complete-domain replacement owned by accepted schema.
- Removes the operation-specific rollback families and duplicate staging paths
  that previously gave CREATE, DROP, and recovery different failure semantics.
- Preserves row-layout and row-commit rollback because those paths protect
  authoritative persisted rows rather than rebuildable derived indexes.

### 🔧 Changed

- Stages the complete accepted-after user-index domain without physical writes,
  after proving the accepted-before projection and current index store agree.
  The stage covers field-path and expression indexes and carries its validated
  cardinality and resource facts into publication.
- Streams authoritative rows directly into the bounded stage. Admission and
  projection limits apply before per-row state is retained, avoiding the former
  raw-row and decoded-row materialisations.
- Publishes CREATE INDEX and DROP INDEX through the existing schema marker,
  rechecks accepted identity immediately before the marker, and performs only
  mechanical application afterward.
- Rebuilds interrupted derived-index state forward from authoritative rows.
  Recovery caches one accepted authority per entity and clears every partial
  rebuilt prefix before returning a failure.
- Keeps failed or interrupted indexes non-ready while marker authority remains
  recoverable. Ready state is published only for the complete accepted-after
  projection.
- Classifies duplicate candidate keys for CREATE UNIQUE INDEX as typed index
  conflicts while retaining corruption classification for duplicates already
  present in accepted-before physical truth.
- Derives dense first, middle, and last DROP INDEX ordinals from the
  accepted-after schema instead of physically remapping later keys and retaining
  reverse-remap state.

### 🩹 Fixed

- Prevents failed unique-index recovery from leaving a partially rebuilt prefix
  visible in a Building index store.
- Prevents complete data-store cloning and repeated accepted-schema resolution
  during forward recovery.
- Applies aggregate staging bounds while rows are streamed instead of after an
  unbounded intermediate collection has already been built.
- Preserves typed unique-candidate conflicts rather than collapsing them into a
  generic persisted-store corruption error.

### 🗑️ Removed

- Deletes CREATE field-path write/rollback plans, expression-index inserted-key
  undo, DROP before-images, physical ordinal remapping, reverse remapping, and
  recovery index-store snapshot restoration.
- Deletes the public `IndexState::Dropping` variant. `Building` is the sole
  non-ready lifecycle state for current derived-index construction and repair.
- Deletes the redundant DROP target ordinal wrapper, discarded target-entry
  count, staged-key metrics decode, and singular schema-publication wrapper.
- Retains no alias, compatibility shim, alternate dispatcher, legacy recovery
  branch, or old persisted representation.

### 🧪 Testing

- Focused coverage exercises field-path and expression CREATE, candidate unique
  rejection, first/middle/last DROP compaction, pre-marker zero-write rejection,
  marker-owned interruption, recovery from partial Building state, startup
  reconciliation, repeated recovery, malformed rows, mixed index shapes, and
  bounded staging at scale.
- Compile-UI coverage, native and Wasm feature checks, strict Clippy, formatting,
  diff hygiene, architecture invariants, feature-matrix checks, documentation,
  and the SQL correctness coverage manifest passed during closeout.
- Full repository and workspace suites remained user-owned push validation under
  the repository agent rules.

### 📊 Audit

- The published release changes 55 files, adds 3,050 lines, and removes 5,771
  lines: a net reduction of 2,721 lines.
- The final SQL performance Wasm is 3,898,854 raw non-gzipped bytes, 7,629 bytes
  smaller (`-0.20%`) than the 0.206.4 baseline. Deterministic gzip is secondary
  evidence at 1,236,711 bytes, 6,334 bytes below baseline.
- The implementation has one schema-owned stage, one commit-owned apply path,
  and one forward recovery direction. It retains no full source-row snapshot,
  derived-index rollback owner, or DROP ordinal transport.
- No dependency, accepted schema, row, index-key, marker, journal, or cursor
  format changed. Query semantics, pagination, and response shapes are also
  unchanged.

### ⚠️ Breaking

- `IndexState::Dropping` is removed as a pre-1.0 hard cut. Code matching index
  lifecycle state must use `Building` as the only current non-ready state; no
  compatibility alias is provided.
