# 0.95 Changelog

## 0.95.0

- `0.95.0` finishes the `HAVING` unification line by deleting the old clause-specific compare-shell path and making planner `Expr` the shared semantic owner for grouped and global aggregate `HAVING`.
- the first part of the cut is at the intent/planner boundary:
  - production `HAVING` no longer enters through `GroupHavingClause` / `GroupHavingSymbol`
  - `having_group(...)` and `having_aggregate(...)` now lower onto the same post-aggregate `Expr` path as the rest of the query model
  - `push_having_expr(...)` is now the production intent entrypoint instead of a wrapper clause shell
- the second part is planner/model cleanup:
  - the old `GroupHavingClause` / `GroupHavingSymbol` model is gone from `crates/icydb-core/src/db`
  - grouped `HAVING` tests, explain fixtures, route fixtures, fingerprint fixtures, and grouped validation helpers now build direct grouped `Expr` instead of carrying a second local `HAVING` AST
  - this removes the last live clause-specific `HAVING` semantic structure from the planning layer
- `0.95.0` also closes the old global aggregate split:
  - zero-key grouped/global aggregate shapes are now admitted on the shared planner path
  - `HAVING` without `GROUP BY` is no longer a SQL-only semantic exception
  - global aggregate `HAVING` now reads as the same post-aggregate boolean model, just with zero group keys, while the stricter global-`DISTINCT` special rules remain bounded to the real field-target `DISTINCT` case
- explain and fingerprinting follow the same ownership cut:
  - explain-side `HAVING` no longer carries `GroupHavingExpr`, `GroupHavingValueExpr`, or case-arm wrappers
  - explain now projects planner `Expr` directly for grouped `HAVING`
  - grouped `HAVING` fingerprinting and shape signatures hash the shared planner `Expr` directly instead of re-owning a second explain/local expression structure
- that means the real `HAVING` ownership model after `0.95.0` is:
  - parser-owned syntax: `SqlExpr`
  - planner-owned semantics: post-aggregate `Expr`
  - grouped/global validation: shared grouped/global aggregate rules
  - boolean admission: the same shared TRUE-only boundary already used elsewhere
- the same patch line also follows through on the broader AST audit that started from the unification work:
  - SQL parser function application is now one generic scalar-expression path instead of several special-case wrappers
  - the old text-function wrapper path is gone
  - aggregate input mini-AST is gone
  - arithmetic projection mini-AST is gone
  - the dedicated `ROUND(...)` AST shell is gone
  - the parser now treats `ROUND(...)` like the rest of the scalar-function family and leaves arity/default-scale handling to lowering
- taxonomy cleanup follows that structural cut:
  - `SqlTextFunction` is now `SqlScalarFunction`
  - parser and lowering helper names now match the real model instead of keeping “text function” naming around a general scalar-function family
  - outward parser diagnostics that used the old “scalar text projection forms” wording were updated to the generic scalar-function wording too
- the important result is architectural, not a new SQL feature family:
  - no clause-specific `HAVING` semantic lane remains
  - no explain-side `HAVING` AST remains
  - global aggregate `HAVING` no longer relies on a semantic side path to be valid
  - the SQL parser no longer keeps the older reduced projection/function wrappers alive beside the generic expression tree
