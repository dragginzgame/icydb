# 0.95 Changelog

## 0.95.2

- `0.95.2` removes one stale grouped Top-K policy gate that was still rejecting `OFFSET` even though the grouped runtime already carried the required bounded window state:
  - grouped planner continuation/window projection already computed `initial_offset_for_page`, `selection_bound`, and `limit + offset + 1` candidate retention for grouped bounded pages
  - grouped Top-K finalization already applied offset after candidate selection and before row shaping
  - the remaining rejection lived only in grouped cursor validation, where aggregate-driven grouped `ORDER BY` still required `offset == 0`
- the actual code change is intentionally narrow:
  - grouped cursor validation no longer rejects `GroupedOrderCursorLane::TopK` when `page.offset > 0`
  - the old `GroupPlanError::OrderOffsetNotSupported` policy variant and constructor were removed because that boundary no longer exists
- user-visible result:
  - grouped aggregate `ORDER BY ... LIMIT ... OFFSET ...` now plans and executes through the normal grouped Top-K path
  - filtered aggregate alias ordering inherits that widening too, so shapes like:

```sql
SELECT
  class_name,
  COUNT(*) FILTER (WHERE is_npc) AS npc_count
FROM Character
GROUP BY class_name
HAVING npc_count > 0
ORDER BY npc_count DESC, class_name ASC
LIMIT 1 OFFSET 1;
```

  now fit the same grouped ordered aggregate surface instead of failing closed at planner policy time
- one important boundary stays the same:
  - grouped Top-K still suppresses continuation cursors
  - this patch widens initial-page `OFFSET` support; it does not change grouped Top-K cursor semantics
- focused verification for `0.95.2` covered planner admission, SQL lowering, and one live filtered-aggregate alias execution shape:
  - `cargo test -p icydb-core grouped_aggregate_order_with_offset_is_allowed_in_planner_cursor_policy -- --nocapture`
  - `cargo test -p icydb-core compile_sql_command_accepts_grouped_filtered_aggregate_order_by_alias_with_offset -- --nocapture`
  - `cargo test -p icydb-core grouped_select_helper_filtered_aggregate_order_alias_supports_offset -- --nocapture`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`

## 0.95.1

- `0.95.1` closes one remaining SQL front-door mismatch after the main `HAVING` unification cut:
  - ordinary session SQL query compilation was still deciding whether a `SELECT` belonged on the dedicated global aggregate lane before prepared-statement normalization
  - that meant alias-shaped queries such as `SELECT COUNT(*) AS total_rows ... HAVING total_rows > 1` were checked too early, missed the aggregate lane, and then fell into the stale non-grouped `HAVING requires GROUP BY` rejection
- the fix is narrow and routing-owned:
  - prepared SQL statements now expose the normalized parsed statement at the compile boundary
  - session SQL aggregate-lane detection now runs on that normalized statement instead of the raw parsed one
  - `EXPLAIN SELECT` applies the same normalized global-aggregate routing rule, so explain and execution no longer disagree on aliased global aggregate `HAVING` shapes
- the test-only reduced SQL compiler follows the same normalized route now:
  - `compile_sql_command(...)` recognizes normalized global aggregate `SELECT` shapes and returns the dedicated aggregate command envelope instead of trying to force them back through the ordinary lowered-query lane
  - this keeps the lowering tests aligned with the real session compile boundary instead of preserving the old stale rejection in test-only code
- user-visible effect:
  - `SELECT COUNT(*) AS total_rows FROM character HAVING total_rows > 1` now compiles and executes through the same dedicated global aggregate path as the equivalent `HAVING COUNT(*) > 1` form
  - plain non-aggregate shapes such as `SELECT * ... HAVING COUNT(*) > 1` still fail closed with `HAVING requires GROUP BY`
- focused verification for `0.95.1` covered the exact repaired surfaces:
  - `cargo test -p icydb-core compile_sql_command_select_global_aggregate_having_alias_lowers_to_global_aggregate_command -- --nocapture`
  - `cargo test -p icydb-core global_aggregate_having_alias_returns_single_row_when_predicate_matches -- --nocapture`
  - `cargo test -p icydb-core sql_surface -- --nocapture`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`

## 0.95.0

- `0.95.0` finishes the `HAVING` unification line by deleting the old clause-specific compare-shell path and making planner `Expr` the shared semantic owner for grouped and global aggregate `HAVING`.
- the first part of the cut is at the intent/planner boundary:
  - production `HAVING` no longer enters through `GroupHavingClause` / `GroupHavingSymbol`
  - `having_group(...)` and `having_aggregate(...)` now lower onto the same post-aggregate `Expr` path as the rest of the query model
  - `push_having_expr(...)` is now the production intent entrypoint instead of a wrapper clause shell
- the second part is planner/model cleanup:
  - the old `GroupHavingClause` / `GroupHavingSymbol` model is gone from `crates/icydb-core/src/db`
  - grouped `HAVING` tests, explain fixtures, route fixtures, fingerprint fixtures, and grouped validation helpers now build direct grouped `Expr` instead of carrying a second local `HAVING` AST
  - this removes the last live clause-specific `HAVING` semantic structure from the planning layer
- `0.95.0` also closes the old global aggregate split:
  - zero-key grouped/global aggregate shapes are now admitted on the shared planner path
  - `HAVING` without `GROUP BY` is no longer a SQL-only semantic exception
  - global aggregate `HAVING` now reads as the same post-aggregate boolean model, just with zero group keys, while the stricter global-`DISTINCT` special rules remain bounded to the real field-target `DISTINCT` case
- explain and fingerprinting follow the same ownership cut:
  - explain-side `HAVING` no longer carries `GroupHavingExpr`, `GroupHavingValueExpr`, or case-arm wrappers
  - explain now projects planner `Expr` directly for grouped `HAVING`
  - grouped `HAVING` fingerprinting and shape signatures hash the shared planner `Expr` directly instead of re-owning a second explain/local expression structure
- that means the real `HAVING` ownership model after `0.95.0` is:
  - parser-owned syntax: `SqlExpr`
  - planner-owned semantics: post-aggregate `Expr`
  - grouped/global validation: shared grouped/global aggregate rules
  - boolean admission: the same shared TRUE-only boundary already used elsewhere
- the same patch line also follows through on the broader AST audit that started from the unification work:
  - SQL parser function application is now one generic scalar-expression path instead of several special-case wrappers
  - the old text-function wrapper path is gone
  - aggregate input mini-AST is gone
  - arithmetic projection mini-AST is gone
  - the dedicated `ROUND(...)` AST shell is gone
  - the parser now treats `ROUND(...)` like the rest of the scalar-function family and leaves arity/default-scale handling to lowering
- taxonomy cleanup follows that structural cut:
  - `SqlTextFunction` is now `SqlScalarFunction`
  - parser and lowering helper names now match the real model instead of keeping “text function” naming around a general scalar-function family
  - outward parser diagnostics that used the old “scalar text projection forms” wording were updated to the generic scalar-function wording too
- the important result is architectural, not a new SQL feature family:
  - no clause-specific `HAVING` semantic lane remains
  - no explain-side `HAVING` AST remains
  - global aggregate `HAVING` no longer relies on a semantic side path to be valid
  - the SQL parser no longer keeps the older reduced projection/function wrappers alive beside the generic expression tree
