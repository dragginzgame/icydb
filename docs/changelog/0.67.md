# 0.67 Changelog

## 0.67.4 - 2026-04-03 - Shared Query Runtime Budget Cuts

### Summary

`0.67.4` keeps the shipped SQL feature surface unchanged and pushes on the
fixed-cost query path instead of widening SQL again. The main goal in this
patch is to stop common generated and typed `SELECT` execution from rebuilding
the same lowered-query, projection, predicate, route-runtime, and row-runtime
setup work on every call when those surfaces already converge on the same
shared scalar execution core.

### Changed

- reduced repeated SQL projection setup in:
  - `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  - `crates/icydb-core/src/db/session/sql/projection/{labels.rs,mod.rs}`
  - typed SQL dispatch no longer binds and discards a full typed `Query<E>` on
    plain `SELECT` before returning to the structural executor path
  - structural SQL projection now derives one projection contract and reuses it
    for both column-label rendering and row materialization
- reduced repeated projection/runtime work in:
  - `crates/icydb-core/src/db/executor/pipeline/entrypoints/scalar/mod.rs`
  - `crates/icydb-core/src/db/executor/kernel/mod.rs`
  - `crates/icydb-core/src/db/executor/terminal/page.rs`
  - `crates/icydb-core/src/db/executor/pipeline/operators/terminal/runtime.rs`
  - the SQL projection lane now skips the duplicate shared projection
    validation pass when it immediately materializes that same projection
  - SQL projection execution now retains decoded slot rows and projects
    directly from those already-decoded values instead of rebuilding
    structural readers from persisted row bytes
- reduced repeated executor preparation and fast-path predicate setup in:
  - `crates/icydb-core/src/db/executor/preparation.rs`
  - `crates/icydb-core/src/db/executor/pipeline/runtime/fast_path/mod.rs`
  - shared scalar load execution now uses a lighter runtime preparation bundle
    that keeps only the compiled row predicate plus slot map instead of also
    building explain/aggregate-only capability state
  - conservative index-predicate programs are now compiled once at the runtime
    preparation boundary and reused during key-stream resolution instead of
    being rebuilt for every query
- reduced repeated scalar execution boundary setup in:
  - `crates/icydb-core/src/db/executor/pipeline/contracts/execution/inputs.rs`
  - `crates/icydb-core/src/db/executor/pipeline/entrypoints/scalar/mod.rs`
  - `crates/icydb-core/src/db/executor/terminal/page.rs`
  - the shared scalar execution stage now borrows the logical plan instead of
    cloning `plan.access` solely to build the runtime adapter
  - row materialization now borrows the adapter-owned structural row-runtime
    state instead of cloning and boxing a fresh row-runtime descriptor before
    every query materialization pass
- refreshed the runtime dependency line in:
  - `Cargo.lock`
  - workspace resolution now lands on `canic-cdk` / `canic-memory` `0.22.6`

## 0.67.3 - 2026-04-03 - Generated Query Route Collapse

### Summary

`0.67.3` keeps the shipped SQL surface unchanged and removes the largest
remaining duplicated SQL-on wasm family on the generated canister query path.
The main goal in this patch is to stop each generated `sql_dispatch::query(...)`
surface from carrying its own copy of authority lookup, query/explain
branching, metadata routing, and generated-surface error shaping when core
already owns the same route family.

### Changed

- reduced duplicated generated query routing in:
  - `crates/icydb-build/src/db.rs`
  - generated `sql_dispatch::query(...)` is now a thin shim that forwards the
    raw SQL string and static authority table into one shared runtime entrypoint
    instead of emitting per-canister routing helpers for `SELECT`, `DELETE`,
    `EXPLAIN`, `DESCRIBE`, `SHOW INDEXES`, `SHOW COLUMNS`, and `SHOW ENTITIES`
  - removed the generated copies of authority lookup, metadata route matching,
    unsupported-entity rendering, and `EXPLAIN` input rewriting helpers from
    the build-emitted SQL layer
- moved the generated query route family under one shared owner in:
  - `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  - core now owns generated-surface authority resolution, supported-entity
    classification, and route branching for the generated canister query lane
    instead of relying on one extra generated routing stack per canister
- kept the facade layer thin in:
  - `crates/icydb/src/db/session/generated.rs`
  - `crates/icydb/src/{lib.rs,db/mod.rs}`
  - `icydb` now only exposes the hidden generated-query bridge plus the final
    public `SqlQueryResult` mapping and the existing unordered-pagination
    `EXPLAIN` rewrite, rather than owning generated-route orchestration itself
- removed a smaller generated-query lowering duplicate in:
  - `crates/icydb-core/src/db/sql/lowering/mod.rs`
  - the generated query surface now reuses the canonical `PreparedSqlStatement`
    and `lower_sql_command_from_prepared_statement(...)` path instead of
    carrying a second prepared-statement enum and lowering function

### Measured

On the `minimal` `wasm-release` `sql-on` audit target, the accepted `0.67.3`
tree moved from:

- built `.wasm`: `1,299,689 -> 779,081` (`-520,608`)
- shrunk `.wasm`: `1,212,099 -> 722,820` (`-489,279`)

Twiggy and size-report follow-up signals on the same accepted tree show:

- retained `export "canister_query query"`: `264,530 -> 22,343`
- function count: `3,227 -> 2,160`
- data section bytes: `171,056 -> 110,752`

After the route collapse, the remaining `minimal` SQL-on premium versus the
matching `sql-off` build is:

- built `.wasm`: `+94,764`
- shrunk `.wasm`: `+85,610`

## 0.67.2 - 2026-04-02 - Batched Write and Save-Path Budget Cuts

### Summary

`0.67.2` keeps the shipped SQL and storage surface unchanged and continues the
`0.67.1` write-path optimization line deeper into save batching and commit
preflight. The main goal in this patch is to stop typed save lanes and batched
insert lanes from rebuilding the same commit metadata, schema fingerprint, and
save-side bookkeeping work on every row.

### Changed

- reduced repeated typed save work in:
  - `crates/icydb-core/src/db/executor/mutation/save.rs`
  - typed full-row save now emits canonical rows directly from the derive-owned
    slot writer instead of routing that same full-row image back through the
    serialized patch bridge first
  - non-atomic batch save reuses one resolved mutation context plus one schema
    / relation-preflight bundle across the whole batch instead of rebuilding
    that outer save setup for every row
  - non-atomic batch save now records one batch-level save span and accumulates
    index delta metrics once per batch, while still flushing committed-prefix
    metrics before fail-fast return
- reduced repeated commit-preflight work in:
  - `crates/icydb-core/src/db/commit/prepare.rs`
  - typed save preflight now resolves the commit schema fingerprint once per
    outer save or batch path instead of re-deriving it per prepared row
  - reverse-index preparation now reuses already-decoded structural slot
    readers and skips reverse-index planning entirely when the model owns no
    strong outgoing relations
  - canonical after-image construction now reuses already-decoded structural
    slots on the accepted fast path instead of re-decoding a second row image
    before materializing the prepared write
- reduced repeated commit-window and relation validation overhead in:
  - `crates/icydb-core/src/db/executor/mutation/commit_window.rs`
  - `crates/icydb-core/src/db/relation/{metadata.rs,reverse_index.rs,validate.rs}`
  - preflight overlay index reads now fall straight through to the native raw
    range reader when no staged overrides exist
  - the staged overlay index-override map now uses the lighter hash-backed hot
    shape instead of the earlier tree-backed lookup path
  - blocked-delete strong-relation checks now test direct target membership
    instead of rebuilding and sorting the full target-key vector just to answer
    one candidate deletion question
  - relation validation now skips irrelevant source models before store
    resolution and hook dispatch, so delete-side strong-reference checks only
    pay for models that can actually point at the target

## 0.67.1 - 2026-04-01 - One-Row Write Path Fixed-Cost Trims

### Summary

`0.67.1` keeps the shipped SQL and storage surface unchanged and trims fixed
cost out of the shared one-row mutation path. The main goal in this patch is
to stop inserts, updates, and deletes from paying generic multi-row commit and
marker overhead when the runtime already knows the operation only touches one
row.

### Changed

- reduced commit-control fixed cost in:
  - `crates/icydb-core/src/db/commit/store.rs`
  - control-slot inspection for marker-presence and begin-commit checks now
    uses a borrowed decoded view instead of materializing owned marker and
    migration buffers on the hot path
  - one-row commit open now has a direct persisted marker write path instead of
    always building a full `CommitMarker { row_ops: Vec<_> }` wrapper first
- reduced one-row commit-window overhead in:
  - `crates/icydb-core/src/db/executor/mutation/commit_window.rs`
  - one-row save/delete preflight now bypasses staged overlay simulation
  - one-row apply now bypasses the rollback-vector path
  - one-row save/delete metrics and index-generation setup now use dedicated
    single-row helpers instead of the batch slice aggregators
- reduced save/update old-row validation overhead in:
  - `crates/icydb-core/src/db/executor/mutation/save.rs`
  - `crates/icydb-core/src/db/executor/mutation/save_validation.rs`
  - persisted existing-row validation now stays on the structural row path and
    uses the canonical slot-reader seam instead of rebuilding a typed entity
  - the runtime field-projection boundary remains intact: the structural check
    now uses `has(...)` plus `required_value_by_contract(...)` instead of
    direct `.get_value(...)` calls in the runtime validation file
- narrowed guard ownership in:
  - `crates/icydb-core/src/db/commit/guard.rs`
  - mutation-path `CommitGuard` now only retains the commit id on the hot save
    and delete lanes
  - migration keeps full marker row ops only where replay actually needs them
    through `crates/icydb-core/src/db/migration/mod.rs`

### Measured

## 0.67.0 - 2026-04-01 - Bounded Direct STARTS_WITH Surface Coherence

### Summary

`0.67.0` opens the line with one bounded SQL coherence slice instead of a
broader lane rewrite. The reduced SQL surface now accepts direct
`STARTS_WITH(...)` predicate spellings for:

- plain fields
- `LOWER(field)`
- `UPPER(field)`

Those forms lower onto the same existing strict or casefold
`CompareOp::StartsWith` planner-bearing intent already used by the bounded
`LIKE '<prefix>%'` family. Broader text-function predicate forms remain
fail-closed.

### Changed

- widened the reduced SQL direct predicate spelling in:
  - `crates/icydb-core/src/db/predicate/parser/mod.rs`
  - `STARTS_WITH(field, '<prefix>')` now lowers directly to the existing strict
    starts-with compare intent
  - `STARTS_WITH(LOWER(field), '<prefix>')` and
    `STARTS_WITH(UPPER(field), '<prefix>')` now lower to the same bounded
    `TextCasefold` starts-with intent already used by `LOWER/UPPER(field) LIKE
    '<prefix>%'`
- kept the direct spelling boundary narrow in:
  - `crates/icydb-core/src/db/predicate/parser/mod.rs`
  - non-casefold wrappers such as `STARTS_WITH(TRIM(field), '<prefix>')`
    remain unsupported
  - the stable unsupported-feature label for that boundary is now
    `STARTS_WITH first argument forms beyond plain or LOWER/UPPER field wrappers`
- reduced duplicate predicate-lowering logic in:
  - `crates/icydb-core/src/db/predicate/parser/mod.rs`
  - bounded `LIKE '<prefix>%'` lowering and direct `STARTS_WITH(...)` lowering
    now share one owner-local operand-to-coercion path instead of each
    re-deriving strict vs casefold mapping separately

### Fixed

- locked the widened bounded family across the main public SQL surfaces in:
  - `crates/icydb-core/src/db/sql/parser/tests.rs`
  - `crates/icydb-core/src/db/sql/lowering/tests.rs`
  - `crates/icydb-core/src/db/session/tests.rs`
  - `crates/icydb/src/db/session/mod.rs`
  - parser, lowering, typed session, and facade coverage now all prove the
    same strict or casefold `StartsWith` intent for the accepted direct forms
- locked generated canister SQL parity in:
  - `canisters/quickstart/src/lib.rs`
  - generated `sql_dispatch::query(...)` now has parity coverage for:
    - direct `STARTS_WITH(field, '<prefix>')`
    - direct `STARTS_WITH(LOWER(field), '<prefix>')`
    - direct `STARTS_WITH(UPPER(field), '<prefix>')`
    - matching `EXPLAIN` forms
  - generated canister SQL also has explicit fail-closed parity coverage for
    unsupported wrappers such as `STARTS_WITH(TRIM(field), '<prefix>')`
- locked PocketIC canister-surface behavior in:
  - `testing/pocket-ic/tests/sql_canister.rs`
  - direct lower/upper bounded `STARTS_WITH(...)` query paths are now covered
    at the canister query boundary
  - unsupported non-casefold wrappers are also checked there so the stable
    runtime `Unsupported` classification survives the generated surface
- refreshed the live contract/design baseline in:
  - `docs/contracts/SQL_SUBSET.md`
  - `docs/design/0.67-feature-coherence/0.67-design.md`
  - `docs/design/0.67-feature-coherence/0.67-surface-matrix.md`
  - `docs/design/0.67-feature-coherence/0.67-evaluation-packets.md`
  - those docs now describe the shipped bounded direct `STARTS_WITH(...)`
    family accurately instead of the older plain-field-only assumption
