# 0.116 Changelog

## 0.116.2

- `0.116.2` keeps the same truth-condition centralization line on the grouped side, but this patch is an owner-consolidation step rather than a behavior widening step. The shared grouped null-aware compare expression builder now lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_having.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_having.rs) as `grouped_having_compare_expr(...)`, with planner semantics owning the one place that decides whether grouped `HAVING <expr> = NULL` becomes `IS NULL`, whether `!= NULL` becomes `IS NOT NULL`, and when the same grouped compare family should remain a binary compare.

- Before this patch, that exact grouped compare/null-aware shaping rule was already logically stable, but it was not structurally centralized. [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs) had its own helper for the grouped compare/null-aware expression surface, and planner-owned test modules were also rebuilding the same `NULL -> IsNull/IsNotNull` versus `non-NULL -> BinaryOp` split locally. That left one grouped truth-condition shape with multiple implementations even though `0.116` is explicitly trying to collapse those adjacent ladders.

- After `0.116.2`, intent and planner-owned tests all read from the same planner-owned grouped expression builder:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs) now imports `grouped_having_compare_expr(...)` from the planner boundary instead of owning a local version
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/tests/mod.rs), [/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/shape_signature/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/shape_signature/tests/mod.rs), and [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/planning/route/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/planning/route/tests/mod.rs) no longer re-derive that grouped null-aware compare shape locally
  - planner-owned grouped plan tests in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/tests/group.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/tests/group.rs) and grouped validation tests in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/tests/grouped.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/tests/grouped.rs) now also route their aggregate compare fixtures through the same helper

- That matters because the grouped truth-condition line now has one cleaner split:
  - planner semantics own grouped compare truth support, grouped compare evaluation, and grouped compare/null-aware expression shaping
  - intent consumes that planner-owned shape
  - explain, fingerprint, routing, and planner-owned grouped tests pin the same shape instead of reproducing it

- The current `0.116.2` follow-through also closes the remaining compare-family bridge splits around that grouped owner move:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_having.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_having.rs) now owns `grouped_having_binary_compare_op(...)`, so grouped validation and grouped streaming checks no longer re-derive `BinaryOp -> CompareOp` locally
  - the same grouped semantics file now also lowers grouped compare expressions through the shared planner compare-family bridge instead of keeping a separate `CompareOp -> BinaryOp` ladder beside the grouped null-aware builder
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs) now owns the generic planner truth-condition compare-family bridge through `truth_condition_compare_binary_op(...)` and `truth_condition_binary_compare_op(...)`
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs) now consumes that planner-owned bridge instead of keeping its own private `CompareOp <-> BinaryOp` ladders during predicate-to-bool lowering, compile-ready bool compare gating, and bool compare compilation

- That extra contraction matters because the truth-condition line is no longer just “grouped compare logic moved to planner semantics.” The scalar predicate bridge and the grouped planner bridge now also share the same planner-owned compare-family mapping, so both sides consume one compare operator seam instead of carrying nearby duplicate ladders.

- This is intentionally not a semantic widening. `0.116.2` does not admit a new grouped `HAVING` family and does not change the grouped compare operators that the planner accepts. It just removes the remaining duplicated grouped compare/null-aware expression ladders so the grouped truth-condition surface is expressed in one owner.

## 0.116.1

- `0.116.1` extends the same truth-condition owner move into compare and null-test admission. [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs) now owns `scalar_where_truth_condition_is_admitted(...)`, which becomes the single planner-owned admission rule for scalar `WHERE` truth semantics instead of leaving compare/null-test admission split between planner wrapper collapse and lowering-local validation.

- That owner move matters because scalar `WHERE` was still carrying two adjacent meaning ladders:
  - planner canonicalization already owned wrapper collapse for the admitted truth-condition family
  - lowering validation in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/validate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/validate.rs) still owned its own compare/null-test truth-family classification

- After `0.116.1`, scalar `WHERE` validation no longer keeps that second ladder. [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/validate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/validate.rs) is now just a clause-admission adapter that asks planner `expr` whether one lowered expression belongs to the admitted scalar truth-condition family. Lowering still owns the clause boundary, but not the compare/null-test meaning underneath it.

- The planner-owned scalar truth-condition family is also richer now in one precise way: compare and null-test truth shaping read from the same bounded compare-operand family that scalar `WHERE` already shipped, including:
  - `LOWER` / `UPPER`
  - the existing bounded text/numeric helper family such as `TRIM`, `ABS`, `ROUND`, `COALESCE`, `NULLIF`, `LENGTH`, `LEFT`, `RIGHT`, `POSITION`, `REPLACE`, and `SUBSTRING`
  - arithmetic subtrees used as compare operands
  - searched `CASE` result arms whose conditions already belong to the admitted scalar truth family

- This is still fail-closed. `0.116.1` does not widen scalar `WHERE` into arbitrary expression truthiness. It only makes compare/null-test truth admission and wrapper collapse read from one planner-owned rule, so the same shipped scalar family is interpreted in one place instead of two.

- The proof surface for `0.116.1` stays bounded:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs) now proves a richer admitted compare wrapper canonicalizes through the same planner-owned scalar truth-condition form
  - the same file now also proves a false-wrapped null test canonicalizes to `NOT <null test>`
  - no new session/cache matrix was needed because this step changes the owner of compare/null-test truth admission, not the already-pinned outward cache/reuse boundary from `0.116.0`

- `0.116.1` also carries that truth-condition owner move over to grouped compare semantics. Grouped `HAVING` compare support and grouped compare evaluation no longer live in [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/semantics.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/semantics.rs) and then route back into the planner through predicate re-exports. That authority now lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_having.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_having.rs).

- That grouped follow-through matters for the same architectural reason as the scalar move. Before this cut, planner-owned grouped semantics still depended on a predicate-owned rule for:
  - which compare operators grouped `HAVING` admits
  - how grouped null-aware equality/ordering behaves under the bounded grouped compare family
  - the test-only grouped compare evaluator used by planner-owned grouped regression coverage

- After the grouped follow-through inside `0.116.1`, planner semantics own that whole grouped compare boundary:
  - `grouped_having_compare_op_supported(...)`
  - `evaluate_grouped_having_compare(...)` for planner-owned tests
  - the grouped compare kind classifier used by those rules

- The grouped boundary imports now match that owner move:

## 0.116.0

- `0.116.0` starts the truth-condition centralization line by moving the shipped truth-wrapper family onto the planner expression owner in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs). Scalar `WHERE` no longer treats outer truth wrappers as a separate local semantic lane. Instead, planner canonicalization now owns the admitted collapse rules for:
  - `expr = TRUE`
  - `TRUE = expr`
  - `expr = FALSE`
  - `FALSE = expr`
  - parser-lowered `expr IS TRUE`
  - parser-lowered `expr IS FALSE`

- That owner move is intentionally scoped, not generalized. The new planner-owned `TruthWrapperScope` keeps one shared wrapper authority while preserving the bounded difference between scalar `WHERE` and grouped `HAVING` admission:
  - scalar `WHERE` uses the already-shipped boolean truth-condition family
  - grouped `HAVING` keeps the slightly wider admitted grouped boolean family
  - both now collapse through the same planner-owned wrapper canonicalization path instead of grouped-only helper ladders

- The scalar side of that new planner-owned wrapper family is bounded and fail-closed. [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/canonicalize.rs) now admits wrapper collapse only for the shipped scalar truth-condition family:
  - bare boolean fields
  - boolean/null literals
  - `NOT`, `AND`, and `OR`
  - admitted compare expressions
  - admitted boolean-result functions
  - searched `CASE`
  - the bounded text-targeted helper family already aligned with scalar compare admission

- Predicate remains the bridge, not the semantic owner. [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs) still exposes the canonicalization entrypoints used by predicate-facing tests and adapters, but the wrapper meaning now comes from planner `expr`. That keeps the predicate layer from owning an independent `= TRUE` / `IS TRUE` / `IS FALSE` interpretation model.

- Lowering follow-through is now pinned for the scalar wrapper family too. [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs) now proves:
  - `WHERE active IS TRUE` lowers onto the same canonical plan identity as `WHERE active`
  - `WHERE active IS FALSE` lowers onto the same canonical plan identity as `WHERE NOT active`
  - the corresponding plan hashes stay equal once planner wrapper canonicalization owns the family

- Session-level semantic identity is also frozen now. [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs) adds a scalar truth-wrapper reuse proof showing:
  - `active IS TRUE` and bare `active` still occupy distinct compiled-command cache entries
  - they collapse onto one structural cache key
  - they share one semantic plan fingerprint
  - they reuse one shared query-plan cache entry at execution time
  - the same holds for `active IS FALSE` and `NOT active`

- `0.116.0` also closes the prepared-vs-non-prepared parity target from the first-step addendum without widening prepared SQL. [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_parameterized.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_parameterized.rs) now proves that zero-parameter prepared queries for `IS TRUE` and `IS FALSE`:
  - stay off prepared template lanes
  - lower through the same canonical structural identity as non-prepared bare / `NOT` truth forms
  - share the same semantic plan fingerprints as those non-prepared canonical forms
  - preserve the same outward result rows at execution time

- One small test-only accessor was added to support that prepared parity proof:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs)
  - `PreparedSqlQuery::statement_for_test(...)`
  This does not widen the production API; it only lets the session test layer compare the prepared empty-binding path against the non-prepared lowering path directly.
