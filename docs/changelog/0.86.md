# 0.86 — Grouped HAVING Expression Widening

## 0.86.0 - 2026-04-16 - Post-Aggregate Grouped HAVING Expressions

This patch starts the `0.86` line by widening grouped SQL `HAVING` onto the
post-aggregate expression layer introduced in `0.85`. Grouped `HAVING` no
longer stops at conservative `field/aggregate op literal` clauses; it now
admits bounded boolean comparisons over grouped keys, finalized aggregate
outputs, literals, arithmetic, and `ROUND(...)`.

What changed:

* grouped plan/runtime now carry and evaluate a real grouped `HAVING`
  expression tree instead of requiring the legacy clause form at execution
  time:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/model.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_model.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_having.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/aggregate/runtime/mod.rs`
* grouped SQL parsing now accepts widened `HAVING` value expressions on both
  sides of a compare, including grouped fields, aggregate leaves, literals,
  bounded arithmetic, and `ROUND(...)` in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/model.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/clauses.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/statement/mod.rs`
* grouped SQL lowering now resolves widened `HAVING` onto the grouped plan
  expression surface instead of rejecting non-legacy forms in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/normalize.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/mod.rs`
* grouped SQL lowering now builds one shared deduplicated aggregate set across
  grouped projection and grouped `HAVING`, so `HAVING`-only aggregates like
  `AVG(age)` are still declared, indexed, and executed even when they are not
  projected.
* grouped projection identity lowering now disables the `ProjectionSelection::All`
  fast path when grouped `HAVING` introduces extra hidden aggregates, so those
  `HAVING`-only aggregate outputs do not leak into visible grouped result rows
  in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/projection.rs`
* grouped static planning semantics now extend the frozen grouped aggregate
  execution-spec bundle with any aggregate indexes referenced by legacy or
  widened grouped `HAVING`, so runtime sees the full post-aggregate surface in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs`

Examples that now work:

```sql
SELECT class_name, COUNT(*)
FROM character
GROUP BY class_name
HAVING COUNT(*) + 1 > 5
ORDER BY class_name
LIMIT 100;
```

```sql
SELECT class_name, AVG(strength)
FROM character
GROUP BY class_name
HAVING ROUND(AVG(strength), 2) >= 10
ORDER BY class_name
LIMIT 100;
```

Deliberately still out of scope:

* grouped `ORDER BY` widening remains unchanged.
* aggregate-input expression widening remains unchanged.
* raw non-grouped field references in grouped `HAVING` remain rejected.
* the existing conservative grouped `HAVING` fluent surface is still carried in
  parallel for compatibility with the older grouped builder path.
