# IcyDB 0.197 Changelog

## 0.197.0

### 📝 Summary

- Starts the deterministic optimizer-canonicalization line with contract
  evidence for primary-key filter spellings.
- Proves strict scalar primary-key equality filters and finite primary-key
  `IN (...)` filters reach bounded `ByKey` / `ByKeys` public-read proofs
  without requiring fake `LIMIT` clauses.
- Proves external scalar primary keys, including `Principal` keys, use the same
  bounded lookup semantics as explicit `by_id(...)` / `by_ids(...)` APIs.
- Keeps the slice test-only: no public API, cursor format, persisted format,
  recovery behavior, diagnostic code, or production runtime path changes.

### 🧪 Testing

- Adds public-read admission guards for generated `id = value` and
  `id IN (...)` filters without explicit limits.
- Adds default fluent execution guards proving primary-key filter spellings
  return the same rows as explicit key APIs and deduplicate repeated `IN`
  values.
- Adds external `Principal` primary-key fixtures and parity tests for scalar
  equality and finite key-set filters.
- Adds residual-safety guards proving exact-key access does not skip remaining
  predicates after admission.
- Adds SQL boundary guards for `WHERE id = ...` and `WHERE id IN (...)`
  primary-key filters, including residual predicate cases.
- Adds EXPLAIN route-fact coverage proving primary-key equality reports
  `ByKey` / `planner_primary_key_lookup`, finite primary-key `IN (...)`
  reports deduped canonical `ByKeys` / `planner_key_set_access`, and residual
  work remains visible after exact-key access.

### 📊 Performance

- No runtime performance win is claimed for this patch.
- Full deterministic performance matrices were not rerun because the slice only
  adds contract coverage for existing behavior.
- Wasm-size delta was not measured.

### 📊 Complexity Delta

- Production complexity is unchanged.
- Test complexity increases through focused session-boundary coverage for
  public admission, default execution, SQL execution, external primary keys,
  residual predicates, and EXPLAIN route facts.
- Implementation shape stays simpler than a production optimizer patch: the
  line starts by proving current canonicalization behavior before adding any
  new runtime machinery.
