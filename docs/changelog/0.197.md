# IcyDB 0.197 Changelog

## 0.197.7

### 📝 Summary

- Hardens finite primary-key `IN (...)` public-read resource evidence without
  changing production behavior.
- Proves the returned-row policy is enforced against the deduplicated exact key
  count for both fluent and SQL primary-key `IN (...)` routes.
- Keeps duplicate input-list handling deterministic while ensuring the admitted
  `ByKeys` row bound cannot exceed the caller-facing endpoint row budget.

### 🧪 Testing

- Adds fluent public-read coverage for a duplicate-containing primary-key
  `IN (...)` list whose deduplicated count exceeds the returned-row policy.
- Adds SQL public-read coverage for the same deduplicated-count-over-policy
  shape.
- Verifies both surfaces keep the planner-owned `ByKeys` access proof visible
  while rejecting through the returned-row policy diagnostic.

### 📊 Performance

- No runtime performance win is claimed for this patch.
- Full deterministic performance matrices were not rerun because the slice only
  adds contract coverage for existing admission behavior.
- Wasm-size delta was not measured.

### 📊 Complexity Delta

- Production complexity is unchanged.
- Test complexity increases through two focused resource-policy tripwires.
- Implementation shape remains narrow and test-only.

### 🧪 Validation

- `cargo fmt --all`
- Focused deduplicated primary-key `IN (...)` row-budget tests
- `cargo test -p icydb-core --all-features read_admission`
- `cargo clippy -p icydb-core --all-features --tests -- -D warnings`
- `git diff --check`

## 0.197.6

### 📝 Summary

- Admits finite primary-key `IN (...)` reads with non-key `ORDER BY` when the
  exact primary-key set itself bounds the candidate rows that must be
  materialized for sorting.
- Keeps public read materialized-order fallback fail-closed unless the selected
  access is exact primary-key access and the materialized-row bound matches the
  exact candidate key count.
- Proves fluent and SQL finite `ByKeys` reads can order by a non-key field
  without adding fake `LIMIT` clauses.

### 🧪 Testing

- Adds admission-unit coverage for exact primary-key-set materialized sorting
  and the underbounded-limit rejection guard.
- Adds fluent public-read coverage for finite primary-key `IN (...)` plus
  `ORDER BY age DESC, id ASC` without an explicit `LIMIT`.
- Adds SQL public-read coverage for the same finite primary-key `IN (...)`
  non-key ordering shape.
- Keeps the existing unresolved materialized-sort rejection tests passing for
  non-exact-key routes.

### 📊 Performance

- No broad performance win is claimed for this patch.
- The intended local behavior change is reduced application workaround pressure:
  exact finite key reads can materialize and order only the proven key set.
- Full deterministic performance matrices were not rerun for this narrow
  admission correctness slice.
- Wasm-size delta was not measured.

### 📊 Complexity Delta

- Production complexity increases slightly through one primary-key-only
  materialized-sort admission gate.
- Test complexity increases through focused admission and SQL/fluent tripwires.
- Implementation shape remains narrow: no cost model, no persisted-format
  change, no cursor-format change, and no general materialized-sort permission.

## 0.197.3

### 📝 Summary

- Hardens deterministic primary-key canonicalization across store states and
  non-canonical predicate shapes.
- Proves exact primary-key filters on heap and journaled entities admit as
  bounded `ByKey` public reads without requiring fake `LIMIT` clauses.
- Proves missing heap and journaled exact-key filters execute as bounded
  optional lookups that return no row.
- Proves deleted heap and journaled exact-key filters do not return stale rows,
  including the journaled tombstone/live-state path.
- Stops scalar SQL `WHERE` affine numeric rewrites from granting exact-key
  access authority, so expression-wrapped primary keys such as `id + 0 = 1`
  no longer explain as planner-owned `ByKey` canonicalization.
- Proves wrong-type SQL primary-key literals fail before read admission.
- Proves secondary unique-index equality and partial composite primary-key
  predicates do not masquerade as scalar primary-key access.

### 🧪 Testing

- Adds heap and journaled public-read admission guards for `id = value` filters
  without explicit limits.
- Adds heap and journaled execution guards for missing `id = value` filters.
- Adds heap and journaled execution guards for deleted rows read back through
  exact-key filters after primary-key delete.
- Adds an EXPLAIN guard for expression-wrapped numeric primary-key predicates.
- Adds a SQL wrong-type literal guard for `WHERE id = 'not-a-ulid'`.
- Adds public-read admission guards for secondary unique-index equality and
  partial composite primary-key predicates.
- Rechecks the existing affine scalar SQL `WHERE` semantics test to prove row
  results are preserved through expression evaluation.

### 📊 Performance

- No runtime performance win is claimed for this patch.
- Scalar SQL affine filters may now keep conservative expression execution
  instead of deriving exact-key access when the field is expression-wrapped.
- Full deterministic performance matrices were not rerun for this focused
  correctness guard.
- Wasm-size delta was not measured.

### 📊 Complexity Delta

- Production complexity decreases slightly by removing one scalar SQL affine
  rewrite step from the access-authority path.
- Test complexity increases through focused non-canonical primary-key
  tripwires.
- Implementation shape remains narrow: predicate-only SQL lanes keep the
  existing affine rewrite, while scalar SELECT filters no longer use it for
  access derivation.

### 🧪 Validation
