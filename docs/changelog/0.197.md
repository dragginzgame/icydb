# IcyDB 0.197 Changelog

## 0.197.2

### 📝 Summary

- Hardens the SQL side of deterministic primary-key canonicalization without
  changing production behavior.
- Proves different literal primary-key SQL values keep distinct compiled SQL
  and shared query-plan cache identities, and return the correct rows.
- Proves current `?` SQL parameter placeholders fail closed through the SQL
  lowering boundary instead of being treated as exact-key bindings.
- Proves fluent primary-key `IN (...)` results follow deterministic encoded key
  order rather than raw input-list order.

### 🧪 Testing

- Adds a SQL cache identity guard for two literal `WHERE id = ...` queries with
  different primary-key values.
- Adds a public-read SQL guard proving `WHERE id = ?` reports unsupported
  parameter placement before read admission.
- Adds a fluent execution guard proving unsorted and deduplicated primary-key
  `IN (...)` filters return the same deterministic row order as canonical input.

### 📊 Performance

- No runtime performance win is claimed for this patch.
- Full deterministic performance matrices were not rerun because the slice only
  adds contract coverage for existing behavior.
- Wasm-size delta was not measured.

### 📊 Complexity Delta

- Production complexity is unchanged.
- Test complexity increases through focused cache, unsupported-parameter, and
  deterministic-order tripwires for exact-key canonicalization.
- Implementation shape remains narrow and test-only.

### 🧪 Validation

- `cargo fmt --all`
- Focused unsupported SQL parameter exact-key guard
- Focused deterministic fluent primary-key `IN (...)` ordering guard
- Focused literal primary-key SQL cache identity guard
- `cargo test -p icydb-core --all-features read_admission -- --nocapture`
- `cargo clippy -p icydb-core --all-features --tests -- -D warnings`

## 0.197.1

### 📝 Summary

- Hardens the deterministic primary-key canonicalization line with direct
  evidence for the original external-key lookup shape that motivated 0.197.
- Proves `filter(pid = value)` and `by_id(Id::from_key(value))` explain through
  the same `ByKey` access strategy for an external `Principal` primary key.
- Adds fail-closed guards so exact-key canonicalization cannot hide invalid
  residual predicates or fall back to scans for contradictory primary-key
  constraints.
- Keeps the slice test-only: no public API, cursor format, persisted format,
  recovery behavior, diagnostic code, or production runtime path changes.

### 🧪 Testing

- Adds structured EXPLAIN evidence proving `filter(pid = value)` and
  `by_id(Id::from_key(value))` use the same `ByKey` access strategy for an
  external `Principal` primary key.
- Adds public-read admission guards proving existing-key and missing-key
  lookups with an unknown residual field fail with the same validation class.
- Adds public-read admission guards proving `pk = a AND pk = b`, `pk IN ()`,
  and `pk = a AND pk IN (b)` use bounded empty access instead of scan fallback.
- Adds a narrowing guard proving `pk = a AND pk IN (a, b)` admits as a single
  `ByKey(a)` proof.
- Adds default fluent execution guards proving contradictory and empty
  primary-key filters return empty result sets without requiring explicit
  limits.

### 📊 Performance

- No runtime performance win is claimed for this patch.
- Full deterministic performance matrices were not rerun because the slice only
  adds contract coverage for existing behavior.
- Wasm-size delta was not measured.

### 📊 Complexity Delta

- Production complexity is unchanged.
- Test complexity increases through focused EXPLAIN, admission, and execution
  guards for 0.197's fail-closed exact-key contract.
- Implementation shape remains narrow because current production behavior
  already satisfies these canonicalization invariants.

### 🧪 Validation

- `cargo fmt --all`
- Focused external primary-key filter-vs-`by_id` EXPLAIN proof test
- Focused invalid-residual, empty-key, and narrowing primary-key tests
- `cargo test -p icydb-core --all-features read_admission -- --nocapture`
- `cargo test -p icydb-core --all-features explain_execution -- --nocapture`

## 0.197.0

### 📝 Summary

- Starts the deterministic optimizer-canonicalization line with contract
  evidence for primary-key filter spellings.
- Proves strict scalar primary-key equality filters and finite primary-key
  `IN (...)` filters reach bounded `ByKey` / `ByKeys` public-read proofs
  without requiring fake `LIMIT` clauses.
- Proves external scalar primary keys, including `Principal` keys, use the same
  bounded lookup semantics as explicit `by_id(...)` / `by_ids(...)` APIs.
- Keeps the slice test-only: no public API, cursor format, persisted format,
  recovery behavior, diagnostic code, or production runtime path changes.

### 🧪 Testing

- Adds public-read admission guards for generated `id = value` and
  `id IN (...)` filters without explicit limits.
- Adds default fluent execution guards proving primary-key filter spellings
  return the same rows as explicit key APIs and deduplicate repeated `IN`
  values.
- Adds external `Principal` primary-key fixtures and parity tests for scalar
  equality and finite key-set filters.
- Adds residual-safety guards proving exact-key access does not skip remaining
  predicates after admission.
- Adds SQL boundary guards for `WHERE id = ...` and `WHERE id IN (...)`
  primary-key filters, including residual predicate cases.
- Adds EXPLAIN route-fact coverage proving primary-key equality reports
  `ByKey` / `planner_primary_key_lookup`, finite primary-key `IN (...)`
  reports deduped canonical `ByKeys` / `planner_key_set_access`, and residual
  work remains visible after exact-key access.

### 📊 Performance

- No runtime performance win is claimed for this patch.
- Full deterministic performance matrices were not rerun because the slice only
  adds contract coverage for existing behavior.
- Wasm-size delta was not measured.

### 📊 Complexity Delta

- Production complexity is unchanged.
- Test complexity increases through focused session-boundary coverage for
  public admission, default execution, SQL execution, external primary keys,
  residual predicates, and EXPLAIN route facts.
- Implementation shape stays simpler than a production optimizer patch: the
  line starts by proving current canonicalization behavior before adding any
  new runtime machinery.

- Focused primary-key equality, finite `IN (...)`, residual, external-key, and
  EXPLAIN route-fact tests

- `cargo clippy -p icydb-core --all-features --tests -- -D warnings`
