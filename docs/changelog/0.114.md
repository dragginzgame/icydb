# 0.114 Changelog

## 0.114.0

- `0.114.0` starts the prepared semantic-authority-collapse line by moving prepared template-lane eligibility fully back under lowering ownership. The shared scoped classifier for expression-owned template barriers now lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs), and [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs) now exposes that decision through `PreparedSqlStatement::uses_general_template_expr_parameters()`.

- Before this step, [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) was still rereading parsed `SELECT` predicate and grouped/global `HAVING` expression semantics locally before choosing whether a prepared query could stay on a template lane. That was structurally wrong for the `0.114` thesis: the session binding layer was consuming prepared SQL, but it still owned one semantic-adjacent decision about expression shape.

- After `0.114.0`:
  - lowering owns the parsed-SQL decision about whether a prepared statement uses general expression-owned parameter shapes that must stay off template lanes
  - the prepared statement envelope carries that decision as a lowering-owned fact
  - the session binding layer now only consumes that fact when selecting a prepared execution template
  - grouped compare-family template admission stays exactly as bounded as before
  - expression-owned grouped `HAVING` still stays off template lanes

- This is intentionally a structural-only ownership move. It does **not** widen prepared SQL, change grouped template admission, or redesign the fallback lane. The point is narrower: remove one more semantic revalidation from the session layer so prepared execution consumes lowering-owned truth instead of recomputing it.

  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs)

That shape check now lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs) as `sql_expr_is_boolean_shape(...)`, and prepared fallback just consumes that lowering-owned classifier.

  - `prepare.rs` no longer owns a dedicated match tree that reclassifies parser nodes into “boolean enough for top-level WHERE”
  - the lowering expression owner now owns that parsed-SQL shape decision
  - prepared fallback still enforces the same fail-closed boundary, but it does so by consuming lowering-owned structure instead of restating the shape semantics itself

## 0.114.1

- `0.114.1` continues the prepared semantic-authority-collapse line by moving the remaining symbolic-template structure off session binding and onto the planner owners that already define those shapes. The session layer still decides whether a prepared symbolic lane is allowed, but it no longer owns the grouped/simple-range parser exception, the symbolic predicate template tree, or the grouped `HAVING` template tree.

- `0.114.1` first moves the grouped simple-range parser detector out of session binding. The grouped symbolic lane in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) still keeps the same bounded exception for simple same-field `field >= ? AND field < ?` style range shapes, but it no longer derives that parser-owned range descriptor locally. [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs) now owns `prepared_sql_simple_range_slots(...)` plus the `SqlRangeCompareDescriptor` / `SqlRangeBoundKind` helper split behind it, and session binding just consumes that lowering-owned range-shape fact.

- That first `0.114.1` move preserves the same grouped symbolic boundary:
  - grouped symbolic access still keeps the simple same-field range exception
  - grouped compound `WHERE` shapes still fail closed unless they match that bounded exception
  - session binding still owns the template-lane policy decision
  - parser-owned range-shape detection is no longer session-owned

- `0.114.1` then moves the symbolic scalar predicate parser classifier out of session binding too. [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) still builds the symbolic scalar predicate template from planner-owned compare metadata, but it no longer matches the parsed SQL tree inline to decide whether the subtree is one bounded `AND`, `OR`, `NOT`, or `field <op> ?` compare shape. [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs) now owns that parser classification through `PreparedSqlPredicateTemplateShape` and `sql_expr_prepared_predicate_template_shape(...)`.

- That second `0.114.1` move keeps the same symbolic template surface:
  - the symbolic scalar predicate builder still only accepts the same bounded compare-family SQL shapes
  - session binding still owns template construction from planner-owned compare metadata
  - parser-owned predicate shape matching is no longer session-owned

- `0.114.1` then contracts one remaining internal drift point inside [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs). Aggregate compare-family inference and aggregate template-sentinel construction no longer rediscover `MIN`/`MAX` input-field shape through separate helper ladders. The prepared compare-target contract path now goes through one shared aggregate owner, so supported aggregate compare contracts read family and template binding from the same validated aggregate input shape instead of parallel local checks.

Direct field compare contracts and field-backed `MIN`/`MAX` aggregate compare contracts no longer each walk the entity model separately to rediscover field kind. [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs) now resolves field kind through one shared helper, keeping compare-family selection and aggregate-input validation on the same model-owned lookup path.

- `0.114.1` then tightens one grouped symbolic-template rebinding seam in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs). Grouped `HAVING` literal slotification no longer uses a local first-hit exemplar matcher that could alias static grouped bool literals onto unrelated bool parameter exemplars. - `0.114.1` then moves one more planner-structure walk out of session binding. The scalar symbolic-template lane in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) no longer owns its own predicate traversal just to ask whether any exemplar runtime literals survived slot lifting. Prepared template predicate rebinding no longer lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) as a local predicate tree walk.
