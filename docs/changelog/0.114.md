# 0.114 Changelog

## 0.114.0

- `0.114.0` starts the prepared semantic-authority-collapse line by moving prepared template-lane eligibility fully back under lowering ownership. The shared scoped classifier for expression-owned template barriers now lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs), and [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs) now exposes that decision through `PreparedSqlStatement::uses_general_template_expr_parameters()`.

- Before this step, [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) was still rereading parsed `SELECT` predicate and grouped/global `HAVING` expression semantics locally before choosing whether a prepared query could stay on a template lane. That was structurally wrong for the `0.114` thesis: the session binding layer was consuming prepared SQL, but it still owned one semantic-adjacent decision about expression shape.

- After `0.114.0`:
  - lowering owns the parsed-SQL decision about whether a prepared statement uses general expression-owned parameter shapes that must stay off template lanes
  - the prepared statement envelope carries that decision as a lowering-owned fact
  - the session binding layer now only consumes that fact when selecting a prepared execution template
  - grouped compare-family template admission stays exactly as bounded as before
  - expression-owned grouped `HAVING` still stays off template lanes

- This is intentionally a structural-only ownership move. It does **not** widen prepared SQL, change grouped template admission, or redesign the fallback lane. The point is narrower: remove one more semantic revalidation from the session layer so prepared execution consumes lowering-owned truth instead of recomputing it.

  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs)

- `0.114.0` then moves one more semantic-adjacent decision out of prepared fallback and into the SQL expression lowering owner. The top-level boolean-shape classifier that prepared `WHERE` fallback was using to decide whether a parameterized expression could only mean “boolean predicate surface” no longer lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs). That shape check now lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs) as `sql_expr_is_boolean_shape(...)`, and prepared fallback just consumes that lowering-owned classifier.

- That second `0.114.0` move matters because it removes one more local meaning ladder from the prepared lane:
  - `prepare.rs` no longer owns a dedicated match tree that reclassifies parser nodes into “boolean enough for top-level WHERE”
  - the lowering expression owner now owns that parsed-SQL shape decision
  - prepared fallback still enforces the same fail-closed boundary, but it does so by consuming lowering-owned structure instead of restating the shape semantics itself
  - no prepared feature surface changed and no grouped template lane widened as a side effect

- `0.114.0` then collapses one more duplicated parser-structure walk out of the prepared lane. The SQL-expression parameter walker no longer exists in multiple local forms across prepared lowering and session binding. [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs) now owns:
  - `sql_expr_contains_param(...)`
  - `sql_expr_first_param_index(...)`
  - `sql_expr_first_param_index_from_args(...)`
  - `sql_expr_first_param_index_from_pair(...)`

- That third `0.114.0` move is still structural-only, but it matters for the same reason as the earlier cuts: prepared lowering and session binding were both walking parsed SQL trees to rediscover runtime-slot presence. That is parser/lowering structure, not prepared-lane meaning. After this step:
  - `prepare.rs` consumes one lowering-owned parameter-slot traversal instead of keeping both a template-admission walker and a second fallback/error-index walker
  - `parameter.rs` no longer keeps its own copy of the same parser walk
  - runtime error attribution still uses the same first-slot behavior, but that attribution now comes from the lowering expression owner
  - no template lane widened and no parameter-contract rules changed

- `0.114.0` then moves one more parser-shape decision out of the session binding layer. The grouped symbolic prepared lane in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) was still classifying parsed SQL `WHERE` trees locally with `sql_expr_is_compound_boolean(...)` to decide whether grouped symbolic access should fail closed on compound residual predicate shapes. That classifier now lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs) as `sql_expr_is_compound_boolean_shape(...)`, and the session layer just consumes the lowering-owned result.

- That fourth `0.114.0` move preserves the same grouped symbolic boundary:
  - grouped symbolic access still fails closed on compound boolean `WHERE` shapes
  - simple same-field range ownership still keeps its existing exception
  - no grouped template lane widens
  - the difference is only authority: parsed-SQL boolean-shape classification is no longer session-owned

- `0.114.0` then removes the remaining template-admission shape ladder from [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs). The parser-shape helpers that decide whether one parameterized SQL subtree still matches the bounded prepared template family now live together in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs):
  - `PreparedSqlTemplateExprScope`
  - `sql_expr_uses_general_template_expr_parameters(...)`
  - the compare-target, null-test-target, and direct text-function shape checks that support that classifier

- That fifth `0.114.0` move keeps the same prepared boundary but makes the code reflect it more honestly:
  - `prepare.rs` still owns prepared statement assembly
  - `expr.rs` now owns the parser-shape ladder used by template-lane eligibility
  - `PreparedSqlStatement::uses_general_template_expr_parameters()` still reports the same fact outward
  - no template ownership widened and no fallback contract rules changed

- `0.114.0` then moves the template-sentinel collision expression walk out of session binding and into the lowering expression owner. [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) still owns the policy question, “should this prepared statement fail closed because one chosen sentinel already exists in the parsed SQL?”, but it no longer owns a second full recursive SQL-expression literal walk to answer it. [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/expr.rs) now provides `sql_expr_contains_any_literal(...)`, and session binding uses that shared parser-owned traversal in its statement/projection wrappers.

- That sixth `0.114.0` move keeps the same sentinel-collision boundary:
  - prepared templates still fail closed when the chosen sentinel value already appears as an ordinary literal in parsed SQL
  - the session layer still owns that prepared-template policy choice
  - the recursive parsed-SQL literal traversal is no longer session-owned
  - no prepared feature surface or template lane changes as a side effect

[/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) no longer walks `SELECT`, `DELETE`, `EXPLAIN`, and projection shapes itself to search for ordinary literals equal to chosen template sentinels. - That seventh `0.114.0` move keeps the authority line clean:
  - lowering owns parsed-statement and parsed-expression traversal
  - session binding still owns the policy question of whether a detected sentinel collision disables the prepared template lane
  - no template lane widened and no execution semantics changed
