# IcyDB 0.168 Changelog

## 0.168.5

### 🔧 Changed

- Closed the diagnostics proof row for runtime store-boundary metadata:
  data, index, and schema store snapshots continue to report `storage: stable`
  alongside stable allocation metadata.

- Closed the negative heap proof rows: `storage(heap())` remains a reserved
  derive syntax error, and generated stable-store wiring still emits no heap
  backend path.

### 🧪 Testing

- Added focused diagnostics coverage for the rendered stable storage-mode
  label.

- Revalidated derive storage parsing, macro compile-fail storage fixtures, and
  generated stable wiring.

## 0.168.4

### 🔧 Changed

- Added store-owned `SchemaStore` raw snapshot traversal with explicit
  early-stop semantics.

- Routed latest-snapshot lookup, schema-store footprint facts, and
  latest-by-entity allocation metadata through the traversal helper instead of
  scanning the backing map directly at each call site.

### 🧪 Testing

- Added focused schema-store traversal coverage proving canonical snapshot key
  order and early-stop behavior.

## 0.168.3

### 🔧 Changed

- Added backend-neutral `IndexStore` entry visitor traversal with explicit
  early-stop semantics.

- Migrated commit recovery rollback snapshots, diagnostics, integrity scans,
  schema reconciliation, schema mutation validation, and affected tests away
  from allocation-shaped index-entry snapshots.

### 🗑️ Removed

- Deleted the old `IndexStore::entries()` surface instead of preserving an
  alias or snapshot helper as a compatibility route.

### 🧪 Testing

- Added focused index traversal coverage proving canonical raw key order and
  early-stop behavior.

## 0.168.2

### 🔧 Changed

- Lowered the public dependency-path MSRV for the `icydb` crate family to Rust
  `1.88.0` while keeping the workspace maintainer toolchain on Rust `1.96.0`.

- Added CI coverage that checks the public dependency path on Rust `1.88.0`:

  ```bash
  cargo check -p icydb --no-default-features
  cargo check -p icydb
  cargo check -p icydb-core
  ```

### 🩹 Fixed

- Gated SQL-style fluent delete returning APIs behind the `sql` feature so
  typed/fluent-only downstream builds compile without default features.

## 0.168.1

### 🔧 Changed

- Added backend-neutral `DataStore` visitor traversal with explicit early-stop
  semantics for full scans, reverse scans, bounded ranges, and entity-prefix
  scans.

- Migrated executor, aggregate COUNT, diagnostics, commit rebuild, and schema
  reconciliation data-row scans away from stable-map iterator-returning
  `DataStore` APIs.

### 🗑️ Removed

- Deleted the old iterator-returning `DataStore::entries()`,
  `DataStore::range(...)`, and `DataStore::range_for_entity(...)` APIs instead
  of keeping aliases, shims, or backwards-compatible fallback routes.

### 🧪 Testing

- Added focused `DataStore` traversal tests proving canonical storage order,
  raw range bounds, compact entity-prefix bounds, and early stop behavior.

## 0.168.0

### 📊 Audit

- Started the runtime store-backend boundary line with a concrete source
  inventory of direct data/index/schema traversal call sites.

- Recorded the initial residual scans proving stable `btreemap::Iter` type
  names are already confined to store modules, while higher layers still depend
  on store traversal APIs such as:

  ```rust
  DataStore::entries()
  DataStore::range(...)
  DataStore::range_for_entity(...)
  IndexStore::entries()
  ```

### 📚 Documentation

- Promoted the 0.168 design to the active runtime store backend-boundary plan:
  stable remains the only implemented backend, `storage(heap())` remains
  reserved and rejected, and removed traversal APIs are hard-cut with no
  aliases, shims, or backwards-compatible fallback routes.
