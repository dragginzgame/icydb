# 0.113 Changelog

## 0.113.1

- `0.113.1` trims one executor wrapper cluster in [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/prepared_execution_plan.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/prepared_execution_plan.rs). The prepared load shell was still exposing several one-hop accessors that only peeled the same immutable residents back out one field at a time:
  - scalar runtime entrypoints were separately asking for authority, prepared projection shape, retained slot layout, lowered prefix specs, lowered range specs, and then the logical plan
  - grouped runtime entrypoints were separately cloning grouped execution preparation and grouped slot layout
  - delete execution was separately extracting the logical plan plus both lowered index-spec vectors from the typed prepared shell

- Those now contract into structural handoff bundles instead of parallel wrapper ladders:
  - `PreparedScalarRuntimeParts`
  - `PreparedGroupedRuntimeParts`
  - `PreparedAccessPlanParts`

- The result is intentionally structural, not behavioral:
  - scalar runtime preparation now crosses one handoff boundary instead of repeating five or six one-hop wrapper calls before every route-runtime assembly
  - grouped runtime preparation now clones its paired grouped residents through one handoff instead of two rename-only wrapper methods
  - delete execution now consumes one prepared access-plan bundle instead of reassembling the same plan/index-spec trio three separate times

- `0.113.1` also contracts the remaining low-value prepared SQL projection handoff in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/projection/runtime/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/projection/runtime/mod.rs). That path was still reading:
  - `authority`
  - `logical_plan`
  - frozen projection shape
  from `SharedPreparedExecutionPlan` through separate wrapper calls before projection execution. It now crosses one explicit shared-prepared projection bundle:
  - `SharedPreparedProjectionRuntimeParts`

- That keeps the SQL projection boundary structural:
  - the session SQL projection layer still owns SQL row shaping
  - the shared prepared shell still owns the immutable executor-side residents
  - only the field-by-field wrapper bounce was removed

- `0.113.1` also trims the last rename-only scalar continuation helpers in [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/planning/continuation/scalar.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/planning/continuation/scalar.rs):
  - `post_access_cursor_boundary()` was only renaming `cursor_boundary()`
  - `continuation_applied()` was only restating `has_cursor_boundary()`

- Callers now use the underlying continuation shape directly at the relevant boundaries:
  - post-access cursor consumers now read `cursor_boundary()`
  - continuation-presence checks now use `has_cursor_boundary()`
  - the continuation module still keeps the real boundary helpers that carry distinct meaning, such as route-mode derivation, access-scan input construction, and load scan-budget validation

- `0.113.1` also removes two low-value explain/access-choice rename wrappers:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/access_projection.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/explain/access_projection.rs) now exports `explain_access_plan(...)` directly instead of routing callers through `ExplainAccessPath::from_access_plan(...)`
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_choice/mod.rs) now builds non-index access-choice snapshots directly from `AccessChoiceExplainSnapshot` constructors instead of keeping rename-only `non_index_access_choice_snapshot_for_*` shells

- That leaves the real ownership edges in place:
  - prepared shells still own executor-ready structural handoff
  - explain projection still owns access-shape rendering
  - access-choice projection still owns its bounded chosen/rejected snapshot policy
  - only the wrapper indirection was removed

- Validation for `0.113.1` covered:
  - `cargo fmt --all`
  - `cargo check -p icydb-core`

## 0.113.0

- `0.113.0` starts the expression-pipeline contraction line with one bounded binding-layer cleanup in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs). The prepared template-admission boundary was still carrying two near-parallel recursive SQL-expression classifiers: one for expression-owned `WHERE`, and one for grouped/global `HAVING`. They now collapse onto one scoped classifier instead of duplicating the same traversal shape under different names.

- The new shared classifier is:
  - `PreparedSqlTemplateExprScope`
  - `sql_expr_uses_general_template_expr_parameters(...)`

- That contraction does **not** flatten the real policy boundary. The scope still keeps the intentional split visible:
  - filter scope keeps predicate/text-template admission rules
  - grouped/global `HAVING` scope keeps compare-family grouped-template admission rules
  - expression-owned grouped/global `HAVING` still stays off template lanes

- So the visible contract after `0.113.0` is:
  - one recursive template-admission walker
  - one explicit scope for the remaining policy difference
  - no widening of prepared template ownership

- `0.113.0` then contracts the top-level prepared contract collectors in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs). `collect_where_param_contracts(...)` and `collect_having_param_contracts(...)` were still carrying the same recursive boolean-clause walk with only a few clause-specific policy edges changed. They now share one scoped collector:
  - `PreparedSqlClauseContractScope`
  - `collect_clause_param_contracts(...)`

- That second contraction keeps the remaining clause-policy split explicit instead of flattening it away:
  - `WHERE` still rejects aggregate clause positions and still routes top-level arithmetic parameter shapes through fallback value-family inference
  - grouped/global `HAVING` still admits aggregate clause positions and still fails closed on top-level arithmetic parameter shapes
  - direct compare-family contracts still preserve their existing clause-local fast path

- So after the second contraction, the prepared lane now has:
  - one shared template-admission classifier in `parameter.rs`
  - one shared top-level clause collector in `prepare.rs`
  - one explicit scope at each remaining policy boundary
  - no widening of prepared fallback or template ownership

- `0.113.0` also contracts one more duplicated symbolic-template rebinding path in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs). `bind_symbolic_scalar_template_plan(...)` and `bind_symbolic_grouped_template_plan(...)` were both rebuilding the same scalar predicate/access instantiation work before the grouped path applied its extra `HAVING` rebind. They now share one bounded scalar-instantiation boundary:
  - `BoundPreparedSqlSymbolicScalarComponents`
  - `instantiate_symbolic_scalar_template_components(...)`
  - `finalize_bound_symbolic_template_plan(...)`

- That third contraction still keeps the grouped-only policy edge visible instead of flattening the grouped path into the scalar one:
  - scalar symbolic template binding now goes through the shared scalar-instantiation path
  - grouped symbolic template binding reuses that same scalar-instantiation path first
  - grouped-only `HAVING` rebinding still stays explicit and separate

- `0.113.0` then contracts the remaining pass-through `WHERE` / `HAVING` wrapper ladders around those shared owners. In [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs), clause collection now maps directly from `PreparedSqlClauseContractScope` to `PreparedSqlFallbackExprScope` instead of bouncing through one `WHERE` wrapper and one `HAVING` wrapper for compare and value collection. In [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs), the template-admission gate now calls the shared scoped classifier directly instead of keeping separate filter-vs-having forwarding helpers that only renamed the same walk.

- That last contraction is small, but it matters for the `0.113` thesis:
  - the scope owner now carries the scope mapping itself
  - the remaining explicit differences are policy edges, not wrapper names
  - the prepared expression/binding hotspot has fewer adjacent ladders that restate the same traversal under `WHERE` and `HAVING`

- `0.113.0` then tightens the ownership line one step further by moving prepared template-lane eligibility fully back under lowering ownership. The shared scoped classifier for general expression-owned template barriers now lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs), and [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/mod.rs) now exposes that fact through `PreparedSqlStatement::uses_general_template_expr_parameters()`. [/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/parameter.rs) no longer re-reads parsed `SELECT` predicate / `HAVING` semantics locally before choosing a template lane.

- `0.113.0` then moves one step out of the prepared lane and into the typing owner in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/type_inference/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/type_inference/mod.rs). Planner function-family ownership was still spread across three parallel ladders:
  - `function_arg_coarse_family(...)`
  - `function_result_coarse_family(...)`
  - `infer_function_expr_type(...)`
  They now share one planner-owned signature classification:
  - `FunctionTypeInferenceShape`
  - `function_type_inference_shape(...)`

- That typing-side contraction keeps the same surface and the same bounded function families, but it removes one overlapping concern:
  - coarse argument-family projection now reads from the shared signature shape
  - coarse result-family projection now reads from the same shape
  - full function type inference now consumes that same shape instead of re-grouping the function families a third time
  - dynamic functions like `COALESCE` and `NULLIF` still stay on their existing unification path instead of being forced into a fake fixed-signature bucket

- So after the current `0.113.0` work, the prepared expression/binding hotspot now has:
  - one shared template-admission classifier in `parameter.rs`
  - one shared top-level clause collector in `prepare.rs`
  - one shared symbolic scalar-instantiation boundary in `parameter.rs`
  - one explicit scope or grouped-only branch at each remaining policy edge
  - no widening of prepared fallback or template ownership
  - one shared function-signature owner in `type_inference`
