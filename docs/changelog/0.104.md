# 0.104 Changelog

## 0.104.1

- `0.104.1` fixes the grouped aggregate-order follow-through for multi-key grouped results. The user-visible repair is that bounded grouped queries may now order by an aggregate output first and then use multiple declared `GROUP BY` keys as deterministic tie-breakers.
- The bug was not in grouped Top-K execution itself. The real contract drift was in planner canonicalization: grouped queries were still inheriting the scalar hidden primary-key tie-break, so a grouped order like `COUNT(*) DESC, class_name ASC, homeland ASC` silently became `COUNT(*) DESC, class_name ASC, homeland ASC, id ASC` before grouped validation ran.
- That extra row-level tie-break was wrong for grouped plans. Grouped ordering is defined over grouped outputs, not over underlying row identities, so the scalar `..., primary_key` canonicalization rule does not belong on grouped logical or grouped access-planning order shapes.
- The fix in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/logical_builder.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/logical_builder.rs) splits canonical order handling by plan family:
  - scalar plans still append the terminal primary-key tie-break
  - grouped plans preserve the explicit grouped `ORDER BY` sequence exactly as declared
- Access planning in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_planner.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/access_planner.rs) now follows the same grouped/scalar split, so grouped access-path selection and grouped logical planning use one consistent grouped order contract instead of diverging around hidden row-level terms.
- Grouped semantic validation in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/core.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/core.rs) now also keeps the primary-key tie-break requirement scalar-only. Grouped plans still validate `ORDER BY` type/admission rules, but they no longer require the row-level deterministic scalar tie-break contract that is irrelevant to grouped output ordering.
- The grouped cursor and grouped strategy helpers in:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/grouped/cursor/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/grouped/cursor/mod.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/grouped_strategy.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/grouped_strategy.rs)
  now treat aggregate-driven grouped ordering as its own bounded Top-K lane more directly:
  - canonical grouped-key ordering still owns the resumable grouped-key prefix path
  - aggregate-led grouped ordering reserves the bounded Top-K lane
  - once that Top-K lane is selected, later grouped-key tie-breakers no longer have to pretend they are part of the canonical grouped-key prefix proof
- This patch also tightens the tests themselves so planner-level grouped aggregate ordering uses real aggregate expression terms instead of fake field labels like `"COUNT(*)"`. That keeps the planner tests aligned with the actual grouped order contract SQL lowering produces.
- Representative query now admitted by `0.104.1`:

```sql
SELECT class_name, homeland, COUNT(*) AS total_count
FROM Character
GROUP BY class_name, homeland
ORDER BY total_count DESC, class_name ASC, homeland ASC
LIMIT 40;
```

  - `cargo test -p icydb-core grouped_plan_accepts_multi_key_aggregate_order_with_group_tie_breakers_when_limited -- --nocapture`

## 0.104.0

- `0.104.0` starts the prepared-typing-alignment slice by removing more of the second nearby type-family system that had accumulated in prepared SQL fallback. The main contract in this patch is that fallback parameter families are now derived from planner-owned expression typing instead of being decided locally from ad hoc SQL-shape matches inside prepared lowering.
- The new shared coarse-family truth lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/type_inference/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/type_inference/mod.rs). That module now exposes boundary-facing helpers that project planner expression typing into the coarse bind families prepared fallback needs:
  - `infer_expr_coarse_family(...)`
  - `coarse_family_for_expr_type(...)`
  - `coarse_family_for_field_kind(...)`
  - `coarse_family_for_literal(...)`
  - `binary_operand_coarse_family(...)`
  - `function_arg_coarse_family(...)`
  - `function_result_coarse_family(...)`
- Prepared fallback in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/prepare.rs) now uses those shared helpers in three places that previously carried their own nearby family truth:
  - non-parameter subtrees now go through `lower_sql_expr(...) -> infer_expr_coarse_family(...) -> coarse bind family`
  - field-kind family projection no longer keeps a local prepared-only mapping table
  - literal family projection no longer keeps a local prepared-only mapping table
- This patch also tightens function-owned fallback typing so outer compare context is allowed to constrain dynamic result-family functions instead of forcing those functions to infer in isolation first. That matters for the expression-owned prepared fallback line because it keeps parameter-carrying trees like:
  - `COALESCE(?, ?) = name`
  - `NULLIF(?, ?) = name`
  - `CASE WHEN ... THEN ? ELSE ? END = name`
  on the correct fallback contract path without widening symbolic templates.
- The same tightening also makes fixed-result functions fail closed earlier when the outer compare family is incompatible. For example, `ABS(?) = name` now fails for the right architectural reason: the numeric function result cannot satisfy the outer text compare family, and prepared fallback no longer tries to smooth that over with a separate local typing approximation.
- What did *not* change in `0.104.0` is just as important:
  - symbolic prepared templates are still limited to predicate/access-owned shapes
  - general expression-owned `WHERE` still prepares through fallback-only contracts
  - runtime semantics and template admission did not widen
- The new prepared-side invariants pinned in tests are:
  - mixed predicate-owned plus expression-owned `WHERE` still stays off template lanes when the expression-owned side survives semantically
  - text expression-owned fallback contracts derive from the shared planner-side family rules
  - boolean wrapper expression-owned fallback contracts derive from the same shared family rules
  - all-parameter `COALESCE`, `NULLIF`, and searched `CASE` result trees can inherit outer compare family when that is the only honest source of type information
  - incompatible fixed-result functions fail closed instead of creating a second semantic gate in prepared lowering
- The most important architectural outcome of `0.104.0` is not that `prepare.rs` became tiny; it did not. The important outcome is narrower ownership:
  - planner typing is now the coarse-family truth
  - prepared fallback still owns some traversal and parameter-flow logic
  - but that traversal is increasingly constrained to contract collection instead of reimplementing expression typing

  - `cargo fmt --all`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`
