# IcyDB 0.200 Changelog

## 0.200.2 - 2026-07-12

### 📝 Summary

- Closes the final persisted-row encoding boundary. Authored values are now
  normalized against the current accepted field contract before encoding, and
  already-canonical values are strictly validated against that contract before
  row bytes are emitted.
- Keeps `AdmittedOwnedValue` limited to catalog authority and its canonical
  value. Persistence obtains a short-lived `AcceptedValueRef` carrying the
  exact kind, storage-decode, and nullability proof instead of cloning field
  contracts into values used by output and query projection.

### 🔧 Changed

- Routes generated typed fields, structural patches, SQL DDL defaults, missing
  accepted slots, and dense row re-emission through the same accepted-value
  encoding boundary.
- Makes authored persistence perform one normalization pass that produces a
  borrowed accepted proof and immediately selects the accepted storage codec.
  It no longer normalizes, discards the proof, and validates the same value a
  second time.
- Makes canonical row replay strictly validate current-format runtime values
  before selecting scalar, structural, or canonical-value storage bytes.
- Requires SQL DDL defaults with values to have accepted catalog authority.
  Catalogless default encoding is no longer a supported internal path.

### 🩹 Fixed

- Rejects root `NULL` input for required structured fields before an admitted
  value can be constructed. Nullable structured fields continue to encode the
  accepted null representation for their storage lane.
- Prevents a value admitted for one accepted kind or storage-decode contract
  from authorizing persistence through a different field contract.
- Ensures recursive `ByKind` structural patch values receive the same bounded
  normalization, decimal-scale handling, collection canonicalization, and enum
  resolution as other authored writes.

### 🗑️ Removed

- Removes the persisted-row encoder that accepted an arbitrary runtime `Value`
  and selected a byte codec without carrying an accepted-value proof.
- Removes its duplicate recursive decimal normalization, list traversal, map
  traversal, scalar projection, and nullable-slot encoding helpers. Accepted
  value admission is now the single normalization owner.
- Removes the structural patch fallback that converted authored `InputValue`
  back into an unproven runtime value before persistence.

### 🧪 Testing

- Passes the full repository gate, including feature-matrix checks, workspace
  Clippy with warnings denied, compile-fail API contracts, workspace tests,
  PocketIC SQL and performance correctness suites, and SQL canister builds.
- Adds focused coverage for accepted input encoding, wrong-contract rejection,
  required and nullable `NULL`, enum ID round-trips, and catalog-backed DDL
  default validation.

### 📊 Audit

- The implementation cleanup touches 12 Rust files with 402 insertions and 470
  deletions, a net reduction of 68 lines. The shape is simpler: one admission
  kernel owns normalization and one proof-consuming encoder owns byte emission.
- `ten_entity_fluent_rows` increases from 2,361,384 to 2,363,086 raw wasm
  bytes, an increase of 1,702 bytes. Canonically shrunk wasm increases from
  2,199,550 to 2,201,226 bytes, an increase of 1,676 bytes. This is the measured
  cost of applying accepted admission to structural write paths that previously
  bypassed it.
- Deterministic gzip context increases from 767,217 to 767,534 bytes for the
  raw artifact and from 731,836 to 732,086 bytes for the shrunk artifact. Raw
  non-gzipped wasm remains the primary size measure.
- No dedicated instruction-count improvement is claimed. The automated
  performance correctness and boundedness guards remain green.

## 0.200.1 - 2026-07-12

### 📝 Summary

- Completes the 0.200 accepted-schema row boundary. Persisted row encoding,
  decoding, patch replay, primary-key materialization, and typed row staging
  now select one accepted field contract instead of branching between accepted
  and generated metadata.
- Keeps generated entity models in their intended proposal and test roles.
  Test fixtures first resolve model proposals into an accepted snapshot and
  enum catalog before constructing row contracts.

### 🔧 Changed

- Makes every structural row contract carry an accepted row-decode contract
  and its catalog authority. Slot names, leaf codecs, absence policy, defaults,
  active slots, and primary-key facts now come from that one contract.
- Routes typed test-row staging through authored-value admission before bytes
  are emitted. Structured maps, decimal scales, enum identities, nullability,
  and storage decode therefore follow the same accepted normalization as
  production writes.
- Makes direct dense reads, sparse reads, lazy slot reads, serialized patches,
  and canonical row replay share the same accepted field validation helpers.
- Renames model-backed test helpers to describe model proposals explicitly,
  avoiding terminology that implied a supported generated-runtime codec path.

### 🩹 Fixed

- Rejects current-format value-wire payloads whose decoded value does not match
  the accepted field kind instead of deferring the mismatch to a later consumer.
- Encodes nullable structural fields according to their accepted storage lane,
  including canonical null bytes for value-backed structural contracts.
- Supplies accepted enum catalog context during serialized-patch typed
  materialization so canonical enum IDs are resolved against the same revision
  that admitted the patch.

### 🗑️ Removed

- Removes the optional generated/accepted state from structural row contracts,
  the generated field table stored beside accepted contracts, and the obsolete
  field-decode contract sum type.
- Removes duplicate generated row decode, encode, primary-key materialization,
  nullable handling, and decimal storage-normalization implementations.
- Removes test-only bridges that bypassed accepted snapshot integrity or
  admitted malformed schema fixtures such as zero-field indexes.

### 🧪 Testing

- Passes the full repository test gate, including feature checks, workspace
  Clippy with warnings denied, compile-fail API contracts, workspace tests,
  PocketIC SQL/performance correctness suites, and SQL canister library builds.
- Keeps focused persisted-row, projection, row-decode, and accepted-schema
  coverage for strict kind validation and model-proposal projection.

### 📊 Audit

- The cleanup touches 31 files with 563 insertions and 1,822 deletions, a net
  reduction of 1,259 lines. The implementation shape is simpler: one row
  authority and one accepted codec path replace the former dual branches.
- `ten_entity_fluent_rows` decreases from 2,368,585 to 2,361,384 raw wasm
  bytes, a reduction of 7,201 bytes. Canonically shrunk wasm decreases from
  2,206,309 to 2,199,550 bytes, a reduction of 6,759 bytes.
- Deterministic gzip context decreases from 769,058 to 767,217 bytes for the
  raw artifact and from 734,127 to 731,836 bytes for the shrunk artifact. Raw
  non-gzipped wasm remains the primary size measure.
- No dedicated instruction-count performance improvement is claimed; the
  standard performance correctness suites remain green.

## 0.200.0 - 2026-07-12

### 📝 Summary

- Makes enum definitions part of the accepted schema instead of repeating
  persisted definitions in every field or reconstructing them from generated
  Rust metadata at runtime.
- Gives every accepted enum type and variant a mandatory, stable, non-zero ID
  scoped to one store catalog. Names remain the authoring and output surface;
  IDs are the runtime, persistence, and equality identity.
- Introduces a crash-safe database format gate and immutable accepted-schema
  revisions so recovery, writes, queries, indexes, and output rendering all use
  one verified catalog snapshot.
- Supports target-typed unit enum values in typed, structural, fluent, and SQL
  paths while keeping payload-enum key operations and SQL enum DDL explicitly
  unavailable until their contracts are complete.

### ⚠️ Breaking - Durable Format

- 0.200 is a hard format cut. A database created by an earlier binary is
  rejected before schema, journal, row, index, or cursor payload decoding.
- Development and test databases must be deleted and recreated. Old fixtures
  must be regenerated rather than migrated, and rollback to a pre-0.200 binary
  is unsupported after a 0.200 store has been created.
- There is no compatibility reader, mutation runner, alias table, spelling
  fallback, dual-read path, read repair, or generated-metadata reconstruction.
  A current binary decodes exactly one current representation or returns a
  typed format/corruption error.
- The current durable inventory includes a database boot record, row envelope
  version 2, raw schema envelope version 1 with schema codec version 6, journal
  and commit-marker version 2, continuation-token version 2, catalog/bundle/root
  envelopes, and the catalog-proved enum index component format.
- Canonical enum values use the bounded `0x84` envelope with fixed type and
  variant IDs and an explicit unit or payload body. Unknown versions, tags,
  IDs, malformed lengths, excessive nesting, and trailing bytes fail closed.

### ➕ Added - Database Format Authority

- Adds one checksummed, permanent database boot record before the existing
  bounded shared commit-control slot. Recovery admits this record before any
  journal, schema, row, or index decoder runs.
- Keeps stable-memory allocation cardinality unchanged: each journaled store
  still owns its four data, index, schema, and journal memories, and the
  canister still has one shared database-control memory. 0.200 does not add a
  fifth memory to every store.
- The control memory contains fixed, bounded records only. It does not contain
  a tree, map, enum catalog, generic value, plan, or cache.
- Virgin initialization writes and flushes the boot record before any other
  database-owned metadata. A crash before that write leaves a virgin database;
  a crash after it can resume current-format initialization idempotently.
- One shared commit marker can continue to cover operations across multiple
  registered stores, preserving database-wide atomic recovery.

### ➕ Added - Accepted Enum Catalog

- Adds one catalog per accepted store revision, keyed by stable enum type IDs
  with path-to-ID lookup and stable variant IDs inside each definition.
- Accepted fields now reference an enum type ID. The catalog stores each enum
  definition once, even when several fields use the same type.
- Models variants as either `Unit` or `Payload { contract }`, eliminating
  states where a unit variant carries decode metadata or a payload variant has
  no contract.
- Allocates new type IDs in canonical path-byte order and new variant IDs in
  canonical name-byte order. Source declaration order, registration order,
  map iteration, and scheduling do not affect identity.
- Rejects zero IDs, overflow, duplicate names, conflicting definitions,
  dangling references, existing-field type rebinding, and direct or indirect
  recursive enum type graphs before publication.
- Requires generated Rust proposals to match the complete accepted canonical
  contract, including identity, variants, unit/payload shape, payload kind,
  storage decode, and ordering policy.

### ➕ Added - Atomic Schema Revisions

- Persists immutable accepted revision bundles containing the enum catalog and
  entity snapshots, then publishes them through a checksummed two-slot current
  root in the existing schema allocation.
- Makes the current-root compare-and-swap the publication linearization point.
  Readers see the complete old revision or complete new revision, never a
  partially updated catalog and entity set.
- Requires publication to name its expected base revision. Concurrent stale
  candidates reject and must be rebuilt, so rejected candidates do not consume
  durable enum IDs.
- Keeps the monotonic accepted revision separate from the semantic fingerprint.
  Exact no-op reconciliation changes neither value and allocates no IDs.
- Pins plans, compiled SQL artifacts, admitted owned values, accepted index
  views, and output rendering to the admitted store scope, revision, and
  fingerprint. Live execution rejects or replans stale authority before row
  access.

### 🔧 Changed - Value Admission

- Separates unresolved `InputValueEnum` names from canonical runtime
  `ValueEnum` IDs and catalog-resolved `OutputValueEnum` names. Public
  authoring cannot construct trusted canonical enum IDs directly.
- Normalization owns unresolved input and produces a canonical value against
  one accepted contract. Strict decode validation only proves already-canonical
  values; it does not normalize, repair, or coerce durable bytes.
- Carries field kind, storage decode, nullability, catalog authority, store
  scope, revision, and fingerprint together so callers cannot accidentally
  combine parts of different contracts.
- Routes generated typed saves, structural patches, same-store batch writes,
  defaults, predicates, membership lists, accepted row decode, index keys, and
  result rendering through the same catalog-backed admission authority.
- Enforces bounded recursive admission and decoding with a maximum depth of 64
  plus the applicable encoded-size limits.

### 🔧 Changed - SQL And Query Behavior

- Target-types bare SQL strings as unit enum labels only when the accepted
  target field supplies the enum contract. Unknown labels, cross-type values,
  and payload variants reject during admission.
- Applies the same rule to `INSERT VALUES`, `UPDATE SET`, read predicates,
  membership predicates, and schema defaults. Fluent enum predicates are also
  canonicalized before planning and cache fingerprinting.

```sql
INSERT INTO tasks (status) VALUES ('Ready');
SELECT * FROM tasks WHERE status = 'Ready';
```

- SQL enum type DDL (`CREATE TYPE`, `ALTER TYPE`, and `DROP TYPE`), SQL enum
  parameter placeholders, payload-enum SQL authoring, and cross-store enum
  import remain unsupported in 0.200.
- Accepted schema is now runtime authority. Generated entity and index models
  remain proposal, reconciliation, model-only convenience, and test inputs;
  accepted execution never falls back to them.

### 🔧 Changed - Equality And Indexing

- Defines enums as equality-only. Declaration order has no semantic meaning,
  and `ORDER BY`, ranges, `MIN`, `MAX`, and query-visible ID ordering reject.
- Freezes an 11-byte, versioned unit-enum equality component containing the
  enum tag, key version, type ID, variant ID, and unit-body tag.
- Allows a unit enum in a composite index as an exact equality-constrained
  prefix followed by an orderable suffix. An unconstrained enum prefix cannot
  become an ordering or range surface.
- Distinguishes pairwise equality from canonical stable-key support. Payload
  enums may be stored and compared pairwise when their accepted payload
  contract supports equality, but they cannot be used for indexes, uniqueness,
  grouping, distinct keys, hash joins/aggregation, deduplication, partitioning,
  or cursor keys.
- Keeps existing unit enum keys stable across source declaration reorder,
  append-only variant additions, and unrelated type additions.

### 🔧 Changed - Native Mutation Policy

- Treats an exact canonical definition and a source-only declaration reorder as
  no-ops.
- Supports deterministic append-only enum type and variant additions when all
  current generated contracts and capability checks agree.
- Rejects type or variant rename, variant removal, aliases, path changes,
  payload-contract changes, ordering-policy changes, and rebinding an existing
  field to a different enum type ID.
- Retains unused catalog definitions after their final field reference is
  removed. Initial 0.200 IDs are never reused.
- Does not build row scanners, rewrite runners, proof runners, or migration
  orchestration for rejected operations.

### 🗑️ Removed

- Removes embedded accepted enum definitions from persisted field kinds and
  removes the possibility of releasing an embedded-registry mode beside the
  catalog model.
- Removes the context-free `EnumValue` conversion trait, generated enum storage
  codec, path/label runtime tuple helpers, and infallible unresolved-enum to
  runtime-value conversion.
- Removes the hidden model-only runtime slot encode/decode facade and makes
  persistence consume accepted-value proof instead of arbitrary runtime
  `Value`.
- Removes headerless schema snapshot acceptance and all other old-format
  decode paths covered by the 0.200 hard cut.

### 🧭 Migration Notes

- Delete and recreate every development or test database before running a
  0.200 binary. Do not attempt to open or repair a pre-0.200 database.
- Regenerate persisted fixtures, snapshots, cursors, and index data with the
  0.200 binary. Old fixtures are not valid migration inputs.
- Use `InputValue` and `InputValueEnum` for structural enum authoring. Runtime
  enum IDs are intentionally opaque and store-local; move values between
  stores by names plus an explicit target contract, not by copying IDs.
- Keep generated enum declarations exactly aligned wherever a shared enum type
  is proposed. Variant declaration order may differ, but membership and every
  payload contract must match.
- Use unit enum labels in target-typed SQL expressions. Do not rely on SQL enum
  DDL, payload literals, enum ordering, or enum parameter placeholders in this
  release.

### 🧪 Testing

- Adds a central persisted-format corpus covering current, older, future,
  malformed, truncated, and representative corrupt bytes for every present
  durable or externally resumable container.
- Adds crash-order and virgin-initialization coverage for the database boot
  record, shared commit marker, journal replay, and current-schema root.
- Adds deterministic catalog allocation, exact generated-contract,
  append-only reconciliation, stale-publication, field-rebinding, recursive
  admission, canonical wire, and store-provenance coverage.
- Adds enum write/read/default/predicate tests across typed, structural, fluent,
  SQL, cursor, index, grouped, output, and generated-facade boundaries.
- Adds frozen unit enum value and equality-key vectors plus stability coverage
  for declaration reorder and append-only catalog changes.

### 📊 Audit

- The final cleanup kept the measured `ten_entity_fluent_rows` wasm footprint
  unchanged at 2,368,585 raw bytes and 2,206,309 shrunk bytes. Gzip context is
  769,058 raw and 734,127 shrunk bytes; raw non-gzipped bytes remain the primary
  decision metric.
- This is a deliberately cross-cutting storage and schema change, so the total
  implementation is larger. The representation model is nevertheless simpler:
  one persisted enum form, one accepted catalog authority, one admission path,
  and no compatibility or generated-runtime fallback branch.
- No general query-performance improvement is claimed. The release prioritizes
  fail-closed persistence, deterministic identity, atomic publication, and
  consistent boundary validation.
