# IcyDB 0.61

This document aggregates detailed release notes for the `0.61.x` minor line.

Patch numbering in active work follows the next release after the workspace
version in `Cargo.toml`. With the workspace currently at `0.61.5`, the active
follow-up work in this line belongs to `0.61.7`.

## 0.61.7 - 2026-03-21 - Structural Delete Execution

### Summary

The seventh `0.61` patch applies the same successful data-flow split to the
next execution entrypoint: `DeleteExecutor<E>::execute`.

After `0.61.6`, the remaining top project-owned duplicated family in the debug
monos scan was the typed delete executor. It still mixed two different kinds of
work in one `E`-specialized body:

- typed delete semantics:
  - decode rows into `E`
  - run delete post-access filtering and ordering
  - format the final typed response rows
- structural orchestration:
  - validate the mutation plan
  - resolve structural access rows
  - validate strong relations
  - assemble rollback row ops
  - cross the commit window

The important constraint here was the same one that already paid off in commit
prep: do not replace generic orchestration with callback-based orchestration.
The rejected commit-prep callback split made raw wasm worse because it
redistributed monomorphization instead of removing it.

So this patch uses the proven shape instead:

- one structural delete execution state
- one structural access/materialization path
- one typed decode-and-filter leaf
- one structural delete commit-preparation core

### Changed

- split delete execution in `db::executor::delete` into data-flow stages:
  - `prepare_delete_execution_state(...)` now owns structural plan validation
    and one-shot predicate/index compilation through structural authority
  - `resolve_delete_candidate_rows(...)` now resolves delete access rows from
    `StoreHandle + EntityAuthority` through `StructuralTraversalRuntime`
  - `prepare_typed_delete_rows::<E>(...)` is now the strict typed leaf that
    decodes rows, applies delete post-access filtering, and packages response
    rows plus rollback rows
  - `prepare_delete_commit(...)` now owns structural strong-relation
    validation, store preflight, and mechanical delete row-op assembly
- added one structural mutation-plan preflight entrypoint in
  `db::executor::mutation`:
  - `preflight_mutation_plan_for_authority(...)`
  - delete now uses that structural entrypoint instead of routing through one
    typed wrapper
- removed dead typed wrappers exposed by the new delete path:
  - `validate_executor_plan::<E>(...)`
  - `preflight_mutation_plan::<E>(...)`
  - `slot_map_for_entity_plan::<E>(...)`
  - the old typed `Context<'_, E>` row-materialization convenience methods
    that were only supporting the previous delete access shell

## 0.61.6 - 2026-03-20 - Structural Relation Validation, Reverse-Index Preparation, And Commit Finalization

### Summary

The sixth `0.61` patch keeps applying the same structural-core rule to the last
large relation and commit-preflight duplication owners.

After `0.61.5`, relation work still had two meaningful typed owners:

- strong-relation delete validation still decoded one full typed source entity
  just to recover the relation field value again
- reverse-index preparation still carried typed old/new source entities through
  commit prep even though the mutation loop only needed relation targets and the
  already-validated source key

Those two cuts landed first, but the real remaining owner after that work was
still `prepare_row_commit_for_entity_impl`. The earlier callback-based attempt
to split that path looked cleaner in debug symbols but made raw `wasm-release`
worse, so it was rejected.

The winning shape was different:

- typed commit preparation now becomes one strict leaf that only decodes the
  old/new entity rows if forward-index planning needs them
- the heavy commit orchestration now runs through one structural finalization
  function fed by pure data, not callbacks
- reverse-index preparation now consumes raw rows plus a validated structural
  source key through one nongeneric reader seam

So `0.61.6` now moves relation validation, reverse-index preparation, and the
rest of commit finalization onto raw row bytes plus structural metadata. The
typed path survives only at true edges:

- blocked-delete diagnostics still reconstruct the typed source key for the
  final error
- forward-index planning still keeps typed entities where it really needs them

### Changed

- extended strong relation metadata so delete validation can reason about the
  source relation field structurally:
  - `StrongRelationInfo` now carries the relation field kind
- added one structural relation-target extraction helper in
  `db::relation::reverse_index`:
  - `relation_target_keys_for_source_row(...)` decodes the relation field
    directly from the persisted row payload
  - the helper only supports storage-key-compatible strong-relation field kinds
    and list/set wrappers around those key kinds
- removed typed entity decode from the strong-relation delete validation hot
  loop in `db::relation::validate`:
  - `validate_delete_strong_relations_for_source::<S>(...)` no longer calls
    `source_raw_row.try_decode::<S>()`
  - it no longer routes through `relation_target_keys_for_source(&source, ...)`
    just to recover relation targets already present in the raw row
- moved reverse-index preparation onto structural row bytes and the already
  validated source storage key:
  - `prepare_reverse_relation_index_mutations_for_source_rows(...)` now
    consumes `RawRow` plus one `StorageKey` and no longer monomorphizes over
    the source entity
  - reverse-index prep no longer extracts source ids through typed entity
    values inside commit preparation
  - one structural index-entry reader seam now feeds that reverse-index path
    from `Context<'_, E>`, trait readers, and the preflight overlay without
    pulling entity typing back into the shared mutation loop
- split commit preparation in `db::commit::prepare` into pure-data stages
  instead of callback stages:
  - `prepare_row_commit_structural_inputs(...)` now owns structural
    path/fingerprint checks, raw key/data key extraction, old/new row loads,
    and no-op detection
  - `prepare_typed_commit_leaf::<E>(...)` is now the strict typed leaf that
    only decodes rows and plans forward-index mutations
  - `finalize_row_commit_structural(...)` now owns reverse-index dispatch,
    data-store resolution, mutation assembly, and final `PreparedRowCommitOp`
    materialization
- widened relation metadata just enough to support the new structural seams:
  - `strong_relations_for_model(...)` now shares the same relation metadata
    lookup without forcing a typed entity shell
  - `ReverseRelationSourceInfo` now crosses the commit/relation boundary as
    structural authority data instead of being rebuilt behind a typed helper
- kept typing only at the final error edge:
  - `source_data_key.try_key::<S>()` is still used when formatting the
    blocked-delete diagnostic for the caller

## 0.61.5 - 2026-03-20 - Narrower SQL Delete Fallback After Shared Lowered-Query Dispatch

### Summary

The fifth `0.61` patch keeps pushing the remaining lowered-SQL query-lane
wrapper toward the same structural shape as the earlier execution and EXPLAIN
cuts.

`0.61.4` already moved the shared `SELECT` path onto one structural core and
left only one smaller typed delete fallback per entity. That fallback was still
paying for more typed control flow than it needed because it rebuilt one full
typed `SqlCommand<E>` and routed back through the mixed load/delete query
helper.

This patch narrows that last SQL delete path again. The typed fallback now
binds one delete `Query<E>` directly from the lowered SQL shape and jumps
straight to one dedicated typed delete execution helper. The old duplicated SQL
query-wrapper families disappear from the top debug monos list, and the raw
simple-entity slope drops again.

### Changed

- narrowed the lowered-SQL delete fallback in
  `db::session::sql`:
  - `execute_lowered_sql_dispatch_delete::<E>(...)` now rejects non-delete
    lowered shapes explicitly
  - it binds one typed delete `Query<E>` directly with
    `bind_lowered_sql_query::<E>(...)`
  - it no longer rebuilds one full typed `SqlCommand<E>` only to route back
    through the mixed query helper
- added one dedicated typed delete SQL execution helper:
  - `execute_typed_sql_delete(...)` now owns the final typed delete query
    execution plus deleted-row projection formatting
- widened the existing lowered-query binder visibility only far enough to let
  the session SQL boundary reuse it:
  - `bind_lowered_sql_query::<E>(...)` is now available within `crate::db`

## 0.61.4 - 2026-03-20 - Shared Scalar Execute, SQL Dispatch, Reverse-Relation, Structural Explain, And Structural SQL Select Dispatch

### Summary

The fourth `0.61` patch attacks the first three large duplicated symbol
families from the new `one_simple` / `ten_simple` audit matrix.

The first target was the unpaged scalar `LoadExecutor<E>::execute` entry path,
which the raw debug `twiggy monos` scan was still showing as one copy per
entity in `ten_simple`.

That slice moved the rows-only entry path onto one canister-shared structural
scalar execution helper. The typed `LoadExecutor<E>::execute` method now only
converts the structural page back into the final typed response at the outer
boundary.

The second target was the generated SQL query-lane callback path. After the
shared scalar execute cut, the next large duplicated family was still the
entity-typed SQL dispatch wrapper under `DbSession<C>`.

This patch then keeps flattening that SQL path in five steps:

- query routes now call one narrowed typed query helper
- explain routes now call one narrowed typed explain helper
- lowered query binding now shares one structural builder path before wrapping
  back into `Query<E>`
- explain plan/json rendering now shares one structural core and only leaves
  execution/global-aggregate fallback on the typed path
- lowered SQL query dispatch now runs through one structural core for `SELECT`
  and leaves only one small typed `DELETE` fallback at the boundary

The fourth target was the residual typed SQL explain fallback for global
aggregate execution explain. Even after the earlier SQL dispatch split, the raw
debug `twiggy monos` report still showed one large duplicated
`explain_lowered_sql_dispatch_typed_execution::<E>` family per entity.

This patch removes that fallback as a meaningful per-entity body by
structuralizing global aggregate EXPLAIN around model-owned query descriptors
and aggregate route planning. The typed SQL explain wrapper now only passes the
entity model into the shared structural core.

The third target was reverse-relation mutation preparation during commit
preflight. That path was still carrying one large typed helper family per
entity while deriving the mechanical reverse-index insert/delete set.

This patch pushes the heavy reverse-index mutation body onto structural source
authority and precomputed storage keys, leaving the typed shell to gather the
old/new relation-target sets and then call the shared mutation helper.

### Changed

- Moved the unpaged scalar rows entry path off the entity-generic load wrapper:
  - `LoadExecutor<E>::execute(...)` now converts `ExecutablePlan<E>` into a
    prepared structural load plan, calls one canister-shared scalar execution
    helper, and decodes the final structural page at the typed boundary
- Added one canister-shared scalar rows execution seam:
  - `execute_scalar_rows_for_canister(...)` in the load entrypoint root
  - `execute_prepared_scalar_rows_for_canister(...)` in the scalar entrypoint
    module
  - `prepare_scalar_route_runtime_for_canister(...)` for the shared scalar
    route/runtime preparation path
- Removed the old scalar-rows orchestrator mode and surface branch:
  - deleted `LoadMode::ScalarRows`
  - deleted the untraced scalar-rows `LoadExecutionSurface` variant
  - collapsed the remaining scalar/grouped orchestration surfaces onto traced
    paged payloads only
- Kept the typed boundary narrow:
  - the shared scalar rows path now returns `StructuralCursorPage`
  - only `page.into_entity_response::<E>()` remains on the typed outer edge
- Split generated SQL dispatch callbacks by lane:
  - `sql_dispatch` route tables now expose distinct `query` and `explain`
    callbacks instead of one combined query-lane callback
  - generated SQL route execution now chooses the narrowed callback from the
    parsed statement route before entity-typed execution begins
- Narrowed typed SQL session entrypoints:
  - replaced the combined `execute_lowered_sql_dispatch_query_lane::<E>(...)`
    path with `execute_lowered_sql_dispatch_query::<E>(...)` and
    `explain_lowered_sql_dispatch::<E>(...)`
  - split the typed SQL command execution helpers so query and explain paths
    no longer share one monomorphized entity wrapper
- Flattened lowered SQL query binding onto a shared structural builder:
  - `bind_lowered_sql_query_structural(...)` now owns the heavy lowered-select
    shape application before `Query::from_inner(...)` wraps the structural
    query back into the typed outer shell
  - the old duplicated `bind_lowered_sql_query` family no longer appears in
    the fresh raw debug `ten_simple` monos report
- Flattened explain plan/json rendering onto a shared structural core:
  - `render_lowered_sql_explain_plan_or_json(...)` now owns the shared
    structural plan/json explain rendering path
  - `explain_lowered_sql_dispatch_core(...)` owns the lane guard plus
    structural-first dispatch for regular lowered EXPLAIN shapes
- Flattened global aggregate EXPLAIN onto a shared structural core:
  - added `assemble_aggregate_terminal_execution_descriptor_with_model(...)`
    so aggregate-terminal execution explain can run from one model-owned
    structural descriptor path instead of `ExecutablePlan<E>`
  - added `StructuralQuery::explain_aggregate_terminal(...)` so global
    aggregate execution explain stays on the generic-free query core
  - added one structural lowered global-aggregate binder so EXPLAIN dispatch
    can build `StructuralQuery` directly from the lowered SQL shape without
    `bind_lowered_sql_command::<E>`
  - moved global aggregate SQL terminal validation onto
    `sql_global_aggregate_terminal_to_expr_with_model(...)`
  - `explain_lowered_sql_dispatch_core(...)` now owns the full EXPLAIN lane:
    regular explain and global aggregate explain both run structurally
  - the old `explain_lowered_sql_dispatch_typed_execution::<E>` family is gone
    from the fresh raw debug `ten_simple` monos report
- Flattened reverse-relation mutation preparation onto structural authority:
  - added one structural `ReverseRelationSourceInfo` boundary for source path
  and tag authority
  - precomputed old/new source `StorageKey` values once at the typed boundary
  instead of inside the per-target mutation loop
  - moved reverse-index entry decode/update/re-encode work into one shared
  `prepare_reverse_relation_index_mutation_for_target(...)` helper
  - updated delete-time strong-relation validation to reuse the same
    structural reverse-index helpers so the reverse-index seam is consistent
    across validation and prepare paths
- Flattened lowered SQL `SELECT` dispatch onto one structural core:
  - added one nongeneric `execute_lowered_sql_dispatch_query_core(...)` path
    that owns lane gating, lowered-query extraction, structural binding, and
    structural projection execution
  - kept `<E>` only on one delete-only fallback:
    `execute_lowered_sql_dispatch_delete::<E>(...)`
  - `execute_lowered_sql_dispatch_query::<E>(...)` now only passes model
    authority, entity authority, and the small typed delete callback into the
    shared core
  - narrowed that delete-only fallback again so it now binds one typed delete
    `Query<E>` directly and jumps straight to one dedicated
    `execute_typed_sql_delete(...)` path instead of rebuilding one full typed
    `SqlCommand<E>` and routing through the mixed load/delete query helper

shared runtime helper seam and resets the wasm audit fixture matrix around
shared helpers under `testing/`.

and newtype wrapper types. Instead of re-emitting the same collection and map
conversion loops per generated schema type, the derive output now calls shared
helper functions in `traits`.

### Changed

## 0.61.0 - 2026-03-20 - Initial Entity Macro Compression

### Summary

The opening `0.61` patch starts the next wasm-focused macro cleanup after
`0.60` removed generated view DTOs.

This patch does not attempt a full descriptor-runtime rewrite. Instead, it
takes the first smaller structural compression steps that match the current
macro code:

- entity runtime model emission no longer fans out into one generated
  `__MODEL_FIELD_*` constant per field
- generated entity and record traversal now runs through a shared
  field-descriptor loop instead of spelling out per-field `drive` bodies
  inline
- visit/sanitize/validate generation now shares one derive-side field-walk
  lowering path instead of each module rebuilding that field metadata shape on
  its own

### Changed

- Reduced runtime model constant fanout in entity macro output:
  - entity generation now emits one direct `__MODEL_FIELDS` array instead of
    one per-field `__MODEL_FIELD_*` constant plus an array wrapper
  - runtime `FieldModel` lowering now lives behind one derive-side helper in
    the inherent-model path
  - this keeps the emitted entity-model authority the same while shrinking the
    generated constant surface
- Lifted entity and record traversal onto a runtime field-descriptor loop:
  - added `VisitableFieldDescriptor<T>` plus shared
    `drive_visitable_fields(...)` helpers in the core visitor module
  - generated entity and record `Visitable` impls now build one static
    descriptor table and run one shared traversal loop over it
  - field-specific traversal still uses typed generated helper functions per
    field, but the repeated inline `drive` call-site bodies are gone
- Introduced one shared derive-side field-walk lowering layer:
  - added a shared field-walk helper module used by entity/record generation
  - `visitable`, `sanitize`, and `validate` now consume that shared lowering
    path instead of each rebuilding field-name/path/member-access metadata
    independently
  - this is primarily a macro-output and maintainability cleanup, but it also
    establishes the exact seam needed for later runtime-loop work
- Marked the `0.61` design line active:
  - the `0.61` design doc now reflects the post-`0.60` reality and is marked
    `In Progress`
