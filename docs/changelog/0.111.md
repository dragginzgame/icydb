# 0.111 Changelog

## 0.111.1

- `0.111.1` is an explicit release-line exception. Instead of opening `0.112`
  for the next cleanup seam, this patch carries the prepared SQL fallback
  typing consolidation work temporarily under the `0.111` line because the
  next minor cannot be pushed right now.
- The scope stays narrow and internal:
  - planner-owned coarse type truth should remain the semantic authority
  - `prepare.rs` should read as prepared-boundary policy/orchestration rather
    than as a second type engine
  - prepared template admission stays predicate/access-only
  - expression-owned prepared `WHERE` stays fallback-only
  - supported dynamic-family parameter inference and fail-closed unsupported
    placements should be pinned more clearly with tests
- This exception should not redefine the normal release shape. After
  `0.111.1`, the next clean topic should move back to a new line instead of
  extending `0.111` again.

## 0.111.0

- `0.111.0` turns grouped omitted-`ELSE` searched `CASE` from a preserved-shape outlier into a proof-gated admitted family. Grouped SQL `HAVING CASE WHEN ... THEN TRUE END` is now admitted only when lowering plus grouped boolean canonicalization can prove that the parser-lowered `ELSE NULL` expansion joins the existing grouped boolean family without leaving raw planner `Case` nodes behind. That admission gate lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs).

- The admitted omitted-`ELSE` family now follows the same grouped semantic surface as explicit `ELSE NULL`:
  - grouped SQL lowering canonicalizes the admitted omitted-`ELSE` family through the shared grouped boolean canonicalizer in [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs)
  - grouped `EXPLAIN` output now matches the explicit `ELSE NULL` form instead of preserving the original omitted-`ELSE` spelling, with coverage in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs)
  - grouped structural/cache identity and shared session query-plan reuse now match the explicit `ELSE NULL` family, with coverage in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests/cache_key.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests/cache_key.rs) and [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs)

- `0.111.0` then tightened the same shipped family instead of widening it. Redundant grouped boolean truth wrappers inside admitted `WHEN` conditions, such as `(COUNT(*) > 1) = TRUE`, now prove onto the same explicit `ELSE NULL` canonical family for grouped SQL lowering, grouped `EXPLAIN`, and grouped session reuse. That hardening is pinned in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs), [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs), and [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs).

- The same omitted-`ELSE` proof gate now applies in the dedicated global-aggregate `HAVING` lane too. `SELECT COUNT(*) ... HAVING CASE WHEN COUNT(*) > 1 THEN TRUE END` now joins the explicit `ELSE NULL` family when the grouped boolean proof succeeds, the value-family outlier still rejects with `UnsupportedSelectHaving`, and the same redundant truth-wrapper spelling now preserves identical `EXPLAIN` output there as well. The global-aggregate follow-through is pinned in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs), [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs), and [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs).

- `0.111.0` also hardens the fail-closed side of that boundary. Grouped omitted-`ELSE` searched `CASE` outside the admitted boolean family now rejects at grouped SQL lowering with `UnsupportedSelectHaving` instead of drifting into mixed planner-era behavior. The direct regression proof lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs).

- The follow-through work exposed one real grouped identity bug: grouped plan hashes were hashing projection identity and returning early, which let grouped `HAVING` differences fall out of the outward plan-hash surface whenever grouped projection identity stayed the same. [/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/hash_parts/grouping.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/hash_parts/grouping.rs) now hashes grouped projection identity and grouped shape together, so grouped `HAVING` remains part of outward logical-plan hash identity.

- That plan-hash fix is now pinned at two levels:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests/explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests/explain.rs) proves the admitted omitted-`ELSE` / explicit `ELSE NULL` family stays hash-equivalent while the explicit-`ELSE FALSE` family stays hash-distinct
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs) proves the same distinction on the outward session `trace_query` and shared query-plan reuse surface

- The shipped `0.111.0` boundary is therefore:
  - omitted-`ELSE` grouped searched `CASE` is admitted only for the already-bounded grouped boolean family
  - admitted omitted-`ELSE` grouped searched `CASE` joins the explicit `ELSE NULL` family for canonicalization, explain, plan-hash, and shared plan identity
  - redundant grouped boolean truth wrappers inside that admitted family join the same canonical `ELSE NULL` family instead of creating a second identity/explain spelling
  - dedicated global aggregate `HAVING` now uses the same proof-gated omitted-`ELSE` admission policy for that family
  - the explicit-`ELSE FALSE` grouped boolean family remains distinct
  - grouped omitted-`ELSE` shapes outside that proven family still fail closed
