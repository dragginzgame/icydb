# IcyDB 0.53

This document aggregates detailed release notes for the `0.53.x` minor line.

## 0.53.2 - 2026-03-13 - Survivability Runtime Hardening

### Summary

`0.53.2` lands the stability-data hardening pass for persisted-row readability,
commit replay compatibility, migration crash safety, and fail-closed recovery
integrity.

### Changed

- Added explicit persisted row format envelopes with `N-1` compatibility-window
  decode rules and classified compatibility failures
  (`incompatible_persisted_format`) at codec boundaries.
- Added explicit commit-marker format envelopes with `N-1` compatibility-window
  decode rules and fail-closed recovery behavior on incompatible marker
  versions.
- Added an explicit migration execution contract:
  - `MigrationPlan` + `MigrationStep` + `MigrationCursor` contracts
  - bounded migration execution via commit-marker row-op steps
  - migration step progress now binds atomically to commit-marker writes so
    crash/upgrade recovery cannot re-execute already-marked steps
  - durable internal migration cursor persistence in shared commit control
    state (`migration_id`, `step_index`, `last_applied_row_key`)
  - deterministic resume behavior across restarts/upgrades without requiring
    external cursor storage
  - classified fail-closed migration-step error reporting.
- Added a new integrity scan surface (`Db::integrity_report` /
  `DbSession::integrity_report`) with deterministic per-store and aggregate
  findings:
  - data-to-index forward consistency checks (missing/divergent index entries)
  - index-to-data reverse consistency checks (orphan index references)
  - explicit classification buckets for corruption, compatibility, and misuse.
- Hardened upgrade/recovery fail-closed behavior:
  - startup recovery now enforces post-rebuild integrity validation before
    clearing persisted marker authority
  - incompatible persisted row envelopes during rebuild keep marker authority
    and preserve persisted rows for explicit operator action
  - added crash-mid-migration replay tests to lock marker-bound progress
    semantics and prevent duplicate step re-execution after recovery.

### Validation

- `cargo fmt --all`
- `cargo check -p icydb-core --locked`
- `cargo test -p icydb-core --locked db::codec::tests:: -- --nocapture`
- `cargo test -p icydb-core --locked db::commit::store::tests:: -- --nocapture`
- `cargo test -p icydb-core --locked db::commit::tests:: -- --nocapture`
- `cargo test -p icydb-core --locked db::migration::tests:: -- --nocapture`
- `cargo test -p icydb-core --locked db::diagnostics::tests:: -- --nocapture`

## 0.53.1 - 2026-03-13 - Ownership Boundary Consolidation

### Summary

`0.53.1` tightens runtime ownership boundaries so session facades,
canonicalization, continuation semantics, and index-range validation stay
authority-local and easier to audit.

### Changed

- Split the runtime session surface from one large `db/session.rs` module into
  a directory module (`db/session/`) with smaller facade modules and explicit
  wiring at the session root boundary.
- Enforced canonicalization ownership by keeping access canonicalization and
  predicate canonicalization in their owner layers, while planner surfaces
  consume canonicalized contracts instead of defining canonicalization helpers.
- Enforced continuation authority under cursor runtime ownership:
  - continuation-envelope and continuation-advance semantics remain owned by
    `cursor/envelope.rs` and `cursor/continuation.rs`
  - executor modules now call cursor-owned continuation helpers instead of
    defining duplicate continuation semantics
  - added structural guards to keep continuation semantic definitions
    cursor-owned.
- Consolidated index-range validation checks so runtime validation authority is
  explicit:
  - semantic range-slot arity and lowering invariants remain in
    `access/lowering.rs`
  - executor index-range spec alignment/required-spec/consumption checks are now
    centralized in `executor/traversal.rs`
  - stream-access modules delegate to traversal-owned validators.
- Fixed CI guard matching for generic function definitions by updating
  continuation-structure guard detection to match both `fn name(...)` and
  `fn name<T>(...)` forms.

### Validation

- `cargo fmt --all`
- `cargo check -p icydb-core --locked`
- `cargo test -p icydb-core --locked db::executor::tests::continuation_structure -- --nocapture`
- `cargo test -p icydb-core --locked db::executor::runtime_context::tests:: -- --nocapture`
- `cargo test -p icydb-core --locked db::executor::stream::access::tests:: -- --nocapture`

## 0.53.0 - 2026-03-12 - Data Survivability Contract Framing

### Summary

`0.53.0` starts the survivability line by tightening the design contract around
persisted-data readability, upgrade safety, and compatibility-aware recovery.
This patch is design-contract work only; no runtime behavior changes are
introduced yet.

### Changed

- Renamed and reframed the `0.53` design document from a deferred
  data-integrity roadmap to an active `0.53` release contract focused on
  survivability and upgrade safety.
- Added an explicit top-level compatibility contract that defines persisted
  format compatibility expectations and fail-closed behavior.
- Made the compatibility window explicit (`N-1` persisted-format compatibility)
  so parser/runtime/planning work can target one stable survivability contract.
- Tightened migration wording from generic rollback phrasing to explicit
  interruption recovery and resume-safe execution semantics.
- Clarified replay guarantees by separating marker wire-format compatibility from
  replay semantic stability.
- Tightened error-taxonomy language so corruption-classified failures remain
  distinct from compatibility-version failures and operator misuse.
- Expanded invariants and exclusions to make survivability boundaries explicit:
  no silent unreadability, no silent row loss, no implicit destructive format
  evolution, and no undeclared semantic reinterpretation.

### Documentation

- Updated `/docs/design/0.53-data-survivability/0.53-design.md` to reflect the
  active `0.53` framing, compatibility contract, revised objective naming, and
  tightened invariant language.
- Split `0.53` planning artifacts into dedicated design + status docs:
  - `/docs/design/0.53-data-survivability/0.53-design.md`
  - `/docs/design/0.53-data-survivability/0.53-status.md`
