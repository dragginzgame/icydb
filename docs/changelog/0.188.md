# IcyDB 0.188 Changelog

## 0.188.4

### 📝 Summary

- Adds an internal SQL write candidate collection.
- Tracks projected selector source rows separately from semantic mutation
  candidates.
- Adds an internal structural DELETE candidate collection that separates loaded
  rows from post-access selected candidates.
- Keeps public SQL write and DELETE behavior unchanged.

### 🔧 Changed

- Splits SQL selector-write collection from mutation payload storage:
  `SqlWriteCandidateCollection` owns internal diagnostics while
  `SqlWriteMutationBatch` owns only row payloads.
- Records projected selector source-row counts when SQL projection
  materialization hands rows to mutation-batch construction.
- Keeps INSERT VALUES on direct source-size admission while UPDATE and
  INSERT SELECT flow through the candidate collection before mutation handoff.
- Adds `StructuralDeleteCandidateCollection` for structural DELETE count and
  `RETURNING` paths.
- Records structural DELETE loaded-row count separately from selected
  post-access candidate count after delete filtering/windowing.
- Keeps bounded structural DELETE validation on the selected-candidate count
  before projection packaging and commit preparation.

### 🧪 Testing

- Added focused coverage for SQL candidate collection projected-source
  diagnostics.
- Rechecked SQL write metrics coverage to keep public staged, matched, mutated,
  and `RETURNING` counters stable.
- Rechecked bounded DELETE characterization coverage for exact bounds,
  over-limit atomicity, `RETURNING` order, and cap precedence.
- Rechecked the SQL write and structural DELETE source guard, SQL-feature
  Clippy, default `icydb-core` Clippy, formatting, and whitespace validation.

### 📚 Documentation

- Updated the 0.188 mutation-candidate collector design status with the SQL
  candidate collection and structural DELETE collection progress.

### 🧭 Migration Notes

- No public SQL syntax, fluent API, generated endpoint behavior, statement
  payload, metrics field, persistence format, schema, cursor, route, EXPLAIN,
  or cache-identity migration is intended.
- The new collection diagnostics are internal implementation vocabulary only.

## 0.188.3

### 📝 Summary

- Names SQL write candidate accounting as the shared successful-write row-count
  seam.
- Adds internal staged-row over-limit source diagnostics.
- Removes the remaining structural binary walker unsafe callback state.
- Adds a workspace unsafe-code lint gate.
- Keeps public SQL write metrics, error codes, and result payloads unchanged.

### 🔧 Changed

- Replaces the SQL write row-attribution helper with
  `SqlWriteCandidateAccounting`, separating semantic candidates from matched,
  mutated, and `RETURNING` row counts before projecting them into the existing
  metrics event.
- Adds internal staged-row bound-check source stages for INSERT VALUES source
  admission, selector-source mutation-batch collection, and final
  mutation-batch handoff.
- Routes INSERT, UPDATE, and DELETE write metrics through the named candidate
  accounting seam without changing exported metric fields.
- Keeps staged-row over-limit failures mapped to the existing
  `StagedRowsTooMany` SQL write-boundary error.
- Replaces structural binary list/map walker raw callback context with safe
  closure-based visitors across accepted-schema, generated-model, relation-key,
  and owned nested-payload decode paths.
- Keeps structural binary parsing, persisted bytes, accepted-schema authority,
  and generated-model compatibility behavior unchanged.
- Adds `unsafe_code = "deny"` to the workspace Rust lints so ordinary
  first-party unsafe blocks, functions, traits, and impls fail workspace
  checks.
- Uses `deny` rather than `forbid` because the current `ctor`-based schema
  registration macro expands through internal `allow(unsafe_code)` plumbing for
  constructor support.

### 🧪 Testing

- Added unit coverage for candidate accounting row counts and over-limit stage
  diagnostics.
- Rechecked SQL write metrics coverage to prove emitted staged, matched,
  mutated, and `RETURNING` counters remain stable.
- Rechecked the SQL write boundary source guard, SQL-feature Clippy, default
  `icydb-core` Clippy, formatting, and whitespace validation.
- Rechecked focused structural-field decode coverage, all-target/all-feature
  `icydb-core` Clippy, wasm size A/B fixtures, and a focused PocketIC
  local-instruction branch-set query comparison.
- Rechecked the full workspace all-target/all-feature check with the
  unsafe-code lint gate enabled.

### 📚 Documentation

- Updated the 0.188 mutation-candidate collector design status with the
  candidate accounting and diagnostic-stage progress.

### 🧭 Migration Notes

- No public SQL, fluent, generated endpoint, metrics field, persistence format,
  schema, cursor, EXPLAIN, or cache-identity migration is intended.
- The new diagnostic source stages are internal vocabulary only.
- Structural binary bytes and accepted-schema decode contracts are unchanged;
  the safe walker cleanup is an internal implementation change.

## 0.188.2

### 📝 Summary

- Extends bounded SQL write collection to `INSERT ... SELECT`.
- Removes the now-unused unbounded structural-query mutation-batch helper.
- Keeps INSERT VALUES source-size prechecks and final mutation-batch validation
  intact.

### 🔧 Changed

- Passes the existing INSERT candidate bounds into the
  `INSERT ... SELECT` structural source-row mutation-batch collector.
- Makes selector-source mutation-batch collection use one bounded mode instead
  of separate bounded and unbounded helper paths.
- Keeps `INSERT ... VALUES` on its direct source-size staged-row precheck
  rather than moving it into selector-style collection.
- Preserves final mutation-batch validation as the last guard before write
  handoff.

### 🧪 Testing

- Rechecked the public update-surface `INSERT ... SELECT` staged-row over-limit
  atomicity test.
- Rechecked INSERT SELECT boundary-code precedence coverage.
- Tightened the SQL write source guard so INSERT SELECT stays on bounded source
  collection.
- Rechecked SQL-feature Clippy, default `icydb-core` Clippy, formatting, and
  whitespace validation.

### 📚 Documentation

- Updated the active 0.188 design status to move `INSERT ... SELECT` out of the
  later-slice bucket and into current follow-up progress.

### 🧭 Migration Notes

- No public SQL syntax, statement result shape, persistence format, generated
  endpoint behavior, metrics payload, route choice, cursor format, EXPLAIN
  semantics, or cache identity is intended to change.
- Oversized bounded `INSERT ... SELECT` writes still report the same public
  staged-row write-boundary error while rejecting during source-row
  mutation-batch construction.

## 0.188.1

### 📝 Summary

- Removes the metrics sink scoped-override unsafe pointer path.
- Keeps production session metrics allocation-free through static sink
  overrides.
- Keeps test-local metrics capture safe through owned `Rc` sink overrides.

### 🔧 Changed

- Replaces the metrics sink thread-local raw pointer override with a safe
  thread-local stack of sink handles.
- Keeps nested metrics overrides and panic-time restoration behavior while
  making the scoped override depth invariant explicit.
- Clones the active sink handle before dispatch so sink implementations may
  record nested metrics without re-entering the thread-local borrow.
- Leaves persisted storage, public metrics payloads, and session execution
  behavior unchanged.

### 🧪 Testing

- Added regression coverage for nested override restoration, panic restoration,
  and reentrant metrics recording.
- Updated executor and session metrics capture helpers to use owned test sinks.
- Rechecked focused metrics, executor metrics, session metrics, heap runtime,
  journaled runtime, formatting, and all-target/all-feature `icydb-core`
  Clippy.

### 🧭 Migration Notes

- No public API, metrics event, persistence format, SQL behavior, or schema
  migration change is intended.

## 0.188.0

### 📝 Summary

- Starts the mutation-candidate collector line for bounded public writes.
- Establishes parity coverage before changing bounded write accounting.
- Keeps the durable commit window, public SQL behavior, row ordering,
  `RETURNING` payload shape, and error families unchanged.

### 🔧 Changed

- Moves bounded structural DELETE row-limit checks to the selected candidate
  boundary after delete post-access filtering/selection and before
  rollback/`RETURNING` packaging or commit preflight.
- Adds bounded UPDATE collection through the staged-row bound so mutation-batch
  construction stops once the projected semantic candidate count exceeds the
  public limit.
- Preserves the final mutation-batch validation for UPDATE as the last guard
  before save/commit execution.
- Keeps INSERT SELECT on the existing collection path for this slice.
- Boxes large lowered SQL command payloads so SQL test Clippy can run without a
  large-enum warning while leaving lowered command behavior unchanged.

### 🧪 Testing

- Added bounded DELETE characterization for exact bounds, zero bounds,
  over-limit atomicity, `RETURNING` order, and current staged/`RETURNING`
  row-cap precedence.
- Added bounded UPDATE characterization for exact bounds, zero bounds,
  over-limit atomicity, `RETURNING` order, and staged/`RETURNING` row-cap
  collision precedence.
- Tightened source guards so DELETE bounds stay at the post-access candidate
  boundary and UPDATE uses bounded collection plus final mutation-batch
  validation.
- Rechecked focused bounded UPDATE/DELETE coverage, the full SQL write module,
  SQL lowering coverage, `sql-explain` compilation, formatting, whitespace
  validation, and SQL-feature Clippy for `icydb-core`.

### 📚 Documentation

- Added the active 0.188 mutation-candidate collector design baseline.
- Documented the first implementation slice and the remaining decision to
  defer a public `MutationCandidateCollection` wrapper until DELETE and UPDATE
  accounting seams are proven stable.

### 🧭 Migration Notes

- No schema migration is required.
- No SQL syntax, fluent API, generated endpoint behavior, result payload,
  cursor format, persistence format, route choice, cache identity, or EXPLAIN
  semantics is intended to change.
- Over-limit bounded UPDATE/DELETE writes should continue to report the same
  public write-boundary error families while rejecting earlier in the internal
  candidate pipeline.
