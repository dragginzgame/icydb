# 0.128.x

## 0.128.2

`0.128.2` continues the persisted-row patch/update narrowing after the
contract-side and writer-side cleanup in `0.128.0` and `0.128.1`. This patch
does not change persisted bytes, update semantics, row-reader behavior, or
error wording. It only makes the sparse serialized patch overlay path read as
one patch-owned contract instead of repeated slot-indexing choreography at each
consumer.

The change lands in:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/patch.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/patch.rs)
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/reader.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/reader.rs)

What changed:
- added the private `SerializedPatchPayloads` owner for the slot-indexed view
  of one `SerializedUpdatePatch`
- moved duplicate-slot last-write-wins indexing into that owner
- moved complete after-image payload lookup behind
  `required_complete_payload(...)`
- moved sparse update baseline-overlay payload selection behind
  `overlay_payload(...)`
- retargeted `SerializedPatchSlotReader`,
  `canonical_row_from_complete_serialized_update_patch(...)`, and
  `apply_serialized_update_patch_to_raw_row(...)` onto that patch-owned view
- narrowed `StructuralSlotReader::validate_storage_key_value(...)` to private
  visibility now that only the public reader-owned `validate_storage_key(...)`
  wrapper needs to call it

That leaves `patch.rs` with a clearer split:
- `SerializedPatchPayloads` owns serialized patch indexing and overlay policy
- `SerializedPatchSlotReader` only adapts that sparse view to typed
  materialization
- contract-side helpers still own canonical row emission
- field-contract encode/decode remains unchanged

The practical cleanup is small but useful for the rest of `0.128`: future patch
work now has one obvious private owner for sparse serialized patch payload
selection instead of reaching back into raw `Vec<Option<&[u8]>>` staging from
multiple call sites.

The small reader visibility trim follows the same `0.128` rule: keep the
storage-key validation entrypoint available where other DB owners use it, but
do not expose the narrower raw-key helper outside the reader owner.

Validation:
- `cargo fmt --all`
- `cargo fmt --all --check`
- `cargo check -p icydb-core --tests`
- `cargo test -p icydb-core persisted_row -- --nocapture`
- `cargo clippy -p icydb-core --all-targets -- -D warnings`

## 0.128.1

`0.128.1` finishes the small follow-through after the `0.128.0` contract-side
contraction by deleting the now-dead `SlotBufferWriter` path from the
persisted-row module, removing the leftover test-only wrapper seams that were
still hanging off the old test access shape, and tightening a few helper
owners now that the broader `0.128.0` contraction has settled.

The change lands in:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/writer.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/writer.rs)
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/mod.rs)
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs)

What changed:
- deleted `SlotBufferWriter`
- removed its test-only module re-export
- rewrote persisted-row fixture tests to build dense slot payload bytes through
  test-local helpers instead of a second production writer type
- removed the old test-only `persisted_row::mod` re-export hub
- deleted:
  - `decode_slot_value_by_contract(...)`
  - `StructuralSlotReader::from_raw_row_lazy(...)`
  - `apply_update_patch_to_raw_row(...)`
- made contract-only helpers owner-local:
  - `encode_slot_payload_from_dense_slot_image(...)`
  - `canonical_row_from_slot_payload_bytes(...)`
- moved `serialized_patch_payload_by_slot(...)` into `patch.rs`
- moved `storage_key_from_scalar_ref(...)` into `reader.rs`
- moved `field_model_for_slot(...)` into `types.rs`

That leaves `writer.rs` with one production writer owner:
- `CompleteSerializedPatchWriter`

It still owns:
- staged dense-slot completeness checks for typed save/update slot images
- writer-local missing-slot error wording

It no longer also owns a second dense-slot writer whose only job was to help
tests flatten one canonical row payload. Tests now do that directly with
owner-local persisted-row helpers, so the production module surface is smaller
without changing bytes, errors, or runtime behavior.

The later follow-through matters for the same reason:
- tests now reach real owners directly instead of through a local convenience
  export layer
- reader-specific and patch-specific helpers no longer live on broader shared
  helper surfaces just because they started there earlier in the refactor
- the remaining shared helpers line up more closely with actual cross-module
  ownership instead of historical staging shape

Validation for `0.128.1` stayed on the touched seam:
- `cargo fmt --all`
- `cargo check -p icydb-core --tests`

## 0.128.0

`0.128.0` opens the persisted-row slot-contract narrowing line with a small
contract-side contraction instead of another broad ownership rewrite. This
patch does not change persisted binary formats, slot-reader behavior, or patch
semantics. It only reduces who owns the canonical row rebuild choreography.

The change lands in:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/contract.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/contract.rs)
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/patch.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/patch.rs)
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/writer.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/writer.rs)
- [/home/adam/projects/icydb/crates/icydb/src/db/mod.rs](/home/adam/projects/icydb/crates/icydb/src/db/mod.rs)
- [/home/adam/projects/icydb/crates/icydb/src/lib.rs](/home/adam/projects/icydb/crates/icydb/src/lib.rs)

Before this patch, several persisted-row paths in `patch.rs` and
`writer.rs` still owned the same choreography locally:
- build or flatten a dense slot image
- then wrap that image into the final canonical row

That choreography is now moved behind contract-side helpers:
- `canonical_row_from_payload_source(...)`
- `canonical_row_from_value_source(...)`
- `canonical_row_from_slot_payload_bytes(...)`
- `encode_slot_payload_from_dense_slot_image(...)`

Those helpers now own:
- dense canonical slot-image rebuilding
- dense slot-image flattening into the canonical slot container
- final canonical row emission from that image

As part of that move, the older internal helpers become owner-local:
- `dense_canonical_slot_image_from_payload_source(...)`
- `dense_canonical_slot_image_from_value_source(...)`
- `emit_raw_row_from_slot_payloads(...)`

That matters because `patch.rs` no longer needs to stage the same rebuild flow
as two separate local steps in multiple places. It now keeps the policy
decisions while delegating the canonical row assembly choreography back to the
contract owner, and `writer.rs` no longer owns its own copy of the dense slot
flattening loop.

The main call sites simplified by this patch are:
- `canonical_row_from_complete_serialized_update_patch(...)`
- `canonical_row_from_entity(...)`
- `canonical_row_from_structural_slot_reader(...)`
- `canonical_row_from_raw_row(...)`
- `apply_serialized_update_patch_to_raw_row(...)`

The same `0.128.0` line also folds in one small facade hard-cut:
- redundant `icydb::db::query::expr::*`
- redundant `icydb::db::query::builder::*`
- the public `icydb::error::*` module path

Those owned surfaces still exist, but now live on one canonical facade path
instead of several equivalent aliases.

This is intentionally a narrow opener for `0.128`:
- it does **not** redesign the broader `Value`-based slot contract
- it does **not** redesign `reader.rs`
- it does **not** redesign patch/update currency
- it does **not** change bytes or error messages

So the shipped win in `0.128.0` is locality:
- fewer contract-side rebuild steps staged in `patch.rs`
- no writer-local dense slot-image flattening loop
- a smaller canonical row re-emission owner surface
- a slightly smaller public `icydb` facade shape
- a cleaner starting point for later `0.128.x` work on the broader slot
  contract

Validation for `0.128.0` stayed on the touched seam:
- `cargo fmt --all`
- `cargo check -p icydb-core --tests`
- `cargo test -p icydb --lib -- --nocapture`
