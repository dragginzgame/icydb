# 0.127.x

## 0.127.3

`0.127.3` carries the direct structured codec line through the next cleanup that was still obscuring ownership after `0.127.2`. Two things happen here:
- `ValueCodec` is deleted entirely and replaced with the split runtime conversion pair `ValueSurfaceEncode` and `ValueSurfaceDecode`
- the duplicated semantic leaf byte rules that had drift pressure across structural scalar, semantic leaf, value-storage, and storage-key lanes are collapsed onto one shared typed payload owner

The runtime conversion split lands in:
- [/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs)
  deletes `ValueCodec`, keeps the runtime `Value` conversion surface on `ValueSurfaceEncode` and `ValueSurfaceDecode`, and moves the generic container helper cluster onto those split traits instead of a single composite owner
- [/home/adam/projects/icydb/crates/icydb-core/src/lib.rs](/home/adam/projects/icydb/crates/icydb-core/src/lib.rs)
  [/home/adam/projects/icydb/crates/icydb/src/lib.rs](/home/adam/projects/icydb/crates/icydb/src/lib.rs)
  trim the hidden macro exports so generated code no longer treats `ValueCodec` as a surface that exists
- [/home/adam/projects/icydb/crates/icydb-schema-derive/src/trait_kind.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/trait_kind.rs)
  [/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs)
  [/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/from.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/from.rs)
  retarget generated value-surface code so it emits and consumes `ValueSurfaceEncode` / `ValueSurfaceDecode` directly instead of naming the deleted composite trait

That deletion matters because it makes the remaining authority split honest:
- runtime lowering into `Value` is encode-side only
- runtime reconstruction from `Value` is decode-side only
- persisted structured-field ownership no longer needs `ValueCodec`
- persisted-row `ByKind` ownership no longer needs `ValueCodec`

The second half of this patch is the byte-authority cleanup. The shared owner is:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/typed.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/typed.rs)
  adds one typed payload authority for the semantic leaf families that were still duplicated across sibling structural lanes

That shared typed owner now owns the payload semantics for:
- `Account`
- `Principal`
- `Subaccount`
- `Timestamp`
- `Ulid`
- `Float32`
- `Float64`
- `Int128`
- `Nat128`
- `Date`
- `Duration`
- `Decimal`

The sibling lanes were then retargeted onto that owner:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/scalar.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/scalar.rs)
  now reuses the shared typed payload decode for float, 128-bit integer, and ULID scalar families instead of keeping parallel local payload reconstruction
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/leaf.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/leaf.rs)
  now reuses the shared date, decimal, and duration payload authority instead of carrying its own second copy of that normalization logic
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs)
  now reuses the same typed payload conversions for account, principal, subaccount, timestamp, ULID, float, 128-bit integer, date, duration, and decimal instead of carrying a separate value-storage-local semantic copy
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key/scalar/encode.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key/scalar/encode.rs)
  [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key/scalar/bytes.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key/scalar/bytes.rs)
  [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key/scalar/primitive.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key/scalar/primitive.rs)
  now reuse the shared typed payload owner for the overlapping storage-key leaf families instead of keeping a separate local semantic reconstruction path

That is the important architectural result of `0.127.3`:
- framing ownership stays separate
  - scalar structural binary framing still belongs to the scalar lane
  - value-storage envelope framing still belongs to the value-storage lane
  - storage-key framing still belongs to the storage-key lane
- but typed semantic payload reconstruction no longer exists in three or four parallel copies

So after `0.127.3`, the split is much cleaner:
- `ValueSurfaceEncode` / `ValueSurfaceDecode` are runtime-only
- persisted structured-field codecs stay on `PersistedStructuredFieldCodec`
- persisted-row `ByKind` codecs stay on `PersistedByKindCodec`
- shared typed leaf payload semantics now have one owner even though framing still has multiple owners

Validation for this follow-through stayed focused on the affected boundaries:
- `cargo fmt --all`
- `cargo clippy -p icydb-core --all-targets -- -D warnings`
- `cargo test -p icydb-core persisted_row -- --nocapture`
- `cargo test -p icydb-core structural_field -- --nocapture`

## 0.127.2

`0.127.2` finishes the next persisted-row `ByKind` cleanup by deleting the remaining blanket `ValueCodec` fallback from that lane. After `0.127.1`, the persisted-row contract already flowed through `PersistedByKindCodec`, but one blanket implementation still let any `T: ValueCodec + 'static` re-enter as the universal fallback. This patch removes that blanket and replaces it with explicit owners.

The main boundary change is in:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs)
  deletes the blanket `impl<T> PersistedByKindCodec for T where T: ValueCodec + 'static` and replaces it with explicit implementations for direct leaf families, explicit value-backed implementations for the semantically heavier residue, and explicit wrapper implementations for the container family

That new shape is intentional:
- direct leaves now own their persisted-row `ByKind` path directly
- semantically heavier leaves that still use runtime conversion now do so through named impls instead of one global escape hatch
- wrapper families now declare their persisted-row ownership explicitly instead of inheriting it accidentally from a blanket fallback

The direct explicit leaf ownership now covers:
- `bool`
- `String`
- `Blob`
- `Float32`
- `Float64`
- signed int family
- unsigned int family
- `Timestamp`
- `Principal`
- `Subaccount`
- `Ulid`
- `Unit`

The explicit residue that still routes through runtime value conversion is now narrow and visible:
- `Account`
- `Date`
- `Decimal`
- `Duration`
- `Int128`
- `Nat128`
- `Int`
- `Nat`

And the container or wrapper family now has explicit persisted-row owners too:
- `Box<T>` where `T: PersistedByKindCodec`
- `Option<T>` where `T: PersistedByKindCodec`
- `Vec<T>` where `Vec<T>: ValueCodec`
- `BTreeSet<T>` where `BTreeSet<T>: ValueCodec`
- `BTreeMap<K, V>` where `BTreeMap<K, V>: ValueCodec`

That matters because the old blanket fallback hid two different kinds of ownership:
- real persisted leaf ownership
- temporary runtime-value reuse for complex residue and wrapper families

After this patch, those are no longer the same thing. The persisted-row `ByKind` boundary is explicit about which types are direct, which types still deliberately reuse runtime conversion, and which wrappers own their own adapter edge.

The supporting validation added in this follow-through is in:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs)
  keeps `direct_persisted_by_kind_leaf_codecs_cover_tier_one_family` as the explicit proof of direct first-path coverage and adds the necessary explicit test-owner implementation for a structured by-kind test type so persisted-row tests no longer rely on the deleted blanket fallback
- [/home/adam/projects/icydb/testing/macro-tests/src/test/structured_field_value.rs](/home/adam/projects/icydb/testing/macro-tests/src/test/structured_field_value.rs)
  keeps the generated structured-wrapper path green after the fallback removal, proving that the persisted custom-field lane and the persisted-row by-kind lane are now both explicit enough to survive without the old blanket bridge

So after `0.127.2`, the remaining `ValueCodec` use in the persisted-row `ByKind` lane is no longer a blanket fallback. It is explicit residue in named implementations, which is the right shape for any later attempt to shrink or delete it further.

## 0.127.1

`0.127.1` follows the direct structured-field codec cut by tightening the persisted-row `ByKind` lane itself. The core change is that persisted-row helpers no longer advertise `ValueCodec` as the `ByKind` contract directly, and the obvious scalar/storage-key leaf family now takes a direct persisted path before the generic runtime-value fallback is allowed to re-enter.

The first ownership cleanup in this patch is contractual:
- [/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs)
  adds `PersistedByKindCodec` as the schema-selected persisted-row authority for `ByKind` fields
- [/home/adam/projects/icydb/crates/icydb/src/traits/mod.rs](/home/adam/projects/icydb/crates/icydb/src/traits/mod.rs)
  keeps `PersistedFieldMetaCodec` off the stable facade trait surface, so persisted-row implementation traits do not leak back into the user-facing facade contract
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs)
  retargets the persisted-row `ByKind` helpers onto `PersistedByKindCodec` instead of binding row helpers directly to `ValueCodec`

That still leaves one localized blanket implementation in place:
- `impl<T> PersistedByKindCodec for T where T: ValueCodec + 'static`

But after this patch, that blanket impl is no longer “always use `ValueCodec`.” It now behaves like a narrowing adapter:
- try the direct scalar/storage-key leaf family first
- only if no direct owner matches, fall back to the older `ValueCodec -> Value` route

The direct first-path now lives across:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/scalar.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/scalar.rs)
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/storage_key/mod.rs)
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/mod.rs)
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs)

The direct `ByKind` first-path now covers the obvious leaf family:
- `bool`
- `String`
- `Blob`
- `Float32`
- `Float64`
- signed int family
- unsigned int family
- `Timestamp`
- `Principal`
- `Subaccount`
- `Ulid`
- `Unit`

The patch also reshapes the helper boundary so the direct lane is readable by owner family instead of one long branch ladder:
- scalar fast-path leaves
- signed-int storage-key leaves
- unsigned-int storage-key leaves
- remaining storage-key leaves

That matters because `0.127.1` is not trying to redesign the fallback again. It is trying to shrink the fallback until the remaining residue is honest:
- composite or non-leaf `ByKind` cases
- semantically heavier leaves such as `Account`, `Decimal`, or the 128-bit families
- any other by-kind cases whose persisted ownership still needs a named authority

The validation proving that narrower shape now exists is in:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs)
  adds `direct_persisted_by_kind_leaf_codecs_cover_tier_one_family`, which pins the direct first-path for the full tier-one leaf family instead of leaving that ownership implied by helper shape alone

So after `0.127.1`, the remaining `ValueCodec` use in the persisted-row `ByKind` lane is localized and explicit. It is no longer the public persisted-row contract, and it is no longer carrying the obvious leaf family by default.

## 0.127.0

`0.127.0` starts the direct structured field codec cut by making typed structured custom field persistence a real owner in its own right instead of always routing that lane through `ValueCodec -> Value -> persisted bytes`. This is deliberately narrower than “remove `Value` from persisted rows.” The broader slot reader, patch, canonicalization, and projection seams remain `Value`-based. What changed here is the derive-owned structured custom field path.

The new owner landed in:
- [/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs)
  added `PersistedStructuredFieldCodec` as the typed `structured field <-> persisted custom payload bytes` boundary. This is the new authority for derive-owned custom field persistence.
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs)
  moved persisted custom-slot helpers onto `PersistedStructuredFieldCodec`, added the blanket recursive edge for `Box<T>`, and added direct persisted codecs for the scalar family now used by generated structured wrappers
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs)
  added the direct structural-bytes leaf and container helpers that the new codec path composes without constructing `Value`
- [/home/adam/projects/icydb/crates/icydb-core/src/db/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/mod.rs)
  [/home/adam/projects/icydb/crates/icydb/src/db/mod.rs](/home/adam/projects/icydb/crates/icydb/src/db/mod.rs)
  added the hidden generated-code wrappers for structural text/list/map/enum payload framing and decode-side error mapping, so derive output does not become the owner of byte layout details

The key architectural shift inside `0.127.0` is Option B: reuse structural writer logic, not `Value`.
- container and wrapper `PersistedStructuredFieldCodec` impls no longer construct `Value` just to get back to bytes
- direct scalar and structural payload helpers now encode and decode the existing custom payload format directly
- the storage helper module still owns the byte layout
- derive output still owns only shape mapping
- the typed codec trait still owns typed entry and exit

The direct-bytes lane now covers:
- direct scalar leaf payloads such as `bool`, `String`, `Blob`, `i8/i16/i32/i64`, `u8/u16/u32/u64`, `Float32`, `Float64`, `Principal`, `Subaccount`, `Timestamp`, `Date`, `Duration`, `Ulid`, `Unit`, `Account`, `Decimal`, `Int128`, `Nat128`, `Int`, and `Nat`
- direct structural container payloads for `Vec<T>`, `Option<T>`, `BTreeSet<T>`, and `BTreeMap<K, V>`
- direct generated wrapper payloads for `list`, `set`, `map`, `newtype`, `record`, `tuple`, and `enum`

The derive-owned move landed in:
- [/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs)
  generated wrappers now implement `PersistedStructuredFieldCodec` directly and recurse on nested members and payloads through that trait instead of relying on a primitive-only direct-admission allowlist

That matters because the direct admission rule is now trait-driven rather than “primitive-backed”:
- records recurse through each field’s `PersistedStructuredFieldCodec`
- tuples recurse through each item’s `PersistedStructuredFieldCodec`
- enums recurse through each payload-bearing variant’s `PersistedStructuredFieldCodec`
- list/set/map/newtype wrappers recurse through their nested item or payload type’s `PersistedStructuredFieldCodec`

So the remaining work is much more isolated than it was before:
- the direct structured codec lane is real
- generated wrappers can recurse through it
- nested direct custom payloads are no longer blocked on another derive-side primitive classifier

The validation added in this patch is what proves that claim:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs)
  pins the new owner with a direct custom type that implements `PersistedStructuredFieldCodec` without implementing `ValueCodec`
- [/home/adam/projects/icydb/testing/macro-tests/src/test/enum_payload.rs](/home/adam/projects/icydb/testing/macro-tests/src/test/enum_payload.rs)
  proves the direct enum payload lane for generated unit and primitive-backed enums
- [/home/adam/projects/icydb/testing/macro-tests/src/test/structured_field_value.rs](/home/adam/projects/icydb/testing/macro-tests/src/test/structured_field_value.rs)
  proves recursive mixed nesting on the generated path:
  - record containing generated enum
  - enum payload containing generated record
  - map of generated structured wrappers
  - malformed nested payloads fail closed
  - canonical payload bytes stay stable after decode and re-encode

This does **not** change the broader persisted-row slot contract:
- `UpdatePatch`
- `FieldUpdate`
- `SerializedUpdatePatch`
- slot-reader `get_value`
- slot-reader `required_value_by_contract`
- canonical row readers and structural projection paths

Those seams are still intentionally `Value`-based after `0.127.0`. The point of this patch is narrower: custom structured field persistence no longer has to use `Value` as its mandatory adapter just because the old derive-owned path never had a typed bytes owner of its own.
