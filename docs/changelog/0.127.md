# 0.127.x

## 0.127.0

`0.127.0` starts the direct structured field codec cut by making typed structured custom field persistence a real owner in its own right instead of always routing that lane through `ValueCodec -> Value -> persisted bytes`. This is deliberately narrower than “remove `Value` from persisted rows.” The broader slot reader, patch, canonicalization, and projection seams remain `Value`-based. What changed here is the derive-owned structured custom field path.

The new owner landed in:
- [/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/traits/mod.rs)
  added `PersistedStructuredFieldCodec` as the typed `structured field <-> persisted custom payload bytes` boundary. This is the new authority for derive-owned custom field persistence.
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/codec.rs)
  moved persisted custom-slot helpers onto `PersistedStructuredFieldCodec`, added the blanket recursive edge for `Box<T>`, and added direct persisted codecs for the scalar family now used by generated structured wrappers
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/structural_field/value_storage.rs)
  added the direct structural-bytes leaf and container helpers that the new codec path composes without constructing `Value`
- [/home/adam/projects/icydb/crates/icydb-core/src/db/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/mod.rs)

The key architectural shift inside `0.127.0` is Option B: reuse structural writer logic, not `Value`.
- container and wrapper `PersistedStructuredFieldCodec` impls no longer construct `Value` just to get back to bytes
- direct scalar and structural payload helpers now encode and decode the existing custom payload format directly
- the storage helper module still owns the byte layout
- derive output still owns only shape mapping
- the typed codec trait still owns typed entry and exit

The direct-bytes lane now covers:
- direct scalar leaf payloads such as `bool`, `String`, `Blob`, `i8/i16/i32/i64`, `u8/u16/u32/u64`, `Float32`, `Float64`, `Principal`, `Subaccount`, `Timestamp`, `Date`, `Duration`, `Ulid`, `Unit`, `Account`, `Decimal`, `Int128`, `Nat128`, `Int`, and `Nat`
- direct structural container payloads for `Vec<T>`, `Option<T>`, `BTreeSet<T>`, and `BTreeMap<K, V>`
- direct generated wrapper payloads for `list`, `set`, `map`, `newtype`, `record`, `tuple`, and `enum`

The derive-owned move landed in:
- [/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs](/home/adam/projects/icydb/crates/icydb-schema-derive/src/imp/value_surface.rs)
  generated wrappers now implement `PersistedStructuredFieldCodec` directly and recurse on nested members and payloads through that trait instead of relying on a primitive-only direct-admission allowlist

That matters because the direct admission rule is now trait-driven rather than “primitive-backed”:
- records recurse through each field’s `PersistedStructuredFieldCodec`
- tuples recurse through each item’s `PersistedStructuredFieldCodec`
- enums recurse through each payload-bearing variant’s `PersistedStructuredFieldCodec`
- list/set/map/newtype wrappers recurse through their nested item or payload type’s `PersistedStructuredFieldCodec`

So the remaining work is much more isolated than it was before:
- the direct structured codec lane is real
- generated wrappers can recurse through it
- nested direct custom payloads are no longer blocked on another derive-side primitive classifier

The validation added in this patch is what proves that claim:
- [/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/data/persisted_row/tests.rs)
  pins the new owner with a direct custom type that implements `PersistedStructuredFieldCodec` without implementing `ValueCodec`
- [/home/adam/projects/icydb/testing/macro-tests/src/test/enum_payload.rs](/home/adam/projects/icydb/testing/macro-tests/src/test/enum_payload.rs)
  proves the direct enum payload lane for generated unit and primitive-backed enums
- [/home/adam/projects/icydb/testing/macro-tests/src/test/structured_field_value.rs](/home/adam/projects/icydb/testing/macro-tests/src/test/structured_field_value.rs)
  proves recursive mixed nesting on the generated path:
  - record containing generated enum
  - enum payload containing generated record
  - map of generated structured wrappers
  - malformed nested payloads fail closed
  - canonical payload bytes stay stable after decode and re-encode

This does **not** change the broader persisted-row slot contract:
- `UpdatePatch`
- `FieldUpdate`
- `SerializedUpdatePatch`
- slot-reader `get_value`
- slot-reader `required_value_by_contract`
- canonical row readers and structural projection paths

Those seams are still intentionally `Value`-based after `0.127.0`. The point of this patch is narrower: custom structured field persistence no longer has to use `Value` as its mandatory adapter just because the old derive-owned path never had a typed bytes owner of its own.
