# 0.81 — Bounded Computed ORDER BY Aliases

## 0.81.4 - 2026-04-14 - Normalize Shared Lower Cache Identity

This patch does not widen the shipped SQL surface. It cleans up the shared
lower query-plan cache identity used by fluent and SQL after they converge
below the frontend split, replacing the old manual structural fingerprint walk
with one explicit canonical cache-key type.

What changed:

* introduced one explicit canonical shared lower cache key:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/cache_key.rs`
  * new owner type:
    * `StructuralQueryCacheKey`
  * the shared lower cache now stores:
    * entity path
    * schema fingerprint
    * visibility
    * `StructuralQueryCacheKey`
  * instead of storing an opaque structural `[u8; 16]` digest
* moved semantic cache identity construction out of the old mixed
  “normalize while writing bytes” path in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs`
  * the query-intent boundary now exposes:
    * `structural_cache_key()`
  * the old low-level `xxh3_write_*` and `hash_*` helper family is no longer
    the live owner for shared lower cache identity
* kept SQL upper caches unchanged while aligning the lower cache story with
  them structurally:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * SQL upper caches still use:
    * `SqlCompiledCommandCacheKey`
    * `SqlSelectPlanCacheKey`
  * the lower shared cache now also uses one explicit derived key family
* removed the current predicate `Debug`-string compromise from shared lower
  cache identity:
  * predicate identity now reuses the predicate subsystem's canonical
    fingerprint authority instead of formatting normalized predicates into text
    during cache-key construction
* updated cache-compatibility tests to assert canonical key equality instead of
  the older structural digest equality:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs`
* added direct unit coverage for the new canonical key owner:
  * identical scalar queries normalize to the same shared cache key
  * order-direction differences normalize to different shared cache keys

Why this patch matters:

* the SQL-side cache story was already explicit:
  * derived key structs at the cache boundary
* the remaining complexity hotspot was only the shared lower fluent/structural
  cache identity path
* `0.81.4` makes the important artifact visible as one real type instead of
  burying semantic identity inside a byte encoder
* this lowers maintenance cost before the broader deterministic compiled-query
  reuse work planned in `0.82`

What this patch does not do:

* no new SQL syntax
* no new fluent feature
* no cache-scope widening
* no new cache tier
* no grouped computed ordering or grouped computed projection widening

Verification:

* `cargo test -p icydb-core structural_query_cache_key -- --nocapture`
* `cargo test -p icydb-core direct_bounded_numeric_order_terms_use_the_normal_sql_cache_path -- --nocapture`

## 0.81.0 - 2026-04-14 - Admit Bounded Computed Scalar ORDER BY Aliases

This patch closes one narrow SQL coherence gap: `ORDER BY <alias>` now works
when the alias resolves to one admitted bounded scalar computed projection
expression. That includes the current scalar arithmetic family and bounded
`ROUND(..., scale)` projection family, while still keeping direct expression
ordering and grouped computed ordering fail-closed.

Examples now admitted:

```sql
SELECT age + 1 AS next_age FROM Customer ORDER BY next_age;
SELECT ROUND(age / 3, 2) AS rounded_age FROM Customer ORDER BY rounded_age DESC;
SELECT dexterity + charisma AS total FROM Character ORDER BY total;
```

What changed:

* widened SQL alias normalization so admitted arithmetic and `ROUND(...)`
  projection aliases canonicalize onto the same internal order-expression form
  as their equivalent non-aliased semantic expression:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/normalize.rs`
* extended the canonical internal order-expression parser/renderer beyond the
  old `LOWER(field)` / `UPPER(field)`-only family so it now admits:
  * plain field terms
  * bounded arithmetic terms with one field on the left and one field or
    literal on the right
  * bounded `ROUND(expr, scale)` over that same admitted scalar family
  * implementation:
    * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/ast.rs`
    * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/order_term.rs`
* added one shared lowering helper so projection lowering and alias-order
  normalization both derive the same canonical scalar `Expr` tree instead of
  carrying separate projection-only and order-only shapes:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/projection.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/mod.rs`
* switched order validation from the old single-field text-wrapper assumption
  to expression type inference:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/order.rs`
  * arithmetic and `ROUND(...)` alias order terms are now validated as
    orderable numeric expressions instead of being rejected as unknown fields
* kept the old strict planner policy only for the existing casefold
  expression-order family that still depends on matching index-backed access:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/validate/core.rs`
  * `LOWER(field)` / `UPPER(field)` still preserve the current
    index-satisfied-expression-order contract
  * admitted numeric alias ordering now flows through normal materialized sort
    instead of being blocked by that casefold-specific policy gate
* generalized resolved-order lowering so supported order expressions no longer
  assume one `FunctionCall(field)` wrapper:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs`
  * the resolved-order path now walks field leaves through bounded arithmetic
    and `ROUND(...)` expressions before compiling one expression-backed order
    source
* updated cursor-boundary validation so order-expression continuation slots no
  longer assume every supported order term maps back to one single source
  field type:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/cursor/boundary.rs`
  * the continuation boundary path now derives expected slot shape from the
    inferred order-expression type where needed
* added parser/lowering/session/explain/canister coverage for the new admitted
  alias family:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/order_term.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_projection.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs`
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/sql_canister.rs`

What this patch does not do:

* no direct `ORDER BY age + 1`
* no direct `ORDER BY ROUND(age / 3, 2)`
* no grouped computed projection widening
* no grouped computed alias ordering
* no aggregate-derived computed alias ordering
* no new cache tier or side cache for order-only semantics

Why this is the right cut:

* the missing piece was alias normalization plus canonical order-expression
  ownership, not a new executor path
* keeping this as alias-only support preserves the bounded SQL surface instead
  of silently turning it into generic direct-expression ordering
* the compiled SQL cache line stays coherent because admitted alias ordering
  canonicalizes onto the same structural order/query representation before
  cache lookup or insertion, instead of creating a second order-only semantic
  shape
* planner/runtime stay alias-neutral after normalization; execution only sees
  the canonical expression-backed order source
