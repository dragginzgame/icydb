# 0.141.x

## 0.141.0

`0.141.0` reorganizes SQL projection materialization so projection execution,
DISTINCT handling, metrics, prepared projection shape, row transport, and final
structural page construction each have a narrow owner. SQL projection output,
`DISTINCT` semantics, cursor behavior, diagnostics counters, public response
shapes, and persisted formats remain unchanged.

### 🔧 Changed

- split `db::executor::projection::materialize` from one large module into
  focused owners:
  - `execute` owns retained-slot and data-row projection loops
  - `plan` owns prepared projection shape and validation contracts
  - `metrics` owns projection materialization counters
  - `distinct` owns projected-row DISTINCT key storage and bounded windowing
  - `row_view` owns local projected-row transport before final materialization
  - `structural` owns `StructuralCursorPage` dispatch and final row wrapping
- replaced the older internal entrypoint names with the shorter executor
  boundary names `project` and `project_distinct`
- introduced a local `RowView` transport so projection execution can carry
  projected rows without committing every path to the final `Vec<Vec<Value>>`
  response backing shape immediately
- kept the structural module as the only place that converts local row views
  into `MaterializedProjectionRows`
- retained direct retained-slot projection, direct data-row field copies, and
  model-identity projection paths as prepared projection shape decisions

### ⚙️ Performance

- keeps SQL `DISTINCT` duplicate checks on borrowed projected rows for common
  non-map values, avoiding a full projected-row clone before duplicate
  rejection
- preserves the owned canonicalization fallback for map-valued projected rows
  so canonical map validation and error behavior remain unchanged
- keeps direct projection paths moving or copying only the requested field slots
  instead of routing every simple projection through scalar expression fallback
- records the same projection materialization metrics through the narrower
  metrics owner

### 🧹 Cleanup

- removes the projection materialization catch-all module shape and makes the
  execution path easier to audit from prepared shape, through row projection,
  through DISTINCT, to structural page output
- keeps tests and callsites on the module-root projection boundary instead of
  importing deep materialization helpers directly
- documents `RowView::Borrowed` as the next zero-copy extension point while the
  current production paths continue to use owned rows where ownership is still
  required

### 🧭 Retained behavior

- SQL projection labels and row ordering are unchanged
- SQL `DISTINCT` still preserves canonical equality behavior, including nested
  decimal normalization and map-value validation
- cursor paging and bounded DISTINCT windowing remain unchanged
- response DTOs still receive the same owned row matrix at the structural
  boundary
