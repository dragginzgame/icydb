# 0.141.x

## 0.141.6

`0.141.6` introduces the parser and planner AST preparation for the next minor
version's table/field/subfield projection work. SQL expressions can now retain
a nested `FieldPath` leaf through parsing, normalization, lowering,
fingerprinting, labels, and structural analysis while execution still rejects or
falls back at the existing unsupported boundaries. This keeps shipped query
behavior unchanged and gives the next minor a typed path shape to wire into
projection and predicate execution.

### ЁЯФз Changed

- added planner-owned `FieldPath` expression leaves rooted at a top-level field
  with a nested path tail
- added SQL parser-owned `FieldPath` leaves for dotted field references in
  projection, aggregate input, function argument, arithmetic, and order
  expression parsing surfaces
- normalized entity-qualified paths such as `users.name` back to direct fields,
  while preserving true nested paths such as `profile.name` and
  `users.profile.name`
- updated expression traversal, hashing, labels, analysis, and compile
  admission checks so nested paths are explicit and fail closed until execution
  support lands

### ЁЯзн Retained behavior

- direct `entity.field` and table-alias field references still normalize to
  normal top-level fields
- execution does not yet evaluate nested field paths
- scalar projection, grouped projection, predicate compilation, aggregate
  lowering, and explain output keep their existing behavior for supported query
  shapes

### ЁЯзк Validation

- added a parser regression for a nested projection expression
- ran `cargo check -p icydb-core`
- ran
  `cargo test -p icydb-core parse_select_statement_with_field_path_projection_expr -- --nocapture`

## 0.141.5

`0.141.5` extends the RowView borrowed-execution cleanup to retained slot-row
projection materialization. Non-DISTINCT retained slot pages now visit borrowed
`RowView` instances backed by one reusable projection buffer before copying
into `MaterializedProjectionRows`. SQL projection output, row ordering,
retained-slot missing-field behavior, DISTINCT row retention, structural page
payloads, and retained-slot storage remain unchanged.

### ЁЯФз Changed

- added retained slot-row projection visitors for the non-DISTINCT structural
  materialization path
- routed direct retained-slot projection and scalar retained-slot projection
  through reusable output buffers before final structural materialization
- reduced the per-output-row temporary allocation between `RetainedSlotRow`
  projection and final `MaterializedProjectionRows` construction
- kept `project_slot_row` as the owned single-row projector for DISTINCT and
  any future row-retaining callers
- documented the 0.141.5 retained-slot borrowing plan, with retained-slot view
  abstractions and scan-level borrowed producers left out of this patch

### ЁЯзн Retained behavior

- retained slot-row projection still consumes `RetainedSlotRow` pages from
  `StructuralCursorPage::SlotRows`
- direct retained-slot projection still uses `take_slot` semantics, including
  duplicate-slot missing-field failures
- retained-slot scalar projection still evaluates through the compiled scalar
  projection evaluator and borrowed slot references
- SQL `DISTINCT` retained-slot projection still uses owned `RowView` values
  because accepted rows may outlive one projection callback

### ЁЯзк Validation

- added projection materialization tests proving direct and wide scalar
  retained slot-row execution visit borrowed `RowView` instances
- added a retained direct-slot missing-field regression for duplicate projected
  slots
- ran `cargo fmt --all`
- ran `cargo check`

  `cargo test -p icydb-core slot_row_materialization -- --nocapture`

## 0.141.4

`0.141.4` begins the RowView borrowed-execution addendum for structural
projection materialization. Identity and non-identity data-row projection now
visit each row as `RowView::Borrowed` from reusable row buffers, then copy into
owned `MaterializedProjectionRows` only at the structural result boundary. SQL
projection output, row ordering, slot-access metrics, DISTINCT behavior, scalar
projection evaluation, retained-slot projection, and persisted row formats
remain unchanged.

### ЁЯФз Changed

- replaced identity data-row `RowView::Owned` staging with a borrowed row-view
  visitor backed by one reusable decode buffer
- added a non-identity data-row visitor so direct field projection and scalar
  fallback projection can borrow each projected row from one reusable output
  buffer before structural materialization
- added `RowLayout::decode_full_value_row_into` so full-row identity decode can
  reuse caller-owned storage across rows
- kept the final `Vec<Vec<Value>>` allocation inside
  `MaterializedProjectionRows`, preserving the existing executor/session
  handoff shape
- removed the now-unused owned full-row decode helper
- updated the RowView borrowed-execution addendum with the 0.141.4 data-row
  progress and retained-slot follow-up boundary

### ЁЯзн Retained behavior

- identity projections still emit canonical model-field order
- structural projection materialization still returns the same owned value-row
  payload to SQL/session callers
- retained slot-row projection and DISTINCT projection paths still use their
  existing owned row-retention shapes
- non-identity direct field projection and scalar fallback projection preserve
  their slot-access metrics, expression evaluation, and row ordering
- dense full-row decode still validates declared-field presence and reports the
  same persisted-row corruption error if a required declared field is missing

### ЁЯзк Validation

- added projection materialization tests proving identity, direct data-row, and
  wide scalar fallback execution visit borrowed `RowView` instances
- ran `cargo fmt --all`
- ran `cargo check`
- ran `cargo clippy -- -D warnings`

## 0.141.3

`0.141.3` keeps aggregate builder behavior, public builder functions, explain
output, executor request mapping, and tests unchanged while splitting the
aggregate builder into expression, explain, and strategy owners. Request enums
now own aggregate identity helpers and numeric DISTINCT classification so
strategy implementations no longer repeat those small semantic mappings.

### ЁЯФз Changed

- split `db::query::builder::aggregate` into focused owners:
  - `expr` owns `AggregateExpr` and the public aggregate builder functions
  - `explain` owns aggregate explain projection contracts and descriptors
  - `strategy` owns existing-row, numeric-field, order-sensitive, projection,
    and scalar terminal strategy preparation
- shortened aggregate-builder-internal strategy and request names after the
  split so module ownership supplies the redundant fluent/prepared context
- standardized strategy request fields and test helpers around `request`
  terminology instead of the older `runtime_request` wording
- moved existing-row, numeric-field, and order-sensitive aggregate identity
  lookup onto their request enums
- made numeric request DISTINCT handling route through one
  `NumericFieldRequest::is_distinct` helper
- moved projection terminal/output explain labels and executor request mapping
  onto `ProjectionRequest`

### ЁЯзн Retained behavior

- public aggregate builder functions such as `count`, `sum`, `avg`, `min_by`,
  and `max_by` keep the same signatures and output shapes
- fluent aggregate explain output remains unchanged
- existing-row, numeric, scalar, order-sensitive, and projection terminal
  executor request mapping remains unchanged
- tests continue to cover request shape and explain projection stability after
  the split

### ЁЯзк Validation

- ran `cargo fmt --all`
- ran `cargo check`
- ran `cargo clippy -- -D warnings`
- ran `cargo test -p icydb-core builder::aggregate::tests -- --nocapture`

## 0.141.2

`0.141.2` finishes the aggregate state cleanup, adds focused grouped extrema
guard coverage, fixes fixed-scale decimal write normalization, and refreshes the
Canic workspace dependencies. Grouped MIN/MAX results, field-target comparison,
expression-value ingestion, key-only ordering, early termination, SQL output,
persisted formats, and corruption checks for already-stored decimal scale drift
remain unchanged.

### ЁЯФз Changed

- shared row-backed scalar expression evaluation between grouped aggregate
  inputs and grouped aggregate `FILTER` expressions while preserving their
  separate invariant labels
- moved SUM/AVG grouped reducer details behind a local `SumLikeKind` instead of
  attaching executor behavior to `AggregateKind`
- routed scalar MIN/MAX terminal updates through the same local extrema
  direction policy used by grouped MIN/MAX
- split `contracts::state` into focused `control`, `distinct`, `factory`,
  `grouped`, `reducer`, and `scalar` owners without changing the
  `contracts::state` import surface
- refreshed workspace Canic dependencies from `0.29.2` to `0.29.4`

### ЁЯРЫ Fixed

- normalized authored decimal writes to the declared field scale before
  persisted-row encoding, so lower-scale inputs such as `140` can be stored in a
  scale-3 decimal field as `140.000`
- kept persisted-row invariant validation strict for already-stored bytes, so
  raw decimal scale drift is still reported as corruption instead of being
  silently repaired

### ЁЯзк Validation

- verified that slot-aware field comparison stays confined to field-target
  extrema paths
- verified that expression MIN/MAX still use generic reducer ingestion through
  `ingest_min_value` and `ingest_max_value`
- added typed, structural, and direct persisted-row codec coverage for decimal
  write normalization while retaining the persisted-row scale-drift rejection
  test
- added a grouped key-path early-break matrix test covering:
  - `MIN + ASC` stops early
  - `MIN + DESC` keeps scanning
  - `MAX + DESC` stops early
  - `MAX + ASC` keeps scanning

## 0.141.1

`0.141.1` keeps grouped aggregate behavior unchanged while consolidating the
terminal grouped aggregate executor. Input resolution, `DISTINCT` admission,
reducer dispatch, and MIN/MAX extrema handling now each have one owner inside
`GroupedTerminalAggregateState`, reducing repeated branching without changing
SQL results, NULL handling, ordering, budgets, error taxonomy, or persisted
formats.

### ЁЯФз Changed

- added one grouped aggregate input resolver so field-target aggregates,
  expression-backed aggregates, and NULL filtering share the same internal
  boundary
- centralized grouped `DISTINCT` admission so value-based deduplication and
  key-based deduplication enter the execution context through one helper while
  preserving filter-before-distinct ordering
- removed the local `AggregateReducerClass` indirection and dispatch grouped
  terminal updates directly from `AggregateKind`
- merged the mirrored grouped MIN and MAX update bodies into one extrema helper
  while keeping field-target comparison, expression ingestion, key-only
  ordering, and early termination explicit

### ЁЯзн Retained behavior

- `COUNT(*)` still counts every admitted row, while `COUNT(field)` and
  `COUNT(expr)` still skip NULL
- grouped `DISTINCT` still skips NULL values before canonical-key admission and
  does not spend distinct budget on skipped NULLs
- field-target MIN/MAX still use slot-aware comparison through
  `compare_orderable_field_values_with_slot`
- expression MIN/MAX still use the generic value reducer ingestion path
- MIN over ascending input and MAX over descending input still stop early
- SUM/AVG numeric coercion, grouped reducer outputs, and invariant error labels
  remain unchanged

## 0.141.0

`0.141.0` reorganizes SQL projection materialization so projection execution,
DISTINCT handling, metrics, prepared projection shape, row transport, and final
structural page construction each have a narrow owner. SQL projection output,
`DISTINCT` semantics, cursor behavior, diagnostics counters, public response
shapes, and persisted formats remain unchanged.

### ЁЯФз Changed

- split `db::executor::projection::materialize` from one large module into
  focused owners:
  - `execute` owns retained-slot and data-row projection loops
  - `plan` owns prepared projection shape and validation contracts
  - `metrics` owns projection materialization counters
  - `distinct` owns projected-row DISTINCT key storage and bounded windowing
  - `row_view` owns local projected-row transport before final materialization
  - `structural` owns `StructuralCursorPage` dispatch and final row wrapping
- replaced the older internal entrypoint names with the shorter executor
  boundary names `project` and `project_distinct`
- introduced a local `RowView` transport so projection execution can carry
  projected rows without committing every path to the final `Vec<Vec<Value>>`
  response backing shape immediately
- kept the structural module as the only place that converts local row views
  into `MaterializedProjectionRows`
- retained direct retained-slot projection, direct data-row field copies, and
  model-identity projection paths as prepared projection shape decisions

### тЪЩя╕П Performance

- keeps SQL `DISTINCT` duplicate checks on borrowed projected rows for common
  non-map values, avoiding a full projected-row clone before duplicate
  rejection
- preserves the owned canonicalization fallback for map-valued projected rows
  so canonical map validation and error behavior remain unchanged
- keeps direct projection paths moving or copying only the requested field slots
  instead of routing every simple projection through scalar expression fallback
- records the same projection materialization metrics through the narrower
  metrics owner

### ЁЯз╣ Cleanup

- removes the projection materialization catch-all module shape and makes the
  execution path easier to audit from prepared shape, through row projection,
  through DISTINCT, to structural page output
- keeps tests and callsites on the module-root projection boundary instead of
  importing deep materialization helpers directly
- documents `RowView::Borrowed` as the next zero-copy extension point while the
  current production paths continue to use owned rows where ownership is still
  required

### ЁЯзн Retained behavior

- SQL projection labels and row ordering are unchanged
- SQL `DISTINCT` still preserves canonical equality behavior, including nested
  decimal normalization and map-value validation
- cursor paging and bounded DISTINCT windowing remain unchanged
- response DTOs still receive the same owned row matrix at the structural
  boundary
