# IcyDB 0.199 Changelog

## 0.199.1

### 📝 Summary

- Continues the 0.199 technical-debt audit line with a focused direct-query
  boundary cleanup.
- Adds hidden diagnostics-only fluent attribution terminals for load and
  partial-window queries.
- Updates the SQL perf audit canister so fluent scenario measurement no longer
  needs raw `Query<E>` extraction.
- Keeps public read behavior, public read admission, cursor tokens, persisted
  formats, generated SQL authority, and normal endpoint APIs unchanged.

### 🔧 Changed

- `FluentLoadQuery` and `PartialWindowLoadQuery` now expose hidden
  diagnostics-only `execute_with_attribution()` methods in both `icydb-core`
  and the public `icydb` facade.
- The SQL perf audit canister measures fluent scenarios through those hidden
  fluent terminals instead of calling direct session attribution helpers with
  extracted `Query<E>` values.
- The public facade guide no longer provides copyable direct-query execution
  examples and points diagnostics tooling toward fluent attribution terminals
  when starting from a fluent wrapper.

### 📚 Documentation

- Adds a 0.199 fluent-attribution boundary note.
- Updates the 0.199 boundary matrix and direct-query boundary trace to mark
  direct `Query<E>` execution as a reduced P3 internal/generated surface after
  fluent perf attribution moved back onto the fluent boundary.

### 🧪 Validation

- `cargo fmt --all`
- `cargo fmt --all --check`
- `cargo check -p canister_audit_sql_perf`
- `cargo check -p icydb-core --all-features`
- `cargo check -p icydb --all-features`
- `cargo clippy -p icydb-core --all-features --all-targets -- -D warnings`
- `cargo clippy -p icydb --all-features --all-targets -- -D warnings`
- `cargo clippy -p canister_audit_sql_perf --all-targets -- -D warnings`
- `cargo test -p icydb --lib --all-features`
- `git diff --check`

### 📊 Performance

- No runtime performance improvement is claimed.
- Full deterministic performance matrices were not rerun.
- The change affects the measurement boundary for fluent perf scenarios, not
  query execution semantics.

### 📊 Complexity Delta

- Public API complexity stays neutral because the new attribution terminals
  are diagnostics-gated and hidden.
- Direct-query boundary complexity decreases slightly because fluent perf
  measurement no longer requires raw fluent `.query()` extraction.
- Implementation complexity stays low: the slice adds small pass-through
  attribution terminals and removes repeated direct-query attribution call
  sites from the audit canister.

## 0.199.0

### 📝 Summary

- Starts the 0.199 technical-debt audit line after the 0.197 optimizer and
  0.198 read-intent hard cuts.
- Adds a boundary matrix for representative read/query shapes so future cleanup
  work starts from evidence rather than broad refactor pressure.
- Normalizes the 0.198 and 0.199 design directories to descriptive names.
- Records focused evidence notes for exact-key proof ownership, direct-query
  facade boundaries, partial-window public docs, read-admission invariant
  checks, diagnostic vocabulary, and exact aggregate helper wiring.
- Keeps public query behavior, read admission, cursor tokens, persisted formats,
  and generated SQL boundaries unchanged.

### 🔧 Changed

- Direct fluent `.query()` extraction remains available for internal/generated
  tooling but is hidden from normal public API documentation.
- Planner-local selected primary-key proof logic now has a named projection
  helper, reducing duplication in the predicate-stripping path without changing
  selected access semantics.
- Read-admission contract examples now use `page(limit)` for public-list fixes
  instead of teaching `partial_window(...).execute_rows()` as a general
  replacement for removed public load `.limit(...)`.
- Read-admission invariant checks still guard section anchors, API tokens,
  diagnostic enum parity, budgets, and forbidden public policy surfaces, but no
  longer require exact explanatory guide sentences.
- Exact aggregate EXPLAIN helpers now share one private read-intent attribution
  path for `ReadIntentKind::ExactAggregate`.

### 📚 Documentation

- Adds 0.199 audit artifacts for:
  - the read/query boundary matrix;
  - exact-key proof ownership;
  - direct-query facade isolation;
  - partial-window public-doc cleanup;
  - read-admission invariant hardening;
  - diagnostic vocabulary ownership;
  - exact aggregate helper DRY cleanup.
- Updates the 0.198 closeout artifact links to the renamed 0.198 design
  directory.
- Keeps historical 0.198 design feedback intact under the renamed design
  directory.

### 🧪 Validation

- `cargo fmt --all`
- focused exact-aggregate EXPLAIN read-intent test
- focused exact/read-intent raw-limit rejection tests
- full focused `read_admission` test filter
- `cargo clippy -p icydb-core --all-features --all-targets -- -D warnings`
- `bash -n scripts/ci/check-read-admission-invariants.sh`

- `cargo test -p icydb --lib --all-features`
- `cargo check -p canister_audit_sql_perf`
- `git diff --check`

### 📊 Performance

- No runtime performance improvement is claimed.
- Full deterministic performance matrices were not rerun.
- Wasm-size delta was not measured.

### 📊 Complexity Delta

- Public API complexity stays neutral; no new public read API was added.
- Documentation complexity increases intentionally through audit artifacts that
  make owner boundaries explicit.
- Implementation complexity decreases slightly in targeted places by removing
  duplicate exact-key proof and exact-aggregate EXPLAIN wiring.
- The read-admission invariant script becomes less brittle because it now
  checks broad concepts and structured facts rather than exact guide prose.
