# IcyDB 0.52

This document aggregates detailed release notes for the `0.52.x` minor line.

## Next Patch (Unreleased) - 2026-03-12 - Grouped and Global SQL Aggregate Paths

### Summary

The next `0.52` patch expands reduced SQL execution with explicit grouped and
global aggregate SQL entrypoints, while tightening reduced-parser fail-closed
boundaries for unsupported aliasing/function forms.

### Changed

- Added grouped SQL session entrypoints in core and facade:
  - core: `DbSession::execute_sql_grouped<E>(sql, cursor_token)`
  - facade: `db::DbSession::execute_sql_grouped<E>(sql, cursor_token)`
- Added global aggregate SQL session entrypoints in core and facade:
  - core: `DbSession::execute_sql_aggregate<E>(sql)`
  - facade: `db::DbSession::execute_sql_aggregate<E>(sql)`
  - result contract: scalar `Value` (`Uint` for `COUNT(*)`, `COUNT(<field>)`,
    `Decimal`/`Null` for `SUM`/`AVG`, field-typed value or `Null` for
    `MIN`/`MAX`)
- Extended reduced SQL lowering to accept one constrained grouped aggregate
  shape:
  - projection list must declare grouped fields first
  - grouped fields must match `GROUP BY` fields
  - aggregate terminals follow grouped fields and lower via existing aggregate
    intent (`COUNT`, `SUM`, `AVG`, `MIN`, `MAX`)
- Added constrained global aggregate lowering for one-terminal aggregate
  statements (no `GROUP BY`, no `DISTINCT`):
  - `COUNT(*)`
  - `COUNT(<field>)`
  - `SUM(<field>)`
  - `AVG(<field>)`
  - `MIN(<field>)`
  - `MAX(<field>)`
- Kept reduced fail-closed boundaries explicit:
  - scalar/projection SQL surfaces (`execute_sql`, `execute_sql_projection`)
    keep global aggregate projection lowering-gated
  - scalar `DISTINCT` remains policy-constrained:
    - executable for `SELECT DISTINCT *`
    - executable for `SELECT DISTINCT <field_list>` only when projection
      includes primary key
    - non-primary-key-only distinct projection shapes remain fail-closed
  - grouped projection/group-by mismatch remains lowering-gated
    (`UnsupportedSelectGroupBy`)
  - unsupported global aggregate terminals (mixed/grouped forms) fail closed as
    unsupported SQL execution shapes
  - scalar SQL execution surfaces (`execute_sql`, `execute_sql_projection`)
    reject grouped SQL intent with `IntentError::GroupedRequiresExecuteGrouped`
- Added SQL grouped coverage:
  - grouped aggregate lowering tests in `db::sql::lowering`
  - session tests for grouped SQL execution, grouped/scalar API boundary
    rejections, and unsupported grouped projection mismatch
- Added SQL global-aggregate coverage:
  - lowering tests for accepted/rejected global aggregate statement shapes
  - session tests for `COUNT(*)`, `COUNT(field)`, `SUM`, `AVG`, `MIN`, `MAX`,
    window semantics, and unknown aggregate-target field rejection
  - session tests for global aggregate `EXPLAIN` plan/execution/json surfaces,
    plus unknown-target fail-closed behavior
- Added constrained scalar DISTINCT coverage:
  - lowering tests for allowed (`DISTINCT *`, PK-inclusive field lists) and
    rejected (non-PK-only projections) distinct shapes
  - session tests for executable constrained distinct paths, explain
    plan/execution/json surfaces, and fail-closed rejection of unsupported
    distinct projection shapes
- Added reduced SQL qualified-identifier normalization coverage:
  - schema-qualified entity names are accepted when the trailing entity segment
    matches the requested model
  - table-qualified field references (`Entity.field`,
    `schema.Entity.field`) normalize to canonical planner field names across
    scalar projection, predicate/order terms, and aggregate target lowering
  - lowering/session tests lock qualified vs unqualified intent parity for
    scalar and global aggregate SQL surfaces
- Added explain-surface parity locks for qualified identifiers:
  - plan/execution/json explain tests assert qualified and unqualified SQL forms
    render identical explain outputs after normalization
  - parser tests lock qualified identifier parsing for `SELECT` and `EXPLAIN`
    statements, plus fail-closed alias rejection on schema-qualified entities
- Extended qualified-identifier parity locks to grouped SQL:
  - grouped lowering/session tests now cover qualified grouped execution
    (`GROUP BY`/`ORDER BY`/predicate/projection with qualified fields)
  - grouped `EXPLAIN` parity is locked across plan/execution/json surfaces for
    qualified vs unqualified equivalent statements
- Added parser unsupported-feature taxonomy lock coverage:
  - one parser test matrix now asserts exact `UnsupportedFeature` labels across
    all reduced out-of-scope grammar branches (`JOIN`, `WITH`,
    `UNION/INTERSECT/EXCEPT`, `UPDATE`, `HAVING`, alias forms,
    multi-statement input, quoted identifiers, unsupported function namespace,
    aggregate `DISTINCT` qualifiers, and table aliases)
- Added session propagation locks for parser unsupported-feature labels:
  - all SQL session entrypoints now preserve parser labels in structured query
    execution error detail for unsupported SQL features (`query_from_sql`,
    `execute_sql`, `execute_sql_projection`, `execute_sql_grouped`,
    `execute_sql_aggregate`, `explain_sql`)
  - session tests assert typed detail parity for representative unsupported
    cases (`JOIN`, quoted identifiers, table aliases)
- Added facade-session propagation locks for parser unsupported-feature labels:
  - facade SQL entrypoints now assert one runtime contract for unsupported SQL
    feature failures:
    `ErrorKind::Runtime(RuntimeErrorKind::Unsupported)` with
    `ErrorOrigin::Query`
  - coverage includes `query_from_sql`, `execute_sql`,
    `execute_sql_projection`, `execute_sql_grouped`,
    `execute_sql_aggregate`, and `explain_sql`
- Aligned workspace `canic-*` dependency pins to published `0.15.1` crates
  (`canic-cdk`, `canic-memory`, `canic-utils`) and refreshed lockfile
  resolution accordingly.
- Updated SQL subset contract (`docs/contracts/SQL_SUBSET.md`) to document the
  grouped and global aggregate SQL execution contracts in this baseline.
- Hardened parser boundary classification for unsupported reduced SQL forms:
  - explicit unsupported rejection for table aliases (`FROM x alias`,
    `FROM x AS alias`)
  - explicit unsupported rejection for quoted identifiers
  - explicit unsupported rejection for `DISTINCT` aggregate qualifiers
    (`COUNT(DISTINCT field)`)
  - normalization contract now documents case/whitespace/terminator handling
    and single-statement input constraint

## 0.52.1 - 2026-03-12 - SQL Projection Surface and Parity Lock

### Summary

`0.52.1` advances the reduced SQL line from parser/lowering baseline to a
projection-aware SQL execution surface, while keeping aggregate SQL projection
shapes explicitly fail-closed.

### Changed

- Expanded SQL lowering for reduced `SELECT`:
  - direct field-list projections now lower into canonical planner projection
    selection (`ProjectionSelection::Fields`)
  - aggregate projection items in SQL `SELECT` remain lowering-gated in this
    patch (`UnsupportedSelectProjection`)
- Added projection-selection mutation plumbing in query intent:
  - query-intent mutation now owns explicit projection-selection assignment
  - query/model boundaries now expose one explicit field-list selection path
    consumed by SQL lowering
- Added a projection-shaped SQL execution surface in core session:
  - `DbSession::execute_sql_projection<E>(...) -> ProjectionResponse<E>`
  - only `SELECT` is accepted by this API; `DELETE` fails with typed
    unsupported execution error
  - existing `execute_sql(...) -> EntityResponse<E>` remains unchanged for
    backwards compatibility
- Added matching facade (`icydb`) forwarding surface:
  - `db::DbSession::execute_sql_projection<E>(...)`
  - public `db::ProjectionResponse<E>` wrapper over projection-row payloads
  - `ProjectionResponse<E>` now supports `iter()` plus `IntoIterator` for both
    borrowed and owned forms, keeping facade ergonomics aligned with strict
    clippy warning gates
- Added one executor projection entrypoint for SQL/session consumption:
  - `LoadExecutor::execute_projection(...)` executes canonical scalar load
    semantics and materializes projected row payloads in planner declaration
    order
- Added SQL/fluent parity and projection-surface coverage:
  - SQL field-list projection lowers to the same normalized planned query as
    equivalent fluent intent
  - parity lock on equivalent plan fingerprint and executable routing contracts
    for SQL vs fluent projection forms
  - session tests cover:
    - field-list projection output shape via `execute_sql_projection(...)`
    - `SELECT *` projected row order/value shape
    - typed rejection of `DELETE` on projection API
    - existing `execute_sql(...)` entity-shaped compatibility behavior
- Updated the SQL subset contract doc (`docs/contracts/SQL_SUBSET.md`) to make
  the two execution surfaces explicit:
  - entity-shaped SQL execution (`execute_sql`)
  - projection-shaped SQL execution (`execute_sql_projection`)

### Validation

- `cargo fmt --all`
- `cargo check -p icydb-core`
- `cargo check -p icydb`
- `cargo test -p icydb-core db::sql::lowering::tests:: -- --nocapture`
- `cargo test -p icydb-core db::session::tests:: -- --nocapture`

### Line Status

- `0.52.1` closes the scalar SQL field-projection runtime gap for a dedicated
  projection API surface.
- Reduced SQL aggregate projection (`SELECT COUNT(*)`, `SUM(...)`, etc.) remains
  intentionally lowering-gated for follow-up patches in this line.

## 0.52.0 - 2026-03-12 - Reduced SQL Parser Foundation

### Summary

`0.52.0` opens the reduced SQL parser line with a deterministic SQL parsing
foundation under `db::sql`, plus initial SQL-to-query lowering and typed
session entrypoints for the minimum executable subset.

### Changed

- Added a new `db::sql` module boundary and parser scaffolding.
- Added parser-owned reduced SQL statement contracts for:
  - `SELECT`
  - `DELETE`
  - `EXPLAIN` (`PLAN` default, plus `EXECUTION` and `JSON` modes)
- Added reduced parser projection and aggregate contracts:
  - `*`, field projections
  - aggregate projection items: `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`
- Added deterministic lexer and parser normalization behavior:
  - keyword case-insensitive parsing
  - insignificant whitespace tolerance
  - consistent optional trailing semicolon handling
- Added predicate parsing for the reduced subset:
  - compare operators (`=`, `!=`, `<`, `<=`, `>`, `>=`)
  - `IN`, `NOT IN`, `BETWEEN`
  - `IS NULL`, `IS NOT NULL`
  - boolean composition (`AND`, `OR`, `NOT`)
- Added explicit unsupported-feature gates aligned with the `0.52` subset
  contract, including:
  - multi-statement input rejection
  - `INSERT`/`UPDATE` rejection
  - `DELETE ... OFFSET` rejection
  - alias rejection
  - quoted identifier rejection
  - SQL function namespace rejection outside supported aggregate forms
- Added parser unit coverage for:
  - `SELECT` parse shape with predicate/order/window
  - `DELETE` parse shape with constrained limit form
  - `EXPLAIN JSON SELECT ...` wrapping behavior
  - unsupported-feature rejection paths
  - normalization parity (case/whitespace-insensitive parse equivalence)
- Added SQL lowering contracts in `db::sql::lowering`:
  - parsed SQL now lowers into canonical typed `Query<E>` intent for:
    - reduced `SELECT *` load shape
    - constrained `DELETE` shape
    - `EXPLAIN` wrappers around those executable shapes
  - executable SQL subset is intentionally narrow in `0.52.0`:
    - `SELECT *` is executable
    - non-`*` projections are currently rejected at lowering
    - SQL `DISTINCT` and `GROUP BY` are currently rejected at lowering
- Added core session SQL entrypoints:
  - `query_from_sql<E>(...)`
  - `execute_sql<E>(...)`
  - `explain_sql<E>(...)`
- Added facade (`icydb`) session SQL entrypoints forwarding to core:
  - `query_from_sql<E>(...)`
  - `execute_sql<E>(...)`
  - `explain_sql<E>(...)`
- Added SQL lowering coverage for:
  - successful `SELECT *`/`DELETE`/`EXPLAIN EXECUTION` lowering
  - entity mismatch rejection
  - non-`*` projection rejection
- Added session-level reduced SQL execution coverage for:
  - `execute_sql` `SELECT *` window semantics (`ORDER BY`/`LIMIT`/`OFFSET`)
  - constrained `DELETE` execution semantics
  - `query_from_sql` and `explain_sql` statement-kind routing guards

### Validation

- `cargo fmt --all`
- `cargo check -p icydb-core -p icydb`
- `cargo test -p icydb-core db::sql::parser::tests -- --nocapture`
- `cargo test -p icydb-core db::sql:: -- --nocapture`
- `cargo test -p icydb-core db::session::tests:: -- --nocapture`

### Line Status

- `0.52` line is open.
- `0.52.0` now includes parser + initial lowering/session integration for the
  minimum basic query subset.
- Broader SQL surface lowering (field projections, grouped SQL shapes, and
  expanded semantics parity) remains intentionally staged for follow-up patches
  in this line.
