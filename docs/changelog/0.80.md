# 0.80 — Compiled SQL Cache

## 0.80.3 - 2026-04-14 - Admit Bounded Field-to-Field Arithmetic Projection

This patch widens one narrow SQL scalar projection boundary: arithmetic
projection items now accept a plain field identifier on the right-hand side in
addition to numeric literals. That makes queries such as
`SELECT dexterity + charisma AS total FROM character` executable without
opening broader arithmetic expression parsing or changing predicate/order
semantics.

The same patch also fixes one SQL display-contract bug around `ROUND(...)`:
aliased rounded projections now keep the requested fixed decimal scale at the
public SQL result boundary instead of dropping trailing zeros when the outward
column label no longer exposes the original `ROUND(..., scale)` text.

What changed:

* widened the reduced SQL projection parser from `field op literal` to
  `field op (literal | field)` for scalar projection items only:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/model.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/projection.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/mod.rs`
* updated SQL statement alias normalization and select normalization so
  alias-qualified arithmetic projection operands keep the same canonical entity
  scoping as other bounded projection items:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/statement/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/normalize.rs`
* lowered field-to-field arithmetic projection items onto the same canonical
  binary expression seam already used for scalar projection evaluation:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/projection.rs`
  * literal RHS still uses the existing bounded numeric-projection helper path
  * field RHS now lowers directly to one `Expr::Binary { left: Field, right: Field }`
    shape instead of forcing the parser to reject identifier operands
* extended the same bounded arithmetic path through `ROUND(expr, scale)` so
  `ROUND(field + field, scale)` stays on the scalar projection seam:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/tests/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_projection.rs`
* fixed aliased `ROUND(..., scale)` rendering by carrying fixed-scale hints
  through SQL projection packaging instead of inferring scale only from the
  outward column label text:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/projection/labels.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/projection/payload.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb/src/db/sql/mod.rs`
  * this keeps outputs like `ROUND(dexterity + charisma, 3) AS dextrisma`
    rendering as `16.000` instead of `16`
* added bounded parser, lowering, session, result-packaging, and live canister
  coverage:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/parser/tests/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_projection.rs`
  * `/home/adam/projects/icydb/crates/icydb/src/db/sql/mod.rs`
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/sql_canister.rs`

What this patch does not do:

* no arithmetic predicates
* no grouped arithmetic widening
* no `ORDER BY` arithmetic widening
* no generic nested arithmetic parser

Why this is the right cut:

* the executor and planner already know how to evaluate binary scalar
  projection expressions with two field reads
* the real missing piece was the parser/lowering boundary, not a new runtime
  subsystem
* keeping this to projection position preserves the bounded scalar-expression
  model instead of silently reopening general arithmetic SQL
* preserving fixed `ROUND(..., scale)` display at the outward SQL boundary is
  also the correct contract match for familiar SQL surfaces like SQLite and
  PostgreSQL, even when the projection label is aliased

## 0.80.2 - 2026-04-14 - Expose SQL Shell Compiler/Planner/Executor Footer Split

This patch makes the local Rust SQL shell show the deeper SQL attribution that
already exists at the session SQL seam instead of only one outer instruction
total. The change is intentionally a hard cut on the dev-shell perf payload:
`query_with_perf(sql)` now returns one current structured footer payload with
total, compiler, planner, and executor counts, and the shell renders that
shape directly.

What changed:

* changed the demo and lightweight SQL shell canisters so `query_with_perf(sql)`
  now calls `execute_sql_query_with_attribution::<E>(...)` instead of wrapping
  `execute_sql_query::<E>(...)` in one outer `performance_counter` delta:
  * `/home/adam/projects/icydb/canisters/demo/rpg/src/lib.rs`
  * `/home/adam/projects/icydb/canisters/test/sql/src/lib.rs`
* expanded the dev-shell perf envelope to carry:
  * total instructions
  * planner instructions
  * executor instructions
  * compiler instructions
* enabled `perf-attribution` by default on those two shell-oriented canisters
  so ordinary local `dfx deploy` runs through `scripts/dev/sql.sh` produce the
  richer footer without extra feature wiring:
  * `/home/adam/projects/icydb/canisters/demo/rpg/Cargo.toml`
  * `/home/adam/projects/icydb/canisters/test/sql/Cargo.toml`
* hard-cut the Rust CLI onto the new perf result shape and new footer format:
  * `/home/adam/projects/icydb/crates/icydb-cli/src/main.rs`
  * footer rendering now prints values like:
    * `7 rows in set (7.9Mi, 4.7Mi comp, 0.6Mi plan, 2.6Mi exec)`
  * the shell no longer keeps compatibility parsing for the old single-number
  perf payload
  * the shell now also treats `query_with_perf(sql)` as the required current
    query surface rather than probing for older canister behavior first
* deepened the shared repeated-query cache one level lower by letting the
  shared lower query cache hold one reusable prepared-plan shell instead of
  only the lower logical plan:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/executor/prepared_execution_plan.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs`
  * SQL and fluent still converge at the same generic-free structural query
    boundary, but cache hits now skip more repeated plan-preparation work on
    both frontends
* moved the shared lower query cache plus the SQL upper caches out of
  `DbSession` instance fields and into store-registry-scoped in-heap caches:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * generated `db()` still returns a fresh `DbSession`, but fresh sessions for
    the same canister/store registry now see the same warmed compiled-command,
    SQL `SELECT` plan, and shared lower prepared-plan entries
  * this keeps cache reuse available across canister calls when the cache is
    warmed from an `update` call, which is the IC-compatible persistence model
    for in-heap query caches
* added explicit cache-layer attribution to the SQL audit path so repeated SQL
  queries can show which cache boundary actually handled the request:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/mod.rs`
  * SQL query attribution now separates:
    * compiled-command cache hits/misses
    * SQL `SELECT` plan cache hits/misses
    * shared lower query-plan cache hits/misses
  * grouped `SELECT` now carries the same SQL cache-layer attribution instead
    of silently dropping back to zeroed cache counters
* extended the dedicated perf audit canister and SQL audit harness so the new
  cache-layer counters are averaged and printed in the recurring audit output:
  * `/home/adam/projects/icydb/canisters/audit/sql_perf/src/lib.rs`
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/sql_perf_audit.rs`
  * the SQL audit can now distinguish:
    * one-shot SQL queries that miss both SQL upper caches and hit the shared
      lower cache build path
    * repeated SQL queries that are satisfied entirely by the SQL-specific
      caches above the shared lower cache
* kept the fluent audit on the existing shared-cache counters:
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/fluent_perf_audit.rs`
  * fluent still reaches the shared lower cache directly, so its cache-hit
    story remains the clean reference for the converged non-SQL boundary
* added live PocketIC regressions that prove update-warmed caches are visible
  to later query calls on both fronts:
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/sql_perf_audit.rs`
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/fluent_perf_audit.rs`
  * the SQL regression warms one `SELECT` through an update endpoint and then
    asserts the next query call hits the compiled-command and SQL `SELECT`
    plan caches
  * the fluent regression warms one fluent read through an update endpoint and
    then asserts the next query call hits the shared lower query-plan cache

Why this is the right cut:

* the compile/execute split already exists in the current SQL session boundary
  and is part of the ongoing `0.80` cache line
* this patch also refines the `SELECT` execute side into planner work versus
  narrower runtime execution, so shell output can stop treating both as one
  opaque bucket
* showing only one outer instruction total hides the part the current work most
  needs to watch: whether compiler cost falls, whether planner cost moves, and
  whether narrower runtime execution actually changes
* this is an internal/dev-shell protocol, so keeping parallel old and new perf
  payloads would add compatibility burden without product value
* the new SQL cache-layer counters close a real observability gap:
  * before this patch, SQL repeat rows could be fast while the audit still
    showed zero shared-cache hits without explaining that SQL had already
    satisfied the query in an upper cache
  * after this patch, the recurring audit can tell whether reuse came from the
    compiled-command cache, the SQL `SELECT` plan cache, or the shared lower
    cache instead of treating those layers as one opaque hit path
* storing the caches on `DbSession` was too shallow for the IC execution
  model:
  * generated canister `db()` accessors construct a fresh session per call
  * query calls cannot persist heap mutations, but update calls can
  * moving the caches to canister-lifetime heap state lets update traffic warm
    later query traffic without pretending that ordinary query calls can commit
    cache state

## 0.80.1 - 2026-04-14 - Share Lower Query Cache Across SQL and Fluent

This patch keeps the explicit SQL compile/execute seam from `0.80.0`, but
moves the next reusable boundary down to the generic-free query layer so the
same repeated-query cache is no longer SQL-only. SQL still owns its frontend
parse/lower caches, but once both frontends reach the generic-free structural
query shape they now reuse the same session-local plan cache and the same
structural query fingerprinting model.

The practical goal of this slice is not to claim a broad one-shot speedup.
The real gain is that repeated queries issued through either frontend now reuse
the same lower plan work, and the audit surface can measure that behavior on
SQL and fluent separately without sharing one test file or one baseline.

What changed:

* added a shared lower query-plan cache in:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs`
  * the new cache is keyed by:
    * authority path
    * query-plan visibility mode
    * generic-free structural query fingerprint
  * the cached artifact is the lower planned structural query reused by the
    session after the frontend-specific compile steps are done
* moved SQL `SELECT` planning onto that lower shared cache instead of keeping
  the reusable plan work entirely inside the SQL session module:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * SQL still keeps its higher raw-SQL and compiled-command caches, but the
    lower structural plan reuse is now shared with fluent
* added structural query fingerprinting at the generic-free query intent
  boundary:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs`
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/query/fingerprint/mod.rs`
  * this gives non-SQL queries a stable cache key without depending on raw SQL
    text
* added fluent-side compile/execute attribution so the non-SQL path can be
  measured the same way as SQL:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/query.rs`
  * `/home/adam/projects/icydb/crates/icydb/src/db/session/mod.rs`
* added a focused core regression proving that equivalent SQL and fluent
  `SELECT` shapes reuse the same lower cached plan instead of creating
  parallel cache entries:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs`
* extended the dedicated audit canister with fluent perf endpoints while
  keeping the canister/schema fixture single-purpose:
  * `/home/adam/projects/icydb/canisters/audit/sql_perf/src/lib.rs`
  * the canister now exposes one-shot and looped fluent query perf endpoints
    over the same audit fixtures and indexes already used by SQL
* added a separate fluent PocketIC perf audit and separate baseline artifact:
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/fluent_perf_audit.rs`
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/fixtures/fluent_perf_audit_baseline.json`
  * this keeps the SQL and fluent audits comparable on the same data surface
    without making one file serve both recurring audit roles
* refreshed the SQL perf baseline after the lower shared cache change:
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/fixtures/sql_perf_audit_baseline.json`
  * the new SQL baseline now reflects the shared-cache state rather than the
    earlier SQL-only cache boundary

Perf notes:

* repeated-query rows improve on both frontends because the lower plan work is
  now shared and cached below the frontend split
* one-shot SQL rows are mixed and should not be described as a broad speedup
* the real shipped change is cache-boundary coherence, not universal
  instruction reduction on every query

## 0.80.0 - 2026-04-14 - Add Explicit SQL Compile/Execute Seam

This patch starts the `0.80` line by making the SQL semantic compile boundary
explicit before any cache machinery is added. The main goal of this slice is to
create one concrete command artifact that owns the planner-facing semantic
result of SQL compilation, so future cache work can reuse that artifact instead
of rebuilding semantic compile stages on repeated queries.

This is intentionally not a cache patch yet. There is no session-local cache,
no cross-session reuse, and no repeated-query speedup claim in this release.
The work here is the narrower prerequisite: define the compile artifact, route
the public SQL surfaces through it, and prove that compile and execute can now
be exercised separately.

What changed:

* added a new internal semantic SQL artifact:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/mod.rs`
  * `CompiledSqlCommand` now owns the compiled command family for:
    * lowered `SELECT`
    * lowered `DELETE`
    * global aggregate commands
    * lowered `EXPLAIN`
    * prepared `INSERT`
    * prepared `UPDATE`
    * entity metadata commands such as `DESCRIBE`, `SHOW INDEXES`,
      `SHOW COLUMNS`, and `SHOW TABLES`
* split the public SQL session flow into explicit compile then execute stages:
  * `execute_sql_query::<E>(...)`
  * `execute_sql_update::<E>(...)`
  * both now parse, validate the public lane, compile to `CompiledSqlCommand`,
    then execute that artifact
* centralized SQL compile routing under one owner:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/route.rs`
  * compilation now decides the semantic command family once, instead of
    mixing that classification back into execution helpers
* added one explicit execution entrypoint for compiled commands:
  * `/home/adam/projects/icydb/crates/icydb-core/src/db/session/sql/execute/mod.rs`
  * `execute_compiled_sql::<E>(...)` now consumes the semantic artifact and
    dispatches onto the existing execution/runtime helpers
* kept the compile artifact semantic rather than executor-prepared:
  * the artifact owns planner-facing lowered output
  * transient execution state remains executor-derived at call time
  * this keeps the cache boundary aligned with semantic compile work instead of
    widening early into scratch/runtime state reuse
* made aggregate compile output cloneable where needed so compiled-command
  execution can borrow the shared artifact cleanly without reintroducing
  semantic compile work through ownership friction
* removed one now-obsolete aggregate compile helper after compile ownership
  moved into the new route owner
* added focused seam tests proving compile and execute can be exercised
  separately across the currently supported SQL families:
  * scalar `SELECT`
  * grouped `SELECT`
  * global aggregate `SELECT`
  * `EXPLAIN`
  * `INSERT`
  * `UPDATE`
  * `DELETE ... RETURNING`

What this patch does not do:

* no SQL cache entries are stored yet
* no schema-fingerprint invalidation path exists yet
* no compile-hit instrumentation exists yet
* no executor scratch-state reuse exists yet
* no new SQL surface area was added

That boundary is intentional. The first `0.80` slice is only about artifact
completeness and seam correctness. The next cache slices can now be judged
against a concrete question: does a cache hit skip semantic compile work, or is
the engine still quietly rebuilding planner-facing state on the hit path?
