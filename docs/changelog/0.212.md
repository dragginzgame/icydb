# IcyDB 0.212: Bounded Resumable Integrity Check

## 0.212.1

### 📝 Summary

- Exposes the accepted-native integrity engine through explicitly trusted Rust
  and direct administrative SQL requests.
- Adds a default-off, controller-gated generated `icydb_integrity` endpoint for
  canisters that opt in with `[canisters.<name>.sql] integrity = true`.
- Completes Quick inspection's bounded control checks and makes durable job
  retention advance fairly during ordinary authorized use.

### ➕ Added

- Adds trusted typed Quick, Deep-start, Deep-continue, and Deep-abort requests
  with typed results. Entity names resolve exactly, jobs remain bound to their
  initiating caller, and private physical checkpoints never enter the public
  request or response surface.
- Adds direct `CHECK INTEGRITY` SQL parsing, lowering, and execution over the
  same controller as the typed API. The SQL frontend adds no separate
  inspection policy or execution path.
- Adds the opt-in generated Candid endpoint and build configuration. The
  endpoint is absent by default and uses the canister controller as its caller
  identity.
- Quick inspection now validates accepted-plan and relation closure, exact
  store capability profiles, committed stable-memory allocation closure, and
  journal control. Readable journal drift produces a bounded finding; corrupt
  load-bearing control authority produces typed `Uninspectable`.

```sql
CHECK INTEGRITY Token QUICK;
```

### 🔧 Changed

- Every well-formed authorized integrity attempt advances one fair, bounded
  progress-retention page. Expiry and terminal cleanup remain controller-owned;
  the heap-only retention cursor is neither persisted authority nor part of a
  proof vector.
- Retention preserves unacknowledged receipts, persists terminal
  acknowledgement, and prevents inactivity expiry or abort from destroying a
  response that must still replay exactly.
- Quick verification reuses accepted schema, relation, allocation, capability,
  and journal owners rather than introducing another scanner or reconstructing
  their semantic facts.

### 🧭 Persistence

- Persisted row, index, relation, accepted-schema, journal, commit-marker,
  query-cursor, and public response formats remain unchanged.
- The current integrity-progress format records the terminal acknowledgement
  and bounded retention state required by exact replay. There is no legacy
  decoder or compatibility translation.

### 📊 Audit

- Relative to `0.212.0`, this patch currently changes 43 files at approximately
  `+2,353/-271` lines, including three new frontend modules.
- The implementation adds public and generated boundary code plus focused
  tests, while retaining one integrity controller and one accepted-native
  verification architecture. It adds no second scanner, lifecycle runner, or
  persisted compatibility path.

### 🧪 Testing

- Focused typed and SQL execution, generated endpoint, configuration, Quick
  control, Deep lifecycle, replay, abort, retention, heap, relation, journal,
  and allocation-closure tests pass.
- A live generated canister test proves the default-off endpoint can be enabled
  and executes the controller-gated Quick path against the production
  stable-memory allocation registry.
- All-feature warnings-denied Clippy, no-default native compilation,
  formatting, diff hygiene, and affected architecture/documentation guards
  pass. Full repository validation remains owned by the human push workflow.
- Raw non-gzipped Wasm, instruction, peak-heap, response-byte, and durable
  job-state measurements remain the next closeout-evidence slice.

## 0.212.0

### 📝 Summary

- Establishes the accepted-native integrity engine beneath the future typed
  and SQL administrative surfaces.
- Adds bounded Quick inspection and durable resumable Deep verification over
  authoritative rows, active derived state, journal tails, and a final
  unchanged-proof boundary.
- Replaces recovery's whole-database aggregate scan with verification of only
  the bounded effects owned by the recovered marker.
- This release does not yet expose the authorized public or SQL
  `CHECK INTEGRITY` workflow. That frontend and final closeout evidence remain
  the next 0.212 slice.

### ⚠️ Breaking

- Hard-cuts the commit control envelope to its current version-1 form carrying
  a nonzero durable database-incarnation identity. Pre-0.212 development
  control bytes fail closed; no compatibility decoder, synthesis, alias, or
  migration path remains.
- Generated journaled canister layouts reserve a distinct stable-memory
  allocation for integrity progress. Existing development stores must be
  recreated where their allocation contract does not contain that role.

### ➕ Added

- Adds one fingerprinted `AcceptedInspectionPlan` compiled from the verified
  accepted catalog selection. It carries accepted entity/store identity,
  schema and catalog authority, row-layout contracts, compiled accepted checks,
  and active index/relation projections without generated-model fallback.
- Adds exact bounded Quick results with durable incarnation, accepted schema
  identity, typed completion state, exact finding totals, a bounded finding
  prefix, and an exact omitted count.
- Adds physical row verification over current and authenticated historical row
  layouts. Exact private checkpoints resume within a row across field,
  primary-key, accepted-check, expected-index, relation-target, and
  reverse-relation verification atoms.
- Adds bounded physical traversal of active accepted forward-index and
  source-owned reverse-relation generations. Verification re-derives each
  witness from authoritative rows and detects malformed, orphaned,
  conditional-membership, projection, and unique-multiplicity divergence.
- Adds a bounded journal phase covering current envelopes, sequence
  continuity, fold-watermark agreement, and duplicate batch identities without
  retaining an unbounded identity set.
- Adds one durable Deep controller over Quick metadata, rows, derived state,
  journal tails, and final proof. Every page compares exact pre/post proof
  vectors and preserves private checkpoints, bounded findings, replay keys,
  counters, and blocked-family state.
- Adds a separate current-form integrity-progress store with bounded job,
  owner, record, receipt, and retention limits. Lost responses replay exactly;
  proof drift terminalizes without publishing candidate findings; abort and
  expiry preserve unacknowledged receipts.

### 🔧 Changed

- Accepted-schema session caching now transports one inspection plan rather
  than parallel snapshot, catalog, identity, and compiled-program facts.
- The accepted write program remains the mutation gate while integrity selects
  only validated accepted checks from the same compiled authority.
- Active logical index and relation identities remain diagnostic authority;
  physical ordinal and generation values delimit only the exact active
  keyspaces being inspected.
- Recovery now verifies the last marker-owned effect for each row, schema
  snapshot, accepted-schema publication, and validation job before clearing
  marker authority. Recovered row puts also prove their rebuilt index and
  reverse-index effects.
- Recovery verifies every marker batch is covered by the durable fold
  watermark and that each journaled store has no physical tail batch remaining.
  It does not invoke Quick, Deep, or any global integrity scan.

### 🗑️ Removed

- Removes the old aggregate integrity scanner and its
  `IntegrityReport`/`IntegrityStoreSnapshot`/`IntegrityTotals` DTO hierarchy.
- Removes the global entity-to-live-key map previously materialized by that
  scanner.
- Removes report-specific tests and internal recovery coupling rather than
  retaining an alias, wrapper, or compatibility route.

### 🧭 Persistence

- Adds one nonzero database-incarnation field to the current commit-control
  envelope and one new checksummed version-1 integrity-progress format.
- The progress record is capped at 512 KiB, with 64 jobs globally and eight per
  owner. Unknown versions, corrupt checksums, invalid proof/checkpoint closure,
  and impossible lifecycle states fail closed.
- Persisted row, index, relation, accepted-schema, journal-batch,
  commit-marker payload, query cursor, and public response encodings are
  unchanged.

### 📊 Audit

- Excluding separately archived historical design documents, the release
  touches 66 files at approximately `+9,678/-1,204` lines.
- The line increase is concentrated in the durable Deep controller, bounded
  physical phase engines, persisted progress codec/store, and focused tests.
  Recovery decoupling itself is a net reduction because it deletes the
  aggregate scanner and report model.
- Authority count decreases at the recovery boundary: explicit Quick/Deep
  inspection owns database-wide integrity, while recovery owns only its
  marker-bounded postconditions.
- No dependency or Cargo package-version change is part of the implementation
  commit. Package version changes remain owned by release tooling.

### 🧪 Testing

- Focused accepted-plan, Quick, row, index, relation, journal, Deep lifecycle,
  proof-drift, receipt replay, recovery, schema-publication, and
  validation-job recovery tests pass.
- `icydb-core` passes all-target/all-feature warnings-denied Clippy plus native
  and Wasm compilation, including no-default-feature checks.
- Formatting, diff hygiene, durability, atomicity, layering, generated-build,
  and production-panic guards pass.
- Full repository tests, release packaging, raw non-gzipped Wasm deltas, and
  instruction measurements remain human-owned release validation.
