# 0.138.x

## 0.138.9

`0.138.9` seals aggregate identity normalization across SQL global and grouped
execution without changing aggregate results, output labels, projection remap
order, `HAVING` behavior, or executor terminal kinds. Raw SQL aggregate
terminals now preserve parsed `DISTINCT` syntax, a new aggregate identity owner
normalizes the executable meaning, and prepared SQL scalar aggregate strategies
store model-bound aggregate identity variants instead of carrying an independent
`distinct_input` flag alongside descriptor state. The follow-through also
renames the owning modules, helpers, tests, and active design-doc framing so
`identity` is used for equivalence/hash/dedup authority while `semantics`
remains reserved for behavior rules.

What changed:
- added planner-owned `AggregateIdentity` as the canonical owner for aggregate
  meaning, `DISTINCT` normalization, filter-aware executable identity, and
  model-bound prepared aggregate identity
- made `SqlGlobalAggregateTerminal` purely syntactic by preserving the raw
  parsed `DISTINCT` marker and removing terminal-owned `MIN`/`MAX` DISTINCT
  erasure
- changed projection terminal deduplication to compare aggregate identity
  instead of raw terminal equality
- changed `PreparedSqlScalarAggregateStrategy` to store
  `PreparedAggregateIdentity`, making `MIN` and `MAX` incapable of carrying a
  runtime DISTINCT bit
- removed the old raw aggregate-kind to descriptor helper path and derive the
  executor descriptor from prepared aggregate identity
- routed grouped aggregate equality, grouped execution-spec matching, HAVING
  aggregate lookup, and aggregate fingerprinting through the same identity
  rules
- renamed the SQL lowering and planner aggregate owner modules to
  `identity.rs`, and renamed stale helper/test vocabulary such as
  `semantic_input_expr_owned` to identity-specific names
- added lowering coverage proving `MIN(age)` and `MIN(DISTINCT age)` dedupe to
  the same identity terminal while distinct-sensitive aggregates such as
  `COUNT(DISTINCT age)` remain separate from their plain forms
- updated the `0.xxx` naming-cleanup design note with an `Identity` versus
  `Semantics` rubric plus a rename follow-through checklist for module/file
  names, re-exports, helpers, tests, comments, changelog text, active design
  docs, old-term scans, and stale empty directories

Retained behavior:
- SQL aggregate values, projection labels, projection remap order, `HAVING`
  semantics, DISTINCT behavior, and executor terminal kinds are unchanged
- `MIN(DISTINCT ...)` and `MAX(DISTINCT ...)` continue to ignore DISTINCT as
  before, but that behavior is now structurally represented by identity enum
  variants that do not carry a DISTINCT flag
- filtered and unfiltered aggregate terminals remain separate executable
  identities
- field-target, expression-input, and `COUNT(*)` row-target aggregates still
  bind through the same model validation and executor terminal construction
  boundaries

## 0.138.8

`0.138.8` removes the production `RowView::Indexed` materialization path from
grouped execution. Grouped row reads no longer construct a per-row retained-slot
`Vec<Option<Value>>`; instead they keep a raw-row-backed view, validate the
planner-selected grouped slots before fold consumption, and decode individual
slots lazily when hashing, equality, filters, grouped aggregates, or global
DISTINCT reducers actually need them. This keeps the grouped fold boundary
fail-closed while cutting the old retained-slot staging allocation out of the
production grouped row path.

What changed:
- replaced the production indexed grouped row view with a raw-row-backed row
  view that carries the persisted row, row layout, and storage key
- removed grouped execution's call to compact retained-slot vector decoding
  from normal row materialization
- kept the existing single-slot grouped fast path for the common one-field
  shape
- updated grouped hash construction, canonical-key equality checks, grouped
  count ingestion, generic grouped aggregate reducers, grouped aggregate input
  and filter expressions, and global DISTINCT field reducers to read grouped
  slots through the lazy row-view boundary
- preserved the previous fail-closed validation order by validating all
  retained grouped slots before returning the lazy row view, without storing
  those decoded values in a per-row vector
- kept owned grouped key materialization on miss/insert and final output paths,
  rather than introducing new key allocations on existing-group probes

Retained behavior:
- SQL output, fluent output, grouped aggregate results, cursor behavior,
  persisted row bytes, and structural value bytes are unchanged
- grouped hashing and equality still compare against canonical `GroupKey`
  values using the same stable hash and canonical comparison rules
- grouped map/bucket ownership and duplicate-group handling are unchanged
- the dense `Vec<Option<Value>>` row-view helper remains test-only; production
  grouped execution no longer uses the retained-slot vector materialization
  path

Validation:
- formatted the workspace with `cargo fmt --all`
- ran `cargo clippy -p icydb-core --all-targets -- -D warnings`
- ran grouped-focused `icydb-core` tests
- ran the ignored native execution hot-path benchmark report to cover grouped
  high-cardinality, low-cardinality, multi-field key, and aggregate scenarios

## 0.138.7

`0.138.7` expands the SQL numeric scalar surface with logarithmic and
exponential functions while keeping the existing checked numeric-evaluation
boundary. SQL now accepts `EXP(x)`, `LN(x)`, `LOG2(x)`, `LOG10(x)`,
`LOG(base, value)`, and `CBRT(x)` in projection, WHERE-owned scalar
expressions, aggregate inputs, and supported ORDER BY function expressions.
The functions lower to the shared planner `Function` taxonomy and evaluate
through the same decimal coercion and `NumericEvalError` path as `SQRT`,
`POWER`, and `MOD`.

```sql
SELECT name, EXP(level - level), LN(level / level), LOG2(8), LOG10(100), LOG(2, 8), CBRT(27)
FROM Character;

SELECT class_name, AVG(LOG2(level + 1)), SUM(CBRT(level))
FROM Character
GROUP BY class_name;
```

What changed:
- added parser support for `EXP`, `LN`, `LOG2`, `LOG10`, `LOG(base, value)`,
  and `CBRT`
- added planner function variants and scalar-function metadata for the new
  unary and binary numeric function families
- added checked decimal helpers for exponent, natural logarithm, base-2
  logarithm, base-10 logarithm, explicit-base logarithm, and cube root
- routed projection evaluation, preview evaluation, aggregate-input constant
  folding, type inference, and ORDER BY expression parsing through the shared
  function semantics
- added SQL projection tests for successful output values and invalid
  logarithm-domain errors
- added SQL lowering tests proving ORDER BY aliases and direct ORDER BY terms
  normalize onto the canonical planner expressions

Retained behavior:
- existing numeric functions, Decimal storage, persisted row formats, cursor
  formats, SQL output labels, and aggregate behavior are unchanged
- non-representable domains such as `LN(0)`, `LOG2(-1)`, and `LOG(1, 10)` use
  the existing structured numeric-not-representable query error path
- `LOG(x)` unary dialect behavior is intentionally not introduced; the
  supported explicit-base form is `LOG(base, value)`

## 0.138.6

`0.138.6` implements the numeric overflow semantics addendum for the `0.138`
SQL numeric-function line. Exact numeric SQL evaluation now reports overflow
and non-representable exact numeric results as structured query execution
errors instead of silently inheriting decimal saturation in projection
arithmetic, scalar numeric functions, aggregate expression inputs, and
aggregate accumulation. Low-level `Decimal` operators keep their existing
behavior; checked semantics live at the database numeric-evaluation boundary
used by SQL-facing execution paths. The public canister proof now lives on the
dedicated `schema/test/sql` and `canisters/test/sql` surface, so the numeric
coverage is not coupled to the RPG demo `Character` entity.

What changed:
- added the numeric overflow semantics addendum under the `0.138` design notes
- introduced a checked `NumericEvalError` boundary for overflow and
  non-representable exact numeric results
- added checked decimal arithmetic, average, square-root, and power helpers
  alongside the existing low-level decimal behavior
- routed scalar projection arithmetic, SQL numeric scalar functions, grouped
  projection functions, and compiled scalar projection operators through the
  checked numeric boundary
- changed SUM/AVG accumulator paths, grouped aggregate reducers, and global
  grouped DISTINCT numeric aggregation to fail on checked numeric overflow
- preserved numeric error classification for grouped aggregate input and
  filter expression failures instead of wrapping checked numeric failures as
  invalid logical-plan errors
- preserved planner constant-folding safety by skipping overflowed literal
  folds instead of materializing saturated constants that could diverge from
  runtime behavior
- added structured query error detail for numeric overflow and
  non-representable numeric results so tests and callers do not need to match
  rendered error text
- added regressions for overflowing SQL projection functions, overflowing SQL
  aggregate expression inputs, checked numeric helper overflow, and checked
  aggregate numeric addition
- added a dedicated `SqlTestNumericTypes` schema/test SQL canister fixture with
  signed integer, unsigned integer, decimal, `Float32`, and `Float64` fields,
  plus PocketIC coverage for mixed numeric projections, global/grouped
  aggregates, overflow, division/modulo by zero, and non-representable square
  root results

Retained behavior:
- low-level `Decimal` arithmetic operators still saturate or return their
  existing primitive fallback values outside the checked SQL evaluation seam
- persisted bytes, binary formats, schema formats, and public APIs are
  unchanged
- DISTINCT equality, floating-point special-value behavior, and aggregate route
  architecture are not reworked in this patch
- no trait-based numeric abstraction or execution-path unification was
  introduced
- demo canister `Character` rows are not used for this test coverage

## 0.138.5

`0.138.5` adds a focused native execution hot-path benchmark harness and
removes live compatibility fallbacks that still referenced older audit layouts
or legacy helper naming. The new ignored benchmark suite covers simple SQL
projection, SQL DISTINCT over wider rows, ordered limited SELECTs, grouped
queries at different cardinalities, scalar/grouped aggregates, DELETE
RETURNING, and persisted-row codec encode/decode cases. The PocketIC SQL and
fluent perf audits remain the source of canister instruction samples, while
the native harness gives a cheaper way to compare relative hot-path shapes
inside `icydb-core`.

This patch also tightens the structural value-storage codec boundary without
changing the structural binary format. The old single value-storage module is
split into encode, decode, skip, walk, primitive, and tag owners behind the
same module facade. Decode hot paths now avoid small fixed-arity tuple staging
allocations for decimal, bigint, and enum payloads, reuse parsed scalar heads
instead of re-parsing the same root value, pre-size list/map materialization
vectors from their encoded lengths, and decode recursive list/map values in one
materialization pass while leaving the borrowed-slice walker APIs intact.
Visitor-based borrowed list/map traversal helpers are also available internally
so future generated callers can consume nested payload slices without first
staging them in a vector.

What changed:
- added an ignored `execution_hot_path_microbenchmarks_report` test under the
  session test boundary for repeatable native query-shape measurements
- kept the benchmark harness test-only and opt-in so normal unit, clippy, and
  check runs do not pay benchmark setup cost
- removed wasm audit baseline lookup fallbacks for pre-current artifact
  layouts, so comparisons now use the maintained
  `artifacts/<scope>/<report>.<canister>.<profile>.<sql-variant>.size-report.json`
  path only
- renamed the session SQL test-surface classifier from its legacy helper name
  to the current `SelectTestSurface` boundary while preserving the exact test
  rejection matrix and messages
- verified that live `testing/`, `canisters/`, `crates/`, and `scripts/`
  sources no longer reference the removed old SQL perf harness names or old
  audit scenario-key patterns
- split structural value-storage encoding, decoding, local tag handling,
  primitive payload parsing, skip traversal, and walk traversal into explicit
  internal owners
- removed fixed-arity `Vec<&[u8]>` tuple staging from decimal, bigint, and enum
  decode paths by using stack arrays for the known two- and three-field shapes
- added parsed-head scalar decode helpers so root scalar value decode can reuse
  the already-parsed `(tag, len, payload_start)` tuple while preserving the
  same truncation, tag, length, and trailing-byte validation behavior
- pre-sized recursive list and map materialization buffers from the encoded
  collection length before walking entries
- changed recursive `Value::List` and `Value::Map` materialization to advance
  the decode cursor directly instead of first finding every nested item with
  skip traversal and then decoding the same slice again
- added private visitor helpers for borrowed list/map payload traversal, then
  reimplemented the existing borrowed split APIs on top of those visitors with
  the same returned slices and ordering
- documented the two traversal models in the value-storage owner: skip remains
  the authoritative boundary detector for structural validation, while decode
  owns runtime `Value` materialization over already-bounded slices
- added focused malformed-payload and borrowed-slice tests for list/map trailing
  bytes, truncated nested payloads, invalid nested local tags, duplicate map
  keys, and visitor-preserved item order

Retained behavior:
- no public SQL, fluent query, grouped query, aggregate, codec, cursor, or
  persisted format behavior changed
- historical changelog and audit report artifacts remain archival records and
  were not rewritten
- SQL and fluent PocketIC perf audit harnesses remain current and unchanged
- wasm size reports still compare against current-format baselines when present
- structural binary envelope bytes, tag values, payload ordering, decode error
  text, fail-closed validation, and public value-storage APIs are unchanged
- existing walker and borrowed-slice split APIs remain available; the new
  visitor helpers are internal preparation for generated callers
- `Value::Map` construction still routes through `Value::from_map`, preserving
  duplicate-key and ordering invariants

## 0.138.4

`0.138.4` keeps aggregate query results unchanged while making value aggregate
semantics a single reducer-core responsibility. Scalar terminals, grouped
value reducers, global aggregate execution, grouped DISTINCT finalization, and
grouped `COUNT(*)` page output now route COUNT finalization through the same
policy helper, and COUNT(value), SUM, AVG, MIN, and MAX reducer behavior is
documented as owned by `executor::aggregate::reducer_core`. New session
coverage compares fluent scalar COUNT/SUM/AVG with SQL global and SQL grouped
aggregate outputs on the same input, while SQL global and grouped paths also
lock MIN/MAX parity through the shared reducer core. Aggregate fast-path stream
requests also now borrow the prepared executable access contract instead of
cloning it before routing, keeping fallback behavior unchanged while removing
an avoidable access-plan copy from secondary, primary, index-range, and
composite aggregate fast-path branches.

What changed:
- added an explicit reducer-core contract documenting that COUNT(value), SUM,
  AVG, MIN, and MAX value aggregate behavior is owned by
  `executor::aggregate::reducer_core`
- introduced one COUNT finalization helper and routed scalar value COUNT,
  grouped value COUNT, grouped `COUNT(*)` page output, key-stream COUNT output,
  and grouped DISTINCT COUNT finalization through it
- kept COUNT storage specialized where it already was, including grouped
  `COUNT(*)` counters and key-stream count payloads, while sharing the final
  public `Value` policy
- added a session-level convergence regression that compares fluent scalar
  COUNT/SUM/AVG, SQL global aggregates, and SQL grouped aggregates over the
  same deterministic input
- locked SQL grouped/global MIN and MAX parity in that regression without
  changing the fluent extrema APIs, which continue to return entity ids ordered
  by a field
- changed fast-stream route requests to borrow `ExecutableAccessPlan` instead
  of owning a cloned plan
- added a borrowed traversal entrypoint for executable access plans so aggregate
  primary-key, primary-scan, secondary-index, index-range, and composite
  fast-path branches can stream without cloning `ExecutableAccess`
- narrowed primary-key fast-stream route verification to borrow the access
  contract because it validates shape only and does not need ownership

Retained behavior:
- aggregate results, projection labels, empty-window semantics, DISTINCT
  behavior, grouped pagination, and cursor behavior are unchanged
- grouped `COUNT(*)` keeps its dedicated storage and ingest path
- scalar, grouped, and global execution strategies remain separate; only shared
  reducer semantics and COUNT finalization policy are centralized
- aggregate fast-path route selection, diagnostics, scan accounting, fallback
  behavior, and error text are unchanged
- no trait objects, boxed execution strategies, or new hot-loop allocations were
  introduced

## 0.138.0

`0.138.0` expands the SQL scalar numeric function surface while keeping the
planner/executor function model canonical. SQL parser aliases now lower onto
one internal function identity, and the runtime implementations reuse the
shared `db::numeric` Decimal boundary where exact fixed-point behavior already
exists. The same patch also folds a small db cleanup pass into the release:
repeated local-instruction timers, simple offset/limit row windows, and one
duplicated grouped-count metrics definition now route through single owners
without changing execution behavior. A follow-up cleanup removes one empty
post-access window shim, derives delete projection row counts from the
materialized rows, and lets SQL projection execution return its row payload
directly instead of constructing a one-use adapter wrapper.

What changed:
- added SQL scalar support for `SIGN(...)`, `SQRT(...)`, `MOD(...)`,
  `POWER(...)`, `POW(...)`, `TRUNC(...)`, and `TRUNCATE(...)`
- mapped SQL aliases onto canonical planner functions, so `POW` renders and
  plans as `POWER`, and `TRUNCATE` renders and plans as `TRUNC`
- added Decimal truncation support for scale-taking `TRUNC` execution
- added shared numeric helpers for sign, approximate square root, and power
  evaluation
- kept `MOD` on the same Decimal remainder semantics as the existing numeric
  arithmetic layer
- generalized the old `ROUND` scale-taking function path so `ROUND` and `TRUNC`
  share parsing, lowering, type inference, preview evaluation, aggregate input
  folding, and projection execution behavior
- added SQL projection and lowering coverage for the new numeric functions and
  aliases
- consolidated repeated db diagnostics timing helpers through
  `db::diagnostics::local_instructions`
- consolidated identical executor offset/limit row-window helpers through one
  executor utility while leaving each caller responsible for its own projection,
  delete, or post-access semantics
- collapsed the duplicate `GroupedCountFoldMetrics` cfg definitions into one
  struct and narrowed covering projection row consumption to the executor
  boundary
- removed the empty post-access window module and import-only covering
  projection window aliases
- removed the one-use `SqlProjectionRows` adapter wrapper and the duplicate
  stored delete-projection row count
- collapsed the duplicated SQL projection materialization metrics capture helper
  across diagnostics and test cfgs
- removed the SQL projection passthroughs for pure-covering local-instruction
  counters
- consolidated grouped-key map normalization, executor page-window state
  conversion, and ranked materialized terminal row/value projection loops so
  each duplicated tail now has one local owner

Example queries:

```sql
SELECT
  SIGN(age - 30),
  SQRT(age - 17),
  MOD(age, 10),
  POWER(age - 30, 2),
  POW(age - 30, 2),
  TRUNC(age / 10, 0),
  TRUNCATE(age / 10, 1)
FROM User
ORDER BY TRUNC(age / 10, 0) DESC;
```

Retained behavior:
- existing `ROUND`, `ABS`, `CEIL`/`CEILING`, and `FLOOR` behavior is unchanged
- SQL projection labels remain canonicalized through the planner function name
- Decimal storage, Decimal field scale validation, index formats, cursor
  formats, and executor route behavior are unchanged
- `SQRT` and non-integral or negative-exponent `POWER` use the explicit lossy
  Decimal-to-f64 bridge and fail closed when the result cannot be represented
  as the current Decimal type
- diagnostics attribution bucket names, query results, cursor behavior, and SQL
  response payloads are unchanged by the db cleanup
