# IcyDB 0.174 Changelog

## 0.174.7

### ➕ Added

- Switches the demo RPG store to the admitted journaled cached-stable storage
  form so the demo canister exercises generated journaled runtime wiring.
- Adds an ignored native timing report that splits stable and journaled storage
  timing across per-row inserts, atomic batch inserts, full SQL scans, and
  indexed SQL scans over the same session workload.
- Adds SQL `SHOW STORES` and typed `show_stores()` catalog metadata for
  registered store storage modes.
- Adds explicit verbose catalog surfaces: `SHOW ENTITIES VERBOSE` and
  `SHOW STORES VERBOSE` render full entity and store paths while default
  shell output stays compact.

### 🔧 Changed

- Changes `SHOW ENTITIES` and typed `show_entities()` from name-only output to
  structured entity catalog metadata carrying entity name, entity path, and
  owning store path.
- Changes shell rendering for `SHOW ENTITIES` to show compact entity/store
  names by default, with full paths available only through `VERBOSE`.
- Changes `SHOW STORES` shell rendering to show compact store names and storage
  modes by default, with full store paths available only through `VERBOSE`.
- Optimizes journaled cached-stable data and index traversal by streaming
  fresh-live, folded-canonical, and mixed canonical-plus-live read views instead
  of materializing full snapshots before scans.
- Optimizes primary-order SQL projection hot paths by keeping filterless
  compiled-command and shared-plan cache hits off accepted `SchemaInfo`
  projection, and by decoding single-slot retained rows through the narrow
  required-slot reader.

### 🗑️ Removed

- Removes the SQL `SHOW TABLES` alias and typed `show_tables()` helper so the
  public catalog vocabulary stays on IcyDB entities and stores.

### 🧪 Testing

- Adds parser, lowering, session catalog, facade rendering, and compile-cache
  coverage for `SHOW STORES`, verbose catalog output, compact shell rendering,
  and the removed `SHOW TABLES` alias.
- Adds regression coverage proving stable and journaled
  `ORDER BY id ASC LIMIT 1` projections scan only one row, plus store-level
  guards proving mixed journaled data and index range traversal do not call the
  old snapshot materializer.

## 0.174.6

### ➕ Added

- Adds journaled storage-report coverage so data, index, and schema diagnostic
  rows render `storage: journaled` with durable allocation, recovery, commit,
  and schema-history capabilities.
- Adds mixed stable-to-journaled strong-relation coverage proving journaled
  targets satisfy durable relation-target capability semantics.

### 🔧 Changed

- Closes the runtime-admission proof matrix against the 0.173 protocol
  decisions: marker-bound journal publication, monotonic tail ordering, fold
  watermark replay, schema replay, and derived index materialization.
- Extends storage-policy source guards to reject concrete journaled backend
  branching in executor, relation, commit, reconciliation, and diagnostics
  policy roots.

### 🧪 Testing

- Adds focused diagnostics, relation, source-guard, stable-regression, and
  heap-regression evidence for the journaled runtime-admission closeout.

## 0.174.5

### ➕ Added

- Adds durable journal fold-watermark storage inside the journal-tail memory,
  preserving the four-memory journaled allocation contract.
- Adds recovery fold for committed journal-tail row and schema records into the
  canonical stable data/schema BTrees.

### 🔧 Changed

- Recovery now uses the fold watermark as the journal replay boundary and
  clears folded tail batches after advancing the watermark.
- Rebuilds journaled indexes as derived materialized state and folds that view
  into the canonical index BTree after recovery rebuild.
- Keeps normal journaled writes out of canonical stable BTrees; canonical
  mutation remains recovery/fold-owned.

### 🧪 Testing

- Adds focused journal-tail watermark tests plus session recovery coverage for
  canonical data/index fold, marker-bound journal repair, and replay-tail
  cleanup.
- Updates source-guard evidence for the 0.174.5 fold boundary and keeps fold
  policy out of executor and relation roots.

## 0.174.4

### ➕ Added

- Adds journaled recovery replay from canonical stable BTrees plus committed
  marker-bound journal tail batches.
- Adds recovery repair for missing journal-tail publication using embedded
  commit-marker journal batches.

### 🔧 Changed

- Rebuilds journaled live data and schema projections during recovery without
  folding row/schema mutations into canonical stable BTrees.
- Extends startup index rebuild to treat journaled indexes as derived
  materialized state from recovered rows.

### 🧪 Testing

- Adds focused session and source-guard evidence for journal-tail recovery,
  marker-bound publication repair, live index rebuild, and no pre-fold
  canonical BTree mutation.

## 0.174.3

### ➕ Added

- Adds journaled data, index, and schema store wrappers with live Rust BTree
  projections over canonical stable BTree bases.
- Adds journaled public write/read/index coverage proving live rows and
  materialized indexes are visible before recovery and fold land.

### 🔧 Changed

- Routes journaled write commits through marker-bound journal-tail appends
  before updating live projections.
- Keeps canonical data/index/schema stable BTrees untouched during normal
  journaled writes; fold remains the only future path into the canonical base.
- Keeps journaled recovery fail-closed until the recovery slice lands.

### 🧪 Testing

- Adds focused session and source-guard evidence for live journaled write/read,
  indexed SQL routing, journal-tail append, and no-normal-write fold behavior.

## 0.174.2

### ➕ Added

- Adds the bounded/fallible journal batch codec, `JournalBatch` envelope, and
  logical row/schema journal records for the journaled cached-stable tail.
- Adds stable BTreeMap-backed journal-tail storage keyed by monotonic
  `journal_sequence`, with ordered read validation for gaps, duplicate batch
  IDs, and physical-key/encoded-sequence mismatches.

### 🔧 Changed

- Hard-cuts the internal commit-marker payload to embed marker-bound journal
  batch payloads under the 0.173 publication strategy.
- Threads generated journaled wiring through a `JournalTailStore` and
  `register_journaled_store` without adding journaled data/index/schema
  runtime wrappers, recovery replay, or fold behavior.

### 🧪 Testing

- Adds codec, journal-tail, commit-marker, registry, generator, and
  source-guard evidence for commit-marker-bound journal batch storage.

## 0.174.1

### ➕ Added

- Admits the full four-ID `storage(journaled(...))` form at the schema,
  derive, and generated-build wiring boundary.
- Adds journaled cached-stable schema metadata, runtime capability projection,
  four-role allocation identity metadata, and generated journal-tail memory
  declaration.

### 🔧 Changed

- Keeps stable and heap generated wiring unchanged while adding journaled
  canonical data/index/schema stable-store wiring.
- Keeps journaled write/recovery behavior fail-closed until the journal
  runtime store, codec, recovery, and fold slices land.

### 🧪 Testing

- Adds schema, derive, registry, generator, compile-pass, compile-fail, and
  source-guard evidence for the schema/build admission boundary.

## 0.174.0

### ➕ Added

- Starts the journaled cached-stable runtime-admission line with an
  admission-readiness check instead of public runtime admission.

### 🔧 Changed

- Records the 0.173 protocol decisions as binding for 0.174 implementation:
  embedded bounded journal batch payload, monotonic `journal_sequence`, fold
  watermark replay boundary, accepted-schema replay, and derived index
  materialization.
- Keeps public `storage(journaled(...))` reserved until generated wiring,
  runtime construction, recovery, fold, diagnostics, relation behavior, and
  source audits all pass in one branch.
- Leaves runtime behavior unchanged for the setup slice.

### 🧪 Testing

- Adds a source/design guard proving 0.174.0 starts from the closed 0.173
  foundation while keeping journaled runtime admission gated.
