# IcyDB 0.181 Changelog

## 0.181.0

### 📝 Summary

- Starts the safe and bounded SQL `UPDATE` line by documenting the current
  session/library write-lane boundary without changing execution behavior.
- Records that `UPDATE` and narrow `UPDATE ... RETURNING` already work through
  `execute_sql_update::<E>(...)`, while generated query and DDL endpoints
  reject row mutation SQL.
- Establishes the next implementation direction: explicit `UPDATE` exposure
  policy gates before any generated/public canister SQL write endpoint can
  expose broader mutation semantics.

### 📚 Documentation

- Marks the 0.181 design as in progress and scopes 0.181.0 to Phase 0 /
  Slice A: baseline docs and boundary tests.
- Updates the SQL subset contract so `UPDATE` availability is stated by
  surface rather than as a uniform SQL capability.
- Updates the SQL surface mapping to distinguish the session/library
  `execute_sql_update::<E>(...)` write lane from generated `__icydb_query` and
  `__icydb_ddl` endpoints.
- Documents that no generated SQL write endpoint is part of the current
  default generated canister surface.

### 🧪 Testing

- Relies on existing focused core and generated-canister coverage for the
  current baseline: `UPDATE`, non-primary-key session updates,
  `UPDATE ... RETURNING`, fail-closed unsupported `RETURNING`, missing
  `WHERE`, primary-key/generated/managed assignment rejection, and generated
  query/DDL endpoint rejection of row mutation SQL.
