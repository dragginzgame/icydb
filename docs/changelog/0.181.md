# IcyDB 0.181 Changelog

## 0.181.16

### 📝 Summary

- Moves the schema-derived public/generated bounded `UPDATE` target-row limit
  into a named update-policy default.
- Routes generated/public schema-derived update admission through one default
  context constructor, so the bounded target limit and `RETURNING` response
  budget cannot drift apart.
- Keeps manually constructed validated plans configurable through their
  explicit policy context.

### 🔧 Changed

- Adds an internal `100` row default for public bounded deterministic
  `UPDATE`.
- Uses that default in policy helpers, schema-derived execution, and focused
  policy tests instead of repeating a literal limit.
- Documents the concrete default in the 0.181 design note.

### 🧪 Testing

- Covers policy rejection for limits above the named bounded-update default.
- Covers schema-derived public bounded execution rejecting an excessive limit
  before mutation and leaving accepted rows unchanged.
- Rechecks core SQL clippy.

## 0.181.15

### 📝 Summary

- Gives schema-derived public/generated SQL `UPDATE RETURNING` endpoints a
  default rendered-projection response budget.
- Enforces that default for both primary-key-only and bounded deterministic
  public update adapters before mutation commit.
- Leaves manually validated internal/session update plans unchanged unless a
  caller supplies an explicit response-byte cap.
- Cleans up `icydb.toml` ownership by renaming the shared config crate to
  `icydb-config`, tightening discovery semantics, and keeping new-config
  placement policy in the CLI.

### 🔧 Changed

- Adds an internal default public `UPDATE RETURNING` response-byte budget of
  `1 MiB`.
- Carries that budget through accepted-schema-derived public update plan
  classification.
- Documents the concrete default in the 0.181 design note.
- Makes shared existing-config discovery an IcyDB ancestor walk instead of a
  Cargo workspace-bounded search.
- Keeps `cargo metadata` limited to `icydb config init` creation placement when
  no existing `icydb.toml` is discoverable.
- Renames `icydb-config-build` to `icydb-config` across live Cargo metadata,
  build-script consumers, CLI config code, and current docs.
- Splits active config loading from ancestor-only existing-config discovery so
  `ICYDB_CONFIG_PATH` remains a read/build override without steering
  `icydb config init` creation decisions.
- Makes `icydb config init` create files with exclusive creation by default,
  replace through a temporary file when `--force` is explicit, and reject
  canister names that cannot be rendered as TOML bare keys.
- Treats Cargo metadata errors as placement diagnostics except for the expected
  non-Cargo fallback case.
- Renames the public config error type from `ConfigBuildError` to
  `ConfigError`.

### 🧪 Testing

- Covers schema-derived primary-key public update rejection when `RETURNING`
  would exceed the default response budget.
- Covers schema-derived bounded deterministic public update rejection for the
  same default response-budget path.
- Covers ancestor-based config discovery, workspace-root init placement,
  standalone-package init placement, non-Cargo fallback placement, and ambient
  config-path isolation.
- Covers directory entries named `icydb.toml`, force replacement, TOML-key
  render validation, and normalized known-canister validation.
- Rechecks policy plan budget propagation and core SQL clippy.
- Rechecks the renamed config crate, CLI config tests, a representative
  build-script consumer, and config crate clippy.

## 0.181.14

### 📝 Summary

- Enforces configured public SQL `UPDATE RETURNING` row-count budgets before
  commit.
- Carries the effective row budget in policy-validated public update plans, so
  primary-key and bounded deterministic updates reject oversized returning row
  sets atomically.
- Leaves count-only public primary-key and bounded updates unaffected because
  no `RETURNING` row payload is emitted.

### 🔧 Changed

- Combines configured returning-row budgets with policy-derived bounds when
  validating public update plans.
- Validates returning row count in the precommit hook alongside the existing
  rendered response-byte budget.
- Adds a compact `ReturningRowsTooMany` SQL write-boundary diagnostic with
  host-side rendering.
- Updates the 0.181 design note to reflect that public/generated
  `UPDATE RETURNING` now checks both row-count and response-size budgets before
  commit.

### 🧪 Testing

- Covers policy-bound lowering for configured row budgets.
- Covers public primary-key and bounded deterministic execution paths where
  oversized returning row sets reject before commit and leave persisted rows
  unchanged.
- Covers count-only public updates with configured returning-row caps.
- Rechecks SQL write behavior, diagnostic code mapping, CLI diagnostic
  rendering, clippy for touched packages, and invariant scans.

## 0.181.13

### 📝 Summary

- Enforces configured public SQL `UPDATE RETURNING` response-byte caps against
  the rendered projection payload instead of rejecting every configured cap.
- Runs the byte-cap check after structural mutation validation has produced
  sanitized after-images and before the save executor opens the commit window.
- Keeps count-only public primary-key and bounded updates unaffected by
  response-byte caps because they do not emit a `RETURNING` projection payload.

### 🔧 Changed

- Adds a SQL structural-batch precommit hook so public/generated write
  adapters can inspect validated after-images before accepted rows are
  published.
- Moves the shared row-projection value renderer into core so SQL write
  execution can size the same rendered values that the public facade packages.
- Adds a compact `ReturningResponseTooLarge` SQL write-boundary diagnostic and
  host-side rendering for the new leaf code.

### 🧪 Testing

- Adds public primary-key and bounded deterministic coverage proving oversized
  `UPDATE RETURNING` payloads reject with the new SQL write-boundary detail and
  leave persisted rows unchanged.
- Adds companion success coverage proving sized `RETURNING` payloads execute
  and still return post-update projection rows.
- Rechecks the broader SQL write suite, diagnostic-code table, CLI diagnostic
  rendering, facade tests, clippy for touched packages, and invariant scans.

## 0.181.12

### 📝 Summary

- Keeps public SQL `UPDATE RETURNING` aligned with accepted row projection
  visibility, including generated and managed fields.
- Preserves `RETURNING *` and ordinary `RETURNING field, ...` projections when
  the requested fields are visible through the accepted public row shape.
- Carries the companion cleanup slice that narrows a few parser, data, and
  grouped-aggregate boundaries without changing query semantics.

### 🔧 Changed

- Keeps generated and managed field assignment rejection as the write-safety
  boundary while allowing those fields to be returned when accepted row
  projection visibility already exposes them.
- Leaves policy-validated public primary-key and bounded deterministic update
  execution guarded by the configured response-byte-cap preflight from
  `0.181.11`.
- Keeps session/library SQL write behavior unchanged.

### 📊 Audit

- Narrows value-storage scalar-view accessors to the data boundary while
  keeping executor projection traversal on the borrowed view surface.
- Removes a duplicate grouped aggregate collector guard after SQL lowering has
  already proven `GROUP BY` is present.
- Narrows the reduced predicate parser cursor entry point to the parser module
  and removes the unused wider re-export.

### 🧪 Testing

- Adds policy coverage proving generated and managed fields are admitted in
  `RETURNING` when accepted row projection visibility includes them.
- Proves existing shape rejections remain stable with schema-owned fields in
  `RETURNING`.
- Adds execution coverage for validated-plan `RETURNING *`, schema-derived
  explicit generated returning fields, managed bounded `RETURNING *`, and the
  ordinary visible-field-list path.

### 🧭 Follow-Up

- Exact encoded response-size preflight remains pending before configured
  public/generated `UPDATE RETURNING` byte caps can be enforced instead of
  rejected.
- A narrower field-hiding policy remains deferred; today's public returning
  visibility follows the accepted row projection used by generated SQL reads.

## 0.181.2

### 📝 Summary

  separate raw parser value.

  row caps for primary-key and bounded deterministic `RETURNING`.
- Extends the hidden SQL facade re-exports so generated-code-facing paths can

- Proves generated query/DDL rejection and other rejected shapes expose no
  partially usable plan.

## 0.181.1

### 📝 Summary

- Adds a reusable SQL `UPDATE` exposure-policy classifier for the safe and
  bounded update line.
- Keeps existing execution behavior unchanged: the session/library write lane
  still owns broad `UPDATE`, while generated query and DDL surfaces remain
  rejection-only for row mutation SQL.
- Aligns the implementation with the first Phase 1 step in the updated design:
  explicit statement classification before any generated/public write surface
  can execute `UPDATE`.

### ➕ Added

- Adds hidden SQL update policy types for current session write, generated
  query, generated DDL, public primary-key-only, public bounded deterministic,
  and admin/bulk policy selection.
- Adds schema-derived policy context for primary-key, generated-owned, and
  managed/internal field categories.
- Adds structured update classification for target entity, assigned fields,
  unsafe assignment categories, `WHERE` proof, explicit primary-key ordering,
  `LIMIT`, `OFFSET`, and narrow `RETURNING` shape.
- Exposes the classifier through the existing hidden SQL facade path so future
  generated routing can opt into policy checks without importing private SQL
  parser modules.

### 🧪 Testing

- Adds focused classifier coverage for broad session admission, generated
  query and DDL rejection, non-`UPDATE` rejection, missing `WHERE`, unsafe
  primary-key/generated/managed assignment, primary-key-only proof, alias
  qualified primary-key predicates, composite primary keys, extra non-PK
  guards, bounded deterministic order/limit rules, `OFFSET` rejection, and
  narrow `RETURNING` classification.

### 🧭 Follow-Up

- The updated 0.181 design now requires successful policy validation to
  produce typed validated update plans before generated/public execution can
  consume them. `0.181.1` establishes the classifier and policy report
  foundation; the typed executable plan handoff remains follow-up work for the
  next 0.181 slice.

## 0.181.0

### 📝 Summary

- Starts the safe and bounded SQL `UPDATE` line by documenting the current
  session/library write-lane boundary without changing execution behavior.
- Records that `UPDATE` and narrow `UPDATE ... RETURNING` already work through
  `execute_sql_update::<E>(...)`, while generated query and DDL endpoints
  reject row mutation SQL.
- Establishes the next implementation direction: explicit `UPDATE` exposure
  policy gates before any generated/public canister SQL write endpoint can
  expose broader mutation semantics.

### 📚 Documentation

- Marks the 0.181 design as in progress and scopes 0.181.0 to Phase 0 /
  Slice A: baseline docs and boundary tests.
- Updates the SQL subset contract so `UPDATE` availability is stated by
  surface rather than as a uniform SQL capability.
- Updates the SQL surface mapping to distinguish the session/library
  `execute_sql_update::<E>(...)` write lane from generated `__icydb_query` and
  `__icydb_ddl` endpoints.
- Documents that no generated SQL write endpoint is part of the current
  default generated canister surface.

### 🧪 Testing

- Relies on existing focused core and generated-canister coverage for the
  current baseline: `UPDATE`, non-primary-key session updates,
  `UPDATE ... RETURNING`, fail-closed unsupported `RETURNING`, missing
  `WHERE`, primary-key/generated/managed assignment rejection, and generated
  query/DDL endpoint rejection of row mutation SQL.
