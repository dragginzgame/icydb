# IcyDB 0.181 Changelog

## 0.181.2

### 📝 Summary

- Upgrades the SQL `UPDATE` exposure-policy gate from classification-only
  admission to typed validated plan construction.
- Keeps execution behavior unchanged: no generated SQL write endpoint is added,
  and the existing session/library `execute_sql_update::<E>(...)` lane remains
  the only broad SQL mutation executor.
- Closes the next Phase 1 design requirement that admitted generated/public
  write policies return a policy-specific success value rather than relying on
  callers to remember that a raw parsed statement was checked earlier.

### ➕ Added

- Adds hidden validated update plan variants for current session write,
  public primary-key-only, public bounded deterministic, and admin/bulk
  policies.
- Carries the policy-validated parsed `UPDATE` statement privately inside each
  validated plan, so future executor adapters can consume the plan instead of a
  separate raw parser value.
- Adds returning bounds metadata to validated plans, including policy-derived
  row caps for primary-key and bounded deterministic `RETURNING`.
- Extends the hidden SQL facade re-exports so generated-code-facing paths can
  use the validated plan types when future write routing is introduced.

### 🧪 Testing

- Adds plan-construction coverage proving each admitted policy produces only
  its own validated plan variant.
- Proves generated query/DDL rejection and other rejected shapes expose no
  partially usable plan.
- Proves validated plans carry `RETURNING` row and response-size bounds.

## 0.181.1

### 📝 Summary

- Adds a reusable SQL `UPDATE` exposure-policy classifier for the safe and
  bounded update line.
- Keeps existing execution behavior unchanged: the session/library write lane
  still owns broad `UPDATE`, while generated query and DDL surfaces remain
  rejection-only for row mutation SQL.
- Aligns the implementation with the first Phase 1 step in the updated design:
  explicit statement classification before any generated/public write surface
  can execute `UPDATE`.

### ➕ Added

- Adds hidden SQL update policy types for current session write, generated
  query, generated DDL, public primary-key-only, public bounded deterministic,
  and admin/bulk policy selection.
- Adds schema-derived policy context for primary-key, generated-owned, and
  managed/internal field categories.
- Adds structured update classification for target entity, assigned fields,
  unsafe assignment categories, `WHERE` proof, explicit primary-key ordering,
  `LIMIT`, `OFFSET`, and narrow `RETURNING` shape.
- Exposes the classifier through the existing hidden SQL facade path so future
  generated routing can opt into policy checks without importing private SQL
  parser modules.

### 🧪 Testing

- Adds focused classifier coverage for broad session admission, generated
  query and DDL rejection, non-`UPDATE` rejection, missing `WHERE`, unsafe
  primary-key/generated/managed assignment, primary-key-only proof, alias
  qualified primary-key predicates, composite primary keys, extra non-PK
  guards, bounded deterministic order/limit rules, `OFFSET` rejection, and
  narrow `RETURNING` classification.

### 🧭 Follow-Up

- The updated 0.181 design now requires successful policy validation to
  produce typed validated update plans before generated/public execution can
  consume them. `0.181.1` establishes the classifier and policy report
  foundation; the typed executable plan handoff remains follow-up work for the
  next 0.181 slice.

## 0.181.0

### 📝 Summary

- Starts the safe and bounded SQL `UPDATE` line by documenting the current
  session/library write-lane boundary without changing execution behavior.
- Records that `UPDATE` and narrow `UPDATE ... RETURNING` already work through
  `execute_sql_update::<E>(...)`, while generated query and DDL endpoints
  reject row mutation SQL.
- Establishes the next implementation direction: explicit `UPDATE` exposure
  policy gates before any generated/public canister SQL write endpoint can
  expose broader mutation semantics.

### 📚 Documentation

- Marks the 0.181 design as in progress and scopes 0.181.0 to Phase 0 /
  Slice A: baseline docs and boundary tests.
- Updates the SQL subset contract so `UPDATE` availability is stated by
  surface rather than as a uniform SQL capability.
- Updates the SQL surface mapping to distinguish the session/library
  `execute_sql_update::<E>(...)` write lane from generated `__icydb_query` and
  `__icydb_ddl` endpoints.
- Documents that no generated SQL write endpoint is part of the current
  default generated canister surface.

### 🧪 Testing

- Relies on existing focused core and generated-canister coverage for the
  current baseline: `UPDATE`, non-primary-key session updates,
  `UPDATE ... RETURNING`, fail-closed unsupported `RETURNING`, missing
  `WHERE`, primary-key/generated/managed assignment rejection, and generated
  query/DDL endpoint rejection of row mutation SQL.
