# IcyDB 0.181 Changelog

## 0.181.11

### 📝 Summary

- Makes validated public SQL update execution fail closed when a configured
  `UPDATE RETURNING` response-byte cap is present.
- Preserves count-only public primary-key and bounded updates when the same
  response-byte cap is configured, because no `RETURNING` payload is produced.
- Documents that exact encoded-response-size preflight remains pending and
  configured caps must reject rather than be silently ignored until it exists.

### 🔧 Changed

- Adds the execution-side guard to the policy-validated public primary-key and
  bounded deterministic update adapters.
- Keeps row-cap metadata carried by validated plans as the current admitted
  `RETURNING` bound, while treating byte caps as fail-closed for now.
- Removes stale schema transition plan publication-status and mutation
  fingerprint accessors now superseded by runner preflight.
- Removes redundant SQL INSERT missing-primary-key validation branching while
  preserving the existing boundary code.

### 🧪 Testing

- Adds regression coverage proving capped public primary-key and bounded
  `UPDATE RETURNING` reject before mutation.
- Adds companion coverage proving capped count-only public updates still
  execute and mutate the admitted target rows.
- Rechecks the validated-plan returning-bounds test and the SQL-feature core
  test Clippy target.

### 🧭 Follow-Up

- Exact encoded response-size preflight still needs to be implemented before
  configured public/generated `UPDATE RETURNING` byte caps can be enforced
  rather than rejected.

## 0.181.10

### 📝 Summary

- Adds source-boundary proof that configured generated SQL update glue routes
  through explicit public-safe policy helpers, not broad session
  `execute_sql_update`.
- Tightens SQL global aggregate projection terminal collection to its
  projection owner.
- Narrows generated-model persisted-row complete patch writer visibility to
  its persisted-row parent module.

### 🧪 Testing

- Adds a generated SQL update glue guard covering configured
  `__icydb_update` routing.
- Keeps the guard focused on the active primary-key and bounded policy helper
  paths so future generated write routing cannot accidentally inherit broad
  session update semantics.

## 0.181.9

### 📝 Summary

- Adds live canister-boundary coverage for generated `UPDATE RETURNING` beyond
  the first bounded field-list case.
- Proves primary-key and bounded `RETURNING *` projections emit post-update
  row images.
- Removes redundant scalar aggregate terminal empty-result bookkeeping now
  owned by shared reducer finalization.

### 🧪 Testing

- Exercises generated primary-key and bounded update endpoints with
  `RETURNING *`.
- Verifies invalid generated `UPDATE RETURNING` projections reject without
  mutating rows.

## 0.181.8

### 📝 Summary

- Adds live canister-boundary coverage for bounded generated
  `UPDATE RETURNING`.
- Proves returned rows are post-update images emitted in the frozen bounded
  target order.
- Proves unsupported computed `RETURNING` rejects at the generated bounded
  update endpoint without mutating rows.

### 🧪 Testing

- Exercises the bounded generated update canister with explicit
  `ORDER BY primary_key ASC LIMIT n RETURNING field, ...`.
- Verifies unsupported computed `RETURNING` keeps the specific compact SQL
  feature code and query-owned origin metadata.
- Rechecks the bounded endpoint group covering accepted bounded updates,
  implicit-order rejection, post-update returning rows, and computed returning
  rejection.

## 0.181.7

### 📝 Summary

- Adds explicit generated SQL update policy selection for bounded deterministic
  `UPDATE` without changing the existing `update = true` primary-key-only
  default.
- Lets `update = "bounded"` route generated `__icydb_update` calls to the
  bounded validated-plan executor, while `update = "primary_key"` and
  `update = true` continue to select the primary-key-only policy.
- Keeps generated update dispatch policy-selected and prevents broad
  session/library `execute_sql_update` from becoming the generated public write
  route.

### ➕ Added

- Adds config/build plumbing for `PublicBoundedDeterministic` from
  `icydb.toml` through generated actor glue.
- Adds `icydb config init --update-policy bounded` for scaffolded bounded
  update endpoints, while preserving `--update` as the conservative
  primary-key shorthand.
- Adds a bounded SQL test canister fixture so primary-key-only and bounded
  generated update policies have separate live canister coverage.

### 📚 Documentation

- Updates the SQL subset, SQL surface mapping, and 0.181 design notes so
  generated `__icydb_update` is documented as opt-in and policy-selected rather
  than hypothetical.
- Documents that generated query and DDL endpoints continue to reject row
  mutation SQL, while generated update exposure depends on the configured
  public-safe update policy.

### 🧪 Testing

- Adds parser, CLI scaffold, config report, endpoint surface, and generated
  token coverage for the bounded policy selector.
- Adds live canister coverage proving the bounded generated update endpoint
  admits explicit primary-key-ordered limited updates and rejects unordered
  limited updates without mutating rows.

## 0.181.5

### 📝 Summary

- Adds the first generated SQL write endpoint, `__icydb_update`, behind an
  explicit `update = true` canister config switch.
- Makes the generated update endpoint use the default public primary-key-only
  validated `UPDATE` policy instead of the broad session/library SQL update
  lane.
- Routes CLI shell `UPDATE` statements to the configured SQL update endpoint
  while leaving `INSERT` and `DELETE` on the query rejection surface until
  their own generated policies exist.

### ➕ Added

- Adds generated-code config plumbing from `icydb.toml` through build options
  to actor glue emission for the primary-key-only SQL update policy.
- Adds `icydb config init --update` so scaffolded configs can opt into
  `__icydb_update` without hand-editing the SQL surface section.
- Extends config reports, endpoint surface checks, and wasm endpoint reporting
  so the generated SQL update surface is visible and gated consistently.

### 🔧 Changed

- Keeps `update = true` as the only accepted public config spelling for this
  first generated update policy; unsupported string policies fail parsing.
- Keeps generated update dispatch entity-scoped and policy-selected, proving
  generated actor glue does not call the broad `execute_sql_update` session
  entry point.

### 🧪 Testing

- Adds config parsing and CLI scaffold coverage for `update = true`,
  `update = false`, invalid update policy values, and `--update`.
- Adds generated-token and wasm endpoint-surface coverage for present and
  absent `__icydb_update` exports.
- Adds live canister-boundary coverage proving primary-key `UPDATE` succeeds,
  non-primary-key `UPDATE` rejects without mutation, and the CLI shell routes
  `UPDATE` to the configured update endpoint.

## 0.181.4

### 📝 Summary

- Locks the generated SQL surface to the current query, DDL, and fixture
  endpoints while the safe public write route is still policy-gated.
- Makes row-mutation SQL routing explicit: `INSERT`, `UPDATE`, and `DELETE`
  classify to the generated query rejection surface, not to a hidden generated
  write route.
- Extends wasm endpoint reporting so an unexpected generated SQL update export
  is visible instead of blending into the normal generated surface.

### 🔧 Changed

- Removes stale generated SQL DDL/write wording now that the generated surface
  is query plus DDL only.
- Documents the query surface as the current rejection lane for row mutation
  SQL until a generated write endpoint explicitly selects an update policy.
- Keeps the broad session/library SQL update executor available for internal
  callers, but proves generated actor glue does not call it or the narrowed
  public update helpers by default.

### 🧪 Testing

- Adds generated-token guards proving default SQL actor glue exports only the
  query, DDL, and fixture endpoints.
- Adds core route-classification coverage for `SELECT`, metadata reads, DDL,
  `INSERT`, `UPDATE`, and `DELETE`.
- Adds wasm-report coverage for absent and unexpected generated SQL update
  exports.
- Expands live canister-boundary coverage so the DDL endpoint rejects
  `INSERT`, `UPDATE`, and `DELETE` without mutating rows.

## 0.181.3

### 📝 Summary

- Adds hidden execution adapters for policy-validated public primary-key and
  bounded deterministic SQL `UPDATE` plans.
- Adds schema-derived helper paths that classify public-safe SQL `UPDATE`
  text against accepted schema field policy before execution, instead of
  trusting raw broad session `UPDATE` text.
- Keeps generated canister SQL write exposure unchanged: no generated
  `__icydb_update` endpoint is added in this slice.

### ➕ Added

- Adds primary-key-only and bounded deterministic execution entry points that
  accept only their matching typed validated update plan variants.
- Adds accepted-schema-derived plan construction for public-safe `UPDATE`,
  including primary-key fields plus generated-owned and managed/internal field
  policy context.
- Exposes matching hidden facade helpers so future generated/public write
  routing can consume the narrowed execution paths without importing parser
  internals.

### 🧪 Testing

- Proves public primary-key execution mutates one row and rejects non-primary
  key predicates before mutation.
- Proves public bounded deterministic execution mutates only the explicit
  ordered `LIMIT` window and rejects implicit-ordering shapes before mutation.
- Proves schema-derived public policy context rejects generated and managed
  field assignment before execution.
- Re-runs the SQL write boundary matrix and the update-policy matrix to keep
  existing broad session write behavior intact.

## 0.181.2

### 📝 Summary

- Upgrades the SQL `UPDATE` exposure-policy gate from classification-only
  admission to typed validated plan construction.
- Keeps execution behavior unchanged: no generated SQL write endpoint is added,
  and the existing session/library `execute_sql_update::<E>(...)` lane remains
  the only broad SQL mutation executor.
- Closes the next Phase 1 design requirement that admitted generated/public
  write policies return a policy-specific success value rather than relying on
  callers to remember that a raw parsed statement was checked earlier.

### ➕ Added

- Adds hidden validated update plan variants for current session write,
  public primary-key-only, public bounded deterministic, and admin/bulk
  policies.
- Carries the policy-validated parsed `UPDATE` statement privately inside each
  validated plan, so future executor adapters can consume the plan instead of a
  separate raw parser value.
- Adds returning bounds metadata to validated plans, including policy-derived
  row caps for primary-key and bounded deterministic `RETURNING`.
- Extends the hidden SQL facade re-exports so generated-code-facing paths can
  use the validated plan types when future write routing is introduced.

### 🧪 Testing

- Adds plan-construction coverage proving each admitted policy produces only
  its own validated plan variant.
- Proves generated query/DDL rejection and other rejected shapes expose no
  partially usable plan.
- Proves validated plans carry `RETURNING` row and response-size bounds.

## 0.181.1

### 📝 Summary

- Adds a reusable SQL `UPDATE` exposure-policy classifier for the safe and
  bounded update line.
- Keeps existing execution behavior unchanged: the session/library write lane
  still owns broad `UPDATE`, while generated query and DDL surfaces remain
  rejection-only for row mutation SQL.
- Aligns the implementation with the first Phase 1 step in the updated design:
  explicit statement classification before any generated/public write surface
  can execute `UPDATE`.

### ➕ Added

- Adds hidden SQL update policy types for current session write, generated
  query, generated DDL, public primary-key-only, public bounded deterministic,
  and admin/bulk policy selection.
- Adds schema-derived policy context for primary-key, generated-owned, and
  managed/internal field categories.
- Adds structured update classification for target entity, assigned fields,
  unsafe assignment categories, `WHERE` proof, explicit primary-key ordering,
  `LIMIT`, `OFFSET`, and narrow `RETURNING` shape.
- Exposes the classifier through the existing hidden SQL facade path so future
  generated routing can opt into policy checks without importing private SQL
  parser modules.

### 🧪 Testing

- Adds focused classifier coverage for broad session admission, generated
  query and DDL rejection, non-`UPDATE` rejection, missing `WHERE`, unsafe
  primary-key/generated/managed assignment, primary-key-only proof, alias
  qualified primary-key predicates, composite primary keys, extra non-PK
  guards, bounded deterministic order/limit rules, `OFFSET` rejection, and
  narrow `RETURNING` classification.

### 🧭 Follow-Up

- The updated 0.181 design now requires successful policy validation to
  produce typed validated update plans before generated/public execution can
  consume them. `0.181.1` establishes the classifier and policy report
  foundation; the typed executable plan handoff remains follow-up work for the
  next 0.181 slice.

## 0.181.0

### 📝 Summary

- Starts the safe and bounded SQL `UPDATE` line by documenting the current
  session/library write-lane boundary without changing execution behavior.
- Records that `UPDATE` and narrow `UPDATE ... RETURNING` already work through
  `execute_sql_update::<E>(...)`, while generated query and DDL endpoints
  reject row mutation SQL.
- Establishes the next implementation direction: explicit `UPDATE` exposure
  policy gates before any generated/public canister SQL write endpoint can
  expose broader mutation semantics.

### 📚 Documentation

- Marks the 0.181 design as in progress and scopes 0.181.0 to Phase 0 /
  Slice A: baseline docs and boundary tests.
- Updates the SQL subset contract so `UPDATE` availability is stated by
  surface rather than as a uniform SQL capability.
- Updates the SQL surface mapping to distinguish the session/library
  `execute_sql_update::<E>(...)` write lane from generated `__icydb_query` and
  `__icydb_ddl` endpoints.
- Documents that no generated SQL write endpoint is part of the current
  default generated canister surface.

### 🧪 Testing

- Relies on existing focused core and generated-canister coverage for the
  current baseline: `UPDATE`, non-primary-key session updates,
  `UPDATE ... RETURNING`, fail-closed unsupported `RETURNING`, missing
  `WHERE`, primary-key/generated/managed assignment rejection, and generated
  query/DDL endpoint rejection of row mutation SQL.
