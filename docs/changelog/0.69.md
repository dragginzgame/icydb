# 0.69 Changelog

## 0.69.0 - 2026-04-06 - True Covering Read Execution

### Summary

`0.69.0` starts the `0.69` line by turning covering reads from a
planner-only admission signal into a real execution contract with explicit
row-presence semantics. `0.68` had already widened the scalar read families
that could stay on index-backed routes; `0.69.0` makes the first part of that
contract executable without row payload reads instead of continuing to optimize
the row-materialized fallback around it.

The first shipped route stays intentionally narrow and explicit. Current
primary-store PK-only reads now fall into two planner-owned cohorts:

- authoritative primary-store traversal (`FullScan` and strict half-open
  `KeyRange`) reports `existing_row_mode = planner_proven` and may skip
  row-existence checks
- exact primary-key lookup (`ByKey` and canonical ascending `ByKeys`) reports
  `existing_row_mode = row_check_required` and keeps explicit row-presence
  checks even though the route is now first-class `CoveringRead`

Secondary covering routes from `0.68` remain conservative. They now compile
through the same execution-grade contract, but they stay
`row_check_required` until a later slice can prove authoritative existing-row
semantics for them.

### Changed

- promoted planner-owned covering admission into an execution-grade contract in:
  - `crates/icydb-core/src/db/query/plan/covering/mod.rs`
  - introduced `CoveringReadExecutionPlan`
  - introduced `CoveringExistingRowMode`
  - introduced `covering_read_execution_plan(...)`
  - moved route payloads away from projection-only covering metadata toward one
    execution-grade contract that runtime can consume directly
- widened the route boundary so covering execution is visible in the execution
  plan instead of remaining an implicit terminal shortcut in:
  - `crates/icydb-core/src/db/executor/route/terminal.rs`
  - `crates/icydb-core/src/db/executor/pipeline/operators/terminal/runtime.rs`
  - `crates/icydb-core/src/db/executor/explain/descriptor/load.rs`
  - current covering routes now surface `existing_row_mode` in
    `EXPLAIN EXECUTION` and `EXPLAIN JSON`
- shipped the first honest planner-proven cohort in:
  - `crates/icydb-core/src/db/query/plan/covering/mod.rs`
  - `crates/icydb-core/src/db/query/intent/model.rs`
  - PK-only projections over authoritative primary-store `FullScan`
  - PK-only projections over strict half-open primary-key `KeyRange`
  - redundant half-open PK predicates are stripped once the exact `KeyRange`
    access path is chosen
- widened the same route to exact primary-key lookup while keeping fail-closed
  row semantics explicit in:
  - `crates/icydb-core/src/db/query/plan/covering/mod.rs`
  - exact `ByKey` PK-only projection now compiles to `CoveringRead` with
    `existing_row_mode = row_check_required`
  - canonical ascending `ByKeys` PK-only projection does the same
  - descending `ByKeys` still fails closed for now rather than inventing an
    executor-local reorder contract
- kept secondary/index-backed covering routes honest in:
  - `crates/icydb-core/src/db/query/plan/covering/mod.rs`
  - the new execution-grade contract carries them forward, but they remain
    `row_check_required` until planner-owned row-authority proof exists
- widened core coverage for the new route contract in:
  - `crates/icydb-core/src/db/query/plan/covering/tests.rs`
  - `crates/icydb-core/src/db/executor/route/tests.rs`
  - `crates/icydb-core/src/db/session/tests/mod.rs`
  - `crates/icydb-core/src/db/executor/tests/semantics.rs`
  - checked-in coverage now locks:
    - route payload retention
    - explain-surface `existing_row_mode`
    - fail-closed `ByKeys DESC`
    - execution snapshot parity for the new PK-only covering nodes
- added outer-surface quickstart and PocketIC coverage for the first shipped
  cohort in:
  - `canisters/quickstart/src/lib.rs`
  - `testing/pocket-ic/tests/sql_canister.rs`
  - typed and generated SQL now both lock the new PK-only `CoveringRead` route
    and its explain/perf story for simple `SELECT id FROM User ORDER BY id ASC LIMIT 1`
    style queries
- recorded the first measurable runtime win from true covering execution on the
  PK-only cohort in:
  - `canisters/quickstart/src/perf.rs`
  - `testing/pocket-ic/tests/sql_canister.rs`
  - generated dispatch `SELECT id FROM User ORDER BY id ASC LIMIT 1` now
    averages about `337,485` local instructions
  - typed dispatch on the same query averages about `334,704`
  - against the older row-backed `SELECT age FROM User ORDER BY id ASC LIMIT 1`
    cohort, that is roughly a `17%` total drop and about a `21%`
    execute-phase drop
