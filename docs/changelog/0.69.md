# 0.69 Changelog

## 0.69.3 - 2026-04-06 - Generated Memory Registration Fix

### Summary

`0.69.3` is a narrow follow-up to `0.69.1`. It does not change the shipped
covering-read authority or executor behavior. Instead, it fixes the generated
canister bootstrap contract around the newer `canic-memory` API shape.

The important boundary correction is:

- generated code now re-queues the application memory range during codegen
- generated bootstrap now flushes pending eager-init registrations instead of
  trying to create the owner range again at runtime
- demo_rpg’s host-parallel test note now reflects that generated `db()`
  startup depends on pending eager-init flush, not runtime owner-range creation

So `0.69.3` is specifically the generated `icydb-build` memory-registration
fix, not another `0.69` executor slice.

### Changed

- updated generated canister memory registration in:
  - `crates/icydb-build/src/db.rs`
  - generated store wiring now emits:
    - `canic_memory::eager_init!(...)`
    - `canic_memory::ic_memory_range!(min, max)`
  - generated `ensure_memory_bootstrap()` now calls:
    - `MemoryApi::bootstrap_pending()`
  - this corrects the new `canic-memory` registration contract by making the
    generated canister flush the pending eager-init queue instead of trying to
    rebuild its owner range dynamically at bootstrap time
- refreshed the supporting demo_rpg test note in:
  - `canisters/demo_rpg/src/lib.rs`
  - the host-parallel bootstrap comment now explains the current generated
    behavior accurately: generated `db()` startup drains pending eager-init
    state, so tests must re-queue the application range before each
    bootstrap-dependent path

### Validation

- no additional tests were rerun for this changelog-only entry
- the code delta covered by this entry is limited to:
  - `crates/icydb-build/src/db.rs`
  - `canisters/demo_rpg/src/lib.rs`

## 0.69.2 - 2026-04-06 - Shared Covering Scan Kernel Win

### Summary

`0.69.2` does not widen the shipped `0.69` covering-read authority contract.
Instead, it ships one real shared execution-kernel improvement for already
admitted cursorless covering-read cohorts, keeps one persisted-row
serializer/deserializer cleanup, and records one measured-but-rejected
secondary covering experiment so the line does not accrete another low-signal
executor specialization.

The kept runtime win is shared, not a new narrow cohort rule:

- the execution kernel now tries the existing cursorless SQL direct covering
  scan materialization path before building a generic ordered key stream
- that change does not widen witness admission or row-authority rules; it only
  makes already-admitted direct-field covering routes cheaper at execution time
- the same kernel-owned change produces real same-query drops on both the
  `Character` and `ActiveUser` witness-backed direct-field cohorts

The kept serializer-side cleanup is still in the canonical row re-emission
path:

- structural row re-emission now rebuilds canonical slot payloads from the
  already-decoded structural value cache instead of re-entering raw field-byte
  decode after the reader has already validated the row
- canonical row emission now flattens slot payload bytes directly into the row
  payload/table envelope instead of restaging those same bytes through the
  mutable slot-writer buffer first

Separately, one earlier attempted `0.69` major-win experiment was measured on
the exact checked-in `Character` composite covering cohort and then rejected:

```sql
SELECT id, level, class_name
FROM Character
ORDER BY level ASC, class_name ASC, id ASC
LIMIT 2
```

The centralized executor-owned multi-component `row_check_required`
optimization only moved the exact `Character` ASC/DESC pair by about
`0.7%–1.1%`, so it was backed out instead of widening the runtime with another
narrow special case. The shipped `0.69.2` SQL win comes from the later shared
kernel bypass below, not from that rejected row-check experiment.

## 0.69.1 - 2026-04-06 - Canic Memory API Cleanup

### Summary

`0.69.1` is a narrow integration follow-up to `0.69.0`. It does not widen the
new covering-read authority contract. Instead, it rebases IcyDB's remaining
stable-memory bootstrap and commit-slot plumbing onto the cleaned-up
`canic-memory 0.25.10` public surface.

The important outcome is boundary cleanup, not new query semantics:

- generated canister bootstrap now uses the renamed owner-range bootstrap API
- commit-marker memory validation now uses the new pending-bootstrap and
  inspection/listing helpers
- the last direct low-level memory-registry init in IcyDB is gone
- facade SQL tests and other local bootstrap helpers now use the same public
  memory API that generated canisters use

So after `0.69.1`, IcyDB no longer mixes the public `canic-memory` boundary
with direct registry-runtime calls for its current integration.

## 0.69.0 - 2026-04-06 - True Covering Read Execution

### Summary

`0.69.0` starts the `0.69` line by turning covering reads from a
planner-only admission signal into a real execution contract with explicit
row-presence semantics. `0.68` had already widened the scalar read families
that could stay on index-backed routes; `0.69.0` makes the first part of that
contract executable without row payload reads instead of continuing to optimize
the row-materialized fallback around it.

The line now ships two explicit secondary authority modes rather than treating
all index-backed covering reads as one blurred class:

- `planner_proven` for primary-store shapes where the chosen access path already
  proves row authority
- `witness_validated` for narrow secondary/index-backed cohorts where IcyDB can
  prove the data store and index store are synchronized and therefore skip the
  extra row-existence probe without weakening stale-row safety

Current primary-store PK-only reads still fall into two planner-owned cohorts:

- authoritative primary-store traversal (`FullScan` and strict half-open
  `KeyRange`) reports `existing_row_mode = planner_proven` and may skip
  row-existence checks
- exact primary-key lookup (`ByKey` and canonical ascending `ByKeys`) reports
  `existing_row_mode = row_check_required` and keeps explicit row-presence
  checks even though the route is now first-class `CoveringRead`

Secondary covering routes from `0.68` no longer all behave the same. Narrow
single-field and filtered-composite cohorts now have an explicit
`witness_validated` route, while broader secondary and expression-backed
covering families still stay `row_check_required` until a later slice can
prove stronger authority semantics for them.
