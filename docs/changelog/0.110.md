# 0.110 Changelog

## 0.110.0

- `0.110.0` starts the grouped semantic-alignment line by teaching grouped `HAVING` to canonicalize one bounded searched-`CASE` family instead of leaving grouped boolean semantics structurally normalized but semantically second-tier relative to scalar `WHERE`.

- The shipped grouped canonicalization scope is intentionally narrow:
  - grouped searched `CASE` must already be admitted in grouped `HAVING`
  - the grouped `CASE` must carry an explicit `ELSE`
  - canonicalization stays inside the existing grouped boolean expression surface
  - omitted-`ELSE` grouped searched `CASE` remains out of scope in this patch

- The new grouped semantic canonicalization seam is shared, not duplicated:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs) now exports `canonicalize_grouped_having_bool_expr(...)`
  - fluent grouped `HAVING` append uses that seam in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/mutation.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/mutation.rs)
  - grouped SQL lowering uses that seam in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/aggregate.rs)

- The first shipped grouped canonical form follows the current grouped boolean expansion exactly, not a stronger simplifier. In particular, `CASE WHEN p THEN TRUE ELSE FALSE END` currently canonicalizes onto the grouped boolean tree equivalent of `COALESCE(p, FALSE) OR FALSE`, and the identity/explain proofs are pinned to that exact emitted form rather than to a further simplified `COALESCE(p, FALSE)` spelling that `0.110.0` does not yet claim.

- `0.110.0` also closes the grouped ownership seam that would otherwise have re-canonicalized omitted-`ELSE` SQL shapes accidentally after lowering. SQL-lowered grouped `HAVING` now binds through a preserve-shape path in:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/select/mod.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/query.rs)

- That split is the important behavioral boundary for this patch:
  - fluent grouped `HAVING` still uses the grouped semantic canonicalizer directly
  - SQL-lowered grouped `HAVING` preserves the grouped shape already chosen by lowering
  - grouped searched `CASE` without `ELSE` therefore stays on the original grouped `CASE` shape instead of collapsing by accident through the later intent append path

- Grouped explain parity now follows the same bounded semantic form. Equivalent explicit-`ELSE` grouped searched `CASE` `HAVING` queries surface the canonical grouped boolean tree in explain instead of echoing the original `CASE` spelling, while omitted-`ELSE` grouped searched `CASE` still surfaces the original grouped `CASE` tree. The new explain coverage landed in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs).

- Grouped identity follow-through is also now pinned at both the SQL/session and structural fluent seams:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_surface.rs) proves that explicit-`ELSE` grouped searched `CASE` and its shipped canonical grouped boolean form share one structural query identity, one plan fingerprint, and one shared query-plan cache entry
  - the same file also proves that omitted-`ELSE` grouped searched `CASE` stays identity-distinct from that explicit-`ELSE` canonical family
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests/cache_key.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/tests/cache_key.rs) pins the same distinction at the structural grouped cache-key seam for fluent grouped queries
