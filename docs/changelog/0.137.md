# 0.137.x

## 0.137.9

`0.137.9` seals the remaining SQL/session execution handoff seams identified by
the post-refactor db/ boundary audit. Projection path selection and grouped SQL
page-shape handling now cross executor-owned structural wrappers, so SQL/session
keeps only request adaptation, diagnostic callback wiring, labels/scales, cursor
display formatting, and statement DTO construction.

What changed:
- added executor-owned `StructuralProjectionRequest` and
  `execute_structural_projection_result` so projection execution orchestration
  lives behind one facade
- moved projection path selection into the executor facade, including
  projected-row `DISTINCT` raw-page suppression, covering projection selection,
  scalar retained-slot fallback execution, and distinct/non-distinct projection
  materialization dispatch
- narrowed the retained-slot scalar projection helper to executor visibility so
  SQL/session no longer imports scalar page execution internals
- changed SQL projection runtime to build the structural projection request,
  pass diagnostic recorders, and shape the returned rows into SQL payloads
- moved delete-returning `KernelRow` to value-row conversion into executor-owned
  `MaterializedProjectionRows`, removing the production SQL/session dependency
  on executor kernel row internals
- made `execute_scalar_aggregate_terminals` private to the executor aggregate
  module so global aggregate execution enters through
  `execute_structural_aggregate_result`
- added executor-owned `StructuralGroupedProjectionResult` so grouped SQL paths
  no longer expose `GroupedCursorPage` fields or page row counts to SQL/session
- added grouped response finalization for `StructuralGroupedProjectionResult`,
  preserving SQL/session ownership of grouped columns, fixed scales, cursor
  string formatting, and `SqlStatementResult::Grouped` construction

Retained behavior:
- SQL projection rows, `DISTINCT` ordering/window behavior, covering projection
  behavior, grouped rows, grouped cursors, aggregate rows, `HAVING`, diagnostics
  counters, cache behavior, and cursor bytes are unchanged
- executor projection still reuses the existing covering and materialization
  implementations; this patch only moves the coordinator boundary
- grouped execution, grouped aggregation, grouped ordering, grouped diagnostics,
  and grouped cursor behavior are unchanged
- SQL/parser/lowering, query/plan semantics, mutation execution, and persisted
  bytes are unchanged

executor move. It keeps the new aggregate handoff neutral, removes stale
SQL-owned wording from executor aggregate comments, and confirms SQL/session
does not retain aggregate expression evaluation or `HAVING` execution.

What changed:
- tightened executor aggregate comments so scalar aggregate terminal preparation
  is described as adapter-neutral structural execution rather than SQL-owned
  execution
- kept `StructuralAggregateResult` as an opaque executor transport wrapper with
  only a consumptive value-row handoff for adapters
- confirmed executor aggregate execution depends on canonical query/plan
  `Expr`, `ProjectionSpec`, and `FieldSlot` types, not SQL parser or lowering
  syntax types
- confirmed production SQL/session no longer imports grouped projection
  evaluators, scalar aggregate terminal internals, or aggregate `HAVING`
  execution helpers

Retained behavior:
- no SQL aggregate result, ordering, `HAVING`, `FILTER`, cache, cursor, or
  projection behavior changes
- SQL/session still owns SQL labels, fixed-scale metadata, cache attribution,
  and `SqlProjectionPayload` construction
- executor remains the owner of global aggregate reduction, `HAVING`, and final
  aggregate projection execution introduced in `0.137.7`

## 0.137.7

`0.137.7` moves global aggregate terminal preparation and post-reduction
execution behind the executor aggregate boundary. SQL/session now adapts lowered
SQL aggregate intent into a neutral structural aggregate request, resolves the
shared prepared plan once, and shapes the returned rows into SQL payloads.

What changed:
- added executor-owned `StructuralAggregateRequest`,
  `StructuralAggregateTerminal`, `StructuralAggregateTerminalKind`, and
  `StructuralAggregateResult` types for non-SQL aggregate execution handoff
- moved scalar aggregate terminal selection, count-equivalent terminal routing,
  scalar aggregate terminal compilation, terminal reduction dispatch, global
  aggregate `HAVING`, and final aggregate projection evaluation into
  `db/executor/aggregate/scalar_terminals`
- changed SQL global aggregate execution so it no longer imports executor
  scalar aggregate terminal internals or grouped projection evaluators
- changed SQL global aggregate execution to resolve one shared prepared plan and
  call `execute_structural_aggregate_result`, preserving shared query-plan cache
  reuse for `COUNT(*)` and non-null `COUNT(field)` shapes
- kept SQL projection labels, fixed-scale metadata, statement payload shaping,
  and SQL cache attribution in SQL/session

Retained behavior:
- SQL global aggregate rows, labels, fixed-scale metadata, `HAVING` semantics,
  `FILTER` semantics, expression aggregate inputs, duplicate aggregate outputs,
  and cache reuse behavior are unchanged
- grouped aggregate execution continues through the existing grouped executor
  runtime; this patch only moves the global aggregate session-owned terminal and
  post-reduction work
- SQL lowering still owns SQL aggregate syntax admission and lowered aggregate
  strategy construction
- projection `DISTINCT` execution remains executor-owned from `0.137.6`

## 0.137.6

`0.137.6` moves SQL `DISTINCT` projection materialization behind the
executor-owned projection boundary. SQL/session still admits SQL, invokes the
shared scalar execution path, and shapes response payloads, but projected-row
deduplication and the post-`DISTINCT` `LIMIT`/`OFFSET` window are now applied by
the executor projection materializer.

What changed:
- added an executor-owned `project_distinct_structural_projection_page` facade
  that consumes structural scalar pages and returns `MaterializedProjectionRows`
  through the same opaque result wrapper used by non-`DISTINCT` projection
  materialization
- moved projected-row `DISTINCT` accumulation, bounded-window handling, direct
  slot-row projection, direct data-row field projection, and scalar data-row
  fallback projection out of `db/session/sql/projection/runtime` and into
  `db/executor/projection/materialize`
- kept SQL projection diagnostics in SQL/session by passing counter callbacks
  through `ProjectionMaterializationMetricsRecorder`, so executor execution
  ownership does not also own adapter-local counter storage
- removed the SQL/session `finalize_sql_projection_rows` and
  `project_distinct_sql_projection_from_structural_page` helpers, eliminating
  the session-owned execution path for projected-row deduplication
- refreshed SQL projection runtime comments so `DISTINCT` is described as
  executor projection materialization while SQL/session remains responsible for
  response-payload shaping

Retained behavior:
- SQL `SELECT DISTINCT` result rows, ordering, `NULL` equality behavior,
  expression projection behavior, and `LIMIT`/`OFFSET` semantics are unchanged
- covering projection fast paths still avoid applying raw-row paging when
  projected-row `DISTINCT` must observe the full ordered input stream
- SQL/session still strips raw scalar paging before shared scalar execution for
  `DISTINCT`, then hands the structural page to executor projection
  materialization
- aggregate execution consolidation is not part of this patch

## 0.137.5

`0.137.5` is a boundary-clarity patch after the SQL/parser/lowering and
executor convergence work. It removes misleading SQL-owned names from shared
planner, fluent projection, and executor projection surfaces, and it narrows one
SQL-specific literal normalization re-export so future work has a cleaner
starting point for projection and aggregate ownership refactors.

What changed:
- renamed the SQL lowering ORDER-expression test seam from `parse_*` helpers to
  `lower_*_order_expr_text`, keeping token parsing visibly owned by
  `db/sql/parser` while lowering remains responsible for semantic adaptation
- renamed planner-owned function and aggregate render labels from `sql_label`
  to `canonical_label`, so canonical query/plan expression types no longer read
  as SQL-specific APIs
- renamed the fluent bounded projection helper method from `sql_label()` to
  `projection_label()`, preserving the same rendered labels while removing SQL
  ownership wording from the shared fluent projection contract
- removed `canonicalize_strict_sql_literal_for_kind` from the `db/query/plan`
  re-export surface; SQL lowering and SQL write handling now import the
  model-owned helper through SQL/model boundaries instead of through the plan
  module
- refreshed stale comments in query/plan expression, aggregate-input, covering,
  and builder projection code so shared semantic behavior is described as
  canonical or adapter-facing rather than SQL-owned
- renamed private SQL/session projection helpers so they read as SQL projection
  adapting from structural prepared plans, structural queries, or structural
  pages instead of suggesting a separate "structural SQL" ownership model
- cleaned executor projection comments so shared prepared projection bundles,
  structural pages, delete returning setup, and materialized projection rows are
  described as executor/adapter boundaries rather than SQL projection internals

Retained behavior:
- SQL parsing, lowering, query results, projection labels, aggregate results,
  cursor behavior, cache behavior, diagnostics meaning, and persisted bytes are
  unchanged
- parser-owned ORDER-expression AST parsing still feeds lowering-owned
  expression adaptation; semantic normalization remains in query/plan
- executor projection boundaries still return structural wrappers such as
  `CoveringProjectionRows` and `MaterializedProjectionRows`; raw
  `Vec<Vec<Value>>` storage remains confined to private executor helpers or
  SQL/session payload shaping
- SQL `DISTINCT` projection materialization still lives in the SQL/session
  projection runtime for now; moving that behind an executor-owned facade is a
  separate follow-up refactor
- aggregate execution consolidation is not part of this patch

## 0.137.4

`0.137.4` starts reducing the shared query execution floor after the `db/`
boundary cleanup. The main scalar path now avoids several common setup costs:
commit recovery skips absent-marker checks after startup, trivial scalar loads
can bypass the full predicate/access planner, and cached scalar prepared plans
stay shared through runtime assembly instead of cloning the full
`AccessPlannedQuery` at the scalar entrypoint.

What changed:
- added a commit-marker presence flag so normal query execution can skip
  `commit_marker_present_fast()` after startup recovery when no marker can be
  present
- added a conservative trivial scalar-load planner fast path for full scalar
  loads with no filter, no distinct, no key-access override, and either no
  paging or primary-key-only ordering
- wired the session shared query-plan cache to use the trivial scalar fast path
  before constructing planner-visible index sets or normalized predicate
  fingerprints
- added equivalence coverage proving the trivial scalar load fast path emits the
  same access-planned shape as the general planner for supported no-order and
  primary-order cases, while rejecting secondary-order cases
- skipped executor authority/access validation when scalar runtime preparation
  is already sourced from the validated prepared-load boundary
- cached scalar `ExecutionPreparation` as a lazy prepared-plan resident so scalar
  execution does not rebuild slot-map/runtime-filter preparation on every run
- changed scalar runtime extraction to carry a shared `PreparedScalarPlanCore`
  handle through route/materialization assembly, borrowing the logical plan and
  lowered index specs instead of cloning owned `AccessPlannedQuery` and spec
  vectors from cached prepared plans
- kept raw SQL retained-slot/materialized scalar helpers on a prepared lowered
  adapter so they preserve existing owned-plan behavior while entering the same
  scalar runtime spine

Measured impact:
- fluent `db().load::<PerfAuditUser>().order_term(asc("id")).limit(2)` moved
  from about `417k` instructions before these common-path cuts to about `342k`
  in the latest PocketIC probe
- repeated fluent ordered scalar loads also moved down materially, with
  `repeat.user.age.order_only.asc.limit3.runs100` measuring about `472k` after
  the shared-plan clone cut
- SQL primary-key-only projection remains essentially flat because that path is
  already dominated by SQL projection/retained-slot shaping rather than the full
  scalar load path
- SQL retained-slot `ORDER BY id` projection still has follow-up overhead to
  remove; this patch does not claim that path is finished

Retained behavior:
- query results, SQL/fluent convergence, cursor behavior, delete behavior,
  grouped execution, diagnostics trace shape, and persisted bytes are unchanged
- the trivial scalar fast path is intentionally narrow and falls back to the
  general planner for predicates, distinct queries, key access, secondary
  ordering, grouped queries, and unsupported pagination shapes
- scalar route selection and residual-retry semantics still use the existing
  executor route/runtime owners
- raw SQL projection paths still own projected-row distinct/window behavior and
  do not use the full scalar-load fast path unless they are explicitly eligible

## 0.137.3

`0.137.3` finishes the post-audit boundary stabilization pass for `db/`.
The SQL statement path now keeps syntax normalization, execution, mutation
staging, projection materialization, and diagnostics handoffs on clearer owners
after the aggregate and write alignment work from the previous patches.

What changed:
- moved table-alias identifier normalization for SQL `SELECT`, `UPDATE`, and
  `DELETE` out of parser statement helpers and into SQL lowering, so the parser
  preserves raw alias-qualified identifiers while lowering owns normalized
  statement shape before binding and planning
- rehomed SQL covering projection execution from `session/sql` into
  executor-owned projection helpers, leaving session code to call one executor
  boundary and shape the SQL response payload
- hid covering projection pure/hybrid implementation functions behind the
  executor projection module and kept the SQL-visible covering API to one
  executor-facing function
- added primary-store covering execution for planner-proven primary-key-only SQL
  projections, so queries such as `SELECT id ... ORDER BY id ... LIMIT n` can
  project from streamed `DataKey`s instead of reading persisted rows
- kept exact primary-key lookup covering plans on the existing row-presence
  checking path, so `ByKey` and `ByKeys` routes still verify stored rows before
  emitting output
- hid `StructuralMutationBatchItem` from SQL write execution by accepting
  lowered structural `(key, patch)` rows at the save executor boundary and
  building private batch items inside mutation/save code
- stopped exposing `StructuralCursorPagePayload` to SQL projection
  materialization; SQL now dispatches through a page adapter instead of matching
  executor payload variants directly
- added diagnostics-only scalar aggregate terminal attribution for base-row
  materialization, reducer ingestion, expression/filter evaluation counts,
  row counts, terminal counts, interned expression counts, and current buffered
  sink mode
- renamed misleading plan-label helpers away from `render_*_sql_label` wording
  and adjusted cursor imports to depend on access contracts rather than reaching
  through executor re-exports
- refreshed stale ownership comments so alias normalization is described as
  lowering-owned, covering projection execution as executor-owned, and the
  generated-index predicate parser as a separate predicate DSL rather than the
  main SQL statement parser path

Retained behavior:
- SQL query results, SQL write behavior, cursor behavior, grouped execution,
  aggregate results, cache behavior, diagnostics meaning, and persisted bytes
  are unchanged
- SQL `DISTINCT` projection still owns projected-row deduplication and
  post-projection windowing at the SQL response boundary
- covering projection remains SQL-specific in name and behavior, but the
  execution path is now executor-owned
- primary-store covering applies only to planner-proven primary full-scan/range
  routes; lookup routes that require row checks still preserve their existing
  row-read behavior
- strong relation validation for SQL statement-atomic `INSERT`/`UPDATE`
  remains committed-store-only
- the standalone generated-index predicate parser remains a separate predicate
  DSL and was not folded into the main SQL statement parser/lowering path

## 0.137.2

`0.137.2` continues the scalar aggregate terminal cleanup by reducing avoidable
work inside SQL global aggregate execution. Repeated aggregate input and filter
expressions are interned once per terminal set, and scalar aggregate reducers now
consume the scalar runtime's post-access/windowed row boundary directly instead
of returning a retained-slot page to the SQL aggregate layer.

What changed:
- interned duplicate scalar aggregate input expressions and aggregate `FILTER`
  expressions at terminal preparation time, so shared expressions such as
  `SUM(x + 1)` and `AVG(x + 1)` evaluate at most once per input row
- kept field-backed aggregate inputs out of the expression table, preserving the
  direct retained-slot read path for simple field aggregates
- added a scalar aggregate row-sink entrypoint that runs the existing scalar
  prepared access/window plan, suppresses external cursor emission, and feeds
  post-access/windowed `KernelRow`s into executor-owned reducers
- removed the retained-slot `StructuralCursorPage` handoff from SQL global
  aggregate terminal execution, so SQL aggregate code no longer consumes an
  intermediate aggregate row page
- kept the residual-retry and scalar-window contracts aligned with normal scalar
  execution by reusing the same route hints, post-access filtering, ordering,
  `LIMIT`, and `OFFSET` handling

Retained behavior:
- SQL aggregate results, aggregate `FILTER` behavior, aggregate-level
  `DISTINCT`, `HAVING`, output expressions, and empty-window semantics are
  unchanged
- `COUNT(*)` and provably non-null unfiltered `COUNT(field)` still use the shared
  scalar count terminal and retain their existing cache behavior
- grouped execution, grouped `DISTINCT` policy, cursor tokens, persisted bytes,
  typed save paths, and SQL write behavior are unchanged
- terminal metadata remains uncached and outside `SharedPreparedExecutionPlan`;
  the shared scalar plan still owns access, filtering, ordering, and window
  semantics

## 0.137.1

`0.137.1` stabilizes the `db/` audit follow-through by removing two remaining
SQL-owned execution/write lanes. SQL global aggregate reduction now runs through
executor-owned scalar aggregate terminals over the shared scalar prepared plan,
and SQL `INSERT`/`UPDATE` now stage internally lowered structural patches before
opening one commit window.

What changed:
- added executor-owned scalar aggregate terminal types for SQL global
  `SUM`, `AVG`, `MIN`, `MAX`, filtered aggregates, expression aggregates, and
  aggregate-level `DISTINCT`
- kept `COUNT(*)` and provably non-null unfiltered `COUNT(field)` on the shared
  scalar count terminal so existing count cache behavior remains intact
- removed the SQL-owned projected `Vec<Value>` aggregate reducer lane, so global
  aggregate reduction no longer materializes one projected value vector per
  aggregate
- preserved SQL scalar window semantics for aggregate input: `WHERE`, `ORDER BY`,
  `LIMIT`, and `OFFSET` still shape the input rows before terminal reduction
- added a private structural mutation batch boundary for internally lowered SQL
  writes, reusing existing structural save preflight and atomic commit-window
  ownership
- routed SQL multi-row `INSERT`, `INSERT SELECT`, and matched-row `UPDATE`
  through one staged structural batch so duplicate keys, late validation
  failures, and unique-index conflicts commit zero rows
- kept SQL `DELETE` on its existing delete executor path, which was already
  aligned with shared delete planning and commit ownership
- documented and tested that strong relation validation remains
  committed-store-only: same-statement SQL relation targets must already exist
  before the statement starts

Retained behavior:
- SQL aggregate query results, `HAVING` handling, aggregate output expressions,
  cursor suppression, and grouped `DISTINCT` policy are unchanged
- terminal metadata remains uncached and outside `SharedPreparedExecutionPlan`;
  the shared plan continues to own scalar access, filtering, ordering, and window
  semantics
- single-row SQL `INSERT` and `UPDATE` results are unchanged, but multi-row
  statements now use one atomic commit window
- typed save paths, SQL `DELETE`, persisted bytes, cursor behavior, and response
  materialization are unchanged

## 0.137.0

`0.137.0` closes the holistic `db/` audit by removing the remaining
pre-semantic SQL parser rewrites from the predicate path. SQL parsing now keeps
raw syntax-shaped predicate intent, SQL expression lowering maps syntax onto
planner expressions, and the shared boolean-expression normalization path owns
the canonical planner form that feeds query planning, cache identity, and
execution.

What changed:
- added a raw `SqlExpr::Like` parser node so `LIKE`, `NOT LIKE`, `ILIKE`, and
  `NOT ILIKE` preserve their source-level intent until expression lowering
- moved prefix-only `LIKE` validation and `LIKE/ILIKE -> STARTS_WITH` lowering
  from parser code into SQL expression lowering
- kept field-first comparison canonicalization out of WHERE parsing and in the
  shared boolean-expression normalization path, preserving the existing
  normalized planner shape
- kept direct WHERE text-predicate casefold behavior, such as
  `STARTS_WITH(UPPER(name), 'AL')`, in boolean-expression normalization instead
  of parser rewrites
- updated parser tests to assert syntax-preserving AST output and lowering
  tests to keep SQL/fluent predicate parity locked
- added projection coverage proving non-WHERE expression lowering preserves raw
  `LIKE` targets before boolean-expression normalization owns predicate cleanup
- taught SQL identifier and alias normalization helpers, plus db-local test
  helpers, to recurse through the new raw `LIKE` expression node

Retained behavior:
- supported prefix `LIKE` and `ILIKE` queries still plan and execute as the
  same `STARTS_WITH` predicates as before
- unsupported non-prefix `LIKE` patterns still fail closed with the same public
  unsupported-feature label
- SQL and fluent predicate parity, plan identity, and direct text-predicate
  behavior are unchanged
- parser-only behavior is intentionally narrower: parser tests now validate raw
  syntax ownership instead of expecting planner-owned predicate shapes
- persisted bytes, cursor behavior, commit behavior, and response materialization
  are unchanged
