# 0.137.x

## 0.137.2

`0.137.2` continues the scalar aggregate terminal cleanup by reducing avoidable
work inside SQL global aggregate execution. Repeated aggregate input and filter
expressions are interned once per terminal set, and scalar aggregate reducers now
consume the scalar runtime's post-access/windowed row boundary directly instead
of returning a retained-slot page to the SQL aggregate layer.

What changed:
- interned duplicate scalar aggregate input expressions and aggregate `FILTER`
  expressions at terminal preparation time, so shared expressions such as
  `SUM(x + 1)` and `AVG(x + 1)` evaluate at most once per input row
- kept field-backed aggregate inputs out of the expression table, preserving the
  direct retained-slot read path for simple field aggregates
- added a scalar aggregate row-sink entrypoint that runs the existing scalar
  prepared access/window plan, suppresses external cursor emission, and feeds
  post-access/windowed `KernelRow`s into executor-owned reducers
- removed the retained-slot `StructuralCursorPage` handoff from SQL global
  aggregate terminal execution, so SQL aggregate code no longer consumes an
  intermediate aggregate row page
- kept the residual-retry and scalar-window contracts aligned with normal scalar
  execution by reusing the same route hints, post-access filtering, ordering,
  `LIMIT`, and `OFFSET` handling

Retained behavior:
- SQL aggregate results, aggregate `FILTER` behavior, aggregate-level
  `DISTINCT`, `HAVING`, output expressions, and empty-window semantics are
  unchanged
- `COUNT(*)` and provably non-null unfiltered `COUNT(field)` still use the shared
  scalar count terminal and retain their existing cache behavior
- grouped execution, grouped `DISTINCT` policy, cursor tokens, persisted bytes,
  typed save paths, and SQL write behavior are unchanged
- terminal metadata remains uncached and outside `SharedPreparedExecutionPlan`;
  the shared scalar plan still owns access, filtering, ordering, and window
  semantics

## 0.137.1

`0.137.1` stabilizes the `db/` audit follow-through by removing two remaining
SQL-owned execution/write lanes. SQL global aggregate reduction now runs through
executor-owned scalar aggregate terminals over the shared scalar prepared plan,
and SQL `INSERT`/`UPDATE` now stage internally lowered structural patches before
opening one commit window.

What changed:
- added executor-owned scalar aggregate terminal types for SQL global
  `SUM`, `AVG`, `MIN`, `MAX`, filtered aggregates, expression aggregates, and
  aggregate-level `DISTINCT`
- kept `COUNT(*)` and provably non-null unfiltered `COUNT(field)` on the shared
  scalar count terminal so existing count cache behavior remains intact
- removed the SQL-owned projected `Vec<Value>` aggregate reducer lane, so global
  aggregate reduction no longer materializes one projected value vector per
  aggregate
- preserved SQL scalar window semantics for aggregate input: `WHERE`, `ORDER BY`,
  `LIMIT`, and `OFFSET` still shape the input rows before terminal reduction
- added a private structural mutation batch boundary for internally lowered SQL
  writes, reusing existing structural save preflight and atomic commit-window
  ownership
- routed SQL multi-row `INSERT`, `INSERT SELECT`, and matched-row `UPDATE`
  through one staged structural batch so duplicate keys, late validation
  failures, and unique-index conflicts commit zero rows
- kept SQL `DELETE` on its existing delete executor path, which was already
  aligned with shared delete planning and commit ownership
- documented and tested that strong relation validation remains
  committed-store-only: same-statement SQL relation targets must already exist
  before the statement starts

Retained behavior:
- SQL aggregate query results, `HAVING` handling, aggregate output expressions,
  cursor suppression, and grouped `DISTINCT` policy are unchanged
- terminal metadata remains uncached and outside `SharedPreparedExecutionPlan`;
  the shared plan continues to own scalar access, filtering, ordering, and window
  semantics
- single-row SQL `INSERT` and `UPDATE` results are unchanged, but multi-row
  statements now use one atomic commit window
- typed save paths, SQL `DELETE`, persisted bytes, cursor behavior, and response
  materialization are unchanged

## 0.137.0

`0.137.0` closes the holistic `db/` audit by removing the remaining
pre-semantic SQL parser rewrites from the predicate path. SQL parsing now keeps
raw syntax-shaped predicate intent, SQL expression lowering maps syntax onto
planner expressions, and the shared boolean-expression normalization path owns
the canonical planner form that feeds query planning, cache identity, and
execution.

What changed:
- added a raw `SqlExpr::Like` parser node so `LIKE`, `NOT LIKE`, `ILIKE`, and
  `NOT ILIKE` preserve their source-level intent until expression lowering
- moved prefix-only `LIKE` validation and `LIKE/ILIKE -> STARTS_WITH` lowering
  from parser code into SQL expression lowering
- kept field-first comparison canonicalization out of WHERE parsing and in the
  shared boolean-expression normalization path, preserving the existing
  normalized planner shape
- kept direct WHERE text-predicate casefold behavior, such as
  `STARTS_WITH(UPPER(name), 'AL')`, in boolean-expression normalization instead
  of parser rewrites
- updated parser tests to assert syntax-preserving AST output and lowering
  tests to keep SQL/fluent predicate parity locked
- added projection coverage proving non-WHERE expression lowering preserves raw
  `LIKE` targets before boolean-expression normalization owns predicate cleanup
- taught SQL identifier and alias normalization helpers, plus db-local test
  helpers, to recurse through the new raw `LIKE` expression node

Retained behavior:
- supported prefix `LIKE` and `ILIKE` queries still plan and execute as the
  same `STARTS_WITH` predicates as before
- unsupported non-prefix `LIKE` patterns still fail closed with the same public
  unsupported-feature label
- SQL and fluent predicate parity, plan identity, and direct text-predicate
  behavior are unchanged
- parser-only behavior is intentionally narrower: parser tests now validate raw
  syntax ownership instead of expecting planner-owned predicate shapes
- persisted bytes, cursor behavior, commit behavior, and response materialization
  are unchanged
