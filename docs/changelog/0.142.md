# 0.142.x

## 0.142.0

`0.142.0` turns the `FieldPath` preparation from `0.141.6` into scan-time
execution for structural value-storage fields. SQL projection and filtering can
now evaluate nested map paths such as `profile.rank` by resolving the
top-level slot once, walking bounded value-storage slices, and materializing
only the selected nested value when the query shape requires it.

### ➕ Added

- added projection execution for nested field paths rooted at existing
  top-level model fields
- added scan-only predicate execution for nested field-path comparisons
- added missing-path handling where projection returns `NULL` and predicates
  reject the row
- added a compiled path representation so nested segment preparation happens
  once during scalar lowering instead of once per scanned row

### 🔧 Changed

- centralized nested map traversal behind the projection path resolver so
  projection and predicate execution share one path pipeline
- routed simple nested scalar predicates through bounded byte slices instead of
  constructing full runtime `Value`s
- added direct bounded scalar decode for `bool`, `i64`, `u64`, and text
  predicate comparisons
- precomputed path segment bytes and used byte equality for map-key lookup,
  avoiding repeated UTF-8 decoding during row scans
- kept unsupported predicate shapes on the existing full-decode fallback path

### 🧭 Boundary notes

- this remains scan-only execution; no planner pushdown, path indexing, string
  interning, or hash lookup is introduced
- nested traversal stays inside the projection resolver and does not expose
  `ValueStorageView` to planner or SQL layers
- direct top-level field behavior is unchanged
- invalid structure still fails closed instead of silently falling back
