# IcyDB 0.193 Changelog

## 0.193.4

### 📝 Summary

- Makes the default read-admission contract discoverable from the main
  developer, setup, public facade, query, and SQL documentation.
- Clarifies that ordinary typed/fluent reads are bounded by default, while
  generated readonly SQL and `execute_sql_query` remain trusted/admin lanes.
- Extends the read-admission invariant guard so these discovery links and
  trusted SQL warnings cannot silently drift.
- Keeps the no-default `icydb-core` test lint target clean without weakening
  production or all-features dead-code linting.

### 📚 Documentation

- Added top-level README guidance that ordinary typed/fluent reads apply the
  default bounded read-admission gate after endpoint-owned caller
  authorization.
- Added README SQL-lane wording warning that `execute_sql_query` is
  trusted/admin SQL and is not public-safe for caller-controlled SQL by itself.
- Added install/setup guidance clarifying that generated readonly SQL is a
  controller-gated admin surface, not a generated public read endpoint.
- Linked the read-admission contract from the foundation, query contract, query
  practice, SQL subset, README, setup, and public facade docs.
- Tightened facade SQL helper rustdoc so hidden attribution helpers keep the
  same trusted/admin lane contract as `execute_sql_query`.
- Scoped no-default test-target dead-code allowance to `icydb-core`, where the
  target intentionally compiles shared test helpers whose consumers live behind
  SQL, SQL-explain, or diagnostics test features.

### 🧪 Testing

- Extended the read-admission invariant guard to require the documentation
  discovery links for developer, setup, query, SQL, and facade entry points.
- Added invariant checks for README default bounded-read guidance, README SQL
  lane wording, generated readonly SQL setup wording, public facade default
  read guidance, and trusted SQL helper warnings.
- Verified formatting, invariant checks, facade no-default compilation, facade
  all-features clippy, core no-default/all-features clippy, and whitespace
  validation.

### 🧭 Migration Notes

- No public API, generated endpoint, persisted format, cursor format, response
  shape, cache identity, execution behavior, or diagnostic-code change is
  introduced.
- This patch is documentation and regression-guard hardening around the
  existing 0.192 read-admission hard cut.

## 0.193.3

### 📝 Summary

- Connects every stable read-admission rejection reason to a documented
  developer-facing query fix.
- Guards the internal rejection enum and public diagnostic-code enum so they
  stay aligned one-for-one.
- Adds CLI diagnostic coverage proving every read-admission reason renders a
  non-empty fix hint.

### 📚 Documentation

- Added a common read-admission rejection/fix table to the read-admission
  contract.

- Documented fixes for missing bounds, unindexed access, full scans, estimated

  budget failures, EXPLAIN lane misuse, disabled introspection, and unsupported
  trusted-SQL lane usage.

  `QueryReadAdmissionCode` variants from source and require a matching
  documentation row for every variant.
- Extended the same guard to compare internal `QueryAdmissionRejection`
  variants against public `QueryReadAdmissionCode` variants one-for-one.
- Added a CLI diagnostic witness that every read-admission rejection code
  renders a fix hint.
- Verified read-admission invariants, CLI diagnostics, CLI no-default
  compilation, CLI clippy, SQL-feature read-admission behavior filters,
  formatting, and whitespace validation.

### 🧭 Migration Notes

- No public API, generated endpoint, persisted format, cursor format, response
  shape, cache identity, execution behavior, or diagnostic-code change is
  introduced.
- This patch improves documentation and regression evidence around the existing
  default read-admission hard cut.

## 0.193.2

### 📝 Summary

- Clarifies public facade method documentation for ordinary typed/fluent read
  execution.
- Makes the default bounded read-admission gate visible directly on the methods

- Keeps trusted execution documented as an explicit controller/admin path.
- Extends the read-admission invariant guard so public facade method docs cannot

- Documented `DbSession::execute_query` as the caller-facing ordinary read path

  response shapes.
- Documented scalar `execute_rows` as a scalar-only ordinary read path and
  pointed grouped callers back to `execute().into_grouped()`.
- Documented cursor-paged `execute` / `execute_paged` as ordinary bounded
  cursor-pagination paths.
- Documented `QueryResponse` variants and shape-specific consumers so scalar
  and grouped response handling are visible from the public response type.
- Tightened trusted-read wording so `trusted_read_unchecked()` and
  `*_trusted` helpers remain visibly admin/maintenance only.
- Added a read-admission contract section describing the invariant guard for
  public facade helper names, generated SQL authority, facade method docs, and
  default budget wording.

### 🧪 Testing

  ordinary execution, grouped response consumption, and cursor-paged execution.
- Added invariant checks for the default returned-row, response-byte, group,
  group-byte, and distinct-entry budget constants alongside their documented
  contract wording.
- Verified facade all-features and no-default compilation.
- Verified read-admission invariants, facade clippy, formatting, and whitespace
  validation.

### 🧭 Migration Notes

  introduced.

  methods.

## 0.193.0

### 📝 Summary

- Starts the public-query ergonomics line after the 0.192 read-admission hard
  cut.
- Makes read-admission CLI diagnostics point developers toward practical fixes
  instead of only reporting the rejection reason.
- Adds executable default-path witnesses for ordinary fluent reads that must
  fail closed before row execution.

### 🔧 Changed

- CLI rendering for `QueryReadAdmission` diagnostics now includes short fix
  hints for common public-query failures.
- Hints point toward concrete next steps: add an index, tighten the predicate,
  lower row or response bounds, use cursor pagination instead of `OFFSET`, add
  grouped hard limits, or move broad scans behind a trusted/admin endpoint.
- Diagnostic codes, diagnostic details, and error taxonomy remain unchanged;
  only the rendered human-facing CLI text is more actionable.

### 🧪 Testing

- Added default fluent execution witnesses proving ordinary `execute()` rejects
  non-zero `OFFSET`, materialized sorts, and grouped reads without query-owned
  hard limits.
- Kept the trusted bypass witness explicit so admin/maintenance reads remain a
  deliberate call-site choice rather than a policy-builder variant.
- Verified focused CLI diagnostic tests, SQL-feature read-admission tests,
  read-admission invariants, CLI clippy, `icydb-core` SQL-feature test clippy,
  CLI no-default compilation, formatting, and whitespace validation.

### 🧭 Migration Notes

- No public API, generated endpoint, persisted format, cursor format, response
  shape, cache identity, or read-admission diagnostic-code change is introduced.
- CLI users will see more detailed read-admission error text for rejected public
  query shapes.
- Ordinary typed/fluent reads remain bounded by default; trusted/admin reads
  remain explicit.
