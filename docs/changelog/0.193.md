# IcyDB 0.193 Changelog

## 0.193.0

### 📝 Summary

- Starts the public-query ergonomics line after the 0.192 read-admission hard
  cut.
- Makes read-admission CLI diagnostics point developers toward practical fixes
  instead of only reporting the rejection reason.
- Adds executable default-path witnesses for ordinary fluent reads that must
  fail closed before row execution.

### 🔧 Changed

- CLI rendering for `QueryReadAdmission` diagnostics now includes short fix
  hints for common public-query failures.
- Hints point toward concrete next steps: add an index, tighten the predicate,
  lower row or response bounds, use cursor pagination instead of `OFFSET`, add
  grouped hard limits, or move broad scans behind a trusted/admin endpoint.
- Diagnostic codes, diagnostic details, and error taxonomy remain unchanged;
  only the rendered human-facing CLI text is more actionable.

### 🧪 Testing

- Added default fluent execution witnesses proving ordinary `execute()` rejects
  non-zero `OFFSET`, materialized sorts, and grouped reads without query-owned
  hard limits.
- Kept the trusted bypass witness explicit so admin/maintenance reads remain a
  deliberate call-site choice rather than a policy-builder variant.
- Verified focused CLI diagnostic tests, SQL-feature read-admission tests,
  read-admission invariants, CLI clippy, `icydb-core` SQL-feature test clippy,
  CLI no-default compilation, formatting, and whitespace validation.

### 🧭 Migration Notes

- No public API, generated endpoint, persisted format, cursor format, response
  shape, cache identity, or read-admission diagnostic-code change is introduced.
- CLI users will see more detailed read-admission error text for rejected public
  query shapes.
- Ordinary typed/fluent reads remain bounded by default; trusted/admin reads
  remain explicit.
