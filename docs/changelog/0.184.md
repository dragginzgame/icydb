# IcyDB 0.184 Changelog

## 0.184.4

### 📝 Summary

- Continues the query-engine audit by extending the lowered expression-analysis
  artifact through more SQL lowering paths.
- Removes repeated expression walks and aggregate relowering in ORDER BY,
  HAVING, grouped aggregate validation, and global aggregate terminal
  interning.
- Keeps this as a cleanup slice: no SQL syntax, payload, cursor, persistence,
  or public endpoint behavior changes are intended.

### 🔧 Changed

- Lowered SQL `ORDER BY` terms now carry expression analysis so DISTINCT
  projection validation can reuse the shared field-proof artifact.
- Lowered SQL `HAVING` clauses now carry analyzed lowered expressions; grouped
  HAVING aggregate-slot resolution and global aggregate HAVING validation
  consume that artifact.
- Grouped SELECT artifacts now carry validated planner-owned aggregate
  expressions through HAVING binding and query application.
- Aggregate-call lowering shapes now retain analyzed grouped aggregate input
  and `FILTER` expressions before final aggregate construction.
- Global aggregate terminal collection now retains semantic keys beside
  terminals, so projection and HAVING terminal interning compare retained keys
  directly.

### 🧹 Cleanup

- Removed repeated direct-field and aggregate-reference walks from DISTINCT
  ORDER BY and HAVING validation paths.
- Removed grouped SELECT relowering of parser aggregate calls after the grouped
  aggregate list has already been validated.
- Removed repeated semantic-key reconstruction for already-retained global
  aggregate terminals during terminal deduplication.
- Kept global aggregate terminal input/filter validation facts deferred for a
  future terminal-preparation artifact instead of folding that broader design
  into this cleanup.

### 🧪 Testing

- Revalidated focused aggregate lowering coverage.
- Revalidated grouped aggregate input and unknown-field diagnostics.
- Revalidated broad grouped SQL/execution coverage and global aggregate
  coverage.
- Revalidated SQL-feature core clippy, formatting, and whitespace checks.

### 🧭 Migration Notes

- No schema migration is required.
- No SQL syntax, result payload, cursor format, persistence format, or public
  endpoint behavior changes are intended.

## 0.184.3

### 📝 Summary

- Continues the query-engine audit by tightening proof around encoded
  index-predicate execution and predicate-pushdown observability.
- Compares compiled byte-key predicates against canonical runtime predicate
  semantics for strict compare trees, membership, sorted large `IN`, text
  prefix bounds, and conservative `AND` prefilters.
- Moves verbose predicate-pushdown labels onto planner-owned diagnostics while
  preserving the existing compact label surface.

### 🔧 Changed

- Encoded index-predicate membership now switches to sorted lookup at 16
  candidates instead of 32, based on the native threshold benchmark.
- Finalized static planning now carries planner-owned predicate-pushdown
  diagnostics beside the residual-filter contract.
- Verbose EXPLAIN now renders predicate-pushdown outcome and reason fields
  from that finalized planning contract, covering no-filter,
  expression-only/no-subset, access-applied, residual-after-access, and
  full-scan fallback cases.
- The legacy compact `diag.p.predicate_pushdown` label remains stable while
  the new outcome/reason diagnostics provide a less ambiguous audit surface.

### 🧹 Cleanup

- Removed late verbose EXPLAIN fallback classification from rendered predicate
  trees.
- Kept strict index-prefilter observability route-owned through
  `diag.r.predicate_stage=index_prefilter(strict_all_or_none)` instead of
  folding executor route-preparation facts into the planner contract.

### 🧪 Testing

- Added encoded index-predicate parity coverage against canonical runtime
  predicate evaluation.
- Added an ignored native threshold benchmark for linear versus sorted encoded
  membership evaluation.
- Added finalized-planning contract tests for no-filter, expression-only,
  access-applied, residual-after-access, and full-scan fallback diagnostics.
- Updated verbose EXPLAIN coverage to lock the new predicate-pushdown outcome
  and reason fields.
- Revalidated focused planner diagnostics, verbose EXPLAIN, formatting,
  whitespace, and SQL-feature library clippy checks.

### 🧭 Migration Notes

- No schema migration is required.
- No SQL syntax, result payload, cursor format, persistence format, or public
  endpoint behavior changes are intended.

## 0.184.2

### 📝 Summary

- Continues the query-engine audit with contract consolidation around SQL
  expression analysis, filter lowering, residual planning, visible index
  candidates, and EXPLAIN diagnostics.
- Keeps runtime query behavior unchanged while making planner/executor
  handoff facts more explicit and less likely to drift.
- Leaves broader pushdown coverage/fallback reason design and operator-level
  planning for later slices.

### 🔧 Changed

- Lowered SQL expression analysis now records aggregate leaves in lowered
  expression order, and global aggregate terminal interning consumes those
  analysis-owned leaves instead of walking the same expression again.
- SQL filter lowering now centralizes visible-expression and
  predicate-pushdown construction in `LoweredSqlFilter` for scalar SELECT,
  grouped SELECT, global aggregate, DELETE, and UPDATE filters.
- Finalized static planning now stores residual expression, residual predicate
  subset, and compiled runtime filter program in one residual-filter contract.
- Runtime-visible accepted indexes now cache one sorted reduced semantic
  candidate list in `VisibleIndexes`, reused by access planning,
  residual-burden reranking, and EXPLAIN access-choice finalization.
- Residual EXPLAIN nodes and verbose route diagnostics now report residual
  filter shape from the planner-owned residual-filter contract.

### 🧹 Cleanup

- Removed another aggregate-only expression walk from global aggregate
  projection lowering.
- Removed loose sibling residual-filter fields from finalized static planning
  in favor of a single contract artifact.
- Removed obsolete accepted-index wrapper/rerank flows after the runtime
  visible-index boundary began carrying reduced semantic candidates directly.
- Kept accepted field-path contracts available for the order-only fallback path
  that still needs field-path-specific order proof.

### 🧪 Testing

- Added expression-analysis coverage for aggregate leaves without field-root
  leakage.
- Added residual-filter contract coverage for expression-only and
  expression-plus-predicate residual shapes.
- Extended visible-index bridge coverage to prove cached accepted semantic
  candidates stay sorted and count-aligned with accepted metadata.
- Updated execution and verbose EXPLAIN snapshots to lock residual-filter
  shape diagnostics.
- Revalidated focused SQL lowering, query-plan, planner, session EXPLAIN,
  verbose diagnostics, SQL EXPLAIN, snapshot, formatting, whitespace, and
  SQL-feature library clippy checks.

### 🧭 Migration Notes

- No schema migration is required.
- No SQL syntax, result payload, cursor format, persistence format, or public
  endpoint behavior changes are intended.

## 0.184.1

### 📝 Summary

- Continues the query-engine audit with targeted convergence work rather than
  a broad planner or operator rewrite.
- Reuses the already compiled source query artifact for `INSERT ... SELECT`
  execution.
- Routes builder preview projection evaluation through the shared
  `CompiledExpr` path.
- Starts turning lowered SQL expression facts into an explicit artifact used by
  grouped and global aggregate projection lowering.

### 🔧 Changed

- `INSERT ... SELECT` write execution now consumes the compiled source query
  artifact instead of preparing and binding the source SELECT again at
  execution time.
- Builder preview projection evaluation now compiles through
  `ScalarProjectionExpr` / `CompiledExpr` over a one-slot preview reader,
  removing the old direct recursive preview evaluator.
- Lowered SQL expression analysis now gathers aggregate presence, direct field
  roots, field-path presence, and unknown-field diagnostics in one pass.
- Grouped projection validation and grouped aggregate collection now consume
  the shared expression-analysis proof for grouped-field authority.
- Grouped and global projection lowering now use an `AnalyzedLoweredExpr`
  artifact when the lowered SELECT item and its analysis facts are consumed
  together.
- Index-store range, entry-read, and prefix-cardinality counters remain
  available to unit tests without enabling runtime diagnostics.

### 🧹 Cleanup

- Removed duplicated preview expression evaluation logic in favor of the shared
  compiled expression evaluator.
- Reduced loose lower-then-analyze pairs in grouped/global projection lowering
  by passing one expression-plus-analysis contract through the local lowering
  helpers.
- Kept the broader typed/analyzed expression artifact deferred until there is a
  short design for type inference, aggregate references, ORDER BY facts, and
  predicate-derivation inputs.

### 🧪 Testing

  expression and analysis proof coupled.
- Revalidated focused SQL lowering analysis coverage under the plain SQL
  feature.
- Revalidated grouped projection diagnostics and global aggregate

- Revalidated the SQL-feature core library check and SQL-feature library clippy

## 0.184.0

### 📝 Summary

- Starts the 0.184 mega-audit line with a SQL membership correctness and
  maintainability cleanup.
- Routes SQL `IN` / `NOT IN` filters through canonical boolean-expression
  lowering before deriving runtime predicate pushdown.
- Preserves SQL three-valued `NULL` membership semantics while retaining
  compact `IN` predicate extraction for safe non-`NULL` true sets.
- Adds the first evaluator parity guardrails for SQL NULL boolean composition
  and preview evaluation versus `CompiledExpr`.
- Adds index-route versus forced full-scan guardrails before deeper executor
  consolidation work.

### 🔧 Changed

- SQL scalar and global aggregate filters now lower the visible boolean
  expression first, then derive the predicate subset from that lowered
  expression.
- SQL membership no longer has a parser-level predicate-only shortcut that can
  bypass normal boolean-expression semantics.
- Ordinary non-`NULL` membership filters still expose compact predicate
  pushdown after expression lowering.
- Wide scalar `IN` predicates recover one compact membership predicate during
  expression predicate compilation without changing fluent/user OR route
  normalization.
- Shared query-plan cache miss classification now walks warmed cache keys once
  while preserving existing miss-reason priority.

### 🩹 Fixed

- `IN (NULL, value)` and `NOT IN (NULL, value)` no longer enter the runtime
  membership shortcut as if `NULL` were an ordinary predicate-list value.
- Indexed and full-scan SQL membership paths now agree on `NULL` membership
  behavior.
- Wide SQL scalar membership no longer degrades to a fixed-prefix scan plus
  residual OR filter after canonical boolean lowering.

### 🧹 Cleanup

- Removed the old parser-owned SQL membership predicate-only path and its
  duplicate membership coercion helpers.
- Narrowed owned predicate normalization helpers back to test-only use after
  the SQL shortcut was removed.

### 🧪 Testing

- Added focused lowering coverage proving membership predicates are extracted
  after boolean-expression lowering.
- Added runtime convergence coverage for indexed non-null and nullable
  full-scan `IN` / `NOT IN` queries containing `NULL`.
- Added branch-aware route coverage proving the admitted composite-index route
  matches a forced full-scan fallback over the same rows.
- Added grouped pagination coverage for rejected order-prefix ties and accepted
  full group-key tie-break ordering.
- Added SQL NULL boolean-composition coverage for `NOT UNKNOWN`,
  `UNKNOWN OR TRUE`, and `UNKNOWN AND TRUE`.
- Added preview-evaluator versus `CompiledExpr` parity tests for NULL
  comparison composition and CASE truth admission.
- Revalidated the full `icydb-core` SQL/diagnostics unit and trybuild suite.
- Revalidated core SQL/diagnostics clippy and the no-default public `icydb`
  SQL feature build.

### 🧭 Migration Notes

- No schema migration is required.
- No SQL syntax, result payload, cursor format, persistence format, or public
  endpoint behavior changes are intended.
