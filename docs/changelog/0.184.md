  projection-boundary cleanup.

  remaining SQL-only tree predicate helper.

## 0.184.12

### 📝 Summary

- Continues the query-engine audit by closing the large SQL membership
  expansion hole that made wide `IN (...)` predicates compile into huge boolean
  trees.
- Admits order-compatible index multi-lookup pages, such as
  `prefix IN (...) ORDER BY id`, when the selected index leaves the primary key
  as the exact remaining suffix.
- Collapses sparse covered prefix-set pages to the direct bounds reader after
  empty-prefix pruning leaves one active prefix, avoiding repeated chunked
  range traversal for that case.
- Adds aggregate EXPLAIN separation and scalar materialization footprint
  metrics so follow-up optimization work is guided by route and runtime
  evidence.

### 🔧 Changed

- SQL membership lowering, predicate recovery, truth-set compilation, and scalar
  evaluation now keep membership as compact `IN_LIST` semantics instead of
  expanding into left-deep `OR` / `AND` chains.
- Sparse key-only multi-lookup pages can use covering reads and linear
  access-proof checks, avoiding row hydration and quadratic canonical literal
  rechecks.
- Sparse prefix-set covering reads now reuse the direct covering resolver when
  only one prefix survives pruning, keeping the runtime route simple instead of
  entering the multi-prefix merge loop.
- Index-prefix-family routes whose consumed prefix leaves the primary-key
  fields as the exact remaining index suffix can now prove primary-key
  `ORDER BY` and stream without a materialized sort.
- Prefix and multi-lookup lazy streams now derive per-prefix primary-key suffix
  resume anchors when that route proof is active, preserving cursor safety for
  the admitted ordered stream.
- Aggregate EXPLAIN now reports semantic aggregate contracts separately from
  physical aggregate execution details.
- Scalar materialization diagnostics now report retained-slot layouts, retained
  values, and byte-length-only retained values in kernel-row attribution.

### 🧹 Cleanup

- Kept secondary-order pushdown separate from primary-key-suffix index stream
  preservation so route naming does not blur two different ordering proofs.
- Left the aggregate DTO as an explicit deferred boundary until it can remove
  real grouped/global duplication or protect cache/explain identity.
- Kept the existing `(collection_id, stage, id)` sparse token page honest: it
  remains bounded and covering, but still scans active collection entries
  because `stage` sits between the lookup prefix and `id`.

### 📊 Audit

- Sparse token `collection_id IN (...)` page compile work dropped from
  billion-scale expansion to about 3.3M compile instructions in the filtered
  audit scenario.
- The sparse token page path now reports about 40.6M total instructions, about
  15.8M planner/route instructions, zero row-store reads for key-only
  projection, and the same 256 index-entry reads on the existing
  `(collection_id, stage, id)` index.
- The matching sparse token count remains metadata-backed at about 4.8M total
  instructions with no index-entry scan and no row-store reads.

### 🧪 Testing

- Added focused route proof coverage for accepted primary-key-suffix
  multi-lookup ordering and rejected non-suffix index shapes.
- Added runtime coverage proving `IN (...) ORDER BY id` on `(bucket, id)` uses
  `IndexMultiLookup`, avoids materialized sorting, stays covering, and merges
  lookup branches by primary key.
- Revalidated the adjacent explicit primary-key suffix covering matrix, sparse
  branch-set multi-lookup coverage, SQL+diagnostics core clippy, no-default SQL
  checks, formatting, and whitespace checks.

### 🧭 Migration Notes

- No schema migration is required.
- No SQL syntax, result payload, cursor format, persistence format, or public
  endpoint behavior changes are intended.
- Existing sparse `collection_id IN (...) ORDER BY id` workloads benefit most
  when an order-compatible index such as `(collection_id, id)` exists. The
  current `(collection_id, stage, id)` shape still needs prefix-distinct
  metadata or future branch-tree routing for further page-cost reduction.
