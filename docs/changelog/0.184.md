# IcyDB 0.184 Changelog

## 0.184.8

### 📝 Summary

- Continues the query-engine audit by turning SQL write staged-row
  observability into an enforced execution boundary for public bounded
  `UPDATE`.
- Adds the same pre-commit row-bound contract to structural `DELETE RETURNING`
  execution, while leaving broad SQL DELETE behavior unbounded until a public
  DELETE exposure policy exists.
- Reduces journaled stable-store bucket amplification for tiny durable writes
  by keeping variable row, schema, and journal payload caps out of stable BTree
  layout hints.

### 🔧 Changed

- Policy-validated public primary-key `UPDATE` plans now carry an execution
  bound of at most one staged row.
- Policy-validated public bounded deterministic `UPDATE` plans now carry a
  staged-row execution bound derived from the admitted `LIMIT`.
- SQL `UPDATE` execution verifies staged candidate rows before opening the
  mutation commit window.
- Structural `DELETE RETURNING` execution can now receive a row-bound contract
  and validates it after delete preparation but before applying the commit
  bridge.
- Raw row, raw schema snapshot, and raw journal batch storage now use
  unbounded stable-structure layout hints while retaining their existing
  runtime encode/decode size checks.

### 🩹 Fixed

- Adds a compact `StagedRowsTooMany` SQL write-boundary diagnostic for
  execution-time staged-row overflow.
- Avoids worst-case MemoryManager bucket reservation for tiny journaled data,
  schema, and journal writes whose payloads are variable sized.

### 🧪 Testing

- Added focused coverage for staged-row bound acceptance/rejection, public
  bounded `UPDATE` metrics, and `DELETE RETURNING` pre-commit rejection without
  deleting rows.
- Added focused stable-store coverage proving raw row, schema, and journal
  payloads no longer amplify tiny durable writes into extra MemoryManager
  buckets.
- Revalidated SQL-feature core checks, SQL/no-default downstream checks,
  SQL-feature clippy, diagnostic-code tests and clippy, formatting, and
  whitespace checks.

### 🧭 Migration Notes

- No schema migration is required.
- Broad session/admin SQL write behavior remains unbounded and unchanged.
- This release adds a new compact diagnostic code for staged-row overflow.
- Generated/public DELETE policy wiring remains deferred; the DELETE
  pre-commit hook is internal until that policy exists.

## 0.184.7

### 📝 Summary

- Continues the query-engine audit by making SQL write materialization pressure
  visible before changing mutation execution semantics.
- Extends SQL write metrics with staged-row counts beside matched, mutated,
  and RETURNING row counts.

- Left INSERT SELECT staged-row attribution covered by core SQL write tests;
  the heap/journaled perf fixtures intentionally reject computed primary-key

  integration perf-audit checks, clippy, formatting, and whitespace checks.

### 🧭 Migration Notes

## 0.184.6

### 📝 Summary

- Continues the query-engine audit with two small cleanup contracts instead of
  broad execution rewrites.
- Tightens SQL global aggregate terminal interning so duplicate aggregate
  terminals compare retained semantic keys before constructing runtime-bound
  terminal state.
- Makes SQL write staging and row attribution explicit for INSERT, UPDATE, and
  DELETE, setting up later boundedness/materialization metrics without
  changing mutation execution semantics.

### 🔧 Changed

- Global aggregate terminal collection now owns executable terminals and their
  semantic keys through one interner contract, replacing loose parallel-vector
  plumbing.
- First-seen global aggregate terminals reuse the precomputed semantic key
  when constructing the runtime-bound terminal shape.
- INSERT, UPDATE, and DELETE now report matched, mutated, and returning row
  counts through one SQL write row-attribution helper.
- INSERT and UPDATE now stage mutation rows through an explicit
  mutation-batch boundary, and metrics derive staged-row attribution from that
  boundary before the batch is consumed by save execution.

### 🧹 Cleanup

- Removed repeated terminal construction for duplicate global aggregate
  semantic keys.
- Removed the terminal `semantic_key()` accessor after interning became owned
  by the terminal interner.
- Removed duplicated row-count saturation and returning-row derivation from
  individual SQL write execution lanes.
- Kept save execution and mutation commit behavior unchanged; the new staged
  batch is a local contract for observability and future bounds work.

### 🧪 Testing

- Revalidated focused global aggregate lowering/session coverage.
- Revalidated SQL write coverage across INSERT, INSERT SELECT, UPDATE, public
  bounded UPDATE, DELETE RETURNING, write metrics, and write residual-filter
  parity.
- Revalidated SQL-feature core clippy, no-default SQL feature check,
  formatting, and whitespace checks.

### 🧭 Migration Notes

- No schema migration is required.
- No SQL syntax, result payload, cursor format, persistence format, or public
  endpoint behavior changes are intended.
- The write-path changes are internal staging/attribution cleanup only; chunked
  mutation execution and explicit write bounds remain deferred.

## 0.184.0

### 📝 Summary

- Starts the 0.184 mega-audit line with a SQL membership correctness and
  maintainability cleanup.
- Routes SQL `IN` / `NOT IN` filters through canonical boolean-expression
  lowering before deriving runtime predicate pushdown.
- Preserves SQL three-valued `NULL` membership semantics while retaining
  compact `IN` predicate extraction for safe non-`NULL` true sets.
- Adds the first evaluator parity guardrails for SQL NULL boolean composition
  and preview evaluation versus `CompiledExpr`.
- Adds index-route versus forced full-scan guardrails before deeper executor
  consolidation work.

### 🔧 Changed

- SQL scalar and global aggregate filters now lower the visible boolean
  expression first, then derive the predicate subset from that lowered
  expression.
- SQL membership no longer has a parser-level predicate-only shortcut that can
  bypass normal boolean-expression semantics.
- Ordinary non-`NULL` membership filters still expose compact predicate
  pushdown after expression lowering.
- Wide scalar `IN` predicates recover one compact membership predicate during
  expression predicate compilation without changing fluent/user OR route
  normalization.
- Shared query-plan cache miss classification now walks warmed cache keys once
  while preserving existing miss-reason priority.

### 🩹 Fixed

- `IN (NULL, value)` and `NOT IN (NULL, value)` no longer enter the runtime
  membership shortcut as if `NULL` were an ordinary predicate-list value.
- Indexed and full-scan SQL membership paths now agree on `NULL` membership
  behavior.
- Wide SQL scalar membership no longer degrades to a fixed-prefix scan plus
  residual OR filter after canonical boolean lowering.

### 🧹 Cleanup

- Removed the old parser-owned SQL membership predicate-only path and its
  duplicate membership coercion helpers.
- Narrowed owned predicate normalization helpers back to test-only use after
  the SQL shortcut was removed.

### 🧪 Testing

- Added focused lowering coverage proving membership predicates are extracted
  after boolean-expression lowering.
- Added runtime convergence coverage for indexed non-null and nullable
  full-scan `IN` / `NOT IN` queries containing `NULL`.
- Added branch-aware route coverage proving the admitted composite-index route
  matches a forced full-scan fallback over the same rows.
- Added grouped pagination coverage for rejected order-prefix ties and accepted
  full group-key tie-break ordering.
- Added SQL NULL boolean-composition coverage for `NOT UNKNOWN`,
  `UNKNOWN OR TRUE`, and `UNKNOWN AND TRUE`.
- Added preview-evaluator versus `CompiledExpr` parity tests for NULL
  comparison composition and CASE truth admission.
- Revalidated the full `icydb-core` SQL/diagnostics unit and trybuild suite.
- Revalidated core SQL/diagnostics clippy and the no-default public `icydb`
  SQL feature build.

### 🧭 Migration Notes

- No schema migration is required.
- No SQL syntax, result payload, cursor format, persistence format, or public
  endpoint behavior changes are intended.
