# IcyDB 0.175 Changelog

## 0.175.6

### 🔧 Changed

- Caches generated-schema reconciliation after recovery for each runtime store
  registry/hook set, so read-side `recovered_store` calls do not rerun schema
  reconciliation on every warmed query.
- Marks the reconciliation cache dirty around real recovery, preserving the
  existing pre/post-recovery schema validation boundary before read paths
  resume.
- Extends scalar query attribution with load-plan, row-layout, continuation,
  handoff, route-plan, and runtime-prepare counters so future perf audits can
  distinguish store traversal cost from recovery/runtime-prep drift.

### 🧪 Testing

- Expands the journaled SQL perf guard to prove warmed primary-key
  `ORDER BY id ASC LIMIT 1` stays bounded below the million-instruction range
  and no longer re-enters recovery reconciliation from executor preparation.
- Records focused stable/journaled measurements showing warmed journaled
  `LIMIT 1` queries return to the sub-million path while cold first-query plan
  construction remains a separate cache-warmup cost.
- Revalidates query, session, recovery, journaled, workspace check, core
  clippy, and diff hygiene coverage after isolating the recovery hot-path fix.

## 0.175.5

### 🔧 Changed

- Adds a journaled SQL perf-audit fixture store and entity so the integration
  harness can compare stable and journaled `ORDER BY id ASC LIMIT 1` behavior
  under the same canister-level instruction counters.
- Caches accepted schema snapshots and schema fingerprints for read-side query
  cache setup, avoiding repeated stable schema reload/fingerprint work before a
  warmed SQL compiled-command cache hit.
- Retains the resolved prepared select plan on cached compiled SQL `SELECT`
  commands, keyed by accepted schema fingerprint, so warmed query execution can
  skip the expensive shared plan lookup while still failing closed across schema
  changes.
- Keeps public SQL perf attribution on the detailed compile/plan/store/executor
  path instead of replacing it with a lower-detail shell-only probe.

### 🧪 Testing

- Adds an integration regression guard proving journaled primary-key
  `ORDER BY id ASC LIMIT 1` returns one row, keeps store traversal bounded, and
  stays close to the stable-store traversal cost.
- Proves a warmed journaled SQL query reuses both the compiled SQL artifact and
  prepared query plan without re-entering the expensive accepted-schema reload
  or plan-lookup paths.
- Revalidates the core stable/journaled projection guard showing both storage
  modes scan exactly one row for primary-key ordered `LIMIT 1` projections.
- Revalidates SQL DDL cache invalidation coverage so cached accepted-schema and
  prepared-plan query state is invalidated across schema publication.

## 0.175.4

### 📚 Documentation

- Adds public storage-mode guidance to the README, distinguishing direct
  `storage(stable(...))`, volatile `storage(heap())`, and journaled
  cached-stable `storage(journaled(...))` without implying heap durability.
- Documents the four journaled memory-ID roles: canonical data, index, and
  schema stable BTrees plus the durable journal-tail memory.
- Extends the SQL subset contract so `SHOW ENTITIES`, `SHOW STORES`, and
  `SHOW MEMORY` expose storage-mode facts as catalog diagnostics, not runtime
  policy authority.

### 🧪 Testing

- Adds a public-docs source guard proving README, SQL contract, and demo RPG
  schema surfaces describe stable, heap, and journaled tradeoffs consistently.
- Runs the 0.175 closeout validation set across journaled, query, executor,
  session, diagnostics, registry, relation, recovery, commit, CLI snapshot,
  build wiring, workspace check, and clippy coverage.
- Closes the 0.175 proof matrix with all rows H1-H24 marked closed.
- Adds a source-audit guard proving stable and heap compatibility contracts
  remain distinct from journaled storage: stable keeps present allocation
  identity, durable commit participation, and stable commit replay while heap
  keeps absent allocation identity, volatile durability, live-only commit
  participation, and no recovery participation.
- Revalidates stable registry allocation identity and heap registry allocation
  absence tests.
- Revalidates heap reinitialization volatility coverage, proving heap rows and
  indexes are not recovered from stable commit state while live schema metadata
  is rebuilt for validation and diagnostics.
- Revalidates existing row/index/schema codec and commit-marker source guards
  so capability policy and journaled codec work stay out of existing stable
  codec and marker-format surfaces.

## 0.175.1

### 🔧 Changed

- Closes the ordered-overlay DRY boundary by recording data, index, and schema
  store traversal evidence against the shared overlay helper introduced in
  `0.175.0`.
- Keeps journaled data and index store wrappers on the same
  canonical-plus-live-minus-tombstone merge primitive for ascending and
  descending traversal, range bounds, live-over-canonical precedence, and
  visitor early-stop.
- Treats schema-store accepted-snapshot traversal as part of the same hot-path
  boundary so latest-snapshot lookup can stream descending canonical/live
  overlays without building a full snapshot map.

### 🧪 Testing

- Confirms mixed canonical/live/tombstone data traversal honors range bounds
  and early stop in both ascending and descending order.
- Confirms journaled schema traversal interleaves canonical/live versions,
  prefers live duplicates, skips tombstoned versions, and can early-stop on
  descending latest-version lookup.
- Extends source-audit evidence that normal query/executor/session paths do not
  call journaled snapshot materializers and that the remaining index
  materializer is purpose-named for fold.

## 0.175.0

### ➕ Added

- Starts the journaled store audit, hardening, and DRY cleanup line with an
  active 0.175 design covering ordered-overlay traversal, snapshot boundaries,
  hot-path guards, recovery/fold edge evidence, and closeout criteria.
- Adds a shared ordered-overlay traversal primitive for journaled canonical
  stable maps, volatile live maps, and tombstones so data, index, and schema
  store wrappers can preserve the same streaming read-view semantics.
- Adds accepted-schema-backed entity metadata to catalog surfaces so
  `SHOW ENTITIES` can report storage mode, column count, index count, relation
  count, and `sv` schema version without widening default output with verbose
  paths.
- Adds SQL `SHOW MEMORY` catalog inspection for stable-memory tags, memory IDs,
  and owning stores.

### 🔧 Changed

- Routes journaled data and index traversal through streaming overlay paths
  that honor visitor early-stop before materializing a full merged view.
- Applies the same streaming overlay boundary to journaled schema snapshot
  reads that are on SQL query hot paths.
- Carries accepted schema, schema fingerprint, and accepted authority through
  cold SQL compile-to-execute paths so query calls do not rely on persisted
  query-cache writes to avoid schema reconstruction.
- Keeps fold-only journaled index materialization isolated behind a
  purpose-named helper.

### 🧪 Testing

- Adds source-audit coverage for journaled traversal materialization,
  fold-only snapshot helpers, diagnostics authority, and generated-model
  fallback boundaries.
- Adds regression coverage for stable and journaled primary-order SQL
  projections, mixed canonical/live/tombstone overlay traversal, schema-store
  latest-snapshot traversal, and compact catalog column rendering.
