# IcyDB 0.169 Changelog

## 0.169.4

### 🔧 Changed

- Stable-source relation declarations targeting heap-backed entities now fail
  closed during schema validation, including scalar relation fields and
  relation-edge tuple metadata.
- Save-time strong-relation validation rejects stable-source rows that would
  own referential integrity against volatile heap targets.
- Heap-source strong relations to heap targets keep existing live validation
  semantics.

### 🧪 Testing

- Adds schema-node coverage for stable -> heap rejection and heap -> heap
  relation admission.
- Adds runtime session coverage proving stable-source strong relation saves
  reject without persisting the source row.
- Adds runtime session coverage proving heap-source strong relations validate
  while the heap target exists live.

## 0.169.3

### 🧪 Testing

- Adds a heap session lifecycle proof that simulates heap store
  reinitialization at the store-registry boundary.
- Confirms heap data and secondary-index contents are absent after reinit and
  are not recovered from stable commit state.
- Confirms heap schema metadata is rebuilt only as live validation/diagnostic
  metadata with `storage: heap` and absent stable allocation identity.

### 📚 Documentation

- Updates the 0.169 proof matrix with the heap syntax, generated wiring,
  diagnostics, live runtime behavior, and lifecycle rows that now have concrete
  evidence.

## 0.169.2

### 🧪 Testing

- Adds public runtime proof that heap-backed stores support typed insert/load
  round trips while the process is live.
- Proves heap-backed secondary indexes are visible through the public SQL query
  entrypoint and retain the indexed access route in public EXPLAIN output.
- Confirms heap-backed data and index diagnostics report `storage: heap`, live
  entry counts, and absent stable allocation metadata after writes.

## 0.169.1

### 🔧 Changed

- Storage snapshots now derive each data, index, and schema row's storage mode
  from the actual runtime store backend.
- Heap-backed store roles report `storage: heap` with no `memory_id` or
  `stable_key`.
- Stable stores with intentionally absent allocation identities still report
  `storage: stable`, keeping storage mode separate from allocation metadata.

### 🧪 Testing

- Adds diagnostics coverage for a heap-backed store registered with absent
  allocation identity.
- Extends storage-mode rendering coverage to include the `heap` label.

## 0.169.0

### ➕ Added

- Admits explicit `storage(heap())` store declarations as the first non-stable
  store backend.
- Adds volatile heap-backed `DataStore`, `IndexStore`, and `SchemaStore`
  runtime wrappers using ordered in-memory maps.
- Generates heap store wiring through `init_heap()` constructors and explicit
  absent allocation identity.

### 🔧 Changed

- `storage(heap())` is now the canonical heap storage form instead of reserved
  syntax.
- Heap stores reject stable-only memory ID arguments instead of silently
  assigning them heap meaning.
- Stable stores keep their existing generated stable wiring and allocation
  identity path.

### 🧪 Testing

- Adds compile-pass coverage for heap store declarations.
- Adds heap traversal tests for data, index, and schema stores covering order,
  range bounds, and early-stop behavior.
- Adds generated wiring tests proving stable stores still use stable memory
  keys while heap stores use heap initializers and absent allocation identity.

### 🧭 Migration Notes

- Stable store declarations are unchanged.
- Heap stores are volatile and do not carry stable-memory IDs, stable keys, or
  durable allocation identity.
