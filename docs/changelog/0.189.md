# IcyDB 0.189 Changelog

## 0.189.2

### 📝 Summary

- Expands the reliability proof with deterministic model-based guarded recovery
  coverage.
- Compares marker replay against a forward-apply row/index oracle across
  generated insert, update, and delete batches.
- Reuses the same deterministic model batch across every classified commit
  failpoint site and interruption mode.
- Tightens the failpoint oracle so fold-watermark state is checked alongside
  rows, indexes, marker presence, and journal-tail visibility.
- Adds literal empty-byte contracts for marker and journal decoding.

### 🔧 Changed

- Adds a deterministic model fixture that produces stable insert, update, and
  delete batches over indexed rows, then proves guarded recovery converges to
  the same raw row/index state as direct forward apply.
- Extends the shared failpoint case helper with pre/post fold-watermark facts
  so interruption and retry expectations use the same classified oracle.
- Keeps empty commit-marker bytes as the explicit durable "no marker" state.
- Keeps empty journal-batch bytes in the malformed input family so they fail
  closed as corruption instead of being accepted as an empty batch.

### 🧪 Testing

- Added deterministic model recovery coverage for repeated marker persistence,
  guarded recovery, marker clear, and empty journal tail after each model batch.
- Added model-batch failpoint coverage across marker write, marker-bound journal
  append, and marker clear for both structured returned errors and host unwind
  interruptions.
- Added focused empty-byte marker and journal tests.
- Rechecked the commit recovery, commit-store, and journal test modules plus
  SQL-feature `icydb-core` Clippy.

### 🧭 Migration Notes

- No public API, SQL/fluent behavior, generated endpoint behavior, persisted
  format, response shape, cursor format, or cache identity changes are intended.
- The new coverage is test-harness and corruption-contract proof only.

## 0.189.1

### 📝 Summary

- Continues the reliability harness by making commit failpoint classification
  executable.
- Names structured returned errors separately from host unwind interruptions.
- Names the durable recovery authority available at each failpoint site.
- Adds explicit interruption oracles for expected snapshot, marker, and
  journal-tail state.

### 🔧 Changed

- Adds a test-only failpoint failure-class contract for structured returned
  errors and host unwind interruptions.
- Adds a test-only recovery-authority contract for no-authority, marker-payload,
  marker-plus-journal-prefix, recovered-with-marker, and recovered-without-marker
  failpoint positions.
- Adds a failpoint recovery oracle that declares the expected pre-commit versus
  marker-authorized snapshot family, marker presence, and journal-tail batch
  count after interruption.
- Routes the existing commit recovery failpoint tests through the shared oracle
  instead of duplicating marker, journal, and snapshot assertions at each site.

### 🧪 Testing

- Added focused coverage proving every current commit failpoint mode has an
  explicit failure class.
- Added focused coverage proving every current durable failpoint site has an
  explicit recovery-authority class.
- Added focused coverage proving every current durable failpoint site has an
  explicit interruption oracle.
- Rechecked the commit recovery test module and SQL-feature `icydb-core` Clippy.

### 🧭 Migration Notes

- No public API, SQL/fluent behavior, generated endpoint behavior, persisted
  format, response shape, cursor format, or cache identity changes are intended.
- The new classifications are test-harness vocabulary for future resource,
  trap, and budget validation.

## 0.189.0

### 📝 Summary

- Starts the reliability-audit cleanup line after the second mega-audit pass.
- Makes the supported no-default feature matrix explicit for public crates and
  CI-style validation.
- Adds an IC-native host failpoint harness for the core commit-marker recovery
  windows.
- Hardens journal-tail storage so oversized persisted journal values cannot be
  materialized as one unbounded stable-structure value before validation.
- Expands deterministic persisted-format corruption coverage for commit markers
  and journal batches.
- Documents volatile heap-store and non-atomic batch write footguns.
- Scopes commit recovery state by commit-memory domain so independent registries
  cannot suppress each other's recovery checks in host/multi-registry contexts.

### 🔧 Changed

- Treats `cargo check --no-default-features` as a supported public-crate matrix
  item and keeps full no-default `icydb-core` tests outside the supported matrix
  until the broad SQL/projection/schema test harness is split.
- Keeps SQL perf-audit canister SQL/attribution DTOs behind the SQL feature
  while leaving typed fixture load/reset helpers available in no-default
  workspace checks.
- Adds failpoint coverage around durable marker writes, marker-bound journal
  publication, and marker clear.
- Makes recovery failpoint tests compare deterministic data/index snapshots,
  marker state, and journal-tail visibility against explicit pre-marker or
  marker-authorized post-commit oracles.
- Replaces whole-batch journal-tail stable values with bounded tail chunks keyed
  by journal sequence and chunk index.
- Preserves tiny append behavior while rejecting sequence gaps, missing chunk
  zero, chunk gaps, oversized reconstructed payloads, corrupt magic, truncated
  payloads, duplicate batch ids, and encoded sequence/key drift.
- Treats already-persisted matching marker-bound journal prefixes as interrupted
  append progress so recovery can fill missing chunks, while mismatched prefixes
  fail closed.
- Keeps the journal-tail storage change as a pre-1.0 internal hard cut, matching
  the project rule that internal persisted formats do not keep compatibility
  fallbacks before `1.0.0`.
- Makes commit-marker stores, marker-presence hints, and startup recovered state
  keyed by commit-memory/storage domain instead of one process-global slot.

### 📚 Documentation

- Public facade and core session docs now state that `*_many_non_atomic` helpers
  are fail-fast prefix-commit APIs and that returned errors do not roll back an
  already committed prefix.
- Atomic same-entity batch helpers are documented as the all-or-nothing option.
- README examples use atomic import helpers unless partial progress is the
  intended behavior.
- Storage-mode docs state that `storage(heap())` is live-only volatile state
  with no stable allocation identity, commit-marker/journal participation, or
  recovery path.
- The 0.189 design doc records the minimum closeout audit, finding
  dispositions, and multi-registry recovery proof status.

### 🧪 Testing

- Added no-default public-crate checks and a no-default facade smoke test.
- Added host failpoint coverage for returned internal errors and panic/unwind
  style interruptions around marker write, marker-bound journal append, and
  marker clear.
- Added deterministic marker corruption coverage for corrupt control-slot magic,
  truncated marker envelopes, oversized stored markers, incompatible marker
  versions, malformed control slots, invalid row keys, oversized row payloads,
  and unknown row-op flags.
- Added deterministic journal corruption coverage for corrupt batch magic,
  truncated payloads, trailing bytes, incompatible versions, oversized raw
  values before payload parsing, unknown record tags, sequence gaps, duplicate
  batch ids, and corrupt/truncated raw tail bytes during ordered traversal.
- Added multi-registry recovery coverage proving that a recovered or no-marker
  primary domain does not consume or suppress marker recovery in a peer domain,
  and vice versa.

### 🧭 Migration Notes

- The journal-tail persisted layout is an internal pre-`1.0.0` hard cut.
- No public SQL syntax, fluent API, generated endpoint behavior, cursor format,
  response type, or cache-identity migration is intended.
- Heap stores remain intentionally volatile; users who need durability should
  use stable/journaled store modes.
- Non-atomic batch helpers still intentionally allow prefix commits; users who
  need all-or-nothing behavior should use atomic batch helpers.
