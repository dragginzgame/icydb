# 0.91 Changelog

## 0.91.1

- finished the SQL front-end unification line by removing the separate parser-owned `SqlPredicate` statement surface and making `WHERE` carry ordinary boolean `SqlExpr` trees instead, so `SELECT`, aggregate inputs, `HAVING` values, `ORDER BY`, and `WHERE` now all enter lowering through the same `SqlExpr -> lower_sql_expr(..., phase)` seam.
- kept runtime `Predicate` as an execution concern rather than a second semantic front end by making `WHERE` lowering do one shared `SqlExpr -> Expr -> Predicate` adaptation step in `db/sql/lowering/predicate.rs`; clause owners no longer parse or lower predicate semantics independently.
- admitted searched `CASE` in `WHERE` on top of that unified seam instead of special-casing it into the old predicate parser, and added focused parser, lowering, runtime, and explain coverage proving:
  - searched `CASE` in `WHERE` parses as `SqlExpr`
  - `WHERE` lowering preserves the same canonical boolean intent as equivalent direct predicates
  - `EXPLAIN` output for `WHERE CASE ...` matches the equivalent canonical boolean form
- tightened `WHERE` boolean/`NULL` semantics at the final predicate boundary so the two-valued runtime predicate layer still matches SQL row-filter behavior for:
  - bare `NULL` in boolean position
  - `NOT NULL`
  - combined boolean expressions that include `CASE` branches returning `NULL`
  - searched `CASE` paths where only the selected branch should determine row inclusion
- fixed the front-end semantic bug where SQL had been collapsing `IS NULL` / `IS NOT NULL` and `= NULL` / `!= NULL` into the same shape too early:
  - parser now preserves dedicated `SqlExpr::NullTest` nodes for `IS NULL` and `IS NOT NULL`
  - ordinary `= NULL` / `!= NULL` remain ordinary comparison expressions
  - `WHERE` lowering now maps explicit null tests onto runtime `Predicate::IsNull` / `Predicate::IsNotNull`
  - ordinary comparisons against `NULL` now stay distinct and collapse to unknown-at-`WHERE` row filtering instead of silently acting like null tests
- followed that null-test split through normalization, aggregate collection, type inference, and expression ownership so the new distinction is stable across parsing, lowering, fingerprints, and runtime evaluation without leaking the internal null-test helpers into projection or order-expression families.
- added focused proof coverage for the distinction at three levels:
  - parser tests prove `age IS NULL` and `age = NULL` now produce different SQL expression trees
  - lowering tests prove those two forms now build different planned query intents
  - runtime tests prove `age IS NOT NULL` and `age != NULL` behave differently even on the current non-nullable scalar fixture, with `IS NOT NULL` keeping rows and `!= NULL` keeping none

```sql
SELECT name
FROM character
WHERE CASE WHEN strength >= dexterity THEN TRUE ELSE NULL END
ORDER BY id ASC
LIMIT 10;
```

## 0.91.0

- introduced a real SQL-side expression boundary for reduced SQL before planner lowering, centered on `SqlExpr` plus one shared `lower_sql_expr(..., phase)` seam, so scalar projection, aggregate inputs, alias/order normalization, and grouped/global `HAVING` value lowering now all run through one planner-owned expression contract instead of clause-specific shells.
- widened the production planner and evaluator expression spine to admit boolean composition, comparisons, unary `NOT`, and planner-owned searched `CASE` nodes, which gives `CASE` one canonical runtime, explain, cache-key, and fingerprint representation instead of another SQL-only execution path.
- admitted searched `CASE WHEN ... THEN ... [ELSE ...] END` on the SQL surfaces that already sit on that shared expression seam:
  - `SELECT` projections
  - aggregate input expressions such as `SUM(CASE WHEN ... THEN ... ELSE ... END)`
  - grouped/global aggregate `HAVING` value expressions
- kept the first shipped `CASE` surface intentionally narrow:
  - searched `CASE` only
  - no simple `CASE x WHEN ...`
  - missing `ELSE` canonicalizes to planner-owned `NULL`
  - branch result typing reuses the existing scalar compatibility rules instead of adding CASE-specific coercions
  - non-boolean `WHEN` conditions and incompatible branch types fail closed during inference/lowering instead of at runtime
- migrated grouped/global `HAVING` value lowering off the older compare-shell-only path and onto the shared SQL-expression seam, so post-aggregate `CASE`, unary `NOT`, and the already-admitted arithmetic/wrapper expressions now lower through the same planner-owned post-aggregate expression contract.
- removed the last SQL front-end ownership split before `0.91.0` by introducing a parser-owned `SqlPredicate` statement model for `WHERE`, `DELETE`, and `UPDATE` selectors; SQL statements no longer carry runtime `Predicate` values directly, and lowering now owns the `SqlPredicate -> Predicate` conversion boundary.
- kept predicate traversal ownership where it belonged while doing that `WHERE` migration: SQL normalization now adapts parser-owned predicate identifiers by delegating through the existing predicate-field rewrite helper instead of taking ownership of predicate-tree traversal inside SQL lowering.
- added focused parser, lowering, inference, runtime, and structural-guard coverage for:
  - searched `CASE` parsing and rejection of simple `CASE`
  - missing-`ELSE` canonicalization to `NULL`
  - searched `CASE` laziness in projection evaluation
  - grouped/global aggregate `HAVING` value lowering through the shared seam
  - the new SQL predicate front-end ownership boundary for statement parsing and normalization

```sql
SELECT
  CASE
    WHEN strength >= dexterity THEN 'frontline'
    ELSE 'support'
  END AS role
FROM character
ORDER BY id ASC
LIMIT 10;
```
