# 0.91 Changelog

## 0.91.5

- followed the `0.91.4` aggregate/pipeline cleanup with a larger deletion-oriented pass over the duplicated load execution and session execution matrices that had still been carrying separate normal and perf-attributed paths for the same runtime work.
- collapsed the scalar load preparation family onto one canonical structural builder in `db/executor/pipeline/entrypoints/scalar/mod.rs`:
  - initial scalar rows
  - resumed scalar rows
  - retained-slot scalar SQL projection rows
  - materialized scalar boundary rows
  now all derive their prepared runtime through the same `prepare_scalar_route_runtime_from_parts(...)` seam plus one small options carrier instead of open-coding authority/store/spec/route-plan setup in parallel helpers.
- followed the same cleanup through grouped execution by deleting the repeated grouped route assembly seam:
  - grouped entrypoints and orchestrator strategy now both prepare grouped runtimes through `prepare_grouped_route_runtime_for_load_plan(...)`
  - normal grouped execution and diagnostics-attributed grouped execution now share one internal grouped runtime spine and only differ in whether they keep the optional phase-attribution payload
  - initial grouped SQL/runtime entrypoints no longer keep a second private “prepare initial grouped runtime” layer
- deleted the duplicated grouped session execution wrapper path in `db/session/query.rs`:
  - grouped cursor decode / prepare / execute now stays on one `execute_grouped_plan_with(...)` helper
  - the diagnostics-only grouped query result wrapper no longer keeps a second cursor/runtime wrapper stack beside the canonical grouped execution path
  - the query attribution surface now maps grouped and scalar outputs directly from the canonical execution path instead of routing through parallel intermediate wrappers
- deleted the broadest duplicated non-`SELECT` SQL attribution matrix in `db/session/sql/execute/mod.rs`:
  - perf-attributed SQL execution for `DELETE`, global aggregate, `EXPLAIN`, `INSERT`, `UPDATE`, `DESCRIBE`, `SHOW INDEXES`, `SHOW COLUMNS`, and `SHOW ENTITIES` now wraps the existing canonical `execute_compiled_sql_with_cache_attribution(...)` path instead of re-implementing each branch with a second match matrix
  - the remaining `SELECT` attribution work now stays inline only where the planner/runtime split genuinely differs, instead of keeping extra one-use wrappers for grouped and scalar `SELECT`
- the net effect is not a new feature, but it materially shrinks one of the last broad “same runtime, different wrapper” seams on the `0.91` line:
  - fewer execution-path wrappers at the session boundary
  - fewer load-preparation owners in scalar and grouped entrypoints
  - fewer diagnostics-only mirror paths
  - fewer places where grouped and scalar routing rebuild the same prepared runtime shell
- followed that runtime-path reduction with another deletion-only cleanup pass over dead compatibility and test scaffolding that had become redundant after the load/session collapse:
  - removed the legacy grouped SQL session test shim and consolidated the old parser/lowering surface onto one shared legacy select-test helper path
  - deleted the unused lowered-SQL diagnostics attribution surface in `db/session/sql/execute/lowered.rs` instead of carrying a second dead read-only attribution API beside the live lowered execution helpers
  - deleted the dead `clear_stores_for_tests(...)` write shim and trimmed a small dead registry/index helper chain exposed by the cleanup
  - removed one more planner-validation test seam by exposing grouped validation helpers directly to crate tests instead of keeping extra `*_for_tests` forwarding wrappers in `db/query/plan/validate/mod.rs`
- the follow-through cleanup kept the same behavior surface but reduced more of the “helper for the helper” scaffolding that had been left behind the canonical runtime path:
  - fewer legacy session-test adapters
  - fewer diagnostics-only dead APIs
  - fewer zero-value test/reset shims
  - fewer planner-validation forwarding layers
- verification for this cleanup stayed focused on the affected canonical paths:
  - `cargo test -p icydb-core db::executor::pipeline:: -- --nocapture`
  - `cargo test -p icydb-core db::session::tests::sql_grouped -- --nocapture`
  - `cargo test -p icydb-core db::session::sql::projection::runtime::tests:: -- --nocapture`
  - `cargo test -p icydb-core db::query::plan::validate::tests::grouped -- --nocapture`
  - `cargo test -p icydb-core db::query::plan::tests::group -- --nocapture`

## 0.91.4

- followed the `0.91.3` `HAVING` collapse with another cleanup-only slice focused on deleting the remaining clause-owned front-end scaffolding that had been left behind the unified `SqlExpr -> Expr` line.
- removed the last duplicate `WHERE` admission owner by deleting the parser-side admitted-surface gate and keeping `db/sql/lowering/predicate/validate.rs` as the only authoritative shipped-surface validator for `WHERE` boolean expressions.
- collapsed one more clause-specific parser normalization seam by deleting separate alias-normalization wrappers for `WHERE` and `HAVING` expressions and replacing them with shared `SqlExpr` normalization helpers:
  - `normalize_sql_expr_for_entity_alias(...)`
  - `normalize_sql_exprs_for_entity_alias(...)`
  so parser entry points still stay clause-local, but expression-tree rewriting no longer has separate `predicate` and `having` owners.
- kept `SELECT`, `WHERE`, aggregate inputs, `HAVING`, and grouped/global post-aggregate expression lowering on the same phase-driven ownership line, with parser entry points only handling syntax and statement assembly while lowering remains the one boundary that decides shipped expression admission.
- trimmed a broad slice of duplicated SQL-lowering test scaffolding in `db/sql/lowering/tests/mod.rs` instead of carrying more one-off parity helpers:
  - repeated typed SQL query compilation now goes through shared helpers
  - repeated SQL-vs-fluent plan-parity assertions now go through shared helpers
  - repeated executable parity assertions now go through shared helpers
  - several field-list, grouped, searched-`CASE`, qualified-identifier, and text-predicate parity tests now reuse those shared paths instead of open-coding the same `compile -> unwrap -> plan -> compare` sequence
- followed the production cleanup with two more LOC-focused reductions behind the same line:
  - simplified fluent policy validation by deleting the small rule-table/context scaffolding in `db/query/fluent/load/validation.rs` and `db/query/plan/validate/fluent_policy.rs`
  - simplified intent policy validation by deleting the rule-table/context scaffolding in `db/query/intent/policy.rs` and `db/query/plan/validate/intent_policy.rs`
- kept behavior unchanged on those policy slices: the cleanup was deletion and direct checks, not new policy semantics.
- the same cleanup pass also removed one more leftover `HAVING` wrapper seam from SQL lowering and grouped execution follow-through, so the current `0.91` line now reads much more cleanly as:
  - parser syntax owners
  - shared phase-driven SQL expression lowering
  - planner-owned semantics
  - structural predicate compilation
  - execution/runtime owners
- followed that cleanup through the aggregate execution boundary by deleting several remaining parallel prepared-carrier families and converging them onto one canonical prepared object per concept:
  - grouped `HAVING` explain and fingerprint no longer project the same post-aggregate meaning into separate private trees
  - global aggregate lowering no longer carries a second typed terminal model alongside the prepared scalar aggregate strategy
  - grouped aggregate planning now carries one `GroupedAggregateExecutionSpec` instead of a second near-copy projection spec for the same aggregate slot
  - scalar aggregate execution no longer keeps tuple-style `(boundary, strategy/payload, prepared)` handoffs for numeric, terminal, and projection lanes
- the aggregate executor cleanup specifically hard-cut the broadest parallel model seams:
  - removed `PreparedAggregateStreamingInputsCore` and kept `PreparedAggregateStreamingInputs` as the only aggregate streaming payload owner
  - moved numeric aggregate execution onto one `PreparedScalarNumericBoundary` that owns its runtime payload family directly
  - moved scalar terminal, order-sensitive terminal, and scalar projection execution onto single prepared boundary objects instead of tuple carriers plus separate strategy handoff
- followed the same “one boundary per prepared runtime” rule through pipeline entrypoints by collapsing the extra scalar-vs-grouped dispatch wrapper in the load orchestrator:
  - `PreparedLoadRouteRuntime` is now the canonical prepared route-runtime envelope for entrypoint dispatch
  - access-stage orchestration now carries that one route-runtime envelope directly instead of wrapping `PreparedScalarRouteRuntime` / `PreparedGroupedRouteRuntime` in a second `ExecutionSpec` enum
  - payload assembly now goes through explicit `LoadExecutionPayload::{scalar, grouped}` and `LoadPayloadState::new(...)` constructors instead of open-coding another wrapper layer around the same result pair
- finished the same cleanup line by removing the last live grouped-aggregate compatibility split:
  - `GroupAggregateSpec` semantic identity now depends on `input_expr` alone instead of keeping a second production `target_field` track beside the canonical aggregate input expression
  - `GroupedAggregateExecutionSpec` now derives field-target behavior from plain `Expr::Field(...)` aggregate inputs instead of carrying a parallel stored field label
  - grouped explain, fingerprint, cache-key, policy, and grouped-`HAVING` matching now compare against the same canonical aggregate input expression shape
  - the old grouped aggregate input-normalization helper path was deleted instead of preserved as another compatibility adapter
- kept older grouped test matrices declarative without leaving production semantics split: test builds still carry a fixture-only `target_field` fallback on `GroupAggregateSpec`, but production code no longer depends on that second representation for grouped aggregate identity, routing, or policy.
- removed one more aggregate-input canonicalization duplicate on the SQL side by reusing the shared builder-owned aggregate-input canonicalizer directly, so SQL aggregate lowering no longer owns a second meaning-preserving normalization path for the same aggregate input surface.
- removed one more grouped aggregate reconstruction duplicate by routing projection-side grouped aggregate rebuilding through the shared grouped semantics helper instead of keeping a second local reconstruction helper in the projection layer.
- taken together, this `0.91.4` follow-through is still cleanup-only, but it materially shrinks the amount of duplicated internal structure behind the shipped SQL expression line:
  - fewer aggregate model families
  - fewer executor handoff tuples and wrapper enums
  - fewer conversions between semantically identical prepared runtime objects
  - clearer “parser -> lowering -> planner -> prepared runtime -> execution” ownership boundaries

## 0.91.3

- finished the last obvious clause-specific semantic wrapper on the `0.91` SQL-expression line by collapsing grouped/global `HAVING` onto shared planner `Expr` instead of carrying dedicated SQL-side and lowering-side `HAVING` value wrapper types.
- deleted the remaining `HAVING`-only front-end adapter layer:
  - `SqlHavingValueExpr`
  - `having_value_expr_from_sql_expr(...)`
  - `normalize_having_value_expr(...)`
  - `normalize_having_value_expr_for_table_alias(...)`
  - lowering-side `ResolvedHaving*` wrapper types
- moved grouped and global aggregate lowering to resolve `HAVING` directly from `SqlExpr -> lower_sql_expr(SqlExprPhase::PostAggregate) -> Expr`, so post-aggregate arithmetic, searched `CASE`, unary `NOT`, and null-test expressions now stay on the same planner-owned expression surface instead of round-tripping through a clause-specific `HAVING` shell.
- followed that collapse through the grouped execution and observability owners too:
  - grouped execution now reuses compiled grouped projection expressions for `HAVING`
  - grouped/global `HAVING` explain output now walks the shared planner `Expr`
  - grouped/global `HAVING` fingerprint and cache-key identity now hash the same shared expression shape
- fixed two semantic seams while deleting that wrapper path:
  - grouped/global `HAVING` now treat final-boundary `NULL` as a non-match instead of surfacing a runtime type error
  - grouped `HAVING` streaming admissibility now recursively inspects widened expression trees instead of only the top compare shell, so arithmetic and wrapped aggregate `HAVING` expressions no longer slip through ordered-streaming checks incorrectly
- kept SQL null semantics aligned with the new shared expression path by preserving explicit `IS NULL` / `IS NOT NULL` nodes through `HAVING` parsing and by normalizing fluent `= NULL` / `!= NULL` grouped `HAVING` shapes onto the same null-test semantics before grouped execution.
- cleaned the follow-through fallout rather than carrying stale compatibility scaffolding:
  - removed stale `large_enum_variant` lint expectations made obsolete by the deletion pass
  - removed a few dead re-exports and test-only helper uses that had only existed to support the old `HAVING` wrapper model
  - kept the grouped aggregate matching checks explicit enough to stay lint-clean without suppressing the new shared-expression comparisons
- revalidated the collapse with the full `icydb-core` suite plus `cargo clippy -p icydb-core --all-targets -- -D warnings`, so the post-cleanup state is both test-green and lint-clean.

```sql
SELECT class_name, AVG(strength)
FROM character
GROUP BY class_name
HAVING CASE
         WHEN AVG(strength) >= 12 THEN TRUE
         ELSE NULL
       END
ORDER BY class_name ASC
LIMIT 10;
```

## 0.91.2

- followed the `0.91.1` `WHERE` unification through the last internal ownership seam by changing `db/sql/lowering/predicate` from one mixed helper file into an explicit three-phase boundary:
  - `validate.rs` rejects unsupported `WHERE` boolean shapes after `SqlExpr -> Expr` lowering
  - `normalize.rs` owns structural shaping only, including compare canonicalization, casefold wrapper normalization, and the light boolean cleanup needed before execution
  - `compile.rs` now treats normalized boolean `Expr` as its input contract and lowers that contract into runtime `Predicate` without returning `Result`
- tightened the `WHERE` lowering contract itself to:
  - `SqlExpr -> lower_sql_expr(SqlExprPhase::PreAggregate)`
  - `validate_where_bool_expr(...)`
  - `normalize_where_bool_expr(...)`
  - `compile_where_bool_expr_to_predicate(...)`
  so rejection, shaping, and execution-form compilation no longer blur together inside one helper.
- kept the runtime `Predicate` layer as an execution concern rather than a second semantic owner by making compilation total over normalized input:
  - unsupported-shape rejection no longer lives in predicate compilation
  - compilation no longer performs SQL-front-end admission decisions
  - the last stage now only maps normalized boolean expression structure onto runtime `Predicate` shape, including the existing true-set / false-set tracking needed for `NOT`, `OR`, and searched `CASE`
- moved the `WHERE` predicate lowering code to a directory-module layout so the boundary is enforced physically as well as logically:
  - `db/sql/lowering/predicate/mod.rs` orchestrates the pipeline
  - `db/sql/lowering/predicate/validate.rs` owns admission
  - `db/sql/lowering/predicate/normalize.rs` owns shaping
  - `db/sql/lowering/predicate/compile.rs` owns structural predicate compilation
- added explicit regression-proofing so semantic normalization is harder to leak back into the compiler:
  - orchestration now `debug_assert!`s that normalization produced a compile-ready boolean expression
  - the compiler itself now `debug_assert!`s its normalized-expression contract before compiling
  - a new structural guard test checks that `compile.rs` does not depend on normalization helpers, SQL lowering helpers, `SqlExpr`, or `SqlLoweringError`
  - a new panic test proves that feeding a deliberately non-normalized `Expr` directly into the compiler trips the compile-boundary invariant instead of silently recompiling semantics there
- kept searched `CASE` compilation structural-only in that final stage:
  - branch order is preserved
  - branches are not simplified or reordered in the compiler
  - `CASE` lowering to runtime predicate form is now clearly downstream of normalization rather than another place where boolean semantics can drift
- reran focused lowering/runtime coverage for the cleaned-up boundary, including searched-`CASE` in `WHERE`, `EXPLAIN` parity, `NULL` boolean semantics, the new structural guard, and the compiler normalization panic proof, and `cargo clippy -p icydb-core --all-targets -- -D warnings` is green on the split layout.

## 0.91.1

- finished the SQL front-end unification line by removing the separate parser-owned `SqlPredicate` statement surface and making `WHERE` carry ordinary boolean `SqlExpr` trees instead, so `SELECT`, aggregate inputs, `HAVING` values, `ORDER BY`, and `WHERE` now all enter lowering through the same `SqlExpr -> lower_sql_expr(..., phase)` seam.
- kept runtime `Predicate` as an execution concern rather than a second semantic front end by making `WHERE` lowering do one shared `SqlExpr -> Expr -> Predicate` adaptation step in `db/sql/lowering/predicate.rs`; clause owners no longer parse or lower predicate semantics independently.
- admitted searched `CASE` in `WHERE` on top of that unified seam instead of special-casing it into the old predicate parser, and added focused parser, lowering, runtime, and explain coverage proving:
  - searched `CASE` in `WHERE` parses as `SqlExpr`
  - `WHERE` lowering preserves the same canonical boolean intent as equivalent direct predicates
  - `EXPLAIN` output for `WHERE CASE ...` matches the equivalent canonical boolean form
- tightened `WHERE` boolean/`NULL` semantics at the final predicate boundary so the two-valued runtime predicate layer still matches SQL row-filter behavior for:
  - bare `NULL` in boolean position
  - `NOT NULL`
  - combined boolean expressions that include `CASE` branches returning `NULL`
  - searched `CASE` paths where only the selected branch should determine row inclusion
- fixed the front-end semantic bug where SQL had been collapsing `IS NULL` / `IS NOT NULL` and `= NULL` / `!= NULL` into the same shape too early:
  - parser now preserves dedicated `SqlExpr::NullTest` nodes for `IS NULL` and `IS NOT NULL`
  - ordinary `= NULL` / `!= NULL` remain ordinary comparison expressions
  - `WHERE` lowering now maps explicit null tests onto runtime `Predicate::IsNull` / `Predicate::IsNotNull`
  - ordinary comparisons against `NULL` now stay distinct and collapse to unknown-at-`WHERE` row filtering instead of silently acting like null tests
- followed that null-test split through normalization, aggregate collection, type inference, and expression ownership so the new distinction is stable across parsing, lowering, fingerprints, and runtime evaluation without leaking the internal null-test helpers into projection or order-expression families.
- added focused proof coverage for the distinction at three levels:
  - parser tests prove `age IS NULL` and `age = NULL` now produce different SQL expression trees
  - lowering tests prove those two forms now build different planned query intents
  - runtime tests prove `age IS NOT NULL` and `age != NULL` behave differently even on the current non-nullable scalar fixture, with `IS NOT NULL` keeping rows and `!= NULL` keeping none

## 0.91.0

- introduced a real SQL-side expression boundary for reduced SQL before planner lowering, centered on `SqlExpr` plus one shared `lower_sql_expr(..., phase)` seam, so scalar projection, aggregate inputs, alias/order normalization, and grouped/global `HAVING` value lowering now all run through one planner-owned expression contract instead of clause-specific shells.
- widened the production planner and evaluator expression spine to admit boolean composition, comparisons, unary `NOT`, and planner-owned searched `CASE` nodes, which gives `CASE` one canonical runtime, explain, cache-key, and fingerprint representation instead of another SQL-only execution path.
- admitted searched `CASE WHEN ... THEN ... [ELSE ...] END` on the SQL surfaces that already sit on that shared expression seam:
  - `SELECT` projections
  - aggregate input expressions such as `SUM(CASE WHEN ... THEN ... ELSE ... END)`
  - grouped/global aggregate `HAVING` value expressions
- kept the first shipped `CASE` surface intentionally narrow:
  - searched `CASE` only
  - no simple `CASE x WHEN ...`
  - missing `ELSE` canonicalizes to planner-owned `NULL`
  - branch result typing reuses the existing scalar compatibility rules instead of adding CASE-specific coercions
  - non-boolean `WHEN` conditions and incompatible branch types fail closed during inference/lowering instead of at runtime
- migrated grouped/global `HAVING` value lowering off the older compare-shell-only path and onto the shared SQL-expression seam, so post-aggregate `CASE`, unary `NOT`, and the already-admitted arithmetic/wrapper expressions now lower through the same planner-owned post-aggregate expression contract.
- removed the last SQL front-end ownership split before `0.91.0` by introducing a parser-owned `SqlPredicate` statement model for `WHERE`, `DELETE`, and `UPDATE` selectors; SQL statements no longer carry runtime `Predicate` values directly, and lowering now owns the `SqlPredicate -> Predicate` conversion boundary.
- kept predicate traversal ownership where it belonged while doing that `WHERE` migration: SQL normalization now adapts parser-owned predicate identifiers by delegating through the existing predicate-field rewrite helper instead of taking ownership of predicate-tree traversal inside SQL lowering.
- added focused parser, lowering, inference, runtime, and structural-guard coverage for:
  - searched `CASE` parsing and rejection of simple `CASE`
  - missing-`ELSE` canonicalization to `NULL`
  - searched `CASE` laziness in projection evaluation
  - grouped/global aggregate `HAVING` value lowering through the shared seam
  - the new SQL predicate front-end ownership boundary for statement parsing and normalization
