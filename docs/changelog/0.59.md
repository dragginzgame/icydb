# IcyDB 0.59

This document aggregates detailed release notes for the `0.59.x` minor line.

## 0.59.6 - 2026-03-20 - Structural Runtime Authority Consolidation

### Summary

`0.59.6` continues the same executor-core closure work, but this patch is more
about eliminating the remaining structural authority drift than landing new
execution features. The main outcome is that both scalar and grouped kernel
dispatch now run from prepared structural runtime bundles, while the executor
no longer passes loose `model` / `entity_tag` / `entity_path` triples through
runtime-preparation APIs.

This still does not widen `0.59` into SQL, session, schema, or mutation work.
Typed entrypoints remain at the outer API boundary on purpose. The change is
that execution-core preparation now resolves typed entity authority once and
then hands runtime code one structural authority contract plus one prepared
runtime bundle.

### Changed

- Removed the remaining typed scalar and grouped kernel dispatch adapters:
  - scalar kernel dispatch now executes a prepared structural runtime bundle
    instead of boxing `LoadExecutor<E>` and `ExecutablePlan<E>` behind a typed
    scalar runtime adapter.
  - grouped kernel dispatch already ran through a prepared grouped runtime
    bundle after `0.59.5`; this patch aligned the grouped preparation shape
    with the new shared structural authority contract.
  - obsolete typed grouped-handoff helper layers and dead executor wrappers
    were deleted instead of being retained as compatibility ballast.
- Consolidated executor structural authority under one contract:
  - introduced one executor-owned `EntityAuthority` bundle carrying
    `EntityModel`, `EntityTag`, and `entity_path` together.
  - executor-boundary validation now consumes `EntityAuthority` instead of
    separate `model` and `path` arguments.
  - prepared scalar, grouped, and aggregate runtime payloads now keep one
    authority object instead of hand-carrying loose structural identity fields.
- Kept structural runtime assembly consistent across execution families:
  - scalar page finalization now uses a path-based structural helper instead of
    a typed `LoadExecutor<E>` finalization shim.
  - grouped runtime preparation now resolves observer/model authority from the
    same structural authority contract used elsewhere in executor prep.
  - aggregate fast paths, field-extrema helpers, numeric streaming folds,
    projection covering readers, and existing-row terminals were updated to
    consume prepared structural authority instead of raw `model` /
    `entity_tag` / `entity_path` fields.
- Closed the patch with cleanup required by strict linting:
  - removed the dead non-test `GroupedExecutorHandoff` root re-export after the
    grouped planner path stopped using the old typed convenience layer.
  - converted a remaining orchestrator payload projection helper into an
    associated function once it no longer depended on executor instance state.

### Validation

- `cargo test -p icydb-core stream_access_module_limits_direct_store_traversal_to_scan_boundary --lib --locked`
- `cargo test -p icydb-core --lib --locked`
- `cargo clippy --workspace --all-targets -- -D warnings`

### Measurement

No fresh wasm-size rerun was recorded for `0.59.6`.

### Next Step

`0.59.6` is a strong execution-core checkpoint. The remaining `0.59` work, if
the line continues at all, should stay narrow:

- remeasure wasm size for the updated executor-core checkpoint
- decide whether any remaining `E` usage inside execution-reachable helpers is
  true kernel duplication or just an outer typed boundary
- close the line once the remaining work no longer has clear structural-size
  payoff

## 0.59.5 - 2026-03-20 - Structural Runtime Route Consolidation

### Summary

`0.59.5` continues the same `0.59` execution-core target, but the focus is now
on tightening runtime ownership boundaries rather than widening de-typing into
new domains. The remaining cleanup in this patch removes more typed fast-path
runtime plumbing, keeps direct store/index traversal behind explicit scan
adapters, and reuses one executor-owned structural ordering contract across
page, ranking, and aggregate-adjacent row paths.

This does not change the `0.59` semantic boundary: schema/model, validation,
SQL lowering, session, and mutation APIs remain typed on purpose. The change is
that more executor internals now consume structural runtime inputs directly
instead of recovering typed context or re-owning cursor/order helpers in
execution-local modules.

### Changed

- Replaced erased typed fast-path runtime plumbing with structural execution
  runtime inputs:
  - the execution runtime adapter now carries one
    `StructuralTraversalRuntime` instead of erased `Context<'_, E>` pointers
    plus a typed vtable.
  - fast-stream route execution now runs through one structural
    `execute_fast_stream_route(...)` boundary instead of entity-typed
    `LoadExecutor<E>` leaf dispatch.
  - fallback executable-access stream resolution now reuses the same
    structural traversal runtime contract as the verified fast-path branches.
- Moved direct store/index traversal ownership onto explicit scan/store
  boundaries:
  - primary and secondary scan helpers now accept structural `StoreHandle` and
    `EntityTag` inputs so traversal/runtime code does not fetch registry handles
    in the middle of stream orchestration.
  - stream-access traversal now stays focused on spec consumption and recursive
    key-stream production; it no longer reaches into the store registry
    directly.
  - runtime helpers that must resolve named stores now do so through one
    structural `StoreLookup` / `StructuralStoreResolver` boundary instead of
    repeatedly threading typed `Context<E>` lookups through executor internals.
- Consolidated structural ordering under executor-owned helpers:
  - the old cursor-local ordering helper module was removed.
  - shared structural ordering now lives in `db::executor::order`, where slot
    resolution, row comparison, and cursor-boundary comparison stay executor
    owned.
  - scalar page materialization now applies ordering and boundary filtering
    through the same structural row contract used by the rest of the executor.
- Pushed more aggregate/materialized helpers onto structural runtime inputs:
  - aggregate fast paths and existing-row terminals now execute from structural
    traversal/store inputs instead of routing back through typed load executor
    shells.
  - materialized aggregate projection and `bytes(field)` covering helpers now
    validate row existence and index-store reads through structural store
    boundaries.
  - ranked materialized row helpers now operate through one structural helper
    that takes the resolved model once, rather than rebuilding typed execution
    context during ranking.

### Validation

- `cargo test -p icydb-core stream_access_module_limits_direct_store_traversal_to_scan_boundary --lib --locked`

No full `make test`, `cargo clippy`, or wasm-size rerun was recorded for this
checkpoint.

### Measurement

No fresh wasm-size rerun was recorded for `0.59.5`.

### Next Step

`0.59.5` keeps pushing the executor toward one shared structural runtime, but
the remaining work is now even narrower:

- run the full validation gate before cutting the release
- decide whether `0.59` closes here or continues only for deeper aggregate
  kernel de-monomorphization
- keep semantic-authority layers typed; any further `0.59` work remains
  execution-core only by design

## 0.59.3 - 2026-03-19 - Structural Materialization Follow-Through

### Summary

`0.59.3` continues the same `0.59` runtime target, but at a narrower and more
executor-local level than `0.59.2`. The remaining high-value work was no
longer broad architectural cleanup. Instead, it was concentrated in a handful
of execution families that still rebuilt typed access state or fully typed
entity responses in the middle of scalar execution.

This patch removes that remaining baggage from prepared scalar/aggregate
execution, executable-plan handoff, projection materialization, `bytes(field)`
fallbacks, and ranked-row terminals. The result is not that typed authority
goes away. It does not. The result is that more of the actual execution loop
now stays on structural access plans, structural row buffers, and slot-reader
evaluation until the final typed API edge.

### Changed

- Removed the remaining prepared-boundary typed access sidecars:
  - scalar and aggregate prepared execution no longer carry duplicated
    `AccessPlan<E::Key>` state after structural planning has already happened.
  - aggregate numeric, aggregate terminal, aggregate projection, and bytes
    materialized fallback paths now borrow structural access directly from the
    planned query contract instead of rebuilding typed access just to re-enter
    structural execution.
  - grouped-fold runtime setup now takes structural access directly through the
    runtime adapter rather than reconstructing typed access during aggregate
    runtime preparation.
- Removed the `ExecutablePlan<E>` typed access sidecar:
  - executable plans now hand off one structural `AccessPlannedQuery` plus
    lowered prefix/range specs rather than carrying an extra reconstructed
    typed access-plan payload.
  - delete scanning, grouped route-stage setup, and DISTINCT execution now
    consume structural plan state directly after boundary validation.
  - executor index-prefix/range lowering now takes explicit `EntityTag`
    authority and structural access-plan inputs instead of depending on typed
    access reconstruction.
- Pushed more materialized execution families onto structural row buffers:
  - scalar projection materialization now evaluates projection expressions
    directly over structural `DataRow` payloads and slot readers instead of
    running `execute(plan)` to rebuild `[(Id<E>, E)]` before projection.
  - `bytes()` and `bytes(field)` materialized fallbacks now read structural
    page payloads directly; `bytes(field)` extracts the requested field through
    slot readers instead of first building `EntityResponse<E>`.
  - ranked row terminals (`take`, `top_k_by`, `bottom_k_by`, and value/id
    ranking variants) now rank structural `DataRow` payloads plus extracted
    field values, and only reconstruct typed rows at the final `Rows` output
    edge.
- Removed more dead typed helpers exposed by the refactor:
  - dead entity-based aggregate field extraction helpers and obsolete scalar
    typed rows-boundary wrappers were removed instead of being retained as
    compatibility ballast.
  - stale boundary comments that still described typed access sidecars were
    updated so the code documentation matches the current structural execution
    model.

- `cargo test -p icydb-core projection_materialization_exposes_projected_rows_payload --lib`

`wasm-release` numbers documented below.

### Next Step

`0.59.3` is a reasonable release checkpoint for the current execution-core
cleanup, but it is probably the point where the remaining size work becomes a
different class of task:

- release this checkpoint once the full validation gate (`make test` and wasm
  size runs) is green
- if the line continues, the next meaningful size target is deeper aggregate
  kernel de-monomorphization rather than more boundary shuffling
- keep typed authority in schema/model, SQL lowering, validation, session, and
  mutation APIs; the remaining work is execution-core only by design

## 0.59.2 - 2026-03-19 - Query-Path Cleanup Follow-Through

### Summary

`0.59.2` follows through on the remaining query-path cleanup that still
mattered after the first scalar prepared-boundary checkpoint in `0.59.1`.
The execution core is now cleaner in four places that were still widening the
runtime surface: key canonicalization no longer has to reconstruct typed keys
in executor hot paths, materialized aggregate fallbacks now operate on
structural rows and slots instead of typed entity responses, grouped/scalar row
runtime stays structural through the last shared loops, and final typed
response assembly has been pushed to the outer response/paging boundary instead
of staying attached to `Context<E>`.

This still does not claim the `0.59` line is finished. Typed authority remains
deliberately in schema/model, SQL lowering, validation, session, and mutation
APIs. The point of this patch is narrower: remove the leftover query-path
duplication that was still sitting around the structural execution core after
`0.59.1`, and make that boundary explicit in the `0.59` design doc.

### Changed

- Removed the remaining query-path typed stream loops that still survived after
  `0.59.1`:
  - aggregate field-extrema streaming now folds over structural slot values and
    canonical `StorageKey` tie-break state instead of carrying `Option<E>`
    through the key-stream loop.
  - streaming numeric field aggregates now read structural row slots directly
    instead of decoding full entities before decimal coercion.
  - aggregate terminal field-value projection now reads one structural row and
    extracts the requested slot value directly instead of reconstructing `E`
    for a single-row terminal fetch.
  - grouped row runtime now reads persisted rows through structural row decode
    and builds `RowView` from slot values instead of decoding `E` and then
    immediately projecting it back into slots.
- Moved key canonicalization and access reconstruction onto structural
  boundaries:
  - executor fast paths, scan traversal, grouped fold setup, and execution
    preparation no longer reconstruct typed access plans just to re-enter the
    structural runtime.
  - `DataKey` now owns structural key canonicalization through
    `try_from_field_value(...)` / `try_from_structural_key(...)`, so executor
    key-range and by-key paths no longer need typed key reconstruction just to
    produce storage keys.
  - structural key dedup and entity-range bound construction now happen through
    one `DataKey` boundary that is covered by direct constructor-equivalence and
    bounds tests.
- Collapsed duplicated persisted-row decode shells:
  - delete execution, save-side existing-row identity checks, and runtime row
    response reconstruction now share one persisted-row decode boundary in
    `db::data` instead of each re-owning their own key/decode/mismatch shell.
  - typed row-buffer construction for `Vec<DataRow> -> EntityResponse<E>` now
    runs through one erased builder so the row iteration itself is no longer an
    executor-owned generic loop.
  - the raw pointer/vtable seam for execution-runtime erasure is now isolated
    in one dedicated `execution::erased` module rather than being duplicated in
    the runtime adapter file.
- Moved final typed response assembly onto the outer response/paging boundary:
  - structural scalar execution now returns structural cursor-page payloads all
    the way to the typed surface adapters.
  - `EntityResponse<E>` and `CursorPage<E>` now own typed reconstruction from
    structural `DataRow` payloads, instead of reusing a `Context<E>`-owned
    response helper.
  - this leaves execution/runtime code structural and keeps typed response DTO
    assembly in the response/paging layer where it belongs.
- Reduced aggregate materialized fallback dependence on typed entity response
  buffers:
  - scalar materialized-page execution now exposes structural page payloads to
    aggregate helpers before the outer typed projection edge.
  - materialized `nth` / `median` / `min_max`, numeric `sum` / `avg`, and value
    projection fallbacks now work from structural `DataRow` plus slot-reader
    extraction instead of rebuilding `EntityResponse<E>` in the middle of the
    aggregate path.
  - materialized aggregate fallback identity tracking now folds on `DataKey`
    and slot values instead of full typed entities where only structural state
    is required.
- Cleaned test-only scaffolding out of production runtime/planner files:
  - dead test-only executor stats fields and obsolete helper shims were removed
    instead of being carried forward as compatibility ballast.
  - larger inline test suites were moved into `tests.rs` or directory test
    modules so production files no longer need to carry wide `#[cfg(test)]`
    helper surfaces just to keep coverage readable.
- Locked the `0.59` design boundary to avoid de-typing semantic-authority
  layers:
  - the `0.59` design doc now explicitly separates typed authority,
    one-way typed-to-structural translation, and the structural runtime
    execution core.
  - SQL lowering, validation, session, schema/model, and mutation APIs remain
    intentionally typed; ongoing `0.59` work is constrained to runtime
    execution duplication.

## 0.59.1 - 2026-03-19 - Scalar Runtime Prepared-Boundary Checkpoint

### Summary

`0.59.1` takes the first substantial scalar execution slice after the grouped
runtime checkpoint in `0.59.0`. Scalar aggregate kernels, scalar terminal
families, scalar projection/numeric families, and scalar row production now run
through prepared structural contracts instead of carrying generic plan-owned
execution branching deeper into runtime code.

This patch does not claim the `0.59` line is finished. Typed API boundaries
still exist, and final typed decode still happens where it owns real entity
authority. The important change is that scalar runtime policy and scalar loop
ownership have moved decisively away from `ExecutablePlan<E>` / `LoadExecutor<E>`
and toward prepared, data-driven execution payloads.

### Changed

- Moved scalar aggregate kernel ownership onto structural runtime state:
  - scalar aggregate reducer state no longer owns `Id<E>` or
    `AggregateOutput<E>` in the execution core.
  - scalar aggregate outputs now flow through structural
    `StorageKey`-backed state and non-generic scalar aggregate output
    containers.
  - scalar aggregate execution no longer reconstructs `ExecutablePlan<E>`
    beneath prepared aggregate execution boundaries.
- Moved scalar load/runtime ownership onto structural execution stages:
  - shared scalar entrypoint orchestration now runs through one structural
    scalar execution stage instead of one typed load-owned loop.
  - scalar page materialization now uses one erased row-runtime handle and one
    structural page payload before typed decode at the outer edge.
  - scalar row-runtime descriptors are recovered once at the typed boundary and
    then reused by the shared scalar runtime.
- Made scalar row decode structural:
  - scalar row production now decodes persisted rows directly into structural
    slot-indexed row values instead of materializing `E` inside the main scalar
    loop.
  - scalar row decode is now owned by a `RowLayout` / `RowDecoder` contract in
    executor runtime code.
  - primary-key consistency is still validated at decode time, but the hot loop
    no longer depends on typed entity deserialization.
- Moved scalar family policy onto prepared contracts:
  - numeric field aggregates now lower once into
    `PreparedScalarNumericBoundary` plus prepared execution state before
    runtime dispatch.
  - scalar field projection terminals now lower once into
    `PreparedScalarProjectionBoundary` plus prepared execution state before
    runtime dispatch.
  - scalar terminal families (`COUNT`, `EXISTS`, `id`, `id_by`) now lower once
    into `PreparedScalarTerminalBoundary` plus prepared execution state before
    runtime dispatch.
  - runtime execution in those families now matches prepared strategy enums
    instead of deriving fast-path policy from `ExecutablePlan<E>` inside the
    execution path.
- Collapsed scalar wrapper fanout and removed dead compatibility surfaces:
  - aggregate terminal/numeric/projection families now prepare once and
    dispatch through shared family boundaries rather than through layered
    plan-owned wrappers.
  - non-aggregate scalar families (ranking and bytes) now share one prepared
    scalar materialized boundary instead of each re-owning generic
    `execute(plan)` setup.
  - dead executor wrapper names, rebuild helpers, and obsolete compatibility
    entrypoints exposed by the refactor were removed instead of being kept as
    long-lived aliases.
- Simplified grouped DISTINCT handoff beneath the scalar boundary:
  - scalar global DISTINCT lowering now prepares one grouped route stage and
    executes that directly.
  - grouped DISTINCT no longer rebuilds a grouped `ExecutablePlan<E>` beneath
    the scalar boundary just to re-enter grouped execution.
- Hardened development/setup support for the new invariant tooling:
  - the index-range invariant script now requires `rg` explicitly and checks
    the current shared traversal ownership sites.
  - the normal setup/update paths now install `ripgrep`, so fresh local and WSL
    environments provision the required invariant-check tool by default.

## 0.59.0 - 2026-03-18 - Execution De-Monomorphization Opens with Grouped Kernels

### Summary

`0.59.0` opens the execution de-monomorphization line by taking the first real
execution-kernel slice instead of continuing the `0.58.x` wrapper and boundary
cleanup work. The grouped aggregate kernel now stores structural reducer state,
finalizes grouped outputs without entity-typed aggregate containers, and runs
through one monomorphic grouped fold path without the old typed grouped hook
surface.

This is the first `0.59` change that materially moves execution specialization
itself rather than only rearranging orchestration. The grouped path still has a
typed stream-build and finalize boundary, but grouped engine construction,
group-admission handling, reducer state, and grouped fold ownership are now
structural.

### Changed

- Moved grouped aggregate reducer state onto structural runtime contracts:
  - grouped reducer state no longer carries `Id<E>` or other entity-typed
    grouped terminal containers.
  - grouped `MIN` / `MAX` / `FIRST` / `LAST` now retain structural
    `StorageKey` state and materialize `Value` only at finalize time.
  - grouped aggregate output/state containers are now non-generic and are owned
    by grouped runtime contracts instead of entity-typed aggregate shells.
- Made grouped aggregate accounting structural:
  - grouped budget tracking, group-admission accounting, and grouped slot-size
    estimation no longer depend on entity type parameters.
  - grouped implicit single-group admission for global DISTINCT now happens
    directly in the grouped fold runtime instead of through a typed hook
    surface.
- Removed the remaining typed grouped fold hook surface:
  - grouped engine construction is now a free structural function.
  - grouped fold execution is now a free structural function.
  - the old `GroupedFoldRuntimeHooks` boundary and typed grouped fold callback
    path were removed.
  - grouped entrypoint routing now calls the structural grouped fold runtime
    directly instead of re-entering a typed fold adapter.
- Kept the typed grouped boundary where it still owns real entity authority:
  - grouped stream construction still resolves typed access and row decode at
    the boundary.
  - grouped final output packaging and stream-finalization remain typed leaves.
  - this keeps the current slice focused on grouped kernel de-monomorphization
    rather than prematurely widening into scalar execution or access-plan
    redesign.
