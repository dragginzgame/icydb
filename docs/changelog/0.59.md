# IcyDB 0.59

This document aggregates detailed release notes for the `0.59.x` minor line.

## 0.59.1 - 2026-03-19 - Scalar Runtime Prepared-Boundary Checkpoint

### Summary

`0.59.1` takes the first substantial scalar execution slice after the grouped
runtime checkpoint in `0.59.0`, and then keeps going through the remaining
query-side cleanup that still mattered for monomorphization. Scalar aggregate
kernels, scalar terminal families, scalar projection/numeric families, scalar
row production, aggregate field-stream folds, and final response assembly now
run through prepared or structural contracts instead of carrying generic
plan-owned execution branching deeper into runtime code.

This patch does not claim the `0.59` line is finished. Typed API boundaries
still exist, and final typed decode still happens where it owns real entity
authority. The important change is that scalar runtime policy and scalar loop
ownership have moved decisively away from `ExecutablePlan<E>` / `LoadExecutor<E>`
and toward prepared, data-driven execution payloads.

### Changed

- Moved scalar aggregate kernel ownership onto structural runtime state:
  - scalar aggregate reducer state no longer owns `Id<E>` or
    `AggregateOutput<E>` in the execution core.
  - scalar aggregate outputs now flow through structural
    `StorageKey`-backed state and non-generic scalar aggregate output
    containers.
  - scalar aggregate execution no longer reconstructs `ExecutablePlan<E>`
    beneath prepared aggregate execution boundaries.
- Moved scalar load/runtime ownership onto structural execution stages:
  - shared scalar entrypoint orchestration now runs through one structural
    scalar execution stage instead of one typed load-owned loop.
  - scalar page materialization now uses one erased row-runtime handle and one
    structural page payload before typed decode at the outer edge.
  - scalar row-runtime descriptors are recovered once at the typed boundary and
    then reused by the shared scalar runtime.
- Made scalar row decode structural:
  - scalar row production now decodes persisted rows directly into structural
    slot-indexed row values instead of materializing `E` inside the main scalar
    loop.
  - scalar row decode is now owned by a `RowLayout` / `RowDecoder` contract in
    executor runtime code.
  - primary-key consistency is still validated at decode time, but the hot loop
    no longer depends on typed entity deserialization.
- Removed more executor-side typed row loops after the first scalar checkpoint:
  - aggregate field-extrema streaming now folds over structural slot values and
    canonical `StorageKey` tie-break state instead of carrying `Option<E>`
    through the key-stream loop.
  - streaming numeric field aggregates now read structural row slots directly
    instead of decoding full entities before decimal coercion.
  - grouped row runtime now reads persisted rows through structural row decode
    and builds `RowView` from slot values instead of decoding `E` and then
    immediately projecting it back into slots.
- Collapsed duplicated persisted-row decode shells:
  - delete execution, save-side existing-row identity checks, and runtime row
    response reconstruction now share one persisted-row decode boundary in
    `db::data` instead of each re-owning their own key/decode/mismatch shell.
  - typed row-buffer construction for `Vec<DataRow> -> EntityResponse<E>` now
    runs through one erased builder so the row iteration itself is no longer an
    executor-owned generic loop.
  - the raw pointer/vtable seam for execution-runtime erasure is now isolated
    in one dedicated `execution::erased` module rather than being duplicated in
    the runtime adapter file.
- Moved scalar family policy onto prepared contracts:
  - numeric field aggregates now lower once into
    `PreparedScalarNumericBoundary` plus prepared execution state before
    runtime dispatch.
  - scalar field projection terminals now lower once into
    `PreparedScalarProjectionBoundary` plus prepared execution state before
    runtime dispatch.
  - scalar terminal families (`COUNT`, `EXISTS`, `id`, `id_by`) now lower once
    into `PreparedScalarTerminalBoundary` plus prepared execution state before
    runtime dispatch.
  - runtime execution in those families now matches prepared strategy enums
    instead of deriving fast-path policy from `ExecutablePlan<E>` inside the
    execution path.
- Collapsed scalar wrapper fanout and removed dead compatibility surfaces:
  - aggregate terminal/numeric/projection families now prepare once and
    dispatch through shared family boundaries rather than through layered
    plan-owned wrappers.
  - non-aggregate scalar families (ranking and bytes) now share one prepared
    scalar materialized boundary instead of each re-owning generic
    `execute(plan)` setup.
  - dead executor wrapper names, rebuild helpers, and obsolete compatibility
    entrypoints exposed by the refactor were removed instead of being kept as
    long-lived aliases.
- Simplified grouped DISTINCT handoff beneath the scalar boundary:
  - scalar global DISTINCT lowering now prepares one grouped route stage and
    executes that directly.
  - grouped DISTINCT no longer rebuilds a grouped `ExecutablePlan<E>` beneath
    the scalar boundary just to re-enter grouped execution.
- Moved final typed response assembly onto the outer response/paging boundary:
  - structural scalar execution now returns structural cursor-page payloads all
    the way to the typed surface adapters.
  - `EntityResponse<E>` and `CursorPage<E>` now own typed reconstruction from
    structural `DataRow` payloads, instead of reusing a `Context<E>`-owned
    response helper.
  - this leaves execution/runtime code structural and keeps typed response DTO
    assembly in the response/paging layer where it belongs.
- Hardened development/setup support for the new invariant tooling:
  - the index-range invariant script now requires `rg` explicitly and checks
    the current shared traversal ownership sites.
  - the normal setup/update paths now install `ripgrep`, so fresh local and WSL
    environments provision the required invariant-check tool by default.

### Next Step

`0.59.1` is still not the end of the `0.59` line. The remaining work is now
mostly outside the hot executor loops:

- keep shrinking outer typed response/materialized DTO boundaries
- continue reducing typed API-edge setup that still survives above the
  structural execution core
- avoid widening SQL or mutation execution before the query-path
  de-monomorphization work is fully settled

## 0.59.0 - 2026-03-18 - Execution De-Monomorphization Opens with Grouped Kernels

### Summary

`0.59.0` opens the execution de-monomorphization line by taking the first real
execution-kernel slice instead of continuing the `0.58.x` wrapper and boundary
cleanup work. The grouped aggregate kernel now stores structural reducer state,
finalizes grouped outputs without entity-typed aggregate containers, and runs
through one monomorphic grouped fold path without the old typed grouped hook
surface.

This is the first `0.59` change that materially moves execution specialization
itself rather than only rearranging orchestration. The grouped path still has a
typed stream-build and finalize boundary, but grouped engine construction,
group-admission handling, reducer state, and grouped fold ownership are now
structural.

### Changed

- Moved grouped aggregate reducer state onto structural runtime contracts:
  - grouped reducer state no longer carries `Id<E>` or other entity-typed
    grouped terminal containers.
  - grouped `MIN` / `MAX` / `FIRST` / `LAST` now retain structural
    `StorageKey` state and materialize `Value` only at finalize time.
  - grouped aggregate output/state containers are now non-generic and are owned
    by grouped runtime contracts instead of entity-typed aggregate shells.
- Made grouped aggregate accounting structural:
  - grouped budget tracking, group-admission accounting, and grouped slot-size
    estimation no longer depend on entity type parameters.
  - grouped implicit single-group admission for global DISTINCT now happens
    directly in the grouped fold runtime instead of through a typed hook
    surface.
- Removed the remaining typed grouped fold hook surface:
  - grouped engine construction is now a free structural function.
  - grouped fold execution is now a free structural function.
  - the old `GroupedFoldRuntimeHooks` boundary and typed grouped fold callback
    path were removed.
  - grouped entrypoint routing now calls the structural grouped fold runtime
    directly instead of re-entering a typed fold adapter.
- Kept the typed grouped boundary where it still owns real entity authority:
  - grouped stream construction still resolves typed access and row decode at
    the boundary.
  - grouped final output packaging and stream-finalization remain typed leaves.
  - this keeps the current slice focused on grouped kernel de-monomorphization
    rather than prematurely widening into scalar execution or access-plan
    redesign.
