# 0.132.x

## 0.132.0

`0.132.0` opens the data-layer boundary cleanup by removing output shaping,
projection fallback behavior, and selective slot decoding from `db::data`. The
storage-facing data module now keeps typed raw-row decode primitives, structural
row byte contracts, and raw-row persistence surfaces, while executor
terminal/runtime ownership constructs response DTOs and chooses sparse slot
decode paths. The patch also splits the remaining persisted-row reader internals
into smaller reader-owned modules and tightens cursor-token ownership around the
cursor subsystem.

The typed response materialization helpers moved from `db::data::entity_decode`
to `executor::terminal::typed_response`. This keeps pagination and public
response DTO construction out of the row storage layer without changing row
decode behavior, key-consistency checks, cursor emission, or terminal response
shape. Scalar entrypoints and ranking terminals now consume the helpers through
the executor terminal boundary.

The generated persisted-row projection fallback was also removed.
`PersistedRow::project_slot(...)` previously allowed projection consumers to
materialize a full typed entity from slot readers and then ask
`FieldProjection::get_value_by_index(...)` for one slot. That behavior made the
persisted-row trait look like a projection runtime owner and kept a
full-materialization fallback inside the data layer. The derive macro no longer
generates that override, and projection tests now validate the projected value
shape through the dedicated `FieldProjection` API.

`DataStore::read_slot_values(...)` was removed. The only caller was the hybrid
SQL covering projection fast path, which now fetches one `RawRow` through
`DataStore::get(...)` and performs sparse slot decode in projection runtime
ownership. The single-slot path calls `RowDecoder::decode_required_slot_value`,
while multi-slot paths call the indexed row-layout decode. This keeps the
storage owner from selecting execution decode strategy while preserving the
existing one-slot optimization and missing-row behavior.

The persisted-row reader was split into a directory module after the data-layer
boundary cuts made its remaining mixed responsibilities explicit. The reader
root keeps `StructuralSlotReader` and the slot-reader trait implementations,
while direct dense/sparse decode, lazy slot-cache state, primary-key validation,
and structural-read diagnostics now live in separate reader-owned files. This
does not change persisted bytes, slot validation, sparse decode behavior, or
diagnostic counters; it only makes the reader internals composable enough for
future storage/runtime boundary audits.

The resulting `db::data` boundary no longer owns executor output shaping,
projection fallback behavior, or storage-trait selective decode helpers. It also
does not import executor or response DTO modules. Decode path selection now
stays with executor terminal/projection runtime code, while the data layer
retains storage-facing row and slot primitives.

The cursor token boundary was also tightened after the cursor audit.
`db::cursor` now owns external cursor token strings, including the lowercase
hex API-facing encode/decode helpers and the external string length limit.
`db::codec::hex` remains a generic hex transform only, and debug-only hex
snippets now call that generic helper directly instead of depending on cursor
token helpers.

The binary cursor token codec was split without changing the token wire format.
`cursor::token::codec` keeps only the scalar/grouped token envelope,
variant/version checks, direction fields, anchor fields, whole-token byte
limit, and scalar/grouped encode/decode orchestration. `cursor::token::bytes`
owns the bounded byte cursor, big-endian primitive helpers, length-prefixed
byte/string helpers, truncation checks, and trailing-byte checks.
`cursor::token::value` owns the cursor-local `Value` tag set and recursive
value payload read/write helpers; it intentionally remains cursor-local rather
than becoming a shared `Value` binary codec. The scalar and grouped token byte
limits now share one `MAX_CURSOR_TOKEN_BYTES` constant, and the external hex
limit derives from it as `MAX_CURSOR_TOKEN_BYTES * 2`.

The executor-facing cursor handoff now requires `PlannedCursor` explicitly
instead of accepting `Option<CursorBoundary>` through a blanket conversion.
This removes the raw-boundary bypass that let tests and future executor callers
construct runtime cursor state without naming the validation boundary. Scalar
load cursor entrypoints and `LoadCursorInput::scalar(...)` now take
`PlannedCursor` directly, while boundary-oriented tests construct planned
cursor state explicitly.

What changed:
- moved typed data-row to `EntityResponse` construction into
  `executor::terminal::typed_response`
- moved typed data-row to `CursorPage` construction into
  `executor::terminal::typed_response`
- removed executor and response DTO imports from `db::data::entity_decode`
- kept `decode_raw_row_for_entity_key(...)` in `db::data` as the storage-facing
  typed raw-row decode primitive
- retargeted scalar load entrypoints and ranking terminals to the executor
  terminal response owner
- removed `PersistedRow::project_slot(...)` from the persisted-row data trait
- stopped the derive macro from generating persisted-row projection overrides
- retargeted the structured field projection regression to
  `FieldProjection::get_value_by_index(...)`
- removed `DataStore::read_slot_values(...)`
- removed the now-unused `SelectiveRowRead` DTO from `db::data::row`
- moved hybrid SQL covering projection sparse row-field decode to projection
  runtime ownership, using explicit one-slot and indexed decode paths
- exposed the one-slot sparse decode path through `RowLayout` so SQL
  projection runtime does not need access to the executor-private `RowDecoder`
  type
- split `persisted_row::reader` into root reader, cache, direct decode,
  primary-key validation, and diagnostics modules
- moved cursor token string formatting from `db::codec` into `db::cursor`
- deleted the generic-looking `db::codec::cursor` helper module
- removed grouped cursor token one-off string formatting from
  `GroupedContinuationToken`
- routed grouped SQL cursor text emission through the cursor-owned string
  helper
- routed prepared execution plan debug hex snippets through
  `db::codec::hex::encode_hex_lower`
- added encode-side binary token size checks for scalar and grouped cursor
  tokens
- replaced separate scalar/grouped token byte constants with one
  `MAX_CURSOR_TOKEN_BYTES`
- derived external cursor string length from the binary token limit times two
- split cursor token byte primitives into `cursor::token::bytes`
- split cursor-local `Value` tag and payload encoding into
  `cursor::token::value`
- removed the blanket `Option<CursorBoundary> -> PlannedCursor` conversion
- narrowed scalar paged executor entrypoints and `LoadCursorInput::scalar(...)`
  to accept `PlannedCursor` directly
- updated cursor pagination tests to construct empty and boundary-backed
  planned cursors explicitly

Retained behavior:
- row decode and key-consistency validation still go through
  `decode_raw_row_for_entity_key(...)`
- typed load and ranking responses keep the same outward `EntityResponse`
  shape
- scalar paged loads keep the same `CursorPage` and cursor behavior
- generated persisted-row materialization and slot writing remain unchanged
- hybrid covering projection still skips missing primary rows and still keeps
  the one-slot sparse decode optimization
- persisted-row reader validation, lazy materialization, sparse decode, and
  structural-read metrics behavior remain unchanged
- cursor token wire vectors remain unchanged
- cursor string tokens remain lowercase hex
- cursor value payload tags remain private to cursor token encoding
- executor cursor validation/revalidation behavior remains unchanged; only the
  implicit raw-boundary construction path was removed
