# 0.155.x

## 0.155.6

`0.155.6` closes the next post-DDL visibility gap by making accepted
DDL-published indexes visible to `DESCRIBE`, `EXPLAIN EXECUTION`, and indexed
reads, while pinning schema-scoped cache and continuation behavior across DDL
publication.

### ­¤Éø Fixed

- Accepted-schema `DESCRIBE` now renders persisted index metadata instead of
  reopening generated model indexes, so DDL-published field-path indexes appear
  in the same schema description surface as generated indexes.
- Planner semantic validation now accepts secondary access contracts that are
  present in `SchemaInfo`, allowing DDL-published accepted indexes to flow from
  planning into EXPLAIN/execution without requiring a generated `IndexModel`.
- Shared query-plan cache coverage now proves DDL publication produces a new
  accepted-schema cache key instead of reusing a pre-DDL plan.
- Cursor coverage now proves a continuation cursor minted before DDL
  publication is rejected when the post-DDL accepted schema changes the planned
  access shape.
- Source-boundary coverage now guards the SQL DDL frontend against direct
  schema-store publication and generated-index metadata authority.

## 0.155.5

`0.155.5` hardens the generated admin SQL and DDL canister boundary and adds
DDL rebuild metrics to successful DDL payloads.

### Ō×Ģ Added

- Successful DDL payloads now include `rows_scanned` and
  `index_keys_written`, threaded from the existing field-path rebuild developer
  report into the public SQL result and shell renderer.
- Added CLI-side DDL response coverage that decodes the same
  `Result<SqlQueryResult, icydb::Error>` shape returned by the `ddl` update
  endpoint and verifies the rendered output includes the rebuild metrics.
- The PocketIC SQL canister boundary tests now exercise the generated
  `icydb_admin_sql_query` endpoint and decode its perf envelope, matching the
  removed demo-only `query`, `query_numeric_types`, and `update` SQL exports.
- The PocketIC SQL canister boundary tests now also exercise the generated
  `ddl` update endpoint for the supported `CREATE INDEX ... ON entity
  (field_path)` path and verify the published index is visible through
  `SHOW INDEXES`.

### ­¤Éø Fixed

- Admin SQL generated dispatch now routes no-entity introspection statements
  such as `SHOW TABLES` and `SHOW ENTITIES` through a canister-owned entity
  when one exists instead of falling through to the empty-canister error path.
- The generated `ddl` endpoint now has live fail-closed coverage proving an
  unknown accepted field path rejects without publishing a new index or changing
  `SHOW INDEXES` visibility.
- The generated `ddl` endpoint now also has live accepted-catalog rejection
  coverage for duplicate index names and duplicate field-path indexes, with
  `SHOW INDEXES` proving rejected candidates do not become visible.
- The generated `ddl` endpoint now has live parser-shape rejection coverage for
  `CREATE UNIQUE INDEX`, multi-field keys, expression keys, and filtered
  indexes, with accepted index visibility unchanged after every rejection.

## 0.155.4

`0.155.4` moves ordinary CLI SQL reads onto the standard controller-gated
admin SQL query surface.

### Ō×Ģ Added

- The demo RPG and lightweight SQL test canisters now expose
  `icydb_admin_sql_query`, a read-only SQL query endpoint that requires the
  caller to be a canister controller and returns the shell perf envelope,
  through the new lifecycle-hooked `icydb::admin_sql_query!` canister macro.
- `icydb sql` now calls `icydb_admin_sql_query` for ordinary read SQL while
  keeping supported DDL on the existing `ddl` update endpoint, so ordinary
  reads keep their perf footer while moving off the demo-specific
  `query_with_perf` method name.
- `icydb canister refresh` now provides a generic destructive
  rebuild/reinstall command for any selected canister, and the repo-specific
  `icydb demo` command family was removed. Fixture loading is now left to
  canister-specific APIs instead of IcyDB CLI self-test commands.
- Added `scripts/dev/sql-start-demo` as a repository-local helper for the demo
  RPG refresh, fixture load, and SQL smoke-check workflow.
- The demo RPG and lightweight SQL test canisters no longer carry their own
  duplicate `query_with_perf` endpoint implementations; the shared admin SQL
  macro owns that perf envelope.
- The demo RPG and lightweight SQL test canisters no longer export their old
  generic `query`, `query_numeric_types`, or `update` SQL endpoints; read SQL
  now goes through the controller-gated admin endpoint and DDL goes through the
  macro-owned controller-gated `ddl` update endpoint.
- Demo fixture reset/load-default exports are now macro-owned controller-gated
  lifecycle methods. Reset clears every generated canister-owned entity across
  stores, and load-default runs that reset before calling the conventional
  canister-local `icydb_admin_sql_load_default` hook.
- SQL DDL-created field-path indexes now persist the owning entity store path
  instead of an index-specific pseudo-store path, so recovery can rebuild them
  through the generated store registry after an upgrade.
- Admin SQL generated dispatch now matches SQL target identifiers against the
  public entity schema name as well as the Rust path, so lowercase shell input
  such as `DESCRIBE character` routes correctly.
- `SHOW TABLES` now renders as a small table with a table-count footer instead
  of the old `surface=entities` metadata lines, so shell perf attribution lands
  on a readable count line.
- The local SQL demo docs now describe the standard admin SQL endpoint instead
  of treating the demo-specific perf query method as the shell contract.

### ­¤Éø Fixed

- No separate bug-fix-only entry was recorded for `0.155.4`; the patch was the
  admin SQL migration slice described above.

## 0.155.3

`0.155.3` executes the one supported SQL DDL path through the schema-owned
field-path rebuild runner.

### Ō×Ģ Added

- `execute_sql_ddl` now routes supported non-unique field-path `CREATE INDEX`
  commands through the same physical rebuild and accepted-snapshot publication
  boundary used by startup reconciliation.
- SQL statement results now include a structured DDL payload. The public
  `icydb` SQL facade renders it as `surface=ddl` with mutation kind, target
  index, target store, field path, and publish status.
- The demo RPG and test SQL canisters now expose a `ddl` update method that
  routes through `execute_sql_ddl`.
- The `icydb sql` shell now detects the supported `CREATE INDEX` DDL shape and
  calls `icp canister call <canister> ddl ... --output hex --environment <env>`
  without `--query`, while read statements continue through `query_with_perf`.
- Accepted-schema reconciliation now treats generated schema plus supported
  DDL-created non-unique field-path indexes as compatible, so later runtime
  schema loads keep the accepted snapshot as authority instead of trying to
  remove DDL-created indexes that generated metadata does not declare.
- SQL `SHOW INDEXES` now reads accepted schema index metadata, so indexes
  created by supported DDL appear in the same developer introspection surface
  as generated indexes.

## 0.155.1

`0.155.1` adds the first SQL DDL frontend slices while keeping execution
fail-closed.

### Ō×Ģ Added

- Added a parsed DDL statement boundary for
  `CREATE INDEX name ON entity (field_path)`. The parsed intent remains
  unresolved and cannot enter mutation planning yet.
- Added accepted-catalog binding for parsed `CREATE INDEX` statements. Binding
  resolves the target entity and field path through `SchemaInfo`, rejects
  unknown fields, duplicate index names, and duplicate field-path indexes, and
  still stops before mutation planning or physical store work.
- Added schema-owned mutation admission for bound `CREATE INDEX` candidates.
  The SQL DDL lowerer now proves the candidate accepted index snapshot lowers
  through the existing schema mutation request, mutation plan, execution plan,
  and supported field-path runner admission path without starting execution.
- Added accepted-after snapshot derivation for bound `CREATE INDEX`
  candidates. The derivation appends the candidate index to the accepted-before
  snapshot, revalidates accepted schema integrity, and proves the resulting
  before/after delta still admits only the supported field-path runner path.
- Added a prepared-only SQL DDL command/report surface. It packages the bound
  request, accepted-after derivation, target index/store/path, mutation kind,
  and explicit `PreparedOnly` execution status for the future execution slice.
- Unsupported DDL shapes such as `CREATE UNIQUE INDEX`, multi-field index keys,
  expression index keys, and filtered indexes fail closed at the parser
  boundary.
- Query, update, and model-only lowering surfaces now reject parsed DDL before
  executable SQL command compilation, query binding, or physical mutation work.

## 0.155.0

`0.155.0` opens the DDL foundation track while keeping implementation scope
fail-closed around the one developer-supported schema mutation path from
0.154.

### ­¤ōÜ Documentation

- Added the 0.155 DDL foundation design. The target surface is deliberately
  narrow: one non-unique field-path secondary-index add through a future
  `CREATE INDEX name ON entity (field_path)` frontend.
- The design keeps accepted persisted schema snapshots as catalog authority,
  keeps SQL DDL as a frontend over schema mutation requests/plans, and rejects
  unsupported DDL before physical work or schema publication.
- The design records execution ordering, developer diagnostics, concurrency
  assumptions, test coverage, and deferred work for unique indexes, expression
  indexes, `DROP INDEX`, `ALTER TABLE`, batch DDL, and online DDL.

### ­¤®╣ Fixed

- Startup reconciliation now rejects the supported field-path index-add rebuild
  path if the physical index store is not ready before rebuild work starts.
  That keeps stale or unpublished physical work from being reused as a publish
  candidate during startup.

### ­¤¦╣ Cleanup

- Moved the runtime-startup field-path schema mutation bridge into a focused
  startup adapter module. General schema reconciliation now delegates the
  physical field-path path instead of carrying the full runner bridge inline.
- Updated write-boundary guards so the supported startup publication path stays
  visible after the split and cannot drift back into direct schema-store writes
  or generated index authority.

### ­¤ōŖ Audit

- Refreshed the dry-consolidation report after the 0.154 closeout baseline.
- Refreshed recovery consistency guidance and report coverage around migration
  replay and non-ready staged physical work.
- Refreshed resource-model and error-taxonomy recurring audit definitions and
  reports so future constructive audits have current method language.
- Refreshed the query instruction footprint audit so the dedicated PocketIC SQL
  and typed/fluent perf harnesses are the recurring authority, while old
  generated demo dispatch sampling is optional compatibility context.
