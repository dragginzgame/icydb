# 0.143.x

## 0.143.4

`0.143.4` finishes the expression/executor separation follow-up. Compiled
grouped aggregate expressions, grouped projection expressions, and residual
filter programs now keep expression-shape dispatch inside the expression and
planning layers. Executor row loops load slots, call compiled evaluators, and
continue with aggregation, projection, filtering, and paging mechanics without
matching on expression or filter variants.

### 🔧 Changed

- introduced expression-layer `CompiledExpr` and `CompiledPredicate`
  interfaces, with `GroupedCompiledExpr`, `PredicateProgram`, and the effective
  runtime filter wrapper implementing the compiled contracts
- moved grouped aggregate input/FILTER expression evaluation out of
  `executor::aggregate::contracts::state::grouped_expr` and into
  `query::plan::expr::compiled_expr`
- moved grouped projection and grouped HAVING expression evaluation out of
  `executor::projection::grouped` and into the same compiled expression owner
- replaced executor-facing `EffectiveRuntimeFilterProgram::{Predicate, Expr}`
  matching with one compiled filter wrapper that exposes evaluation and
  slot-marking methods
- wrapped expression-backed residual filters as TRUE-only compiled predicates
  while keeping predicate-backed residual filters on the existing predicate
  runtime program
- added slot/value reader adapters so executor row views provide loaded values
  without expression evaluators depending on executor row-view concrete types
- updated retained-slot and grouped-row layout code to ask the compiled filter
  for referenced slots instead of inspecting predicate-vs-expression shape

### 🧭 Boundary notes

- query results, SQL truth admission, predicate semantics, and projection
  output are unchanged
- expression recursion still exists, but it is contained inside the expression
  layer rather than executor row loops
- the predicate-vs-expression residual filter choice is private to the compiled
  filter wrapper; executor modules no longer match on that filter shape
- predicate runtime may still select scalar-vs-generic execution internally,
  but scan and grouped execution call the compiled filter surface
- grouped projection and grouped HAVING now share the expression-layer compiled
  grouped projection evaluator

## 0.143.3

`0.143.3` closes the cleanup and grouped-execution optimization slice. Fluent
load terminals are now monomorphic descriptors that call matching `DbSession`
execution/explain entrypoints directly, aggregate cache identity follows
aggregate semantics, and grouped aggregation does less work in its hot loop by
borrowing group-key inputs and keeping SUM/AVG reducer state typed until
finalization.

### 🔧 Changed

- replaced existing-row, scalar id/extrema, order-sensitive, projection, and
  numeric terminal strategies with concrete descriptors such as
  `CountRowsTerminal`, `MinIdBySlotTerminal`, `SumDistinctBySlotTerminal`, and
  `ValuesBySlotWithIdsTerminal`
- removed fluent terminal transport output enums and the associated `into_*`
  decode layer
- moved typed result shaping to direct descriptor-specific session entrypoints
- deleted the old `query/fluent/load/runtime.rs` terminal adapter layer
- made projection explain generic over descriptor-owned `ProjectionExplain`
  metadata instead of a shared projection strategy type
- aligned structural query cache keys with aggregate semantic identity, so
  `MIN(DISTINCT field)` shares a prepared-plan cache key with `MIN(field)`
  while `COUNT(DISTINCT field)` and `SUM(DISTINCT field)` remain distinct from
  their non-DISTINCT forms
- added a source guard that rejects the old shared strategy/request/transport
  names in the fluent terminal surfaces
- split runtime `Value` responsibilities so data representation, behavior
  operations, semantic classification, map normalization, and partial-ordering
  logic have separate owners
- moved grouped fold key lookup onto stable-hash buckets that can compare
  borrowed row-slot values before materializing an owned group key
- pre-sized grouped hash state from cheap candidate-count hints where available
  so grouped fold avoids unnecessary hash-table growth
- kept the common grouped `COUNT(*)` path on direct slot hashing and count
  updates, with per-stage instruction counters preserved for audit visibility
- specialized grouped `SUM` and `AVG` reducers to store typed decimal
  accumulators directly instead of routing every update through a generic
  `ValueReducerState`
- resolved direct grouped `SUM`/`AVG` field inputs as borrowed row-slot values
  before numeric admission, avoiding cloned slot `Value`s on that path
- collapsed one-arm grouped searched `CASE` expressions with slot-literal or
  boolean-slot conditions into direct branch evaluators, and selected constant
  `TRUE`/`FALSE`/`NULL` CASE conditions during grouped expression setup

### 🧭 Boundary notes

- query planning, SQL execution, and executor boundary DTOs are unchanged
- terminal descriptors still build the same executor boundary requests as the
  old strategies, but each descriptor now has one fixed operation and output
  shape
- aggregate cache identity is semantic for aggregate DISTINCT behavior; SQL
  compiled-command cache identity can still preserve spelling where it owns the
  SQL surface cache
- aggregate-local FILTER expressions remain part of the structural query cache
  key, so filtered aggregate calls do not alias just because their aggregate
  function and input match
- grouped key equality still uses canonical `Value` equality semantics; the new
  borrowed path only skips allocation when the existing owned group key can be
  proven equivalent to the current row slots
- grouped SUM/AVG still use the shared numeric coercion and checked decimal
  arithmetic rules, and grouped output still converts back to `Value` only at
  finalization
- grouped HAVING remains a post-aggregation/finalization concern; this slice
  only removes repeated CASE condition work from grouped aggregate input
  evaluation
- public fluent method signatures and result shapes are unchanged

## 0.143.2

`0.143.2` tightens grouped aggregate execution after the expression and SQL
compile-pipeline cleanup. Grouped fold now decodes each required row slot once
into a retained row view, then evaluates aggregate input and FILTER expressions
through grouped-local slot-indexed programs instead of re-entering the generic
scalar projection evaluator for every aggregate row.

### 🔧 Changed

- decoded grouped retained slots once at the grouped row-runtime boundary and
  reused borrowed slot values across residual filters, group-key handling, and
  aggregate input evaluation
- replaced the grouped aggregate input/FILTER scalar-evaluator callback path
  with `GroupedCompiledExpr`, a grouped-fold-local evaluator that stores field
  references as resolved slot indexes
- added direct grouped evaluator variants for slot reads, slot-slot arithmetic,
  and slot-literal binary expressions while preserving shared NULL, numeric,
  comparison, and error semantics
- kept searched CASE, unary, function-call, and fallback binary behavior in the
  grouped evaluator so large or complex aggregate inputs remain semantically
  equivalent to scalar projection evaluation
- preserved the common grouped `COUNT(*)` fast path and the single-slot row-view
  path while moving multi-slot grouped execution onto retained decoded slots

### 🧪 Tests

- added owner-local `GroupedCompiledExpr` coverage for direct slot reads,
  slot-slot arithmetic, searched `CASE` TRUE-only branch selection,
  `NULL`/`FALSE` fallthrough, scalar function calls, and missing-slot
  diagnostics

### 🧭 Boundary notes

- planner and SQL compilation behavior are unchanged
- grouped expression compilation happens when the grouped aggregate bundle is
  prepared, not during the SQL compile pipeline or per-row fold loop
- grouped row views remain immutable after decode; expression evaluation borrows
  slot values and does not mutate or invalidate retained storage
- shared projection operators still own scalar arithmetic, comparison, unary,
  function, NULL, and numeric error semantics
- the grouped fold perf audit still identifies remaining cost in grouped fold
  work outside the removed AST/scalar-evaluator traversal

## 0.143.1

`0.143.1` aligns expression truth admission with predicate compilation after
the expression-pipeline sealing work, then tightens the session-owned SQL
compile pipeline into a clearer parse → compile-entry → compile-core →
statement-compiler flow. The shared truth-value contract now defines the only
TRUE-admitting value, treats `NULL` as SQL UNKNOWN, and is used by scalar
filters, scalar `CASE` conditions, aggregate FILTER checks, and
grouped/HAVING-style evaluation boundaries. SQL compilation now carries a
single semantic shape descriptor alongside compiled command artifacts so
compile-time facts such as aggregate, row-returning, and mutation shape are
recorded once at the compile boundary.

### 🔧 Changed

- documented `truth_value` as the single owner for evaluated boolean admission
- added a borrowed truth-admission helper so callers holding `&Value` or
  `Cow<Value>` do not allocate on the successful path
- routed scalar filter, scalar `CASE`, and aggregate terminal filter checks
  through the shared truth-value helper
- documented predicate compilation as a TRUE-set lowering boundary where SQL
  UNKNOWN is represented by neither the TRUE set nor the FALSE set
- moved SQL surface validation into the compile entry boundary so callers no
  longer have to satisfy an external precondition before compilation
- split SQL statement compilation into dedicated SELECT, DELETE, INSERT,
  UPDATE, EXPLAIN, DESCRIBE, and SHOW helpers while keeping cache and
  measurement ownership outside the per-statement compilers
- replaced the ad-hoc aggregate compile flag with `SqlQueryShape`, a compact
  compile-owned descriptor for aggregate, row-returning, and mutation
  classification
- added construction-time debug invariants that keep `SqlQueryShape` aligned
  with the compiled command payload without adding release-build recomputation
- kept INSERT and UPDATE compilation on the minimal prepare → extract →
  artifact path because those write statements do not need full lower/bind
  work before execution

### 🧪 Tests

- added predicate-bridge coverage comparing actual `PredicateProgram`
  execution with expression evaluation followed by `truth_value`
- covered simple comparisons, `NULL` input, boolean composition, scalar
  searched `CASE`, and HAVING-like searched `CASE`

### 🧭 Boundary notes

- predicate runtime behavior does not change
- projection evaluation still owns scalar value materialization
- SQL three-valued expression evaluation remains separate from TRUE-only
  admission
- SQL compiled-command cache behavior is unchanged; the cache shell still owns
  key construction, lookup, parse, compile-entry dispatch, and insertion
- SELECT and DELETE compiled commands still retain shared query ownership where
  cached command reuse needs cheap cloning
- compile attribution fields are unchanged, but miss-path wiring now flows
  through a small builder so cache, parse, compile-core, and insert counters
  stay ordered in one place

## 0.143.0

`0.143.0` opens a bugfix, refactor, optimization, and cleanup slice by fixing
one value-storage validation bug in generated enum payload handling. Fresh rows
whose enum variant payload stored a ULID could fail reload because the enum
payload splitter checked for `NULL` with the generic Structural Binary parser
instead of the value-storage skipper. The generic parser does not own local
value-storage tags such as the ULID tag, so valid fresh bytes were rejected
before the ULID payload decoder ran.

### 🩹 Fixed

- fixed generated enum payload splitting for variants carrying local-tagged
  value-storage payloads such as `Value::Ulid`
- fixed fresh-row reloads for simple enum-backed fields whose selected variant
  carries a ULID payload

### 🔧 Changed

- routed the value-storage null probe through the value-storage skipper so the
  validation path understands the same local tag set as value decode
- kept local value-storage tag ownership inside the value-storage boundary
  instead of widening the generic Structural Binary parser

### 🧭 Boundary notes

- persisted bytes do not change
- by-kind structural field decoding remains separate from value-storage decode
- `NULL` payload detection still validates that the nested payload contains
  exactly one bounded value

  `cargo test -p icydb-core binary_value_storage_enum_splitter_accepts_local_tagged_payload -- --nocapture`
- ran `cargo test -p icydb-core binary_value_storage -- --nocapture`
