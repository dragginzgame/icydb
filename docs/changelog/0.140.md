# 0.140.x

## 0.140.0

`0.140.0` records the completed aggregate identity convergence design and the
large-file audit cleanup that followed it. Query behavior, SQL results, grouped
aggregate results, executor semantics, cursor behavior, projection output,
error surfaces, and persisted formats remain unchanged.

### 📚 Documentation

- finalized the aggregate identity convergence design under
  `docs/design/0.140-aggregate-identity-convergence/`
- documented `AggregateIdentity` as the shared semantic core for aggregate kind,
  input identity, and observable `DISTINCT` behavior
- documented `AggregateSemanticKey` as the filter-aware equivalence boundary:
  aggregate-local `FILTER` expressions stay separate from aggregate identity
  while remaining part of deduplication and comparison
- clarified that SQL global aggregate semantics is a SQL-facing wrapper over
  shared identity, not an independent DISTINCT-normalization owner
- clarified that grouped aggregate specs may keep raw construction fields only
  before identity construction; semantic decisions should consume identity or a
  semantic key
- explicitly kept grouped planner rules, reducer behavior, projection ordering,
  output shapes, and execution architecture out of scope

### 📊 Audit

- audited structural value-storage decode paths for allocation pressure,
  recursive traversal cost, skip/decode overlap, and future zero-copy seams
- audited `db/session/query` as a session-boundary module and split the major
  responsibility clusters into separate owners
- audited grouped execution cost at the instruction level, with focus on
  grouping key materialization, row views, hash-table updates, and aggregate
  fold/finalize phases
- audited scalar execution from entrypoint through kernel, projection
  materialization, predicate evaluation, and expression evaluation
- audited large scalar entrypoint code for concentrated runtime construction,
  materialized execution, streaming sink execution, finalization, public
  entrypoints, and diagnostics concerns

### 🔧 Changed

- split `db/session/query` into focused modules for cache, execution, paging,
  fluent terminal adapters, explain/trace surfaces, and diagnostics attribution
- made session query execution converge on a single prepared execution pipeline
  so diagnostics reuses the same scalar/grouped/delete branching as normal
  execution
- reorganized scalar executor entrypoints into runtime construction,
  materialized execution, streaming sink execution, finalization, public
  entrypoints, and diagnostics modules
- removed production grouped row materialization through per-row
  `Vec<Option<Value>>` staging and kept grouped reads on borrowed row views
- unified aggregate identity naming around `AggregateIdentity` so grouped and
  SQL aggregate comparisons share one identity vocabulary
- added focused structural value-storage decode improvements, including
  pre-sized list/map materialization, fixed-arity tuple splitting without heap
  staging, parsed scalar helpers, single-pass list/map decode, and visitor-style
  borrowed slice traversal helpers
- added scalar projection fast paths for direct single-slot projections and
  model-identity projections while preserving the final `Value` row
  representation

### 🧹 Cleanup

- reduced large-file overlap by moving logic into owner modules instead of
  letting session, executor entrypoints, and projection materialization act as
  catch-all boundaries
- kept public APIs, binary formats, cursor formats, projection labels, and
  response DTO shapes stable
- left walker APIs, borrowed slice split helpers, and map normalization behavior
  intact in structural value-storage code
- documented traversal ownership where skip-based traversal remains the
  authoritative structural validation model and decode-based traversal owns
  materialization over bounded slices

### 🧭 Retained behavior

- `MIN(DISTINCT ...)` and `MAX(DISTINCT ...)` remain equivalent to plain
  `MIN` / `MAX`
- `COUNT(DISTINCT ...)`, `SUM(DISTINCT ...)`, and `AVG(DISTINCT ...)` remain
  distinct-sensitive
- aggregate-local `FILTER` expressions continue to prevent otherwise matching
  aggregates from deduplicating
- grouped and SQL global aggregates remain separate systems with shared
  identity rules rather than one unified aggregate execution model
