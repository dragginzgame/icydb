# 0.118 Changelog

## 0.118.0

- `0.118.0` starts the expression-pipeline flow-collapse line by removing predicate as the owner of the branchy normalized-`Expr` -> runtime predicate compile lane. The heavy compile/readiness path that decides whether one normalized planner boolean expression can collapse onto the runtime predicate shell now lives under planner expression ownership in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/predicate_compile.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/predicate_compile.rs), while [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs) is reduced to the actual predicate adapter path.

- This is a flow contraction, not a semantic widening. The admitted boolean-expression family does not change. The runtime predicate shell still supports the same bounded normalized shapes as before:
  - bare boolean fields
  - boolean and null literals
  - `NOT`, `AND`, and `OR`
  - the admitted compare family
  - the admitted null-test, text, missing/empty, and collection-contains function family
  - normalized searched-`CASE` trees that remain inside that compile-ready surface

- Before this patch, predicate still owned two large flow-participant ladders in [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs):
  - the compile-readiness classifier for normalized boolean expressions
  - the recursive truth-set compiler that turned normalized planner `Expr` trees into runtime `Predicate` truth/false branches

- That meant the runtime flow for normalized boolean expressions still looked like:
  - planner owned semantic normalization and truth-condition meaning
  - predicate then re-branching on the normalized expression tree to decide compile readiness and runtime predicate lowering

- After `0.118.0`, that branching owner moves behind planner `expr`:
  - `compile_normalized_bool_expr_to_predicate(...)` owns the normalized planner-expression -> runtime predicate compilation path
  - `derive_normalized_bool_expr_predicate_subset(...)` owns the compile-ready subset gate for normalized planner expressions
  - the associated membership-collapse, compare/function truth-set, null/text target, and compile-ready ladders moved with that owner

- Predicate now keeps the actual adapter boundary that still belongs there:
  - `predicate_to_bool_expr(...)` still converts runtime predicates onto planner-compatible boolean `Expr`
  - `canonicalize_predicate_via_bool_expr(...)` still routes the runtime predicate shell through the planner-owned normalization seam and back
  - the public/internal predicate-facing function names stay stable, but they delegate into the planner-owned compile lane instead of owning it

- This matters because it is the first real `0.118` move that reduces module participation instead of just centralizing meaning:
  - planner now owns semantic boolean normalization and the compile/readiness branching over normalized planner `Expr`
  - predicate is closer to an adapter shell and less of a second expression-routing stage
  - lowering, intent, explain, and planner semantics now call the planner-owned compile lane directly instead of routing normalized planner `Expr` back through predicate-facing compile helpers

- The structural follow-through is also frozen now:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/tests/structural_guards.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/tests/structural_guards.rs) now reads the shared boolean compiler from the planner expression boundary instead of assuming it still lives in predicate
  - the existing lowering-side guard still proves the WHERE predicate compiler remains a thin wrapper over the shared compiler instead of reopening semantic normalization or SQL lowering locally

- That direct-caller cleanup now reaches the concrete non-predicate flow participants that used to keep predicate in the middle of the path:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/mod.rs) now derives the optional runtime predicate subset from planner `expr` directly
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/compile.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/predicate/compile.rs) now compiles normalized boolean expressions through planner `expr` directly
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/intent/model.rs), [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs), and [/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/shared/predicate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/executor/explain/descriptor/shared/predicate.rs) now consume the planner-owned subset compiler directly
  - the old crate-wide predicate re-exports for the normalized `Expr` compile/subset lane are gone because they are no longer part of the shared flow surface

- The same flow collapse now also covers the runtime-predicate -> planner-expression bridge itself:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/predicate_compile.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/predicate_compile.rs) now owns `canonicalize_runtime_predicate_via_bool_expr(...)` plus the structural `Predicate -> Expr` rebuild helpers behind it
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/logical.rs) now uses that planner-owned canonicalizer directly instead of routing planner-side predicate equivalence checks back through predicate
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs) is now effectively test-only glue around that planner-owned bridge instead of the owner of the transport lane

- The patch also includes one small internal cleanup that follows from the owner move:
  - the moved compile lane is now isolated in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/predicate_compile.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/expr/predicate_compile.rs), so planner expression flow owns the branch-heavy compile/readiness path directly instead of keeping it embedded under predicate glue
