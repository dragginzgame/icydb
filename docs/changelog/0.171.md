# IcyDB 0.171 Changelog

## 0.171.4

### 🔧 Changed

- Centralizes mutation durability trace emission behind the executor-owned
  helper used by commit-window paths, including the save-row fast path.

### 🧪 Testing

- Adds a source-audit guard proving mutation durability classification remains
  proof-facing trace evidence rather than diagnostics authority.
- Proves diagnostics continue to project store capabilities without owning
  mutation durability classification.

## 0.171.3

### 🧪 Testing

- Adds mixed durable/live recovery proof for a successful heap-source to
  stable-target relation write followed by heap store reinitialization.
- Proves recovery does not restore the heap source row after reinitialization.
- Proves stable relation targets remain usable after recovery purges volatile
  relation state by deleting the stable target through the public SQL update
  entrypoint.

## 0.171.2

### 🔧 Changed

- Gates durable commit-marker recovery payloads through store recovery
  capability so live-only heap row operations are not persisted as replayable
  recovery work.
- Makes recovery replay skip row operations whose store capability is
  `recovery: none`.
- Makes startup secondary-index rebuild operate only on stores that participate
  in stable commit replay.

### 🧪 Testing

- Adds heap recovery regression coverage proving a persisted heap-only marker
  is cleared without replaying heap data rows or rebuilding heap index entries.
- Extends the source-audit guard to require recovery replay/rebuild code to
  consume recovery capabilities.
- Keeps the existing stable commit/recovery regression suite passing.

## 0.171.1

### 🧪 Testing

- Hardens public write proof for capability-derived mutation commit
  classification.
- Proves a stable write that only carries a non-enforcing heap target reference
  still classifies as durable-only.
- Proves heap-source relation maintenance against a heap target still
  classifies as live-only.
- Proves fail-closed stable-source strong relations to volatile heap targets do
  not emit commit classifications.
- Keeps mixed durable/live classification evidence on the existing heap-source
  to stable-target relation path.

## 0.171.0

### ➕ Added

- Starts the capability-gated recovery and write-semantics line.
- Adds the 0.171 design and boundary inventory for turning the existing
  durable/live/mixed mutation classification into the authoritative write,
  commit, and recovery policy contract.
- Records the current classification owners, affected-store behavior,
  write-stage mapping, failure-injection seams, and source-audit guard
  baseline.

### 🔧 Changed

- Clarifies that 0.171 builds on the 0.170 capability model rather than adding
  a parallel storage-mode or mutation-classification vocabulary.
- Defines the affected store set for durability classification: mutated data
  stores, maintained index stores, relation-maintenance stores when they are
  represented as prepared mutation operations, and schema stores only for
  schema-changing operations.
- Documents that validation-only reads do not make target stores mutation
  commit participants.

### 🧪 Testing

- Ties the .0 inventory to existing durable-only, live-only, and mixed
  classification coverage.
- Identifies existing failure-ordering seams and the remaining gaps that later
  0.171 patches must either test directly or record explicitly.
- Leaves runtime behavior unchanged.
