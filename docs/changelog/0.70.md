# 0.70 Changelog

## 0.70.2 - 2026-04-07 - Flat Classifier Seam and Narrow Composite Migration

### Summary

`0.70.2` does not widen the broader covering policy. It changes where the
already-proven authority decisions live.

The patch finishes centralizing the single-component store-backed secondary
line and then widens exactly one more explicitly proven family:

* single-component secondary reads are now classifier-owned end to end
* one composite synchronized-witness order-only family is now classifier-owned
  too
* stale composite and storage-witness composite families remain on the older
  richer covering profile

That is the important boundary of this patch. The new flat classifier is not a
replacement for every covering authority rule yet; it is the centralized owner
for the admitted synchronized-witness cohorts that can be stated cleanly under
the current invariant.

### Stop Point

The current `0.70.2` state is intentionally dual-mechanism:

* the flat classifier owns single-component reads plus one proven composite
  synchronized-witness family
* the richer covering profile still owns stale/storage-witness composite and
  richer structural cases

This is an explicit boundary, not an accidental half-migration. Later design
work should start from that serialized split instead of assuming the flat
classifier is already the authority owner for all covering routes.

### Changed

- introduced one explicit owner seam for store-backed secondary read authority
  in:
  - `crates/icydb-core/src/db/executor/authority/read.rs`
  - `SecondaryReadAuthorityOwner` now makes the split explicit:
    - `FlatSingleComponentClassifier`
    - `FlatCompositeWitnessValidatedClassifier`
    - `RichCoveringProfile`
- finished routing single-component secondary reads through the shared flat
  classifier for:
  - route promotion
  - `EXPLAIN EXECUTION` authority labels
  - canonical execution-descriptor JSON authority labels
- made the classifier monotonic and explicit:
  - once a route already carries a probe-free mode, the classifier preserves it
    instead of recomputing a weaker answer
- removed the last route/explain-local single-component authority
  reconstruction, so the current single-component scope now flows through:
  - context
  - classifier
  - decision/reason
  - route + explain
- widened exactly one composite family behind the new seam:
  - synchronized composite order-only covering
  - this is the proven `witness_validated` family only
  - it does **not** include stale composite/storage-witness cases
- kept stale composite families explicitly profile-owned:
  - storage-witness composite order-only
  - storage-witness composite leading-component
  - equality-prefix/storage-witness composite cases
  - these still route through the richer explicit covering profile because
    their authority depends on richer witness structure than the current flat
    classifier carries

## 0.70.1 - 2026-04-07 - Authority Observability and Downgrade Locks

### Summary

`0.70.1` keeps the new `0.70.0` validity gate unchanged at execution time, but
it makes that gate much more visible and much harder to regress.

The main theme of this patch is:

* make the final store-backed covering authority decision observable from the
  outside
* keep lifecycle mutation seams narrow and fixture-scoped
* lock the explicit downgrade contract across core, generated SQL parity, and
  PocketIC

This is intentionally **not** a new widening and **not** a new execution-mode
change. It is an observability and regression-safety follow-up for the
`IndexState::Valid` covering rule that `0.70.0` introduced.

### Changed

- centralized the final store-backed secondary covering authority decision in:
  - `crates/icydb-core/src/db/executor/authority/read.rs`
  - this keeps `Valid` gating and witness precedence in one shared profile
    instead of re-deriving the final outcome in multiple route-local helpers
- surfaced runtime index state on inspection paths in:
  - `crates/icydb-core/src/db/schema/format.rs`
  - `crates/icydb-core/src/db/session/mod.rs`
  - `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  - `SHOW INDEXES` now includes `[state=valid|building|dropping]` when a live
    store handle is available
- extended structured diagnostics in:
  - `crates/icydb-core/src/db/diagnostics/mod.rs`
  - `IndexStoreSnapshot` now includes runtime `state`
- surfaced store-backed secondary covering authority on `EXPLAIN EXECUTION` in:
  - `crates/icydb-core/src/db/executor/explain/descriptor/load.rs`
  - the covering load root now exposes:
    - `index_state`
    - `authority_decision`
    - `authority_reason`
  - the current flat authority contract is intentionally explicit rather than
    normalized:
    - `authority_decision`
      - `row_check_required`
      - `witness_validated`
      - `storage_existence_witness`
    - `authority_reason`
      - `index_not_valid`
      - `synchronized_pair_witness`
      - `stale_storage_existence_witness`
      - `authoritative_witness_unavailable`
      - `probe_required`
- locked the same authority metadata on the canonical execution-descriptor JSON
  surface in:
  - `crates/icydb-core/src/db/session/tests/mod.rs`
  - this keeps structured execution explain consumers on the same flat
    authority contract as the text-tree renderer
- tightened the invalid-state fixture seam so it stays lifecycle-scoped instead
  of becoming a reusable public API:
  - hidden core helper remains in:
    - `crates/icydb-core/src/db/session/mod.rs`
  - the public facade re-export was removed from:
    - `crates/icydb/src/db/mod.rs`
  - `sql_parity` now depends on `icydb-core` directly for the narrow fixture
    seam in:
    - `canisters/test/sql_parity/Cargo.toml`
  - the canister test hook remains one-purpose and lifecycle-shaped in:

## 0.70.0 - 2026-04-07 - Explicit Index Validity for Probe-Free Covering

### Summary

`0.70.0` starts the next read-authority line by making one important lifecycle
rule explicit instead of leaving it implicit inside route and storage behavior:
probe-free covering authority is only available when the backing index is
explicitly `Valid`.

This patch does not remove the older fail-closed safety checks and it does not
broaden read authority. It adds a small explicit index lifecycle model:

- `Building`
- `Valid`
- `Dropping`

That model is then used to gate the already-shipped probe-free covering modes:

- `WitnessValidated`
- `StorageExistenceWitness`

So the new rule is intentionally narrow and honest:

> A probe-free covering read may only promote when the backing index is
> `IndexState::Valid`.

Everything else still fails closed back to the older probe-based path.

Just as importantly, `0.70.0` does **not** yet claim that every index-backed
shortcut is now validity-gated. Aggregate `COUNT` / `EXISTS` shortcuts are
still outside this slice because they depend on a different missing-row
correctness model.

### Changed

- added explicit `IndexState` lifecycle state to index storage in:
  - `crates/icydb-core/src/db/index/store.rs`
  - states are:
    - `Building`
    - `Valid`
    - `Dropping`
  - existing index stores initialize as `Valid`
  - invalid states now clear the narrower secondary witness bits so probe-free
    covering authority cannot drift forward on a stale lifecycle state
- exposed the new state through the index/store registry boundary in:
  - `crates/icydb-core/src/db/index/mod.rs`
  - `crates/icydb-core/src/db/mod.rs`
  - `crates/icydb-core/src/db/registry.rs`
- made recovery rebuild explicitly validity-aware in:
  - `crates/icydb-core/src/db/commit/rebuild.rs`
  - `crates/icydb-core/src/db/commit/recovery.rs`
  - touched indexes now move through `Building` during rebuild and are only
    restored to `Valid` after successful rebuild plus integrity validation
- made successful commit-apply restore `Valid` on touched index stores before
  re-enabling the narrower secondary authority bits in:
  - `crates/icydb-core/src/db/executor/mutation/commit_window.rs`
- gated the two probe-free covering promotions on index validity in:
  - `crates/icydb-core/src/db/executor/route/terminal.rs`
  - `WitnessValidated` promotion now requires `index_is_valid()`
  - `StorageExistenceWitness` promotion now requires `index_is_valid()`
  - when an index is `Building` or `Dropping`, the route now stays on
    `RequiresRowPresenceCheck`
- added shared covering-kernel debug assertions in:
  - `crates/icydb-core/src/db/executor/covering.rs`
  - these assertions document and enforce the current `0.70` invariant during
    probe-free covering execution:
    - if the engine is running a probe-free covering mode, the backing index
      must already be `Valid`
- kept the `0.70` scope explicit in:
  - `docs/design/0.70-centralized-read-authority/0.70-design.md`
  - aggregate `COUNT` / `EXISTS` fast paths are still out of scope for this
    validity rule

### Testing

- added session-level route tests in:
  - `crates/icydb-core/src/db/session/tests/mod.rs`
  - these lock:
    - `Building` index states downgrade previously probe-free covering reads
      back to `row_check_required`
    - `Valid` indexes restore the earlier probe-free route after a temporary
      invalid state
    - stale `StorageExistenceWitness` routes stay valid only while the index is
      still `Valid`
    - `Dropping` indexes fail closed back to `row_check_required`
- added direct route-boundary tests in:
  - `crates/icydb-core/src/db/executor/route/tests.rs`
  - these prove the route planner keeps eligible covering plans on
    `RequiresRowPresenceCheck` once the backing index is no longer `Valid`
