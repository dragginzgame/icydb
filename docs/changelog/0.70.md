# 0.70 Changelog

## 0.70.5 - 2026-04-08 - Planner Visibility Gating for Session-Backed Reads

### Summary

`0.70.5` finishes the current `0.70` direction by collapsing index correctness
into planner visibility for the session-backed read path.

The important rule after this patch is:

* planner-visible secondary indexes are the only indexes session-backed query
  planning may use
* if the recovered store is not in the `Valid` lifecycle state, planner sees no
  secondary indexes for that store
* execution no longer carries a separate secondary-read authority decision
  about index correctness

That is the real stop point of this patch. It is not another covering-policy
expansion. It is the boundary-tightening slice that makes the new Option A
direction actually true for the session-owned planner/executor path.

### Changed

- moved secondary-index visibility to the session/planner boundary in:
  - `crates/icydb-core/src/db/session/mod.rs`
  - `DbSession::visible_indexes_for_store_model(...)` now resolves the
    recovered store once and returns:
    - `model.indexes()` when `store.index_state() == IndexState::Valid`
    - `[]` otherwise
- threaded one explicit planner-visible index slice through the structural
  planning path in:
  - `crates/icydb-core/src/db/query/intent/model.rs`
  - `crates/icydb-core/src/db/query/intent/query.rs`
  - `crates/icydb-core/src/db/query/plan/access_planner.rs`
  - `crates/icydb-core/src/db/query/plan/planner/*.rs`
  - load/query planning no longer assumes `model.indexes()` is always visible;
    the visible index set is now caller-supplied at the planning boundary
- switched session-backed explain/SQL dispatch surfaces to the same
  planner-visible index set in:
  - `crates/icydb-core/src/db/session/query.rs`
  - `crates/icydb-core/src/db/session/sql/explain.rs`
  - `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  - `crates/icydb-core/src/db/session/sql/dispatch/lowered.rs`
  - fluent/session SQL `EXPLAIN`, `EXPLAIN EXECUTION`, and structural query
    execution now all replan against the current visible secondary-index slice
    instead of carrying the old authority/snapshot seam forward
- updated load execution-descriptor assembly so explain stays derived from the
  same planner-visible access set in:
  - `crates/icydb-core/src/db/executor/explain/descriptor/load.rs`
  - access-choice explain snapshots are now projected against the same explicit
    visible index list the planner used
- removed more dead Option B compatibility surface in:
  - `crates/icydb-core/src/db/query/plan/access_choice/mod.rs`
  - `crates/icydb-core/src/db/sql/lowering/mod.rs`
  - `crates/icydb-core/src/db/query/intent/query.rs`
  - this patch continues the earlier authority removal by deleting now-dead
    helpers that still assumed model-only index visibility or authority-era
    explain fallback paths

### Behavioral Result

For session-backed reads, the current rule is now simple:

* `Valid` secondary index:
  - planner may choose the index
  - covering routes may still emit `existing_row_mode=planner_proven`
* non-`Valid` secondary index:
  - planner does not see the index at all
  - route planning falls back to whatever non-index path remains correct
  - there is no executor-side downgrade from “chosen index” to
    “row_check_required”

The most visible user-facing consequence is the former building-index secondary
covering explain path:

* before this patch’s final rewrite, those tests still encoded the old
  “CoveringRead + row_check_required” fallback
* after this patch, the same building-index query explains as:
  - `FullScan`
  - `OrderByMaterializedSort`
  - no `CoveringRead`
  - no `existing_row_mode`
  - no `authority_decision` / `authority_reason` / `index_state`

That is the intended Option A contract: invisible index, not visible-but-softly
downgraded index.

runtime lifecycle visibility, because those call sites do not own a recovered
store handle. The shipped invariant here is narrower and concrete:

* session-backed planning now enforces `visible index == correct index`
* execution trusts the resulting plan contract

## 0.70.4 - 2026-04-08 - Authority Snapshot Boundary and Aggregate Query Coverage

It does two narrower follow-ups instead:

* it separates secondary-read authority resolution from live store access by

* it broadens aggregate query coverage and PocketIC measurement on the typed
  aggregate lane without widening any aggregate authority shortcut policy

That is the real boundary of this patch. `0.70.4` is a seam-tightening and
coverage-baseline slice, not another authority-policy widening.

## 0.70.3 - 2026-04-08 - Canonical Resolved Profile Cleanup

### Summary

`0.70.3` does not widen the broader covering policy or change query results.
It finishes the authority cleanup that `0.70.2` started by making the
resolved profile the only production behavior source.

The final state of this follow-up is:

* one canonical resolved profile is now the only production source of
  secondary-read authority behavior
* flat authority labels are now a derived, optional projection from that
  resolved profile
* the already-admitted flat-projection families from `0.70.2` stay intact
* stale composite and storage-witness composite families still remain
  resolved-profile-only and still do not pick up the flat labels by
  resemblance

That is the important boundary of this patch. The flat classifier is now a
compatibility projection layered on top of the canonical resolved profile,
not a second behavior source.

### Stop Point

The current `0.70.3` state is intentionally narrow but cleaner than the
earlier transition slices:

* the production authority object is always one
  `ResolvedSecondaryReadAuthorityProfile`
* that resolved profile always carries:
  * `index_state`
  * `existing_row_mode`
  * `flat_classifier: Option<...>`
* the flat projection currently exists only for the cohorts already admitted in
  `0.70.2`:
  * the centralized single-component secondary line
  * one explicitly proven synchronized-witness composite order-only family
* stale composite/storage-witness and broader richer structural cases remain
  resolved-profile-only

This is an explicit boundary, not an accidental half-migration. Later design
work should start from this serialized state instead of assuming the flat
projection already covers all covering routes.

### Changed

- kept rich behavior primary and flat projection secondary in:
  - `crates/icydb-core/src/db/executor/authority/read.rs`
  - the canonical resolved profile now carries:
    - `index_state`
    - `existing_row_mode`
    - `flat_classifier: Option<...>`
  - probe-free `existing_row_mode` is resolved first
  - flat classifier projection is derived only afterward when the profile
    clears the explicit admissibility gate
- finished routing route promotion and store-backed explain through that
  resolved profile for:
  - route promotion
  - `EXPLAIN EXECUTION` authority labels
  - canonical execution-descriptor JSON authority labels
- made the flat projection explicitly monotonic and lossy:
  - once a resolved profile already carries a probe-free mode, the flat
    projection must preserve that mode rather than recomputing a weaker answer
  - the flat classifier remains a lossy summary of admitted fast-path profiles,
    not a new behavior source
- removed the last owner-era production vocabulary and the internal
  second-profile helper:
  - `SecondaryReadAuthorityOwner` is gone from production
  - the internal `SecondaryCoveringAuthorityProfile` helper struct is gone
  - one direct `secondary_covering_probe_free_mode_for_store(...)` helper now
    feeds the resolved profile instead
- kept stale composite families explicitly resolved-profile-only:
  - storage-witness composite order-only
  - storage-witness composite leading-component
  - equality-prefix/storage-witness composite cases
  - these still keep their covering-specific `existing_row_mode` because their
    authority depends on richer witness structure than the current flat
    projection carries
- kept the external inspection split honest:
  - resolved profiles with an admitted flat projection emit
    `authority_decision` + `authority_reason`
  - resolved-profile-only routes keep `existing_row_mode` without inheriting
    those flat labels

### Testing

  - `crates/icydb-core/src/db/executor/authority/read.rs`
- added route seam locks in:
  - `crates/icydb-core/src/db/executor/route/tests.rs`
  - these prove:
    - single-component routes expose a flat projection
    - synchronized composite order-only exposes a flat projection only when
      the stronger synchronized witness is present
    - stale composite/storage-witness does **not** drift onto the flat

- kept the external authority metadata locks in:
  - `crates/icydb-core/src/db/session/tests/mod.rs`
  - `canisters/test/sql_parity/src/tests.rs`
  - `testing/pocket-ic/tests/sql_canister.rs`
  - these prove:
    - classifier-admitted routes still expose flat authority labels

## 0.70.0 - 2026-04-07 - Explicit Index Validity for Probe-Free Covering

### Summary

`0.70.0` starts the next read-authority line by making one important lifecycle
rule explicit instead of leaving it implicit inside route and storage behavior:
probe-free covering authority is only available when the backing index is
explicitly `Valid`.

This patch does not remove the older fail-closed safety checks and it does not
broaden read authority. It adds a small explicit index lifecycle model:

- `Building`
- `Valid`
- `Dropping`

That model is then used to gate the already-shipped probe-free covering modes:

- `WitnessValidated`
- `StorageExistenceWitness`

So the new rule is intentionally narrow and honest:

> A probe-free covering read may only promote when the backing index is
> `IndexState::Valid`.

Everything else still fails closed back to the older probe-based path.

Just as importantly, `0.70.0` does **not** yet claim that every index-backed
shortcut is now validity-gated. Aggregate `COUNT` / `EXISTS` shortcuts are
still outside this slice because they depend on a different missing-row
correctness model.

### Changed

- added explicit `IndexState` lifecycle state to index storage in:
  - `crates/icydb-core/src/db/index/store.rs`
  - states are:
    - `Building`
    - `Valid`
    - `Dropping`
  - existing index stores initialize as `Valid`
  - invalid states now clear the narrower secondary witness bits so probe-free
    covering authority cannot drift forward on a stale lifecycle state
- exposed the new state through the index/store registry boundary in:
  - `crates/icydb-core/src/db/index/mod.rs`
  - `crates/icydb-core/src/db/mod.rs`
  - `crates/icydb-core/src/db/registry.rs`
- made recovery rebuild explicitly validity-aware in:
  - `crates/icydb-core/src/db/commit/rebuild.rs`
  - `crates/icydb-core/src/db/commit/recovery.rs`
