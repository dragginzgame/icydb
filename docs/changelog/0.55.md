# IcyDB 0.55

This document aggregates detailed release notes for the `0.55.x` minor line.

## 0.55.7 - 2026-03-15 - Legacy and Alias Surface Cleanup

### Summary

`0.55.7` removes remaining alias-style compatibility surfaces and stale
legacy/shim wording in runtime-adjacent tests/docs. The patch keeps persisted
decode compatibility behavior unchanged by explicitly retaining the row and
commit-marker fallback paths.

### Changed

- Removed API aliases that only preserved transitional naming:
  - removed PocketIC staging alias `stage_sql_test_canister_for_dfx(...)`;
    staging now uses `stage_canister_for_dfx(...)` directly.
  - removed `Date::get()` alias and converged date-byte consumers on
    `Date::as_days_since_epoch()`.
- Cleaned stale legacy/shim/backwards-compatible wording from:
  - SQL dispatch generation guard assertions.
  - index key-item parity docs/messages.
  - executor structural guard assertion text.
  - session SQL projection docs.
  - UI architecture guard notes.
  - testing guidance that still referenced `LegacyTestEntityModel` (replaced
    with `entity_model_from_static` guidance).
- Kept persisted-format compatibility behavior unchanged:
  - retained legacy row decode fallback in `db::codec`.
  - retained legacy commit-control/commit-marker decode fallbacks in
    `db::commit::store`.

### Validation

- `cargo check -p icydb-core -p icydb-schema -p icydb-testing-integration -p canister_sql_test`
- `rg -n -i "\blegacy\b|\bshim\b|backward.?compatible|backwards.?compatible" crates canisters testing`

## 0.55.6 - 2026-03-15 - Load Pipeline Containment Follow-Up

### Summary

`0.55.6` continues executor-load containment by splitting remaining
post-access/orchestrator/operator hubs into narrower owner seams. This patch
keeps behavior stable while reducing mixed-concern module pressure and preserving
stage-order and layer-authority invariants.

### Changed

- Continued post-access containment decomposition:
  - split post-access into explicit contract/coordinator/terminal seams under
    `db::executor::pipeline::operators::post_access::*`.
  - moved ordering/paging/delete-limit terminal operators under one dedicated
    terminal boundary and kept coordinator runtime focused on phase sequencing.
- Continued orchestrator containment decomposition:
  - converted orchestrator hubs to directory modules and split ownership into
    `contracts`, `dispatch`, `payload`, `state`, and `strategy` seams under
    `db::executor::pipeline::orchestrator::*`.
  - removed prior flat orchestrator module files that mixed stage envelope,
    payload, and dispatch responsibilities.
- Continued operator-level seam cleanup:
  - split reducer ownership into
    `operators::reducer::{contracts, runner, aggregate}`.
  - converted DISTINCT operator wiring to a directory module with separate
    runtime and test surfaces.
  - converted terminal row-collector path to
    `operators::terminal::{collector, runtime}` with entrypoint wiring kept in
    `terminal::mod`.
- Kept layer-authority invariant gates aligned after refactor:
  - updated post-access coordinator imports to stay compliant with the
    pipeline-planner import invariant scan.

### Validation

- `cargo check -p icydb-core`
- `cargo test -p icydb-core load_execute_stage_order_matches_linear_contract -- --nocapture`
- `cargo test -p icydb-core load_pipeline_optional_stage_slots_stay_within_soft_delta -- --nocapture`
- `make check-invariants`

## 0.55.5 - 2026-03-15 - Continuation Envelope Consolidation and Audit Publication

### Summary

`0.55.5` follows the `0.55.4` semantic-authority work by moving continuation
envelope helpers to one index-owned boundary, tightening invariant leak checks
for generic helper signatures, hardening PocketIC integration teardown behavior
so primary failures are preserved, and publishing the same-day crosscutting
audit report set.

### Changed

- Consolidated continuation-envelope helper ownership under the index boundary:
  - moved strict continuation advancement, anchor-envelope containment checks,
    and resume-bound rewriting into `db::index::envelope`.
  - removed `db::cursor::envelope`; cursor APIs now consume these helpers via
    index-owned exports.
  - expanded `db::index::envelope` test coverage to keep continuation and
    envelope semantics locked under one owner module.
- Tightened layer-authority invariant scanning for generic helper declarations:
  - `scripts/ci/check-layer-authority-invariants.sh` now matches helper
    signatures like `fn helper<T>(...)` when checking for ownership leaks.
  - this prevents generic helper declarations from bypassing canonical-owner
    detection rules.
- Hardened PocketIC integration test teardown failure handling:
  - added `run_with_pocket_ic(...)` in
    `testing/pocket-ic/tests/sql_canister.rs`.
  - test execution now rethrows the primary test panic and suppresses secondary
    teardown panics so root failures remain actionable.
- Published recurring crosscutting audit artifacts for 2026-03-15:
  - added canonical semantic-authority and DRY-consolidation reports with
    helper evidence tables under `docs/audits/reports/2026-03/2026-03-15/`.
  - added `summary.md` for the subset run (`complexity-accretion`,
    `canonical-semantic-authority`, `dry-consolidation`) with reported risks
    of `5.0/10`, `3.9/10`, and `5.0/10` respectively.

### Validation

## 0.55.4 - 2026-03-15 - Predicate Semantic-Authority Audit and PocketIC Test Wiring

### Summary

`0.55.4` applies the predicate-focused semantic-authority audit follow-up by
moving structural OR-equality canonicalization and identifier-tree adaptation
to predicate-owned boundaries, then hardens PocketIC integration test execution
for local `make test` and CI by requiring explicit server-binary resolution.

### Changed

- Centralized canonical OR-equality rewriting under predicate ownership:
  - `predicate::normalize` now collapses same-field `Eq` disjunctions into one
    canonical `In` compare when coercion/literal constraints are safe.
  - planner-local OR->IN rewrite logic was removed from
    `query::plan::planner::predicate`; planning now consumes canonical predicate
    models from the predicate authority boundary.
  - added parity coverage proving canonical equivalence across predicate digest
    hashing plus query fingerprint/continuation-signature generation for
    same-field `Eq OR Eq` vs `In` forms.
- Added predicate-owned identifier adaptation helper and delegated SQL lowering:
  - introduced `db::predicate::identifiers::rewrite_field_identifiers(...)`.
  - SQL lowering now delegates predicate-field traversal to that helper instead
    of owning local recursive predicate traversal helpers.
  - added nested qualified-SQL predicate parity coverage against unqualified
    fluent query intent.
- Reduced duplicate semantic helpers outside owner boundaries:
  - EXPLAIN planning now consumes canonical predicate payloads directly from
    planner-owned scalar semantics (no extra EXPLAIN-side normalize pass).
  - access-choice candidate evaluation now uses shared planner compatibility
    helpers (`index_literal_matches_schema`, `IndexModel::is_field_indexable`)
    instead of local duplicate compatibility/operator helpers.
  - structural-guard tests were expanded to lock these ownership boundaries and
    reject regressions that reintroduce duplicate canonicalization helpers.
- Hardened PocketIC integration test setup for local and CI runs:
  - `testing/pocket-ic/tests/sql_canister.rs` now builds PocketIC with explicit
    `POCKET_IC_BIN` + `with_application_subnet()` wiring to avoid implicit
    server-binary downloads and invalid empty-subnet builder configs.
  - added `scripts/ci/ensure-pocket-ic-bin.sh` to resolve/reuse/download the
    matching PocketIC server binary version.
  - `make test-unit` now exports `POCKET_IC_BIN` through the resolver script,
    and CI sets `POCKET_IC_BIN` before running the `sql_canister` integration
    test target.
  - SQL test canister build profile selection now recognizes
    `SQL_TEST_WASM_PROFILE=wasm-release` and defaults to `wasm-release` for
    `DFX_NETWORK=mainnet|staging|ic`.

## 0.55.2 - 2026-03-15 - Expression-Index Eligibility and Route-Parity Expansion

### Summary

`0.55.2` expands expression-index eligibility and route-parity coverage for the
supported `LOWER(field)` lookup subset while tightening fail-closed boundaries
for unsupported expression kinds and range-family access paths.

### Changed

- Introduced one canonical expression-key derivation authority:
  - moved expression key-value transforms into
    `db::index::derive_index_expression_value`.
  - index-key derivation and planner key-item lookup lowering now share this
    transform boundary.
- Made expression lookup capability explicit in the index model:
  - added `IndexExpression::supports_text_casefold_lookup`.
  - current matrix is explicit: only `LOWER(field)` supports `TextCasefold`
    lookup lowering for `Eq`/`In` in this release.
- Hardened planner/access-choice fail-closed behavior for unsupported shapes:
  - unsupported expression lookup kinds (for example `UPPER(field)` under
    `TextCasefold`) are rejected deterministically.
  - expression-key indexes are explicitly excluded from range-family
    (`Gt`/`StartsWith`) eligibility in this slice.
- Added S5 parity coverage across planner/access-choice/runtime for supported
  expression lookups:
  - `LOWER(email)` + `TextCasefold` `Eq` lowers to deterministic
    `IndexPrefix`/`IndexPrefixScan`.
  - `LOWER(email)` + `TextCasefold` `In` lowers to canonical deduplicated
    `IndexMultiLookup`/`IndexMultiLookup`.
- Improved local SQL harness development ergonomics:
  - moved SQL helper script to `scripts/dev/sql.sh` with explicit
    `--deploy`/`--reset`/`--init` control flow.
  - updated README and fixture seeding module layout (`seed::base`,
    `seed::rpg`) for clearer local harness usage.

## 0.55.0 - 2026-03-15 - Expression Index Foundations

### Summary

`0.55.0` establishes canonical expression-index foundations and closes the main
string-authority drift seams identified after the filtered-index line. This
release adds typed key-item metadata, canonical expression-key derivation and
eligibility rules, and parity checks across planner, EXPLAIN/access-choice, and
mutation/replay paths.

### Changed

- Added canonical index key-item metadata authority:
  - introduced typed key-item model (`Field` / `Expression`) in schema/runtime
    index contracts.
  - introduced typed canonical expression metadata (`IndexExpression`) instead
    of raw expression text payloads.
  - schema/runtime/facade exports now expose canonical expression-key model
    types.
- Added schema-time expression key-item validation for supported deterministic
  subset:
  - text transforms: `LOWER`, `UPPER`, `TRIM`, `LOWER(TRIM(...))`
  - temporal transforms: `DATE`, `YEAR`, `MONTH`, `DAY`
  - field type constraints are enforced (text vs date/timestamp inputs).
- Added mutation-path expression key derivation and unique validation parity:
  - index key build now materializes canonical derived values for expression key
    items.
  - unique conflict checks now validate against canonical derived key
    components.
  - conditional + expression composition is covered: predicate-false rows bypass
    membership/unique checks, predicate-true activation enforces canonical
    uniqueness.
  - live-vs-replay conflict classification parity is locked for expression
    unique conflicts.
- Added planner/access eligibility support for the smallest useful expression
  subset:
  - planner prefix and multi-lookup eligibility now evaluates canonical key
    items rather than field-only assumptions.
  - `TextCasefold` Eq/In (and same-field OR-equality rewrite) can use
    `LOWER(field)` expression indexes via canonical lookup-value lowering.
  - filtered + expression composition is covered: filtered predicate implication
    remains required before expression-index eligibility.
- Added EXPLAIN access-choice parity with planner semantics:
  - access-choice candidate/rejection logic now reasons over canonical key
    items.
  - planner and access-choice share one key-item matcher/lowering authority to
    avoid duplicate semantic owners.
- Hardened semantic fingerprinting and canonicalization boundaries:
  - schema fingerprint now includes canonical index key-item semantics
    (including expression kind + field) and filtered predicate semantics.
  - SQL dispatch wrappers and identifier/projection-label side paths were moved
    onto canonical lowering/normalization boundaries so semantic routing is not
    owned by ad hoc string scans.
