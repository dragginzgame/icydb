# IcyDB 0.55

This document aggregates detailed release notes for the `0.55.x` minor line.

## 0.55.3 - 2026-03-15 - Expression-Index Closure and Fail-Closed Parity

### Summary

`0.55.3` closes the `0.55` expression-index line by locking unsupported
expression lookup behavior to fail closed across planner, access-choice,
runtime, and startup recovery paths. This patch also introduces a dedicated
recurring crosscutting audit track for canonical semantic authority.

### Changed

- Added unsupported-expression fail-closed parity coverage for
  `UPPER(email)` indexes under `TextCasefold` `Eq`/`In` predicates:
  - planner access remains `FullScan`.
  - access-choice diagnostics remain non-index (`full_scan`,
    `non_index_access`).
  - runtime route selection remains full-scan aligned while preserving
    canonical casefold filtering semantics.
- Added startup recovery rebuild coverage for unsupported expression lookup
  kinds:
  - startup recovery now proves stale `UPPER(email)` expression index entries
    are purged and canonical expression keys are rebuilt from authoritative row
    data.
- Added recurring crosscutting canonical semantic-authority audit ownership:
  - introduced
    `docs/audits/recurring/crosscutting/crosscutting-canonical-semantic-authority.md`.
  - updated recurring-audit run-order and summary-order governance to include
    canonical semantic authority checks in the fixed crosscutting sequence.
  - added runtime metrics artifact for the 2026-03-15 crosscutting run.
- Updated `0.55` status tracking to mark S6 (unsupported-expression fail-closed
  parity) complete.

## 0.55.2 - 2026-03-15 - Expression-Index Eligibility and Route-Parity Expansion

### Summary

`0.55.2` expands expression-index eligibility and route-parity coverage for the
supported `LOWER(field)` lookup subset while tightening fail-closed boundaries
for unsupported expression kinds and range-family access paths.

### Changed

- Introduced one canonical expression-key derivation authority:
  - moved expression key-value transforms into
    `db::index::derive_index_expression_value`.
  - index-key derivation and planner key-item lookup lowering now share this
    transform boundary.
- Made expression lookup capability explicit in the index model:
  - added `IndexExpression::supports_text_casefold_lookup`.
  - current matrix is explicit: only `LOWER(field)` supports `TextCasefold`
    lookup lowering for `Eq`/`In` in this release.
- Hardened planner/access-choice fail-closed behavior for unsupported shapes:
  - unsupported expression lookup kinds (for example `UPPER(field)` under
    `TextCasefold`) are rejected deterministically.
  - expression-key indexes are explicitly excluded from range-family
    (`Gt`/`StartsWith`) eligibility in this slice.
- Added S5 parity coverage across planner/access-choice/runtime for supported
  expression lookups:
  - `LOWER(email)` + `TextCasefold` `Eq` lowers to deterministic
    `IndexPrefix`/`IndexPrefixScan`.
  - `LOWER(email)` + `TextCasefold` `In` lowers to canonical deduplicated
    `IndexMultiLookup`/`IndexMultiLookup`.
- Improved local SQL harness development ergonomics:
  - moved SQL helper script to `scripts/dev/sql.sh` with explicit
    `--deploy`/`--reset`/`--init` control flow.
  - updated README and fixture seeding module layout (`seed::base`,
    `seed::rpg`) for clearer local harness usage.

## 0.55.1 - 2026-03-15 - Expression-Index Follow-Up Hardening

### Summary

`0.55.1` is the post-publish hardening patch for the expression-index line. It
adds the highest-priority parity locks for filtered+expression composition and
replay/integrity behavior so planner eligibility, access-choice/EXPLAIN, and
mutation/recovery semantics stay aligned.

### Changed

- Added planner composition coverage for filtered + expression indexes:
  - expression-index eligibility remains gated by filtered-index predicate
    implication.
  - covered `TextCasefold` + `LOWER(email)` eligibility under
    predicate-missing vs predicate-implied query shapes.
- Added mutation-path conditional + expression unique coverage:
  - predicate-false rows bypass conditional unique checks.
  - predicate-false -> predicate-true activation enforces canonical
    `LOWER(email)` uniqueness.
- Added replay/integrity parity coverage for expression-unique conflicts:
  - live save-path classification and replay-path classification are now locked
    for casefold unique conflicts.
- Updated `0.55` status tracking to record these follow-up parity/composition
  checks under the expression-index slice tracker.

## 0.55.0 - 2026-03-15 - Expression Index Foundations

### Summary

`0.55.0` establishes canonical expression-index foundations and closes the main
string-authority drift seams identified after the filtered-index line. This
release adds typed key-item metadata, canonical expression-key derivation and
eligibility rules, and parity checks across planner, EXPLAIN/access-choice, and
mutation/replay paths.

### Changed

- Added canonical index key-item metadata authority:
  - introduced typed key-item model (`Field` / `Expression`) in schema/runtime
    index contracts.
  - introduced typed canonical expression metadata (`IndexExpression`) instead
    of raw expression text payloads.
  - schema/runtime/facade exports now expose canonical expression-key model
    types.
- Added schema-time expression key-item validation for supported deterministic
  subset:
  - text transforms: `LOWER`, `UPPER`, `TRIM`, `LOWER(TRIM(...))`
  - temporal transforms: `DATE`, `YEAR`, `MONTH`, `DAY`
  - field type constraints are enforced (text vs date/timestamp inputs).
- Added mutation-path expression key derivation and unique validation parity:
  - index key build now materializes canonical derived values for expression key
    items.
  - unique conflict checks now validate against canonical derived key
    components.
  - conditional + expression composition is covered: predicate-false rows bypass
    membership/unique checks, predicate-true activation enforces canonical
    uniqueness.
  - live-vs-replay conflict classification parity is locked for expression
    unique conflicts.
- Added planner/access eligibility support for the smallest useful expression
  subset:
  - planner prefix and multi-lookup eligibility now evaluates canonical key
    items rather than field-only assumptions.
  - `TextCasefold` Eq/In (and same-field OR-equality rewrite) can use
    `LOWER(field)` expression indexes via canonical lookup-value lowering.
  - filtered + expression composition is covered: filtered predicate implication
    remains required before expression-index eligibility.
- Added EXPLAIN access-choice parity with planner semantics:
  - access-choice candidate/rejection logic now reasons over canonical key
    items.
  - planner and access-choice share one key-item matcher/lowering authority to
    avoid duplicate semantic owners.
- Hardened semantic fingerprinting and canonicalization boundaries:
  - schema fingerprint now includes canonical index key-item semantics
    (including expression kind + field) and filtered predicate semantics.
  - SQL dispatch wrappers and identifier/projection-label side paths were moved
    onto canonical lowering/normalization boundaries so semantic routing is not
    owned by ad hoc string scans.
