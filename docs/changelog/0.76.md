## 0.76.10 - 2026-04-12 - Explicit Generator Family Widening

This patch does not reopen generic SQL defaults. It keeps reduced SQL
insert omission schema-first and explicit by widening `generated(insert = "...")`
only to one more admitted generator family, while also finishing the managed
timestamp ownership cleanup so typed saves, structural writes, and typed-dispatch
SQL writes all converge on one save-preflight execution path.

What changed:

* widened the schema-owned insert-generation allowlist in:
  * `crates/icydb-schema-derive/src/node/field.rs`
  * `crates/icydb-core/src/model/field.rs`
  * `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  so the current admitted `generated(insert = "...")` cohort is now:
  * `Ulid::generate` on single-value primitive `Ulid`
  * `Timestamp::now` on single-value primitive `Timestamp`
* kept generation explicit and fail-closed:
  * no generic reuse of `default = ...`
  * no arbitrary generator expressions
  * no widening to optional, repeated, relation, or custom `is = ...` fields
* added schema conflict validation in:
  * `crates/icydb-schema-derive/src/node/field.rs`
  so the derive layer now rejects:
  * `generated(insert = ...)` combined with auto-managed write fields
  * `generated(insert = ...)` whose primitive field kind does not match the generator family
  * `generated(insert = ...)` plus `default = ...` when the two paths do not use the same supported generator contract
* widened reduced SQL insert synthesis accordingly in:
  * `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  so typed-dispatch inserts can now synthesize generated non-primary `Timestamp`
  fields on the same explicit field-policy path already used for `Ulid`
* finished the managed timestamp ownership move by keeping derive as the

## 0.76.9 - 2026-04-12 - Schema-Owned Insert Generation

This patch moves reduced SQL insert omission onto one explicit schema/runtime
contract instead of leaving it as a write-lane `Ulid` type special case.
Fields can now opt into insert-time synthesis through a separate schema
`generated(insert = "...")` clause, while ordinary schema `default = ...` expressions stay a
typed Rust/entity-construction concern and are no longer something reduced SQL
may silently consume by omission. The same patch also moves auto-managed
timestamp ownership onto explicit runtime field metadata, so derive-injected
`created_at` and `updated_at` stay boilerplate-free without leaving the SQL
write lane dependent on literal field names.

What changed:

* added one separate schema field-generation contract in:
  * `crates/icydb-schema/src/node/field.rs`
  * `crates/icydb-schema-derive/src/node/field.rs`
  so schema fields can now carry:
  * `generated(insert = "...")`
  separately from:
  * `default = ...`
* kept the first admitted `generated(insert = "...")` cohort intentionally narrow:
  * `generated(insert = "...")` currently validates only on single-value primitive `Ulid`
    fields
  * unsupported uses fail at macro/derive validation instead of degrading into
    runtime guesswork
* emitted the schema-owned insert-generation metadata into runtime field models
  in:
  * `crates/icydb-core/src/model/field.rs`
  * `crates/icydb-schema-derive/src/imp/inherent/model.rs`
  so reduced SQL write execution now consumes one runtime field contract
  instead of asking whether a field kind happens to be `Ulid`
* replaced the old reduced SQL insert `Ulid` special case in:
  * `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  with:
  * field-by-field insert-generation lookup from `FieldModel`
  * insert-time synthesis for omitted schema-generated fields
  * explicit rejection when omitted insert columns are not schema-generated
* moved derive-owned managed timestamp behavior onto explicit runtime field
  metadata in:
  * `crates/icydb-schema/src/node/field.rs`
  * `crates/icydb-schema-derive/src/node/field.rs`
  * `crates/icydb-core/src/model/field.rs`
  so auto-added `created_at` / `updated_at` fields now emit one
  schema/runtime-managed write contract instead of forcing SQL writes to infer
  ownership from literal field names
* replaced the old SQL write name-based timestamp checks in:
  * `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  with:
  * metadata-driven insert synthesis for managed fields
  * metadata-driven `updated_at` refresh on update
  * explicit rejection when SQL tries to author managed fields directly
* widened the admitted generated insert behavior accordingly:
  * schema-generated non-primary fields can now be omitted and synthesized too
  * omitted-column-list positional insert now admits omission only when tuple
    width matches the non-generated field set
* tightened reduced SQL insert omission policy:
  * `default = ...` alone no longer implies reduced SQL omission rights
  * omitted non-generated fields now fail closed with an explicit boundary
    message

Coverage added/updated:

* derive-node validation coverage in:
  * `crates/icydb-schema-derive/src/node/field.rs`
  now covers:
  * accepted `generated(insert = "Ulid::generate")` on primitive `Ulid`
  * rejected `generated(insert = "Ulid::generate")` on non-`Ulid` fields
* public macro compile-fail coverage in:
  * `testing/macro-tests/tests/ui/generated_requires_ulid.rs`
  * `testing/macro-tests/tests/ui/generated_requires_ulid.stderr`
  now locks the outward macro error for unsupported `generated(insert = "...")` usage
* session SQL write coverage in:
  * `crates/icydb-core/src/db/session/tests/mod.rs`
  * `crates/icydb-core/src/db/session/tests/sql_write.rs`
  now covers:
  * schema-generated primary-key omission on typed-dispatch insert
  * schema-generated non-primary-field omission on typed-dispatch insert
  * explicit rejection for omitted non-generated insert fields

Architectural notes:

  * reduced SQL insert omission now follows field metadata emitted by the

  * auto-managed timestamp writes now follow field metadata emitted by the

* `default` and `generated(insert = "...")` are now different contracts:

  * `generated(insert = "...")` is about insert-time synthesis at the write boundary

  so trailing `RETURNING` clauses now map to one stable unsupported-feature
  label instead of falling through as generic invalid syntax
* kept the reduced SQL write-result contract intentionally narrow:

  * no `UPDATE ... RETURNING`
  * no `DELETE ... RETURNING`
* kept the generated canister SQL surface unchanged too:
  * `sql_dispatch::query(...)` remains query-only
  * `RETURNING` does not open a back door into write-result semantics there

## 0.76.7 - 2026-04-12 - Narrow SQL Insert-Select Admission

This patch keeps the `0.76` line on the same session-owned typed SQL write
boundary and admits the narrowest useful `INSERT ... SELECT` shape without
moving SQL writes into planner-owned mutation execution. Typed SQL dispatch can
now copy rows through `INSERT ... SELECT` when the source stays on the same
scalar entity lane, reuses deterministic primary-key ordering, and projects
either plain fields or one admitted scalar computed text projection before
structural insert replay.

What changed:

* widened parsed `INSERT` state in:
  * `crates/icydb-core/src/db/sql/parser/model.rs`
  so `SqlInsertStatement` now carries one explicit source contract instead of
  only literal tuple lists:
  * `Values(...)`
  * `Select(...)`
* widened `INSERT` parsing in:
  * `crates/icydb-core/src/db/sql/parser/statement.rs`
  so reduced SQL now accepts:
  * `INSERT INTO entity (field, ...) SELECT field, ... FROM entity ...`
* kept the new source shape intentionally narrow:
  * same-entity typed-dispatch lane only
  * scalar `SELECT` source only
  * field-only source projection, or one admitted scalar computed text
    projection
  * no grouped source query
  * no aggregate source query
* implemented the source-query replay at the session SQL write boundary in:
  * `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
  by:
  * validating the admitted `SELECT` source shape locally
  * forcing deterministic primary-key-backed source ordering when needed
  * lowering and executing field-only sources through the existing scalar SQL
    projection path
  * routing admitted scalar computed sources through the existing
    session-owned computed projection lane
  * validating projection width against the resolved target insert columns
  * replaying the existing structural insert path per projected source row
* kept generated-key insert support intact for the new copy-insert path:
  * `Ulid`-key entities can still omit the generated primary key from the
    target insert column list
  * the write lane still synthesizes that key locally before structural replay
* kept the existing narrowed write ownership unchanged:
  * writes still stay on typed dispatch
  * no mutation lowering was added to `query_from_sql(...)`
  * no generic SQL write planner lane was introduced

Coverage added/updated:

* parser coverage in:

  now covers:
  * field-only `INSERT ... SELECT` parsing
  * computed scalar `INSERT ... SELECT` parsing

  * `crates/icydb-core/src/db/session/tests/sql_write.rs`
  now covers:
  * generated-key same-entity copy-insert through `INSERT ... SELECT`
  * generated-key same-entity computed copy-insert through `INSERT ... SELECT`
  * explicit rejection when the source projection widens into aggregates
* PocketIC SQL harness coverage in:

  now includes:
  * one typed-dispatch generated-key `Customer` field-only insert-select probe

  * one typed-dispatch generated-key `Customer` computed insert-select probe

Architectural notes:

* this patch still keeps reduced SQL writes session-owned rather than lowering
  them into planner-owned mutation execution
* the new source `SELECT` lane is reused, not reinvented:
  * prepare/lower one scalar field-only `SELECT`
  * execute the existing structural SQL projection path for field-only sources
  * execute the existing session-owned computed projection path for admitted
    scalar computed sources
  * replay the existing structural insert mutation path
* the important boundary is unchanged:
  * SQL reads and SQL writes still meet at the session layer
  * executor/runtime contracts remain structural and alias-neutral

Explicit current boundary:

* `INSERT ... SELECT` is admitted only on the same typed-dispatch entity lane
* source projection must remain field-only or one admitted scalar computed
  text projection
* grouped, aggregate, and broader unsupported computed source projections
  remain fail-closed for insert-select
* writes still stay on typed dispatch rather than `query_from_sql(...)` or
  `execute_sql(...)`
* the generated `sql_dispatch::query(...)` canister surface remains query-only
  and still rejects SQL write lanes, including `INSERT ... SELECT`

## 0.76.5 - 2026-04-12 - Predicate-Based SQL Update Widening

This patch keeps the `0.76` line on the same reduced SQL write boundary and
widened the next admitted write shapes without changing planner/runtime
ownership. Typed SQL dispatch `UPDATE` is no longer restricted to
`WHERE <primary_key> = literal`; it now admits the existing reduced predicate
surface on the typed dispatch lane, accepts single-table aliases on that
narrowed `UPDATE` surface, and locks the still-blocked write tails behind
stable explicit unsupported-feature errors.

What changed:

* widened typed SQL dispatch `UPDATE` from one primary-key-addressed row to
  one deterministic predicate-selected row set on:
  * `execute_sql_dispatch::<E>(...)`
* kept the parser contract structurally unchanged:
  * `SqlUpdateStatement` already carried one optional reduced `Predicate`
  * the widening still happens at the session SQL write boundary, not in
    planner-owned mutation lowering
* admitted single-table aliases on the narrowed `UPDATE` surface:
  * `UPDATE entity alias SET ... WHERE alias.field ...`
  * `UPDATE entity AS alias SET ... WHERE alias.field ...`
