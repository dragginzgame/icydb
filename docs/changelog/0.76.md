# 0.76 — SQL Surface Completion

## 0.76.3 - 2026-04-11 - Reduced SQL Write Surface

This patch widens the reduced `0.76` SQL surface in one narrow place: typed
SQL dispatch now admits a first write lane for explicit-column `INSERT` and
primary-key-targeted `UPDATE`. It does not widen generic planner/runtime write
ownership. The implementation stays on the existing typed mutation path and
keeps broader SQL write shapes fail-closed.

What changed:

* admitted parser support for:
  * `INSERT INTO entity (field, ...) VALUES (...)`
  * `UPDATE entity SET field = literal [, ...] WHERE <primary_key> = literal`
* wired those statements through the typed SQL dispatch boundary in:
  * `crates/icydb-core/src/db/session/sql/dispatch/mod.rs`
* kept the admitted write surface intentionally narrow:
  * `INSERT` still requires an explicit column list
  * `INSERT` still requires the primary-key column to be present
  * `UPDATE` still requires `WHERE pk = literal`
  * primary-key mutation remains rejected
* normalized SQL numeric write literals against target field kinds before patch
  assembly so the typed mutation path receives the same semantic value shape it
  already expects:
  * non-negative `Int` SQL literals can normalize to `Uint` fields
  * in-range `Uint` SQL literals can normalize to `Int` fields
* synthesized derive-managed timestamp fields on the SQL write lane:
  * `INSERT` now fills `created_at` and `updated_at` when those fields are part
    of the target derived entity schema
  * `UPDATE` now refreshes `updated_at`
* kept non-dispatch SQL surfaces fail-closed for writes:
  * `query_from_sql(...)`
  * `execute_sql(...)`
  * grouped SQL dispatch
  * aggregate SQL dispatch
  all continue to reject `INSERT` / `UPDATE` instead of inventing fake generic
  write routes
* added one dedicated SQL write probe entity to the `sql_parity` schema and
  canister harness so PocketIC perf coverage can measure the new write lane
  without overloading an unrelated query fixture entity
* added the matching PocketIC SQL write probe surfaces for:
  * typed SQL dispatch insert
  * typed SQL dispatch update
* fixed local compile/test/clippy drift around the new write and projection
  support while keeping the changes local:
  * restored the minimal test-only projection helpers needed by session SQL
    projection tests
  * fixed stale borrowed-vs-owned planner test inputs
  * removed a few dead helper functions and store wrappers surfaced by strict
    all-target clippy
  * kept one large orchestrator enum inline and documented that choice with one
    scoped clippy expectation instead of turning a cleanup patch into an
    ownership rewrite
* pulled SQL projection/materialization ownership back behind the session SQL
  boundary instead of widening shared executor ownership:
  * shared scalar execution now returns structural pages only
  * final SQL projection/value/text shaping now lives under the session SQL
    projection module
  * shared executor contracts no longer carry SQL final payload families or
    SQL-prepared projection state
  * the shared terminal page/kernel path no longer finalizes SQL rows directly
* trimmed the remaining executor-side fallout from the old fused SQL terminal
  mode:
  * removed dead covering/materialization adapter state and unused helpers
  * renamed the remaining delete-side prepared projection helper so executor
    internals no longer carry `*Sql*` ownership names
  * rewrote executor comments/docs to describe structural or outer-surface
    ownership instead of SQL-owned behavior

Architectural notes:

* no new executor write pipeline was introduced
* no generic SQL mutation planner was introduced
* writes reuse the existing typed mutation/save path after narrow SQL parsing
  and normalization
* SQL write admission remains session-owned dispatch behavior, not a widened
  generic query/runtime contract
* SQL projection/materialization also moved back to a session-owned boundary:
  the executor now owns structural pages, retained-slot rows, and shared row
  materialization mechanics, while session SQL owns final projection shaping
* the supporting cleanup in this patch is local hygiene only:
  * test-boundary visibility fixes
  * dead helper deletion
  * clippy-only simplifications
  * executor/session boundary compression for SQL projection shaping
  none of that changes admitted SQL write semantics, and the executor cleanup
  restores the intended ownership boundary rather than widening behavior

Explicit current boundary:

* `INSERT` without an explicit column list remains fail-closed
* multi-row `INSERT ... VALUES (...), (...)` remains out of scope
* `UPDATE` with non-primary-key predicates remains out of scope
* SQL write routing still lives only on typed dispatch, not the generic
  `execute_sql(...)` surface

Verification:

* `cargo test -p icydb-core sql_write -- --nocapture`
* `cargo test -p icydb-core db::sql::parser::tests -- --nocapture`
* `cargo test -p icydb-testing-integration --test sql_canister sql_canister_perf_typed_dispatch_sql_write_probe_ -- --nocapture`
* `cargo check -p icydb-testing-integration --test sql_canister`
* `cargo clippy -p icydb-core --all-targets -- -D warnings`

## 0.76.2 - 2026-04-11 - SQL Surface Follow-Up

This follow-up keeps the `0.76` SQL-surface line narrow. It does not add a new
grouped execution family. Instead, it removes two remaining SQL-only grouped
restrictions while keeping the change local to SQL lowering and the session SQL
boundary rather than widening generic grouped planner/runtime ownership. It
also admits reduced SQL projection aliases and narrow `ORDER BY` aliases while
keeping alias ownership local to parser/session shaping and SQL prepare
normalization rather than generic query planning.

What changed:

* admitted top-level grouped `SELECT DISTINCT ... GROUP BY ...` for the
  existing grouped projection lane
* normalized redundant grouped top-level `DISTINCT` away during SQL lowering,
  so:
  * grouped logical plans match the non-`DISTINCT` form
  * grouped `EXPLAIN` output matches the non-`DISTINCT` form
  * grouped execution rows match the non-`DISTINCT` form
* admitted session-owned grouped computed text projection for grouped SQL
  shapes where text functions wrap grouped fields before aggregate outputs
* rewrote that grouped computed lane onto the canonical grouped base query, so:
  * grouped direct execution transforms grouped key values after the base
    grouped query runs
  * unified SQL dispatch returns grouped payloads with computed grouped column
    labels
  * grouped `EXPLAIN` output matches the rewritten base grouped query
* admitted reduced SQL projection aliases in both `AS alias` and bare trailing
  alias form for projection items
* kept aliases presentation-only by applying them only to outward session SQL
  column labels on:
  * scalar projection dispatch
  * grouped dispatch
  * global aggregate dispatch
  * session-owned computed projection dispatch
* kept lowering, planning, runtime execution, and `EXPLAIN` alias-neutral, so
  aliased queries plan identically to their unaliased forms
* admitted reduced SQL `ORDER BY <alias>` when the alias resolves to:
  * one projected field
  * one grouped projected field
  * one session-owned computed `LOWER(field)` / `UPPER(field)` target
* normalized field aliases during shared SQL prepare/lowering, so:
  * scalar execution matches the canonical unaliased order target
  * grouped execution matches the canonical unaliased order target
  * `EXPLAIN` output matches the canonical unaliased order target
* normalized computed `LOWER/UPPER` aliases on the session-owned computed SQL
  lane before that lane rewrites to its base field-only query
* kept unsupported `ORDER BY` alias targets fail-closed for:
  * broader computed aliases such as `TRIM(...)`
  * aggregate aliases
* consolidated the repeated PocketIC SQL query-lane coverage in:
  * `testing/pocket-ic/tests/sql_canister.rs`
  * the checked-in canister harness now folds the hand-written
    `projection` / `EXPLAIN EXECUTION` / `EXPLAIN JSON` / `DELETE` parity
    families into shared named case tables and assertion helpers for:
    * covering-route windows
    * strict-prefix `LIKE` / `STARTS_WITH` / text-range parity
    * casefold `LOWER(...)` / `UPPER(...)` parity
    * bounded planner-choice route selection
    * fail-closed unsupported `STARTS_WITH(TRIM(...))` surfaces
  * this keeps the same outer SQL contracts locked while making missing axis
    combinations such as `ASC` / `DESC`, offset vs non-offset, and
    projection vs explain surfaces visible as case rows instead of one-off
    test bodies
* added PocketIC perf harness coverage for:
  * `SELECT DISTINCT name, COUNT(*) FROM Customer GROUP BY name ORDER BY name ASC LIMIT 10`
  * `SELECT TRIM(name), COUNT(*) FROM Customer GROUP BY name ORDER BY name ASC LIMIT 10`
  * `SELECT name AS customer_name, age years FROM Customer ORDER BY customer_name ASC LIMIT 2`
  * `SELECT TRIM(name) AS trimmed_name, COUNT(*) total FROM Customer GROUP BY name ORDER BY name ASC LIMIT 10`
  * `SELECT LOWER(handle) AS normalized_handle, id FROM CustomerAccount WHERE active = true ORDER BY normalized_handle ASC, id ASC LIMIT 2`
  * `SELECT age years, COUNT(*) total FROM Customer GROUP BY age ORDER BY years ASC LIMIT 10`

Architectural notes:

* no grouped DISTINCT runtime was widened
* no grouped adjacency policy was relaxed
* no generic grouped expression ownership was widened
* no generic alias or expression-wrapper ownership was widened
* no planner/runtime order semantics were widened for alias support
* no planner or runtime semantics changed as part of the PocketIC query-lane
  matrix consolidation; that work stays in the outer test harness only
* top-level grouped `DISTINCT` stays local to SQL lowering because the admitted
  grouped SQL lane already emits one row per grouped key plus declared
  aggregates
* grouped computed text projection stays local to the session SQL boundary by
  rewriting onto the admitted grouped base query and transforming grouped key
  cells only after grouped execution
* projection aliases stay local to parser/session output shaping by carrying
  only parser-owned alias metadata into the final column-label packaging step
* field `ORDER BY` alias normalization stays in shared SQL prepare/lowering,
  while computed `LOWER/UPPER` alias normalization stays on the session-owned
  computed projection rewrite boundary because that lane rewrites its base
  select list before shared lowering runs

Explicit current boundary:

* generic grouped projection-expression widening remains out of scope for the
  current line even though the narrower session-owned grouped computed text
  lane is now admitted
* broader computed or aggregate `ORDER BY` aliases remain fail-closed
* JOIN remains parser-gated and outside the reduced SQL surface

## 0.76.1 - 2026-04-11 - Initial SQL Surface Completion Slice

The current `0.76` working line continues by widening several non-JOIN SQL
surfaces that were still narrower than the normalized executor and grouped
runtime already allowed. This is not a new execution architecture. The work
reuses the existing single-path planner/runtime stack and removes SQL-only
gaps or artificial restrictions where the underlying structural engine was
already capable of expressing the shape cleanly.

What changed:

* normalized scalar SQL `DISTINCT` by:
  * removing the PK-projection lowering restriction for scalar
    `SELECT DISTINCT ...`
  * moving immediate SQL DISTINCT semantics to the projected-row boundary
  * applying cursorless SQL `LIMIT/OFFSET` after projected-row dedup on that
    immediate SQL surface
* added global aggregate DISTINCT qualifiers for:
  * `COUNT(DISTINCT field)`
  * `SUM(DISTINCT field)`
  * `AVG(DISTINCT field)`
  using the existing aggregate runtime plus one SQL-side distinct-input adapter
  instead of widening the aggregate kernel
* admitted grouped field-target extrema in the grouped SQL/planner/runtime line:
  * `MIN(field)`
  * `MAX(field)`
  and fixed grouped extrema state so grouped field-target extrema compare and
  store actual field values rather than failing closed as unsupported
* admitted ordered SQL delete windows by:
  * parsing `DELETE ... OFFSET`
  * carrying ordered delete window state through lowering/planning
  * applying delete `OFFSET` and `LIMIT` after predicate filtering and ordering
    on the delete lane
  * keeping the explicit ordering requirement for delete windows
* normalized redundant DISTINCT extrema qualifiers:
  * `MIN(DISTINCT field)`
  * `MAX(DISTINCT field)`
  now lower to the existing `MIN/MAX(field)` path instead of failing closed
* admitted grouped aggregate DISTINCT qualifiers for:
  * `COUNT(DISTINCT field)`
  * `SUM(DISTINCT field)`
  * `AVG(DISTINCT field)`
  by teaching grouped runtime state to deduplicate aggregate input values per
  group for those admitted field-target aggregate kinds instead of reusing
  row-identity distinctness

Architectural notes:

* no duplicate execution flow was introduced
* no new aggregate runtime family was introduced
* no grouped algorithm was replaced
* the delete change widened the existing ordered delete boundary from
  limit-only to ordered window semantics instead of inventing a delete-only
  special path
* grouped DISTINCT support stayed inside the current grouped runtime by
  narrowing distinctness to the per-group aggregate-input boundary for
  `COUNT/SUM/AVG`, instead of inventing a second grouped execution lane

Explicit current boundary:

* generic grouped projection-expression widening remains out of scope for the
  current line
* JOIN remains parser-gated and outside the reduced SQL surface

Verification:

* `cargo fmt --all`
* `cargo check -p icydb-core -p icydb-testing-integration`
* targeted parser, lowering, session, grouped, and aggregate tests for:
  * scalar `SELECT DISTINCT` without PK projection
  * global aggregate DISTINCT qualifiers
  * grouped `MIN/MAX(field)`
  * grouped `COUNT/SUM/AVG(DISTINCT field)`
  * ordered `DELETE ... OFFSET`
  * `MIN/MAX(DISTINCT field)`
* targeted PocketIC perf coverage for:
  * scalar DISTINCT projection
  * global aggregate DISTINCT qualifiers
  * grouped `MIN/MAX(field)`
  * grouped `COUNT/SUM/AVG(DISTINCT field)`
  * ordered `DELETE ... OFFSET`
* targeted fail-closed tests for:
  * grouped projection-expression widening through the normal SQL lowering lane

## 0.76.0 - 2026-04-11 - SQL Surface Completion Audit Baseline

This patch opens the `0.76` line by turning the remaining reduced-SQL surface
gaps into one explicit, testable work queue before widening any new SQL
shapes. It does not add a new execution path or ship new SQL behavior on its
own. Instead, it establishes the baseline that the later `0.76.1` and
`0.76.2` feature slices build on.

What changed:

* created the dedicated `0.76` SQL surface-completion tracker in:
  * `docs/design/0.76-sql-surface-completion/0.76-status.md`
* enumerated the remaining non-`JOIN` SQL gaps that still fit the normalized
  single-path execution model
* locked the early `0.76` scope around SQL-surface widening only, rather than
  reopening planner/runtime architecture work
* documented the deliberate fail-closed boundaries for:
  * generic grouped projection-expression widening
  * broader computed or aggregate alias-ordering surfaces
  * `JOIN`
* established the verification expectation that later `0.76` work must land
  with parser, lowering, session, and PocketIC coverage for each newly
  admitted SQL shape

Architectural notes:

* no execution behavior changed in this baseline patch
* no new runtime, planner, or grouped execution abstraction was introduced
* this patch exists to make the `0.76` line explicit and auditable instead of
  letting the remaining SQL surface gaps drift as undocumented follow-up work

Verification:

* design/status review only
* no runtime or parser behavior changed in this baseline patch
