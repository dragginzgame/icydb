# 0.101 Changelog

## 0.101.2

- `0.101.2` closes the remaining practical scalar-function parity work on the current reduced SQL surface instead of opening another planner or runtime slice. The shipped end state now keeps the admitted scalar families aligned across the ordinary expression-bearing clause owners:
  - projection
  - scalar `WHERE`
  - grouped `HAVING`
  - aggregate input
  - aggregate `FILTER (WHERE ...)` via the widened `WHERE` seam
  - direct bounded `ORDER BY`
  - grouped bounded `ORDER BY`, including wrapped post-aggregate alias parity
- Unary numeric wrappers (`ABS`, `CEIL`, `CEILING`, `FLOOR`) now ride the same shared expression-argument seam instead of staying on the older unary-field-only path. That covers direct scalar expressions like `ABS(strength + dexterity)`, aggregate-wrapped terms like `FLOOR(AVG(level))`, aggregate inputs like `SUM(ABS(age - 15))`, and direct bounded order terms like `ORDER BY FLOOR(dodge_chance + 0.10)`.
- The null/value-selection family (`COALESCE`, `NULLIF`) now moves beyond projection-only ownership. SQL `WHERE` can preserve these shapes as semantic `filter_expr` while falling back to residual execution when the derived predicate compiler cannot represent them, grouped `HAVING` can use the same wrapped post-aggregate value-selection expressions, aggregate inputs can nest them directly, and aggregate `FILTER (WHERE ...)` inherits the widened `WHERE` surface without a separate special-case parser line.
- The unary text family (`LOWER`, `UPPER`, `LENGTH`, `TRIM`, `LTRIM`, `RTRIM`) now accepts nested expression arguments on the same shared scalar-expression seam instead of only bare field inputs. That keeps text wrappers aligned with the numeric and value-selection families across projection, direct bounded `ORDER BY`, grouped `HAVING`, aggregate input, and residual `WHERE` evaluation.
- Grouped bounded ordering no longer stops at the direct wrapped post-aggregate form. Direct terms like `ORDER BY COALESCE(NULLIF(AVG(age), 40), 99)` and the alias form `SELECT ..., COALESCE(NULLIF(AVG(age), 40), 99) AS adjusted_avg ... ORDER BY adjusted_avg` now both lower and execute through the same grouped Top-K lane. The grouped helper boundary remains intentionally narrower only for computed grouped projections that still depend on raw grouped field access rather than pure post-aggregate expressions.

```sql
SELECT name, COALESCE(NULLIF(AVG(age), 40), 99) AS adjusted_avg
FROM Character
GROUP BY name
ORDER BY adjusted_avg DESC, name ASC
LIMIT 2;
```

## 0.101.1

- `0.101.1` carries the `0.101` scalar-function line across the remaining ordinary expression-bearing clause families instead of leaving the first projection-only cut as a special case. This patch still stays inside the existing reduced SQL and expression-first `WHERE` architecture; it does not reopen predicate planning or prepared-template design.
- Unary numeric wrappers (`ABS`, `CEIL`, `CEILING`, `FLOOR`) move off the older unary-field-only parser seam and onto the shared scalar-expression path for the admitted clause owners. That covers:
  - projection expressions such as `ABS(strength + dexterity)`
  - aggregate-wrapped grouped expressions such as `FLOOR(AVG(level))`
  - aggregate inputs such as `SUM(ABS(age - 15))`
  - direct bounded `ORDER BY` terms such as `ORDER BY FLOOR(dodge_chance + 0.10)`
- `COALESCE` and `NULLIF` stop being projection-only. SQL `WHERE` now admits these shapes by preserving them as semantic `filter_expr` and falling back to residual execution when the narrower derived predicate compiler cannot represent the expression. The same value-selection family also becomes available in grouped `HAVING`, aggregate inputs, and aggregate `FILTER (WHERE ...)` through the widened `WHERE` seam.
- Grouped bounded ordering widens on the direct wrapped post-aggregate path. Direct grouped order terms like `ORDER BY COALESCE(NULLIF(AVG(age), 40), 99)` now lower and execute on the grouped Top-K lane instead of staying fail-closed at grouped order admissibility. Alias-based parity is still not part of this patch line yet; that follow-through remains for the next patch.
- This patch keeps one boundary explicit instead of pretending the whole grouped computed-projection story is solved. At the end of `0.101.1`, direct wrapped grouped order terms work, but grouped alias forms like `... AS adjusted_avg ORDER BY adjusted_avg` are still a separate follow-up slice.

```sql
SELECT name, AVG(age)
FROM Character
GROUP BY name
ORDER BY COALESCE(NULLIF(AVG(age), 40), 99) DESC, name ASC
LIMIT 2;
```

## 0.101.0

- `0.101.0` widens the projection-first scalar function surface without reopening `WHERE` again. The current line admits the new unary numeric functions on direct field arguments and the null/value-selection family on projection expressions, including nested `COALESCE(NULLIF(...), ...)` forms, while `COALESCE(...)` and `NULLIF(...)` in `WHERE` remain intentionally fail-closed for now.

```sql
SELECT name,
       COALESCE(NULLIF(guild_rank, background), class_name) AS preferred_label
FROM Character
ORDER BY preferred_label ASC, name ASC
LIMIT 20;
```
