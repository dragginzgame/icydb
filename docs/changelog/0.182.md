# IcyDB 0.182 Changelog

## 0.182.6

### 📝 Summary

- Hard-cuts public/schema direct `storage(stable(...))` now that journaled
  cached-stable stores are the durable public path.
- Removes the stable storage mirror from the SQL performance audit matrix so
  storage comparisons focus on volatile heap versus supported journaled
  durable stores.
- Deletes the accumulated write-boundary source-scan integration test and the
  stale stable-syntax resurrection fixtures, reducing maintenance pressure
  from tests that only protected removed paths.

### 🗑️ Removed

- Removes the schema-level direct stable storage model, stable store parser
  lowering, generated stable store wiring, wasm-helper stable fixture branch,
  and stable audit-schema declarations.
- Removes stable-backed SQL perf canister endpoints and integration-test
  matrix rows; the storage mirror now compares heap and journaled surfaces.
- Removes stale trybuild fixtures for stable memory-id diagnostics, replacing
  them with journaled equivalents where the behavior still matters.
- Removes the source-text write-boundary guard target instead of carrying
  broad anti-resurrection assertions for retired authority paths.

### 🧭 Migration Notes

- Canister schemas that still declare `storage(stable(...))` must switch to
  `storage(journaled(...))` for durable storage or `storage(heap())` for
  volatile fixtures.
- Direct stable storage remains rejected at parse time; this slice removes
  the old compatibility fixtures, not the fail-closed parser behavior.

### 📊 Audit

- The slice removes roughly `2.8k` lines net, with the largest single deletion
  coming from the obsolete write-boundary source-scan integration test.
- The remaining direct stable syntax hit is the parser rejection branch itself;
  dedicated resurrection fixtures and ignored trybuild wip outputs are gone.

### 🧪 Testing

- Validates formatting, schema-derive tests, core compile-fail/compile-pass
  trybuild tests, schema macro compile-fail tests, workspace clippy, invariant
  checks, and whitespace checks.
- A broad `icydb-core --tests --no-run` check remains unsuitable because
  no-`sql` feature-gated lib-test imports fail independently of this slice.

## 0.182.5

### 📝 Summary

- Adds stable, heap, and journaled storage-mirror SQL surfaces to the
  deterministic generated perf matrix so durable and volatile backend read
  costs are visible beside ordinary query hotspots.
- Extends the matrix Markdown report with a side-by-side storage comparison
  table that pairs stable, heap, and journaled rows by generated scenario
  suffix and reports total/store instruction deltas.
- Adds focused primary-key `LIMIT 1` storage-read attribution so the storage
  audit prints direct stable-versus-journaled instruction deltas.
- Keeps top-level SQL membership predicates compact by lowering parser-owned
  membership forms directly into predicate-only filters when the WHERE clause
  can be represented completely by the runtime predicate subset.

### 📊 Audit

- The focused storage audit passed and showed journaled primary-key SQL
  `LIMIT 1` at `663,495` total instructions versus stable at `772,272`, a
  `108,777` instruction reduction; cached journaled was `270,860` versus
  cached stable at `378,947`.
- The full deterministic generated SQL matrix passed with `1,824` generated
  scenarios, `1,743` executed scenarios, and `81` expected fail-closed
  `QueryPlan` rejections.
- The largest total-instruction hotspots are stable-backed mirror scans. In
  the worst stable mirror rows, heap and journaled equivalents are roughly
  half the total instruction cost and the store phase falls from about
  `28.6M` instructions to about `7.0M`.
- Outside the storage mirror rows, the largest ordinary query hotspots remain
  blob payload-returning projections, followed by introspection-heavy
  `SHOW ENTITIES` and `EXPLAIN` rows.
- Compile hotspots remain concentrated in filtered expression-index account
  query families around `LOWER(handle) LIKE ...` plus expression ordering.

### 🔧 Changed

- Adds `stable_user`, `heap_user`, and `journaled_user` matrix surfaces backed
  by the existing shape-matched perf audit entities.
- Adds deterministic mirror projections, predicates, orders, and report
  pairing logic for those storage surfaces.
- Adds a synthetic report-format regression test proving the storage mirror
  table keeps heap and journaled paired against the stable baseline.
- Adds an explicit stable-versus-journaled primary-key `LIMIT 1` assertion in
  the focused storage perf audit while preserving the existing stable, heap,
  and journaled bounded checks.
- Skips visible SQL filter-expression lowering for parser-owned top-level
  membership predicates that can lower directly to a complete runtime
  predicate subset.

### 🧪 Testing

- Validates formatting, whitespace, generated SQL matrix shape tests, storage
  mirror report pairing, focused integration clippy for the matrix and storage
  perf audit targets, focused core SQL membership lowering, the full SQL core
  lib suite, the focused PocketIC storage comparison, and the full
  deterministic PocketIC SQL perf matrix.

## 0.182.4

### 📝 Summary

- Keeps top-level SQL `IN` / `NOT IN` membership predicates compact through
  the parser-owned WHERE handoff instead of expanding them into OR/AND
  expression trees before deriving the runtime predicate subset.
- Adds a negated-prefix route guard proving `NOT LIKE 'prefix%'` keeps a
  residual filter and cannot claim a pure covering read without a complement
  scan contract.

### 📊 Audit

- Re-runs the deterministic generated SQL performance matrix after the
  membership lowering change.
- The `age IN (...)` scenario family keeps the same pass/fail matrix shape
  while average compile instructions drop from roughly 693k to roughly 656k,
  saving about 36.7k compile instructions per affected scenario.
- The remaining top hotspots stay centered on row fetch/materialization after
  access planning and on broader planner/executor work for field-to-field
  predicates, ordered primary-key ranges with non-covered projections, and
  expression ordering.

### 🔧 Changed

- Adds a SQL-owned predicate-subset derivation helper that can use parser
  context for top-level membership while preserving the visible expression
  for semantics, explain output, and query identity.
- Routes strict SQL predicate lowering through the same compact-membership
  path so SELECT, UPDATE selector, and aggregate filter paths no longer keep
  separate membership derivation behavior.
- Leaves visible SQL filter-expression retention unchanged; dropping or
  compacting fully covered visible filters remains a separate optimization
  because it affects explain and cache-identity surfaces.

### 🧪 Testing

- Validates formatting, whitespace, focused top-level membership lowering,
  SQL/fluent predicate convergence, direct prefix regression coverage,
  focused SQL clippy, the full `icydb-core` SQL lib test suite, and the
  deterministic generated SQL performance matrix.

## 0.182.3

### 📝 Summary

- Makes generated SQL performance-matrix runs explicitly choose either the
  deterministic baseline or seeded-random exploration, rejecting random
  controls without `ICYDB_SQL_PERF_MATRIX_MODE=random` instead of producing a
  mixed report.
- Adds stable diagnostic labels to generated SQL matrix failure reports so
  fail-closed query shapes show names such as `QueryPlan (3)` rather than only
  compact numeric diagnostics.
- Fixes local PocketIC fixture builds so generated canister build scripts
  receive the local IcyDB build target and keep SQL introspection available for
  canister tests.
- Fixes the release workflow default-empty wasm artifact so it builds the
  current empty audit canister package instead of referencing the removed
  package ID.
- Teaches the deterministic secondary-order contract to treat trailing explicit
  primary-key index components as the normal primary-key tie-break, allowing
  key-only ordered queries over indexes such as `(bucket, id)` to stay on
  covering reads instead of hydrating rows.

### 📊 Audit

- The latest mixed pre-split SQL artifact showed no obvious unexpected
  failures: the visible rejects were fail-closed query-planner shapes around
  expression ordering, `age IN`, and `LOWER(handle)` combinations.
- The same artifact keeps the next performance targets clear: blob payload
  return dominates total instructions, `SHOW ENTITIES` and `EXPLAIN` are
  expensive introspection paths, and ordinary query hotspots still center on
  avoidable row/value materialization after access planning.
- The first execution optimisation from the matrix targets redundant
  row/value materialization: when an index explicitly ends with the entity
  primary key, the planner now recognises the order as already satisfied and
  can project primary-key-only rows directly from index/key material.
- The write-boundary source audit now keeps negative authority-leak checks
  while dropping brittle implementation-shape mirrors and old design-document
  assertions, reducing the guard suite to the boundaries that still carry
  signal.

### 🧹 Cleanup

- Moves shared source-scan helpers out of the `write_boundary_guards`
  integration test and into small test-support modules for integration and
  unit guard suites.
- Strengthens the shared `#[cfg(test)]` source stripper so source guards can
  ignore multi-line test-only helpers without treating them as production
  authority leaks.

### 🧪 Testing

- Validates formatting, focused SQL perf matrix shape tests, stable diagnostic
  label coverage, focused clippy for the generated matrix audit target, and
  whitespace checks.
- Adds focused SQL-feature planner, covering-plan, and session covering checks
  for explicit primary-key suffix index order compatibility.
- Validates the trimmed write-boundary guard suite with focused test and
  clippy runs.
- The ignored PocketIC hotspot scan remains manual because this slice changes
  audit report structure and a core planner/executor contract; full matrix
  reruns remain opt-in hotspot validation.

## 0.182.2

### 📝 Summary

- Adds a generated SQL performance-matrix audit around the existing `sql_perf`
  audit canister, giving optimisation work broad deterministic coverage plus
  optional seeded-random query generation.
- Starts the shared row-backed projection optimisation line by reusing repeated
  direct raw-row source slots within a single SQL output row.
- Makes generated SQL introspection target-aware: local builds keep `EXPLAIN`,
  `DESCRIBE`, and `SHOW` enabled by default, while IC builds reject them unless
  explicitly enabled.
- Pins the current materialization hotspot families in the fixed SQL perf
  audit suite so future work keeps measuring row-backed predicates,
  expression-order paths, filtered expression indexes, and blob payload
  boundaries.

### 📊 Audit

- The generated SQL matrix can run all deterministic scenarios or a bounded
  prefix, add seeded-random cases, and emit ranked JSON/Markdown hotspot
  reports for later comparison.
- The fixed SQL perf audit now includes explicit scenarios for
  `age > rank ORDER BY LOWER(name)` and
  `age IN (...) ORDER BY age + rank`, alongside the existing filtered
  `LOWER(handle)` and blob byte-length/payload coverage.

### 🔧 Changed

- The direct raw-row projection materializer now detects repeated direct source
  slots and caches the decoded value for the rest of that output row.
- Projection semantics and output ordering are unchanged; the optimisation only
  avoids duplicate value materialization when one SQL projection repeats the
  same direct field.
- The blob coverage remains deliberately split between byte-length projections
  and actual payload projections so future work can prove payload bytes are
  touched only when returned.
- Adds `[canisters.<name>.sql.introspection]` with independent `local` and
  `ic` booleans; defaults are `local = true` and `ic = false`.
- Generated query dispatch now gates `EXPLAIN`, `DESCRIBE`, and `SHOW` before
  entity routing and reports a dedicated runtime-boundary diagnostic when
  introspection is disabled.
- The CLI passes the generated build target into build/deploy commands so local
  and IC builds resolve the policy deterministically, while unknown direct
  builds fail closed for introspection.

### 🧭 Migration Notes

- IC canisters that intentionally expose generated SQL introspection must set
  `ic = true` in `icydb.toml`; local-only introspection requires no config.

### 🧪 Testing

- Validates formatting, whitespace, focused SQL projection materialization,
  expression-index covering execution, blob byte-length guards, fixed perf
  scenario registration, and focused clippy for the touched core and
  integration-test targets.
- Adds focused coverage for local/IC introspection config defaults and
  overrides, generated query rejection, build-target environment propagation,
  diagnostic rendering, and SQL dispatch classification.
- The full ignored generated matrix remains an opt-in audit run because it
  requires the local canister test environment and is intended for hotspot
  discovery rather than every release-prep check.

## 0.182.1

### 📝 Summary

- Adds a scalar rows-only fluent load execution path so ordinary row reads no
  longer retain grouped and delete executor branches through the broad
  `LoadQueryResult` boundary.
- Moves public row materializers and the row-only fluent audit canisters onto
  the new rows-only path while leaving broad `.execute()` available for callers
  that need the full result surface.
- Renames the default wasm-footprint canister matrix around the dimension being
  measured: empty generated wiring, rows-only fluent reads, broad fluent
  execute, SQL query, and entity-count scaling.

### 📊 Audit

- The default wasm-footprint matrix is now `default_empty`,
  `one_entity_fluent_rows`, `one_entity_fluent_execute`,
  `one_entity_sql_query`, and `ten_entity_fluent_rows`.
- Metrics-only and complex-schema audit canisters are removed from the default
  run because metrics is a feature-cost audit and complex schema shape is no
  longer a high-signal size axis.
- Current shrunk raw `.wasm` sizes are: `default_empty` 6,552 bytes,
  `one_entity_fluent_rows` 1,938,760 bytes,
  `one_entity_fluent_execute` 2,113,550 bytes,
  `one_entity_sql_query` 2,624,953 bytes, and
  `ten_entity_fluent_rows` 1,955,283 bytes.
- Current shrunk gzipped `.wasm` sizes are: `default_empty` 4,087 bytes,
  `one_entity_fluent_rows` 623,838 bytes,
  `one_entity_fluent_execute` 690,437 bytes,
  `one_entity_sql_query` 907,835 bytes, and
  `ten_entity_fluent_rows` 625,912 bytes.
- The rows-only fluent path reduces the one-entity fluent row-read canister by
  roughly 174 KiB raw against the broad fluent execute path, while the clean
  one-to-ten entity slope is roughly 1,836 raw bytes per extra entity.

### 🔧 Changed

- Adds `execute_rows()` to the core fluent load terminal and public facade.
- Routes public row materializers such as row/entity/id collection helpers
  through the scalar rows-only execution path instead of broad `.execute()`
  followed by `into_rows()`.
- Keeps broad `.execute()` as the full row/grouped result surface and makes the
  rows-only path reject grouped plans before executor dispatch.
- Removes the unused complex audit schema fixture macros after dropping the
  complex audit canisters from the active matrix.
- Updates the recurring wasm-footprint audit contract and integration canister
  mapping to the renamed default matrix.

### 🧪 Testing

- Validates formatting, whitespace, focused core rows-only behavior, focused
  core/public clippy, renamed audit canister package checks, renamed audit
  canister clippy, and the default wasm-size report.

## 0.182.0

### 📝 Summary

- Starts the 0.182 audit and optimisation line with a fixed wasm-footprint
  baseline for the default canister matrix.
- Captures the current SQL and fluent query-performance matrices so later
  optimisation slices can compare against a concrete snapshot.
- Applies the first SQL metadata execution optimisation while preserving
  accepted-schema authority.
- Splits SELECT/grouped execution helpers out of the root SQL execution hub so
  follow-up optimisation work starts from a narrower command-routing module.

### 📊 Audit

- Captures the regenerated 2026-06-13 wasm-footprint audit with profile
  `wasm-release` and SQL variant `sql-on`; this current-tree report is the
  0.182 optimisation baseline.
- Canonical shrunk raw `.wasm` sizes are now: `minimal` 6,516 bytes,
  `minimal_metrics` 288,647 bytes, `one_simple` 2,113,542 bytes,
  `one_sql_query` 2,624,946 bytes, `one_fluent_query` 2,113,535 bytes,
  `one_complex` 2,131,293 bytes, `ten_simple` 2,130,041 bytes, and
  `ten_complex` 2,147,851 bytes.
- Canonical shrunk gzipped `.wasm` sizes are now: `minimal` 4,096 bytes,
  `minimal_metrics` 111,774 bytes, `one_simple` 690,381 bytes,
  `one_sql_query` 907,806 bytes, `one_fluent_query` 690,313 bytes,
  `one_complex` 696,242 bytes, `ten_simple` 691,819 bytes, and
  `ten_complex` 698,307 bytes.
- The audit script keeps the 2026-06-11 report as comparability metadata, but
  follow-up 0.182 optimisation slices should compare against this regenerated
  2026-06-13 current-tree baseline.
- Captures 94 SQL query-performance scenarios, including primary-key,
  secondary-index, expression-index, grouped, metadata, blob payload, repeat,
  cache-warm, and update-invalidation cases.
- Captures 17 fluent query-performance scenarios, including cold compile,
  warm cache, direct-scan attribution, grouped attribution, and repeat-query
  cases.
- Re-runs the SQL performance harness after the metadata optimisation. In the
  captured comparison, `DESCRIBE PerfAuditUser` drops from 5,543,430 to
  224,342 instructions, `SHOW COLUMNS PerfAuditUser` drops from 5,534,014 to
  215,471 instructions, and `SHOW ENTITIES` drops from 18,321,354 to
  17,323,018 instructions.

### 🔧 Changed

- Establishes SQL metadata output construction as the first runtime
  optimisation target, especially `SHOW ENTITIES`, `DESCRIBE`,
  `SHOW COLUMNS`, and `EXPLAIN`.
- Establishes `one_sql_query` as the first wasm-footprint target because the
  current baseline shows the SQL result surface is materially larger than the
  fluent/simple query baseline.
- Reuses accepted catalog snapshots for `DESCRIBE` and `SHOW COLUMNS`
  execution instead of rebuilding equivalent metadata through the slower
  typed metadata path.
- Lets `SHOW ENTITIES` read compatible accepted snapshots before falling back
  to reconciliation, reducing the path modestly while leaving the larger
  all-entity catalog-summary design deferred.
- Moves SQL metadata command dispatch into the metadata execution module and
  moves SELECT/grouped prepared-plan execution into a dedicated SELECT module,
  reducing the root SQL execution hub from roughly 1000 lines to roughly 609
  lines.

### 🧪 Testing

- Validates the wasm-footprint audit, including per-canister size reports and
  Twiggy top, dominator, retained, and monomorphization artifacts.
- Validates the focused SQL query-performance harness against the repo-local
  PocketIC binary.
- Validates the focused fluent query-performance harness against the repo-local
  PocketIC binary.
- Validates the metadata optimisation and execution split with focused SQL
  metadata, projection, grouped, aggregate, diagnostics, clippy, formatting,
  and whitespace checks.
