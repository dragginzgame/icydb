# 0.79 — Predicate Expressions

## 0.79.1 - 2026-04-13 - Add NOT BETWEEN

This patch widens the same bounded predicate surface slightly by admitting
`NOT BETWEEN` on the shared SQL and fluent predicate path, and by accepting
ergonomic symmetric compare spellings that normalize back onto the same
canonical field-first predicate forms.

The shipped surface is:

* SQL:

  ```sql
  SELECT * FROM Character WHERE strength NOT BETWEEN 10 AND 20;
  SELECT * FROM Character WHERE 5 < strength;
  SELECT * FROM Character WHERE dexterity = strength;
  ```

* fluent:

  ```rust
  db.load::<Character>().filter(field("strength").not_between(10, 20))
  ```

What changed:

* added one predicate-owned `not_between(...)` helper to the public predicate
  model as the direct dual of the existing `between(...)` lowering
* added one matching fluent builder helper,
  `field(\"...\").not_between(lower, upper)`
* widened the reduced SQL predicate parser to admit `field NOT BETWEEN lower AND upper`
  without creating a new runtime node family
* kept the canonical lowering deliberately small:
  * `BETWEEN` stays `field >= lower AND field <= upper`
  * `NOT BETWEEN` stays `field < lower OR field > upper`
* widened reduced SQL predicate parsing to accept literal-leading compare forms
  like `5 < strength`
* normalized those symmetric forms back onto the canonical field-first compare path:
  * `5 < strength` -> `strength > 5`
  * `5 <= strength` -> `strength >= 5`
  * `5 = strength` -> `strength = 5`
* added deterministic canonical ordering for symmetric field-to-field equality
  and inequality, so swapped field spellings lower to the same
  `CompareFields` representation
* kept the runtime/planner boundary unchanged:
  both `NOT BETWEEN` and symmetric compare forms stay on the existing
  predicate path instead of widening into generic predicate expressions or
  adding a new runtime shape
* added direct proof for:
  * standalone predicate parsing
  * SQL/fluent session parity
  * live canister-boundary execution
  * statement-level SQL parsing for symmetric compare forms
  * literal-leading mixed-type compare rejection after normalization

Still intentionally excluded:

* field-to-field `NOT BETWEEN`
* arithmetic bounds
* symmetric arithmetic predicates
* literal-leading `BETWEEN`
* grouped predicate-expression widening
* generic expression predicates

Verification:

* `cargo test -p icydb-core parse_sql_predicate_normalizes_literal_leading_compare_to_field_first -- --nocapture`
* `cargo test -p icydb-core parse_sql_predicate_normalizes_swapped_field_equality_to_deterministic_order -- --nocapture`
* `cargo test -p icydb-core parse_select_statement_with_symmetric_predicate_forms -- --nocapture`
* `cargo test -p icydb-core execute_sql_scalar_not_between_matches_fluent_runtime_result -- --nocapture`
* `cargo test -p icydb-core execute_sql_scalar_symmetric_compare_forms_match_canonical_results -- --nocapture`
* `cargo test -p icydb-core execute_sql_scalar_literal_leading_mixed_type_compare_still_rejects_semantically -- --nocapture`

## 0.79.0 - 2026-04-13 - Add Field-To-Field Predicate Comparison

This patch opens the `0.79` line with one bounded predicate-expression slice:
single-entity field-to-field comparison in predicate position, on the shared
SQL and fluent predicate path.

The shipped surface is:

* SQL:

  ```sql
  SELECT * FROM Character WHERE strength > dexterity;
  SELECT * FROM Character WHERE strength = dexterity;
  ```

* fluent:

  ```rust
  db.load::<Character>().filter(field("strength").gt_field("dexterity"))
  db.load::<Character>().filter(field("strength").eq_field("dexterity"))
  ```

What changed:

* added one predicate-owned `CompareFields` leaf instead of widening the scalar
  projection expression system again
* lowered both SQL and fluent field-to-field comparisons to the same canonical
  predicate leaf before planning
* kept the executable compare runtime on one unified compare family with typed
  operands (`field slot` or `literal`) instead of maintaining separate
  field-vs-literal and field-vs-field runtime node families
* kept field-to-field comparison residual-only in planning, so literal/indexable
  clauses can still drive access choice while field-to-field checks remain
  row-filter work
* kept grouped query lanes fail-closed for this slice
* fixed mixed numeric equality and inequality normalization so field-to-field
  `=` / `!=` comparisons widen supported numeric pairs instead of failing strict
  coercion when the two field kinds differ (for example `Uint` vs `Int`)
* added semantic rejection coverage for the important fail-closed cases:
  incompatible type ordering, ordered boolean comparison, and missing
  right-hand fields
* locked the explain/debug surface so field-to-field predicates stay visible as
  `CompareFields { ... }` in the predicate tree instead of collapsing into
  opaque expression text
* locked the same-field edge case (`field = field`) as valid and stable
  residual compare behavior
* tightened the runtime after the first cut so scalar `field-slot` vs
  `field-slot` compares now stay on the scalar slot seam too; the compare model
  is unified and the remaining difference versus `field op literal` is only
  optimization eligibility, not a separate execution shape
* moved the local SQL dev shell from a shell-owned formatter plus example
  helper into one maintained Rust CLI crate rooted at
  `/home/adam/projects/icydb/crates/icydb-cli/src/main.rs`, with
  `scripts/dev/sql.sh` reduced to deploy/reset/init orchestration plus launcher
  wiring
* kept the shell output on the canonical Rust SQL render path, including boxed
  tables, `NULL` cell rendering, and footer-style row counts
* added perf-aware shell footer rendering through the canister-local
  `query_with_perf(sql)` surface, so interactive and one-shot shell queries now
  report instruction counts like `4 rows in set (7.8M instructions)` when the
  canister exposes the perf wrapper
* made the Rust shell handle semicolon-terminated multiline input, persistent
  history, blank-line reprompts, and top-level `\\q` / `quit` / `exit` without
  sending empty SQL into the engine
* added dedicated regressions for:
  * SQL/fluent runtime parity on the same field-to-field filter
  * mixed signed/unsigned equality on the live SQL session path
  * strict rejection of near-miss expression predicates like
    `strength = dexterity + 1`
  * live canister-boundary proof that a mixed literal plus field-to-field
    predicate (`age > 18 AND age > rank`) keeps the literal clause compatible
    with ordinary access planning while the field-to-field clause remains a
    residual filter

Still intentionally excluded:

* grouped field-to-field predicate execution
* arithmetic predicate expressions
* field-to-field `IN`, `CONTAINS`, `STARTS_WITH`, or other non-comparison forms
* planner attempts to rewrite field-to-field compares into indexable ranges
