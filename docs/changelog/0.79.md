This patch adds a dedicated recurring SQL performance audit surface instead of
continuing to piggyback instruction sampling onto the lightweight SQL smoke
tests. The new audit has its own schema fixture crate, canister crate, PocketIC
test entrypoint, and checked-in baseline artifact so future perf drift can be
measured against one stable matrix without coupling that work to smoke or
parity harness maintenance.

What changed:

* added one dedicated audit schema fixture crate:
  * `/home/adam/projects/icydb/schema/audit/sql_perf`
* added one dedicated audit canister crate:
  * `/home/adam/projects/icydb/canisters/audit/sql_perf`
* added one dedicated recurring PocketIC audit:
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/sql_perf_audit.rs`
* added one checked-in baseline artifact for scenario deltas:
  * `/home/adam/projects/icydb/testing/pocket-ic/tests/fixtures/sql_perf_audit_baseline.json`
* kept the new perf surface isolated from the existing SQL smoke/parity suites:
  * no `PerfAudit*` entities were added to the lightweight `sql_canister`
    correctness harness
  * the only shared wiring is the generic PocketIC helper knowing how to boot
    the dedicated `sql_perf` canister package
* widened the dedicated audit matrix so it covers a broad stable slice of the
  current SQL planner/runtime surface, including:
  * primary-key order-only and range scans
  * secondary order-only, range, prefix, and equality routes
  * covering versus materialized projection pairs
  * `LIMIT/OFFSET` windows
  * expression-index `ORDER BY LOWER(...)` routes
  * prefix `LIKE` / `NOT LIKE`
  * prefix `ILIKE` / `NOT ILIKE`
  * `IN` / `NOT IN`
  * field-to-field comparison
  * mixed signed/unsigned numeric field comparison
  * field-bound `BETWEEN` / `NOT BETWEEN`
  * grouped count, scalar count, and explain metadata routes
  * filtered and filtered-expression account routes
* changed the recurring report shape so every row now prints:
  * scenario key
  * SQL string
  * average local instructions
  * absolute delta versus the checked-in baseline
  * percent delta versus the checked-in baseline
* kept the scenario builders explicit and query-family-local instead of hiding
  the matrix behind generated data, so future additions stay easy to audit and
  narrow clippy cleanup can keep those builders small without weakening lint
  policy
* refreshed the macro `trybuild` stderr snapshots for the new index diagnostic
  span shape, since expression-index validation now points directly at the
  `index(fields = "...")` argument instead of the whole `#[entity(...)]`
  attribute block

The dedicated audit is intentionally not a general correctness harness. Its job
is to measure one stable instruction matrix repeatedly and compare that matrix
against a checked-in baseline. That means fixture shape, query naming, and
baseline updates should be treated as perf-audit changes, not as routine smoke
test maintenance.

## 0.79.4 - 2026-04-14 - Add IS NOT TRUE/FALSE and Field-Bound BETWEEN

This patch keeps the `0.79` line on the same bounded predicate SQL seam, but
widens two existing families slightly so boolean negation and sibling-field
range checks can stay on canonical predicate forms instead of opening generic
predicate-expression support.

The shipped surface is:

* SQL:

  ```sql
  SELECT * FROM Character WHERE active IS NOT TRUE;
  SELECT * FROM Character WHERE active IS NOT FALSE;
  SELECT * FROM Character WHERE strength BETWEEN dexterity AND charisma;
  SELECT * FROM Character WHERE strength NOT BETWEEN dexterity AND charisma;
  ```

* fluent:

  ```rust
  db.load::<Character>()
      .filter(field("strength").between_fields("dexterity", "charisma"))
  db.load::<Character>()
      .filter(field("strength").not_between_fields("dexterity", "charisma"))
  ```

What changed:

* widened plain-field `IS` parsing to admit:
  * `IS NOT TRUE`
  * `IS NOT FALSE`
* kept that widening intentionally small and canonical:
  * `field IS NOT TRUE` lowers to `NOT(field = true)`
  * `field IS NOT FALSE` lowers to `NOT(field = false)`
* locked the null and missing-value edge on the current predicate runtime:
  `IS NOT TRUE` and `IS NOT FALSE` keep null and absent values on the
  "not true / not false" side instead of introducing a second boolean semantic
  lane
* widened plain-field `BETWEEN` / `NOT BETWEEN` parsing to admit plain-field
  lower and upper bounds as well as literals
* kept that widening intentionally small and canonical:
  * `field BETWEEN lower_field AND upper_field`
    lowers to `field >= lower_field AND field <= upper_field`
  * `field NOT BETWEEN lower_field AND upper_field`
    lowers to `field < lower_field OR field > upper_field`
* added matching fluent helpers:
  * `between_fields(lower, upper)`
  * `not_between_fields(lower, upper)`
* kept field-bound range checks on the existing compare-fields predicate seam
  instead of introducing a dedicated range leaf or widening into generic
  predicate expressions
* added direct proof for:
  * standalone predicate parsing
  * statement-level SQL parsing
  * SQL/fluent lowering parity for `IS NOT TRUE` / `IS NOT FALSE`
  * runtime null/missing semantics for `IS NOT TRUE` / `IS NOT FALSE`
  * SQL/fluent parity for field-bound `BETWEEN`
  * session execution parity for field-bound `BETWEEN` and `NOT BETWEEN`

Still intentionally excluded:

* `IS NOT UNKNOWN`
* grouped `HAVING ... IS NOT TRUE/FALSE`
* wrapped-field `BETWEEN` / `NOT BETWEEN`
* arithmetic range bounds
* generic expression predicate bounds

Verification:

## 0.79.3 - 2026-04-14 - Add Prefix ILIKE / NOT ILIKE

This patch keeps the `0.79` line on the same bounded predicate SQL seam, but
adds familiar case-insensitive prefix aliases so callers no longer need to
spell those filters through `LOWER(...) LIKE ...` or `UPPER(...) LIKE ...`.

The shipped surface is:

* SQL:

  ```sql
  SELECT * FROM Character WHERE name ILIKE 'al%';
  SELECT * FROM Character WHERE name NOT ILIKE 'al%';
  ```

What changed:

* widened reduced SQL predicate parsing to admit `ILIKE` and `NOT ILIKE`
* kept that widening intentionally small and alias-like:
  * `name ILIKE 'al%'` lowers to the existing casefolded prefix path
  * `name NOT ILIKE 'al%'` lowers to `Not(...)` around that same casefolded
    prefix path
* kept `ILIKE` / `NOT ILIKE` on the same bounded accepted pattern family as the
  existing `LIKE` surface:
  * prefix-only patterns ending in `%`
  * no general wildcard widening
  * no suffix, infix, or arbitrary pattern execution
* kept the runtime and planning surface unchanged by reusing the existing
  casefolded starts-with predicate family instead of introducing a new
  predicate leaf or a second text-matching runtime
* added direct proof for:
  * standalone predicate parsing
  * statement-level SQL parsing
  * SQL lowering parity onto the canonical casefolded prefix path
  * session execution parity against the established casefolded prefix behavior

Still intentionally excluded:

* non-prefix `ILIKE` / `NOT ILIKE` patterns
* field-to-field `ILIKE`
* grouped `HAVING ... ILIKE`
* broader SQL pattern-matching grammar

## 0.79.2 - 2026-04-14 - Add Small Predicate SQL Ergonomics

This patch keeps the `0.79` line on the same bounded predicate surface, but
adds a few small SQL quality-of-life widenings that reuse the existing
canonical predicate model instead of opening new runtime or planning families.

The shipped surface is:

* SQL:

  ```sql
  SELECT * FROM Character WHERE name NOT LIKE 'Al%';
  SELECT * FROM Character WHERE strength <> dexterity;
  SELECT * FROM Character WHERE strength IN (1, 2, 3,);
  SELECT * FROM Character WHERE active IS TRUE;
  SELECT * FROM Character WHERE active IS FALSE;
  ```

What changed:

* widened the bounded prefix-LIKE seam to admit `NOT LIKE 'prefix%'`
* kept that lowering intentionally small:
  * `name LIKE 'Al%'` stays `StartsWith("Al")`
  * `name NOT LIKE 'Al%'` stays `Not(StartsWith("Al"))`
* kept `NOT LIKE` on the same accepted operand family as the existing prefix
  LIKE surface:
  * plain field
  * `LOWER(field)`
  * `UPPER(field)`
* added direct proof that `<>` is accepted as a SQL alias for `!=` and lowers to
  the same canonical `Ne` compare path
* widened reduced SQL list parsing to tolerate one trailing comma in
  `IN (...)` and `NOT IN (...)`
* kept list parsing fail-closed for:
  * empty lists
  * double commas
  * broader tuple/list grammar widening
* widened plain-field `IS` parsing to admit `IS TRUE` and `IS FALSE`
* kept `IS TRUE` / `IS FALSE` deliberately narrow:
  * plain fields only
  * same canonical strict boolean equality path as `field = true/false`
  * no grouped `HAVING ... IS TRUE/FALSE`
  * no `IS NOT TRUE` / `IS NOT FALSE` in this slice
* kept all of the above on the existing predicate/parser/lowering/runtime seams
  instead of introducing new predicate leaf families

Still intentionally excluded:

* non-prefix `LIKE` / `NOT LIKE` patterns
* grouped `HAVING ... IS TRUE/FALSE`
* `IS NOT TRUE` / `IS NOT FALSE`
* field-to-field `NOT LIKE`
* arithmetic predicate expressions
* generic expression predicates

## 0.79.1 - 2026-04-13 - Add NOT BETWEEN

This patch widens the same bounded predicate surface slightly by admitting
`NOT BETWEEN` on the shared SQL and fluent predicate path, and by accepting
ergonomic symmetric compare spellings that normalize back onto the same
canonical field-first predicate forms.

The shipped surface is:

* SQL:

  ```sql
  SELECT * FROM Character WHERE strength NOT BETWEEN 10 AND 20;
  SELECT * FROM Character WHERE 5 < strength;
  SELECT * FROM Character WHERE dexterity = strength;
  ```

* fluent:

  ```rust
  db.load::<Character>().filter(field("strength").not_between(10, 20))
  ```

What changed:

* added one predicate-owned `not_between(...)` helper to the public predicate
  model as the direct dual of the existing `between(...)` lowering
* added one matching fluent builder helper,
  `field(\"...\").not_between(lower, upper)`
* widened the reduced SQL predicate parser to admit `field NOT BETWEEN lower AND upper`
  without creating a new runtime node family
* kept the canonical lowering deliberately small:
  * `BETWEEN` stays `field >= lower AND field <= upper`
  * `NOT BETWEEN` stays `field < lower OR field > upper`
* widened reduced SQL predicate parsing to accept literal-leading compare forms
  like `5 < strength`
* normalized those symmetric forms back onto the canonical field-first compare path:
  * `5 < strength` -> `strength > 5`
  * `5 <= strength` -> `strength >= 5`
  * `5 = strength` -> `strength = 5`
* added deterministic canonical ordering for symmetric field-to-field equality
  and inequality, so swapped field spellings lower to the same
  `CompareFields` representation
* kept the runtime/planner boundary unchanged:
  both `NOT BETWEEN` and symmetric compare forms stay on the existing
  predicate path instead of widening into generic predicate expressions or
  adding a new runtime shape
* added direct proof for:
  * standalone predicate parsing
  * SQL/fluent session parity
  * live canister-boundary execution
  * statement-level SQL parsing for symmetric compare forms
  * literal-leading mixed-type compare rejection after normalization

Still intentionally excluded:

* field-to-field `NOT BETWEEN`
* arithmetic bounds
* symmetric arithmetic predicates
* literal-leading `BETWEEN`
* grouped predicate-expression widening
* generic expression predicates

## 0.79.0 - 2026-04-13 - Add Field-To-Field Predicate Comparison

This patch opens the `0.79` line with one bounded predicate-expression slice:
single-entity field-to-field comparison in predicate position, on the shared
SQL and fluent predicate path.

The shipped surface is:

* SQL:

  ```sql
  SELECT * FROM Character WHERE strength > dexterity;
  SELECT * FROM Character WHERE strength = dexterity;
  ```

* fluent:

  ```rust
  db.load::<Character>().filter(field("strength").gt_field("dexterity"))
  db.load::<Character>().filter(field("strength").eq_field("dexterity"))
  ```

What changed:

* added one predicate-owned `CompareFields` leaf instead of widening the scalar
  projection expression system again
* lowered both SQL and fluent field-to-field comparisons to the same canonical
  predicate leaf before planning
* kept the executable compare runtime on one unified compare family with typed
  operands (`field slot` or `literal`) instead of maintaining separate
  field-vs-literal and field-vs-field runtime node families
* kept field-to-field comparison residual-only in planning, so literal/indexable
  clauses can still drive access choice while field-to-field checks remain
  row-filter work
* kept grouped query lanes fail-closed for this slice
* fixed mixed numeric equality and inequality normalization so field-to-field
  `=` / `!=` comparisons widen supported numeric pairs instead of failing strict
  coercion when the two field kinds differ (for example `Uint` vs `Int`)
* added semantic rejection coverage for the important fail-closed cases:
  incompatible type ordering, ordered boolean comparison, and missing
  right-hand fields
* locked the explain/debug surface so field-to-field predicates stay visible as
  `CompareFields { ... }` in the predicate tree instead of collapsing into
  opaque expression text
* locked the same-field edge case (`field = field`) as valid and stable
  residual compare behavior
* tightened the runtime after the first cut so scalar `field-slot` vs
  `field-slot` compares now stay on the scalar slot seam too; the compare model
  is unified and the remaining difference versus `field op literal` is only
  optimization eligibility, not a separate execution shape
* moved the local SQL dev shell from a shell-owned formatter plus example
  helper into one maintained Rust CLI crate rooted at
  `/home/adam/projects/icydb/crates/icydb-cli/src/main.rs`, with
  `scripts/dev/sql.sh` reduced to deploy/reset/init orchestration plus launcher
  wiring
* kept the shell output on the canonical Rust SQL render path, including boxed
  tables, `NULL` cell rendering, and footer-style row counts
* added perf-aware shell footer rendering through the canister-local
  `query_with_perf(sql)` surface, so interactive and one-shot shell queries now
  report instruction counts like `4 rows in set (7.8M instructions)` when the
  canister exposes the perf wrapper
* made the Rust shell handle semicolon-terminated multiline input, persistent
  history, blank-line reprompts, and top-level `\\q` / `quit` / `exit` without
  sending empty SQL into the engine
* added dedicated regressions for:
  * SQL/fluent runtime parity on the same field-to-field filter
  * mixed signed/unsigned equality on the live SQL session path
  * strict rejection of near-miss expression predicates like
    `strength = dexterity + 1`
  * live canister-boundary proof that a mixed literal plus field-to-field
    predicate (`age > 18 AND age > rank`) keeps the literal clause compatible
    with ordinary access planning while the field-to-field clause remains a
    residual filter

Still intentionally excluded:

* grouped field-to-field predicate execution
* arithmetic predicate expressions
* field-to-field `IN`, `CONTAINS`, `STARTS_WITH`, or other non-comparison forms
* planner attempts to rewrite field-to-field compares into indexable ranges
