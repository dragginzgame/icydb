# 0.99 Changelog

## 0.99.2

- `0.99.2` continues the symbolic prepared-template migration without widening the admitted `?` SQL surface. The prepared SQL line still only supports the existing compare-family subset from `0.98`, but more of that admitted fixed-route path now binds through slot-owned symbolic templates instead of sentinel-literal substitution.
- the symbolic scalar lane is now the normal owner for the admitted scalar compare-family path:
  - scalar `WHERE` compare-family predicates keep rebuilding from slot metadata
  - indexed secondary prefix lookups such as `WHERE name = ?` now stay on the symbolic scalar lane too
  - exact primary-key prepared lookups such as `WHERE id = ?` now bind their selected access payload symbolically as well
- the symbolic grouped lane remains in place for compare-family grouped `HAVING`:
  - grouped post-aggregate rebinding still comes from planner-owned grouped expressions
  - SQL execution still does not re-derive grouped semantics locally
  - branch-ownership invariants remained green while this lane widened around it
- `0.99.2` also tightens admission on the symbolic scalar lane:
  - partially-symbolified scalar predicates are now rejected back to fallback instead of silently reusing exemplar literals
  - this keeps symbolic template selection honest when a scalar plan still carries runtime-bound values that the current symbolic slice did not actually lift into slot ownership
- one candidate widening was intentionally not shipped:
  - symbolic secondary `IndexRange` rebinding was attempted for prepared queries like `WHERE name >= ? AND name < ?`
  - live testing showed that route was not yet correct on the symbolic lane
  - the symbolic range claim was removed instead of freezing a broken path
  - the shipped boundary is now explicit: secondary index range prepared queries still prepare and execute correctly, but they remain on the legacy prepared-template lane for now
- the access-boundary and SQL branch-ownership contracts remained clean throughout this follow-through:
  - no new raw `AccessPath::...` matching leaked back into the session layer
  - grouped and aggregate semantic decisions stayed lowering-owned
- focused verification for `0.99.2` covered the widened symbolic access slices, the explicit secondary-range fallback boundary, and the relevant invariants:
  - `cargo fmt --all`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`
  - `bash scripts/ci/check-sql-branch-ownership-invariants.sh`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_indexed_scalar_compare_contracts -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_primary_key_lookup_contracts -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_indexed_range_contracts_preserve_fallback_results -- --nocapture`
  - `cargo test -p icydb-core runtime_raw_access_path_references_stay_within_access_boundary -- --nocapture`

## 0.99.1

- `0.99.1` closes one reduced-SQL ordered-compare coercion gap for float-backed fields. Decimal-looking SQL literals such as `0.20` still parse through the shared decimal literal path, but ordered comparisons against `Float64` fields now canonicalize onto the numeric-widen compare contract instead of being preserved as strict literal matches and then failing schema validation.
- the bug lived in the bool-expression predicate bridge, which was still treating ordered `Decimal` / `Float32` / `Float64` literal compares as `Strict` on one canonicalization seam even though the parser/lowering surface already models ordered numeric SQL compares through `NumericWiden`
- this keeps the existing stricter behavior for the other literal families that still rely on it, and narrows the change to the ordered compare operators:
  - `<`
  - `<=`
  - `>`
  - `>=`
- shipped regression coverage now locks the fix at three levels:
  - bool-expression predicate canonicalization
  - reduced SQL lowering vs fluent intent parity
  - live session SQL execution over one `Float64` fixture entity
- focused verification for `0.99.1` covered the direct coercion regression and the end-to-end SQL path:
  - `cargo fmt --all`
  - `cargo test -p icydb-core predicate_bridge_promotes_ordered_decimal_literal_compares_to_numeric_widen`
  - `cargo test -p icydb-core sql_query_lowering_float_field_decimal_literal_order_compare_matches_fluent_intent -- --nocapture`
  - `cargo test -p icydb-core execute_sql_scalar_float_field_decimal_literal_order_compare_matches_expected_rows -- --nocapture`

## 0.99.0

- `0.99.0` starts the symbolic prepared-template line by replacing the first internal sentinel-backed template routes with slot-owned symbolic templates. This does not widen the admitted `?` SQL surface; it changes how the already-supported fixed-route prepared lane represents and rebinds values internally.
- scalar compare-family prepared queries are now on a symbolic template lane:
  - admitted scalar `WHERE` compare-family shapes no longer depend on collision-resistant sentinel literals during execution
  - the symbolic scalar template keeps planner-owned compare metadata and re-instantiates predicate trees directly from slot indexes at execute time
  - the first scalar symbolic slice covers:
    - single compare leaves
    - `AND`
    - `OR`
    - `NOT`
  - this is the slice that finally removes the old small-domain bool blocker from the scalar template path
- grouped compare-family `HAVING` prepared queries are now on a symbolic grouped lane too:
  - grouped prepared execution still freezes one exemplar grouped plan for routing and projection
  - post-aggregate `HAVING` expressions now rebuild bound literal leaves from slot identity instead of sentinel replacement
  - the shipped grouped symbolic slice is intentionally narrower than a full grouped symbolic rewrite:
    - compare-family `HAVING` only
    - no grouped scalar predicate in the same symbolic grouped lane
    - full-scan grouped access only for this first grouped slice
- `0.99.0` also takes the first symbolic access-payload step:
  - the symbolic scalar lane can now own one real planner-selected access payload shape:
    - single-path `IndexPrefix`
  - prepared indexed scalar queries like `WHERE name = ?` on the indexed session fixture no longer need the legacy sentinel-era `plan.access.bind_runtime_values(...)` path
  - this is the first proof that symbolic slots now own both:
    - logical predicate rebinding
    - selected access payload rebinding
  - broader access-path payload families still stay on the legacy fallback lane for now
- the legacy sentinel-backed prepared template lane still exists in `0.99.0`, but its role is reduced:
  - it now mainly covers access-payload-dependent shapes that have not yet moved onto symbolic slot ownership
  - scalar and grouped compare-family logical binding no longer depend on it on the main admitted prepared-query path
- the prepared-query safety model remains fail-closed:
  - unsupported symbolic access shapes fall back to the legacy prepared lane instead of widening or changing the SQL surface
  - existing SQL branch-ownership rules still hold; grouped `HAVING` symbolic binding was derived from planner-owned grouped expressions rather than by re-deriving grouped semantics from the SQL surface at execution time
- focused verification for `0.99.0` covered the new symbolic scalar lane, grouped symbolic lane, the first symbolic indexed access slice, and normal invariant hygiene:
  - `cargo fmt --all`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`
  - `bash scripts/ci/check-sql-branch-ownership-invariants.sh`
  - `cargo test -p icydb-core execute_prepared_sql_query_bool_compare_contracts_preserve_fallback_results -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_mixed_numeric_and_text_compare_contracts -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_mixed_or_compare_contracts -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_grouped_mixed_having_compare_contracts -- --nocapture`
  - `cargo test -p icydb-core execute_prepared_sql_query_templates_indexed_scalar_compare_contracts -- --nocapture`
