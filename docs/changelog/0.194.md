# IcyDB 0.194 Changelog

## 0.194.0

### 📝 Summary

- Starts the query performance optimization line with measurement and
  attribution cleanup rather than speculative runtime changes.
- Splits the wasm-footprint audit matrix so metrics-off runtime fixtures and
  the generated compact metrics endpoint are measured separately.
- Captures broad deterministic and seeded-random SQL query hotspot baselines
  and verifies the existing fluent and curated SQL perf harnesses still run.

### 📊 Audit

- Added an explicit `default_empty_metrics` wasm audit fixture so the compact
  generated metrics endpoint cost is isolated from the zero-export runtime
  floor.
- Kept `default_empty`, one-entity fluent rows, one-entity fluent execute,
  one-entity SQL query, and ten-entity fluent rows as the default small
  footprint attribution matrix.
- Kept `sql_perf` registered and buildable as the broad instruction-sampling
  canister while documenting why it is deliberately excluded from the default
  wasm footprint matrix.
- Captured a deterministic SQL perf matrix with 1,755 generated scenarios,
  1,674 executed scenarios, and 81 expected `QueryPlan` failures.
- Captured a seeded-random SQL perf matrix with 500 generated scenarios, 486
  executed scenarios, and 14 expected `QueryPlan` failures.
- Wrote a compact local hotspot summary that groups raw matrix output by
  scenario family and phase so optimization work can start from ranked
  evidence instead of individual spot checks.

### 🔧 Optimization Notes

- The largest admitted query hotspot is sparse token `IN` routing over many
  missing collection IDs at roughly 34.9M local instructions, split mainly
  across planner route construction and executor merge/dedup work.
- The broadest repeated hot family is unindexed heap/journaled mirror scans
  ordered by non-route fields; small `LIMIT` values do not reduce the dominant
  full-scan and post-access ordering cost.
- The fluent harness confirms that branch-set token full-entity reads and wide
  branch sets are expensive compared with narrow projected reads.
- Curated SQL samples keep diagnostic metadata and EXPLAIN surfaces visible as
  high-cost admin/trusted paths rather than ordinary public read paths.
- Warm query-plan and SQL compile caches materially reduce repeated-call cost,
  so future optimization work should keep cold and warm results separate.

### 📚 Documentation

- Updated the recurring wasm-footprint audit guide with the current default
  canister set and each fixture's attribution role.
- Documented that `sql_perf` is a broad perf/access-shape fixture and should
  be measured explicitly when a scenario needs it, not treated as the default
  small wasm-footprint baseline.

### 🧪 Testing

- Verified all audit canister packages compile.
- Verified SQL audit canisters compile without default features.
- Rebuilt the full six-canister default wasm-size report.
- Staged the `sql_perf` fixture under the wasm-release profile.
- Verified generated build-config invariants, shell syntax, formatting, and
  whitespace validation.
- Ran the deterministic SQL matrix, seeded-random SQL matrix, fluent perf
  harness, and curated SQL perf harness with PocketIC.

### 🧭 Migration Notes

- No public API, generated endpoint contract, persisted format, cursor format,
  recovery behavior, response shape, diagnostic-code, or successful query
  semantics change is introduced.
- This slice establishes measurement baselines and fixture attribution for the
  optimization line; it does not yet apply runtime optimizer changes.
