# IcyDB 0.194 Changelog

## 0.194.11

### 📝 Summary

- Continues the 0.194 optimization line with a narrow indexed prefix/range
  lowering cleanup.
- Moves encoded component bytes directly into execution prefix specs,
  range-prefix specs, and direct-count cardinality specs instead of cloning or
  re-encoding intermediate batches.
- Keeps the slice internal: no query semantics, public API, cursor format,
  persisted index-key bytes, diagnostic code, planner route contract, or
  successful result shape changes.

### 🔧 Changed

- Single-component indexed `IN (...)` lowering now encodes one lookup value and
  moves the owned component bytes into the lowered execution spec.
- Direct-count cardinality specs for single-component indexed `IN (...)` now
  reuse the same owned encoded component bytes instead of cloning from a batch.
- Index range lowering now returns raw bounds plus the equality-prefix bytes
  produced by the same canonical range-bound pass, avoiding a second prefix
  encoding step for prefixed range scans.
- Bounds-only callers keep the existing shape and do not build execution prefix
  byte vectors when they only need cursor/range bounds.

### 📊 Performance

- Sparse indexed `IN (...)` count dropped from 4,060,408 to 3,811,058 local
  instructions (`-249,350`, `-6.14%`) against the 0.194.10 canary baseline.
- The same count canary's plan-build bucket dropped from 1,049,837 to 804,412
  local instructions.
- Sparse indexed `IN (...)` page dropped from 17,961,141 to 17,760,438 local
  instructions (`-200,703`, `-1.12%`) against the 0.194.10 canary baseline.
- The same page canary's plan-build bucket dropped from 8,352,272 to 8,117,535
  local instructions.
- The indexed range regression canary stayed noise-level flat against the
  latest comparable full-matrix baseline (`+9,352`, `+0.04%`).

### 📊 Complexity Delta

- Code footprint: six files touched at release-prep time (`+164/-38`).
- Runtime complexity judgment: lower on the hot path. The slice centralizes
  byte ownership at the canonical lowering boundary and removes duplicate
  clone/re-encode work instead of adding a new planner route.
- Test footprint: one new access-layer boundary test pins that prefixed range
  lowering carries the canonical equality-prefix bytes used by execution.

### 🧪 Testing

- Re-ran access-lowering boundary tests, including the new prefixed range
  prefix-byte regression.
- Re-ran canonical index range codec witnesses.
- Re-ran cursor tests to guard bounds-only callers.
- Re-ran `icydb-core` all-target/all-feature clippy with warnings denied.
- Re-ran a three-scenario PocketIC SQL performance matrix for sparse indexed
  `IN (...)` count, sparse indexed `IN (...)` page, and the indexed range
  regression canary.

## 0.194.3

### 📝 Summary

- Hard-cuts the fluent response helper names that were easy to misread in
  downstream application code.
- Renames optional single-entity materialization from `one_opt()` to
  `try_one()` so the method name matches the existing `try_row()`,
  `try_entity()`, and `try_id()` convention.
- Aligns identity materialization so `id()` means "the one id",
  `try_id()` means "zero or one id", and `ids()` means "all ids".
- Adds a host-side CLI diagnostic lookup command so compact public errors like
  `E7` and `E190` can be decoded without keeping rich error strings in canister
  wasm.

### 🔧 Changed

- Removed `FluentLoadQuery::one_opt()` from the public facade.
- Added `FluentLoadQuery::try_one()` as the optional single-entity
  convenience alias over `try_entity()`.
- Changed fluent `id()` from "first id if any" to exact-cardinality identity
  materialization.
- Removed facade `require_id()` naming in favor of `id()` for value-returning
  identity materialization.
- Renamed the core sealed `ResponseCardinalityExt::require_id()` helper to
  `id()` so the public facade and core cardinality trait use the same naming
  model.
- Added `icydb diagnostic <CODE>` with support for compact code inputs such as
  `E7`, `7`, `E190`, and `190`.
- The diagnostic command renders whether the code is known, its broad class,
  its default origin, the stable symbolic diagnostic label, and any registered
  fix guidance.

### 🧭 Migration Notes

Use this naming model for fluent query materialization:

| Wanted result | Required | Optional zero-or-one | All |
| --- | --- | --- | --- |
| Row | `row()` | `try_row()` | `rows()` |
| Entity | `entity()` / `one()` | `try_entity()` / `try_one()` | `entities()` / `all()` |
| Id | `id()` | `try_id()` | `ids()` |

- Replace `.one_opt()` with `.try_one()`.
- Replace `.require_id()` with `.id()`.
- If code used fluent `.id()` for "first id if any", switch to `first()` for
  response-order first-id semantics or `try_id()` for zero-or-one cardinality.
- `require_one()` and `require_some()` remain validation-only helpers that
  return `()`.
- Use `icydb diagnostic E7` when a canister or frontend only reports the compact
  public error string. `E7` maps to `E_QUERY_NOT_FOUND`; `E190` maps to
  `E_QUERY_READ_ADMISSION` with read-admission fix guidance.
