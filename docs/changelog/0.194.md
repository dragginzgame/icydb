# IcyDB 0.194 Changelog

## 0.194.2

### 📝 Summary

- Continues the query optimization line by reducing cold-plan and row-decode
  overhead in the indexed `IN`, SQL covering-read, retained-slot, and
  persisted-row decode paths.
- Keeps the work as an internal hygiene/performance slice: no public API,
  query semantics, cursor format, persisted format, or diagnostic code changes.

### 🔧 Changed

- Borrow multi-lookup and branch-set access-bound value slices during
  residual-filter proof instead of rebuilding equivalent list values.
- Reuse prepared lowered index-prefix/range specs for SQL covering projection
  runtime preparation instead of lowering the same access plan a second time.
- Let direct persisted-row decode and primary-key/slot decode borrow the
  selected structural row contract through the scan boundary.
- Let `StructuralSlotReader` borrow a row-shape contract when callers already
  own the contract, while preserving owned-reader construction for call sites
  that build a contract and move the reader across a boundary.
- Use single-threaded sharing for retained-slot layouts and prepared grouped
  runtime residents so single-threaded IC runtime data does not pay atomic
  reference-counting overhead.

### 📊 Performance

- The targeted three-query PocketIC rerun completed with 3 executed scenarios
  and 0 failures after running outside the local-network sandbox.
- Compared with the prepared-covering baseline, the branch-set token query
  improved by roughly 0.47% total instructions and 0.97% executor
  instructions.
- The wide user covering read improved by roughly 0.47% total instructions and
  0.69% execute instructions.
- The blob primary-key read improved by roughly 0.14% total instructions and
  1.23% kernel row-read instructions.

### 🧪 Testing

- Re-ran focused persisted-row and retained-slot tests.
- Re-ran `icydb-core` no-default check, `icydb-core` clippy with warnings as
  errors, formatting checks, whitespace validation, and invariant checks.
- Re-ran the targeted PocketIC SQL perf sample for the three scenarios touched
  by this slice.

### 🧭 Migration Notes

- No public API, generated endpoint contract, persisted format, cursor format,
  recovery behavior, response shape, diagnostic-code, or successful query
  semantics change is introduced.
- The ownership changes are internal runtime representation cleanup only.

## 0.194.1

### 📝 Summary

- Starts runtime optimization work from the 0.194.0 hotspot evidence by
  reducing the sparse indexed `IN` page-query path.
- Keeps the optimization local to prefix-family discovery, access lowering,
  and bounded materialized prefix-family execution; no public query semantics
  or persisted formats change.
- Tightens `significant_drop_tightening` cleanup sites so scoped guard
  lifetimes are explicit in commit, build, and schema validation code.

### 🔧 Changed

- Batch sparse multi-lookup child-prefix expansion through synchronized
  index-prefix cardinality metadata instead of probing one parent prefix at a
  time.
- Lower single-component indexed multi-lookups from pre-encoded values so large
  `IN` lists avoid repeated value encoding and repeated scan-contract string
  allocation.
- Share exact-prefix bound construction through index-owned encoded-prefix
  helpers, keeping range-bound authority in the index module.
- Add a guarded materialized prefix-family fast path for bounded page reads
  when the request has no cursor boundary, uses primary-key suffix resume
  semantics, and stays below active-prefix and scanned-key caps.

### 📊 Performance

- The deterministic sparse token collection `IN` page query improved from
  34.85M to 31.22M local instructions in the targeted PocketIC matrix run.
- The same query's execute phase dropped by roughly 11.2%, planner
  plan-build work by roughly 8.3%, store instructions by roughly 20.2%, and
  executor instructions by roughly 13.1%.
- The full deterministic matrix retained the same generated/executed/failure
  shape as the 0.194.0 baseline and showed no scenario above a 3% total
  regression threshold.

### 🧪 Testing

- Added access-lowering parity tests that compare multi-lookup lowered prefix
  bounds against individual prefix lowering and canonical exact-prefix bounds.
- Added index-prefix-cardinality batch coverage for sparse child-prefix
  discovery.
- Re-ran targeted access/cardinality tests, core clippy, the full `make
  clippy` validation path, and the targeted sparse `IN` PocketIC perf
  scenario.
- Re-ran the full deterministic PocketIC SQL perf matrix after the
  optimization slice.

### 🧭 Migration Notes

- No public API, generated endpoint contract, persisted format, cursor format,
  recovery behavior, response shape, diagnostic-code, or successful query
  semantics change is introduced.
- The bounded materialized prefix-family path is an internal executor
  optimization and falls back to the existing merge path outside its narrow
  guard conditions.

## 0.194.0

### 📝 Summary

- Starts the query performance optimization line with measurement and
  attribution cleanup rather than speculative runtime changes.
- Splits the wasm-footprint audit matrix so metrics-off runtime fixtures and
  the generated compact metrics endpoint are measured separately.
- Captures broad deterministic and seeded-random SQL query hotspot baselines
  and verifies the existing fluent and curated SQL perf harnesses still run.

### 📊 Audit

- Added an explicit `default_empty_metrics` wasm audit fixture so the compact
  generated metrics endpoint cost is isolated from the zero-export runtime
  floor.
- Kept `default_empty`, one-entity fluent rows, one-entity fluent execute,
  one-entity SQL query, and ten-entity fluent rows as the default small
  footprint attribution matrix.
- Kept `sql_perf` registered and buildable as the broad instruction-sampling
  canister while documenting why it is deliberately excluded from the default
  wasm footprint matrix.
- Captured a deterministic SQL perf matrix with 1,755 generated scenarios,
  1,674 executed scenarios, and 81 expected `QueryPlan` failures.
- Captured a seeded-random SQL perf matrix with 500 generated scenarios, 486
  executed scenarios, and 14 expected `QueryPlan` failures.
- Wrote a compact local hotspot summary that groups raw matrix output by
  scenario family and phase so optimization work can start from ranked
  evidence instead of individual spot checks.

### 🔧 Optimization Notes

- The largest admitted query hotspot is sparse token `IN` routing over many
  missing collection IDs at roughly 34.9M local instructions, split mainly
  across planner route construction and executor merge/dedup work.
- The broadest repeated hot family is unindexed heap/journaled mirror scans
  ordered by non-route fields; small `LIMIT` values do not reduce the dominant
  full-scan and post-access ordering cost.
- The fluent harness confirms that branch-set token full-entity reads and wide
  branch sets are expensive compared with narrow projected reads.
- Curated SQL samples keep diagnostic metadata and EXPLAIN surfaces visible as
  high-cost admin/trusted paths rather than ordinary public read paths.
- Warm query-plan and SQL compile caches materially reduce repeated-call cost,
  so future optimization work should keep cold and warm results separate.

### 📚 Documentation

- Updated the recurring wasm-footprint audit guide with the current default
  canister set and each fixture's attribution role.
- Documented that `sql_perf` is a broad perf/access-shape fixture and should
  be measured explicitly when a scenario needs it, not treated as the default
  small wasm-footprint baseline.

### 🧪 Testing

- Verified all audit canister packages compile.
- Verified SQL audit canisters compile without default features.
- Rebuilt the full six-canister default wasm-size report.
- Staged the `sql_perf` fixture under the wasm-release profile.
- Verified generated build-config invariants, shell syntax, formatting, and
  whitespace validation.
- Ran the deterministic SQL matrix, seeded-random SQL matrix, fluent perf
  harness, and curated SQL perf harness with PocketIC.

### 🧭 Migration Notes

- No public API, generated endpoint contract, persisted format, cursor format,
  recovery behavior, response shape, diagnostic-code, or successful query
  semantics change is introduced.
- This slice establishes measurement baselines and fixture attribution for the
  optimization line; it does not yet apply runtime optimizer changes.
