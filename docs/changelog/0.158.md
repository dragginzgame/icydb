# 0.158.x

## 0.158.7

### 🔧 Changed

- `ALTER TABLE ... ALTER COLUMN ... SET DEFAULT` now publishes metadata-only
  accepted schema changes for DDL-owned fields when the SQL literal can be
  encoded for the accepted field contract.
- `ALTER TABLE ... ALTER COLUMN ... DROP DEFAULT` now publishes metadata-only
  accepted schema changes for nullable DDL-owned fields.
- Matching default changes now report `no_op` instead of publishing redundant
  accepted snapshots.
- Invalid default literals reject without changing accepted schema visibility.
- Generated accepted fields reject SQL default changes and remain owned by the
  Rust schema.
- `DROP DEFAULT` on required defaulted fields remains fail-closed, because
  historical rows may still depend on the accepted default to materialize
  missing slots.
- DDL reports now expose `set_field_default` and `drop_field_default` mutation
  kinds.

```
ALTER TABLE Character ADD COLUMN score nat;
ALTER TABLE Character ALTER COLUMN score SET DEFAULT 7;
ALTER TABLE Character ALTER COLUMN score SET DEFAULT 7;
ALTER TABLE Character ADD COLUMN nickname text DEFAULT 'anonymous';
ALTER TABLE Character ALTER COLUMN nickname DROP DEFAULT;
DESCRIBE Character;
```

### 🧪 Validation

- Added binder coverage for executable `SET DEFAULT` and `DROP DEFAULT` no-op
  handling.
- Added session execution coverage for `SET DEFAULT`, nullable `DROP DEFAULT`,
  matching-default no-op handling, invalid-default rejection, and
  required-field `DROP DEFAULT` rejection.
- Added canister endpoint coverage for published default changes and
  unsupported nullability/default removals.

## 0.158.6

ALTER TABLE Character ALTER COLUMN score DROP DEFAULT;
ALTER TABLE Character ALTER COLUMN score SET NOT NULL;

### 🧪 Validation

- Added binder coverage proving accepted catalog validation happens before the

- Added binder coverage for unknown accepted columns.
- Added canister endpoint coverage proving rejected `ALTER COLUMN` statements

## 0.158.5

### 🔧 Changed

- `icydb schema check` now prints targeted recommendations for generated-only
  fields/indexes, accepted-only generated fields/indexes, default drift,
  nullability drift, and same-name index contract drift.
- Generated-only fields recommend an accepted additive transition before
  deployment.
- Accepted-only generated fields recommend waiting for an explicit retained-slot
  removal policy instead of treating the drift as valid DDL-owned catalog
  state.
- Default and nullability mismatches recommend explicit future
  `ALTER COLUMN SET/DROP DEFAULT` and `ALTER COLUMN SET/DROP NOT NULL` flows.
- Generated-only, accepted-only generated, and same-name-different-contract
  indexes recommend explicit publication, removal/restoration, or replacement
  rather than falling through to the generic mismatch text.

```
icydb schema check demo_rpg -e demo
```

### 🧪 Validation

- Added schema-check rendering coverage for generated-only field
  recommendations.
- Added schema-check rendering coverage for accepted-only generated-field
  recommendations.
- Added schema-check rendering coverage for default and nullability drift
  recommendations.
- Added schema-check rendering coverage for generated-only, accepted-only
  generated, and same-name-different-contract index recommendations.

## 0.158.3

### 🔧 Changed

- `ALTER TABLE ... ADD COLUMN ...` now publishes accepted schema metadata for
  nullable no-default columns and supported defaulted columns.
- SQL-added columns are stored as DDL-owned accepted fields, with a durable
  field ID, accepted row slot, persisted kind, nullability/default metadata,
  and `origin=ddl`.
- DDL publication remains metadata-only for this slice; existing shorter rows
  materialize missing accepted slots through accepted-schema absence policy:
  `NULL` for nullable additions and encoded database defaults for defaulted
  additions.
- Generated-compatible runtime schema checks now allow trailing DDL-owned
  accepted fields that have a safe missing-field policy.
- Typed writes after field DDL now materialize omitted DDL-owned slots through
  accepted schema policy instead of generated Rust construction.

```sql
ALTER TABLE Character ADD COLUMN nickname text;
ALTER TABLE Character ADD COLUMN score nat DEFAULT 7 NOT NULL;
SHOW COLUMNS Character;
DESCRIBE Character;
SELECT name, score FROM Character;
```

### 🛡️ Guardrails

- `ALTER TABLE ... ADD COLUMN ... NOT NULL` without a default still rejects
  before accepted schema publication.
- Defaults that cannot encode through the accepted field contract reject before
  accepted schema publication.
- This patch keeps the supported SQL-added scalar type set narrow: `text`,
  `bool`, `int`, and `nat`.

### 🧪 Validation

- Added binding coverage for nullable SQL-added fields as accepted DDL-owned
  metadata.
- Added binding coverage for defaulted SQL-added fields as accepted DDL-owned
  metadata.
- Added execution coverage proving nullable `ADD COLUMN` publishes accepted
  schema metadata and appears in `SHOW COLUMNS`.
- Added execution coverage proving defaulted `ADD COLUMN` publishes accepted
  schema metadata and materializes omitted values from the accepted default.
- Added rejection coverage for unencodable SQL-added defaults and `NOT NULL`
  fields without defaults.

## 0.158.2

### 🔧 Changed

- Hard-cut schema defaults to one authoring concept before executable field DDL
  expands.
- In schema macros, `default = ...` is now the single schema/database default
  syntax.
- The Rust schema macro is now documented as the schema authoring frontend:
  generated Rust types are a typed access layer, and generated Rust `Default`
  is only construction convenience.
- Removed `db_default = ...` with no pre-1.0 compatibility shim for new macro
  source.
- Identity-like `Type::default()` constructors, including
  `Principal::default()`, `Account::default()`, `Subaccount::default()`, and
  `Ulid::default()`, are no longer valid schema/database defaults. `Principal`,
  `Account`, and `Ulid` no longer implement Rust `Default`; `Subaccount` keeps
  its ordinary all-zero Rust default, but schema defaults still require an
  explicit constructor or persisted literal when that value is intentional.
- Rust `Default` is no longer a separate schema-default authority. Generated
  Rust construction may reuse explicit schema defaults, nullable fields, empty
  collections, and explicit generation rules, but database defaulting is
  controlled only by accepted schema metadata.
- SQL/MySQL column defaults are the defaulting analogy, with IcyDB-specific
  accepted-schema snapshots and row-slot mechanics remaining authoritative.
- Missing-field decode now materializes only from accepted nullability or an
  explicit accepted database default; required missing fields without an
  accepted default still fail closed.
- Schema-check default drift output now includes a short payload fingerprint so
  default mismatches are readable without comparing raw bytes.

### 🧪 Validation

- Added macro and compile-fail coverage for the new default contract.
- Kept missing-field, additive-field, default-drift, and SQL DDL parser/binder
  coverage on the accepted-schema path.
- Added schema-check rendering coverage for readable default mismatches.

### 🛡️ Guardrails

- Existing accepted schema snapshots that already contain encoded database
  defaults continue to mean database defaults.
- This patch removes the old authoring split; it does not add executable SQL
  `ALTER TABLE ... ADD COLUMN` publication yet.

## 0.158.1

### 🔧 Changed

- Added `__icydb_schema_check` for canisters with `schema.enabled = true`.
- Changed the CLI schema surface into an explicit command family:
  `schema show` reads accepted metadata, and `schema check` compares generated
  proposal metadata with the accepted live catalog.
- `schema check` reports accepted-only fields, DDL-owned accepted indexes, and
  generated-vs-accepted mismatches so field DDL work has a clear drift
  inspection surface before executable `ADD COLUMN` lands.
- Added an accepted field-origin contract to persisted schema snapshots.
- Generated fields now persist as `generated`; future SQL-added fields can be
  persisted as `ddl` instead of appearing as unclassified accepted drift.
- Schema transition policy now preserves accepted DDL-owned fields that are
  absent from the generated Rust model, while still rejecting accepted-only
  generated fields as unsafe drift.
- Schema describe payloads now expose field origin, and rendered `DESCRIBE` /
  `SHOW COLUMNS` tables include an `origin` column.

```bash
icydb schema show demo_rpg
icydb schema check demo_rpg
icydb schema check demo_rpg -e demo
```

```sql
DESCRIBE Character;
SHOW COLUMNS Character;
```

### 🛡️ Guardrails

- Schema check treats `accepted-only origin=ddl` as expected DDL drift and
  keeps `accepted-only origin=generated` as unsafe drift.
- Field-origin metadata and reconciliation preservation landed before
  executable field DDL so DDL-owned columns can survive generated schema
  reconciliation.

### 🧪 Validation

- Added CLI parsing and rendering coverage for the new schema-check command.
- Added endpoint-name coverage for the generated `__icydb_schema_check`
  surface.
- Added persisted-schema codec coverage proving generated and DDL field origins
  round-trip through accepted schema snapshots.
- Added transition-policy coverage proving DDL-owned accepted fields remain
  compatible with generated schema metadata.
- Updated describe, CLI schema report, and SQL table-renderer coverage for the
  new field-origin payload.

## 0.158.0

### 🔧 Changed

- Started the schema-DDL line after the 0.157 index-DDL closeout.
- SQL now parses `ALTER TABLE ... ADD COLUMN ...` into the DDL frontend model,
  including nullable/default intent.
- `ALTER TABLE ... ADD COLUMN ...` now routes through the DDL surface and binds
  the target accepted entity before failing closed with an explicit unsupported
  schema-DDL diagnostic.
- The CLI SQL shell now routes `ALTER TABLE ...` statements to the generated
  DDL update endpoint instead of the readonly query endpoint.

```sql
ALTER TABLE Character ADD COLUMN nickname text;
ALTER TABLE Character ADD COLUMN score nat DEFAULT 0 NOT NULL;
```

### 🛡️ Guardrails

- `ALTER TABLE ... ADD COLUMN ...` does not publish accepted schema metadata or
  rewrite rows in this patch.
- Field DDL remains fail-closed until accepted field-transition publication,
  missing-slot decode policy, and write encoding are wired end to end.

### 🧪 Validation

- Added parser coverage for additive column DDL intent, nullable flags, and
  default literals.
- Added DDL binding coverage proving additive column DDL validates the target
  entity before returning the unsupported schema-DDL diagnostic.
- Added CLI shell routing coverage for `ALTER TABLE ... ADD COLUMN ...`.
