# IcyDB 0.164 — Composite Relation Targets

## 0.164.8

### 🔧 Changed

- Moved save-time strong-relation validation for accepted relation edges onto
  ordered local component tuples, lowering source field values directly into
  `PrimaryKeyValue::Scalar` or `PrimaryKeyValue::Composite` before target row
  membership checks.
- Moved reverse-index mutation preparation for accepted relation edges onto the
  same ordered local component metadata, decoding component field bytes from
  accepted row contracts and preserving full target tuple identity.
- Kept existing scalar, list, and set relation fields on their established
  relation-field payload path while explicit relation-edge metadata can use
  tuple decoding.
- Rejected relation-edge tuples whose local component fields mix required and
  optional cardinality, keeping runtime all-or-none null handling fail-closed.

### 🧪 Testing

- Added executor coverage for existing composite target tuples, missing target
  tuples, and delete blocking through reverse-index membership.
- Added schema and derive validation coverage for mixed required/optional
  relation tuple components.
- Re-ran focused relation and reverse-index suites, including write-boundary
  guards for accepted relation contracts.

## 0.164.7

### 🔧 Changed

- Added source-local relation-edge contracts to persisted accepted schema
  snapshots, including relation name, target entity path, and ordered local
  field IDs.
- Updated the persisted schema snapshot codec to carry relation-edge metadata
  under a new hard-cut wire version.
- Projected generated `RelationEdgeModel` proposal rows into initial persisted
  relation-edge snapshots during schema proposal compilation.
- Carried accepted relation-edge rows through `AcceptedRowDecodeContract` so
  save validation and reverse-index preparation can prefer persisted edge
  metadata when it exists.
- Preserved existing scalar relation-field behavior for declarations that have
  not yet moved through explicit relation-edge metadata.

### 🧪 Testing

- Added codec, proposal, snapshot integrity, runtime descriptor, and
  reverse-index coverage for accepted relation-edge metadata.
- Re-ran the focused relation suite to prove existing scalar relation
  validation, reverse-index updates, and recovery behavior remain intact.

## 0.164.6

### 🔧 Changed

- Moved accepted-schema `DESCRIBE` relation rows onto persisted relation field
  authority instead of generated relation descriptors. Generated DESCRIBE still
  uses generated relation metadata for proposal/model-only output.
- Hardened save-time strong-relation preflight so scalar relation fields verify
  the accepted target primary key is exactly one matching scalar kind before
  relation membership checks build a target data key.
- Renamed the shared accepted relation descriptor field to
  `scalar_target_key_kind` so current scalar relation lanes are not confused
  with future full target tuple identity.

### 🧹 Cleanup

- Removed stale `storage-key` wording from relation primary-key-component
  structural codec comments and diagnostics.

### 🧪 Testing

- Added accepted DESCRIBE coverage proving relation rows come from persisted
  accepted field metadata.
- Added save-time preflight guard coverage for composite target PK rejection
  and relation/target key-kind drift.

## 0.164.5

### 🔧 Changed

- Added strict scalar `field = "..."` shorthand alongside `fields = [...]` for
  derive primary-key and index declarations.
- Kept both scalar shorthand and list syntax normalized to the same ordered
  field-list model.
- Kept `field` and `fields` mutually exclusive and kept comma-string field
  lists rejected.

### 🧪 Testing

- Updated derive parser coverage for scalar shorthand, ordered field-list
  normalization, duplicate rejection, and malformed comma-string rejection.

## 0.164.4

### 🔧 Changed

- Centralized accepted relation primary-key component kind admissibility in one
  relation-domain gate shared by save validation and reverse-index
  preparation.
- Added a schema-side scalar relation target descriptor that compares primitive
  kind plus scale, length, and byte bounds without treating scalar relation
  syntax as composite admission.
- Added an ordered local-component constructor for accepted relation metadata
  so current scalar relation declarations lower through the same tuple-capable
  boundary future composite relation edges need.
- Renamed the remaining scalar target primary-key validator to make its
  single-component contract explicit.

### 🧪 Testing

- Added schema relation validation coverage for scalar kind mismatches,
  matching scalar kinds, scalar descriptor bounds, and 128-bit scalar targets.
- Added accepted relation metadata coverage for ordered local component tuples
  and empty local component rejection.

## 0.164.3

### 🔧 Changed

- Centralized accepted relation-target descriptor extraction so save
  validation and reverse-index preparation consume one relation-domain
  metadata owner instead of duplicating tuple-shaped target parsing.
- Centralized accepted target identity validation so save validation and
  reverse-index preparation share one path for target path, name, tag, and
  store checks.
- Made relation cardinality explicit in the accepted target descriptor so
  direct, list, and set relation shapes no longer need to be re-matched by
  each runtime relation path.

### 🧹 Cleanup

- Removed the now-dead invalid relation field-kind error constructor that was
  only needed while relation runtime paths revalidated accepted field shape
  locally.

## 0.164.2

### 🔧 Changed

- Moved relation target extraction onto an explicit target-key domain instead
  of returning raw `Vec<PrimaryKeyValue>` values through the relation path.
- Made relation target cardinality explicit with none, one, and many
  constructors so scalar relation payloads and collection relation payloads do
  not communicate cardinality through vector length.
- Renamed relation target extraction helpers away from broad
  `primary_key_values` wording and toward target-key terminology.
- Renamed the remaining target primary-key helper from scalar wording to a
  single-component contract so composite relation work cannot mistake it for
  general target identity authority.

### 🧪 Testing

- Added relation-target domain coverage for none, one, and many key cases.

## 0.164.1

### 🩹 Fixed

- Fixed scalar `int128` / `nat128` relation target key validation so accepted
  strong-relation structural paths admit the same 128-bit key lanes as the
  relation payload codec.
- Kept `int128` / `nat128` relation payloads on the structural field-bytes
  decode path instead of sending them through the scalar-slot fast path.

### 🔧 Changed

- Moved accepted strong-relation source metadata onto ordered local component
  descriptors. Current scalar relation declarations still populate one local
  component, but the accepted descriptor now matches the composite-relation
  edge shape.
- Changed accepted strong-relation target metadata construction to accept an
  ordered primary-key component list. Current scalar relations still pass a
  one-component list, while composite target metadata can use the same target
  identity shape.
- Made relation target identity metadata fail closed when accepted target
  primary-key component metadata is missing.
- Added a relation metadata arity guard so local relation components must
  match the accepted target primary-key component count before scalar
  target-key lanes are used.

### 🧪 Testing

- Added schema-node coverage proving single-field relation declarations reject
  composite target primary keys even when the first target component has the
  same primitive kind as the relation field.
- Added relation metadata coverage for ordered local components and scalar
  128-bit key lanes.
- Added a guard regression proving a single local relation field cannot
  validate against composite target primary-key metadata.

## 0.164.0

### 🔧 Changed

- Started the composite relation target line by moving reverse-relation target
  identity onto complete `PrimaryKeyValue` encoding instead of scalar target
  components.
- Changed reverse-index storage for relation targets to encode the full target
  `EncodedPrimaryKey` as one system index component while preserving source
  identity as the row suffix.
- Added ordered target primary-key component metadata to accepted
  strong-relation descriptors. Existing scalar relation declarations populate a
  one-component descriptor; composite admission can extend the same shape.
- Moved relation target extraction from accepted source rows to return
  `PrimaryKeyValue` identities, keeping current scalar relation fields as
  `PrimaryKeyValue::Scalar` without exposing a scalar-only target API.

### 🧪 Testing

- Added focused relation tests proving reverse-relation keys and bounds can
  carry complete composite target identity.
- Updated write-boundary guards so relation runtime paths assert accepted
  contracts and `PrimaryKeyValue` identity boundaries instead of scalar target
  component boundaries.

### 🧭 Notes

- Composite relation target declarations are not admitted yet in `0.164.0`.
  This patch prepares the runtime identity and metadata boundary first.
