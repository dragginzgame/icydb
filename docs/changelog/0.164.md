# IcyDB 0.164 — Composite Relation Targets

## 0.164.2

### 🔧 Changed

- Moved relation target extraction onto an explicit target-key domain instead
  of returning raw `Vec<PrimaryKeyValue>` values through the relation path.
- Made relation target cardinality explicit with none, one, and many
  constructors so scalar relation payloads and collection relation payloads do
  not communicate cardinality through vector length.
- Renamed relation target extraction helpers away from broad
  `primary_key_values` wording and toward target-key terminology.
- Renamed the remaining target primary-key helper from scalar wording to a
  single-component contract so composite relation work cannot mistake it for
  general target identity authority.

### 🧪 Testing

- Added relation-target domain coverage for none, one, and many key cases.

## 0.164.1

### 🩹 Fixed

- Fixed scalar `int128` / `nat128` relation target key validation so accepted
  strong-relation structural paths admit the same 128-bit key lanes as the
  relation payload codec.
- Kept `int128` / `nat128` relation payloads on the structural field-bytes
  decode path instead of sending them through the scalar-slot fast path.

### 🔧 Changed

- Moved accepted strong-relation source metadata onto ordered local component
  descriptors. Current scalar relation declarations still populate one local
  component, but the accepted descriptor now matches the composite-relation
  edge shape.
- Changed accepted strong-relation target metadata construction to accept an
  ordered primary-key component list. Current scalar relations still pass a
  one-component list, while composite target metadata can use the same target
  identity shape.
- Made relation target identity metadata fail closed when accepted target
  primary-key component metadata is missing.
- Added a relation metadata arity guard so local relation components must
  match the accepted target primary-key component count before scalar
  target-key lanes are used.

### 🧪 Testing

- Added schema-node coverage proving single-field relation declarations reject
  composite target primary keys even when the first target component has the
  same primitive kind as the relation field.
- Added relation metadata coverage for ordered local components and scalar
  128-bit key lanes.
- Added a guard regression proving a single local relation field cannot
  validate against composite target primary-key metadata.

## 0.164.0

### 🔧 Changed

- Started the composite relation target line by moving reverse-relation target
  identity onto complete `PrimaryKeyValue` encoding instead of scalar target
  components.
- Changed reverse-index storage for relation targets to encode the full target
  `EncodedPrimaryKey` as one system index component while preserving source
  identity as the row suffix.
- Added ordered target primary-key component metadata to accepted
  strong-relation descriptors. Existing scalar relation declarations populate a
  one-component descriptor; composite admission can extend the same shape.
- Moved relation target extraction from accepted source rows to return
  `PrimaryKeyValue` identities, keeping current scalar relation fields as
  `PrimaryKeyValue::Scalar` without exposing a scalar-only target API.

### 🧪 Testing

- Added focused relation tests proving reverse-relation keys and bounds can
  carry complete composite target identity.
- Updated write-boundary guards so relation runtime paths assert accepted
  contracts and `PrimaryKeyValue` identity boundaries instead of scalar target
  component boundaries.

### 🧭 Notes

- Composite relation target declarations are not admitted yet in `0.164.0`.
  This patch prepares the runtime identity and metadata boundary first.
