# 0.157.x

## 0.157.14

### 🔧 Changed

- SQL `CREATE INDEX` and `CREATE UNIQUE INDEX` now accept explicit `ASC`
  ordering on field-path key components.
- `ASC` is treated as syntax for IcyDB's default ascending physical key order
  and is not stored as a separate DDL contract.

```sql
CREATE INDEX character_level_class_idx ON Character (level ASC, class_name ASC);
CREATE UNIQUE INDEX character_level_class_unique_idx ON Character (level ASC, class_name ASC);
```

### 🛡️ Guardrails

- `DESC` remains unsupported until descending index keys have a real
  storage/planner contract.
- Expression and filtered `CREATE INDEX` shapes remain fail-closed.

### 🧪 Validation

- Focused parser tests cover `ASC` normalization and continued `DESC`
  rejection.
- Focused SQL DDL tests cover publishing explicit-`ASC` indexes through the
  same accepted-catalog/rebuild path as default-order indexes.
- Live canister coverage proves generated DDL endpoints publish explicit `ASC`
  as default-order index metadata.

## 0.157.13

### 🔧 Changed

- Enabled SQL `CREATE INDEX` and `CREATE UNIQUE INDEX` for multi-field
  field-path indexes.
- Parser DDL intent now carries ordered field-path key lists instead of a
  single field path.
- SQL DDL binding now resolves every composite key component through accepted
  catalog metadata before deriving the candidate accepted index snapshot.
- `DROP INDEX` now handles DDL-created composite field-path indexes.
- DDL reports preserve the existing `field_path` payload shape by reporting
  composite keys as a comma-separated field list.

```sql
CREATE INDEX character_level_class_idx ON Character (level, class_name);
CREATE UNIQUE INDEX character_level_class_unique_idx ON Character (level, class_name);
SHOW INDEXES FROM Character;
DROP INDEX character_level_class_idx ON Character;
```

### 🛡️ Guardrails

- Expression, filtered, and descending ordered-key `CREATE INDEX` shapes remain
  fail-closed at the SQL surface.
- Generated/model-owned index drops remain rejected; this only widens
  DDL-owned field-path secondary indexes.

### 🧪 Validation

- Focused parser tests cover multi-field `CREATE INDEX` intent.
- Focused SQL DDL tests cover multi-field binding, non-unique and unique
  publication, `SHOW INDEXES`, and `DROP INDEX`.

## 0.157.12

### 🔧 Changed

- Enabled SQL `CREATE UNIQUE INDEX` for single field-path indexes.
- Added staged unique field-path rebuild validation below the SQL DDL surface.
- Unique rebuild staging now compares accepted index components while ignoring
  the primary-key suffix used to distinguish physical index entries.
- Widened schema-owned field-path index mutation admission so accepted unique
  field-path snapshots lower into the same rebuild path as non-unique
  field-path snapshots.
- Widened SQL `DROP INDEX` so DDL-created unique secondary indexes can be
  removed through the same DDL path.
- Renamed internal schema mutation vocabulary from non-unique field-path index
  addition to field-path index addition.
- Renamed SQL DDL report labels to `add_field_path_index` and
  `drop_secondary_index`.

```sql
CREATE UNIQUE INDEX character_name_unique_idx ON Character (name);
SHOW INDEXES FROM Character;
DROP INDEX character_name_unique_idx ON Character;
```

### 🛡️ Guardrails

- Duplicate unique index components fail closed before staged physical work can
  become publishable.
- Expression, filtered, and ordered-key `CREATE INDEX` shapes remain
  fail-closed at the SQL surface.

### 🧪 Validation

- Focused schema mutation tests cover duplicate unique component rejection.
- Focused schema mutation tests cover distinct unique component acceptance.
- Focused planning tests cover unique field-path index lowering and supported
  developer physical-path admission.
- Focused SQL DDL tests cover unique index binding, publication, duplicate
  value rejection, and unique index drop.

## 0.157.11

### 🔧 Changed

- CLI SQL shell routing now classifies `CREATE UNIQUE INDEX ...` as DDL-shaped
  input, so it uses the DDL-gated update endpoint instead of the readonly query
  endpoint.
- Fixed `UNIQUE` keyword tokenization, allowing the parser's specific
  `SQL DDL CREATE UNIQUE INDEX` unsupported-feature diagnostic to surface.

### 🛡️ Guardrails

- `CREATE UNIQUE INDEX` remains unsupported in this patch. The change only
  moves the command to the correct DDL surface so fail-closed diagnostics and
  config gating match its syntax.
- Other unsupported `CREATE UNIQUE ...` shapes remain outside the index-DDL
  router.

### 🧪 Validation

- Focused CLI tests cover supported `CREATE INDEX`, idempotent `CREATE INDEX`,
  unsupported `CREATE UNIQUE INDEX`, `DROP INDEX`, and `DROP INDEX IF EXISTS`
  routing.
- Parser guard coverage pins the `CREATE UNIQUE INDEX` unsupported-feature
  label that the DDL surface now returns.
- Lexer coverage pins `UNIQUE` as a keyword so future DDL diagnostics do not
  drift back to generic identifier parsing.
- Live canister coverage proves `__icydb_ddl` surfaces the specific
  `CREATE UNIQUE INDEX` unsupported-feature diagnostic without publishing
  rejected DDL metadata.

## 0.157.9

### ✨ Added

- Added conservative support for `CREATE INDEX IF NOT EXISTS`.
- Added conservative support for `DROP INDEX IF EXISTS`.
- Added `no_op` DDL execution status for idempotent DDL commands that do not
  mutate accepted schema or physical index storage.

### Samples

```
CREATE INDEX IF NOT EXISTS character_level_idx ON Character (level);
DROP INDEX IF EXISTS character_level_idx ON Character;
```

### 🛡️ Guardrails

- `CREATE INDEX IF NOT EXISTS` no-ops only when the accepted catalog already
  has the exact requested non-unique field-path index contract.
- Conflicting existing index definitions still reject; `IF NOT EXISTS` does
  not hide schema drift.
- `DROP INDEX IF EXISTS` no-ops only when the target index is absent.
- Generated/model-owned and otherwise unsupported index drops still reject even
  with `IF EXISTS`.

### 🧪 Validation

- Focused parser tests cover preserving explicit idempotency intent.
- Focused session tests cover no-op binding, no-op execution reports, and
  rejection of conflicting/generated-index cases.

## 0.157.8

### 🔧 Changed

- SQL DDL idempotency forms now fail with explicit unsupported-feature
  diagnostics for `CREATE INDEX IF NOT EXISTS` and `DROP INDEX IF EXISTS`
  instead of falling through to generic syntax errors.
- `CREATE INDEX ... (field ASC|DESC)` now fails with an explicit unsupported
  key-ordering diagnostic instead of a generic syntax error.
- Interactive `icydb sql` help now includes the supported
  create/show/describe/drop index workflow.

### Samples

```
CREATE INDEX character_level_desc_idx ON Character (level DESC);
```

### 🛡️ Guardrails

- `CREATE INDEX IF NOT EXISTS` and `DROP INDEX IF EXISTS` remain unsupported in
  this patch; `.8` only makes their rejection precise and discoverable.
- SQL DDL index key ordering modifiers remain unsupported; existing generated
  and DDL-created indexes still use IcyDB's current deterministic ordering.
- The shell examples use the already supported DDL and introspection surfaces;
  no new mutation semantics are introduced.

### 🧪 Validation

- Focused parser tests cover the new unsupported-feature labels.
- Focused CLI tests cover the updated interactive shell help text.

## 0.157.7

### 🔧 Changed

- Added index `origin` metadata to structured schema/describe index payloads.
- `DESCRIBE` now renders an `origin` column for indexes, matching the
  generated-vs-DDL distinction already visible through `SHOW INDEXES`.
- `icydb schema` now expands accepted fields, indexes, and relations into
  readable tables instead of only showing aggregate entity-level counts.
- The schema report header now includes total field, index, and relation counts
  so operators can scan schema size before reading the detailed tables.
- Accepted persisted schema remains the source of truth for DDL-created index
  origins; generated model descriptions continue to label model-declared
  indexes as `generated`.

### 🛡️ Guardrails

- This is a hard-cut DTO shape change before `1.0.0`; no legacy Candid shape or
  compatibility fallback is kept.
- The origin label remains descriptive metadata. DDL authority still comes from
  accepted-catalog mutation admission.

### 🧪 Validation

- Focused tests cover the updated Candid shape, shell `DESCRIBE` rendering, and
  canister DDL publication visibility through structured `DESCRIBE`.

## 0.157.3

### 🔧 Changed

- Widened generated index rename reconciliation to preserve extra accepted
  DDL-created indexes while publishing current generated index names.
- Metadata-only rename reconciliation now builds a merged accepted snapshot from
  generated index contracts plus accepted non-generated DDL index contracts.
- Made generated/model-declared `DROP INDEX` rejection actionable at the public
  `execute_sql_ddl` boundary.
- Documented the current SQL DDL contract in the SQL subset contract, including
  the split between DDL-created indexes and generated entity-model indexes.

### 🛡️ Guardrails

- Extra accepted indexes are preserved only when they still lower as supported
  non-unique field-path DDL indexes.
- Generated index contracts still must match by ordinal, store, key,
  uniqueness, and predicate; only the generated name may drift.
- `DROP INDEX` still rejects generated/model-declared indexes; developers must
  remove those indexes from the entity schema macro and let accepted-schema
  reconciliation publish the change.
- SQL DDL can currently drop only non-unique field-path indexes created through
  SQL DDL.

### 🧪 Validation

- Focused transition and reconciliation tests cover generated index renames in
  snapshots that also contain accepted DDL-created indexes.
- Added focused coverage for the user-facing `execute_sql_ddl` generated-index
  drop rejection message.

## 0.157.1

### ✨ Added

- Added SQL `DROP INDEX <name> ON <entity>` support for non-unique
  DDL-published field-path indexes.
- Routed `DROP INDEX` through accepted-catalog binding, schema-owned mutation
  admission, physical index-key cleanup, and accepted-after schema publication.
- Routed `DROP INDEX` from the CLI SQL shell to the generated DDL update
  endpoint instead of the readonly query endpoint.

### 🛡️ Guardrails

- Hard-cut `SHOW INDEXES` to the SQL-style `SHOW INDEXES FROM <entity>` shape,
  with `IN` accepted as the MySQL synonym, instead of keeping IcyDB's older
  `SHOW INDEXES <entity>` shortcut or adding an `ON` variant.
- Hard-cut generated secondary index names from delimiter-style
  `Entity|field` values to SQL-friendly identifiers:
  `idx_<entity>__<field>` for non-unique indexes and
  `uniq_<entity>__<field>` for unique indexes.
- `DROP INDEX` rejects generated/model-declared indexes; only accepted indexes
  added through SQL DDL are droppable in this slice.
- Kept generated-index authority and schema-store publication outside the SQL
  DDL frontend boundary.

### Samples

```
SHOW INDEXES FROM Character;
SHOW INDEXES IN Character;
DROP INDEX character_level_idx ON Character;
```

### 🧪 Validation

- Focused `icydb-core` DDL tests cover parsing, binding rejection for generated
  indexes, accepted schema publication, physical index cleanup, and query-plan
  invalidation guardrails.

## 0.157.0

`0.157.0` opens the DDL continuation line after the 0.156 config work.

### 🔧 Changed

- Re-centered the active minor-line target on SQL DDL and accepted-catalog
  mutation work.
- Left explicit room for focused developer improvements when they support the
  DDL path or simplify the public IcyDB API without changing runtime behavior.
- Recorded the initial design boundary for 0.157 DDL continuation work in
  [docs/design/0.157-ddl-continuation/0.157-design.md](../design/0.157-ddl-continuation/0.157-design.md).
- Reused the shared CLI table formatter for `icydb config show` and clarified
  config command help text around optional ICP environment comparison.

### 📚 Documentation

- Updated README and INSTALLING guidance for `icydb.toml`, `config init`,
  `config show`, `config check`, generated endpoint switches, and local
  troubleshooting.

### 🧭 Migration Notes

- No runtime or persistence migration is introduced by this initial 0.157
  changelog/design slice.
- Future DDL slices should continue to treat accepted schema snapshots as the
  runtime authority. SQL DDL remains a frontend over catalog-native mutation
  semantics, not the source of truth.

### 🧪 Validation

- Documentation-only slice.
