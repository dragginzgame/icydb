# 0.157.x

## 0.157.23

### 🔧 Changed

- SQL DDL now supports `CREATE UNIQUE INDEX` for deterministic expression
  keys backed by the accepted expression-index publication path.
- Unique expression-index rebuilds validate normalized expression keys before
  publication, so duplicate values reject without publishing accepted schema
  metadata or physical index entries.
- `CREATE INDEX IF NOT EXISTS` now recognizes exact existing expression-index
  contracts and returns a `no_op` DDL report instead of attempting to republish.
- Expression-index DDL now rejects duplicate accepted expression-key contracts
  even when the duplicate uses a different index name.

```sql
CREATE UNIQUE INDEX character_lower_name_unique_idx ON Character (LOWER(name));
CREATE INDEX IF NOT EXISTS character_lower_name_idx ON Character (LOWER(name));
```

### 🛡️ Guardrails

- Unique expression-index DDL stays limited to the same supported deterministic
  expression key functions as non-unique expression DDL: `LOWER(field)`,
  `UPPER(field)`, and `TRIM(field)`.
- Duplicate expression keys fail closed during staged rebuild validation before
  the accepted-after schema snapshot is published.
- Conflicting expression index names and duplicate expression key contracts
  remain hard errors; `IF NOT EXISTS` only covers exact accepted-catalog
  matches.

### 🧪 Validation

- Added core execution coverage for successful unique expression-index
  publication and duplicate normalized-key rejection.
- Added live PocketIC coverage proving generated canister DDL publishes a
  unique `LOWER(field)` expression index and exposes it through `SHOW INDEXES`
  with `origin=ddl`.
- Added core and live canister coverage for expression-index
  `CREATE INDEX IF NOT EXISTS` no-op execution.

## 0.157.22

### 🔧 Changed

- SQL DDL expression indexes now bind into accepted catalog metadata instead of
  stopping at the old field-path-only binder gate.
- `CREATE INDEX ... (LOWER(field))`, `UPPER(field)`, and `TRIM(field)` prepare
  as persisted expression key-item metadata with canonical `expr:v1:*`
  expression text.
- Mixed field/expression index keys now preserve key-item order in the
  accepted DDL candidate.
- Accepted-after derivation now classifies expression-index additions as a
  schema mutation shape instead of treating them as an incompatible snapshot
  delta.
- SQL DDL now publishes supported non-unique expression indexes through a
  physical rebuild path that stages expression keys from accepted row slots,
  validates the target physical index entries, marks the index store ready, and
  publishes the accepted-after schema snapshot.
- `DROP INDEX` now supports DDL-published expression indexes as well as
  DDL-published field-path indexes.

### 🛡️ Guardrails

- Expression key sources must still resolve through accepted schema metadata,
  and text functions remain limited to text-backed fields.
- Expression-index schema publication uses the same accepted-before row/schema
  gate as field-path DDL so row changes or schema changes during rebuild fail
  closed before publication.
- Accepted schema compatibility now treats supported DDL-published expression
  indexes as extra accepted indexes, so normal SQL metadata reads keep working
  after publication.

### 🧪 Validation

- Added focused DDL binding coverage for single expression keys and mixed
  field/expression keys.
- Added preparation coverage proving expression-index DDL carries an accepted
  schema derivation and expression rebuild target.
- Added execution coverage proving expression-index DDL publishes physical
  keys, appears in `SHOW INDEXES` with `origin=ddl`, and can be dropped with
  `DROP INDEX`.

## 0.157.21

### 🔧 Changed

- Updated generated IcyDB memory wiring for the Canic 0.38 stable-memory ABI.
- Canister and store schema declarations now carry canonical `memory_namespace` and
  `store_name` values used to build durable allocation keys.
- Generated store memories now use explicit stable-key registration instead of
  the legacy keyless Canic memory macro.
- Commit marker memory configuration now validates both the fixed memory ID and
  the fixed stable key before opening the marker store.
- Fixture, audit, demo, and macro-test schemas now use application-owned stable
  memory IDs in the `100-254` range.
- Renamed the canister-level stable-memory identity from `db_name` to
  `memory_namespace` so schema declarations describe the Canic allocation
  namespace directly instead of implying a separate IcyDB database concept.
- Kept stores on a single `store_name` identity; store memory keys are derived
  from the canister `memory_namespace`, the store `store_name`, and the
  physical memory role.
- Hard-cut the intermediate `db_name` and `stable_name` naming from schema,
  derive, generated-memory wiring, docs, and compile-fail fixtures.
- Generated commit-memory stable keys now use the canonical `commit` segment
  instead of the invalid `__commit` segment required by the intermediate
  naming pass.

### 🛡️ Guardrails

- IcyDB stable keys are canonical lowercase ASCII dot-separated keys ending in
  `.v1`, use underscores rather than hyphens, and reject the reserved
  `canic.` prefix.
- Stable memory keys now read as durable allocation identities such as
  `icydb.demo_rpg.main.data.v1` and
  `icydb.demo_rpg.commit.control.v1`.
- The schema shape remains canister-oriented: one canister memory namespace,
  many named stores, and fixed per-role physical allocations.
- Allocation identity is the `(memory_id, stable_key)` pair. Schema version and
  fingerprint metadata are treated as diagnostics/evolution metadata, not as a
  replacement for the physical allocation identity.
- Schema validation rejects duplicate memory IDs with different keys and
  duplicate stable keys with different memory IDs.

### 🧪 Validation

- Added schema allocation tests proving adding a store or reordering schema
  declarations does not change existing store memory identities.
- Added guard coverage for duplicate key/id conflicts, app memory IDs below
  `100`, reserved `canic.` keys, and schema metadata changes that must not alter
  allocation identity.
