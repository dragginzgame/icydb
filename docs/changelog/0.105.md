# 0.105 Changelog

## 0.105.1

- `0.105.1` extends the same bounded normalization line onto compare orientation, so equivalent admitted compare spellings now converge onto one planner-owned filter shape even when the literal appears on the left-hand side.
- The concrete change is in [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs). `normalize_bool_compare_expr(...)` now flips literal-left compare forms onto the canonical operand order whenever the right-hand side is already one normalized non-literal compare operand. This keeps the scope narrow:
  - no new `WHERE` admission
  - no new partial extraction capability
  - no widened `OR` policy
  - no broader algebraic rewrite
- That means both extractable and residual compare families now have exact parity for equivalent flipped spellings such as:

```sql
WHERE 20 = age
WHERE age = 20
WHERE 'AlphA' = REPLACE(name, 'a', 'A')
WHERE REPLACE(name, 'a', 'A') = 'AlphA'
```

- The extracted-predicate side stays bounded too. This patch does not invent new pushdown shapes; it just ensures the already-admitted compare family lands on one canonical planner expression before predicate subset derivation and explain rendering happen.
- Coverage was added across the same three surfaces used by `0.105.0`:
  - bool-expression normalization tests in [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs)
  - SQL lowering plan-identity tests in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs)
  - exact `EXPLAIN EXECUTION` parity tests in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs)
- As part of the cleanup around these parity tests, the one-off explain helper in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs) was removed. The one remaining searched-`CASE` explain exception is now explicit at the test site instead of being hidden behind a special-purpose helper.
- `0.105.1` still leaves that searched-`CASE` exception in place deliberately. Removing it would require boolean semantic canonicalization for searched `CASE`, which is a broader follow-up than this compare-orientation normalization patch.
- Focused verification for `0.105.1` covered:

## 0.105.0

- `0.105.0` starts the predicate-extraction-normalization slice by making already-admitted boolean `WHERE` trees normalize onto more stable planner-owned shapes before predicate subset derivation runs. The core goal in this patch is predictability, not broader pushdown power.
- The main normalization work lives in [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/bool_expr.rs). Associative boolean trees now normalize through one bounded structural pass that:
  - flattens `AND` chains
  - flattens `OR` chains
  - orders associative children deterministically using the existing planner-owned scalar filter renderer
  - collapses duplicate associative children
- That means equivalent spellings like:

```sql
WHERE age = 20 AND name = 'alpha'
WHERE name = 'alpha' AND (age = 20)
WHERE age = 20 AND age = 20
```

  now converge much more directly onto one canonical boolean-expression shape instead of preserving front-door tree noise.
- `0.105.0` also tightens the extracted predicate side, not just the expression side. `compile_bool_expr_to_predicate(...)` now routes its result through predicate normalization so equivalent normalized boolean trees no longer leave behind nested `Predicate::And` / `Predicate::Or` shells from truth-set compilation.
- The important boundary here is what did *not* change:
  - clause legality is still decided before this phase
  - `0.105.0` does not widen SQL `WHERE` admission
  - `0.105.0` does not add new partial extraction power across mixed trees
  - `0.105.0` does not relax conservative `OR` extraction policy
- The shipped effect is consistency across three layers:
  - planner `filter_expr` shape
  - derived predicate subset shape
  - `EXPLAIN EXECUTION` public output
- Coverage added in this patch pins that consistency for:
  - equivalent pure `AND` trees
  - equivalent pure `OR` trees
  - equivalent mixed `AND` / `OR` trees
  - duplicate associative children such as `a AND a` and `a OR a`
  - both extractable and residual-only boolean subtrees
- SQL lowering parity was pinned in [/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/sql/lowering/tests/mod.rs), proving equivalent admitted `WHERE` spellings now lower onto identical planned query identity for:
  - extractable associative trees
  - residual-only associative trees
  - mixed boolean trees
  - duplicate associative children
- Public explain parity was pinned in [/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/session/tests/sql_explain.rs), and most of the older “ignore `filter_expr` spelling” comparison surface is now gone for the new equivalence families. One older searched-`CASE` explain parity case still intentionally compares routed shape instead of exact semantic filter spelling because the front-door `CASE` spellings remain semantically equivalent but not textually canonicalized at the `filter_expr` label layer yet.
