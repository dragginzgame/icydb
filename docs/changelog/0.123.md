# 0.123 Changelog

## 0.123.1

- `0.123.1` continues the structural-classification consolidation line by moving two more coarse family classifiers onto their root enums: grouped global-DISTINCT aggregate support now lives on `AggregateKind` itself, and intrinsic compare-operator families and range-bound semantics now live on `CompareOp` instead of being re-derived across normalization, validation, simplification, capability, and index compile seams.

- The support-family owner move lands in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/model.rs):
  - added `GlobalDistinctAggregateKind`
  - `AggregateKind` now owns:
    - `global_distinct_kind()`
    - the derived `supports_global_distinct_without_group_keys()` answer now routes through that owner instead of repeating the support set inline

- The grouped semantic projection now consumes that owner in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_distinct.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/semantics/group_distinct.rs):
  - grouped global-DISTINCT aggregate construction no longer matches raw `AggregateKind::{Count, Sum, Avg}` directly
  - the supported grouped DISTINCT aggregate expression is now built through one associated method on `GlobalDistinctAggregateKind`

- The grouped executor handoff now consumes that same owner in [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/group.rs):
  - `GroupedDistinctExecutionStrategy` now owns `from_supported_global_distinct(...)`
  - grouped DISTINCT lowering no longer carries its own second `COUNT | SUM | AVG` strategy-lowering match over raw `AggregateKind`

- `CompareOp` now owns its own intrinsic family and bound semantics in [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/model.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/model.rs):
  - added:
    - `is_equality_family()`
    - `is_ordering_family()`
    - `is_membership_family()`
    - `is_contains_family()`
    - `is_text_pattern_family()`
    - `supports_field_compare()`
    - `lower_bound_inclusive()`
    - `upper_bound_inclusive()`
  - canonical field-order normalization for symmetric field compares now consumes that owner instead of repeating the `Eq | Ne` subset locally

- The predicate normalization, validation, and simplification seams now consume those `CompareOp` owners instead of open-coding the same compare-family buckets:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/normalize.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/normalize.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/schema/validate.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/schema/validate.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/simplify.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/simplify.rs)

- Predicate capability and index compile now also read the same operator-family owner instead of keeping their own first-pass compare ladders:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/capability.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/predicate/capability.rs)
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/index/predicate/compile.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/index/predicate/compile.rs)
  - the index compiler keeps one local `CompareOp -> IndexCompareOp` mapping helper because index opcode selection is still an index-boundary concern, but the branch partitioning now consumes the root operator-family owner instead of repeating raw `CompareOp` subsets twice

- Root plan exports were widened just enough to carry the new planner-owned support family through the normal plan boundary:
  - [/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/mod.rs](/home/adam/projects/icydb/crates/icydb-core/src/db/query/plan/mod.rs)

- This keeps the `0.123.1` cut narrow:
  - grouped global-DISTINCT support ownership moved onto `AggregateKind`
  - grouped semantics and grouped executor handoff now read that owner
  - intrinsic compare-operator family and bound semantics moved onto `CompareOp`
  - normalization, validation, simplify, capability, and index compile now read that owner
  - the larger SQL aggregate lowering classification surface is intentionally left for a separate follow-up instead of widening this patch

- Validation covered:
  - `cargo fmt --all`
  - `cargo check -p icydb-core`
  - `cargo test -p icydb-core grouped_distinct -- --nocapture`
  - `cargo test -p icydb-core compare_predicate_builders_preserve_operator_shape -- --nocapture`
  - `cargo test -p icydb-core grouped_order -- --nocapture`
  - `cargo test -p icydb-core query::fingerprint::hash_parts::grouping -- --nocapture`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`
