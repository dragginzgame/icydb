# 0.90 Changelog

## 0.90.5

- widened the scalar `ORDER BY` expression family just enough to admit the already-supported unary text-function projections `TRIM(field)`, `LTRIM(field)`, `RTRIM(field)`, and `LENGTH(field)` on the same canonical order-expression path that already handled `LOWER(field)` and `UPPER(field)`.
- removed the narrower alias-only failure boundary for those same shapes by resolving unary text-function projection aliases through the planner-owned canonical order-expression renderer instead of leaving them in the `UnsupportedOrderByAlias` bucket, so `ORDER BY trimmed_name` now lowers the same way as `ORDER BY TRIM(name)`.
- representative `0.90.5` query shapes are `SELECT TRIM(name) AS trimmed_name FROM character ORDER BY trimmed_name ASC LIMIT 10` and `SELECT LENGTH(name) AS name_len FROM character ORDER BY name_len DESC LIMIT 10`.
- kept the widening bounded: literal-argument text functions like `LEFT`, `RIGHT`, `STARTS_WITH`, `ENDS_WITH`, `CONTAINS`, `POSITION`, `REPLACE`, and `SUBSTRING` are still outside the admitted scalar order-expression family in this patch.
- added focused parser, lowering, projection, and explain coverage that proves unary text-function aliases and their canonical direct `ORDER BY` spellings stay aligned for execution and explain output.

## 0.90.4

- widened grouped SQL again by removing the grouped-only policy gate that had been rejecting `Predicate::CompareFields(...)` leaves after lowering, so single-entity shapes like `SELECT class_name, COUNT(*) FROM character WHERE strength = dexterity GROUP BY class_name ...` no longer fail just because the same field-to-field predicate moved from scalar SQL into a grouped query.
- kept that fix on the existing grouped residual predicate path instead of inventing a grouped-only compare-fields evaluator: grouped validation now admits the already-lowered predicate tree, and grouped execution keeps using the same compiled predicate program and retained-row filtering contract that scalar SQL already uses for field-to-field comparisons.
- replaced the old fail-closed coverage with plan, lowering, and session tests that prove grouped field-to-field `WHERE` now survives grouped planning and filters rows before aggregation, including a deterministic grouped count case over `SessionDeterministicRangeEntity`.
- the newly admitted grouped field-compare surface includes both equality and ordered compares, so representative `0.90.4` query shapes are `SELECT class_name, COUNT(*) FROM character WHERE strength = dexterity GROUP BY class_name ORDER BY class_name ASC LIMIT 10` and `SELECT level, AVG(strength) FROM character WHERE strength > dexterity GROUP BY level ORDER BY level ASC LIMIT 20`.
- followed that grouped residual-predicate widening through the bounded aggregate-order lane too: grouped strategy selection now reserves `top_k_group` before the residual-predicate streaming fallback fires, so grouped queries like `SELECT class_name, ROUND(AVG((strength + dexterity) / 2), 2) AS avg_balanced_power FROM character WHERE strength >= dexterity GROUP BY class_name ORDER BY avg_balanced_power DESC, class_name ASC LIMIT 10` keep the admitted post-aggregate Top-K path instead of downgrading into scalar order resolution and failing with `unknown field`.

## 0.90.3

- widened the existing bounded arithmetic expression surface from one binary step to recursive chained and parenthesized arithmetic shapes in SQL projection and aggregate-input parsing, so forms like `age + 1 * 2`, `age - 1 - 2`, and `ROUND((age + rank) / 2, 2)` no longer fail just because they need precedence or explicit grouping.
- kept that widening on the existing shared scalar-expression path instead of inventing a second evaluator: recursive `SqlProjectionOperand::Arithmetic(...)` trees now normalize, lower, collect aggregate leaves, and execute through the same planner/runtime expression contracts already used by scalar projections, grouped post-aggregate expressions, and aggregate-input expressions.
- preserved aggregate-input canonicalization while widening the parser by letting constant subexpressions still fold before terminal deduplication, so shapes like `AVG(age + 1 * 2)` keep the right runtime value while the typed aggregate terminal can still canonicalize the constant `1 * 2` fragment down to one stable semantic input.
- added new focused parser, lowering, and session coverage for precedence and parenthesized aggregate-input expressions, plus hard-coded PocketIC sql-canister assertions over the `SqlTestUser` fixture rows that prove the actual returned values for chained scalar arithmetic and chained global aggregate expressions instead of only checking internal AST shape.
- cleaned up another singleton global aggregate combination gap while closing the same line: inert output-row `ORDER BY` on singleton global aggregate results now strips both direct and wrapped aggregate output aliases, so `ORDER BY AVG(...) AS alias` and `ORDER BY ROUND(AVG(...), 2) AS alias` no longer misclassify those output expressions as unknown base-row fields.
- fixed the remaining grouped wrapped-aggregate alias ordering drift on the same admitted surface by canonicalizing numeric literal leaves recursively inside aggregate-input expressions, so grouped Top-K shapes like `ROUND(AVG(age + 1 * 2), 2) AS alias ... ORDER BY alias` now resolve the same aggregate identity through projection lowering, grouped handoff, and grouped order parsing instead of failing with `unknown aggregate expression` once one side had folded `1 * 2` earlier than the other.
- finished the same grouped aggregate alias follow-through for parenthesized arithmetic too: parenthesized arithmetic inside grouped aggregate-input aliases now survives alias normalization and grouped Top-K planning, with the shared SQL label/rendering boundary preserving precedence and the grouped post-aggregate order parser widened to the same bounded chained-arithmetic surface, so shapes like `ROUND(AVG((age + age) / 2), 2) AS avg_balanced ... ORDER BY avg_balanced` no longer flatten into a different expression and fail closed.

## 0.90.2

- widened the dedicated global aggregate SQL lane again so `HAVING` now works over the single implicit aggregate group, which means shapes like `SELECT COUNT(*) ... HAVING COUNT(*) > 1` and `SELECT ROUND(AVG(age), 0) ... HAVING AVG(age) > 30` no longer fail closed just because they omit `GROUP BY`.
- lowered global aggregate `HAVING` onto the same `GroupHavingExpr` contract already used by grouped SQL instead of inventing a second aggregate-result boolean model, so aggregate-only `HAVING` conditions now reuse the existing post-aggregate compare, arithmetic, and `ROUND(...)` expression semantics on one shared planner-owned shape.
- kept the runtime side equally narrow by evaluating global aggregate `HAVING` through the existing grouped post-aggregate evaluator on an empty-key grouped row, which avoids duplicating boolean reduction logic while still keeping direct-field global `HAVING` references fail-closed.
- preserved the global aggregate output projection contract even when `HAVING` rejects the implicit group, so statement payloads still return the same columns and fixed decimal scales with zero rows instead of changing shape based on the filter result.
- added focused lowering and session coverage for matching and failing global aggregate `HAVING` cases, then revalidated the aggregate surface with `cargo fmt --all`, `cargo test -p icydb-core global_aggregate -- --nocapture`, and `cargo clippy -p icydb-core --all-targets -- -D warnings`.

## 0.90.1

- widened the dedicated global aggregate SQL lane from pure aggregate terminal lists to full post-aggregate projection expressions over admitted aggregate leaves, so the engine now executes global shapes like `ROUND(AVG(age), 4)`, `COUNT(*) + 1`, and `MAX(age) - MIN(age)` instead of rejecting them as mixed scalar/aggregate projection lists.
- moved the global aggregate lane onto a planner-owned output `ProjectionSpec` instead of parser-owned label overrides, which means aliases, canonical labels, fixed decimal scale metadata, and aggregate-leaf deduplication now all flow through the same projection contract that runtime evaluates after reducing the unique aggregate terminals.
- taught scalar-lane lowering to detect wrapped aggregate leaves before planning, so `ROUND(AVG(age), 4)` and similar shapes now route into the dedicated global aggregate lane consistently for normal execution and `EXPLAIN` instead of drifting through the scalar projection surface.
- kept the older terminal-dedup contract intact for direct aggregate lists while intentionally stopping the structural runtime from depending on the legacy top-level terminal remap once wrapped post-aggregate output expressions are present.
- added focused coverage for lowering, execution values, payload labels, and fixed-scale metadata on the new post-aggregate global shapes, then validated the slice with `cargo fmt --all`, `cargo test -p icydb-core global_aggregate -- --nocapture`, and `cargo clippy -p icydb-core --all-targets -- -D warnings`.

## 0.90.0

- started the SQL-surface closure line by turning several broad lowering fallback buckets into clause-specific semantic errors, so the admitted single-entity SQL boundary is now enforced at the lowering layer instead of leaking through generic `UnsupportedSelect*` paths.
- added a dedicated global-aggregate projection boundary for the pure aggregate lane, which now rejects mixed scalar-plus-aggregate lists and grouped shapes through explicit global-aggregate lowering errors instead of reusing the older generic grouped bucket.
- made `HAVING` without `GROUP BY` a first-class lowering error on both the normal SQL lane and the dedicated global-aggregate lane, so forms like `SELECT * ... HAVING COUNT(*) > 1` now fail as the specific semantic mistake they are.
- split grouped projection rejection into narrower user-facing failures:
  - grouped `SELECT *` now fails as an explicit grouped projection shape error instead of one generic unsupported-select fallback
  - grouped field-only projection lists now fail as grouped projection shapes that are missing an aggregate term
  - grouped projection terms that escape grouped-key authority still report the first offending projection index
  - grouped scalar projection terms that appear after the aggregate phase still fail with a dedicated post-aggregate ordering error
- kept the closure work mechanically clean by fixing the stray grouped-fold test import drift and the missing prepared-execution snapshot lint annotation that workspace `clippy` surfaced while validating the new SQL boundary.
- validated the shipped `0.90.0` slice with `cargo fmt --all`, `make clippy`, and `make test-unit`, including the PocketIC-backed `sql_canister` integration tail and the focused SQL lowering regressions for each new semantic error.
