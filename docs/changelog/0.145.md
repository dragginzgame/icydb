# 0.145.x

## 0.145.8

`0.145.8` narrows SQL projection row ownership in two local materialization
paths. DISTINCT rows accepted only to advance the post-DISTINCT OFFSET window
are now consumed directly into the canonical key set. One-component pure
covering projections now move the decoded index value into the projected SQL
row at its final use instead of borrowing it and cloning every occurrence, and
multi-component pure covering rows now do the same for each decoded component.
Hybrid covering projections now do the same for their per-row decoded covering
component and sparse row-field maps.
Public DISTINCT ordering, LIMIT/OFFSET, NULL, expression, duplicate projection
column, and map-canonicalization behavior stay unchanged.

### DISTINCT key storage

- Added an owned insertion lane for projected DISTINCT rows that are accepted
  before the output window.
- Moved owned projected rows into `GroupKey::from_group_values(...)` for
  pre-offset accepted keys instead of cloning `row.values().to_vec()`.
- Preserved the existing borrowed insertion lane for rows that are emitted,
  because those rows must still be retained for output.
- Kept map-valued rows on the grouped-key canonicalization surface so malformed
  map behavior and canonical map equality remain unchanged.

### Covering projection row assembly

- Changed the one-component pure covering projection shaper to consume its
  decoded `Value`.
- Moved the decoded index value into the final matching projected column.
- Preserved cloned values for repeated references to the same component because
  each output column remains independently owned.
- Added an owned multi-component pure covering shaper that moves each decoded
  component into its final matching projected column.
- Changed hybrid covering row assembly to consume its per-row decoded component
  and sparse row-field maps.
- Moved decoded hybrid covering and row-backed values into their final matching
  projected columns while preserving clones for repeated references.
- Left DISTINCT, reordered, and cursor-sensitive materialization boundaries
  unchanged.

## 0.145.7

`0.145.7` removes remaining owned-copy boundaries from aggregate terminal
helpers. Row-returning `top_k_by(field, k)` and `bottom_k_by(field, k)` still
rank over the same materialized effective response window, but selected rows
now move out of that window into the typed response instead of cloning full
`DataRow` payloads after ranking. `top_k_by_with_ids(field, k)` and
`bottom_k_by_with_ids(field, k)` now move selected `DataKey`s out of the same
materialized window without wrapping every source row in a temporary option
slot. The materialized `min_max(field)` helper now also clones the projected
field value only when one candidate must seed or replace both extrema states;
min-only and max-only replacements move the value directly into the selected
state.

### Ranked row terminals

- Changed row-returning ranked field terminals to consume the materialized page
  only for `Rows` output.
- Changed values-with-ids ranked field terminals to consume the materialized
  page and move selected `DataKey`s out instead of cloning them.
- Moved ranked winners with descending-index `swap_remove` so the selected
  winner set is extracted without building one `Option<DataRow>` wrapper per
  source row.
- Kept value-only ranked projections on the borrowed page path because they do
  not return full rows or ids.
- Preserved deterministic `(field value, primary key)` ordering and the
  existing ranked window selection.

### Paired extrema helper

- Reordered `min_max(field)` candidate updates so comparisons finish before
  ownership moves into the selected extrema state.
- Kept the one required clone when the same projected value must be retained
  as both min and max.

### DISTINCT scalar reducer

- Replaced the retained NULL admission clone with a direct `Value::Null`
  marker insert.
- Added a consuming value-reducer finalization path for scalar aggregate
  terminals so selected extrema move out of reducer state instead of cloning at
  terminal finalization.

## 0.145.6

`0.145.6` removes the raw-row staging layer from DELETE execution and narrows
route-ordered covering projection page windows. The delete key-stream
resolver now maps each present row directly into the caller's post-access
candidate shape: typed delete rows for fluent/typed delete paths and
structural kernel rows for SQL `DELETE ... RETURNING` projection. The existing
post-access vector boundary remains because DELETE still needs the same
filtering, ordering, LIMIT/OFFSET, and rollback selection semantics.
Route-ordered covering `bytes_by(field)` pages now fold byte lengths only for
the requested page window after preserving row-presence checks, route-ordered
scalar projection terminals now decode only values inside their effective page
window, and pure SQL covering projections now drop offset-discarded raw
component pairs before one- and multi-component decode.

### DELETE row resolution

- Replaced `resolve_delete_candidate_rows(...) -> Vec<DataRow>` with a
  mapper-based resolver that returns the final delete candidate row vector and
  the same present-row metric count.
- Typed delete paths now decode raw rows into `DeleteRow<E>` during key-stream
  consumption instead of staging `DataRow` first.
- SQL structural delete projection now decodes present rows into `KernelRow`
  during key-stream consumption, then runs the existing delete post-access and
  RETURNING packaging flow unchanged.

### Covering bytes windows

- Added an index-order covering `bytes_by(field)` fold that keeps the existing
  row-presence checks over raw covering candidates but decodes byte lengths only
  for rows inside the effective page window.
- Primary-key-reordered covering byte paths stay on the existing materialized
  decoder because they must compare the full candidate set before windowing.
- Added parity coverage against the full-scan materialized baseline for paged
  index-ordered `bytes_by(rank)`.

### Covering projection windows

- Added an index-order covering scalar projection fold that keeps row-presence
  accounting over all covering candidates while decoding only values inside the
  effective `OFFSET/LIMIT` window.
- Kept primary-key-reordered covering projection paths on the existing
  materialized decoder because they must reorder the full candidate set before
  paging.
- Preserved covering DISTINCT behavior by keeping the same effective-window
  ordering before DISTINCT admission.

### Pure covering SQL projections

- Dropped index-ordered raw covering component pairs before decode when the
  planner-proven pure covering route has already bounded the scan to the page
  window.
- Kept DISTINCT, row-presence-checking, and primary-key-reordered pure covering
  paths on the existing materialized decode boundary.
- Preserved public projection output rows while reducing decode work for
  `SELECT field FROM table ORDER BY indexed_field LIMIT/OFFSET` style pure
  covering routes.

## 0.145.5

`0.145.5` starts the deferred `INSERT ... SELECT` staging cleanup and narrows
one remaining sparse covering projection window. SQL `INSERT` now builds the
structural mutation batch directly from each source shape: `VALUES` rows are
borrowed from the parsed statement instead of cloned into a temporary
source-row vector, and `INSERT ... SELECT` projected rows are consumed into
patches after width validation. Hybrid covering projections now also skip
OFFSET-discarded index-component decode and projection work for index-ordered,
non-`DISTINCT` pages after row-presence filtering.

### SQL INSERT staging

- Removed the full `VALUES` source-row clone from SQL INSERT execution.
- Consumed `INSERT ... SELECT` projected rows directly into structural patches
  instead of routing both VALUES and SELECT through one shared source-row
  staging vector.
- Preserved tuple-width validation, INSERT SELECT width errors, generated and
  managed field checks, atomic batch mutation behavior, and RETURNING shaping.

### Hybrid covering projection paging

- Moved index-ordered, non-`DISTINCT` hybrid covering OFFSET handling into the
  sparse-row assembly loop.
- Kept row-presence filtering before the skip so OFFSET still counts only rows
  that would have been projected.
- Avoided decoding projected index components and building projected rows for
  skipped OFFSET candidates.

## 0.145.4

`0.145.4` closes more scalar projection staging. `distinct_values(field)` still
retains accepted values because it returns them, but `COUNT(DISTINCT field)`
now counts decoded admissions directly in both materialized and covering-index
scalar projection paths. Covering-index first/last value terminals now also
select from the already-windowed structural pair vector instead of allocating a
second value-only vector and cloning the selected value. Global DISTINCT scalar
aggregate decode now moves the singleton grouped output value out of the owned
page row instead of cloning it. Scalar aggregate terminal DISTINCT ingestion now
also borrows admitted values for reducer inspection before moving them into the
retained DISTINCT admission set. SQL `RETURNING field, ...` projection over
owned rows no longer builds a full-row optional slot vector just to move the
selected fields out. Scalar aggregate field/expression reducers now also
borrow source values for reducer ingestion instead of cloning every input
candidate before the reducer knows whether it needs ownership.

### Count-distinct scalar projections

- Materialized scalar projection counts decode field values and update the
  canonical DISTINCT set in one pass, without first collecting `Vec<Value>`.
- Covering-index scalar projection counts from the already-windowed
  `(DataKey, Value)` pairs instead of allocating a second value-only vector.
- Removed the old count-only covering helpers that accepted owned value vectors.

### First/last scalar value terminals

- Covering-index `first_value_by(field)` and `last_value_by(field)` now move
  the selected value out of the ordered structural pair vector.
- Preserved prepared windowing, row-presence policy, and first/last terminal
  validation.

### Global DISTINCT scalar decode

- Global DISTINCT scalar aggregate decode now consumes the owned singleton
  grouped output row and moves the one aggregate value out.
- Kept the existing zero-key and single-aggregate-value invariant checks.

### Scalar DISTINCT reducer ingestion

- Scalar aggregate terminal DISTINCT ingestion now checks duplicate admission,
  lets the reducer inspect the accepted value by reference, and then moves the
  value into the retained DISTINCT set.
- Scalar aggregate terminal field and expression reducers now ingest borrowed
  source values. Non-DISTINCT COUNT/SUM/AVG paths avoid cloning entirely, and
  extrema clone only when the reducer stores a selected candidate.
- Preserved NULL skipping, numeric coercion, extrema comparison, and duplicate
  admission behavior.

### SQL RETURNING projection

- Shared SQL `RETURNING field, ...` shaping now precomputes the selected input
  slots and moves selected values into an output-width buffer.
- Removed the per-row full-column `Vec<Option<Value>>` staging previously used
  to take selected fields out of owned rows.
- Kept duplicate-field and row/column alignment errors on the same invariant
  surface.

## 0.145.1

`0.145.1` continues the streaming-boundary cleanup on paged scalar loads,
covering SQL projections, and aggregate field terminals. The slice preserves
ordering, cursor, SQL response, and aggregate behavior while removing staging
that only existed to discard rows later.

### Direct scalar load paging

- Taught the route-satisfied direct `DataRow` scan lane to skip cursorless page
  offset rows during collection instead of staging them and dropping them in the
  final page-window pass.
- Taught the cursorless retained-slot short path to apply the same scan-time
  offset skip, with the post-scan tail explicitly recognizing that the page
  window was already applied.
- Preserved existing scan accounting by tracking matched rows separately from
  retained rows, so diagnostics still report the pre-page matched candidate
  count.
- Added lane-metric coverage for an `OFFSET 1 LIMIT 2` direct-row page to pin
  both the early stop and the returned post-offset rows.

### SQL covering projection paging

- Taught non-`DISTINCT`, planner-proven pure covering projections in index or
  primary-key order to skip page offset rows before projected row assembly.
- Taught primary-store pure covering projection to skip offset keys while
  collecting planner-proven primary-key rows, so it no longer stages keys that
  the projected-row assembly phase would immediately discard.
- Kept missing-row-checking, reordered, and `DISTINCT` covering paths on their
  existing materialized paging boundaries.
- Reused the existing scan-limit contract so bounded covering projections still
  scan only `offset + limit` input candidates where the planner proves that is
  safe.

### Field aggregate staging

- Materialized aggregate field projection terminals that return only values now
  decode directly to `Vec<Value>` instead of first building `(DataKey, Value)`
  pairs.
- `values_with_ids(field)` stays on the keyed projection path because that
  public terminal returns ids with values.
- Scalar aggregate terminal finalization now writes reducer outputs directly
  into terminal-order slots instead of first collecting a finalized-output
  vector.
- Left `nth(field)` and `median(field)` materialized because those ordered
  terminals require sorted candidate vectors.

## 0.145.0

`0.145.0` starts the streaming-boundary cleanup by removing accidental staging
buffers from bounded grouped output, SQL write-returning projection, and
materialized `min_max(field)` aggregate helpers. The slice preserves existing
ordering, cursor, HAVING, SQL response, and aggregate result behavior while
pushing ownership to the point where each path actually needs it.

### Bounded grouped output

- Streamed HAVING/resume-qualified grouped `COUNT(*)` window candidates directly
  into the retained bounded heap instead of first collecting a `qualifying`
  vector.
- Replaced grouped page-finalization pre-collection with a visitor-style
  candidate handoff, so bounded grouped pages can retain only the selected
  candidates.
- Kept full candidate materialization for unbounded grouped output, where global
  ordering still requires the complete candidate set.

### SQL write-returning projection

- Taught `INSERT` and `UPDATE ... RETURNING field_list` to project requested
  fields directly from typed mutation outputs instead of building complete rows
  first.
- Updated shared SQL `RETURNING` field-list shaping over owned rows to move
  selected values out of rows instead of cloning them.
- Kept `RETURNING *` on the full-row path because the public response requires
  every field.

### Field aggregate helper staging

- Removed the second projected-row vector from the materialized `min_max(field)`
  aggregate helper.
- The helper now decodes each projected field value and folds it directly into
  the min/max candidate state.
- Left `nth(field)` and `median(field)` materialized because those ordered
  terminals require sorted candidate vectors.
