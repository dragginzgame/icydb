# 0.97 Changelog

## 0.97.1

- `0.97.1` is a narrow DISTINCT follow-through patch over the shipped `0.97.0` contract rather than another new DISTINCT design slice:
  - it keeps the same SQL-only scope
  - it does not widen fluent distinct
  - it does not add scalar SQL continuation tokens
  - it focuses on locking the already-admitted projected-tuple and post-dedup paging rules across more real execution and front-door surfaces
- the most important fix in `0.97.1` is one real runtime bug on the expression-indexed SQL path:
  - `SELECT DISTINCT LOWER(name) ... LIMIT ... OFFSET ...` exposed that the SQL projection runtime was still letting the original scalar page window leak into execution before final DISTINCT dedupe on one ordered expression-backed path
  - that violated the intended `0.97` contract because paging could still depend on raw pre-dedup rows instead of the ordered deduped projected stream
  - the repair stays SQL-local:
    - SQL projection execution now runs DISTINCT queries against an unpaged scalar plan clone
    - the original `LIMIT` / `OFFSET` stays deferred to final SQL DISTINCT materialization after projected-row dedupe
  - this preserves the ordinary non-DISTINCT scalar paging path while making the DISTINCT runtime contract real on expression-backed covering routes too
- `0.97.1` then widens the proof surface around that repaired contract:
  - live SQL session coverage now proves `SELECT DISTINCT LOWER(name) ... ORDER BY LOWER(name)` dedupes compiled expression outputs rather than row identity or source-text casing
  - high-offset expression-backed DISTINCT paging now proves empty pages return cleanly without panicking or depending on raw scanned-row count
  - EXPLAIN, EXPLAIN JSON, and EXPLAIN EXECUTION now all preserve the same projected-tuple rejection when `SELECT DISTINCT` tries to order by non-projected fields
- the projected-tuple boundary itself is also tightened further in `0.97.1`:
  - direct non-projected fields were already rejected in `0.97.0`
  - `0.97.1` now locks the same fail-closed behavior for:
    - wrapped non-projected fields such as `ORDER BY LOWER(age)`
    - source fields hidden behind expression projections such as `SELECT DISTINCT LOWER(name) ... ORDER BY name`
  - that keeps the DISTINCT rule honest: ordering must be derivable from the outward projected distinct tuple, not recoverable from hidden source-row payloads through wrappers or expression backdoors
- `0.97.1` also makes one boundary explicit in the design notes:
  - top-level scalar SQL statement payloads still do not expose continuation tokens
  - grouped SQL remains the only SQL statement surface with `next_cursor`
  - so the shipped `0.97` DISTINCT proof is intentionally about `LIMIT` / `OFFSET` semantics on scalar SQL, not scalar SQL continuation semantics
- the architectural result after `0.97.1` is clearer and more defensible:
  - projected-tuple ordering rules are now pinned on plain fields, wrapped fields, and source fields behind expression projections
  - post-dedup DISTINCT paging is now pinned on plain scalar values, nullable values, expression outputs, and high-offset empty pages
  - EXPLAIN surfaces follow the same fail-closed DISTINCT ordering boundary as live SQL execution
- focused verification for `0.97.1` covered the new runtime repair and the widened DISTINCT boundary matrix:
  - `cargo fmt --all`
  - `cargo test -p icydb-core explain_sql_distinct_rejects_order_by_non_projected_field -- --nocapture`
  - `cargo test -p icydb-core execute_sql_projection_distinct_dedupes_expression_outputs -- --nocapture`
  - `cargo test -p icydb-core execute_sql_projection_distinct_expression_high_offset_returns_empty_rows -- --nocapture`
  - `cargo test -p icydb-core compile_sql_command_rejects_distinct_order_by_wrapped_non_projected_field -- --nocapture`
  - `cargo test -p icydb-core execute_sql_select_distinct_rejects_order_by_wrapped_non_projected_field -- --nocapture`
  - `cargo test -p icydb-core compile_sql_command_rejects_distinct_order_by_direct_field_from_expression_projection -- --nocapture`
  - `cargo test -p icydb-core execute_sql_select_distinct_rejects_order_by_source_field_from_expression_projection -- --nocapture`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`

## 0.97.0

- `0.97.0` starts the DISTINCT follow-through line with a deliberately narrow SQL contract repair rather than a broader distinct redesign:
  - it does not widen fluent distinct
  - it does not add `DISTINCT ON`
  - it does not add new index-aware distinct execution
  - it focuses on making the shipped SQL `DISTINCT` surface fail closed where needed and on locking the runtime ordering/paging semantics that were already intended
- the first repair is at SQL lowering time for `SELECT DISTINCT ... ORDER BY ...`:
  - scalar SQL `DISTINCT` now rejects order terms that are not derivable from the projected distinct tuple
  - the admitted rule is intentionally narrow:
    - exact projected expressions are allowed
    - expressions that can be written entirely from directly projected fields are allowed
    - hidden non-projected base-row fields are rejected
  - this prevents the engine from pretending it can rank the post-distinct stream by information that is no longer present after projection
- user-visible result:
  - forms such as:

```sql
SELECT DISTINCT name
FROM Character
ORDER BY age ASC;
```

  now fail cleanly with:
  - `SELECT DISTINCT ORDER BY terms must be derivable from the projected distinct tuple`
  - while bounded projected-tuple-derived forms such as `ORDER BY LOWER(name)` remain admitted
- the second repair is a runtime contract lock for paging over SQL `DISTINCT` results:
  - the final SQL projection materializer now documents and preserves the intended order of operations:
    - projection
    - distinct-row dedupe
    - `LIMIT` / `OFFSET`
  - the pure covering and hybrid covering fast paths now also make the same contract explicit:
    - no pre-window truncation under `DISTINCT`
    - no local `LIMIT` / `OFFSET` application before projected-row dedupe
  - this keeps `SELECT DISTINCT ... ORDER BY ... LIMIT ... OFFSET ...` defined over the ordered deduped stream rather than over raw scanned rows
- that paging lock-in is now covered with duplicate-heavy SQL session tests:
  - `LIMIT` returns the first `N` distinct ordered rows, not the first `N` raw rows
  - `OFFSET` skips distinct ordered rows, not raw rows
  - duplicate-heavy fixtures now prove this directly instead of relying on implicit behavior
- `0.97.0` also locks one nullable `DISTINCT` edge that commonly regresses when paging and dedupe logic move:
  - repeated `NULL` projected values now have explicit SQL session coverage proving they collapse to one distinct row on the projected stream
  - paging over that stream is also pinned, so `NULL` participates once in the ordered deduped result before `LIMIT` / `OFFSET` are applied
- the important architectural boundary after `0.97.0` is still narrow:
  - this is a SQL `DISTINCT` contract repair
  - it does not change the broader query planning model
  - it does not add a new distinct runtime family
  - it makes the shipped DISTINCT line more explicit, more predictable, and easier to keep correct
- focused verification for `0.97.0` covered lowering, live session behavior, null-dedup behavior, formatting, and linting:
  - `cargo fmt --all`
  - `cargo test -p icydb-core compile_sql_command_accepts_distinct_order_by_expression_derived_from_projected_field -- --nocapture`
  - `cargo test -p icydb-core compile_sql_command_rejects_distinct_order_by_non_projected_field -- --nocapture`
  - `cargo test -p icydb-core execute_sql_select_distinct_rejects_order_by_non_projected_field -- --nocapture`
  - `cargo test -p icydb-core execute_sql_projection_distinct_limit_and_offset_use_deduped_ordered_stream -- --nocapture`
  - `cargo test -p icydb-core execute_sql_projection_distinct_treats_null_as_one_deduped_ordered_value -- --nocapture`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`
