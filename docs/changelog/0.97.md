# 0.97 Changelog

## 0.97.0

- `0.97.0` starts the DISTINCT follow-through line with a deliberately narrow SQL contract repair rather than a broader distinct redesign:
  - it does not widen fluent distinct
  - it does not add `DISTINCT ON`
  - it does not add new index-aware distinct execution
  - it focuses on making the shipped SQL `DISTINCT` surface fail closed where needed and on locking the runtime ordering/paging semantics that were already intended
- the first repair is at SQL lowering time for `SELECT DISTINCT ... ORDER BY ...`:
  - scalar SQL `DISTINCT` now rejects order terms that are not derivable from the projected distinct tuple
  - the admitted rule is intentionally narrow:
    - exact projected expressions are allowed
    - expressions that can be written entirely from directly projected fields are allowed
    - hidden non-projected base-row fields are rejected
  - this prevents the engine from pretending it can rank the post-distinct stream by information that is no longer present after projection
- user-visible result:
  - forms such as:

```sql
SELECT DISTINCT name
FROM Character
ORDER BY age ASC;
```

  now fail cleanly with:
  - `SELECT DISTINCT ORDER BY terms must be derivable from the projected distinct tuple`
  - while bounded projected-tuple-derived forms such as `ORDER BY LOWER(name)` remain admitted
- the second repair is a runtime contract lock for paging over SQL `DISTINCT` results:
  - the final SQL projection materializer now documents and preserves the intended order of operations:
    - projection
    - distinct-row dedupe
    - `LIMIT` / `OFFSET`
  - the pure covering and hybrid covering fast paths now also make the same contract explicit:
    - no pre-window truncation under `DISTINCT`
    - no local `LIMIT` / `OFFSET` application before projected-row dedupe
  - this keeps `SELECT DISTINCT ... ORDER BY ... LIMIT ... OFFSET ...` defined over the ordered deduped stream rather than over raw scanned rows
- that paging lock-in is now covered with duplicate-heavy SQL session tests:
  - `LIMIT` returns the first `N` distinct ordered rows, not the first `N` raw rows
  - `OFFSET` skips distinct ordered rows, not raw rows
  - duplicate-heavy fixtures now prove this directly instead of relying on implicit behavior
- `0.97.0` also locks one nullable `DISTINCT` edge that commonly regresses when paging and dedupe logic move:
  - repeated `NULL` projected values now have explicit SQL session coverage proving they collapse to one distinct row on the projected stream
  - paging over that stream is also pinned, so `NULL` participates once in the ordered deduped result before `LIMIT` / `OFFSET` are applied
- the important architectural boundary after `0.97.0` is still narrow:
  - this is a SQL `DISTINCT` contract repair
  - it does not change the broader query planning model
  - it does not add a new distinct runtime family
  - it makes the shipped DISTINCT line more explicit, more predictable, and easier to keep correct
- focused verification for `0.97.0` covered lowering, live session behavior, null-dedup behavior, formatting, and linting:
  - `cargo fmt --all`
  - `cargo test -p icydb-core compile_sql_command_accepts_distinct_order_by_expression_derived_from_projected_field -- --nocapture`
  - `cargo test -p icydb-core compile_sql_command_rejects_distinct_order_by_non_projected_field -- --nocapture`
  - `cargo test -p icydb-core execute_sql_select_distinct_rejects_order_by_non_projected_field -- --nocapture`
  - `cargo test -p icydb-core execute_sql_projection_distinct_limit_and_offset_use_deduped_ordered_stream -- --nocapture`
  - `cargo test -p icydb-core execute_sql_projection_distinct_treats_null_as_one_deduped_ordered_value -- --nocapture`
  - `cargo clippy -p icydb-core --all-targets -- -D warnings`
